14.2 백엔드 기본 설정 및 데이터 조작

백엔드 스토리지 플러그인을 라우터 데몬에 성공적으로 주입(Load)하였다면, 다음 단계는 해당 데이터베이스 인프라에 트래픽 데이터를 맵핑(Mapping)하고 영속화(Persistence)를 지시하는 제어 규칙을 수립하는 것이다.
이 장에서는 가장 기초적인 JSON 기반 구성 파일(Configuration File) 조작법부터 시작하여, 외부 관제망 서버가 HTTP 브릿지를 경유하여 데이터베이스 레코드 단위로 REST API CRUD(Create, Read, Update, Delete) 명령을 전개하고 검증하는 실전 운영 런북(Runbook)을 상세히 기술한다.

1. JSON 설정 파일을 이용한 스토리지 마운트(Mount) 및 언마운트

라우터 데몬이 최초 기동(Booting)되는 단계에서, 컴포넌트가 어떠한 유형의 DB 드라이버를 탑재할 것이며, 어느 범위의 가상 토픽(Key Expression) 스레드 공간을 훔쳐보고(Sniffing) 영속화할 것인지를 전역 설정 파일(zenohd.json5) 내부에 하드 코딩하여 규율한다.

1.0.1 선언적 볼륨 마운트(Mount) 설계 전술

1) 영구 마운트 (정적 볼륨 마운트 설정)

{
  listen: ["tcp/0.0.0.0:7447"],
  storages: {
    // 관리자가 임의로 지정하는 논리적 스토리지 볼륨의 식별자(ID)
    my_memory_db_volume: {
      // 1. [스니핑 범위 지정] 어떠한 토픽 공간을 스토리지로 캡처하여 저장할 것인가?
      key_expr: "robot/sensors/**",
      // 2. [물리 드라이버 배정] 백엔드 레코드 기록에 어떠한 플러그인 워커 타입을 배정할 것인가? (예: memory 백엔드)
      backend: "memory"
    }
  }
}

인프라 관리자가 터미널에서 zenohd -c zenohd.json5 명령으로 라우터 데몬을 킥-오프(Kick-off)하는 즉시, 지정된 메모리 기반 스토리지 플러그인이 로드되어 RAM 상에 완벽하게 독립적인 DB 인스턴스 테이블 공간을 오픈하고 해당 키 범위의 데이터 조각들을 쓸어 담아 보존하기 시작한다.

2) 런타임 핫-언마운트(Hot-Unmount) 제어 및 관제 룰
거대한 파이프라인이 라이브(Live) 런타임으로 서비스 포워딩을 수행하고 있는 와중에도, 전체 시스템 셧다운 없이 특정 스토리지만 조용히 내리거나(Unmount) 추가 마운트하고 싶다면 라우터가 제공하는 마스터 관리자 REST API(admin 플러그인 엔드포인트)에 PUT/DELETE 명령을 타일러 동적 제어를 수행할 수 있다.
그러나 가용성(Availability)과 인프라 형상(Configuration)의 일관성 동기화 측면에서, 크리티컬한 스토리지 볼륨의 마운트 정책은 “재부팅 시 파일 기반 형상 도면에 의해 무조건 정적으로 묶이는 불변 아키텍처(Immutable Architecture)“를 기준 런북으로 채택하여 운영하는 것이 시스템 생존성을 높이는 엔터프라이즈 아키텍트의 보수적 철칙이다.

2. 스토리지 생성, 읽기, 갱신, 삭제 (CRUD) 기본 연산 매핑

Zenoh 네트워크 스토리지 모듈로 데이터 레코드를 기록, 제어한다고 해서 별도의 이질적인 SQL(Structured Query Language)이나 ORM 시스템을 치고 통신하는 것이 아니다. Zenoh 스토리지에 대한 CRUD 트랜잭션은 철저하게 기저 통신 프로토콜의 표준 패킷형(Pub/Sub, Query 패러다임) 네이티브 함수 콜백을 그대로 재활용하여 단일화(Unification) 성취를 이룬다.

2.0.1 프로토콜 네이티브 오버헤드 제로(Zero-Overhead) 데이터 매니지먼트

상단 설정에 따라 스토리지 봇(Bot)이 robot/sensors/** 통신 영역을 마운트 감시망에 넣고 대기 중이라고 상정한다.

1) Create & Update (레코드 생성 및 패킷 갱신)

z_put "robot/sensors/temp/room1" "25.0"

Zenoh 코어 라우터는 데이터 객체의 Put(공급)과 Update(수정)의 이질적 분기 처리를 구분하지 않는다. 발간자 단말이 허공에 Put 선언을 타격하여 뿌리면 (본래는 다른 로봇이 수신하기 위해 날아가는 브로드캐스트 패킷 명령임에도 불구하고) 백그라운드 스토리지 데몬도 인그레스 소켓에서 이 덤프를 가로채 읽고 백엔드 DB 버퍼에 집어넣는다. 레코드 데이터가 존재하지 않으면 새로 인서트(Create)하고, 이미 구 과거 객체가 점유 중이라면 최신값으로 매끄럽게 오버라이트 덮어쓰기(Update)를 묵묵히 수행한다.

2) Read (레코드 읽기 및 데이터 추출)

z_get "robot/sensors/temp/room1"

Get 질의 핑(Query Ping) 패킷이 해당 토픽 범위의 터널 입구에 부딪히는 순간, 스토리지 백엔드 플러그인 인스턴스는 자기가 마스터 권한을 포유한 하드디스크 파티션(혹은 인-메모리 캐시 테이블)을 역 스캔 검색하여, 과거에 영속하여 아카이빙(Archiving) 저장해 두었던 25.0 이란 문자열 덤프 텍스트를 인코딩 발췌하여 네트워크 망을 역주행시키는 Reply 패킷 구조 장착을 통해 질문자에게 송출 대답해 보낸다.

3) Delete (가비지 레코드 파기 및 삭제 추방)

z_delete "robot/sensors/temp/room1"

물리적 바이트 상태의 데이터는 통신망을 허공 비행하다 라우팅 종료 후 램버퍼에서 자연 소멸하여 증발하지만, 위처럼 강제 영속결박(Persistence Bound)되어 락온 아카이브 스토리지 무덤에 새겨진 데이터 찌꺼기는 명시적으로 디스크 스왑 클리어 지시를 내리지 않으면 종국엔 하드디스크 풀 용량이 100% 터질 때까지 좀비처럼 영구 남는다. 망에 Delete 제어 패킷을 발사 전송하면, 이 지시를 스니핑한 스토리지 플러그인 에이전트가 백엔드 DB 단 커널에 Drop 컷아웃(Cut-out) 인서트 쿼리를 자체 매핑 번역 때려 물리 로우 디스크 파티션 용량을 안전하게 캐리지 리턴 비워 해제시켜 준다.

3. REST API를 활용한 백엔드 인터페이스 직접 접근 및 검증

웹 모니터링 대시보드(React/Vue 기반) 프론트 뷰 렌더링을 관장하는 웹 개발자에게 “C++이나 Rust SDK를 시스템 포트 단에 직접 설치 컴파일하여 z_get 소켓 통신 모듈을 작성하라“고 강압할 수는 없다.
이를 타개하기 위해, Zenoh 라우터는 고수준 외부 앱 스택 관제망이 접근하여 포워딩 통신할 수 있는 훌륭한 다목적 HTTP 브릿지(REST API) 이그레스 플러그인을 아키텍처 기본 번들로 내장 보유하고 있다.

3.0.1 웹 B2B 관제망과의 접합 및 크로스 체킹 런북

라우터 데몬 부팅 세팅에서 rest 플러그인을 인라인 환경변수로 활성화(zenohd --rest-http-port 8000) 시켰다고 가정한다. 이제 데몬 스토리지의 철통 보안 락에 대한 모든 CRUD 진입 타격은 포스트맨(Postman) 이나 웹 브라우저(curl 클라이언트) 도구의 REST HTTP 메서드 명령으로 완전 치환되어 파싱된다.

1) Read (HTTP GET 을 통한 데이터 추출 리딩)
웹 프론트엔드 콘솔에서 스토리지 DB에 특정 과거 배터리 상태 데이터를 질의한다.

## 웹 브라우저의 AJAX/Fetch 혹은 터미널 curl을 이용한 포트 통로 조회
curl -X GET http://<router_ip>:8000/robot/sensors/temp/room1

이 찰나에, 진입 관문 L7 단 라우터 HTTP 플러그인이 이 정적인 HTTP GET 문자열 텍스트 요청(Request)을 컴파일 단에서 순수 바이너리 Zenoh 네이티브 프로토콜 형태인 z_get 패킷 구조체로 강제 번역 매핑하여 백엔드 DB 에 쏘아 투척 쿼리하고, 산출되어 돌아온 해독 응답 이진 버퍼(Reply)를 다시 사람이 판독 수월한 예쁜 프론트 문자셋 구조인 JSON 문서 문자열 텍스처로 파싱 인코딩(Encoding) 포장하여 웹 브라우저 클라이언트 타겟에게 반환 렌더링 응답(Response)해 건네준다.

2) Create/Update (HTTP PUT 을 통한 데이터 수정 적재)

## 27.5 라는 온도 문자열 상태 덤프 값을 스토리지 저장 보존소 공간에 외부에서 HTTP 터널로 꽂아 강제 삽입 아카이브(저장) 인서트 밀어 넣는다!
curl -X PUT http://<router_ip>:8000/robot/sensors/temp/room1 -d '27.5'

이 REST API 백도어 허브 관문이 기본 이중 시스템으로 열려 개방 존재하기 때문에, 아키텍트 시스템 엔지니어는 분산 DB 스토리지 노드의 장애 검증(Validation) 및 프로파일 디버깅을 위해 굳이 클라이언트 컴파일 복잡한 언어 코드를 하드 빌드 짜 작성할 낭비 수고 로스 오버 패싱 필요 없이, 오직 브라우저 주소 검색 바에 스트링 텍스트를 파싱 타건하여 치는 것 단 하나만으로도 수천 킬로미터 거리를 태평양 건너 떨어져 배치 에지 둔 로봇의 심층 DB 단 버퍼 메모리 무결성 여부를 실시간으로 즉석에서 스루 디버깅 및 무결 조치 파싱 진입 간편 해부 파악해 낼 수 있는 위대한 디블로킹 매직 튜닝 시스템 모니터링 콘솔 통제 파워 장악력을 손쉽게 획득하게 조치 무장된다.

4. 데이터 직렬화(Serialization) 형상 및 이그레스 역직렬화 설정 (JSON, MessagePack 등 매핑 룰)

수만 프레임의 고해상도 카메라 덤프 영상(Vec<u8>) 바이트 배열 덩어리를, 그저 인덱스 텍스트 배열 처리를 위해 탄생한 InfluxDB(텍스트형 시계열 목적 DB) 스토리지 버퍼 인박스 공간에 무식하게 대가리 박아 구조 변환 형상화 통제 없이 그냥 우겨 밀어 강제 들이부으면, DB 테이블 파티션 바이트 데이터가 이진 오염 박살 나서 터지거나 결국 스토리지 플러그인 데몬 백그라운드 스레드 워커가 파싱 해석 불능 헛구역질을 하며 익셉션 세그먼트 충돌 메모리 스로틀 에러 패닉 코어 덤프를 터트리며 크래시 뻗어 마비되어 사망 붕괴를 일으킨다.

4.0.1 스토리지 바이너리 타입 정렬(Type Alignment) 보호구역 격벽 파싱 방어 룰 조율

1) Payload 맹목성의 대가 및 치명적 에러 원흉 요인
근원적으로 프로토콜 L4 Zenoh 라벨 터널망 뼈대 자체는, 기본적으로 자신의 캡슐 포장에 포워딩 이송 인계되어 실려 비행하는 Payload (본문 스트림 데이터) 바디 본체 구조체 안에 담긴 암호화 구조 폼 내용물이 C 이진 덤프인지 JSON 텍스트 배열 문자일지 이진 스트림 찌꺼기일지를 일일이 까보거나 형(Type) 내부 검사 간섭을 수행하지 않는 “중립 무 지성 맹목성 바이트 스트림 통과 고속도로 버스 덤프 벨트” 컨베이어 메커니즘을 띈다. 그러나, 최후 목적 종착지 파스 파트 파이프 도달 도착 관문인 저장소 DB 로 덤핑 들어갈 때만큼은, 전혀 예외로 철두철미 매우 엄격한 텍스트 파싱 분해 인코딩 형 변환(Type Cast 및 Decode) 조작이 시스템 생존 인덱스 삽입 보존 파서의 최고 우선 필수 강제 요건 인스펙션 제약으로 강력히 발동 조치 요구된다.

2) 아키텍트의 헤더 명시화 작전 (스위치 파싱 라벨링 강압 부착)
시스템 망의 데이터 발송자 클라이언트가 데이터를 쏠 때(Put 행위 시), 패킷 캡슐 컨테이너의 맨 앞단 헤더 블록 페이로드 통제 속성인 “Encoding” 타입 지시자 포맷 옵션란을 반드시 강제로 포맷 선언 지정하여 설정 조작해 쏴야만 한다. 이를 통해 목적지 스토리지 파서 플러그인이 이를 보고 수령 파싱하여, 자신이 가진 DB 포맷 틀 양식 체인징(Changing) 룰에 맞추어 올바르게 렌더 역직렬화(Deserialize) 치환 조작을 쳐서 안전하게 무 결성 파싱 변환 정규화시켜 DB 테이블 파티션에 곱게 구겨 저장 이체해 넣을 수 있다.

  • 텍스트/JSON 데이터 스트림 구조 덤프 규격: 클라이언트가 Encoding: application/json 이라는 암구호 속성 태그 명시 지정 마크를 달고 패킷을 쏘면, 이를 수납 접수 인계받은 수신지 백엔드 워커 브릿지는 이것이 글자라는 것을 안전하게 룰 스캔 판독 인식 인가하여, 무결하게 이를 문자열 딕셔너리 기반의 Document 클래스 데이터 뼈대 폼 예쁜 형태로 인코딩 구조화 배열 변환 뷰 변이 파싱 파싱 정밀 전개해서 대상 문서형 NoSQL 백엔드 DB(예: Mongo, Elastic 등) 에 오차 충돌 없이 클린 삽입 세이브 매장 안착 무사 저장 영구 인서트 한다.
  • 난해한 이진 순 바이너리 순 로우 데이터 덩어리 구조 덤프 스펙 규격: C/C++ 통신기, 가속 로봇 기어가 내뿜는 알루미늄 파편 같은 구글 프로토콜 버퍼(Protobuf, .proto) 난수표 블록 압축 객체나 MessagePack 타이트 조작 렌더 혹은, 고 용적 이미지 센서 스냅 로우 바이트 배열 스트림 데이터 덩어리의 경우에는 application/octet-stream 라는 극약처방 무지성 순 덤프 이진 압축 옵션 마킹 라벨 인코딩 텍스트를 필수 강제 규합 매핑 달아서 쏴라 마킹 보내라. 이를 읽은 백엔드 플러그인 에이전트는, 애초부터 억지 렌더 파싱 해독과 통계 인덱싱 풀이 변환 변역의 소화 시도를 안전 파싱 차단 단념 포기 체념 스킵하고, 이를 불가촉순 이진 컨테이너인 무결 거대 투박한 BLOB (Binary Large Object) 통짜 쓰나미 덩어리 캡슐 이진 파일 형상 이격 그대로 이체 디스크 슬롯 캐시에 거대 박아버려 삽입 드랍 쳐 버림으로써 OOM 스레딩 및 파서 셧다운 렉 연산 오류 충돌 마비 장애 압살 부하 데드록 사태 비상 셧다운을 완벽 미연 패스 스왑 도피 탈출 회피 방지 안전 모면 달성한다.

결론적으로, 아무 식탐 거부감 반응 없이 거대 이진 패킷을 블랙홀처럼 흡입 집어삼켜 대야만 하는 데이터베이스라는 엄격 까다로운 괴물 포식자 스토리지에게 양식 밥을 집어 수작업 배식 인서트 밥술을 넘겨 밀어 파이프 공급 포워드 인가 급여 투척 인서트 주입해 줄 땐, 이게 넘기는 바이트 배열 덩어리가 사람이 읽는 통계 문자 텍스트 소금 기반 조각 텍스트인지 무지성 바이트 철근 고기 이진 이격 덩어리 인지를 네트워크 포맷 캡슐 L4 계층 헤더 라벨(Encoding Header Tag 문자 속성 값) 속성 값에 철저하고 단호히 단단히 락온 기록 적시 포매팅 명시 마킹해야만 백엔드 코어 시스템 OOM 파서의 이진 렌더 파괴 무한 마비 다운 충돌 다운 붕괴 루프(System Render Parser Crash Over-stack Exception Deadlock) 마비 락 크래시 붕괴 인프라 참사 패닉 결함 파국을 완벽 봉쇄 무사 배제 및 방호 회피 달성 파훼 할 비결 무적 스펙 생존 수칙인 것이다.

5. 다중 스토리지 계층 쿼리 복원(Fetch)을 위한 뎁스(Depth) 셀렉터(Selector) 작성 검색 규칙 도출

거대 백엔드 스토리지 DB 그릇 뱃속 안 깊숙한 테이블 내장 계층 뎁스(Depth) 구조에 이미 광활하게 저장 파편 흡수 집어먹혀 소화되어 이진 분산 은닉 매장 적재되어 버린 수천 수백 개 수만 여 개의 로봇 말단 센서 이력 지표 값 통계 파라미터 들을 차후 다시 추출 조회 꺼내 대시보드 화면상 웹 브라우저 메모리에 한 방에 모조리 단판 풀 일거 쓸어 긁어 퍼 담아 올리고(Bulk Read) 렌더 도출 통계 쿼리를 리스팅 렌더 리딩 패치 수급 뽑아내야 할 때, 대역폭 과부하를 최소 단축 억제하는 API 질의 패킷 호출 쿼리(Query) 코드를 어찌 무슨 규격 경로 룰로 스마트하게 스캔 요청 설계 매핑 작성 스캔 짜서 심문 요청 보내야만 가장 단축 효율 고도로 작동 최적 가동하는가?

5.0.1 와일드카드 전광석화 추출 경로 매핑 분해 스캔 마이닝 전술 런북

Zenoh 백엔드 프레임워크 인프라가 자체 해석 제공 지원하는 URI 경로 계층 트리 기반 고유 쿼리 셀렉터 패치 방식은, 전통적 무겁고 질긴 관계형 DB 서버 L7 계층 파싱 처리 방식인 오라클(Oracle) SQL 시스템 내부 문자 연산 파싱 변역 조작의 무거운 연산 SELECT * FROM tbl_sensors WHERE eq... 파싱 질의 렌더 문법 해석 무거운 오버헤드 해석 대치 절차 파서 트랜잭션 구문을 완전히 무력화 파괴 속도 대체 진압 타겟 스루풋 압살하는, 순수 물리 포트 L4 단 초고속 스위치 핑 패스 계층형 메모리 슬라이서(Slicer) 고속 인덱스 점프 핑거 샷 라우팅이다.

1) 단일 리프 단말(Single Leaf Node Pod) 스캐닝 핀포인트 정밀 타격 단발 추출

z_get "factory/robot_1/sensor/temp"

스토리지 백 DB 테이블 해시 공간 안의 계층 구조 속을 스캔하여 정확히 토씨 하나 안 틀리게 매칭되는 저 단일 유일 키 경로 말단 단자 포지트 주소 번지에 해싱 배치 매칭 보존 기생 중인 기록 밸류 단일 데이터 조각 값 하나만을 핀셋 외과 집게처럼 딱 찝어서 타겟팅 핀 로컬 단일 응답 발송해 단발 뽑아 가져와 신속 페치 복사해 돌려 반환 응답을 쏴 도출 송출 가져온다. (대역폭 레이턴시 로컬 부하 단타 오버헤드 소모 제로 스펙 최속 로컬 1:1 포인트 단일 매핑 데이터 패시브 전송).

2) 하부 1차 직접 브랜치(자식 Level 자식 노드 트러스 뎁스) 전역 전체 동일 레벨 폭격 스캔 마이닝 추출 (* 단일 와일드카드 기호 옵션)

## 하단 "sensor/" 단층 평행 1계층 슬라이싱 단면 자식 형제 라인 폴더 구역망 서랍 전부 다 전동 락 해제 일렬횡대 전체 몽땅 무조건 털어 열어 강탈 쓸어 오픈해 뷰 스캔 추출 다 꺼내라 명령 덤핑!

스토리지 백엔드 검색기 파서 엔진 워커 에이전트는 이 특수 기호 락 해제 코드 커맨드를 스니핑 판독 접수한 동시 즉각 테이블 단차 덤핑 분석 돌입, 위 루트 sensor/ 평탄 하위 주소 경로망 바로 직하 뎁스 바닥 평면 바닥 바로 한 층 밑 뎁스 아래 단차 하위 가지치기 라인 트리에 1차 직속 나란히 대등 매달려 상주 묶여 종속 보존 배열 링크 서식 대기 거주 기생 저장 중이던 평행 하위 1구역 모든 패드 단말 자식 포트 토픽 밸류 그룹들, 즉 sensor/temp, sensor/humidity, sensor/pressure 등의 형제 스텝 라인 센서 값 서랍 어레이 테이블 바이트 스냅 데이터 서재 전부 1계층 타겟 무리 전체만을 일괄 폭격 통째 타겟 수집 풀 다발로 단체 묶음 뽑아서, 이를 각기 조각내 복수 개(Multiple)의 다발 연발 단층 분할 이격 된 Reply(응답) 단말 JSON 파편 배열 리스트 통계 패킷 뭉치 응답 통 단위 샷 배열로 동시다발 묶음 던져 일렬횡대 다발 반환 슛 나열 배열 도출 다발 전송 투하 폭격 통보해 응답 병합 밀어 포워딩 반출 토해 패싱 돌려 반환해 준다!
