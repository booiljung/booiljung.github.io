2.4.1 결정론적(Deterministic) 출력에서 확률적(Stochastic) 출력으로의 전환

기존 소프트웨어 엔지니어링 1.0의 가장 큰 축복은 ’재현성(Reproducibility)’이었다. 어제 버그가 발생한 시스템에 어제와 똑같은 조건과 입력을 다시 주입하면, 시스템은 그 버그를 머리카락 한 올 틀리지 않고 완벽하게 재현해 내며 디버깅(Debugging)의 힌트를 제공했다.

그러나 AI, 특히 대규모 언어 모델(LLM)은 구조 자체가 철저한 확률적(Stochastic) 룰렛 보드 위에 설계되어 있다.

1. Next Token Prediction의 저주

LLM의 작동 원리는 본질적으로 문맥을 파악한 뒤 “이 다음에 올 가장 확률 높은 단어(Token)가 무엇인가?“를 추측하는 과정(Next Token Prediction)의 무한 반복이다.
온도(Temperature) 파라미터가 0.0을 초과하는 순간부터 모델은 소프트맥스(Softmax) 계층에서 도출된 확률 분포에 따라 매번 다른 단어를 샘플링(Sampling)하게 된다.

  • 상황: “100달러를 환불받으려면 어떻게 해야 하나요?”
  • 월요일의 응답: “고객 센터로 전화를 주시면 즉시 100달러를 환불해 드립니다.”
  • 화요일의 응답: “이메일을 통해 영수증을 첨부해야만 환불 절차가 진행됩니다.”

단순히 어투만 달라지는 것이 아니라, 토큰을 샘플링하는 트리의 물리적 경로가 완전히 엇갈림으로써 도출되는 비즈니스 로직(환불 정책) 결괏값 자체가 뒤틀려버린다.

2. 일관성 오라클(Consistency Oracle)의 사망 선고

이러한 확률적 출력은 2.2장에서 다룬 방어선인 ’일관성 오라클’에 대한 즉각적인 사망 선고를 의미한다.
비즈니스적으로는 아무런 하자가 없는 코드일지라도, AI가 어제는 “안녕하세요”, 오늘은 “반갑습니다“라고 말하는 순간 assertEquals(Yesterday_Response, Today_Response) 구문은 여지없이 테스트 실패(Fail)와 함께 알람을 울려대기 때문이다. 개발자는 이것이 진짜 버그인지 아니면 단순한 확률적 변덕인지 스택 트레이스(Stack trace) 없이 눈으로 직접 읽어가며 구분해야 하는 절망적인 상황에 놓인다.
