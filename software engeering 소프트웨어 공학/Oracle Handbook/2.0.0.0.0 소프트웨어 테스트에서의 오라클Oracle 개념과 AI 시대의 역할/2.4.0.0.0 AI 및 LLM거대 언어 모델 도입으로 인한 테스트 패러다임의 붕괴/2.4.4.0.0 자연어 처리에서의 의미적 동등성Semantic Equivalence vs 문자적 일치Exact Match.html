2.4.4 자연어 처리에서의 의미적 동등성(Semantic Equivalence) vs 문자적 일치(Exact Match)

과거 단위 테스트(Unit testing) 시절의 오라클은 Assert.assertEquals("Order Success", responseText) 처럼 철자 하나 공백 하나도 틀리지 않는 문자적 일치(Exact Match)를 정답의 척도로 삼았다.

그러나 AI가 도출해 내는 자연어 문자열 앞에서 문자적 일치 체계는 철저히 파탄 난다.

1. 의미적 동등성(Semantic Equivalence)의 세계

  • 기대 결과: “비밀번호는 8자리 이상이어야 합니다.”
  • AI 출력 A: “비밀번호 형식이 맞지 않습니다. 8자 이상 작성하세요.”
  • AI 출력 B: “The password must be at least 8 characters long.”

이 세 가지 문장은 문자열 레벨에서는 해시(Hash)값이 완전히 다르지만, 의미론적으로는(Semantically) 100% 동일한 비즈니스 정책을 안내하고 있다. 만약 레거시 방식의 문자열 비교 오라클을 그대로 둔다면, A와 B는 모두 억울하게 Fail 판정을 받고 폐기된다.

classDiagram
    class LegacyOracle {
        + Exact Match (문자열 등호 판별)
        + 매우 빠름 
        + 동의어, 유의어 판독 불가 (유연성 0%)
    }
    class SemanticOracle {
        + 임베딩 벡터 거리 계산 (Cosine Similarity)
        + 연산 리소스 매우 큼
        + 동의어 및 어순 변환 파악 가능 (유연성 100%)
    }
    
    LegacyOracle --|> Fail : "의미는 같아도 철자가 다를 시 무조건 사살"
    SemanticOracle --|> Pass : "임베딩 공간 좌표가 0.95 이상 가깝다면 인가"

이 난제는 QA 엔지니어들로 하여금 어쩔 수 없이 임베딩(Embedding) 기반의 의미적 동등성 비교 모델을 파이프라인에 추가하도록 강제한다. 이는 테스트 실행 속도를 극악으로 떨어뜨릴 뿐만 아니라, “유사도(Similarity) 0.85를 합격으로 볼 것인가, 0.90을 합격으로 볼 것인가?” 라는 또 다른 골치 아픈 ’경계값 튜닝(Threshold Tuning)’의 딜레마를 낳게 된다.
