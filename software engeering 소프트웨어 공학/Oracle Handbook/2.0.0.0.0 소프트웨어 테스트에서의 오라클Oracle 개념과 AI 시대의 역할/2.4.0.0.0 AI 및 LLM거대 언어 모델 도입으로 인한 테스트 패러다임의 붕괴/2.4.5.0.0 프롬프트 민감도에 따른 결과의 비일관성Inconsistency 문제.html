2.4.5 프롬프트 민감도에 따른 결과의 비일관성(Inconsistency) 문제

정형화된 API 스키마를 던지면 정형화된 응답을 주는 과거의 방식(REST API, gRPC 등)과 달리, LLM 기반 파이프라인은 인간의 자연어 프롬프트(Prompt) 자체를 제어 코드로 쓴다. 이때 치명적인 문제는 모델이 프롬프트의 미세한 공백, 조사 하나, 심지어는 나열된 퓨샷(Few-shot) 예제의 순서에 이르기까지 병적으로 민감하게 반응하여 아예 다른 차원의 코드베이스나 텍스트를 뱉어낸다는 점이다.

1. 깨지기 쉬운 프롬프트의 유리성

  • 프롬프트 A: “이 파이썬 스크립트에서 보안 취약점을 찾아줘.”
  • 프롬프트 B: “보안 전문가로서, 아래 파이썬 스크립트 코드 스니펫에서 보안 취약점을 찾아.”

인간이 볼 때 A와 B는 100% 동일한 요구사항이다. 하지만 모델 내부의 어텐션 메커니즘 관점에서 B는 “보안 전문가“라는 토큰에 강력한 가중치가 부여되면서 아키텍처 관점의 복잡한 취약점을 반환하고, A는 단순 문법 에러를 나열할 확률이 높아진다.

이러한 프롬프트 민감도(Prompt Sensitivity)는 회귀 테스트(Regression testing)를 파괴한다. 개발자가 프롬프트 효율성을 위해 ’코드’라는 단어를 ’코드 스니펫’으로 바꾸기만 했을 뿐인데, 기존 오라클 성능 점수가 90점에서 50점으로 폭락(Performance flaking)하는 현상이 벌어진다. 즉, 시스템이 망가진 것인지 단순 프롬프트 매칭이 안 맞은 것인지 테스트 파이프라인상에서 구분할 방도가 완전히 소실되는 것이다.
