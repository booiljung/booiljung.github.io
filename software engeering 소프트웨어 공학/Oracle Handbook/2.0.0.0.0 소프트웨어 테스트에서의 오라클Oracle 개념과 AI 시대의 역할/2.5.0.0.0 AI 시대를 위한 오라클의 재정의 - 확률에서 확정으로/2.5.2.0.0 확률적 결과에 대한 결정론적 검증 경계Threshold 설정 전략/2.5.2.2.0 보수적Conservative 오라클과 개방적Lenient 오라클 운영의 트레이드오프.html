2.5.2.2 보수적(Conservative) 오라클과 개방적(Lenient) 오라클 운영의 트레이드오프

경계값(Threshold)을 설정할 때는 반드시 두 가지 극단적인 운영 모델 사이에서 외타기 곡예를 해야 한다. 이 두 모델은 AI 제품의 고객 경험(UX)과 서비스 확장성에 정반대의 효과를 가져온다.

1. 보수적 오라클 (Conservative Oracle)

오라클의 문턱(Threshold)을 극도로 높인 형태이다. “완벽하게 검증되지 않은 것은 전부 오답 처리한다“는 철학을 따른다.

  • 장점: 거짓 양성(False Positive: 틀렸는데 맞다고 통과시키는 경우)이 제로에 한없이 수렴한다. 치명적인 AI 에러가 프로덕션에 나가는 것을 원천 봉쇄하여 시스템의 보안성과 안정성을 철통같이 보장한다.
  • 단점 (Trade-off): 쓸만하고 정상적인 AI의 생성물들(창의적이고 약간 변칙적이지만 질이 좋은 답변)마저 오라클의 기준을 넘지 못했다며 줄줄이 차단당한다(False Negative). 결국 고객 입장에서는 “AI가 너무 단답형만 말한다”, “내가 뭐만 물어보면 자꾸 ’답변해 드릴 수 없습니다’라고만 한다“는 등 시스템의 지능 파이프라인이 멍청해 보이는 부작용을 직면한다.

2. 개방적 오라클 (Lenient Oracle)

오라클의 문턱을 한껏 낮춘 형태이다. “어지간한 헛소리나 문맥 이상만 아니면 창의성을 존중하여 통과시킨다“는 철학이다.

  • 장점: AI 특유의 다채로움, 창의성, 문제 해결의 다양성을 극한으로 끌어올릴 수 있다. 사용자 경험 측면에서 시스템이 매우 유창하고 똑똑해 보인다. 거짓 음성(False Negative) 리스크가 줄어든다.
  • 단점 (Trade-off): 교묘하게 가려진 환각(Hallucination) 발언이나 사내 보안 규약을 아슬아슬하게 넘어서는 답변들이 여과 없이 고객 클라이언트에 그대로 렌더링 된다.

결국 QA 엔지니어링의 미학은, 보수적 오라클과 개방적 오라클을 기능별로 쪼개어 혼합하는 것이다. 예를 들어, JSON 스키마나 API 함수 호출(Function Calling) 검사 부분은 보수적 오라클을, 사용자에게 건네는 인사말이나 안내 문구의 스타일 검증에는 개방적 오라클을 분리 결합하는 식의 투-트랙(Two-Track) 오라클 아키텍처가 권장된다.
