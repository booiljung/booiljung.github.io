2.5.5 의미 기반 오라클(Semantic Oracle): 임베딩 벡터 유사도(Cosine Similarity)를 활용한 비교

구조를 통과한 AI 출력물이 정답과 ‘의미론적으로(Semantically)’ 얼마나 일치하는지 평가할 때, 기존의 문자 조합(==) 매칭 오라클로는 다중 정답(Multiple valid answers) 앞에서는 속수무책이다. 이를 우회하기 위해 자연어 처리(NLP) 스택의 고차원 수리 모델을 비교기(Comparator)로 장착한 것이 바로 의미 기반 오라클(Semantic Oracle)이다.

1. 임베딩을 통한 다차원 공간 매핑

의미 기반 오라클의 작동 원리는 다음과 같다.

  1. 인간이 작성한 완벽한 기준 지식(Ground Truth) 텍스트를 임베딩(Embedding) 모델에 통과시켜 n-차원 벡터(Vector) 좌표를 얻어낸다.
  2. 테스트 대상 LLM이 뱉어낸 실제 응답 텍스트 역시 동일한 임베딩 모델에 넣어 방대한 다차원 공간의 벡터 값으로 변환한다.
  3. 두 개의 수학적 벡터들 사이의 각도를 측정하는 코사인 유사도(Cosine Similarity) 를 도출한다. 이 값이 사전에 정의한 임계값(Threshold, 예: 0.9)을 상회하면 의미가 일치하는 것으로 판단하여 테스트 통과(Pass) 판정을 내린다.

2. 의미 기반 오라클의 한계성

유사도 기반의 오라클은 어미나 유의어 변형에 몹시 강한 장점이 있지만, 결정론적 완결성 측면에서는 한계가 지적된다.

  • 정반대의 논리 무시: 문맥 전체는 유사하지만 “따라서 환불이 불가합니다” 와 “따라서 환불이 가능합니다” 처럼 마지막 한 글자의 ‘Not’ 연산 차이가 서비스 정책상 치명적인 차이임에도, 벡터 유사도 공간에서는 대부분 0.98 수준의 아주 흡사한 점수로 계산되어 오라클을 통과해 버리는 ’거짓 양성 현상’이 빈번히 일어난다.
  • 즉, 의미 흐름 자체를 대조할 수는 있으나, 절대적으로 지켜져야 할 중요한 비즈니스 변수나 참/거짓 매핑은 Semantic 오라클을 신뢰해서는 안 되며 룰 베이스 필터와 결합되어야만 한다.
