2.5.4 구조적 오라클(Structural Oracle): JSON, XML 등 출력 형식을 통한 1차 검증

아무리 오라클이 정답의 의미를 해석하려 발버둥 쳐도, 소프트웨어 파이프라인 뒷단에 연결된 데이터베이스나 파서(Parser)가 이해할 수 없는 형태의 문자열 포맷이라면 그 시스템은 에러를 뿜으며 즉각 붕괴한다.

AI 시대에 소프트웨어 무결성을 방어하는 가장 선봉에 서는 제1 징병 부대는 바로 구조적 오라클(Structural Oracle) 이다.

1. JSON Schema 기반의 강성 필터링

구조적 오라클은 텍스트의 ’내용’이 올바른지는 눈곱만큼도 신경 쓰지 않는다. 오직 텍스트의 ’형태’가 엔터프라이즈 서버 어플리케이션과 연동될 수 있도록 기계 독해가 가능한(Machine-readable) 마크업 규칙을 완벽하게 따랐는지만을 철저히 단언(Assertion)한다.

flowchart LR
    AI([LLM Agent]) --> Output["{ 'name': 'John', 'age': 'N/A' }"]
    Output --> S_Oracle{구조적 오라클\n(JSON Schema Validation)}
    
    S_Oracle -- "타입 오류! Age는 반드시 Integer" --> Reject[테스트 Fail / AI 재호출 로직 발동]
    S_Oracle -- "Schema 100% 만족" --> Backend[로직 백엔드 및 다음 단계 오라클로 전달]

  • JSON 괄호가 중간에 끊겼는가?
  • 명세에 정의된 필수 키(Key) 값인 email 항목이 이탈(Missing key)되었는가?
  • Integer 타입이 들어와야 할 age 속성에 AI가 마음대로 String 형식의 구연동화를 펼쳐놓았는가?

이러한 구조적 오라클은 기존의 명세 기반(Specification-based) 오라클의 성질을 그대로 지니고 있으며, Pydantic 모델이나 OpenAPI 스웨거 체계를 그대로 오라클의 뼈대로 이식할 수 있다는 엄청난 이점을 지닌다. LLM의 자유도를 코드 레벨 파이프라인에서 1진으로 억제해 내는 가장 효율적이고 절대적인 결정론적 관문이다.
