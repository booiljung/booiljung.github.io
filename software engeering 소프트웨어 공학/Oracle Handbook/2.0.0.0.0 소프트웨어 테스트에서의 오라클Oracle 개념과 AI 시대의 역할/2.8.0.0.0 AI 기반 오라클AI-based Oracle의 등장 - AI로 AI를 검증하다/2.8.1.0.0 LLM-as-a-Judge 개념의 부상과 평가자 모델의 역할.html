2.8.1 LLM-as-a-Judge 개념의 부상과 평가자(Evaluator) 모델의 역할

기계 학습 모델의 자유분방한 출력을 정규표현식으로 막는 데는 한계가 있다. 가장 널리 쓰이는 대안은 LLM-as-a-Judge, 즉 대규모 언어 모델 그 자체를 코드에 심어진 ’심판관(Oracle)’으로 활용하는 기법이다.

1. 평가자 모델의 구동 방식 (Evaluator Pipeline)

LLM-as-a-Judge 기법에서는 테스트 코드가 Assert 구문 대신 Judge 모델의 API를 호출한다.

  1. 테스트 대상(SUT) 동작: 사내 챗봇 AI에게 “비밀번호 초기화 방법“을 묻는다.
  2. 응답 수집: 챗봇 AI가 “비밀번호는 인사팀에 이메일을 보내세요“라는 문자열을 반환한다.
  3. 심판관(Judge) 호출: 테스트 스크립트는 이 응답과 ‘사내 IT 규정집(Ground Truth)’ 텍스트를 하나로 묶어, GPT-4 등 훨씬 성능이 거대한 심판관 LLM에게 프롬프트로 전송한다.
  • 심판관 프롬프트 예시: “너는 사내 QA 엔지니어다. 주어진 규정집에 근거할 때, 앞선 챗봇의 답변이 팩트 기반이라면 합격(Pass)을, 환각이라면 불합격(Fail)을 외치고 그 이유를 JSON으로 덤프하라.”
  1. 결과 분석: 심판관 LLM이 반환한 Fail이라는 응답을 파싱(Parsing)하여 테스트를 폭파시킨다.

이 방식은 수천 명의 인간 QA 테스터가 달라붙어도 수개월이 걸리는 감성 평가와 문맥적 일치도 평가를 파이프라인 안에서 몇 초 만에 완료할 수 있다는 강력한 스케일링(Scaling) 장점을 지닌다.
