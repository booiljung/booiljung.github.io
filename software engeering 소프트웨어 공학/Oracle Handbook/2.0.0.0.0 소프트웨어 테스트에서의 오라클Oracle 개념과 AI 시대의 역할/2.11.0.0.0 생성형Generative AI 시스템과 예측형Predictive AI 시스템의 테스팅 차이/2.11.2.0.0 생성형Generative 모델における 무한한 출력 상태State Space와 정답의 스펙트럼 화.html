LLM 시대로 돌입하며, 예측형 시대의 견고했던 오라클은 일거에 파괴되었다. 생성형 AI 모델의 역할은 주어진 카테고리 중 하나를 고르는 것이 아니라, 아예 백지상태에서 새로운 수십 줄의 텍스트 토큰(Token)이나 파이썬(Python) 코드를 스스로 ’창작(Generation)’하여 나열하는 것이다.

1. 무한대(Infinity) 차원에서의 오라클의 길잃음

“사과에 대해 시를 지어줘.”
이 프롬프트에서 도출될 수 있는 정답의 경우의 수 즉 출력 상태 공간(Output State Space)은 글자 수의 허가 범위에 따라 우주의 원자 개수보다도 많은 차원으로 발산(Divergence)한다.

  • 정답 판별 불가: 여기에 혼동 행렬(Confusion Matrix)과 True/False 기반의 1:1 매칭 모델을 들이대는 것은 바위를 바늘로 찌르는 격이다. 수십억 가지의 ‘아마도 꽤 괜찮은 정답들(Acceptable Answers)’ 사이에서 유일한 절대 기준이 사라지기 때문이다.
  • 정답의 스펙트럼(Spectrum) 화: 패스(Pass) 아니면 페일(Fail)이 아니라, 이 시가 92점짜리 정답인지 85점짜리 정답인지 그 퀄리티 자체가 연속적인 스펙트럼 위를 부유하게 된다.

결국 생성형 모델 앞에서는 “정답이 무엇인가?“라는 질문 모델을 버리고, **“이 대답이 최소한 우리의 비즈니스 제약 조건을 어기지 않고, 고객이 불쾌하지 않을 정도의 경계(Threshold)를 넘었는가?”**라는 새로운 패러다임의 평가 엔진(Evaluation Engine)으로 오라클 아키텍처 자체가 완전히 이사해야만 하는 것이다.
