2.1.3 테스트 대상(SUT)과 오라클 간의 상호작용 메커니즘

소프트웨어 아키텍처에서 테스트 환경을 구축할 때 테스트 대상 시스템(System Under Test, SUT)과 테스트 오라클(Test Oracle)은 서로 긴밀하면서도 완벽하게 격리된(Isolated) 상호작용 메커니즘을 가져야 한다. 두 개체는 입력을 공유하지만, 결과 도출 과정에서는 서로의 블랙박스(Black-box)에 일절 관여하지 않아야 객관적인 심판이 성립된다.

1. 동기식(Synchronous) 상호작용 vs 비동기식(Asynchronous) 상호작용

가장 일반적인 SUT와 오라클의 상호작용은 런타임 상에서 순차적으로 벌어진다.

  1. 상태(State) 주입: 테스트 프레임워크가 SUT와 오라클에게 테스트 시작 환경(데이터베이스 초기값 등)을 셋업(Setup)한다.
  2. 평가(Evaluation) 사이클: SUT가 함수의 반환(Return)을 마치면, 그제야 오라클 파이프라인이 구동되어 결과를 채점한다.

하지만 마이크로서비스 아키텍처(MSA)나 메시지 큐(Message Queue), 스트리밍 데이터 구조에서는 비동기식 상호작용이 주를 이룬다. SUT가 카프카(Kafka) 파이프라인에 이벤트를 던지는 즉시 흐름이 종료되며, 오라클은 백그라운드에서 별개의 프로세스로 구독(Subscribe)을 진행하면서 수집된 메시지의 로그가 정답 데이터 셋과 일치하는지 백그라운드 추적(Background Tracing)을 수행한다.

2. 시스템 오염(Pollution)과 격리성(Isolation)

SUT와 오라클의 상호작용에서 엔지니어가 가장 빈번하게 저지르는 실수는 동일한 데이터베이스 인스턴스나 동일한 캐시(Cache) 상태를 공유하게 만드는 것이다.

오라클의 계산 소스코드나 참조 변수가 SUT의 로직 흐름과 메모리 포인터 공간을 공유하게 되면, 로직 결함의 전이(Defect Contagion) 현상이 발생한다. 즉 SUT에서 잘못된 로직이 타서 쓰레기 값(Garbage)이 발생했는데, 오라클마저 동일한 쓰레기 값을 참조하여 “음, 방금 생성된 결과와 기대치가 일치하는군. Pass!“라고 외치는 거짓 긍정(False Positive) 대참사가 일어나는 것이다.

결론적으로 SUT와 오라클 간의 상호작용은 ’감시자(Watcher)’와 ’실행자(Executor)’의 엄격한 상호 분리(Separation of Concerns)를 전제로 해야만 오라클 본연의 ‘심판’ 권위가 손상되지 않는다.
