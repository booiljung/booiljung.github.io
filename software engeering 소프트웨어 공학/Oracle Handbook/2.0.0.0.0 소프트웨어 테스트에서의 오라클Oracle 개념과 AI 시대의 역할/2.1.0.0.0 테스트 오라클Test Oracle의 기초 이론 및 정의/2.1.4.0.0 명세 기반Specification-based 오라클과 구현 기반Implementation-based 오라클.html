2.1.4 명세 기반(Specification-based) 오라클과 구현 기반(Implementation-based) 오라클

소프트웨어 시스템에서 오라클이 정답 시나리오를 구성할 때, 그 정답의 체계를 어디에서 뼈대로 삼았느냐에 따라 크게 ‘명세 기반(Specification-based)’ 오라클과 ‘구현 기반(Implementation-based)’ 오라클로 양분된다. 이는 오라클의 성격과 테스트의 통과 기준을 정하는 데 지대한 철학적 차이를 발생시킨다.

1. 명세 기반 오라클 (Specification-based Oracle)

명세 기반 오라클은 기획자(PM)가 작성한 비즈니스 요구사항 문서, API 프로토콜 문서(Swagger 등), 수리적 명세서, 혹은 아키텍처 설계도 등 “시스템이 어떻게 동작해야만 하는가(What it should do)” 를 담은 외부 문서에 논리적 기반을 둔다.

  • 장점: 설계 결함 그 자체를 찾을 수 있다. 만약 개발자가 요구사항을 곡해하여 로직을 처음부터 잘못 구현(Wrong implementation)했다 하더라도, 기획 문서 기반의 명세 오라클은 이를 무자비하게 Fail로 처리해 결함을 걷어낸다. 블랙박스 테스팅(Black-box Testing)의 훌륭한 파트너이다.
  • 예시: JSON Schema 형식을 강제하거나, API 명세서에 정의된 데이터 타입 규약(userId는 UUID 포맷이어야 함)을 철저히 검사(Validation)하는 필터.

2. 구현 기반 오라클 (Implementation-based Oracle)

구현 기반 오라클은 테스트 대상이 되는 코드의 실제 설계 방식(Source Code Structure), 데이터베이스 스키마, 화이트박스 흐름(Control Flow) 등 “시스템이 현재 어떻게 짜여 있는가(How it is built)” 에 기반하여 정답을 판별한다. 소프트웨어 엔지니어가 직접 내부 모델을 파악하여 단위 수준(Unit Level)에서 촘촘하게 삽입하는 방식이다.

  • 장점: 코드 레벨에서 상태 변이, 메모리 유출, 특정 변수의 임계값 초과 등을 빠르고 직관적으로 검사할 수 있다.
  • 예시: 특정 함수를 실행한 후 array.length가 정확히 기존 값보다 +1 증가했는지 내부 상태를 까보는 내부 단언(Internal Assertion), 과거의 안정적인 레거시(Legacy) 코드를 참조 모델로 삼아 두 결과값을 직접 비교하는 형태 등.

3. 상호 보완성

엔터프라이즈 환경에서의 견고한 자동화 체계는 위 두 가지 오라클을 모두 필요로 한다.

완벽한 소스코드 문법과 객체 지향 패턴에 의해 내부적으로 무결하고 에러를 뱉지 않는 화이트박스 모듈(구현 통과)이라 할지라도, 그 로직이 고객과 맺은 계약 약관에 위배되는 잘못된 금액을 뱉어낼 경우 비즈니스는 실패한다(명세 위반). 반대로 명세를 만족하는 것처럼 보이는 출력이지만 뒷단에서 데이터를 중복 저장하거나 성능 누수를 내고 있다면 구현 오라클 파이프라인에서 즉시 경고음을 울려야만 한다. 따라서 현대적 QA 파이프라인은 두 오라클이 스위스 치즈 모델(Swiss Cheese Model)처럼 촘촘히 포개어 방어막을 쳐야 한다.
