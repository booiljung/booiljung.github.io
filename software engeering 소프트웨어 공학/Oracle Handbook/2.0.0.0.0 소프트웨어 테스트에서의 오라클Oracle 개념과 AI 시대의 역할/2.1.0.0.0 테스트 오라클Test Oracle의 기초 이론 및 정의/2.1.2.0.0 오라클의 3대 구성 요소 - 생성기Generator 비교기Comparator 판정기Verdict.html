2.1.2 오라클의 3대 구성 요소: 생성기(Generator), 비교기(Comparator), 판정기(Verdict)

추상적인 개념으로서의 테스트 오라클(Test Oracle)을 실제 소프트웨어 아키텍처나 자동화 프레임워크(Automation Framework) 상에 구현하기 위해서는, 입력부터 최종 판정에 이르는 과정을 논리적인 파이프라인으로 분해해야 한다.

일반적으로 성숙한 자동화 오라클은 다음의 3대 핵심 모듈로 구성된다.

1. 예상 결과 생성기 (Expected Result Generator)

생성기는 특정 입력(Input)이 주어졌을 때, 시스템이 내뿜어야 할 정답(Ground Truth)을 사전에 계산하거나 추출해 내는 모듈이다.

  • 정적 생성(Static Generation): 사전에 하드코딩된 값이나 외부 JSON 파일, 정답지 데이터베이스에서 값을 조회해 온다.
  • 동적 생성(Dynamic Generation): 참조 시스템(Reference System)이나 병렬 알고리즘을 즉석에서 구동하여 비교 대상 값을 계산해 낸다.

예를 들어, 암호화 알고리즘을 테스트할 때 입력 문자열에 대한 예상 해시(Hash) 값을 표준 라이브러리를 통해 미리 생성해 두는 모듈이 이에 속한다.

2. 비교기 (Comparator)

비교기는 테스트 대상 시스템(SUT)에서 흘러나온 실제 결과(Actual Result)와 생성기에서 도출된 예상 결과(Expected Result)를 맞대고 차이점(Diff)을 연산해 내는 모듈이다.

  • 가장 기초적인 비교기는 단순한 바이트 어레이 일치나 문자열 등호(==)를 사용한다.
  • 더 복잡한 엔터프라이즈 환경에서는 객체 구조, JSON 트리(Tree)의 깊숙한 계층 비교, 혹은 허용 오차(Tolerance Margin) 범위 내의 부동소수점 근사치(Approximation)를 비교하는 휴리스틱 비교 알고리즘이 적용된다.

3. 판정기 (Verdict / Evaluator)

판정기는 비교기에서 도출된 차이성(Differences) 데이터를 바탕으로 테스트의 최종 성공/실패 여부를 확정 짓는 최종 관문이다. 비교기가 “A와 B 값에 0.001의 차이가 있다“라고 보고하면, 판정기는 비즈니스 도메인의 중요도를 고려하여 이 차이를 통과(Pass)시킬지, 아니면 아키텍처 상의 예외(Exception/Fail)로 던질지 결정한다.

오라클을 생성기, 비교기, 판정기의 3단 구조로 모듈화(Modularization)하는 것은 매우 중요하다. AI 시스템 테스트에서는 완전히 일치하는 기대값(Expected)을 생성해 내는 생성기 메커니즘이 무너지기 때문에, 비교기를 문자열 매칭(Exact Match)에서 코사인 유사도(Cosine Similarity) 혹은 임베딩(Embedding) 비교로 갈아 끼우고, 판정기의 허용 오차를 유연하게 튜닝하는 아키텍처적 대응이 필요하기 때문이다.
