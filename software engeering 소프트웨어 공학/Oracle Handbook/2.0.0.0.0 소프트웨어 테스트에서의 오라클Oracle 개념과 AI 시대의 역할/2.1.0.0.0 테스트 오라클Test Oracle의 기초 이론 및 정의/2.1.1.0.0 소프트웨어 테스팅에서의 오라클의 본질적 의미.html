2.1.1 소프트웨어 테스팅에서의 오라클의 본질적 의미

소프트웨어 시스템에서 ’버그(Bug)’란 코드가 실행되지 않는 멈춤(Crash) 현상만을 의미하지 않는다. 코드가 완벽히 작동하더라도, 고객이 당초 요구했던 사양(Requirements)과 다른 결론을 만들어낸다면 그것은 로직의 결함이다. 따라서 테스팅이란 “이 프로그램이 주어진 입력에 대해 ’참(True)’인 결과를 도출하는가?“를 묻는 진실 게임과 같다.

이 진실 게임에서 답변이 참인지 거짓인지 판별해 주는 절대적인 재판관의 역할을 수행하는 메커니즘을 테스트 오라클(Test Oracle)이라고 명명한다. 이는 고대 그리스 신화에서 신의 뜻을 받아 인간에게 진실을 판별해 주던 델포이의 신탁(Oracle of Delphi)에서 그 용어를 빌려온 것이다.

1. 오라클의 수학적 투영

소프트웨어를 하나의 거대한 수학적 함수 f(x)라고 가정해 보자.
사용자가 시스템에 입력값 x를 전달하면, 우리가 구축한 시스템 즉, 테스트 대상(System Under Test, SUT)은 출력값 y_{actual}을 반환한다.

y_{actual} = f_{SUT}(x)

여기서 이 결과가 정상인지 결함인지 파악하기 위해서는, 이상적이고 완벽한 참조 함수 O(x)가 존재해야 한다. 작동하는 오라클은 이론상 완벽한 정답 y_{expected}를 산출해 낸다.

y_{expected} = O(x)

테스트의 통과(Pass) 여부는 궁극적으로 y_{actual} == y_{expected} 라는 명제가 참인지 거짓인지에 달려 있다.

flowchart LR
    Input[입력값 x] --> SUT[테스트 대상 시스템 SUT]
    Input --> Oracle[테스트 오라클 O]
    
    SUT --> Y_act[실제 결과 y_actual]
    Oracle --> Y_exp[기대 결과 y_expected]
    
    Y_act --> Compare{비교 연산\ny_actual == y_expected}
    Y_exp --> Compare
    
    Compare -- 일치함 --> Pass((Pass))
    Compare -- 불일치 --> Fail((Fail/Bug))

2. 진리(Truth)를 향한 접근

소프트웨어 공학의 정수는 완벽한 로직을 작성하는 것을 넘어, 완벽한 오라클 O(x)를 어떻게 저렴한 비용으로, 그리고 자동화된 코드로 구성할 수 있는가에 대한 끝없는 싸움이다.

개발자는 개발 과정에서 무의식적으로 이 오라클을 사용한다. assert(sum(2,3) == 5)라는 코드에서 숫자 5는 개발자 스스로가 인간 오라클(Human Oracle)이 되어 도출해 낸 기대 결과(Expected Output)를 코드에 영구적으로 주입한 것이다.

이처럼 오라클은 시스템의 출력값을 투영(Projection)하여 그것이 비즈니스 요구사항이라는 거울에 올바르게 맺혔는지 확인하는 가장 본질적인 평가 기준(Evaluation Criteria)이자, 테스트 자동화의 시작과 끝을 담당하는 핵심 매개체이다.
