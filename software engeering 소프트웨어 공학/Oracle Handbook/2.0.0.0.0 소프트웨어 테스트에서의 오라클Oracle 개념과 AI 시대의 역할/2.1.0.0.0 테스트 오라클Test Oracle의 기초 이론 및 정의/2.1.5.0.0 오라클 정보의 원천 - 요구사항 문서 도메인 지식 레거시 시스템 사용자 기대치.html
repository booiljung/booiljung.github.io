2.1.5 오라클 정보의 원천: 요구사항 문서, 도메인 지식, 레거시 시스템, 사용자 기대치

우리가 ’정답’이라고 일컫는 무결점의 지표는 진공 상태에서 태어나지 않는다. 테스트 대상(SUT)을 압박하기 위해 오라클 파이프라인(Oracle Pipeline)을 구축할 때, 시스템 엔지니어들이 정답지 데이터 셋업(Ground Truth Setup)을 위해 참고하고 의존하는 지적 자원의 원천(Sources of Information)은 크게 네 가지 갈래로 나눌 수 있다.

이 원천 데이터가 튼튼할수록 오라클의 권위는 절대적이어지며, 원천 데이터가 부유(Drifting)하거나 모호할수록 오라클은 오판을 남발하는 허수아비로 전락한다.

1. 요구사항 문서 (Requirements & Specifications)

가장 명시적이고 법적 구속력이 있는 오라클의 정보 원천이다.

  • PRD(Product Requirements Document), API 스웨거(Swagger), UI/UX 디자인 피그마(Figma), 약관 규약 등이 여기에 포함된다.
  • “입력값이 공백일 경우 HTTP 400 Bad Request를 반환해야 한다“라는 치밀한 문장이 그대로 오라클의 하드코딩된 단언문(assertEquals(response.status, 400))으로 진화한다.

2. 레거시 시스템 (Legacy System / Reference Implementation)

코드 리팩토링(Refactoring)이나 언어 전환, 또는 새로운 마이크로서비스 전환 프로젝트에서 강력하게 의존하는 원천이다.

  • 과거에 안정적으로 수년째 구동되어 온 모듈 자체를 하나의 거대한 ’정답 생성기’로 취급한다.
  • 섀도우 런(Shadow Run)이나 트래픽 미러링(Traffic Mirroring) 환경에서, 새로운 버전의 시스템 출력값과 과거 레거시 시스템의 출력값을 1:1 바이트 비교(Byte-by-byte Comparison) 함으로써 무결성을 보장받는다.

3. 도메인 지식 체계 (Domain Knowledge Base)

자연 법칙, 업계 표준, 수학적 공리 등 문서를 뛰어넘는 특정 산업군의 절대 불변하는 도메인 룰(Domain Rules)이다.

  • 예시: 은행 시스템의 경우 이체 금액 > 0 이고, 입금자와 출금자의 증감치 합이 0(Zerosum)이어야 한다는 무결성 공식.
  • 예시: 의료 시스템에서 특정 약물 A와 B는 절대 동시 피가 불가능하다는 약리학적 금기(Contraindications).
  • 이 지식들은 주로 휴리스틱 오라클과 부분 불변량 오라클의 뼈대로 작용하여, 시스템 내의 치명적 논리 오작동을 차단한다.

4. 사용자 기대치 및 암묵적 규약 (User Expectations & Implicit Rules)

명문화되지 않았으나 사용자라면 당연히 보장될 것이라 믿는 상식의 영역이다.

  • 버튼을 눌렀을 때 애플리케이션이 크래시(Crash)되지 않아야 한다.
  • 웹사이트 내비게이션 바는 각 페이지마다 동일한 위치에 있어야 한다.
  • 비록 오차율 기반의 모호한 비교라 할지라도, 사용자가 직관적으로 “이것은 버그다“라고 느끼게 만드는 시스템의 먹통 현상(Hang)이나 무응답(Timeout) 자체가, 그 어떤 명세 문서보다도 암묵적(Implicit) 오류를 선언하는 강력한 인간 오라클의 원형으로 작용한다.
