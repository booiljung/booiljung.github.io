2.3.5 인간 오라클(Human Oracle)의 개입 필요성과 병목 현상

오라클 문제(The Oracle Problem)가 기계적인 코드와 검증 알고리즘을 휩쓸고 지나가 초토화된 전장 위에서 소프트웨어 품질을 지켜내는 최후의 방어선은 결국 인간의 인지 능력이 개입하는 **인간 오라클(Human Oracle)**이다.

인간은 문맥(Context)을 이해하고, 시각 구조의 미적 조화를 판별하며, 확률적 문장이 띠고 있는 윤리적 편향성을 즉각적으로 판독해 내는 지상 최고의 초정밀 추론 엔진이다. 테스트할 수 없는(Non-testable) 프로그램조차 도메인 전문가의 눈앞에 가져다주면 즉시 정답/오답 판별이 가능해진다.

1. 수동 테스팅(Manual Testing)과 휴먼-인-더-루프(HITL)

  1. 탐색적 테스팅(Exploratory Testing): 자동화 오라클이 커버하지 못한 예외 공간에 뛰어들어, QA 엔지니어 개인의 경험과 상식을 바탕으로 즉흥적인 입력을 날려가며 시스템의 내구성을 수동으로 파괴해 보는 행위. 여기서 정답은 테스터 머릿속의 직관이다.
  2. AI 영역에서의 HITL (Human-in-the-Loop): AI가 도출한 법률 요약본을 변호사가 3등급 척도(완벽함/오류 약간/치명적 오답)로 수동 검수하는 구조. RLHF(인간 피드백 기반 강화학습)의 중추이다.

2. 확장성의 절망적인 병목(Bottleneck)

그러나 이 완벽해 보이는 인간 오라클의 최고 치명타는 초당 처리량(Throughput)이 끔찍하게 낮고, 인건비가 비정상적으로 높다는 점이다.

  • CI/CD 파이프라인에서 1초에 500개의 테스팅 연산이 수행되지만, 인간 도메인 전문가는 하나의 스크린샷 렌더링 결함을 잡아내기 위해 최소 5초를 응시해야 한다.
  • 피로 누적과 주의력 결핍으로 인해 오라클 판정 결과마저 비일관성(Inconsistency) 오류를 일으킨다.

따라서 21세기 테스팅의 승패는 “인간 오라클 인력을 몇 명이나 더 고용할 수 있는가?“가 아니라 “어떻게 인간의 인지적 암묵지(Tacit Knowledge) 룰셋을 프롬프트나 머신러닝 판정기(Evaluator)로 빠르게 증류(Distill)시켜 고속의 기계 오라클로 이식할 것인가?“에 완벽히 종속되어 있다.
