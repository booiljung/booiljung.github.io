2.3.2 테스트 불가능한 프로그램(Non-testable Programs)의 특징

오라클 문제(The Oracle Problem)가 치명적인 임계점을 넘어서게 되면, 개발자는 해당 컴포넌트를 이른바 ’테스트 불가능한 프로그램(Non-testable Programs)’으로 격하시키게 된다. E.J. Weyuker가 1982년에 처음 제안한 이 개념은, 코드가 빌드되지 않는다거나 문법 에러가 났다는 의미가 아니라 “기계적으로(Automatically) 정답을 판별해 낼 수단이 원천 봉쇄된 로직” 을 뜻한다.

테스트 불가능한 프로그램들은 다음과 같은 고유한 특징들을 공유한다.

1. 정답의 스펙트럼 화 (Spectrum of Correctness)

흑백의 논리(True/False)가 아니라 회색 지대(Gray Area)가 끝없이 펼쳐지는 도메인이다.

  • 기계 번역(Machine Translation) 알고리즘: 영어 문장을 한국어로 번역할 때 “이것만이 유일한 정답이다“라는 1:1 매칭은 존재하지 않는다. 조사 하나의 차이, 단어 선택의 뉘앙스 차이 등 수천 가지의 ’괜찮은 번역(Acceptable Answers)’이 존재하므로 명확한 불린(Boolean) 단언이 불가능하다.

2. 예측 불가능한 외부 상태 변경 (Side-effects outside control)

시스템 자체가 외부에 제어 불가능한 상태 변이를 무작위로 발생시키거나 종속될 때 테스트가 불가능해진다.

  • 하드웨어 센서 노이즈가 밀리초(ms) 단위로 섞여 들어오는 임베디드 코어 로직. 외부에서 어떤 값이 튀어나올지 런타임 예측기(Generator)가 도저히 모방할 수 없는 도메인이다.

3. 기하급수적 상태 공간 (Exponential State Space)

가능한 경로와 분기점(Branch)이 무한정 폭발하여 기하급수적 경우의 수를 지닐 때 오라클은 붕괴한다.

  • 체스나 바둑 플레이를 위한 강화 학습 알고리즘(AlphaGo 등)은 다음 턴의 가장 완벽한 한 수(Expected Action)를 계산하기 위해 10의 100승이 넘는 상태 공간을 탐색해야 한다. 이러한 복잡도는 단순 비교 연산기로는 검증할 수 없다.

결과적으로, AI 시대의 비결정적 생성 모델(LLM)들은 그 탄생 구조부터가 본질적으로 가장 거대한 형태의 ’테스트 불가능한 프로그램’의 특징을 전부 보유하고 있다. 따라서 이러한 프로그램을 억제하려면 기존의 참/거짓 매핑 오라클이 아닌, 부분 속성을 평가하는 메타모픽 테스팅(Metamorphic Testing) 등의 완전히 새로운 수학적 접근이 강제된다.
