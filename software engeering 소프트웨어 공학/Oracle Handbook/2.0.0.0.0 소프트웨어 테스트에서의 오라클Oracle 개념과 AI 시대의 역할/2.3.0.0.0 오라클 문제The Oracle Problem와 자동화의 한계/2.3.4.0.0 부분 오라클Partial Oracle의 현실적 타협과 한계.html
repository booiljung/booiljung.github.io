2.3.4 부분 오라클(Partial Oracle)의 현실적 타협과 한계

오라클 비용(Oracle Cost)이 감당할 수 없을 정도로 치솟거나 오라클 문제(The Oracle Problem)로 인해 완벽한 정답 생성이 불가능할 때, 업계가 타협점으로 채택하는 것이 부분 오라클(Partial Oracle) 전략이다. 이는 거대한 성벽 전체를 보수하는 대신 적군이 진입 불가능한 핵심 통로만 방어하겠다는 방어적 소프트웨어 테스팅의 일환이다.

1. 부분 검증의 매커니즘

부분 오라클은 출력 결과물의 ’전체 값 일치’가 아닌 결괏값이 절대로 범해서는 안 되는 **필수 조건(Post-conditions)**과 **불변 법칙(Invariants)**만을 검출한다.

  • 금융 알고리즘 예시: 수십 가지 파생상품 알고리즘이 뒤섞인 복잡한 포트폴리오 산출 엔진. SUT가 도출한 최종 포트폴리오의 이윤율(Profit rate)이 완벽한 정답인지 증명하기는 너무 어렵다(참 오라클 부재). 그러나 최소한 “모든 주식 종목의 보유 비율(%)의 합은 정확히 100%여야 한다“거나 “현금 자산은 절대 음수(-)가 될 수 없다“는 성질은 확실하다. 부분 오라클은 오직 이 두 가지 제약 조건만을 assert 구문으로 감사(Audit)한다.

2. 치명적인 사각지대의 양산 (한계점)

그러나 부분 오라클은 태생적으로 거짓 양성(False Positive)의 리스크는 줄이되 거짓 음성(False Negative)의 리스크를 눈감는 치명적 약점을 노출한다. 즉, 시스템이 미친 짓을 저지르고 있음에도 오라클은 “음, 현금이 음수도 아니고 보유 비율도 100%네요. 테스트 통과 (Pass)!” 라고 오판해버리는 심각한 버그 누수(Bug Leak) 현상을 맞이할 수 있다.

마치 AI 모델에게 “환불 규정에 대해 응답할 것“이라고 지시했을 때, 모델이 욕설을 섞어가며 완벽히 틀린 환불 정책을 생성해 냈음에도 불구하고(실패), 오라클은 단지 출력물 안에 “환불” 이라는 키워드가 존재했으므로 “Pass” 파란불을 띄우는 나태한 심판관으로 전락해버리는 현상이다. 따라서 부분 오라클은 절대 단독으로 사용되어선 안 되며, 수십 개의 다차원적 부분 룰셋을 겹겹이 쌓아 올려 방어막의 밀도를 메워야만 한다.
