2.3.1 오라클 문제의 정의: 예상 결과를 산출하기 어렵거나 불가능한 상황

소프트웨어 엔지니어링에서 오라클 문제(The Oracle Problem)란, “주어진 테스트 케이스 입력(Input)에 대하여, 시스템이 반환해야 할 ’완벽히 올바른 기대 결과(Expected Result)’를 실용적인 비용과 시간 내에 기계적으로 산출하거나 검증할 수 없는 한계 상황” 을 의미한다.

즉, 버그가 있는지 알고 싶어도 “비교할 정답지 자체를 만들 수 없어서 채점을 포기해야 하는 딜레마“이다.

1. 오라클 문제의 발생 근거

오라클 문제는 일시적인 기술 부족이 아니라 도메인이 지닌 본질적인 성질에서 기인한다. 프로그램이 다루는 변수가 무한대로 발산하거나 외부 환경 통제가 불가능할 때 주로 나타난다.

  1. 복잡계 연산(Complex Computation): 기상청의 날씨 시뮬레이션 프로그램을 생각해 보라. 내일 서울 특정 지역의 온도가 23.5^\circ\text{c}라는 프로그램 결괏값이 나왔을 때, 이 결괏값이 로직 결함 없는 100% 정답인지 알려면 내일 그 시간까지 기다리는 것 외엔 방법이 없다.
  2. 비결정성(Non-determinism): 병렬 다중 스레딩(Multi-threading) 환경, 분산 머신 러닝, 그리고 대규모 언어 모델(LLM)과 같이 확률적 샘플링(Probabilistic Sampling)이 내부 깊숙이 개입된 경우, 동일한 상태와 입력을 주어도 매번 결괏값이 달라지므로 하나의 확고한 예상 결과를 1:1로 매핑할 수 없다.

2. 자동화 파이프라인의 붕괴

이 딜레마는 CI/CD와 같은 지속적 통합 파이프라인의 숨통을 강하게 조인다. Assertion 구문을 작성하려면 expected_value를 하드코딩하거나 런타임 중에 연산해야 하는데, 이 값을 모르는 상태에서는 아무리 훌륭한 Gherkin 문법(기능 명세)과 Cucumber 프레임워크를 도입하더라도 테스트 자체가 무용지물이 되기 때문이다.
