2.2.5 파생 오라클(Derived Oracle): 기존 시스템이나 대체 알고리즘의 결과를 정답으로 간주

새로운 알고리즘이나 프레임워크를 개발할 때, 개발자 머릿속에서 모든 기대 결과를 0에서부터 도출해 내는 것은 비효율적이다. 파생 오라클(Derived Oracle), 혹은 참조 기반 오라클(Pseudo-Oracle) 이라 불리는 이 방법은 정답을 ’창조’하는 대신 다른 믿음직한 엔진에서 결과를 ‘빌려와(Derive)’ 정답지로 사용하는 방식이다.

1. 백-투-백 테스팅(Back-to-Back Testing)의 교본

가장 대표적인 활용 사례는 알고리즘 효율성 최적화(Optimization)를 진행할 때 발생한다.

수치 병렬 처리를 위해 기존의 느린 버블 정렬(Bubble Sort) 로직을 버리고 새로운 C++ CUDA 기반의 다중 스레드 퀵 정렬(Quick Sort) 시스템 SUT를 개발했다고 가정하자. 이 빠른 산술 코드의 결과가 모든 시나리오에서 정상인지 검증하기란 까다롭다.

이때 ‘느리지만 이미 완벽함이 증명된 버블 정렬 시스템’ 을 파생 오라클로 설정한다. 완전히 동일한 수만 개의 무작위 난수 배열을 느린 레거시 시스템과 새로 만든 시스템에 동시에 슛팅(Shooting)하고 두 결과물을 비교한다. 이 이중 실행 방법론을 백투백(Back-to-Back) 오라클 테스팅이라고 칭한다.

flowchart TD
    InputData[무작위 배열 입력\n1만 세트] --> SUT[새로운 초고속 다중 스레드 정렬 알고리즘]
    InputData --> Pseudo[기존의 안정된 레거시 정렬 알고리즘\nDerieved Oracle]
    
    SUT --> Out_new((결과 집합 A))
    Pseudo --> Out_old((결과 집합 B))
    
    Out_new --> Compare{정합성 비교\nA == B}
    Out_old --> Compare
    
    Compare -- 모두 일치 --> Output_Pass[Pass: 신규 시스템 완벽함]
    Compare -- 한 줄이라도 불일치 --> Output_Fail[Fail: 로직 내부 경계값 오류 탐지]

2. AI 성능 판독을 위한 N-Version 프로그래밍

이 기법은 소프트웨어 2.0 시대의 AI를 테스트하는 데에도 폭넓게 활용된다. 여러 회사의 서로게이트 모델(Surrogate Models)을 묶어서 N-Version 프로그래밍(예: 동일한 질문을 Claude 3.5, GPT-4, Gemini-1.5 3가지에 날린 후 다수결로 판정)을 시키거나, 가벼운 On-Premise SLM(소형 모델)의 로컬 결과를 값비싼 메인 클라우드의 GPT-4o라는 파생 오라클과 정직성 매핑 교차 비교를 수행함으로써 자체 품질을 우회적으로 보증받는 등, 현업에서 가장 널리 쓰이는 실전 오라클 패턴이라 할 수 있다.
