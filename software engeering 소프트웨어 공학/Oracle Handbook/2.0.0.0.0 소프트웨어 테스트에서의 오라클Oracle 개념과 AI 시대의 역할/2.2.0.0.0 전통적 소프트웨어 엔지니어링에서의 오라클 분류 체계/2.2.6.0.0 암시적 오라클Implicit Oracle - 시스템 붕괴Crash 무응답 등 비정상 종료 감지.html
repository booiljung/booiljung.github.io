2.2.6 암시적 오라클(Implicit Oracle): 시스템 붕괴(Crash), 무응답 등 비정상 종료 감지

전수 테스트와 철저한 명세 기반 검증을 위한 예산이 전무하거나, 외부 모듈을 단순 호출(Third-party Library)해야만 하는 환경이 있다. 이때는 결과값이 도메인 논리에 정확히 맞아들어가는지 한 줄 한 줄 따질 겨를조차 없다. 이런 극단적 상황에 기대는 최후의 수단, 혹은 시스템 강건성(Robustness) 검증의 기초가 되는 것이 바로 암시적 오라클(Implicit Oracle)이다.

1. 최소한의 생존 조건 감지

암시적 오라클은 기획자나 도메인 전문가가 별도의 정답지 룰(Rules)을 만들어주지 않더라도 프로그램이라면 “당연히 보여야 할 생존의 암묵적 기준“에 기대어 실패 여부를 선언한다. 이 오라클의 기준은 지극히 1차원적이고 생물학적(?)이다.

  1. 치명적 붕괴(System Crash): NullPointerException이나 Segmentation Fault 등을 발생시키며 코어 덤프(Core Dump)와 함께 프로그램이 돌연사했는가?
  2. 무한 루프와 타임아웃(Hang/Timeout): 시스템이 10초 이상 아무런 API 응답을 주지 않고 교착 상태(Deadlock)에 빠졌는가?
  3. 자원 소진 감시(Resource Exhaustion): CPU 사용률 100% 지속, 혹은 메인 메모리(RAM)를 미친 듯이 점유하는 메모리 누수(Memory Leak) 징후가 나타났는가?

단순히 브라우저 콘솔이나 서버 로그에 빨간색 Fatal Error가 찍혔다면, 출력 텍스트가 아무리 아름답게 생성되었건 말건 암시적 오라클은 이를 가차 없이 Fail로 규정한다.

2. 원숭이 테스팅과 혼돈 공학의 핵심망

암시적 오라클은 퍼즈 테스팅(Fuzz Testing)이나 멍키 테스팅(Monkey Testing)을 진행할 때 파이프라인의 핵심 센서가 된다. 수조 단위의 악의적인 가비지 데이터나 문자열 폭탄을 애플리케이션에 빗발치듯 던졌을 때(Fuzzing), 테스터는 이 수조 개의 데이터가 어떤 비즈니스 결괏값을 내는지 알 수 없고 관심도 없다. 오로지 “마이크로서비스 애플리케이션의 메모리가 터졌는가? 터지지 않았는가?” 만을 체크함으로써 보안 취약점과 침투 임계치를 밝혀낸다.
