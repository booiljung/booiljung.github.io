2.2.3 일관성 오라클(Consistency Oracle): 동일 입력에 대한 이전 실행 결과와의 비교

참 오라클(True Oracle)이 도저히 불가능한 복잡계 애플리케이션에서 광범위하게 쓰이는 또 하나의 핵심 무기는 바로 ’일관성 오라클(Consistency Oracle)’이다. 이 방식은 시스템이 무슨 계산을 하고 있는지 그 내부 원리와 정답을 굳이 외부에서 만들거나 알려고 하지 않는다. 대신 “어제도 이렇게 입력했을 때 50이 나왔으니, 배포 후인 오늘 똑같이 입력해도 50이 나오면 이 코드는 정상이겠지“라는 시계열적 안정성에 베팅하는 철학이다.

1. 회귀 테스트(Regression Testing)의 지배자

일관성 오라클은 사실상 현대 CI/CD 파이프라인의 회귀 테스트 스위트를 90% 이상 장악하고 있는 방법론이다.

  1. 골든 데이터 캡처(Golden Data Capture): 앱 배포 전 안정적인 상태에서 수만 개의 사용자 입력을 흘려보내고, 그 출력값을 모두 데이터베이스에 덤프(Dump)하여 스냅샷(Snapshot)을 생성한다.
  2. 코드 변경 및 릴리즈: 데이터베이스 마이그레이션을 거치거나 컴팩트한 로직 리팩토링(Refactoring)을 수행한다.
  3. 일관성 검증: 이전에 넣어보았던 수만 개의 동일한 입력을 다시 시스템에 통과시킨 뒤, 이번에 뽑힌 출력과 과거의 스냅샷 덤프값을 1:1 바이트 단위로 대조하여 불일치(Diff)를 감지한다.

이 모델에서 오라클의 정답 역할은 ’어제의 시스템(Yesterday’s System)’이 대신 수행한다.

sequenceDiagram
    participant History as 어제의 시스템 버전 v1.0
    participant System as 오늘의 시스템 버전 v1.1
    participant Oracle as 일관성 오라클
    
    activate History
    Note over History: 1. 기존 안정화 버전에서<br/>[Input X] -> [Output Y] 도출 및 저장
    History-->>Oracle: 스냅샷 Y 저장 (Golden Record)
    deactivate History
    
    activate System
    System->>Oracle: 2. 코드 개선 후 [Input X] -> [Output Y'] 도출
    deactivate System
    
    Oracle->>Oracle: 3. 비교 (Y == Y') 수행
    alt 두 출력이 완벽 일치함
        Oracle-->>System: 일관성 유지 (Pass)
    else 누락이나 변조 발생
        Oracle-->>System: 일관성 붕괴 (Fail / Regression!)
    end

2. AI 시대에서의 파열음

그러나 일관성 오라클은 소프트웨어가 결정론적(Deterministic)이라는 확고한 믿음 하에 성립된다. 입력이 같으면 무조건 어제와 똑같은 결과가 나와야 한다는 대전제가 무너지는 순간 이 오라클은 고장을 일으킨다.

즉, 매일 온도(Temperature) 샘플링에 따라 출력이 달라지는 LLM을 도입하거나, 외부 API의 잠수함 패치(Model Drift)가 발생하여 출력 포맷과 어조가 미세하게 변경되면, 비즈니스상 하등 문제가 없는 로직 변경이었음에도 일관성 오라클은 100% Fail 스크린을 띄워버리게 된다. 이는 1.4.3장에서 언급한 ’AI 도입으로 인한 회귀 테스트의 복잡성’이 발생하는 근원적 매커니즘이라 할 수 있다.
