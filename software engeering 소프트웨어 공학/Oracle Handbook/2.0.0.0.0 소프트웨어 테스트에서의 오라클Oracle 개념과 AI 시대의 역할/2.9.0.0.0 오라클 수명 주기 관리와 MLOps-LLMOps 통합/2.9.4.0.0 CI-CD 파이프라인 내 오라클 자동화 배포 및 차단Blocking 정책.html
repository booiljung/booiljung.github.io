2.9.4 CI/CD 파이프라인 내 오라클 자동화 배포 및 차단(Blocking) 정책

“테스트는 통과했는데 배포는 수동으로 하겠습니다“와 같은 정책은 릴리즈 주기 단축을 생명으로 하는 현대 소프트웨어 공학에 역행한다. 이 과정의 허리를 이어주는 것이 완벽히 조율된 CI/CD 파이프라인과 그 파이프라인의 심장인 ’오라클 자동 차단기’다.

1. 무인(Unmanned) 릴리즈 파이프라인 구축

  1. 지속적 통합(CI - Continuous Integration):
    개발자가 프롬프트나 백엔드 로직 코드를 수정하여 커밋(Commit)하면 젠킨스(Jenkins)나 깃허브 액션(GitHub Actions) 러너가 즉시 켜진다.
  • 이 러너 내에서 Pytest나 Jest 등의 테스트 프레임워크가 가동되어, 사전에 정의해둔 수천 개의 골든 데이터셋과 실버 데이터셋을 새 모델에 질의한다.
  • 이때 2.7장에서 규정한 강성 오라클(예: 주민등록번호 검출 시 에러)과 확률적 오라클(코사인 유사도 > 0.85 만족)이 채점관으로 개입한다.
  1. 차단 깃발(Blocking Flag) 발동:
    만약 강성 오라클에서 단 1건의 Fail이 떨어지거나, 확률적 오라클 스코어의 총합이 조직이 지정한 Deployment_Threshold 밑으로 하회하는 순간 프로세스는 붉은색 알람(Red Alert)과 함께 동결된다. 슬랙(Slack)으로 실패 메시지가 전파되며 프로덕션 서버로의 도킹(Merge)이 원천 봉쇄된다.

이 고도화된 막을 통과한 소프트웨어만이 운영 환경(Production)으로 나가는 자격을 얻음으로써 인간의 개입 없는 완벽한 “Continuous Deployment(지속적 전개)“를 쟁취할 수 있게 된다.
