2.9.3 버전 관리 시스템 내에서의 테스트 케이스 및 오라클 이력 관리

프롬프트 엔지니어링이나 RAG 검색 로직에서 단 한 줄의 프롬프트만 변경되어도 출력 결과 전체가 완전히 요동치는 환경이기 때문에, 오라클 정답지에 대한 형상 관리(Version Control)는 서버 백엔드의 메인 소스코드보다 몇 배는 더 철저하게 통제되어야 한다.

1. Git 연동 기반 멀티-버전(Multi-Version) 오라클 아키텍처

AI 모델이나 프롬프트 버전을 업그레이드할 때, 이전 버전의 오라클 잣대를 그대로 대입하면 테스트는 필연적으로 모두 박살 난다. 따라서 소스코드 릴리즈 태그(Tag)와 오라클 데이터셋의 태그 체인은 완벽히 페어링(Pairing)되어야 한다.

  • v1.2.0 AI 모델의 배포 파이프라인은 반드시 dataset-v1.2-oracle 이라는 리비전(Revision) 해시를 바라보도록 Git LFS(Large File Storage)나 DVC(Data Version Control)에 명시해야 한다.
  • 만약 평가 점수 산출 로직(예: 길이 패널티 코드를 삭제함) 자체를 갈아엎었다면, 이전 버전에서 90점을 맞았던 모델이 신규 오라클 스위트에서는 60점으로 폭락할 수도 있다. 이것이 모델의 퇴행(Regression)인지, 오라클 잣대의 변동 때문인지를 명확히 분별하기 위해 깃허브 커밋 로그에 오라클 룰 개정 역사가 소상명백하게 문서화(Documentation)되어야만 한다.

“오라클 패러미터 변경” 자체가 시스템 전반의 브레이킹 체인지(Breaking Change)로 취급받는 엄격한 이력 관리 정책이야말로 LLMOps의 성패를 가른다.
