3.1.3 결정론적 정답지의 3요소: 명확성(Clarity), 검증 가능성(Verifiability), 불변성(Immutability)

결정론적 정답지(Deterministic Ground Truth)는 단순한 데이터베이스 텍스트 더미가 아니다. 시스템의 오라클(Oracle)이 채점을 수행할 때 ’법전’으로 기능해야만 하므로 극도로 엄밀한 설계 요소의 뒷받침을 받아야 한다.

소프트웨어 품질 보증(QA) 기저에 박아야 하는 결정론적 정답지의 3대 근본 요소는 다음과 같다.

1. 명확성 (Clarity)

오직 하나의 뜻으로만 해석(Unambiguous interpretation)되어야 한다. 자연어나 수치를 막론하고, 읽는 이나 파싱(Parsing)하는 기계가 전혀 다른 비즈니스 문맥으로 오해할 수 있는 회색 지대(Gray Area)가 존재해서는 안 된다.

  • 나쁜 정답지: “상품 설명은 친절하고 자세하게 적혀야 한다.” (평가 기계가 ’친절’을 이해할 수 없음)
  • 명확한 정답지: “상품 설명 배열 리스트(Array)에는 반드시 [가격, 반품 정책, 원산지] 키워드가 존재해야 한다.”

2. 검증 가능성 (Verifiability)

파이프라인 안의 코드가 정답지의 값을 읽고 곧바로 AI의 출력값과 수학적 혹은 문자열 검사 함수로 참/거짓 매핑을 전개할 수 있어야 한다.

  • 설명 불가능한 검증 불가 정답: “디자인적으로 화려한 레이아웃에 어울리는 색상이어야 한다.”
  • 검증 가능한 정답: “CSS Hex 코드가 회사의 백색/청색 브랜드 팔레트 배열 요소 안에 Includes 되는가?”

3. 불변성 (Immutability)

시간이 흘러도, 혹은 테스트 코드가 수천 번의 브랜치(Branch) 환경에서 재실행(Re-run)되어도, 외부의 개입에 의해 정답 데이터 자체가 훼손되거나 무단으로 조작되어서는 안 된다.

  • 정답지가 담긴 저장소(Repository) 파일은 철저한 Read-Only 권한 모델로 포장되어야 하며, 버전 관리 체계(Git, DVC)의 해시값에 의해 위변조가 통제되는 순수 불변 함수(Pure function constant)의 상태를 이뤄야만 한다.

이 3대 요소가 완비될 때, 비로소 AI의 미쳐 날뛰는 변동성(Volatility)을 거대한 닻(Anchor)의 힘으로 잡아당겨 CI/CD 파이프라인 내부로 강제 정착시킬 수 있게 된다.
