3.1.1.2 엔터프라이즈 시스템에서 요구하는 멱등성(Idempotency)과 재현성(Reproducibility)

주사위 굴리기로 무작위 텍스트를 조립하는 AI 모델을 들고 은행의 ‘모기지론 대출 심사 시스템’ 앞에 섰다고 가정해 보자. 소프트웨어 공학 파이프라인이 AI를 거부하며 토해내는 이유는 명백하다. 엔터프라이즈가 시스템에 목숨처럼 요구하는 제1의 원칙인 멱등성(Idempotency) 과 재현성(Reproducibility) 이 철저히 붕괴되기 때문이다.

1. 진리의 두 기둥: 멱등과 재현

  1. 멱등성 (Idempotency):
    동일한 함수나 API를 한 번 호출하든 1만 번 호출하든, 그로 인해 서버에 변경되는 상태나 최종적인 결괏값의 속성은 언제나 동일해야 한다는 설계 원칙이다. 결제 버튼(Payment)을 실수로 두 번 눌렀을 때 고객 통장에서 돈이 두 번 빠져나가는 사태를 막는 최후의 저지선이 멱등성이다.
  2. 재현성 (Reproducibility):
    에러가 터진 시점의 메모리 덤프와 사용자 입력값을 다시 시스템에 흘려보내면, 정확히 똑같은 에러 라인이 튀어나와야 한다. 이것이 없으면 스택 트레이스(Stack Trace) 기반의 원인 분석(Debugging)은 완벽히 불가능하다.

LLM 기반 시스템은 동일한 “내 계좌 잔고를 알려줘“라는 요청을 두 번 날릴 때, 첫 번째는 “잔고는 100달러입니다”, 두 번째는 “비밀번호를 먼저 입력하세요“라며 완전히 엇나간 출력을 제공할 위험성을 항상 내포하고 있다.
멱등성과 재현성이 파괴된 소프트웨어는 무너지는 모래성이다. 이 모래성이 무너지는 것을 억지로라도 붙잡아 기둥을 세우는 무거운 콘크리트 인프라가 바로 우리가 설계해야 할 ’결정론적 정답지(Deterministic Ground Truth)’의 소명이다.
