3.2.3.1 오류 발생 시 모델의 문제인지 데이터의 문제인지 판별

RAG(검색 증강 생성) 구조 등 최신 AI 파이프라인에서 고객이 “A 상품의 환불 규정을 알려줘“라고 물었는데 시스템이 엉망진창인 답변을 냈을 경우, 원인은 보통 두 가지 중 하나다.

  1. 데이터 검색기(Retriever)의 문제: 애초에 사내 벡터 DB(Vector DB)에서 A 상품이 아닌 B 상품의 규정을 뽑아 모델에게 던져준 경우. (Garbage In)
  2. 생성기(Generator / LLM)의 문제: DB에서 뽑아준 규정 문서는 완벽했는데, 모델이 컨텍스트를 오독하여 환각을 일으킨 경우. (Garbage Out)

1. 정답지가 꽂아주는 판별의 이정표

이 환장할 노릇 속에서 결정론적 정답지(Ground Truth)는 책임을 명확하게 분배한다.

정밀한 정답지 셋에는 ’이 질문에 대답하기 위해 추출되어야 하는 골든 컨텍스트(Golden Context Document ID)’가 정답 메타데이터로 함께 매핑되어 있다.

  • 만약 검색기가 골든 컨텍스트 ID를 인출하는 데 실패했다면, 백엔드 데이터베이스 팀이나 임베딩 전략을 수정해야 한다. (리트리버 튜닝)
  • 검색기는 골든 컨텍스트를 물고 왔는데 모델의 출력이 오라클에서 튕겨냈다면, 당장 LLM 벤더를 교체하거나 프롬프트를 공격적으로 조여야 한다. (제너레이터 튜닝)

이처럼 결정론적으로 매핑된 진붓 정답지는 장애 조사 시 시스템 오동작의 근본 원인 분리(Root Cause Isolation)를 자동화하는 유일한 열쇠다.
