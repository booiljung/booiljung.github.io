3.2.2.1 모델 업데이트 및 프롬프트 변경에 따른 성능 변화 추적

AI 비즈니스 서비스는 백엔드의 거대 언어 모델(API 모델 변경, 파인튜닝, 압축 기법 등)이나 시스템의 핵심 프롬프트를 주기적으로 교체(Update)한다.

예를 들어 OpenAI의 gpt-4-0613에서 gpt-4o로 베이스 모델을 마이그레이션(Migration)할 때, 개발팀은 막연히 “새로 나온 모델이 응답이 빠르네“라고 짐작해서는 안 된다.

1. 정답지가 만드는 ‘변화의 삼각측량’

정답지는 모델 업데이트 전후를 관통하는 측정의 ’잣대’다.

  1. 구형 모델에 1,000개의 프롬프트를 날리고, 출력값을 결정론적 정답지와 매핑해 통과율(Pass Rate: 예 95\%)을 박제한다.
  2. 프롬프트를 개선하거나 신형 모델을 연결한 뒤, 똑같은 파이프라인으로 똑같은 결정론적 정답지와 비교한다.
  3. 통과율이 92\%로 폭락했다면, 비록 새 모델이 생성해 낸 문장이 더 유창하다 할지라도 이 릴리즈는 치명적인 퇴행(Regression)을 일으킨 것이므로 즉각 롤백(Rollback)해야만 한다.

    Old_Model[기존 모델 (GPT-4)] --> Test[회귀 테스트 파이프라인]
    New_Model[신규 모델 (GPT-4o)] --> Test
    
    Test --> GT[(결정론적 정답지\nGround Truth DB)]
    
    GT -- Match --> Score_Old["기존 점수: 95/100 (Pass)"]
    GT -- Match --> Score_New["신규 점수: 85/100 (Fail)"]
    
    Score_New --> Reject[신규 브랜치 배포 거부]

이런 확고한 정답지 잣대 없이는 시스템이 조금씩 좀먹어 들어가는 서서히 끓는 물(Boiling frog) 현상을 막아낼 방법이 전혀 없다.
