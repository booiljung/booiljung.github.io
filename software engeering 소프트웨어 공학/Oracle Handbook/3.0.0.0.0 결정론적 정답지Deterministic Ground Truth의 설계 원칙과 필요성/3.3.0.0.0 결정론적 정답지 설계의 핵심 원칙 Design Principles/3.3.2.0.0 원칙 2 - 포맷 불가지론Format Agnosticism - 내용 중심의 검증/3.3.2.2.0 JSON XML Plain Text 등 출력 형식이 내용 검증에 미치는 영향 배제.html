3.3.2.2 JSON, XML, Plain Text 등 출력 형식이 내용 검증에 미치는 영향 배제

LLM이 때로는 친절한 줄글(Plain Text)로, 때로는 프로그램이 읽기 좋은 JSON 스크립트로 답변을 제공할 때, 오라클의 정답지가 렌더링 형식 자체에 구속당하게 되면 테스트 시스템의 수명은 극도로 짧아진다.

flowchart LR
    Output1["응답 A: JSON 형태\n{ 'name': 'John', 'age': 30 }"] --> Parser{포맷 어댑터\n(Adapter Layer)}
    Output2["응답 B: XML 형태\n<user><name>John</name><age>30</age></user>"] --> Parser
    Output3["응답 C: Plain Text\n사용자의 이름은 John이고 나이는 30입니다."] ---> Parser
    
    Parser -- "의미 벡터/객체 추출" --> Truth["결정론적 정답지\n(Target: Name==John AND Age==30)"]
    Truth --> Verdict[Pass]

정답지 인프라 앞단에는 반드시 어댑터 계층(Adapter Layer) 이 세워져야 한다.
JSON이든 XML이든 줄글이든 상관없이, 이 출력물들을 오직 파이썬 딕셔너리(Dictionary)나 추상 구문 트리(AST) 같은 순수한 논리 구조체로 ’디코딩(Decoding)’한 이후에야, 비로소 결정론적 정답지(Target Value)와 assert 매핑을 허용해야만 한다. 포맷에 독립적인 이러한 정답지 설계만이 진정한 회귀 방어력을 제공한다.
