3.4.1.2 RAG 시스템을 위한 문서 청크(Chunk) 매핑 전략

RAG(검색 증강 생성) 아키텍처에서는 사실 기반 정답지를 일일이 사람의 손으로 타이핑해 두는 것이 불가능하므로, ‘참조해야 할 원본 문서의 조각(Document Chunk)’ 자체를 정답지의 본질로 매핑하는 문서 청크 매핑(Chunk Mapping) 전략을 사용한다.

1. 원문 앵커링(Context Anchoring) 체계

RAG 스위트용 오라클 정답지는 “무엇을 대답해야 하는가“를 하드코딩하지 않고, **“이 대답은 사내 DB 문서의 몇 단락째 조각(Chunk ID)에서 파생되어야만 하는가”**를 정답으로 규정한다.

graph LR
    Query["사용자: 24년 휴가 규정은?"] --> RAG[RAG 파이프라인]
    RAG --> Answer["LLM 응답: 총 15일 제공"]
    
    GT[(결정론적 정답지\nExpected Chunk ID: HR-24-001)] -. "비교 스캔" .-> Answer
    
    Answer --> Oracle{오라클 판별기}
    
    Oracle -- "응답의 Citation이 HR-24-001과 일치함" --> Pass
    Oracle -- "응답이 위키백과 외부 지식에 오염됨" --> Fail

즉, AI의 응답이 아무리 올바른 팩트라 할지라도, 그 답변의 뿌리(Citation)가 우리가 정답지로 지정해둔 HR-24-001 이라는 문서 파편에서 출발한 것이 아니라면(예: 모델 자체 내부 파라미터 지식에만 의존한 경우), 이 오라클은 Fail을 격발한다. 이것이 RAG 시스템 환각 통제의 첫 번째 단계다.
