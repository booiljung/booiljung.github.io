3.9.1 비즈니스 로직(약관, 프로모션 등) 변경에 따른 테스트 스위트의 진화

회사가 ‘배송비 무료 기준 3만 원’ 정책을 내일부터 ’5만 원’으로 변경하기로 결정했다고 상상해 보자. 즉각적으로 RAG 시스템의 문서(Vector DB)가 업데이트되고 모델의 응답은 5만 원으로 바뀔 것이다.
하지만 만약 QA 파이프라인 안의 결정론적 정답지가 과거의 ’3만 원’을 붙잡고 있다면, 내일 배포되는 모든 AI 시스템 빌드는 오류(Fail) 처리 백스핀을 맞게 된다.

1. 정답지 자동 무효화(Auto-Invalidation) 파이프라인

골든 데이터셋은 더 이상 수동 파일 뭉치로 존재해서는 안 되며, 회사의 메인 비즈니스 룰 엔진(Business Rule Engine) 과 실시간으로 파이프라인이 뚫려 있어야 한다.

  • 회사의 약관 DB가 업데이트되어 Git에 커밋(Commit)이 발생하는 순간, CI/CD 웹훅(Webhook)이 작동한다.
  • 웹훅은 해당 약관과 매핑(Mapping)되어 있던 오라클 정답지 세트 블록(예: Test_Suite_Shipping_Policy) 전체의 상태를 일괄적으로 Stale(상태 만료) 혹은 Invalid(무효)로 트리거한다.
  • 이 순간부터 해당 테스트는 전체 통과율(Pass Rate) 산정에서 제외되며, 새로운 비즈니스 룰을 주입받은 상태로 데이터 팀(혹은 자동화 스크립트)에 의해 덮어쓰기(Overwrite) 될 때까지 테스트 락(Lock) 상태에 방치된다.

이러한 즉각적인 무효화(Evolution by Invalidation)만이, 과거의 정답이 오늘의 오답이 되어 시스템을 마비시키는 재앙을 단절시킬 수 있다.
