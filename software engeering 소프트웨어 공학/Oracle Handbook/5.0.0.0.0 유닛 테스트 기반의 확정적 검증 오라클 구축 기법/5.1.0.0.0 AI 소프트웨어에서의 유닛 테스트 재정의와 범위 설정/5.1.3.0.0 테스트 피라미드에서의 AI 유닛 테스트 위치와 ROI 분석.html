5.1.3 테스트 피라미드에서의 AI 유닛 테스트 위치와 ROI 분석

로버트 C. 마틴(Robert C. Martin) 등이 제창한 ’테스트 피라미드’의 하단은 가볍고 빠르며 저렴한 유닛 테스트가 떠받쳐야 한다. 그러나 AI 컴포넌트를 무작정 유닛 테스트에 쑤셔 넣으면, 매 빌드마다 OpenAI API 호출 비용 폭탄과 수십 분의 지연 시간(Latency)이라는 재앙에 직면한다.

AI 컴포넌트 환경의 ROI(Return on Investment) 최적화 전략:

  • 80%의 기계적인 코드(파싱 로직, 프롬프트 빌더, DB 연동)는 목(Mocking) 객체를 사용하여 비용 ‘0원’, 실행 시간 ’1ms’의 진성 유닛 테스트 계층에 둔다.
  • 20%의 절대 양보할 수 없는 ’실제 LLM 추론 성능 확인’은 하루에 단 두어 차례만 돌아가는 통합 테스트(Integration Test) 또는 프롬프트 CI(DSPy 계층)로 승격시켜 예산을 세이브한다.
  • 즉, 모델이 어떤 대답을 할지 본질적으로 예측하기 힘든 창의적 태스크는 유닛 테스트 대상에서 과감히 제외하는 것이 오라클 구축의 황금률이다.
