5.1.4 성공 정의의 전환: ’정확한 문장 일치’에서 ’의도 및 제약 조건 충족’으로

기존 assertEquals("Hello World", response) 방식은 AI의 유닛 테스트에서 즉시 폐기되어야 한다.
오라클은 이제 다음과 같이 “제약 조건(Constraints)” 을 평가하는 방식으로 어써션(Assertion) 라이브러리를 진화시켜야 한다.

  1. 상태(State) 충족 검증:
    “요약문의 단어 수가 50개를 넘지 않는가?” (len(response.split()) <= 50)
  2. 비즈니스 의도(Intent) 검증:
    “개인 정보 블라인드 처리가 지시되었을 때, 응답에 숫자 13자리(주민번호 패턴)가 완전 배제되었는가?” (not re.search(r'\d{6}-\d{7}', response))
  3. 스키마(Schema) 정합성 검증:
    반환된 텍스트가 정확한 JSON 형태를 띠며 필수 키(Key)인 status와 reason을 포함하고 있는가?

이러한 계약 기반(Contract-based) 테스팅으로 사고를 전환하면, 수천 개의 노드가 돌아가는 AI 파이프라인에서도 흔들림 없는 자동판독 오라클을 구축할 수 있다.
