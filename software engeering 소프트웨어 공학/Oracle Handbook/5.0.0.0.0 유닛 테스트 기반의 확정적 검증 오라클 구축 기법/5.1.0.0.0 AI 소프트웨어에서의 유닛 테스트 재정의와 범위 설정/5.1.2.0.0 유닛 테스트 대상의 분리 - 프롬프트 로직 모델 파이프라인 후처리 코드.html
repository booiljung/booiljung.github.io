5.1.2 유닛 테스트 대상의 분리: 프롬프트 로직, 모델 파이프라인, 후처리 코드

AI라고 해서 모든 덩어리를 뭉뚱그려 “프롬프트 넣고 결과 받아서 테스트“하는 식의 ’Integration-like Unit Test’를 짜면 원인 추적이 불가능해진다.
AI 오라클 시스템을 구축할 때는 반드시 파이프라인을 3단계로 쪼개어 각각 격리된 유닛 테스트를 구성하라.

  1. 프롬프트 템플릿 로직 (Prompt Logic)
  • 목적: 문자열 보간(Interpolation) 자체가 안전한가? 외부 데이터가 주입될 때 구획 문자(Delimiter)가 깨지지 않는가?
  • 태스크: LLM API를 호출하지 않는다(Mocking). 오직 렌더링 된 프롬프트 텍스트 자체를 정규식으로 검수하여 로직 버그를 잡는다.
  1. 모델 엔드포인트 파이프라인 (Model Pipeline)
  • 목적: 지정된 온도(Temperature=0) 파라미터가 잘 세팅되었고, API 타임아웃이나 재시도(Retry)가 정상 동작하는가? 파운데이션 모델 자체의 버전 변화에 따른 환각 상태를 감지한다.
  • 태스크: 가장 통제 가능한 입력을 넣고, JSON 포맷이 잘 반환되는지 등 ’모델 고유의 행동 결함’만을 테스트한다.
  1. 후처리 코드 (Post-processing Parser)
    • 목적: 모델이 마크다운 블록(json) 혹은 쓸데없는 인삿말을 섞어 보냈을 때 사내 파서(Parser)가 이를 잡아내어 구조화된 객체(DTO)로 잘 변형하는가?
    • 태스크: 모델은 철저히 Mocking하고, ’의도적으로 꼬아 놓은 가짜 모델 응답 텍스트’를 주입하여 파서의 복원력을 검증한다.
