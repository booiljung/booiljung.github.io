5.3.3 시스템 프롬프트와 유저 프롬프트 조합에 따른 단위 테스트 케이스 매트릭스

오라클 시스템은 ’고정된 룰(System Prompt)’과 ’변하는 대상(User Prompt)’이 충돌하는 컨텍스트(Context) 윈도우다. 이 두 힘의 조합을 무작위 성격으로 놔두면 견고한 오라클을 구축할 수 없다.

테스트 엔지니어는 반드시 기능 명세서(TDD)를 토대로 2D 조합 매트릭스를 구축해야 한다.

  • X축 (System Prompt의 제약조건): “보안 엄격 모드 (금지어 즉시 FAIL)”, “보안 일반 모드 (금지어 경고 후 PASS)”, “자유 모드”
  • Y축 (User 데이터의 성격): “정상 코드”, “문법이 깨진 코드”, “악의적인 프롬프트 인젝션 시도”, “거짓 긍정(False Positive)을 유발하는 엣지 케이스”

이 3x4 = 12개의 조합 매트릭스 각각에 대해 기대되는 결과값(PASS, FAIL, UNKNOWN)을 엑셀이나 YAML 스키마로 하드코딩해 두고, 스위트가 매번 기계적으로 반복 테스트를 돌게 해야 한다. 이를 충족하지 못한 프롬프트 패치는 절대로 상용 파이프라인(Main Branch)에 병합(Merge) 되어서는 안 된다.
