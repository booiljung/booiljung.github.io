5.3.1 거대 프롬프트의 분해(Decomposition)와 단위 기능별 테스트 격리

복잡한 비즈니스 로직을 한 번의 LLM 호출로 풀어내려는 욕심은 테스트 실패를 부른다.
“수십 개의 조건을 확인하고, JSON으로 응답하되, 고객 이름은 가리고, 한국어로 번역해서 출력해라“라는 프롬프트를 테스트하려면 수백 개의 조건 갈래 분기(Branch)가 생겨 유닛 테스트 케이스 체계를 짤 수 없게 된다.

해결책: 파이프라인 분해 (Decomposition)
거대 프롬프트를 “단일 책임 원칙(Single Responsibility Principle)“에 따라 잘게 쪼개어 다중 LLM 호출 파이프라인으로 재구성하라.

  1. 분석 프롬프트: “결함 여부만 (PASS/FAIL)로 분석하라.” -> 이 함수에 대해서는 상태 텍스트 어써션(assertEqual)만 테스트한다.
  2. 변환 프롬프트: “위에서 FAIL이 났으면 사유를 한국어로 번역하라.” -> 이 함수에 대해서는 정규식으로 한글 유무만 판단하는 테스트 격리가 가능해진다.

이러한 분해는 LLM API 비용을 소폭 증가시킬 수 있으나, 컴포넌트 단위의 완벽한 멱등성과 테스트 격리성(Test Isolation)을 가져다준다.
