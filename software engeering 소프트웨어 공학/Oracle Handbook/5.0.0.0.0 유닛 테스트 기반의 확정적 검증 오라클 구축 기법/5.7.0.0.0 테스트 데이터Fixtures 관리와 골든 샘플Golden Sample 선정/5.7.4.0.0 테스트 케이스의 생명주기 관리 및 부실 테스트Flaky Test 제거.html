5.7.4 테스트 케이스의 생명주기 관리 및 부실 테스트(Flaky Test) 제거

결정론적 환경에서 어제는 성공했는데 오늘은 실패하는 테스트(Flaky Test)는 무시의 대상이 아니라 최우선 박멸 대상이다.

  • 캐싱(Caching) 레이어 오염 확인: Mocking되지 않고 실제 호출된 API가 모델 업데이트로 인해 포맷을 살짝 바꾼 경우, 즉각 해당 테스트 데이터를 Deprecated 처리하거나 프롬프트를 튜닝(Tuning)하여 고정해야 한다.
  • 온도(Temperature) 누수 점검: Flaky Test 발생 시 제일 먼저 체크할 것은 프롬프트 러너(Runner)의 매개변수가 Temperature=0 으로 강제 세팅되어 있는지다.
  • 좀비(Zombie) 데이터셋 청소: 1년 전의 오래된 비즈니스 룰을 기반으로 만들어진 Golden Sample은 모델이 헛소리를 한다고 착각하게 만들 수 있다. 정책이 바뀌면 룰 엔진(System Prompt)과 함께 테스트 픽스처(Fixture)도 동시에 PR(Pull Request)에 묶여 업데이트되는 생명주기 동기화가 필수적이다.
