7.3.5 코드 품질 평가를 위한 기능적 정확성과 스타일 준수 여부 분리

AI가 코드를 짰을 때 이를 검증하는 LLM 심판은 다른 어떤 도메인보다 엄격한 이원화 메커니즘을 가져야 한다.

  1. 기능적 오라클(Functional Accuracy): “이 파이썬 코드가 SyntaxError 없이 실행되는가? 시간 복잡도 O(N) 조건을 위배하지 않았는가?” -> 이는 가급적 LLM 심판보다는 9장의 정적 분석기(AST/Compiler)를 통해 1차로 기계적 필터링을 거치는 것이 옳다.
  2. 스타일 및 맥락 오라클(Style Compliance): “이 코드가 PEP8을 준수하는가? 사내 코딩 컨벤션(예: DB 접근 시 항상 TypeORM을 쓸 것)을 따르고 있는가?” -> 바로 이 지점이 LLM-as-a-Judge의 핵심 전장이다. 정적 분석기가 잡을 수 없는 ’엔터프라이즈의 인덴트(Indent)’와 ’주석의 질도’를 평가하는 정밀한 룰셋을 프롬프트에 제공하라.
