7.9.1 RAG 시스템의 검색 정확도(Retrieval Accuracy)와 답변 생성(Generation) 품질 분리 평가

RAG 파이프라인이 엉뚱한 대답을 했다면 죄인은 둘 중 하나다. 1) DB에서 문서를 아예 잘못 찾은 색인/검색기 거나, 2) 문서는 제대로 찾았는데 LLM이 헛소리한 것이다.

이를 분리해 책임을 묻는 오라클 아키텍처 (RAGAS / TruLens 차용):

  1. 검색 품질 평가: 사용자의 질문(Query)과 검색 모델이 찾아온 Context 덩어리만을 LLM 심판에게 던져, “이 문단 안에 질문에 대답할 정보가 충분히 포함되어 있는지 문서의 관련성(Context Relevance)을 1~5점으로 평가하라“고 지시.
  2. 생성 품질 평가: LLM 심판에게 검색된 Context와 봇이 생성한 최종 Answer를 제시하고, “봇의 답변이 100% 한 치의 오차 없이 저 문서 내부의 내용만으로 도출되었는지(Faithfulness) 검토하라“고 지시.
    이 두 지표를 쪼갬으로써 엔지니어는 DB 튜닝을 할지 프롬프트를 고칠지를 기계적으로 결정할 수 있게 된다.
