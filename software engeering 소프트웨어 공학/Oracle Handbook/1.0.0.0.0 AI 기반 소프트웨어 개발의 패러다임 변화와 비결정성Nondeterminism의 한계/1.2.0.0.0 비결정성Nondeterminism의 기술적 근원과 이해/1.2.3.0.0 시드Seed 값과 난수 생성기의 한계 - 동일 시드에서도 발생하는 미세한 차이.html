1.2.3 시드(Seed) 값과 난수 생성기의 한계: 동일 시드에서도 발생하는 미세한 차이

개발자들이 대규모 언어 모델(LLM)의 비결정성(Nondeterminism)을 억제하기 위해 가장 먼저 시도하는 방법은 난수 생성기(Random Number Generator)의 시드(Seed) 값을 고정하는 것이다. 이론적으로 컴퓨터 공학에서 의사 난수 생성기(Pseudorandom Number Generator, PRNG)는 초기 시드 값이 동일하다면 언제나 동일한 난수 시퀀스를 배출해야 하며, 이를 통해 완벽한 재현성(Reproducibility)을 확보할 수 있어야 한다. 그러나 LLM 환경에서 시드 값의 고정은 ’절대적인 결정론’을 보장하는 해결책이 되지 못한다.

1. 시스템 아키텍처 내 난수 생성기의 파편화

일반적인 파이썬(Python) 스크립트에서는 random.seed(42) 한 줄로 난수가 통제되지만, LLM 추론 환경은 다중 계층의 복잡한 시스템 스택 위에서 구동된다.

  1. 프레임워크 레벨: 파이토치(PyTorch)나 텐서플로우(TensorFlow) 자체의 난수 생성기
  2. CUDA / cuDNN 레벨: GPU 커널(Kernel)에서 난수를 다루는 방식의 비결정성
  3. 분산 처리 환경: 여러 노드(Node)나 여러 텐서 병렬화(Tensor Parallelism) 그룹에 시드를 동기화하는 과정의 타이밍 이슈

이러한 레이어 간의 단절로 인해 모델을 통과하는 데이터 스트림 어딘가에서 외부 요인에 의해 난수 상태가 미세하게 어긋나면, 소프트맥스(Softmax) 이후 확률 분포에서 샘플링되는 최종 토큰이 변경될 수 있다.

2. API 레벨에서의 시드(Seed) 지원과 실효성

OpenAI 등 주요 API 제공자들은 seed 파라미터를 제공하여 일관된 출력을 유도할 수 있는 기능을 지원하고 있으나, 공식 문서조차 “동일한 시드와 동일한 파라미터가 완벽한 재현을 보장하지는 않으며, 단지 일관될 가능성을 극대화(Best-effort)할 뿐“이라고 명시하고 있다. 특히 모델이 클라우드 환경에서 동적 배치(Dynamic Batching)로 다른 사용자의 쿼리와 함께 묶여 처리될 때, 메모리 정렬(Memory Alignment) 과정에서 패딩(Padding)의 미묘한 차이가 부동소수점 연산 결과에 영향을 미치기도 한다.

이처럼 시드 값의 고정은 테스트 환경에서 어느 정도의 일관성을 부여하는 베이스라인(Baseline) 역할은 수행할 수 있으나, 금융권의 원장(Ledger) 관리나 로보틱스 제어 같은 미션 크리티컬(Mission-Critical)한 상황을 담보하는 확정적 오라클(Deterministic Oracle)로 취급하기에는 공학적으로 심각한 결함을 지니고 있다.
