1.1.2 AI 기반 소프트웨어 개발(Software 2.0)의 부상: 데이터와 확률적 모델링

소프트웨어 엔지니어링의 패러다임을 뿌리째 흔드는 혁신적인 변화는 안드레이 카파시(Andrej Karpathy)가 주창한 소프트웨어 2.0(Software 2.0)이라는 개념에서 출발한다. 소프트웨어 2.0은 인간이 명시적으로 코드를 작성하여 규칙을 정의하는 소프트웨어 1.0(Software 1.0)과 본질적으로 다르다. 이 새로운 접근 방식에서는 데이터셋(Dataset)과 목적 함수(Objective Function)가 알고리즘의 동작을 정의하며, 신경망(Neural Networks) 구조가 최적화 과정을 거쳐 스스로 로직을 학습한다.

1. 데이터 중심(Data-Centric) 패러다임

소프트웨어 2.0 체제에서 코드는 로직을 담는 그릇이 아니라 로직을 도출하기 위한 ’학습 파이프라인(Learning Pipeline)’으로 역할이 축소된다. 진정한 프로그램의 명세(Specification)는 모델의 아키텍처나 파이프라인용 스크립트가 아니라 전처리된 방대한 양의 학습 데이터(Training Data)이다. 신경망은 역전파 알고리즘(Backpropagation)을 통해 손실 함수(Loss Function)를 최소화하는 방향으로 수십억 개의 가중치(Weights) 매개변수를 갱신해 나가며, 궁극적으로 데이터에 내재된 패턴과 로직을 암묵적으로 내재화(Internalization)한다.

예를 들어 고양이를 인식하는 프로그램을 작성한다고 가정해 보라. 고양이의 귀(Ear), 꼬리(Tail), 수염(Whisker)의 형태학적(Morphological) 특징을 RGB 픽셀 값의 임계치 조건(If-Else Conditions)으로 정의하는 소프트웨어 1.0의 접근은 사실상 불가능하다.

# Software 1.0 접근: 불가능에 가까운 명시적 로직
def is_cat(image):
    if count_edges(image) > 50 and check_fur_texture(image):
        return True
    return False

# Software 2.0 접근: 데이터를 통한 확률적 분류
def is_cat(image, model):
    # model은 수만 장의 고양이 사진을 보고 스스로 가중치를 조정한 신경망 객체
    probability = model.predict(image)
    return True if probability > 0.8 else False

이와 같이 소프트웨어 2.0의 본질은 인간이 규명하기 어려운 복잡한 특징 공간(Feature Space)을 신경망의 수많은 은닉층(Hidden Layer) 내부에 고차원 벡터들의 상관관계로 분산시킨다는 데 있다. 결국 모델의 예측은 참(True) 또는 거짓(False)이라는 결정론적 진리가 아니라, 주어진 데이터상황에서 정답일 확률 분포(Probability Distribution)를 반환하는 통계적 추정이 된다.

2. 확률적 아키텍처 모델링

소프트웨어 2.0 모델 기반 시스템의 동작 구조는 근본적으로 확률 모델(Probabilistic Model)에 기초한다. 이것은 기존의 결정론적 제어 흐름과는 매우 다른 양상을 띤다.

flowchart TD
    A[입력 데이터 Input Data] --> B[신경망 아키텍처 Neural Network]
    B -- 파라미터 W1, W2.. Wn --> C[가중치 행렬 연산 Matrix Multiplication]
    C --> D[활성화 함수 Activation Function]
    D --> E{소프트맥스 / 시그모이드 Softmax / Sigmoid}
    E -- 클래스별 확률 제공 --> F[출력: 고양이일 확률 83%]
    F -- 임계값 Threshold 반영 --> G[최종 식별값 반환]

위의 아키텍처에서 보이듯, 로직은 직접 구성되는 것이 아니라 매트릭스 안에 수치적으로 암호화된다. 이러한 특성은 명백한 혜택을 수반한다. 예외 케이스 처리(Exception Handling)를 수천 줄의 코드로 작성하지 않아도, 충분히 많은 다양성을 가진 데이터가 확보된다면 모델이 어느 정도의 일반화(Generalization) 능력을 보여주게 된다.

그러나 확률적 모델링이 수반하는 가장 큰 맹점은 소프트웨어의 응답이 결코 100% 확실할 수 없다는 것이다. 가장 훈련이 잘된 모델이라 하더라도, 이전에 관측하지 못한 데이터 분포 아웃라이어(Out-of-Distribution, OOD)나 적대적 예제(Adversarial Example)를 만날 경우 전혀 예상치 못한 엉뚱한 결론을 도출할 수 있다. 이러한 소프트웨어 2.0의 내재적 확률성(Stochasticity)은 소프트웨어 공학의 ‘검증 가능성(Verifiability)’ 문제로 직결되며, 이를 극복하는 것이 새로운 엔지니어링 패러다임의 가장 큰 과제로 부상하였다.
