1.5.4 비용 예측의 어려움: 재시도(Retry) 로직 증가에 따른 토큰 비용 및 레이턴시 폭증

전통적인 클라우드(Cloud) 컴퓨팅 환경에서 서버의 연산 비용(Compute Cost)과 응답 지연(Latency)은 시간 복잡도(Time Complexity)와 요청량에 거의 정확히 비례하는 선형적 성격을 띤다. 그러나 비즈니스 파이프라인 정중앙에 대규모 언어 모델(LLM)이 편입되면 경제성 타당성(Economic Feasibility) 모델은 산산조각이 나고 만다. 비결정적 출력을 회복(Recovery)하기 위해 가동되는 악명 높은 재시도(Retry) 로직 때문이다.

1. 토큰(Token) 기반 과금 구조의 잔혹성

주요 API 제공자(OpenAI, Anthropic 등)는 철저하게 사용된 토큰 수량(입력 + 출력)을 기준으로 비용을 청구한다. 만약 LLM 시스템이 약속된 JSON 규격이나 특정 마크다운 구조를 정확히 뱉어내지 않아 파서(Parser)에서 JSONDecodeError를 리턴한다면, 시스템이 할 수 있는 일은 크게 두 가지뿐이다.

  1. 사용자에게 파멸적인 에러 화면을 노출하고 종료하기.
  2. 예외를 잡아낸 뒤(Catch Exception) 백그라운드에서 다시 한 번 LLM을 호출하여 ’올바른 형식으로 한 번 더 대답해 줘’라고 Retry 핑을 때리기.

어느 정상적인 서비스라면 당연히 2번의 자기 수정성 재시도 파이프라인(Self-Correction Pipeline)을 설계한다.

2. 레이턴시와 비용 계산의 기하급수적 발산

하지만 확률 모델의 무작위성 때문에 정답 포맷이 1번에 떨어진다는 보장은 어디에도 없다. 어떤 요청은 1회 호출(Cost: 1x, Time: 2초)에 끝나지만, 형태가 조금 복잡한 응답은 내부에서 실패를 4~5번 거듭하다가(Cost: 5x, Time: 10초) 간신히 정답을 내뱉기도 한다.

graph TD
    User([사용자 API 요청]) --> Invoke1[1차 LLM 호출]
    Invoke1 --> Validator{유효성 검증 Validation Oracle}
    Validator -- 통과 --> Success([비용 1배, Latency 2초 응답])
    
    Validator -- 실패 환각/포맷파괴 --> Retry1[포맷 강제 지시어 추가 + 2차 호출]
    Retry1 --> Validator2{재검증}
    
    Validator2 -- 실패 --> Retry2[3차 호출...]
    Retry2 -- 간신히 통과 --> FailCost([비용 3배, Latency 8초 응답])

비즈니스 플래너(Business Planner)가 한 달 트래픽 100만 건, 건당 평균 0.01달러로 월 1만 달러의 클라우드 예산을 기안 올렸다고 해도, 시스템의 불안정성(Flakiness)이 짙어져서 재시도율이 30%를 기록한다면 예산은 순식간에 두 배 가까이 치솟는다. 게다가 10초 가까운 레이턴시 지연은 사용자의 이탈(Churn Rate)을 가속화시킨다.

따라서 AI 시스템에서 출력의 비결정성은 단순한 알고리즘의 오류로 취급되어서는 안 되며, 기업의 인프라스트럭처 재무(FinOps) 구조를 무너뜨리는 현금 유출의 블랙홀(Sinkhole)로 여겨져야 한다.
