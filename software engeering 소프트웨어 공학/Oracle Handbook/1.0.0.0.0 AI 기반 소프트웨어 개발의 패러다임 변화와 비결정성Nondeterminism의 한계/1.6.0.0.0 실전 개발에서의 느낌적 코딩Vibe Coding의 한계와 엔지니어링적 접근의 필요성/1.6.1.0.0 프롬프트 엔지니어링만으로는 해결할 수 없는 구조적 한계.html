1.6.1 프롬프트 엔지니어링만으로는 해결할 수 없는 구조적 한계

개발자들이 AI의 비결정성을 통제하기 위해 가정 먼저 도입하는 수단은 프롬프트 엔지니어링(Prompt Engineering)이다. “너는 20년 차 시니어 개발자다”, “반드시 JSON 포맷만 출력하라”, “환각을 일으키면 패널티를 주겠다“와 같은 장황한 명령어들을 통해 모델의 출력 공간(Output Space)을 강제로 좁히려 시도한다. 그러나 본질적으로 소프트웨어 엔지니어링은 이러한 자연어 기반의 통제 수단만으로 결코 대체될 수 없다.

1. 자연어의 선천적 모호성

프로그래밍 언어(C++, Python, Rust 등)는 문맥 의존성이 배제된 ’형식 언어(Formal Language)’로, 컴파일러나 인터프리터에 의해 오직 단 하나의 의미로만 해석되도록 설계되었다. 반면 인간의 자연어는 다의성(Polysemy)과 모호성(Ambiguity) 덩어리다.

# 형식 언어의 엄밀함
if error_count > 5:
    alert_admin()

# 프롬프트의 모호성
# "에러가 너무 많이 발생하면 관리자에게 알려줘."

위의 지시에서 “너무 많이(Too many)“의 기준 모델의 훈련 데이터 문맥과 샘플링에 따라 count > 3이 될 수도, count > 100이 될 수도 있다. 개발자가 아무리 장황한 수식어를 덧붙여 제한을 가한다 하더라도, 그것은 여전히 해석의 여지를 남겨두는 확률 기계에 파라미터를 입력하는 것에 지나지 않는다.

2. 컨텍스트 손실과 지시망각의 벽

더욱이 비즈니스 요구사항이 복잡해질수록 지시문은 수천 토큰에 달하는 방대한 문서로 변모한다. 앞에서 언급했듯이 대규모 언어 모델은 입력된 컨텍스트 길이에 따라 중간의 지시사항을 잃어버리는 ‘Lost in the middle’ 증후군을 겪는다.

결국 프롬프트 튜닝은 “초록색 셔츠를 입은 고양이” 그림을 생성하기 위해 “매우 쨍한 초록색, 4개의 발, 고양이, 귀여운” 같은 키워드를 끊임없이 추가하는 시도일 뿐이며, 소프트웨어 공학에서 요구하는 100% 무결성 명세를 대체할 수 없는 근본적인 구조적 한계(Structural Limitation)를 안고 있다.
