1.4.1 전통적 단위 테스트(Unit Test)의 한계: assert(output == expected) 구문의 무력화

오랜 기간 동안 소프트웨어 엔지니어들은 테스트 주도 개발(Test-Driven Development, TDD) 철학 아래 코드를 작성해 왔다. 이 방법론의 정수에 자리하는 것이 바로 assert(기대값 == 실제값) 비교 구문이다. 특정한 입력을 함수에 전달하고 리턴된 값이 정확히 엔지니어가 예측한 문자열이나 숫자와 일치하는지, 바이트 단위까지 엄격하게 검사함으로써 로직의 유효성을 보장한다. 하지만 AI를 호출하는 함수 앞에서는 이 강력했던 등호 연산자(==)가 힘을 잃는다.

1. 등호(==) 연산자와 일치성 검증의 종말

언어 모델(LLM) 기반의 기능 테스트를 시도해 보면, 동일한 의미(Semantics)를 내포하고 있더라도 모델은 매 실행마다 미세하게 다른 포맷, 동의어, 조사, 어조를 섞어 응답을 반환한다.

# Software 1.0의 결정론적 테스트
def test_string_uppercase():
    assert to_uppercase("hello") == "HELLO"  # 영원히 통과함

# Software 2.0의 확률적 테스트 (실패 가능성 100%)
def test_ai_translation():
    result = ai_translate("hello", target="Korean")
    
    # 1차 시도 반환값: "안녕하세요."
    # 2차 시도 반환값: "안녕"
    # 3차 시도 반환값: "안녕하세요! 무엇을 도와드릴까요?"
    assert result == "안녕하세요" # 거의 대부분 실패함

단위 테스트 환경 내에서 assert(output == "안녕하세요") 구문은 모델이 점(.) 하나를 더 찍거나, 띄어쓰기를 바꾸거나, 정중한 맺음말을 추가하는 순간 무자비하게 실패(Fail)를 반환한다. 테스트의 실패는 사실 로직의 버그를 의미하는 것이 아니라, 테스팅 프레임워크가 AI의 동적 유연성을 수용할 수 있을 만큼 똑똑하지 않음을 자백하는 현상에 불과하다.

이에 대한 대안으로 일부 개발자들은 정규표현식(Regular Expression)이나 분리된 키워드 포함 검사(예: assert "안녕" in result)를 사용하기도 한다. 그러나 이는 부분적인 미봉책에 그칠 뿐이다. 문자열의 부분 일치만을 허용하게 되면 모델이 “안녕이라고 말하지 않을 것입니다.“라는 치명적으로 반대되는 결과를 내뱉어도 테스트가 모순적으로 통과(False Positive)해 버리는 대참사가 발생한다. 결국 단순한 매칭을 넘어선 새로운 차원의 오라클(Oracle) 엔진이 필요해진다.
