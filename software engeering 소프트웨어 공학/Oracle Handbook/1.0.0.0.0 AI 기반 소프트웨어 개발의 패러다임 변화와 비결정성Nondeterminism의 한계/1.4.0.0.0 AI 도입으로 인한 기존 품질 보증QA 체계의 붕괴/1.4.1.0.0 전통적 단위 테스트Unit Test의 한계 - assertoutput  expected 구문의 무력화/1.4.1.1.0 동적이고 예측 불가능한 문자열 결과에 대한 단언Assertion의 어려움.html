1.4.1.1 동적이고 예측 불가능한 문자열 결과에 대한 단언(Assertion)의 어려움

대규모 언어 모델(LLM)이 생성하는 문자열 출력(String Output)은 구조적으로 예측 불가능(Unpredictable)하며 정형화된 형태를 따를 의무를 지니지 않는다. 특히 마크다운(Markdown), JSON, YAML과 같은 프로그래밍 호환 포맷을 요구할 때조차, 결과물의 래핑(Wrapping) 방식이 동적으로 변경된다.

1. 동적 문자열의 형태 변환 문제

비즈니스 파이프라인에서 AI에게 정보를 요약하여 JSON 객체 하나만 뱉어내라고 지시하는 경우를 생각해보라. AI 모델은 매 실행 시마다 다른 래퍼(Wrapper) 텍스트를 제공한다.

// Case A: 올바른 형태
{ "status": "ok", "value": 100 }

// Case B: 마크다운 래핑 삽입
```json
{ "status": "ok", "value": 100 }

// Case C: 대화형 서두 추가
알겠습니다. 요청하신 JSON 데이터를 결과로 제공합니다.
{
“status”: “ok”,
“value”: 100
}


위의 3가지 케이스는 의미론적으로는 정확히 100% 동일한 비즈니스 로직 결과를 담고 있다. 하지만 이를 코드로 파싱(Parsing)하여 검증하는 기존의 `JSON.parse(output)` 명령어는 Case B와 Case C에서 치명적인 문법 오류(Syntax Error)를 일으키며 즉시 프로그램 실행을 중단시킨다.

## 2. 단언(Assertion) 작성의 피로도 증가


이러한 출력의 동적 변동성은 개발자로 하여금 단언(Assertion)을 작성하는 데 극도의 피로를 조장한다. 단 하나의 테스트 케이스를 통과시키기 위해 결과값 문자열의 서두와 후미를 잘라내고(Trimming), 특정 마크다운 기호를 정규표현식으로 치환(Replace)하며, 대소문자를 정규화(Normalization)하는 파싱 코드가 배보다 배꼽이 큰 수준으로 불어나는 것이다.

위 흐름도에서 보듯 기존에는 Assert 하나로 끝났을 검증 파이프라인이 AI를 만나는 순간 복잡한 문자열 전처리 및 추론 파이프라인으로 뻗어나가게 된다. 이는 QA 스위트(QA Suite)의 실행 시간을 폭증시키며, 유지보수의 난이도를 돌이킬 수 없는 지점으로 끌어올리는 주된 요인이다.
