1.4.1.2 모킹(Mocking) 및 스터빙(Stubbing) 패러다임의 붕괴

단위 테스트(Unit Test)를 고립된(Isolated) 환경에서 빠르고 안전하게 실행하기 위해 소프트웨어 엔지니어들은 외부 시스템의 응답을 가짜 객체로 대체하는 모킹(Mocking)과 스터빙(Stubbing) 기법을 애용해 왔다. 데이터베이스 호출이나 외부 API 지연 문제를 해결하기 위해, 테스트 환경에서는 특정 함수가 항상 정해진(Static) JSON 결과를 즉각 반환하게끔 인터셉터(Interceptor)를 설정하는 식이다.

1. AI 시대, 무엇을 어떻게 흉내 낼 것인가?

모킹의 본질적 전제는 ’외부 의존성의 응답 규격이 명확하고 변하지 않는다’는 것이다. 그러나 AI 기능이 통합된 로직에서는 API 계층을 모킹하는 것 자체가 테스트의 가치를 사실상 파괴(Invalidate)한다.

개발자가 파이프라인 테스트를 위해 대규모 언어 모델(LLM)의 API 반환값을 "이 문서는 계약서입니다"라고 모킹(Mocking)했다고 가정해 보자. 이 테스트는 1밀리초 만에 성공적으로 통과할 것이다. 그러나 이는 비즈니스 로직 파이프라인 중간에 존재하는 수많은 텍스트 전처리 패널티, 환각, 그리고 예측 불가능성을 완벽히 무시한 ’잘못된 위도(False Comfort)’에 불과하다.

    class TraditionalTest {
        <<기존 방식>>
        + Database (Mocked: 빠른 응답)
        + Network API (Stubbed: 고정 상태코드)
        + 결과: 예측 가능한 로직 흐름 보장
    }
    class AITest {
        <<LLM 컴포넌트 포함 방식>>
        + LLM_API (Mocked? 무결성 훼손)
        + 실제 호출 시 토큰 수 제한, 환각, 트림 누락 등 이슈 존재
        + 결과: Mock이 실제 프로덕션 리스크를 대변하지 못함
    }

2. 모의 응답(Mock Response) 구조의 한계

AI 시스템에서는 프롬프트(Prompt) 지시어 하나만 바뀌거나 주어지는 컨텍스트 윈도우 데이터의 일부분이 변경되면 결과의 도메인이 극단적으로 틀어진다. 테스트 코드 안에서 수백 가지 변형을 모두 모킹 데이터로 구성하는 것은 불가능하며 유지보수 역시 파탄에 직면한다.

결국 LLM이 관여하는 테스트는 가짜 응답을 주입하는 Mocking 계층을 걷어내고, 실제 라이브(Live) 모델이나 평가 전용 프록시(Proxy) 모델을 직접 호출해서 검증해야(End-to-End Testing) 의미를 지닌다. 그러나 이는 테스트 속도를 기하급수적으로 저하시키고 API 과금 비용의 폭탄을 낳으므로 전통적인 테스트 주도 개발(TDD)의 빠른 순환 피드백 고리(Fast Feedback Loop)를 근본적으로 파괴하는 결과를 낳게 된다.
