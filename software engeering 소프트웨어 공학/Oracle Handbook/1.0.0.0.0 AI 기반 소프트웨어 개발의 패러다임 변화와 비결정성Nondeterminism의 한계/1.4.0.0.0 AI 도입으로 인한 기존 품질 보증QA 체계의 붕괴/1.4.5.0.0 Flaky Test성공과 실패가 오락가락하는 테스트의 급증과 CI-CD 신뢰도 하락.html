1.4.5 Flaky Test(성공과 실패가 오락가락하는 테스트)의 급증과 CI/CD 신뢰도 하락

자동화 배포 파이프라인(CI/CD Pipeline)을 지탱하는 가장 큰 미덕은 일관성에 대한 ’신뢰 지표(Confidence Level)’다. 개발자가 커밋(Commit) 푸시 1번을 발생시킬 때마다 수백 개의 엔드투엔드(E2E) 스위트가 실행되며, 만약 초록불(Green)이 떴다면 이 기능은 무조건 실무에 배포해도 결함이 없음(Safe to Ship)을 보증해야 한다. 하지만 비결정적 AI 컴포넌트가 이 흐름에 내장되는 순간 심각한 품질 감염인 플레이키 테스트(Flaky Test) 현상이 급증한다.

1. 시스템 신뢰도를 좀먹는 Flaky Test 파충류

플레이키 테스트란 코드를 1바이트도 건드리지 않고 단순히 테스트를 여러 번 반복해서 실행(Re-run)했을 뿐임에도 불구하고, 어떤 때는 패스(Pass)하고 어떤 때는 실패(Fail)하는 오락가락한 현상을 일컫는다.

AI 로직을 테스트하는 환경에서는 이 현상이 만연하다. 확률적 샘플링이나 무작위성(Temperature, Top-p)에 의해 모델의 출력이 미세하게 변형되며 검증 필터(Regex 기반 혹은 키워드 매칭 등)의 선을 아슬아슬하게 넘나들기 때문이다. 처음 테스트 시에는 단언(Assert)을 통과했으나, 30분 뒤 동일한 빌드 서버에서 다시 트리거했을 때는 모델 응답 지연으로 발생한 타임아웃, 예기치 않은 인사말(Greeting) 추가로 인한 파싱 에러로 붉은 불(Red)이 켜진다.

2. “다시 한 번 돌려봐(Just Restart It)” 문화의 위험성

이러한 현상은 엔지니어링 퀄리티 및 조직 문화에 엄청난 치명타를 가한다. 특정 단위 테스트가 실패했을 때, 개발자들은 “아, AI가 일으킨 간헐적 오류(Flakiness)일 거야. 재시작 버튼 누르면 이번엔 통과되겠지“라고 반응하기 시작한다. 이 행동이 세 번만 반복되면, 그 팀은 자동화된 테스트 파이프라인(CI/CD)의 알람에서 완전히 눈을 닫아버리게 된다(Alert Fatigue).

  1. 테스트가 실패해도 아무런 치명감(Sense of urgency)을 느끼지 않는다.
  2. 진짜 심각하게 부서진(Broken) 코드 로직 변경조차 AI의 플레이키 테스트 결과와 섞여 무시된 채로 운영 환경에 병합(Merged)된다.
  3. 자동화 배포에 대한 조직적 불신(Distrust)이 고착화된다.

결국 Flaky Test를 억제하고 확률적 텍스트에서 명백한 객관성(Objectiveness)을 도출하기 위해서는 통과 기준의 허용 오차(Tolerance Level)를 정밀하게 설계하고, 판정 과정을 결정론적 틀 안에 강제로 포획해내는 오라클(Oracle) 평가 프레임워크가 CI/CD 체계의 심장부에 필수적으로 삽입되어야만 한다.
