1.4.2 테스트 케이스 설계의 난이도 증가: 무한에 가까운 입력 공간과 예측 불가능한 출력 공간

기존의 품질 보증(QA) 엔지니어가 시스템을 부수어(Break) 결함을 찾기 위해 사용하는 무기는 경계값 분석(Boundary Value Analysis)과 동치 분할(Equivalence Partitioning) 기법이다. 예를 들어, 10대 할인을 적용하는 함수가 있다면 입력값 9, 10, 19, 20의 4가지 숫자만 치밀하게 찔러보면 해당 모듈의 무결성을 99.9% 보증할 수 있다. 입력 가능한 정수의 한계(Input Space)는 넓을지라도 논리적 동치 클래스(Class)는 명확히 제한되기 때문이다.

1. 자연어 입력의 폭발적 무한성 (Infinity of Input Space)

하지만 AI를 상대로 한 프롬프트나 사용자의 텍스트 질의는 유한하지 않다. 동일한 의미를 가진 질문이라도 수천 가지의 변형(Augmentations)이 존재한다.

  • “배송 언제 와?”
  • “물건 언제 도착함?”
  • “제가 주문한 물품의 수령 예정일이 궁금합니다.”
  • “where is my parsing (오타 포함)”

전통적인 시스템에서는 정해진 버튼 클릭이나 API 헤더의 규격만 검사하면 되었지만, LLM 앞에서는 고객이 시도할 수 있는 모든 언어적 조합, 오탈자, 은어, 은유법이 전부 시스템을 위협하는 유효 공격 표면(Attack Surface)이 된다.

2. 예측 불가한 다차원 출력 공간 (Unpredictable Output Space)

무한한 입력을 시스템이 어떻게든 처리했다 하더라도 튀어나오는 출력 역시 예측이 불가하다. QA 엔지니어가 예상한 범주(“환불 완료” 또는 “환불 불가”)를 가볍게 무시하고 “오늘은 날씨가 좋네요, 근데 환불은 어려워요“처럼 사담을 끼워 넣거나, 윤리적 방어 필터가 과도하게 작용하여 단순 기능 문의에도 “도덕적으로 답변할 수 없다“는 응답을 반환하기도 한다.

입력 공간과 출력 공간이 모두 ‘무한(Infinity) 곱하기 무한’ 차원으로 발산(Divergence)하는 상황에서, 모든 경우의 수를 커버하는 코드 커버리지(Code Coverage) 100%라는 목표는 신기루일 뿐만 아니라 개념적으로 달성 불가능하다. 이는 시스템 테스트의 패러다임을 ’사전 예방(Preventation) 및 예측(Prediction)’에서 ‘사후 감시(Monitoring) 및 평가적 방벽(Evaluative Guardrails)’ 체계로 180도 전환시킬 것을 강요한다.
