1.3 결정론적(Deterministic) 시스템 vs 확률적(Probabilistic) 시스템의 충돌

소프트웨어 1.0 시대를 지탱해 온 공학의 뿌리는 철저하게 결정론적(Deterministic) 패러다임에 근간을 두고 있다. 그러나 데이터를 기반으로 스스로 로직을 추론하는 대규모 언어 모델(LLM)이 현업의 프로덕션 단계(Production Stage)에 침투하면서, 우리는 완벽한 결정론과 내재된 확률성이라는 완전히 상반되는 두 철학의 맹렬한 충돌을 목도하고 있다.

1. 상반된 두 철학의 구조적 갈등

전통적인 소프트웨어 아키텍처는 톱니바퀴처럼 빈틈없이 맞물려야 하는 기계 메커니즘과 같다. 반면 AI, 즉 확률적 시스템(Probabilistic System)은 생물학적 뇌의 시냅스 연결과 같이 유연하지만 결코 100% 동일성을 띠지 않는다. 이 둘을 하이브리드(Hybrid) 형태로 단일 서비스 아키텍처 내에 통합할 때, 시스템 경계(System Boundary)에서 필연적으로 데이터의 신뢰성 충돌이 일어난다.

classDiagram
    class DeterministicSystem {
        <<Software 1.0>>
        + 명시적 로직 (Explicit Logic)
        + 100% 재현성 (Reproducibility)
        + 불변의 상태 (Immutable State)
        + 항상 동일한 응답 분기
        +RuleBasedValidation()
    }
    class ProbabilisticSystem {
        <<Software 2.0 / LLM>>
        + 확률적 분포 (Probability Distribution)
        + 모델 드리프트 (Model Drift)
        + 무작위 샘플링 (Random Sampling)
        + 매번 다른 텍스트/코드 생성
        +GenerativeInference()
    }
    
    DeterministicSystem <--x ProbabilisticSystem : 1. 데이터 파싱 오류
    DeterministicSystem <--x ProbabilisticSystem : 2. 통제 범위 이탈

2. 충돌이 발생하는 주요 영역

이러한 충돌은 다음의 세 가지 차원에서 극명하게 드러난다.

  1. 데이터 무결성(Data Integrity): 관계형 데이터베이스(RDBMS)나 백엔드 로직은 단 한 글자의 오타도 시스템 예외(Exception)로 처리한다. 반면 확률 시스템은 “User“라는 단어를 어느 날은 “사용자”, 어느 날은 “Client“로 매핑하여 시스템에 전달한다.
  2. 에러 핸들링(Error Handling): 결정론적 시스템은 실패 시 콜스택(Call Stack)과 명확한 에러 코드를 제공하여 복구를 유도한다. 그러나 확률 시스템은 엉뚱한 로직을 마치 성공적인 대답인 것처럼 포장하여(환각) 통과시키므로, 전통적인 가드레일(Guardrails)을 무력화한다.
  3. 배포 주기와 형상 관리(Version Control): 전통적 코드는 깃(Git) 브랜치를 통해 철저히 버전업되며 과거 코드는 완전히 보존된다. 하지만 API 형태의 LLM은 실시간으로 가중치가 조정될 수 있어 특정 커밋(Commit)에서의 완벽한 동작을 미래에 보장하지 못한다.

결과적으로, 결정론적 시스템의 강건함을 보호하기 위해 확률적 시스템의 출력을 방역하고 검증하는 ’확정적 오라클(Deterministic Oracle)’의 배치가 그 어느 때보다 중요해졌다. 본 장에서는 이 거대한 충돌에서 나타나는 비즈니스 및 엔지니어링 한계점들을 하나씩 해부한다.
