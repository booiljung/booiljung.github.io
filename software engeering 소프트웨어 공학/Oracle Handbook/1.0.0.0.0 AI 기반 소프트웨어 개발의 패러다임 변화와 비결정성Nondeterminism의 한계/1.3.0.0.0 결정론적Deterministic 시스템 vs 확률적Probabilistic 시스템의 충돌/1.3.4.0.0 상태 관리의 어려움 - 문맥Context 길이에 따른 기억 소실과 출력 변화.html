1.3.4 상태 관리의 어려움: 문맥(Context) 길이에 따른 기억 소실과 출력 변화

결정론적 애플리케이션에서는 변수에 할당된 값이나 데이터베이스에 커밋(Commit)된 레코드는 전원이 차단되거나 명시적인 Update/Delete 쿼리가 발생하지 않는 한 영구적으로 상태(State)가 관리된다. 그러나 대규모 언어 모델(LLM)을 상태 관리의 코어 혹은 파서(Parser)로 활용하는 순간, ’문맥 길이(Context Window)’라는 치명적인 물리적 한계표와 ’기억 소실(Amnesia)’이라는 인지적 한계표가 동시에 작동하기 시작한다.

1. 컨텍스트 윈도우(Context Window)의 제약

LLM은 본질적으로 무상태(Stateless) 시스템이다. 모델이 긴 대화나 방대한 양의 코드를 ’기억’하는 것처럼 보이는 이유는, 매 요청(Request)마다 과거의 모든 이력(History)을 하나의 거대한 프롬프트 모음으로 전부 묶어서 다시 전송하기 때문이다.

모델의 버전에 따라 다르지만 8K, 32K, 128K 등으로 컨텍스트 길이가 엄격하게 제한되어 있으며, 이 허용 토큰 수를 단 한 글자라도 초과하는 순간 앞부분의 텍스트가 잘려나가는(Truncated) 정보의 하드 드롭(Hard Drop)이 일어난다. 중요하게 선언했던 글로벌 변수 규칙이나 페르소나(Persona) 지침이 대화의 앞쪽에 위치했다면, 그 지침은 영구적으로 소실된다.

2. Lost in the Middle 증후군

더 심각한 문제는 컨텍스트 한도 내에 정보가 위치하더라도 발생하는 이른바 ‘중간 유실(Lost in the Middle)’ 현상이다. 여러 연구(“Lost in the Middle: How Language Models Use Long Contexts” 등)에 따르면, 최신 LLM들은 프롬프트의 맨 앞(Primacy Effect)과 맨 뒤(Recency Effect)에 주어진 정보는 비교적 정확하게 파악하고 출력에 반영하지만, 텍스트 뭉치 중간에 삽입된 중요한 조건이나 논리는 무시하거나 잊어버리는 치명적인 패턴을 보인다.

3. 결정론적 일관성의 붕괴

이러한 특성으로 인해 10줄의 간단한 코드를 생성할 때의 출력 퀄리티와, 2,000줄의 전체 모듈을 컨텍스트로 제공한 뒤 특정 함수 수정을 지시했을 때 모델이 내뱉는 코드의 퀄리티가 극명하게 달라진다.

방대한 문맥을 제공하면 오히려 모델은 그 방대함에 노이즈(Noise)를 느끼고 주의(Attention) 분배에 실패하여 이전에 잘 지키던 포맷팅이나 코딩 컨벤션(Coding Convention)을 갑작스럽게 위반하기 시작한다. 이는 문맥의 길이가 가변적인 프로덕션 운영 환경에서 AI가 내놓는 결과물의 일관성을 도저히 신뢰할 수 없게 만드는 또 다른 비결정성의 지뢰밭이자 거대한 상태 관리의 난관이다.
