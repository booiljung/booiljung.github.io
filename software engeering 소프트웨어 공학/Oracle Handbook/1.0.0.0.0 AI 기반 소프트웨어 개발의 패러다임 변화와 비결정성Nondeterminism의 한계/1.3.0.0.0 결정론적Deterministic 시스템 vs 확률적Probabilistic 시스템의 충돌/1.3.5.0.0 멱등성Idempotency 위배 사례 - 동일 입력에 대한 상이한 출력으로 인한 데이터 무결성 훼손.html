1.3.5 멱등성(Idempotency) 위배 사례: 동일 입력에 대한 상이한 출력으로 인한 데이터 무결성 훼손

컴퓨터 과학과 분산 시스템 아키텍처에서 멱등성(Idempotency)이란, 동일한 연산이나 요청을 여러 번 수행하더라도 그 결과나 시스템의 최종 상태가 한 번 수행했을 때와 동일하게 유지되는 성질을 의미한다. RESTful API에서 PUT이나 DELETE 메서드가 대표적인 멱등적 연산이다. 엔터프라이즈 시스템이 복잡한 네트워크 장애 환경에서도 데이터 정합성(Data Consistency)을 유지할 수 있는 것은 이 멱등성 메커니즘이 강제되어 있기 때문이다.

그러나 시스템의 제어 흐름에 대규모 언어 모델(LLM)이 개입하면, 이 멱등성의 원칙은 가장 먼저 파괴되는 희생양이 된다.

1. 비결정성이 파생하는 멱등성 붕괴

사용자가 동일한 자연어로 “A 고객의 배송지를 ’서울 강남구’로 업데이트 쿼리를 작성해 실행해줘“라고 요청했다고 가정해 보자.

  • 첫 번째 호출 시도: LLM이 정상적으로 UPDATE users SET address = '서울 강남구' WHERE user_id = 'A' 쿼리를 생성하여 실행한다 (성공).
  • 네트워크 타임아웃 발생: 시스템이 일시적인 멈춤 현상으로 재시도(Retry) 로직을 백그라운드에서 발동시킨다.
  • 두 번째 호출 시도 (자동 Retry): LLM의 내부 샘플링(Temperature)이나 노이즈에 의해, 이번에는 뜻을 과도하게 해석하여 UPDATE users SET address = '서울 강남구' WHERE name LIKE '%A%' 라는 광범위 파괴 쿼리를 생성해버린다.

결과는 참담하다. 단 1건의 고객 정보가 바뀌어야 할 시스템에서 이름에 ’A’가 들어간 수백 명의 배송지가 일괄 변경되는 대형 사고가 터지는 것이다.

2. 복잡성 폭발과 데이터 오염

이러한 사례는 단순히 SQL 생성 오류에만 국한되지 않는다. 문서를 파싱(Parsing)해서 JSON으로 저장할 때도, 동일 문서를 넣어도 배열(Array)의 개수가 달라지거나 키(Key) 이름이 대소문자로 혼용(customerName vs Customer_Name)되어 들어가면 백엔드의 비즈니스 로직 스키마는 파싱 에러를 내며 연쇄적으로 다운된다.

엔지니어링 시스템은 ’거의 같은 입력이면 멱등적으로 거의 같은 결과를 보장’하는 낭만적인 관용을 절대 베풀지 않는다. AI 모델의 비결정적 출력을 그대로 시스템의 Write/Update 단계를 제어하는 스티어링 휠(Steering wheel)에 연결하는 행위는 본질적으로 폭주하는 무작위 난수기를 데이터베이스에 직결시켜버리는 최악의 아키텍처 실수이며, 반드시 중간에 출력값을 멱등적 포맷으로 격리 및 검증해내는 오라클(Oracle) 계층이 필수 불가결하다.
