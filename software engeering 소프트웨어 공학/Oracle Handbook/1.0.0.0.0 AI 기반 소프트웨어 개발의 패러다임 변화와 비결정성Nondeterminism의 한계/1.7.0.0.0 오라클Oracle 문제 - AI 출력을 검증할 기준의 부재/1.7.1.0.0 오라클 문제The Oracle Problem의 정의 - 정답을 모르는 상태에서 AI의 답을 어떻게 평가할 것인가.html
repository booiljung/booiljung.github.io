1.7.1 오라클 문제(The Oracle Problem)의 정의: 정답을 모르는 상태에서 AI의 답을 어떻게 평가할 것인가?

소프트웨어 공학에서 ’오라클(Oracle)’이란 고대 그리스 신화에서 신의 뜻을 전달받아 진실을 판별해 주던 델포이의 신탁에서 유래한 용어로, 주어진 입력에 대해 시스템이 “어떤 출력을 내놓아야만 올바른 것인지 결정해 주는 메커니즘이나 기준” 을 뜻한다.

따라서 ’오라클 문제(The Oracle Problem)’란, 시스템을 테스트하기 위한 완벽한 절대 정답을 사전에 알지 못하거나, 정답을 생성하고 판독하는 행위 자체가 테스트 대상 시스템을 모방하는 것만큼이나 극도로 복잡하고 비용이 많이 들어서, 사실상 결과의 참/거짓 매핑을 확정적으로 수행할 수 없는 딜레마를 의미한다.

1. AI 시스템에서 극대화되는 오라클 딜레마

전통적인 계산기 프로그램은 add(2, 3)의 결과가 5인지 아닌지를 확인하는 데 있어 완벽한 산술적 오라클을 보유하고 있다. 컴퓨터 내장 연산자(+)나 인간의 뇌가 완벽한 비교 오라클로 기능하기 때문이다. 그러나 대규모 언어 모델(LLM)에게 복잡한 프로그래밍 코드를 짜달라고 명령하거나, 법률 데이터를 해석해 달라고 지시했을 때는 이 오라클이 자취를 감춘다.

  1. 절대적 정답의 부재(Absence of Absolute Correctness): “효율적인 정렬 알고리즘을 짜줘“라는 질문에 정답은 수백 가지 존재한다. Quick Sort, Merge Sort 등 환경에 따라 정답이 변하므로 1:1 매칭이 불가능하다.
  2. 동일한 복잡도: AI가 출력한 200줄짜리 스마트 컨트랙트(Smart Contract) 소스 코드가 보안 취약점 없이 작동하는지 인간이 파악하려면, 사실상 개발자가 그걸 처음부터 직접 짜는 것과 동일한 수준의 인지적 노력이 필요하다. 즉, 오라클 검증 비용이 생성 비용보다 역전되는 현상이 벌어진다.

이처럼 “이 답이 맞는가?“를 기계적으로(Automated) 판독해 줄 수단이 붕괴된 상태, 이것이 바로 프롬프트 엔지니어링 생태계를 위협하는 오라클 문제의 핵심이다.
