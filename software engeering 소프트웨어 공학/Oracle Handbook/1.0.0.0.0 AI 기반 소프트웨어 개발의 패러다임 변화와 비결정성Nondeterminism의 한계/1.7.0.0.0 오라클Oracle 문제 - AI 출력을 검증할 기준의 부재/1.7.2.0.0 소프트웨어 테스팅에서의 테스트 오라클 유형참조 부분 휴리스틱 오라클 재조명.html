1.7.2 소프트웨어 테스팅에서의 테스트 오라클 유형(참조, 부분, 휴리스틱 오라클) 재조명

정답이 부재한 끔찍한 오라클 문제를 타파하기 위해, 소프트웨어 공학자들은 시스템의 출력값이 완벽한지 증명하는 대신 “최소한 이 테두리 안에는 속해야 한다“는 차선의 검증막을 설계해 왔다. 기존 테스팅 철학에 뿌리를 둔 이 오라클의 종류들은 AI의 출력을 가두는 데에도 핵심적인 모티브를 제공한다.

1. 참조 오라클 (Reference Oracle / Pseudo-Oracle)

완벽하게 동일한 비즈니스 로직을 수행하는 기존의 잘 알려진 신뢰 시스템(Legacy System)을 정답지로 활용하는 방식이다.

  • 예시: C++로 작성된 기존의 결제 모듈 엔진과, 새롭게 AI가 Python으로 작성한 결제 모듈 엔진에 동일한 100만 개의 테스트 입력을 넣는다. 두 출력 결과가 소수점 끝까지 일치한다면, 기존 C++ 프로그램을 참조 오라클로 신뢰하는 것이다. 이 기법은 마이그레이션(Migration) 검증에서 매우 강력하다.

2. 부분 오라클 (Partial Oracle)

결과값 전체를 완벽하게 검사할 수는 없지만, 참(True)이 되기 위해 반드시 만족해야 하는 하위 조건(Post-conditions)이나 불변성(Invariants)을 검사하는 방식이다.

  • 예시: AI에게 특정 데이터베이스 조회를 위한 SQL을 짜라고 지시했을 때, 쿼리의 완전한 로직은 알 수 없더라도 최소한 DROP이나 DELETE 명령어 집합이 포함되어서는 안 된다고 검증하는 가이드라인 체계가 부분 오라클에 해당한다. (스키마 Validation이 주된 요소)

3. 휴리스틱 오라클 (Heuristic Oracle)

경험적 법칙이나 통계적 임계값에 의존하여 “이 결과가 합리적으로 타당한가(Reasonably valid)?“를 평가하는 체계이다. 주로 메타모픽 테스팅(Metamorphic Testing) 기법과 결합된다.

  • 예시: “주어진 이미지 속 객체를 분석하라“는 AI 파이프라인에서, 이미지를 좌우 반전(Flip) 시켜서 다시 넣어도 결과값이 95% 이상 유사하게 나와야 한다는 ’관계성 법칙’을 휴리스틱 오라클로 설정할 수 있다.

이 세 가지 과거의 오라클 체계 구조는, 제어할 수 없는 확률적 언어 모델을 기업 파이프라인 안에 포획하기 위한 든든한 설계 패턴이 된다.
