1.7.4 정답지(Ground Truth) 확보의 비용과 현실적 제약

오라클(Oracle)을 설계하기 위해 모든 엔지니어가 가장 먼저 떠올리는 이상적인 방식은, 시스템이 처리해야 할 수많은 케이스들에 대해 완벽한 인간 검증 정답지(Golden Dataset 혹은 Ground Truth)를 사전에 마련해두고, AI가 통과할 때마다 이 정답 데이터베이스와 대조해 보는 것이다. 그러나 이론적으로 완벽해 보이는 이 접근법은 실무에서 극심한 경제성 파탄에 직면한다.

1. 절대적 비용(Cost)의 폭주

특정 의료 문서 요약 AI를 검증하기 위해 데이터 1만 건에 대한 완벽한 정답지(요약본)를 준비해야 한다고 가정하자.

  • 데이터 전처리에 걸리는 시스템 비용
  • 도메인 전문가(의사 등 전문 지식인)를 고용하여 요약본을 수기로 작성 및 검수하는 압도적인 인건비(Human Captioning Cost)
  • 정답지가 편향되지(Biased) 않았는지 교차 검증하는 시간적 비용

이러한 정답 데이터를 구축하는 데 들어가는 비용이 사실상 모델을 운영하고 서비스를 런칭하여 얻는 잠재 수익을 아득히 초과하게 된다. 이는 대다수의 중소 규모 프로젝트나 애자일(Agile) 조직에서 ’완벽한 참조 오라클(Reference Oracle)’을 포기하게 만드는 가장 큰 진입장벽이다.

2. 정답지의 부패(Decay) 현상

더욱 심각한 것은 도메인 로직이나 현행법이 변경될 때마다 힘들게 구축해 둔 정답지 세트가 하룻밤 사이에 쓸모없는 휴지 조각(Outdated Data)으로 전락해버리는 현상이다. 2023년에 완벽하게 매핑해 둔 세법 정답지는 2024년 세법 개정안 앞에서는 모델에게 잘못된 학습을 강요하는 ’오답지’로 역변한다.

결국 방대한 절대 정답(Ground Truth)의 저수지를 구축하는 것은 막대한 예산을 지닌 초거대 테크 기업의 기초 AI 모델(Foundation Model) 학습 단계에서나 겨우 지속 가능할 뿐, 변화가 극심한 애플리케이션 개발 단계의 오라클 대응법으로는 부적합하다. 따라서 막대한 데이터셋 라벨링에 집착하기보다는, 런타임(Runtime) 환경에서 지켜내야 할 ’명제적 규칙(Propositional Rules)’과 ‘결정론적 문법 검사기(Syntax/Schema Validator)’ 형태의 유연한 제약형 오라클 체재로 넘어가야 할 당위성이 성립된다.
