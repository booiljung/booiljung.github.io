4.2.5 Max Tokens 설정을 통한 응답 길이의 물리적 제한과 일관성

Max Tokens는 모델이 지식을 어디까지 동원할지 제어하지는 못하지만, API 과금을 제어하고 할루시네이션의 꼬리를 강제로 잘라내는 최종 수비수다.

1. “짧게 답하라“의 한계와 Max Tokens의 강제력

프롬프트 상단에 “단답형으로만 말해“라고 백 번 지시하는 것보다, max_tokens: 10 이라는 물리적 파라미터 세팅 한 번이 훨씬 강력한 통제력을 발휘한다.

  • Yes/No(True/False) 오라클:
    주어진 코드가 시큐리티 룰을 어겼는가? 단순히 “PASS” 또는 “FAIL“만을 뱉어내는 1글자짜리 오라클이라면, max_tokens를 과감히 3에서 5 정도로 조여버린다. (토크나이저마다 변환 길이가 다르므로 버퍼를 약간 둔다).
  • 논리적 지연의 차단:
    토큰 수를 극단적으로 제한하면 모델은 불필요한 서론(“분석 결과 확인해 드리겠습니다”)을 스스로 싹둑 자르고 결론 토큰부터 앞으로 전진 배치시키는 압박을 받는다. 이는 응답 형식의 일관성을 크게 끌어올리는 역결과를 낳는다.

단, 이 기법은 모델 내부의 CoT(추론 과정) 기능이 필요한 구조(예: OpenAI의 o1이나 복잡한 판별 로직)에서는 오히려 논리의 경로를 조기 절단하여 엉뚱한 값으로 수렴하게 만들 위험이 있으므로, 단순 라우팅 및 1차원 필터 판독용 오라클에만 제한적으로 사용해야 한다.
