4.7.1 사용자 입력의 공백, 특수문자 처리가 모델 응답에 미치는 나비 효과

BPE(Byte-Pair Encoding)나 Tiktoken과 같은 LLM 토크나이저(Tokenizer)는 띄어쓰기에 극도로 예민하다.

예를 들어 "Test"와 " Test"(앞에 공백 포함)는 각각 전혀 다른 고유 번호(Token ID)로 매핑된다. 만약 자동화 스크립트가 파싱한 로그 파일 끝에 후행 공백(Trailing Whitespace)이 지저분하게 묻어서 LLM에 전달된다면, 모델은 이 의미 없는 공백 토큰까지 문맥의 일부로 해석하느라 자원을 낭비하고 쓸데없는 접속사를 생성하며 응답 궤적을 이탈하게 된다.

방어 전략:

  • 텍스트 주입 전 항상 trim() / strip() 을 통해 전후방 공백을 날려라.
  • 이모지나 의미를 알 수 없는 유니코드(제어 문자, BOM 등)는 정규식을 통해 모두 안전한 ASCII 텍스트로 치환하거나 필터링(Sanitize) 하라.
