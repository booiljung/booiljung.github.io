4.1.2 소프트웨어 엔지니어링에서의 ’멱등성(Idempotency)’과 AI 응답의 가변성

’멱등성(Idempotency)’은 현대 소프트웨어, 특히 분산 시스템과 RESTful API 설계에서 가장 중요한 원리로 꼽힌다. 동일한 입력(Input)으로 여러 번 연산을 수행하더라도, 그 결과(Output)와 시스템의 상태가 첫 번째 연산과 동일하게 유지되는 성질을 말한다. 수학적으로 표현하면 f(f(x)) = f(x)가 성립하는 것이다.

1. 멱등성이 파괴된 AI API의 실상

고전적인 결제 API를 생각해 보라. 고객이 중복해서 ‘결제’ 버튼을 10번 누르더라도, 시스템은 멱등성을 보장하여 단 1번만 결제되도록 통제한다. 즉, POST /pay는 입력값 재현에 대해 예측이 가능하다.

그러나 LLM의 API는 이 근본적인 룰을 무시한다.
동일한 프롬프트(Prompt)로 POST /v1/chat/completions를 10번 호출하면, AI는 10번 모두 미세하게, 또는 완전히 다른 텍스트를 반환할 수 있다.

// 호출 1의 결과
{"role": "assistant", "content": "네, 그 접근 방식이 맞습니다."}

// 호출 2의 결과 (동일 프롬프트)
{"role": "assistant", "content": "그렇게 할 수도 있지만, 보안상 위험할 수 있습니다."}

2. 가변성이 초래하는 아키텍처의 위기

AI 응답의 가변성(Variability)은 다음과 같은 연쇄적인 시스템 붕괴를 초래한다.

  1. 캐싱(Caching) 전략 무력화: 동일 입력에 대해 다른 결과가 나오므로, 전통적인 Key-Value 기반의 캐시 시스템을 프롬프트 레벨에 단순 적용하기 어렵다.
  2. 트랜잭션(Transaction) 동기화 실패: AI의 응답을 파싱하여 다음 마이크로서비스로 넘기는 파이프라인에서, JSON 구조가 조금만 틀어지거나 단어가 바뀌어도 후속 로직(예: DB Insert)이 예외(Exception)를 뱉으며 중단된다.
  3. 사용자 경험(UX) 훼손: 새로고침을 할 때마다 챗봇의 대답과 스탠스(Stance)가 뒤바뀌면, 사용자는 해당 AI 인프라가 미성숙하다고 판단하여 즉각적인 불신을 갖게 된다.
