4.1.3 비결정적 응답이 자동화 테스트 및 오라클 구축에 미치는 악영향

테스트 주도 개발(TDD)과 CI/CD 파이프라인의 핵심은 ’자동화된 기계적 검증’에 있다. assert(expected == actual) 이라는 하나의 확고한 등식을 중심으로 전체 품질 관리 시스템이 돌아간다.

그러나 AI가 반환하는 응답이 비결정적(Nondeterministic)이 되면, 이 테스트 파이프라인은 일대 혼란(Chaos)에 빠지게 된다.

1. Flaky Test(불안정한 테스트)의 양산

소프트웨어 품질관리(QA)에서 가장 기피 대상 1호는 돌릴 때마다 성공(Pass)과 실패(Fail)가 무작위로 뒤바뀌는 플래키 테스트(Flaky Test) 이다.

  • 정답 문자열의 변형: 정답지가 "Warning"을 기대하고 있는데, AI가 어느 날은 "Warning", 어떤 날은 "Warnings", 또 어느 날은 "Alert"라고 대답한다면 어떻게 될까? 단순한 단위 테스트(Unit Test)를 돌릴 때마다 파이프라인은 붉은색 실패 경고를 뿜어댈 것이다.
  • 오탐(False Positive)과 미탐(False Negative)의 지옥: 테스트 파이프라인이 계속해서 무작위로 깨지면 개발자들은 결국 알람을 무시하는 ’양치기 소년 효과(Cry Wolf Effect)’에 빠진다. 진짜 코드가 고장나서 생기는 심각한 오류와, 단순히 AI 텍스트 생성의 표면적 변형으로 인한 오류를 구분하지 않은 채 모든 경고를 패스시켜버리는 참사가 발생한다.

2. 검증 불가능한 오라클(Unverifiable Oracle)

이 책의 핵심인 오라클(Oracle)은 AI 또는 알고리즘의 결괏값을 판정하는 ’절대적인 잣대’다. 그런데 이 판정의 기준선 자체가 바람 앞의 갈대처럼 흔들린다면 오라클은 존재 가치를 잃는다.
“어제는 버그라고 하더니, 오늘은 버그가 아니라고 하는” 판정 시스템은 차라리 없는 것만 못하다. 따라서 오라클을 세우기 위해 가장 선결되어야 할 과제는, AI 모델 내부의 주사위(확률 연산)를 강제로 멈추게 만들거나, 그 주사위가 던져지는 반경을 극도로 좁히는 ’제어 기법’을 획득하는 것이다.
