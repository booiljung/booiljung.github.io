4.12.1 프롬프트 엔지니어링의 한계와 프롬프트 프로그래밍(Prompt Programming)의 부상

프롬프트 엔지니어링은 기본적으로 “눈먼 자들의 코딩“과 같다. 개발자가 “너는 훌륭한 전문가야“라고 쓰든, “엄격히 판단해라“라고 쓰든, 모델의 신경망 레이어가 그 텍스트를 정확히 어떤 가중치로 해독했는지 증명할 방법이 없다.

또한 GPT-3.5에 최적화하여 3개월에 걸쳐 깎아 만든 프롬프트가 LLaMA-3 모델에서는 전혀 다르게 작동하여 에러를 뿜어낸다. 이를 유지보수하기 위해 또다시 단어를 끼워 맞추는 노가다를 한다.

이러한 수작업(Manual) 프롬프트 튜닝의 시대가 끝나고, 인풋(Input)과 아웃풋(Output)의 인터페이스(Signature)만 선언(Declarative)해주면 프레임워크(컴파일러)가 모델에 맞춰 최적의 ’시스템 프롬프트와 퓨샷 예제’를 자동으로 찾아 기록해 주는 프롬프트 프로그래밍(Prompt Programming) 의 시대가 도래했다. 이는 어셈블리어(수작업 프롬프트)에서 C 언어(DSPy)로 넘어가는 프로그래밍 패러다임의 혁명과 궤를 같이한다.
