Chapter 30. 자동 비행 모드(Mission, Hold, Return) 소스 코드 분석

무인항공기(UAS)의 자율 비행 시스템에서 자동 비행 모드(Auto Flight Mode)는 조종사의 개입 없이 시스템이 독립적으로 경로를 계획하고 비행을 수행하는 핵심 기능이다. PX4-Autopilot은 이 자동 비행 모드를 구현하기 위해 고도화된 상태 머신(State Machine)과 비동기식 이벤트 구동(Event-driven) 아키텍처를 채택하고 있다. 본 장에서는 자동 비행 모드의 대표적인 세 가지 유형인 임무(Mission), 정지 비행(Hold), 복귀(Return/RTL) 모드의 소스 코드 구현체계와 내부 제어 논리를 심층적으로 분석한다.

1. 자동 비행 모드(Auto Mode)의 개념과 특징

PX4-Autopilot의 자동 비행 모드는 멀티로터, 고정익, VTOL 등 다양한 위치역학적(Kinematic) 특성을 가진 기체 플랫폼에서 동일하게 동작하도록 고도로 추상화된 제어 인터페이스를 제공한다. 자동 비행 모드가 활성화되면, PX4의 navigator 모듈이 경로점(Waypoint) 중심의 궤적을 3차원 공간 상의 설정점(Setpoint)으로 변환하여 하위 제어기(Position Controller)로 비동기 송출한다.

  • 임무(Mission) 모드: 지상 관제 시스템(GCS) 또는 온보드(Onboard) 컴퓨터를 통해 비휘발성 저장소(SD 카드)에 미리 입력된 경로점(Waypoint) 배열을 순차적으로 탐색하고 비행하는 모드이다. 단순히 다음 위치로 이동하는 것뿐만 아니라, 특정 속도나 고도에 도달하기, 서보(Servo) 모터를 트리거하여 화물 투하하기 등 복합적이고 스크립트화된 임무를 수행할 수 있다.
  • 정지 비행(Hold) 모드: 비행 중 예기치 못한 비상 상황이 발생하거나 조종사가 즉각적인 통제권을 잠시 유보할 때, 현재의 3차원 공간 좌표(GPS 기반 LLA 좌표계)에 기체를 완전히 고정시키는 모드이다. Ardupilot의 Loiter 모드와 유사한 목적을 가지나, PX4의 궤적 제동(Braking) 및 호버링 유지 계산 수식에는 바람 외란(Wind Disturbance)을 상쇄하는 EKF2(Extended Kalman Filter 2) 혁신(Innovation) 데이터가 깊이 관여한다.
  • 복귀(Return) 모드: 레거시 관행상 RTL(Return To Launch)로도 널리 불리며, 기체가 시스템에 기록된 이륙 위치(Home position) 또는 사전에 지정된 복귀 지점(Rally point)로 자율 비행하여 안전하게 착륙하는 방어적 비행 모드이다. 조종기 통신 단절(Data Link Loss) 및 배터리 임계치 도달(Low Battery) 시 시스템 백업 매커니즘인 Failsafe의 주축으로 작동한다.

2. Ardupilot 대비 자동 비행 아키텍처의 설계 대조

오픈소스 비행 제어기 생태계를 양분하고 있는 PX4와 Ardupilot은 동일한 자동 비행 목표를 성취하기 위해 완전히 상반된 소프트웨어 설계 철학을 준수한다.

  1. 상태 관리 및 프로세스 모듈화 구조:
  • PX4-Autopilot: 철저히 POSIX 표준 호환성을 지향하는 NuttX RTOS 환경 위에서 독립된 태스크(Task) 혹은 워크 큐(Work Queue) 스레드 형태로 모듈들이 분리 실행된다. 자동 모드 계산은 navigator라는 격리된 백그라운드 데몬(Daemon)이 전담하며, 이 프로세스가 uORB (Micro Object Request Broker) 메시지를 통해 commander(상태 제어), mc_pos_control(멀티로터 위치 제어), fw_pos_control_l1(고정익 위치 제어) 등의 외부 계층과 철저히 비동기적으로 통신한다.
  • Ardupilot: 과거 마이크로컨트롤러 자원이 극도로 제한적이었던 시기의 유산으로, 거대한 메인 폴링 루프(Main Scheduler Loop) 내에서 함수 테이블 포인터를 호출하여 각 모드의 run() 함수를 동기적으로 실행하는 모놀리식(Monolithic) 접근 방식을 취한다.
  1. 궤적 생성(Trajectory Generation) 인터페이스 방식:
  • PX4: 급격한 가속도 변화(Jerk)를 최소화하기 위해 C2(Continuous Acceleration) 수준의 궤적 연속성 보장을 목표로 한다. 이를 위해 설정점 삼중체(Setpoint Triplet: position_setpoint_triplet_s) 토픽을 정의한다. 이 토픽은 한 번에 ‘이전 경로점(Previous)’, ‘현재 향해가는 경로점(Current)’, ’그 다음 경로점(Next)’에 대한 3개의 컨텍스트를 하위 제어기 코어에 배열 형태로 동시에 넘겨주어 코너リング 시 기하학적으로 최적화된 감속/가속(Cornering Smoothing) 궤적을 연산하도록 유도한다.
  • Ardupilot: 웨이포인트 간의 L1 가이던스(L1 Guidance) 또는 구조화된 스플라인(Spline) 네비게이션 방식을 사용하며, 상대적으로 설정점 데이터 구조가 선형적이고 제어 계층 간 커플링(Coupling)이 강하게 결합되어 있다.

3. 시스템 아키텍처 및 내부 데이터 파이프라인

PX4의 모드 제어 시스템은 QGroundControl부터 델타 PWM 시그널 출력단에 이르는 여러 계층의 인터페이스를 거치도록 고안되어 있다. 데이터 송수신 흐름은 철저히 출처(Source)와 목적지(Destination)를 분리하는 발행-구독(Publish-Subscribe) 패턴으로 통일된다.

    A["QGroundControl <br/> (GCS 관제 시스템)"] -->|MAVLink Protocol <br/> MISSION_ITEM_INT| B[mavlink_receiver <br/> 비동기 파싱 스레드]
    B -->|uORB: vehicle_command| C["commander 모듈 <br/> (마스터 상태 머신)"]
    C -->|명령 인가 대조 <br/> vehicle_status 변경| D["navigator 모듈 <br/> (Mission, Hold, RTL 상태 계산)"]
    D -->|"임무 아이템 페치 (Dataman)"| E["(SD Card / RAM <br/> Mission Storage / LRU Cache)"]
    D -->|uORB: position_setpoint_triplet| F["위치 제어기 모듈 <br/> (mc_pos_control / fw_pos_control)"]
    F -->|uORB: vehicle_attitude_setpoint| G["내부 자세 제어 루프 <br/> (PID Rate Controller)"]
    G -->|uORB: actuator_controls_0| H[Mixer & Output 모듈 <br/> 액추에이터 구동]

위의 아키텍처 도표에서 알 수 있듯, navigator 모듈은 GCS로부터 수신한 웨이포인트를 해석하여 추상적인 위치 목표(position_setpoint_triplet)를 방출하는 역할에 집중한다. 실제 모터의 출력을 어떻게 물리적으로 분배하여 해당 좌표까지 이동할 것인지는 완벽히 하위 제어 모듈(mc_pos_control)에 의존성이 역전(Inverted Dependency)된 형태로 위임된다.

4. 임무(Mission) 설정 및 MAVLink 트랜잭션의 이해

자동 비행 모드는 필연적으로 지상 관제 시스템(GCS)과의 안정적인 원격 통신 데이터베이스 동기화를 전제로 한다. QGroundControl(QGC)과 작동할 때, MAVLink 통신 마이크로 서비스 모델은 무선 통신의 패킷 손실(Packet Loss)을 감수하고서라도 데이터 무결성(Integrity)을 보장해야 한다.

4.1 MAVLink 기반 프로토콜 협상 과정

임무 편집기에서 조종사가 웨이포인트를 업로드할 때, 기체와 GCS 간에는 엄격한 핸드셰이킹(Handshaking)이 발생한다. MAVLink의 MISSION_COUNT를 먼저 전송하여 항목의 총개수를 알리고, 기체가 이를 인지하여 MISSION_REQUEST_INT로 특정 인덱스를 역으로 요청하면, GCS가 최종적으로 개별 MISSION_ITEM_INT 구조체를 전송하는 방식이다. 이 프로토콜은 통신 링크가 상대적으로 불안정한 텔레메트리 환경 900MHz/433MHz 대역망에서의 생존성을 극대화하는 소프트웨어 공학적 설계이다.

4.2 주요 제어가 파라미터(Configuration Parameters) 구성

기체의 비행 성능 및 특임무 한계 포락선(Flight Envelope)을 정의하는 파라미터는 param 시스템을 통해 비휘발성 저장소에 기록된다. 자동 모드 동작 시 필수적으로 검토해야 될 파라미터는 다음과 같다.

  • NAV_ACC_RAD: 경로점 수용 구역 반경(Acceptance Radius). 기체가 특정 항로 좌표점에 논리적으로 도달했음을 선언하는 구역(Sphere)의 크기를(미터 단위) 지정한다. 지나치게 좁으면 기체가 경로점을 맴도는 현상(Orbiting)이 발생할 수 있다.
  • RTL_RETURN_ALT: 통신이 두절되거나 복귀(Return) 명령이 하달되었을 때, 기체가 위치를 고정하고 제일 먼저 상승해야 할 ’안전 보장 최소 고도’를 명시한다. 주로 경로상의 나무나 철탑 등 장애물 충돌 회피를 위한 기초적인 세이프가드로 기능한다.
  • MIS_TAKEOFF_ALT: 임무 스크립트 상점(1번 웨이포인트)에 이륙(Takeoff) 명령이 포함되어 있을 때, 강제적으로 지면으로부터 떠올라 보장해야 할 최소 안전 고도를 정의한다.

5. 자율 비행의 확장: ROS2와 uXRCE-DDS 오프보드(Offboard) 연동

전통적인 Mission, Hold 모드는 사전에 GCS를 통해 주입된 정적(Static) 데이터를 추종하는 수동적 자율 비행 시스템이다. 오늘날 최첨단 항공 로봇 공학 서비스(예: 컴퓨터 비전을 응용한 실시간 군집 비행, 즉석 추적(Tracking))에서는 연산 자원이 풍부한 동반 컴퓨터(Companion Computer, 예: Raspberry Pi, NVIDIA Jetson)와의 긴밀한 파트너십을 요구한다.

이러한 로컬 인공지능 기반의 동적 궤적 생성 모델을 수용하기 위해, PX4는 Offboard라는 고차원적 외부 자동 모드를 지원한다. 내부적으로는 Micro XRCE-DDS (uXRCE-DDS) 미들웨어 에이전트가 브리지(Bridge) 역할을 수행하여, ROS2 네트워크 생태계 상의 퍼블리싱 데이터를 PX4 내부망의 uORB 토픽(예: trajectory_setpoint_s)으로 심리스(Seamless)하게 변환시킨다.

이를 통하여, PX4 메인 컨트롤러는 외부 ROS2 노드들이 생성한 회피 공간 역학 연산(Obstacle Avoidance)과 VIO(Visual Inertial Odometry) SLAM 알고리즘 추정치를 아무런 코드 수정 없이 실시간 설정점(Setpoint)으로 취급할 수 있게 되며, 자동 비행 모드는 완전한 로봇 공학 모듈 중심 아키텍처로 탈바꿈하게 된다.
