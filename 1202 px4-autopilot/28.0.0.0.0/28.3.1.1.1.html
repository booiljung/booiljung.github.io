28.3.1.1.1. activate(): Task 전환 시 초기 상태 동기화 및 적분기(Integrator) 리셋 로직

조종사가 매뉴얼 모드에서 오토 미션 모드로 토글 스위치를 튕기는 ’모드 전환(Mode Transition)’의 찰나, 드론 내부에서는 거대한 소프트웨어 객체의 교대식이 벌어진다. 이 교대식의 첫 번째 관문이자, 새로운 비행 객체가 시스템의 통제권을 쥐기 위해 반드시 통과해야 하는 통과 의례가 바로 activate() 가상 함수다.

새롭게 메모리(RAM)에 로드된 자식 FlightTask 객체는 activate() 함수를 통해 자신의 초기 상태(Initial State)를 현재 기체의 물리적 현실과 동기화(Synchronization)하고, 과거의 잔재인 적분기 에러를 청소하는 막중한 책임을 진다.

1. 초기 상태 동기화 (Initial State Synchronization): 충격 없는 바톤 터치 (Bumpless Transfer)

모드 전환 시 가장 위험한 순간은 ’목표 궤적(Setpoint)’의 불연속성이다.
예를 들어, 조종사가 수동으로 초당 10m로 맹렬하게 전진하던 중 스틱을 놓고 고도 유지(Altitude Hold) 모드로 전환했다고 가정하자. 만약 새로 켜진 고도 유지 객체가 가상 공간의 목표 속도(Setpoint Velocity)를 갑자기 0 m/s로 설정해 버리면, 기체는 허공에 가상의 벽을 들이받은 것처럼 모터를 미친 듯이 역회전시켜 급브레이크를 밟게 되고, 이는 심각한 기체 불안정이나 추락으로 이어진다.

이를 방지하기 위해 activate() 함수는 ‘범플리스 트랜스퍼(Bumpless Transfer: 충격 없는 제어권 전환)’ 철학을 구현한다.

  • 현재 상태의 캡처 및 Setpoint 치환:
    activate() 루틴이 켜지면, 센서가 읽고 있는 ’현재 기체의 실제 속도와 위치(Ground Truth)’를 복사하여, 자신이 산출해야 할 첫 번째 루프의 ‘목표 궤적(Setpoint)’ 초기값으로 그대로 덮어씌운다(Overwriting).
  • 관성의 존중:
    결과적으로 새로 켜진 태스크는 기체가 현재 가지고 있는 관성(초당 10m 전진)을 자신의 첫 번째 목표로 삼게 되며, 이후 수십 밀리초에 걸쳐 부드럽게 감속(Deceleration) 궤적을 새로 그어나간다. 이 동기화 과정을 통해 제어기(Controller) 입장에서는 모드가 바뀌었는지조차 모를 정도로 부드러운 위치/속도 연속성이 보장된다.

2. 적분기 리셋 (Integrator Reset): 과거의 망령 지우기

PID 제어기, 특히 적분기(Integrator, I-term) 항은 과거의 오차를 시간의 흐름에 따라 누적하여 현재의 출력을 교정하는 기억 장치(Memory) 역할을 한다.
만약 이전 비행 모드에서 거센 측풍을 이겨내느라 롤(Roll) 축 적분기에 거대한 오차 보상값이 +500만큼 쌓여있었다고 가정해 보자.

  1. 초기화의 부재 시 참사: 모드가 전환되었는데 새로 램에 올라온 태스크가 이 +500이라는 적분기 쓰레기 값(Wind-up)을 그대로 물려받거나 0으로 명시적 청소를 해주지 않으면, 새 모드가 시작되자마자 모터는 이유 없이 기체를 확 뒤집어버리는 ‘점프(Jump)’ 현상을 일으킨다.
  2. reset_integrators()의 강제 호출:
    따라서 모든 FlightTask 자식 객체의 activate() 함수 내부에는 하위 PID 제어기나 속도 연산 파이프라인에 묻어있는 누적된 변수들을 0.0f로 새하얗게 날려버리는 reset() 로직이 필수적으로 포함되어 있다. 이는 “새 술은 새 부대에 담는다“는 원칙 하에, 새로운 비행 임무가 오염되지 않은 클린 룸(Clean Room)에서 궤적 연산을 시작할 수 있도록 보장하는 소프트웨어적 살균 작업이다.

3. 조건부 활성화 거부권 (Conditional Activation Rejection)

가장 우아한 객체 지향적 설계 중 하나는 이 activate() 함수의 리턴 타입이 bool이라는 점이다.
새로 태어난 임무 객체는 스스로 활성화 가능성을 판단하여 이의가 제기될 경우 **자기 파괴적인 거부권(Rejection)**을 행사할 수 있다.

bool FlightTaskAutoMission::activate() {
    // 1. 필요한 필수 하드웨어 센서나 조건이 충족되었는지 검증
    if (!_global_position_valid) {
        // 미션 모드인데 GPS가 튀어 글로벌 좌표를 모른다면?
        // 시작도 하지 않고 즉시 false를 뱉어 활성화 거부
        return false; 
    }
    
    // 2. 초기 상태 동기화 및 적분기 리셋
    _resetSetpointToCurrentState();
    _resetIntegrators();
    
    // 3. 정상 부팅 완료 승인
    return true;
}

앞선 ‘상태 불일치(State Mismatch)’ 장에서 설명했듯, 이 activate() 가상 함수가 내뱉는 false 반환값은 상위 FMM 모듈에게 던져지는 강력한 에러 예외(Exception) 처리 트리거 역할을 한다. FMM은 즉시 생성 중이던 이 객체를 포기(delete)하고 생존 가능한 백업 모드(Fallback Mode) 객체를 뽑아냄으로써, 데이터가 없는 텅 빈 상태로 제어 연산 루프(update)가 돌기 시작하는 재앙의 시발점 자체를 원천 봉쇄해 버린다.
