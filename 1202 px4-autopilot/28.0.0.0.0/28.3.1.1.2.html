28.3.1.1.2. updateInitialize(): 매 루프 센서/사용자 입력 데이터의 내부 버퍼 갱신 및 전처리

PX4 아키텍처에서 비행 모드 객체(FlightTask 파생 클래스)의 메인 궤적 산출 함수인 update()가 수학적으로 순결한 ’수술실’이라면, updateInitialize()는 그 수술실에 들어가기 전 환자의 혈압을 재고 차트를 정리하는 ’수술 전 처치실(Pre-op)’에 비유할 수 있다.

상부 모듈인 FMM(Flight Mode Manager)은 팩토리에서 찍어낸 현재 FlightTask 객체 포인터의 updateInitialize() 가상 함수를 제어 주기(예: 400Hz, 매 2.5ms)마다 어김없이 가장 먼저 호출한다. 이 함수는 외부 미들웨어(uORB)와 내부 수학(Math) 알고리즘 사이의 거친 데이터를 매끄럽게 정제하는 전처리(Preprocessing) 필터 역할을 전담한다.

1. uORB 데이터의 일괄 수확 (Batch Harvesting) 및 동기화

수술(제어 계산) 도중에 간호사(OS 스케줄러)가 불쑥 들어와 환자 차트(센서 데이터)를 갈아치우면 수술에 치명적인 에러(Race Condition)가 발생한다.
과거 레거시 코드들에서는 계산 중간중간에 생각날 때마다 센서 데이터를 읽어오는 난잡함을 보였으나, FlightTask 프레임워크는 이 난잡함을 updateInitialize() 단일 블록으로 격리시켰다.

  • 동기화된 스냅샷(Synchronized Snapshot):
    부모 클래스 FlightTask 단에 구현된 updateInitialize()는 호출되자마자 위치 추정기(EKF2), 배터리, 조종기 조작량, 비행 모드 변경 플래그 등 관련된 모든 uORB 토픽들의 캐시(Cache) 버퍼를 순식간에 읽어들인다(updated() 및 copy() 호출).
  • 원자적(Atomic) 데이터 봉인:
    이렇게 한 큐에 긁어온 데이터들은 구조체 변수에 고스란히 저장되며, 이후 진행될 update() 수식 계산이 끝날 때까지 0.0025초 동안 절대 갱신되지 않고 그 시점을 스냅샷처럼 유지한다. 덕분에 궤적 X축을 계산할 때와 Z축을 계산할 때의 타임스탬프가 서로 틀어지는 끔찍한 위상 지연 에러를 아키텍처적으로 근절한다.

2. ’물리적 불변량’의 치환 및 델타 타임(dt) 정제가공

데이터를 읽어오는 것만으로 전처리가 끝나는 것은 아니다. FlightTask 기저 클래스는 자식 클래스들이 수학 연산에만 전념할 수 있도록 계산이 귀찮은 기계적(Mechanical) 변수들을 먼저 정제하여 밥상을 차려준다.

  • 정밀 델타 타임(_deltatime) 산출:
    앞서 다루었던 hrt_abstime 기반의 지터(Jitter) 보상 델타 타임 계산이 바로 이 함수 안에서 이루어진다. 현재 OS 절대 시간에서 직전 루프 시간을 빼어 float 타입의 초(Second) 단위 _deltatime 변수에 미리 박아둔다. 자식 클래스의 PID 계산식들은 그저 이 변수를 가져다 적분기에 곱하기만 하면 된다.
  • 고도 홈(Home) 기준계 편차 보정:
    기압계 센서 중심의 로우 레벨 데이터들은 고도 편차가 섞여 들어오기 쉽다. updateInitialize()는 EKF2에서 넘어온 위치 정보에 Home 지점의 고도 오프셋이나 지형(Terrain) 높낮이 데이터를 미리 가산 및 감산 치환하여, 자식 객체가 “지면으로부터 상대적으로 몇 미터인가?“에 대한 직관적인 Z축 높이 정보(_position(2))만 순수하게 취할 수 있도록 전처리 매핑을 수행한다.

3. 자식 클래스의 오버라이딩(Overriding)을 통한 확장

원칙적으로 updateInitialize()는 부모 클래스(FlightTask)가 공통 인프라를 처리해 주는 구역이지만, 자식 클래스(예: FlightTaskAutoMapper)가 특별한 추가 전처리가 필요할 경우 override 키워드를 통해 이 파이프라인에 자신의 로직을 소량 얹을 수 있다.

// 자식 클래스의 오버라이딩 예시
bool FlightTaskAutoMapper::updateInitialize() {
    // 1. 부모의 공통 전처리 루틴을 먼저 의무적으로 호출 (super() 개념)
    bool ret = FlightTask::updateInitialize();
    
    // 2. 자식 특유의 추가 미션 데이터(예: 카메라 트리거 간격) 읽어오기
    _sub_camera_trigger.update();
    
    return ret;
}

이처럼 자식 클래스는 상속 지향 설계의 다형성을 십분 활용하여, 부모가 닦아놓은 거대한 uORB 센서 스위핑(Sweeping) 고속도로 위에 자신만의 지선(Branch) 데이터를 부드럽게 합류시킬 수 있다. 이 철저한 분업이야말로 비행 제어 알고리즘 프로그래머가 OS나 미들웨어를 이해하지 못해도 복잡한 수식을 안전하게 전개할 수 있게 만들어주는 FlightTask 추상화 프레임워크의 가장 아름다운 가치 중 하나다.
