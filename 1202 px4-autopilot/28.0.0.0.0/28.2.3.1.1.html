28.2.3.1.1. vehicle_local_position 토픽 업데이트 인터럽트에 의한 루프 깨우기(Wake-up) 로직

PX4 비행 제어 스택에서 가장 핵심적인 진자의 추 역할을 하는 Flight Mode Manager (이하 FMM)는, 앞서 설명한 대로 자신만의 고집스러운 while(1) 타이머 슬립 루프를 돌지 않는다.
대신 FMM은 철저하게 **“내가 지금 우주 공간 어디에 위치해 있는지 알고 나서야, 다음 목표로 어디로 갈지 계산할 수 있다”**는 지극히 상식적이고 수동적인(Reactive) 철학을 따른다. 바로 이 철학명제를 코드 레벨 시스템 스레드 스케줄링으로 구현해 낸 핵심 트리거(Trigger)가 vehicle_local_position 토픽 업데이트 구독 인터럽트다.

1. Primary 스케줄링 트리거의 록인(Lock-in) 설정

FMM 메인 부모 클래스가 초기화 데몬 런칭(init()) 단계를 거쳐 운영체제 커널의 nav_and_controllers 워크 큐에 올라타기 직전, FlightModeManager.cpp 초기 셋업 소스 코드에서는 스케줄러를 깨울 유일한 ’오디오 알람’을 커널에 등록하는 매우 치명적인 1줄의 C++ 코드가 실행된다.

// FMM을 기상시킬 유일한 프라이머리 알림 소스를 커널 스케줄러에 등록
_vehicle_local_position_sub.registerCallback();

이 registerCallback() 인터페이스 함수 호출은, 하위 uORB::SubscriptionCallbackCallbackWorkItem 기반의 특수 구독자 객체가 OS 미들웨어 계층(uORB Node)을 향해 쏘아 올리는 하드웨어 인터럽트 연동 신청서와 같다.

이 선언을 통해 FMM은 수많은 구독 토픽들(vehicle_status, battery_status 등) 중에서, 오직 단 하나, “EKF2(칼만 필터)가 3차원 위치 추정을 끝마치고 vehicle_local_position 데이터를 갱신(Publish)했을 때만 내 Run() 메인 함수를 큐에서 깨워줘!” 라고 워크 큐 스케줄러에게 절대적인 편애성 편파 트리거를 못 박아 설정(Lock-in)하게 된다.

2. EKF2 (Estimator) 주도형 트리거 체인(Cascade Trigger Chain)의 지배

왜 FMM은 다른 귀중한 토픽들(예: 조종기 스틱 입력)을 무시하고 하필이면 위치 데이터(vehicle_local_position) 업데이트를 자신의 기상(Wake-up) 메인 트리거로 삼았을까? 이는 현대 자율 비행 제어 아키텍처의 ‘상태 추정 주도형 파이프라인(Estimator-driven Pipeline)’ 사상 때문이다.

  1. 추정의 절대성: 드론이 아무리 똑똑한 회피 알고리즘(FlightTask)을 품고 있든, 조종사가 조종기를 미친 듯이 꺾어대든 간에, ’현재 내 기체 무게 중심이 땅에서 몇 미터 떠 있고 어느 방향으로 초당 몇 미터로 흘러가고 있는지’에 대한 물리적 상태 편행(Local Position)을 알지 못하면 그 어떤 다음 타겟 궤적 산출 수식도 휴지조각 수학에 불과하다.
  2. 동기화된 주파수 체인의 압수: 픽스호크 보드의 IMU 센서가 1000Hz로 가속도를 샘플링하면, EKF2 필터 데몬이 이를 게걸스럽게 받아먹어 융합한 뒤 약 400~200Hz 언저리의 주파수로 vehicle_local_position을 토해낸다. FMM이 바로 이 토픽의 갱신 시점을 인터럽트 기상 트리거로 잡음으로써, FMM 자신의 Run() 궤적 산출 루프 주파수 역시 자연스럽게 EKF2의 출력 심장 박동수에 1:1로 완벽하게 하드웨어 종속(Slave Sync) 정합되는 놀라운 톱니바퀴 동기화 효과를 거둔다.
  3. 최소 지연(Minimum Latency): EKF2가 계산 결과를 메모리에 써넣고 퇴장(Publish)하는 그 마이크로초 단위의 찰나에, 미들웨어가 즉각 이벤트 콜백을 튕겨 FMM의 Run()을 깨워주므로, 과거의 낡은 위치 데이터를 바탕으로 엉뚱한 미래 궤적을 산출하는 치명적인 제어 위상 지연(Phase Lag) 에러를 시스템 아키텍처 레벨에서 근본적으로 억제한다.

3. 예외 상황 처리: 위치 데이터 유실 시의 자력 기상 (Fallback Timeout)

그렇다면 비전 센서나 GPS 박스가 공중에서 폭발해 끊어져서 EKF2가 더 이상 vehicle_local_position 토픽을 발행하지 못하고 침묵(Timeout)해 버리면 FMM 데몬 스레드는 영원히 기절(Starvation)하여 드론은 추락할 수밖에 없는 것일까?

이 절대적 단일 트리거 의존성에 의한 단일 장애점(Single Point of Failure) 아키텍처의 한계를 극복하기 위해, FMM Run() 루프의 마지막 끝자락 꼬리 부분에는 다음과 같은 최후의 보루(Fallback) 안전장치 코드가 박혀 있다.

// 이 시간(현재 약 50ms) 내에 콜백이 안 오면 OS가 강제로 내 함수를 호출하라 
ScheduleDelayed(BACKUP_SCHEDULE_TIMEOUT_US);

만약 폭풍우 치는 하늘에서 위치 추정기 EKF2가 죽어버려 vehicle_local_position 갱신 인터럽트가 평소 주기(2.5ms)를 한참 넘겨 50ms 동안 단 한 번도 울리지 않으면, NuttX 타임 스케줄러 모듈 위에서 똑딱대던 백업 와치독(Watchdog) 알람이 매정하게 터지며 FMM의 Run() 함수를 **시간 초과 강제 기상(Timeout Wake-up)**시켜 버린다.

강제로 눈을 뜬 FMM은 updated() API로 위치 데이터가 없음을 싸늘하게 확인하고는, 현재 비행 모드(FlightTask)를 위치 제어 불능에 대비한 최후의 고도 유지 혹은 강제 비상 착륙(Failsafe / Descend Task) 모드로 팩토리 스니핑(Sniffing) 전환하여, 데이터 체인이 끊어지는 최악의 참사 속에서도 기체를 최소한의 물리 법칙으로 맨땅에 안전하게 꽂아 살려내는 아키텍처적 불사조 로직을 완성해 낸다.
