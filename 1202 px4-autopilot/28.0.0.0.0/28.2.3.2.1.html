28.2.3.2.1. 센서 데이터 지연(Latency) 발생 시 제어기 발산 방지용 데이터 폐기 로직

자율 비행 제어 시스템 철학에서 ’느리게 도달한 정답’은 ’빠르게 도달한 오답’보다 치명적이다.
비전(Vision) 센서나 컴패니언 컴퓨터(Companion Computer)에서 처리되어 직렬 미들웨어(ROS2/MAVLink)를 타고 Flight Mode Manager (이하 FMM)로 느릿느릿 올라오는 외부 위치 추정 데이터들은 본질적으로 무거운 지연(Latency)을 내포하고 있다. FMM은 이런 지연된 과거의 유령 수치들이 제어 적분기(Integrator)를 오염시켜 기체를 추락(발산)시키는 것을 막기 위해, 고도의 시간축 타겟팅 데이터 폐기(Discarding) 아키텍처를 가동한다.

1. 늦은 데이터가 유발하는 제어기 발산(Divergence)의 수학적 악몽

드론이 바람에 밀려 오른쪽으로 30cm 벗어났다고 가정해 보자.
만약 비전 카메라 센서가 이 30cm 오차를 찍어서 연산하고, UART 선을 타고 FMM에 도달하기까지 0.2초(200ms)의 라우팅 지연(Latency)이 걸렸다면 어떻게 될까?

  1. 지연된 오차 인식: 0.2초 후, FMM은 “우측으로 30cm 밀렸다“는 보고를 받는다. FMM의 자식 객체(Position Controller)는 즉시 왼쪽으로 강하게 기울어 복구하라는 모터 가속도 명령(trajectory_setpoint)을 내린다.
  2. 물리적 복구와 과거의 망령: 기체는 왼쪽으로 이동하여 이미 정중앙(0cm)으로 성공적으로 복귀했다. 그러나 비전 센서의 데이터는 여전히 0.2초 전의 과거를 찍어 보내고 있으므로, 0.2초 동안 FMM은 “아직도 우측에 있네? 더 강하게 왼쪽으로 틀어!“라고 착각하며 적분기(I-term) 에러를 미친 듯이 누적(Wind-up)시킨다.
  3. 오버슈팅(Overshooting) 발산: 0.2초 뒤, 드론은 왼쪽으로 너무 과도하게 밀려가 왼쪽 -50cm 지점으로 튕겨 나간다(Overshoot). 이 진자 운동의 진폭은 지연 시간에 비례하여 계속 커지며, 결국 드론은 하늘에서 미친 듯이 좌우로 흔들리다가(Toilet-bowl Effect) 뒤집어져 물리적 발산(Divergence) 추락을 맞이하게 된다.

2. 지연 파라미터(Delay Parameter) 보상과 Time-travel 매칭

PX4의 FMM 및 하위 위치 제어 모듈들은 이런 끔찍한 오버슈팅 발산을 철학적으로 용납하지 않는다. 이를 방어하기 위해 외부 센서의 태생적 물리적 지연값을 수학적으로 보상하는 메커니즘을 둔다.

  • EKF2_EV_DELAY 파라미터 체계:
    PX4 설계자들은 사용자가 튜닝할 수 있는 외부 비전 센서 타임 딜레이(예: 175ms 지연) 파라미터를 제공한다.
  • 링 버퍼(Ring Buffer) 타임머신 비교:
    메인 상태 추정기(EKF2)와 FMM 제어기는 현재 흘러가는 기체의 찰나의 센서 상태(관성 항법 등 지연이 없는 값)를 즉각 갖다 쓰지 않고 링 버퍼(Ring Buffer)라는 타임머신 메모리 배열에 0.1밀리초 단위로 수백 개씩 차곡차곡 저장해 둔다. 외부 비전 센서 데이터가 175ms 늦게 도착하면, FMM 연계 제어 시스템은 “지금 현재 기체 상태“와 이 늦은 데이터를 비교하는 무식한 짓을 하지 않는다. 대신 링 버퍼를 과거로 175ms 거슬러 올라가, **“정확히 175ms 전의 내 관성항법 자세”**와 **“방금 도착한 175ms 전 과거를 찍은 비전 데이터”**를 타임라인 상에서 수평으로 완벽하게 도킹(Matching) 시켜버린다.

이 놀라운 시간여행(Time-travel) 매칭 기술 덕분에, 제어 적분기는 미래와 과거가 뒤섞인 쓰레기 오차(Error)를 먹지 않게 되고 제어기 발산을 수학적으로 근절한다.

3. 유효성 한계선(Validity Deadline)을 넘긴 데이터의 무자비한 폐기(Discarding) 로직

하지만 링 버퍼로 보상할 수 있는 지연에도 한계(수백 ms 수준의 버퍼 사이즈)가 있다. 만약 짐벌 카메라 CPU가 과부하가 걸려 위치 계산 데이터를 1.5초(1500ms)나 늦게 FMM 버스에 뒤늦게 토해냈다면?
PX4 설계자들은 이 정도로 썩어버린 데이터를 애써 살려내려 하지 않고 가차 없이 휴지통에 폐기(Drop)해버린다.

// 가상 코드화 한 타임아웃 킬 스위치 로직의 예
const uint64_t MAX_LATENCY_TOLERANCE_US = 500_000; // 0.5초 절대 방어선

if ((hrt_absolute_time() - msg_timestamp) > MAX_LATENCY_TOLERANCE_US) {
    // 이 데이터는 링 버퍼로 살려낼 수 없는 썩은 고기다. 즉시 버린다.
    _sensor_valid = false;
    discard_stale_data();
    trigger_failsafe_routine();
}

FMM은 이전 섹션에서 다룬 timestamp 뺄셈 로직을 더욱 엄격하게 적용하여, **제어기가 버틸 수 있는 최대 지연 한계선(Validity Deadline)**을 넘긴 데이터 프레임은 버퍼에서 아예 지워버린다.
“잘못된 정보로 절벽으로 달려가느니, 차라리 장님이 되어 강제로 제자리에 멈춰 서는 것(Failsafe Loiter / Land)이 기체를 살린다“는 PX4 자율제어 아키텍처의 냉혹하고도 아름다운 실용주의 생존 본능을 가장 잘 보여주는 로직이다.
