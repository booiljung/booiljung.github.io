28.3.1.1.3. update(): 순수 가상 함수(Pure Virtual)에 가까운 핵심 Setpoint 연산 인터페이스

앞선 activate()가 부팅(Booting)이고 updateInitialize()가 데이터 수집(Fetching)이었다면, update()는 FlightTask 프레임워크의 진정한 존재 계기이자 알고리즘의 꽃이 피어나는 **메인 제어 연산 루프(Main Control Loop)**이다.

상위 모듈인 Flight Mode Manager (이하 FMM)는 매 루프의 정점에서 이 가상 함수를 맹렬하게 호출하며, 자식 클래스(각각의 커스텀 비행 모드)는 오직 이 update() 블록 안에서 0.0025초 뒤의 드론의 미래 궤적을 산출해 내야만 한다.

1. 순수 연산식(Pure Computation)의 격리 공간 (Sandbox)

PX4 설계자 아키텍트들은 update() 함수 내부를 철저한 ’결정론적 수학(Deterministic Mathematics)’의 샌드박스로 격리하고자 노력했다.

  • I/O 프리(I/O Free): 잘 짜인 FlightTask 자식 클래스의 update() 본문에는 uORB로부터 직렬 버스를 폴링(Polling)하거나 타임스탬프를 비교하는 잡스러운 인프라 코드가 단 한 줄도 들어가선 안 된다. (이는 윗단 updateInitialize()가 전부 끝내놓았다).
  • 수학 모델링만 남은 세계: 개발자는 그저 매개변수로 주어진 조종기 스틱의 x, y, z 퍼센티지(역치) 값을 기체의 최대 설계 속도(m/s)로 선형 보간(Linear Interpolation)하고, 앞서 계산된 _deltatime(dt)을 곱하여 “다음 목표 위치(Setpoint)는 원래 위치보다 +5cm 전진한 곳이다“라는 순수한 위치/속도/가속도 벡터 연산에만 집중하게 된다.
    이러한 설계는 새로운 비행 모드를 개발하려는 제어공학 연구자가 굳이 복잡한 RTOS(Real-Time OS) 스레드 개념을 몰라도 선형대수학 지식만으로 C++ 코드를 작성할 수 있게 해주는 거대한 장벽(Abstraction Layer)이 된다.

2. 유일한 아웃풋(Output): _evaluateVehicleLocalPositionSetpoint() 의 강제

update() 함수의 심장부에서 수천 번의 행렬 곱셈이 일어난다 한들, 그 결과 값이 외부로 제대로 배출되지 않으면 드론의 모터는 단 1RPM도 회전하지 않는다.

최상위 기저 클래스 FlightTask는 자식 객체의 update()가 수학 연산을 마친 직후, 최종 결과물을 반드시 부모가 마련해 둔 표준 규격의 그릇(구조체)에 담아 제출하도록 강제한다.
그 결과물이 담기는 최종 종착지 그릇이 바로 클래스 내부의 멤버 변수인 _trajectory_setpoint 다.

  • 표준화된 산출물: 자식 클래스는 수동 비행을 하든, 라이다(LiDAR)로 장애물을 피하든 간에, 자신의 복잡한 연산 결과를 결국 _trajectory_setpoint의 삼차원 위치(X,Y,Z), 속도(Vx,Vy,Vz), 가속도(Ax,Ay,Az), ヨー 각도(Yaw)로 매핑해 넣어야 한다.
  • 리턴(Return)의 무거움: update() 연산이 성공적으로 Setpoint를 산출하여 구조체에 값을 채웠다면 true를 반환하고 루프를 빠져나간다. 반대로 연산 도중 매트릭스 역행렬 계산이 꼬여서 NaN(Not a Number)이 발생하거나 한계치(Limit)를 벗어나 산출이 불가능해지면, 즉시 false를 뱉어버린다.
    이 false 신호는 곧바로 상위 FMM에게 “이번 루프 계산 붕괴됨“이라는 하드웨어 예외(Exception)와 동일한 급의 경고를 날리며, 앞서 설명한 폴백(Fallback) 안전 상태로의 핫스와핑을 즉각 유발한다.

3. 발행(Publishing) 책임의 분리 (Single Responsibility Principle)

가장 놀라운 점은, 이토록 치열하게 궤적을 산출한 update() 함수가 끝날 때, 그 어디에도 orb_publish(ORB_ID(trajectory_setpoint), ...) 같은 결과 송출(Publish) 코드가 들어있지 않다는 사실이다.

  • 단일 책임 원칙 (SRP): PX4는 객체 지향 공학의 핵심인 단일 책임 원칙을 극한으로 지키고 있다. 자식 객체(FlightTask)의 임무는 ‘오직 궤적 수치를 채워 넣는 계산기’ 역할로서만 끝이 난다.
  • 부모의 몫(Publishing): 계산이 끝나고 제어권이 상부(FMM의 Run() 루프)로 반환되면, FMM 본체 데몬이 방금 자식 객체가 정갈하게 채워 넣은 _trajectory_setpoint 멤버 그릇의 포인터를 휙 채어간다. 그리고 FMM 데몬 본인이 직접 uORB 미들웨어 망을 통해 하위 자세 제어기(Attitude Controller)를 향해 토픽을 시원하게 퍼블리싱(Publishing)한다.

이러한 출력 책임의 분리를 통해, FlightTask 자식 객체는 언제든 마음대로 껐다 켤 수 있는 순수 수학 샌드박스로 남게 되며, 네트워크 발행(I/O)의 교착 상태(Deadlock)나 미들웨어 에러로부터 완벽히 자유로워지는 우아한 백엔드 구조를 뽐내게 된다.
