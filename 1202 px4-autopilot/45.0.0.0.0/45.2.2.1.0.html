45.2.2.1. 주 비행 모드(Main Mode) 및 보조 비행 모드(Sub Mode)의 32비트 부호 없는 정수형(Unsigned Integer) 공간 내 비트 할당(Bit Allocation) 밀도 최적화

MAVLink 프로토콜을 기반으로 운용되는 PX4-Autopilot의 통신 아키텍처에서, 비행 모드(Flight Mode)는 custom_mode라는 32비트 부호 없는 정수형(uint32_t) 공간에 할당된다. 단순 열거형(Enumeration) 방식을 취하는 Ardupilot과 달리, PX4는 이 32비트 데이터 공간을 고도로 최적화된 비트 할당(Bit Allocation) 밀도로 나누어, 주 비행 모드(Main Mode)와 보조 비행 모드(Sub Mode)를 논리적으로 분리하고 패킹(Packing)한다.

1. 32비트 공간 내 비트 할당 철학과 밀도 최적화

32비트(4바이트)의 공간은 이론적으로 2^{32} (약 42억) 개의 상태를 표현할 수 있다. 비행 모드의 개수는 수십 개에 불과하므로 단순히 1부터 50까지의 일련번호를 부여하는 것이 직관적일 수 있다. 그러나 다수의 드론을 동시에 관제하거나 임베디드 코어에서 고속 연산이 필요한 경우, 분기문(if-else 또는 switch-case)이 길어질수록 CPU의 파이프라인 분기 예측(Branch Prediction) 실패율이 증가한다.

PX4 펌웨어(기준 버전: v1.14.0)는 이를 해결하기 위해 총 32비트 중 16비트만을 제어 상태 관리에 할당하며, 이를 다시 8비트(1바이트) 단위로 완전히 정렬(Byte-Aligned)하여 주 비행 모드와 보조 비행 모드로 분리한다.

  • 주 비행 모드 (Main Mode, 8비트): Manual, Altitude Control, Position Control, Auto, Acro, Offboard 등 최상위 비행 제어기(Flight Controller)의 동작 범주를 결정한다. 최대 256개의 주 모드를 할당할 수 있다.
  • 보조 비행 모드 (Sub Mode, 8비트): 주 모드가 Auto일 때 세부적인 동작 시나리오를 정의한다. 예를 들어 Auto-Ready, Auto-Takeoff, Auto-Loiter, Auto-Mission, Auto-RTL(Return To Launch), Auto-Land 등이다. 이 역시 최대 256개의 확장을 보장한다.

이러한 8비트 경계(Boundary)에 맞춘 비트 단층화 설계는 논리 비트 밀도를 높이는 동시에, 마이크로컨트롤러 단에서 바이트 단위 레지스터 접근 속도를 극대화하는 밀도 최적화(Density Optimization) 전략이다. 하위 16비트는 예약 공간(Reserved)으로 두어 추후 기능 확장을 도모한다.

2. 모드 추출을 위한 C/C++ 논리 연산 최적화

QGroundControl(v4.3.0 이상)이나 MAVSDK를 통해 수신된 32비트 custom_mode 값에서 실제 비행 모드를 확인하려면 비트 마스크(Bitmask) 추출을 수행해야 한다. 8비트 단위로 딱 맞아 떨어지게 비트가 할당되어 있기 때문에, 컴파일러(Compiler) 수준에서 다음과 같이 단 2회의 비트 시프트(Bit Shift)와 논리곱(Bitwise AND)만으로 완벽하게 상태를 도출한다.

// 32비트 부호 없는 정수 변수 (예: 0x00040300)
uint32_t custom_mode_val = heartbeat.custom_mode;

// 비트 연산을 통한 주/보조 모드의 고속 추출 파이프라인
// 0xFF는 8비트 전체 1 (비트마스크)
uint8_t main_mode = (custom_mode_val >> 16) & 0xFF; // [23:16] 위치의 값을 추출
uint8_t sub_mode  = (custom_mode_val >> 24) & 0xFF; // [31:24] 위치의 값을 추출

이는 분기 조건문 없이 산술 논리 장치(ALU) 내 기본 명령어만 수행하므로 매우 속도가 빠르며, 메모리 연산 자원이 부족한 임베디드 관제 환경에서 특히 유리하다.

4. 자율 비행(ROS2) 시스템 구축을 위한 시사점

자율 에이전트 비행, 군집 비행(Swarm) 제어 등의 상위 명령 계층에서 ROS2 미들웨어를 통해 PX4를 관제할 때, 차량의 현재 상태(VehicleStatus uORB 토픽)를 기반으로 다음 행동을 결정하는 유한 상태 머신(Finite State Machine, FSM) 설계가 중요해진다.

PX4 펌웨어의 이러한 주 모드/보조 모드 분리 설계는 상태 머신의 트리(Tree) 구조를 자연스럽게 그려준다. 개발자는 상위 상태 머신에서 먼저 주 비행 모드(nav_state 범주 중 Main Level)의 전이를 확인하고, 주 모드가 Auto 범주에 들어왔을 때만 하위 모드(보조 비행 모드) 트리로 분기를 유도할 수 있다. 이로써 제어 로직 파이프라인이 간결해지며, 상태 천이(State Transition) 검증 프로세스에서의 논리적 무결성(Logical Integrity)을 확보할 수 있다.
