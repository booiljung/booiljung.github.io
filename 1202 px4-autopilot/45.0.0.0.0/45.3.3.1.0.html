45.3.3.1 부동소수점(IEEE 754 표준) 매개변수 공간을 활용한 이산 상태 지정(Discrete State Designation) 시의 가수부(Mantissa) 양자화(Quantization) 오차 회피 기법

COMMAND_LONG 체계 내에서 7개의 파라미터 영역은 모두 32비트 IEEE 754 단정밀도 부동소수점(Single-Precision Floating-Point) 형식(float)으로 전송된다. 연속적인 무빙 포인트를 표현하는 부동소수점은 본질적으로 정확한 정숫값(Integer Value) 표현에 한계를 가진다. 비행 제어기(PX4-Autopilot)가 이런 부동소수점 변수를 받아 비행 모드 번호, 카메라 트리거 개수 등과 같은 완벽한 이산 상태(Discrete State) 데이터로 사용할 때 발생하는 가수부 양자화 오차(Mantissa Quantization Error)의 원인과, 이를 회피하기 위한 PX4의 내부 설계 기법에 대해 상세히 분석한다.

(설명 기준: PX4-Autopilot v1.14.0)

1. 이산 상태 매핑 시의 가수부(Mantissa) 양자화 한계

IEEE 754 32비트 포맷은 부호(1비트), 지수(8비트), 가수(Mantissa/Significand, 23비트)로 나뉜다.
수학적 관점에서 23비트 가수가 손실 없이 정확히 1단위로 표현할 수 있는 최대 정수는 2^{24} - 1인 16,777,215 정밀도이다. 일상적인 10진수 비행 모드(예: 4번 모드, 5번 서브모드)를 다룰 때는 23비트의 가수부 정밀도로도 공간이 충분해 보일 수 있다.

그러나 문제는 **플랫폼 간 직렬화(Serialization)/역직렬화(Deserialization) 및 연산 과정에서 누적되는 라운딩 에러(Rounding Error)**에 있다.

  • 통상 GCS 환경인 Python(MAVSDK)이나 JavaScript 프론트엔드는 기본 수치형으로 64비트 배정밀도(Double-Precision, double)를 사용한다.
  • 이 배정밀도에서 단정밀도 지수-가수 폼으로 다운캐스팅(Down-casting)되어 RF 통신으로 전파를 타는 과정, 또는 시뮬레이터 런타임(예: SITL/Gazebo)을 거치며 4.0이라는 정수값이 PX4 수신부에 3.9999998이나 4.0000001로 변질되어 들어오는 양자화(Quantization) 현상이 흔하게 발생한다.

수신된 3.9999998을 C++의 일반적인 (int)param 방식으로 캐스팅(형변환)하면, 소수점 이하가 버림(Truncation) 처리되어 3이라는 전혀 다른 비행 모드로 오인식하는 치명적 교착 상태에 빠질 수 있다.

2. PX4-Autopilot의 양자화 오차 회피(Avoidance) 파이프라인

PX4-Autopilot은 위와 같은 부동소수점의 태생적 결함으로부터 이산 상태 지정을 강력하게 보호하기 위해(Robustness), 단순한 형변환(Casting)을 금지하고 철저한 안전망(Safeguard) 수식 트리를 거치도록 규정하고 있다.

2.1 -1. 수학적 반올림(Rounding to Nearest) 필연성 적용

PX4의 커맨더(commander_module) 또는 MAVLink 수신 레이어에서 이산 상태값(비행 모드 ID, 마운트 짐벌 각도 인덱스 등)을 파싱할 때는 <math.h> 또는 PX4 내부 mathlib 라이브러리의 roundf()를 사용하여 가장 가까운 정수로 이산화(Discretization)한다.

  • ❌ 잘못된 캐스팅 래퍼: int mode = (int)param2; (3.9998 -> 3으로 처리됨)
  • ✅ PX4 권장 매핑 방식: int mode = (int)roundf(param2); (3.9998 -> 4로 처리됨)

경우에 따라 프로세서 클록 소모를 억제하기 위해 int mode = (int)(param2 + 0.5f); 형태의 부동소수점 옵셋(Offset) 매크로 체계를 사용하는 코드베이스도 존재한다.

2.2 -2. 불리언(Boolean) 상태의 스레시홀드(Threshold) 필터링

특정 컴포넌트를 동작시키거나 멈추는(예: MAV_CMD_COMPONENT_ARM_DISARM의 Arming/Disarming) 불리언 동작의 파라미터는 이론적으로 1.0f와 0.0f로 수신되어야 한다. PX4는 임계값(Threshold)을 넓게 설정하여 잡음을 걸러낸다.
통상 0.5라는 가드 밴드(Guard Band)를 두어, param > 0.5f 수식을 참(True)으로 간주하고, 미만인 경우 거짓(False)으로 간주하도록 논리적으로 필터링한다. 이를 통해 양자화 오차로 인해 0.98f나 0.01f가 들어오더라도 정상적인 상태 머신 천이를 보장한다.

2.3 -3. NaN 검증 및 안전 폴백(Fallback) 방어

PX4는 파라미터를 다룰 때, 그 값이 사용되지 않는 Null-equivalent한 상태임을 입증하기 위해 NaN (Not a Number)인지를 항상 먼저 체크한다. (std::isnan(param))
만약 이산 상태로 파싱해야 할 파라미터가 유효한 숫자 범위(예: 비행 모드 Enum 범위 0~20)를 벗어나거나 NaN 또는 Infinity 일 경우, 명령 승인(Ack) 모듈은 즉결 거부(MAV_RESULT_DENIED) 메시지를 반환하여 미정의 상태(Undefined Behavior)로 빠지는 것을 철벽 방어한다.

3. 요약 및 관제 소프트웨어 개발 시 고려사항

PX4-Autopilot은 COMMAND_LONG의 부동소수점 가수부 한계에서 파생되는 데이터를 안전한 이산 상태 코드로 파싱하기 위해 방어적 프로그래밍(Defensive Programming) 기법인 반올림 래핑(Rounding Wrapping), 스레시홀드 필터링(Threshold Filtering), NaN 우선 검증 정책을 구사한다.

GCS(QGroundControl 및 MAVSDK 애플리케이션) 개발자의 경우, 기체로 보내는 명령 패킷을 생성할 때 정확히 배정밀도 프레임에서 단정밀도 float 셋업으로 변환 시 자릿수 초과 현상이 없는지를 검토해야 하며, 임의의 식별용 시드(Seed) 값과 같은 너무 큰 정수(예: 2^{24} 이상)를 param 배열에 태워서 보내는 안티 패턴(Anti-pattern)을 피해야 한다. 큰 정수나 해시(Hash) 식별자가 필요한 프로토콜 확장의 경우 MAVLink의 새로운 자료구조를 재정의하도록 아키텍처를 잡는 것이 바람직하다.
