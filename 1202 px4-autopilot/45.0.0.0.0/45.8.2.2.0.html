45.8.2.2 임무 플래시 메모리 섹터의 체크섬(Checksum) 및 암호화 해시(Cryptographic Hash) 무결성 사전 검사

무인항공기(UAV)가 자율 임무(Mission) 비행을 수행하기 위해서는 지상 관제 시스템(GCS)으로부터 전달받은 다수의 웨이포인트(Waypoint)와 명령 데이터 집합이 비행 제어기(Flight Controller)의 비휘발성 저장 장치(주로 플래시 로컬 메모리나 SD 카드)에 안전하게 기록되어야 한다. 이 과정에서 저장 매체의 물리적 배드 섹터(Bad Sector), 전원 차단으로 인한 쓰기 실패, 또는 통신 중의 비트 스위칭 오류로 인해 메모리가 오염될 가능성이 존재한다.

PX4-Autopilot은 이와 같은 치명적인 임무 데이터 손상을 방지하고 비행 제어 로직 구조의 무결성을 유지하기 위해 임무 데이터를 실행(Mission Start)하기 전과 부팅 시점에 체크섬(Checksum) 및 암호화 해시(Cryptographic Hash)를 활용한 무결성 사전 검사(Pre-flight Integrity Check)를 수행한다. 본 절에서는 데이터 매니저(dataman) 계층과 MAVLink 임무(Mission) 프로토콜을 중심으로 임무 플래시 메모리의 구조와 해시 알고리즘 적용 방식을 심도 있게 분석한다.

1. MAVLink 기반 임무 데이터 무결성 검증 로직

QGroundControl(QGC) 등 지상 관제 소프트웨어는 기체와 MAVLink의 Mission Protocol 버전을 협상하여 임무 데이터를 업로드한다. 데이터가 기체에 정상적으로 도달했는지를 확인하기 위해, MAVLink 프로토콜상에서 전체 미션 항목 수와 각 내용에 대해 CRC(Cyclic Redundancy Check) 또는 자체적인 해싱 구조를 통해 송수신 무결성을 1차적으로 보증한다.

  • MISSION_REQUEST_INT 및 MISSION_ITEM_INT 통신 중에 수신된 데이터는 기체의 메모리에 직렬화되어 쓰여지며, 모든 기록이 끝난 뒤 QGC는 기록된 미션이 원본과 동일한지 재차 확인하기 위해 MISSION_ACK 단계를 거친다.
  • 그러나 이는 ’통신 상의 오류’를 잡는 단계이며, 기계 자체 내에 영구 저장될 때의 무결성을 보장하기 위해서는 추가적인 스토리지 단위의 무결성 검증이 필요하다.

2. PX4의 dataman 모듈 중심의 메모리 무결성 검사

PX4 시스템은 비행 데이터를 고속으로 빈번하게 읽고 쓰기 위해 파일 시스템 부하가 큰 VFAT/FAT32 대신 전용 데이터 관리자인 dataman(Data Manager)을 사용한다. dataman 은 사전 할당된 플래시 메모리 섹터를 사용하여 일정한 크기의 아이템 배열(예: 임무 항목, 지오펜스, 랠리 포인트)을 관리한다. 임무 데이터는 시스템이 재부팅되어도 보존되는 구조(Persistent Storage)로 보관된다.

2.1 섹터의 체크섬(Checksum) 및 서명 구조

메모리 블록(섹터)을 관리하기 위해, PX4는 저장되는 각 항목이나 배열의 메타데이터 블록에 헤더(Header) 정보와 검증 필드를 둔다.

  • 데이터 페이로드(Payload): 실제 웨이포인트(위도, 경도, 고도 및 파라미터) 데이터
  • 길이(Length): 유효한 데이터의 크기
  • 체크섬/해시(Checksum/Hash): CRC32 또는 간단한 16-bit 덧셈 체크섬, 최근 빌드 생태계에서는 Fletcher-32 알고리즘 또는 외부 요구 시 SHA 계열 암호화 해시(다소 무거운 알고리즘이므로 보통 컴패니언 컴퓨터 단위에서 검수)를 페이로드에 대한 검증용으로 추가 메타데이터 영역에 기록한다.

2.2 부팅 및 임무 모드 진입 시 검사 파이프라인

임무 데이터의 플래시 메모리 무결성 검증은 주로 두 가지 시점에 수행된다.

  1. 초기 부팅 시점(Boot Initialization): dataman 모듈이 초기화될 때 저장된 파일을 읽어들이고 메타데이터의 체크섬 필드를 검증한다. 해시/체크섬 값이 불일치하면 해당 메모리 영역을 ’손상됨(Corrupted)’으로 마킹하고 캐시를 초기화하며, 임무 데이터가 없음을 비행 제어기에 보고한다.
  2. 임무 모드 진입 시점(Mode Transition Pre-condition): 커맨더(Commander) 또는 내비게이터(Navigator) 모듈에서 임무 비행 모드(Mission Mode)로 진입할 수 있는지 여부를 상태 공간 로직에서 확인할 때 데이터 무결성을 검증한다. 무결성 플래그가 True이고 임무 데이터의 해시값이 유효할 때만 상태 천이가 허가된다.

4. Ardupilot 대비 암호화 해시 검증 아키텍처의 차이점

Ardupilot은 전통적으로 EEPROM이나 DataFlash를 사용하여 Mission 데이터를 저장할 때 무결성을 관리하기 위해 구조적 파라미터 기반의 스토리지 시스템(AP_Mission) 내부에서 간단한 체크섬 비교와 사이즈 일치 여부를 판별한다. 반면, PX4-Autopilot은 dataman 이라는 별도의 독립 워커 데몬(Worker Daemon)을 두고 IPC(Inter-Process Communication)/uORB 메시징 구조 하에 독립적인 I/O 관리로 접근한다는 구조적 차이가 있다.

최소한의 CPU 사이클만을 사용하여 메모리 병목 현상을 막도록 스케줄링이 설계된 PX4의 dataman 모듈은, 보안성 강화를 위해 필요 시 임베디드 칩 레벨의 하드웨어 가속기(Crypto Accelerator)와 묶어 섹터 단위의 무결성 암호화 해시 검증(예: Secure Boot 체인과 유사한 서명 검증)으로 그 구조의 확장성을 제시하고 있다.

5. 소스 코드 레벨에서의 검증 로직 이해

실제 구현에서는 임무 데이터를 로드할 때 오류를 필터링하는 파라미터가 사용된다. C++ 구조체 관점에서 mission_s 의 헤더부에 기록된 체크섬을 대조하는 방식은 시스템 안정성에 직결된다.

// PX4 dataman 서브시스템 무결성 검증 간략 예시

bool verify_mission_sector_integrity(unsigned item_index) {
    uint8_t buffer[dm_sector_size];
    // 플래시 메모리에서 데이터 로드
    ssize_t bytes_read = dm_read(DM_KEY_WAYPOINTS_OFFBOARD, item_index, buffer, sizeof(buffer));
    
    if (bytes_read < 0) {
        return false; // I/O 오류
    }

    mission_item_s *item = reinterpret_cast<mission_item_s *>(buffer);
    
    // 계산된 해시/CRC 값
    uint32_t computed_crc = calculate_crc32(buffer + sizeof(uint32_t), bytes_read - sizeof(uint32_t));
    
    // 저장된 메타데이터 해시값 대조
    if (item->checksum != computed_crc) {
        PX4_ERR("Mission sector corruption detected! Index: %d", item_index);
        return false;
    }
    
    return true; // 무결성 입증
}

이와 같이 PX4는 비행 전 철저한 사전 검사(Pre-flight Test) 프로세스의 일부로서, 임무 플래시 메모리 섹터의 무결성을 보장하는 고도화된 체계를 구현하고 있다.
