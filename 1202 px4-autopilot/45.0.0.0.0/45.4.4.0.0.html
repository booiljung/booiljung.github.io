45.4.4. 커맨더 스레드(commander.cpp) 내 유한 상태 머신 천이 행렬(Transition Matrix)의 분기 예측(Branch Prediction) 및 제어 흐름 추적

1. 개요 및 이론적 배경

PX4-Autopilot의 심장부에는 기체의 생명 주기(오프라인, 시동 대기, 비행 중, 비상 사태, 종료) 전반을 관장하는 중앙 통제 모듈인 **커맨더(Commander)**가 존재한다. 커맨더 논리의 핵심은 거대한 **유한 상태 머신(FSM, Finite State Machine)**이다.
비행 모드의 변경이나 시동(Arming) 과정은 단순한 if-else 분기 처리가 아니라, 수학적인 천이 행렬(Transition Matrix) 개념을 바탕으로 현재 상태(Current State)에서 목표 상태(Target State)로 넘어가는 것이 타당한지를 수치적, 논리적으로 예측하고 검증하는 과정을 거친다.

본 절에서는 commander.cpp 소스 코드를 중심으로 FSM 내부의 천이 매트릭스 구현, 조건부 분기 예측(Branch Prediction) 기법, 그리고 복잡하게 얽혀 있는 제어 흐름(Control Flow) 추적 아키텍처를 집중적으로 조명한다.

2. 커맨더 모듈(commander.cpp)의 아키텍처와 FSM 메커니즘

PX4의 커맨더 모듈은 별도의 데몬(Daemon) 스레드로 동작하며, 20Hz~50Hz의 주기로 계속 시스템 전체의 상태(Sensor Health, Battery, RC/MAVLink Input, EKF Validity)를 순회(Polling) 및 구독(Subscribe)한다.

FSM은 크게 세 가지 축으로 기체의 상태를 정의한다.

  1. Arming State (시동 상태): INIT, STANDBY, ARMED, SHUTDOWN, IN-AIR RESTORE 등
  2. Main State (주 상태 / 비행 모드): MANUAL, ALTCTRL, POSCTRL, AUTO_MISSION, AUTO_RTL, OFFBOARD 등
  3. Navigation State (내비게이션 상태): Main State의 세부적인 오퍼레이션 상태

2.1 천이 행렬(Transition Matrix)의 역할

상태 머신 내에서 무작위적인 상태 천이는 치명적이다(예: 공중에서 ARMED \rightarrow STANDBY로 천이 시 추락). commander.cpp의 main_state_transition() 함수는 각 상태 간의 천이를 허가하는 거대한 논리 행렬(Logical Matrix) 역할을 수행한다. 천이를 시도할 때 제어 흐름은 현재 상태의 Row와 목표 상태의 Column을 대조해보며, 사전 결정된 제약 조건(Constraints) 집합을 모두 통과해야만 상태 변경을 승인한다.

3. 분기 예측(Branch Prediction) 기반 전제 조건 검사

통상적인 FSM은 “트리거 발생 \rightarrow 상태 변경“의 절차를 밟지만, 항공 우주 소프트웨어인 PX4의 커맨더는 소프트웨어적 **분기 예측(Forward-looking Branch Prediction)**과 유효성 검증(Validation) 파이프라인을 사전에 거친다.

3.1 소프트웨어적 천이 예측 제어 흐름

모드 변경 트랜잭션이 발생할 때, 커맨더는 해당 분기(Branch)를 즉시 타는 것이 아니라, 목표 모드가 요구하는 “미래의 센서와 액추에이터 활성도“를 가상으로 점검한다.

  • 예측 로직: Position 모드로의 천이 명령이 수신되었다고 가정할 때, 제어 흐름은 즉각 천이 플래그를 세우지 않고, commander_helper.cpp 및 프리플라이트 점검(Pre-flight check) API를 호출해 로컬 위치 추정치(local_position.valid)가 향후 1~2초 내에 지속적으로 유효할 것인지 (EKF 혁신 벡터의 동향 분석) 관찰 및 예측한다.
  • 예측 결과 불안정성이 높다면 (예: GPS Jamming 비율 상승 중), 상태 천이 행렬의 해당 교차점은 논리적 False로 막히게 되며, 커맨더 스레드는 분기를 우회(Fallback)하거나 그대로 Hold 상태에 머무른다.
// 커맨더 내부 FSM 분기 구조 (Pseudo-Code)
transition_result_t transition_main_state(State current, State target, HealthFlags flags) {
    // 1단계: 천이 행렬 기반의 기본 허구 검사 (Is transition topologically valid?)
    if (!valid_transition_matrix[current][target]) return REJECTED_TOPOLOGY;
    
    // 2단계: 예측 기반 전제 조건(Pre-condition) 평가
    if (target == MAIN_STATE_AUTO_MISSION) {
        if (!predict_gnss_stability(flags) || !is_mission_loaded()) {
             return REJECTED_PRECONDITION; 
        }
    }
    
    // 3단계: 제어 흐름 상태 업데이트 인가
    apply_state(target);
    return TRANSITION_ACCEPTED;
}

4. 파라미터 기반 FSM 설정 및 구성

사용자는 QGroundControl을 경유해 여러 파라미터를 변경함으로써 커맨더 FSM의 분기 가중치를 물리적으로 조작할 수 있다.

  • COM_FLTMODE1 ~ COM_FLTMODE6: 조종기 스위치 PWM 채널 값에 매핑되는 하드웨어 FSM 천이 타겟 행렬.
  • COM_RC_IN_MODE: RC 수신 불량 시 FSM의 상태 예측 흐름을 변경하여, 오프보드 모드로만의 강제 천이나 수동 모드 잠금 해제를 설정한다.
  • COM_ARM_WO_GPS: GPS Lock 없이 시동을 걸 수 있는지 여부. 이 파라미터가 비활성화되어 있으면 Standby \rightarrow Armed 분기 파이프라인에서 GPS 체커(Checker)가 무조건 차례를 블로킹(Blocking)한다.

5. Ardupilot 대비 아키텍처 차이

오픈소스 제어기 진영의 양대 산맥인 PX4와 Ardupilot은 상태 관리 철학에서 뚜렷한 대비를 띈다.

  • PX4-Autopilot 의 중앙집중형 FSM: PX4는 commander라는 하나의 강력한 모놀리식(Monolithic) 데몬이 전체 기체의 상태 지도를 Unified Transition Matrix로 들고 있다. 멀티로터든 고정익이든 FSM의 최상단 제어 흐름은 동일한 파이프라인을 통과하여 각 서브 모듈을 켜고 끄는 방식이다. 에러 디버깅 시 논리적 병목을 추적하기 편하며 시스템 수학적 증명(Validation)이 쉽다.
  • Ardupilot의 분산형 FSM: Ardupilot은 기체별(Copter, Plane, Rover) mode.cpp, mode_rtl.cpp 등으로 상태 제어 로직이 객체 지향적으로 철저히 분산되어 있다. 하나의 거대한 천이 행렬을 가지기보다는, 모드 인스턴스가 호출될 때 체인 구조로 자신의 init() 조건만을 자체 점검한다. 유연성이 극대화되나, 전역적인 관점에서의 분기 예측 추적도는 PX4에 비해 시각화하기 까다롭다.

6. MAVLink 및 ROS2 텔레메트리 관찰

GCS 운영자와 자율 비행 개발자는 실시간으로 커맨더의 FSM 상태 분기를 추적해야 한다.

  • MAVLink HEARTBEAT (ID: 0): 커맨더가 상태를 천이할 때마다 base_mode와 custom_mode 필드를 갱신하여 1Hz 주기로 송신한다. 이 하트비트 트래킹을 통해 QGroundControl은 상단 인디케이터에 현재 모드(PosCtl, Mission 등)를 표시한다.
  • MAVLink COMMAND_INT (ID: 75) & COMMAND_ACK (ID: 77): 모드 전환을 외부에서 시도할 때, 커맨더 분기 예측부가 이를 거절하면 즉각 COMMAND_ACK의 결괏값으로 MAV_RESULT_DENIED를 반환하며, 거절 사유에 대한 정보를 콘솔 텍스트(STATUSTEXT)로 뱉는다.
  • ROS2 VehicleStatus 메시지: uORB 토픽인 vehicle_status를 캡슐화한 메시지이다. arming_state, nav_state 등의 멤버 변수를 보유하고 있어 Companion Computer 측에서 PX4 커맨더 FSM의 실시간 현재 핀(Pin) 위치를 모니터링하여 복합 지능형 에이전트 설계의 근간으로 사용한다.
