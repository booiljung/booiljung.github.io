45.3.4 핸드셰이킹(Handshaking) 제어 흐름 내에서의 유한 상태 머신 교착 상태(Deadlock) 및 라이브락(Livelock) 방지 로직

지상 관제 시스템(GCS) 또는 오프보드(Offboard) 애플리케이션(예: MAVSDK, ROS2 노드)에서 무인항공기(UAV)로 비행 모드 변경이나 핵심 제어 명령을 내릴 때, MAVLink 프로토콜은 안정성과 무결성을 보장하기 위해 COMMAND_LONG (또는 COMMAND_INT) 송신과 COMMAND_ACK 수신 체계인 핸드셰이킹(Handshaking) 프로세스를 거친다. 하지만 무선 데이터 링크의 본질적인 패킷 손실(Packet Loss), 높은 지연 시간(Latency) 및 기체 내부 시스템의 동역학적 조건 불만족 등의 제약으로 인하여, 양 시스템의 유한 상태 머신(Finite State Machine, FSM)은 쉽게 교착 상태(Deadlock)나 라이브락(Livelock)의 늪에 빠질 수 있다. 본 절에서는 이러한 분산 상태 동기화 파이프라인에서 발생하는 위험성과 이를 방어하기 위한 로직을 기술한다.

1. 핸드셰이킹(Handshaking) 트랜잭션의 교착 상태(Deadlock) 메커니즘

교착 상태(Deadlock)는 GCS의 상태 머신이 비행 모드 변경 명령을 내린 후 PX4로부터 COMMAND_ACK 응답을 영구적으로 기다리는 동안 블로킹(Blocking)되어, 이후의 새로운 명령어나 관제 흐름을 처리하지 못하는 마비 현상을 의미한다.

특히 PX4-Autopilot의 최상위 상태 제어 모듈인 Commander.cpp는 Ardupilot의 느슨한 전이 조건과 달리 대단히 엄격한 전제 조건(Pre-conditions)을 검증한다. 예를 들어 GPS의 수평 위치 오차 공분산이 일시적으로 한계값을 초과하거나 배터리 전압이 일시 강하하는 순간 Acro나 Position 모드로의 전환 명령을 즉각적으로 거부(MAV_RESULT_DENIED)하거나 묵살할 수 있다.

이때 GCS 내부의 상태 머신이 오직 ‘수락(Accepted)’ 응답만을 기다리도록 설계되어 있다면 GCS는 무한 대기 루프에 들어가는 데들락 상태에 봉착하게 된다.

2. 라이브락(Livelock)과 대역폭 공격 유사 현상

라이브락(Livelock)은 교착 상태와 달리 시스템의 상태가 지속적으로 변하지만, 결코 다음 성공 상태로 전진하지 못하고 특정 흐름의 무한 반복에 갇히는 현상이다.

개발자가 단순한 타임아웃(Timeout) 및 무한 재시도(Retry) 방식을 적용할 경우 흔히 발생한다. GCS가 명령 송신 후 0.5초 안에 ACK가 오지 않았다고 해서 동일한 COMMAND_LONG을 재차 송신하고, PX4가 여전히 내부적인 EKF(Extended Kalman Filter) 초기화 지연 문제로 MAV_RESULT_TEMPORARILY_REJECTED를 회신하는 상황이 반복된다. 이는 GCS와 PX4 간의 활발한 통신 트래픽을 유발하며 가시적으로는 시스템이 살아있는 것처럼 보이지만, 실질적으로는 비행 모드 전환이라는 목표에 영원히 도달하지 못하는 라이브락에 빠진 것이다.

극단적인 경우, 너무 짧은 주기의 재시도 로직은 PX4의 mavlink_receiver 데몬에 페이로드 부하를 가중시켜 다른 정상적인 텔레메트리나 핵심 uORB 메시지 처리(예: vehicle_command 구독)까지 지연시키는, 이른바 ’의도치 않은 타이밍 공격(Timing Attack)’과 유사한 자원 고갈 현상을 유발할 위험이 있다.

3. 유한 상태 머신의 교착 및 라이브락 방지 설계 패턴

고도의 신뢰성을 확보하기 위해 QGroundControl 매니저나 MAVSDK 등의 관제 소프트웨어 계층에서는 다음과 같은 구조적 안전화(Structural Stabilization) 로직을 탑재해야 한다.

3.1 지터링(Jittering) 및 지수 백오프(Exponential Backoff) 기반 재전송

라이브락 및 대역폭 포화를 막기 위하여 단순한 선형 고정 타이머(예: 500ms 주기 반복) 대신 지수 백오프 알고리즘을 사용해야 한다. 명령 실패나 연기(Temporarily Rejected) 판정을 받았을 때 1차 재시도 대기는 0.5초, 2차는 1초, 3차는 2초 등으로 대기 시간을 기하급수적으로 이완시키며, 이 과정에 미세한 난수 지속시간(Jitter)을 가산하여 통신 버퍼 충돌을 분산시킨다. 최대 재시도 횟수(예: 3~5회)를 초과하면 즉각 명령 트랜잭션을 포기하고 상태 머신을 Abort/Failed로 강제 천이시킨다.

3.2 상태 머신의 비동기적 타임아웃 분리 탈출 로직

GCS 제어 로직에서 COMMAND_LONG에 대한 비동기(Asynchronous) 호출 메커니즘을 십분 활용해야 한다. 호출 함수가 블로킹(Blocking)되지 않고 백그라운드 스레드에서 ACK를 리스닝하도록 설계하며, 독립적인 타이머 타이머가 한계선(Deadline)을 넘으면 즉각 ’타임아웃 이벤트’를 발생시켜 FSM이 예외 처리(Exception Handling) 분기로 벗어나 데들락을 방어하도록 해야 한다.

3.3 PX4의 MAV_RESULT Enum 의 명시적/선택적 해석 파이프라인

PX4 관제에서 신뢰도 확보의 핵심은 COMMAND_ACK 페이로드 중 결과 코드(result)를 단순히 성공/실패의 이분법으로 치부하지 않는 데 있다.

  • MAV_RESULT_ACCEPTED: 즉각 트랜잭션 성공 종료.
  • MAV_RESULT_TEMPORARILY_REJECTED: 센서 초기화나 일시적 딜레이 상태임. 이 경우에만 재시도(Retry) 로직을 구동한다.
  • MAV_RESULT_DENIED 또는 MAV_RESULT_UNSUPPORTED: 해당 고등 비행 모드로의 천이가 원천적으로 금지되었거나 스펙을 지원하지 않는 경우이다. 이 결과 코드가 수신된 즉시 모든 재시도 루프를 파괴(Break)하고 사용자 또는 상위 시스템에 강한 경고 알람을 표출하여 라이브락을 차단한다.

4. 결론

GCS 개발자가 단순히 패킷 송신만을 목적으로 제어 로직을 짠다면, PX4의 보수적인 시스템 검증 사상은 수시로 명령을 거부할 것이고 전체 제어망은 교착 상태와 라이브락의 늪에 빠질 것이다. 따라서 명령 하달 계층은 PX4 Commander의 반응 특성을 온전히 반영하여, 타임아웃 이벤트, 횟수 제한 기재, 지수 백오프 및 상세한 예외 코드를 반영하는 강건 제어(Robust Control) 로직이 요구된다.
