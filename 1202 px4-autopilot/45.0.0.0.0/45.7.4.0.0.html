45.7.4. GCS 운영자 인지 부하(Cognitive Load) 저감을 위한 상태 천이 실패 요인 텍스트 스트림의 정규 표현식(Regular Expression) 휴리스틱 파싱(Parsing) 기법

지상 관제 시스템(GCS, Ground Control Station)에서 무인기(UAV) 운영 중 비행 모드 천이(Flight Mode Transition)와 같은 주요 이벤트가 실패할 경우, 시스템은 실패의 원인을 포함한 다량의 진단 메시지를 STATUSTEXT MAVLink 메시지 형태로 GCS에 전송한다. 이러한 메시지들은 주로 원시 텍스트 스트림(Raw Text Stream) 형태로 표출되며, 긴박한 상황에서 운영자가 이를 실시간으로 독해하고 조치하는 과정은 인지 부하(Cognitive Load)를 급격히 가중시킨다. 특히, 다수의 기체를 동시 관제하는 상황이나 예기치 못한 비상 상황(Emergency)에서는 인적 오류(Human Error)를 유발하는 핵심 요인이 된다.

본 절에서는 이러한 문제를 해결하기 위해, 텍스트 스트림에서 유의미한 상태 천이 실패 요인을 추출해 내는 정규 표현식(Regular Expression) 기반의 휴리스틱 파싱(Heuristic Parsing) 기법에 대해 상세히 다룬다. 본 내용은 PX4-Autopilot 펌웨어 v1.14 이상 및 QGroundControl v4.3 이상 환경을 기준으로 설명한다.

1. 상태 천이 거부와 STATUSTEXT 스트림

PX4-Autopilot은 비행 모드 천이(예: Position 모드에서 Mission 모드로의 전환) 요건이 불충족될 경우, 커맨더(Commander) 모듈에서 모드 전환을 거부(Reject)하고 그 원인을 MAVLink의 STATUSTEXT (메시지 ID #253) 형식으로 브로드캐스트한다.

이때 전달되는 메시지는 다음과 같은 특성을 갖는다.

  • 비정형성(Unstructuredness): 오류 메시지는 가독성을 위해 작성된 자연어 문자열이므로 데이터 속성(예: 센서 ID, 오류 코드)이 정형화되어 있지 않다.
  • 지속적 스트리밍(Continuous Streaming): 단일 오류 요인이 연쇄 반응을 일으켜 다수의 경고 메시지 스트림이 연속적으로 반환될 수 있다.

이러한 특성으로 인해 GCS 측에서는 단순한 문자열 매칭(String Matching)만으로는 정확한 실패 요인을 추론하기 어려우며, 정규 표현식과 시스템적 지식(Heuristic)을 결합한 지능형 파싱 파이프라인이 요구된다.

2. 휴리스틱 기반 정규 표현식 파싱 설계 (Heuristic Regex Parsing)

정규 표현식(Regex)을 활용한 파싱은 텍스트 내에서 특정한 규칙을 갖는 패턴을 찾아내는 강력한 방법이다. 그러나 PX4의 오류 메시지 형식은 펌웨어 버전에 따라 미세하게 변경될 수 있으므로, 엄격한 매칭보다는 키워드와 패턴의 조합을 찾는 휴리스틱(Heuristics) 기법을 함께 적용해야 강건성(Robustness)을 확보할 수 있다.

2.1 주요 상태 천이 실패 패턴 및 정규식 예시

다음은 PX4 운영 중 흔히 발생하는 상태 천이 실패 메시지와 이를 추출하기 위한 대표적인 정규 표현식 패턴이다. (C++의 std::regex 또는 Python의 re 모듈에서 사용 가능한 표준 문법 기준)

사례 1: EKF(Extended Kalman Filter) 초기화 및 상태 불량

  • 발생 메시지: "Preflight Fail: EKF HGT error" 또는 "Preflight Fail: EKF pos horiz variance"
  • 정규화 패턴: (?i)Preflight\s+Fail:\s+EKF\s+(.*)
  • (?i): 대소문자를 구분하지 않음 (Case-insensitive)
  • \s+: 하나 이상의 공백
  • (.*): 오류의 세부 요인을 캡처 그룹(Capture Group)으로 추출
  • 파싱 결과: “HGT error” 또는 “pos horiz variance” 추출. GCS는 이를 인지하여 운영자에게 “고도 추정기 이상“이라는 직관적인 UI 경고 팝업을 표시할 수 있다.

사례 2: GPS 수신 불량으로 인한 위치 제어 모드 진입 실패

  • 발생 메시지: "Reject Position Control: High GPS DoP" 또는 "Failsafe enabled: no global position"
  • 정규화 패턴: (?i)(Reject|Failsafe).*(Position|GPS).*
  • 이 경우 “Reject” 또는 “Failsafe” 키워드와 “Position” 또는 “GPS” 키워드가 동일 문장 내에 존재하는지 확인한다.
  • 파싱 결과: 위치 추정 관련 실패 문맥임을 인지(Context Recognition)하고, GCS 맵 뷰에서 HDOP 수치를 붉은색으로 강조점등(Blinking)하도록 연동한다.

사례 3: 배터리 전압 강하 (Low Battery)

  • 발생 메시지: "Critical battery: Return to launch condition"
  • 정규화 패턴: (?i)(Critical|Low)\s+battery.*(Return|Land)
  • 파싱 결과: 즉각적인 회항 또는 착륙이 필요한 배터리 고갈 상태임을 추출.

3. 운영자 인지 부하 관점의 설계 및 적용 효과

휴리스틱 파싱 결과를 UI 단위와 연동함으로써 GCS 운영 환경은 다음과 같은 변화를 가져온다.

  1. 정보의 시각적 추상화(Visual Abstraction of Information): 복잡하고 원시적인 텍스트 스트림(“Preflight Fail: EKF pos horiz variance”)을 즉각적이고 직관적인 아이콘(예: 붉은색 EKF 경고등)으로 치환한다. 이는 주의 자원(Attentional Resources)을 여러 데이터에 분산시키지 않도록 돕는다.
  2. 행동 유도(Affordance) 제공: 정해진 패턴의 에러가 검출되면, GCS는 오류 메시지 표출에 그치지 않고 해결을 위한 버튼(예: “센서 캘리브레이션 재시작”, “강제 시동(Force Arm)”)을 팝업창 내 동적으로 제공한다. 이는 대응 시간을 비약적으로 단축시킨다.
  3. 음성 합성(TTS, Text-to-Speech) 연동 최적화: 원시 텍스트를 그대로 TTS로 읽어주는 것은 오히려 청각적 인지 부하를 높인다. 파싱을 통해 정제된 문장(예: “Position Mode Reject. High GPS DoP.” -> “GPS 품질 저하로 위치 제어 모듈에 진입할 수 없습니다.”)으로 변환하여 송출함으로써 음성 피드백의 효율성을 극대화한다.

4. Ardupilot의 오류 메시징 체계와의 차이

Ardupilot은 PX4 대비 GCS와의 상태 동기화에 있어 오류의 상태 변수를 세분화하여 메시지를 송출하는 경향이 있다. 예를 들어, SYS_STATUS나 특수 확장을 사용한 상태 비트를 상세하게 전달하기도 한다.
반면 PX4는 MAVLink의 표준 장명령(COMMAND_LONG)의 NACK 응답 시 부가적인 STATUSTEXT로 실패 사유를 구어체 형태로 전달하는 방식이 잦으며, 이로 인해 PX4 기반의 범용 GCS를 설계할 때에는 본 절에서 다룬 텍스트 스트림 파싱 파이프라인의 구축이 더욱 중요한 구조적 요소로 작용한다.

5. 요약

고등 비행 모드로의 천이 시 실패 요인은 기체 안전성에 직결된다. GCS 설계자는 MAVLink로 유입되는 STATUSTEXT 스트림을 단순히 콘솔에 표시하는 수준을 넘어, 정규 표현식 엔진과 휴리스틱 데이터베이스를 활용한 실시간 파싱 시스템을 갖추어야 한다. 이를 통해 운영자의 인지 부하를 줄이고, 복잡한 오류 요인을 시각적·청각적으로 추상화하여 안전하고 신속한 의사 판단(Decision Making)을 지원하는 강건한 관제 시스템 아키텍처를 구현할 수 있다.
