45.8.1. 외부 제어(Offboard) 궤적 추종 모드 진입의 연속성 제약 조건

PX4-Autopilot의 로봇 운영 체제(ROS/ROS 2) 연동 및 군집 비행, 컴퓨터 비전 기반 자율 비행의 핵심은 보조 컴퓨터(Companion Computer)가 기체를 통제하는 외부 제어(Offboard) 모드이다. 보조 컴퓨터는 실시간으로 위치, 속도, 가속도, 추력 등의 목표값(Setpoint)을 계산하여 비행 제어기(FCC)로 전송하며, 기체는 이 궤적(Trajectory)을 추종하게 된다.

외부 제어 모드는 자율성이 매우 높은 반면, 보조 컴퓨터의 소프트웨어 오류나 통신 지연으로 인해 기체의 통제력을 상실할 위험이 크다. 따라서 PX4는 Offboard 모드로의 진입과 유지에 있어 목표값 스트림의 시간적 **연속성 제약 조건(Continuity Constraint)**을 매우 엄격하게 강제한다. 본 절에서는 이러한 연속성 제약 조건의 동작 원리와 구현 방식에 대해 상세히 다룬다.

1. 궤적 추종 모드의 연속성 제약 조건 개요

PX4 비행 제어 스택에서 “연속성 제약 조건“이란, 외부 컴퓨터로부터 전달되는 제어 목표값 메시지가 설정된 최소 주파수(Minimum Frequency) 이상으로 끊임없이(Continuously) 갱신되어야 함을 의미한다.

  • 배경: 기체가 고속 운항 중 외부 컴퓨터의 연산 처리 지연(Latency) 혹은 프로세스 다운 현상으로 인해 다음 목표 궤적이 제때 도착하지 않으면 제어 불능 상태에 빠지게 된다.
  • 제약 조건 성립: PX4는 Offboard 모드로 전환 시도 시 제어 목표값이 충분히 최신의 값이며, 일정한 주기로 유입되고 있는지(시간적 무결성) 사전 검사를 수행한다.

2. 모드 진입 시의 시간적 연속성(Pre-condition) 검증

Offboard 모드로 전환하라는 명령어(MAV_CMD_DO_SET_MODE)를 GCS나 MAVSDK를 통해 전송했을 때, PX4의 커맨더(commander) 모듈은 진입을 허가하기에 앞서 다음과 같은 제약 조건을 확인한다.

2.1 사전 목표값 수신 제약 (Prior Target Streaming)

PX4는 보조 컴퓨터가 Offboard 모드로 전환 명령을 내리기 전, 최소 0.5초(500ms) 동안 2Hz 이상의 주기로 목표값을 펌웨어로 먼저 전송하고 있어야만 모드 전환을 승인한다.

  • 만약 목표값 스트림이 사전에 존재하지 않는 상태에서 Offboard 모드 변경 명령만 단독으로 들어오게 되면, 커맨더는 이를 거부(Reject)하고 MAVLink STATUSTEXT로 상태 천이 거부 메시지를 발생시킨다. 이는 통신 링크가 완전히 확립되지 않은 상태에서의 위험한 권한 위임을 원천 차단하기 위함이다.

2.2 통신 단절 한계치 (Timeout Threshold) 파라미터

Offboard 모드 유지 중 목표값 스트림의 통신 단절을 판별하는 기준은 COM_OF_LOSS_T 파라미터에 의해 결정된다.

  • COM_OF_LOSS_T (기본값: 0.5초): 마지막 목표값 데이터를 수신한 지 이 파라미터에 설정된 시간이 경과하면, PX4는 연속성 제약 조건이 훼손된 것으로 판단하고 통신 단절(Offboard Data Link Loss) 상태를 선언한다.

3. 연속성 유지를 위한 소프트웨어 아키텍처 예시

ROS 2 및 uXRCE-DDS 브리지를 사용하여 제어 스트림을 퍼블리싱(Publishing)할 때, 연속성 제약을 충족하기 위한 권장 타이머 및 이벤트 흐름은 다음과 같다.

sequenceDiagram
    participant ROS2_Node as ROS 2 Companion Node
    participant uORB as PX4 uORB Middleware
    participant Commander as PX4 Commander Module
    
    Note over ROS2_Node: Initialize Control Loop (e.g., 50Hz)
    rect rgb(240, 248, 255)
    loop Every 20ms
        ROS2_Node->>uORB: Publish TrajectorySetpoint
    end
    end
    Note over Commander: Receives Valid Setpoints for > 0.5s
    ROS2_Node->>Commander: Send Mode Change: VehicleCommand(Offboard)
    Commander-->>ROS2_Node: Mode Accepted (Transition to Offboard)
    
    Note over ROS2_Node: Network interruption / Node Crash
    ROS2_Node--XuORB: Stop Publishing Setpoints
    Note over Commander: Timer > COM_OF_LOSS_T (0.5s)
    Note over Commander: Continuity Constraint Violated!
    Commander->>Commander: Trigger Failsafe (RTL, Hold, Return)

4. Ardupilot (Guided Mode)와의 제약 조건 차이

Ardupilot 환경에서는 PX4의 Offboard 모드와 유사한 형태인 Guided 모드가 존재한다.
Ardupilot의 Guided 모드는 목표 위치나 속도 명령이 단발성으로 전송되더라도 기체가 해당 목표를 향해 이동하며, 주기적인 데이터 전송(Streaming)을 엄격한 선결 요건으로 진입 단계에서부터 차단하지 않는 편이다. (물론 속도 제어 등 일부 하위 모드에서는 타임아웃이 존재한다.)

반면 PX4-Autopilot은 이 연속성 제약(Continuity Constraint)에 타협이 없다. 이는 PX4가 외부 보조 컴퓨터를 신뢰성 있는 “실시간 제어 시스템의 연장선“으로 취급하며, 통신 무결성이 보장되지 않는 외부 명령 입력으로부터 내부 비행 제어 스택을 견고하게 방어하려는 아키텍처 철학을 극명하게 보여주는 설계적 차이점이다.

5. 요약

PX4에서 외부 제어(Offboard) 모드 진입 및 유지는 지속적이고 규칙적인 목표값 스트림 발송이라는 **시간적 연속성 전제(Temporal Continuity Pre-condition)**가 충족될 때만 허용된다. GCS 인프라 구축이나 자율 비행(Autonomous Flight) 에이전트용 관제 백엔드를 설계할 개발자는, 일시적인 네트워크 트래픽 지연(Jitter)이나 마이크로 서비스의 병목 처리로 인해 제어 프레임이 누락될 경우 기체가 지연 타임아웃 판정을 받고 다른 비행 모드로 강제 천이 발동(Failsafe)될 수 있음을 명심해야 한다.

따라서 궤적 데이터 발행을 담당하는 통신 루프를 시스템 내 최우선 순위 쓰레드(Highest Priority Thread)로 할당하거나 실시간 운영체제(RTOS) 환경 위에서 엄밀한 주기를 보장하도록 스케줄링하는 방어적인 아키텍처 구현이 필수적으로 선행되어야 한다.
