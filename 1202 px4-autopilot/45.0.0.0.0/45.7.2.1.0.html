45.7.2.1 이진 지수 백오프(Binary Exponential Back-off) 기반 재전송 스케줄링 및 무선 대역폭 포화(Saturation) 방지 기법

본 절에서는 무인항공기(UAV)와 지상 관제 시스템(GCS) 통신 과정에서 돌발적인 무선 간섭이나 채널 포화(Saturation)가 발생했을 때, 네트워크 붕괴(Network Collapse)를 예방하고 통신을 안정적으로 복구하기 위한 이진 지수 백오프(Binary Exponential Back-off, 이하 BEB) 기반 재전송 스케줄링 기법에 대해 상세히 설명한다. 이는 PX4-Autopilot(펌웨어 v1.14 기준)과 QGroundControl(v4.3 기준)의 강건한 관제 구조를 이해하는 데 필수적인 요소이다.

1. 개요 및 통신 포화(Saturation) 문제

무인항공기의 텔레메트리(Telemetry) 링크는 제한된 대역폭(Bandwidth)을 가지며 반이중(Half-Duplex) 통신을 사용하는 경우가 많다. 이 환경에서 MAVLink 프로토콜의 중요한 제어 명령(COMMAND_LONG 등)이나 파라미터(Parameter) 동기화 요청이 여러 이유로 응답(ACK)을 받지 못할 경우, 송신 측(GCS 또는 PX4)은 타임아웃(Timeout) 이후 재전송(Retransmission)을 시도하게 된다.

만약 재전송을 일정한 고정 간격으로 무한정 반복한다면, 무선 채널의 가용 대역폭은 재전송 패킷만으로 가득 차게 되어 다른 중요한 텔레메트리 메시지 송수신까지 불가능해지는 채널 포화 통신 마비 현상을 초래한다. 이를 방지하기 위해 IEEE 802.11 MAC 계층 등에서 널리 활용되는 충돌 회피 알고리즘 논문인 “Performance Analysis of the IEEE 802.11 Distributed Coordination Function“에서 소개된 이진 지수 백오프 방식이 응용된다.

2. 이진 지수 백오프(BEB) 알고리즘 모델

이진 지수 백오프 알고리즘은 타임아웃 또는 패킷 충돌(Collision)이 빈번하게 발생할 때, 다음 재전송까지의 대기 시간(Back-off Time)을 지수 함수적으로 증가시키는 메커니즘이다.

수식으로 표현하면 k번째 재전송 시도에서의 대기 시간 T_w는 다음과 같이 정의된다.

T_w = \text{Random}(0, 2^k - 1) \times T_{slot}

단, k \le MAX\_RETRY 이며, GCS-드론 간의 MAVLink 애플리케이션 계층에서는 무작위 대기 시간 배분보다는 결정론적(Deterministic) 지수 함수 증가를 주로 응용한다. 즉, 수신 확인이 지연될 때마다 재전송 시간 간격을 직접 2배씩 늘려 채널 안정화 기간을 부여한다.

T_{retry}(n) = T_{initial} \times 2^{n-1}

상기 식에서 T_{initial}은 최초 설정된 타임아웃 임계치이며, n은 재전송 시도 횟수이다.

    A[명령 송신] --> B{ACK 수신?}
    B -- Yes --> C[정상 완료]
    B -- No --> D[패킷 유실 감지]
    D --> E{재전송 횟수 < MAX?}
    E -- No --> F[명령 실패 알림 및 예외 처리]
    E -- Yes --> G[재전송 대기 간격 2배 임시 증가]
    G --> H[백오프 타이머 대기]
    H --> A

3. QGroundControl 및 PX4의 적용 아키텍처

QGroundControl(GCS) 프론트엔드와 C++ 백엔드를 살펴보면 Vehicle 관리 매니저 내부에 통신 신뢰성을 담보하기 위한 백오프 스케줄링 구조가 존재한다.

  1. GCS에서 PX4로의 명령어 송신 (Command Request)
    GCS의 Vehicle::sendMavCommand 호출 시, COMMAND_LONG 메시지는 지정된 타임아웃(예: 500ms) 내에 COMMAND_ACK를 기다린다. ACK가 도달하지 않을 경우 GCS는 재전송 논리를 가동한다. 이때, 단순 반복이 아닌 네트워크 지연 상태를 감안한 재시도 간격 알고리즘이 동작하며, 극심한 혼잡 상태가 탐지될 경우 지수 백오프와 유사하게 딜레이 페널티가 부여된다.

  2. 파라미터 프로토콜 최적화 (Parameter Sync)
    기체와 GCS 연결 초기 단계에서는 대량의 파라미터 동기화가 이루어진다. 이 과정에서 한 번에 쏟아지는 파라미터 리스트 요청이 무선 망을 마비시키지 않도록, 누락 파라미터 조회를 요청할 때에도 무조건적인 연속 요청을 지양하고 응답 수신율을 계산하여 요청 속도를 조절하는 백오프 타이머가 개입한다.

4. Ardupilot과 PX4-Autopilot 관제 철학 차이

관제 소프트웨어 환경 및 응용 프로토콜 처리 방식 측면에서 두 시스템 간의 차이가 존재한다.

  • PX4-Autopilot: 시스템 및 통신 리소스가 낭비되는 것을 강력하게 제어한다. 통신 포화가 감지되면 GCS뿐 아니라 기체 상의 텔레메트리 스트리머 또한 특정 MAVLink 스트림의 송신율를 능동적으로 저하시킨다. GCS 계층에서도 이러한 백오프 스케줄링과 연동되어 송신 큐(Tx Queue)의 점유율을 적극 낮춘다. 이는 철저하게 ROS 2 노드와 같이 대용량 데이터 통신을 주고받는 최신 시스템 환경 기반의 실시간성(Real-time)을 유지하기 위한 철학의 산물이다.
  • Ardupilot: Mission Planner 기반의 GCS 시스템을 주로 활용하며, 재전송 스케줄링에서 고정된 타임아웃 주기를 사용하여 파라미터를 더욱 끈질기게 요청하는 성향을 띠고 있다. 대역폭 포화보다는 설정 값 수신의 안정성과 무결성에 더 높은 가중치를 두는 설계 철학을 보여준다. 즉, 대기 시간을 지수적으로 증가시키기보다는 반복적 확인 요청(Polling) 방식을 좀 더 적극적으로 사용한다.

5. 결론

무인항공기의 제어 및 텔레메트리 관제 시스템에서 통신 장애 및 대역폭 포화 현상은 비행의 안전과 직결되는 핵심 문제이다. 이진 지수 백오프(BEB) 알고리즘과 같은 적응형 재전송 스케줄링 구조를 적용함으로써 트래픽 폭주로 인한 MAVLink 채널의 마비 현상을 미연에 방지할 수 있다. PX4-Autopilot 및 QGroundControl은 이러한 동적 재전송 모델을 채택함으로써, 외부 간섭이나 돌발적인 신호 저하 시에도 통신 신뢰성을 탄력적으로 유지하고 안정적인 비행 복구(Recovery) 기반을 제공한다.
