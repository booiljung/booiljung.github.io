45.7.3 임계 재전송 허용치 초과 시 시스템 복구 파이프라인

본 절에서는 무인항공기(UAV)와 지상 관제 시스템(GCS) 간의 통신에서 명령어 타임아웃(Timeout) 및 임계 재전송 횟수(Threshold of Retransmission)를 초과하여 복구가 불가능한 통신 두절(Data Link Loss) 상태에 이르렀을 때 활성화되는 시스템 복구 파이프라인(System Recovery Pipeline)에 대해 다룬다. 본 내용은 PX4-Autopilot(펌웨어 v1.14 기준)과 QGroundControl(v4.3 기준)의 안전 장치(Failsafe) 메커니즘을 기반으로 설명한다.

1. 개요 및 재전송 임계치의 의미

MAVLink 프로토콜을 통하여 기체에 전달되는 상태 제어 명령(예: COMMAND_LONG 메시지)이나 파라미터(Parameter) 요청은 이진 지수 백오프(Binary Exponential Back-off) 등의 스케줄링에도 불구하고 극심한 무선 간섭으로 인해 최종적으로 실패할 수 있다. QGroundControl 혹은 PX4의 MAVLink 인터페이스가 사전에 정의된 최대 재전송 횟수(Maximum Retries, 일반적으로 3~5회)를 소진하면, 해당 제어 트랜잭션은 완전히 실패한 것으로 간주된다.

이러한 임계 재전송 허용치 초과는 단순한 패킷 드롭(Packet Drop)을 넘어서, 텔레메트리(Telemetry) 링크가 유효성을 상실했음을 시스템에 알리는 핵심 트리거(Trigger)가 된다. 이 기점을 바탕으로 시스템은 자율 복구(Autonomous Recovery) 모드로 전환하게 된다. “Design of a Robust Flight Control System for a Small Unmanned Aerial Vehicle” 등 주요 학술 논문에서도 통신 시스템의 하드 타이머 한계 도달 시 상태 천이를 강제하는 복구 메커니즘의 당위성을 제시하고 단기적 통신 회복 실패율을 줄이는 방안을 거론하고 있다.

2. PX4-Autopilot의 복구 파이프라인 아키텍처

PX4-Autopilot은 기체 내부에서 통신 및 텔레메트리 수신 스레드(Thread)를 통해 GCS와의 연속적인 Heartbeat 및 명령어 수신 상태를 모니터링한다.

  1. 상태 감지 및 전파 (State Detection & Propagation)
    GCS로부터의 유효한 트래픽이 임계 시간 이상 단절되거나, MAVLink 통신 모듈(mavlink_receiver 및 commander 모듈)에서 ACK/NACK 응답 및 재전송 교환이 완전히 마비된 경우, telemetry_status uORB 토픽(Topic)에 텔레메트리 유실 플래그(Data Link Loss Flag)가 세팅된다.

  2. 커맨더(Commander) 유한 상태 기계(State Machine)의 개입
    telemetry_status의 변경을 구독(Subscribe)하고 있는 PX4 커맨더(Commander) 데몬은 해당 상황을 비상(Failsafe) 상황으로 인지한다. 사용자가 파라미터(NAV_DLL_ACT)에 설정한 정책에 따라 다음의 복구 파이프라인 파라미터 분기가 실행된다.

  • Return 모드 전환: 기체가 즉각적으로 이륙 지점(Home Position) 혹은 랠리 포인트(Rally Point)로 복귀(Return) 상태로 천이한다.
  • Hold / Loiter 모드 전환: 통신이 잠시 후 복구될 가능성을 염두에 두고 현재 위치에서 공중 정지(Hold) 상태를 유지한다.
  • Land 모드 전환: GPS 등의 추가 센서도 신뢰할 수 없을 때, 제자리에서 안전하게 착륙(Land)을 시도한다.

3. QGroundControl (GCS) 측의 복구 경고 파이프라인

임계 재전송 허가 횟수를 초과하면 기체뿐만 아니라 GCS 단의 상태 머신도 복구 파이프라인에 진입한다. QGC의 C++ 백엔드(차량 매니저 클래스 산하)는 다음과 같은 조치를 취한다.

  • 명령 큐(Command Queue) 초기화 및 폐기: 지속적으로 전송 실패가 일어난 COMMAND_LONG 태스크 뿐만 아니라 연계된 후속 명령 큐를 모두 비움(Flush)으로써 시스템의 대기 상태 병목을 해소한다.
  • 오디오 피드백 및 UI 알림 강제: 운영자(Operator) 텍스트 스트림 UI와 오디오 출력 스레드에 비동기적으로 “Communication Lost” 이벤트를 송출하여 조종자가 인지 모드(Cognitive Mode)를 시각 조준 및 수동 대비 모드로 전환하도록 유도한다.

4. Ardupilot과 PX4-Autopilot의 비상 복구 철학 비교

두 플랫폼은 통신 임계치 초과 시 복구 파이프라인 진입에 있어 설계 차이를 보인다.

비교 항목PX4-AutopilotArdupilot
Failsafe 전이의 단호함COMMAND_ACK 타임아웃 및 Heartbeat 유실을 기반으로 한 매우 엄격한 상태 천이 우선 정책. 통신 두절(DLL) 시 설정된 Failsafe 트리거(Return, Land 등)를 유예 없이 즉시 실행.통신 타임아웃 발생 시 ’Short Failsafe’와 ‘Long Failsafe’ 등 단계적(Multi-stage) 지연을 두고 복구 행동의 수위를 점진적으로 격상하는 파이프라인 아키텍처 채택.
상태 머신 모듈화커맨더(Commander)라는 단일화된 중앙 통제 모듈이 uORB 브로드캐스팅을 통하여 모든 비상 트리거를 일원화하여 통제.GCS Failsafe 루틴이 각 비행 제어기(Copter, Plane) 내부 서브루틴에 강하게 결합되어 있어, 기체 종류에 따라 튜닝 파라미터 풀이 상이함.
복구 후 통신 재연결 처리Failsafe 상태(Return 모드 등)에 한 진입하면, 이후 통신이 재개되더라도 운영자가 모드 변경(예: Position 모드로 재전환)을 명시적으로 인가(Acknowledge)할 때까지 Failsafe 기반 비행을 유지(안전 최우선).특정 파라미터 설정 환경 하에서는 통신 복구 시 곧바로 이전 모드(예: Auto)로 롤백(Rollback)하는 옵션 활용이 가능하여 임무 연속성을 강조할 수 있음.

5. 결론

무인항공기의 제어 명령어 재전송 행위가 허용 임계치를 돌파하는 순간, 시스템은 통신 문제를 자가 치유 불가능한 치명적 결함으로 판단한다. 이를 대비하기 위해 PX4-Autopilot은 커맨더 모듈 및 uORB 통신 버스를 경유하는 신속하고 단호한 안전 복구 파이프라인(Safety Recovery Pipeline)을 전개하며, GCS 역시 명령 큐를 선제적으로 소거하고 운영자에게 경고 체계를 발동시킨다. 이러한 견고한 비상 파이프라인 아키텍처의 설계는 비가시권 비행(BVLOS) 등 통신 가시선이 확보되지 않는 악조건 속에서도 가장 높은 확률의 생존력과 회복 가능성을 기체에 부여한다.
