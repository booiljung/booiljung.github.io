45.9.1.2. Qt 프레임워크의 메타 오브젝트 시스템(Meta-Object System) 기반 시그널/슬롯(Signal/Slot) 이벤트 루프 블로킹(Event Loop Blocking) 회피 설계 전략

자율비행 관제 환경에서 지상 관제 시스템(GCS)은 수백 헤르츠(Hz)로 쏟아지는 원격 단말기(Telemetry) 데이터 흐름 속에서도 부드러운 렌더링(Rendering) 프레임률과 즉각적인 사용자 입력 반응성을 유지해야 한다. QGroundControl(이하 QGC)은 이러한 가혹한 실시간(Real-time) 비동기 처리 요구사항을 Qt 프레임워크 고유의 메타 오브젝트 시스템(Meta-Object System, MOS) 기반 시그널/슬롯(Signal/Slot) 메커니즘을 통해 돌파한다. 본 절에서는 QGC 내부에서 MAVLink 메시지 수신과 UI 렌더링 스레드(Thread) 간의 이벤트 루프 블로킹(Event Loop Blocking)을 방지하는 구조적 설계 전략을 분석한다.

1. Qt 이벤트 루프(Event Loop)와 화면 멈춤(Freezing) 현상의 근원

모든 Qt 애플리케이션은 메인 스레드(Main Thread, 통상 GUI 스레드)에서 QCoreApplication::exec()를 호출함으로써 무한 이벤트 루프에 진입한다. 이 루프는 마우스 클릭, 터치, 타이머 이벤트, 그리고 화면 갱신 이벤트를 끊임없이 수집하여 적절한 객체에 배분한다.

만약 메인 스레드 내부에서 비행 제어기(PX4-Autopilot)로부터 특정 MAVLink 응답(예: 파라미터 읽기, 트래픽이 몰릴 때의 COMMAND_ACK)을 받기 위해 동기식(Synchronous) 대기 상태(예: while 문을 통한 폴링 대기, sleep(), 또는 타임아웃 대기)에 빠진다면 어떻게 될까?
이벤트 루프의 실행 흐름이 해당 함수 호출에 묶여 다음 연산을 진행하지 못하므로, QGC의 QML 프론트엔드 버튼은 눌리지 않고 지도(Map) 화면은 업데이트되지 않는 심각한 화면 멈춤(Freezing) 현상 즉, 이벤트 루프 블로킹(Event Loop Blocking)이 발생하게 된다.

2. 메타 오브젝트 시스템(MOS)의 교차 스레드(Cross-Thread) 비동기 큐잉

QGC는 이벤트 루프 블로킹을 근본적으로 차단하기 위해 **다중 스레드 아키텍처(Multi-Thread Architecture)**와 Qt::QueuedConnection 타입의 시그널/슬롯 조합을 활용한다.

Qt의 C++ 확장 전처리기인 **moc(Meta-Object Compiler)**는 QObject를 상속받는 모든 클래스에 대해 런타임 타입 정보(RTTI)와 동적 메서드 호출이 가능한 메타데이터를 삽입한다. 이 메타데이터를 기반으로 동작하는 시그널/슬롯 연결 방식 중 교차 스레드 통신의 핵심은 다음과 같다.

  • 통신 스레드(MAVLink Receiver Thread): 물리적 직렬/네트워크 포트(UART, UDP)에서 패킷을 바이트 단위로 파싱하고 mavlink_message_t 구조체를 완성해낸다.
  • 시그널 방출(Signal Emission): 완성된 메시지가 유효성 검증을 통과하면, 통신 스레드는 messageReceived(mavlink_message_t) 같은 시그널을 방출(emit)한다.
  • 이벤트 큐(Event Queue) 삽입: 시그널을 수신하는 Vehicle 객체가 메인 스레드 소속일 경우, Qt 시스템은 두 객체의 스레드 컨텍스트(Context)가 다름을 인지한다. 즉시 슬롯 함수를 실행(Direct Connection)하는 대신, 슬롯 함수 실행 요청과 파라미터(MAVLink 메시지의 복사본)를 메인 스레드의 이벤트 큐 맨 뒤에 안전하게 삽입(Qt::QueuedConnection)하고 통신 스레드는 즉시 자신의 루프로 복귀한다.
  • 메인 스레드의 지연 실행(Deferred Execution): 현재 그리고 있던 UI 렌더링 작업을 마친 메인 스레드의 이벤트 루프는, 다음 틱(Tick)에 큐에 적재된 슬롯 함수(예: 고도 업데이트 로직)를 꺼내어 순차적으로 실행한다.

이러한 설계는 두 스레드 간의 락(Lock, Mutex) 경합을 최소화하면서도 완벽한 비동기식 생산자-소비자 모델(Producer-Consumer Model)을 구현하게 해준다.

3. MAVLink 명령 송수신 비동기 파이프라인 구조

아래 시퀀스 다이어그램은 QGC 내부에서 파라미터 값 변경(PARAM_SET)이나 비행 모드 명령과 같은 응답성(ACK) 트랜잭션을 처리할 때, 이벤트 큐 기반 비동기 파이프라인이 어떻게 블로킹 현상을 회피하는지를 보여준다.

    UI->>CommandQueue: 9. ACK 검증 후 타이머 해제 및 콜백 완료
  1. 사용자가 UI에서 버튼을 누르면, 슬롯 함수는 명령을 MAVLink 트랜잭션 큐(VehicleCommandList)에 등록하고 타이머(QTimer)만 켜둔 채 즉시(Direct Return) 종료된다.
  2. 메인 이벤트 루프는 즉시 복귀하여 부드러운 UI 렌더링을 계속한다.
  3. 별도의 네트워크/직렬 수신 스레드는 블로킹 소켓(Blocking Socket) 읽기를 수행하며 대기하다가 패킷이 들어오면 시그널을 방출한다.
  4. 방출된 데이터는 Q_DECLARE_METATYPE을 통해 안전하게 패키징되어 메인 루프 큐에 적재되고, UI 스레드는 유휴 시간 산출 시 이 큐 소모하여 상태를 최신화한다.

4. 임베디드 GCS 설계자들을 위한 기술적 통찰

ArduPilot 관제에 널리 쓰이던 초창기 Windows 형태의 GCS 응용 프로그램들은 종종 단일 스레드 기반 타이머나 윈도우 메시지 후킹에 의존하여 텔레메트리 대량 수신 시 일시적인 ‘Not Responding(응답 없음)’ 상태에 빠지기 일쑤였다. 반면 최근의 QGC 아키텍처는 C++11의 std::thread와 무거운 동기화 팩터를 직접 다루는 대신, Qt 메타 오브젝트 시스템의 signal/slot을 스레드 경계를 넘나드는 메시지 패싱(Message Passing) 수단으로 적극 채택하였다.
이는 ROS2 아키텍처의 노드(Node) 간 Executor 큐잉 모델과도 궤를 같이한다. PX4를 타깃으로 커스텀 GCS나 MAVSDK 기반 관제 앱을 설계할 때, 동기식 함수 호출(Synchronous Call)로 인한 블로킹의 유혹을 뿌리치고 철저한 비동기 시그널 큐(Asynchronous Signal Queue) 기반의 콜백(Callback) 설계를 강제해야만 수십 기를 관제할 다중 에이전트 환경(Swarm)에서도 무너지지 않는 안정성을 확보할 수 있다.
