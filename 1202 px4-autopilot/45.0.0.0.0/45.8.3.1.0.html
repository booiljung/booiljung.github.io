45.8.3.1 스레드 간 우선순위 역전(Priority Inversion) 방지를 위한 우선순위 상속 프로토콜(Priority Inheritance Protocol) 적용 및 모드 선점(Preemption) 알고리즘

PX4-Autopilot은 실시간 운영체제(RTOS)인 NuttX를 기반으로 수많은 비행 제어 및 미들웨어 스레드(Task/Thread)를 병렬로 구동하는 복잡한 소프트웨어 아키텍처를 가지고 있다. 이처럼 다중 스레드 환경에서 센서 데이터를 읽거나 하드웨어 리소스(I2C, SPI 버스 등), 혹은 공유 메모리(uORB 토픽)에 배타적으로 접근할 때 뮤텍스(Mutex)나 세마포어(Semaphore)와 같은 동기화 프리미티브(Synchronization Primitives)를 사용한다.

이 과정에서 가장 치명적인 문제인 **우선순위 역전(Priority Inversion)**이 발생할 수 있으며, PX4는 이를 방지하기 위해 **우선순위 상속 프로토콜(Priority Inheritance Protocol, PIP)**을 적극적으로 적용하고 있다. 또한, 비상 상황(예: RTL)에서 커맨더(Commander) 스레드가 다른 모드를 즉각적으로 탈취하는 모드 선점(Mode Preemption) 알고리즘이 시스템의 결정론적(Deterministic) 반응성을 보장한다.

1. 우선순위 역전(Priority Inversion) 현상과 위험성

우선순위 역전은 운영체제 스케줄링에서 고순위(High-priority) 스레드가 저순위(Low-priority) 스레드에 의해 간접적으로 블로킹(Blocking)되는 현상을 말한다.

  • 상황 예시:
  1. 저순위 스레드 T_L이 버스(I2C) 뮤텍스를 획득하여 센서를 읽고 있다.
  2. 고순위 스레드 T_H(예: 자세 제어기)가 활성화되어 동일한 버스 뮤텍스를 요청하지만, T_L이 소유하고 있어 대기 리스트에 들어간다(Blocked).
  3. 이때, 중순위 스레드 T_M(예: 로깅 스레드)이 깨어나 실행된다. T_M은 버스 뮤텍스가 필요 없으므로 리소스를 점유하며 실행을 이어간다.
  4. 결과적으로 T_L은 T_M에 의해 선점되어 CPU를 할당받지 못해 뮤텍스를 반환하지 못하고, T_H는 T_M이 끝날 때까지 무한정 기다려야 하는 우선순위 역전 현상이 발생한다.

수 밀리초(ms) 단위의 반응성이 중요한 비행 제어기(Flight Controller)에서 이러한 역전 현상은 자켓 제어 실패나 추락으로 이어진다.

2. 우선순위 상속 프로토콜(Priority Inheritance Protocol)의 적용

PX4-Autopilot 및 기반 OS인 NuttX는 이러한 문제를 해결하기 위해 우선순위 상속(Priority Inheritance) 기법을 뮤텍스 설정 단계에서 표준으로 채택하고 있다.

2.1 메커니즘 개요

고순위 스레드 T_H가 저순위 스레드 T_L이 쥐고 있는 락(Lock)을 요청할 경우, 운영체제 스케줄러는 일시적으로 T_L의 우선순위를 T_H와 동일하게 끌어올린다(Inherit). 이에 따라 중순위 스레드 T_M은 T_L을 선점할 수 없게 되며, T_L은 신속하게 작업을 마치고 락을 해제한다. 락이 해제됨과 동시에 T_L의 우선순위는 원래대로 복구되고, T_H가 즉시 실행을 이어간다.

2.2 PX4 뮤텍스(Mutex) 소스 코드 구현체

PX4 소스 내에서 스레드 뮤텍스를 초기화할 때 POSIX 표준 스레드 속성 중 PTHREAD_PRIO_INHERIT 속성을 부여하는 방식으로 구현되어 있다.

// PX4 내부 pthread 뮤텍스 초기화 예시 (Priority Inheritance 강제)

pthread_mutex_t _lock;
pthread_mutexattr_t attr;

pthread_mutexattr_init(&attr);
// 우선순위 상속 프로토콜 명시적 할당
pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT); 
pthread_mutex_init(&_lock, &attr);
pthread_mutexattr_destroy(&attr);

이와 같이 드라이버 단이나 uORB 내부의 공유 큐(Queue)에 접근할 때 우선순위 상속이 활성화된 락 메커니즘을 사용하여 시스템의 실시간(Real-time) 보장성을 해치지 않는다.

3. 임계 결함 감지 및 모드 선점(Mode Preemption) 알고리즘

우선순위 역전을 내부 스케줄링 단계에서 방지했다면, 시스템 논리 구성 관점에서는 비상 제어 로직이 기존의 비행 궤적 로직을 완전히 탈취(Preempt)해야 한다. 앞 절에서 다룬 임계 결함에 따른 RTL 강제 개입 역학과 밀접한 관련이 있다.

graph TD
    A[Normal Flight Task e.g. Mission] -->|CPU 점유| B(Flight Control Stack)
    C[Failsafe Trigger: Commander Task]
    
    C -->|1. Event Interrupt| D{Priority Assessment}
    D -->|High Priority| E[Preemption Signal 발송]
    E -->|uORB: vehicle_command| F[Navigator 태스크]
    
    F -->|2. Abort Current Thread Context| G[Terminate Mission Context]
    G -->|3. Establish New Context| H[RTL Setpoint Generation]
    H -->|최고 권한 제어| B

모드 선점 과정:

  1. 이벤트 트리거: 커맨더 스레드는 높은 스케줄링 우선순위(Higher OS Task Priority)를 가지며 주기적 혹은 인터럽트 기반으로 구동된다.
  2. 논리적 선점 (Logical Preemption): uORB 토픽(vehicle_status.nav_state)을 변경함으로써, 현재 구동 중인 하위 궤적 제어기 서브모듈(예: Auto-Mission 스레드의 State Machine)의 조건을 무효화(Invalidate)시킨다.
  3. 태스크 스위칭: 하위 스레드는 매 루프마다 nav_state의 변경을 확인(Polling)하며, 상태가 변경되었음을 감지하면 즉시 로컬 실행(Local Execution)을 return하거나 중단하여 RTL 내비게이션 태스크가 제어 루프를 점유하도록 양보(Yield)한다.

4. Ardupilot 대비 아키텍처 비교

Ardupilot은 전통적으로 단일 스레드 기반의 코퍼레이티브 멀티태스킹(Cooperative Multitasking) 구조에서 출발하여, 최근 ChibiOS를 포팅하면서 선점형 멀티태스킹(Preemptive Multitasking)을 부분 도입하기 시작하였다. 하지만 코어 비행 제어 루프(Fast Loop) 자체를 동기적으로 실행하는 비중이 매우 높다. 따라서 AP_Scheduler에 의해 정밀한 시간 단위로 슬라이싱되어 실행되므로, 락(Lock) 경합으로 인한 우선순위 역전에 상대적으로 적게 노출되는 구조를 추구해왔다.

반대로 PX4-Autopilot은 철저한 마이크로서비스(Microservices) 및 멀티스레드 기반 비동기 미들웨어(uORB) 구조를 지향한다. 따라서 OS 차원에서의 커널 스위칭 및 락 모델에 훨씬 강하게 의존한다. 이로인해 PX4는 복잡한 다중 연산(VIO 알고리즘과의 병렬 구동 등) 동시 처리와 컴팩트한 모듈화에 강세를 보이나, 설계 시 우선순위 상속 및 선점 알고리즘의 세밀한 튜닝이 필수 불가결한 아키텍처적 특성을 갖는다.
