45.9.1. QGroundControl 아키텍처 내 비행체 관리 클래스(Vehicle::setFlightMode) 분석

QGroundControl(이하 QGC)은 지상 관제 시스템(GCS, Ground Control Station)으로서 다양한 무인기 오픈소스 비행 제어기(Autopilot)를 추상화하여 관리하는 강력한 백엔드(Back-end) 아키텍처를 지니고 있다. 본 절에서는 QGC 애플리케이션 프레임워크 내에서 개별 무인기를 논리적으로 대변하는 Vehicle 클래스의 핵심 기능 중 하나인 setFlightMode 메서드를 낱낱이 해부한다. 특히 PX4 펌웨어(버전 1.14 기준)와 QGC(버전 4.3 기준)의 상호작용 속에서 이 메서드가 어떻게 동작하는지 심층적으로 다룬다.

1. Vehicle 클래스의 아키텍처적 위상

QGC의 C++ 백엔드 아키텍처에서 Vehicle 클래스는 무인 항공기(UAV, Unmanned Aerial Vehicle) 객체를 추상화한 최상위 논리 단위이다. 다중 기체(Multi-Vehicle) 관제 환경에서 QGC 매니저 클래스(예: MultiVehicleManager)는 연결된 MAVLink 시스템마다 고유한 Vehicle 인스턴스를 생성하여 관리한다.

Vehicle 클래스는 다음과 같은 주요 역할을 담당한다.

  • 상태 추적(State Tracking): 기체의 고도, 속도, 배터리 상태, 현재 비행 모드(Flight Mode) 등을 실시간으로 모니터링한다.
  • 컴포넌트 추상화(Component Abstraction): 펌웨어에 종속적인 속성(PX4, ArduPilot 등)을 FirmwarePlugin 클래스를 통해 추상화하여, 상위 프론트엔드(QML) 컴포넌트에는 동일한 형태의 인터페이스(Interface)를 제공한다.
  • 명령 수행(Command Execution): 사용자 또는 자율 에이전트로부터 전달받은 명령을 MAVLink 메시지 스택으로 변환하여 송출한다.

2. Vehicle::setFlightMode 메서드의 심층 분석

조종자나 자율 임무 시스템이 비행 모드를 변경하고자 할 때, QML 프론트엔드 계층에서 시작된 이벤트는 최종적으로 C++ 백엔드의 Vehicle::setFlightMode(const QString& flightMode)를 호출하게 된다.

이 메서드는 단순한 텍스트 문자열(String) 형태의 모드 식별자를 인자로 받아, 기체에 적합한 MAVLink 패킷으로 변환하고 전송을 보장하는 트랜잭션(Transaction) 과정을 수행한다.

2.1 커스텀 모드(Custom Mode) 매핑 전략

MAVLink 프로토콜 명세 내에서 기본 비행 모드(base_mode) 외에 제조사 또는 펌웨어 고유의 확장 모드는 custom_mode라는 32비트(bit) 비트필드(Bitfield)로 관리된다. PX4-Autopilot과 ArduPilot은 이 custom_mode를 해석하는 체계가 완전히 다르다.

  • PX4의 커스텀 모드: 메인 모드(Main Mode)와 서브 모드(Sub Mode)의 조합으로 구성된다. 예를 들어, ‘Position’ 모드나 ‘Offboard’ 모드는 지정된 비트 마스크에 맞게 구조화되어 있다.
  • ArduPilot의 커스텀 모드: 단순히 단일 정수값(예: Copter의 경우 0=Stabilize, 3=Auto 등)으로 열거형(Enumeration) 매핑을 사용한다.

Vehicle::setFlightMode는 직접 하드코딩된 모드 값을 전송하지 않는다. 대신 의존성 주입 체계에 따라 할당된 FirmwarePlugin::setFlightMode를 호출하여 다형성(Polymorphism)을 구현한다. 이를 통해 동일한 “Return” 명령이라도 PX4용 PX4FirmwarePlugin과 ArduPilot용 APMFirmwarePlugin에서 각각 알맞은 custom_mode 바이트 스트림(Byte Stream)으로 변환된다.

2.2 제어 명령 생성 및 전송 아키텍처

모드 전환 요청이 QGC에 입력되었을 때 MAVLink 프로토콜까지 도달하는 내부 흐름은 아래의 시퀀스 다이어그램(Sequence Diagram)과 같다.

sequenceDiagram
    participant QML UI as 프론트엔드 (QML)
    participant Vehicle as Vehicle 클래스 (C++)
    participant FirmwarePlugin as FirmwarePlugin (PX4)
    participant MAVLinkSys as MAVLink Protocol Stack
    participant PX4 as PX4-Autopilot (FCU)

    QML UI->>Vehicle: setFlightMode("Offboard")
    Vehicle->>FirmwarePlugin: setFlightMode("Offboard", &base_mode, &custom_mode)
    FirmwarePlugin-->>Vehicle: base_mode 및 custom_mode 반환
    Vehicle->>MAVLinkSys: sendMavCommand(MAV_CMD_DO_SET_MODE, base_mode, custom_mode, ...)
    MAVLinkSys-->>PX4: MAVLink COMMAND_LONG 패킷 전송
    PX4-->>MAVLinkSys: COMMAND_ACK (수신 확인)
    MAVLinkSys-->>Vehicle: ACK 처리 및 상태 업데이트
    Vehicle-->>QML UI: flightModeChanged() 시그널 발생(Signal Emission)
  1. 플러그인 델리게이션(Delegation): 텍스트 모드명(“Offboard”, “Position” 등)을 FirmwarePlugin의 가상 함수(Virtual Function)에 넘기고, 대응하는 base_mode와 custom_mode 스칼라 변수를 참조(Reference)로 받아온다.
  2. 명령 인코딩(Command Encoding): QGC 4.x 이상의 버전에서는 장명령(COMMAND_LONG) 방식 중 MAV_CMD_DO_SET_MODE를 사용하여 비행 모드를 설정한다. 추출된 모드 인자를 페이로드(Payload) 파라미터 영역에 적재한다.
  3. 트랜잭션 및 강건 제어(Robust Control): 단순히 메시지를 던지고 끝나는 것(Fire-and-forget)이 아니다. 패킷 손실률이 높은 무선 데이터 링크 환경에서 신뢰성을 확보하기 위해 내부적으로 VehicleCommandList 또는 전담 커맨드 관리 객체가 응답(ACK)을 기다린다.
  4. 피드백 검증(Verification): 설정된 Timeout 시간 이내에 COMMAND_ACK 메시지가 수신되면 성공으로 간주한다. 실패하거나 거부당한 경우 재전송(Retry) 로직을 거치거나, 사용자 UI에 오류 로그(예: “모드 전환 거부됨: 위치 센서 초기화 실패”)를 메타 오브젝트 시그널망을 통해 표출한다.

3. PX4와 ROS2 연동 관점에서의 setFlightMode 제어 차이점

QGC를 통해 사람의 손으로 직접 setFlightMode를 호출하는 방식과, 자율 에이전트인 ROS2 노드(Node)에서 모드를 전환하는 방식에는 다소 차이가 존재한다.

ROS2 환경(uXRCE-DDS 방식 연동)에서 Offboard 모드 기반 제어를 수행할 경우, MAVLink 프로토콜을 그대로 쓰기보다는 uORB vehicle_command 토픽(Topic) 메시지를 직접 발행(Publish)하는 방식이 선호된다.
QGC의 C++ 코드는 UART 나 UDP를 거치는 원격 단말기(Telemetry) 레이어의 관점이지만, ROS2 브릿지는 비행 제어기 보드의 미들웨어 수준 메시지로 우회하여 직접 개입하게 된다. 결과적으로 QGC에서 처리하는 Vehicle::setFlightMode의 동기화 검증(ACK 수신 대기) 로직과 유사한 상태 모니터링 체계를 ROS2 측에서도 별개의 타이머 콜백(Timer Callback) 등을 통해 직접 구현해 주어야 하는 부담을 개발자가 갖게 됨을 유의해야 한다.

4. 요약

QGroundControl의 Vehicle::setFlightMode 메서드는 단순한 속성(Property) 변경자(Setter)가 아니라, 멀티 플랫폼(Multi-platform) 지원 플러그인 아키텍처와 MAVLink 신뢰성 트랜잭션이 결합된 핵심 관제 파이프라인(Pipeline)이다. 이를 올바르게 이해함으로써 향후 맞춤형 GCS(Custom GCS) 구축이나 MAVSDK 기반 자동화 시스템을 설계할 때, 명령 전송 계층의 타임아웃 방어 로직과 다형성 설계 패턴을 손쉽게 이식할 수 있을 것이다.
