45.1.2 지상 관제 시스템(GCS)과 비행 제어기(FC) 간의 이산 사건 시스템(Discrete Event System) 동기화

현대의 무인 항공기(Unmanned Aerial Vehicle, UAV) 관제 환경에서 지상 관제 시스템(Ground Control Station, GCS)과 비행 제어기(Flight Controller, FC) 간의 통신은 본질적으로 이산 사건 시스템(Discrete Event System, DES)의 형태를 띤다. 본 절에서는 PX4-Autopilot(기준 펌웨어: v1.14.0 이상)과 QGroundControl(기준 버전: v4.3.0 이상)을 중심으로 두 시스템 간의 상태(State)가 어떻게 동기화(Synchronization)되고 일치성을 유지하는지 심도 있게 분석한다.

1. 이산 사건 시스템(Discrete Event System, DES)의 개념적 모델링

연속 시간 관점에서 제어되는 비행 동역학(Flight Dynamics) 시스템과 달리, GCS와 FC 사이의 상호작용은 MAVLink와 같은 통신 프로토콜을 기반으로 제한된 이산형 메시지를 주고받으며 이루어진다. 이처럼 시간이 아닌 특정 사건(Event)의 발생에 의해 시스템의 상태가 전이되는 제어 구조를 이산 사건 시스템(DES)이라 정의한다.

이를 수학적 상태 공간(State Space) 모델로 표현하면, GCS와 FC는 각각 자신만의 유한 상태 기계(Finite State Machine, FSM)를 구동하는 독립적 주체(Agent)이다. 두 주체는 MAVLink 데이터 링크라는 불안정한 비동기적 통신 채널을 통하여 동기화를 시도한다.

이때, 관할 시스템 전체의 동작은 오토마타(Automata) G 로 모델링할 수 있다.
G = (X, E, f, \Gamma, x_0, X_m)

  • X: 시스템이 가질 수 있는 유한한 상태의 집합 (예: DISARMED, ARMING, ARMED, TAKEOFF 등)
  • E: 상태 전이를 유발하는 유한한 사건의 집합 (예: 관제사의 Arming 명령 입력, 센서 캘리브레이션 완료 이벤트 등)
  • f(x, e): 상태 x에서 사건 e가 발생했을 때의 전이 함수 (Transition Function)
  • \Gamma(x): 상태 x에서 발생 가능한 유효 사건의 집합 (Active Event Set)
  • x_0: 초기 상태 (Initial State)
  • X_m: 목적 또는 정상 상태의 집합 (Marked States)

무선 통신 링크(RF Data Link)의 지연(Latency) 및 패킷 손실(Packet Loss)이 존재하므로, GCS가 인지하는 상태 집합 X_{GCS}와 FC가 실제 유지하는 상태 집합 X_{FC}는 특정 시점(Time t)에서 불일치(Mismatch)할 수 있다. 따라서 GCS 관제 소프트웨어 아키텍처는 이를 극복하기 위해 지속적인 상태 추정(State Estimation)과 이벤트 피드백 구조를 내장해야 한다.

2. 상태 불일치(State Mismatch) 및 비동기적 통신 한계점

비행 제어기 내부의 커맨더(Commander) 모듈은 현재 기체의 수많은 상태 변수(예: 자이로 센서 초기화 여부, GPS 3D Fix 여부, 배터리 전압 상태 등)를 초당 수백 회 이상 갱신하며 내부 전제 조건(Pre-conditions)을 검증한다. 반면, 데이터 대역폭(Bandwidth)의 한계로 인해 GCS는 내부 상태의 일부분만 간헐적으로 전달받는다. Ardupilot과 대비하여 PX4-Autopilot은 상태 변화 시 더 엄격한 내부 검증 과정을 거치며, GCS 로의 상태 보고 주기도 차이를 보인다.

만약 관제사가 QGroundControl을 통해 시동(Arming) 명령 트리거(Trigger)를 발생시키면, 이 이벤트는 COMMAND_LONG 메시지로 캡슐화되어 FC로 송신된다. 이때 발생할 수 있는 DES 관점의 예외 사항은 다음과 같다.

  1. 지시 누락(Command Omission): 통신 음영 지역으로 인해 COMMAND_LONG 메시지가 FC에 도달하지 못하여 X_{GCS}는 천이 대기 상태가 되나 X_{FC}는 이전 상태에 머무는 현상.
  2. 응답 누락(Acknowledgment Omission): FC가 명령을 수신하고 X_{FC}를 ARMED로 천이시켰으나, 결과인 COMMAND_ACK 피드백이 GCS에 도달하지 않아 X_{GCS}가 갱신되지 못하는 환상 오류(Phantom Error).
  3. 거부(Rejection): FC가 명령을 수신하였으나, 내부 사전 조건(예: GPS 정밀도 미달)을 만족하지 못해 명령 수행을 거부하고, COMMAND_ACK의 반환 값으로 거절 사유(예: MAV_RESULT_TEMPORARILY_REJECTED)를 보내는 경우.

3. PX4 및 QGC의 상태 동기화 아키텍처 (State Synchronization Architecture)

3.1. 하트비트(HEARTBEAT) 기반 주기적 폴링(Periodic Polling)

상태 불일치 문제를 해결하는 가장 기초적이고 강력한 기법은 주기적인 상태 브로드캐스트(Broadcast)이다. PX4-Autopilot은 흔히 1Hz 주기로 HEARTBEAT 메시지를 발행(Publish)한다. 이 메시지는 시스템의 현재 기본 모드(Base Mode)와 사용자 정의 모드(Custom Mode), 그리고 시스템 상태(System Status)를 담고 있다.

GCS는 특정한 명령어(COMMAND_LONG) 전송 직후 응답(COMMAND_ACK)에 의한 즉각적인 상태 천이를 기대하지만, 만약 COMMAND_ACK가 소실되더라도 후속 수신되는 HEARTBEAT 메시지를 파싱하여 기체가 실제로 시동이 걸렸는지(MAV_MODE_FLAG_SAFETY_ARMED), 비행 모드가 변경되었는지 판별하여 상태 동기화를 복구한다. 즉, HEARTBEAT 메시지는 DES 모델에서 관측자(Observer)가 FC의 실제 상태 공간 X_{FC}를 확인하는 보증된 관측 주기(Guaranteed Observation Horizon)를 제공한다.

3.2. 명령-응답 트랜잭션(Command-Response Transaction) 구조

QGroundControl 측의 C++ 백엔드(Backend) 소스 코드(예: Vehicle 매니저 객체)를 분석해보면, GCS는 명령 전송에 대하여 타임아웃(Timeout) 기반의 재전송(Retry) 상태 머신을 병렬로 운용한다. Ardupilot 계열의 Mission Planner보다 QGC는 비동기 이벤트 큐와 시그널 슬롯 구조에 크게 의존하여 이러한 트랜잭션을 관리한다.

  1. QGC는 FC에 상태 변경 이벤트를 발생시킬 때 전역 큐(Global Queue)에 해당 커맨드를 등록한다.
  2. 커맨드가 발송됨과 동시에 로컬 타이머를 작동시킨다.
  3. 정해진 기한(예: 500ms) 내에 COMMAND_ACK를 수신하면 트랜잭션을 성공으로 처리하고 타이머를 해제한다.
  4. 기한이 초과(Timeout)되면 지정된 최대 횟수만큼 동일 명령을 재발송한다.
  5. 여기서 중요한 점은 MAVLink의 시동 및 모드 변경 명령들은 **멱등성(Idempotence)**을 가지도록 설계되어야 한다는 점이다. 만약 FC가 이미 ARMED 상태일 때 ARM 명령이 중복 도달하면, 시스템에 치명적인 영향을 주지 않고 “이미 승인되었음(MAV_RESULT_ACCEPTED)“을 반환해야 이산 사건 시스템의 안정성을 해치지 않는다.

5. 설계 시 고려사항 및 강건 제어(Robust Control) 로직 구현

자율 비행 제어 아키텍처 및 GCS 개발 시 이산 사건 시스템 모델을 기반으로 강건한 관제 시스템을 개발하기 위해 다음의 철학을 엄격히 적용해야 한다.

  • 낙관적 상태 전이 지양: GCS 측 코드를 작성할 때, 제어 명령을 보냈다는 즉발적 사실만으로 GCS 내부의 모델 상태를 즉시 목표 상태(Target State)로 갱신하지 마라. GCS의 UI 및 내부 변수는 오직 FC로부터 직접 발송된 명시적 피드백(ACK) 또는 상태 알림 메시지(HEARTBEAT, EXTENDED_SYS_STATE 등)를 확인한 진리(Ground Truth)에 입각하여 상태를 변경시켜야 한다.
  • 불확실성 내성(Uncertainty Tolerance): GCS의 FSM은 과도 상태(Transient State, 예: ARMING_REQUESTED 동작 대기 등)를 반드시 정의하여, 무한 루프 상태에 돌입하는 문제를 차단하라. 유한 상태 기계의 설계 결함은 교착 상태(Deadlock)를 초래하여 기동 불능의 원인이 된다.
  • 예외적 상태 반전 방지: QGroundControl 소스는 사용자 및 FC 데이터에 따른 경합 조건(Race Condition)을 방어하기 위해 객체의 Mutex 메커니즘을 적극적으로 활용한다. GCS 내 상태 제어 스레드와 MAVLink 파싱 인터페이스 간 데이터 동시 접근에 대한 제반 처리를 철저히 구성하라.

6. 결론 및 학술적 접근 한계

PX4 기반의 비행 시스템은 MAVLink 프로토콜의 설계 철학을 바탕으로 비동기 환경에서의 이산 사건 시스템 동기화를 훌륭하게 수행하고 있다. 그러나 군집 비행(Swarm Flight)이나 극단적 통신 지연이 존재하는 심우주(Deep Space) 및 위성 통신 링크(BVLOS) 관제 조건에서는 기본적 타임아웃 기반 응답 수신 구조만으로는 복잡한 임무의 정합성을 담보하기 모호하다. 따라서 최근 연구에서는 이러한 이산 사건 제어 시스템을 추계적 펫리넷(Stochastic Petri Nets)이나 감독 통제 이론(Supervisory Control Theory)을 이용해 확장하려는 시도가 지속되고 있다.
