45.8.3 임계 결함(Critical Fault) 감지 시 자동 복귀(RTL) 제어기 강제 개입 역학

무인항공기(UAV) 운영 중 센서 고장, 배터리 임계 전압 도달, 지상 관제 시스템(GCS)과의 통신 단절(Data Link Loss) 및 RC(Radio Control) 조종 신호 상실과 같은 임계 결함(Critical Fault)이 발생할 경우, 비행 제어기(Flight Controller)는 기체와 주변 환경의 안전을 위해 운영자의 명령을 오버라이드(Override)하고 자동 복귀(Return To Launch, RTL 혹은 Return) 제어기를 강제로 개입시켜야 한다.

PX4-Autopilot(Firmware v1.14 이상)은 이러한 비상 상황을 다루기 위해 매우 정교한 안전 장치(Failsafe) 상태 머신(State Machine)과 커맨더(Commander) 모듈 아키텍처를 내장하고 있다. 본 절에서는 임계 결함 감지부터 RTL 제어기가 강제로 권한을 탈취하여(Preemption) 기체를 복귀시키는 역학적 흐름과 그 소스 코드 구조를 살펴본다.

1. 임계 결함(Critical Fault)의 정의와 종류

PX4 시스템 내에서 커맨더 모듈(commander.cpp)은 각종 모듈(센서, 배터리 관리자, Datalink 모니터 등)로부터 uORB 메시지를 구독하여 시스템의 전반적인 건강 상태(Health Status)를 실시간으로 모니터링한다. RTL을 강제하는 주요 임계 결함은 다음과 같다.

  1. 원격 측정 및 조종 신호 상실: GCS와의 MAVLink 통신(Heartbeat) 단절(Target Timeout), 조종기(Radio/RC) 신호 일시적 또는 영구적 상실.
  2. 배터리 임계 수준(Critical Battery): 배터리 잔량이나 셀 전압이 설정된 Failsafe 임계값(예: BAT_CRIT_THR) 미만으로 떨어졌을 때.
  3. 위치 추정(Estimator) 편차: 외부 교란이나 GPS 손실로 인해 EKF2(Extended Kalman Filter)의 위치 추정 분산이 허용 오차를 초과할 때.
  4. 지오펜스(Geofence) 이탈: 기체가 허용된 비행 영역을 벗어났을 때.

이러한 결함이 감지되면 시스템은 즉각 상태 공간 궤적에서 ‘비정상(Abnormal)’ 상태로 전환하며, 설정된 페일세이프 정책(Failsafe Action Parameter, 예: NAV_DLL_ACT, NAV_RCL_ACT)에 따라 대응한다. 대부분의 임계 상황에서 가장 널리 쓰이는 기본 방어 기재는 RTL(Return)이다.

2. 자동 복귀(RTL) 제어기 강제 개입 프로세스

임계 결함이 발생하면 커맨더 모듈은 현재의 비행 모드(예: Auto-Mission, Offboard, Position 등)를 즉각 중단하고, Navigator 모듈 내의 RTL 상태 머신을 최우선 순위로 활성화(Activation)한다.

2.1 이벤트 파이프라인과 상태 전환 흐름

  1. 결함 감지 (Fault Detection): 커맨더 모듈이 vehicle_status 및 각종 센서 uORB 토픽의 에러 플래그를 감지한다.
  2. 모드 강제 전환 요청 (Force Mode Switch): 커맨더가 현재 내비게이터 태스크를 선점(Preempt)하고 내부적으로 vehicle_command를 발생시켜 RTL 모드로 시스템 상태를 전이(Transition)시킨다.
  3. RTL 상태 머신 활성화 (Navigator - RTL): Navigator 서브모듈 중 하나인 RTL 객체가 실행되며, 사전에 기록된 홈 위치(Home Position) 혹은 랠리 포인트(Rally Point)를 목적지로 설정한다.
  4. 비행 제어 스택 인가 (Flight Control Stack): Position Controller 레이어에서 기존 궤적 추종(Trajectory Tracking) 명령을 폐기하고, RTL 서브모듈이 생성하는 회피 및 상승/이동 궤적(Setpoints)을 추종하기 시작한다.

3. 소스 코드 레벨 분석: Commander 모듈의 개입 역학

PX4의 src/modules/commander/Commander.cpp 내부를 살펴보면, 주기적인 루프 안에서 상태 검사를 수행하는 로직을 확인할 수 있다. 임계 상황 발생 시 사용자나 GCS의 일반 명령어 입력 처리를 잠그고(Lock), 강제 상태 변이를 트리거하는 코드가 존재한다.

// PX4 Commander 모듈 내 RTL 강제 개입 역학 예시

void Commander::handle_failsafe(vehicle_status_s &status)
{
    // 통신 단절(Data Link Loss) 감지 로직
    if (_datalink_loss_detected && status.nav_state != vehicle_status_s::NAVIGATION_STATE_AUTO_RTL) {
        
        // 파라미터 정책에 따라 RTL 액션 트리거
        if (_param_nav_dll_act.get() == FAILSAFE_ACTION_RTL) {
            PX4_WARN("Failsafe: Datalink lost, forcing RTL");
            
            // 네비게이션 상태를 RTL로 강제 오버라이드 (Preemption)
            status.nav_state = vehicle_status_s::NAVIGATION_STATE_AUTO_RTL;
            status.failsafe = true;
            
            // uORB에 변경된 상태를 발행하여 하위 제어 체계 트리거
            _vehicle_status_pub.publish(status);
        }
    }
    // 배터리 임계 및 기타 결함 생략...
}

이와 같이 커맨더 스레드는 MAVLink 명령어 수신부(MAVLink Receiver)나 조종기 입력보다 상위의 논리적 우선순위를 가지며, 결함이 해소되거나 사용자가 의도적으로 개입하여 비상 권한을 회수(RC Override)하기 전까지 기체의 통제권을 Navigator 서브계층(RTL 모듈)에 위임한다.

4. Ardupilot 대비 Failsafe 및 RTL 제어 차이점

Ardupilot은 전통적으로 비행 코드루프(Flight Code Loop) 내부에서 페일세이프 인터럽트를 직접 검사하여 서브루틴(AP_Vehicle::failsafe_foo()) 단위로 분기시키는 방식을 취한다. 또한 Ardupilot은 스마트 RTL(SmartRTL)이라는 기능을 제공하여 비행 중 기록된 경로를 역추적하는 고도화된 스택을 기본 포함하고 있다.

반면, PX4-Autopilot은 미들웨어 중심의 아키텍처 특성상 커맨더 모듈이 독립된 마이크로 태스크(Micro Task)로서 백그라운드에서 동작하며 uORB 메시지 지향적으로 상태를 제어한다. 이는 소프트웨어 컴포넌트 간 결합도(Coupling)를 낮추어, RTL 모듈이 커맨더의 로직과 독립적으로 분리 개발(Decoupled Development)될 수 있도록 도와준다. 최근 PX4에도 Return type 파라터를 통해 역추적(Reverse trajectory)이나 근접 랠리 포인트(Rally point)로의 복귀 등 다양한 복귀 로직 역학을 선택할 수 있는 유연한 아키텍처를 제공하고 있다.
