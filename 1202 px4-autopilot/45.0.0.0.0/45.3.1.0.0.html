45.3.1 레거시 단일 명령(SET_MODE)의 페이로드 대역폭 병목(Bandwidth Bottleneck) 현상과 프로토콜 확장 메커니즘

비행 제어 시스템의 발전 과정에서 통신 트래픽의 효율화는 지속적으로 요구되어온 주요 과제 중 하나이다. 드론의 기능이 다양화됨에 따라 지상 관제 시스템(GCS)과 기체 간의 데이터를 주고받기 위한 명령 대역폭 요구량도 급증했다. 본 절에서는 과거에 사용되던 레거시(Legacy) 단일 명령 체계인 SET_MODE 메시지의 구조적 한계를 분석하고, 이를 극복하기 위해 PX4-Autopilot 아키텍처에 도입된 프로토콜 확장 메커니즘(Protocol Extension Mechanism)에 대해 다룬다.

(설명 기준: PX4-Autopilot v1.14.0, QGroundControl v4.3.0, MAVLink 2 표준)

1. 레거시 SET_MODE 메시지의 구조적 한계

초기 MAVLink 프로토콜(MAVLink 1) 시절, 비행 모드를 설정하기 위한 전용 메시지로 SET_MODE (#11)가 고안되었다. 이 메시지는 다음과 같은 단순한 구조를 가지고 있었다.

  • target_system (1 byte)
  • base_mode (1 byte): 시스템의 기초 상태 (예: Arming 상태 유무 등)
  • custom_mode (4 bytes): 오토파일럿(운영체제)별로 정의할 수 있는 사용자 정의 모드 (예: PX4의 Hold 또는 Mission 모드 플래그)

SET_MODE 방식은 패킷 크기가 작아 단순한 비행 모드 전환에는 직관적으로 보이나, 다음과 같은 심각한 대역폭 병목(Bandwidth Bottleneck) 현상을 초래했다.

  1. 컨텍스트(Context) 전환의 한계
    시스템의 상태를 단편적인 custom_mode 숫자 하나에 구겨 넣어야 했기 때문에, 점차 세분화되는 하위 모드(Sub-mode)를 명시하기에 비트(Bit) 공간이 턱없이 부족해졌다.
  2. 다중 명령 체계 유지 비용
    복잡한 무인기의 임무 수행(예: “위치 A로 이동 후 고도를 10m 내리고 카메라 트리거를 수행하는 특정 미션 모드”)을 명령하기 위해서는 모드 전환 패킷(SET_MODE)과 제어 설정 패킷들을 각각 별개의 트랜잭션으로 연달아 전송해야 했다. 이는 곧 데이터 링크의 혼잡을 유발하고 수신 측에서의 트랜잭션 동기화 실패 우려를 키웠다.
  3. 확장성 결여
    기체가 VTOL(수직이착륙기) 등 복합적인 운용 양태를 보이게 되면서 단일 32비트 포맷(custom_mode)으로는 모조리 매핑(Mapping)할 수 없었다.

2. 대역폭 병목 타개를 위한 프로토콜 확장 메커니즘

MAVLink 프로토콜은 SET_MODE라는 단일 전용 메시지를 점진적으로 폐기(Deprecated) 수준으로 낮추고, 모든 명령을 규격화된 범용 컨테이너에 담아 전송하는 COMMAND_LONG 프로토콜 기반의 확장 메커니즘을 채택했다. 이를 통해 비행 모드 설정은 MAV_CMD_DO_SET_MODE라는 통합 열거형(Enum) ID를 지닌 명령의 형태로 전송된다.

2.1 COMMAND_LONG 및 MAV_CMD_DO_SET_MODE의 도입

새로운 확장 메커니즘 체계하에서는 7개의 부동소수점 파라미터 배열(param1 ~ param7)을 지원하는 넓은 메모리 풀을 활용할 수 있다. MAV_CMD_DO_SET_MODE 명령을 수신한 PX4의 수신기는 다음과 같이 파라미터를 파싱한다.

  • param1: base_mode (또는 MAV_MODE 플래그)
  • param2: custom_mode (PX4의 메인 모드 식별자, 예: Auto 모드)
  • param3: custom_sub_mode (세부 하위 모드, 예: Auto-Mission 또는 Auto-RTL)

여러 패킷으로 분할해서 전송했어야 할 “단일 기능의 하위(Sub) 파라미터 설정” 트랜잭션들을 1개의 확장형 데이터 패킷 속으로 병합할 수 있게 되면서, 통신의 헤더 오버헤드를 줄이고 데이터 대역폭 효율성을 극대화했다.

3. ArduPilot과 PX4에서의 페이로드 처리 패러다임 차이

  • ArduPilot의 모드 처리 방식
    ArduPilot은 여전히 전통적인 운영 환경과의 역호환성(Backward Compatibility)을 강력하게 유지하려는 경향이 있어, 수신기가 직렬 인터페이스 구조상에서 레거시 모드 지정 방식과 최신 방식을 거의 동일선상의 우선순위로 두고 파싱하는 코드가 넓게 분포해 있다. 이는 소스 코드가 다소 분산되는 특징을 가진다.
  • PX4-Autopilot의 모드 처리 방식
    PX4-Autopilot은 미들웨어 중심 구조(uORB 기반)의 이점(장점)을 극대화하기 위하여, GCS로부터 직렬화(Serialized)된 레거시 명령이 들어올 경우 MAVLink Receiver 모듈 내에서 최신 메시지 구조 체계로 캐스팅(Casting) 및 상향 변환(Up-conversion)을 하여 내부 트랜잭션을 일원화한다.
    따라서 vehicle_command uORB 토픽에는 언제나 동일한 정규화(Normalized)된 규격만이 실려 커맨더 모듈로 전달되므로, 코어 비행 제어 스택에서는 레거시 메시지의 존재를 인지할 필요도 없게 처리하여 시스템 복잡도를 크게 낮춘다.

4. 요약

레거시 단일 단위 메시지 지정 방식(SET_MODE)의 가장 큰 패착은 ’비행 모드’라는 중요한 상태 머신 전이를 숫자 몇 개로 정의하려 했던 경직성에서 기인했다. 결과적으로 패킷 플래그먼테이션(Fragmentation)이 빈번해지며 데이터 통신망에서 페이로드 대역폭 병목 현상이 일어났으나, PX4-Autopilot은 MAVLink 프로토콜의 확장 메커니즘인 COMMAND_LONG을 채용하고 내부적으로 uORB 레이어를 통해 일원화 전송함으로써 효율적이며 안정적인 무인기 상태 제어를 완성하였다.
