45.4.2. 액추에이터 제어 주기(Control Loop) 활성화 상태 및 기하학적 비행 궤적 곡률(Curvature) 제한에 따른 운동학적 천이 제약

1. 개요 및 이론적 배경

고도화된 자율 비행 제어 시스템인 PX4-Autopilot은 비행 모드(Flight Mode)를 천이(Transition)할 때, 단순한 내부 상태 기계(State Machine)의 논리적 조건만을 살피지 않는다. 고수준(High-level)의 궤적 제어기와 저수준(Low-level)의 액추에이터 속도/자세 제어기가 원활히 맞물려 돌아가기 위해 액추에이터 제어 주기(Control Loop)의 포화 및 활성화 상태를 점검한다. 또한 고속 비행 중에 갑작스럽게 임무(Mission) 모드에서 정지(Hold) 모드로 천이하거나 웨이포인트를 급격하게 선회할 경우, 기체의 물리적 한계점(최대 가속도 및 저크)을 넘지 않도록 **비행 궤적 곡률(Curvature) 제한 기반의 운동학적 제약(Kinematic Constraints)**을 부과한다.

본 절에서는 이러한 PX4 내부의 운동학적 천이 제약 파이프라인, 위치 제어기(Position Controller) 단의 한계 검사 로직, 그리고 기하학적 궤도 곡률의 처리 메커니즘을 소스 코드(펌웨어 1.14+ 기준) 관점에서 깊이 있게 해부한다.

2. 제어 주기(Control Loop) 활성화 역학 및 플래그 검증

기체는 모터, 서보 등 액추에이터의 실시간 한계(예: PWM Saturation)에 직면할 수 있다. 커맨더(Commander)가 비행 모드를 전환하려면 하위 제어기들이 전환으로 발생할 급격한 델타 제어 입력(Control Effort Delta)을 수용할 상태인지 확인해야 한다.

2.1 vehicle_control_mode 토픽을 통한 권한 활성화

PX4에서 각 모드는 자신이 점유할 제어 루프의 권한을 vehicle_control_mode uORB 메시지로 퍼블리싱(Publishing)한다.

  • flag_control_manual_enabled
  • flag_control_attitude_enabled
  • flag_control_velocity_enabled
  • flag_control_position_enabled

상위 모드(예: Auto, Mission)로 천이할 경우, 커맨더는 EKF 에스티메이터가 출력하는 속도/위치 유효성(v_z_valid 등)뿐 아니라, 현재 액추에이터 할당기(Control Allocator) 및 모터 제어 루프가 락(Lock)에 걸려있지 않고 정상 틱(Tick)으로 활성화되어 있는지 상태 점검을 수행한다.

3. 기하학적 비행 궤적 곡률(Curvature) 및 운동학적 제약 이론

비행 중 기체가 곡선 궤적을 비행할 때 발생하는 구심 가속도(Centripetal Acceleration) a_c는 속도 v와 궤도 곡률 반경 R에 의해 결정된다.

a_c = \frac{v^2}{R} = v^2 \cdot \kappa

여기서 \kappa는 궤적 곡률(Curvature)이다. 기체가 발생시킬 수 있는 최대 수평 가속도는 기체의 최대 틸트각(Tilt Angle)에 의해 물리적인 한계 a_{max} = g \cdot \tan(\theta_{max}) (단, g는 중력가속도) 를 가지게 된다.

3.1 위치 제어기(mc_pos_control) 내 저크 통제 궤적(Jerk-limited Trajectory)

멀티로터의 위치 제어 모듈에서는 급격한 웨이포인트 이동 명령 시 곡률 제한을 준수하기 위해 저크 통제 궤적 생성기(Jerk-limited Trajectory Generator)를 가동한다.
목표 지점이 급격히 꺾인다면, 제어 루프는 최대 허용 가속도(MPC_ACC_HOR_MAX) 및 최대 저크(MPC_JERK_MAX) 이내에서 구심 가속도가 수용 가능한 곡률 R_{min}을 확보할 수 있도록 진입 속도 v를 능동적으로 감속시킨다.

3.2 고정익 L1 Navigation (fw_pos_control_l1) 내의 곡률 제한

고정익의 경우 선회 반경은 양력과 직접적으로 결부되므로 한계 제약이 훨씬 엄격하다. L1 내비게이션 알고리즘은 목표 거리 L_1에 대해 목표 가속도 a_{s_{cmd}}를 다음 수식으로 도출한다.

a_{s_{cmd}} = 2 \frac{V^2}{L_1} \sin(\eta)
(여기서 \eta는 경로 이탈각)

PX4의 fw_pos_control_l1 모듈은 선회 시의 롤(Roll) 각도를 파라미터 FW_R_LIM으로 캘리브레이션하며, 이에 따라 최대로 그릴 수 있는 궤적 곡률(Maximum Curvature)이 하드코드되거나 동적으로 계산되어 비행 궤도가 천이하는 것을 억제한다.

5. PX4 파라미터 구조체 및 설계 로직 분석

이와 같은 궤적 기반의 운동학 제약은 엔지니어들이 설정하는 핵심 파라미터들을 바탕으로 이루어진다.

  • MPC_XY_CRUISE / MPC_Z_VEL_MAX_DN: 천이 단계로 들어가기 전 허용되는 기본 관성 순항 속도.
  • MPC_JERK_MAX: 기체가 급감속하거나 천이 명령을 수행할 때 운동 방정식의 미분(Jerk) 허용치를 제한하여 부드러운 전환형 곡선을 유도한다.
  • MPC_ACC_HOR / MPC_ACC_UP_MAX: 수평 방향 가속도 한계를 한정하여 선회 궤도의 최소 곡률 반경(Minimum Turn Radius) 물리 엔진 리미터를 건다.

만일 Hold 모드로 스위치(전환) 시 진입 속력이 한계를 초과하면, PX4 위치 제어기 내의 FlightTaskAutoMapper 클래스 내 로직은 현재 속력 벡터에서 제어 한계(MPC_ACC_HOR)로 멈출 수 있는 정지점(Stopping Point)을 새 목표점(Setpoint)으로 재할당하여 액추에이터의 응답 지연이나 제어 주기 포화를 사전에 예방한다.

6. Ardupilot (SCurve Navigation) 대비 아키텍처 차이

Ardupilot 단에는 Leonard Hall이 주도하여 개발한 독자적인 Kinematic SCurve 네비게이션 컨트롤러 구조가 존재한다.

  • PX4-Autopilot의 저크 제한 생성기(Jerk-limited Trajectory Generator)는 주로 위치 스텝(Position Step) 명령에 대한 연속적인 궤적 변환 최적화에 특화되어 있으며, 각 축 파라미터를 비교적 간결하게 구성한다.
  • Ardupilot의 SCurve 로직은 웨이포인트(WP) 구간 통과(Cornering) 시의 최대 곡률 제약 처리에 있어 WPNAV_RADIUS, WPNAV_ACCEL 등을 코너 단위로 매우 세분화하여 수학적 프로파일을 명확하게 파티셔닝(Partitioning)한다.
    따라서, 가상의 운동학적 한계 도달 시 PX4는 모드 천이 전 상태 트랙의 ’연장 및 스무딩(Smoothing)’에 조금 더 주력하는 설계인 반면, Ardupilot은 다소 보수적인 제동(Brake) 팩터와 함께 사전 설정 경로를 융통성 있게 베어내는 데 초점이 맞추어져 있다.

7. MAVLink을 활용한 곡률 추적 및 QGroundControl 관제

GCS(예: QGroundControl)에서는 기체가 운동학적 제어 천이에 부하를 겪고 있는지 실시간으로 모니터링할 수 있다.

  • POSITION_TARGET_LOCAL_NED (ID: 85): 기체가 당장 목표로 삼은 위치, 속도, 가속도 타겟팅 벡터. 만약 천이 발생 시점에서 가속도 타겟(afx, afy)이 지속적으로 파라미터의 한계값에 포화(Saturation)되어 있다면 곡률 제약을 겪고 있는 상태다.
  • NAV_CONTROLLER_OUTPUT (ID: 62): nav_roll, nav_pitch, crosstrack_error 등의 지표. 곡률 제약을 초과하여 선회를 완만하게 도는 경우 Crosstrack Error가 눈에 띄게 큰 폭으로 증가한다.

GCS 운영자는 꺾임각이 큰 고속 비행 임무일 경우, 이러한 관제 텔레메트리 값을 바탕으로 코너링 구간 전 진입 속력을 낮추는 추가 임무 명령(Speed Change)을 사전 삽입할 필요가 있다.

8. ROS2 환경에서의 자율 제약 및 Offboard 모드 응용

ROS2를 이용해 자율 에이전트(Companion Computer)에서 비행 경로 시퀀스를 제어할 때, Offboard 모드에서는 기체의 내부 궤적 생성기를 우회(Bypass)할 위험이 존재한다.

ROS2 노드 측에서 uXRCE-DDS 브리지를 통해 TrajectorySetpoint uORB 토픽을 발행할 때, 속도 공간이나 위치 공간의 지시 포인트의 거리가 멀면 기체 제어기가 자체적인 저크와 곡률 제한을 바탕으로 따라오지 못하고 추종 오차 역학(Tracking Error Dynamics)에 빠질 수 있다.

# ROS2 Python TrajectorySetpoint 명령 제약 부여 예시
from px4_msgs.msg import TrajectorySetpoint
import math

# 최대 허용 곡률을 초과하지 않는 스무스한 경로 생성 로직 필요
# a_c = v^2 * kappa 조건 검증
def check_kinematic_constraints(current_vel, next_waypoint, dt):
    required_accel = get_required_accel(current_vel, next_waypoint, dt)
    max_accel = 5.0 # m/s^2 (MPC_ACC_HOR 파라미터 기반 추정)
    
    if required_accel > max_accel:
         # 운동학적 안전 곡선(Spline 등)으로 곡률 리샘플링 생성
         return generate_smoothed_spline(current_vel, next_waypoint, max_accel)
    return next_waypoint

이와 같이 높은 수준의 자율 비행에서는 ROS2 경로 계획(Path Planning) 알고리즘 단에서 이미 기체의 제어기 활성화 역학과 최대 곡선 가속도(Curvature Acceleration) 한계를 계산에 넣어 Setpoint를 하달(Down-streaming)하는 설계가 가장 완벽한 관제 아키텍처로 인정받는다.
