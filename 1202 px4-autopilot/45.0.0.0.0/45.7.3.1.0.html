45.7.3.1 GCS 내부 소프트웨어 워치독 타이머(Watchdog Timer) 인터럽트 발생 및 안전 복구(Fallback) 유한 상태 기계로의 강제 천이 메커니즘

본 절에서는 MAVLink 비동기 통신 중 특정 응답이 영구적으로 상실되어 복구 불능 상태에 이르렀을 때, 지상 관제 시스템(QGroundControl, 이하 QGC)의 내부 시스템 무결성을 유지하기 위한 소프트웨어 워치독 타이머(Software Watchdog Timer)의 동작 원리와, 예외 발생 시 시스템이 안전 복구(Fallback) 유한 상태 기계(Finite State Machine, FSM)로 강제 천이하는 메커니즘에 대해 면밀히 고찰한다. 본 내용은 QGC 버전 v4.3 기준의 소스 코드 아키텍처를 토대로 서술한다.

1. 개요: 소프트웨어 워치독 타이머의 필요성

일반적으로 하드웨어 워치독 타이머(Hardware WDT)는 임베디드 장치에서 CPU가 무한 루프에 빠지는 등 데드락(Deadlock)에 직면했을 때 시스템을 하드웨어적으로 재부팅하는 데 사용된다. 이와 달리 GCS 단에 구현되는 소프트웨어 워치독 타이머는 UI 프론트엔드나 백엔드 스레드가 특정 명령어(COMMAND_LONG 등) 전송 후 무기한 대기(Infinite Wait) 상태에 빠지는 것을 방지하는 애플리케이션 계층의 생존성 메커니즘이다.

GCS가 기체의 치명적인 결함이나 통신 두절로 인해 영구적으로 ACK를 받지 못할 때, 계속해서 재전송(Retransmission) 스케줄링 루트를 따라가면 GCS의 명령 큐(Command Queue)가 가득 차고 UI 스레드가 블로킹(Blocking)되는 치명적인 오류가 발생할 수 있다. QGC 백엔드에서는 이를 예방하기 위해 워치독 데몬 타이머를 가동하여, 설정된 한계 시간이 지나면 C++ 이벤트 인터럽트를 발생시키고 시스템을 즉시 통제 가능한 상태로 강제 복구(Fallback)시킨다. 이는 “Design and Implementation of A Reliable Unmanned Aerial Vehicle Ground Control Station” 등 여러 논문에서 제시하는 결함 허용(Fault-tolerant) GCS 아키텍처의 필수 구성 요소이다.

2. QGroundControl 백엔드의 워치독(Watchdog) 설계 아키텍처

QGroundControl은 Qt 프레임워크 기반의 비동기 이벤트 루프(Event Loop) 아키텍처를 채택하고 있다. Vehicle 객체가 생성될 때, 내부적으로 명령어 재전송 타이머뿐 아니라 트랜잭션 전반의 무효화를 감시하는 QTimer 기반의 소프트웨어 워치독이 통합된다.

  1. 타이머 활성화 (Timer Activation)
    운영자가 비행 모드 전환(Flight Mode Change)이나 센서 캘리브레이션 개시 명령을 내릴 때, 큐(Queue)에 명령어가 적재됨과 동시에 최상위 워치독 타이머가 기동된다.

  2. 이벤트 루프 기반 인터럽트 (Event-loop based Interrupt)
    해당 트랜잭션의 총 소요 시간이 워치독 임계치(예: 3,000ms 등, 명령어 종류에 따라 동적 할당됨)를 초과하게 되면, Qt의 시그널-슬롯(Signal-Slot) 체계를 통해 워치독 만료 인터럽트 시그널(Timeout Signal)이 방출된다.

3. 안전 복구(Fallback) 유한 상태 기계로의 강제 천이

워치독 시그널이 발생하면, 현재 진행 중이던 명령어 송신 FSM은 즉시 파기되며 GCS 내부의 상태는 **안전 복구 유한 상태 기계(Fallback FSM)**로 강제 천이(Forced Transition)하게 된다.

stateDiagram-v2
    direction TB
    [*] --> Idle
    Idle --> Command_Sent : 사용자 명령 발생
    
    state Command_Sent {
        [*] --> Wait_For_Ack
        Wait_For_Ack --> Retry : ACK 타임아웃
        Retry --> Wait_For_Ack
    }
    
    Command_Sent --> Success : 정상 ACK 수신
    Command_Sent --> Fallback_FSM : 워치독 타이머 인터럽트 발생
    
    state Fallback_FSM {
        [*] --> Queue_Flush : 진행 중인 큐 삭제
        Queue_Flush --> Disconnect_State : 통신 객체 일시 정지/초기화
        Disconnect_State --> Notify_UI : 운영자 예외 경고
    }
    
    Success --> Idle
    Fallback_FSM --> Idle : 복구 처리 완료 후 UI 대기

상황이 복구 상태로 강제 천이될 때, GCS는 다음 세 가지 조치를 순차적으로 실행한다.

  • 큐 초기화(Queue Flushing): 후속된 제어 명령을 모두 버려(Drop) 오동작 누적을 방지한다.
  • UI 동기화 단절 예외 처리: QML 프론트엔드로 실패 시그널을 전파하여 기체 상태 패널(Vehicle Status Panel)이 ’비정상(Unknown) 혹은 오류 상태’임을 나타내도록 렌더링을 갱신한다.
  • 재연결 대기 및 재초기화: MAVLink 링크 레이어를 명시적으로 재초기화하여 소켓 버퍼(Socket Buffer) 고갈을 차단한다.

4. Ardupilot 대비 PX4 GCS 환경의 예외 처리 철학

QGroundControl(PX4 주력 GCS)과 Mission Planner(Ardupilot 주력 GCS)는 예외 처리에서 다른 소프트웨어 공학적 관점을 갖는다.

비교 항목QGroundControl (PX4 환경)Mission Planner (Ardupilot 환경)
워치독 인터럽트 구조QTimer 기반의 비동기적(Asynchronous) 이벤트 루프 인터럽트를 통해 워치독을 처리. 스레드 프리징(Freezing) 가능성이 극히 낮음.C# 환경의 폼(Form) 스레드와 백그라운드 스레드 간의 타이머 위임(Delegation)을 통해 처리. 상황에 따라 순간적인 UI 렉(Lag)이 발생할 소지가 존재함.
Fallback 천이 강도실패 요인이 감지되면 해당 컴포넌트(예: 카메라 설정, 비행 모드 설정)의 트랜잭션을 매우 공격적으로 소거하고 즉시 기본(Idle) 폴백 상태로 귀환함.재시도 및 프로시저 복구 기회를 가능한 충분히 부여하며, 사용자가 직접 수동으로 취소(Cancel)하기 직전까지 대기하는 경향이 강함.
연관 명령어 종속성(Dependency) 제거워치독이 발동하면 종속성 있는 큐(Cascade Commands) 전체를 한꺼번에 연쇄 삭제하여 잠재적 치명 결함을 제거함.오류의 범위를 국소화(Localization)하여, 실패한 특정 명령어 외에 다른 큐 스레드가 개별적으로 동작할 수 있는 여지를 비교적 허용함.

PX4 방식은 QGC와 결합하여 자율 비행(Autonomous Flight) 시 예측 불가능한 이상 현상이 연쇄 작용을 일으키는 것을 막는 데 주안점을 두고 있다. 따라서 강제 천이와 트랜잭션 즉각 폐기 전략은 시스템 복원을 보장하는 최고의 방패막이 된다.

5. 결론

GCS 내부의 소프트웨어 워치독 타이머 설계는 통신 트러블이나 제어 대상의 응답 불능 상황으로부터 지상 컴퓨터 소프트웨어의 안정성을 지켜내는 관문 역할을 한다. 인터럽트 만료 시 강제적으로 발동되는 안전 복구 유한 상태 기계(Fallback FSM) 파이프라인은 진행 중이던 불안정한 명령 트랜잭션을 원천적으로 파기함으로써, 관제 시스템이 데이터 교착 상태(Deadlock)에 빠지지 않고 항시 조종자에게 최적의 상황 인지성(Situational Awareness)을 제공할 수 있도록 보장한다.
