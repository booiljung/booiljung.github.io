45.9.2. MAVSDK(C++/Python) 프레임워크의 동시성(Concurrency) 제어 아키텍처

지상 관제 시스템(GCS)이 QGroundControl(QGC)처럼 인간 조종자를 위한 시각적 인터페이스에 집중한다면, MAVSDK(MAVLink SDK)는 자율 비행 애플리케이션, 군집 비행 제어, 머신 비전 연동 등 소프트웨어 에이전트(Software Agent)를 위한 미들웨어적 성격이 강하다. MAVSDK는 고성능 C++ 코어 브릿지(Core Bridge)를 중심으로 Python, Swift, Java 등 다국어 래퍼(Wrapper)를 결합하는 하이브리드 아키텍처를 취하고 있다. 본 절에서는 MAVSDK가 MAVLink 패킷 송수신과 사용자 스크립트 실행 간의 충돌을 막고 비동기(Asynchronous) 이벤트를 처리하기 위해 구축한 동시성(Concurrency) 제어 아키텍처를 심도 있게 다룬다.

1. MAVSDK 백엔드(C++)의 동시성 제어 기반

MAVSDK의 C++ 코어(Core)는 멀티 스레딩(Multi-threading) 환경에 최적화되어 설계되었다. MAVSDK 인스턴스가 생성되면 내부적으로 다음과 같은 스레드 풀(Thread Pool) 및 작업 분리 구조가 형성된다.

  • 수신 스레드(Receive Thread): 직렬 포트(UART), UDP, 또는 TCP 링크를 통해 들어오는 바이트 스트림을 수신하고 mavlink_message_t로 파싱(Parsing)한다. 파싱된 메시지는 각 플러그인(Plugin, 예: Action, Telemetry, Mission)의 메시지 처리기(Message Handler)로 분배된다.
  • 콜백 스레드(Callback Thread): 사용자(개발자)가 등록한 콜백 함수(Callback Function)를 실행하는 전담 스레드 풀이다. 특정 MAVLink 응답(ACK)이나 텔레메트리 스트림 업데이트가 발생했을 때, 수신 스레드가 직접 콜백을 실행하지 않고 큐(Queue)에 넘긴다. 이는 사용자의 콜백 로직 내부에서 블로킹 작업(예: 디스크 I/O)이 발생하더라도 핵심 MAVLink 패킷 파싱이 지연되지 않도록(Decoupling) 하기 위함이다.
  • 미들웨어 통신: C++ 표준 라이브러리의 std::thread, std::mutex, std::condition_variable을 활용하여 스레드 간 동기화를 보장하며, std::promise와 std::future를 활용한 미래 객체 반환 설계로 비동기 API를 제공한다.

2. Python 래퍼(Wrapper)와 asyncio 코루틴(Coroutine) 아키텍처

MAVSDK-Python 등 하이레벨 언어용 래퍼는 C++ 코어와 직접 링크(Link)되지 않고 gRPC(Google Remote Procedure Call) 채널을 통해 소통한다. 이는 언어 종속성을 제거하고 마이크로서비스 아키텍처(MSA) 관점에서의 분산 배치를 가능하게 한다.

특히 Python 모듈은 gRPC의 비동기 호출 기능을 극대화하기 위해 Python 3.6 이상부터 표준으로 자리 잡은 asyncio 프레임워크를 전면 채택하였다.

# MAVSDK-Python의 전형적인 비동기 호출 패턴 예시
import asyncio
from mavsdk import System

async def run():
    drone = System()
    await drone.connect(system_address="udp://:14540")
    
    # 여기서 await는 이벤트 루프를 차단하지 않고 양보(yield)한다.
    async for position in drone.telemetry.position():
        print(f"고도: {position.relative_altitude_m}")
        break  # 단일 수신 후 탈출

if __name__ == "__main__":
    asyncio.run(run())

위 파이썬 코드가 실행될 때 내부적으로 일어나는 동시성 동작 원리는 다음과 같다.

  1. await drone.connect()를 호출하면, 파이썬 객체는 내부적으로 코어 서버(C++ gRPC Server)로 원격 프로시저 호출을 예약하고, 파이썬 메인 이벤트 루프에 제어권을 양보(yield)한다.
  2. 메인 루프는 다른 코루틴 태스크를 병렬로 스케줄링할 수 있다.
  3. gRPC 응답이 도착하면 파이썬의 asyncio 이벤트 루프로 콜백이 인입되고, 앞서 중단되었던 await 지점부터 함수의 실행이 재개된다.

결과적으로 개발자는 time.sleep()과 같은 치명적인 스레드 블로킹(Thread Blocking) 함수 대신 asyncio.sleep()과 await를 조합하여, 수십 대의 드론(Swarm) 제어 로직을 단일 스레드 내에서 비동기로 병력 제어할 수 있는 강력한 코루틴 아키텍처를 누리게 된다.

C++ 코어의 멀티 스레딩과 Python 클라이언트의 코루틴이 gRPC 프로토콜을 매개로 어떻게 동시성 제어를 조화롭게 수행하는지는 아래 나열된 구조도를 통해 직관적으로 확인할 수 있다.

4. 아키텍처적 의의 및 차별점

ROS2가 uXRCE-DDS 미들웨어 기반으로 토픽(Topic) 중심의 데이터 교환에 최적화되어 있다면, MAVSDK는 이러한 이벤트 드리븐(Event-Driven) 및 명령형(Imperative) 비동기 호출 구조가 완비되어 있다. ArduPilot의 DroneKit 모듈이 과거 단일 스레드와 글로벌 락(Global Interpreter Lock, GIL)의 한계를 극복하기 위해 다소 억지스러운 몽키 패치(Monkey Patching) 체제를 동원했던 것에 비해, MAVSDK의 설계는 언어(네이티브 및 스크립트)와 네트워크 계층, 컴파일 환경을 완전히 분리해내는 데에 중점을 두었다.

이를 통해 GCS 개발자나 관제 통합 관리자는 “패킷 전송 주기, 네트워크 지연(Latency)“과 같은 하위 레이어의 멀티 스레드 경합(Deadlocks) 이슈 등으로부터 해방되어, “이륙 -> 웨이포인트 이동 -> 착륙“과 같은 고차원적인 비행 상태 머신(State Machine) 흐름 제어에만 집중할 수 있는 쾌적한 개발 환경을 제공받는다.
