45.2.2.3. 비트 시프트(Bit Shift) 및 비트 논리곱(Bitwise AND) 연산 시 중앙처리장치(CPU) 레벨의 연산 원자성(Atomicity) 확보

PX4-Autopilot의 핵심 제어 모듈인 커맨더(Commander) 스레드와 통신을 담당하는 MAVLink 수신 스레드는 NuttX RTOS 환경 내에서 서로 다른 우선순위(Priority)를 가지고 동시에 실행되는 멀티스레드(Multithread) 아키텍처를 지닌다. 이 환경에서 비행 모드(Flight Mode)나 주요 기체 상태명령을 변경하려면 32비트 혹은 64비트로 구성된 상태 변수(State Variable)의 특정 비트(Bit) 구역을 켜거나 끄는 비트 시프트(Bit Shift) 및 비트 논리곱(Bitwise AND) 연산을 빈번하게 수행해야 한다.

하지만 이러한 비트 조작 연산이 중앙처리장치(CPU) 레벨에서 원자성(Atomicity)을 완벽하게 확보하지 못한 채 병렬로 실행된다면, 심각한 경쟁 상태(Race Condition)가 발생하여 예상치 못한 통제 불능의 비행 제어 모드로 시스템 상태가 오염될 수 있다. 따라서 PX4 펌웨어 코어 아키텍처에서는 이기종 스레드 간의 데이터 동기화와 원자성을 보장하기 위한 정교한 제어 기법이 필수적으로 수반된다.

1. 비트 연산의 Read-Modify-Write(RMW) 문제와 한계

대부분의 무인항공기(UAV) 비행 제어기 하드웨어(예: Pixhawk 시리즈)에서 널리 채택하고 있는 ARM Cortex-M 계열의 RISC(Reduced Instruction Set Computer) 마이크로프로세서는 메모리 셀에 직접적으로 비트 논리 연산을 수행하지 못한다. C/C++ 언어 코드 레벨에서 mode_bitmask |= (1 << 3);와 같은 일견 단순해 보이는 단일 구문으로 작성된 비트 시프트 및 논리합(OR) 연산은 사실 컴파일된 어셈블리 기계어 레벨에서 다음과 같은 3단계의 분할된 명령어(Instruction)로 동작하게 된다.

  1. Read (LDR): 타겟 메모리 주소에서 현재의 비트마스크 데이터를 CPU 레지스터(Register)로 로드(Load)한다.
  2. Modify (ORR): 물리적 레지스터 내부에서 비트 연산을 수행하여 특정 비트 필드의 값을 갱신(Modify)한다.
  3. Write (STR): 갱신된 결과값을 다시 동일한 타겟 메모리 주소에 저장(Store)한다.

만약 MAVLink 데이터 수신 스레드가 제1단계와 제2단계를 수행하는 찰나의 순간에 하드웨어 인터럽트(Hardware Interrupt)가 발생하여 문맥 교환(Context Switching)이 일어나게 되고, 더 높은 우선순위의 센서 처리 스레드나 안전 감시(Failsafe) 스레드가 동일한 비트마스크 변수에 접근하여 값을 갱신해버린다면 어떻게 될까? 인터럽트가 종료되고 문맥 교환에서 원래대로 돌아온 기존의 MAVLink 스레드는 지연된 채로 제3단계를 비로소 수행하게 되며, 이전에 안전 감시 스레드 등 다른 런타임 시스템이 수행한 필수 변경 사항을 완전히 덮어씌우는(Overwrite) 오류를 낳는다.

2. PX4의 하드웨어 종속적 및 независи적 원자성 확보 설계 기법

저지연 및 고신뢰성을 추구하는 PX4 소프트웨어 아키텍처는 비트 조작 모델의 동시성 문제를 해결하기 위해 시스템 계층 전반에 걸쳐 크게 두 가지의 방어 기제를 혼용하여 사용한다.

2.1 C++11 표준 std::atomic 템플릿의 적극적 활용

PX4 최신 메인라인(버전 1.14 이후 시스템 베이스라이닝) 모듈들은 가급적 하드웨어 RTOS 종속성이 낮은 std::atomic<uint32_t> 등의 표준 원자적 변수 타입을 활용하도록 리팩토링되었다. 시스템 개발자가 C++ 컴파일러를 향해 atomic_status_flags.fetch_or(1 << CUSTOM_MAIN_MODE_BIT, std::memory_order_relaxed);와 같은 구문으로 비트 논리합 연산을 요청하면, 컴파일러는 동작하는 대상 아키텍처가 제공하는 하드웨어 지원 원자적 명령어(예를 들어 ARM 백엔드의 LDREX 및 STREX 배타적 모니터 장치 명령어 쌍)를 적절히 호출해 다른 인터럽트 루틴에 의해 절차적 동작이 파편화되지 않도록 원자성을 보장한다. 메모리 배리어(Memory Barrier) 전략과 엮여 동작하므로, 안전 장치 트리거 갱신과 주 비행 모드의 통신 갱신을 완벽하게 격리해 낼 수 있다.

2.2 RTOS 수준의 인터럽트 임계 구역(Critical Section) 방어

아키텍처 상 다수의 플래그를 복합적으로 비교 및 조작해야 하는 거대 모듈이나 아직 추상화가 덜 이뤄진 레거시(Legacy) 코드 블록의 경우에는, 여전히 NuttX RTOS 커널 계층에서 제공하는 최상위 권한의 인터럽트 마스킹(Interrupt Masking) 기반의 임계 구역 락킹(Locking)을 적용한다.

// NuttX 환경에서의 임계 구역(Critical Section) 진입 예제
irqstate_t flags = enter_critical_section(); // 글로벌 하드웨어 인터럽트 비활성화
uint32_t current_mode = vehicle_status.nav_state; // Read
current_mode &= ~(0b1111 << SHIFT_OFFSET); // 기존 모드 비트 클리어 (Bitwise AND MASK)
current_mode |= (NEW_MODE << SHIFT_OFFSET); // 새로운 모드 인가 (Bitwise OR SHIFT)
vehicle_status.nav_state = current_mode; // Write
leave_critical_section(flags); // 인터럽트 활성화 및 처리 복구

위처럼 CPU 사이클을 물리적으로 일시 정지시키는 인터럽트 차단 방식은 가장 확실한 연산 원자성을 제공한다. 그러나 이 방식을 지나치게 남용할 경우 시스템 전체의 실시간(Real-Time) 응답 지연 시간(제어 루프 Latency)을 전반적으로 증가시킬 리스크가 상존하므로, 제어 연산 주기가 극도로 짧은 상태 갱신 매커니즘 등의 마이크로초 단위 최소 조작에만 선별적으로 설계해야 하는 부담이 따른다.

4. 컴파일러 최적화 방지(Compiler Optimization Prevention)를 위한 아키텍처 규범

소프트웨어 엔지니어들이 종종 오해하는 부분은 단순히 volatile 키워드를 상태 변수에 선언하는 것만으로 멀티코어 파이프라인(Multicore Pipeline) 또는 다중 스레드 환경에서 원자성을 확보할 수 있을 것으로 기대하는 것이다. 그러나 volatile은 컴파일러가 해당 변수를 캐시에서 읽어오는 최적화를 방지하고 메인 주소에서 읽어오도록 지시할 뿐, Read-Modify-Write 기계어 동작 자체의 하위 연산 분리 현상이나 분리 스케줄링을 근원적으로 차단해주지는 않는다.

따라서 PX4 미들웨어(uORB 기반 메시징) 계층이나 관제 모듈(GCS Interface) 단의 개발자는 반드시 std::atomic과 같은 메모리 배리어를 내포한 전용 원자적 형(Type) 추상화를 사용하여야 한다. 이를 통해 레지스터 파이프라인(Register Pipeline) 상에서의 CPU 독립적인 명령어 재배치(Instruction Reordering) 등 예측 불가능한 패널티를 억제하고, 컴파일러(예: GCC/LLVM Clang) 레벨 및 타일(Tile) 아키텍처 수준의 연산 원자성을 동시에 보장할 수 있는 산업 표준형 우주항공 제어 소프트웨어를 구현해야 한다.
