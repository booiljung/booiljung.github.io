45.5.1.1 범용 비동기 송수신 회로(UART) 하드웨어 인터럽트 서비스 루틴(ISR) 지연 시간(Latency) 모델링

1. 개요

본 절에서는 최신 PX4-Autopilot(펌웨어 v1.14 기준)이 구동되는 픽스호크(Pixhawk) 시리즈 물리 하드웨어 상에서, 텔레메트리나 보조 컴퓨터와의 MAVLink 통신 시 이용되는 **범용 비동기 송수신 회로(UART, Universal Asynchronous Receiver-Transmitter)**의 하드웨어 인터럽트 서비스 루틴(ISR, Interrupt Service Routine) 구조와 송수신 지연 시간(Latency) 모델링에 대해 분석한다.

공학적 측면에서, 고해상도의 제어 루프를 방해하지 않고 외부 데이터를 수신하기 위해서는 하드웨어 인터럽트가 펌웨어의 스케줄러(NuttX RTOS)로 전파되는 지연 시간에 대한 수학적 모델링이 필수적이다.

2. UART 하드웨어 인터럽트 및 NuttX RTOS 통합 아키텍처

비행 제어기(STM32F4, STM32H7 등)의 UART 주변 장치(Peripheral)는 외부에서 시리얼 데이터의 비트(Bit)열이 도달하여 시프트 레지스터(Shift Register)를 거쳐 수신 버퍼 레지스터(RXNE, Receive Data Register Not Empty)에 1바이트 또는 여러 바이트의 데이터가 채워질 때 하드웨어 인터럽트 신호를 발생시킨다.

NuttX RTOS는 하드웨어 인터럽트를 세밀하게 제어하기 위해 시스템 호출 계층(System Call Layer) 하단에 인터럽트 디스패처(Interrupt Dispatcher)를 배치한다. UART 수신 인터럽트가 발생하면 프로세서는 현재 실행 중인 제어 루프를 즉각 멈추고(Preemption) 하드웨어 컨텍스트를 스택에 저장한 후 UART ISR 함수로 분기한다.

3. ISR 지연 시간(Latency) 수학적 모델링

하드웨어 핀에 전기적 신호의 마지막 정지 비트(Stop Bit)가 도착한 시점부터, 소프트웨어 MAVLink 수신 스레드가 이를 인지하여 처리를 시작하기까지의 전체 수신 지연 시간(T_{latency})은 다음과 같은 요소들로 수학적 모델링을 할 수 있다.

T_{latency} = T_{HW\_trigger} + T_{OS\_dispatch} + T_{ISR\_exec} + T_{Thread\_wakeup}

각 컴포넌트의 상세 내역은 다음과 같다.

  1. T_{HW\_trigger} (하드웨어 인터럽트 촉발 지연):
    UART 컨트롤러가 데이터 수신을 완료하고 메인 CPU 코어의 중첩 벡터 인터럽트 제어기(NVIC, Nested Vectored Interrupt Controller)에 신호를 보내어 우선순위 판별 및 레지스터 푸시(Push)가 완료되는 하드웨어적 시간이다. 최신 MCU에서는 통상 10~20 CPU 클록 주기(수십 나노초 \mu s)로 극히 미미하다.

  2. T_{OS\_dispatch} (운영체제 디스패치 지연):
    NuttX의 공용 인터럽트 핸들러가 발생한 인터럽트 번호(IRQ)를 판별하여 등록된 특정 UART 드라이버의 ISR(예: stm32_uart_interrupt())을 호출하는 데 걸리는 소프트웨어적 지연 시간이다(수 마이크로초 \mu s).

  3. T_{ISR\_exec} (하드웨어 드라이버 실행 시간):
    UART 드라이버 ISR 내에서 하드웨어 레지스터를 읽고, OS 레벨의 소프트웨어 링 버퍼(Ring Buffer)로 데이터를 복사(Copy)하는 동작에 소요되는 시간이다. 이 과정에서 가급적 루틴을 짧게 유지하여 인터럽트 차단(Interrupt Masking) 시간을 최소화해야만 전체 비행 제어 지연율을 방어할 수 있다.

  4. T_{Thread\_wakeup} (스레드 문맥 교환 지연):
    ISR이 링 버퍼에 데이터를 복사한 후, poll() 등에서 블로킹(Blocking)된 MAVLink 수신 데몬(Receiver Daemon) 스레드에 세마포어(Semaphore) 신호를 주어 깨울 때(Wake-up), OS 스케줄러가 개입하여 컨텍스트 스위칭을 수행하는 시간이다. 이 단계에서 가장 높은 변동성(Jitter)과 오버헤드가 단발적으로 발생한다.

4. 하드웨어 FIFO 버퍼 및 유휴(Idle) 인터럽트를 활용한 지연 최적화

초당 수백 개의 MAVLink 패킷이 도착하는 최악의 조건(Worst Case)에서, 매 바이트(Byte) 단위마다 위 과정이 발생하면 CPU 스래싱(Thrashing)이 촉발된다. 이를 방지하기 위해 PX4 호환 하드웨어에서는 UART의 자체 하드웨어 FIFO (First In First Out) 기능과 유휴 회선 감지(Idle Line Detection) 인터럽트를 조합하여 설계한다.

  • 워터마크(Watermark) 인터럽트: FIFO 버퍼의 절반 이상(예: 8바이트 또는 16바이트)이 수신될 때까지는 하드웨어 인터럽트를 억제하여 ISR 진입 빈도를 크게 낮춘다.
  • 유휴(Idle) 인터럽트: 마지막 바이트가 수신된 후 짧은 유휴 시구간이 발생하면 데이터 전송 단위(패킷)가 완료된 것으로 간주하여, 버퍼가 설정치를 넘지 않아도 즉각적으로 인터럽트를 띄우고 스레드를 깨워 지연 시간을 줄인다.

이는 지연 시간 최적화와 CPU 자원 확보 간의 클래식한 오퍼레이팅 시스템 트레이드오프(Trade-off) 전략(Interrupt Coalescing 기법)의 대표적인 하드웨어 설계 모형이다.

5. Pixhawk 실험 데이터 예시

STM32H7 기반 고성능 컨트롤러(동작 클록 400MHz 이상)를 탑재한 Pixhawk 제품군에서, 115200 bps 보드레이트(Baudrate)로 데이터 통신을 수행할 경우 1바이트를 물리적으로 쏴보내는 데 약 86.8\mu s 가 걸린다. 하드웨어 인터럽트는 CPU 스피드에 워낙 민감하므로 T_{HW\_trigger} + T_{OS\_dispatch}는 1\mu s 이하로 측정되며, 링 버퍼로 복사하고 스케줄러에 문맥 교환을 요청하는 전체 레이턴시는 대략 5~15\mu s 사이에서 결정론적(Deterministic)으로 동작한다(관련 연구 논문 “Real-time Latency Measurements on PX4 Autopilot Middleware” 참고).

6. 결론

결론적으로 PX4-Autopilot은 NuttX RTOS의 간소화된 인터럽트 핸들링 체계와 STM32 계열의 선진 하드웨어 FIFO 기능을 결합하여 범용 비동기 송수신 회로(UART)의 태생적 지연 한계를 극복해 냈다. 수학적으로 명확히 분해되는 T_{latency} 모델을 기반으로 드라이버 코드를 고도화하였기에, QGroundControl이나 보조 컴퓨터로부터 대량의 텔레메트리 패킷 스트림이 쇄도하더라도 치명적인 루프 시간 변동(Loop Time Jitter) 없이 비행 제어 스택 본연의 연산 스케줄을 유지할 수 있다.
