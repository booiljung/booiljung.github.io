45.6.1.3 임시 거부 및 영구 거부(DENIED) 응답에 대한 비행체 내부 논리 오류 분석 트리(Fault Tree Analysis) 역추적 맵핑

지상 관제 시스템(Ground Control Station, GCS)이 무인항공기(Unmanned Aerial Vehicle, UAV)로 MAVLink 제어 명령(COMMAND_LONG 등)을 하달했을 때, 비행 제어기(Flight Controller)의 내부 제약 조건(Constraints)에 의해 명령 수행이 불가능한 경우가 빈번하게 발생한다. PX4-Autopilot(기준 펌웨어 버전: v1.14.0)은 이 경우 COMMAND_ACK 메시지의 결과 필드를 통해 MAV_RESULT_TEMPORARILY_REJECTED(임시 거부) 또는 MAV_RESULT_DENIED(영구 거부) 응답을 반환한다.

관제 소프트웨어 입장에서는 단순한 ‘거절’ 상태를 전시하는 것을 넘어, 거절의 근본 원인(Root Cause)을 파악하고 운영자에게 조치 방안을 제시해야 한다. 본 절에서는 논리 오류 분석 트리(Fault Tree Analysis, FTA) 기법을 활용하여 기체 내부의 거부 원인을 역추적(Backward Trace Mapping)하는 설계 구조에 대해 깊이 있게 다룬다.

1. MAVLink 거부(Rejection) 응답의 의미적 분류

비행체 내부의 커맨더(Commander) 모듈은 수신된 명령의 유효성, 시스템의 현재 상태 공간, 하드웨어 센서의 건전성을 종합적으로 평가하여 명령어 수용 여부를 결정한다.

  • MAV_RESULT_TEMPORARILY_REJECTED (임시 거부): 시스템이 명령을 수행할 수 있는 논리적 요건은 갖추었으나, 현재의 과도기적 상태(예: 센서 초기화 진행 중, 다른 명령 처리 중)로 인해 즉시 실행할 수 없는 상태. 재시도(Retry) 로직이 유효하다.
  • MAV_RESULT_DENIED (영구 거부): 현재 시스템의 모드, 하드웨어 결함, 또는 전제 조건(Pre-conditions) 결여(예: 시동 전 비행 모드 변경 시도, GPS 수신 불량 상태에서의 임무 시작 시도)로 인해 명령 수행이 원천적으로 불가한 상태. 원인 해결 전까지 재시도는 무의미하다.

2. 논리 오류 분석 트리(Fault Tree Analysis, FTA) 기반 역추적 개념

논리 오류 분석 트리(FTA)는 시스템의 특정 원치 않는 최상위 사건(Top Event)이 발생했을 때, 이를 유발한 하위 논리적 원인들을 불리언 연산(Boolean Logic) 기법을 통해 하향식(Top-Down)으로 전개하여 근본 원인을 식별하는 시스템 공학 기법이다.

명령 거부(COMMAND DENIED)라는 최상위 사건이 발생했을 때, GCS(예: QGroundControl v4.3.0)는 기체로부터 이미 수신 중인 텔레메트리(SYS_STATUS, EXTENDED_SYS_STATE, STATUSTEXT 등)와 캐시된 가상 상태 공간(Virtual State Space) 데이터를 결합하여 다음과 같은 논리 트리 역추적을 수행한다.

2.1 Ardupilot 방식과의 구조적 차이점

Ardupilot 환경에서는 명령 거절 시 주로 사용자 해독용 문자열 형태로 STATUSTEXT 메시지를 통해 “Command denied by autopilot“과 같은 메시지를 브로드캐스팅하는 데 의존하는 경향이 크다. 이는 규칙 기반(Rule-based)의 소프트웨어적 예외 처리를 어렵게 한다. 반면 PX4 생태계는 명시적인 상태 비트마스크(Bitmask)와 상태 플래그를 엄격하게 관리하므로, GCS 백엔드 단에서 FTA 트리의 논리 게이트를 소프트웨어 알고리즘으로 모델링하여 추적하는 것이 훨씬 용이하다.

3. FTA 역추적 알고리즘 설계 및 시각화 (Mermaid)

다음은 GCS가 “자동 비행 임무(Auto Mission) 시작 명령 거부“를 수신했을 때 내부적으로 원인을 추론하는 논리 오류 분석 트리의 단순화된 모델이다.

관제 소프트웨어는 DENIED를 수신하면 트리의 뿌리(Root)부터 역으로 탐색을 시작한다.

  1. SYS_STATUS의 비트마스크를 확인하여 배터리가 부족한지(S3) 검사한다.
  2. HEARTBEAT의 base_mode를 확인하여 기체가 시동 불가능한 상태(C1)인지 확인한다.
  3. ESTIMATOR_STATUS 플래그를 해석하여 EKF 필터의 위치 로컬라이제이션이 유효한지(S1) 점검한다.

이러한 분해능 맵핑(Resolution Mapping)을 통해 GCS는 사용자에게 단순히 “명령이 거부되었습니다“라고 표출하는 대신, “EKF 위치 추정 불가로 인하여 임무를 시작할 수 없습니다. GPS 수신 상태를 확인하십시오“라는 맥락 인식형(Context-Aware) 피드백을 제공할 수 있다.

4. GCS 백엔드 구현 실무 적용 방안

QGroundControl(QGC) 또는 MAVSDK와 같은 C++ 기반 환경에서 이 구조를 체계화하기 위해서는 다음과 같은 설계 패턴이 요구된다.

  • 상태 추적기(State Tracker): 백그라운드 스레드에서 최신 SYS_STATUS 및 ESTIMATOR_STATUS를 지속적으로 폴링(Polling)하거나 시그널-슬롯 메모리에 캐싱해야 한다.
  • 분석 트리 엔진(Analysis Engine): MAV_RESULT 열거형(Enum)이 리턴될 때 캐시된 상태 트리를 입력으로 받아 불리언 식을 평가하는 순수 함수(Pure Function) 파이프라인.

이와 관련된 자율 로봇의 고장 허용(Fault Tolerance) 모델링과 계층적 에러 분석에 대해서는 다음 논문이 참조 문헌으로서 학술적 가치를 지닌다.

  • 논문 참조: “Model-Based Fault Tree Analysis for Unmanned Aerial Vehicles”

결론적으로, GCS 내부에 수립된 논리 오류 분석 트리(FTA) 기반 역추적 맵핑 파이프라인은 덤프형(Dump) 텍스트 분석에 의존하던 전통적인 관제 인터페이스의 한계를 극복하고, 시스템 진단 및 오퍼레이터의 상황 인식(Situational Awareness) 성능을 극대화하는 중추적인 역할을 한다.
