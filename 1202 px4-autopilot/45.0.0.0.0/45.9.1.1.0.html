45.9.1.1. C++ 백엔드 가상 함수 테이블(Virtual Function Table) 동적 바인딩(Dynamic Binding) 및 호출 스택(Call Stack) 추적

QGroundControl(이하 QGC)은 단일 지상 관제 애플리케이션으로 PX4-Autopilot과 ArduPilot 등 전혀 다른 철학과 구조를 가진 비행 제어기(Autopilot)를 동시에 지원한다. 이러한 강력한 이식성(Portability)과 확장성의 핵심에는 C++의 객체 지향 프로그래밍(OOP, Object-Oriented Programming) 패러다임, 특히 가상 함수 테이블(vtable, Virtual Function Table)을 활용한 동적 바인딩(Dynamic Binding) 설계가 자리 잡고 있다. 본 절에서는 Vehicle::setFlightMode 호출 시 내부적으로 어떠한 동적 바인딩이 일어나는지와 그에 따른 호출 스택(Call Stack)의 흐름을 기술한다.

1. 다형성(Polymorphism)과 펌웨어 플러그인(Firmware Plugin) 아키텍처

QGC 아키텍처에서 비행 제어기의 펌웨어 종속적인 논리는 FirmwarePlugin이라는 추상 기저 클래스(Abstract Base Class)로 분리되어 있다. Vehicle 클래스는 기체가 연결될 때 MAVLink의 HEARTBEAT 메시지에 포함된 autopilot 필드(예: MAV_AUTOPILOT_PX4 또는 MAV_AUTOPILOT_ARDUPILOTMEGA)를 판별하여, 알맞은 파생 클래스(Derived Class)의 인스턴스를 동적으로 할당받는다.

  • FirmwarePlugin (기저 클래스)
  • PX4FirmwarePlugin (PX4 전용 파생 클래스)
  • APMFirmwarePlugin (ArduPilot 전용 파생 클래스)

Vehicle 클래스는 자신이 관리하는 기체가 구체적으로 어떤 펌웨어인지 알 필요 없이, 오직 FirmwarePlugin 포인터(FirmwarePlugin*)만을 지니고 인터페이스를 호출한다. 비행 모드 전환 시 호출되는 setFlightMode 인터페이스 역시 FirmwarePlugin 내에 순수 가상 함수(Pure Virtual Function) 또는 오버라이딩(Overriding) 가능한 가상 함수로 선언되어 있다.

2. 가상 함수 테이블(vtable) 기반 동적 바인딩(Dynamic Binding) 원리

C++ 컴파일러(예: GCC, Clang)는 가상 함수가 하나라도 선언된 클래스에 대해 숨겨진 가상 함수 테이블(vtable)을 생성한다. 이 테이블은 각 가상 함수에 대한 실제 메모리 주소(Function Pointer) 목록을 담고 있다.

기체 연결 시 QGC 팩토리 패턴(Factory Pattern)에 의해 PX4FirmwarePlugin 객체가 메모리에 인스턴스화되면, 해당 객체의 숨겨진 vptr(Virtual Pointer)은 PX4FirmwarePlugin 전용 vtable을 가리킨다.

결과적으로 Vehicle::setFlightMode 내부에서 실행되는 다음 코드는 컴파일 타임(Compile-time)이 아닌 런타임(Runtime)에 실제 주소를 찾아 동적으로 바인딩(Dynamic Binding)된다.

// Vehicle.cc 내부의 모드 설정 로직 예시
uint8_t base_mode;
uint32_t custom_mode;

// 여기서 _firmwarePlugin은 FirmwarePlugin* 타입이다.
// 다이나믹 바인딩: 객체의 vptr -> vtable -> 실제 구현체(PX4 또는 ArduPilot)의 함수 주소 호출
_firmwarePlugin->setFlightMode(flightMode, &base_mode, &custom_mode);

이러한 설계로 인해, 조종자가 “Return” 모드를 선택하더라도 QGC 엔진은 PX4FirmwarePlugin::setFlightMode로 진입하여 PX4 펌웨어 버전(1.14 기준)에 맞는 RTL(Return To Launch) custom_mode 비트필드(Bitfield)로 변환하거나, ArduPilot 기체일 경우 APMFirmwarePlugin을 통해 정해진 열거형(Enum)으로 자동 변환해 준다. 분기문(if/switch) 오버헤드 없이 O(1)의 성능으로 펌웨어 확장이 가능한 구조적 장점을 지닌다.

3. setFlightMode 호출 스택(Call Stack) 추적

QML 프론트엔드 버튼 클릭에서 시작되어 MAVLink 패킷이 생성되기까지의 호출 스택(Call Stack) 흐름은 다음과 같은 깊이(Depth)를 가진다. 이를 추적함으로써 QGC 백엔드 계층 간 책임 분리를 명확히 이해할 수 있다.

  1. [QML UI Layer] 사용자 입력 발생
  • FlightDisplayView.qml 또는 FlightModeSelector.qml에서 조종자가 특정 모드를 클릭한다.
  • QML 엔진은 내부적으로 QGCVehicle::flightMode 프로퍼티(Property)의 Setter를 호출한다.
  1. [C++ Vehicle Layer] Vehicle::setFlightMode(const QString& flightMode) 진입
  • QML 측에서 전달된 텍스트 문자열(“Mission”, “Hold” 등)을 인자로 받는다.
  1. [C++ FirmwarePlugin Layer] FirmwarePlugin::setFlightMode(const QString& flightMode, uint8_t* base_mode, uint32_t* custom_mode) 동적 바인딩 호출
  • 동적 바인딩 핵심 구간: vtable을 참조하여 PX4FirmwarePlugin 쪽 구현체로 점프(Jump)한다.
  • PX4 펌웨어 규격에 맞춰 base_mode와 custom_mode 메모리 공간에 적절한 값을 기록한다.
  1. [C++ Vehicle Layer] MAVLink 명령 송신 준비
  • FirmwarePlugin으로부터 반환된 모드 값들을 검증한 뒤, Vehicle::sendMavCommand 호출을 준비한다.
  • 명령 전송 객체(VehicleCommandList 또는 큐)를 통해 타임아웃과 재전송을 추적하도록 등록한다.
  1. [QGC MAVLink Protocol Stack Layer] MAVLinkProtocol::sendMessage 또는 Vehicle::sendMessageOnLink
  • COMMAND_LONG 메시지 중 MAV_CMD_DO_SET_MODE 값을 채워넣어 직렬화(Serialization)를 수행한다.
  1. [Network / Serial Link Layer] UDPLink 또는 SerialLink 패킷 전송
  • 물리적 링크(UART, UDP 시리얼 텔레메트리 등)를 통해 텔레메트리 모듈로 바이트 스트림(Byte Stream)을 송출한다.

4. 임베디드 관제 개발자를 위한 요약 및 교훈

QGC의 펌웨어 지원 아키텍처는 전형적인 개방-폐쇄 원칙(OCP, Open-Closed Principle)을 따른다. Vehicle 클래스는 변경에 닫혀 있으며, 새로운 비행 제어기가 등장하더라도 FirmwarePlugin을 상속받는 새로운 파생 클래스만 추가(확장에 열려 있음)하면 된다. C++의 가상 함수 테이블(vtable)이 제공하는 동적 바인딩 메커니즘 덕분에, 복잡한 시스템에서도 MAVLink 기반 제어 명령 로직의 호출 스택을 우아하고 결합도(Coupling)를 낮게 유지할 수 있음을 상기해야 한다.
