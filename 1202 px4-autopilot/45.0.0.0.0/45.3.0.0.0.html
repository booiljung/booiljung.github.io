45.3 장명령(COMMAND_LONG) 프로토콜의 트랜잭션 계층(Transaction Layer) 제어 기법

PX4-Autopilot과 지상 관제 시스템(GCS, Ground Control Station) 간의 통신에서 가장 중요한 요소 중 하나는 신뢰성 있는 명령 전달 구조를 확보하는 것이다. 단순한 상태 보고나 텔레메트리 데이터 전송과 달리, 기체의 상태를 직접적으로 변경하는 제어 명령은 철저한 트랜잭션 계층(Transaction Layer) 제어 기법을 통해 관리되어야 한다. 본 절에서는 MAVLink 프로토콜의 핵심 명령 전송 방식인 COMMAND_LONG 메시지를 기반으로, PX4-Autopilot 내부에서 수행되는 트랜잭션 제어 기법과 아키텍처를 심도 있게 분석한다.

(설명 기준: PX4-Autopilot v1.14.0, QGroundControl v4.3.0)

1. COMMAND_LONG 메시지 구조와 트랜잭션의 개념

MAVLink 프로토콜에서 기체 내부의 동작을 지시하는 주요 메시지는 COMMAND_LONG (#76) 메시지이다. 과거에는 파라미터 개수가 제한적인 COMMAND_INT 메시지 등이 주로 사용되었으나, 현대의 복잡한 무인항공기 제어를 위해 최대 7개의 부동소수점 매개변수(Parameters)를 포함할 수 있는 COMMAND_LONG이 표준으로 자리 잡았다.

COMMAND_LONG 메시지의 구성 필드는 다음과 같다:

  • target_system, target_component: 명령을 수신해야 할 대상 시스템 및 컴포넌트 ID
  • command: 수행할 명령의 고유 식별자(ID) (예: MAV_CMD_DO_SET_MODE)
  • confirmation: 재전송 횟수 또는 확인(Confirmation) 플래그
  • param1 ~ param7: 명령 수행에 필요한 최대 7개의 파라미터 값

트랜잭션(Transaction)은 요청(Request)과 응답(Response)이 쌍(Pair)을 이루는 일련의 과정을 의미한다. GCS가 COMMAND_LONG을 전송하면 대상 기체는 이를 수신하여 처리 가능 여부를 검토하고, 최종적으로 COMMAND_ACK (#77) 메시지를 통해 성공(MAV_RESULT_ACCEPTED), 거부(MAV_RESULT_DENIED), 일시 불가(MAV_RESULT_TEMPORARILY_REJECTED) 등의 결과를 반환해야 한다. 이 과정이 완료될 때까지 하나의 트랜잭션 주기가 유지된다.

2. PX4-Autopilot 내부의 트랜잭션 처리 파이프라인

PX4-Autopilot 내에서 MAVLink 명령이 도착하여 최종적으로 수락되기까지는 여러 컴포넌트를 거치는 논리적인 파이프라인이 존재한다.

  1. MAVLink 수신기 (MAVLink Receiver Layer)
    PX4의 mavlink_receiver.cpp 모듈은 GCS로부터 수신된 직렬화(Serialized)된 MAVLink 패킷을 파싱(Parsing)한다. 이 단계에서 수신된 패킷이 COMMAND_LONG으로 판별되면, 타깃 시스템 및 컴포넌트 ID가 현재 PX4 기체와 일치하는지 검증한다. 일치할 경우, 메시지의 내용을 복호화하여 명령을 라우팅할 준비를 한다.

  2. uORB 미들웨어를 통한 명령 전달 (Command Transfer via uORB)
    PX4는 프로세스 간 통신을 위해 uORB(Micro Object Request Broker) 미들웨어를 사용한다. 수신된 COMMAND_LONG 메시지는 uORB의 vehicle_command 토픽(Topic)으로 변환되어 발행(Publish)된다. 이로 인해 통신 모듈(MAVLink)과 비행 제어 및 런타임 모듈이 완벽히 분리(Decoupling)되는 구조적 장점을 가진다.

  3. 커맨더 모듈의 명령 평가 (Commander Module Evaluation)
    vehicle_command 토픽을 구독(Subscribe)하는 주체는 주로 커맨더(Commander) 모듈이다. 커맨더 모듈은 현재 기체의 내부 상태(비행 모드, 시동 여부, 센서 상태 점검 결과 등)를 고려하여 해당 명령을 수행할 수 있는 전제 조건(Pre-conditions)이 성립하는지 평가한다.

  4. 명령 결과 전송 (Ack Transfer)
    커맨더 모듈의 판단이 완료되면, 그 결과는 다시 uORB 토픽인 vehicle_command_ack으로 발행된다. MAVLink 송신기 모듈(mavlink_messages.cpp)은 이 토픽을 구독하여 최종적으로 COMMAND_ACK MAVLink 메시지로 변환 후 GCS로 반환한다.

3. 트랜잭션 신뢰성 보장 및 재전송(Retry) 제어

통신 환경이 불안정한 무선 링크 상태에서는 트랜잭션이 유실될 가능성을 배제할 수 없다. COMMAND_LONG 프로토콜 계층은 이러한 손실을 관리하기 위해 타임아웃(Timeout) 및 재전송 매커니즘을 내포한다.

3.1 타임아웃(Timeout) 및 확인(Confirmation) 기법

COMMAND_LONG 패킷에 포함된 confirmation 필드는 해당 명령이 첫 번째 시도인지 아니면 재전송된 시도인지를 기록한다.

  • GCS 측: 명령을 보낸 후 즉시 타이머를 가동한다. 일정 시간 내에 기체로부터 타당한 COMMAND_ACK이 도착하지 않으면, confirmation 계수 수치를 증가시켜 동일한 COMMAND_LONG 패킷을 전송한다.
  • PX4 측: 동일한 트랜잭션(동일한 Command ID 및 파라미터)이 confirmation 값이 증가되어 들어오면, PX4 내부에서는 재전송 패킷임을 인지하고 멱등성(Idempotence, 연산을 여러 번 적용하더라도 결과가 같은 성질)을 기반으로 중복 수행 오류를 억제하면서 COMMAND_ACK을 재발송한다.

3.2 Ardupilot 대비 PX4-Autopilot의 관제 처리 차이

PX4-Autopilot과 ArduPilot은 명령 트랜잭션 처리 방식에서 구조적인 차이를 보인다.

  • Ardupilot의 처리 구조: 직렬 연산 위주의 강력한 락(Lock) 체계를 지향하는 접근 방식을 가지며, 각 명령의 처리가 MAVLink 스레드와 밀접하게 맞물려 평가되는 경향이 있다. 처리 자체가 비교적 즉각적으로 진행되며 메모리 참조 구조가 얕은 편이다.
  • PX4-Autopilot의 처리 구조: 앞서 언급한 uORB를 기반으로 하는 철저한 비동기적(Asynchronous) 이벤트 주도(Event-driven) 아키텍처에 중점을 둔다. 명령의 파싱, 논리 평가, 결과 전송이 각기 다른 스레드(모듈)에서 비동기 전파된다. 결과적으로 높은 모듈성을 확보하지만, 내부적으로 미들웨어를 거치며 발생하는 마이크로 단위의 지연(Latency) 속에서 상태 불일치가 일어나지 않도록 정밀한 트랜잭션 제어를 필수로 한다.

4. 상태 전이 방어선 (State Transition Defensive Line)

커맨더 모듈에서 COMMAND_LONG을 평가할 때는 단순한 수락 통과 여부만을 결정하지 않는다. PX4는 상태 전이 안전 장치(State Transition Setup and Failsafe) 파이프라인을 구축해 두었다. 예를 들어, MAV_CMD_DO_SET_MODE (비행 모드 변경 명령)가 수신될 경우, vehicle_status의 내부 논리 회로가 작동하여 변경하려는 모드에 대한 센서 준비 상태(Navigational State)와 이중화(Redundancy) 요건이 만족되는지 교차 검증한다. 이를 통해 조작자의 실수나 외부 공격에 의한 악의적인 제어 명령이 시스템을 위험한 상태로 몰아가는 것을 원천적으로 차단한다.

5. 요약

COMMAND_LONG 프로토콜의 트랜잭션 계층 제어 기법은 신뢰성 높은 무인항공기 원격 제어의 기반이다. PX4-Autopilot의 비동기적 아키텍처는 이를 uORB 라우팅 시스템을 통해 고도로 체계화하였으며, 상태 검증 논리와 명확한 승인 반환 절차(COMMAND_ACK)를 통해 비행 제어 소프트웨어의 강건성(Robustness)을 보장한다.
