45.8. 고등 비행 모드 제약 조건 체계의 구조적 무결성(Structural Integrity) 검증

PX4-Autopilot의 비행 모드는 조종사의 개입 정도와 시스템의 자율성 수준에 따라 수동(Manual), 보조(Assisted), 자율(Autonomous) 및 고등(Advanced) 비행 모드로 분류할 수 있다. 이 중 외부 제어(Offboard) 모드나 임무(Mission) 모드와 같은 고등 비행 모드는 기체의 궤적 및 자세 제어 권한을 온전히 비행 제어 컴퓨터(FCC) 또는 보조 컴퓨터(Companion Computer)에 위임한다. 이러한 권한의 위임은 시스템의 안전성에 직접적인 영향을 미치므로, 고등 비행 모드로의 천이(Transition)와 유지(Maintenance)를 위해서는 엄격한 제약 조건(Constraints)의 충족 여부를 실시간으로 논리적이고 구조적으로 검증해야 한다.

본 절에서는 PX4-Autopilot에서 고등 비행 모드를 운영하기 위해 요구되는 제약 조건 체계의 구조적 무결성(Structural Integrity) 검증 아키텍처를 상세히 분석한다.

1. 제약 조건의 구조적 무결성이란?

비행 제어 시스템에서 구조적 무결성이란, 단순히 개별 센서 데이터의 유효성을 확인하는 것을 넘어, 시스템에 입력되는 제어 명령이나 참조 궤적(Reference Trajectory)이 동역학적 한계, 시간적 연속성, 그리고 공간적 제약을 종합적으로 만족하는 수학적, 논리적 완전성을 의미한다.

구조적 무결성 검증은 크게 다음 세 가지 차원으로 나뉜다.

  • 데이터의 유효성(Data Validity): 수신된 목표값(Setpoint)이나 임무 항점(Waypoint)이 정의된 자료형의 범위를 벗어나지 않는가? (예: NaN 또는 Inf 값 검출)
  • 시간적 연속성(Temporal Continuity): 제어 입력이 시스템의 나이퀴스트-섀년 표본화 정리(Nyquist-Shannon Sampling Theorem)를 만족할 수 있는 주기로 지속해서 갱신되고 있는가?
  • 공간적 유계성(Spatial Boundedness): 목표 위치나 속도가 기체의 동역학적 한계(Kinematic Limits) 및 지정된 지오펜스(Geofence) 내부에 존재하는가?

2. PX4 커맨더(Commander) 모듈의 역할

PX4 아키텍처에서 비행 모드 천이의 승인과 거부를 결정하는 핵심 주체는 커맨더(commander) 모듈이다. 사용자가 QGroundControl(GCS) 또는 보조 컴퓨터(ROS 2 연동 등)를 통해 고등 비행 모드로의 변경을 명령하면, 커맨더는 내부 상태 머신(State Machine)을 통해 비행 전 단계 점검(Pre-flight Checks)과 모드 유효성 검사(Mode Validity Checks)를 수행한다.

flowchart TD
    A[Mode Change Request : Offboard / Mission] --> B(Commander Module)
    B --> C{Structural Integrity Check}
    C -->|1. Data Validity| D[Check Setpoint / Mission Item formats]
    C -->|2. Temporal Continuity| E[Check uORB Topic Update Rates]
    C -->|3. Spatial Boundedness| F[Check Geofence & Kinematic Limits]
    
    D --> G{All Passed?}
    E --> G
    F --> G
    
    G -- Yes --> H[State Transition: ACCEPTED]
    G -- No --> I[State Transition: REJECTED]
    I --> J[Publish MAVLink STATUSTEXT Error]

3. 고등 비행 모드별 구조적 무결성 검증 사례

3.1 외부 제어(Offboard) 모드

외부 제어(Offboard) 모드는 보조 컴퓨터(Companion Computer)가 MAVLink 마이크로 서비스 또는 ROS 2(uXRCE-DDS)를 통해 실시간으로 목표 위치, 속도, 가속도, 질량 중심 추력 등을 조작하는 고도의 자율 비행 모드이다.

  • 무결성 검증 로직: PX4는 Offboard 모드 진입 전과 비행 중에 외부 컴퓨터가 일정한 주파수(기본 설정 COM_OF_LOSS_T 파라미터에 따름, 보통 2Hz 이상)로 제어 목표값 스트림(예: TrajectorySetpoint uORB 메시지)을 전송하고 있는지 감시한다.
  • 안전 장치(Failsafe): 제어 스트림의 무결성이 훼손될 경우(시간적 연속성 단절), PX4는 즉각적으로 구조적 무결성이 깨진 것으로 간주하고 체공(Loiter) 또는 복귀(Return) 모드로 강제 천이하여 기체의 생존성을 확보한다.

3.2 임무(Mission) 자율 비행 모드

임무(Mission) 모드는 GCS에서 사전에 업로드한 웨이포인트(Waypoint)를 순차적으로 추종하는 모드이다.

  • 무결성 검증 로직: Mission 모드로 진입하기 위해서는 내부의 mission_feasibility_checker 모듈이 활성화된다. 여기서는 모든 임무 항점의 고도가 안전선 내에 있는지 유효성(Validity)을 검사하며, 인접한 항점 간의 거리와 상대 각도가 기체의 최대 선회 역학 특성을 초과하지 않는지 물리적 제약 한계를 검증한다.
  • Ardupilot과의 차이점: Ardupilot의 경우 Mission Planner와 같은 GCS에서 사전에 무결성을 크게 강제하여 업로드하는 경향이 강한 반면, PX4는 기체 내부의 Flight Task 레벨에서 임무 항목들의 구조적 무결성을 실행 환경에 맞추어 한 번 더 강도 높게 검증하는 견고한 미들웨어 구조를 갖추고 있다.

4. 구조적 무결성 파괴 시의 강건 제어(Robust Control) 로직

제약 조건의 무결성이 런타임(Run-time)에 파괴될 경우, 제어기는 발산(Divergence)을 막기 위해 강건 제어로 전환된다. 예를 들어 외부 컴퓨터의 연산 지연으로 임무 수행 중 유효하지 않은 NaN 값이 제어 루프에 유입될 경우, 비행 제어기(Flight Controller)의 수학 라이브러리(mathlib) 필터가 이를 즉각적으로 걸러내어 비정상적인 액추에이터 출력을 방지한다. 이후 시스템 관리자(System Manager)는 상태 추정 오류로 감지하고 MAVLink STATUSTEXT를 통해 GCS에 치명적 경보를 송출하게 된다.

5. 요약

고등 비행 모드의 체계를 유지하기 위해 제약 조건의 구조적 무결성 검증은 단순한 에러 검출 이상의 중대한 통제 의미를 갖는다. PX4-Autopilot은 데이터 형식, 시간 간격, 공간 제약이라는 세 가지 축을 중심으로 외부 입력의 수학적, 논리적 완전성을 검사하여 자율 에이전트 비행의 신뢰성을 극한으로 끌어올리도록 설계되었다. GCS 운용자 및 ROS 2 소프트웨어 개발자는 이러한 기체 내부의 체계적인 검증 아키텍처를 사전에 완전히 인지하고 제어 스트림을 논리적으로 무결하게 설계해야만 의도하지 않은 모드 천이 거부나 안전 장치(Failsafe) 발동을 사전에 차단할 수 있다.
