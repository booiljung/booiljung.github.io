Chapter 17. NuttX RTOS 기반 PX4 시스템 동작 원리 및 스케줄링

비행 제어 컴퓨터(Flight Controller)의 CPU는 하늘을 나는 동안 단 1 밀리초(ms)의 정지도 용납받지 못한다. 가속도 센서가 초당 1,000번 데이터를 쏟아내고, 모터는 초당 400번 새로운 PWM 제어값을 요구하며, GCS는 쉴 새 없이 원격 제어 명령을 날린다. 이 모든 난장판 속에서 수백 개의 C++ 모듈이 충돌 없이, 그리고 ‘정해진 데드라인(Deadline) 안에 무조건’ 실행되도록 통제하는 보이지 않는 신(God)이 바로 실시간 운영체제(RTOS, Real-Time Operating System)다.

PX4-Autopilot은 이 절대적인 타이밍 무결성을 확보하기 위해 POSIX 표준을 준수하는 경량 커널인 NuttX RTOS를 그 심장으로 채택하고 있다. 자유로워 보였던 PX4의 소스 코드는 사실 이 NuttX의 엄격한 스레드(Thread) 스케줄링과 메모리 통제망 안에서 위태롭게 춤을 추고 있는 것에 불과하다.

이 장에서는 어플리케이션 개발자의 시선에서 벗어나, 시스템 하부(Under the Hood) 커널 영역으로 침투한다. 다음의 핵심 메커니즘들을 메스(Scalpel)로 해부하듯 분석할 것이다.

  • 실시간성(Real-Time)의 본질: 왜 리눅스 대신 NuttX를 써야만 하는가? 데드라인 스케줄링의 원리.
  • NuttX 커널 구조: POSIX 표준 VFS(Virtual File System)와 메모리 관리 기법.
  • PX4 부스트스트랩(Bootstrap): 전원이 켜진 순간부터 메인 제어기가 돌기까지 5초간의 극비 초기화 스크립트 시퀀스.
  • 동기화 파동(Synchronization): 다중 스레드 환경에서 세마포어(Semaphore)와 뮤텍스(Mutex)로 공유 자원을 지켜내는 혈투.

NuttX의 스케줄링 원리를 지배하지 못하는 자는, 간헐적으로 수십 마이크로초 단위로 튀어 오르는 지연(Latency) 버그를 결코 잡아낼 수 없다. 비행 중 기체가 흔들리는 현상의 원인이 물리적 진동인지, 커널의 우선순위 역전(Priority Inversion) 인지를 꿰뚫어 보는 통찰력을 이 장에서 무장하라.
