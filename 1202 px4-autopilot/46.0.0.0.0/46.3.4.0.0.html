46.3.4. 비행 임무(FlightTask) 아키텍처 연동 및 기구학적 궤적 생성(Kinematic Trajectory Generation) 알고리즘

지상 관제 시스템(GCS) 또는 네비게이터(Navigator) 모듈로부터 단일 목적지 이동(Goto/Reposition)과 같은 고수준(High-level)의 위치 제어 명령이 전달되면, PX4-Autopilot의 위치 제어기(Position Controller)는 이를 부드럽고 물리적으로 추종 가능한 기구학적 궤적(Kinematic Trajectory)으로 변환해야 한다. PX4-Autopilot은 이러한 기능을 위해 비행 임무(FlightTask) 아키텍처를 채택하고 있다 (기반 펌웨어 버전: PX4-Autopilot v1.14 기준). 본 섹션에서는 FlightTask 아키텍처의 연동 구조와 부드러운 기동을 보장하는 기구학적 궤적 생성 알고리즘에 대해 분석한다.

1. 비행 임무(FlightTask) 아키텍처 설계

PX4의 비행 제어 스택에서 네비게이터(Navigator) 모듈과 다중 콥터 위치 제어기(mc_pos_control) 사이에는 논리적인 분리 계층이 존재한다. 네비게이터는 단순한 목표 지점(Setpoint Triplet)만을 제공하며, 위치 제어 모듈 내부에 존재하는 FlightTask 아키텍처가 이 지점들을 실시간 궤적으로 변환한다.

FlightTask 아키텍처는 다형성(Polymorphism)을 활용한 객체 지향적 설계 패턴을 따른다. 모든 궤적 생성 로직은 FlightTask라는 최상위 부모 클래스(Base Class)를 상속받아 구현된다. 대표적인 파생 클래스(Derived Class)는 다음과 같다.

  • FlightTaskManual: 조종기의 수동 입력(Manual Stick Input)을 받아 위치 및 속도 셋포인트를 생성한다.
  • FlightTaskAuto: 네비게이터에서 요구한 자율 비행(Mission, Reposition, RTL 등)에 대한 궤적을 셍성한다.
  • FlightTaskOffboard: 외부 동반 컴퓨터(Companion Computer, 예: ROS 2 환경)에서 전송된 셋포인트를 처리한다.

1.1 구성 요소 간 데이터 파이프라인(Data Pipeline)

명령이 하달되어 모터 제어로 이어지기까지의 단계는 다음과 같은 통합 연동 체계를 따른다.

graph TD
    A[Navigator Module] -->|position_setpoint_triplet| B(FlightTaskMapper)
    B --> C{Active FlightTask}
    C -->|FlightTaskAuto| D[Kinematic Trajectory Generator]
    C -->|FlightTaskManual| E[Manual Smoothing]
    D -->|vehicle_local_position_setpoint| F[Position Controller]
    F -->|vehicle_attitude_setpoint| G[Attitude / Rate Controller]
    G -->|actuator_controls| H[Mixer / Motors]

2. 기구학적 궤적 생성(Kinematic Trajectory Generation) 알고리즘

멀티로터 기체가 단순히 현재 위치에서 목표 위치로 선형 이동하도록 PID 제어기를 구성하면, 목표점 갱신 순간에 불연속적인 가속도(Acceleration)나 저크(Jerk: 가속도의 미분값)가 발생하여 기체가 심하게 튀는 현상(Twitching)이 발생한다. 이를 방지하기 위해 FlightTaskAuto 모듈 등에서는 저크 제한 궤적 생성기(Jerk-limited Trajectory Generator) 또는 연속 곡선 기반 보간(Interpolation)을 사용한다.

2.1 저크 제한(Jerk-limited) 기동의 원리

소프트웨어 구현체(WeatherVane, Trajectory 라이브러리 등)는 목표 지점 도달을 위해 시간에 따른 S자형 속도 프로파일(S-curve Velocity Profile)을 생성한다.

  1. 제약 조건 반영: 최대 속도(V_{max}), 최대 가속도(A_{max}), 최대 저크(J_{max}) 파라미터 제약 조건을 기반으로 설정한다.
  2. 운동학 방정식 미분/적분: 위치(Position)의 3차 미분인 저크 단위를 상수로 유지하면서, 이를 적분하여 가속도 \rightarrow 속도 \rightarrow 위치 프로파일을 연속적으로 도출한다.
  3. 실시간 재계산(Real-time Replanning): 비행 중 갑작스러운 외부 외란(Disturbance)이 발생하거나, 목적지가 동적으로 변경될 경우 진행 중인 궤적을 폐기하고 현재 기구학적 상태(위치, 속도, 가속도)를 초기 조건으로 하여 새로운 궤적을 실시간 스플라인 연산으로 부드럽게 재결합한다.

이를 통해 기체는 시작점과 끝점에서 속도와 가속도가 부드럽게 증가하고 감소하며, 기상 악화나 센서 노이즈가 있는 상황에서도 기체의 모터 마모를 줄이고 카메라 짐벌(Gimbal) 통제에 유리한 비행 안정성을 확보한다.

3. ArduPilot의 웨이포인트 네비게이션(Waypoint Navigation)과의 구조적 차이

ArduPilot의 아키텍처에서는 이러한 궤적 생성 기능이 AC_WPNav (Waypoint Navigator) 라이브러리에 강하게 결합되어 있다. AC_WPNav 모듈이 경로 생성(Path Generation)과 위치 추종(Position Tracking)의 책임을 동시에 가져, 코드가 집중되는 특징을 지닌다.

반대급부로 PX4-Autopilot은 FlightTask라는 인터페이스 층을 도입함으로써 **제어 법칙(Control Law)과 궤적 생성(Trajectory Generation)을 분리(Decoupling)**하였다. 이는 새로운 비행 모드나 자체적인 회피(Obstacle Avoidance) 로직, 혹은 대학 및 연구소 환경에서의 새로운 기구학적 모델링(예: Minimum Snap Trajectory 등)을 적용해야 할 때, 기존 Position Controller의 PID 루프 수정 없이 새로운 FlightTask 자식 클래스를 상속받아 구현할 수 있도록 하는 강력한 확장성(Extensibility)을 제공한다.
