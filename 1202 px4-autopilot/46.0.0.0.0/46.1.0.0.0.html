46.1 단일 목적지 이동(Goto/Reposition) 제어 시스템 아키텍처 및 선행 조건

본 절에서는 단일 목적지 이동(Goto/Reposition) 명령이 PX4-Autopilot(v1.14 기준) 내에서 어떠한 시스템 아키텍처를 거쳐 처리되는지 살펴보고, 해당 기능이 무결하게 실행되기 위해 요구되는 시스템적, 환경적 선행 조건(Pre-conditions)에 대해 심층적으로 분석한다.

1. 단일 목적지 이동 제어 시스템 아키텍처

PX4의 다계층 제어 아키텍처에서 단일 목적지 이동은 실시간 운영체제(NuttX RTOS) 위에서 동작하는 여러 모듈 간의 유기적인 메세지 교환으로 이루어진다. 이 구조는 외부 명령을 해석하고, 궤적을 계획하며, 액추에이터 제어 명령을 생성하는 세 가지 주요 논리적 블록으로 나뉜다.

1.1 아키텍처 토폴로지 및 uORB 통신망

단일 목적지 이동 명령이 하달될 때 활성화되는 핵심 모듈과 uORB(Micro Object Request Broker) 통신 파이프라인은 아래 다이어그램과 같다.

    TrajectoryGenerator -- vehicle_local_position_setpoint --> PosCtrl[Position Controller]:::ctrl
    PosCtrl -- vehicle_attitude_setpoint --> AttCtrl[Attitude Controller]:::ctrl
  1. 명령 수신부 (Command Receiver): QGroundControl(v4.3 기준)이나 MAVSDK는 MAVLink 프로토콜을 통해 MAV_CMD_DO_REPOSITION을 전송하며, 이는 PX4의 MAVLink 수신 모듈에서 파싱(Parsing)된다. ROS 2 환경에서는 uXRCE-DDS 브릿지를 통해 vehicle_command 구조체로 직렬화(Serialization)되어 유입된다.
  2. 커맨더 모듈 (Commander Module): 입력된 명령을 가장 먼저 수신하여 기체의 현재 비행 모드와 센서 상태를 바탕으로 명령의 수용 가능 여부를 평가한다.
  3. 내비게이터 모듈 (Navigator Module): 커맨더의 승인을 득하면, Reposition 임무를 위한 상태 기계(State Machine)를 가동한다. 목표 지점을 기반으로 이전(Previous), 현재(Current), 다음(Next) 웨이포인트 정보를 담은 position_setpoint_triplet 메시지를 발행한다.
  4. 위치 제어기 (Position Controller): 이동 간의 저크(Jerk)와 가속도를 제한하는 궤적 생성기(Trajectory Generator)를 활용하여 부드러운 이동 궤적을 산출하고, 이를 바탕으로 속도 및 자세 설정값(vehicle_attitude_setpoint)을 캐스케이드(Cascade) 제어기에 전달한다.

2. 제어 공학적 실행 선행 조건 (Pre-conditions)

단일 목적지 이동은 기체의 위치를 3차원 공간 상에서 전역적으로 즉시 변경하는 고권한(High-authority) 명령이므로, 펌웨어 내부적으로 엄격한 선행 조건 검증 로직을 거친다. 조건이 하나라도 누락되면 커맨더는 명령 거부(NACK)를 응답한다.

2.1 전역 위치 추정(Global Position Estimate)의 유효성 검증

가장 중요한 조건은 EKF2(Extended Kalman Filter) 또는 ECL(Estimation and Control Library)이 출력하는 위치 데이터가 유효해야 한다는 점이다.

  • GPS 3D Fix: 최소한 3D Fix 판정 및 설정된 EPH(Horizontal Position Error), EPV(Vertical Position Error) 임계값 이내의 정밀도가 요구된다.
  • VIO 연동: 외부 비전 통계(Visual Inertial Odometry)나 RTK GPS를 결합하는 경우, 센서 퓨전 엔진(Sensor Fusion Engine) 단계에서 vehicle_global_position 메시지가 지연 없이 정상적으로 퍼블리시되고 있어야 한다.

2.2 비행 상태 (Flight State) 전제 조건

  • 시동 상태 (Armed): 기체의 모터가 회전할 준비가 된 시동 상태여야 한다. Disarmed 상태에서 들어온 Reposition 명령은 완전히 무시된다.
  • 이륙 완료 (Takeoff Complete) 및 고도 조건: 기체가 지면에 있는 경우(Landed 상태) Reposition 명령은 수행되지 않는다. 이륙이 완전히 완료되어 공중에 체공 중(In-air)인 상태여야만 위치 이동을 개시할 수 있다.

2.3 소스 코드 레벨의 상태 검증 구조

다음은 PX4 소스 코드 내에서 선행 조건을 검증하는 의사 코드(Pseudo-code) 구성의 예이다. 이는 펌웨어 확장을 고려하는 GCS 개발자나 ROS 2 애플리케이션 개발자가 반드시 이해해야 하는 검증 절차이다.

bool is_reposition_valid(const vehicle_status_s &status, const vehicle_global_position_s &global_pos) {
    // 1. 시동 여부 확인
    if (status.arming_state != vehicle_status_s::ARMING_STATE_ARMED) {
        return false;
    }
    
    // 2. 체공 상태 확인
    if (status.hil_state != vehicle_status_s::HIL_STATE_ON && status.condition_landed) {
        return false;
    }

    // 3. 글로벌 위치 유효성 검사
    if ((global_pos.timestamp == 0) || !global_pos.xy_valid || !global_pos.z_valid) {
        return false;
    }

    return true; // 모든 선행 조건 만족
}

해당 검증을 통과하지 못한 명령은 MAVLink를 통해 MAV_RESULT_DENIED 또는 MAV_RESULT_TEMPORARILY_REJECTED 오류 코드와 함께 QGroundControl로 회신되며, 이 때 QGC는 운용자에게 오디오 모니터링 경고나 팝업 메시지 “Reposition command rejected“를 출력한다.

3. ArduPilot 아키텍처 대비 차별성

단일 목적지 이동 기능에 대해 PX4와 ArduPilot 펌웨어는 상태 전이 관점에서 명확한 설계 철학 차이를 보인다.

ArduPilot은 목적지 이동을 수행하기 위해 전통적으로 Guided 비행 모드로의 모드 변경(Mode Switch)을 요구한다. 즉, GCS는 1) SET_MODE 명령을 수반하여 기체를 Guided 모드로 진입시킨 후, 2) 위치 제어 목표 지점을 전송하는 두 단계의 트랜잭션을 거친다.

근래의 PX4 펌웨어 버전에서는 MAV_CMD_DO_REPOSITION 처리 모듈이 비행 모드의 상위에 해당하는 내비게이터(Navigator) 인스턴스로 편입되어, 기체가 Hold(대기) 모드나 Mission(자동 임무) 모드를 수행하던 중이라도 별도의 모드 변경 명령(SET_MODE) 없이 즉각적으로 이동을 수락(Override)하고 자동 비행 상태 기계를 변경하는 구조적 유연성을 지니고 있다. 이러한 인터럽트 주도형(Interrupt-driven) 아키텍처는 다수 기체의 군집 비행(Swarm) 제어나 ROS 2 기반의 동적 장애물 회피 알고리즘 구동 시 불필요한 통신 오버헤드를 줄이는 데 크게 기여한다.
