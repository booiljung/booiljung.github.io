46.3.3. 네비게이터(Navigator) 모듈 내 Reposition 상태 기계의 유한 상태 전이 루프 및 스레드 생명주기(Thread Lifecycle) 분석

단일 목적지 이동(Goto/Reposition) 제어 명령이 수신되면, PX4-Autopilot의 내부 아키텍처에서는 네비게이터(Navigator) 모듈이 이를 처리하게 된다. 네비게이터 모듈은 기체의 자율 비행 작업(Auto Flight Task)을 관장하는 핵심 컴포넌트이며, 이 중 Reposition 상태 기계(State Machine)는 기체가 현재 위치에서 새로운 목적지로 이동하여 호버링(Hovering)하게 만드는 제어 흐름을 관리한다. 본 섹션에서는 네비게이터 모듈 내 Reposition 상태 기계의 유한 상태 전이(Finite State Machine, FSM) 동작 원리와 관련된 스레드 생명주기(Thread Lifecycle)를 분석한다 (기반 펌웨어 버전: PX4-Autopilot v1.14 기준).

1. 네비게이터(Navigator) 모듈의 스레드 생명주기(Thread Lifecycle)

PX4-Autopilot은 실시간 운영체제(RTOS)인 NuttX 위에서 동작하며, 네비게이터 모듈 역시 독립적인 스레드(태스크)로 인스턴스화(Instantiation)되어 실행된다. 네비게이터 스레드의 생명주기는 생성(Creation), 초기화(Initialization), 메인 실행 루프(Main Execution Loop), 그리고 종료(Termination)의 단계로 나뉜다.

1.1 스레드 초기화 및 실행

시스템 부팅 시 또는 초기화 스크립트에 의해 네비게이터 모듈이 시작되면, Navigator::task_spawn() 메서드를 통해 새로운 스레드가 생성된다. 스레드가 할당되면 Navigator::run() 메서드가 메인 루프로서 지속적으로 실행된다.

메인 루프의 주요 주기는 다음과 같은 흐름을 따른다.

  1. uORB 메시지 대기(Poll): 네비게이터는 지속적으로 폴링(Polling)을 수행하여 vehicle_local_position, vehicle_global_position, vehicle_command 등의 주요 uORB 메시지가 업데이트될 때까지 스레드를 대기 상태로 둔다.
  2. 데이터 업데이트: 위치 데이터나 커맨드가 수신되면, _vehicle_command_sub.update() 등을 통해 최신 상태를 복사한다.
  3. 상태 기계 평가(Evaluate State Machine): 현재 활성화된 네비게이터 모드(Navigator Mode)의 상태 기계를 평가하고, 필요한 경우 모드 전이(Mode Transition)를 수행한다.
  4. 궤적 생성기(Trajectory Generator)로의 명령 전달: 활성화된 모드(Reposition, Mission, RTL 등)의 연산 결과를 기반으로 position_setpoint_triplet 메시지를 발행(Publish)하여 하위 제어기(비행 임무 아키텍처, FlightTask)로 전달한다.

2. Reposition 상태 기계의 유한 상태 전이(FSM) 모형

Reposition 모드는 단일 목적지로 이동 후 대기하는 단순한 목적을 갖지만, 안정성을 보장하기 위해 내부적으로 정교하게 정의된 유한 상태 전이(FSM) 구성을 따른다.

Reposition 모듈(Reposition 클래스)은 NavigatorMode라는 부모 클래스를 상속받아 구현되어 있으며, 주로 세 가지 핵심 콜백(Callback) 함수를 통해 생명주기가 관리된다.

  1. on_activation(): Reposition 모드가 최초로 활성화될 때 한 번 호출된다. 목적지 좌표 설정, 초기 궤적 생성기의 상태 초기화 등을 수행한다.
  2. on_active(): Reposition 모드가 활성화되어 있는 동안 네비게이터 메인 루프에 의해 주기적으로 호출된다. 여기서 기체와 목적지 간의 거리를 계산하고, 도달 여부(Arrival Determination)를 지속적으로 검사한다.
  3. on_inactivation(): 다른 모드(예: Mission, RTL)로 전이되거나 수동 조작 개입으로 모드가 종료될 때 호출되어, 리소스를 정리하고 상태를 리셋한다.

2.1 Reposition 상태 전이도(State Diagram)

다음은 Reposition 동작 과정에서의 내부 상태 전이도이다.

stateDiagram-v2
    [*] --> Idle : 시스템 부팅 및 초기화
    
    Idle --> Activation : Goto/Reposition 명령 수신 (uORB)
    
    state Activation {
        [*] --> Set_Destination : 목표 좌표 확인
        Set_Destination --> Check_Validity : 이동 가능 여부 (Geofence 등) 검증
        Check_Validity --> Publish_Setpoint : 초기 위치 셋포인트(Setpoint) 발행
    }
    
    Activation --> Active : 초기화 완료
    
    state Active {
        [*] --> Moving : 목적지를 향해 이동 중
        Moving --> Trajectory_Tracking : 주기적 궤적(Trajectory) 오차 보정
        Trajectory_Tracking --> Checking_Arrival : 도달 조건(Acceptance Radius) 확인
        Checking_Arrival --> Moving : 도달하지 않음
        Checking_Arrival --> Loitering : 목적지 반경 진입 완료
    }
    
    Active --> Inactivation : 타 비행 모드 변경 통보 또는 RC 개입
    Inactivation --> Idle : 상태 초기화 및 제어권 반환

Reposition 모드에 진입하면, 기체의 위치 제어기(Position Controller)를 위해 position_setpoint_triplet_s 타입의 uORB 토픽을 발행(Publish)한다. 이 토픽은 현재 위치(current), 이전 위치(previous), 다음 위치(next)의 세 가지 좌표를 포함할 수 있으며, Reposition의 경우 단일 목적지이므로 주로 current 항목이 갱신되어 전송된다.

3. 제어 루프 측면에서의 시간 복잡도와 설계 의도

네비게이터 스레드는 통상적으로 비행 제어기 루프(예: Rate Controller, Attitude Controller)보다 느린 약 10~50Hz 수준으로 실행된다. 즉, Reposition 상태 기계가 목적지에 대한 도달 판별(예: 이동경로 계산 및 상태 체크)을 수행하는 빈도는 위치 제어나 자세 제어 대비 저빈도로 수행된다.

이는 시스템 아키텍처 층위에서 관심사의 분리(Separation of Concerns) 원칙을 엄격하게 적용한 결과이다.

  • 고빈도(High-Rate) 제어 루프: 모터의 믹싱(Mixing) 및 자세/속도 보정은 FlightTask, Position Controller, Rate Controller 단계에서 수백 Hz로 매우 빠르게 수행된다.
  • 저빈도(Low-Rate) 의사결정 루프: 네비게이터는 상태 머신의 전이 허용 여부 점검, 현재 좌표 기반의 목표지점 도달 확인 등 전략적인 비행 논리(Flight Logic)에 집중하며 CPU 부하를 최적화한다.

4. ArduPilot의 Guided 모드와의 아키텍처 구조 비교

ArduPilot 아키텍처에서 단일 목적지 이동 기능은 주로 Guided 모드로 통합되어 실행된다. mode_guided.cpp 내에서 대상 위치(Target Location) 갱신 및 궤적 관리가 함께 이뤄지는 형태이다.

반면, PX4-Autopilot은 네비게이터(Navigator)라는 상위 수준의 경로 및 미션 매니저를 두어, Mission, RTL, Reposition 등의 각각의 비행 행동들을 NavigatorMode 클래스를 상속받은 객체 지향적(Object-Oriented) 구조로 분리하여 관리한다.
따라서 PX4를 기반으로 GCS(Ground Control Station) 확장을 구현하거나 상태 기계를 수정할 때에는, 전체 제어 루프가 아닌 Reposition 객체 내의 on_active() 루프만 별도로 격리하여 검증(Testing)할 수 있어 소프트웨어 공학적 유지보수성(Maintainability)이 뛰어나다.
