46.7.2.1. 기체 텔레메트리 스트림(Telemetry Stream)의 패킷 지연율(Packet Latency) 및 손실률(Drop Rate)을 보상하기 위한 잔여 거리(Remaining Distance) 실시간 선형 보간(Linear Interpolation) 알고리즘

지상 관제 시스템(GCS; Ground Control Station)에서 무인 항공기(UAS)의 상태를 실시간으로 모니터링하는 것은 임무 수행의 안전성과 직결된다. 특히 PX4-Autopilot 기반의 기체가 단일 목적지 이동(Goto)이나 연속적인 웨이포인트(Waypoint) 비행을 수행할 때, GCS는 기체로부터 MAVLink 프로토콜을 통해 전송되는 텔레메트리 스트림(Telemetry Stream)을 기반으로 현재 위치, 속도, 그리고 목적지까지의 잔여 거리(Remaining Distance)를 계산하여 사용자 인터페이스(UI)에 시각화한다.

그러나 무선 통신 환경, 즉 RF 데이터링크(RF Datalink) 영역에서는 본질적으로 패킷 지연율(Packet Latency)과 데이터 손실률(Drop Rate)이 발생할 수밖에 없다. 이러한 통신 품질 저하는 GCS 상에서 기체의 위치가 비연속적으로 뛰어넘듯 이동하는 스터터링(Stuttering) 현상을 유발하며, 이는 운용자로 하여금 기체가 목표 지점에 도달했는지 여부를 오판하게 만드는 주요 원인이 된다. 본 절에서는 QGroundControl(v4.3 이상) 및 맞춤형 관제 소프트웨어에서 이러한 텔레메트리 한계를 극복하고, 잔여 거리를 부드럽고 정확하게 산출하기 위한 실시간 선형 보간(Linear Interpolation) 알고리즘의 수학적 설계와 구현 방안을 심층 분석한다. (해당 기술 내용은 PX4-Autopilot v1.14 펌웨어와 MAVLink v2 통신 체계를 기준으로 설명한다.)

0.1 텔레메트리 패킷 지연 및 손실 현상의 분석

지상 관제 시스템과 PX4-Autopilot 간의 데이터 교환 구조에서, 기체의 위치 및 속도 정보는 주로 GLOBAL_POSITION_INT 또는 LOCAL_POSITION_NED MAVLink 메시지를 통해 전달된다. 일반적으로 이 메시지들은 5Hz에서 10Hz의 주기(Streaming Rate)로 전송된다.

패킷 지연시간(\tau)은 기체의 비행 제어기(Flight Controller)에서 데이터가 생성된 시점(Timestamp)부터 GCS 애플리케이션의 콜백 포트에 수신되어 파싱이 완료되는 시점까지의 시간차를 의미한다. 데이터 통신 링크(Wi-Fi, 텔레메트리 모뎀, LTE 등)의 대역폭 한계나 전파 간섭이 존재할 경우 패킷은 유실되거나 지연 연착된다.
알고리즘에서 패킷 유실을 감지하기 위해서는 연속된 두 메시지의 시스템 타임스탬프 차이 \Delta t_{msg}를 계산하고, 이것이 설정된 주기(\frac{1}{f_{stream}})를 초과하는지 모니터링해야 한다.

0.2 잔여 거리 산출을 위한 실시간 선형 보간 알고리즘 기하학

패킷 유실이 발생한 구간(Blind Spot) 동안 관제 소프트웨어는 기체가 정지해 있는 것으로 렌더링하는 대신, 직전 수신된 속도 벡터(Velocity Vector) 정보를 활용하여 기체의 추정 위치를 데드 레코닝(Dead Reckoning) 방식으로 보간해야 한다. 이는 학계의 “State Estimation and Localization for Autonomous Vehicles“와 같은 연구에서 제안하는 1차 마르코프 모델(First-order Markov Model) 기반의 상태 추정 기법에 착안할 수 있다.

기체의 직전 수신 위치 벡터를 \mathbf{X}_{t_0}, 직전 수신 속도 벡터를 \mathbf{V}_{t_0}라고 정의한다.
GCS의 최신 렌더링 프레임의 시점(Current Time)을 t_{now}라고 할 때, 수신 지연 시간 \Delta t는 다음과 같이 정의된다.
\Delta t = t_{now} - t_{0}

1차원 선형 보간식(Linear Interpolation Equation)을 통해 추정되는 기체의 현재 보상 위치 \mathbf{X}_{est}(t_{now})는 아래와 같이 수식화된다.
\mathbf{X}_{est}(t_{now}) = \mathbf{X}_{t_0} + \mathbf{V}_{t_0} \cdot \Delta t

여기서 목표 목적지(Target Waypoint)의 위치 벡터를 \mathbf{W}_{target}이라고 할 때, 관제 인터페이스 상에 표출되어야 할 보상된 잔여 거리 D_{rem\_comp}는 두 벡터 간의 유클리디안 거리(Euclidean Distance)로 도출할 수 있다.
D_{rem\_comp} = \Vert \mathbf{W}_{target} - \mathbf{X}_{est}(t_{now}) \Vert

0.3 통신 두절 대비를 위한 타임아웃 컷오프(Cutoff) 처리

단순한 선형 보간은 기체가 등속 직진 운동을 한다는 강한 가정을 기반으로 한다. 만약 기체가 실제로 방향을 선회(Turning)하거나 제동(Braking)을 걸고 있는 시점에 패킷 유실이 장기화될 경우, 속도 벡터 \mathbf{V}_{t_0}를 무한정 적용하면 추정 위치 \mathbf{X}_{est}가 실제 기체의 위치와 극심한 오차를 발생시키는 발산(Divergence) 현상을 초래한다.

따라서 GCS의 원격 측정 상태 동기화 기법에서는 최대 외삽 시간(Maximum Extrapolation Time Limit, T_{ext\_max})을 도입해야 한다. 일반적으로 T_{ext\_max}는 1초에서 2초 내외로 설정된다. \Delta t > T_{ext\_max} 조건이 성립되면 더 이상의 선형 보간을 강제 중지하고, GCS 인터페이스 상의 기체 아이콘을 흐리게(Dimming) 처리하거나 점멸시켜 운용자에게 통신 링크 지연 또는 상태 동기에 심각한 문제가 생겼음을 경고해야 한다.

0.4 QGroundControl 아키텍처 내 적용 메커니즘 및 시스템 흐름도

QGroundControl 내부는 C++ 기반의 백엔드와 QML 기반의 프론트엔드로 나뉜다. 텔레메트리 패킷 지연 및 선형 보간은 Vehicle 클래스의 내부 타이머 리스너(Timer Listener)를 통해 백그라운드 스레드에서 주기적으로 계산된다. 계산된 보간 위치 프로퍼티는 Qt의 시그널-슬롯(Signal-Slot) 아키텍처를 통해 QML의 FlightMap과 상태 위젯에 비동기로 전달된다.

다음은 패킷 유실 시 텔레메트리 보간을 수행하여 QGroundControl 내비게이션 상태를 업데이트하는 전체 데이터 흐름을 명시한 아키텍처 다이어그램이다.

0.5 C++ 구조체 기반 관제 소프트웨어 보간 로직 예시 (MAVLink 기반)

오픈 소스 관제 도구나 독자적 GCS에서 실시간 보간 로직을 구현할 때는 MAVSDK 프레임워크나 POSIX C/C++ 표준 라이브러리를 통해 시간 편차를 추출하여 반영한다. 다음은 패킷 지연 상황하에서 선형 보간을 지속적으로 산출해 내는 백엔드 타이머 슬롯 함수의 핵심 코드 예시다.

#include <chrono>
#include <cmath>

// 드론 상태 구조체 정의
struct VehicleState {
    double lat_deg;
    double lon_deg;
    float alt_m;
    float vx_m_s; // 속도 성분 북(North)
    float vy_m_s; // 속도 성분 동(East)
    float vz_m_s; // 속도 성분 하(Down)
    uint64_t last_update_timestamp_ms;
};

// 최근 수신된 실제 텔레메트리 상태 전역/멤버 변수
VehicleState last_known_state;
const uint64_t T_EXT_MAX_MS = 1500; // 최대 외삽 허용 시간 1.5초

// 위도를 미터로 근사 변환하는 상수 (적도 기준)
const double LAT_TO_METER = 111320.0;

void updateGCSDisplay() {
    uint64_t t_now = std::chrono::duration_cast<std::chrono::milliseconds>(
        std::chrono::system_clock::now().time_since_epoch()).count();
        
    uint64_t dt_ms = t_now - last_known_state.last_update_timestamp_ms;
    
    if (dt_ms > T_EXT_MAX_MS) {
        // 통신 장기 지연 발생: 보간 중단 및 링크 경고 시그널 발생
        triggerLinkWarning();
        return;
    }
    
    // dt_ms를 초 단위 지연 시간으로 변환 
    float dt_sec = static_cast<float>(dt_ms) / 1000.0f;
    
    // 선형 보간(Linear Interpolation) 적용 (단거리 평면(Flat Earth) 이동 근사)
    double estimated_lat = last_known_state.lat_deg + 
        ((last_known_state.vx_m_s * dt_sec) / LAT_TO_METER);
    double estimated_lon = last_known_state.lon_deg + 
        ((last_known_state.vy_m_s * dt_sec) / (LAT_TO_METER * cos(last_known_state.lat_deg * M_PI / 180.0)));
    
    // UI 업데이트 (산출된 좌표를 지도 렌더러에 반영)
    updateUIPosition(estimated_lat, estimated_lon);
}

0.6 Ardupilot 관제 철학과의 차이 및 시스템 강건성(Robustness) 고려사항

Ardupilot은 전통적으로 Mission Planner 등에서 수신된 원천 위치 데이터를 보간 모듈 없이 그대로 스크린 화면에 점차적으로 업데이트하는 보다 직관적이고 날것에 가까운 방식을 채택하는 경우가 많다. 이는 패킷 지연 발생 시 화면 상의 데이터 멈춤 현상(Lagging)을 엔지니어가 적나라하게 모니터링하고 시각적으로 직시하도록 설계된 철학에 기인한다.

반면, 현대 기조의 PX4-Autopilot과 밀접하게 디자인된 QGroundControl의 관제 철학에서는 조종자의 인지적 이질감을 최소화하고 부드러운 상태 추이에 초점을 맞추는 사용자 경험(UX) 최적화에 주안을 둔다. 그렇기 때문에 이와 같은 잔여 거리 및 좌표 보간 매커니즘이 더욱 가치있게 사용된다.

다만, 실제 운용 현장에서 GPS 글리치(GPS Glitch)나 외부 전자기 간섭으로 인해 MAVLink로 수신된 텔레메트리의 속도 성분(\mathbf{V}_{t_0})의 값이 순간적으로 튀거나 오염되어 있을 경우에는, 선형 보간 추이 또한 부정확한 궤적을 그리며 치솟는 오버슈트(Overshoot) 현상을 겪게 된다.
따라서 산업용 상용 GCS 설계 시에는 단순 1차원 선형 보간 로직에만 의존하지 않고, 관제 시스템의 프론트엔드 직전에 경량화된 칼만 필터(Lightweight Kalman Filter)나 저주파 통과 필터(Low Pass Filter/LPF)를 접목하여 속도 모델 노이즈를 평활화(Smoothing)하는 구조주의적 강건성 제어(Robustness Control) 설계를 병행해야 한다.
