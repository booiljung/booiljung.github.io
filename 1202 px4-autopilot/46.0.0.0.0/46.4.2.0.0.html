46.4.2. 3차원 유클리드 공간(Euclidean Space) 도달 판별을 위한 비선형 기하학적 교집합(Intersection) 알고리즘

단일 목적지 이동(Goto)이나 다중 웨이포인트(Mission) 비행을 수행할 때, 드론은 3차원 공간상의 한 지점(목표 위치)을 향해 비행한다. 기체가 해당 목표 지점에 ’도달’했음을 수학적, 시스템적으로 선언하기 위해서는 3차원 유클리드 공간(Euclidean Space) 내에서의 오차 거리를 기하학적 볼륨(Volume)으로 해석하는 과정이 필요하다. PX4-Autopilot은 단순한 3차원 구면식 거리 계산을 넘어, 수평과 수직의 한계를 분리시킨 원통형(Cylindrical) 구조의 비선형 기하학적 교집합(Intersection) 알고리즘을 채택하고 있다 (기반 펌웨어 버전: PX4-Autopilot v1.14 기준). 본 절에서는 이 알고리즘의 제어공학적 설계 의도 수리 물리적 모델을 분석한다.

1. 유클리드 공간(Euclidean Space) 상의 기하학적 제어 볼륨 정의

PX4-Autopilot에서 수용 반경(Acceptance Radius)은 하나의 3차원 구(Sphere) 형태가 아닌, 수평면 제어와 수직축 제어를 분리한 원통 모양(Cyllinder)의 제어 볼륨(Control Volume)으로 정의된다.

이러한 비선형적 볼륨 분리를 수행하는 이유는 멀티로터의 동역학적 비대칭성 때문이다. 멀티로터는 수평(XY) 이동 시 반드시 롤(Roll)/피치(Pitch) 기동이 수반되어 동역학적 반응이 복잡하지만, 수직(Z) 이동 시에는 모터의 총 추력(Thrust) 조절만으로 독립적인(Decoupled) 단위 제어가 가능하다. 따라서 도달 판정의 수용 한계 또한 x-y 평면 오차와 z축 오차를 엄격히 분리하여, 각각 독립된 매개변수(NAV_ACC_RAD, NAV_MC_ALT_RAD)로 설정하고 평가한다.

1.1 두 공간의 교집합(Intersection) 연산 구조

이 교집합 알고리즘을 논리적 회로로 표현하면 두 가지 독립된 조건의 논리곱(Logical AND) 연산으로 귀결된다.

  1. 수평면(XY-Plane) 조건: 현재 2차원(XY) 위치 곡면 좌표와 목표 좌표 간의 유클리드 벡터의 크기(Euclidean Norm)가 수용 반경 한계 내에 있는지 평가한다. 이는 무한한 높이를 갖는 가상의 기둥 영역(Cylindrical projection)을 의미한다.
  2. 수직축(Z-Axis) 조건: 현재 높이와 목표 고도 간의 절대 오차(Absolute Error)가 설정된 고도 수용 한계 내에 있는지 평가한다. 이는 무한한 면적을 갖는 가상의 상하 평행판(Parallel plates) 영역을 의미한다.

이 무한한 두 공간의 교집합(Intersection) 영역이 유한한 부피를 갖는 **최종 3차원 도달 허용 원통(Acceptance Cylinder)**이 된다.

2. 비선형 계산 복잡도 및 제어 루프 파급력

수학적으로 3차원 구(Sphere) 방정식 \sqrt{x^2 + y^2 + z^2} < R 단 하나를 계산하는 것 보다 두 개의 수식을 독립적으로 분리하여 연산(비선형 분기, Non-linear Branching)하는 것이 CPU 명령어 파이프라인 상 조건부 분기(Conditional Branch)를 증가시킬 여지가 있다.

그러나 PX4-Autopilot 체계 내에서 이 분리 연산은 다음과 같은 치명적인 예외 상황(Edge Case)을 방지하는 필수적인 역할을 수행한다.

  • Z축 우선 접근 상황의 보정: 만약 3차원 구 모델을 사용한다면 고도는 일치하지 않는데 (예: 수직 오차 2m), 수평 위치는 완벽히 일치하여 (XY 오차 0m), 식의 결과 반경이 2m로 계산될 수 있다. 이 때 만약 설정 반경이 2.5m라면, 고도를 다 맞추기도 전에 도달이 선언(False Positive)되어 다음 웨이포인트(Mission item)로 넘어가는 어처구니없는 참사가 발생한다.
  • XY와 Z의 동역학 응답 차이를 고려한 격리 효과 보장: 강풍 환경에서 기후에 저항하느라 XY 오차 수렴이 늦어지더라도, Z축 강하 임무를 끝까지 강제하도록 만든다.

3. ArduPilot 대비 3차원 좌표 데이터 처리 구조

ArduPilot 체계 역시 수직과 수평의 수용치를 다르게 두는 원칙(WPNAV_RADIUS) 자체는 매우 유사하게 가져간다.

차이점이 있다면, PX4-Autopilot의 경우 네비게이터(Navigator) 객체가 궤도(FlightTask 등)를 구성하기 위한 별도의 계층에서 이 모든 벡터 행렬(Matrix) 연산을 표준 Matrix 기법(Mathlib)으로 일상화하여, vehicle_local_position_setpoint 안의 (x,y,z) 행렬 튜플로 한 번에 처리한다는 점이다. 반면 구버전 혹은 일부 고정익 관련 ArduPilot 라이브러리들은 위/경도를 기반으로 하는 구면 삼각법(방위각, Haversine formula) 등 위도/경도를 스칼라 변수로 해체하여 직접 비교하는 성향이 강해 소스코드의 구조적 가독성이 달랐으나, 최근에는 EKF 기반 유클리드 오차로 거의 통합되어 체계적 유사성이 매우 높아진 상황이다.
