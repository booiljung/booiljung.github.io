46.2.4 지상 관제 시스템(GCS)과의 비동기 트랜잭션 제어 및 응답 패킷(COMMAND_ACK) 수신에 따른 유한 상태 기계(Finite State Machine) 상태 전이 모델

1. 개요

지상 관제 시스템(GCS; Ground Control Station)과 무인항공기(UAS) 간의 위치 제어 및 명령 하달은 무선 데이터 링크(Telemetry, Wi-Fi, LTE 등)를 매개로 이루어지며, 해당 링크는 태생적으로 패킷 손실(Packet Loss)이나 지연(Latency)이 발생할 수 있는 비신뢰성(Unreliable) 네트워크의 특성을 지닌다. 따라서 QGroundControl과 같은 GCS가 MAV_CMD_DO_REPOSITION 등과 같은 미션 혹은 제어 명령을 MAVLink 프로토콜을 통해 기체에 하달할 때, 시스템은 이를 동기적(Synchronous) 단일 호출 방식이 아닌 비동기 트랜잭션(Asynchronous Transaction) 형태로 관리해야 한다.

명령 트랜잭션의 신뢰성을 보장하기 위해, MAVLink 표준은 제어 측에서 COMMAND_LONG 또는 COMMAND_INT 패킷을 전송하면, 수신 측(비행 제어기)이 처리 성공 여부를 담은 COMMAND_ACK 패킷으로 응답하는 1:1 핸드쉐이크 체계를 갖추고 있다. PX4-Autopilot의 커맨더(Commander) 모듈은 이 비동기 트랜잭션 스트림을 감시하며, 수신된 명령의 가결성에 따라 기체의 유한 상태 기계(Finite State Machine, FSM)를 안전하게 전이(State Transition)시키는 코어 아키텍처를 내장하고 있다.

2. MAVLink 비동기 트랜잭션 파이프라인

QGroundControl 관제 모듈(Vehicle Manager)에서 Goto 명령 등의 액션이 트리거(Trigger)되면, 다음과 같은 제어공학적 비동기 트랜잭션 파이프라인이 가동된다.

  1. 명령 전송 및 타이머 시작: GCS는 대상 시스템(target_system), 컴포넌트(target_component), 명령 인덱스(command), 인자(param1 ~ param7)를 조립하여 COMMAND_INT 패킷을 브로드캐스트한다. 동시에 GCS 내부에 타임아웃(Timeout) 타이머를 구동한다.
  2. PX4 수신 및 유효성(Validity) 검증: MAVLink 수신 쓰레드(mavlink_receiver.cpp)가 이를 파싱하여 내부 uORB 메시지인 vehicle_command를 퍼블리시한다.
  3. Commander FSM 평가: 커맨더 데몬(src/modules/commander)이 명령을 인터셉트(Intercept)하고, 현재 기체 상태(Armed/Disarmed, 비행 모드 락팅 여부 등)와 좌표 유효성을 검사한다.
  4. COMMAND_ACK 응답: 처리 결과(MAV_RESULT)를 매핑하여 vehicle_command_ack uORB를 퍼블리시하면, MAVLink 송신 모듈이 이를 GCS로 반환한다.
  5. GCS 상태 동기화 및 재전송 로직: GCS가 지정된 타임아웃 시간(일반적으로 500ms~1000ms) 내에 COMMAND_ACK를 받지 못하면 명령을 재전송(Retry)하며, 최다 재시도(예: 3회) 실패 시 제어 링키지(Linkage) 유실 상태로 간주하고 UI에 사용자 경고를 스로잉(Throwing)한다.

3. PX4-Autopilot의 COMMAND_ACK 응답 구조

vehicle_command_ack에 실어 전송되는 COMMAND_ACK 페이로드(Payload)에는 핵심 상태 머신 전환 지표인 result 필드가 존재하며, 이는 다음 중 하나의 상수값을 지닌다.

  • MAV_RESULT_ACCEPTED (0): 명령이 정상적으로 수용되었으며, 즉각적으로 내비게이터(Navigator) 인스턴스에 의해 상태가 전이되고 실행된다.
  • MAV_RESULT_TEMPORARILY_REJECTED (1): 명령 자체의 문법은 유효하나 현 시점의 기체 FSM 상태(예: 이륙 전, 또는 하드웨어 초기화 중)로 인해 실행을 유보한다.
  • MAV_RESULT_DENIED (2): 잘못된 파라미터, 센서 캘리브레이션 누락, 제어 권한 탈취 상태 등으로 인해 명령이 즉거 거부되었다.
  • MAV_RESULT_UNSUPPORTED (3): 펌웨어에서 지원하지 않거나 인식할 수 없는 MAVLink Command ID이다.

PX4의 Commander 클래스는 입력받은 제어 변수가 FSM 전환 제약 조건을 통과할 때만 MAV_RESULT_ACCEPTED를 할당한다.

    state "Goto / Reposition State" as Goto_Mode_Active {

    }

    Goto_Mode_Active --> CmdValidation : 새로운 타겟 명령 인가

5. Ardupilot과 PX4-Autopilot의 FSM 및 응답 처리 설계 철학 비교

  • 상태의 원자성(Atomicity): PX4는 src/modules/commander/state_machine_helper.cpp에 명시된 엄격한 서브 함수를 통과하지 못하면 본래 FSM 모드 엔진을 절대 스위칭하지 않는다. 즉, 전이 자체가 원자적(Atomic) 트랜잭션으로 취급된다. 반면 Ardupilot은 제어 Mode 객체의 서브 클래스(ModeGuided, ModeAuto 등) 내부 init() 함수에서 전제를 판별하고 반환값(Boolean)에 따라 부드럽게 복호 처리하여 트랜잭션 유실에 조금 더 관대한 흐름을 가진다.
  • ACK 우선순위: PX4는 내비게이션(Navigator) 스레드가 스케줄을 처리하기 이전, 커맨더(Commander) 데몬이 즉각 파라미터를 스캔하고 vehicle_command_ack를 선행 퍼블리시하여 GCS가 블로킹(Blocking) 없이 명령 성공 여부를 확신하도록 설계된 이벤트 드리븐(Event-driven) 형태를 지향한다.

6. ROS2 연동: uXRCE-DDS의 비동기 트랜잭션 확장

자율 비행을 제어하는 오프보드 에이전트 환경(ROS2) 역시 동일한 비동기 트랜잭션 문제에 봉착한다. PX4의 px4_msgs::msg::VehicleCommand를 ROS2 노드 측에서 퍼블리시할 경우, 신뢰성 있는 QoS(Quality of Service, 예: RELIABLE, TRANSIENT_LOCAL)를 적용하더라도 미들웨어 레벨에서의 전달 보장이지, FSM 상태 기계 측면에서의 수락을 보장하는 것은 아니다.

따라서 ROS2 오프보드 애플리케이션 개발자는 px4_msgs::msg::VehicleCommandAck 토픽을 반드시 서브스크라이빙(Subscribing)하는 비동기 콜백(Callback)을 작성해야 하며, 자신이 부여한 command 필드와 매칭되는 메시지의 result 변수가 VehicleCommandAck.MAV_RESULT_ACCEPTED로 확인될 때까지 유한 상태 전이를 대기(Wait)하는 엣지 케이스 로직 구조를 구현해야 한다.

7. 결론

GCS 및 자율 에이전트와 PX4-Autopilot 비행 제어기 간의 통신은 이상적인 선형 데이터 흐름이 아니다. 이들 간격에는 타임아웃, 재전송 로직, 그리고 기체 커맨더의 까다로운 명령 거부 리포트가 내재된 비동기 트랜잭션 제어가 얽혀있다. 유효한 응답 패킷(COMMAND_ACK)의 코드를 철저하게 해석하고, 그에 동기화되는 Commander 중심의 유한 상태 기계(FSM) 전이 모델의 작동 생리를 이해하는 것은, 예측 불허의 상황에서도 신뢰성 있게 비행하는 강건한 관제 시스템 및 MAVSDK/ROS2 애플리케이션을 성공적으로 구현하기 위한 프로그래밍의 핵심 전제 지식이다.
