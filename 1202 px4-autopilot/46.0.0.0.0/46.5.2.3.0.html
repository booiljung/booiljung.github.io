46.5.2.3 교착 상태 타개를 위한 강제 도달 인정(Forced Arrival Acceptance) 타임아웃(Timeout) 릴레이 트리거(Relay Trigger) 기법

무인항공기(UAV)가 단일 목적지 이동(Goto/Reposition) 비행을 수행할 때, 강풍(Headwind)과 같은 대기 환경적 외란(Wind Disturbance)이나 기체의 추력 한계 초과 등으로 인해 설정된 수용 반경(Acceptance Radius) 내로 진입하지 못하는 상황이 발생할 수 있다. 이러한 상태가 지속되면 기체는 목표 위치에 도달하기 위해 무한히 에너지를 소모하며, 다음 임무나 명령으로 전환되지 못하고 특정 위치에 갇히는 **교착 상태(Deadlock)**에 빠지게 된다.

본 절에서는 이러한 제어 루프 정상 상태 오차(Steady-State Error)로 인한 교착 상태를 타개하기 위해, PX4-Autopilot(펌웨어 v1.14.0 기준) 및 지상 관제 시스템(QGroundControl v4.3.0 기준) 환경에서 적용되는 강제 도달 인정(Forced Arrival Acceptance) 및 타임아웃 릴레이 트리거(Timeout Relay Trigger) 기법의 소프트웨어 공학적 설계와 제어 논리를 심층 분석하라.

1. 교착 상태(Deadlock) 타개의 제어공학적 필요성

목표 웨이포인트(Waypoint)나 단일 목적지에 도달했음을 판별하는 기본 기준은 삼차원 유클리드 거리(3D Euclidean Distance)가 시스템에 정의된 허용 반경(예: 파라미터 NAV_ACC_RAD) 이내로 들어왔는지를 확인하는 것이다. 하지만 기체가 수용 반경 경계선 바로 밖에서 제자리 비행(Hovering)을 유지하며 오차를 좁히지 못하는 현상이 발생하면, 비행 제어기(비행 관리자; Navigator 또는 Commander 모듈) 내의 유한 상태 기계(Finite State Machine, FSM)는 \text{State} \neq \text{REACHED} 로 판별하여 로직이 전진하지 못한다.

이를 해결하기 위해 위치 오차를 허용하면서도 다음 비행 시퀀스로 안전하게 넘어가기 위한 비선형적 릴레이 감지 구조(Non-linear Relay Detection Scheme)가 도입되었다. 이는 주로 시간적 한계(Time Limit) 기법과 에너지/속도 감쇠 모델을 기반으로 설계된다.

2. PX4-Autopilot의 타임아웃 릴레이 트리거 메커니즘

PX4의 Navigator 모듈에서는 위치 도달을 검증하는 로직에 공간적 제약(Spatial Constraint)뿐만 아니라, 시간적 제약(Temporal Constraint)을 보완적으로 결합한다.

2.1 속도 감쇠 모니터링을 통한 정체 감지 (Velocity Decay Monitoring)

PX4는 단순히 시간만을 측정하는 것이 아니라, 기체가 목표점를 향해 이동하려는 의도(속도 벡터)를 상실했는지를 함께 평가한다. 목표 지점 근처에 도달했음에도 전진 속도(Forward Velocity)가 특정 임계값(NAV_MC_ALT_RAD나 최소 이동 임계치 파라미터들과 연동) 이하로 오랜 시간 유지될 경우, 제어기는 기체가 외란에 의해 물리적 한계에 봉착했음을 인지한다.

2.2 강제 도달(Forced Acceptance) 검증 로직 흐름

다음의 Mermaid 상태 다이어그램은 PX4 내부에서 구현된 타임아웃 기반 강제 도달 인정(Forced Arrival Acceptance) 판별 알고리즘의 동작 과정을 나타낸다.

stateDiagram-v2
    [*] --> Goto_Moving : Goto 명령 수신 및 비행 시작
    Goto_Moving --> Distance_Check : 위치 추정 데이터 수신
    
    state Distance_Check {
        direction LR
        Check_R(반경 내 진입 여부)
        Check_T(목표 근처 정체 시간 감지)
    }

    Distance_Check --> Reached_Normal : ||p_err|| < NAV_ACC_RAD
    Distance_Check --> Timeout_Check : ||p_err|| >= NAV_ACC_RAD

    state Timeout_Check {
        direction TB
        Check_Spd : ||v_est|| < Threshold 감지
        Check_Timer : 정체 타이머(t_stalled) 증가
        Limit_Eval : t_stalled > Timeout_Max ?
        
        Check_Spd --> Check_Timer
        Check_Timer --> Limit_Eval
    }

    Timeout_Check --> Goto_Moving : t_stalled <= Timeout_Max (비행 계속)
    Timeout_Check --> Reached_Forced : t_stalled > Timeout_Max (릴레이 트리거 발동)

    Reached_Normal --> [*] : 다음 상태(Hold 또는 Mission)로 전환
    Reached_Forced --> [*] : 강제 도달 인정 후 다음 상태로 전환

이 알고리즘에서 **릴레이 트리거(Relay Trigger)**는 FSM이 무한 루프에 빠지는 것을 차단하는 일종의 ‘소프트웨어 차단기(Software Circuit Breaker)’ 역할을 수행한다. 목표물 접근 후 특정 반경 외곽(R > R_{acc})에서 기체가 전진하지 못하는 상태(v \approx 0)가 Timeout_Max 이상 지속되면, FSM은 현재 궤적점을 생략(Skip)하거나 도달한 것으로 간주하여 시스템의 전체 임무 연속성을 보장한다.

3. Ardupilot의 대응 기법과의 제어 아키텍처 비교

두 오픈소스 비행 제어 스택은 교착 상태를 타개하는 철학에서 명확한 차이를 보인다.

비교 항목PX4-AutopilotArdupilot (Copter)
강제 도달(Forced Arrival) 판별 철학위치 제어 루프와 독립적인 Navigator의 상태 기계에서 도달 거리와 상태 유지 시간을 종합하여 도달 여부(is_mission_item_reached)를 판단. 기계적이고 엄격한 조건 트리거 적용.고정익기(Plane)의 L1 내비게이션이나 멀티로터의 WP_YAW_BEHAVIOR 및 웨이포인트 크로스 트랙 에러(Crosstrack Error) 지연(Delay) 파라미터(WP_NAVALT_MIN 등)를 통해 궤적 상의 정체를 보다 유연하게 넘기려 시도.
타임아웃 파라미터 매핑MIS_YAW_ERR와 같이 헤딩 및 위치 정렬에 대한 타임아웃 조건을 부여하거나, 특정 임무 고도 도달 불능 시 자동 타임아웃을 거쳐 다음 항목으로 이동하는 명시적 릴레이 구조 구비.웨이포인트 간 이동 경로 중 WP_RADIUS 및 WP_ACCEL 제약을 통해 물리적 도달 불가 지점을 회피하는 ‘플라잉 쓰루(Flying Through)’ 경로 최적화 기법을 우선 사용.
MAVLink 이벤트 발생 유무강제 도달 조건 만족 시 즉시 MISSION_ITEM_REACHED 메시지를 방출하여 GCS와 상태를 동기화.정체된 경우 Failsafe(바람 초과 등) 메시지를 발생시키면서 오퍼레이터의 개입을 유도하는 경향이 강함.

PX4는 Ardupilot 대비 내비게이터 모듈 내부에서 강제 타임아웃 릴레이를 상태 변경의 트리거 조건으로 적극 채택하고 있으며, 이는 대단위 군집 비행(Swarm)이나 로봇 운영 체제(ROS2) 연동 시 하나의 기체로 인해 전체 미션 파이프라인이 정지하는 현상을 방지하는 데 더 적합한 구조를 지닌다.

4. GCS (QGroundControl) 측면의 교착 상태 관제 및 개입

임베디드 비행 제어 시스템 내부에서 릴레이 트리거가 발동하기 이전, 또는 설정된 타임아웃 임계값이 지나치게 길게 설정된 경우, 지상 관제 시스템(QGroundControl)과 원격 조종자는 이를 선제적으로 타개할 수 있어야 한다.

4.1 지상 관제 시스템(GCS) 기반 관찰 및 원격 개입 (Human-in-the-Loop)

MAVSDK나 QGroundControl을 통해 개발된 맞춤형 GCS는 MAVLink LOCAL_POSITION_NED (메시지 ID: 32)와 VELOCITY_ESTIMATE 데이터를 10Hz 이상의 주기로 수신한다.

  1. 에러 튜빙(Error Tubing) 모니터링: 궤적 제어 중 요구 위치(Position Setpoint)와 현재 위치의 편차 놈(Norm) \vert\vert \mathbf{p}_{err} \vert\vert의 미분값이 0에 수렴하는 현상을 감지한다.
  2. MAVLink 개입: QGroundControl의 백엔드(C++) 매니저는 교착 상태 감지 시, COMMAND_LONG 메시지를 이용해 다음 웨이포인트로 명시적 점프(Set Current Item)를 명령하거나, 현재 위치를 기반으로 하는 새로운 Reposition 명령(예: MAV_CMD_DO_REPOSITION, ID: 192)을 발행하여 교착 상태 FSM을 강제로 초기화(Reset) 시킨다.

4.2 자율 에이전트 비행을 위한 ROS2 연동

마이크로 XRCE-DDS(uXRCE-DDS) 브리지를 통해 연결된 ROS2 Offboard 제어 환경에서는 로컬 플래너(Local Planner)가 상태 기계를 관리한다.
ROS2 노드 측에서는 VehicleLocalPosition uORB 토픽의 동적 변화율을 모니터링하여, 이동 명령 하달 후 T_{timeout} 초간 목표 지점 반경 반경 이내로 수렴하지 않으면 Offboard 모드 상태 머신 자체에서 새로운 경로 궤적을 재계획(Path Re-planning)하여 회피 혹은 우회 궤적 세트포인트를 퍼블리시(Publish) 하도록 아키텍처를 설계해야 한다.

5. 결론

Goto 및 이동 명령 수행 중 발생하는 환경적 외란에 의한 교착 상태(Deadlock)는 단순한 오버슈트 문제를 넘어 전체 시스템 무결성을 위협한다. PX4-Autopilot은 위치 및 속도의 정상 상태 모니터링과 타임아웃 릴레이 트리거 연동을 통해 수용 반경 내 진입이라는 엄격한 공간적 조건을 시간적 개념과 융합하여 안전하게 우회(Bypass)한다. 제어 엔지니어는 기체의 추력 성능과 운영 환경의 극한 풍속 등을 고려하여 시간 임계치 및 속도 감쇠 판단 계수를 통합적으로 캘리브레이션해야 하며, ROS2나 GCS를 통한 계층적 Failsafe 관제 로직을 함께 구현해야만 진정한 의미의 강건한 자율 에이전트 운용이 가능하다.
