46.3.4.1. FlightTaskAutoMapper를 이용한 객체 지향적 다형성(Polymorphism) 기반 자동 비행 태스크 동적 할당 및 인스턴스화(Instantiation) 로직

단일 목적지 이동(Goto/Reposition)을 포함한 자동 비행 명령이 네비게이터(Navigator)로부터 하달될 때, PX4-Autopilot의 위치 제어기(Position Controller)는 목적지의 특성이나 비행 모드에 따라 궤적 생성 전략을 동적으로 변경해야 한다. 이를 위해 PX4 아키텍처는 C++의 객체 지향 프로그래밍(Object-Oriented Programming, OOP) 원리인 다형성(Polymorphism)을 활용하여, 실행 시간에 알맞은 비행 임무(FlightTask) 객체를 생성하고 할당한다. 본 섹션에서는 FlightTaskAutoMapper 클래스를 중심으로 한 동적 할당 및 인스턴스화(Instantiation) 로직을 분석한다 (기반 펌웨어 버전: PX4-Autopilot v1.14 기준).

1. FlightTaskAutoMapper의 역할 및 계층 구조

FlightTaskAutoMapper는 팩토리 패턴(Factory Pattern) 또는 매핑(Mapping) 레이어에 해당하는 핵심 클래스이다. 네비게이터에서 전달받은 position_setpoint_triplet 메시지 내부의 타입(Type) 플래그를 분석하여, 어떠한 세부 FlightTaskAuto 하위 클래스가 현재 비행 상태에 적합한지 결정하는 논리적 스위치 역할을 수행한다.

FlightTaskAuto 자체는 자동 비행 궤적 생성의 기반이 되는 부모 클래스(Base Class)이며, 이를 상속(Inheritance)받은 구체적인 임무 클래스들은 다음과 같다.

  • FlightTaskAutoLine: 두 웨이포인트(Waypoint) 사이를 선형(Linear)으로 부드럽게 연결하는 기본적인 궤적 생성기.
  • FlightTaskAutoLineSmooth: FlightTaskAutoLine을 확장하여, 경유점을 멈추지 않고 곡선으로 지나가는 평활화(Smoothing) 로직을 포함.
  • FlightTaskAutoMapper: 위와 같은 다양한 FlightTask를 런타임에 결정하고 다형적으로 호출하기 위한 스위칭 클래스.

1.1 매퍼(Mapper)의 동적 할당(Dynamic Allocation) 흐름

기체가 새로운 셋포인트(Setpoint)를 수신하면, FlightTaskAutoMapper::update()가 호출되어 내부적으로 타입 검증을 실시한다. 다음의 상태 검사를 거쳐 알맞은 인스턴스로 분기(Branching)한다.

  1. 셋포인트 유효성 검사: position_setpoint_triplet.current의 유효성(valid)을 검사한다.
  2. 임무 유형(Type) 식별: 이륙(Takeoff), 착륙(Land), 배회(Loiter), 단순 이동(Position/Goto) 중 현재 셋포인트의 모드를 파악한다.
  3. 인스턴스 활성화(Active Instance): _task라는 다형적 포인터 인스턴스(Polymorphic Pointer Instance)가 현재 요구사항에 맞춰 알맞은 자식 클래스의 메서드를 가리키게 하여(Virtual Function 오버라이딩 활용), 궤적 연산을 위임(Delegation)한다.

2. 객체 지향적 다형성(Polymorphism)을 통한 아키텍처 추상화

다음은 FlightTaskAutoMapper가 셋포인트를 해석하여 궤적을 갱신하는 일련의 과정을 나타내는 시퀀스 전이도이다.

sequenceDiagram
    participant Nav as Navigator
    participant Ctrl as Position Controller
    participant Mapper as FlightTaskAutoMapper
    participant Task as Active FlightTaskAuto (e.g. Line)
    
    Nav ->> Ctrl: Publish `position_setpoint_triplet`
    Ctrl ->> Mapper: updateInitialize()
    Mapper ->> Mapper: Check setpoint type (Goto, Loiter, etc.)
    alt Type == Position (Goto)
        Mapper ->> Task: setLineTask() 및 포인터 맵핑
    else Type == Velocity
        Mapper ->> Task: setVelocityTask()
    end
    Ctrl ->> Mapper: update()
    Mapper ->> Task: update() <br/> (동적 인스턴스의 다형적 연산)
    Task -->> Ctrl: 반환: vehicle_local_position_setpoint

위의 로직에서 보듯, 최상위 제어기(mc_pos_control)는 FlightTaskAutoMapper 내부에 어떤 세부적인 자식 클래스가 할당되어 수식을 풀고 있는지 알 필요가 없다. 단지 update() 멤버 함수만을 반복 호출할 뿐이다.
이러한 다형성(Polymorphism) 기반 추상화(Abstraction) 덕분에, ArduPilot의 거대한 분기문(예: switch-case 지옥)에 비해 PX4-Autopilot의 코드는 결합도(Coupling)가 낮고 응집도(Cohesion)가 높아져 GCS 명령에 대응하는 새로운 비행 곡선 방정식(예: 베지에 곡선, B-스플라인)을 플러그인(Plug-in) 형태로 손쉽게 추가할 수 있다.

3. 동적 인스턴스화 과정의 C++ 컴파일러 레벨 이해

네비게이터 모듈로부터 Goto(Reposition) 명령이 도착하면, 매퍼는 초기화 루틴인 on_activation() 단계를 거치면서 객체 포인터를 정렬한다. 다만 하드웨어 리소스가 제한적인 임베디드 실시간 운영체제(RTOS) 환경에서는 메모리 파편화(Memory Fragmentation)를 방지하기 위해, 매번 new 및 delete 키워드로 런타임 동적 할당을 하기보다는, 내부 메모리 풀(Memory Pool) 또는 유니온(Union)/배치 뉴(Placement New) 기법을 활용하여 안정적으로 인스턴스의 메모리를 관리한다. 이를 통해 멀티로터의 고속 제어 주파수(약 250Hz) 내에서 객체의 생성과 소멸로 인한 시간 지연(Latency) 현상을 원천적으로 차단한다.
