46.3.2 제어 명령의 uORB 메시지 발행/구독(Publish/Subscribe) 흐름 및 위상 동기화(Phase Synchronization) 추적

1. 개요

무인항공기(UAS)의 제어 시스템은 GCS(Ground Control Station)나 자율 에이전트로부터 전달받은 상위 명령(High-level Command)을 모터의 물리적 회전수(RPM)로 변환하는 복잡한 데이터 파이프라인(Data Pipeline)을 가진다. PX4-Autopilot 아키텍처의 가장 큰 핵심은 이 모든 모듈 간의 통신이 uORB(Micro Object Request Broker) 메커니즘을 통해 발행(Publish)과 구독(Subscribe)의 비동기적 형태로 이루어진다는 점이다.

서로 다른 주파수(Frequency)로 동작하는 수십 개의 태스크(Task) 모듈이 동일한 비행 상태를 제어하기 위해서는, 데이터가 이동할 때마다 오차 누적이나 데이터 레이스(Data Race)가 없도록 **위상 동기화(Phase Synchronization)**를 유지하는 것이 치명적으로 중요하다. 본 문서에서는 수신된 제어 명령이 uORB 파이프를 관통하며 어떻게 위상을 조율하고 하위 제어기(Low-level Controller)로 흘러가는지 그 심층적인 구조를 분석한다.

2. uORB 제어 명령 발행/구독(Pub/Sub) 핵심 흐름

QGroundControl 또는 ROS2 연동 오프보드(Offboard) 노드에서 타겟 목적지로의 이동을 지시하는 MAV_CMD_DO_REPOSITION 제어 명령이 수신되었을 때, uORB 토픽의 동적 흐름은 다음과 같은 다단계 발행/구독 사이클을 거친다.

  1. MAVLink 수신 및 디코딩 (mavlink_receiver): COMMAND_INT 패킷이 수신되면 MAVLink 모듈이 이를 파싱하여 vehicle_command uORB 토픽으로 발행한다.
  2. 커맨더 모듈의 상태 인지 (commander): commander 데몬이 vehicle_command를 구독하여 FSM 권한을 검증한 후, 이상이 없으면 vehicle_command_ack로 GCS에 응답하고, 내비게이터(Navigator) 상태 트리거를 가동한다.
  3. 경로 계획 생성 (navigator -> flight_task): navigator 스레드는 제어권을 넘겨받고, 현재 EKF2의 vehicle_local_position 토픽을 구독하여 출발점을 동기화한다. 이후, 부드러운 이동 궤적을 묘사하는 타겟 추종 데이터인 trajectory_setpoint 토픽을 약 50Hz의 주기로 발행한다.
  4. 위치 및 자세 제어 루프 (mc_pos_control -> mc_att_control): 위치 제어기(mc_pos_control)는 trajectory_setpoint를 구독해 속도/위치 PID 연산을 수행하여 vehicle_attitude_setpoint(자세 목표치)를 퍼블리시한다. 이를 바탕으로 자세 제어기(mc_att_control)가 250Hz 이상의 고속 주기로 actuator_controls를 산출해 낸다.

3. 서로 다른 주사율 간의 위상 동기화(Phase Synchronization) 추적

위 흐름에서 가장 난해한 소프트웨어 공학적 도전은 **비동기성(Asynchrony)**이다. navigator는 50Hz로 목표치를 내뱉지만, 하위의 mc_att_control은 250Hz, 혹은 IMU 센서 드리프트(sensor_gyro)는 1000Hz 주기로 작동한다.

3.1 타임스탬프(Timestamp) 기반 위상 정렬

PX4는 모든 uORB 토픽 데이터 구조체(C struct)의 첫 번째 맴버로 64비트 정수형 마이크로초 단위의 timestamp를 의무적으로 포함시킨다. 이는 단순한 이벤트 로깅을 넘어 하위 제어기가 상위 제어기의 제어 주기를 파악하게 하는 척도이다.

  • 제어 데이터의 Freshness 검증: mc_pos_control은 trajectory_setpoint를 Poll(폴링)할 때마다 최신 토픽의 timestamp를 자신이 마지막으로 연산한 시점과 비교한다. Delta-T (\Delta t)가 비정상적으로 길어지면 제어 명령이 지연도(Staleness)를 겪고 있다고 판단하여 페일세이프(Failsafe) 멈춤(Hovering) 로직을 가동한다.

3.2 블로킹 Poll 및 깨우기(Wake-up) 메커니즘

CPU 사이클 낭비를 막고 즉각적인 위상 대기를 위해 NuttX RTOS의 파일 식별자(File Descriptor) poll 체계를 도입한다.

  • 센서나 상위 제어기의 uORB 발행 행위는 근본적으로 OS 레벨의 write() 함수와 매핑되며, 이는 서브스크라이버 태스크들의 이벤트를 큐잉하여 인터럽트처럼 대기 중인(blocking) 태스크를 즉각 깨운다(poll() 리턴). 이 방식으로 1000Hz의 센서 데이터를 딜레이 없이 1000Hz로 추종하는 제어 쓰레드 위상 락(Phase Lock)이 형성된다.

5. Ardupilot의 스케줄러 보상 메커니즘과의 구조적 차이점

  • PX4-Autopilot: 완전한 다중 쓰레드/태스크 모델을 차용하여 데이터를 전달받는 수신처 쓰레드가 uORB의 poll() 타임아웃 락킹(Locking)을 통해 동적으로 스스로의 위상을 정렬한다. 이는 RTOS의 선점형 스케줄러(Preemptive Scheduler)의 역량을 극대화하는 방식이다.
  • Ardupilot: 단일 메인 루프(Single Main Loop) 기반의 코루틴(Coroutine) 스케줄러를 선호한다. 미리 정의된 스케줄 타임 테이블(AP_Scheduler 테이블 구조체)에 맞추어 Fast Loop(예: 400Hz 레이트 제어) 안에 특정 조건에서만 분기되는 Slow Loop(예: 50Hz 위치 제어, MAVLink 수신) 함수를 호출하는 고정형 페이징 구조를 지향한다. PX4보다 병렬성은 낮으나 제어 연산 순서에 대한 직관성이 높고 디버깅이 쉬운 아키텍처이다.

6. 결론

PX4의 uORB 발행/구독 아키텍처는 제어 명령을 즉각적이고 유연하게 계층화시키는 훌륭한 디자인 패턴이다. 비동기식 메시지 통신의 한계로 지적될 수 있는 제어 스텝 지연(Phase Lag) 및 데이터 레이스 가능성은 RTOS 기반의 정교한 파일 시스템 Polling과 타임스탬프-\Delta t 추적 알고리즘을 통해 수학적으로 엄밀히 보상 및 동기화된다.

특히 ROS2 환경에서 Micro XRCE-DDS 래퍼(Wrapper)를 통해 수천 킬로미터 밖의 원격 클라우드 컨테이너 명령을 직접적으로 이 uORB 파이프 내부에 융합(Integration)하더라도, 코어 제어기는 이 위상 동기화 트리거 구조 덕분에 전혀 이질감 없이 250Hz 이상의 정지 비행 퍼포먼스를 유지할 수 있게 된다.
