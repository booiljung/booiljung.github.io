46.7.2.2. 목표 지점 도달 전이 상태(Transitional State)의 이산적(Discrete) 시각화 렌더링(Visual Rendering) 및 도달 성공/실패 여부에 따른 비동기 이벤트 리스너(Asynchronous Event Listener) 설계

지상 관제 시스템(GCS; Ground Control Station)에서 무인 항공기(UAS)에 단일 목적지 이동(Goto/Reposition) 명령을 하달했을 때, 조종자는 기체가 안전하게 장애물을 회피하며 최종 목적지에 무사히 안착했는지 시각적으로 명확하게 파악할 수 있어야 한다. 이를 위해 QGroundControl(이하 QGC)을 비롯한 고도화된 관제 소프트웨어는 기체의 연속적인 물리적 이동 궤적을 몇 가지 단계로 나누어 이산적(Discrete)인 전이 상태(Transitional State) 모델로 정의한다.

본 절에서는 기체가 목적지로 이동하는 과정의 전이 상태를 어떻게 정의하고 이를 프론트엔드에 시각화(Visual Rendering)하는지 그 구조적 접근법을 분석한다. 아울러, PX4-Autopilot 비행 제어기와의 비동기 통신 환경에서 도달 성공(Reached) 또는 실패(Failed/Aborted) 이벤트를 견고하게 감지하고 처리하기 위한 비동기 이벤트 리스너(Asynchronous Event Listener)의 아키텍처 설계 기법을 상세히 다룬다.

0.1 전이 상태(Transitional State)의 이산적 모델링

공기역학적 외란과 통신 지연 속에서 연속적으로 변하는 기체의 위치 정보를 조종자에게 그대로 노출하는 것은 인지적 과부하를 초래한다. 따라서 GCS 내부의 비행 제어 관리자(Flight Manager) 로직은 기체의 현재 비행 단계를 유한 상태 기계(Finite State Machine) 형태의 이산적 모델로 추상화한다.

Goto 미션의 전이 상태는 일반적으로 다음과 같이 이산화할 수 있다.

  1. STATE_IDLE (대기 상태): 기체가 정지해 있거나, 현재의 위치 제어(Position Control) 모드에서 새로운 목적지 명령을 대기하는 상태.
  2. STATE_IN_TRANSIT (이동 중 전이 상태): Goto 명령(e.g., MAVLink MAV_CMD_DO_REPOSITION)이 승인(ACK)된 직후, 실제 기동을 시작하여 목표 지점을 향해 순항하는 상태.
  3. STATE_APPROACHING (도달 임박 상태): 기체와 목표 지점 간의 거리가 설정된 감속 반경(Deceleration Radius) 내로 진입하여, 기체의 속도 벡터가 줄어들고 제동(Braking) 메커니즘이 활성화된 상태.
  4. STATE_REACHED (도달 완료 상태): 시스템 수용 반경(Acceptance Radius; PX4 파라미터 NAV_ACC_RAD) 이내로 진입한 후 위치 유지가 안정화되었다고 판별된 상태.
  5. STATE_ABORTED (도달 실패/취소 상태): 조종자의 개입, 배터리 부족 시나리오 장착(Failsafe RTL), 통신 단절(Link Loss), 혹은 심각한 장애물 조우로 인해 이동이 강제 취소된 상태.

0.2 이산적 시각화 렌더링(Discrete Visual Rendering) 기법

각 전이 상태에 따라 GCS의 지도 인터페이스(Map Interface) 상에 렌더링되는 그래픽 컴포넌트의 특성이 역동적으로 변화해야 운용자의 직관을 극대화할 수 있다. QGC의 경우 QML 기반의 선언적 UI 프레임워크를 활용하여 다음과 같은 렌더링 전략을 취한다.

  • 목표 지표(Target Indicator) 렌더링: STATE_IN_TRANSIT 상태에서는 목적지를 지시하는 아이콘 주변에 진행률(Progress)을 나타내는 점선 궤적이나 투명도(Opacity) 펄스 애니메이션이 가동된다.
  • 수용 반경(Acceptance Radius)의 시각화: STATE_APPROACHING 상태에 진입하면 목표 지점을 중심으로 MapCircle 요소가 렌더링되며, 조종자는 기체 아이콘이 이 원 내부로 겹쳐지는 물리적 과정을 눈으로 확인할 수 있다.
  • 상태 전이 애니메이션(State Transition Animation): Qt의 ColorAnimation과 PropertyAnimation을 활용하여 상태 변화 시 기체와 목표 핀의 색상이 즉각적으로 변환(예: 대기 시 주황색 \rightarrow 도달 시 녹색)되어, 성공적 안착 여부를 비언어적 큐(Non-verbal Cue)로 제공한다.

0.3 도달 성공/실패 여부 판별과 비동기 이벤트 리스너(Asynchronous Event Listener) 설계

PX4-Autopilot은 GCS를 향해 기체의 이동 경과를 실시간 MAVLink 텔레메트리로 브로드캐스트한다. GCS 백엔드의 통신 스레드(Network Thread)는 비동기적으로(Asynchronously) 쏟아지는 패킷들을 파싱하게 되며, 특정 이동 명령에 대한 최종 결과를 콜러(Caller) 스레드로 전달하기 위해서는 ‘비동기 이벤트 리스너’ 인터페이스 패턴을 엄격히 적용해야 한다.

도달 여부를 평가하는 이벤트 판별식(Event Evaluator)은 다음과 같은 복합 조건으로 이뤄진다.

  • 도달 성공(Success) 조건: 기체의 현재 위치 추정치 Pos(\mathbf{x}, \mathbf{y})와 타겟 위치 Tgt(\mathbf{x}, \mathbf{y})의 거리가 파라미터 NAV\_ACC\_RAD 미만이고, 시스템에서 MISSION_ITEM_REACHED 또는 포지션 홀드 상태로의 전환 플래그가 수신되었을 때.
  • 도달 실패(Failure) 조건: 최대 타임아웃(Timeout)이 경과하였으나 여전히 이격 거리가 큰 경우, 혹은 이동 도중 비행 모드(Flight Mode)가 Goto/Offboard에서 수동 제어로 강제 전환(Interrupted)된 경우 이벤트 시스템은 실패 콜백을 반환한다.

0.4 C++ 스레드 풀 및 콜백(Callback)을 활용한 비동기 리스너 구현 예제

현대적인 C++11 이상의 GCS 개발 환경(예: MAVSDK 활용 백엔드)에서는 std::promise와 std::future, 혹은 std::function을 통한 람다 표현식(Lambda Expression) 콜백 구조를 채택하여 스레드 안전성(Thread Safety)을 도모한다.

다음은 기체의 좌표를 비동기적으로 감시하며 상태 전이 및 도달 성공/실패 이벤트를 발송(Emit)하는 백엔드 리스너의 기술적 예제이다.

#include <iostream>
#include <functional>
#include <cmath>
#include <mutex>
#include <atomic>

// 이산적 전이 상태 정의 (Enum)
enum class GotoState {
    STATE_IDLE,
    STATE_IN_TRANSIT,
    STATE_APPROACHING,
    STATE_REACHED,
    STATE_ABORTED
};

class GotoEventManager {
private:
    std::mutex mtx_;
    std::atomic<GotoState> current_state_{GotoState::STATE_IDLE};
    
    double target_lat_ = 0.0;
    double target_lon_ = 0.0;
    const double ACCEPT_RADIUS_M = 1.5; // 수용 반경 1.5m
    
    // UI 업데이트용 콜백 함수 (QML로 시그널 전송)
    std::function<void(GotoState)> ui_render_callback_;

public:
    // 리스너 콜백 등록 등록 (Subscribe to UI Render System)
    void setRenderCallback(std::function<void(GotoState)> callback) {
        ui_render_callback_ = callback;
    }

    // Goto 명령 발행 시 호출됨
    void initiateGoto(double lat, double lon) {
        std::lock_guard<std::mutex> lock(mtx_);
        target_lat_ = lat;
        target_lon_ = lon;
        changeState(GotoState::STATE_IN_TRANSIT);
        std::cout << "[Command] Goto 명령 전송 완료. 비동기 감시 시작." << std::endl;
    }

    // 텔레메트리 수신 스레드에서 주기적으로 호출되는 비동기 리스너 로직
    void onPositionTelemetryReceived(double current_lat, double current_lon, bool is_mode_changed) {
        std::lock_guard<std::mutex> lock(mtx_);
        
        if (current_state_ == GotoState::STATE_REACHED || current_state_ == GotoState::STATE_ABORTED) {
            return; // 분기 종료 (State Machine Exit)
        }
        
        // 예외 상황 리스닝 (사용자가 돌연 비행 모드를 변경한 경우)
        if (is_mode_changed) {
            std::cout << "[Event] 타의에 의한 모드 변경 감지. Goto 취소." << std::endl;
            changeState(GotoState::STATE_ABORTED);
            return;
        }

        // 헤이버사인(Haversine) 공식을 통한 거리 근사 계산 (단순화)
        double distance_m = calculateDistance(target_lat_, target_lon_, current_lat, current_lon);

        if (distance_m <= ACCEPT_RADIUS_M) {
            std::cout << "[Event] 수용 반경(Acceptance Radius) 진입 및 위치 유지 안정화." << std::endl;
            changeState(GotoState::STATE_REACHED);
        } 
        // 도달 전 감속 구간 시각적 전이 상태 처리
        else if (distance_m <= ACCEPT_RADIUS_M * 3.0 && current_state_ != GotoState::STATE_APPROACHING) {
            changeState(GotoState::STATE_APPROACHING);
        }
    }

private:
    void changeState(GotoState new_state) {
        if (current_state_ != new_state) {
            current_state_ = new_state;
            if (ui_render_callback_) {
                ui_render_callback_(current_state_); // UI 렌더링 콜백 비동기 격발
            }
        }
    }

    double calculateDistance(double lat1, double lon1, double lat2, double lon2) {
        // 실제 수식은 Haversine 공식을 사용해야 하나, 예제 간소화를 위해 가상의 값 반환 구조 모델링
        double lat_to_m = (lat1 - lat2) * 111320.0;
        double lon_to_m = (lon1 - lon2) * 111320.0 * cos(lat1 * M_PI / 180.0);
        return sqrt(lat_to_m * lat_to_m + lon_to_m * lon_to_m);
    }
};

0.5 아키텍처 의의

위와 같은 이벤트 기반 모델은 하드웨어와 무선 Datalink의 불안정성에 종속되지 않게 GCS의 메인 렌더링 스레드를 보호(Isolate)하는 훌륭한 패턴이다.
조종자는 기계적인 위도/경도 숫자표기가 아닌 상태 머신 기반의 반응형 색상 애니메이션 및 시각적 피드백을 통해 찰나의 전이 시간 동안에도 드론의 움직임을 확신할 수 있으며, 시스템 안정성 향상과 사용자 경험 고도화라는 두 마리 토끼를 잡는 효과를 가져온다. PX4-Autopilot의 커맨더(Commander) 로직이 내부적으로 수행하는 비행기반 상태 머신 전이를 GCS 프론트엔드에도 거울처럼 비추어 동기화하는 것이 이 설계의 핵심이라 볼 수 있다.
