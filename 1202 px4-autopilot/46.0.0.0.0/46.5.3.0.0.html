46.5.3 시스템 장애 전파(Fault Propagation) 억제 및 원격 조종자 개입(Human-in-the-Loop Intervention) 대응 메커니즘

무인항공기(UAV)의 자율 비행(Autonomous Flight) 중 발생하는 하드웨어 또는 소프트웨어 결함은 단일 모듈의 에러로 끝나지 않고 시스템 전체의 치명적 오작동으로 확산되기 쉽다. 이러한 연쇄 반응을 **장애 전파(Fault Propagation)**라 칭한다. 특히, 단일 목적지 이동(Goto/Reposition)과 같이 공간적 이동을 수반하는 임무 중 상태 추정기(Estimator) 무효화, 액추에이터 포화(Actuator Saturation) 또는 통신 지연이 발생할 경우, 비행 제어기(Flight Controller)는 신속히 에러의 확산을 억제(Containment)하고 안전 상태(Safe State)로 진입해야 한다.

본 절에서는 PX4-Autopilot(펌웨어 v1.14.0 기준)이 취하고 있는 예외 상황 대응 아키텍처와 장애 전파 억제 메커니즘을 분석하고, 지상 관제 시스템(QGroundControl v4.3.0)을 통한 원격 조종자 개입(Human-in-the-Loop Intervention) 프로토콜을 다각도에서 조명하라.

1. PX4-Autopilot의 장애 전파 억제(Fault Containment) 아키텍처

PX4는 마이크로 서비스 아키텍처(Microservices Architecture) 패턴과 유사하게 uORB(Micro Object Request Broker) 메시징 버스를 통해 모듈 간 의존성을 분리(Decoupling)하였다. 이 구조는 특정 센서 드라이버나 제어 노드가 크래시(Crash)되더라도 다른 독립적 모듈로 장애가 무분별하게 전파되는 현상을 억제한다.

1.1 계층적 Failsafe 상태 머신 (Hierarchical Failsafe State Machine)

단일 목적지 이동 제어(Position Control) 루프 내에서 장애가 발생할 경우, Commander 모듈은 에러의 심각도(Severity)에 따라 계층적으로 대응한다.

  1. 로컬 복구(Local Recovery) 단계: 예를 들어, GPS 신호가 일시적으로 튀는 경우(GPS Glitch), EKF2(Extended Kalman Filter 2) 상태 추정기는 곧바로 비행 모드를 Failsafe로 전환하지 않고, 광학 흐름 센서(Optical Flow)나 VIO(Visual Inertial Odometry) 데이터로 위치 추정 소스를 동적 전환(Fallback)하여 Goto 명령을 지속하려 시도한다. 이를 EKF_선택적_융합(Selective Fusion)이라 한다.
  2. 글로벌 비상(Global Emergency) 단계: 센서 퓨전으로도 현재 위치 속도 추정이 불가능해지면, Commander는 위치 제어 루프를 강제로 차단하고 고도 제어 모드(Altitude Mode)나 수동 안정화 제어 모드(Stabilized Mode)로 하향 전환(Downgrade)하여 제어 연산폭증이나 난기류 비행(Erratic Flight)을 방지한다.

1.2 예외 상황 대응을 위한 Mermaid 상태 다이어그램

다음 다이어그램은 Goto 비행 중 센서/위치 제어 에러에 따른 계층적 장애 억제 및 사용자 개입 절차를 보여준다.

    [*] --> Auto_Goto : Goto 비행 중
    
    state Auto_Goto {
        Normal_Flight : 정상 궤적 추종
        EKF_Glitch : 위치 추정 오차율 폭증
        
        Normal_Flight --> EKF_Glitch : 위상/센서 노이즈 발생
        EKF_Glitch --> Normal_Flight : EKF 내부 Fallback 성공
    }
    
    Auto_Goto --> Position_Control_Lost : Fallback 실패 (장애 전파 위협 감지)
    
    state Position_Control_Lost {
        direction TB
        Mode_Downgrade : 비행 모드 강등 (Auto -> Altitude)
        Auto_Loiter : 제자리 비행 (비상 Hold)
        
        Mode_Downgrade --> Auto_Loiter
    }
    
    Position_Control_Lost --> Human_Intervention : 원격 조종자 개입 감지 (RC 스틱 등)
    Position_Control_Lost --> RTL_or_Land : 타임아웃 지속 시 자동 귀환/착륙

        Manual_Override : 수동 조종 오버라이드 (Manual Override)

    }

2. 원격 조종자 개입 (Human-in-the-Loop Intervention) 기법

최고 수준의 안전성을 담보하기 위해 자율 비행 중에도 인적 판단(Human Judgment)이 최우선 권한을 가져야 한다. PX4는 다양한 레이어에서 즉각적인 원격 개입을 허용한다.

2.1 스틱 오버라이드 (Stick Override)를 통한 권한 탈취

비행 관리자(Flight Manager)는 COM_RC_OVERRIDE 파라미터 활성화 시, 자율 비행(Auto Mode) 도중 조종기(Radio Control)의 스틱 움직임이 설정된 데드밴드(Deadband)를 초과하는 즉시 비행 모드를 수동 기반의 위치 제어(Position 모드)로 전환한다. 이는 장애로 인해 기체가 예기치 않은 방향으로 급발진(Flyaway)하는 현상을 물리적이고 직관적인 동작으로 즉각 억제(Veto)할 수 있게 한다.

2.2 GCS(QGroundControl) 기반 소프트웨어 오버라이드

QGroundControl에서의 오버라이드는 주로 MAVLink 명령 계층에서 이루어진다.
QGroundControl 측에서는 기체의 상태(LOCAL_POSITION_NED)가 기존 설계 범위를 넘어서는 에러를 보고할 경우, 다음과 같은 MAVLink 기반 명령을 통해 개입한다.

  • MAV_CMD_DO_PAUSE_CONTINUE (ID: 193): теку Goto 이동 명령을 강제로 멈추고 현 위치에서 브레이킹(Braking) 및 호버링(Hovering)을 지시한다.
  • MAV_CMD_NAV_LOITER_UNLIM (ID: 17): 위치 불확실성이 크거나 추가적인 판단 시간이 필요할 때 사용하는 무제한 체공 명령이다.

3. Ardupilot의 대응 기법과의 제어 철학 비교

장애 전파 억제 및 오버라이드 기능에서 두 펌웨어는 뚜렷한 설계 철학의 차이를 공유한다.

비교 기준PX4-AutopilotArdupilot (Copter)
비행 모드 하향 전환(Mode Downgrade)센서 장애 시 Auto \rightarrow Altitude \rightarrow Stabilized 순으로 비행 제어 모드를 체계적으로 그리고 자동으로 강등시켜 시스템 루프 붕괴를 막는 유연한 연성 실패(Soft-fail)를 추구한다.GPS 타임아웃 등 에러 감지 시 즉각적이고 확정적인 EKF Failsafe 모드(Land 또는 AltHold)로 진입하여 결정론적(Deterministic) 강경 대응 구조를 선호한다.
원격 개입 민감도 및 조건RC 입력에 의한 자동 비행 중단(COM_RC_OVERRIDE)을 기본적으로 매우 직관적이고 즉각적인 인터럽트로 취급하여 조종자 친화적 구조이다.조종자 개입 시 현재 미션 스위치를 명시적으로 변경하거나(Mode Switch Trigger), 설정된 브레이크(Brake) 스위치를 별도로 지정해야 반응하는 등, 의도치 않은 조작 실수를 막으려는 논리가 강하다.

PX4 방식은 ROS2 기반의 로보틱스 자율 제어나 연구 목적에서 예외 상황 시 조종자의 개입 피드백을 동적으로 수용하기 용이하게 구성되어 있다.

4. 자율 에이전트를 위한 ROS2 (uXRCE-DDS) 기반 개입 로직

ROS2 Offboard 모드로 단일 목적지 이동을 수행하는 고도화된 시스템에서는, 사람의 조작이 아닌 “외부 감시 노드(External Monitor Node)“가 Human-in-the-Loop 역할을 대신하는 “자율 개입(Autonomous Intervention)” 메커니즘이 수반된다.

ROS2 환경에서 DDS 퍼블리셔(Publisher) 노드는 VehicleCommand uORB 토픽 내에 타임스탬프와 함께 새로운 VehicleCommand::VEHICLE_CMD_DO_SET_MODE 값을 즉각 발행한다. 에러 확산 방지를 위해 궤적 생성 주기(예: 50Hz) 내부에서, VIO 센싱이나 LiDAR 기반 충돌 위험이 임계값(Threshold)을 넘는 경우 Offboard 노드 자체가 스스로 Goto 궤적 생성을 정지하고 안전 고도 대기(Hold) 상태 명령을 전송하도록 노드 그래프(Node Graph)를 구성해야 한다.

5. 결론

Goto 비행 중 발생하는 시스템 에러의 전파(Fault Propagation)를 막는 것은 치명적 추락(Catastrophic Crash)을 방지하는 핵심 소프트웨어 공학 기술이다. PX4-Autopilot은 uORB 메시지 기반의 모듈화 아키텍처와 계층적 모드 강등(Mode Downgrade) 메커니즘을 통해 에러를 국소화(Localization)한다. 개발자는 이 과정에서 MAVLink 조이스틱 오버라이드 및 QGroundControl의 릴레이 기반 개입 수단을 적극적으로 분석하고, ROS2 환경의 에러 감시 노드를 추가하여 예상치 못한 엣지 케이스(Edge Case) 하에서도 기체가 안전성을 잃지 않도록 강건한 제어 파이프라인(Robust Control Pipeline)을 설계해야 한다.
