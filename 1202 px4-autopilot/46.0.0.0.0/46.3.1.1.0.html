46.3.1.1 무장(Armed) 상태 및 사전 비행 점검(Pre-flight Check) 통과 여부에 따른 모드 전환 블로킹(Blocking) 로직

1. 개요

무인항공기(UAS)의 생존성과 안전을 제어하는 가장 중요한 1차원적 상태값은 바로 기체의 무장(Armed) 상태 여부이다. PX4-Autopilot의 FSM(Finite State Machine) 아키텍처에서 Armed 상태는 모터 액추에이터로 실제 전력 제어 신호(PWM/DShot 등)가 흐르고 기체가 언제든 이륙할 수 있거나 이미 공중에 떠 있는 위험 상태를 의미하며, 반대로 Disarmed 상태는 센서는 동작하되 물리적인 구동계 전원이 차단된 대기 상태를 가리킨다.

커맨더(Commander) 모듈은 이러한 Armed/Disarmed 상태를 런타임 제약 조건(Runtime Constraint)의 최우선 기준으로 삼는다. 특히 비행 모드를 전환(Mode Switch)하거나 임무(Mission)를 시작하기 전, 기체는 반드시 **사전 비행 점검(Pre-flight Check)**을 통과해야 하며, 각 상태에 부합하지 않는 모드 전이 요청은 커맨더 산하의 강건한 블로킹(Blocking) 로직에 의해 철저하게 거부된다.

2. 사전 비행 점검(Pre-flight Check) 파이프라인

QGroundControl 관제석에서 기체를 무장(Arm)시키려 하거나, Mission, Takeoff 등의 핵심 모드로 사전 전환하려 할 때, PX4의 FSM 컨트롤러는 src/modules/commander/PreflightCheck.cpp 모듈에 정의된 일련의 엄격한 테스트 벤치를 즉각 실행한다.

주요 평가(Check) 레이어는 다음과 같다.

  • 센서 건전성 (Sensor Health): 가속도계, 자이로스코프, 나침반(지자기 센서) 및 기압계의 측정 상태와 캘리브레이션 유효성 판별.
  • 포지셔닝 및 EKF 품질: EKF2 프레임워크가 GPS(또는 VIO) 데이터 기반으로 수평 및 수직 위치 공분산(Variance)이 특정 임계치(Threshold, 예: 위치 오차 3m 이내) 이하로 수렴(Convergence)했는지의 판단.
  • 링크 및 전력: RC 컨트롤러 혹은 GCS와의 데이터 링크(Heartbeat) 상태, 그리고 배터리 잔량의 임계치(안전 모드 발동 전압 강하 여부 등).

이러한 수십 개의 플래그(Flag) 중 단 하나라도 false로 산출되면, 시스템은 차단(Blocked) 상태 플래그를 올려 모드 전환 및 Arming 요청에 명시적인 DENIED를 반환한다.

3. 무장(Armed) 상태 기반의 모드 역전 블로킹 로직

PX4의 커맨더 데몬은 비행 모드의 유효성을 오직 현재의 무장 상태와 연동하여 동적으로 평가한다. 이는 Disarmed 시점의 지상 테스트나 Armed 시점의 공중 작전 등 맥락(Context)에 따라 안전한 모드 천이 여부가 달라지기 때문이다.

3.1 Disarmed (대기 상태)에서의 모드 전환

  • 지상에 기체가 정지해 있는 경우(Landed == true 및 Armed == false), 조종사는 Manual, Stabilize, Position 모드를 자유롭게 스위칭하며 센서의 상태나 조종기의 매핑을 사전에 점검할 수 있다.
  • 하지만 위성 포지셔닝 락(GPS Lock)이 없는 실내에서 Position 모드로 변경하려 하면 조향 알고리즘 기저의 사전 비행 점검 레이어에서 반려되어 블로킹당한다.

3.2 Armed (작전 상태)에서의 블로킹 메커니즘

Armed 및 이륙 상태(In-Air)에서는 더욱 복잡하고 엄격한 페일세이프(Failsafe) 룰이 개입한다.

  • 수동(Manual) 모드로의 하향(Downgrade) 억제: 특정 에어프레임이나 안전 스위치 설정에 따라, 자율 비행(Mission) 도중 GPS가 일시적으로 상실되었다고 하여 기체가 무방비한 1차원 Manual 상태로 강제 전환되는 것을 블록한다. 대신 위치 추론 없는 자이로 안정화 모드인 Altitude나 조종 개입 모드로 안전하게 차상위 복원(Fallback)을 시도한다.
  • Auto 모드 진입 제한: 사용자가 공중에서 MAV_CMD_DO_SET_MODE를 이용해 임무(Mission) 모드로 전환하려 할 때, EKF2의 EPH(Estimated Position Horizontal) 수치가 비행 전 점검에서 규정한 허용 반경을 이탈했거나, 웨이포인트(Waypoint) 궤적 데이터가 시스템 RAM에 완벽히 캐싱되어 있지 않으면, 커맨더는 MAV_RESULT_TEMPORARILY_REJECTED 응답과 함께 이를 블로킹한다.

4. 커맨더 내부 로직 아키텍처 다이어그램

이러한 상태 기반 모드 필터링(Mode Filtering) 파이프라인은 아래와 같은 흐름으로 전개된다.

flowchart TD
    CMD_REQ[MAVLink / RC<br/>모드 전환 요청] --> PARSER(명령어 파서)
    PARSER --> ARMING_STATE{현재 무장 상태 판별}
    
    %% Armed Case
    ARMING_STATE -->|Armed & In_Air| IN_AIR_CHECKS[동적 작전 평가 모듈<br/>(Dynamic In-air Rules)]
    IN_AIR_CHECKS --> |수동 복구 / 하향 통제| SAFE_FALLBACK{안전 복원 임계 허용?}
    
    %% Disarmed Case
    ARMING_STATE -->|Disarmed| PFC_MODULE[사전 비행 점검<br/>(Pre-flight Check)]
    PFC_MODULE --> PFC_EVAL{센서/시스템 건전성<br/>ALL PASS?}
    
    %% Output
    SAFE_FALLBACK -->|Yes| MODE_ACCEPT(모드 전환 수락)
    SAFE_FALLBACK -->|No| MODE_REJECT(모드 전환 블로킹<br/>경고 메시지 발송)
    
    PFC_EVAL -->|Yes| MODE_ACCEPT
    PFC_EVAL -->|No| MODE_REJECT

5. Ardupilot Arming Check 아키텍처와의 비교 포인트

PX4-Autopilot과 Ardupilot은 둘 다 오픈소스 최고 수준의 안전 표준을 제공하지만, 내부 소프트웨어 엔지니어링 뎁스(Depth)에서 약간의 차별점을 가진다.

  • Ardupilot의 비트마스크(Bitmask) 파라미터화: Ardupilot 시스템 코어(AP_Arming.cpp)는 ARMING_CHECK 파라미터를 통해 관리자가 검사할 하드웨어 스택(예: Baro, GPS, Compass 등)의 항목을 정수형 비트 연산 기반으로 임의 설정하고 무력화할 권한을 직관적으로 제공한다.
  • PX4의 FSM 회로 우선주의: 반면 PX4는 CBRK_* (Circuit Breaker) 파라미터로 개별 검사를 오버라이드(Override)할 수 있긴 하나, 커맨더 메인 루프 자체가 vehicle_status 구조체의 생명 주기에 철저히 귀속되어 작동한다. 즉, 단순히 체크를 통과하는 것을 넘어, “현재 FSM 다이어그램 선(Edge) 위에서 해당 상태 전이가 수학적으로 정의된 정합성(Validity)을 지니고 있는가“에 대하여 FSM 모델(src/modules/commander/state_machine_helper.cpp)에서 더욱 타이트하게 로직을 통제한다.

6. 결론

PX4의 커맨더 모듈 내 사전 비행 점검(Pre-flight Check)과 무장(Armed) 상태에 연동된 모드 단위 블로킹 로직은, 단순히 MAVLink의 에러를 던져주는 1차원적인 인터페이스가 아니다. 이는 비행기가 공중에 뜰 자격이 있는지 측정하는 하드웨어 인증 테스트베드이며, 동시에 고공에서 예측 불허의 외부 조작이나 노이즈에 의해 치명적인 모드로 전환되는 것을 막는 소프트웨어 방파제 역할을 겸임한다. 자율 관제 환경을 구성할 때, 이러한 블로킹 조건을 GCS의 UI로 적절히 라우팅하여 명시적인 시각 알람으로 피드백해주는 과정은 견고한 무인항공 체계 구축의 핵심이라 볼 수 있다.
