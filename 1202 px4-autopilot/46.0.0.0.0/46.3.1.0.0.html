46.3.1 커맨더(Commander) 모듈의 명령어 디코딩(Decoding) 및 비행 모드 전이(Flight Mode Transition) 권한 검증 상태도(State Diagram) 분석

1. 개요

지상 관제 시스템(GCS; QGroundControl)이나 오프보드(Offboard) 애플리케이션으로부터 MAVLink 프로토콜을 이용해 비행 기체에 전달되는 모든 제어 명령(Command)과 모드 전환(Mode Switch) 요청은, 비행 시스템 내부에서 전적으로 커맨더(Commander) 모듈에 의해 인터셉트(Intercept)되고 디코딩(Decoding)된다. PX4-Autopilot의 커맨더 모듈(src/modules/commander)은 비행 제어기 운영체제 내에서 최고 수준의 상태 관제 기관이며, 수용된 명령의 위험성, 선행 조건(Pre-condition) 충족 여부, 런타임 하드웨어 센서 헬스 스탯(Health Status)에 근거하여 유한 상태 기계(Finite State Machine, FSM)를 구동하는 핵심적인 아키텍처이다.

본 문서에서는 Goto 미션, Return 모드 및 사용자 임의 모드 변경 등을 포괄하는 비행 모드 전이(Flight Mode Transition) 시, 커맨더(Commander) 모듈 내부에서 수행되는 트랜잭션 수락 시스템의 구조와 통과 여부에 따른 FSM 상세 상태도를 소프트웨어 공학적 관점에서 분석한다.

2. 모드 전이(Mode Transition)의 권한 검증 파이프라인

QGroundControl 또는 ROS2 MAVROS/MAVSDK 노드에서 특정 모드로 전환하라는 명령(MAV_CMD_DO_SET_MODE)이 uORB 토픽 vehicle_command로 수신되면, 커맨더 모듈의 메인 쓰레드는 이 토픽을 구독(Subscribe)하고 디코딩 로직을 가동시킨다. 이때 PX4는 무분별한 상태 변경으로 인한 치명적인 추락을 막기 위해 3계층 방어 체계를 거쳐 권한을 검증한다.

  1. 내비게이션 모드 검증 (Navigation Mode Validity): 요청된 비행 모드(예: Position, Mission, RTL)가 활성화 기체의 에어프레임(Airframe, 예: 멀티로터, 고정익, VTOL)에 기술적으로 유효한 모드인지 1차로 검사한다.
  2. 센서 품질 확인 (Estimator Health Check): EKF2(Estimator)가 퍼블리시하는 vehicle_local_position 토픽 타임스탬프와 분산 공분산(Variance) 행렬을 추적한다. GPS의 EPH/EPV(Position Dilution of Precision)가 높거나 시각 측위(VIO) 연결이 단절된 상태에서는 위치 제어가 강제되는 자동(Auto) 모드(예: Mission, Goto) 진입을 엄격히 블로킹(Blocking)한다.
  3. 상태 천이 조건 검사 (FSM Rule Verification): 현재 기체의 FSM 상태 트리(vehicle_status)를 대조하여 허가되지 않는 천이 다이어그램을 배제한다. (예: 공중 비행 상태(In-air)에서 갑작스러운 Manual 모드로의 전환은 안전 스위치 설정에 따라 통제됨.)

이 과정 중 하나라도 반려되면 커맨더는 오류 원인을 인코딩한 COMMAND_ACK(MAV_RESULT_DENIED) 메시지를 즉시 스트리밍하여 관제 화면에 알림을 띄운다.

3. 비행 모드 전이 유한 상태 기계(FSM) 상태도(State Diagram)

다음은 커맨더 패키지의 state_machine_helper.cpp 모듈에 설계된 주요 상태 판별 및 권한 양도 구조도이다.

stateDiagram-v2
    [*] --> Standby
    
    state Standby {
        [*] --> Disarmed
        Disarmed --> Pre_Flight_Check : Arming 명령 인가
        Pre_Flight_Check --> Armed : 검증 통과 (Validity OK)
        Pre_Flight_Check --> Disarmed : 센서 이상, 권한 반려
    }

    state Armed {
        [*] --> Current_Mode (e.g. Position, Altitude)
        
        state "명령어 디코딩/검증 (Mode Switch)" as ModeValidation {
            state "MAV_CMD_DO_SET_MODE 수신" as CMD_RCV
            state "비행 데이터 헬스 점검 (Health Check)" as HEALTH
            state "FSM 룰 매핑 (FSM Rules)" as FSM_MAP
            
            CMD_RCV --> HEALTH
            HEALTH --> FSM_MAP
        }
        
        Current_Mode --> ModeValidation : 체제 변환 이벤트 (uORB trigger)
        
        FSM_MAP --> Target_Mode : 최종 검증 승인 (MAV_RESULT_ACCEPTED)
        FSM_MAP --> Current_Mode : 검증 반려 (MAV_RESULT_DENIED)
        
        state "Target Mode" as Target_Mode {
            Action : 내비게이터(Navigator) 인스턴스에 제어권 인계
        }
    }
    
    Target_Mode --> Current_Mode : 모드 종료 혹은 자동 복귀 조건 트리거
    Armed --> Disarmed : 랜딩(Landed) 감지 후 Disarm 시간 초과

4. Ardupilot Mode 클래스 시스템과의 아키텍처 비교

  • 결합도(Coupling) 및 단일 책임 원칙:
  • PX4의 모드 권한 처리는 Commander 모듈에 완벽히 집중(Centralized)되어 있어 비행 스택(Flight Stack)의 하위 동작 모듈(Navigator 나 각 단위 Controller)은 FSM 상태 전환의 유효성 검사에 전혀 개입하지 않는다.
  • 반면 Ardupilot은 서브 클래스화된 각 Mode 객체 단위(예: ModeAuto, ModeGuided) 내에 상태 변경 허용 함수 모델(예: init() 메서드 반환값)을 개별 할당하여 로직을 띄운다. 따라서 모드 변경 시 해당 클래스 인스턴스가 각각 고유한 하드웨어 검증 루틴을 분산해서 호출하는 분산-권한 아키텍처(Decentralized Architecture)를 갖는다. PX4의 중앙 집중식 설계가 코드 추적성에 유리하지만, 모드 수가 급증할수록 커맨더 데몬 코드의 복잡성을 가중시키는 원인이 되기도 한다.

5. 결론

PX4-Autopilot의 커맨더(Commander) 모듈은 단순한 MAVLink 문자열 파서(Parser)를 넘어서, 비행 상태 머신(FSM)의 신뢰도를 유지하는 중앙 관제탑 역할을 수행한다. 디코딩 알고리즘 내부의 직교적인 검증 루틴(센서, 상태, 에어프레임 제약 조건 등)을 통과한 모드 전이(Flight Mode Transition) 권한만이 하위 조향 컴포넌트로 전가된다. GCS 서드파티 개발자나 자율 운용 ROS2 애플리케이션 개발자는 강제 런타임 천이를 시도할 때 반드시 이 FSM 권한 검증 상태 구조도를 숙지해야 하며, 거부 응답이 올 경우 맹목적인 재전송(Retry)을 회피하고 먼저 시스템 센서의 vehicle_status를 읽어 반려 사유를 디버깅하는 소프트웨어 설계가 필수적이다.
