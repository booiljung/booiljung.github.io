46.7.3.1. Goto 애플리케이션 프로그래밍 인터페이스(API)의 추상화 계층(Abstraction Layer) 호출 구조 및 비동기 프로그래밍 모델(Async/Await/Promise) 적용 패턴

임베디드 탑재 컴퓨터(Companion Computer) 기반의 자율 비행 시스템을 설계할 때, 개발자가 로우 레벨(Low-level)의 물리적 통신 프로토콜과 MAVLink 페이로드 바이트 구조에 매몰되지 않고 애플리케이션 비즈니스 로직(예: 비전 기반 장애물 회피, 추적 기능)에 집중할 수 있도록 돕는 것이 MAVSDK 프레임워크의 본질적 목표다.

본 절에서는 MAVSDK의 단일 목적지 이동(Goto) API가 하위 MAVLink 시스템을 어떻게 추상화(Abstraction)하는지 그 계층 도메인을 분석한다. 더불어, 현대 소프트웨어 공학의 필수 패러다임인 비동기 프로그래밍(Asynchronous Programming) 모델(C++의 Promise/Future 패턴과 Python의 Async/Await 패턴)이 단일 목적지 제어 흐름에 어떻게 적용되어 스레드 블로킹(Thread Blocking)을 파훼하는지 심도 있게 논의한다.

0.1 Goto API 추상화 계층(Abstraction Layer) 호출 아키텍처

MAVLink 프로토콜을 원시적으로 다룰 때 기체를 특정 좌표로 이동시키려면, 복잡한 비트마스크(Bitmask)를 설정하여 SET_POSITION_TARGET_GLOBAL_INT 메시지를 주기적으로 퍼블리시(Publish)하거나, 시스템 ACK 응답 처리가 요구되는 COMMAND_LONG (Command: MAV_CMD_DO_REPOSITION) 형식을 엄격한 규칙 아래 전송해야 한다.

MAVSDK의 Action 플러그인은 이러한 과정을 직관적인 객체 지향 함수 호출로 캡슐화(Encapsulation)하는 API 추상화 계층을 제공한다.

개발자가 고수준 API인 goto_location(lat, lon, alt, yaw)를 호출할 때, 내부 프레임워크 하단에서는 다음과 같은 콜 스택(Call Stack) 변환 작업이 연속적으로 일어난다.

  1. API 호출 계층 (Application Layer): 언어 바인딩(Python, C++ 등) 인터페이스 검증 (자료형 안정성 확인).
  2. gRPC 중간 계층 (RPC/Protobuf Layer): 프로토콜 버퍼(Protocol Buffers)를 기반으로 고수준 파라미터가 직렬화(Serialization)되어 C++ Core 서버에 위임된다. (동일 C++ 아키텍처 환경에서는 이 계층이 생략된다.)
  3. 코어 트랜슬레이션 계층 (Core Translation Layer): MAVSDK Core 엔진이 Action::goto_location 요청을 인가받고, 이 요청을 적법한 MAVLink MAV_CMD_DO_REPOSITION 커맨드 패킷으로 조립한다.
  4. 네트워크-링크 통신 계층 (Transport Layer): 직렬 포트(Serial Port)나 UDP/TCP 소켓을 통해 최종 덤프된 바이너리 MAVLink 패킷이 기체의 비행 제어기(Flight Controller) 로 전달된다.

0.2 동기적(Synchronous) 제어의 한계와 비동기(Asynchronous) 철학의 필요성

동기적 프로그래밍 패턴에서는 API를 호출한 즉시 애플리케이션의 메인 스레드(Main Thread)가 반환 값을 기다리며 정지(Blocking)된다. 드론의 Goto 비행 제어는 물리적 한계로 인해 짧게는 수 초에서 길게는 수십 분이 소요될 수 있는 매우 긴 트랜잭션의 작업이다.
만약 메인 제어 루프가 goto_location의 도달 완료 또는 승인 응답을 대기하느라 블로킹될 경우, 시스템은 실시간으로 수신되는 카메라 비전 영상 프레임 분석이나 VIO 오도메트리 파싱, 그리고 Failsafe 인터럽트 등의 치명적인 작업을 동시다발적으로 처리할 수 없게 되어 치명적인 추락 사고(Crash)를 유발한다.

이러한 문제의 해법이 바로 **비동기 프로그래밍 모델(Async/Await/Promise)**이다. 콜러(Caller) 스레드는 Goto 명령을 비행 제어기에 밀어 넣고(Push) 곧바로 시스템 제어권을 반환받아 다른 이벤트 루프를 순회하며, 명령의 결과(성공/실패)는 백그라운드 스레드의 약속(Promise/Future) 객체나 이벤트 콜백(Callback)을 통해 차후에 수신받는 구조를 취해야 한다.

0.4 핵심 언어별 비동기 모델 구현 아키텍처

PX4 기반 관제 애플리케이션 프로그래밍에서 C++과 Python 양측 진영은 각기 다른 철학의 비동기 문법 패러다임을 지원한다.

0.4.1 C++ 환경의 Promise/Future 패턴 설계

MAVSDK-C++ 컴포넌트는 goto_location_async()와 같은 이름 규칙으로 비동기 호출을 지원한다. 콜백 함수(Callback Function) 헬(Hell) 현상을 예방하고 객체 지향적 흐름 제어를 위해, C++ 표준 쓰레드 라이브러리인 std::promise와 std::future를 응용하여 동기적 코드처럼 보이지만 강력한 비동기 지연 평가 체계를 완성할 수 있다.

#include <iostream>
#include <future>
#include <mavsdk/mavsdk.h>
#include <mavsdk/plugins/action/action.h>

using namespace mavsdk;

void executeAsyncGoto(System& system, double lat_deg, double lon_deg, float alt_m, float yaw_deg) {
    auto action = Action{system};
    
    // 비동기 통지(Notification)를 전달받을 Promise 선언
    std::promise<Action::Result> prom;
    // Promise와 쌍으로 연결되어 결과값을 언박싱할 Future 선언
    std::future<Action::Result> fut = prom.get_future();
    
    std::cout << "[Command] 비동기 Goto 명령을 요청합니다..." << std::endl;

    // 비동기 논블로킹 함수 호출 (콜백으로 람다 함수 전달)
    action.goto_location_async(
        lat_deg, lon_deg, alt_m, yaw_deg,
        [&prom](Action::Result%20result) {
            // 명령 결과가 도착하면 스레드 안전하게 promise를 결정(Resolve)함
            prom.set_value(result);
        });
        
    // 메인 스레드는 블로킹 되지 않고 즉시 아래 코드를 마저 수행함
    std::cout << "[Info] 명령 전달 확인 전, 다른 VIO 센서 폴링 작업 수행 가능..." << std::endl;

    // 만약 결과 판단이 필요할 때만 future.get() 호출 (여기서 Wait 발생)
    Action::Result final_result = fut.get();
    if (final_result == Action::Result::Success) {
        std::cout << "[Success] 기체 측에서 Goto 명령 승인 (ACK 수신 완료)." << std::endl;
    } else {
        std::cerr << "[Error] 제어 거부. 코드: " << final_result << std::endl;
    }
}

0.4.2 Python의 Async/Await (Event Loop) 기반 패턴 설계

Python 진영(MAVSDK-Python)은 Python 3.7+ 표준인 asyncio 라이브러리를 전면 채용하여 코루틴(Coroutine) 아키텍처를 강제한다. 즉 Action.goto_location 내부가 이미 비동기 트랜잭션으로 구성되어 있어, await 키워드를 삽입하는 것만으로 이벤트 루프 스케줄러(Scheduler)가 유휴 스레드 자원을 타 프로세스(카메라 수신 루프 등)로 쉽게 양보(Yield)할 수 있다.

import asyncio
from mavsdk import System
from mavsdk.action import ActionError

async def monitor_battery_async(drone: System):
    """지속적으로 병렬 수행되어야 할 비동기 배터리 모니터링 태스크"""
    async for battery in drone.telemetry.battery():
        print(f"[Telemetry] 배터리 수치: {battery.remaining_percent * 100:.1f}% 제한 없이 실행 중.")
        await asyncio.sleep(2.0)

async def auto_navigate_async(drone: System):
    """도달 전이 모델 제어 및 목적지 할당 비동기 태스크"""
    try:
        print("[Event] 1번 목적지 이행 하달.")
        # 코루틴을 중지시키며 백그라운드로 PX4 응답을 대기 (Block이 아님)
        await drone.action.goto_location(37.1234, 127.1234, 50.0, 0.0)
        print("[Event] 기체가 목적지 이동을 진행합니다.")

        # 이 와중에도 asyncio.sleep 등 기타 이벤트 감지는 정상적으로 스핀(Spin) 한다
        await asyncio.sleep(5) 
        
    except ActionError as e:
        print(f"[ERROR] 이동 명령 실패: {e}")

async def main_application():
    drone = System()
    await drone.connect(system_address="udp://:14540")
    
    # 두 개의 비동기 태스크를 동시 실행(Concurrent Execution)
    await asyncio.gather(
        monitor_battery_async(drone),
        auto_navigate_async(drone)
    )

if __name__ == "__main__":
    # 단일 스레드 기반의 논블로킹 이벤트 루프 구동
    asyncio.run(main_application())

0.5 아키텍처 결론

임베디드 로보틱스 관제에 있어 하드웨어 커뮤니케이션을 캡슐화한 API 추상화 계층은 시스템의 유지 보수성과 타 기종 무인기(예: Ardupilot MAVLink 호환성) 포팅을 압도적으로 유리하게 이끈다. 특히, 단일 스레드 기반 혹은 풀링 구조 안에서 Promise/Future나 Async/Await와 같은 논블로킹 철학을 근본적으로 접목한 소프트웨어는 기체 멈춤, UI 동결, 제어권 붕괴를 예방하는 가장 완전한 아키텍처적 무기가 된다.
