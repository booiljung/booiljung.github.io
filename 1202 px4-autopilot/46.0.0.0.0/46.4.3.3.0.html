46.4.3.3 기수각 정렬 완료 전 병진 운동(Translational Motion) 제어 명령의 위상 지연(Phase Delay) 삽입 및 다자유도(Multi-DOF) 운동학적 동기화(Kinematic Synchronization)

무인항공기(UAV)의 자동 비행 제어 시스템에서 목적지(Waypoint)로 이동할 때, 기체의 기수각(Heading/Yaw)이 목표 방향으로 완전히 정렬되기 전에 병진 운동(Translational Motion)이 시작되면 비효율적인 비행 궤적이 발생하거나 센서(카메라, 라이다 등)가 목표 지점을 적절히 지향하지 못하는 문제가 발생할 수 있다. PX4-Autopilot은 이러한 현상을 방지하기 위해 기수각 정렬 상태에 따라 병진 운동 제어 명령에 위상 지연(Phase Delay)을 인위적으로 삽입하고, 다자유도(Multi-DOF) 운동학적 동기화(Kinematic Synchronization)를 수행하는 정교한 알고리즘을 내장하고 있다. 본 절에서는 해당 제어 로직의 공학적 원리와 PX4 내부 아키텍처, 그리고 통신 프로토콜 관점에서의 동기화 메커니즘을 심층적으로 분석한다. (기준: PX4-Autopilot v1.14.0, QGroundControl v4.3.0)

1. 운동학적 동기화(Kinematic Synchronization)의 필요성

멀티로터 기체는 6자유도의 움직임을 가지지만, 주요 구동력은 로터의 추력 방향(일반적으로 제트축)에 의존한다. 따라서 수평 방향의 병진 운동을 위해서는 기체가 롤(Roll) 또는 피치(Pitch) 방향으로 기울어져야 한다. 이때 요(Yaw) 방향의 회전 운동과 수평 방향의 병진 운동이 동시에 명령될 경우, 기체의 비행 역학적 교차 결합(Cross-coupling) 현상으로 인해 제어 성능이 저하될 수 있다.

기수각 오차(Heading Error)가 큰 상태에서 즉각적인 병진 운동 속도를 인가하면 다음과 같은 단점이 존재한다:

  • 비행 궤적의 비효율성: 기체가 옆으로 미끄러지듯(Crab Flight) 비행하게 되어 공기역학적 항력(Drag)이 증가한다.
  • 페이로드(Payload) 지향 오류: GCS(QGroundControl)에서 설정한 임무가 특정 타겟 촬영을 목적으로 하는 경우, 카메라가 목표를 향하기 전에 기체가 이동해버려 원하는 영상을 획득하지 못한다.
  • 센서 퓨전 오차 증가: 측면 비행 시 피토관(Pitot-tube) 등 특정 방향 특성을 가지는 센서들의 신뢰성이 하락할 수 있다.

이를 해결하기 위해 기수각 정렬이 완료될 때까지 전진 속도(Forward Velocity)의 크기를 제한하거나 지연시키는 위상 지연(Phase Delay) 전략이 필수적이다.

2. 병진 운동과 회전 운동의 위상 지연 제어 모델 (Mathematical Formulation)

PX4-Autopilot의 위치 제어기(Position Controller)는 현재 기체의 기수각 \psi_{current}와 목표 기수각 \psi_{target} 간의 오차 \Delta\psi를 바탕으로 병진 운동의 속도 지령 v_{cmd}를 스케일링(Scaling)한다.

기수각 오차는 다음과 같이 정의된다.
\Delta\psi = \vert \psi_{target} - \psi_{current} \vert

병진 속도 조절을 위한 위상 지연 함수 f(\Delta\psi)는 오차가 특정 임계값(Threshold) \psi_{th} 이상일 때 속도를 0으로 제한하거나 점진적으로 감소시키는 비선형 함수로 모델링할 수 있다.

v_{cmd} = v_{target} \times f(\Delta\psi)

일반적인 감쇠 모델은 다음과 같은 형태를 띤다.
f(\Delta\psi) = \begin{cases} 1 & \text{if } \Delta\psi \le \psi_{th\_low} \\ \frac{\psi_{th\_high} - \Delta\psi}{\psi_{th\_high} - \psi_{th\_low}} & \text{if } \psi_{th\_low} < \Delta\psi < \psi_{th\_high} \\ 0 & \text{if } \Delta\psi \ge \psi_{th\_high} \end{cases}
이 모델을 통해 기수각 정렬이 불량할 경우 v_{cmd}가 0으로 수렴하게 되어, 기구학적으로 회전 운동이 완료될 때까지 병진 운동의 시작을 지연(Phase Delay)시킨다.

3. PX4-Autopilot 소프트웨어 아키텍처 및 파라미터 제어

PX4 내부의 FlightTaskAuto 모듈 및 PositionControl 클래스에서 이러한 다자유도 운동학적 동기화를 수행한다. 해당 과정은 사용자가 설정한 파라미터에 의해 동적으로 제어된다.

핵심 제어 파라미터

  • MPC_YAW_MODE: 임무 비행 중 요(Yaw) 동작 방식을 결정한다.
    • 0: 헤딩이 목표 웨이포인트를 향함 (Towards Waypoint)
    • 1: 이륙 시의 헤딩 유지
    • 2: 헤딩이 홈(Home) 위치를 향함
    • 3: 헤딩이 홈 위치 반대 방향을 향함
  • MIS_YAW_ERR: 임무를 수행할 때 병진 운동을 허용하기 위한 최대 허용 기수각 오차(Max Yaw Error)이다. 이 값 내로 \Delta\psi가 진입해야만 Phase Delay가 해제되고 기체가 앞으로 전진한다.

다음은 기수각 오차 검증 및 속도 동기화 로직의 흐름을 나타낸 Mermaid 다이어그램이다.

    A[GCS: 다음 Waypoint 이동 명령 수신] --> B(목표 위치 및 목표 기수각 계산)
    B --> C{MPC_YAW_MODE 분석}
    C -- Towards Waypoint --> D[현재 기수각과 목표 기수각 오차 검출]
    C -- 기타 모드 --> H[기준 속도 그대로 인가]
    D --> E{오차 < MIS_YAW_ERR ?}
    E -- No (Phase Delay 인가) --> F[요 조향 속도(Yaw Rate) 최대화 & 병진 속도 명령 v_cmd = 0]
    F --> D
    E -- Yes (동기화 완료) --> G[병진 속도 명령 v_cmd 가속도 프로파일 적용 후 인가]
    G --> I[TrajectorySetpoint 메시지 발행]
    H --> I
    I --> J(PositionController & AttitudeController 인가)

4. 소스코드 레벨의 운동학적 동기화 검증

PX4의 소스 코드에서 위상 지연과 동기화가 실제로 어떻게 구현되어 있는지 살펴본다. src/modules/flight_mode_manager/tasks/Auto/FlightTaskAuto.cpp 내부의 궤적 생성 로직을 확인하면 다음과 같은 패턴을 발견할 수 있다.

// 목표 방향과의 각도 오차 계산
float yaw_error = wrap_pi(_yaw_sp - _yaw_current);

// 오차가 MIS_YAW_ERR 설정값보다 큰 경우
if (fabsf(yaw_error) > _param_mis_yaw_err.get()) {
    // 병진 속도(Translational Velocity) Setpoint를 0으로 감쇠 (Phase Delay 삽입)
    _velocity_sp(0) = 0.0f;
    _velocity_sp(1) = 0.0f;
    
    // 이 상태에서는 자세 제어기가 Yaw 회전에만 집중하도록 유도
} else {
    // 오차 범위 내에 들어오면 부드럽게 병진 가속재개 (Kinematic Synchronization)
    compute_velocity_setpoint_to_target();
}

이러한 구현을 통해 멀티로터는 목표 웨이포인트를 향해 제자리에서 회전을 선행한 후, 회전이 지정된 임계각 이내로 수렴하면 전진을 시작하는 전형적인 “회전 후 전진(Turn-then-move)” 거동을 보이게 된다.

5. MAVLink 및 ROS2 (uXRCE-DDS) 통신 기반 제어 연동

지상 관제 시스템(QGroundControl)이나 ROS2 기반의 자율 에이전트(Autonomous Agent)가 외부에서 제어 명령을 내릴 때도 운동학적 동기화를 고려해야 한다.

MAVLink 기반 외부 제어 (Offboard Mode)

GCS 또는 외부 컴패니언 컴퓨터가 SET_POSITION_TARGET_LOCAL_NED MAVLink 메시지를 사용하여 제어할 때, 위치(Position), 속도(Velocity), 가속도(Acceleration) 비트마스크(Type Mask)와 별도로 요(Yaw) 각도 및 요 회전율(Yaw Rate) 비트마스크를 지정할 수 있다.
이때 관제 솔루션을 개발하는 경우 컴패니언 컴퓨터가 직접 위상 지연을 구현하여 PX4로 전송하도록 설계할 수도 있다. 즉, MAVLink 명령 생성 단계에서 자체적인 \Delta\psi를 검사한 후, 병진 속도를 0으로 설정한 메시지를 주입하다가 정렬이 완료되면 속도를 점진적으로 증가시키는 방식이다.

ROS2 Offboard 제어 (uXRCE-DDS)

ROS2 환경에서는 TrajectorySetpoint uORB 토픽을 직접 발행(Publishing)하여 6자유도 운동을 설정한다. ROS2 노드(Node)에서 경로 계획(Path Planning) 알고리즘을 짤 때 운동학적 한계를 고려한 궤적 생성기(Trajectory Generator)를 사용하면, 매끄러운 다자유도 동기화 궤적을 PX4로 스트리밍할 수 있다.

// ROS2 노드에서의 위상 지연 제어 패턴 예시 (C++)
px4_msgs::msg::TrajectorySetpoint msg;
msg.yaw = target_yaw;

if (std::abs(current_yaw - target_yaw) > yaw_tolerance) {
    // 위상 지연 삽입: 기수 정렬 중에는 변위 속도를 인가하지 않음
    msg.velocity[0] = 0.0;
    msg.velocity[1] = 0.0;
} else {
    // 운동학적 동기화 완료 후 이동 개시
    msg.velocity[0] = planned_vel_x;
    msg.velocity[1] = planned_vel_y;
}
publisher_->publish(msg);

6. PX4와 Ardupilot 간의 다자유도 운동학적 동기화 제어 철학 비교

이기종 펌웨어 간 제어 방식을 비교하면 아키텍처의 철학적 차이가 명확해진다.

  • PX4-Autopilot: PX4는 기체 역학의 안정성과 부드러운 궤적 제어에 초점을 맞추어 MIS_YAW_ERR 등의 파라미터를 통해 엄격한 상태 전이(State Transition) 방식을 취한다. 즉, 조건이 맞을 때까지 병진 궤적 생성을 보류하는 성향이 강하다. 이를 통해 곡예 비행보다는 정밀한 촬영 및 임무 수행 시 물리적인 오차 발생을 차단한다.
  • Ardupilot (Copter): Ardupilot의 경우 WP_YAW_BEHAVIOR 파라미터에 의존한다. 기본적으로 웨이포인트 이동 시 기수각 정렬을 수행하지만, 비행 선(Spline)을 따라 이동하면서 요(Yaw)를 혼합(Blending)하여 변경하는 방식에 조금 더 초점이 맞춰져 있다. 따라서 고속 임무에서는 곡선 궤적 상에서 미끄러지면서 회전과 전진을 동시에 동역학적으로 수행하는 특성을 보인다.

7. 시스템 운영 시 주의사항 및 디버깅

위상 지연 제어와 다자유도 동기화 기능과 관련하여 실무에서 접할 수 있는 주요 예외 상황(Edge Cases)은 다음과 같다.

  1. 외란으로 인한 제자리 회전 지연: 기수 방향으로 강한 맞바람이 불거나 모터 출력이 불균형하여 실제 요율(Yaw Rate)이 MPC_YAWRAUTO_MAX 설정값에 도달하지 못하면 기수각 오차가 좁혀지지 않는다. 이 경우 기체가 전진하지 않고 제자리에 영구적으로 머무는 데드락(Deadlock)에 빠질 수 있으므로 임무 설계 시 비행 기상의 면밀한 검토가 필요하다.
  2. 센서 캘리브레이션 불량: 자기장 나침반(Magnetometer)의 교정이 불량하여 헤딩 추정치(ECL/EKF2)가 떨리거나 드리프트하면, \Delta\psi의 측정값이 끊임없이 요동쳐 병진 속도 제어기가 급가속과 제동을 반복하는 떨림(Chattering) 현상이 유발될 수 있다. QGroundControl의 ‘Flight Review’ 분석 도구를 사용하여 로깅 데이터 내 vehicle_local_position 메시지의 v_xy_valid 플래그 및 헤딩 상태 오차 분산을 상세히 트러블슈팅하라.
