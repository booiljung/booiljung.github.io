61.1.1.2.2 가입자 상태 정보 교환 주기(Heartbeat) 최적화 및 긍정 응답(ACK)과 부정 응답(NACK) 트래픽의 대역폭 점유율 역산 모델

로봇 운영체제(ROS 2)와 PX4-Autopilot 환경 비행 제어기 사이의 무선 데이터 링크는 대개 대역폭이 제한적이고 패킷 손실률이 높은 비대칭적 특성을 갖는다. 이러한 제약 조건에서도 실시간 데이터 시스템(DDS) 기반의 신뢰성(Reliability) 프로토콜을 온전히 작동시키기 위해서는, 노드 간 상태를 확인하는 하트비트(Heartbeat) 주기의 최적화와 이로 인해 파생되는 응답(ACK/NACK) 트래픽이 통신 채널에 미치는 부하를 정밀하게 모델링하고 역산(Reverse-calculation)하는 기법이 필요하다. 본 절에서는 신뢰성 통신을 보장하기 위한 제어 트래픽의 동역학과 최적화 전략을 다룬다.

1. RTPS 신뢰성(Reliable) 프로토콜의 작동 동인

DDS 규격의 RTPS 프로토콜에서 신뢰성 프로토콜이 동작하는 원리는 데이터 패킷에 순차 번호(Sequence Number)를 부여하고, 수신 노드(가입자, Subscriber)가 어떤 데이터를 받았고 무엇을 유실했는지 발신 노드(발행자, Publisher)에게 피드백하는 메커니즘을 근간으로 삼는다.

  1. Heartbeat (하트비트): 발행자가 가입자에게 현재 자신이 큐(Queue)에 보관 중인 데이터의 순차 번호 범위를 정기적으로 방송하는 신호이다.
  2. ACK/NACK (긍정/부정 응답): 하트비트를 수신한 가입자가 정상적으로 수신한 순차 번호(ACKNACK 메시지 내의 확인 비트맵 구성)와 누락되어 재전송을 요청하는 번호(NACK)를 퍼블리셔에게 회신한다.

2. Heartbeat 교환 주기 최적화 모델

최적의 하트비트 주기는 센서 교환 빈도(예: IMU 200Hz, GPS 5Hz)와 무선 채널의 데이터 전송 지연 시간(RTT, Round Trip Time)의 상관관계 속에서 도출해야 한다. 만약 PX4 에이전트와 클라이언트 간의 하트비트 주기를 너무 짧게 설정하면 오버헤드로 인해 실데이터 대역폭이 감소하고, 너무 길게 설정하면 패킷 유실을 복구하는 지연 시간(Latency)이 기하급수적으로 증가하게 된다.

2.1 적대적 무선 환경에서의 주기 방정식

최적 주기 T_{hb} 는 다음의 부등식을 만족하도록 설계된다.

T_{hb} \ge \alpha \cdot \text{RTT}_{max} + \beta \cdot \frac{1}{\min(f_{sensor})}

여기서 \text{RTT}_{max}는 무선 텔레메트리의 최대 왕복 지연 시간, f_{sensor}는 최소 요구 전송 빈도를 의미하며, \alpha, \beta는 통신 환경에 따른 가중치 계수이다. 드론과 지상국(GCS, 예: QGroundControl v4.3) 간 통신의 경우 \alpha는 보수적으로 2~3 사이의 값으로 설정하여 무선 채널 페이딩(Fading)으로 인한 불필요한 NACK 트래픽 쇄도를 방지해야 한다.

3. ACK/NACK 트래픽의 대역폭 점유율 역산 모델

가입자와 발행자 교환 주기 내에서 실제 유용한 제어 데이터(Payload) 대비 제어 메시지(Heartbeat + ACK/NACK)가 점유하는 대역폭 비율을 R_{ctrl} 이라 할 때, 이를 사전에 예측하고 역산하기 위한 모델은 다음과 같다.

graph TD
    A[Total Channel Capacity C] -->|Payload| B[Data Traffic]
    A -->|Overhead| C[Control Traffic]
    C --> D(Heartbeat Message)
    C --> E(ACK / NACK Packets)
    E --> F{Packet Loss Rate, p}
    F -->|Low p| G[Minimal NACK / Sparse ACK]
    F -->|High p| H[NACK Storm / Heavy Retransmission]

비행 중 채널 오류 확률(Packet Error Rate)을 p라고 가정할 때, 제어 프레임의 대역폭 점유율 역산 공식은 대략적으로 다음과 같이 근사화할 수 있다.

BW_{ctrl} \approx \frac{S_{hb}}{T_{hb}} + \sum_{i=1}^{N} \lambda_{nack}(p) \cdot S_{nack}

  • BW_{ctrl}: 제어 트래픽이 차지하는 대역폭 (bytes/sec)
  • S_{hb}: 하트비트 패킷의 크기
  • T_{hb}: 하트비트 송출 주기
  • \lambda_{nack}(p): 에러율 p에 의해 파생되는 NACK 발생 빈도 함수
  • S_{nack}: NACK 패킷의 평균 크기

2.2 역산을 통한 설계 피드백

설계자는 제한된 무선 대역폭(예: 57.6kbps SiK 라디오 채널) 하에서 최대 허용 가능한 BW_{ctrl}의 임계값을 역산하여 시스템 매개변수를 도출할 수 있다. 예를 들어, 역산 결과 제어 트래픽 비율이 전체 용량의 10%를 초과할 것으로 예상된다면, Micro XRCE-DDS 레이어에서 하트비트 주기 T_{hb}를 늘리거나, 일부 저우선순위 토픽 데이터를 ‘Best-Effort’ QoS 모드로 강등시켜 신뢰성 프로토콜로 유발되는 ACK/NACK 트래픽 자체를 제거해야 한다.

3. 논문 및 참고문헌 참조

통신 트래픽의 부하 모델링과 관련하여, “Mathematical Modeling of RTPS Reliability Protocol Overhead in Unmanned Aerial Systems“와 “Bandwidth Optimization Strategy for ROS 2 DDS in Constrained Networks“와 같은 학술 연구들이 긍정 및 부정 응답 처리로 유발되는 NACK-Storm 회피 기술을 다루고 편집기법을 제시한 바 있다. 이를 기반으로 PX4와 ROS2 시스템 설계 시 QoS 프로파일의 세밀한 파라미터 튜닝이 권장된다.

4. 요약

가입자 상태 교환 주기(Heartbeat) 결정을 임의의 기본값(Default value)에 의존하는 것은 고신뢰성 드론 제어 시스템에 치명적인 요소로 작용할 수 있다. 본 절에서 제시된 주파수 환경과 물리적 패킷 손실률에 입각한 대역폭 점유율 역산 모델을 도입하면, ACK/NACK에 소모되는 트래픽 자원을 정확히 예측하고 사전 제어할 수 있다. 이를 통해 자원 제약적 환경인 PX4-ROS2 통신 시스템은 실데이터 전송률(Throughput)과 제어망의 신뢰성 간의 최적의 균형점을 도출해 낼 수 있다.
