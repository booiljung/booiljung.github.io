61.2.3.1. 다중 코어 프로세서의 동시성(Concurrency) 한계 효용을 극대화하기 위한 작업 큐(Task Queue) 분할 및 방향성 비순환 그래프 기반 병렬 컴파일

1. 개요

오프보드 컨테이너 혹은 동반 컴퓨터(예: Raspberry Pi 4, NVIDIA Jetson 시리즈) 환경에서 ROS2와 PX4-Autopilot 제어 모듈을 통합 컴파일할 때, 단일 코어 기반의 링킹(Linking)은 치명적인 시간적 병목을 유발한다. 반면 프로세서 코어를 무제한으로 사용하려 할 경우, 캐시 미스(Cache Miss) 한계와 주 메모리(RAM) 고갈 현상이 발생하여 오히려 컴파일 속도가 저하되는 동시성(Concurrency)의 한계 효용 체감 법칙이 작용한다. 본 절에서는 메타 빌드 시스템 colcon이 방향성 비순환 그래프(Directed Acyclic Graph, DAG)를 이용하여 빌드 작업을 수학적으로 모델링하고, 이를 최적의 워커 스레드(Worker Thread)가 처리할 수 있도록 작업 큐(Task Queue)로 분할하는 엔지니어링 원리를 다룬다.

2. 방향성 비순환 그래프(DAG) 기반 모델링

수십 개의 ROS2 패키지가 포함된 거대 워크스페이스를 빌드하기 위해, colcon 내장 파서는 각 패키지의 package.xml과 CMakeLists.txt를 스캔하여 패키지 간의 선행-후행 의존성 위상 그래프를 형성한다. 이 그래프는 순환 참조(Cyclic Reference)가 없는 유향 비순환 그래프(DAG)로 표현된다.

만일 노드 V를 개별 패키지로, 간선 E를 의존성 관계로 정의하는 그래프 G(V, E)가 있을 때, 패키지 A가 패키지 B를 의존한다면 선행 관계는 B \rightarrow A가 된다. colcon은 위상 정렬(Topological Sorting)을 통해 내차수(In-degree)가 0인 기초 독립 패키지 리스트를 최우선으로 선별하여 작업 큐의 첫 번째 배치(Batch)로 편입시킨다.

3. 작업 큐 분할의 동시성 및 한계 효용

위상 정렬 상 동일 뎁스(Depth)에 위치한 다수의 패키지들은 이론상 무한한 병렬 처리가 가능하지만, 소프트웨어의 동시성은 가용한 하드웨어인 다중 코어 프로세서의 산술 논리 장치(ALU) 개수 및 메모리 대역폭 한계에 종속된다.

3.1 코어 스로틀링(Throttling) 메커니즘

colcon은 시스템 호출을 통해 사용 가능한 CPU 논리 코어 수(보통 nproc 또는 os.cpu_count())를 동적으로 파악한 후, 그 숫자만큼의 워커 스레드로 구성된 논리적 작업 큐(Task Queue) 풀(Pool)을 생성한다.
이 방식은 코어 간의 빈번한 컨텍스트 스위칭(Context Switching)에 따른 L2/L3 캐시 펜싱(Fencing) 오버헤드를 막고, 프로세스의 스레드 동기화 록(Lock) 경합 빈도를 억제하여 코어별 순수 처리량 스루풋(Throughput)을 최대로 유지하도록 돕는다.

3.2 암달의 법칙(Amdahl’s Law)과 컴파일 한계 효용

현존하는 C++ 컴파일 파이프라인 중 링커(ld) 단계 등 특정 공정은 본질적으로 직렬(Serial) 처리 병목을 갖는다. 암달의 법칙에 따르면, 프로세서를 아무리 늘려도 프로그램의 직렬 처리 비율에 의해 병렬화 한계 효용이 감소한다. 따라서 하나의 거대 패키지를 N개의 스레드로 무리하게 나눠 짓는 것보다는, DAG 상에서 서로 완전히 분리된 여러 개의 중소형 패키지를 패키지별로 분할하여 작업 큐에 동시에 푸시(Push)하는 방식이 훨씬 높은 스루풋을 달성한다. 이 지점이 바로 colcon 메타 병렬 빌드 시스템 설계의 수학적 정당성이다.

4. 동반 컴퓨터를 위한 작업 큐 커스텀 분할 (OOM 회피)

4GB 수준의 제한된 RAM을 지닌 동반 컴퓨터 환경에서 GCS 백엔드나 PX4 브릿지를 병렬 빌드할 때, C++17 이상의 고중량 템플릿 메타 프로그래밍(Template Metaprogramming) 컴파일 시 각 스레드 당 기가바이트 단위의 고립된 메모리를 사용하게 된다. 이 때 작업 큐 관리를 튜닝하지 않으면 OS 스와퍼(Swapper)가 한계에 부딪혀 강제 스레드 사살(OOM-Kill)이 발생한다.

이를 방지하기 위해 colcon 레벨의 패키지 병렬 큐 제한과 cmake 빌더 파이프라인 단의 내부 코어 제한을 혼합 도출하는 공학적 접근이 사용된다.

# PX4 Companion Computer Safe Build Schema
# --parallel-workers 2 : colcon 작업 큐에서 동시에 실행될 패키지 최대 개수를 2개로 강제 분할
# MAKEFLAGS="-j2"      : 각 패키지 내부 컴파일러(make/ninja)가 사용할 스레드도 2개로 제한
MAKEFLAGS="-j2" colcon build --parallel-workers 2 --symlink-install

5. 정리

colcon 시스템은 PX4-ROS2의 거대 트리를 무작정 빌드하는 것이 아니라, 방향성 비순환 그래프(DAG) 구조 탐색을 통해 상호 독립적인 패키지를 찾아내서 묶어내고, 가용 하드웨어 자원의 코어 한계 효용 법칙을 반영하여 작업 큐에 분산 파견하는 지능형 스케줄러(Intelligent Scheduler)이다. 적절한 스레드 분할과 직렬/병렬 교차 투입을 모델링함으로써 운영체제의 메모리 크래시 없이 고속으로 오프보드 제어 환경의 바이너리 아티팩트(Binary Artifacts)를 성공적으로 생산할 수 있다.
