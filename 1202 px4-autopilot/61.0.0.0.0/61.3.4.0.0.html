61.3.4. 에이전트 프로세스 생명주기 관리 및 파일 디스크립터 상태 전이 검증

오프보드(Offboard) 제어 환경에서 PX4-Autopilot과 ROS 2 미들웨어 사이의 데이터 교환을 담당하는 Micro XRCE-DDS 에이전트(Agent)는 끊김 없는 통신과 안정적인 비행 제어를 보장해야 한다. 이를 위해서는 단순한 프로세스 실행을 넘어 에이전트의 전체 생명주기(Lifecycle)를 엄격히 관리하고, 통신 채널인 파일 디스크립터(File Descriptor)의 생합성 및 상태 전이(State Transition)를 지속적으로 검증하는 메커니즘이 필수적이다.

본 절에서는 리눅스(Linux) 호스트 운영체제 기반에서 uXRCE-DDS 에이전트 데몬(Daemon)의 생명주기를 확립하는 방법과, 파일 디스크립터 리소스 누수(Leak)를 방지하기 위한 상태 전이 검증 기법을 고찰한다.

1. Micro XRCE-DDS 에이전트 프로세스 생명주기 관리 아키텍처

로봇 운영 환경에서 에이전트 프로세스는 시스템 부팅 시 자동 시작되어야 하며, 비정상적인 크래시(Crash)나 세그멘테이션 결함(Segmentation Fault) 발생 시 즉시 재시작(Restart)될 수 있는 견고성(Robustness)을 갖추어야 한다. 현대적인 리눅스 환경에서는 systemd와 같은 서비스 관리자를 통해 이러한 생명주기를 시스템 레벨에서 위임 관리한다.

에이전트 프로세스의 상태는 기본적으로 아래와 같은 유한 상태 머신(FSM, Finite State Machine)의 형태를 띤다.

  • 초기화(Initializing): 소켓 바인딩(Socket Binding) 또는 직렬 포트(Serial Port) 파싱 권한 획득 스텝.
  • 실행(Running): DDS와 uORB 간 퍼블리싱/서브스크라이빙 루프 활성화.
  • 오류/좀비 상태(Fault/Zombie): 통신 장치 파일의 연결 단절이나 데드락에 의한 블로킹 상태.
  • 종료(Terminated): 정상 시그널 수신 후 정리 작업 완료.

아래는 데몬 기반 에이전트의 생명주기 제어 흐름을 도식화한 다이어그램이다.

stateDiagram-v2
    [*] --> Initializing : systemctl start
    Initializing --> Running : 리소스 할당 성공
    Initializing --> Failed : 소켓 오류/권한 부족
    
    Running --> Terminated : SIGINT / SIGTERM 수신 (systemctl stop)
    Running --> Fault : 세그멘테이션 결함 / 통신 대기 시간 초과
    
    Fault --> Initializing : systemd Restart=always 정책
    
    Terminated --> [*]
    Failed --> [*]
    
    note right of Running
        이 상태에서 파일 디스크립터(FD)를 
        활용한 송수신(RX/TX)가 실시간으로 일어남.
    end note

이러한 상태 전이를 자동화하기 위하여 다음과 같이 서비스 유닛(Service Unit) 파일을 구성하여 관리할 수 있다. (예시: /etc/systemd/system/uxrce_dds_agent.service)

[Unit]
Description=Micro XRCE-DDS Agent for PX4 ROS2 Bridge
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/MicroXRCEAgent udp4 -p 8888
Restart=always
RestartSec=3
LimitNOFILE=4096

[Install]
WantedBy=multi-user.target

2. 파일 디스크립터(File Descriptor) 상태 전이 및 리소스 누수 검증

PX4-Autopilot은 직렬(Serial) 통신(예: /dev/ttyUSB0) 혹은 UDP 베이스 타임아웃 통신 채널을 사용하여 에이전트와 교신한다. 이 때 리눅스 커널은 통신 포트를 파일 디스크립터(File Descriptor, FD)라는 정수형 식별자로 추상화하여 관리한다.

불안정한 하드웨어 통신 단절(예: 텔레메트리 모듈의 순간적인 전압 강하)이나 소프트웨어의 미숙한 예외 처리는 할당된 FD를 정상적으로 닫지 못하고 파기하는 리소스 누수(Resource Leak)를 유발할 수 있다. 에이전트 프로세스가 열어둔 FD가 커널에서 자립적으로 누적되면, 종국에는 오퍼레이팅 시스템의 리소스 제한(Limit)에 도달하여 “Too many open files” 계열의 치명적인 EMFILE 예외를 뱉으며 에이전트가 다운되게 된다.

2.1 상태 전이 검증 메커니즘

올바른 상태 전이를 위해서는 프로세스 내의 파일 디스크립터 모니터링 체계가 요구된다. 운영 환경에서는 아래와 같은 커널 수준 모니터링 기법들을 활용 참조한다.

  1. lsof 및 procfs 관측: /proc/[PID]/fd/ 경로의 폴링(Polling)을 통해 현재 에이전트 프로세스가 사용 중인 FD 개수와 소켓 상태를 수치화된 메트릭으로 시각화한다. 이를 통해 특정 임계치 이상 활성 FD가 급증할 경우 메모리 누수를 감지할 수 있다.
  2. 이벤트 기반 상태 감지(epoll / select): 에이전트의 메인 루프에서 멀티플렉싱 파일 I/O 시스템 콜(epoll 등)을 사용하여 각 통신 채널의 상태(READ, WRITE, ERROR)를 비동기적으로 검증한다. 만약 단절(HUP)이 확인되면 즉시 해당 FD를 닫고(close(fd)) 연결 상태를 재구축하는 소프트 리셋(Soft Reset)을 수행한다.

실무 개발에서는 에이전트의 생명주기를 외부에서 강제화하기 위한 스크립트 모니터를 병행 구동하기도 한다. 예를 들어 에이전트의 프로세스의 자원 상태를 감지하여 일정 수준 오류가 누적될 시, 강제 종료(kill -9) 대신 우아한 종료(Graceful Shutdown) 신호인 SIGTERM을 전달하여 가용 자원을 커널에 정상 반환하도록 유도하는 오케스트레이션(Orchestration) 방식을 권장한다.

결론적으로 PX4 연동 환경에서 uXRCE-DDS 에이전트의 안정성은 어플리케이션 계층에서의 생명주기 관리 루틴과 호스트 커널 계층 간의 파일 디스크립터 정합성이 완벽히 동기화되었을 때 비로소 달성된다.
