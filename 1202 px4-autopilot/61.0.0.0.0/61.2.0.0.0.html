61.2. 실시간 제어를 위한 호스트 운영체제 커널 튜닝 및 ROS2 환경 구축

1. 개요

현대의 무인항공기(UAV) 제어 시스템은 PX4-Autopilot(펌웨어 버전 v1.14 이상)과 지상 관제 시스템(GCS, 예: QGroundControl v4.3 이상) 간의 안정적인 실시간 통신을 필수적으로 요구한다. 특히 ROS2 미들웨어를 도입하여 복잡한 자율 비행(Autonomous Flight)과 군집 비행(Swarm Flight)을 구현할 경우, 호스트 운영체제(Host Operating System) 수준에서 실시간성(Real-Time Performance)이 보장되어야 제어 루프의 지연(Latency)과 지터(Jitter)를 최소화할 수 있다. 본 절에서는 오프보드(Offboard) 제어 환경 구성을 위해 호스트 리눅스(Linux) 커널을 실시간 환경으로 튜닝하는 방법과 ROS2(Robot Operating System 2) 환경을 최적화하여 구축하는 절차를 다룬다.

2. 호스트 커널 실시간성 보장의 필요성

PX4-Autopilot의 내부 비행 제어 루프는 NuttX RTOS 기반에서 마이크로초(μs) 단위의 엄격한 경성 실시간성(Hard Real-Time)을 보장하며 실행된다. 하지만 외부 컴퓨팅 장치(동반 컴퓨터, Companion Computer)에서 ROS2 기반 알고리즘이 동작할 때, 범용 운영체제(General Purpose Operating System, 예: Ubuntu Linux)의 스케줄러 정책에 의해 병목이 발생할 수 있다.
범용 리눅스 커널의 완전 공정 스케줄러(Completely Fair Scheduler, CFS)는 처리량(Throughput)과 공정성에 초점을 맞추므로, 특정 제어 프로세스가 CPU를 선점(Preemption)해야 하는 결정론적(Deterministic) 타이밍을 보장하지 못한다. 이를 해결하기 위해 PREEMPT_RT 패치를 적용하거나, CPU 코어 격리(CPU Isolation)와 같은 커널 튜닝 기법을 적용해야 한다.

3. 커널 튜닝의 핵심 기술

호스트 머신의 실시간 튜닝 과정에서는 다음과 같은 주요 기법이 적용된다.

  1. 완전 선점형 커널 패치(PREEMPT_RT Patch): 리눅스 커널 스핀락(Spinlock) 구조를 실시간 뮤텍스(rt_mutex)로 변경하여 커널 코드 내에서도 우선순위 역전(Priority Inversion) 방지 및 강제 선점이 가능하게 한다.
  2. CPU 고립(CPU Isolation) 및 친화도(Affinity) 설정: isolcpus 커널 파라미터를 사용하여 특정 CPU 코어를 스케줄러의 부하 분산(Load Balancing) 대상에서 제외하고, 해당 코어를 ROS2의 중요 제어 프로세스 전용으로 할당한다.
  3. 인터럽트 스레딩(Interrupt Threading): 모든 하드웨어 인터럽트를 스레드로 처리함으로써 시스템의 응답성을 높이고 제어 소프트웨어의 스레드 우선순위를 하드웨어 인터럽트 스케줄러보다 상위로 격상시킬 수 있는 유연성을 제공한다.
  4. 가상 메모리 잠금(Memory Locking): 스와핑(Swapping)으로 인한 페이지 폴트(Page Fault)를 방지하기 위해 프로세스 시작 단계에서 mlockall() 시스템 호출을 이용해 사용자 공간 메모리를 물리적 주소 공간에 강제로 고정한다.

4. ROS2 환경 구축 및 최적화

실시간 커널 튜닝이 완료되면, 그 위에서 작동할 ROS2(예: Ubuntu 22.04 기반 ROS2 Humble) 미들웨어 스택에 대해 PX4-Autopilot과의 네트워크 통신 중계(Micro XRCE-DDS) 역할을 최적으로 지원하도록 세부 파라미터를 설정한다.
기본 설정의 DDS(Data Distribution Service)는 높은 신뢰성과 유연성을 표방하나, PX4와 같이 높은 전송 빈도수(e.g., 센서 데이터 100Hz 이상)를 요구하는 환경에서는 적절한 서비스 품질(QoS, Quality of Service) 프로파일 튜닝이 동반되어야 한다. 네트워크 상황에 따라 RELIABLE 정책 혹은 오버헤드를 줄이기 위한 BEST_EFFORT 정책을 적재적소에 혼용 적용한다.

4.1 ROS2 프로세스의 스케줄링 승격 사례

비결정성을 제거하기 위해 호스트에서 구동되는 단편적인 스크립트조차도 POSIX 실시간 스케줄링 정책을 적용하여 실행하는 접근이 권장된다.

import os
import sched

def set_realtime_priority():
    """ 
    제어 프로세스의 스케줄링 정책을 실시간(SCHED_FIFO)으로 상향 조정한다. 
    동반 컴퓨터 프로세스의 실행 불확실성을 낮추기 위한 필수 단계이다.
    """
    pid = os.getpid()
    
    # SCHED_FIFO 방식으로 우선순위 90 설정 (최상위 우선순위 99)
    param = os.sched_param(90)
    try:
        os.sched_setscheduler(pid, os.SCHED_FIFO, param)
        print(f"[Info] PID {pid} is successfully elevated to SCHED_FIFO with priority 90.")
    except PermissionError:
        print("[Error] 실시간 우선순위 설정에는 root 권한 또는 CAP_SYS_NICE 기능(Capability)이 요구된다.")

if __name__ == '__main__':
    set_realtime_priority()

이와 같이 호스트 운영체제의 네트워크 대역폭 및 스케줄러 스택의 공정성에 의한 비결정성(Non-determinism)을 제거하는 작업은 강력하고 안정적인 오프보드 제어를 성공적으로 수행하기 위한 근본적인 기반 엔지니어링 단계이다. 이어지는 하부 절들에서는 각각의 세부 커널 스케줄러 정책의 위상학적 구조를 분석하고 메모리 고정 기법과 의존성 관리 전략에 대해 더욱 심층적으로 다룬다.
