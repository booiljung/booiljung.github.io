61.7.4. 자원 제약 환경에서의 메모리 고갈(Out-of-Memory) 방지 및 자가 치유(Self-Healing) 시스템 프로그래밍

PX4-Autopilot(Firmware Version 1.14 이상)이 탑재된 비행 제어기(FCU, 예: Pixhawk 6C)는 범용 리눅스 컴퓨터와 달리 메인 메모리(SRAM)가 512\text{KB}에서 2\text{MB} 수준에 불과한 극단적인 자원 제약(Resource-constrained) 환경이다. 이 비좁은 메모리 공간 안에서 수십 개의 제어 코어 모듈과, 컴패니언 컴퓨터와의 고대역폭 데이터 통신을 전담하는 microRTPS_client (과거) 펌웨어 도달 이후의 uXRCE-DDS 에이전트/클라이언트 데몬이 동시에 경합한다.

장시간(수십 시간 이상) 비행하는 고정익(Fixed-wing) 윤형 비행이나 연속 임무(Continuous Mission) 환경에서 통신 모듈의 메모리 누수(Memory Leak)나 단편화(Fragmentation)가 누적되어 힙(Heap) 공간이 고갈되면, RTOS(NuttX)는 치명적인 Out-of-Memory(OOM) 오류를 일으켜 하드 폴트(Hard Fault)와 함께 모터 제어권을 상실하게 된다.

본 절에서는 자원 제약 하드웨어 관점에서 uXRCE-DDS 통신 모듈의 메모리 고갈 원인을 규명하고, 동적 할당(Dynamic Allocation)의 위험성을 배제하기 위한 정적 메모리 파라다임 및 런타임 OOM 발생 시 프로세스를 강제 재시작하여 시스템을 복구하는 자가 치유(Self-Healing) 아키텍처를 분석한다.

1. RTOS 환경에서의 메모리 고갈 동역학

FCU 내부의 컴팩트한 메모리는 스택(Stack)과 힙(Heap)으로 양분된다. 통신 미들웨어 계층의 코딩 패러다임이 메모리 고갈을 어떻게 벼랑 끝으로 몰고 가는지 이해해야 한다.

1.1 동적 메모리 할당(malloc/new)의 덫과 단편화(Fragmentation)

네트워크 통신은 본질적으로 비동기적(Asynchronous)이며, 도달하는 패킷의 크기(Payload Size)를 예측할 수 없다. 만일 ROS2와 통신하는 XRCE 클라이언트 드라이버 내부에서, 수신된 메시지를 버퍼링하기 위해 malloc() 이나 C++의 std::vector (동적 크기 변경 배열)를 반복적으로 사용한다면 끔찍한 단편화가 시작된다.

  • 외부 단편화(External Fragmentation): 다양한 크기의 패킷 수신과 소멸이 반복되면서, 힙 메모리의 빈 공간들이 치즈 구멍처럼 산산조각 난다. 총 남은 메모리가 100\text{KB}임에도 불구하고, 연속된 5\text{KB}의 덩어리 빈 공간을 찾지 못해 커널이 메모리 할당을 거부(Allocation Failed)하는 상황이 벌어진다.
  • 단편화는 켤레(Conjugate)를 이룰 수 없는 시스템의 ‘엔트로피 증가’ 함수이며, 가동 시간에 비례해 필연적으로 힙을 질식시킨다.

1.2 대기열 포화(Queue Saturation)에 의한 역압(Backpressure) 팽창

무선 통신 상태가 급격히 악화(Data Link Loss)되어 컴패니언 컴퓨터가 PX4가 보내는 sensor_combined 토픽을 제때 받아 가지 못할 경우, DDS 미들웨어의 송신 큐(QoS 신뢰성 프로파일 적용 시)에는 재전송을 기다리는 과거 패킷들이 기하급수적으로 적체된다. 이 대기열(Queue)이 팽창하면서 가용 힙 메모리를 모두 집어삼키게 된다.

2. OOM 방지 및 정적 풀 메모리(Static Pool Memory) 설계

PX4 아키텍트는 펌웨어 수준에서 메모리 붕괴의 변수를 원천 봉쇄하기 위해 RTOS 프로그래밍의 대원칙을 고수한다.

2.1 힙 리스(Heap-less) 아키텍처와 객체 풀(Object Pool)

생명 유지(Life-critical) 소프트웨어에서는 부팅 과정이 끝난 이후인 무한 루프 런타임 시점에서는 절대로 malloc() 이나 new 연산자가 단 한 번도 호출되어서는 안 된다.

  • uXRCE-DDS 클라이언트는 통신을 개시하기 전에, 설정된 MTU (Maximum Transmission Unit, 예: 512\text{Bytes}) 크기를 가진 고정 배열(Array) 버퍼 수십 개를 전역 변수(BSS 영역)나 스택의 정적 풀(Static Pool) 형태로 미리 일괄 할당(Pre-allocation)해 둔다.
  • 패킷이 들어오면 무조건 정해진 블록 하나를 빌려 쓰고(Checkout), 처리가 끝나면 풀에 반환(Return)한다.
  • 사이즈가 크든 작든 무조건 동일한 크기의 거푸집(Block)을 사용하므로, 외부 단편화 발생 확률은 리눅스 커널의 페이징(Paging) 체계 없이도 0\%로 수학적으로 억제된다.

3. 워치독(Watchdog) 기반 자가 치유(Self-Healing) 시스템 프로그래밍

설사 정적 설계가 완벽하다 하더라도, 예측할 수 없는 우주선(Cosmic Ray)에 의한 비트 플립이나 알 수 없는 포인터 무한 루프 결함이 XRCE 데몬 스레드를 교착 상태(Deadlock)에 빠뜨릴 수 있다. 이때 비행기의 메인 프로펠러 제어 권한(Attitude Control)까지 침범당하는 것을 막기 위해 에이전트 격리 및 자가 복구 로직이 가동된다.

3.1 NuttX Task 워치독 아키텍처

PX4의 메인 커맨더(Commander) 데몬은 자신을 제외한 부수적 데몬(uXRCE-DDS, MAVLink, SD 로깅 등)을 하위 태스크(Task) 프로세스로 띄운다.

stateDiagram-v2
    [*] --> Init
    Init --> Running : uxrce_dds_client start
    Running --> Heartbeat_Check
    
    Heartbeat_Check --> Running : Telemetry Tx/Rx Alive
    Heartbeat_Check --> Stalled : Queue Full or OOM Exception Block
    
    Stalled --> Kill_Task : Watchdog Timeout (e.g. 5sec)
    Kill_Task --> Restarting : Clear Socket & Memory Flush
    Restarting --> Running : uxrce_dds_client start (Self-Healed)
  1. 상태 모니터링(Heartbeat): uXRCE-DDS 클라이언트 프로세스는 정상 구동 시 특정 전역 변수 플래그(Alive Timestamp)를 지속하여 갱신한다.
  2. 이상 감지(Stall Detection): 통신 버퍼가 가득 차 메모리 할당 지연 상태에 빠지거나 코드가 블록(Block)되면 이 타임스탬프 갱신이 멈춘다. 워치독 데몬은 5초 이상 갱신이 없으면 이를 ‘프로세스 빈사’ 상태로 판정한다.
  3. 가스 자르기(Surgical Kill) 및 재시작: 워치독은 즉시 해당 프로세스의 파일 디스크립터(UART/UDP Socket)를 커널 단에서 강제로 닫고(Force Close), 태스크에 할당되었던 메모리 풀 공간을 반환(Free)시킨 뒤, 데몬을 백지상태에서 완전히 재부팅(Respawn/Self-Heal) 시킨다.

놀라운 점은, 이 통신 모듈 재부팅의 1\text{초} ~ 2\text{초} 휴보기 기간 동안 PX4의 코어 비행 제어 모듈(Rate Controller)은 아무런 영향도 받지 않고 독립 스펙트럼에서 \sim 800\text{Hz}로 기체의 수평을 잡고 평형 비행(Hold Mode)을 지속한다는 것이다. ROS2 컴패니언 컴퓨터 입장에서는 일시적인 “Connection Lost” 후 “Connection Restored” 로그만을 보게 되며, 시스템의 완전한 파국(Crash)을 면하고 임무를 속행할 수 있게 된다.

결론적으로 자원 제약 환경 하에서의 OOM 방어와 자가 치유 로직은, 단순히 메모리를 아끼는 기술이 아니라 제한된 자원이라는 잔혹한 물리적 한계 환경을 모듈화된 파티션(Partition)의 설계미학으로 극복하는 내결함성(Fault-tolerance) 프로그래밍의 진화적 정점이다.
