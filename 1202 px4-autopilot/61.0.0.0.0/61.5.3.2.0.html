61.5.3.2. 군집 비행(Swarm) 제어를 위한 서브넷 마스크 연산 기반 IP 주소 공간 분할 및 주소 해석 프로토콜(ARP) 캐싱 폭주 현상(Broadcast Storm) 억제를 위한 스위칭 허브 포트 분리 알고리즘

단일 지상 관제 시스템(GCS)이 수십 대에서 수백 대에 이르는 PX4-Autopilot 기반의 무인항공기를 통제하는 대규모 군집 비행(Swarm Flight) 아키텍처에서는, 개별 에이전트의 컴퓨팅 파워보다 텔레메트리 데이터를 실어나르는 백본(Backbone) 네트워크의 스위칭 구조가 선행적인 병목 포인트로 작용한다. 특히 Ethernet 기반의 스웜 통신망이나 고해상도 VIO/LiDAR 맵핑 데이터의 실시간 수집망에서 다수의 기체가 동일한 스위칭 허브(Switching Hub)에 연결된 경우, 주소 해석 프로토콜(ARP, Address Resolution Protocol)의 브로드캐스트 캐싱 요구가 기하급수적으로 폭증하여 네트워크 대역폭을 마비시키는 “브로드캐스트 폭풍(Broadcast Storm)” 현상이 발생한다.

본 절에서는 펌웨어 버전 1.14 이상의 PX4-Autopilot과 ROS2 미들웨어가 연동된 군집 환경에서, 서브넷 마스크(Subnet Mask)를 이진 연산(Binary Operation) 수준에서 엄밀히 통제하여 IP 주소 공간을 분할하는 방법론과, 물리적 스위칭 허브의 다중 포트를 가상 랜(VLAN) 또는 격리 규칙(Isolation Rule)으로 쪼개어 ARP 캐싱 에너지를 국소화(Localization)하는 네트워크 엔지니어링 알고리즘을 분석한다.

1. 군집 환경에서의 ARP 폭주(Broadcast Storm) 발생 동역학

네트워크 계층(IP, Layer 3)의 데이터그램이 물리 계층 링크를 타고 전송되려면, 라우터나 송신 호스트는 반드시 목적지 호스트의 데이터 링크 계층 주소(MAC 주소, Layer 2)를 알아야 한다. 이를 위해 IPv4 환경에서는 ARP 패킷을 서브넷 전체에 브로드캐스트(FF:FF:FF:FF:FF:FF)하여 “이 IP를 가진 녀석은 MAC 주소로 응답하라“고 묻는다.

1.1 O(N^2) 복잡도의 브로드캐스트 패널티

만일 100대의 드론(각각 PX4와 ROS2 컴패니언 컴퓨터 동재)이 서브넷 마스크 255.255.255.0 (즉, /24 대역, 254 호스트 가용)이라는 단일 브로드캐스트 도메인에 통째로 물려있다고 가정해보자.
시스템 부팅 시점 혹은 특정 트리거에 의해 에이전트들이 일제히 GCS나 이웃 에이전트의 MAC 주소를 찾고자 ARP Request를 발송하면, 하나의 ARP 브로드캐스트 프레임은 스위칭 허브에 의해 나머지 99대의 에이전트 NIC(Network Interface Card)로 복제(Flooding)된다.
100대의 에이전트가 동시에 ARP를 발송하면 순간적으로 100 \times 99 = 9,900개의 프레임이 스쳐 지나가며, 이는 각 드론의 리눅스 커널과 PX4 NuttX 네트워크 스택(인터럽트 처리기)을 강타하여 CPU 사용률을 급등시킨다. 이 현상은 스웜 대역폭의 페이로드 전송률을 기하급수적으로 갉아먹는다.

2. 서브넷 마스크 연산 기반 IP 공간(Address Space) 분할

ARP 브로드캐스트는 본질적으로 송신 호스트의 IP와 목적지 호스트의 IP를 서브넷 마스크와 비트 논리곱(AND) 연산하여, 두 IP가 “동일한 서브넷 네트워크(Network ID)에 속해있는가?“를 판정할 때만 발생한다. 만일 서로 다른 서브넷(Subnet)에 속해 있다면, 브로드캐스트를 포기하고 유니캐스트(Unicast)로 라우터(Gateway)의 MAC 주소에 프레임을 날린다.

2.1 가변 길이 서브넷 마스킹 (VLSM) 기반 편대 분할

가장 효과적인 억제 알고리즘은 전체 100대의 스웜을 10대씩 10개의 편대(Squadron)로 나누고, 각 편대별로 서브넷을 쪼개는 것이다.
192.168.1.0/24의 대역을 /28 (서브넷 마스크 255.255.255.240)로 분할하면:

  • 편대 A: 192.168.1.0 ~ 15 (호스트 14대 가용) -> 브로드캐스트 주소 192.168.1.15
  • 편대 B: 192.168.1.16 ~ 31 (호스트 14대 가용) -> 브로드캐스트 주소 192.168.1.31
  • …

이와 같이 쪼개서 각 기체의 interfaces (Linux) 또는 PX4 파라미터(SYS_AUTOCONFIG 제외, 고정 IP 할당)에 이식하면, 편대 A 내부의 드론 통신 시 발생하는 ARP는 다른 편대 B, C의 드론 NIC에 도달하지 않고 (혹은 도달하더라도 L3 단에서 즉시 드롭되어) 폭주 복잡도가 O(N_{squad}^2) \times 10 수준으로 극적으로 완화된다.

3. 스위칭 허브 포트 분리(Isolation) 알고리즘 및 VLAN 적용

서브넷 분할만으로는 동일한 물리적 스위치 허브(Unmanaged Switch)의 하드웨어적 브로드캐스트 복제 특성을 완전히 막지 못한다. 물리 계층(L2) 수준에서의 격리를 위해 매니지드 스위치(Managed Switch) 단의 알고리즘 튜닝이 필수적이다.

3.1 GCS 집중형 스웜의 포트 아이솔레이션(Port Isolation) 기법

군집 비행 시 에이전트 간 직접 통신(P2P)이 필요 없고, 오직 모든 드론이 GCS(또는 중앙 서버)와만 1:N 형태로 MAVLink/DDS 데이터를 취합하는 별모양(Star) 토폴로지라면 Private VLAN (Port Isolation) 기능이 가장 압도적인 성능을 발휘한다.

  • 업링크 포트(Uplink Port): GCS 서버가 연결된 스위치 포트. (Promiscuous Mode)
  • 다운링크 포트(Downlink Port): 개별 PX4 기체가 연결된 스위치 포트들. (Isolated Mode)

알고리즘: 스위치 펌웨어 설정에서 모든 다운링크 포트 간의 패킷 교환(Switching) 룰을 완전히 차단(Deny)한다.
결과적으로 드론 A가 발송한 ARP 브로드캐스트는 오직 업링크 포트(GCS)로만 전달되며, 다른 드론 B, C 등의 포트로는 복제(Flooding)되지 않는다. 이는 L2 브로드캐스트 스톰의 물리적 가능성 자체를 절연시키는 방법이다.

3.2 분산 자율(P2P) 스웜의 정적 ARP(Static ARP) 테이블 선 주입

드론 간 상호 충돌 회피(Collision Avoidance)나 편대 제어를 위해 로컬 P2P 통신이 반드시 필요한 경우(예: ROS2 Multi-Agent DDS Discovery) 포트들을 격리시킬 수 없다.
이때 궁극적인 해결책은 부팅 시점에 네트워크 상주 데몬을 끄고, PX4 펌웨어의 커널이나 컴패니언 컴퓨터의 리눅스 커널 테이블에 이웃 드론들의 IP-MAC 매칭 정보를 강제로 하드코딩(Static ARP Table Injection) 시켜버리는 것이다.
리눅스 부팅 쉘스크립트에 arp -s 192.168.1.20 00:11:22:33:44:55 형태의 리스트를 밀어 넣음으로써, 운영체제 네트워크 스택은 ARP 브로드캐스트 과정 전체를 영구적으로 우회(Bypass)하게 되고, 브로드캐스트 폭풍의 씨앗 자체를 소멸(Extinction)시킬 수 있다.
