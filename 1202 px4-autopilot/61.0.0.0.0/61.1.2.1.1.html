61.1.2.1.1 최악 실행 시간(WCET) 보장을 위한 결정론적(Deterministic) 블록 메모리 병합 및 순회 알고리즘

PX4-Autopilot과 같은 실시간 임베디드 제어 시스템에서는 고도의 안전성과 신뢰성을 요구한다. 특히 무인항공기(UAV)의 경우 센서 데이터를 읽어 들이고, 제어 루프를 연산하며, 구동기에 명령을 내리는 모든 과정이 엄격한 데드라인(Deadline) 안에 완료되어야 한다. 이를 위해 실시간 운영체제(RTOS) 환경 하에 구동되는 애플리케이션이나 미들웨어(예: Micro XRCE-DDS)는 기능의 실행이 지정된 시간 내에 반드시 종료됨을 보장해야 하는데, 이를 최악 실행 시간(WCET: Worst-Case Execution Time) 보장이라고 한다.

본 장에서는 동적 메모리 할당으로 인해 발생하는 비결정론(Non-determinism)과 단편화(Fragmentation) 문제를 극복하고 WCET를 보장하기 위해 도입된 결정론적 블록 메모리 병합(Deterministic Block Memory Coalescing) 및 **순회 알고리즘(Traversal Algorithm)**에 대하여 심도 있게 다룬다. 본 내용은 PX4 v1.14와 ROS 2 Humble 환경을 기준으로 작성되었다.

1. 동적 메모리와 최악 실행 시간(WCET)의 관계

전통적인 소프트웨어 아키텍처에서 가장 흔히 쓰이는 malloc()과 free() 함수는 요구하는 크기에 맞추어 메모리 블록을 힙(Heap) 공간에서 할당하고 반환한다. 하지만 임베디드 및 실시간 로보틱스 프레임워크인 PX4-Autopilot 환경에서는 이러한 표준 동적 할당 방식이 다음과 같은 두 가지 치명적 문제를 발생시킨다.

  1. 지연 시간의 예측 불가능성: 가용 블록을 찾기 위해 자유 공간 목록(Free list)을 순회하는 시간이 힙 단편화 정도에 따라 매번 달라진다.
  2. 단편화로 인한 할당 실패: 충분한 가용 메모리가 남아있음에도 불구하고 연속된 공간이 확보되지 않아 메모리 할당에 실패하게 되어, 치명적인 비행 제어 오류 체인(Error chain)을 유발할 수 있다.

이 때문에 PX4 펌웨어에서는 비행에 핵심적인 객체 요청 브로커(uORB: Micro Object Request Broker) 및 미들웨어 계층의 통신 엔진(Micro XRCE-DDS Client)에서 런타임에 동적 할당을 철저히 배제하거나, 사전에 할당된 정적 메모리 풀(Static Memory Pool)을 운용하는 전략을 택한다.

2. 정적 풀 기반 블록 메모리 병합 알고리즘

기존 동적 메모리의 단점을 극복하기 위해 통신 미들웨어나 비행 제어 스택은 부팅 혹은 모듈 초기화 시점에 큰 메모리 풀을 미리 할당받거나 정적 배열을 선언한다. 이 공간을 일정한 크기의 청크(Chunk) 또는 블록들로 나누어 논리적인 힙으로 자체 운용하게 된다.

사용을 마친 블록이 반환될 때, 파편화된 빈 공간을 연속된 하나의 큰 영역으로 합쳐주는 과정이 바로 **메모리 병합(Memory Coalescing)**이다. 일반적인 가비지 컬렉터(Garbage Collector)의 병합 과정은 처리 시간이 비결정론적이어서 WCET가 보장되지 않으므로, 실시간 시스템을 위한 결정론적 병합 설계가 요구된다.

2.1 인접 블록 병합 (Buddy Memory Allocation 활용)

결정론적으로 병합 시간을 설계하는 가장 훌륭한 방법의 하나는 블록의 크기를 항상 2의 거듭제곱으로 유지하는 버디 메모리 할당(Buddy Memory Allocation) 기법을 활용하는 것이다.

  • 할당 단계: 크기가 2^n인 요청이 들어왔을 때 해당 크기가 없으면 한 단계 더 큰 2^{n+1} 블록을 정확히 반씩 분할(Split)하여 반환한다.
  • 반환 단계: 블록이 반환될 때, 메모리 주소를 비교하여 자신과 “버디(Buddy, 짝꿍)“인 블록이 빈 상태라면 지연 없이 즉시 하나의 2^{n+1} 블록으로 병합한다.

병합 시나리오의 O(1) 실행 시간 보장
버디 시스템 하에서는 블록을 병합하기 위해 힙 전체를 뒤질 필요가 없으며, 자신의 주소와 크기를 이용한 단순한 비트 시프트 연산(Bit-shift operation)과 XOR 비트 연산만으로 버디 블록의 주소를 O(1) 시간에 찾을 수 있다. 버디가 비어있다면 합치고, 상위 노드에 대해 이 과정을 재귀적으로 수행하되 그 깊이(Depth)의 상한선이 고정적이므로 WCET가 엄격하게 보장된다.

3. WCET 보장 메모리 순회(Traversal) 알고리즘

메모리를 관리하는 데이터 구조를 탐색하는 순회 알고리즘 역시 예측 불가능한 반복 연산을 제거해야 한다. Ardupilot과 PX4 공통적으로 비행 중에는 루프 타임을 엄수해야 하는데, 연결 리스트(Linked list)로 빈 블록을 관리하면 할당 가능한 블록을 찾는(순회하는) 데 O(N)의 시간이 걸려 실시간성을 저해한다.

PX4와 uORB 코어 아키텍처에서는 이를 해결하기 위해 두 가지 방식 중 하나를 사용한다.

3.1 O(1) 고정 풀 접근 프레임워크

서로 다른 크기의 가변 블록을 할당하기보다는, 동일한 크기의 객체를 다수 생성해야 할 경우 **고정 크기 객체 풀(Fixed-size Object Pool)**을 운용한다.
단일 크기 블록 풀은 스택(Stack)이나 큐(Queue) 형태로 가용 블록들의 포인터(인덱스)를 관리한다.

  • 메모리가 필요할 경우 인덱스를 통해 O(1) 시간 복잡도로 블록 포인터를 얻어낸다.
  • 반환 시에도 스택의 상단 혹은 큐의 꼬리에 포인터를 밀어넣는 것으로 처리가 종료된다. 이는 병합이나 순회를 완전히 제거한 극한의 WCET 보장 기법이다.

3.2 분리 가용 리스트 (Segregated Free List) 접근

서로 다른 크기의 페이로드(Payload)를 다뤄야 하는 Micro XRCE-DDS 버퍼 관리 등에서는 크기 범위별로 빈 블록의 리스트를 여러 개 운용하는 분리 가용 리스트 아키텍처를 도입한다.
원하는 크기에 해당하는 연결 리스트로 바로 접근하므로 전체 순회를 수행하지 않아도 되며, 블록 탐색에 필요한 최대 순회 횟수를 상수로 제한할 수 있어 WCET를 안전하게 예측 범위 내로 수렴하게 만든다.

4. 실시간성 검증 관점의 차이와 아키텍처 설계 지침

  • 비결정성과 소프트 실시간(Soft Real-Time): 로깅(Logging) 애플리케이션 등은 메모리 병합 시 다소의 순회 시간이 들더라도 비행 안정성을 침해하지 않으므로 유연함을 허용한다.
  • 결정성과 하드 실시간(Hard Real-Time): 자세 제어기(Attitude Controller), 이스케이프/페일세이프(Failsafe) 핸들러, 미들웨어의 내부 토픽 동기화 루틴은 하드웨어 타이머에 의존적으로 돌아가야 하므로, 블록 탐색 및 병합 로직의 WCET가 스케줄러 루프 주기(예: 1ms 또는 2.5ms)를 초과해서는 안 된다.

결론적으로 PX4-Autopilot에서 결정론적 메모리 병합 알고리즘 및 순회 모델 설계는 단순한 소프트웨어 최적화 이상의 과제이며, 무인 항공기가 공중에서 예기치 못한 메모리 고갈과 타이밍 미스(Timing miss)로 추락하지 않게 하는 안전-비판적(Safety-critical) 자원 관리의 핵심 축이다.
