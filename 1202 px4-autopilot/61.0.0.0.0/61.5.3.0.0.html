61.5.3. 네트워크 전송 계층(UDP/TCP) 기반 논리적 브로드캐스트 도메인 격리

단일 기체 중심의 관제 아키텍처를 벗어나 다중 에이전트(Swarm) 환경이나 대규모 HIL(Hardware-In-the-Loop) 시뮬레이션 환경으로 스케일 아웃(Scale-out)할 때, 가장 빈번하게 발생하는 통신 장애의 원흉은 무분별한 패킷 브로드캐스팅(Broadcasting)으로 인한 네트워크 대역폭 포화(Saturation) 현상이다. 특히 PX4-Autopilot 생태계(Firmware Version 1.14 이상)와 ROS2 미들웨어 환경이 이더넷(Ethernet) 스위치를 공유할 때, 물리적 도메인 분리 방안 없이 기본 동작을 방치하면 모든 노드가 불필요한 트래픽의 수신 및 파싱(Parsing)에 CPU 사이클을 낭비하게 된다.

본 절에서는 텔레메트리 트래픽(MAVLink)과 uXRCE-DDS 에이전트 브리지 트래픽이 공존하는 복합 네트워크에서, 네트워크 전송 계층(Transport Layer: UDP/TCP)의 특성을 활용하여 다수의 노드가 얽힌 논리적 브로드캐스트 도메인(Logical Broadcast Domain)을 어떻게 격리(Isolation)하는지 학술적으로 분석한다.

1. MAVLink 및 DDS 미들웨어의 기본 브로드캐스트 작동 원리 관찰

무인기 시스템 초기 설정에서 “플러그 앤 플레이(Plug and Play)“의 편의성을 제공하기 위해, 네트워크 계층은 상대편 호스트의 IP 주소를 몰라도 통신 객체를 자동으로 찾는 발견 메커니즘(Discovery Mechanism)을 기본으로 가동한다.

1.1 MAVLink UDP 브로드캐스트 방식

지상 관제 시스템인 QGroundControl(QGC)을 실행하면, QGC는 기본적으로 로컬 네트워크의 서브넷(Subnet) 상에서 255.255.255.255 형태의 글로벌 브로드캐스트 주소 또는 서브넷 제한 브로드캐스트 주소를 사용하여 주기적으로 UDP 심장박동(Heartbeat) 메시지를 살포(Spray)한다. PX4 펌웨어 내의 mavlink 모듈 역시 mavlink start -u 14550 명령을 통해 특정 UDP 포트를 개방하고 이 브로드캐스트 도메인에 합류하여 자동 페어링(Auto-Pairing)을 성사시킨다.

1.2 RTPS/DDS 기반 단순 멀티캐스트(Multicast)의 한계

마찬가지로 ROS2가 구동되는 컴패니언 컴퓨터나 오프보드(Offboard) 터미널들 역시, Fast DDS 등 기반 미들웨어를 통해 표준 멀티캐스트 라우팅(예: 239.255.0.1)을 이용하여 구독/발행자 개체 탐색(Entity Discovery) 트래픽을 송출한다.

겉보기에는 멀티캐스트가 브로드캐스트보다 진보해 보이나, 일반적인 더미 스위칭 허브(Unmanaged Switch) 환경에서 스위치의 MAC 테이블은 멀티캐스트 프레임을 브로드캐스트 프레임과 동일하게 취급하여 스위치의 **모든 핀치(Port)로 플러딩(Flooding)**하는 역효과를 빚는다. 결과적으로, A 드론의 관성 센서 구독 토픽이 본질적으로 B 드론의 컴패니언 컴퓨터 네트워크 인터페이스(NIC)까지 침범하여 로컬 커널의 IP 역직렬화(Deserialization) 부하를 유발하는 브로드캐스트 폭풍(Broadcast Storm)이 초래된다.

2. 논리적 домен 격리(Logical Domain Isolation) 아키텍처 수립

수십 대의 에이전트가 단일 스위치 허브 체인(Switch Hub Chain)에 물려 있는 스웜(Swarm) 아키텍처에서 이러한 스케일 병목을 차단하려면, IP 서브넷 마스킹(Subnet Masking) 수준을 넘어선 전송 계층 기반의 논리적 격리 기법을 투입해야 한다.

2.1 유니캐스트(Unicast) 지향 UDP 터널링 설정

가장 확실한 브로드캐스트 도메인 억제 방법은 MAVLink 및 uXRCE-DDS 라우팅의 종단점을 **포인트-투-포인트(Point-to-Point) 유니캐스트 트래픽으로 강제 상호 결속(Binding)**하는 것이다. 상호간의 자동 디스커버리 기능을 비활성화(Disable)하고, 각 드론의 IP와 QGC 측의 IP, 혹은 컴패니언 컴퓨터 상의 브리지 에이전트(Agent) IP를 정적(Static)으로 하드코딩(Hardcoding)한다.

  • MAVLink 스태틱 매핑: PX4 터미널 시작 스크립트(rc.board 등)에서 mavlink start -t <GCS Target IP>의 -t 옵션을 강제 부여하여 단일 타겟에만 IP:UDP포트 기반으로 데이터를 직사(Direct Shot)하도록 구성한다.
  • XRCE-DDS 터널링: PX4 내부의 마이크로 클라이언트(Micro Client)가 서버 역할을 하는 ROS2 MicroXRCEAgent로 연결할 때, 멀티캐스트 기반의 탐색 프로토콜을 우회하여 IP/Port 기반의 UDP/TCP 프로필만을 사용하도록 초기화 파라미터(UXRCE_DDS_CFG)를 엄격하게 제한한다.

2.2 TCP 기반의 세션(Session) 레이어 연결 분산

오프보드 비행 환경에서는 고속의 상태 추정 업데이트(e.g., 200\text{Hz} Odom)가 요구되므로 UDP가 기본으로 쓰이나, MAVLink의 미션(Mission) 업로드/다운로드 프로토콜이나 파라미터 동기화(Parameter Sync)와 같이 ’오류 없는 순차 전송 100% 보장’이 필요한 트래픽의 경우, 전송 계층을 TCP로 분리하는 듀얼 아키텍처(Dual Architecture)가 강력한 이점을 지닌다.

PX4 mavlink 모듈 인스턴스는 한 기체에서 여러 개를 동시 구동시킬 수 있다.

  • 텔레메트리(UDP): 자세, 위치, 배터리 잔량 등 순간적인 지연(Latency)이 치명적이지만 중간에 한 프레임 유실되어도 곧 복구되는 특성의 데이터 군은 UDP 기반의 보조 포트(e.g., 14551) 유니캐스트로 브로드캐스트 도메인을 탈피한다.
  • 가용한 파라미터 제어(TCP): 미션 경로점, GCS 설정 인터페이스 등은 PX4 내부 모듈을 TCP 서버(-m custom -p 5760)로 바인딩하여, 연결 지향형 안정 송신을 보장함과 동시에 TCP 핸드셰이크(Handshake) 자체 구조로 타 노드와의 난입(Interruption)을 커널 수준에서 완벽하게 격리해 낸다.

3. 요약 및 실무 적용 지침

이더넷 인터페이스 하나에 다수의 비행 기체 운영 및 로컬 ROS2 관제망 레이어를 혼재시킬 때는, 전송 계층의 논리적 격리 부재 시 나타나는 CPU 인터럽트 폭주 및 통신 대역폭 고갈 사태에 각별히 유의해야 한다.

  • PX4-Autopilot 및 GCS 소프트웨어 양측에서 묵시적 브로드캐스트 기반의 “자동 찾기(Auto-Discovery)” 의존을 근절한다.
  • IP 라우팅 테이블(Routing Table) 및 유니캐스트 -t (Target) 옵션을 통한 “1:1 소켓 바인딩(Socket Binding)” 원칙을 철저히 준수하여 스위치(Switch)에서 패킷이 플러딩되는 것을 원천 차단해야 한다.
