61.5.1.2. 사용자 데이터그램 프로토콜(UDP) 환경에서 인터넷 프로토콜(IP) 및 매체 접근 제어(MAC) 헤더 캡슐화 크기를 감안한 최대 전송 단위(MTU) 기반 최적 패킷 단편화 한계 산출

현대의 PX4-Autopilot 기반 다중 에이전트 시스템(Multi-Agent System) 및 고해상도 센서 퓨전 환경에서는 단순 직렬(Serial) 통신을 뛰어넘어, 로컬 무선 랜(Wi-Fi) 또는 전용 5G 무선 데이터 링크를 통한 이더넷(Ethernet) 기반의 사용자 데이터그램 프로토콜(UDP, User Datagram Protocol) 통신이 GCS 및 컴패니언 컴퓨터 연동의 주축을 이룬다. Micro XRCE-DDS 프로토콜 또한 UDP 위에서 동작하며, 실시간성 보장을 위해 연결형인 TCP(Transmission Control Protocol) 대신 비연결형인 UDP 멀티캐스트(Multicast) 및 유니캐스트(Unicast)를 기본 전송 계층으로 채택한다.

그러나 고속 UDP 환경이라 할지라도 네트워크 하드웨어가 한 번에 처리할 수 있는 물리적 덩어리인 프레임(Frame)의 크기에는 구조적 한계가 존재한다. 비행 제어를 위한 대용량 페이로드(예: 고해상도 VIO 오도메트리 파티클 행렬, 포인트 클라우드 배열 등) 전송 시 이 한계를 무시하면 네트워크 스택 단에서 강제적인 패킷 단편화(IP Fragmentation)가 발동되어 시스템 전체의 통신 지연(Latency) 폭증과 패킷 손실률(Packet Loss Rate) 상승을 야기한다.

본 절에서는 UDP 통신 시 발생하는 MAC 보조 헤더 및 IP 캡슐화(Encapsulation) 오버헤드를 수학적으로 분석하고, 이를 감안하여 단편화(Fragmentation)가 발생하지 않는 최적의 페이로드 크기 한계인 최대 전송 단위(MTU, Maximum Transmission Unit)를 산출하는 메커니즘을 심도 있게 파헤친다.

1. 이더넷 프로토콜 스택의 캡슐화(Encapsulation) 오버헤드

PX4 탑재 기체(예: Pixhawk 6X의 내장 이더넷 포트)가 이더넷 케이블 또는 무선 공유기를 통해 ROS2 노드(Target)로 데이터를 쏘아 보낼 때, 응용 계층의 데이터(uORB 토픽)는 하위 계층으로 내려갈수록 점차 헤더(Header)가 덧붙여지며 캡슐화된다.

표준 이더넷(IEEE 802.3) 규격에서 정의된 각 계층별 기본 오버헤드 바이트(Bytes) 수는 다음과 같다.

  1. 데이터 링크 계층 (MAC Layer):
  • 목적지 MAC 주소(6\text{ bytes}) + 출발지 MAC 주소(6\text{ bytes}) + EtherType(2\text{ bytes}) = 14 \text{ bytes}
  • 프레임 뒷부분의 오류 검출용 FCS(Frame Check Sequence): 4\text{ bytes}
  • (참고: 프레임 간 갭(IFG) 및 프리앰블(Preamble)은 물리적 신호 타이밍이므로 소프트웨어 페이로드 크기 계산에서는 배제한다.)
  1. 네트워크 계층 (IPv4 Header):
  • 버전, 헤더 길이, ToS, 전체 길이, 식별자, TTL, 프로토콜, 헤더 체크섬, 출발지 IP, 목적지 IP 등을 포함한 최소(고정) 크기: 20\text{ bytes} (옵션 부재 시)
  1. 전송 계층 (UDP Header):
  • 출발지 포트, 목적지 포트, UDP 길이, 체크섬: 고정 8\text{ bytes}

1.1 표준 MTU 기반 네트워크 프레임 경계 분석

전통적으로 보편화된 이더넷 환경에서 하드웨어 스위치(Switch)와 라우터(Router)들이 자르지 않고(Fragment-free) 온전히 넘길 수 있는 IP 패킷(IP Header + Data)의 최대 크기를 MTU라고 명명하며, 기본값은 일반적으로 **1500\text{ bytes}**로 고정되어 있다.

이 MTU 1500\text{ bytes}는 MAC 헤더(14\text{ bytes})와 FCS(4\text{ bytes})를 제외한 3계층(IP) 이상의 순수 IP 데이터그램(Datagram) 최대 길이를 지칭한다.

2. IP 패킷 단편화(Fragmentation) 한계 산출 알고리즘

소프트웨어 개발자(또는 ROS2 애플리케이션 엔지니어) 입장에서 가장 중요한 것은 “운영체제의 네트워크 소켓(Socket)에 sendto() 함수를 사용하여 밀어 넣을 수 있는, 절대로 분절되지 않는 순수 응용 계층 페이로드(Pure Payload)의 최대 크기는 몇 바이트인가?“이다.

이 한계점(Threshold)은 위에서 도출한 오버헤드 공식을 역산하여 다음과 같이 명확하게 한정할 수 있다.

\text{Max Unfragmented Payload} = \text{MTU} - (\text{IPv4 Header Size} + \text{UDP Header Size})
= 1500\text{ bytes} - (20\text{ bytes} + 8\text{ bytes}) = 1472\text{ bytes}

따라서, PX4-Autopilot에서 ROS2로 송출되는 **단일 UDP 데이터그램 내부의 내용물 크기가 1472\text{ bytes}를 초과하는 순간 IP 단편화 현상이 트리거(Trigger)**된다.

    A[XRCE-DDS Agent Buffer] --> B{Payload > 1472 Bytes ?}
    B -- Yes --> C[IP Fragmentation 발생]
    C --> D(단편화된 다중 IP 패킷 전송)
    D --> E[수신측 커널 단 재조립 대기]
    E --> F{일부 패킷 유실?}
    F -- Yes --> G((전체 프레임 드롭 및 지연 폭증))

    B -- No --> I[단일 IP 패킷 캡슐화]
    I --> J(즉각적인 선로 전송)
    J --> K[재조립 없는 고속 인터럽트 수신]
    K --> H

2.1 IP 단편화의 위험성 (Ardupilot/PX4 GCS 관제 공통 적용)

단편화가 위험한 이유는, 1472\text{ bytes}를 넘는 1개의 커스텀 uORB 메시지(예: 대용량 Array를 포함한 메시지)를 보내기 위해 IP 계층이 패킷을 여러 개(Fragment 1, Fragment 2 …)로 쪼갤 때, 분할된 패킷 중 단 한 개라도 채널 혼잡(Congestion) 상태에서 라우터에 의해 버려지면(Drop), 수신 측 커널(Linux Network Stack)은 나머지 정상 수신된 파편들을 재조립(Reassembly) 버퍼에 들고 있다가 타임아웃(Timeout) 판정이 내려질 때까지 기다린 뒤 전체 메시지를 모두 폐기해버리기 때문이다. 이는 통계적으로 비행 제어 루프의 지터(Jitter)를 폭증시키는 원흉이 된다.

3. 최적 패킷 페이로드 설계 및 프로비저닝 전략

PX4 펌웨어와 XRCE-DDS 환경에서 대규모 파라미터 리스트를 불러오거나, 복잡한 군집 로봇의 경로점(Target Waypoints) 배열을 한 번에 동기화하려 할 때 이러한 단편화 한계를 마주하게 된다. 이를 회피하기 위한 최적 통신 아키텍처는 다음과 같다.

  1. uORB 토픽의 소프트웨어 차원 크기 제한 (Constraint):
    .msg 파일 설계 단계에서, 가변 배열(Bounded VLA)이나 고정 배열의 최대 크기를 조합했을 때 전체 구조체의 직렬화(Serialized) 풋프린트(Footprint)가 XRCE-DDS 프로토콜 헤더(보통 24\text{ bytes} 내외)를 합쳐 1472\text{ bytes}를 넘지 않도록 규격을 제한한다.
    \text{Target Message Size} \le 1472 - \text{XRCE\_Header\_Overhead}
  2. 시퀀스(Sequence) 기반의 애플리케이션 계층 분할 송출:
    도저히 1472 바이트 안에 들어가지 않는 대용량 맵(Map) 데이터나 비전 기반 포인트클라우드가 존재할 경우, UDP 프로토콜 수준에 이를 통째로 맡겨 IP 단편화를 유발하기보다는 애플리케이션 레이어(uORB Publish 로직 내부 또는 ROS2 브리지 노드)에서 인위적으로 ~1\text{ KByte} 단위의 청크(Chunk)로 쪼개어 각각 독립적인 패킷 시퀀스 번호(Sequence Number)를 부여한 뒤 순차 송신하는 패턴(Application-level Segmentation)을 채택해야 한다.

요컨대, 실시간성이 생명인 무인항공기 관제 및 오프보드(Offboard) 분산 제어 시스템에서 MTU 1500 기반의 1472\text{ bytes} 페이로드 장벽은 단순한 네트워크 상식을 넘어, 끊김 없는 비행 동역학 데이터 전송 루프를 지탱하는 보이지 않는 “하드웨어의 레드라인(Red-Line)“으로 간주되어야 한다.
