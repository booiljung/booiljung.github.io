61.5.2.1. 가상 파일 시스템(VFS) 노드 할당을 통한 문자 디바이스(Character Device) 매핑 및 직렬 포트 제어(termios) 구조체의 비동기 플래그 세팅 원리

PX4-Autopilot의 핵심 운영체제인 NuttX RTOS와 컴패니언 컴퓨터(Companion Computer) 환경인 Linux 시스템은 모두 POSIX(Portable Operating System Interface) 규격을 준수하는 유닉스 계열 아키텍처를 기반으로 한다. 이러한 아키텍처의 가장 근본적인 철학은 “모든 것은 파일이다(Everything is a file)“라는 명제에 있다. 따라서 비행 제어기 하드웨어 핀(Pin)에 물리적으로 종속된 직렬 포트(Serial Port, UART)조차도 응용 계층(Application Layer)에서는 단순한 파일 입출력(File I/O)으로 취급되어야 한다.

본 절에서는 하드웨어 추상화 계층(HAL) 위에서 가상 파일 시스템(VFS, Virtual File System) 노드가 어떻게 문자 디바이스(Character Device)로 매핑(Mapping)되는지, 그리고 XRCE-DDS 에이전트 브리지(또는 MAVLink) 모듈이 직렬 포트의 데이터 스트림을 비차단(Non-blocking) 비동기 모드로 다루기 위해 POSIX termios 구조체의 제어 플래그를 어떻게 튜닝하는지 학술적 관점에서 상세히 기술한다.

1. 가상 파일 시스템(VFS) 기반 문자 디바이스 매핑 메커니즘

PX4 펌웨어 부팅 초기화 단계(Boot Initialization)에서, 마이크로컨트롤러(MCU)에 내장된 고정 채널 UART 하드웨어 자원은 운영체제의 디바이스 드라이버 로드 파이프라인을 거쳐 VFS 상의 노드(Node)로 거듭난다.

1.1 하드웨어 인스턴스의 장치 노드 변환 과정

  1. 드라이버 등록(Driver Registration): PX4-Autopilot(Firmware 1.14 기준)의 보드 특화(Board-specific) 초기화 코드(/src/boards/.../init.c 등)는 MCU 초기화 직후 nuttx/drivers/serial/ 하위의 직렬 포트 등록 함수(예: uart_register())를 호출한다.
  2. 문자 디바이스 생성(Character Device Creation): 운영체제 커널 계층에서 /dev/ttyS0, /dev/ttyS1, /dev/ttyACM0 등의 장치 명칭(Device Name)을 VFS 계층 트리에 동적으로 바인딩한다. 문자 디바이스는 블록 디바이스(Block Device)와 달리 데이터의 순차적 문자 단위(Byte-by-Byte 스트림) 입출력을 전제로 설계된다.
  3. 가상 파일 시스템 오퍼레이션(VFS Operations) 바인딩: 파일 입출력 시스템 콜인 open(), read(), write(), ioctl() 호출이 접수되었을 때 이를 실제로 하드웨어 레지스터 제어 루틴으로 넘겨줄(Dispatch) 함수 포인터 테이블(file_operations 구조체)이 장치 노드에 사상(Map)된다.

이러한 VFS 추상화 덕분에, 컴패니언 컴퓨터의 ROS2 시스템이나 기체 내부의 오프보드(Offboard) 애플리케이션 등은 하드웨어 종속성을 완전히 탈피하고 오직 /dev/ttyS1이라는 유일무이한 식별자 노드(Node)만으로 직렬 데이터의 송수신 및 인터럽트 제어를 통제할 수 있다.

2. POSIX termios 구조체와 직렬 포트 제어(Control) 패러다임

장치 노드 매핑 이후 애플리케이션이 이 포트와 대화하기 위해서는, 포트의 하드웨어 특성(보드 레이트, 패리티, 시작/정지 비트 등)과 운영체제 커널 단의 입출력 버퍼 특성을 섬세하게 조율해야 한다. POSIX 표준은 이러한 세팅을 관장하는 데이터 구조체로 termios 객체를 제공한다.

2.1 원시 통신 모드(Raw Mode) 구조체 플래그 세팅 원리

지상 관제 시스템(GCS) 또는 컴패니언 컴퓨터 통신 데이터(px4_msgs)는 사람이 읽기 위한 단순 텍스트 터미널(TTY) 기반의 상호작용 문자가 아니라, 철저하게 해상도 짙은 구조화된 이진 바이너리(Binary) 데이터이다. 운영체제 커널의 TTY 드라이버가 문자열 치환(예: \n을 \r\n으로 자동 변환)이나 텔넷(Telnet) 제어 문자 처리를 개입시키는 것을 원천 차단하기 위해 Raw 모드(원시 모드) 전개가 핵심이다.

이를 달성하기 위한 termios 플래그 초기화(Initialization)는 다음의 비트 마스킹(Bit-masking) 공식을 따른다.

  1. 입력 옵션(Input Flags, c_iflag):
    IGNBRK | IGNCR | INLCR | ICRNL | IXON | IXOFF 등을 모두 Bit-Clear(& ~) 하여, 수신된 바이트 스트림을 커널이 임의로 조작하거나 변형하는 것을 막는다.
  2. 출력 옵션(Output Flags, c_oflag):
    OPOST 처리를 제거하여 개행 문자의 로컬 변환을 막고 원형 그대로 밀어낸다.
  3. 제어 옵션(Control Flags, c_cflag):
    CS8 (8 데이터 비트), -PARENB (패리티 없음), -CSTOPB (1 정지 비트)를 통해 8N1 텔레메트리 하드웨어 전송 양식을 수립한다. 만일 하드웨어 흐름 제어(Hardware Flow Control)가 필요하다면 CRTSCTS 마스크를 삽입한다.
  4. 로컬 옵션(Local Flags, c_lflag):
    ICANON (캐노니컬 모드), ECHO (로컬 에코), ISIG (신호 생성) 처리를 무효화하여 모든 데이터를 응용 프로그램으로 직행시킨다.

3. 텔레메트리 레이턴시 제로화를 위한 비동기 및 비차단(Non-blocking) 플래그 최적화

수백 \text{Hz}의 제어 대역폭을 소화하는 드론 분산 운영 환경에서는 read()나 write() 시스템 콜 호출 시 하드웨어의 입출력 완료 및 운영체제의 버퍼 채움을 기다리며 현재 쓰레드(Thread)가 정지하는 스레드 차단(Thread Blocking) 사태를 엄격히 금한다.

이러한 지연(Latency) 방지를 위해 개발자는 장치 파일 노드를 개방(open())할 때 O_NONBLOCK 및 O_ASYNC 상태 플래그를 삽입하고, 추가로 termios 구조체 내부의 특별 제어 변수(VMIN, VTIME)를 최적화한다.

\text{VMIN} = 0, \quad \text{VTIME} = 0

위와 같은 공식 조합은 커널의 직렬 드라이버에게 “현재 수신 FIFO 버퍼에 단 1바이트의 데이터라도 도달해 있으면, 타임아웃을 대기하지 말고 즉각 응용 스레드(Application Thread) 공간으로 복사하여 반환하라“는 가장 엄밀한 제로 레이턴시(Zero-latency) 폴링 구조를 강제한다. 이는 다중 스레드(Multi-thread) 환경에서 select()나 poll() API 시스템 콜과 융합되어, 수신 데이터가 존재하는 찰나에만 비동기로 콜백(Callback)을 동작시키는 저전력 고효율 비차단 IO 모델(Asynchronous Non-blocking IO)을 완성한다.

결론적으로 VFS를 통한 다형적(Polymorphic) 문자 디바이스 매핑과 termios 구조체의 정교한 비트 마스킹 메커니즘은, PX4-Autopilot과 이기종 플랫폼 간의 직렬 통신 파이프라인이 임의의 커널 오버헤드를 우회하고 순수한 마이크로초(Microsecond) 단위의 데이터 무결성을 보존토록 하는 프로그래밍적 기반 기술이다.
