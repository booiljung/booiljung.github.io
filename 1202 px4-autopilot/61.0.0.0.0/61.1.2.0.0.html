61.1.2 극소형 확장 리소스 제약 환경(Micro XRCE-DDS)의 메모리 및 처리기 최적화 설계

안전 필수(Safety-critical) 실시간 운영체제(RTOS) 환경인 NuttX 상에서 구동되는 PX4-Autopilot은 비행 제어 알고리즘(Position/Attitude Control)과 상태 추정기(EKF2) 연산에 CPU와 메모리의 상당 부분을 할당해야 한다. 따라서 ROS 2와의 통일된 연동을 위해 도입된 외부 통신 스택은 극도로 가벼워야 하며 제어 주기에 간섭을 일으켜서는 안 된다. 이를 위해 고안된 것이 바로 “극소형 확장 리소스 제약 환경용 데이터 분배 서비스(Micro XRCE-DDS: eXtremely Resource Constrained Environments DDS)“이다. 본 절에서는 PX4 환경 내에서 Micro XRCE-DDS 클라이언트가 어떠한 방식으로 메모리와 처리기를 최적화하도록 설계되었는지 그 아키텍처 원리를 심층적으로 분석한다.

1. Micro XRCE-DDS 설계의 본질적 목표

전통적인 완편(Full-fledged) DDS 스택 로직(예: Fast DDS, Cyclone DDS)은 동적인 토픽 발견(Discovery), 이력(History) 캐싱, 복잡한 QoS 재전송 버퍼 관리 등으로 인해 수십 메가바이트의 메모리를 요구하며 빈번한 동적 메모리 할당(Dynamic Allocation)을 수반한다.
반면, 비행 제어기가 탑재된 마이크로컨트롤러(예: Pixhawk FMUv5/v6X의 STM32)는 보통 수백 킬로바이트(KB) 남짓한 가용 SRAM만을 갖는다. Micro XRCE-DDS는 이러한 극단적인 제약 환경에 맞추어 완전한 P2P 통신망 릴레이 기능을 **에이전트(Agent)**로 오프로딩(Offloading)하고, PX4 내부에는 상태 저장소를 최소화한 **클라이언트(Client)**만을 탑재하는 비대칭 아키텍처를 취했다.

2. 메모리 최적화: 정적 메모리 관리(Static Memory Management)

RTOS 환경에서 잦은 malloc()과 free()의 호출은 힙 단편화(Heap Fragmentation)를 유발하여 커널 패닉(Kernel Panic)이나 치명적인 시스템 오동작(Out Of Memory)을 초래할 수 있다. PX4 펌웨어(v1.14 기준) 내에 내장된 Micro XRCE-DDS 클라이언트 라이브러리(Micro-CDR 기반)는 다음과 같은 메모리 최적화 원칙을 고수한다.

2.1 정적 버퍼 사전 할당(Pre-allocated Static Buffers)

시스템 부팅(Boot-up) 시점에 필요한 모든 스트림 버퍼와 통신 객체(Session, Stream)를 BSS 영역 혹은 사용자 지정 메모리 풀(Pool)에 정적으로 할당한다.

  • MTU(Maximum Transmission Unit) 기반 스트림 분리: 시리얼 모뎀(UART/Telem) 계층 최적화를 위해 패킷이 직렬화되는 송수신 버퍼 크기를 플랫폼 MTU에 맞추어 사전에 고정 배열(Fixed Array) 형태로 정의한다. 이를 통해 통신 중 메모리 재할당으로 인한 지연이나 단편화를 원천적으로 차단한다.

2.2 제로 카피(Zero-Copy) 지향 직렬화

내부 비행 제어기 객체인 uORB 메시지는 송신을 위해 DDS용 Payload 포맷으로 직렬화(Serialization)되어야 한다. Micro-CDR 프로토콜은 PX4의 C 구조체(struct) 메모리 레이아웃 패딩을 최적화하여 값비싼 중복 복사(Deep Copy) 없이 포인터 캐스팅 및 최소한의 바이트스왑(Byte-swap)만으로 송신 버퍼에 데이터를 패킹하는 제로 카피 지향 직렬화 로직을 채택하고 있다.

3. 처리기(CPU) 스케줄링 간섭 최소화 설계

PX4의 비행 제어 루프는 최대 400Hz (2.5ms 주기) 이상으로 동작하며, 통신 스택이 이 루프를 지연시켜서는 안 된다. Micro XRCE-DDS 클라이언트 작업(Task)은 비행 제어 루프와 격리된 아키텍처를 지닌다.

  1. 비동기식 이벤트 구동(Asynchronous Event-driven) 모델: 통신 클라이언트 쓰레드는 uORB 토픽의 업데이트 이벤트를 poll() 함수로 대기하며 휴면(Sleep) 상태를 유지한다. 새로운 센서 데이터나 제어 명령이 도착할 때만 깨어나 프로세서를 점유함으로써 CPU 유휴 시간(Idle time)을 보존한다.
  2. 저우선순위 스케줄링(Low-priority Scheduling): XRCE-DDS 송수신 스레드는 EKF2 추정기나 PID 제어기 등 핵심 제어 워크로드보다 상대적으로 낮은 RTOS 우선순위를 할당받아, 통신 병목이 비행 안정성을 침해하지 않도록 격리(Isolate)된다.

4. 논문 및 오픈소스 생태계 참조

Micro XRCE-DDS 프로토콜 구조에 관한 개념은 eProsima 사 및 OSRF의 주도 하에 “Micro XRCE-DDS: A New Definition of DDS for Resource-Constrained Environments” 등의 설계 제안서를 통해 구체화되었다. 현재 이 아키텍처는 PX4 펌웨어 메인 브랜치 내의 src/modules/uxrce_dds_client 디렉토리에 오픈소스 C 코드로 충실하게 구현되어 있으며, PX4의 시그니처 네트워킹 스택으로 공식 채택되었다.

5. 결론 및 실무적 고려사항

PX4-Autopilot에 적용된 Micro XRCE-DDS 아키텍처는 정적 메모리 모델과 비동기식 이벤트 스케줄링, 그리고 완전한 P2P 네트워킹 스택의 에이전트 위임 모델이라는 세 가지 축을 기반으로 설계되었다. 개발자들은 커스텀 QGroundControl 혹은 ROS 2 연동 모듈을 작성할 때, PX4 펌웨어 환경의 이러한 제약적(Micro) 성격을 깊이 이해하고 과도한 빈도의 대용량 토픽 퍼블리싱(예: 고해상도 이미지 전송)이 XRCE-DDS 클라이언트 스레드나 시리얼 대역폭에 과부하를 초래하지 않도록 설계 단계부터 철저히 대역폭 오토 쓰로틀링(Auto-throttling) 기법을 반영해야 한다.
