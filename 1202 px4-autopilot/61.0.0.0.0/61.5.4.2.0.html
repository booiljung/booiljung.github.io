61.5.4.2. 비순환 방향 그래프(DAG) 형태의 토픽 트리에서 상위 계층 노드명 프리픽스(Prefix) 래핑을 통한 글로벌 토픽 문자열(String) 충돌 완전 회피 검색 알고리즘

현대의 ROS2 기반 자율 무인항공기 관제 시스템은 단일 기체의 단순 데이터 통신망을 넘어, 수십 대의 PX4-Autopilot 기체(에이전트)가 동시에 데이터를 생산(Publish)하고 소비(Subscribe)하는 복잡한 데이터 분배 서비스(DDS) 네트워크 위에서 동작한다. 펌웨어 버전 1.14 이상에서 도입된 uXRCE-DDS 미들웨어는 PX4 내부의 uORB 토픽(예: fmu/out/vehicle_local_position)을 직렬화하여 ROS2의 글로벌 토픽 공간(Global Topic Space)으로 여과 없이 쏟아낸다.

이때 서로 다른 에이전트들이 생성하는 토픽의 문자열(String) 이름이 완전히 동일하다면, 중앙 관제 지상국(GCS, e.g., QGroundControl 하이브리드 ROS 노드)의 그래프 구조는 치명적인 데이터 오염(Data Collision)을 겪게 된다. 본 절에서는 ROS2가 채택한 비순환 방향 그래프(DAG, Directed Acyclic Graph) 형태의 토픽 트리 아키텍처를 분석하고, 최상위 노드명(Root Node Name)에 고유 프리픽스(Prefix)를 씌우는 네임스페이스 래핑(Namespace Wrapping) 알고리즘을 통해 글로벌 분산 환경에서 이름 충돌을 수학적으로 완전 회피하는 메커니즘을 규명한다.

1. ROS2 컴퓨테이션 그래프(Computation Graph)와 문자열 충돌의 역학

ROS2 아키텍처는 노드(Node), 토픽(Topic), 서비스(Service)가 상호 연결된 위상학적 비순환 방향 그래프(DAG)를 이룬다.

1.1 글로벌 토픽 명칭의 구조적 취약점

기본적으로 ROS2 노드가 특정 이름(.msg 또는 .idl 규격에 종속된 토픽 명칭)으로 데이터를 발행하면, 이 이름은 슬래시(/) 기호로 구분된 전역 네임스페이스 /fmu/out/vehicle_local_position에 등록된다.
만약 100대의 드론이 모두 별도의 식별자 조치 없이 MicroXRCEAgent를 통해 동일한 ROS2 도메인으로 브리지(Bridge)를 개통할 경우, GCS의 구독 노드(Subscriber Node) 입장에서는 자신이 수신하고 있는 /fmu/out/vehicle_local_position 데이터 프레임이 1호기 것인지, 50호기 것인지, 혹은 99호기 것인지 판별할 수 있는 메타데이터가 파괴된다.

이러한 글로벌 이름 충돌(Global Name Collision) 현상은 단순한 디버깅의 어려움을 넘어, A 드론의 관성 센서 위치 정보를 근거로 B 드론의 제어 루프가 갱신되는 치명적인 제어 혼선(Flight Control Interruption)을 초래한다.

2. 계층적 네임스페이스 래핑(Namespace Wrapping) 알고리즘

ROS2 네트워크 철학은 패킷 내부의 몸체(Payload)를 열어보고 발신자를 찾는 대신, 구독 및 발행의 식별자(Identifier) 자체를 트리(Tree) 구조의 고유 디렉터리로 완전히 분할하여 디자인 타임부터 충돌을 방어하도록 설계되었다.

2.1 토픽 트리 DAG의 노드 분할(Segmentation) 메커니즘

해결의 핵심은 각 드론(에이전트)이 ROS2 네트워크에 참여할 때, 자신이 발행하거나 구독하는 모든 토픽 및 서비스의 최상단 루트(Root)에 자신의 고유한 명칭(기체 식별 번호, MAV_SYS_ID 등 기반)을 **프리픽스(Prefix)로 강제 래핑(Push-down)**하는 것이다.

  • 드론 1호기 래핑: Push Namespace "/drone1"
  • 드론 2호기 래핑: Push Namespace "/drone2"

이 연산이 적용되면, 기존의 평면적인 그래프 구조는 드론 식별자를 분기점(Branch)으로 삼는 깊고 견고한 DAG 트리로 재구성된다.

f_{wrap}(T_{base}, ID) = \text{"/"} \oplus ID \oplus \text{"/"} \oplus T_{base}

(여기서 \oplus는 문자열 결합 연산자, T_{base}는 원본 fmu/out/... 토픽 명칭을 나타낸다.)

이를 통해 /fmu/out/vehicle_local_position 이라는 단일 토픽(String)은 물리적으로 완전히 격리된 독립 변수 /drone1/fmu/out/vehicle_local_position 과 /drone2/fmu/out/vehicle_local_position 로 수학적으로 분리되며, 두 문자열은 글로벌 도메인 상에서 절대 교차(Intersection)하지 않는다.

3. 프리픽스 래핑의 실무 적용 및 정규식 검색 트래픽 필터링

실제 PX4-ROS2 군집 환경을 세팅할 때, 위 알고리즘의 구현은 컴패니언 컴퓨터 상의 브리지 노드 구동 스크립트(Launch File) 단계에서 이루어진다.

3.1 ROS2 Launch 시스템을 이용한 동적 래핑

ROS2의 Python 기반 LaunchConfiguration과 PushRosNamespace 액션을 활용하여 에이전트의 구동 인자(Command line arguments)로 부여된 이름을 노드 전체 범위(Node Scope)의 디폴트 네임스페이스로 지정한다. 이로써 MicroXRCEAgent를 포함한 여러 센서 포워딩 노드들이 해당 에이전트명(예: /px4_sw12) 하위로 자동으로 말려들어간다.

3.2 GCS의 동적 토픽 정규식(Regex) 서칭 알고리즘

수백 대의 방대한 문자열 노드가 생성되었을 때, 지상 관제 시스템(GCS)은 이 래핑된 모든 토픽을 일일이 하드코딩해서 구독할 수 없다. 여기서 트리 탐색의 위력이 발휘된다.
GCS의 모니터링 노드는 특정 에이전트의 가입(Join)을 모니터링하기 위해 ROS2 Graph API 체층에서 제공하는 동적 노드 탐색기를 백그라운드 스레드로 구동하며, 지정된 정규표현식(예: ^/drone\d+/fmu/out/vehicle_local_position$)에 매칭되는 새로운 래핑 토픽이 발견될 때마다 메모리에 새로운 에이전트 상태 구조체(Struct)를 할당(Allocate)하고 해당 토픽에 대해서만 콜백(Callback)을 연결(Bind)한다.

결론적으로, 네임스페이스 프리픽스 래핑은 단순한 이름 바꾸기(Renaming) 편의성을 넘어선다. 이는 ROS2라는 오픈 데이터 버스 구조론 속에서 각 무인항공기 객체(Object)에 배타적 소유권을 부여함으로써, 분산 컴퓨팅(Distributed Computing) 네트워크 특유의 통신 변조 및 스레딩 콜전(Collision) 버그를 원자(Atomic) 수준에서 예방하는 확고한 아키텍처적 보안 기법이다.
