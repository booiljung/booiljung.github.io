61.3.2.2 시스템 권한 분리 모델에 입각한 사용자 식별자(UID) 권한 승격 및 리눅스 보안 모듈(AppArmor/SELinux)의 프로세스 자원 접근 제어 정책

본 장에서는 PX4-Autopilot과 연동되는 ROS2 컴패니언 컴퓨터(Companion Computer) 환경에서, Micro XRCE-DDS 에이전트와 같은 시스템 데몬을 안전하게 구동하기 위한 보안 모델을 심층 분석한다. 특히 최고 관리자 권한(root)의 남용을 방지하는 시스템 권한 분리(Privilege Separation) 모델과, 사용자 식별자(UID, User Identifier) 기반의 제한적 권한 승격 기법, 그리고 리눅스 보안 모듈(LSM: AppArmor 및 SELinux)을 활용한 네트워크 포트 및 직렬 통신 자원 접근 제어 정책을 구체적으로 다룬다.

1. PX4-Autopilot 컴패니언 컴퓨터의 보안 위협 모델

PX4 기반의 드론 시스템이 상용화되고 LTE/5G 망을 통한 원격 제어(BVLOS, Beyond Visual Line of Sight)가 보편화됨에 따라, 컴패니언 컴퓨터(예: Raspberry Pi, UP Board 등)의 보안 계층은 비행 안전의 핵심으로 대두되었다.

  • 권한 탈취 시나리오: 외부 네트워크와 맞닿아 있는 통신 데몬(ROS2 노드, uXRCE-DDS Agent, MAVLink Router 등) 프로세스가 침해당할 경우, 해당 프로세스가 root 권한으로 동작 중이었다면 공격자는 컴패니언 컴퓨터의 전체 제어권을 즉각 획득하게 된다. 이후 비행 제어기(Pixhawk)로 임의의 악성 명령(Kill Switch 구동, 위치 속임 등)을 전송할 수 있다.

2. 시스템 권한 분리 모델과 UID 권한 승격 최소화

전통적인 연구 개발 환경이나 Ardupilot 커뮤니티의 일부 사례에서는 편의성을 이유로 모든 GCS/통신 프로세스를 sudo 혹은 root 사용자로 실행하는 안티 패턴(Anti-pattern)이 존재했다. PX4-Autopilot의 실무 상용 배포 가이드라인은 이를 엄격히 배제하며, **최소 권한의 원칙(Principle of Least Privilege)**을 강제한다.

2.1 UID/GID (User/Group Identifier) 분할 정책

Micro XRCE-DDS 에이전트는 전용 비특권 사용자(예: uxrce, ros2) 계정으로 구동되어야 한다.

  1. 사용자 생성: 시스템에 셸(Shell) 접근 권한이 없는 데몬 전용 사용자 및 그룹을 생성한다 (useradd -r -s /bin/false uxrce).
  2. 그룹 권한 승격: 에이전트가 통신에 필요한 직렬 포트(예: /dev/ttyTHS1)에 접근할 수 있도록 해당 사용자를 dialout (또는 tty) 그룹에만 선별적으로 포함시킨다 (usermod -aG dialout uxrce).

이렇게 구성된 환경에서 에이전트는 직렬 통신 외의 파일 시스템 조작, 커널 모듈 적재 등의 치명적 행위를 원천적으로 수행할 수 없게 된다.

2.2 리눅스 Capabilities를 통한 선택적 권한 승격

간혹 에이전트가 1024번 이하의 특정 포트(Privileged Port) 바인딩이나 실시간 스케줄링(Real-time scheduling, SCHED_FIFO) 등 기존에는 root만 가능했던 커널 자원을 요구할 수 있다. 이때 전체 프로세스를 root로 띄우는 대신, 리눅스 Capabilities 스펙을 활용하여 프로세스 바이너리 자체에 핀셋형 권한만을 승격(Setcap)시킨다.

  • setcap 'cap_net_bind_service,cap_sys_nice=ep' /usr/local/bin/MicroXRCEAgent

3. 리눅스 보안 모듈(LSM)에 의한 프로세스 자원 접근 제어

UID 계층의 보호막(DAC, Discretionary Access Control)만으로는 커널의 제로데이(Zero-day) 취약점이나 사용자 계정 설정 실수에 기인한 심층 방어가 불가능할 수 있다. PX4-Autopilot 상용 시스템 관리자는 강제 접근 제어(MAC, Mandatory Access Control) 메커니즘인 AppArmor(주로 Ubuntu/Debian 계열) 또는 SELinux(RHEL/Fedora 계열) 체계를 적용한다.

3.1 AppArmor 프로파일 작성 예시

AppArmor는 각 프로세스 파일(/usr/local/bin/MicroXRCEAgent) 경로에 묶인 화이트리스트(Whitelist) 프로파일을 커널 단위에서 강제한다. 다음은 uXRCE-DDS 데몬을 위한 보안 프로파일의 의사(Pseudo) 예시이다.

# /etc/apparmor.d/usr.local.bin.MicroXRCEAgent
#include <tunables/global>

/usr/local/bin/MicroXRCEAgent {
  #include <abstractions/base>
  #include <abstractions/nameservice>

  # 실행에 필요한 동적 라이브러리 읽기 허용
  /usr/lib/** r,
  
  # 직렬 포트 및 설정 파일 접근 명시적 허용
  /dev/ttyACM[0-9]* rw,
  /dev/ttyUSB[0-9]* rw,
  /etc/uxrce_dds/agent.conf r,
  
  # 네트워크 소켓(UDP/TCP) 사용 허용
  network inet dgram,
  network inet stream,
  
  # 명시되지 않은 모든 경로 및 명령어 트리거(Trigger) 강력 차단 (Deny by default)
}

3.2 프로세스 격리의 실무적 효과

이러한 SELinux/AppArmor 정책(Policy)이 활성화(Enforcing mode)되면, 설령 통신 데몬 모듈 자체에 원격 코드 실행(RCE, Remote Code Execution) 취약점이 존재하여 공격자가 셸 코드(Shell code)를 주입했다 하더라도, 프로파일에 등재되지 않은 /bin/sh 실행 요구나 /etc/shadow 읽기 시도는 커널이 즉시 차단(EPERM)하고 로그를 남기게 된다.

4. 요약

전통적인 Ardupilot이나 단순 연구 목적의 시스템 구축에서 소홀하기 쉬운 것이 바로 운영체제 레벨의 보안 아키텍처이다. 하지만 PX4-Autopilot 생태계 내에서 UAM(도심항공교통) 및 군용 체계급의 무결성을 달성하기 위해서는, UID를 철저히 분리한 상태에서 Capabilities 기반의 제한적 승격을 적용하고, 최종적으로 리눅스 보안 모듈(LSM)의 화이트리스트 MAC 정책 샌드박스(Sandbox) 내에서 프로세스를 구동하는 강력한 데몬 격리 패러다임을 도입해야만 한다.
