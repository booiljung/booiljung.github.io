61.4.2.2. 정적 타이핑 메모리 해제 로직(C++) 및 동적 타이핑 언어의 참조 횟수 계산(Reference Counting) 기반 가비지 컬렉션(GC) 대응 바인딩 인터페이스 자동 컴파일

무인항공기 시스템(UAS)의 제어는 하드웨어 추상화 계층(HAL)과 맞닿아 있는 정적 메모리 환경(주로 C/C++)과, 상위 인공지능(AI) 혹은 군집 로직 추론을 수행하는 동적 메모리 환경(주로 Python) 간의 유기적인 상호작용을 통해 이루어진다. 두 이기종 환경이 ROS2라는 동일한 미들웨어 버스 위에서 데이터 분산 서비스(DDS)를 공유할 때, 메모리 누수(Leak) 현상을 방지하기 위해서는 서로 다른 생명주기(Lifecycle) 관리 철학을 완벽하게 이어붙이는 바인딩(Binding)이 요구된다. 본 절에서는 PX4 아키텍처 연동을 위해 생성된 언어 래퍼(Wrapper) 층이 어떻게 C++의 수동/RAII 메모리 해제 모델과 Python의 참조 카운팅 기반 가비지 컬렉터 형식을 통합하는지 고찰한다.

1. 정적 타이핑 생태계(C++)에서의 RAII 메커니즘과 메모리 안전성 보장

PX4-Autopilot의 기본 통로인 uORB부터 이와 연동되는 px4_msgs의 ROS2 C++ 노드(rclcpp)에 이르기까지, 저수준 제어 환경에서는 스택(Stack)과 힙(Heap) 메모리의 할당/해제 주기가 매우 엄격하게 관리된다.

자동 생성된 C++ 메시지 헤더는 기본적으로 초기화 시 자원 획득(RAII, Resource Acquisition Is Initialization) 패턴을 따르도록 설계된다.

  • 생성자(Constructor): 인스턴스가 스택에 할당될 때, 배열이나 하위 메시지 구조체에 대해 기본값(Default values) 또는 NaN으로 포인터 및 내용을 즉시 초기화한다.
  • 소멸자(Destructor): 동적으로 힙(Heap) 메모리를 점유하는 가변 배열 구조 요소(e.g., std::vector 확장형)를 사용할 경우, 스코프(Scope)를 벗어나면 C++ 런타임이 즉시 메모리 플러싱(Flushing)을 수행하도록 한다.

특히 고주파 제어 루프(예: IMU 센서 데이터의 400Hz 파이프라인)에서는, 객체의 생성과 파괴가 빈번히 일어날 경우 메모리 단편화(Fragmentation)가 유발되어 RTOS의 치명적인 스케줄링 딜레이를 유발시킬 수 있다. 때문에 MAVLink나 uORB에서는 가능하면 가변 크기 메시지 생성을 회피하고, 정적 크기의 직렬화 버퍼를 C 배열 패러다임으로 고수하는 경향이 매우 강하다. 만약 동적 할당이 꼭 필요한 노드라면 std::shared_ptr나 std::unique_ptr를 사용하는 스마트 포인터 기반의 객체 래핑으로 누수를 차단한다.

2. 동적 언어 바인딩(Python, rclpy)과 가비지 컬렉션(GC) 간극

Python과 상위 계층의 컴패니언 컴퓨터 프레임워크(e.g., ROS2 rclpy)의 생태계는 전혀 다른 메모리 사상을 지니고 있다. Python 프로그래머는 구조체(메시지)를 new 연산이나 malloc으로 직접 제어하지 않으며, 단지 객체 레퍼런스(Reference)만을 다룬다. Python 객체의 생존 주기는 순전히 **참조 횟수(Reference Count)**의 모니터링을 통해 결정되며, 이 수가 0에 도달할 때야 비로소 내장된 가비지 컬렉터(Garbage Collector)가 비동기적으로 메모리를 수거한다.

ROS2 IDL 파이프라인(e.g., rosidl_generator_py)은 이 간극을 메우기 위해 C-Python 바인딩 인터페이스(C Extension)를 자동으로 컴파일한다. 변환 흐름은 아래의 다이어그램과 같다.

graph LR
    A[Python ros_msg Object] -->|Set Field| B(C-Extension Setter)
    B -->|Type Check & Cast| C[Underlying C Struct]
    C -->|DDS Serialization| D((Micro XRCE-DDS Bus))
    
    D -->|DDS Deserialization| E[Underlying C Struct]
    E -->|C-Extension Getter| F(Create PyObject & INCREF)
    F --> G[Python ros_msg Object]

2.1 C-Extension 계층에서의 참조 계산(RefCount) 방어 로직

코드 제너레이터가 생성한 브릿지 모듈은 C++에서 할당된 메시지 구조체를 Python 객체(PyObject)로 반환할 때 반드시 Py_INCREF()를 호출하여 참조 카운트를 강제로 올려야 한다. 반환된 변수가 Python 메모리 공간에서 더 이상 참조되지 않고 가비지 컬렉팅될 때는, C-Extension 층에 등록된 파괴자 콜백(Destructor Callback)이 작동되어 백그라운드에 남아있던 원시 C/C++ 객체의 할당을 free()하거나 C++ 소멸자를 우회 호출하여 진정한 자원 반환을 이루어 낸다.

3. GC 개입 지연과 RTOS 관제 분리의 당위성

로봇 제어 공학적 시각에서 볼 때, Python의 동적 GC 메커니즘을 저지연(Low-Latency) 요구 계층인 PX4-Autopilot 펌웨어까지 침투시키지 않는 것은 아키텍처 설계 상 극히 합리적인 판단(Rational Choice)이다.

가비지 컬렉터가 작동하는 ‘Stop-The-World’ 페이즈(정지기)에는 수백 밀리초(ms) 정도의 스레드 중단이 발생할 수 있으며, 만약 이러한 딜레이가 자세 제어(Attitude Control)나 모터 믹서(Mixer) 루프에 영향을 준다면 무인항공기의 치명적 추락 오차로 이어진다.

PX4의 다중 언어 환경 지원 아키텍처는 이를 방지하고자, 비결정론적 시스템 동작을 보이는 동적 타이핑 및 지연된 가비지 수집 언어 계층은 철저하게 오프보드 컴패니언 컴퓨터(ROS2) 쪽에 논리적/물리적으로 격리시킨다. 즉, 통신 링크(Micro XRCE-DDS)가 양극단의 메모리 생태계를 완충하는 버퍼(Buffer) 역할을 담당하도록 설계함으로써, 하드웨어 추상화 계층에서의 나노초(ns) 단위 정밀 제어 안정성을 보존하는 것이다. 이는 Ardupilot 시스템이 단일 보드 컴퓨터와 컴패니언을 엮을 때 MAVLink 프로토콜의 하드 리밋(Hard Limit)으로 동적 변화를 억제하는 방식과 철학적 맥락을 같이하지만, ROS2-DDS 노드 구조상 더욱 세련된 데이터 객체 캐싱 모델(Caching Model)을 확보할 수 있다는 점에서 차이를 지닌다.
