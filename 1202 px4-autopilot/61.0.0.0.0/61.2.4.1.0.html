61.2.4.1 파일 시스템 디렉토리 섀도잉(Shadowing) 원리에 기반한 오버레이(Overlay) 워크스페이스의 동적 링커(Dynamic Linker) 경로(LD_LIBRARY_PATH) 덮어쓰기 로직

현대의 복잡한 로보틱스 소프트웨어 스택, 특히 PX4-Autopilot과 ROS2 미들웨어를 통합하는 환경에서는 다양한 버전의 공유 라이브러리가 혼재하게 된다. 이때 전역 시스템의 안정성을 훼손하지 않고 특정 애플리케이션에만 커스텀 라이브러리(예: 사용자 정의된 px4_msgs 또는 DDS 플러그인)를 안전하게 연결하기 위해 섀도잉(Shadowing) 기법과 동적 링커 경로 덮어쓰기(Dynamic Linker Path Overwriting) 로직이 핵심적으로 사용된다. 본 절에서는 오버레이(Overlay) 워크스페이스가 시스템의 동적 링커(Dynamic Linker)를 어떻게 조작하여 의존성을 가로채는지 심도 있게 분석한다.

1. 디렉토리 섀도잉(Shadowing)의 개념과 오버레이(Overlay) 모델

소프트웨어 시스템에서 섀도잉(Shadowing)이란, 이름이 동일한 두 개 이상의 자원(변수, 파일, 라이브러리 등)이 존재할 때 탐색 우선순위가 높은 자원이 우선적으로 선택되어 우선순위가 낮은 자원이 “가려지는(Shadowed)” 원리를 말한다.

ROS2의 colcon 빌드 시스템은 이 원리를 극대화하여 **언더레이(Underlay)**와 **오버레이(Overlay)**라는 계층적 작업 공간(Workspace) 아키텍처를 구성한다.

  • 시스템 전역 또는 ROS2 코어가 설치된 경로는 언더레이 환경이 되며, 가장 낮은 우선순위를 갖는다.
  • 개발자가 직접 컴파일한 패키지가 놓이는 오버레이 환경은 언더레이보다 탐색 우선순위가 높게 설정된다.

이러한 설계는 시스템 디렉토리를 물리적으로 수정하거나 삭제하지 않아도, 사용자 환경 변수만 조작하여 특정 라이브러리를 동적으로 대체할 수 있는 논리적 샌드박스(Sandbox)를 제공한다.

2. 동적 링커(Dynamic Linker)와 LD_LIBRARY_PATH 동작 원리

리눅스(Linux) 운영체제에서 실행 파일(ELF, Executable and Linkable Format 바이너리)은 구동 시점에 자신이 의존하는 공유 라이브러리(.so 파일)들의 메모리 주소를 해석(Resolution)해야 한다. 이 과정을 수행하는 주체가 바로 동적 링커(Dynamic Linker/Loader, 일반적으로 ld-linux.so)이다.

동적 링커는 공유 라이브러리를 탐색하기 위해 정립된 규칙과 경로 우선순위를 따른다. 가장 대표적이고 강력한 탐색 경로가 바로 환경 변수 LD_LIBRARY_PATH이다.

flowchart TD
    A[바이너리 실행 / ELF Loader 구동] --> B{RPATH / RUNPATH 존재 확인}
    B -- 없음 --> C{LD_LIBRARY_PATH 확인}
    B -- 있음 --> Z[RPATH 경로에서 라이브러리 탐색]
    C -- 존재함 --> D[LD_LIBRARY_PATH에 명시된 디렉토리 순차 탐색]
    C -- 없음 --> E[ld.so.cache (/etc/ld.so.conf) 탐색]
    D --> F{라이브러리 발견됨?}
    E --> F
    F -- Yes --> G[메모리에 라이브러리 맵핑 (mmap)]
    F -- No --> H[시스템 기본 경로 /lib, /usr/lib 탐색]
    H --> I{발견됨?}
    I -- Yes --> G
    I -- No --> J[런타임 링킹 에러 발생 (Fatal Error)]

LD_LIBRARY_PATH는 콜론(:)으로 구분된 디렉토리의 목록(List)으로 구성되며, 링커는 왼쪽에서 오른쪽으로 경로를 탐색한다. 따라서 이 문자열의 가장 왼쪽에 주입된(Prepended) 디렉토리 경로가 가장 높은 선점(Preemption) 권한을 가진다.

3. 오버레이 환경의 LD_LIBRARY_PATH 덮어쓰기 로직

사용자가 터미널에서 source install/setup.bash 명령을 실행할 때, ROS2의 셸 훅(Shell Hooks) 스크립트들은 기존의 LD_LIBRARY_PATH 값을 소거하지 않는다. 대신, 현재 오버레이 워크스페이스의 lib/ 디렉토리를 기존 문자열의 “가장 앞(앞단)“에 이어 붙인다.

# 언더레이(Base ROS2) 환경만 소싱된 상태
$ echo $LD_LIBRARY_PATH
/opt/ros/humble/opt/rviz_ogre_vendor/lib:/opt/ros/humble/lib:/usr/lib/x86_64-linux-gnu

# 오버레이(사용자 워크스페이스) 환경을 추가로 소싱할 경우 동작하는 내부 논리
# export LD_LIBRARY_PATH="/home/dev/px4_ros_ws/install/px4_msgs/lib:$LD_LIBRARY_PATH"

$ source install/setup.bash
$ echo $LD_LIBRARY_PATH
/home/dev/px4_ros_ws/install/px4_msgs/lib:/opt/ros/humble/opt/rviz_ogre_vendor/lib:/opt/ros/humble/lib:/usr/lib/x86_64-linux-gnu

결과적으로 시스템에 /opt/ros/humble/lib/libpx4_msgs.so (언더레이 라이브러리)와 /home/dev/px4_ros_ws/install/px4_msgs/lib/libpx4_msgs.so (오버레이 라이브러리)가 동시에 존재하더라도, 링커는 LD_LIBRARY_PATH의 첫 번째 요소인 /home/dev/px4_ros_ws/... 경로에서 라이브러리를 먼저 발견하게 되며 즉시 탐색을 종료한다. 이를 통해 하위 시스템의 라이브러리 변경 없이 오직 현재 셸 세션에서만 안전하게 사용자 정의(Custom) DDS 통신 타입이나 패킷 포맷을 테스트할 수 있다.

4. 실무 예제: ldd와 readelf를 활용한 동적 링크 경로 추적

PX4와 ROS2 기반의 로보틱스 소프트웨어를 개발하다 보면, “분명히 코드를 변경하고 새로 빌드했는데 변화가 없다“는 문제에 자주 직면한다. 이는 대부분 셸 환경의 LD_LIBRARY_PATH가 제대로 섀도잉되지 않아서 이전 버전의 라이브러리가 링크되었기 때문이다. 이 경우 리눅스 시스템 유틸리티를 활용하여 링커의 참조 상태를 디버깅할 수 있다.

  • ldd 유틸리티 활용: 특정 바이너리나 라이브러리가 물리적으로 어떤 파일 경로에 연결되었는지 런타임 상태를 확인한다. 
    $ ldd ~/px4_ros_ws/build/my_px4_app/my_px4_app | grep px4_msgs
    libpx4_msgs.so => /home/dev/px4_ros_ws/install/px4_msgs/lib/libpx4_msgs.so (0x00007fc123abc000)

위 출력에서 연결 경로가 `/opt/ros/...`라면, 오버레이 덮어쓰기가 정상적으로 수행되지 않은 것이다.

* **`readelf` 유틸리티 활용:** 환경 변수보다 앞서 적용될 수 있는 하드코딩된 내부 경로(`RPATH` 또는 `RUNPATH`)가 혹시 존재하는지 정적 분석(Static Analysis)을 실시한다.
  ```bash
  $ readelf -d ~/px4_ros_ws/build/my_px4_app/my_px4_app | grep PATH
   0x000000000000001d (RUNPATH)            Library runpath: [/home/dev/px4_ros_ws/install/px4_msgs/lib]

결론적으로, LD_LIBRARY_PATH를 활용한 오버레이 워크스페이스의 섀도잉 로직은 컨테이너(Container) 기술 없이도 프로세스 단위의 가벼운 라이브러리 격리 상태를 만들어주는 가장 전통적이면서도 강력한 아키텍처 패턴이다. 이를 명확히 이해해야만 PX4-ROS2의 이산 라이브러리 종속성 충돌(Dependency Collision)을 근본적으로 회피하고 디버깅할 수 있다.
