61.1.2.1 동적 메모리 할당(Dynamic Memory Allocation)에 의한 힙 단편화(Heap Fragmentation) 억제를 위한 정적 메모리 풀(Static Memory Pool) 사전 할당 모델

오픈소스 비행 제어기인 PX4-Autopilot은 자원이 극도로 제한된 마이크로컨트롤러(MCU) 상에서 실시간 운영체제(NuttX RTOS)를 기반으로 구동된다. 이러한 환경에서 ROS 2 플랫폼과의 고속 통신을 위해 도입된 Micro XRCE-DDS 레이어는 메모리 안정성을 최우선으로 확보해야 한다. 본 절에서는 통신 스택 구동 중 동적 메모리 할당(malloc, free)으로 인해 발생하는 힙 단편화(Heap Fragmentation) 문제의 심각성을 분석하고, 이를 근본적으로 억제하기 위한 Micro-CDR 및 Micro XRCE-DDS 클라이언트의 정적 메모리 풀(Static Memory Pool) 사전 할당 아키텍처를 심도 있게 다룬다.

1. RTOS 환경에서의 힙 단편화(Heap Fragmentation) 위험성

전통적인 데스크톱 환경의 통신 미들웨어는 데이터 패킷의 가변적인 크기에 대응하기 위해 런타임에 동적 메모리를 할당하고 해제하는 방식을 선호한다. 하지만 1MB 내외의 가용 SRAM만을 확보한 비행 제어 하드웨어(예: STM32F7 기반 Pixhawk)에서 이러한 동적 메모리 할당 패러다임을 도입할 경우 다음과 같은 치명적 결함이 발생한다.

  1. 메모리 파편화(Fragmentation): 크기가 서로 다른 토픽 메시지를 빈번하게 할당하고 해제하면, 사용 가능한 힙 메모리 공간이 조각나게 된다. 결국 전체 가용 여유 용량은 충분하더라도, 연속된(Contiguous) 메모리 블록을 찾지 못해 메모리 할당에 실패(OOM, Out-Of-Memory)하는 현상이 유발된다.
  2. 비결정론적 지연(Non-deterministic Latency): 커널 단에서 적합한 자유 블록을 탐색하고 병합(Coalescing)하는 알고리즘은 일정하지 않은 수행 시간(Execution Time)을 갖는다. 이는 400Hz 이상의 고정밀 실시간 제어 루프를 스케줄링하는 비행 제어기의 데드라인(Deadline)을 침해하여 제어 발산을 초래할 수 있다.

2. Micro XRCE-DDS 클라이언트의 정적 버퍼 아키텍처

위의 문제를 원천적으로 방지하기 위해 PX4 펌웨어(v1.14 이상) 내의 uxrce_dds_client 모듈은 “부팅 시점의 단발성 정적 할당(One-time Static Allocation in Boot-up)” 원칙에 기반한 메모리 풀 기법을 채택한다.

2.1 스트림 메모리 영역의 분절과 MTU 기반 크기 결정

통신 릴레이를 담당하는 직렬(Serial) 모뎀 혹은 이더넷(UDP) 계층 간의 패킷 송수신을 위해 Micro XRCE-DDS 클라이언트는 세션(Session)과 스트림(Stream) 구조체를 운용한다. 이때 가변 메모리를 사용하지 않고, 기저에 놓인 하드웨어 전송 계층의 최대 전송 단위(MTU, Maximum Transmission Unit)에 기반하여 버퍼의 크기를 컴파일 타임(Compile-time)에 상수로 결정한다.

  • Best-Effort 스트림: 손실을 허용하는 센서 데이터 전송용으로 주로 1개의 패킷을 담을 크기의 정적 버퍼가 할당된다.
  • Reliable 스트림: 상태 명령(Command)이나 임무(Mission) 데이터처럼 무결성이 요구되는 통신은 전송 이력(History) 보관을 위해 여러 개의 여유분 정적 버퍼 슬롯을 할당받는다.

2.2 Micro-CDR 레이어의 제로 카피(Zero-Copy) 직렬화

uORB 통신 토픽을 직렬화하는 과정에서도 임시 객체를 힙에 생성하지 않는다. ROS 2 메시지 포맷에 맞춘 px4_msgs의 C 구조체 인스턴스가 생성되면, 고정 할당된 송신 버퍼 메모리의 특정 오프셋(Offset)으로 포인터를 직접 캐스팅(Pointer Casting)하여 값을 채워 넣는다. 이는 추가적인 메모리 복사를 수반하지 않기 때문에 CPU 사이클 저감을 도모할 뿐만 아니라 힙 단편화 제로(Zero-fragmentation) 정책을 일관되게 유지한다.

3. 정적 메모리 매개변수 설정 및 튜닝 실무

PX4의 uxrce_dds_client 구동 시, 개발자는 환경별 가용 메모리와 통신 목표(Throughput) 간의 트레이드오프(Trade-off)를 계산해야 한다.
QGroundControl(QGC)이나 ROS 2 브릿지 연결을 전제로 시스템을 튜닝할 때 PX4 파라미터 UXRCE_DDS_MTU를 물리 계층(예: 텔레메트리의 패킷 사이즈)에 정확히 일치시켜야만 무용지물인 메모리 점유(Memory Waste)를 막을 수 있다. 높은 대역폭을 요구하는 고해상도 VIO 오도메트리 토픽을 퍼블리시해야 한다면, 버퍼 슬롯의 갯수를 무분별하게 조절하는 대신 버퍼당 페이로드 한계치를 올려 시스템 충돌 한계선 내에서 조심스럽게 정적 메모리 풀 크기를 재컴파일(Re-compile)해야 한다.

4. 참조 문헌

본 설계 원리는 ROS 2 프레임워크의 코어 미들웨어 백서인 “ROS 2 on Microcontrollers: Seamless Integration of Embedded Devices“에 자세히 소개된 정적 할당(Static Allocation) 프로파일에 기조를 두고 있다. 임베디드 코딩 표준 가이드(예: MISRA C++:2008) 또한 고신뢰성 코어에서 런타임 동적 메모리 할당의 배제를 강력하게 권고하고 있으며, PX4는 이 지침을 충실히 다르고 있다.

5. 결론

마이크로컨트롤러 기반의 항공 제어 시스템에서 통신 기능 확장을 이룩하면서도 제어 성능의 일관성을 사수하는 열쇠는 메모리의 결정론적(Deterministic) 통제에 있다. Micro XRCE-DDS 클라이언트에 깊숙이 각인된 정적 메모리 풀 사전 할당 모델은 동적인 malloc/free 체인을 완전히 제거함으로써, 극한의 진동과 라디오 간섭 속에서도 드론의 메인 두뇌가 메모리 고갈로 인한 소프트웨어 패닉에 빠지지 않도록 돕는 최후의 안전 로직으로 작동한다.
