61.5.4.1. 도메인 식별자(Domain ID) 매개변수를 이용한 인터넷 그룹 관리 프로토콜(IGMP) 스누핑(Snooping) 토폴로지 구성 및 멀티캐스트 패킷의 물리적 스위치 포트 매핑 제한

ROS2 미들웨어의 기반이 되는 데이터 배포 서비스(Data Distribution Service, DDS)는 중앙 서버 없이 노드 간 자동 탐색(Discovery)을 수행하기 위해 근본적으로 UDP 멀티캐스트(Multicast) 프로토콜에 의존한다. PX4-Autopilot 생태계(Firmware Version 1.14 이상)에서 Micro XRCE-DDS 에이전트 브리지를 거쳐 나온 uORB 토픽들은 결국 이 DDS 멀티캐스트 망에 편승하게 되는데, 멀티캐스트 역시 제대로 통제하지 않으면 로컬 서브넷 전체를 마비시키는 플러딩(Flooding)의 주범이 된다.

본 절에서는 분산 군집(Swarm) 환경에서 서로 독립적인 임무를 수행하는 편대(Squadron) 간의 간섭을 물리적으로 완벽히 차단하기 위해, DDS의 도메인 식별자(Domain ID)와 인터넷 그룹 관리 프로토콜(IGMP, Internet Group Management Protocol) 스누핑(Snooping) 기능을 결합한 지능형 네트워크 토폴로지 구성 알고리즘을 분석한다.

1. DDS Domain ID와 멀티캐스트 주소/포트 바인딩 알고리즘

ROS2 환경에서 개발자가 ROS_DOMAIN_ID 환경 변수(예: export ROS_DOMAIN_ID=5)를 설정하면, DDS 스택(Fast DDS, Cyclone DDS 기본 구현체)은 OMG(Object Management Group)의 RTPS(Real-Time Publish-Subscribe) 표준 규약에 따라 해당 도메인 전용의 멀티캐스트 포트(Port)를 동적으로 계산한다.

1.1 RTPS 표준 포트 계산식(Port Calculation Equation)

Domain ID가 D로 주어질 때 (단, 0 \le D \le 232), 기본 멀티캐스트 탐색 주소는 통상 239.255.0.1로 고정되며 제어용 포트(Port) P_{discovery}는 다음 공식에 의해 결정된다.

P_{discovery} = PB + (DG \times D) + d0

여기서,

  • PB (Port Base Number): 보통 7400
  • DG (Domain Identifier Gain): 통상 250
  • d0 (Offset for Multicast discovery): 통상 0

예를 들어, Domain ID가 0인 노드 집합은 포트 7400 번을 점유하여 멀티캐스트 패킷을 주고받으며, Domain ID가 5인 에이전트 집단은 포트 8650 번에서 패킷을 주고받는다.
이처럼 애플리케이션 계층(L7)의 파라미터가 네트워킹 전송 계층(L4)의 포트를 수십 가닥으로 완전히 분리해 냄으로써, 운영체제(Linux 커널) 수준에서 서로 다른 포트를 노리는 패킷들을 상호 폐기(Drop)하게끔 유도하는 논리적 격리의 첫 번째 관문이 완성된다.

2. L2 멀티캐스트 스톰(Multicast Storm)의 한계와 스위칭 역학

그러나, 포트가 달라도 목적지 IP가 239.255.0.1이라는 동일한 멀티캐스트 D-Class 주소를 공유한다면, 중간에 놓인 더미 허브(Unmanaged Switch)나 일반적인 Wi-Fi 공유기는 이 패킷의 MAC 주소(01:00:5E:7F:00:01)만을 보고 스위치의 모든 포트로 플러딩(Flooding)을 감행한다.

즉, 포트 번호 다변화는 목적지 드론의 CPU 인터럽트 단에서 패킷을 걷어내는 효과는 있으나, 물리적인 케이블과 스위치 패브릭(Switch Fabric)의 대역폭 점유율(Bandwidth Overhead)을 전혀 경감시키지 못한다. 수백 대의 드론이 송출하는 고주파 오도메트리 토픽들로 인해 스웜(Swarm) 중계 네트워크가 물리적으로 질식하는 현상을 막기 위해서는 매니지드 스위치(Managed Switch) 계층의 개입이 필수적이다.

3. IGMP 스누핑(Snooping)을 활용한 제약적 포트 매핑(Port Mapping) 알고리즘

물리적 네트워크 대역폭 보호를 위한 핵심 솔루션은 스위칭 장비에 탑재된 IGMP 스누핑(Snooping) 알고리즘을 활성화하는 것이다.

3.1 IGMP 스누핑의 동작 원리

IGMP 스누핑은 스위치가 라우터(Router)와 각 에이전트 사이에 교환되는 IGMP 제어 메시지(Join, Leave 메시지 등)를 몰래 엿듣고(Snoop), 어떤 스위치 핀치(Port)에 물려있는 드론이 특정 멀티캐스트 그룹(예: 239.255.0.x)에 가입되어 있는지를 포트-MAC-IP 테이블로 동적으로 학습(Learning)하는 메커니즘이다.

3.2 분산 군집 환경 맞춤형 토폴로지 튜닝 적용

PX4-ROS2 다중 에이전트 환경의 파편화를 위해, 앞선 1절의 도메인 튜닝과 IGMP를 다음과 같이 연쇄적으로 결합한다.

  1. Domain 별 멀티캐스트 주소 분리: 단순히 포트만 바꾸는 것이 아니라, DDS XML 튜닝(예: FASTRTPS_DEFAULT_PROFILES_FILE)을 통해 미들웨어가 사용하는 탐색 멀티캐스트 IP 주소 자체를 분할한다.
    • 편대 A (Domain 1): 239.255.1.1 사용
    • 편대 B (Domain 2): 239.255.2.1 사용
  2. 리눅스 커널의 IGMP Join 트리거: 편대 A에 속한 드론 컴패니언 컴퓨터에서 ROS2 노드가 구동되면, 운영체제는 네트워크 스위치로 IGMP Report (Join 239.255.1.1) 패킷을 발송한다.
  3. 스위치의 동적 MAC 매핑 갱신: IGMP 스누핑이 활성화된 매니지드 L2 스위치는 이 패킷을 가로채고, 내부 캐시 매핑 테이블을 갱신한다.
    • "포트 5번과 포트 6번에 물려있는 기기만이 편대 A(239.255.1.1)의 트래픽을 원한다."
  4. 포트 지향적 하드웨어 필터링: 편대 A의 1번 드론이 px4_msgs::msg::SensorCombined 토픽을 멀티캐스트로 쏠 때, 스위치는 전체 포트로 플러딩하지 않고 오직 테이블에 등록된 5번과 6번 포트로만 프레임을 포워딩한다. 나머지 편대 B가 물려있는 포트로는 일말의 전류조차 흐르지 않는다.
