61.4.2. 다중 프로그래밍 언어 지원을 위한 자동 코드 생성기(Code Generator) 아키텍처

무인항공기 시스템(UAS) 제어 소프트웨어는 단일 프로그래밍 언어로 작성되지 않는다. 하위 비행 제어기(PX4-Autopilot)는 주로 C 및 C++ 구문을 사용하여 실시간 운영체제(RTOS)의 제약 하에 고성능으로 구동되며, 상위 컴패니언 컴퓨터(Companion Computer)에서는 로봇 운영체제(ROS2)를 통해 C++, 파이썬(Python), 혹은 러스트(Rust) 등 다중 프로그래밍 언어로 인공지능(AI)과 컴퓨터 비전 알고리즘을 수행한다. 이처럼 다양한 계층과 언어가 혼재된 환경에서 데이터 직렬화 및 구조체 일관성을 수동으로 유지하는 것은 사실상 불가능하다. 본 절에서는 동일한 메시지 명세(.msg)에서 각 언어별 자료형 클래스와 바인딩 코드를 자율적으로 도출하는 PX4의 다중 프로그래밍 언어 지원 데이터 파이프라인과 자동 코드 생성기(Code Generator) 아키텍처를 심도 있게 고찰한다.

1. 모델 주도 아키텍처(MDA) 기반의 코드 생성 사상

PX4 시스템은 .msg 형식의 범용 메시지 템플릿을 단일 진실 공급원(Single Source of Truth)으로 취급한다. 시스템 아키텍처는 이 단일 명세 파일로부터 분산 시스템에서 요구되는 여러 언어와 플랫폼의 결과물을 자동으로 파생(Derivation)시키는 모델 주도 아키텍처(Model-Driven Architecture, MDA) 사상을 도입하고 있다.

코드 생성기의 주요 역할 파이션(Partition)은 다음과 같다.

  1. 내부 C++ 헤더 생성: PX4 펌웨어 내(uORB 버스)에서 즉각적으로 스케줄링될 struct 구조체 헤더.
  2. MAVLink 직렬화 C 코어: 텔레메트리(Telemetry) 라디오를 통한 QGroundControl(QGC) 등 지상 관제 시스템 송수신용 C 매크로 및 구조체 배열 매핑 코드.
  3. uORB \leftrightarrow DDS 마이크로브리지 IDL (C/C++): Micro XRCE-DDS 클라이언트가 ROS2 환경으로 데이터를 발행/구독(Publish/Subscribe)할 수 있도록 중개하는 OMG IDL 파일 및 해당 구조체 변환 함수(Serialization Code).
  4. 고수준 ROS2 메시지 (Python/C++): px4_msgs 및 rclpy, rclcpp를 거쳐 사용자 레벨의 코드가 읽고 쓸 수 있는 객체 지향 클래스 프레임.

이 생성기(Generator)는 파이썬(Python)의 Jinja2 또는 자체 AST(Abstract Syntax Tree) 렌더링 엔진을 사용하여 문자열 치환과 로직 분기를 수행한다.

2. 코드 제너레이터(Code Generator) 파이프라인의 구조

생성기 모듈은 빌드 시스템인 CMake에 긴밀히 통합되어 운영되며, 의존성 트리(Dependency Tree) 상에서 메시지 파일이 수정될 경우 관련된 모든 타겟 타깃 소스코드를 재구성(Rebuild)하도록 설계되어 있다. 이 과정은 컴파일 단계 이전(Pre-compile Phase)에 동적으로 이루어진다.

파이프라인의 작업 흐름은 크게 3단계로 요약된다.

    A[uORB .msg Definition] -->|Python Parser| B(Abstract Syntax Tree <br> Tokenization & Verification)
    B -->|Jinja2 Template Injection| C{Language Backend Selector}
    
    C -->|C/C++ Backend| D(uORB C struct headers<br>Firmware)
    C -->|C Backend| E(Micro XRCE-DDS Serialization<br>Firmware Client)
    C -->|IDL Backend| F(ROS2 .idl / .msg<br>px4_msgs)
    
    D --> G[NuttX RTOS Build]
    E --> G
    F --> H[Colcon Build - rclcpp/rclpy bindings]

2.1 정규식(Regex) 기반의 파싱 전처리부

파서(Parser)는 정규 표현식을 활용하여 .msg 파일 내의 자료형별 식별자(Identifier), 상수, 크기 배열 요소, 그리고 주석 메타데이터를 추출한다. 파서는 타입 호환성 테이블(Type Compatibility Table)을 참조하여 유효한 스칼라(Scalar) 데이터 타입인지 검증하며, 중첩된 커스텀 구조체를 사용할 경우 의존하는 서브 .msg 파일까지 재귀적(Recursive)으로 파싱 경로를 확장시킨다.

2.2 언어별 템플릿(Language-Specific Templates) 렌더링

구축된 AST 노드는 각 타겟 언어를 위한 Jinja2 기반 템플릿 엔진으로 넘겨진다.
예를 들어 C++용 템플릿은 uint8을 uint8_t로 바꾸고 constexpr 상수들을 선언하는 매크로를 작성한다. Python 메시지 패키징을 위한 ROS2 msg 백엔드 템플릿은 동일 필드를 파이썬의 int 스칼라와 타입 힌트(Type Hint)로 맵핑(Mapping)하는 형태의 선언문 구문으로 변환한다.

3. Ardupilot의 단일 링키지 구조론과의 철학적 대비

다중 언어를 수렴하는 구조에 있어서 PX4의 생성 파이프라인은 펌웨어 외부에 강력한 범프용(Bump) 브릿지를 두고 유연한 객체 생성을 허용하는 반면, Ardupilot 생태계는 MAVLink XML 기반의 코드 생성기(e.g., mavgen)에 집중하여 전 계층을 단일화 된 바이트 배열 스택(Byte-array Stack)으로 취급하는 철학을 갖는다.

Ardupilot의 경우에도 Python Pymavlink 모듈과 C 헤더가 자동 생성되나, ROS2로 바로 편입시켜 rclpy 형태의 메시지를 즉시 구독하기에는 추가적인 DDS 트랜스레이션(Translation) 에이전트 노드가 필요하며 자원 오버헤드가 크다.
이에 반해 PX4의 다중 언어 코드 생성기 모델은 Micro XRCE-DDS를 활용해 펌웨어 구조체와 ROS2 Python/C++ 노드를 양단(End-to-end)에서 유연하게 직결(Directly Connect)시키는 독자적 아키텍처 우위를 선점하게 된다.

결과적으로, PX4의 모델 주도 자동화 파이프라인은 복잡한 이기종 군집 드론망에서 C 레벨의 극초단파 제어루프(Control Loop)와 고수준 언어 기반의 AI 추론(Inference) 계층 간의 데이터 무결성과 시스템 확장성(Scalability)을 보장하는 핵심 인프라 스트럭처이다.
