61.4.2.1. 정규 문법(Regular Grammar) 파서를 이용한 IDL 토큰화(Tokenization) 및 컴파일러 전단부(Front-end)의 추상 구문 트리(AST) 노드 순회(Traversal) 메커니즘

PX4-Autopilot의 코드 생성(Code Generation) 파이프라인은 단순히 텍스트를 치환하는 매크로 엔진이 아니라, 그 자체로 하나의 마이크로 컴파일러(Micro Compiler) 전단부(Front-end) 역할을 수행한다. 이 파이프라인의 핵심은 커스텀 언어로 정의된 파일(.msg)을 컴퓨터가 이해할 수 있는 수학적 구조인 추상 구문 트리(Abstract Syntax Tree, AST)로 변환하는 데 있다. 본 조에서는 이러한 변환의 첫 번째 관문인 정규 문법(Regular Grammar) 파싱 구조, 그리고 생성된 AST를 횡단(Traversal)하며 타겟 코드로 번역하는 알고리즘을 소스 코드 수준에서 상세히 고찰한다.

1. 정규 표현식 기반 어휘 분석기(Lexer) 및 토큰화(Tokenization)

컴파일의 첫 단계는 일련의 문자열로 구성된 문서를 의미 있는 최소 단위인 토큰(Token)으로 분해하는 어휘 분석(Lexical Analysis)이다. PX4의 메시지 생성 스크립트 모듈(예: 파이썬 기반의 genmsg 또는 자체 파서)은 정규 표현식(Regular Expressions) 집합을 사용하여 문자열을 토큰 스트림(Token Stream)으로 변환한다.

.msg 파일에 등장하는 주요 문법 패턴은 다음 세 가지 토큰 카테고리로 분류된다.

  1. 내장 자료형(Built-in Types): uint8, float32, bool 등의 프리미티브 타입 스칼라 변수.
  2. 배열(Arrays) 및 길이 선언: [3], [10] 등 고정 크기를 명시하는 대괄호 기호.
  3. 상수 선언(Constants): = 기호 뒤에 상수가 할당되는 NUM_MAX_SENSORS=3 형태의 디렉티브.
  4. 주석 및 여백(Comments & Whitespace): # 문자로 시작하는 주석 라인은 메타데이터 처리(예: @description) 외의 경우 토큰 파이프라인에서 버려진다(Ignoring).

이러한 패턴을 판별하는 어휘 분석기는 입력 스트림에서 텍스트 라인을 읽어 들일 때마다, 매칭되는 문법 규칙을 확인하여 내부적으로 FieldType, FieldName, ArraySize, ConstantValue 등의 속성(Attribute)을 지닌 객체를 생성하게 된다.

2. 구문 분석기(Parser)와 추상 구문 트리(AST) 구축

생성된 토큰 스트림들은 다시 구문 분석(Syntax Analysis) 과정을 거쳐 논리적인 집합체로 구성된다. PX4에서는 이것이 객체 지향적(Object-Oriented) 데이터 모델, 즉 **추상 구문 트리(AST)**로 메모리에 적재된다.

만약 my_sensor.msg 파일이 다른 time_stamp.msg 타입을 필드로 가지는 복합 자료형(Composite Type) 이라면, 트리 구조 상 루트(Root) 노드는 my_sensor가 되며, 그의 자식 노드로 각 필드(Field)가 매달리게 된다. 부모-자식 의존성이 발생할 경우(예컨대 타입 내 요소가 또 다른 사용자 정의 클래스인 경우), 파서는 그 파일까지 재귀적으로(Recursively) 열어서 AST 브랜치(Branch)를 통합한다.

graph TD
    Root[Root: SensorData.msg]
    
    Root --> Field1[Field: timestamp]
    Field1 -.-> Type1(Type: uint64)
    
    Root --> Field2[Field: accelerometer]
    Field2 -.-> Type2(Type: float32)
    Field2 -.-> Prop2(Array: [3])
    
    Root --> Field3[Constant: MAX_SAMPLES]
    Field3 -.-> Value3(Value: 200)
    
    Root --> Field4[Field: status]
    Field4 -.-> Type4(Custom Type: HardwareStatus.msg)
    Type4 --> SubRoot[AST of HardwareStatus]

이 트리는 프로그램 구문의 모든 정보(의미론적 종속성 포함)를 내포하고 있으면서도, 특정 타겟 프로그래밍 언어(C++, Python 등)에 얽매이지 않는 순수한 중간 표현(Intermediate Representation) 상태이다.

3. 방문자 패턴(Visitor Pattern)을 활용한 노드 순회(Traversal) 및 코드 방출

AST가 메모리에 완벽히 구성되면, 마지막 단계로 각 노드를 방문하며 실제로 필요한 디스크 파일 단위의 결과물을 뿜어내는 ‘코드 방출(Code Emission)’ 과정이 이어진다. 이 과정은 컴파일러 이론에서의 전형적인 순회(Traversal) 알고리즘 혹은 소프트웨어 공학의 디자인 패턴인 방문자 패턴(Visitor Pattern) 아키텍처를 따른다.

ROS2 IDL 매핑이나 uORB C 구조체를 만들기 위해, Generator 스크립트는 루트 노드부터 시작하여 깊이 우선 탐색(DFS, Depth-First Search) 방식으로 트리를 횡단한다.

  1. 상수 노드 순회: 노드가 ‘Constant’ 속성을 가지면 C++ 헤더의 경우 constexpr 선언문을, Python 클래스의 경우 클래스 변수 선언(Class Variable)을 출력한다.
  2. 배열 특성 적용: 노드가 스칼라(Scalar)인지 정적 배열(Array)인지에 따라 타겟 코드의 렌더링 방식이 다르다. (예: C++에서는 float accel[3];, IDL에서는 float accel[3]; 등)
  3. 사용자 타입 해결(Resolution): 순회 중 서브 AST 루트(Sub-Root)를 만나면, 해당 타입이 이미 생성되었는지 Dependency Graph를 참조한 뒤 #include 구문을 주입한다.

이러한 정밀한 노드 순회 메커니즘을 통해, PX4 펌웨어는 고도화된 ROS2/DDS 생태계와 상호 작용할 때 가장 기초적인 블록인 ’메시지 스트럭처(Message Structure)’를 수동 코드 작성 없이 무결점(Zero-Defect)으로 생산해낸다. 이는 Ardupilot 계열의 단일 구조체 직렬화 방식(MAVLink Dictionary 파싱 위주) 대비, 분산 다중 에이전트 시스템에 최적화된 강력한 유연성을 제공한다.
