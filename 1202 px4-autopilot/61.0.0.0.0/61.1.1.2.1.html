61.1.1.2.1 멀티캐스트 도메인 내 피어 투 피어(Peer-to-Peer) 인증을 위한 유니버설 고유 식별자(UUID) 공간 분할 및 매핑 원리

PX4-Autopilot 기반의 무인항공기 군집 비행(Swarm) 시스템이나 복잡한 다중 에이전트 환경에서 ROS 2 통신을 활용할 때, 네트워크 상의 각 노드를 유일하게 식별하는 것은 데이터의 무결성과 송수신자의 신뢰성을 보장하는 핵심 기술이다. 실시간 발행-구독 프로토콜(RTPS)은 이를 위해 전역 고유 식별자(GUID, Globally Unique Identifier)를 사용한다. 본 절에서는 멀티캐스트 도메인 내에서 피어 투 피어(Peer-to-Peer) 통신 노드를 인증하고 식별하기 위한 GUID(또는 범용 고유 식별자 UUID)의 공간 분할 기법과 상호 매핑 원리에 대해 기술한다.

1. RTPS GUID 아키텍처 개요

DDS-RTPS 표준에서 각 도메인 참여자(Domain Participant)와 종단점(Endpoint, 즉 발행자나 구독자)은 네트워크 전체에서 유일한 16바이트(128비트) 길이의 GUID를 부여받는다. 이 16바이트 공간은 단일한 난수로 구성되는 범용 UUID와 달리, 라우팅과 식별의 효율성을 극대화하기 위해 두 가지 주요 공간으로 구조적으로 분할된다.

classDiagram
    class GUID_128bit {
        +byte[12] guidPrefix
        +byte[4] entityId
    }
    class GuidPrefix_96bit {
        +byte[4] hostId
        +byte[4] appId
        +byte[4] instanceId
    }
    class EntityId_32bit {
        +byte[3] entityKey
        +byte[1] entityKind
    }
    GUID_128bit *-- GuidPrefix_96bit
    GUID_128bit *-- EntityId_32bit
  1. GUID 접두사(GuidPrefix, 12바이트): 네트워크 상에서 도메인 참여자(애플리케이션 스택)를 식별하는 데 사용된다. 일반적으로 호스트(Host) 식별자, 애플리케이션 프로세스(App) 식별자, 그리고 시스템 인스턴스(Instance) 값을 조합하여 생성하므로 피어 간의 충돌(Collision)을 원천적으로 방지한다.
  2. 엔티티 식별자(EntityId, 4바이트): 단일 참여자 내부에 존재하는 구체적인 종단점(발행자 또는 구독자)을 식별한다. 마지막 1바이트는 해당 엔티티가 내장된(Built-in) 관리용 엔티티인지, 혹은 사용자 정의(User-defined) 데이터 스트림인지 구분하는 매직 넘버(Entity Kind)로 사용된다.

2. PX4/Micro XRCE-DDS에서의 GUID 매핑 원리

PX4는 연산 자원의 한계로 인해 내부에 완전한 RTPS 스택을 구현하지 않고, Micro XRCE-DDS 에이전트(ROS 2 네트워크 측)에 통신을 위임한다. 이 브릿지 아키텍처 내부에서 식별자 매핑은 다음과 같이 이루어진다.

2.1 도메인 참여자 식별 위임

PX4 클라이언트 펌웨어(v1.14 이상)에서 생성된 uORB 센서 토픽(예: SensorCombined)은 브릿지 컴포넌트인 클라이언트 애플리케이션에 의해 에이전트로 전송된다. 에이전트는 자신만의 12바이트 GuidPrefix를 할당받아 멀티캐스트 도메인 상에 ROS 2 도메인 참여자 자격으로 존재를 알린다. 즉, PX4 내부의 다수 uORB 토픽들은 네트워크 상에서 에이전트의 단일 GuidPrefix 아래에서 서빙(Serving)된다.

2.2 토픽 엔티티별 공간 분할(EntityId Mapping)

Micro XRCE-DDS는 에이전트 내부에서 수신되는 물리/가상 토픽들에 대해 동적으로 EntityId를 분할하여 매핑한다.

  • 예를 들어, PX4의 VehicleOdometry 발행 종단점과 VehicleCommand 구독 종단점은 동일한 GuidPrefix를 공유하지만 각기 다른 EntityId를 동적으로 부여받아 네트워크 상의 ROS 2 노드와 독립적으로 매칭된다.
  • 이 공간 분할 메커니즘 덕분에 GCS나 외부의 컴퓨터 비전(CV) 노드들은 어떤 데이터가 특정 드론의 어느 종단점에서 발생했는지 암호학적으로 안전하게(충돌 없이) 역추적할 수 있다.

3. 피어 투 피어 동적 인증(Peer-to-Peer Authentication) 모델

멀티캐스트 도메인(예: 동일한 UDP 서브넷 상의 239.255.0.1) 내에서 노드들이 활성화되면 단순 참여자 발견 프로토콜(SPDP)을 통해 자신의 GUID와 메타 데이터를 브로드캐스트한다.

  1. 초기 수신 및 필터링: 각 노드는 수신된 SPDP 패킷 내의 GuidPrefix를 추출하여 자신이 이미 알고 있는 피어 리스트와 비교한다.
  2. 논리적 도메인 분리: 비록 동일한 물리적 V-LAN이나 멀티캐스트 그룹에 속하더라도 DDS 도메인 ID(DDS Domain ID)와 GuidPrefix 기반의 상호 인증을 통해 인가된(Authorized) 군집 피어 데이터만 처리한다. 이 원리를 활용하면 다수 드론 군집(Swarm) 환경에서 호스트 GCS는 각 기체(Agent)별 고유 식별 공간을 분리하여 개별 드론을 식별하고 지휘할 수 있다.

4. 실전 구성 및 분석 팁

실제 군집 비행 설정 시 QGroundControl(v4.3 이상) 및 다중 PX4 인스턴스를 ROS 2와 연동할 때, 각 기체의 Micro XRCE-DDS 클라이언트/에이전트 인스턴스가 고유한 참여자 식별자를 갖도록 네임스페이스(Namespace)와 도메인 ID를 명시적으로 분리해 주어야 한다.

  • 만약 식별자 공간 분할 원리를 위배하여 여러 PX4 에이전트 호스트의 ROS 2 ROS_DOMAIN_ID가 동일하고 UUID 충돌 방지 로직이 미비할 경우, 토픽 루프백(Loop-back) 현상이나 데이터 혼선(Crosstalk)과 같은 치명적 네트워크 데드락 상태가 유발될 수 있다.

5. 요약

RTPS의 유니버설 고유 식별자(GUID) 128비트 아키텍처는 공간을 접두사(참여자)와 엔티티(종단점) 계층으로 명확히 분할함으로써, 멀티캐스트 도메인 상에서 중앙 서버의 중재 없이 피어 투 피어(P2P) 검색과 식별을 완벽히 보장한다. PX4 기반 시스템에서는 Micro XRCE-DDS 에이전트가 이 체계를 상속받아 호스트 시스템의 통신 복잡성을 혁신적으로 줄여준다.
