61.1.3.2 uORB 다중 생산자-다중 소비자(MPMC) 락 프리(Lock-Free) 원형 큐(Circular Queue) 원리와 브릿지 노드의 데이터 폴링(Polling) 주기 동기화

PX4-Autopilot의 내부 통신 아키텍처인 uORB(Micro Object Request Broker)는 고도로 병렬화된 환경에서 수많은 센서 모듈과 제어 모듈이 실시간으로 데이터를 교환할 수 있도록 설계되었다. 이 구조에서 가장 중요한 특징 중 하나는 여러 모듈이 동시에 동일한 토픽을 발행(Publish)하거나 여러 모듈이 동시에 구독(Subscribe)할 수 있는 다중 생산자-다중 소비자(MPMC: Multi-Producer Multi-Consumer) 패턴을 지원한다는 점이다.

이러한 MPMC 병렬 처리 환경에서 데이터를 안전하고 매우 빠르게 전달하기 위해 uORB는 **락 프리 원형 큐(Lock-Free Circular Queue)**를 채택하고 있다. 본 장에서는 이 로우 레벨 데이터 구조의 원리와, 외부 네트워크로 데이터를 전송하는 브릿지 노드(Micro XRCE-DDS Client)가 이 큐를 어떻게 효율적으로 가져가는지(Polling)에 대한 동기화 기법을 다룬다.

1. 뮤텍스(Mutex)의 위험성과 락 프리(Lock-Free) 큐의 도입

기존의 병렬 프로그래밍에서는 여러 스레드가 동시에 접근하는 데이터(공유 메모리)를 보호하기 위해 뮤텍스(Mutex)나 세마포어(Semaphore)와 같은 락(Lock) 메커니즘을 사용했다. 그러나 실시간 운영체제(NuttX RTOS) 기반의 비행 제어기에서는 이러한 락 기반 접근이 다음과 같은 치명적인 문제를 야기한다.

  1. 우선순위 역전(Priority Inversion): 중요도가 낮은 로깅 태스크가 큐에 락을 걸고 연산을 수행하는 동안, 중요도가 매우 높은 자세 제어기(Attitude Controller)가 큐에 접근하지 못해 비행 타이밍(Loop Time)을 놓치는(Miss) 현상이 발생한다.
  2. 블로킹 지연(Blocking Delay) 및 교착 상태(Deadlock): 인터럽트(ISR) 컨텍스트에서는 락을 획득하기 위해 대기(Sleep)할 수 없으며, 예기치 않은 타이밍 문제로 시스템 전체가 정지할 위험이 존재한다.

1.1 원자적 연산(Atomic Operation) 기반의 큐 설계

uORB는 이러한 지연을 O(1)의 확정적 시간으로 제어하기 위해, 하드웨어가 지원하는 **원자적 연산(예: Compare-And-Swap, CAS)**을 활용하여 락 없이도 안전하게 큐의 삽입(Push) 위치를 업데이트하는 락 프리 원형 큐를 구현했다.
생산자(Publisher)가 토픽을 발행할 때 메모리에 데이터를 복사하는 작업과 큐의 꼬리(Tail) 인덱스를 증가시키는 작업이 락 없이 원자적으로 수행되므로, 다른 소비자(Subscriber)들의 읽기 연산이 결코 방해받지 않는다.

2. 외부 브릿지(XRCE-DDS) 노드의 데이터 폴링(Polling) 메커니즘

비행 제어기 내부의 일반적인 제어 모듈들은 uORB 토픽이 업데이트되었을 때 이벤트 기반(Event-driven)으로 깨어나 데이터를 처리한다. 반면, 외부 컴패니언 컴퓨터(ROS 2 노드)로 데이터를 릴레이하는 **Micro XRCE-DDS 클라이언트(브릿지 노드)**는 여러 uORB 토픽들의 큐를 주기적으로 돌며 읽어가는 폴링(Polling) 방식을 띠는 경우가 많다.

2.1 큐 오버플로우(Queue Overflow)와 데이터 유실

uORB 원형 큐는 지정된 크기(예: 센서 토픽의 경우 보통 수 개 정도의 버퍼 길이)를 초과하면 가장 오래된 데이터를 덮어쓰게 된다(Ring Buffer 동작).
만약 센서가 500Hz로 데이터를 큐에 삽입하고 있는데, 브릿지 노드가 100Hz로 폴링하여 외부 네트워크로 퍼블리싱한다면 어떻게 될까?

  • 브릿지가 한 번 폴링할 때 큐에서 가장 최신 데이터만 읽어간다면, 중간에 덮어씌워진 4개의 센서 데이터는 ROS 2로 넘어가지 못하고 영구적으로 유실(Drop)된다.
  • 이는 자율 비행 알고리즘(Visual Inertial Odometry 등)의 성능 저하와 추적 오차를 급증시키는 원인이 된다.

3. 폴링 주기 동기화 및 큐 버퍼(Queue Size) 최적화 전략

uORB 브릿지 시스템에서 무결성 있는 데이터를 외부로 내보내기 위해서는 큐의 크기(Queue Size)와 브릿지의 폴링 퍼블리싱 주기(Publishing Rate) 사이의 치밀한 동기화 및 튜닝이 필수적이다.

3.1 동적 폴링 레이트 최적화

고주파수 센서 데이터(IMU 등)를 브릿징할 때, 브릿지 클라이언트의 업데이트 주기는 원본 데이터의 생산 주기에 나이퀴스트-섀넌(Nyquist-Shannon) 정리를 고려하여 여유 있게 맞추거나 그 이상의 속도로 데이터를 비워주어야 한다. PX4에서 이는 .yaml 형태의 설정 파일이나 커맨드 라인 인자를 통해 토픽별 XRCE-DDS 퍼블리싱 레이트를 구성하여 해결한다.

3.2 uORB Queue Size 조정

만약 네트워크 대역폭 한계로 브릿지의 전송 주기를 높일 수 없다면, uORB 토픽의 **동적 큐 사이즈(Dynamic Queue Size)**를 튜닝하여 브릿지가 1회 폴링 시 여러 개의 배치(Batch) 데이터를 한 번에 가져갈 수 있도록 완충 지대(Buffer)를 마련해야 한다.
실무적으로 px4_msgs 개발 시 커스텀 메시지를 추가할 때, 메시지의 *.msg 파일 안에 구조체뿐만 아니라 uORB 컴파일러가 참고할 큐 사이즈 힌트 인자(예: 큐 길이를 10 이상으로 설정)를 부여할 수 있다.

이와 같은 락 프리 원형 큐의 비차단(Non-blocking) 특성과 브릿지 노드의 정교한 폴링 동기화는, PX4 기반 비행 제어기가 한정된 CPU 사이클 내에서 초당 수천 개의 메시지를 지연 없이 컴패니언 컴퓨터로 쏘아 올릴 수 있게 해주는 가장 중추적인 시스템공학적 기반이다.
