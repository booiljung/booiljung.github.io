61.1.2.2 데이터 직렬화(Serialization) 단계의 중앙 처리 장치(CPU) 사이클 절감을 위한 바이트 정렬(Byte Alignment) 및 패딩(Padding) 비트 최소화 로직

PX4-Autopilot과 컴패니언 컴퓨터(Companion Computer, ROS 2 노드) 간의 통신은 네트워크를 통해 이루어지며, 이를 위해 내부 메모리의 데이터 구조체를 바이트 스트림(Byte Stream)으로 변환하는 데이터 직렬화(Serialization) 과정이 필수적이다. Micro XRCE-DDS 미들웨어는 객체 요청 브로커(uORB)의 메시지를 공용 데이터 표현(CDR: Common Data Representation) 표준에 맞추어 직렬화한다.

이 과정에서 임베디드 마이크로컨트롤러(MCU)의 리소스를 가장 많이 소모하는 병목 지점 중 하나가 바로 메모리 정렬(Memory Alignment) 불일치로 인한 추가 연산과 낭비되는 패딩(Padding) 비트의 전송이다. 본 장에서는 CPU 사이클을 절감하고 네트워크 대역폭을 최적화하기 위한 바이트 정렬 및 패딩 최소화 로직을 심층적으로 다룬다.

1. 바이트 정렬(Byte Alignment)과 CPU 사이클의 상관관계

대부분의 현대 프로세서(Pixhawk 시리즈에 탑재되는 ARM Cortex-M 등)는 메모리에서 데이터를 읽고 쓸 때, 데이터의 크기에 맞춘 특정 주소 배수(Alignment Boundary)에서 가장 높은 성능을 발휘한다.
예를 들어, 4바이트 크기의 uint32_t 데이터는 주소가 4의 배수인 곳에 위치해야 정렬된 접근(Aligned Access)이 이루어진다.

만약 데이터가 정렬되지 않은 주소(Unaligned Address)에 위치해 있다면 프로세서의 아키텍처에 따라 다음과 같은 성능 저하가 발생한다.

  1. 소프트웨어 에뮬레이션(Software Emulation) 오버헤드: 하드웨어가 비정렬 접근을 지원하지 않을 경우 하드볼트(HardFault) 예외가 발생하거나, 여러 번의 바이트 단위 읽기(Read) 연산과 비트 마스킹(Bit-masking) 연산을 통해 CPU가 강제로 정렬을 맞추어 데이터를 로드해야 하므로 엄청난 CPU 사이클 낭비가 초래된다.
  2. 이중 메모리 접근(Double Memory Access): 메모리 버스가 한 번에 읽을 수 있는 경계를 데이터가 걸치고 있는 경우, CPU는 두 번의 메모리 트랜잭션을 수행한 뒤 조립(Assemble) 과정을 거쳐야 한다.

Micro XRCE-DDS 클라이언트(PX4 내부)는 직렬화 과정에서 이러한 비정렬 패널티를 없애기 위해 데이터를 CDR 스트림 버퍼에 쓸 때 동적 바이트 정렬 로직을 수행한다. 즉, 직렬화 함수 내부에서 다음 쓸 포인터 위치가 현재 데이터 타입의 경계에 맞지 않으면, 맞을 때까지 강제로 버퍼 인덱스를 전진(Advance)시킨 후 데이터를 복사한다. 이 강제 전진 구간이 바로 **패딩(Padding)**이다.

2. 미정렬 구조체가 유발하는 패딩(Padding) 비트의 낭비

C/C++ 언어의 컴파일러는 메모리 접근 효율을 높이기 위해 구조체의 각 멤버 필드 사이에 보이지 않는 더미(Dummy) 바이트, 즉 패딩 비트를 자동 삽입하여 바이트 정렬을 맞춘다. 메시지 구조체의 배치에 따라 패딩의 양은 기하급수적으로 늘어날 수 있다.

아래는 비효율적으로 설계된 ROS 2 커스텀 메시지(.msg)의 예시이다.

// 비효율적인 메시지 정의 (Bad Example)
struct BadMsg {
    bool flag;         // 1 byte
    // [3 bytes Padding] 가 삽입됨
    double value1;     // 8 bytes
    uint8_t status;    // 1 byte
    // [3 bytes Padding] 가 삽입됨
    uint32_t count;    // 4 bytes
    char id;           // 1 byte
    // [7 bytes Padding] 가 구조체 끝에 삽입됨 (8의 배수 맞춤)
};

총 15 바이트의 유효 데이터가 있지만, 패딩이 13 바이트나 삽입되어 직렬화 버퍼 트랜잭션의 크기가 28 바이트로 불어난다. 이는 제한된 무선 대역폭(Telemetry)과 처리 성능을 지닌 PX4 시스템에서 심각한 오버헤드이다.

3. 메모리 구조체 재배치(Reordering)를 통한 패딩 최소화 로직

PX4 펌웨어와 ROS 2 px4_msgs 간의 메시지 정의에서는 위와 같은 낭비를 막기 위해, 데이터 크기가 가장 큰 타입부터 가장 작은 타입 순으로 내림차순 정렬하는 구조체 팩킹(Struct Packing) 또는 재배치(Reordering) 가이드라인을 따른다.

3.1 최적화된 메시지 재배치

가장 크기가 큰 8바이트 데이터(double, uint64_t)를 최상단에 배치하고, 마지막에 1바이트 데이터(bool, uint8_t)를 모아두면 중간에 삽입되는 패딩 비트를 수학적으로 제거할 수 있다.

// 최적화된 메시지 정의 (Good Example)
struct GoodMsg {
    double value1;     // 8 bytes
    uint32_t count;    // 4 bytes
    bool flag;         // 1 byte
    uint8_t status;    // 1 byte
    char id;           // 1 byte
    // [1 byte Padding] 가 구조체 끝에만 1개 삽입됨
};

위와 같이 구조체를 선언하면 동일한 유효 데이터를 운반함에도 직렬화 크기가 28 바이트에서 16/15 바이트로 대폭 감소한다.

3.2 패딩과 직렬화 연산 절감 알고리즘 메커니즘

Micro XRCE-DDS의 Fast CDR C 라이브러리 엔진은 직렬화를 수행할 때 루프를 돌며 정렬 조건문(if align…)을 검사한다.
패딩이 최소화된 데이터 구조는 직렬화 엔진 내에서 다음과 같은 최적화를 이끌어낸다.

  1. 분기 예측 성공률 증가: 포인터가 늘 이미 정렬된 상태에 놓이므로, 패딩을 추가하기 위해 포인터를 건너뛰는 조건 분기(Branch)가 실행되지 않는다. 따라서 CPU 파이프라인(Pipeline)의 캐시 적중률과 분기 예측(Branch Prediction) 성공률이 급증한다.
  2. 블록 복사(Block Copy) 활성화: 데이터가 패딩 없이 연속적으로 팩킹(Packing)되어 있는 경우, 변수들을 하나씩 memcpy() 하지 않고 메모리 블록 통째로 직렬화 버퍼에 덤프(Dump)하는 컴파일러 최적화(Loop Unrolling 및 Vectorization)가 수월해진다.

4. 실무 적용 지침 및 검증

새로운 uORB 토픽(Topic)이나 px4_msgs를 설계하는 GCS 및 자율 비행 개발자는 반드시 다음을 검증해야 한다.

  • .msg 파일을 작성할 때 변수명 중심의 논리적 순서가 아니라, Data Type 크기(8 -> 4 -> 2 -> 1 바이트) 기준으로 배열하라.
  • sizeof() 연산자를 이용해 실제 사용 메모리가 예상치와 일치하는지, offsetof() 구조체 매크로를 이용해 메모리 홀(Hole/Padding)이 없는지 단위 테스트(Unit Test)를 작성하여 CI 과정에서 자동 검출되도록 하라.

바이트 정렬을 고려한 패딩 최적화는 단일 메시지에서는 수 마이크로초 단위의 미미한 차이일 수 있으나, 초당 수백 헤르츠(Hz)로 유통되는 수십 개의 uORB 토픽 처리 관점에서는 컴패니언 컴퓨터와 비행 제어기 통신 전체의 대기 시간(Latency)과 전력 소모를 비약적으로 개선하는 가장 강력한 로우 레벨 최적화 기법이다.
