61.4.3.2. 운영체제 런타임 동적 링커(ld.so)의 가상 메모리 공간 내 심볼 재배치(Relocation) 테이블 갱신 및 외부 기호 참조(External Symbol Reference) 결정 과정

PX4-Autopilot과 ROS2의 결합 환경에서, 특히 uXRCE-DDS 에이전트(Agent)가 구동되는 컴패니언 컴퓨터(Companion Computer, 주로 Linux 기반) 측에서는 수많은 커스텀 uORB 메시지 타입에 대응하기 위하여 플러그인(Plugin) 형태의 동적 공유 라이브러리(Shared Objects, .so 파일) 기반 아키텍처를 채용한다. 새로운 메시지 토픽이 발견되었을 때, 전체 에이전트 프로세스를 재시작하지 않고도 해당 메시지의 직렬화/역직렬화(Serialization/Deserialization) 코드를 프로세스 가상 메모리에 즉각적으로 병합하는 기술은 시스템의 유연성과 실시간성 측면에서 매우 중요하다.

본 절에서는 런타임 동적 링커(Runtime Dynamic Linker, Linux의 ld.so)가 어떻게 가상 메모리 내에서 심볼 재배치(Relocation)를 수행하고 미해결 외부 기호 참조(External Symbol Reference)를 런타임에 결정(Resolution)하여, PX4 데이터 파이프라인의 핵심인 DDS 데이터 처리 객체를 인스턴스화하는지에 대한 심층적인 운영체제 수준의 메커니즘을 서술한다.

1. 동적 공유 객체(Shared Object)의 메모리 맵 및 위치 독립 코드(PIC)

일반적인 정적 링킹(Static Linking)과 달리 플러그인 라이브러리들은 컴파일 시점에 자신이 메모리의 어느 특정 주소에 적재될지 알 수 없다. 운영체제는 메모리 보호와 보안 메커니즘인 주소 공간 배열 무작위화(ASLR, Address Space Layout Randomization)를 적용하기 때문에, 함수나 변수의 절대 주소(Absolute Address)를 빌드 시점의 실행 파일(ELF) 내에 고정할 수 없다.

이를 해결하기 위해 gcc나 clang 컴파일러는 -fPIC(Position Independent Code) 옵션을 강제하여 위치 독립 코드를 생성한다.

  • PIC의 원리: 명령어 계수기(Program Counter, PC)에 상대적인 주소 지정(PC-Relative Addressing)을 사용하여 내부 루프와 조건문 분기를 처리한다.
  • 글로벌 오프셋 테이블(GOT, Global Offset Table): 라이브러리 내부에서 참조하지만 메모리 적재 시점 전까지 주소가 확정되지 않는 외부 기호들의 실제 절대 주소들을 런타임에 보관해 두는 테이블이다.

2. 런타임 동적 링커(ld.so)의 심볼 재배치(Relocation) 테이블 갱신 과정

PX4 메시지 규격(예: px4_msgs::msg::SensorCombined)을 처리하는 동적 라이브러리가 에이전트의 구동 도중 dlopen() 시스템 콜에 의해 호출되면, 운영체제의 동적 링커 ld.so가 개입하여 다음의 재배치 과정을 수행한다.

2.1 GOT 및 PLT의 동적 바인딩(Dynamic Binding)

런타임에 함수 호출의 오버헤드를 최소화하기 위해 동적 링커는 지연 바인딩(Lazy Binding) 메커니즘을 지원하는 PLT(Procedure Linkage Table)를 활용한다. 컴패니언 컴퓨터 상의 uXRCE-DDS 에이전트 워커 쓰레드(Worker Thread)가 새 메시지 파서를 호출할 때 발생하는 과정은 다음과 같다.

  1. 첫 번째 호출 시: 라이브러리 내의 코드가 외부 함수(예: ROS2 미들웨어의 메모리 할당 함수 rcutils_alloc)를 호출하면, 제어권은 PLT의 특수 스터브(Stub) 코드로 점프한다.
  2. ld.so 진입: 이 스터브는 아직 대상 주소가 확정되지 않았음을 인지하고, 스택에 기호 인덱스를 푸시한 뒤 동적 링커(_dl_runtime_resolve)를 호출한다.
  3. 심볼 탐색 및 주소 결정: 링커는 에이전트 프로세스 이미지 내에서 해당 이름(rcutils_alloc)을 가진 심볼의 위치를 해시 테이블을 통해 찾는다. 발견되면 그 메모리 주소를 추출한다.
  4. GOT 갱신 및 재귀 호출: 링커는 추출된 절대 주소를 GOT 내의 해당 엔트리에 기록(Overwriting)한다. 이후 방금 불렸던 원래 함수(PLT를 경유하지 않고 직접 GOT 참조)로 제어권을 반환한다.
sequenceDiagram

    participant P as PLT (Procedure Linkage Table)
    participant G as GOT (Global Offset Table)
    participant L as Dynamic Linker (ld.so)
    participant R as External API (e.g. ROS2 Allocator)

    C->>P: 1. call func@PLT (첫 호출)
    P->>G: 2. 간접 점프 (Initial: PLT 스터브로 롤백)
    G-->>P: 진행
    P->>L: 3. 기호 확인 및 위치 재배치 요청
    L->>L: 4. 외부 기호 테이블 탐색 (External Symbol Table)
    L->>G: 5. 탐색된 실제 주소로 GOT 엔트리 덮어쓰기 (갱신)
    L->>R: 6. 실제 함수 호출
    R-->>C: 반환

이러한 지연 바인딩은 모든 심볼을 적재 시점(Load Time)에 한 번에 매핑하는 방식(Immediate Binding, LD_BIND_NOW=1 설정)에 비해 플러그인 로딩의 초기 지연(Initial Latency)을 극적으로 감소시킨다. 수백 개의 PX4 메시지 타입 라이브러리가 존재할 수 있는 드론 관제 컴퓨터의 조건에서는 초기 부팅 시간을 단축시키는 핵심 요소가 된다.

3. 외부 기호 참조(External Symbol Reference) 결정 알고리즘의 최적화

에이전트가 dlopen을 수행할 때 RTLD_LAZY 플래그 외에도 RTLD_GLOBAL 플래그의 사용 여부가 외부 기호 참조 결정에 지대한 영향을 미친다.

  • RTLD_LOCAL (기본값): 적재된 라이브러리의 심볼은 오직 자신과 의존성 체인 내에서만 참조 가능하다. uXRCE-DDS 에이전트는 서로 다른 PX4 메시지 타입 간(예: VehicleLocalPosition과 VehicleGlobalPosition)에 심볼 이름 충돌(예: 동일한 이름의 파싱 보조 함수) 현상을 방지하기 위해 이 옵션을 적용하여 네임스페이스 격리(Namespace Isolation) 효과를 얻는다.
  • $ORIGIN RPATH 활용: PX4 펌웨어 버전업 시, 대응되는 메시지 라이브러리는 빌드 시스템에 의해 일괄 생성되어 특정 디렉토리(Workspace/install/lib)에 위치한다. 에이전트 바이너리가 동적 링커에게 라이브러리를 요청할 때, 시스템 전역 경로(/usr/lib) 대신 상대 경로 식별자인 $ORIGIN을 기반으로 한 RPATH(Run-path) 또는 RUNPATH를 ELF 헤더에 기록해둠으로써, 링커가 호환되는 버전의 라이브러리를 정확히 찾을 수 있도록 유도한다.

3.1 런타임 타입 검사와의 결합

이전 절에서 다룬 페이로드 해시 무결성 검증과 본 운영체제 수준의 심볼 재배치 메커니즘을 융합하는 최종 논리는 다음과 같다.
수신된 메시지의 해시(Hash) 식별자를 바탕으로 에이전트는 해당 해시에 맵핑된 라이브러리 파일(.so)을 동적 링커 캐시에서 찾거나 파일시스템에서 호출(dlopen)한다. 발견된 객체의 진입점(Entry Point, dlsym을 통해 획득한 팩토리 심볼 포인터)에 외부 참조를 바인딩함으로써, 메인 루프를 방해하지 않고도 미지의 센서 데이터를 파싱할 수 있는 능력을 동적으로 획득하게 된다.

이처럼 PX4-Autopilot 생태계는 MAVLink 기반 텔레메트리가 가진 타입의 경직성을 극복하고, ROS2 인터페이스의 가상 메모리 공간을 유연하게 활용하는 심볼 재배치 시스템을 통해 현대 드론 시스템에 필수적인 유연한 모듈화 및 장애 고립 아키텍처를 완성하고 있다.
