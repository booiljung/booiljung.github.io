61.5. 물리 계층 및 네트워크 라우팅 계층의 통신 채널 프로비저닝 모델링

분산 자율 비행 시스템(Distributed Autonomous Flight System)을 구성하는 PX4-Autopilot과 컴패니언 컴퓨터(예: ROS2 노드가 구동되는 Jetson, Raspberry Pi 등) 간의 통신 아키텍처는 단순한 데이터 교환을 넘어 실시간 제어 루프의 안정성을 결정짓는 가장 핵심적인 인프라이다. 이종(Heterogeneous) 하드웨어 간 결합을 위해 채택된 uXRCE-DDS 미들웨어 생태계에서, 데이터 처리를 담당하는 응용 계층(Application Layer) 프로세스가 예측 가능한 성능을 발휘하기 위해서는 하단부의 물리 계층(Physical Layer)과 네트워크 라우팅 계층(Network Routing Layer)에 대한 엄밀한 채널 프로비저닝(Channel Provisioning) 모델링이 선행되어야 한다.

본 장에서는 PX4-Autopilot의 실시간 텔레메트리 대역폭(Bandwidth) 요구사항을 수학적으로 모델링하고, UART 직렬 통신(Serial) 및 유무선 이더넷(UDP/IP) 기반 계층에서의 자원 할당 및 병목 현상을 예측하는 이론적 프레임워크를 심도 있게 다룬다.

1. 통신 채널 프로비저닝(Provisioning)의 중요성과 환경적 제약

PX4 환경에서 채널 프로비저닝이란, 기체의 자세(Attitude), 위치(Position), 센서 페이로드 등의 uORB 메시지가 ROS2 환경으로 안전하게 넘어가기 위해 요구되는 네트워크 용량을 사전에 계산하고, 병목이 발생하지 않도록 물리적 전송 속도(Baudrate)나 네트워크 토폴로지(Topology)를 할당하는 일련의 수학적 시스템 설계 과정을 의미한다.

1.1 Ardupilot 기반 통신과의 아키텍처적 차원

전통적인 Ardupilot이나 초창기 PX4 모델에서는 MAVLink 텔레메트리를 통해 57,600 bps 또는 115,200 bps 수준의 저속 직렬 통신 채널 하나에 의존하여 지상 관제 시스템(GCS)과 데이터를 송수신하였다. 이러한 구조는 제어 주기율(Control Rate)이 낮거나 단순한 웨이포인트(Waypoint) 비행에는 무리가 없었다.

그러나 현대의 PX4-Autopilot 기반의 오프보드(Offboard) 제어기나 고정밀 VIO(Visual Inertial Odometry) 퓨전 환경에서는 초당 수백 회(100Hz ~ 250Hz 이상)의 고속 센서 데이터 교환이 ROS2 노드와 실시간으로 이루어져야 한다. uXRCE-DDS 프로토콜 기반의 패킷은 MAVLink 대비 데이터 추상화 및 직렬화 과정에서 필연적인 오버헤드 구조(수 바이트의 헤더 및 해시 메타데이터)를 지니고 있기 때문에, 주먹구구식의 통신 채널 설정은 텔레메트리 큐 플러딩(Queue Flooding)과 심각한 제어 지연(Control Latency)을 초래하여 비행체의 추락을 유발할 수 있다.

따라서 이론적 최대치와 실제 가용 스루풋(Throughput) 간의 갭을 인지하고 이를 사전에 모델링하는 계층별 프로비저닝이 필수적이다.

2. 물리 계층(Physical Layer) 모델링: 직렬 통신(UART)

Pixhawk와 같은 비행 제어기(Flight Controller) 하드웨어와 단일 보드 컴퓨터(SBC) 간의 가장 보편적인 원시(Primitive) 물리 계층 연결은 범용 비동기 송수신(UART, Universal Asynchronous Receiver-Transmitter)이다.

2.1 대역폭 한계와 오버헤드 정량화

UART 통신 모델링에 있어서 전송 속도인 보드 레이트(Baudrate, R_b)는 단순히 초당 전송되는 비트 수를 나타낸다. 가장 권장되는 초고속 통신 속도인 921,600 bps를 사용할 경우, 이론적인 최대 용량은 초당 약 900 킬로비트이다. 그러나 이 수치는 순수 페이로드(Payload) 전송률을 의미하지 않는다.

UART 프로토콜 전송에는 다음의 오버헤드가 발생한다.

  • Start Bit / Stop Bit: 통상 8비트 데이터 프레임마다 1비트의 Start 플래그와 1비트의 Stop 플래그가 붙는다 (총 10비트 사용).
  • Parity Bit: 옵션에 따라 오류 검출용으로 추가(보통 PX4 통신에서는 None으로 설정하여 10-bit 구조(8N1)를 사용).
  • 하드웨어 딜레이: UART 컨트롤러의 FIFO(First In First Out) 버퍼 스위칭 및 인터럽트 서비스 루틴(ISR) 지연 등.

만약 L_{data}가 패킷 페이로드의 총 바이트 수이고, H_{dds}가 XRCE-DDS 프로토콜 자체 헤더, 프레임 오버헤드를 O_{uart}(보통 바이트당 2비트)라 할 때, 하나의 논리적 메시지를 전송하는데 걸리는 시간 T_{tx}는 다음과 같이 수학적으로 추상화할 수 있다.

T_{tx} = \frac{(L_{data} + H_{dds}) \times (8 + O_{uart})}{R_b} \quad (\text{초})

이 모델을 기반으로 오프보드 제어를 위해 200\text{Hz} 주기로 200\text{ bytes} 크기의 구조체를 전송한다고 가정하면, 초당 약 40,000\text{ bytes}가 필요하며, UART 오버헤드를 포괄하면 대략 400,000\text{ bps} 이상의 채널 대역폭이 확보되어야 함을 계산할 수 있다. 이를 만족하기 위해 921,600 bps 이상의 보드 레이트 할당(Provisioning)이 강제된다.

3. 네트워크 계층(Network Layer) 모델링: 이더넷 및 UDP/IP

PX4 기반의 다중 에이전트(Swarm) 환경이나 대용량 네트워크(WiFi, 전용 5G LTE) 모델에서는 직렬 통신의 대안으로 UDP/IP 기반의 이더넷 인터페이스 매핑이 요구된다. ROS2 DDS 자체도 본래 이더넷 멀티캐스트(Multicast) 기반으로 설계된 미들웨어이므로, 이 모델이 아키텍처적 조화도가 높다.

3.1 UDP/IP 스택 캡슐화에 따른 최대 전송 단위(MTU) 모델

네트워크 라우팅 계층 모델의 핵심 요소는 최대 전송 단위(MTU, Maximum Transmission Unit)이다. 이더넷의 표준 MTU는 1500\text{ bytes}이며, 이 안에서 여러 프로토콜 헤더가 누적 캡슐화(Encapsulation)된다.

  • MAC 헤더 + FCS: 18\text{ bytes}
  • IP 헤더: 통상 20\text{ bytes}
  • UDP 헤더: 8\text{ bytes}
  • XRCE-DDS 프레이밍 헤더: (제조사 구현에 따라 변동) 약 12 \sim 24\text{ bytes}

결과적으로 단일 패킷 안에서 순수하게 전송할 수 있는 px4_msgs 데이터 크기 한계 공간(Payload Limit)은 대략 1400\text{ bytes} 남짓으로 제한된다.
기체에 탑재된 LiDAR 포인트 클라우드 센서 데이터나 VIO 이미지 프레임 등 단일 크기가 이 한계를 초과하는 대규모 구조체가 uORB로 퍼블리싱(Publishing)되면, XRCE 에이전트 브리지 계층 또는 운영체제 네트워크 계층(IP 단편화, IP Fragmentation)에서 패킷이 쪼개지게 된다. 단편화가 자주 발생할수록 일부 파편(Fragment) 유실 시 전체 데이터 프레임을 재조립(Reassembly)하지 못하고 폐기(Drop)하게 되어 급격한 스루풋 하락 및 비선형 지연이 발생한다.

3.2 분산 네트워크 라우팅 계층 프로비저닝 메커니즘

    A[PX4 uORB 토픽] -->|대용량 데이터| B(uXRCE-DDS Agent)
    B -->|단편화 분석| C{크기가 MTU 1500 초과?}
    C -->|Yes| D[IP 단편화 및 다중 프레임 전송]
    C -->|No| E[단일 프레임 UDP 패킷]
    D --> F[네트워크 스위치/라우터]
    E --> F
    F -->|큐잉 지연 및 랜덤 패킷 로스 발생| G[ROS2 노드]
    G --> H{재조립(Reassembly) 가능?}
    H -->|일부 로스 시| I((프레임 전체 Drop))
    H -->|성공 시| J((역직렬화 및 콜백 실행))

위의 모식도처럼, 분산 환경에서는 시스템 적합성 검토를 통해 MTU 한계를 초과하는 토픽(예를 들어 이미지나 포인트클라우드는 MAVLink나 XRCE가 아닌 GStreamer 등 별도 사이드 채널을 분할 할당하는 방식)을 네트워크 모델 설계 초기에 차단해야 안정적인 프로비저닝 도출이 가능해진다.

결론적으로, PX4-Autopilot과 이종 미들웨어 간의 물리적 장벽을 허물고 고신뢰성의 자율 제어를 달성하기 위해서는, 물리 계층의 비트 오버헤드와 네트워크 라우팅 계층의 단편화 패널티를 런타임 이전에 철저히 수치 모델링하고 적정 전송 매체를 프로비저닝하는 설계 공학적 접근이 필수불가결하다.
