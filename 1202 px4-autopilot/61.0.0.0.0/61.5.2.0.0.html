61.5.2. 하드웨어 추상화 계층(HAL)을 통한 직렬 통신(Serial) 하드웨어 인터럽트 제어

안정적이고 결정론적인(Deterministic) 자율 에이전트 비행 제어를 보장하기 위해, PX4-Autopilot은 이기종(Heterogeneous) 비행 제어기 하드웨어(e.g., Pixhawk 4, 5X, 6C 등) 위에서 동일한 애플리케이션 코드를 구동할 수 있도록 정밀하게 설계된 하드웨어 추상화 계층(HAL, Hardware Abstraction Layer)을 채택하고 있다. 특히, 지상 관제 시스템(GCS) 또는 온보드 컴패니언 컴퓨터(ROS2 노드)와의 핵심 통로인 직렬 통신(Serial) 포트는 센서 수집 및 제어 명령 수신의 대동맥 매체로서, 인터럽트(Interrupt) 기반의 비동기 제어 구조가 필수적이다.

본 절에서는 단일 칩 마이크로컨트롤러(MCU) 레벨의 레지스터(Register) 조작을 추상화하여, PX4 펌웨어(Version 1.14 이상) 환경에서 어떻게 직렬 통신의 하드웨어 인터럽트를 제어하고 uORB 토픽의 저지연(Low-Latency) 송수신을 보장하는지 학술적 입장에서 분석한다.

1. 하드웨어 추상화 계층(HAL)의 역할과 아키텍처적 의의

자율 무인항공기 생태계에서는 수백 종의 서로 다른 MCU 제조사(STMicroelectronics, NXP 등)와 아키텍처(Cortex-M4, M7 등)가 혼용되고 있다. Ardupilot과 PX4-Autopilot의 핵심 경쟁력 중 하나는 이러한 하드웨어의 파편화를 극복하고, 텔레메트리(Telemetry) 라우팅 코드 등 페이로드 로직이 특정 MCU의 UART 레지스터 이름에 종속되지 않게 만드는 것이다.

PX4-Autopilot 시스템은 기본적으로 NuttX 실시간 운영체제(RTOS)의 장치 드라이버(Device Driver) 프레임워크를 HAL로 차용한다. MCU에 종속된 하위 레벨의 USART(Universal Synchronous/Asynchronous Receiver Transmitter) 인터럽트 서비스 루틴(ISR, Interrupt Service Routine)은 arch/arm/src/stm32.../ 경로 등에 캡슐화되어 있으며, 상위 계층의 MAVLink 모듈이나 uXRCE-DDS 에이전트 브리지는 POSIX(Portable Operating System Interface) 규격인 표준 open(), read(), write(), ioctl() 시스템 콜을 통해 이 직렬 포트에 균일하게 접근한다.

1.1 인터럽트 주도형(Interrupt-Driven) 입출력(I/O) 모델

수백 헤르츠(Hz) 주기로 갱신되는 비행 동역학 모델에서, 중앙 처리 장치(CPU)가 통신 포트의 데이터 수신 여부를 스핀락(Spinlock)하며 지속적으로 감시하는 폴링(Polling) 방식은 CPU 사이클의 막대한 낭비를 초래하고 스케줄링의 실시간성을 파괴한다.

이에 대응하여, 하드웨어 추상화 계층에는 다음과 같은 인터럽트 주도형 입출력 로직이 구현된다.

  1. 상향식(Bottom-Up) 수신(Rx) 인터럽트: 직렬 포트 핀에 데이터 비트가 물리적으로 도착하여 UART 하드웨어 버퍼(FIFO)에 1바이트가 온전히 조립되면, 하드웨어는 즉시 CPU에 수신 인터럽트(RXNE: RX Not Empty)를 발생시킨다. ISR은 해당 바이트를 읽어 소프트웨어 링 버퍼(Ring Buffer)에 밀어 넣고 즉시 복귀하여 메인 루프 실행 지연을 최소화한다.
  2. 하향식(Top-Down) 송신(Tx) 인터럽트: 상위 계층 모듈이 write() 시스템 콜로 다량의 페이로드(예: 직렬화된 px4_msgs)를 송명하면, HAL 드라이버는 이를 전송용 링 버퍼에 적재한다. 하드웨어 UART 송신 레지스터가 비워질 때마다 송신 인터럽트(TXE: TX Empty)가 발생하며, ISR은 링 버퍼에서 1바이트씩 꺼내어 전송 레지스터로 밀어 넣는다.

2. 인터럽트 오버헤드와 콘텍스트 스위칭(Context Switching) 상쇄 제어

NuttX RTOS 기반의 PX4 환경에서 인터럽트를 발생시키는 것은 무료가 아니다. 고속 보드 레이트(예: 921,600\text{ bps})로 동작하는 GCS 통신 라인의 경우, 인터럽트가 바이트 단위로 트리거(Trigger)될 경우 초당 약 100,000번의 하드웨어 인터럽트가 발생할 수 있으며, 이로 인한 콘텍스트 스위칭 비용은 자칫 EKF2(Extended Kalman Filter 2)나 물리 타겟 위치 제어기(Position Controller)의 CPU 할당량을 박탈하는 결과로 이어진다.

이러한 인터럽트 폭풍(Interrupt Storm)을 억제하기 위해 HAL은 다음과 같은 고도의 하드웨어 제어 기법을 노출하거나 추상화한다.

2.1 하드웨어 수신 버퍼(Hardware FIFO) 임계치 튜닝

최신 MCU(예: Cortex-M7 커널이 탑재된 Pixhawk 6C) 내부에 통합된 UART 브리지는 자체적으로 16바이트 이상의 하드웨어 FIFO를 가지고 있다. HAL 드라이버 계층에서는 수신 인터럽트 임계치(Rx FIFO Watermark)를 조절할 수 있도록 구성된다.
단일 바이트가 아닌 특정 개수(예: 8바이트 또는 14바이트)의 데이터가 하드웨어 버퍼에 모이거나 버퍼에 데이터가 머무른 채 잠시 선로가 유휴(Idle) 상태가 되었을 때(Idle Line Detection)만 복합적인 인터럽트를 발생시키도록 제어 레지스터를 세팅한다. 이는 이론상 CPU의 ISR 호출 빈도를 최대 1/8 수준으로 격감시키는 결정적 최적화 기술이다.

2.2 하드웨어 흐름 제어(Hardware Flow Control)의 자동화

페이로드가 집중되어 소프트웨어 링 버퍼(수신 버퍼)가 가득 차기 직전에 이르면, 데이터 조각화나 폐기(Drop)를 막기 위해 하드웨어 트랜시버가 논리적으로 송신 측(GCS나 Companion Computer)에 전송 중지(RTS/CTS 핀 활용)를 실시간으로 요청해야 한다.
이전 세대의 시스템에서는 소프트웨어 ISR 내에서 남은 버퍼 용량을 계산하여 수동으로 GPIO 핀 스윙 상태를 바꾸었다면, 현대의 PX4 HAL 계층 설정(e.g., termios API의 CRTSCTS 플래그 활성화)은 이를 MCU의 UART 주변 장치(Peripheral) 모듈에 전적으로 위임(Offloading)한다. 즉, 지정된 버퍼 여유 임계점에 도달하면 CPU 개입 없이 하드웨어 적으로 즉각 CTS(Clear To Send) 신호를 비활성화함으로써 인터럽트 반응 지연으로 인한 페이로드 손상을 완벽히 회피한다.

3. 결론

사용자 어플리케이션 계층(MAVSDK, ROS2 Node)에서 명령 몇 줄로 텔레메트리 데이터를 자연스럽게 읽고 쓰는 이면에는, 이처럼 운영체제의 VFS 레이어 하단에서 초미세 타이밍으로 동작하는 HAL 기반의 직렬 인터럽트 제어가 버티고 있다. 이러한 인터럽트 주도형 및 FIFO 오프로딩(Offloading) 기반의 시스템 스택 캡슐화야말로, 아두파일럿의 루프 타이머 기반 폴링 아키텍처나 전통적인 펌웨어 모델에서 진일보한 PX4-Autopilot만의 현대적 실시간 제어 철학을 대변하는 핵심 메커니즘이라 할 수 있다.
