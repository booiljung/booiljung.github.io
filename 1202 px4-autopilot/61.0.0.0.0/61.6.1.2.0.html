61.6.1.2. 동적 발행-구독 매칭 그래프(Matching Graph) 생성 알고리즘에 따른 노드 간 방향성(Directionality) 벡터 검증 및 토픽 타입 호환성 트리 분석

PX4-Autopilot(Firmware Version 1.14 이상 탑재)과 로봇 운영체제(ROS2) 컴패니언 컴퓨터가 uXRCE-DDS 브리지를 통해 결합된 오프보드(Offboard) 환경은 다수의 제어 노드와 센서 노드가 맞물린 거대한 방향성 그래프(Directed Graph) 컴파일 아키텍처를 형성한다. 이 아키텍처에서 노드(Node) 간 데이터 파이프라인이 정상적으로 개통되기 위해서는, 송신자와 수신자가 물리적 네트워크 주소뿐만 아니라 논리적인 토픽 이름(Topic Name), 토픽 타입(Topic Type, .msg IDL 구조), 그리고 통신 품질(QoS) 파라미터까지 정확히 일치하는 동적 매칭(Dynamic Matching)을 성사시켜야 한다.

본 절에서는 uXRCE-DDS 에이전트에 의해 로컬 서브넷 상에서 전개되는 파티서펀트(Participant) 탐색 메커니즘을 규명하고, 발행-구독 간 방향성 벡터(Directionality Vector)의 일관성 검증 및 추상 구문 트리(AST) 수준의 토픽 타입 호환성 붕괴 원인을 심도 있게 분석한다.

1. 퍼블리셔-서브스크라이버 동적 매칭(Matching) 메커니즘

ROS2 네트워크(DDS 기반)는 마스터 노드(Master Node) 없이 순수 P2P(Peer-to-Peer) 기반의 단순 탐색 프로토콜(SDP, Simple Discovery Protocol)을 통해 동적 매칭을 수행한다.

1.1 참가자 탐색(Participant Discovery Phase, PDP)

에이전트(ROS2 브리지)가 구동되면, 가장 먼저 239.255.0.1 기본 멀티캐스트 주소를 통해 “나는 Participant UUID [A] 이다” 라는 신원 선언 패킷(DATA(p))을 주기적으로 살포한다. 이를 수신한 GCS나 이웃 드론의 Participant들은 상호 간에 존재를 인식하고, 유니캐스트(Unicast) 세션을 통해 직접 통신할 준비를 마친다.

1.2 엔터티 탐색(Endpoint Discovery Phase, EDP)

서로의 존재를 확인한 Participant들은 자신이 보유한 데이터 쓰기 엔터티(DataWriter, 퍼블리셔)와 읽기 엔터티(DataReader, 서브스크라이버)의 세부 명세(Specification) 교환을 시작한다.
이 명세(DATA(w) / DATA(r)) 안에는 다음과 같은 3대 필수 매칭 요소가 포함되어 있다.

  1. 토픽 명칭 (Topic Name): 네임스페이스가 래핑된 정확한 문자열 (예: /drone1/fmu/in/offboard_control_mode)
  2. 토픽 타입 (Topic Type): 해시화된 데이터 구조체 명칭 (예: px4_msgs::msg::dds_::OffboardControlMode_)
  3. QoS 파라미터 규약: 요구되는/제공하는 통신 품질 (예: Best Effort, Reliable)

이 세 가지 조건이 이항 연산적으로 완벽히 교집합(Intersection)을 이룰 때 수학적인 “매칭(Matched)” 이벤트가 발생하고, 비로소 방향성을 가진 데이터 페이로드(Payload) 벡터가 열린다.

2. 노드 간 방향성 벡터(Directionality Vector) 불일치 진단

DDS 그래프에서 간선(Edge)은 데이터의 흐름을 나타내는 일방향 벡터이다. PX4 관제 파이프라인 아키텍처에서 빈번하게 발생하는 오류 패턴은 이 방향성 벡터가 누락되거나 역전되는 경우다.

2.1 토픽 스푸핑(Spoofing) 및 혼선으로 인한 이방향성(Bi-directionality) 붕괴

PX4는 철저히 fmu/in/* (입력 제어 명령) 토픽은 구독(Subscribe)만 하고, fmu/out/* (출력 텔레메트리) 토픽은 발행(Publish)만 하도록 하드코딩 되어 있다.
하지만 ROS2 GCS를 잘못 코딩하여 fmu/out/vehicle_odometry를 구독하는 대신 실수로 발행(Publish) 노드로 선언해버렸다고 가정하자.

  • PX4 (Publisher) \rightarrow fmu/out/vehicle_odometry
  • ROS2 Node (Publisher) \rightarrow fmu/out/vehicle_odometry

이 경우 양측 모두 “송신자(DataWriter)” 신분만을 가지므로, EDP 탐색 과정에서 매칭 그래프 알고리즘은 교착 상태(Unmatched)를 반환한다. 즉 물리적 네트워크가 완벽히 살아있음에도 불구하고 방향성 불일치(Type Mismatch in Direction)로 인해 어떠한 트래픽도 교환되지 않는 “조용한 단절(Silent Disconnect)” 현상이 초래된다.

3. 토픽 타입(Topic Type) 호환성 트리 검증 알고리즘

방향성과 토픽 이름이 완벽히 맞더라도, ROS2 노드가 수신한 페이로드 직렬화 바이트 패킷을 원래의 C++ 구조체로 역직렬화(Deserialization)하지 못한다면 시스템은 곧바로 크래시(Crash)를 일으키거나 쓰레기값(Garbage Data)을 제어기에 밀어넣게 된다. 이것이 토픽 타입 붕괴(Type Mismatch Breakdown) 메커니즘이다.

3.1 추상 구문 트리(AST) 레벨의 IDL 구조체 해시(Hash) 불일치

PX4의 px4_msgs 디렉토리 내의 .msg 파일들은 빌드 타임에 C++ 헤더 및 DDS IDL(Interface Definition Language) 파일로 컴파일된다. 이때 구조체 내부의 필드 순서, 자료형(e.g., float32 vs float64), 혹은 구조체 이름 중 단 하나라도 변형되면, 해당 타입의 근본 구조를 나타내는 원본 256비트 해시 고유값(Type Identifier Hash)이 완전히 다르게 생성된다.

  • PX4 펌웨어 버전 불일치: PX4 사이드 커스텀 펌웨어는 필드 A, B, C를 가지는데, 지상 ROS2 컴퓨터의 소스 작업공간(Workspace)은 구버전 필드 A, B만을 가지는 경우.
  • EDP(Endpoint Discovery Phase) 과정에서 에이전트와 클라이언트는 자신의 토픽 타입을 단순한 문자열(px4_msgs/msg/VehicleLocalPosition)뿐만 아니라 빌드 타임의 IDL 해시 핑거프린트(Fingerprint)로 상호 검증한다. 해시가 1비트라도 엇갈리면 매칭 그래프는 즉각 Type Mismatch 오류를 내고 브리지 생성을 거부(Reject)한다.

3.2 런타임 진단(Diagnostics) 파라미터 활용 기법

이러한 동적 매칭 실패는 컴파일 에러가 아닌 런타임 네트워크 묵살(Drop)로 나타나므로, 개발자는 ros2 topic info -v /fmu/in/offboard_control_mode 와 같은 진단 명령어를 통해 현재 Publishers 수와 Subscribers 간의 노드 매칭 그래프 엔터티 불균형을 추적해야 한다.

결과적으로, 분산 시스템 아키텍처에서 발행-구독 간의 이항 관계는 이름, 방향성, 타입 핑거프린트, QoS 파라미터가 유기적으로 교차 검증되는 4차원적 수학 행렬의 완결 조건 하에서만 데이터 브리지를 승인(Handshake Validation)하는 고밀도 방어 기제임을 엔지니어는 반드시 인지해야 한다.
