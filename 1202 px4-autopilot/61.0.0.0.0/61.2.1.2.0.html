61.2.1.2. 가상 메모리 페이징(Paging) 스와핑(Swapping) 현상 억제를 위한 프로세스 주소 공간 물리 메모리 강제 잠금(mlockall) 시스템 호출 메커니즘

1. 개요

호스트 리눅스 커널의 스케줄러를 PREEMPT_RT 패치 등을 통해 최적화하여 선점 지연(Preemption Latency)을 극복하더라도, 메모리 관리 하위 시스템(Memory Management Subsystem)에서 발생하는 지연은 여전히 실시간 시스템의 치명적인 문제로 남는다. ROS2 및 Micro XRCE-DDS 클라이언트 애플리케이션이 실행되는 과정에서 가상 메모리 스와핑(Swapping)이 발생할 경우, 디스크 I/O에 의한 심각한 지연이 유발되어 PX4-Autopilot의 오프보드 제어 주기(예: 100Hz~1kHz)를 만족시킬 수 없게 된다. 본 절에서는 이러한 페이징(Paging) 현상의 위험성을 분석하고, POSIX 표준 시스템 호출인 mlockall() 메커니즘을 통하여 가상 주소 공간을 물리 메모리에 강제로 잠금(Lock) 처리함으로써 메모리 접근 지연을 확정적(Deterministic) 한계 내로 억제하는 공학적 방법론을 다룬다.

2. 가상 메모리 스와핑과 메이저 페이지 폴트(Major Page Fault)의 위협

현대의 범용 운영체제는 가용한 물리 메모리(RAM)보다 더 큰 주소 공간을 제공하기 위해 페이징 기법을 사용하며, 사용 빈도가 낮은 페이지(Page)를 보조기억장치(스왑 영억, Swap Area)로 이동시킨다(Swap-out).
실시간 제어 시스템 관점에서 스와핑의 치명적인 단점은 다음과 같다.

  1. 메이저 페이지 폴트(Major Page Fault) 발생: ROS2 제어 노드가 일시적으로 유휴(Idle) 상태에 있다가 센서 인터럽트에 의해 깨어날 때, 그 사이 스왑 아웃된 메모리에 접근하려 하면 CPU는 메이저 페이지 폴트 예외(Exception)를 발생시킨다.
  2. 디스크 I/O 지연으로 인한 데드라인 누락(Deadline Miss): 스왑 영역에서 물리 메모리로 데이터를 다시 적재(Swap-in)하려면 느린 스토리지(SSD/HDD) 장치 컨트롤러의 인터럽트를 거쳐야 한다. 메모리 접근 시간(수십 ns 수준)이 디스크 대기 시간(수 ms 수준)으로 수십만 배 이상 급증하게 되며, 이는 드론 제어 제세동(Control Instability) 및 MAVLink 프로토콜 타임아웃을 유발한다.

3. 프로세스 공간 강제 잠금 메커니즘 (mlockall)

지상 관제 시스템(GCS) 애플리케이션이나 동반 컴퓨터의 ROS2 퍼블리셔(Publisher)가 스와핑의 희생양이 되는 것을 방지하기 위해, 프로세스 구동 초기화 단계에서 메모리를 잠가야 한다. POSIX mlockall(int flags) 시스템 호출은 호출한 프로세스의 모든 가상 메모리 페이지를 물리 메모리에 핀(Pin) 고정하여 커널의 스왑 데몬(kswapd)이 이를 스왑 아웃하는 것을 원천 차단한다.

3.1 매개변수 플래그 분석

mlockall() 함수는 주로 다음과 같은 두 가지 비트 매스크(Bitmask) 플래그를 인자로 받는다.

  • MCL_CURRENT: 현재 프로세스 주소 공간에 매핑된 모든 페이지(코드, 데이터, 힙, 스택, 공유 라이브러리 영역)를 즉시 잠근다.
  • MCL_FUTURE: 호출 시점 이후 프로세스가 동적으로 할당받을(예: malloc(), mmap()) 모든 새로운 메모리 페이지에도 잠금 속성을 강제한다.

3.2 메모리 한도 설정 (ulimit -l) 과 권한 문제

물리 메모리를 강제 점유하는 행위는 악의적인 프로세스에 의한 시스템 서비스 거부 상태(OOM, Out of Memory)를 유발할 수 있으므로, 기본적으로 리눅스는 일반 사용자 계정이 고정할 수 있는 메모리 양을 엄격히 제한(예: 기본 64KB)한다.
따라서 ROS2 제어 노드를 구동하기 전, /etc/security/limits.conf 파일을 수정하여 해당 사용자의 memlock 한도를 증가시키거나 무제한(unlimited)으로 설정해야 한다.

# /etc/security/limits.conf 설정 예시
# ROS2를 구동하는 'px4-user'에게 무제한 물리 메모리 잠금 권한 부여
px4-user    hard    memlock    unlimited
px4-user    soft    memlock    unlimited

3.3 C++ 및 ROS2 애플리케이션 내 적용 예제

ROS2 애플리케이션(C++) 작성 시, 센서 데이터를 처리하거나 모터 PWM 제어 신호를 퍼블리시하기 직전, main() 함수의 최상단에서 다음과 같이 잠금 코드를 삽입하는 것이 권장된다.

#include <sys/mman.h>
#include <iostream>
#include <cstring>
#include "rclcpp/rclcpp.hpp"

// 실시간 권한을 위한 메모리 잠금 함수
void lock_memory() {
    // 현재 및 향후 할당될 모든 메모리를 물리적 RAM에 고정
    if (mlockall(MCL_CURRENT | MCL_FUTURE) == -1) {
        std::cerr << "[Warning] 메모리 고정 실패: " << std::strerror(errno) << std::endl;
        std::cerr << "지연(Latency)이 발생할 수 있습니다. ulimit -l 설정을 확인하십시오." << std::endl;
    } else {
        std::cout << "[Info] 메모리 페이징 방지를 위한 프로세스 공간 잠금 처리 완료." << std::endl;
    }
}

int main(int argc, char **argv) {
    lock_memory();
    
    // 이후 ROS2 노드 초기화 및 실행 로직
    rclcpp::init(argc, argv);
    auto node = rclcpp::Node::make_shared("px4_realtime_control_node");
    rclcpp::spin(node);
    rclcpp::shutdown();
    return 0;
}

4. 정리

위와 같은 mlockall 호출을 통해 커널의 스케줄러 최적화(PREEMPT_RT)로 확보한 CPU 자원 선점 능력과 맞물려, 메모리 액세스 타임(Memory Access Time)까지 상수를 보장받게 된다. 따라서 페이지 폴트에 기인하는 불필요한 시스템 호출과 I/O 블로킹을 사전에 제거함으로써, PX4-Autopilot과 이종 미들웨어 환경 간의 극도로 짧은 통신 주기에서도 단 한 건의 메시지 데드라인 위반 없이 데이터를 안정적으로 교환할 수 있는 정밀 제어 시스템 아키텍처가 완성된다.
