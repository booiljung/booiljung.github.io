61.7.2.1. 순환 중복 검사(CRC) 다항식 나눗셈의 모듈로-2(Modulo-2) 연산 실패 시, 손상된 버퍼 블록을 즉시 폐기하고 슬라이딩 윈도우(Sliding Window) 상태를 초기화하는 플러싱(Flushing) 복구 로직

PX4-Autopilot(Firmware Version 1.14 이상)이 GCS 혹은 ROS2 컴패니언 컴퓨터와 교환하는 장거리 원격 측정(Telemetry) 데이터는, 가혹한 점대점(Point-to-Point) 무선 전파 환경을 횡단하며 필연적으로 비트 오류(Bit Error)에 노출된다. 프로토콜 계층(Protocol Layer, 예: MAVLink 2.0 또는 uXRCE-DDS)은 이러한 노이즈의 위협으로부터 자율 비행 데이터의 무결성을 사수하기 위해, 강력한 에러 검출 알고리즘인 순환 중복 검사(CRC, Cyclic Redundancy Check)를 페이로드(Payload) 검증의 최전선 구역에 배치한다.

본 절에서는 수신단 파서(Parser)가 프레임을 조립하는 과정에서 CRC 다항식 나눗셈의 모듈로-2(Modulo-2) 연산이 불일치를 일으켰을 때, 즉각적으로 오염된 데이터 블록을 폐기(Drop)하고 무너진 슬라이딩 윈도우(Sliding Window) 전송 상태 머신을 깨끗하게 재동기화(Resynchronization)시키는 플러싱(Flushing) 복구 아키텍처를 수학적, 시스템적 관점에서 상세히 기술한다.

1. CRC 모듈로-2 다항식 연산과 데이터 오염 감지 메커니즘

CRC는 단순한 해시(Hash)나 체크섬(Checksum, 단순 덧셈) 방식을 넘어, 갈루아 필드(Galois Field, GF(2)) 상의 다항식(Polynomial) 나눗셈 원리를 이용해 단일 비트 오류뿐만 아니라 연쇄적인 버스트 에러(Burst Error)까지 탁월하게 잡아내는 감별기이다.

1.1 수신단의 CRC 검증 수학적 모델

송신단은 원본 메시지 다항식 M(x) 에 특정 생성 다항식(Generator Polynomial) G(x) 를 기반으로 모듈로-2 나눗셈을 수행하여, 그 나머지인 R(x) 를 프레임 꼬리(Tail)에 CRC 체크섬(통상 16비트, X.25 규격)으로 덧붙여 전송한다.

수신된 프레임을 꼬리까지 모두 포함한 거대 다항식을 T(x) 라고 할 때, 수신 파서(Parser)는 도달한 전체 비트 스트림 T(x) 를 다시 동일한 G(x) 로 나누는 모듈로-2(XOR) 연산을 수행한다.

T(x) = M(x) \cdot x^n + R(x)
\text{Check} = T(x) \pmod{G(x)}

  • 성공(매칭): 전송 과정에서 에러가 단 한 비트도 없었다면, 이 나눗셈의 나머지(Remainder)는 완벽하게 0 이 된다. (즉, \text{Check} == 0)
  • 실패(손상 감지): 공중의 전자기적 간섭(EMI)으로 인해 T(x) 에 노이즈 다항식 E(x) 가 더해졌다면, T'(x) \pmod{G(x)} \neq 0 이 도출되어 파서는 메시지가 오염되었음을 즉각적으로 수학적 증빙과 함께 확정 짓는다.

2. 손상 블록 폐기 및 슬라이딩 윈도우 플러싱(Flushing) 로직

CRC 연산이 0이 아닌 값을 뱉어내는 순간, PX4 수신 로직은 해당 메시지가 단순 텔레메트리이건, 기체를 무장 해제(Disarm) 시키는 치명적 제어 명령(Control Command)이건 일말의 타협 없이 쓰레기통으로 던져버린다(Hard Drop). 그러나 진정한 문제는 떨어진 한 개의 패킷이 아니라, 이로 인해 연쇄적으로 무너지는 흐름 제어(Flow Control) 상태 머신이다.

2.1 신뢰성 있는 채널(Reliable Channel)에서의 시퀀스 붕괴

uXRCE-DDS의 Reliable 프로파일이나 MAVLink의 패킷 재전송(Retransmission) 스택은 통상적으로 슬라이딩 윈도우(Sliding Window) 기법을 사용한다.
수신단은 시퀀스 번호(Sequence Number)를 윈도우에 쌓아두고 있는데, 중간에 CRC 에러로 Seq #45가 날아가 버리면 Seq #45 이후에 도착하는 Seq #46, 47 등은 물리적으로 온전하더라도 인-오더(In-order) 보장 원칙에 의해 애플리케이션 계층(FCU 로직)으로 올라가지 못한 채 펜딩(Pending) 큐에 기약 없이 적체된다.

2.2 플러싱(Flushing) 및 재동기화(Resynchronization) 아키텍처

이러한 교착(Deadlock)을 타파하기 위해, 수신 파서(Parser)는 CRC 에러를 감지한 즉시 미세 튜닝된 플러싱(Flushing) 복구 로직을 트리거(Trigger)한다.

stateDiagram-v2
    [*] --> WAIT_FOR_MAGIC_BYTE : Parser Idle
    WAIT_FOR_MAGIC_BYTE --> PARSING_PAYLOAD : 0xFD 획득
    PARSING_PAYLOAD --> CRC_CHECK : 페이로드 수신 완료
    
    CRC_CHECK --> PARSING_PAYLOAD : Modulo-2 == 0 (Valid, Push to App)
    
    CRC_CHECK --> FLUSHING_RECOVERY : Modulo-2 != 0 (Corrupted)
    
    state FLUSHING_RECOVERY {
        direction LR
        A[Drop Current \n Buffer Block] --> B[Clear Partial \n Rx State]
        B --> C[Send NACK \n (if Reliable QoS)]
        C --> D[Reset Parser \n Pointer Index]
    }
    
    FLUSHING_RECOVERY --> WAIT_FOR_MAGIC_BYTE : 재동기화 대기
  1. 오염 버퍼 즉각 폐기(Hard Drop): 수신 버퍼 링(Ring)에 쌓여있던 현재 프레임 구간(Header ~ CRC Tail)의 인덱스를 무효화(Invalidate) 처리하여 메모리에서 논리적으로 소거한다.
  2. 파서 상태 강제 초기화(Parser Reset): 파서의 상태 머신을 강제로 IDLE (혹은 WAIT_FOR_MAGIC_BYTE) 상태로 끌어내려, 이전에 조립 중이던 모든 맥락(Context)을 백지화한다.
  3. NACK(Negative ACK) 전송 및 슬라이딩 윈도우 조정: 신뢰성 프로파일 하에서는 즉시 송신단에 “패킷 #45가 손상되었으니 여기서부터 다시 윈도우를 동기화하여 쏴라“는 NACK 피드백을 신속히 날린다.
  4. Resynchronization(재동기화): 수신단은 버퍼에서 다음 0xFD (또는 프로토콜 헤더 매직 바이트)를 바이트 단위로 샅샅이 검색하며(Byte-hunting), 성공하는 순간 다시 온전한 슬라이딩 윈도우 수신 상태 지위를 스스로 회복(Self-heal)한다.

결론적으로, CRC 모듈로-2 연산 실패로 촉발되는 플러싱 로직은 단순한 에러 검출 도구가 아니다. 이는 노이즈로 떡칠된 바이트 스트림 구덩이 속에서, 오염된 명령이 비행 제어기의 심장부로 파고드는 것을 결사적으로 막아내는 동시에 프로토콜의 연속성을 찰나의 순간에 복구해 내는, 극도로 정교한 면역계(Immune System)의 발동이라 정의할 수 있다.
