61.1.3 비행 제어기 내부 객체 요청 브로커(uORB)와 외부 네트워크 브릿지 결합

초기 무인항공기 개발 생태계에서는 비행 제어기(Flight Controller, FC)와 지상 관제 시스템(GCS) 사이의 통신 혹은 외부 컴패니언 컴퓨터(Companion Computer)와의 연동이 주로 MAVLink 프로토콜을 통해 이루어졌다. 하지만 MAVLink는 단순한 직렬 통신 구조에 최적화되어 있어, 실시간으로 방대한 센서 데이터와 제어 명령을 다수의 노드(ROS 2 등)와 복잡하게 주고받기에는 대역폭과 확정성 측면에서 한계를 드러냈다.

이를 극복하기 위해 최신 PX4-Autopilot 아키텍처에서는 비행 제어기 내부의 핵심 미들웨어인 **객체 요청 브로커(uORB: Micro Object Request Broker)**를 외부 네트워크와 직접적으로 연결하는 브릿지 결합(Bridge Integration) 기술이 전면 도입되었다. 본 장에서는 PX4 내부의 uORB 메커니즘과 외부 분산 통신망을 매끄럽게 결합하는 구조적 원리에 대해 자세히 다룬다.

1. 비행 제어기 내부 데이터 통신의 척추: uORB 개요

PX4는 NuttX Real-Time Operating System (RTOS) 환경 위에서 수많은 독립적인 모듈(센서 드라이버, 자세 제어기, 위치 추정기 등)이 스레드 또는 태스크 형태로 동시에 구동되는 마이크로커널(Microkernel) 유사 아키텍처를 가진다.

**uORB(Micro Object Request Broker)**는 이 수많은 내부 모듈들이 서로 데이터를 공유 채널로서, 발행-구독(Publish-Subscribe) 패턴을 기반으로 하는 비동기적(Asynchronous) 메시징 시스템이다.

  • 공유 메모리 기반: uORB 메시지는 RAM의 중앙 데이터 구조체에 기록되며, 메모리 복사를 최소화하여 데이터를 교환한다.
  • 주소 지정이 없는(Data-centric) 통신: 모듈들은 누가 데이터를 생성하고 소비하는지 알 필요 없이, 오직 토픽(Topic) 이름(예: sensor_combined, vehicle_local_position)만을 바탕으로 독립적으로 작동한다.

2. 외부 네트워크 브릿지의 필요성과 패러다임 변화

uORB는 단일 칩(MCU) 내부의 메모리 통신에는 완벽하게 작동하지만, 라즈베리 파이(Raspberry Pi)나 젯슨 나노(NVIDIA Jetson)와 같은 외부 컴패니언 컴퓨터에 구동되는 ROS 2 생태계와 데이터를 투명하게 주고받을 수는 없다.

이 단절을 해소하기 위해 만들어진 아키텍처가 “미들웨어 브릿지(Middleware Bridge)“이다. 과거 PX4 v1.13 이전 버전에서는 microRTPS_bridge 앱이 이 역할을 수행했으나, 현재 최신 PX4 펌웨어에서는 eProsima의 Micro XRCE-DDS (eXtremely Resource Constrained Environments DDS) 클라이언트 미들웨어를 자체 브릿지로 채택하고 있다. 브릿지의 결합은 다음과 같은 패러다임의 변화를 가져왔다.

  1. 상태 동기화의 투명성: 외부의 ROS 2 노드가 마치 PX4 내부의 uORB 모듈 중 하나인 것처럼, 지연을 최소화한 상태로 로컬 uORB 토픽을 발행하고 구독할 수 있다.
  2. MAVLink 종속성 탈피: MAVLink 메시지 파싱 연산을 거치지 않고, px4_msgs라는 C++ 구조체를 CDR 직렬화를 통해 외부로 직접 쏘아주는 데이터 하이웨이가 구축되었다.

3. 브릿지 결합 아키텍처 구조 (uORB to ROS 2 DDS)

PX4 내부 uORB 네트워크가 외부의 데이터 분배 서비스(DDS: Data Distribution Service) 네트워크로 확장되는 아키텍처는 크게 세 영역으로 맵핑된다.

3.1 비행 제어기 영역: Micro XRCE-DDS Client

PX4 내부에서 하나의 백그라운드 태스크(uxrce_dds_client)로 동작한다. 이 클라이언트는 사용자가 지정한 설정 파일(.yaml 또는 소스코드 내 구성)을 읽어들여, 외부로 내보낼 로컬 uORB 토픽을 **구독(Subscribe)**하고, 반대로 외부에서 들어와야 할 토픽을 로컬 uORB 공간에 **발행(Publish)**한다. 이때 uORB의 C 구조체 객체는 CDR 형식의 바이트 스트림으로 패킹된다.

3.2 직렬/네트워크 채널 영역

클라이언트와 에이전트를 연결하는 물리적 매개체다. 전통적으로는 UART 연결을 사용했으나, 현대에는 이더넷(Ethernet) 포트를 내장한 비행제어기(예: Pixhawk 6X)가 보급됨에 따라 높은 대역폭을 보장하는 UDP 포트가 주로 활용된다.

3.3 컴패니언 컴퓨터 영역: Micro XRCE-DDS Agent

Linux 기반의 환경에서 백그라운드 데몬으로 수행되는 에이전트는, 비행 제어기로부터 받은 제한된 직렬화 데이터 스트림의 압축을 풀고 완전한 형태의 ROS 2 DDS 퍼블리셔/서브스크라이버 객체로 변환시켜 광역 ROS 2 네트워크 생태계(Global Topic Space)로 뿌려준다. 결과적으로, 내부의 sensor_accel 토픽은 외부 ROS 2 네트워크에서 /fmu/out/sensor_accel 이라는 토픽으로 완벽히 매핑되어 관측된다.

4. 로봇 제어에서의 기술적 시사점

과거 MAVLink를 사용할 당시에는 컴퓨터 비전(Computer Vision) 기반의 오도메트리(VIO) 정보나 조이스틱 회피 제어 명령 등을 비행 제어기로 전달하려면, MAVLink의 규격화된 메시지로 변환하는 변환 노드(Translation Node, 예: MAVROS)가 반드시 중간에 개입해야 했다.

하지만 uORB와 브릿지가 직접 결합된 현재의 구조에서는 데이터 구조 변환의 오버헤드가 사라진다. 설계자는 단순히 msg 폴더에 원하는 데이터 타입 방식을 정의하기만 하면, 빌드 타임에 uORB 구조체와 ROS 2 메시지 타입 체계가 1:1로 자동 생성(Code Generation)된다. 이로 인해 임베디드 펌웨어 개발자(C/C++)와 자율주행 알고리즘 개발자(Python/C++) 간의 협업 장벽이 허물어지고, 고도의 실시간 군집 비행(Swarm) 스택을 손쉽게 구축할 수 있는 소프트웨어 생태계가 마련되었다.
