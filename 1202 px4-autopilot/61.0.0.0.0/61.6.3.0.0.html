61.6.3. 시스템 병목 탐지 지표 및 제어 루프 대역폭(Control Loop Bandwidth) 프로파일링

PX4-Autopilot(Firmware Version 1.14 이상)과 ROS2 컴패니언 컴퓨터가 결합된 자율 비행 시스템은 본질적으로 다중 스레드(Multi-threaded) 및 다중 노드(Multi-node)가 경합하는 복잡한 실시간(Real-time) 파이프라인이다. 이 파이프라인에서 센서 데이터의 획득부터 액추에이터 명령 하달까지 걸리는 시간이 특정 임계치를 초과할 경우, 시스템은 단순한 성능 저하를 넘어 물리적인 제어 불안정성(Instability) 상태로 진입하게 된다.

본 절에서는 분산 제어 아키텍처의 건전성을 진단하기 위해 필수적인 시스템 병목(Bottleneck) 탐지 지표를 정립하고, 폐루프 제어(Closed-loop Control)의 핵심 성능 척도인 제어 루프 대역폭(Control Loop Bandwidth)을 정량적으로 프로파일링(Profiling)하는 방법론을 상세히 해설한다.

1. PX4-ROS2 파이프라인의 시스템 병목 탐지 지표

PX4 내부의 uORB 메시징 버스와 외부의 uXRCE-DDS 네트워크 망을 넘나드는 데이터 흐름 속에서, 병목 현상은 CPU 스케줄링 간섭, 통신 대역폭 포화, 자금 잠금(Lock Contention) 등 다양한 원인에 의해 촉발된다. 이를 색출하기 위해 3대 핵심 지표(KPI)를 실시간으로 모니터링해야 한다.

1.1 종단 간 지연 시간(End-to-End Latency)

지상 관제 시스템(GCS)이나 온보드 컴퓨터에서 오프보드(Offboard) 제어 명령(예: TrajectorySetpoint)을 하달한 순간부터, PX4의 Rate Controller가 이를 수신하여 모터 믹서(Mixer)에 반영하기까지 걸리는 총 체공 시간이다.

  • 정상 범주: 이더넷 기반 브리지에서 2\text{ms} ~ 5\text{ms} 안정 유지.
  • 병목 징후: 지연 시간이 20\text{ms} 이상으로 튀는 지터 범프(Jitter Bump) 주기적 발생 시 통신 버퍼나 QoS 설정 점검 요구.

1.2 토픽 드롭율(Topic Drop Rate)

지정된 업데이트 주기(Hz) 대비 실제 수신된 메시지의 비율이다. 이는 통신 대역폭의 물리적 포화 상태를 대변한다.

  • 진단법: ros2 topic hz /fmu/out/vehicle_odometry 명령을 통해 원래 퍼블리시 주기(예: 200\text{Hz})와 실제 수신 주기 간의 편차를 계산.
  • 병목 징후: 10\% 이상의 지속적인 패킷 드롭 발생 시, 직렬 통신(UART)의 보드 레이트(Baudrate) 한계 초과 또는 DDS의 Best-Effort QoS에 의한 강제 폐기를 의심.

1.3 CPU 부하 및 스레드 블로킹(Thread Blocking)

PX4 NuttX RTOS 상의 Work Queue 지연 시간과 리눅스 컴패니언 컴퓨터의 Context Switching Overhead를 측정한다.

  • 진단법: PX4 콘솔에서 top 명령어 및 perf 도구를 활용하여 uXRCE-DDS 에이전트 태스크의 CPU 점유율을 모니터링.

2. 제어 루프 대역폭(Control Loop Bandwidth) 프로파일링

제어 이론(Control Theory)에서 대역폭은 시스템이 얼마나 빠른 주파수의 명령 변화에 추종(Tracking)할 수 있는지를 나타내는 척도이다. PX4의 자세 제어(Attitude Control) 및 각속도 제어(Rate Control) 루프의 대역폭 한계는 기체의 기계적 특성과 네트워크 지연의 결합 함수로 결정된다.

2.1 주파수 응답(Frequency Response) 테스트 주입(Injection)

시스템의 진정한 대역폭을 알아내기 위해서는 “스윕(Sweep)” 방식의 정현파(Sine Wave) 명령 주입 기법이 사용된다.

  1. 테스트 신호 주입: ROS2 노드에서 PX4의 VehicleTorqueSetpoint (또는 Rate Setpoint)로 0.1\text{Hz}부터 100\text{Hz}까지 주파수가 점진적으로 상승하는 정현파 제어 명령을 스윕(Sweep) 형태로 인가한다.
  2. 응답 데이터 로깅: PX4의 vehicle_angular_velocity (실제 기체 반응) 데이터를 고주기로 로깅한다.
  3. 보드 선도(Bode Plot) 추출: 명령된 정현파 신호(Input)와 측정된 각속도 신호(Output) 간의 진폭비(Magnitude Ratio)와 위상차(Phase Shift)를 계산하여 주파수 도메인 상에 플로팅한다.

2.2 대역폭 한계점(Cut-off Frequency)의 결정 및 해석

도출된 보드 선도(Bode Plot)에서 대역폭 f_{bw} 는 입력 대비 출력의 진폭이 -3\text{dB} (약 70.7\%)로 떨어지는 지점의 주파수로 정의된다.

  • 기계적 한계와 통신 한계의 교점:
  • 통상 5인치 레이싱 드론의 경우 기계적 대역폭은 30\text{Hz} ~ 50\text{Hz}에 달하지만, 대형 상업용 드론은 관성 모멘트(Moment of Inertia)가 커서 5\text{Hz} 수준에 머문다.
  • 여기에 PX4-ROS2 브리지의 네트워크 지연 시간(T_d)이 추가되면, 나이퀴스트 안정도 판별법에 의해 위상 지연(Phase Lag \phi = -2\pi f T_d)이 급격히 발생하여 추가적인 위상 여유(Phase Margin)를 빠르게 잠식한다.
  • 통신 병목으로 지연 시간이 상승하면, 이 -3\text{dB} 대역폭 지점이 왼쪽(저주파 대역)으로 밀려나게 되며, 이는 기체가 외부의 빠른 외란(Wind Gust)이나 고속의 조향 명령에 민첩하게 반응하지 못하고 출렁거리게 됨을 의미한다.

결론적으로, 오프보드 연산 환경을 구축할 때 단순한 기능적 통신 개통을 넘어 시스템 병목 지표와 제어 대역폭을 주기적으로 프로파일링하는 것은, 해당 기체가 궤적 추종 오차(Tracking Error)를 목표 임계치 내로 영구히 속박할 수 있는지를 보증하는 가장 강력한 제어 공학적 감사(Audit) 절차이다.
