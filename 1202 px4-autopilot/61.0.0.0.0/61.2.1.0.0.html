61.2.1. 경수준 실시간성(Soft Real-Time) 보장을 위한 리눅스 커널 공간 최적화

1. 개요

이종 분산 시스템이자 드론 제어의 중추인 PX4-Autopilot 시스템에 있어, 지상 관제망 또는 동반 컴퓨터(Companion Computer)에서 동작하는 ROS2(Robot Operating System 2) 애플리케이션의 지연율(Latency)은 비행 안전성과 직결된다. PX4 비행 제어기 자체가 제공하는 마이크로초 단위의 파이프라인과 완벽하게 동기화하기 위해서는, ROS2가 구동되는 호스트 리눅스(Linux) 환경에서도 메시지 전송 및 처리 속도에 대한 경수준 실시간성(Soft Real-Time)을 충분히 보장해야 한다. 본 절에서는 오프보드(Offboard) 환경 통신 시 발생할 수 있는 지터(Jitter)를 억제하고 경수준 실시간성을 달성하기 위한 구체적인 리눅스 커널 공간(Kernel Space) 최적화 기법에 대해 철저히 다룬다.

2. 범용 리눅스 환경과 실시간성의 간극

우분투(Ubuntu)를 비롯한 대부분의 범용 리눅스 배포판은 공평성(Fairness)과 시스템 전반의 전체 스루풋(Throughput) 최적화를 목표로 설계된 완전 공정 스케줄러(Completely Fair Scheduler, CFS)를 사용한다. 이러한 아키텍처는 다수의 사용자가 동시에 시스템을 활용할 때는 유리하나, 센서 융합이나 경로 계획(Path Planning)처럼 짧고 규칙적인 주기로 CPU를 획득해야 하는 단일 제어 프로세스에는 치명적인 선점 지연(Preemption Latency)을 야기한다.

또한, 범용 커널에서는 커널 모드 진입(System Call)이나 하드웨어 인터럽트(Hardware Interrupt) 처리 시 다른 스레드가 CPU를 선점하지 못하도록 전역 락(Global Lock)이나 런큐(Runqueue) 락을 획득하는 구간이 존재한다. 이로 인해 Micro XRCE-DDS 에이전트와 같이 데이터를 지속 송수신하는 프로세스가 간헐적으로 스케줄링 밖으로 밀려나게 되며, 이는 결과적으로 PX4가 최신 센서 및 제어 명령 데이터를 제시간에 수신하지 못하는 데이터 손실(Drop) 문제를 초래한다.

3. 커널 공간 최적화(Tuning) 핵심 기법

경수준 실시간성을 성공적으로 달성하기 위해, 커널의 컴파일 단계 혹은 런타임 매개변수 조정을 통해 다음과 같은 세부 공간 최적화를 적용한다.

3.1 CPU 고립(CPU Isolation)을 통한 캐시 및 스케줄러 간섭 방지

다중 코어(Multi-core) 아키텍처를 채택한 동반 컴퓨터의 경우, ROS2 제어 노드와 PX4-ROS2 브릿지(Micro XRCE-DDS) 전용으로 물리적 CPU 코어를 할당하고, 해당 코어를 시스템의 기본 태스크 스케줄링 대상에서 제외해야 한다. 이를 통해 컨텍스트 스위칭(Context Switching) 비용을 대폭 절감할 수 있다.

부트로더(GRUB) 매개변수 적용 예시:

# /etc/default/grub 파일을 수정하여 시스템 부팅 시 커널 파라미터 전달
# 예: 4코어 환경에서 2, 3번 코어를 격리 (0, 1번 코어에서만 범용 작업 수행)
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash isolcpus=2,3 rcu_nocbs=2,3 nohz_full=2,3"

위 명령어에서 적용된 핵심 매개변수들의 역할은 다음과 같다.

  • isolcpus: 지정된 CPU를 사용자 공간의 기본 인터랙티브 스케줄러 대상에서 분리한다.
  • nohz_full (Tickless Kernel): 격리된 CPU에서 작동하는 스레드가 1개일 경우 스케줄링을 위한 타이머 틱(Timer Tick) 발생을 중단시켜 컨텍스트 스위칭 간섭을 없앤다.
  • rcu_nocbs: 해당 코어에서 Read-Copy-Update(RCU) 콜백을 처리하지 않도록 분리하여 커널 내부 작업에 의한 지연을 회피한다.

3.2 IRQ(Interrupt Request) 프로세서 친화도(Affinity) 조정

하드웨어 인터럽트는 현재 실행 중인 태스크에 상관없이 CPU를 강압적으로 선점한다. 따라서 네트워크 카드(NIC), USB 버스 등의 IRQ를 ROS2 제어 노드가 구동 중인 코어에서 분리(Offloading)하여 다른 코어로 우회(Routing)시키는 과정이 필수적이다. 리눅스 환경에서는 smp_affinity 마스크를 수정하여 이와 같은 처리가 가능하다.

3.3 스레드 인터럽트화(Threaded Interrupts) 활성화

소프트 실시간성 구현 시, threadirqs 부트 파라미터를 추가하여 모든 인터럽트 핸들러를 별도의 스레드로 동작하게 할 수 있다. 이 기법은 인터럽트 핸들러 자체에 rt_prio를 부여할 수 있기 때문에, 상대적으로 덜 중요한 하드웨어 인터럽트보다 제어 데이터 송수신 스레드 영역에 더 높은 실시간 우선순위(SCHED_FIFO)를 스케줄러상에서 보장받게끔 유도한다. 이는 QGroundControl과 PX4 간의 MAVLink 통신 시 발생할 수 있는 패킷 드랍 및 프로토콜 타임아웃(Timeout) 억제에 크게 기여한다.

4. 시스템 프로파일링 및 기대 효과

상기 명시된 공간 최적화 튜닝을 시스템에 적용한 이후에는 cyclictest를 비롯한 실시간성 지핑 툴을 활용하여 시스템 응답 시간의 상한선이 제어 주파수 이내로 유지되는지 검증해야 한다.
위와 같은 리눅스 스케줄러 튜닝은 하드 하드웨어 변경 없이도 MAVLink나 ROS2 미들웨어의 패킷 유실을 극적으로 감소시키고, 센서 데이터 타임스탬프(Timestamp) 오차를 낮추어 VIO(Visual Inertial Odometry)와 같은 민감도 높은 퓨전 알고리즘 정밀도를 비약적으로 향상시킨다.
본 최적화를 기반으로 하여, 이어지는 절에서는 리눅스 표준 스케줄러 역학 한계 분석 및 가상 메모리 스와핑 회피 방안 등 한 단계 심화한 커널 공간 최적화 기법에 대해 논의한다.
