61.1.3.1 공용 데이터 표현(CDR) 표준을 활용한 이종 아키텍처 간의 바이트 순서(Endianness) 자동 역전 및 부동소수점 캐스팅(Casting) 알고리즘

PX4-Autopilot 생태계는 매우 다양한 하드웨어와 운영체제가 결합하는 이종(Heterogeneous) 분산 제어 환경으로 진화하고 있다. 비행 제어기(Pixhawk 시리즈의 ARM Cortex-M)는 지상 관제 시스템(x86_64 기반의 Windows/Linux PC) 및 컴패니언 컴퓨터(ARM 기반의 Raspberry Pi 또는 특정 Big-Endian 프로세서)와 실시간으로 데이터를 교환해야 한다.

이때 서로 다른 시스템 아키텍처가 메모리에 바이트를 싣는 순서, 즉 **엔디언(Endianness)**이 다르거나 부동소수점(Floating-Point)을 해석하는 방식이 다르면 센서 데이터와 제어 명령이 심각하게 오염된다. Micro XRCE-DDS가 채택한 공용 데이터 표현(CDR: Common Data Representation) 표준은 이러한 이종 아키텍처 간의 데이터 불일치를 하드웨어의 오버헤드 없이 수학적으로 매끄럽게 해결하는 핵심 프로토콜이다.

1. 이종 아키텍처 간 엔디언(Endianness) 붕괴 문제

엔디언은 다중 바이트 데이터(예: 4바이트 정수형 uint32_t)를 메모리에 배열하는 순서이다.

  • 리틀 엔디언(Little-Endian): 가장 낮은 자리의 바이트(LSB)를 가장 낮은 메모리 주소에 저장. (예: ARM, x86)
  • 빅 엔디언(Big-Endian): 가장 높은 자리의 바이트(MSB)를 가장 낮은 메모리 주소에 저장. (네트워크 표준 및 일부 고성능 서버 아키텍처)

만약 리틀 엔디언 기반의 비행 제어기가 자이로스코프 데이터 0x12345678을 전송하고, 빅 엔디언 기반의 기기가 이를 수신하여 그대로 해독한다면 0x78563412라는 완전히 엉뚱한 값으로 해석되어 기체의 즉각적인 추락(Crash)을 유발할 수 있다.

2. CDR 표준의 수신기 부담(Receiver-Makes-It-Right) 원칙과 자동 역전 메커니즘

과거 네트워크 프로토콜에서는 전송을 하기 전에 반드시 송신자가 데이터를 네트워크 표준 엔디언(Big-Endian)으로 변환하는 방식(예: htonl(), ntohl() 함수 사용)을 취했다. 그러나 이는 MCU의 CPU 사이클을 송수신 양측에서 불필요하게 낭비하는 결과를 초래했다.

DDS 프로토콜 기반의 CDR(Common Data Representation) 표준은 이른바 **수신기 부담 원칙(Receiver-Makes-It-Right)**을 채택하여 송신자의 오버헤드를 완전히 제거했다.

2.1 메타데이터(Header)를 통한 엔디언 플래그 전송

CDR은 직렬화된 데이터 스트림의 맨 앞(Header)에 항상 2바이트의 메타데이터 비트를 포함시켜 데이터 페이로드의 인코딩 방식을 아나운스한다.

  • 0x00: Big-Endian으로 인코딩됨
  • 0x01: Little-Endian으로 인코딩됨

2.2 바이트 자동 역전(Byte Reversal) 알고리즘 동작

  1. 송신측 (PX4 내부): 자신이 사용하는 호스트의 네이티브(Native) 엔디언 방식 그대로 구조체를 직렬화 버퍼에 덤프(Dump)하고 역전 변환 코드를 전혀 수행하지 않는다. (O(1) 블록 복사)
  2. 수신측 (ROS 2 노드 등): 헤더에 기록된 엔디언 플래그를 확인한다.
  • 만약 수신측의 네이티브 아키텍처 환경과 플래그가 일치한다면, 아무 변환 없이 메모리로 맵핑한다.
  • 플래그가 불일치한다면, 역직렬화(Deserialization) 과정에서 SWAP 연산을 수행하여 바이트 순서를 반대로 뒤집는다(Byte Reversal).
// 바이트 순서 자동 역전을 수행하는 eProsima Fast CDR 내부 로직 예시
if (msg_endianness != host_endianness) {
    // 32-bit (4-byte) 데이터 역전
    uint32_t raw_val = read_from_buffer();
    uint32_t unswapped = ((raw_val >> 24) & 0xff)      | 
                         ((raw_val << 8)  & 0xff0000)  | 
                         ((raw_val >> 8)  & 0xff00)    | 
                         ((raw_val << 24) & 0xff000000);
    assign_to_struct(unswapped);
}

이러한 설계 덕분에, 현존하는 대부분의 시스템이 리틀 엔디언을 범용적으로 사용한다는 점을 감안할 때 99%의 통신 무결성 검사에서 바이트 스왑 함수가 실행조차 되지 않아 시스템 성능이 극대화된다.

3. 부동소수점 데이터(Float / Double)의 무손실 캐스팅(Casting) 알고리즘

무인항공기의 제어공학 관점에서 위치, 속도, 자세 제어 PID 이득(Gain)과 같은 데이터는 모두 부동소수점 데이터(float 32-bit 또는 double 64-bit)로 관리된다. 그러나 이를 직렬화할 때 실수값을 그대로 비트 시프트(Bit-shift)하는 것은 문법적으로 유효하지 않거나 데이터 손실(Precision Loss)을 유발할 수 있다.

이를 방지하기 위해 CDR은 IEEE 754 (IEC 60559) 표준에 의거하여 메모리 레벨의 딥 카피(Deep Copy) 및 캐스팅을 수행한다.

3.1 Union 기반의 안전한 메모리 앨리어싱(Aliasing)

직렬화 시 부동소수점을 바이트 스트림 정수형으로 변형하기 위해 C/C++에서 흔히 쓰이는 포인터 형변환(예: *(uint32_t*)&float_val)은 엄밀한 문법(Strict Aliasing Rule) 규칙을 위배하여 컴파일러 최적화 도중 의도치 않은 버그를 양산할 가능성이 있다.

따라서 px4_msgs 및 DDS 직렬화 모듈은 Union 구조나 C++11의 std::memcpy, 혹은 현대 C++20의 std::bit_cast 메커니즘을 응용하여 부동소수점 트랜잭션을 구현한다.

// 부동소수점 무손실 캐스팅 트랜잭션 예시 (Union 방식)
union FloatIntCast {
    float f_value;
    uint32_t i_value;
};

FloatIntCast caster;
caster.f_value = 3.14159f; // 비행 제어기 내부의 자세 추정값
// 이후 caster.i_value 를 바이트 스트림에 uint32_t 처럼 전송

위 앨리어싱 과정을 통해 IEEE 754 표준의 부호 바이트(Sign), 지수부(Exponent), 가수부(Mantissa)가 단 1비트의 손실도 없이 4바이트 정수 배열로 변환되어 네트워크 스트림에 올라탈 수 있게 된다.

4. 시스템 확장에 대한 기술적 메리트

uORB 브릿지가 XRCE-DDS의 CDR 인코딩 철학을 온전히 수용한 것은 시스템 확정성(Scalability) 관점에서 막대한 강점을 부여한다. 개발자는 PX4 펌웨어가 x86 시뮬레이션 환경(SITL)에서 구동되든, ARM 칩 위(Hardware)에서 구동되든, 수신하는 GCS가 모바일 앱이든 서버 환경이든 상관없이 데이터 해석의 일관성을 100% 보장받는다.

결론적으로 바이트 순서 자동 역전 플래그와 무손실 부동소수점 캐스팅 알고리즘은 분산 무인 시스템 데이터 아키텍처에 있어 보이지 않는 “만국 공통어 통역기“로서 핵심적인 역할을 수행 중이라 할 수 있다.
