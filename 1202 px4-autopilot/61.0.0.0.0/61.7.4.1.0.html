61.7.4.1. 클라이언트-에이전트 간 송신 버퍼의 최대 전송 단위(MTU) 크기 상한선 제어를 통한 운영체제 네트워크 스택 단의 지수적(Exponential) 데이터 파편화 현상과 큐 지연(Queueing Delay) 억제 기법

PX4-Autopilot(Firmware Version 1.14 이상) 기반의 드론과 ROS2 컴패니언 컴퓨터가 이더넷(Ethernet)이나 직렬(UART) 무선 텔레메트리로 결합될 때, 데이터의 효율적 전송을 담보하는 가장 근원적인 매개변수 집합 중 하나가 프레임 구성 정책이다. 특히 uXRCE-DDS 클라이언트(PX4)와 에이전트(ROS2) 간에 전송되는 데이터 덩어리는 물리 계층을 통과하기 전 적절한 크기로 재단되어야 하는데, 이를 주관하는 척도가 최대 전송 단위(MTU, Maximum Transmission Unit)이다.

만약 애플리케이션 계층에서 생성된 sensor_combined 토픽 배열이 네트워크 스택의 통과 허용량(MTU)을 무분별하게 초과하게 되면, 운영체제(NuttX 커널 혹은 Linux) 내부에서 끔찍한 데이터 파편화(Fragmentation) 현상과 큐 지연(Queueing Delay) 폭주가 필연적으로 발생한다. 본 절에서는 MTU 상한선 제어의 기하학적 메커니즘을 분석하고, 지수적 파편화가 초래하는 실시간 제어기의 붕괴를 하향식 설정(Top-down configuration)으로 억제하기 위한 네트워킹 스택 프로파일링 기법을 해설한다.

1. 운영체제 네트워크 스택과 데이터 파편화(Fragmentation) 동역학

네트워크 인터페이스 카드(NIC)나 직렬 드라이버는 한 번의 인터럽트(Interrupt) 구동으로 전송할 수 있는 바이트의 물리적 한계치(MTU)를 명확히 규정하고 있다. 표준 이더넷의 경우 통상 1500\text{ Bytes}, 시리얼 기반(SLIP 등) 환경의 uXRCE-DDS 디폴트는 하드웨어 가용성에 따라 512\text{ Bytes} 내외로 극도로 제한된다.

1.1 MTU 초과 시의 지수적 단편화(Exponential Fragmentation) 메커니즘

DDS 퍼블리셔가 4000\text{ Bytes}짜리 거대한 상태(State) 베열을 송신하려 한다고 가정하자.
설정된 \text{MTU} = 512\text{ Bytes} 라면, TCP/IP 혹은 UDP/IP 네트워크 스택은 이 원시 데이터를 512\text{ Bytes} (정확히는 IP/UDP 헤더 길이를 제외한 페이로드 크기) 단위의 조각(Fragment) 8개 이상으로 무참히 쪼개버려야 한다.

  • 파편화 페널티(Fragmentation Penalty): 쪼개진 각 조각마다 20\sim40\text{ Bytes} 단위의 MAC/IP/UDP 프레임 헤더가 중복해서 기생수처럼 들러붙는다. 오버헤드(Overhead)의 지수적 팽창이 시작되는 것이다.
  • 재조립(Reassembly) 리스크: 수신단 운영체제 커널 계층에서 8개의 파편이 모두 안전하게 도착할 때까지 대기(Wait)해야 한다. 만일 직렬 통신 환경의 잡음으로 인해 8개의 파편 중 단 1개라도 증발(Drop)해버린다면, 타임아웃 경과 후 4000\text{ Bytes} 원본 전체를 무효화(Discard)하고 전체를 재전송(Retransmission)받아야 하는 확률적 참사가 빚어진다.

\text{Drop Rate}_{total} = 1 - (1 - P_{frag\_drop})^N
(N: 파편화 개수. 단 1개 조각이라도 버려질 확률은 조각 수가 늘어날수록 급진적으로 1에 수렴한다.)

2. 큐 지연(Queueing Delay) 파동과 HOL(Head-of-Line) Blocking

조각난 메시지 파편들이 NIC 버퍼 큐(Queue)를 맹렬히 점령(Saturation)하면, 그 뒤에 줄을 서 있는 아주 작지만 생판(Life-critical) 중요한 패킷—예를 들어 24\text{ Bytes}에 불과한 오프보드 기체 제어기 모드 변경 명령—이 길을 뚫고 지나가지 못하는 교착 현상이 벌어진다. 이를 헤드-오브-라인 블로킹(HOL Blocking)이라 칭한다.

파편들에 의해 물리 채널이 완전히 독점당하는 수백 밀리초 동안 대기열 뒤편에 갇힌 오도메트리와 제어 명령의 지연 시간(Queueing Delay)은 치솟게 되며, 제어기 위상 여유(Phase Margin)를 송두리째 무너뜨린다.

3. 클라이언트-에이전트 송신 버퍼 MTU 상한선 최적화 전략

가장 우아한 해결책은 운영체제 단의 파편화가 아예 일어날 수 없도록, 원천지(Source)인 미들웨어 계층의 버퍼 상한선과 애플리케이션 메시지 크기를 통제하는 것이다.

3.1 UXRCE_MTU 프로비저닝 (Firmware Tuning)

PX4 모듈 소스 코드 트리 내부 혹은 환경 변수 설정 스크립트에서 초기화되는 uXRCE-DDS 클라이언트의 MTU 상한선을 현재 물리 매체 환경에 완벽하게 일치시켜야 한다.

  • 이더넷(Ethernet) 전용 기체: \text{MTU} = 1500\text{ Bytes} (점보 프레임 지원 시 최대화 가능) 설정하여 조각화를 축소한다.
  • UART 라디오 모뎀 기체: 하드웨어 FIFO 버퍼 오버플로우를 막기 위해 \text{MTU} = 128\sim256\text{ Bytes} 단위로 강제 하향(Drop down) 조정한다.

3.2 애플리케이션 계층 Payload 쪼개기(Sharding)

메시지 설계의 본질적 수정 지침이다. 절대 단일 ROS2 .msg 파일 내부에 무식하게 큰 바이트 동적 배열(예: 4\text{K} 해상도 PointCloud)과 중요 헤더를 동시에 우겨넣지 마라. 거대 데이터는 반드시 청크(Chunk) 단위로 쪼개어 각각 독립적인 토픽(Topic)으로 분할 퍼블리싱해야 미들웨어 스택의 HOL 블로킹 없이 흐름 제어(Flow Control)를 유연하게 태울 수 있다.

결과적으로 MTU 상한선의 의도적인 통제는 패킷 파편화의 지수형 팽창을 원천 차단하는 방파제이며, 실시간성이 목숨과도 같은 비행 제어 라우팅을 매끄럽게 유지하는 로우레벨(Low-level) 지휘 철학의 백미라 할 것이다.
