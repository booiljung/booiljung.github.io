61.3 Micro XRCE-DDS 에이전트 소프트웨어 크로스 컴파일 및 시스템 프로비저닝

PX4-Autopilot의 uORB 메시지 생태계와 ROS2의 DDS(Data Distribution Service) 토픽 통신을 유기적으로 연결하기 위해서, 필수적으로 도입해야 하는 미들웨어 브릿지가 바로 Micro XRCE-DDS 기술이다. 이는 기존의 microRTPS 아키텍처를 완벽하게 대체(Deprecated from PX4 v1.14)하는 현대적인 솔루션이다. 본 절에서는 동반 컴퓨터(Companion Computer, 예: 라즈베리 파이, Jetson Nano 등)에 Micro XRCE-DDS 에이전트(Agent)를 구축하기 위한 크로스 컴파일(Cross-Compilation) 전략과, 안정적인 운영 환경을 보장하기 위한 시스템 프로비저닝(Provisioning) 기법을 상세히 다룬다.

1. 클라이언트(Client) - 에이전트(Agent) 아키텍처의 이해

Micro XRCE-DDS 시스템은 비대칭적인 리소스 소모 환경에 특화되어 설계되었다.

  • 클라이언트(Client): 상대적으로 리소스가 극히 제한적인 마이크로컨트롤러(MCU) 내부, 즉 Pixhawk와 같은 비행 제어기(FC)의 NuttX RTOS 상에서 동작한다. uORB 토픽 데이터를 XRCE 포맷으로 직렬화(Serialization)하여 내보낸다.
  • 에이전트(Agent): 넉넉한 컴퓨팅 파워를 가진 리눅스 기반 동반 컴퓨터(Companion Computer) 위에서 완전한 Fast DDS 백엔드를 가지고 실행된다. 클라이언트로부터 수신한 경량 메시지를 정식 ROS2 / DDS 토픽으로 변환(Bridging)하여 퍼블리싱한다.

2. 크로스 컴파일(Cross-Compilation)을 고려한 빌드 시스템 구성

실제 자율 에이전트 비행을 구성할 때, 에이전트 소프트웨어는 x86 호스트 PC가 아닌 ARM 아키텍처(aarch64 등)를 가진 임베디드 보드에 탑재된다. 대상 보드(Target) 내에서 직접 컴파일(Native Compile)을 수행하는 것도 가능하나, 의존성 라이브러리의 용량이나 컴파일 속도의 한계로 인해 성능이 우수한 호스트 PC에서 크로스 툴체인(Toolchain)을 사용하여 에이전트 바이너리를 뽑아내는 방법이 널리 쓰인다.

크로스 컴파일을 수행하기 위해서는 CMake 툴체인 파일을 활용해야 한다. 다음은 aarch64-linux-gnu 컴파일러를 지정하기 위한 toolchain.cmake 작성 예시이다.

# toolchain.cmake 구성 예시
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR aarch64)

# 크로스 컴파일러의 절대 경로 지정
set(CMAKE_C_COMPILER /usr/bin/aarch64-linux-gnu-gcc)
set(CMAKE_CXX_COMPILER /usr/bin/aarch64-linux-gnu-g++)

# 호스트 환경과 타겟 환경의 루트 경로 분리
set(CMAKE_FIND_ROOT_PATH /usr/aarch64-linux-gnu)
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)

이후 Micro-XRCE-DDS-Agent 소스 코드를 복제하여 아래와 같은 방식으로 빌드 파이프라인을 구동한다. ROS2 colcon 빌드를 사용해도 되지만, 순수 에이전트 데몬만 필요할 때는 독립된 CMake 빌드가 시스템 결합도를 낮추는 이점이 있다.

git clone https://github.com/eProsima/Micro-XRCE-DDS-Agent.git
cd Micro-XRCE-DDS-Agent && mkdir build && cd build

# 앞서 작성한 toolchain 파일을 명시하여 CMake 구성
cmake -DCMAKE_TOOLCHAIN_FILE=../toolchain.cmake -DUAGENT_BUILD_EXECUTABLE=ON ..
make -j$(nproc)

3. 시스템 라이프사이클 데몬화 및 프로비저닝(Systemd Provisioning)

컴파일된 에이전트 바이너리를 무인항공기의 동반 컴퓨터에서 실행할 때, 가장 중요한 신뢰성 요구사항은 “전원 인가 직후 (OS 부팅 성공 시) 에이전트가 사용자 개입 없이 즉시 가동되어야 하며, 장애 시 스스로 복구(Restart) 되어야 한다“는 점이다. 이를 위해 데비안 계열 리눅스 시스템의 표준 이니트(Init) 시스템인 systemd 기반 프로비저닝 작업을 수행해야 한다.

동반 컴퓨터의 /etc/systemd/system/uxrce_dds_agent.service 파일을 다음과 같이 작성한다.

[Unit]
Description=Micro XRCE-DDS Agent for PX4 ROS2 Bridge
After=network.target

[Service]
Type=simple
# 에이전트 바이너리의 절대 경로 및 통신 인터페이스 지정 (예: 직렬 통신 /dev/ttyTHS1, 보드레이트 921600)
ExecStart=/usr/local/bin/MicroXRCEAgent serial --dev /dev/ttyTHS1 -b 921600
Restart=on-failure
RestartSec=3

# 보안 강화를 위한 최소 구동 권한 설정 (루트 권한 지양)
User=px4dev
Group=dialout

[Install]
WantedBy=multi-user.target

작성이 완료되면 시스템 데몬을 리로드하고 서비스를 영구적으로 활성화시킨다.

sudo systemctl daemon-reload
sudo systemctl enable uxrce_dds_agent
sudo systemctl start uxrce_dds_agent

4. QGroundControl(QGC)과 파라미터 매치 검증

동반 컴퓨터의 에이전트 환경이 완성되었더라도, 픽스호크 펌웨어 내부의 클라이언트 모듈이 적절한 통신 매체를 통해 활성화되지 않으면 연결(Bridging)은 이루어지지 않는다.

사용자는 QGroundControl을 통해 기체와 연결한 후, Parameters 탭에서 UXRCE_DDS_CFG 파라미터를 찾아야 한다. 이 값은 통신 프로토콜과 인터페이스를 결정한다(예: 0=Disabled, 1=TELEM1, 2=TELEM2, …).
비행 제어기와 동반 컴퓨터 간의 하드웨어 와이어링(UART TX/RX 교차 연결) 포트가 TELEM2에 연결되었다면, UXRCE_DDS_CFG를 TELEM2로 활성화하고 기체를 재부팅해야 클라이언트 태스크가 생성되며, 에이전트 데몬 로그 상에서 [Session Established]라는 연결 성공 상태를 확인할 수 있다.

이러한 크로스 컴파일과 시스템 레벨의 프로비저닝 설계는 실험실 환경(SITL)을 넘어 실제 실외 자율 비행(Offboard) 미션을 수행하는 상용 GCS 관제 기술의 필수적인 개발 기반암(Bedrock)이 된다.
