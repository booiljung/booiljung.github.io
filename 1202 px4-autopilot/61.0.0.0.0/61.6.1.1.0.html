61.6.1.1. 에이전트와 클라이언트 간의 핸드셰이크(Handshake) 상태 전이 과정에서의 암호화 논스(Nonce) 교환 및 세션 생명주기 타이머 동기화 로직

PX4-Autopilot 생태계(Firmware Version 1.14 이상 탑재)에서 uXRCE-DDS 미들웨어가 구동될 때, 온보드(Onboard) 비행 제어기와 오프보드(Offboard) 관제 컴퓨터 간의 통신은 단순 무상태(Stateless) 패킷 전송이 아닌, 확고한 세션(Session) 기반의 유한 상태 기계(Finite State Machine, FSM) 통제 하에 이루어진다. 이 세션의 수립 단계인 핸드셰이크(Handshake) 과정은 통신의 무결성과 보안성, 그리고 데이터 파이프라인의 생명주기를 결정짓는 가장 핵심적인 초기화 관문이다.

본 절에서는 uXRCE-DDS 프로토콜을 기반으로 에이전트(ROS2 브리지)와 클라이언트(PX4)가 논리적 채널을 개통하는 교섭 동역학(Negotiation Dynamics)을 분석하고, 암호학적 논스(Nonce) 교환을 통한 세션 하이재킹(Hijacking) 방지 및 세션 생명주기(Session Lifecycle)의 타이머 동기화 알고리즘을 학술적으로 고찰한다.

1. XRCE-DDS 핸드셰이크(Handshake) 상태 전이 모델

PX4 내장 micro_client는 이더넷 혹은 직렬(UART) 버스를 통해 첫 패킷을 전송하는 순간부터, 서버 역할을 하는 micro_agent와 정교하게 정의된 상태 지향형 통신을 시작한다. 이 연결의 수립은 CREATE_CLIENT 오퍼레이션을 통해 시작된다.

1.1 3-Way Handshake와 연결 컨텍스트(Context) 할당

  1. 상태 초기화 (Init State): 클라이언트는 특정 에이전트 IP:Port 또는 활성 직렬 채널로 연결(Connect) 요청을 발송한다. 이 메시지는 XRCE 헤더 버전에 클라이언트의 Session ID(무작위 할당)가 포함된다.
  2. 세션 검증 (Negotiation State): 에이전트는 해당 수신 요청에 대해 가용한 리소스(Resource Limit)를 확인한다. 수용 가능한 연결이라면, 에이전트 역시 자신의 XRCE 컨텍스트를 활성화시키고 승인(Acknowledge) 메시지를 회신한다.
  3. 토픽 엔터티 생성 (Established State): 상호 간 Session ID가 확인되면, 클라이언트는 본격적으로 데이터 파이프라인의 핵심인 퍼블리셔(Publisher), 서브스크라이버(Subscriber), 데이터라이터(DataWriter), 데이터리더(DataReader) 엔터티(Entity)를 에이전트 측에 원격으로 동적 생성(Create)해달라는 명령 트랜잭션을 실행한다.

각 엔터티의 생성이 STATUS_OK로 반환될 때 비로소 핸드셰이크가 종료되며 동기화 검증을 위한 본격적 데이터 통신 상태(Ready)로 전이된다.

2. 암호화 논스(Nonce) 교환 및 보안 터널링 원리

열린 네트워크(Open Network)나 다중 에이전트 군집(Swarm) 환경에서 무단 접근이나 재생 공격(Replay Attack)을 차단하기 위해, 최신 통신 규약은 초기 핸드셰이크 궤적 상에 암호학적 논스(Nonce) 개입을 허용한다.

2.1 세션 식별자(Session ID)와 논스(Nonce) 기반 인가

논스(Number used once)는 한 번만 사용되는 무작위 난수(Random Salt)이다. 연결 초기화 시:

  • Client Request: 클라이언트는 1회용 대형 난수(Nonce_C)를 생성하여 연결 메시지 해시에 포함시켜 에이전트로 전송한다.
  • Agent Challenge: 에이전트는 이를 기반으로 자체 논스(Nonce_A)를 섞어(Mix) 반환하며, 클라이언트의 유효성을 묻는다.
  • Session ID 생성: 이 논스 교환이 성공적으로 검증된 이후에만, 에이전트와 클라이언트는 고유한 1\text{Byte} (또는 내부적으로 2\text{Bytes})의 Session ID를 상호 동기화한다.

만일 제3의 악의적 장치가 과거의 패킷을 가로채 똑같이 주입하는 재생 공격(Replay Attack)을 시도하더라도, 다음 세션 수립 시 동적으로 변동하는 논스 요구값을 맞출 수 없으므로 에이전트 단계에서 STATUS_ERR_INCOMPATIBLE을 반환하며 연결을 거부(Deny)한다.

3. 세션 생명주기(Lifecycle) 및 타이머(Timer) 동기화 파이프라인

핸드셰이크로 수립된 세션은 영구적으로 지속되는 것이 아니다. 통신 지연(Latency)이 불안정하거나 물리적 케이블 단선(Link Loss)이 발생할 경우, 시스템은 고착 상태(Deadlock)에 빠진 ‘좀비(Zombie)’ 세션을 신속하게 판별하고 폐기해야 한다.

3.1 하트비트(Heartbeat) 및 Keep-Alive 타이머 롤오버(Roll-over)

세션 유지의 근간은 양측의 파이어월(Firewall)에서 관리하는 논리적 타이머의 엄격한 동기화에 있다.

  • 클라이언트 타이머(Client Timer): PX4 클라이언트는 일정 주기(예: 1000\text{ms})마다 제어 패킷이나 PING 메시지를 송수신하며 Last_Activity_Time 틱(Tick)을 갱신한다.
  • 에이전트 만료 타이머(Agent Lease Time): 에이전트는 클라이언트를 승인할 때 특정 생명주기 임계값(Lease Duration, 예: 5000\text{ms})을 부여한다. 만일 지정된 제한 시간이 흐르도록 클라이언트로부터 어떠한 수신도 확인되지 않으면, 타이머는 강제 만료(Expired)된다.

3.2 FSM 상실 감지 메커니즘과 세션 복구(Recovery) 역학

타이머가 만료되면 에이전트 측의 세션은 소멸 전이(Terminated State)로 넘어간다. 흥미로운 점은 PX4 클라이언트 측의 생존성 보장 로직이다.

stateDiagram-v2
    [*] --> Init: Booting
    Init --> Negotiation: SEND_CONNECT_REQ
    Negotiation --> Established: RECV_ACK & CREATE_ENTITIES
    
    Established --> Established: SEND_HEARTBEAT / PING (Reset Timer)
    
    state "Timeout Event" as Timeout {
        Established --> SessionLost: No Ack for > 5000ms
    }
    
    SessionLost --> Teardown: Clear Memory Context
    Teardown --> Init: Auto-Reconnect Algorithm Trigger

클라이언트는 지속적으로 데이터 전송을 시도하다 수신측의 응답(Ack)이 Lease Duration 이상 누락되면 독자적으로 세션 상실(Session Lost) 상태를 판정한다. 이 순간 PX4 펌웨어는

  1. 자신에게 할당되었던 메모리 풀 내부의 기존 엔터티(Publisher/Subscriber) 컨텍스트를 전부 초기화(Tear-down)하고,
  2. 즉시 다시 난수를 발생시켜 새로운 암호화 논스를 조립한 후, Init 상태로 되돌아가 전혀 다른 새로운 Session ID 생성을 위한 CREATE_CLIENT를 재송출한다.

이러한 타이머 동기화 기반의 자가 복구 루프(Self-Healing Loop)야말로, 거친 비행 환경과 극심한 주파수 페이딩(Fading) 속에서도 PX4-ROS2 오프보드 제어 체계가 신속하게 통신 건전성을 회복해 낼 수 있는 설계 아키텍처의 백미이다.
