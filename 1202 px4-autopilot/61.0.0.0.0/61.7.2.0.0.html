61.7.2. 프로토콜 계층 데이터 프레임 손상 감지 및 역직렬화(Deserialization) 예외 처리

PX4-Autopilot(Firmware Version 1.14 이상)과 GCS 혹은 ROS2 컴패니언 컴퓨터 간의 데이터 교환 과정에서 물리 계층(Physical Layer)을 무사히 통과한 비트(Bit) 신호들은 프로토콜 계층(Protocol Layer)에 도달하여 비로소 논리적인 ’프레임(Frame)’으로 조립된다. MAVLink 프로토콜이나 uXRCE-DDS 미들웨어는 C++ 구조체(Struct)를 바이트 스트림(Byte Stream)으로 쪼개어 날려 보내는 직렬화(Serialization)를 수행하며, 수신단은 이를 다시 구조체 메모리로 복원하는 역직렬화(Deserialization) 과정을 거친다.

그러나 비행 중 덮치는 가혹한 EMI 환경이나 직렬 포트(UART) 버퍼의 오버런(Overrun) 현상은 이 바이트 스트림의 중간 일부를 절단하거나(Drop) 이빨을 빠뜨려 프레임 자체의 구문을 손상(Corruption)시킨다. 이렇게 손상된 데이터 덩어리를 여과 없이 역직렬화 알고리즘에 밀어 넣게 되면, 메모리 세그멘테이션 결함(Segmentation Fault)으로 인한 컴패니언 컴퓨터 노드의 크래시(Crash)나 기체 추락으로 직결된다.

본 절에서는 프로토콜 계층에서 손상된 데이터 프레임을 감지하기 위한 해시 및 체크섬 구조를 분석하고, 역직렬화 과정의 메모리 침범을 방어하기 위한 강력한 예외 처리(Exception Handling) 아키텍처를 해설한다.

1. 프레임 캡슐화(Encapsulation)와 무결성 검증 아키텍처

MAVLink 2.0 및 XRCE 프로토콜은 악의적이거나 우발적인 데이터 손상을 색출하기 위해 프레임 앞뒤에 엄격한 식별자와 검증 꼬리표(Tail)를 부착한다.

1.1 Magic Byte와 프레임 동기화(Frame Synchronization)

수신 측의 하드웨어 UART 인터럽트는 바이트가 도착할 때마다 버퍼(Buffer)에 데이터를 무지성으로 쌓아둔다. 미들웨어 파서(Parser)가 이 뒤죽박죽인 버퍼 속에서 “어디서부터가 유효한 메시지의 시작인가?“를 찾아내는 과정을 프레임 동기화라 한다.

  • MAVLink 2.0의 경우: 프레임의 헤더 첫 바이트는 항상 0xFD (Magic Byte)로 시작한다. 파서는 바이트 스트림을 순회하다가 0xFD를 발견하면 그 지점을 새 프레임의 머리(Header)로 간주하고 상태 머신(State Machine)을 대기(IDLE) 상태에서 길이 파싱(Parsing) 상태로 락온(Lock-on)시킨다.
  • 만일 중간에 데이터가 드롭되어 프레임 길이가 어긋났더라도, 다음 0xFD를 만나는 순간 파서는 이전의 불완전한 덩어리를 즉각 버리고(Flush) 파싱 기계를 재설정한다.

1.2 순환 중복 검사 (CRC, Cyclic Redundancy Check)

프레임의 머리부터 꼬리 직전까지의 모든 페이로드 바이트 배열을 다항식(Polynomial) 분젯값으로 나누어 얻은 16비트(\text{X.25} 방식 등) 나머지 해시(Hash) 값이다.
PX4 수신단은 도달한 메시지의 페이로드를 직접 CRC 연산해보고, 이 결과가 프레임 꼬리에 적혀 수신된 원본 CRC [CRC_Low, CRC_High] 값과 단 1비트라도 일치하지 않으면 해당 전체 프레임을 즉시 폐기(Drop)한다. 이는 노이즈에 의해 변조된 1\text{m/s} 속도 명령이 운 나쁘게 비트 역전(Bit Flip)을 겪어 201\text{m/s}의 살인적 명령으로 돌변하는 현상을 완벽히 차단한다.

2. 역직렬화(Deserialization) 메모리 붕괴 동역학과 방어책

CRC 검증마저 통과했다고 해서 안심할 수 없다. uXRCE-DDS의 경우, 동적 길이(Dynamic Length) 배열이나 문자열(String) 타입 데이터를 역직렬화할 때 치명적 취약점이 도사린다.

2.1 배열 범위 초과(Out-of-Bounds) 취약점

수신된 페이로드 내부 구조가
[Array_Length : uint32] [Data 1] [Data 2] ...
의 형태로 직렬화되어 오는데, 만일 CRC 검증을 우연히 비껴간 노이즈로 인해 Array_Length 필드 숫자가 비정상적으로 부풀려져 1,000,000 으로 해석되었다고 가정하자.

C++ 기반의 ROS2 노드 역직렬화 로직이 이 미친 길이를 믿고 메모리의 다음 바이트들을 무작정 힙(Heap) 공간 배열로 형 변환(Casting)하며 복사(memcpy)해 나가면, 결국 할당되지 않은 운영체제 커널의 보호 영역을 건드리게 되어 Segmentation Fault (보호 위반)를 띄우고 즉사하게 된다.

2.2 방비 아키텍처: Size-Bounded 파싱 및 예외 처리

이를 막기 위해 Micro-XRCE-DDS-Client(PX4 내부)와 Fast-DDS(컴패니언 컴퓨터) 양측의 역직렬화 코어에는 Bound Checking(경계 검사) 파라다임이 내장되어 있어야 한다.

flowchart TD
    A[UART/UDP 수신 버퍼] --> B{Magic Byte 검출}
    B -- 실패 --> C[Flush & Continue]
    B -- 성공 --> D[Header 및 Length 필드 파싱]
    
    D --> E{Size-Bound Check:\n Header.Length <= Max_Buffer?}
    E -- 초과 시 --> F[Throw Exception \n (Buffer Overflow 막음)]
    E -- 정상 --> G[CRC Checksum 검증]
    
    G -- 불일치 --> H[Drop Packet \n (Corrupted Payload)]
    G -- 일치 --> I[Memory Safe Deserialization \n C++ Struct 복원]
    
    I --> J((Send to \n EKF2/Controller))
  1. 최대 페이로드 크기 고정: 설정 파일에서 XRCE_MAX_PAYLOAD를 기체 제어에 필요한 타당한 맥시멈 사이즈(예: 512\text{Bytes})로 엄격히 고정(Clamp)한다. 헤더에 명시된 길이가 이 매크로 상수 값을 초과하면 CRC 연산조차 수행하지 않고 선제적으로 프레임을 버린다.
  2. C++ try-catch 예외 핸들링: 컴패니언 컴퓨터 측 ROS2 콜백 내부에서 DDS CDR(Common Data Representation) 역직렬화 함수 호출 시, BadParamException 메타 클래스를 catch 블록으로 래핑(Wrapping)하여 코딩한다. 만일 구조체 크기 불일치 예외가 던져지면(Thrown), 노드 크래시 대신 ROS RCLCPP_WARN 에러 로그만 콘솔에 뱉고 다음 프레임 루프를 구동시키도록 생존력을 보장한다.

요컨대, 물리 계층이 진흙탕(EMI)이라면 프로토콜 계층은 시스템의 ’면역 감시망’이다. 타협 없는 프레임 폐기(Drop) 전략과 무자비한 메모리 경계 검사(Boundary Check)만이, 분산 관제망 속에서 비상식적으로 오염된 데이터가 자율 비행의 조종간(Stick)을 탈취하는 파국을 막아내는 최후의 보루이다.
