61.5.4. 분산 다중에이전트 식별자 격리 및 계층적 네임스페이스(Namespace) 래핑

로보틱스 텔레메트리 프로토콜인 MAVLink와 글로벌 데이터 배포 미들웨어인 ROS2(Data Distribution Service, DDS 기반)를 동시에 운용하는 차세대 PX4-Autopilot 군집(Swarm) 환경에서는, 네트워크 상에서 각 에이전트(무인항공기)가 생산하고 소비하는 데이터의 소유권(Ownership)을 명확하게 분리하는 것이 관건이다. 만일 다수의 기체가 동일한 식별 시스템을 공유할 경우, A 기체의 위치 데이터(VehicleLocalPosition)가 B 기체의 제어기(Controller)로 잘못 주입되어 치명적인 추락 사고(Cross-Feeding Crash)를 유도할 수 있다.

본 절에서는 펌웨어 버전 1.14 이상의 PX4-Autopilot 구조 하에서, MAVLink 프로토콜의 시스템 식별자(System ID) 기반 격리 기법과 ROS2 미들웨어의 도메인/네임스페이스(Namespace) 래핑(Wrapping) 기법을 융합하여 상호 간섭 없는 분산 다중에이전트 식별자 아키텍처를 설계하는 알고리즘적 원리를 규명한다.

1. MAVLink 아키텍처 내 시스템 식별자(System ID/Component ID) 기반의 필터링

가장 고전적이고 확실한 MAVLink의 토폴로지 격리 방법은 패킷 헤더(Header) 수준에서 발신자(Sender)와 수신자(Target)의 식별 번호를 엄격히 통제하는 것이다.

1.1 식별자 할당과 송수신 메커니즘

모든 MAVLink 메시지는 sysid(System ID, 1~255)와 compid(Component ID, 1~255)라는 두 가지 메타데이터를 포함한다.

  • MAV_SYS_ID 파라미터는 편대 내 각 기체의 고유 번호(예: 드론 1호기는 1, 드론 2호기는 2 …)를 나타낸다.
  • 지상 관제 시스템(QGroundControl)은 통상 sysid=255, compid=190 등의 별도 공간을 점유한다.

PX4 펌웨어 내의 mavlink_receiver.cpp 모듈은 인터럽트를 통해 패킷을 수신하면 가장 먼저 이 식별자를 디코딩(Decoding)한다. 패킷 내의 타겟 식별자(target_system) 필드가 존재할 경우, 이 값이 자신의 MAV_SYS_ID와 일치하지 않거나 브로드캐스트 ID(0)가 아니면 메인 파서(Parser)를 거치지 않고 패킷을 즉각 폐기(Drop)한다.

결과적으로, 물리적 통신 채널(UDP 브로드캐스트 혹은 RF 다중접속)이 공유되어 다른 기체의 트래픽이 자신의 하드웨어로 흘러 들어오더라도, MAVLink 파싱 레이어에서 산술적인 조건문(if (target_sys == my_sys)) 하나로 어플리케이션 계층의 오염을 원천 차단한다.

2. ROS2/DDS 미들웨어의 논리적 격리: Domain ID

ROS2 생태계(Fast DDS, Cyclone DDS 기본)에서 1차적인 격리 장벽(Isolation Barrier)은 ROS_DOMAIN_ID 환경 변수를 통한 도메인 분리이다.

2.1 도메인 식별자를 이용한 멀티캐스트 포트 다변화 메커니즘

DDS 규격에 따르면, 도메인 ID(0 ~ 232)는 각 노드가 바인딩(Binding)하는 UDP 멀티캐스트 포트(Multicast Port) 번호를 결정하는 수학적 시드(Seed)로 작용한다.
예를 들어, Default Domain ID인 0번을 사용하면 탐색 멀티캐스트 포트는 대략 7400 번에 할당되지만, Domain ID를 1번으로 올리면 포트가 7650 (도메인 게인 250 가정)으로 시프트(Shift)된다.

따라서 서로 다른 도메인 ID를 부여받은 기체 군(Group)은 소켓 레벨에서 아예 포트가 다르므로 운영체제(OS) 네트워킹 커널 스택 수준에서 서로의 탐색 트래픽 자체를 보지 못하게 되어 완벽한 다중에이전트 격리가 달성된다.
(상세한 멀티캐스트 포트 계산 알고리즘은 후속 61.5.4.1절의 IGMP 스누핑 편에서 다룬다.)

3. 계층적 네임스페이스(Namespace) 래핑(Wrapping) 아키텍처

로컬 서브넷 상의 모든 에이전트가 단일 GCS(ROS2 기반 중앙 관제탑)에 데이터를 쏟아부어야 하거나, 드론 간 충돌 회피(Swarm Avoidance)를 위해 상호 위치 정보를 교환해야 하는 “개방형 군집 환경(Open Swarm)“이라면, 도메인 ID를 분리하여 서로 소통할 수 없게 만드는 것은 해결책이 될 수 없다. 이때 등장하는 필연적인 기술이 **네임스페이스(Namespace) 래핑(Wrapping)**이다.

3.1 토픽 이름 충돌(Topic Name Collision)의 역학

PX4 내부 uORB 토픽(예: fmu/out/vehicle_local_position)이 Micro XRCE-DDS 에이전트를 거쳐 ROS2 네트워크로 발행(Publishing)될 때, 별도의 조작을 가하지 않으면 모든 100대의 기체가 정확히 같은 이름의 토픽으로 데이터를 쏟아낸다. GCS 입장에서는 이 데이터가 1호기 것인지 100호기 것인지 구분할 방법이 없어, 구독(Subscription) 노드의 콜백(Callback) 함수 내 버퍼가 심각하게 오염된다.

3.2 글로벌 토픽 프리픽스(Prefix) 래핑 연산 알고리즘

이를 해결하기 위해 런타임(Runtime) 시점에 각 기체의 고유 식별 번호(MAV_SYS_ID 또는 ROS2 호스트명)를 추출하여, DDS 발행자(Publisher)를 초기화하는 트리에 프리픽스(Prefix)를 덧씌우는(Wrapping) 계층화 작업이 수행된다.

    B -->|SYS_ID = 1<br/>Namespace Wrapping| C1[ROS2 Topic<br/>'/drone1/fmu/out/vehicle_local_position']

    C1 --> D{Swarm GCS / 중앙 모니터링 ROS Node}
    C2 --> D
    C3 --> D
    
    D -.->|동적 정규식 필터링| E[개별 에이전트 상태 테이블 업데이트]

리눅스 컴패니언 컴퓨터 상에서 ROS2 에이전트를 Launch 파일이나 스크립트로 기동할 때, Push Namespace 옵션을 통하여 해당 노드가 발행하는 모든 토픽과 서비스의 상위 디렉터리를 /<에이전트_명칭>으로 치환한다.
이렇게 계층형(Tree) 디렉토리 구조로 분리된 토픽명 공간(Namespace) 내에서는 전역적 문자열(String) 충돌이 원천적으로 발생할 수 없다.

또한 GCS 서버는 ROS2가 제공하는 정규식 구독 기능이나, 노드 생성 콜백 이벤트를 활용하여 /<새로운_명칭>/fmu/out/vehicle_local_position가 감지될 때마다 내부 C++ 객체 맵(Map) 컨테이너에 새로운 군집 에이전트를 동적으로 인스턴스화하여 격리된 관리 풀(Pool)을 구축한다. 이로써 아무리 많은 드론이 추가되더라도 데이터 버퍼 오염이 전혀 없는 확장 가능한 시스템 식별 모델이 완성된다.
