51.1.3. Return 유한 상태 머신(FSM)의 전이 행렬(Transition Matrix) 및 이산 시간(Discrete Time) 이벤트 트리거

1. 개요 및 아키텍처 이해 (Architecture Understanding)

PX4-Autopilot(펌웨어 v1.14 기준)에서 복귀(Return, RTL) 모드는 기체가 예기치 않은 데이터 링크 상실(Data Link Loss) 상태나 사용자의 명시적인 명령에 의해 안전하게 이륙 지점 또는 지정된 랠리 포인트(Rally Point)로 돌아오는 핵심적인 페일세이프(Failsafe) 기능이다. 이 과정은 연속적인 시간(Continuous Time) 영역에서의 동역학적 기동과 이산 시간(Discrete Time) 단위로 발생하는 이벤트 처리 결과의 결합으로 유도된다. PX4는 이러한 상태의 안정적인 전이를 보장하기 위해 유한 상태 머신(Finite State Machine, FSM) 아키텍처를 적용하여, 각 비행 단계를 수학적인 전이 행렬(Transition Matrix) 관점에서 엄격하게 통제한다. 특정 조건 즉, 이벤트 트리거(Event Trigger)가 만족될 때만 다음 상태로 넘어가도록 설계되어 있어 강건한 비행 안전성을 보장한다.

2. 유한 상태 머신(FSM) 모델 및 전이 행렬(Transition Matrix)

복귀 모드의 유한 상태 머신은 고도 확보, 이동, 착륙 등을 위해 크게 상승(Climb), 순항(Transit), 체공(Loiter), 하강(Descend), 착륙(Land)의 5가지 핵심 단계로 구분된다. 상태 천이는 이산 시간에서 발생하는 특정한 이벤트(예: 목표 고도 도달, 목표 위치 도달 등)에 의해 트리거(Trigger)된다. 이를 수식으로 나타내면 다음과 같다.

상태 벡터를 S = [s_1, s_2, s_3, s_4, s_5]^T로 정의할 때, 전이 행렬 T는 각 상태 간의 전이 조건을 포함한다. 다음 상태 S_{t+1}은 모델 관점에서 S_{t+1} = T \cdot S_t 수식의 형태로 전이되며, 실제 비행 제어기(Flight Controller) 코드 구현에서는 불리언(Boolean) 기반의 논리 게이트 파이프라인을 거쳐 행렬곱의 논리적 동치 형태로 상태가 업데이트된다.

2.1 메인 상태 전이 로직 도식화

다음은 복귀(Return) 상태 머신의 전이 과정을 나타낸 다이어그램이다.

stateDiagram-v2
    [*] --> Climb : Return 명령 수신 (이벤트 발생)
    Climb --> Transit : 목표 복귀 고도(Return Altitude) 도달
    Transit --> Loiter : 복귀 목적지(Home/Rally Point) 상공 반경 내 진입
    Loiter --> Descend : 체공 지연 시간(Delay Time) 이산 시간 도과
    Descend --> Land : 지면 접근 및 접촉 (Touchdown) 예상 한계 진입
    Land --> [*] : 시동 해제(Disarm) 완료
    
    Climb --> Descend : 치명적 에러 발생 및 상승 불가 시 비상 하강
    Transit --> Descend : 비상 착륙 조건 충족 (배터리 고갈 임박 등)

3. 이산 시간(Discrete Time) 이벤트 트리거 매커니즘

이벤트 트리거 매커니즘은 일차적으로 내측 비행 제어(Flight Control) 루프의 주기적 업데이트율(통상 250Hz~400Hz)에 의존하나, 내비게이터(Navigator) 모듈은 보통 10Hz~50Hz 단위의 이산 시간 스텝(Discrete Time Step)을 기준으로 상태 전이를 비동기적으로 평가한다.

  • 상승 조건(Climb Event): 현재 고도 h가 파라미터 RTL_RETURN_ALT보다 낮은 경우 상승 성분의 제어 벡터를 생성한다.
  • 순항 조건(Transit Event): h \ge \text{RTL\_RETURN\_ALT} - \epsilon (\epsilon은 위치 추정 허용 오차)를 만족하는 이산 시간에서 수평 위치 제어기(Position Controller)를 활성화시켜 전방으로 기동한다.
  • 하강 조건(Descend Event): 위치 반경 오차 \vert \Delta x \vert < \text{RTL\_ACCEPTED\_RAD}를 만족하는 이산 시간 인덱스 k에서 체공 이후, 하강 제어기 갱신 트리거 신호를 발생시킨다.

4. 제어 기술 및 센서 이해 (Control Technology & Sensor Understanding)

전이 행렬에 의해 FSM 상태 전환이 활성화될 때마다, 하위 계층 제어 루프(Control Loop)에 할당되는 목표 제어량(Setpoint)이 동적으로 재계산된다.

  • 제어 기술: 위치 제어(Position Control) 모듈은 캐스케이드(Cascade) P-PID 제어 구조로 구현되어 있다. FSM이 Transit 상태일 경우 일정한 수평 반환 속도 파라미터(RTL_SPEED)에 맞춘 저크 제한(Jerk-limited) 궤적을 셍성한다. 전이 순간 기체의 자세와 속도에 불연속성 충격(Bump)이 발생하지 않도록 궤적의 1차 및 2차 미분값에 부드러운 스무딩(Smoothing) 프로파일 필터링이 적용된다.
  • 센서 이해: 이산 시간 이벤트 트리거의 무결성은 센서 융합 모듈인 확장 칼만 필터(Extended Kalman Filter, EKF2)의 상태 추정치 안정성에 크게 기대고 있다. 특히 GPS(GNSS) 기반의 3차원 위치(3D Position) 분산과 기압계(Barometer) 및 VIO 기반 상대 고도(Relative Altitude) 데이터의 공분산(Covariance)이 지나치게 커지면, FSM은 전이를 일시 보류하거나 즉각적인 센서 기반 페일세이프(Sensor Failsafe) 모드로 강제 개입하여 비정상 거동을 차단한다.

5. Ardupilot 대비 차이 (Comparison with Ardupilot)

오픈소스 비행 제어 소프트웨어의 양대 산맥인 Ardupilot과 PX4-Autopilot은 복귀 모드 내 FSM을 구현하는 아키텍처 철학에 뚜렷한 차이를 보인다.

  • PX4-Autopilot: PX4의 내비게이터(Navigator) 백엔드 내 클래스(RTL)는 철저한 객체 지향 및 상태 패턴(State Pattern) 설계 방식을 따른다. 각 상태(Climb, Transit, Loiter, Descend) 진입(OnEnter), 유지(OnActive), 종료(OnExit) 처리가 분리되어 스크립팅 방식 없이도 이산 시간에 맞춰 정형화되어 순차 호출되므로 상태 전이 경로(Transition Pathway)가 매우 직관적이다.
  • Ardupilot: Ardupilot의 mode_rtl.cpp 등은 오랜 역사 동안 실전 피드백이 누적됨에 따라 거대한 제어 흐름(Control Flow) 지향적 분기를 통해 상태를 관리한다. 조건문 기반의 상태 간 결합도가 높아 상태 전이 행렬 형태의 독립적 관리보다는, 연속된 루프 안에서 분기문이 직접 제어 명령을 호출하는 절차적 프로그래밍(Procedural Programming) 성격이 더 짙다.

6. 소스 코드 이해 (Source Code Understanding)

소스 코드 트리 src/modules/navigator/rtl.cpp 구조를 살펴보면, 내부에서 상태 전이 행렬 제어기 역할을 수행하는 주기적 평가 함수 set_rtl_item()을 확인할 수 있다.

// PX4-Autopilot RTL State Machine 구현의 개념적 추상화 구조
void RTL::on_activation() {
    _rtl_state = RTL_STATE_CLIMB; // 초기 활성화 시 상승 트리거 삽입
}

void RTL::on_active() {
    switch (_rtl_state) {
        case RTL_STATE_CLIMB:
            // 이산 시간 데이터에서 최소 고도 도달 판단
            if (_navigator->get_altitude() >= _return_alt) {
                _rtl_state = RTL_STATE_TRANSIT; // 상태 전이 (Transition)
            }
            break;
            
        case RTL_STATE_TRANSIT:
            // 도달 반경 내 진입에 따른 FSM 전환
            if (_navigator->get_position_xy_error() < _accepted_radius) {
                _rtl_state = RTL_STATE_LOITER;
            }
            break;
            
        case RTL_STATE_LOITER:
            // 지정된 시간 대기 후 하강 (이산 시간 이벤트 딜레이)
            if (hrt_elapsed_time(&_loiter_start_time) > _land_delay) {
                _rtl_state = RTL_STATE_DESCEND;
            }
            break;
    }
}

위 코드는 FSM이 단일 주기 프로세스 내에서 어떻게 각 단계별 조건을 확인하고 안전하게 다음 페이즈로 이동(Transition)하는지 잘 보여준다.

7. MAVLink 및 ROS2 연동 이해 (MAVLink and ROS2 Integration)

지상 관제국 애플리케이션(QGroundControl v4.3 기준) 또는 ROS2 기반 자율 제어 노드에서, 복귀 모드의 내부 상태 전이 양상을 실시간으로 파악하기 위해서는 특정 계층의 메시징 처리가 필수적이다.

  • MAVLink 이해: PX4는 상태가 전이되는 순간 MAVLink의 HEARTBEAT 메시지 내 custom_mode 필드를 송출하여 현재 주 비행 모드가 RTL임을 QGroundControl에 알린다. 더 나아가, NAV_CONTROLLER_OUTPUT 패킷을 활용하여 기체와 웨이포인트(Return 도착 지점) 간의 교차 오차(Crosstrack Error)와 목표 거리를 전송한다. 또한 상태 전이 시 사람이 읽을 수 있는 형태로 STATUSTEXT 메시지("RTL: climb to 50m" 등)를 발행하여 관제 화면상 알림 패널에 표시한다.
  • ROS2 연동 이해: 마이크로 XRCE-DDS(uXRCE-DDS) 브릿지를 경유하여 PX4의 내부 uORB 주제(Topic)가 ROS2 토픽으로 노출된다. ROS2를 운용하는 임베디드 컴퓨터(Companion Computer) 환경에서는 px4_msgs::msg::VehicleCommand 인터페이스 기반으로 복귀 모드 진입을 강제할 수 있다. 더 중요한 것은, 노드가 px4_msgs::msg::NavigatorMissionItem 등을 서브스크라이빙(Subscribing)하여 FSM이 현재 TRANSIT인지 DESCEND인지 확인하며, 필요한 경우 Offboard 제어권을 통해 RTL 모드를 선점(Preemption)하고 객체 회피 기동에 직접 개입할 수 있다는 점이다.

8. 설정 및 구성 (Configuration and Setup)

Return 유한 상태 머신의 행렬망 내에서 행동의 양식과 전이 속도를 지배하는 펌웨어 파라미터 그룹은 다음과 같다.

  • RTL_RETURN_ALT: 기체가 장애물을 회피하여 복귀하기 위해 보장해야 하는 최소한의 안전 고도. 상승(Climb) 단계의 전이를 결정짓는 주요 임계값이다.
  • RTL_DESCEND_ALT: 복귀 목적지에 도달하고 대기(Loiter) 한 후 하강(Descend) 단계를 개시할 때 초기에 목표로 삼는 하향 고도값.
  • RTL_LAND_DELAY: 최종 착륙(Land) 모드로 진입하기에 앞서, 착륙 지점의 안정성을 파악하도록 체공하며 대기하는 이산 시간 설정치.
