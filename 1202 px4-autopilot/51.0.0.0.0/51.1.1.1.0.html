51.1.1.1. 커맨더(Commander) 모듈과 내비게이터 모듈 간의 비동기적 상태 전이(Asynchronous State Transition) 요청 메커니즘

PX4-Autopilot 아키텍처의 핵심 원칙 중 하나는 각 소프트웨어 구동 모듈 간의 결합도를 낮추는 것이다. 복귀(Return) 명령이 인가될 때 시스템 상태를 관장하는 커맨더(Commander) 모듈과 실제 비행 경로를 생성하는 내비게이터(Navigator) 모듈은 직접적인 함수 호출(Synchronous Call) 대신, uORB(Micro Object Request Broker) 메시징 통신을 이용한 비동기적(Asynchronous) 상태 전이 메커니즘을 따른다.

1. 비동기 통신의 필요성과 아키텍처 개요

Ardupilot의 메인 스레드 기반 동기적 처리 방식과 달리, PX4는 커맨더와 내비게이터를 서로 다른 데몬 프로세스(Daemon Process) 혹은 워크 큐(Work Queue) 위에서 실행시킨다. 만약 GCS(QGroundControl)로부터 복귀 명령이 들어왔을 때 커맨더가 내비게이터의 상태 전이 처리가 끝날 때까지 블로킹(Blocking)된다면, 시스템 전체의 주 제어 루프가 지연(Latency)되어 치명적인 비행 불안정을 초래할 수 있다.

따라서 커맨더는 상태 전이 ’요청’만을 uORB를 통해 퍼블리시(Publish)하고, 내비게이터는 자신의 갱신 주기(Tick)에 맞추어 상태 전이를 ’수행’하는 이벤트 주도형(Event-Driven) 비동기 파이프라인을 갖는다.

2. 상태 전이 시퀀스 분석 (State Transition Sequence)

지상 관제 시스템에서 복귀 버튼을 클릭하는 순간부터 기체의 실질적인 궤적이 변경되기까지의 흐름은 다음과 같다.

sequenceDiagram
    participant GCS as QGroundControl
    participant MAVLink as MAVLink / MAVSDK
    participant Cmd as Commander
    participant uORB
    participant Nav as Navigator

    GCS->>MAVLink: MAV_CMD_NAV_RETURN_TO_LAUNCH
    MAVLink->>uORB: vehicle_command 발행
    uORB-->>Cmd: vehicle_command 수신 이벤트
    Cmd->>Cmd: 활성화 조건 검증 (GPS 수신, Arming 상태 파악 등)
    Cmd->>uORB: vehicle_status (nav_state = AUTO_RTL) 발행
    Cmd->>MAVLink: COMMAND_ACK (전이 수락 회신)
    MAVLink->>GCS: 명령 접수 확인 표시
    
    uORB-->>Nav: vehicle_status 변경 감지
    Nav->>Nav: 이전 Task 종료 및 Return Task 초기화
    Nav->>uORB: trajectory_setpoint 발행 시작

2.1 단계: 커맨더의 명령어 수신 및 조건 검사

커맨더 모듈 내 로직(src/modules/commander/Commander.cpp)은 vehicle_command 토픽을 폴링(Polling)한다. MAV_CMD_NAV_RETURN_TO_LAUNCH 커맨드를 파싱한 뒤, 커맨더는 다음을 검증한다.

  • 기체의 전역 위치(Global Position) 유효성 여부
  • 시동(Arming)이 걸린 상태인지 여부
    위 조건이 충족되면 모드 전환 트랜잭션이 활성화된다.

2.2 단계: uORB 기반 vehicle_status 전이 요청

커맨더는 통과된 전환 요청을 시스템 전역에 전파하기 위해 vehicle_status uORB 메시지의 nav_state 필드를 명시적인 복귀 상태 변수인 NAVIGATION_STATE_AUTO_RTL로 변환하여 발행한다.

2.3 단계: 내비게이터의 비동기 수신 및 Task 전이

내비게이터 모듈은 메인 실행 사이클 주기(예: 50Hz)마다 vehicle_status 토픽의 업데이트 여부를 관찰(Poll)한다.
상태가 NAVIGATION_STATE_AUTO_RTL로 변경됨을 확인하면, 기존에 실행 중이던 비행 작업(예: Mission Task 또는 Loiter Task)의 실행을 중단하고 Return 인스턴스의 on_activation() 메서드를 호출한다. 이때 비로소 기체의 실제 물리적 궤적 변화를 위한 trajectory_setpoint 계산이 시작된다.

3. 동기화 지연(Synchronization Latency)의 안전 메커니즘 보완

비동기 트랜잭션의 특성상 커맨더가 명령을 발행한 시점과 실제 내비게이터가 작동을 개시하는 시점 사이에는 미세한 마이크로초 단위의 시차가 존재한다. 이를 보완하기 위해 커맨더는 명령이 접수된 즉시 vehicle_command_ack 메시지를 통해 GCS로 회신한다. 이는 무선 링크 상에서의 타임아웃(Timeout) 및 불필요한 재전송(Retry) 트래픽을 예방하여 패킷 지연률을 극도로 낮추는 관제 시스템 최적화의 핵심 기반이 된다.
