51.3 GCS-Autopilot 간 MAVLink 복귀 제어 트랜잭션 및 원격 측정(Telemetry) 비동기 동기화

비행 중인 기체(PX4-Autopilot)가 지상 관제 시스템(GCS, 예: QGroundControl)으로부터 복귀(Return/RTL) 명령을 수신하고, 이를 수행하며 자신의 현재 상태를 GCS에 보고하는 일련의 과정은 MAVLink 프로토콜을 통해 이루어진다. 이 과정은 단방향 통신이 아니라, 제어 명령의 트랜잭션(Transaction) 처리와 원격 측정 데이터(Telemetry)의 비동기적 동기화(Asynchronous Synchronization)가 정교하게 맞물려 작동하는 분산 제어 시스템의 핵심이다.

이 장에서는 PX4(v1.14.0 이상)와 QGroundControl(v4.3 이상)이 MAVLink를 통해 복귀 명령을 주고받는 핸드셰이크(Handshake) 과정, 비동기 상태 동기화 매커니즘, 그리고 MAVLink 통신 손실 발생 시의 예외 처리 아키텍처를 심층적으로 분석하라.

1. MAVLink 명령 트랜잭션 구조와 COMMAND_LONG

GCS에서 사용자 인터페이스(UI)를 통해 복귀 버튼을 클릭하면, QGC 백엔드(C++)의 Vehicle 매니저 컴포넌트가 MAVLink COMMAND_LONG (메시지 ID: #76) 또는 COMMAND_INT 패킷을 활용하여 Autopilot에 MAV_CMD_NAV_RETURN_TO_LAUNCH (ID: 20) 제어 명령을 비동기 전송한다.

비동기 전송 환경에서는 패킷의 변조, 지연, 유실 가능성이 항상 존재하므로 PX4는 이를 수신한 즉시 수신 확인 응답인 COMMAND_ACK (#77) 메시지를 전송하여 트랜잭션을 종결지어야 한다.

    GCS (QGC)->>Autopilot (PX4): COMMAND_LONG (MAV_CMD_NAV_RETURN_TO_LAUNCH)

2. PX4-Autopilot의 커맨더(Commander) 모듈 처리 흐름

명령 트랜잭션이 성공하기 위해서는 PX4 내부의 커맨더(Commander) 로직이 명령을 유효한 것으로 수용해야 한다. src/modules/commander/Commander.cpp 와 연관된 MAVLink 커맨드 파싱 파일에서는 MAV_CMD_NAV_RETURN_TO_LAUNCH가 수신되었을 때 시스템 상태를 바탕으로 다음의 검증을 거친다.

  • 글로벌 포지션 유효성(Global Position Validity): 복귀(RTL) 궤적을 계획하기 위해서는 GPS나 VIO 등을 통해 Home Position과 현재 Position 데이터가 확립되어 있어야 한다. EKF2가 local_position과 global_position 유효 플래그를 참(True)으로 보고해야만 MAV_RESULT_ACCEPTED 응답을 낼 수 있다.
  • Flight Task 스위칭: 수용 조건이 충족되면 PX4는 내부 상태 머신을 변경하고(Mode 변경), FlightTaskAutoRtl 클래스로 제어 권한을 이양한다.

만약 배터리 전압이 너무 낮거나 위치 정보가 유실된 상태라면, PX4는 COMMAND_ACK 트랜잭션을 통해 MAV_RESULT_TEMPORARILY_REJECTED 또는 MAV_RESULT_DENIED를 GCS에 반환하고 비동기 제어 트랜잭션을 실패로 종결한다.

3. 원격 측정(Telemetry)의 비동기 동기화 구조

PX4 시스템이 Return 모드로 변경되었다 하더라도 GCS 측 UI에 이를 실시간 정합하기 위해 빈번한 동기화가 필요하다.
QGroundControl 측에서는 PX4가 퍼블리싱(Publishing)하는 다음의 원격 측정 패킷들을 비동기적으로 수신하여 GUI 상태 머신을 업데이트한다:

  1. HEARTBEAT (#0): 시스템의 base_mode 와 custom_mode 필드를 파싱하여 현재 기체의 비행 모드가 ‘RTL’ 혹은 ’Return’으로 정상적으로 동기화되었는지 확인한다(약 1Hz 퍼블리싱).
  2. NAV_CONTROLLER_OUTPUT (#62): 남은 웨이포인트(Home)까지의 거리(wp_dist), 타겟 고도 오차, 속도 오차 등을 GCS의 비행 화면 (Fly View)에 오버레이(Overlay)하기 위해 전송된다. Return 궤적이 갱신될 때마다 비동기적으로 QGC에 반영된다.
  3. STATUSTEXT (#253): RTL 모드로 진입하거나, 근접 과정에서 단계(Climb -> Transit -> Land)가 바뀔 때 사람이 읽을 수 있는 텍스트 형태(예: “RTL: climb to 30m”)로 알림이 전달된다.

이러한 데이터들은 UDP 통신망(보통 포트 14550) 위에서 전송되며 비연결성(Connectionless) 특성을 띠므로, GCS는 패킷 누락에 대비해 타임아웃과 이전 상태 유지(Caching) 아키텍처를 보유하고 있다.

4. Ardupilot과의 관제 철학 및 동기화 차이점

Ardupilot은 전통적으로 데이터의 양이 많은 SYS_STATUS나 고유(Custom) 패킷들을 통해 복합적인 피드백을 제공하는데, Return/RTL 상태 변이에 대해 비교적 넓은 관용을 둔다.
반면 PX4의 MAVLink 마이크로서비스 관제 철학은 철저한 모드 의존적(Mode-dependent) 스트리밍을 지향한다. 즉, PX4는 현재 활성화된 모드에 따라 MAVLink 스트리밍 레이트를 uORB 계층에서 동적으로 조율한다. RTL 모드가 시작되면 홈 위치 추론 패킷(HOME_POSITION)이나 컨트롤러 출력(NAV_CONTROLLER_OUTPUT)의 우선순위를 높여 MAVLink 트래픽 혼잡도 속에서도 필수적인 동기화 정보가 GCS에 먼저 도달할 수 있도록 대역폭 분배(Bandwidth Allocation)를 제어한다.

5. MAVLink 데이터 링크 소실(Data Link Loss)에 대한 예방

통신 비동기 동기화 중 가장 위험한 상황은 명령 자체는 전송되었으나, 이로 인해 GCS와 PX4 간 MAVLink Data Link가 단절되는 경우다.

  • PX4 파라미터 COM_DL_LOSS_T: 해당 시간(초) 동안 HEARTBEAT를 받지 못하면 통신 두절로 간주하여 독자적인 안전장치(Failsafe)인 RTL 모드로 진입하게 한다.
  • 만약 GCS가 복귀 명령 트랜잭션을 보냈는데 COMMAND_ACK을 받지 못한 채 통신이 끊어지면, QGC는 수 회(Retry Logic) 재전송을 시도하다가 연결 없음(Disconnected) 아이콘을 띄운다. 반면 기체는 이미 명령을 수신하여 복귀 중이거나, COM_DL_LOSS_T가 만료되어 자체적으로 복귀 궤적을 생성해 되돌아오고 있을 것이다.

결론적으로 GCS-Autopilot 간 MAVLink 복귀 제어 트랜잭션은 단일 패킷의 응답에 의존하지 않고, 상태 추정 및 원격측정 데몬들의 다중 다단계 피드백(Multi-stage Feedback) 통신을 통해 높은 강건성(Robustness)을 확보하게 된다.
