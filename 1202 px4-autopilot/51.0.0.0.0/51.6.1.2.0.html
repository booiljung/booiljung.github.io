51.6.1.2 GNSS 혁신 일관성(Innovation Consistency) 실패 시 광학 흐름(Optical Flow) 또는 대기 속도(Airspeed) 기반 추측 항법(Dead Reckoning)으로의 무충돌 상태 강등(Bumpless Degradation)

무인항공기가 임무를 수행하는 동안 GNSS(위성 항법 시스템) 신호는 다중 경로(Multipath), 전파 방해(Jamming), 기만(Spoofing) 등 다양한 외부 요인에 의해 왜곡될 수 있다. PX4-Autopilot(Firmware 버전 1.14 이상 기준)은 강력한 확장 칼만 필터(Extended Kalman Filter, EKF2)를 통해 이러한 이상 신호를 감지하고, 상태 추정의 신뢰성을 유지하기 위해 혁신 일관성(Innovation Consistency) 검사를 수행한다. GNSS 데이터의 혁신(예측값과 측정값의 차이)이 허용 임계치를 초과하여 실패로 판정될 경우, 기체는 즉각 광학 흐름(Optical Flow)이나 대기 속도(Airspeed) 센서를 활용한 추측 항법(Dead Reckoning) 상태로 전환된다. 이때 비행 궤적과 제어 명령에 급격한 튐(Jump)이 발생하지 않도록 하는 조치를 **무충돌 상태 강등(Bumpless Degradation)**이라고 한다.

1. 아키텍처 및 제어 기술의 이해

PX4-Autopilot의 ekf2 모듈(Estimator)은 측정 데이터를 융합할 때 각 센서 데이터가 필터의 현재 추정 상태와 일치하는지 확인하는 Innovation Gate 메커니즘을 사용한다.

1.1 혁신 일관성(Innovation Consistency) 검증

EKF에서는 매 타임스텝마다 센서로부터 들어오는 실제 측정값(Z)과 필터 모델에서 예측한 측정값(H\hat{X})의 차이인 **혁신(Innovation, \tilde{y})**을 계산한다.
\tilde{y} = Z - H\hat{X}
혁신 일관성 평가는 정규화된 혁신 제곱(Normalized Innovation Squared, NIS)인 \tilde{y}^T S^{-1} \tilde{y} 값이 카이제곱 분포 기반의 임계치(Gate)를 초과하는지를 통해 이루어진다. 여기서 S는 혁신 공분산(Innovation Covariance)이다. GNSS 데이터의 혁신값이 설정된 게이트(EKF2_GPS_V_GATE, EKF2_GPS_P_GATE)를 지속적으로 벗어나면, EKF2는 해당 GNSS 데이터가 신뢰성을 잃었다고 판단하고 융합을 중단(Reject)한다.

1.2 무충돌 상태 강등(Bumpless Degradation) 메커니즘

GNSS 융합이 중단되면 기체는 대체 센서 융합 모드로 강등(Degradation)된다.

  • 멀티로터(Multirotor): 하방 카메라 센서인 광학 흐름(Optical Flow) 데이터와 거리 측정 센서(Range Finder)를 결합하여 지면 대비 상대 속도 및 위치 상태를 유지한다.
  • 고정익(Fixed-wing): 피토관 등 대기 속도(Airspeed) 센서를 활용하여 진대기속도(True Airspeed, TAS)를 측정하고, 이전에 추정된 바람 상태(Wind State)를 결합하여 추측 항법(Dead Reckoning)으로 위치 및 속도를 갱신한다.

이 전환 과정에서 상태 변수(위치, 속도)의 값이 급변하여 기체가 급기동하는 것을 방지하기 위해 필터 공분산(Covariance) 매트릭스를 재설정(Reset)하고 상태 오프셋(State Offset)을 맞추어 부드럽게 천이하는 “무충돌(Bumpless)” 기법을 적용한다.

2. 상태 천이 동작 흐름 (Mermaid 도표)

다음은 GNSS 실패 감지부터 추측 항법으로 전환하기까지의 EKF 상태 천이 흐름도이다.

    B -- 정상 --> A
    B -- 허용치 초과 / 실패 --> C[GNSS 융합 타임아웃/중단 탐지]
    C --> D{비행체 타입 및 <br> 가용 센서 확인}
    D -- 멀티로터 / Optical Flow 가용 --> E[Optical Flow + Range Finder <br> 기반 속도 추정 모드 전환]
    D -- 고정익 / Airspeed 가용 --> F[Airspeed + Wind States <br> 기반 추측 항법 전환]
    D -- 가용 센서 없음 --> G[고도 유지(Altitude/Stabilized) <br> 모드로 Fallback 요청]
    E --> H[필터 상태 및 공분산 리셋 <br> Bumpless Degradation 수행]
    F --> H
    H --> I[GCS MAVLink 경고 발생 <br> 'Failsafe: Auto mode disabled']

3. 센서 이해

  • 광학 흐름 센서(Optical Flow): 지면의 텍스처(Texture) 이동을 분석하여 기체의 2D 각속도를 측정한다. 거리 측정 센서의 고도 데이터와 결합하면 미터(m/s) 단위의 지면 상대 속도 벡터를 계산할 수 있다. GNSS가 차단된 상태에서 호버링(Hovering)이나 저속 정밀 비행을 유지하는 핵심 센서이다.
  • 대기 속도 센서(Airspeed Sensor): 동압(Dynamic Pressure)을 측정하여 기체를 스쳐 지나가는 공기의 속도를 측정한다. 추측 항법 모델에서 Ground Speed = Airspeed + Wind speed라는 벡터 합을 통해 위치를 갱신해 나가며, GNSS 실패 시 고정익 기체의 유일한 수평 생존 수단이 된다.

4. 소스 코드 분석 (ECL - EKF2)

PX4 펌웨어 소스에서 이 과정은 주로 src/modules/ekf2/ 디렉터리의 Estimator and Control Library(ECL)에서 처리된다. control.cpp 및 velocity_pos_fusion.cpp를 중심으로 살펴보라.

// src/lib/ecl/EKF/control.cpp 내부의 GNSS 검증 및 로직 축약 예시

void Ekf::controlFusionModes()
{
    // ...
    // GNSS 혁신 일관성 타임아웃 및 실패 감지
    if (_time_last_imu - _time_last_gps_yaw_fuse > _params.gps_timeout_max) {
        _control_status.flags.gps_yaw = false;
        // 위성 항법 실패 플래그 활성화
    }

    // 상태 강등 조건: GNSS가 실패하고 Optical Flow 데이터가 유효한 경우
    if (!_control_status.flags.gps && _control_status.flags.opt_flow) {
        // Optical Flow를 메인 위치/속도 센서로 전환 (Bumpless)
        resetVelocityToVision();
        resetPositionToVision();
        _information_events.flags.reset_pos_to_vision = true;
    }
    // 고정익 추측 항법 조건 (대기 속도 센서 활용)
    else if (!_control_status.flags.gps && _control_status.flags.fixed_wing) {
        // Airspeed와 기존 Wind 추정치를 통한 추측 항법(Dead Reckoning) 지속
        startDeadReckoning();
    }
}

소스 코드 내부의 resetVelocityTo... 함수들은 위치와 속도 상태를 오프셋 처리하여 제어기(Position Controller) 단에서 위치/속도의 불연속적 도약으로 인한 갑작스러운 피치/롤 명령 출력을 억제한다. 즉 필터(Estimator)의 내부 원점을 이동시켜 연속적인 위치를 보장하는 것이 핵심이다.

5. 파라미터(Parameter) 설정 및 구성

이러한 강등 시스템이 의도한 대로 부드럽게 동작하려면 QGroundControl을 통해 다음 파라미터들을 올바르게 튜닝해야 한다.

  • EKF2_GPS_V_GATE / EKF2_GPS_P_GATE: GNSS 속도 및 위치 혁신 일관성 게이트 값. 게이트 크기가 너무 작으면 정상적인 약전계 상황에서도 쉽게 타임아웃이 발생하며, 너무 크면 스푸핑이나 오차량이 큰 데이터까지 융합하여 기체가 요동칠 수 있다. (기본값: 속도 5.0, 위치 5.0 가량)
  • EKF2_OF_GATE: 광학 흐름 센서 데이터의 일관성 확인 게이트 임계값.
  • EKF2_TAS_GATE: 진대기속도 센서의 혁신 임계값. 고정익 추측 항법 상태의 정확도에 결정적 영향을 미친다.
  • COM_SPOOLUP_TIME 및 COM_POS_FS_DELAY: 포지션 제어 실패 감지 후 강등 페일세이프(Failsafe) 발동까지 견디는(Delay) 시간. 짧을수록 빠르게 경고하고 모드를 전환한다.

6. PX4와 Ardupilot의 차이점

  • PX4-Autopilot (ECL 기반): 단일 EKF 인스턴스 내에서 센서의 상태 편이를 추적하고, 특정 센서가 타임아웃되거나 게이트 검사에 실패하면 해당 센서의 융합을 끄고, 분산 공분산 매트릭스와 상태 원점 오프셋을 조절(Reset)하여 연속적(Bumpless) 제어 상태를 이어가는 데 중점을 둔다.
  • Ardupilot (EKF3 레인 스위칭 기반): 여러 가지 센서 조합(Core 또는 Lane)을 활용해 독립적인 EKF3 코어를 여러 개 동시에 병렬 구동하는 레인(Lane) 아키텍처를 자주 사용한다. Primary Lane의 GNSS 혁신 수치가 급증하면, Optical Flow나 다른 센서만을 참조하던 대기 중인 다른 정상 Lane으로 메인 제어 참조 권한 자체를 빠르게 절체(Lane Switch)하는 방식에 더 최적화되어 있다. PX4가 상태 조작에 관대하다면, Ardupilot은 병렬 다중 검증에 더 큰 비중을 둔 구조를 취한다.

7. MAVLink 관제와의 연계 및 Mode 제어

이 강등 조건이 발생하면 PX4 Commander 모듈은 즉각적으로 MAVLink 시스템 상태를 통해 GCS(QGroundControl)에 긴급 메시지를 타전한다.

  • 송신 메시지: STATUSTEXT (Severity: EMERGENCY 또는 CRITICAL)
  • 메시지 내용: Failsafe enabled: no global position 또는 Nav mode degraded: using dead reckoning.

이 경우 Commander의 유한 상태 머신(State Machine)은 Position 모드나 Mission 모드처럼 전 지구적 위치가 필요한 모드를 강제로 비활성화시키고, 기착 시스템에 따라 Altitude 모드 또는 옵티컬 플로우 기반의 Position Slow 모드로 내부 모드를 동적 강등 변환시킨다. 관제사는 이 알람을 수신하는 즉시 수동 제어권(Human-in-the-loop) 개입을 통해 기체의 안전을 확보해야 한다.
