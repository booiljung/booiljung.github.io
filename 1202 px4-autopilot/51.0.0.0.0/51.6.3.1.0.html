51.6.3.1. 수동 조종(Manual Control) 스틱 변위 입력 발생 시 Return 유한 상태 머신의 강제 선점 해제(Forced Preemption Release) 및 모드 롤백(Mode Rollback) 로직

자동 복귀(Return/RTL) 비행 중에 예기치 못한 비상 상황(예: 갑작스러운 유인 항공기 접근, GCS 모니터링 상의 경로 이탈 등)이 발생할 경우, 조종사는 즉각적인 개입을 통해 기체 제어권을 원상 복구해야 한다. PX4-Autopilot은 이러한 상황에서 조종기(RC) 스틱의 변위 입력을 감지하고, 기존에 실행 중이던 복귀 유한 상태 머신(FSM)을 강제로 선점 해제(Forced Preemption Release)한 뒤, 안전하게 수동 제어가 가능한 모드(예: Position 또는 Altitude 모드)로 전환하는 모드 롤백(Mode Rollback) 로직을 정밀하게 구현하고 있다.

본 절에서는 해당 오버라이드 절차가 PX4 내장 커맨더(Commander) 모듈과 비행 작업(Flight Task) 프레임워크 간에 어떻게 상호작용하며 상태를 천이(Transition)하는지 상세히 분석한다. (분석 기준: PX4-Autopilot v1.14 아키텍처)

1. 선점 해제(Preemption Release) 조건 및 트리거 로직

PX4에서는 조종사의 스틱 입력을 지속적으로 모니터링하여, 자동 모듈(Auto Mode)의 궤적 제어를 강제로 선점할지 판단한다. 오버라이드를 판단하는 주체는 주로 Commander 모듈 내부의 handle_command 및 상태 점검 루프이다.

1.1 입력 감지 및 파라미터 제약

RC 송신기 또는 QGroundControl의 가상 조이스틱을 통한 입력은 uORB 토픽인 manual_control_setpoint로 비행 제어 스택에 전달된다. 시스템은 롤(Roll), 피치(Pitch), 요(Yaw), 스로틀(Throttle) 채널 중 어느 하나라도 사전에 정의된 물리적 사각지대(Deadzone)를 벗어나면 इसे 유의미한 제어권 회수 시도로 간주한다.

  • COM_RC_OVERRIDE: RC 오버라이드 기능을 켤지 결정하는 핵심 파라미터. (0=비활성화, 1=활성화)
  • COM_RC_STICK_OV: 스틱의 변위량(%)이 이 임계값을 초과할 때만 오버라이드를 트리거하도록 설정하여, 미세한 노이즈나 조종사의 실수로 인한 의도치 않은 해제를 방지한다.

1.2 모드 롤백 메커니즘 도식화

입력이 임계값을 초과하는 순간, Commander 모듈은 내부 상태 머신 디스패처를 호출하여 모드 롤백(Rollback)을 명령한다.

이 다이어그램이 보여주듯, 스틱 변위가 트리거되면 시스템은 Global Position(GPS) 상태 유무를 판단하여 Position 모드로 롤백할지, Altitude 모드로 롤백할지를 지능적으로 결정한다. 이는 강제 선점 과정에서도 기체의 안전성을 최우선으로 확보하기 위함이다.

2. Flight Task 프레임워크 상의 상태 전이 아키텍처

PX4의 다중 콥터 위치 제어(Multicopter Position Control, MPC) 스택에서는 여러 비행 거동을 FlightTask라는 독립된 객체로 관리한다. 복귀 모드는 FlightTaskAutoMapper를 경유한 FlightTaskAuto 또는 FlightTaskAutoReturn 객체가 담당하고 있다.

2.1 커맨더(Commander)에 의한 Task 교체

모드 롤백이 결정되면, 커맨더(Commander) 모듈은 vehicle_command 토픽을 통해 모드 변경 명령(VEHICLE_CMD_DO_SET_MODE)을 발행하거나, 내부 navigation_state를 직접 NAVIGATION_STATE_POSCTL로 천이시킨다.

  1. MPC 모듈은 navigation_state가 변경되었음을 uORB로 감지한다.
  2. 현재 구동 중인 FlightTaskAutoReturn 객체의 실행을 즉각 중지(Preempt)하고 파기한다.
  3. 곧바로 휴먼-인-더-루프 처리에 적합한 FlightTaskManualPosition (Position 모드의 태스크) 객체를 생성하고 활성화(Activate)한다.

2.2 ArduPilot과의 제어 철학 비교

ArduPilot은 전통적으로 특정 모드 내에서 “스틱 믹싱(Stick Mixing)“이라는 방식으로 자동 궤적에 조종사의 스틱 변위를 더해주는 유연한 방식을 차용해 왔다. 예를 들어, RTL 모드 중 피치 스틱을 밀면 기속이 증가하거나 복귀 위치가 미세 조정될 수 있다.
반면 PX4-Autopilot은 상태 머신의 순수성과 의도성(Intent) 에 매우 엄격한 설계 철학을 가진다. 자동 모드는 시스템이 온전히 장악하며, 개입이 필요할 경우 모드 체계를 완전히 분리(Decoupling)하여 명시적인 “수동 전환(Rollback)“으로만 이를 허용한다. 이는 소프트웨어 아키텍처 관점에서 상태 추적(Traceability)을 극대화하고, 예측 불가능한 궤적 왜곡을 없앤다는 뚜렷한 이점이 있다.

강제 선점 로직의 실제 코드는 src/modules/commander/Commander.cpp 내부의 메인 루프에서 실행된다.

// 가상 코드 예시: 오버라이드 로직의 기본 형태
if (rc_updated && _param_com_rc_override.get() != 0) {
    bool stick_deflected_significantly = 
           (fabsf(manual.x) > _param_com_rc_stick_ov.get() || 
            fabsf(manual.y) > _param_com_rc_stick_ov.get() || ... );

    if (stick_deflected_significantly && is_auto_mode(internal_state.main_state)) {
        // Return 스케줄의 강제 선점 해제 및 모드 롤백 수행
        transition_status = main_state_transition(
                                internal_state, 
                                commander_state_s::MAIN_STATE_POSCTL, 
                                ... );
        // GCS 로 MAVLink 통보
        mavlink_log_info(&_mavlink_log_pub, "Overriding auto mode to POSCTL");
    }
}

이와 같이 PX4 제어 스택은 단일 이벤트 루프 내에서 조종기 권한을 시스템 지령치보다 최상위의 인터럽트 요소로 취급한다. 이를 통해 강제 선점 해제는 어떠한 지연시간(Latency) 병목 없이 즉각적으로 모드 롤백을 단행하게 되며, 이는 극한 상황에서 시스템 생존성을 보장하는 핵심 아키텍처로 작용한다.
