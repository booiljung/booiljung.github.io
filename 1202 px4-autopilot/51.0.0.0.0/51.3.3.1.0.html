51.3.3.1 COMMAND_ACK 메시지의 결과 코드(Result Code) 비동기 콜백(Asynchronous Callback) 함수 매핑 구조

1. 개요

PX4-Autopilot 및 지상 관제 시스템(GCS, 예: QGroundControl) 간의 MAVLink 통신에서 제어 명령(Command) 트랜잭션의 신뢰성을 보장하기 위해 COMMAND_ACK 메시지가 핵심적인 역할을 수행한다. 이 절에서는 명령 요청에 대한 결과 코드(Result Code)가 비동기적으로 수신되었을 때, 이를 적절한 비동기 콜백(Asynchronous Callback) 함수로 매핑(Mapping)하고 내부 상태를 업데이트하는 소프트웨어 아키텍처와 구조에 대해 매우 상세히 기술한다. 본 내용은 PX4-Autopilot 펌웨어(v1.14 이상) 및 QGroundControl(v4.3 이상)의 구조적 특성을 기반으로 설명한다.

2. MAVLink COMMAND_ACK 메시지와 Result Code의 이해

GCS나 동반 컴퓨터(Companion Computer)가 기체에 특정한 동작(예: 이륙, 모드 변경, 웨이포인트 이동 등)을 지시하기 위해 COMMAND_LONG 또는 COMMAND_INT 메시지를 전송하면, 수신측인 PX4-Autopilot은 해당 명령의 수용 및 실행 여부를 COMMAND_ACK 메시지로 회신한다.

이 회신 메시지 내에는 명령의 실행 결과를 나타내는 MAV_RESULT 열거형(Enum) 타입의 결과 코드가 포함되어 있으며, 주요 코드는 다음과 같이 정의되어 있다.

  • MAV_RESULT_ACCEPTED (0): 명령이 정상적으로 수용되어 실행되었거나 실행을 시작함.
  • MAV_RESULT_TEMPORARILY_REJECTED (1): 시스템의 현재 상태나 조건이 맞지 않아 명령이 일시적으로 거부됨(예: 기체가 공중에 있는 상태에서 이륙 명령 수신).
  • MAV_RESULT_DENIED (2): 명령이 영구적으로 거부됨(예: 관리자 권한 부족, 유효하지 않은 매개변수 값 포함).
  • MAV_RESULT_UNSUPPORTED (3): 수신측 시스템이 해당 명령 자체를 지원하지 않거나 구현되어 있지 않음.
  • MAV_RESULT_FAILED (4): 명령을 수용하여 실행을 시도했으나, 치명적인 오류나 하드웨어 문제로 인해 실행에 실패함.
  • MAV_RESULT_IN_PROGRESS (5): 명령이 수용되어 현재 장기적인 실행이 진행 중임을 알림. 이 경우 작업 완료 시 최종 결과가 담긴 별도의 COMMAND_ACK가 다시 전송된다.

3. 비동기 콜백(Asynchronous Callback) 함수의 필요성 및 설계 철학

네트워크 통신의 지연(Latency) 변동성이 큰 무선 원격 측정(Telemetry) 환경에서는 동기식(Synchronous) 블로킹(Blocking) 호출을 사용할 경우, 응답이 올 때까지 메인 스레드나 인터페이스UI가 멈추는 프리징(Freezing) 현상이 발생한다. 이는 시스템의 실시간 반응성을 심각하게 저해한다.

따라서 QGroundControl과 같은 프론트엔드 환경 및 PX4의 MAVLink 수신 모듈은 명령을 큐(Queue)에 적재하여 전송한 후 쓰레드를 즉시 반환(Return)하는 비동기식 이벤트 주도(Event-Driven) 아키텍처를 채택하고 있다. 명령 전송 시점에 결과 코드를 처리할 콜백 함수(Callback Function)를 등록하고, 실제 COMMAND_ACK 메시지가 도착(이벤트 발생)했을 때 사전에 등록된 해당 콜백 함수가 호출되도록 매핑하는 것이 이 구조의 핵심 설계 철학이다.

4. 매핑(Mapping) 아키텍처 구현 메커니즘

콜백 매핑 메커니즘은 요청자(Requester)가 여러 명일 수 있는 복잡한 시스템에서, 수신된 응답이 정확히 어느 요청에 대한 것인지를 식별하여 처리 상태를 라우팅(Routing)하는 역할을 한다.

  1. 명령 전송 및 콜백 객체 등록: MAVLink 명령을 전송할 때, 명령 식별자인 Command ID, 대상 시스템 및 컴포넌트 정보, 그리고 타임아웃(Timeout) 값과 재전송 시도 횟수(Retry Count)를 묶어 트랜잭션 객체(Transaction Object)를 생성한다. 이 객체 내부에는 결과 처리를 위한 콜백 함수 포인터나 람다(Lambda) 함수가 속성으로 보관되며, 이를 해시 맵(Hash Map)이나 딕셔너리(Dictionary)와 같은 조회 최적화 자료구조에 등록한다.
  2. 트랜잭션 대기열(Pending Queue) 관리: 등록된 트랜잭션은 대기열에 들어가며, 별도의 타이머 스레드가 정기적으로 대기열을 순회하면서 타임아웃이 발생한 명령의 재전송을 트리거하거나 최종 실패(Timeout Failure) 콜백을 발생시킨다.
  3. 메시지 수신 버퍼링 및 파싱: MAVLink 파서(Parser)가 데이터 스트림으로부터 COMMAND_ACK 메시지를 디코딩하면, 해당 메시지 내부의 Command ID와 응답 결과 코드(MAV_RESULT)를 추출한다.
  4. 콜백 탐색 및 비동기 호출: 추출된 식별자를 키(Key)로 활용하여 대기열에서 매칭되는 트랜잭션 객체를 탐색한다. 객체가 발견되면 대기열에서 이를 삭제(Dequeue)하고 보관된 콜백 함수에 추출한 결과 코드를 인자로 주입하여 실행한다. 이를 통해 상태 머신의 상태 전이(State Transition)가 비동기적으로 완료된다.

6. PX4-Autopilot과 Ardupilot 간 비동기 아키텍처 비교

이러한 메시지 콜백 매핑 메커니즘에 있어서 PX4와 Ardupilot은 내부 구조적 설계에서 명확한 차이를 보인다.

PX4-Autopilot은 uORB (Micro Object Request Broker) 기반의 발행-구독(Publish-Subscribe) 미들웨어를 심층적으로 채택하고 있다. MAVLink 데이터 수신 모듈과 비행 제어를 담당하는 커맨더(Commander) 로직이 강하게 결합되어 있지 않기 때문에(Decoupled), 비동기 응답(vehicle_command_ack)이 이벤트 버스를 타고 자연스럽게 흘러간다. 이로 인해 다중 컴포넌트 환경이나 복잡한 비동기 콜백 처리에 있어 매우 유연하고 확장성이 높다.

반면, Ardupilot은 전통적으로 중앙 집중식 스케줄러 내에서 상태 머신이 직접 MAVLink 응답 플래그나 글로벌 공유 메모리 구조체를 폴링(Polling)하거나 검사하는 방식에 더 의존하는 경향이 있다. 최근에는 이벤트 드리븐 성격이 강화되었으나, PX4처럼 완전히 구독자 기반으로 분리된 콜백 매핑 인프라보다는 내부 객체의 동기화 틱(Tick) 주기에 맞추어 결과 코드를 전파 및 처리하는 패턴이 여전히 큰 비중을 차지한다.

7. QGroundControl 비동기 콜백 매핑 구현 예시 (C++ Qt)

실제 오픈소스 지상 관제 시스템인 QGroundControl(QGC) 내에서는 Qt 프레임워크의 Signal/Slot 메커니즘을 융합하여 콜백 매핑 관리를 구현한다. 구조를 단순화한 예시는 다음과 같다.

// GCS 차량 관리 모듈 내부 명령 전송 파트

// 1. 콜백 함수 원형을 정의 (결과 코드 및 타임아웃 상태 수신)
typedef std::function<void(MAV_RESULT result, bool noResponseTimeout)> CommandCallback;

// 2. 대기열 맵 선언 (Command ID -> 콜백 매핑)
QMap<int, CommandCallback> _pendingCommandCallbacks;

// 3. 비동기 명령 전송 메서드
void Vehicle::sendMavlinkCommandAsync(int commandId, float param1, CommandCallback callback) {
    // 트랜잭션을 저장 (단순화를 위해 Command ID를 키로 사용; 실제로는 target/comp id 고려)
    _pendingCommandCallbacks.insert(commandId, callback);
    
    // 타임아웃 타이머 시작 등 추가 로직 생략...
    
    // MAVLink 메시지 포장 및 물리 계층 전송
    mavlink_message_t msg;
    mavlink_msg_command_long_pack(system_id, comp_id, &msg, ...);
    _linkManager->sendMessage(msg);
}

// 4. COMMAND_ACK 수신 처리 슬롯 메서드
void Vehicle::_handleCommandAck(const mavlink_message_t& message) {
    mavlink_command_ack_t ack;
    mavlink_msg_command_ack_decode(&message, &ack);
    
    int receivedCommandId = ack.command;
    MAV_RESULT resultCode = static_cast<MAV_RESULT>(ack.result);
    
    // 매핑된 콜백 탐색
    if (_pendingCommandCallbacks.contains(receivedCommandId)) {
        CommandCallback callback = _pendingCommandCallbacks.take(receivedCommandId); // 맵에서 꺼내고 삭제
        
        // 동기화된 컨텍스트에서 비동기 콜백 함수 실행
        callback(resultCode, false);
    } else {
        // 등록되지 않은 ACK 수신 파싱 오류 등 예외 처리
        qWarning() << "Unmapped COMMAND_ACK received for command:" << receivedCommandId;
    }
}

이와 같은 매핑 구조를 통해, 상위 로직은 명령이 전송된 후 코드가 블로킹되지 않으면서도, 비동기 람다(Lambda) 함수 컨텍스트 내에서 명령의 성공 혹은 권한 거부(Denied)에 대한 개별적인 복구 및 사용자 피드백 절차를 모듈화하여 유려하게 처리할 수 있게 된다.
