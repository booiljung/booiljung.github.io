51.1.1. 비행 제어 스택(Flight Control Stack) 내 내비게이터(Navigator) 모듈의 계층적 역할

PX4-Autopilot의 비행 제어 스택(Flight Control Stack)은 고도의 모듈화(Modularization)를 통해 다양한 비행 궤적 및 다기종 에이전트를 통합적으로 지원하도록 설계되었다. 특히 복귀(Return) 모드와 같은 자율 비행(Autonomous Flight) 기능이 호출될 때, 추정기(Estimator)로부터 올라오는 센서 융합 데이터와 MAVLink를 통한 외부 관제 명령을 제어기가 처리할 수 있는 수학적 궤적(Trajectory)으로 변환하는 중추적인 역할을 내비게이터(Navigator) 모듈이 수행한다.

1. 내비게이터(Navigator) 모듈의 논리적 계층 위치

내비게이터 모듈은 비행 스택 아키텍처 상향식(Bottom-up) 구조상 상태 추정기(EKF2/ECL)의 상단, 그리고 위치 제어기(Position Controller, 예: mc_pos_control, fw_pos_control_l1)의 상단에 위치하며, 하향식(Top-down) 관점에서는 커맨더(Commander) 모듈 아래에 자리한다. 즉, 미들웨어 구조에서 상위 계층의 ’의사 결정(Decision)’을 하위 계층의 ’물리적 제어 목표(Setpoint)’로 번역(Translation)하는 브리지(Bridge) 역할을 담당한다.

    A[QGroundControl / MAVLink] -->|MAV_CMD_NAV_RETURN_TO_LAUNCH| B(Commander Module)
    B -->|vehicle_status / navigator_mode 설정| C{Navigator Module}
    D[Estimator / EKF2] -->|vehicle_global_position / home_position| C
    C -->|trajectory_setpoint 생성| E(Position Controller)
    E -->|vehicle_attitude_setpoint| F(Attitude Controller)
    F -->|actuator_outputs| G[Mixer / Motors]

2. Ardupilot과 PX4-Autopilot의 내비게이터 구조 설계 차이

PX4-Autopilot과 Ardupilot 모두 자율 복귀 기능을 수행하지만 소프트웨어 설계 철학은 극명히 구분된다.

  • 독립적 프로세스 분리(Process Isolation): PX4의 navigator는 독립된 데몬(Daemon) 스레드로 백그라운드에서 동작한다(NuttX RTOS 기반). 이는 제어 주파수(예: 250Hz 이상의 위치/자세 제어 루프)와 임무 갱신 주파수(예: 50Hz)를 분리함으로써 시스템 자원 할당을 최적화한다.
  • Ardupilot의 통합형 루프: 반면 Ardupilot은 특정 비행 모드(예: RTL) 로직을 메인 제어기 내에서 직접 분기(Switch-Case) 처리하는 경우가 빈번하다. 이러한 구조는 내부 지연을 최소화하는 장점이 있지만, 새로운 비행 임무 알고리즘을 추가할 때 기존 제어 코드를 대대적으로 수정해야 하는 단점이 존재한다.

3. uORB 메시징 통신 및 센서 데이터 구독 구조

내비게이터 모듈은 동작을 위해 비동기 퍼블리시-서브스크라이브 패턴인 uORB 네트워크에 깊게 관여한다.

  1. 입력 데이터(Subscriptions): 기하학적 복귀 경로를 산출하기 위해 필수적으로 home_position(이륙 시 기록된 출발 좌표), vehicle_global_position(현재 절대 좌표계상의 위치), 그리고 MAVLink에서 파싱된 vehicle_command를 구독한다. 통신 두절 상황을 감지하기 위한 telemetry_status 혹은 배터리 경고 battery_status 등도 함께 모니터링한다.
  2. 데이터 생산(Publications): 복귀 알고리즘(예: 상태 머신에 따른 상승 및 순항) 계산 후, 로컬 좌표계 기준의 trajectory_setpoint(위치, 속도, 가속도 목표치)를 발생시킨다.

4. 소스 코드 분석: 메인 스케줄링(Main Scheduling) 루프

PX4 펌웨어(v1.14 기준)의 소스 디렉토리 src/modules/navigator/navigator_main.cpp를 분석하면 내비게이터의 심장부인 메인 루프를 확인할 수 있다.

이 루프는 다음과 같은 주기적 타스크(Run())로 구성된다.

  1. 커맨더의 최신 모드(Mode) 변경 요청 상태를 확인한다.
  2. 현재 모드가 Return으로 확인되면, 내부 다형성을 통해 Return 인스턴스의 on_active()를 호출한다.
  3. GPS/VIO 데이터 갱신 여부를 체크하고, return.cpp 블록 내의 FSM 노드를 업데이트하여 새 셋포인트를 생성한다.

결과적으로 내비게이터 모듈은 센서(GPS, 기압계 등) 추정 품질 붕괴와 같은 돌발 상황(Failsafe Condition)을 인지하고, 이를 기반으로 비행 모드를 자율적으로 전환하거나 시스템을 종료(Flight Termination)하도록 커맨더에 권고하는 지능적 모니터링 주체로 작용한다. 이러한 계층 구조는 다양한 센서 융합과 제어기를 독립적으로 교체할 수 있게 하는 PX4-Autopilot의 가장 강력한 코어 아키텍처적 특성이다.
