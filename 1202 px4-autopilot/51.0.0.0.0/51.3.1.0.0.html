51.3.1 MAVLink 프로토콜의 명령 직렬화(Serialization) 및 마이크로서비스(Microservices) 아키텍처

MAVLink(Micro Air Vehicle Link)는 무인항공기(UAV)와 지상 관제 시스템(GCS) 간의 통신을 위해 고안된 경량의 메시징 프로토콜이다. PX4-Autopilot 아키텍처에서 MAVLink 통신은 단순한 구조체(Struct) 전송을 넘어, 페이로드 직렬화(Serialization) 기법과 마이크로서비스(Microservices) 기반 트랜잭션 모델을 채택하여 신뢰성과 확장성을 도모하고 있다. 특히 복귀(RTL/Return)와 같은 중요 제어 트랜잭션을 처리할 때 이러한 아키텍처적 특성이 극대화된다.

이 장에서는 PX4(v1.14.0 이상)의 네트워크 통신 모듈 내부에서 MAVLink 메시지가 C/C++ 구조체로부터 네트워크 바이트 스트림으로 어떻게 직렬화되는지, 그리고 명령 트랜잭션을 관장하는 마이크로서비스 아키텍처가 어떻게 동작하는지 소스 코드 레벨에서 심층적으로 분석하라.

1. MAVLink 페이로드 직렬화(Serialization) 원리

직렬화는 메모리에 할당된 데이터 구조체나 객체 상태를 통신망으로 전송할 수 있는 바이트 스트림(Byte Stream) 형태로 변환하는 과정이다. MAVLink v2 프로토콜은 C 언어의 구조체 패딩(Struct Padding) 문제를 방지하고 플랫폼의 엔디안(Endianness, Byte Order) 독립성을 보장하기 위해 고유한 직렬화 컴파일러(MAVLink Generator, C 헤더 파일 등 자동 생성)를 사용한다.

  • 데이터 정렬(Data Alignment): MAVLink는 메시지 정의(XML)에 기재된 순서가 아니라, 데이터 타입의 크기가 큰 것(예: uint64_t)부터 작은 것(uint8_t) 순으로 메모리에 강제 재배치하여 빈 공간(Padding)을 완전히 제거한다. 이를 통해 페이로드 오버헤드를 최소화하고 시리얼/RF/WiFi 통신 링크의 대역폭 효율을 높인다.
  • 리틀 엔디안(Little-Endian) 스탠다드: MAVLink 패킷은 아키텍처의 엔디안과 상관없이 항상 리틀 엔디안 포맷으로 네트워크에 전송된다. PX4가 탑재된 ARM Cortex-M(주로 리틀 엔디안) 환경에서는 오버헤드가 없지만, GCS나 타 아키텍처에서는 수신 후 언패킹(Unpacking) 과정에서 매크로(예: _mav_get_uint16_t)를 통해 바이트 오더를 맞춘다.

2. PX4 내부의 MAVLink 마이크로서비스(Microservices) 구조

PX4 시스템의 src/modules/mavlink 디렉토리 아래에는 MAVLink 통신을 관장하는 다수의 스레드(혹은 워크 큐)가 존재한다. PX4는 복잡한 GCS-Autopilot 연동 시나리오를 효과적으로 처리하기 위해 “마이크로서비스(Microservices)” 패턴을 차용하여 MAVLink 기능을 분리 구현했다.

3. COMMAND_LONG 직렬화 및 스트리밍 과정

복귀(Return) 명령을 지시하는 COMMAND_LONG 패킷이 QGC에서 송출될 때의 직렬화 과정을 추적해 보자.

  1. 메시지 팩킹(Message Packing): QGC에서는 MAVLink 라이브러리의 mavlink_msg_command_long_pack() 함수를 호출한다. 패킷 길이는 33바이트(MAVLink v2 규격)로 압축된다.
  2. PX4 파싱(Parsing): 기체의 시리얼 포트 프론트엔드가 바이트 스트림을 수신하면 mavlink_parse_char() 함수가 Magic Byte(0xFD)를 시작으로 시퀀스 번호, 컴포넌트 ID를 확인하고 CRC 무결성 검증을 마친 후 내부 버퍼로 복사한다.
  3. 디스패치(Dispatch) 및 uORB 브릿징: 언패킹된 데이터 구조체는 MavlinkReceiver로 넘어가 Command Service에 배분되고, 이는 PX4 내부의 메시징 버스인 uORB의 vehicle_command 토픽으로 변환(Translation)하여 발행(Publish)된다.

이러한 모듈 간 분리성은 ROS2 환경과의 연동 시에도 장점이 된다. Micro XRCE-DDS 에이전트가 ROS2에서 내리는 비행 명령 역시 수신부만 다를 뿐, vehicle_command라는 동일한 uORB 토픽으로 변환되어 Commander 모듈에 전달되도록 아키텍처가 설계되어 있다.

4. Ardupilot 대비 아키텍처 차이점

Ardupilot은 전통적으로 거대한 GCS_MAVLink 클래스가 다중 포트(Serial 0~3)에 대한 메시지 라우팅과 인코딩/디코딩 업무를 다소 모놀리식(Monolithic)하게 수행해왔다(최근 분할 추세이나 여전히 강결합 속성이 존재).
반면 PX4의 MAVLink 모듈 아키텍처는 개별 포트별로 독립된 인스턴스(Instance) 앱이 구동되며, 각 인스턴스 내에서 마이크로서비스 클래스가 플러그인처럼 조립된다. 이는 QGroundControl 관제 시 복수의 MAVLink 링크(예: 텔레메트리 보드와 LTE 컴패니언 컴퓨터 동시 사용)에 대해 각 마이크로서비스가 전송 대역폭(Stream Rate)을 독립적으로 제어하기 유리한 환경을 제공한다.

명령 직렬화의 엄격성과 마이크로서비스 분산 설계는, 드론 자율 비행의 고도화와 확장성(예: 군집 비행 제어, 사용자 정의 페이로드 추가)을 지원하기 위해 PX4가 채택한 강력한 소프트웨어 공학적 기반이다.
