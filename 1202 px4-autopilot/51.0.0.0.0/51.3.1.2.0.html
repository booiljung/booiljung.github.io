51.3.1.2 시스템 ID 및 컴포넌트 ID 필터링을 통한 다중 에이전트(Swarm) 환경 내 개별 기체 타겟팅 및 브로드캐스트 억제

다중 에이전트(Swarm) 환경에서 여러 대의 무인항공기(UAV)가 동시에 운용될 때, 지상 관제 시스템(GCS) 또는 동반 컴퓨터(Companion Computer)가 특정 기체에게만 제어 명령을 내리거나, 전체 기체에 브로드캐스트(Broadcast)를 수행하는 것은 MAVLink 통신의 핵심 로직 중 하나이다. 특히, 특정 기체만 복귀(RTL)시키거나 비행 모드(Flight Mode)를 변경해야 할 때, 타겟 시스템 ID(Target System ID)와 타겟 컴포넌트 ID(Target Component ID)를 통한 필터링 메커니즘은 매우 중요하다.

본 절에서는 PX4-Autopilot(펌웨어 v1.14 기준)과 QGroundControl(v4.3.0 이상)을 기반으로 다중 에이전트 환경에서 개별 기체를 타겟팅하는 원리와 브로드캐스트 억제를 통한 통신망 부하 경감(Traffic Mitigation) 기법에 대해 상세히 분석한다.

1. MAVLink 라우팅 및 타겟 ID 시스템 아키텍처

MAVLink 프로토콜은 모든 메시지에 송신자의 출처를 식별하기 위한 sysid (System ID)와 compid (Component ID)를 포함한다. 특히 제어 명령 기반 메시지(COMMAND_LONG 등)의 경우 수신 대상을 명확히 지정하기 위해 target_system과 target_component 필드를 추가로 포함한다.

  • 시스템 ID(System ID): 네트워크 상에서 물리적인 기체 하나 혹은 GCS를 구분하는 고유 식별자이다. 1부터 254까지의 값을 가질 수 있으며, GCS는 주로 255를 할당받는다.
  • 컴포넌트 ID(Component ID): 단일 기체 시스템 내에 존재하는 하위 제어 모듈(예: Autopilot, 짐벌 제어기, 동반 컴퓨터 등)을 식별한다. 기본 비행 제어 컴포넌트는 대부분 MAV_COMP_ID_AUTOPILOT1 (1)을 사용한다.

1.1 브로드캐스트(Broadcast) 주소 지정 원리

메시지의 target_system 또는 target_component 필드가 0으로 설정될 경우, 이는 브로드캐스트(Broadcast) 명령으로 해석된다.

  • target_system = 0: 네트워크 내에 접속된 모든 기체(System)에 메시지 전달
  • target_component = 0: 특정 기체(System) 내의 모든 로컬 컴포넌트에 메시지 전달
  • target_system = 0, target_component = 0: 네트워크 내의 모든 기체 및 모든 컴포넌트에 완전한 글로벌 브로드캐스트 수행

1.2 개별 기체 타겟팅 원리

특정 기체(예를 들어 System ID가 3인 드론)에만 명령을 전달하고자 한다면, target_system = 3, target_component = 1 로 설정하여 명령 메시지를 전송한다. 다른 System ID를 가진 기체는 MAVLink 수신단(Receiver)의 파싱 계층에서 이 메시지가 자신을 향한 것이 아님을 인지하고 즉각 무시(Drop)한다.

2. PX4-Autopilot의 메시지 필터링 및 브로드캐스트 억제 메커니즘

수십 대 이상의 기체를 운영하는 군집 비행(Swarm Flight) 환경에서는 단일 기체 혹은 GCS가 무분별하게 퍼뜨리는 브로드캐스트 메시지가 전체 통신 대역폭(Bandwidth)을 단숨에 포화 상태(Saturation)로 만들어 브로드캐스트 폭풍(Broadcast Storm) 현상을 유발할 수 있다. PX4-Autopilot 메인 소스 코드의 MAVLink 수신 모듈(mavlink_receiver.cpp)에서는 이러한 무분별한 메시지를 엄격하게 필터링하여 네트워크 병목 구간의 부하를 억제한다.

2.1 라우팅 판단 로직 흐름도 (Routing Decision Logic)

아래는 제한된 대역폭 하에서 드론이 자신에게 수신된 MAVLink 메시지를 어떻게 판단하고 필터링하는지 나타내는 로직 흐름도이다.

flowchart TD
    A[MAVLink 패킷 수신] --> B{target_system == 본인 기체 ID?}
    B -- Yes --> C{target_component == 본인 컴포넌트 ID OR 0?}
    B -- No --> D{target_system == 0? Broadcast}
    
    C -- Yes --> E[메시지 처리 수락 및 내부 uORB 토픽 퍼블리싱]
    C -- No --> F[기타 컴포넌트로 포워딩 Forwarding 또는 폐기]
    
    D -- Yes --> G{브로드캐스트 허용 대상 메시지인가?}
    D -- No --> H[메시지 무시 및 폐기 Drop]
    
    G -- Yes --> E
    G -- No --> H

2.2 브로드캐스트 억제(Suppression) 기법 분석

PX4는 군집 모드에서의 안전성과 대역폭 관리를 위해, 특정 커맨드를 제외하고는 브로드캐스트 명령 수신을 시스템 레벨에서 강력히 제한한다. 주기적인 상태 정보를 포함하는 HEARTBEAT 등은 브로드캐스트를 통해 전이되지만, 치명적인 액션을 유발하는 핵심 제어 명령(예: 시동 해제, 모드 전환, 복귀 등)은 target_system이 정확히 일치하는 유니캐스트(Unicast)이거나, 매우 특수한 전제 조건 하단에서만 허용된다. 이는 네트워크 트래픽 과부하를 막는 방어 기제 역할을 수행한다.

3. 다중 에이전트 환경(Swarm)에서의 타겟팅 사례 분석

지상 관제 시스템(QGroundControl)이나 ROS2 기반 커스텀 관제 애플리케이션을 통해 수많은 에이전트를 조율할 때 사용하는 패킷 전송 사례는 다음과 같다.

3.1 개별 기체 복귀(RTL) 명령 하달 패턴

5대의 군집 드론 중 배터리가 부족한 “기체 5번(System ID: 5)“만을 기지로 강제 복귀(Return To Launch)시켜야 할 경우, GCS는 아래와 같이 패킷 필드를 구성한다.

  • command: MAV_CMD_NAV_RETURN_TO_LAUNCH (20)
  • target_system: 5
  • target_component: 1 (MAV_COMP_ID_AUTOPILOT1)

기체 5번의 PX4 MavlinkReceiver는 수신 즉시 이 명령을 파싱하고 수락한 뒤 COMMAND_ACK 트랜잭션을 응답하여 궤적 생성 모듈에 유효함을 알린다. 반면, 기체 1, 2, 3, 4번은 텔레메트리 데이터 버스(Bus) 상에서 물리적으로 동일한 패킷 신호를 감지하더라도, 수신 계층에서 target_system 값의 불일치를 판정하여 내부 상태 머신에 전달하기 전 즉각 폐기(Drop)한다.

3.2 전체 기체 동시 상태 제어 (Swarm Broadcast Control)

반대로 비상 상황 발생으로 전체 드론 편대를 제자리에 대기유지(Hold) 시켜야 할 경우에는 제한적 브로드캐스트를 사용할 수 있다.

  • target_system: 0
  • target_component: 0

이 경우 모든 기체가 명령을 파싱한다. 단, 시스템이 이를 모두 수락하게 되면 수십 대의 기체가 동시에 COMMAND_ACK 응답을 GCS로 되돌려 보내는 ACK 폭풍(ACK Storm) 현상이 발생한다. PX4는 최신 버전으로 갈수록 이러한 무분별한 브로드캐스트 사용을 지양하며, 각 기체별 독립 제어 트랜잭션을 시분할(Time-division)하여 유니캐스트 멀티플렉싱(Unicast Multiplexing) 구조로 보내는 방식을 더욱 권장한다.

4. Ardupilot 대비 관제 시스템 통신 철학의 차이점

Ardupilot 관제 통신 계층도 동일한 MAVLink 표준 프레임워크를 기반으로 구성되지만, Swarm(군집) 및 다중 기체 처리에 대한 구현 철학과 제어 관점은 다음과 같이 차이가 있다.

  • PX4-Autopilot 분리 및 모듈화 철학: 마이크로서비스(Microservices) 기반의 모듈화에 중점을 두어 동작한다. QGroundControl은 각 기체의 파라미터(Parameter), 미션(Mission), 그리고 트랜잭션을 비동기식(Asynchronous)으로 매우 철저히 독립 관리한다. 물리적 연결은 한 회선이라도 논리적인 기체 연결 객체(Vehicle Object) 인스턴스는 메모리 상에 완전히 독립 생성되며, 타겟팅된 System ID를 기준으로 맞춤형 UI/UX 렌더링을 격리해 수행한다. 브로드캐스트보다는 유니캐스트 의존도가 높다.
  • Ardupilot 통신 철학: Mission Planner 등의 관제 소프트웨어를 구동할 때, 동일한 무선 링크(Telemetry Link)를 타고 수신되는 MAVLink 패킷들을 다루는 데 있어, Swarm 목적으로 브로드캐스트 명령어의 수용 및 확장을 조금 더 유연하게 허용하는 경우가 있다. 이로 인해 다중 기체 실험에서 제어를 간편히 구현할 수는 있으나, 통신망 신뢰도(Reliability) 측면에서는 PX4의 방어적 억제 로직이 더 엄격한 편이다.

5. 소스 코드 구현 관점: 시스템 ID 검증 메커니즘 딥다이브

실제 PX4 펌웨어 소스 트리 내 src/modules/mavlink/mavlink_receiver.cpp 파일의 Command 파서(handle_message_command_long) 구조를 분석해 보면 타겟 ID 검증과 필터링 소프트웨어 아키텍처를 가장 명확히 이해할 수 있다.

void MavlinkReceiver::handle_message_command_long(mavlink_message_t *msg)
{
    mavlink_command_long_t cmd_mavlink;
    mavlink_msg_command_long_decode(msg, &cmd_mavlink);

    // [핵심 검증 로직]
    // 1. target_system 필드가 0(Broadcast)이거나, 본인 기체의 system ID와 정확히 일치하는가?
    bool target_system_ok = (cmd_mavlink.target_system == 0) || 
                            (cmd_mavlink.target_system == mavlink_system.sysid);

    // 2. target_component 필드가 0이거나, 본인 기체의 component ID와 일치하는가?
    bool target_component_ok = (cmd_mavlink.target_component == 0) || 
                               (cmd_mavlink.target_component == mavlink_system.compid);

    // 타겟팅에 해당하지 않는다면 즉각 리턴(Return)하여 프로세싱 비용과 자원의 낭비를 막는다.
    if (!target_system_ok || !target_component_ok) {
        // 필터링 메커니즘: 해당 기체의 명령이 아니므로 포워딩을 수행하거나 메모리 상에서 폐기(Drop)한다.
        return;
    }

    // 조건 부합 시: 수신된 외부 MAVLink 명령을 내부의 표준 uORB 메시지(vehicle_command)로 변환
    vehicle_command_s vcmd{};
    vcmd.command = cmd_mavlink.command;
    vcmd.target_system = cmd_mavlink.target_system;
    vcmd.target_component = cmd_mavlink.target_component;
    // ... 세부 제어 파라미터 매핑 작업 수행 ...

    // 파싱된 최종 명령을 제어 모듈이 청취할 수 있도록 토픽 퍼블리싱 수행
    _cmd_pub.publish(vcmd);
}

소스 코드에서 관찰할 수 있듯, 타겟 ID의 검증 로직 자체는 짧지만 군집 비행 제어에서 이는 가장 절대적이고 강력한 방어선(Defense Line)이 된다. 이 필터를 온전히 거친 패킷만이 내부의 vehicle_command 구조체로 변환되어 실질적인 기체의 제어기(Controller)로 하달될 수 있다.

6. 결론

최신 다중 에이전트(Swarm) 관제 및 비행 제어 시스템을 결점 없이 완벽하게 설계하기 위해서는 MAVLink 프로토콜 상의 System ID와 Component ID 타겟팅 원리를 엄밀히 파악해야 한다. PX4-Autopilot은 target_system 필드 필터링을 통해 유니캐스트 트랜잭션의 신뢰성을 엄격하게 보장하고 있으며, 통신 지연(Latency)의 발생 확률이 높고 데이터 대역폭이 비좁은 다중 기체 통신 환경에서 불필요한 브로드캐스트의 소모를 철저히 억제한다. 이는 대형 스웜 시스템 운영 중 유발될 수 있는 네트워크 폭풍(Network Storm)과 이로 인한 치명적인 조종권 일시 소실(Loss of Control) 현상을 방어하는 강력한 아키텍처적 기반이 된다.
