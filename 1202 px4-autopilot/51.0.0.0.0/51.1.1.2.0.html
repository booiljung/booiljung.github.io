51.1.1.2. 자율 비행 작업(Flight Task) 다형성(Polymorphism)을 활용한 Return Task 인스턴스 생성 및 추상화(Abstraction) 기법

최신 무인항공기 소프트웨어는 단순한 하드웨어 제어기를 넘어 복잡한 비즈니스 로직을 처리하는 임베디드 운영체제 수준의 복잡도를 지닌다. PX4-Autopilot은 복귀(RTL) 궤적을 포함한 다양한 실시간 비행 임무(Task)의 궤적을 안전하게 캡슐화(Encapsulation)하기 위해 현대 C++의 객체 지향 프로그래밍(OOP) 기법 중 하나인 **다형성(Polymorphism)**을 적극적으로 활용한다.

1. Flight Task의 추상화(Abstraction) 구조

PX4 내비게이터(Navigator) 모듈에서 비행 궤적을 생성하는 모든 기능적 모듈은 최상단의 기본 클래스(Base Class)인 MissionBlock 또는 제어 스택 내의 FlightTask를 상속받아 구현된다. 다형성(Polymorphism)의 핵심은 시스템이 현재 수행 중인 구체적인 비행 모드가 무엇인지 알 필요 없이, 부모 클래스의 포인터를 통해 예측 가능하고 통일된 인터페이스로 궤적을 요청할 수 있다는 점이다.

2. Return Task 인스턴스의 라이프사이클(Lifecycle)

GCS(예: QGroundControl)에서 MAV_CMD_NAV_RETURN_TO_LAUNCH 커맨드를 전송하여 복귀 명령이 활성화될 때, 내비게이터 모듈 내부에서는 Return 클래스의 인스턴스가 호출되며 다음과 같은 라이프사이클을 거친다.

  1. 객체 활성화 (on_activation): 내비게이터가 상태 전이를 결정하면, 이전 Task(예: 수동 제어 또는 임무 비행)의 on_inactivation()을 호출하여 메모리와 상태를 정리한 후, Return 인스턴스의 on_activation() 메서드를 오버라이딩(Overriding)하여 호출한다. 이 시점에 현재 기체 위치, RTL_RETURN_ALT 등의 파라미터가 평가되어 FSM의 초기 상태(주로 상승 상태)가 결정된다.
  2. 동적 궤적 갱신 (on_active): 루프 주기마다 다형성을 통해 호출되는 이 함수는 set_return_item()과 같은 구체적 메서드를 이용하여 uORB trajectory_setpoint 메시지를 갱신한다.
  3. 메모리 및 제어권 해제 (on_inactivation): 랠리 포인트 도달 후 착륙하여 시동이 해제(Disarmed)되거나, 조종자가 수동 모드로 개입하면 on_inactivation()이 호출되어 안전하게 제어권을 반환한다.

3. 아키텍처적 이점: Ardupilot과의 비교

이러한 추상화(Abstraction) 기법은 PX4-Autopilot이 Ardupilot과 구별되는 가장 구조적인 강점 중 하나이다.
Ardupilot의 경우 mode_rtl.cpp와 같이 특정 제어 루프가 메인 비행 클래스 내에 비교적 절차적(Procedural)으로 연동되어 있어, 복귀 로직을 확장하거나 고정익/멀티로터에 따른 세부 분기를 만들 때 소스 코드의 결합도(Coupling)를 높이는 경향이 있다.

반면, PX4는 인터페이스가 강제된 다형성을 통해 기능 추가 시 기존 시스템 코드를 수정하지 않고 새로운 클래스(예: 장애물 회피를 포함한 Smart Return Task)만 상속 구현하여 런타임에 인스턴스를 교체(Dependency Injection 유사 패턴)하는 방식으로 손쉽게 확장이 가능하다. 이는 방대한 학술 연구와 자율 에이전트 GCS 관제 개발에 있어 탁월한 유지보수성을 제공한다.
