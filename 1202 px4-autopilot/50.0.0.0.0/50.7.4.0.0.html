50.7.4. 지상 관제 데이터 링크(Data Link) 지속 단절 시의 온보드(Onboard) 컴퓨터 자율 의사 결정 트리(Decision Tree) 시스템

비가시권(BVLOS, Beyond Visual Line of Sight) 자율 비행 임무 중 지상 관제 시스템(GCS)과의 텔레메트리 무선 데이터 링크(Data Link)가 끊어지는 현상은 환경 내 전파 도달 거리 물리적 제한, 거대 지형적 차폐, 또는 기체 이중 안테나 송수신 불량 등의 물리적 이유로 필연적으로 수시 발생할 수 있다. PX4-Autopilot은 데이터 링크 단절(Data Link Loss, DLL) 상황을 방어할 수 없는 단순한 일회성 오류 타임아웃으로 치부하지 않고, 온보드 컴퓨터(내부 비행 제어기 및 Companion Computer) 뇌관이 현재 기체의 상태와 임무 진행도, 센서 건전성을 종합적으로 실시간 연산하여 최적의 생존 및 임무 지속 전략을 결정해내는 자율 의사 결정 트리(Decision Tree) 기반의 심층 Failsafe 시스템 아키텍처를 내장하고 있다.

1. 데이터 링크 단절(DLL) 감지 메커니즘

PX4-Autopilot은 GCS와의 정상적인 통신 상태를 지속 모니터링하기 위해 MAVLink 프로토콜의 생명선인 HEARTBEAT (Message ID: 0) 구조체 메시지 연속성을 최우선적으로 신뢰한다.

  • 정상 헬스 상태 (Normal State): QGroundControl(GCS) 소프트웨어 라우터 및 탑재된 텔레메트리 라디오 모듈은 대략 1Hz(초당 1회) 주기로 HEARTBEAT 메시지를 무선 교환하며 상호 생존을 증명한다.
  • 단절 확정 판별 (Loss Detection): 코어 커맨더(Commander) 모듈 내부의 독립 인스턴스인 failsafe_statemachine은 NAV_DLL_ACT (Data Link Loss Action 행동 파라미터)에 정의된 작동 바에 따라 감시 대역을 분리한다. GCS로부터 수신되는 HEARTBEAT 패킷 타임스탬프가 설정된 허용 타임아웃 파라미터인 COM_DL_LOSS_T (기본값 약 10초) 동안 도달하지 않을 때 이를 ’데이터 링크 지속 붕괴 단절(DLL)’로 확정 짓는다.

이러한 단절 여부가 코드 상으로 확정되는 즉시, uORB 내부 주제인 telemetry_status 버스의 data_link_lost 불리언(Boolean) 플래그가 true로 덮어씌어 확정되며 자율 의사 결정 트리가 시스템 전역에 트리거(Trigger)되어 활성화된다.

2. 온보드 자율 의사 결정 트리(Decision Tree) 아키텍처 계층

데이터 통신이 끊어졌다고 해서 원시적으로 무조건 시작점 구역으로 맹목적인 귀환 복귀(RTL, Return To Launch)를 수행하는 것은 2차 재난 인명 구조나 장거리 핵심 물류 배송 등 고도화된 특수 목적 산업 임무에서 대단히 치명적이고 비효율적인 조치일 수 있다. 따라서 PX4의 제어 스택은 파라미터 NAV_DLL_ACT에 사용자가 지정해 둔 운용 정책과 현재 진행 중인 비행 모드 양방항을 교차 검증 연산하는 강력한 의사 결정 트리를 내부로 통과시킨다.

2.1 Decision Tree 상세 경로 분석

  1. 초기 상태 진입 (Data Link Lost 감지):
  • 현재 물리 조종기(RC) 주파수 링크도 함께 소실되었는가? (이중 통신 전면 단절 확인 절차 선행)
  1. 파라미터 기반 1차 분기 (NAV_DLL_ACT 평가):
  • 0 (Disabled 옵션): 아무런 행동 보호 조치도 취하지 않고 위태롭지만 현재 궤도 명령을 그대로 유지한다. (개발자 디버깅 전용)
  • 1 (Hold 옵션): 즉시 현재 허공 고도 및 3차원 위치에 정지브레이크를 걸고 호버링(Hovering, Loiter) 체공하며 GCS 통신 복구를 무기한/유기한 대기한다. (COM_DL_REG_T 복구 파라미터 초과 시 2차 조치 발동 여부 연계)
  • 2 (Return mode 옵션): 가장 일반적이고 산업계 권장되는 설정치. 즉각 RTL(Return To Launch) 시퀀스 노드로 돌입하여 장애물 회피 목적의 절대 안전 고도로 상승 후 이륙 지점 홈(Home)으로 복귀 이동한다.
  • 3 (Land 옵션): 추가적인 고도 상승이나 위치 이동 없이 수직 강하하여 제자리 즉각 비상 착륙을 수행한다.
  • 6 (Terminate 옵션): 시동을 강제 차단하고 모터 회전을 즉시 꺼트려 궤도를 물리적으로 비상 종료(Crash)시킨다.
  1. 임무 계속성 검증 분기 (AUTO.MISSION 자율 임무 모드 개입 시):
  • 만약 기체가 사전 업로드된 자동 임무(Mission)를 순항 수행 중이고, NAV_DLL_ACT가 무시(0) 또는 Hold(1)로 설정되어 있더라도, PX4는 임무 개별 프로토콜 파라미터인 MIS_OBL_ACT (Mission Obligation Action) 등에 따라 ‘텔레메트리 통신이 끊어져도 남은 임무 웨이포인트를 끝까지 소화하여 강행(Continue Mission)’ 할 것인지 분리하여 개별 연산 결정한다.
  1. 글로벌 위치(GNSS) 유효성 복합 분기:
  • 통신 단절 상태에서 Return(2)을 수행해야 하는데, 설상가상으로 동시에 GNSS 신호마저 유실 파괴되었다면 기체는 어떻게 행동해야 할 것인가? (다중 결함 동시 발생 시나리오)
  • 위치 솔루션이 소멸되었으므로 동역학적으로 Return 곡선을 단독 수행할 수 없다. 그러므로 커맨더 로직은 의사 결정 계층을 주권적으로 한 단계 임시 낮추어 Position Failsafe 로직(즉, 수직 고도 유지 비상 강제 하강 ALTCTRL 모드 전환)을 하이재킹(Hijacking)하여 우선적으로 신속 적용시킨다.

3. ArduPilot과의 복합 결함 처리 철학 차이

경쟁 플랫폼인 ArduPilot(ArduCopter 등)의 FS_GCS_ENABLE 및 FS_GCS_TIMEOUT 메커니즘 코어 또한 지상 관제 데이터 링크 상실 시 매우 유사한 방어적 행동(RTL, Land 등) 규칙을 정의할 수 있다.
그러나 PX4-Autopilot은 다중 센서 및 다중 통신 결함(Multiple Faults)이 예측 불가능하게 복합적으로 동발(동시 발생)하였을 때, 이를 안전하게 관할하고 억제하는 ‘계층적 우선순위(Hierarchical Priority)’ 철학 아키텍처가 상대적으로 시스템 전반에 더 깊숙이 그리고 명확하게 설계되어 있다.

  • PX4-Autopilot: 상태 머신 내부의 예외 처리(Exception Handling)가 고수준의 트리 형태로 구조화되어 있다. 예를 들어 데이터 링크 단절(DLL)과 배터리 부족(Low Battery), GNSS 상실(Position Loss)이 정확히 같은 순간 동시 다발적으로 겹쳐 발생했을 때, 커맨더(Commander) 내부 회로가 즉각적으로 생존 위험도의 가중치를 계산하여 가장 치명적인 에러(예: 배터리 전압 임계치 도달에 의한 즉각 Land)의 행동 양식을 최상위 의사로 덮어쓰기(Override)하여 즉각 수행하도록 클래스와 로직이 엄격히 분리되어 동작한다.
  • ArduPilot: GCS Failsafe 함수 내에서 개별적으로 절차적 RTL 등을 병렬 호출하는 절차적 프로그래밍 색채가 비교적 강하며, 복합 결함 발생 시 예외 동작들이 메모리 상에서 상호 의도치 않게 충돌(Conflict)하거나 상쇄되지 않도록 경험적으로 하드코딩된 분기(If-else 덩어리)에 상대적으로 더 크게 의존하는 소프트웨어 경향성을 띠곤 한다.

4. 커맨더(Commander) 및 MAVLink 관찰자 소스 코드 분석

PX4 Firmware(v1.14 이상 최신 버전 기준)의 소스 코드 구조에서 DLL 타임아웃을 평가하는 논리적 단위는 주로 /src/modules/commander/failsafe/Failsafe.cpp 내의 로직에 의해 런타임 구동된다.

// Data Link Loss (GCS 연결망 유실) 체킹 정밀 의사 코드 요약
void Failsafe::checkTelemetryLink(const hrt_abstime &time_now)
{
    // 마지막 HEARTBEAT 수신 동기화 시간과 시스템 현재 시간의 차이를 절대 계산
    const hrt_abstime telemetry_timestamp = _telemetry_status.timestamp;
    if (time_now - telemetry_timestamp > _param_com_dl_loss_t.get() * 1_s) {
        
        // DLL 설정 임계치를 막 초과한 통신 단절 치명적 발생!
        if (!_telemetry_link_lost) {
            _telemetry_link_lost = true;
            // 의사 결정 트리를 통한 Failsafe Action 시퀀스 체인 실행
            updateState(FailsafeState::DATALINK_LOST);
        }
    } else {
        // 끊어졌던 통신망이 다시 전파권으로 진입하여 복구(Recovery) 되었을 경우 
        if (_telemetry_link_lost) {
            _telemetry_link_lost = false;
            mavlink_log_info(&_mavlink_log_pub, "Data link regained. Telemetry active.");
            // GCS 설정(복구 액션)에 따라 원래의 비행 모드로 롤백(Rollback) 가능성 타진 연산
        }
    }
}

이 때, updateState(FailsafeState::DATALINK_LOST) 핵심 공통 메서드가 호출되면 앞서 장황하게 설명한 NAV_DLL_ACT 파라미터를 기반으로 스위치-케이스(Switch-case) 룩업 테이블을 거쳐 모드를 계층적으로 조율하여 안전하게 전환하게 된다.

5. 지상 관제망(QGroundControl) 및 자율 에이전트(ROS2)의 감쇠 대처 기술 혁신

QGroundControl(v4.3 이상 최적화 기준)은 이러한 역방향적 상황, 즉 ’자신(GCS)이 상공의 기체로부터 응답 데이터를 전혀 받지 못하는 블랙 아웃(Black-out) 상황’을 자체 감지하면, GUI의 최상단 관제 툴바에 무시무시한 빨간색 경고등과 함께 **“Communication Lost”**라는 치명적 오류 텍스트를 출력하고 기체의 아이콘 수신율을 0% 및 회색(음영 비활성화)으로 즉시 표기한다. 이후, 만약 데이터 링크가 끊어진 암흑 상태에서 기체가 내장된 온보드 의사 결정 트리에 따라 자율적으로 복귀(RTL) 행동 중이라면 지상의 관제사 조종사는 오로지 기체가 다시 사정거리(LOS) 내 주파수망으로 진입하여 텔레메트리 연결 구조가 시스템적으로 자동 복구(Auto Reconnect) 되기만을 가만히 기다려야만 한다.

반면, 멀티 에이전트 군집 비행(Swarm) 환경 조직망이나 보조 컴퓨터(Companion Computer, 예: Nvidia Jetson Nano, Raspberry Pi 5 등)가 탑재되어 연동된 최상의 통합 아키텍처 패러다임에서는 이를 극복할 혁신적 구조를 구축할 수 있다.

  • 비행 제어기(Pixhawk 기반 FC)와 연결된 GCS로 향하는 외부 텔레메트리 라디오가 끊어지더라도, 기체 내부 직렬 하드와이어 통신(UART/USB)으로 견고하게 연결된 보조 온보드 컴퓨터와 FC 사이의 Micro XRCE-DDS 로컬 데이터 버스 인터페이스 통신은 여전히 끊기지 않고 쌩쌩하게 살아있을 물리적 확률이 100%에 가깝다.
  • 따라서 ROS2 네트워크 프레임워크 상에서 구동되는 자체 자율 에이전트 노드는 지속적으로 /fmu/out/telemetry_status 토픽을 초당 구독하다가 GCS 장비 통신망 단절 기점(data_link_lost = true) 플래그 확정을 감지하게 되면, 보조 에이전트 스스로가 무능화된 GCS를 대신하여 **“Local Sub-GCS”**로서의 역할을 소프트웨어적으로 계승(Takeover)할 수 있다.
  • 권한을 인계받은 에이전트는 기체를 Offboard 외부 제어 모드로 안전하게 재구독하여 PX4에 새로운 임무 경로를 스케줄링해주거나, 혹은 우회 통신망 서브 시스템(예: LTE/5G 모듈 모뎀 백업 활성화, 위성 통신 Iridium 재다이얼 등)을 긴급 구동하는 등의 지능화된 분산 의사 결정 처리 로직을 완벽에 가깝게 구현해낼 수 있다.
