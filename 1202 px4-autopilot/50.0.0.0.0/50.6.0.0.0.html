50.6. 지상 관제 시스템(GCS) 소프트웨어 아키텍처의 착륙 상태 원격 모니터링 및 객체 지향 상호작용

무인항공기(UAV)가 비행 임무를 마치고 지면에 접촉하여 착륙(Landing)하는 과정은 물리적인 현상이지만, 이를 지상 관제 시스템(Ground Control Station, GCS) 화면상에서 원격으로 인지하고 제어 모드를 전환하기 위해서는 정교한 통신 및 소프트웨어 상호작용 아키텍처가 요구된다. PX4-Autopilot 생태계의 대표적인 GCS인 QGroundControl(QGC)은 C++ 프레임워크인 Qt(Qt Framework)를 기반으로 작성된 고도로 모듈화된 객체 지향(Object-oriented) 프론트엔드/백엔드 아키텍처를 가진다.

본 절에서는 비행 제어기(Flight Controller, FC) 기체가 착륙 상태에 돌입했을 때, MAVLink 프로토콜을 통과하여 QGroundControl 내부에 객체(Object)의 상태로 캐싱되고, 결국 사용자 인터페이스(UI)에 반영되기까지의 아키텍처 및 원격 모니터링 메커니즘을 상세히 분석한다. (해당 설명은 QGroundControl v4.2+ 및 PX4 v1.13+ 구조를 기준으로 한다.)

1. MAVLink 기반 상태 동기화 아키텍처 개요

기체가 착륙했음을 GCS가 인지하기 위해, 기체는 주기적으로 자신이 공중에 있는지, 아니면 지면에 접촉하여 모터가 멈춘 상태인지를 텔레메트리(Telemetry) 무선 링크를 통해 지속적으로 보고해야 한다.
Ardupilot과 PX4-Autopilot 모두 MAVLink 통신 프로토콜을 공통적으로 사용하지만, 내부 시스템 상태(System State)를 추론하고 브로드캐스트하는 주체 모듈은 다르다. PX4에서는 commander 모듈이 vehicle_land_detected uORB 토픽을 구독(Subscribe)한 후, 최종적으로 MAVLink의 EXTENDED_SYS_STATE 메시지에 landed_state 필드를 담아 GCS로 1Hz 이상의 주기로 송신한다.

QGroundControl 측에서는 이렇게 수신된 패킷을 백엔드에서 비동기적으로 가로채어 파싱(Parsing)하고, 내부적으로 유지하고 있는 기체 모델 클래스의 속성(Property)으로 변환해 프론트엔드인 QML 계층으로 신호(Signal)를 쏘아 올린다.

2. QGroundControl 내 객체 지향적 기체 모델 추상화

QGroundControl 백엔드는 복수의 기체(Swarm) 관제가 가능하도록 각 기체를 추상화한 Vehicle이라는 범용 C++ 클래스를 인스턴스화(Instantiation)하여 관리한다.

  • Vehicle 클래스 (src/Vehicle/Vehicle.h): 단일 무인기에 대한 거의 모든 정보(고도, 배터리, 비행 모드, 시동 상태, 착륙 상태 등)를 캡슐화(Encapsulation)한 핵심 객체이다.
  • 컴포넌트 객체 모듈화: Vehicle 클래스 내부는 각 서브 시스템(예: 펌웨어 플러그인, MAVLink 통신 링크, 임무 관리자, 팩터 매니저 등) 객체들의 합성(Composition)으로 구성된다.

기체의 “착륙(Landed)” 여부는 Qt 프레임워크의 프로퍼티 속성인 Q_PROPERTY 매크로를 사용하여 Vehicle 객체의 멤버 변수로 등록되어 있다. 객체 지향 아키텍처 덕분에 특정 기체 인스턴스의 속성이 변하면, 이를 구독하고 있는 다수의 UI 위젯 객체들이 즉시 이를 통보받아 업데이트된다.

4. 시그널과 슬롯(Signal and Slot) 기반의 비동기 상호작용 상속 구조

QGroundControl은 Qt 핵심 기술인 메타 객체 시스템(Meta-Object System) 기반의 시그널 앤 슬롯(Signal and Slot) 메커니즘을 적극 차용하여 원격 모니터링 모듈 간의 결합도(Coupling)를 낮추었다.

4.1 MAVLink 수신과 역직렬화 체인

네트워크 스레드(Network Thread)에서 UDP 혹은 직렬(Serial) 포트로부터 MAVLINK_MSG_ID_EXTENDED_SYS_STATE 패킷이 도달하면, Vehicle::_mavlinkMessageReceived 메서드(슬롯)가 호출된다. 내부의 switch(msg.msgid) 구문을 거쳐 각각의 전용 디코딩(Decoding) 처리 함수로 분기된다. 이때 착륙 정보는 mavlink_msg_extended_sys_state_decode() C 함수를 통해 페이로드 변수에 풀린다.

4.2 상태 전이 로직 및 시그널 방출(Emit)

Vehicle 객체는 직전에 가지고 있던 기체의 착륙 상태(예: MAV_LANDED_STATE_IN_AIR)와 현재 수신된 상태(예: MAV_LANDED_STATE_ON_GROUND)를 비교한다.

  • 상태 불일치 감지: 변위가 발생했을 때만 백엔드 내부의 멤버 변수인 _flying 혹은 _landed_state 값을 갱신(Update)한다.
  • 이벤트 방출: 속성(Q_PROPERTY) 선언부와 연동된 특정 시그널 구문(예: emit flyingChanged(false))을 강제로 방출한다. 이 신호는 스레드 안전(Thread-safe) 큐를 통해 메인 UI 스레드 영역으로 전달된다.

4.3 프론트엔드(QML)의 선언적(Declarative) 상태 전이

UI를 담당하는 QML 스크립트는 해당 Vehicle 인스턴스를 ActiveVehicle 컨텍스트 프로퍼티로 주입(Inject)받는다. QML 코드 내부에서 activeVehicle.flying 변수를 참/거짓으로 조건 분기하여 화면 상단의 기체 상태 텍스트 배지(Badge) 색상을 녹색(동작 중)에서 회색 또는 주황색(대기/착륙 완료)으로 실시간 변경한다. 또한 슬라이더(Slider)를 조작하여 “Takeoff(이륙)” 명령 대신 “Arm(시동)” 기능이 나타나도록 문맥 메뉴(Context Menu) 항목을 능동적으로 바꾼다.

5. 관제 최적화를 위한 텔레메트리 대역폭 한계 및 타임아웃(Timeout) 방어 기작

SD 카드에 로깅되는 고주파수(High frequency) 데이터와는 다르게, 텔레메트리 대역폭(일반적으로 57600 baud rate 수준) 한계로 인해 MAVLink 시스템 상태 메시지는 보통 1Hz 또는 2Hz 속도로 느리게 보고된다. PX4 기체가 지면에 닿고 나서 GCS가 이를 인지하기까지 최대 약 1초의 시간 차이(Latency)가 발생할 수 있다.

  • Heartbeat 타임아웃 감시: 착륙과 무관하게, Vehicle 객체는 약 3.5초 이내에 HEARTBEAT이나 텔레메트리를 수신하지 못하면, 통신 단절(Link Loss) 상태로 간주하고 UI에 경고를 표시한다. 결국 기체가 착륙 중 충격이나 배터리 단절이 발생하여 메인 보드가 정지한 것인지, 아니면 정상적으로 착륙하여 MAV_LANDED_STATE_ON_GROUND를 보낸 것인지를 구별하는 논리적 추론도 GCS 아키텍처에 구현되어 있어야 한다.

6. 결론

지상 관제 시스템(GCS) 소프트웨어 아키텍처는 원격지에 존재하는 무인기의 착륙 물리 상태를 MAVLink 스트림 -> 객체 지향 파서 -> 멤버 변수 속성 캐싱 -> Qt 시그널 발행 -> UI 바인딩 이라는 일련의 파이프라인을 거쳐 정밀하게 통합한다. QGroundControl은 이 같은 구조를 통해, 여러 대의 기체(Swarm) 관제에서도 특정 기체의 이벤트 기반 상태 변화만을 추적, 최소한의 컴퓨팅 자원 오버헤드로 다수의 Vehicle 객체를 안정적으로 렌더링하고 사용자에게 직관적 통제 기능을 실시간 조력한다.
