50.5.4. 고속 비행 데이터 로깅 시스템(Logger)의 동기화 및 범용 비동기 송수신기(UART) 버퍼 처리 종료

무인항공기의 착륙 후 시스템을 안전하게 종료하는 과정에서, 물리적인 모터 정지만큼 중요한 소프트웨어적 제어 영역이 존재한다. 그것은 바로 비행 중 수집된 방대한 텔레메트리(Telemetry) 및 내부 상태(State) 데이터를 저장 매체(주로 SD 카드)에 안전하게 병합 및 동기화(Sync)하고, 통신 채널에 남아있는 찌꺼기 버퍼(Buffer)를 말끔히 비워내는 처리 절차이다. 본 절에서는 PX4-Autopilot의 ULog 데이터 로깅 시스템과 UART MAVLink 스트림이 시동 해제(Disarming) 시퀀스 동안 어떠한 방식으로 종료 절차를 밟는지 아키텍처 관점에서 서술한다.

1. 개요 및 비행 로그(ULog) 종료의 중요성

비행 데이터 로그(ULog)는 추락 분석, PID 튜닝 검증, 상태 추정기(Estimator) 진단 등 사후 분석(Flight Review)에 필수적인 “운항 기록 장치(Black box)” 역할을 한다. PX4-Autopilot은 비행 중 센서 및 제어 데이터를 일정한 고주파수(예: 200Hz ~ 1kHz 이상)로 버퍼 메모리에 적재한 뒤, I/O 스레드를 통해 파일 시스템(NuttX RTOS 기반 SD 카드)에 비동기(Asynchronous)로 기록한다.

착륙 후 배터리를 즉각적으로 분리하거나 전원 스위치를 끄게 되면, 램(RAM) 영역의 버퍼에 남아있던 수 바이트(Byte)에서 수 킬로바이트(Kilobyte)의 최종 로그 데이터가 누락되거나, 심할 경우 파일 시스템의 커럽션(File Corruption, 파일 구조 훼손)이 발생하여 해당 비행 로그 전체를 열람하지 못할 수 있다. 따라서 PX4 시스템은 시동 해제 이벤트 발생 시 로거 엔진과 범용 비동기 송수신기(UART) 채널에 능동적인 플러시(Flush) 명령을 하달하여 데이터 무결성을 보장한다.

2. PX4-Autopilot 로거 모듈의 소스 코드 및 동기화(fsync) 아키텍처

로깅 동작은 전역 파라미터 SDLOG_MODE 등에 의해 제어되며, 기본적으로 ‘시동(Armed) 시 로깅 시작, 시동 해제(Disarmed) 시 로깅 종료’ 메커니즘을 따른다.

2.1 파일 버퍼 동기화 (POSIX fsync 메커니즘)

PX4의 로거 스레드(/src/modules/logger/logger.cpp)는 커맨더 모듈에서 비행 상태 변경(vehicle_status 내 arming_state)을 수신한다. 기체의 상태가 ARMING_STATE_DISARMED로 전이되면, 로거 모듈은 더 이상의 uORB 토픽 구독을 중지하고 다음과 같은 종료 루틴을 밟는다.

  1. Drop Notification: 버퍼 오버플로우로 인해 소실된 메시지 수(Drop count)를 최종 메타데이터 블록에 기록.
  2. Write Thread 동기화: 링 버퍼(Ring Buffer)에 아직 파일 시스템으로 푸시(Push)되지 않은 찌든(Dirty) 메모리 블록을 디스크 I/O 스레드에 강제 할당.
  3. 파일 동기화와 종료: POSIX 파일 제어 시스템 명령어인 fsync()와 close()를 연쇄적으로 호출하여, 파일 할당 테이블(FAT)과 데이터 블록 패리티가 SD 카드 물리 레이어에 확실하게 각인되도록 처리한다.
/* PX4 로거 내부 I/O 버퍼 종료 구조 (Pseudo-code) */
void Logger::stop_log()
{
    // 로깅 플래그 해제 및 링 버퍼 잠금 해제
    _logging_enabled = false;
    
    // 남은 버퍼 잔여량을 디스크로 완전 복사 (Flush)
    flush_ring_buffer();
    
    // NuttX 파일 시스템 드라이버 레벨에서의 강제 동기화 지시
    if (_log_file_fd >= 0) {
        fsync(_log_file_fd);
        close(_log_file_fd);
        _log_file_fd = -1;
    }
}

3. 통신 모듈 (UART 버퍼) 강제 처리 및 MAVLink 전송 종료

ULog 기록과 별개로 기체 외부의 텔레메트리나 컴패니언 컴퓨터(ROS 2 연동 등)와 연결된 UART 인터페이스의 버퍼 역시 정리가 필요하다.

3.1 MAVLink 노드의 대기 큐 플러싱(Flushing)

관제 시스템(GCS)으로 데이터를 전송하는 MAVLink 데몬(/src/modules/mavlink/mavlink_main.cpp) 패킷 구조상 시스템 내부가 착륙 및 Disarm 상태로 전환되면, MAVLink는 대규모 데이터 스트리밍 토픽(예: 고주파 IMU 피드백) 송출 대역폭을 급격히 축소시킨다. 이때 직렬 통신 포트 하드웨어(UART TX Buffer)에 병목되어 대기 중이던 패킷들을 강제로 배출시키거나 만료 처리함으로써 GCS 쪽에 HEARTBEAT(base_mode: Disarmed 적용) 및 최종 STATUSTEXT(“Armed time: Xs”)가 가장 최우선적으로 지연 없이 도달하도록 버퍼의 제어 우선순위를 교착 회피 관점에서 재조정한다.

5. Ardupilot (DataFlash) 대비 아키텍처적 차이

비행 데이터를 로깅하는 철학은 비슷하나 파일 구조 처리 방식에서 차이가 발생한다.

  • Ardupilot (DataFlash 로깅): 과거 고속 플래시 메모리 보드 설계를 계승한 DataFlash 방식을 일부 상속받아 사용하며, 내부 C++ 클래스인 AP_Logger를 통해 마이크로초 단위의 이벤트 타이밍 태그를 부여한다. 동기화 관점에서 Log_Disarm 이벤트를 기록하고 점진적으로 버퍼 큐를 해소하는 방식을 선호한다.
  • PX4-Autopilot (ULog): ULog는 자체 서술적(Self-describing) 파일 포맷이다. 파일 헤더(Header)에 파라미터와 메시지 형식을 모두 정의해 두기 때문에 비행 중 파일이 강제로 끊어지더라도 파일 복구 스크립트로 상당량 복원이 가능하도록 아키텍처 자체가 강건하게 설계되어 있다. 그럼에도 불구하고 NuttX 상의 파일 fsync 호출 구조를 로거 스레드 최상단으로 분리시켜, 메인 제어 루프가 SD 카드 쓰기 지연(I/O Blocking)의 영향을 전혀 받지 않도록 비동기 아키텍처에 극한의 튜닝이 집중되어 있는 것이 큰 차이점이다.

6. QGroundControl (QGC) 연동 기법 및 관제 데이터 무결성 검증

비행이 종료되고 시스템이 로버 버퍼 처리를 완료하면, QGroundControl을 통해 비행 기록을 관제 측으로 온전하게 마이그레이션할 수 있다.

  1. 다운로드 세션 준비 검사: QGC 화면 우측 상단의 “Analyze Tools” -> “Log Download” 메뉴 진입 시, QGC는 MAVLink LOG_REQUEST_LIST 메시지를 통해 기체 SD 카드 내의 ULog 파일 목록을 호출한다. 정상적인 close() 시스템 콜이 이루어진 파일만이 온전한 파일 크기 정보(Size)를 갖추고 디렉토리 엔트리에 등록되어 GCS에 안전하게 표출된다.
  2. 데이터 소실에 대한 관제 알람: 조종자가 fsync() 완료 시간(약 1~2초)을 기다리지 않고 배터리를 강제로 차단했을 경우, 파일 시스템 복구가 필요할 수 있다. 향후 전원 재설정 시 QGC가 기체와 연결될 때 시스템 내부의 “File System dirty” 또는 “MicroSD format required” 계열의 경고음을 방출하여 오퍼레이터의 주의를 환기시킨다.

7. ROS2 기반 에이전트의 로깅 병합 체계 (uXRCE-DDS)

로봇 운영체제(ROS2)를 연결하여 자율화된 로보틱스 관제를 수행하는 시스템은 로깅의 주체가 이중화(Dual-logging)되는 경우가 잦다. (하위 계층: PX4의 ULog, 상위 계층: ROS 2의 rosbag2).

이때 ROS2 상위 알고리즘은 PX4의 착륙 종료 동기화(Disarmed 상태)를 관측하는 즉시 자신(ROS 2 Node)의 rosbag2 레코더를 동시에 정지시키고 스토리지 동기화 메서드를 호출하여야, 향후 Time-sync 로그 병합(예: ULog의 센서 데이터 타임스탬프와 rosbag2 상의 VIO/Computer Vision 인지 데이터 타임스탬프 동기화 정렬)을 실행할 때 끊어진 엔드포인트(End-point) 불일치로 인한 오차를 시스템적으로 방어할 수 있다.
