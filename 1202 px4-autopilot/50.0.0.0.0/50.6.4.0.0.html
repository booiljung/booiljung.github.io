50.6.4 GCS 백엔드 통신 매니저의 시동 해제(Disarm) 이벤트 구독 및 임무 제어기(Mission Controller)의 완전 종료(Mission Complete) 판별을 위한 분산 상태 머신 동기화

비행 임무(Mission)의 최종 단계로서 기체가 지면에 안착(Landed)한 이후, 물리적인 프로펠러 구동을 중단하는 시동 해제(Disarm) 이벤트와 논리적인 임무의 완전 종료(Mission Complete) 판별이 이루어져야 한다. 이 과정은 단순히 단일 상태 플래그의 변경을 넘어, 무인항공기 기체 내부의 비행 제어기(FC, Flight Controller) 시스템과 지상 관제 시스템(GCS, Ground Control Station) 간의 고도로 동기화된 시스템적 절차를 거친다. 본 절에서는 PX4-Autopilot 환경 하에서 QGroundControl(QGC)의 백엔드 통신 매니저 및 임무 제어기(Mission Controller)가 어떻게 기체 측의 시동 해제 이벤트를 구독하고, 분산 상태 머신(Distributed State Machine) 측면에서 임무의 완벽한 종료를 판별하는지 그 아키텍처를 상세히 서술한다. (적용 기준: PX4 v1.14 이상, QGC v4.3 이상)

1. 분산 상태 머신(Distributed State Machine) 개요

단일 프로그램 내의 상태 머신과 달리, 기체(UAV)와 지상 관제 시스템(GCS) 간의 결합은 물리적, 공간적으로 분리되어 있으며 무선 데이터 링크 트러블로 인해 메시지 유실 가능성이 항상 존재하는 비동기 분산 통신 환경에 놓인다.
따라서 PX4 펌웨어 측의 Commander 모듈과 QGC 내부의 Vehicle 및 MissionController 모듈은 서로 독립적인 상태 머신을 유지하면서도 MAVLink라는 브리지(Bridge)를 통해 논리적 동기화를 꾀한다.

1.1 기체(PX4) 측 상태와 GCS(QGC) 측 상태의 맵핑 관계

  • 기체(PX4) 측 상태: 임무 인덱스, 현재 착륙 상태 플래그(landed_state), 그리고 가장 강력한 안전 하드웨어 개입 상태인 무장/무장 해제(armed / disarmed) 상태를 갖는다.
  • GCS(QGC) 측 상태: MissionController 객체는 다운로드된 임무 항목(WPs) 리스트, 활성 임무 진행 상황, 기체의 시동(Arm) 상태 이벤트를 기반으로 프론트엔드 UI에 임무 완료(Mission Complete) 메시지를 띄우기 위한 상태 변수를 관리한다.

2. 임무 제어기의 완전 종료(Mission Complete) 판별 조건

지상 관제 시스템의 UI 단에 조종자를 향해 “Mission Complete” (임무 완료) 라는 안내를 표출하고, 기체에 대한 후속 임무(예: 새로운 웨이포인트 업로드)를 조작할 수 있도록 록(Lock)을 해제하기 위해서는 단순히 마스킹 된 단일 플래그에 의존할 수 없다.
QGC 내부의 C++ MissionController 로직은 다음의 복합(Heuristic) 조건을 충족할 때 비로소 임무 완전 종료를 판별한다.

2.1 다중 판별 조건 알고리즘

  1. 임무 시퀀스 검증 (Mission Sequence Completion)
    : 수신 중인 MAVLink MISSION_CURRENT 메시지에 포함된 웨이포인트 시퀀스 번호(Sequence seq)가 GCS 내에 업로드/동기화된 임무 리스트의 최종 항목 인덱스와 일치하거나 그 이상일 것.
  2. 비행 모드 및 지면 상태 검증 (Flight Mode & Landed State)
    : 기체의 비행 모드(Vehicle::flightMode())가 착륙 관련 위상(예: Land 또는 자동 착륙 속성을 포함한 Mission)을 수행 완료하였고, PX4 Extended Sys State의 지면 접지 판별 파라미터가 착륙됨(MAV_LANDED_STATE_ON_GROUND)으로 세팅될 것.
  3. 최종 확인 조건 - 시동 해제 이벤트 구독 (Disarm Signal Subscription)
    : PX4 측에서 프로펠러 회전을 멈추고 HEARTBEAT 메시지의 base_mode 마스크에서 무장 비트(MAV_MODE_FLAG_SAFETY_ARMED)가 해제될 것. 즉, GCS 백엔드의 시그널인 Vehicle::armedChanged(false) 이벤트 수신.

3. 시동 해제(Disarm) 이벤트 구독 및 통신 매니저 동작

비행 제어기가 착륙을 감지(Land Detector 트리거)한 후 자동으로, 혹은 사용자의 수동 개입에 의해 COMMAND_LONG(MAV_CMD_COMPONENT_ARM_DISARM, 0)이 전달되어 시동 해제가 발생하면 다음의 데이터 스트림이 발생한다.

3.1 MAVLink 수신 및 속성 갱신 (QGC 백엔드)

QGC 내부 백엔드 MAVLink Protocol Manager 블록은 주기적인 HEARTBEAT 메시지를 수신하여 파싱한다.

// QGroundControl / Vehicle.cc 내부 로직 요약
void Vehicle::_handleHeartbeat(const mavlink_message_t& message) {
    mavlink_heartbeat_t heartbeat;
    mavlink_msg_heartbeat_decode(&message, &heartbeat);

    bool newArmed = heartbeat.base_mode & MAV_MODE_FLAG_SAFETY_ARMED;
    if (_armed != newArmed) {
        _armed = newArmed;
        emit armedChanged(_armed); // 메타 객체 시그널 방출
    }
}

여기서 발생한 C++ 시그널 armedChanged(bool)는 QGC 전역에 등록된 리스너(Listener), 특히 MissionController와 QML 프론트엔드로 파급(Propagation)된다.

3.2 임무 제어기(Mission Controller)의 상태 동기화 응답

MissionController 인스턴스는 해당 armedChanged(false) 시그널과 이전의 임무 마지막 WP 도달 정보 등을 결합(And 논리합)하여 내부 메서드의 _checkMissionComplete() 펑션을 수행한다.
성공적으로 모든 조건이 파싱되면, GCS 프레임워크는 내부적인 임무 객체 메모리 릴리스, 진행률 초기화, 비행 로그 다운로드 유도 모달(Modal) 팝업 등을 순서대로 실행하는 트리거를 당기게 된다.

5. 결론 및 관제 구현의 핵심 포인트

GCS 백엔드 통신 매니저의 임무 완전 종료 판별 로직은 비동기 네트워크 특성에 대비한 견고한(Robust) 인터록(Interlock) 장치이다.
일부 상용 GCS나 초급 단계의 커스텀 관제 앱의 경우 단순히 MISSION_ITEM_REACHED 메시지 수신 즉시 임무 완료 모달 팝업을 띄우는 실수를 범하곤 한다. 이는 만에 하나 기체가 착륙 전 호버링 상태에 머물러 있거나 돌발 강풍에 의해 시동이 안 꺼진 위험 상태에 있을 때 조종자의 주의를 분산시켜 대형 사고로 이어질 수 있다.

PX4-QGC 아키텍처는 이를 방지하고자, MAVLink Heartbeat의 시동 비트 마스크의 변경 여부(Disarm 이벤트)를 궁극적인 안전 클리어런스(Clearance)로 구독함으로써 **물리적 동역학 정지 상태(분산 머신의 기체 측 관점)**와 **임무 리스트 종결(분산 머신의 GCS 측 관점)**을 논리적으로 크로스 동기화시킨다. 따라서 현장 실무에서 독자적 GCS를 구성할 때에도 QGC 내부 MissionController의 다중 판별 철학을 아키텍처 레벨에서 적극 차용해야 한다.
