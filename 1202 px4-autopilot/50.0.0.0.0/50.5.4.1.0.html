50.5.4.1 SD 카드 컨트롤러 내부에 캐싱(Caching)된 ULog 포맷 비행 데이터의 강제 플러시(Flush) 및 파일 할당 테이블(FAT) 메타데이터 업데이트 보장 루틴

드론이 성공적으로 착륙(Landing)하고 지면 접촉 상태(Landed State)가 확정된 이후, 비행 제어기(Flight Controller, FC)의 소프트웨어 아키텍처에서 가장 중요하게 처리해야 할 물리적 동작 중 하나는 바로 내장된 SD 카드에 비행 데이터 로그를 안전하게 보존하는 과정이다. PX4-Autopilot은 비행 중 발생하는 모든 상태 정보와 센서 데이터를 고속의 ULog 포맷으로 기록(Logging)한다. 그러나 기록 효율을 극대화하기 위해 데이터는 SD 카드로 즉각 기록되지 않고 시스템 메모리 버퍼(Buffer)나 SD 컨트롤러 내부 캐시(Cache)에 임시 저장된다. 기체가 안전하게 착륙을 완료하여 전원 차단(Power-off)이 허용되는 상태에 돌입할 때, 이 캐싱된 데이터를 물리적인 플래시 메모리에 최종적으로 기록(강제 플러시, Forced Flush)하고 파일 할당 테이블(File Allocation Table, FAT)의 메타데이터를 갱신하는 보장 루틴은 필수적이다.

본 절에서는 착륙 직후 시퀀스에서 수행되는 SD 카드 버퍼 플러시 동작 원리와 FAT 메타데이터 갱신 알고리즘, 그리고 이를 보장하기 위한 PX4-Autopilot의 내부 루동을 심층적으로 분석한다. (해당 내용은 주요하게 PX4 v1.13 및 v1.14 버전을 기준으로 서술되며, 구동되는 실시간 운영체제인 NuttX RTOS 기반의 파일 시스템(VFS) 동작을 포함한다.)

1. ULog 데이터 로깅과 캐싱 메커니즘의 근본 원리

비행 중 제어기는 수백 헤르츠(Hz) 이상의 주기로 비행 역학(Flight Dynamics), 센서 계측 데이터, 에스티메이터(Estimator) 상태, 제어기 출력 등 다양한 uORB(Micro Object Request Broker) 메시지 토픽을 발행한다. PX4의 로거(Logger) 모듈은 이러한 고주파 형태의 uORB 토픽을 지속적으로 구독(Subscribe)하여 ULog 바이너리 포맷으로 묶어낸다.

그러나 SD 카드(일반적으로 NAND 플래시 메모리 기반)의 데이터 기록 속도는 램(RAM) 영역에 비해 현저히 느리며 가비지 컬렉션(Garbage Collection), 마모 평준화(Wear Leveling) 등의 내부 컨트롤러 작업으로 인해 일시적인 병목 지연(Latency)이 발생할 수 있다. 이러한 입출력(I/O) 지연을 극복하지 못하면 주요 비행 루프(Main Flight Loop)의 차단(Block) 현상인 ’드롭아웃(Dropout)’이 발생하기 때문에, 비행 제어 시스템은 다음과 같은 다중 캐싱을 이용한다.

  1. 로거 모듈의 RAM 링 버퍼(Ring Buffer): 로깅 스레드(Logging Thread)는 우선 데이터를 FC의 주 메모리에 할당된 고정 크기 버퍼(일반적으로 수십~수백 KB)에 적재한다.
  2. NuttX VFS (Virtual File System) 버퍼 캐시: 로거가 write() 시스템 콜을 호출할 때, 데이터를 즉시 물리 장치에 기록하지 않고 OS 레벨의 버퍼(Page Cache)에 임시로 둔다.
  3. SD 컨트롤러 하드웨어 캐시: SD 카드 하드웨어 자체에도 호스트로부터 전송받은 쓰기 커맨드를 임시로 담아두는 내부 SRAM이 존재한다.

비행 중단이나 착륙 직후 모터 시동 해제(Disarm) 시점에 사용자가 기체 전원을 즉각 차단할 경우, 위 세 가지 계층에 머물고 있는 데이터들이 물리 소자에 쓰이지 못하고 손실될 가능성이 매우 높다. 따라서 시스템은 정상적인 착륙 프로세스의 마지막 단계로서 명시적인 동기화 명령(fsync)을 내려야만 한다.

2. FAT32 파일 시스템의 구조와 메타데이터 손상의 치명성

대부분의 소형 무인기 SD 카드는 호환성을 위해 FAT32(File Allocation Table 32) 형식으로 포맷되어 있다. 비행 데이터 로그는 단일 파일(예: log001.ulg 등) 형태로 용량이 점진적으로 늘어나며 쓰인다. 이때 파일의 메타데이터(Metadata)란 파일의 실제 크기, 수정 시간, 디렉토리 엔트리(Directory Entry), 클러스터 체인(Cluster Chain) 정보 등을 의미한다.

데이터 쓰기 작업 시, 데이터 블록 자체는 SD 카드에 써지더라도 해당 파일의 최종 크기 정보를 갖는 디렉토리 엔트리 영역이나 파일 할당 테이블(FAT 영역)이 갱신되지 않으면, 파일 시스템을 재마운트 시 운영체제가 해당 파일의 실제 크기를 0 바이트로 인식하거나 파일 자체가 손상(Corrupted)된 것으로 취급할 수 있다. 즉, 하드웨어 적으로는 기록이 된 데이터일지라도 논리적으로 파일을 닫고(Close) 메타데이터를 확정하는 플러시(Flush) 보장 루틴이 호출되지 않으면 방대한 ULog 데이터를 온전히 복원할 수 없게 된다.

3. 착륙 후 ULog 강제 플러시 보장 아키텍처 (Mermaid 도표)

PX4-Autopilot에서 로거 메인 스레드 커맨드 루프와 하위 시스템에서 일어나는 강제 플러시 및 파일 닫기 루틴은 다음과 같은 구조로 상호작용한다.

    participant LM as Logger Module (로거)
    participant NX as NuttX VFS (파일 시스템)
    participant SD as SDIO / SD Card HW

    Note over CM, LM: 지면 접촉 및 착륙(Landed) 감지
    CM->>LM: 로깅 종료 명령 또는 Disarm 이벤트 전송
    Note over LM: 남은 ULog RAM 버퍼 처리

    Note over NX: VFS 캐시 비우기 및 FAT 엔트리 갱신
    NX->>SD: CMD24/25 (Block Write) 데이터 전송

    NX-->>LM: fsync() 완료 반환

4. 강제 플러시 루틴의 소프트웨어 구현 분석

실제 PX4 코드베이스 내 로거 모듈(주로 src/modules/logger/logger.cpp 파일 내)의 구현 로직을 살펴보면 강제 플러시를 보장하는 세부 스텝을 확인할 수 있다. 비행 제어기가 물리적으로 Landed 상태로 전환될 때, 혹은 사용자나 지상 관제 시스템(GCS)으로부터 명시적으로 비행 기록 정지(Stop Logging) 명령이 하달되었을 때 작동된다.

4.1 램 버퍼(RAM Buffer) 누수(Drain) 처리

로거 스레드가 종료 권한을 위임받으면 제일 먼저 uORB 토픽 구독을 중지한다. 그 후, 메모리 버퍼 큐에 아직 SD카드로 내려보내지 못한 잔류 데이터 청크(Data Chunk)들이 있는지 확인한다. 이 청크들을 파일 스왑(Swap) 영역이나 POSIX 호환 API인 write()를 호출하여 NuttX 커널 계층으로 모두 밀어낸다(Drain).

4.2 fsync() 시스템 콜을 통한 동기화

로거는 잔류 버퍼 기록을 마친 후 fsync(fd) (또는 fflush() 및 fclose()) 시스템 콜을 호출하여 OS 계층에서 지연된 쓰기 블록(Dirty Pages)을 즉시 블록 디바이스 드라이버로 보내도록 강제한다.

  • 동작 매커니즘: NuttX의 FAT32 파일 시스템 드라이버는 fsync가 호출되면 디렉토리 엔트리의 파일 생성 날짜와 파일 크기를 최신화한다. 이 메타데이터 수정 내용 자체도 내부 버퍼에서 SD 카드 영역으로 덮어씌워져 기록된다.

4.3 파일 포인터의 논리적 닫힘(File Closure)

강제 플러시 후, close(fd)를 안전하게 수행하여 시스템에 열려있는 파일 식별자(File Descriptor) 자원을 해제시킨다. 이 일련의 루틴이 반환(Return)에 성공해야만, 파일이 완벽하게 닫힌 온전한 단일 ULog 파일로서 확정된다. 사용자가 기체 전원선을 뽑아도 데이터 오염 확률은 사실상 없게 된다.

5. 지연 쓰기(Delayed Write) 시의 Failsafe 고려사항

강제 플러시 작업 자체는 마이크로세컨드(us)에서 수 밀리세컨드(ms) 범위 안에서 끝나지만, SD 카드의 물리적 상태(예: 불량 섹터 우회나 내부 가비지 컬렉션 타임)에 따라 수백 밀리세컨드(Latency Spikes)가 걸리기도 한다.
착륙 후 Disarm 명령이 내려지고 사용자가 거의 동시에 배터리를 분리하는 극한의 경우, 플러시 루틴이 완료되지 못한 채로 전원이 끊어져(Power loss) 앞선 메타데이터가 파괴될 수 있다. 이를 방지하기 위해 PX4는 아래와 같은 방어 기작(Failsafe)을 마련해 두었다.

  • 배터리 컷오프 지연(보완 회로): 고급 자율주행 드론 하드웨어(고급형 Pixhawk 등)는 메인 전원이 차단되더라도 하드웨어적으로 짧은 시간(수십 ms ~ 수 초 내외)을 버틸 수 있는 캐패시터 기반 전원 유지 장치를 포함하여 FAT이 손상되지 않게 전원을 연장한다.
  • 파일 시스템 저널링/복구 유틸리티: FAT32 자체가 저널링(Journaling)을 자체 지원하지 않으므로, 손상된 .ulg의 경우 파일의 끝(EOF 마커)을 정상적으로 닫지 못했을 때 호스트 GCS에서 이를 부분적으로 파싱(Parsing)할 수 있는 에러 보완 알고리즘을 외부 분석 툴(비행 데이터 리뷰 툴)에 탑재한다.
  • 정기적인 백그라운드 프레임 단위 플러시(Background Flush): 착륙 시 강제 플러시에만 전적으로 의존하는 것은 위험도 요소를 가지므로 상시적인 백그라운드 태스크에서 로깅 중 일정한 임계 타임스탬프 간격마다 간헐적으로 메타데이터를 사전 갱신 해놓는 방식도 병합 활용된다.

6. 결론 및 요약

SD 카드 모듈 내부에 존재하는 버퍼, NuttX VFS 레벨의 버퍼, 그리고 PX4 Logger 스레드의 링 버퍼 간의 지연(Delay)을 완전하게 해소하는 강제 플러시 루틴 및 파일 구조 테이블 메타데이터 갱신 작업은 비행 제어 및 해석의 피드백루프에 없어서는 안 되는 중요한 마침표이다.
착륙 및 모터 Disarm을 인식하는 순간 PX4 시스템은 상위 어플리케이션(QGroundControl 등)에 종료 상태를 보고하기 전까지 잔류 중인 모든 바이너리 토픽을 플래시 메모리 영역에 강제 기록(fsync(), close())하고 디렉토리 엔트리에 실제 비행 파일의 크기를 갱신 및 봉인하여, 데이터 오염 방지(Data corruption prevention)에 대한 강력한 신뢰도를 구축하고 있다.
비행 역학 분석이나 사후 사고 원인 파악, 그리고 알고리즘 개선에 있어서 오류 없이 종결된 완벽한 ULog 파일의 생성은 자율비행 공학의 가장 기본이 되는 데이터 자산 보증 행위이다.
