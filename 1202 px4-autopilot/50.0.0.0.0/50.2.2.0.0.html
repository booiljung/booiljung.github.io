50.2.2 Commander 모듈의 중앙 상태 머신(Central State Machine) 비행 모드 천이 평가 논리 회로

PX4-Autopilot 아키텍처에서 기체의 현재 비행 상태(예: Arming, Disarming) 및 비행 모드(예: Manual, Altitude, Position, Mission, Land)를 통합적으로 관리하고 전환(Transition)을 승인하는 핵심 모듈은 Commander 스레드이다. 지상 관제 시스템(GCS)으로부터 착륙 명령(Land Command)과 같은 비행 모드 변경 요청이 uORB 미들웨어를 통해 유입될 때, Commander는 중앙 상태 머신(Central State Machine)의 평가 논리 회로를 가동하여 해당 명령이 현재 동역학적, 시스템적 안전 조건을 만족하는지 엄격히 검증한다. 본 장에서는 최근 PX4 펌웨어를 기준으로 이 평가 회로의 동작 구조를 깊이 있게 분석한다.

1. uORB vehicle_command 구독 및 초기 상태 평가

앞선(50.2.1) 단계에서 mavlink_receiver가 발행한 vehicle_command 토픽은 Commander 내의 비동기 이벤트 루프(Event Loop)에 의해 즉시 수신된다. Commander는 내부적으로 지속적인 폴링(Polling) 혹은 콜백(Callback)을 통해 명령 대기열(Command Queue)을 점검한다.

착륙 유도 명령(VEHICLE_CMD_NAV_LAND)이 감지되면, Commander는 맹목적으로 모드를 전환하지 않고 사전 필터링(Pre-filtering)을 수행한다.

  1. 커맨드 유효성 검사 (Command Validity Check): 해당 명령의 타임스탬프(Timestamp)가 허용 한계(일반적으로 수백 밀리초)를 초과하여 지연된 과거의 메시지인지 확인한다.
  2. 명령 중복 처리 방지 (Duplicate Prevention): 동일한 커맨드 식별자(Command ID)가 이미 처리 중인지 내부 트랜잭션 상태를 확인한다.

2. 모드 천이 평가 논리(Transition Evaluation Logic)

명령이 유효하다고 판명되면 Commander의 메인 천이 평가 회로(Transition Evaluation Circuit)인 set_main_state 혹은 set_nav_state 계열의 C++ 함수가 호출된다. 이 과정은 논리곱(AND) 기반의 복합적인 상태 검증 로직으로 구성된다.

2.1 주요 검증 파라미터(Condition Parameters)

착륙 모드로의 천이를 위해 Commander는 여러 uORB 센서 토픽과 기체 상태 변수를 논리적으로 결합하여 판단한다.

  • 무장 상태 (Arming State): 기체가 현재 무장(Armed) 상태이며, 모터가 활성화되어 비행 중(vehicle_status.arming_state == ARMING_STATE_ARMED)이어야 착륙 절차가 유효하다. 만약 디스암(Disarmed) 상태에서 착륙 명령이 유입되면, 이는 논리적 모순이므로 즉시 거부(Rejected)된다.
  • 위치 락(Position Lock) 및 기본 유도 정보: 착륙 제어기(Landing Controller)는 정확한 수직 강하를 위해 로컬 고도 추정치 및 속도 벡터가 필요하다. EKF2/ECL 모듈이 발행하는 vehicle_local_position의 검증 비트(Status Bit)가 참(True)인지 확인한다.
  • 배터리 및 안전 장치(Failsafe) 충돌 차단: 만약 현재 기체가 심각한 배터리 부족(Critical Battery Failsafe)으로 인해 강제 비상 착륙이나 비행 종료(Flight Termination) 절차에 진입한 상태라면 일반적인 GCS의 착륙 명령보다 시스템의 안전 모드가 우선순위(Preemption Priority)를 가진다.

결과적으로 평가 논리는 다음과 유사한 불리언(Boolean) 검증 수식을 따른다.
\text{Is Transition Valid} = (\text{Is Armed}) \land (\text{Local Position Valid}) \land \neg(\text{Failsafe Overridden})

2.2 상태 머신 전환 흐름도 (Mermaid)

다음은 Commander 내에서 착륙 명령이 처리되고 상태가 전이되는 과정을 나타내는 상태도(State Diagram)이다.

stateDiagram-v2
    [*] --> Polling_uORB: Wait for vehicle_command
    Polling_uORB --> Validity_Check: VEHICLE_CMD_NAV_LAND Received
    
    Validity_Check --> Evaluation_Circuit: Command is fresh & valid
    Validity_Check --> Command_Rejected: Command is stale

    state Evaluation_Circuit {
        Check_Arming: Check if Armed
        Check_Position: Check EKF Local Position Valid
        Check_Priority: Check Preemption (e.g. Failsafe)
        
        Check_Arming --> Check_Position
        Check_Position --> Check_Priority
    }
    
    Evaluation_Circuit --> Mode_Transition: All conditions MET
    Evaluation_Circuit --> Command_Denied: Any condition FAILED
    
    Mode_Transition --> Update_Vehicle_Status: Set nav_state = NAVIGATION_STATE_AUTO_LAND
    Update_Vehicle_Status --> Publish_Ack: COMMAND_ACK (Accepted)
    Command_Denied --> Publish_Ack: COMMAND_ACK (Denied/Failed)

3. 모드 전이 확정 및 결과 회신 (Acknowledgement)

모든 동역학적, 시스템적 조건이 충족되어 상태 천이가 확정(Approved)되면, Commander는 내부 vehicle_status 구조체의 특정 멤버(예: nav_state)를 NAVIGATION_STATE_AUTO_LAND로 변경한다. 이후 아래의 후속 조치를 원자적으로(Atomically) 실행한다.

  1. 상태 버스 브로드캐스트: 변경된 vehicle_status 구조체를 uORB를 통해 발행하여, 즉각적으로 Navigator 모듈이 경로 비행 로직을 강하(Descent) 유도 로직으로 초기화할 수 있도록 촉발(Trigger)한다.
  2. ACK 피드백: GCS의 제어 신뢰성을 보장하기 위해 mavlink_commander (혹은 관련 응답 모듈)를 통해 COMMAND_ACK (MAV_RESULT_ACCEPTED) 메시지를 MAVLink로 송출한다.

4. Ardupilot 대비 아키텍처 차이

Ardupilot의 비행 모드 전환(예: set_mode(Mode::Number::LAND))은 각 모드 객체 내부 스코프에 분산된 init() 가상 함수(Virtual Function)와 메인 루프에서 순차적으로 호출되며 평가된다. 즉, 모드 전환 로직이 분산 객체 지향 형태로 각각의 ModeLand.cpp 같은 파일에 정의된다.
반면 PX4-Autopilot은 Commander라는 철저하게 격리된 단일 시스템 데몬(Daemon)이 중앙 통제 센터 역할을 수행하며, 거시적인 모드 전환 로직과 사전 안전 검증을 한 곳에서 독점적으로 처리한다. 이는 복잡한 다중 에이전트 간의 모드 경쟁을 막고 검증 로직을 추적하기 용이하다는 장점을 지닌다.

5. 요약 및 실무 적용 시 고려사항

Commander의 중앙 상태 머신은 외부의 비정상적이거나 악의적인 명령으로부터 기체의 물리적 무결성을 보호하는 “최종 방어선(Last Line of Defense)“이다. 연구자나 관제 소프트웨어 개발자는 MAVLink 명령 송출 시 왜 COMMAND_ACK에서 거부(Denied) 상태가 반환되는지 분석하기 위해 기체의 EKF 추정 상태(Position/Velocity Validity) 및 Failsafe 상태를 ULog 혹은 QGroundControl의 MAVLink 인스펙터(Inspector)로 상시 모니터링해야 한다. 모드 천이 조건을 무단으로 우회하도록 Commander 소스코드를 수정하는 행위는 비행 안정성을 심각하게 테스팅하거나 훼손하므로 극히 제한되어야 한다.
