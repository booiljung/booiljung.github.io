50.1.2.1 명령어 식별 코드 파싱 및 대상 시스템/컴포넌트 식별자(System/Component ID) 기반의 마이크로 라우팅(Micro-routing) 알고리즘

본 절에서는 기준 버전인 PX4-Autopilot v1.14.0을 바탕으로, QGroundControl(이하 QGC) 및 ROS2 미들웨어에서 전송된 MAV_CMD_NAV_LAND 명령어가 기체 내부에서 올바른 목적지로 도달하도록 보장하는 식별자(ID) 기반의 마이크로 라우팅 알고리즘을 분석한다.

1. 개요 및 식별자(ID) 시스템의 목적

여러 대의 무인기(Swarm)가 동시에 운용되거나, 단일 기체 내에 비행 제어기(FC), 짐벌(Gimbal), 컴패니언 컴퓨터(Companion Computer) 등 다수의 독립 단위(Component)가 MAVLink 네트워크를 공유하는 복합 환경에서는, 수신된 명령어가 누구를 향한 것인지 명확히 식별하는 것이 필수적이다.

MAVLink 프로토콜은 패킷 구조의 페이로드 앞단에 target_system과 target_component라는 1바이트 크기의 식별자 속성을 포함하며, PX4의 수신 모듈은 이 식별 코드 쌍을 기반으로 패킷을 내부의 적절한 uORB 토픽으로 매핑(Mapping)하거나 다른 직렬 포트로 포워딩(Forwarding)하는 마이크로 라우팅 과정을 수행한다.

2. System ID와 Component ID의 정의

  • System ID (sysid): 네트워크 상에서 독립된 물리적/논리적 무인기 개체 또는 GCS를 식별한다. 기본적으로 PX4 기체는 1번을 사용하며, QGC는 255번을 사용한다.
  • Component ID (compid): 동일한 System ID 내의 하위 장치(Subsystem)를 식별한다. 비행 제어 역할을 하는 메인 스택은 주로 MAV_COMP_ID_AUTOPILOT1 (1번)을 사용하며, 카메라(100번), 회피 스택(196번) 등과 구분된다.
  • 브로드캐스트(Broadcast) ID: 0번. target_system이나 target_component 값이 0인 경우, 해당 네트워크 내의 모든 시스템 또는 컴포넌트가 명령을 무조건 수용해 수신해야 함을 뜻한다.

3. PX4 마이크로 라우팅(Micro-routing) 알고리즘

착륙 지시(MAV_CMD_NAV_LAND, 식별 코드: 21) 명령이 통신 포트를 통해 mavlink_receiver 모듈에 인입되면, PX4는 다음과 같은 알고리즘을 거쳐 명령 수용 여부를 결정한다.

flowchart TD
    A[GCS에서 MAVLink 명령 패킷 수신] --> B{target_system == 본 기체의 sysid?<br>OR target_system == 0?}
    B -- 아니오 --> C[무시 또는 외부 MAVLink 포트로 라우팅<br>Mavlink Forwarding]
    B -- 예 --> D{target_component == 본 FC의 compid?<br>OR target_component == 0?}
    D -- 아니오 --> E[내부 카메라 등 타 컴포넌트로 전달 무시]
    D -- 예 --> F[명령어 식별 코드 확인<br>Command ID == 21]
    F --> G[uORB vehicle_command 메시지로 디코딩 후 발행]

4. 소스 코드 구현체 분석

코어 모듈인 src/modules/mavlink/mavlink_receiver.cpp 내의 handle_message_command_int 함수 구문을 살펴보면 위 로직이 소프트웨어적으로 어떻게 구현되었는지 확인할 수 있다.

해당 모듈은 mavlink_msg_command_int_decode 인라인 함수를 사용하여 메모리상의 바이트 열을 MAVLink C 구조체로 역직렬화한 다음, 기체 고유의 저장된 시스템 파라미터인 MAV_SYS_ID 및 MAV_COMP_ID 값과 패킷 내의 타겟 ID 쌍을 대조하는 비교 로직을 거친다. 이 식별자 적합성 검증 필터를 통과해야만 비로소 vehicle_command_s C++ 구조체에 데이터가 논리적으로 채워지며, command 멤버 변수에 정수 21이 할당된 후 uORB::Publication을 통해 커맨더(Commander) 스레드로 데이터가 전파된다.

만약 target_system 필드가 기체의 ID와 불일치한다면, PX4는 해당 패킷을 내부 메모리로 맹목적으로 적재하지 않고, 설정된 라우팅 테이블(Routing Table) 파라미터 체계에 따라 물리적으로 연결된 다른 텔레메트리(Telemetry) 인터페이스로 패킷을 재송신(Forward)하여 메쉬(Mesh) 네트워크 체인 상의 다른 유효 기체가 명령을 적절히 전달받을 수 있도록 버퍼링 역할을 돕는다.

5. Ardupilot 대비 통신 라우팅 아키텍처의 차이

명령어 식별 및 타겟 검증을 수행하는 기본 원리는 Ardupilot의 GCS_MAVLink 클래스의 패킷 처리 프레임워크 흐름과 본질적으로 수학적 일치를 보인다. 그러나 세부 라우팅 인터페이스 생태계에서는 핵심적인 아키텍처 설계 차이가 두드러진다.

  • PX4-Autopilot: mavlink_receiver 스레드와 mavlink_main 스레드 간의 비동기 메시지 큐(Queue) 통신을 활용하며, 생성된 각 통신 인스턴스(Instance)별로 포워딩 규칙을 파라미터로 세밀하게 독립적으로 조율하는 방식이 POSIX 기반 아키텍처에 깊게 내재되어 있다. 이는 ROS2 Micro XRCE-DDS 에이전트와 MAVLink 모듈 간의 분리된 병렬 파이프라인 형성을 본질적으로 유리하게 돕는다.
  • Ardupilot: FCU 전체 시스템이 강력한 단일 거대한 프록시 라우터 구실을 하는 경향이 더 강하다. 여러 물리/논리 포트(Telem1, Telem2, USB)를 능동적으로 넘나드는 소프트웨어 포워딩 정책이 GCS_MAVLink::routing 함수 수준에서 고도로 동기적 최적화되어 있어, 별도의 라우팅 테이블 트리 설정 없이도 짐벌이나 OSD와 같은 플러그앤플레이(Plug and Play) 스마트 컴포넌트를 연동하기 쉬운 구조적 편의 철학을 지니고 있다.

결론적으로, PX4의 식별자 파싱과 독립적 라우팅 로직은 다중 에이전트(Multi-agent) 군집 비행이나 이종 기기 간 복합 미들웨어 운영 제어 측면에서 연구자 및 시스템 엔지니어에게 명확한 권한 격리 및 제어 경계를 제공하는 보안성 높은 아키텍처로 작동한다.
