50.6.3.1 현재 고도 추정치와 기준 접지면(Ground Elevation) 간의 잔여 거리 비례 연산을 통한 진행 상태바(Progress Bar) 그래픽 렌더링 스레드

지상 관제 시스템(GCS)에서 기체의 착륙은 가장 직관적이면서도 안전 지향적인 모니터링이 필요한 비행 단계이다. QGroundControl(이하 QGC)은 기체가 지면으로 접근하는 동안 조종자에게 진행 상태바(Progress Bar) 형태의 UI를 제공하여 하강 절차의 진척도를 시각화한다. 본 절에서는 원격 측정(Telemetry) 데이터로 수신된 기체의 ’현재 고도 추정치’와 지형 데이터 또는 거리 센서에 의해 산출된 ‘기준 접지면(Ground Elevation)’ 정보를 기반으로 어떻게 잔여 거리를 비례 연산하고, 이를 Qt/QML 프레임워크의 렌더링 스레드(Rendering Thread)상에서 부드러운 그래픽으로 표출하는지 그 내부 아키텍처를 상세히 분석한다. (기준: PX4-Autopilot v1.14 이상, QGroundControl v4.3 이상)

1. 잔여 거리 비례 연산 메커니즘과 상태 머신

기체가 착륙(Land) 명령을 수신하여 하강을 시작하면, QGC 내부의 기체(Vehicle) 객체 속성은 착륙 초기 고도를 스냅샷(Snapshot)으로 저장하여 진행률 연산의 분모를 구성한다. 이후 지속적으로 갱신되는 GLOBAL_POSITION_INT 또는 라이다(LiDAR) 기반의 DISTANCE_SENSOR MAVLink 메시지를 바탕으로 현재 고도 추정치(H_{est})를 업데이트하며, 목표 기준 접지면(H_{gnd})까지의 잔여 거리(D_{rem}) 비례식을 연산한다.

1.1 거리 비례식의 수학적 모델링

착륙 진행률을 P라고 할 때, 연산은 0.0 에서 1.0(또는 0% ~ 100%) 범위의 정규화된 값(Normalized value)을 반환해야 한다.
착륙 모드 진입 지점의 고도를 H_{init}라 할 때, 임의의 시점 t에서의 진행률 P(t)는 다음 수식으로 표현된다.

P(t) = \min \left( 1.0, \max \left( 0.0, \frac{H_{init} - H_{est}(t)}{H_{init} - H_{gnd}} \right) \right)

예외 처리 로직(Edge-case handling):
바람이나 외란으로 인해 일시적으로 기체가 초기 착륙 고도(H_{init})보다 상승하거나 지표면 데이터의 오름차순 에러가 유입되어 H_{est}(t) > H_{init} 또는 H_{init} - H_{gnd} \approx 0 이 될 가능성이 존재한다. QGC 백엔드 로직은 분모가 0이 되거나 진행 상태가 음수로 떨어지는 것을 방지하는 클리핑(Clipping) 로직을 포함한다. Ardupilot용 GCS(Mission Planner)의 경우 고도 프로파일을 상대 고도(Relative Altitude) 단일 축으로 갱신하는 경향이 있으나, PX4 최신 아키텍처용 QGC는 지형 데이터 모델(Terrain Extension)과 연동하여 H_{gnd}를 동적으로 반영하는 점이 특징이다.

2. QML 씬 그래프(Scene Graph)와 그래픽 렌더링 스레드의 분리

연산된 진행률 데이터는 화면에 그려지기 위해 Qt 프레임워크의 그래픽 파이프라인(Graphics Pipeline)을 통과해야 한다. Qt 구동 환경은 크게 메인 스레드(Main/GUI Thread)와 렌더링 스레드(Render Thread)로 분리되어 구동되며, 이는 UI의 응답성과 프레임 레이트(Frame Rate)를 보장하기 위한 핵심 구조이다.

2.1 메인 스레드(Main Thread) - 데이터 바인딩 로직

MAVLink 처리 스레드에서 생성된 데이터가 Vehicle C++ 객체의 Q_PROPERTY로 반영되면, NOTIFY 시그널이 발동하여 QML 엔진에 변경 사실을 알린다. 이 시그널 처리는 프로그램의 메인 스레드에서 이루어지며, QML 엔진은 자바스크립트(JavaScript) 평가기 내의 프로퍼티 바인딩(Property Binding) 트리를 갱신한다.

2.2 렌더링 스레드(Render Thread) - 하드웨어 가속 렌더링

데이터 속성값 처리가 완료되면 Qt는 렌더링 그래프(Scene Graph) 단계를 준비한다.
메인 스레드에서 QML DOM 트리의 시각적 노드 속성(크기, 색상, Y 좌표 시프트 등) 변경점이 도출되면, 이 변경 사항(State Changes)을 렌더링 스레드의 씬 그래프(Scene Graph)로 동기화(Synchronization)한다. 이후 메인 스레드는 즉각 비지니스 로직 처리로 복귀하고, 렌더링 스레드는 OpenGL, Vulkan, 또는 Metal 등 기반 하드웨어 가속 API를 호출하여 프레임버퍼(Framebuffer)에 픽셀을 Draw한다. 이 병렬 아키텍처 덕분에 대용량 원격 측정 데이터가 실시간으로 수신되더라도 애니메이션의 멈춤(Jank) 현상이 발생하지 않는다.

3. 시각적 진행 상태바(Progress Bar)의 프론트엔드 구현 사례

착륙 잔여 거리가 시각화되는 컴포넌트 내부 로직은 QML에서 제공하는 시각적 애니메이션 클래스와 커스텀 페인팅(Custom Painting) 로직을 결합하여 구성된다. 진행 상태바의 값 갱신 시 시각적인 부드러움(Smoothness)을 유지하기 위해 선형 보간 애니메이션이 필수적으로 추가된다.

// LandingProgressBar.qml 구현 예제
import QtQuick 2.15
import QtQuick.Controls 2.15
import QGroundControl.Vehicle 1.0

ProgressBar {
    id: landingProgress
    width: 20
    height: 150
    orientation: Qt.Vertical
    from: 0.0
    to: 1.0
    
    // activeVehicle 객체로부터 고도 관련 C++ 프로퍼티 수신
    readonly property real currentAlt: activeVehicle.altitudeRelative.value
    readonly property real groundAlt: activeVehicle.terrainAltitude.value
    property real initialAlt: 0.0
    
    // 비행 모드가 Land로 변경되는 시점을 포착하여 initialAlt 설정
    Connections {
        target: activeVehicle
        function onFlightModeChanged() {
            if (activeVehicle.flightMode === "Land") {
                landingProgress.initialAlt = landingProgress.currentAlt;
            }
        }
    }
    
    // 잔여 거리 비례 연산 및 예외 처리
    value: {
        let totalDist = Math.max(0.1, initialAlt - groundAlt); // 0으로 나누기 방지
        let currentDist = currentAlt - groundAlt;
        let progress = 1.0 - (currentDist / totalDist);
        return Math.max(0.0, Math.min(1.0, progress)); // 0.0 ~ 1.0 클리핑
    }
    
    // Render Thread에서 처리될 부드러운 애니메이션
    Behavior on value {
        NumberAnimation {
            duration: 150 // 150ms 프레임 보간
            easing.type: Easing.OutQuad  // 사용자 친화적인 감속 곡선
        }
    }
    
    // Qt Scene Graph로 전달될 Custom 렌더링 지시자
    background: Rectangle {
        color: "#333333"
        radius: 4
    }
    contentItem: Item {
        Rectangle {
            width: parent.width
            height: parent.height * landingProgress.value
            anchors.bottom: parent.bottom // 아래에서 위로 차오르는 UI
            color: landingProgress.value >= 0.95 ? "#28a745" : "#007bff"
            radius: 4
        }
    }
}

위의 예제처럼 메인 스레드 공간에서 데이터의 논리적 연산과 분기(value 프로퍼티 도출)가 수행된 후 지정된 애니메이션 속도(Behavior on value)에 따라 중간 프레임 데이터가 파생되며, 이는 최종적으로 렌더링 스레드의 Scene Graph Node 트리로 넘어가 그래픽 카드로 명령(Draw Call)을 내리게 된다.

5. 결론

PX4를 제어하는 GCS 프론트엔드의 착륙 진행 상황 렌더링은 단순한 수치 변화의 나열이 아니다. “고도 변위의 비례 연산”, “이상치에 대한 상태 클리핑(Clipping)”, 그리고 “Qt 메인 스레드와 렌더링 스레드 간의 비동기적 씬 그래프(Scene Graph) 데이터 동기화“가 유기적으로 조합된 결과물이다.
특히 잔여 거리 도출 로직에 있어 착륙 지점의 지형 고도를 센서나 DB를 통해 실시간 보정( H_{gnd} 갱신)하는 현대적인 PX4-QGC 생태계는 구형 펌웨어/관제 도구 배포 형태보다 안전하고 부드러운 하강 경험을 제공한다. 독자적인 GCS나 모바일 기반의 관제 앱을 개발할 때에도 이처럼 UI 렌더링 스레드 분리와 백엔드 수신 모델의 분리, 그리고 애니메이션 속성 보간에 대한 지식을 숙지하여 적용해야 한다.
