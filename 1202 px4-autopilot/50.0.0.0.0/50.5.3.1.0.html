50.5.3.1. 비례-적분-미분 제어기의 적분항 누적 방지 기법(Anti-windup) 활성화 상태 강제 해제 및 누적 오차 메모리 블록(Memory Block) 영점 초기화

비례-적분-미분(PID) 제어기에서 적분(Integral)항은 정상 상태 오차(Steady-state Error)를 제거하는 핵심 요소이다. 그러나 구동기의 물리적 한계점(Saturation) 도달 시에도 오차가 존재하면, 적분항이 기하급수적으로 누적되는 ‘적분 와인드업(Integral Windup)’ 현상이 발생한다. 무인항공기가 지면에 접지하여 착륙된 상태에서는 더 이상 오차를 줄이기 위해 자세를 움직일 수 없기 때문에 와인드업이 극대화될 수 있다. 본 절에서는 이러한 상황에서 시스템의 안정을 보장하는 기법인 Anti-windup 활성 상태 강제 해제 로직 및 누적 오차 메모리 블록 영점 초기화 루틴에 대하여 설명한다.

1. 개요 및 적분 와인드업(Integral Windup)의 위험성

착륙 직후 또는 이륙 직전 지면에 닿아있는 기체는 조종기나 지상 관제 시스템(GCS)으로부터 들어오는 자세 제어 요구치(Setpoint)를 물리적으로 추종할 수 없다(지면에 고정되어 있으므로). 이때 제어 시스템이 여전히 활성화되어 있다면, 목표 자세와 실제 자세 간의 미세한 오차가 PID 제어기의 적분 연산기에 누적된다.

이 상태로 이륙을 시도하거나 스로틀이 상승할 경우, 누적된 방대한 에너지(에러 총량)가 모터로 한꺼번에 쏟아져 기체가 급격히 뒤집어지는(Flip over) 대형 사고로 이어진다. 이를 방지하기 위해 PX4 시스템은 비행 종료 시점에 명확하게 한계점(Limit) 조건과 누적된 오차 블록을 소프트웨어적으로 리셋(Reset)해야 한다.

2. PX4의 Anti-windup 아키텍처 및 강제 해제 메커니즘

PX4는 비행 제어 알고리즘의 유연성을 위해 자세 제어(Attitude Control)와 위치 제어(Position Control) 모듈이 각각 독립적인 PID 연산 객체(Object)를 소유한다. (소스 코드: src/lib/pid/pid.c 계열 및 멀티로터 제어 모듈 내부 참조)

PX4의 Anti-windup 기법은 주로 Back-calculation 방식이나 Conditional Integration(조건부 적분) 방식을 혼용하여 구현된다. 즉, 제어량이 액추에이터 포화 한도에 도달했을 때 적분을 중지시킨다. 그러나 기체가 착륙(Landed)한 것으로 판별되거나 시동 해제(Disarmed) 상태로 전이하면, 기존의 조건부 적분 플래그마저 무효화하고 메모리 레지스터 자체를 강제로 영점화(Zeroing)하는 프로세스가 호출된다.

/* PX4 내부 PID 객체의 적분항 영점 초기화 수도 코드 (Pseudo-code) */
void PIDController::reset_integral()
{
    // 1. 메모리 블록 상의 연속된 누적 오차 값(Float) 강제 영점 할당
    _integral_error = 0.0f;
    
    // 2. Anti-windup 플래그 및 클램핑(Clamping) 변수 초기화
    _is_saturated_upper = false;
    _is_saturated_lower = false;
    
    // 3. 적분 스케줄링 타이밍용 델타-t(dt) 변수 초기 지연 회로 재설정
    _last_update_time = hrt_absolute_time();
}

이 루틴은 커맨더(Commander) 모듈의 콜백과 밀접하게 연동된다. vehicle_land_detected 모듈이 착륙을 선언하면 각 제어 파이프라인(Rate Controller, Attitude Controller, Position Controller)은 일제히 reset_integral()에 준하는 내부 상태 정리 함수를 실행한다.

3. 메모리 블록 플러시 및 Anti-windup 다이어그램 (Mermaid)

비행 중 지속적으로 제어 루프를 도는 PID 제어기가 기체의 착륙 이벤트를 분기점으로 삼아 어떠한 방식으로 루프를 차단하고 메모리를 클리어하는지를 나타낸 상태 전이도이다.

stateDiagram-v2
    [*] --> Standby (Disarmed)
    Standby (Disarmed) --> Flying (Armed): Takeoff
    
    state Flying (Armed) {
        Normal_Integration --> Windup_Detected: Actuator Saturation Reached
        Windup_Detected --> Anti_Windup_Active: Clamp Integral Term
        Anti_Windup_Active --> Normal_Integration: Error Reverses / Saturation Resolved
    }
    
    Flying (Armed) --> Landing_Detected: vehicle_land_detected == TRUE
    
    state Memory_Flush_Sequence {
        Landing_Detected --> Suspend_Integration: 1. 조건부 적분 연산 강제 중지
        Suspend_Integration --> Clear_Memory_Block: 2. 적분항 누적 변수(_int_err) = 0.0f
        Clear_Memory_Block --> Disable_Anti_Windup_Flag: 3. 포화(Saturation) 감지 플래그 해제
    }
    
    Memory_Flush_Sequence --> Standby (Disarmed): System Reset Complete

4. Ardupilot 대비 Anti-windup 제어기 초기화 차이

두 플랫폼(PX4 및 Ardupilot) 모두 드론의 안전성을 우선시하므로 Anti-windup 기법을 적용하나, 구현의 중심점과 메모리 관리 로직에 있어 설계 철학의 차이를 보인다.

  • Ardupilot (PI_Controller / AC_AttitudeControl): relax_attitude_controllers()와 같은 범용 이완(Relaxation) 함수를 상위 클래스 계층에서 호출함으로써 누적 오차를 점진적으로 소산시키거나 즉시 리셋한다. 특히 AC_AttitudeControl의 init() 메서드를 통해 모드 진입이나 착륙 시 객체 생성자의 초기화 로직을 모방하여 변위를 비운다.
  • PX4-Autopilot: uORB 다중 쓰레드 구조상, 개별 제어기는 자신의 스레드 스케줄 내에서 vehicle_control_mode 또는 vehicle_land_detected 토픽을 구독(Subscribe)하다가 이벤트를 포착하면 스스로 수학 연산기(ControlMath 관련 메서드)를 호출해 제로화(Zeroing) 매트릭스를 적용한다. 이는 Ardupilot 대비 각 제어기(자세, 위치)가 독립적으로 자신의 타이밍에 메모리를 해제하는 자율성이 더 강함을 의미한다.

5. QGroundControl (QGC)를 응용한 제어 모니터링 방식 제한

QGC는 제어기 내부의 적분항 그 자체(Raw Float Value)를 GCS 화면에 직접적으로 브로드캐스팅하는 MAVLink 패킷 디스플레이를 기본 UI로 가지고 있지 않다. GCS 운용자는 간접적 확인법을 채택해야 한다.

  1. MAVLink Inspector 활용: QGroundControl의 개발자 도구 (Analyze Tools) 탭에서 MAVLink Inspector를 실행 한 뒤, PID_TUNING 메시지 팩이나 HIGHRES_IMU 패킷의 추세를 모니터링한다. (특정 빌드 스위치에서만 전송될 수 있음)
  2. 그래픽 피드백 부재 시 Flight Review 의존: GCS 관제에서 Anti-Windup 리셋을 실시간으로 보는 것은 어렵기 때문에, 착륙 시 기체가 바닥에서 구르는 등의 거동 이상이 발견되면 GCS의 실시간 기능보다 비행 완료 후 ULog 기반의 Flight Review 툴을 활용하여 estimator_status와 vehicle_rates_setpoint 다이어그램 상의 피크 파형(Peak Waveform)을 분석하는 기술이 요구된다.

6. ROS2 기반 자율 제어 및 오차 피드백 데이터 연동 (uXRCE-DDS)

로봇 운영체제(ROS2) 환경 하에서 에이전트를 개발할 경우, 외부 보드(Companion Computer)에서 PX4의 상태를 모니터링하면서 제어기 리셋을 보장받는 것은 자율화 시스템에 결정적이다.

ROS2 개발자는 외부 MAVLink 프로토콜 또는 uXRCE-DDS (PX4 v1.14 이상 권장) 환경을 통해 기체의 vehicle_status 메시지를 서브스크라이브 한다. 만약 ROS2에서 PositionSetpoint 명령을 비정상적으로 계속 내리고 있는 도중 드론이 Failsafe 로 인해 착륙했을 경우, 드론의 내부 PID의 와인드업은 PX4에서 초기화되었을지라도, ROS2 애플리케이션의 내부 제어 로직(ROS2 기반 컨트롤러)에서는 여전히 누적 오차가 쌓이고 있을 수 있다.

  • 제어 아키텍처 연동 팁: ROS2 자율 비행 노드 내에 자체 구현된 제어기 루프가 존재한다면, PX4 측 토픽인 vehicle_status.nav_state 필드가 NAVIGATION_STATE_AUTO_LAND 단계에서 완료로 넘어가는 즉시 ROS2 내부 제어 객체의 reset_integral() 콜백 함수를 연쇄적으로 호출하여 양방향(Dual-layer) 제어 초기화를 반드시 달성해야 한다.
