50.2.2.2. 현재 실행 중인 비행 모드의 우선순위와 착륙 명령의 인터럽트 우선순위 간의 선점(Preemption) 스케줄링 로직

PX4-Autopilot의 비행 제어 아키텍처는 다수의 비행 모드(Flight Mode)와 페일세이프(Failsafe) 로직이 동시에 시스템 제어권을 확보하기 위해 경쟁하는 다중 상태(Multi-state) 환경이다. 따라서 지상 관제 시스템(GCS)으로부터 착륙(Land) 명령이 유입되었을 때, 이 명령이 현재 실행 중인 비행 모드를 즉각적으로 선점(Preemption)할 수 있는지 판단하는 엄격한 우선순위(Priority) 스케줄링 로직이 Commander 모듈 내부에 구현되어 있다. 본 절에서는 착륙 명령의 인터럽트 우선순위가 다른 비행 모드들과 어떻게 상호작용하는지 심도 있게 다룬다.

1. PX4 비행 모드의 우선순위 계층 구조(Priority Hierarchy)

비행 제어 소프트웨어는 예기치 못한 충돌이나 명령 엇갈림을 방지하기 위해 각 모드에 절대적인 계층적 우선순위를 부여한다. 우선순위는 크게 사용자 편의성 모드(User Assist), 자율 비행 모드(Autonomous), 그리고 생존을 위한 비상 모드(Failsafe/Emergency)로 분류된다.

  • 최상위 (비상/생존 모드): Flight Termination, Kill Switch, Critical Battery Failsafe, Sensor Failure
  • 상위 (명령형 자율 모드): Auto Land, Return To Launch (RTL), Mission (Auto)
  • 중위 (보조형 수동/반자동 모드): Position Control, Altitude Control, Offboard
  • 하위 (수동 모드): Manual, Stabilized, Acro

GCS를 통한 VEHICLE_CMD_NAV_LAND 커맨드는 “상위 (명령형 자율 모드)” 계층에 속하는 Auto Land를 트리거한다.

2. 선점(Preemption) 평가 메커니즘

Commander 스레드가 착륙 명령을 수신하면, 내부적으로 set_nav_state() 함수를 호출하기 전 현재 유지 중인 nav_state의 우선순위와 새로 요청된 Auto Land의 우선순위를 비교하는 선점 평가 메커니즘을 가동한다.

  1. 상향 선점 (Upward Preemption - 허용): 현재 기체가 Position Control(중위) 혹은 Mission(상위-일반) 상태에서 비행 중일 때 착륙(Land) 명령이 들어오면, 이는 조종사나 관제사의 명시적이고 긴급도 높은 개입으로 간주되어 기존 모드를 즉각 선점하고 착륙 유도 제어로 전환된다.
  2. 하향 혹은 동위에 의한 거부 (Downward Preemption - 거부): 만약 기체가 이미 Low Battery Failsafe에 의해 자동 복귀(RTL) 후 강제 하강(Land) 중이거나, 통신 두절(Data Link Loss) 페일세이프 상태라면, GCS의 일반적인 착륙 명령은 시스템 생존 모드보다 설계상 우선순위가 낮으므로 수용되지 않고(명시적 MAV_RESULT_TEMPORARILY_REJECTED 또는 DENIED 응답) 기존 비상 시퀀스를 계속 유지한다.

4. Ardupilot 대비 아키텍처적 차이점

  • 모드 전환의 유연성과 위계: Ardupilot의 비행 제어기(예: Copter)는 사용자 명령이나 GCS 커맨드에 의해 set_mode()가 호출될 때 대부분의 경우 사용자의 의도를 최우선으로 존중하여 강제로 모드를 덮어쓰는(Overwrite) 경향이 강하다. 반면, PX4-Autopilot은 vehicle_status의 내부 상태 머신이 다중 트리 기반(Hierarchical State Machine)으로 정교하게 짜여 있어, 시스템 스스로 판단한 비상 우선순위(Failsafe Priority)가 사용자 개입(Manual Command)보다 우위에 있는 경우가 더 많다. 이는 자율 주행의 시스템적 안전성(Autonomy Safety)을 인간의 오판으로부터 보호하려는 PX4의 설계 철학을 반영한다.

5. 요약 및 실무 적용 시 고려사항

착륙 명령이 하위 모드를 선점하여 실행되는 과정은 단순한 switch-case 구문이 아니라, 기체의 안위를 보장하기 위한 철저한 우선순위 스케줄링 결과이다. 관제 응용 프로그램(Custom GCS)을 개발하는 실무자는 착륙 명령 송출 시 무조건 기체가 반응할 것이라 가정(Assumption)해서는 안 된다. 항상 반환되는 COMMAND_ACK 결과 코드를 비동기 콜백으로 대기(Awaiting) 처리하고, 만약 명령이 선점당하지 못하고 거부되었다면 기체가 현재 더 높은 계층의 페일세이프 상태에 빠져 있는지 원격 원격도구(Telemetry) 측면에서 사용자 인터페이스(UI)로 경고해야만 온전한 관제 루프를 달성할 수 있다.
