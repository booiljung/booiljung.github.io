50.2.3.1. 유휴 임무 상태(Idle)에서 착륙 임무 상태(Landing)로의 내부 트리거 동기화

PX4-Autopilot의 Navigator 모듈은 고수준의 궤적 제어와 상태 기계를 관리하며, 실행할 임무가 명확하지 않거나 자동 임무(Auto-Mission)가 모두 완료된 경우 기체를 일시적인 ’유휴 임무 상태(Idle)’로 전환하여 대기시킨다. 이 유휴 상태에서 시스템 안전 보장 논리(Failsafe)가 개입하거나 지상 관제 시스템(GCS)으로부터 명시적인 착륙(Land) 명령이 접수되었을 때, 이를 즉각적이고 매끄럽게 착륙 임무 상태(Landing)로 천이시키기 위해 내부 트리거(Internal Trigger) 동기화 메커니즘이 동작한다. 본 절에서는 Navigator 스택 내부에서 Idle과 Landing 사이의 상태 천이가 어떻게 동기화되는지 분석한다(PX4 펌웨어 v1.14 기준).

1. 유휴 임무 상태(Idle State)의 시스템적 정의

자율 비행 중인 기체의 Navigator 모듈 내에서 Idle 상태란 ’경로 추종 제어기(Position Controller)가 활성화되어 XY 좌표와 고도(Z축)를 자체적으로 유지(Position Hold / Loiter)하고 있으나, 목표 지점(Target Setpoint) 갱신은 중단된 상태’를 의미한다.

  • 진입 조건:
    사전 프로그래밍된 웨이포인트(Waypoint) 배열이 모두 소진되었거나, 오프보드(Offboard) 노드와의 연결 단절 직후 체공을 유지하는 경우 진입한다.
  • 동역학적 특성:
    position_setpoint_triplet_s uORB 토픽의 목표가 자신의 현재 위치로 고정되어 기체가 바람이나 외란에 대항하며 공간상에 정지 상태를 유지한다.

2. 상태 전환 트리거(Trigger) 및 uORB 동기화 아키텍처

Idle 상태에 머무르던 중, 배터리 임계치 도달(Low Battery Failsafe) 혹은 MAVLink COMMAND_LONG (MAV_CMD_NAV_LAND) 수신을 통해 상위 제어 모듈인 Commander가 모드 변경을 승인하면 트리거 동기화 로직이 개시된다.

2.1 이벤트 폴링(Polling)과 서브클래스 활성화

Navigator 메인 루프 스레드는 고정된 주기(Tick)마다 vehicle_command와 vehicle_status uORB 토픽의 업데이트 여부(Poll)를 확인한다.
모드가 AUTO_LAND 등 특정 유도(Guidance) 모드로 갱신되었음이 포착되면(Edge Trigger), Navigator 내부의 글로벌 상태 머신인 on_idle() 트리거 핸들러가 이를 감지하여 유휴 상태 루틴을 탈출한다.

2.2 핫스왑(Hot-swap) 방식의 제어권 이양

Navigator는 내부적으로 다수의 상속된 Task 클래스(예: Mission, RTL, Land 등)를 포인터로 관리한다. 트리거 조건이 충족되는 즉시, 기존의 활성 태스크 포인터를 NULL(또는 Loiter)에서 Land 인스턴스로 동기화하여 교체(Hot-swap)한다.
활성화된 Land 인스턴스는 즉시 자신의 on_activation() 메서드를 구동하고 기체의 현재 공간 좌표를 초기 조건(Initial Condition)으로 삼아 새로운 셋포인트 생성을 시작한다.

3. 내부 트리거 동기화 시퀀스 타이밍 다이어그램

4. ArduPilot과의 아키텍처 비교 및 분석

  • 명시적 분기 구조(Explicit Branching):
    ArduPilot(Copter)의 경우, 임무 제어 코어(AP_Mission) 내에서 웨이포인트 배열 처리가 끝나는 지점에 암시적(Implicit)으로 착륙 명령을 심거나 RTL 로직으로 분기하는 구조적 종속성이 존재한다. 즉 자동 비행 흐름 내에서 미션 종료가 상태 전환과 논리적으로 강하게 연결되어 있다.
  • 분산형 트리거와 펍섭 동기화(Distributed Trigger via Pub/Sub):
    반면 PX4-Autopilot은 Navigator 모듈이 vehicle_status 토픽 하나만을 모니터링하여 자신을 동기화(Synchronize)시키는 분산형 관점에 기반한다. 임무가 완료되면 단순히 시스템을 Idle 상태의 위치 홀딩으로 두고, 외부 트리거나 Failsafe 시스템(Commander 측)이 모드를 강제 변경해 주기를 기다린다. 이는 다단계로 복잡해질 수 있는 비상 처리 로직(예: 비행선박 위로의 동적 착륙, ROS2 궤적 보정 등)을 도입할 때 매우 강건하고 확장성 높은 구조적 이점을 제공한다.
