50.2.1.2 수신된 MAVLink 스트림의 uORB vehicle_command 구조체로의 데이터 복사 및 원자적 연산(Atomic Operation) 기반 발행(Publishing)

PX4-Autopilot 아키텍처에서 외부 지상 관제 시스템(GCS)으로부터 전달되는 제어 명령은 mavlink_receiver 모듈을 거쳐 내부 실시간 퍼블리시-서브스크라이브(Publish-Subscribe) 미들웨어인 uORB(Micro Object Request Broker) 네트워크로 유입된다. 이 장에서는 수신된 외부 MAVLink 데이터 버퍼가 직렬화 해제(Deserialization)를 거쳐 내부 C++ 구조체인 vehicle_command_s로 변환된 후, 다중 스레드 환경에서 데이터 무결성을 보장하기 위해 원자적 연산(Atomic Operation)을 기반으로 시스템 버스에 발행(Publishing)되는 전 과정을 기술한다. 작성된 내용은 최근의 PX4 v1.14 아키텍처를 바탕으로 서술한다.

1. MAVLink 명령 메시지의 역직렬화 메커니즘

mavlink_receiver 백그라운드 스레드는 DMA 인터럽트에 의해 깨어난 후 직렬 통신 버퍼에서 데이터를 읽어와 MAVLink C 라이브러리의 파싱 함수(mavlink_parse_char)를 연속적으로 호출한다. 올바른 체크섬(Checksum)과 함께 단일 프레임이 완성되면 해당 메시지의 ID를 식별하기 위한 switch-case 구문으로 진입한다.

예를 들어 GCS에서 이륙, 착륙, 혹은 특정 제어 모드 진입을 지시하는 COMMAND_LONG (메시지 ID: 76) 혹은 COMMAND_INT (메시지 ID: 75) 메시지가 감지되면, 시스템은 이를 인지하고 handle_message_command_long() 혹은 handle_message_command_int()와 같은 전담 처리 핸들러(Handler) 함수를 호출한다. 이 과정에서 유선 상의 리틀 엔디안(Little Endian) 페이로드가 내부 메모리 영역으로 복사되어 의미 있는 변수(Param 1~7, Command ID, Target System/Component)로 디코딩된다.

2. uORB vehicle_command 구조체로의 데이터 매핑

역직렬화가 완료된 MAVLink 커맨드는 PX4 내부에서 전역적으로 통용되는 vehicle_command_s C++ 구조체로 변환되어야 한다.

  1. 대상 식별 검증 (Target Identification Validation): 역직렬화된 메시지의 target_system과 target_component 필드를 확인하여 현재 기체(Vehicle)의 MAVLink ID와 일치하는지, 혹은 브로드캐스트(Broadcast) 명령인지 유효성을 검사한다.
  2. 구조체 초기화 및 매핑 (Struct Initialization & Mapping):
    내부 변수 struct vehicle_command_s vcmd {};를 선언하고, MAVLink의 인자들을 해당하는 param1 ~ param7, command 식별자 필드에 1:1로 복사한다.
  3. 메타데이터 추가 (Metadata Attachment): 보안 스탬프(Timestamp) 및 명령의 출처(Source System/Component) 정보를 구조체 메타데이터로 함께 삽입하여, 차후 명령을 수신하는 Commander 혹은 Navigator 모듈이 해당 명령의 발생 시점과 생존 주기(Time-To-Live, TTL)를 평가할 수 있도록 한다.

3. 다중 스레드 환경 하의 원자적 연산(Atomic Operation) 기반 발행 (Publishing)

비행 제어 소프트웨어는 높은 실시간성을 요구하므로, mavlink_receiver 스레드에서 생성된 vehicle_command 구조체 데이터가 uORB 토픽으로 발행되는 과정은 반드시 원자적(Atomic)이거나 안전한 락 프리(Lock-free) 혹은 짧은 스핀락(Spin-lock) 구간 내에서 처리되어야 한다.

PX4의 uORB 미들웨어는 데이터 레이스(Data Race)와 컨텍스트 스위치(Context Switch)의 부작용을 원천 차단하기 위해 내부적으로 다음과 같은 매커니즘을 따른다.

  • 원자적 쓰기 (Atomic Write): uORB의 내부 구현 객체(Device Node)에 write() 오퍼레이션을 수행할 때, 여러 비행 태스크(Task)가 메모리를 동시에 오염시키는 것을 방지하기 위해 인터럽트 비활성화(Interrupt Disable) 및 원자적 포인터 교환(Atomic Pointer Swap) 기술을 적용한다.
  • 락 프리 알림 (Lock-Free Notification): 데이터 복사가 완료되면, 조건 변수(Condition Variable)나 세마포어(Semaphore)를 증가시킴으로써 구독(Subscriber) 중인 스레드 대기열을 즉각적으로 깨운다(Wake-up).

4. Ardupilot 대비 아키텍처적 차이점

  • 메시지 라우팅 구조: Ardupilot은 GCS 명령이 유입되면 메인 이벤트 루프의 GCS_MAVLink 클래스에서 직접 처리 루틴을 라우팅하여 각 서브시스템(예: Mode::takeoff_run())의 메서드를 즉각적으로 동기 호출(Synchronous Call)하는 경향성이 짙다. 반면, PX4-Autopilot은 철저하게 uORB라는 메시지 버스를 중간에 둠으로써 MAVLink 수신 인프라와 비행 제어 논리를 독립된 스레드 영역으로 분리(Decoupling)시켜 비동기(Asynchronous) 체계를 구축한다.
  • 확장성 이점: 이러한 PX4의 퍼블리시-서브스크라이브 패턴은 ROS2(예: uXRCE-DDS)와 결합 시 동일한 vehicle_command 구조체 인터페이스만 준수하면 MAVLink 뿐만 아니라 외부 에이전트에 의해서도 제어권을 투명하게 넘길 수 있는 강력한 개방성을 제공한다.

5. 요약 및 실무 적용 시 고려사항

vehicle_command_s 토픽 발행 과정에서 원자적 오퍼레이션이 보장된다는 사실은 개발자가 외부 앱 수백 개를 덧붙여도 코어 비행 제어기(Flight Controller) 스레드의 타이밍 안정성이 유지됨을 의미한다. 사용자 커스텀 MAVLink 명령을 추가하고자 한다면, 기존 handle_message_command_long 흐름 내에 명령 ID를 매핑하고, uORB 토픽 생태계를 거쳐 하위 앱(Custom Application)이 구조체를 구독(Subscribe)하도록 시스템 흐름을 통제하라 보호 구역 밖에서의 무절제한 글로벌 변수 활용은 엄격히 금지된다.
