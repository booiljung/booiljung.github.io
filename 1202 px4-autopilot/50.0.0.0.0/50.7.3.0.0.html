50.7.3. 착륙 유도 단계에서의 글로벌 항법 위성 시스템(GNSS) 위치 측정 솔루션 유실 시의 계층적 비상 감쇠 체계(Emergency Failsafe Degradation)

다중 로터 및 고정익 무인항공기가 자동 착륙(Auto Land) 모드 혹은 임무 단계의 최종 착륙 유도(Landing Guidance)를 수행할 때, 절대적인 위치 기준인 글로벌 항법 위성 시스템(Global Navigation Satellite System, GNSS)은 필수적인 요소다. 그러나 도심지 빌딩 숲(Urban Canyon), 두꺼운 구름, 태양 플레어로 인한 이온층 교란, 혹은 의도적인 전파 방해(Jamming 및 Spoofing) 환경에서는 GNSS 신호가 급격히 유실(Loss of Lock)되거나 품질이 심각하게 저하될 수 있다.

이와 같이 지면과 극히 가까워지는 착륙 단계에서 위치 솔루션을 잃어버리는 현상은 추락이나 구조물과의 충돌로 직결될 수 있는 치명적인(Critical) 결함이다. 따라서 최신 PX4-Autopilot(Firmware v1.14 이상 기준)은 GNSS 신호 유실 시 즉각적으로 시스템을 포기하는 대신, 가용한 다른 센서의 조합을 통해 안전을 확보하는 계층적 비상 감쇠 체계(Hierarchical Emergency Failsafe Degradation)를 운영한다.

1. 계층적 비상 감쇠(Degradation) 체계의 개념과 구조

GNSS 유실 시 PX4는 내부 상태 머신(State Machine)인 커맨더(Commander) 모듈을 통해 사전 설정된 논리에 따라 비행 모드를 점진적으로 하향(Fallback)시킨다. 이는 완전히 수동 제어로 넘어가는 충격을 방지하고, 기체가 제어 불능 상태에 빠지지 않도록 유도하는 목적을 지닌다.

이를 ’비상 감쇠(Degradation)’라 부르는 이유는 전체 센서 융합(Sensor Fusion) 시스템이 제공하는 상태 추정치(State Estimate)의 품질이 한 단계 낮아짐에 따라 제어 모드 또한 위치 제어(Position Control)에서 상대 고도 및 자세 제어(Altitude/Attitude Control)로 단계적으로 낮아지기 때문이다.

1.1 시스템 감쇠 계층(Hierarchy of Degradation)

  1. 정상 상태 (Normal Operation): GNSS(절대 위치) + 기압계/Lidar(고도) + IMU(자세) + 나침반(방위각)을 결합하여 EKF2 모듈이 전 지구적 3차원 위치 융합을 수행한다.
  2. 1단계 감쇠 (GNSS 유실, 옵티컬 플로우/VIO 사용): GNSS가 단절되더라도 시각 기반의 옵티컬 플로우(Optical Flow)나 시각 관성 오도메트리(Visual Inertial Odometry, VIO)가 사용 가능한 경우, 절대 좌표 이동은 불가하나 상대적인 지면 속도를 0으로 억제하여 ‘위치 고정(Position Hold)’ 상태는 유지할 수 있다.
  3. 2단계 감쇠 (수평 위치 측정 불가, 상대 고도만 유지): GNSS 및 시각 센서가 모두 불가능한 상황이다. 기체는 Position 모드에서 Altitude 모드로 자동 강등된다. 제어기는 수직(Z축) 고도 강하율만 통제하며 수평(X-Y축)으로는 바람에 의해 표류(Drift)될 수 있다.
  4. 3단계 감쇠 (완전한 비상 착륙, Emergency Landing): 현재 상태에서 수동 제어로의 전환이 지연되거나 조종기(RC) 및 데이터 링크(Data Link)마저 없는 경우, 즉각적으로 Land 모드로 진입하거나 비상 엔진 정지(Flight Termination)를 단행한다.

2. EKF2 (Extended Kalman Filter) 및 COM_POS_FS 파라미터 기반 설정

PX4의 비상 통제는 주로 매개변수 COM_POS_FS_EPH (Position Failsafe Ephemeris), COM_POS_FS_EPV 등을 통해 GNSS 예상 수평/수직 오차(Expected Horizontal/Vertical Position Error) 허용 한계를 넘어서는지를 모니터링한다.

  • COM_POS_FS_DELAY: GNSS 유실 조건이 만족된 이후, 실제로 비행 모드를 강등(Degradation)시키기 전까지 대기하는 지연 시간(기본값 1초)이다. 일시적인 신호 단절상황에서 불필요한 안전 장치(Failsafe) 발동을 막는다.
  • COM_POS_FS_PROB: 위치 혁신(Position Innovation) 테스트 실패 확률의 임계값.

QGroundControl(v4.3 이상 기준)의 Safety 탭에서 지정할 수 있는 “Position Loss Failsafe” 동작은 다음과 같이 설정할 수 있다.

  • Altitude / Manual 모드로 전환
  • Land(제자리 착륙)

특히 착륙 유도 단계에서는 지면과의 거리가 짧으므로, 빠른 속도로 Land 명령을 실행하거나 고도를 천천히 낮추는 제어가 우선적으로 요구된다.

3. ArduPilot과의 비상 감쇠 시스템 아키텍처 차이

ArduPilot 체계 역시 내부적으로 EKF(Extended Kalman Filter) 3을 사용하여 동일한 문제를 제어한다. 그러나 철학적 차이가 분명하다.

  • PX4-Autopilot: 커맨더(Commander) 모듈이라는 중앙 집중화된 상태 머신 기반 제어 방식을 선호한다. 커맨더가 EKF의 혁신(Innovation) 오류와 상태 플래그(local_position_valid 등)를 확인하면, 선제적으로 미들웨어를 통해 비행 모드 강등을 uORB 토픽(vehicle_status)으로 전역에 발행(Publish)한다. 감등 시점의 예외 처리가 매우 엄격하며 모듈 간 소프트웨어적 결합도가 낮다.
  • ArduPilot: 모드 제어와 EKF가 상대적으로 조금 더 밀접하게 상호작용한다. ArduPilot의 EKF Failsafe 메커니즘은 Copter 비행 루프 자체에서 직접 트리거되며, 이른바 ’EKF Failsafe Action’이 별도의 비행 모드(예: Land 또는 직전의 안전 모드)로 즉각 전환시킨다. 상대적으로 사용자 파라미터(예: FS_EKF_ACTION) 의존도가 크고 경험적(Heuristic) 튜닝 요소가 많이 반영되어 있다.

4. 커맨더(Commander) 구조와 소스 코드 분석

PX4의 소스코드(Firmware v1.14 기준)를 살펴보면 src/modules/commander/Commander.cpp 내부에서 상태 전이에 대한 중추적인 로직을 담당한다.

// GNSS 유실 및 EKF 솔루션 붕괴 감지 시 (Pseudo Code)
if (!status_flags.condition_local_position_valid &&
    vehicle_status.nav_state == vehicle_status_s::NAVIGATION_STATE_AUTO_LAND) {
    
    // 이전에 옵티컬플로우 등 속도를 추정할 수 있는 대안 센서가 유효한지 검사
    if (status_flags.condition_local_velocity_valid) {
        // 속도 추정은 가능하므로, 위치 정보가 없어도 속도 기반 하강 제어는 유지
        main_state_transition(vehicle_status, commander_state_s::MAIN_STATE_ALTCTRL);
    } else {
        // 완전한 위치 및 속도 솔루션 유실, 즉각적인 오픈 루프 착륙(Open-loop Land) 또는 비상 강하
        failsafe_transition(failsafe_reason::POSITION_UNRELIABLE);
    }
}

GNSS 신호 불량으로 인한 uORB 메시지 sensor_gps의 위성 수효(satellites_used) 및 HDOP(Horizontal Dilution of Precision) 증가가 발생하면, ekf2 모듈 내부에서 한계점 초과를 인식하고 위치 유효 플래그(position_valid)를 false로 떨어뜨린다. Commander는 이 플래그를 정기적으로 폴링(Polling)하다가 안전 장치(Failsafe) 시퀀스로 전환하게 된다.

5. 비상 감쇠 상태 천이 다이어그램 (State Transition Diagram)

다음은 착륙 도중 GNSS 유실이 감지되었을 때 시스템이 어떠한 판단 경로를 거치는지를 시각화한 상태 머신 다이어그램이다.

stateDiagram-v2
    [*] --> AutoLand : 착륙 시동 진입 (GNSS 정상)
    
    AutoLand --> EKF_Error_Check : 위치 오차(Innovation) 증가 검출
    EKF_Error_Check --> AutoLand : 오차 허용치 이내 (자체 복구)
    
    EKF_Error_Check --> ComPosFs_Trigger : COM_POS_FS_DELAY 초과
    ComPosFs_Trigger --> OptFlow_Check : 대안 센서 확인
    
    OptFlow_Check --> AltitudeHold_Land : VIO/옵티컬플로우 부재 (수직 고도만 유지하며 유지강하)
    OptFlow_Check --> VelocityHold_Land : VIO/옵티컬플로우 정상 (상대 속도를 통제하며 유지강하)
    
    AltitudeHold_Land --> OpenLoop_Descend : 기압계/Lidar마저 오작동 발생 시
    VelocityHold_Land --> Landed : 안전하게 도달
    
    OpenLoop_Descend --> Emergency_Terminate : 임계치를 초과하는 빠른 하강률 감지
    OpenLoop_Descend --> Landed : 지면 강제 접촉 (Crash/Land)
    Landed --> [*]

위의 단계 중 OpenLoop_Descend 상태는 최후의 수단으로 모터 출력을 호버링 출력 스로틀(Hover Throttle) 대비 일정한 비율로 서서히 낮추는 제어를 의미한다. 이 구간에서는 바람에 의한 횡방향 이탈을 시스템이 전혀 보정할 수 없게 된다.

6. GCS 및 MAVLink 기반 위치 유실 통보 체계

QGroundControl(GCS)과 PX4-Autopilot 기체 사이의 관제 통신은 MAVLink 프로토콜상 STATUSTEXT 구조체를 통하여 텍스트 포맷 형태의 치명적 경고를 발하게 된다. QGC 4.3 이상에서는 빨간색 배너 팝업을 통해 사용자 혹은 관제사에게 긴급하게 다음과 같은 문자열을 출력하여 상황을 전파한다.

  • “Failsafe: Position estimate lost”
  • “Switching to Altitude mode”

동시에 기체가 주기적으로 전송하는 HEARTBEAT 메시지의 custom_mode 필드가 변경되어, GCS UI에 나타나는 기체의 활성 모드는 AUTO.LAND에서 ALTITUDE (또는 지정된 비상 모드)로 즉각 재표출된다.

이 때 지상 관제망에서는 MAVSDK 기반의 범용 관제 애플리케이션 또는 자율 에이전트(Autonomous Agent)가 해당 MAVLink 이벤트를 구조적으로 구독(Subscribe)하여 실시간으로 개입할 수 있다. 특히 최근의 ROS2 기반 통합 환경의 경우, VehicleStatus 토픽의 변화를 감지하고 uXRCE-DDS 브리지를 활용하여 신속하게 비상 프로토콜 스크립트(예: 가장 가까운 낙하산 전개 구역으로 유도, 비상 조향 명령 전파 등)를 추가로 하달할 수 있도록 관제 기술이 설계되어야 한다.

결론적으로 PX4-Autopilot의 자동 착륙 중 GNSS 유실 대응 모델은 단순히 이전 임무를 취소하는 단일 단계(Single-step) 회피에 머무르지 않는다. 센서 데이터의 유효성을 세분화하여 가용한 한도 내 최대한의 부분 제어 권한을 유지하는 ‘계층적 감쇠’ 메커니즘을 설계함으로써, 기체의 파손 및 대형 사고를 최소화하는 정교한 공학적 기법을 제시하고 있다.
