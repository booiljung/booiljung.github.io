50.6.2. QGroundControl 내 Vehicle C++ 클래스 객체의 속성 업데이트 및 상태 이벤트 기반(Event-driven) 시그널 슬롯(Signal-Slot) 연결 매커니즘

비행 중인 PX4 무인기가 지면에 완전하게 착륙(Landed)했음을 지상 관제 시스템(GCS)이 처리하는 과정은, 단순한 데이터 값을 화면상에 출력하는 수준을 넘어선다. GCS 애플리케이션 프레임워크인 QGroundControl(QGC) 내부에서는 MAVLink 메시지의 원시 바이트(Raw Bytes)가 의미 있는 객체 지향 모델(Object-Oriented Model)로 승격되어 관리된다. 이 과정의 중심에는 다중 기체 동시 관제를 가능케 하는 핵심 추상화 계층인 Vehicle C++ 클래스가 존재한다.

본 절에서는 QGroundControl 아키텍처 관점에서, 기체의 이착륙 및 비행 모드 상태가 MAVLink 수신 패킷에서 추출된 후 Vehicle 클래스 인스턴스의 속성(Property)에 저장되고, 이로 인해 연쇄적인 상태 이벤트(Event-driven)가 시그널 슬롯(Signal-Slot) 구조를 타고 QML 프론트엔드 UI 요소에 도달하는 매커니즘을 상세히 다룬다.

1. Vehicle C++ 클래스의 역할 및 선언적 속성(Property) 구조

QGroundControl(v4.2 기준) 데이터베이스 내에서 단일 드론 기체는 1:1로 매칭되는 Vehicle 클래스(소스 경로: src/Vehicle/Vehicle.h)의 인스턴스로 관리된다. 이 객체는 기체의 시스템 ID 등 고유 정보뿐만 아니라 실시간으로 변동하는 배터리 잔량, 비행 모드(Flight Mode), 그리고 이착륙 상태(Flying / Landed state) 정보를 모두 캡슐화(Encapsulate)한다.

Qt 프레임워크의 강력한 메타 객체 시스템(Meta-Object System)을 사용하는 Vehicle 클래스는 속성 값을 선언적으로 QML과 연동하기 위해 헤더 파일 상단에 다수의 Q_PROPERTY 매크로를 정의하고 있다. 기체의 시동 및 착륙 상태와 관련된 주요 프로퍼티는 아래와 같이 선언된다.

// src/Vehicle/Vehicle.h 일부 발췌 구조
Q_PROPERTY(bool armed READ armed WRITE setArmed NOTIFY armedChanged)
Q_PROPERTY(bool flying READ flying NOTIFY flyingChanged)
Q_PROPERTY(bool landed READ landed NOTIFY landedChanged)

이 매크로는 데이터를 읽는 접근자(Accessor, READ) 및 값이 변동되었을 때 이를 통지(Notify, NOTIFY)하기 위한 시그널 함수를 C++ 컴파일러(보다 정확히는 moc, Meta-Object Compiler)에게 지시한다. 만약 백엔드 코드에서 기체가 착륙했다는 사실을 인지하여 해당 변수 값을 변경한다면, 반드시 약속된 시그널(landedChanged)을 방출(Emit)해야 한다.

2. 상태 이벤트 기반(Event-driven) 업데이트 알고리즘 (Mermaid 도표)

착륙 완료 상태를 통지하는 전체 파이프라인의 이벤트 흐름은 네트워크 인터럽트를 통해 패킷을 수신한 뒤, 순수 데이터가 객체의 속성 변이(State Mutation)로 이어지는 흐름을 갖는다.

    participant MAV as MAVLink Parser
    participant Veh as Vehicle::handleExtendedSysState

    MAV->>Veh: EXTENDED_SYS_STATE 디코딩 완료 이벤트
    Note over Veh: 기존 상태값과 수신된 상태값 비교
    alt 상태 변화 감지됨 (In Air -> On Ground)
        Veh->>Prop: _flying = false, _landed = true 등 값 갱신
        Note over Veh: Qt 시그널 발행 (Emit)
        Veh->>UI: emit flyingChanged(false)
        Veh->>UI: emit landedChanged(true)
    else 상태 변화 없음
        Note over Veh: 속성 갱신 건너뜀 (오버헤드 방지)
    end
    Note over UI: UI 프레임워크가 시그널 큐를 소비하여 <br> 착륙 아이콘 및 버튼 재계산(Re-rendering)

3. 업데이트 최적화와 시그널 슬롯(Signal-Slot) 연결의 중요성

GCS 내부에서 각 기체는 초당 수 십 개의 상태 메시지를 전송한다. 만약 상태 값의 변화 유무를 따지지 않고 무조건적으로 UI를 갱신하게 된다면, 렌더링 스레드(Rendering Thread)에 막대한 부하가 걸려 소프트웨어 병목 현상(Bottleneck)이 초래된다. 이를 방어하기 위해 상태 이벤트 기반 아키텍처는 다음과 같은 특성을 보장한다.

3.1 변위 검출을 통한 시그널 최소화 기법

Vehicle 객체의 메서드는 파싱된 패킷 인자의 값(mavlink_msg_extended_sys_state_t 내의 landed_state)이 이전에 저장된 변수 스냅샷 값(_landed_state)과 일치하는지를 내부적으로 비교 로직(if (_landed_state != new_state))을 거친다. 오직 값이 다를 때만 내부 변수를 갱신하고 emit 키워드를 호출한다. 이 조건 분기(Condition Branching)는 불필요한 이벤트 트래픽을 원천 차단하는 핵심 방어선이다.

3.2 독립 스레드 간 비동기 연결(Queued Connection)

QGC 설계상 MAVLink 패킷은 메인 스레드와 분리된 하위 통신 스레드에서 해석될 수 있다. Qt 프레임워크의 시그널 슬롯 메커니즘은 서로 다른 스레드에 상주하는 객체 간에 통신이 일어날 때, 자동으로 큐 연동(Queued Connection) 방식을 차용한다.
따라서 Vehicle 클래스의 백엔드 스레드에서 시그널(emit landedChanged())을 던지면, Qt의 메인 이벤트 루프(Event Loop)가 이를 안전하게 큐잉(Queueing)했다가 UI 스레드가 유휴 상태일 때 QML 바인딩 연산을 비동기적으로(Asynchronously) 수행한다.

4. 프론트엔드(QML)의 반응형 동작 연쇄

Vehicle 클래스로부터 도달하는 시그널은 QML 계층의 프로퍼티 바인딩(Property Binding) 엔진에 의해 즉각 수용된다.

  1. 아이콘 및 알림 표출: activeVehicle.landed가 true로 변화함에 따라, 툴바(Toolbar)의 기체 상태 뱃지 레이블이 “In Air“에서 “Landed” 또는 “Ready” 컴포넌트로 리렌더링된다.
  2. 슬라이더(Slider) 문맥 변경: 사용자가 조작 가능한 제어 위젯 하부 영역에서, 기존에 떠있던 ‘Land(착륙) / RTL(복귀)’ 커맨드 슬라이더가 닫히고, 기체가 바닥에 내려왔으므로 ‘Arm(시동)’ 및 ’Takeoff(이륙)’을 위한 슬라이더 UI 팝업 논리 조건이 다시 활성화된다.

5. 결론

QGroundControl(GCS)에서 무인기의 착륙 완료 시점과 같은 치명적 상태 전이는 절차적(Procedural)으로 폴링(Polling)되어 화면에 그려지지 않는다. PX4가 하달하는 MAVLink 메시지는 QGC 백엔드의 Vehicle C++ 객체 내 추상화 계층을 통과하며 기존 속성값과의 차분 분석(Delta Analysis)을 거치며, 실제 전이가 발생한 경우에 한해 시그널 앤 슬롯 매커니즘이라는 이벤트 버스(Event Bus)를 타고 메인 UI 스레드를 안전하게 일깨운다. 이러한 객체 지향 및 이벤트 구조는 복잡한 GCS 환경에서 코드를 직관적으로 유지함과 동시에, CPU 부하 오버헤드를 획기적으로 감축시키는 가장 교과서적인 분산 상태 동기화 기법이라 할 수 있다.
