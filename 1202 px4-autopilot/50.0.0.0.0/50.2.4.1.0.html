## 0.1 유도 제어기에서 발행된 vehicle_local_position_setpoint 토픽을 위치 제어기가 폴링(Polling) 방식으로 구독(Subscribe)하는 스레드 동기화 기법

유도 제어기(Guidance Controller)와 위치 제어기(Position Controller) 간의 데이터 파이프라인 결합에서 가장 핵심적인 부분은 제어 목표값(Setpoint)의 실시간 전달 및 스레드 동기화(Thread Synchronization)이다. PX4-Autopilot(펌웨어 버전 1.14 이상 기준) 시스템 내에서 유도 제어기 역할을 수행하는 Navigator 또는 비행 작업(Flight Task) 모듈은 계산된 궤적을 기반으로 vehicle_local_position_setpoint 토픽(Topic)을 uORB(Micro Object Request Broker) 브로커를 통해 주기적으로 발행(Publish)한다. 이를 수신하여 실제 액추에이터 출력을 위한 자세 제어 목표를 산출하는 위치 제어기(예: mc_pos_control)는 이벤트 기반의 콜백(Callback) 방식 대신, 능동적인 폴링(Polling) 방식을 통해 해당 토픽을 구독(Subscribe)하여 스레드 간 동기화를 보장한다.

0.1.1 폴링(Polling) 방식 구독의 시스템적 근거

멀티로터의 위치 제어 모듈은 비행 동역학을 안정화하기 위해 엄격한 실시간성(Real-time Constraints)을 요구하며, 보통 최고 250Hz 이상의 고정된 주기로 실행되어야 한다. 만약 콜백(Callback)이나 인터럽트 기반으로 다양한 토픽을 수신할 경우, 각기 다른 주기로 발행되는 토픽들로 인해 제어 루프의 실행 주기(Jitter)가 불안정해질 수 있다. 따라서 위치 제어기는 시스템의 메인 루프 관성을 유지하고 락(Lock) 경합을 최소화하기 위해 논블로킹(Non-blocking) 폴링을 사용하거나, px4_poll() 함수를 통한 타임아웃 기반 블로킹 폴링 기법을 적용한다.

이러한 기법은 실시간 운영체제(RTOS)인 NuttX 환경에서 컨텍스트 스위칭(Context Switching) 오버헤드를 현저히 줄이고, 결정론적(Deterministic) 제어 루프를 보장하기 위해 필수적으로 요구된다.

0.1.2 vehicle_local_position_setpoint 동기화 파이프라인

유도 제어기가 vehicle_local_position_setpoint 토픽을 업데이트할 때, 위치 제어기의 실행 스레드는 동기화를 위해 다음과 같은 절차를 거친다.

  1. uORB 노드 선언(Subscription Node Initialization): 위치 제어기 초기화 단계에서 uORB::SubscriptionInterval 또는 uORB::SubscriptionCallbackWorkItem 기반 구독 객체를 생성하여 폴링 주기나 콜백 워크 큐를 설정한다. 메인 제어 루프 구동 시에는 명시적인 폴링 함수 처리를 통해 동기화를 검증한다.
  2. 이벤트 대기(Event Wait) 및 폴링 타임아웃: 위치 제어기 스레드는 vehicle_local_position (현재 기체 위치) 업데이트에 맞춰 메인 루프를 동기화하고, vehicle_local_position_setpoint (목표 위치) 토픽에 대해 updated() 메서드를 호출하여 새로운 제어 목표의 수신 여부를 검사한다.
  3. 데이터 복사(Data Fetching): 새로운 목표값이 확인되면 copy() 함수를 통해 메모리 공간으로 메시지를 복사한다. 이 과정은 uORB 내부의 원자적(Atomic) 연산에 의해 보호되므로, 발행자와 구독자 스레드 간의 데이터 경합 현상(Race Condition)을 방지한다.

아래는 유도 제어기와 위치 제어기 간의 폴링에 기반한 동기화 처리 흐름을 도식화한 다이어그램이다.

sequenceDiagram
    participant GC as Guidance Controller <br/> (Navigator)
    participant uORB as uORB Middleware
    participant PC as Position Controller <br/> (mc_pos_control)

    Note over GC,uORB: 비동기 발행 (Asynchronous Publish)
    loop Trajectory Generation
        GC->>GC: 궤적 연산 및 <br/> 목표값(Setpoint) 산출
        GC->>uORB: orb_publish(vehicle_local_position_setpoint)
    end

    Note over uORB,PC: 동기 폴링 (Synchronous Polling)
    loop 250Hz Control Loop
        PC->>uORB: poll() 또는 updated() 검사
        alt New Setpoint Available
            uORB-->>PC: true 반환
            PC->>uORB: copy()
            uORB-->>PC: vehicle_local_position_setpoint 구조체 반환
        else No New Data
            uORB-->>PC: false 반환
            PC->>PC: 기존 목표값 유지 (Zero-Order Hold)
        end
        PC->>PC: 추력(Thrust) 및 자세(Attitude) 목표값 연산
        PC->>uORB: orb_publish(vehicle_attitude_setpoint)
    end

0.1.3 C++ 클래스 구현 예시

실제 소스 코드에서 위치 제어기 모듈 내 폴링 아키텍처는 아래와 유사한 패턴으로 작성된다.

#include <uORB/Subscription.hpp>
#include <uORB/topics/vehicle_local_position_setpoint.h>

class PositionControl {
private:
    uORB::Subscription _local_pos_sp_sub{ORB_ID(vehicle_local_position_setpoint)};
    vehicle_local_position_setpoint_s _local_pos_sp{};

public:
    void Run() {
        // 메인 제어 루프 내부
        while (!should_exit()) {
            
            // 1. 현재 토픽에 새로운 데이터가 발행되었는지 폴링 방식으로 확인 (논블로킹)
            if (_local_pos_sp_sub.updated()) {
                // 2. 동기화된 메모리 복사 수행 (Lock-free 보장)
                _local_pos_sp_sub.copy(&_local_pos_sp);
            }

            // 3. 복사된 제어 목표값(_local_pos_sp)을 기반으로 제어 알고리즘 수행
            calculate_position_control(&_local_pos_sp);
            
            // 다음 제어 주기까지 슬립(Sleep) 처리 또는 타임아웃 이벤트 대기
            px4_usleep(4000); // 250Hz 주기 대기
        }
    }
};

위의 예시 코드에서 볼 수 있듯이, updated() 메서드 호출은 메인 제어 스레드의 실행 흐름을 차단하지 않고 상태(Flag)만을 검사하는 폴링 기법의 대표적 사례이다. 이러한 구조적 분리는 QGroundControl(버전 4.3 이상)과 같은 지상 관제 시스템에서 실시간으로 웨이포인트(Waypoint)를 변경하거나 복귀(Return to Launch) 명령을 송신할 때, 유도 제어기로부터 생성된 갑작스러운 궤적의 변화를 위치 제어기가 유연하면서도 지연 없이 분리된 타이밍에(Zero-delay synchronization) 수용할 수 있게 해준다.

결론적으로, 폴링 기반 스레드 동기화 기법은 다중 스레드 환경에서 분산된 제어 모듈 간 데이터 통신 시 발생할 수 있는 락 경합과 데드락(Deadlock)을 원천적으로 차단하며, 가장 신뢰성 높은 자율 비행 제어 연산을 실시간으로 수행하게 하는 안정적인 백본망 역할을 한다.
