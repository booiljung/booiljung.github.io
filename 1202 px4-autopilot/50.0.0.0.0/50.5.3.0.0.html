50.5.3. 하위 수준(Low-level) 자세 및 위치 피드백 제어기의 내부 연산 변수 초기화(Reset) 및 포화 해제(Desaturation)

무인항공기가 공중에 체공하다 지면에 접지하여 착륙(Landed) 상태가 확정되고 시동 해제(Disarming)가 명령되면, 시스템 내부의 제어 루프는 더 이상 물리적인 추력을 발생시키지 않는다. 하지만 프로그래밍 관점에서 제어 알고리즘의 루프는 여전히 활성화되어 있거나 메모리 상에 이전의 제어 누적 오차 변수를 보유하고 있을 수 있다. 본 절에서는 PX4-Autopilot 환경에서 착륙 직후 하위 수준(Low-level) 제어기(자세 제어기, 위치 제어기 등)의 내부 변수를 안전하게 초기화(Reset)하고 포화(Saturation)된 상태를 해제(Desaturation)하는 소프트웨어적 구조 및 제어 공학적 의미를 다룬다.

1. 제어기 내부 변수 초기화 및 포화 해제의 필요성 (제어 공학적 관점)

기체가 착륙한 시나리오에서는 센서(IMU 등)가 지면의 미세한 진동이나 외부 요인(바람)을 측정할 수 있다. 이때 제어 루프가 여전히 활성화되어 있으면, 위치 편차 또는 각속도 편차를 줄이기 위해 제어기 내부의 비례-적분-미분(PID) 모듈은 지속적으로 오차를 누적하게 된다.

만약 제어기 초기화가 이루어지지 않는다면 발생할 수 있는 문제 결과:

  1. 초기 발산 (Initial Divergence): 누적된 적분 에러(Integral Error)로 인해 다음번 이륙(Takeoff) 명령이 내려졌을 때 엑추에이터(Actuator)로 비정상적으로 큰 신호가 급격히 전달되어 기체가 전복되거나 튀어 오를 수 있다(Flip).
  2. 적분 와인드업 (Integral Windup): 착륙 상태에서 요구 위치(Setpoint)와 실제 위치(State) 간의 미세한 오차가 무한정 누적되어 적분항이 시스템 한계점까지 가득 차는 포화(Saturation) 현상이 유지된다.

따라서 PX4-Autopilot은 착륙 상태로 전환되거나 제어 모드가 활성화/비활성화될 때, 제어기 내부의 메모리 블록을 강제로 영점(Zero)으로 돌리고, 와인드업 방지 로직을 초기화하는 절차를 수행한다.

2. PX4 소스 코드 아키텍처에서의 제어기 초기화 (Source Code Analysis)

PX4-Autopilot은 멀티콥터의 경우 주로 mc_pos_control (멀티콥터 위치 제어)과 mc_att_control (멀티콥터 자세 제어) 모듈을 사용하여 계층형 제어(Cascaded Control)를 수행한다. 이 모듈들은 기체의 시동 무장(Arming) 상태나 비행 모드의 변동 상태를 uORB 메시지인 vehicle_control_mode 토픽을 통해 감지한다.

소스 코드 관점에서, 상태가 활성화(Active)에서 비활성화(Inactive)로 전환되거나 그 반대의 경우, 특정 reset() 메서드가 명시적으로 호출된다.

/* PX4 멀티로터 위치 제어 루틴 초기화 예제 기저 논리 
   src/modules/mc_pos_control/PositionControl/PositionControl.cpp */
void PositionControl::reset()
{
    // 위치 예측 변수 및 상태 초기화
    _pos_int.setZero();     // 위치 적분 제어기(Integral Controller) 값 영점(0) 강제 할당
    _vel_int.setZero();     // 속도 적분 제어기 초기화
    
    // PID 포화(Saturation) 해제 및 와인드업 가드 타이머 리셋
    _vel_sp.setZero();
    _acc_sp.setZero();
    
    // 제어기가 다시 시작할 때 부드러운 시작(Soft Start)을 위한 제어 플래그 오버라이드
    _lim_z_thrust_int = false;
}

위 로직은 vehicle_control_mode에 명시된 시동 상태(flag_armed)가 false (즉, 시동이 해제됨)로 트랜지션할 때 위치 제어 스택 및 자세 제어 스택에서 동시다발적으로 트리거되어 제어 포화 상태를 해제하게 된다.

3. 제어기 상태 변경 모니터링 시퀀스 (Mermaid Diagram)

아래 다이어그램은 착륙 이후 커맨더 모듈이 생성하는 상태 전이를 제어기에서 수신하고 변수를 해제하는 시스템 틱(Tick) 단위의 순서를 묘사한다.

sequenceDiagram
    participant LD as Land Detector
    participant C as Commander (State Machine)
    participant uORB as uORB Middleware
    participant MC_Pos as Position Controller
    participant MC_Att as Attitude Controller

    LD->>uORB: vehicle_land_detected (landed = true)
    uORB->>C: 토픽 감지 및 시동 해제 타이머 초과 확인
    C->>uORB: vehicle_control_mode 발행 (flag_armed = false)
    
    uORB->>MC_Pos: 토픽 업데이트 수신
    activate MC_Pos
    MC_Pos->>MC_Pos: reset() 함수 호출<br>(위치/속도 적분항 초기화)
    deactivate MC_Pos
    
    uORB->>MC_Att: 토픽 업데이트 수신
    activate MC_Att
    MC_Att->>MC_Att: 각속도/자세 오차 메모리 블록 영점화<br>포화 경계선 리셋
    deactivate MC_Att

이와 같은 다중 모듈 간 uORB 동기식 통신 구조 덕분에, 각 제어기는 독립적으로 동작하면서도 단일 장애점(Single Point of Failure)을 방지하며 하위 로직을 안전하게 소멸시킬 수 있다.

4. Ardupilot 대비 제어기 아키텍처 차이

Ardupilot은 AC_AttitudeControl 및 AC_PosControl 라이브러리를 사용하여 하위 수준 제어를 수행한다.
Ardupilot의 경우 Copter::update_flight_mode()를 필두로 메인 비행 루프(Main Loop) 내에서 단일 스레드로 제어 계층의 리셋 함수(relax_attitude_controllers() 또는 init() 등)를 모놀리식(Monolithic)하게 호출하는 구조를 취한다. 반면, PX4는 마이크로 서비스 아키텍처(uORB 기반 퍼블리셔-서브스크라이버 구조)를 채택하고 있으므로, 커맨더(Commander) 모듈이 명시적으로 reset 함수를 지시하는 것이 아니라 통제 모드의 변경(Topic)을 선언하면 각 모듈이 이를 자율적으로 해석하여 reset()을 실행한다는 핵심적인 아키텍처 로직 차이를 지닌다.

5. QGroundControl (QGC)를 통한 관제 현황 및 로그 분석 연동

QGC v4.3 및 동급 버전에서는 착륙 후 기체의 하위 수준 제어기가 정상적으로 리셋되었는지 명시적으로 확인할 수 있는 전용 그래픽 UI(User Interface) 버튼을 제공하지는 않는다. 그러나 GCS 관제사 혹은 오퍼레이터 차원에서는 다음의 방법을 통해 제어 포화가 해제되었는지 간접적으로 인지한다.

  • MAVLink ACTUATOR_CONTROL_TARGET 모니터링: QGC의 MAVLink Inspector를 통해 ACTUATOR_CONTROL_TARGET 패킷 내부의 제어 할당 지시값이 시동 해제 후 모두 0 또는 중립값으로 스윕(Sweep)되는 과정을 관찰할 수 있다.
  • 비행 기록(ULog): 착륙 이후의 로그가 정지될 때까지 남겨진 vehicle_local_position_setpoint 및 estimator_states 토픽의 변화량을 살펴보면 위치/속도 요구치(Setpoint)가 강제로 0으로 리셋되는 구간을 확인할 수 있다. 이는 추후 Flight Review 툴을 활용한 기체 진단 시 와인드업 현상 유무를 판별할 중요한 지표가 된다.

6. ROS2 연동 기반 (uXRCE-DDS) 고급 관제 에이전트 개발 기술

외부 자율 에이전트(ROS 2 / Companion Computer) 환경에서 Offboard 제어를 사용하다가 착륙 후 로직이 종료될 때에도 리셋 메커니즘을 동일하게 적용하는 것이 중요하다.
Offboard 제어기 노드(ROS 2 Node)는 제어 명령 발행을 중지함과 동시에, 이륙 후 새로운 궤적(Trajectory)을 기체로 전송하기 전에 ROS 2 상의 TrajectorySetpoint 메시지를 초기화된 오프셋 평면에 할당하여 발행(Publish)하여야 초기 발산과 포화 누적(Desaturation Error)을 원시적으로 방지할 수 있다.

// ROS 2 자율 비행 노드 내 초기화 로직 구현 구조 예시
if (current_vehicle_status.arming_state == ARMING_STATE_DISARMED) {
    // 궤적 생성기의 내부 메모리 스테이트 리셋 지시
    trajectory_generator_->reset();
    
    px4_msgs::msg::TrajectorySetpoint msg{};
    msg.position = {0.0f, 0.0f, 0.0f}; // 오프셋 초기화
    msg.velocity = {0.0f, 0.0f, 0.0f}; // 속도 벡터 초기화
    trajectory_setpoint_publisher_->publish(msg);
}

이와 같이 자율 주행 로봇 제어(Agent Control)에서 위치(P) 및 속도(V) 제어 루프를 하드웨어(PX4)와 소프트웨어(ROS 2) 양방향으로 동기화하여 초기화하는 절차는 비행 안전 확보의 핵심 토대가 된다.
