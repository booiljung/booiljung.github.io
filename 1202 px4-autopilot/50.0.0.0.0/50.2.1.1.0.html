50.2.1.1. 직접 메모리 접근(DMA) 방식 수신 버퍼의 인터럽트 서비스 루틴(ISR) 호출 구조

PX4-Autopilot은 고속 통신 환경에서 중앙 처리 장치(CPU)의 부하를 최소화하고, 안정적인 MAVLink 메시지 수신을 보장하기 위해 직접 메모리 접근(Direct Memory Access, DMA) 방식을 적극적으로 활용한다. 이 장에서는 MAVLink 메시지가 물리적 계층(Physical Layer)을 거쳐 수신 버퍼(Reception Buffer)에 적재되고, 인터럽트 서비스 루틴(Interrupt Service Routine, ISR)이 호출되는 일련의 구조를 심도 있게 분석한다. 본 절의 기술적 내용은 PX4 v1.14 및 하부의 NuttX RTOS 기반 하드웨어 추상화 계층(Hardware Abstraction Layer, HAL)을 기준으로 한다.

1. MAVLink 데이터 수신과 DMA(Direct Memory Access)의 필요성

무인 항공기(UAS) 제어 시스템에서 지상 관제 시스템(GCS)과의 통신은 통상적으로 UART(Universal Asynchronous Receiver-Transmitter) 채널을 통해 이루어진다. 수많은 텔레메트리(Telemetry) 데이터와 제어 명령이 빈번하게 교환되는 상황에서 수신되는 바이트(Byte)마다 인터럽트를 발생시키는 고전적인 방식은 시스템의 과도한 인터럽트 오버헤드(Interrupt Overhead)를 초래한다.

이러한 문제를 해결하기 위해 PX4의 시리얼 드라이버는 NuttX 운영체제 하에서 DMA 파이프라인(DMA Pipeline)을 구성한다. DMA 컨트롤러는 주변기기(UART RX 레지스터)에서 수신된 데이터를 CPU의 개입 없이 메모리 영역(Ring Buffer)으로 직접 전송(Transfer)하며, 전송이 완료되거나 버퍼의 절반이 찼을 때(Half-transfer), 혹은 회선 활성 상태가 종료(Idle 라인 감지)되었을 때만 CPU에 인터럽트를 발생시켜 ISR을 호출한다.

2. DMA 파이프라인 및 인터럽트 서비스 루틴(ISR) 동작 구조

DMA 컨트롤러가 데이터를 메모리에 적재하는 과정과 인터럽트가 발생하는 절차는 다음과 같이 상세화할 수 있다.

  1. 하드웨어 초기화 (Hardware Initialization): 비행 제어기(Flight Controller, FC) 부팅 시, 특정 UART 포트(예: TELEM 1, ttyS1)에 할당된 RX 채널에 대해 DMA 스트림이 활성화(Enable)된다.
  2. 환형 버퍼 적재 (Ring Buffer Loading): MAVLink 스트림이 UART RX 핀을 통해 들어오면, 하드웨어 UART 페리페럴(Peripheral)은 이를 데이터 레지스터에 기록하고, DMA는 이를 사전에 지정된 환형 버퍼(Ring Buffer) 주소로 이동시킨다.
  3. 인터럽트 트리거 (Interrupt Trigger):
  • 버퍼 절반 전송 (Half-Transfer, HT): 버퍼의 절반이 데이터로 채워지는 시점에 ISR을 호출하여, 나머지 반환 영역이 채워지는 동안 CPU가 앞서 수신된 데이터를 처리하게 한다.
  • 버퍼 완료 전송 (Transfer Complete, TC): 전체 버퍼 도달 시 인터럽트를 발생시켜 랩어라운드(Wraparound) 처리를 지시한다.
  • 유휴 상태 감지 (Idle Line Detection): 전송 중 데이터 스트림이 일시적으로 중단되었을 때(즉, 라인이 유휴 상태로 진입하면), 대기 시간(Latency)을 줄이기 위해 즉각적으로 인터럽트를 트리거하여 짧은 MAVLink 패킷이라도 지연 없이 상위 계층으로 전달한다.

3. NuttX 시리얼 드라이버와 인터럽트 서비스 루틴(ISR) 분석

PX4 하부의 NuttX RTOS 소스 코드(boards/px4/ 또는 nuttx/arch/arm/ 경로)를 살펴보면 시리얼 DMA의 ISR 핸들러는 주로 up_dma_rxinterrupt와 같은 구조체 및 함수로 매핑된다. 해당 함수의 호출 흐름은 아래의 기술적 메커니즘을 따른다.

  1. 상태 레지스터 검증 (Status Register Verification): ISR 진입 즉시, 인터럽트가 발생한 원인 레지스터(Status Register)를 읽어 HT, TC, 혹은 Idle 상태인지를 판단한다.
  2. 포인터 갱신 (Pointer Update): DMA 컨트롤러 내부의 잔여 데이터 개수 레지스터(NDTR, Number of Data Register)를 읽어 현재 링 버퍼에서 데이터가 쓰인 정확한 위치(Head Pointer)를 계산하고 갱신한다.
  3. 컨텍스트 스위치 알림 (Context Switch Notification): 갱신된 버퍼에 읽을 수 있는 유효 데이터가 존재함을 알리기 위해 sem_post() 등의 동기화 객체를 통해 차단(Blocked)되어 있던 상위 poll() 함수나 파일 읽기 대기열(Read Queue)을 깨운다(Wake up). 이를 통해 mavlink_receiver 백그라운드 스레드가 활성화된다.

4. Ardupilot 대비 아키텍처적 차이점

Ardupilot 역시 DMA 기반 직렬 통신 구조를 제공하여 CPU 효율을 배가한다. 하지만 구현된 운영체제와 하드웨어 추상화 계층(HAL) 철학에서 차이를 보인다.

  • 운영체제 관점: PX4는 POSIX 심리스(Seamless) 환경을 제공하는 NuttX RTOS 구조를 기반으로 하여 표준 파일 디스크립터(File Descriptor) 및 poll()/select() API를 통해 ISR에서 사용자 공간(User Space) 쓰레드로 데이터를 자연스럽게 전달한다.
  • Ardupilot의 ChibiOS: Ardupilot은 내부적으로 ChibiOS를 사용하며 해당 HAL에서 정의하는 시리얼 드라이버 API(예: hal.console, hal.uartA)를 통해 직렬화된 이벤트 루프 기반으로 통신 처리를 이어나간다.

5. 요약 및 실무 적용 시 고려사항

PX4-Autopilot 환경에서 DMA를 활용한 MAVLink 수신 버퍼의 인터럽트 처리 메커니즘은 매우 결정론적(Deterministic)이고 효율적인 구조를 띄고 있다. 새로운 텔레메트리 보드 호환성을 확보하거나 커스텀 하드웨어를 설계(Board Bring-up)하는 개발자는 반드시 해당 포트(Port) 마이크로컨트롤러(MCU) 핀 맵에 DMA 하드웨어 스트림이 할당되어 있는지 점검하라. 만약 DMA 채널 충돌이나 설정 누락이 발생하면, Fallback으로 일반 글자 단위 인터럽트(Byte-level Interrupt)가 활성화되어 높은 트래픽 환경하에서 CPU 점유율이 비정상적으로 치솟고 결국 비행 제어 루프의 타이밍을 위협할 수 있음을 명심하라.
