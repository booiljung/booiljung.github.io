50.2.3. Navigator 모듈의 하강 유도(Guidance) 시스템 활성화 및 비행 경로 스택(Path Stack) 재구성

PX4-Autopilot의 아키텍처에서 Commander 모듈이 착륙(Land) 명령에 대한 유효성 검증을 마치고 중앙 비행 모드를 착륙 모드로 전환시키면, 실제 비행 궤적의 생성 및 유도(Guidance) 책임은 Navigator 모듈로 이관된다. Navigator 모듈은 고수준의 자율 비행(Mission, RTL, Land 등) 동작을 추상화하여, 하위 위치 제어기(Flight Task 또는 Position Controller)가 추종할 구체적인 위치 셋포인트(Position Setpoint Triplet)를 생성하는 핵심 구성 요소이다. 본 절에서는 착륙 모드 진입 시 Navigator 모듈이 기존 비행 경로 스택(Path Stack)을 어떻게 재구성하고, 안전한 지면 접촉을 위해 하강 유도 시스템을 어떠한 메커니즘으로 활성화하는지 상세히 분석한다(논의 기준: PX4 펌웨어 v1.14 아키텍처).

1. Navigator 모듈의 착륙 모드 감지 메커니즘

Navigator 노드는 백그라운드에서 실시간 미들웨어 버스인 uORB를 통해 vehicle_status 토픽을 지속적으로 구독(Subscribe)한다. Commander에 의해 해당 토픽 내부의 nav_state 열거형 값이 NAVIGATION_STATE_AUTO_LAND 등으로 변경됨이 수신되면, Navigator 내부의 제어 흐름 분기가 동작하여 현재 작동 중이던 유도 클래스스(예: Mission, Loiter 등)의 실행을 중단시킨다.

  • 유도 인스턴스 스위칭(Guidance Instance Switching):
    Navigator 모듈은 내부적으로 여러 가지 내비게이션 태스크(Task) 클래스를 보유하고 있다. 상태 갱신 이벤트 파서가 동작하면, 활성 상태의 태스크 포인터를 Mission 클래스에서 Land 태스크 전담 클래스로 변경(스위칭)하여 착륙(Land) 전용 유도 루틴을 활성화한다.

2. 비행 경로 스택(Path Stack)의 초기화 및 재구성 프로세스

기존에 수행 중이던 자동 임무나 귀환 경로점(Waypoint)들은 착륙이라는 긴급하고 최종적인 상태에 진입함으로써 의미를 잃게 된다. 따라서 Navigator는 기체가 하강에 전념할 수 있도록 내부 비행 경로 스택을 전면 개편한다.

2.1 기존 궤적 스택 파기 (Stack Flushing)

메모리 및 유도 로직 상에 적재되어 있던 이전 단계의 목표 셋포인트 큐(Queue)는 모두 소거(Flush) 처리된다. 이를 통해 낡은 위치 정보가 위치 제어루프로 유입되어 예상치 못한 수평 전진 거동을 유발하는 현상을 방지한다.

2.2 하강 지향 셋포인트 트리플릿(Position Setpoint Triplet) 생성

경로 스택이 비워지면, Navigator는 Land 태스크 클래스를 통해 현재 기체의 로컬 좌표(XY 반경)를 홀드(Hold) 축으로 고정한다. 그 다음, 오직 Z축(고도 하강 방향)만을 타겟으로 삼는 새로운 데이터 패키지인 position_setpoint_triplet_s를 구성하여 uORB에 게시한다.

  • Previous Setpoint: 하강을 시작한 현재 공간 좌표 위치.
  • Current Setpoint: Z 벡터 상의 최종 도달 목표인 가상의 지면 좌표점 (추가적인 강하 회피 동작이 필요 없는 수직 경로).
  • Next Setpoint: 미할당(Null) 처리됨. (착륙이 가장 로우 레벨 모드이자 최종 모드이므로 이어지는 임무점이 없음)

이러한 삼단 스택(Triplet) 구조 재편을 통해 비행 프론트엔드는 완전히 동결된 XY 평면상의 점과 하향하는 Z 타겟에만 집중하는 궤도 스택을 유지하게 된다.

3. 하강 유도 및 셋포인트 매핑 아키텍처 흐름

아래의 Mermaid 시퀀스 다이어그램은 Commander에 의한 상태 전이 시점부터 Navigator가 경로 스택을 비우고 위치 제어기에 하강 목표점 세트를 매핑하는 전이(Transition) 과정을 보여준다.

sequenceDiagram
    participant Commander
    participant uORB as uORB Bus
    participant Navigator
    participant LandTask as Land Task (Guidance)
    participant PosCtrl as Position Controller

    Commander->>uORB: vehicle_status.nav_state = AUTO_LAND 퍼블리시
    uORB-->>Navigator: nav_state 변경 이벤트 수신
    activate Navigator
    Navigator->>Navigator: 기존 Mission/Hold 태스크 비활성화 및 스택 Flushing
    Navigator->>LandTask: on_activation() 호출 (착륙 유도 트리거)
    activate LandTask
    LandTask->>LandTask: 현재 XY 좌표 고정 (수평 호버링 축성립)
    LandTask->>LandTask: 지향 Z 좌표(지면 레벨) 설정
    LandTask-->>Navigator: 신규 position_setpoint 반환
    deactivate LandTask
    Navigator->>uORB: position_setpoint_triplet_s 퍼블리시
    deactivate Navigator
    
    uORB-->>PosCtrl: Triplet 데이터 수신
    PosCtrl->>PosCtrl: 고도 하향(Descend) 지향 위치/속도 제어 루프 개시

4. ArduPilot과의 아키텍처 철학적 대비

PX4-Autopilot의 Navigator 시스템 구조는 위치 목표(Target) 생성과 실제 위치 궤적 제어(Tracking) 책임을 매우 엄격하게 분리(Decoupling)하고 있다.

  • PX4-Autopilot:
    Navigator는 그저 점(Setpoint)을 찍어 uORB 버스에 올려둘 뿐, 기체가 어떠한 곡선 반경 혹은 어떠한 가속도 리미트로 이동할지는 전적으로 PositionController나 FlightTask 스택에 일임된다.
  • ArduPilot:
    ArduPilot(주로 Copter 아키텍처 기준)의 경우, 모드 클래스(ModeLand) 내부 또는 AC_WPNav와 같은 내비게이션 코어에서 위치, 속도 제한, 궤적 매핑이 다소 강하게 결합되어 동작한다. (모드가 직접 제어기의 Kinematics 호출을 전담)

이러한 PX4 특유의 완전 분리형 버스 기반( Bus-centric) 아키텍처 덕분에, 향후 ROS 2 브리지 커스텀 노드를 통해 Navigator를 완전히 우회하는 로직(Offboard Mode 기반의 자율 에이전트 관제 기술)을 구성하더라도, 위치 제어 모듈에 부하를 주지 않으며 매우 유동적이고 안정적인 자율 경로 스택(Path Stack) 재구성이 보장된다.
