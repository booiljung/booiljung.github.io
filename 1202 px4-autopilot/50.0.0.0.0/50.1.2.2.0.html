50.1.2.2 마이크로초(Microsecond) 단위 타임스탬프 동기화를 통한 네트워크 지연(Latency) 보상 및 시계열 데이터 정렬

본 절에서는 기준 버전인 PX4-Autopilot v1.14.0 및 QGroundControl(이하 QGC) v4.3.0을 바탕으로, MAVLink 통신 과정에서 발생하는 네트워크 지연(Latency)을 보상하고, 비행 제어기(FC) 내부에서 시계열 데이터를 정렬하기 위한 마이크로초(Microsecond) 단위의 타임스탬프 동기화 아키텍처를 분석한다.

1. 개요 및 정밀 시간 동기화의 필요성

자율 착륙(Autonomous Landing)이나 정밀 회피 기동과 같이 지면/장애물과 근접하여 수행되는 임무에서는 지상 관제 시스템(GCS) 또는 보조 컴퓨터(Companion Computer)가 내리는 제어 명령의 타이밍이 기체의 생존과 직결된다. 무선 텔레메트리나 Wi-Fi를 경유하는 MAVLink 패킷은 필연적으로 비결정적(Non-deterministic)인 전송 지연을 동반한다.

따라서 PX4-Autopilot은 GCS와 기체 간의 시스템 클록(System Clock)을 마이크로초 단위로 동기화(TIMESYNC)하고, 모든 송수신 메시지에 고정밀 타임스탬프를 부여하여 “언제 생성된 명령인지“를 기반으로 네트워크 지연을 보상하고 데이터의 시계열적 유효성을 판단한다.

2. TIMESYNC 프로토콜 기반 클록 오프셋 추정

PX4는 GCS와의 통신 지연 및 시스템 시계 차이를 계산하기 위해 MAVLink의 TIMESYNC 메시지(ID: 111)를 활용하여 NTP(Network Time Protocol)와 유사한 핑퐁(Ping-Pong) 동기화 알고리즘을 수행한다.

sequenceDiagram
    participant GCS as QGroundControl (GCS)
    participant FC as PX4-Autopilot (FC)
    
    Note over GCS, FC: t1: GCS 송신 시간, t2: FC 수신 시간<br>t3: FC 응답 시간, t4: GCS 수신 시간
    
    GCS->>FC: TIMESYNC (tc1=t1, ts1=0)
    activate FC
    FC->>FC: hrt_absolute_time()으로 t2 기록
    FC-->>GCS: TIMESYNC (tc1=t1, ts1=t2)
    deactivate FC
    
    GCS->>GCS: t4 시간에 수신 완료
    GCS->>GCS: Round Trip Time (RTT) = (t4 - t1)
    GCS->>GCS: Clock Offset = t2 - (t1 + RTT/2)

위 과정을 통해 GCS는 기체의 내부 고해상도 타이머(hrt_absolute_time()) 기준의 표준 시간을 얻게 되며, 이후 송신하는 MAV_CMD_NAV_LAND 등의 착륙 지시 패킷에 동기화된 타임스탬프를 삽입하여 전송한다.

3. PX4 미들웨어(uORB)의 시계열 데이터 정렬 아키텍처

PX4의 실시간 미들웨어인 uORB(Micro Object Request Broker) 구조에서 모든 메시지(.msg 파일)는 반드시 암묵적으로 첫 번째 필드로 uint64 timestamp를 갖도록 강제된다. 이는 설계 철학 상 매우 중요한 특징이다.

  1. 지연 보상(Latency Compensation): 수신 모듈(mavlink_receiver.cpp)은 수신된 패킷 내부의 타임스탬프와 현재의 hrt_absolute_time() 값을 비교하여, 패킷이 허용 가능한 안전 지연 시간(Timeout, 통상 0.5초 이내)을 초과했는지(Stale Data)를 즉각 판별한다. 만약 지연이 크다면 착륙 명령은 무효화된다.
  2. 시계열 버퍼 정렬(Time-series Ring Buffer): 확장 칼만 필터(EKF2)와 같은 센서 융합 모듈이나 커맨더(Commander) 로직은 도착 순서가 아닌 타임스탬프 순서대로 큐(Queue) 버퍼에 데이터를 정렬 및 재구성하여 위상(Phase) 오차를 보정한다.

4. 소스 코드 구현체 분석

코어 모듈인 src/modules/mavlink/mavlink_timesync.cpp 의 MavlinkTimesync::handle_message 함수 내부를 살펴보면, MAVLink 패킷 인터럽트가 발생할 때 하드웨어 타이머 레지스터를 직접 참조하여 1마이크로초 해상도의 시스템 타임을 채득하는 로직이 구현되어 있다.

이후 변환된 vehicle_command uORB 메시지는 수신 측 시스템 클록을 기준으로 재검교정된 타임스탬프를 획득하며, Commander 스레드에서는 vehicle_command.timestamp와 현재 시간의 델타(Delta) 값을 검사하여 낡은 명령(Old Command)의 실행을 원천 차단하는 방어 로직이 수행된다.

5. Ardupilot 대비 패킷 시간 스탬핑(Timestamping) 아키텍처의 차이

데이터 시간 정렬을 다루는 관점에서 PX4와 Ardupilot은 근본적인 아키텍처적 차이를 지닌다.

  • PX4-Autopilot: POSIX 아키텍처와 uORB 코어 스택 상에서 모든 메시지가 의무적(Mandatory)으로 64비트 정밀 타임스탬프를 보유한다. 이는 분산형 멀티스레딩(Multi-threading) 환경에 최적화되어, 어떤 스레드에서 데이터를 소비하든 데이터의 절대적 수명을 추적하고 버퍼링 기반 지연 보상을 일관되게 수행할 수 있다.
  • Ardupilot: 모놀리식(Monolithic) 구동 모델 중심의 AP_HAL 라이브러리 상에서 실행되므로, 스레드 간 지연이 상대적으로 적다. 따라서 시스템 전역에 걸친 개별 객체 단위의 무거운 타임스탬핑 강제보다는, 전역 변수와 단일 스케줄러 기준의 시간(AP_HAL::millis(), micros())을 참조하여 로컬 타임아웃을 관리하는 편의주의적(Pragmatic) 접근을 더 많이 채택하고 있다.

결론적으로, PX4의 엄격한 마이크로초 단위 타임스탬프 동기화 및 시계열 보상 모델은 통신 지터(Jitter)가 심한 ROS2 네트워크 환경이나 군집(Swarm) 관제 프로토콜 최적화에 있어서 뛰어난 수학적 무결성을 보장한다.
