50.5.1 지면 감지 모듈의 최종 Landed 상태 인터럽트(Interrupt) 신호 발생 및 커맨더 모듈로의 콜백(Callback) 전달 체계

본 절에서는 PX4-Autopilot(기준 펌웨어 버전 1.14 이상)의 지면 감지 데몬(land_detector)이 기체의 최종적인 착륙(Landed) 상태를 확정한 순간, 이를 시스템 전반에 알리는 이벤트 처리 및 커맨더(commander) 모듈로의 신호 전달 체계에 대해 심층 분석한다. 전통적인 실시간 운영체제(RTOS)의 하드웨어 인터럽트(Hardware Interrupt)와 유사한 논리적 콜백(Logical Callback) 메커니즘이 uORB(Micro Object Request Broker) 미들웨어를 통해 어떻게 구현되는지 이해하라.

1. 개요 및 설계 철학

네트워크로 분산된 현대적인 비행 제어 아키텍처에서 센서 데이터의 판단과 제어 모드의 변경은 철저히 모듈화(Modularization)되어 있다. 지면 접촉 여부를 순수하게 물리적 변수(추력, 가속도, 속도)로만 측정하는 land_detector 모듈은 기체의 상태(시동 상태 해제 등)를 직접적으로 변경할 권한을 갖지 않는다. 대신, 해당 상태가 충족되었을 때 논리적 인터럽트(Logical Interrupt) 형태의 메시지를 발행(Publish)하여 시스템 전반을 조율하는 주체인 커맨더 모듈이 비동기적으로 이를 수신하여 후속 로직을 처리하도록 설계되어 있다.

2. 최종 Landed 상태의 판별 및 메시지 발행(Publish)

land_detector 모듈의 메인 루프는 일정 주기로 여러 휴리스틱 조건(ground_contact, maybe_landed, landed)을 다단 필터링한다. 최종적으로 Landed 상태가 검증 회로를 통과하면 다음과 같은 절차로 신호를 발생시킨다.

  1. 상태 천이 로직 완료: 기체가 공회전 추력 임계치 이하에 도달하고, 수직 위치 변화율이 거의 0이며, 설정된 시간(히스테리시스 타임, Hysteresis Time) 동안 이 상태를 유지할 경우 조건이 충족된다.
  2. uORB 토픽 업데이트: 모듈 내부 구조체인 vehicle_land_detected_s의 landed 불리언(Boolean) 멤버 변수를 true로 설정한다.
  3. 블로킹 해제(Unblocking) 메인 이벤트: orb_publish(ORB_ID(vehicle_land_detected), ...); 함수를 호출하여 운영체제 계층에 변경된 데이터를 등록한다.

3. 커맨더 모듈의 콜백(Callback) 수신 및 논리적 인터럽트 처리 체계

3.1 파일 디스크립터(File Descriptor) 기반의 폴링(Polling) 대기

커맨더 모듈은 NuttX RTOS 기반에서 poll() 또는 px4_poll() 함수를 활용하여 자신이 관심 있는 다수의 uORB 토픽들의 파일 디스크립터(FD)를 구독 등록해 둔 상태로 대기한다. vehicle_land_detected 토픽도 이 모니터링 체인의 핵심 요소로 등록되어 있다.

3.2 콜백(Callback) 및 인터럽트 유사 전가 로직

land_detector에서 orb_publish()가 호출되는 순간, uORB 미들웨어는 해당 토픽을 구독하고 있는 모든 태스크의 FD에 쓰기(Write) 이벤트를 발생시킨다.
커맨더 모듈의 폴링 스레드는 즉각적으로 깨어나며(Wake-up), 마치 하드웨어 인터럽트를 받은 것과 유사한 지연 시간(Low Latency)으로 vehicle_land_detected 이벤트 비트를 파싱하게 된다.

  • 데이터 복사: orb_copy(...) 로 최신 토픽 데이터를 가져온다.
  • 상태 핸들링(Callback 함수 실행): 이전 사이클에서 landed == false 였으나 현재 true로 엣지(Edge) 트리거가 감지되면, 커맨더 내부의 착륙 후처리 함수인 콜백(Callback) 로직을 실행시킨다.
  • 후속 처리 병행: 시동 해제 타이머(Auto Disarm Timer)를 가동시키고, 비행 중단(Flight Termination) 안전성 루틴을 초기화한다.

5. Ardupilot 대비 아키텍처 비교 및 장단점

Ardupilot은 전통적으로 중앙 집중식 모놀리식(Monolithic) 아키텍처에 가까워, 비행 모드 단위(mode.cpp) 또는 AP_Vehicle 클래스 메인 루프 안에서 직접 AP_LandingDetector_update() 함수를 호출하고 리턴값을 확인하여 즉각 상태 분기를 진행한다.
이는 함수 콜 오버헤드가 적어 루프 처리가 직관적이라는 장점이 있다. 반면, PX4-Autopilot은 uORB라는 IPC(Inter-Process Communication) 미들웨어를 거치는 형태를 택하여, 이와 같은 논리적 콜백 체인 모듈화를 이끌어냈다. 그 결과 다소의 컨텍스트 스위칭(Context Switching) 오버헤드는 존재하나, 센서 모듈, land_detector 모듈, commander 모듈의 결합도를 낮추어 유연하게 알고리즘 전반을 확장하거나 단위 테스트(Unit Test)를 수행할 수 있는 탁월한 학술적 장점을 지닌다.

6. 결론

지면 감지 모듈의 Landed 상태 확정은 기체의 물리적 행위에 대한 결론이며, 이것이 시스템 전반의 상태 변환으로 이어지기 위해서는 커맨더 모듈로의 논리적 인터럽트 트리거와 콜백 체계가 필수적이다. PX4의 uORB 파일 디스크립터 폴링(Polling) 메커니즘을 통한 이 일련의 구조를 숙지함으로써, 독자적인 추가 제어 알고리즘 구현 시 다중 스레드 환경에서 데이터 변경 이벤트를 지연 없이 신뢰성 있게 캡처하는 핵심 기법을 배양하라.
