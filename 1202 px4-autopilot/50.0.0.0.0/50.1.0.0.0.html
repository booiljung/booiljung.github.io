50.1. 착륙(Land) 명령의 시스템적 정의와 MAVLink 데이터 링크 계층의 역직렬화(Deserialization) 구조

무인항공기(UAV)의 착륙 명령은 기체의 운용 시나리오에 따라 지상 관제 시스템(GCS, Ground Control Station)에서 발동될 수도 있고, 동반 컴퓨터(Companion Computer) 내부의 자율 에이전트(Autonomous Agent) 알고리즘에 의해 오프보드(Offboard) 형태로 궤적이 생성될 수도 있다. 어떤 방식이든 물리적인 비행 제어기(Flight Controller)에 최종 착륙 의도를 전달하는 표준 인터페이스는 MAVLink 데이터 링크 계층(Data Link Layer)을 통해 이루어진다. 본 절에서는 착륙 명령의 시스템적 정의를 확립하고, MAVLink 패킷이 PX4-Autopilot 내부로 인입되어 처리되는 역직렬화(Deserialization) 구조를 아키텍처 관점에서 분석한다.

1. 착륙 명령의 시스템적 정의

기본적으로 기체에 하달되는 제어 명령(Control Command)은 크게 두 가지로 분류할 수 있다. 첫째는 조종기의 스틱을 통한 **수동 제어 입력(Manual Control Input)**이고, 둘째는 디지털 프로토콜을 이용한 **자동화된 차량 명령(Automated Vehicle Command)**이다. ‘착륙(Land)’ 명령은 후자에 속하며, 단순한 수직 하강 제어를 넘어서 기체 내부의 복합적인 유한 상태 기계(FSM, Finite State Machine) 전환을 동반하는 논리적 명령으로 정의된다.

PX4-Autopilot 구조에서 착륙 명령이 유효하기 위해서는 다음의 전제 조건 모니터링 체계가 성립해야 한다.

  • 안전 센서 검증: GPS, 고도계(기압계, 거리 측정계 등) 시스템이 정상이어야 한다.
  • 수락 가능성(Acceptability) 평가: 현재 기체의 비행 모드(Flight Mode)에서 착륙 모드로 전이(Transition)할 수 있는 상태여야 한다. 예를 들어 고정익(Fixed-wing) 기체의 경우 활공각과 착륙 지점이 유효한지 커맨더(Commander) 내부 로직에서 1차적인 평가가 수행된다.

1.1 Ardupilot과의 차별점

Ardupilot 환경에서는 착륙 제어가 개별 비행 코드(Copter, Plane) 내부의 ModeLand 클래스로 파편화되어 다루어지는 반면, PX4-Autopilot은 Commander라는 전역 객체가 명령의 유효성을 총괄 검증(Validation)한 이후, Navigator로 제어권을 넘겨 일관성 있는 궤적을 그리도록 설계되었다는 점에서 모듈화 완성도가 높다.

2. MAVLink 데이터 링크 계층과 역직렬화 동작 원리

지상 관제 시스템인 QGroundControl(QGC, v4.3 기준)이나 ROS2 MAVROS/uXRCE-DDS 노드에서 발송된 착륙 명령은 직렬 포트(UART)나 네트워크 인터페이스(UDP/TCP)를 통해 비행 제어기로 인입된다.

2.1 명령어 패킷 사양 (MAV_CMD_NAV_LAND)

MAVLink 표준에서 착륙은 MAV_CMD_NAV_LAND (Command ID: 21) 명령으로 구현되며, 일반적으로 COMMAND_LONG (Message ID: 76) 또는 COMMAND_INT (Message ID: 75) 메시지 포맷에 실려 전송된다. 주요 파라미터는 다음과 같다.

  • param1: Abort Alt (착륙 취소 고도 / 사용하지 않을 경우 0)
  • param2: Land Mode (정밀 착륙 등 세부 모드 지정)
  • param3: 빈 칸 (Reserved)
  • param4: Yaw / Heading (착륙 시 기수 방향)
  • param5, param6, param7: Target Latitude(X), Target Longitude(Y), Target Altitude(Z) 또는 로컬 좌표계 정보

2.2 역직렬화 및 uORB 브릿징(Bridging) 프로세스

PX4-Autopilot 내부의 mavlink 모듈(수신부 구동 프로세스)은 바이트 스트림(Byte Stream)을 수신하면 이를 식별(Parsing)하여 다음과 같은 구조체에 값을 매핑하는 역직렬화 로직을 수행한다.

// 수신 패킷을 디코딩하는 MAVLink 라이브러리 함수 호출 예시
mavlink_command_long_t cmd_mavlink;
mavlink_msg_command_long_decode(msg, &cmd_mavlink);

역직렬화가 완료된 MAVLink 메시지는 시스템 내부의 실시간 운영체제(RTOS, NuttX) 간 통신 미들웨어인 uORB(Micro Object Request Broker) 브릿지 계층을 거쳐 순수 PX4 자료 구조인 vehicle_command_s 구조체로 변환된다.

해당 처리 파이프라인(Pipeline)을 다이어그램으로 도식화하면 다음과 같다.

도표에서 보듯, 역직렬화 루틴(Deserialization Routine)은 단순한 데이터 복사를 넘어서 페이로드 크기나 프로토콜 버전(MAVLink v2 서명 검증 등)을 체크하며 보완재 역할을 수행한다. 변환 결과물인 vehicle_command 토픽이 uORB 트리에 발행(Publish)되면, 커맨더 모듈 등의 구독(Subscriber) 노드들이 즉시 이벤트를 폴링(Polling)하여 착륙 진입 판단 로직을 시작하게 된다.

이처럼 MAVLink 메시지가 내부 uORB 통신 버스로 매핑되는 브릿징 패턴은 GCS나 ROS2와 같은 외부 에이전트(Agent)의 요구사항과 비행 제어 알고리즘(Flight Control Algorithm) 내부의 논리를 완벽히 분리시켜 시스템의 확장성을 높이는 데 핵심적인 역할을 한다.
