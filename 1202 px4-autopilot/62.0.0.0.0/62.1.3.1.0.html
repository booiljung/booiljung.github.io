62.1.3.1 바이트 순서(Byte Order) 불일치 해결을 위한 엔디안(Endianness) 변환 선형 연산 로직

다양한 임베디드 장치와 컴퓨팅 보드가 맞물려 동작하는 로봇 OS 생태계에서, 메모리에 데이터를 기록하는 바이트 순서(Byte Order), 즉 엔디안(Endianness)의 불일치는 심각한 통신 오류를 유발하는 주원인 중 하나이다. 예를 들어, PX4-Autopilot의 주력 하드웨어인 Pixhawk 소속의 STM32 마이크로컨트롤러는 리틀 엔디안(Little-Endian)을 사용하는 반면, 일부 레거시 센서 모듈이나 특정 네트워크 프로토콜 스택은 빅 엔디안(Big-Endian)을 표준으로 채택하기도 한다. 이 절에서는 Micro XRCE-DDS가 상이한 아키텍처 간의 바이트 순서 불일치를 해결하기 위해 적용하는 엔디안 변환 선형 연산 로직과 그 최적화 원리를 분석한다.

1. CDR 모델의 “수신자 변환(Receiver Makes it Right)” 패러다임

과거의 많은 네트워크 프로토콜들은 전송 효율성이나 통일을 위해 송신자가 무조건 ’네트워크 바이트 순서(일반적으로 빅 엔디안)’로 변환하여 보내도록 강제했다. 그러나 OMG의 DDS CDR 표준은 비행 제어기 같은 송신 단말의 연산 부하를 줄이기 위해 송신자는 자신의 네이티브(Native) 엔디안 그대로 데이터를 전송하고, 변환의 책임을 수신자에게 넘기는 “수신자 변환(Receiver Makes it Right)” 패러다임을 따른다.

이를 위해 직렬화된 데이터 페이로드의 최상단(Encapsulation Header)에는 해당 메시지가 리틀 엔디안인지 빅 엔디안인지를 명시하는 1바이트의 플래그(Endianness Flag)가 반드시 포함된다.

2. 수신 측의 엔디안 검증 및 변환 상태 기계

수신 측(예: 컴패니언 컴퓨터 상의 에이전트 또는 제어 명령을 수신하는 PX4 클라이언트)은 메시지의 헤더 플래그를 읽고 자신의 시스템 아키텍처(Endianness)와 비교하여 변환 여부를 결정한다.

flowchart TD
    A[페이로드 수신] --> B[Encapsulation 헤더 파싱]
    B --> C{메시지 엔디안 == 로컬 시스템 엔디안?}
    C -- Yes (일치) --> D[변환 생략: Zero-Copy 또는 memcpy 수행]
    C -- No (불일치) --> E[자료형 크기에 따른 바이트 스왑(Byte Swap) 연산 분기]
    
    E --> F[2바이트(16-bit) 스왑]
    E --> G[4바이트(32-bit) 스왑]
    E --> H[8바이트(64-bit) 스왑]
    
    F --> I[역직렬화 및 uORB 퍼블리싱]
    G --> I
    H --> I
    D --> I

만약 두 엔디안이 일치한다면 배열을 그대로 읽어 들이며(memcpy), 불일치할 경우에만 자료형의 선언된 길이에 맞추어 바이트의 앞뒤 순서를 뒤집는 선형 블록 연산을 수행한다.

3. 크기별 바이트 스왑(Byte Swap) 선형 연산 로직

바이트 순서를 역방향으로 재배치하는 작업은 비트 단위 논리 연산(Bitwise Operations)의 조합인 시프트(Shift)와 마스킹(Masking) 기법을 사용한다. PX4의 uxrce_dds_client 내부에 포함된 microCDR 라이브러리는 다음과 같은 수학적 선형 로직을 통해 빠른 속도로 배열의 비트 값을 변환한다.

3.1 16비트 (2바이트) 변환

16비트 정수형(int16_t, uint16_t)의 경우 상위 8비트와 하위 8비트의 자리를 바꾼다.
Y = ((X \ \& \ \text{0x00FF}) \ll 8) \ \vert \ ((X \ \& \ \text{0xFF00}) \gg 8)
비트 시프트 연산과 논리 합을 통해, 하위 바이트를 좌측으로 8칸 밀고 결합하여 상위 바이트와 교체한다.

3.2 32비트 (4바이트) 변환

32비트 데이터(예: int32_t, float)는 1바이트씩 쪼개어 가장 끝에 있는 바이트가 가장 앞쪽으로 오도록 역순 배치한다.

uint32_t swap_32(uint32_t x) {
    return ((x & 0x000000FF) << 24) |
           ((x & 0x0000FF00) <<  8) |
           ((x & 0x00FF0000) >>  8) |
           ((x & 0xFF000000) >> 24);
}

여기서 부동 소수점(float) 데이터는 메모리 상에서 직접적인 비트 시프트 연산이 불가능하므로, 포인터 캐스팅을 통해 메모리를 임시로 uint32_t 형태로 읽어 들인 후 바이트 스왑 연산을 수행하고, 그 결과를 다시 float 메모리 블록에 덮어쓰는 포인터 앨리어싱(Pointer Aliasing) 우회 기법을 주로 사용한다.

3.3 64비트 (8바이트) 변환

double이나 uint64_t와 같은 8바이트 데이터 구조는 가장 연산이 무겁기 때문에, 32비트 변환 연산을 양분하여 대칭적으로 결합하거나 하드웨어 지원 명령어를 최우선적으로 활용하도록 포팅(Porting) 되어 있다.

4. 컴파일러 내장 함수(Intrinsics)를 이용한 하드웨어 가속 최적화

수학적 수치 연산만으로 구성된 스왑 함수는 반복문에 수십 바이트 이상 적용될 경우 PX4의 타임-크리티컬 루프(Time-critical loop) 안에서 필연적인 병목을 일으킬 수 있다. 따라서 최신 ARM-GCC 컴파일러 툴체인(Toolchain)을 기반으로 빌드되는 PX4 펌웨어는, C언어 수준의 수학적 스와핑 로직 대신 컴파일러 내장 함수(Compiler Intrinsics)를 사용하여 어셈블리(Assembly) 레벨의 타겟 칩셋 가속 명령어로 교체한다.

  • 예를 들어 GCC 환경에서 __builtin_bswap32()나 __builtin_bswap64() 매크로를 호출하면, ARM 아키텍처는 이를 전용 바이트 반전 하드웨어 명령어(REV 패밀리) 단 한 사이클만으로 처리하여 결과를 반환한다.
  • 이러한 최적화 아키텍처를 통해 엔디안 불일치가 발생하더라도, 마이크로컨트롤러에 가해지는 역직렬화 지연 시간(Latency) 패널티를 획기적으로 낮출 수 있다.

5. 결론

이기종 간의 데이터를 통합하는 XRCE 통신 브리지 체계에서, 바이트 순서 불일치는 회피할 수 없는 난제이다. 그러나 수신자 판단 기반의 합리적인 변환 시스템 배치와 함께 비트 시프트 기반 선형 연산, 더 나아가 컴파일러 고유의 단일 명령어(REV) 최적화를 융합함으로써, PX4-Autopilot은 이질적인 ROS 2 컴퓨터와의 인터페이스 환경에서도 컴퓨팅 파워 손실 없이 정교한 데이터 무결성 확보와 안전한 비행 제어를 완수할 수 있게 된다.
