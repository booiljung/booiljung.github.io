62.4.4 비동기 명령(Asynchronous Command) 하달: VehicleCommand 메시지 직렬화 및 확인(Acknowledgment) 프로토콜

최신 PX4-Autopilot(펌웨어 버전 1.14 이상)과 ROS 2(Robot Operating System 2)가 결합된 아키텍처에서, 비행체의 동작 모드 변경, 시동(Arming), 이륙(Takeoff), 특정 액추에이터 제어 등은 오프보드(Offboard) 노드에 의한 비동기 명령(Asynchronous Command)으로 하달된다. 본 절에서는 이러한 비동기 명령을 전달하는 핵심 매개체인 VehicleCommand 메시지의 직렬화(Serialization) 구조와, 명령 수행의 신뢰성을 보장하기 위한 VehicleCommandAck 기반의 확인응답(Acknowledgment) 프로토콜에 대해 심층적으로 다룬다.

1. 비동기 명령 시스템의 개요

단기적인 궤적(Trajectory)이나 자세 추종을 위해 백헤르츠(100Hz) 이상의 고빈도로 연속 퍼블리싱(Continuous Publishing)되는 동기식 스트림 데이터(예: TrajectorySetpoint)와 달리, 기체의 근본적인 상태 천이(State Transition)를 유발하는 제어 명령은 비동기적(Asynchronous) 이벤트로 처리된다. PX4 내외부에서 발생하는 이러한 이벤트는 VehicleCommand uORB 토픽을 통해 전달된다.

과거에는 MAVLink의 COMMAND_LONG 또는 COMMAND_INT 메시지를 지상 관제 시스템(GCS, 예: QGroundControl)이나 보조 컴퓨터(Companion Computer)가 직접 전송했으나, ROS 2 환경(Micro XRCE-DDS)에서는 px4_msgs::msg::VehicleCommand 메시지를 퍼블리시하여 동일한 효과를 낸다. 이 메시지는 DDS 네트워크를 거쳐 직렬화된 후 PX4 내부의 uORB 토픽으로 매핑(Mapping)된다.

2. VehicleCommand 메시지 구조 및 직렬화 (Serialization) 구조

VehicleCommand 메시지는 MAVLink의 커맨드 구조를 uORB 생태계로 투영한 형태를 지닌다. 주요 멤버 변수와 그 직렬화 페이로드(Payload) 특성은 다음과 같다.

  • command (uint32): 수행할 특정 명령의 고유 식별자(ID). (예: VEHICLE_CMD_COMPONENT_ARM_DISARM = 400)
  • param1 ~ param7 (float32): 단정밀도 부동소수점(Single-precision floating-point) 규격을 따르는 7개의 매개변수. 각 명령(Command ID)의 정의에 따라 매개변수의 의미가 동적으로 결정된다.
  • target_system, target_component (uint8): 다중 에이전트(Swarm) 환경에서 명령 수신자의 논리적 주소를 지정한다. 일반적으로 싱글 에이전트 브릿지 환경에서는 각각 1로 설정된다.
  • source_system, source_component (uint8): 명령 발신자의 식별자.
  • confirmation (uint8): 동일한 명령의 재전송 횟수 등을 나타내어, 네트워크 유실 시 커맨더 모듈이 중복 처리를 판별하는 데 사용된다.

ROS 2 노드에서 이 구조체 인스턴스를 생성하면 커스텀 메시지 타입 지원을 위해 RTPS/CDR(Common Data Representation) 표준에 맞춰 바이트 배열로 직렬화(Serialization)된다. 직렬화된 버퍼는 XRCE-DDS 에이전트에 의해 UART/UDP를 거쳐 PX4 클라이언트로 전송되고, PX4 내부 보편 언어인 uORB 객체로 역직렬화(Deserialization)된다.

3. 확인응답 프로토콜 (Acknowledgment Protocol) 및 신뢰성 확보

비동기 제어 시스템에서는 명령이 유실되거나 조건 불만족으로 인해 기체가 명령을 거부(Deny)할 가능성이 항상 존재한다. 이를 극복하고 폐루프 통신 무결성을 검증하기 위해, PX4 커맨더(Commander) 모듈은 VehicleCommand를 수신 및 처리한 직후 VehicleCommandAck 메시지를 발행한다.

응답 상태를 나타내는 VehicleCommandAck::result 필드의 주요 값은 다음과 같다.

  • VEHICLE_CMD_RESULT_ACCEPTED (0): 명령이 정상적으로 수용 및 실행됨.
  • VEHICLE_CMD_RESULT_TEMPORARILY_REJECTED (1): 시스템 초기화 결여 등 일시적 사유로 거부. 재전송 시도 가능.
  • VEHICLE_CMD_RESULT_DENIED (2): 비행 상태 상 명령 수행이 불가함 (예: 비행 중 시동 해제 명령).
  • VEHICLE_CMD_RESULT_UNSUPPORTED (3): 펌웨어에서 지원하지 않는 명령어 ID.

ROS 2 제어 노드는 명령을 퍼블리시한 후, 타이머 인터럽트(Timer Interrupt) 또는 콜백 블로킹을 통해 VehicleCommandAck를 서브스크라이브(Subscribe)하여 상태 전이가 완료되었는지 확인상태를 평가해야 한다.

5. ROS 2 C++ 기반 명령 및 확인응답 구현 예제

기체에 시동(Arm)을 거는 비동기 명령을 전송하고, 반환되는 확인응답을 처리하는 비동기 객체 지향 구현의 일부를 간략히 나타낸다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_command.hpp>
#include <px4_msgs/msg/vehicle_command_ack.hpp>

using namespace std::chrono_literals;

class AsyncCommandNode : public rclcpp::Node {
private:
    rclcpp::Publisher<px4_msgs::msg::VehicleCommand>::SharedPtr cmd_pub_;
    rclcpp::Subscription<px4_msgs::msg::VehicleCommand_ack>::SharedPtr ack_sub_;

    void ack_callback(const px4_msgs::msg::VehicleCommandAck::SharedPtr msg) {
        if (msg->command == px4_msgs::msg::VehicleCommand::VEHICLE_CMD_COMPONENT_ARM_DISARM) {
            if (msg->result == px4_msgs::msg::VehicleCommandAck::VEHICLE_CMD_RESULT_ACCEPTED) {
                RCLCPP_INFO(this->get_logger(), "Arming command accepted by PX4.");
            } else {
                RCLCPP_ERROR(this->get_logger(), "Arming command rejected. Error Code: %d", msg->result);
            }
        }
    }

public:
    AsyncCommandNode() : Node("async_cmd_node") {
        cmd_pub_ = this->create_publisher<px4_msgs::msg::VehicleCommand>("/fmu/in/vehicle_command", 10);
        
        ack_sub_ = this->create_subscription<px4_msgs::msg::VehicleCommandAck>(
            "/fmu/out/vehicle_command_ack", 10,
            std::bind(&AsyncCommandNode::ack_callback, this, std::placeholders::_1)
        );
    }

    void request_arm() {
        px4_msgs::msg::VehicleCommand msg{};
        msg.target_system = 1;
        msg.target_component = 1;
        msg.source_system = 1;
        msg.source_component = 1;
        
        msg.command = px4_msgs::msg::VehicleCommand::VEHICLE_CMD_COMPONENT_ARM_DISARM;
        // param1 = 1.0f (Arm), 0.0f (Disarm)
        msg.param1 = 1.0f;
        
        msg.from_external = true;
        msg.timestamp = this->now().nanoseconds() / 1000ULL;

        cmd_pub_->publish(msg);
        RCLCPP_INFO(this->get_logger(), "VehicleCommand: Arm requested.");
    }
};

6. 결론 및 참고 문헌

ROS 2 및 PX4 오프보드 환경에서의 제어 신뢰성은 단순히 메시지를 발행하는 것에 그치지 않고, VehicleCommand의 하달과 VehicleCommandAck의 처리 루프를 온전히 구현하는 데 달려있다. 개발자는 네트워크 손실이나 예기치 않은 상태로 인한 거부(Deny) 현상을 고려하여, 강건한 재전송(Retry) 및 예외 처리 로직(Exception Handling)을 자율 에이전트(Autonomous Agent) 소프트웨어 스택에 포함시켜야 한다.
