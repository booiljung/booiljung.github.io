62.6.5. 마이크로 에이전트 커널 로그의 정형 데이터 변환 및 상태 전이 그래프(State Transition Graph) 이상 징후 패턴 인식(Pattern Recognition) 기법

현대의 무인항공기 시스템(UAS)은 고도로 복잡한 분산 아키텍처를 기반으로 작동한다. 특히 PX4-Autopilot(v1.14.0 기준)과 ROS2(vHumble 기준)가 연동되는 환경에서는 수많은 노드(Node) 간의 비동기적 통신이 발생하며, 이를 중재하는 마이크로 에이전트(Micro Agent, 주로 uXRCE-DDS)의 역할이 매우 중요하다. 본 절에서는 마이크로 에이전트의 커널 로그(Kernel Log)를 기계가 읽을 수 있는 정형 데이터(Structured Data)로 변환하는 과정과, 변환된 데이터를 바탕으로 상태 전이 그래프(State Transition Graph)를 구축하여 이상 징후(Anomaly) 패턴을 인식하는 고급 기법에 대해 상세히 다룬다.

1. 마이크로 에이전트 커널 로그의 특성과 정형화 파이프라인

기체 내부에 탑재된 비행 제어기(Flight Controller)의 실시간 운영체제(RTOS, 예: NuttX)와 컴패니언 컴퓨터(Companion Computer) 사이에서는 브릿지(Bridge)를 통한 방대한 양의 직렬화(Serialization)된 데이터가 교환된다. 이때 에이전트 단에서 발생하는 커널 로그는 시스템의 상태를 진단할 수 있는 가장 원초적인 정보를 담고 있다.

1.1 비정형 로그의 한계 및 데이터 파싱(Parsing) 전략

기본적으로 커널 로그는 사람이 읽기 위한 비정형 텍스트(Unstructured Text) 형태로 출력된다. 이를 자동화된 이상 징후 탐지 시스템에 입력하기 위해서는 개별 파라미터로 속성을 분리해 내는 정형화(Structuring) 과정이 필수적이다. 정규 표현식(Regular Expression)이나 AST(Abstract Syntax Tree) 분석 기법을 활용하여 다음의 핵심 요소들을 추출한다.

  • 타임스탬프(Timestamp): 마이크로초 단위의 이벤트 발생 시간
  • 토픽 식별자(Topic ID): 메시지 교환에 사용된 uORB 토픽(예: VehicleLocalPosition)
  • 버퍼 상태(Buffer Status): 송수신 큐(Queue)의 포화도 및 오버플로우 발생 여부
  • 오류 코드(Error Code): 통신 거부(Request Denied)나 타임아웃(Timeout) 등에 해당하는 시스템 반환 값

예시: 비정형 로그 문자열을 JSON 형태의 정형 데이터로 변환하는 과정

// 비정형 원본 로그: "[uXRCE] ERROR: Drop packet, QOS incompatibility in topic VehicleOdometry"
{
  "timestamp": 1693820491.123456,
  "level": "ERROR",
  "component": "uXRCE",
  "event_type": "Packet_Drop",
  "topic": "VehicleOdometry",
  "reason": "QOS_Incompatibility"
}

2. 상태 전이 그래프(State Transition Graph) 모델링

마이크로 에이전트와 PX4-Autopilot의 커맨더(Commander) 모듈 간의 상호작용은 수학적인 가설인 유한 상태 기계(Finite State Machine, FSM)로 모델링될 수 있다. 상태 전이 그래프는 각 시스템의 논리 상태를 노드(Node)로 취급하고, 특정 이벤트(Event)에 의한 상태 변경을 간선(Edge)으로 표현한다.

2.1 정상 전이와 이상 경로의 교차 분석(Cross-Analysis)

아래의 Mermaid 다이어그램은 마이크로 에이전트의 통신 세션 수립 단계부터 데이터 발행(Publishing) 과정까지의 상태 전이를 보여준다. 붉은 점선은 통신 병목이나 데이터 패킷 유실로 인해 발생할 수 있는 대표적인 이상 징후(Anomaly) 전이 경로를 의미한다.

stateDiagram-v2
    [*] --> Initialization : 부팅 및 에이전트 시작
    Initialization --> Handshake : 직렬 통신 포트 설정
    Handshake --> Session_Established : uXRCE-DDS 세션 수립 완료
    Session_Established --> Subscribing : uORB 토픽 구독 (QoS 체크)
    Subscribing --> Data_Transfer : 정상적인 데이터 퍼블리싱
    
    %% 이상 징후 경로 모델링
    Handshake --> Handshake_Failed : 타임아웃 발생 (이상 징후)
    Subscribing --> QoS_Mismatch : 대역폭 및 호환성 오류 (이상 징후)
    Data_Transfer --> Buffer_Overflow : 데이터그램 혼잡 (이상 징후)
    Buffer_Overflow --> Session_Dropped : 연결 단절 및 Failsafe
    
    Handshake_Failed --> Initialization : 재시도 로직
    Session_Dropped --> Initialization : 예외 복구 프로세스 전환

이와 같이 Ardupilot의 단일 텔레메트리 기반 로깅 구조(DataFlash)에 비해, PX4-Autopilot의 분산 통신 모델(uORB 및 XRCE-DDS)은 각 노드별 상태 전이를 모니터링할 수 있어 시스템 오작동 원인을 더 깊숙이 추적할 수 있는 장점이 있다.

3. 이상 징후 패턴 인식(Pattern Recognition) 기법

상태 전이 그래프로 표현된 시스템의 로그 궤적을 기반으로 이상 징후를 판별하기 위해, 통계적 분석(Statistical Analysis) 및 머신 러닝(Machine Learning) 기술이 융합되어 사용된다.

3.1 통계적 거리 측정 및 마르코프 체인(Markov Chain) 기반 감지

정상적인 비행 시나리오에서는 각 상태(State)에서 다른 상태로 넘어갈 확률이 일정하게 수렴한다. 마르코프 체인 모델을 활용하여 현재 상태에서 확률적으로 매우 낮은 전이(Zero-Probability Transition)가 발생했을 때 이를 이상 징후로 판단한다.

QoS 호환성이나 대역폭의 한계로 인해 주기적으로 패킷 손실이 발생할 경우, 상태 전이 간의 지연 시간(Latency) 분포는 가우스 분포(Gaussian Distribution)를 벗어나 꼬리가 긴(Heavy-tailed) 분포를 보이게 된다. 이때, 마할라노비스 거리(Mahalanobis Distance) 수식을 통해 데이터의 산포도 이상을 탐지할 수 있다. 테이블 형식으로 표현된 비교 매트릭스는 다음과 같다.

판별 기준 (Criteria)정상 상태 전이 (Normal)이상 징후 발생 시 (Anomaly)
전이 확률 (P)P(S_{t+1} \vert S_t) > 0.95P(S_{t+1} \vert S_t) < 0.01 (희소 전이)
지연 시간 분산\mu \pm 2\sigma 이내 수렴\sigma 범위를 초과하는 극단적 아웃라이어 존재
커널 로그 패턴[INFO], [DEBUG] 메시지 주류[WARN], [ERROR], [FATAL] 군집화 발생

3.2 머신 러닝 알고리즘의 적용: 시계열 은닉 마르코프 모델(HMM)

단순한 임계치(Threshold) 검사를 넘어, 시간의 흐름에 따른 상태 변화 시퀀스를 포착하기 위해 은닉 마르코프 모델(Hidden Markov Model, HMM)이나 장단기 메모리(LSTM) 신경망을 활용한다. 이는 겉으로 드러나는 에이전트의 로그(관측 상태)를 통해 시스템 내부 깊숙한 곳의 교착 상태(Deadlock)나 자원 부족(Hidden State)을 성공적으로 추정해 낼 수 있게 해준다.

4. QGroundControl 연동 및 Failsafe 트리거

이러한 패턴 인식 파이프라인 컴패니언 컴퓨터 상에서 동작하는 독립적인 노드로 구현된다. 이상 징후가 패턴 인식 알고리즘에 의해 확정되면, 해당 노드는 MAVLink 프로토콜 또는 ROS2 토픽을 통해 지상 관제 시스템(QGroundControl, QGC)에 알람 명령을 즉시 송출한다. QGC v4.3.0 이상 버전에서는 수신된 사용자 정의 경고 텍스트를 파싱하여 화면상에 팝업을 띄우거나, 기체의 모드를 비상 모드(Return to Launch 또는 Land)로 강제 변환하도록 Failsafe 로직을 프로그래밍할 수 있다.
