62.1.4.2 데이터 내구성(Durability) 및 이력(History) 정책: 비휘발성(Non-volatile) 메모리 매핑 및 원형 큐(Circular Queue) 포인터 연산 메커니즘

PX4-Autopilot 기반의 무인항공기 운영 중에는 통신 링크가 일시적으로 단절되거나 지상 관제 시스템(GCS) 및 보조 컴퓨터(Companion Computer) 내부의 ROS 2 노드가 기체 기동 시점보다 늦게 구동(Late-Joiner)되는 상황이 빈번하게 발생한다. 이 경우, 이미 퍼블리싱된 과거의 로직이나 주요 데이터가 새로 참여한 시스템에 어떻게 전달되어야 하는지에 대한 기준이 필요하다. 본 절에서는 이러한 데이터 생존 및 저장 방식을 통제하는 데이터 내구성(Durability) 및 이력(History) QoS 정책의 아키텍처와, 마이크로컨트롤러 환경에서의 원형 큐(Circular Queue) 포인터 연산 및 메모리 매핑 기법을 다룬다.

1. 데이터 내구성(Durability) 정책

내구성(Durability) 정책은 퍼블리셔(Publisher)가 생산한 데이터가 네트워크 상에서, 혹은 메모리 버퍼 상에서 얼마나 오랫동안 생존하여 새로운 서브스크라이버(Subscriber)에게 전달될지를 결정한다.

1.1 내구성 정책의 종류 (PX4/uXRCE-DDS 지원 기준)

  • VOLATILE (휘발성): 데이터가 발행되는 시점에 연결되어 있는 구독자에게만 전송된다. 데이터는 전송 직후 즉시 파기되며, 늦게 참여한 노드는 이전 데이터를 수신할 수 없다. sensor_gyro나 vehicle_local_position과 같이 과거 데이터의 의미가 없는 고주파 텔레메트리 스트림에 적용된다.
  • TRANSIENT_LOCAL (지연 전달 보장): 퍼블리셔의 생존 기간(Lifespan) 동안 데이터가 메모리에 유지되며, 늦게 연결된(Late-joining) 구독자에게도 과거 데이터(History 내에 보관된 분량)가 전송된다. 특정 모듈의 초기 파라미터 상태나 vehicle_status와 같은 저주파 상태 정보에 필수적이다. (단, 퍼블리셔 프로세스가 종료되면 데이터도 소멸한다.)
  • TRANSIENT / PERSISTENT: 퍼블리셔의 종료 이후에도 데이터 생존을 보장하기 위해 비휘발성(Non-volatile) 매체에 저장하는 정책이다. (다만, uXRCE-DDS 환경 등 리소스가 제약된 마이크로 미들웨어에서는 일반적으로 지원하지 않거나 제한적으로만 응용된다.)

1.2 비휘발성(Non-volatile) 메모리 매핑의 개념적 결합

PX4 내부에서 TRANSIENT에 준하는 지속성을 확보하기 위해, 중요 파라미터나 비행 이력 모드 값들은 SD 카드나 FRAM, EEPROM과 같은 비휘발성(Non-volatile) 메모리 매핑(Memory Mapping) 기법과 연계되어 관리된다. ROS 2 측에서 TRANSIENT_LOCAL로 최신 상태를 요구할 때, PX4 측 퍼블리셔는 RAM의 큐 이력뿐만 아니라 Flash 메모리에 백업된 시스템 내부 파라미터 트리를 조회하여 데이터를 동기화하는 백엔드 로직을 거치기도 한다.

2. 데이터 이력(History) 정책

내구성 정책이 “늦게 참여한 노드에게 데이터를 보낼 것인가“를 다룬다면, 이력(History) 정책은 “그렇다면 과거 데이터를 몇 개나 보낼(보관할) 것인가“를 정의한다.

  • KEEP_ALL: 메모리가 허용하는 한 모든 데이터를 보관한다. 임베디드 장치인 픽스호크(Pixhawk) 시리즈에서는 메모리 버퍼 오버플로우를 유발할 수 있으므로 극도로 제한적으로 사용된다.
  • KEEP_LAST: 가장 최신의 데이터 N개(Depth 파라미터로 지정)만을 보관한다. 새로운 데이터가 생성되면 가장 오래된 데이터는 삭제(Overwritten)된다. 시스템 리소스를 통제할 수 있어 PX4-ROS 2 연동에서 가장 널리 쓰이는 표준 방식이다.

3. 원형 큐(Circular Queue)를 이용한 이력 버퍼 관리 시스템

PX4 내부의 uORB 아키텍처 및 미들웨어 클라이언트는 KEEP_LAST 속성의 윈도우 인덱싱을 고효율로 처리하기 위해 원형 큐(Circular Queue) 또는 링 버퍼(Ring Buffer) 구조를 적용하여 이력을 관리한다.

3.1 원형 큐 아키텍처

원형 큐는 고정된 크기의 배열을 할당하고, 헤드(Head)와 테일(Tail) 포인터를 사용하여 데이터의 삽입(Push)과 삭제/오래된 값의 갱신(Overwrite)을 단일 사이클로 처리할 수 있는 O(1) 연산 복잡도를 지닌 자료구조이다.

  • 삽입(Push) 동작: 새로운 데이터가 들어올 때 테일(Tail) 포인터 위치에 기록하고 포인터를 1칸 이동시킨다.
  • 오버라이트 메커니즘: 배열의 끝에 도달하면 포인터를 0으로 래핑(Wrapping)하며, 이때 Depth를 초과하게 되면 헤드(Head) 포인터가 함께 전진하여 가장 오래된 데이터가 자동으로 소멸된다.

3.2 포인터 산출 모듈로(Modulo) 연산

마이크로컨트롤러에서 가장 빈번하게 발생하는 연산 지연(Latency) 중 하나가 큐의 인덱스를 계산하는 과정이다. 일반적인 원형 큐 구현에서는 다음과 같이 모듈로 분배를 사용한다.

Pointer_{next} = (Pointer_{current} + 1) \bmod L
(단, L은 큐의 길이 표본값)

그러나 나눗셈 및 나머지(Modulo) 연산은 ARM Cortex-M 시리즈에서 클럭 사이클 소모가 크기 때문에, PX4 시스템 개발진은 큐의 크기(L)를 반드시 **2의 거듭제곱(Power of Two, 예: 2, 4, 8, 16 …)**으로 설정하도록 강제(Constraint)하여 비트 마스킹(Bit Masking) 기법으로 최적화한다.

Pointer_{next} = (Pointer_{current} + 1) \mathbin{\&} (L - 1)

이 최적화 스케줄링 방식을 통해, TRANSIENT_LOCAL과 KEEP_LAST QoS가 적용된 데이터 링크에서 새로운 ROS 2 노드가 연결되더라도, 시스템 오버헤드 없이 즉각적으로 과거의 N개 이력 데이터를 원형 큐에서 추출하여 일괄 전송(Burst Transmission)할 수 있는 안정성을 확보하게 된다.
