62.2.2.1 추상 구문 트리(Abstract Syntax Tree) 기반의 C/C++ 및 Python 언어 바인딩(Language Binding) 템플릿 인스턴스화

PX4-Autopilot과 ROS 2 간의 인터페이스 정의 언어(IDL) 분석의 1차 결과물은 단순한 텍스트 파싱을 넘어선 **추상 구문 트리(Abstract Syntax Tree, AST)**이다. 드론 제어 시스템의 이기종(Heterogeneous) 환경—임베디드 비행 제어기(C/C++)와 상위 레벨의 임무 컴퓨터(Python/C++)—이 안전하게 협력하기 위해서는 이 동일한 트리를 바탕으로 각 프로그래밍 언어의 문법과 메모리 제약에 맞춘 네이티브 코드(Native Code)가 파생되어야 한다. 본 절에서는 AST 구성 원리와, 이를 다양한 언어로 변환(Language Binding)하기 위해 사용되는 템플릿 인스턴스화(Template Instantiation) 기법을 다룬다.

1. 추상 구문 트리(Abstract Syntax Tree, AST) 모델링

앞서 .msg 파일에서 어휘와 토큰이 추출되면, 빌드 시스템 내장 컴파일 스크립트(주로 rosidl_parser 및 PX4 내부 파서)는 이를 논리적 계층 구조인 AST로 변환한다.

1.1 AST의 자료 구조

하나의 메시지 파일은 트리 최상단의 **Message Node(메시지 노드)**로 정의되며, 그 아래에 여러 개의 **Field Node(필드 노드)**와 **Constant Node(상수 노드)**를 자식으로 갖는다.
각 Field Node는 다시 데이터의 속성을 저장하는 메타 노드를 가진다.

[Message Node] : VehicleCommand
 ├── [Field Node]
 │    ├── Name: command
 │    ├── Type: uint32
 │    └── Constraints: None
 ├── [Constant Node]
 │    ├── Name: VEHICLE_CMD_DO_REPOSITION
 │    ├── Type: uint32
 │    └── Value: 192
 └── [Field Node]
      ├── Name: param1
      ├── Type: float32
      └── Constraints: None

이 AST 자료 구조는 컴파일 타임(Compile-time)의 메모리상에 Python 객체(Object)의 모음으로 적재되며, 이후 구동될 다양한 코드 생성기의 범용적인 입력값(Single Source of Truth)으로 철저하게 활용된다.

2. 언어 바인딩(Language Binding)과 템플릿 인스턴스화

AST가 구성되면 이를 각 타겟 프로그래밍 언어로 번역(Binding)해야 한다. C++은 구조체와 포인터 개념을 사용하고, Python은 클래스(Class)와 동적 타이핑 메커니즘을 사용하므로 번역 규칙이 완전히 다르다. 이 간극을 메우기 위해 템플릿 엔진(Jinja2 또는 EmPy)이 사용된다.

2.1 템플릿 엔진(Template Engine) 구동 원리

템플릿 철학의 핵심은 “로직이 배제된 채 구멍(Hole)만 뚫려 있는 틀(Template)에 AST의 객체 데이터들을 채워 넣는(Instantiation) 것“이다.

아래는 C/C++ 구조체를 생성하기 위한 Jinja2 템플릿(*.h.em 혹은 *.h.in)의 가상화된 예시이다.

// C++ Header Template Example
struct @(message.name)_s {
@[for field in message.fields]@
    @(field.type.cpp_type) @(field.name);
@[end for]@
};

위 템플릿 엔진에 VehicleCommand AST가 주입되면, 반복 구문(for field in message.fields)이 트리의 자식 노드들을 순회(Traversal)하며 다음과 같은 C++ 코드를 인스턴스화(Instantiate)해 낸다.

// Instantiated C++ Header
struct VehicleCommand_s {
    uint32_t command;
    float param1;
    // ...
};

2.2 이기종 언어 간 바인딩 정밀도 확보

파이썬(Python) 노드를 위해 바인딩할 때도 구조는 동일하다. 파이썬용 템플릿은 AST를 순회하며 __slots__ 나 타입 힌트(Type Hinting)가 포함된 데이터클래스(Dataclass) 형태로 코드를 생성한다.

# Instantiated Python Binding Example
class VehicleCommand:
    __slots__ = ['command', 'param1']
    
    def __init__(self, command: int = 0, param1: float = 0.0):
        self.command = command
        self.param1 = param1

3. 템플릿 인스턴스화의 아키텍처적 이점

이러한 AST 기반의 템플릿 언어 바인딩 아키텍처는 PX4 프레임워크와 ROS 2 브리지에 다음과 같은 결정론적 안전성을 부여한다.

  1. 포인터와 오프셋의 완벽한 일치: C/C++(PX4)과 Python(ROS 2) 애플리케이션이 동일한 객체의 크기와 바이트 오프셋(Byte Offset) 순서를 동일하게 취급하게 된다. 이는 미들웨어를 통해 메모리 덤프가 직렬화되어 넘어갈 때 데이터 형식 붕괴(Corrupted Data Frame) 현상을 원천 차단한다.
  2. 언어 확장성(Scalability): 만약 차후 드론 제어 시스템을 Rust나 Go 언어로 작성하길 원한다면, 파서나 AST를 수정할 필요 없이 오직 Rust용 Jinja 템플릿 파일 하나만 추가하면 모든 기존 .msg 파이프라인과 완벽히 바인딩된다.
  3. 유지보수의 중앙집중화: 새로운 제어 파라미터 param2가 필요할 때, C++, C, Python 소스 코드를 각각 수정할 필요가 없다. 오직 .msg 파일의 텍스트 한 줄만 지시하면, AST 갱신과 템플릿 주입을 통해 빌드 파이프라인이 수십 개의 연관 코드를 자동으로 일괄 갱신(Overhaul)한다.

이처럼 추상화된 트리 모델과 템플릿 주입 기법의 결합은 드론이라는 생명 유지 크리티컬(Safety-Critical) 하드웨어 환경에서, 인간의 반복적인 코딩 실수를 배제하고 컴파일러 수준의 이기종 언어 호환성을 담보하는 최고의 소프트웨어 공학적 설계이다.
