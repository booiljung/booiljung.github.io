62.1.3 공통 데이터 표현(Common Data Representation) 규격 기반 메모리 직렬화(Memory Serialization) 및 역직렬화 메커니즘

서로 다른 하드웨어 아키텍처와 운영체제를 사용하는 시스템들이 통신하기 위해서는 메모리 상의 데이터 구조체(Struct)를 네트워크로 전송 가능한 일차원 바이트 배열로 변환하는 직렬화(Serialization) 과정과, 수신한 바이트 배열을 다시 원래의 구조체로 복원하는 역직렬화(Deserialization) 과정이 필수적이다. PX4-Autopilot과 ROS 2 간의 Micro XRCE-DDS 통신에서는 OMG(Object Management Group)의 국제 규격인 공통 데이터 표현(Common Data Representation, CDR) 포맷을 채택하여 이 역할을 수행한다. 본 절에서는 CDR 규격을 기반으로 하는 직렬화 및 역직렬화의 작동 메커니즘과, 자원이 제약된 PX4 시스템에서의 메모리 최적화 처리 기법을 심층적으로 분석한다.

1. MAVLink 직렬화와의 차이점 및 CDR의 도입 배경

기존 PX4에서 지상관제소(GCS)와의 통신에 널리 쓰이는 MAVLink 프로토콜은 대역폭을 절약하기 위해 페이로드를 극도로 압축하고 고정된 타입 순서로 패킹(Packing)하는 독자적 방식을 사용한다. 반면, ROS 2의 기반이 되는 DDS 미들웨어는 다양한 이기종 플랫폼(Windows, Linux, RTOS 등)에서의 범용성과 확장성을 중시하여 CDR이라는 표준화된 이진(Binary) 데이터 직렬화 형식을 사용한다.
Micro XRCE-DDS가 PX4에 도입됨에 따라, PX4 내부의 uORB 메시지는 ROS 2가 즉각 파싱할 수 있는 CDR 형태로 직렬화되어 전송되어야 한다. 이를 위해 PX4 펌웨어 빌드 시 microCDR이라는 고속 경량 변환 라이브러리가 함께 컴파일된다.

2. CDR 직렬화 메모리 레이아웃 및 정렬(Alignment) 규칙

CDR 규격의 가장 큰 특징 중 하나는 데이터의 메모리 접근 속도(Memory Access Speed)를 극대화하기 위해 자연 정렬(Natural Alignment) 원칙을 따른다는 점이다. 이는 바이트 배열 내부에서 각 데이터 타입이 자신의 크기의 배수가 되는 오프셋(Offset) 위치에 저장되도록 강제하는 규칙이다.

  • 1바이트 char나 int8_t: 모든 위치에 정렬 가능
  • 2바이트 int16_t: 짝수(2의 배수) 인덱스에 위치해야 함
  • 4바이트 int32_t나 float: 4의 배수 인덱스에 위치해야 함
  • 8바이트 int64_t나 double: 8의 배수 인덱스에 위치해야 함

정렬 규칙을 맞추기 위해 실제 데이터와 데이터 사이에 의미 없는 빈 공간(Padding)이 삽입될 수 있다. 직렬화 과정에서 CPU는 단순히 메모리 블록을 복사(memcpy)할 수 있으므로, 복잡한 비트 시프트(Bit-shift) 연산이 제거되어 비행 제어기(Flight Controller)의 연산 부하가 대폭 감소한다.

3. 메모리 직렬화 및 역직렬화 수행 과정

PX4에서 uORB 메시지를 ROS 2로 퍼블리싱(Publishing)할 때와 ROS 2에서 보낸 메시지를 서브스크라이빙(Subscribing)할 때 각각 직렬화와 역직렬화가 발생한다.

3.1 직렬화(Serialization) 메커니즘

uxrce_dds_client는 특정 uORB 토픽(*.msg 파일 기반)에 새로운 데이터가 발행되는지 수신 대기(Listening)한다.

  1. 데이터 복사: uORB 메시지 인스턴스(구조체)의 값을 읽어온다.
  2. microCDR 직렬화: 자동 생성된 직렬화 함수를 호출한다. 이 함수는 사전에 할당된 전송용 스태틱 버퍼(Static Buffer) 포인터에 차례대로 데이터 타입의 크기와 CDR 정렬 규칙에 맞추어 변수들을 집어넣으며 마이크로 버퍼(microBuffer) 오프셋을 전진시킨다.
  3. 메시지 전송: 직렬화가 완료된 버퍼가 전송 계층으로 넘겨진다.

3.2 역직렬화(Deserialization) 메커니즘

역직렬화는 메모리의 바이너리 배열을 의미 있는 구조체로 다시 캐스팅하는 과정이다.

  1. 안전성 검증: 전송 계층으로부터 페이로드가 수신되면, 데이터의 실질 크기와 조립 상태가 완전한지 검증한다.
  2. 마이크로 버퍼 초기화: 수신된 배열 포인터를 microCDR의 리더(Reader) 구조체에 연결한다.
  3. 데이터 추출: 송신 시와 동일한 순서 및 정렬 규칙을 적용하여 버퍼의 메모리를 읽어 PX4의 uORB C++ 구조체의 멤버 변수들에 각각 할당한다. 타입 오류 혹은 배열 경계 초과(Out-of-bounds)가 발생하지 않도록 강형(Strongly-typed) 파싱을 엄격히 수행한다.
  4. uORB 퍼블리싱: 값이 정상적으로 산출된 구조체를 PX4 내부망에 발행한다.

4. 제약 환경 최적화를 위한 튜닝 전략

Micro XRCE-DDS는 이러한 구조 덕분에 직렬화/역직렬화 타임을 O(1)에 가깝게 유지할 수 있다. 그러나 이진 패딩에 의해 MAVLink보다 전반적인 페이로드 크기가 비대해질 수 있다는 단점이 있다. 따라서 시스템 자원을 최적화하기 위해 다음과 같은 프로그래밍 기법이 요구된다.

  • 배열(Array)의 선형 복사 최적화: 큰 비전 데이터나 궤적(Trajectory) 데이터를 담을 때는 배열 내부 데이터가 동일한 데이터 타입일 경우 반복문을 사용하지 않고 memcpy 계열의 블록 단위 복사를 수행하도록 컴파일러 최적화를 유도해야 한다.
  • Zero-Copy 기법 설계 도모: RTOS 상에서 네트워크 스택에서 수신 버퍼로 넘어온 데이터를 또 다른 직렬화/역직렬화 전용 버퍼로 복사하는 오버헤드를 최대한 줄여야 한다. 따라서 메모리 풀(Memory Pool) 기법과 결합하여 송수신 버퍼를 uORB 데이터 포인터 영역과 물리적으로 가깝게 연동시키는 방향으로 아키텍처를 설계해야 한다.

5. 결론

공통 데이터 표현(CDR) 규격 기반의 메모리 직렬화와 역직렬화 메커니즘은 구조가 상이한 임베디드 레벨의 PX4 운영체제와 ROS 2 간의 자료 교환을 유연하고 매우 빠르게 만들어 준다. 데이터 타입의 메모리 정렬을 강제하여 CPU 연산 사이클을 비약적으로 아끼는 CDR의 철학은, 처리 성능에 한계가 뚜렷한 무인항공기 하드웨어 환경에서 ROS 2 미들웨어를 실용적으로 운영할 수 있게 하는 훌륭한 해답을 제공한다.
