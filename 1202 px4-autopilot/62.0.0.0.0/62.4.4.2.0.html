62.4.4.2 단정밀도 부동소수점(Single-precision Floating-point) 규격 기반의 매개변수 페이로드(Parameter Payload) 정밀도 손실 방지 할당 규칙

PX4-Autopilot 아키텍처 내부 및 ROS 2 통신 브릿지(Micro XRCE-DDS)를 관통하는 비동기 명령 포맷인 VehicleCommand 메시지는, 7개의 데이터 페이로드를 IEEE 754 규격의 단정밀도 부동소수점(float32, param1 ~ param7) 변수로 저장한다. 본 절에서는 이러한 32비트 메모리 할당 구조가 야기하는 정밀도 손실(Precision Loss) 메커니즘을 분석하고, 이를 방지하기 위한 소프트웨어 공학적 할당 규칙을 제시한다.

1. IEEE 754 단정밀도 부동소수점의 한계

단정밀도 부동소수점 스칼라 값 구조는 1비트의 부호(Sign), 8비트의 지수(Exponent), 그리고 23비트의 가수(Fraction/Mantissa)로 구성된다. 이로 인해 보장되는 십진 유효숫자는 약 7.2자리에 불과하다.

만약 전 지구적 좌표계(WGS84)의 위도(Latitude)나 경도(Longitude)와 같은 큰 스케일의 스칼라를 10진수 도(Degree) 단위로 param5 등에 직접 캐스팅하여 할당할 경우, 센티미터 단위의 물리적 위치 제어를 수행하는 무인항공기에는 치명적인 절대 공간 양자화(Spatial Quantization) 및 절사 오차(Truncation Error)가 발생한다. 이는 정밀 착륙이나 국소 회피 기동 시 예기치 않은 동작 궤적을 산출하는 원인이 된다.

2. 해결 및 할당 규칙 (Allocation Rules)

PX4 펌웨어 버전 1.14 이상의 환경에서 오프보드 제어 컴포넌트를 설계할 때, 개발자는 다음의 페이로드 할당 규칙을 준수해야 한다.

2.1 규칙 1: 고정밀 데이터의 스칼라 우회(Bypass)

절대 좌표 필드나 마이크로초(Microsecond) 단위의 타임스탬프와 같이 높은 유효숫자를 요구하는 데이터는 결코 단정밀도 매개변수 필드(param1 ~ 7)에 할당해선 안 된다. 위치 지정이 필요한 임무(Mission) 명령이나 타겟 지정의 경우, COMMAND_LONG 대신 높은 해상도의 전용 uORB 테마(예: TrajectoryWaypoint)를 이용하거나, 기준점(Local Origin) 대비 상대 이동 거리(X, Y, Z 미터 스케일)를 사용하여 지수부의 희생을 방지해라.

2.2 규칙 2: 미사용 인터페이스에 대한 NaN 초기화 (NaN Initialization)

QGroundControl과 같은 지상 관제 시스템(GCS) 컴포넌트는 모든 파라미터가 사용되지 않는 명령을 하달할 때, 반드시 비활성 필드를 NaN (Not a Number)으로 초기화한다. 0.0f로 초기화할 경우 PX4 커맨더(Commander) 모듈 내 상태 기계가 이를 유효한 제어 변수(예: 자이로스코프 영점, 고도 0미터 설정)로 오인할 수 있다.

3. MAVLink 데이터 패킷과의 비교 (Ardupilot 및 GCS 관제 차이)

전통적인 Ardupilot이나 초기 PX4는 MAVLink의 COMMAND_LONG 메시지를 광범위하게 사용했다. 하지만 단정밀도 좌표계 표현의 한계를 극복하기 위해 MAVLink 프로토콜은 COMMAND_INT 메시지를 도입하였다. COMMAND_INT는 위치 좌표(x, y)를 int32_t형 (1e7 스케일링)으로 전송하여 센티미터 이하의 정밀도를 보존한다.

하지만 ROS 2 환경에 구축된 VehicleCommand uORB 메시지는 이 COMMAND_INT의 속성 또한 param5 및 param6을 통하거나, 별도의 내부 변환 레이어를 통해 처리하게 된다. 따라서 ROS 2를 활용해 GCS 애플리케이션이나 자율 비행 인공지능을 구현하는 퍼블리셔(Publisher) 엔드포인트는 페이로드 인코딩 과정에서 이러한 형 변환(Type Conversion) 정책을 면밀히 숙지하고 있어야 한다.

5. C++ 구현 시 정밀도 보존 및 NaN 연산 예제 (Code Example)

ROS 2(C++14 이상)에서 px4_msgs::msg::VehicleCommand를 안전하게 생성하고 NaN 할당 규칙을 준수하는 방법은 다음과 같다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_command.hpp>
#include <cmath> // std::isnan, NAN

class PrecisionSafeCommandNode : public rclcpp::Node {
private:
    rclcpp::Publisher<px4_msgs::msg::VehicleCommand>::SharedPtr cmd_pub_;

public:
    PrecisionSafeCommandNode() : Node("precision_safe_command_node") {
        cmd_pub_ = this->create_publisher<px4_msgs::msg::VehicleCommand>("/fmu/in/vehicle_command", 10);
    }

    void publish_gimbal_set_pitch_yaw(float pitch_deg, float yaw_deg) {
        px4_msgs::msg::VehicleCommand msg{};
        msg.command = px4_msgs::msg::VehicleCommand::VEHICLE_CMD_DO_MOUNT_CONTROL;

        // 타겟 시스템 및 컴포넌트 할당
        msg.target_system = 1;
        msg.target_component = 1;      // Payload/Gimbal component ID
        msg.source_system = 255;
        msg.source_component = 0;

        // 규칙 1: 정밀도가 허용하는 범위(각도 등) 내에서의 스칼라 대입
        msg.param1 = pitch_deg;        // Pitch
        msg.param2 = 0.0f;             // Roll (사용함)
        msg.param3 = yaw_deg;          // Yaw

        // 규칙 2: 사용하지 않는 페이로드 필드는 반드시 IEEE 754 NaN 특수값으로 할당
        msg.param4 = NAN;
        msg.param5 = NAN;
        msg.param6 = NAN;
        msg.param7 = NAN;

        msg.timestamp = this->now().nanoseconds() / 1000ULL;

        cmd_pub_->publish(msg);
        RCLCPP_INFO(this->get_logger(), "Published command with safe precision and NaN boundaries.");
    }
};

위의 예제는 짐벌 제어(Gimbal Control) 명령을 내리는 전형적인 코드이다. param4부터 param7까지는 로직 수행과 무관하므로, 0.0f 등을 할당해 오역합착(Mismatch)을 유발하는 대신 NAN 매크로 상수를 직접 주입하여 모듈이 명시적으로 필드를 묵살(Ignore)하도록 설계하였다.

6. 결론

비동기 메시지 기반의 로보틱스 시스템에 있어 단정밀도 부동소수점의 남용은 위치 표류(Position Drift)와 같은 제어 불안정의 주요 원인이다. 그러므로 자율 에이전트 개발자는 변수 구조체의 물리적 메모리 속성을 이해하고, NaN 초기화 준칙 등을 강제함으로써 PX4 내장 컴포넌트와의 데이터 프로토콜 불일치(Data Protocol Discrepancy)를 원천적으로 제거해야 한다.
