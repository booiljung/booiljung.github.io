62.6.4 서비스 품질(QoS) 호환성 검증 알고리즘 불일치에 따른 요청 거부(Request Denied) 예외 발생 시 스택 트레이스(Stack Trace) 분석 기법

PX4-Autopilot(펌웨어 버전 1.14 이상)과 ROS 2(Micro XRCE-DDS)가 구성하는 분산 비행 제어망에서, 퍼블리셔(Publisher)와 서브스크라이버(Subscriber) 간의 토픽 이름과 데이터 형식(IDL)이 완벽히 일치하더라도 통신이 단절되는 가장 까다로운 원인은 서비스 품질(Quality of Service, QoS) 프로파일의 호환성 매칭 실패이다. 본 절에서는 DDS 엔드포인트 간의 QoS 불일치로 인해 발생하는 ‘요청 거부(Requested Incompatible QoS)’ 예외 현상을 규명하고, 컴패니언 컴퓨터 상에서 이를 스택 트레이스(Stack Trace) 수준으로 추적하는 C++ 분석 기법을 다룬다.

1. 아키텍처 상의 QoS 호환성 매트릭스 (Compatibility Matrix)

DDS 표준에 기반한 ROS 2 미들웨어는 각 데이터 스트림의 성격에 맞추어 네트워크 신뢰도를 조절하는 멀티플렉스 QoS 구조를 갖는다. PX4 펌웨어의 dds_topics.yaml 설정과 ROS 2 노드의 C++ 소스 코드는 다음의 핵심 프로파일을 공유하며, 수학적 호환성 매트릭스를 반드시 통과해야만 브릿지 연결이 승인된다.

  • 신뢰성(Reliability):
  • 송신자가 Best Effort(데이터 유실 허용)로 발송할 때, 수신자가 Reliable(수신 확인 및 재전송 요구)을 강제하면 **호환성 실패(Incompatible)**가 발생한다. 반대로 송신자 Reliable, 수신자 Best Effort는 호환된다.
  • 영속성(Durability):
  • 송신자가 Volatile(과거 데이터 즉시 파기)로 설정했을 때, 수신자가 Transient Local(늦게 연결된 구독자에게도 최신 과거 데이터 제공)을 요구하면 호환성 실패가 발생한다.

2. Ardupilot 파라미터 NACK와 ROS 2 QoS 예외의 차이 비교

QGroundControl과 기체 간 MAVLink를 통한 Ardupilot 스택 제어 시에는 잘못된 요청(예: 범위를 벗어난 비행 모드 변경 명령)이 접수되면 비행 제어기는 가벼운 구조체인 COMMAND_ACK 패킷 내의 결과 코드(예: MAV_RESULT_DENIED)를 발행하여 즉각적이고 수동적인 응답을 거쳤다.

이에 반해, ROS 2 환경의 QoS 매칭 검증은 RMW(ROS Middleware) 및 RTPS 발견(Discovery) 단계에서 자율적으로 집행된다. 즉, 프로그래머가 명시적으로 예외 핸들러를 등록해두지 않으면 매칭 실패 오류는 터미널 로그를 거치지 않고 **조용히 드롭(Silent Drop)**되며 시스템은 무한 대기 상태에 빠지게 된다. 이는 마이크로 서비스 아키텍처 특유의 투명성이 주는 부작용이므로, 엄격한 콜백 추적(Callback Tracing) 메커니즘 설계가 필수적이다.

3. 스택 트레이스 및 콜백 이벤트 후킹 아키텍처

ROS 2 C++ 클라이언트 라이브러리(rclcpp)는 퍼블리셔 및 서브스크라이버 객체 생성 시, QoS 호환성이 깨졌을 때 콜백을 발생시킬 수 있는 SubscriptionEventCallbacks 구조체를 지원한다.

네트워크 엔지니어는 이 이벤트 핸들러를 훅(Hook)하여, 단순히 ’통신 실패’라는 표면적 징후를 넘어, **어떤 QoS 정책(Reliability 혹은 Durability 여부)**이 퍼블리셔 측과 상충하여 예외가 던져졌는지를 명확히 분리(Isolation)하는 스택 트레이스를 콘솔에 출력할 수 있다.

5. C++ 이벤트 핸들러 기반 스택 트레이스 예제 (Implementation Example)

다음은 컴패니언 컴퓨터 측에서 PX4의 VehicleStatus 토픽을 서브스크라이브 할 때, QoS 불일치 이벤트를 가로채어 원인을 추적하는 ROS 2 C++14 예제이다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_status.hpp>

class QoSDiagnosticNode : public rclcpp::Node {
private:
    rclcpp::Subscription<px4_msgs::msg::VehicleStatus>::SharedPtr sub_;

    // 상태 토픽 수신 콜백
    void status_callback(const px4_msgs::msg::VehicleStatus::SharedPtr msg) {
        RCLCPP_INFO_ONCE(this->get_logger(), "Successful Connection & QoS Matched.");
    }

    // QoS 호환성 충돌 예외 발생 시 콜백
    void incompatible_qos_callback(rclcpp::QOSRequestedIncompatibleQoSInfo& info) {
        RCLCPP_ERROR(this->get_logger(), "CRITICAL FAULT: QoS Incompatibility Detected!");
        
        switch (info.last_policy_kind) {
            case rmw_qos_policy_kind_t::RMW_QOS_POLICY_RELIABILITY:
                RCLCPP_ERROR(this->get_logger(), "-> Conflict Root Cause: Reliability Mismatch.");
                RCLCPP_ERROR(this->get_logger(), "-> Check if PX4 dds_topics.yaml is Best Effort while ROS 2 is Reliable.");
                break;
            case rmw_qos_policy_kind_t::RMW_QOS_POLICY_DURABILITY:
                RCLCPP_ERROR(this->get_logger(), "-> Conflict Root Cause: Durability Mismatch.");
                break;
            case rmw_qos_policy_kind_t::RMW_QOS_POLICY_DEADLINE:
                RCLCPP_ERROR(this->get_logger(), "-> Conflict Root Cause: Deadline Mismatch.");
                break;
            default:
                RCLCPP_ERROR(this->get_logger(), "-> Conflict Root Cause: Unknown Policy (%d).", info.last_policy_kind);
                break;
        }
        RCLCPP_ERROR(this->get_logger(), "Total mismatched policies count: %d", info.total_count);
    }

public:
    QoSDiagnosticNode() : Node("qos_diagnostic_node") {
        // 의도적으로 잘못된 QoS를 요구하여 테스트 (PX4는 기본 Best Effort로 전송함)
        rclcpp::QoS bad_qos_profile = rclcpp::QoS(10).reliable();

        // 콜백 옵션 객체 생성 및 이벤트 핸들러 바인딩
        rclcpp::SubscriptionOptions sub_options;
        sub_options.event_callbacks.incompatible_qos_callback = 
            std::bind(&QoSDiagnosticNode::incompatible_qos_callback, this, std::placeholders::_1);

        sub_ = this->create_subscription<px4_msgs::msg::VehicleStatus>(
            "/fmu/out/vehicle_status", 
            bad_qos_profile,
            std::bind(&QoSDiagnosticNode::status_callback, this, std::placeholders::_1),
            sub_options
        );
    }
};

int main(int argc, char * argv[]) {
    rclcpp::init(argc, argv);
    rclcpp::spin(std::make_shared<QoSDiagnosticNode>());
    rclcpp::shutdown();
    return 0;
}

6. 결론

“어디서 지연되는가“를 확인하기 전에 “연결이 자격 증명을 마쳤는가“를 확인하는 역량은, 분산 실시간 로봇 운영체제를 다루는 개발자의 필수 덕목이다. QoS 호환성 예방은 컴파일러가 잡아주지 않는 분산 런타임의 맹점이다. dds_topics.yaml을 수정할 때마다 상반되는 파이프라인의 C++ 소스 코드 프로파일을 대조해야 하며, 프로덕션 베이스의 자율 비행 노드라면 반드시 위와 같은 인컴패터블(Incompatible) 콜백 추적기를 삽입하여 트러블슈팅의 첫 번째 방벽을 구성해야 한다.
