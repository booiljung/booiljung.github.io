62.6.3 중앙 처리 장치(CPU) 스케줄링 지연(Scheduling Latency) 및 네트워크 지터(Jitter)의 확률 밀도 함수(Probability Density Function) 분석

PX4-Autopilot(펌웨어 버전 1.14 이상)과 다중 코어 컴패니언 컴퓨터 간의 ROS 2(Micro XRCE-DDS) 통신망은 엄격한 경성 실시간(Hard Real-Time) 요구사항을 지닌다. 그러나 물리적 인터페이스 버퍼링, 미들웨어 스레딩, 운영체제의 컨텍스트 스위칭(Context Switching)이 결합되면서 데이터 도착 시간은 결코 일정하지 않게 된다. 본 절에서는 이러한 비결정적 타이밍 변동성인 **지터(Jitter)**를 확률 밀도 함수(Probability Density Function, PDF) 모델링을 통해 분석하고, CPU 스케줄링 레이턴시가 제어 대역폭에 미치는 영향을 규명한다.

1. 지터(Jitter)의 아키텍처적 유발 요인

오프보드(Offboard) 모드에서 100Hz(10ms 주기)로 TrajectorySetpoint를 퍼블리시하더라도, PX4의 위치 제어기(Position Controller)가 정확히 10ms마다 명령을 수신한다는 보장은 없다. 지터의 주요 원천은 다음과 같다.

  1. 미들웨어 스레딩 지연(Middleware Threading Latency): 컴패니언 컴퓨터의 리눅스(Linux) OS, 혹은 Fast DDS의 내부 수신/송신 비동기 워커(Worker) 스레드가 스케줄러에 의해 선점(Preempted)당하는 현상.
  2. 네트워크 버퍼 지연(Network Buffering Delay): 이더넷 스위치나 시리얼 포트(UART)의 하드웨어 FIFO 버퍼가 가득 차면서 발생하는 큐잉 타임(Queueing Time).
  3. NuttX RTOS 컨텍스트 스위칭(Context Switching): 픽스호크 내부에서 microdds_client 태스크가 수신 인터럽트(RX Interrupt)를 받아 uORB로 데이터를 역직렬화하는 과정에서 우선순위 역전(Priority Inversion)이나 선점 회피에 의해 스케줄링이 밀리는 레이턴시.

2. Jitter의 통계적 분석: 확률 밀도 함수(PDF) 매핑

네트워크 지터는 이상적으로는 평균이 0인 좁은 **가우시안 분포(Gaussian Distribution)**를 따라야 한다. 즉, 대부분의 패킷 도착 주기가 \Delta T \approx 10\text{ms} 에 밀집해야 한다.
그러나 실제 프로덕션 시스템에서 지터의 확률 밀도 함수(PDF)는 우측으로 길게 늘어지는 롱테일(Long-tail) 분포를 띤다.

P(\Delta t) = \frac{1}{\sqrt{2\pi\sigma^2}} \exp\left( -\frac{(\Delta t - \mu)^2}{2\sigma^2} \right) + P_{anomaly}(\Delta t)

이 P_{anomaly}(\Delta t) 영역에 속하는, 즉 도착 주기가 15ms나 20ms 이상 벌어지는 아웃라이어(Outlier) 패킷들은 고대역폭 드론 제어 루프(ECL/EKF2)에서 위상 마진(Phase Margin)을 깎아먹고 앨리어싱(Aliasing)을 유발하여 진동(Oscillation)을 초래한다.

3. MAVLink 폴링(Polling) 대 DDS I/O 이벤트 타이밍 철학의 차이

기존 Ardupilot 기반 시스템 혹은 QGroundControl 관제에서는 GCS와 기체 간 MAVLink 통신이 대개 메인 태스크 루프 내의 단순한 타이머 기반 폴링(Polling) 주기로 송수신되었다. 이는 구조가 단순하여 P_{anomaly}가 직관적이고 상대적으로 예측 가능했다.

반면, 현대의 분산 ROS 2 프레임워크는 I/O 다중화(Event-driven Multiplexing)와 복잡한 QoS(Quality of Service) 재전송 스택을 포함한다. 따라서 Reliable QoS를 적용할 경우, 패킷 소실 시 재전송(Retransmission) 매커니즘이 발동하면서 롱테일 Jitter가 급격히 비대해진다. 무인항공기 오프보드 제어처럼 ’늦게 온 과거 데이터’가 가치가 없는 영역에서는, Jitter의 악영향을 최소화하기 위해 반드시 Best Effort QoS를 채택해야 하는 수학적 근거가 바로 이 확률 밀도 곡선의 꼬리(Tail) 형상에 있다.

5. 지터 트래킹 및 확률 분석을 위한 Python 스크립트 예제

단순한 평균 핑(Ping) 테스트로는 스케줄링 지연의 스파이크(Spike)를 잡아낼 수 없다. 다음은 rclpy를 활용하여 메시지 도착 인터벌(Interval)의 델타 타임(Delta Time)을 실시간으로 추적하고, Jitter의 PDF 롱테일 아노말리를 스크리닝하는 구현체이다.

import rclpy
from rclpy.node import Node
from px4_msgs.msg import VehicleOdometry
import time
import numpy as np

class JitterPDFAnalyzerNode(Node):
    def __init__(self):
        super().__init__('jitter_pdf_analyzer')
        self.create_subscription(VehicleOdometry, '/fmu/out/vehicle_odometry', self.odom_callback, 10)
        self.last_rx_time = time.time()
        self.target_interval = 0.01  # 목표 주기 100Hz -> 10ms
        
        self.jitter_history = []
        self.MAX_SAMPLES = 1000  # 통계 분석을 위한 샘플 윈도우

        self.timer = self.create_timer(2.0, self.report_statistics)

    def odom_callback(self, msg):
        current_time = time.time()
        delta_t = current_time - self.last_rx_time
        self.last_rx_time = current_time
        
        # Jitter: 기대 수행 간격과 실제 간격의 오차
        jitter = delta_t - self.target_interval
        self.jitter_history.append(jitter)
        
        if len(self.jitter_history) > self.MAX_SAMPLES:
            self.jitter_history.pop(0)

    def report_statistics(self):
        if len(self.jitter_history) < 100:
            return
            
        data = np.array(self.jitter_history) * 1000.0  # ms 단위 변환
        mean_jitter = np.mean(data)
        std_dev = np.std(data)
        
        # 롱테일 검출: 3-Sigma를 넘어서는 아웃라이어 비율 계산
        threshold = 3.0 * std_dev
        outliers = np.sum(np.abs(data - mean_jitter) > threshold)
        outlier_ratio = (outliers / len(data)) * 100.0
        
        self.get_logger().info(f"--- Jitter PDF Analysis (N={len(data)}) ---")
        self.get_logger().info(f"Mean Jitter: {mean_jitter:.3f} ms | StdDev (σ): {std_dev:.3f} ms")
        self.get_logger().info(f"Max Spike: {np.max(data):.3f} ms")
        self.get_logger().info(f"Long-tail Outliers (>3σ): {outliers} packets ({outlier_ratio:.2f}%)")
        
        if np.max(data) > 10.0:
            self.get_logger().error("CRITICAL: CPU Scheduling Anomaly or Severe Network Jitter detected!")

def main(args=None):
    rclpy.init(args=args)
    analyzer = JitterPDFAnalyzerNode()
    rclpy.spin(analyzer)
    analyzer.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()

6. 결론

실시간 제어 네트워크에서 평균 지연 시간이 낮다는 사실만으로는 결코 시스템 무결성을 보장할 수 없다. 분산 운영체제의 스케줄러와 미들웨어가 만들어내는 무작위적인 지터 통계 곡선(PDF)을 분석하고, 롱테일 아웃라이어가 비행기체 안정성(Stability)에 타격을 주기 전에 감지해 내는 런타임 분석 툴체인이 필수적이다. 개발자는 반드시 상술한 통계적 스니퍼를 가동하여, 자신의 ROS 2 노드 환경 조합이 ‘결정론적 한계 마진(Deterministic Threshold Margin)’ 내부에서 수렴하고 있는지를 상시 증명해야 한다.
