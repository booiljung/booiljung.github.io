62.3.3.2. 상태 전이 행렬(State Transition Matrix) 기반의 시스템 페일세이프(Failsafe) 트리거(Trigger) 조건 부울 대수(Boolean Algebra) 연산

최신 무인항공기(UAV) 운영 환경에서는 기체의 안전과 임무 성공을 보장하기 위해 고도로 복잡한 페일세이프(Failsafe) 논리가 요구된다. PX4-Autopilot의 안전 장치 시스템은 단순한 1차원적 센서 결함 감지를 넘어, 기체의 비행 모드, 배터리 잔량, 데이터 링크(Data Link) 단절 여부, GEOFENCE(지오펜스) 이탈 여부 등 다수의 플래그를 종합하여 가장 안전한 전이 상태(Transition State)를 결정한다.

본 섹션에서는 복합 결함 상황에서 시스템이 취해야 할 최적의 생존 동작(예: 복귀, 즉시 착륙, 비행 강제 종료)을 판별하기 위해, 상태 전이 행렬(State Transition Matrix)과 부울 대수(Boolean Algebra) 연산을 결합한 트리거 아키텍처를 분석한다.

1. 1차원 예외 처리를 넘어서: 복합 결함의 문제 공간

드론이 비행 중 겪게 되는 장애 요소는 단일하게 나타나지 않는 경향이 있다. 예를 들어, 라디오 컨트롤(RC) 조종기 신호가 단절됨과 동시에 배터리 전압이 경고 수준 이하(Critical)로 떨어질 수 있다. 이때 PX4의 시스템 커맨더(Commander)는 아래와 같은 복합적인 상황 판단을 수행해야 한다.

  • 조종기 단절 시 기본 페일세이프 동작은 “비행 지점으로 복귀(Return to Launch, RTL)“이다.
  • 하지만 배터리 전압이 임계치 이하일 때 RTL을 시도하면 복귀 도중 추락할 위험이 크므로 “즉시 착륙(Land)“으로 행동 지침이 수정되어야 한다.

이와 같이 여러 개의 이산 상태 플래그 값들이 중첩될 때, 결론(Action)을 명확하게 도출하기 위해 시스템은 부울 대수에 기반을 둔 상태 전이 행렬을 활용하게 된다.

2. 부울 대수(Boolean Algebra)를 이용한 상태 평가 및 트리거

온보드 탑재 컴퓨터의 ROS 2 환경 혹은 지상 관제 시스템(GCS)에서는 PX4로부터 수신되는 수십 개의 상태 플래그를 조합하여 외부 에이전트 관점의 2차 페일세이프 트리거를 설계할 수 있다.

  • RC_Lost: 조종기 신호 단절 (VehicleStatus.rc_signal_lost == true)
  • GPS_Lost: 측위 정보 신뢰성 상실 (VehicleLocalPosition.xy_valid == false)
  • Batt_Critical: 배터리 임계치 이하 (BatteryStatus.warning == BATTERY_WARNING_CRITICAL)

이러한 개별 평가 값들은 논리합(Logical OR, ||)과 논리곱(Logical AND, &&)으로 결합되어 최종 트리를 형성한다.

[ 예비 산술식 예시 ]

IF (RC_Lost AND NOT GPS_Lost) THEN Action = AUTO_RTL
IF (RC_Lost AND GPS_Lost) THEN Action = AUTO_LAND
IF (Batt_Critical OR (GEOFENCE_Breach AND NOT GPS_Lost)) THEN Action = AUTO_LAND
IF (Motor_Failure AND Altitude_Low) THEN Action = FLIGHT_TERMINATION

4. C++ 기반 외부 에이전트의 Failsafe 연산 계층 설계

자율 에이전트가 QGroundControl을 대체하거나 보조하여 기체의 상태를 관측하고, 펌웨어가 미처 감지하지 못하는 복합 시스템 장애(예: 페이로드 컴퓨터의 열폭주 등)를 부가하여 제어권을 행사하는 설계 예시이다.

#include <iostream>

// 시스템 검증을 위한 구조체 선언
struct SystemDiagnostics {
    bool rc_lost = false;
    bool gps_valid = true;
    bool battery_critical = false;
    bool geofence_breach = false;
    bool companion_cpu_overheated = false; // PX4 외부 요인
};

// 트리거 액션 열거형
enum class FailsafeAction {
    NONE,
    RETURN_TO_LAUNCH,
    EMERGENCY_LANDING,
    FLIGHT_TERMINATION
};

// 부울 대수 행렬 기반의 전이 상태 도출 함수
FailsafeAction evaluate_failsafe_algebra(const SystemDiagnostics& diag) {
    
    // 규칙 1: 치명적 장애 (복합 요인) -> 비행 강제 종료
    bool condition_terminate = (diag.battery_critical && diag.Companion_cpu_overheated) || 
                               (diag.rc_lost && !diag.gps_valid && diag.battery_critical);
    if (condition_terminate) { return FailsafeAction::FLIGHT_TERMINATION; }


    // 규칙 2: 심각한 장애 -> 즉시 착륙
    bool condition_land = diag.battery_critical || 
                          diag.geofence_breach || 
                          (diag.rc_lost && !diag.gps_valid);
    if (condition_land) { return FailsafeAction::EMERGENCY_LANDING; }


    // 규칙 3: 경미한 장애 -> 복귀
    bool condition_rtl = diag.rc_lost || diag.companion_cpu_overheated;
    if (condition_rtl) { return FailsafeAction::RETURN_TO_LAUNCH; }


    // 기본 상태
    return FailsafeAction::NONE;
}

void supervision_loop(const SystemDiagnostics& current_diags) {
    FailsafeAction required_action = evaluate_failsafe_algebra(current_diags);

    switch(required_action) {
        case FailsafeAction::RETURN_TO_LAUNCH:
            std::cout << "[Failsafe] 트리거 발동: RTL 복귀 명령 송신." << std::endl;
            break;
        case FailsafeAction::EMERGENCY_LANDING:
            std::cout << "[Failsafe] 트리거 발동: 긴급 LAND 명령 송신." << std::endl;
            break;
        case FailsafeAction::FLIGHT_TERMINATION:
            std::cout << "[CRITICAL] 트리거 발동: 비행 종료(Kill) 명령 송신." << std::endl;
            break;
        case FailsafeAction::NONE:
            // 정상 비행 유지
            break;
    }
}

위의 예제처럼, ROS 2 C++ 애플리케이션 내에 명시적인 부울 대수 판별 트리를 구성함으로써, PX4 펌웨어 자체의 안전판 외에도 임무 특정적(Mission-specific)인 고차원 페일세이프 아키텍처를 유연하게 구축할 수 있다. 상태 전이 행렬 기반의 설계는 조건의 확장과 검증 시스템(Unit Test) 작성에 절대적인 유지보수성을 제공한다.
