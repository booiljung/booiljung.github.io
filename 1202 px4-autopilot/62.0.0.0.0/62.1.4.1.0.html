62.1.4.1 전송 신뢰성(Reliability) 정책: 부정 응답(Negative Acknowledgment) 기반의 패킷 재전송 스케줄링 및 윈도우 크기(Window Size) 할당

로봇 제어 시스템에서 일부 데이터는 손실되어서는 안 되는 매우 높은 중요도를 지닌다. PX4-Autopilot과 ROS 2 간의 uXRCE-DDS 기반 통신에서 전송 신뢰성(Reliability) 정책은 메시지가 구독자(Subscriber)에게 반드시 도달해야 하는지, 혹은 가급적 전송하되 누락을 허용할 것인지를 결정한다. 본 절에서는 DDS의 RELIABLE 정책이 채택하는 부정 응답(Negative Acknowledgment, NACK) 기반의 패킷 재전송 스케줄링 메커니즘과, 제한된 임베디드 장치 환경에서 요구되는 윈도우 크기(Window Size) 할당 기법에 대해 상세히 분석한다.

1. 전송 신뢰성(Reliability) 정책 개요

DDS 표준 및 uXRCE-DDS 미들웨어는 데이터 전송 계층의 신뢰성을 제어하기 위해 크게 두 가지의 정책 프로파일을 제공한다.

  1. BEST_EFFORT (최선 노력): 발행자(Publisher)는 데이터를 한 번만 전송하며, 수신자의 확인 응답을 기다리지 않는다. 네트워크 지연이 가장 짧고 대역폭을 적게 차지하므로 sensor_accel, vehicle_odometry와 같이 높은 주파수로 스트리밍되는 센서 데이터 버스에 적합하다.
  2. RELIABLE (신뢰 보장): 수신자가 데이터를 받을 때까지 발행자가 데이터를 재전송한다. vehicle_command, offboard_control_mode와 같은 기체의 상태 천이 및 핵심 제어 지시명령을 송수신할 때 필수적으로 적용된다.

2. NACK 기반 패킷 재전송 스케줄링 메커니즘

RELIABLE QoS가 활성화된 통신에서, uXRCE-DDS는 전통적인 TCP 통신의 긍정 응답(ACK) 방식보다 오버헤드가 적은 부정 응답(NACK) 지향적 프로토콜을 혼합하여 사용한다. 이는 다수의 노드 시스템(Swarm 등)이나 불안정한 무선 텔레메트리 환경에 최적화된 기법이다.

2.1 하트비트(Heartbeat)와 NACK 상호작용

발행자는 데이터를 전송할 때마다 혹은 정기적으로 자신의 송신 이력을 알려주는 하트비트(Heartbeat) 메시지를 발송한다. 구독자는 이 하트비트를 수신한 후 연번(Sequence Number)을 대조하여 누락된 패킷이 있는 경우에만 **부정 응답(ACKNACK 메시지)**을 발행자에게 회신한다.

sequenceDiagram
    participant PX4 as PX4-Autopilot (Publisher)
    participant Agent as Micro XRCE-DDS Agent
    participant ROS2 as ROS 2 (Subscriber)

    PX4->>Agent: Send Message [Seq 1]
    Agent->>ROS2: Forward Message [Seq 1]
    Note right of ROS2: Received (No ACK sent yet)
    
    PX4--xC Agent: Send Message [Seq 2] (Packet Loss)
    
    PX4->>Agent: Send Heartbeat [Available: 1~2]
    Agent->>ROS2: Forward Heartbeat
    Note right of ROS2: Oh! I missed Seq 2
    
    ROS2->>Agent: Send ACKNACK [Missing: 2]
    Agent->>PX4: Forward ACKNACK
    
    PX4->>Agent: Retransmit Message [Seq 2]
    Agent->>ROS2: Forward Message [Seq 2]

2.2 NACK 억제 메커니즘 (NACK Suppression)

네트워크가 불안정하여 다수의 구독자가 동시에 패킷 손실을 경험할 경우, 모든 구독자가 일제히 ACKNACK을 발송하면 “NACK 폭풍(ACKNACK Storm)“이 발생하여 브리지(Bridge)의 대역폭이 붕괴될 수 있다. 이를 방지하기 위해 구독자는 패킷 누락을 인지하더라도 임의의 백오프 시간(Random Back-off Time)만큼 대기한 후 NACK을 발송하며, 대기 시간 중 다른 노드가 전송한 NACK이나 재전송된 동일 패킷을 엿듣게(Overhear) 되면 스스로 NACK 발송을 취소하는 억제 메커니즘을 지원한다(PX4 시퀀스 상 멀티캐스트 환경 설계 시 적용).

3. 윈도우 크기(Window Size) 할당과 메모리 한계 최적화

RELIABLE 통신을 논리적으로 완성하기 위해서는, 발행자가 수신된 사실을 확인(Acknowledged) 받을 때까지 이전에 발송한 메시지들을 버퍼(메모리)에 임시로 보관해야 한다. 이를 관리하는 공간의 단위가 바로 전송 윈도우 크기(Transmission Window Size)이다.

3.1 윈도우 캐시와 버퍼 고갈(Buffer Starvation)

  • 최대 메시지 이력: 버퍼에 보관할 수 있는 메시지의 최대 개수이다.
  • 발행자는 메시지를 발송할 때마다 윈도우의 가용 공간을 차지하게 된다.
  • NACK에 따른 재전송 처리를 완료하거나 정기적인 긍정 응답(ACK)을 수신하면 해당 메시지의 메모리를 해제(Free)하고 윈도우 슬롯을 확보한다.

하지만 PX4가 동작하는 STM32F4/STM32H7과 같은 마이크로컨트롤러 환경에서는 힙(Heap)과 스택(Stack) 사이즈가 수백 킬로바이트(KB) 수준으로 매우 엄격하게 제한된다. 만약 지상 관제 시스템(GCS)과의 연결 상태 불량으로 ACK가 지연된다면, 송신 윈도우 버퍼는 순식간에 차오르게 되고 결국 버퍼 고갈(Buffer Starvation) 상태에 빠지게 된다.

3.2 PX4 측의 실무적 대응 및 산출

버퍼 고갈 발생 시 PX4 내부의 uxrce_dds_client는 더 이상 ROS 2 측으로 메시지를 퍼블리싱하지 못하고 블로킹(Blocking)되거나 신규 메시지를 폐기(Drop)하게 된다.

이를 예방하기 위해서는 다음의 최적화 할당 원칙이 필요하다.

  1. 필수 데이터만 RELIABLE 지정: 센서 데이터에 RELIABLE을 적용하는 치명적인 실수를 피하라.
  2. 슬라이딩 윈도우(Sliding Window) 튜닝: 링크의 평균 왕복 지연 시간(Round Trip Time, RTT)과 제어 메시지 발행 주파수(f)를 바탕으로 최소 윈도우 크기(W)를 산출한다.
    W \ge \lceil RTT \times f \rceil + \alpha
    여기서 \alpha는 지터(Jitter) 처리를 위한 안전 여유분(Margin)이다.
  3. PX4 Kconfig 또는 CMake 환경 블록 내의 MTU(Maximum Transmission Unit) 설정과 동기화하여 청크 사이즈를 분배한다.

결론적으로, 부정 응답(NACK) 지향의 신뢰성 정책은 DDS 미들웨어의 패킷 지연을 획기적으로 낮춰 주지만, PX4의 제한된 시스템 리소스 하에서는 재전송 부하와 윈도우 버퍼 고갈을 고려하여 꼭 필요한 오프보드 지시 명령과 핵심 상태 값들에 한정적으로 배포해야만 하드 리얼타임 결정론을 확보할 수 있다.
