62.1.4 실시간 제어 시스템의 결정론적(Deterministic) 성능 보장을 위한 서비스 품질(Quality of Service) 정책 정밀 튜닝

무인항공기(UAV)와 같은 하드 리얼타임(Hard Real-time) 제어 시스템에서 시스템의 안정성과 신뢰성은 제어 루프의 결정론적(Deterministic) 지연 시간 보장에 직결된다. PX4-Autopilot은 내부적으로 NuttX RTOS 기반의 스케줄링을 통해 마이크로초(µs) 단위의 실시간성을 보장하지만, 이를 외부 시스템인 ROS 2 환경과 연동할 경우 네트워크 지연(Network Latency), 패킷 손실(Packet Loss), 대역폭(Bandwidth) 제한 등 비결정론적(Non-deterministic) 요소들이 개입하게 된다.

이러한 한계를 극복하기 위해 uXRCE-DDS(Micro XRCE-DDS) 미들웨어는 데이터 분산 서비스(Data Distribution Service, DDS) 표준에 정의된 서비스 품질(Quality of Service, QoS) 정책을 활용하여 네트워크 통신에서도 일정 수준의 결정론적 성능을 보장할 수 있는 메커니즘을 제공한다. 본 장에서는 PX4와 ROS 2 간의 연동 시, 각 센서 데이터와 제어 명령의 특성에 맞춰 QoS 프로파일을 정밀하게 튜닝하는 아키텍처 및 설정 기준을 분석한다. (적용 기준: PX4-Autopilot v1.14 이상, ROS 2 Humble 이상)

1. 결정론적 성능 보장과 QoS 메커니즘 개요

로봇 운영체제(ROS 2) 네트워크 토폴로지 내부 혹은 원격 지상 관제 시스템(QGroundControl 등)을 거쳐 전달되는 메시지는 항상 일정한 시간 안에 도착한다는 보장이 없다. 특히 비전 센서 기반의 오도메트리(VIO)나 다수의 액추에이터 제어 명령이 빈번하게 오가는 오프보드(Offboard) 모드 제어 상황에서, 메시지 전송의 신뢰도가 떨어지면 기체의 추락이나 경로 이탈과 같은 치명적인 결함(Fault)이 발생한다.

DDS 표준은 발행자(Publisher)와 구독자(Subscriber) 간의 데이터 전송 요구 수준을 협상할 수 있도록 다양한 QoS 정책 파라미터를 제공한다. 시스템 설계자는 데이터의 성격(예: 고주파수 센서 데이터 vs. 1회성 제어명령)에 맞추어 QoS를 조합함으로써 제한된 네트워크 자원을 극대화하고 실시간 처리 성능의 상한 및 하한 곡선을 예측할 수 있다.

1.1 주요 QoS 정책의 분류

실시간 제어 시스템에서 주로 튜닝해야 하는 핵심 제어 인자는 다음과 같다.

  1. 전송 신뢰성 (Reliability): 패킷 손실 시 재전송 여부 결정 (BEST_EFFORT vs RELIABLE)
  2. 데이터 이력 관리 (History): 수신하지 못한 데이터를 큐에 몇 개까지 보관할 것인지 설정 (KEEP_LAST vs KEEP_ALL)
  3. 데이터 내구성 (Durability): 늦게 연결된 구독자를 위해 데이터를 유지할지 여부 결정 (VOLATILE vs TRANSIENT_LOCAL)
  4. 마감 시간 (Deadline): 데이터가 지정된 시간 주기로 도착하지 않을 때 이벤트 트리거 여부 설정
  5. 활성 상태 (Liveliness): 시스템 간 연결 상태(Heartbeat) 감시 주기 및 주체 설정

이러한 정책들은 발행자와 구독자 간에 **호환성(Compatibility)**을 충족해야만 정상적인 통신 링크가 확립된다. 예를 들어, 발행자가 BEST_EFFORT로 전송하는데 구독자가 RELIABLE을 요구하는 경우 통신 연결이 거부(QoS Mismatch)된다.

2. PX4-ROS 2 브리지 간 QoS 프로파일 매핑 아키텍처

PX4 펌웨어 내의 uxrce_dds_client 모듈은 내부 uORB 메시지를 시리얼(UART), 이더넷(Ethernet) 등 물리적 인터페이스를 거쳐 ROS 2 미들웨어 환경의 MicroXRCEAgent로 전달한다. 이때 모든 uORB 토픽이 동일한 QoS 프로파일을 사용하는 것은 비효율적이므로, PX4는 uORB와 ROS 2의 브리지 역할을 담당하는 브리지 설정(Client-side Configuration)에서 토픽 단위로 QoS를 매핑(Mapping)한다.

위의 아키텍처 다이어그램에서 보듯, 시스템의 특성을 고려하여 각 데이터 버스 트래픽별로 차등화된 QoS 정책을 부여한다.

2.1 고주파 센서 서브시스템 매핑 전략

가속도계(sensor_accel)나 자이로스코프(sensor_gyro)와 같이 초당 수백 회(100Hz ~ 1kHz) 이상 퍼블리싱되는 센서 데이터의 경우, 지나간 과거의 데이터는 가치가 급격히 하락한다. 만약 단일 패킷이 유실되었다 하더라도 다음 주기의 패킷이 즉각 이를 대체할 수 있으므로, 재전송(Retransmission) 스케줄링이 유발하는 대역폭 낭비와 타이밍 지연(Stale Data Issue)을 방지하는 것이 매우 중요하다.

  • 권장 속성: Reliability = BEST_EFFORT, Durability = VOLATILE, History = KEEP_LAST (Depth=1)

2.2 중요 제어 및 상태 서브시스템 매핑 전략

반면 시동 여부를 결정하는 vehicle_command나 비행 모드 설정(offboard_control_mode)과 같은 메시지는 단 한 번이라도 누락될 경우 기체 제어 상실(Loss of Control)을 야기할 수 있다. 이러한 명령 및 시스템 상태 메시지는 약간의 지연이 발생하더라도 반드시 전달되어야 하며, 일시적인 네트워크 간섭에서도 강건하게 유지되어야 한다.

  • 권장 속성: Reliability = RELIABLE, Durability = TRANSIENT_LOCAL (또는 VOLATILE), History = KEEP_LAST (Depth=N)

3. 정밀 튜닝 시나리오와 트러블슈팅

QoS 정책의 정밀 튜닝은 단순한 옵션 설정 이상의 엔지니어링 역량을 요구한다. 제한된 임베디드 메모리(STM32의 SRAM 제약 등)에서 RELIABLE 연결을 유지하기 위해서는 발급된 메시지가 수신 확인(ACK)을 받을 때까지 버퍼(큐)에 상주해야 하므로, 윈도우 크기(Window Size) 할당에 주의를 기울여야 한다.

또한 무분별한 RELIABLE 정책 적용은 네트워크 트래픽 과부하 현상인 “메시지 폭풍(Message Storm)“을 촉발시킬 수 있다. 따라서 PX4 개발자는 실제 비행 환경의 데이터 링크 대역폭(예: 57600bps 텔레메트리 연동 시)을 분석하고, 시뮬레이터 벤치마킹 도구를 통해 종단 간 데이터 계층(End-to-End Data Layer)의 최대 처리량(Max Throughput) 및 지연율(Latency Profile)을 맵핑한 후 결정론적 상한선을 초과하지 않도록 QoS 파라미터를 최적화해야 한다.

이어지는 하위 절들에서는 이 핵심 QoS 정책들(신뢰성, 내구성, 이력, 마감시간, 활성상태)을 구체화하여, 이러한 제약사항들이 실제 PX4 C++ 소스 레벨과 ROS 2 메시징 아키텍처에 어떻게 적용되고 수학적으로 산출되는지 심층적으로 다룬다.
