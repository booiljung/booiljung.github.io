62.3.1. uORB(Micro Object Request Broker) 버스 인터페이스와 DDS 글로벌 데이터 공간 간의 토픽 해시(Topic Hash) 기반 매핑 알고리즘

PX4-Autopilot의 내부 통신 아키텍처인 uORB(Micro Object Request Broker)와 ROS 2의 기반이 되는 DDS(Data Distribution Service)는 본질적으로 상이한 통신 패러다임을 갖는다. uORB는 단일 주소 공간(단일 보드 컴퓨터 물리 메모리) 내에서 작동하는 경량화된 게시-구독(Publish-Subscribe) IPC(Inter-Process Communication)인 반면, DDS는 네트워크로 연결된 다수의 분산 노드 간의 통신을 관장하는 글로벌 데이터 공간(Global Data Space, GDS) 사양이다.

이 둘을 실시간으로 잇는 uXRCE-DDS 브리지에서는 매크로 오버헤드를 최소화하면서 토픽을 상호 매핑(Mapping)하는 고속 알고리즘이 필요하다. 본 절에서는 토픽의 이름 문자열(String) 대신 토픽 해시(Topic Hash) 를 사용하여 uORB와 DDS 글로벌 데이터 공간 간의 데이터를 연결하는 알고리즘적 메커니즘을 심층적으로 분석한다.

1. 문자열 식별(String Identification)의 한계와 해시(Hash)의 도입

DDS 사양에서 퍼블리셔(Publisher)와 서브스크라이버(Subscriber)는 토픽 이름(예: rt/fmu/out/vehicle_odometry)이라는 긴 문자열을 통해 서로를 식별하고 데이터를 교환한다. 그러나 자원이 극도로 제한된 마이크로컨트롤러(MCU) 상에서 구동되는 PX4 시스템이 매 메시지가 생성될 때마다 이 긴 문자열을 비교(String Matching)하거나 페이로드(Payload)에 담아 직렬화(Serialization)하여 전송한다면, 심각한 CPU 클록 낭비와 대역폭(Bandwidth) 고갈이 발생하게 된다.

이를 극복하기 위해 uXRCE-DDS 클라이언트(PX4 측)와 에이전트(ROS 2 측)는 초기 세션(Session) 생성 단계에서만 문자열 정보를 교환하고, 실시간 스트리밍 시에는 이 문자열을 짧은 정수형 식별자로 인코딩한 토픽 해시(Topic Hash) 만을 매핑 키(Mapping Key)로 사용한다.

2. 토픽 해시 매핑 알고리즘 파이프라인

해시(Hash) 기반의 매핑 파이프라인은 크게 두 단계—1단계: 정적 생성 및 메타데이터 교환, 2단계: 런타임 O(1) 해시 룩업(Lookup)—로 이루어진다.

    participant px4_client as uXRCE-DDS Client (PX4)
    participant ros_agent as uXRCE-DDS Agent (ROS 2)
    participant GDS as DDS Global Data Space

    Note over px4_client, ros_agent: 초기화(Initialization) 단계
    px4_client->>ros_agent: CREATE_TOPIC (String: "vehicle_odometry", ID: 0x1A2B)
    ros_agent->>GDS: Register "vehicle_odometry" in GDS
    ros_agent-->>px4_client: STATUS_OK

    Note over uORB, GDS: 실시간(Runtime) 스트리밍 단계
    uORB->>px4_client: Topic Data Published
    px4_client->>ros_agent: WRITE_DATA (Topic ID: 0x1A2B, Payload: [Binary])
    ros_agent->>GDS: Map 0x1A2B -> "vehicle_odometry" -> Publish

2.1 토픽 ID(Topic ID)의 해시 생성 로직

PX4 빌드 시스템 내에서 메시지(예: msg/VehicleOdometry.msg)가 파싱될 때, C/C++ 소스 코드가 생성됨과 동시에 각 uORB 토픽에는 16비트(16-bit) 혹은 32비트(32-bit) 크기의 고유 식별자인 topic_id가 부여된다. 이 식별자는 일반적으로 FNV-1a와 같은 빠르고 충돌(Collision) 확률이 낮은 해싱 알고리즘을 토픽 문자열에 적용하여 생성된다.

// [단순화된 FNV-1a 기반 해시 매핑 예시]
constexpr uint32_t val_32_const = 0x811c9dc5;
constexpr uint32_t prime_32_const = 0x1000193;

inline constexpr uint32_t hash_topic_name(const char* const str, const uint32_t value = val_32_const) noexcept {
    return (str[0] == '\0') ? value : hash_topic_name(&str[1], (value ^ uint32_t(str[0])) * prime_32_const);
}

// 컴파일 타임에 'vehicle_odometry' 문자열을 32비트 해시로 변환
constexpr uint32_t VEHICLE_ODOMETRY_TOPIC_HASH = hash_topic_name("vehicle_odometry");

최근 PX4 시스템에서는 컴파일 타임(Compile-time) 매크로 평가를 통해 이러한 해시 값을 사전에 모두 계산해 두므로, 런타임 동작에서는 연산 비용이 0(Zero) 에 수렴한다.

2.2 글로벌 데이터 공간(GDS)에서의 매핑(Mapping)

ROS 2 상에서 실행되는 uXRCE-DDS 에이전트는 내부에 std::map<uint32_t, std::string> 또는 해시 테이블 기반의 자료구조를 갖고 있다. 에이전트는 PX4 클라이언트로부터 초기화 패킷을 받을 때 각 해시 ID와 실제 토픽 문자열, 자료형 메타데이터(Type Metadata)를 세팅한다.

데이터 스트리밍 중에는 XRCE 프로토콜 헤더에 포함된 Topic ID 필드(해시 값)를 읽고, 해당 해시 테이블에서 상수 시간 O(1) 혹은 로그 시간 O(log N) 만에 실제 DDS 토픽 객체를 찾아 글로벌 데이터 공간(GDS)의 서브스크라이버들에게 페이로드를 전달한다.

3. 메모리 및 대역폭 최적화 효과

이 알고리즘이 제공하는 시스템적 이점은 명확하다.

  1. 대역폭(Bandwidth) 절감: 토픽 이름 문자열(약 20~40 바이트)을 매 메시지마다 담지 않고 2바이트 또는 4바이트의 해시 식별자만 전송하므로 직렬(Serial) 통신 대역폭을 획기적으로 보존할 수 있다.
  2. RAM 점유율(Memory Footprint) 최소화: uORB 버스 컨트롤러와 통신 클라이언트 양측 모두에서 내부 룩업 테이블(Lookup Table)이 해시 숫자로 구성되므로 구조체 크기가 줄어들며, 캐시 적중률(Cache Hit Ratio)이 상승한다.
  3. 지연 시간(Latency) 단축: strcmp 함수 등을 통한 문자열 비교 오버헤드를 완전히 제거하고 단순 정수 비교(Integer Comparison)로 처리하여, 250Hz 이상의 고속 센서 데이터를 처리할 때의 병목(Bottleneck) 현상을 해소한다.

4. 요약

PX4-Autopilot의 로컬 버스인 uORB와 ROS 2의 글로벌 네트워크 공간을 이어주는 연결 고리에서, 토픽 해시 기반의 매핑 알고리즘은 실시간성(Real-time Constraints)과 네트워크 대역폭 한계를 동시에 극복하게 하는 핵심 메커니즘이다. 문자열의 해닝(Hashing) 과정을 컴파일 주기로 추방시키고 브리지 통신 전반을 정수 키 체계로 전환함으로써, 시스템 엔지니어는 고빈도(High-Frequency) 관성 항법 데이터부터 대용량 비전 상태 정보까지 신뢰성 있게 서브스크라이빙할 수 있는 견고한 통신 라우팅 척추망을 얻게 된다.
