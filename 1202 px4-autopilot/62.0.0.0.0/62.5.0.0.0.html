62.5 아키텍처 확장성(Scalability): 사용자 정의 자료형(Custom Data Type) 인터페이스 설계 및 통신 버스 통합

현대 무인항공기(UAV) 산업에서는 표준화된 항법 스택 외에도 광학 검사장비, 방사선 센서, 다관절 로봇 팔 등 비표준 페이로드(Payload)를 장착하는 사례가 급증하고 있다. 최신 PX4-Autopilot(펌웨어 버전 1.14 이상) 아키텍처는 이러한 비표준 통신 요구를 수용하기 위해, ROS 2의 Micro XRCE-DDS와 연동되는 강력한 사용자 정의 자료형(Custom Data Type) 인터페이스를 제공한다. 본 절에서는 통신 버스의 병목 현상을 최소화하면서도 사용자 정의 메시지를 아키텍처에 통합하는 설계 기법을 다룬다.

1. MAVLink 확장 한계 및 ROS 2 중심의 아키텍처 패러다임

과거 Ardupilot 및 구형 PX4 시스템에서는 새로운 센서를 통합하기 위해 MAVLink 프로토콜의 common.xml이나 custom.xml 다이얼렉트(Dialect)를 수정하고 QGroundControl, 펌웨어, MAVSDK를 모두 재배포(Re-distribution)해야 하는 경직성(Rigidity) 한계가 존재했다. 지정된 메시지 식별자 범위 내에서만 개발해야 하므로, 복합적인 사용자 정의 배열이나 텍스트를 전송하려면 프로토콜 오버헤드가 극심했다.

반면, PX4 v1.14 이후 도입된 ROS 2 브릿지 아키텍처는 데이터 중심 퍼블리시-서브스크라이브(Data-Centric Publish-Subscribe, DCPS) 모델을 기반으로 한다. 개발자는 PX4 펌웨어 내부에 단순한 형식을 지닌 uORB 메시지 포맷(.msg)을 정의하고, 이를 Micro XRCE-DDS 브릿지 구성 파일에 선언하기만 하면, 미들웨어가 런타임에 동적으로 토픽 라우팅을 수행하게 된다. 이는 기존 C++ 컴포넌트 간 강한 결합(Tight Coupling)을 끊어내고 모듈 형식의 높은 확장성을 보장한다.

2. 사용자 정의 uORB 메시지(.msg) 인터페이스 설계 규칙

사용자 정의 인터페이스를 설계할 때에는 시스템의 메모리 정렬(Memory Alignment) 원칙을 준수해야 한다. NuttX RTOS 상에서 동작하는 마이크로컨트롤러 환경의 특성상, 64비트 변수부터 8비트 변수까지 크기가 큰 순서대로 멤버 변수를 선언하여 컴파일러의 패딩(Padding) 공간 낭비를 억제하라.

  • 원시 자료형 구성: 표준 .msg 파일은 uint64, float32, bool 등의 원시 자료형(Primitive Type)으로만 구성된다. 복잡한 구조체는 다른 .msg 파일의 포함(Inclusion)을 통해 계층적으로 설계될 수 구될 수 있다.
  • 안정성 보장: 배열의 크기는 반드시 컴파일 타임 런타임 정적 바운드(Static Bound) 스칼라 값 상수로 제한되어야 한다. 동적 크기(Dynamic Allocation)의 데이터형은 시스템의 결정론적(Deterministic) 타이밍 분석을 해칠 수 있어 허용되지 않는다.

3. 통신 버스 통합 아키텍처 흐름

새로운 자료형이 마이크로컨트롤러 메모리 내 uORB 토픽으로 할당된 후, ROS 2 글로벌 데이터 스페이스로 통합되는 빌드 및 라우팅 파이프라인은 다음과 같다.

    subgraph 1_Msg_Definition
        A[Define CustomPayload.msg] --> B[Add to CMakeLists.txt]
    end
    
    subgraph 2_Compilation_Layer
        B --> C[msg_gen Toolchain]

        D --> F(PX4 Internal uORB Topic)

        H <--> G
    end

개발자는 dds_topics.yaml (또는 microdds_client 설정 스크립트) 내에서 해당 토픽을 퍼블리시할지 서브스크라이브할지, 그리고 대역폭 제한을 위한 QOS(Quality of Service) 프로필(예: Best Effort, Reliable)을 명시하여 통신 버스를 최적화한다.

4. 커스텀 자료형 인터페이스 정의 예제 (Implementation Example)

다음은 사용자 정의 방사선량 측정 센서와 그 상태를 PX4 커스텀 제어 믹서(Custom Mixer)로 전달하기 위해 msg/CustomRadiationSensor.msg 파일을 정의한 예시이다. (크기순 정렬 적용)

# 파일 경로: px4-autopilot/msg/CustomRadiationSensor.msg
# 타임스탬프는 시스템 동기화를 위해 필수적인 멤버 헤더이다 
uint64 timestamp          # time since system start (microseconds)

# 64비트, 32비트 변수 우선 선언
float64 radiation_level   # µSv/h 기반 절대 피폭량 측정값
float32 temperature       # 센서 내부 온도 (섭씨 도)

# 16비트, 8비트, 불리언 타입 선언
uint16 error_code         # 자체 진단 고장 식별 코드
uint8  sensor_id          # 다중 센서 구성 시 고유 식별자
bool   is_calibrated      # 영점 보정 완료 여부

# 상수 선언 매크로
uint8 ERROR_NONE = 0
uint8 ERROR_OVERHEAT = 1

위 텍스트 명세서를 PX4 빌드 스크립트에 등록 후 소스를 컴파일하면, 런타임에 uORB 브로커가 이를 관리하게 된다. 그 후 ROS 2 워크스페이스(Workspace)의 px4_msgs 패키지에 동일한 .msg 파일을 복제하고 colcon build 파이프라인을 통과시키면, DDS 네트워크상에 완전한 인터페이스 브릿지가 성립된다.

5. 결론 및 참고 문헌

현대 무인기 임무의 다변화에 발맞추어, PX4의 uORB 메시지 포맷과 Micro XRCE-DDS 통신 버스의 조합은 하드웨어와 소프트웨어 간의 완벽한 느슨한 결합(Loosely Coupled Integration)을 가능하게 한다. C++ 마이크로컨트롤러 메모리 계층에 대한 정확한 이해를 바탕으로 사용자 자료형을 정렬 및 설계한다면, GCS 시스템 교체 없이도 강력하고 확장성 있는 전용 비행 서비스를 구축할 수 있다.
