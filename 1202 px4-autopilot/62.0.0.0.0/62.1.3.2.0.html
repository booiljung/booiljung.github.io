62.1.3.2 데이터 버스 아키텍처 정렬 제약(Alignment Constraints)을 충족하기 위한 메모리 패딩(Memory Padding) 및 오프셋(Offset) 산출 방식

PX4-Autopilot과 ROS 2 간의 통신은 uXRCE-DDS(Micro XRCE-DDS) 미들웨어 계층을 통해 이루어지며, 이 과정에서 공통 데이터 표현(Common Data Representation, CDR) 규격에 따른 메모리 직렬화(Serialization)가 필수적으로 요구된다. 임베디드 시스템에서 메모리에 데이터를 읽고 쓸 때, CPU 아키텍처 및 데이터 버스(Data Bus)의 특성에 따라 특정 크기의 데이터는 반드시 특정 메모리 오프셋(Offset) 배수(Multiple) 주소에 위치해야 하는 정렬 제약(Alignment Constraints)이 발생한다. 본 절에서는 이러한 제약을 충족하기 위해 적용되는 메모리 패딩(Memory Padding) 기법과 오프셋 산출 방식에 대해 상세히 다룬다. (기준: PX4-Autopilot v1.14 이상)

1. 데이터 버스 아키텍처와 정렬 제약의 원리

대부분의 현대 마이크로컨트롤러(MCU)와 CPU(예: ARM Cortex-M 시리즈, x86 등)는 데이터 버스의 폭(Width) 단위로 메모리에 접근한다. 예를 들어 32비트(4바이트) 아키텍처 시스템은 메모리 주소가 4의 배수인 위치에서 데이터를 한 번의 명령 사이클(Instruction Cycle)로 읽거나 쓸 수 있다.

만약 4바이트 크기의 데이터(예: int32_t)가 4의 배수가 아닌 주소지(Unaligned Address)에 저장되어 있다면, 프로세서는 해당 데이터를 읽기 위해 두 번의 메모리 접근을 수행하고 비트 시프트 연산을 통해 데이터를 조합해야 한다. 이는 시스템 버스의 오버헤드를 증가시키고, 일부 아키텍처(예: ARM Cortex-M0/M3)에서는 하드웨어 예외 처리(HardFault Exception)를 발생시켜 시스템 크래시를 유발하기도 한다.

1.1 기본 자료형의 정렬 규칙 (Alignment Rules)

CDR 직렬화 프로토콜에서는 각 데이터 타입에 대해 엄격한 정렬 규칙을 정의한다.

  • 1바이트 자료형 (int8_t, uint8_t, char, bool): 1바이트 정렬 (제약 없음)
  • 2바이트 자료형 (int16_t, uint16_t): 2의 배수 주소에 정렬 (2바이트 정렬)
  • 4바이트 자료형 (int32_t, uint32_t, float): 4의 배수 주소에 정렬 (4바이트 정렬)
  • 8바이트 자료형 (int64_t, uint64_t, double): 8의 배수 주소에 정렬 (8바이트 정렬)

2. 메모리 패딩(Memory Padding) 메커니즘

직렬화된 바이트 스트림(Byte Stream)을 생성할 때, 서로 다른 크기의 자료형이 연속적으로 직렬화되면 정렬 제약으로 인해 빈 공간이 발생하게 된다. 이 빈 공간을 의미 없는 더미 바이트(Dummy Bytes, 보통 0x00)로 채워 맞추는 과정을 패딩(Padding)이라 한다.

2.1 패딩 발생 예시

다음과 같은 구조체의 직렬화 과정을 가정해 보자.

struct SensorData {
    uint8_t  id;        // 1 bytes
    float    velocity;  // 4 bytes
    uint16_t status;    // 2 bytes
};

위 구조체가 CDR 기반으로 직렬화될 때 메모리에 배치되는 형태는 다음과 같다.

  • id 요소: 1바이트 크기이므로 오프셋 0에 배치된다. (크기: 1)
  • velocity 요소: 4바이트 크기이므로 오프셋은 4의 배수여야 한다. 현재 오프셋은 1이므로, 3바이트의 패딩이 추가된 후 오프셋 4에 배치된다. (크기: 4)
  • status 요소: 2바이트 크기이므로 오프셋은 2의 배수여야 한다. velocity 기록 후 현재 오프셋은 8(4+4)이며, 이는 2의 배수이므로 패딩 없이 배치된다. (크기: 2)

결론적으로 전체 직렬화 길이는 1 (id) + 3 (패딩) + 4 (velocity) + 2 (status) = 10바이트가 된다.

2.2 메커니즘 다이어그램

block-beta
  columns 10
  block:id:1
    ID("id (1B)")
  end
  block:pad1:3
    PAD1("Padding (3B)")
  end
  block:vel:4
    VEL("velocity (4B)")
  end
  block:stat:2
    STAT("status (2B)")
  end

위 다이어그램은 velocity 데이터가 4바이트 정렬을 맞추기 위해 3바이트의 보상된 오프셋 구간(Padding)을 갖는 것을 나타낸다.

3. 오프셋(Offset) 산출 로직

효율적인 직렬화 및 역직렬화를 위해서는 런타임에 동적으로 오프셋을 계산하는 수식이 필요하다. uXRCE-DDS 미들웨어 및 Fast CDR 라이브러리에서는 패딩을 산출하기 위해 다음과 같은 비트 연산 또는 모듈로(Modulo) 연산을 사용한다.

3.1 수식에 의한 패딩 크기 도출

특정 데이터 타입의 크기를 A (사용되는 정렬 기준 크기, 예: 2, 4, 8)라 하고, 현재 직렬화 버퍼의 오프셋을 C (Current Offset)라고 할 때, 정렬을 위해 추가되어야 할 패딩 크기 P는 다음과 같이 산출된다.

P = (A - (C \bmod A)) \bmod A

위 식에서 파생되는 개념을 통해 다음 기록을 위한 최종 오프셋(C_{new})은 아래 식을 따른다.

C_{new} = C + P + A

만약 프로세스의 연산 성능을 극대화하기 위해 비트 연산(Bitwise Operation)을 활용할 경우, A가 2의 거듭제곱인 성질을 이용하여 다음과 같이 표현할 수 있다.

P = (A - (C \mathbin{\&} (A - 1))) \mathbin{\&} (A - 1)

이 방식은 나눗셈 및 모듈로(Modulo) 연산 명령어가 비싼 임베디드 환경(PX4가 구동되는 STM32 등)에서 필수적인 최적화 기법이다.

3.2 패딩 및 오프셋 연산 코드 예제

실제 C/C++ 미들웨어 아키텍처 내부에서 오프셋과 패딩 크기를 계산하고 정렬하는 로직은 아래와 같이 전처리 매크로나 인라인 함수로 구현된다.

// 매크로를 이용한 CDR 얼라인먼트 패딩 계산 (PX4/uXRCE-DDS 소스 내부 구조 참조)
#define CDR_ALIGNMENT(current_alignment, data_size) \
    (((data_size) - ((current_alignment) & ((data_size) - 1))) & ((data_size) - 1))

void serialize_float(FastBuffer* buffer, float value, uint32_t* current_offset) {
    // 1. 필요한 패딩 크기 계산 (float는 4바이트 정렬)
    uint32_t padding = CDR_ALIGNMENT(*current_offset, sizeof(float));
    
    // 2. 오프셋 증가시켜 패딩 적용
    *current_offset += padding;
    
    // 3. 버퍼의 해당 위치에 값 쓰기
    memcpy(buffer->data + *current_offset, &value, sizeof(float));
    
    // 4. 값을 쓴 후 오프셋 갱신
    *current_offset += sizeof(float);
}

이 연산 과정에서 계산된 오프셋 값을 올바르게 추적하지 않으면 직렬화된 페이로드(Payload) 데이터가 손상되며, 이는 PX4와 ROS 2 간의 MAVLink 또는 uORB 메시지 파싱(Parsing) 시 심각한 결함(Fault)으로 이어진다.

4. 메시지 정의 최적화(Message Definition Optimization)

위에서 서술한 패딩 메커니즘으로 인해, 메시지 내의 변수 선언 순서에 따라 불필요한 네트워크 대역폭(Bandwidth) 낭비와 메모리 오버헤드가 발생할 수 있다. PX4 설계 시, uORB 메시지(*.msg)나 ROS 2의 인터페이스를 정의할 때는 패딩 발생을 최소화하기 위하여 데이터 크기가 큰 자료형부터 내림차순으로 선언하는 규칙을 적용한다.

비효율적인 구조 (패딩 다수 발생):

uint8_t  a;  // 1바이트 + 패딩 7바이트
uint64_t b;  // 8바이트
uint8_t  c;  // 1바이트 + 패딩 3바이트
uint32_t d;  // 4바이트

(총 크기: 24바이트)

최적화된 구조 (패딩 최소화):

uint64_t b;  // 8바이트 (크기가 가장 큰 자료형)
uint32_t d;  // 4바이트
uint8_t  a;  // 1바이트
uint8_t  c;  // 1바이트
// 뒤에 이어지는 패딩(Tail Padding) 2바이트 추가 후 버퍼가 종료됨 (필요시)

(총 크기: 16바이트)

이처럼 데이터 크기의 정렬 제약과 오프셋 매핑 방식을 명확히 이해하고 데이터 구조를 설계함으로써, 제한된 네트워크 환경에서도 데이터 버스 병목을 줄이고 시스템 처리 속도를 비약적으로 향상시킬 수 있다.
