62.4.4.1 이산 사건 제어 시스템(Discrete Event Control System)에서의 상태 천이(State Transition) 유발을 위한 명령어 식별자 인코딩

최신 무인항공기 비행 제어기, 특히 PX4-Autopilot(펌웨어 버전 1.14 이상)은 근본적으로 연속 시간(Continuous-time) 제어계와 결합된 하이브리드 자동기(Hybrid Automata) 구조를 띤다. 그중에서도 이륙, 시동, 모드 변경과 같은 비유동적 이벤트 처리는 전형적인 이산 사건 제어 시스템(Discrete Event Control System, DECS)으로 모델링된다. 본 절에서는 이러한 상태 천이(State Transition)를 유발하기 위해 외부 명령을 어떻게 고유한 식별자(ID)로 인코딩(Encoding)하고, 내부 커맨더(Commander) 모듈로 전달하는지에 대한 공학적 기법을 다룬다.

1. 이산 사건 제어 시스템 패러다임과 상태 천이

PX4-Autopilot의 최상위 제어 로직인 Commander 모듈은 내부적으로 거대한 유한 상태 기계(Finite State Machine, FSM)로 동작한다. 이 상태 기계는 ‘시동 대기(Standby)’, ‘시동(Arm)’, ‘이륙(Takeoff)’, ‘비행(In-air)’, ‘착륙(Land)’ 등 상호 배타적인 기체 상태(Vehicle State)의 집합을 가진다.

이산 사건(Discrete Event)은 외부 지상 관제 시스템(GCS, 예: QGroundControl)이나 ROS 2 환경의 컴패니언 컴퓨터(Companion Computer)로부터 전달되는 비동기적 트리거(Trigger)를 의미한다. 외부에서 기체의 상태를 변경하려면 단순히 임의의 문자열이나 함수를 호출하는 것이 아니라, 사전에 정의된 정수형의 **명령어 식별자(Command ID)**를 메시지에 인코딩하여 전송해야 한다.

2. 명령어 식별자(Command ID) 인코딩 아키텍처

PX4의 비동기 명령은 VehicleCommand uORB 토픽의 command 필드로 전달된다. 이 필드는 부호 없는 32비트 정수(uint32) 데이터형을 가지며, MAVLink 통신 규약(MAVLink Protocol)에 정의된 MAV_CMD 열거형(Enumeration) 상수를 그대로 차용하여 아키텍처의 일관성을 유지한다.

  • 식별자 매핑 예시:
  • VEHICLE_CMD_COMPONENT_ARM_DISARM = 400: 시동 상태 변경 유발 트리거
  • VEHICLE_CMD_NAV_TAKEOFF = 22: 이륙 고도 및 지점 도달 유발 트리거
  • VEHICLE_CMD_DO_SET_MODE = 176: 비행 모드(Flight Mode) 강제 오버라이드 트리거

식별자를 인코딩할 때에는 MAVLink 마이크로 서비스와 달리, ROS 2의 Micro XRCE-DDS 브릿지 환경에서는 px4_msgs::msg::VehicleCommand C++ 구조체의 멤버 변수에 이 스칼라 값을 직접 대입 연산하는 방식으로 이루어진다.

3. 유한 상태 기계(FSM) 모듈에서의 명령어 매핑 및 디코딩

PX4 펌웨어의 커맨더(src/modules/commander/Commander.cpp)는 외부 모듈에서 발행된 VehicleCommand가 수신되면, 등록된 이산 사건으로서 이를 디코딩(Decoding)한다.

디코딩 과정에서는 거대한 switch-case 구조가 작동하며, 수신된 식별자를 기반으로 현재 기체 상태(vehicle_status)와의 호환성을 검증한다. 예를 들어 이륙 중인 기체에 다시 VEHICLE_CMD_NAV_TAKEOFF 이벤트가 주입되면 예외 처리(Deny)를 수행하고, 이륙 대기 상태라면 서브루틴을 호출하여 상태 천이를 승인(Accept)한다.

4. QGroundControl (MAVLink) 대 ROS 2 (uORB) 인코딩 방식 비교

과거 QGroundControl을 포함한 Ardupilot 호환 GCS는 직렬 통신(UART/USB)이나 UDP 위에서 COMMAND_LONG 텔레메트리 패킷 바이트스트림을 직접 구성하여 식별자를 인코딩했다. 반면 최신 PX4의 ROS 2 환경 연동에서는 직접적인 바이트 오프셋(Byte Offset) 계산이나 체크섬(Checksum) 생성 스크립트 작성 로직이 생략된다. 개발자는 ROS 2 메시지의 추상화된 정수 형식 프로퍼티에 명명된 상수(const uint32_t)만 할당함으로써 더 강력한 형 안정성(Type Safety)을 확보할 수 단 있다.

6. ROS 2 환경을 위한 C++ 구현 예제

비행 모드를 Offboard 모드로 전환시키는 이산 사건을 인코딩하여 전송하는 ROS 2 C++ 예제이다. 여기서 식별자 VEHICLE_CMD_DO_SET_MODE가 핵심적인 트리거 역할을 한다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_command.hpp>

class ModeTransitionNode : public rclcpp::Node {
private:
    rclcpp::Publisher<px4_msgs::msg::VehicleCommand>::SharedPtr cmd_pub_;

public:
    ModeTransitionNode() : Node("mode_transition_node") {
        cmd_pub_ = this->create_publisher<px4_msgs::msg::VehicleCommand>("/fmu/in/vehicle_command", 10);
    }

    void trigger_offboard_mode() {
        px4_msgs::msg::VehicleCommand msg{};
        msg.target_system = 1;
        msg.target_component = 1;
        msg.source_system = 1;
        msg.source_component = 1;
        msg.from_external = true;
        msg.timestamp = this->now().nanoseconds() / 1000ULL;

        // 상태 천이를 위한 명령어 식별자 인코딩
        msg.command = px4_msgs::msg::VehicleCommand::VEHICLE_CMD_DO_SET_MODE;
        
        // Mode Transition을 위한 Custom Mode 매개변수 (PX4 Offboard 모드)
        msg.param1 = 1.0f;           // Base mode (Custom mode 활성화)
        msg.param2 = 6.0f;           // PX4 Custom Main Mode = PX4_CUSTOM_MAIN_MODE_OFFBOARD (6)
        msg.param3 = 0.0f;           // Sub Mode (0)

        cmd_pub_->publish(msg);
        RCLCPP_INFO(this->get_logger(), "Published Discrete Event ID: %d (DO_SET_MODE)", msg.command);
    }
};

7. 결론

단순한 데이터 전송 체계를 넘어, PX4-Autopilot의 명령 처리 기법은 철저하게 이산 사건 제어이론에 그 기반을 두고 있다. MAV_CMD 기반의 식별자(ID)를 ROS 2 추상화 계층 위에서 올바르게 인코딩하는 법을 이해하면, 미션 시퀀싱 및 다중 에이전트 자율 운용 시 복잡한 동적 궤적을 안전하게 구동할 상태 머신 동기화 논리를 구축할 수 있다.
