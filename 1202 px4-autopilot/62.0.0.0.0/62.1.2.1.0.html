62.1.2.1 클라이언트(Client)와 에이전트(Agent) 간의 상호 작용 상태 기계(Interaction State Machine) 분석

자원 제약이 있는 마이크로컨트롤러 환경(Micro eXtremely Resource Constrained Environments)에서, PX4-Autopilot 기반의 무인항공기와 임무 컴퓨터(Companion Computer)에 탑재되는 ROS 2 간의 통신 신뢰성을 확보하는 것은 필수적이다. Micro XRCE-DDS 프로토콜에서는 이를 위해 클라이언트와 에이전트 간의 상호 작용을 엄격한 상태 기계(State Machine)로 정의하여 세션(Session)의 연결, 유지 및 오류 복구를 관리한다. 본 절에서는 클라이언트(PX4)와 에이전트(ROS 2 측)가 어떠한 상태 전이를 통하여 통신 채널을 확립하고 안정적으로 운영하는지 심층 분석한다.

1. 상호 작용 상태 기계의 필요성

전송 채널로 시리얼(UART)이나 UDP를 이용할 경우, 물리적 연결의 단선이나 네트워크 지연, 패킷 유실 등이 빈번하게 발생할 수 있다. 단순히 메시지를 주고받는 방식을 넘어서, 양단 간에 동일한 연결 상태를 공유(Synchronization)하고 장애 발생 시 신속하게 복구하기 위해서는 규격화된 상태 기계 체계가 필요하다. Micro XRCE-DDS는 세션 기반의 상호 작용 모델을 사용하여 통신의 영속성(Persistence)과 무결성(Integrity)을 보장한다.

2. 클라이언트-에이전트 상태 전이 모델

클라이언트 모듈은 백그라운드 스레드에서 주기적으로 상태 기계를 갱신하며 에이전트와의 세션을 조율한다. 주요 상태(State)와 전이(Transition) 과정은 다음과 같이 규정할 수 있다.

stateDiagram-v2
    [*] --> UNINITIALIZED
    UNINITIALIZED --> DISCONNECTED : 초기화 완료(Init)
    DISCONNECTED --> HANDSHAKING : 연결 요청(Create Session)
    HANDSHAKING --> DISCONNECTED : 타임아웃/연결 거부(Reject)
    HANDSHAKING --> CONNECTED : 상태 수락(Session Configured)
    CONNECTED --> RUNNING : 엔티티(Entity) 생성 완료
    RUNNING --> RECONNECTING : 하트비트 누락(Keep-Alive Timeout)
    RECONNECTING --> HANDSHAKING : 세션 복구 시도
    RECONNECTING --> DISCONNECTED : 리셋(Reset) 및 포기

2.1 초기화 및 단절 상태 (UNINITIALIZED / DISCONNECTED)

PX4가 부팅되고 uxrce_dds_client 커맨드가 실행되면 시스템은 자원 할당 및 네트워크 소켓(또는 시리얼 포트)을 여는 UNINITIALIZED 상태를 거친다. 하드웨어적 초기화가 끝나면 DISCONNECTED 상태로 진입하며, 이때 클라이언트는 설정된 주기마다 에이전트를 탐색(Discovery)하거나 명시적으로 세션 생성 요청을 준비한다.

2.2 핸드셰이크 상태 (HANDSHAKING)

HANDSHAKING 상태에서 클라이언트는 에이전트 측으로 CREATE_CLIENT 서브메시지를 포함한 연결 요청 페이로드를 전송한다. 이 메시지에는 클라이언트 고유의 식별자(Client ID), 임대 시간(Session Timeout/Lease Duration), 스트림 채널 구성(신뢰성/비신뢰성 채널) 등의 필수 제원 정보가 포함된다.
에이전트는 이 요청을 검증한 뒤 STATUS_AGENT 또는 STATUS 속성의 서브메시지로 응답한다. 허가될 경우 양측은 CONNECTED 상태로 동기화된다. 장애 회피를 위해, 응답이 지정된 기한 내에 도착하지 않으면 클라이언트는 연결 시도를 중단하고 상태를 초기화한다.

2.3 연결 및 운영 상태 (CONNECTED / RUNNING)

세션이 확립된 CONNECTED 상태에서는 클라이언트가 발행자와 구독자 같은 구체적인 통신 엔티티(Entity)를 에이전트 상에 대리 생성(Creation)한다. 이 과정이 모두 끝나면 시스템은 정상 작동 상태인 RUNNING 모드로 전환한다.
이 상태에서는 실제 uORB 데이터(예: 센서 데이터, 오도메트리 등)가 직렬화되어 에이전트로 지속 전송되며, ROS 2 측의 경로 제어(Offboard Control) 명령이 에이전트를 경유하여 클라이언트로 반환된다.

2.4 재연결 상태 (RECONNECTING)

무선 간섭이나 케이블 접촉 부량 등으로 인해 에이전트로부터의 상태 확인(Keep-Alive/Heartbeat) 메시지가 지정된 임대 시간 이상 수신되지 않으면 상태 기계는 즉각 RECONNECTING 상태로 전이한다. 이 시점에서는 이미 구성된 엔티티 정보를 파기할지, 아니면 세션 지속성(Session Persistence) 스펙에 근거하여 통신만 재개할지 결정하게 된다. PX4-Autopilot은 비행 안전 정책(Failsafe State Machine)과 연동하여, 이 통신 단절이 치명적인 데이터 링크 단절(Data Link Loss)로 이어질지를 자체적으로 판단하는 추가적 메커니즘을 작동시킨다.

3. 동기화와 신뢰성 확보 메커니즘 (Heartbeat 및 ACK/NACK)

상태 기계가 RUNNING 상태를 지속하기 위해 Micro XRCE-DDS 프로토콜은 신뢰성 스트림(Reliable Stream) 운영을 지원한다.

  • Heartbeat 발행: 상태 머신 내부의 타이머에 의해 양방향으로 하트비트 신호가 발생한다. 이는 채널이 정상적으로 열려 있음을 상대방에게 알리는 동시에, 송신한 패킷 중 유실된 일련번호(Sequence Number)가 없는지 대조하는 역할을 수행한다.
  • ACK/NACK 처리 메커니즘: 수신 측은 받은 데이터의 순서가 맞지 않거나 누락이 감지되면 부정 응답(NACK) 메시지로 재전송을 요구하며, 정상 수신 시에는 긍정 응답(ACK) 메시지를 보낸다. 마이크로컨트롤러의 메모리는 극도로 제약되므로 패킷 재전송용 송신 버퍼(Tx Buffer) 용량을 적절히 산정하여 오버플로우를 막는 것이 이 상태 기계 관리의 핵심 과제이다.

4. 결론

Micro XRCE-DDS의 클라이언트-에이전트 상태 기계는 불안정한 물리적 데이터 링크 위에서 데이터의 흐름과 무결성을 논리적으로 보증하는 체계이다. 이러한 명확한 상태 분리 아키텍처 덕분에 PX4-Autopilot은 비행 제어 루프(Control Loop)에 지연을 유발하지 않고서도 외부의 고사양 연산 보드와 메시지를 주고받는 비동기식 상호작용 체계를 매우 안정적으로 유지할 수 있다.
