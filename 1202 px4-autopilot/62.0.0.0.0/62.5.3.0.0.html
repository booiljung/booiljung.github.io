62.5.3 클라이언트-에이전트 브릿지 구성을 위한 YAML 마크업 텍스트의 정규 표현식(Regular Expression) 파싱 및 토픽 라우팅(Topic Routing) 테이블 갱신 메커니즘

최신 PX4-Autopilot(펌웨어 버전 1.14 이상)과 ROS 2 간의 상호 운용성은 Micro XRCE-DDS 미들웨어에 의해 달성된다. 그러나 수백 개에 달하는 모든 uORB 토픽(Topic)을 무분별하게 브릿지 레이어로 개방할 경우 심각한 네트워크 대역폭(Bandwidth) 고갈 및 직렬화 오버헤드가 발생한다. 본 절에서는 개발자가 의도한 데이터 스트림만을 선별적으로 통과시키기 위해 사용되는 dds_topics.yaml 파일의 정규 표현식(Regular Expression) 파싱 기반 필터링 및 런타임 토픽 라우팅(Topic Routing) 테이블 갱신 기법을 다룬다.

1. uORB-DDS 브릿지 아키텍처와 YAML 설정의 역할

일반적인 ROS 2 환경의 노드들은 자유롭게 임의의 토픽을 퍼블리시/서브스크라이브 할 수 있지만, PX4 펌웨어 내부에서 동작하는 DDS 클라이언트(Micro XRCE-DDS Client) 모듈은 어떤 uORB 토픽을 DDS 글로벌 스페이스로 내보내거나(Publish) 또는 밖에서 읽어들일지(Subscribe)를 빌드 타임 단계에서 엄격히 컴파일해야 한다.

  • 브릿지 필터링(Bridge Filtering): 펌웨어 내 수많은 고빈도 센서 토픽(sensor_gyro, sensor_accel 등) 중 외부 로직 처리에 필요한 최소한의 집합만을 브릿지하여 직렬화 부하를 제한한다.
  • dds_topics.yaml: 이 목적을 위해 PX4 소스 트리의 src/modules/microdds_client/ 하위에 위치하는 YAML 파일. 이곳에 허용할 토픽의 송수신 방향 및 QoS(Quality of Service) 프로파일 명세를 마크업(Markup) 텍스트로 기술한다.

2. 정규 표현식(Regex) 기반 파싱 및 추상화

dds_topics.yaml 환경 설정은 단순한 1:1 토픽 이름 나열을 넘어, 정규 표현식 매칭(Regular Expression Matching)을 활용하여 다수의 유사한 성격의 토픽을 일괄적으로 그룹화하는 기능을 제공한다.

전처리 과정에서 수행되는 빌드 스크립트(주로 px4_ros_com 하위 또는 PX4 환경의 Python 스크립트 엔진)는 CMake 빌드 단계에서 이 마크업 텍스트를 파싱한다.
예를 들어 topic: vehicle_.* 와 같은 정규 표현식 패턴이 주어졌을 때, 파서는 내부 uORB 메시지 저장소(예: msg/*.msg)의 전체 목록과 교차 검증(Cross Validation)하여 매칭되는 모든 커스텀/프라이머리 토픽들을 탐색한다.

파싱이 완료된 토픽들은 SEND 및 RECEIVE 지시자에 따라 Micro XRCE-DDS 클라이언트 C++ 소스 코드 내에 구조체 배열 형태의 하드 코딩된 **토픽 라우팅 초기화 블록(Routing Initialization Block)**으로 렌더링(Rendering) 및 갱신된다.

3. 토픽 라우팅 테이블(Topic Routing Table) 갱신 메커니즘

이러한 정규식 파싱의 결과물은 컴파일 환경 내의 정형 데이터를 생성하여, 비행 제어기 보드(Pixhawk 등)가 부팅 시 microdds_client 데몬을 구동할 때 메모리에 정적 토픽 라우팅 테이블(Static Topic Routing Table)로 상주하게 된다.

  • 펌웨어가 초기화되면 Micro XRCE-DDS 에이전트 측으로 생성된 라우팅 테이블의 엔티티(Entity) 정보를 전달한다.
  • 에이전트(Companion Computer 측)는 수신한 명세서에 따라 ROS 2 글로벌 데이터 스페이스 내에 퍼블리셔 및 서브스크라이버 프록시(Proxy) 노드를 생성한다.
  • 이후 발생되는 모든 페이로드 트래픽은 이 정적 라우팅 트리에 포함된 해시 식별자(Hash Identifier)와 매핑될 때에만 허가된다.

4. MAVLink 스트림 구동과의 차이점 비교 (Ardupilot 대비)

Ardupilot 환경에서 구동되는 QGroundControl이나 Mission Planner는 MAVLink MAV_CMD_SET_MESSAGE_INTERVAL 커맨드를 전송하여 “실행 중(On-the-fly)” 특정 데이터 스트림의 송신 빈도를 동적으로 활성화/비활성화 시킬 수 있다.

반면, ROS 2의 Micro XRCE-DDS 기반 PX4 토픽 라우팅은 빌드 타임(Build-Time) 에 생성된 dds_topics.yaml 기반 C++ 배열에 종속되므로 런타임에 새로운 토픽을 추가 개폐할 수 없는 폐쇄성을 갖는다. 이는 동적 유연성은 떨어뜨리나 RTOS 동작 상의 안정성을 비약적으로 담보하며, MAVLink의 무작위 인터럽트로 인한 병목을 거부함으로써 자율 에이전트 수준의 보안(Security)을 제공한다.

6. YAML 파싱 규칙 및 라우팅 설정 예제

다음은 컴패니언 컴퓨터가 기체의 상태(vehicle_status)를 수신하고, 외부 궤적 제어(trajectory_setpoint)를 기체로 송신하도록 설정하기 위한 dds_topics.yaml의 일부 예시이다.

# FILE: src/modules/microdds_client/dds_topics.yaml

publications:
  - topic: vehicle_status
    type: px4_msgs::msg::VehicleStatus # 단일 토픽 명시 지정
  
  - topic: sensor_.*                   # 정규 표현식을 이용한 다중 토픽 배포 설정
    type: RegexFilter                  # sensor_gyro, sensor_mag 등이 일괄 등록 배포됨

subscriptions:
  - topic: trajectory_setpoint
    type: px4_msgs::msg::TrajectorySetpoint
  
  - topic: vehicle_command
    type: px4_msgs::msg::VehicleCommand

개발자는 위와 같이 정규식과 타입 지시자를 이용해 효율적으로 필터 테이블을 갱신하며, 빌드 시스템(make px4_fmu-v6x_default)을 재구동하여 펌웨어 내부의 브릿지 메커니즘을 동기화해야 한다.

7. 결론 및 참고 문헌

가벼운 텍스트 양식인 YAML 마크업의 정규 표현식 파싱을 통해 microdds_client의 토픽 라우팅 스페이스를 제한하는 것은, 임베디드 장비 메모리 파편화를 막고 대역폭 안정성을 보장하기 위한 가장 뛰어난 시스템 구성 기법이다. ROS 2 개발자는 MAVLink의 런타임 유연함에 길들여진 관성을 벗고, 빌드 타임 기반의 구조적이고 정적인 필터링 시스템이 주는 결정론적 하드웨어 무결성 구조를 체득해야만 한다.
