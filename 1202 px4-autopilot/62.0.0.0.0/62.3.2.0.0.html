62.3.2. 비동기 이벤트 구동(Asynchronous Event-driven) 모델에서의 스레드 안전(Thread-safe) 콜백(Callback) 처리 아키텍처

최신 무인항공기(UAV) 제어 시스템, 특히 PX4-Autopilot(펌웨어 v1.14 이상 기준)과 ROS 2를 연동하는 uXRCE-DDS 미들웨어 환경에서는 고주파(High-frequency)로 발생하는 다량의 센서 데이터와 기체 상태 정보가 비동기적(Asynchronous)으로 처리된다. 이러한 비동기 이벤트 구동(Asynchronous Event-driven) 아키텍처에서는 다중 스레드(Multi-thread) 환경에서 발생하는 동시성 문제(Concurrency issues)를 해결하기 위한 스레드 안전(Thread-safe) 콜백(Callback) 처리 구조의 설계가 매우 중요하다. 본 문서에서는 분산 로봇 시스템에서 데이터 무결성을 보장하기 위한 스레드 안전 콜백 아키텍처와 이를 구현하기 위한 구체적 방법론을 다룬다.

1. 비동기 이벤트 구동 모델과 ROS 2 실행기(Executor) 아키텍처

ROS 2(Robot Operating System 2)의 통신 구조는 기본적으로 비동기 이벤트 구동 모델을 채택하고 있다. 노드(Node)가 특정 토픽(Topic)을 구독(Subscribe)하면 데이터가 수신될 때마다 미들웨어를 통해 등록된 콜백 함수가 호출된다. PX4-Autopilot과 연동 시 자세 제어(Attitude Control)나 오도메트리(Odometry) 등 고주파로 발행되는 uORB(Micro Object Request Broker) 메시지는 ROS 2 브리지(Bridge)를 통해 DDS 객체로 전환되며, 이때 콜백 함수는 ROS 2 실행기(Executor)에 의해 관리되고 실행된다.

ROS 2의 스핀(Spin) 동작은 단일 스레드 실행기(Single-threaded Executor)와 다중 스레드 실행기(Multi-threaded Executor)로 나뉜다. 다중 스레드 실행기를 사용할 경우, 복수의 콜백이 동시에 병렬적으로 실행될 수 있어 데이터 전송 지연(Latency)을 단축하고 시스템 처리량(Throughput)을 향상시킬 수 있다. 하지만 프로세스 내의 동일한 자원(예: 전역 변수, 상태 관리 객체)에 동시에 접근하게 됨으로써 경쟁 상태(Race Condition)가 발생할 위험이 커진다.

1.1 스레드 안전성 보장을 위한 콜백 그룹(Callback Group) 분류

ROS 2에서는 콜백 함수의 동시 실행 여부를 제어하기 위해 콜백 그룹(Callback Group) 메커니즘을 제공한다.

  • 상호 배제 콜백 그룹(Mutually Exclusive Callback Group): 이 그룹에 속한 콜백들은 동시에 실행되지 않는다. 하나의 콜백이 실행 중일 경우 다른 콜백은 대기하게 되며, 스레드 안전성을 보장하기 쉽지만 병렬 처리의 이점을 상실할 수 있다.
  • 재진입 가능 콜백 그룹(Reentrant Callback Group): 이 그룹에 속한 콜백들은 다중 스레드 실행기에 의해 동시에 병렬로 실행될 수 있다. 성능을 극대화할 수 있지만, 콜백 내부에서 공유 자원을 처리할 때 반드시 개발자가 명시적으로 스레드 안전(Thread-safe) 설계를 구현해야 한다.

2. 공유 자원 접근 제어와 동기화(Synchronization) 메커니즘

수백 헤르츠(Hz) 주기로 갱신되는 VehicleOdometry 데이터나 VehicleStatus 메시지가 ROS 2 노드의 재진입 가능 콜백에서 처리될 때, 로컬 노드의 메모리 데이터를 최신 상태로 갱신하는 연산과 이 데이터를 기반으로 임무(Mission) 로직을 판단하는 연산이 동시에 발생할 수 있다. 이때 데이터 데이터 레이스(Data Race)를 방지하기 위하여 메커니즘이 필요하며, 주로 상호 배제(Mutex) 락과 조건 변수(Condition Variable)가 사용된다.

위의 단계에서 콜백 함수가 실행될 때, 표준 라이브러리의 std::mutex를 통한 잠금(Lock)을 획득하여 공유 데이터 영역의 일관성을 유지한다. 잠금을 획득하지 못한 다른 스레드의 콜백은 잠금이 해제될 때까지 스레드 컨텍스트를 블로킹(Blocking) 상태로 둔다.

3. 스레드 안전 콜백 설계 예제 (C++ 기반)

비행체의 가속도 및 각속도 정보를 포함하는 IMU 센서 데이터를 구독하여 전역 상태 저장소에 병합하는 ROS 2 C++ 노드의 스레드 안전 콜백 설계 예시이다. std::lock_guard를 사용하여 콜백 블록 내의 예외 발생 시에도 안전하게 락이 해제됨을 보장하는 RAII(Resource Acquisition Is Initialization) 패턴을 적용한다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/sensor_combined.hpp>
#include <mutex>

class ThreadSafeStateObserver : public rclcpp::Node {
public:
    ThreadSafeStateObserver() : Node("thread_safe_observer") {
        // 재진입 가능한 다중 스레드 콜백 그룹 설정
        callback_group_ = this->create_callback_group(
            rclcpp::CallbackGroupType::Reentrant);

        rclcpp::SubscriptionOptions options;
        options.callback_group = callback_group_;

        // 고주파 센서 데이터를 위한 구독자(Subscriber) 선언
        sensor_sub_ = this->create_subscription<px4_msgs::msg::SensorCombined>(
            "/fmu/out/sensor_combined",
            rclcpp::QoS(10).best_effort(),
            std::bind(&ThreadSafeStateObserver::sensor_callback, this, std::placeholders::_1),
            options);
    }

private:
    void sensor_callback(const px4_msgs::msg::SensorCombined::SharedPtr msg) {
        // 상호 배제 락을 통한 임계 구역(Critical Section) 보호
        std::lock_guard<std::mutex> lock(data_mutex_);
        
        // 공유 변수 갱신 (Race condition 방지)
        latest_gyro_rad_[0] = msg->gyro_rad[0];
        latest_gyro_rad_[1] = msg->gyro_rad[1];
        latest_gyro_rad_[2] = msg->gyro_rad[2];
        
        /* 추가적인 상태 추정 연산 블록 */
    }

    rclcpp::Subscription<px4_msgs::msg::SensorCombined>::SharedPtr sensor_sub_;
    rclcpp::CallbackGroup::SharedPtr callback_group_;
    
    std::mutex data_mutex_; // 데이터 레이스 방지를 위한 Mutex 객체
    float latest_gyro_rad_[3];
};

위 코드에서는 rclcpp::CallbackGroupType::Reentrant 속성을 통해 여러 콜백이 병렬로 진입할 수 있게 하되, data_mutex_를 통하여 클래스 멤버 변수인 latest_gyro_rad_ 배열을 조작하는 임계 영역(Critical Section)에서의 실행 컨텍스트 간 간섭을 격리한다.

4. 아키텍처 시사점 및 고도화 전략

이러한 스레드 안전 관리를 위해 락(Lock)을 과도하게 사용할 경우 락 경합(Lock Contention)에 의한 오버헤드가 발생하여 실시간성(Real-time Performance)을 훼손할 수 있다. 특히 드론 제어 시스템에서 수백 밀리초의 대기 지연은 치명적인 제어 실패를 야기할 수 있다.

따라서 락이 유지되는 구역(임계 구역의 길이)을 최소화하고, 단순 변수 병합의 경우 락이 없는(Lock-free) 자료구조나 원자적 연산(std::atomic) 객체를 도입하는 아키텍처로 고도화하는 것이 바람직하다. 아키텍처를 설계함에 있어 제어 데이터의 지연시간(Latency), 갱신 주기(Frequency) 등을 종합적으로 검토해야 안정적인 PX4-ROS 2 유도 및 제어망을 구축할 수 있다.
