62.2.1 ROS 2 패키지 파일 시스템 구조 정형화 및 의존성 그래프(Dependency Graph) 위상 정렬(Topological Sort)

드론 통제 시스템을 비롯한 대규모 로보틱스 소프트웨어를 구축할 때, 수십 개의 노드(Node)와 수백 개의 메시지 정의가 복잡한 관계망을 형성하게 된다. 이러한 복잡도를 관리하고 성공적인 교차 컴파일(Cross-Compilation)을 완수하기 위해 ROS 2 생태계는 엄격한 패키지(Package) 파일 시스템 구조를 강제하며, 패키지 간의 순서를 결정하기 위해 의존성 그래프(Dependency Graph)의 위상 정렬(Topological Sort) 매커니즘을 사용한다. 본 절에서는 PX4-ROS 2 브리지 구성 요소들이 어떠한 파일 시스템 체계를 따르고, 빌드 시 어떻게 정렬되는지 분석한다.

1. ROS 2 패키지 파일 시스템 구조 정형화

ROS 2(Humble 기준)의 기본 빌드 단위는 패키지(Package)이며, 각 패키지는 ament_cmake 또는 ament_python 빌드 시스템이 인식할 수 있도록 정형화된(Standardized) 파일 시스템 계층 구조를 갖추어야 한다. 특히 PX4와 통신하는 커스텀 메시지 패키지(예: px4_msgs)나 제어 알고리즘 패키지의 전형적인 구조는 다음과 같다.

px4_ros2_workspace/
 └── src/
     ├── px4_msgs/
     │   ├── CMakeLists.txt         # ament_cmake 빌드 구성 파일
     │   ├── package.xml            # 패키지 메타데이터 및 의존성 명세 (Manifest)
     │   ├── msg/                   # 사용자 정의 메시지 인터페이스 (.msg) 정의 디렉토리
     │   │   ├── VehicleOdometry.msg
     │   │   └── VehicleCommand.msg
     │   └── mapping_rules.yaml     # (선택) uXRCE-DDS 자료형 변환 룰
     └── offboard_control/
         ├── CMakeLists.txt
         ├── package.xml
         ├── src/                   # C++ 소스 코드 디렉토리
         │   └── offboard_control_node.cpp
         └── launch/                # 런칭 스크립트 디렉토리

1.1 메타데이터 파일 (package.xml)의 역할

package.xml은 독립된 패키지가 다른 어떤 패키지에 기대어 동작하는지(Dependencies)를 명시하는 선언적 파일(Declarative File)이다. 이곳에 명시된 <depend>, <build_depend>, <exec_depend> 태그의 내용들이 모여 전체 워크스페이스(Workspace)의 거대한 의존성 그래프를 구성하는 핵심 노드 및 간선(Edge) 데이터로 활용된다.

2. 의존성 그래프(Dependency Graph) 관점의 아키텍처 해석

여러 패키지를 묶어서 빌드하는 colcon (Collective Construction) 도구는, 빌드 명령이 하달되면 가장 먼저 각 패키지의 package.xml을 파싱(Parsing)하여 방향성 비순환 그래프(Directed Acyclic Graph, DAG) 형태의 의존성 그래프를 메모리상에 구축한다.

2.1 의존성 그래프 다이어그램

PX4와 ROS 2 간 오프보드(Offboard) 노드 빌드 시 발생할 수 있는 이상적인 의존성 그래프는 대략 다음과 같다.

graph TD
    %% 노드 정의
    A[std_msgs (ROS 2 기본 모듈)]
    B[px4_msgs (PX4 커스텀 메시지)]
    C[uxrce_dds_client (통신 미들웨어)]
    D[offboard_control (제어 비행 노드)]
    E[vio_estimator (비전 오도메트리 노드)]

    %% 의존성 관계 (화살표 방향: "A가 B에 의존성을 제공함", 즉 하위 -> 상위 빌드 흐름)
    A -->|1. Data Types| B
    B -->|2. Generated Structs| D
    B -->|2. Generated Structs| C
    B -->|2. Generated Structs| E
    C -->|3. DDS Link| D
    C -->|3. DDS Link| E

이 그래프에서 특정 코드를 컴파일하기 위해 사전에 반드시 컴파일이 완료되어야 하는 선수 패키지(Prerequisite Packages)들이 분명하게 나타난다. 예를 들어 px4_msgs의 C++ 헤더가 생성되기 전에 offboard_control 노드를 컴파일하려 시도하면 #include <px4_msgs/msg/vehicle_command.hpp> 단계에서 치명적 경로 탐색 오류(Fatal Error)가 발생한다.

3. 위상 정렬(Topological Sort) 알고리즘과 병렬 빌드 스케줄링

의존성이 꼬여있는 상황(순서 불일치)을 방지하기 위해 colcon 빌드 엔진은 생성된 DAG를 상대로 위상 정렬(Topological Sort) 알고리즘을 수행한다.

3.1 위상 정렬의 수학적 원리

위상 정렬은 방향 그래프의 꼭지점(Vertex)들을 선형 서열로 나열하는 것이다. 임의의 유향 간선 U \to V (U 빌드 완료 후 V 빌드 가능)에 대해, 정렬된 배열에서 노드 U가 배열의 앞쪽에 위치하고 인접 정점 V가 뒤쪽에 위치하도록 배치하는 연산이다.

가장 대표적인 Kahn’s Algorithm 기반의 위상 정렬 로직은 다음과 같이 스케줄링 그룹을 나눈다.

  1. 진입 차수(In-Degree)가 0인 노드(다른 것에 의존하지 않는 근본 패키지, 예: std_msgs)를 식별하여 그룹 1로 분류하고 병렬 빌드 워커(Worker)에 투입한다.
  2. 그룹 1의 빌드가 완료되면, 이 노드들에서 뻗어나가는 간선(해당 패키지에 대한 의존성)을 제거한다.
  3. 새롭게 진입 차수가 0이 된 노드들(px4_msgs)을 그룹 2로 묶고 병렬로 빌드한다.
  4. 이 과정을 그래프에 남은 잔여 노드가 없을 때까지 반복한다. 그래프에 사이클(순환 의존성, Circular Dependency)이 발견되면 즉시 빌드를 중단하고 에러를 뿜는다.

3.2 PX4와 ROS 2 연동 시 순환 의존성(Circular Dependency) 경고

드론 제어 시스템을 설계할 때 가장 조심해야 하는 것이 ‘순환 의존성’ 함정이다. 만약 개발자가 설계 실수로 A 패키지가 B 패키지의 메시지를 참조하고, 동시에 B 패키지가 A 패키지의 서비스를 참조하도록 package.xml과 CMakeLists.txt를 짜게 되면 위상 정렬이 불가능해져 해당 워크스페이스 전체의 통합 컴파일이 붕괴된다.

따라서 PX4-ROS 2 아키텍처에서는 오직 데이터 인터페이스만을 정의하는 전용 순수 패키지(Pure Message Package, 예: px4_msgs)를 로직 패키지와 철저히 격리시키는 패턴을 채택하고 있으며, 이러한 독립적 설계 철학이 대규모 시스템에서도 빠르고 안전하게 빌드 의존성을 해소(Resolution)할 수 있는 원동력이 된다.
