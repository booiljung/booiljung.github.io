62.1.1. 발행-구독(Publish-Subscribe) 패러다임 기반 데이터 분산 서비스(Data Distribution Service) 엔티티(Entity) 관계 모델링

무인항공기(UAS) 제어 시스템과 오프보드(Offboard) 컴퓨터 간의 실시간 통신 환경에서는 데이터를 중앙에서 통제하는 클라이언트-서버(Client-Server) 아키텍처보다, 각 노드가 독립적으로 데이터를 송수신할 수 있는 유연한 상호 작용 체계가 요구된다. PX4-Autopilot(이하 PX4)은 ROS 2와의 네이티브 연동을 완벽히 지원하기 위해 데이터 분산 서비스(Data Distribution Service, DDS) 기반의 발행-구독(Publish-Subscribe) 패러다임을 도입하였다. 본 절에서는 DDS 표준 규격에서 정의하는 핵심 엔티티(Entity)들의 역할과, PX4 시스템 내부에서의 관계 모델링 설계 방식에 대해 상세히 분석한다.

1. 발행-구독(Publish-Subscribe) 패러다임의 개요

발행-구독 모델은 네트워크 상에서 데이터를 교환할 때, 데이터의 생성자(Publisher)와 소비자(Subscriber)가 서로의 존재(IP 주소나 포트 등 물리적 위치)를 직접 알지 못하더라도 자유롭게 통신할 수 있도록 설계된 비동기 분산 통신 모델(Asynchronous Distributed Communication Model)이다.

전통적인 프레임워크인 ROS 1 시스템이나 기본적인 TCP/IP 소켓 지점 간 통신과 달리, DDS는 전역 데이터 공간(Global Data Space, GDS)이라는 가상의 분산 데이터 공유 영역에 토픽(Topic) 형태로 데이터를 발행(Publishing)하고, 필요한 컴포넌트에서만 조건에 맞춰 이를 구독(Subscribing)하는 방식을 취한다. PX4는 내부의 코어 통신 아키텍처인 uORB가 이미 발행-구독 모델을 완벽하게 채택하고 있기에, DDS 기반의 외부 통신망과 구조적으로 매우 높은 동형(Isomorphic) 유사성을 지니며 프로토콜 상호 간 변환 브리지 구성에 매우 유리하다.

2. DDS 핵심 엔티티(Entity) 관계 모델링 구조

DDS 표준 명세(Specification) 상의 통신을 실현하는 핵심 엔티티 객체들은 다음과 같은 위계적인 상속 관계를 형성한다. 각 엔티티는 객체 지향적 시스템 설계 사상에 입각하여 미들웨어 통신의 특정 제어 범주를 추상화한다.

2.1 도메인 참여자(Domain Participant)

DDS 통신 네트워크 내에서 모든 개별 통신의 최상위 캡슐화 단위이자 논리적 격리 공간인 도메인(Domain)에 소속된 최초 진입점을 도메인 참여자(Domain Participant)라고 칭한다. PX4 통신 환경에서 펌웨어 내의 Micro XRCE-DDS 클라이언트는 하나의 도메인 참여자로 인스턴스화되며, 내부적으로 하위 발행자와 구독자를 생성하고 관리하는 팩토리(Factory)의 역할을 수행한다.

2.2 퍼블리셔(Publisher)와 데이터 라이터(DataWriter)

  • 퍼블리셔(Publisher): 도메인 참여자로부터 생성되어 하나 이상의 데이터 라이터(DataWriter)를 관리하고 구동 그룹화하는 상위 객체이다. 다중 데이터 전송 시 일괄적으로 적용될 공통의 서비스 품질(QoS, Quality of Service) 설정 프로파일을 제어할 수 있다.
  • 데이터 라이터(DataWriter): 특정 토픽(Topic)에 대해 실제 바이너리 데이터 값을 전역 데이터 공간(GDS)에 기록하는(Write) 역할을 전담하는 최종 하위 엔티티이다. PX4에서 센서 값(예: EKF2 산출값, IMU 센서 데이터 등)을 ROS 2 측으로 주기적으로 송출할 때 이 데이터 라이터 인터페이스를 통해 바이트 스트림이 출력된다.

2.3 서브스크라이버(Subscriber)와 데이터 리더(DataReader)

  • 서브스크라이버(Subscriber): 하나 이상의 데이터 리더(DataReader) 객체를 논리적 배열로 관리하며, 수신된 데이터에 대한 인터럽트형 콜백(Callback)이나 주기적 이벤트 폴링(Polling) 구조의 흐름을 총괄한다.
  • 데이터 리더(DataReader): 전역 데이터 공간에서 자신이 구독 설정한 특정 토픽의 데이터를 물리적으로 읽어들이는(Read/Take) 엔티티이다. 외부 오프보드 제어 환경으로부터 유입되는 제어 궤적 명령(Trajectory Setpoint) 등은 클라이언트 단의 데이터 리더가 읽어들여 역직렬화한 뒤 로컬 uORB 버스로 전달된다.

2.4 토픽(Topic)과 타입(Type)

토픽은 데이터를 발행하는 측과 구독하는 측을 논리적으로 연결시키는 공유 데이터 기반 채널이다. 각 식별 토픽은 고유한 문자열 이름(Name)과 엄격하게 정의된 데이터 타입(Data Type) 구조체를 가진다. DDS 통신망에서는 송신 및 수신 측의 토픽명, 그리고 직렬화 해시에 대응하는 데이터 타입이 완벽하게 일치해야만 데이터 바인딩(Data Binding) 통신이 성립된다.

3. PX4 통신 시스템 내에서의 모델링 적용 및 GCS 구조 비교

초기 부팅 과정에서 PX4 펌웨어에 내포된 uxrce_dds_client 서비스 모듈은 컴파일 타임(Compile time)에 파싱(Parsing)되어 생성된 dds_topics.yaml 명세 테이블을 참조하여 상술한 각 엔티티(DataWriter 및 DataReader)를 동적으로 할당하고 초기화한다.

반면, ArduPilot의 관제 통신 생태계는 역사적으로 MAVLink 텔레메트리 채널의 다이렉트 메시지 라우팅 시스템 체계에 강하게 의존하고 있다. ArduPilot의 MAVLink 아키텍처 구조에서는, 네트워크 모델 상 각각의 통신 주체가 IP나 고유한 System ID / Component ID 속성 기반 연결 위에서 송수신을 집중적으로 처리하지만, DDS 기반의 엔티티(Entity) 객체 모델링처럼 세밀하고 정교한 종단 간 서비스 품질(QoS) 제어 보장이나 동적인 타입 디스커버리(Dynamic Type Discovery)를 네이티브 시스템 수준에서 완벽하게 지원하기에는 한계가 존재한다.

이러한 관점에서 PX4 플라이트 스택 내부에 공식 적용된 DDS 엔티티 모델링은 ROS 2의 RMW(ROS Middleware) 레이어 아키텍처 철학을 임베디드 단속(Firmware)까지 온전히 계승할 수 있어, 학술적 융합 연구나 산업용 대규모 다중 드론 군집 관제(Swarm Control) 시스템 구현 시 논리적으로 그 확장성(Scalability)이 매우 뛰어나다는 지대한 강점을 확보하게 된다.
