62.2.3. CMake 메타 빌드 시스템 기반의 병렬 컴파일(Parallel Compilation) 최적화

PX4-Autopilot(펌웨어 버전 v1.14 기준)과 ROS 2(Robot Operating System 2)의 연동 환경을 구축할 때, uXRCE-DDS 인터페이스 및 다수의 ROS 2 패키지를 소스 코드로 빌드하는 과정은 상당한 시간이 소요된다. 특히 대규모 분산 통신 마이크로 아키텍처를 교차 컴파일(Cross-Compilation)로 구성하려면 빌드 시스템(Build System)을 고도화할 필요가 있다. 본 절에서는 CMake 메타 빌드 시스템(Meta-Build System)을 활용하여 병렬 컴파일(Parallel Compilation)을 최적화하고 빌드 시간을 극대화로 단축하는 기법을 상세히 설명한다.

1. PX4-Autopilot 및 ROS 2 체계의 CMake 빌드 구조 이해

전통적인 ROS 1 시스템 체계에서는 catkin이 빌드를 담당했으나, ROS 2에서는 colcon을 기반으로 내부에서 CMake 메타 빌드 시스템을 호출하여 빌드를 수행한다. PX4-Autopilot 역시 메인 빌드 시스템으로 CMake를 채택하고 있다. 이는 단일 명령어(예: make px4_fmu-v6x_default)가 입력되었을 때, 내부적으로 ninja 또는 make 프로세스를 통해 실제 컴파일러(Compiler)인 GCC나 Clang을 호출하는 방식이다.

CMake는 본질적으로 크로스 플랫폼(Cross-Platform) 빌드 파일 생성기(Generator)이다. 따라서 CMake 자체가 코드를 컴파일하는 것이 아니라, 빌드 설정 파일(CMakeLists.txt)을 파싱(Parsing)하여 병렬 처리에 유리한 build.ninja나 전통적인 Makefile을 생성한다.

2. 병렬 컴파일(Parallel Compilation) 최적화 원리

단일 스레드(Single-Threaded) 환경에서는 소스 코드 파일(예: .cpp, .c)이 순차적으로 목적 파일(Object File, .o)로 변환된다. 반면, 현대의 멀티 코어(Multi-Core) 중앙처리장치(CPU)를 적극적으로 활용하는 병렬 컴파일 환경에서는 의존성(Dependency)이 없는 다수의 소스 파일을 동시에 컴파일할 수 있다.

아래는 단일 컴파일과 병렬 컴파일의 차이를 보여주는 프로세스 다이어그램이다.

    subgraph Single Compilation
        A1[Source 1] --> B1[Compile Task 1]
        B1 --> C1[Object 1]
        C1 --> A2[Source 2]
        A2 --> B2[Compile Task 2]
        B2 --> C2[Object 2]
    end

    subgraph Parallel Compilation
        P1[Source 1] --> Q1[Compile Task 1 - CPU Core 1]

        P3[Source 3] --> Q3[Compile Task 3 - CPU Core 3]
        Q1 --> R1[Object 1]
        Q2 --> R2[Object 2]
        Q3 --> R3[Object 3]
    end

3. Ninja 빌드 시스템의 채택과 병렬 처리 효율성

CMake는 기본 제너레이터(Generator)로 GNU Make를 사용하지만, 최근 PX4-Autopilot 개발 환경에서는 Ninja 빌드 시스템의 사용이 권장된다(QGroundControl v4.3.x 이상 버전을 위한 Qt 6 빌드 환경에서도 기본적으로 Ninja가 널리 사용된다).

Ninja는 작은 규모의 빠른 빌드를 목표로 개발된 경량 빌드 시스템으로, 파일의 수정 시간(Timestamp)이 아닌 파일 내용의 해시(Hash)를 추적하여 재빌드(Rebuild) 여부를 결정한다. 가장 큰 특징은 별도의 플래그(Flag) 입력 없이도 가용한 CPU 코어 수를 감지하여 명시된 한도 내에서 최대한의 병렬 컴파일을 자동으로 수행한다는 점이다.

CMake에서 Ninja 엔진을 활성화하는 커맨드는 다음과 같다.

mkdir build && cd build
cmake -G Ninja ../
ninja

ROS 2의 colcon 환경에서 Ninja를 사용하여 PX4 인터페이스를 병렬로 컴파일하려면 다음 옵션을 부여하라.

colcon build --cmake-args -G Ninja

이 커맨드를 사용하면 각 ROS 2 노드와 uXRCE-DDS 메시지 인터페이스가 서로 의존성이 얽히지 않은 상태일 때 여러 패키지를 동시에 병렬로 빌드하므로, 빌드 속도가 드라마틱하게 향상된다.

4. CMake 3.12 이상의 병렬 빌드(Parallel Build) 명령 옵션 활용

CMake 환경 자체가 제공하는 독립된 병렬 빌드 인터페이스(Interface)를 활용할 수도 있다. CMake는 컴파일 시 -j 옵션을 통해 병렬 코어의 수를 직접 지정할 수 있도록 지원한다.

다음 명령어는 CMake가 프로젝트를 16개의 작업(Job)으로 분할하여 병렬로 빌드하도록 지시한다.

cmake --build build_dir --parallel 16
# 또는
cmake --build build_dir -j 16

이는 개발 PC 플랫폼의 가용 코어가 16개일 경우 시스템 자원 사용률(Resource Utilization)을 최대치(100%)로 끌어올려 빌드를 수행하게 한다. 이러한 병렬 처리 시스템은 “Fast Parallel Builds with CMake (2018)” 등의 논문에서도 지속적으로 연구되어 메타 빌드 시스템의 핵심 최적화 요소로 자리 잡았다.

5. 컴파일 캐시(ccache)를 통한 반복 빌드 최적화

병렬 컴파일 성능을 한층 더 끌어올리는 강력한 툴은 컴파일 캐시 매니저(Compile Cache Manager)인 ccache이다. ccache는 이전에 컴파일을 완료한 목적 코드(Object Code)를 로컬 디스크의 캐시 스토리지에 저장하고, 코드 변경이 발생하지 않은 상태에서 재빌드가 요청되면 실제 컴파일러를 호출하는 대신 캐시된 파일을 즉각적으로 반환(Hit)한다.

PX4 펌웨어 메인 저장소를 클론(Clone)하고 빌드할 때 환경 변수를 적절하게 주입하면, CMake 빌드 시스템이 C/C++ 컴파일러를 ccache로 감싸서(Wrap) 초기화를 진행한다.

export CC="ccache gcc"
export CXX="ccache g++"

CMake의 내장 속성(Property)을 이용하여 빌드 시스템 전체에 ccache를 적용할 수도 있다. ROS 2 환경의 워크스페이스를 빌드할 때 다음과 같은 CMake 인자(Argument)를 전달하라.

colcon build --cmake-args -DCMAKE_C_COMPILER_LAUNCHER=ccache -DCMAKE_CXX_COMPILER_LAUNCHER=ccache

5.1 ccache와 Parallel Compilation 연동의 효과

ccache와 병렬 컴파일을 결합하면 최초 빌드(Cold Build) 시에는 모든 코어 자원을 사용해 컴파일을 빠르게 마치고, 이후의 반복 빌드(Warm Build 또는 Incremental Build) 시에는 병렬화된 작업을 통해 아주 빠른 속도로 캐시 검증 및 목적 파일을 로드하게 된다. 이는 로직 수정과 빌드를 반복해야 하는 오프보드(Offboard) 제어 개발이나, 방대한 수의 메시지가 포함된 uXRCE-DDS 데이터 직렬화 코드(Data Serialization Code)를 설계할 때 엄청난 시간적 이득을 제공한다.

6. 결론

PX4-Autopilot과 ROS 2 간의 인터페이스 애플리케이션이나 펌웨어를 빌드할 때, CMake 메타 빌드 시스템의 설정을 최적화하는 것은 개발 생산성과 직결된다. 단순히 make 커맨드를 사용하는 것을 넘어서, Ninja 빌드 시스템 기반의 제너레이션, --parallel 속성 적용, 그리고 ccache 시스템 통합이라는 삼원 조합(Triad Configuration)을 구축하라. 이는 단일 컴파일 환경 대비 십수 배 이상의 컴파일 효율성을 제공하는 최적의 개발 환경이다.
