62.3. 상태 관측기(State Observer) 설계: 다중 센서 데이터 및 기체 상태 정보 서브스크라이빙(Subscribing)

현대의 무인항공기(UAV) 제어 시스템에서 자율 비행(Autonomous Flight) 알고리즘이나 컴퓨터 비전(Computer Vision) 기반의 추적 제어를 수행하기 위해서는 기체의 현재 상태를 실시간으로 정확하게 파악하는 것이 필수적이다. PX4-Autopilot(펌웨어 버전 v1.14 기준)과 ROS 2(Robot Operating System 2)의 연동 환경에서는 uXRCE-DDS(Micro XRCE-DDS) 브리지를 통해 고속의 상태 데이터 동기화가 이루어진다. 본 절에서는 ROS 2 오프보드(Offboard) 보조 컴퓨터 환경에서 다중 센서 데이터와 확장 칼만 필터(EKF2) 기반의 기체 추정 상태 정보를 효율적으로 수집하는 상태 관측기(State Observer)의 설계 패턴과 서브스크라이빙(Subscribing) 기법에 대해 상세히 다룬다.

1. 아키텍처 아웃라인 및 데이터 파이프라인

PX4-Autopilot의 내부 통신 버스인 uORB(Micro Object Request Broker)는 다양한 센서(IMU, GPS, 기압계 등)와 상태 추정기(Estimator)의 데이터를 토픽(Topic) 형태로 지속적으로 발행(Publishing)한다. 상태 관측기(State Observer) 노드는 ROS 2 네트워크 인터페이스에 존재하며, 이 uORB 토픽들이 uXRCE-DDS 에이전트를 거쳐 ROS 2 토픽으로 변환될 때 이를 구독(Subscribe)하여 자율 에이전트 알고리즘에 공급하는 데이터 허브 역할을 수행한다.

2. 주요 상태(State) 토픽 및 자료형

상태 관측기를 설계할 때 가장 우선적으로 고려해야 할 것은 어떠한 데이터를 취합할 것인지에 대한 정의이다. 일반적으로 오프보드 제어 및 경로 계획(Path Planning)에서 주로 서브스크라이빙하는 px4_msgs의 토픽들은 다음과 같다.

  1. SensorCombined: 내부 IMU 단위의 원시 센서 데이터(가속도계, 자이로스코프)를 통합하여 제공한다. 제어 주파수가 250Hz 이상의 고속 동기화가 필요한 로우 레벨 자세 제어 알고리즘에 활용된다.
  2. VehicleOdometry: 기체의 현재 3차원 위치(Position), 속도(Velocity), 자세(Orientation, Quaternion 기반), 그리고 각속도(Angular Velocity) 추정치를 포함한다. 경로 계획 및 장애물 회피 알고리즘에서 기하학적 기준 좌표계(일반적으로 NED 또는 FRD 좌표계)를 획득하기 위해 가장 빈번하게 사용된다.
  3. VehicleStatus: 현재 기체의 시동 상태(Arming State), 비행 모드(Flight Mode - Nav State), 배터리 인가 상태 등 다방면의 운영/관리 메타데이터를 제공한다.

3. 서비스 품질(QoS, Quality of Service) 프로파일의 중요성

ROS 2의 분산 통신 인프라에서는 데이터의 특성에 따라 서로 다른 QoS 프로파일을 적용해야 한다. 상태 관측기가 실시간 센서 데이터를 서브스크라이빙할 때는 데이터의 손실 지연(Loss Retransmission Overhead)보다 최신 데이터의 즉각적인 반영이 중요하다. 따라서 VehicleOdometry나 SensorCombined와 같은 고주파수 스트리밍 토픽은 반드시 Best Effort (최선 노력) 신뢰성(Reliability) 프로파일을 적용하여야 한다.

반면, 모드 변경이나 시동 상태를 알려주는 VehicleStatus는 한 번의 메시지 유실이 치명적인 논리적 오류를 발생시킬 수 있으므로 Reliable (신뢰성 보장) 프로파일을 유지하는 것이 바람직하다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_odometry.hpp>
#include <px4_msgs/msg/vehicle_status.hpp>

class StateObserver : public rclcpp::Node {
public:
    StateObserver() : Node("state_observer") {
        // 센서/상태 추정 데이터 (고속 타임리얼, Best Effort 적용)
        rclcpp::QoS qos_sensor_profile(10);
        qos_sensor_profile.best_effort();
        qos_sensor_profile.volatility(rclcpp::VolatilityPolicy::Volatile);

        // 시스템 상태 메타데이터 (저속/이벤트 드리븐, Reliable 적용)
        rclcpp::QoS qos_status_profile(10);
        qos_status_profile.reliable();
        qos_status_profile.durability(rclcpp::DurabilityPolicy::TransientLocal);

        // Vehicle Odometry 서브스크라이버 초기화
        odom_sub_ = this->create_subscription<px4_msgs::msg::VehicleOdometry>(
            "/fmu/out/vehicle_odometry", qos_sensor_profile,
            std::bind(&StateObserver::odom_callback, this, std::placeholders::_1)
        );

        // Vehicle Status 서브스크라이버 초기화
        status_sub_ = this->create_subscription<px4_msgs::msg::VehicleStatus>(
            "/fmu/out/vehicle_status", qos_status_profile,
            std::bind(&StateObserver::status_callback, this, std::placeholders::_1)
        );
    }

private:
    void odom_callback(const px4_msgs::msg::VehicleOdometry::SharedPtr msg) {
        // 타임스탬프 동기화 및 EKF 추정 위치 데이터 갱신 로직
        current_x_ = msg->position[0]; 
        current_y_ = msg->position[1];
        current_z_ = msg->position[2];
        // ... (이후 궤적 생성기에 데이터 전달)
    }

    void status_callback(const px4_msgs::msg::VehicleStatus::SharedPtr msg) {
        // 기체 Arming 및 Offboard 관제 가능 상태 체크 로직
        is_armed_ = (msg->arming_state == px4_msgs::msg::VehicleStatus::ARMING_STATE_ARMED);
    }

    rclcpp::Subscription<px4_msgs::msg::VehicleOdometry>::SharedPtr odom_sub_;
    rclcpp::Subscription<px4_msgs::msg::VehicleStatus>::SharedPtr status_sub_;

    float current_x_{0.0f}, current_y_{0.0f}, current_z_{0.0f};
    bool is_armed_{false};
};

4. 다중 센서 데이터 동기화 기법

드론의 제어 루프를 구성할 때 여러 센서에서 발행되는 토픽의 발행 주기(Publishing Rate)는 제각기 다르다. IMU는 250Hz 이상이지만, GPS는 통상 5~10Hz로 동작한다. 이를 단일 제어 알고리즘에서 조합(Fusion)하기 위해서는 상태 관측기 내부에 시간 동기화 모델(Time Synchronization Model)이 구현되어야 한다.

ROS 2의 message_filters 라이브러리를 활용하면 특정 시간 공차(Time Tolerance) 이내에 수신된 다중 토픽을 정확한 타임스탬프 단위로 매치(Match)시켜 콜백 함수에 동기화된 상태로 전달할 수 있다. PX4-Autopilot의 uXRCE-DDS 브리지는 PX4 운영체제(NuttX) 상의 timestamp를 그대로 ROS 2 구조체에 실립하여 전송하므로, 상태 관측기는 ROS 로컬 머신의 타임스탬프(System Clock) 대신 메시지 페이로드에 포함된 타임스탬프를 기준으로 데이터를 재정렬(Reordering)하고 동기화하는 전략을 택해야 시스템 오차 및 지연 시간 측정을 정밀하게 수행할 수 있다.

5. 결론

상태 관측기(State Observer) 노드는 복잡한 오프보드 자율 비행 알고리즘 개발을 위한 데이터 허브이다. uXRCE-DDS를 통한 PX4-Autopilot의 px4_msgs 포맷을 C++ 구조체로 다룰 때, 주파수에 따른 QoS 프로파일 분리, 페이로드 기반 타임스탬프 동기화, 시스템 결점을 최소화하기 위한 논블로킹(Non-blocking) 비동기 콜백 체인 구성은 메모리와 CPU 자원을 최적화하면서 관제 안정성을 확보하는 아키텍처적 핵심이다.
