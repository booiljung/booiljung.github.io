62.2.2 인터페이스 정의 언어(Interface Definition Language) 구문 분석(Parsing) 및 목적 코드(Object Code) 자동 생성 아키텍처

로봇 운영체제(ROS 2)와 PX4-Autopilot은 독립적인 소프트웨어 스택으로 발전해 왔으나, 두 시스템을 묶어주는 핵심 매개체는 **인터페이스 정의 언어(Interface Definition Language, IDL)**이다. 개발자가 px4_msgs 패키지 등에 .msg 형태로 변수들을 나열해 두면, 빌드 시스템은 이 텍스트 파일을 분석(Parsing)하여 로우 레벨 환경에서 동작하는 C/C++ 구조체와 직렬화/역직렬화용 목적 코드(Object Code)를 런타임 이전에 자동 생성(Auto-generation)한다. 본 장에서는 이러한 IDL 구문 분석기가 어떻게 동작하며, 이로부터 파생되는 코드 생성 아키텍처가 PX4 내에서 어떻게 구현되는지 다룬다.

1. 인터페이스 정의 언어(IDL)와 ROS 2 .msg 포맷

통신에 참여하는 두 엔티티(예: 비전 노드와 픽스호크 보드)는 주고받을 데이터의 구조를 동일하게 이해해야 한다. 이를 강제하는 것이 IDL이다.
ROS 2에서 널리 쓰이는 .msg 파일은 궁극적으로 OMG(Object Management Group) 표준인 순수 .idl 포맷으로 변환되어 처리되는데, 다음과 같이 극도로 단순화된 문법을 지닌다.

# VehicleLocalPosition.msg 예시
uint64 timestamp        # time since system start (microseconds)
float32 x               # Position in local NED frame
float32 y
float32 z
float32[3] v            # Velocity in NED frame

이 코드는 특정 프로그래밍 언어(C++이나 Python)에 종속되지 않은 선언적 명세(Declarative Specification)이다.

2. 구문 분석(Parsing) 엔진 메커니즘

CMake 빌드 인프라가 가동되면, 가장 먼저 rosidl_adapter 및 px4_generate_uorb_topic과 같은 파이썬 기반 파서(Parser)가 호출되어 텍스트 파일을 기계가 이해할 수 있는 트리(Tree) 원형 구조로 변환한다.

2.1 Lexical Analysis (어휘 분석) 및 토크나이징(Tokenizing)

파서는 .msg 파일을 읽고 공백, 주석(# 기호 이후)을 제거한 뒤, 의미 있는 문자열 조각(Token)으로 분해한다.

  • 타입 식별자(Type Identifier): uint64, float32
  • 심볼(Symbol): timestamp, x, v
  • 배열 스펙(Array Specifier): [3]

2.2 심볼릭 매핑(Symbolic Mapping) 및 메타데이터 추출

어휘가 추출되면 각 토큰의 의미론(Semantics)을 검사한다. float32는 4바이트 부동소수점이라는 것을 검증하며, 배열의 길이가 고정 길이(Fixed Array, [3])인지 동적 길이(Dynamic Array, [])인지 식별한다.
PX4와 같은 uXRCE-DDS 클라이언트 환경에서는 동적 메모리 할당(malloc/new)이 극도로 제한되므로 동적 배열(sequence 타입)에 대해 엄격한 바인딩 제약을 가하거나 최대 길이(Bounded Sequence) 메타데이터를 필수적으로 도출한다.

3. 목적 코드(Object Code) 자동 생성 아키텍처

구문 분석을 통해 데이터 모델이 메모리에 온전히 구축되면, 다음 단계인 **코드 생성 엔진(Code Generator)**이 가동된다.

PX4-ROS 2 프레임워크에서는 주로 Jinja2나 EmPy와 같은 템플릿 엔진을 사용하여, C/C++ 소스 코드를 찍어내는 틀(Template)에 분석된 메타데이터를 주입(Injection)한다.

코드 생성기가 만들어내는 출력물은 크게 세 가지 계층으로 나뉜다.

3.1 1계층: uORB C++ 타입 추상화 헤더

가장 먼저 PX4 커널 로직을 위한 순수 C/C++ 구조체(struct)가 생성된다.

/* Auto-generated header: vehicle_local_position.h */
struct vehicle_local_position_s {
    uint64_t timestamp;
    float x;
    float y;
    float z;
    float v[3];
};

이 구조체는 컴파일 타임(Compile-time)의 정렬 제약(Alignment)과 패딩 규칙에 맞게 배치되어 드론 비행 제어 알고리즘(FlightTask 등) 내부에서 곧바로 사용된다.

3.2 2계층: 직렬화/역직렬화 (Serialization) C 소스 코드

구조체를 바이트 스트림(Byte Stream)으로 변환하는 로직을 사람이 일일이 작성하는 것은 위험도가 높다(휴먼 에러). 코드 생성기는 이 변환 과정을 수행하는 마이크로-CDR(Common Data Representation) 인터페이스 코드를 토해낸다.

/* Auto-generated CDR logic */
bool serialize_vehicle_local_position(FastBuffer* buffer, const vehicle_local_position_s* input) {
    serialize_uint64(buffer, input->timestamp);
    serialize_float(buffer, input->x);
    // ... 나머지 필드 직렬화
    return true;
}

3.3 3계층: 엔드포인트 바인딩 로직

마지막으로 이 직렬화 함수들을 uXRCE-DDS 미들웨어의 토픽 트리거 함수에 연결해 주는 브리지 백엔드(Bridge Backend) C++ 코드가 생성된다. PX4 클라이언트는 이 생성된 코드를 의무적으로 링크(Link)함으로써 외부 네트워크에서 들어온 패킷을 로컬 구조체로 변환하여 내부 uORB 네트워크로 무사히 발행(Publish)할 수 있게 된다.

이러한 **단일 소스 진실(Single Source of Truth, 즉 .msg 파일)**에 기반한 자동 생성(Auto-Generation) 아키텍처는, 개발자가 통신 프로토콜의 하위 디테일에 신경 쓰지 않고 비행 제어 및 경로 계획 등 상위 수준의 알고리즘에만 집중할 수 있는 완벽한 추상화 계층(Abstraction Layer)을 선물한다.
