62.3.4.3 기준 좌표계 선형 변환(Linear Transformation): 북동하(North-East-Down) 직교 좌표계에서 동북상(East-North-Up) 직교 좌표계로의 변환을 위한 직교 행렬(Orthogonal Matrix) 곱셈 연산

무인항공기(UAV) 제어를 위한 소프트웨어 아키텍처 환경에서 시스템 간 데이터 연동 시 가장 빈번히 마주하는 근본적인 장벽 중 하나는 기준 좌표계(Frame of Reference)의 불일치이다. 특히 PX4-Autopilot은 항공우주 분야의 통상적인 표준형인 북-동-하(NED: North-East-Down) 직교 좌표계를 내부 연산 기반으로 채택하고 있는 반면, ROS2 및 일반적인 로보틱스 프레임워크 진영에서는 동-북-상(ENU: East-North-Up) 직교 좌표계를 널리 표방한다. 본 절에서는 두 좌표계 간의 공간 벡터를 호환하기 위해 필수적으로 수행되는 선형 변환(Linear Transformation)의 기하학적 의미와 직교 행렬(Orthogonal Matrix)을 이용한 선명한 곱셈 연산 과정에 대해 수학적으로 상세히 기술한다.

1. 직교 좌표계(Orthogonal Coordinate System)의 특성

NED와 ENU 좌표계는 모두 서로 직각으로 교차하는 세 축으로 이루어진 3차원 유클리드 공간(Euclidean Space) 내의 직교 좌표계이다. 두 좌표계 간의 위치(Position)나 속도(Velocity) 벡터를 변환하는 과정은 물리적 공간의 크기(Scale)나 형태를 임의로 변형시키지 않고 오직 축 구성을 향한 회전(Rotation)과 반전(Reflection)만을 수반하는 강체 변환(Rigid Transformation) 유도 혹은 선형 변환(Linear Transformation)으로 규정된다.

  • NED (PX4 기준): 공간의 X축은 진북(True North), Y축은 진동(True East), Z축은 중력 방향(Down)을 향해 늘어져 있다.
  • ENU (ROS2 기준): 공간의 X축은 진동(True East), Y축은 진북(True North), Z축은 반중력 방향인 상단(Up)을 향하도록 축이 구성된다.

2. 좌표 변환을 위한 직교 행렬(Orthogonal Matrix) 연산 모델

NED 좌표계로 표현된 3차원 상태 벡터 V_{NED} = [x_{N}, y_{E}, z_{D}]^T를 ENU 좌표계 지향 벡터인 V_{ENU} = [x_{E}, y_{N}, z_{U}]^T로 올바르게 변환하기 위해서는 다음과 같은 차원 매핑 관계가 논리적으로 확립되어야 한다.

  • ENU의 X축 물리량 (동) = NED의 Y축 물리량 (동)
  • ENU의 Y축 물리량 (북) = NED의 X축 물리량 (북)
  • ENU의 Z축 물리량 (상) = 거꾸로 대치된 NED의 Z축 물리량 (하)

이를 선형 대수학적인 직교 행렬(Orthogonal Matrix, R_{NED}^{ENU})의 곱셈 형태 단일 연산으로 명쾌하게 묶어 표현하면 다음과 같은 수식이 도출된다.

V_{ENU} = R_{NED}^{ENU} \cdot V_{NED}

$$
\begin{bmatrix}
x_E \
y_N \
z_U
\end

\begin{bmatrix}
0 & 1 & 0 \
1 & 0 & 0 \
0 & 0 & -1
\end{bmatrix}
\begin{bmatrix}
x_N \
y_E \
z_D
\end{bmatrix}
$$

설계된 행렬 R_{NED}^{ENU}는 행렬식(Determinant)이 -1이며, 그 자신의 전치 행렬(Transpose Matrix)이 역행렬(Inverse Matrix)과 정확히 동일한 직교 행렬의 공리적 성질(R^T = R^{-1})을 만족한다. 따라서 반대 방향인 로보틱스 ENU에서 항공 NED로의 타깃 역변환 역시 단순히 기존 행렬의 전치 행렬을 전위 곱셈하는 것만으로 신속하게 연산될 수 있다.

V_{NED} = (R_{NED}^{ENU})^T \cdot V_{ENU} = (R_{NED}^{ENU}) \cdot V_{ENU}

(위에서 유도된 특수 변환 행렬은 스스로 대칭 행렬 구조를 취하고 있기도 하므로, R = R^T가 성립하여 순방향과 역방향의 변환 연산자가 외견상 완벽히 동일하다.)

3. ROS2 - PX4 간 오도메트리 위치/속도 변환 아키텍처 (Mermaid)

PX4의 분산 미들웨어인 uXRCE-DDS 클라이언트를 통하여 ROS2 환경으로 기체의 위치 및 속도 오도메트리 데이터를 송달하거나 상위 제어 명령을 하강(Publish)시킬 때, 통신 브릿지 노드는 다음의 시퀀스 다이어그램 패턴과도 같은 변환 처리를 필연적으로 거쳐야 오류가 소멸된다.

graph TD
    A[PX4 EKF2 Estimator <br/> NED Frame] -->|Position/Velocity Data| B[uXRCE-DDS Agent]
    B -->|px4_msgs::msg::VehicleOdometry| C{ROS2 Frame Transformer Node}
    
    C -->|Extract V_NED| D[Matrix Multiplication<br/>V_ENU = R * V_NED]
    D -->|R Matrix Application| E[Construct Target Msg<br/>nav_msgs::msg::Odometry]
    E -->|Publish V_ENU| F[ROS2 Control Stack <br/> ENU Cartesian Frame]
    
    F -->|Command Velocity V_ENU| C
    C -->|V_NED = R^T * V_ENU| A

위의 시스템 연계 구조도에서 가리키듯, VehicleOdometry 혹은 외부 자율 제어 타깃인 TrajectorySetpoint 등 공간의 병진 및 지향을 지시하는 메시지가 이기종 시스템 경계망(ROS2 \leftrightarrow PX4)을 횡단할 때, 위치와 선속도 벡터 필드류 전반에 걸쳐 앞서 언급된 3 \times 3 규모의 직교 행렬 연산이 파이프라인 상류에서 일괄 수행되어야 한다. 만일 이를 누락하거나 혼동할 경우 드론이 조종사나 상위 자율 비행 알고리즘의 의도 궤적을 벗어나 치명적인 이탈 비행(Fly-away) 현상을 일으킬 다분한 위험성이 존재하게 된다.
