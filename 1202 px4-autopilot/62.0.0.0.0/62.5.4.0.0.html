62.5.4 양방향 폐루프(Bi-directional Closed-loop) 데이터 스트림 무결성 검증을 위한 핑퐁(Ping-Pong) 지연 시간(Latency) 측정 노드 설계

PX4-Autopilot(펌웨어 버전 1.14 이상)과 ROS 2(Micro XRCE-DDS) 기반의 분산 비행 제어 아키텍처에서는, 에이전트와 클라이언트 간의 물리적 통신 버스(UART, UDP, CAN 등)와 미들웨어 직렬화/역직렬화에 따른 내재적 **네트워크 지연 시간(Latency)**이 필연적으로 발생한다. 본 절에서는 오프보드 제어 시 데이터 스트림의 실시간 무결성(Data Integrity)을 보장하고, 타이밍 위반(Timing Violation)을 감지하기 위한 폐루프(Closed-loop) 핑퐁(Ping-Pong) 노드 설계 이론을 다룬다.

1. 분산 오프보드 환경에서의 지연 시간(Latency) 위협

수백 헤르츠(Hz) 주기로 궤적 추종(Trajectory Tracking)을 수행하는 ROS 2 비전/AI 어플리케이션에 있어 통신 딜레이는 단순한 데이터 도착 지연을 넘어, 위상 지연(Phase Lag)을 유발하여 PID 등 폐루프 제어기의 발산(Instability)을 초래하는 핵심 보안 위협이다.

기체 내부의 IMU나 이스케이프 페이로드 센서의 최신 상태 메타데이터가 외부 노드에서 너무 늦게 파악되거나, 외부 노드에서 계산된 자율 회피 궤적(TrajectorySetpoint)이 PX4 내부 믹서 큐에 뒤늦게 도달하여 타당성을 잃는 현상을 실시간으로 모니터링할 핑퐁(Ping-Pong) 루프백 측정 메커니즘이 요구된다.

2. 핑퐁(Ping-Pong) 아키텍처와 시퀀스 무결성 논리

핑퐁 측정 메커니즘의 핵심은 ROS 2 측에서 시작된 질의(Ping)가 PX4 내부 uORB 네트워크를 거쳐 응답(Pong)으로 돌아오는 데 걸리는 **총 왕복 시간(Round-Trip Time, RTT)**을 산출하는 것이다.

  • 타임스탬프 반사(Timestamp Reflection): ROS 2 노드는 고유 시퀀스 번호(Sequence Number)와 발송 시간표(Unix Epoch 64비트 마이크로초)를 담은 트랜잭션을 퍼블리시한다.
  • 루프백(Loopback): PX4 펌웨어에는 해당 핑 토픽을 수신하자마자 하드웨어 클럭을 재기록하지 않고, 수신된 발송 시간표를 그대로 복사하여 퐁(Pong) 토픽으로 발행하는 초경량 에코(Echo) 모듈이 구현된다.
  • 무결성 검출(Integrity Detection): ROS 2 노드는 수신된 퐁 토픽의 시퀀스 누락 여부로 패킷 유실(Packet Loss)을 감지하고, 수신 시간과 반사된 발송 시간 간의 차이로 RTT 레이턴시를 도출한다. 이때 RTT가 임계치(예: 20ms)를 넘을 경우 즉각 제어권을 수동 모드로 반환(Fail-safe)하도록 동작한다.

3. MAVLink TIMESYNC와의 아키텍처 및 성능 특성 비교

과거 MAVLink 기반 QGroundControl 환경에서는 TIMESYNC 메시지(ID: 111)를 사용하여 지상국과 픽스호크 클럭 간의 주기적 동기화를 수행했다. 그러나 MAVLink 통신은 상대적으로 직관적이고 가벼운 헤더를 가진 반면 동기적(Synchronous) 단일 버스 특성이 강했다.

최신 ROS 2 분산 아키텍처 하의 Micro XRCE-DDS 구조는 RTPS(Real-Time Publish-Subscribe) 프로토콜로 작동한다. 데이터 직렬화(CDR Format), 에이전트 내의 토픽 라우팅 큐(Queue), 송수신 스레드 스위칭 오버헤드가 추가된다. 따라서 ROS 2 브릿지 상에서의 핑퐁 레이턴시 변동성(Jitter)은 기존 MAVLink 시리얼 통신에 비해 분산되어 나타나며, QoS 정책(Best Effort vs Reliable) 설정에 따라 측정되는 지연 프로파일의 확률 밀도(PDF) 곡선이 확연히 달라진다.

5. C++ 구현 시 레이턴시 측정 및 RTT 평가 로직 (Implementation Example)

다음은 C++14 기반 ROS 2에서 커스텀 핑 토픽을 송출하고 퐁응답을 받아 무결성 및 지연 시간을 진단하는 측정 노드의 핵심 구현 구조이다. (미리 생성한 CustomPingPong.msg에 uint32 seq 및 uint64 ping_time이 존재한다고 가정한다.)

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/custom_ping_pong.hpp>
#include <chrono>

using namespace std::chrono;

class LatencyWatchdogNode : public rclcpp::Node {
private:
    rclcpp::Publisher<px4_msgs::msg::CustomPingPong>::SharedPtr ping_pub_;
    rclcpp::Subscription<px4_msgs::msg::CustomPingPong>::SharedPtr pong_sub_;
    rclcpp::TimerBase::SharedPtr timer_;

    uint32_t sequence_id_ = 0;
    const uint64_t MAX_ALLOWED_RTT_US = 25000; // 허용 최대 왕복 지연시간 (25ms)

    void pong_callback(const px4_msgs::msg::CustomPingPong::SharedPtr msg) {
        // 도착 시간 기록 (마이크로초)
        uint64_t rx_time = time_point_cast<microseconds>(steady_clock::now()).time_since_epoch().count();
        uint64_t rtt_us = rx_time - msg->ping_time;

        // 패킷 누락 검출
        if(msg->seq != sequence_id_ - 1) {
            RCLCPP_WARN(this->get_logger(), "Packet Loss detected! Expected SEQ: %d", sequence_id_ - 1);
        }

        // 지연 시간 임계치 진단
        if (rtt_us > MAX_ALLOWED_RTT_US) {
            RCLCPP_ERROR(this->get_logger(), "Critical Latency Warning: RTT %lu us exceeds limit! Trigger Failsafe.", rtt_us);
            // 이곳에 통신 레이턴시 초과 시 호버링(Hold) 혹은 귀환(RTL) 명령 하달 트리거 로직 추가
        } else {
            RCLCPP_INFO(this->get_logger(), "Stream Healthy. RTT: %.2f ms (SEQ: %d)", rtt_us / 1000.0, msg->seq);
        }
    }

    void publish_ping() {
        px4_msgs::msg::CustomPingPong px_msg;
        px_msg.seq = sequence_id_++;
        // 출발 시간 기록 (마이크로초)
        px_msg.ping_time = time_point_cast<microseconds>(steady_clock::now()).time_since_epoch().count();

        ping_pub_->publish(px_msg);
    }

public:
    LatencyWatchdogNode() : Node("latency_watchdog_node") {
        ping_pub_ = this->create_publisher<px4_msgs::msg::CustomPingPong>("/fmu/in/ping_topic", rclcpp::QoS(10).best_effort());
        pong_sub_ = this->create_subscription<px4_msgs::msg::CustomPingPong>(
            "/fmu/out/pong_topic", rclcpp::QoS(10).best_effort(),
            std::bind(&LatencyWatchdogNode::pong_callback, this, std::placeholders::_1));

        // 50Hz (20ms 주기)로 Ping 전송 타이머 구동
        timer_ = this->create_wall_timer(20ms, std::bind(&LatencyWatchdogNode::publish_ping, this));
    }
};

6. 결론

데이터 스트림의 RTT 및 유실률을 런타임에 정량적으로 스크리닝(Screening)하는 핑퐁 측정 노드는, 고속 임무(High-speed Mission)를 집행하는 자율 무인항공기의 ’통신계 센서’와 동일한 전략적 지위를 갖는다. 아키텍트는 분산 제어 하드웨어의 병목점을 예의주시해야 하며, 레이턴시 허용 임계값을 초과했을 때 신속히 제어권을 반환하는 능동형 감시견(Watchdog) 패턴을 오프보드 필수 유스케이스로 규정해야 한다.
