62.2.3.1. 메시지 생성 매크로(Macro) 전개 및 동적 링크 라이브러리(Dynamic Link Library) 기호(Symbol) 노출 설정

PX4-Autopilot 기반 무인항공기(UAV) 체계와 ROS 2(Robot Operating System 2) 모듈이 uXRCE-DDS(Micro XRCE-DDS) 플랫폼을 통해 연동할 때, 가장 중요한 파이프라인 중 하나는 인터페이스(IDL, Interface Definition Language) 파일로부터 실제 프로그래밍 언어의 소스 코드를 생성해 내는 과정이다. 본 절에서는 CMake 환경 내에서 메시지 생성 매크로(Macro)가 전개(Expansion)되는 원리와, 이로 인해 빌드된 동적 링크 라이브러리(Dynamic Link Library, DLL 혹은 Shared Library .so)에서 기호(Symbol) 노출을 제어하는 최적화 기법에 대해 상세히 다룬다.

1. 메시지 생성 매크로(Macro) 전개 파이프라인

ROS 2 및 PX4가 연동되는 빌드 파이프라인에서 커스텀 메시지, 서비스, 액션 단위의 .msg, .srv, .action 파일들은 C, C++, Python 등의 목적 코드로 치환되어야 한다. 이를 위해 CMakeLists.txt 파일 내부에서는 다양한 추상화된 매크로를 제공하며, 대표적으로 rosidl_generate_interfaces() 매크로가 사용된다.

이 매크로는 단일 명령어로 보이지만, 내부적으로는 타겟 언어(Target Language)에 맞는 생성기(Generator)들—예: rosidl_generator_c, rosidl_generator_cpp, rosidl_typesupport_introspection_cpp 등—을 연쇄적으로 호출하는 거대한 파이프라인으로 전개된다.

1.1 매크로 동작 다이어그램

아래 다이어그램은 인터페이스 정의 파일에서 시작하여 CMake 매크로 전개를 통해 소스 코드가 생성되고 공유 라이브러리로 컴파일되는 절차를 보여준다.

2. 동적 링크 라이브러리의 기호(Symbol) 노출 제어

위 과정을 통해 수많은 구조체 선언과 함수들이 자동 생성된다. 이때 컴파일러가 아무런 지시 없이 이 코드들을 라이브러리로 링킹(Linking)하게 되면, 생성된 모든 함수와 클래스에 대한 기호(Symbol) 테이블이 동적 라이브러리의 전역 공간에 노출(Export)된다. 이는 다음과 같은 치명적인 단점을 야기한다.

  1. 라이브러리 크기 증가: 노출된 기호 테이블 메타데이터로 인해 바이너리의 크기(Memory Footprint)가 비대해진다.
  2. 이름 충돌(Name Collision): 다른 의존성 라이브러리와 중복된 이름이 존재할 가능성이 높아진다.
  3. 심볼 탐색 시간(Lookup Time) 지연: 프로그램 로드 시간에 동적 링커(Dynamic Linker)가 기호를 해석(Resolve)하는 시간이 증가하여 부팅 또는 노드 실행 속도가 느려진다.

이를 해결하기 위해, ROS 2의 메시지 생성 파이프라인은 컴파일러 의존적인 심볼 가시성 속성(Visibility Attribute)을 제어하는 매크로 헤더(예: visibility_control.h)를 자동 생성한다.

3. 기호 노출(Visibility) 제어를 위한 매크로 설계 기법

리눅스(Linux) 환경의 GCC 및 Clang 컴파일러에서는 __attribute__((visibility("default"))) 및 __attribute__((visibility("hidden"))) 지시어를 사용하여 심볼의 노출 여부를 결정한다. 윈도우(Windows)의 경우 __declspec(dllexport)와 __declspec(dllimport)를 사용한다.

자동화된 빌드 시스템에서는 각 플랫폼에 맞는 매크로를 전개할 수 있도록 플랫폼 독립적인 코드가 삽입된다. 일반적인 메시지 패키지의 visibility_control.h는 다음과 같이 구성된다.

#ifndef PX4_MSGS__VISIBILITY_CONTROL_H_
#define PX4_MSGS__VISIBILITY_CONTROL_H_

// Windows 플랫폼을 위한 로직
#if defined _WIN32 || defined __CYGWIN__
  #ifdef __GNUC__
    #define PX4_MSGS_EXPORT __attribute__ ((dllexport))
    #define PX4_MSGS_IMPORT __attribute__ ((dllimport))
  #else
    #define PX4_MSGS_EXPORT __declspec(dllexport)
    #define PX4_MSGS_IMPORT __declspec(dllimport)
  #endif
  #ifdef PX4_MSGS_BUILDING_LIBRARY
    #define PX4_MSGS_PUBLIC PX4_MSGS_EXPORT
  #else
    #define PX4_MSGS_PUBLIC PX4_MSGS_IMPORT
  #endif
  #define PX4_MSGS_PUBLIC_TYPE PX4_MSGS_PUBLIC
  #define PX4_MSGS_LOCAL
// Linux 및 macOS(POSIX) 플랫폼을 위한 로직
#else
  #define PX4_MSGS_EXPORT __attribute__ ((visibility("default")))
  #define PX4_MSGS_IMPORT
  #if __GNUC__ >= 4
    #define PX4_MSGS_PUBLIC __attribute__ ((visibility("default")))
    #define PX4_MSGS_LOCAL  __attribute__ ((visibility("hidden")))
  #else
    #define PX4_MSGS_PUBLIC
    #define PX4_MSGS_LOCAL
  #endif
  #define PX4_MSGS_PUBLIC_TYPE
#endif

#endif  // PX4_MSGS__VISIBILITY_CONTROL_H_

위의 매크로 시스템에서 알 수 있듯이,CMake 레벨에서 플래그 -fvisibility=hidden 을 사용하여 기본적으로 모든 심볼을 은닉(Hide)시킨다. 이후, 외부 어플리케이션(예를 들어 사용자 정의 오프보드 제어기)에서 명시적으로 접근해야 하는 특정 구조체와 함수에 한해 PX4_MSGS_PUBLIC 매크로를 수식하여 명시적인 노출을 허용한다.

4. CMakeLists.txt에서의 기호 가시성 설정 최적화

만약 독자적인 C++ 인터페이스 라이브러리를 구축하고, 이것이 uXRCE-DDS 에이전트와 연동된다면, 개발자는 CMakeLists.txt 에 명시적으로 CXX 컴파일러 플래그 설정을 기입하여 기호 테이블 최적화를 보장해야 한다. 다음은 이를 구현한 예시이다.

# C++14 이상 권장
set(CMAKE_CXX_STANDARD 14)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# 기본 기호 가시성을 숨김(hidden) 처리
set(CMAKE_CXX_VISIBILITY_PRESET hidden)
set(CMAKE_C_VISIBILITY_PRESET hidden)
# 인라인 함수들의 가시성 역시 숨김 처리
set(CMAKE_VISIBILITY_INLINES_HIDDEN YES)

# 타겟 라이브러리 정의
add_library(px4_custom_control SHARED
  src/trajectory_controller.cpp
  src/mode_manager.cpp
)

# 라이브러리 빌드 시 엑스포트(Export) 매크로 선언을 트리거
target_compile_definitions(px4_custom_control 
  PRIVATE "PX4_CUSTOM_CONTROL_BUILDING_LIBRARY"
)

이와 같이 메타 빌드 시스템에서 가시성 제어 속성을 부여함으로써 생성된 라이브러리는 Ardupilot 기반 시스템의 메시지 파서 구현체와 대비하여 런타임 심볼 해석의 오버헤드가 극도로 축소되며, 메모리가 제한적인 보조 컴퓨터(Companion Computer) 환경에서도 빠른 로드 속도와 낮은 자원 점유율을 달성할 수 있다.

5. 요약

PX4-Autopilot 및 ROS 2 체제하에서의 메시지 정의 파일(.msg)은 rosidl 기반의 CMake 매크로 전개를 통해 다량의 C/C++ 소스 구조체로 변환된다. 빌드된 동적 링크 라이브러리의 용량을 줄이고 로딩 성능을 최적화하기 위해서는 반드시 기본 가시성 은닉(Default Visibility Hidden) 전략을 취하여야 하며, 이는 컴파일러 종속성을 타파하는 자동 생성 visibility_control.h 매크로를 통해 달성된다. 이러한 최적화는 복잡도가 점화되는 현대 멀티-로터 및 자율 비행 시스템 관제 체계 구축에서 코어 런타임 성능을 유지시키는 중요한 소프트웨어 공학적 발판이 된다.
