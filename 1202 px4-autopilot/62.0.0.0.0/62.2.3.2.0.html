62.2.3.2. 실행 시간(Runtime) 의존성과 빌드 시간(Build-time) 의존성의 논리적 분리를 통한 공유 라이브러리 메모리 발자국(Memory Footprint) 최소화

자원(Resource)이 극도로 제한된 비행 제어 컴퓨터(Flight Control Computer, FCC) 환경에서 작동하는 PX4-Autopilot 및 이를 제어하는 오프보드(Offboard) 보조 컴퓨터의 ROS 2 환경을 설계할 때 중요한 엔지니어링 지표 중 하나는 컴파일된 공유 라이브러리(Shared Library)가 소비하는 메모리 발자국(Memory Footprint)이다. 이 절에서는 CMake 환경에서 실행 시간 의존성(Runtime Dependency)과 빌드 시간 의존성(Build-time Dependency)을 어떻게 논리적으로 분리하고, 이를 통해 공유 라이브러리의 크기 및 로딩 시간을 획기적으로 줄이는 아키텍처적 접근법을 상세히 기술한다.

1. 의존성(Dependency)의 이원화 개요

현대의 C/C++ 기반 프레임워크(예: ROS 2 rclcpp, PX4 uORB)에서 하나의 패키지(Package)가 다른 패키지의 요소를 사용하려면 의존 관계를 설정해야 한다. 이러한 의존성은 그 쓰임의 생명주기(Lifecycle)에 따라 두 가지로 명확히 분리할 수 있다.

  1. 빌드 시간 의존성(Build-time Dependency): 특정 모듈을 컴파일(Compile)할 때만 필요하며, 최종 링킹(Linking)이 끝나고 실제 런타임에는 동적 링커(Dynamic Linker)에 의해 로드될 필요가 없는 모듈이다. 설계 패턴 관점에서 전방 선언(Forward Declaration)이나 Pimpl 묵시적 포인터 이디엄을 사용할 때, 혹은 헤더-온리(Header-only) 라이브러리를 사용할 때 주로 발생한다.
  2. 실행 시간 의존성(Runtime Dependency): 빌드뿐만 아니라 프로그램이 실제 메모리에 적재되어 실행(Execution)될 때 반드시 동적 링크 라이브러리(.so) 형태로 로드되어야 하는 의존성이다.

1.1 문제점: 무분별한 의존성 전파(Dependency Propagation)

개발 과정에서 편의성을 위해 CMake의 target_link_libraries(target PUBLIC libA) 명령 체계를 남용하면, 해당 타겟(Target)에 의존하는 모든 상위 모듈로 libA 에 대한 실행 시간 의존성이 폭포수(Cascade)처럼 전파된다. 결과적으로 PX4-Autopilot 관제 코어 라이브러리를 하나 로드하는데, 실제 런타임에는 사용되지 않는 수십 개의 불필요한 기호(Symbol)와 동적 라이브러리 파일이 함께 적재되어 엄청난 메모리 오버헤드와 초기 로딩 지연(Latency)을 초래한다.

2. CMake 타겟 속성(Target Properties)을 활용한 논리적 분리

이러한 문제를 방지하고 메모리 풋프린트를 최소화하려면, 빌드 파이프라인(CMakeLists.txt)에서 제공하는 가시성 제어 키워드인 PRIVATE, PUBLIC, INTERFACE를 엄격하게 구분하여 사용해야 한다.

  • PRIVATE: 해당 의존성이 현재 타겟의 구현(Implementation, .cpp 소스 파일) 내에서만 사용될 때 지정한다. 이 의존성은 빌드 시간 의존성으로만 처리되며, 이를 참조하는 외부 타겟으로 의존성이 전파되지 않는다.
  • INTERFACE: 해당 의존성이 구현 코드가 아닌 헤더(Header, .h 또는 .hpp) 에 노출되어 다른 프로젝트들이 이를 인클루드(include)할 때 필요할 경우 선언한다. 이 속성은 자신에게는 링크되지 않고 오직 의존성만을 하위 타겟에 전파한다.
  • PUBLIC: 해당 타겟이 구현과 헤더 양쪽 모두에서 타겟에 의존할 때 사용한다. (PRIVATE + INTERFACE)

3. Pimpl 관용구(Pointer to Implementation) 결합

단순히 CMake 설정 변경으로 끝나면 안 되며, 코드 베이스 상에서 C++의 Pimpl Idiom(Pointer to Implementation) 컴파일 방화벽 메커니즘을 적극적으로 활용해야 논리적 분리가 완성된다.

PX4-Autopilot의 센서 제어용 사용자 정의 클래스의 헤더 파일 내에서 외부 복잡한 서드파티 라이브러리(예: OpenCV 또는 ROS 2 특정 스택)의 헤더를 직접 포함(Include)하는 대신, 불완전 형식(Incomplete Type)을 이용한 전방 선언을 채택하라.

// [나쁜 예] 헤더에서 직접 포함하여 PUBLIC/Runtime 의존성 유발
#include <rclcpp/rclcpp.hpp> 

class Px4RosBridge {
public:
    void init();
private:
    rclcpp::Node::SharedPtr node_; // 불필요한 헤더 노출
};

위의 구조를 Pimpl 구문으로 리팩토링(Refactoring)하면 다음과 같다.

// [좋은 예 - Header] 전방 선언 사용으로 의존성 분리
#include <memory>

class Px4RosBridge {
public:
    Px4RosBridge();
    ~Px4RosBridge();
    void init();
private:
    struct Impl; // 전방 선언 (Forward Declaration)
    std::unique_ptr<Impl> pimpl_;
};

// [좋은 예 - Source (.cpp)] 소스 파일에서만 의존성 포함
#include "px4_ros_bridge.hpp"
#include <rclcpp/rclcpp.hpp>

struct Px4RosBridge::Impl {
    rclcpp::Node::SharedPtr node_;
};

Px4RosBridge::Px4RosBridge() : pimpl_(std::make_unique<Impl>()) {}
Px4RosBridge::~Px4RosBridge() = default;

void Px4RosBridge::init() {
    pimpl_->node_ = std::make_shared<rclcpp::Node>("px4_bridge");
}

4. CMake에서의 논리적 분리 구현

위와 같이 컴파일러 단에서 의존성을 소스 레벨(.cpp)로 완전히 격리시켰다면, CMakeLists.txt 파일에서는 다음과 같이 명시적으로 분리 선언이 가능하다.

add_library(px4_ros_bridge SHARED
    src/px4_ros_bridge.cpp
)

# Implementation 내부에서만 사용하는 라이브러리는 PRIVATE(빌드 시간/내부 의존성)
target_link_libraries(px4_ros_bridge PRIVATE
    rclcpp::rclcpp
)

# 외부로 노출되어야 하는 모듈만 INTERFACE 혹은 PUBLIC 채택
target_link_libraries(px4_ros_bridge INTERFACE
    px4_msgs::px4_msgs
)

5. 결론

위와 같이 소프트웨어 아키텍처 수준에서의 Pimpl 관용구 결합과, 빌드 시스템(CMake) 레벨에서의 링크 가시성 제어(PRIVATE 활용)를 통해, 실행 시간(Runtime) 의존성과 빌드 시간(Build-time) 의존성은 완벽히 논리적으로 격리(Isolation)된다. 이러한 격리 체계는 패키징된 바이너리의 최종 사이즈를 드라마틱하게 감소시키며, 시스템 부팅 및 동적 링킹 단계를 최소화하여 PX4-Autopilot과 같은 경시성(Hard Real-time) 시스템 환경에서 높은 관제 성능 가용성을 보장하는 가장 강력한 아키텍처 설계 지침이라 할 수 있다.
