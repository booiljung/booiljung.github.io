62.3.3. 이산 상태(Discrete State) 모니터링: VehicleStatus 구조체의 논리적 해석 및 전이 상태 추론

무인항공기(UAV)의 제어 시스템에서는 기체의 공간적 위치나 속도 등을 다루는 연속 상태(Continuous State) 추정과 더불어, 논리적인 비행 모드나 시스템 건전성(Health) 상태를 평가하는 **이산 상태(Discrete State)**의 모니터링이 필수적이다. PX4-Autopilot 시스템 내에서 이산 상태는 다양한 개별 플래그(Flag)와 열거형(Enumeration) 값들의 집합 형태로 관리된다.

본 절에서는 uORB 메시지 중 기체의 거시적 이산 상태를 통제하는 최상위 계층 데이터 구조체체인 VehicleStatus 메커니즘을 심층 분석하고, ROS 2 미들웨어 환경에서 기체의 동적 상태 전이를 추론(Inference)하는 설계 기법을 다룬다.

1. 연속 상태(Continuous State)와의 대비 및 이산 상태 모니터링의 중요성

비행 제어 컴퓨터(Flight Controller)의 Estimator(ECL/EKF2) 모듈은 IMU, GPS, 기압계 센서를 융합하여 기체의 위치, 속도, 자세(Quaternion) 등의 연속 상태를 추정한다. 이와 대비되는 이산 상태는 기체의 시동 및 무장(Arming) 여부, 네비게이션 모드(Navigation Mode, 예: Offboard, RTL, Mission), 그리고 고도 제한 붕괴나 통신 두절로 촉발된 각종 페일세이프(Failsafe) 플래그를 포함한다.

이러한 이산 상태 정보는 기체의 비행 권한을 제어하거나 비상 상황에 대처하는 오프보드(Offboard) 알고리즘의 최우선 판단 기준이 된다. 예를 들어, 오프보드 제어기로부터 아무리 정밀한 TrajectorySetpoint(연속 궤적)가 인가되더라도 기체의 이산 상태가 nav_state != NAVIGATION_STATE_OFFBOARD이거나 arming_state != ARMING_STATE_ARMED인 상황에서는 제어 명령이 활성화되지 않는다.

2. VehicleStatus 구조체의 논리 필드 분석

PX4의 VehicleStatus (ROS 2 인터페이스 기준 px4_msgs::msg::VehicleStatus) 구조체는 다음과 같은 핵심 이산 변수를 내포한다.

  • arming_state: 무장 수준. 초기화(INIT), 대기(STANDBY), 무장(ARMED), 에러(ERROR) 등의 단계를 갖는다.
  • nav_state: 기체의 현재 네비게이션 모드. 주요 상태로는 수동(MANUAL), 안정화(STAB), 미션(AUTO_MISSION), 오프보드(OFFBOARD), 복귀(AUTO_RTL), 착륙(AUTO_LAND) 등이 있다.
  • failsafe 및 failsafe_and_user_took_over: 치명적 결함 조치 또는 조종자 개입 상태 표시.
  • rc_signal_lost: 데이터 링크(Radio Control)의 유실 감지 플래그.
  • vehicle_type: 고정익(Fixed-wing), 멀티로터(Rotary-wing), 로버(Rover) 등 구조적 비행 플랫폼 변형을 지시. (VTOL 기체의 경우 비행 중 전환 상태 모니터링에 힌트를 제공)

3. 이산 상태 전이 추론(Inference) 라우팅 논리 - 유한 상태 기계(FSM) 접근

지상 관제 시스템(GCS) 또는 온보드 컴패니언 컴퓨터에 탑재된 자율 에이전트(Autonomous Agent)는 단순한 플래그 값을 1차원적으로 읽는 것에 그치지 않고, 프레임 간 상태의 변화량을 감지하여 “전이(Transition)“를 추출해내야 한다. 이는 유한 상태 기계(Finite State Machine, FSM) 모델에 부합한다.

이를테면 “비행 중 임무 수행에서 조종기 단절에 의한 자동 복귀(RTL)로의 진입“은 다음과 같은 필드의 교차 변화로 확인된다.

  • 이전 상태: nav_state == AUTO_MISSION
  • 현재 상태: nav_state == AUTO_RTL 및 rc_signal_lost == true

이를 자동화된 모니터링 알고리즘으로 시각화하면 아래와 같다.

stateDiagram-v2
    [*] --> STANDBY : 전원 인가 및 부팅 완료
    
    STANDBY --> ARMED : Arming 명령 (Commander 승인)
    ARMED --> AUTO_MISSION : 자동 임무 모드 전환
    ARMED --> OFFBOARD : 자율 비행(ROS 2) 모드 전환
    
    AUTO_MISSION --> AUTO_RTL : 배터리 경고 / 링크 두절
    OFFBOARD --> AUTO_RTL : Heartbeat 0.5초 이상 누락
    OFFBOARD --> STABILIZED : 조종사 오버라이드(RC 개입)
    
    AUTO_RTL --> AUTO_LAND : 착륙 지점 도달
    AUTO_LAND --> STANDBY : Disarming (엔진 정지)
    STANDBY --> [*]

4. ROS 2 환경 기반 이산 상태 추론 알고리즘 설계 방안 (C++ 구현 구조)

ROS 2의 VehicleStatus 서브스크라이버를 선언하고, 콜백의 매 프레임마다 이전 상태 값(Previous State)과 최신 상태 값(Current State)을 배타적 논리합(XOR) 또는 비교 연산하여, 상태 전이의 에지(Edge)를 점출하는 Observer 설계를 적용해야 한다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_status.hpp>

class DiscreteStateObserver : public rclcpp::Node {
public:
    DiscreteStateObserver() : Node("discrete_state_observer") {
        // 초기 상태를 미정(UNKNOWN)으로 세팅
        prev_status_.nav_state = px4_msgs::msg::VehicleStatus::NAVIGATION_STATE_MAX; 
        prev_status_.arming_state = px4_msgs::msg::VehicleStatus::ARMING_STATE_MAX;

        status_sub_ = this->create_subscription<px4_msgs::msg::VehicleStatus>(
            "/fmu/out/vehicle_status",
            rclcpp::QoS(1),     // 최신 이산 상태만 중요하므로 Depth=1
            std::bind(&DiscreteStateObserver::status_callback, this, std::placeholders::_1)
        );
    }

private:
    void status_callback(const px4_msgs::msg::VehicleStatus::SharedPtr msg) {
        // 1. 네비게이션 모드 전이(Transition) 감지
        if (prev_status_.nav_state != msg->nav_state) {
            handle_nav_state_transition(prev_status_.nav_state, msg->nav_state);
        }

        // 2. 무장 상태(Arming) 전이 감지 (상승 엣지 및 하강 엣지)
        if (prev_status_.arming_state != msg->arming_state) {
            if (msg->arming_state == px4_msgs::msg::VehicleStatus::ARMING_STATE_ARMED) {
                RCLCPP_INFO(this->get_logger(), "[Event] Vehicle Armed.");
            } else if (msg->arming_state == px4_msgs::msg::VehicleStatus::ARMING_STATE_STANDBY) {
                RCLCPP_INFO(this->get_logger(), "[Event] Vehicle Disarmed.");
            }
        }

        // 상태 병합
        prev_status_ = *msg;
    }

    void handle_nav_state_transition(uint8_t old_state, uint8_t new_state) {
        // 예시: 자율 비행 권한을 상실했을 때의 예외 처리 루틴
        if (old_state == px4_msgs::msg::VehicleStatus::NAVIGATION_STATE_OFFBOARD && 
            new_state != px4_msgs::msg::VehicleStatus::NAVIGATION_STATE_OFFBOARD) {
            RCLCPP_WARN(this->get_logger(), "CRITICAL: Offboard control authority lost. Nav State -> %d", new_state);
            // 궤적 발행 노드의 타이머 정지 혹은 자율 임무 중단 로직 전개
        }
    }

    rclcpp::Subscription<px4_msgs::msg::VehicleStatus>::SharedPtr status_sub_;
    px4_msgs::msg::VehicleStatus prev_status_; // 이전 사이클의 이산 상태 보존
};

위의 아키텍처는 매 콜백 호출 시점을 트리거(Trigger)로 사용하여, 논리 연산을 통해 오직 전이 시점에만 행동 방침을 결단하는(Event-triggered Action) 구조를 갖는다. 이를 고도화하면, 상태 천이 정보만을 추려서 별도의 이산형 이벤트 토픽(Event Topic)으로 추상화시켜 발행(Publish)하는 시스템 상위 진단 모듈로 발전시킬 수 있다.
