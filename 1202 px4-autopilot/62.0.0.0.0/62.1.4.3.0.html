62.1.4.3 마감 시간(Deadline) 및 활성 상태(Liveliness) 임계치 설정을 통한 결함 검출(Fault Detection) 시간 최소화 기법

하드 리얼타임(Hard Real-Time) 기반의 무인항공기 비행 제어에서 가장 치명적인 사고 원인 중 하나는 ‘침묵하는 실패(Silent Failure)’ 현상이다. 즉, 지상 관제 시스템(GCS) 통신 링크 단절, ROS 2 측 비전 컴퓨터의 전원 차단, 또는 프로세스 교착 상태(Deadlock)가 발생하여 제어 지시가 끊겼음에도 불구하고, 비행 제어기(PX4-Autopilot)가 이를 신속하게 감지하지 못해 기체가 기존 경로로 무한정 비행하는 사태를 뜻한다. uXRCE-DDS 구조에서는 이러한 결함을 즉각적으로 식별하고 예방 및 대응(Failsafe) 메커니즘을 가동하기 위해 **마감 시간(Deadline)**과 활성 상태(Liveliness) QoS 정책의 임계치(Threshold)를 정밀하게 설정한다.

1. 마감 시간(Deadline) QoS 방어 기법

마감 시간(Deadline) 정책은 주기적(Periodic)으로 발생해야 하는 데이터가 사전에 정의된 특정 시간 내에 반드시 발행되어야 함을 약속(Contract)하는 기능이다.

1.1 마감 시간 정책 아키텍처

퍼블리셔(Publisher)와 서브스크라이버(Subscriber)는 각각 자신의 최대 허용 마감 시간 주기를 선언한다.

  • 퍼블리셔의 약속: “나는 최소한 시간 T_{pub} 안에는 새로운 데이터를 발행하겠다.”
  • 서브스크라이버의 요구: “나는 최소한 시간 T_{sub} 안에는 새로운 데이터를 받아야만 한다.”

데이터 통신이 성립되기 위해서는 T_{pub} \le T_{sub} 조건이 성립해야 한다. 만약 퍼블리셔가 지정된 T 시간 내에 메시지 퍼블리싱을 실패하면, 미들웨어는 데이터 스레드에 Deadline Missed 상태 이상(Event) 알림을 트리거하며, 애플리케이션 계층(PX4 Failsafe 모듈이나 ROS 2 Safety 로직)은 즉시 안전 복구 기동(Return to Launch, Hovering 등)에 진입하게 된다.

1.2 센서 처리 시간 최소화를 위한 임계치 산출

비전 기반 장애물 회피(VIO) 알고리즘이 초당 20회(20Hz, 주기 50ms)의 빈도로 경로 제어 명령(trajectory_setpoint)을 PX4에 송신한다고 가정해 보자.
단순히 주기와 마감 시간을 50ms로 일치시키면 네트워크 미세 지터(Jitter)로 인해 과도한 거짓 알림(False Alarm)이 빗발칠 수 있다. 따라서 임계치(T_{Deadline}) 산출 시 센서 주기 여유율(\beta)을 더하여 아래와 같이 설정한다.

T_{Deadline} = \frac{1}{Frequency} \times (1 + \beta)

일반적으로 \beta = 0.5를 부여하여, 50ms \times 1.5 = 75ms를 임계치로 구성함으로써 결함 감지 시간과 시스템 탄력성의 비율을 최적화한다.

2. 활성 상태(Liveliness) 모니터링 기법

센서 데이터처럼 계속해서 갱신되지 않는 1회용 설정값이나 드문 제어 명령의 경우, 마감 시간(Deadline) 방식으로는 통신 단절을 판별하기 어렵다. 이 때 노드의 생존 여부를 모니터링하는 것이 활성 상태(Liveliness) QoS 정책이다.

2.1 Liveliness 정책의 종류

  1. AUTOMATIC (자동): 가장 흔하고 가벼운 방식으로, 퍼블리셔가 속한 노드가 네트워크 상에 존재하기만 하면 미들웨어가 묵시적으로 살아있다고 판단한다. 미들웨어 계층의 백그라운드 스레드가 하트비트(Heartbeat)를 전송한다.
  2. MANUAL_BY_PARTICIPANT / TOPIC (수동): 애플리케이션 로직 내부에서 개발자가 특정 토픽이나 노드가 살아있음을 명시적으로(Explicitly) 선언하는 assert_liveliness() API를 호출해야만 활성 상태로 인정받는다.

2.2 결함 검출(Fault Detection) 시간 최소화를 위한 적용

고정익이나 멀티로터의 자율 편대비행(Swarm) 환경에서 특정 기체의 물리적 교착 상태(예: 메인 루프 데드락으로 인한 프로그램 정지)를 검출하기 위해 AUTOMATIC 방식은 적절하지 않다. 미들웨어 통신 칩셋 모듈은 살아있어서 데이터를 응답하지만 중앙 루프가 멎어있을 수 있기 때문이다.

따라서 PX4-ROS 2 구조에서는 MANUAL_BY_TOPIC 속성과 근접 임계 주기 단위의 하트비트를 조합한다.

  • 구독자(GCS 또는 관제 ROS 2 노드)는 Liveliness 주기 지속 시간(Lease Duration)을 300ms 등 기체의 파괴 한계 시간(Crash-limit time) 이내로 설정한다.
  • PX4의 센서 루프는 100ms마다 assert_liveliness를 발급한다.
  • 만약 메인 루프에 정지(Freeze)가 발생하면 발급이 멈추고 제어 시스템은 Lease Duration 만료(Expired) 직후 300ms 안에 즉각적인 비상 조치(Flight Termination 등)를 발령하여 결함이 전파되는 시간을 최소화할 수 있다.

결론적으로 마감 시간과 활성 상태 QoS는 하드 리얼타임 드론 제어 환경에서 “소프트웨어와 네트워크 신뢰성을 어떻게 측정하고 의심할 것인가“에 대한 가장 수학적이고 결정론적인 안전장치로서 작동한다.
