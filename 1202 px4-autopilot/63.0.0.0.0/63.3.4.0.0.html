63.3.4. Offboard 타임아웃 파라미터(COM_OF_LOSS_T)와 네트워크 토폴로지 단절 감지 및 대처 메커니즘

무인항공기의 분산 제어 시스템(Distributed Control System)에서 지상 관제 시스템(GCS)이나 동반 컴퓨터(Companion Computer)와 비행 제어기(FC: Flight Controller)를 잇는 통신 링크는 가장 취약한 단일 장애점(SPOF: Single Point of Failure)이다. PX4-Autopilot(v1.14 이상)은 이 데이터 채널의 생명 유무를 감시하고, 네트워크 단절 시 기체의 생존을 도모하기 위해 COM_OF_LOSS_T 파라미터를 식역값(Threshold)으로 하는 ‘오프보드 타임아웃(Offboard Timeout)’ 감시 메커니즘을 펌웨어 안쪽에 치밀하게 엮어두었다. 본 절에서는 해당 파라미터의 제어 공학적 의미와 비행 제어기 단에서 발동되는 비상 안전(Failsafe) 대처 메커니즘 구조를 규명한다.

1. COM_OF_LOSS_T의 정의와 워치독(Watchdog) 기반 단절 감지

Ardupilot 통제 체계는 통상적으로 GCS Heartbeat 타임아웃(FS_GCS_ENAB)이나 원격 조종기(Radio Control) 신호 유실 등에 대비한 통합 페일세이프(Failsafe) 룰을 적용한다. Ardupilot의 GUIDED 모드 상태에서도 목적지에 도달하면 단순히 정지하여 대기하는 특성이 짙다. 반면, PX4의 오프보드(Offboard) 모드 아키텍처는 에이전트 노드와의 고빈도 주기적 텔레메트리(High-frequency Periodic Telemetry) 갱신을 생명력의 증거로 삼으며, 이에 특화된 강건한 워치독(Watchdog) 타이머를 구동한다.

1.1 오프보드 제어 스트림 감시 로직

PX4 커맨더(Commander) 내부에는 오프보드 모드 진입과 동시에 백그라운드에서 돌아가는 데드맨 스위치(Dead-man’s Switch)가 활성화된다.

  • 이 타이머의 제한 시간은 파라미터 COM_OF_LOSS_T(기본값 0.5\text{초})로 정의된다.
  • uORB 버스를 통해 TrajectorySetpoint 또는 OffboardControlMode 메시지가 하나라도 갱신되는 순간, 이 내부 타이머는 0\text{초}로 리셋(Reset)된다.
  • 만약 와이파이(Wi-Fi) 간섭, ROS2 미들웨어 QoS 버퍼 마비, 커스텀 USB 케이블 손상 등 물리적/논리적 네트워크 토폴로지의 단절이 발생하여 타겟 업데이트가 COM_OF_LOSS_T의 임계 간격을 초과하게 되면, FC 펌웨어는 더 볼 것도 없이 현 제어자를 축출하고 오프보드 모드 권한을 즉각 회수(Revoke) 한다.

2. 타임아웃 감지 이후의 오토파일럿 대처 메커니즘 (Failsafe Action)

데이터 단절 감지는 단순히 경고 로그(CRITICAL: Offboard control lost)를 띄우는 것으로 끝나지 않는다. FC의 스케줄러는 미리 약속된 비행 모드로 강제 상태 천이(State Transition)를 감행하여 기체의 생존을 보장하는 자구책을 편다.

2.1 예외 처리 우회 경로 (Bypass Route)

COM_OF_LOSS_T 타이머가 만료되는 순간 작동하는 대처 시나리오는 파라미터 설정과 기체의 현재 상태(고도, GPS 픽스 등)에 따라 계층적으로 발동한다.

  1. 지상 대기 중인 경우 (Landed 상태): 오프보드 대기 중에 스트림이 끊기면 평온하게 모드를 해제(Disarm) 시키거나 Hold 모드로 진입한다.
  2. 비행 중인 경우 (In-air 상태):
  • Hold 모드 강제 전이: 가장 일반적인 기본 방어책이다. 기체는 과거의 관성(Inertia)이나 조향 각도를 즉각 폐기하고(Brake), 현재 3D 공간의 GPS 좌표에 고정(Hold) 호버링하여 에이전트 노드가 네트워크 접속을 복구할 때까지 그 자율 비행을 일시 정지 상태로 동결시킨다.
  • Return 모드 복귀: 만약 통신 단절이 구조적/영구적이라고 판단되는 경우(예: GCS Heartbeat Loss와 동시다발적 발생), 또는 파라미터 NAV_DLL_ACT(Data Link Loss Action)가 Return to Launch(RTL)로 설정된 경우, 기체는 즉시 안전 고도로 상승한 후 이륙 지점이나 사전에 지정된 홈(Home) 포인트로 귀환 시퀀스를 가동한다.

2.2 제어 공학적 지연 보상 한계선

왜 하필 기본값이 0.5\text{초}(2\text{Hz})일까? 앞서 영차 파지(Zero-Order Hold) 모델에서 살펴보았듯, 위치 제어의 대역폭 한계를 고려할 때 마지막 제어 지령을 반 초(500\text{ms}) 이상 맹신하여 기체를 그대로 날려 보내는 것은, 드론이 통제 불능의 발사체(Uncontrolled Projectile)가 되는 치명적 런어웨이 궤도로 빠지는 마지노선(Deadline)이기 때문이다.

결과적으로 PX4의 COM_OF_LOSS_T 파라미터 매핑 아키텍처는, 무선 매체나 스레드 공유 환경 등 신뢰할 수 없는 분산 컴퓨팅(Distributed Computing) 시스템을 믿고 하늘을 날아야 하는 드론에게 부여된 마지막 **동적 무결성(Dynamic Integrity) 이스케이프 해치(Escape Hatch)**라 할 수 있다.
