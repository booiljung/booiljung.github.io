63.8.1. ROS2 네임스페이스(Namespace) 지정자를 활용한 개별 물리 기체 데이터 스트림 분할 및 논리적 객체 지향 트리 구조 매핑

군집 비행 시스템을 구축함에 있어 가장 초기에 직면하는 과제는 단일 네트워크(DDS Domain) 내호에 수많은 기체가 쏟아내는 수백 가지의 uORB 토픽(Topic)이 상호 충돌(Collision)하지 않도록 분리하는 작업이다. 만약 다용도의 비행 기체가 데이터 스트림의 식별자 없이 /fmu/out/vehicle_odometry와 같은 기본 토픽명으로 퍼블리싱한다면, 관제 노드는 어떤 데이터가 어느 기체로부터 왔는지 구분할 수 없게 되어 치명적인 자세 제어(Attitude Control) 오작동을 유발한다.

본 절에서는 PX4-Autopilot v1.14.x와 ROS 2 Humble 환경을 기반으로, ROS 2의 핵심 식별 기능인 네임스페이스(Namespace) 지정자를 활용하여 각 기체의 데이터 스트림을 논리적인 트리(Tree) 구조로 매핑하고, 단일 소스 베이스를 통해 확장이 용이한 다중 에이전트 객체 지향 노드 작성 기법을 서술한다.

1. uORB-XRCE-DDS 브릿지와 네임스페이스 환경 매핑 원리

일반적인 MAVLink 망에서는 MAV_SYS_ID 파라미터를 1, 2, 3과 같이 할당하여 고유 식별자로 활용한다. 이와 비교하여 ROS 2 통신망은 데이터 분포 서비스(DDS, Data Distribution Service)를 토대로 하므로, 메시지 프레임 레벨이 아닌 토픽 URL 경로 자체에 접두어(Prefix) 형태의 네임스페이스를 삽입하여 도메인 간섭을 원천적으로 차단한다.

SITL(Software In The Loop, Gazebo 시뮬레이션) 환경에서는 PX4_SYS_AUTOSTART와 인스턴스 번호 -i 플래그를 통해 자동으로 /px4_1/fmu/in/, /px4_2/fmu/in/ 등의 네임스페이스 트리 구조가 생성된다. 반면, 물리적인 실제 기체(Hardware In The Loop)나 컴패니언 컴퓨터(Companion Computer)에서는 Micro XRCE-DDS 클라이언트(Client) 프로세스를 구동할 때 명시적으로 네임스페이스 파라미터를 전달하여야 한다.

PX4의 콘솔(NuttX 터미널 혹은 텔레메트리 쉘)에서 XRCE-DDS 에이전트를 개시할 때 uxrce_dds_client start -t udp -p 8888 -h 192.168.0.100 -n uav1 과 같이 -n 옵션을 명시하게 되면 uav1 이라는 고유 네임스페이스가 DDS Global Domain 상의 프리픽스로 일관되게 적용된다.

2. ROS 2 네임스페이스 지정자를 적용한 통신 트리 구조

명시적 네임스페이스를 적용하게 되면, GCS 혹은 중앙 집중 군집 제어기는 ROS 2 토픽의 디렉터리 경로만으로 객체(기체)를 논리적으로 분류할 수 있게 된다. 이를 도식화하면 다음과 같은 객체 지향적 트리 구조를 형성하게 된다.

이러한 수직적 논리 매핑은 노드 작성 시 대상(Target)을 개별적인 객체(Object) 인스턴스로 취급할 수 있도록 만들어주는 기반이 된다.

3. [실습] C++ 기반의 객체 지향적 다중 기체 제어 노드 캡슐화

ROS 2(rclcpp)에서 여러 기체를 조종하기 위한 코드를 작성할 때, 하드코딩된 토픽 이름(/uav1/fmu/in/...)을 여러 번 복사하여 사용하는 것은 설계적 관점에서 타당하지 않다. 효율적인 군집 제어 스크립트를 위해서는 싱글턴(Singleton)이나 단일 클래스를 정의한 뒤 노드의 생성자 인자(Constructor Argument)로 네임스페이스 식별자를 주입해주는 템플릿(Template)화된 설계를 권장한다.

아래는 단독 C++ 소스 코드를 통해 다수의 PX4 기체 상태를 모니터링하고 제어 집합점을 인가하는 객체 지향 구조 실습 코드이다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/vehicle_status.hpp>
#include <px4_msgs/msg/trajectory_setpoint.hpp>
#include <string>
#include <memory>
#include <vector>

using namespace std::chrono_literals;

// 개별 기제(에이전트)의 동작을 추상화한 단일 제어 클래스 캡슐화
class SingleAgentNode : public rclcpp::Node
{
public:
    // 생성자에서 namespace 이름을 주입받음으로써 독립적인 노드로 작동하게 함.
    explicit SingleAgentNode(const std::string& vehicle_namespace) 
    : Node(vehicle_namespace + "_offboard_node")
    {
        // 네임스페이스를 포함하여 절대 경로로 토픽 동적 매핑
        std::string status_topic = "/" + vehicle_namespace + "/fmu/out/vehicle_status";
        std::string setpoint_topic = "/" + vehicle_namespace + "/fmu/in/trajectory_setpoint";

        status_sub_ = this->create_subscription<px4_msgs::msg::VehicleStatus>(
            status_topic, 10,
            [this, vehicle_namespace](const%20px4_msgs::msg::VehicleStatus::SharedPtr%20msg) {
                RCLCPP_INFO(this->get_logger(), "[%s] Vehicle Mode: %d | Arming State: %d", 
                            vehicle_namespace.c_str(), msg->nav_state, msg->arming_state);
            });

        setpoint_pub_ = this->create_publisher<px4_msgs::msg::TrajectorySetpoint>(
            setpoint_topic, 10);

        // 주기적 제어 루프
        timer_ = this->create_wall_timer(
            100ms, std::bind(&SingleAgentNode::control_loop, this));
            
        RCLCPP_INFO(this->get_logger(), "Agent %s initialized.", vehicle_namespace.c_str());
    }

private:
    rclcpp::Subscription<px4_msgs::msg::VehicleStatus>::SharedPtr status_sub_;
    rclcpp::Publisher<px4_msgs::msg::TrajectorySetpoint>::SharedPtr setpoint_pub_;
    rclcpp::TimerBase::SharedPtr timer_;

    void control_loop()
    {
        // 실제 스웜 제어 물리 로직이 구현되는 구간 (로컬 좌표 변환 등)
        px4_msgs::msg::TrajectorySetpoint msg;
        msg.timestamp = this->get_clock()->now().nanoseconds() / 1000ULL;
        // 예시로 각 기체마다 NaN으로 초기화하여 안전 상태 대기
        msg.position = {NAN, NAN, NAN};
        msg.yaw = NAN;
        setpoint_pub_->publish(msg);
    }
};

int main(int argc, char **argv)
{
    rclcpp::init(argc, argv);

    // ROS 2의 Multi-Threaded Executor를 사용하여 다중 에이전트를 동시 비동기 스케줄링 처리
    rclcpp::executors::MultiThreadedExecutor executor;

    // uav1, uav2, uav3 객체 인스턴스를 동적으로 생성하여 병렬 확장
    std::vector<std::string> swarm_list = {"uav1", "uav2", "uav3"};
    std::vector<std::shared_ptr<SingleAgentNode>> node_list;

    for (const auto& uav_name : swarm_list) {
        auto node = std::make_shared<SingleAgentNode>(uav_name);
        node_list.push_back(node);
        executor.add_node(node);
    }

    // 모든 기체의 이벤트를 단일 메인 루프에서 멀티스레드로 처리
    executor.spin();
    rclcpp::shutdown();
    return 0;
}

위 코드가 보여주듯, 네임스페이스 기반의 객체 캡슐화 기법을 사용하면 군집 비행기 대수(swarm_list)가 100대로 증가하더라도 상용 비즈니스 로직 수정 없이 MultiThreadedExecutor에 노드만 동적으로 추가(Add)하여 할당할 수 있다. 이는 고도의 재사용성(Reusability)과 객체 지향성(Object-Oriented)을 제공한다.

4. ArduPilot과의 네임스페이스 철학 차이 및 관제(GCS) 관점에서의 비교

ArduPilot 관점 하의 Mission Planner나 QGroundControl에서는 일반적으로 단일 시리얼 패킷 통로(RF 모뎀 등)에 멀티플렉싱(Multiplexing)된 MAVLink 스트림이 도달한다. GCS 내부의 모듈은 패킷 헤더(Header)의 시스템 식별자(SYSID_THISMAV) 필드를 필터링하여 기별 상태 창(UI)을 복제(Cloning)해서 그려낸다. 즉, 통신망 자체가 논리적으로 분할된 것이 아니라 수신자(Receiver) 입장에서의 디코딩(Decoding) 필터에 의존하는 구조다.

반대로 PX4와 ROS 2 브릿지 연동 생태계에서는 퍼블리셔(기체) 단에서부터 네임스페이스 지정자를 달고 DDS 통신 도메인 내의 격리된 파이프(Pipe)를 선점하게 된다. 따라서 관제 응용 프로그램(Custom GCS Node)은 필터링 로직 없이도 자신이 듣고 싶은 특정 네임스페이스 트리 구조만 즉각적으로 엮거나 그룹핑(Grouping)할 수 있는 탁월한 망 분리성을 누리게 된다. 이는 무선 트래픽이 기하급수적으로 폭증하는 Swarm 제어 환경 하에 네트워크 토픽 충돌을 운영 체제(OS) 수준에서 사전 차단하는 극단적인 이점을 제공한다.

5. 결론

다수 기체의 군집 비행(Swarm) 제어를 온전하게 이루어 내기 위해서는 물리적인 기체의 하드웨어 자원을 논리적 데이터 트리로 정확히 투영(Mapping)하는 과정이 선행되어야 한다. PX4-Autopilot 및 ROS 2 미들웨어 환경은 Micro XRCE-DDS 에이전트의 구동 인자를 통해 네임스페이스를 일원화 지원하며, 제어 노드의 객체 지향적(Object-Oriented) 캡슐화를 통해 소프트웨어의 단일 구조 유지보수와 극적인 확장성을 동시에 보장한다. 이어지는 편대 비행 시분할 제어 실습에서도 이러한 네임스페이스 트리의 분리 개념을 심층적으로 활용하게 된다.
