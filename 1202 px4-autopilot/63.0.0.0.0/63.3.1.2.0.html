63.3.1.2. 비동기 콜백(Asynchronous Callback) 동시성(Concurrency) 환경에서의 임계 구역(Critical Section) 경쟁 상태(Race Condition) 방지 및 상호 배제(Mutex) 잠금 기법

PX4-Autopilot(v1.14 이상)을 오프보드(Offboard) 모드에서 정밀 제어하기 위해 ROS2 프레임워크를 기반으로 동반 컴퓨터(Companion Computer) 노드(Node)를 설계할 때, 필연적으로 직면하는 가장 난해한 소프트웨어 공학적 과제는 ’동시성(Concurrency) 제어’이다. 센서 피드백을 수신하는 서브스크라이버(Subscriber) 콜백(Callback) 스레드와, 제어 지령(TrajectorySetpoint)을 최적화하여 퍼블리싱(Publishing)하는 타이머 스레드가 동일한 상태 변수(State Variable)를 점유하기 위해 다투는 병렬 처리(Parallel Processing) 환경이 조성되기 때문이다. 본 절에서는 이 비동기 콜백 구조에서 발생하는 경쟁 상태(Race Condition)의 위험성을 규명하고, 임계 구역(Critical Section)을 보호하기 위한 상호 배제(Mutex: Mutual Exclusion) 기법의 올바른 적용 방안을 다룬다.

1. ROS2 콜백 그룹(Callback Group)과 공유 메모리 동시 접근의 위험성

Ardupilot 환경의 커스텀 MAVLink 스크립트 작성 시에는 단일 스레드(Single-thread) 기반의 순차적(Sequential) 폴링(Polling) 방식이 주를 이루어 동시성 문제에서 비교적 자유롭다. 반면, ROS2의 rclcpp/rclpy 아키텍처는 효율적인 I/O 처리를 위해 MultiThreadedExecutor를 통해 여러 콜백을 스레드 풀(Thread Pool)에 분산시켜 비동기적으로(Asynchronously) 실행한다.

1.1 오프보드 제어기 내 임계 구역(Critical Section)의 형성

일반적인 궤적 제어 노드는 최소 두 개의 핵심 스레드로 분기된다.

  1. 수신 스레드: 상향 링크(Uplink) 토픽인 VehicleOdometry나 VehicleStatus가 도달할 때마다 콜백 함수가 트리거되어 기체의 전역 상태 변수 current_pose를 갱신한다.
  2. 발행 스레드: 50\text{Hz} 주기의 WallTimer 콜백이 이 current_pose를 참조하여 다음 목표 궤적을 연산하고 TrajectorySetpoint를 하향(Downlink)으로 송출한다.

이때 current_pose 구조체가 메모리 상에서 갱신(Write)되는 도중에 타이머 스레드가 이를 부분적으로 참조(Read)하게 되면, 소위 찢어진 읽기(Torn Read)라 불리는 치명적인 데이터 오염(Data Corruption)이 발생한다. 이 오염된 좌표가 비행 제어기(FC)로 주입될 경우 기체는 불연속적인 위치 입력으로 인해 순간적인 저크(Jerk)를 발생시키거나 제어 발산(Divergence)에 빠질 수 있다.

2. 상호 배제(Mutex) 기반의 락(Lock) 메커니즘 설계

이러한 경쟁 상태(Race Condition)를 원천 차단하기 위해서는 운영체제(OS) 수준에서 제공하는 락(Lock) 매커니즘을 동원하여 해당 변수 영역을 임계 구역(Critical Section)으로 격리해야 한다.

2.1 C++ std::mutex 및 std::lock_guard의 적용 지침

ROS2 C++ 노드 개발 시 다중 스레드 환경에서 가장 안전하게 채택할 수 있는 표준 방법론은 RAII(Resource Acquisition Is Initialization) 패턴에 기반한 std::lock_guard의 활용이다.

  • 잠금의 단위(Granularity): 콜백 함수 내부 전체를 락(Lock)으로 걸어버리면(Coarse-grained Lock) 멀티스레드의 장점이 퇴색되고 병목(Bottleneck)이 발생하여 50\text{Hz} 주기를 맟추지 못하는 데드라인 미스(Deadline Miss)가 발생한다. 락은 오직 current_pose 변수를 복사(Copy)하는 수 마이크로초의 찰나에만 적용되어야 한다(Fine-grained Lock).
  • 데이터 방어(Data Defense): 수신 스레드에서는 mutex.lock() 획득 직후 구조체를 업데이트하고 반환하며, 타이머 스레드 역시 제어기 연산(Controller Maths)을 수행하기 이전에 락을 걸어 안전한 지역 변수(Local Variable)로 상태를 깊은 복사(Deep Copy)해 온 뒤 락을 즉시 해제(unlock)하여 다른 스레드가 멈추지 않도록 설계해야 한다.

2.2 데드락(Deadlock)과 락 없는(Lock-free) 자료구조 대안

서로 다른 두 개의 뮤텍스가 역순으로 호출될 때 발생하는 교착 상태(Deadlock)는 비행 중인 기체에게 즉각적인 추락(Crash)을 의미한다. 이를 컴파일타임 및 런타임에 방지하기 위해 스마트 포인터(std::shared_ptr) 기반의 원자적 교환(Atomic Swap) 메커니즘이나 락 프리(Lock-free) 큐(Queue)를 도입하는 것이 최신 자율 비행 소프트웨어 공학의 트렌드이다.

결론적으로 PX4의 오프보드 제어 성능은 미분 방정식(Differential Equation)의 해를 찾는 수학적 완벽함뿐만 아니라, 컴퓨터 과학(Computer Science)의 근본 과제인 메모리 스레드 안전성(Thread Safety)이 담보될 때 비로소 실제 하드웨어의 생존성으로 직결된다.
