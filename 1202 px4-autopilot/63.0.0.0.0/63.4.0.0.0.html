63.4. ROS2 기반 Offboard 제어 노드(Node) 설계 및 멀티스레딩(Multi-threading) 프로그래밍

본 절에서는 ROS 2 환경에서 PX4-Autopilot의 Offboard 제어 권한을 획득하고 차량을 자율적으로 조종하기 위한 제어 노드(Node)의 설계 원리와 멀티스레딩(Multi-threading) 프로그래밍 기법을 상세히 다룬다. 본 내용은 PX4 v1.14.0 펌웨어와 ROS 2 Humble 버전 환경을 기준으로 기술한다.

1. ROS 2 Offboard 제어 노드의 기본 아키텍처

ROS 2(Robot Operating System 2)를 사용하여 PX4를 제어하는 시스템은 마이크로 XRCE-DDS(Micro XRCE-DDS) 미들웨어를 통해 PX4의 uORB 메시지와 통신한다. 여기서 중심 역할을 수행하는 독립적인 실행 단위를 제어 노드(Control Node)라 명명한다.

제어 노드는 기본적으로 두 가지 핵심 기능을 동시에 수행해야 한다.
첫째, PX4의 현재 상태(위치, 속도, 자세, 배터리 상태 등)를 고속으로 구독(Subscribe)하여 추정기(Estimator) 상태를 파악해야 한다.
둘째, 주기적인 제어 명령(설정점, Setpoint)과 상태 기계(State Machine) 전환 명령을 10Hz 이상의 고주파수(High Frequency)로 발행(Publish)하여 Offboard 모드가 해제되지 않게 강제 유지해야 한다.

따라서 단일 노드 안에서 다수의 구독자(Subscriber)와 발행자(Publisher)가 필연적으로 공존하게 되며, 이들 간의 실행 스케줄링(Scheduling)을 정밀하게 통제하는 것이 시스템 안정성의 핵심 지표가 된다.

2. 단일 스레드(Single-thread) 한계와 멀티스레딩 환경의 필요성

ROS 2 노드는 기본적으로 단일 스레드 실행자(Single-threaded Executor)에 의해 관리된다. 단일 스레드 환경에서는 들어오는 메시지 콜백(Callback), 타이머(Timer) 콜백 등이 큐(Queue)에 쌓여 도착 순서대로 순차적으로 처리된다. 만약 특정 비전 처리 알고리즘이나 상태 측정 콜백 처리에 병목(Bottleneck)이 발생하여 지연(Latency)이 생기면, 다음과 같은 중대한 치명적 한계에 직면한다.

  1. Offboard 모드 유지 실패: 제어 루프가 지연되어 명령 발행 타이머 콜백이 지정된 주파수(예: 10Hz 이상)를 만족시키지 못하면 시그널 손실(Signal Loss)로 간주되어 기체는 즉각 RTL(Return to Launch)이나 호버링(Hovering) 등 비상 안전 모드(Failsafe)로 전환된다.
  2. 제어 지터(Jitter) 발생: 콜백 실행 주기가 일정하지 않고 파동(Fluctuation)을 일으키면, 위치 제어 및 자세 제어의 불안정성을 초래하여 드론 비행 중 진동(Oscillation) 현상을 유발할 위험이 있다.

이러한 한계를 극복하기 위해 다중 스레드(Multiple Threads)를 통해 여러 콜백을 병렬적으로 분산하여 실행하는 패러다임이 요구된다. ROS 2의 MultiThreadedExecutor를 도입하여 독립된 실행 스케줄과 문맥 교환(Context Switching) 환경을 선제적으로 설계하라.

3. 다중 스레드 실행자(Multi-threaded Executor) 시스템 아키텍처

ROS 2의 스케줄링 및 콜백 실행 계층 메커니즘은 다음과 같이 구조화할 수 있다.

graph TD
    subgraph Multi-Threaded Executor
    Thread1[Worker Thread 1]
    Thread2[Worker Thread 2]
    Thread3[Worker Thread 3]
    end

    subgraph Control Node
        CB_Grp_MutuallyExclusive[Mutually Exclusive Callback Group]
        CB_Grp_Reentrant[Reentrant Callback Group]
        
        Timer_Control[High-frequency Control Timer] --> CB_Grp_MutuallyExclusive
        Sub_Odometry[Odometry Subscriber] --> CB_Grp_Reentrant
        Sub_Sensor[Camera/Sensor Subscriber] --> CB_Grp_Reentrant
    end

    CB_Grp_MutuallyExclusive --> Thread1
    CB_Grp_MutuallyExclusive --> Thread2
    CB_Grp_Reentrant --> Thread2
    CB_Grp_Reentrant --> Thread3

설계 시 각 콜백을 어떠한 그룹에 배치할 것인지 명확히 산정하라.

  • 상호 배타적 콜백 그룹 (Mutually Exclusive Callback Group): 그룹 내 소속된 콜백들이 절대로 동시에 실행되지 않음을 보장한다. 순서 의존성이 높거나, 동일한 데이터 구조체에 변경을 가할 때 경합 조건(Race Condition)을 원천적으로 차단할 목적에 적합하다. 안전 비행 루프 타이머에 우선 적용하라.
  • 재진입 가능 콜백 그룹 (Reentrant Callback Group): 스레드 자원이 가용하다면 그룹 내 콜백들이 병렬로 동시 실행되는 것을 허용한다. 데이터의 변경을 수반하지 않는 독립적인 상태 감시나 대용량 센서 데이터 처리에 대응할 목적으로 사용하라.

4. 멀티스레딩 노드 프로그래밍 구현 전략

멀티스레딩을 적용하여 C++ rclcpp API 기반 혹은 Python rclpy 기반으로 Offboard 제어 노드를 설계할 때는 아래의 엔지니어링 설계 규칙을 철저히 준수해야 한다.

  1. 콜백 분리 설계: 고주파 타임 크리티컬(Time-critical) 제어명령 발행 콜백(예: 50Hz 제어 루프)과, 연산 소요 시간이 불명확한 수신 콜백(예: 객체 딥러닝 인식 프로세스)을 완벽히 분리된 서로 다른 상호 배타적 콜백 그룹에 할당하여, 통신 대기열 비대에 의한 실시간 제어 지연을 방어하라.
  2. 데이터 경합 보호 설계: 두 개 이상의 서로 다른 스레드가 동일한 클래스 멤버 변수(Member Variable)에 읽기/쓰기를 동시에 시도한다면, 메모리 충돌을 피비 위해 반드시 표준 라이브러리의 std::mutex 또는 std::shared_mutex를 활용한 락(Lock) 매커니즘을 부여하여 스레드 안전성(Thread Safety)을 확보하라.
  3. 오토 락 체계 도입: 자원의 획득은 곧 초기화원칙(RAII, Resource Acquisition Is Initialization)에 입각하여 std::lock_guard 문법을 적극 활용함으로써, 런타임 예외 상황 발생 시 락 해제 누수로 인한 교착 상태(Deadlock)를 체계적으로 방지하라.

4.1 C++ 구현 아키텍처 예제

아래 서술된 C++ 코드 작성 패턴을 참고하여, 멀티스레딩 기반 객체 지향 제어기를 구축하라.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/offboard_control_mode.hpp>
#include <mutex>

class OffboardControlNode : public rclcpp::Node {
public:
    OffboardControlNode() : Node("offboard_control_node") {
        // 독립적인 제어 타이머 처리를 위한 별도의 상호 배타적 그룹 생성
        timer_cb_group_ = this->create_callback_group(
            rclcpp::CallbackGroupType::MutuallyExclusive);
        
        // 빈번하게 유입되는 구독자 메시지 처리를 위한 재진입 가능 그룹 생성
        sub_cb_group_ = this->create_callback_group(
            rclcpp::CallbackGroupType::Reentrant);

        // 옵션 객체를 생성하여 콜백 그룹 배정
        rclcpp::SubscriptionOptions sub_options;
        sub_options.callback_group = sub_cb_group_;

        // 50Hz 고정 주기 작동을 보장하는 고주파수 발행 타이머 등록
        timer_ = this->create_wall_timer(
            std::chrono::milliseconds(20), 
            std::bind(&OffboardControlNode::timerCallback, this),
            timer_cb_group_);
    }

private:
    void timerCallback() {
        std::lock_guard<std::mutex> lock(data_mutex_);
        // 드론에 요구되는 설정점(Setpoint) 업데이트 및 전송 로직 수행
    }

    rclcpp::CallbackGroup::SharedPtr timer_cb_group_;
    rclcpp::CallbackGroup::SharedPtr sub_cb_group_;
    rclcpp::TimerBase::SharedPtr timer_;
    std::mutex data_mutex_;
};

int main(int argc, char ** argv) {
    rclcpp::init(argc, argv);
    
    auto node = std::make_shared<OffboardControlNode>();
    
    // 호스트 프로세서의 코어(Core) 가용성을 고려한 멀티스레드 인스턴스 구축
    // 4개의 스레드를 할당하여 병목을 제거한다.
    rclcpp::executors::MultiThreadedExecutor executor(
        rclcpp::ExecutorOptions(), 4); 
    executor.add_node(node);
    
    executor.spin();
    rclcpp::shutdown();
    return 0;
}

제시된 예제와 같이 운영체제 하드웨어 자원의 수용 능력을 최대한 활용함으로써 실시간 비행 제어 및 복잡한 항법 데이터를 동시에 수용할 수 있는 신뢰성 높은 아키텍처를 구현하라. 이를 통해 무인기가 유동적이고 위험한 외부 환경 조건 속에서도 Offboard 제어권을 안정적으로 보존하도록 시스템을 설계해야 한다.
