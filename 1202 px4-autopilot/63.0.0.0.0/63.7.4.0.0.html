63.7.4. [실습] 펄스 폭 변조(PWM) 및 계측 제어기 통신망(CAN) 버스 보조 채널을 활용한 외부 임무 하중(투하 장치, 광학 짐벌 등)의 논리적 릴레이(Relay) 펄스열 동기화 제어 구현

무인항공기(UAV)의 응용 분야가 다각화됨에 따라 비행 기체의 물리적 이동을 넘어서 외부 임무 하중(Payload)에 대한 복합적인 제어가 필수적으로 요구되고 있다. 물자 투하 장치(Drop Mechanism), 다분광 광학 짐벌(Optical Gimbal), 로봇 그리퍼(Gripper) 등은 기체의 메인 비행 컴퓨터(FC, Flight Controller)와 논리적으로 분리되어 동작하지만, 임무 수행의 성공을 위해서는 비행 궤적 및 제어 주기와 엄밀하게 동기화되어야 한다.

본 절에서는 PX4-Autopilot의 보조 출력(AUX Output) 포트를 활용하여 펄스 폭 변조(PWM, Pulse Width Modulation) 신호 및 계측 제어기 통신망(CAN, Controller Area Network) 버스 인터페이스를 기반으로 하는 외부 하중 제어 실습을 진행한다. 특히, 단순히 On/Off를 수행하는 아날로그 스위치 레벨을 넘어서 ROS 2 Offboard 환경에서 펄스열(Pulse Train)의 위상을 동기화하고 논리적 릴레이(Relay)를 구현하기 위한 C++ 제어 노드 작성 기법을 다룬다. (본 실습은 PX4 v1.14.x, QGroundControl v4.3.x 및 ROS 2 Humble 환경을 기준으로 서술한다.)

1. 외부 임무 하중 제어를 위한 엑추에이터 출력 아키텍처

비행 제어 컴퓨터는 주로 FMU(Flight Management Unit) MAIN 포트와 AUX 포트로 I/O 시스템이 나뉜다. MAIN 포트가 멀티로터의 브러시리스 DC(BLDC) 전자 변속기(ESC) 제어 등 기체 동역학에 직접 관여한다면, AUX 포트는 랜딩 기어, 낙하산 사출, 짐벌 및 투하 장치 제어를 위한 여분의 타이머(Timer) 채널을 제공한다. 최근에는 UAVCAN(DroneCAN) 프로토콜 기반의 스마트 노드가 도입되면서, CAN 버스를 통한 분산 제어가 기존의 아날로그 PWM 방식을 빠르게 대체하고 있다.

PX4-Autopilot v1.14 아키텍처부터 기본 적용된 동적 제어 할당 체계(Dynamic Control Allocation) 설계 철학에 따르면, 믹서(Mixer) 파일 매핑 없이도 MAVLink나 ROS 2(Micro XRCE-DDS)를 통해 논리적 제어 메시지를 할당기에 직접 주입할 수 있다. 이는 외부 페이로드의 모터 스텝 제어나 릴레이 래치(Latch) 타이밍 제어 시 매우 유용하게 작용한다.

2. 펄스열 동기화(Pulse Train Synchronization)와 릴레이 로직의 수학적 지연 처리

논리적 릴레이(Relay) 제어는 근접 센서나 광학 비전 시스템의 트리거(Trigger) 시점을 기점으로 특정 임무 장비에 개방(Open) 통신 펄스를 인가하는 로직이다. 일반적인 단순 릴레이는 상태(State) 값인 0 또는 1 만을 인가하지만, PWM 및 CAN 버스 구동기를 사용하는 다기능 임무 장비는 1000µs ~ 2000µs의 정밀한 펄스폭 변화를 상태 전이(State Transition) 조건으로 요구한다.

이를 위해 ROS 2 기반의 임무 프로세스는 컴패니언 컴퓨터(Companion Computer)와 PX4 타이머 사이의 네트워크 레이턴시(Network Latency)를 고려해야 한다. 오차를 최소화하고 결정론적(Deterministic) 펄스를 생성하기 위해 메시징 루프 내의 타임스탬프(Timestamp) 동기화는 필수적이다. 즉, 시각 T에 컴퓨터 비전 인식 알고리즘이 표적을 포착하면, 릴레이 작동 시각 T + \Delta t 에는 PX4 내부 할당기를 통해 정확히 1500\mu s 에서 1900\mu s 로 펄스 폭을 변조해야 한다.

3. PX4 및 ArduPilot의 페이로드 제어 파라미터 비교

ArduPilot은 임무 플래너(Mission Planner) 상에서 RCx_OPTION이나 SERVOx_FUNCTION 파라미터를 통해 릴레이 스위치를 정형화된 기능으로 매핑한다. 짐벌 제어를 위해 MNT_MODE 등을 지원하여 훌륭한 사용자 경험을 제공하지만, 개발자가 소스 코드를 통해 완전히 독자적인 형태의 다단계 위상 동기화 투하 프로토콜을 구현하려고 할 때는 Lua 스크립트에 주로 의존해야 하므로 성능상의 제약이 따를 수 있다.

이와 대조적으로 PX4는 ActuatorServos 토픽을 활용해 특정 Control 프로파일 요소별 비율(Ratio: -1.0 ~ 1.0)을 직접 조립하여 전송할 수 있는 강력한 인터페이스를 갖추고 있다. 이는 복잡하게 얽혀 있는 페이로드를 조작할 때 파라미터 튜닝 없이 프로그래머 중심적인 I/O 맵핑 기능을 보장한다.

4. 논리적 릴레이 토폴로지 동역학 모델

아래 다이어그램은 ROS 2 기반의 외부 임무 제어 시스템이 비전 인식을 기반으로 트리거 신호를 생성하고, 이를 PWM 및 CAN 제어 노드에 동기화하여 인가하는 일련의 과정을 묘사한다.

5. [실습] C++ 기반 짐벌 및 투하 장치 논리적 릴레이 제어 노드 작성

ROS 2 Humble 기반의 C++ 프로그래밍을 활용하여 특정한 센서 이벤트(여기서는 단순 타이머 트리거로 모사)에 대응하여 투하 장치 릴레이를 해제하고 짐벌의 셔터를 작동시키는 펄스열 동기화 코드를 작성한다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/actuator_servos.hpp>
#include <chrono>

using namespace std::chrono_literals;

class PayloadRelayController : public rclcpp::Node
{
public:
    PayloadRelayController() : Node("payload_relay_controller")
    {
        // 엑추에이터 서보 제어 명령 퍼블리셔 선언
        servo_pub_ = this->create_publisher<px4_msgs::msg::ActuatorServos>(
            "/fmu/in/actuator_servos", 10);

        // 이벤트 트리거를 모사하는 10Hz 제어 타이머
        timer_ = this->create_wall_timer(
            100ms, std::bind(&PayloadRelayController::control_loop, this));
            
        // 초기화 시점 기록
        start_time_ = this->get_clock()->now();

        RCLCPP_INFO(this->get_logger(), "Payload Relay Controller Node Started.");
    }

private:
    rclcpp::Publisher<px4_msgs::msg::ActuatorServos>::SharedPtr servo_pub_;
    rclcpp::TimerBase::SharedPtr timer_;
    rclcpp::Time start_time_;
    
    bool payload_dropped_ = false;

    void control_loop()
    {
        auto current_time = this->get_clock()->now();
        double elapsed_sec = (current_time - start_time_).seconds();

        px4_msgs::msg::ActuatorServos servo_msg;
        servo_msg.timestamp = current_time.nanoseconds() / 1000ULL;
        
        // C++ 배열 초기화 (미사용 채널은 기본적으로 배제)
        for (int i = 0; i < 8; i++) {
            servo_msg.control[i] = NAN;
        }

        /*
         * 채널 매핑 가정:
         * Control[4] (AUX 5번 채널) -> 물자 투하 장치 릴레이 제어 (-1.0: 닫힘, 1.0: 열림)
         * Control[5] (AUX 6번 채널) -> 짐벌 셔터 트리거 펄스 (-1.0: 대기, 1.0: 셔터 동작)
         */

        if (elapsed_sec > 10.0 && !payload_dropped_) {
            // 이륙 후 10초 경과 시 비전 이벤트가 발생했다고 가정 (논리적 릴레이 전환)
            RCLCPP_WARN(this->get_logger(), "Target Reached! Triggering Drop Mechanism & Gimbal Shutter!");
            
            // 투하 장치 개방 (하이 펄스열 인가)
            servo_msg.control[4] = 1.0f;
            
            // 카메라 셔터 기동
            servo_msg.control[5] = 1.0f; 
            
            servo_pub_->publish(servo_msg);
            payload_dropped_ = true;
        } 
        else if (payload_dropped_) {
            // 지속적인 상태 유지를 위한 래치(Latch) 기능 활성화
            servo_msg.control[4] = 1.0f;
            
            // 카메라 셔터는 단발성 펄스(Shot)를 요구하므로 짧은 지연 후 복원
            if (elapsed_sec > 11.0) {
                servo_msg.control[5] = -1.0f; 
            } else {
                servo_msg.control[5] = 1.0f; 
            }
            servo_pub_->publish(servo_msg);
        }
        else {
            // 대기 상태(Standby State)
            servo_msg.control[4] = -1.0f; // 닫힘 유지
            servo_msg.control[5] = -1.0f; // 셔터 대기 유지
            servo_pub_->publish(servo_msg);
        }
    }
};

int main(int argc, char **argv)
{
    rclcpp::init(argc, argv);
    rclcpp::spin(std::make_shared<PayloadRelayController>());
    rclcpp::shutdown();
    return 0;
}

위 코드는 ActuatorServos 내의 특정 요소 배열(Array)에 접근하여 논리적 하이(High) 및 로우(Low) 신호를 시계열에 맞춰 매핑한다. NAN 값을 사용하여 본 노드에서 다루지 않는 채널(예 : VTOL의 틸트 서보 등)의 간섭을 원천적으로 차단하는 기법은 멀티 에이전트 시스템에서 제어 권한 분리 원칙을 준수하는 모범 사례(Best Practice)에 해당한다.

6. 결론

본 절에서는 단순한 페이로드 조작을 넘어, 펄스열 제어 동기화와 논리적 릴레이 상태 머신(Relay State Machine)을 PX4-Autopilot 및 ROS 2 Offboard 아키텍처에 구현하는 방안을 살펴보았다. PWM과 CAN 통신을 아우르는 제어 할당 체계(Control Allocation)의 유연성을 통해, 개발자는 복잡한 임무 하중의 순차적인 트리거 및 상태 유지를 실현할 수 있다. 이러한 통합 통신망 제어는 컴퓨터 비전 추적 제어 및 자율 비행 시스템 구현을 위한 기반 하드웨어 추상화 지식으로서 중추적인 역할을 담당한다.
