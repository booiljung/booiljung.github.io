63.1.4.2 생존 신호(Heartbeat) 동기화 결함 감지 및 제어 권한 상실 방지를 위한 슬라이딩 윈도우(Sliding Window) 기반 동적 타임아웃 메커니즘

1. 개요

비행 제어기(FC, Flight Controller)와 임무 컴퓨터(Companion Computer) 간의 오프보드 모드(Offboard Mode) 기반 시스템에서는 지속적이고 안정적인 통신이 필수적이다. PX4-Autopilot은 생존 신호(Heartbeat) 및 제어 설정점(Setpoint) 수신이 중단될 경우 이를 제어 권한 상실(Control Loss)로 간주하고 즉각적인 비상 안전 조치(Failsafe)를 수행한다. 본 절에서는 네트워크 지터(Jitter)로 인한 잦은 오탐지(False Positive)를 방지하고 제어 권한을 안정적으로 유지하기 위해, 슬라이딩 윈도우(Sliding Window) 알고리즘을 활용한 동적 타임아웃(Dynamic Timeout) 결함 감지 메커니즘의 아키텍처와 원리를 상세히 분석한다. (기준: PX4 v1.14 및 ROS 2 Humble)

2. 오프보드 제어 상실(Offboard Loss)의 원인 및 기존 방식의 한계

제어 스택과 임무 컴퓨터 간의 통신은 MAVLink 또는 ROS 2 uXRCE-DDS 미들웨어를 통해 이루어진다. 비행 제어기는 임무 컴퓨터로부터 주기적(통상 10Hz 이상)인 제어 명령을 기대한다.

  • PX4의 기본 타임아웃 정책: PX4-Autopilot은 COM_OF_LOSS_T 파라미터를 통해 오프보드 신호 상실까지의 타임아웃 기준(기본값 0.5초)을 정의한다.
  • 고정 타임아웃의 한계: 와이파이(Wi-Fi)나 LTE 기반의 외부 통신망, 혹은 임무 컴퓨터 측 운영체제의 높은 프로세스 부하로 인해 패킷 송신이 일시적으로 지연(Latency Peak)될 수 있다. 고정된 타임아웃(Static Timeout) 방식을 적용할 경우, 단발성의 패킷 지연이 곧바로 제어 상실로 판별되어 비정상적인 비상 복구(RTL 혹은 Hovering) 상태가 촉발된다.

3. 슬라이딩 윈도우(Sliding Window) 기반 동적 타임아웃 구조 설계

이러한 네트워크 불안정성과 연산 지연의 변동성을 허용하기 위해, 최근의 메시지 도달 시간의 통계적 분포를 이용하여 타임아웃 임계치를 실시간으로 갱신하는 슬라이딩 윈도우 기법을 적용한다.

3.1 작동 알고리즘 흐름

  1. 관측 윈도우(Observation Window) 구성: 최신 N개의 제어 메시지(Heartbeat 또는 Trajectory Setpoint) 수신 간격(Inter-arrival Time, \Delta t) 데이터를 선입선출(FIFO) 기반의 원형 큐(Circular Queue)에 저장한다.
  2. 동적 임계값(Dynamic Threshold, T_{dynamic}) 연산:
    윈도우에 적재된 데이터의 평균(\mu)과 표준편차(\sigma)를 계산하여, 상황에 맞는 허용 한계값을 산출한다.
    수식은 T_{dynamic} = \mu + k \cdot \sigma 로 정의되며, k는 민감도를 조절하는 가중치 팩터이다.
  3. 결함 감지 판별(Fault Detection): 가장 마지막 수신된 메시지로부터 경과한 시간이 산출된 T_{dynamic}을 초과하되 절대 최대 허용 한계(Absolute Maximum Delay)에 도달하기 전이라면 경고(Warning)를 발생시키고, 최대 허용 한계를 초과해야만 실제 제어 권한 상실로 확정(Fault Confirmed)한다.

4. Ardupilot 대비 결함 통제 체계의 차이점

  • Ardupilot의 구조: Ardupilot은 GCS 및 Offboard 컴퓨터 단절 시 FS_GCS_ENAB 파라미터를 기반으로 고정된 타임아웃 주기로 통신 단절(Data Link Loss) 여부를 판별한다. GCS와 Companion Computer 역할이 겹칠 경우 예외 처리 분기가 비교적 단순하며 복잡한 동적 윈도우 메커니즘의 자체 적용이 제한적이다.
  • PX4의 구조: PX4-Autopilot은 COM_RC_LOSS_T 패러다임 외에, 데이터 링크를 COM_DL_LOSS_T 및 COM_OF_LOSS_T 등으로 세분화하여 분리 감지한다. 특히 ROS 2와 결합 시 uXRCE-DDS 레이어의 QoS(Quality of Service) 데드라인(Deadline QoS) 정책과 결합하면, 슬라이딩 윈도우 레이어를 수신단(FC) 뿐만 아니라 미들웨어 계층 및 송신단에서도 손쉽게 융합적(Hybrid)으로 구현할 수 있는 우수한 확장성을 제공한다.

5. 슬라이딩 윈도우 기반 통계 처리 구현 예시 (ROS 2 C++)

동적 타임아웃 알고리즘 구성을 위한 기초적인 수학적 통계 산출 과정의 일부 구현 예시이다.

#include <vector>
#include <numeric>
#include <cmath>

class SlidingWindowMonitor {
private:
    std::vector<double> deltas; // 수신 타이밍 델타 버퍼
    size_t window_size;
    size_t current_index;
    double max_absolute_timeout; 
    double k_factor;

public:
    SlidingWindowMonitor(size_t size, double max_timeout, double k) 
        : window_size(size), max_absolute_timeout(max_timeout), k_factor(k), current_index(0) {
        deltas.resize(size, 0.05); // 초기 예상 주기 50ms 삽입
    }

    void add_delta(double delta_time) {
        deltas[current_index] = delta_time;
        current_index = (current_index + 1) % window_size;
    }

    double get_dynamic_timeout() {
        double sum = std::accumulate(deltas.begin(), deltas.end(), 0.0);
        double mean = sum / window_size;

        double sq_sum = std::inner_product(deltas.begin(), deltas.end(), deltas.begin(), 0.0);
        double stdev = std::sqrt(sq_sum / window_size - mean * mean);

        double dynamic_timeout = mean + (k_factor * stdev);
        
        // 상한선 제한(Maximum Capping)
        return (dynamic_timeout > max_absolute_timeout) ? max_absolute_timeout : dynamic_timeout;
    }
};

위 로직은 T_{dynamic}을 실시간 갱신하며, 산출된 타임아웃 내에 다음 제어 데이터가 송신 단계를 밟지 못한다면 어플리케이션 계층에서 궤적을 보간(Interpolation)하여 임의 생성해 송신하거나, 즉각적인 자체 페일세이프 시퀀스를 호출할 수 있도록 기능한다.

6. 결론

슬라이딩 윈도우를 활용한 동적 타임아웃 감지 메커니즘을 오프보드 제어 아키텍처에 구현하면, 실시간 운영체제(RTOS)가 아닌 모호한 우선순위를 가진 범용 리눅스 환경과 무선 네트워크 환경에서도 튼튼한 내결함성(Fault Tolerance)을 달성할 수 있다. 이 메커니즘은 통신 지터로 인한 고정 타임아웃 검출 오류를 줄이고 PX4-Autopilot이 요구하는 제어 권한의 안정성을 탁월하게 향상시킨다.
