63.4.2. 고정 주기 국소 발진기(Local Oscillator) 기반 하드웨어 타이머 콜백을 활용한 동기화된 궤적 설정점 발행 패턴 구현

본 절에서는 자율 비행 로봇 제어 시스템에서 필수적으로 요구되는 고정 주기(Fixed-rate) 제어 루프를 보장하기 위해, 국소 발진기(Local Oscillator)에 기반한 하드웨어 타이머를 ROS 2 환경에 연동하는 기법을 다룬다. 특히 PX4-Autopilot의 Offboard 모드 유지를 위한 가장 핵심적인 궤적 설정점(Trajectory Setpoint) 동기화 발행 메커니즘을 심도 있게 분석한다. 본 내용은 PX4 v1.14.0 펌웨어 및 ROS 2 Humble 미들웨어 생태계를 기준으로 명세한다.

1. 국소 발진기(Local Oscillator)와 하드웨어 타이머의 원리

비행 제어 컴퓨터(Flight Control Computer, FCC) 및 지상 관제 목적의 보조 컴퓨터(Companion Computer) 내부에는 물리적 시간 경과를 정밀 측정하기 위한 수정 진동자(Quartz Crystal) 기반의 국소 발진기가 내장되어 있다. 이 발진기는 매 순간 아주 정밀하게 고정된 주파수의 클럭 펄스(Clock Pulse)를 생성하며, 프로세서는 이 펄스를 계수(Count)하여 정밀도 높은 하드웨어 타이머(Hardware Timer) 인터럽트를 시스템에 릴레이한다.

ROS 2의 벽시계 타이머(Wall Timer) 로직은 궁극적으로 운영체제(OS) 커널 계층을 거쳐 이 하드웨어 타이머의 시스템 콜(System Call) 인터페이스를 이용한다. 제어 명령의 전송 주기가 불안정할 경우(기대 주파수가 무너지며 지터(Jitter) 오차가 커질 경우), 기체의 내부 상태 추정기(Estimator)와 외부에서 수신된 목표 명령 간의 가파른 위상 차이(Phase Lag)가 유발되어 치명적인 제어 발진(Control Oscillation)을 야기하게 된다. 따라서 국소 발진기 기반의 타이머 콜백 규칙 적용을 통해 정확한 델타 타임(\Delta t) 연속성을 담보하는 것이 Offboard 궤적 추종 성능의 성패를 좌우한다.

2. 동기화된 궤적 설정점 통신 아키텍처

아래는 하드웨어 타이머 인터럽트에서 기인하는 ROS 2와 PX4 간의 동기화 제명령 처리 메커니즘을 묘사한 시스템 아키텍처이다.

sequenceDiagram
    participant LO as Local Oscillator (Hardware)
    participant OS as OS Kernel & ROS 2 Executor
    participant Node as Offboard Control Node
    participant DDS as Micro XRCE-DDS Middleware
    participant PX4 as PX4-Autopilot (FMU)

    loop Every 20ms (Fixed 50Hz)
        LO->>OS: Hardware Interrupt (Clock Tick)
        OS->>Node: Trigger Wall Timer Callback
        Node->>Node: Calculate Trajectory Setpoint (x, y, z)
        Node->>Node: Populate OffboardControlMode
        Node->>DDS: Publish TrajectorySetpoint & OffboardControlMode
        DDS->>PX4: Serialize & Transmit via UART/UDP
        PX4->>PX4: Pass to Position Controller
    end

이 고정 패러다임 안에서 제어 노드는 센서 수신(Subscribe) 타이밍이나 수신 지연(Delay)에 종속되지 않고, 철저히 독립적이고 고정적인 클럭 주기에 엄격히 맞추어 목표 궤적을 발행(Publish)한다. 즉, 특정 센서 프로세싱의 처리 지연이 발생하더라도 제어 하달 명령은 일정한 주기를 방어하게 되어 Offboard 타임아웃(Timeout) 판정을 원천 차단하는 핵심 설계 방식이다.

3. 동기화된 궤적 설정점 발행 패턴 구현 패러다임

하드웨어 타이머에 전적으로 의존하는 동기화 발행 루틴을 구현할 때는 다음 세 가지의 핵심 설계 요구사항을 충족해야 한다.

  1. 상호 배타적 고순위 스케줄링 설계: 타이머 콜백은 다른 서브프로시저에 의해 선점(Preemption)되거나 블로킹(Blocking)되어서는 안 된다. 따라서 앞서 다룬 멀티스레드 실행자(Multi-threaded Executor) 내부에서도 이콜백을 반드시 상호 배타적(Mutually Exclusive) 콜백 그룹 단독으로 분배하여 최우선 처리권을 보장받도록 위임해야 한다.
  2. 이중 토픽 발행(Dual Publishing)의 원자성(Atomicity): PX4의 Offboard 모드 진입 및 활성 상태를 연속적으로 유지하기 위해서는 단순히 목표 좌표값만을 지시하는 TrajectorySetpoint 메시지 개체와 더불어, 어떠한 제어 루프(위치, 속도, 가속도, 자세 등) 포트폴리오를 적용할 것인지를 PX4 커맨더(Commander)에 선포하는 OffboardControlMode 메시지를 반드시 동일한 주기로, 동일한 하드웨어 콜백 함수 바운더리 내에서 융합해 순서대로 발행해야 한다. 이는 제어 모드와 데이터 값의 시간적 불일칠을 예방하는 PX4 생태계의 절대적 수칙이다.
  3. 단위 시간당 균일성 측정 모니터링 체계 도입: rosbag 등의 서드파티 기록 도구를 통해 추후 메시지 타임스탬프(Timestamp) 오프셋을 역추적 분석했을 때 시그니처 편차가 기준 시간 대비 \pm 2ms 대역 이내로 수렴하도록 해야 한다. 이를 점검하기 위해 std::chrono 라이브러리의 고해상도 시계(std::chrono::high_resolution_clock)를 활용하여 제어 코드 내부에 지터(Jitter) 모니터링 및 실시간 경고(Diagnostic) 로직을 통합하라.

4. C++ rclcpp::TimerBase를 활용한 타이머 콜백 코드 템플릿

상기 요구사항을 응용 소프트웨어 구조로 안전하게 반영한 C++ 구현체 템플릿은 다음과 같이 작성될 수 있다.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/offboard_control_mode.hpp>
#include <px4_msgs/msg/trajectory_setpoint.hpp>
#include <chrono>

using namespace std::chrono_literals;

class SynchronizedTrajectoryPublisher : public rclcpp::Node {
public:
    SynchronizedTrajectoryPublisher() : Node("sync_tray_pub_node") {
        
        // 제어 모드 및 설정점 퍼블리셔 선언
        offboard_control_mode_pub_ = this->create_publisher<px4_msgs::msg::OffboardControlMode>(
            "/fmu/in/offboard_control_mode", 10);
        trajectory_setpoint_pub_ = this->create_publisher<px4_msgs::msg::TrajectorySetpoint>(
            "/fmu/in/trajectory_setpoint", 10);

        // 50Hz (20ms) 고정 주기 하드웨어 연계 타이머 인스턴스 할당
        timer_ = this->create_wall_timer(
            20ms, std::bind(&SynchronizedTrajectoryPublisher::timer_callback, this));
            
        RCLCPP_INFO(this->get_logger(), "Hardware Timer Callback Successfully Initialized at 50Hz.");
    }

private:
    void timer_callback() {
        // 단일 콜백 스코프 안에서 모드와 설정점의 이중 발행 원자성을 보장함
        publish_offboard_control_mode();
        publish_trajectory_setpoint();
    }

    void publish_offboard_control_mode() {
        px4_msgs::msg::OffboardControlMode msg{};
        msg.timestamp = this->get_clock()->now().nanoseconds() / 1000ULL;
        msg.position = true;   // 위치 제어 활성화 플래그 On
        msg.velocity = false;
        msg.acceleration = false;
        msg.attitude = false;
        msg.body_rate = false;
        
        offboard_control_mode_pub_->publish(msg);
    }

    void publish_trajectory_setpoint() {
        px4_msgs::msg::TrajectorySetpoint msg{};
        msg.timestamp = this->get_clock()->now().nanoseconds() / 1000ULL;
        // z축은 NED(North-East-Down) 기체 기준 좌표계 관례 상 하단 방향이 양수(+)이다.
        // 그러므로 -5.0m는 지표면 대비 상승 고도 5m 지점을 의미함
        msg.position = {0.0f, 0.0f, -5.0f}; 
        msg.yaw = -3.14f; // -PI (서쪽 방향 조향 유지)
        
        trajectory_setpoint_pub_->publish(msg);
    }

    rclcpp::Publisher<px4_msgs::msg::OffboardControlMode>::SharedPtr offboard_control_mode_pub_;
    rclcpp::Publisher<px4_msgs::msg::TrajectorySetpoint>::SharedPtr trajectory_setpoint_pub_;
    rclcpp::TimerBase::SharedPtr timer_;
};

int main(int argc, char *argv[]) {
    rclcpp::init(argc, argv);
    // 단일 프로세스 루프를 관장하는 스핀(Spin) 활성화
    rclcpp::spin(std::make_shared<SynchronizedTrajectoryPublisher>());
    rclcpp::shutdown();
    return 0;
}

예시 코드는 단일 타이머 스레드 슬롯 영역 내에서 제어 파라미터를 담는 OffboardControlMode와 실질적 삼차원 좌표 수치를 넘기는 TrajectorySetpoint를 순차적으로 발행하여 시스템 간의 타임스탬프 동기화를 하드웨어 레벨에서 보장한다. 하드웨어 콜백 주기에 강결합된 이 방식을 보수적으로 관철함으로써, 제어 패킷의 불규칙한 누락으로 인한 예상치 못한 Failsafe 강제 착륙 절차가 기동되는 리스크를 실무 환경에서 완벽하게 예방하라.
