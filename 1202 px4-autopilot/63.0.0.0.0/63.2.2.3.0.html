63.2.2.3. 전역 기체 명령 토픽(VehicleCommand)을 통한 마르코프 결정 과정(Markov Decision Process) 기반 유한 상태 기계(Finite State Machine) 상태 전이 트리거 로직

무인항공기(UAS) 제어 소프트웨어 아키텍처에 있어 거시적인 비행 상태(Flight State)의 천이는 시스템의 안전과 직결되는 핵심 연산이다. PX4-Autopilot은 상태 전이(State Transition) 메커니즘을 마르코프 결정 과정(MDP: Markov Decision Process)의 관점에서 설계 모델링하였으며, 이러한 전역 유한 상태 기계(FSM: Finite State Machine)를 이끌어가는(Trigger) 지배적 매개체가 곧 VehicleCommand 토픽이다. 본 절에서는 지상 관제 시스템(GCS) 또는 오프보드(Offboard) 노드에서 발송된 기체 명령이 내부 커맨더(Commander) 모듈에서 어떠한 논리 구조를 거쳐 확정적 상태 전이를 유발하는지 분석한다.

1. VehicleCommand 구조체와 MDP 기반 상태 전이 모델

비행 제어기의 유한 상태 기계는 근본적으로 현재 기체의 상태(State, S)와 인가된 행동(Action, A)을 바탕으로 다음 상태(S')로 천이할 확률적 타당성 혹은 결정론적 가부(Deterministic Validation)를 판단한다. MDP 모델에 기반하여 구동되는 PX4 커맨더 로직 내에서 VehicleCommand는 곧 외부 환경(QGroundControl, MAVLink, ROS2)이 에이전트(FC)에게 강구하는 의도된 행동(A) 집합에 해당한다.

1.1 VehicleCommand의 논리 구조 분해

Ardupilot이 COMMAND_LONG 메시지를 수신하여 중앙 집중식으로 분기 처리하는 것 대비, PX4는 uORB 상의 다형성(Polymorphism) 기반 토픽 배포(Distribution) 모델을 지향한다. VehicleCommand 구조체(v1.14 기준)는 다음의 핵심적인 필드를 내포하고 있다.

  • command (명령 ID): 수행하고자 하는 주 명령의 열거형(Enum) 식별자 (예: VEHICLE_CMD_COMPONENT_ARM_DISARM, VEHICLE_CMD_DO_SET_MODE)
  • param1 ~ param7 (매개변수 벡터): 7개의 단정밀도 부동소수점(Float) 필드. 각 명령 ID에 따라 암묵적 형변환(Implicit Casting)되어 완전히 다른 의미를 갖도록 오버로딩(Overloading)됨.
  • target_system / target_component: 다중 에이전트(Swarm) 통신 환경 및 컴포넌트 라우팅 제어 시 수신 대상을 좁히는(Filtering) 필드. 멀티캐스트 네트워크 상에서 필수적이다.
  • source_system / source_component: 명령을 발행한 클라이언트(예: MAVSDK 애플리케이션 등)의 식별자. MAVLink 명령어 전달 신뢰성을 보장하는 ACK 메시지 라우팅에 활용비례된다.

1.2 커맨더(Commander) 내부의 암묵적 보상(Implicit Reward)과 전제 조건(Precondition) 검증

MDP에서 최적의 정책(Policy)이 보상(Reward) 함수를 토대로 결정되듯, PX4의 FSM 로직은 특정 명령(A)을 수락하여 상태(S')로 진입하기에 앞서 ’시동 전제 조건(Pre-arm Checks)’과 같은 논리적 방벽을 우선 평가한다.
예를 들어 VEHICLE_CMD_COMPONENT_ARM_DISARM 명령 수신 시, 커맨더는 다음의 전제 조건 벡터 \mathbf{C} 들을 논리곱(Logical AND) 연산한다.

  1. \text{C}_{ekf} : EKF2 확장 칼만 필터의 위치/자세 추정 오차 공분산이 허용 임계치 이내인가?
  2. \text{C}_{cal} : 가속도계/자이로스코프(IMU) 및 나침반 센서의 캘리브레이션이 유효한가?
  3. \text{C}_{pwr} : 배터리 잔량 및 내부 프로세서 구동 전압 시스템이 안정적인가?
  4. \text{C}_{rc} : (선택적) 안전 파일럿을 위한 RC 수신기 연결 상태가 무결한가?

이러한 사전 점검(Pre-flight Check) 결과가 하나의 상태 평가 함수(Value Function)로서 ’가용(Valid)’으로 반환될 때 비로소 액추에이터 활성상태(ARMED)로 천이하며, 실패 시 VehicleCommandAck 메시지의 result 필드에 실패 사유를 포장(Deny)하여 지상 관제 환경으로 피드백한다.

2. 모드 역이행(Fallback)과 응답(ACK) 동기화 프로토콜

지상 관제 시스템(QGC v4.3) 또는 자율 에이전트 개발 시 가장 빈번히 마주하는 문제는 모드 전환 명령의 타임아웃(Timeout) 및 비동기적 지연 현상이다. 커맨더 노드는 FSM 사이클마다 큐(Queue)에 적재된 VehicleCommand를 소비(Consume)한다.

만일 임무(Mission) 모드 전환을 요청하는 VEHICLE_CMD_DO_SET_MODE가 유효하게 처리되지 못한 채 COMMAND_ACK 응답 지연이 발행할 경우, 클라이언트는 유한 시간 경과 후 명령을 재전송(Retry)하는 구조를 취해야 한다. 흥미롭게도 커맨더 내부 구조 해석 및 학술 논문(예: “Designing Verification and Validation Workflows for ArduPilot/PX4”, 2021)에 따르면 두꺼운 미들웨어 계층(uORB 링 버퍼 등)이 명령 처리의 병목으로 작용할 수 있음을 지적한다.

2.1 다이어그램: VehicleCommand 기반 상태 전이 수락 파이프라인

sequenceDiagram
    participant GCS as GCS(QGroundControl) / ROS2
    participant MAV as MAVLink / DDS Bridge
    participant CMD as Commander (FC FSM)
    participant SENS as Sensor Hub (EKF2)
    
    GCS->>MAV: Send COMMAND_LONG (Arming)
    MAV->>CMD: Publish VehicleCommand (uORB)
    CMD->>SENS: Validate System Preconditions
    SENS-->>CMD: Return Readiness Boolean
    alt is Readiness True
        CMD->>CMD: FSM Transition: STANDBY -> ARMED
        CMD->>MAV: Publish VehicleCommandAck (ACCEPTED)
    else is Readiness False
        CMD->>MAV: Publish VehicleCommandAck (DENIED / TEMPORARILY_REJECTED)
    end
    MAV-->>GCS: Relay ACK Response

결론적으로 PX4의 VehicleCommand 기반 제어 파이프라인은 정적 함수 호출이 아닌 큐잉(Queueing) 기반의 비동기적 이벤트 드리븐(Event-driven) 상태 기계 진입 방식을 표방한다. 이는 오프보드 제어 환경의 복잡한 비선형적(Non-linear) 예외 처리 시 안정망을 담보하는 필수불가결한 아키텍처라 하겠다.
