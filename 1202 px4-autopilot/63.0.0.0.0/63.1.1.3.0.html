63.1.1.3. 이산 시간(Discrete-time) 제어 시스템에서의 실시간성(Real-time Constraints) 보장을 위한 컴퓨팅 자원 할당 이론

1. 서론

현대 무인 항공기(UAV)의 비행 제어 및 자율 비행 시스템은 본질적으로 이산 시간(Discrete-time) 제어 시스템의 특성을 지닌다. 유도, 항법 및 제어(GNC: Guidance, Navigation, and Control) 알고리즘은 연속 시간(Continuous-time)의 물리적 동역학(Dynamics)을 센서 샘플링 속도 및 제어 루프 주기에 맞춰 이산화(Discretization) 과정을 거쳐 디지털 컴퓨터 상에서 실행된다. 특히, 임무 컴퓨터(Mission Computer)와 비행 제어기(Flight Controller, FC)가 연동되는 분산 아키텍처—예컨대 ROS 2(Robot Operating System 2)를 기반으로 하는 오프보드(Offboard) 제어 모델—에서는 실시간성(Real-time Constraints) 보장을 위한 컴퓨팅 자원의 엄격한 할당 및 스케줄링 이론이 시스템의 안정성(Stability)을 결정짓는 절대적인 기준이 된다. 본 절에서는 PX4-Autopilot(Firmware v1.14 기준)과 ROS 2 시스템(Humble 또는 Iron 기준) 연동 환경을 바탕으로 하여 실시간성 확보를 위한 컴퓨팅 자원 할당 이론과 실제 적용 기법에 대해 상세히 분석하라.

2. 이산 시간 제어 시스템에서의 연산 지연(Latency)과 시스템 안정성 분석

이산 시간 제어 환경에서 센서 데이터를 수집하고 제어 입력을 계산하여 목적 구동기(Actuators)로 전달하는 일련의 과정에서는 필연적으로 계산 지연(Computational Delay)과 통신 지연(Communication Delay)이 수반된다. 제어 이론적 관점에서 볼 때 이러한 시간 지연(Time Delay)은 주파수 영역(Frequency Domain) 내에서 위상 여유(Phase Margin)를 급격하게 감소시켜 폐루프(Closed-loop) 제어 시스템의 응답 품질 하락과 더불어 극단적 발산(Divergence) 현상을 유발할 수 있다.

비행 제어 소프트웨어에서 각 루프는 독립적인 주기적 마감 시간(Periodic Deadline)을 통제받는다. PX4의 내부 역학 제어기(Inner-loop Controller인 Rate 및 Attitude Controller)는 최소 250Hz에서부터 400Hz 수준(2.5ms~4ms 주기) 이상의 높은 대역폭으로 구동되며 하드 리얼타임(Hard Real-time) 특성을 요구받는다. 반면에 컴패니언 컴퓨터(Companion Computer) 상에서 주로 실행되는 컴퓨터 비전 처리 파이프라인이나 고수준 3차원 유클리드 공간 궤적 생성기(Trajectory Generator) 노드는 30Hz~50Hz(20ms~33ms 주기) 환경에서 실행되며, 상대적으로 여유로운 소프트 리얼타임(Soft Real-time) 제약을 지닌다. 즉, 이 두 이기종(Heterogeneous) 프로세서 영역 간의 연산 특성을 올바르게 인지하고 마감 시간 이내에 데이터 통신을 완료하도록 스케줄링하는 것이 자원 할당 이론의 대전제이다.

3. 실시간 스케줄링 이론의 PX4 및 Linux 적용 기법

NuttX RTOS를 커널로 채택하는 PX4 비행 제어 시스템의 내부, 그리고 통상적인 리눅스(Linux) 운영체제에 탑재되는 임무 컴퓨터 기반 ROS 2 환경에서는 스케줄링 알고리즘이 시스템의 결정론적 타이밍(Deterministic Timing)을 수호한다. 주로 우선순위 기반 선점형 스케줄링(Preemptive Priority Scheduling)이 핵심이 된다.

3.1 비율 단조 스케줄링 (RMS: Rate Monotonic Scheduling) 이론

정적인 주기성을 갖는 태스크(Task) 모델에서 수행 주기가 매우 짧은(빈도수가 높은) 태스크일수록 더 높은 실행 우선순위를 고정적으로 부여하는 전형적 스케줄링 기법이다. PX4 내부 모듈 설계 시 자이로스코프(Gyroscope) 데이터 폴링 스레드 및 각속도(Rate) 제어 루프는 가장 짧은 실행 주기를 가지므로 최상위 우선순위를 수여받는다. 반면, 상대적으로 주기가 길고 중요도가 낮은 SD 카드 로깅(Logging)이나 배터리 모니터링은 하위 스레드 우선순위로 밀려나게 된다.
Liu & Layland가 발표한 “Scheduling Algorithms for Multiprogramming in a Hard-Real-Time Environment” (1973) 연구에서 검증된 바와 같이, 일정 한계 수준 이하의 CPU 사용률(Utilization bound) 내에서 RMS는 다중 주기적 태스크 간의 데드라인 만족 확률을 이론적으로 완벽히 보장한다.

3.2 마감 시간 우선 스케줄링 (EDF: Earliest Deadline First)

동적인 관점의 스케줄링 기법으로서, 각 실행 태스크의 데드라인 타임스탬프가 다가오는 순서대로 즉각 최고 우선순위를 업데이트 부여한다. ROS 2의 rclcpp 실행기(Executor) 레벨 내지 이벤트 큐 관리에 일부 적용하여 활용성을 높일 수 있으며, CPU 자원 활용률 관점에서 RMS 대비 스케줄링 능력이 뛰어나다. 그러나, 시스템 과부하 상태(Overload condition) 혹은 캐시 미스(Cache Miss) 현상 발생 시 순간적인 도미노 타임오버(Domino Effect Rate Miss) 현상에 취약하다는 분명한 한계점이 교차한다.

4. 임무 컴퓨터 시스템 내의 자원 분리 전략 (Resource Partitioning)

고도의 계산 복잡도를 요하는 오프보드 제어를 달성하기 위해 비행 제어기와 임무 컴퓨터가 이더넷(Ethernet) 채널이나 고속 직렬 터미널(Serial) 채널 상의 uXRCE-DDS 프로토콜로 직접 결합되는 환경의 연산 분산 모델을 구축하라. 이는 즉 PX4 기반 FC는 확장칼만필터(EKF2) 기반 상태 추정(State Estimation)과 자세(Attitude) 제어와 같이 초정밀 타이밍 억제가 요구되는 로우 레벨 루프 연산을 수행하고, 무거운 컴퓨팅 파워 연산—카메라 또는 LiDAR 센서 입력, VIO/SLAM 알고리즘 수행, 글로벌 경로 계획(Global Path Planning)—은 임무 컴퓨터에게 전담시키는 **계층적 제어 권한 분리 모델(Hierarchical Control Authority Separation Model)**을 의미한다.

5. Linux PREEMPT_RT 및 ROS 2 오프보드 노드의 제어 스케줄링 적용

QGroundControl(v4.3 이상)과 같은 관제 소프트웨어가 단순 관제 기능 위주로 활용된다면, 실제 비행 궤적을 제어하고 명령을 내보내는 ROS 2 오프보드 에이전트 측에서의 제어기 품질 무결성이 더욱 중요하다. 따라서 컴패니언 계층 내 타이밍 위반 오류를 방지하고 제어 루프를 스케줄링하기 위해 리눅스 커널의 실시간성을 획득하는 조치를 반영하라.

  1. CPU 선호도 고정(CPU Affinity & Isolation) 및 Cgroups 정책 설정:
    임무 컴퓨터 시스템(예: 8-core CPU 구조) 환경 내에서 특정 CPU 코어(예컨대 Core 6, 혹은 7)를 ROS 2의 핵심 오프보드 제어 위치(Position), 속도(Velocity) 스트리밍 노드 전용으로만 점유하도록 격리(isocpus 커널 파라미터 적용)시켜, 리눅스의 여타 일반 프로세서들이 해당 컴퓨팅 스레드를 뺏어가는 CPU 선험 현상(Preemption)을 철저하게 방지하라.

  2. 스케줄링 우선순위 등급업 (SCHED_FIFO / SCHED_RR):
    운영체제단의 기본인 SCHED_OTHER 시분할(Time-sharing) 정책 대신, POSIX 스레드 라이브러리 pthread 내지는 chrt 유틸리티를 통한 SCHED_FIFO(First In First Out) 스케줄러 정책으로 등급을 상승 기입시켜, 선점형 제약 속성을 획득함으로써 핵심 제어 워크로드 구동 중의 유휴 문맥 교환(Context Switching Latency) 시간 손실을 수 마이크로초 단위로 극소화하라.

  3. QoS 오프로드 및 무복사 전송 체계 수립 (Data Distribution Service - Zero Copy):
    uXRCE-DDS 환경과 Fast DDS 설정을 연동하여 신뢰할 수 있는 전송 규격(RELIABLE Reliability QoS Profile) 프로파일과 적시성의 마감 시간 규제(Deadline QoS) 프로파일을 동시에 선언 적용시켜라. 이는 이종 프로세서 직렬 버스 간의 송수신 정체 및 지터(Jitter) 현상을 낮춤으로써 자원 대역폭 제약을 근본적으로 완화하게 된다.

6. Ardupilot과 대비된 PX4 고유 자원 분리 철학

가장 큰 틀에서 바라보면, PX4는 최하위 시스템 계층에서부터 uORB (Micro Object Request Broker) 기반의 극대화된 모듈형 컴포넌트 아키텍처 사상을 바탕으로 한다. NuttX RTOS 환경 하에서 각각 독립된 스레드/프로세스(Thread/Process) 형태로 기동하므로 태스크 간의 자원 간섭을 선천적으로 차단한다. 이에 비해 구버전 아두파일럿(Ardupilot) 아키텍처는 주로 단일 스레딩(Single-threaded) 환경에서의 협력적 다중 코어(Cooperative Multitasking) 설계에 무게를 두었던 역사가 있어 시스템적 차이가 존재했다(현재는 아두파일럿도 ChibiOS 및 다중 스레드 구동 체계로 이식됨).
특히 PX4는 ROS 2 생태계 통합(DDS Middleware 탑재)에 있어 설계 단계부터 각별한 아키텍처 고려가 포함되어 있어, 대규모 실시간 분산 노드들과 상호 연산 자원 할당 관점에서 데이터 통신 파이프라인(Broker-less)을 구성하고 보장하는 데 상대적 우위를 보유한다.

결과적으로 최신 이산 시간 하이브리드 제어기(Hybrid Controller) 환경의 안정성 유지는 프로세서를 점진적으로 고도화(Scale-up)하는 물리적 확충 이외에도 각 태스크와 GNC 요소 프로그램들의 실행 우선순위와 주기를 수리적 제약 모델로 분할, 격리, 선점 보장하는 고도의 아키텍처링 프로세스를 반드시 수반하며 구축되어야 한다.
