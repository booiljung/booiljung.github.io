63.4.1. Offboard 제어 목적의 객체 지향적 발행자(Publisher) 및 구독자(Subscriber) 소프트웨어 디자인 패턴 설계

본 절에서는 ROS 2 기반의 PX4-Autopilot Offboard 제어 시스템에서 유지보수성, 확장성, 그리고 테스트 용이성(Testability)을 극대화하기 위한 객체 지향적(Object-Oriented) 소프트웨어 디자인 패턴의 설계 원칙과 실제 구현 패러다임을 기술한다. 본 내용은 PX4 v1.14.0 펌웨어 및 ROS 2 Humble 미들웨어 환경을 기준으로 명세한다.

1. 객체 지향적 노드 아키텍처의 필요성

자율 비행 드론 제어 노드(Control Node)는 통상적으로 수많은 센서 데이터 및 기체 상태를 연속적으로 구독(Subscribe)하고, 동시에 다양한 제어 명령과 궤적 설정점(Setpoint)을 빈번하게 발행(Publish)해야 한다. 기존의 단순 절차적(Procedural) 프로그래밍 방식으로 이를 하나의 거대한 단일 클래스나 콜백 함수 덩어리로 구현하게 되면, 멤버 변수 간의 결합도(Coupling)가 비정상적으로 상승하여 시스템의 신뢰성에 치명적인 동기화 버그(Synchronization Bug)를 초래한다.

따라서 단일 책임 원칙(Single Responsibility Principle, SRP)에 철저히 입각하여 발행자(Publisher)와 구독자(Subscriber) 역할을 독립적인 캡슐화(Encapsulation) 클래스로 분리하고, 주 제어 루프 알고리즘을 관리하는 퍼사드(Facade) 역할의 코어 메인 클래스가 이들을 유기적으로 통제하도록 아키텍처를 설계하라.

2. 소프트웨어 디자인 패턴 아키텍처 구조도

아래는 객체 지향 원칙에 따른 Offboard 제어 시스템의 논리적 클래스 연관 관계를 나타내는 UML(Unified Modeling Language) 구조도이다.

3. 구독자(Subscriber) 컴포넌트 설계 전략

구독자 컴포넌트 클래스는 PX4-Autopilot으로부터 인입되는 고빈도 상태 변수들을 실시간 캐싱(Caching)하고, 멀티스레드 제어기에 맞추어 스레드 안전성(Thread Safety)을 완벽히 보장하는 접근자(Getter) 인터페이스를 제공해야 한다.

  • 순수 콜백 로직 유지: 구독자 콜백 함수 내부에서는 연산 비용이 큰 알고리즘 처리를 절대 병행 실행하지 마라. 단순히 수신된 최신 메시지를 내부 멤버 구조체 혹은 메모리 공간에 원자적(Atomic)으로 갱신하는 역할만 수행해야 한다.
  • 최신 상태 보존 원칙: VehicleOdometry(EKF2 위치 및 속도 추정치) 및 VehicleStatus(Arming 및 항법 모드 상태) 메시지는 제어 루프 타이머가 언제든지 뮤텍스(Mutex) 등의 오버헤드를 최소화한 상태로 동기화된 갱신 값을 참조 가능하도록 구성하라.

4. 발행자(Publisher) 컴포넌트 설계 전략

발행자 컴포넌트 클래스는 시스템 제어기의 용도 목적에 철저하게 맞추어 발행 토픽 계층의 역할을 수직적으로 구분해야 한다.

  • 상태 전이 명령어 인터페이스(CommandPublisher): VehicleCommand 메시지 체계를 활용하여 시동(Arming), 이륙(Takeoff), Offboard 모드 전환 등 단발성 비행 상태 전이 및 커맨더(Commander) 명령을 트리거하는 역할을 독립 캡슐화하라. 또한 Offboard 제어 권한을 마이크로 XRCE-DDS에 송출하기 위한 OffboardControlMode 메시지를 내부 클럭 타이머에 따라 주기적으로 발행하는 부가루틴 인터페이스를 마련하라.
  • 실시간 궤적 데이터 발행(TrajectoryPublisher): TrajectorySetpoint 메시지를 관장하여 글로벌 포지셔닝이나 로컬 위치, 속도, 가속도 기반의 실시간 목표 값을 송출하는 역할을 위임하라. ROS 2 기준 좌표계(ENU)에서 PX4의 내제적 기준 좌표계(NED)로 변환하는 좌표 변환(Coordinate Transformation) 로직은 메인 제어 루프에 종속시키지 말고 철저히 이 클래스 내부의 어댑터(Adapter) 단에서 자동 변환되도록 처리하여 응용 로직(Application Logic)의 복잡성을 대폭 낮추라.

5. C++ 컴포넌트 기반 소스 코드 설계 템플릿

소개된 디자인 패턴을 실무에 반영한 C++ rclcpp 인터페이스 기반의 핵심 컴포넌트 분리 구현 구조 예제이다. 아래 아키텍처 가이드라인을 엄수하여 설계하라.

#include <rclcpp/rclcpp.hpp>
#include <px4_msgs/msg/trajectory_setpoint.hpp>
#include <px4_msgs/msg/vehicle_odometry.hpp>
#include <mutex>

// 단일 목적 관점에 따라 분리된 상태 데이터 구독자 컴포넌트 클래스
class VehicleStateSubscriber {
public:
    VehicleStateSubscriber(rclcpp::Node* node) {
        // QoS(Quality of Service) 설정을 Sensor Data 혹은 Best Effort 방식에 일치시킴
        odom_sub_ = node->create_subscription<px4_msgs::msg::VehicleOdometry>(
            "/fmu/out/vehicle_odometry", rclcpp::QoS(10).best_effort(),
            [this](const%20px4_msgs::msg::VehicleOdometry::SharedPtr%20msg) {
                std::lock_guard<std::mutex> lock(mutex_);
                latest_odom_ = *msg;
            });
    }

    px4_msgs::msg::VehicleOdometry getLatestOdometry() {
        std::lock_guard<std::mutex> lock(mutex_);
        return latest_odom_;
    }

private:
    std::mutex mutex_;
    px4_msgs::msg::VehicleOdometry latest_odom_;
    rclcpp::Subscription<px4_msgs::msg::VehicleOdometry>::SharedPtr odom_sub_;
};

// 메인 제어기 통제 관리자 역할을 수행하는 코어 노드 (퍼사드 패턴)
class OffboardControlNode : public rclcpp::Node {
public:
    OffboardControlNode() : Node("offboard_control_node") {
        // 하위 구독 및 발행 역할을 담당하는 객체들의 초기화 및 생명주기 병합
        state_sub_ = std::make_shared<VehicleStateSubscriber>(this);
        
        // 50Hz 빈도로 작동하는 메인 제어 루프 타이머
        timer_ = this->create_wall_timer(
            std::chrono::milliseconds(20),
            std::bind(&OffboardControlNode::controlLoop, this));
    }

private:
    void controlLoop() {
        // 객체 지향적 접근자 인터페이스를 통한 스레드 세이프(Thread-safe) 상태 데이터 호출
        auto odom = state_sub_->getLatestOdometry();
        
        // TODO: 위치/속도 추종 피드백 루프 등 고도화된 궤적 생성 메커니즘 연산 후
        // 분리된 TrajectoryPublisher 클래스의 메서드 지시자를 활용해 안전한 명령 하달 수행
    }

    std::shared_ptr<VehicleStateSubscriber> state_sub_;
    rclcpp::TimerBase::SharedPtr timer_;
};

상기 명시된 방식처럼 모놀리식(Monolithic) 제어 루프를 완벽히 분할된 컴포넌트 아키텍처로 대체 설계하면, 소스 코드 유닛 테스트(Unit Test) 작성 과정에서 각 단위 모듈별 기능 검증, 그리고 MOCK 데이터를 활용한 분산 의존성 주입(Dependency Injection) 전략 수립이 한층 유연해진다.
