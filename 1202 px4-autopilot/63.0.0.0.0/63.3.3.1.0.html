63.3.3.1. 모드 전환 명령 패킷 전송 및 상태 토픽 내 주 비행 모드(Main State) 식별자 변경 상태의 교차 상관관계(Cross-correlation) 확인 알고리즘

동반 컴퓨터(Companion Computer)가 지상 관제 시스템(GCS)을 배제하고 단독으로 비행 모드를 오프보드(Offboard)로 전환할 때, 패킷의 전송만으로 제어권 획득을 확신하는 오픈 루프(Open-loop) 제어는 무선 환경에서 치명적인 결함을 갖는다. PX4-Autopilot(v1.14 이상)은 명령 수신 즉시 모드를 변경하는 것이 아니라 깐깐한 유효성(Validity) 검사를 마친 후에야 내부 상태를 변경한다. 본 절에서는 하향 링크(Downlink)를 통한 모드 전환 명령 발달과, 상향 링크(Uplink)를 통한 상태 갱신 피드백 사이의 시간적 차이(Time Lag)를 극복하기 위한 교차 상관관계(Cross-correlation) 검증 알고리즘을 설계한다.

1. 개루프(Open-loop) 판별의 위험성과 교차 검증의 필요성

Ardupilot MAVLink 스트림에서는 COMMAND_ACK 메시지만 확인하여 모드 전환의 성패를 가늠하는 경우가 잦다. 그러나 ROS2 기반의 uXRCE-DDS 분산 네트워크 구조에서 VehicleCommandAck 만을 맹신하는 것은, 실제 기체의 제어 루프가 그 모드 위로 온전히 올라탔음을 100\% 보증하지는 못한다.

예를 들어, 커맨더(Commander) 노드가 오프보드 전환 조건을 아슬아슬하게 통과시켜 ACCEPTED ACK를 발송했으나, 불과 수 밀리초(ms) 뒤에 TrajectorySetpoint 타임아웃이 발생하여 기체가 즉시 호버링(Hold 모드)으로 튕겨져 나가는(Failsafe) 상황이 발생할 수 있다. 이 경우 에이전트는 기체가 오프보드 모드에 있다고 착각하여 계속 궤적을 쏘지만, 기체는 그 명령을 무시(Ignore)하게 되는 ’상태 불일치(State Mismatch)’의 지옥에 빠지게 된다.

1.1 VehicleCommandAck와 VehicleStatus.nav_state의 이원화 구조

이러한 상태 전이의 비동기적 특성 때문에, 강건한 자율 에이전트 설계는 명령의 수락(ACK)을 확인하는 1차 관문에 이어, 기체의 내재적 상태(Intrinsic State)를 나타내는 VehicleStatus 토픽의 nav_state 필드가 정확하게 NAVIGATION_STATE_OFFBOARD (14)로 자리를 잡았는지를 검사하는 2차 관문(Cross-correlation)을 반드시 거쳐야 한다.

sequenceDiagram
    participant ROS as Companion Node (ROS2)
    participant DDS as uXRCE-DDS Agent
    participant CMD as PX4 Commander
    
    ROS->>DDS: CMD_DO_SET_MODE (OFFBOARD)
    DDS->>CMD: uORB: VehicleCommand
    CMD-->>DDS: uORB: VehicleCommandAck (ACCEPTED)
    DDS-->>ROS: ACK Callback Triggered
    Note left of ROS: 1st Verification Pass
    
    CMD->>CMD: Update Internal State Machine
    CMD-->>DDS: uORB: VehicleStatus (nav_state = 14)
    DDS-->>ROS: Status Callback Triggered
    Note left of ROS: 2nd Verification Pass (Cross-check)
    
    ROS->>ROS: State Mismatch Resolved!

2. 교차 상관관계(Cross-correlation) 기반의 재시도(Retry) 유한 상태 기계 알고리즘

명령 송출과 상태 갱신은 필연적으로 통신 대역폭과 ROS2 큐(Queue) 사이즈에 종속된 전파 지연(Propagation Delay)을 수반한다.

2.1 지연 허용(Delay Tolerance) 폴링(Polling) 루프 설계

두 정보의 교차 검증을 위해 오프보드 제어기(Offboard Controller Node)는 비동기 호출 사이에서 안전판 역할을 할 윈도우(Window) 기반의 폴링 루프를 가동해야 한다.

  1. 명령 래치(Latch): 에이전트는 DO_SET_MODE 명령을 발송한 직후, 로컬 래치 변수 last_mode_request_time에 현재 타임스탬프를 기록한다.
  2. 이중 콜백 모니터링 분석:
  • VehicleCommandAck 콜백에서 해당 명령이 거부(DENIED 또는 TEMPORARILY_REJECTED)된 것으로 나타나면 즉각 재전송 로직으로 회귀한다.
  • ACK가 ACCEPTED를 가리키더라도 에이전트는 오프보드 활성화를 선언(Declaration)하지 않는다. 대신 VehicleStatus 구독 콜백을 살피며 기체의 실제 nav_state가 14(오프보드)로 덮어씌워졌는지 감시한다.
  1. 타임아웃 및 동기화 무결성: 명령 송출 후 \sim500\text{ms}의 인내 창(Patience Window) 내에 nav_state == 14 조건이 달성되면, 오프보드 제어권이 100\% 확보되었음을 확정 짓고 실제 동역학 타겟 생성을 시작한다. 만약 \text{ACK}만 오고 nav_state가 변경되지 않는 교착 현상이 500\text{ms}를 초과한다면, 노드는 시스템 붕괴를 경계하며 다시 예열 스트림(Warm-up Streaming) 단계로 롤백(Roll-back)하여 모드 전환을 1단계부터 재시도해야 한다.

결론적으로, 패킷 전송 이력과 수신 측 상태 토픽 간의 교차 상관관계를 끈질기게 추적하는 FSM(Finite State Machine) 아키텍처는 보이지 않는 전파 공백과 운영체제 스케줄러의 변덕 속에서도 통제권을 놓치지 않는 가장 견고한(Robust) 소프트웨어 공학적 예방주사이다.
