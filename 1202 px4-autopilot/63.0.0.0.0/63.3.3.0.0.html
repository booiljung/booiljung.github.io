63.3.3. 비행 모드 전환 명령을 통한 Offboard 모드 강제 진입 제어기 및 상태 천이 알고리즘

지상 관제 시스템(GCS) 통제나 조종기 수동 입력이 아닌, 동반 컴퓨터(Companion Computer) 내부의 로직 판단만으로 기체의 주 비행 제어권(Main Flight Control Authority)을 빼앗는 ‘모드 전환(Mode Switch)’ 로직은 오프보드(Offboard) 자율 비행의 중추이다. PX4-Autopilot(v1.14 이상) 체계에서는 이 모드 전환 과정이 단순한 플래그 변경이 아니며, 커맨더(Commander) 내부의 상태 기계(State Machine)를 비틀어 강제로 오프보드 가지(Branch)로 진입시키는 적극적인 액션(Action)이다. 본 장에서는 VehicleCommand를 이용한 오프보드 모드 강제 진입의 제어 공학적 알고리즘과 그 전제 조건들을 상세히 서술한다.

1. MAVLink DO_SET_MODE와 VehicleCommand 매핑 구조

Ardupilot 통제 시스템에서는 MAV_CMD_DO_SET_MODE 명령을 사용하거나, 커스텀 비행 모드 번호(예: GUIDED 모드의 경우 4)를 직접 할당하는 방식으로 외부 제어를 활성화한다. 상대적으로 관대한 Ardupilot의 GUIDED 모드와 달리, PX4의 OFFBOARD 모드는 진입 장벽이 높으며 커스텀 모드(Custom Mode) 서브 파라미터를 이용한 다중 계층(Multi-layered) 인식 체계를 사용한다.

ROS2 uXRCE-DDS 브릿지를 통과하여 PX4 펌웨어 내부로 유입되는 모드 전환 지령은 VehicleCommand 토픽을 통해 처리된다. 오프보드 모드로의 강제 진입을 위해 에이전트는 다음 파라미터 셋을 정확히 일치시켜 발행(Publish)해야 한다.

  • command: VehicleCommand::VEHICLE_CMD_DO_SET_MODE (MAVLink 176 대응)
  • param1: 1.0 (베이스 모드, VEHICLE_MODE_FLAG_CUSTOM_MODE_ENABLED)
  • param2: 6.0 (메인 모드, PX4_CUSTOM_MAIN_MODE_OFFBOARD)

이 세 가지 파라미터의 교집합적 타격(Hit)만이 비행 제어기 커맨더의 상태 천이 트리거(Transition Trigger)를 당길 수 있다.

2. 모드 전환 강제 진입을 위한 상태 천이(State Transition) 알고리즘

논리적으로 완결된 자율 비행 노드(Node)는 단순히 모드 전환 명령을 한 번 쏘고 끝나는 오픈 루프(Open-loop) 제어로 작성되어서는 안 된다.
모드 전환 명령은 무선 링크의 패킷 드롭(Packet Drop)이나 GCS 측의 동시 제어(Race Condition)에 의해 무시될 수 있으므로, 반드시 확인 응답이 수반되는 폐루프(Closed-loop) 상태 기계 폴링(Polling) 알고리즘으로 구성되어야 한다.

2.1 예열(Warm-up) 스트리밍 강제성

가장 직관을 거스르는 PX4 오프보드 모드의 특징은 “허락을 받기 전에 먼저 일을 시작해야 한다“는 역설이다. 커맨더 모듈은 DO_SET_MODE(OFFBOARD) 명령을 수신한 시점(Tick)에서 과거를 되돌아본다.

만약 모드 전환 명령 수신 시점 이전에 최소 0.5초 이상 TrajectorySetpoint 토픽 스트림이 2\text{Hz} 이상의 주기로 도착했던 이력이 없다면, 커맨더는 “외부 에이전트가 아직 제어 준비가 되지 않았다“고 간주하여 즉각적으로 모드 전환을 거부(Deny)한다. 따라서 에이전트 노드의 FSM 알고리즘은 다음과 같이 전개되어야 한다.

stateDiagram-v2
    [*] --> STREAM_INIT
    
    STREAM_INIT --> PRE_WARM_UP : Start 50Hz Timer
    Note right of PRE_WARM_UP: Publish Setpoint (Current Pos)
    
    PRE_WARM_UP --> REQUEST_OFFBOARD : After 500ms
    Note right of REQUEST_OFFBOARD: Publish CMD_DO_SET_MODE(Offboard)
    
    REQUEST_OFFBOARD --> CHECK_ACK : Wait ACK
    CHECK_ACK --> REQUEST_OFFBOARD : ACK = DENIED / Timeout\n(Retry Loop)
    CHECK_ACK --> OFFBOARD_ACTIVE : ACK = ACCEPTED
    
    OFFBOARD_ACTIVE --> [*] : Active Control Phase

2.2 비동기 재전송(Asynchronous Retry) 및 폴링 타이머 설계

오프보드 진입 노드는 진입 거부 시 무한 루프 블로킹(Blocking)에 빠지지 않도록 유의해야 한다. ROS2 타이머(Timer) 콜백 안에서 비동기 상태 기계를 설계하여, 특정 타임아웃(예: 1.0초) 내에 모드 전환 승인(ACK)이 도착하지 않거나 VehicleStatus 상의 nav_state가 NAVIGATION_STATE_OFFBOARD로 갱신되지 않으면, 다시 재전송(Retry)을 시도하는 능동성(Liveness) 보장 로직이 배후에 깔려 있어야 한다.

  • 유의점: 재전송 시도 중에도 앞선 50\text{Hz} 주기의 TrajectorySetpoint 발행은 결코 멈추어서는 안 된다. 타겟 발행 주기가 멈추면 앞서 공들여 쌓은 ’0.5초 스트림 이력’이 다시 붕괴되어 영영 오프보드 모드에 진입하지 못하는 데드락(Deadlock)에 빠질 수 있다.

이처럼 PX4의 오프보드 강제 진입 체계는 단방향 명령 하달(Command Dispatch) 구조가 아니라, 에이전트의 제어 의지와 비행 제어기의 수용 능력이 실시간으로 교섭(Negotiation)하는 쌍방향, 시계열 기반의 상태 천이 프로토콜(State Transition Protocol)로 설계되어 있음을 명심해야 한다.
