63.9.3. 하드웨어 인터럽트(Hardware Interrupt) 기반 독립형 워치독 타이머(Watchdog Timer)를 이용한 제어 노드 무한 루프 이탈 회피 및 코어 덤프(Core Dump) 로깅을 포함한 안전 종료 마이크로 아키텍처

자율 비행 통합 시스템의 임무 컴퓨터(Mission Computer)에서 구동되는 운영체제(Linux 커널) 또는 ROS2 미들웨어 자체가 완전히 응답을 멈추는 커널 패닉(Kernel Panic), 하드웨어 종속적인 교착 상태, 혹은 인터럽트 폭풍(Interrupt Storm)에 빠질 경우, 앞서 다룬 소프트웨어 레벨의 감시 체계(Systemd Watchdog 등)는 CPU 자원을 얻지 못해 무용지물이 된다. 이러한 궁극적인 물리적 프리징(Freezing) 현상을 극복하고 시스템의 결정론적 생존성을 부여하기 위해서는 메인 컴퓨팅 로직과 물리적으로 분리된 하드웨어 인터럽트(Hardware Interrupt) 기반의 독립형 워치독 타이머(Watchdog Timer, WDT) 설계가 필수적이다.

본 절에서는 임무 컴퓨터 내장 WDT 하드웨어를 활용하여 ROS2 제어 노드의 무한 루프(Infinite Loop)를 회피하고, 셧다운 전 짧은 유예 시간(Grace Period)을 이용하여 코어 덤프(Core Dump) 로깅을 시도하는 안전 종료 마이크로 아키텍처(Fail-Safe Micro Architecture)를 심도 있게 다룬다.

1. 하드웨어 워치독 타이머(WDT)의 작동 원리와 독립성

하드웨어 워치독 타이머는 임무 컴퓨터의 메인 브레인 역할을 하는 SoC(System on Chip) 내부에 독립적인 클럭 도메인으로 내장되거나, I2C/SPI 버스로 연결된 별도의 외부 마이크로컨트롤러(MCU) 칩 형태로 존재한다.

  • 정상 구동 상태 (Kicking/Feeding): ROS2 오프보드 제어 노드 혹은 전용 헬스 모니터 프로세스가 일정한 주기(예: 1초 간격)로 /dev/watchdog 디바이스 노드에 특정 매직 문자(Magic Character, 예: V)를 기록하거나 하드웨어 핀(WDI)을 토글링하여 워치독 내부의 카운터를 초기화(Reset)한다.
  • 장애 상태 (Timeout): 임무 컴퓨터 커널이 정지하거나 제어 프로세스가 무한 루프에 빠져 Kicking 주기를 놓치면, 하드웨어 WDT의 카운터가 0에 도달하여 장애(Timeout) 이벤트가 발동된다.

이때 WDT는 시스템 전원 관리 칩(PMIC)에 직접 전기적 리셋 신호를 인가하여 강제 재부팅(하드 리셋, Hard Reset)을 시도함으로써, 미지의 교착 상태를 탈출하는 “궁극적인 소프트웨어 재생(Rejuvenation)” 스위치 역할을 수행한다.

2. Pre-Timeout (사전 타임아웃) 기반 코어 덤프(Core Dump) 로깅 아키텍처

강제 리셋은 시스템을 교착 상태에서 해방시키지만, 개발자 관점에서는 ’왜 뻗었는지’에 대한 디버깅 단서를 모두 날려버리는 치명적인 부작용을 갖는다. 사후 분석(Post-Mortem Analysis)의 데이터 보존을 위해 현대 컴퓨팅 아키텍처는 WDT의 사전 타임아웃 인터럽트(Pre-Timeout Interrupt) 기능을 활용한다.

이 기법은 하드 리셋을 가하기 수 초 전(예: 타임아웃 10초 설정 시, 8초 경과 시점)에 우선순위가 매우 높은 NMI(Non-Maskable Interrupt, 마스크 불가능 인터럽트) 혹은 프로세서 고유의 하드웨어 인터럽트를 유발하는 구조이다.

  1. Pre-Timeout 감지: WDT 타이머가 1차 임계값에 도달하면 즉시 NMI가 커널로 전달된다. NMI는 다른 어떤 소프트웨어 레벨의 인터럽트 차단(Interrupt Masking)이나 무한 루프 상태라도 강제로 실행 흐름을 가로챈다.
  2. 커널 패닉 유도 및 코어 덤프 (Graceful Logging): NMI 핸들러는 시스템이 파국에 이르렀음을 인지하고 즉각 강제 커널 패닉(Panic) 루틴을 호출한다. 커널은 kdump와 같은 크래시 덤프 체계를 이용해 완전히 멈추기 전 확보한 레지스터(Register) 스냅샷, 프로세스 메모리 상태, 최근 커널 로그 트레이스(dmesg)를 eMMC 플래시나 비휘발성 메모리에 신속하게 코어 덤프로 기록(Flush)한다.
  3. 최종 하드 리셋: 코어 덤프 기록이 완료될 즈음, 2차 임계값이 만료되면서 WDT가 직접 하드웨어 파워 라인을 끊었다 다시 켬으로써 극적인 재부팅을 성사시킨다.

4. 아키텍처 설계 시 PX4-Autopilot 설정 연동 고려사항

이 마이크로 아키텍처가 실효성을 거두기 위해서는 “임무 컴퓨터가 덤프 메모리를 기록하고 완전히 하드 리셋되어 재부팅되는 시간” 동안 기체가 안정적으로 하늘에 떠 있어야 한다는 전제가 따른다. 통상적인 임무 컴퓨터의 콜드 부팅(Cold Booting) 시간은 최소 15초 이상이 소요된다.

따라서 PX4 파라미터 COM_OF_LOSS_T (오프보드 상실 타임아웃)은 WDT 타임아웃보다 충분히 짧게 설정되어야 하며, COM_OBL_ACT 파라미터는 1(Hold 모드)이나 0(RTL 모드)로 세팅되어 기체가 비상 비행(Failsafe Flight) 상태로 자구책을 유지할 수 있어야 한다. 부팅이 완료된 임무 컴퓨터는 곧이어 백업 덤프가 쓰여졌는지 점검하고, MAVLink나 uORB VehicleCommand 메시지를 통해 PX4로 Offboard 인계 요청을 발송하여 무결점 비행 재개(Seamless Resumption) 시나리오를 완성하게 된다. 이처럼 소프트웨어적 파국을 하드웨어 인터럽트로 탈출하는 전략은 군용 및 상업용 자율비행 통합 관제 시스템 설계에서 심장과 같은 역할을 담당한다.
