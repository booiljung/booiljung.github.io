63.2.1 uXRCE-DDS 브릿지를 통한 제어 데이터 직렬화 및 역직렬화(Serialization/Deserialization) 메커니즘

1. 개요

PX4-Autopilot과 ROS 2 간의 오프보드(Offboard) 제어 환경에서는 이기종 시스템 간의 빈번한 이진 데이터(Binary Data) 교환이 발생한다. 비행 제어기(FC)의 메모리에 상주하는 uORB(Micro Object Request Broker) 데이터 객체 구조체는 네트워크 전송 매체를 통과하기 위해 1차원의 평면적인 바이트 배열(Byte Array)로 변환되어야 하며, 이를 직렬화(Serialization)라 일컫는다. 반대로 수신된 바이트 스트림의 포맷을 해독하여 본래의 ROS 2 메시지 객체로 복원하는 과정을 역직렬화(Deserialization)라 한다. 본 절에서는 uXRCE-DDS 브릿지 아키텍처 환경에서 어떠한 원리로 데이터의 손실 없이 고속의 직/역직렬화가 일어나는지, 뼈대가 되는 Micro CDR(Common Data Representation) 표준의 메커니즘을 상세히 다룬다.

2. Micro CDR (Common Data Representation) 규격의 이해

DDS(Data Distribution Service) 미들웨어 네트워크망은 객체의 상태를 공유하기 위해 OMG(Object Management Group)에서 제정한 CDR 직렬화 표준 포맷을 이용한다. PX4-Autopilot은 MCU(Microcontroller Unit) 레벨의 제한된 메모리와 연산 속도 환경을 극복하기 위하여, eProsima사에서 경량 모델로 설계한 Micro CDR 라이브러리를 채택하였다.

  • 플랫폼 하드웨어 독립성: 서로 다른 프로세서 명령어 아키텍처(예: ARM Cortex-M 기반 비행 제어기와 x86 혹은 ARM Cortex-A 기반 임무 컴퓨터) 간의 차이점, 특정한 바이트 정렬 방식인 엔디안(Endianness) 불일치와 구조체의 메모리 패딩(Memory Alignment) 차이를 극복하도록 설계되었다.
  • 통신 프로토콜 경량성: XML이나 JSON 형식과 같은 무거운 메타데이터 텍스트 전송을 완전히 배제하고, 순수하게 데이터 타입(Payload) 기준의 바이트 길이로 패키징하여 무선 시리얼(UART) 통신처럼 좁은 대역폭(Bandwidth) 환경하에서도 월등히 높은 처리량(Throughput)을 보장한다.

3. 직렬화 및 역직렬화 처리 파이프라인 (Pipeline) 구조

통신 속도의 극대화를 위해 PX4의 직렬화 코드는 펌웨어 빌드 시점에 .msg 구조체 정의 파일로부터 파이썬(Python) 기반의 코드 제너레이터(Code Generator) 도구를 통해 C/C++ 최적화 코드로 자동 생성(Auto-generation)된다.

3.1 단계별 메커니즘 분석

  1. uORB 메시지 갱신 (Publishing): 비행 제어기의 내부 모듈이나 센서 시스템이 VehicleOdometry 혹은 TrajectorySetpoint와 같은 제어용 uORB 토픽을 갱신한다.
  2. uXRCE-DDS Client의 폴링(Polling) 수집: 펌웨어 내부의 브릿지 클라이언트 모듈이 갱신된 uORB 인스턴스의 메모리 블록을 감지하고 확보한다.
  3. Micro CDR 직렬화 (Serialization) 수행:
  • 내부의 자동 생성 코드가 호출되어, 각 데이터 타입(예: 8바이트의 uint64_t, 4바이트의 float32) 길이 규격에 맞게 전송용 버퍼 버스에 순차적으로 값을 기록한다.
  • 워드(Word) 메모리 접근 속도 효율을 극대화하기 위해, 4바이트 또는 8바이트 자료형이 등장할 경우 적절한 패딩(Padding) 비트를 삽입하여 정렬(Memory Alignment)을 강제한다.
  • 통신망의 표준 규격으로 리틀 엔디안(Little-Endian) 포맷이 설정되며, 빅 엔디안 시스템일 경우 바이트 스와핑(Byte Swapping) 로직을 거친다.
  1. 프레이밍 및 송출: XRCE 프레이밍(Framing) 규격 가이드라인에 편승하여 스트림 헤더(Header) 및 무결성 검증을 위한 체크섬(Checksum/CRC)을 바이트 꼬리에 덧붙인 뒤 물리 계층(UART/UDP)을 통해 임무 컴퓨터(Agent) 측으로 송출한다.
  2. 역직렬화 (Deserialization) 수행: ROS 2 환경에 안착한 Micro-XRCE-DDS Agent가 수신된 바이트 스트림 묶음의 헤더를 검사한 후, 내장된 Micro CDR 역직렬화 루틴을 되짚어 호출함으로써 px4_msgs::msg::TrajectorySetpoint와 같은 완전한 ROS 2 IDL(Interface Definition Language) C++ 객체로 복원(Reconstruction)한다.

4. Ardupilot의 고전적 직렬화 체계(MAVLink)와의 기술적 차이

Ardupilot을 포함한 전통적 GCS 환경은 주로 MAVLink 프로토콜의 C 라이브러리가 제공하는 패키징 방식에 전적으로 의존하여 직렬화를 수행한다. MAVLink 규격은 메시지 고유 ID와 고정된 구조체 형식을 하드코딩으로 매핑하여 직렬화를 수행하므로 매우 가볍지만, 동적 배열(Dynamic Array) 문자열이나 극도로 복잡하게 중첩된 구조체(Nested Structures)를 직렬화 및 확장하는 데에는 심각한 문법적 한계점을 낳는다.

반면에 현재 PX4의 설계인 uXRCE-DDS(Micro CDR) 방식은 객체지향 미들웨어 표준을 뿌리째 계승하고 있으므로, IDL을 통해 복잡하게 정의된 고차원 데이터 타입이나 다차원 행렬을 구조적 오버헤드 없이 직렬화하여 ROS 2 토픽으로 즉시 투영(Projection)해낼 수 있다. 이는 복잡화된 코드를 단순화하는 객체 상호운용성(Interoperability)의 극대화라는 측면에서 Ardupilot의 기존 관제 체계와 확연히 대비되는 차별점이다.

5. 결론

uXRCE-DDS 브릿지를 통과하는 직렬화 및 역직렬화 메커니즘은 PX4-Autopilot과 ROS 2 간에 필연적으로 존재하는 이기종 컴퓨터 하드웨어 아키텍처의 물리적/소프트웨어적 차이를 완벽히 덮어버리는 추상화(Abstraction) 기술의 정수를 보여준다. Micro CDR 기반의 효율적인 바이트 정렬과 컴파일 타임(Compile-time)에 일괄 생성된 고속 직렬화 로직을 기반으로, 드론의 뇌와 컴퓨터 비전을 구동하는 임무 컴퓨터는 논리적으로 완벽하게 동일한 메모리 구조체 객체를 지연 없이 고속으로 공유 및 연산할 수 있다.
