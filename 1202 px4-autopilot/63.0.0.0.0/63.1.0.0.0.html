63.1. Offboard 모드(Offboard Mode)의 핵심 개념과 제어 권한 분리 설계

무인항공기 시스템(UAS)의 자율성을 극한으로 끌어올리기 위해서는 비행 제어기(Flight Controller) 단일 컴퓨팅에 의존하는 것을 넘어, 임무 컴퓨터(Mission Computer)와의 유기적인 결합이 필수적이다. PX4-Autopilot 환경에서 이러한 결합의 핵심 매개체가 바로 Offboard 모드(Offboard Mode)이다. 본 절에서는 Offboard 모드의 근본적인 개념을 확립하고, 제어 권한(Control Authority)이 어떻게 임무 컴퓨터와 비행 제어기 사이에서 안전하게 분리 및 위임되는지 아키텍처 관점에서 심도 있게 분석한다.

1. Offboard 모드의 핵심 프레임워크 이해

Offboard 모드는 기체의 조종사(인간) 혹은 내장된 기초적인 웨이포인트(Waypoint) 미션이 제어하는 단계를 뛰어넘어, 기체 외부(Off-board)에 실장된 고성능 컴퓨팅 모듈이 실시간으로 궤적, 속도 벡터, 가속도 등을 연산하여 기체를 제어하도록 허용하는 비행 모드이다.

이러한 설계의 기저에는 ’관심사의 분리(Separation of Concerns)’라는 소프트웨어 공학적 철학이 깔려 있다. 비행 제어의 안정화 알고리즘(PID 루프 기반 자세 제어 및 모터 믹싱)은 실시간성이 극도로 보장된 PX4 내장 실시간 운영체제(NuttX RTOS)가 전담하고, 장애물 회피, 영상 기반 객체 추적, 그리고 3차원 공간 복원력 연산 등 연산 복잡도(Computational Complexity)가 높은 알고리즘은 비전 센서나 카메라 데이터 처리에 특화된 외부 보조 컴퓨터(Companion Computer, 예: Raspberry Pi, NVIDIA Jetson 등)가 수행하도록 각자의 역할을 명확히 분리한다.

1.1 MAVLink 로직과 ROS2 uXRCE-DDS 기술의 병립

외부에서 PX4를 제어하기 위한 수단으로는 오랫동안 MAVLink 프로토콜의 SET_POSITION_TARGET_LOCAL_NED 메시지가 표준으로 군림해왔다. 이 메시지 구조는 GCS 관제 환경이나 QGroundControl 소스코드를 이해하고 활용하는 GCS 개발을 위한 관제 기술에서 중추적 역할을 수행한다. 그러나 자율 에이전트 비행을 위한 관제 기술이 고도화됨에 따라 더 빠르고 데이터 대역폭이 높은 ROS2 미들웨어 연동이 필요해졌다. 따라서 최신 PX4에서는 Micro XRCE-DDS 레이어를 통해 ROS2 도메인과 직접적으로 제어 권한을 공유하는 방식을 전면적으로 채택하고 있다.

    participant MC as Mission Computer (ROS2/MAVLink)
    participant uORB as PX4 uORB Middleware

    MC->>uORB: Offboard Control Mode Enable (Heartbeat 1Hz+)

2. 권한 분리 설계(Authority Delegation Design)의 차이: Ardupilot 대비

PX4-Autopilot과 Ardupilot은 자동 비행을 구현하기 위한 제어 위임 아키텍처에서 확연한 철학적 차이를 보여준다.

Ardupilot은 전통적으로 개별 플라이트 모드(Flight Mode)보다는 GUIDED 모드 하나로 모든 외부 명령 체계를 통폐합하여 관리하는 경향이 강하다. 사용자가 단발적으로 좌표(Takeoff, Goto 등)를 던져주면 Ardupilot 내부의 네비게이터(Navigator)가 그 과정까지의 모든 경로 궤적 처리를 알아서 해결하는 방식이 주를 이룬다.

반면, PX4-Autopilot의 Offboard 모드는 제어 권한을 임무 컴퓨터에 전적으로 ’양도’하는 개념을 명확히 한다. 즉, PX4 내부 플래너는 Offboard 궤적 개입 구간 동안에는 잠들고, 임무 컴퓨터가 매 순간(Interval) 다음번 기체가 놓여야 할 상태 모델(Kinematic State)을 지속적으로 주입해야만 한다. 이를 통해 예측 불가능한 비선형적 제어(예: 실시간 장애물 회피로 인한 급격한 곡률 반경 수정)를 구현할 때 내장 메모리의 한계에 얽매이지 않고 임무 컴퓨터 측에서 자유자재로 연산할 수 있다.

3. 오프로딩 컴퓨팅을 위한 센서 데이터 파이프라인

조종사 개입을 배제한 완벽한 자율 Offboard 통제를 유지하기 위해서는 기체의 현재 위치를 알 수 있는 상태 추정기(Estimator/EKF2)의 신뢰성이 권한 분리의 핵심 전제이다.

  • GPS/GNSS 센서: 실외 환경에서의 기본적인 글로벌 좌표 획득 수단이다. QGroundControl 소프트웨어를 통해 센서 캘리브레이션을 진행하고, MAVLink 프로토콜을 통해 수신 위성 개수와 HDOP(수평 정밀도 저해 지수)를 모니터링하여 EKF2 혁신(Innovation) 값을 수용한다.
  • VIO(Visual Inertial Odometry) 센서: 비GPS(GPS-Denied) 실내 환경에서는 PX4-Autopilot이 지원하는 T265와 유사한 스테레오 비전 처리 센싱 기술이 반드시 융합되어야 한다. 이 센서 정보 역시 임무 컴퓨터 측에서 융합 연산을 거쳐 MAVLink ODOMETRY 메시지나 ROS2의 VehicleOdometry 토픽을 통해 PX4 비행 제어기로 파이프라인 처리되며, 비행 제어기는 이 외부 데이터를 자신의 IMU 등과 결합해 최종 위치를 결정한다.

4. 제어 기술 이해 및 소스코드 아키텍처

PX4 소스코드 내부를 들여다보면 Offboard 상태 관리는 궁극적으로 커맨더 모듈(commander.cpp)을 지나 비행 로직 스택 내부의 FlightTaskOffboard.cpp 파일에 의해 위임 받게 된다.

// 예제: FlightTaskOffboard.cpp 내부 제어 수락 로직의 간소화된 설계 개념
bool FlightTaskOffboard::updateInitialize()

    // 로직 검증: 오프보드 제어 모드 메시지가 유효하고 타임아웃 이내에 수신되었는가?
    bool offboard_control_mode_valid = _sub_offboard_control_mode.updated();

    if (offboard_control_mode_valid) {
        // 메시지의 세부 요구사항 검증 (Position 제어인지 자세 제어인지 플래그 평가)

        return true; // 권한 수용(Authority Accepted)
    }
    return false; // 권한 박탈 및 Failsafe 이행
}

이와 같이 PX4-Autopilot은 단순히 외부의 입력을 수동적으로 맹신하는 것이 아니라, 수신되는 제어 스트림의 타임스탬프와 플래그 상태를 능동적으로 평가함으로써 권한 분리를 달성한다. 즉, 임무 컴퓨터가 제어 권한을 양도받았을지라도, PX4 비행 제어기는 이 제어 신호가 수학적·시간적으로 무결한지 감시하며 시스템의 최소 안전을 보장하는 감시자(Supervisor)의 권한은 스스로 유지하는 독특한 이원적 안전(Dual Safety) 아키텍처를 취하고 있다. 나아가 이러한 권한 검증 알고리즘은 다중 에이전트 통신망 구성 시 개별 에이전트의 내부 결함 여부를 실시간으로 격리하는 데 필수적인 핵심 제어 기술로 기능한다.
