63.2.3.3. 기체의 전기적 무장(Arming) 상태, 비행 모드 식별자 및 에너지 시스템 배터리 잔량 피드백 토픽(VehicleStatus) 활용

오프보드(Offboard) 제어기나 지상 관제 시스템(GCS)이 제어 명령의 인가 여부를 결정하고 임무(Mission)의 계속성을 판단하기 위해서는 비행 제어기(FC)의 ‘현재 상태’ 확증이 필수적이다. PX4-Autopilot의 상향 링크(Uplink) 아키텍처에서 이 역할을 전담하는 핵심 브로드캐스트 토픽 체계가 바로 VehicleStatus와 BatteryStatus이다. 본 절에서는 기체의 무장(Arming) 전이, 주 비행 모드(Nav State)의 식별, 그리고 배터리 잔량 피드백 논리가 제어 공학의 생존성(Survivability) 설계에 어떻게 융합되는지 분석한다.

1. 전역 시스템 상태(System State) 피드백 구조

Ardupilot은 MAVLink의 HEARTBEAT 메시지에 베이스 모드(Base Mode)와 커스텀 모드(Custom Mode), 그리고 시스템 상태(System Status)를 융합하여 1Hz(기본값) 주기로 송출한다. 반면, PX4-Autopilot(v1.14 이상)은 이와 매핑되는 uORB 내부의 VehicleStatus 토픽을 별도로 분리하여 관리하며, ROS2 uXRCE-DDS 에이전트 환경에서 텔레메트리로 전송함으로써 상태 천이(State Transition) 모니터링 주기를 독립적으로 제어할 수 있도록 보장한다.

1.1 VehicleStatus의 핵심 식별자 구성

VehicleStatus 구조체를 구동하는 주축 논리는 다음과 같다.

  • arming_state: 물리적 모터 구동계의 활성/비활성 상태(예: ARMING_STATE_STANDBY, ARMING_STATE_ARMED, ARMING_STATE_STANDBY_ERROR 등)를 나타낸다. О프보드 제어 명령의 수락을 결정짓는 절대적 전제 조건(Precondition)이다.
  • nav_state: 현재 기체의 주 비행 모드를 식별하는 열거형(Enum) 변수이다(예: NAVIGATION_STATE_OFFBOARD, NAVIGATION_STATE_AUTO_MISSION). 커맨더(Commander) 노드에 의한 권한 스위칭 검증용 피드백으로 작용한다.
  • failsafe_and_user_cmd: 비상 제어(Failsafe) 플래그의 발동 여부를 직관적으로 표상하며, 동반 컴퓨터(Companion Computer)가 자신이 수행 중인 로직의 강제 무효화를 인지하는 데 쓰인다.

2. 에너지 시스템 토픽(BatteryStatus) 기반 강건성 설계

멀티로터형 기체는 추진 시스템 특성상 에너지 소모가 극심하며, 임무의 지속 가능성은 오로지 배터리 잔량과 방전율 에스컬레이션(Discharge Escalation) 커브에 종속된다. PX4는 스마트 배터리 모듈이나 전력 관리 보드(PMB)로부터 취득한 1차적 센싱값을 EKF 필터처럼 단일화하여 BatteryStatus 토픽으로 상향 전송한다.

2.1 자율 비행과 배터리 고갈 방어 로직

오프보드 경로 계획기(Path Planner)에게 BatteryStatus 토픽의 remaining 필드(0.0 ~ 1.0 비율)와 voltage_v 필드는 장애물 유무만큼이나 가중치가 높은 임무 비용 함수(Cost Function) 팩터이다.

  • 현존하는 고급 무인항공기 관제 기술(QGroundControl v4.3 및 최신 자율 에이전트)에서는 기체가 배터리 warning_level에 도달하기 전, 반환 가능 지점(Point of No Return)을 계산해야 한다.
  • MAVLink 기반으로 관제될 시 QGC는 배터리 잔량 퍼센티지를 임계치와 단순 비교하여 경고를 띄우지만, ROS2 기반 동반 컴퓨터 제어 체계에서는 수신된 voltage_v가 모터 부하로 인해 순간적으로 강하(Voltage Drop)하는 현상을 1차 통과 필터(Low-pass Filter)로 스무딩(Smoothing)하여 실제 소모율 예측 알고리즘에 보정 파라미터로 통합하는 것이 학술적/산업적 관례이다.

3. 오프보드 연계: 교착 상태(Deadlock) 방지 트리거

가장 위험한 통신 오류 시나리오 중 하나는 ’비대칭 상태 인지’이다. 동반 컴퓨터는 타겟을 향해 TrajectorySetpoint를 퍼블리싱하고 있으나, 비행 제어기 측에서는 이미 배터리 저전압 Failsafe가 발동하여 nav_state를 NAVIGATION_STATE_AUTO_RETURN으로 강등(Downgrade)시킨 상황을 들 수 있다.

이 경우 PX4의 안전 철학에 따라 동반 컴퓨터의 제어권은 즉각 박탈된다. 이때 오프보드 제어 프로그램(Node)은 주 제어 루프 진입 전 반드시 VehicleStatus 토픽의 nav_state가 여전히 OFFBOARD인지, arming_state가 ARMED를 유지하는지를 폴링(Polling)해야 한다. 만일 일치하지 않는다면 프로그램 내의 타겟 추종 스레드를 보류(Suspend)하고 클라이언트(GCS) 측에 제어 모순 경고(Control Logic Conflict)를 발송해야 교착 상태(Deadlock)나 자원 낭비를 방지할 수 있다.

정리하자면, VehicleStatus와 관련된 일련의 상향 링크 토픽들은 제어 시스템에서 권한의 타당성을 되묻는 일종의 안전망(Safety Net) 프로토콜이자 에너지 동역학에 대한 최상위 모니터링 인터페이스로 기능한다.
