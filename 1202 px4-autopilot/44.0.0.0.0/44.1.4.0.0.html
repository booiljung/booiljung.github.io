44.1.4. 시동(Arming) 상태 유지 확인 및 하드웨어 인터록(Interlock) 해제 검증

무인항공기(UAS)의 자율 비행 시스템에서 이륙(Takeoff) 명령을 수행하기 위한 가장 기본적인 전제 조건은 시스템이 안전하게 시동(Arming) 상태를 유지하고 있으며, 하드웨어 인터록(Hardware Interlock) 장치가 완전히 해제(Disengaged)되었음을 검증하는 것이다. 시동 상태는 비행 제어기(Flight Controller, FC)가 모터 및 액추에이터(Actuator)에 동력을 전달할 준비가 완료되었음을 의미하며, 하드웨어 인터록은 예기치 않은 모터 작동으로 인한 사고를 방지하기 위한 최후의 물리적, 또는 논리적 안전 장치(Failsafe)이다. 본 절에서는 PX4-Autopilot(기준 펌웨어: v1.14.0)을 중심으로 시동 상태를 유지하고 인터록을 해제하는 메커니즘을 상세히 분석한다. 아울러 QGroundControl(기준 버전: v4.3.0) 관제 환경에서의 동작과 Ardupilot과의 차이점도 함께 고찰한다.

1. 시동(Arming) 상태 유지 및 검증 메커니즘

이륙 제어 로직이 정상적으로 작동하려면 기체가 시동(Armed) 상태로 전환된 후, 이륙 완료 시점까지 해당 상태가 안정적으로 유지되어야 한다. PX4-Autopilot의 커맨더(Commander) 모듈은 현재의 시동 상태를 지속적으로 모니터링한다.

1.1 커맨더 모듈의 상태 전이(State Transition)

PX4 내부에서 기체의 시동 상태는 vehicle_status uORB 토픽의 arming_state 필드를 통해 관리된다. 이륙 명령이 MAVLink를 통해 수신되면(예: MAV_CMD_NAV_TAKEOFF), Commander 모듈은 즉시 이 필드가 ARMING_STATE_ARMED 상태인지 검증한다.

만약, 시동이 걸리지 않은 상태(Standby 또는 Disarmed)에서 이륙 명령이 하달될 경우, PX4는 해당 명령을 거부(Command Denied)하고 관제 시스템(GCS)으로 COMMAND_ACK 메시지 내에 거부 사유를 반환한다.

1.2 자동 시동 해제(Auto-Disarm) 타임아웃

시동이 걸린 후 일정 시간 동안 이륙(Takeoff)이나 스로틀(Throttle) 입력이 없으면, 안전을 위해 기체는 스스로 시동을 해제하도록 설계되어 있다. 이 기능은 조종자의 부주의나 통신 지연으로 인한 사고를 예방한다.

  • PX4 파라미터: COM_DISARM_PRFLT 옵션을 통해 이륙 전 자동 시동 해제 타임아웃(Timer)을 초 단위로 설정할 수 있다 (기본값: 보통 10초).
  • 이륙 명령이 이 타임아웃 시점을 초과하여 도달하면 명령은 실패하며, 관제 애플리케이션에서는 이를 재시도(Retry)하거나 사용자에게 시동을 다시 걸도록 유도해야 한다.

    participant GCS as QGroundControl
    participant Commander as PX4 Commander
    participant Actuator as Actuator Control
    
    GCS->>Commander: MAV_CMD_COMPONENT_ARM_DISARM (Arm)
    Commander-->>GCS: COMMAND_ACK (Accepted)
    note over Commander: Arming State: ARMED
    Commander->>Actuator: 동력 인가 준비 (Idle Thrust)
    
    opt Timeout Occurs (ex: 10초 무입력)
        Commander->>Commander: Auto-Disarm 타임아웃 발생
        Commander-->>GCS: Status Text (Auto-Disarmed)
        note over Commander: Arming State: STANDBY
    end
    
    GCS->>Commander: MAV_CMD_NAV_TAKEOFF
    alt is Armed
        Commander-->>GCS: COMMAND_ACK (Accepted)
        Commander->>Actuator: 이륙 추력 할당
    else is Disarmed
        Commander-->>GCS: COMMAND_ACK (Denied)
    end

2. 하드웨어 인터록(Hardware Interlock) 검증 메커니즘

인터록(Interlock) 장치는 본래 설비 등에서 조건이 맞지 않으면 동작을 차단하는 기계/전자적 잠금 장치를 일컫는다. 무인항공기에서 하드웨어 인터록은 모터가 갑작스럽게 회전하는 것을 방지하기 위해 사용되는 물리적 안전 스위치(Safety Switch)를 말한다.

2.1 안전 스위치(Safety Switch)의 역할

Pixhawk 시리즈 등 대부분의 호환 하드웨어에는 별도의 외부 안전 스위치가 존재한다. 이 스위치를 길게 누르면 LED 상태가 깜빡임(Blinking)에서 점등(Solid)으로 변경되며, 이때 비로소 하드웨어 인터록이 해제(Disengaged)되고 PWM 또는 DSHOT 신호가 ESC(Electronic Speed Controller)로 정상 출력될 수 있다.

  • 상태 관리: PX4는 actuator_armed uORB 토픽 내에 ready_to_arm 및 locked 상태를 관리한다.
  • 이륙 전제 조건: 소프트웨어적으로 시동(Arming) 명령이 인가되더라도 하드웨어 인터록 스위치가 해제되지 않았다면 모터는 회전하지 않는다. PX4 시스템은 이 불일치를 치명적인 오류(Critical Error)로 판단하거나, 아예 시동 명령 자체를 거부하도록 동작한다(CBRK_IO_SAFETY 파라미터로 이를 강제 우회할 수 있으나 권장되지 않는다).

2.2 Ardupilot 대비 PX4의 인터록 처리 철학

Ardupilot과 PX4-Autopilot은 안전 스위치를 다루는 철학에서 미세한 차이를 보인다.

  • PX4-Autopilot: 철저한 다중 안전장치를 지향한다. 하드웨어 버튼(Safety Switch) 해제와 소프트웨어 시동(Arming) 명령이라는 이중 인증(Dual Authorization) 과정을 엄격히 요구한다. QGroundControl은 시스템 상태를 인지하여 UI 화면 상단에 분명한 경고 메시지(‘Safety Switch is engaged’)를 표시한다.
  • Ardupilot: Ardupilot의 경우 BRD_SAFETYENABLE 파라미터를 통해 안전 스위치의 활성화를 조절하며, Mission Planner에서 비교적 유연하게 인터록 옵션을 해제할 수 있는 구성을 제공한다. 하지만 PX4는 회로기판 수준의 I/O 리소스 모듈(PX4IO)에서부터 PWM 펄스를 하드웨어적으로 원천 차단하는 방식에 더욱 의존하는 경향이 있다.

3. 이륙 제어를 위한 데이터 유효성 및 상태 모니터링 실무

자율 에이전트 시스템(예: ROS 2 프로토콜을 사용하는 보조 컴퓨터와의 연동)에서 자동 이륙을 수행하고자 할 때, 통합 제어 로직 안에 반드시 다음과 같은 방어적 프로그래밍(Defensive Programming) 기법을 구현해야 한다.

3.1 상태 모니터링 로직 구현

보조 컴퓨터(Companion Computer) 또는 GCS 개발 시, MAVLink의 HEARTBEAT 메시지에 포함된 base_mode (내부 MAV_MODE_FLAG_SAFETY_ARMED 비트 플래그)를 주기적으로 검사하라.
시동 상태가 정상적으로 유지 중이고(비트값이 1), 시스템의 SYS_STATUS 메시지 내 onboard_control_sensors_health 플래그에서 모터/액추에이터 상태가 건강함(Healthy)으로 보고될 때 비로소 이륙 명령(Takeoff)을 송신해야 단다.

3.2 안전 검증 프로세스

  1. 시동 상태 유지 확인 (Checking Arming State): 기체가 ARMED 상태에 안정적으로 머물러 있는가? 제한 시간(Auto-Disarm 타임아웃) 이내인가?
  2. 하드웨어 인터록 해제 검증 (Validating Hardware Interlock): SYS_STATUS 메시지를 수신하여 하드웨어 스위치 에러 플래그가 없는지 확인하였는가?
  3. 액추에이터 상태 정상 확인 (Actuator Health Check): ESC 원격 측정(Telemetry) 데이터 피드백이나 RPM 출력이 대기(Idle) 상태에 도달하였는가?

이러한 상태 판별 루프(State Decision Loop)가 정확하게 동작하도록 구축되어야만, 오작동을 미연에 방지하고 예기치 못한 스로틀 급상승이나 로터(Rotor) 충돌로 인한 물리적 피해를 원천 차단할 수 있다.

이상의 검증 로직을 모두 통과하면 기체 제어 시스템은 최종적으로 액추에이터 출력을 스케일링하며 본격적인 고도 상승 절차를 위한 캐스케이드(Cascaded) 제어 단계로 돌입하게 된다.
