44.4 이륙 시 PX4-Autopilot 내부 유한 상태 기계(FSM) 및 다중 루프 제어(Cascaded Control) 메커니즘

본 절에서는 MAVLink를 통해 이륙 명령(Takeoff Command)이 전달되었을 때, PX4-Autopilot 내부에서 이를 처리하는 유한 상태 기계(FSM, Finite State Machine)의 구조와 이를 바탕으로 액추에이터 출력을 생성하는 다중 루프 제어(Cascaded Control) 메커니즘을 심층적으로 분석한다. (기준 펌웨어: PX4 v1.14.0 이상, QGroundControl v4.3.0 이상)

1. 이륙을 통제하는 유한 상태 기계(FSM) 모듈

PX4의 최상위 논리 제어는 Commander 모듈이 담당한다. 커맨더 모듈은 기체의 모드 변경, 시동(Arming), 그리고 이륙과 같은 중대한 전이(Transition)를 중앙 집중식 유한 상태 기계(FSM)로 관리한다.

이륙 절차는 다음의 주요 FSM 상태 전이를 거친다.

  1. 상태 평가 및 가드 조건(Guard Condition): 이륙 명령이 수신되면, 커맨더 모듈은 Pre-flight Check(센서 유효성, GPS 3D Lock, 배터리 상태 등)를 1차 가드 조건으로 검사한다.
  2. 시동(Arming): 가드 조건을 통과하면 기체 상태는 ARMED로 천이되며 액추에이터에 최소 PWM이 인가된다.
  3. 이륙 상태(Takeoff State): Auto 모드의 서브 모드인 Takeoff로 진입한다. 내부적으로는 FlightTaskTakeoff 클래스가 활성화되어 궤적 생성을 주도한다.
  4. 비행 상태(Flight State): 이륙 고도 한계에 도달하면 Auto-Loiter (Hold 모드) 또는 다음 임무(Mission)를 위한 Auto-Mission 상태로 자동 천이한다.

2. 이륙 궤적 생성과 FlightTask 아키텍처

이륙 FSM이 활성화되면 위치 제어기(Position Controller) 내부의 FlightTaskAutoMapper를 통해 FlightTaskTakeoff가 실행된다. 이 태스크 클래스는 기체가 지면에 있는 상태(On-ground)에서 목표 고도(Target Altitude)에 도달할 때까지 연속적인 위치 및 속도 설정값(Setpoint)을 생성한다.

  • Ramp-up (추력 점진적 증가): 지면에 있을 때 센서 노이즈로 인한 급격한 스로틀(Throttle) 증가를 방지하기 위해 이륙 추력을 점진적으로 상승시킨다.
  • Trajectory Generation: 이륙 가속도 제약과 저크(Jerk) 제약을 고려한 위상 궤적 곡선(Phase Trajectory Curve)을 그려 부드러운 속도 명령을 하위 제어기에 전달한다.

3. 다중 루프 제어(Cascaded Control) 아키텍처 해석

생성된 이륙 궤적은 기체의 다중 루프 제어기를 거쳐 최종적으로 모터의 PWM 신호로 변환된다. 다중 루프(Cascaded) 구조는 외부 루프(위치/속도)와 내부 루프(자세/각속도)가 분리되어 직렬로 연결된 형태를 의미한다. 이는 대역폭(Bandwidth) 차이를 이용해 시스템을 안정화하는 고전적이면서도 강력한 제어 기법이다. (참고 논문: “Multicopter Design and Control Practice”)

이륙 단계에서의 각 제어 루프 역할은 다음과 같다.

  1. 위치 제어기 (Position Controller, P-Control)
  • 입력: 목표 수직 이륙 고도(z_{\text{sp}}) 및 현재 시스템 고도(z).
  • 출력: 위치 오차(e_z)에 비례 이득(Proportional Gain)을 곱하여 목표 수직 속도(v_{z, \text{sp}})를 산출한다.
  1. 속도 제어기 (Velocity Controller, PID-Control)
  • 입력: 목표 수직 속도(v_{z, \text{sp}}) 및 현재 수직 속도(v_z).
  • 출력: 이륙 시 목표 추력 벡터(Thrust Setpoint)와 목표 자세각(Attitude Setpoint)을 산출한다. 수직 상승 중에는 롤과 피치 목표값은 0으로 유지된다.
  1. 자세 제어기 (Attitude Controller, P-Control)
  • 입력: 목표 자세각(q_{\text{sp}}) 및 추력.
  • 출력: 현재 자세각과의 오차를 기반으로 회전 각속도 목표값(Angular Rate Setpoint, \omega_{\text{sp}})을 산출한다.
  1. 각속도 제어기 (Rate Controller, PID-Control)
  • 입력: 목표 각속도(\omega_{\text{sp}}) 및 자이로스코프 측정치(\omega). 가장 빠른 주파수(루프 타임스텝 \sim 4ms 이내)로 동작한다.
  • 출력: 믹서(Mixer) 또는 Control Allocation 모듈을 거쳐 최종 전기 변속기(ESC) 제어 명령(토크 및 추력)을 스팬(Span)한다.

위 다중 루프에서 가장 중요한 수식은 속도 제어기의 PI 연산을 통해 추력 \mathbf{u}_T를 생성하는 과정이다. 적분기(Integrator)는 이륙 전 지면(Ground) 상태에서 Wind-up 현상에 노출되지 않도록 초기화가 엄격히 제한된다.

\mathbf{u}_T = K_p(v_{z, \text{sp}} - v_z) + \int K_i(v_{z, \text{sp}} - v_z)dt + K_d \frac{d}{dt}(v_{z, \text{sp}} - v_z)

(단, 이륙 등 비선형 궤적 추종 시 수식에 피드포워드(Feedforward) 텀이 추가된다.)

4. Ardupilot 대비 다중 루프 및 FSM 차이 분석

PX4와 Ardupilot 모두 다중 루프 제어 아키텍처를 기반으로 하나, 구조적 유연성과 구현 철학에 차이가 존재한다.

  1. 상태 기계(FSM)의 결합도
  • PX4-Autopilot: 커맨더 인스턴스라는 하나의 독립된 모듈이 상태 유효성을 책임지며 상태를 강력히 정의한다. uORB 메시지 브로커를 통해 각 서브 모듈이 vehicle_status 토픽을 구독(Subscribe)하여 철저히 동기화된다.
  • Ardupilot: 글로벌 상태 객체(AP_State 등)에 의존하는 경향이 크며, 비행 모드 모듈 내부(mode_auto.cpp, mode_takeoff.cpp 등)에서 자율적으로 상태를 전환하고 예외 처리를 병행하는 분산형 제어 코드 베이스의 특징을 띈다.
  1. 제어기 튜닝 체계 (Control Allocation)
  • PX4-Autopilot: 위치/속도 컨트롤러와 자세/각속도 컨트롤러가 명확한 클래스로 격리되어 있으며, 공력 행렬(Mixer Matrix, 최근 Control Allocation으로 업데이트 됨)이 완전히 분리되어 범용적인 기체 형상을 손쉽게 적용할 수 있다.
  • Ardupilot: 멀티로터의 경우 AC_AttitudeControl 라이브러리가 모터 라이브러리(AP_Motors)와 긴밀하게 밀착되어 있어 특정 멀티로터 형상에 고도로 최적화된 성능(P-PID 구조)을 뽑아내는 것에 중점을 둔다.

5. 결론 및 실무적 주의사항

이륙 시 PX4의 FSM 및 다중 루프 제어를 분석해 보면, 상위 계층에서 발생한 제어 명령이 안전하고도 강건하게 하위 물리 계층으로 스며드는 것을 확인할 수 있다. 실무적으로 GCS나 ROS 기반 자율 에이전트 개발 시, 단순히 Takeoff MAVLink 패킷을 보내는 것에 그치지 않고 현재 FSM 상태가 이륙 가드 조건을 통과했는가, 그리고 기저 다중 루프의 Rate Controller 단에서 발진(Oscillation) 없이 궤적을 쫓아가는가를 QGroundControl의 “Tuning” 화면이나 비행 로그(ULog 분석)를 통해 지속적으로 모니터링해야 한다.
