44.3.3. PX4 내부 파라미터 트리(Parameter Tree)의 이륙 속도 및 고도 연동 구조

1. 개요: 개방형 파라미터 아키텍처 (Open Parameter Architecture)

PX4-Autopilot(Firmware 버전 v1.14 기준)은 모듈형(Modular) 로보틱스 프레임워크 설계 철학을 바탕으로, 비행 제어 및 동작에 관여하는 모든 변수를 파라미터 트리(Parameter Tree) 라는 일원화된 데이터베이스로 관리한다. 이 파라미터 시스템은 단순한 상수(Constant)의 집합이 아니라, 비행 중에도 uORB(Micro Object Request Broker) 메시지를 통해 실시간으로 변경 사항을 퍼블리싱(Publishing)하고 서브스크라이빙(Subscribing)할 수 있는 동적 구조를 띤다.

특히 기체가 지면을 박차고 오르는 이륙(Takeoff) 프로세스는 모터의 급출력, 지면 효과(Ground Effect), 그리고 고도 제어 시스템의 활성화가 복합적으로 일어나는 과정이므로, 이륙 속도와 고도에 대한 파라미터 연동 구조를 이해하는 것은 안정적인 GCS 관제 기술 및 커스텀 비행 제어기 개발에 필수적이다.

2. 핵심 이륙 파라미터(Takeoff Parameters) 계층 구조

멀티로터(Multicopter) 기반 PX4 시스템에서 이륙의 프로파일(Trajectory Profile)을 결정짓는 핵심 파라미터들은 주로 MIS_ (Mission) 계열과 MPC_ (Multicopter Position Control) 계열로 나뉘어 연동된다.

  1. MIS_TAKEOFF_ALT (Mission Takeoff Altitude):
  • 설명: GCS(예: QGroundControl)로부터 별도의 고도 지정 없이 이륙(Takeoff) 명령만 하달되었을 때, 혹은 임무(Mission) 모드에서 자동 이륙 시 기체가 도달하여 호버링(Hovering)해야 할 지면 대비 상대 고도(AGL, Above Ground Level)의 기본값을 지정한다.
  • 단위: 미터 (m), 기본값: 2.5m
  1. MPC_TKO_SPEED (Multicopter Takeoff Speed):
  • 설명: 이륙 시 기체가 상승하는 최대 수직 속도를 정의한다.
  • 단위: m/s, 기본값: 1.5m/s
  1. MPC_TKO_RAMP_T (Takeoff Ramp Time):
  • 설명: 이륙 명령이 떨어지고 모터가 공회전(Idle) 상태에서 이륙 필요 추력(Takeoff Thrust)에 도달하기까지 스로틀(Throttle)을 부드럽게 증가시키는 스루 레이트(Slew-rate)의 시간 상수이다.
  • 단위: 초 (s), 기본값: 3.0s

3. 이륙 속도 및 고도의 내부 연동 로직 (Source Code Architecture)

위에서 정의된 파라미터들은 단순히 제어기에 그대로 전달되는 것이 아니라, 궤적 생성기(Trajectory Generator)인 FlightTaskTakeoff 클래스를 거쳐 부드러운 위치 및 속도 설정점(Setpoint)으로 변환된다.

    A[GCS Takeoff Command] --> B[Commander Module]
    B --> C{Parameter Evaluation}
    C -->|Fetch MIS_TAKEOFF_ALT| D[Target Altitude Z_target]
    C -->|Fetch MPC_TKO_SPEED| E[Max Ascent Velocity Vz_max]
    C -->|Fetch MPC_TKO_RAMP_T| F[Ramp Up Time]
    
    D --> G[(FlightTaskTakeoff \nTrajectory Generator)]
    E --> G
    F --> G
    
    G -->|Smoothly Increment Z_sp| H[Position Controller (PosControl)]
    G -->|Slew-limited Vz_sp| H
    H --> I[Mixer & Actuator Drivers]

3.1 궤적 생성의 수학적 스무딩(Smoothing)

  • 기체가 이륙할 때 시스템은 단순히 목표 고도(MIS_TAKEOFF_ALT)와 목표 속도(MPC_TKO_SPEED) 스텝 입력(Step Input)을 위치 제어기(PID)로 던지지 않는다.
  • 대신, 현재 바닥에 있는 고도 상태에서 시작하여 매 타임 스텝(dt)마다 MPC_TKO_SPEED를 넘지 않는 선에서 점진적으로 증가하는 중간 목표 고도 Z_{sp}(t) 곡선을 수학적으로 생성한다(Jerk-limited Trajectory).
  • 이로 인해 제어 포화(Actuator Saturation)를 방어하고 부드러운 스풀업(Spool-up)이 가능해진다.

4. QGroundControl(GCS)과의 동적 연동 메커니즘

운영자가 QGC(v4.3.0)에서 비행 전 이륙 고도를 슬라이더나 텍스트 박스로 조정하면, QGC는 백엔드 통신 모듈을 통해 PARAM_SET MAVLink 메시지를 PX4로 전송하여 MIS_TAKEOFF_ALT 파라미터 값을 실시간으로 덮어쓴다(Overwrite).

  • PX4의 파라미터 서버는 변경을 감지하고 parameter_update uORB 토픽을 브로드캐스트한다.
  • FlightModeManager와 Navigator 모듈은 이 이벤트를 서브스크라이빙(Subscribe)하여 즉각적으로 다음번 이륙이나 오프보드(Offboard) 모드 진입 시의 고도 한계 기준을 새로운 값으로 갱신한다.

5. Ardupilot과 PX4 파라미터 철학의 차이 (비교)

이륙 프로파일을 다루는 파라미터 트리 구조에서 PX4와 Ardupilot(Mission Planner)은 서로 다른 아키텍처 설계 철학을 보여준다.

  • PX4-Autopilot의 계층 구조: PX4는 파라미터를 기능 모듈별 접두사(MPC_, MIS_, EKF2_)로 매우 엄격하고 체계적으로 구분한다. 이륙 속도 하나를 제어하더라도 코어 비행 제어 모듈인 멀티로터 위치 제어(MPC_)에 파라미터를 귀속시켜 전역적인 모드 간 간섭을 분리한다.
  • Ardupilot의 편의성 중심 구조: Ardupilot은 WPNAV_SPEED_UP (웨이포인트 상승 속도)이나 PILOT_TKOFF_ALT와 같이 사용자나 조종사(Pilot) 관점에서의 직관적인 네이밍을 선호하며, 하나의 파라미터가 임무 주파수나 여러 비행 모드에 다중으로 영향을 미치는 통합적인(Tight-coupled) 설계 성향이 짙다.

파라미터 중심의 트리 구조는 개발자가 재컴파일(Recompile) 과정 없이 기체의 동역학에 맞춰 무한히 시스템을 튜닝할 수 있게 만드는 오픈소스 오토파일럿의 본질이며, GCS 개발자는 이 파라미터 모듈 시스템의 메타데이터(Metadata) 명세를 완벽히 이해해야만 커스텀 제어 환경을 올바르게 구축할 수 있다.
