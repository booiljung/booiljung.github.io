44.2.1.1. 패킷 페이로드(Payload) 내 파라미터의 바이트 정렬(Byte Alignment) 및 리틀 엔디언(Little-Endian) 처리 규칙

무인항공기(UAS) 지상 관제 시스템(GCS)과 PX4-Autopilot 간의 신뢰성 있는 텔레메트리(Telemetry) 통신을 보장하기 위해서는 MAVLink 메시지 패킷 내부의 데이터 레이아웃을 정확하게 이해하는 것이 필수적이다. 특히 MAV_CMD_NAV_TAKEOFF와 같은 제어 명령은 다수의 부동소수점(Floating-point) 파라미터를 포함하므로, 이기종(Heterogeneous) 하드웨어 아키텍처 간의 데이터 교환 시 패킷 페이로드(Payload)의 바이트 정렬(Byte Alignment) 및 엔디언(Endianness) 처리 규칙이 엄격하게 적용되어야 한다. 본 절에서는 MAVLink 프로토콜의 메모리 접근 구조와 리틀 엔디언(Little-Endian) 직렬화 메커니즘을 소스 코드 수준에서 상세히 분석한다.

1. MAVLink 데이터 직렬화(Serialization)와 엔디언(Endianness)

비행 제어 컴퓨터(Flight Control Computer, FCC)와 지상 관제 시스템(예: QGroundControl), 그리고 동반 컴퓨터(Companion Computer, 예: ROS 2 환경)는 주로 서로 다른 CPU 아키텍처(ARM, x86, RISC-V 등)를 사용한다. 서로 다른 아키텍처는 다중 바이트(Multi-byte) 데이터를 메모리에 저장하는 바이트 순서, 즉 엔디언이 상이할 수 있다.

1.1 리틀 엔디언(Little-Endian) 표준 규칙

MAVLink 프로토콜은 데이터 전송 표준으로 **리틀 엔디언(Little-Endian)**을 엄격하게 채택하고 있다. 이는 하위 바이트(Least Significant Byte, LSB)가 메모리의 하위 주소(Lowest Address)에 먼저 저장됨을 의미한다.

  • 네트워크 통신 표준인 TCP/IP가 빅 엔디언(Big-Endian)을 사용하는 것과는 대조적이다.
  • 대부분의 소형 임베디드 장비(ARM Cortex-M 등)와 최신 데스크톱(Intel x86)이 기본적으로 리틀 엔디언 아키텍처를 사용하므로, 송수신부에서 바이트 스와핑(Byte Swapping)에 소모되는 CPU 오버헤드(Overhead)를 최소화하기 위한 설계 철학이다.

1.2 MAVSDK 및 QGroundControl의 바이트 변환

GCS 혹은 ROS 2용 Micro XRCE-DDS 에이전트(Agent)가 빅 엔디언 시스템에서 구동될 경우, MAVLink 패커/언패커(Packer/Unpacker) 라이브러리 내부의 매크로 연산(_MAV_PUT_uint32_t 등)을 통해 전송 전 리틀 엔디언으로 강제 변환된다. PX4-Autopilot 수신단 모듈은 이를 항상 리틀 엔디언으로 가정하고 파싱(Parsing)한다.

2. 페이로드 바이트 정렬(Byte Alignment) 및 패딩(Padding) 철학

C/C++ 환경에서 컴파일러는 메모리 접근 효율을 높이기 위해 변수 타입의 크기에 맞춰 더미 바이트(Dummy Byte)를 삽입하는 구조체 패딩(Struct Padding)을 수행한다. 그러나 무선 통신에서는 1바이트의 낭비도 허용되지 않으므로, MAVLink는 매우 독창적인 구조체 정렬(Structure Alignment) 방식을 사용한다.

2.1 크기 기반 내림차순 정렬 알고리즘

MAVLink 메시지 생성기(Generator)는 XML 파일에 정의된 변수의 순서와 무관하게, 페이로드 구조체 멤버를 데이터 크기(64비트 \rightarrow 32비트 \rightarrow 16비트 \rightarrow 8비트) 순으로 내림차순 정렬하여 메모리를 할당한다.

이러한 재배열(Reordering) 메커니즘은 컴파일러가 암묵적인 패딩 바이트를 추가하지 않도록 원천적으로 차단한다.

예를 들어 COMMAND_LONG 메시지 페이로드를 C 언어 구조체로 나타내면 다음과 같이 가장 크기가 큰 데이터(32비트 float 배열)부터 선언된다.

// PX4-Autopilot MAVLink 헤더 내부 매핑 예시
typedef struct __mavlink_command_long_t {
    float param1;       // 4 Bytes (Offset 0)
    float param2;       // 4 Bytes (Offset 4)
    float param3;       // 4 Bytes (Offset 8)
    float param4;       // 4 Bytes (Offset 12)
    float param5;       // 4 Bytes (Offset 16)
    float param6;       // 4 Bytes (Offset 20)
    float param7;       // 4 Bytes (Offset 24)
    uint16_t command;   // 2 Bytes (Offset 28)
    uint8_t target_system;    // 1 Byte  (Offset 30)
    uint8_t target_component; // 1 Byte  (Offset 31)
    uint8_t confirmation;     // 1 Byte  (Offset 32)
} mavlink_command_long_t;     // Total : 33 Bytes (No Padding)

2.2 #pragma pack(1) 및 GCC 속성 처리

다양한 컴파일러 환경(GCC, Clang 등)에서 패딩 동작을 기계적으로 방지하기 위해, MAVLink 라이브러리는 #pragma pack(1) 디렉티브 또는 __attribute__((packed)) 속성(Attribute)을 부여하여 1바이트 단위 정렬을 강제한다. PX4 펌웨어 빌드 시스템은 이를 통해 MCU(Microcontroller Unit) 내부 버스 접근 오작동(Alignment Fault)을 제어 로직 차원에서 무결하게 해결하고 있다.

3. PX4와 ROS 2 (uXRCE-DDS) 통신에서의 정렬 관점

최신 PX4 환경에서는 MAVLink 외에도 ROS 2와의 네이티브 통신을 위해 uXRCE-DDS (Micro XRCE-DDS) 브리지를 적극 활용한다.

  • MAVLink: 가상 크기 순서 정렬(Wire-level Reordering)을 사용하여 페이로드 밀도를 극한으로 높인다.
  • DDS 계층(ROS 2): CDR(Common Data Representation) 직렬화 포맷을 사용하여, 플랫폼의 엔디언 특성을 감지하고 패딩을 수용하는 방식으로 유연성을 확보한다.

따라서, 보조 컴퓨터의 ROS 2 노드에서 이륙 명령어(Takeoff Command)의 파라미터를 넘길 때, uORB 구조체의 크기 및 DDS 브리지의 데이터 배열 방식이 MAVLink의 패밀리 트리(Architecture)와 차이가 있음을 인지하고, 개발 시 자료형 캐스팅에 주의를 기울여야 한다.

결론적으로, GCS 개발자나 관제 시스템 아키텍트는 MAVLink 프로토콜의 리틀 엔디언 전송 규칙과 크기 기반 정렬 방식이 PX4 제어 메커니즘의 초저지연(Ultra-low Latency) 명령어 처리를 실현하는 핵심적인 배경임을 이해해야만 하드웨어 독립적인 자율 비행 시스템을 설계할 수 있다.
