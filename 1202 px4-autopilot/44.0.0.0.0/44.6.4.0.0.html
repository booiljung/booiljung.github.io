44.6.4. 이륙 완료 후 비행 모드 자동 전환(Takeoff -> Hold/Mission) 트리거의 동시성(Concurrency) 제어 및 교착 상태(Deadlock) 방지

자율 비행 시스템에서 이륙(Takeoff) 완료가 기하학적 노름(Norm) 연산과 체류 시간(Dwell Time) 검증을 통해 최종 승인되면, 기체는 즉각적으로 다음 비행 모드로 전환되어야 한다. PX4-Autopilot(펌웨어 v1.14.x 기준)의 비행 제어 스택에서 이 전환은 주로 Takeoff 상태에서 Hold (지정 위치 대기) 또는 Mission (임무 수행) 상태로의 전이를 의미한다. 이 구간에서 비행 모드 변경은 실시간 운영체제(NuttX RTOS) 내의 복수 스레드(Thread) 간에 공유되는 비행 상태 자원에 대한 동시성(Concurrency) 제어 문제와 직결된다. 잘못된 자원 점유나 예외 처리가 겹치면 교착 상태(Deadlock)에 빠져 기체가 허공에서 멈추거나 예기치 않은 추락을 유발할 수 있다. 본 절에서는 이러한 모드 전환 과정의 스레드 안전성(Thread Safety)과 동시성 제어 아키텍처를 상세히 다룬다.

1. 이륙 완료 트랜잭션의 비동기적 성쇠와 동시성 문제

PX4-Autopilot 아키텍처는 navigator 모듈이 궤적을 계획하고 위치 추종 도달 여부를 평가하는 반면, 최상위 모드 전환의 최종 권한은 commander 모듈이 장악하는 이원화된 구조를 가진다.

  1. 내비게이터(Navigator) 스레드가 “이륙 목표 고도 도달 완료” 이벤트(Flag)를 발생시킨다.
  2. 이 이벤트는 uORB(Micro Object Request Broker) 메시지 버스를 통해 커맨더(Commander) 로 비동기 전달된다.
  3. 커맨더는 현재의 안전 상태(Failsafe, RC Loss 등)를 확인하고 최종 비행 모드를 변경(State Update)한다.

이러한 메시지 기반 통신(Pub/Sub) 구조는 모듈 간 결합도를 낮추지만, 이륙 완료 시점과 동시에 배터리 경고(Low Battery) 시그널이 발생하거나 지상 관제 시스템(GCS)에서 수동 개입(Manual Override) MAVLink 명령을 송신할 경우, 커맨더 내부에서 상호 충돌(Race Condition)이 발생할 위험을 내포한다.

2. 모드 전환 시 교착 상태(Deadlock)의 논리적 시나리오

자율 비행 로직에서 교착 상태란 두 개 이상의 프로세스나 제어 루프가 서로의 작업이 끝나기만을 맹목적으로 기다리는 현상을 뜻한다. 이륙 완료 시점의 대표적인 교착 상태 시나리오는 다음과 같다.

  • 원인 (Cause): 기체는 이륙 완료 시 자동으로 Mission 모드로 넘어가도록 설정(파라미터 COM_OBS_AVOID 등과 연동)되어 있으나, 이륙 도중 GCS와 통신이 끊기거나(Telemetry Loss), 유효한 웨이포인트(Waypoint) 리스트가 메모리에 업로드되지 않은 경우이다.
  • 결과 (Effect): Navigator는 Mission으로의 전환 명령을 발행하지만, Commander는 임무 데이터 부재를 이유로 전환을 거부(NACK)한다.
  • 교착 (Deadlock): Navigator는 다음 웨이포인트 지시를 받지 못해 지속해서 Takeoff 완료 신호만을 재발행하고, 기체는 다음 동작을 결정하지 못한 채 바람에 밀리며 표류하게 된다.

3. PX4의 교착 상태 방지 및 원자적(Atomic) 폴백(Fallback) 제어 기법

PX4-Autopilot은 이와 같은 동시성 충돌 및 교착 상태를 방지하기 위해 엄격한 폴백(Fallback: 예비 동작) 상태 트리를 구현하여 원자적(Atomic)으로 처리한다.

3.1 상태 전환의 원자성(Atomicity) 보장

모드 전환 로직은 commander 루프 내에서 뮤텍스(Mutex) 또는 상태 검증 락(Lock)에 의해 한 번의 제어 주기(Tick) 내에 단일 스레드 전용으로 처리된다. 전환 트리거가 발생하면 MAVLink 명령이나 외부 시그널 처리는 해당 틱이 종료될 때까지 대기 큐(Queue) 블로킹 된다.

3.2 안전 폴백 (Safe Fallback) 트리거

이륙 완료 직후 Mission 전환이 거부되는 즉시 시스템은 무한 대기하지 않고 가장 안전한 하위 모드로 강제 스위칭한다.

\text{Transition}(Takeoff \to Mission) \implies \begin{cases} \text{Success} & (\text{Valid Mission Data}) \\ \text{Fallback to Hold} & (\text{Invalid Mission Data / Error}) \end{cases}

이를 다이어그램으로 나타내면 다음과 같다.

5. 지상 관제 프론트엔드 (GCS) 개발 시 유의사항

QGroundControl(QGC)와 같은 GCS 애플리케이션을 전문적으로 개발할 때, 교착 상태를 유발하지 않기 위한 동시성 제어 원칙을 UI 철학에 반영해야 한다.
예를 들어, 기체가 이륙 목표 고도에 도달하기 직전의 Takeoff 진행률(Progress) 90% 이상 구간에서는 조종사가 수동으로 Mission이나 Land 버튼을 누르도록 허용하면 안 된다. 이 타이밍에 MAVLink Command가 전송될 경우, PX4 내부의 원자성(Atomicity) 락(Lock)에 걸려 COMMAND_ACK(NACK)가 발생하거나 커맨더 상태 큐가 과부하에 걸릴 수 있다. 따라서 UI 레벨에서 잠시 커맨드 입력 버튼을 비활성화(Disable) 처리하거나, Hold 상태로 폴백(Fallback) 되었음을 확인한 이후에 후속 임무(Upload)를 동기화하는 프론트엔드 상태 머신(Frontend State Machine) 설계가 필수적이다.
