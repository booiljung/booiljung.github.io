44.3.2. GCS 관제 애플리케이션의 이륙 고도 입력 한계(Limit) 지정 및 제어 포화(Actuator Saturation) 방어 로직

1. 개요: 지정 고도 한계 설정의 중요성

무인항공기(UAS)의 완전 자율 비행(Autonomous Flight)을 시작하는 첫 번째 단계인 이륙(Takeoff) 명령은 지상 관제 시스템(GCS, 예: QGroundControl)을 통해 하달된다. 이때 오퍼레이터는 이륙 후 기체가 대기할 목표 고도(Target Altitude)를 입력하게 되는데, 만약 소프트웨어적인 입력 한계(Hard/Soft Limit)가 지정되지 않는다면 치명적인 비행 사고로 이어질 수 있다.

예를 들어, 사용자의 실수로 목표 고도가 10,000m로 입력되거나, 0.1m와 같이 물리적으로 지면 효과(Ground Effect)를 벗어날 수 없는 고도가 입력될 경우, 비행 제어기(Flight Controller) 내의 위치 제어기(Position Controller)는 불가능한 제어 명령을 추종하려다 제어 포화(Actuator Saturation) 상태에 빠져 기체가 전복되거나 시스템이 멈추는 현상이 발생한다. 본 절에서는 QGroundControl(v4.3.0 기준)의 프론트엔드 입력 검증 로직과 PX4-Autopilot(v1.14 기준) 펌웨어 내부의 제어 포화 방어 아키텍처를 상세히 분석한다.

2. QGroundControl(GCS)의 이륙 고도 입력 검증 및 전송 구조

QGroundControl(이하 QGC)은 사용자의 비이성적인 입력을 MAVLink 명령어(MAV_CMD_NAV_TAKEOFF)로 패키징하기 전에, GUI 프론트엔드 레벨과 백엔드의 Vehicle 객체 레벨에서 2단계로 검증한다.

2.1 소프트 리미트(Soft Limit)와 하드 리미트(Hard Limit)

  • 최소 고도 제한 (Minimum Altitude Limit): QGC 소스 코드 내에서 이륙 고도의 하한선은 통상적으로 회전익(Multicopter) 기준 최소 1.5m ~ 2.5m 이상으로 강제(Hard Limit)된다. 이는 기체가 지면 근처로 압축되는 공기 역학적 현상인 지면 효과(Ground Effect) 구간을 확실하게 벗어나 안정적인 호버링(Hovering) 난류를 피하기 위함이다.
  • 최대 고도 제한 (Maximum Altitude Limit): PX4의 내부 파라미터 MPC_Z_VEL_MAX_UP 및 기체의 배터리 용량을 고려하여 과도한 고도(예: 1,000m 이상) 입력 시 사용자에게 UI 경고창(Warning Dialog)을 표출하여 한 번 더 확인을 거치는 소프트 리미트(Soft Limit) 로직이 적용되어 있다.

QGC가 검증을 마친 유효한 고도는 MAVLink의 Param 7 (Altitude) 필드에 적재되어 기체로 송신된다.

3. PX4-Autopilot 내부의 제어 포화(Actuator Saturation) 발생 메커니즘

목표 고도가 정상적인 범위를 초과하여 설정되었을 때, 이를 그대로 추종하려 할 경우 발생하는 핵심적인 제어 공학적 문제가 바로 제어 포화(Actuator Saturation) 이다.

PX4의 고도 제어는 PID(Proportional-Integral-Derivative) 중첩 루프 구조로 이루어져 있다. 위치 제어 루프에서 목표 고도와 현재 고도의 차이(오차, Error)를 계산하여 속도 제어 루프로 넘겨주면, 속도 제어 루프는 모터의 추력(Thrust Setpoint)을 결정한다.

graph TD
    A[Target Altitude Setpoint\nZ_sp] --> B((+))
    C[Current Altitude\nZ_est] -->| - | B
    B -->|Altitude Error e_z| D[Position P-Controller]
    D --> E[Target Vertical Velocity Setpoint]
    E --> F((+))
    G[Current Vertical Velocity] -->| - | F
    F -->|Velocity Error e_vz| H[Velocity PID Controller]
    H -->|Thrust Command 0.0 ~ 1.0| I{Mixer / ESCs}
    I --> J[Motors]

만약 오퍼레이터가 이륙 고도를 기체의 상승 능력을 아득히 초과하는 값으로 설정한다면, 속도 제어 루프의 적분기(Integrator, I-term)가 오차를 시간에 따라 계속 누적하게 된다.

  1. 추력 명령(Thrust Command)이 최대치인 1.0 (100%)에 도달한다.
  2. 모터는 이미 최대 속도(Saturation)로 돌고 있음에도 고도 오차는 줄어들지 않는다.
  3. 적분기 값이 비정상적으로 거대해지는 적분 통제 불능(Integral Windup) 현상이 발생한다.
  4. 이 상태에서 제어기가 바람의 교란(Disturbance)을 받아 자세를 교정하려 해도 제어 여유(Control Margin)가 없기 때문에 기체는 추락하거나 뒤집히게 된다.

4. PX4의 제어 포화 방어 로직 (Anti-Windup 및 Setpoint Slew-rate Limiting)

PX4 개발진은 이러한 사태를 막기 위해 위치 제어기(Flight Task 기능 내)에 강력한 방어 로직을 소스 코드 단에서 구축해 두었다.

4.1 스루 레이트 제한 (Slew-rate Limiting) 알고리즘

GCS가 이륙 고도를 50m로 주었다 하더라도 위치 제어기는 타겟 목표점(Z_{sp})을 단숨에 50m로 설정하지 않는다. MPC_TKO_SPEED (이륙 속도) 파라미터를 기반으로 매 제어 루프(통상 250Hz)마다 상승할 수 있는 궤적(Trajectory)을 부드럽게 생성(Slew)하여 가상의 실시간 타겟 궤적을 갱신한다. 이를 통해 PID 제어기에 들어가는 오차가 항상 일정 수준 이하로 유지되도록 통제한다.

4.2 안티 와인드업 (Anti-Windup) 아키텍처

속도 제어기의 PID 알고리즘 내에는 클램핑(Clamping) 기반의 안티 와인드업 로직이 구현되어 있다.

  • 만약 산출된 추력 명령이 MPC_THR_MAX (최대 추력 파라미터, 통상 0.9 ~ 1.0)에 도달하여 하드웨어가 포화(Saturation) 상태에 이르렀다고 판단되면,
  • 알고리즘은 즉시 적분기(I-term)의 누적 연산을 중지(Freeze) 한다.
  • 이를 통해 모터가 100%로 구동 중일 때 불필요한 에러 누적이 차단되며, 언제든 자세 제어(Attitude Control)를 위한 여유 출력(Margin)을 확보할 수 있도록 믹서(Mixer)가 스로틀을 배분할 수 있는 공간을 지켜낸다.

5. Ardupilot과의 이륙 한계 검증 아키텍처 비교 차이

Ardupilot 플랫폼 (Mission Planner 및 EKF3 기반) 역시 동일한 물리적 오류를 방어하기 위한 로직을 가지고 있으나, 통제 철학에서 다소 차이가 있다.

  • PX4-Autopilot: 궤적 생성기(Trajectory Generator) 중심의 철학을 가진다. 사용자가 터무니없는 한계치를 주더라도 시스템이 알아서 부드러운 궤적 곡선(Jerk-limited Trajectory)으로 쪼개어 제어기 내부 포화를 미연에 방어하는 ’비행기 내부 방어’에 치중되어 있다. MIS_TAKEOFF_ALT 값을 바탕으로 유효한 이륙 궤적을 능동적으로 생성한다.
  • Ardupilot: 파라미터 한계와 GCS 검증을 더 강력하게 신뢰하는 편이다. PILOT_TKOFF_ALT 값을 사용하여 이륙 명령을 수행하며, 궤적 스루 레이트 처리 방식보다는 파라미터 단에서 상/하한선 클리핑(Clipping)을 명시적으로 제한하고, 과도한 목표값이 입력될 경우 GCS 상에 즉각적인 MAVLink “Command Rejected” 메시지를 반환하는 형태의 ‘명시적 거부(Explicit Rejection)’ 기법을 더욱 적극적으로 활용한다.

결론적으로 GCS의 이륙 고도 한계 지정과 오토파일럿 내부의 포화 방어 로직은, 운영자의 휴먼 에러(Human Error)가 드론의 물리력 상실(Loss of Control)로 직결되는 것을 소프트웨어 구조적으로 단절시키는 핵심 안전 고리라 할 수 있다.
