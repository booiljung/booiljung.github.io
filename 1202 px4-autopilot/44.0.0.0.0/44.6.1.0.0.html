44.6.1. GCS 측 폐루프(Closed-loop) 상태 모니터링 아키텍처 및 네트워크 전송 지연(Transmission Latency) 보상 핑-퐁(Ping-Pong) 측정

1. 개요

지상 관제 시스템(Ground Control Station, GCS)이 비행 제어기(PX4-Autopilot)에 이륙과 같은 중대한 상태 변이 명령을 인가할 때, 단순히 명령 전송(Open-loop)에 그치지 않고 기체의 실제 상태 변화를 지속적으로 피드백받아 확인하는 폐루프(Closed-loop) 상태 모니터링이 필수적이다. 특히, 원격지에서 드론을 관제하는 경우 RF 데이터 링크(Telemetry, LTE/5G 등) 특성상 발생하는 네트워크 전송 지연(Transmission Latency)은 실시간 상태 동기화를 방해하는 주요 원인이다. 본 절에서는 QGroundControl(QGC)의 폐루프 모니터링 아키텍처와, MAVLink 통신 라우팅 상의 지연을 극복하기 위한 핑-퐁(Ping-Pong) 측정 및 보상 알고리즘을 분석한다.

2. GCS 측 폐루프(Closed-loop) 상태 모니터링 아키텍처

QGC(버전 4.3 기준)는 C++ 백엔드의 구조적 이점을 살려, 백그라운드 스레드에서 MAVLink 메시지를 처리하고 기체의 상태를 실시간으로 추적하는 견고한 폐루프 아키텍처를 보유하고 있다.

  • 명령-확인 응답(Command-ACK) 체인: GCS가 COMMAND_LONG 캡슐을 통해 이륙(Takeoff) 등의 상태 전이 명령을 전송하면, FC(비행 제어기)는 수신 후 처리 결과를 COMMAND_ACK로 반환한다. 이는 전송의 신뢰성을 보장하는 가장 기본적인 형태의 폐루프 제어 구조이다.
  • 실시간 상태 폴링(Polling) 및 이벤트 수신: COMMAND_ACK 수신 이후에도 기체가 실제로 이륙 고도에 도달했는지 확인하기 위해, GCS는 GLOBAL_POSITION_INT나 EXTENDED_SYS_STATE 같은 주기적 텔레메트리 스트림을 지속적으로 리스닝(Listening)한다.
  • 상태 머신 브릿지(State Machine Bridge): QGC 내 Vehicle 클래스는 수신된 상태 값들을 융합하여 독자적인 소프트웨어 상태 머신(Flying, Landed, Unknown 등)을 관리하며, 이를 Qt의 시그널-슬롯(Signal-Slot) 메커니즘을 통해 QML 프론트엔드로 비동기 통지하여 UI를 동기화한다.

3. 네트워크 전송 지연(Transmission Latency)의 기원과 특성

GCS와 PX4 시스템 사이의 무선 링크(SiK Telemetry, WiFi, LTE 등)는 필연적으로 물리적 파동 전파 시간, 패킷 인코딩/디코딩 오버헤드, 그리고 라우팅 대기 시간 등을 포함한다.
특히 다중 에이전트 환경(Swarm)이나 BVLOS(Beyond Visual Line of Sight) 비행에서는 지연 분산 가변성(Jitter)이 커진다. 이러한 레이턴시는 GCS 화면에 렌더링되는 기체의 현재 고도 및 자세가 수백 밀리초 단위의 “과거 상태“임을 의미하며, 정밀 통제 시 치명적인 상태 오판을 야기할 수 있다.

4. 핑-퐁(Ping-Pong) 측정을 통한 지연 보상 알고리즘

지연 시간에 따른 관제 오차를 극복하기 위해, PX4 및 QGC는 MAVLink 프로토콜의 네이티브 프로브 메시지인 PING을 활용한 왕복 지연 시간(Round Trip Time, RTT) 연속 측정 모델을 구현한다.

sequenceDiagram
    participant QGC as QGroundControl (GCS)
    participant Link as Data Link Layer
    participant PX4 as PX4-Autopilot

    QGC->>QGC: $T_{send}$ 기록
    QGC->>Link: MAVLink PING (seq: 1, target_msg)
    Link->>PX4: PING 수신
    PX4->>PX4: 즉각 응답 루틴 트리거
    PX4->>Link: MAVLink PING 대답 (seq: 1)
    Link->>QGC: PING_REPLY 수신
    QGC->>QGC: $T_{recv}$ 기록 및 RTT 계산
    QGC->>QGC: 단방향 지연(One-way Delay) 갱신: RTT / 2

4.1 타임스탬프 계산 및 필터링

GCS는 주기적(예: 1~5Hz)으로 PING 패킷을 대상 System ID로 전송한다. PING 패킷 내부에는 전송 주체 시스템의 마이크로초 단위 타임스탬프(Unix Epoch Base 또는 부트 타임)가 포함된다. PX4는 이를 수신하는 즉시 헤더를 변경하여 그대로 반송(Echo)한다.

  • 수학적 모델링: RTT는 RTT = T_{recv} - T_{send} 로 도출되며, 네트워크 상향/하향 채널이 대칭적(Symmetric)이라는 가정 하에 단방향 지연시간 t_{delay} 는 \frac{RTT}{2} 로 근사된다.
  • 이동 평균 필터(Moving Average Filter): 무선 링크의 패킷 드롭 및 전송 스파이크(Spike) 현상에 대응하기 위해, QGC는 수집된 최근 N개의 t_{delay} 샘플에 대하여 저역 통과 필터(Low-pass Filter)나 이동 평균을 취하여 평활화(Smoothing)된 베이스라인 지연값을 산출한다.

4.2 시스템 클럭 오프셋(Clock Offset) 동기화

단순히 전송 지연만을 측정하는 것을 넘어, PX4-Autopilot의 로컬 부트 타임(Uptime)과 GCS를 구동하는 호스트 머신의 시스템 클럭 사이의 오프셋(Offset)을 지속적으로 보정하는 TIMESYNC 메시지 교환 프로세스도 병행된다. 이를 통해 텔레메트리 데이터에 기록된 발생 시간(Timestamp)이 GCS 상의 특정 절대 시간 기준으로 언제 일어난 측정값인지 소급하여 역산할 수 있다.

5. 지연 보상이 이륙 상태 판별에 미치는 영향

위에서 보정된 t_{delay} 값은 폐루프 상태 모니터링의 정확성을 비약적으로 상승시킨다.
예를 들어, GCS가 현재 수신한 텔레메트리 상의 기체 고도가 z(t)이고 지연 보상 시간이 t_{delay}이며 기체의 Z축 상승 속도가 v_z일 때, GCS는 기체의 “현재 실제 고도 추정치” z'(t) 를 다음과 같이 1차 선형 보간(Linear Interpolation) 궤적으로 예측할 수 있다.
z'(t) = z(t) + v_z \times t_{delay}

이러한 로직을 응용함으로써, GCS는 아직 네트워크를 타고 전송 중인 “이륙 완료” 패킷 상태 전환을 미리 예측적으로 대비하거나, UI의 진행률 바(Progress Bar) 등을 끊김 현상(Stuttering) 없이 부드럽게 렌더링(Rendering)하여 관제사의 상황 인식 성능을 극도로 향상시킨다.

6. 결론

QGC 중심의 상태 모니터링 아키텍처는 단순한 패킷 폴링(Polling)을 초월하여, Ping-Pong 타임 동기화와 지연 추정(Latency Estimation) 알고리즘을 결합한 강력한 폐루프 시스템을 이룩하였다. 이 정교한 아키텍처는 PX4-Autopilot과의 협력을 통해 수백 밀리초의 물리적 통신 지연 환경 하에서도 운영자에게 고도로 일관된 상태 추적 능력을 유지하게 해주는 기반 관제 기술이라 분석할 수 있다.
