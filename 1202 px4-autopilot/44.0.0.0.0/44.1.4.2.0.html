44.1.4.2. 시스템 안전 장치(Failsafe) 개입 여부의 인터럽트(Interrupt) 기반 실시간 모니터링 및 천이 차단

자율 비행 로직이 이륙(Takeoff) 과정을 수행하는 도중에는, 기체의 하드웨어 결함이나 통신 단절, 배터리 전압 급감과 같은 치명적인 상태 변화가 언제든 발생할 수 있다. PX4-Autopilot은 이러한 비상 상황을 감지하고 시스템 상태를 안전 장치(Failsafe) 모드로 즉각 천이(Transition)시키는 강력한 메커니즘을 내장하고 있다. 본 절에서는 이륙 과정에서 발생하는 Failsafe 개입 조건과 실시간 인터럽트(Interrupt) 기반 모니터링 기법, 그리고 안전 장치 발동 시 기존 임무나 이륙 프로세스를 차단하는 과정을 상세히 분석한다.

1. Failsafe 시스템의 역할과 개입 조건

무인항공기의 오류 감지(Fault Detection) 메커니즘은 단순히 오류를 보고하는 데 그치지 않고, 위험 수준에 따라 기체의 행동을 즉각적이고 물리적으로 통제하는 방향으로 개입한다.

1.1 주요 Failsafe 감지 요소

PX4(v1.14.0 기준)에서 커맨더(Commander) 트리(Tree)와 상태 관리자(State Manager)는 다음과 같은 주요 에러 이벤트를 지속적으로 모니터링한다.

  • 통신 단절 (Data Link Loss / RC Loss): GCS와의 MAVLink Heartbeat가 설정된 시간(예: NAV_DLL_AH_T 파라미터) 이상 유실되거나, 조종기 데이터가 끊길 경우.
  • 배터리 임계치 도달 (Battery Failsafe): 전압이나 남은 용량 비율을 기반으로 Warning, Low, Critical, Emergency의 네 단계 구분에 따라 개입.
  • 자세 추정기 오류 (Estimator Failure): EKF2(Extended Kalman Filter 2) 모듈이 GPS, IMU, 나침반 사이의 혁신(Innovation) 오차가 커서 위치 추정을 포기하는 현상(Position Reject).
  • 위치 정보 상실 (Geofence / Loss of Position): 사전에 설정된 지오펜스(Geofence) 범위를 이탈하거나 유효한 3D Fix를 상실하는 경우.

1.2 상태 변이 차단(Transition Blocking)의 원리

Failsafe가 트리거(Trigger)되면, 기존에 진행 중이던 자율 이륙(Takeoff Command) 명령은 강제로 취소(Aborted)된다. 이는 시스템 내부에서 인터럽트 우선순위 최상단에 위치한 안전 관리 루프가 vehicle_status.nav_state를 Return, Land, 또는 Terminate 상태로 덮어쓰기(Overwrite)하기 때문이다.

2. 인터럽트 기반 실시간 모니터링

PX4 백엔드 시스템(주로 commander.cpp 및 관련된 워커 스레드)은 각 모듈에 대한 상태 검사를 고속 루프로 순회(Polling)함과 동시에, uORB(Micro Object Request Broker) 통신의 콜백 메커니즘을 이용해 비동기적 인터럽트(Interrupt) 방식으로 이벤트를 처리한다.

2.1 uORB 구독(Subscription) 및 이벤트 트리거

센서 데이터나 배터리 상태 등 치명적인 정보가 발행(Publish)되면, 폴링 주기까지 기다리지 않고 스케줄러가 Commander 스레드를 깨워(Wake-up) 즉시 대응하도록 설계되어 있다.

  • vehicle_status, battery_status, telemetry_status 등의 uORB 토픽은 시스템의 안전도 계산 모델(Safety State Machine)로 인가된다.

    PREFLIGHT --> ARMED : 시동 인가 (Arming)
    ARMED --> TAKEOFF : 이륙 명령 수신 (Takeoff Command)
    
    TAKEOFF --> FAILSAFE : 치명적 에러 감지 (Interrupt)

        Evaluate_Severity --> Warning : 알림 및 GCS 경고

3. GCS(QGroundControl)와의 연동 및 관제 실무

이륙 시점에서 Failsafe가 돌발적으로 개입하게 되면 GCS 측 제어 애플리케이션(QGC 또는 ROS 2 연동 시스템) 또한 이 상태 변화를 즉시 추적하고 관제 시스템의 흐름을 동기화해야 한다.

3.1 MAVLink 기반 상태 동기화 매핑

QGroundControl 측 시스템 아키텍처는 MAVLink의 HEARTBEAT 및 SYS_STATUS 메시지를 기반으로 백엔드(Vehicle Object)에서 차량의 Failsafe 돌입 여부를 판별한다.

  • SYS_STATUS 메시지 내의 비트마스크(Bitmask)인 onboard_control_sensors_health와 주요 상태 플래그를 정밀 분석하여, 디스플레이 상단에 빨간색 경고 배너(“CRITICAL: Battery Failsafe” 등)를 렌더링한다.

3.2 Ardupilot 대비 PX4의 Failsafe 철학

PX4와 Ardupilot 모두 훌륭한 안전 장치 처리를 지원하지만 처리 철학에 구조적 차이가 존재한다.

  • PX4: 중앙 집중형(Centralized) 커맨더 아키텍처를 통해 uORB 상태 변수를 단일 진실 공급원(Single Source of Truth)으로 삼아 전체 모듈의 시스템 상태를 일괄 중지시킨다. 즉, 이륙 모드(Takeoff Nav State) 자체를 탈취하여 비상 착륙 모드로 엄격히 천이한다.
  • Ardupilot: 각 비행 모드 모듈(Flight Mode Module) 내에서 Failsafe 상황 시 서브루틴을 호출하거나, 전역 조건에 의해 모드를 강제 변경(Change Mode)하는 분산 처리 성격을 조금 더 강하게 띤다.

결과적으로, 이륙 명령이 MAVLink를 통해 승인되었다 할지라도, 개발자는 ROS 2 로직이나 GCS 프로그래밍 과정에서 항상 Failsafe에 의한 상태 강제 천이가 발생할 수 있음을 염두에 두어야 한다. 이륙 명령 송신 이후 특정 시간(Timeout) 내에 목표 고도(Target Altitude) 도달에 실패하거나, 수신되는 상태가 이륙 모드를 이탈했다면 코드를 통해 이륙 포기(Takeoff Abort) 처리를 구현해야 매우 견고한 자율 에이전트 시스템을 완성할 수 있다.
