44.4.2 Flight Task 아키텍처 내 Auto-Takeoff 모드 진입 스케줄링 및 태스크 선점(Preemption) 과정

1. 개요 및 아키텍처 철학

현대의 무인항공기(UAV) 자율 비행 시스템에서 이륙(Takeoff)은 기체가 지면의 제약을 벗어나 3차원 공간으로 진입하는 가장 핵심적이고 위험한 단계이다. PX4-Autopilot(버전 1.14 기준)은 이러한 비행 상태의 복잡성을 관리하기 위해 플라이트 태스크 아키텍처(Flight Task Architecture)를 도입하여 제어 로직을 캡슐화하고 있다. 플라이트 태스크 아키텍처는 비행 궤적 생성 및 위치 제어 추종 명령을 모듈화하여, 특정 비행 모드에서 요구되는 동작(예: 이륙, 착륙, 위치 유지 등)을 독립적인 객체로 처리한다. 본 절에서는 지상 관제 시스템(QGroundControl v4.3 기준)이나 보조 컴퓨터(Companion Computer)로부터 자동 이륙(Auto-Takeoff) 명령이 하달되었을 때, 내부적으로 어떻게 태스크 스케줄링(Task Scheduling)이 이루어지고 기존 태스크를 안전하게 교체(Preemption)하는지에 대한 수학적 및 시스템적 과정을 상세히 분석한다.

2. Flight Task 기반 계층적 제어 구조

PX4 시스템에서 위치 제어기(Position Controller)는 단일한 거대한 제어 루프로 동작하지 않고, 현재 비행 모드에 따라 적절한 FlightTask 객체를 동적으로 활성화하여 궤적을 생성한다. 모든 개별 태스크는 최상위 가상 클래스(Virtual Class)인 FlightTask를 상속받는다.

자동 이륙 모드의 경우 FlightTaskAutoMapper 또는 FlightTaskAutoTakeoff 클래스를 통해 인스턴스화되며, 내부적으로 고도 목표점 및 상승 속도 프로파일을 계산하여 하위 위치 제어기에 전달한다. 이는 ArduPilot이 주로 전통적인 플라이트 모드(Flight Mode) 함수 호출 방식을 사용하여 모드별 제어 루프를 분기하는 것과 큰 차이가 있다. PX4의 객체 지향적 플라이트 태스크 설계는 코드의 응집도(Cohesion)를 높이고 모드 간 전환 시의 부작용(Side-effect)을 최소화하는 장점이 있다.

3. Auto-Takeoff 태스크 진입 스케줄링 로직

이륙 명령이 유효하다고 판명되면 커맨더(Commander) 모듈은 내비게이터(Navigator)를 거쳐 멀티로터 위치 제어기(mc_pos_control)에 플라이트 태스크 전환을 요청한다. 이 과정에서의 진입 스케줄링 알고리즘은 다음과 같이 수행된다.

  1. 상태 검증(State Verification): 기체가 현재 시동이 걸려있는지(Armed), 그리고 지면 접촉 상태(Landed State)인지 확인한다.
  2. 목표 파라미터 초기화(Target Parameter Initialization): 지정된 이륙 고도(h_{target})와 최대 상승 속도(v_{z,max}) 파라미터(MIS_TAKEOFF_ALT, MPC_TKO_SPEED 등)를 로드한다.
  3. 태스크 전환 요청(Task Switch Request): 위치 제어기 내부의 태스크 매니저(Task Manager)에게 Takeoff 태스크로의 전환을 큐(Queue)에 삽입한다.
  4. 활성화 및 궤적 융합(Activation and Trajectory Blending): 새로운 태스크가 활성화될 때, 급격한 제어 입력의 도약을 방지하기 위해 현재 기체의 센서 융합 상태(위치, 속도)를 초기 상태로 설정한다.

\begin{bmatrix} p_z(0) \\ v_z(0) \\ a_z(0) \end{bmatrix} = \begin{bmatrix} p_{z, est} \\ v_{z, est} \\ 0 \end{bmatrix}

여기서 p_{z, est}와 v_{z, est}는 EKF2(Extended Kalman Filter 2)로부터 추정된 현재의 수직 위치 및 수직 속도를 나타낸다.

4. 태스크 선점(Preemption) 및 충돌 회피 과정

태스크 선점이란, 우선순위가 높은 새로운 이벤트가 발생했을 때 현재 수행 중인 이벤트(태스크)를 중단하고 해당 자원을 새로운 태스크에 할당하는 운영체제 및 시스템 제어의 핵심 기법이다. PX4의 Flight Task 아키텍처에서는 무분별한 선점을 방지하기 위한 강력한 보호 계층을 갖추고 있다.

예를 들어, 조종사가 수동 위치 유지(Position Mode: FlightTaskManualPosition) 상태에서 비행 지시를 내리던 중 MAVLink를 통한 자동 이륙(Auto-Takeoff) 명령이 수신되었다고 가정하자.

  • 우선순위 역전 방지: 관제 시스템에 의한 자동 명령(Auto)이 수동 명령(Manual)을 선점하기 위해서는 현재 RC 입력 신호가 중립(Neutral) 부근에 있거나 Offboard 제어 권한이 허용된 상태여야 한다.
  • 연속성 보장(Smooth Transition): 태스크가 교체되는 정확한 시점(t = t_{switch})에 제어기 내부의 적분기(Integrator) 값을 초기화하거나 승계하여 모터 출력 신호(PWM)의 튀는 현상(Spike)을 방지한다.
  • ArduPilot과의 비교: ArduPilot의 경우에는 set_mode() 함수를 통해 강제적인 상태 레지스터 변경을 수행하며, 모드 진입 초기에 init() 함수에서만 적분기 초기화가 이루어진다. 반면 PX4는 FlightTask::updateInitialize() 함수가 매 루프 선점 상태를 감지하고 궤적의 연속성을 실시간으로 평가(Evaluation)하여 더 유연한 전환을 제공한다.

6. 학술적 시사점

PX4의 Flight Task 기반 선점 스케줄링 모델은 자율 항법의 아키텍처 논문(“Software Architecture for Autonomous Navigation”) 등에서 강조하는 다형성(Polymorphism)과 모듈화(Modularity)의 우수 사례로 볼 수 있다. 이를 통해 비행 제어 알고리즘 개발자는 전체 시스템의 안정성을 위협하지 않으면서도, 이륙 궤적 제어와 같은 특정 알고리즘만을 손쉽게 교체하거나 업그레이드할 수 있다. 이러한 독립적인 태스크 교체 메커니즘은 로보틱스 미들웨어 환경, 특히 ROS2 (uXRCE-DDS) 기반의 고도화된 스웜(Swarm) 비행 및 객체 추적 비행 제어 시스템을 구축할 때 그 확장성이 극대화된다.
