44.2.1.2. IEEE 754 표준 부동소수점(Floating-point) 데이터의 양자화(Quantization) 오차 및 전송 해상도(Resolution) 제약 분석

현대의 자율 에이전트 시스템(Autonomous Agent System)은 MAVLink 프로토콜을 통해 지상 관제 시스템(GCS)과 데이터를 교환하며 이착륙 및 경로 비행을 수행한다. 이때 COMMAND_LONG 메시지 페이로드에 적재되는 위치 좌표(위도, 경도) 및 고도 데이터는 32비트 단정밀도 부동소수점(Single-Precision Floating-point) 형식을 사용한다. 본 절에서는 IEEE 754 표준에 기반한 32비트 부동소수점 데이터가 글로벌 이동 및 이륙 제어 시 발생시킬 수 있는 양자화(Quantization) 오차의 수학적 원인과, 이를 극복하기 위한 PX4-Autopilot 및 프로토콜 설계 측면의 제약 및 대응 방안을 심도 있게 분석한다.

1. IEEE 754 단정밀도 부동소수점의 구조적 한계

MAVLink v1.0 및 초창기 v2.0 명령 체계에서 이륙 명령의 파라미터(Param 5, 6, 7)는 4바이트(32비트) 크기의 float 자료형으로 매핑된다. 이 자료형은 IEEE 754 표준을 따르며 부호(Sign) 1비트, 지수부(Exponent) 8비트, 그리고 가수부(Mantissa/Fraction) 23비트로 구성된다.

1.1 WGS84 좌표계상에서의 공간적 해상도(Resolution) 제약

지구 타원체 모델인 WGS84(World Geodetic System 1984) 좌표계에서 적도 부근의 1도(Degree)는 약 111,320 미터(m)의 물리적 거리에 해당한다.
가수부 23비트가 표현할 수 있는 유효 숫자(Significant Decimal Digits)는 십진수 기준으로 약 7.2자리(\log_{10}(2^{24}))에 불과하다. 위도와 경도 값은 절대적 수치로 최대 \pm180^{\circ}까지 값이 커질 수 있으므로, 지수부가 커질수록 가수부가 표현하는 최하위 비트(LSB, Least Significant Bit)의 실제 단위 거리도 커지게 된다.

  • 최대 오차 범위: 절대 좌표값이 100^{\circ} 근처일 경우, IEEE 754 float의 LSB 이동 크기는 약 1.52 \times 10^{-5} 도로 계산되며, 이를 거리로 환산하면 최소 해상도 오차가 약 1.7미터(m) 상하로 발생하게 된다.
  • 이러한 양자화 오차(Quantization Error)는 이착륙 패드의 정밀 타겟팅(Precision Landing)이나 좁은 공간에서의 자율 제어에 있어서 치명적인 불안정성을 유발한다.

1.2 고도 제어(Target Altitude) 데이터의 양자화 특성

고도(Altitude, Param 7)의 경우, 값싸고 넓은 스케일 범위를 다루기에는 유리하나 미세 조정 시 이산화(Discretization)가 발생한다. 지표면 대비 100m 이하의 낮은 고도에서는 \text{오차} < 1\text{cm} 구조를 만족하지만, 전역 좌표값(AMSL)이 1,000m 단위를 넘어가면 수 센티미터 수준의 단차(Step)가 발생하여 PID 제어기의 미분기(Derivative)에 고주파 노이즈(Noise) 증폭을 유발할 수 있다.

2. 오차 극복을 위한 프로토콜 진화와 PX4-Autopilot의 아키텍처

MAVLink 커뮤니티와 PX4 개발진은 이러한 수학적 본질을 극복하기 위해 통신 및 제어 아키텍처 레벨에서 다각적인 해결책을 도입하였다.

2.1 COMMAND_INT 기반 정밀도 향상

단정밀도 float 구조체인 COMMAND_LONG의 대안으로, 센티미터 이하 단위까지 위치를 매핑하기 위해 위치 파라미터(위도, 경도)를 32비트 정수형(int32_t)으로 선별 변환한 COMMAND_INT 메시지(Message ID: 73)가 표준으로 자리잡았다.

  • 스케일링 인자(Scaling Factor): 위경도 값에 10^7을 곱하여 정수로 캐스팅(Casting)한다. 이 변환을 통해 좌표 해상도는 일률적으로 **1.11 \text{cm}**의 분해능 상수(Constant Resolution)를 획득한다.
  • PX4-Autopilot의 내부 인터페이스인 vehicle_command uORB 토픽 역시 int32_t 구조와 이중 정밀도(double, 64비트) 변수를 선택적으로 수용하도록 진화하였다.

2.2 Ardupilot과 PX4 제어 메커니즘의 차이 비교

Ardupilot과 PX4-Autopilot 두 오픈소스 비행 제어 스택 모두 COMMAND_INT 프로토콜을 적극 지원하나, 내부 변환(Internal Conversion) 철학에는 다소 차이가 존재한다.

  • PX4-Autopilot: 내비게이션 좌표계 처리에 있어 64비트 단위의 계산(map_projection 라이브러리의 이중 정밀도 적용)을 통해 로컬 NED(North-East-Down) 평면 투영 시 오차를 최소화한다. uXRCE-DDS 브리지를 활용한 ROS 2 연동에 있어서도 기본적으로 double 타입 전송을 우선시하여 에이전트단의 손실을 차단한다.
  • Ardupilot: 메모리 최적화를 중시하는 오랜 레거시(Legacy) 코드의 영향으로 내부 루프의 많은 부분에서 32비트 스케일링 체제에 의존해왔다. 특히 EKF(Extended Kalman Filter) 단계 이전에 정수 기반 위치 옵셋 처리를 통해 부동소수점 오차 누적을 우회하는 방식을 역사적으로 더 선호하였다.

결론적으로, GCS 통신 시스템 구축이나 자율 관제 제어기 스택(Controller Stack)을 개발하는 엔지니어는 MAVLink COMMAND_LONG이 지닌 부동소수점 양자화 한계를 철저히 이해하고, 이륙 좌표나 임무 경로 설정 시 의무적으로 COMMAND_INT 기반의 고해상도 명령 트랜잭션 프로토콜을 채택해야 한다.
