44.2.1.3. 순환 중복 검사(CRC)를 통한 전송 계층 무결성 및 시퀀스 번호(Sequence Number) 기반 패킷 누락 검출

무인항공기(UAS) 지상 관제 시스템(GCS)과 탑재형 비행 제어기(Flight Controller, FC) 간의 통신은 주로 노이즈(Noise)가 심한 RF(Radio Frequency) 무선 텔레메트리(Telemetry)를 통해 이루어진다. 이륙(Takeoff) 명령과 같은 치명적인(Mission-Critical) 제어 패킷이 전송 중 비트 반전(Bit Flip)이나 간섭(Interference)으로 인해 변조된다면, 기체는 예측 불가능한 궤적으로 오작동할 위험이 몹시 크다. 본 절에서는 MAVLink 프로토콜이 전송 계층(Transport Layer) 수준에서 패킷의 무결성(Integrity)을 보장하기 위해 채택한 X.25 기반 순환 중복 검사(CRC, Cyclic Redundancy Check) 알고리즘과, 제어 명령의 손실(Drop)을 감별해내는 시퀀스 번호(Sequence Number) 아키텍처를 상세히 분석한다.

1. MAVLink 패킷 무결성: ITU X.25 CRC-16 메커니즘

MAVLink 프로토콜(v1.0 및 v2.0 모두 해당)은 패킷의 끝부분에 2바이트(16비트) 길이의 체크섬(Checksum) 필드를 부착한다. 이 체크섬은 전송 중 데이터 변조를 수학적으로 검출해내는 핵심 안전 장치이다.

1.1 CRC-16-CCITT 사양과 다항식(Polynomial) 연산

MAVLink는 0x1021 (수학적 표현: x^{16} + x^{12} + x^5 + 1) 다항식을 사용하는 ITU X.25 규격의 CRC-16 알고리즘을 사용한다.
송신단(예: QGroundControl)은 페이로드(Payload)와 헤더 속성값을 모두 바이트 단위로 순회(Iterate)하며 CRC 레지스터의 값을 갱신(Update)한다. 수신단(PX4-Autopilot)은 메시지를 파싱(Parsing)할 때 동일한 연산을 독립적으로 수행한 후, 패킷에 동봉된 2바이트 체크섬 결과와 비교한다.

  • 결과가 일치하지 않으면(CRC Mismatch), PX4의 mavlink_receiver 모듈은 해당 패킷을 조용히 폐기(Drop)하고 통신 통계의 msg_rx_errors 카운터를 증가시킨다.
  • 이 방식은 단순한 체크섬(합계) 연산보다 훨씬 더 높은 확률로 다중 비트 반전 오류(Burst Error)를 검출해낸다.

1.2 CRC_EXTRA: 버전 불일치 및 디코딩 오류 차단

프로토콜의 독창적인 안전 요소 중 하나는 CRC_EXTRA 바이트의 추가이다.

  • GCS와 드론 양측은 메시지 정의(XML 기반)에 따라 COMMAND_LONG (Message ID: 76) 메시지의 페이로드 길이나 파라미터 타입이 해시(Hash)된 1바이트의 추가 시드(Seed) 값(CRC_EXTRA)을 공유한다.
  • CRC 계산의 가장 마지막 단계에 이 CRC_EXTRA 값을 연산에 포함시킨다.
  • 결과적으로, GCS와 PX4-Autopilot이 서로 호환되지 않는 MAVLink 다이얼렉트(Dialect) 버전을 사용하고 있다면 메시지 구조는 파싱되더라도 최종 CRC 검사 단계에서 완전히 실패하게 되어, 잘못된 명령어로 인한 기체의 제어기(Controller) 오작동을 원천적으로 차단한다.

2. 시퀀스 번호(Sequence Number) 기반 패킷 누락 검출

무결성 검사를 통과하더라도 패킷 자체의 유실 현상을 감지하기 위해 MAVLink 헤더에는 seq (시퀀스 번호, 1바이트) 필드가 존재한다.

2.1 패킷 누락(Drop) 검출 원리

QGroundControl이 이륙 명령 전송 시 부여한 시퀀스 번호가 104번이었다면, 다음 송신 패킷 메세지는 반드시 시퀀스 식별자 105를 가져야 한다. (값은 0~255 범위에서 순환(Wrap-around)한다.)
PX4의 통신 모듈은 이전에 수신된 seq 값과 현재 수신된 seq 값을 비교하여 라우팅 지연이나 RF 드롭아웃(Dropout) 현상을 감지해낸다.

sequenceDiagram
    participant GCS as QGroundControl
    participant RF as Radio Telemetry
    participant PX4 as PX4-Autopilot
    
    GCS->>RF: COMMAND_LONG (seq=12) [이륙 명령]
    note right of RF: CRC Check 통과
    RF->>PX4: COMMAND_LONG (seq=12) 수신
    PX4-->>GCS: COMMAND_ACK (seq=12) 수락
    
    GCS->>RF: MISSION_ITEM (seq=13) [노이즈로 인하여 소실]
    note over RF,PX4: Message Dropped in Air
    
    GCS->>RF: PARAM_REQUEST_READ (seq=14)
    RF->>PX4: PARAM_REQUEST_READ (seq=14)
    note right of PX4: 예상 seq(13)과 수신 seq(14) 불일치 감지!
    PX4->>PX4: 패킷 누락 로깅 (Packet Loss Detected)

2.2 Ardupilot 대비 통신 상태 관리 차이

  • PX4-Autopilot: mavlink_receiver 스레드 내에서 시퀀스 누락을 감지하면 내부 uORB 통계 토픽(telemetry_status)에 지표(Metric)로 실시간 반영한다. 패킷 손실률(Packet Loss Rate)이 임계치를 초과할 경우 시스템 보안 기화기(Commander)가 능동적으로 Failsafe 메커니즘(Return 또는 Land)을 즉시 발동하도록 아키텍처가 긴밀히 결합되어 있다.
  • Ardupilot: 유사하게 시퀀스 누락을 카운트하고 통계 지표로 활용하지만, GCS 측(Mission Planner 등)의 Heartbeat 타임아웃 룰에 대한 반응 의존도가 약간 더 높은 경향이 있다. 즉, 부분적인 패킷 손실에 대해 EKF(Estimator) 보정이나 자체 제어기 유지를 통해 조금 더 관대하게 지켜보는 성향이 강하다.

결론적으로, 로봇 운영체제(ROS 2)나 독자적인 GCS 환경을 구축하여 이륙 명령을 송신하는 개발자는, COMMAND_LONG 메시지를 전송하고 단순히 대기할 것이 아니라 패킷 시퀀스 번호가 정상적으로 증분되었는지와 대상 기체로부터 올바른 COMMAND_ACK 응답 지표가 수신되었는지를 필수적으로 모니터링해야 한다.
