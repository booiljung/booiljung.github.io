44.6.3. 고도 도달 판별 알고리즘의 유클리드 기하학적 구성

자율 비행 기반의 무인항공기(UAV) 관제 시스템에서 이륙(Takeoff)이나 특정 웨이포인트(Waypoint)로의 이동 시, 기체가 목표 고도에 정상적으로 도달했는지를 판단하는 알고리즘은 비행 상태 머신(State Machine)의 전환을 결정하는 핵심 요소이다. PX4-Autopilot(펌웨어 v1.14.x 기준)에서는 단순한 1차원적 Z축 고도 측정치 비교를 넘어서, 3차원 공간상의 위치 오차 벡터를 활용하는 유클리드 기하학(Euclidean Geometry)적 기법을 통해 고도 도달 여부를 판정한다. 본 절에서는 이러한 고도 도달 판별 알고리즘의 수학적 모델링과 PX4 내부 아키텍처에 구현된 검증 로직을 분석한다.

1. 유클리드 기하학 기반 고도 도달 판별의 필요성

일반적으로 고도(Altitude)는 Z축에 대한 스칼라 값으로 이해하기 쉽지만, 실제 비행 제어 환경에서 기체는 항상 3차원 공간 즉, 수평 평면(X-Y Plane) 상의 외란(바람 등) 및 위치 제어기의 오버슈트(Overshoot)와 결합된 동역학적 궤적을 그린다. 단순히 Z축 고도 오차만을 비교하여 도달 여부를 판단할 경우, 기체가 목표 수평 위치에서 크게 벗어나 있음에도 불구하고 고도만 일치하면 다음 임무로 전환되는 위험한 논리적 오류(Logical Error)가 발생할 수 있다.

따라서 PX4-Autopilot의 내비게이터(Navigator) 모듈은 목표 위치를 중심으로 하는 실린더형 허용 구역(Cylindrical Acceptance Volume) 또는 구형 반경(Spherical Radius) 기반의 유클리드 기하학 구성을 채택하여, 수평 거리(Horizontal Distance)와 수직 오차(Vertical Error) 조건을 동시에 충족할 때 고도 도달로 인정한다.

2. 오차 허용 구역(Acceptance Volume)의 수학적 모델링

기체의 현재 3차원 위치 벡터를 \mathbf{p} = [x, y, z]^T, 목표 궤적 상의 목표 위치 벡터를 \mathbf{p}_{ref} = [x_{ref}, y_{ref}, z_{ref}]^T 라 정의한다. 고도 도달 판별은 목표 지점 주변에 기하학적인 오차 허용 영역을 설정하는 것으로 귀결된다.

2.1 수직 반경 오차 평가 (Vertical Error Evaluation)

고도 제어의 가장 기본적인 조건은 수직 방향의 편차 e_z가 시스템에서 정의한 허용 반경(Acceptance Radius), 즉 파라미터 NAV_MC_ALT_RAD (멀티콥터 기준 고도 허용 오차 반경) 이내로 들어와야 한다는 것이다.
e_z = \vert z - z_{ref} \vert \le R_{z, accept}

전체 3차원 유클리드 노름 검증 (3D Euclidean Norm)

수직 오차가 충족되었다고 하더라도, 수평 위치 오차 반경 e_{xy} = \sqrt{(x - x_{ref})^2 + (y - y_{ref})^2} 역시 수용 가능 구역 이내이어야 안전한 고도 도달로 판정한다. 엄격한 임무 요건(예: 정밀 정지 비행)에서는 전체 위치 오차에 대한 유클리드 노름(Euclidean Norm) 검사를 수행한다.

\lVert \mathbf{e} \rVert_2 = \sqrt{e_x^2 + e_y^2 + e_z^2} \le R_{accept}

여기서 R_{accept}는 파라미터 NAV_ACC_RAD 설정값에 해당한다. 이 기하학적 검사는 QGroundControl(버전 v4.3.x 이상)을 통해 사용자가 기체 파라미터를 조정함으로써 임무의 정밀도를 동적으로 추종하고 설정할 수 있게 해준다.

3. PX4-Autopilot 내부 판별 알고리즘 흐름도

PX4 소스코드에서 navigator 애플리케이션 내의 위치 도달 판정은 이 유클리드 계산을 기반으로 상태(State)를 주기적으로 갱신한다. 다음은 기체가 이륙 명령을 받고 목표 고도에 도달하는지 여부를 검증하는 로직의 흐름을 나타낸 것이다.

graph TD
    A[이륙 상태(Takeoff State) 진입] --> B{현재 텔레메트리 위치 수신<br/>x, y, z 업데이트}
    B --> C[수직 오차 계산<br/>ez = \vert z - z_ref \vert]
    C --> D[수평 오차 계산<br/>exy = sqrt x_err^2 + y_err^2]
    D --> E{ez <= NAV_MC_ALT_RAD ?}
    E -- Yes --> F{exy <= NAV_ACC_RAD ?}
    E -- No --> B
    F -- Yes --> G[방향각(Yaw) 정렬 오차 확인]
    F -- No --> B
    G --> H{고도 및 위치 도달<br/>최종 판정 완료}
    H --> I[상태 전환: Hold 또는 Mission]

위의 흐름도에서 보듯, 고도 축의 검증 로직은 단절된 1차원 제어가 아니라, X, Y축 위치 정보와 깊은 기하학적 연관성(Geometrical Correlation)을 갖는 다차원 제어 로직이다.

4. PX4와 Ardupilot 간의 기하학적 판별 관제 차이점

두 오픈소스 비행 제어 시스템은 위치 도달 검증(Arrival Verification)에서 다소 구분되는 철학적 접근을 취한다.

판별 구조 형태수직 오차 반경과 수평 오차 반경을 조합한 실린더형 허용 구역(Cylinder) 검사를 엄격히 처리한다.WPNav 라이브러리의 방사 거리(Along-track distance)와 고도 편차를 융합하여 판별한다.
파라미터 모델링NAV_ACC_RAD, NAV_MC_ALT_RAD 파라미터로 명시적인 기하학적 유클리드 구역을 설정한다.WPNAV_RADIUS, TECS 등의 파라미터가 롤/피치 동역학 및 수직 이동 지연과 결합된다.
수학적 엄밀성상태 머신 상에서 명확한 유클리드 벡터 스칼라 거리를 연산하여 타겟 조건을 적용한다.제어기의 Z-컨트롤 오차 수렴성(Error Convergence)의 동적 지연 시간에 일부 의존한다.

5. 소스 코드 구현 요약 시사점

결과적으로 PX4-Autopilot의 고도 도달 판별 알고리즘은 1차원 수직 데이터에 대한 평면적 조건문이 아니며, 3차원 유클리드 기하학에 기초한 공간 종속적 문제이다. 지상 관제 시스템(GCS) 애플리케이션 개발 시, QGroundControl의 상태 메시지뿐 아니라 PX4 내부에서 정의하는 NAV_ACC_RAD 제약 조건이 펌웨어 레벨에서 어떻게 동적으로 연산되는지를 이해해야 한다. 이를 통해 다중 에이전트(Swarm) 환경에서 속도-위치 교착 상태(Deadlock)—기체가 고도에는 도달했으나 수평 위치 불일치로 상태가 전환되지 않는 문제—를 선제적으로 방지하고 안전한 관제 망을 구성할 수 있다.
