44.7.2.1. 재전송 폭주(Broadcast Storm) 방지를 위한 지수 백오프(Exponential Backoff) 알고리즘 및 패킷 생존 시간(TTL) 적용

비동기 무선 통신(Asynchronous Wireless Communication) 환경에서 무인항공기(UAV) 지상 관제 시스템(GCS)과 비행 제어기(Flight Controller) 간의 상태 기반 제어 명령을 주고받을 때, 무선 링크의 불안정성은 필연적으로 패킷 유실을 동반한다. 이러한 유실에 대처하기 위해 단순한 고정 인터벌 재전송(Fixed-interval Retransmission) 방식을 채택할 경우, 통신 음영 지역이나 간섭 심화 구간에서 지속적인 패킷 적체와 MAVLink 트래픽 폭주(Broadcast Storm) 현상을 초래할 수 있다. 특히 이륙(Takeoff) 명령과 같이 치명적이고 상태 의존적인(State-dependent) 제어 입력을 처리할 때는, 네트워크 대역폭 포화를 방지하기 위한 지수 백오프(Exponential Backoff) 알고리즘과 더불어 지연 도달 패킷에 의한 오작동을 차단하는 패킷 생존 시간(Time-to-Live, TTL) 적용 메커니즘이 필수적으로 요구된다.

이 문서에서는 최신 펌웨어(PX4-Autopilot v1.14.0 이상)와 QGroundControl(v4.3 이상)의 통신 아키텍처를 기반으로, GCS 단에서 MAVLink 제어 패킷의 생애 주기(Lifecycle)를 관리하고 브로드캐스트 스톰을 회피하기 위한 제어 시스템의 논리적 메커니즘을 상세히 다룬다.

1. 지수 백오프(Exponential Backoff) 알고리즘의 논리적 아키텍처

지수 백오프(Exponential Backoff)는 초기 재전송 대기 시간(Timeout)을 점진적으로 증가시켜, 일시적인 네트워크 혼잡을 해소하고 RF 모뎀 버퍼 오버플로우(Buffer Overflow)를 방지하는 충돌 회피 알고리즘이다. QGroundControl은 중요한 상태 천이 명령어(예: MAV_CMD_NAV_TAKEOFF) 전송 시 ACK/NACK 응답 수신 대기를 위한 유한 상태 기계(Finite State Machine, FSM)를 구동하며, 재전송 타이머를 가변적으로 연장한다.

1.1 수학적 모델 및 가중치 계산 체계

GCS에서 n번째 재전송(Retransmission) 스케줄링이 발생할 때의 대기 시간(Timeout) T_{n}은 다음과 같이 지수승 단위로 증가하도록 설계된다.

T_{n} = \min \left( T_{max}, \ T_0 \times \beta^n \right) \quad (\text{단, } n \ge 0)

  • T_0: 기본 응답 대기 시간 (Base Timeout, 통상적으로 MAVLink COMMAND_LONG 프로토콜에서는 500ms 내외로 설정됨)
  • \beta: 백오프 승수 (Backoff Multiplier, 통상 1.5 ~ 2.0 사이의 값이 적용되어 지수적 증가를 유도)
  • T_{max}: 최대 대기 시간 (Maximum Timeout Threshold), 명령어의 상태 동기화가 무의미해지지 않도록 지수 팽창의 상한선을 제어. 지정된 시간(예: 3초)을 초과하는 대기 상태는 시스템 장애(Fail-state)로 즉시 처분(Abortion)된다.

1.2 QGroundControl 및 PX4의 적용 사례와 ArduPilot과의 차이

QGroundControl(QGC)의 Vehicle 클래스 및 내부 MAVLink 프로토콜 추상화 계층에서는, COMMAND_LONG 또는 COMMAND_INT 전송 인스턴스에 가변 타이머를 결합하여 백오프를 관리한다. 특정 명령에 대해서는 NACK를 명시적으로 수신하기 전까지 닫힌 루프(Closed-loop)를 구성하여 응답 여부를 모니터링한다.

반면, 레거시 ArduPilot 환경과 Mission Planner의 조합에서는 MAVLink 통신 유실 보상을 위해 단순한 N주기 반복 전송(예: 초당 2회 재요청)을 사용하는 경우가 존재했다. 이는 네트워크 링크 여유 대역폭(Bandwidth)이 충분할 때는 빠른 회복을 보장하지만, 무선 교란 모델(Wireless Interference Model) 하에서는 반송파 감지 충돌(Carrier Sense Collision) 빈도를 높여 전체 시스템의 패킷 처리량(Throughput)을 수직 하강시키는 원인(Collision Avalanche)이 된다. QGroundControl의 지수 백오프 적용은 이러한 하위 RF 모뎀(예: SiK Radio) 측 버퍼(Tx Buffer) 포화를 원천적으로 차단하는 진보된 접근이다.

2. 패킷 생존 시간(TTL: Time-To-Live) 적용 메커니즘

유선 이더넷(Ethernet) 프로토콜의 IP 헤더에서 볼 수 있는 홉 기반(Hop-count) TTL과 달리, 무선 텔레메트리에서의 TTL은 “시간 기반의 논리적 만료 시간(Temporal Expiration Period)“을 의미한다. 일시적인 통신 단절(Link Loss) 상태에서 GCS가 비동기적으로 내린 명령이 오랜 시간 뒤 링크가 복구됨과 동시에 기체로 일괄 전달되는 현상(Stored Command Flush)은 극도로 위험하다.

2.1 시지연 명령 처리의 위험성 방어 로직

사용자가 QGroundControl에서 ‘이륙(Takeoff)’ 버튼을 탭(Tap)했을 때, 해당 명령이 즉결 처리되지 않고 10초 후에야 PX4의 Commander FSM 계층으로 유입될 경우, 기체는 사용자의 예측을 벗어난 시점에서 급작스런 고도 상승을 야기할 수 있다. PX4-Autopilot 및 GCS 프로토콜 계층은 이러한 “유통기한(TTL)이 경과된(Stale)” 명령을 폐기(Discard Processing)하기 위해 다음과 같은 검증 파이프라인을 거친다.

  1. 송신측 (QGroundControl) 큐 매니저 검열: GCS 내부의 Queue Manager는 송신 대기열(Tx Queue)에 명령어 패킷을 적재한 시점으로부터 타임스탬프(Timestamp)를 추적한다. 운영자가 지정한 임계 생존 시간(Critical TTL Payload, 예: 2500ms)이 만료되기 이전에 텔레메트리 트랜시버(Transceiver) 계층으로 DMA 인가되지 못했다면, 해당 패킷은 즉시 파기되고 사용자 UI에 타임아웃 오류(Timeout Error)를 표출한다.
  2. 수신측 (PX4-Autopilot) 로컬 시스템 타임 스니핑(Time Sniffing): 시간 동기화(Time Sync) 프로토콜에 의해 GCS-기체 간 마이크로초 단위의 시계열(Timesync)이 구성된 경우, PX4의 시스템 백그라운드는 수신된 명령어 인스턴스의 하프-라이프(Half-Life) 파손 여부를 추론한다. 만일 GCS의 발송 시간으로부터 현재 수신 시간의 델타(\Delta t)가 비행 동역학(Flight Dynamics)의 데드라인을 침범했다고 판단되면, PX4 모듈은 해당 COMMAND_LONG을 수리하지 않고 MAV_RESULT_DENIED 와 함께 STATUSTEXT로 경고를 회신한다.

3. 이륙 상태 제어 과정에서의 백오프와 TTL 처리 흐름

    Cpp->>TxTx: 2차 전송 시도 (Tx1)

    Cpp->>TxTx: 3차 전송 시도 (Tx2)

    alt TTL 생존 시간 초과 분석 (Drop Detection)

        Cpp-->>UI: 명령 폐기(Command Discarded) 알람 및 진행 상태 롤백
    else 무선 링크 복구 시
        TxTx->>FC: 수신 성공 (MAV_CMD_NAV_TAKEOFF)
        FC->>FC: 현재 기체 상태 및 안전 조건(Failsafe) 재검증
        FC-->>Cpp: ACK 판정 (MAV_RESULT_ACCEPTED)
        Cpp-->>UI: 이륙 정상 진행 (Progress Bar)
    end

4. 고려 사항

본질적으로 지수 백오프 알고리즘과 TTL 폐기 로직의 조합은, 분산 실시간 처리 시스템(Distributed Real-time System)이 직면하는 과도한 트래픽 지연(Congestion Delay)과 데이터 왜곡 현상을 우아하게 억제한다.

  • GCS 개발 및 PX4 펌웨어 파라미터 튜닝 시, COM_RC_LOSS_T(조종기 신호 유실 임계값) 및 COM_DL_LOSS_T(데이터 링크 유실 임계값)와 긴밀한 동기화가 이루어져야 한다. 백오프의 최대 초과 한계(T_{max})가 데이터 링크 유실 판단 임계값보다 크면, GCS에서는 전송을 재시도하고 있는 도중에 FC(비행 제어기) 단에서 커넥션 스킵(Connection fail-safe) 트리거가 발동하여 예기치 않은 RTL(복귀 기동)이 수행되는 원치 않는 경쟁 상태(Race Condition)를 초래할 수 있다.
  • 따라서, 이륙 전 전송 프로토콜 스케줄러(Scheduler)의 논리적 주기는 항상 하위 모터/액추에이터(Actuator) 제어 주기와 시스템 페일세이프 주기의 서브 도메인 내에서 안전하게 관리되어야 함을 명심해야 한다.
