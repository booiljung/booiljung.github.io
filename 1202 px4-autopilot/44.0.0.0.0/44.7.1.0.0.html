44.7.1 이륙 명령 거부(NACK) 수신 시 MAVLink 결과 코드 파싱 및 상태 머신 롤백(State Machine Rollback) 트랜잭션

지상 관제 시스템(GCS)이 다기능 무인항공기(UAS) 제어를 위해 PX4-Autopilot으로 이륙(Takeoff) 명령을 전송할 때, 해당 명령이 항상 성공적으로 수용되는 것은 아니다. PX4는 다양한 비행 전 안전 검사(Pre-flight Checks)와 센서 상태, 그리고 현재 구동기(Actuator)의 준비 상태를 엄격하게 평가한다. 만약 조건이 충족되지 않으면 PX4는 이륙 명령을 거부(NACK, Negative Acknowledge)하며, GCS는 이를 정확히 파싱(Parsing)하여 사용자에게 원인을 알리고 시스템을 안전한 상태로 롤백(Rollback)해야 한다. 본 절에서는 MAVLink 프로토콜의 COMMAND_ACK 메시지를 기반으로 한 제어 명령 거부 메커니즘과, 이에 수반되는 상태 머신(State Machine)의 트랜잭션 복구 로직을 심층적으로 분석한다.

1. MAVLink COMMAND_ACK 메시지 및 결과 코드 파싱 로직

지상 관제 시스템(예: QGroundControl v4.x 이상)에서 비동기적으로 전송하는 MAV_CMD_NAV_TAKEOFF 명령에 대해, PX4(v1.14.x 펌웨어 기준)는 MAVLink의 COMMAND_ACK (#512) 메시지로 응답한다. 이 응답 메시지의 핵심은 result 필드이며, 이는 MAV_RESULT 열거형(Enum) 값을 포함하고 있다. 이륙 명령이 정상적으로 수용되면 MAV_RESULT_ACCEPTED가 반환되지만, 거부될 경우 다음과 같은 세부적인 NACK 코드가 반환된다.

  • MAV_RESULT_TEMPORARILY_REJECTED (임시 거부):
    현재 기체가 수 초 내에 해결될 수 있는 초기화 작업을 진행 중이거나, GPS 3D Fix가 아직 완전히 안정화되지 않은 경우 발생한다. GCS는 사용자 개입 없이 일정 시간(Timeout) 대기 후 자동으로 재전송(Retransmission) 스케줄링을 트리거할 수 있다.
  • MAV_RESULT_DENIED (명령 거부):
    기체 상태가 이륙을 절대적으로 허용하지 않는 경우 반환된다. 예를 들어, 기체가 아직 시동(Arm)되지 않았거나, 치명적인 하드웨어 결함 플래그가 설정된 채로 이륙 명령이 강제 전송된 경우이다.
  • MAV_RESULT_UNSUPPORTED (지원되지 않음):
    펌웨어 버전 차이 혹은 특정 다중 로터(Multi-rotor) 프레임 구성에서 해당 MAVLink 이륙 커맨드 파라미터 규격을 지원하지 않는 경우 반환된다.

GCS의 통신 파서(Parser)는 수신 스레드에서 COMMAND_ACK의 command 필드가 이륙(MAV_CMD_NAV_TAKEOFF)인지 확인한 후, result 필드의 값에 따라 적절한 예외 처리 루틴으로 진입해야 한다.

2. 상태 머신 롤백(State Machine Rollback) 트랜잭션

이륙 명령이 거부된 상황은 단순한 통신 실패를 넘어, 시스템 내부의 상태(State) 불일치를 유발할 수 있다. 예를 들어, GCS의 UI 프론트엔드는 사용자가 “이륙” 슬라이더를 밀어 이미 “이륙 진행 중” 상태로 화면을 갱신했지만, PX4 시스템은 명령을 수용하지 않아 여전히 “지상 대기(Landed)” 상태에 머물러 있을 수 있다. 이러한 분산 시스템 구조의 상태 비동기성(Asynchrony)을 해결하기 위한 절차가 상태 머신 롤백(Rollback) 트랜잭션이다.

2.1 롤백 트랜잭션의 주요 절차

  1. 상태 불일치 감지(Mismatch Detection):
    GCS 내부의 기체(Vehicle) 관리 매니저는 명령 전송 후, 타임아웃 타이머를 시작함과 동시에 예상되는 ‘이륙’ 상태 전이를 대기(Pending) 상태로 등록한다.
  2. NACK 수신 및 롤백 트리거:
    PX4로부터 MAV_RESULT_DENIED와 같은 거부 ACK를 수신하면, GCS 매니저는 즉시 대기 상태(Pending State) 트랜잭션을 취소(Abort)한다.
  3. UI 및 내부 모델 롤백:
    C++ 백엔드는 비동기 시그널을 방출하여 QML UI에 이륙 실패를 통보하고, 기체 고도 및 모바일 프로그레스 바 상태를 이전의 안정 상태(예: Arming 상태에서의 지상 대기)로 롤백한다. 사용자에게는 “Takeoff Denied: Condition Not Met“과 같은 알림(Statustext)을 출력한다.

2.2 PX4-Autopilot 내의 커맨더(Commander) 롤백

PX4 내부적으로도 커맨더 모듈은 롤백을 수행한다. 이륙 트랜지션을 준비하기 위해 잠시 할당했던 제어기(Controller)의 적분기(Integrator) 상태를 초기화하고, 추력(Thrust) 출력 파라미터를 최소화하여 예기치 않은 모터 튀어오름(Spike)을 방지한다. 이는 미션 수행 중 실패 시 치명적인 돌발 비행을 차단하는 매우 강력한 안전 장치로 작용한다.

4. 아키텍처적 의의 및 비교

과거의 단순한 RC 조종 기반 비행 제어에서는 명령이 수용되지 않았을 때 모터가 반응하지 않는 수준에 그쳤다. 하지만 MAVLink 기반의 GCS-Flight Controller 아키텍처에서는, 비동기 명령에 대한 명시적인 부정 응답(NACK)과 이에 따른 트랜잭션 롤백 시스템이 필수적이다.

특히 Ardupilot 생태계의 Mission Planner 등은 NACK 발생 시 단순히 로그를 화면에 출력하는 방식을 취하며 트랜잭션 관리 계층이 다소 약한 경향이 있었으나, 최근 QGroundControl과 PX4의 조합은 데이터베이스(DB) 수준의 ACID(원자성, 일관성, 고립성, 지속성) 특성을 차용한 엄격한 상태 머신 롤백 메커니즘을 적용하여 예측 불가능한 시스템 오라클(Oracle) 오류를 방어한다. 이는 상업적 지상 관제망 운용 시 시스템 신뢰성을 담보하는 가장 기초적이고 강력한 설계 원칙 중 하나이다.
