44.5.1 PX4 Land Detector 모듈의 논리적 아키텍처 및 실시간 운영체제(RTOS) 내 워커 스레드(Worker Thread) 상태 판별 주기

PX4-Autopilot은 단순한 펌웨어를 넘어 하드웨어 실시간 제어를 보장하기 위해 POSIX 호환 실시간 운영체제(RTOS)인 NuttX 위에서 구동되는 고도화된 소프트웨어 아키텍처를 가진다. 비행 중 기체의 지면 이탈 여부(Ground Contact / In-air)를 판별하는 Land Detector 모듈(버전 1.14 이상 기준)은 비행 메인 루프에서 분리되어 독립적인 워커 스레드(Worker Thread) 형태로 실행된다. 본 장에서는 객체 지향적으로 설계된 Land Detector 모듈의 논리적 아키텍처와 RTOS 태스크 스케줄링 메커니즘을 상세히 다룬다.

1. Land Detector 모듈의 객체 지향 논리적 아키텍처(Class Architecture)

PX4의 Land Detector 모듈은 기체의 기구학적 특성(회전익, 고정익, 수직이착륙기)에 따라 지면 접촉 모델링 방정식이 상이하다는 점에 착안하여 다형성(Polymorphism)을 활용한 추상 클래스 구조를 채택하고 있다.

1.1 계층적 클래스 설계

src/modules/land_detector 디렉터리에 위치한 소스 코드는 다음과 같은 상속 관계를 갖는다.

  1. ModuleBase<LandDetector> 및 ModuleParams: PX4 시스템 데몬(Daemon)으로 인스턴스화되고 파라미터 업데이트를 수신하기 위한 최상위 기반 클래스들이다.
  2. LandDetector (Abstract Base Class): 지면 판별에 필요한 공통 uORB 메시지 구독(예: IMU, 센서 검증) 스텁(Stub)과 상태를 발행(Publish)하는 범용 로직이 추상화되어 있다.
  3. 구현체 클래스(Implementation Classes):
  • MulticopterLandDetector: 추력 및 수직 속도 한계치 등을 이용한 멀티로터 전용 알고리즘을 구현한다.
  • FixedwingLandDetector: 익면 하중과 대기 속도(Airspeed)를 활용한 고정익 착륙 알고리즘을 구현한다.
  • VtolLandDetector: VTOL 전환 상태(Transition state)에 따라 위 두 구현체 논리를 동적으로 하이브리드(Hybrid) 적용한다.

2. 실시간 운영체제(RTOS) 내 워커 스레드(Worker Thread) 할당

NuttX 운영체제는 수십 개의 제어 모듈이 한정된 CPU 자원을 경합하는 환경이므로, 컨텍스트 스위칭(Context Switching) 오버헤드를 줄이기 위해 작업 큐(Work Queue) 모델을 적극 채택한다.

2.1 커스텀 작업 큐(WorkQueue) 스케줄링

Land Detector 객체는 인스턴스화된 후 nav_and_controllers라는 시스템 내부 워크 큐(Work Queue)에 배치된다. 이 큐는 기체 자세 제어나 위치 제어만큼 극단적인 실시간성(Hard Real-time, 예: 250Hz~400Hz)을 요구하지는 않으나, 센서 로거(Logger)나 일반 텔레메트리 전송보다는 높은 우선순위로 스케줄링된다.

2.2 상태 판별 주기 (Update Rate)

Land Detector 워커 스레드는 비동기적(Asynchronous) 폴링 방식으로 동작하며, 내부적인 슬립 타이머(Sleep Timer) 또는 센서 uORB 토픽의 업데이트 인터럽트(Event Callback)에 의존하여 깨어난다.
일반적으로 **50Hz (20ms 주기)**를 타깃으로 스케줄링 메커니즘(Run())이 순환한다. 이 50Hz라는 판별 주기는 착륙 순간 발생하는 로터 진동(Vibration chattering) 주파수 대역을 LPF(Low Pass Filter)로 스무딩할 때 가장 안정적인 나이퀴스트-섀넌(Nyquist-Shannon) 샘플링 공간을 제공하는 최적의 수치임이 경험적으로 증명되었다.

3. 워커 스레드 실행 주기의 시퀀스 다이어그램

다음 구조는 nav_and_controllers 워크 큐에 의해 LandDetector::Run() 메서드가 20ms 간격으로 스케줄링되는 제어 흐름을 도식화한 것이다.

sequenceDiagram
    participant OS as NuttX (WorkQueue)
    participant LD as LandDetector (Base)
    participant Child as MulticopterLandDetector
    participant uORB as uORB Sub/Pub

    OS->>LD: 20ms 경과 (Run() 함수 호출)
    activate LD
    LD->>uORB: uORB 센서 토픽 갱신 여부 폴링
    alt 토픽 갱신 안됨
        LD-->>OS: WorkQueue 반환 (Sleep)
    else 로컬 포지션 등 센서 정보 업데이트 됨
        LD->>Child: _get_ground_contact_state() 오버라이드 호출
        activate Child
        Child-->>LD: 접촉 여부 (true/false) 반환
        deactivate Child
        
        LD->>Child: _get_landed_state() 오버라이드 호출
        activate Child
        Child-->>LD: 최종 상태 (In-air / Landed) 반환
        deactivate Child

        LD->>LD: 히스테리시스 (Hysteresis) 필터 검증
        LD->>uORB: vehicle_land_detected 메시지 발행 (Publish)
    end
    deactivate LD

위 로직에서 착안할 수 있는 중요한 아키텍처 철학은, 부모 클래스(LandDetector)가 워커 큐 사이클, uORB 토픽 동기화 관리, 그리고 시계열 히스테리시스 필터링 무결성을 총괄하고, 자식 구현체 클래스가 순수한 공력/기구학 기반 불리언 논리(Boolean Logic) 연산 처리만을 전담한다는 점이다.

4. Ardupilot 등 타 컨트롤러 대비 아키텍처 특장점

Ardupilot은 전통적으로 주기적 루프 배열형(scheduler.tasks)의 단일 모놀리식 메인 스레드에 가깝게 동작한다. 반면, PX4의 데몬 기반 작업 스레드 구조는 다음과 같은 구조적 우위를 제공한다.

  1. 상태 격리성(State Isolation): Land Detector 연산이 딜레이되더라도 위치 제어(Position Control)의 정밀 PID 제어 주기(Rate Loop)를 지연시키거나 간섭(Interference)하지 않는다.
  2. 모듈라리티(Modularity): QGroundControl에서 터미널(MAVLink Console) 접속 시 land_detector status 명령을 하달하면, 현재 스레드의 스케줄링 상태와 누적 실행 통계만 독립적으로 디버깅할 수 있어 트러블슈팅이 매우 용이하다.

5. 결론

PX4 Land Detector 모듈은 NuttX RTOS의 객체 중심(Object-oriented) 워크 큐 스케줄링의 모범적 지향점을 보여준다. 하위 디바이스 드라이버(Device Driver)와 독립되어 약 50Hz 단위로 엄밀하게 스케줄링되는 이 모듈은, 시스템의 부하가 높아지는 이륙 및 회피 기동 시점에서도 착륙 판정 루틴이 무한 대기(Starvation) 상태에 빠지지 않도록 보장한다. 이와 같이 다중 상속과 독립 스레드 배분을 통한 논리적 분할 아키텍처는 틸트로터나 군집 에이전트와 같은 복잡한 커스텀 모델로 확장(Extension)할 때 개발 효율을 획기적으로 향상해 주는 핵심 원동력이다.
