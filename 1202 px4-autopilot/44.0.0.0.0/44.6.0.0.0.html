44.6. 이륙 완료(Takeoff Completion) 상태 판별의 제어 루프 검증 및 GCS 동기화 알고리즘

1. 개요

이륙(Takeoff) 완료 상태 판별은 기체가 비행을 시작하기 위한 준비 단계를 마치고 본격적인 자율 비행(Mission, Offboard 등)으로 전환하기 위한 핵심적인 논리 게이트(Logical Gate)이다. PX4-Autopilot은 이 판별 과정을 내부의 다중 제어 루프(Multi-loop Control) 피드백과 상태 추정기(Estimator)의 텔레메트리 데이터에 의존하여 수행한다. 본 절에서는 이륙 완료 상태가 PX4의 위치 제어기(Position Controller) 및 커맨더(Commander) 내에서 어떻게 검증되는지, 그리고 지상 관제 시스템(GCS, 일례로 QGroundControl)과 어떤 동기화 알고리즘을 통해 관제 상태가 갱신되는지 심층적으로 다룬다.

2. 제어 루프 수준에서의 이륙 완료 조건 검증

이륙 완료를 판단하기 위해서는 기체의 동적 상태가 안정적인 호버링(Hovering) 내지는 목표 상승 고도를 확보했다는 확증이 필요하다. 단순히 목표 고도(Setpoint)에 도달했다는 위치 오차(Position Error)만을 판별하는 것이 아니라, 수직 속도와 가속도의 수렴성을 복합적으로 검증하는 과정이 요구된다.

flowchart TD
    A[이륙 명령 개시] --> B[Z축 상승 궤적 생성]
    B --> C[위치 제어기: 고도 도달 여부 확인]
    C --> D{목표 고도 - 현재 고도 < 오차 한계?}
    D -- Yes --> E{Z축 수직 속도 추정치 \approx 0 ?}
    D -- No --> C
    E -- Yes --> F[이륙 완료 상태(Takeoff Reached) 선언]
    E -- No --> C
    F --> G[비행 상태 천이 및 MAVLink 하트비트 갱신]
  1. 고도 도달 임계값 판별: 확장 칼만 필터(EKF2)로부터 추정된 기체의 수직 고도(Z_{est})가 이륙 목표 고도(Z_{sp})와의 차이가 허용 마진(\epsilon_{z}) 내에 들어왔는지 지속적으로 확인한다. 수식으로 표현하면 \vert Z_{sp} - Z_{est} \vert < \epsilon_{z} 조건이 만족되어야 한다.
  2. 속도 및 가속도 잔차 확인: 위치 제어기는 이륙 기동이 끝물에 다다랐을 때 기체가 관성에 의해 지속적으로 상승하는 이른바 ‘오버슈트(Overshoot)’ 상태를 방지하기 위해 Z축 속도 피드백 검증을 병행한다. 목표 수직 속도가 0으로 수렴해야 하며, 이 조건이 일정 시간(Time Window) 이상 유지될 때 상태 머신은 비로소 이륙 기동이 종료되었음을 승인한다.

3. GCS(QGroundControl)와의 관제 상태 동기화 알고리즘

비행 제어기(Flight Controller, FC)가 내부적으로 이륙을 완료했다 하더라도, 원격지에 있는 관제 시스템이 이를 인지하지 못하면 후속 비행 모드 전환(예: 미션 개시, Offboard 전환)에서 타이밍 불일치(Race Condition)가 발생할 수 있다. PX4-Autopilot과 QGroundControl(QGC v4.3 기준)은 이러한 비동기화를 극복하기 위해 MAVLink 프로토콜의 확장 상태 메시징을 활용한다.

3.1 EXTENDED_SYS_STATE 및 HEARTBEAT 메시지를 통한 동기화

PX4는 내부의 vehicle_status uORB 토픽이 변경될 때마다 이를 기반으로 MAVLink HEARTBEAT 및 EXTENDED_SYS_STATE 메시지를 GCS로 전송한다. EXTENDED_SYS_STATE 메시지 내의 landed_state 필드는 다음과 같이 천이한다:

  • MAV_LANDED_STATE_ON_GROUND (지면 대기/시동 전후)
  • MAV_LANDED_STATE_TAKEOFF (상승 중)
  • MAV_LANDED_STATE_IN_AIR (이륙 완료 후 안정적 체공 중)

QGC는 이 메시지 이벤트를 수신하면 내부의 차량(Vehicle) 관리 매니저 클래스(Vehicle.cc)에 정의된 상태 머신을 갱신하고, 사용자 인터페이스(UI) 상에서 이륙 팝업 상태를 해제한 뒤 비행 중(Flying) 정보 화면으로 전환한다.

3.2 상태 폴링(Polling) 대 이벤트 트리거(Event-triggered) 방식의 최적화

이동 통신망이나 저속의 원격 측정(Telemetry) 환경에서는 대역폭 제약으로 인해 폴링 방식으로 지속적인 상태를 확인하는 것은 비효율적이다. 따라서 PX4-Autopilot은 이륙 전후의 중요한 상태 변이가 발생할 때 상태 동기화 이벤트를 트리거하여 우선순위가 높은 MAVLink 메시지를 즉시 버퍼로 푸시(Push)하는 최적화 알고리즘을 채택하고 있다.

4. Ardupilot 대비 동기화 아키텍처 비교

  • PX4-Autopilot: 다중 상태 플래그(nav_state, vehicle_status, landed_state)가 uORB라는 명시적인 Publish/Subscribe 미들웨어를 거쳐 MAVLink로 매핑된다. 따라서 GCS는 내비게이션 상태(Takeoff -> Loiter/Auto)와 착륙 상태(Landed -> In Air)를 이원화하여 추적함으로써 이탈, 돌풍 등 예기치 않은 제어 이상 상황을 매우 미세하게 캡처할 수 있다.
  • Ardupilot: 모드 변경(Guided -> Auto 등)을 위주로 이륙 완료 상태를 묵시적으로 동기화하는 성향이 강하다. 미션 플래너(Mission Planner) 등의 GCS는 Ardupilot으로부터 수신한 비행 모드와 목표 고도 대비 현재 고도 데이터를 취합하여 자체 소프트웨어 로직으로 이륙 완료(Takeoff Phase End)를 추론(Deduction)하는 방식에 가깝다. 반면 PX4 생태계는 기체 측에서 이륙 완료라는 뚜렷한 논리적 이벤트 플래그를 내려준다는 점에서 중앙 집중형 관제(Centralized Fleet Management)에 더 친화적인 구조를 갖는다.

5. 요약 및 결론

이륙 완료 상태 판별은 복잡한 속도/고도 피드백 루프의 안정성 검증과, GCS와의 지연 제로에 가까운 통신 동기화 단위가 결합된 고차원적 관제 과정이다. 이 체계적인 이륙 검증 알고리즘은 단일 기체의 이륙 안정성뿐만 아니라 향후 ROS2 uXRCE-DDS 브리지를 이용한 수십 대 규모의 군집 비행 제어기(Swarm Flight Controller) 통합 관리 목적의 견고한 기반 구조로 작용한다.
