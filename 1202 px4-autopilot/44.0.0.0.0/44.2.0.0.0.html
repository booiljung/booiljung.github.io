44.2. MAVLink 프로토콜 기반 이륙 명령(Takeoff Command)의 직렬화(Serialization) 및 구조 분석

지상 관제 시스템(GCS)과 무인항공기(UAS) 간의 오프보드 제어(Offboard Control) 및 임무 전달 과정에서 가장 널리 사용되는 통신 규격은 MAVLink 프로토콜이다. 이륙(Takeoff) 명령은 기체가 지면을 박차고 지정된 고도까지 비행 모드를 자율적으로 전환하도록 지시하는 매우 중요한 명령어이다. 본 절에서는 PX4-Autopilot 환경에서 이륙 명령이 어떠한 MAVLink 구조로 패키징(Packaging)되며, 네트워크를 통해 직렬화(Serialization)되어 전송되는지 심층적으로 구조 모델링을 진행한다. 이를 통해 GCS 개발자나 ROS 2 연동 시스템 설계자가 패킷 레벨의 분석을 수행할 수 있도록 돕는다.

1. MAVLink 프로토콜의 직렬화(Serialization) 메커니즘

MAVLink는 대역폭이 제한된 무선 통신(Telemetry) 환경에 최적화된 초경량 헤더 기반 이진(Binary) 직렬화 프로토콜이다.

1.1 직렬화 프레임 구조 (MAVLink v2.0)

MAVLink v2.0 버전을 기준으로 메시지 프레임은 가변 길이(Minimum 12바이트 ~ Maximum 280바이트)로 구성된다. 메모리 직렬화 구조는 아래와 같다.

  • Magic Marker (1 Byte): 프레임의 시작을 알리는 식별자 (v2.0 기준 0xFD).
  • Payload Length (1 Byte): 페이로드의 바이트 크기(0~255).
  • Incompatibility/Compatibility Flags (2 Bytes): 서명 처리 및 라우팅 호환성 식별 플래그.
  • Sequence (1 Byte): 패킷 누락 감지를 위한 순차 번호 (0~255).
  • System ID / Component ID (2 Bytes): 발신 기체(System ID) 및 발신 컴포넌트(예: Autopilot, Gimbal 등)의 고유 번호.
  • Message ID (3 Bytes): 전달하고자 하는 메시지의 종류(예: MAV_CMD_NAV_TAKEOFF). 24비트로 확장되어 1,600만 개의 메시지를 정의할 수 있다.
  • Payload (0~255 Bytes): 실제 데이터가 담기는 적재 공간. 리틀 엔디안(Little-Endian) 방식으로 정렬되어 전송된다.
  • Checksum (2 Bytes): ITU X.25 방식의 CRC(Cyclic Redundancy Check) 검사 코드로 무결성을 보증한다.

2. 이륙 명령(Takeoff Command) 메커니즘

이륙 명령 전송을 위해 MAVLink에서는 주로 COMMAND_LONG (Message ID: #76) 또는 COMMAND_INT (Message ID: #73) 메시지 페이로드를 사용하며, 그 내부에 Command ID로서 MAV_CMD_NAV_TAKEOFF (Enum Value: 22)를 담아 전송한다.

2.1 COMMAND_LONG 메시지의 구조 분석

가장 고전적이고 보편적으로 사용되는 COMMAND_LONG 메시지는 7개의 부동 소수점(Float) 매개변수(Param)를 가진다. 이륙 명령에서 각 필드의 의미는 다음과 같이 매핑된다.

파라미터(Parameter)자료형설명 및 용도 (MAV_CMD_NAV_TAKEOFF 기준)
Param 1FloatMinimum pitch: 이륙 시 목표로 하는 최소 피치(Pitch) 상승각도. (회전익에서는 주로 무시됨)
Param 2Float비어있음(Reserved/Empty)
Param 3Float비어있음
Param 4FloatYaw angle: 목표 요(Yaw) 앵글. (지원하는 기체의 경우) \vert NaN 사용 시 기존 헤딩 유지
Param 5FloatLatitude: 목표 위도 (Global 좌표계 기준).
Param 6FloatLongitude: 목표 경도 (Global 좌표계 기준).
Param 7FloatAltitude: 목표 비행 고도 (기본적으로 AMSL, 즉 평균 해수면 기준 척도).

참고: 파라미터 5, 6, 7의 경우 위치가 비어있으면(예: NaN 값 입력), PX4는 현재 위치를 수직 이륙 허용 좌표로 대체(Overwrite) 처리하여 제자리에서 이륙하게 된다.

2.2 Ardupilot과 PX4의 이륙 명령 좌표 처리 방식 차이

Ardupilot과 PX4-Autopilot은 Global 좌표와 상대 고도를 다루는 방식에 있어 구조적 차이가 있다.

  • PX4-Autopilot: COMMAND_LONG 전송 시 글로벌 좌표계가 비어있다면(NaN) 현재 위치를 기준으로 곧바로 참조 모델을 갱신한다. 또한 고도 프레임(Altitude Frame) 처리를 명확히 하기 위해 COMMAND_INT 메시지(고정 소수점을 사용하여 정밀도 향상)의 사용을 더욱 권장하는 방향으로 진화하고 있다.
  • Ardupilot: Takeoff 명령을 수행하기 위해서는 기본적으로 Home 위치가 명확해야 하며, Copter 버전에서는 위도와 경도 값(Param 5, 6)이 보통 무시(Ignored)되고 수직 이륙 고도만이 우선적으로 참조되는 특징을 지닌다.

3. 직렬화 과정의 데이터 패킹(Data Packing) 아키텍처

위와 같은 명령어를 전송할 때 GCS의 내부 엔진(예컨대 QGroundControl의 C++ 백엔드)은 구조체를 선언하고 이를 바이트 스트림(Byte Stream)으로 변환하는 직렬화 알고리즘을 수행한다.

graph LR
    A[사용자 UI Takeoff 버튼 클릭] --> B[QGC Vehicle Object]
    B --> C{COMMAND_LONG 구조체 생성}
    C -->|Param 1~7 할당| D[MAV_CMD_NAV_TAKEOFF 매핑]
    D --> E[mavlink_msg_command_long_pack()]
    E --> F[Little-Endian 바이트 배열 직렬화]
    F --> G[CRC Checksum 추가]
    G --> H[Telemetry (UART/UDP) 전송]
    H --> I[PX4 mavlink_receiver 모듈 수신]

이 과정에서 가장 중요한 점은 7개의 실수 파라미터가 네이티브(Native) 메모리 상에서 구조체(Struct) 오프셋(Offset)에 맞춰 빈틈없이 채워지는(Packed) 방식이다. MAVLink의 헤더 생성기는 각 변수의 크기를 기준으로 가장 큰 변수(예: uint64_t나 double)부터 먼저 배치하여 메모리 접근 효율을 높이고 구조체 패딩(Padding)을 원천 차단하는 최적화를 수행한다. 이 직렬화 메커니즘 덕분에, 매우 낮은 컴퓨팅 자원을 가진 보드에서도 인터럽트 없이 이중 루프(Cascaded Loop) 처리를 원활히 수행할 수 있다.
