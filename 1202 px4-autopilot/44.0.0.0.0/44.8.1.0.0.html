44.8.1. `MAV_CMD_NAV_TAKEOFF` 처리 시 Ardupilot(Copter/Plane)과 PX4의 내부 파서(Parser) 구문 트리(Syntax Tree) 및 메시지 라우팅 토폴로지(Routing Topology) 차이

GCS(지상 관제 시스템)에서 송신된 MAV_CMD_NAV_TAKEOFF(MAVLink 명령 번호 22) 패킷이 물리적인 비행 제어기(FC) 하드웨어의 통신 포트(UART, USB 등)에 수신된 이후, 이를 파싱(Parsing)하고 실제 비행 동력학 제어 로직에 전달하는 라우팅 토폴로지(Routing Topology)는 PX4와 Ardupilot 간에 뚜렷한 아키텍처적 차이를 보인다.

본 절에서는 두 프레임워크가 MAVLink 바이트 스트림을 해석하여 내부 구문 트리(Syntax Tree)를 구성하는 방법론, 그리고 이륙 로직을 기체 형태(멀티콥터, 고정익 등)에 맞게 배분하는 메시지 라우팅 과정을 소스 코드 레벨에서 비교 분석한다.

1. PX4-Autopilot의 메시지 라우팅 토폴로지: 구조체 기반 uORB 퍼블리싱

PX4 프레임워크는 MAVLink 메시지를 처리할 때 특정 모듈들 간의 강한 결합(Tight Coupling)을 피하고, 철저히 uORB(Micro Object Request Broker) 메시징 버스를 위주로 한 비동기 이벤트 릴레이(Relay) 방식을 취한다.

1.1 내부 파서(Parser) 구문 트리: mavlink_receiver.cpp

PX4의 시작점은 mavlink 모듈 내부의 MavlinkReceiver::handle_message_command_long 함수이다.

  1. 디코딩(Decoding) 및 역직렬화: MAVLink 패킷이 들어오면 mavlink_msg_command_long_decode() 함수를 통해 바이트 스트림이 C 구조체로 역직렬화(Deserialization)된다.
  2. 명령어 치환(Command Translation): 추출된 command(예: 22번 MAV_CMD_NAV_TAKEOFF)와 7개의 param 배열은 PX4의 내부 추상화 구조체인 vehicle_command_s로 1:1 매핑(Mapping)된다.
  3. uORB 퍼블리싱 (Routing 시작): mavlink_receiver는 이 명령이 누구에 의해 수행될지(Copter인지, Plane인지) 판단하지 않는다. 단순히 구문 분석된 vehicle_command_s 인스턴스를 uORB 버스에 퍼블리시(Publish)한다. 이를 퍼블리시-서브스크라이브 토폴로지(Pub-Sub Topology) 라 한다.

1.2 메시지 라우팅과 Commander의 역할

uORB 버스에 올라온 vehicle_command 구조체는 이 시스템의 전역 유한 상태 관리 모듈인 Commander가 서브스크라이브(Subscribe)한다.

  • Commander 내부의 command_handler.cpp에서 거대한 switch-case 구문 트리가 전개된다.
  • case vehicle_command_s::VEHICLE_CMD_NAV_TAKEOFF: 분기점에서 기체의 현재 Armed 상태와 비행 모드(Nav State)를 검증한다.
  • 검증에 성공하면 Commander는 다시 내부 플래그를 변경하고, 최종적으로 이륙 이벤트를 관장하는 Navigator 모듈이 이를 인지하여, 고정익(Plane) 위치 제어 모듈(fw_pos_control_l1) 또는 멀티콥터 위치 제어 모듈(mc_pos_control)에 목표 궤적(Setpoint) 구성을 지시한다.

2. Ardupilot의 메시지 라우팅 토폴로지: 객체 지향적 메소드 디스패치(Method Dispatch)

반면, Ardupilot은 C++의 다형성(Polymorphism)과 상속(Inheritance) 메커니즘을 적극 활용하여 메시지를 라우팅한다. MAVLink 패킷은 거대한 싱글톤 방식의 GCS 클래스에서 파싱된 즉시, 현재 활성화된 비행 역학(Vehicle Dynamics) 객체로 직접 하향 전파(Top-down Dispatch)된다.

2.1 내부 파서(Parser) 구문 트리: GCS_MAVLink의 거대 분기문

Ardupilot 메인 코드베이스의 GCS_MAVLink::handle_command_long() 함수는 MAVLink 패킷을 받아 파싱 트리를 전개한다.

  • 패킷은 역직렬화된 이후, 프레임워크 전반에 공유되는 거대한 switch(packet.command) 구문 트리를 통과한다.
  • case MAV_CMD_NAV_TAKEOFF: 블록에 도달하면, Ardupilot은 uORB와 같이 버스에 데이터를 던지는 것이 아니라, 현재 제어 주도권을 가진 기체 최상위 인터페이스(vehicle 포인터) 또는 비행 모드 객체 멤버 함수를 즉시 호출한다.

2.2 메세지 라우팅: Copter와 Plane의 다형성(Polymorphism)

Ardupilot은 이륙 명령어 처리를 Copter와 Plane (그리고 Rover 등) 서브클래스에서 재정의(Override)한 가상 함수(Virtual Function)로 라우팅한다.

  • Copter (멀티콥터): GCS 파서는 Copter의 Mode_Guided 또는 Mode_Auto 객체에 존재하는 do_user_takeoff(altitude)를 직접 서브루틴 호출(Synchronous Call)한다. 이 함수는 Z축(수직) 제어기인 AC_POSCONTROL 인터페이스에 목표 이륙 고도를 넘겨줌으로써 수직 상승 절차를 개시한다.
  • Plane (고정익): 고정익은 수직 이륙이 불가능하므로 이륙 런(Takeoff Run)이 필요하다. GCS 파서가 Plane 로직을 가리키고 있을 경우, Plane의 Mode_Auto는 활주로 이륙을 위한 스로틀 개방(Throttle Max) 및 일정 피치각(Pitch Angle) 유지를 위한 AP_TECS (Total Energy Control System) 라이브러리로 제어 벡터를 전달한다. 활주로 조향 제어를 위해 이륙 명령 하나가 스로틀(Throttle), 피치(Pitch), 러더(Rudder) 제어 레이어에 동시에 분산 라우팅되는 방사형(Radial) 라우팅 형태를 띤다.

두 시스템이 이륙 파라미터(고도, 피치 등)를 구문 분석하여 실제 액추에이터 신호 발생 레이어까지 이동시키는 토폴로지는 다음과 같은 트레이드오프(Trade-off)를 가진다.

5. 결론

PX4와 Ardupilot 모두 MAV_CMD_NAV_TAKEOFF라는 동일한 ID(22번)를 통해 이륙 명령을 받는다. 하지만 PX4는 파서에서 읽어들인 바이트 스트림을 vehicle_command 구조체로 추상화하여 미들웨어(uORB)에 뿌리는 ’상위-지향형 느슨한 결합(Loose Coupling)’을 채택하고 있다. 이 구조는 안정성과 모듈 독립성은 뛰어나지만 분산 구조 특유의 상태 롤백(Rollback) 추적 난이도가 높다.
반면 Ardupilot은 GCS 수신부 파서가 C++ 다형성 객체인 Copter/Plane 모드 클래스 메소드를 직접 찔러주는(Dispatching) 방식을 채택하여, 실행 흐름(Execution Flow)의 직관성과 디버깅 편의성을 극대화하였다. GCS 관제 프로그램 개발자는 이와 같이 두 프레임워크가 응답 ACK를 생성하기까지 거치는 구문 트리 탐색과 메모리 릴레이 횟수 차이를 인지하고 명령 전송 지연시간(Latency)을 설계해야 한다.
