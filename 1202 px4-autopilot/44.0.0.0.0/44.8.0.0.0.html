44.8. PX4와 Ardupilot의 이륙 관제 아키텍처 및 명령어 처리 체계의 비교 분석

오픈소스 기반의 무인항공기 생태계를 양분하고 있는 PX4-Autopilot과 Ardupilot 프레임워크는 각기 다른 철학과 아키텍처를 바탕으로 비행 시스템을 제어한다. GCS(지상 관제 시스템)에서 이륙(Takeoff) 명령을 하달하여 기체가 대기 상태(Hovering)에 도달하기까지의 과정은 두 펌웨어가 내부적으로 메시지를 어떻게 파싱(Parsing)하고, 상태 머신(State Machine)을 어떻게 갱신하며, 예외 상황(Exception)에 어떻게 대처하는지를 극명하게 보여주는 대표적인 사례이다.

본 절에서는 MAVLink 프로토콜을 통한 이륙 명령 처리의 관점에서 PX4(v1.14.0 기준)와 Ardupilot(Copter 4.3 기준)의 아키텍처와 명령어 처리 체계의 근본적인 차이를 비교 분석한다.

1. MAVLink 이륙 명령어(MAV_CMD_NAV_TAKEOFF) 패킷 처리 동선

GCS(예: QGroundControl, Mission Planner)에서 MAV_CMD_NAV_TAKEOFF 명령을 전송했을 때, 이를 수신하고 해석하여 액추에이터(Actuator) 출력까지 도달하는 일련의 파이프라인(Pipeline)은 크게 다르다.

1.1 PX4-Autopilot의 처리 동선: 마이크로 서비스 모델

PX4는 철저하게 멀티스레딩(Multi-threading) 기반의 퍼블리시-서브스크라이브(Publish-Subscribe) 미들웨어인 uORB 메시징 버스 위에서 구동된다.

  1. 커맨더(Commander) 모듈: MAVLink 인터페이스 모듈에서 명령을 수신하면, 커맨더(Commander)가 이를 받아 기체의 Arming 상태, 현재 비행 모드(Nav State), 배터리 및 센서 상태 등 전역 상태(Global State)를 가장 먼저 평가한다.
  2. 내비게이터(Navigator) 추상화: 명령이 타당(Valid)하다고 판별되면 uORB를 통해 vehicle_command 토픽이 발행되며, 이륙 전용 논리 객체인 Navigator 모듈 내부의 Takeoff 임무 항목(Mission Item)이 활성화된다.
  3. 위치 제어기(Position Control)와 비행 작업(Flight Task): 이후 제어 레벨(mc_pos_control)에 존재하는 FlightTaskAutoMapper를 거쳐 FlightTaskAutoTakeoff 객체가 생성된다. 이 객체는 지리적 목표 고도 성분을 역학적 궤적(Trajectory Setpoint)으로 변환하여 최종 자세 제어기(Attitude Controller)로 보낸다.

1.2 Ardupilot의 처리 동선: 다형성 및 모드 객체화 (Mode Objectification)

Ardupilot 역시 멀티스레드와 실시간 운영체제(ChibiOS)를 활용하지만, 전통적으로 모놀리식(Monolithic) 메인 제어 루프(Main Scheduler Loop) 성향과 강하게 결합된 객체 지향적(Object-Oriented) 비행 모드 상속 플로우를 갖는다.

  1. GCS_MAVLink 모듈: 라우터 역할을 하는 이 모듈에서 MAVLink 커맨드를 파싱한다.
  2. 비행 모드 클래스 메서드 직접 호출: vehicle 객체(Copter 등)는 현재 활성화된 비행 모드(예: Mode_Guided 또는 Mode_Auto)의 do_user_takeoff와 같은 가상 함수(Virtual Function)를 직접 호출한다.
  3. 위치 제어 인터페이스(AC_PosControl): 각 모드(Mode) 객체가 이륙 목표(Target) 지점을 수립하면 AC_PosControl 라이브러리의 init_takeoff() 함수를 호출한다. 이때 궤적 생성기(Trajectory Generator, AC_WPNav)가 고도 변경 프로필을 그리며 모터 출력을 점진적으로 상승시킨다.

2. 상태 머신(State Machine)과 모드 검증 기준의 차이

명령이 이행되기 직전, 기체는 **이륙할 준비가 되었는가?**를 내부 상태 머신을 바탕으로 검증한다.

2.1 PX4: 엄격한 선결 조건(Pre-condition)과 전역 상태 검증

PX4는 커맨더(Commander) 중심의 트리 구조 기반 전역 유한 상태 기계(FSM)를 사용한다. 이륙 명령을 수신했을 때 기체의 모드가 Offboard, Mission, Hold 등 자율(Auto) 기반의 모드 중에 어느 상태에 있는지 검증함과 동시에 Local/Global Position 유효성 플래그가 엄격하게 체크된다.

  • 만약 GCS가 Takeoff 명령을 내리기 이전에 PX4를 올바른 자율 비행 상태(Nav State)로 전환하지 않았다면 제어 천이 실패로 즉각 NACK(MAV_RESULT_DENIED 등)를 반환한다.
  • PX4는 상태 이상 발견 시 ’거부(Deny)’로 단호하게 응답하는 편이다.

2.2 Ardupilot: 유연한 절차(Procedure)적 전환

Ardupilot은 이륙 명령을 수신하면 내부 로직에서 상태(비행 모드) 변경을 알아서(Implicity) 수행하는 경우가 많다.

  • 예를 들어, Guided 모드에서 MAV_CMD_NAV_TAKEOFF 명령을 받으면, 기체가 이륙 조건을 만족(Pre-arm Check 통과 및 Armed 상태)할 경우 암묵적으로 내부 위치 제어(PosHold/Loiter와 유사한 알고리즘) 루프를 가동시키며 이륙을 강행한다.
  • GCS 관점에서는 모드 전환 오류에 대한 관용성(Tolerance)이 높다고 볼 수 있다.

5. 결론

GCS 엔진 개발자는 목표로 하는 기체가 PX4-Autopilot인지 Ardupilot인지에 따라 이륙 명령에 대한 관제 논리 구조를 독립적으로 구성해야 한다. MAVLink라는 공통 언어를 사용함에도 불구하고, 내부 파서 구문 트리 구조와 제어기(Controller)로 매핑(Mapping)되는 라우팅 양상이 완전히 다르며, 이는 관제 소프트웨어가 통신 상태 이상(NACK)과 이벤트 천이(State Flip)를 감시하고 보완하는 트랜잭션 방어 로직 설계에 매우 중대한 영향을 미친다. 이어지는 하위 절에서는 두 펌웨어의 구문 파서, 호버링(Hovering) 시 적분기 유지, 그리고 객체 지향적 관제 레이어 구축 패턴에 대해 계층별로 면밀히 비교하도록 한다.
