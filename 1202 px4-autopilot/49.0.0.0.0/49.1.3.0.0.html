49.1.3 자율 비행(Autonomous Flight) 고도화를 위한 외부 제어 루프(External Control Loop)의 임무 수행 토폴로지

PX4-Autopilot은 태생적으로 실시간 운영체제(NuttX RTOS) 기반의 하위 레벨 비행 제어기(Low-level Flight Controller) 역할에 최적화되어 있다. 단일 컴퓨터 내에서 복잡한 임무 계획(Mission Planning), 장애물 회피, 컴퓨터 비전(Computer Vision), 인공지능 알고리즘 연산을 모두 처리하기에는 마이크로컨트롤러(MCU)의 연산 한계(Compute Bound)와 실시간성 충돌 문제가 발생한다. 따라서 고도화된 자율 비행 시스템을 구현하기 위해서는 로우 레벨을 관장하는 PX4-Autopilot과 하이 레벨을 관장하는 외부 임무 컴퓨터(Mission Computer, 예: ROS2 노드 구동 환경) 간의 외부 제어 루프(External Control Loop) 토폴로지(Topology) 설계가 필수적이다. 본 섹션에서는 이러한 외부 제어 아키텍처의 사상과 임무 연결 토폴로지를 상세히 분석한다.

1. 외부 제어 루프(External Control Loop)의 사상

외부 제어 루프의 철학은 “인지(Perception), 계획(Planning), 제어 로직(Control Logic)“을 외부 컴퓨터에 위임(Delegate)하고, PX4는 “실행(Execution) 및 기초 생존(Basic Survival)“에만 집중하는 것이다.

  • 임무 컴퓨터(Mission Computer, High-level): 라이다(LiDAR), 심도 카메라(Depth Camera), AI 가속기 등을 활용하여 슬램(SLAM), 전역/지역 경로 계획(Global/Local Path Planning), 객체 추적, 그리고 스웜(Swarm) 통제 등 연산 집약적인 작업을 처리한다. 산출된 결과는 설정값(Setpoint)으로 변환된다.
  • PX4-Autopilot (Low-level): 오프보드(Offboard) 모드 상태에서 외부로부터 설정값 스트림(예: 위치, 속도, 가속도, 자세, 추력 등)을 지속적으로 수신하여 자체적인 다단 PID 루프(Cascaded PID Loop)를 통해 모터와 액추에이터를 정밀 제어한다.

2. 임무 수행의 3계층(3-Tier) 토폴로지 아키텍처

고도화된 자율 비행 에이전트를 위한 시스템 토폴로지는 보통 센서 데이터의 흐름과 통제 권한에 따라 세 가지 계층으로 설계된다. ROS2와 uXRCE-DDS 브리지를 활용한 현대적인 PX4 토폴로지는 다음과 같다.

2.1 Tier 1: 인지와 전역 계획 (Perception & Global Planning)

가장 바깥쪽 루프(Outer-most Loop)로, 딥러닝 기반의 객체 검출(Object Detection) 노드나 전역 지도를 활용한 A* (A-Star) 등 경로 탐색 노드가 위치한다.

  • 연산 주기가 비교적 느리다(1~10Hz).
  • 최종 종착점(Destination)과 통과해야 할 거시적인 경유지(Macro Waypoints)를 생성한다.

2.2 Tier 2: 지역 계획 및 궤적 생성 (Local Planning & Trajectory Generation)

중간 계층 루프(Middle Loop)로, 실시간 장애물 회피(Obstacle Avoidance) 알고리즘(예: VFH, DWA)이나 부드러운 스플라인 궤적(Spline Trajectory)을 생성하는 노드가 담당한다.

  • 연산 주기가 중간 정도이다(10~50Hz).
  • Tier 1에서 생성한 경로를 따라가되, 갑작스럽게 튀어나온 동적 장애물을 회피할 수 있는 미시적인 회피 궤적(Micro Evasive Trajectory)을 연산하여 부드러운 위치/속도 프로파일(Jerk-limited Profile)을 생성한다.

2.3 Tier 3: 실시간 설정값 전송 (Real-time Setpoint Streaming)

가장 안쪽 루프(Inner-most Loop)이자 PX4의 직접적인 인터페이스 접점이다.

  • 연산(데이터 펌핑) 주기가 매우 빠르다(50Hz 이상 권장).
  • ROS2의 TrajectorySetpoint 메시지 퍼블리셔(Publisher)가 이 역할을 수행한다. 이 계층은 상위 계층에서 만들어진 연속적인 궤적 함수 위에서, 현재 시점 t에서의 정확한 목표 위치, 속도, 가속도, 시선 방향(Yaw)을 샘플링하여 uXRCE-DDS 브리지를 통해 PX4의 커맨더로 끊임없이 밀어 넣는다.

3. 오프보드 제어 토폴로지의 안전장치 (Failsafe Topology)

외부 컴퓨터가 기체의 통제권을 완전히 쥐게 되므로, 임무 컴퓨터의 OS 다운, 프로세스 크래시(Crash), 혹은 일시적인 연산 지연 시 기체의 안전을 담보하는 페일세이프 토폴로지 설계가 절대적으로 중요하다.

  1. 하트비트 신호 기반 생존 검증(Heartbeat & Timeout):
    PX4는 파라미터 COM_OF_LOSS_T를 통해 오프보드 신호의 유실을 감지한다. 외부 컴퓨터의 TrajectorySetpoint 스트림이 일정 시간(예: 0.5초) 이상 끊기면, PX4 커맨더는 외부 컴퓨터가 “죽었다“고 판단하고 즉시 오프보드 통제권을 박탈한다.
  2. 다중 강등 계층(Fallback Hierarchy):
    오프보드 제어 유실 시, 시스템은 사전 설정(Parameter)된 페일세이프 트리거에 따라 다음 중 하나의 상태로 강등(Fallback)된다.
  • Hold 모드 전환: 그 자리에 정지(Hovering)하여 상위의 명령 재개를 기다림.
  • Return 모드 전환: 장애물 회피 루프가 작동하지 않더라도 기본 고도로 상승한 후 이륙 지점으로 복귀.
  • Land 모드 전환: 배터리가 부족하거나 GPS 신뢰성마저 의심될 때 즉시 수직 하강.

ROS2 애플리케이션 개발자는 이 PX4 내부의 페일세이프 발동 조건을 명확히 인지하고, 프로세스 크래시가 났을 때 노드가 자동으로 재시작되도록 systemd 구성을 하거나 쿠버네티스(Kubernetes/K3s) 등 컨테이너 오케스트레이션을 활용하여 컴포넌트 생존 토폴로지를 구성해야 한다.

4. 제어 지령 믹싱 토폴로지 (Position-Velocity-Acceleration Setpoints)

PX4의 TrajectorySetpoint 메시지는 매우 유연하여 여러 도함수 차원의 제어 지령을 믹싱하여 전달할 수 있다.

  • 위치만 제어(Position only): 지정한 좌표로 이동. 궤적 생성은 PX4 내부의 Jerk-limited 궤적 생성기(Flight Task)가 알아서 처리.
  • 속도만 제어(Velocity only): 전방향 카메라 기반의 추적(Tracking) 어플리케이션에서 주로 사용. 목표물이 도망가는 속도만큼 기체의 목표 속도를 지정.
  • 위치 + 속도 피드포워드 (Position + Velocity Feed-forward): 가장 권장되는 형태로, 외부 컴퓨터가 스플라인 궤적을 만들고 특정 시점의 위치와 그 접선 속도(Feed-forward)를 동시에 전달하여 PID 추종 오차(Tracking Error)를 최소화.

5. ArduPilot과의 아키텍처 토폴로지 차이 (Ardupilot 대비 차이)

ArduPilot 체계에서 이와 유사한 개념을 **가이드 모드(Guided Mode)**라고 부른다.

  • 데이터 브리지(Bridge): ArduPilot은 MAVLink와 MAVROS를 이용한 통신 토폴로지가 지배적이나, PX4는 ROS2와의 네이티브한 메모리 통합 토폴로지를 지향하며 uXRCE-DDS (Micro-ROS) 프로토콜을 백본으로 채택하여 데이터 퍼블리시 주기와 오버헤드 측면에서 압도적인 실시간성을 자랑한다.
  • 제어 엄격성: 앞서 다루었듯 PX4의 오프보드 제어는 지속적인 주기 스트림(Continuous Stream)을 강제하는 토폴로지이다. 반면 ArduPilot의 가이드 모드는 단발성 목적지 명령(Single Goto Command)을 던져 놓고 외부 컴퓨터가 대기하는 형태의 느슨한(Loose) 루프도 비교적 관대하게 수용하는 편이다.

6. 결론

자율 비행의 고도화를 달성하기 위한 외부 제어 루프의 토폴로지는 단순한 지령 하달을 넘어 네트워크, 분산 처리, 실시간 시스템의 신뢰성에 대한 깊은 고민을 담고 있다. ROS2 기반의 임무 컴퓨터 아키텍처를 도입하여 복잡한 인지와 계획을 오프로드(Offload) 하되, 비행의 생리학적 안전 메커니즘은 PX4의 파라미터 구성과 강력한 결합을 유지하도록 설계해야만 상용 수준의 드론 시스템을 완성할 수 있다.
