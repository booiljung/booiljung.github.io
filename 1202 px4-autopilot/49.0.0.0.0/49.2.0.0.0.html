49.2 유한 상태 머신(Finite State Machine) 기반 Offboard 모드 천이(Transition) 결정론

현대의 무인항공기(UAV) 제어 시스템 로직은 예측 불가능한 비행 환경과 비동기식으로 입력되는 통신 지령 속에서 기체의 안전과 임무 수행의 결함을 방지하기 위해 엄격한 상태 관리 구조를 취한다. PX4-Autopilot은 이러한 상태 관리를 위해 수학적으로 완결성이 증명된 유한 상태 머신(FSM, Finite State Machine) 모델을 채택하고 있다. 특히 기체 외부의 컴퓨터(예: ROS 2 컴패니언 컴퓨터 또는 지상 관제 시스템)가 직접 비행 궤적을 제어하는 오프보드(Offboard) 모드로의 천이(Transition)는 가장 위험도가 높은 동작 중 하나이므로, FSM의 엄격한 유효성 검증을 거쳐야만 모드 전환이 승인된다.

본 절에서는 PX4-Autopilot 아키텍처(펌웨어 1.14 기준) 내에서 Offboard 비행 모드로 진입하기 위한 조건, 커맨더(Commander) 모듈의 상태 천이 사양, MAVLink 제어 이면의 결정론(Determinism), 그리고 Ardupilot과의 철학적 차이에 대해 심층적으로 분석한다.

1. 유한 상태 머신(FSM) 아키텍처의 제어 공학적 의의

유한 상태 머신은 시스템이 가질 수 있는 “상태(State)“의 개수가 유한(Finite)하며, 특정 이벤트(Event)나 조건(Condition)이 충족되었을 때만 정의된 경로를 따라 다른 상태로 “천이(Transition)“하는 수학적 연산 모델이다.

항공우주 소프트웨어공학에서 FSM은 다음과 같은 장점을 제공한다.

  • 결정론 보장(Determinism): 예상치 못한 이벤트(예: 통신 두절, 센서 오류)가 발생했을 때 소프트웨어가 미정의된(Undefined) 동작을 수행하는 것을 물리적으로 차단한다.
  • 추적성(Traceability): 비행 상태의 변화 로그를 통해 사고 조사(Crash Analysis) 시 어느 시점에 어떤 조건 누락으로 페일세이프(Failsafe)가 발동했는지 명확하게 증명할 수 있다.

PX4의 비행 모드 관리자 모듈인 커맨더(commander)는 메인 FSM을 호스팅하며, 오프보드 모드 전환 역시 단순한 스위치 조작이 아닌 “다중 선행 조건(Pre-conditions)이 충족된 유한 상태로의 이동“으로 철저하게 규정된다.

2. Offboard 모드로의 천이 결정론: 선행 조건 및 검증

Offboard 모드는 기체의 메인 비행 제어기(Flight Controller)가 자체 생성하는 궤적 대신, 외부 연결 장치에서 고주기로 전송하는 위치, 속도, 자세 또는 추력 지령(Setpoint)을 신뢰하고 따르는 상태를 의미한다.

PX4의 FSM은 Offboard 모드로 천이하기 전, 그리고 천이된 상태를 유지하기 위해 다음의 엄격한 논리적 규칙(결정론)을 강제한다.

2.1 천이 선행 조건 (Pre-conditions for Transition)

  1. 사전 지령 수신 (Setpoint Pre-loading):
  • 오프보드 모드로 전환하기 전, 최소 0.5초 동안 연속적으로 MAVLink 스웜 제어 메시지(SET_POSITION_TARGET_LOCAL_NED 등)나 ROS 2 uXRCE-DDS 오프보드 타겟 토픽이 수신되어야 한다.
  • 결정론적 이유: 천이 직후 추종할 목표가 없어 기체가 표류(Drifting)하거나 급강하하는 현상을 방지하기 위해 제어 버퍼를 사전에 채우도록 강제하는 것이다.
  1. 동적 위치 추정기(Estimator)의 절대적 안정성:
  • 확장 칼만 필터(EKF2) 등의 항법 추정기가 글로벌 포지셔닝(GPS 등) 혹은 로컬 포지셔닝(VIO, Optical Flow 등) 상태에 대해 유효성 플래그(Validity Flag)를 “True“로 유지하고 있어야 한다. 속도와 위치에 대한 내부 공분산(Covariance) 허용치를 초과하면 천이 요청은 묵살(Reject)된다.

2.2 유지 및 이탈 조건 (Hold & Fallback Conditions)

Offboard 상태로 진입한 후, FSM은 외부 지령의 “심장 박동(Heartbeat)“을 지속적으로 모니터링한다.

  • 지령 업데이트 주기가 약 0.5초(타임아웃 파라미터 COM_OF_LOSS_T 등에 의존)를 초과하여 지연되면 FSM은 타임아웃 예외(Exception) 이벤트를 발생시킨다.
  • 이 이벤트는 상태 머신의 천이 테이블(Transition Table)에 따라 즉각 오프보드 상태를 취소하고, 스스로 제어 권한을 회수하여 안전한 상태(Hold 모드 또는 Position 모드)로 강제 회귀(Fallback)하는 논리 연산을 구동한다.

3. PX4 메인 커맨더 모듈 내 FSM 구조 변환 과정

오프보드 천이를 승인하는 commander 모듈 내부의 동작 흐름은 다음과 같이 도식화할 수 있다.

stateDiagram-v2
    [*] --> Manual_Hold_State
    
    Manual_Hold_State --> Setpoint_Check_State : Mode Switch Request (Offboard)
    note right of Setpoint_Check_State: Check if setpoint flux > 2Hz<br/>for the last 0.5 seconds
    
    Setpoint_Check_State --> Rejected_State : [No Valid Setpoints]
    Rejected_State --> Manual_Hold_State : Warning Generated
    
    Setpoint_Check_State --> Offboard_State : [Setpoints Valid] & [EKF Valid]
    
    state Offboard_State {
        direction LR
        Validating --> Pursuing : Trajectory Tracking
        Pursuing --> Validating : Check Next Setpoint
    }
    
    Offboard_State --> Manual_Hold_State : Setpoint Timeout (> 0.5s)
    Offboard_State --> Failsafe_State : Ext. RC Loss or EKF Failure

이 상태 전이도는 소스 코드 상에서 state_machine_helper.cpp 및 분리된 비행 모드 관리자(Flight Mode Manager) 내의 플래그 검사(Bitwise OR/AND 연산)를 통해 엄격하게 보호된다.

4. Ardupilot 시스템과의 관제 철학 및 모드 전환 차이

동일한 오픈소스 자율 비행 운영 체제인 PX4와 Ardupilot(펌웨어 4.4 이상 기준)은 외부 제어를 수용하는 방식과 사용자 안전 철학에서 뚜렷한 설계 차이를 보인다.

  1. 천이의 엄격성과 융통성
  • PX4 (Offboard Mode): 서론에서 언급한 바와 같이 아주 엄격하고 결정론적인 선행 조건(0.5초 선 수신 원칙)을 갖는다. 개발자가 외부 컴퓨터 제어 파이프라인의 순서를 실수하면 기체는 단호하게 전환 요청을 거부(CMD_ACK_DENIED)한다. 철저히 로봇 공학 및 자율 주행 연구 우선의 철학에 기반하여 예외를 배제한다.
  • Ardupilot (Guided Mode): 외부 지령을 수용하는 모드를 “Guided” 모드라고 부른다. Ardupilot은 GCS 비행 운용자의 사용자 친화적 관제를 중시하여, 사용자가 Guided 모드로 먼저 강제 전환해 둔 상태(기체는 그 자리에 위치 제어로 Hold됨)에서 외부 지령이 수신되기를 느긋하게 기다리는 느슨한(Slack) 상태 천이 모델을 허용한다.
  1. 단절 시 복구(Fallback) 행위
  • PX4: 오프보드 지령 신호가 끊어지면 COM_OBL_RC_ACT 등의 파라미터 정책에 따라 즉시 Hold(정지 비행), Land(착륙) 또는 Return(회귀) 모드로 전환되는 강력한 페일세이프 천이 트리거가 발동된다.
  • Ardupilot: Guided 명령의 최신화가 끊기더라도, 설정된 마지막 웨이포인트나 목표 벡터를 소화한 후 그 자리에서 안전하게 정지하여 추가 명령을 무한 대기하는 유연함을 기본 특성으로 갖는다.

5. MAVLink 및 ROS 2 uXRCE-DDS 관점의 천이 제어 실무

오프보드 상태 천이를 GCS나 분산형 외부 노드에서 소프트웨어 코드로 정상 트리거할 때는 단순한 모드 변경 커맨드 단일 전송만으로는 부족하다.

1. MAVLink 인터페이스 기반:
MAVSDK 등의 라이브러리를 사용할 때, 모드 전환인 MAV_CMD_DO_SET_MODE 명령어 패킷을 GCS 채널로 송신하기 이전에, 반드시 독립된 백그라운드 스레드(Thread)를 통하여 SET_POSITION_TARGET_LOCAL_NED 등의 제어 지령 페이로드를 초당 2Hz 이상(통상 10~50Hz) 연속 퍼블리싱하는 데이터 파이프라인 구축 작업을 선행시켜야 한다. 데이터 선행 입력 타이밍이 맞지 않으면 MAV_RESULT_TEMPORARILY_REJECTED 응답 코드가 매몰차게 회신된다.

2. ROS 2 인터페이스 (Micro XRCE-DDS) 기반:
ROS 2 분산 메시지 생태계 통신에서는 모드 변경 요청조차 uORB 인터페이스를 DDS 도메인으로 사상(Mapping)한 퍼블리싱 동작으로 수행된다.
절차적으로 외부 스마트 노드가 먼저 OffboardControlMode 메시지를 퍼블리시하여 오프보드 제어 토폴로지(위치 추종인지, 속도 지령 추종인지 선언) 활성화를 알리고, 이어서 연속적으로 TrajectorySetpoint 메시지 스트림으로 위치나 속도 지령 값을 흘려보낸다.
그 안전한 데이터 흐름이 확인된 이후에 VehicleCommand ટો픽을 사용해 VEHICLE_CMD_DO_SET_MODE (마스킹 파라미터 Custom Mode = Offboard 적용) 이벤트를 전송해야, 비로소 PX4 FSM이 이를 논리적 결점(Logical Fault)이 구별되지 않는 정상적인 시스템 천이로 승인하고 비행 제어 연산을 외부로 오프로딩(Off-loading)한다.

요약하자면, PX4-Autopilot의 FSM 기반 Offboard 천이는 외부 노드(GCS 혹은 ROS 2) 제어권 위임에 있어 예외를 원천 차단하는 “신뢰할 수 없는 환경에서의 보장된 신뢰(Trustless Determinism)“를 구현하는 가장 강력한 자율 비행 내결함성 보증 수단이다.
