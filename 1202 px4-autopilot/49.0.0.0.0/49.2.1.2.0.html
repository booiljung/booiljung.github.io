49.2.1.2 통신 단절(Data Link Loss) 및 조종기 신호 상실(RC Loss) 감지 시의 모드 강등(Mode Downgrade) 메커니즘

1. 개요 및 설계 철학

무인항공기(UAV)의 비행 중 지상 관제 시스템(GCS, Ground Control Station)과의 주기적인 통신 수단인 데이터 링크(Data Link)나 조종자의 직접적인 수동 제어 신호(RC, Radio Control)가 상실되는 상황은 시스템의 안전성을 심각하게 위협하는 주요 요인이다. PX4-Autopilot(Firmware v1.14 기준)은 이러한 비상 상황을 감지하고, 기체의 생존성과 더불어 지상 인프라의 안전을 보장하기 위해 강력하고 유연한 Failsafe(안전 장치) 메커니즘을 내장하고 있다.

이러한 메커니즘의 핵심은 **모드 강등(Mode Downgrade)**이다. 정상적인 비행 모드(예: 위치 보존 제어가 수행되는 Position 모드, 또는 외부 컴퓨터 지령에 의존하는 Offboard 모드) 상태에서 통신이 두절될 경우, PX4-Autopilot의 내부 상태 머신(State Machine)은 즉시 안전을 보장할 수 있는 우선순위가 높은 모드(RTL, Hold, Land 등)로 자동 천이(Transition)한다. 이는 Ardupilot 시스템이 FS_GCS_ENABL 및 FS_THR_VALUE 파라미터 등을 기반으로 Failsafe를 수행하는 것과 유사하지만, PX4는 중앙의 Commander 모듈을 통해 모든 내비게이션 상태(Navigation State) 변화를 일원화하여 관리하므로 아키텍처의 투명성이 더 높다.

2. 신호 상실 감지 메커니즘(Detection Mechanism)

2.1 데이터 링크 상실(Data Link Loss) 감지

PX4-Autopilot은 GCS(QGroundControl v4.3 기준)와 실시간으로 MAVLink 프로토콜을 교환한다. 연결의 유효성을 평가하기 위해 시스템은 GCS로부터 수신되는 MAVLink HEARTBEAT 메시지의 주기를 감시한다.

  • 정상적인 상황에서는 1Hz(1초에 1회) 수준의 주기로 Heartbeat가 도달한다.
  • PX4는 COM_DL_LOSS_T 파라미터(기본값: 10초)에 명시된 시간 동안 유효한 HEARTBEAT 메시지를 수신하지 못하면, 데이터 링크 상실(Data Link Loss) 이벤트를 발생시킨다.

2.2 조종기 신호 상실(RC Loss) 감지

수동 조종(Manual Control)이 이루어지고 있는 상황에서의 RC Loss 감지는 수신기(Receiver) 하드웨어와 프로토콜 라우팅 계층에서 처리된다. SBUS, CRSF, Spektrum 등 다양한 수신기 프로토콜은 각각 고유하게 신호 상실 플래그(Failsafe Flag)를 지원한다.

  • 수신기가 송신기의 신호를 놓치면 자체적으로 Failsafe 프레임을 생성하거나 유효 채널 값의 업데이트를 중지한다.
  • PX4는 COM_RC_LOSS_T 파라미터(기본값: 0.5초)를 초과하여 유효한 RC 신호 업데이트가 이루어지지 않으면 RC Loss를 선언한다. (참고: 과거 버전의 경우 특정 채널의 펄스 폭 저하를 이용하기도 하였으나, 최신 펌웨어에서는 프로토콜 자체의 플래그 검증 메커니즘을 선호한다.)

3. 모드 강등(Mode Downgrade) 결정 아키텍처

신호 상실이 감지되면, Commander 모듈(비행 상태 관리자) 내의 상태 머신은 현재 비행 모드와 설정된 Failsafe 파라미터를 대조하여 어느 모드로 강등할지 결정한다.

3.1 상태 천이 다이어그램

다음은 모드 강등 로직의 흐름을 보여주는 유한 상태 머신(FSM) 아키텍처 다이어그램이다.

stateDiagram-v2
    [*] --> NormalFlight : Arming & Takeoff
    
    state NormalFlight {
        [*] --> MissionMode
        [*] --> OffboardMode
        [*] --> PositionMode
    }
    
    NormalFlight --> FailsafeEvaluation : 감지 (COM_DL_LOSS_T\nor COM_RC_LOSS_T 트리거)
    
    state FailsafeEvaluation {
        state "파라미터 평가\n(NAV_DLL_ACT / NAV_RCL_ACT)" as ParamCheck
        ParamCheck --> Mode_Return : 설정이 RTL일 때
        ParamCheck --> Mode_Hold : 설정이 Hold일 때
        ParamCheck --> Mode_Land : 설정이 Land일 때
        ParamCheck --> Mode_Terminate : 설정이 Flight Termination일 때
    }
    
    FailsafeEvaluation --> RecoveryMode
    
    state RecoveryMode {
        Mode_Return : Return (귀환)
        Mode_Hold : Hold (위치 고수)
        Mode_Land : Land (착륙)
        Mode_Terminate : System Terminate (추락)
    }
    
    RecoveryMode --> NormalFlight : 통신 복구 및\n조종자 제어권 회수(Override) 승인

위 다이어그램에서 알 수 있듯, GCS 또는 조종기 중 하나라도 복구될 경우 비행 모드는 일시적으로 Failsafe 상태에 머무르다가, 조종자가 스틱을 움직이는 등 제어권 회수 의지(Override)를 명시하면 다시 원래 모드로 복귀할 수 있다.

3.2 핵심 제어 파라미터 구성 (Configuration)

QGroundControl을 통해 관리되는 모드 강등 관련 핵심 파라미터는 다음과 같다.

  • NAV_DLL_ACT (Data Link Loss Action): 데이터 링크 상실 시 어떤 제어 모드로 강등될 것인지 결정한다. 0=Disable(아무 조치 안 함), 1=Hold(정지비행), 2=Return(RTL), 3=Land, 6=Terminate 등이 있다.
  • NAV_RCL_ACT (RC Loss Action): 조종기 상실 시 동작을 결정한다. 값의 범례는 NAV_DLL_ACT와 거의 동일하다.
  • COM_RC_IN_MODE: 다중 제어 환경에서 RC 입력의 필수 여부를 정의한다. 만약 완전한 자율 에이전트 비행(Mission-only 또는 Offboard)으로 설계된 기체라면 해당 값을 조정하여 조종기 없이도 조종기 신호 상실 예외를 회피(Bypass)하도록 설정할 수 있다.

4. 모드 강등 중의 소스 코드 내부 동작(Source Code Understanding)

PX4-Autopilot의 Failsafe 진입 로직은 주로 src/modules/commander/Commander.cpp 파일과 state_machine_helper.cpp 내 함수에서 다루어진다.

RC Loss 이벤트 모니터링은 메인 스레드의 주기적인 작업(Run Loop) 내부에서 실행된다. 구조적인 분석은 다음과 같다.

  1. uORB 메시징 폴링: Commander는 vehicle_status 및 rc_channels uORB 토픽(Topic)을 구독(Subscribe)한다. 유효한 타임스탬프 갱신이 COM_RC_LOSS_T를 초과하여 이루어지지 않았음을 타임아웃 타이머가 감지한다.
  2. 플래그 업데이트: 시스템 상태(System State) 데이터 구조 내의 rc_signal_lost 플래그가 true로 설정된다.
  3. 상태 머신 호출: statemachine_step() 함수가 Failsafe 조건의 우선순위에 따라 평가를 수행한다. 예를 들어 Return(RTL) 액션이 요구되면 commander는 main_state_transition() 함수를 호출하여 현재 상태(main_state)를 MAIN_STATE_RETURN으로 변경한다.
  4. 제어기 하달: 변경된 main_state는 다시 vehicle_status 토픽으로 발행(Publish)되며, Position Controller는 이를 구독하여 기체를 홈 위치(Home Position)로 유도하는 궤적(Trajectory) 생성 로직을 개시한다.

5. 실무에서의 관제 설계 제안 및 비유

학술적이고 체계적인 관점에서 볼 때, 모드 강등(Mode Downgrade) 메커니즘을 제대로 튜닝하지 않으면 임무 성공률이 치명타를 입을 수 있다.

통신 두절로 인한 자동 귀환 과정을 비유하자면, 지휘 통제소(GCS)와의 무전이 끊긴 군부대(UAV)가 사전에 하달받은 ’비상 작전 지침서(Failsafe Parameters)’를 개봉하여 베이스 캠프(Home)로 복귀할지, 제자리 사수(Hold)를 할지 결정하는 것과 같다.

가시권 비행(VLOS) 시에는 조종기를 통한 즉각적인 개입이 가능하므로 NAV_RCL_ACT를 Return 또는 Hold로 두는 것이 유리하지만, 비가시권 자율 비행(BVLOS) 환경 중 특히 LTE/5G 등 고지연 혹은 간헐적 단절이 잦은 망에서 운용할 경우 짧은 HEARTBEAT 손실에도 잦은 Return이 발생하여 효율성이 덜어질 수 있다. 따라서 GCS 백엔드 개발 시에는 COM_DL_LOSS_T의 타임아웃 마진을 통신 망 상태에 맞게 적절히 늘리고, 기체의 자율 생존성을 극대화할 수 있도록 설계해야 한다. 더 나아가, 논문 “Network Performance Evaluation for BVLOS UAV Operations” 등에서 자주 다뤄지는 바와 같이 상공망의 셀 핸드오버 지연시간(Cell Handover Delay)에 기인한 일시적 MAVLink 단절을 Failsafe 트리거와 혼동하지 않도록 임계치(Threshold)를 튜닝하는 것이 공학적으로 필수라 하겠다.
