49.2.2 세트포인트(Setpoint) 스트리밍의 워치독(Watchdog) 감시 아키텍처

1. 개요 및 워치독(Watchdog)의 필요성

PX4-Autopilot의 Offboard 모드나 ROS2 기반의 자율 에이전트 제어는 외부 컴패니언 컴퓨터(Companion Computer) 또는 지상 관제 시스템(GCS)이 계산한 궤적 지령, 즉 **세트포인트(Setpoint: 위치, 속도, 가속도, 자세 등)**를 기체에 실시간으로 주입(Streaming)하는 방식으로 이루어진다.

멀티로터의 제어 시스템은 본질적으로 불안정한 동역학(Dynamics)을 내포하기 때문에, 제어 지령의 갱신이 지연되거나 중단되면 기체는 잘못된 목표점을 항해 폭주하거나 치명적인 추락을 겪을 수 있다. 이를 방지하기 위해 PX4는 소프트웨어 레벨의 워치독(Watchdog) 감시 아키텍처를 구축하고 있다. 이 아키텍처는 외부로부터의 세트포인트가 사전에 정의된 타임아웃(Timeout) 이내에 지속적으로 유입되고 있는지를 실시간으로 모니터링하여, 통신 또는 프로세스 장애 시 즉각적으로 기체의 통제권을 내부 Failsafe 로직으로 회수(Fallback)하는 안전망 역할을 한다.

2. 워치독 감시 아키텍처 상세

2.1 스트리밍 및 감시 메커니즘

외부 제어기(예: ROS2 노드 또는 MAVSDK 스크립트)는 특정한 주파수(일반적으로 10Hz ~ 50Hz)로 제어 지령을 퍼블리싱(Publishing)해야 한다. MAVLink의 SET_POSITION_TARGET_LOCAL_NED 메시지나 ROS2의 TrajectorySetpoint uORB 토픽이 이에 해당한다.

  • 사전 요구 조건(Pre-condition): PX4는 Offboard 모드로의 진입(Transition)을 허가하기에 앞서, 이미 1초 이상 세트포인트 스트리밍이 유효하게 들어오고 있는지를 검사한다. 이는 초기 진입 시 발생할 수 있는 과도 상태(Transient State)를 예방하는 워치독의 1차 관문이다.
  • 스트리밍 감시(Runtime Monitoring): 진입 후 비행 중에는 통상 0.5초(파라미터 COM_OF_LOSS_T 기본값)로 설정된 워치독 타이머가 동작한다. 유효한 제어/하트비트 메시지가 하나 도착할 때마다 이 워치독 타이머는 리셋(Reset)된다.

2.2 아키텍처 데이터 흐름도

다음은 워치독이 세트포인트 스트리밍을 감시하고, 예외 발생 시 어떻게 방어 로직을 전개하는지를 보여주는 아키텍처 다이어그램이다.

        WD-->>WD: 타이머 리셋 (Reset)
        uORB->>Ctrl: 현재 세트포인트 전달 및 제어 수행

    loop 내부 폴링 주기
        WD->>WD: 경과 시간 계산
    end
    
    note over WD: 타임아웃 발생 (시간 > COM_OF_LOSS_T)
    
    WD->>Ctrl: Offboard 제어 지령 차단
    WD->>Ctrl: 내부 위치 고수(Hold) 또는 RTL 궤적 생성 강제

이 다이어그램에서 보듯, 워치독 타이머가 타임아웃에 도달하게 되면 시스템은 더 이상 잘못되었거나 지연된 세트포인트를 추종하지 않으며, 자체적인 FSM(Finite State Machine) 규칙에 따라 생존을 위한 동작 모드로 강제 스위칭된다.

3. Ardupilot의 Guided 모드 감시 방식과의 차이

Ardupilot 환경에서는 이에 대응하는 모드로 Guided 모드가 존재한다. Ardupilot 역시 타임아웃 감지(예: FS_GCS_ENABL에 종속된 타임아웃 등) 메커니즘을 가지고 있지만, 아키텍처의 엄격성에 있어서는 PX4와 차이를 보인다.

  • 진입의 엄격성: Ardupilot은 GCS 명령어로 즉각적인 Guided 모드 진입을 허용하는 경향이 크지만, PX4의 Offboard 모드는 활성화되기 이전에 반드시 “연속된 세트포인트의 수신 레이트(Rate) 유효성“을 선행 검증한다. 앞서 진입 전 1초간의 사전 스트리밍 요구 로직이 부재 시 모드 전환 자체를 Commander가 거절(Reject)한다.
  • 분배의 일관성: PX4는 uORB 아키텍처를 근간으로 실시간 스케줄링을 하므로 워치독 로직 자체가 uORB 토픽의 수신 타임스탬프(timestamp 필드) 델타(Delta) 값을 직접 평가하는 수학적 방식으로 강결합되어 있다.

4. 제어 공학적 고려 및 파라미터 (Configuration)

워치독의 타임아웃 여유값(Margin)은 기체의 동적 특성(Dynamic Characteristics)과 직결된다.
기본 설정인 COM_OF_LOSS_T = 0.5s는 보수적(Conservative)인 안전 기준으로, 멀티로터의 빠른 자세 붕괴를 막는 데 적당하다. 그러나 다음과 같은 개발 환경에서는 추가적인 튜닝이 요구된다.

  1. 무선 네트워크 불안정: LTE/5G나 Wi-Fi를 경유하는 uXRCE-DDS 브릿지를 사용할 경우, 네트워크 지터(Jitter)로 인해 순간적인 패킷 지연이 0.5초를 초과할 수 있다. 이 경우 기체가 끊임없이 Hold 모드로 강등되는 ‘Mode Flapping’ 현상이 발생하므로, 타임아웃을 1.0초~2.0초 사이로 늘리는 것을 고려해야 한다.
  2. 논문 “Real-time Guarantee in UAV Control Systems” 등에서 언급되는 워치독 전략: 소프트웨어 에이전트의 GC(Garbage Collection) 또는 스레드 블로킹으로 인한 일시적인 데드라인 미스(Deadline Miss) 시 기체가 어떻게 과도 상태(Transient State)를 버틸 것인가에 관한 연구들이다. PX4는 워치독 타임아웃 즉시 기존 궤적의 가속도 성분을 0으로 감쇠시켜 급격한 거동 변화를 흡수하는 범플리스 보상(Bumpless Transfer) 메커니즘을 병행 호출한다.

5. 소스 코드 컴포넌트 분석 (Source Code Understanding)

소스 코드 레벨(v1.14 기준)에서 src/modules/commander/Commander.cpp 내부의 폴링 루프(Polling Loop)를 살펴보면 워치독 감시 로직을 명확히 볼 수 있다.
Commander는 offboard_control_mode 토픽을 구독(subscribe)하며 다음 수식에 따라 타임아웃 여부를 결정한다.

\Delta t = t_{current\_time} - t_{last\_valid\_setpoint}

만약 \Delta t > \text{COM\_OF\_LOSS\_T} 라면, 비행 모드 관리자 함수 내부의 offboard_lost 플래그가 true로 활성화되며, Failsafe 상태 머신이 즉각 실행되어 Hold 모드 또는 파라미터로 설정된 백업 비행 모드로 강제 스위칭된다.

6. 결론

PX4-Autopilot의 세트포인트 워치독 아키텍처는 유연한 외부 제어 개입을 허락함과 동시에, 그 외부 제어 루프의 소프트웨어적·네트워크적 결함으로부터 기체 하드웨어를 완벽하게 분리하여 보호하는 핵심 중추이다. 지상 관제 시스템(GCS)이나 다중 에이전트 군집 알고리즘 프로그래머는 이 워치독의 깐깐한 타임아웃 및 진입 요구 사항을 아키텍처 설계 초기부터 반드시 고려해야만 안정적인 미션 비행을 담보할 수 있다.
