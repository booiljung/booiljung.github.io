49.1.1.1 실시간 운영체제(RTOS) 기반 내부 루프(Inner Loop)와 비실시간/연성 실시간(Soft Real-time) 외부 루프(Outer Loop) 간의 통신 비동기성

1. 개요

자율 비행 로봇 시스템에서 비행 제어기(Flight Controller)의 내부 루프(Inner Loop)는 기체의 물리적 추락을 막기 위해 1밀리초(ms) 단위의 경성 실시간성(Hard Real-time) 을 반드시 보장해야 한다. 이를 위해 PX4-Autopilot은 NuttX RTOS 환경 위에서 동작하며, 정해진 데드라인(Deadline)을 칼같이 준수하는 결정론적(Deterministic) 스케줄링을 사용한다.
반면 임무 컴퓨터(Mission Computer)에서 구동되는 객체 인식, 슬램(SLAM), 경로 계획 등은 복잡한 연산을 요하므로 Linux 기반의 범용 운영체제나 ROS 2와 같은 연성 실시간(Soft Real-time) 환경, 혹은 비실시간(Non Real-time) 환경에서 동작하는 외부 루프(Outer Loop)를 형성한다. 이 두 이질적인 시스템 간의 데이터를 교환할 때 발생하는 통신 비동기성(Communication Asynchrony) 의 원인과 극복 방안을 분석하는 것은 오프보드(Offboard) 제어기 설계의 핵심이다.

2. 스케줄링 패러다임 차이와 지터(Jitter)의 발생

NuttX RTOS 상의 자세 제어기(Attitude Controller) 및 Rate Controller는 타이머 인터럽트를 통해 250\text{Hz} ~ 1\text{kHz} 속도로 항상 똑같은 주기에 깨어난다. 반면 Linux(예: Ubuntu 22.04) 환경의 임무 컴퓨터(MC) 파이프라인에서는 운영체제 스케줄러가 여러 프로세스의 공평성(Fairness)과 처리량(Throughput)을 고려해 CPU 자원을 나누어 가지므로 프로세스의 실행 주기가 완벽히 일정하지 않다.
이로 인해 외부 루프에서 주기적으로 20\text{Hz} 로 제어 데이터를 퍼블리시(Publish)하더라도, 비행 제어기에 도착하는 패킷의 인터벌(Interval)은 45\text{ms} 에서 55\text{ms} 사이를 오가는 지터(Jitter) 현상이 반드시 발생한다. 지터가 심화되면 비행 제어기 입장에서는 데이터가 “한꺼번에 몰려오거나”, “데드라인 전까지 도착하지 않는(Data Starvation)” 현상을 겪게 된다.

3. 통신 비동기성 아키텍처: 락프리 버퍼와 uORB

이러한 두 세계 간의 마찰을 줄이기 위해, 비행 제어기와 임무 컴퓨터는 양방향 통신에서 동기화 블록(Blocking Synchronization)을 사용하지 않는다. 한쪽이 데이터를 보낼 때 상대방의 수신 상태를 기다려주면 경성 실시간 시스템 전체가 멈춰버리는(Deadlock 또는 Priority Inversion) 치명적 오류가 발생하기 때문이다.

PX4-Autopilot(v1.14 이상)은 임무 컴퓨터와의 비동기 통신을 위해 내부 데이터 버스인 uORB(Micro Object Request Broker) 와 uXRCE-DDS 클라이언트를 연동하여 완전한 비동기식 독립성을 확보한다.

4. 비동기 환경에서의 타임스탬프와 지연 보상

이종 시스템 간의 통신 비동기성에서 가장 민감한 공학적 문제는 “이 데이터가 언제 생산되었는가“를 식별하는 타임스탬프(Timestamp) 오차이다. 비실시간 OS에서 보낸 목표 수치(Setpoint)가 통신 경로 내 큐(Queue)에 잠시 머무르다 뒤늦게 비행 제어기에 도착한 경우, 비행 제어기가 이를 최신 정보로 착각하면 심각한 제어 위상 지연(Phase Lag)이 유발된다.

  1. 시간 동기화(Timesync): MAVLink 및 uXRCE-DDS는 하부에서 두 OS의 시스템 시간(System Clock) 오프셋을 계산하여 임무 컴퓨터가 FC의 마이크로초 단위 timestamp와 동일한 기준을 갖도록 보정한다.
  2. 폐기 시간(Staleness) 모니터링: FC의 위치 제어 모듈(Position Controller) 등은 TrajectorySetpoint uORB 토픽을 꺼냈을 때, 자체 RTOS 현재 시간과 메시지에 찍힌 timestamp의 차이를 비교한다. 차이가 허용 오차 이상 벌어졌다면(예: 네트워크 버퍼링 지연), 이 데이터를 폐기(Drop)하거나 이를 기반으로 이전 속도 성분을 외삽법(Extrapolation)으로 보정하여 사용한다.

5. 공학적 시사점

오프보드 제어를 완벽하게 구현하고자 하는 개발자는 단순히 “주기적인 루프(While/Rate Loop)를 돌려 명령을 쏜다“는 개념을 넘어, 임무 컴퓨터 자체의 스케줄링 지터를 최소화하기 위해 노력해야 한다.

  • 임무 제어 프로그램을 Linux 환경에서 작성할 때는 시스템 콜 호출을 최소화하고, 실행 스레드에 **스케줄링 우선순위(SCHED_FIFO / SCHED_RR 등)**를 부여하거나 ROS 2 QoS(Quality of Service) 정책을 BEST_EFFORT 와 고속 전송 주기로 조합하여 일정한 레이턴시를 획득해야 한다.
  • PX4-Autopilot은 Ardupilot보다 훨씬 좁은 타임아웃 윈도우(Timeout Window)를 운용하므로 연성 실시간 OS의 가비지 컬렉션(Garbage Collection)이나 커널 인터럽트에 의해 0.5초(기본값) 이상 제어 신호가 지연되지 않도록 주의해야만 추락이나 의도치 않은 비행 모드 변경(Failsafe)을 방지할 수 있다.
