49.1 실시간 제어(Offboard/Guided) 아키텍처의 비행 동역학 및 제어 공학적 기초

1. 개요

멀티로터 및 고정익 형태의 무인항공기(UAV)에서 PX4-Autopilot의 오프보드 모드(Offboard Mode)나 Ardupilot의 가이드 모드(Guided Mode)를 활용하여 실시간으로 위치 및 속도를 제어하는 아키텍처는 제어 공학과 비행 동역학(Flight Dynamics)의 원리를 밀접하게 결합하고 있다. 임무 컴퓨터(Mission Computer)에서 계산된 목표 궤적(Target Trajectory)이 기체의 실제 조종면(Control Surfaces)이나 모터의 회전수(RPM)로 변환되기까지는 매우 빠른 시간 내에 다단계의 계층적 제어 루프(Hierarchical Control Loop)를 통과해야 한다.
본 절에서는 외부 컴퓨터가 기체에 내리는 제어 명령이 물리적인 동역학 및 제어 이론 관점에서 어떻게 모델링되고 실행되는지에 대한 기초를 다룬다.

2. 강체 비행 동역학(Rigid Body Flight Dynamics) 모델링

무인항공기를 6자유도(6-DoF)를 가지는 강체(Rigid Body)로 가정할 때, 시스템의 상태 벡터(State Vector)는 관성 좌표계(Inertial Frame)에서의 위치, 속도, 그리고 동체 좌표계(Body Frame)에서의 자세(Euler Angles: \phi, \theta, \psi 혹은 Quaternion)와 각속도(Angular Velocity: p, q, r)로 정의된다.

Newton-Euler 방정식에 기반한 쿼드로터의 병진 운동(Translational Motion)과 회전 운동(Rotational Motion) 방정식은 다음과 같이 기술된다.

  • 병진 운동: m \ddot{\mathbf{r}} = \mathbf{F}_T + \mathbf{F}_g + \mathbf{F}_d
  • 회전 운동: \mathbf{J} \dot{\mathbf{\omega}} + \mathbf{\omega} \times (\mathbf{J} \mathbf{\omega}) = \mathbf{\tau}

여기서 m은 기체 질량, \mathbf{r}은 위치 벡터, \mathbf{F}_T는 모터 추력, \mathbf{F}_g는 중력, \mathbf{F}_d는 공기 저항력(Aerodynamic Drag)을 나타낸다. \mathbf{J}는 관성 텐서(Inertia Tensor) 행렬, \mathbf{\tau}는 롤(Roll), 피치(Pitch), 요(Yaw) 축의 제어 모멘트(Control Moment)이다.

임무 컴퓨터가 내리는 오프보드 제어 위치(Position/Velocity Target) 명령이란, 이러한 뉴턴 역학 시스템에서 \mathbf{r} 이나 \dot{\mathbf{r}} 이 목표 상태 도달에 필요한 일련의 과도 응답(Transient Response) 곡선을 결정해주는 과정과 같다.

3. 계층적 제어 아키텍처(Hierarchical Control Architecture)

실시간성을 극대화하기 위하여, PX4-Autopilot(v1.14 기준)의 제어 시스템은 내부적으로 다단 폭포수 형태(Cascaded Structure)를 이룬다. 임무 컴퓨터는 “어디로 갈 것인가”(Outer Loop)만 결정하고, 비행 제어기(FC)는 “어떻게 물리적으로 달성할 것인가”(Inner Loop)를 계산한다.

3.1 외부 루프(Position & Velocity Control)

오프보드 명령을 수신한 PX4의 위치 제어 모듈(Position Controller)은 외란(Wind Disturbance 등)에 강건하면서도 응답성을 확보하기 위해 PID 제어기나 좀 더 고도화된 선형 2차 임무 제어기(LQR) 형태의 게인 스케줄링(Gain Scheduling)을 채택할 수 있다. 목표 위치 벡터와 EKF(Extended Kalman Filter) 기반 현재 상태 추정치 사이의 오차 공간(Error Space)을 생성하여, 목표 속도 및 가속도를 도출한다.

3.2 내부 루프(Attitude & Rate Control)

목표 가속도를 만들기 위해 기체는 반드시 특정한 기울기, 즉 목표 자세(Target Attitude)를 형성해야 한다. 목표 추력 방향 벡터 \mathbf{z}_B를 계산해 목표 쿼터니언(Quaternion)으로 변환 후, 자세 제어기(P 제어 위주)와 속도 제어기(PID 제어 기반)에 차례로 넣는다. 회전 관성이 작아 신속하게 변하는 각속도를 다루어야 하므로 Rate Control의 업데이트 주기는 매우 짧아야 한다.

4. 시스템 위상 지연(Phase Lag)과 대역폭(Bandwidth) 문제

제어 공학 설계의 핵심 과제 중 하나는 전체 시스템 제어 루프의 응답 주파수 성분을 나타내는 대역폭(Bandwidth)을 적절하게 분할하고, 통신 라인에서 발생하는 비동기 지연 시간(Network Delay/Transport Lag)이 닫힌 루프(Closed-loop) 시스템의 안정도 여유(Stability Margin)를 깎아내리지 않도록 분석하는 것이다.

보통 가장 안쪽 루프(각속도 루프)의 대역폭은 20\sim50\text{Hz}에 달하지만, 바깥쪽 루프(위치 루프)나 임무 컴퓨터와의 통신에 따른 나이퀴스트 주파수(Nyquist Frequency) 한계, 계산 지연 등을 포함하면 이를 대폭 줄여야 오버슈트(Overshoot) 없는 안정적인 조향이 가능하다. 학술 논문 “A Survey on Trajectory-Tracking Control for Quadrotor UAVs” 등 다수의 연구에서 실시간 통신 시간 지연(Delay \tau_d)이 있을 경우 다음과 같은 지연 요소를 라플라스 영역(Laplace Domain)에서 고려하여(예: e^{-s \tau_d}) 제어 게인을 축소하거나 리드-래그(Lead-Lag) 보상기를 추가하는 방식을 권장하고 있다.

5. PX4 와 Ardupilot 제어 아키텍처 해석 차이점

  • PX4-Autopilot: 철저하게 모듈화 된 uORB 토픽 브로커 구조를 사용하여, 서로 다른 주파수의 제어기가 비동기적으로(Asynchronous) 설계되어 있다. 입력된 오프보드 궤적 정보는 스무스 트래젝토리 제너레이터(Smooth Trajectory Generator)를 통과하여 이산적인 입력(Step Input)이 들어와도 급격한 저크(Jerk) 없이 매끄러운 궤적으로 재생성된 후 PID를 거친다.
  • Ardupilot (Guided Mode): 위치/속도 컨트롤러(AC_PosControl, AC_AttitudeControl) 라이브러리를 통해 상대적으로 덜 모듈화된 일체형 C++ 객체 지향 형태로 처리한다. 가이드 모드는 타겟 지점이 갱신되면 로컬 버퍼에 담아 Kinematic 제약을 기반으로 최대 가속 등을 연산하며, 모드 유지의 타임아웃 규칙이 단발 명령 갱신을 허용하도록 더 느슨하게 튜닝되는 경향이 있다.

6. 결론

실시간 오프보드 제어를 성공적으로 달성하기 위해서는 동체 거동의 물리적 한계점(추력, 회전 모멘트 대역폭)을 벗어나지 않는 타당한 목표 값(Feasible Setpoint)을 임무 컴퓨터에서 생성해주어야 한다. 동시에 PX4 내부 제어 계층의 시간 상수(Time Constant)와 보드 간 통신 지연 시간을 면밀히 분석함으로써 위상 여유(Phase Margin)를 충분히 확보하는 보수적인 제어기를 설계하는 것이 공학적으로 필수적이다.
