49.7.1.1 POSIX 스레드(Pthreads) 및 Qt QThread 기반의 타이머(Timer) 콜백(Callback) 이벤트 루프 구성

1. 개요

지상 관제 시스템(GCS) 애플리케이션인 QGroundControl(QGC) 내부의 백엔드 아키텍처는 끊임없이 반복(Looping)되는 비동기 주기 작업들로 가득 차 있다. 정해진 주사율에 맞추어 MAVLink Heartbeat 패킷을 발송하거나, 실시간 조이스틱 제어 입력을 기체로 퍼블리시(Publish)하며, 수신된 Telemetry 데이터를 UI에 갱신하는 모든 일상적 책무가 “타이머(Timer)” 기반의 폴링(Polling) 루틴 또는 콜백(Callback) 이벤트로 구현된다. 이를 안정적으로 백그라운드에서 지연 없이 수행하기 위하여 Qt의 QThread와 저수준의 POSIX 스레드(Pthreads) 모델이 혼합 사용된다.

2. QThread 내부의 이벤트 루프(Event Loop) 아키텍처

Qt 프레임워크의 코어가 제공하는 QThread는 사실 단순히 OS가 제공하는 원시 스레드(Raw Thread) 래퍼(Wrapper)가 아니다. 자체적인 큐(Queue)와 대기 처리 로직을 지닌 강력한 이벤트 루프(Event Loop: QEventLoop) 를 내장하고 있다.

  • QGC의 통신 매니저나 MAVLink 파서(Parser) 객체를 특정 QThread의 작업자(Worker)로 moveToThread() 시키면, 해당 스레드는 자체적인 스레드 안전(Thread-Safe) 이벤트 루프를 갖게 된다.
  • 이때 스레드 내부에서 QTimer 객체를 생성하여 시작(start()) 명령을 내리면, 지정된 밀리초(ms) 주기마다 timeout() C++ 시그널(Signal)이 발생하게 된다.
  • 이 시그널은 즉시 동일 스레드 혹은 다른 스레드의 처리 함수(Slot)를 호출(Callback)하여, 메인 GUI 스레드를 1ms도 지연시키지 않고 백그라운드 데이터 전송 처리를 완수할 수 있게 돕는다.

3. 타이머 해상도(Timer Resolution)와 시스템 스케줄링의 맹점

문제는 운영체제(Linux, Windows, macOS)의 스케줄러가 소프트웨어 타이머를 정밀하게 1ms 단위로 보장해 주지 않는다는 것이다. 특히 Windows 환경의 경우, 기본 커널 스케줄러의 타임 슬라이스(Time Slice) 및 타이머 해상도가 15.6ms 수준에 머물러 있어 QTimer에 10ms 호출을 지시해도 실제로는 15~16ms 단위로 무작위 지연 호출이 발동되는 현상(Jitter)이 빈번하게 일어난다.

실시간(Real-time) 센서 관제나 50Hz(20ms 간격)의 Offboard/Guided 명령 송신이 절대적으로 유효해야 하는 PX4 기체 특성상, GCS의 지터는 곧 ‘통신 유효성 위반(Timeout)’ 에러로 직결된다.

따라서 QGroundControl이나 MAVSDK 백엔드는 정밀도를 위해 다음과 같은 우회 하드웨어 타이머 접근 전략을 사용한다.

  1. QTimer::setTimerType(Qt::PreciseTimer) 모드 명시적 지정.
  2. 시스템 API인 POSIX 스레드의 실시간 스케줄링 권한(Linux의 경우 SCHED_FIFO나 SCHED_RR)을 획득.

4. 고성능 타이밍 유지를 위한 POSIX 스레드(Pthreads) 결합

소프트웨어 이벤트 루프(QTimer)가 신뢰할 수 없을 만큼 빠른 처리(예: 1kHz급, 1ms 이하 지연)가 요구되는 비디오 디코딩 프레임 버퍼링이나 원시(Raw) 시리얼 포트 인터럽트 핸들링에서는, Qt 시스템 계층조차 제거하고 가장 가벼운 POSIX 스레드(pthread) 를 날 것으로 띄워 무한 루프 스풀링을 진행한다.

예를 들어, 매우 엄격한 실시간성을 띄는 특정 루프 내부 메커니즘은 아래와 같다.

  • pthread_create() 로 스레드를 분리한다.
  • std::this_thread::sleep_until() 혹은 nanosleep() 등을 사용해 틱(Tick)을 맞춘 무한 while(true) 루프를 동작시킨다.
  • 조건 변수(Condition Variable, pthread_cond_t)를 사용해 I/O 인터럽트를 대기하다가 신호가 떨어지면 즉각(Zero-latency) 반응하는 콜백 로직을 구사한다.

이 구조는 QGC와 ROS 2 브리지(Bridge) 미들웨어 간의 통신(micro-ROS, uXRCE-DDS)을 중계하는 지상 제어 PC의 하위 데몬(Daemon) 프로세스 등에서도 공통적으로 차용하는 핵심적 로우 레벨 최적화 패턴이다.

5. 결론

QGroundControl이 여러 개의 PX4 드론을 동시에 관제함에 있어서, 그 뚝심 있는 성능은 바로 보이지 않는 곳에서 쉬지 않고 돌아가는 수십 개의 QThread 기반 이벤트 루프와 QTimer 콜백 시스템에 의해 지탱된다. 고성능이 요구되는 병목 지점에서는 기저 운영체제의 시스템 리눅스 표준인 POSIX Pthreads 기반의 마이크로 단위 하드 리얼타임 루프가 개입한다. 이 이중적인 비동기 스레딩 모델 결합은 GUI 렌더링 프레임 방어를 필수로 하는 GCS 개발 엔지니어링의 정석(Best Practice)이라 할 수 있다.
