49.8. 비행 데이터 분석(Flight Data Analysis)을 통한 Offboard 제어 성능 평가 및 디버깅

지상 관제 시스템(GCS)이나 보조 컴퓨터(Companion Computer)에서 인가하는 오프보드(Offboard) 제어 명령은 기체의 역동적인 기동과 직결되므로, 제어 성능에 대한 치밀한 평가와 디버깅(Debugging)은 자율 비행 시스템 개발에 있어 필수적인 과정이다. PX4-Autopilot은 이 과정에서 발생하는 방대한 비행 데이터를 시스템 내부적으로 정밀하게 기록하는 고도화된 로깅 아키텍처를 제공한다. 본 절에서는 비행 데이터 분석을 통해 Offboard 제어의 성능을 수치적으로 평가하고, 잠재적인 오류를 추적하여 시스템을 안정화하는 기법에 대해 상세히 다룬다. (참고 기준: PX4-Autopilot 펌웨어 v1.14 이상, QGroundControl v4.3 이상)

1. 비행 데이터 분석의 중요성과 로깅 체계

Offboard 모드에서는 외부의 자율 에이전트(Autonomous Agent) 시스템이 MAVLink 메시지(예: SET_POSITION_TARGET_LOCAL_NED) 또는 ROS 2 uXRCE-DDS 클라이언트를 통해 trajectory_setpoint 토픽을 높은 빈도(10Hz ~ 50Hz)로 전송한다. 이 과정에서 발생하는 통신 지연, 메시지 유실, 혹은 제어기의 동역학적 한계로 인한 추종 오차(Tracking Error)를 면밀히 분석하기 위해서는 블랙박스 형태의 비행 로그가 요구된다.

PX4-Autopilot은 이를 위해 ULog 포맷을 표준으로 채택하고 있다. ULog는 시스템의 uORB 메시지를 시계열(Time Series) 데이터로 직접 직렬화(Serialization)하여 저장하므로, 제어 지령(Setpoint)과 실제 상태(State) 간의 미세한 위상 지연(Phase Lag)이나 진폭 감쇠(Amplitude Attenuation)를 주파수 및 시간 영역에서 분석할 수 있는 기반을 제공한다.

1.1 Ardupilot과의 로깅 체계 비교

PX4-Autopilot의 ULog 시스템은 uORB(Micro Object Request Broker) 아키텍처에 깊이 통합되어 있어, 단순히 정해진 변수들을 기록하는 Ardupilot의 DataFlash 로그 포맷 대비 더욱 유연한 구조를 지닌다. Ardupilot의 경우 LOG_BITMASK 파라미터를 통해 특정 범주의 데이터 기록 여부를 결정하지만, PX4는 logger 모듈이 설정된 프로파일(예: default, high-rate)에 따라 uORB 토픽 단위로 로깅 주파수와 대상을 결정한다. 이러한 토픽 기반 로깅은 ROS 2의 “rosbag” 기록 방식과 개념적으로 유사하며, 개발자가 사용자 정의 uORB 토픽을 추가하더라도 별도의 로깅 코드 수정 없이 손쉽게 데이터를 수집할 수 있다는 강력한 이점을 제공한다.

2. Offboard 제어 데이터 흐름과 로깅 아키텍처

로깅 체계가 어떠한 방식으로 Offboard 제어 지령을 기록하는지 이해하기 위해 데이터의 흐름을 시각화하면 다음과 같다.

sequenceDiagram
    participant Companion as Companion Computer (ROS2/MAVSDK)
    participant MAVLink_uXRCE as MAVLink / uXRCE-DDS 인터페이스
    participant Commander as PX4 Commander (모드 판단)
    participant FlightController as 비행 제어기 (ECL/Position Controller)
    participant Logger as PX4 Logger 시스템
    
    Companion->>MAVLink_uXRCE: 목표 위치/속도 지령 전송
    MAVLink_uXRCE->>FlightController: trajectory_setpoint (uORB 발행)
    FlightController->>Logger: trajectory_setpoint 로깅
    FlightController->>FlightController: 제어 루프 수행 (PID 계산)
    FlightController->>Logger: vehicle_local_position 등 (현재 상태 로깅)
    Commander->>Logger: vehicle_status (Offboard 모드 상태 기록)
    Logger->>Logger: SD 카드에 ULog 포맷으로 기록

위의 도표에서 확인할 수 있듯이, 외부에서 인가된 제어 명령은 최종적으로 trajectory_setpoint 등의 uORB 토픽으로 변환되며, Logger 모듈은 이 토픽을 구독하여 SD 카드에 저장한다. 동시에 EKF2 기반의 추정기(Estimator)가 산출한 실제 기체 상태인 vehicle_local_position 혹은 vehicle_local_velocity 토픽 역시 저장되므로, 두 데이터 간의 상관관계를 통해 성능을 평가할 수 있다.

3. 성능 평가 및 디버깅 기법

Offboard 제어의 성능을 평가하기 위한 주된 지표는 추종 오차(Tracking Error)와 시지연(Time Delay)이다. 비행 데이터를 분석하여 다음의 과정으로 문제점을 디버깅한다.

3.1 ) 응답 지연 및 상승 시간 분석

MAVLink나 ROS 2를 통해 새로운 위치 지령을 전송했을 때, 기체가 목표 지점을 향해 가속을 시작하기까지의 지연(Delay)과, 목표 속도의 10%에서 90%에 도달하는 상승 시간(Rise Time)을 분석한다. 비행 기록 분석 도구(Flight Review 등)를 통해 trajectory_setpoint.x (지령값)와 vehicle_local_position.x (실제값)의 시계열 그래프를 겹쳐보면 위상 차이를 명확히 파악할 수 있다. 오차가 과도할 경우, MAVLink 네트워크의 대역폭 문제나 오프보드 제어 루프의 타이밍 이슈를 의심해야 한다.

3.2 ) 제어 대역폭(Control Bandwidth) 평가

보조 컴퓨터에서 사인파(Sine wave) 형태의 궤적 지령 주파수를 점진적으로 증가시키며 인가하는 주파수 스윕(Frequency Sweep) 테스트를 통해 시스템의 제어 한계 대역폭을 측정할 수 있다. 보데 선도(Bode Plot) 상에서 크기(Magnitude)가 -3dB로 감쇠하거나 위상(Phase) 지연이 -90도를 초과하는 지점을 찾아 기체가 추종할 수 있는 동역학적 한계를 정량화한다. 이러한 평가는 빠른 회피 기동이 요구되는 군집 비행(Swarm Flight)이나 장애물 회피(Obstacle Avoidance)에서 매우 중요한 성능 지표가 된다.

3.3 ) Offboard 진입 및 이탈(Dropout) 추적

Offboard 모드는 지속적인 하트비트(Heartbeat) 및 유효한 제어 지령(Timeout 통상 0.5초)이 유지되어야만 상태를 유지할 수 있다. 비행 중 갑자기 Offboard 모드가 해제되고 기본 위치 유지 모드(Position Mode) 혹은 복귀 모드(RTL)로 전환되었다면, vehicle_status.nav_state의 변경 시점과 MAVLink 수신 주기를 로깅 데이터를 통해 대조해야 한다. 시스템의 CPU 부하, 통신 채널의 간섭, 프로세스 레벨의 경쟁 조건(Race Condition) 등이 주요 원인일 수 있다.

4. 요약 및 실무 적용

비행 데이터 분석(Flight Data Analysis)은 단순한 사후 검증이 아닌 오프보드(Offboard) 알고리즘의 최적화를 위한 필수적인 피드백 루프이다. 개발자는 PX4의 포괄적인 ULog 데이터 시스템과 제공되는 pyulog 시각화 라이브러리, 또는 Flight Review 웹 도구를 적극 활용하여 시스템의 입출력 특성을 파악해야 한다. 수학적이고 공학적인 접근을 통한 객관적인 데이터 분석만이 고도의 신뢰성을 요구하는 자율 에이전트 시스템을 완성하는 핵심 경쟁력이다.
