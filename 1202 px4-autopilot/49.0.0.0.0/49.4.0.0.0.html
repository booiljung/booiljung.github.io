49.4 주기적 제어 지령 전송을 위한 네트워크 통신 이론 및 지연 보상 최적화

1. 개요: Offboard 제어와 실시간 네트워크 통신의 필요성

PX4-Autopilot에서 가장 진보된 자율 비행(Autonomous Flight) 제어 방식인 Offboard 모드는 지상 관제 시스템(GCS) 또는 동반 컴퓨터(Companion Computer)가 직접 기체의 위치, 속도, 또는 자세 제어 지령(Setpoint)을 실시간으로 계산하여 비행 제어기(Flight Controller)에 주입하는 아키텍처이다. 이 과정에서 제어 지령은 무선 데이터 링크(예: Wi-Fi, LTE, RF Telemetry)를 통해 전송되는데, 무선 통신 환경의 본질적인 특성상 지연(Latency), 지터(Jitter), 패킷 손실(Packet Loss)이 필연적으로 발생한다.

비행 제어 시스템과 같은 폐루프 제어계(Closed-loop Control System)에서 통신 지연은 제어기의 위상 여유(Phase Margin)를 감소시켜 시스템을 발산(불안정)하게 만드는 주된 요인이다. 따라서 주기적인 지령 전송 채널의 물리적 한계를 수학적으로 모델링하고, 이를 보상(Compensation)하기 위한 최적화 기법을 적용하는 것은 고성능 자율 비행을 달성하기 위한 공학적 필수 과제이다.

2. 네트워크 지연의 이론적 모델링

지상국에서 비행 제어기로 지령이 도달하기까지의 총 지연 시간 T_d는 다음과 같이 모델링할 수 있다.

T_d = T_{proc\_tx} + T_{trans} + T_{prop} + T_{proc\_rx}

  • T_{proc\_tx}: 지상국 또는 동반 컴퓨터에서 제어 알고리즘을 연산하고 MAVLink 또는 ROS2 메시지로 패키징하는 처리 지연
  • T_{trans}: 통신 인터페이스(UART, Ethernet) 병목에 의한 전송 지연
  • T_{prop}: 무선 전파 공간을 이동하는 물리적 전파 지연 (빛의 속도에 종속되나 무선 모뎀의 변복조 처리 시간 포함)
  • T_{proc\_rx}: PX4-Autopilot이 메시지를 수신하여 uORB 토픽으로 발행하기까지의 처리 지연

특히 T_{trans}와 T_{prop}는 네트워크 트래픽이나 무선 간섭에 의해 확률적으로 변동(지터, Jitter)하므로, 수학적으로 볼 때 지연 시간 T_d는 확률 밀도 함수를 갖는 확률 변수(Random Variable)로 취급되어야 한다. TCP(전송 제어 프로토콜)는 신뢰성을 보장하기 위해 패킷 손실 시 재전송을 수행하여 T_d를 무한대에 가깝게 증폭(Head-of-Line Blocking)시킬 수 있으므로, Offboard 제어 지령 전송에는 원칙적으로 UDP(사용자 데이터그램 프로토콜) 와 같은 비연결지향 프로토콜을 사용한다.

3. PX4-Autopilot의 지연 보상 및 안전 아키텍처

PX4-Autopilot은 이러한 통신 지연 및 단절에 대응하기 위해 다음과 같은 아키텍처 수준의 안전 장치 및 보상 로직을 구현하고 있다.

3.1. Offboard 모드 안전성 검증 (Offboard Timeout)

PX4의 commander 모듈은 Offboard 제어 지령(예: SET_POSITION_TARGET_LOCAL_NED)이 일정 주기 이상 수신되지 않으면 네트워크에 심각한 지연이나 단절이 발생한 것으로 간주하고 즉각적인 안전 조치(Failsafe)를 취한다.

  • PX4는 최소 2Hz(0.5초) 주기로 지령이 갱신되어야 Offboard 모드를 유지할 수 있다. 파라미터 COM_OF_LOSS_T를 통해 이 타임아웃을 설정하며, 타임아웃 발생 시 자동으로 Hold 또는 Return 모드로 전환하여 기체의 추락을 방지한다.
  • 안정적인 제어를 위해서는 통신 주기(Control Rate)를 기하학적 제어기의 시상수(Time Constant)보다 훨씬 짧은 50Hz ~ 100Hz로 유지하는 것이 권장된다.

3.2. 시간 동기화(Time Synchronization)

지연 보상의 첫걸음은 송신측(Companion Computer)과 수신측(PX4) 간의 정확한 시간 동기화이다.

  • MAVLink 통신 시 TIMESYNC 메시지를 교환하여 두 시스템의 시계(Clock) 오프셋과 왕복 지연 시간(RTT, Round Trip Time)을 동적으로 추정한다.
  • ROS2 uXRCE-DDS 미들웨어를 사용하는 경우, Micro XRCE-DDS 클라이언트 자체의 시간 동기화 프로토콜을 바탕으로 uORB 메시지 타임스탬프(timestamp 및 timestamp_sample)를 일치시켜 시스템 내부의 EKF2(상태 추정기)가 해당 지령이 언제 생성되었는지 정확히 인지할 수 있도록 한다.

3.3. 예측 제어 및 보상 알고리즘 적용

실제 고도화된 시스템에서는 통신 지연 T_d를 보상하기 위해 지상국 측에서 스미스 예측기(Smith Predictor) 나 통신 지연을 고려한 모델 예측 제어(MPC, Model Predictive Control) 알고리즘을 채택한다. 지상 관제 시스템은 기체의 현재 상태가 아니라, 지연 시간 T_d 후 기체가 도달할 미래의 상태를 예측하여 그 시점에 맞는 지령을 선제적으로 전송(Feedforward)함으로써 위상 지연으로 인한 불안정성을 상쇄시킨다.

    GCS->>GCS: 위치/속도 제어 지령 계산

    PX4->>PX4: Commander 타임아웃 체크 (< 0.5s)
    PX4->>PX4: uORB (trajectory_setpoint) 퍼블리시
    PX4->>PX4: PID Control Loop 추종

4. Ardupilot 대비 통신 제어 아키텍처의 차이점

Ardupilot의 경우 Guided 모드를 통해 지상국으로부터 제어 명령을 수신한다. 두 시스템 모두 타임아웃 메커니즘을 지원하지만 설계 최적화 방향에서 차이가 있다.

  • PX4-Autopilot: ROS 및 ROS2 기반 시스템 연동을 염두에 두고 설계되어, 고주파수(Offboard Stream) 실시간 스트리밍에 대해 매우 엄격한 uORB 동기화 및 타임스탬프 관리를 수행한다. XRCE-DDS(PX4 v1.14 이상)를 이용해 외부 네트워크를 사실상 내부 uORB 네트워크의 연장선으로 취급하는 데이트 구조를 지닌다.
  • Ardupilot: 전통적으로 MAVLink 기반 명령 처리에 더 높은 비중을 두며, Guided 모드의 내부 상태 머신이 비교적 추상적으로 다뤄져 통신 지연에 대한 파라미터 미세 조정보다는 범용 제어 루프의 강건성에 주로 등기대어 대응한다.

5. 결론 및 공학적 시사점

실무에서 Offboard 제어를 구현할 때 단순히 코딩적인 반복문 루프 스피드를 올리는 것만으로는 지연 문제를 해결할 수 없다. 네트워크 통신 이론에 근거하여 운영 주파수를 결정하고, 패킷 손실에 대비한 타임아웃 튜닝 메커니즘을 구축하며, 필요에 따라 예측 제어 필터를 통한 선제적 보상을 동반해야만, 고속 회피 기동이나 정밀 대형 유지 비행(Swarm Flight)에서 요구하는 극도의 추종 성능을 확보할 수 있다. QGroundControl 혹은 독자적 GCS 개발 시 이러한 확률 통계적 지연 모델을 반드시 소프트웨어 구조 설계에 반영해야 한다.
