49.4.2 클록 동기화(Clock Synchronization) 및 타임스탬프(Timestamp) 오프셋 보정 기술

1. 개요(Introduction)

지상 관제 시스템(GCS, Ground Control Station)과 무인항공기 탑재 비행 제어기(PX4-Autopilot)는 물리적으로 완전히 독립된 시스템이므로, 각기 고유한 하드웨어 오실레이터(Hardware Oscillator) 및 운영체제 클록(OS Clock)을 바탕으로 구동된다. Offboard 궤적 제어나 고정밀 외부 센서 융합(Sensor Fusion), 중앙 집중형 다중 에이전트(Swarm) 분산 시스템 통신 환경에서는 양단의 시간 기준선(Time Reference)을 엄격하게 일치시키는 과정이 필수적이다.
이를 클록 동기화(Clock Synchronization)라고 총칭하며, 두 시스템 간의 상대적인 시간 격차를 나타내는 타임스탬프 오프셋(Timestamp Offset)을 정확히 연산하여 보정(Compensation)해야만 통신 채널 트랜잭션의 지연(Latency) 정량화와 시계열 데이터(Time-series Data)의 인과적 유효성 검증이 가능해진다.

2. PX4-Autopilot 내 타임스탬프 오프셋 보정의 당위성

  1. 단방향 지연 시간(One-way Latency) 연산: 수신된 패킷(Setpoint, Telemetry 등)이 무선 트랜시버 네트워크를 통과하여 도달하기까지 소요된 물리적인 실제 시간을 계산하기 위해서는 양단 시스템의 논리적 계시(Timekeeping)가 동기화되어 있어야 한다.
  2. 센서 데이터 및 좌표계 매핑(Mapping): 비전 기반 RTK-GPS 시스템, Mocap(Motion Capture), VIO(Visual Inertial Odometry) 등의 외부 기구학 데이터(External Vision/Odometry)가 PX4 시스템 내부의 EKF2(Estimator) 상태 방정식으로 융합될 때, 타임스탬프 불일치로 인한 오염된 데이터 입력은 상태 추정기의 발산을 초래하는 치명적 에러 인자로 작용한다.
  3. 로깅 및 사후 분석(Flight Data Analysis): 기체 내부에 저장되는 ULog 파일은 기반 RTOS(NuttX) 커널의 부트 타임(Boot Time)을 기점(t=0)으로 마이크로초(microseconds) 단위 단조 증가 타임스탬프로 로깅된다. GCS의 임무 기록 시스템(Mission Log)이나 비디오 스트리밍 텔레메트리와 상관 분석(Correlation Analysis)을 수행하기 위해서는 글로벌 타임스탬프 오프셋 사상(Mapping) 변환이 선결되어야만 한다.

3. MAVLink TIMESYNC 프로토콜 기반 오프셋 산출 원리

PX4 펌웨어 및 QGroundControl(v4.3 기준) 표준 관제 통신 생태계에서는 MAVLink의 TIMESYNC (메시지 ID #111) 메시지를 상호 간 주기적으로 교환하여 시스템 간 시간 오프셋과 왕복 복귀 시간(RTT, Round Trip Time)을 측정한다.

3.1 TIMESYNC 메시지 규격

TIMESYNC 메시지는 시간 동기화를 위한 두 개의 64비트(uint64) 핵심 필드로 구성된다.

  • tc1 (Time Component 1): 송신자가 인지하고 있는 상대 타겟 시스템의 시스템 시간 (초기 질의 시 0으로 채워짐, 응답 패킷에서 랩어라운드됨)
  • ts1 (Time Stamp 1): 송신자 본연의 현재 시스템 시간

3.2 오프셋(Offset) 수학적 산출 모델

GCS가 PX4(Autopilot)로 타임싱크 핑(Ping)을 전송하고, PX4가 MAVLink 스레드를 거쳐 GCS로 응답(Pong)하는 트랜잭션 처리 과정을 통해 오프셋을 역산한다. 타임스탬프 샘플링 지점은 다음과 같다.

  • T_{GCS, tx}: GCS의 TIMESYNC 송신 시점
  • T_{PX4, rx}: PX4 패킷 수신 시점
  • T_{PX4, tx}: PX4 응답 패킷 송신 시점
  • T_{GCS, rx}: GCS 응답 패킷 최종 수신 시점

트랜잭션에 소요된 왕복 지연 시간(RTT)은 통신 링크가 상하향 대칭적(Symmetric)이라고 가정하는 NTP(Network Time Protocol) 모델을 차용하여 다음과 같이 근사된다.
RTT = (T_{GCS, rx} - T_{GCS, tx}) - (T_{PX4, tx} - T_{PX4, rx})

GCS를 기준으로 한 PX4의 클록 오프셋 추정치(Offset)는 단방향 전파 지연 \frac{RTT}{2} 를 보완 항으로 고려하여 아래와 같이 계산된다.
Offset = T_{PX4, rx} - T_{GCS, tx} - \frac{RTT}{2}

이때 1회성 트랜잭션으로 구해진 난수 성격의 Offset은 네트워크 지터(Jitter)를 그대로 포함하고 있다. 따라서 GCS 백엔드 컴포넌트는 해당 값을 지수 이동 평균(EMA, Exponential Moving Average)이나 선형 칼만 필터(Kalman Filter)와 같은 평활화(Smoothing) 알고리즘을 거쳐 분산을 억제한 최종 오프셋 값으로 정제하여 동기화 메커니즘에 적용한다.

5. 애플리케이션 레벨 정밀 클록 동기화의 구조적 한계와 차이점

  • 근본적 한계 분석: MAVLink 수준의 상위 TIMESYNC 메커니즘은 L7 수준의 애플리케이션 계층(Application Layer) 데몬에서 동작하므로, 호스트 OS의 스케줄링 간섭, 시리얼 포트 버퍼 언더런/오버런, Wi-Fi 매체 접근(MAC) 딜레이 등 하위 계층에서 발생한 비결정론적 네트워크 지연 시간 변동(Jitter)에 사실상 무방비로 노출된다는 구조적 한계를 지닌다. IEEE 1588 규격과 같은 하드웨어 타임스탬핑에는 미치지 못한다.
  • Ardupilot과의 관점 비교: Ardupilot의 SYSTEM_TIME 메시지 방송(Broadcast) 방식과 PX4의 TIMESYNC 폴링(Polling) 방식은 개념의 뿌리를 공유한다. 그러나 PX4 생태계는 MAVROS, microRTPS(PX4 v1.13 이전) 및 uXRCE-DDS(PX4 v1.14 이후)와 같은 이기종(Heterogeneous) 통신 브리지 확장 환경에서 동작하는 것을 전제로 설계되었기에 수신 시스템의 타임스탬프 스무딩, 필터링 로직 분리가 더 명시적이다. 특히 ROS2-PX4 연동 환경에서는 TIMESYNC를 우회하여 XRCE-DDS 미들웨어 내부에 자체 탑재된 XRCE Time Sync Agent를 브릿지 레벨에서 직접 연계함으로써 보다 깊이 있는 시간축 일원화를 도모하는 설계 철학적 진보를 보여준다.

6. 결론

비동기 통신 채널에 의존하는 GCS-무인기 비행 제어 아키텍처에서 타임스탬프 클록 오프셋에 대한 방관은 시스템의 제어 위상(Phase) 지연과 센서 융합 관측기의 수학적 발산으로 직결된다. GCS의 통신 프로토콜 스택 개발 관점에서는 필연적으로 TIMESYNC 등의 동기화 루틴을 구현하여, MAVLink 네트워크 상의 PX4 내부 시간(NuttX Uptime)과 외부 제어 컴퓨터의 Wall-clock Time 간의 오프셋을 철저하게 추종 및 필터링 관리해야 한다. 이는 신뢰성 높은 자율 에이전트(Autonomous Agent) 운용의 근본 인프라 전제 조건임을 주지하여야 한다.
