49.1.2 비행 상태 추정기(State Estimator)의 공분산(Covariance) 행렬 유효성과 Offboard 제어 권한의 상관관계

PX4-Autopilot에서 오프보드(Offboard) 제어 모드는 외부 임무 컴퓨터(Mission Computer)나 지상 관제 시스템(GCS)이 설정값(Setpoint)을 실시간으로 기체에 전송하여 비행을 제어하는 방식이다. 이러한 외부 제어는 기체의 상태 추정기(State Estimator)가 제공하는 위치(Position), 속도(Velocity), 자세(Attitude) 데이터에 전적으로 의존한다. 본 섹션에서는 PX4 펌웨어(v1.14 기준)의 EKF2(Extended Kalman Filter)가 산출하는 공분산(Covariance) 행렬의 유효성이 오프보드 제어 권한(Control Authority)의 획득 및 유지에 어떠한 상관관계를 가지는지 상세히 분석한다.

1. 비행 상태 추정기(State Estimator)와 EKF2의 역할

PX4는 기본 상태 추정기로 EKF2 모듈을 사용한다. EKF2는 관성 측정 장치(IMU), 나침반(Magnetometer), 기압계(Barometer), GPS(Global Positioning System) 및 시각 관성 오도메트리(VIO) 등의 센서 데이터를 융합(Fusion)하여 기체의 현재 상태를 추정한다.

상태를 역학적으로 완벽히 공간상에 매핑하기 위해 상태 추정기는 현재 상태 추정치와 함께 그 **불확실성(Uncertainty)**을 나타내는 **공분산 행렬(Covariance Matrix)**을 지속적으로 계산한다.

1.1 공분산 행렬의 물리적 의미

공분산 행렬의 대각 성분(Diagonal Elements)은 각 상태 변수(예: x축 위치, y축 속도 등)의 분산(Variance)을 의미하며, 비대각 성분(Off-diagonal Elements)은 상태 변수 간의 상관관계(Correlation)를 나타낸다. 값이 작을수록 해당 추정치에 대한 시스템의 ’자신감(Confidence)’이 높음을 의미하며, 반대로 값이 과도하게 커지면 센서 노이즈나 측정 지연 등으로 인해 추정값을 신뢰할 수 없음을 나타낸다.

2. 오프보드(Offboard) 제어 권한의 전제 조건

오프보드 모드(Offboard Mode)는 외부 시스템(예: ROS2, MAVSDK)에서 기체를 제어하므로, 내부적인 상태 추정에 오류가 있을 경우 그대로 기체가 추락하거나 위험 상태에 빠질 수 있다. 따라서 PX4의 커맨더(Commander) 모듈은 오프보드 모드로의 천이(Transition) 과정에서 엄격한 검증 로직을 수행한다.

오프보드 제어를 시작하기 위해서는 다음의 조건이 충족되어야 한다.

  1. 유효한 위치 및 속도 추정치 확보: 시스템(EKF2)이 현재 로컬(Local) 또는 글로벌(Global) 위치와 속도 추정치에 대해 충분한 신뢰도를 가져야 한다.
  2. 지속적인 지령 수신: 외부 시스템에서 최소 2Hz 이상의 속도로 제어 지령(Setpoint) 스트림이 들어오고 있어야 한다.

3. 공분산 행렬 유효성과 오프보드 제어 권한의 상관관계 알고리즘

PX4 소스 코드 내에서 EKF2의 상태가 유효한지 판별하는 핵심은 혁신(Innovation) 테스트 및 분산(Variance) 임계값 테스트에 있다.

EKF2는 새로운 센서 측정값이 들어올 때마다 예측 모델의 추정치와 실제 측정값 간의 차이인 ’혁신(Innovation)’을 계산한다. 이때 시스템의 오차 공분산 행렬 \mathbf{P} 와 측정 모델을 기반으로 혁신 공분산 행렬 \mathbf{S} 가 산출된다.

\mathbf{S} = \mathbf{H} \mathbf{P} \mathbf{H}^T + \mathbf{R}

여기서 \mathbf{H}는 관측 행렬(Observation Matrix), \mathbf{R}은 측정 시스템 노이즈 공분산(Measurement Noise Covariance) 행렬이다.

3.1 커맨더(Commander)의 거절(Rejection) 로직 매커니즘

만약 특정 축(예: 수평 위치)의 공분산 값이 파라미터 COM_POS_FS_EPH 와 같이 미리 설정된 임계값을 초과하면, PX4 커맨더는 오프보드 모드 전환(Mode Switch) 요청을 즉시 **거부(Reject)**한다.

다음의 인과관계를 따른다.

  • 분산 상승: GPS 수신율 하락 혹은 자기장 교란(Magnetic Interference) 발생.
  • 플래그 상실: EKF2 내부 로직에서 local_position_valid 혹은 global_position_valid 플래그를 false 로 전환.
  • 제어권 박탈: 오프보드 모드로 비행 중이었다면 안전 장치(Failsafe) 발동하여 Position 모드나 Altitude 모드, 최종적으로는 Land 또는 Return 모드로 강제 강등(Fallback) 발생.

아래 예시는 상태 추정기의 공분산 평가를 통한 모드 천이 과정을 보여주는 상태도(State Diagram)이다.

stateDiagram-v2
    [*] --> EKF2_Running
    EKF2_Running --> Evaluate_Covariance : 센서 데이터 융합

    state Evaluate_Covariance {
        direction LR
        Valid(공분산 < 내정 임계값) 
        Invalid(공분산 > 임계값)
    }

    Evaluate_Covariance --> Offboard_Mode : Valid (지령 2Hz 이상 수신)
    Evaluate_Covariance --> Failsafe_Trigger : Invalid (위치 신뢰 상실)

    Offboard_Mode --> Evaluate_Covariance : 루프 반복 모니터링
    Failsafe_Trigger --> Altitude_Mode : 예외 처리 (Fallback)
    Altitude_Mode --> [*]

4. QGroundControl (QGC) 및 MAVLink 기반 상태 모니터링

QGroundControl(v4.3 기준)은 이러한 EKF 상태를 사용자에게 실시간으로 피드백한다.
MAVLink 통신상에서 EKF의 상태는 ESTIMATOR_STATUS (ID: 163) 메시지로 GCS 및 외부 시스템에 전파된다. 이 메시지 안에는 속도, 수평/수직 위치, 나침반 등의 혁신 비율(Innovation Ratio)과 공분산 플래그(Flags)들이 비트마스크(Bitmask) 형태로 포함되어 있다.

QGC 소스 코드를 분석해 보면, 차량(Vehicle) 관리 매니저 클래스가 이 ESTIMATOR_STATUS를 수신할 때, 특정 비트(예: 추정기 결함 플래그)가 세트되어 있으면 즉각적으로 사용자 인터페이스(UI)에 “EKF Position Error” 또는 “Compass Error“와 같은 치명적인 경고(Critical Alert) 레이블을 점멸시킨다. 동시에 오프보드 모드로 전환하려는 MAVLink SET_MODE 또는 COMMAND_LONG 에는 MAV_RESULT_TEMPORARILY_REJECTED 라는 거절 메시지가 회신(ACK)된다.

5. Ardupilot의 Guided 모드와의 차이점 비교 (Ardupilot 대비 차이)

PX4의 오프보드(Offboard) 모드와 유사한 개념으로 ArduPilot에는 가이드(Guided) 모드가 있다.

  • PX4-Autopilot: 공분산 행렬에 기반한 상태 유효성 검사에 대하여 매우 경직되고 수학적으로 엄격한(Strict) 철학을 유지한다. 위치 추정 유효 플래그가 하나라도 일시적으로 해제되면 즉시 위치 제반 모드를 차단하고 강등을 실시한다.
  • ArduPilot: EKF3의 EKF_Failsafe 로직이 존재하나, 특정 설정에서는 “Dead Reckoning(추측 항법)” 등을 일시적으로 과감히 허용하면서 가이드 모드를 유지하려는 관대함(Tolerance)을 제공하는 편이다.

따라서 PX4 환경에서 자율 비행용 ROS2(또는 MAVSDK) 애플리케이션을 개발할 때는, 기체 외부에서 단지 제어 명령을 주기적으로 밀어 넣는 데 국한하지 말고, 지속적으로 uORB의 vehicle_local_position 토픽이나 MAVLink의 ESTIMATOR_STATUS를 구독하여 공분산 및 EKF 혁신율을 모니터링하는 로직(Health Check Loop)을 구현해야 한다.

6. ROS2 연동: Micro XRCE-DDS 퍼블리싱과 제어권 검증

ROS2 환경에서 uXRCE-DDS 브리지를 활용한 오프보드 제어를 할 때, 이 상관관계는 더욱 두드러진다. ROS2 노드(Node)는 OffboardControlMode 메시지와 기하학적 설정값(예: TrajectorySetpoint) 메시지를 PX4 측으로 퍼블리시(Publish)한다.

이때 ROS2 노드는 PX4 측에서 발행하는 EstimatorStatus 메시지를 구독(Subscribe)하도록 시스템 아키텍처를 설계해야 한다. 기체가 터널 구간이나 건물 사이(Urban Canyon)와 같이 GPS 음영 지역에 진입하면 공분산 값이 기하급수적으로 발산한다. 모범적인 자율 에이전트 비행을 위한 제어 기술 설계 시에는 이러한 공분산 발산이 임계값(Threshold)에 도달하기 직전에 외부 컴퓨터(ROS2)가 주도적으로 VIO 데이터의 가중치를 조작하거나 보조 센서(옵티컬 플로우 등) 참조 비중을 재조정하는 로직을 갖추어야만 오프보드 권한의 상실을 예방할 수 있다.

7. 결론

요약하면, PX4-Autopilot의 오프보드 제어 권한은 EKF2 상태 추정기 공분산 행렬의 ’인질’과도 같다. 제어 공학적 측면에서 상태 피드백 제어기의 안정도는 추정된 상태의 불확실성에 묶여 있기 때문이다. 이를 정확히 이해하고 ROS2 코드와 MAVLink 제어 루프를 구성하는 것이 고도화된 스웜(Swarm) 제어나 스마트 자율 비행 에이전트를 개발하는 필수적인 선결 조건이다.
