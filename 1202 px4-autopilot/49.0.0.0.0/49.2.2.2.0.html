49.2.2.2 PX4 내부 0.5초 타임아웃(Timeout) 발생 시의 위치 고수(Loiter) 및 비상 귀환(RTL) 궤적 생성 분기

1. 개요 및 안전 관리 철학

PX4-Autopilot의 Offboard 제어 모드에서 비행 중, GCS 또는 컴패니언 컴퓨터(ROS2 노드 등)로부터 수신되던 세트포인트(Setpoint) 지령이 끊기면 앞서 다룬 워치독 감시 체계에 의해 타임아웃(Timeout, 기본 파라미터 COM_OF_LOSS_T = 0.5초) 이벤트가 발생한다.

타임아웃은 단순한 소프트웨어 경고(Warning) 플래그를 띄우는 것으로 끝나지 않는다. 0.5초라는 찰나의 순간에 기체는 즉각 기존의 추종 궤적을 폐기하고, 물리적인 추락이나 비정상적인 폭주를 막기 위해 스스로 새로운 **안전 궤적(Failsafe Trajectory)**을 생성하는 단계로 진입해야 한다. 이 장에서는 워치독 트리거 직후에 발생하는 두 가지 주요 분기 경로인 **위치 고수(Loiter/Hold)**와 비상 귀환(RTL, Return To Launch) 시의 내부 궤적 생성 메커니즘을 상세히 분석한다.

2. 모드 강등 분기(Mode Branching) 파이프라인

타임아웃이 발생하면 Commander 모듈은 내부 유한 상태 머신(FSM)을 통해 기체가 어떤 생존 기동을 취해야 할지 파라미터 설정을 기반으로 평가(Evaluation)한다. 주로 영향을 미치는 파라미터는 COM_OBL_ACT(Offboard Loss Action)와 COM_OBL_RC_ACT(RC 사용 시 Offboard 상실 시 액션)이다.

이 파라미터들의 설정값에 따라 시스템은 크게 두 가지 궤적 생성 모듈인 FlightTask 엔진으로 제어권을 넘긴다.

2.1 제동 및 위치 고수(Loiter / Hold) 분기 메커니즘

외부 제어 지령이 상실되었을 때 가장 기본적이고 안전한 일차적 대응은 “기체가 현재 위치에 멈춰 서서 다음 지시를 기다리는 것(Hold)“이다.

  • 관성 제동(Kinematic Braking): 외부로부터 속도 지령 v_{target}을 받아 전진 중이었을 때 지령이 끊기면, 단순히 모터 출력을 줄이는 것이 아니라 내부의 FlightTaskAutoMapper가 속도를 0으로 수렴시키는 부드러운 감속 궤적을 파생시킨다.
  • Bumpless Transfer: 갑자기 위치 고수 모드로 전환 시, 타임아웃 직전까지 기체가 가지고 있던 관성과 목표 가속도 간의 차이로 인해 덜컹거리는 서지(Surge)를 막기 위해 Jerk-limited(저크 제한) 제동 곡선 알고리즘이 적용된다.
  • 제어 루프 이관: 외부 세트포인트(TrajectorySetpoint) 구독이 중단되고, 내부의 Position Controller가 현재 상태 추정치(Estimator로부터 받은 EKF2 위치)를 기반으로 고정된 공간 좌표 좌표(x_{hold}, y_{hold}, z_{hold})를 생성하여 호버링(Hovering)을 유지한다.

2.2 비상 귀환(Return / RTL) 분기 메커니즘

통신 두절이 장기화될 경우를 대비하거나, 처음부터 적극적인 모 보전(Safekeeping)을 하도록 파라미터를 설정한 경우 기체는 Return 모드로 분기한다. Return 궤적 생성은 다음의 엄격한 시퀀스(Sequence)를 따른다.

  1. 상승 반경(Climb to Return Altitude): 현재 기체의 고도가 파라미터 RTL_RETURN_ALT보다 낮다면, 충돌 회피를 위해 제자리에서 해당 고도까지 수직으로 상승(Climb)하는 궤적이 먼저 계산된다.
  2. 이동(Cruise to Home): 목표 고도에 도달하면 GCS 연동 초기 설정된 홈 위치(Home Position)의 위/경도 좌표로 직선 수평 비행(Cruise) 궤적을 잇는다.
  3. 강하 및 착륙(Descend & Land): 홈 위치 상공에 도달한 뒤, RTL_DESCEND_ALT까지 1차 강하 이후 최종 랜딩 시퀀스를 수행한다.

Commander가 main_state를 MAIN_STATE_HOLD나 MAIN_STATE_RETURN으로 플래그 스위칭만 하면, 실질적인 비행 동역학적 경로 계산은 위치 제어 스택 내부의 FlightTask 클래스들(src/modules/flight_mode_manager/tasks)이 객체지향적 다형성(Polymorphism)을 활용하여 즉각적으로 자체 궤적(Velocity/Position Setpoint)을 내뿜도록 설계되어 있다.

4. Ardupilot 대비 아키텍처적 고찰

Ardupilot은 전통적으로 Guided 모드 상실 시 Failsafe 로직이 비교적 단일한 흐름(예: 무조건 RTL로 전환되거나 파라미터가 요구하는 단순 행동 취합)을 유지하지만, PX4는 FlightTask라는 비행 태스크(Task) 기반의 모듈식 아키텍처를 채택하였다. 이로 인해 Timeout 발생 직후의 궤적 제어 전환이 훨씬 매끄러우며, 개발자가 FlightTask 클래스를 상속받아 커스텀(자체 규칙 기반의) 비상 회피 궤적 로직을 플러그인(Plug-in) 형태로 손쉽게 확장할 수 있다는 것이 아키텍처 상의 가장 큰 관제 차이점이다.

5. 결론 및 관제 기술 응용

오프보드 제어를 활용한 자율 에이전트 개발자는 이 0.5초 직후에 발동되는 내부 궤적 생성 분기 로직을 투명하게 이해해야 한다. 일시적인 통신 지연이나 핑(Ping) 튀김 현상으로 의도치 않게 기체가 제동을 걸거나 상승 로직을 발동하면 자율 주행 임무 시간과 연료(배터리) 낭비가 초래될 수 있기 때문이다. GCS 사이드에서는 기체의 HEARTBEAT 상태와 통신 품질(Link Quality)을 모니터링하여, 기체가 Failsafe 궤적에 진입하기 전에 스스로 속도를 능동적으로 줄이도록 하거나 복구 알고리즘(Retry Handling)을 선행 설계하는 것이 바람직하다.
