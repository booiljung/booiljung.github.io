49.5.3. SET_ATTITUDE_TARGET 메시지를 이용한 내부 자세 제어 루프(Inner Attitude Loop) 직결 기법

무인항공기의 제어 아키텍처는 보편적으로 위치 및 속도를 관장하는 외부 루프(Outer Loop, Navigation Loop)와 기체의 유체역학적 기울기 및 각속도를 제어하는 내부 루프(Inner Loop, Attitude Loop)의 캐스케이드(Cascade) 형태로 구성된다. 일반적인 MAVLink SET_POSITION_TARGET 기반의 Offboard 제어는 외부 루프에서부터 시작하여 내부 루프까지 모든 동역학 연산을 비행 제어기(FC; Flight Controller) 내부 알고리즘에 일임한다. 반면, 복잡한 비선형 제어(Non-linear Control) 연구나 신경망 강화학습(RL; Reinforcement Learning) 기반의 궤적 생성 모델을 동반 컴퓨터(Companion Computer)에 탑재할 경우, 외부 루프 연산 결과물인 **목표 자세(Attitude)**와 추력(Thrust) 스칼라를 펌웨어의 내부 자세 제어기에 “직결(Direct Injection)“하는 기법이 필수적으로 대두된다.

본 절에서는 SET_ATTITUDE_TARGET 메시지를 활용하여 PX4-Autopilot의 위치 제어망을 우회(Bypass)하고, 내부 자세 제어 루프를 직접 타격하는 오프보드 통신 기법과 그 백엔드 아키텍처를 상세히 분석한다. 서술 기준 펌웨어는 PX4 v1.14 아키텍처 및 QGroundControl v4.3, 통신 규약은 MAVLink v2를 상정한다.

1. SET_ATTITUDE_TARGET MAVLink 메시지 구조와 비트마스크

해당 메시지는 드론의 3차원 자세 목표치를 오일러 각의 짐벌 락(Gimbal Lock) 특이점을 극복한 쿼터니언(Quaternion) 수학 모델로 수신하며, 부가적으로 기축 단위의 각속도(Angular Rate)와 상대적으로 정규화된 추력(Thrust) 데이터를 일괄 전달한다. 메시지의 주요 페이로드(Payload) 구성 프레임은 다음과 같다.

  • q (float[4]): 공간상 목표 절대 자세를 나타내는 쿼터니언 배열 [w, x, y, z]. 동역학 시뮬레이션 및 ROS2 융합 환경에서 표준적으로 사용된다.
  • body_roll_rate, body_pitch_rate, body_yaw_rate (float): 기축(Body Frame) 기준의 목표 롤, 피치, 요 각속도(단위: 라디안/초, [rad/s]).
  • thrust (float): 0.0 에서 1.0 사이로 정규화(Normalized)된 총 추력 지령치. (0.0은 모터 아이들링 혹은 정지, 1.0은 배터리 한계 허용 최대 추력을 의미함)
  • type_mask (uint8_t): 8비트 길이의 플래그 마스크로, 특정 지령 벡터를 무시하는 네거티브 논리(Negative Logic) 스위치 역할을 수행한다.

1.1 비트마스크(Type Mask) 논리 할당

  • 비트 0 (0x01): IGNORE_ROLL_RATE (롤 각속도 무시)
  • 비트 1 (0x02): IGNORE_PITCH_RATE (피치 각속도 무시)
  • 비트 2 (0x04): IGNORE_YAW_RATE (요 각속도 무시)
  • 비트 6 (0x40): IGNORE_ATTITUDE (자세 쿼터니언 변수 무시, 오직 각속도만 통제할 때 쓰임)
  • 비트 7 (0x80): IGNORE_THRUST (추력 무시 - 단, 수직 추력이 생략될 경우 기체 추락을 초래하므로 실제 비행에서 활용성은 극히 제한적)

통상적인 외부 자세 루프 제어에서는 각속도를 무시 비트(0x07)로 세팅하여 드론 내부 각속도 연산 PID에 의해 P이득으로 각속도가 자동 도출되도록 하거나, 고등 제어기에서는 자세 배열(q)과 피드포워드 각속도를 모두 활성화(Mask=0x00)하여 극한의 고주파 추종 성능을 도모한다.

2. PX4 아키텍처 내부의 직결 처리 경로 (Bypass Path)

관제 명령이 텔레메트리나 직렬 포트(Serial Port)를 통해 수신되면 PX4의 수신 데몬(mavlink_receiver.cpp)은 메시지 구조체를 파싱(Parsing)하여 vehicle_attitude_setpoint (자세) 스레드와 vehicle_rates_setpoint (각속도) uORB 토픽으로 즉각 분산 발행(Publish)한다.

2.1 위치 제어망의 완전한 배제 (Navigation Loop Decoupling)

이 구조적 메시지가 인가된 직후 기체가 Offboard 비행 모드를 점유하고 있다면, 최상단의 PositionControl 혹은 FlightTask 서브루틴 연산은 철저하게 **침묵(Bypass)**한다. 외부 지점(Waypoint)에서 연산해야 할 방위 및 위치 오차 PID 프로세스를 GCS 백엔드의 CPU나 외부 ROS2 에이전트의 신경망이 모두 대신 처리했으므로, 펌웨어는 내부 컴퓨팅 자원을 대폭 절약함과 동시에 최소 레이턴시(Zero-latency)로 들어오는 쿼터니언 벡터를 내부 자세 제어 블록(mc_att_control 혹은 mc_rate_control)에 시냅스 꽂듯 꽂아 넣는다. 이 무결성 기법은 공학 수학적으로 보장된 50Hz ~ 100Hz 이상의 고주파수 지령 인가를 필수 생존 전제로 삼는다.

3. C++ 및 MAVSDK를 통한 직결 제어망 구현 예시

외부 고성능 컴퓨터(예: NVIDIA Jetson)의 비전 노드에서 동적 객체를 인식하고 딥러닝(Deep Learning) 제어 정책(Policy) 모델이 산출한 롤/피치 각도와 정규화 추력(Thrust)값을 통해 드론을 직관적으로 조종해나가는 C++ 로직 구현체의 뼈대는 다음과 같다.

// MAVLink 프로토콜 페이로드 할당 예시 (C++ 헤더 mavlink_msg_set_attitude_target.h 참조)
mavlink_set_attitude_target_t att_setpoint = {};

// 시스템의 현재 부팅 타임스탬프 동기화 (네트워크 폐쇄 감지를 위해 매우 중요)
att_setpoint.time_boot_ms = get_current_time_ms();

// 네거티브 논리에 따라 각속도를 무시하고 오직 쿼터니언 자세와 추력 지령 인가
att_setpoint.type_mask = 1 | 2 | 4; // 0b00000111 즉 0x07 (Roll/Pitch/Yaw Rate 완전 Ignore)

// 우측 타겟을 쫓기 위한 목표 각도(Roll 15도 우측 강제 기울임)를 쿼터니언으로 스케일 변환 맵핑
float target_roll  = 15.0f * (M_PI / 180.0f);
float target_pitch = 0.0f;
float target_yaw   = current_estimated_yaw; 
EulerToQuaternion(target_roll, target_pitch, target_yaw, att_setpoint.q);

// 기체 스펙에 맞춰 정규화된 추력 할당 
// (호버링 추력(Hover_thrust) 파라미터가 0.45 근방이라 가정할 시 고도를 방어하기 보수적 수치 부여)
att_setpoint.thrust = 0.48f;

// 직렬 텔레메트리 포트(Serial) 혹은 UDP/TCP망을 통하여 데이터 스트림 발사
send_mavlink_message(&att_setpoint);

이렇게 고속 전송망을 타고 전달된 값은 드론의 내부 관성 측정 장치(IMU) 및 필터망에서 추정된 현재 쿼터니언(Current Quaternion)과 즉각 대수적으로 합차 비례 연산되어 펌웨어의 멀티로터 믹서 행렬(Mixer Matrix)로 하달된다.

4. Ardupilot (Copter) 시스템과의 아키텍처 관제 차별성 통찰

PX4와 Ardupilot 모두 업계 표준인 SET_ATTITUDE_TARGET 메시지를 지원하지만, 이를 파라미터로 소화하는 Flight Mode의 관제 스택 체계 철학은 현격한 차이를 보인다.

  • Ardupilot (Guided_NoNav 모드 및 Guided API 스택): Ardupilot 생태계 내에서 외부망이 자세 제어 통제권을 강제 박탈(Override)해 사용하려면 단순 포괄적 Guided 모드가 아닌, 항법(Navigation) 파트너 기능이 소프트웨어적으로 완전히 격리 해제된 특정 서브모드(Guided_NoNav)를 명시적으로 트리거해야 하는 엄격한 제약 서브루틴이 존재한다. GCS 개발자는 반드시 시스템이 외부 자세/추력 지령을 수용할 안전 상태인지 AC_AttitudeControl 및 파생 오버라이드 가드 플래그를 지속적으로 폴링(Polling) 동기화해야 시스템적 충돌이 야기되지 않는다.
  • PX4-Autopilot (Offboard 단일 모드 스레드 유연성): PX4는 Offboard라는 단일 통제 모드명 내에서, uORB 파이프를 통해 들어오는 메시지의 종류(위치 제어용 trajectory_setpoint 인가 아니면 자세 제어용 vehicle_attitude_setpoint 인가) 및 도달 빈도를 능동적으로 판별하여 펌웨어가 어느 계층 루프부터 대체(Bypass)할 것인지를 실시간 동적(Dynamic)으로 스마트 판단한다. 즉, 직결 기법이 모드 전환 오버헤드나 승인 핸드셰이크(Handshake) 딜레이 없이 즉각 즉시 수용되는 압도적 장점이 있어 전위적 로보틱스 모션 제어 연구에 월등히 유화적인 스택을 제공한다.

5. 결론 및 실무 구현 현장에서의 고찰

SET_ATTITUDE_TARGET을 이용한 내부 루프 직결 분기 우회 기법은 자율 비행의 지능적 판단 요소(Intelligence)를 기체의 제한된 하드웨어 내부에서 외부 GCS 서버군 혹은 고성능 동반 컴퓨터 그래픽 카드로 완전히 이관 및 외주화(Offloading)하기 위한 핵심적 중추 통신 브리지망이다. 머신 비전을 탑재한 동적 시각 기반 서보 제어(Dynamic Visual Servoing)나 강화학습(DDPG, SAC 등) 아키텍처 기반 신경망 곡예비행 논문 등 최첨단 도메인 제어 기법은 예외 없이 이 아키텍처적 터널을 토대로 실증이 이루어진다.

그러나 아키텍트가 필연적으로 조심해야 할 중대한 위험도는, 외부망의 운영체제(Linux 혹은 GCS 스레드 풀)의 스케줄러 지연에 의해 이 메시지의 스트리밍 전송 주기(일반적 최소 방어선 50Hz)가 버퍼링(Buffering)되거나 끊길 경우, PX4 내경 고유 감시 모듈은 Offboard Loss 혹은 Data Link Loss 비상 페일세이프(Failsafe)를 즉각 발동하여 시스템 권한을 회수, Loiter 대기나 RTL 복귀 모드로 도망쳐버린다는 사실성에 있다.
따라서 이 기법을 도입하는 GCS 통신 엔지니어는 제어 데이터의 맹렬한 실시간성(Real-time Constraints)을 보증할 수 있는 통신 채널망 확립과 저지연 멀티 스레드 프로그래밍 기법 확보, 더불어 실험 데이터로 보정·산출된 정밀한 추력(Thrust) 대 질량 비선형 매핑 제어 곡선 테이블을 필수적으로 확보해야만 치명적 비행 사고를 차단하고 외력 직결 지령의 극한의 이점을 온전히 활용할 수 있다.
