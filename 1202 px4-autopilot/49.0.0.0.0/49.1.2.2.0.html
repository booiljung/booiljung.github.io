49.1.2.2 시각 관성 오도메트리(VIO) 연동 시의 지연 시간(Latency) 보상 및 외부 자세 및 방향 추정치(External Vision/Heading) 융합

오프보드(Offboard) 환경에서의 자율 비행은 본질적으로 GPS 음영 지역(GPS-Denied)이나 실내 환경에서의 정밀한 항법 능력을 요구한다. 이를 위해 PX4-Autopilot은 Companion Computer(예: Raspberry Pi, NVIDIA Jetson 등)에서 구동되는 시각 관성 오도메트리(Visual Inertial Odometry, VIO) 및 SLAM 알고리즘의 결과를 수용할 수 있는 강력한 외부 추정치(External Estimation) 융합 기능을 제공한다. 본 절에서는 EKF2 시스템이 외부 VIO 데이터를 수용할 때 발생하는 지연 시간(Latency) 문제를 어떻게 수리적으로 보상하는지, 그리고 **외부 비전 자세 및 헤딩(External Vision/Heading)**을 어떤 메커니즘으로 융합하는지 상세히 분석한다.

1. 외부 비전(External Vision) 데이터의 uORB 연동 아키텍처

외부 임무 컴퓨터(Mission Computer)에서 연산된 기체의 3차원 위치(x, y, z)와 자세(통상적으로 쿼터니언 q 포맷) 데이터는 MAVLink의 ODOMETRY (ID: 331) 또는 VISION_POSITION_ESTIMATE (ID: 73) 메시지를 통해 비행 제어기(Flight Controller)로 전송된다.

전송된 메시지는 PX4의 mavlink_receiver 모듈에 의해 파싱(Parsing)되어 vehicle_visual_odometry uORB 토픽으로 변환 및 퍼블리시(Publish)되며, 최종적으로 EKF2 모듈이 이를 구독(Subscribe)하여 상태 측정값(Measurement)으로 활용한다.

2. 지연 시간(Latency) 발생의 원인과 치명성

카메라 영상을 캡처하고, 특징점(Feature)을 추출 및 매칭하며, 최적화(Optimization)를 통해 자세를 추정하기까지의 연산 과정은 필연적으로 **시스템 지연(System Latency)**을 수반한다.

지연의 흐름:

  1. 카메라 셔터 타이밍 (Sensor Exposure)
  2. 이미지 전송 지연 (MIPI/USB 통신)
  3. VIO/SLAM 알고리즘 연산 시간
  4. MAVLink 시리얼(UART/Ethernet) 전송 지연
  5. PX4 내부 uORB 메시지 처리 지연

만약 VIO 데이터가 EKF2에 도달했을 때 이 데이터가 기체의 과거 상태를 나타냄에도 불구하고 현재 상태로 융합(Fusion)된다면, EKF2의 혁신(Innovation)이 급격히 증가하고 필터가 발산(Divergence)하여 기체가 불안정하게 진동하거나 추락하는 치명적인 결과를 초래한다.

3. 지연 시간 보상(Latency Compensation) 알고리즘 메커니즘

PX4의 EKF2는 지연 시간을 정교하게 구조적으로 처리하기 위해 **과거 상태를 저장하는 버퍼(History Buffer)**를 운용한다. 이 지연 보상 아키텍처는 관성 항법 시스템(INS)의 보정에서 매우 중요한 공학적 기법이다.

3.1 타임스탬프 기반 동기화 및 EKF_EV_DELAY 파라미터

모든 VIO 시스템은 영상이 캡처된 순간의 정확한 타임스탬프를 MAVLink 메시지 내 usec 필드에 기록해야 한다. 하지만 타임스탬프 동기화(Time Sync)가 완벽하지 않거나 하드웨어적인 래그가 명확한 경우를 대비하여, PX4는 EKF2_EV_DELAY 파라미터를 제공한다.

이 파라미터는 외부 비전 측정치가 캡처된 시점부터 EKF2에 인식되기까지의 고정적인 ’평균 지연 시간(밀리초)’을 나타낸다. 시스템은 메시지가 도달한 시간에 이 파라미터 값을 차감하여 **실제 측정 발생 시간(True Measurement Time)**을 추정한다.

3.2 링 버퍼(Ring Buffer)를 활용한 상태 역산(Rewind)

EKF2는 고주파수(통상 250Hz 이상)의 IMU 측정을 기반으로 기체의 상태를 예측하면서, 일정 시간(통상 수백 밀리초 분량) 동안의 과거 예측 상태 \hat{x} 분산을 **링 버퍼(Ring Buffer)**에 저장해 둔다.

외부 비전 데이터 관측 시간 t_v가 현재 시간 t_{now}보다 지연된 채로 도착하면, 보상 로직은 다음 순서로 동작한다.

  1. 상태 버퍼 역탐색(Rewind): 버퍼에서 t_v와 가장 가까운 시점의 기체 상태 \hat{x}_{t_v}를 검색한다.
  2. 혁신(Innovation) 계산: 현재 시간 t_{now}의 융합 모델이 아닌, 과거 t_v 시점의 상태 \hat{x}_{t_v}를 사용하여 혁신 행렬 y_v를 산출한다.
    \quad y_{v} = z_{v} - h(\hat{x}_{t_v})
  3. 가중치 보정(Correction): 이 과거 시점의 혁신을 바탕으로 공분산을 업데이트한다.
  4. 재적분(Forward Propagation): t_v 시점에서 보정된 상태를 기반으로, 버퍼 안에 저장된 t_v에서 t_{now} 사이의 IMU 기록을 다시 적분하여(재연산) 현재 시점 t_{now}의 최종 보정된 상태 \hat{x}_{t_{now}}^{corrected}를 완성한다.

이러한 수리적 역산(Rewind/Fast-Forward) 방식을 통해, 지연된 VIO 데이터가 EKF2 시스템에 동적인 충격을 주지 않고 부드럽게 융합될 수 있다.

4. 외부 자세(Attitude) 및 자북(Heading) 데이터의 융합 (External Vision Fusion)

VIO 시스템은 위치(3D Position)뿐만 아니라 기체의 3차원 자세 요소(Roll, Pitch, Yaw)를 출력한다. VIO와 EKF2 간의 효율적인 융합을 위해, 사용자는 EKF2 파라미터를 통해 어떤 형태의 외부 비전 데이터를 필터에 반영할지 신중하게 결정해야 한다.

4.1 위치 및 높이 데이터의 융합 설정

EKF2_EV_CTRL 파라미터는 비전 데이터의 어떤 요소를 칼만 필터 예측 모델의 관측 모델에 포함할 것인지를 비트마스크(Bitmask)로 정의한다.

  • 수평 위치 융합 (Horizontal Position Fusion)
  • 수직 속도/위치 융합 (Vertical Velocity/Position Fusion)

기압계 높이나 GPS가 없는 완전한 실내 비행에서는 외부 비전의 Z축 추정치를 Primary Height Source(EKF2_HGT_REF = Vision)로 설정하여 동작해야 한다.

4.2 헤딩(Heading/Yaw) 융합의 핵심: 자북계와의 분리

실내에서의 오프보드 제어 시 가장 고질적인 문제는 자기장 교란에 의한 나침반(Magnetometer)의 왜곡이다. 철골 구조물이나 모터 전원 케이블이 뿜어내는 자기장은 나침반의 3D 벡터를 뒤틀고 헤딩 추정에 치명적인 악영향을 준다.

따라서 VIO를 구비한 기체는 통상적으로 나침반을 완전히 비활성화하고 VIO의 요(Yaw) 각도를 유일한 헤딩 정보로 사용하도록 구성한다.

  • EKF2_MAG_TYPE을 0(None)으로 세팅하거나, VIO 헤딩을 우선 참조하게 설정한다(EKF2_EV_CTRL 비트 중 Yaw Fusion 활성화).
  • VIO 알고리즘의 기준 좌표계(Reference Frame)가 시작 시점의 방향을 0도로 잡는 국소 좌표계(Local Frame)인 경우, EKF2는 VIO의 로컬 플랫 지구 좌표계(Flat Earth Frame)와 내부 NED 좌표계 사이의 오프셋(Offset) 각도를 계산하고 보상하는 내부 정렬 상태 머신(Alignment State Machine)을 실행한다.

5. ArduPilot과의 차이점 비교 (Ardupilot 대비 차이)

시각 관성 오도메트리와 같은 비-GPS 센서의 지연 시간 처리 및 융합에 있어 PX4와 ArduPilot은 아키텍처적 유사성이 있으나, 파라미터 구성 및 버퍼 운용에 작은 차이가 존재한다.

  • 버퍼 크기 및 운용: 두 자동조종장치 모두 지연 보상을 위한 히스토리 버퍼를 사용하지만, PX4 프로세스 설계 최적화는 EKF2의 메모리 관리 기법에 영향을 미쳐 극단적으로 큰 딜레이 시스템(예: 500ms 이상의 무거운 SLAM) 사용 시 EKF2_EV_DELAY 임계값 조율에 민감하다. 너무 오랜 시간을 역전(Rewind)하면 CPU 부하(IMU 재적분 부하)로 인해 워치독(Watchdog)에 걸리거나 메인 제어 루프가 지연될 수 있다.
  • 설정의 명확성: ArduPilot(EKF3)은 EK3_SRCn 시리즈 파라미터를 통해 센서 소스를 1, 2, 3차 백업(Fallback) 배열로 시각화하여 지정할 수 있는 매우 직관적인 유연성을 제공하지만, PX4는 EKF2_EV_CTRL이나 EKF2_HGT_REF 등의 조합을 요구하여 초기 설정 복잡도가 높은 편이다. 그러나 일단 설정되면 PX4의 EKF2는 파라미터를 엄격하게 준수하며 모드 이탈 조건이 칼같이 명쾌하다.

6. 결론

자율 비행을 위한 오프보드 제어 루프(External Control Loop)가 정밀하고 부드러운 궤적 생성을 수행하기 위해서는 VIO 시스템의 지연 시간 보상 버퍼 운용 원리를 개발자가 정확히 이해해야 한다. 지연 시간이 보상되지 못한 위치-자세 데이터는 EKF 발산의 가장 큰 원인이다. 개발자는 MAVLink 패킷과 타임스탬프 동기화 정밀도를 개선하여 EKF2_EV_DELAY 값을 최소화하고, 자기장 교란 환경에서 VIO 기반 헤딩 융합 파라미터 설정을 적절히 분리함으로써 외부 컴퓨터 중심의 제어권한 획득을 확고하게 만들 수 있다.
