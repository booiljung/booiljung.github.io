49.5. MAVLink 위치, 속도 및 자세 지령(Setpoint) 페이로드의 물리적 다중화 메커니즘

무인항공기(UAV)의 오프보드(Offboard) 제어는 근본적으로 닫힌 루프(Closed-loop) 제어 시스템 내에서 외부의 지상 관제 시스템(GCS)이나 동반 컴퓨터(Companion Computer)가 지속적으로 목표 궤적(Trajectory)을 하달하는 행위이다. 이를 위해 MAVLink 프로토콜은 네트워크 대역폭의 한계와 제어의 실시간성(Real-time)을 동시에 충족하기 위하여, 단일 메시지 페이로드(Payload) 내에 위치(Position), 속도(Velocity), 가속도(Acceleration), 힘(Force), 자세(Attitude) 및 요율(Yaw Rate) 지령을 혼합하여 제한 없이 전송할 수 있는 고도의 물리적 다중화(Physical Multiplexing) 메커니즘을 제공한다. 본 절에서는 해당 다중화 메커니즘의 소프트웨어 아키텍처와 비트스트림(Bitstream) 처리 구조를 심도 있게 분석한다. (참조 펌웨어 버전: PX4-Autopilot v1.14, QGroundControl v4.3)

1. 지령 페이로드 다중화(Multiplexing)의 시스템 공학적 필요성

지상 관제 시스템(GCS)이나 ROS2 노드에서 생성된 최적 비행 궤적(Optimal Trajectory)은 시간에 대한 연속 방정식이다. 이를 이산적인 형태의 MAVLink 데이터그램으로 변환해 비행 제어기(Flight Controller)에 전달할 때, 다음과 같은 아키텍처적 딜레마가 발생한다.

  • 각각의 지령(위치, 속도, 가속도, 요)을 독립된 MAVLink 토픽 메시지로 분할하여 송신할 경우: 통신 계층의 헤더 파싱 오버헤드(Overhead)가 기하급수적으로 극대화되며, 메시지 도달 시간의 미세한 불일치(Timestamp Mismatch)가 내부 제어 루프 진동(Oscillation)을 유발할 수 있다.
  • 반대로, 모든 제어 지령을 무조건 하나의 거대한 고정 페이로드로 송신할 경우: 속도만 제어하려는 상황에서 무의미한 위치 제어 파라미터를 강제 포함하게 되어 대역폭(Bandwidth) 낭비를 초래한다.

이러한 공차를 해결하기 위해 PX4-Autopilot 환경에서는 대표적으로 SET_POSITION_TARGET_LOCAL_NED, SET_POSITION_TARGET_GLOBAL_INT, 그리고 SET_ATTITUDE_TARGET 메시지 내에 **타입 마스크(Type Mask, type_mask)**라는 16비트(16-bit) 스위칭 필드를 투입하여 다중화 메커니즘을 구체화하였다.

2. Type Mask 기반 페이로드 다중화 디멀티플렉싱(Demultiplexing) 아키텍처

MAVLink 메시지는 엄격히 고정된 크기의 C-Struct 메모리 레이아웃 특성을 지닌다. SET_POSITION_TARGET_LOCAL_NED 메시지를 예로 들면, 총 53 바이트 패킷 내에 모든 동역학계 차원(위치, 속도, 가속도 등) 배열이 강제 할당되어 있다. 하지만 실제 가동 중인 비행 제어기의 내측 루프(Inner Loop)로 갱신 투입되는 변숫값은 전적으로 type_mask 비트 플래그 연산에 의해 다중화(Multiplexing) 및 선택 기반 디멀티플렉싱(Demultiplexing)된다.

2.1 비트마스크 필터링 아키텍처 및 데이터 흐름도

다음 다이어그램은 GCS나 MAVSDK 체계에서 생성된 한 덩어리의 페이로드가 내부 Offboard 모듈로 전달되어 각 계층 제어기로 스위칭되는 과정을 묘사한다.

    B -->|type_mask & 0x07| C{Position Enabled?}
    B -->|type_mask & 0x38| D{Velocity Enabled?}

    I --> K
    J --> K
    K --> L[Actuator Mixer / Control Allocation]

type_mask 필드의 핵심은 바로 **역논리(Inverse Logic)**를 취하고 있다는 점이다.
해당 비트가 1로 설정(Set)되면 해당 차원의 제어 지령을 **내부적으로 무시(Ignore)**하고, 0으로 설정(Clear)되면 해당 차원 파라미터를 **활성화(Enable)**하여 수용한다.
예를 들어 하위 3비트(0x07, 111_2)를 세팅할 경우 X, Y, Z 측의 ‘위치’ 지령을 배제하고 그 이외의 속도나 가속도 루프만 구동할 수 있다. 이와 같이 0을 활성화로 채택한 시스템공학적 근거는, 페이로드 전체를 0으로 초기화한 빈 상태(Zero-initialized base state)가 가장 완전하고 방대한 지령 즉, ’모든 물리량을 전부 사용’이라는 결정론적 전제조건을 만족할 수 있도록 아키텍처를 설계했기 때문이다.

3. 다중화 메시지 투입과 내부 uORB 메커니즘의 사상(Mapping)

MAVLink 기반으로 디코딩된 다중화 페이로드는 PX4 내부 발행/구독 네트워크 통신망인 uORB(Micro Object Request Broker) 레이어로 진입하여 다음과 같이 구체적으로 사상(Mapping)된다.

  1. 오프보드 제어 데몬(Offboard Control Module): 내부 소스코드 mavlink_receiver.cpp 클래스는 수신된 SET_POSITION_TARGET 류의 type_mask를 비트연산으로 면밀히 파싱(Parsing)한다.
  2. 트래젝토리 셋포인트(trajectory_setpoint) 변수 치환: 타입 마스크에 의해 수용(활성화)된 특성(예: 속도 벡터 \mathbf{v} = [v_x, v_y, v_z]^T와 위치 벡터 \mathbf{p} = [p_x, p_y, p_z]^T)은 그 값을 유지하여 내부 uORB 구조체의 부동소수점 데이터로 복사되고, 무시되도록 마스킹(Masking)된 항목의 값은 IEEE 표준의 NAN(Not a Number) 기호로 명시적 치환된다.
  3. 오프보드 비행 태스크(Flight Task Offboard): 제어 사이클의 종단인 FlightTaskOffboard.cpp 단계에서는, NAN으로 치환된 변수는 다단계 캐스케이드(Cascade) PID 제어기의 산술 투입 및 피드포워드(Feedforward) 연산에서 구조적으로 제외되며 유효값이 존재하는 물리량만이 계산 루틴에 투입된다.

이러한 “단일 패킷 집중 다중화(Single-Packet Concentrated Multiplexing)” 설계 패턴에 의해, 대역폭 점유를 최저로 유지하면서도 드론의 6-DoF 상태를 단 1회의 타임스탬프 하에서 유기적이고 완벽하게 병합 동기화할 수 있다.

4. Ardupilot 모델 대비 PX4 페이로드 다중화의 분석적 철학 차이

경쟁 플랫폼인 Ardupilot 환경에서는 동일하게 SET_POSITION_TARGET_LOCAL_NED 메시지를 활용하여 GUIDED 모드를 제공하지만, 내부 위치 제어기(AC_WPNav)의 계층적 특성 탓에 위치 지령과 속도 지령을 동시 인가하더라도, 물리량들이 상호 보완적으로 피드포워드 합성을 이루기보다는 하나의 주도적인 제어 변수(Dominant Control Parameter)만이 궤적을 지배하는 양상을 보인다. Ardupilot은 전통적으로 보수적 궤적 탐색을 선호해 페이로드 복합 마스크(위치 + 속도) 구현에서 버전에 따른 변형이나 기능 제약을 가질 때가 많았다.

반면에 PX4-Autopilot은 태생부터 학술적 제어 공학의 코어 아키텍처인 캐스케이드 피드백-피드포워드 루프(Cascade Feedback/Feedforward Loop)의 원형 보존을 그 철학으로 추구했다. 위치 오차 비례 제어에 의한 도출값과, 네트워크로 동시에 인용된 속도/가속도 지령 페이로드를 매끈하게 피드포워드로 합산(Summation)하여 하위 모터 추력 할당기(Control Allocator)에 하달한다. 이로인해 Position, Velocity, Yaw 동시 다중화 페이로드를 입력했을 때, 고속 코너링이나 급기동(Aggressive Maneuver) 시 궤적 추종 오차가 현저히 줄어드는 강력한 모델 예측 성향을 발휘한다.

결론적으로 GCS 및 로봇 컴패니언 애플리케이션 프레임워크 설계자는, PX4 통신 프로토콜을 다룸에 있어 단순히 원하는 고도 수치를 채워 넣는 초보적 관점에 정지해서는 아니되며, MAVLink 데이터그램의 물리적 다중화 메커니즘을 온전히 직시하여 매 타임 프레임마다 제어기의 물리적 해석을 판가름하는 type_mask 필드를 공학적으로 치밀하게 조율할 의무를 지닌다.
