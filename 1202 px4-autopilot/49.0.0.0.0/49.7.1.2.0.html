49.7.1.2 공유 메모리(Shared Memory) 접근 충돌 방지를 위한 뮤텍스(Mutex) 잠금(Lock) 및 세마포어(Semaphore) 동기화

1. 개요

지상 관제 시스템(GCS) 애플리케이션인 QGroundControl(QGC) 내부의 백엔드 아키텍처는 수십 개의 QThread와 POSIX 스레드가 병렬로 동작하며 엄청난 양의 데이터를 쏟아낸다. 메인 GUI 스레드는 이 데이터를 읽어와 사용자 인터페이스(UI)를 갱신해야 하고, 통신 스레드는 동일한 데이터 버퍼에 쉴 새 없이 새로운 Telemetry 측정값을 기록해야 한다. 이렇게 2개 이상의 스레드가 동일한 데이터 영역(예: 기체의 배터리 전압 변수, 위치 좌표 등)에 동시에 접근하려 할 때 발생하는 경쟁 상태(Race Condition) 와 이로 인한 데이터 무결성 훼손 방지를 위해 동기화 메커니즘이 강제된다.

2. 데이터 접근 충돌(Data Race)과 뮤텍스(Mutex)의 필요성

메모리 영역의 갱신(Writing)은 원자적(Atomic) 연산이 아니며, 여러 개의 CPU 명령(Instruction) 사이클을 거친다. 만약 통신 스레드가 배열 형태의 궤적(Trajectory) 데이터를 50%쯤 메모리에 기록하고 있을 때, CPU 스케줄러가 메인 GUI 스레드로 제어권을 넘겨버린다면 어떻게 될까? GUI 스레드는 반쪽만 갱신된 비정상적인 쓰레기 값(Garbage Value)을 화면에 렌더링하게 되어 버그 방출이나 프로그램 크래시(Segmentation Fault)를 일으키게 된다.

이를 막기 위해 QGC 코어 소스 코드에서는 주요 데이터 객체마다 잠금 장치인 뮤텍스(Mutex; Mutual Exclusion) 를 배치한다.

  • 어떤 스레드(예: MAVLink 파서)가 Vehicle 인스턴스 내의 특정 데이터 변수를 변경하기 전에 잠금(lock())을 건다.
  • 이 잠금이 활성화된 동안, 값을 읽으러 온 GUI 스레드는 잠금이 풀릴 때(unlock())까지 일시 정지(Blocked)된 채로 큐에서 얌전히 대기해야만 한다.

3. QGC 구조 내부의 QMutex와 QReadWriteLock 활용

C++ std::mutex를 사용하기도 하지만, Qt 환경에서는 그와 대응되는 QMutex와 고성능 확장을 위한 다중 잠금 개념들이 두루 쓰인다.

3.1 QMutexLocker 기법 (RAII 패턴)

QGC 백엔드에서는 C++의 생성자/소멸자 원리를 이용한 RAII(Resource Acquisition Is Initialization) 패턴이 광범위하게 쓰인다.
코드 블록 내부에서 QMutexLocker locker(&_mutex); 객체를 선언하면 생성 시점에 자동으로 lock()이 수행되고, 해당 블록({ ... })의 끝 범위를 벗어나거나 심지어 도중에 예외(Exception)가 발생하여 return을 맞더라도 소멸자가 호출되며 무조건 unlock()을 실행해 준다. 프로그래머의 실수로 인한 교착 상태(Deadlock) 가능성을 수학적으로 제거해 주는 핵심 코딩 스탠다드이다.

3.2 QReadWriteLock을 통한 최적화

모든 상황에 무식하게 뮤텍스를 걸어버리면, 데이터를 그저 ‘읽기만(Read-only)’ 하려는 여러 개의 무해한 스레드들까지도 서로를 멈추게 만들어 GCS의 성능이 치명적으로 저하되는 스레드 병목(Thread Contention) 현상이 터진다. QGroundControl은 이를 피해 가기 위해 읽기-쓰기 잠금(Read-Write Lock) 전략인 QReadWriteLock을 통신 파라미터 매니저 등에 적극 차용한다.

  • 데이터 갱신 시에는 lockForWrite()를 써서 다른 모든 읽기/쓰기 스레드를 통제한다.
  • 단순히 UI 표기용으로 값만 가져갈 때는 lockForRead()를 쓰는데, 이 상태의 스레드가 100개건 1,000개건 서로 싸우지 않고 동시다발적으로 메모리에 병렬 접근하는 것을 허용하며 극적인 성능 최적화를 부여한다.

4. 생산자-소비자 패턴과 세마포어(Semaphore) 조율

단순 메모리 보호를 위한 뮤텍스가 아닌, “쓰레드 A가 데이터를 일정량 생산해야만 쓰레드 B가 구동된다“는 구조적 신호 제어(Signaling)가 필요할 때가 있다. 예를 들어, 소켓(Socket)에서 들어온 로우 바이트(Raw Byte) 버퍼링 과정이 이에 해당한다.

이때는 특정 크기의 카운터 값을 가지는 세마포어(QSemaphore) 가 활약한다.

  1. 소켓 수신 스레드(생산자)가 네트워크 패킷 5개를 받아 링 버퍼(Ring Buffer)에 넣으면 세마포어 값을 +5 올린다(release(5)).
  2. 파싱 스레드(소비자)는 대기 루프를 돌다가, 세마포어 수가 늘어난 것을 확인하면 필요한 만큼 -1씩 차감(acquire(1))하며 버퍼의 패킷을 조용히 꺼내 처리한다.
    버퍼가 텅 비게 되면(세마포어=0), 파싱 스레드는 CPU 점유율을 1%도 먹지 않은 채로 OS 단말에서 영면 대기(Sleep)에 돌입하는 고효율 파이프라인이 완성되는 것이다.

5. 결론 및 실무적 교훈

공유 메모리에 대한 뮤텍스와 세마포어 동기화 기법은 다수의 스레드 기반으로 무거운 I/O 트래픽 처리를 감당해야 하는 QGroundControl의 ‘백본 네트워크(Backbone Network)’ 인프라 그 자체다. 이러한 동기화(Synchronization) 프리미티브들이 C++ 코드 전반에 얼마나 촘촘하고 현명하게(읽기 최적화 및 RAII 도입) 직조되어 있느냐에 따라, GCS 구동 피시의 메모리 단편화 발생률과 GUI 정체 여부, 그리고 PX4-Autopilot 기체와의 연결 수명이 판가름 나게 된다.
