49.1.1 분산 제어 시스템(Distributed Control System) 관점의 임무 컴퓨터(Mission Computer)와 비행 제어기(Flight Controller) 간 인터페이스 설계

1. 개요

현대의 무인항공기(UAV) 시스템은 센서 데이터 처리, 컴퓨터 비전(Computer Vision) 기반 객체 인식, 복잡한 경로 계획(Path Planning) 등의 고부하 연산을 필요로 한다. 이러한 연산을 하드웨어 자원이 제한적인 비행 제어기(Flight Controller, FC) 단독으로 수행하는 것은 불가능에 가깝다. 따라서 고성능 마이크로프로세서를 탑재한 임무 컴퓨터(Mission Computer, MC)와 비행 제어기를 분리하여 역할을 분담하는 형태의 분산 제어 시스템(Distributed Control System, DCS) 구조가 필수적으로 도입되고 있다.

본 절에서는 임무 컴퓨터와 비행 제어기가 물리적, 논리적인 계층에서 어떻게 안전하고 강건한 인터페이스를 유지하는지 그 설계 기준을 논의한다.

2. 물리적 하드웨어 인터페이스 분류 및 특징

임무 컴퓨터(예: Raspberry Pi, NVIDIA Jetson 시리즈)와 비행 제어기(예: Pixhawk 6C 등) 간 통신을 위해 가장 보편적으로 사용되는 인터페이스는 직렬 통신(UART)과 이더넷(Ethernet)이다. 각 하드웨어별 특성은 다음과 같다.

  1. UART(비동기 직렬 통신) 기반 인터페이스
  • 장점: 모든 비행 제어기(PX4-Autopilot 및 Ardupilot 하드웨어 호환 규격 포트 TELEM 1/2)에서 범용적으로 지원되며 구현이 간단하다.
  • 단점: 보드레이트(Baud Rate)가 보통 921600 bps 수준으로 제한되어, 고대역폭 데이터(예: 원시 영상, 고주파 포인트 클라우드)를 온보드로 전송하기에는 병목 현상이 발생한다. 전기적으로 민감하여 장거리 배선이나 노이즈가 심한 환경에서는 CRC(Cyclic Redundancy Check) 에러가 유발될 가능성이 있다.
  1. 이더넷(Ethernet) 인터페이스
  • 장점: 최신 규격인 Pixhawk 5X/6X 등 표준화된 고사양 하드웨어에서 채택되기 시작했다. TCP/UDP 통신을 기반으로 100Mbps ~ 1Gbps에 달하는 고속 데이터 통신과 안정적인 절연(Isolation) 설계가 용이하다. 멀티캐스트(Multicast)를 활용하여 다중 임무 컴퓨터 토폴로지 구성에 적합하다.

3. 소프트웨어/데이터 버스 인터페이스 설계 (PX4 v1.14 이상 기준)

물리 계층 위에서 데이터를 교환하는 방식은 무인기의 실시간성에 절대적인 영향을 미친다. 과거에는 오직 MAVLink 프로토콜만이 주로 쓰였으나, ROS 2 환경의 보급화로 데이터 파이프라인의 양상이 바뀌었다.

3.1 기존 MAVLink 기반 인터페이스

  • 비행 제어기 내부의 mavlink 모듈이 UART 포트를 폴링(Polling)하며 패킷을 파싱하고 생성하는 구조이다.
  • 장점: 오버헤드가 극도로 적으며 QGroundControl과 같은 기존 관제 시스템과 직접적으로 호환된다.
  • 단점: ROS 2 환경의 임무 컴퓨터 코드에서 이를 파싱하기 위해 MAVROS와 같은 노드(Node) 변환기를 한 번 더 거쳐야 하므로 아키텍처의 복잡성 및 변환 지연(Translation Latency)을 초래한다.

3.2 uXRCE-DDS (ROS 2) 인터페이스 아키텍처

최신 PX4 환경에서는 MAVROS를 대체할 수 있는 uXRCE-DDS (Micro XRCE-DDS) 아키텍처가 표준 인터페이스로 자리 잡았다. 이는 비행 제어기 내의 uORB 토픽을 임무 컴퓨터의 DDS 도메인(ROS 2 토픽) 환경으로 직접 투영(Mapping) 시켜주는 인터페이스이다.

  • 비행 제어기 측에는 Micro XRCE-DDS Client가 데몬 형태로 실행되고, 임무 컴퓨터 측에는 Micro XRCE-DDS Agent가 동작하여 두 장치 사이에 직렬 케이블이나 UDP 페이로드를 터널링(Tunneling)한다.
  • 임무 컴퓨터의 코드는 px4_msgs 라이브러리를 임포트(Import)하여 마치 다른 ROS 2 노드를 호출하듯 퍼블리셔(Publisher) / 서브스크라이버(Subscriber) 로직을 전개할 수 있다.

4. 인터페이스 간 타이밍 동기화 (Time Synchronization)

분산 컴퓨팅에서 가장 주의해야 할 문제는 두 컴퓨팅 장치 간의 RTC(Real-time Clock)가 서로 일치하지 않아 발생하는 타임스탬프 오류(Timestamp Skew)이다.
임무 컴퓨터에서 발생한 위치 명령이 정확한 시점을 반영해야만 상태 추정기(EKF2)나 제어 루프에서 노이즈로 해석되지 않기 때문이다.

  • MAVLink의 경우 TIMESYNC 메시지를 주고받으며 지연 시간을 역산산하여(RTT, Round-Trip Time) 두 보드 간의 시간 오프셋을 추적한다.
  • uXRCE-DDS 아키텍처의 클라이언트/에이전트 프로토콜 역시 세션 계층(Session Layer) 내부에서 클록 동기화 과정(Time Sync)을 수행하여, 메시지의 timestamp 필드가 양측 모두 마이크로초 단위에서 정렬되도록 인터페이스를 통합 제공한다.

5. 강건성: Heartbeat와 인터페이스 오류 감지 메커니즘

분산 시스템 아키텍처 설계에는 한쪽 구성 요소에 고장이 발생했을 경우 치명적 장애를 방지하는 페일세이프(Failsafe) 요소가 필요하다.
임무 컴퓨터와 FC는 상호 간에 주기적으로(일반적으로 1Hz 단위) HEARTBEAT 메세지(MAVLink의 경우) 혹은 이에 준하는 생존 핑(uXRCE Ping)을 교환해야 한다.
FC의 Commander 모듈은 오프보드 제어 중 임무 컴퓨터로부터 이 핑(Ping)과 제어 값이 명시된 데드라인(예: 0.5초) 내에 당도하지 않으면 즉시 임무 컴퓨터 연결 두절(Companion Computer Lost)로 판정하고 안전 메커니즘을 작동시킨다.

6. 결론

결과적으로, PX4-Autopilot과 임무 컴퓨터는 소프트웨어 공학적으로 결합도가 낮으면서(Loosely Coupled), 데이터 처리량을 보장하는 강건한 데이터 통합 토폴로지를 요구한다. 현재 추세에 비추어볼 때, 단순 관제나 구버전 시스템과의 연동은 효율적인 MAVLink를 취하고, 복잡한 군집 비행 혹은 인공 지능을 비롯한 딥러닝/최신 자동화 알고리즘이 적용된 임무 컴퓨터에서는 ROS 2 기반의 uXRCE-DDS 체계적 인터페이스를 적용하는 것이 시스템 유연성 향상 측면에서 가장 설득력 있는 공학적 해법이다.
