49.8.2.1 Commander 모듈 내부의 EKF 혁신 거부(Innovation Rejection) 플래그 및 센서 타임아웃 오류 식별

본 절에서는 PX4-Autopilot(기준 최신 펌웨어 버전 v1.14 및 최신 메인 브랜치)과 지상 관제 시스템인 QGroundControl(QGC, v4.3.x 이상) 환경에서 Commander 모듈 내부의 확장 칼만 필터(EKF, Extended Kalman Filter) 혁신 거부(Innovation Rejection) 플래그와 센서 타임아웃(Sensor Timeout) 오류를 어떻게 식별하고 처리하는지에 대해 심층적으로 다루고자 한다.

비행 제어 시스템의 안전성과 신뢰성을 확보하기 위해서는 센서 융합 과정에서 발생하는 데이터 이상 상태를 정확히 감지하는 것이 필수적이다. 특히 기체가 Offboard 모드나 자동 비행(Auto) 모드에서 운용될 때, EKF 필터의 건전성(Health Status)은 비행 모드 전환의 핵심 결정 요인으로 작용한다.

1. EKF 혁신(Innovation)과 혁신 거부(Innovation Rejection)의 메커니즘

자율 비행 시스템의 내비게이션 추정기(Estimator)로 주로 사용되는 EKF2 모듈은 다양한 센서(IMU, GPS, 기압계, 시각적 관성 주행거리 측정기(VIO, Visual Inertial Odometry))의 데이터를 융합하여 기체의 현재 상태(위치, 속도, 자세)를 추정(Estimation)한다.

필터 갱신(Update) 단계에서, 모델을 통해 예측된 센서 측정값과 실제 센서에서 수신된 측정값 간의 차이를 계산하며, 이를 혁신(Innovation) 혹은 잔차(Residual)라고 정의한다.

시스템 동역학 모델과 실제 물리적 환경 간의 일치도가 높고 센서 노이즈가 허용 오차 범위 내에 있다면 이 혁신 값은 0에 가까운 값을 유지한다. 그러나 외부 환경의 급격한 간섭(예: 도심 지역에서의 건물에 의한 GPS 다중 경로 오차, 고압선으로 인한 자기장 교란)이나 하드웨어 결함 등으로 인해 실제 측정값이 EKF의 예측 모델과 허용치 이상으로 벗어날 경우, 필터는 해당 데이터의 신뢰성을 부정하여 융합 과정에서 배제한다. 이 결정 과정을 **혁신 거부(Innovation Rejection)**라고 한다.

EKF 혁신 거부 현상이 지속되면 오류 발산(Divergence)을 막기 위해 필터는 해당 센서 입력의 의존도를 낮추거나 비활성화해야 한다. 이는 심각한 자세 지령(Setpoint) 편차 및 내비게이션 오류를 초래할 수 있으므로, 제어 최상위 모듈인 Commander는 이를 지속적으로 감시해야만 한다. 참고할 주요 문헌으로는 EKF 융합 이론에 관한 논문인 “A Robust Multi-Sensor Fusion Approach for Unmanned Aerial Vehicles“를 들 수 있다.

2. Commander 모듈 내부의 상태 판별 아키텍처

PX4 아키텍처 내부의 중추 신경인 Commander 모듈(src/modules/commander 하위 파일 참조)은 기체의 상태 머신(State Machine) 전환을 수행하며 시동(Arming) 및 비행 모드 천이(Transition)의 최고 권한을 갖는다.

Commander는 uORB(Micro Object Request Broker) 메시징 통신망을 통해 EKF2 메인 쓰레드에서 발행(Publish)되는 상태 정보를 구독(Subscribe)한다. 혁신 거부 및 에러 상태 모니터링에 사용되는 핵심 메시지는 estimator_status이다.

이 메시지는 EKF 추정기의 전반적인 상태를 표현하며, 각 센서(GPS, Baro, Mag, Vision)에 대한 혁신 테스트(Innovation Test) 통과/실패 결과를 비트마스크(Bitmask) 형태로 전달한다.
Commander 내부의 검증 루틴(주로 preflight_check.cpp 또는 이벤트 처리기)은 이 비트마스크 맵을 확인하여 검증을 수행한다.

예를 들어, GPS 수신 단의 오차가 설정된 임계 파라미터(예: EKF2_GPS_V_GATE, EKF2_GPS_P_GATE)를 초과하여 거부될 경우 estimator_status 메시지의 내부 플래그는 REJECT_HORz_POS 실패로 설정된다. Commander는 이러한 플래그의 활성화가 특정 시간 유지되면, Failsafe(안전 장치) 절차를 가동하거나 사용자의 명시적인 Offboard 모드 전환 명령을 거부(Rejection)한다.

3. 센서 타임아웃(Sensor Timeout) 오류의 발생과 식별

센서 오류의 또 다른 주요 원인은 데이터의 지연(Delay)이나 하드웨어적 통신 두절로 인해 일정 주기 동안 데이터 갱신이 발생하지 않는 현상인 센서 타임아웃(Sensor Timeout)이다.

물리 인터페이스(I2C, SPI, CAN/UART)의 결함, 전자파 간섭(EMI), 데이지 체인 상의 버스 트래픽 마비 등으로 인해 센서 드라이버가 측정값을 uORB 토픽으로 정상 발행하지 못하면, 메시지 갱신 주기는 설정된 제한치(예: 고주파 IMU 센서의 최장 지연 제한 10ms) 초과와 같은 문제를 야기한다.

PX4 메인 제어 루프의 타임아웃 감시 로직은 시스템 클럭에 의존한다. 센서 구독을 처리하는 각각의 파이프라인에서 직전 수신된 타임스탬프(timestamp)와 현재 시스템 기준 시간(hrt_absolute_time())의 시차(Delta Time)를 비교하여 감지한다.

이 시간차가 센서 타임아웃 파라미터(COM_SENS_TIMEOUT 등)를 초과하면 시스템의 각 서브 모듈, 특히 Commander는 심각한 시스템 결함(Fault) 상태로 식별하고, 비행 모드 강제 변경 및 시동 해제(Disarming) 플래그를 세팅한다. 이는 지상 관제에서 QGroundControl(QGC)을 활용하여 “CRITICAL: SENSOR TIMEOUT“과 같은 오류 알람으로 원격 관제자에게 전파된다.

4. Offboard 모드 진입 거부 로직 모델링 (Mermaid)

다음 다이어그램은 외부 MAVLink 단말(Companion Computer)이나 GCS에서 Offboard 전환 명령을 송신했을 때, Commander가 어떻게 내부의 EKF 혁신 거부 및 타임아웃 플래그를 식별하여 거부하는 지를 유한 상태 머신 형태로 모델링한 것이다.

stateDiagram-v2
    direction TB
    [*] --> Standby: 전원 인가 및 대기(Disarmed) 상태
    
    state "Commander 모듈 평가 엔진" as Eval {
        state "EKF 품질(Quality) 검사" as CheckEKF
        state "센서 주파수(Timeout) 검사" as CheckTimeout
        
        CheckEKF --> CheckTimeout: 혁신 거부 플래그 없음
        CheckTimeout --> Valid: 갱신 지연(Delay) 없음
    }
    
    Standby --> Eval: Offboard 모드 전환 명령 (MAVLink Cmd)
    
    Valid --> Offboard_Mode: 전환 승인(ACK 전송)
    
    CheckEKF --> Rejection: GPS/지자기/시각 혁신 거부 검출
    CheckTimeout --> Rejection: 센서 통신 불가/타임아웃 발생
    
    Rejection --> Standby: 전환 거부 및 상태 롤백(NACK 전송)
    Rejection --> Failsafe: 공중 비행 중일 경우 RTL/Land 천이
    
    Offboard_Mode --> Eval: 런타임(Runtime) 지속 감시 주기 수행

위의 모드 전환 및 판별 흐름도에 따라, PX4 시스템은 비행 제어 지령 이전에 추정기의 상태 건전성을 우선시하는 페일 세이프 중심의 방어형 프로그래밍 전략을 취하고 있음을 숙지하라.

5. 비행 데이터 분석(Flight Data Analysis)과 추적 튜닝

현장 비행 실험이나 시뮬레이션 인 더 루프(SITL) 운용 중에 센서 혁신 발생 등으로 시스템 거부 현상을 경험하였다면, 이를 해결하기 위해 원격 측정(Telemetry) 기반의 로깅 데이터(ULog) 아키텍처를 면밀히 분석해야만 한다.

  1. QGroundControl 실시간 모니터링:
    QGroundControl의 MAVLink Console 위젯을 이용하거나 Analyzer 패널의 ESTIMATOR_STATUS 계기 뷰를 띄워 실시간 런타임 상황을 확인하라. EKF 거부 플래그의 비트 패턴 변화 양상을 직관적으로 감시할 수 있다.

  2. ULog 시계열(Time Series) 플롯 분석:
    기체의 SD 카드 또는 MAVLink 링크를 토대로 추출한 .ulg 바이너리 로그 파일을 Flight Review 온라인 웹 도구나 오프라인 유틸리티인 pyulog 디버거를 통해 파싱(Parsing)하라.
    그래프 분석 시 ekf2_innovations 메시지의 자력계(Magnetometer X/Y/Z) 혹은 GPS 좌표축 단위 필터 오차 값 플롯을 집중 확인해야 한다. 임계선을 수 차례 반복적으로 벗어나는 스파이크(Spike) 현상과 Commander 상태 변경 알람 시간대가 정확히 매핑되는지 대조하라.

  3. 게인 조정 및 캘리브레이션 튜닝:
    시스템 노이즈 환경 측정 결과에 따라 타당한 융합 지표 범위 확보를 위해 파라미터들을 조정할 필요성이 있다. 자력 센서는 추가적인 공간 캘리브레이션(Calibration)을 필수적으로 진행해야 하며, 여전히 혁신 편차가 심할 경우 EKF2_MAG_GATE 의 게이트 폭 매개변수를 조정하라. 이와 유사하게 GPS 음영 및 반사가 심한 구역에서는 EKF2_GPS_P_GATE 값을 부분적으로 완화시켜야 할 수 있다. (단, 과도한 완화는 자율비행 제어 안정성을 근본적으로 파괴함을 지각하라.)

결론적으로, PX4-Autopilot를 운용하는 무인항공기 개발 시스템 엔지니어는 Commander 내부의 이중 삼중 EKF 검증 스택과 하드웨어 지연에 대비한 센서 타임아웃 트리거 조건들을 명확하게 통달해야 최적화된 고급 위치 제어 알고리즘 및 궤적 추적 연산을 안정적으로 런타임 적용할 수 있을 것이다.
