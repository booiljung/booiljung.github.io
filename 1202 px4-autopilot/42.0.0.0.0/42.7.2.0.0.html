42.7.2 STATUSTEXT 메시지를 통한 심각도(Severity) 수준 분류 로직

1. 개요 및 STATUSTEXT 메시지의 아키텍처적 목적

비행 제어 컴퓨터(Flight Controller)를 구동하는 임베디드 소프트웨어 내부에서 발생하는 다양한 디버깅 이벤트, 센서 캘리브레이션 진행 상태, 상태 머신 전환(State Transition), 그리고 치명적인 하드웨어 중단 오류 등은 32비트 비트마스크(Health Bitmask)만으로는 그 발생 원인과 구체적 맥락을 온전히 GCS에 표현하기 어렵다. 이를 보완하기 위하여, MAVLink 프로토콜은 사람이 직접 읽을 수 있는 ASCII 형식의 텍스트 문자열을 기체에서 지상 관제 시스템(Ground Control Station, GCS)으로 지속 전송할 수 있는 STATUSTEXT 메시지(Message ID: 253) 표준 규격을 최상위 수준표로써 제공한다.

특히 STATUSTEXT 패킷은 단순한 텍스트 배열 나열에 그치지 않고, 각 메시지 인스턴스마다 8단계의 엄격한 심각도(Severity) 수준 플래그를 헤더에 할당한다. 이를 통해 수신 측인 GCS가 텍스트를 파싱하기도 전에 선행적으로 해당 알림의 시각적 인터페이스 차단율, 청각적 피드백 수위를 동적으로 렌더링하고 라우팅(Routing)할 수 있도록 고안된 핵심 관제 피드백 메커니즘이다.

2. 심각도(Severity) 분류 체계의 척도 및 MAV_SEVERITY 열거형

STATUSTEXT 패킷의 첫 번째 데이터 페이로드는 severity 필드(1바이트 uint8_t 레지스터)로 구성되며, 이는 컴퓨터 시스템의 전통적인 유닉스 시스템 로그 규격인 Syslog(RFC 5424) 심각도 분류 알고리즘을 그대로 차용한 MAV_SEVERITY 열거형(Enum) 자료구조에 1:1로 매핑된다. 주요 수준(Level) 계층도는 다음과 같이 정의된다.

MAV_SEVERITY 레벨 지수패킷 내 코드(Hex)설명 및 GCS 렌더링 동작 기준
MAV_SEVERITY_EMERGENCY0시스템이 완벽하게 사용 불가능한 직전 상태. 즉각적인 공중 추락 위험, 배터리 서지 등에 의한 전원 차단 임박 경고. (가장 최우선 순위의 심각함)
MAV_SEVERITY_ALERT1즉각적인 오퍼레이터의 조종 개입(Override)이 개시되어야 하는 상태. 다중 센서 결함 및 심각한 액추에이터 컴포넌트 오류 발현.
MAV_SEVERITY_CRITICAL2치명적 프로세스 상태. 배터리 고갈 임박에 따른 RTL 고정 진입 상태, EKF2 추정기의 노이즈 발산(Divergence) 등.
MAV_SEVERITY_ERROR3제어 에러 상태. 보조 센서 타임아웃, 커맨더 컴포넌트에서의 사용자 명령(Command) 강제 거부(NACK) 등의 일반적 오류.
MAV_SEVERITY_WARNING4주의 경고 상태. 간헐적 GPS HDOP 품질 저하 등 당장은 추락하지 않으나 향후 치명적 문제로 전개될 가능성을 시사.
MAV_SEVERITY_NOTICE5일반적이나 비행 흐름상 중요한 상황(Normal but significant). 자동 비행 모드 전환 성공 타임스탬프, 센서 캘리브레이션 진입 등.
MAV_SEVERITY_INFO6일상적인 시스템 정보 프로토콜 기술. 초기화 부트 프로세스 콘솔 내용, 하드웨어 버스 프로빙(Probing) 상태 결과.
MAV_SEVERITY_DEBUG7시스템 개발자 디버깅용 메시지. 일반적인 관제 릴리즈 UI 환경에서는 필터(Filtered)되어 콘솔 파일 로깅(ULog)에만 기록됨.

3. PX4-Autopilot의 로깅 및 텍스트 메시지 송출 아키텍처 인프라

PX4-Autopilot 펌웨어 내부(v1.14 아키텍처 기준)에서는 mavlink_log_info(), mavlink_log_critical(), mavlink_log_emergency()와 같은 전역 로그 매크로 함수 집합(src/modules/mavlink/mavlink_log.h 헤더 인터페이스)을 제공하여, 수십 개의 개별 애플리케이션 계층 전반에서 텍스트 로그를 완벽히 비동기적(Asynchronous)으로 발생시킬 수 있도록 시스템을 디자인하였다.

sequenceDiagram
    participant A as Application (예: Commander, EKF2, VIO)
    participant U as uORB (mavlink_log 통합 토픽)
    participant M as MAVLink Streamer 데몬 스레드
    participant G as QGroundControl (지상 관제)

    A->>U: mavlink_log_critical("Reject Auto: GPS Signal Lost") 호출
    U->>U: 심각도(Severity=2)와 원문 텍스트를 메시지 Ring Buffer Queue에 보관
    M->>U: Log 토픽 구독 (이벤트 기반 비동기 Non-blocking 대기)
    M->>M: 50바이트 ASCII 배열 청크 단위로 클리핑(Clipping) 및 패키징
    M->>G: STATUSTEXT 전송 (Severity=2)
    G->>G: Payload Parse 및 Severity 추출
    G-->>G: GCS UI 색상 테마 '적색(Red)' 오버라이드 및 TTS 보이스 알람 송출
    G-->>G: Mission Log 텍스트 창에 영구 기록

이 PX4 로깅 아키텍처의 가장 강력한 지향점은, 수많은 구동 어플리케이션 스레드가 MAVLink 송신 모듈 포트나 UART 하드웨어 장치에 직접 의존(Coupling)하지 않는다는 것이다. 디바이스 드라이버나 제어 로직은 단지 uORB의 중앙 mavlink_log 토픽 버퍼에 메시지 구조체만 밀어 넣으면(Push) 그 즉시 할 일이 끝난다. MAVLink 스트리머 컴포넌트는 자신이 여유가 되는 스케줄 폴링 레이트 내에서 큐(Queue)를 소모하며 GCS로 패킷을 캡슐화하여 전송한다. 이는 1kHz 이상으로 구동되어야 하는 실시간성(RTOS)의 핵심 레이어(예컨대 내부 제어 루프 레이트)에 어떠한 입출력 블로킹 지연(I/O Blocking delay)도 유발하지 않는 극한의 효율적인 결합형 설계이다.

4. QGroundControl(QGC)의 심각도 파싱 로직 및 시각화 피드백 메커니즘

QGC 백엔드의 Vehicle.cc 및 UI 프론트엔드의 알림 센터 모듈은, 원격지에서 STATUSTEXT 메시지가 도달하면 메시지 구조체 내부의 severity 값을 기반으로 조건 분기 필터를 작동시켜 조종자(Operator)에게 전달할 사용자 경험(UX) 및 경보의 수위를 엄격하게 차등 적용한다.

  1. 시각적(Visual) 인터페이스 경보 오버라이드:
  • CRITICAL, ALERT, EMERGENCY 수준의 심각한 오류(0~2 레벨): QGC 상단 중앙 도구 모음(Toolbar)의 시스템 뱃지가 적색 격자로 점멸하며 애니메이션 효과를 내뿜고, 메시지 히스토리 컴포넌트 창이 다른 모달 창을 무시하고 최상단에 강제 팝업(Pop-up Overlay)되어 사용자의 인지 우선순위를 강탈한다.
  • WARNING (4 레벨): 황색(Yellow) 뱃지를 표출하여 무인기에 잠재적 부하가 생기고 주의를 요망함을 무언으로 지시한다.
  • INFO / NOTICE (5~6 레벨): 녹색(Green) 혹은 회색 테마 플래그로 렌더링되며, 상단 패널을 간헐적으로 가리기보다는 하단 확장 로그 뷰어에 조용히 순차적으로 적재(Appended)된다.
  1. 청각적(Audio) 경보 연동 (TTS Engine 아키텍처): QGC는 탑재 운용체제 레벨의 텍스트 음성 변환(TTS, Text-to-Speech) 코어 클래스를 통하여, WARNING以上の 등급인 경우 텍스트 원문 내용을 즉석에서 사람의 오디오 메시지로 신서시스(Synthesis)하여 스피커로 고출력 스피커 채널로 출력한다. 이는 조종자가 GCS 터치스크린 패널에서 잠시 시계(Eye-sight)를 떼고 육안 비행(VLOS)에 집중하거나 물리적 조종기(Remote Controller) 스틱을 다루고 있는 멀티태스킹 상황에서도, 청각 채널만으로 기체의 페일세이프(Failsafe) 오류 원인을 직관적으로 인지하도록 보장하는 결정적인 인적 대응(Human In the Loop) 트리거 요건이다.

5. Ardupilot 대비 PX4 텍스트 로깅 시스템 철학의 특징점 분석

Ardupilot (APM 기반 빌드) 및 보조 컴패니언 라이브러리 역시 동일한 MAVLink STATUSTEXT 정규 프로토콜 패킷을 빌드하므로, 코어 통신망(Tx/Rx) 계층의 송출 구조는 거의 유사하게 구동된다. 그러나 GCS에 도달했을 때의 텍스트 토큰 어휘 구성과 백엔드 메시지 노이즈 철학에서 약간의 분기점이 두드러진다.

  • Ardupilot (APM C++ 프레임워크): 인터럽트 수준의 디바이스 드라이버에서 STATUSTEXT를 다루는 GCS_SendText 함수군이 매우 고도로 투영되며, 내부 디버그 성격의 확장 C 구조체 상태나 이진 정보의 나열이 포맷팅(Formatting) 문자열로 포함되는 빈도가 잦다. 특히 비행 전 PreArm 검증 실패 시, “PreArm: ~” 으로 시작하는 수동 트리거 실패 메시지가 수십 줄에 걸쳐 순차적으로 GCS 콘솔 화면을 가득 채우며 엔지니어 측면의 정보량을 선호하는 특징이 있다.
  • PX4-Autopilot: 전체 스택에서 메시지의 과도한 남발(Spamming)을 억제하고 “최소한의 가독성을 갖추어, 엔지니어가 아닌 일반 오퍼레이터-유저 친화적인” 텍스트 출력을 유도하려는 명문화된 아키텍처 디자인 지침을 가진다. 따라서 내부의 메모리 포인터 구조나 난해한 변수값 덤프(Dump)보다는 [cal] calibration started: 2 accel, GPS Signal Lost: EKF2 Failsafe Start 와 같이 엄격히 규격화되고 축약된 상태 서술자(Descriptor) 고지가 주력으로 포진되어 텔레메트리 화면을 깔끔하게 구축한다.

6. 결론

PX4-Autopilot의 MAVLink STATUSTEXT 메시지에 탑재된 심각도(Severity) 기반 분류 로직 프레임워크는 기체의 단순 1차원적인 상태 덤핑을 넘어서, 인공지능 기반 드론 및 유무인 복합 체계의 시각·청각적 휴먼-컴퓨터 상호작용(HCI, Human-Computer Interaction)의 근간을 단단히 지탱한다. 8단계의 세밀화된 열거형 버퍼 구조와, 메인 루프를 방해하지 않는 uORB 미들웨어 기반 비동기 스트리밍 결합 시스템, 그리고 GCS 소프트웨어 측의 TTS 파싱 인프라가 하나의 거대한 시스템 레벨 사상으로 정밀하게 맞물려 작동하고 있다.
이를 통해 극한 환경에서 다중 무인기를 통제하는 인적 자원(조종사)의 비행 중 화면 분석에 대한 인지 부하(Cognitive Load)를 극한으로 최소화하고, 다급한 구조적 개입이 불가피한 치명적인 고장에 대한 인간의 반응 지연(Reaction Delay) 갭을 메가헤르츠 단위로 메워넣는 현대 항공 우주 통신 설계의 모범 사례로 군림하고 있다.
