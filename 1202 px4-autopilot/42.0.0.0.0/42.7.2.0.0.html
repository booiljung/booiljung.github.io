## 0.1 STATUSTEXT 메시지를 통한 심각도(Severity) 수준 분류 로직

PX4-Autopilot과 지상 관제 시스템(Ground Control Station, GCS) 간의 상태 통보 및 경고 안내는 MAVLink 프로토콜의 STATUSTEXT 메시지를 통해 수행된다. 시스템의 무결성(Integrity) 확인 또는 비행 전 점검(Pre-flight Check) 과정에서 발생하는 다양한 로그 및 상태 메시지는 단순한 텍스트 전달을 넘어 중요도를 기준으로 철저히 분류되어 보고되어야 한다. 본 절에서는 STATUSTEXT 메시지에 명시되는 심각도(Severity) 분류 체계와 PX4-Autopilot의 내부 처리 로직을 상세히 분석한다. (기준 버전: PX4-Autopilot v1.14 및 QGroundControl v4.3)

0.1.1 STATUSTEXT 메시지의 구조 및 Severity 열거형(Enum) 체계

MAVLink 프로토콜에서 정의하는 STATUSTEXT 메시지(#253)는 최대 50바이트의 문자열(Text) 정보와 함께 메시지의 위급성 수준을 나타내는 severity 필드를 포함한다. 이 필드는 해당 메시지가 시스템의 안전과 핵심적으로 직결되는지, 오류로 인한 즉각적인 비상 조치가 필요한지, 혹은 단순한 시스템 상태 알림(Information)인지를 지시한다.

심각도 수준은 MAV_SEVERITY 열거형에 기초하여 정의되어 있으며, 이는 RFC 5424(Syslog 프로토콜) 표준의 심각도 수준(Severity Levels) 체계를 따른다.

MAV_SEVERITY값설명QGroundControl에서의 처리
MAV_SEVERITY_EMERGENCY0시스템을 사용할 수 없음 (System is unusable). 치명적인 패닉 로직과 결합됨.적색 경고 알림(Critical Alert) 및 음성 강제 출력(TTS) 발생
MAV_SEVERITY_ALERT1즉각적인 조치가 필요함 (Action must be taken immediately).적색 경고 및 음성 강제 출력
MAV_SEVERITY_CRITICAL2심각한 조건 (Critical conditions). 하드웨어 고장 등에 직면한 상태.적색 경고 알림
MAV_SEVERITY_ERROR3오류 조건 (Error conditions). 센서 데이터 유실 등.주황색 또는 적색 경고 알림
MAV_SEVERITY_WARNING4경고 조건 (Warning conditions). 비행 전 점검 실패 등 주의 촉구.주황색 알림 타일 표시
MAV_SEVERITY_NOTICE5정상이지만 중요한 알림 (Normal but significant condition).단순 텍스트 로깅 및 회색/흰색 알림
MAV_SEVERITY_INFO6유용한 정보성 메시지 (Informational messages). 파라미터 저장 완료 등.GCS 콘솔 뷰어에 텍스트만 출력
MAV_SEVERITY_DEBUG7디버그 단위의 메시지 (Debug-level messages). 개발자 전용.무시되거나 콘솔 하위 탭에만 출력

위 테이블에 명시된 심각도 객관화 기준에 따라 PX4 내부 시스템은 적절한 MAV_SEVERITY 값을 산출하여 STATUSTEXT 구조체에 실어 전송하게 된다.

0.1.2 PX4 Commander 모듈에서의 메시지 발행 메커니즘

PX4-Autopilot의 핵심인 Commander 메인 루프에서는 지속적인 주기로 시스템 상태를 정밀 모니터링한다. 특히, 사용자가 아밍(Arming) 시퀀스를 개시할 때 PreFlightCheck::preflightCheck() 함수가 인보크(Invoke)되어, 각종 센서, 배터리 상태, 시스템 모드, RC 연결 무결성 등을 순차적으로 엄격히 검증한다. 검증에 실패 시, mavlink_log_critical() 또는 mavlink_log_warning() 등의 매크로 함수를 호출하여 GCS로 특정 텍스트 메시지를 배포한다.

PX4의 독특한 아키텍처 특성 상 시스템 프로세스에서 MAVLink 메시지를 직접 구성하여 원격 터미널로 전송하지 않는다. 대신, uORB(Micro Object Request Broker) 통신 기반의 펍/섭(Pub/Sub) 미들웨어 토픽인 mavlink_log (데이터 구조적으로는 mavlink_log_s 활용)를 이용해 브로드캐스트한다.

sequenceDiagram
    participant Sensor as 센서 컴포넌트 (IMU, GNSS)
    participant Commander as Commander 모듈
    participant uORB as uORB 버스 (mavlink_log 토픽)
    participant MAVLink as MAVLink 송출 모듈 (mavlink_messages)
    participant GCS as QGroundControl

    Sensor->>Commander: 센서 캘리브레이션 누락 혹은 잡음 발생 보고
    Commander->>Commander: Pre-flight Check 로직 검증 실패 판정 산출
    note over Commander: 위협 인지 및 심각도 결정<br/>(예: MAV_SEVERITY_CRITICAL)
    Commander->>uORB: orb_publish(ORB_ID(mavlink_log), &log_msg)
    uORB->>MAVLink: 구독 중인 메시지 수신(Poll 이벤트) 감지
    MAVLink->>MAVLink: STATUSTEXT MAVLink 패킷으로 직렬화 (Serialization)
    MAVLink->>GCS: 텔레메트리 채널(UART/UDP)을 통한 전송
    GCS->>GCS: 메시지 수신 후 Severity 파싱 및 UI/사운드 이펙트 라우팅
    GCS-->>운용자: "Preflight Fail: Accel uncalibrated" 시각/청각 경고 발송

이 과정을 요약하자면 시스템 내의 특정 모듈이 uORB의 mavlink_log 토픽을 발행하면, MAVLink 네트워크 스트리밍을 단독 제어하는 mavlink_main.cpp 태스크가 이를 비동기 수집(Subscribe)하고 최종적으로 STATUSTEXT 구조로 패키징하여 QGroundControl로 전파한다. 이러한 디자인 패턴은 코어 로직과 통신 프로토콜 직렬화를 완전히 분리하고자 하는 PX4의 설계 원칙과 완벽하게 부합한다.

0.1.3 ArduPilot 대비 메시지 처리 규칙의 설계 지향점 비교

ArduPilot 체계와 PX4 기반 환경은 동일하게 MAVLink 개방형 프로토콜을 계승하지만, 실전 적용에 있어서 텍스트 기반 상태 로깅을 대하는 철학적 차이가 존재한다.

  1. 상태 천이 안내 메시지의 이벤트 범람(Flooding) 제어
    ArduPilot 아키텍처는 비행 모드 변경 상황이나 내비게이션 상의 웨이포인트 접근 등의 빈번한 이벤트에 대해서도 STATUSTEXT를 지속 발행하여 지상 기지에 알리는 경향이 상대적으로 짙다. 반면에 PX4-Autopilot은 GCS 채널 상의 메시지 플래딩 현상을 최소화하기 위해 노력한다. 필수불가결한 이벤트(즉, 치명적 하드웨어 이탈, 이착륙 가능 여부 등)에 자원과 심각도를 집중시키고 잉여의 MAV_SEVERITY_INFO 발행은 최대한 지양하도록 통제된다.
  2. QGroundControl의 연동 및 의존성 최적화
    일반적으로 선호되는 GCS인 QGroundControl은 그 내부 파서의 특성 상 MAV_SEVERITY_WARNING 구역 이하의 메시지에 대해 즉각적이고 다소 공격적인 오디오 렌더링(Text-To-Speech) 및 적색 모달(Modal) 차단 위젯 렌더링을 시도한다. PX4 코어 생태계는 관제 소프트웨어가 지닐 행동 패턴을 고려하여, 기체의 물리적인 완전성이 크게 훼손될 가능성이 높거나 당장 이륙이 차단된 특단 상황에서만 EMERGENCY~CRITICAL 심각도를 보수적으로 투입한다.

0.1.4 시각적/청각적 제어 시스템의 지상 관제 연동 아키텍처

QGroundControl 프론트엔드 모듈(v4.3 기준)의 구조를 분석해 보면 Vehicle C++ 객체 내 _handleStatustext 메서드가 프로토콜 언래핑 및 데이터 파싱을 담당한다. 추출된 severity 수치에 따라 UX/UI 로직 분할 처리가 수행된다.

  • 음성 경보 합성기(Speech Synthesizer) 연계: MAV_SEVERITY_ERROR 단계 이상의 위험성이 감지될 경우, 프레임워크 내 QGC::AudioOutput 서비스 시스템이 즉각적으로 텍스트 구조를 변환하여 백그라운드 스피치 출력 큐(Queue)에 데이터를 주입한다. 시스템의 즉응성 확보는 기체가 가시선 외(BVLOS) 환경을 활공 중인 긴급 국면에서 다층 계기판을 관측하지 않고도 운영 주체가 상황을 즉각인지하게 함을 목표로 한다.
  • 파일 로깅 기로 및 사후 감시(Forensic) 지원: 추출 메시지는 _handleStatustext 내부 로깅 미들웨어를 관통해 .tlog 파일 (Telemetry Log)에 타임스탬프 순서대로 기입 저장된다. 추후 발생된 문제 현상에 관한 포렌식 및 원인 분석(Root Cause Analysis) 진행 시 해당 텍스트 레코드들이 증빙 데이터로 직결된다.

0.1.5 소스 코드 레벨 분석 및 헬퍼 구문 확장 구현

운영체제(NuttX/Linux) 레벨의 각기 타겟에서 임베딩 C/C++ 개발의 편의성을 높이기 위해 src/modules/mavlink/mavlink_log.h 헤더 파일 내 헬퍼 매크로(Helper Macros)를 광범위하게 배포한다. 아래의 의사 코드(Pseudo-Code) 형태의 C++ 구문은 센서 초기화 실패 상황에 대한 대응 전략을 상징적으로 보여준다.

#include <systemlib/mavlink_log.h>

// 퍼블리셔(Publisher) 핸들러 초기화 선언
orb_advert_t mavlink_log_pub = nullptr;

// 센서 무결성 테스트 프로시저 내 적용
if (!sensor_mag_valid) {
    // 최고 등급 경고 발동 조치. 안전 비행 불가 (Severity: MAV_SEVERITY_CRITICAL)
    // 수식에서의 세로바 문자는 \vert 등으로 인코딩 되거나 그대로 전송 가능함
    mavlink_log_critical(&mavlink_log_pub, "Preflight Fail: Compass missing or invalid (%d)", error_code);
    return false;
} else if (sensor_mag_noisy) {
    // 요주의 조건 성립. 잠재적 오류의 소지 다분 (Severity: MAV_SEVERITY_WARNING)
    mavlink_log_warning(&mavlink_log_pub, "Preflight Warning: High magnetic interference detected");
} else {
    // 무결성 패스 증거 통지 (Severity: MAV_SEVERITY_INFO)
    mavlink_log_info(&mavlink_log_pub, "Preflight: Compass check passed normally.");
}

이렇듯 헬퍼 매크로는 서브루틴의 명칭 자체에 목적하고자 하는 severity 레벨을 이미 명시하고 있다. 코드 가독성을 현격하게 높임과 동시에 C 기반 라이브러리의 printf 스타일의 버퍼 포맷팅 및 메모리 관리를 안전하게 핸들링한다. 이로써 단말 모듈의 어떠한 상황적 변화에도 시스템 레벨의 위험 감지 등급을 적절히 주입하여 GCS와의 직관적인 통신 연결을 매끄럽게 유지할 수 있도록 설계된 것이다.
