42.7.3.1 비동기 원격 프로시저 호출(Asynchronous RPC) 관점에서의 명령어 전달 및 수락 기한(Deadline) 검증

지상 관제 시스템(GCS; Ground Control Station) 혹은 외부 파트너 컴퓨터(ROS2 Companion Computer)에서 무인항공기(UAV) 방향으로 명령을 송신하고 그 결과를 회신받는 체계는, 분산 시스템 아키텍처 관점에서 비동기 원격 프로시저 호출(Asynchronous RPC; Remote Procedure Call)의 대표적인 형태라 할 수 있다. 무선 링크 기반의 드론 통신 환경은 패킷 손실률이 높고 지연 시간(Latency)이 가변적이다. 따라서 명령어의 신뢰성 있는 전달과 수락 기한(Deadline)의 엄격한 관리는 기체의 안전 보장에 가장 기본이 되는 아키텍처이다.

본 절에서는 최신 PX4 펌웨어(v1.14 기준) 및 QGroundControl(v4.3 기준) 환경을 중심으로 MAVLink 및 ROS2 연동 시점에서의 비동기 RPC 동작 원리와 명령어 타임아웃(Timeout)/기한 검증 메커니즘을 심층 분석한다.

1. MAVLink 프로토콜의 비동기 RPC 설계 철학

네트워크 상단에서 볼 때, QGroundControl(클라이언트)이 PX4-Autopilot(서버)에게 COMMAND_LONG (혹은 COMMAND_INT) 메시지를 송신하는 행위는 특정 제어 함수를 원격으로 “호출“하는 행위이다.

하지만 임베디드 실시간 운영체제(NuttX 등)를 구동하는 기체의 특성상, 수신 즉시 모터 시동(Arming)이나 모드 변경(Mode Switch)과 같은 무거운 상태 천이 로직을 동기식(Synchronous)으로 끝마치고 응답을 반환할 여력이 없다. 따라서 PX4는 다음과 같은 비동기(Asynchronous) 디커플링 구조로 설계되어 있다.

  1. 요청 수신: mavlink_receiver.cpp 모듈이 COMMAND_LONG 패킷을 해석하여 즉시 uORB 메시징 버스 상에 vehicle_command 토픽을 발행(Publish)한다. 수신 모듈의 역할은 여기까지이며, 더 이상 호출 블로킹(Blocking)에 걸리지 않는다.
  2. 명령 처리: 커맨더(Commander.cpp) 스레드 등에서 해당 토픽을 폴링 스케줄에 따라 가져가서 유효성 점검, 파라미터 점검 등을 수행한 후 실제 기체 상태를 변경한다.
  3. 결과 반환: 처리가 완료되면 vehicle_command_ack 내부 토픽을 재발행하고, 이를 송신 담당 모듈(mavlink_messages.cpp)이 낚아채어 GCS 방향으로 MAVLink COMMAND_ACK 메시지를 재전송한다.

이러한 논블로킹(Non-blocking) 비동기 파이프라인의 핵심은 GCS 측에서 명령어의 처리 완료 시점을 보장받기 위해 “수락 기한(Deadline)” 및 “유효 타임아웃(Timeout)” 메커니즘을 엄격히 구현해야 한다는 점이다.

2. 수락 기한(Deadline) 검증 및 재전송(Retry) 아키텍처

비동기 시스템에서 GCS는 자신이 보낸 RPC 요청이 공중에서 유실되었는지, 기체가 처리 중인지, 아니면 기체가 응답을 보냈으나 ACK가 유실되었는지를 즉각적으로 구별할 수 없다. 이를 위해 QGroundControl 매니저 클래스 내부에는 각 명령 별로 개별적인 타이머(Timer)가 구동된다.

sequenceDiagram
    participant GCS as QGroundControl (v4.3)
    participant UART as Radio Link (RF)
    participant PX4 as PX4 Commander Subsystem

    GCS->>UART: COMMAND_LONG 전송 (Attempt 1)
    activate GCS
    Note over GCS: Deadline Timeout 타이머 시작 (예: 500ms)
    UART-->>PX4: [Packet Lost]
    Note over GCS: 타임아웃 만료! (No ACK Recv)
    GCS->>UART: COMMAND_LONG 재전송 (Attempt 2)
    Note over GCS: 타이머 재시작
    UART->>PX4: 패킷 정상 수신
    activate PX4
    PX4-->>PX4: uORB 변환 및 상태 머신 실행
    PX4->>UART: COMMAND_ACK (Result: 0=Accepted)
    deactivate PX4
    UART->>GCS: ACK 수신
    deactivate GCS
    Note over GCS: 비동기 RPC 트랜잭션 정상 종료

QGC는 명령 송신 후 규정된 기한(일반적으로 500ms ~ 1000ms 내외) 이내에 COMMAND_ACK를 받지 못하면 실패 횟수(Retry Count)를 차감하며 동일 패킷을 재시도한다. 완전히 횟수가 소진되면 UI 스레드 상에 “명령 전송 실패“를 팝업으로 표출하게 된다.

2.1 PX4 소스 코드 상의 Stale 데이터 검증 로직

기체 역시 지연된 패킷(Stale Command)이 뒤늦게 도착하여 기체가 원치 않는 타이밍에 예기치 못한 거동을 하는 사태를 막아야 한다. PX4 내부 vehicle_command_s 구조체는 고정밀 타임스탬프(timestamp) 필드(단위: 마이크로초)를 포함한다.
Commander 모듈은 명령을 수신한 시점의 내장 클록 시간(hrt_absolute_time())과 vehicle_command 구조체의 생성 타임스탬프를 비교하여, 너무 오래 전(예: 수 초 전)에 유입되었거나 통신 지연으로 인해 스케줄러 큐에 정체되어 있던 명령은 과감하게 폐기(Drop)하도록 구현되어 있다. (방어적 프로그래밍 관점)

3. ROS2 (uXRCE-DDS) 브리지 환경에서의 기한(Deadline) 관리

자율 에이전트 비행을 구현하기 위해 ROS2의 Micro XRCE-DDS 프로토콜을 이용할 때, 명령어 전달 체계의 신뢰성은 DDS의 QoS(Quality of Service) 프로파일 설정을 통해 조율된다.

ROS2의 Action Server 개념을 차용하거나, 오프보드 제어를 위해 VehicleCommand 메시지를 직접 퍼블리시할 경우, 송신 노드(ROS2) 측에서는 DDS QoS 정책 중 Deadline QoS와 Reliability = RELIABLE 속성을 복합적으로 조정하여, 설정된 기한 내에 명령 토픽이 안정적으로 전파되었는지 모니터링한다. 무선 와이파이(Wi-Fi)나 LTE 환경 상의 ROS2 네이티브 연동 시에는, 기존 시리얼 기반의 타임아웃 로직보다 이와 같은 미들웨어 수준의 세밀한 네트워크 패킷 재송신 및 트래픽 폭주(Congestion) 제어 관리가 더욱 강력한 효과를 낸다.

4. Ardupilot 대비 관제 시스템의 차이

Ardupilot을 주로 관제하는 Mission Planner(C# 기반) 역시 비동기 MAVLink 처리를 수행하지만, 재시도 주기의 스케줄러 및 타임아웃 큐 대기 시간 설정값에 차이가 있다. Ardupilot은 특정 제어 명령(예: 캘리브레이션 등 시간이 오래 걸리는 프로시저)에 대해서 COMMAND_ACK 중 ’처리 중(IN_PROGRESS)’이라는 응답 상태를 적극적으로 반환하여 GCS가 타임아웃에 의해 불필요한 재전송을 반복하지 않도록 통제하는 기법(MAV_RESULT_IN_PROGRESS)을 상대적으로 광범위하게 활용하는 편이다.

반면 PX4 생태계 시스템은 특정 트리거 명령어들을 uORB에 즉결 캐싱(Caching)하고 최대한 단시간에 처리하여 직관적이고 상태 확정적인 ACCEPTED 혹은 DENIED를 피드백하는 구조적 단결성(Simplicity)을 띄기 위해 노력한다.

5. 요약: 아키텍처 관점에서의 안정성 보장 지침

  1. GCS나 오프보드 에이전트 개발자는 PX4 API나 MAVSDK를 호출할 때 동기식 콜(Synchronous block method)을 설계하기보다는, 항상 Callback 기반 혹은 Future/Promise 디자인 패턴 구조를 통해 백드롭(Backdrop) 타임아웃 기한을 설정해야 한다.
  2. 장거리 혹은 노이즈가 많은 비가시권(BVLOS) 비행에서 원격 프로시저 호출(RPC)의 데드라인 제어를 간과할 경우, 재전송 폭풍(Retry Storm)으로 텔레메트리 대역폭이 소진될 가능성이 대두된다.
  3. PX4 펌웨어 수정 단계에서는 새로 추가하는 커스텀 vehicle_command가 혹여나 1,000 마이크로초 이상의 지연 시간을 유발하지 않도록 연산 주기를 최적화해야 RPC 응답 실패를 회피할 수 있다.
