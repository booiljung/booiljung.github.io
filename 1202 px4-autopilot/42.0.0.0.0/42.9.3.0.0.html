42.9.3 에러 메시지 렌더링 방식 및 워크플로우의 시스템 자원 점유 비교

비행 전 점검(Pre-flight Check) 과정이나 비행 중 발생하는 예외 상황에서 지상 관제 시스템(GCS)으로 상태를 보고하기 위해서는 에러 메시지의 신속하고 정확한 전달이 필수적이다. 본 절에서는 PX4-Autopilot(기준 펌웨어 버전: PX4 v1.14)과 Ardupilot의 에러 메시지 생성 및 전송 아키텍처를 비교하고, 텍스트 렌더링 워크플로우가 비행 제어기(FC)의 시스템 자원(CPU, 메모리) 점유율에 미치는 영향을 분석한다.

1. MAVLink STATUSTEXT 메시지의 역할과 한계

두 자동비행장치 모두 GCS(QGroundControl 등)와의 텍스트 기반 사용자 피드백 통신 표준으로 MAVLink 프로토콜의 STATUSTEXT (문자열 최대 50바이트 길이) 메시지를 사용한다.

하지만 커맨더 모듈(Commander Module) 또는 각종 센서 드라이버에서 문제 상황(예: “Preflight Fail: Gyro not calibrated”)을 감지했을 때, 이 문자열을 메모리 상에 조합(String Formatting)하고 MAVLink 패킷화하여 시리얼 큐에 담는 내부 워크플로우에는 아키텍처적 뚜렷한 차이가 존재한다.

2. PX4-Autopilot의 이벤트(Events) 기반 에러 렌더링 아키텍처

PX4는 비행 제어 및 텔레메트리 관련 자원 점유를 분리하기 위해 철저한 독립형 스레드 아키텍처를 지향한다.

2.1 분리된 mavlink_log 및 이벤트 처리 구조

PX4는 개별 모듈에서 직접 직렬 변환(Serialization)을 수행하지 않는다. 오류나 경고가 발생했을 때 모듈 구동 스레드에서는 mavlink_log_info() 또는 새로운 이벤트 인터페이스(events::send())를 호출한다.

  1. 메시지 발행(Publishing): 커맨더 모듈은 오류 텍스트를 uORB 메시지 형태로 메모리 버퍼 파이프(mavlink_log 토픽)에 발행만 하고 즉시 다음 검증 로직으로 복귀한다.
  2. 비동기 렌더링(Asynchronous Rendering): 백그라운드에서 동작하는 전용 MAVLink 스트리밍 스레드(mavlink_receiver 및 송신 루프)가 uORB로부터 에러 로그를 읽어와 MAVLink 패킷으로 직렬화 및 스트리밍한다.

2.2 자원 점유(Resource Footprint) 특성

  • 문자열 조합(Formatted Print): sprintf 등을 통한 문자열 조합 비용이 높은 편임에도 불구하고, 이를 즉시 네트워크 버퍼로 렌더링하지 않고 uORB 링 버퍼(Ring Buffer)를 활용하기 때문에 실시간 제어 스레드의 반응성(Responsiveness) 저하를 완벽히 막아낸다.
  • CPU 점유: 에러 전송 중에도 비행 제어 루프의 타이밍 틱(Timing Tick)인 주기적 마이크로초 지연(Micro-latency) 현상이 발생하지 않는다.

3. Ardupilot의 순차적(Sequential) 텍스트 전송 워크플로우

Ardupilot 아키텍처에서는 전통적으로 GCS 클래스 객체를 직접 호출하는 방식을 사용한다.

3.1 GCS_MAVLINK::send_text 직접 호출

주요 센서나 Pre-flight 로직에서 오류가 감지되면, 코드 베이스 상에서 전역 객체화된 gcs().send_text(MAV_SEVERITY_CRITICAL, "Error %s", reason)를 직접적으로 호출한다.

  1. 직접 렌더링(Direct Rendering): 함수 내부에서 곧바로 가변 인자(Variadic Arguments)를 파싱 및 문자열 완성(Format) 단계를 거친다.
  2. MAVLink 큐 삽입: 완성된 문자열은 GCS 클래스의 스케줄링 큐에 즉시 삽입된다.

3.2 자원 점유 특성

  • 단일 스레드 병목 현상(Single-thread Block): 만약 send_text() 함수 호출 빈도가 동시에 급증하거나 문자열 버퍼링 과정이 복잡한 경우, 해당 로직을 수행 중인 현재의 메인 루프 실행 시간이 약간 지연(Penalty)될 가능성이 잔존한다.
  • 오버헤드 관리: 이를 상쇄하기 위해 Ardupilot은 send_text() 루틴이 너무 자주 불리지 않도록 인터벌 제한(Rate Limiting/Throttling)을 두어 메인 루틴 방어를 수행한다.

5. 설계 철학 결론 종합

두 가지 접근법 모두 자원의 심각한 낭비를 유발하지는 않지만, 설계 기준점(Design Paradigm)에서 근본적 차이를 보인다. PX4-Autopilot은 uORB 비동기 메시징 아키텍처에 의존하므로, 오류 렌더링에 필요한 여분의 마이크로초(Microseconds) 통산 시간마저 비행 제어 핵심 루프로부터 원천 차단(Decoupling)하였다. 이는 확장 칼만 필터(EKF2)나 제어 루프가 완벽한 실시간성을 유지하도록 도우나 메모리 버퍼 오버헤드는 소폭 상승한다.

반면 Ardupilot은 GCS 객체 체인 및 포인터 직접 제어를 통한 로직 일원화로 코드 가독성과 메모리 구조의 단순화 측면에서는 유리하나, 무수히 많은 에러 로그가 복합적으로 터져 나오는 스웜(Swarm) 제어 및 복합 페이로드 결함 상황 등에서는 STATUSTEXT 포맷팅 자체로 인해 미세한 타이밍 지연 가능성을 내포한다.
