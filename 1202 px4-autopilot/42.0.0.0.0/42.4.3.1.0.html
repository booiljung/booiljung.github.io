42.4.3.1 측정치 갱신(Measurement Update) 시의 잔차 공분산(Residual Covariance) 일관성 평가

확장 칼만 필터(EKF)의 수학적 연산은 크게 예측(Prediction)과 갱신(Update) 두 단계로 나뉜다. 센서로부터 새로운 GNSS(Global Navigation Satellite System) 데이터가 인입되면, EKF2는 상태 예측치와 실제 관측치의 차이인 잔차(Residual, 혹은 이노베이션)를 구하고 이를 통해 시스템의 상태를 보정하는 측정치 갱신(Measurement Update) 과정을 수행한다.

이 과정에서 가장 치명적인 문제 중 하나는 필터 알고리즘이 스스로 발산(Divergence)하여 잘못된 수학적 확신을 갖는 것이다. 이를 방지하기 위해 PX4-Autopilot의 EKF2 아키텍처는 측정치 갱신 직전 잔차 공분산(Residual Covariance, S)의 핵심적인 일관성(Consistency) 평가를 시행한다.

1. 잔차 공분산(Residual Covariance, S)의 정의

측정치 갱신 단계에서 칼만 이득(Kalman Gain, K)을 계산하기 위해 필수적으로 요구되는 행렬이 바로 잔차(이노베이션) 공분산 행렬 S이다. 수식으로는 다음과 같이 정의된다.

S = H \cdot P \cdot H^T + R

  • H: 관측 모델의 자코비안(Jacobian) 행렬
  • P: EKF 내부의 사전 상태 오차 공분산(State Error Covariance)
  • R: 센서(GNSS) 관측 노이즈 공분산(Measurement Noise Covariance)

이 수식의 의미는 “현재 EKF가 가진 상태 불확실성(P)“과 “새로 들어온 GPS 데이터의 불확실성(R)“을 합쳐 실제 측정치가 존재할 만한 통계적 영역(S)을 구축하는 것이다.

2. PX4-Autopilot의 잔차 공분산 일관성 검증 한계치 검사

PX4-Autopilot 펌웨어(버전 1.14 기준)의 EKF2 모듈은 수학적 무결성을 담보하기 위해 행렬 S에 대한 일관성 검사를 수행한다. 즉, 행렬 S의 분산(대각 성분)이 음수(-)가 되거나, 과도하게 커지는 등 물리적으로 불가능한 수학적 모순이 발생했는지를 점검한다. 관련 코드는 주로 ekf2_main.cpp 및 각 센서 융합 메서드(예: fuseGps()) 내부에서 확인할 수 있다.

  1. 양의 정부호(Positive Definiteness) 점검: S의 대각 성분(분산)은 반드시 0보다 커야 한다. 수치적 불안정성(Numerical Instability)이나 과도한 자코비안 선형화 오차로 인해 이 값이 0 이하로 계산되면, EKF2는 즉각 갱신을 취소하고 분산 재초기화(Covariance Reset)를 수행한다.
  2. 이노베이션 일관성 테스트(Innovation Consistency Test): 측정된 잔차(이노베이션, y)가 S 행렬로 규정된 타원체(Ellipsoid) 한계선(Gate) 내에 위치하는지 점검한다. 수식으로는 정규화된 이노베이션 \gamma = y^T S^{-1} y 가 지정된 게이트 임계치 한계점(EKF2_GPS_P_GATE 등)보다 작은지 확인한다.
flowchart TD
    A[GNSS 데이터 수신 및 예측 상태 벡터 도출] --> B[R 행렬 및 H 행렬 구성]
    B --> C[잔차 공분산 계산: S = H*P*H^T + R]
    C --> D{일관성 평가 1: S의 대각 성분이 양수인가?}
    D -- No (음수/0) --> F[수치적 결함 감지: 칼만 갱신 중단 및 P 행렬 강제 리셋]
    D -- Yes --> E{일관성 평가 2: 노멀라이즈 인베이션(y^T * S^-1 * y) < Gate?}
    E -- No (Gate 통과 실패) --> G[다중 경로/글리치 감지: 융합 보류 (Measurement Rejected)]
    E -- Yes --> H[칼만 이득(K) 계산 및 상태 벡터/P 행렬 갱신 (Fusion Success)]

3. 일관성 붕괴(Inconsistency)의 원인과 대응 파라미터 튜닝

GNSS 융합 시 일관성 붕괴가 자주 일어난다면 필터 설정에 문제가 있을 확률이 높다. 개발자는 다음과 같은 PX4 파라미터를 점검해야 한다.

  • EKF2_GPS_DELAY: GPS 수신기와 Flight Controller(Pixhawk) 간의 지연 시간. 이 값이 실제 물리적 모델보다 너무 적거나 크면 예측(P)과 관측(R)의 타이밍이 어긋나 S의 일관성이 무너진다. (기본값: ~110ms)
  • EKF2_EVP_NOISE, EKF2_EVS_NOISE: EKF 내부적으로 가정하는 최소한의 GPS 위치 및 속도 관측 노이즈. u-blox가 보고하는 정확도가 이 파라미터 임계치들보다 더 낮을 때(센서가 스스로 “완벽하다“고 착각할 때) 이 값들이 R 행렬의 하한선 역할을 해주어 S가 수치적 발산으로 가지 않도록 돕는다.

4. Ardupilot 아키텍처와의 비교

  • Ardupilot: Ardupilot의 EKF3도 동일하게 잔차(Innovation) 분산의 양의 정부호성 검사를 실시하며 실패 시 분산 값을 리셋(Reset)하는 구조를 지닌다. 다만, Ardupilot은 다중(복수) IMU 기반 EKF 코어를 독립적으로 병렬 구동하여 한 코어에서 일관성이 깨지면 즉시(Lane Switching) 가장 건강한 코어로 주도권을 넘기는 방식에 비중을 둔다.
  • PX4-Autopilot: PX4도 Multi-EKF 체계(버전 1.13 이후)를 도입하여 병렬 인스턴스를 운영하지만, 단일 인스턴스 내에서 잔차 공분산의 일관성 테스트 실패 횟수 통계를 관리하여 인스턴스의 ’건강 점수(Health Score)’를 산출하고, 이를 토대로 메인 EKF 코어를 스위칭한다는 점에서 아키텍처적 차별성이 있다.

5. QGroundControl에서의 모니터링 관제

관제 시 QGroundControl(버전 4.3 기준)은 EKF2의 일관성 평가 실패를 개발자 위젯의 EKF 상태 패널을 통해 표출한다. estimator_status의 pos_horiz_test_ratio (위치 검사율) 및 vel_test_ratio (속도 검사율)가 이노베이션 게이트 S를 기준으로 1.0에 도달하면 적색 알람이 표출된다.

실제 비행 중 이 Test Ratio가 1.0을 넘어서는 스파이크가 빈번하다면, 기체의 진동(Vibration)이 너무 심하여 P 행렬 자체가 거대해져 있거나, 하드웨어 컴파스(Compass)의 방위각과 GNSS 트랙이 어긋나 잔차(y)가 폭증하는 징후이므로 로그 파일 분석(Flight Review)이 필수적으로 요구된다.
