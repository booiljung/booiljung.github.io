42.2.2.1.1 전이 행렬(Transition Matrix)의 특이점(Singularity) 회피 및 결정론적 회귀 보장

1. 개요

유한 상태 머신(Finite State Machine, FSM)을 모델링할 때 전이 가능성을 행렬(Matrix) 형태로 구성할 수 있다. 무인기(UAV)와 같은 고신뢰성 제어 시스템에서 FSM이 특이점(Singularity), 즉 “어떤 안전한 상태(State)로도 빠져나갈 수 없는 교착 무한 루프“에 빠지는 것은 치명적이다. 본 절에서는 PX4-Autopilot의 커맨더(Commander) 모듈이 상태 전이 행렬의 특이점을 어떻게 수학적/논리적으로 회피하고, 예기치 못한 센서 붕괴 상황에서도 반드시 안전한 페일세이프(Failsafe) 상태로 수렴하는 결정론적 회귀(Deterministic Regression)를 보장하는지 분석한다.

2. 전이 행렬의 특이점(Singularity)과 악순환

PX4 기체의 상태 전이는 유효한 트리거(Trigger) 조건들 간의 매핑으로 이해할 수 있다. 특이점은 주로 중복된 페일세이프 조건의 모순에서 발생한다.

예를 들어:

  • 기체가 AUTO_MISSION 중 배터리 부족을 감지하여 RTL (Return to Launch) 모드로 전이하려 시도한다. (A \rightarrow B 전이)
  • 그러나 동시에 GPS 신호가 끊겨 RTL 모드 진입 조건인 ’글로벌 위치 유효성’이 실패한다. (B \rightarrow C 전이 강제)
  • 다시 ALT_HOLD로 전이하려 하나, 고도계 이중화 매칭이 실패하여 고도 제어조차 불가능하다. (C \rightarrow D 전이 강제)

이때 적절한 전치 행렬이나 예외 복구 분기(Branch)가 없다면 상태 머신은 A \rightarrow B \rightarrow A \rightarrow B 등 유효성을 만족할 때까지 순환 참조를 반복하며 특이점에 수렴한다. 이 시간 동안 기체의 엑추에이터 출력은 멈추거나 통제력을 잃어 추락하게 된다.

3. 특이점 회피를 위한 결정론적 회귀(Deterministic Regression)

PX4-Autopilot은 상태 간의 우선순위를 엄격히 둔 폴백(Fallback) 계층 구조를 통해 특이점(Singularity)을 회피한다. 전이가 거부(Deny)될 때마다, 커맨더(Commander) 트리거는 다음 우선순위의, 조건 제약이 더 적은 안전 상태로 회귀(Regression)하도록 하드코딩되어 있다. 이 전이는 무조건 단방향(Directed)으로 이루어지며 사이클을 형성하지 않는다.

graph TD
    A[초기 페일세이프 발생<br/>Mission 실패] --> B{RTL 모드 시도}
    
    B -- GPS 정상 --> C[RTL 상태로 전이 및 복귀]
    B -- GPS 불가 --> D{Hold (Loiter) 시도}
    
    D -- 로컬 위치/광학 흐름 정상 --> E[Hold 상태로 전이 및 대기]
    D -- 센서 전체 붕괴 (특이점 위협) --> F{Altitude / Stabilized 회귀}
    
    F -- 고도계 정상 --> G[Descend (천천히 자동 하강)]
    F -- 고도계까지 고장 --> H[비상 엔진 정지<br/>Flight Termination / Parachute]
    
    C -.- I((결정론적 회귀 완료))
    E -.- I
    G -.- I
    H -.- I

이 다이어그램이 보여주듯, 가장 우측에 위치한 상태(State)들은 센서 의존성이 거의 제로에 수렴하는 단방향 리프(Leaf) 노드이다. PX4 상태 머신의 전이 행렬은 항상 가장 안전하고 센서 의존성이 적은 리프 노드로 확정 도달(Deterministic Guarantee)하도록 성질을 가진다.

4. PX4 펌웨어 소스 코드 레벨 분석

결정론적 회귀는 state_machine_helper.cpp 내부의 중첩된 switch-case 및 if-else 블록에서 우선순위가 높은 모드부터 순차적으로 시도(Try-Fall-Through)하는 기법으로 구현된다.

// PX4 Commander의 결정론적 회귀(Fallback) 논리 추상화
void commander_failsafe_transition() {
    // 1순위: RTL 시도
    if (can_enter_RTL()) {
        set_nav_state(NAVIGATION_STATE_AUTO_RTL);
        return;
    } 
    // 특이점을 막기 위해 루프를 돌지 않고 즉시 하위 안전 모드로 Fallback
    // 2순위: Hold
    else if (can_enter_Hold()) {
        set_nav_state(NAVIGATION_STATE_AUTO_LOITER);
        return;
    }
    // 가장 의존성이 낮은 조건
    // 3순위: 긴급 하강(Descend) 또는 비행 종료(Termination)
    else {
        // 이 블록은 실패 조건을 가지지 않음 = 무조건 수렴 보장(Deterministic)
        set_nav_state(NAVIGATION_STATE_DESCEND);
    }
}

위와 같은 계단식(Cascading) 조건문은 전이 행렬 관점에서 대각선 아래의 하삼각 행렬(Lower Triangular Matrix) 형태를 띠게 강제하므로 고유값에 의한 무한 순환을 피하고 특이점을 발생시키지 않는다.

5. Ardupilot과 페일세이프 전이 구조의 철학적 차이

Ardupilot의 경우 특정 센서 실패 시 사용자(조종사)에게 통제권을 돌려주는(Manual Override) 로직과 비상 착륙 로직이 결합되어 있다. AP_System의 페일세이프 모듈은 이벤트를 큐(Queue)에 넣고, 조종사의 스틱 반응 유무에 따라 후속 전이를 결정하는 유연한 접근을 시도한다.

반면, PX4-Autopilot은 기계적인 자율성(Autonomy)과 체계적 회귀 보장에 더욱 방점을 둔다. 조종사의 개입이 당장 어렵거나 연결이 끊어진 환경을 기본 페일세이프 공간으로 가정하고, 상태 전이의 끝단(Terminal State)이 절대로 블로킹(Blocking)되지 않도록 프로그래밍된 룰(Rule)에 의해 최하위 모드로 즉각 전이시킨다. 이는 상용 무인기가 보장해야 하는 결정론적(Deterministic) 안정성의 모범 사례이다.

6. 결론

PX4-Autopilot 시스템에서 유한 상태 머신의 전이 특이점을 예방하는 아키텍처는 수학적 행렬 공간에서 절대적인 싱크(Sink) 노드(예: Descend, Termination)를 두는 것과 동일하다. 개발자는 임의의 커스텀 비행 모드를 시스템에 추가할 때, 기존의 페일세이프 콜 스택(Call Stack)과 회귀 루틴을 끊어먹는 역방향 전이(Backward Transition)나 무한 조건을 설계하지 않도록 극도로 주의하라. 모든 상태 전이는 최종적으로 안전하게 바닥(Ground)으로 도달한다는 전제를 만족해야 한다.
