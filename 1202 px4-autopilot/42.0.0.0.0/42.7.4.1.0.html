42.7.4.1. 확인응답 패킷 손실 시의 지수 백오프(Exponential Backoff) 재전송 알고리즘 연계

1. 개요

무인기-지상 관제 시스템(PX4-QGC) 간의 무선 통신 환경(Telemetry, Wi-Fi 등)은 본질적으로 불안정하다. 기체(PX4-Autopilot)가 지상 관제 시스템(QGroundControl)으로부터 시동(Arming)이나 비행 모드 전환과 같은 중요 제어 명령(COMMAND_LONG)을 성공적으로 수신하고 실행했음에도 불구하고, 기체가 반환하는 확인 응답(COMMAND_ACK) 패킷이 GCS로 도달하는 도중 무선 링크 환경의 악화로 인해 손실(Packet Loss)될 수 있다.

이때 GCS가 맹목적으로 짧은 주기로 명령을 재전송하면 네트워크 혼잡(Congestion) 상태를 가중시키고 시스템의 치명적인 대역폭 고갈을 야기할 수 있다. 본 절에서는 이러한 상황을 해결하기 위해 GCS와 제어 SDK(MAVSDK 등)에 구현된 지수 백오프(Exponential Backoff) 기반의 적응형 명령 재전송(Retry) 아키텍처를 심층 분석한다. (최신 PX4 Firmware 및 QGroundControl v4.3 기준)

2. 무선 통신 환경에서의 ACK पै킷 손실 메커니즘

MAVLink는 기본적으로 UDP 또는 시리얼(Serial) 기반의 비연결 지향(Connectionless), 비신뢰성(Unreliable) 프로토콜로 작동한다.

  1. GCS는 MAV_CMD_COMPONENT_ARM_DISARM 명령을 PX4로 전송한다.
  2. PX4의 Commander 모듈은 명령을 수락하고 COMMAND_ACK (ACCEPTED) 패킷을 회신하지만, 간헐적 전파 방해로 인해 GCS에 수신되지 못한다.
  3. PX4 기체는 실제로 시동 상태(Armed State)로 전환하여 프로펠러를 회전시키지만, GCS 화면 상에는 여전히 “Disarmed” 상태로 남아있거나 타임아웃 오류 창이 발생하는 심각한 불일치(Inconsistency)가 발생한다.

3. 지수 백오프(Exponential Backoff) 재전송 아키텍처

GCS 패러다임에서 가장 교과서적인 해결책은 타임아웃(Timeout) 발생 시 동일한 명령을 재지시하는 것이다. 그러나 동시에 여러 텔레메트리 트래픽이 몰리는 상황에서 고정된 주기의 재전송(예: 0.5초마다 무한 반복)은 무선 채널의 패킷 충돌(Collision)을 극대화시킨다. 이를 방지하기 위해 이더넷의 CSMA/CD에서 기원한 지수 백오프 알고리즘이 MAVLink 명령 트랜잭션 매니저에 도입된다.

3.1 QGroundControl(QGC) 내부의 백오프 설계 (C++ 소스 분석)

QGC 소스 트리(src/Vehicle/Vehicle.cc 및 src/MissionManager/MissionManager.cc)를 살펴보면, MAVLink 트랜잭션 관리자가 재전송 로직을 관장한다.

  • 초기 타임아웃(Base Timeout): 일반적으로 500ms(0.5초)로 설정된다.
  • 재전송 횟수 제한(Max Retries): 무한정 차단(Blocking)되는 것을 막기 위해 통상 3~4회로 제한한다.
  • 백오프 계수(Backoff Factor): 타임아웃이 발생할 때마다 다음 대기 시간은 이전 대기 시간의 2배(지수적 증가)로 늘어난다. (0.5초 \rightarrow 1.0초 \rightarrow 2.0초)

3.2 MAVSDK를 통한 프로그래밍 지원

C++ 또는 Python 기반의 자율 에이전트 제어 라이브러리인 MAVSDK 내부의 Action 또는 Mission 플러그인 구현체 역시 내부적으로 이 지수 백오프 로직을 탑재하고 있다.
명령 지시 후 std::future나 Promise-Async 패턴으로 Non-blocking 대기를 수행하며, 응답 불가 시 내부적으로 백오프 타이머를 비동기 콜백 체인으로 걸어 안전하게 재전송을 시도한다.

4. Ardupilot과 PX4 GCS 환경의 재전송 처리 차이

지상 관제 시스템의 명령 트랜잭션 관점에서는 PX4와 Ardupilot 모두 COMMAND_LONG / COMMAND_ACK 프로토콜을 공통으로 사용하므로 GCS 측(QGroundControl 활용 시)의 백오프 로직은 대동소이하게 작동한다.
다만, **Ardupilot 전용 GCS인 Mission Planner(C# 구현)**의 경우, 전통적으로 지수 백오프 모델보다는 선형적인(Linear) 폴링 타임아웃 구조를 기반으로 사용자에게 에러를 직접적으로 빠르게 반환하는 경향이 짙다. QGC 방식의 지수 백오프는 혼잡 네트워크에서 통신 복원 능력이 더욱 우수하지만, 사용자가 시동 실패를 인지하기까지 최대 트랜잭션 타임아웃 합계(예: 0.5+1+2 = 3.5초)만큼의 응답 지연(Lag)을 체감하게 만드는 단점(Trade-off)도 존재할 수 있다.

5. 시스템 설계 시사점

PX4나 ROS2 브리지를 기반으로 직접 GCS 류의 어플리케이션을 개발할 때, 개발자는 반드시 MAVLink 명령 송출 함수(mavlink_msg_command_long_send 등)를 Fire-and-Forget(쏘고 잊기) 방식으로 구현해서는 안 된다. 상태 머신 객체와 타이머를 연동하여 확인 응답(ACK)을 대기하는 큐(Queue)를 구현해야 하며, 이 때 백오프 타이머 로직을 적극적으로 차용해야만 군집 비행(Swarm) 환경 등에서 RF 채널의 가용 스펙트럼이 붕괴되는 현상을 예방할 수 있다. 참고 문헌: Reliability in Wireless Sensor Networks using Exponential Backoff.
