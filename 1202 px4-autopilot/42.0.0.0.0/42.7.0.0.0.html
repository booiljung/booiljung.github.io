42.7 GCS와의 관제 연동: MAVLink 상태 보고 및 승인 프로토콜의 정밀 분석

유무인 복합 체계(MUM-T) 및 장거리 자율 비행(BVLOS) 임무가 증가함에 따라, 무인항공기(UAV)와 지상 관제 시스템(GCS: Ground Control Station) 간의 신뢰성 있는 상태 동기화가 필수적이다. 비행 전 점검(Pre-flight Check) 단계에서 기체 내부의 센서와 시스템이 자체적으로 안전성을 판단하는 것뿐만 아니라, 이 상태를 GCS에 완벽하게 보고하고 GCS 운용자(혹은 관제 서버)로부터 최종 비행 승인을 이끌어내는 상호 통신 아키텍처가 동반되어야 한다. 본 절에서는 PX4-Autopilot과 QGroundControl 간의 MAVLink 프로토콜 기반 상태 보고 및 승인 체계의 핵심 동작 원리를 소스 코드 단위에서 심층 분석한다. (기준 버전: PX4-Autopilot v1.14.0)

1. MAVLink SYS_STATUS 및 HEARTBEAT 아키텍처 원리

PX4-Autopilot의 운영체제인 NuttX 위에서 구동되는 여러 모듈(센서 구동기, EKF2 상태 추정기, 배터리 모니터 등)은 각자의 건강 상태(Health Status)를 비동기적으로 uORB 메시지(예: sensor_preflight_mag, vehicle_status 등)를 통해 퍼블리싱한다. 이 내부 데이터들은 GCS로 직접 전송되지 않으며, mavlink 모듈이 통합하여 MAVLink 데이터 스트림으로 직렬화(Serialization)한다.

1.1 HEARTBEAT 메시지를 통한 기본 모드 및 전역 상태 동기화

HEARTBEAT 메시지는 보통 1Hz 주기로 발생하며, 무인기의 현재 비행 모드(custom_mode), 무장 상태(base_mode 내 MAV_MODE_FLAG_SAFETY_ARMED), 그리고 시스템 상태(system_status)를 포함한다. 시스템 상태 벡터(system_status 필드)는 MAV_STATE 열거판(Enum) 값을 지니며, MAV_STATE_UNINIT, MAV_STATE_BOOT, MAV_STATE_CALIBRATING, MAV_STATE_STANDBY (Pre-flight 완료 대기 상태), MAV_STATE_ACTIVE (비행 중) 등으로 구분된다. GCS는 이 스트림을 통해 기체가 현재 비행 준비가 되었는지, 캘리브레이션 중인지를 1초 단위로 트래킹한다.

1.2 SYS_STATUS 를 통한 세부 모듈 건강 보고

GCS가 “어떤 센서 때문에 비행 전 점검(Pre-flight Check)이 실패했는가?“를 구체적으로 파악하기 위해서는 SYS_STATUS 메시지가 사용된다. 이 메시지에는 32비트 비트마스크(Bitmask) 변수인 3총사가 포함되어 있다.

  • onboard_control_sensors_present: 하드웨어적으로 장착 및 인식된 센서/모듈
  • onboard_control_sensors_enabled: 현재 사용하도록 활성화된 센서/모듈
  • onboard_control_sensors_health: 현재 정상 작동 중인 (Pre-flight 검증을 통과한) 센서/모듈

예를 들어 GCS는 present 비트에는 GPS 모듈 트리거가 1로 설정되어 있으나, health 비트에는 0으로 나타나는 것을 교차 검증하여 “GPS 하드웨어는 연결되었으나 3D 픽업(Fix)을 잡지 못해 Unhealthy 상태임“을 GUI에 붉은색 알림으로 시각화한다.

2. GCS 승인 프로토콜 및 커맨드 지연 처리 메커니즘

PX4는 수동 비행 시 조종기의 스위치 조작뿐만 아니라, GCS 시스템에서의 원격 이륙(Takeoff), 임무 시작(Mission Start), 무장(Arming) 명령을 수락하기 위해 결정론적인(Deterministic) 상태 전이 프로토콜을 사용한다.

2.1 COMMAND_LONG 및 COMMAND_INT 트랜잭션

GCS에서 전송되는 MAVLink 관제 명령은 주로 COMMAND_LONG (이전 레거시)이나 COMMAND_INT (고정밀도 위치 포함 포맷) 메시지를 통해 내려보내진다.

비행 전 점검 상태를 강제로 승인(Override)하거나 무장을 명령할 때, GCS는 MAV_CMD_COMPONENT_ARM_DISARM 명령을 전송한다.
비행 제어 컴퓨터 측 commander 모듈은 이 명령을 수신하면 즉각 실행하는 것이 아니라 다음의 검증 단계를 수행한다.

  1. 상태 조건 검증: Commander는 기체의 현재 vehicle_status.nav_state가 무장이 허용되는 상태인지, 그리고 모든 Pre-flight Check(센서 무결성, 지오펜스 로드 완료 여부 등) 플래그가 참(True)인지 평가한다.
  2. 응답(Acknowledge) 메시지 발송: 결과에 따라 COMMAND_ACK 메시지로 응답한다.
  • MAV_RESULT_ACCEPTED: 명령이 정상적으로 수락되고 파라미터가 적용됨.
  • MAV_RESULT_TEMPORARILY_REJECTED: 시스템이 아직 준비되지 않아 거부됨 (예: EKF 정렬 중).
  • MAV_RESULT_DENIED: Pre-flight 실패 등 안전상의 이유로 영구 거부됨. GCS 화면에 자세한 STATUSTEXT 오류 메시지(예: “Preflight Fail: Gyro not calibrated”)를 동반 출력한다.

3. Ardupilot 관제 통신 아키텍처와의 차이 비교

Ardupilot은 MAVLink 프로토콜을 PX4보다 먼저 설계하고 확장하는 데 크게 기여했으며, 라우팅 테이블(Routing Table) 처리와 파라미터 동기화 속도에 최적화된 내부 프로토콜을 갖추고 있다. Mission Planner와 연동될 때 Ardupilot은 GCS 주도적인(GCS-Centric) 의사결정 체계에 유연성을 더 두고 있어, 강제 승인(Force Arming) 기능 파라미터(ARMING_CHECK)를 통해 개별 오류 항목을 GCS 단에서 쉽게 무시하고 강제 이륙할 수 있다.

반면, PX4-Autopilot은 비행 컴퓨터 중심적(Autopilot-Centric) 철학에 기반한다. QGroundControl과 PX4의 연동 환경에서는 안전 임계(Safety-critical) 로직이 기체 내부에 절대적으로 의존한다. 파라미터 CBRK_ (Circuit Breaker) 시리즈를 통하지 않고서는 특정 점검(예: USB 연결 모드 시 암 거부 등)을 소프트웨어 GCS 명령만으로 단기적으로 우회(Bypass)하기가 상대적으로 까다롭다. 이는 상업용 다중 기체 관제나 비가시권 통신망에서 통신 두절로 인한 GCS의 통제력 상실 상황이 발생해도, 기체 자체가 갖는 상태 평가의 독립성과 엄격한 Failsafe 귀환 보장을 유지하기 위한 설계적 결단이다.

4. 관제 통합을 위한 설계 제언

새로운 생태계인 ROS2 및 uXRCE-DDS가 부상하고 있으나, 기존 항공 규법(예: FAA, EASA)의 체계적 인증 관리를 받는 GCS는 여전히 MAVLink 스트림의 보장성에 의존한다. 따라서 GCS 엔지니어나 커스텀 관제 서버 개발자는 무작정 제어 명령을 반복 전송(Spamming)하기보다는, PX4에서 방출하는 SYS_STATUS의 Bitmask 상태와 COMMAND_ACK 트랜잭션을 정밀하게 트래킹하는 비동기 관제 아키텍처(Asynchronous Dispatching Architecture)를 구성해야 한다. MAVLink 프로토콜의 이 승인/거절 피드백 루프야말로 물리적 파손을 미연에 방지하는 최후의 소프트웨어 방어선이다.
