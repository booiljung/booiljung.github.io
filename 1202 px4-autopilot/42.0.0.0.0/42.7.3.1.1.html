42.7.3.1.1. 분산 시스템에서의 클락 스큐(Clock Skew)를 고려한 타임스탬프 기반 명령 유효 기간(TTL) 제어

1. 개요

드론 시스템은 비행 제어기(PX4-Autopilot), 지상 관제 시스템(QGroundControl), 그리고 임무 컴퓨터(Companion Computer)가 네트워크(MAVLink, ROS2)를 통해 상호작용하는 전형적인 분산 시스템(Distributed System) 모델을 따른다. 각 독립적인 컴퓨팅 노드는 자체적인 하드웨어 오실레이터에 다라 구동되는 로컬 클락(Local Clock)을 가지므로 시간이 흐름에 따라 물리적 시간 차이, 즉 클락 스큐(Clock Skew)가 필연적으로 발생하게 된다. (최신 PX4 Firmware v1.14 이상, QGroundControl v4.3 이상 기준)

이러한 현상은 타임스탬프(Timestamp)를 기반으로 제어 명령을 실행하고 검증하는 환경에서, 뒤늦게 도착한 과거의 명령이 현재 시점에 실행되는 치명적인 제어 오류(예: 철 지난 비상 정지 명령의 지연 실행)를 야기할 수 있다. 본 절에서는 PX4-Autopilot 환경에서 클락 스큐를 보정하고 명령 유효 기간(TTL, Time-To-Live)을 안전하게 통제하는 메커니즘을 소스 코드와 아키텍처 관점에서 심층 분석한다.

2. 분산 관제 환경에서의 클락 스큐(Clock Skew) 문제와 위험성

클락 스큐란 네트워크 상의 서로 다른 노드가 유지하는 시스템 시간의 차이를 의미한다. 온도 변화, 전원 불안정, 하드웨어 특성 차이로 인해 각 시스템의 시간은 미세하게 다르게 흐른다(Clock Drift).

  • 지연된 명령의 위험성: GCS에서 전송한 비동기 스레드 기반의 중요 제어 명령(예: MAV_CMD_COMPONENT_ARM_DISARM, MAV_CMD_DO_SET_MODE)이 통신 큐 지연이나 간헐적 무선 링크 단절로 인해 뒤늦게 수신될 수 있다. 타임스탬프 TTL 검증이 누락될 경우, 이미 상황이 종료되었거나 조건이 맞지 않는 상황에서 명령이 즉시 실행되어 추락 등 대형 사고를 유발할 수 있다.
  • 재생 공격(Replay Attack) 취약성: 악의적인 무선 노드가 과거에 정상적으로 승인된 제어 패킷을 캡처하여 동일하게 재전송할 경우, 정밀한 타임스탬프 기반 TTL 검증 윈도우(Validation Window)가 없다면 기체는 이를 유효한 명령으로 착각하여 수행하는 보안 취약점에 노출된다. 참고 논문: Security Analysis of the MAVLink Protocol for Unmanned Aerial Vehicles.

3. MAVLink 시간 동기화(Time Synchronization) 프로토콜

PX4는 이러한 클락 스큐 문제를 원천적으로 해결하고 정밀한 타임스탬프 기반 TTL 제어를 수행하기 위해 MAVLink의 TIMESYNC 메시지(Message ID: 111)를 활용한다. 이는 인터넷의 NTP(Network Time Protocol) 모델과 유사한 방식으로 노드 간 타임스탬프를 주기적으로 동기화한다.

3.1 동기화 메커니즘 아키텍처 설계도

아래는 QGroundControl과 PX4-Autopilot 간의 정밀 시간 동기화 절차를 나타내는 시퀀스 다이어그램이다.

sequenceDiagram
    participant GCS as QGroundControl
    participant PX4 as PX4-Autopilot (MavlinkReceiver)
    
    Note over GCS, PX4: TIMESYNC를 통한 시스템 클락 오프셋 계산
    GCS->>PX4: MAVLink TIMESYNC (tc1=0, ts1=GCS 송신 타임스탬프)
    Note right of PX4: 도착 시점의 기체 시간 기록 (tc1)
    PX4->>GCS: MAVLink TIMESYNC (tc1=수신된 기체 시간, ts1=본래 GCS 타임스탬프)
    GCS->>GCS: RTT(Round Trip Time) 산출 및 Time Offset 계산
    Note left of GCS: Offset = tc1 - ts1 - (RTT / 2)
    GCS->>PX4: [명령 전송 시] 로컬 타임스탬프 + Offset이 적용된 명령 하달

이 프로세스를 통해 QGroundControl은 무선망 전파 지연(Propagation Delay)을 수리적으로 보정하고, 기체와 동기화된 일원화 시스템 타임스탬프를 획득하여 제어 메시지를 패키징한다.

4. 타임스탬프 기반 명령 유효 기간(TTL) 검증 로직 구현 구조

동기화된 시간을 바탕으로 PX4는 수신되는 MAVLink 명령(COMMAND_LONG, COMMAND_INT)의 TTL 유효성을 철저하게 검증한다. PX4 소스 트리 내부의 컴포넌트는 모든 이벤트를 마이크로초 단위의 고해상도 타이머인 hrt_absolute_time()를 기준으로 식별한다.

4.1 소스 코드 레벨 검증 파이프라인 (uORB 연계)

  1. Mavlink 파싱 단계: src/modules/mavlink/mavlink_receiver.cpp 내에서 수신된 명령 패킷을 해독한다. 이때 패킷 자체의 레이턴시를 확인하여 1차적인 유효 기간 초과 여부를 필터링한다.
  2. uORB Topic 변환: 수신된 명령은 vehicle_command uORB 토픽으로 퍼블리싱(Publishing)된다. 이 토픽 구조체 내에는 timestamp 필드(생성 시점)가 필수적으로 기록된다.
  3. Commander 소비 단계: src/modules/commander/Commander.cpp에서 해당 토픽을 구독(Subscribe)하여 상태 머신 액션으로 연결한다. 이때 다음과 같은 TTL 검증을 진행한다:
  • 현재 시간 - vehicle_command.timestamp > 임계값 (예: 500ms) 조건에 성립하면, 해당 명령을 NACK (부정 응답) 처리하고 폐기(Drop)한다.
  • 이는 GCS의 요청이 비록 네트워크를 살아서 거쳐왔으나, 이미 적시성(Timeliness)을 잃어 안전성을 보장할 수 없기 때문이다.

이러한 구조는 지속적인 스트림으로 들어오는 실시간 속도 제어(Offboard/Guided 모드) 데이터에서 더욱 엄격하게 적용된다. ROS2를 통해 uXRCE-DDS 미들웨어로 인입되는 TrajectorySetpoint 메시지는 500ms(0.5초) 이상 갱신 지연 발생 시 (TTL 초과) 즉각적으로 Failsafe 상태로 천이시켜 호버링(Hovering) 혹은 귀환(Return) 모드로 강제 강등시킨다.

5. Ardupilot과 PX4의 TTL 제어 아키텍처 차이

  • PX4-Autopilot: 완전한 비동기식 Publisher/Subscriber 통신(uORB) 아키텍처 특성 상, 모듈 간 큐 메시지 대기 시간 및 처리 지연까지 촘촘하게 고려한다. 메시지가 MavlinkReceiver에서 uORB로 변환되어 Commander에 도달하는 일련의 파이프라인 내에서도 타임스탬프를 기반으로 신선도(Freshness)를 엄격히 감별하는 Fail-Fast(빠른 실패) 설계 철학을 가진다.
  • Ardupilot: 전통적인 모놀리식(Monolithic) 메인 루프 기반 스케줄링 폴링 구조를 상대적으로 많이 유지하므로, 외부 GCS 네트워크 지연에 대한 거시적인 타임아웃(Data Link Loss)에는 엄격하지만, 내부 스레드나 모듈 사이에 메시지가 머무르는 지연 자체를 각 패킷의 고유 TTL 타임스탬프 방벽으로 끊어내는 방식은 PX4에 비해 유연한(관대한) 편이다.

6. QGroundControl 관점의 수명 주기 대응 및 디버깅 전략

지상 관제용 소프트웨어인 QGroundControl 관점에서도 클락 스큐 보정을 위한 정밀한 설계가 포함되어 있다.

QGC 소스 코드 내 src/Vehicle/Vehicle.cc 클래스에서는 연결된 드론 객체별로 MAVLink 시스템 시간 오프셋 변수를 독자적으로 트래킹한다. _timesyncMessageReceived() 타이머 콜백을 통하여 정밀한 클락 스큐 변화량을 흡수(Smoothing) 필터링한다.

  • TTL 만료로 인해 기체 측에서 떨어지는 NACK 응답에 직면할 경우, GCS는 단순히 “명령 실패“만을 출력하는 것이 아니라 디버그 콘솔을 통해 재전송(Retry 백오프 알고리즘) 루프를 통제하고 타임스탬프 스큐 오류를 감시한다.

7. 자율비행 에이전트 개발을 위한 응용 시사점

응용 개발자가 PX4-Autopilot을 이용하여 ROS2 프레임워크 기반의 자율 비행(Computer Vision 기반 충돌 회피, Swarm 군집 비행 등) 컴퓨터를 구성할 경우 시분할 리눅스(Linux) OS의 특성상 I/O 스큐 딜레이가 심하게 튈 수 있다.

이때는 반드시 Micro XRCE-DDS 에이전트 내에서 제공하는 Time Sync 기능을 활성화하여 컴패니언 컴퓨터와 비행 제어기 간 클락 오프셋을 1ms 이내로 단일화해야만 한다. 이를 무시할 경우 정상적으로 산출된 제어 명령(Setpoint)조차 PX4 측의 수신 TTL 검증 로직에 걸려 지속적인 명령 강제 폐기로 불안정한 비행(Jerky movement)을 야기하므로 아키텍처 설계 초기에 반드시 시간 단위의 정렬을 확립해야 한다.
