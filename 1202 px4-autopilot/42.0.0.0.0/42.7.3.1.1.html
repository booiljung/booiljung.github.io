42.7.3.1.1 분산 시스템에서의 클락 스큐(Clock Skew)를 고려한 타임스탬프 기반 명령 유효 기간(TTL) 제어

복수의 컴퓨팅 노드가 네트워크로 연동되는 무인항공기(UAV)와 지상 관제 시스템(GCS; Ground Control Station) 간의 통신은 전형적인 분산 시스템(Distributed System) 아키텍처를 따른다. 이 분산 환경에서 제어 명령을 비동기적으로 주고받을 때 가장 심각한 안전상 위협 중 하나는 네트워크 지연(Latency)에 의해 정체되었던 오래된 명령(Stale Command)이 갑작스럽게 기체로 유입되어 의도치 않은 거동을 유발하는 것이다. 이를 방지하기 위해 명령어에 타임스탬프(Timestamp)와 유효 기간(TTL; Time-To-Live)을 부여하는 제어 메커니즘이 필수적으로 요구된다.

그러나 송신측(QGroundControl 혹은 ROS2 기반 에이전트)과 수신측(PX4 픽스호크 보드) 간의 시스템 클록(System Clock)은 물리적으로 분리된 크리스털 발진기(Crystal Oscillator)에 의해 구동되므로 미세한 주파수 오차에 의한 클락 도메인 분리 현상, 즉 클락 스큐(Clock Skew)가 필연적으로 발생한다. 본 절에서는 최신 PX4 펌웨어(v1.14 기준) 환경을 중심으로, 이러한 클락 스큐 환경 하에서 명령의 TTL을 어떻게 검증하고 안전하게 필터링하는지 심층 분석한다.

1. PX4-Autopilot의 TIMESYNC 동기화 메커니즘

서로 다른 노드 간에 생성된 타임스탬프를 비교하여 해당 명령이 “유효 기한 내에 도착했는가“를 판별하기 위해서는, 송수신 양단의 시간 기준(Time Reference)을 0에 가깝게 일치시키는 작업이 선행되어야 한다. 이를 위해 MAVLink 프로토콜은 TIMESYNC 규격을 제공한다.

PX4의 mavlink_receiver 스레드는 백그라운드에서 QGroundControl(v4.3 기준)과 주기적으로 TIMESYNC 메시지를 핑퐁(Ping-Pong)하여 왕복 지연 시간(RTT; Round-Trip Time)의 절반을 추정한다. 시간 오프셋(Time Offset) 보정이 완료되면, GCS가 전송하는 COMMAND_LONG 내부의 시간 기반 필드와 PX4 내부의 실시간 운영체제(NuttX) 고정밀 타이머인 hrt_absolute_time() 간의 비교가 논리적 타당성을 확보하게 된다.

2. 클락 스큐(Clock Skew)를 가정한 명령 TTL 제어의 수학적 모델

송신 노드 A(GCS)와 수신 노드 B(PX4) 간의 이상적인 기준 시간을 t라고 할 때, 발진기 주파수 차이에 의해 누적되는 클락 스큐(Drift) 오차를 S(t)라 정의한다. 노드 A가 네트워크로 명령 버킷을 전송한 시점의 내장 타임스탬프를 T_{A}, 노드 B에 도달한 시점의 내장 클록 측정값을 T_{B}라고 한다면, 수신단 B가 해당 명령에 대해 인지하는 ‘겉보기 체류 시간(Apparent Age)’ \Delta T_{age}는 다음과 같이 유도된다.

\Delta T_{age} = T_{B} - (T_{A} + Offset_{AB} \pm \epsilon)

명령의 유효를 허용하는 임계 시간(유효 기간, TTL)을 \tau_{max} (예: 500\text{ms} 내외)라고 할 때, 커맨더(Commander) 모듈은 다음 조건을 만족하는 경우에만 명령을 인가(Accept)한다.

\mathit{Condition}: \vert \Delta T_{age} \vert < \tau_{max}

명령이 네크워크 상에서 혼잡(Congestion) 상태에 놓이거나, 무선 링크 절단 후 순간적으로 재연결되는(Burst Reconnect) 경우 \vert \Delta T_{age} \vert \ge \tau_{max} 조건을 만족하게 되며, 해당 제어 모듈은 이를 “Stale(오래된)” 명령으로 판단하고 폐기(Drop) 목록으로 분류한다. 여기서 \epsilon은 분산 시스템 내 클락 보정 오차 한계치로서, 이 수치가 커질수록(즉 TIMESYNC의 신뢰도가 떨어질수록) 정상 명령마저 버려지는(False Positive) 확률이 높아지게 된다.

3. 타임스탬프 필터링 아키텍처 처리 다이어그램

아래 도표는 시스템 간의 클락 동기화를 전제로, 지연된 MAVLink 제어 패킷이 PX4 내부에서 소멸되는 메커니즘을 나타낸다.

sequenceDiagram
    participant GCS as GCS (Clock A)
    participant MAV_RX as PX4 MAVLink Rx
    participant UORB as uORB Bus
    participant CMD_EXEC as PX4 Commander Subsystem (Clock B)

    GCS->>MAV_RX: `TIMESYNC` 교환 (Offset 동기화)
    Note over GCS: Time_A = 1000ms 송신 (TTL=500ms)
    GCS-->>MAV_RX: `COMMAND_LONG` (네트워크 지연 2000ms 발생)
    Note over MAV_RX: 도달 시점 Time_B = 3000ms
    MAV_RX->>UORB: `vehicle_command` 매핑 및 발행
    UORB->>CMD_EXEC: 명령 폴링 수신
    activate CMD_EXEC
    CMD_EXEC-->>CMD_EXEC: hrt_absolute_time() 확인
    CMD_EXEC-->>CMD_EXEC: Delta = 3000ms - 1000ms = 2000ms
    CMD_EXEC-->>CMD_EXEC: Delta > TTL(500ms). 유효 기한 만료 판정
    CMD_EXEC->>UORB: `vehicle_command_ack` (DENIED_STALE)
    deactivate CMD_EXEC

이와 같이 PX4 펌웨어 레퍼런스(특히 vehicle_command 수집 루틴) 내부에는 과도기적 시간차 발생 방지 코드가 강하게 옭아매어져 있다. 만약 \Delta값이 음수로 도출되는 심각한 시각 역전 현상(Time Inversion)이 감지되면, 이는 클락 보정 로직(TIMESYNC) 자체가 파괴되었음을 암시하므로 마찬가지로 패킷을 드롭(Drop)한다.

4. ROS2 기반 자율 제어 환경(Micro XRCE-DDS)에서의 클락 동기화 위협

오프보드(Offboard) 환경으로 진화하면서 MAVLink가 아닌 ROS2 생태계의 uXRCE-DDS 미들웨어가 도입될 경우, 클락 스큐의 양상은 더욱 복잡해진다. ROS2 컴패니언 컴퓨터(라즈베리 파이, 젯슨 Nano 등)는 통상 리눅스 환경의 NTP(Network Time Protocol) 서버나 Chrony 데몬을 통해 독자적으로 UTC 타임을 맞추려는 경향이 있다.

이때 ROS 노드에서 발행하는 타임스탬프 기반 헤더 메타데이터가 픽스호크 메인보드의 NuttX hrt_time과 심하게 괴리되면(예를 들어 부팅 후 픽스호크는 상대 시간, 라즈베리는 절대 시간을 쓰는 경우), DDS 상의 로직 처리가 비정상적으로 만료되거나 퓨전 에러를 낳는다. 따라서 실무 환경 최적화를 위해서는 ROS2 노드의 시계 도메인을 네트워크 시간에 의존케 하지 않고, XRCE 브리지를 통해 uORB timesync_status 토픽을 기준으로 슬레이브 록킹(Slave Locking) 처리하는 특수 동기화 패치 과정이 권장된다.

5. Ardupilot 철학과의 시간 동기화 처리 대비

Ardupilot은 MAVLink 프로토콜의 표준인 TIMESYNC 처리에 대해 유사한 패킷 교환 메커니즘을 지원한다. 하지만 수신 측(기체) 큐 모델에서 타임아웃을 강제로 필터링하는 파라미터 제약(Hard boundary) 구배가 PX4보다 상대적으로 느슨한 구석이 있다. Mission Planner 등 클라이언트에서 발생시킨 낡은 제어 명령이 수신되었을 경우, Ardupilot은 우선순위 큐(Priority Queue) 체계에서 밀어내거나 덮어쓰기(Overwrite)하는 논리로 대응하는 경우가 다수 존재하는데 반해, 엄격한 실시간 객체 버스를 지향하는 PX4 uORB 시스템은 타임스탬프 필드 위반(Timestamp Violation) 시 철저한 “Fail-fast” 원칙에 따라 무응답 혹은 명시적 취소(NACK) 상태 천이로 차갑게 돌아서는 성향을 지닌다.

결론적으로 자율 에이전트 인프라스트럭처를 구성하는 클라이언트 설계자는, 분산 시스템 간 클락 스큐 현상을 무시하고 임의의 무선망(Wi-fi, LTE)에 제어 명령을 푸시(Push)하는 것의 치명적인 비행 위험성을 깊이 직시하여야 한다.
