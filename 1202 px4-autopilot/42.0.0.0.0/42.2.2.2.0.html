42.2.2.2 방향성 비순환 그래프(Directed Acyclic Graph) 기반의 점검 프로세스 종속성 모델링

PX4-Autopilot(기준 펌웨어 버전: v1.14.0 이상)의 Commander 모듈에서 수행되는 비행 전 점검(Pre-flight Check)은 단순한 순차적 실행이 아닌 복잡한 종속성(Dependency)을 지닌 검증 프로세스들의 집합이다. 이러한 종속성을 효과적으로 모델링하고 검증하기 위해 방향성 비순환 그래프(DAG, Directed Acyclic Graph) 개념을 적용하여 아키텍처를 설계하고 해석할 수 있다. QGroundControl(기준 버전: v4.3.0 이상)과 같은 지상 관제 시스템(GCS, Ground Control Station)은 이러한 점검 결과를 수신하여 사용자에게 시각적으로 피드백을 제공한다.

1. 비행 전 점검(Pre-flight Check)에서의 종속성 개요

무인항공기 시스템(UAS, Unmanned Aerial System)의 안전한 구동을 위해서는 다양한 서브시스템(Subsystem)이 정상 작동하는지 확인해야 한다. 센서 캘리브레이션(Sensor Calibration), 추정기(Estimator, EKF2), 전력 시스템(Power System) 등은 독립적으로 동작하는 것처럼 보이지만 실제로는 명확한 선후 관계를 가진다.
예를 들어, 확장 칼만 필터(EKF, Extended Kalman Filter) 기반의 추정기가 유효한 위치 및 자세 데이터를 산출하려면, 우선적으로 IMU, 나침반(Compass), 기압계(Barometer) 및 위성항법장치(GNSS) 등의 초기화 및 데이터 무결성(Integrity) 검증이 선행되어야 한다. 이처럼 구성 요소 간의 “A가 확인되어야 B를 검증할 수 있다“는 관계를 모델링하는 데 DAG가 매우 적합하다.

관련 논문인 “A Formal Verification Approach for Safety-Critical Flight Control Systems” 등의 연구에서도 복잡한 소프트웨어 검증 시 DAG를 활용한 프로세스 모델링 기법의 중요성을 강조하고 있다. 이러한 원리를 바탕으로 전반적인 시스템을 파악하라.

2. 방향성 비순환 그래프(DAG) 구조 모델링

방향성 비순환 그래프를 점검 프로세스에 적용하면, 각 노드(Node)는 개별 점검 항목(Check Item)을 나타내고, 간선(Edge)은 종속성을 의미한다. DAG의 주요 특징인 순환(Cycle)이 없다는 점은 프로세스가 무한 루프(Infinite Loop)나 데드락(Deadlock)에 빠지지 않고 반드시 유한 시간 내에 완료되거나 실패함을 보장한다.

아래는 PX4-Autopilot의 전형적인 비행 전 점검 프로세스 종속성을 단순화한 구조도이다. 이 아키텍처를 통해 종속성을 시각적으로 파악하라.

graph TD
    A[전원 시스템 검증<br>Power System Check]
    B[센서 무결성 및 캘리브레이션 검증<br>Sensor Calibration & Integrity]
    C[비행 데이터 기록 시스템 확인<br>Logger Check]
    D[GNSS 수신 상태 검증<br>GNSS Reception Check]
    E[추정기(EKF2) 상태 검증<br>Estimator Status Check]
    F[안전장치(Failsafe) 설정 확인<br>Failsafe Configuration Check]
    G[최종 시동 승인 검토<br>Final Arming Approval]

    A --> B
    A --> C
    A --> F
    B --> E
    D --> E
    E --> G
    C --> G
    F --> G

이 그래프에서 최종 시동 승인 검토 노드로 진행하기 위해서는 상위 노드인 센서, 추정기, 안전장치 설정 등이 모두 정상 상태(Healthy)임을 통과해야 함을 알 수 있다. 이러한 직관적인 파이프라인(Pipeline) 구조는 개발자나 운영자가 시스템 검증 실패 시 오류의 근본 원인(Root Cause)을 추적(Tracing)하는 데 매우 유용하다.

3. PX4 소스 코드 수준의 종속성 평가 구현

PX4의 최고 수준(Top-level) 상태 머신을 다루는 Commander 하위 모듈에는 각 점검 항목들이 구현된 클래스(Class)와 함수들이 존재한다. Ardupilot 시스템이 AP_Arming 라이브러리를 통해 상대적으로 직렬화된 검사를 수행하는 경향이 있는 반면, PX4는 uORB(Micro Object Request Broker) 메시징 미들웨어를 통해 완전히 비동기적(Asynchronous)으로 서브시스템의 상태를 수집한다.

그러나 논리적 검증(Logical Verification) 과정에서는 DAG 기반의 평가를 적용해야 한다. 즉, uORB 트리를 통해 sensor_preflight_mag, sensor_preflight_gyro 메시지를 먼저 구독하여 유효성을 판별한 뒤, 이를 바탕으로 estimator_status를 검증하는 계층적 확인 절차를 거친다.

아키텍처 관점에서, PreFlightCheck::checkEstimator() 함수가 호출될 때 하드웨어 센서 노드에 해당하는 검사가 실패했다면 EKF 검사 단계를 수행하지 않고 즉시 조기 종료(Early Exit)를 실행하여 프로세서를 절약함과 동시에 선행 종속성 오류를 QGroundControl에 보고하게 된다.

3.1 코드 구조 예시: 종속성에 기반한 판별 로직

이해를 돕기 위해 아래와 같이 종속성을 평가하는 로직의 컨셉을 C++ 형태의 예시로 구성할 수 있다. 실제 PX4 내부에서도 이와 유사한 조건 분기(Conditional Branching) 방식을 취함을 확인하라.

bool PreFlightCheck::run_all_checks() {
    // 1단계 노드 검사 (선행 종속성 없음)
    if (!checkSystemPower()) {
        report_error("Preflight Fail: Power System Error");
        return false;
    }
    
    if (!checkSensors()) {
        report_error("Preflight Fail: Sensor Integrity Error");
        return false;
    }

    // 2단계 노드 검사 (1단계 노드 통과 시에만 수행)
    if (!checkEstimator()) {
        report_error("Preflight Fail: EKF2 Estimator not ready (Depends on Sensors)");
        return false;
    }

    // 3단계 노드 최종 통과 판별
    return validateFinalRequirements();
}

이와 같이 코드 베이스 내부에 내포된 DAG 구조를 명확히 숙지하라. 시스템 아키텍처를 설계할 때 각 하위 모듈이 어떠한 선행 조건(Precondition)을 요구하는지를 명확히 규명하는 것은 소프트웨어 안정성(Software Reliability)을 확보하는 핵심 기술이다.

4. Ardupilot과의 통제 흐름 차이점 및 개선 방향

PX4의 아키텍처는 uORB라는 강력한 발행-구독(Publish-Subscribe) 모델 위에서 비동기 상태를 모으고, Commander 중심에서 통합된 종속성 검사를 실행한다. 이 구조는 서브시스템의 추가와 삭제에 유연하며, 모듈화(Modularity) 측면에서 Ardupilot의 거대 단일 구조 체계에 비해 확장성이 높다.
예를 들어, 어떤 추가적인 기능(예: 시각 관성 주행거리계(VIO, Visual Inertial Odometry) 기반 실내 항법 등)을 DAG 모델에 편입시키려 할 때, 새로운 검사 노드를 추가하고 기존 노드와의 간선(Edge)만 새롭게 정의해 주면 전체 로직이 무리 없이 유지된다.

DAG 기반 점검 프로세스는 향후 모델 기반 시스템 공학(MBSE, Model-Based Systems Engineering)이나 위상 정렬(Topological Sorting) 알고리즘을 강력하게 도입하여 런타임에 동적으로 점검 순서를 최적화하는 단계로 발전할 수 있다.
이를 통해 자율 에이전트(Autonomous Agent) 등 외부 개입 없는 상태에서 기체가 자체적으로 비행 불가 사유를 추론(Reasoning)하고, QGroundControl을 거치지 않더라도 스스로 복구(Self-Recovery) 로직을 구사하는 수준까지 도달할 수 있도록 개선 방안을 연구하라.
