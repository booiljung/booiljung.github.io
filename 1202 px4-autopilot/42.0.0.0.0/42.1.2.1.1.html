42.1.2.1.1. 결정론적 유한 오토마타(Deterministic Finite Automata) 기반의 시동 불능(Disarmed) 상태 고착화 방지망

1. 개요

이 장에서는 PX4-Autopilot의 핵심 안전 메커니즘 중 하나인 결정론적 유한 오토마타(DFA, Deterministic Finite Automata) 기반의 상태 검증 철학을 다룬다. 특히 비정상적인 전압 강하, 센서 고장, 통신 단절 등의 치명적인 오류가 발생했을 때, 기체가 예측할 수 없는 동작(예: 지상 대기 중 예기치 않은 모터 회전)을 방지하기 위해 시동 불능(Disarmed) 상태로 어떻게 “고착화(Lock-down)“되는지 분석한다. 시스템의 각 상태는 매우 제한적이고 명시적인 조건 교집합(Intersection)을 통해서만 전이될 수 있도록 엄격하게 제어된다.

2. 결정론적 유한 오토마타(DFA) 기반 상태 전이 원리

PX4 펌웨어의 Commander 모듈은 내부적으로 기체의 상태를 DFA 모델로 관리한다. DFA는 현재 상태(Current State)와 입력 이벤트(Input Event)에 의해 오직 하나의 명확한 다음 상태(Next State)로만 전이될 수 있는 수학적 모델이다. 비행 제어에서 이 모델을 도입하는 가장 큰 이유는 “예측 가능성(Predictability)“을 보장하기 위함이다.

시동 불능(Disarmed) 상태는 유한 오토마타에서 가장 안전한 상태(Safe State)로 정의된다.
기체가 Armed 상태로 전이하기 위해서는 오직 유효성 검사를 모두 통과한 제한된 이벤트 집합(예: GCS의 Arm 명령 수신 AND 모든 센서 Status 정상 AND GPS 3D Lock 달성)만이 긍정적 전이(Transition)를 유발한다.

2.1 시동 불능(Disarmed) 고착화의 필요성

비행 중에 하드웨어 결함이 발생할 경우, PX4는 페일세이프(Failsafe) 절차에 따라 착륙(Land)하거나 복귀(RTL)를 시도한다. 그러나 기체가 지상에 있는 상태(Landed State)에서 결함이 감지된 경우, 가장 확실한 방어 메커니즘은 시스템을 영구히(또는 리부팅 전까지) Disarmed 상태에 고착화하는 것이다. 이는 작업자가 기체 주변에 있을 때 발생할 수 있는 인명 사고를 원천 차단한다.

    [*] --> Disarmed
    
    state Disarmed {
        [*] --> System_Ready
        System_Ready --> PreArm_Error : 하드웨어/센서 결함(Event)

        PreArm_Error --> System_Ready : 오류 해소

3. Ardupilot의 Pre-arm Checks와의 차별점

DFA를 기반으로 상태를 엄격히 제한하는 것은 Ardupilot 대비 PX4의 확고한 철학적 특징 중 하나이다.

  • PX4-Autopilot: 각 비행 모드(Flight Mode)는 독립적인 상태(State)로 간주되며, 이 상태들의 전이 테이블(Transition Table)이 하드웨어 코드 레벨(src/modules/commander/state_machine_helper.cpp 등)에 하드코딩되어 있다. 즉, 조건이 단 하나라도 누락되면 DFA는 그 입력(Arm command)을 무효 처리(Invalid Input)하고 현재의 Disarmed 상태를 유지(Self-Loop)한다.
  • Ardupilot: 비행 모드에 따른 검사를 유연하게 파라미터화하고, 이벤트 중심(Event-driven)보다는 체크리스트 중심(Checklist-driven)에 가깝게 동작한다. 사용자 측면에서 문제를 회피(ARMING_CHECK = 0)하기 쉽다는 장점이 있으나, “규칙을 강제“하는 오토마타의 엄격함에서는 차이가 있다.

4. Disarmed 고착화를 결정하는 주요 트리거 메커니즘

DFA가 기체를 Lockdown 상태로 전이시키는 주요 치명적 결함(Fatal Error) 조건은 다음과 같다.

  1. EKF2/ECL 발산(Divergence): 확장 칼만 필터의 위치(Position) 또는 속도(Velocity) 혁신(Innovation) 값이 파라미터(예: EKF2_REQ_VDRIFT) 임계치를 초과할 때.
  2. 데이터 링크 타임아웃(Data Link Timeout): 외부 관제 시스템이나 Companion Computer(ROS2/MAVROS)의 Heartbeat가 일정 시간(예: COM_DL_LOSS_T) 동안 유실될 때.
  3. 메모리 스래싱 및 CPU 과부하: NuttX RTOS 상에서 스케줄러의 부하가 극한에 달해, 실시간성이 보장되지 않는 경우(System Load Check).

이러한 조건에 도달하면 GCS(QGroundControl) 화면에 “PREFLIGHT FAIL” 또는 “CRITICAL ERROR“와 같은 메시지가 출력되며, 강제 시동 명령을 내려도 오토마타는 이를 수용하지 않는다.

5. 강제 우회(Override) 메커니즘의 위험성과 올바른 관제 개발

GCS 개발자나 연구자는 테스트 목적으로 이러한 고착화 상태를 임의로 우회하고 싶어 하는 경우가 많다. MAVLink 명령으로 강제 시동 (예를 들어 MAV_CMD_COMPONENT_ARM_DISARM 전송 시 param2: 21196(Force Arming Magic Number) 사용)을 사용하면 일부 제한적인 상황에서는 시동이 걸릴 수 있다.

하지만, 항공전자 안전 무결성 기준(DO-178C 등) 관점에서는 이러한 “강제 우회“는 매우 위험한 백도어(Backdoor)로 간주된다.
ROS2 환경에서 Offboard 제어를 구현할 때도, 기체가 Disarmed 상태를 고수한다면 강제로 명령을 푸시(Push)할 것이 아니라, vehicle_status 토픽을 심층 분석하여 오토마타가 전이를 거부하는 근본 원인(센서 캘리브레이션 불량, GPS 수신 불량 등)을 GCS 운영자에게 명확히 피드백하는 방향으로 아키텍처를 설계하라.

5.1 요약

  • DFA 철학 수용: 시동 검증(Pre-arm Check) 통과 실패 시 기체가 Disarmed 상태로 스스로를 잠그는 행위(Lock-down)는 버그가 아니라 고도로 의도된 안전 메커니즘이다.
  • QGroundControl 소스코드 활용 방안: QGC 소스 내 Vehicle.cc 의 State 머신 구현부를 참고하여, PX4가 보내오는 MAVLink Heartbeat의 base_mode 및 custom_mode 플래그를 정밀하게 파싱하고 사용자 UI와 동기화하라.
