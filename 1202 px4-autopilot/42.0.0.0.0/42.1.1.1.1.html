42.1.1.1.1 다중화(Redundancy) 구조에서의 공통 원인 고장(Common Cause Failure) 배제 논리

1. 개요

현대의 무인항공기(UAV) 비행 제어 시스템은 통상적으로 단일 하드웨어 고장에 대응하기 위해 센서 집합(예: IMU, 나침반, 기압계)을 2개 이상 구축하는 다중화(Redundancy) 설계를 채택한다. 그러나 3개의 독립된 센서 보드를 갖췄다고 할지라도, 특정 외부 노이즈나 구조적 결함이 3개의 센서 모두에 동시에 영향을 미친다면 다중화 설계의 본질적인 목적은 상실된다. 이렇게 하나 이상의 시스템에서 동시에 발생하는 종속적인 고장을 가리켜 공통 원인 고장(Common Cause Failure, CCF) 이라 정의한다.

안전 임계 시스템(Safety-Critical System)인 PX4-Autopilot(v1.14 기준)은 다중화 설계를 맹신하지 않는다. 비행 전 점검(Pre-flight Check) 단계에서 물리적/논리적 결함을 식별할 때, CCF의 잠재적 가능성을 배제하기 위해 이질성(Heterogeneity) 및 분리(Separation) 논리를 강력하게 채택하고 있다.

2. 공통 원인 고장(CCF)의 수학적 모델링 메커니즘

확률론적 위험 평가에서 다중 다중화(Redundancy) 시스템의 고장 확률을 산출할 때 CCF를 제외하면, n개의 동일 센서가 독립적 고장률 \lambda를 가질 때 전체 시스템의 고장률은 \lambda^n으로 극단적으로 낮게 산출된다. 하지만 β-factor 모델 등을 적용하여 CCF 요인을 고려하면 실제 병렬 시스템의 위험도는 훨씬 높아진다.

수학적으로 CCF 요건은 두 가지 원인으로 분류된다.

  1. 내적 공통 결함(Internal Common Cause): 모든 센서를 동일한 로트(Lot), 동일 모델(예: 동일한 제조사의 MEMS 센서 3개)로 사용할 경우 칩셋 내부의 마이크로코드 버그나 특정 진동 주파수에서의 공진이 모든 센서를 동시에 무너뜨리는 현상이다.
  2. 외적 환경 결함(External Common Cause): 단일 전원 공급기 변동(Voltage Ripple), EMI(전자기 간섭), 또는 심각한 프레임 진동이 센서 버스(SPI/I2C)나 센서 회로 전체를 오염시키는 현상이다.

PX4-Autopilot 설계 철학은 이러한 CCF 확률 변수를 최소화하기 위해 소프트웨어적인 보팅(Voting) 메커니즘을 넘어, 초기 아키텍처 및 폼팩터 수준에서 CCF 배제 논리를 설정하라.

3. PX4-Autopilot 내 CCF 배제 설계: 이질성(Heterogeneity)

PX4 기반의 최신 픽스호크 하드웨어(e.g., Pixhawk 6X, Cube Orange)는 이질성(Heterogeneity) 원칙을 도입하여 센서 모듈의 공통 원인 고장을 설계 단계에서 단절시킨다.

  • 서로 다른 물리적 특성의 센서 조합: IMU1, IMU2, IMU3를 구성할 때 모두 같은 제조사의 칩을 사용하지 않는다. 보통 InvenSense 제품(예: ICM-42688)과 Bosch 제품(예: BMI088) 등을 혼용 배치하여, 특정 주파수 대역의 진동이나 자기장에만 취약한 센서 제조상의 CCF를 차단한다.
  • 물리적 분리(Physical Separation): 기판 위에서 가열 소자 곁의 온도 구배나 특정 축의 구조적 공진점(Node)으로부터 벗어나도록 다중 센서의 실장 위치를 상호 이격시키고, 폼팩터 댐핑(Damping) 처리를 달리한다.

4. 비행 전 점검을 통한 공통 원인 고장(CCF) 필터링 아키텍처

소프트웨어적으로 QGroundControl(v4.3)과 PX4-Autopilot의 커맨더(Commander) 모듈은 이륙 전 CCF 징후를 격리(Isolate)하기 위해 다음과 같은 센서 데이터 독립성 검증(Independence Check) 절차를 거친다.

이 다이어그램이 뒷받침하는 핵심 원리는 단순 비교가 아니다. 3개의 센서가 “가장 정확하고 동일한 형태의 노이즈 곡선“을 내뿜는다면, 오히려 PX4 엔진은 이를 개별 센서의 독립적 측정이 아니라, (어떤 외란 체계에 의해) 센서 보드 자체가 통째로 흔들리거나 간섭받는 CCF의 징후로 해석한다.

5. Ardupilot과 PX4의 CCF 대응 양상 차이

결함 허용 관점에서 Ardupilot은 메인 제어 루프 내부에서 EKF 코어(EKF3)들을 병렬 스레드로 복제하여 각각 독립적인 센서 데이터를 취합하는 소프트웨어 다중화에 훌륭한 수준을 구현했다.

다만 PX4-Autopilot은 CCF 대응을 한 계층 더 내려서 sensors 백그라운드 구동 모듈과 uORB 메시지 계층에서 해결하려 든다. PX4의 Sensor Selection 큐는 본질적인 이질성을 유지하기 위해 서로 다른 샘플링 레이트(f_s)를 갖는 센서 간의 데이터 위상(Phase)을 다이내믹하게 조정(Aliasing Mitigation)한다. 이를 통해 동일한 주기로 인한 공명형 CCF(Resonant CCF)까지 배제하며, QGroundControl에는 최종적으로 필터링된 “투표 전 통합 신뢰도(Pre-voted Trust Index)“를 제공하여 시각적 직관성을 향상시킨다.

6. 결론

“여러 개의 센서가 있으니 안전하다“는 고전적인 믿음은 다중화 설계를 무력화하는 공통 원인 고장(CCF) 앞에서 무의미해질 수 있다. PX4-Autopilot이 채택한 하드웨어의 이질적 조합, 소프트웨어 계층 구조에서의 측정치 독립성 검증 메커니즘은 시스템 전체가 동시에 오류 상태에 빠지는 것을 가장 효율적으로 방어하는 수단이다. 최상위 설계자는 Pre-flight Check 로직 내부에서 센서들의 편차(Variance)뿐만 아니라 노이즈 간의 상호 상관성(Cross-correlation)에 기반한 CCF 식별 알고리즘을 시스템에 반드시 투영시켜야 한다.
