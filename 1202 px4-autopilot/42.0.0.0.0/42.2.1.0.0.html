42.2.1 Commander 모듈 내 Pre-flight Check 소스 코드의 제어 흐름(Control Flow) 추적

1. 개요

PX4-Autopilot의 핵심 관리자인 커맨더(Commander) 모듈은 기체 시동 전 안전성을 확보하기 위해 복잡하고 정교한 비행 전 점검(Pre-flight Check) 과정을 거친다. 본 절에서는 PX4 펌웨어 소스 코드(v1.14 기준) 레벨에서 Commander.cpp와 PreFlightCheck.cpp를 중심으로 제어 흐름(Control Flow)을 추적하고, 센서 및 하위 시스템들이 어떻게 순차적으로 검증되는지 상세히 분석한다.

2. 소스 코드 구조 및 진입점(Entry Point)

PX4의 커맨더 모듈은 별도의 NuttX 태스크(Task) 모델 위에서 무한 루프를 돌며 uORB 메시지를 구독(Subscribe)하고 처리한다. 이 루프 내부에서 기체 상태 머신(State Machine)의 전이가 요구될 때(예: 시동 전) PreFlightCheck::preflightCheck() 함수가 호출된다.

전체적인 소스 코드의 디렉토리 구조는 다음과 같다.

  • 진입점: src/modules/commander/Commander.cpp
  • 점검 로직 구현: src/modules/commander/PreFlightCheck/PreFlightCheck.cpp
  • 개별 컴포넌트 점검 클래스: src/modules/commander/PreFlightCheck/ 디렉토리 하위의 AirspeedCheck.cpp, MagnetometerCheck.cpp 등

3. 제어 흐름(Control Flow) 상세 분석

비행 전 점검의 제어 흐름은 다형성(Polymorphism)과 모듈화된 함수 호출의 정교한 오케스트레이션(Orchestration)이다.

  1. 사용자 암/디스암 요청 인지: Commander::run()의 주 루프에서 vehicle_command 토픽 구독을 통해 MAV_CMD_COMPONENT_ARM_DISARM 명령을 인지한다.
  2. 시동 전 평가 (Arming Evaluation): arm_disarm_command() 내부에서 PreFlightCheck::preflightCheck()를 호출한다. 이때 파라미터로 요구되는 비행 모드(Nav State)와 현재 센서 인스턴스, 그리고 엄격성(Strictness) 레벨을 전달한다.
  3. 순차적 컴포넌트 점검: preflightCheck() 함수는 하드코딩된 단일 블록이 아니라, 각 서브 시스템의 점검 루틴을 논리적 AND(&&) 혹은 OR(||)로 결합하여 평가하라. 특정 항목이 실패하더라도 모든 오류를 수집하기 위해 일부 독립적인 점검 엔진은 플래그(Flag) 기반으로 동작한다.

4. 핵심 소스 코드 레벨 분석

PreFlightCheck.cpp 내부의 핵심 제어 흐름은 다음과 같이 간소화하여 살펴볼 수 있다.

bool PreFlightCheck::preflightCheck(vehicle_status_s *status, vehicle_local_position_s *local_pos, ... , bool report)
{
    bool passed = true;

    // 1. 센서 캘리브레이션 및 무결성 점검
    passed &= preflightCheckMagnetometer(status, report);
    passed &= preflightCheckAirspeed(status, report);
    passed &= preflightCheckBaro(status, report);
    passed &= preflightCheckIMU(status, report);

    // 2. 위치 및 상태 추정기(EKF2) 점검
    // 비행 모드가 글로벌 위치를 요구하는 경우에만 작동하는 조건부 로직 포함
    if (requires_global_position(status->nav_state)) {
        passed &= preflightCheckGlobalPosition(status, report);
    }

    // 3. 전원 및 시스템 상태
    passed &= preflightCheckBattery(status, report);

    return passed;
}

제어 흐름 추적의 핵심은 passed &= check_function(); 패턴이다. 이는 비트 연산 기반의 누적(Accumulation) 방식이 아니라, 논리적(Logical) AND를 통해 하나의 컴포넌트라도 검증에 실패하면 passed가 false가 되는 결정론적 구조를 보장한다. (단, C++의 && 단축 평가(Short-circuit Evaluation)와 달리 &=를 사용하거나 상태 변수에 값을 누적시키는 기법을 사용하여, 첫 실패 시점에 점검을 중단하지 않고 전체 에러 메시지를 QGC에 보고할 수 있도록 처리하는 부분도 유의 깊게 관찰하라).

5. Ardupilot과 제어 흐름의 차이점 파악

Ardupilot의 제어 구조(Control Structure)는 AP_Arming::pre_arm_checks() 메인 루틴이 Copter::init_ardupilot()와 같은 기체 전용 코드 베이스 안팎에서 폴링되는 구조이다. 이 과정에서 각 비트마스크(Bitmask) 파라미터를 읽어 들여 if (!(arming_check & ARMING_CHECK_BARO))와 같이 동적으로 브랜칭(Branching)하는 흐름을 갖는다.

반면, PX4-Autopilot은 비트마스크를 활용하되 PreFlightCheck 모듈 클래스의 정적인 계층 구조로 제어 흐름을 분리하여 유지보수성을 극대화한다. 즉, PX4는 하향식 객체 호출(Top-down Function Call)을 철저히 모듈별로 격리하며, Ardupilot보다 분산된 코드가 아닌 모아진 뷰(Centralized View)를 제공한다.

6. 결론 및 개발자 가이드

커맨더 모듈 내의 Pre-flight Check 소스 코드를 추적하는 것은 PX4 개발자가 맞춤형 센싱 장비나 페이로드(Payload)를 연동할 때 필수적인 관문이다. 만약 사용자가 만든 커스텀 장애물 회피 모듈이 이륙 전에 반드시 점검되어야 한다면, 이 제어 흐름에 개입하여 PreFlightCheck.cpp 내에 독자적인 평가 함수를 삽입하고 논리 곱 체인(passed &=)에 연결하라. 이를 통해 코드의 일관성과 무인기 안전성을 동시에 달성할 수 있다.
