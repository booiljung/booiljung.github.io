42.7.4.1.1. 상태 기계 동기화 오류 방지를 위한 멱등성(Idempotency) 기반 명령 재시도 상태 모델링

1. 개요

앞선 장에서 다룬 ’지수 백오프 기반 재전송 알고리즘’에 의해, 지상 관제 시스템(QGroundControl)은 무선 링크가 불안정할 경우 동일한 제어 명령(예: 시동, 비행 모드 전환)을 기체 측에 반복적으로 전송한다. 만약 기체의 상태 기계(State Machine)가 유입되는 명령을 순차적인 ‘토글(Toggle)’ 액션이나 ’무조건적 재수행’으로 받아들인다면, 중복 전송된 시동 명령으로 인해 오히려 시스템이 치명적인 오류 상태에 빠질 수 있다.

이러한 분산 시스템 간의 상태 불일치(Inconsistency) 현상을 원천적으로 방지하기 위해 PX4-Autopilot의 커맨더(Commander) 내부 상태 모델은 수학적 멱등성(Idempotency) 원리를 기반으로 설계되어 있다. 본 절에서는 PX4가 중복된 MAVLink 명령을 어떻게 안전하게 무시하고 동기화를 복구하는지 아키텍처 관점에서 심층 분석한다. (최신 PX4 Firmware 및 QGC 기준)

2. 명령 재전송 환경에서의 멱등성(Idempotency) 개념

분산 컴퓨팅 및 네트워킹 이론에서 ’멱등성’이란, 어떤 연산을 여러 번 반복 적용하더라도 그 결과가 단 한 번 적용했을 때와 동일한 성질을 의미한다 (f(f(x)) = f(x)).

  • 비멱등적 설계의 위험성: 만약 GCS가 “시동(Arm)” 버튼을 눌렀을 때 내부적으로 “현재 상태를 반전시켜라(Toggle)“라는 명령이 날아간다면, 재전송이 발생할 때마다 기체의 상태는 Arm \rightarrow Disarm \rightarrow Arm으로 미친 듯이 요동치게 된다.
  • 멱등적 명령 설계: MAV_CMD_COMPONENT_ARM_DISARM 명령은 파라미터 1번에 명시적인 대상 상태(1=Arm, 0=Disarm)를 담아서 보낸다. 기체에 “목표 상태를 1(Arm)로 만들어라“라고 지속적으로 지시할 경우, 백 번을 수신하더라도 기체는 1(Arm) 상태를 유지해야 한다.

3. PX4 Commander의 멱등성 기반 상태 평가 로직

PX4의 심장부인 Commander 모듈(src/modules/commander/Commander.cpp)은 수신된 제어 토픽(vehicle_command)을 평가할 때 가장 먼저 현재 기체의 vehicle_status를 들여다본다.

3.1 소스 코드 레벨의 방어선 처리 루틴

  1. 명령 수신: Commander::handle_command() 함수 내부의 VEHICLE_CMD_COMPONENT_ARM_DISARM 분기 진입.
  2. 상태 비교 (Idempotency Check): GCS가 요구한 파라미터(param1 == 1.0f)와, 현재 기체의 상태(_vehicle_status.arming_state == vehicle_status_s::ARMING_STATE_ARMED)를 비교한다.
  3. 요구 상태가 이미 달성된 경우 (ACK 복구): 만약 기체가 이미 시동(Armed) 상태인데 또다시 시동(Arm) 명령이 들어왔다면, PX4는 하드웨어 액추에이터 활성화를 이중으로 트리거하거나 “이미 시동이 걸려 있다“는 에러로 NACK 처리를 구태여 하지 않는다. 대신 조용히 MAV_RESULT_ACCEPTED라는 긍정의 COMMAND_ACK 트랜잭션을 다시 퍼블리싱한다.

이 설계 덕분에 GCS의 지수 백오프 기반 재시도(Retry) 폭격은 시스템 안전성에 아무런 위해를 가하지 않으며, 잃어버린 ACK 패킷을 복원하는 순기능만을 온전히 수행하게 된다. (Fail-Safe 및 Idempotent Actor 아키텍처 구현)

4. 비-멱등적(Non-Idempotent) 명령의 한계와 주의점

모든 MAVLink 명령이 멱등성을 지니는 것은 아니다.

  • 카메라 트리거(MAV_CMD_IMAGE_START_CAPTURE)나 서보 모터 직접 제어, 특정 횟수의 웨이포인트 스킵 명령 등은 성격상 누적(Cumulative)될 수 있는 비-멱등적 연산이다.
  • 이러한 명령들의 경우, 네트워크 손실 시 무한정 재전송을 가동하면 동일한 사진이 연속으로 찍히거나 미션이 의도치 않게 대거 스킵될 위험이 있다.
  • PX4는 이에 대해서는 타임아웃 윈도우(TTL)와 명령 시퀀스(Sequence Number)를 교차 검증하여 동일 시퀀스의 트랜잭션 도배를 걷어내는 상태 해시(State Hash) 기반의 방어 로직을 혼용하기도 한다.

5. Ardupilot과 PX4의 중복 상태 처리(Idempotency) 차이점 비교

지상 관제 소프트웨어 층위에서는 MAVLink를 공용으로 쓰기에 비슷하지만, 명령이 펌웨어 코어로 진입했을 때의 철학적 차이가 존재한다.

  • PX4-Autopilot: 완전한 상태 머신(State Machine) 구조에 입각하여 각 모듈(Commander, Navigator 등)의 목표 상태와 현재 상태 사이의 차등(Delta)만을 최소한으로 컴퓨팅하도록 고도화되어 있다. 무한 재시도를 시스템의 동기화 복원으로 부드럽게 흡수하는 측면에 매우 강점을 지닌다.
  • Ardupilot: 모놀리식 메인 루프에서 플래그(Flag) 기반으로 절차지향적 검사를 다수 수행하며, PX4의 Commander처럼 단일화된 거대 상태 통제처가 전담하기보다 각 서브 시스템 루틴별 조건식으로 중복 처리를 방어한다. 여전히 Mission Planner 위주의 환경에서 이미 시동된 상태에서의 시동 재전송 시, 콘솔 창에 불필요한 “Already Armed” 류의 경고 스팸성 STATUSTEXT가 올라오는 경향이 PX4 대비 더 잦다.

6. 관제 소프트웨어(GCS) 및 ROS2 컴패니언 개발 시 시스템 뷰어(Viewer) 구축 전략

응용 개발자가 독자적인 지상 통합 관제 애플리케이션이나 ROS2(uXRCE-DDS) 기반 중앙 관제 서버를 구축할 때, 멱등성을 고려하지 않고 상태 동기화 모델을 짜게 되면 화면 속 위젯이 미친 듯이 깜빡이거나 토글되는 UI 글리치(Glitch) 현상을 겪기 쉽다.

반드시 PX4로부터 실시간 방출되는 주기적 상태 토픽(vehicle_status 혹은 HEARTBEAT)을 Observer 패턴으로 구독하여 UI 상태를 반영해야 하며, 명령 재시도 루프(Retry loop)가 작동 중일지라도 GCS 내부의 자체적인 “가상 상태“를 임의로 확정 짓지 않고 기체의 실제 수신 상태에 동기화되는 “최종적 일관성(Eventual Consistency)” 모델을 따라야 안정적인 무인기 다중 다이얼로그 시스템을 완성할 수 있다.
