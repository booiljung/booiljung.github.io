42.4.2 위성항법장치(GNSS) 신호 무결성 및 정밀도 저하율(DOP) 평가

무인항공기 시스템(UAS)의 자율 비행에 있어 위성항법장치(GNSS)의 역할은 절대적이다. 하지만 도심지, 수목 지대 등 다양한 환경 요인으로 인해 GNSS 신호는 다중 경로(Multipath) 오차, 신호 단절, 전리층 지연 등의 문제를 겪을 수 있다. PX4-Autopilot은 이러한 신호 왜곡을 사전에 감지하고 차단하기 위해 EKF2(Extended Kalman Filter 2) 아키텍처 내부에 엄격한 신호 무결성 검증 및 정밀도 저하율(DOP, Dilution of Precision) 평가 로직을 내장하고 있다. 이 장에서는 PX4-Autopilot 펌웨어(버전 1.14 기준) 및 QGroundControl(버전 4.3 기준)을 바탕으로 GNSS 평가 프로세스의 구조와 알고리즘을 분석한다.

1. 정밀도 저하율(DOP)의 개념과 기하학적 의미

정밀도 저하율(DOP)은 위성들의 기하학적 배치가 위치 결정 정밀도에 미치는 영향을 수치화한 지표이다. 하늘에 떠 있는 위성들이 좁은 구역에 밀집해 있을 경우 측정 오차가 증폭배수로 작용하여 DOP 수치가 커지고, 넓게 퍼져 있을수록 오차가 상쇄되어 DOP 수치가 낮아진다.

PX4-Autopilot에서 주로 확인하는 DOP 지표는 다음과 같다.

  • HDOP (Horizontal DOP, 수평 정밀도 저하율): 2차원 평면(위도, 경도) 상의 정밀도를 나타낸다. 보통 2.5 이하일 때 양호한 것으로 평가한다.
  • VDOP (Vertical DOP, 수직 정밀도 저하율): 고도 측정의 정밀도를 나타낸다.
  • PDOP (Position DOP, 위치 정밀도 저하율): 3차원 공간상의 종합적인 정밀도를 나타낸다.

2. PX4-Autopilot의 GNSS 무결성 검증 통과 조건

PX4-Autopilot의 EKF2 엔진은 GNSS 데이터를 추정기에 융합(Fusion)하기 전에 다양한 무결성 지표를 평가한다. 이 통과 조건(GNSS Check)이 모두 충족되어야만 비행 제어기(Flight Controller)는 GPS가 유효하다고 판단하며 위치 제어 기반 비행 모드(Mission, Hold, Return 등)로 진입할 수 있다.

관련 파라미터는 EKF2_GPS_CHECK 비트마스크(Bitmask)에 의해 설정되며, 주요 검사 항목은 다음과 같다.

  1. 위성 개수 (Minimum Satellites): COM_RC_IN_MODE 및 GPS 관련 파라미터로 설정되는 최소 위성 수(보통 6개 이상).
  2. HDOP 상한치: EKF2_GPS_CHECK에서 HDOP 검사가 활성화된 경우, 지정된 임계값(예: 2.5) 이하인지 확인한다.
  3. 위도/경도 정확도 (hacc): 수평 위치 오차 추정치가 임계치(기본 5.0m) 이내인지 검사한다.
  4. 고도 정확도 (vacc): 수직 위치 오차 추정치가 임계치(기본 8.0m) 이내인지 검사한다.
  5. 속도 정확도 (s_var/sacc): 3차원 속도 오차 추정치가 지정된 신뢰 구간을 만족하는지 파악한다.
  6. 위치 변동(Drift) 검사: 정지 상태일 때 GPS 위치가 튀는 현상(Drift)의 크기가 임계치보다 작은지 확인한다.

3. 관련 소스 코드 및 알고리즘 분석

PX4-Autopilot의 ECL(Estimation and Control Library) 모듈 내에서 GNSS 검증을 수행하는 핵심 코드는 src/modules/ekf2/EKF/gps_checks.cpp에 구현되어 있다. (Firmware 1.14 기준)

추정기는 매 주기마다 collect_gps() 함수를 호출하여 uORB를 통해 vehicle_gps_position 메시지를 획득한 후 runGpsChecks()를 수행한다. 이 함수 내부의 핵심 검증 로직은 다음과 같이 요약된다.

// 수평 위치 오차 한계 검사
if (_gps_status.eps > _params.req_hacc) {
    _gps_check_fail_status.flags.hacc = true;
}

// 수직 위치 오차 한계 검사
if (_gps_status.epv > _params.req_vacc) {
    _gps_check_fail_status.flags.vacc = true;
}

// HDOP 한계 검사
if (_gps_status.hdop > _params.req_hdop) {
    _gps_check_fail_status.flags.hdop = true;
}

// 속도 오차 한계 검사 (sacc)
if (_gps_status.sacc > _params.req_sacc) {
    _gps_check_fail_status.flags.sacc = true;
}

위의 조건 중 하나라도 실패로 판별될 경우 _gps_check_fail_status의 비트맵 플래그가 세팅되며, 이는 상태 머신(State Machine)에 플래그 정보로 갱신되어 QGroundControl을 통해 사용자에게 “GPS Check Failed“라는 알림으로 전송된다.

4. Ardupilot과의 무결성 검증 아키텍처 비교

Ardupilot은 PX4-Autopilot과 유사한 EKF 구조를 가지고 있으나, 통과 조건과 내부 파라미터 명명 등에서 차이를 드러낸다.

  • 파라미터 체계: PX4-Autopilot은 EKF2_GPS_CHECK 단일 비트마스크 파라미터로 검증 항목을 토글(Toggle)하는 반면, Ardupilot은 EK3_GPS_CHECK 파라미터 외에도 각 오차들의 한계치를 개별 파라미터(EK3_GLITCH_RAD, GPS_HDOP_GOOD 등)로 명확하게 분리하여 관리한다.
  • 데이터 구조화 결합: Ardupilot은 여러 GNSS 수신기가 장착된 듀얼 GPS 환경에서 두 수신기간 BLEND(블렌딩) 알고리즘을 보다 전역적으로 수행하지만, PX4는 ekf2 안에서의 EKF 혁신(Innovation) 분산 상태를 기준으로 메인 GPS를 동적으로 스위칭하는 방식을 선호한다.

5. QGroundControl에서의 모니터링 및 관제

QGroundControl 사용자는 관제 화면 상단 상태 표시줄의 위성 아이콘을 통해 실시간 GNSS 상태를 모니터링할 수 있다. 색상을 통한 직관적인 표시 방식을 취한다.

  • 적색(Red): 위성 수가 부족하거나 DOP가 지나치게 높아(통상 2.5 이상) 무결성 검사를 통과하지 못함. Autonomous 비행(Return, Mission)을 거부한다.
  • 황색(Yellow): 무결성 통과 조건을 간신히 만족하는 상태로 비행 환경이 좋지 못함을 암시한다.
  • 녹색(Green): DOP가 우수하고 위치 오차 반경이 충분히 좁아 EKF2 융합이 원활한 상태이다. MAVLink HIGHRES_IMU 및 GPS_RAW_INT 메시지가 지속적으로 GCS 측으로 전송되며 상태 머신은 락이 안정적임을 판단한다.

개발자는 QGC의 소스 코드에서 Vehicle.cc 인스턴스가 수신하는 GPS_RAW_INT 구조체의 eph(수평 불확실도), epv(수직 불확실도), 그리고 satellites_visible 값을 통해 커스텀 HUD를 구성하거나 특정 조건 이탈 시 경보 알고리즘을 추가할 수 있다.

이와 같이 PX4-Autopilot은 GNSS 기반의 정밀 위치 제어를 보장하기 위하여 위성 신호에 대해 매우 엄밀하고 다양한 검사를 수행한다. 공분산에 치중되는 칼만 필터의 근본적 취약성을 하드웨어 센서에서 올라오는 DOP 지표 및 오차 정보들을 선제적으로 방어함으로써, 자율 비행 시스템의 신뢰성과 항행 안정성을 고도화하고 있다.
