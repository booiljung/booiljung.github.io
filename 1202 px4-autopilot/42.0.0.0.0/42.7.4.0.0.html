42.7.4. 시동 명령 응답 프로토콜: NACK(Negative Acknowledge) 처리 및 피드백 디버깅

1. 개요

무인기 시스템 관제에서 시동(Arming) 명령은 기체의 프로펠러 등 액추에이터 구동을 물리적으로 허가하는 가장 민감하고 중요한 절차이다. 지상 관제 시스템(QGroundControl)이나 컴패니언 컴퓨터가 시동을 요청하면, PX4-Autopilot은 단순히 수락 여부만을 결정하는 것이 아니라 내부적인 비행 전 점검(Pre-flight Check) 로직을 거친다. 조건이 충족되지 못한 경우 시동 명령은 거부되며, 이를 명확한 사유와 함께 관제 시스템으로 피드백하는 프로토콜이 필수적이다.

본 절에서는 MAVLink 프로토콜 상의 시동 명령 전달 체계와, PX4 내부에서 부정 응답(NACK, Negative Acknowledge)이 발생했을 때 이를 어떻게 처리하고 디버깅 가능한 형태로 GCS에 피드백하는지 소스 코드와 아키텍처 관점에서 심층 분석한다. (최신 PX4 Firmware v1.14 이상, QGroundControl v4.3 이상 산출물 기준)

2. MAVLink 기반 시동 명령과 COMMAND_ACK 프로토콜 구조

GCS는 기체 시동을 위해 MAVLink의 COMMAND_LONG 혹은 COMMAND_INT 메시지를 사용하여 MAV_CMD_COMPONENT_ARM_DISARM (명령 ID: 400) 명령을 전송한다. 이 명령에 대한 처리 결과는 반드시 COMMAND_ACK (메시지 ID: 77) 메시지를 통해 피드백되어야 한다.

2.1 COMMAND_ACK의 Result 필드

COMMAND_ACK 메시지는 처리 결과를 result 필드(Enum 형식 MAV_RESULT)로 반환한다.

  • MAV_RESULT_ACCEPTED (0): 명령 수락 (시동 성공)
  • MAV_RESULT_TEMPORARILY_REJECTED (1): 일시적 거부 (예: 아직 센서 캘리브레이션 중이거나 시스템 초기화 중)
  • MAV_RESULT_DENIED (2): 거부 (시동 전제 조건 위반, 가장 잦은 NACK 유형)
  • MAV_RESULT_UNSUPPORTED (3): 미지원 명령
  • MAV_RESULT_FAILED (4): 명령 실행 중 실패

시동 거부 시 PX4는 주로 MAV_RESULT_DENIED 형태의 NACK를 반환한다.

2.2 시동 명령 요청 및 NACK 처리 시퀀스 (Mermaid)

sequenceDiagram
    participant GCS as QGroundControl
    participant PX4 as PX4-Autopilot (통신/제어)
    participant Cmdr as Commander 모듈
    
    GCS->>PX4: COMMAND_LONG (MAV_CMD_COMPONENT_ARM_DISARM, param1=1)
    PX4->>Cmdr: uORB: vehicle_command 발행
    Cmdr->>Cmdr: Arming 조건 및 Pre-flight Check 검증
    alt 검증 실패 (예: GPS Fix 없음, 센서 캘리브레이션 미비)
        Cmdr-->>PX4: uORB: vehicle_command_ack (Result=DENIED)
        Note right of Cmdr: 이벤트 시스템(Event System)을 통한 구체적 사유 생성
        PX4->>GCS: COMMAND_ACK (Result=MAV_RESULT_DENIED)
        PX4->>GCS: EVENT / STATUSTEXT (Severity=ERROR, "Preflight Fail: No GPS Fix")
        GCS->>GCS: NACK 코드 파싱 및 텍스트 알림 UI 팝업
    else 검증 성공
        Cmdr-->>PX4: uORB: vehicle_command_ack (Result=ACCEPTED)
        PX4->>GCS: COMMAND_ACK (Result=MAV_RESULT_ACCEPTED)
        PX4->>GCS: STATUSTEXT (Severity=INFO, "Armed")
    end

3. PX4 Commander 모듈의 시동 거부(NACK) 처리 아키텍처

PX4의 심장부인 Commander 모듈(src/modules/commander/Commander.cpp) 내부에는 시동을 통제하는 arm_auth 하위 시스템과 거대한 상태 머신(State Machine)이 존재한다.

  1. 명령 수신 및 검사: GCS가 전송한 명령은 Commander::handle_command() 함수 내부의 case vehicle_command_s::VEHICLE_CMD_COMPONENT_ARM_DISARM: 분기에서 가로채어진다.
  2. 사전 조건 검증 (arm() 함수 호출): 내부에 구현된 HealthAndArmingChecks 서브클래스에서 각 센서, 배터리 상태, EKF(확장 칼만 필터) 상태를 순회(Iteraton)하며 점검한다.
  3. NACK 결정 및 이벤트 발생: 어느 한 항목이라도 미달할 경우 arm() 호출은 실패로 돌아가며, vehicle_command_ack uORB 토픽의 result 필드에 거절 원인을 맵핑하여 퍼블리싱한다.

3.1 구체적 사유 전달: STATUSTEXT와 Event 패러다임 차이

과거 PX4 버전에서는 NACK를 보냄과 동시에 코드 내에서 직접 mavlink_log_critical() 매크로를 호줄하여 문자열(STATUSTEXT)을 송출하였다.
최신 PX4 버전에서는 유지보수성과 다국어 지원, 그리고 구조화를 위해 **이벤트 체계(Event Interface)**를 적극 도입했다. 시동이 실패하면 events::send<...>(events::ID("commander_arm_rejected"), ...); 형태의 이벤트가 트리거되고, 이는 MAVLink 이벤트 프로토콜과 호환되어 GCS로 포워딩된다. 이를 통해 GCS는 단순히 “시동 거부” 텍스트만 띄우는 것이 아니라, “나침반 센서 캘리브레이션 필요“라는 구체적인 해결책 조언을 UI 상에 반영할 수 있게 되었다.

4. Ardupilot과 PX4의 NACK 처리 및 피드백 구조 비교

  • PX4-Autopilot: NACK 발생 요인을 이벤트 모듈을 통해 구조화된 ID 맵핑 체계로 변환하여 텍스트 및 사운드(Tones)와 연동한다. GCS에 구체적인 디버깅 로그를 명확하게 전달하며, MAVSDK 및 uXRCE-DDS(ROS2) 환경에서도 해당 NACK 콜백과 사유를 객체 지향적으로 캐치업(Catch-up)할 수 있는 API 프레임워크를 제공한다.
  • Ardupilot: 전통적으로 문자열 로깅에 상당히 집착하는 구조다. AP_Arming 모듈에서 PreArm: [문제부위] 형식의 평문 STATUSTEXT 메시지를 COMMAND_ACK와ほぼ 동시에 대량으로 살포한다. 구조화된 이벤트 ID보다는 직관적인 텍스트 로깅 중심이라 미션 플래너(Mission Planner) 콘솔 창에서 텍스트 기반의 피드백 디버깅이 주를 이룬다.

5. QGroundControl 관점의 피드백 디버깅 메커니즘

QGC에서 NACK 수신 및 처리는 사용자 경험(UX) 및 디버깅 관점에서 매우 핵심적인 로직으로 설계되어 있다.

  1. Ack 수집: src/Vehicle/Vehicle.cc 인스턴스에서 커맨드 트랜잭션을 관리하는 매니저가 COMMAND_ACK를 대기한다.
  2. 결과 파싱: MAV_RESULT_DENIED와 같은 NACK를 수신하면 트랜잭션 실패 이벤트 루프가 돌아가고, 즉시 경고음을 발생시킨다.
  3. 피드백 결합: QGC는 NACK 직후 수백 밀리초 단위로 수신되는 STATUSTEXT 혹은 EVENT 프로토콜 메시지들 중 ‘Preflight’ 필터링을 거친 문자열을 버퍼링하여 화상 중앙의 메시지 배너 박스(Message Banner) 상단 붉은색 알림 UI에 출력한다.
  4. 로깅: 이러한 NACK 사유 이력은 파일 백엔드 시스템 로그에 [NACK] 태그로 저장되어 사후 비행 기록 리뷰 파일(.tlog) 디버깅에 활용된다.

6. ROS2 자율 비행 응용 환경에서의 NACK 디버깅 기법

자율적인 컴패니언 컴퓨터가 ROS2 브릿지를 통해 직접 시동을 명령할 때는 GCS의 화려한 UI 백업이 없다. 개발자는 ROS2 노드(Node) 안에서 단순히 시동 서비스를 요청하고 True/False 결과만 기다려서는 안 된다. NACK 처리 시 디버깅을 원활히 하기 위해서는 다음의 아키텍처를 구현해야 한다.

  1. 동기식 서비스 한계 극복: ROS2 vehicle_command를 발행한 뒤 비동기 상태로 vehicle_command_ack 토픽을 명시적으로 서브스크라이빙하여 NACK 코드를 캡처한다.
  2. 사유 트래킹 노드: event 혹은 mavlink_log uORB 토픽을 브릿지를 통해 같이 구독하여, result == 2 (DENIED) 발생 시점 즈음의 텍스트 로그를 로컬 리눅스의 표준 에러 로그(ROS_ERROR)로 바인딩하여 백트래킹(Backtracking) 디버깅 환경을 최적화해야 실무에서 센서 오류 파악이 빠르다.
