42.7.4 시동 명령 응답 프로토콜: NACK(Negative Acknowledge) 처리 및 피드백 디버깅

지상 관제 시스템(GCS; Ground Control Station)에서 무인항공기(UAV)로 하달되는 제어 명령 중 가장 중대한 권한 이양 행위는 바로 시스템의 프로펠러나 액추에이터에 동력을 인가하는 시동(Arming) 명령이다. MAVLink 프로토콜 상에서 이는 MAV_CMD_COMPONENT_ARM_DISARM 명령으로 송출된다. 그러나 비행 제어기(Flight Controller)는 센서 이상, 배터리 부족, 안전 스위치 체결 상태 등을 이유로 이 명령을 수락할 수 없는 경우가 빈번하게 발생한다. 이 때 기체가 단순한 무응답으로 일관하지 않고 명시적인 반려 의사와 그 사유를 반환하는 체계를 NACK(Negative Acknowledge) 응답 프로토콜이라 칭한다.

본 절에서는 최신 PX4 펌웨어(v1.14 기준) 및 QGroundControl(v4.3 기준) 환경을 중심으로, 시동 명령 거절 시 발생하는 펌웨어 내부의 NACK 생성 아키텍처와 이를 지상 관제 시스템 혹은 ROS2 연동 환경에서 어떻게 디버깅하고 렌더링하는지 심층적으로 다룬다.

1. PX4-Autopilot의 NACK 생성 아키텍처 및 철학

GCS가 COMMAND_LONG 폼을 빌려 암(Arm) 명령을 지시하면, 기체 내 mavlink_receiver를 거쳐 vehicle_command uORB 토픽 형태로 PX4 커맨더(Commander.cpp) 스레드에 도착한다. 커맨더 스레드는 현재 시스템 상태 머신(State Machine) 상에서 시동이 가능한지 검문사격과 같은 ‘Pre-flight check’ 혹은 강제 권한(Override) 제약 조건을 모두 스캔한다.

만일 단 하나의 조건이라도 위배되어 시동을 인가할 수 없는 경우, 커맨더는 즉시 vehicle_command_ack 토픽을 다음과 같은 결과 코드(Result Code)를 실어 발행한다.

  • MAV_RESULT_DENIED (4): 논리적으로 해당 상태에서 절대 실행 불가능한 명령(예: 센서 캘리브레이션 미수행)
  • MAV_RESULT_TEMPORARILY_REJECTED (3): 시스템이 아직 초기화 중이거나 잠시 후 정상 수락될 가능성이 있는 경우

하지만 이러한 1바이트 크기의 상수형 결과 코드만으로는 관제석에 앉은 조종사나 에이전트가 “동력 인가가 왜 거절되었는지” 원인을 특정할 수 없다. 따라서 PX4 펌웨어 설계 철학은 NACK 전송 시 반드시 인간 단위에서 식별 가능한 원인 규명 텍스트 이벤트를 동반하도록 강제하고 있다.

2. NACK 피드백 병합을 통한 GCS의 팝업 디버깅 파이프라인

QGroundControl (v4.3) 수신단은 시동 거절 사태를 매끄럽게 사용자 인터페이스(UI)로 파싱해 내기 위해, 다음과 같은 병렬 트랜잭션을 수집하는 아키텍처를 지니고 있다.

sequenceDiagram
    participant UI as QGC 사용자 UI
    participant MAV_MGR as QGC MAVLink Manager
    participant PX4 as PX4 Commander Subsystem

    UI->>MAV_MGR: 슬라이더 조작 (Arm 명령 실행 지시)
    MAV_MGR->>PX4: COMMAND_LONG (MAV_CMD_..._ARM_DISARM)
    activate PX4
    PX4-->>PX4: 상태 점검: GPS 3D Fix 상실 확인
    PX4->>MAV_MGR: COMMAND_ACK (Result: 4 = DENIED)
    PX4->>MAV_MGR: STATUSTEXT ("PREFLIGHT FAIL: NO 3D FIX")
    PX4->>MAV_MGR: EVENT ("gps_not_fixed_error")
    deactivate PX4
    MAV_MGR-->>MAV_MGR: ACK 결과와 수신된 텍스트 이벤트 병합 타이머 구동
    MAV_MGR->>UI: 시동 실패 알림 및 "NO 3D FIX" 에러 배너 표출

위의 위상 연계도에서 알 수 있듯, GCS의 프로토콜 해석 매니저는 COMMAND_ACK에서 거절(DENIED) 값을 감지하게 되면, 즉각적으로 UI에 에러를 표출하기보다는 짧은 타이머 한도 내에 병렬로 도착하는 STATUSTEXT 혹은 MAVLink v2.0 기반의 EVENT 메시지를 낚아챈 뒤 이들을 논리적으로 결합(Merge)하여 상세한 디버깅 피드백 창을 완성한다. 이를 통해 엔지니어는 굳이 ULog 파일을 열어보지 않고도 기체의 문제를 신속하게 규명할 수 있다.

3. ROS2 (uXRCE-DDS) 기반 오프보드 연동 시의 예외 처리 로직

근래 드론 응용 분야에서 부각되고 있는 로봇 운영 체제(ROS2) 환경 하에서는 사람이 콘솔을 보지 않고 자율 에이전트가 스크립트 기반으로 비행을 주도한다.

C++ 혹은 Python으로 작성된 오프보드(Offboard) 노드에서 VehicleCommand 토픽 구조체를 채워 시동을 지시했을 때, 에이전트 노드는 반드시 VehicleCommandAck 서브스크라이버(Subscriber) 콜백 함수를 구성하여 result 필드 값을 검증해야 한다. 코드가 NACK 처리에 대한 대비 없이 강압적으로 다음 스테이지(임무 이동)를 트리거하게 되면 시스템 루프가 데드록(Deadlock)에 빠질 수 있기 때문이다.
ROS2 설계 시에는 NACK 결과값을 획득했을 때 지수 백오프(Exponential Backoff) 알고리즘을 사용한 재시도 로직을 구성하거나, 안전 기동 모드로 강제 복귀하도록 Failsafe 트리를 짜는 프로그래밍 가이드라인이 강조된다.

4. Ardupilot의 NACK 철학 대비 아키텍처 비교

Ardupilot 환경 또한 MAV_RESULT_DENIED를 통해 시동 거절을 표현하지만, PX4와 비교 시 다음과 같은 구조적 차이가 존재한다.
Ardupilot은 내부 모듈별 비트마스크(Bitmask) 검증 로직 결과 자체를 지상 관제 시스템(예: Mission Planner)이 질의하여 가져갈 수 있도록 확장된 파라미터 세트와 ARMING_CHECK 하위 식별 코드를 심층적으로 운용한다. 즉 관제 시스템이 기체의 어느 파트 모듈에서 암(Arm) 제어를 블로킹하고 있는지 시스템 변수 레벨에서 추적하기가 상대적으로 직관적이다.

반면 PX4-Autopilot 생태계는 MAVLink EVENT 채널을 이용한 상태 코드 기반의 이벤트 로깅(Logging) 철학에 큰 무게를 둔다. PX4 v1.14 이후부터는 문자열을 통째로 보내 시스템 대역폭을 낭비하는 대신, 고유 식별자가 담긴 이벤트 패킷을 반환하여 GCS 단에서 번역(Translation) 로직과 매핑하는 방식(메타데이터 기반 디버깅)으로 진화하여 NACK 사유 추적의 효율을 높이고 텔레메트리 혼잡도를 극적으로 낮추는 방향성을 고수하고 있다.

결과적으로 PX4 소스 코드를 최적화하여 독자적인 GCS 환경을 구축하려는 엔지니어 조직에서는, 단순한 COMMAND_ACK 트래킹에 머물 것이 아니라 PX4의 이벤트(events 및 statustext) 파이프라인을 매끄럽게 관제 프론트엔드로 연동 파싱(Parsing)해 내는 디버깅 아키텍처 역량을 내재화해야 한다.
