42.6.3. 오프보드(Offboard) 제어기와의 제어 명령 스트림 연속성 검증

복잡한 임무 수행이나 정밀한 자율 비행(Autonomous Flight)을 위해 무인항공기 시스템(UAS)은 종종 비행 제어기(Flight Controller, FC) 외부의 고성능 연산 장치인 커패니언 컴퓨터(Companion Computer)를 활용한다. 이를 오프보드(Offboard) 제어라 일컬으며, 컴퓨터 비전(Computer Vision)이나 장애물 회피(Obstacle Avoidance)와 같은 고차원 알고리즘의 결과물이 실시간 제어 명령 스트림(Control Command Stream) 형태로 PX4-Autopilot에 전달된다.
비행 전 점검(Pre-flight Check) 및 비행 중에 이 오프보드 제어 명령 스트림이 끊기거나 지연되는 현상은 기체의 통제 상실(Loss of Control)로 이어질 수 있는 매우 위험한 상황이다. 따라서 제어 명령 스트림의 연속성을 엄격하게 검증하는 아키텍처는 필수 불가결하다. 본 절에서는 PX4의 Offboard 제어 모드에서 명령 스트림의 연속성을 담보하는 메커니즘과 연동 기술을 심층 분석한다.

1. 오프보드(Offboard) 제어 스트림의 개념 및 요구사항

오프보드 모드는 PX4-Autopilot이 외부(Offboard)에서 생성된 위치(Position), 속도(Velocity), 가속도(Acceleration), 자세(Attitude) 또는 추력(Thrust) 설정값(Setpoint)을 수용하여 기체를 제어하는 비행 모드이다.
이러한 설정값들은 MAVLink의 SET_POSITION_TARGET_LOCAL_NED와 같은 메시지를 이용하거나, 최근 부상하고 있는 ROS2 기반의 uXRCE-DDS 미들웨어를 통하여 TrajectorySetpoint 혹은 VehicleCommand uORB 토픽 형태로 PX4에 인가된다.

제어 주기 이론에 따르면, FC 내부의 위치/자세 제어 루프(Position/Attitude Control Loop)는 지속적인 외부 레퍼런스를 요구한다. 스트림 연속성 측면에서의 핵심 요구사항은 다음과 같다.

  • 최소 송출 주파수(Minimum Send Rate): 오프보드 모드 진입 전 그리고 유지 중에, 제어기(Companion Computer)는 설정값을 최소한 2Hz (이상적으로는 10Hz~50Hz) 이상의 주기로 송출해야 한다.
  • 데이터 흐름의 무결성(Stream Integrity): 스트림 패킷이 지터(Jitter) 한계를 넘지 않고 일정한 타임스탬프 간격으로 도착해야 한다.

2. PX4-Autopilot의 Offboard 스트림 연속성 검증 아키텍처

PX4에서는 Offboard 제어 명령 스트림의 연속성을 Commander 모듈 내부의 OffboardControl 로직과 수신된 offboard_control_mode uORB 토픽 타임아웃 검증을 통해 처리한다.

2.1 진입(Entry) 전제 조건 로직

비행 전 점검 단계 혹은 비행 중 Offboard 모드로 전환 시, PX4는 제어 스트림이 사전에 안정적으로 확립되었는지를 최우선으로 검증한다. 이를 ’스트림 예열(Stream Pre-warming)’이라고도 한다.

stateDiagram-v2
    [*] --> Idle: 초기 상태
    Idle --> Streaming_Valid: 커패니언 컴퓨터가 Setpoint 스트림 전송 개시 ( > 2Hz )
    Streaming_Valid --> Offboard_Mode: GCS 또는 API를 통한 Offboard 모드 전환 명령 (ACK)
    Offboard_Mode --> Streaming_Valid: 스트림 타이머 갱신
    Offboard_Mode --> Stream_Lost (Failsafe): 타임아웃 발생 (COM_OF_LOSS_T 초과)
    Stream_Lost (Failsafe) --> Hold_Mode: 연속성 상실로 인한 비상 정지 및 체공
  • Setpoints 수신 확인: PX4는 모드 전환 명령을 수신한 시점을 기준으로, 최근 N초(보통 1초 이내) 안에 유효한 오프보드 설정값이 수신되었는지를 검열한다. 수신 이력이 없다면, 모드 전환은 거절(Mode Change Denied)된다.

2.2 연속성 상실(Signal Loss) 감지 및 안전 장치(Failsafe) 파라미터

Offboard 모드 동작 중에 데이터 스트림이 단절되는 상황에 대비하여, PX4 파라미터 시스템은 COM_OF_LOSS_T (Offboard Loss Timeout) 매개변수를 제어한다.

3. ROS2 (uXRCE-DDS) 연동과 QoS를 통한 제어망 동기화

최근 자율 시스템에서는 고전적인 MAVLink 시리얼 통신을 넘어 PX4-ROS2 아키텍처가 널리 채택된다. 커패니언 컴퓨터상의 ROS2 노드(Node)는 uXRCE-DDS 에이전트(Agent)와 클라이언트(Client) 구조를 매개로 uORB 토픽을 직접 퍼블리싱(Publishing)한다.

DDS(Data Distribution Service) 계층의 가장 큰 이점은 서비스 품질(Quality of Service, QoS) 설정에 있다.

  • Deadline QoS: 발행자(Publisher)가 특정한 마감 시간(예: 100ms)마다 메시지를 퍼블리싱하겠다고 선언하고, 수신자(Subscriber, PX4 측 클라이언트)는 이 주기가 어긋날 경우 ’마감 시간 초과 이벤트(Offered Deadline Missed)’를 즉각적으로 인식한다.
  • 스트림 연속성의 DDS 레벨 검증: 기존 MAVLink에서는 커맨더가 자체적으로 타임아웃을 연산했다면, ROS2 연동 하에서는 DDS의 Liveliness 및 Deadline 아키텍처가 결합되어 더욱 즉각적이고 확정적인 스트림 무결성 검증이 가능해진다.

4. Ardupilot 대비 Offboard 제어 사상 비교

Ardupilot은 GUIDED 모드를 통하여 오프보드 제어를 수행한다. Ardupilot의 GUIDED 모드 아키텍처는 목표 지점(Target Waypoint)이나 속도 벡터를 MAVLink로 수신하면, 내부적으로 ModeGuided 클래스가 제어를 전담하는 방식을 취한다.

  • 상태의 유지: Ardupilot은 한 번 GCS나 커패니언 컴퓨터로부터 설정값이 들어오면, 다음 명령이 들어오지 않더라도 (타임아웃 등에 대한 엄격한 내부 연속성 제한 없이) 해당 상태를 유지하려는 경향이 있다 (예: 특정 좌표로 가라는 명령 시). 속도 제어(Velocity Control)의 경우에만 타임아웃(기본 3초)이 하드코딩되어 있다.
  • PX4의 확정적(Deterministic) 연속성 보장: 반면 PX4는 Offboard 체계를 훨씬 좁고 엄격하게 정의한다. PX4는 상태(State) 유지가 아닌 지속적인 흐름(Continuous Stream)의 갱신을 원칙으로 삼는다. 어떠한 형태의 설정값(Pos, Vel, Attitude)이든 COM_OF_LOSS_T 시간이 초과하면 예외 없이 Failsafe를 발동(Trigger)시킴으로써, 의도치 않은 비행기 돌진(Flyaway) 사고를 원천 차단하는 데 중점을 둔다.

5. QGroundControl과 MAVSDK를 활용한 관제 기술 활용

오프보드 애플리케이션 개발 시(예: Python 기반 MAVSDK 활용), 개발자는 루프(Loop)를 차단하는 동작(Blocking call, 예: time.sleep())을 최대한 회피하고 비동기(Asynchronous) 설계 또는 독립된 발행 스레드(Publishing Thread)를 유지해야 한다.

  • MAVSDK 관제 유의점: MAVSDK의 .offboard.set_velocity_ned()와 같은 명령어를 사용한다면, 제어 로직이 수행되는 동안 백그라운드 태스크(Background Task) 체계가 설정값을 주기적으로 갱신하도록 구성해야 개 구성해야 한다. MAVSDK 파이썬 인터페이스는 자체적으로 await action.arm()과 await offboard.start() 호출 전, 연속성 검증을 회피하지 못하도록 2~3번 이상의 설정값 선행 전송을 강제한다.
  • QGC 모니터링: QGroundControl UI 내부에서는 오프보드 스트림 단절 발생 시 ‘Offboard mode rejected’ 또는 ‘Failsafe: Offboard loss’ 플래시 메시지를 표출하며 즉시 기체 상태를 경고 인디케이터로 전환시킨다.

자율 군집 제어(Swarm Control) 애플리케이션을 구상한다면 이러한 스트림 연속성의 구조적 한계점과 PX4의 Fail-fast(오류 조기 발생) 설계 철학을 깊이 이해하고 코딩하는 것이 무엇보다 중요하다.
