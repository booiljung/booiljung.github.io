42.6.3. 오프보드(Offboard) 제어기와의 제어 명령 스트림 연속성 검증

무인항공기 시스템(UAS)에서 외부 컴퓨터(Companion Computer)나 지상 관제 시스템(GCS)을 통해 기체를 직접 제어하는 방식을 오프보드 제어(Offboard Control)라 한다. 고도의 자율 비행, 컴퓨터 비전 기반의 회피 기동, 복잡한 경로 계획(Path Planning) 등이 요구되는 현대의 임무 환경에서는 오프보드 제어가 필수적이다. 그러나 외부 시스템에 제어 권한을 위임하는 만큼, 제어 명령 스트림(Control Command Stream)의 연속성이 확보되지 않을 경우 기체는 치명적인 추락이나 이탈 등 심각한 위험에 노출될 수 있다. 따라서 비행 전 점검(Pre-flight Check) 단계 및 비행 과정 전반에 걸쳐 제어 명령이 끊김 없이 수신되는지(스트림 연속성 검증)를 철저히 확인해야 한다. 본 절에서는 PX4-Autopilot의 오프보드 제어 아키텍처 내에서 발생하는 명령 스트림 연속성 검증 알고리즘과 안전 매커니즘을 심층적으로 분석한다. (기준 버전: PX4-Autopilot v1.14 이상, QGroundControl v4.3 이상)

1. 오프보드 제어 명령 스트림 연속성의 개념과 중요성

오프보드 제어 모드에서 PX4 비행 제어기(Flight Controller)는 조종기의 입력이나 내부 임무(Mission) 로직 대신 외부 반려 컴퓨터(Companion Computer, 예: Raspberry Pi, NVIDIA Jetson 등)나 GCS로부터 유입되는 실시간 시점 이동 명령(Setpoints)을 추종한다. 이러한 명령은 위치(Position), 속도(Velocity), 가속도(Acceleration), 자세(Attitude) 혹은 추력(Thrust) 형태로 전달된다.

제어 명령 스트림(Control Command Stream)은 이진 데이터나 MAVLink 패킷 형태로 일정한 주파수(예: 10Hz 이상)를 가지고 비행 제어기에 지속적으로 전파되어야 한다. 만약 네트워크 혼잡, 반려 컴퓨터의 프로세스 다운, 혹은 통신 링크 단절로 인해 명령 스트림이 끊기면, 비행 제어기는 마지막 명령을 무기한 추종하게 되어 궤도 이탈이나 기체 파손을 초래할 수 있다. 이러한 사태를 미연에 방지하기 위해 스트림 연속성 검증(Stream Continuity Verification) 메커니즘이 동작한다.

1.1 스트림 연속성 검증의 수학적 조건

오프보드 모드로의 진입 및 유지를 위해, 외부 제어기는 최소 T_{min}의 주기로 명시적인 제어 명령을 비행 제어기로 송신해야 한다. 매 패킷 수신 시간 t_k에 대하여 연속성 조건은 다음과 같이 정의된다.
\Delta t = t_k - t_{k-1} \le T_{timeout}
여기서 T_{timeout}은 임계 허용 지연 시간(Timeout Threshold)이며, 이 조건을 만족하지 못하는 경우 스트림이 단절된 것으로 간주하고 즉시 안전 장치(Failsafe) 메커니즘을 가동한다.

2. PX4-Autopilot의 오프보드 제어 스트림 검증 아키텍처

PX4-Autopilot 내부에서는 commander 모듈과 FlightTaskOffboard 모듈이 긴밀하게 협력하여 오프보드 명령 스트림을 검증한다. MAVLink 또는 ROS 2(uXRCE-DDS) 인터페이스를 통해 제어 입력이 들어오면, uORB 메시지(예: trajectory_setpoint, vehicle_command) 형태로 변환되어 시스템 내부로 배포된다.

오프보드 검증 상태 머신 모델

다음은 PX4에서 오프보드 모드 진입 및 스트림 단절 시 모드 전환을 결정하는 상태 머신의 흐름이다.

stateDiagram-v2
    [*] --> Manual_Hold : 부팅 및 대기
    Manual_Hold --> Offboard_Requested : 오프보드 모드 요청
    
    state Offboard_Requested {
        [*] --> Check_Stream
        Check_Stream --> Stream_Valid : 최근 0.5초 이내 Setpoint 수신
        Check_Stream --> Stream_Invalid : Setpoint 없음
    }
    
    Offboard_Requested --> Offboard_Active : 스트림 유효 (Stream_Valid)
    Offboard_Requested --> Manual_Hold : 스트림 무효 (명령 거절)
    
    state Offboard_Active {
        [*] --> Monitor_Stream
        Monitor_Stream --> Timeout_Detected : Setpoint 수신 지연 (> 0.5초)
    }
    
    Offboard_Active --> Failsafe_Triggered : Timeout_Detected 발생
    Failsafe_Triggered --> RTL_Land_Hover : 파라미터(COM_OBL_*) 설정에 따른 대처

위의 상태 다이어그램에서 보듯, 오프보드 모드로 전환하기 위해서는 명령을 전환하기 전 최소 0.5초간 스트림이 이미 수신되고 있어야 한다. 이는 전환 직후 기체가 명령 부재로 인해 불안정한 상태에 빠지는 것을 선제적으로 차단하기 위한 통제 장치이다.

3. 제어 명령 스트림 단절 감지 및 안전 장치(Failsafe) 동작 메커니즘

스트림 연속성이 무너지고 타임아웃 조건이 발생할 때, PX4-Autopilot은 파라미터 설정에 기반하여 시스템을 즉시 안전 모드로 전환한다.

안전 감시 루프(Safety Monitoring Loop)

commander의 메인 루프는 uORB 토픽(offboard_control_mode)의 타임스탬프를 매 주기마다 확인한다.

  • 정상 수신 상태: timestamp_last_valid_msg > current_time - timeout_interval
  • 타임아웃 상태: 수신 주기가 임계값을 초과할 경우, 내부 lost_offboard 플래그가 true로 설정된다.

주요 파라미터 구성 가이드

  • COM_OF_LOSS_T: 오프보드 명령 패킷의 타임아웃 임계 시간을 설정한다. 기본값은 0.5초이다. 네트워크 지연이 심한 환경에서는 수치를 올릴 수 있으나, 안전을 위해 1.0초 미만 유지를 권장한다.
  • COM_OBL_ACT: 오프보드 스트림이 상실(Data Link Loss)되었을 때 취할 비행 모드를 결정한다.
    • 0: Return mode (기본값, 즉시 이륙 위치로 복귀)
    • 1: Land mode (현재 위치에서 즉시 착륙)
    • 2: Hold mode (현재 고도와 위치에서 일시정지 호버링)
  • COM_OBL_RC_ACT: 만약 조종기(RC) 명령 스트림이 아직 살아있는 상태에서 오프보드 명령만이 끊기면 어떠한 모드로 전환할지 규정한다. (예: Position 모드로 절체하여 조종사가 수동 개입 가능토록 지원)

ROS 2 및 uXRCE-DDS 통신 환경의 고려사항

최신 PX4 펌웨어 버전에서는 ROS 2와의 직접적인 연동을 위해 uXRCE-DDS 미들웨어 통신을 지원한다. Micro XRCE-DDS 구조에서는 DDS 토픽이 직접 uORB 메시지로 매핑된다. 이 과정에서 네트워크 QoS(Quality of Service) 설정, 특히 RELIABILITY (Best Effort vs Reliable) 설정에 유의해야 한다. 스트림 연속성 관점에서는 지연(Latency)이 짧고 주기적 갱신이 이루어지는 제어 명령 특성상 Best Effort 방식이 패킷 적체 방지에 유리하며 연속성 확보에 실질적으로 도움이 된다.

4. Ardupilot의 Guided 모드와의 아키텍처 비교

PX4-Autopilot의 오프보드(Offboard) 모드에 대응하는 Ardupilot의 기능은 가이디드(Guided) 모드이다. 두 플랫폼 모두 반려 컴퓨터의 명령을 처리하지만, 명령 스트림의 연속성 보장 철학에는 구조적인 차이가 존재한다.

  • 안전 검증 진입점 (Entry Criteria):
    • PX4-Autopilot: 오프보드 모드 상태로 전환하기 위해 “전환 요청 수신 직전 일정 시간(예: 0.5초) 이전부터 setpoint 메시지가 지속적으로 수신되고 있어야 함“이라는 엄격한 사전 전제(Pre-flight / Pre-mode-switch Check)를 강제한다.
    • Ardupilot: Guided 모드로 전환 요청 시 즉시 전환이 이루어지며, 이후 타임아웃이나 유효하지 않은 명령이 들어오면 대기(Hold) 상태를 취한다. PX4에 비해 진입 과정이 상대적으로 유연한 편이다.
  • 연속성 상실 보상 (Failsafe Behavior):
    • PX4-Autopilot: 통신 상실 후 타임아웃 도달 시, 시스템 차원에서 강력하게 COM_OBL_ACT 파라미터에 명시된 복귀/착륙 로직 트리거로 즉각 절체된다. 시스템의 무결성을 전제로 하는 학술 및 상용 오프보드 연구 아키텍처 설계에 더 맞닿아 있다.
    • Ardupilot: 타임아웃 발생 시 페일세이프(FS) 로직이 발생하긴 하지만, 단순히 현재 명령을 플러시(Flush)하고 새로운 명령이 오기를 대기하거나 속도를 0으로 수렴시키는 로직이 강하게 작용한다.

5. QGroundControl (QGC)를 통한 스트림 상태 관제 및 모니터링

개발자 및 연구자는 GCS 환경에서 스트림 연속성을 직관적으로 모니터링하기 위해 QGroundControl(QGC)의 기능을 적극 활용해야 한다.

  1. 상태 알림 피드백: QGC는 MAVLink의 HEARTBEAT 및 시스템 STATUSTEXT 메시지를 분석하여 오프보드 모드 전환 실패 사유를 출력한다. “Failsafe enabled: no offboard” 와 같은 에러 텍스트는 오프보드 스트림 단절 상태에서 관제화면 상단 영역 배너에 시각화된다.
  2. 명령 전송 빈도 분석: MAVLink Inspector(MAVLink 검사기) 도구를 사용하여 반려 컴퓨터에서 전송되는 SET_POSITION_TARGET_LOCAL_NED 등의 메시지 송신 주파수가 타임아웃 기준(COM_OF_LOSS_T)보다 높은지 실시간으로 주파수(Hz)를 확인하고 검증해야 한다.
  3. QGC 커스터마이징 (Custom App): GCS 단에서 특수 목적을 위해 스트림 지연율 차트를 그릴 필요가 있을 경우, QGC 내부 C++ 소스에서 Vehicle 객체가 수신하는 타임스탬프(QGC 밀리초 단위 시간)를 추출하여 자체적인 타임 아웃 경고 UI를 추가하는 방식이 자주 활용된다.

6. 요약

PX4-Autopilot 시스템에서 오프보드(Offboard) 제어 명령 스트림 연속성 검증은 기체의 완전성을 결정짓는 핵심 안전 요건이다. 사전에 외부 컴퓨터로부터 최소 0.5초 이상 제어 데이터가 안정적으로 공급되어야만 모드 전환이 승인되는 구조적 철학은, 오프보드 제어 과정에서 발생할 수 있는 잠재적 위험을 시스템적으로 봉쇄하는 PX4만의 정교한 아키텍처 우수성을 증명한다. 스트림 단절 상황에 대비하여 파라미터 구성을 정교화하고, ROS 2와 MAVLink 간의 QoS 미세 조율(Tuning)을 통해 시스템 신뢰성을 극대화해야 한다.
