42.7.3 MAV_CMD_RUN_PREARM_CHECKS 온디맨드(On-demand) 강제 점검 트리거 시스템

지상 관제 시스템(GCS; Ground Control Station) 혹은 외부 파트너 자율 에이전트(ROS2 Companion Computer 등)가 무인항공기(UAV)의 시동(Arming)을 요청하기 이전에, 비행 제어기(Flight Controller) 내부의 모든 하위 시스템이 안전한 상태인지 확인하는 과정은 매우 중요하다. PX4-Autopilot은 백그라운드 태스크를 통해 주기적으로 시스템의 상태를 확인하지만, MAVLink 프로토콜은 관제 측에서 명시적으로 전체 시스템 점검을 즉시 수행하고 그 결과를 피드백받을 수 있는 온디맨드(On-demand) 수단을 제공한다. 이때 활용되는 것이 바로 MAV_CMD_RUN_PREARM_CHECKS 기능이다.

본 절에서는 최신 PX4 펌웨어(v1.14 이상) 및 QGroundControl(v4.3 기준) 아키텍처를 바탕으로 MAV_CMD_RUN_PREARM_CHECKS 명령의 내부 처리 구조, uORB 파이프라인, 그리고 Ardupilot과의 아키텍처 대비를 심층적으로 분석한다.

1. 개요 및 MAVLink 프로토콜 특성

MAVLink 메시지 세트 규격에서 MAV_CMD_RUN_PREARM_CHECKS (Command ID: 401) 명령은 모터에 전력을 인가하기 전 기체의 비행 전 점검(Pre-flight Check) 루틴을 수동으로 혹은 외부 시스템의 강제 호출로 실행하도록 지시한다.
통상적으로 PX4의 커맨더(Commander) 모듈은 상태 머신(State Machine) 전환 시 점검 로직을 실행하지만, 이 외부 명령을 이용하면 기체가 굳이 이륙 준비 단계로 전환되지 않더라도 센서 캘리브레이션 상태, EKF(Extended Kalman Filter) 오차 한계, 전원 배터리 및 지오펜스(Geofence)와 같은 하드웨어/소프트웨어 제약 조건을 사전에 검증하고 관제 시스템에 상세 리포트(STATUSTEXT)를 요구할 수 있다.

2. PX4-Autopilot 환경에서의 수신 및 처리 아키텍처

GCS에서 전송된 COMMAND_LONG 혹은 COMMAND_INT 메시지 내부에 MAV_CMD_RUN_PREARM_CHECKS 명령이 담겨 수신되면, PX4의 명령 처리 파이프라인이 기동된다.

  1. mavlink_receiver.cpp 모듈 수신: 시리얼(UART) 혹은 이더넷(UDP) 리시버가 MAVLink 패킷을 파싱하고, MAV_CMD 식별자를 확인한다.
  2. uORB vehicle_command 토픽 발행: mavlink_receiver는 해당 명령을 PX4 내부의 메시징 버스인 uORB의 vehicle_command 구조체로 매핑(Mapping)한 뒤 퍼블리싱(Publishing)한다.
  3. Commander.cpp 수신 및 트리거: 시스템의 메인 상태 머신을 관장하는 커맨더 스레드가 이 토픽을 폴링(Polling)하다가 수신을 감지하면 내부 함수인 PreFlightCheck::preflightCheck()를 백그라운드가 아닌 전경(Foreground) 트리거 형태로 강제 실행한다.

이때 검증되는 주요 항목은 다음과 같다.

  • 센서(Sensors): 자이로, 가속도계, 자력계의 캘리브레이션 유효성 및 축의 편차 검증.
  • 상태 추정기(Estimators): EKF2 모듈의 융합 품질 검사, 위치 오차 분산치 허용 범위 이내 여부.
  • 안전(Safety): 배터리 전압 조건, RC 연결 상태(수동 비행 시), 킬 스위치(Kill Switch) 트리거 여부.

3. QGC와 PX4의 통신 트랜잭션 (상태 머신 다이어그램)

아래 도표는 QGroundControl에서 사용자가 혹은 자동화 스크립트가 해당 명령을 발생시킨 후의 피드백 트랜잭션을 나타낸다.

sequenceDiagram
    participant GCS as QGroundControl (v4.3)
    participant MAV_RX as PX4 MAVLink Rx
    participant UORB as uORB Bus
    participant CMD as PX4 Commander (Pre-flight)

    GCS->>MAV_RX: MAV_CMD_RUN_PREARM_CHECKS (CMDID: 401)
    MAV_RX->>UORB: vehicle_command 발행
    UORB->>CMD: 커맨드 이벤트 수신
    activate CMD
    CMD->>CMD: PreFlightCheck::preflightCheck(report=true)
    alt 점검 통과 (Passed)
        CMD->>UORB: vehicle_command_ack (ACCEPTED)
        UORB->>MAV_RX: ACK 발생
        MAV_RX->>GCS: COMMAND_ACK (Result: 0)
    else 점검 실패 (Failed)
        CMD->>UORB: vehicle_command_ack (DENIED)
        UORB->>MAV_RX: NACK 및 로깅 발생
        MAV_RX->>GCS: COMMAND_ACK (Result: 4)
        CMD->>MAV_RX: EVENT / STATUSTEXT 발생 (실패 사유)
        MAV_RX->>GCS: "PREFLIGHT FAIL: ACCEL 0 UNCALIBRATED"
    end
    deactivate CMD

PX4 펌웨어 설계의 핵심은, 이 명령을 호출할 때 report 플래그가 전경 호출(true) 상태로 설정되어 QGC UI상에 STATUSTEXT를 통한 치명적(Critical) 진단 결과가 직관적으로 전시되도록 유도한다는 점이다. 기본 백그라운드 체크에서는 관제석 화면에 과도한 로깅 도배를 막기 위해 일회성 상태 천이 실패로만 넘기지만, 이 온디맨드 명령은 구체적인 오류의 문자열 추적을 지원한다.

4. Ardupilot 대비 아키텍처 및 철학 비교

Ardupilot 환경에서는 비행 전 점검이 ARMING_CHECK 파라미터 비트마스크를 통해 철저하게 모듈별로 관리된다. Ardupilot의 지상 관제 시스템인 Mission Planner에서 ‘Arm’ 버튼을 누르면 내부 프로세스상 암(Arm) 시퀀스의 일환으로 암 점검이 자동으로 수행되며, 온디맨드 검사를 전용 MAVLink 명령으로 개별 호출하기보다는 MAV_CMD_COMPONENT_ARM_DISARM 요청 자체를 점검의 최종 트리거로 활용하는 경향이 짙다.

반면, PX4-Autopilot 환경에서는 시동 시의 점검과 병렬적으로, 외부 API(예: MAVSDK 혹은 ROS2 오프보드 에이전트) 기반 애플리케이션 개발 편의를 위해 MAV_CMD_RUN_PREARM_CHECKS 호출을 권장한다. 특히 로봇 운영 체제(ROS2) 환경의 uXRCE-DDS 브리지를 통해 연결된 컴패니언 컴퓨터가 비전(Vision) 기반의 임무를 수행하기 전, 에이전트 측에서 선제적으로 픽스호크의 물리적 상태를 질의할 때 이 명령이 매우 유용하게 쓰인다. 이를 통해 치명적 예외 상황(예: 시동 전 센서 단선)을 ROS2의 네비게이션 스택이나 자율 제어 노드가 안전하게 포착하고 플로우를 중단할 수 있다.

5. 소스 코드 깊은 이해 요약

이 메커니즘을 수정하거나 확장하려는 개발자는 PX4 소스코드의 src/modules/commander/Commander.cpp 내의 제어 루프를 분석해야 한다. _vehicle_command_sub.update() 블록 내부에서 vehicle_command_s::VEHICLE_CMD_RUN_PREARM_CHECKS 조건을 분기 처리하는 구조를 확인할 수 있다.
나아가 자신이 개발하는 독자적인 하드웨어 페이로드(Payload)나 센서의 안전성 점검 조건을 PX4 커맨더에 이식하고자 한다면, PreFlightCheck 모듈에 해당 하드웨어의 uORB 상태 객체를 참조하는 평가 함수(check_custom_payload())를 주입하는 설계 방법론이 요구된다.
