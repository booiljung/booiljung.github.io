42.3 센서(Sensor) 무결성 및 캘리브레이션 정밀 검증 알고리즘

고도화된 자율 비행 시스템에서 관성측정장치(IMU), 자력계(Magnetometer), 기압계(Barometer) 등 탑재된 센서(Sensor)들이 제공하는 원시 데이터의 질은 비행 제어기와 확장 칼만 필터(EKF2)의 안정성을 좌우하는 가장 1차적인 척도이다. PX4-Autopilot의 비행 전 점검(Pre-flight Check) 아키텍처 내에서 센서 무결성(Integrity) 검증과 캘리브레이션 정밀 검증은 하드웨어의 생존성뿐만 아니라 센서 결착/조립의 정확도까지 수리적으로 평가하는 깐깐한 수문장 역할을 수행한다.

본 절에서는 Commander 모듈 내에 구현된 서브시스템별 센서 무결성 추적 로직과 캘리브레이션 파라미터(Parameter)의 동적 일치도를 검증하는 알고리즘을 상세히 분석하라.

1. 센서 검증 아키텍처의 이론적 개요

드론 시스템 내 센서 검증은 크게 두 가지로 분류된다.

  1. 무결성 검증 (Integrity Verification)
  • 센서 칩셋 자체의 생존 여부와 버스 채널(I2C/SPI)의 통신 상태 확인.
  • 센서 데이터의 갱신 속도(Update Rate)와 미작동(Staleness/Timeout) 여부 검사.
  1. 캘리브레이션 정밀 검증 (Calibration Validation)
  • 주입된 캘리브레이션 오프셋(Offset)과 스케일 팩터(Scale Factor)가 현재 장착된 센서 칩의 ID와 매칭되는지 확인.
  • 정지 상태(Static Condition)에서의 필터링된 센서 출력값이 대기 상태 규격(예: 자이로의 경우 0\text{ rad/s} 주변)을 만족하는지 검사.

2. PX4 소스 코드 내 센서 무결성 검증(Integrity Check) 메커니즘

PX4의 Commander 클래스(src/modules/commander/)는 비행 전 PreflightCheck::sensor_check() 함수 군을 주기적으로 호출하여 uORB 토픽을 통해 드라이버 퍼블리시(Publish) 상태를 감시한다.

2.1 타임아웃(Timeout) 및 데이터 갱신(Staleness) 검사

드라이버가 초기화에 성공했더라도 런타임에 전원 딥(Power Dip)이나 버스 간섭으로 측정을 멈출 수 있다.

  • 동작 방식: uORB 함수인 orb_check() 또는 orb_stat()을 이용해 해당 센서의 timestamp가 시스템 현재 시간(hrt_absolute_time()) 대비 특정 타임아웃 범위를 넘었는지 판단하라. 예를 들어 IMU 데이터가 20ms 이상 갱신되지 않으면 해당 센서는 죽은 것(Dead)으로 간주되어 치명적인 에러 플래그가 세워진다.

2.2 다중 센서 불일치도 (Multi-sensor Inconsistency) 판별

최신 PX4 하드웨어(Pixhawk 4, Pixhawk 6C 등)는 2~3개의 중복(Redundant) IMU 스위트를 장착한다. PreflightCheck는 Primary(주) 센서와 Secondary(부) 센서들의 데이터 차이를 지속해서 비교한다.

  • 센서 A와 B 간 유클리드 거리 델타 \Delta = |Sensor_A - Sensor_B| 가 사전에 허용된 임계값 파라미터(예: COM_ARM_IMU_ACC, COM_ARM_MAG_ANG)를 초과하고, 이 상태가 앞서 배운 히스테리시스 필터링 윈도우 한계를 넘어서면 무결성 에러로 판단하고 이륙을 불허하라.

3. 캘리브레이션 정밀 검증 알고리즘 (Calibration Validation)

캘리브레이션 데이터는 센서의 공장 출고 오차 현상을 소프트웨어적으로 바로잡은 렌즈와 같다. PX4는 단순히 “캘리브레이션이 수행되었는지” 여부 상태 플래그(true/false)만 묻는 수준을 넘어서서 데이터 레벨의 다차원 검증을 실시한다.

3.1 Device ID 매칭 (Device ID Matching Algorithm)

센서의 물리적 조립 순서가 바뀌었거나, 펌웨어 업데이트로 인해 EEPROM 파라미터가 꼬였을 경우 기존 캘리브레이션 정보를 다른 센서에 적용하면 조종 불가(Flyaway) 상태가 발생한다.

  • 검증 로직: 저장된 파라미터 CAL_GYRO0_ID 등에 기록된 32비트 센서 고유 디바이스 ID와 현재 버스에서 탐색(Probe)된 device_id 값이 정확히 Bit-to-Bit로 매칭되는지 검사하라. 불일치할 경우 “Accels uncalibrated” 에러를 동반하며 QGroundControl(QGC) 화면에 재교정을 요구한다.

3.2 캘리브레이션 적용 후 데이터 정밀 평가 (Post-Calibration Static Check)

비행 직전 기체가 정지해 있을 때, 자이로(Gyroscope)의 데이터는 0^\circ/s이어야 하며, 자력계(Magnetometer)의 벡터 길이는 지구의 지역적 자기장 세기 크기 범위 안이어야 한다.

|Mag_{Meas}|_{\text{calibrated}} = \sqrt{M_x^2 + M_y^2 + M_z^2} \approx Magnitude_{\text{WMM}}

이처럼 World Magnetic Model(WMM) 데이터 또는 QGroundControl에서 사전 설정된 가우스(Gauss) 한계점과 최종 측정값을 비교하여, 외부 자기장 왜곡 스파이크(예: 주변 철골 구조물 간섭)가 있는지까지 최종 점검망을 형성한다.

4. Ardupilot 대비 아키텍처(Architecture) 및 설계 철학적 강점

Ardupilot 역시 AP_Arming::pre_arm_checks() 내부에서 센서들의 캘리브레이션 완료 상태와 healthy 속성을 체크한다.

하지만 PX4-Autopilot의 아키텍처는 EKF2(Estimator Control Library)가 주도하는 혁신 분산(Innovation Variance)과 파라미터(Parameter) 시스템이 더욱 융합된 형태를 띤다. 즉, PX4에서는 단순히 Commander가 센서 원시 데이터를 점검하는 1차 검증에 더해, 상태 추정기(EKF2) 자체가 센서 데이터와 운동학적 예측(Kinematic Prediction) 간의 오차를 판단하는 2차 검증 체계가 서로 이원화되어 작동한다는 데 차이점이 있다.

이러한 이중 레이어(Dual-layer) 검증 로직은 센서 캘리브레이션 결함이 숨어들어 생기는 예측 불가능한 추락을 막는 데 매우 탁월한 효과를 발휘하며, MAVLink의 STATUSTEXT 메시지로 사용자에게 구체적으로 어떤 센서의 어떤 ID 축에서 얼마나 오차가 발생하는지 추적된 친절한 경보를 제공할 수 있다.
