42.6.2.1. 왕복 시간(Round-Trip Time) 분산 추정을 통한 제어망 지터(Jitter) 한계 검사

무인기 시스템의 지상 관제망에서 단방향 지연 시간(Latency)만큼이나 중요한 성능 지표는 지연 시간의 변동성, 즉 **지터(Jitter)**이다. 특히 군집 비행 제어나 정밀한 오프보드(Offboard) 제어가 요구되는 환경에서는 지상 관제 시스템(GCS)과 비행 제어기(FC) 간 제어 명령 도달 시간의 일관성이 보장되어야 한다. 네트워크 혼잡이나 전파 간섭으로 인해 왕복 시간(RTT, Round-Trip Time)의 분산(Variance)이 커지면, 제어 샘플링 주기와 실제 명령 실행 시점 간의 비동기성이 심화되어 제어 루프의 위상 여유(Phase Margin)를 감소시키고 시스템의 불안정성을 유발할 수 있다. 본 절에서는 PX4-Autopilot 환경에서 RTT 분산을 추정하여 제어망의 지터 한계를 검사하는 원리를 분석한다.

1. 왕복 시간(RTT) 및 지터 추정 이론

지터는 패킷 도달 시간의 통계적 편차로 정의된다. GCS가 기체로 i번째 패킷을 송신한 시각을 S_i, 기체가 패킷을 수신하고 응답을 송신하여 GCS가 최종 수신한 시각을 R_i라고 할 때, i번째 패킷의 왕복 시간 RTT_i는 다음과 같다.

RTT_i = R_i - S_i

지터를 추정하기 위해 단기 RTT 평균 추정치 \bar{R}_i와 RTT 분산(편차) 추정치 V_i를 지수 가중 이동 평균(EWMA, Exponentially Weighted Moving Average) 기법으로 다음과 같이 갱신할 수 있다.

\bar{R}_i = (1 - \alpha) \bar{R}_{i-1} + \alpha RTT_i
V_i = (1 - \beta) V_{i-1} + \beta \vert RTT_i - \bar{R}_i \vert

여기서 \alpha와 \beta는 필터 계수(0 < \alpha, \beta < 1)이다. 이때 제어망이 수용 가능한 최대 지터 한계 임계값 J_{limit}를 초과하는지 검사하는 것이 지터 한계 검사의 핵심이다. 계산된 V_i가 J_{limit}를 상회하면, 시스템은 타이밍 불확실성이 위험 수위에 도달했다고 간주하여 오프보드 제어를 일시 중단하거나 보수적인 비행 모드로 전환을 유도해야 한다.

PX4-Autopilot의 TIMESYNC 아키텍처

PX4-Autopilot은 네트워크 지연 시간과 기체-GCS 간 시간 동기화를 위해 MAVLink의 TIMESYNC 메시지를 활용한다. 이 메시지는 단순한 연결 유지를 위한 HEARTBEAT와 달리, 고정밀 타임스탬프를 포함하여 RTT 및 오프셋을 계산하는 데 직접적으로 사용된다.

    participant FC as PX4-Autopilot
    participant GCS as GCS / Companion Computer
    
    Note over FC,GCS: 1. GCS 주도 동기화 요청

    Note over FC: 2. FC 수신 및 타임스탬프 첨부

    Note over GCS: 3. RTT 산출 및 Jitter 평가
    
    Note over FC,GCS: 반대 방향 동기화 (FC 주도)
    FC->>GCS: TIMESYNC (tc1=Timestamp_FC, ts1=0)
    GCS-->>FC: TIMESYNC (tc1=Timestamp_FC, ts1=Timestamp_GCS)
    Note over FC: FC 내부 RTT 산출 로직 동작

mavlink_timesync.cpp 모듈은 위와 같은 패킷 교환을 통해 GCS와 비행 제어기 간의 시간 오프셋을 보정(_time_offset)하고 RTT를 계산한다. 이 메커니즘을 통해 도출된 지연 시간 변동성(Jitter) 데이터는 telemetry_status uORB 토픽에 결합되거나 보조 컴퓨터(Companion Computer) 측의 미들웨어(ROS 2 등)에서 제어 주기 보정 메트릭으로 활용된다.

제어망 지터 한계 검사의 중요성 (Offboard 제어 관점)

네트워크 지터가 비행 제어에 가장 치명적인 영향을 미치는 영역은 Offboard 모드에서의 실시간 궤적 제어다. GCS나 보조 컴퓨터에서 생성된 목표 위치, 속도, 가속도(SET_POSITION_TARGET_LOCAL_NED 등) 명령은 일정한 주파수(예: 10Hz ~ 50Hz)로 유입되어야 한다.

  • 타임아웃 및 홀드: 지터로 인해 패킷 도달 간격이 불규칙해지면, 짧은 순간 타임아웃(Timeout ~0.5초 선)을 초과하는 경우가 잦아진다. 이때 PX4 Commander는 제어 권한을 상실한 것으로 파악하여 Position 모드나 Loiter 모드로 Fallback(안전 조치)을 시도하게 되며, 이는 드론의 덜컹거림(Jerky Movement)으로 나타난다.
  • 예측 및 지연 보상: 확장 칼만 필터(EKF2)나 속도 제어기 아키텍처 내부에서는 데이터의 도달 지연을 스무딩(Smoothing) 처리하지만, 지터 한계치(J_{limit})를 초과하는 심한 편차는 옵저버 레이어에서 제어 위상을 흔들어 발산(Oscillation)을 유발할 수 있다.

ArduPilot과의 제어망 지터 처리 대비

  • PX4-Autopilot: Offboard 명령의 도달 주기 안정성에 대해 매우 엄격한 잣대를 적용한다. 만약 명령 스트림의 주기성이 파괴되거나 지터 및 타임아웃 임계를 초과하면, 내부 상태 머신에서 즉시 Failsafe를 발동(Offboard lost)하여 자체 위치 제어로 강제 전환한다. 따라서, 네트워크 지터 평가와 억제가 오프보드 비행의 필수 전제조건이다.
  • ArduPilot: Guided 모드 명령 스트림 간격 편차에 대해 상대적으로 관대한 편이며, 명령 간 보간(Interpolation) 메커니즘과 더불어 지연 시간에 유연하게 대응하도록 설계된 파라미터 계층이 두텁다. 지터로 인한 즉각적인 모드 이탈(Fallback) 보다는 마지막 수신 명령을 부드럽게 유지시키며 복구를 대기하는 경향이 강하다.

소스 코드 로직 및 시스템 구성

PX4의 시간 동기화 모듈(mavlink_timesync.cpp) 내에는 RTT의 이상치를 필터링하기 위해 지수 가중 이동 평균 방식이 적용되어 있다.

// src/modules/mavlink/mavlink_timesync.cpp (개념적 요약)
void MavlinkTimesync::sync_converge(int64_t time_offset)
{
    // 필터링을 통해 지터의 영향을 최소화하는 로직
    if (_time_offset == 0) {
        _time_offset = time_offset;
    } else {
        // alpha(지수 가중치)를 활용한 스무딩 필터: jittering 상쇄
        _time_offset = (_time_offset_filter_alpha * _time_offset) + 
                       (1.0 - _time_offset_filter_alpha) * time_offset;
    }
}

왕복 시간(RTT)을 측정하고, 위 필터를 거치기 전후의 편차를 관측함으로써 지터 정도를 모니터링할 수 있다. 높은 신뢰성이 요구되는 애플리케이션의 경우, GCS 내부 혹은 MAVSDK 기반의 제어 노드에서 TIMESYNC 응답 시간을 수집한 뒤 수식에 입각한 분산 V_i를 독립적으로 계산한다. 분산이 미리 설정한 한계를 넘어서면, 네트워크 홉(Hop) 최적화, 무선 라디오 세팅(예: 백오프 타임 최적화, 안테나 점검) 등의 경고 알람을 띄우거나 선제적으로 제어 명령 주파수를 하향 조정하는 방어적 로직을 활성화해야 한다.
