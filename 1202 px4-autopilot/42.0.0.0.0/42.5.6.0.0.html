42.5.6 하드웨어 안전 스위치 논리적 잠금 해제 및 파일 시스템 무결성 점검

무인항공기(UAV)의 이륙 직전 상태인 비행 전 점검(Pre-flight Check) 단계에서는 소프트웨어적인 센서 및 통신 검증뿐만 아니라 물리적인 하드웨어 조작과 데이터 저장 매체의 건전성이 평가된다.

가장 대표적인 물리적 보호 장치는 기체 외부에 장착된 **하드웨어 안전 스위치(Safety Switch)**이며, 데이터 보존 측면에서는 비행 로그(ULog)와 기체 파라미터를 보관하는 **파일 시스템(MicroSD 카드 및 내장 Flash)**의 무결성(Integrity)이 핵심 점검 대상이다. 이 절에서는 PX4-Autopilot 아키텍처가 이 두 가지 이질적인 시스템 요소를 어떻게 논리적으로 통합하고 검증하는지 분석한다.

1. 하드웨어 안전 스위치(Safety Switch)의 논리적 잠금 해제 메커니즘

하드웨어 안전 스위치는 인명 피해를 방지하기 위해 드론의 프로펠러가 예기치 않게 회전하는 것을 물리적·논리적으로 차단하는 이중 안전장치이다. 주로 I/O 보드(예: Pixhawk의 PX4IO 코프로세서)에 직접 연결되며, 사용자가 버튼을 1~2초간 길게 누르면 빨간색 LED 패턴이 변경되면서 시스템 잠금이 해제된다.

1.1 PX4-Autopilot의 Safety 스위치 처리 구조

PX4 시스템(NuttX 커널 기반)에서 스위치의 상태 변화는 다음과 같은 소프트웨어 파이프라인을 거친다.

  1. 인터럽트 및 I/O 폴링: I/O 프로세서가 하드웨어 스위치의 핀 상태 변화를 감지하고, 디바운싱(Debouncing) 처리를 수행하여 일시적인 전기적 노이즈를 걸러낸다.
  2. uORB 메시지 발행: 안전 스위치가 해제(Un-safed) 상태로 확정되면, 시스템은 actuator_armed uORB 토픽 내의 manual_lockdown 또는 armed 연관 플래그를 업데이트한다. 최근 H/W HIL(Hardware-in-the-Loop) 철학의 변화에 따라, 최신 PX4 펌웨어에서는 I/O 칩셋 수준에서 타이머 레지스터를 직접 잠금 해제(PWM 출력 활성화)함과 동시에 메인 FMU 스레드에 이를 통보한다.
  3. Commander 모듈의 상태 병합: 소프트웨어 시동(Arming) 명령어(QGroundControl이나 외부 RC 스위치 조작)가 내려오더라도, Commander 모듈의 Pre-flight Check은 이 safety_button_available 및 safety_off 플래그가 True인지를 최우선 조건으로 평가한다. 만약 물리 스위치가 눌리지 않았다면, GCS로 “Pre-flight Check: Safety Switch is ON” 메시지를 반환하고 시동을 거부한다.

2. 파일 시스템(File System) 및 저장소 무결성 점검

비행 로그의 누락이나 캘리브레이션 파라미터의 변조는 추락 사고의 원인 규명을 불가능하게 만들거나 제어기의 비정상 거동을 유발한다. 따라서 기체 부팅 직후 및 Pre-flight 단계에서 파일 시스템은 엄격하게 검사된다.

2.1 비휘발성 저장 매체 검증 항목

  1. SD 카드 마운트 및 배드 섹터(Bad Sector) 확인:
  • PX4는 NuttX의 VFS(Virtual File System) 계층을 사용하여 FAT32 형식의 MicroSD 카드를 /fs/microsd 경로에 마운트한다.
  • 마운트 과정에서 파일 할당 테이블(FAT) 오류나 읽기/쓰기 응답 지연이 발생할 경우, SD 카드가 없거나 손상된 것으로 판별한다.
  1. 로깅 대역폭(Logging Bandwidth) 병목 방어:
  • logger 모듈은 고속 버퍼 링(Buffer Ring)을 생성하여 ULog를 작성한다. SD 카드의 Class 스펙이 낮거나 파일 단편화(Fragmentation)가 심하여 Write 속도가 센서 로깅 주파수를 따라가지 못할 경우, Drop_out율이 상승하며 이는 시스템 헬스 경고로 이어진다.
  1. MTD 파라미터 파티션 검증:
  • 파라미터(BSON 포맷)는 주로 보드 내부에 탑재된 FRAM이나 별도의 EEPROM (또는 SD 카드의 특정 파일)에 저장된다. 부팅 시 CRC32 해시 검증을 수행하여, 이전 비행 종료 시 전원 차단으로 인한 데이터 코럽션(Data Corruption) 유무를 확인한다.

3. PX4와 Ardupilot의 무결성 처리 및 데이타 로깅 철학 비교

파일 시스템 오류로 인한 페일세이프(Failsafe) 처리나 비행 제한 정책에 있어 두 오픈소스 플랫폼은 약간의 차이를 보인다.

  • PX4-Autopilot: PX4는 연구 수준의 엄밀한 비행 피드백 메커니즘을 강조한다. 기본적으로 SD 카드가 삽입되어 있지 않거나 파일 시스템 오류가 감지되면, 로깅이 불가능하다는 판단 하에 Pre-flight Check 단계에서 시동을 강력하게 거부(Arming Denied)하는 파라미터(COM_ARM_SDCARD)가 활성화되어 있다.
  • Ardupilot: 실전 응용과 임무 완수에 초점을 맞춘 Ardupilot도 DataFlash 로그의 중요성을 인지하지만, 파일 시스템 마운트 오류나 SD 카드 누락 시 조종사에게 강한 지상관제 오디오 및 텍스트 경고를 내보내면서도 강제 비행 옵션 설정 시 암(Arm)을 허용하는 편이다. LOG_BITMASK 등을 통해 유연하게 로깅 전략을 회피할 수 있는 여지를 준다.

4. QGroundControl(GCS) 관제 연동 로직

QGroundControl(v4.3.x) 환경에서 GCS 운용자는 MAVLink 프로토콜을 통해 기체의 물리적, 논리적 준비 상태를 투명하게 관제한다.

  • 안전 스위치 알림 (SYS_STATUS): MAVLink SYS_STATUS 메시지 내의 onboard_control_sensors_health 비트마스크를 통해 하드웨어 버튼의 해제 여부가 실시간으로 GCS 상단 아이콘 상태(빨간색/초록색)로 반영된다.
  • SD 카드 상태 표출 (STATUSTEXT): “Pre-flight Check: Missing SD card” 또는 “Parameter Error: CRC mismatch” 형태의 Severity Alert가 GCS 메시지 패널에 노출되어, 사용자가 강제 비행 전 반드시 플래시 스토리지를 포맷하거나 펌웨어 초기화 절차를 거치도록 유도한다.

결론적으로, PX4-Autopilot은 파일 시스템과 물리적 하드웨어 스위치라는 서로 다른 도메인의 요소를 하나의 Commander 상태 머신 아래로 정밀하게 편입시켜, 인간의 조작 실수(Human Error)와 매체 노후화에 의한 시스템 붕괴를 이륙 전에 차단하는 고도화된 아키텍처를 실현하고 있다.
