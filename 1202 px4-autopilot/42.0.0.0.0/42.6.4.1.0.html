42.6.4.1. 광선 투사 알고리즘(Ray-casting Algorithm)을 응용한 비볼록(Non-convex) 다각형 내 위상 수학적 포함(Inclusion) 여부 검증

무인항공기(UAV) 지오펜스(Geofence) 시스템에서 기체가 허용 구역 내부에 존재하는지 판별하는 것은 기하학적 ‘다각형 내 점의 포함(Point-In-Polygon, PIP)’ 문제로 귀결된다. 실제 비행 임무에서 요구되는 지오펜스는 모든 내각이 180도 미만인 단순한 볼록 다각형(Convex Polygon)이 아닌, 지형이나 사유지 경계를 따라 오목하게 파인 비볼록 다각형(Non-convex Polygon)인 경우가 대다수이다. PX4-Autopilot은 이러한 임의의 비볼록 다각형 경계 조건에서도 기체의 위치를 빠르고 정확하게 검증하기 위해 **광선 투사 알고리즘(Ray-casting Algorithm)**을 펌웨어 내부에 구현하였다. 본 절에서는 해당 위상 수학적 알고리즘의 원리와 PX4 내부 소스 코드 레벨에서의 최적화 구조를 분석한다. (기준 버전: PX4-Autopilot v1.14 이상, QGroundControl v4.3 이상)

1. 광선 투사 알고리즘(Ray-casting Algorithm)의 수학적 원리

광선 투사 알고리즘은 **조르당 곡선 정리(Jordan Curve Theorem)**라는 위상 수학의 기본 정리에 뿌리를 둔다. 이 정리에 따르면, 닫힌 단일 곡선(자기 교차가 없는 다각형)은 평면을 명확히 ’내부(Inside)’와 ‘외부(Outside)’ 두 영역으로 분할한다.

따라서 판별하고자 하는 임의의 점 P(x_p, y_p)(즉, 기체의 현재 로컬 좌표)에서 무한대를 향해 임의의 방향(일반적으로 X축 양의 방향)으로 광선(Ray)을 투사했을 때, 이 광선이 다각형의 경계선(선분)들과 교차(Intersection)하는 횟수를 센다.

  • 홀수(Odd) 번 교차: 점 P는 다각형 **내부(Inside)**에 존재한다.
  • 짝수(Even) 번 교차: 점 P는 다각형 **외부(Outside)**에 존재한다.

이 원리는 다각형이 볼록하든 오목하든(비볼록) 관계없이 항상 성립하므로 범용적인 지오펜스 알고리즘으로 채택된다.

2. PX4-Autopilot의 광선 투사 알고리즘 구현

PX4 내부의 commander 및 navigator 계통 소스 코드(주로 Geofence 관련 C++ 클래스)는 임베디드 장비(NuttX RTOS)의 연산 제약을 고려하여 부동 소수점 연산을 최소화하고 예외 상황을 안전하게 처리하도록 설계되어 있다.

2.1 알고리즘 동작 로직

기체의 현재 좌표 (x, y)와 N개의 꼭짓점 (x_i, y_i)을 가진 다각형이 주어졌을 때, 검사 루프는 다각형의 둘레를 따라 각 선분 (V_i, V_{i+1})에 대해 다음 연산을 반복한다.

  1. Y축 범위 검사 (Bounding Box 필터): 점의 Y좌표가 선분의 y_i와 y_{i+1} 사이에 존재하는지 확인한다. 이 조건을 만족하지 않으면 광선이 선분과 교차할 수 없으므로 무시(Skip)하여 연산 속도를 높인다.
  2. X축 교차점 계산: Y축 범위를 만족하는 선분에 대해, 점 P의 Y좌표를 지나는 수평선이 선분과 만나는 X좌표 값인 x_{intersect}를 비례식을 통해 계산한다.
    x_{intersect} = (x_{i+1} - x_i) \frac{(y - y_i)}{(y_{i+1} - y_i)} + x_i
  3. 교차 횟수 카운트: 기체의 X좌표 x가 x_{intersect}보다 작으면(즉, 광선을 우측으로 투사했을 때 선분과 만나면) 교차 카운트(Intersection 상태 플래그)를 반전(Toggle)시킨다.

교차 판별 구조 다이어그램

graph TD
    A[Start: Point P(x, y)] --> B[Initialize: inside = false]
    B --> C[Loop: For each edge V_i to V_i+1]
    C --> D{Is P.y between V_i.y and V_i+1.y?}
    D -->|No| G[Next Edge]
    D -->|Yes| E{Is P.x < intersection X?}
    E -->|No| G
    E -->|Yes| F[inside = !inside]
    F --> G
    G --> H{More Edges?}
    H -->|Yes| C
    H -->|No| I[Return inside state]

3. 엣지 케이스(Edge Case)와 예외 처리

순수 기하학적 광선 투사 알고리즘은 부동 소수점(Floating-point) 환경에서 특정 엣지 케이스를 만났을 때 오작동할 위험이 있다. PX4는 다음과 같은 방법론으로 안전하게 대응한다.

  • 광선이 꼭짓점(Vertex)을 정확히 관통할 때: 광선이 선분의 끝점을 지날 경우, 인접한 두 선분에서 교차가 두 번 카운트되어 내부/외부 판별이 뒤집히는 오류가 생길 수 있다. 이를 방지하기 위해 PX4의 검사 로직(C++ 코드 선에서는 y < y_i 및 y >= y_{i+1} 등의 부등호 조건)은 한쪽 끝점은 포함하고 다른 끝점은 배제하는 반개 구간(Half-open Interval) 방식을 사용하여 중첩 카운트를 차단한다.
  • 수평 선분과의 평행성: 다각형의 특정 선분이 X축과 완벽하게 수평을 이룰 때(즉 y_i = y_{i+1}), Y축 범위 검사 과정에서 분모가 0이 되는(Divide by Zero) 예외가 발생할 수 있다. PX4 로직은 Y좌표 범위를 먼저 평가하는 과정에서 수평 선분을 아예 교차 후보군에서 제외하도록 하여 이 문제를 자연스럽게 회피(Bypass)한다.

4. Ardupilot의 Ray-casting 아키텍처 비고

Ardupilot 역시 비볼록 지오펜스 폴리곤 판별을 위해 구조적으로 매우 유사한(수학적 근본이 동일한) Ray-casting 알고리즘을 사용한다.
하지만 Ardupilot 시스템은 추가적으로 Winding Number 알고리즘(회전수 방식: 점을 중심으로 다각형 둘레를 회전한 총 각도의 합이 360^\circ인지 0^\circ인지 판별)을 혼용하거나 보완적으로 사용하기도 한다. PX4-Autopilot은 Winding Number 대비 임베디드 환경에서 분기문이 적고 삼각함수 연산 부하가 월등히 적은 순수 Ray-casting 방식을 극도로 최적화하여 채택함으로써 제어 루프 레이트에 악영향을 주지 않도록 고안되었다.

5. 실무 관점에서의 제언

비볼록 다각형 경계를 QGroundControl에서 작성할 때는 너무 조밀한 꼭짓점(수백 개 이상)을 생성하지 않도록 유의해야 한다. Ray-casting 알고리즘의 시간 복잡도는 O(N)(단, N은 선분의 갯수) 이다.
PX4의 연산능력이 뛰어나더라도 N이 극도로 커지면 Navigator 스레드의 CPU 점유율이 일순간 스파이크(Spike)를 일으켜 Task Schedule Delayed 경고가 뜰 수 있다. 따라서 GCS 단에서 꼭짓점 단순화 툴(Polygon Simplification)을 거친 후 필수적인 10~50개 내외의 최적화된 Vertex 형태로 MAVLink를 통해 업로드하는 것이 가장 이상적이다.
