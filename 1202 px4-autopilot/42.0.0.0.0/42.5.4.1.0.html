42.5.4.1. CAN 버스 중재 손실(Arbitration Loss) 및 비트 스터핑(Bit Stuffing) 오류 프레임 빈도 분석

무인항공기(UAV) 네트워크 아키텍처가 발전함에 따라, PX4-Autopilot 생태계(v1.14 이상)에서는 기존의 PWM, I2C 통신을 대체하여 고신뢰성 드론캔(DroneCAN) 또는 실시간 UAVCAN 프로토콜을 탑재한 CAN(Controller Area Network) 버스의 채택이 가속화되고 있다. CAN 버스는 다중 마스터(Multi-Master) 구조로 동작하므로, 여러 노드(센서 단말, ESC, 비행 제어기 등)가 동시에 데이터를 전송할 수 있는 강력한 이점을 가지지만, 충돌 해소를 위한 중재 손실(Arbitration Loss) 및 동기화 무결성을 지키기 위한 비트 스터핑(Bit Stuffing) 영역에서 발생 가능한 오류를 면밀히 분석하는 것이 비행 안전성에 필수적이다.

이 문서에서는 CAN 버스의 패킷 레벨(Packet Level) 물리 계층에서 일어나는 중재 손실과 비트 스터핑 오류의 원인, 그리고 PX4-Autopilot 아키텍처가 이 빈도를 모니터링하고 예외를 캐치(Catch)하는 알고리즘을 분석한다.

1. CAN 버스 중재 손실(Arbitration Loss) 메커니즘

CAN 버스는 CSMA/CD+AMP (Carrier Sense Multiple Access with Collision Detection and Arbitration on Message Priority) 방식의 미디어 액세스 제어 프로토콜을 사용한다. 즉, 여러 노드가 전송을 시도할 때, 패킷의 식별자(ID)를 비교하여 무손실 비트 단위 중재(Bit-wise Arbitration)를 수행한다.

메시지가 버스에 실릴 때, 0은 우성(Dominant) 비트, 1은 열성(Recessive) 비트로 정의된다. 만약 노드 A가 전송을 시도하며 1(열성)을 보냈는데, 버스에서 0(우성)이 감지된다면, 식별자 우선순위가 더 높은 노드 B가 전송 중임을 의미한다. 이때 노드 A는 전송을 즉각 중단하고(Arbitration Loss 발생), 다음 버스 유휴(Idle) 상태가 될 때까지 기다린다.

  • PX4 측면의 위험성: 중재 손실 자체는 CAN 프로토콜의 지극히 정상적인 충돌 해소 메커니즘이다. 그러나 대역폭(Bandwidth) 포화 상태로 인해 후순위 노드(예: 중요도가 낮게 설정된 페이로드 데이터)가 지속적으로 중재에서 밀린다면, PX4의 Commander는 해당 노드가 타임아웃(Timeout)된 것으로 착각할 수 있으며, 이는 센서 누락 또는 Failsafe 트리거로 직결된다.

2. 비트 스터핑(Bit Stuffing) 개요와 오류 프레임(Error Frame)

CAN 프로토콜은 별도의 클럭(Clock) 라인이 존재하지 않는 비동기 통신이므로 수신 측 노드의 동기화를 돕기 위하여 비트 스터핑(Bit Stuffing)을 사용한다.

  • 전송되는 데이터 내에서 **연속으로 5개의 동일한 비트(예: 00000 또는 11111)**가 나타나면, 송신기는 하드웨어 칩 레벨에서 강제로 반대 극성의 비트 1개를 삽입(Stuffing)한다.
  • 수신기는 스트림을 읽다가 삽입된 이 반전 비트를 역으로 제거하여 원본 데이터를 복원한다.

2.1 비트 스터핑 오류(Bit Stuffing Error)의 발생 원인

만약 수신 노드가 연속으로 동일한 극성 6개의 비트를 감지한다면, 이는 비트 스터핑 규칙이 위반되었음을 알리는 명백한 전기적 및 논리적 물리 버스 결함(Stuff Error)이다.
드론 내부에서는 이 오류가 모터의 대전류(High Current) 동작 시 발생하는 강력한 전자기 간섭(EMI, Electromagnetic Interference), 케이블 쉴딩(Shielding) 결함, 또는 커넥터 체결 불량 등에 의해 프레임 물리 신호가 왜곡될 때 가장 빈번하게 발생한다.

3. PX4 통신 드라이버의 오류 분석 및 대응 아키텍처

PX4 비행 펌웨어(NuttX RTOS 기반) 내의 src/drivers/uavcan 모듈과 마이크로컨트롤러의 주변장치(CAN Controller)는 위 에러들을 통계적으로 캡처(Capture)한다. 오류 발생 빈도(Error Frame Rate)에 따라 노드는 다음 세 가지 상태(State) 로직을 전이한다.

  1. Error Active: TEC(Transmit Error Counter)와 REC(Receive Error Counter)가 127 이하인 통상적 상태. 모터 과도 응답 시 스터핑 오류가 산발적으로 발생해도 자가 복구된다.
  2. Error Passive: 에러 카운터가 127을 초과하면 돌입하며, 해당 노드의 통신 속도가 억제된다. PX4는 이 시점부터 uORB can_interface_status 인터페이스를 통해 GCS(지상 관제 시스템)로 점진적 경고를 띄울 준비를 시작한다.
  3. Bus Off: 지속적인 링잉 및 EMI 결함으로 에러 카운터가 255를 돌파하여 노드가 버스에서 논리적으로 강제 퇴출되는 오프라인 상태.

3.1 결함 상태 트리 다이어그램 시각화

    state ErrorActive {

        ArbitrationLoss : 우선순위 중재 실패 대기

    }
    
    ErrorPassive --> BusOff : 결함 빈도 극증 (TEC > 255)
    
    state BusOff {

    }

4. 설정 및 구성 (QGroundControl 연동)

CAN 네트워킹의 물리적 통신 신뢰도 저하를 막기 위해 QGroundControl(v4.3 이상) 환경 하 드론 배포 전 다음 파라미터 기반 구성(Configuration)과 점검 루틴 체계를 수행해야 한다.

  1. UAVCAN_ENABLE 활성화: 최신 펌웨어 기준, 파라미터 트리 메뉴를 거쳐 내부 통신 노출 포트의 CAN 데몬을 시작(Enable)하라.
  2. UAVCAN_NODE_ID 및 우선순위 할당: 중재 손실(Arbitration Loss) 트래픽의 누적을 예방하기 위하여, 매우 빠른 응답성이 보장되어야 하는 센서(메인 GPS, RTK 컴파스 모듈 등)는 DroneCAN DNA(Dynamic Node Allocation)를 통해 할당하거나 가급적 정적(Static)으로 낮은 ID(우선순위 높음)를 부가하도록 논리적으로 구성하라.
  3. MAVLink Console 통신 상태 진단: 비행 전(Pre-flight) 단계에서 터미널 창에 접속한 뒤 uavcan status 커맨드를 직접 기입하여 노드별 RX/TX 에러 프레임 누적 빈도수를 필히 파악하라. 비트 스터프 오류로 인한 Error Passive 전이 이력이 노출된다면 대전류가 흐르는 배선 라우팅을 재배치하거나 백업용 통신 차폐(Shielding) 케이블로 교환하라.

4.1 Ardupilot 버스 아키텍처와의 차별점 요약

Ardupilot은 전통적으로 CAN 인터페이스 운용 시 SLCAN 하드웨어 툴을 통한 칩셋 분배 방식과 고정적인 노드 캘리브레이션을 중시하는 구조이다. Ardupilot 환경에서 버스 중재 손실은 Data Flash 로그 파트를 통해 사후적으로 분절 분석되는 형태가 통상적이다.
반면 PX4-Autopilot의 아키텍처는 다수의 CAN 디바이스를 can_interface_status 등 uORB 토픽(Topic) 체계 하에 동기화시키며, Pre-flight HealthAndArmingChecks 알고리즘이 빈도수(Error Frequency)의 허용치를 일별 초과할 시 동적인 시동(Arming) 잠금 에러 정책을 실시간으로 포괄적이자 중앙집중적으로 대응케 한다.

요약하자면, PX4-Autopilot은 CAN 통신의 하부 물리 계층적 특성인 버스 중재 손실 및 단발적인 비트 스터핑 오류를 관대하게 시스템 내부에 자가 수용하는 듯하지만 파라미터와 노드 칩셋의 물리적 한계 임계점(Bus Off 진입 지표)으로 도달하는 특정 순간에 한하여 유연하면서도 강력한 페일세이프(Failsafe) 시퀀스로 전환하여 제어 체계의 무결성을 완수한다.
