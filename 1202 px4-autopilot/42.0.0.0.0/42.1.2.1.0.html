42.1.2.1. 항공전자시스템 설계 표준(DO-178C/DO-254) 관점에서의 소프트웨어 상태 검증 철학

1. 개요

이 장에서는 무인항공기(UAV) 및 자율 비행 시스템이 유인기에 준하는 안전성을 확보하기 위해 요구되는 항공전자시스템 설계 표준(DO-178C 및 DO-254)의 관점에서, PX4-Autopilot의 소프트웨어(SW) 상태 검증 철학을 집중적으로 살펴본다. 최근 도심항공교통(UAM) 기체나 대형 군용 무인기의 제어 시스템은 전통적인 오픈소스 관행을 넘어 감항 인증(Airworthiness Certification) 수준의 엄격한 시스템 안전 무결성(Safety Integrity)을 요구받고 있다. 따라서 시스템 기획, 하드웨어 계층, 펌웨어 상태 천이 검증이 각각 어떻게 유기적으로 연계되어 치명적인 장애(Fatal Failure)를 차단하는지 공학적으로 이해하고 설계에 반영하라.

2. DO-178C(소프트웨어) 및 DO-254(하드웨어) 인증 개요

항공기 탑재 소프트웨어 및 복잡한 전자 하드웨어의 안전을 보증하기 위해 국제적으로 널리 통용되는 가이드라인이다. 두 표준 모두 시스템 결함 발생 시 항공기와 승객, 지상에 미치는 영향도에 따라 등급(Design Assurance Level, DAL)을 A(재앙적, Catastrophic)부터 E(영향 없음, No Effect)까지 다섯 단계로 분류한다.

  • DO-178C (Software Considerations in Airborne Systems and Equipment Certification): 소프트웨어의 요구사항 분석, 설계, 코딩, 테스트 전 과정에 걸쳐 투명성과 추적성을 확보하도록 강제한다.
  • DO-254 (Design Assurance Guidance for Airborne Electronic Hardware): FPGA, PLD, ASIC 등 복잡한 전자 장치의 무결성을 보장하기 위한 하드웨어 설계 라이프사이클을 다룬다.

PX4-Autopilot 소스 코드는 자체적으로 DO-178C 인증 모델을 완벽하게 준수하도록 배포되지는 않는다. 그러나 상위 레벨의 안전 철학(코드 리뷰의 철저화, 정적 분석 툴 도입, Fail-Safe 아키텍처 적용)은 이 표준들의 요구사항을 강하게 반영하고 있다.

3. PX4-Autopilot의 DO-178C 지향적 소프트웨어 상태 검증 철학

DO-178C는 비행 제어 소프트웨어에서 “명확하지 않은 상태(Undefined State)“로의 진입을 엄격하게 금지한다. PX4는 상태 머신(State Machine) 설계 및 데이터 흐름 분리를 통해 이러한 위험에 대처한다.

3.1 결정론적 제어 루프 (Deterministic Control Loop)

PX4의 심장부인 NuttX RTOS 환경 하에서 각각의 제어기(Controller)와 추정기(Estimator) 모듈은 매우 엄격한 주기로 스케줄링(Scheduling)된다. 센서 데이터 취득부터 액추에이터 명령 하달까지의 과정이 마이크로초 단위의 시차(Jitter) 내에서 예측 가능하도록 설계되었다. 즉, 임의의 인터럽트나 스레드 기아(Thread Starvation) 현상으로 인해 상태 검증이 누락되는 상황을 최소화하라.

3.2 사전 정의된 상태 전이 (Pre-defined State Transition)

비행 제어 로직은 반드시 허용된 경로(Path)를 통해서만 다른 비행 모드나 시동 상태로 전이해야 한다. PX4의 Commander 모듈은 복잡한 다중 조건 논리를 단일 책임으로 묶어 상태 관리의 무결성을 높인다. 유효하지 않은 MAVLink 명령을 수신하더라도 이 모듈은 현재 상태 공간(State Space) 내에서 명령을 폐기(Reject)하여 기체를 보호한다.

다음 다이어그램은 DO-178C에서 요구하는 제어 불능 상태 방지를 위한 분리 검증 프로세스 개념을 보여준다.

4. Ardupilot과의 설계 기반 및 검증 철학 비교

PX4와 Ardupilot 모두 우수한 안전성을 목표로 하지만, 상태를 검증하고 분리하는 아키텍처 관점에서는 차이가 있다.

  • PX4 (uORB & 모듈화 기반): 독립된 각 프로세스(제어, 추정, 통신)가 uORB(Micro Object Request Broker)라는 비동기 메시징 버스를 통해 데이터를 주고받는다. 이 구조는 특정 모듈의 크래시(Crash)가 시스템 전체의 붕괴로 이어지는 것을 막는 결함 격리(Fault Isolation) 설계 관점에서 DO-178C의 철학에 더 친화적인 구조를 가진다.
  • Ardupilot (단일 스택 메모리 공유): 객체 지향적(C++) 철학에 따라 서브시스템이 강하게 결합되어 있어, 공유 메모리를 기반으로 빠른 연산과 유연한 커스터마이징을 제공한다. 반면, 하나의 객체 오류가 전체 동작 스레드에 영향을 줄 잠재적 가능성 측면에서는 PX4 대비 구조적 격리성이 낮다.

5. QGroundControl(GCS) 연동을 위한 MAVLink 무결성 검증

GCS는 원격에서 소프트웨어 상태 천이를 유발하는 핵심 매개체이므로, DO-178C 관점에서 GCS의 명령은 본질적으로 ‘비신뢰(Un-trusted)’ 입력으로 간주하여야 한다.
QGroundControl이 COMMAND_LONG 메시지를 통해 타겟 웨이포인트(Waypoint)나 모드 변경을 지시하면, PX4는 commander.cpp 내부에서 다음과 같은 다중 유효성 검증을 수행한다.

  1. SysID / CompID 일치 검증: 해당 명령이 반드시 자기 자신 기체(Vehicle)를 향한 것인지 확인하라.
  2. 전제 조건 매칭: “Arming 조건이 되지 않았는데 Takeoff 명령이 내려오는 상황“과 같은 논리적 데드락(Deadlock) 가능성을 즉시 차단(NACK 전송)하라.
  3. 지오펜스(Geofence) 및 배터리 조건 검토: 해당 임무가 물리적으로 가능한 범위를 벗어나지 않았는지 평가하라.

따라서 GCS를 커스터마이징하여 새로운 자율 비행 관제 시스템을 설계할 경우, 단순히 패킷을 보내는 데 그치지 말고, 상태 변화 거부(Command NACK, MAV_RESULT_DENIED 등) 결과를 수신할 때의 GCS측 Failsafe 상태 머신 또한 빈틈없이 설계해야 한다.

6. 설계 지침 (Engineering Guidelines)

항공전자시스템 설계 표준의 지침을 실무 관제 및 기체 설계에 도입하려면 다음 원칙을 준수하라.

  • 단일 고립의 원칙 적용: 새로운 드라이버나 ROS2 기반의 오프보드(Offboard) 노드를 개발할 때, 코어 제어기 모듈과 데이터 흐름을 철저히 격리(Isolation)하라.
  • 하드코드된 상태 천이 금지: 임시방편(Hack)으로 특정 함수 내에서 강제로 armed_status = TRUE로 변경하는 행위를 엄격히 배제하라.
  • 명시적 파라미터화(Explicit Parameterization): 비행 전 점검 조건을 수정하려면, 소스 코드를 편집하지 말고 반드시 PX4 Parameter 시스템(CBRK_* 등) 내에서 추적 가능하도록 설정하라.

소프트웨어 상태 검증은 단순한 버그 없는 코딩이 아니라, 논리적 흐름 상 사고로 이어질 수 있는 모든 “예측 불가능성“을 시스템 아키텍처 단계에서 제거하는 공학적 책임임을 상기해라.
