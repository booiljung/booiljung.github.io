42.6.2. GCS와의 양방향 데이터 링크 품질 및 Heartbeat 지연 시간(Latency) 점검

무인항공기(UAS) 운용에 있어서 지상 관제 시스템(GCS)과의 안정적인 양방향 데이터 링크(Bi-directional Data Link)는 텔레메트리 데이터 수신과 기체 제어 명령 송신을 위한 가장 핵심적인 인프라다. 통신 단절(Data Link Loss)이나 극심한 지연 시간(Latency)은 조종 불능 상태를 초래할 수 있으므로, 비행 전 점검(Pre-flight Check) 및 비행 중 상태 모니터링 단계에서 이를 엄격하게 점검해야 한다. PX4-Autopilot은 MAVLink 프로토콜의 HEARTBEAT 메시지를 기반으로 양방향 통신의 무결성을 평가하며 네트워크 지터(Jitter) 및 지연을 지속적으로 감시한다.

1. 양방향 데이터 링크 무결성 평가 원리

데이터 링크는 기체에서 GCS로 향하는 하향 링크(Downlink, 텔레메트리 스트림)와 GCS에서 기체로 향하는 상향 링크(Uplink, 제어 명령 및 파라미터 설정)로 구성된다. 통신 품질에 이상이 생기면 패킷 손실, 비트 에러율(BER, Bit Error Rate) 증가, 대기열 지연(Queueing Delay) 누적으로 인해 상호 간의 통신 지연 시간이 급증하게 된다.

PX4-Autopilot은 이 양방향 링크의 살아있음(Liveness)을 확인하기 위해 HEARTBEAT 메시지의 송수신 빈도를 측정한다. MAVLink 프로토콜 규격에 따라 GCS와 비행 제어기(Flight Controller)는 통상적으로 1Hz의 주기로 HEARTBEAT 패킷을 브로드캐스팅(Broadcasting)한다.

1.1 지연 시간(Latency)과 타임아웃 메커니즘

수신된 HEARTBEAT 메시지의 타임스탬프를 현재 시스템 시간과 비교함으로써 통신망의 일방향 지연(One-way Delay) 징후를 파악할 수 있다. 보다 정확한 왕복 지연 시간(RTT, Round-Trip Time)은 시간 동기화(Timesync) 메시지인 TIMESYNC 패킷을 전송하여 측정되지만, Failsafe 트리거의 관점에서는 GCS의 마지막 HEARTBEAT가 언제 도달했는지를 나타내는 **수신 타임아웃(Timeout)**이 절대적인 기준으로 작용한다.

네트워크 혼잡이나 전파 음영 지역 진입으로 인해 지정된 시간 임계치(예: 10초) 동안 GCS의 HEARTBEAT가 수신되지 않으면, 모듈은 GCS 연결 손실 상태(GCS Connection Lost)로 전환하며, 이는 곧 계층적인 Failsafe 절차를 촉발한다.

2. PX4-Autopilot 데이터 링크 GCS 점검 설정 및 구성

PX4-Autopilot(v1.14 이상)에서는 데이터 링크 손실(Data Link Loss) 판별 및 후속 안전 조치를 위해 다음의 파라미터들이 유기적으로 결합되어 작동한다.

  1. COM_DL_LOSS_T (Data Link Loss Timeout): GCS 등으로부터 텔레메트리(특히 Heartbeat) 수신이 중단된 시간을 감지하는 임계값(초 단위)이다. 이 시간이 초과되면 데이터 링크 손실로 간주된다. 기본값은 통상 10초로 설정되어 있으나, 장거리 비행 시 일시적인 페이딩을 고려하여 보수적으로 15~20초 수준으로 조정하기도 한다.
  2. NAV_DLL_ACT (Data Link Loss Action): 링크 손실이 확인되었을 때 기체가 취해야 할 행동을 정의한다. Return mode(RTL), Land mode, Hold mode, 비행 종료(Flight Termination) 등을 선택할 수 있으며, 이 파라미터 세팅에 따라 상태 머신의 천이가 결정된다.
  3. COM_RC_IN_MODE: RC 제어와 GCS 제어 중 어떤 것을 신뢰할 것인지 결정한다. 조종기(RC Radio) 없는 완전자율 비행(GCS Only) 환경이라면 GCS 링크의 무결성이 곧 조종 권한의 상실로 직결되므로 지연 시간에 더욱 민감하게 반응해야 한다.

2.1 ArduPilot 대조: GCS 관측 철학의 차이

ArduPilot과 PX4는 모두 GCS 페일세이프(GCS Failsafe) 개념을 갖추고 있으나 처방과 내부 상태 모델에서 차이가 존재한다.

  • PX4-Autopilot: Commander 모듈을 통제타워로 하여 GCS 연결 상태를 매우 엄격한 State-based 트리거 조건으로 취급한다. 통신 두절 시 정의된 복구 정책에 따라 즉각적으로 모드 전환(예: Auto -> Return)을 강제하며, 복구(Re-acquisition) 시에도 사용자 설정(COM_DL_REGAIN_T)에 따라 점진적으로 권한을 회복하는 보수적인 접근을 취한다.
  • ArduPilot: ArduPilot의 GCS Failsafe(FS_GCS_ENABLE) 역시 타임아웃 감지 방식으로 동작하지만, RC 신호가 살아있거나 AUTO 미션 수행 중일 때는 GCS 연결 유실을 덜 차단적으로(Non-blocking) 취급하도록(Continue with Mission in Auto) 개별 예외 처리가 유연하게 분산되어 있다.

3. 소스 코드 분석: MAVLink Heartbeat 감지 아키텍처

PX4의 내부 구조에서 GCS 연결 상태는 mavlink_receiver.cpp를 거쳐 telemetry_status 토픽으로 유통되며, 최종적으로 commander.cpp의 Failsafe 상태 머신에서 평가된다.

// src/modules/mavlink/mavlink_receiver.cpp (개념적 요약)
void MavlinkReceiver::handle_message_heartbeat(mavlink_message_t *msg)
{
    mavlink_heartbeat_t heartbeat;
    mavlink_msg_heartbeat_decode(msg, &heartbeat);

    // GCS 컴포넌트인지 확인 (시스템 ID 및 컴포넌트 ID 필터링)
    if (msg->compid == MAV_COMP_ID_MISSIONPLANNER || 
        heartbeat.type == MAV_TYPE_GCS) {
        
        telemetry_status_s tstatus{};
        tstatus.heartbeat_time = hrt_absolute_time(); // 마지막 Heartbeat 도달 시간 기록
        tstatus.telem_time = tstatus.heartbeat_time;
        
        // GCS 접속 유지 상태를 uORB로 전달
        _telemetry_status_pub.publish(tstatus);
    }
}

생성된 telemetry_status 데이터는 Commander 상주 스레드에서 주기적으로 (보통 수십 Hz 주기로) 검사된다. Commander는 현재 시각과 telemetry_status.heartbeat_time 간의 오프셋(Offset)을 계산하고, 이 값이 파라미터 COM_DL_LOSS_T를 초과하는 순간 다음과 같은 상태 플래그를 변형시킨다.

// src/modules/commander/Commander.cpp 의 Data Link Loss 감지 로직 예
const hrt_abstime time_since_last_gcs = hrt_absolute_time() - telemetry_status.heartbeat_time;

if (time_since_last_gcs > (hrt_abstime)(_param_com_dl_loss_t.get() * 1_s)) {
    // 타임아웃 발생
    if (!_gcs_connection_lost) {
        _gcs_connection_lost = true;
        // Data Link Loss Failsafe (NAV_DLL_ACT 동작) 시퀀스 진입
        set_failsafe_state(FailsafeState::DATA_LINK_LOSS);
    }
} else {
    // GCS 연결 회복 및 Latency 안정화 
    if (_gcs_connection_lost && (time_since_last_gcs < (hrt_abstime)(_param_com_dl_regain_t.get() * 1_s))) {
         _gcs_connection_lost = false;
         recover_failsafe_state();
    }
}

위와 같은 로직을 통해 단일 MAVLink 메시지 누락으로 인한 펄스성 오작동을 방지하고, 지속적인 통신망 단절 및 지연시간 폭주 상황에서만 정밀하게 회피 기동을 인가하게 된다.

4. QGroundControl 관점의 지연 시간 하한계 검사

GCS 측(QGroundControl 최신 버전)에서도 역으로 기체 통신 상태를 모니터링한다. QGC 내부의 Vehicle.cc 및 MAVLinkProtocol.cc 모듈은 타겟 노드(드론)로부터 넘어오는 HEARTBEAT 간의 수신 인터벌(Interval)을 누적 분석(Rolling Average)한다.

  • 지연 시간 시각화: 기체로부터의 지연(Latency)이 특정 범위를 넘어서면 QGC 화면 상단의 연결 상태 아이콘이 황색 또는 적색으로 경고를 띄운다.
  • Pre-flight Check 보류: 이륙 전 점검 단계에서 양방향 지연 시간이 임계값을 상회하여 제어 명령(COMMAND_LONG)의 ACK 응답 속도를 보장할 수 없다고 판단되면, 시스템 아밍(Arming) 버튼 자체를 비활성화(Disable) 처리하기도 한다.

따라서 항공 로보틱스 연구자는 PX4 시스템 개발 시 통신 채널에 유의미한 데이터 트래픽이 병목을 일으키지 않도록(예: 불필요한 고해상도 로그 및 영상 스트리밍 대역폭 잠식 등) 텔레메트리 레이트 조절 파라미터(MAV_1_MODE, MAV_1_RATE)를 사전에 조율해야 한다. 이러한 지연 시간의 근본적인 한계 점검은 기체 자체 시스템과 지상 관제 시스템 간의 강력한 커플링을 통해 무인기 비행 무결성을 담보한다.
