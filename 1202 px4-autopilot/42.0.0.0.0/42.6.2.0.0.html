42.6.2. GCS와의 양방향 데이터 링크 품질 및 Heartbeat 지연 시간(Latency) 점검

무인항공기(UAV) 시스템의 비행 전 점검(Pre-flight Check) 단계에서 지상 관제 시스템(Ground Control Station, GCS)과 비행 제어기(Flight Controller) 간의 양방향 데이터 링크 건전성은 필수 검증 요소이다. 단순히 물리적인 전파 강도(RSSI)를 측정하는 것을 넘어서, MAVLink 프로토콜의 생존 신호인 Heartbeat의 전송 지연 시간(Latency)과 수신율 분석을 통해 응용 계층(Application Layer)에서의 실제적 통신 품질을 담보해야 한다. 본 절에서는 PX4-Autopilot 아키텍처를 중심으로 GCS 양방향 데이터 링크 품질의 점검 메커니즘을 상세히 다룬다.

1. 개요 및 이론적 배경

PX4와 QGroundControl(QGC) 등 GCS 간의 통신은 주로 MAVLink (Micro Air Vehicle Link) 프로토콜을 기반으로 이루어진다. 이 프로토콜은 HEARTBEAT (Message ID: 0) 메시지를 일정한 주기(일반적으로 1Hz)로 양방향 교환하며 시스템의 연결 상태, 비행 모드(Flight Mode), 그리고 에이전트의 현재 상태 인지 여부를 갱신한다.

양방향 데이터 링크의 품질은 다음 두 가지 주요 지표로 평가된다.

  • 수신 패킷 손실률(Packet Loss Rate): 전송된 패킷의 시퀀스 번호(Sequence Number) 누락을 기반으로 산출되는 손실 비율.
  • 지연 시간(Latency / Round-Trip Time, RTT): 특정 명령어(예: PING 또는 시스템 시간 동기화 메시지) 송출 후 응답이 도달할 때까지 소요되는 총 시간.

네트워크 이론상 지연 시간이 임계치를 초과하거나 불안정(Jitter)하면, 자율 에이전트의 상태 동기화가 실패하거나 자동 복귀(RTL)와 같은 안전 장치(Failsafe)가 오작동할 수 있다. 따라서 비행 전 체계적인 통신 링크 무결성을 수학적, 구조적으로 평가하는 과정이 요구된다.

2. PX4-Autopilot의 Heartbeat 모니터링 아키텍처

PX4 내부에서는 MAVLink 인스턴스가 독립적인 모듈로 동작하며, GCS로의 Heartbeat 송신과 GCS의 Heartbeat 수신을 분리하여 관리한다.

2.1 uORB 토픽 기반의 연결 상태 관리

GCS로부터 HEARTBEAT 메시지가 MAVLink 수신 스레드(Receiver Thread)에 도달하면, 시스템은 telemetry_status uORB 토픽을 갱신한다. PX4 Commander 모듈은 이 토픽을 구독하여 GCS 연결의 건전성을 판단한다.

sequenceDiagram
    participant QGC as QGroundControl (GCS)
    participant MAVLink_RX as PX4 MAVLink 수신 모듈
    participant COMMANDER as PX4 Commander 모듈
    participant Failsafe as 안전 장치 (Failsafe)
    
    loop Every 1 Hz
        QGC->>MAVLink_RX: MAVLink HEARTBEAT Msg
        MAVLink_RX->>MAVLink_RX: 수신 Timestamp 및 시퀀스 검증
        MAVLink_RX->>COMMANDER: uORB: telemetry_status (Heartbeat 수신 갱신)
        COMMANDER->>COMMANDER: 타임아웃 감시 (COM_DL_LOSS_T)
        alt 타임아웃 초과 판단 (Latency > Threshold)
            COMMANDER->>Failsafe: 데이터 링크 상실 (Data Link Loss) 이벤트 트리거
        end
    end

2.2 통신 품질 지표: 타임아웃과 지연 시간 측정

PX4 파라미터 중 COM_DL_LOSS_T (기본값 10초)는 GCS로부터의 Heartbeat가 끊겼다고 최종 판단하는 타임아웃 임계치이다. 비행 전 점검 파이프라인에서, Commander는 GCS 연결이 일정 시간(예: 최근 2~3초) 내에 안정적으로 유지되었는지 검사한다.

또한 핑(PING, Message ID: 4) 메시지나 시스템 시간(SYSTEM_TIME, Message ID: 2) 메시지를 통해 RTT를 계측한다. RTT 계측 알고리즘 모델은 다음과 같이 정의된다.

RTT = T_{receive} - T_{send} - \Delta T_{processing}

비행 전 점검 시, 텔레메트리 레이블에서의 RTT 누적 분포의 편차(Jitter)가 지정된 허용도를 초과할 경우, 데이터 처리량 폭주 상태로 간주하여 이륙을 보류할 수 있다.

3. QGroundControl 아키텍처를 통한 MAVLink 관제

QGroundControl(이하 QGC)의 경우, C++ 백엔드의 구조에서 차량(Vehicle)과의 연결 상태를 감시하는 시스템이 마련되어 있다. QGC 소스 코드 내 관리 객체(src/Vehicle/Vehicle.cc)는 PX4-Autopilot에서 보내오는 HEARTBEAT와 텔레메트리 데이터 속도를 실시간 실시간으로 감시한다.

  • LinkManager 모니터링: QGC는 다중 링크 접속(예: UDP 연결과 시리얼 텔레메트리 동시 접속)을 보조한다.
  • Heartbeat 프로그레스바 및 UI 렌더링: QML 프론트엔드는 백엔드에서 집계된 지연 시간 통계(Latency Info)를 기반으로 헤더 바에 연결 상태를 그래픽으로 표시한다. 만일 GCS 측에서 PX4로부터의 Heartbeat가 규정된 시간(기본 3~5초 범위) 안에 감지되지 않으면, UI 상에서 통신 오류 알림을 표출함과 동시에, 조종자의 즉각적인 개입을 요구한다.

4. Ardupilot 대비 관제 차이점

Ardupilot 아키텍처는 보편적으로 GCS_Failsafe 서브루틴을 통해 Heartbeat 단절을 모니터링하며, AP_Mission 등 서브시템과 강하게 결합된 하드코딩(Hard-coding)된 타이머 스레드에 주로 의존한다. Mission Planner와 같은 Ardupilot 지향 GCS는 패킷 손실률에 보다 직접적으로 인터페이스 경고를 뿌리는 성향이 강하다.

반면, PX4-Autopilot은 상태 동기화 및 전반적인 이벤트 라우팅(uORB 아키텍처)의 이점을 적극 활용한다. 즉, GCS의 Heartbeat 상실은 단순한 통신 두절 에러에 그치는 것이 아니라, Commander 모듈 내 다중 계층 상태 머신(Multi-tier State Machine)의 전환 조건으로 통합된다. 이를 통해 텔레메트리 장애 시 대기, 귀환, 혹은 보조 통신망(Companion Computer의 LTE/5G DDS 등)으로의 매끄러운 롤오버(Rollover) 처리 능력을 유연하게 제공한다.

5. 자율 비행 응용: ROS2 연동 관점에서의 데이터 링크 품질

근래의 자율 에이전트 개발은 MAVLink에 국한되지 않고, PX4-ROS2 브리지인 uXRCE-DDS 구조를 빈번하게 이용한다. ROS2 Offboard 마이크로서비스 아키텍처 하에서 GCS의 개념은 분산 제어 시스템(Distributed Control System)의 노드들로 확장된다.

ROS2 환경에서 보조 컴퓨터의 제어 명령이 PX4로 인가될 때, uXRCE-DDS 미들웨어의 QoS 프로파일(예: Reliable vs Best Effort)에 따라 레이턴시는 극적으로 달라진다. 따라서 자율 비행 시스템의 비행 전 점검 단계에서는 DDS 노드 간의 상태(Ping/Pong 또는 ROS2 헬스체크) 지연 시간이 엄격한 기준(예: 실시간성을 보장해야 하는 50ms 미만) 내에 들어오는지 별도의 노드 모니터링 유닛이 검증해야 하며, 실패 시 PX4로 반려 트리거(Abort Trigger)를 푸시(Push)하는 추가 관제 기술이 요구된다.
