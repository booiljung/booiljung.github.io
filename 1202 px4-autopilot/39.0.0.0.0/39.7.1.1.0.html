39.7.1.1. MAV_RESULT 상태 전이 시계열 로그 분석을 통한 논리적 교착 상태(Logical Deadlock) 원인 추적

1. 개요

명령어가 기체에 무사히 도착했음에도 불구하고 시스템이 조종사의 의도대로 움직이지 않는다면, 그 원인은 십중팔구 PX4의 Commander 상태 머신이 해당 명령을 논리적으로 거부(Reject)하고 있기 때문이다. 통신망은 완벽하게 열려 있지만, 소프트웨어 로직이 장벽을 치고 있는 이 상태를 파악하기 위해서는 GCS 텔레메트리 로그(.tlog) 내의 COMMAND_ACK 패킷을 시간의 흐름에 따라 나열해 보는 시계열 분포 분석이 필수적이다. 본 절에서는 MAV_RESULT의 상태 전이(State Transition) 패턴을 통해 숨어있는 논리적 교착 상태(Logical Deadlock)를 끄집어내는 분석 기법을 다룬다.

2. MAV_RESULT 시계열 분포 모델링

분석의 첫걸음은 GCS가 COMMAND_LONG을 발송한 타임스탬프 t_{tx}와 기체로부터 COMMAND_ACK를 수신한 타임스탬프 t_{rx}, 그리고 그 속의 result 코드를 2차원 그래프의 타임라인 위에 매핑(Mapping)하는 것이다.

만약 조종사가 ‘자동 이륙(Auto Takeoff)’ 버튼을 눌렀을 때의 트랜잭션을 분석한다고 가정해 보자.

  • t_0: GCS 발송 (이륙 명령)
  • t_0 + 0.1s: 수신 MAV_RESULT_IN_PROGRESS (PX4가 명령을 접수하고 실행 준비 중임)
  • t_0 + 0.5s: 수신 MAV_RESULT_DENIED (이륙 준비 중 무언가 결함이 발생하여 취소됨)

이러한 상태 전이 로그는 단순한 “실패” 한 마디보다 훨씬 깊은 정보를 제공한다. 기체가 아예 명령을 이해 못한 것(UNSUPPORTED)도 아니고, 바빠서 무시한 것(TEMPORARILY_REJECTED)도 아니다. 실행을 시도(IN_PROGRESS)했으나 내부적인 검증 로직에 부딪혀 거절(DENIED)된 명백한 논리적 차단 증거다.

3. 논리적 교착 상태(Logical Deadlock)의 진단

시스템 트러블슈팅에서 가장 골치 아픈 현상이 바로 이 **논리적 교착 상태(Logical Deadlock)**다. 운영체제의 스레드 데드락과는 달리, 제어 로직 간의 선행 조건이 꼬여서 영원히 다음 상태로 넘어가지 못하는 현상을 말한다.

GCS 로그에서 다음과 같은 MAV_RESULT 시계열 패턴이 발견된다면 이는 전형적인 데드락이다.

  • GCS는 1초 간격으로 계속해서 ‘Arming(시동)’ 명령을 보낸다.
  • PX4는 그때마다 0.05초 만에 칼같이 MAV_RESULT_DENIED를 반환한다.
  • GCS는 기체가 왜 거부하는지 영문을 모른 채 타임아웃 횟수(최대 5회)를 다 채울 때까지 바보처럼 똑같은 명령을 반복한다.

이 교착 상태의 이면에는 “조종사는 시동을 원하지만, 기체는 시동 전제 조건(Pre-flight Check)이 만족되지 않아 시동을 걸 수 없는” 팽팽한 논리적 모순이 자리 잡고 있다. 명령-응답 파이프라인 그 자체는 기계적으로 초당 수십 회씩 완벽하게 작동하고 있지만, 시스템 전체의 임무 수행 관점에서는 완전히 멈춰버린(Deadlocked) 것이다.

4. 이종(Heterogeneous) 토픽 크로스 래퍼런싱 기법

MAV_RESULT_DENIED가 반복되는 원인을 밝혀내려면, COMMAND_ACK 로그 하나만 쳐다보고 있어서는 답이 나오지 않는다. 엔지니어는 동일한 시간대(Time Window)에 수신된 다른 텔레메트리 토픽들을 교차 검증(Cross-Referencing)해야 한다.

  1. STATUSTEXT (메시지 ID #253) 분석: PX4 Commander는 DENIED를 뱉을 때 보통 그 사유를 텍스트로 브로드캐스팅한다. “Preflight Fail: Compass absolute offsets too high“와 같은 문자열이 COMMAND_ACK 직전이나 직후의 타임스탬프에서 발견된다면 게임 끝이다.
  2. SYS_STATUS (메시지 ID #1) 비트마스크 분석: 텍스트 메시지가 유실되었더라도, SYS_STATUS 패킷의 onboard_control_sensors_health 비트마스크를 까보면 특정 센서(예: GPS나 광학 흐름 센서)의 플래그가 0(Unhealthy)으로 떨어져 있는 것을 시계열로 확인할 수 있다.
  3. ESTIMATOR_STATUS (메시지 ID #230) 확인: EKF2의 혁신(Innovation) 분산값이 임계치를 초과하여 위치 추정 신뢰도 플래그가 깨져 있는 경우도 잦은 거절의 원인이다.

이처럼 GCS 텔레메트리 로그 마이닝은 단순히 통신 연결망의 살고 죽음을 판별하는 것을 넘어, 파이프라인의 응답 딜레이와 상태 코드의 패턴을 추적하여 펌웨어 내부 깊숙한 곳에서 벌어지는 상태 머신 톱니바퀴의 논리적 충돌을 스캐닝(Scanning)하는 강력한 디버깅 도구로 활용된다.
