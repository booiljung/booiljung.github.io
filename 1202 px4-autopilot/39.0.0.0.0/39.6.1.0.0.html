39.6.1. 왕복 지연 시간(Round-Trip Time, RTT)의 동적 추정(Estimation) 모델

1. 개요

드론이 산등성이 뒤로 넘어가 전파 미약 지대(Shadow Zone)에 진입하면, 초당 10번씩 쾌적하게 들어오던 위치 데이터가 갑자기 1초에 한 번씩 뚝뚝 끊기며 들어오기 시작한다. 이때 지상 관제 시스템(GCS)이 여전히 “정상적인 네트워크“라고 착각하여 200ms 단위의 짧은 타임아웃을 적용하고 있다면, 시스템은 허공에 대고 불필요한 재전송 폭격을 퍼부어 그나마 남아있던 한 줌의 대역폭마저 질식(Choking)시키게 된다.

이러한 재앙을 피하기 위해서는 통신 환경의 변화를 GCS가 스스로 인지하고 실시간으로 타임아웃 한계선을 늦추는 왕복 지연 시간(RTT: Round-Trip Time) 추정 모델이 필수적이다. 본 절에서는 MAVLink 프로토콜 체계 내에서 어떻게 통신 지연의 척도를 동적으로 추정(Estimation)하는지 살펴본다.

2. MAVLink 내장 핑(Ping)과 하트비트(Heartbeat)를 통한 샘플링

RTT를 추정하기 위해서는 쉴 새 없이 네트워크의 맥박을 재는 탐침(Probe) 데이터가 필요하다. MAVLink에서는 크게 두 가지 메시지가 이 역할을 수행한다.

  1. PING 메시지 (Message ID #4): 네트워크 진단을 위해 설계된 전용 패킷이다. GCS가 스탬프 타임과 시퀀스 번호를 얹어 PING을 던지면, 기체는 이를 그대로 복사하여 반사(Echo)한다. GCS는 반사되어 온 패킷의 수신 시간과 패킷 내부의 원본 발송 시간을 빼서 정확한 1회 왕복 지연 시간(Sample RTT)을 마이크로초 단위로 얻어낸다.
  2. HEARTBEAT 주파수 분석: 1Hz(1초에 1번)로 들어와야 할 HEARTBEAT가 1.5초, 혹은 2초 간격으로 불규칙하게 도달한다면, 이는 곧 업링크/다운링크 채널 어딘가에 극심한 큐잉 지연(Queueing Delay)이 발생하고 있음을 암시한다.

GCS 네트워크 백엔드는 이렇게 수집된 원시 RTT 샘플(Raw Sample) 값들을 버퍼에 끝없이 밀어 넣으며 실시간 지연 시간의 근사치를 찾아가게 된다.

3. 단순 이동 평균(SMA)의 통계적 한계

가장 원시적인 RTT 추정 방식은 최근 N개의 샘플을 모아 단순히 평균을 내는 단순 이동 평균(SMA: Simple Moving Average) 기법이다. (예: 최근 5번의 핑이 10ms, 12ms, 15ms, 400ms, 13ms 라면 평균 RTT는 90ms로 산출)

그러나 무선 통신 환경에서 이 단순 평균 방식은 치명적인 약점을 노출한다.
이상치(Outlier)에 대한 취약성 때문이다. 위 예시처럼 우연히 전파 간섭으로 400ms라는 비정상적인 지연(Spike)이 단 한 번 발생했을 뿐인데, 단순 평균은 전체 타임아웃 기대치를 90ms로 훌쩍 끌어올려 버린다. 반대로 과거의 오래된 10ms 데이터들이 계속 윈도우 안에 남아있어, 실제로 드론이 멀어져 RTT가 200ms로 완전히 악화(Degradation)되었음에도 GCS가 통신망 상태가 나빠졌음을 깨닫는 데(Average를 끌어올리는 데) 지나치게 오랜 시간이 걸리는 관성 현상을 겪게 된다.

4. 동적 타임아웃(Dynamic Timeout) 마진의 기초

뛰어난 GCS와 PX4 시스템은 단순 과거의 평균을 믿지 않는다. 현재의 통신망이 “얼마나 빨리 변하고 있는가“라는 가중치를 끊임없이 계산하여 미래의 지연 시간을 예측(Predict)해 내야 한다.

이렇게 산출된 추정 RTT(Estimated RTT)는 단순히 통계 창(UI)에 핑 숫자를 띄워주는 데 그치지 않고, VehicleCommandManager 내부의 비동기 트랜잭션 타이머의 만료 기준선(Timeout Threshold)을 수시로 조정하는 핵심 파라미터로 주입된다. 즉, 핑이 느려지면 명령어가 실패했다고 포기하기까지 GCS가 더 오래 인내심을 갖고 끈질기게 기다려주게 되는 적응형(Adaptive) 파이프라인의 핵심 톱니바퀴가 되는 것이다.

이를 구현하기 위해 항공우주 네트워크 공학에서는 TCP 통신 이론에서 기원한 보다 고차원적인 시계열 평활화(Time-series Smoothing) 수학 모델을 GCS 큐에 이식하게 되는데, 이어지는 절에서 그 필터링 기법들을 구체적으로 해부한다.
GCS->>Channel: MAVLink PING 송출 (Seq: N, Time: T0)
activate GCS
Channel->>PX4: PING 수신
activate PX4
Note over PX4: Payload 무변경 인터럽트
최단 시간 반환 루틴
PX4->>Channel: PING 응답 (원형 Payload 유지)
deactivate PX4
Channel->>GCS: 응답 패킷 유입 (Time: T1)
deactivate GCS
Note over GCS: 신규 샘플 R = T1 - T0 도출
Estimator 큐 삽입


QGC 소스 코드 구현 양상을 들여다보면, 비동기 스레드 풀(Thread Pool) 또는 Qt Event Loop 내에서 위와 같은 무부하 트랜잭션을 1~5Hz 빈도로 스케줄링하며(가변적), 관제 채널에 가중을 주지 않는 오차범위 내에서 주기성을 확보하여 획득된 신규 RTT 통계치 $R_i$를 재전송 정책 결정 모듈로 전달한다.

## 39.6.1.3. 이상치(Outlier) 배제 필터와 전처리(Preprocessing) 무결성

무선 매질의 무작위성(Randomness)으로 인해, 일회성 하드웨어 스톨(Stall)이나 물리적 간섭(차량 통과 등)이 순간적으로 작용할 경우 RTT 샘플의 단층이 100배 이상의 폭등을 그리는 스파이크성(Spiky) 노이즈 이상치가 수집된다. GCS의 백엔드 모델 추정기(Estimator)는 이렇게 취득된 가공 전(Raw Phase) RTT 샘플 $R_i$를 곧장 시계열 함수에 대입하지 않고, 1차적으로 저역 통과 대역이나 윈도잉 기법의 이상치 절단 방호 체계를 가동해야 한다.

GCS 네트워크 설계 시 실장해야 할 필수적인 샘플 정제 기술은 다음과 같다.
1.  **합리적 임계 범위 차단(Clamping & Clipping)**: 계산된 $R_i$ 샘플이 타이머 해상도보다 명백히 작거나(물리 법칙 배제, 예: 0.1ms), 너무나도 긴 극단적 지연(시간적 이탈, 예: 5,000ms 초과)의 지표를 산출할 경우 모델 붕괴 방지를 위해 필터 문턱에서 기각(Reject) 처리한다.
2.  **모호성 해소를 위한 Karn's Algorithm 차용 원탁**: TCP 환경에서 차용된 강건 체계로서, 이미 최소 1회라도 재전송(Retransmission) 절차가 개입한 큐 패킷에 대해 나중에야 도착한 ACK는 과연 원본 패킷의 늦깎이 반환분인지 아니면 최근의 재전송 요청분에 기인한 반환분인지 귀속성 판독이 모호하다. 따라서 해당 트랜잭션의 왕복 측정치는 RTT 추정 모델 업데이트에서 배제시켜 통계적 순도를 지키는 규칙을 고수한다.

이처럼 논리적인 전처리 과정을 순조롭게 거친 정예 유효 샘플군만이 추정기 내부의 지수 가중 이동 함수(EWMA)로 최종 침투되어 평활화를 진행한다. 이상치 필터링이 견고하게 확립되지 못할 경우, GCS의 타임아웃 마진 임계치가 단시간에 부풀어 올라 긴박한 이양 조작(Preemption, Failsafe 회피 타각)이 요망될 시 제어 지령 딜루전(Dilution) 현상 등 전이성 기능 마비를 낳을 가능성이 팽배하다. 최근 채비율을 높이고 있는 PX4 기반의 ROS2(uXRCE-DDS) 플랫폼 환경에서는 이와 동일 선상의 통계적 채널 모델링 작업을 미들웨어 RTPS 레이어가 투명하게 자동 대행하므로 GCS를 개발함에 있어 최상단 레벨의 허가된 예산에만 집중할 수 있는 진보적 아키텍처를 지원한다.
