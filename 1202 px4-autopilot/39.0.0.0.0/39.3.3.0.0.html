39.3.3. 비동기 장기 실행(Long-Running) 명령의 피드백 루프 아키텍처

1. 서론: 단발성(One-shot) 제어의 한계와 비동기 장기 프로세스의 대두

MAVLink 기반의 비행 제어기 명령어망은 액추에이터 활성, 모드 변경 등 수십 밀리초(ms) 이내에 즉각적으로 완료되는 단발성(One-shot) 이벤트를 처리하는 데 최적화되어 있으나, 현대 무인기 시스템에서는 수 초에서 수 분이 소요되는 장기 실행(Long-Running) 프로세스의 비중이 점차 증대되고 있다. 대표적으로 관성 측정 장치(IMU)의 온도 보상 캘리브레이션, 방대한 양의 임무(Mission) 플래시 메모리 기록, 소프트웨어(Firmware) OTA 업데이트 진행 등이 이에 속한다.

단일 COMMAND_ACK 응답만으로 이러한 블로킹(Blocking) 작업의 결과만 기다리게 되면, 제어망은 해당 기간 동안 ’명령 유실’인지 ’정상 처리 중이나 늦는 것’인지를 분간할 수 없어 통신 타임아웃(Timeout) 오류를 양산하게 된다. 본 장에서는 PX4(펌웨어 v1.14 기준)가 MAV_RESULT_IN_PROGRESS(상태 값: 5) 열거형을 활용하여 QGroundControl(v4.3)과 어떻게 지연 없는(Non-blocking) 비동기 피드백 루프 아키텍처를 구성하는지 논의한다.

2. MAV_RESULT_IN_PROGRESS 기반 지연 피드백 루프(Latent Feedback Loop)

비동기 장기 실행 명령 전달 시 Commander 모듈은 명령의 문법과 사전 조건을 즉시 평가(Pre-condition Evaluation)한 후, 본 작업은 백그라운드 스레드(Background Worker Thread)로 넘긴다. 그리고 GCS를 향해 즉각 MAV_RESULT_IN_PROGRESS 플래그가 세팅된 첫 번째 COMMAND_ACK를 송신함으로써 네트워크 타임아웃 붕괴를 예방한다.

2.1 연속 피드백(Continuous Feedback)과 진행 스칼라(Scalar) 필드

COMMAND_ACK 패킷 내부의 progress 필드는 장기 실행 이벤트에 대한 선형 보간용(Linear-interpolation ready) 0~100 사이의 스칼라 백분율(Percentage) 데이터를 담는다.

  1. 상태 머신 브리지(State Machine Bridge): PX4 내부의 각 캘리브레이션 루틴 혹은 태스크(Task)는 vehicle_command_ack uORB 토픽을 초당 1~5회(1~5Hz) 정도 지속적으로 퍼블리시(Publish) 하도록 설계되어 있다.
  2. 소켓 유지 메커니즘: 이 과정에서 MAVLink Receiver 모듈은 이 uORB 토픽을 수신할 때마다 GCS로 IN_PROGRESS ACK 패킷을 연속 스트리밍(Streaming)한다.
  3. 최종 상태의 캡슐화(Final State Encapsulation): 백그라운드 작업이 100% 완료되거나 중간에 오류로 중단되면, 마지막 COMMAND_ACK 패킷이 각각 MAV_RESULT_ACCEPTED 또는 MAV_RESULT_FAILED로 변이되어 발송되며 장기 실행 큐 루프를 종료(Terminate)시킨다.

3. Ardupilot과 PX4의 UI 바인딩(Binding) 철학적 차이

이 IN_PROGRESS 응답을 받아 조종자의 화면에 어떻게 가시화할 것인가에 대한 처리 철학은 PX4와 Ardupilot 진영이 사뭇 다르다.

  • PX4-Autopilot의 명시적 상태 파이프라인 (Explicit State Pipeline):
    PX4는 이 IN_PROGRESS 기반의 정량적 진행률(progress 필드) 활용을 시스템 전반에 걸쳐 아주 표준적으로 하드코딩해 두었다. 특히 캘리브레이션 과정에서 조종자가 기체를 각 축(X, Y, Z)으로 돌릴 때의 진행 상황이 QGroundControl(QML)의 원형 프로그레스 바(Circular Progress Bar)에 숫자 베이스로 매우 부드럽고 정확하게 바인딩(Binding)된다.
  • Ardupilot의 텍스트 의존적 피드백 (Text-dependent Feedback):
    Ardupilot의 스크립트 기반 생태계(Mission Planner 주도)에서는 IN_PROGRESS 스칼라 값을 0~100으로 세분화해서 보내주는 과정 구현이 일관되지 않은 모듈들이 더러 존재한다. 대신 이벤트 중간중간 STATUSTEXT 메시지로 “Calibration 30%…”, “Rotate vehicle…“과 같은 원시 문자열(Raw String)을 빈번히 전송하여 GCS 화면 하단의 텍스트 콘솔에 이를 띄우는 레거시(Legacy) 방식을 여전히 혼용한다.

QGroundControl(v4.3)에서는 PX4 기체가 연결되었을 때 이 MAV_RESULT_IN_PROGRESS 패킷이 도착할 때마다 백엔드 큐(Wait Queue)의 **타임아웃(Timeout) 타이머를 0으로 리셋(Reset)**하여 커넥션 드랍을 억제하는 고정밀 안전 설계가 내장되어 있다. 즉, IN_PROGRESS 루프는 시스템의 비동기적 응답성을 인간 조종자의 체감 인지 속도와 결합하는 가장 효율적인 인터페이스 공학 기술이라 할 수 있다.
