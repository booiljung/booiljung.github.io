39.2.3. COMMAND_INT (#73) 메시지의 고정 소수점(Fixed-Point) 보정 아키텍처

1. 개요

앞선 39.2.2절에서 상세히 논증했듯, WGS84 위도/경도와 같은 정밀한 글로벌 좌표 데이터를 32비트 단정밀도 부동소수점(float, 유효숫자 7자리) 슬롯에 담아 전송하는 COMMAND_LONG(#76) 아키텍처는 필연적으로 1미터 이상의 공간 해상도 붕괴(Truncation Error)를 유발한다. 이 치명적인 수학적 한계를 극복하기 위해 MAVLink 설계자들은 기존의 범용 명령어 껍데기(Container)를 완전히 재설계한 새로운 목적지 특화형 제어 패킷을 창안해 냈다. 그것이 바로 COMMAND_INT (Message ID #73) 패킷이다.

COMMAND_INT는 부동소수점이 가진 ’넓은 다이내믹 레인지(Dynamic Range)’라는 범용성을 과감히 포기하는 대신, 글로벌 좌표 데이터를 10^7 스케일링(Scaling)된 32비트 고정 소수점(Fixed-Point) 정수(int32_t) 포맷으로 강제 형변환(Type Casting)하여 탑재한다.

본 절에서는 이 COMMAND_INT 패킷의 메모리 레이아웃을 분석하고, 어떠한 기작으로 부동소수점의 절단 오차를 완벽하게 회피(Bypass)하며 오차 없는 무결점(Zero-loss) 공간 해상도 전송을 달성하는지, 제한된 대역폭 하에서의 데이터 인코딩 철학을 기술적으로 심층 분석한다.

2. COMMAND_INT 구조체의 하이브리드(Hybrid) 메모리 레이아웃

COMMAND_INT 메시지는 이전의 COMMAND_LONG이 지녔던 7개의 float 슬롯 구조를 버리고, 정수와 실수가 혼합된 하이브리드 바이트 패킹(Hybrid Byte Packing) 구조를 취한다. 전체 페이로드 길이는 35바이트로 COMMAND_LONG보다 단 2바이트만 증가하여 네트워크 대역폭 부담을 거의 늘리지 않았다.

C 구조체(mavlink_command_int_t)의 메모리 맵은 역순 크기 재배치(Size-Descending Reordering) 원칙에 따라 다음과 같이 구성된다.

필드명 (Field)자료형 (Type)크기 (Bytes)설명
param1 ~ param4float16 (4 \times 4)정밀도가 크게 요구되지 않는 파라미터 4개 유지 (예: 각도, 속도)
x (위도/X축)int32_t410^7 스케일링된 WGS84 위도 (또는 로컬 X 좌표)
y (경도/Y축)int32_t410^7 스케일링된 WGS84 경도 (또는 로컬 Y 좌표)
z (고도/Z축)float4미터(m) 단위의 절대/상대 고도

가장 주목해야 할 혁신은 바로 param5와 param6라는 애매한 float 슬롯을 폐기하고, 그 자리에 int32_t 자료형을 갖는 명시적인 x, y 필드를 박아 넣었다는 점이다.

3. 고정 소수점(Fixed-Point) 스케일링 알고리즘 원리

GCS(예: QGroundControl)에서 사용자가 선택한 WGS84 좌표 37.5665350 (북위, double 형)이 드론(PX4)으로 넘어가기까지의 직렬화 파이프라인은 다음과 같이 재정의된다.

3.1 단계: 10^7 (천만) 배율 뻥튀기 (Scaling Up)

GCS 백엔드는 double 좌표값에 정확히 부호 있는 10,000,000 (10^7)을 곱하여 소수점을 우측으로 7칸 이동시킨다.

  • 37.5665350 \times 10^7 = 375665350.0

3.2 단계: 양자화(Quantization) 및 32비트 정수형 캐스팅 (int32_t)

소수점 이하에 남아있는 잔여 무효 숫자들을 절삭하고 순수한 32비트 부호 있는 정수(int32_t) 껍데기에 이 데이터를 집어넣는다.

  • int32_t x_encoded = static_cast<int32_t>(375665350.0);
  • int32_t 자료형은 대략 -21 억부터 +21 억까지 무오류(Zero-Error) 전수 저장이 가능하므로, 최댓값이 \pm 180,000,000 (경도 180^\circ \times 10^7)인 WGS84 스케일링 데이터를 오버플로우 없이 완벽하게 감싸 안을 수 있다. 즉, 소수점 이하 7째 자리의 정밀도 데이터가 절단(Truncation) 오차의 간섭비 없이 메모리에 “정수“로서 안전하게 박제(Preservation)되는 것이다.

3.3 단계: 전송 및 PX4 역직렬화 복원 (Scaling Down)

이 거대한 정수 375665350은 바이트 훼손 없이 COMMAND_INT 네트워크 패킷에 실려 드론 비행 제어기(Pixhawk)에 도달한다. PX4 펌웨어 내부의 내비게이션 추정기 모듈(Navigator/EKF2)은 이를 꺼내 다시 10^7 로스리스(Lossless) 실수 나눗셈을 수행하거나, 시스템 내부 구조체에 그대로 int32_t 형태로 보관하여 위치 제어기(Position Controller)의 피드백 루프 참조값으로 사용한다.

4. 정밀도 확보(Precision Assurance) 및 1cm 이내 공간 해상도 구현

이 10의 7승 스케일링 아키텍처는 WGS84 좌표 체계의 지표면 거리 계산에서 놀라운 마술을 부린다.
위도(Latitude) 소수점 아래 7째 자리(0.0000001^\circ) 하나가 지표면 상에서 가지는 실제 거리를 계산해 보면 다음과 같다.

  • 1^\circ \approx 111,320 \text{ m} 이므로,
  • 0.0000001^\circ \times 111,320 \text{ m/}^\circ \approx 0.0111 \text{ m} = \mathbf{1.11 \text{ cm}}

과거 COMMAND_LONG의 float 한계로 인해 조종사가 감수해야 했던 강제 반올림 **오차 1.11 미터(111 \text{ cm})**가, 오직 이 프로토콜 슬롯 형변환 하나(\text{float} \to \text{int32\_t})를 통해 무려 100배의 해상도 향상 효과를 누리며 1.11 센티미터(1.11 \text{ cm}) 수준까지 정교하게 조율된 것이다.

이는 곧 GCS에서 1cm 단위로 심어놓은 초정밀 웨이포인트를 픽스호크 메모리 끝단까지 한 치의 열화(Degradation) 없이 순수하게 도달시킬 수 있는 고속도로가 개통되었음을 의미한다.

5. 공학적 시사점 및 프로토콜 마이그레이션(Migration) 원칙

PX4 및 ArduPilot 커뮤니티는 이 치명적 오차 해결을 자축하며 소프트웨어 개발자들에게 다음과 같은 강력한 개발 철학과 가이드라인을 반포하였다.

“위치나 목적지 좌표(Latitude/Longitude)가 포함된 어떠한 형태의 글로벌 명령 제어(Global Waypoint, RTL, Land 등)라도, 절대 과거의 COMMAND_LONG(#76)을 사용하지 마라. GCS 클라이언트를 코딩할 때는 무조건 새로운 COMMAND_INT (#73) 패킷과 MAV_FRAME_GLOBAL_INT 궤적 프레임 규범을 최우선(Primary) 객체로 사용하라.”

오늘날 QGroundControl의 임무 계획(Mission Planning) 모듈 소스코드를 열어보면, 겉으로는 사용자가 COMMAND_LONG을 클릭한 것처럼 보일지라도 백엔드의 트랜잭션 매니저는 이를 하부 네트워크 레이어에서 보이지 않게 COMMAND_INT 패킷 큐(Queue)로 암묵적 대체(Implicit Replacement)하여 발송하고 있음을 확인할 수 있다.

이처럼 COMMAND_INT의 고정 소수점 보정 아키텍처는 “단순히 숫자를 어떻게 메모리에 구겨 넣을 것인가“하는 컴퓨터 구조학(Computer Architecture)의 근본적인 자료형(Data Type) 고민이, 어떻게 하늘을 날아다니는 물리적 무인기의 치명적 충돌 궤도를 교정하는 훌륭한 항공 공학적 수학 솔루션이 되었는가를 입증하는 가장 극적인 사례비(Case Study)로 꼽힌다.
