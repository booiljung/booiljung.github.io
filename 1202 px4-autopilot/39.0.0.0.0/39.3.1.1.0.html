39.3.1.1. 발생지 명령어 고유 식별자(Command ID)와 응답 패킷의 1:1 매핑 알고리즘

1. 서론: 1:1 매핑의 시스템 버스(System Bus) 수준 요구사항

드론 지상 관제 시스템(GCS: Ground Control Station)과 비행 제어기(Flight Controller) 간의 제어 프레임워크는 기본적으로 원격 프로시저 호출(RPC: Remote Procedure Call) 구조와 유사한 양상을 띤다. GCS에서 조종자가 특정 기능을 활성화하거나, 자율 비행 에이전트 측 ROS2 노드가 임무(Mission)를 하달할 때, 시스템은 송신된 제어 패킷(COMMAND_LONG 등)과 수신될 결과 패킷(COMMAND_ACK)을 논리적으로 연결해야 한다. 이 상호 참조를 결속하는 알고리즘이 바로 MAVLink 프로토콜의 ’발생지 명령어 고유 식별자(Command ID) 기반 1:1 매핑’이다.

본 장에서는 PX4(펌웨어 v1.14 기준)와 QGroundControl(v4.3 기준)의 아키텍처 내부에서 이 1:1 매핑이 어떠한 스레드 세이프(Thread-safe)한 비동기 큐(Asynchronous Queue) 자료구조를 통해 달성되는지 심층적으로 논의한다.

2. :1 매핑 알고리즘의 동작 메커니즘

COMMAND_ACK 패킷 설계의 핵심은 단일 16비트(bit) 정수형 command 필드에 있다. 이 필드는 단순히 명령어 열거형(Enum) 숫자를 반환하는 것을 넘어, 메모리 상의 ’대기 중인 명령 해시 테이블(Pending Command Hash Table)’의 인덱스 키(Key)로 기능한다.

2.1 QGroundControl과 PX4 간의 인메모리(In-Memory) 매핑 로직

  1. 송신측 (QGroundControl) 해시 테이블 등록 (Registration)
    GCS에서 비행체로 MAV_CMD_DO_REPOSITION(#192)과 같은 제어 명령을 보낼 때, QGroundControl의 C++ 백엔드인 VehicleCommandManager 클래스는 내부적으로 고정 슬롯(Fixed Slot) 배열이나 해시 테이블에 현재 타임스탬프(Timestamp)와 함께 192라는 고유 식별자를 기록한다. 또한, 이 기록 객체(Record Object)에는 타임아웃(Timeout) 콜백이 바인딩된다.
  2. 수신측 (PX4 Commander) 상태 평가 및 콜백
    PX4 내부의 commander 디먼은 uORB를 통해 vehicle_command 토픽을 수신하면, 이를 vehicle_command_ack로 처리한 후 다시 MAVLink 파이프라인으로 넘긴다. 이때 페이로드 생성 함수는 원본 메시지의 command 필드(192)를 그대로 하드카피(Hard-copy)하여 응답 구조체에 주입한다.
  3. 송신측 (QGroundControl) 식별자 충돌 회피 (Collision Resolution) 및 팝(Pop)
    응답이 GCS로 되돌아오면 메인 텔레메트리 파서(Telemetry Parser)는 즉각 테이블에서 해당 ID(192)를 조회(Lookup)한다. 매치율(Match Rate) 100%가 확인되면 대기 중이던 콜백 함수(Callback Function)가 MAV_RESULT 코드 0(설정 성공) 혹은 기타 에러 코드와 함께 비동기로 실행(Resolve)되며 메모리에서 제거(Pop)된다.

3. Ardupilot과의 1:1 매핑 큐(Queue) 관리 차이

비단 PX4 뿐만 아니라 전 세계 무인기 생태계를 양분하고 있는 Ardupilot 역시 MAVLink를 사용하므로 기본적인 1:1 매핑 알고리즘 자체는 공유한다. 하지만 두 시스템 간에는 응답 지연 시 GCS에서 큐를 비워내는 아키텍처 철학이 상이하다. 맞춤형 GCS를 개발하거나 ROS2로 자율 에이전트 시스템을 꾸릴 때 이 부분은 치명적인 버그의 원인이 되곤 한다.

  • PX4-Autopilot의 엄격한 동기성 (Strict Synchrony):
    명령 전달 시점으로부터 특정 타임아웃(예: 500ms) 구간 내에 식별자 매칭이 되지 않으면 QGroundControl과 PX4 로직은 매우 신속하게 해당 Command ID 해시를 만료(Expire)시켜버리고, 이후 도착하는 ’지연된 매핑 메시지(Stale ACK)’는 좀비 패킷(Zombie Packet)으로 간주하여 즉시 파기한다.
  • Ardupilot의 유연한 비동기 지연 허용 (Permissive Latency Tolerance):
    Ardupilot 환경에서는 다중 큐잉(Multiple Queuing) 시 간혹 GCS가 이전 매핑 레코드를 꽤 길게 유지하는 경향성을 띤다. 네트워크 지연(Latency)이 심한 환경(예: 장거리 SATCOM 연동)에서 매핑이 늦게 완료되더라도 이를 수용하고 에이전트 상태를 사후 갱신(Post-Update)해주는 로직에 더 특화되어 있다.

4. 비동기 프로그래밍 모델에서의 한계점 및 시퀀스(Sequence) ID의 부재

현재 MAVLink 2.0 프로토콜 레벨에서 COMMAND_ACK 매핑 알고리즘이 가진 가장 큰 태생적 한계는 동일한 명령을 연속으로 두 번 송신했을 때 각각의 ACK를 완벽하게 구별해낼 Sequence ID가 존재하지 않는다는 것이다.

예를 들어 GCS에서 카메라 트리거 명령(MAV_CMD_IMAGE_START_CAPTURE)을 0.1초 간격으로 2번 보냈다고 가정하자. 두 응답 모두 동일한 command ID 필드 값을 가지고 돌아오므로, GCS의 매핑 알고리즘은 첫 번째 응답이 첫 번째 명령에 대한 것인지, 혹은 운 나쁘게 첫 번째 응답은 유실되고 두 번째 명령에 대한 응답이 빠르게 먼저 도착한 것인지 식별할 보장된 방법이 없다. 따라서 고정밀 관제 시스템 설계 시, 한 명령어의 1:1 매핑 루프가 완전히 Pop 되기 전까지 동일 ID 명령의 전송을 뮤텍스(Mutex)로 직렬화(Serialization)하거나, ROS2 DDS QoS 계층으로 올려 시퀀스 무결성을 별도로 담보하는 방식론을 도입해야만 한다.
