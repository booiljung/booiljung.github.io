39.4.1.2. ITU-T X.25 규격의 순환 중복 검사(CRC) 다항식 연산을 통한 페이로드 무결성 검증 블록

1. 서론: 페이로드 오염(Payload Corruption)의 치명성과 해시(Hash) 기반 1차 방어선

무선 텔레메트리 채널은 대기근(Atmospheric Noise), 모터의 전자기 간섭(EMI), 다중 경로 페이딩(Multipath Fading) 등에 의해 바이트 스트림(Byte stream)이 언제든 왜곡될 수 있는 본질적 취약성을 내포한다. 만약 “고점을 10m로 설정하라“는 커맨드의 페이로드 중 단 1비트(Bit)가 뒤집혀 “고도를 110m로 설정하라“로 오인식된다면, 무인기는 통제 불능의 치명타를 입게 된다.

이러한 하드웨어 물리 계층 단의 비트 변환(Bit Flip) 오류를 논리 계층으로 전파시키지 않기 위해, MAVLink 프로토콜과 PX4의 mavlink_receiver 모듈은 패킷의 끝자락에 16비트 순환 중복 검사(Cyclic Redundancy Check, CRC) 코드를 반드시 포함하여 수신단에서 유효성을 교차 검증(Cross-validation)하도록 강제한다. 본 절에서는 ITU-T X.25 표준을 따르는 CRC-16-CCITT 사양이 어떻게 PX4 파이어월(Firewall)로서 기능하는지 그 공학적 메커니즘을 분석한다.

2. MAVLink CRC-16 다항식(Polynomial) 연산과 Extra CRC 블록 구조

PX4 펌웨어에서 mavlink_parse_char() 함수가 패킷의 종단(END)에 도달하면, 수신한 페이로드 바이트들을 몽땅 집어넣고 자체적인 CRC 생성기(Generator)를 돌려 계산한 16비트 해시값과, 패킷 꼬리에 실제로 실려온 16비트 CRC 해시값을 대조(Verification)한다.

  1. ITU-T X.25 다항식의 채택:
    비행 제어 소프트웨어는 연산 부하가 적으면서도 오류 검출률이 100%에 가까운 해시 펑션을 필요로 한다. MAVLink는 x^{16} + x^{12} + x^5 + 1 이라는 X.25 규격 규약의 생성 다항식을 채택하였다. 이 알고리즘은 단일 비트 오류는 물론, 홀수 개의 비트 오류, 그리고 16비트 이하 길이의 연속된 버스트(Burst) 오류를 수학적으로 무조건 검출해 낸다.
  2. MAVLink만의 독특한 서명: Extra CRC Seed (CRC Extra):
    기본적인 CRC 연산이 [메시지 길이 + 헤더 + 페이로드]를 대상으로 순회(Iteration) 연산을 마친 직후, MAVLink는 마지막으로 그 메시지 ID 고유의 매직 바이트(Magic Byte, Extra CRC Seed) 하나를 추가 연산에 집어넣는다. 이 매직 바이트는 각 메시지(예: HEARTBEAT, COMMAND_LONG)의 XML 정의(XML Definition)에 기술된 필드명, 타입 설정 등에 기반하여 빌드 타임(Build Time) 파이썬 스크립트에 의해 자동으로 생성된 8비트 기호이다.

3. Ardupilot 대비 다이얼렉트(Dialect) 버전 충돌 방지 아키텍처

이러한 Extra CRC 메커니즘의 도입은 단순한 노이즈 필터링을 넘어, Ardupilot 및 PX4 생태계가 겪었던 ’버전 불일치 오류(Version Mismatch Error)’라는 고질병을 해결한 천재적인 아키텍처 설계로 평가받는다.

  • MAVLink 1.0 초기 시절의 재앙 (Legacy Vulnerability):
    과거에는 “메시지 ID 73번“이 버전 1.0에서는 [A, B]라는 두 필드만 갖다가, 버전 1.1에서 [A, B, C] 필드로 확장(Extension)되었을 때, GCS(신버전)가 송신한 패킷을 FC(구버전)가 받을 경우, 길이가 맞지 않아 메모리 파손(Memory Corruption)이나 심각한 버퍼 오버런(Buffer Overrun)을 발생시켰다.
  • Extra CRC를 통한 프로토콜 진화 방어 (Evolutionary Defense):
    PX4의 mavlink_receiver는 오직 자신의 펌웨어에 컴파일된 MAVLink 헤더(Header) 상의 Extra CRC Seed 값만을 사용하여 체크섬(Checksum)을 검증한다. 만약 QGroundControl이 새로운 필드가 추가된(버전이 다른) COMMAND_LONG 맵을 기초로 패킷을 쏘았다면, 조립된 X.25 다항식 끝단에 섞이는 매직 바이트가 달라지므로 CRC 대조가 반드시 실패하게 된다.
    즉, “구조체 설계도(Definition)가 1비트라도 다른 송신자와 수신자” 간의 대화는 커널 딴에서 조용히, 그러나 완벽하게 기각(Drop)된다.

이 설계는 GCS와 드론 펌웨어 개발 사이클이 완전히 엇박자(Asynchronous Update)로 돌아가는 오픈소스 로보틱스 생태계 환경에서, 상호 호환되지 않는 언어(Dialect)로 인한 치명적인 메모리 매핑 사고를 수학적으로 봉쇄하는 최강의 호환성 게이트키퍼(Compatibility Gatekeeper) 역할을 수행한다.
