39.1.1. 무선 주파수(RF) 및 셀룰러 데이터 링크에서의 프로토콜 한계 분석

현대의 무인항공기(UAV) 운영 환경은 시계 비행 범위를 넘어서는 비가시권(BVLOS, Beyond Visual Line of Sight) 비행을 위해 다양한 무선 데이터 링크를 사용한다. 전통적으로 사용되어 온 433MHz, 900MHz(915MHz) 대역의 ISM(Industrial, Scientific, and Medical) RF 텔레메트리 라디오부터 최근 도입이 활발해진 LTE/5G 등의 셀룰러 데이터 링크에 이르기까지, 각각의 통신 채널은 비행 제어 및 관제 메시지(MAVLink)를 전송하는 데 있어 물리적, 구조적인 한계를 내포하고 있다. 본 절에서는 이러한 특수한 무선 네트워크 환경이 전송 프로토콜 계층에 어떠한 제약과 한계를 가하는지 공학적 관점에서 분석한다.

1. 물리적 채널의 특성과 통신 오류 요인

GCS(QGroundControl)와 PX4-Autopilot 기반의 기체 간 연결은 일반적으로 유선망이나 근거리 Wi-Fi 네트워크와는 현격히 다른 물리적 채널 환경에 놓인다.

1.1 RF 텔레메트리 환경의 한계

일반적인 RF 텔레메트리 라디오 모듈(예: SiK Radio)은 시리얼(UART) 인터페이스를 통해 MAVLink 데이터를 송수신한다. 이 라디오 링크는 주로 Frequency Hopping Spread Spectrum(FHSS) 기술을 사용하여 간섭을 회피하지만, 본질적으로 대역폭(Bandwidth)이 매우 좁다(예: 57.6kbps 또는 115.2kbps).
가장 큰 한계는 다음과 같다.

  • 높은 패킷 오차율(PER, Packet Error Rate): 거리가 멀어짐에 따라 신호 감쇠능력이 급격히 떨어지고 노이즈 플로어(Noise Floor)가 상승하여 패킷의 손상이 잦아진다.
  • 비대칭적 가시거리(Asymmetric Line-of-Sight): 도심지 빌딩숲이나 산악 지형에서는 다중 경로 페이딩(Multipath Fading)으로 인해 일시적인 통신 음영 지역(Blackout)이 빈번하게 발생한다.

1.2 셀룰러(LTE/5G) 네트워크의 한계

셀룰러 데이터 링크는 이론적 대역폭이 넓어 영상(Video) 피드와 컨트롤 채널을 동시에 통합 운영할 수 있다는 장점이 있으나, 제어 공학 측면에서는 다음과 같은 프로토콜 한계를 동반한다.

  • 지터(Jitter)와 가변 지연시간(Variable Latency): 기지국(eNodeB/gNodeB) 상황과 패킷 라우팅 경로에 따라 수 밀리초(ms)에서 수 초 단위까지 전송 지연이 불규칙하게 발생한다. 이는 정밀한 타이밍 제어가 필요한 동기식 명령(Synchronous Command) 처리에 맹점으로 작용한다.
  • 핸드오버(Handover) 딜레이: 기체가 고속으로 이동하며 기지국의 서비스 영역을 넘어갈 때 발생하는 핸드오버 과정에서 일시적인 패킷 드랍과 물리 계층의 일시적 단절 현상이 초래된다.

2. 전송 계층(Transport Layer) 프로토콜의 딜레마

위와 같은 열악한 물리 계층(PHY) 및 데이터 링크 계층(MAC) 특성은 상위 전송 계층 네트워크 프로토콜 설계 시 큰 딜레마를 안겨준다.

2.1 TCP의 Head-of-Line(HoL) Blocking 문제

보편적인 신뢰성 프로토콜인 전송 제어 프로토콜(TCP, Transmission Control Protocol)을 셀룰러 네트워크나 Wi-Fi를 통해 드론 원격 제어망으로 사용할 경우, 데이터 스트림의 순서 보장 원칙이 오히려 장애가 된다.
TCP는 네트워크의 특정 패킷 하나가 손실되면, 해당 패킷이 재전송되어 수신자에게 도달할 때까지 후속 패킷들을 버퍼에 가두고 상위 애플리케이션(이 경우 MAVLink 수신 노드)에 넘겨주지 않는다. 이를 Head-of-Line(HoL) Blocking 현상이라 한다.
무인기 환경에서는 1초 전의 유실된 자세 제어 패킷을 다시 받기 위해 현재 가장 시급한 명령어나 최신 자세 데이터(ATTITUDE)가 버퍼에 갇혀 버리는 것은 시스템 추락으로 이어지는 매우 치명적인 약점이다.

2.2 UDP/시리얼(MAVLink)의 상태 비저장(Stateless) 아키텍처

이러한 한계로 인해 PX4-Autopilot를 포함한 MAVLink 통신망은 주로 시리얼 통신 위에 시퀀스 넘버(Sequence Number)를 포함한 프레이밍을 사용하거나, 네트워크에서는 **사용자 데이터그램 프로토콜(UDP, User Datagram Protocol)**을 채택하여 설계된다.
UDP는 상태 비저장(Stateless) 아키텍처를 가지며 단일 패킷의 송수신만 책임질 뿐 앞뒤 패킷의 순서나 수신을 보장하지 않는다. HoL Blocking이 발생하지 않으므로 가장 최신의 중요한 텔레메트리 데이터를 즉시 처리할 수 있다는 이점을 제공한다.
그러나 이는 MAVLink 애플리케이션 계층이 누락된 명령어나 패킷 순서를 자체적으로 검증(Checksum 및 Sequence Counter 확인)하고 명령에 대한 NACK/ACK 및 재전송(Retransmission) 메커니즘을 직접 구현해야 함을 의미한다. (이는 명령어 전달 신뢰을 보장하는 COMMAND_LONG 및 ACK 처리 로직의 근간이 된다.)

3. PX4-Autopilot의 통신 링크 소프트웨어적 대응 아키텍처

PX4의 mavlink 모듈은 RF 환경과 네트워크 한계를 극복하기 위해 대역폭 및 트래픽 양을 제어하는 세밀한 구성 매개변수(Parameter)를 제공한다.

  • MAV_X_RATE: 특정 스트림(예: MAVLink Instance X)의 최대 전송 속도 제한. 라디오 대역폭의 한계를 넘어 지연율이 급상승하는 병목 현상(Congestion)을 막기 위해 사용된다.
  • MAV_X_FORWARD: 네트워크 파티셔닝(Network Partitioning) 대비용 라우팅 옵션. 여러 통신 포트를 하나의 시스템 안에서 포워딩하여 이중화(Redundancy)된 RF 및 셀룰러 링크를 동시에 활용할 수 있도록 지원한다.
  • 패킷 폭주/손실 시나리오 처리: Ardupilot의 GCS_MAVLink 클래스와 마찬가지로 PX4도 자체적인 MAVLink 프레이밍 버퍼를 가지고 있으며, 손실된 프레임이 생기면 Drop 카운터를 증가시키고 이후에 들어오는 온전한 패킷부터 다시 해석하는 상태 머신(Parsers State Machine)을 사용한다.

결론적으로, 무선 주파수 및 셀룰러 네트워크는 불안정성과 제어 불가능한 지연을 지니고 있기 때문에, 무인기 통신의 하위 계층에서 무결성을 보장하려는 시도는 실패할 수밖에 없다. 따라서 PX4와 지상 관제 시스템 간의 설계는 프로토콜 계층에 신뢰성을 일임하지 않고, 애플리케이션 계층인 MAVLink 프로토콜의 핸드셰이크(Handshake, COMMAND/ACK)와 재전송 타이머를 통해 시스템 레벨에서 신뢰성을 달성하는 아키텍처로 진화하게 되었다.
