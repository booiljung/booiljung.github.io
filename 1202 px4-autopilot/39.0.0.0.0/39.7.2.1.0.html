39.7.2.1. vehicle_command와 vehicle_command_ack 토픽 간의 마이크로초 단위 타임스탬프 델타 연산

1. 개요

PX4 펌웨어 내부에서 명령어 처리의 심장부는 uORB(Micro Object Request Broker) 통신망이다. MAVLink 수신 모듈이 파싱한 명령어는 vehicle_command 토픽으로 변환되어 뿌려지고, 이를 낚아챈 Commander 모듈은 처리가 끝난 뒤 vehicle_command_ack 토픽을 발행하여 내뱉는다.
이 두 토픽의 생성 시간 차이는 PX4 운영체제(NuttX RTOS)가 얼마나 빠르고 쾌적하게 명령어를 소화하고 있는지를 나타내는 가장 순수하고 직관적인 생체 지표(Biomarker)다. 본 절에서는 ULog에 기록된 마이크로초(Microsecond, \mu s) 단위의 타임스탬프를 델타(\Delta t) 연산하여 스케줄링 지연(Latency)을 정량화하는 기법을 다룬다.

2. uORB 타임스탬프(Timestamp) 구조체의 정밀도

PX4의 모든 uORB 메시지 구조체(.msg 파일)는 항상 첫 번째 필드로 uint64 timestamp를 가진다.
이 값은 기체가 부팅된 시점(또는 시스템 클럭 동기화 시점)으로부터 흘러간 절대 시간을 마이크로초 단위로 기록한 위상 상수다.

  • t_{cmd} (vehicle_command.timestamp): 직렬 포트(UART) 인터럽트 핸들러를 거친 mavlink_receiver 스레드가 패킷의 CRC 오류 검증을 마치고, uORB 버스에 명령어를 Publish한 정확한 찰나의 시간.
  • t_{ack} (vehicle_command_ack.timestamp): Commander 스레드가 자신의 워크 큐(Work Queue)에서 깨어나 해당 명령어를 평가(Validation)하고 펌웨어 내부 상태를 변경한 뒤, 그 결과(Result)를 uORB 버스에 Publish한 시간.

3. 델타(\Delta t) 연산과 스케줄링 지연(Latency)의 정량화

시스템 엔지니어는 Flight Review나 PlotJuggler와 같은 ULog 분석 도구를 열어 동일한 command ID를 가진 두 토픽의 타임스탬프 차이를 계산한다.

\Delta t = t_{ack} - t_{cmd}

정상적이고 건강한 PX4 시스템에서 이 \Delta t 값은 Commander 스레드의 구동 주파수(보통 50Hz, 즉 20ms 주기)의 절반인 10ms ~ 20ms 이내로 일정하게 수렴해야 한다.
만약 GCS가 타임아웃을 겪었던 문제의 시간대에서 이 \Delta t 값이 100ms 심지어 500ms 이상으로 폭증(Spike)하는 현상이 발견된다면, 이는 무선 통신망의 문제가 아니라 100% 드론 내부의 CPU 코어나 메모리 버스가 극심한 변비(Bottleneck)에 걸려 명령어 처리 스레드가 마비되었음을 의미한다.

39.7.2.1.3. 델타 스파이크(Delta Spike)의 원인 추론

이렇게 \Delta t 연산을 통해 내부 스케줄링 지연이 범인으로 지목되었다면, 엔지니어는 지연이 발생한 타임스탬프 구간에서 시스템(OS) 부하를 가중시킨 다른 요인들을 ULog를 통해 찾아 나선다.

  1. SD 카드 I/O 블로킹(Blocking): 로거(Logger) 스레드가 비트레이트가 높은 센서 데이터를 SD 카드에 무리하게 쓰려다가 파일 시스템 병목에 걸리면서, 메인 데이터 버스를 독점해 버린 경우.
  2. CPU 점유율(Load) 폭주: EKF2(칼만 필터)나 비전 처리 동기화 루틴 등 연산 집약적인(Compute-intensive) 스레드가 찰나의 순간 CPU 점유율을 100%로 끌어올리며 Commander 스레드의 실행할 기회(Time Slice)를 빼앗아 버린 경우.

이처럼 마이크로초 단위의 uORB 타임스탬프 델타 연산은 허공에 떠드는 GCS의 불평이 아니라, PX4 펌웨어 스스로가 기록한 하드웨어 및 OS 레벨의 명백한 병목 증거를 제시하는 강력한 수사 기법이다.
