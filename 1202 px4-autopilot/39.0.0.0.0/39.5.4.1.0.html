39.5.4.1 VehicleCommandManager 클래스 내부 트랜잭션의 상태 머신 격리(Isolation)

단일 비행체 내에 다수의 센서와 액추에이터가 존재하고 여러 모듈(임무 계획기, 설정 창, 조이스틱 등)이 동시다발적으로 명령을 내리는 QGroundControl(QGC) 관제망에서는, 각 명령 트랜잭션(Transaction)이 충돌 없이 온전히 생애 주기를 마칠 수 있도록 상태 머신(State Machine) 격리(Isolation) 아키텍처가 필수적이다. QGC 백엔드의 VehicleCommandManager 클래스는 수많은 비동기 명령어들의 내부 상태를 독립적인 상태 머신 다발(FSM Array)로 캡슐화하여, 다른 트랜잭션의 타임아웃이나 재전송 로직이 현재의 패킷 제어 흐름에 간섭하지 못하게 하는 고도화된 스레딩 관리 기법을 채택하고 있다.

1. 전역 상태(Global State) 관리의 위험성

초기 드론 관제 인터페이스나 취미용 스크립트에서는 기체의 명령 처리 상태를 전역 변수(예: is_command_pending, current_waiting_command) 하나로 플래그(Flag) 처리하곤 하였다. 이러한 디자인 패턴은 극심한 한계를 갖는다.

  1. 병렬 명령 실행 불가: 이륙(Takeoff) 명령의 응답을 기다리는 동안 짐벌(Gimbal) 제어 명령을 동시에 내릴 수 없다.
  2. 상태 오염(State Corruption): 명령 A의 ACK가 늦게 도착하여, 최근에 발송한 명령 B의 응답으로 잘못 오인되어 전역 상태가 해제되는 논리적 결함(Race Condition)이 발생한다.

2. VehicleCommandManager 내부 상태 머신(FSM) 격리 설계

이러한 문제를 원천 차단하기 위해, 최신 QGC 아키텍처는 개별 COMMAND_LONG / COMMAND_INT 트랜잭션을 독자적인 유한 상태 기계(FSM, Finite State Machine)로 추상화하여, VehicleCommandManager 내부에 격리(Isolation)시킨 채 병렬로 운용한다.

각 CommandTransaction 객체 내부는 다음과 같은 상태 전이 구조를 독자적으로 보유한다.

이 다이어그램이 시사하듯, 수십 개의 트랜잭션 객체들이 각자의 타이머 모니터 틱(Tick)과 카운터를 지니고 독자적으로 상태를 전이한다.

2.1 트랜잭션 격리(Isolation) 메커니즘

  1. 내부 변수의 은닉(Encapsulation): 트랜잭션의 재전송 횟수(Retry Count), 최초 발송 타임스탬프, 연관된 람다 콜백 등의 메타데이터는 오직 해당 CommandTransaction 객체 스코프 내부에만 존재한다.
  2. 독립적 타이머 클럭: VehicleCommandManager는 주기적인 시스템 타이머 콜백에서 자신의 해시 맵(Hash Map)에 등록된 개별 트랜잭션들의 Pending 상태를 스윕(Sweep)하며 점검한다. 트랜잭션 A가 타임아웃되어 재전송(Retry 상태 진입)되더라도, 트랜잭션 B의 WaitAck 시간 계측에는 그 어떠한 간섭도 주지 않는다.

3. C++ 객체 지향 설계를 곁들인 구체적 구현 사례

QGroundControl 프레임워크 로직을 유사하게 추상화하면 다음과 같은 클래스 아키텍처가 성립한다.

#include <functional>
#include <chrono>

enum class TransactionState { Init, Pending, Success, Failed, TimeoutAbort };

class CommandTransaction {
private:
    uint16_t _commandId;
    TransactionState _currentState;
    int _retryCount;
    const int _maxRetries = 3;
    uint64_t _lastSentTimeMs;
    std::function<void(bool)> _resultCallback;

public:
    CommandTransaction(uint16_t cmd, std::function<void(bool)> cb)
        : _commandId(cmd), _currentState(TransactionState::Init), _retryCount(0), _resultCallback(cb) {}

    // 격리된 상태 검사 및 재전송 로직
    void checkTimeoutAndRetry(uint64_t currentTimeMs, std::function<void()> resendFunc) {
        if (_currentState != TransactionState::Pending) return;

        if (currentTimeMs - _lastSentTimeMs > 1000 /* 1초 타임아웃 */) {
            if (_retryCount >= _maxRetries) {
                _currentState = TransactionState::TimeoutAbort;
                if (_resultCallback) _resultCallback(false);
            } else {
                _retryCount++;
                _lastSentTimeMs = currentTimeMs;
                resendFunc(); // 개별 트랜잭션의 독자적 재전송 실행
            }
        }
    }

    void handleAck(bool accepted) {
        _currentState = accepted ? TransactionState::Success : TransactionState::Failed;
        if (_resultCallback) _resultCallback(accepted);
    }
};

각 명령 요청마다 생성되는 CommandTransaction 객체 내부에 변수들이 프라이빗(Private)하게 은닉되어 있어, VehicleCommandManager는 단지 순회하며 checkTimeoutAndRetry만 호출해주면 수십 개의 다른 명령 트랜잭션들이 스스로 자신의 상태 생명주기를 완결짓는다.

4. Ardupilot 모델과의 구조적 차이점

Mission Planner와 MAVProxy 등 Ardupilot 주요 관제 소프트웨어 생태계는 종종 서브 시스템(예: Waypoint 매니저, Parameter 매니저) 관점에서의 상태 격리만 수행하며 일반 비행 명령어(Flight Command)는 전역 큐에 의존해 한 번에 하나씩 병목형 전송(Synchronous-like Blocked Queueing)을 취하는 경향을 보여왔다.

반대로 최신 QGroundControl(PX4-Autopilot 특화) 아키텍처에서의 VehicleCommandManager는, 그 어떤 명령어라도 유일한 트랜잭션 객체 단위로 미시 분할(Micro-Segmentation)하여 격리 운용한다. 이 설계 철학의 차이는 다중 에이전트 다기능 스웜 제어 시, 미션 플래너가 체감할 수 있는 큐 적체(Queue Congestion) 스파이크를, PX4 중심의 QGC 관제망에서는 마법처럼 해소해버리는 핵심적인 기술 격차를 낳는다.

5. 요약 및 결론

방대한 텔레메트리 데이터와 비동기 제어 명령이 교차하는 GCS 관제 파이프라인에서 ’전역 상태’를 이용한 제어 기법은 시스템 붕괴를 초래하는 안티패턴(Anti-Pattern)이다. 따라서 QGroundControl 내부의 VehicleCommandManager 클래스는 모든 MAVLink 2.0 커멘드를 고립된 형태의 개별 상태 머신(Isolated State Machine)으로 취급한다.

각 트랜잭션 캡슐 단위로 타임라인과 재전송 전략이 완전히 독립적으로 구동됨으로써, 병행 전송 중인 수많은 명령 스레드들이 서로의 시간축을 일절 간섭하지 않는 스레드-세이프(Thread-Safe)한 최강의 관제 신뢰성을 이룩할 수 있었다.
