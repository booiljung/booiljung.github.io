39.1.3.2. 분산 제어 시스템의 멱등성(Idempotence) 설계 원칙과 수신자 주도 중복 제거(Deduplication) 알고리즘

지상 관제 시스템(GCS)이 UDP 기반의 네트워크 환경에서 COMMAND_LONG 명령어의 신뢰성을 확보하는 가장 1차원적이고 확실한 전략은 ’재전송(Retransmission)’이다. GCS는 COMMAND_ACK 응답이 정해진 타임아웃 이내에 오지 않으면 패킷 유실을 가정하고 동일한 명령을 다시 전송하는 “적어도 한 번(At-Least-Once)” 전달 방식을 취한다. 그러나 이 전술은 본의 아니게 기체(PX4 수신단)에 동일한 명령이 중복하여 도달하는 문제(Duplicate Delivery)를 야기한다. 본 절에서는 이렇게 중복된 명령어가 시스템을 교란하는 것을 방지하기 위한 소프트웨어 공학의 두 가지 핵심 방어 원칙인 **멱등성(Idempotence)**과 **수신자 주도 중복 제거(Receiver-driven Deduplication)**에 대해 설명한다.

1. 멱등성(Idempotence) 설계 원칙

수학 및 컴퓨팅에서 **멱등성(Idempotence)**이란 어떤 연산을 여러 번 반복적으로 수행하더라도 단 한 번만 수행했을 때와 시스템의 최종 결과가 동일하게 유지되는 성질을 말한다. PX4-Autopilot의 커맨더(Commander) 상태 머신은 멱등성을 지니도록 철저하게 설계되어 있다.

1.1 드론 제어에서의 멱등성 위반 사례 vs 준수 사례

비행 제어 시스템의 함수가 멱등성을 위반하도록 짜여 있다면 치명적이다.

  • 멱등성 위반(Bad Case): 예를 들어 “고도를 10m 높이라“는 상대적(Relative) 명령을 내린 후 ACK가 오지 않아 3번 재전송했는데, 기체가 이를 모두 개별 명령으로 해석하면 고도를 30m나 높이는 사고가 발생한다.
  • 멱등성 준수(Good Case): GCS가 “목표 절대 고도를 50m로 설정하라(MAV_CMD_NAV_WAYPOINT)“고 지시하고, 이를 3번 받더라도 기체는 최종 조향 고도를 50m로 한 번만 세팅한다.

1.2 모드 변경 및 Arming 동작의 내부 논리

PX4 커맨더는 외부에서 시동(Arm) 명령(MAV_CMD_COMPONENT_ARM_DISARM, param1=1)이 들어올 때 이 멱등성을 충실히 이행한다. 기체가 이미 Arming되어 추진체가 회전 중일 때, 라디오 딜레이로 인해 뒤늦게 또 다른 Arm 명령이 도착했다고 하자. PX4는 메인 루프를 재설정하거나 모터 캘리브레이션을 처음부터 다시 시작하지 않는다. 내부 vehicle_status.arming_state 레지스터가 이미 ARMED임을 검증하고, 시스템 상태에 변경을 주지 않은 채 즉각 MAV_RESULT_ACCEPTED라는 긍정 응답(ACK)만을 MAVLink 레이어로 올려보낸다. 이를 통해 GCS 측의 빈번한 재전송 시도에도 오토파일럿의 논리적 일관성이 동요하지 않는다.

2. 수신자 주도 중복 제거 (Receiver-Driven Deduplication)

멱등성 아키텍처가 “같은 명령을 실수로 여러 번 실행해도 결과가 똑같게 만드는” 방패라면, **주도적 중복 제거(Deduplication)**는 애초에 “같은 명령이 여러 번 실행되는 것 자체를 차단하는” 필터 역할을 한다.
UDP 기반 통신에서는 패킷이 네트워크 상에서 복제되어 분신이 도착할 수 있으며, 이로 인해 GCS는 재전송을 하지 않았는데도 PX4는 두 개의 같은 명령을 수신할 수 있다. PX4-Autopilot은 이 중복 제거에 대한 짐을 GCS 송신자에게 떠넘기지 않고, 기체 내부(수신단)의 로직으로 해결한다.

2.1 중복 감지와 캐싱 메커니즘

PX4 내의 MAVLink 메시지 파서인 mavlink_receiver.cpp는 패킷을 uORB에 발행하기 전에 필터 처리를 진행한다. 명령어 식별은 주로 **송신 시스템 ID(System ID), 컴포넌트 ID(Component ID), 명령어 종류(Command ID), 그리고 프레임 시퀀스 번호(Sequence Number)**의 조합으로 구성된 해시(Hash) 성격의 키(Key)를 사용한다.

시스템은 가장 최근에 처리하여 승인(COMMAND_ACK)한 명령의 서명(Signature) 또는 시퀀스 윈도우(Sliding Window)를 메모리 캐시에 잠시 보관한다. 만약 방금 전과 완전히 동일한 발신자 서명과 시퀀스를 가진 식별자의 COMMAND_LONG이 밀려들어오면, 파서는 이를 “새로운 지시“로 해석하지 않고 “재전송된 과거 지시” 혹은 “패킷 복제” 현상으로 간주하여 vehicle_command를 통해 하위 로직(Commander)으로 내리는 것을 차단(Drop)한다.

2.2 중복 패킷에 대한 무성(Silent) 폐기 금지

중복 제거를 수행하면서 주의해야 할 가장 중요한 공학적 디테일은, 중복 패킷을 발견했다고 해서 아무런 대답 없이 무성 폐기(Silent Drop)해서는 안 된다는 것이다.
통신 지연으로 인해 첫 번째 명령에 대한 PX4의 COMMAND_ACK가 GCS에 도착하지 않아 GCS가 타임아웃을 겪고 두 번째 동일 명령을 보냈을 가능성이 크기 때문이다. 따라서 PX4 수신단은 해당 명령을 처리 파이프라인에서 폐기(Deduplicate)하되, GCS의 타임아웃 재전송 루프를 달래기 위해 **이전에 보냈던 바와 똑같은 COMMAND_ACK 결과 코드를 즉시 재-브로드캐스트(Re-ACK)**해 주어야 한다.

이처럼 PX4와 QGroundControl로 대변되는 엔터프라이즈 모빌리티 통신 스택은 단순히 네트워크 레이어(TCP/UDP)에 의존하는 대신, 멱등성이 확보된 상태 머신과 식별자에 기반한 영리한 중복 제거 로직을 통해 혹독한 가스관, 해저, 우주에 이르기까지 어떠한 통신 간섭 속에서도 우아한 회복 탄력성(Resilience)을 선보인다.
