39.4.4.1. 명령 수행 완료 시 vehicle_command_ack uORB 토픽의 발행 지연(Publishing Latency) 최소화 설계

1. 서론: “언제 대답할 것인가?” 응답 레이턴시의 임계성

지상 관제 시스템(GCS)과 비행 제어기(Flight Controller) 간의 제어 루프에서, 명령의 수신부터 응답(ACK)의 발송까지 걸리는 왕복 지연 시간(Round Trip Time, RTT)은 전체 시스템의 반응성(Responsiveness)을 결정짓는 핵심 지표이다.

만약 드론이 GCS의 “웨이포인트 비행 시작” 명령을 수신하여 내부적으로는 이미 이동을 시작했음에도 불구하고, 응답 패킷(COMMAND_ACK)의 생성이 지연되어 GCS가 500ms(밀리초) 타임아웃을 겪는다면, GCS는 명령이 실패한 것으로 오판(False Negative)하여 동일한 명령을 재전송(Retransmission)하게 된다. 이는 네트워크 대역폭의 낭비를 넘어 상태 동기화의 치명적 오류를 유발할 수 있다. 본 절에서는 PX4 아키텍처(펌웨어 v1.14 기준)가 Commander 로직 내부에서 명령어 처리 직후 vehicle_command_ack 토픽을 어떻게 ’지연율 0(Zero-Latency 정책)’에 가깝게 발행(Publish)해 내는지 그 구조적 설계를 분석한다.

2. 인라인(Inline) 응답 처리와 컨텍스트(Context) 유지

PX4의 Commander 모듈은 명령의 유효성 검사, 상태 전이(State Transition), 그리고 응답 토픽 발행이라는 세 가지 작업을 단일 호출 스택(Single Call Stack) 안에서 즉시 완결 짓는 ‘인라인 처리(Inline Processing)’ 패턴을 채택하고 있다.

  1. 상태 변경과 응답의 원자성(Atomicity):
    특정 커맨드(예: MAV_CMD_COMPONENT_ARM_DISARM)가 Commander의 평가 로직을 통과하여 기체의 시동 상태(arming_state) 변수를 변경하는 데 성공했다면, Commander는 해당 함수가 반환(Return)되기 직전에 하드코딩된 send_vehicle_command_ack() 헬퍼 함수를 동기식(Synchronous)으로 즉각 호출한다. 이는 상태 변화와 응답 생성을 논리적으로 분리할 수 없는 하나의 원자적 트랜잭션(Atomic Transaction)으로 묶어버린다.
  2. 지연 큐(Delayed Queue)의 배제:
    Ardupilot 등 일부 비동기 이벤트 시스템에서는 “처리 완료된 커맨드 목록“을 별도의 응답 대기열(Response Queue)에 적재해 두고, 후속 루프(Next Loop)나 별도의 백그라운드 스레드가 이를 수거하여 ACK를 일괄 송신하는 방식을 취하기도 한다. 그러나 PX4는 ACK 생성을 결코 미루지 않는다. 메모리 복사 오버헤드가 극히 미미한 uORB 아키텍처의 장점을 살려, Commander 스레드가 쥐고 있는 CPU 제어권(Time slice) 내에서 즉시 orb_publish를 때려버린다.
/* PX4 Commander.cpp 내부 로직 단순화 블록 */
bool Commander::execute_command(const vehicle_command_s &cmd)
{
    uint8_t result = vehicle_command_ack_s::VEHICLE_CMD_RESULT_UNSUPPORTED;

    switch (cmd.command) {
        case vehicle_command_s::VEHICLE_CMD_COMPONENT_ARM_DISARM:
            // 시동 조건 검증 및 기체 상태 변경 시도
            if (arm_disarm(cmd.param1)) {
                result = vehicle_command_ack_s::VEHICLE_CMD_RESULT_ACCEPTED;
            } else {
                result = vehicle_command_ack_s::VEHICLE_CMD_RESULT_DENIED;
            }
            break;
        // 다른 커맨드 처리 분기들...
    }

    // 커맨드 평가가 끝나자마자 (루프를 넘기지 않고) 즉시 ACK 토픽 발행
    send_vehicle_command_ack(cmd, result);
    return true; 
}

3. 비동기 발행(Asynchronous Publish)의 커널 레벨 마이크로 최적화

send_vehicle_command_ack()가 호출되어 vehicle_command_ack 토픽이 uORB 버스로 던져질 때, 발행 지연(Publishing Latency)을 극소화하기 위한 NuttX 운영체제 단의 보이지 않는 조력이 개입한다.

  • Non-blocking uORB Publish:
    uORB의 orb_publish() 커널 콜은 근본적으로 락-프리(Lock-free)에 가까운 초경량 메커니즘을 사용하거나, 잠금을 획득하더라도 매우 짧은 크리티컬 섹션(Critical Section)만을 갖도록 설계되어 있다. 즉, Commander가 ACK를 발행하는 순간 uORB 데몬이 멈춰 서서 데이터를 하드 디스크립터로 복사하는 무거운 과정을 거치지 않으며, 단지 메모리 포인터 갱신과 구독자(Subscriber)들을 깨우는 이벤트 시그널만을 던지고 즉시 반환(Return)된다.
  • 구독자(mavlink_messages)의 즉각적인 Wake-up:
    이 시그널은 송신을 담당하는 mavlink_messages 스레드에 연결된 poll() 디스크립터를 즉시 활성화시킨다. Commander가 ACK 발행을 마치고 다음 커맨드를 처리하거나 다시 수면(Sleep) 상태로 들어가는 찰나의 마이크로초(us) 단위 시간 안팎으로, 송신 스레드가 스케줄러에 의해 선점(Preempt)되어 칩셋의 물리 포트(Tx Register)에 데이터를 밀어 넣기 시작한다.

이처럼 PX4의 응답 메커니즘은 로직 수행 완료점과 응답 발행점 사이의 간격을 단일 C++ 함수의 실행 스코프(Scope) 이내로 압축함으로써, GCS 관점에서 볼 때 커맨드가 전파를 타고 드론에 도달하는 즉시 메아리(Echo)가 돌아오는 듯한 극한의 로우 레이턴시(Low Latency) 피드백 루프를 완성하게 된다.
