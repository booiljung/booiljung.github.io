39.3. COMMAND_ACK (#77) 기반 유한 상태 기계(FSM) 제어 및 응답 라우팅

1. 개요 및 통신 아키텍처

PX4-Autopilot의 지상 관제 시스템(GCS: Ground Control Station) 제어망 아키텍처에서 시스템의 무결성(Integrity)과 결정론적 제어 수렴성을 보장하는 핵심 메커니즘은 MAVLink 프로토콜의 COMMAND_ACK(#77) 메시지를 기반으로 한 유한 상태 기계(FSM: Finite State Machine)의 상호 동기화이다. 본 장에서는 PX4(펌웨어 v1.14 기준)와 QGroundControl(v4.3 기준) 간의 비동기 제어 명령 릴레이 및 이에 대한 엄격한 수락/거절 상태 검증 파이프라인(Accept/Reject Validation Pipeline)을 분석한다.

MAVLink 기반 네트워크 환경(UART, UDP, TCP 라우팅)에서 GCS가 비행체로 COMMAND_LONG(#76) 또는 COMMAND_INT(#75) 패킷을 송신하면, 비행체의 커맨더(Commander) 디먼 프로세스는 이를 즉각 실행하지 않고 현재 비행체의 동역학적 상태(Pre-condition)와 하드웨어의 모드 권한(Mode Authorization)을 검사한다. 평가가 끝나면 비행체는 COMMAND_ACK 메시지를 발송하여 GCS로 수리 여부를 라우팅한다. 이를 기반으로 PX4와 GCS는 양방향 통신에서의 분산형(Distributed) 상태 동기화를 보장한다.

2. MAVLink COMMAND_ACK 기반 응답 라우팅 작동 원리

비행 제어 컴퓨터와 GCS 간 명령어 처리 방식에서, 유실 가능한 무선 데이터 링크를 통한 비동기 명령 응답 방식은 필수적으로 상태 지연(State Latency) 보상 체계를 요구한다. COMMAND_ACK 응답은 단순히 수신 여부를 확인하는 기초적 핸드셰이크(Handshake)를 넘어, 분산 FSM 동기화를 위한 6단계 상태 결과 피드백을 내포한다.

2.1 COMMAND_ACK 핵심 페이로드 구성

COMMAND_ACK 패킷은 다음과 같은 데이터 필드 구조를 채택한다.

  • command: 응답 대상이 되는 원래 명령어의 고유 ID(MAV_CMD 열거형).
  • result: 처리 결과를 명시하는 상태 코드(MAV_RESULT 열거형). ACCEPTED, DENIED 등 6가지 상태 반환.
  • progress: (0-100%) 비동기 장기 실행 명령어의 진행도를 GCS의 UI로 보간할 수 있도록 하는 스칼라(Scalar) 필드.
  • result_param2: 센서 캘리브레이션 실패 이유 등 세부적인 디버그/거부 사유를 다중화하여 담는 매개변수 필드.
  • target_system / target_component: 이중화(Redundancy) 라우팅 네트워크 내에서 특정 GCS(호스트) 클라이언트의 정확한 컴포넌트(예: 메인 화면의 카메라 위젯 등)로 응답 패킷을 라우팅하기 위한 주소 체계.

    else 상태 승인 (수락)
        FSM->>SubModule: 제어 권한 위임 및 루프 시작
        FSM-->>MAVLink: vehicle_command_ack (MAV_RESULT_ACCEPTED)
        MAVLink-->>GCS: COMMAND_ACK 전송 (UI 업데이트)
    end

3. PX4-Autopilot과 Ardupilot의 응답 파이프라인 구조적 차이

명령어의 수락 가능성(Acceptability) 판별 프로세스에 있어서 PX4와 Ardupilot은 근본적인 아키텍처 철학의 차이를 보인다.

  • PX4-Autopilot: 다중 스레드 기반의 엄격한 중앙 상태 허브(Commander 디먼) 모델을 지니고 있다. GCS에서 들어온 명령어는 Commander가 최상위 권한자로서 FSM 조건을 엄격하게 판별한 후, 조건이 1%라도 맞지 않으면 즉각 MAV_RESULT_DENIED를 발생시키고 실행을 파기하는 방어적 프로그래밍(Defensive Programming) 기조를 따른다. 이 방식을 통해 예상치 못한 모드 간 교착 상태(Deadlock)를 방지한다.
  • Ardupilot: 상대적으로 관대한 예외 보상 체계를 채택하고 있다. 명령어가 들어왔을 때 해당 모드의 권한이 없더라도, 조건을 충족시킬 수 있는 자율 기능 개입(Autonomous Compensation) 로직이 분산된 각 모드 스크립트(Mode Object) 내에 배치되어 일부 논리적 공백을 프레임워크가 자가 보정하려는 경향이 있다.

4. ROS2 연동 및 MAVSDK를 통한 프로그래매틱(Programmatic) FSM 제어

오늘날 자율 에이전트(Autonomous Agent) 기반 군집 비행(Swarm) 환경에서는 QGroundControl을 넘어 ROS2 아키텍처나 MAVSDK를 이용한 자동화된 제어가 주를 이룬다. Micro XRCE-DDS 에이전트를 통해 ROS2 노드(Node)에서 직접 오프보드(Offboard) 제어 명령을 하달할 경우에도 시스템 내부적으로는 동일한 FSM 및 COMMAND_ACK 라우팅 경로를 거친다.

ROS2 애플리케이션 프레임워크 레벨에서는 vehicle_command_ack 토픽을 지속적으로 구독(Subscribe)해야만 모드 변경이나 액추에이터 활성 명령이 실제 물리적 제어기 메모리 공간(Controller Memory Space)에 할당되었는지 추적할 수 있다. 만약 COMMAND_ACK 라우팅 루프를 통한 피드백을 관찰하지 않고 명령을 강제로 밀어넣는다면(Fire and Forget 개방 루프 전송), 센서 포화(Saturation)나 통신 유실(Data Link Loss) 하에서 제어기 상태가 동기화되지 못해 심각한 이탈 현상이 발생할 수 있다. 이러한 관제 기술은 단일 논문 “Fault-Tolerant Control and FSM Synchronization in Unmanned Aerial Systems” 등 학술 논문에서 필수적인 분산 토폴로지 요구사항으로 거론된 바 있다.
