39.4.2.2. 발신자 정보(Source System/Component)의 메모리 캐싱 및 동적 역방향 라우팅 테이블 생성

1. 서론: 응답(ACK)의 방향타, “누구에게 대답할 것인가?”

명령 수신 파이프라인에서 직렬화된 패킷을 기체 내부의 vehicle_command uORB 토픽으로 매핑(Mapping)하는 데 성공했다면, 비행 제어기(PX4 Core)는 이제 해당 명령을 수행할 준비를 마친 것이다. 그러나 비동기 통신망에서 ’성공적인 명령 수행’은 절반의 완성에 불과하다. 시스템은 명령 수행의 결과(예: MAV_RESULT_ACCEPTED 또는 MAV_RESULT_DENIED)를 담은 COMMAND_ACK 패킷을 원래 명령을 내렸던 **바로 그 발신자(Originator)**에게 정확히 되돌아 보내야만 비로소 하나의 트랜잭션(Transaction)이 종결된다.

현대의 드론 운용은 단일 GCS 제어 수준을 넘어섰다. 태블릿의 QGroundControl(System ID 255), 메인보드의 컴패니언 컴퓨터(System ID 1, Component ID 191), 그리고 군집 비행 통제 서버 등 다수의 마스터(Master) 노드가 하나의 기체에 동시다발적으로 MAVLink 커맨드를 던지는 멀티 에이전트 환경이다. 본 절에서는 PX4 아키텍처(펌웨어 v1.14 기준)가 어떻게 수신 패킷의 발신자 메타데이터를 메모리에 캐싱(Caching)하고, 이를 바탕으로 정확한 응답 배송지(Reverse Routing Table)를 동적으로 확보해 내는지 분석한다.

2. MAVLink 헤더 메타데이터와 uORB 토픽의 동적 메멘토(Memento) 패턴

물리 계층(e.g., UART)에서 MAVLink 2.0 패킷이 들어왔을 때, 해당 패킷의 헤더(Header)에는 페이로드 내용물과는 별개로 구조적인 발신자 증명(Proof of Origin) 정보가 각인되어 있다. mavlink_receiver는 역직렬화(Deserialization) 과정 중 이 메타데이터를 결코 버리지 않고 uORB 토픽의 전용 필드로 복제하여 메모리에 남겨둔다.

  1. SYSID 및 COMPID의 추출:
    MAVLink 패킷 프레이밍이 끝나면 헤더에 위치한 sysid(Source System ID)와 compid(Source Component ID)를 추출한다. 예를 들어 QGroundControl에서 보낸 명령이라면 통상적으로 sysid: 255, compid: 190 이 발견된다.
  2. uORB 토픽 내 메멘토 저장 (Memento Caching):
    이 값은 앞서 살펴본 동형 매핑을 통해 vehicle_command_s 구조체의 source_system 및 source_component 멤버 변수에 하드카피(Hard-copy)된다. 명령 본문(command, param1~7)과 더불어 자신이 **’어디서부터 왔는지’에 대한 기억(메멘토, Memento)**을 스스로 캡슐화한 런타임 객체가 탄생하는 것이다.
  3. 타겟 식별 (Target Identification):
    MAVLink의 범용 명세(Global Specification)에는 브로드캐스트(Broadcast, 타겟 ID가 0) 개념이 존재한다. mavlink_receiver는 이 타겟 시스템/컴포넌트 ID가 아예 브로드캐스트(0)이거나, 자기 자신(PX4의 SYSID)과 일치할 때만 이를 유효한 수신으로 간주하여 로컬 캐싱을 마친 후 uORB 버스에 퍼블리시한다.

3. 역방향 라우팅을 위한 Source-to-Target 스와핑(Swapping) 아키텍처

이 메타데이터 캐싱의 궁극적인 목적은 결국 응답 전송 시 목적지 주소를 찾아내는 알고리즘 비용을 최소화하기 위함이다.

  • 명령 수행 완료 시점의 역-매핑 (Reverse Mapping):
    Commander 디먼이 험난한 안전망 체크를 마치고 시동(Arming)에 성공했다고 가정해 보자. Commander는 내부적으로 vehicle_command_ack 토픽을 새로 선언한다. 이때 ACK 결과(MAV_RESULT)를 채우는 것과 동시에, 자신이 방금 전 받았던 vehicle_command 안에 고이 모셔져 있던 source_system과 source_component 값을, 새로 만들 ACK 토픽의 target_system과 target_component로 X자 스와핑(Swapping)하여 집어넣는다.
  • 다중 연결 MAVLink 인스턴스의 하드웨어 포트 라우팅:
    이렇게 타겟 주소가 기입된 ACK 토픽이 uORB 버스에 던져지면, 이번에는 송신을 담당하는 mavlink_messages 디먼이 이를 낚아챈다. PX4에는 종종 텔레메트리 포트 1, 텔레메트리 포트 2, USB 등 다수의 MAVLink 인스턴스(Instances)가 동시에 실행 중이다. 송신 디먼은 스스로 관리하고 있는 동적 라우팅 테이블(해당 컴포넌트 ID가 어느 시리얼 물리 포트와 묶여 있는지에 대한 내부 캐시)을 조회하거나, 단순히 연결된 모든 인터페이스의 MAVLink 채널에 조건부 송신을 때려버림으로써(가장 보수적인 필터링 채택), 정확히 명령을 내렸던 GCS의 안테나로 ACK 패킷을 정확히 유도해 낸다.

이는 Ardupilot이 특정 GCS 채널(Chan)을 전역 변수나 정적 배열 체계(Static Array) 내에 묶어놓고 송신 포트를 결정하는 하향식(Top-Down) 라우팅 방식과 궤를 달리한다. PX4는 상태 데이터 객체(uORB 토픽) 자체가 스스로의 발신자 캐싱 데이터를 품고 런타임을 떠도는 상향식(Bottom-Up) 데이터 지향(Data-Driven) 설계를 채택함으로써, 외부 노드가 몇 대이든 유연하게 대처하는 동적 역방향 라우팅 아키텍처를 완성하였다.
