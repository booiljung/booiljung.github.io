39.2. MAVLink 2.0 명령어 패킷의 페이로드 메모리 아키텍처 및 직렬화

초창기 무인항공기 통신 프로토콜들은 각 제조사마다 독자적인 시리얼 통신 규격을 사용하여 상호 호환성이 결여되어 있었다. 이를 극복하고 범용적인 오픈소스 드론 생태계를 구축하는 데 결정적인 역할을 한 것이 바로 MAVLink(Micro Air Vehicle Link) 프로토콜이다. GCS(QGroundControl)와 비행 제어기(PX4-Autopilot) 간의 제어 명령을 담당하는 핵심 메시지인 COMMAND_LONG (설계상 제한이 많은 구형 패킷) 및 COMMAND_INT 등은 한정된 무선 대역폭 속에서도 다양한 형태의 데이터를 누수 없이 효율적으로 전달해야 하는 임무를 띤다. 본 장에서는 MAVLink 프로토콜 최신 버전인 2.0 규격 하에서 명령어 패킷의 메모리 페이로드 구조가 어떻게 설계되어 있으며, 서로 다른 컴퓨터 아키텍처 간의 데이터 직렬화(Serialization)가 어떻게 이루어지는지 심도 있게 해부한다.

1. MAVLink 2.0 프로토콜 개요와 메시지 구조

MAVLink 프로토콜은 매우 경량화된(Lightweight) 바이너리(Binary) 직렬화 프로토콜이다. 2017년부터 도입된 MAVLink 2.0 규격은 기존 1.0 규격을 확장하여, 24비트의 Message ID(수십만 개의 메시지 정의 가능)와 보안 통신을 위한 메시지 서명(Message Signing) 기능을 추가함으로써 엔터프라이즈 및 군용 무인기 환경의 요구사항을 충족하게 되었다.

명령어 패킷을 포함한 모든 MAVLink 2.0 패킷은 다음과 같은 프레임(Frame) 구조를 가진다.

2. 제어 명령어(Command) 페이로드 아키텍처

위 프레임 구조에서 시스템의 핵심 제어 동작을 지시하는 내용은 바이트 스트림 구역인 통신 페이로드(DATA) 안에 구체적인 C 언어 구조체(Struct) 형태로 패킹(Packing)된다. PX4-Autopilot에서 가장 대표적으로 처리하는 두 가지 명령어 구조체는 다음과 같다.

2.1 COMMAND_LONG (Message ID: #76)

초기 MAVLink 1.0 시절부터 가장 범용적으로 쓰인 메시지이다. “모든 것을 이 메시지 하나로 처리할 수 있도록 하자“는 설계 철학 하에, 무려 7개의 float 형 파라미터를 제공한다. (전체 페이로드 길이: 33 Bytes)

  • param1 ~ param7 (float, 4비트씩 총 28 Bytes): 명령어 성격에 따라 다목적으로 쓰이는 인자. WGS84 좌표계 데이터도 여기에 실린다.
  • command (uint16_t, 2 Bytes): MAV_CMD 열거형으로 정의된 명령어 식별 번호 (예: 400번=Arm/Disarm).
  • target_system (uint8_t, 1 Byte): 명령을 수행해야 할 시스템의 ID.
  • target_component (uint8_t, 1 Byte): 명령을 수행해야 할 시스템 내 컴포넌트 단위의 ID.
  • confirmation (uint8_t, 1 Byte): 재전송 시퀀스 횟수 식별 용도.

2.2 COMMAND_INT (Message ID: #73)

COMMAND_LONG이 지닌 부동소수점 데이터 표현의 치명적 한계(단정밀도 float가 지리 좌표계를 담을 때 나타나는 공간 오차)를 극복하기 위해 MAVLink 2.0 시대에 표준으로 승격된 메시지이다. 공간 좌표가 포함된 임무 설정에 주로 쓰인다. (전체 페이로드 길이: 35 Bytes)

  • param1 ~ param4 (float, 16 Bytes): 부동소수점이 필요한 보조 인자.
  • x (int32_t, 4 Bytes): 위도(Latitude). 보통 원본 값에 10^7을 곱해 부호 있는 정수로 표현한다.
  • y (int32_t, 4 Bytes): 경도(Longitude). 10^7 스케일링.
  • z (float, 4 Bytes): 고도 데이터.
  • command, target_system, target_component 등 기타 구조는 COMMAND_LONG과 동일하다.

3. 메모리 직렬화(Serialization)와 통신 엔디안(Endianness)

GCS(예: QGroundControl)가 작동하는 환경은 주로 x86_64 기반의 데스크톱(Windows/Linux)이거나 ARM 아키텍처 기반의 스마트폰(iOS/Android)이다. 그리고 데이터를 수신하는 PX4-Autopilot 장비(Pixhawk 등)는 ARM Cortex-M 시리즈 기반의 마이크로컨트롤러(MCU)이다.
이러한 이종(Heterogeneous) 플랫폼 간에 바이너리 데이터(특히 4바이트 크기의 float나 int32_t)를 바로 메모리 복사로 전송하면 엔디안(Endianness) 문제와 메모리 정렬(Alignment) 문제로 인해 데이터가 깨지게 된다.

3.1 Little-Endian 규격 강제

MAVLink 프로토콜 명세는 네트워크 통신망에서 데이터를 보낼 때 TCP/IP 표준 네크워크 바이트 오더인 Big-Endian 대신 Little-Endian을 표준 직렬화 방식으로 채택했다. 이는 현대의 대부분 기기(x86 프로세서 및 디폴트 ARM 마이컴)가 Little-Endian 메모리 구조를 지배적으로 사용하기 때문이다. Little-Endian을 사용하면 데이터를 보낼 때나 받을 때 별도의 바이트 스왑(Byte-swapping, 예: htons(), ntohs()) 오버헤드 연산을 생략할 수 있어 임베디드 소형 CPU(픽스호크)의 파싱(Parsing) 연산 부하를 크게 줄일 수 있다.

3.2 C 구조체 페이로드 패킹의 공학

C/C++ 언어에서 구조체 컴파일러는 메모리 접근 효율성을 위해 변수들 사이에 패딩(Padding, 의미 없는 공백 바이트)을 삽입하여 메모리를 정렬(Alignment)하려는 본능이 있다. 그런데 이 구조체를 그대로 통신 패킷에 복사하면 전송 대역폭이 낭비될 뿐 아니라 양단 플랫폼의 컴파일러가 패딩을 다르게 처리할 경우 페이로드가 깨진다.

이를 방지하기 위해 PX4의 MAVLink 파서는 #pragma pack(1) 또는 __attribute__((packed)) 와 같은 컴파일러 지시자를 사용하여 MAVLink 메시지 구조체의 모든 멤버를 빈틈없이 1바이트 단위로 압축하여 직렬화한다.
또한 MAVLink 페이로드 정렬 알고리즘은 패딩을 본질적으로 최소화하기 위해 메시지 정의(XML 파일)에서 생성된 구조체의 필드 순서를 설계자의 정의 순서와 무관하게 ‘데이터 타입 크기가 큰 것부터 작은 것 순으로’ (예: uint64_t -> uint32_t -> uint16_t -> uint8_t) 메모리에 재배치하여 바이트 시퀀스를 직렬화한다.

결론적으로 GCS와 PX4가 주고받는 COMMAND_LONG 등의 제어 명령어 페이로드는 RF 통신 환경의 제약을 극복하기 위해 바이트 스왑 생략(Little-Endian), 패딩 제거(Packed Struct), 메시지 데이터 크기 순 정렬이라는 고도화된 메모리 아키텍처 공학의 산물이다. 이어지는 하위 절들에서는 이 페이로드 구조체 구석구석을 채우고 있는 바이트 데이터들이 어떤 한계와 연산 모델을 지니는지 개별적으로 상세히 해부한다.
