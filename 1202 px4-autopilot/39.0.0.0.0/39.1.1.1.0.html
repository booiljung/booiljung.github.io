39.1.1.1. 사용자 데이터그램 프로토콜(UDP)의 비연결성 및 상태 비저장(Stateless) 아키텍처 검토

지상 관제 시스템(GCS)과 무인항공기(PX4-Autopilot) 간의 통신은 단순한 시리얼(UART) 기반의 RF 라디오 계층을 넘어, 현대에는 IP 기반의 네트워크 망(Wi-Fi, 셀룰러 LTE/5G, 위성 통신 등)을 광범위하게 활용하고 있다. 이러한 네트워크 상에서 MAVLink 메시지를 실어 나르기 위한 전송 계층(Transport Layer) 프로토콜로 가장 널리 채택되는 것은 **사용자 데이터그램 프로토콜(UDP, User Datagram Protocol)**이다. 본 절에서는 UDP의 핵심 특성인 비연결성(Connectionless)과 상태 비저장(Stateless) 아키텍처가 무인기 제어망에서 왜 필수적인지, 그리고 이를 기반으로 구축된 MAVLink 통신 시스템의 공학적 장단점을 심층적으로 검토한다.

1. 전송 제어 프로토콜(TCP)의 부적합성과 UDP의 대두

일반적인 IT 웹 서비스나 파일 전송의 경우 데이터의 무결성(Integrity)과 순서를 완벽히 보장하는 전송 제어 프로토콜(TCP)이 필수적이다. 그러나 로보틱스 및 무인기 실시간 제어 분야에서는 TCP의 엄격한 연결 지향적 특성이 오히려 치명적인 시스템 리스크를 유발한다.

  1. 3-Way Handshake로 인한 초기 지연: TCP는 통신을 시작하기 전 양방향 연결을 수립하기 위한 3-Way Handshake 과정을 거친다. 비행 도중 네트워크 링크가 끊어졌다가 복구(Re-connection)되는 상황에서 이러한 핸드셰이크는 수십~수백 밀리초(ms)의 귀중한 제어 시간을 낭비하게 만든다.
  2. Head-of-Line (HoL) Blocking: 무선 링크의 특성상 패킷 유실은 빈번하게 발생한다. TCP는 패킷 N이 유실되면, 이후에 도착한 더 신선한 데이터 패킷인 N+1, N+2를 애플리케이션(MAVLink Parser)에 전달하지 않고 N번 패킷이 재전송될 때까지 큐(Queue)에 대기시킨다. 고속 비행 중인 무인기에게 1초 전의 유실된 자세 데이터를 받기 위해 현재의 긴급한 회피 명령이나 자세 제어 명령이 블로킹되는 현상은 결코 수용될 수 없다.

이러한 이유로 PX4와 QGroundControl(QGC)은 기본 네트워크 통신 규격으로 UDP를 압도적으로 선호하며, UDP의 오버헤드 없는 단순한 메커니즘을 적극 활용한다.

2. UDP의 비연결성과 상태 비저장(Stateless) 아키텍처 특성

UDP는 인터넷 프로토콜(IP)의 단순한 확장으로, 송신자는 수신자의 상태(수신 가능 여부 등)를 묻지 않고 단순히 데이터그램(Datagram) 포맷으로 패킷을 네트워크 버퍼에 밀어 넣는다(보낸다).

2.1 비연결성(Connectionless)의 이점

QGroundControl 측에서는 기체의 IP 주소와 포트 번호(기본적으로 PX4는 14550 포트를 주로 사용)만 알면, 즉시 MAVLink 메시지를 캡슐화한 UDP 패킷을 전송할 수 있다. 연결 유지를 위한 하트비트(Heartbeat)나 세션(Session) 관리 오버헤드가 네트워크 계층에 존재하지 않으므로 데이터 처리 지연시간(Latency)을 최소화할 수 있다. 특히 LTE나 5G를 통해 셀 타워(Cell Tower)를 빈번하게 변경(Handover)해야 하는 BVLOS(비가시권) 비행에서, IP가 변동되는 상황에 보다 빠르고 유연하게 대처할 수 있는 기반을 제공한다.

2.2 상태 비저장(Stateless)의 한계와 책임 전가

상태 비저장이란 전송 계층이 통신 세션의 상태(State)를 기억하지 않는다는 뜻이다. 즉, 내가 보낸 COMMAND_LONG (이륙, 암(Arming) 수행 등) 메시지가 목적지에 도달했는지, 유실되었는지 네트워크(UDP) 계층은 전혀 알지 못하며 기록도 남기지 않는다.
이는 필연적으로 신뢰성 관리의 책임을 상위 계층인 **애플리케이션 계층(MAVLink 프로토콜)**으로 전가시킨다. PX4-Autopilot 및 GCS의 개발자는 UDP의 Stateless 특성을 극복하기 위해 MAVLink 명세상에 다음과 같은 방어 메커니즘을 직접 구현해야만 했다.

  • 각 프레임의 순서를 확인하는 Sequence Number (0~255)
  • 명령어 승인 절차인 COMMAND_ACK 피드백 루프
  • GCS 소프트웨어(QGroundControl) 내부의 타이머 기반 재전송(Retransmission) 큐

    MAC->>TCP_Stack: Packet N (재수신 성공)
    TCP_Stack->>MAVLink: N, N+1, N+2 일괄 전달 (Timing 손실)

    Note over MAC, MAVLink: === UDP 기반 통신 시나리오 (GCS - PX4) ===
    MAC--xUDP_Stack: Datagram N (유실)
    MAC->>UDP_Stack: Datagram N+1 도착
    UDP_Stack->>MAVLink: Datagram N+1 즉시 전달
    MAC->>UDP_Stack: Datagram N+2 (긴급 회피 명령) 도착
    UDP_Stack->>MAVLink: Datagram N+2 즉시 전달
    Note right of MAVLink: MAVLink 파서가 Sequence 번호를 보고<br>N이 유실됨을 인지하나, 긴급 상황 제어는 즉시 실행됨.

결론적으로 사용자 데이터그램 프로토콜(UDP)의 비연결성 및 상태 비저장 아키텍처는 그 자체로 전송의 신뢰성을 담보할 수는 없으나, 드론과 같은 고도 동적 시스템(Highly Dynamic System)에게 필수적인 최저 지연 요건을 충족시킨다. 유실된 커맨드 데이터는 궁극적으로 PX4와 GCS 사이에 구축된 MAVLink의 ACK 메커니즘 등 애플리케이션 레벨의 로직이 해결하므로, 관제 시스템을 설계할 때 이러한 계층별 역할을 명확히 인지하고 아키텍처를 수립해야 한다.
