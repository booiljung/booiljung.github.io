39.4.4. 응답(ACK) 메시지의 uORB 발행(Publishing) 및 MAVLink 송신 변환

1. 서론: 피드백 루프(Feedback Loop)의 마침표

명령어(Command) 처리 아키텍처에서 가장 중요하면서도 간과하기 쉬운 단계는 바로 응답(Acknowledge, ACK)의 생성과 반환이다. 지상 관제 시스템(GCS)은 드론을 향해 “비행 모드를 변경하라“고 명령한 뒤, 기체로부터 MAV_RESULT_ACCEPTED라는 명시적인 ACK 메시지가 돌아올 때까지 해당 상태 전환을 사용자(조종자)의 UI에 반영하지 않고 대기(Blocked)하거나, 타임아웃(Timeout) 발생 시 재전송(Retransmission)을 시도한다.

만약 이 응답 파이프라인 구간에서 단락(Short circuit)이 발생한다면, 기체는 실제로는 명령을 온전히 수행했음에도 불구하고 GCS 화면에는 “명령 거부됨” 혹은 “응답 없음“으로 표출되는 치명적인 ‘기체-GCS 간 인지 부조화(Cognitive Dissonance)’ 사태를 초래한다. 본 장에서는 PX4(v1.14 기준) 내부에서 Commander가 생성한 응답 결과가 어떻게 uORB 토픽으로 변환되고 다시 외부 세계의 MAVLink 패킷으로 직렬화되어 날아가는지, 그 하행선(Downlink) 파이프라인 메커니즘을 추적한다.

2. 런타임 역방향 전파: vehicle_command_ack의 탄생과 발행

Commander 스레드가 상태 벡터 기반의 논리곱 검증을 마치고 모드의 전환이나 액추에이터 활성화를 완료(또는 거부)하면, 그 즉시 내장된 응답 생성 로직 콜백을 실행한다.

  1. 내부 응답 객체(vehicle_command_ack_s)의 인스턴스화:
    가장 먼저 vehicle_command_ack라는 명칭의 uORB 구조체를 메모리에 C-Struct 형태로 0으로 초기화(Zero-initialization)하여 할당한다.
  2. 필수 필드의 주입 (Injection of Mandatory Fields):
  • command: 자신이 어떤 커맨드(예: MAV_CMD_DO_REBOOT)에 대해 대답하는 것인지 원본 커맨드 ID를 복사한다.
  • result: 실행 결과 코드를 주입한다. 성공(ACCEPTED), 일시적 거부(TEMPORARILY_REJECTED), 또는 권한 없음(DENIED) 등의 Enum 값이 기록된다.
  • target_system & target_component: 앞서 수신 역방향 라우팅을 위해 임시 저장해 두었던 발신자(GCS)의 정보(Source System/Component)를 목적지(Target) 필드로 스와핑(Swapping)하여 채워 넣는다.
  1. uORB 버스로의 발행 (Publishing):
    완성된 ACK 토픽 구조체는 orb_publish() 커널 API를 통해 기체 내부 시스템 버스에 뿌려진다. 이로써 Commander의 역할은 완전히 종료되며 다음 이벤트 수신 대기(Poll) 상태로 넘어간다.

3. 송신 브릿지: mavlink_messages 디먼의 폴링과 직렬화(Serialization)

Commander가 버스에 던져 놓은 vehicle_command_ack 구조체를 주워 담아 전파(Radio Wave)로 쏘아내는 역할은 mavlink_messages (또는 mavlink_main의 송신 스레드 부분) 디먼이 전담한다.

  • 비동기 픽업 (Asynchronous Pickup):
    송신 모듈은 기체의 자세 정보(vehicle_attitude) 등 수십 개의 토픽들과 함께 vehicle_command_ack 토픽을 orb_subscribe 해둔 상태이다. ACK 토픽이 갱신되었다는 이벤트 시그널을 받으면, 큐에서 데이터를 빼내어 옵니다.
  • Isomorphic Reverse Mapping (동형 역매핑):
    수신 파이프라인에서 겪었던 과정의 정확한 역방향이 수행된다. 로컬 uORB 구조체의 command, result, target_system, target_component 등의 멤버 변수 값들이 mavlink_command_ack_t라는 MAVLink C-라이브러리 구조체로 1:1 대입(Assignment)된다.
  • 직렬화 및 CRC 패키징 (Serialization & Framing):
    채워진 mavlink_command_ack_t 구조체를 mavlink_msg_command_ack_send_struct()와 같은 패키징 함수에 통과시킨다. 이 함수는 메모리 상의 구조체를 바이트 배열로 납작하게 누르고(Serialization), 앞에는 매직 번호(0xFD)와 헤더를 달며, 뒤에는 X.25 다항식으로 계산한 CRC-16(Extra Seed 포함) 무결성 체크섬 바이트를 이어 붙여 완벽한 MAVLink 2.0 프레임을 주조해 낸다.
  • DMA 전송 요청 (DMA Write Request):
    최종적으로 완성된 바이트 스트림 버퍼는 POSIX write() 시스템 콜을 통해 UART나 UDP 파일 디스크립터로 넘겨지며, NuttX RTOS의 DMA 하드웨어 컨트롤러는 CPU를 우회하여 이를 송신 안테나로 고속 배출한다.

이와 같은 퍼블리시-서브스크라이브 기반의 응답 라우팅 구조는 “커맨드의 실행자“와 “응답 패킷의 전송자” 사이를 완전히 디커플링(Decoupling)시켜 놓았다. 덕분에 다중 텔레메트리 포트가 연결되어 있거나 ROS2 DDS 네트워크가 병렬로 돌아가는 복잡한 기체 환경에서도, 시스템 락(Lock)이나 교착 상태(Deadlock) 없이 매끄럽게 발신자에게 응답이 배달되는 튼튼한 하행선 파이프라인이 유지된다.
