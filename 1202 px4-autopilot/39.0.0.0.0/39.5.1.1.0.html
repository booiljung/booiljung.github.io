39.5.1.1. C++ Qt 프레임워크 기반의 스레드 풀(Thread Pool) 및 QEventLoop 논블로킹(Non-blocking) I/O 처리

PX4-Autopilot 기반의 무인기 시스템을 지상에서 완벽하게 제어하기 위한 QGroundControl(QGC) 백엔드는 단일 프로세스 내에서 수많은 커넥션과 비동기 연산을 지연 없이 처리해야 한다. 이를 달성하기 위해 코어 아키텍처는 C++의 원시(Native) 스레드 인터페이스를 직접 제어하기보다는, Qt 프레임워크가 제공하는 추상화된 고성능 동시성(Concurrency) 제어 기법인 스레드 풀(Thread Pool)과 QEventLoop 기반의 논블로킹(Non-blocking) I/O 모델을 적극적으로 도입하고 있다.

1. Qt의 스레드 풀(QThreadPool) 기반 자원 풀링(Pooling)

지상 관제 시스템이 처리해야 하는 연산 중에는 수 밀리초 내에 즉각 완료되어야 하는 작업(예: 하드웨어 인터럽트 수신)도 있지만, 대용량 임무(Mission) 파일 파싱, 다중 기체에 연결된 플러그인 초기화, 파일 I/O 작업 등 상대적으로 연산 시간이 긴 블로킹 작업도 혼재되어 있다.

이때 모든 단일 작업마다 새로운 OS 수준의 스레드를 생성하고 소멸시키는 것은 컨텍스트 스위칭(Context Switching) 및 커널 오버헤드가 극심하기 때문에, QGC는 QThreadPool 아키텍처를 활용한다.

1.1 QRunnable을 통한 일회성 연산 위임

복잡한 연산 로직은 상태를 유지할 필요가 없는 경우 QRunnable 인터페이스를 상속하여 객체화된다. Vehicle 매니저 스레드는 이 QRunnable 객체를 QThreadPool::globalInstance()->start()를 통하여 스레드 풀에 던져버리고(Fire-and-forget), 즉각 본연의 상태 관리 이벤트 루프로 복귀한다.

  • 장점: 시스템의 가용 코어 수(Ideal Thread Count)에 맞춘 최대 병렬 처리 효율을 보장하며, 비동기 파라미터 업로드/다운로드에서 GCS 인터페이스의 병목 부하를 극도로 완화할 수 있다.
  • 적용 사례: PX4 기체의 대용량 ULog 파일 로그 데이터를 GCS 측 스토리지로 다운로드하여 병렬 직렬화(Serialization)를 수행하는 워커 스레드 등에서 활용된다.

2. QEventLoop와 논블로킹(Non-blocking) I/O 처리의 이론과 실제

컴퓨터 과학의 네트워크 프로그래밍 하에서 일반적인 블로킹 I/O 함수(recv(), read() 등)는 버퍼에 데이터가 도착할 때까지 스레드 자체를 정지점(Suspend)에 둔다. 그러나 QGC의 통신 스레드 블록(LinkManager 하위)은 결코 정지되어서는 안 된다. 따라서 Qt가 제공하는 백그라운드 QEventLoop 객체를 결합하여 완벽한 비동기 I/O를 실현한다.

2.1 QEventLoop의 동작 원리

QEventLoop는 내부적으로 OS 커널의 비동기 I/O 채널 조사 기능(리눅스의 epoll, 윈도우의 IOCP 등)을 래핑(Wrapping)한다. 통신 스레드 내의 무한 루프는 이벤트를 대기하다가 QUdpSocket 또는 QSerialPort에 데이터 도착(readyRead) 인터럽트가 감지되는 즉시 매핑된 콜백(Callback, 슬롯 함수)을 호출한다.

    A[Hardware Network Card] -->|MAVLink Packets| B(OS Kernel Buffer)
    B -->|Interrupt / epoll| C{QEventLoop in LinkThread}
    C -->|Trigger: readyRead() Signal| D[QUdpSocket]
    D -->|Read Datagram (Non-blocking)| E[MAVLink Parser Engine]
    E -->|Route Payload| F[Signal Queue to Vehicle]

2.2 QUdpSocket과 MAVLink 인터페이스 연동

PX4-Autopilot은 이더넷(Ethernet)이나 Wi-Fi 텔레메트리 연결 시 전송 제어 오버헤드가 없는 UDP 프로토콜을 선호한다. 통신 스레드는 QUdpSocket을 논블로킹 모드로 객체화하며, 다음의 절차로 네트워크 지연을 극복한다.

  1. 소켓 바인딩(Binding): GCS 실행 시 백그라운드 통신 스레드가 특정 포트(예: 14550)를 비동기 방식 수신 모드로 열어둔다.
  2. 시그널 모니터링: 데이터 입력을 기다리며 스레드를 정지(Sleep)시키는 대신, 이벤트 루프를 활성화(exec() 호출)하여 readyRead() 시그널만 대기한다.
  3. 논블로킹 읽기(Drain): MAVLink 데이터 버스트(Burst)가 도달하면, hasPendingDatagrams()가 거짓이 될 때까지 readDatagram()으로 모든 페이로드 메모리를 퍼낸다(Drain). 읽기 행위 자체는 메모리 복사만을 수행하므로 지연 시간이 거의 발생하지 않는다.

3. 동시성 보장을 위한 설계적 경고(Caveat)

QEventLoop 기반의 논블로킹 I/O가 완벽하게 작동하기 위해서는 절대 스레드 이벤트 루프 자체를 차단하는 연산을 스레드 내 콜백(Slot)에서 수행해서는 안 된다.

  • 만약 MAVLink 패킷 해독(Decoding) 함수가 지나치게 무겁거나, 디스크 I/O를 동기식으로 진행하여 이벤트 루프 제어권을 수십 밀리초 이상 운영체제로 양보하지 못할 경우, readyRead() 이벤트를 처리하지 못해 운영체제의 커널 소켓 수신 버퍼(Receive Buffer)가 가득 차 패킷 드롭(Packet Drop)이 발생하게 된다.
  • 이를 예방하기 위해 QGC 백엔드는 수신된 바이트 스트림을 최소 규격 단위(mavlink_message_t)로 1차 조립만 완료한 후, 해석 연산 자체는 QThreadPool의 백그라운드 워커에 이관하거나 Vehicle 스레드 로직 큐로 던지도록 설계되어 있다.

이처럼 C++ Qt의 이벤트 루프(Event Loop) 철학과 스레드 풀링 모델을 조화롭게 결합한 QGroundControl의 아키텍처는, 다수의 PX4 기체가 발산하는 막대한 양의 MAVLink 트래픽 앞에서도 매우 견고하고 선형 확장 가능한(Scalable) 네트워크 처리 유연성을 제공한다.
