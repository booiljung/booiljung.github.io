39.4.1. mavlink_receiver 모듈의 물리 계층 데이터 스트림 처리

1. 서론: 아날로그 신호와 디지털 객체의 경계선

무인기의 비행 제어기(Flight Controller) 펌웨어 프레임워크 내에서 MAVLink 프로토콜 처리를 담당하는 디먼(Daemon)은 크게 송신을 전담하는 mavlink_messages와 수신을 전담하는 mavlink_receiver로 양분된다. 지상 관제 시스템(GCS)에서 송출된 제어 명령(Command) 패킷이 기체 내부의 uORB 버스에 진입하기 위해서는 반드시 이 중 후자인 mavlink_receiver의 깐깐한 입국 심사를 거쳐야 한다.

본 장에서는 PX4(펌웨어 v1.14 기준)의 mavlink_receiver 클래스가 NuttX 실시간 운영체제(RTOS)의 하위 디바이스 드라이버 계층과 어떻게 직결되어 물리적인 바이트 스트림(Byte Stream)을 수집하고, 이를 의미 있는 단일 MAVLink 객체(MAVLink Object)로 프레이밍(Framing)해 내는지 물리 계층(Physical Layer)과의 인터페이스 관점에서 분석한다.

2. mavlink_receiver의 POSIX 기반 블로킹/논블로킹(Blocking/Non-blocking) I/O 아키텍처

PX4 아키텍처의 근간을 이루는 NuttX RTOS는 POSIX(Portable Operating System Interface) 규격을 강력하게 준수한다. 따라서 UART 시리얼 포트(예: /dev/ttyS1), USB 컴포짓 디바이스(Composite Device), 혹은 이더넷 소켓(UDP) 등 통신 물리 매체의 종류와 무관하게 mavlink_receiver는 파일 디스크립터(File Descriptor, FD)를 통한 일관된 I/O 인터페이스를 유지한다.

  1. 파일 디스크립터링 및 폴링 (File Descriptoring and Polling):
    mavlink_receiver 스레드는 시작과 동시에 자신에게 할당된 시리얼 포트 파일 디스크립터(_uart_fd)에 대해 poll() 함수를 호출하여 대기(Wait) 상태에 진입한다. 이는 CPU 사이클을 무의미하게 소모하는 바쁜 대기(Busy Wait)를 방지하고, 하드웨어 인터럽트에 의해 버퍼에 데이터가 쌓였을 때만 스케줄러(Scheduler)로부터 스레드 제어권을 넘겨받기 위함이다.
  2. 청크(Chunk) 단위의 논블로킹 읽기 (Non-blocking Read):
    수신 알림(POLLIN 이벤트)이 발생하면, mavlink_receiver는 POSIX read() 시스템 콜을 사용하여 Rx 버퍼에 쌓여 있는 바이트들을 가져온다. 이때 한 번에 1바이트씩 읽어오는 비효율성을 피하기 위해, 내부적으로 수십 바이트 이상의 덩어리(Chunk) 단위 블록 읽기를 수행하여 커널 공간(Kernel Space)과 사용자 공간(User Space) 간의 컨텍스트 스위칭(Context Switching) 오버헤드를 극소화한다.
/* PX4 mavlink_receiver.cpp 의 단순화된 I/O 폴링 구조 예시 */
void MavlinkReceiver::receive()
{
    pollfd fds[1];
    fds[0].fd = _uart_fd;
    fds[0].events = POLLIN;

    // Rx 버퍼에 데이터가 들어올 때까지 대기 (Timeout 100ms)
    int poll_ret = poll(fds, 1, 100);

    if (poll_ret > 0 && (fds[0].revents & POLLIN)) {
        uint8_t buf[MAVLINK_MAX_PACKET_LEN];
        // 커널 버퍼로부터 청크 단위로 가져오기 (논블로킹)
        ssize_t nread = read(_uart_fd, buf, sizeof(buf));
        
        if (nread > 0) {
            parse_bytes(buf, nread);
        }
    }
}

3. 바이트 파싱(Byte Parsing) 및 MAVLink 프레이밍 상태 머신

드론의 무선 채널은 전파 간섭(Interference)과 감쇠(Attenuation)로 인해 패킷의 유실, 순서 뒤바뀜, 중간 짤림 현상이 빈번한 극도로 적대적인 환경이다. 따라서 read()로 가져온 바이트 배열(Byte Array) 안에는 완전한 MAVLink 패킷뿐만 아니라, 깨진 가비지(Garbage) 데이터나 반 토막 난 패킷 파편이 혼재되어 있을 수 있다.

이를 극복하기 위해 mavlink_receiver 모듈 내부에서는 mavlink_parse_char() C-라이브러리 함수 기반의 마이크로 상태 머신(Micro State Machine)이 구동된다.

  • 패킷 동기화(Packet Synchronization) 확보:
    파서는 입력되는 바이트스트림을 한 바이트씩 훑으며 MAVLink 1.0의 매직 넘버(Magic Number)인 0xFE 또는 MAVLink 2.0의 0xFD (STX 바이트)를 찾는다. 이를 발견하면 상태 머신은 ‘패킷 수신 중’ 상태로 전환된다.
  • 길이(Length) 예측 및 페이로드 조립:
    STX 바로 다음 바이트에서 페이로드 길이를 추출하고, 시퀀스 번호(SEQ), 발신자 ID(SYSID), 컴포넌트 ID(COMPID), 그리고 메시지 ID 식별자(MSGID)를 차례로 파싱용 내부 버퍼체(Buffer Structure)에 덧붙여 나가는 슬라이딩 윈도우(Sliding Window) 집성 과정을 거친다.

이와 같은 mavlink_receiver의 하단부 처리 철학은, 상위 로직(PX4 Commander)이 무선 노이즈나 직렬화된 바이트 분절(Fragmentation) 문제에 전혀 신경 쓰지 않고 오로지 “논리적으로 온전한 메시지 덩어리“만을 받아서 처리할 수 있게끔 더러운 일(Dirty Work)을 도맡아 하는 방파제(Breakwater) 역할을 완벽히 수행하는 것이다.
