39.4.2.1. MAVLink C 구조체를 vehicle_command uORB 토픽 구조체로 치환하는 동형 매핑(Isomorphic Mapping) 알고리즘

1. 서론: 시스템 결합도(Coupling)를 낮추는 인터페이스 분리

무인기 비행 제어 시스템 아키텍처에서 외부 통신 인터페이스(MAVLink)와 내부 상태 인터페이스(uORB)를 어떻게 연결할 것인가는 아키텍트들의 영원한 숙제이다. 만약 조종기 스틱 값, 자율 비행 웨이포인트(Waypoint), 모터 제어 명령 등을 처리하는 내부 디먼(Daemon)들이 MAVLink C 헤더 파일을 직접 #include 하여 사용한다면, 시스템 전반적으로 최악의 결합도(Tight Coupling)를 형성하게 된다. 프로토콜 변경이 펌웨어 코어 전반의 연쇄 수정을 강제하기 때문이다.

PX4(v1.14 기준)는 이 문제를 해결하기 위해 mavlink_receiver 모듈을 격벽(Bulkhead)으로 세우고, 외부 통신 객체(MAVLink Struct)를 내부 공용 객체(uORB Topic)로 일대일 변환하는 동형 매핑(Isomorphic Mapping) 설계 패턴을 채택하였다. 본 절에서는 COMMAND_LONG 맵 메시지를 예시로, 이 치환 알고리즘이 런타임에 어떻게 수행되며 어떤 아키텍처적 이득을 주는지 분석한다.

2. 데이터 전사의 메커니즘: MAVLink 구조체에서 uORB로

mavlink_receiver의 파서가 COMMAND_LONG (Message ID 76) 패킷의 역직렬화(Deserialization)를 마치면, 메모리 상에는 mavlink_command_long_t 형식의 C 구조체가 인스턴스화된다. 이를 기체 제어를 전담하는 Commander 디먼이 안전하게 소비할 수 있도록, PX4는 vehicle_command_s라는 이름의 uORB 토픽 구조체를 새로 생성하여 필드 단위의 얕은 전사(Field-by-Field Transcription)를 수행한다.

  1. 타임스탬프(Timestamp) 재할당:
    무엇보다 중요한 것은 시간 동기화(Time Synchronization)이다. MAVLink 구조체는 GCS가 발송한 시점의 시간이나 시퀀스를 담고 있을지언정 시스템 내부 타이머와는 무관하다. 매핑 함수는 hrt_absolute_time() 커널 API를 호출하여 uORB 토픽의 timestamp 필드에 “이 명령이 우리 시스템 메모리에 도착한 절대 시간(Microseconds)“을 박아 넣는다. 이는 추후 사후 비행 분석(Flight Review) 및 ULog 추적 시 핵심적인 동기화 기준점(Pivot)이 된다.
  2. 동형 필드 변환(Isomorphic Field Transformation):
    데이터의 본질(Semantics)을 구성하는 명령어 ID(command)와 7개의 매개변수(param1 ~ param7)는 변수 타입(자료형)만이 통일된 상태로 1:1 단순 대입된다.
  • mavlink_cmd.command (uint16_t) \to vcmd.command (uint32_t)
  • mavlink_cmd.param1 (float) \to vcmd.param1 (float)
  1. 메타데이터 캐싱(Metadata Caching) 및 안전 필터링:
    단순 값 전달에 그치지 않고, 수신된 명령의 발신지 주소(sysid, compid) 및 명령이 향하는 목적지(target_system, target_component) 정보 또한 vcmd 구조체에 그대로 매핑된다. 만약 수신된 메시지의 타겟 시스템 ID가 내 기체의 ID와 다르다면, 이 매핑 과정 이전에 패킷은 소거 파이프(Bit bucket)로 버려진다(Drop).
/* PX4 mavlink_receiver.cpp 내 COMMAND_LONG 매핑 의사 코드(Pseudo Code) */
void MavlinkReceiver::handle_message_command_long(mavlink_message_t *msg)
{
    mavlink_command_long_t cmd_mavlink;
    mavlink_msg_command_long_decode(msg, &cmd_mavlink); // 역직렬화

    vehicle_command_s vcmd{}; // uORB 구조체 0으로 초기화

    // 1. 타임스탬프 스탬핑 (생명 주기 시작 지점 부여)
    vcmd.timestamp = hrt_absolute_time();

    // 2. 파라미터 및 명령어 동형 매핑
    vcmd.param1 = cmd_mavlink.param1;
    vcmd.param2 = cmd_mavlink.param2;
    // ... param3 ~ param7 매핑 생략
    vcmd.command = cmd_mavlink.command;
    vcmd.target_system = cmd_mavlink.target_system;

    // 3. 발신자(GCS) 정보 메타데이터 저장
    vcmd.source_system = msg->sysid;
    vcmd.source_component = msg->compid;
    vcmd.from_external = true;

    // 4. 로컬 IPC 버스로 퍼블리시
    _cmd_pub.publish(vcmd);
}

3. 시스템 격리(Isolation)의 효과와 ROS2 연동 이점

이러한 동형 매핑 알고리즘은 단 몇 줄의 할당 연산(Assignment Operation)에 불과하지만, 아키텍처 전반에 미치는 나비 효과는 어마어마하다.

  • Commander 모듈의 프로토콜 불가지론(Agnosticism):
    명령을 직접 수행하는 Commander나 센서 모듈들은 MAVLink라는 특정 프로토콜의 존재조차 알지 못한다. 오직 vehicle_command uORB 토픽만을 대기하다가 조건이 맞으면 실행할 뿐이다. 이는 비행 제어 로직을 100% 순수하게 유지하며, MAVLink 프로토콜 1.0 \to 2.0 판올림이나 구조체 형상 변경에 대하여 Commander 코드 영역을 완벽하게 보호(Shielding)한다.
  • Micro XRCE-DDS (ROS2) 와의 플러그 앤 플레이(Plug-and-Play) 통합:
    PX4-ROS2 연동 시, ROS2의 Offboard 제어 노드가 보내제어 명령도 결국 Micro XRCE-DDS 에이전트를 거쳐 vehicle_command uORB 토픽으로 매핑(Publish)된다. 즉, MAVLink로 들어오든(UART), ROS2 퍼블리셔(UDP/DDS)로 들어오든, 최종적으로 IPC 버스 위에서 굴러다니는 로컬 객체(vehicle_command_s)의 형태는 완전히 동형(Isomorphic)이 된다. Commander는 두 발신 채널을 구별할 필요 없이 동일한 논리 회로로 대응할 수 있으며, 이는 다중 에이전트 환경(Multi-agent Environment)이나 외부 컴패니언 컴퓨터 확장에 치명적인 유연성을 가져다준다.

이처럼 PX4의 역직렬화 및 동형 매핑 메커니즘은, 극도로 제한된 임베디드 리소스 하에서도 ’분산 마이크로서비스 아키텍처(Distributed Microservices Architecture)’의 이상을 구현해 낸 핵심 디자인 패턴이다.
