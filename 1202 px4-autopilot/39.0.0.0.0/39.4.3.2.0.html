39.4.3.2. 기체 상태 벡터(Vehicle Status Vector)와 명령어 요구 조건의 논리곱(Logical AND) 기반 실행 인가 판별

1. 서론: “할 수 있는가?“와 “해도 되는가?“의 분리

MAVLink 기반 지상 관제 시스템(GCS)이 “고도 50m로 자동 이륙하라(Takeoff)“는 COMMAND_LONG 메시지를 전송했을 때, PX4 비행 제어기의 Commander 모듈은 이 명령을 곧이곧대로 모터 제어기에 전달하지 않는다. 무인기 시스템은 동역학적, 환경적 제약 속에서 동작하므로, 현재 기체의 상태가 해당 명령을 수행하기에 안전한지 스스로 검열(Censorship)하는 두터운 방어 기제가 필요하다.

본 장에서는 PX4(v1.14 기준)가 수신된 커맨드의 ’실행 요구 조건(Pre-condition)’과 현재 기체의 종합적인 ’상태 벡터(Vehicle Status Vector)’를 논리적으로 교차 검증(Cross-validation)하여 최종 실행 여부(Execution Authorization)를 판별해 내는 논리곱(Logical AND) 메커니즘을 심층 해부한다.

2. 상태 벡터(State Vector) 집계와 모드별 전환 방어선

PX4의 Commander 스레드는 백그라운드에서 끊임없이 20여 종이 넘는 여러 uORB 토픽(센서 데이터, 배터리 상태, EKF 위치 추정치 신뢰도, 데이터 링크 상태 등)을 취합하여 하나의 거대한 전역 구조체인 vehicle_status_s로 집성(Aggregation)한다. 이 상태 벡터는 현재 드론이 ’할 수 있는 것과 없는 것’의 경계를 명확히 규정하는 신분증명서와 같다.

명령어 처리 시에는 이 상태 벡터를 기반으로 철저한 조건부 인가(Conditional Authorization)가 진행된다.

  1. 사전 조건 판별 매트릭스 (Pre-condition Matrix):
    각 MAVLink 커맨드(예: MAV_CMD_NAV_TAKEOFF)는 내부적으로 함수 포인터나 switch-case 블록으로 연결되며, 각각 고유의 검문소를 가진다. 이륙 명령의 경우, is_armed == true (시동 상태인가?), nav_state_valid == true (GPS 3D 픽스가 확보되었는가?), home_position_valid == true (홈 위치가 저장되었는가?) 등의 다중 조건이 요구된다.
  2. 논리곱(Logical AND) 연산에 의한 전이 허가:
    Commander 로직은 앞서 집계된 vehicle_status 벡터의 불리언(Boolean) 필드들과, 특정 커맨드가 요구하는 필수 조건들을 비트 단위 혹은 논리적 AND 연산으로 묶는다. 이 연산 체인(Chain) 중 단 하나라도 false가 도출된다면, 해당 커맨드는 즉각 거부되며 MAV_RESULT_DENIED 또는 MAV_RESULT_TEMPORARILY_REJECTED 전이 상태를 촉발시킨다.

3. Ardupilot과 PX4 간 방어적 프로그래밍(Defensive Programming) 철학 비교

이러한 상태 기반 인가 시스템은 드론 펌웨어의 아키텍처 사상과 안전 주관(Safety Philosophy)을 여실히 투영한다.

  • PX4의 ‘명시적 상태 머신 통제’ (Explicit FSM Governance):
    PX4의 Commander 내부 코드를 뜯어보면 방대한 양의 if-else 분기문이 존재하는데, 이는 매우 보수적인 방어적 프로그래밍의 산물이다. 어떤 모듈도 직접 모터(Actuator)나 내부 모드를 켤 수 없으며, 모든 제어 요청은 상태 벡터와의 논리곱(AND) 검증이라는 단일 병목점(Single Choke Point)을 의무적으로 통과해야 한다. 이는 코드가 무거워 보이는 단점은 있으나, 예상치 못한 상황(예: 실내에서 GPS 없이 위치 제어 모드를 강제하는 등)의 발생을 수학적으로 완벽히 소거(Elimination)한다.
  • Ardupilot의 ‘유연한 예외 처리’ (Flexible Exception Allowance):
    Ardupilot(v4.x)의 경우, PX4 대비 상대적으로 더 유연한(Lenient) 전환 허용 정책을 갖는 모드들이 존재한다. 특정 조건이 완벽히 만족되지 않더라도, 하위 비행 제어 코어에서 추정(Dead Reckoning) 기술이나 폴백(Fallback) 모드를 통해 어느 정도 커맨드를 수용하고 자율적으로 보정하려는 경향이 있다. 즉, 명령어 거부(Refusal)보다는 어떻게든 수행 가능한 대체 경로를 찾는 것에 비중을 두는 설계 코드가 곳곳에 녹아 있다.

결과적으로 PX4의 Commander 모듈은 GCS가 던져주는 MAVLink 명령어에 대해 “무조건 순응하는 종(Servant)“이 아니라, 기체의 전역 상태 벡터를 무기로 “현재 물리적, 논리적 상황에 비추어 허락 여부를 따지는 심사관(Auditor)“으로 동작한다. 이 폐쇄적이고 엄격한 논리곱 검증 시스템이야말로, 복잡한 다중 센서와 수많은 오류 가능성이 혼재하는 실 비행 환경에서 비행체의 생존성을 보장하는 최후의 보루(Last Line of Defense)이다.
