39.3.4. Result Param 2 필드의 다중 오류 캡슐화 기법

1. 서론: 문자열(String) 디버깅의 한계와 기계 판독성(Machine Readability)의 요구

과거 MAVLink 1.0 프로토콜 시절, 비행 제어기(Flight Controller)가 GCS의 특정 제어 명령을 거부(FAILED 또는 DENIED)했을 때, 조종자는 오직 제어기에서 산발적으로 날아오는 STATUSTEXT (자유 형식 텍스트) 메시지를 통해서만 그 사유를 짐작할 수 있었다. 그러나 텍스트 기반의 오류 통보는 다국어(Localization) 지원이 불가능하고, ROS2와 같은 자율 에이전트(Autonomous Agent) 시스템이 프로그램적으로 예외를 파싱(Parsing)하고 복구 논리를 전개하는 데 치명적인 걸림돌로 작용했다.

MAVLink 2.0 규격이 도입되면서 COMMAND_ACK 패킷에는 32비트 정수형(Int32)의 확장 필드인 result_param2가 추가되었다. PX4(펌웨어 v1.14 기준)는 이 필드를 활용하여, 텍스트에 의존하지 않고도 실패의 세부 원인(Sub-reason)을 기계가 즉각 해독할 수 있는 “다중 오류 캡슐화(Multiple Error Encapsulation)” 기법을 고도화하였다.

2. Result Param 2의 아키텍처적 활용: 열거형(Enum) 및 비트마스크(Bitmask) 캡슐화

Commander 디먼 내부에서 특정 명령의 수행이 기각될 때, 반환되는 MAV_RESULT 상태 값뿐만 아니라 **‘왜 거부되었는가’**에 대한 세부 메타데이터가 result_param2 필드 내에 정수 형태로 인코딩(Encoding)된다. 이는 주로 두 가지 기법으로 활용된다.

  1. 단일 상세 에러 코드 (Specific Error Enum):
    특정 임무(Mission) 명령이 거부되었을 때, result_param2에는 MAV_MISSION_RESULT 열거형에 해당하는 오류 코드(예: 3=MAV_MISSION_UNSUPPORTED, 4=MAV_MISSION_NO_SPACE)가 삽입된다.
  2. 비트마스킹을 통한 다중 사유 동시 전송 (Bitmask Aggregation):
    이 기법의 백미는 시동(MAV_CMD_COMPONENT_ARM_DISARM) 명령 실패 시 드러난다. 아밍(Arming)이 거부되는 이유는 종종 복합적이다. GPS가 수렴하지 않았고(Bit 0), 나침반 캘리브레이션이 틀어졌으며(Bit 1), 배터리가 부족한(Bit 2) 상황이 동시에 발생할 수 있다. PX4 코어 프레임워크는 이 방해 요소들을 32비트 마스크의 각 비트 자리에 OR(|) 연산으로 누적(Aggregation)하여 하나의 result_param2 정수로 압축(Encapsulation)한 뒤 전송한다.

3. Ardupilot과 PX4의 예외 피드백(Exception Feedback) 패러다임 차이

두 오픈소스 시스템 간에 오류를 사용자에게 전달하는 공학적 취향은 극명하게 갈린다.

  • PX4-Autopilot의 구조화된 데이터 지향 (Structured Data-Driven):
    PX4는 이 result_param2 메커니즘을 적극적으로 활용한다. GCS 백엔드(QGC C++ Core)는 이 32비트 정수를 수신하여 내부 딕셔너리와 매핑한 뒤, 프랑스어, 한국어, 아랍어 등 GCS에 설정된 언어 팩 리소스(.ts)를 거쳐 번역된 UI를 조종자에게 띄운다. 또한 MAVSDK나 ROS2 Offboard 노드는 이 정수형 사유를 바탕으로 ‘배터리가 없으면 RTL’, ‘GPS가 없으면 Optical Flow로 전환’ 등 즉각적이고 결정론적인(Deterministic) 자율 분기 전환을 실시할 수 있다.
  • Ardupilot의 휴리스틱 문자열 지향 (Heuristic String-Driven):
    Ardupilot 진영은 COMMAND_ACK의 서브 파라미터 활용에 상대적으로 덜 엄격하며, 대신 STATUSTEXT MAVLink 메시지를 통해 “PreArm: GPS Horiz error 3.2m (need < 2.5)” 와 같이 사람이 읽기 편하고 구체적인 문자열을 던지는 휴리스틱(Heuristic) 로직이 파이프라인 전반을 강하게 지배하고 있다. 이는 인간 조종자(Human-in-the-loop)에게는 직관적일지 모르나, 다국어 번역이 사실상 불가하며, ROS2 알고리즘이 문자열을 정규식(Regex)으로 파싱해야 하는 심각한 취약점을 야기한다.

결과적으로 PX4의 result_param2 기반 다중 오류 캡슐화 기법은 인간 종속성을 탈피하고 ’네트워크 트래픽의 경제성’과 ’컴퓨터 에이전트의 기계 판독성’을 동시에 확보한 현대 무인기 인터페이스 설계의 모범이라 평가할 수 있다.
