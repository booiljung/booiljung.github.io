39.2.5.2. 8비트 정수 오버플로우(Overflow) 발생 시의 순환 주기 제어 및 예외 처리

1. 개요

MAVLink 2.0 프로토콜의 COMMAND_LONG 및 COMMAND_INT 메시지에서 재전송 횟수를 추적하는 confirmation 필드는 그 데이터 모델이 1바이트 부호 없는 정수(uint8_t)로 정의되어 있다. 이는 컴퓨터 공학적으로 해당 필드가 표현할 수 있는 최댓값이 2^8 - 1 = 255 임을 의미한다.

본 절에서는 “단조 증가(Monotonically Increasing)“라는 핵심 대전제를 가진 이 계수기가 한계치인 255를 넘어설 때 발생하는 정수 오버플로우(Integer Overflow) 현상을 분석하고, 시스템이 이러한 순환(Wrap-around) 주기를 어떻게 공학적으로 제어하며 치명적인 상태 충돌(State Collision) 예외를 방지하는지 논증한다.

2. 오버플로우와 원형 큐(Circular Queue)로서의 변질

GCS가 명령어 재전송을 지속하여 confirmation 필드의 값이 254, 255에 도달했다고 가정한 극단적 사고 실험(Thought Experiment)을 해보자.

255회째 재전송 이후 또다시 재전송을 시도할 때, C/C++ 기반의 GCS 백엔드 코드에서 confirmation++ 연산이 수행되면 변수는 하드웨어 아키텍처의 모듈로 산술(Modulo 256 Arithmetic) 제약에 의해 0으로 랩어라운드(Wrap-around) 된다.

이는 앞서 39.2.5.1절에서 강조했던 방어 기제인 “단조 증가(\text{New} > \text{Old})” 규칙이 최초로 역전(0 < 255)되는 순간이다. 기체의 비행 제어기(PX4) 입장에서 confirmation=0 인 패킷은 “사용자가 방금 버튼을 눌러 새로 지시한 최초의 원본 명령“으로 규정되어 있다.
결과적으로 오버플로우가 발생하면, 수신단은 이것이 수백 번 지연된 낡은(Stale) 명령인지, 아니면 우연히 Command ID 마저 겹쳐버린 완전한 새 트랜잭션인지 논리적으로 분간할 수 없는 순환 주기 충돌(Wrap-around Collision) 에 빠지게 된다.

3. 무한 재전송의 현실적 차단: 타임아웃(Timeout) 캡 지정

수학적으로는 오버플로우가 MAVLink의 상태 머신에 치명적인 교착 상태(Deadlock)나 오동작을 유발할 수 있지만, PX4와 QGroundControl(QGC) 같은 상용 GCS의 현실적인 엔지니어링 코드 베이스에서는 이런 일이 결코 발생하지 않도록 **다중 방어벽(Defense-in-Depth)**을 구축해 두었다.

가장 대표적인 방어벽이 바로 **최대 재전송 허용치(Maximum Retry Limit)**의 하드 코딩이다.

  1. 상식적인 재전송 캡(Cap): GCS 트랜잭션 매니저는 명령어를 무한정 재발송하지 않는다. 통상적으로 최대 3회에서 5회(설정에 따라 최대 10회 미만) 재전송을 시도한 후에도 COMMAND_ACK 가 돌아오지 않으면, confirmation 밸류를 255까지 올리기도 전에 해당 트랜잭션을 강제 종료(Abort) 처리한다.
  2. 제어권 이양(Failsafe Trigger): 5회 이상의 연속 패킷 유실은 통신 채널의 심각한 혼잡이나 무선 링크 단절(Data Link Loss)을 의미한다. GCS는 사용자에게 통신 장애 경고(UI Alert)를 띄우며 명령을 포기하고, 기체(PX4) 역시 하트비트(Heartbeat) 타임아웃 감지 로직과 연계하여 즉각적인 RTL(복귀)이나 홀드(Hold) 모드로 강제 전이(Failsafe Transition)한다.

따라서 정상적인 운용 시나리오 하에서 confirmation 변수가 오버플로우 경계인 255에 닿는 일은 시스템 아키텍처상 원천 봉쇄되어 있다.

4. 디도스(DDoS) 성 패킷 난입에 대비한 PX4의 예외 처리 로직

그럼에도 불구하고, 악의적인 공격자(Spoofer)가 MAVLink 망에 난입하여 임의로 조작된 패킷(confirmation=0, 1, 255...)을 섞어 무작위(Random)로 쏟아내는 상황을 가정해 볼 수 있다.

이 경우 PX4의 mavlink_receiver 및 시퀀서(Sequencer) 모듈은 confirmation 번호에만 100% 의존하지 않고, 시간 창(Time Window, 슬라이딩 윈도우) 기반의 이중 캐시 검증 알고리즘을 가동하여 예외 처리를 수행한다.

  • PX4는 최근 처리한 명령어의 MAV_CMD 번호뿐만 아니라 해당 명령이 도달했던 **절대 시간(Timestamp)**을 저장한다.
  • 만약 짧은 시간 창(예: 1000\text{ms}) 이내에 confirmation=255 수신 직후 논리를 뒤집는 confirmation=0 패킷이 도달한다면, 상태 관리자(State Manager)는 타임스탬프 델타 연산을 통해 “이토록 짧은 시간에 256번의 정상적인 재전송 루프가 돌았을 리 없다“고 합리적으로 판단(Heuristics)한다.
  • 결과적으로 시스템은 이 비정상적인 순서 역전(오버플로우)이 발생한 패킷을 타임아웃 슬라이딩 윈도우 규칙 위반으로 간주하고, 무조건 폐기하여 상태 전이를 거부(Reject)함으로써 기체의 비행 안정성을 확보한다.

결론적으로 uint8_t라는 매우 제한적인 8비트 메모리 공간만으로도, MAVLink 통신망은 상위 비즈니스 로직(Retry Limit, Time Window)과의 유기적인 결합을 통해 정수 오버플로우의 아킬레스건을 완벽하게 보완하며 최고 수준의 트랜잭션 무결성을 달성하고 있다.
