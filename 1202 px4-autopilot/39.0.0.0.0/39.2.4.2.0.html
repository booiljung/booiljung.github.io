39.2.4.2. 식별자 0(Zero ID) 할당을 통한 브로드캐스트(Broadcast) 및 멀티캐스트(Multicast) 전파 로직

1. 개요

앞선 39.2.4.1절에서 강조했듯, MAVLink 네트워크의 대역폭 최적화와 제어 신뢰성 보장을 위한 대전제는 1:1 유니캐스트(Unicast) 통신이다. 하지만 군집 드론 전체에 즉각 비상 정지를 내려야 하는 상황이나 네트워크에 처음 진입한 기체가 주변 환경을 스캔하는 탐색(Discovery) 이벤트 등, 특정 단일 수신자를 미리 알 수 없거나 한 번에 다수에게 명령을 하달해야 하는 예외적 상황은 반드시 발생한다.

수많은 통신 프로토콜들(예: IP의 서브넷 255.255.255.255 브로드캐스트)과 마찬가지로, MAVLink는 이러한 다대다 전파(Propagation)를 지원하기 위해 **식별자 0(Zero ID)**을 특별한 ‘와일드카드(Wildcard)’ 혹은 ’모두(All)’의 의미로 예약해 두었다.

본 절에서는 target_system과 target_component 필드에 값 0이 할당되었을 때 패킷 라우팅 파이프라인에서 어떠한 확산(Spreading) 알고리즘이 발동하는지, 그리고 이 브로드캐스트/멀티캐스트 모델이 가지는 아키텍처적 한계와 취약성을 분석한다.

2. System ID 0: 전역 브로드캐스트(Global Broadcast) 메커니즘

패킷의 target_system 필드에 예약된 값 0이 기입되면, 이 명령어는 **“이 무선 네트워크(Mesh)에 물려 있는 모든 기체와 GCS가 수신해야 함”**을 선언하는 전역 브로드캐스트가 된다. (target_component를 0으로 함께 두는 경우가 일반적이다.)

이러한 전파 로직이 사용되는 대표적인 시나리오는 다음과 같다.

  1. 상태 알림 및 생존 신고 (Heartbeat):
    HEARTBEAT 메시지 자체는 명령(Command) 패킷은 아니지만, 본질적으로 목적지를 0으로 세팅하여 공중에 뿌려진다. 주변의 GCS나 다른 드론들은 이 메시지를 잡아서 자신의 동적 라우팅 테이블(Routing Table)에 새로운 이웃 노드로 추가(Discovery)한다.
  2. 글로벌 파라미터 요청 (Parameter Discovery):
    GCS가 네트워크 전체 구조를 파악하기 위해 처음 연결되었을 때 사용하는 PARAM_REQUEST_LIST의 타겟을 0으로 설정하면, 연결된 모든 기체가 일제히 자신의 파라미터 항목들을 응답하게 된다.
  3. 군집망 일제 비상 정지 명령 (Emergency Kill Switch):
    다수의 편대 비행 드론이 위성망 결함이나 기상 악화로 제어 불능 상태에 빠질 위험이 있을 때, GCS는 MAV_CMD_DO_FLIGHTTERMINATION 명령어의 타겟을 SysID=0으로 실어 던진다. 네트워크 상의 모든 PX4 펌웨어는 수신 즉시 예외 없이 액추에이터 출력을 차단하고 기체를 추락(또는 비상 낙하산 전개)시킨다.

3. Component ID 0: 기체 내 멀티캐스트(Intra-vehicle Multicast)

만약 target_system에는 특정 드론의 ID(예: 1)가 적혀 있는데 target_component에만 0이 적혀 있다면 어떻게 될까? 이는 해당 1번 드론을 향한 유니캐스트 패킷이지만, 일단 1번 드론 **내부망에 진입한 직후부터는 내부의 모든 컴포넌트들에게 확산되는 ‘기체 내 멀티캐스트’**로 작동한다.

PX4의 mavlink_receiver는 이 수신 패턴(SysID=1, CompID=0)을 감지하면 vehicle_command uORB 토픽으로 메시지를 전달함과 동시에, 시리얼 포트들로 연결된 짐벌(CompID 154)과 컴패니언 컴퓨터(CompID 191) 쪽으로도 해당 패킷을 고스란히 복사(Duplicate)하여 흘려보낸다.

이 기능은 **기체 전반의 상태 변경(State Change)**을 동기화해야 할 때 매우 유용하다.

  • 예시: 무장 해제(Disarming) 명령(MAV_CMD_COMPONENT_ARM_DISARM)을 CompID=0으로 내리면, PX4는 모터를 끄고, 동시에 짐벌 제어기는 액추에이터 전력을 차단해 대기 모드(Standby)로 전환하며, 컴패니언 컴퓨터는 영상 기록 프로세스를 일시 중지하는 등 기체 시스템 전체가 일사불란하게 동기화된 셧다운 루틴을 밟을 수 있다.

4. 브로드캐스트 토폴로지의 공학적 한계 요소 (Limitations)

편리함에도 불구하고, 0번 식별자를 남용하는 것은 극도로 위험하며 MAVLink 프로토콜 표준에서도 강력하게 사용을 지양(Deprecate)하도록 가이드하고 있다. 이유는 비교환성(Non-commutativity)과 신뢰성 붕괴(Breakdown of Reliability) 속성 때문이다.

명령어를 다수에게(Broadcast/Multicast) 뿌려버리면, 송신자(GCS)는 수신 확인 메커니즘(COMMAND_ACK)을 이용할 수 없다.
대기열 제어기(Queue Controller) 입장에서 N 대의 드론으로부터 산발적으로 날아오는 ACK 메시지들의 순서와 유실 여부를 Command ID 단 하나만으로 매핑하여 추적하는 것은 유한 상태 기계(FSM) 관점에서 상태 공간 폭발(State Space Explosion)을 일으킨다.

즉, 브로드캐스트 명령은 “Fire and Forget(쏘고 잊어버려라)” 방식일 수밖에 없으며, 패킷 유실률이 잦은 무선 RF 구간에서 이는 치명적이다. 10대의 드론 중 1대가 전파 간섭으로 비상 정지 브로드캐스트를 듣지 못했다면 그 1대는 군집 대열을 이탈해 독자 행동을 하게 되며, GCS는 누가 패킷을 못 받았는지 알 방법이 없다.

결론적으로 식별자 0을 활용한 광역 라우팅은 초기 네트워크 토폴로지 구성(Discovery)이나 타임아웃 처리를 포기하는 비상 스위치 용도로만 엄격하게 제한(Isolation)되어야 하며, 드론 관제 아키텍처를 스케일업(Scale-up) 하기 위한 근본적인 해결책이 될 수 없음을 공학적으로 인지해야 한다. 군집 관제 시나리오라 할지라도, GCS의 미들웨어(Backend) 단에서 내부 루프(For-loop)를 순회하며 각 기체 ID별로 유니캐스트 명령을 직렬 발송하고 개별 ACK 트랜잭션을 관리하는 것이 현대 다중 무인기 제어의 표준 아키텍처 설계 사상이다.
