39.3.2.4. MAV_RESULT_UNSUPPORTED: 명령어 사전(Command Dictionary) 검색 실패 및 인터페이스 불일치 예외 처리

1. 서론: 이기종(Heterogeneous) 플랫폼 간 인터페이스 계약의 한계

단일 GCS 운영체제(예: QGroundControl) 하나로 멀티로터, 고정익, 잠수정, 지상 로버 등 모든 이기종(Heterogeneous) 자율 에이전트를 관제하기 위해 무인기 생태계는 MAVLink라는 범용 프로토콜의 MAV_CMD 열거형을 인터페이스 계약(Interface Contract)으로 삼았다. 그러나 모든 비행 제어기(Flight Controller) 펌웨어가 이 방대한 사전(Dictionary) 전체를 구현하는 것은 임베디드 메모리 한계와 기체 형상적 제약 때문에 불가능하다.

본 장에서는 GCS 측에서 전달한 명령어를 목적지 펌웨어(PX4 v1.14 기준)가 아키텍처 상 전혀 인식하지 못하거나(Not Implemented), 논리 구조 상 해당 기체 시맨틱(Semantics)과 불일치할 때 발생하는 방어 기제인 MAV_RESULT_UNSUPPORTED(상태 값: 3) 전이 논리를 분석한다.

2. MAV_RESULT_UNSUPPORTED 발생의 런타임 원인

Commander 디먼 내부의 메시지 핸들러(e.g., VehicleCommand::handle_command)는 본질적으로 거대한 해시 맵(Hash Map) 혹은 스위치 분기(Switch Branching) 구조를 갖는다. 패킷을 언패킹(Unpacking)하여 추출한 Command ID가 내부 레지스트리에 존재하지 않으면 폴백(Fallback) 라우팅을 거쳐 최종적으로 UNSUPPORTED가 에코(Echo)된다.

주요 발생 원인은 다음 세 가지 범주로 분류된다.

  1. 하드웨어 호환성 부재 (Hardware Incompatibility):
    QGroundControl에서 쿼드콥터를 향해 MAV_CMD_DO_VTOL_TRANSITION(고정익/멀티로터 형태 변환) 명령을 전송했을 때 발생한다. Commander는 현재 기체의 믹서(Mixer) 프레임이 순수 멀티로터로 설정되어 있음을 인지하고, 요구된 명령의 인터페이스 자체가 존재하지 않음을 UNSUPPORTED로 반환한다.
  2. 펌웨어 버전 파편화 (Firmware Version Fragmentation):
    미래의 MAVLink 표준 제안(RFC)에 새로 등재된 MAVLink 2.0 커스텀 명령어를 GCS가 전송했으나, 기체에 탑재된 PX4 펌웨어가 이를 구현하기 전의 레거시(v1.13 이하) 버전인 경우이다. 프로토콜 파서는 해당 ID를 식별하지 못해 거절한다.
  3. 특정 서브 디먼(Sub-Daemon)의 미운영 상태:
    명령어가 지목하는 타겟 컴포넌트(Target Component) 모듈(예: 커스텀 페이로드 제어기 혹은 VIO 퓨전 모듈)이 로딩(NuttX Task Spawn)되지 않았을 때 명령은 고아(Orphan) 상태가 되고, 최종 수신처의 부재를 UNSUPPORTED 코드로 GCS에 보고한다.

3. Ardupilot과 PX4 간의 명령어 지원 스펙트럼 차이

오픈소스 비행 제어 코드베이스의 양대 산맥인 PX4와 Ardupilot(펌웨어 v4.x 계열)은 지원하는 명령어 사전의 볼륨과 채택 철학에서 뚜렷한 차이를 보인다.

  • PX4-Autopilot의 엄격한 모듈화 (Strict Modularity):
    PX4는 범용적인 명령(표준 MAVLink 규격) 채택에는 적극적이나, 비표준적이고 제조사 종속적인(Vendor-specific) 하드웨어 제어 명령(특정 카메라 프로토콜, 특정 스프레이어 등)을 코어 Commander 사전에 등록하는 것에는 매우 보수적이다. 따라서 사용자 정의 드론 개발 시 UNSUPPORTED를 가장 빈번히 접하게 되며, 이를 극복하기 위해 사용자 정의 uORB 토픽과 커스텀 MAVLink 모듈을 직접 C++로 작성하는 과정이 필수적으로 수반된다.
  • Ardupilot의 확장적 포용성 (Expansive Inclusivity):
    Ardupilot은 전통적으로 AP_Camera, AP_Gripper, AP_Sprayer 등 다양한 레거시 써드파티 장비를 제어하기 위한 수 백 가지의 MAV_CMD 분기 처리가 방대하게 하드코딩(Hard-coding)되어 있다. 이로 인해 임의의 명령어를 던졌을 때 인터페이스 검색에서 UNSUPPORTED가 뜰 확률은 PX4 대비 체감상 훨씬 낮다. 즉 Ardupilot은 다기능 플랫폼 통합 지향성이 강하고, PX4는 클린 아키텍처 기반의 확장성(Extensibility) 유지에 중점을 두어 인터페이스 불일치에 대한 경계가 확실하다.

QGroundControl과 같은 관제 소프트웨어는 텔레메트리를 초기화할 때 기체의 AUTOPILOT 열거형 값(PX4 vs Ardupilot)을 우선 파악하여, UI 상호작용 방식 자체를 분기함으로써 조종자가 UNSUPPORTED 응답 루프와 불필요하게 대면하지 않도록 프로그래매틱한 전처리(Preprocessing) 필터링을 수행한다.
