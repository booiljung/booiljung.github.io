39.2.4. 라우팅 토폴로지 설계: 네트워크 계층적 분기 필터링

1. 개요

하나의 지상 관제 시스템(GCS)이 오직 한 대의 단순한 드론(Pixhawk 단일 보드)과 통신하던 초창기(Point-to-Point) 시절에는, GCS가 발송하는 모든 명령어(COMMAND_LONG 등)의 수신자는 자명하게 비행 제어기(Flight Controller) 단 하나뿐이었다.

그러나 오늘날의 무인기 제어망은 수십 대의 드론이 편대를 이루는 군집 비행(Swarm), 기체 내부에 별도의 영상 처리 보드(Companion Computer)와 지능형 짐벌(Gimbal), 페이로드(Payload) 로봇 팔 등이 복잡하게 얽혀 있는 거대한 **분산 로컬 영역 네트워크(Distributed LAN)**로 진화했다.
이처럼 수많은 통신 노드(Node)가 하나의 무선 주파수(RF) 대역을 공유하는 환경에서, 수신자를 명시하지 않은 무분별한 명령어 브로드캐스팅(Broadcasting)은 네트워크 폭주(Broadcast Storm)와 심각한 오작동을 유발한다.

본 절에서는 MAVLink 2.0 프로토콜이 다중 에이전트 다중 컴포넌트(Multi-Agent, Multi-Component) 환경에서 특정 장비만을 정확하게 타격하여 명령을 전달하기 위해 어떻게 네트워크 계층적 분기 필터링(Hierarchical Branch Filtering) 토폴로지를 설계하였는지 심층 분석한다.

2. MAVLink의 2계층 주소 지정(Addressing) 아키텍처

네트워크 계층 모델(OSI 7 Layer)의 IP 주소와 포트(Port) 번호가 인터넷 라우팅을 담당하듯, MAVLink 생태계에서는 모든 명령어 패킷의 헤더 및 페이로드(target_system, target_component 필드)에 다음과 같은 2계층의 고유 식별 주소가 부여된다.

2.1 계층: 네트워크 노드 식별자 - System ID (1 ~ 255)

System ID는 독립적인 전원을 가지고 네트워크에 참여하는 거시적인물리적 ’개체(Entity)’를 식별한다.

  • 예시: 드론 1번 기체(SysID=1), 드론 2번 기체(SysID=2), QGroundControl 관제 PC(SysID=255).
  • System ID 필터링은 마치 우편물의 ‘도/시/군/구’ 주소와 같아서, 해당 패킷이 내 기체(Vehicle)를 향해 날아온 것이 맞는지 일차적으로 판별하는 거름망 역할을 한다.

2.2 계층: 서브시스템 식별자 - Component ID (1 ~ 255)

동일한 기체(System ID) 내부에서도 여러 보드 프로세서 단위의 다중 컴포넌트 환경이 존재할 수 있다. Component ID는 기체 생태계 내부의 세부적인 ’부서’를 식별한다.

  • 예시: 비행 제어기(PX4 Autopilot, CompID=1/MAV_COMP_ID_AUTOPILOT1), 짐벌 카메라 제어기(CompID=154), 비전 처리용 컴패니언 컴퓨터(CompID=191).
  • Component ID 필터링은 우편물의 ‘수신 부서/세부 담당자’ 주소와 같아서, 기체 내부에 패킷이 무사히 도착한 후 어떤 프로세서가 이 명령을 실행해야 할지 이차적으로 라우팅한다.

3. 계층적 분기 필터링(Hierarchical Branch Filtering) 메커니즘

GCS에서 특정 드론의 짐벌 카메라 각도를 내리라는 명령(MAV_CMD_DO_MOUNT_CONTROL)을 전송했다고 가정해 보자. 패킷의 타겟은 target_system=2(2번 드론), target_component=154(짐벌)로 설정되어 RF 모뎀을 통해 공중으로 뿌려진다.

이 패킷이 물리적인 무선 안테나를 통해 각 노드에 도달했을 때, MAVLink 파서(Parser)를 거치며 작동하는 **분기 필터링 결정 트리(Decision Tree)**는 다음과 같다.

  1. 패킷 스니핑 및 System ID 1차 분기
  • 1번 드론(SysID=1)의 텔레메트리 모듈이 패킷을 수신한다. MAVLink 헤더의 target_system 필드를 파싱한 결과 2임을 확인한다.
  • 결정: “나에게 온 명령이 아니다.” \to 패킷을 메모리에서 즉각 폐기(Drop) 하거나, 설정에 따라 Mesh 네트워크의 다음 노드로 릴레이(Relay Forwarding) 전달만 수행하고 자신의 제어 루프에는 올리지 않는다.
  1. System ID 일치 및 Component ID 2차 분기
  • 2번 드론(SysID=2)의 픽스호크 보드가 패킷을 수신한다. target_system==2 조건이 일치하므로 패킷 검사기(Validator)의 문과 방화벽이 열린다.
  • 픽스호크(PX4-Autopilot, CompID=1) 커널의 mavlink_receiver 스레드가 내용을 검사하여 target_component가 154임을 확인한다.
  • 결정: “우리 기체에 온 명령은 맞지만, 비행 제어기(나)가 처리할 일이 아니라 짐벌이 처리할 일이다.” \to PX4는 해당 명령어를 자신이 직접 실행(Execute)하지 않고, 기체 내부의 시리얼(UART) 포트로 연결된 짐벌 보드로 패킷을 텔레포트(Routing Forward) 시켜준다.
  1. 최종 타겟 도달
  • 짐벌 제어기가 패킷을 전달받고 target_system==2 및 target_component==154가 모두 일치함을 최종 확인한 뒤, 마침내 짐벌 모터의 피치(Pitch) 각도 제어 알고리즘을 구동시킨다.

4. PX4 내부 uORB 브릿징을 통한 라우팅 아키텍처

PX4-Autopilot은 단순히 MAVLink 패킷을 처리하는 것을 넘어, 들어온 명령어를 내부 IPC(Inter-Process Communication) 규격인 uORB(Micro Object Request Broker) 토픽으로 브릿징(Bridging)하는 복잡한 중계 역할을 수행한다.

mavlink_receiver 모듈에서 들어온 서브 컴포넌트 타겟의 MAVLink 패킷은 vehicle_command uORB 토픽으로 변환되어 발행(Publish)된다. 이때 토픽의 구조체 내부에는 여전히 target_system 및 target_component ID 값이 그대로 보존되어 흐른다.

PX4 커널 내에서 구동되는 수십 개의 백그라운드 프로세스(App)들은 각자 이 vehicle_command 토픽을 구독(Subscribe)하고 있다가, 새로운 메시지가 뜨면 가장 먼저 계층적 ID 필터링 조건문을 실행한다.

// PX4 Commander 모듈 내부의 추상화된 필터링 로직 예시
if (cmd.target_system == mavlink_system.sysid) {
    if (cmd.target_component == mavlink_system.compid || 
        cmd.target_component == MAV_COMP_ID_ALL) {
        // 비행 제어기 본연의 명령이므로 즉각 State Machine 처리 돌입
        process_command(cmd);
    } else {
        // Component ID가 다르므로 무시. 
        // 짐벌이나 컴패니언 컴퓨터로 향하는 다른 라우터 스레드가 알아서 낚아채 갈 것임.
        ignore_command();
    }
}

5. 설계 철학적 의의

MAVLink의 System ID / Component ID 기반 계층적 라우팅 토폴로지는 단순한 식별표가 아니라, 저대역폭의 무선 통신망에서 중앙 집중형(Centralized) 허브 라우터 없이도 패킷의 충돌이나 교착(Deadlock) 없는 완벽한 분산 라우팅(Distributed Routing)을 가능케 하는 근간이다.

이 필터링 아키텍처 덕분에 글로벌 관제 시스템(GCS)은 드론 내부의 복잡한 시리얼 배선도어나 uORB 토픽 토폴로지를 알 필요 없이, 단순히 2계층 주소(SysID=X, CompID=Y)만 적어 허공에 패킷을 쏘아 올리는 것만으로도 원격지의 특정 센서나 액추에이터를 핀포인트(Pin-point) 타격하여 제어할 수 있는 완벽한 투명성(Location Transparency)을 획득하게 되었다.
