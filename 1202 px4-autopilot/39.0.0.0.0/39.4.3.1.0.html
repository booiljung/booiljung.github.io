39.4.3.1. vehicle_command 토픽의 비동기 폴링(Polling) 대기열 및 스레드 우선순위(Priority) 상속 모델

1. 서론: “누가 먼저 실행될 것인가?” 다중 커맨드의 병목 현상 제어

하나의 비행 제어기(PX4 Core)에는 조종기(Radio Control), 지상 관제 시스템(QGroundControl), 그리고 온보드 컴패니언 컴퓨터(ROS2 / MAVSDK) 등 다수의 마스터(Master) 노드가 동시다발적으로 명령(Command)을 주입할 수 있다. 만약 0.01초 간격으로 “비행 모드를 변경하라“는 커맨드와 “즉시 이륙하라“는 커맨드가 연달아 수신된다면, 단일 스레드 기반의 Commander 모듈은 이들을 어떤 순서로 처리해야 할까?

비동기 메시징 아키텍처에서는 퍼블리셔(Publisher)의 생산 속도가 서브스크라이버(Subscriber)의 소비 속도를 압도할 때 발생하는 병목(Bottleneck) 현상을 제어하기 위한 대기열(Queue) 설계와, 급박한 스레드 선점(Preemption) 기법이 필수적이다. 본 절에서는 PX4(v1.14 기준)가 vehicle_command uORB 토픽에 대해 어떠한 폴링 큐(Polling Queue) 깊이를 설정하고 있으며, NuttX의 스레드 우선순위 상속(Priority Inheritance) 메커니즘이 어떻게 우선순위 역전(Priority Inversion)을 방지하는지 심층 분석한다.

2. uORB Subscription Queue 깊이(Depth) 메커니즘

uORB(Micro Object Request Broker) 인터페이스 하단에는 단순한 공유 메모리(Shared Memory)를 넘어, 다중 메시지의 유실을 막기 위한 원형 큐(Circular Queue)가 존재한다.

  1. 상태 데이터(Telemetry) vs 명령어(Command)의 태생적 차이:
    자이로스코프 센서 데이터(sensor_gyro)와 같은 텔레메트리 토픽은 최신 값 1개만 유효하므로 큐 사이즈가 1로 설정된다. 과거 데이터는 폐기(Overwrite)되는 것이 당연하다. 하지만 vehicle_command 토픽은 성격이 다르다. QGC가 “웨이포인트 1번으로 가라(1번 커맨드)”, 바로 이어서 “속도를 5m/s로 올려라(2번 커맨드)“를 던졌을 때, 최신 값인 2번만 살아남고 1번 커맨드가 소실(Drop)된다면 드론은 엉뚱한 임무 행동을 보이게 된다.
  2. 명령어 대기열의 할당 (Queue Size Allocation):
    이러한 소실 선견(Foreseeable Drop)을 방지하기 위해, PX4 Commander 모듈은 vehicle_command 토픽을 구독(Subscribe)할 때 uORB 채널의 큐 길이를 수 개(통상 3~4개 수준)로 확장해 둔다.
  3. 비동기 폴링(Asynchronous Polling):
    Commander 스레드가 poll() 함수에서 깨어나면 즉시 큐의 앞단(Head)부터 뒷단(Tail)까지 루프를 돌며 미처리된 MAVLink 커맨드 구조체들을 전부 꺼내어 하나씩 순차적으로 상태 머신 로직에 던진다.

3. 스레드 우선순위 상속 (Priority Inheritance)과 역전 방지

Commander 디먼이 큐에 쌓인 커맨드들을 처리하는 와중에, NuttX RTOS의 선점형 스케줄러 환경에서는 골치 아픈 ‘우선순위 역전(Priority Inversion)’ 문제가 발생할 가능성이 상존한다.

  • 우선순위 역전의 위협 (The Threat of Priority Inversion):
    Commander (중간 우선순위)가 uORB 내부의 데이터를 읽기 위해 락(Mutex Lock)을 쥐고 있는 찰나에, 로깅 모듈(낮은 우선순위)이 실행되면서 CPU 블로킹 I/O를 일으키고, 동시에 자세 제어기(가장 높은 우선순위)가 해당 uORB 버스에 쓰기를 시도하다가 락에 걸려버리는 상황이다. 이 경우 최고 우선순위인 자세 제어기가 최하 우선순위 모듈 때문에 제때 실행되지 못하는 참사가 벌어진다.
  • NuttX의 POSIX 우선순위 상속 프로토콜 (Priority Inheritance Protocol):
    NuttX 커널은 이 문제를 해결하기 위해 시스템 전반의 Mutex에 ‘우선순위 상속’ 속성을 부여한다. 만약 최고 우선순위인 자세 제어 스레드가 락을 획득하고자 대기열에 진입하면, 커널은 락을 쥐고 있는 Commander 스레드의 우선순위를 한시적으로 자세 제어기의 수준(최상단)으로 끌어올린다(Inherit). 덕분에 Commander 스레드는 하위 우선순위 타스크들의 방해를 받지 않고 순식간에 커맨드 파싱과 uORB 읽기/쓰기를 완료한 후 락을 반환하며 본래의 우선순위로 강등(Demote)된다.

이러한 Ardupilot의 스케줄러에서는 찾아보기 힘든 NuttX 내장형 하드 리얼타임 방어 기제(Priority Inheritance Protocol) 덕분에, 수많은 외부 커맨드가 폭풍처럼 uORB 대기열(Queue)에 쌓이더라도 PX4의 코어 비행 제어 루프는 1밀리초의 지연도 허용하지 않는 철옹성 같은 동시성(Concurrency) 안전 영역을 구축하게 되었다.
