39.1.2.2. 제어 명령어의 비마르코프적 원자성(Atomicity) 요구 조건과 순차적 일관성(Sequential Consistency)

비행 역학 데이터인 텔레메트리 스트림이 과거 데이터에 의존하지 않는 ’마르코프 특성(Markov Property)’을 지니고 있다면, 이와 대조적으로 기체의 상태(State) 전환을 촉발하는 제어 명령어(Control Command, 예: COMMAND_LONG)는 비마르코프적(Non-Markovian) 성질을 강력하게 띤다. 제어 명령은 과거의 기체 상태와 앞서 내려진 명령의 맥락에 완전히 종속되며, 반드시 의도한 순서대로 온전히 실행되어야 한다. 본 절에서는 이러한 제어 명령어가 갖추어야 할 데이터베이스 및 분산 시스템 이론 관점의 원자성(Atomicity)과 순차적 일관성(Sequential Consistency)에 대해 논의하고, PX4-Autopilot이 이를 어떻게 시스템적으로 보장하는지 분석한다.

1. 제어 명령어의 비마르코프적 특성

텔레메트리에서는 t-1 시점의 자세 데이터를 유실해도 t 시점의 최신 데이터를 덮어쓰면 그만이다. 그러나 제어 명령어는 그렇지 않다.
예를 들어, GCS에서 다음과 같은 논리 흐름으로 세 개의 명령을 순차적으로 전송한다고 가정하자.

  1. MAV_CMD_DO_SET_MODE (비행 모드를 Offboard로 변경)
  2. MAV_CMD_COMPONENT_ARM_DISARM (기체 시동(Arming) 수행)
  3. MAV_CMD_NAV_TAKEOFF (이륙 수행)

만약 1번과 2번 명령 패킷이 유실되고 3번 이륙 명령만 기체에 도달한다면, 기체는 시동조차 걸리지 않은 상태(Disarmed)이므로 이륙 명령을 수행할 수 없어 즉각 거부(MAV_RESULT_DENIED)하게 된다. 이처럼 제어 명령어는 시스템의 이전 상태(History) 및 선행 명령어의 완수 여부에 극도로 의존하므로, 유실된 과거의 데이터를 덮어쓸 수 없는 비마르코프적(Non-Markovian) 특성을 지닌다.

2. 명령어의 원자성(Atomicity) 요구 조건

데이터베이스 트랜잭션의 ACID 특성 중 하나인 원자성(Atomicity)은 “작업 사이클이 완전히 수행되거나, 아예 하나도 수행되지 않아야 함(All-or-Nothing)“을 의미한다. 분산 제어 시스템인 무인기 통신망에서도 제어 명령어는 원자성을 충족시켜야 한다.

임무(Mission) 플랜 데이터를 QGroundControl에서 PX4로 업로드하는 MISSION_ITEM_INT 통신 과정을 예로 들 수 있다. 수십 개의 웨이포인트를 업로드하는 중 네트워크 단절로 절반만 전송되었다면, 기체는 이 불완전한 임무를 “부분적으로 승인“해서는 안 된다. PX4의 내비게이터(Navigator)와 커맨더(Commander) 모듈은 GCS로부터 MISSION_ACK가 상호 교환되어 전체 트랜잭션이 완료되기 전까지는 새로운 임무를 활성화하지 않는 원자성을 보장하도록 소스 코드가 설계되어 있다. 즉, 네트워크 레이어(UDP)에서 유실이 발생하더라도 애플리케이션 레이어(MAVLink FSM)에서 불완전한 파편화 명령이 실행되는 것을 원천 차단한다.

3. 순차적 일관성(Sequential Consistency)

무선 링크를 넘나드는 데이터는 경로 다변화나 지연(Latency Jitter)으로 인해 송신된 순서와 다르게 수신 안테나에 도착할 수 있다(Out-of-Order). **순차적 일관성(Sequential Consistency)**은 분산 노드(GCS와 PX4) 간에 명령어의 실행 순서가 송신자가 의도한 시간적 선후 관계를 정확히 일치시키는 성질을 뜻한다.

이를 보장하기 위해 MAVLink 프로토콜과 PX4 내부 아키텍처는 다음과 같은 기법을 적용한다.

3.1 MAVLink Sequence 필드 검증

MAVLink 프레임 헤더에는 0에서 255까지 순환하는 1바이트의 시퀀스 번호(seq)가 존재한다. PX4의 mavlink_receiver 클래스는 수신된 패킷의 시퀀스 번호를 트래킹하여, 이전 번호 대비 너무 동떨어진 번호가 들어오거나 순서가 뒤집힌 비정상 패킷을 폐기장(Drop Count)으로 보내어 낡은 명령이 뒤늦게 실행되는 것을 방지한다.

3.2 uORB 비동기 메시징의 타임스탬프 동기화

Ardupilot은 전통적으로 중앙 집중형 단일 루프에서 MAVLink 명령을 직렬 변환하여 처리하는 경향이 강해 순차 검증이 비교적 간단하다. 반면, PX4는 COMMAND_LONG 류의 메시지가 uORB vehicle_command 토픽으로 Publish 되어 비동기적으로 Commander 모듈 등에 전달된다.
비동기 환경에서 동시성(Concurrency) 문제가 발생하여 처리 순서가 역전되는 것을 막기 위해, PX4는 vehicle_command 구조체 내에 고해상도 타임스탬프(timestamp)를 기입한다. Commander 모듈은 들어온 명령의 타임스탬프와 내부 큐의 논리적 시계를 비교하여, 이미 만료된(Expired) 지연 명령이거나 순서에 위배되는 명령은 무효화 처리(NACK 반환)한다.

이러한 비마르코프적 통신 요건들은 결국 무선 구간(Airborne Link)의 신뢰할 수 없는 UDP 프로토콜을 보완하여, 원격으로 드론을 조종하는 GCS 사용자에게 ‘유선망으로 연결된 듯한’ 확정적이고 일관된 제어 감각을 제공하는 필수적인 공학적 기반이 된다.
