39.2.3.2. 정수 연산 기반의 위도/경도 데이터 파이프라인의 연산 복잡도 이점

1. 개요

COMMAND_INT(#73) 메시지가 도입한 10^7 스케일링된 정수형(int32_t) 위도/경도 데이터 포맷은, 앞서 다룬 ’공간 해상도 보존(Zero-quantization Error)’이라는 수학적 무결성 외에도 임베디드 하드웨어 아키텍처 관점에서 또 다른 거대한 이점을 제공한다. 바로 **‘연산 복잡도(Computational Complexity)의 획기적인 감소’**이다.

드론의 두뇌 역할을 하는 픽스호크(Pixhawk) 등의 비행 제어기(Flight Controller)는 데스크톱 PC와 달리 제한된 전력과 제한된 클럭 속도(수백 MHz)를 지닌 ARM Cortex-M 마이크로컨트롤러(MCU) 기반으로 구동된다. 이러한 마이크로컨트롤러 환경에서 실수(Floating-Point) 연산, 특히 배정밀도(double) 연산은 극도로 값비싼 컴퓨팅 자원을 갉아먹는 병목 덩어리다.

본 절에서는 MAVLink 프로토콜 레이어에서의 정수형 데이터 패키징이 PX4-Autopilot의 커널(NuttX) 및 위치 제어기(Position Controller), 추정기(EKF2/Estimator)로 흘러갈 때 하드웨어 레벨(ALU vs FPU)에서 어떠한 연산 속도의 이득을 가져오고, 분산 제어 시스템의 결정론적(Deterministic) 비행 궤적 생성에 어떻게 기여하는지 분석한다.

2. ALU와 FPU 관점의 프로세서 클럭 사이클 비교

컴퓨터의 코어 코어 아키텍처에서 덧셈/뺄셈 연산을 처리할 때, 다루는 숫자가 ’정수(Integer)’이냐 ’부동소수점(Floating-Point)’이냐에 따라 동원되는 하드웨어 논리 회로가 완전히 달라진다.

  • 정수 연산 (ALU, Arithmetic Logic Unit 활용):
    int32_t 계열의 데이터는 ARM 아키텍처의 가장 근원적이고 기본적인 하드웨어인 ALU를 거친다. 최신의 Cortex-M4/M7 프로세서에서 32비트 정수의 덧셈, 뺄셈, 비트 시프트 연산은 대개 1 클럭 사이클(Clock Cycle) 만에 초고속으로 처리된다.
  • 부동소수점 연산 (FPU, Floating-Point Unit 혹은 Soft-Float 활용):
    float나 double 데이터를 연산하려면 하드웨어 FPU가 가동되어야 한다. double 덧셈 연산 하나를 수행하기 위해 지수(Exponent)를 정렬하고, 가수(Mantissa)를 더한 뒤 다시 정규화(Normalization)하는 복잡한 과정을 거치며 수 클럭에서 수십 클럭 사이클을 소모한다. 심지어 구형 MCU와 같이 하드웨어 배정밀도 FPU를 지원하지 않는 칩(Soft-Float 방식 적용)에서는 이 과정이 수십 줄의 어셈블리 소프트웨어 코드로 대체되므로 수백 사이클의 치명적인 지연(Latency) 오버헤드를 유발한다.

COMMAND_LONG 체계에서 float로 넘겨받은 위/경도를 정밀한 미션 처리를 위해 double로 캐스팅(Casting)하고 연산하는 일련의 파이프라인은 MCU에게 상당한 스트레스였다. 그러나 COMMAND_INT 체계에서는 GCS 쪽에서 이미 10^7 스케일링이라는 하드워크(Heavy-lifting)를 대신 처리해 준 뒤 순수한 int32_t 형태로 건네주므로, PX4 프로세서는 값비싼 FPU를 켤 필요 없이 1사이클짜리 ALU 정수 덧셈/뺄셈 연산만으로 글로벌 웨이포인트의 거리 산출 및 로컬 직교 좌표계로의 삼각 변환(Tangent Plane Projection)을 쾌속으로 수행할 수 있게 되었다.

3. EKF2 추정기(Estimator) 파이프라인의 데이터 캐싱 및 변환 고속화

PX4-Autopilot의 항법 추정기인 EKF2(Extended Kalman Filter)는 GPS, IMU, 기압계 등에서 올라오는 초당 수백 개의 데이터를 결합해 기체의 현재 위치를 연산한다. EKF2의 내부 코어 역시 성능 극대화를 위해 글로벌 지구 좌표(Lat/Lon) 대신, 이륙 지점 등을 원점(0,0,0)으로 하는 미터(m) 단위의 로컬 직교 좌표계(NED: North-East-Down)를 베이스 레이어로 사용한다.

이때 MAVLink COMMAND_INT를 통해 들어온 글로벌 목적지(예: x=375665350, y=1269779690)를 내부 NED 미터 단위로 변환하는 ‘투영(Projection)’ 알고리즘에서도 정수 연산의 이점이 폭발적으로 발휘된다.

// PX4의 위치 제어 모듈 및 map_projection 모듈의 C++ 로직 추상화
// double을 무분별하게 사용하는 대신, 정수 오프셋 뺄셈을 선행하여 FPU 부담을 최소화
int32_t delta_lat_int = target_lat_int - current_home_lat_int; // 단 1 클럭 소모!
int32_t delta_lon_int = target_lon_int - current_home_lon_int; // 단 1 클럭 소모!

이처럼 현재 위치 원점과 목적지와의 거리를 구할 때, 굳이 무거운 double 좌표 두 개를 로드하여 FPU 뺄셈을 거칠 필요 없이, 수신된 10^7 int32_t 상태 그대로 가벼운 ’정수 오프셋 뺄셈(Integer Offset Subtraction)’을 먼저 수행해 버린다. 그리고 그 도출된 작은 잔차(Residual) 오프셋만을 최종적으로 미터(m) 단위로 환산(라디안 파이 및 지구 반지름 곱셈 등)하는 하이브리드 투영 방식을 적용하여, 연산 복잡도를 O(1)의 극한으로 최적화해 낸 것이다.

4. 결정론적 비행 궤적(Deterministic Flight Trajectory)의 보장

연산 복잡도의 이점 못지않게 중요한 또 다른 공학적 열매는 ‘연산의 결정론성(Determinism)’ 확보이다.

WGS84 실수(Floating-point) 좌표계를 이용해 GCS(Windows/Intel Core i7 아키텍처) 컴퓨터에서 계산한 두 목적지 간의 거리와, PX4(NuttX/ARM Cortex-M7 아키텍처)에서 계산한 두 목적지 간의 거리는 하드웨어 FPU의 부동소수점 반올림 처리 규격 차이 및 컴파일러 플래그 차이로 인해 미세한 비결정성(Non-determinism) 오차를 유발한다.

반면, 10^7 스케일링 **정수 연산(Integer Arithmetic)은 CPU의 종류(x86이든 ARM이든)와 운영체제, 컴파일러 버전을 막론하고 우주 어디에서든 1더하기 1은 무조건 완전한 2(Mathematical Precision)가 나오는 완전한 이산 논리(Discrete Logic)**의 영역에 있다.

이러한 정수형 프로토콜 릴레이 덕분에, 드론 개발자들은 QGroundControl (GCS) 화면의 임무 계획기에서 시뮬레이션으로 시각화하여 확인한 비행 궤적 길이와 각도가, 실제 현장에서 드론의 픽스호크 보드 내부 메인 스레드 연산 결과와 단 1비트의 마이크로 오차도 없이 100\% 완벽히 일치할 것임을 공학적으로 보증할 수 있게 되었다.

5. 소결: 하드웨어-친화적(Hardware-Friendly) 통신 규약의 미학

아키텍트들이 MAVLink COMMAND_INT 메시지에 도입한 정수형 하이브리드 설계는 “단순히 부동소수점의 한계를 우회하는 꼼수” 정도의 소프트웨어적 회피 수단이 아니다. 이는:

  1. WGS84 위경도 좌표의 물리적 공간 해상도 손실 방지 (데이터 무결성)
  2. 제한된 텔레메트리 대역폭의 낭비 방지 (double 증설 회피)
  3. 픽스호크 마이크로컨트롤러 단의 FPU 연산 병목 제거 (하드웨어 최적화)
  4. 크로스 플랫폼 간 수학적 이질성 제거 (결정론적 동기화)

이 네 가지를 동시에 만족시키는 대업을 이루어냈다. 데스크탑 PC(QGC)의 막강한 리소스를 활용해 복잡한 스케일링 연산을 먼저 처리해 주고 픽스호크(PX4)에게는 씹어 넘기기 편한 잘게 부서진 알약(int32_t) 형태로 먹여주는 이 비동기 설계 철학은 실시간 엣지 컴퓨팅(Edge Computing) 및 IoT 분산망 제어 인프라 스트럭처의 가장 모범적이고 예술적인 최적화 구조 원리로 평가받고 있다.
