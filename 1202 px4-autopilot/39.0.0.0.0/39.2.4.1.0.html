39.2.4.1. System ID 및 Component ID 기반의 유니캐스트(Unicast) 타겟 지정 모델

1. 개요

현대의 다중 무인기(Multi-UAV) 제어 네트워크 구조에서, 전송되는 모든 명령어 패킷이 모든 노드에 뿌려진다면 네트워크 대역폭은 순식간에 고갈될 것이며 제어의 신뢰성(Reliability)을 보장할 수 없다. 이를 극복하기 위해 MAVLink 프로토콜은 송신자가 수신자의 고유 주소를 명시하여 1:1로만 통신하는 유니캐스트(Unicast) 타겟 지정 모델을 가장 기본적인 명령어 라우팅 원칙으로 채택하고 있다.

본 절에서는 COMMAND_LONG 및 COMMAND_INT 등 MAVLink의 모든 명령어 패킷에 내장된 target_system과 target_component 필드가 어떻게 결합하여 무인기 네트워크 내에서 유일무이한(Unique) 1:1 유니캐스트 논리 터널을 형성하는지, 그리고 이 모델이 시스템 아키텍처 관점에서 어떠한 공학적 이점을 창출하는지 분석한다.

2. System ID: 거시적 비행 개체(Vehicle) 및 GCS의 식별

유니캐스트 라우팅의 첫 번째 관문은 System ID (1 ~ 255, 8비트 부호 없는 정수)이다. 이 값은 네트워크에 참여하는 ’물리적으로 독립된 큰 덩어리’를 식별한다.

  • 기체(Vehicle) 할당: 통상적으로 드론은 1번부터 순차적인 ID를 갖는다. (예: System ID = 1은 1호기, System ID = 2는 2호기)
  • 지상 관제 시스템(GCS) 할당: QGroundControl과 같은 관제 PC는 통상적으로 가장 높은 번호 대역인 255 번을 할당하여 비행 기체들과의 ID 충돌을 방지한다. 보조 GCS나 안테나 트래커(Antenna Tracker) 등은 254, 253 등을 사용한다.

GCS(SYSID 255)가 1번 드론을 이륙(Takeoff)시키고 싶을 때, 명령어 패킷의 target_system 필드에 명시적으로 1을 기록하여 발송한다. 무선 주파수(RF)를 통해 이 패킷이 수백 대의 드론 편대에 동시에 도달하더라도, System ID가 1이 아닌 우주 상의 모든 주변 기체들은 운영체제(RTOS)의 최하단 MAVLink 파싱 레이어에서 이 패킷을 즉각 폐기(Silent Drop)함으로써 CPU 리소스와 uORB 대역폭 낭비를 원천 차단한다.

3. Component ID: 기체 내 마이크로 서비스(Micro-service) 지정

네트워크 패킷이 1번 드론에 무사히 진입했더라도, 드론 내부에는 비행 제어기(PX4-Autopilot)뿐만 아니라 카메라, 짐벌, 보조 컴퓨터 등 다양한 하위 시스템이 존재한다. 이때 유니캐스트의 두 번째 관문인 Component ID (1 ~ 255)가 작동한다.
이는 현대 소프트웨어 공학의 마이크로 서비스 아키텍처(MSA, Microservices Architecture)와 매우 유사한 모델이다.

MAVLink 표준 위원회는 주요 컴포넌트들에 대해 사전 정의된(Pre-defined) 매직 넘버(Magic Number)를 부여하고 있다.

  • MAV_COMP_ID_AUTOPILOT1 (1): 메인 비행 제어기 (PX4 커널)
  • MAV_COMP_ID_CAMERA (100): 메인 페이로드 카메라
  • MAV_COMP_ID_GIMBAL (154): 카메라를 제어하는 3축 짐벌 마이크로컨트롤러
  • MAV_COMP_ID_MISSIONPLANNER (190): 컴패니언 컴퓨터(예: Raspberry Pi) 위에서 도는 자율 비행 노드

GCS가 짐벌 각도를 내리는 명령을 내릴 때, target_system=1, target_component=154 로 패킷을 인코딩한다. PX4의 mavlink_receiver 스레드는 이 패킷을 수신한 뒤, 목적지가 자신이 아님(CompID != 1)을 인지하고 패킷을 버리는 것이 아니라 기체 내부망(Internal Bus, 예: UART, CAN)을 통해 154번 노드(짐벌) 방향으로 바이패스(Bypass) 라우팅을 수행한다.
이처럼 Component ID 층위의 유니캐스트 모델은 PX4-Autopilot이 중앙 집권적 폭군(Monolithic Tyrant)이 아닌, 영리한 메인 라우터(Main Router)로서 기체 내부의 독립적인 주변기기(Peripherals)들을 조율하게 만드는 훌륭한 설계 장치이다.

4. 유니캐스트 모델이 신뢰성 제어(Reliability Control)에 미치는 파급력

“명령어의 수신자를 정확하게 한 명으로 지정한다“는 이 단순한 유니캐스트 원칙은, 뒤이어 다룰 39.3절의 핵심인 타임아웃(Timeout) 및 무결성 보장(ACK/NACK) 메커니즘을 성립시키는 정보-이론적(Information-theoretic) 절대 필수 조건(Prerequisite)이다.

명령어를 브로드캐스트(Broadcast)로 살포해버리면, “누가 명령을 받았고, 누가 명령을 거부했는지“를 추적하는 유한 상태 기계(FSM, Finite State Machine)를 GCS 단에서 구축하는 것이 수학적으로 불가능해진다. 10대의 드론 중 9대만 응답(ACK)을 보냈을 때 누구에게 재전송해야 하는지 특정할 수 없기 때문이다.

반면, SysID와 CompID로 묶인 견고한 1:1 유니캐스트 파이프라인은 송신주체(GCS)로 하여금 특정 타겟에 대한 전용 트랜잭션 메모리(Transaction Memory)를 할당하게 해 준다. GCS는 패킷을 보낸 직후 해당 ID 쌍(Pair)에 대한 타이머(Timer)를 가동시키고, 오직 그 수신자로부터 도달한 COMMAND_ACK 응답 메시지만을 교차 검증(Cross-Validation)하여 상태 전이를 일으킬 수 있게 된다.

결과적으로 두 개의 1바이트짜리 주소 필드로 구성된 이 간결한 타겟 지정 모델은, 예측할 수 없는 무선 통신 환경 위에서 거대한 자율 비행 생태계가 신뢰성 붕괴(State Mismatch) 없이 유기적으로 동작하도록 지탱하는 가장 거대한 구조적 주춧돌이라 할 수 있다.
