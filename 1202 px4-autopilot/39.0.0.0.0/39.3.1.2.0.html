39.3.1.2. 타겟 컴포넌트 유효성 검사를 통한 비인가 응답(Spoofed Response) 차단 로직

1. 서론: 다중 에이전트 통신망에서의 비인가 응답(Spoofed Response) 위협 모델

단일 무인기 관제 환경을 넘어 군집 비행(Swarm Flight)이나 복합 유무인 체계(MUM-T)가 도입되면서, 단일 지상 관제 시스템(GCS)이 다수의 MAVLink 노드(드론, 페이로드 센서, 안테나 트래커 등)와 동시에 오버레이 네트워크(Overlay Network) 브로드캐스트를 기반으로 통신하는 시나리오가 보편화되었다. 이러한 공유형 주파수 채널에서는 특정 드론 B가 드론 A의 COMMAND_ACK 응답과 동일한 타이밍에 동일한 Command ID의 응답을 무작위로 발산할 때, GCS가 이를 오인하여 드론 A의 제어 상태(State Machine)를 강제로 전이해 버리는 ‘응답 교차 오염(Cross-Talk Contamination)’ 또는 악의적인 ‘비인가 응답(Spoofed Response) 인젝션’ 공격이 발생할 수 있다.

본 장에서는 PX4(펌웨어 v1.14)와 QGroundControl(v4.3)의 내부 아키텍처가 MAVLink target_system 및 target_component 필드를 이용해 이러한 치명적 식별 결함을 어떻게 구조적으로 차단하는지 공학적으로 분석한다.

2. 타겟 시스템 및 컴포넌트 유효성 검사 메커니즘

MAVLink 프로토콜의 모든 패킷은 헤더(Header)에 송신자(Sender)의 sysid(System ID)와 compid(Component ID)를 기본적으로 담고 있다. 이에 더해 COMMAND_ACK(#77) 패킷의 페이로드에는 해당 응답의 최종 수신자가 되어야 할 대상을 지명하는 target_system 필드와 target_component 필드가 내장되어 있다.

QGroundControl 프론트엔드 및 백엔드 C++ 인프라(Vehicle 모듈의 다중 인스턴싱 구조)는 이 변수들을 활용해 3계층(3-Tier)의 유효성 검사 아키텍처를 가동한다.

2.1 계층 필터링(3-Tier Filtering) 아키텍처

  1. 물리적 엔드포인트 필터(Endpoint SysID Check)
    MAVLink 패킷이 GCS 물리 소켓(UDP/UART)에 디코딩되면, 패킷 헤더의 sysid가 현재 QGroundControl에 논리적 객체로 등록된 Vehicle 인스턴스 해시 테이블에 존재하는지 확인한다. 만약 미등록된 sysid(예: 비인가 하드웨어)라면 패킷은 즉각 폐기(Drop)된다.
  2. 명령 큐 매너저 타겟 일치(Target ID Matching)
    인지된 Vehicle 인스턴스로 패킷이 라우팅된 후, 대기 중이던 명령어 큐(Command Wait Queue)는 발생지 명령을 기록할 때 저장했던 타겟 ID와 현재 도착한 COMMAND_ACK 내부의 target_system, target_component 값이 GCS 자신(일반적으로 GCS의 SysID는 255)을 정확히 지목하고 있는지 교차 검증을 수행한다.
  3. 컴포넌트 세분화(Component Granularity) 검사
    명령어가 비행 제어기 보드(COMP_ID_AUTOPILOT1 = 1)를 향한 것인지, 아니면 동반 컴퓨터(Companion Computer) 내부의 특정 프로세스나 별조의 짐벌(Gimbal) 제어기(COMP_ID_GIMBAL = 154)를 향한 것인지 엄격히 파싱한다. 일치율이 100%가 아니면 상태 업데이트(Progress Bar 등) 트리거 이벤트를 무효화(Deny)한다.

3. Ardupilot 대비 PX4의 방어적 프로그래밍 관점

다중 시스템 환경 및 컴포넌트 식별(Component Identification) 처리에 있어 PX4와 Ardupilot은 기본 로직을 공유하지만, 예외가 발생했을 때 백엔드 프레임워크가 대응하는 방어적 격리 수준에는 미세한 차이가 존재한다.

  • PX4-Autopilot의 타겟 지정 강제성 (Strict Target Enforcement):
    PX4 내부 로직(mavlink_receiver.cpp)은 수신된 COMMAND_LONG이 자신을 명시적으로 지칭하지 않고 브로드캐스트(target_system=0) 목적으로 들어온 경우, 명령어의 성격(예: 동기화 불필요한 Heartbeat 관련 등)에 따라 매우 보수적인 태도를 취한다. 특히 치명적 물리 제어 명령이 브로드캐스트로 들어오면 아예 큐 배열에 싣지 않고 거부(Silently Ignore)함으로써, 애초에 혼란스러운 비인가 응답 상황을 생성 시점부터 구조적으로 원천 봉쇄한다.
  • Ardupilot의 컴포넌트 릴레이(Relay) 철학:
    Ardupilot은 CAN 버스 또는 시리얼 브리지를 통해 연결된 하위 센서(마이크로컨트롤러 등)로의 MAVLink 패킷 릴레이 기능이 프레임워크 전반에 관대하게 녹아있다. 따라서 GCS가 타겟을 모호하게 설정하더라도, Ardupilot은 메인 제어기를 거쳐 하위 컴포넌트로 일단 전송을 시도하고 응답을 대행(Proxy)하는 다중 노드 릴레이 패턴을 취할 때가 많다. 이는 구성 편의성을 높이지만, 반대로 비인가 응답이 릴레이 과정에서 교차될 보안 취약점의 표면적(Attack Surface)을 다소 넓히는 방향으로 작용할 수 있다.

이와 같이 타겟 컴포넌트 필터링 로직은 ROS2의 DDS 보안 규격(DDS-Security)으로 넘어가기 이전에, 물리적 시리얼 버스 레벨에서 비행 시스템의 신뢰성(Reliability)을 보장하는 최후의 1차 방어선 역할을 담당한다.
