39.7.1.2. 다중 명령 간의 실행 순서 역전(Out-of-Order Execution) 현상 탐지 및 큐잉 딜레이(Queueing Delay) 분석

1. 개요

GCS 로그(tlog) 마이닝의 두 번째 심화 목표는 “시간의 흐름이 뒤틀린 현상“을 포착하는 것이다. MAVLink 데이터 링크는 기반 프로토콜로 주로 비연결성(Stateless)인 UDP 통신이나 시리얼 라이선스(Serial Line) 라디오(RF)를 사용한다. 이 기저 체계들은 패킷이 GCS에서 튜닝된 순서대로 드론에 도착할 것이라는 어떠한 맹세도 하지 않는다.

조종사나 지상 관제 자동화 시스템(Autopilot GCS)이 짧은 찰나에 여러 개의 제어 명령을 다발로 쏟아낼 때, 이 패킷들의 순서가 공중에서 뒤바뀌어 버리는 실행 순서 역전(Out-of-Order Execution) 현상은 가끔씩 드론이 완전히 마친(Done) 줄 알았던 이전 임무를 다시 수행하러 돌아가는 유령 현상(Ghosting)을 만들어낸다. 본 절에서는 다중 명령 전송 시 발생하는 큐잉 딜레이(Queueing Delay)와 실행 순서 역전 현상을 tlog에서 어떻게 적발해 내는지 탐구한다.

2. 비연결성 프로토콜의 병폐: 실행 순서 역전(Out-of-Order Execution)

비행 로직의 충돌은 대개 상태(State)를 급격하게 변화시키는 두 명령이 수십~수백 밀리초(ms) 이내에 연달아 발송될 때 발생한다.

예를 들어, 다수(Swarm) 관제 프로그램이 드론에게 다음과 같이 연속 명령을 내렸다고 하자.

  1. Command A: MAV_CMD_DO_SET_MODE (웨이포인트 임무 모드로 전환) - t_0 발송
  2. Command B: MAV_CMD_NAV_RETURN_TO_LAUNCH (비상 복귀 모드로 전환) - t_0 + 0.05초 발송 (사용자가 마음을 바꿈)

GCS의 의도는 응당 기체가 B(RTL) 명령을 최종적으로 받들어 고향으로 돌아오는 것이다. 하지만 무선망의 다중 경로 페이딩(Multipath Fading)이나 텔레메트리 모듈 내부의 패킷 재조립 과정에서 B가 A보다 기체의 안테나에 먼저 도달(또는 A가 유실되었다가 재전송되어 뒤늦게 도달)한다면 어떻게 될까?
기체는 RTL 모드로 잘 전환했다가, 0.1초 뒤에 늦게 도착한 A 명령을 곧이곧대로 받아들여 다시 웨이포인트 임무를 수행하러 기수를 휙 돌려버릴 것이다. 관제 화면의 조종사 입장에서는 드론이 미쳐버린(Crazy) 것처럼 보이게 된다.

3. 통신 버퍼 내의 큐잉 딜레이(Queueing Delay) 축적 현상

이러한 패킷 역전과 지연의 근본 원인을 파헤치기 위해, 엔지니어는 GCS tlog를 뜯어 **큐잉 딜레이(Queueing Delay)**를 산출해야 한다.
텔레메트리 라디오(SiK Radio, Microhard 등)는 일반적으로 내부 메모리 버퍼가 작으며, TDD(Time Division Duplex) 방식으로 동작하므로 일시적으로 대역폭(Bandwidth)이 포화되면 패킷을 내부 큐에 쌓아두기(Buffering) 시작한다.

GCS가 COMMAND_LONG 5개를 0.01초 간격으로 무자비하게 쐈을 때, tlog 상에 기록된 COMMAND_ACK 5개의 수신 타임스탬프를 확인해 보라.
1번 응답은 0.1초 만에 왔지만, 2번은 0.3초, 3번은 0.6초, 4번은 1.1초 뒤에 도착하는 식으로 응답 시간이 선형적(Linear) 또는 기하급수적으로 늘어진다면, 이는 명백히 물리 계층(PHY/MAC) 하드웨어 버퍼에 트래픽이 쌓이면서 발생하는 병목 큐잉 딜레이다.

4. 시퀀스 매칭을 통한 다중 명령 역전 현상 진단

이 미세한 순서 역전과 큐잉 병목을 증명하기 위해, 로그 마이닝 도구는 tlog 스트림에서 다음 두 가지 데이터를 크로스 체킹(Cross-Checking)한다.

  • 명령 발송-수신 타임라인 상관관계: 여러 명령이 연달아 발송되었을 때, 응답(ACK)이 돌아온 순서가 발송 순서와 교차(Crossing)되는 선형을 그리는지 확인한다.
  • HEARTBEAT.custom_mode의 플리핑(Flipping) 탐지: HEARTBEAT 패킷(1Hz) 혹은 그보다 빠른 빈도로 내려오는 EXTENDED_SYS_STATE의 모드 플래그가 단 몇 백 밀리초 사이에 MISSION -> RTL -> MISSION 등으로 미친 듯이 깜빡이는 흔적(Flipping)을 남긴다면, 이는 두 명령어가 공중에서 꼬여서 순서대로 실행되지 못하고 운영체제의 커맨드 처리 파이프라인 안에서 뒤섞여 폭발했다는 확실한 부검(Autopsy) 결과다.

이러한 분석 결과를 바탕으로, 소프트웨어 엔지니어는 GCS의 전송 큐에 “앞선 명령의 ACK를 받기 전에는 다음 상태 전이 명령을 보내지 않는다(Strict Sequential Pipeline)” 거나, “최소 0.5초의 명령어 간 강제 지연(Inter-command Gap)을 둔다“는 식의 아키텍처 수준의 동시성 제어 방어벽을 추가하여 시스템의 강건성을 확보하게 된다.
