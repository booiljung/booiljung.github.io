37.3.3. 데이터 링크 손실(Data Link Loss) 및 열악한 통신 환경에서의 결함 허용(Fault Tolerance) 시스템

오토파일럿 기반 무인항공기(UAV) 운영 환경에서 무선 데이터 링크(Data Link)는 본질적으로 간섭, 잡음, 물리적 차폐물로 인해 높은 확률로 단절(Loss)되거나 열화(Degradation)될 수밖에 없다. 지상 관제 시스템(GCS)과의 COMMAND_LONG 텔레메트리나 자율 에이전트(ROS 2)의 실시간 Offboard 제어 신호가 일시적 또는 영구적으로 소실되는 상황은 시스템 붕괴로 직결될 수 있는 가장 치명적인 위험 요인이다.
이 절에서는 PX4-Autopilot이 열악한 통신 환경에서 안전한 시스템 연착륙(Soft Landing)을 유도하기 위해 채택하고 있는 제어 공학 측면의 결함 허용(Fault Tolerance) 아키텍처와 그 내부 논리를 심층 분석한다.

1. 데이터 링크 손실(DLL, Data Link Loss)의 정의와 식별 기작

데이터 링크 손실은 시스템이 GCS나 Companion Computer와의 MAVLink 심박(Heartbeat) 동기화를 상실했을 때 발생한다.

PX4의 Commander (또는 최신버전의 FailsafeStateMachine) 모듈은 다음과 같은 이산 감시 타이머(Watchdog Timer) 메커니즘을 통해 데이터 링크 상태를 모니터링한다.

  • NAV_DLL_ACT 파라미터 계열을 기반으로 트리거된다.
  • RC Loss (조종기 연결 단절): SBUS/PPM 수신기에서 신호 프레임이 일정 횟수 이상(통상 0.5초 지속) 유실되거나 송신기 전원이 차단될 때 발생. 기본 타임아웃 파라미터는 COM_RC_LOSS_T 로 설정된다.
  • Datalink Loss (GCS 연결 단절): MAVLink 채널을 통해 수신되는 HEARTBEAT 메시지 스트림이 끊겼을 때 감지. COM_DL_LOSS_T 파라미터(기본 10초)에 의해 지배된다.
  • Offboard Loss (반려 컴퓨터 단절): OffboardControlMode를 서브스크라이빙하던 ROS 2 브리지 연결이 끊기거나 주기가 지연되었을 때 감지. COM_OF_LOSS_T (기본 0.5초)로 매우 가혹하게 설정된다.

이러한 독립적인 워치독 타이머가 임계치를 초과(Timeout)하면, 시스템은 관측기(Observer) 계층에서 치명적인 결함 상태 플래그를 활성화하고, 상태 기계(FSM)를 결함 허용(Fail-safe) 노드로 강제 천이시킨다.

2. 결함 허용 시스템(Fault Tolerance System)의 안전 우선순위 체계

결함 허용 시스템의 목적은 오직 하나, “비정상적 상황에서도 추락과 인명 및 대물 피해를 최소화하는 상태 방어“이다. 데이터 단절이 식별되면 PX4 커맨더는 다음의 방어 로직을 순차적/조건적으로 발동한다.

2.1 예외 처리 천이 트리(Failsafe Transition Tree)

GCS와 단절되었을 때의 행동(Action)은 파라미터 세팅에 따라 크게 4가지 복구 기작으로 분기된다. 시스템은 현재 기체의 생존 가능성(배터리 잔량, 비행 고도, 홈 위치 저장 여부)을 평가하여 최적의 복귀 방식을 스스로 결정(Arbitration)한다.

  1. Hold (위치 유지/배회 모드): 외부 궤적 제어 루프를 끊고 현재의 Local Position을 기준으로 제자리 체공(Hovering)하며 데이터 링크 재연결을 대기한다. 짧은 통신 음영 지역에서 유용하다.
  2. Return (돌아오기, RTL): GPS 기반 글로벌 포지션이 유효할 경우, 사전 설정된 안전 고도(Return Altitude)로 상승 후 이륙 지점(Home) 또는 랠리 포인트(Rally Point)로 자율 복귀비행을 전개한다. 가장 보편적인 강제 복구 절차이다.
  3. Land (현 위치 착륙): GPS 상태가 불량하거나(Position Invalid), Return을 수행할 배터리 에너지가 만성적으로 부족할 경우 현재 좌표를 기준으로 조심스럽게 수직 하강 착륙을 실시한다.
  4. Terminate / Flight Termination (시스템 강제 종료): Failsafe 로직마저 붕괴될 우려가 있는 심각한 다중 결함(예: DLL 발생과 동시에 치명적 IMU 융합 실패) 시 낙하산을 전개하거나 모터를 정지(Disarm)시켜 비행을 강제로 영구 중단(Terminate)한다.

3. 이기종 아키텍처 관점(PX4 vs Ardupilot)에서의 안전 철학 차이

통신 단절에 대한 결함 허용 설계에서 두 플랫폼은 흥미로운 차이를 보여준다.

  • Ardupilot (절차 중심적): GCS Failsafe(FS_GCS_ENABLE)가 트리거되면 즉시 1차원적인 RTL 또는 지정된 모드로 스위칭된다. RC Failsafe와 GCS Failsafe가 약간 독립적인 모듈들로 이분화되어 있으며, 각 모듈이나 서브시스템에서 failsafe() 함수를 반복적으로 호출하여 이벤트를 처리하는 다소 산개형 시스템을 갖는다.
  • PX4 (중앙 집중식 FSM 상태 기반 제어): 최근 PX4는 커맨더 모듈 이외에 FailsafeStateMachine (나아가 최신 버전에 도입되는 Lua 기반 백엔드 등)라는 시스템 전역 상태 관리 트리(State Management Tree)를 도입하였다. 모든 단절 오류(Data Link, RC Link, GCS 타임아웃, VIO 센서 드롭)는 무작위로 RTL을 때리는 것이 아니라, 중앙 이벤트 허브로 모여 “현재 기구학적 상태(Velocity, Altitude) 및 다중 센서 생존율“을 교차 검증한 후, 위상 공간상 가장 안전한 결함 허용 루트를 수학적 조합(Boolean evaluation table)에 의해 결정한다.

4. 자율 비행(ROS 2/Offboard) 관제 기술 통합 시사점

자율 주행 에이전트 개발자는 통신 단절 시 발생하는 Failsafe FSM의 즉각적 권한 탈취(Override) 현상에 각별히 유의해야 한다.

Offboard 제어 모드 중 ROS 2 노드의 CPU 스파이크나 Linux 커널의 스케줄링 지연으로 vehicle_command(또는 OffboardControlMode 토픽)의 발행이 0.5초 이상 지연되면, 시스템은 자율 에이전트의 경로 생성기를 즉결적으로 신뢰성 상실(Untrusted)로 낙인찍고 Failsafe 모드로 튕겨낸다. (GCS나 텔레메트리 단절보다 훨씬 엄격하다.)
이후 ROS 2 노드가 통신을 회복하더라도(Link Regain), PX4는 오퍼레이터가 개입하여(Human in the loop) 다시금 Offboard 모드를 재명령(SET_MODE)할 때까지는 현재의 Failsafe 복귀/배회 동작을 절대 자체적으로 취소하지 않는다(Latch Effect).

이 원칙은 로보틱스에서 “죽은 자가 부활하여 좀비처럼 통제 불능 상태로 다시 개입하는 현상“을 원천 차단하기 위한 통신 결함 허용의 가장 핵심적인 이산 사건 제어 공학 원리이다.
