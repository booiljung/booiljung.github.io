37.6.2. 메모리 풋프린트(Memory Footprint) 최소화를 위한 상태 플래그(State Flag) 추상화(Abstraction) 설계 비교

무인항공기(UAV)의 비행 제어기(Flight Controller)는 STM32 등 자원 제약이 심한 마이크로컨트롤러(MCU) 상에서 구동되는 임베디드 실시간 운영체제(RTOS) 환경이다. 따라서 GCS나 자율 에이전트와 교환해야 하는 무수히 많은 전역 상태(Global State)와 기체 제어 플래그(Control Flag)들을 메모리에 어떻게 적재하고 통신 채널에 어떻게 싣느냐(Packing)는 전체 시스템의 메모리 풋프린트(Memory Footprint)와 직결된다. 본 절에서는 PX4-Autopilot과 Ardupilot이 각각 어떠한 추상화(Abstraction) 기법을 사용하여 상태 플래그를 아키텍처 레벨에서 최적화하고 있는지 대조한다.

1. 임베디드 통신 환경에서의 상태 플래그(State Flag) 최적화 필요성

GCS 통제를 받는 UAV는 시동 상태(Armed/Disarmed), 통신 연결 상태(Link Status), 배터리 위험 수위, GPS 락(Fix) 상태, 각 센서의 이중화(Redundancy) 건전성 등 수십 가지의 불리언(Boolean) 기반 상태 플래그를 실시간으로 갱신하고 평가해야 한다.

만약 이러한 상태들을 각각 개별적인 바이트(Byte) 단위의 bool 변수나 복잡한 객체(Object) 형식으로 메모리에 상주시키고 MAVLink로 직렬화(Serialization)한다면, 다음과 같은 문제가 발생한다.

  • 메모리 단편화(Fragmentation) 및 패딩(Padding): 구조체(Struct) 내의 무분별한 bool 선언은 컴파일러의 메모리 정렬(Alignment) 규칙에 의해 미사용 패딩 바이트를 대량으로 양산한다.
  • 통신 대역폭(Bandwidth) 낭비: 무선 텔레메트리(Telemetry) 환경(예: 433MHz / 915MHz SiK Radio)은 매우 한정된 대역폭을 가지므로, 오버헤드가 큰 페이로드 구조는 텔레메트리 지연(Latency)을 유발한다.

2. PX4의 uORB 비트 필드(Bitfield) 기반 메모리 압축 패러다임

PX4-Autopilot은 데이터 지향 설계(Data-Oriented Design)에 기초한 uORB 마이크로 미들웨어를 핵심 통신망으로 사용한다. PX4는 메모리 최적화를 위해 상태 변환 플래그들을 구조체 내의 비트 필드(Bitfield) 또는 비트마스크(Bitmask) 정수형으로 강하게 압축(Packing)한다.

2.1 vehicle_status 및 센서 건전성 마스킹 (Masking)

PX4 소스 코드 내 상태를 관장하는 주요 메시지(예: vehicle_status.msg 또는 sensor_combined.msg)를 살펴보면, 복잡한 상태 조건들이 uint32_t나 uint64_t 하나의 변수 안에 비트 논리합(OR) 연산을 통해 누적된다.

// PX4의 서브시스템 상태 비트마스크 예시
uint32_t onboard_control_sensors_present; // 어떤 센서가 인식되었는가
uint32_t onboard_control_sensors_enabled; // 어떤 센서가 활성화되었는가
uint32_t onboard_control_sensors_health;  // 어떤 센서가 정상(Healthy)인가
  • 장점: 캐시 친화도(Cache Friendliness)가 극대화된다. MAVLink의 SYS_STATUS 메시지를 통해 GCS로 전송될 때도, 단 3개의 32비트 정수(총 12바이트)만으로 32여 종의 내부 센서 및 서브시스템(예: Gyro, Accel, Mag, GPS, VIO, RC 등)에 대한 ‘존재/활성/건전’ 상태를 동시에 전달할 수 있다. 이는 GCS 통신 대역폭 최적화에 결정적으로 기여한다.

3. Ardupilot의 객체 지향적(Object-Oriented) 상태 캡슐화 패러다임

Ardupilot은 C++의 객체 지향 프로그래밍(OOP) 개념을 임베디드 펌웨어에 광범위하게 적용하여 시스템을 추상화한다.

3.1 객체 내재화된 상태 플래그 구조 (AP_Vehicle 및 모듈)

Ardupilot의 비행 관리 구조는 거대한 전역 비트마스크를 공유하기보다는, 각각의 인스턴스화된 모듈 객체(예: AP_GPS, AP_Baro, AP_InertialNav)가 본인만의 상태 플래그를 클래스 멤버 변수(Member Variable) 형태로 은닉(Encapsulation)하는 철학을 선호한다.

예를 들어 GCS에서 특정 센서의 상태를 요구할 때, 메인 루프 관제부는 다음과 같이 특정 하위 센서 객체의 Getter 함수를 호출하여 상태를 취합한다.

// Ardupilot의 상태 조회 추상화 예시 (단순화)
if (ahrs.healthy() && gps.is_healthy()) {
    // 상태 조건 충족 로직
}

3.2 메모리 풋프린트와 유지보수성의 트레이드오프

이러한 OOP 기반 상태 캡슐화는 구조적 직관성과 각 모듈 단위의 테스트 유연성을 크게 높인다. 개발자는 특정 서브시스템의 상태 갱신 로직을 파악하기 위해 전체 전역 변수를 추적할 필요 없이 해당 클래스 내부만 살펴보면 된다. (유지보수성 극대화)
다만, 순수한 데이터 연속성 관점에서 보면 VTable(가상 함수 테이블) 유지 비용과 각 객체별로 분산된 멤버 변수 접근 시 발생하는 포인터 연산 오버헤드, 그리고 메모 정렬 패딩 등으로 인해 PX4 방식의 일괄 압축된 uORB 메시지 구조보다는 메모리 풋프린트가 상대적으로 크게 잡히는 특징이 있다.

4. GCS 상태 번역기(State Translator) 계층 구조 설계 전략

MAVLink 기반 중앙 관제 혹은 ROS2(uXRCE-DDS) 기반의 플릿 매니저 시스템을 구축할 때, 이기종 간의 메모리 최적화 철학 차이는 GCS 쪽 텔레메트리 파서(Parser) 구현 방식에 영향을 미친다.

  • PX4 타겟 플러그인(Plugin): GCS 측 상태 관측기는 비트 연산(& 연산자 기반 비트 디코딩) 중심의 경량 트랜잭션 함수들로 설계하여 CPU 오버헤드를 낮추는 전략을 취해야 한다. 하나의 32비트 변수 변화 이벤트를 통해 다중 UI 위젯(알림 패널, 센서 상태 표시등)을 일괄 갱신하는 발행-구독 모델이 유리하다.
  • Ardupilot 타겟 플러그인(Plugin): 기능 그룹별로 분할되어 들어오는 여러 종류의 MAVLink 메시지 단위(Message ID 기반 라우팅)로 파싱 함수들을 다형성(Polymorphism)을 활용해 매핑하는 것이 자연스럽다.

두 아키텍처 모두 극도로 제한된 자원을 효율화하기 위해 분투하지만, PX4는 데이터(Data) 지향 압축과 중앙집중식 버스 통신을 통해 “공간적” 효율(메모리 패킹)을 추구하였고, Ardupilot은 행위(Behavior) 지향 객체 캡슐화를 통해 “인지적” 효율(다형성 구조)을 확보하였다고 정리할 수 있다.
