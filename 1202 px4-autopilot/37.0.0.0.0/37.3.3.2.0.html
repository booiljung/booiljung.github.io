37.3.3.2. 통신 단절 시 자율 비상 상태(Failsafe State) 천이 임계값(Threshold) 설정의 제어 공학적 근거

무인항공기(UAV) 운영 중 지상 관제 시스템(GCS) 또는 반려 컴퓨터(Companion Computer)와의 데이터 링크(Data Link) 단절 시, 기체가 즉시 자율 비상 상태(Failsafe State)로 천이(Transition)하는 기준을 설계하는 것은 제어 공학과 시스템 안전 공학의 핵심 과제이다. PX4-Autopilot은 이 천이 임계값(Threshold)을 단순한 휴리스틱(Heuristics)이 아닌, 제어 대상(Plant)의 물리적 관성, 통신 채널의 노이즈 밀도(Noise Density), 그리고 센서 추정망의 오차 누적률(Drift Rate)을 종합적으로 고려하여 도출하였다.

1. 개방형 제어 루프(Open-loop) 상황 진입 시의 제어 공학적 한계

데이터 링크가 끊어진다는 것은 자율 에이전트(ROS 2)가 하달하는 오프보드 제어 위치 표적(Setpoint)이나 GCS 조종자의 원격 스틱 제어망이 끊어짐을 의미한다. 이때 기체의 외부 제어 루프 구조는 순간적으로 닫힌 루프(Closed-loop) 제판에서 피드백 없는 개방형 추종 주기(Open-loop)로 강제 변환된다.

  • 제어 유예 시간(\tau_{tol}) 동안 외부 명령이 도착하지 않으면, 기체의 현재 속도 및 가속도 벡터 요소가 댐핑(Damping)되지 못하고 운동 방정식에 따라 계속 적분된다.
  • 이로 인해 기하급수적인 위치 오차가 발생하며 구조물과 충돌(Crash)하게 된다.
  • 따라서 무결성 필터는 기체가 제어 불능 상태 한계(Control horizon limit)에 부딪히기 전, 즉 임계값 체공 시간(T_{thresh}) 에 도달하기 전에 내부 로컬 제어기 추종 모드(RTL 또는 Hold)로 Failsafe 천이를 인가해야만 한다.

2. PX4 데이터 링크 타임아웃(Timeout) 파라미터의 산출 근거

PX4는 비행 환경의 동역학적 특성에 따라 타임아웃 임계값 체계를 차등 분리하여 적용한다. (COM_DL_LOSS_T, COM_RC_LOSS_T, COM_OF_LOSS_T)

2.1 오프보드(Offboard) 저지연 임계값: 0.5초 (COM_OF_LOSS_T)

ROS 2 노드 등 외부 에이전트 컴퓨터는 근거리(주로 기체 내부 네트워크 배선) 통신을 전제로 10Hz~100Hz 이상의 고빈도로 세트포인트(Position/Velocity Setpoint)를 발행한다.

  • 근거: 에이전트의 경로 계획기(Path Planner)는 주로 정밀 장애물 회피(Obstacle Avoidance) 로직과 매핑된다. 따라서 이 신호가 단절되었다는 것은 로봇이 맹인(Blind) 상태로 고속 돌진 중일 확률이 높다는 뜻이다.
  • 수학적으로 기체가 10m/s로 순항 중일 때 0.5초만 제어를 잃어도 5m를 목표 궤적과 상관없이 활공한다. 좁은 문을 통과하는 등 극도로 정밀한 동적 튜닝에서는 이러한 변동 자체가 Failsafe Trigger를 요구하는 충돌 반경(Collision Radius) 내에 진입했음을 의미한다.
  • 결과적으로 Offboard의 Failsafe 임계값은 통신 프로토콜의 Jitter 여유 범위(\approx 0.5s) 상한에 딱 맞게 매우 빡빡하게 타게팅된다.

2.2 메인 텔레메트리 데이터 링크 임계값: 10초 (COM_DL_LOSS_T)

GCS와의 MAVLink 심박(Heartbeat) 통신인 경우, 이는 1~2Hz 주기로 통신 지연(Delay)이 긴 장거리 RF 모뎀(예: 900MHz 텔레메트리, LTE망)에 크게 의존한다.

  • 근거: 장거리 RF 망은 물리적 매체의 반사, 다중 경로(Multipath), 대기 전파 감쇠 등으로 인한 패킷 로스(Packet Loss)가 매우 빈번하다.
  • 만일 이를 1~2초로 짧게 설정하면(민감하게 반응), 기체는 수시로 비행 임무(Mission)를 포기하고 RTL을 켜며 이탈하는 비정상적인 신경질적 거동(Nervous behavior)을 보이게 된다. GCS의 역할은 실시간 하위 자세 제어가 아닌 상위 수준의 ’감시 및 관제(Supervision)’이므로, 기체 스스로 탑재된 모듈로 웨이포인트(Mission)를 임무 지향적으로 추종 중이라면 일시적인 Heartbeat 중단은 10초까지 용인(Tolerance)하는 것이 제어 설계상 합리적이다.

4. Ardupilot과 PX4의 통신 감도 스케줄링 패러다임 차이

Ardupilot은 FS_GCS_ENABL 및 FS_LONG_TIMEOUT 파라미터를 통해 유사한 구조를 제공한다. 다만 특징적인 부분은 Ardupilot이 특정 환경(예: 장기 비행 드론/고정익)에서 GCS 연결이 끊겨도 설정해 둔 “Mission Continue(계속 비행)” 옵션을 선호하는 경향이 있는 사용자 층을 의식해 Failsafe 회피 설정을 꽤 다양한 경로로 제공한다는 것이다.

반면, 멀티로터 산업용과 자율 비행(ROS) 중심이 근간인 PX4-Autopilot은 데이터 전송 보장에 훨씬 민감하다. 최근 도입된 Control Interface(예: FailsafeStateMachine)는 임계값 초과를 단순한 Flag 처리가 아니라, 내비게이션 상태(Nav State) 랭크 강등(Demotion)이라는 우선순위 큐(Priority Queue) 체계로 관리한다. 즉, GCS의 Heartbeat나 RC가 10초간 끊어지면, 개발자가 아무리 자율 비행 앱(App)을 돌리고 싶어 하여도 Commander가 최상위 인터페이스 수준에서 FSM의 근본 경로를 RTL(Return To Launch) 체계로 강제 오버라이드(Override)해버리는 철저한 이산 사건 중앙 검열(Centralized Censorship) 패턴을 견지한다.

이를 통해 PX4는 통신망 두절 시 발생할 수 있는 ‘눈먼 폭주 기관차’ 상태를 어떠한 응용 레이어 오류에서도 방어할 수 있는 가장 공학적으로 안전한 소프트웨어 아키텍처를 자랑한다.
