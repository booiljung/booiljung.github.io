37.6.1.1. Ardupilot의 표준 MAVLink 모드 열거형(Enumeration) 체계의 범용성과 한계성 분석

지상 관제 시스템(GCS)이 무인기(UAV)와 통신하여 비행 모드를 설정할 때, 가장 기저에 깔려 있는 프로토콜 표준은 MAVLink이다. Ardupilot은 MAVLink의 근본적인 설계 사상과 역사적 궤적을 함께 해온 오토파일럿이므로, 그 상태 제어 및 모드 변경 체계가 MAVLink의 기본 명세(Specification)와 매우 직관적으로 일치(1:1 결합)하는 특징을 가진다. 본 절에서는 Ardupilot이 채택한 표준 열거형(Enumeration) 기반 다이렉트 매핑 구조의 범용적 장점과 현대적 자율 비행 시스템 통합 관점에서의 아키텍처적 한계성을 분석한다.

1. MAVLink 표준 MAV_MODE 열거형과 Ardupilot의 기본 동작 구조

MAVLink 프로토콜은 기체의 상태와 모드를 정의하기 위해 전통적으로 MAV_MODE 형식을 사용해 왔다. HEARTBEAT 메시지에는 base_mode 필드와 시스템의 사용자 정의 모드를 나타내는 custom_mode 필드가 존재한다.

1.1 Ardupilot의 다이렉트 매핑(Direct Mapping) 방식

Ardupilot은 기체 유형(예: Copter, Plane, Rover)별로 독자적인 모드 열거형 집합(Set)을 가지고 있다. 예를 들어, ArduCopter의 경우 소스 코드(예: defines.h) 내에 비행 모드가 1차원적인 enum으로 명시되어 있다.

// ArduCopter 비행 모드 열거형 (단순화)
enum control_mode_t {
    STABILIZE = 0,
    ACRO = 1,
    ALT_HOLD = 2,
    AUTO = 3,
    GUIDED = 4,
    LOITER = 5,
    RTL = 6,
    // ...
};

GCS에서 모드 변경 명령(MAVLink의 COMMAND_LONG ➔ MAV_CMD_DO_SET_MODE)을 전송할 때, 매개변수 param1에 MAV_MODE_FLAG_CUSTOM_MODE_ENABLED를 나타내는 기본 모드 값을 주고 param2에 위 열거형의 정수 단위(예: RTL의 경우 6)를 그대로 삽입하여 전달한다. Ardupilot은 수신한 정수 값을 스위치-케이스(Switch-Case) 구문이나 함수 포인터 배열을 통해 내부 모터 제어 루프로 직접(Directly) 투영한다.

1.2 설계의 직관성과 범용성 장점

이 아키텍처의 가장 큰 강점은 높은 직관성과 극단적으로 낮은 추상화 장벽(Low Abstraction Barrier) 이다.

  • 간편한 GCS 개발: 미들웨어 혹은 GCS 개발자는 기체 타입과 모드 인덱스 번호만 알고 있으면 별도의 상태 번역기(State Translator) 없이도 기체를 제어할 수 있다.
  • 쉬운 디버깅: MAVLink 패킷 캡처 도구(예: MAVLink Inspector)에서 custom_mode 값이 6으로 떨어지면 즉각적으로 RTL 모드에 진입함을 인식할 수 있다.

2. 모놀리식(Monolithic) 모델의 한계성 분석

그러나 임무의 복잡도가 증가하고 외부 의존성(예: VIO, RTK GPS, Companion Computer)이 결합되면서 이러한 단일 1차원 배열(1D Array) 형태의 열거형 구조는 여러 한계점을 노출하게 되었다.

2.1 차원 폭발(Dimensional Explosion) 문제

Ardupilot 형식의 단일 열거형은 직교(Orthogonal)하는 상태 속성들을 모두 독립된 모드로 명명해야만 한다. 예를 들어, 동일한 위치 유지(Position Hold) 동작이라 하더라도, 회피(Avoidance) 처리가 내장된 모드인지, 기체 추종 기능(Follow Me)이 내장된 모드인지에 따라 새로운 번호를 지속적으로 추가해야 한다.
실제로 Copter의 경우 모드 번호가 20번대 중반을 넘어서고 있으며, Plane이나 Rover와 공유되지 않는 독자적인 상태 번호들로 인해 GCS-오토파일럿 간 일관성(Consistency) 유지 비용이 상승하고 있다.

2.2 컨텍스트 손상(Context Loss) 및 전환 불연속성

단일 모드 변경만으로 전체 시스템의 상태가 재결정되기 때문에, 전환 이전의 상태 히스토리(State History)나 백그라운드 태스크(Background Task)의 컨텍스트(Context)를 보존하기 어렵다.
임무(AUTO) 수행 중 일시 정지(Hold)가 필요할 때, Ardupilot은 임무 트래킹 로직의 상태를 전역적으로 일시 정지시킨 채 새로운 비행 모드(예: GUIDED 또는 LOITER)로 완전히 탈출(Escape)하게 된다. 이는 미들웨어나 상위 에이전트 관점에서 볼 때, 현재 기체가 “임무 중 일시 정지된 상태“인지, 아니면 “단순 공간 고정(Loiter) 상태“인지를 명확히 짚어내기 위한 별도의 플래그(예: MISSION_CURRENT) 교차 분석을 요구하게 만든다.

2.3 센서 계층 검증 로직과의 결합성

PX4는 모드 전이가 이루어질 때 전역 네비게이션 상태(Nav State) 머신이 센서 스트림의 유효성을 점검하는 철저한 중앙 통제식 메커니즘을 가진다. 하지만 Ardupilot의 모드 열거형은 전이가 각 모드 클래스의 생성자 혹은 init() 함수에 분산 의존하므로, 외부 MAVLink로 STABILIZE에서 AUTO로 명령을 내릴 때, 기체의 글로벌 포지션 유효성이 검사되는 시점과 모드가 수용되어 ACK를 보내는 시점 간의 미스매치(Mismatch) 엣지 케이스가 발생할 우려가 있다.

3. 결론 및 상위 호환 통신 설계의 요구

Ardupilot의 MAVLink 열거형 매핑은 QGroundControl이나 Mission Planner와 같은 보편적인 GCS를 연동할 때 직관성과 효율성을 제공한다. 하지만 비행 모드 자체가 시스템의 물리적 제어와 상태 전이 논리를 캡슐화한 1차원 구조에 머무르고 있기 때문에, 고도화된 자율 비행 군집이나 오프보드 연동(ROS2 DDS)을 구현하기 위해 미들웨어 설계자는 기체의 의도(Intent) 와 물리적 상태(Status) 간의 논리적 틈새를 매끄럽게 채워주는 두터운 추상화 계층(Abstraction Layer)을 개별적으로 구축해 주어야 한다.
