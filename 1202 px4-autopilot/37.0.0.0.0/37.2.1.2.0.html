37.2.1.2 상태 변수(State Variable)의 직교성(Orthogonality)과 데카르트 곱(Cartesian Product)을 통한 전체 상태 공간의 차원 해석

오토마타 이론에서 유한 상태 기계(FSM)의 설계 복잡도는 관리해야 할 상태 변수(State Variable)의 개수에 따라 지수적으로 증가한다. 이를 상태 폭발(State Explosion) 문제라 일컫는다. 다수의 센서와 액추에이터를 정밀 통제해야 하는 PX4-Autopilot 아키텍처는 이러한 폭발적 복잡성을 제어하기 위해 상태 변수들을 직교(Orthogonal)하는 부분 공간(Subspace)으로 설계하고, 데카르트 곱(Cartesian Product)을 통해 전체 시스템 상태 공간을 합성하는 수학적 구조론을 차용하였다. 본 절에서는 상태 공간의 차원 분리 원리와 그로 인한 소프트웨어 아키텍처의 안정성 향상 효과를 심층 분석한다.

1. 전역 상태 공간의 단일 차원화(Flattening)가 초래하는 한계

초창기 비행 제어 소프트웨어나 직관적인 절차적(Procedural) 타이머 로직에서는 기체의 모든 상태를 1차원의 열거형(Enum) 리스트로 평탄화하여 관리하려는 경향이 있었다.
예를 들어 STANDBY, ARMED_MANUAL, ARMED_AUTO, ARMED_RTL, FAILSAFE_MANUAL 방식처럼 모든 경우의 수를 하나하나 명명한다면, 모터 상태(2개), 비행 모드(10개), GPS 에러 상태(2개)의 단순 조합만으로도 40개의 거대한 상태 정점이 생성된다. 이러한 단일 차원 설계는 전이 함수 \delta 행렬을 기하급수적으로 팽창시키며, 특정 조건에서 처리 누락에 의한 버그나 비정상 종료(Segmentation Fault 등) 리스크를 기하급수적으로 키우는 태생적 결함을 내재한다.

2. 상태 변수의 직교성(Orthogonality) 확보

직교성(Orthogonality) 관점에서, 특정 부분 상태 공간의 변이는 다른 직교 상태 공간에 물리적인 간섭 함수를 유발하지 않아야 한다. PX4의 커맨더(src/modules/commander/Commander.cpp) 내부 로직은 주요 시스템 파라미터를 다음과 같이 직교하는 다차원 상태 변수로 분리하여 고립(Isolation)시킨다.

2.1 PX4의 3대 직교 부분 공간 (Orthogonal Subspaces)

  1. 무장 상태 벡터 (S_{arm}): {INIT, STANDBY, ARMED, SHUTDOWN}
  • 오직 ESC(Electronic Speed Controller)와 모터 계통을 통한 동력 활성화 여부만을 관장한다.
  1. 내비게이션 모드 벡터 (S_{nav}): {MANUAL, POSCTL, MISSION, RTL, OFFBOARD, 등}
  • Position, Velocity, Attitude 제어기가 어떤 목표 궤적(Setpoint) 추종 알고리즘을 수행할 것인지 결정한다.
  1. 안정성(Failsafe) 가이드 벡터 (S_{fail}): {NORMAL, BAT_LOW, GPS_LOSS, RC_LOSS, COMMS_LOSS}
  • 비정상 기체 센서 상황이나 통신 두절, 전원 이상 정보를 저장한다.

이 세 가지 벡터는 개별적으로 유지되는 독립적 오토마타(M_{arm}, M_{nav}, M_{fail})와 같이 거동하며, uORB 토픽 내에 각각의 uint8_t 플래그로 분산 관리된다.

3. 데카르트 곱(Cartesian Product)에 의한 전체 상태 추론

직교하는 부분 공간들이 조합되어 기체의 현재 동역학적 행위를 완벽하게 묘사하는 전역 시스템 상태 공간 S_{total}은 부분 공간들의 데카르트 곱(Cartesian Product)으로 정의된다.

S_{total} = S_{arm} \times S_{nav} \times S_{fail}

즉, 각 시점의 기체의 전체 상태 객체 s \in S_{total} 은 3-튜플(Tuple) 구조인 s = (s_{arm}, s_{nav}, s_{fail}) 로 매핑된다.

  • 장점 1: 제어 로직의 단순화.
    기체가 특정 비행 모드(s_{nav} = \text{MISSION})를 수행 중에 사용자가 킬 스위치(Kill Switch)를 발동시켰다고 가정할 때, 시스템은 전체 3-튜플을 점검하거나 내비게이션 로직을 수정할 필요 없이, 오직 무장 상태 벡터 S_{arm} 의 값을 \text{SHUTDOWN}으로 독립 전이시키는 1차원 연산만으로 기계적 출력을 완전히 차단할 수 있다.
  • 장점 2: 안전 필터(Safety Filter) 구축 용이성.
    Commander 로직의 가드(Guard) 조건 평가 시 직교 상태 변수들을 참조한다. 예를 들어 “현재 s_{arm} = \text{ARMED} 이고 s_{nav} = \text{POSCTL} 인데 s_{fail} 공간에서 \text{GPS\_LOSS} 이벤트가 수신되면, s_{nav} 벡터를 강제로 \text{ALTCTL}(고도 제어) 로 폴백(Fallback)시킨다“는 식의 선형적이고 계층적인 정책 치환(Policy Override) 코드를 매우 간결하게 작성할 수 있다.

도해에서 볼 수 있듯, 각 부분 공간 내의 변이(Transition) 사이클은 독립적으로 루프를 구성하지만(직교성), 상위 계층인 Failsafe 상태가 하위 내비게이션 상태에 전제 조건(Pre-condition) 논리로서 횡단 간섭(Cross-sectional Override)을 행사하며 전체 시스템 상태를 규정한다.

5. 결론

비행 제어 소프트웨어에서 상태 변수의 직교성을 확보하고 데카르트 곱으로 차적(Dimensional Product) 상태 공간을 합성하는 방식은, 방대한 분량의 스파게티 코드(Spaghetti Code) 생성을 억제하는 아키텍처적 방어선이다. PX4는 무장 상태, 비행 모드 상태, 그리고 예외 상태라는 명확한 직교 벡터 변수들을 차용하였고, 이를 통해 조종자와 GCS에게 극도로 정제되고 예측 가능한 형태의 MAVLink 인터페이스를 제공한다. ROS2 기반의 uXRCE-DDS 미들웨어 설계 시나 제어 알고리즘의 확장 이식을 준비할 때, 이러한 직교적 상태 변수 관리 논리를 침해하지 않고 독립성을 준수하는 것이 버그 없는 상호 연동(Seamless Integration)의 핵심 요건임을 명심해야 한다.
