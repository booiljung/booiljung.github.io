37.6. 이기종 아키텍처(PX4 vs Ardupilot)의 상태 제어 패러다임 비교 및 미들웨어 설계

무인항공기(UAV) 제어 시스템 아키텍처의 양대 산맥인 PX4-Autopilot과 Ardupilot은 근본적으로 상이한 상태 제어(State Control) 패러다임과 미들웨어(Middleware) 설계 철학을 채택하고 있다. 지상 관제 시스템(GCS, Ground Control Station)이나 자율 임무 에이전트(Autonomous Mission Agent)를 설계할 때, 이 두 이기종(Heterogeneous) 비행 제어기(Flight Controller)의 아키텍처 특성을 명확히 이해하고 추상화 계층(Abstraction Layer)을 구축하는 것은 플릿 관리(Fleet Management) 미들웨어 설계의 핵심 과제이다. 본 절에서는 PX4(펌웨어 버전 v1.14 기준)와 Ardupilot 간의 상태 처리 철학과 이를 조율하기 위한 미들웨어적 통신 아키텍처, QGroundControl(QGC, 버전 v4.3 기준)의 구조적 대응 방식을 심층적으로 분석한다.

1. 아키텍처 설계 철학 및 패러다임의 근본적 대비

1.1 단일 통합 루프(Ardupilot) vs 분산형 마이크로 아키텍처(PX4)

Ardupilot은 역사적으로 아두이노(Arduino) 기반의 단일 스레드(Single-threaded) 루프 아키텍처에서 진화해 왔다. 비록 현재는 실시간 운영체제(RTOS)인 ChibiOS 등을 도입하여 멀티 스레딩(Multi-threading)을 지원하지만, 코어 비행 제어 로직과 상태 전이 로직은 여전히 거대한 모놀리식(Monolithic) 객체 구조 안에 밀접하게 결합(Tightly-coupled)되어 있다. 상태 기계(State Machine)의 전이는 각 비행 모드 클래스의 내부 메서드(Method) 호출을 통해 암시적으로 이루어진다.

반면, PX4-Autopilot은 탄생 시점부터 병렬 분산 처리와 모듈화에 방점을 둔 마이크로 서비스 아키텍처(Microservices Architecture) 지향으로 설계되었다. NuttX RTOS 환경 위에서 독립된 프로세스(또는 스레드) 단위로 각 모듈이 구동되며, 모듈 간의 상태 전이 및 데이터 교환은 전적으로 발행-구독(Publish-Subscribe) 기반의 uORB(Micro Object Request Broker) 통신 미들웨어를 통해 비동기적으로 수행된다.

1.2 상태 제어(State Control)와 모드(Mode) 결정 메커니즘 차이

PX4는 기체의 전역 상태(Global State)인 vehicle_status와 사용자가 요청하는 이동 의도인 네비게이션 상태(Navigation State), 그리고 이들을 조율하여 최종 허가를 결정하는 커맨더(Commander) 모듈을 엄격히 분리한다. PX4의 상태 제어 패러다임은 명시적인 방향성 비순환 그래프(DAG, Directed Acyclic Graph)와 유사한 형태를 띠며, 모든 모드 변경 요청은 commander.cpp의 철저한 사전 조건(Precondition) 검사를 통과해야만 허용된다.

Ardupilot은 비행 모드(Flight Mode) 자체가 제어의 주체가 되는 캡슐화(Encapsulation) 특성을 가진다. 각 비행 모드(예: ModeLoiter, ModeAuto)가 부모 클래스인 Mode를 상속받아 고유의 진입조건(Enter)과 갱신(Run) 로직을 처리한다. MAVLink 명령이 들어오면 해당 루프 내에서 모드 객체의 init() 메서드를 호출하고, 반환값이 true이면 상태가 즉시 전이되는 방식을 취한다.

2. 미들웨어 설계 및 ROS2/MAVLink 연동 관점 차이

이기종 플랫폼을 통합하는 상위 에이전트나 GCS 미들웨어 관점에서는, 각 오토파일럿이 외부의 명령(MAVLink, ROS2 DDS)을 수용하는 방식의 차이를 반드시 이해해야 한다.

2.1 외부 제어 권한 위임: Offboard vs Guided 모드

로봇 운영체제(ROS 2)나 자율 에이전트에 의한 컴퓨터 비전 추적, 강화학습 기반 경로 계획(Path Planning) 등의 기능을 수행할 때, 오토파일럿은 외부 명령 수용 모드로 넘어가야 한다.

  • PX4의 Offboard 모드: PX4는 실시간 외부 제어를 위해 특화된 Offboard 모드를 제공한다. 오프보드 모드로 진입하기 위해서는 반드시 사전에 주기적인 외부 제어 명령(예: SET_POSITION_TARGET_LOCAL_NED) 셋포인트(Setpoint) 스트림이 2Hz 이상 지속적으로 수신되고 있어야 한다는 엄격한 록인(Lock-in) 조건이 존재한다. 이는 통신 단절(Data Link Loss) 시 예측 불가능한 거동을 막기 위한 강건한 방어적 설계다.
  • Ardupilot의 Guided 모드: Ardupilot은 동일 목적을 위해 Guided 모드를 사용한다. 가이디드 모드는 PX4 대비 진입 조건이 상대적으로 유연하며, 명시적인 타임아웃(Timeout) 전까지 마지막으로 수신된 웨이포인트(Waypoint)나 속도 벡터를 자율적으로 추종하는 조건부 허용(Conditional Acceptance) 패러다임에 가깝다.

2.2 MAVLink 미들웨어 데이터 동기화 패러다임

QGroundControl과 같은 GCS 애플리케이션이나 온보드 미들웨어를 구축할 때, 오토파일럿의 상태를 추적하기 위해 MAVLink의 심장박동(HEARTBEAT) 메시지 및 상태 메시지(SYS_STATUS)를 동기화해야 한다.

3. QGroundControl(QGC) 내 이기종 관리를 위한 파사드 패턴(Facade Pattern)

QGroundControl은 백엔드(C++) 아키텍처에 PX4 및 Ardupilot 모두를 효과적으로 지원하기 위해 객체 지향의 파사드 패턴(Facade Pattern) 및 어댑터 패턴(Adapter Pattern) 기반 펌웨어 플러그인(Firmware Plugin) 아키텍처를 채택하고 있다.

3.1 펌웨어 추상화 계층(Firmware Abstraction Layer)

QGC 소스 코드 내의 FirmwarePlugin 클래스는 기체의 기본 동작(시동, 모드 변경, 매개변수 동기화)을 추상화한다. 그리고 이를 상속받는 구체적 클래스인 PX4FirmwarePlugin과 APMFirmwarePlugin을 통해 이기종 비행 제어기 각각에 특화된 상태 변환 및 명령 생성 규칙을 캡슐화한다.

예를 들어, GCS에서 “복귀(Return to Launch)” 버튼을 클릭했을 때, QGC의 통신 미들웨어는 추상화된 메서드를 호출하지만, 실제로 전송되는 MAVLink 패킷은 이기종 펌웨어의 규격에 따라 다르게 빌드된다. Ardupilot의 경우 MAV_MODE 열거형 내의 RTL 모드로의 전이 명령이 전파되는 반면, PX4의 경우 커스텀 모드 비트마스크 연산을 통해 custom_mode 필드에 특정한 AUTO.RTL 매핑 값이 실리게 된다.

4. 미들웨어 아키텍처 설계 시 고려사항

스웜(Swarm) 비행관제나 이기종 플릿 통합 관리를 위한 미들웨어를 개발할 때 다음의 아키텍처 제약 사항을 수용해야 한다.

  1. 비대칭적 오류 허용도(Asymmetric Error Tolerance): PX4는 상대적으로 엄격한 상태 확인을 요구하고 예외 발생 시 회복(Failsafe) 로직으로 강제 전이시키는 성향이 강하다. 미들웨어는 PX4 제어 시 상태 관측기(State Observer) 루프 주기를 더 짧게 가져가고 명령 거부(NACK)에 대비한 재시도 큐(Retry Queue) 로직을 견고히 설계해야 한다.
  2. 공통 상태 매개변수 정규화(Normalization): 각기 다른 센서 축(Sensor Axis) 기준(예: 좌표계)과 자이로/가속도 센서 교정 명령(MAV_CMD_PREFLIGHT_CALIBRATION) 파라미터가 다르게 매핑되므로 미들웨어 내에서 데이터 정규화 및 변환을 수행하는 추상화 계층 노드(ROS2의 경우 tf2 및 어댑터 노드) 배치를 강제해야 한다.

결론적으로, PX4-Autopilot은 독립성에 기반한 엄격한 메시지 교환 아키텍처(uORB 중심)를, Ardupilot은 객체 내부 결합도 기반의 실용적 아키텍처를 채용하고 있으므로, 이를 수용하는 GCS나 미들웨어 개발 시에는 디자인 패턴을 활용하여 이 이질적(Heterogeneous) 상태 공간을 단일 논리(Unified Logic)로 투영하여야 한다.
