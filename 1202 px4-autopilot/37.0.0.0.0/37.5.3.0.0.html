37.5.3. 데이터베이스 트랜잭션 기법을 차용한 2단계 커밋(Two-Phase Commit, 2PC) 기반 안전 롤백(Safe Rollback)

본 절에서는 무인항공기(UAV)의 비행 상태 제어기 모듈(Commander)이 지상 관제 시스템(GCS)으로부터 전달받은 고수준(High-level)의 상태 천이 명령을 안전하게 적용하기 위해 차용한 2단계 커밋(Two-Phase Commit, 2PC) 아키텍처를 분석한다. 이는 분산 데이터베이스 컴퓨터 과학에서 기원한 검증론적 기법으로, 비결정론적 센서 오류나 소프트웨어 예외 상황으로 인해 기체의 내적 결함(Inconsistency) 현상이 초래되는 것을 엄격히 봉쇄하고 이전의 안정된 상태로 회귀하는 안전 롤백(Safe Rollback) 메커니즘의 기반 토대가 된다.

설계적 관점의 본문은 모듈식(Modular) 유한 상태 기계(FSM) 아키텍처가 확립된 PX4 Firmware v1.14 이상의 시스템 및 내부 VehicleStatus 갱신 로직망을 기준으로 전개된다.

1. 분산 및 이기종 서브시스템 간 동기화의 본질적인 어려움

UAV 오토파일럿 보드(예: Pixhawk 시리즈) 내부에서 구동되는 PX4 펌웨어는 단일한 단일체(Monolithic) 프로그램이 아니다. uORB(Micro Object Request Broker) 비동기 메시징 버스를 통해 수십 개의 독립된 스레드 모듈(센서 드라이버, 자세 제어기, 위치 제어기, 네비게이터, 시동 관리자 등)이 각자의 주기에 맞춰 맞물려 돌아가는 고도의 비동기 분산 시스템(Asynchronous Distributed System) 이다.

여기서 GCS가 “현재 상태(Position 모드)에서 미션 자동 비행(Mission Auto 모드)으로 전환하라“는 단일 명령을 발령했다고 가정해보자. 이 하나의 천이(Transition) 명령이 무결성을 띠고 성공적으로 적용되기 위해서는 다음 하위 모듈들의 상태가 동시다발적으로 준비되어 있어야 한다.

  1. Estimator(EKF2): “글로벌 3D 위치 추정이 유효한가?” (GPS Fix 보증)
  2. Navigator: “메모리에 유효한 Waypoint 미션 아이템이 업로드되어 있는가?”
  3. Sensor Hub: “비행에 치명적인 나침반 교란(Magnetic Interference) 경고가 없는가?”

만약 커맨더 모듈이 위 조건들을 사전에 철저히 묶어 검증하지 않은 채, 섣불리 일부 모듈(예: LED 모드 표시기 변경, 제어 루프 스위칭)에만 먼저 상태 천이를 지시(Commit)해 버렸다가, 뒤늦게 Navigator 모듈이 “미션 데이터가 없다“며 파업을 선언하는 아찔한 상황이 도래한다면 기체는 어떻게 될까?
기체는 앞으로 가야 할지(Mission), 제자리에 있어야 할지(Position)를 모르는, 공중에서 제어권의 주인이 소실된 ‘상태 교착불능(State Deadlock/Schizophrenia)’ 의 파국적 상태에 빠져 추락하게 된다.

2. 2단계 커밋(Two-Phase Commit, 2PC) 아키텍처 설계망

이러한 모듈 간의 파괴적 비동기 충돌을 원천 차단하기 위해, PX4 커맨더(Commander) 내부의 set_main_state() 혹은 arming_state_transition() 등의 마이크로 트랜잭션 함수들은 전통적인 DB 공학의 2단계 커밋(2PC) 알고리즘을 강건하게 모방 차용하였다.

2.1 제 1위상 (Phase 1): 준비 및 투표(Voting & Preparation Phase)

GCS로부터 상태 천이 트랜잭션 요청이 도달하면, 커맨더 노드는 코어 상태 변수를 결코 즉시 변경하지(Write) 않는다. 대신 해당 천이에 얽힌 모든 의존성 하위 모듈과 센서 헬스(Health) 플래그를 대상으로 “너희들은 이 새로운 상태로 진입할 준비(Readiness)가 되었는가?” 하고 일제히 사전 검증(Pre-Check / Pre-flight Check) 투표를 징수한다.

이 위상에서는 임시 메모리(캐시) 상에서만 논리 연산이 이루어진다.

  • 만일 EKF2 모듈이 “GPS 정밀도가 HDOP 임계치를 초과하여 Auto 비행을 보증할 수 없다“고 부정(Veto/Abort) 표를 던지면, 1위상 투표는 즉시 부결된다.
  • 커맨더는 곧장 트랜잭션을 전격 파기하고, GCS 측에 MAV_RESULT_TEMPORARILY_REJECTED 통보를 하달하여 무결성(Integrity)을 온전히 수호한다.

2.2 제 2위상 (Phase 2): 확정 실행(Commit) 및 원자적 브로드캐스트

모든 참여 모듈이 “준비 완료(Ready to Commit)“라는 만장일치 회신을 보내왔다면, 커맨더 노드는 비로소 2위상으로 기어를 변속한다.
커맨더는 내부 자료구조인 VehicleStatus 구조체의 nav_state, arming_state 멤버 변수 필드를 전격적으로 동시 덮어쓰기(Overwriting) 한다. 그리고 업데이트된 이 중앙 상태 명세 구조체를 uORB 버스에 “단 1회의 원자적(Atomic) 토픽 발행(Publish)” 행위로 브로드캐스트 해버린다.

uORB 버스를 청취(Subscribe)하고 있던 비행 제어기, 네비게이터, 액추에이터 믹서 모듈들은 이 확정된 글로벌 상태 토픽을 동시에 수신하고 각자의 제어 루프를 스위칭한다. 중간에 특정 모듈만 갱신이 누락되는 부분 커밋(Partial Commit)이 일체 발생하지 않는 원자성(Atomicity: All-or-Nothing) 을 완벽하게 구현해 낸 것이다.

3. 잔류 불안정성과 안전 롤백(Safe Rollback/Fallback) 회귀 로직

2단계 커밋(2PC)이 제아무리 완벽히 성공하여 제2위상(Commit)을 통과했더라도, 런타임 환경에 노출된 드론은 실행 직후 예측 불허의 동역학적 이변을 마주할 수 있다.
예를 들어, Auto-Mission 모드로의 트랜잭션 커밋까지 완벽하게 성공하여 1번 웨이포인트(Waypoint)로 날아가기 시작한 지 단 3초 만에, 거대한 구조물 그늘에 진입하며 GPS 전파를 상실(GPS Loss)해 버린 상황을 상정해 보자.

이제 기체는 Auto 모드를 유지하기 위한 전제 조건(위치 피드백)이 사후적으로 파괴되었으므로, 불완전 궤적 비행(Drifting)이라는 런타임 예외 결함에 봉착한다. 이때 PX4 Failsafe 예외 처리기(Exception Handler)는 안전 롤백(Safe Rollback) 및 회귀(Fallback) 루틴을 무자비하게 발동시킨다.

  • PX4 예외 관리자는 현재 모드(Auto)의 유지가 불가능함을 포착 즉시 해당 트랜잭션을 소프트웨어적으로 Abort (철회) 시킨다.
  • 이후 하드코딩된 직전 최상위 안정 상태(Prior Stable State Regression) 혹은 Failsafe 트리거 맵을 참조한다.
  • GPS를 상실했으므로 포지션 제어(Position Control) 역량이 상실되었다 치부하고, 오직 상하 기압계 고도 제어 및 자세(Attitude) 제어만으로 허공에 체공 버티기가 가능한 Altitude mode 혹은 최악의 경우 다 집어던지고 조종기 스틱에 전적으로 의존하는 Manual mode 로 FSM 핀을 강제 회귀(Fallback Reversion)시킨다.
  • 결과적으로 기체는 치명적 폭주를 모면하고, 에너지를 허공에 보존하며 관제사(인간)의 구조적 개입을 애타게 기다리는 ‘안전한 교착(Safe Hovering)’ 상태로 착륙 내지 롤백하게 된다.

4. PX4 상태 천이 2PC 및 Rollback 아키텍처 (Mermaid Diagram)

UAV 내부 커맨더(Commander) 통제 하에 2PC(Two-Phase Commit) 사전 투표와 사후 안전 롤백 검증이 이루어지는 내부 메커니즘을 아래의 시스템 흐름도로 명세화한다.

graph TD
    A[GCS로부터 MAVLink 기반 모드 전환 명령 수신] 
    --> B[Commander 내부 대기열에서 트랜잭션 개시]
    
    subgraph Phase 1: Preparation & Voting
        B --> C{하위 모듈 상태 Pre-flight 센서/논리 검증}
        C -- GPS/추정기/배터리 등 의존성 모듈 VETO 부결 --> D[명령 파기 및 GCS에 NACK 반환]
        C -- 모든 하위 노드 READY 만장일치 가결 --> E[Phase 2 진입 인가]
    end
    
    subgraph Phase 2: Atomic Commit
        E --> F[내부 메모리 VehicleStatus 벡터 일괄 변경]
        F --> G[uORB 버스를 통한 단일 Atomic 상태 브로드캐스트 퍼블리싱]
        G --> H[전체 시스템 스레드 동시다발적 제어 루프 전환 완료]
        H --> I(GCS에 MAV_RESULT_ACCEPTED 발송 통보)
    end
    
    subgraph Runtime Fault & Safe Rollback Guard
        H --> J{런타임 실행 도중 치명적 센서 단절/예외 탐지 (e.g. GPS 상실)}
        J -- 안정 유지 --> K(정상 제어 순항 지속)
        J -- 치명적 런타임 결함 포착 --> L[안전 롤백 트리거: 직전 상태 파훼 선고]
        L --> M[Failsafe/Fallback 매핑 테이블 참조]
        M --> N[Altitude/Manual 혹은 비상 RTL 등 '최우선 안정 상태(Stable State)'로 강제 FSM 회귀 천이]
    end

이 흐름도는 이질적이고 다발적인 소프트웨어 모듈이 동시에 상호작용하는 맹렬한 실시간(Real-time) 비행 환경 속에서, 상태 기계의 무결성을 어떠한 논리 공학적 계층으로 보호하고 포디움(Podium)을 확립지에 두는지 전형적으로 예시한다.

5. 결론 및 요약 도출

고지능화된 상용 비행 에이전트 환경에서 무인기의 유한 상태 기계(FSM) 천이를 일차원적 변수 조작 수준(Spaghetti Variable Assignment)으로 방치하는 것은 곧장 추락 사고의 비상벨을 누르는 것과 동일하다.

이를 극복하기 위해 PX4-Autopilot의 코어 아키텍트는 분산 데이터베이스 이론에서 증명된 2단계 커밋(2PC) 패러다임을 차용하여, 모든 상태 천이 트랜잭션을 엄격한 ’사전 준비 득표(Voting)’와 ’원자적 동시 일괄 적용(Atomic Commit)’의 거푸집 틀에 가두었다. 더욱이 런타임 비행 중 기체 하드웨어의 예측 불허한 결함 발생 시에는, 불완전해진 현재의 상태 벡터를 파기하고 즉각적으로 에너지를 유지 보존할 수 있는 가장 가까운 과거의 제어 권역 모드로 비상 롤백(Safe Rollback/Fallback) 시키는 위상 보존 메커니즘을 견고히 이식해 내었다.

이러한 고도의 트랜잭셔널(Transactional) 마이크로 아키텍처 설계 사상은, 통신과 센서 인프라가 붕괴되는 처참한 조건(Degraded Environment) 내에서도 무인기가 파괴적 폭주를 일삼지 않고 조용히, 그리고 예측 가능한(Predictable) 비행 정지 혹은 회귀 상태로 되돌아올 수 있게끔 만드는 자가 면역 안전망의 정수로 평가받는다.
