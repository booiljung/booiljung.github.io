37.2.2 커맨더(Commander) 모듈의 다중 스레드(Multi-thread) 환경 내 상태 동시성(Concurrency) 제어

PX4-Autopilot은 단순한 단일 루프(Single-loop) 마이크로컨트롤러 프로그램이 아니라, NuttX RTOS(Real-Time Operating System) 위에서 수십 개의 프로세스와 스레드(Thread)가 병렬로 구동되는 고도의 동시성(Concurrency) 시스템이다. MAVLink 통신 스레드, 센서 수집 스레드, 내비게이션 스레드가 동시에 기체의 상태(State)를 읽거나 변경하려 시도할 때, 동시성 제어가 엄밀히 이루어지지 않으면 치명적인 경쟁 상태(Race Condition)가 발생한다. 본 절에서는 PX4 커맨더 모듈이 다중 스레드 환경에서 어떻게 시스템 상태의 무결성을 지켜내는지 그 동시성 통제 원리를 분석한다.

1. 다중 스레드 환경에서의 상태 관리 리스크

현대의 비행 제어 시스템은 분산형 아키텍처이므로 상태 변경 요청이 여러 스레드로부터 비동기적(Asynchronous)으로 폭주할 수 있다.

  • Race Condition 예시: MAVLink Receiver 스레드에서 GCS의 ‘MISSION’ 시작 명령에 따라 상태 변수를 AUTO_MISSION으로 덮어쓰려 하는 정확히 같은 클럭(Clock) 사이클에, Sensors 스레드에서 배터리 고갈을 감지하여 상태 변수를 AUTO_RTL로 덮어쓰려 시도할 수 있다. 두 스레드가 보호받지 않는 공유 메모리(Shared Memory)의 상태 변수에 동시 접근할 경우, 원인 불명의 상태 천이 지연이나 시스템 크래시(Crash)가 유발될 수 있다.

2. ‘단일 진실 공급원(Single Source of Truth, SSOT)’ 패러다임

PX4는 이러한 경쟁 상태를 방지하기 위해, 전역 기체 상태를 직접 수정할 수 있는 권한을 오직 단 하나의 스레드 즉, 커맨더(Commander) 메인 스레드에만 독점적으로 부여한다.

2.1 상태 변이 요청(Request)의 분산, 처리(Execution)의 집중

다른 모든 모듈(센서, 통신, GPS 등)은 기체의 상태를 직접 덮어쓰지 못한다. 대신 uORB 메시징 버스를 통해 vehicle_command 또는 상태 변경 “요청(Request)” 메시지만을 발행(Publish)한다.

  • 예측 가능성: 커맨더 모듈 내의 메인 FSM 루프는 주기적(보통 수백 Hz)으로 깨어나 이 요청 큐(Queue)들을 하나씩 순차적으로(Sequentially) 읽어들인다.
  • 우선순위 역전(Priority Inversion) 방지: 커맨더 내부에서 요청들을 평가할 때, 사전에 정의된 상태 위계(Hierarchy) 및 인터락(Interlock) 논리에 따라 Failsafe (RTL) 요청을 GCS Mission 요청보다 우선 처리한다. 이 구조는 스레드 수준의 동시성을 단일 스레드 버퍼 처리로 직렬화(Serialization)함으로써 상태 무결성을 달성한다.

3. 락 프리(Lock-free) 설계와 uORB 메시지 지연 최소화

NuttX RTOS 상에서 스레드 간 상호 배제를 위해 고전적인 세마포어(Semaphore)나 뮤텍스(Mutex)를 전역 상태 변수에 적용할 수도 있으나, PX4는 이를 지양한다.
과도한 pthread_mutex_lock()의 사용은 다음과 같은 실시간 시스템의 치명적 결함을 낳는다:

  • 블로킹(Blocking) 지연: 상태를 읽으려는 비행 제어기(Position Controller) 스레드가 커맨더 스레드의 락(Lock) 해제를 기다리며 멈춰버릴 수 있다. 이는 제어 주파수(Control Frequency)의 지터(Jitter)로 직결된다.

대신 PX4는 uORB(Micro Object Request Broker)라는 락 프리(Lock-free) 기반의 발행-구독(Publish-Subscribe) 미들웨어를 활용한다.

  • 커맨더는 변경된 최종 상태를 vehicle_status 토픽에 원자적(Atomic)으로 퍼블리싱한다.
  • 다른 제어 스레드들은 이 토픽의 버퍼를 뮤텍스 블로킹 없이 빠르게 폴링(Polling)하거나 파일 디스크립터(File Descriptor)를 통해 비동기로 읽어 들임으로써, 언제나 지연 없는 스레드 세이프(Thread-safe) 상태 읽기가 가능하다.

5. 결론

다중 스레드 환경에서 소프트웨어의 신뢰성은 동시성 리소스 관리에 달려 있다. PX4-Autopilot의 커맨더 모듈은 락(Lock) 메커니즘을 남용하여 발생할 수 있는 교착 상태(Deadlock) 및 우선순위 역전 리스크를 배제하고, uORB 미들웨어 기반의 ‘단일 진실 공급원(SSOT) 및 요청 버퍼링’ 구조를 택했다. 이러한 상태 동시성 제어 아키텍처 덕분에 PX4 기체는 외부의 GCS 자율 비행 명령과 내부의 하드웨어 센서 경고가 동시에 몰아치는 극한의 엣지 케이스(Edge Case) 환경 속에서도 스레드 충돌 없이 일관된 비행 상태(State) 판단을 내릴 수 있다.
