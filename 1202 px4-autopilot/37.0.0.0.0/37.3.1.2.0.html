37.3.1.2. 선형 시제 논리(Linear Temporal Logic, LTL)를 이용한 비행 모드 시퀀스 무결성 검증

선형 시제 논리(Linear Temporal Logic, LTL)는 시간의 흐름에 따라 변화하는 시스템 상태(States)와 이벤트(Events) 간의 인과적 및 시계열적 제약 조건을 수학적으로 엄밀하게 форма화(Formalization)하는 기법이다. PX4-Autopilot의 핵심 제어 모듈인 커맨더(Commander) 내부에는 이러한 시제 논리적 속성이 내재된 형태의 상태 천이 검증(State Transition Validation) 로직이 구현되어 있어, 조작자나 자율 에이전트로부터 하달되는 비행 모드 시퀀스(Flight Mode Sequence)의 무결성을 엄격하게 보증한다.

1. 비행 모드 시퀀스의 LTL 규격화(Specification)

UAV의 제동 및 비행 과정은 임의의 무작위 상태 천이를 허용하지 않으며, 특정 모드(Mode)에 진입하기 위해서는 반드시 만족되어야 할 시간적 선행조건(Temporal Pre-conditions)이 연쇄적으로 존재한다. LTL 연산자를 활용하여 PX4의 주요 비행 모드 트랜잭션을 규격화하면 다음과 같다.

  • 항상(Always, \square): 시스템 안전 불변성(Safety Invariant)
  • \square (\text{Mode} == \text{Mission} \implies \text{Global Position Valid})
  • 시스템이 임무(Mission) 모드에 있는 동안에는 “항상” GPS나 RTK 기반의 전역 위치 식별(Global Position Estimate)이 유효해야 한다. 이 조건이 위배되는 순간 시스템은 즉각 Hold 모드나 Failsafe로 강제 천이된다.
  • 결국(Eventually, \lozenge): 상태 도달 보증(Liveness)
  • \text{Command}(\text{Takeoff}) \implies \lozenge_{\le T_{max}} (\text{State} == \text{In_Air} \land \text{Altitude} \ge h_{min})
  • 이륙 명령이 인가되었다면, 기체는 제한 시간 T_{max} 내에 “반드시” 공중(In_Air) 상태에 도달하고 최소 고도를 확보해야 한다.
  • 다음(Next, \bigcirc): 원자적 트랜잭션 검증
  • 비행 모드의 직접적인 변경(예: Manual \rightarrow Offboard)을 요청할 때, 다음 Tick에서의 \bigcirc (\text{Mode} == \text{Offboard}) 트랜잭션이 유효하기 위해서는 이전 Tick들에서 Setpoint 메시지가 연속적으로 유입되고 있었어야 한다는 시계열적 의존성이 적용된다.
  • ~할 때까지(Until, \mathcal{U}): 상태 유지 제약
  • \text{Arming State} \ \mathcal{U} \ (\text{Landed} \land \text{Throttle} \le \text{Idle})
  • 기체가 완전 착륙 상태로 식별되고 스로틀이 아이들 임계치 이하로 지속될 때“까지“ 시동 상태(Armed)는 해제되어서는 안 된다.

2. PX4 Commander 모듈의 시퀀스 무결성 검증 아키텍처

PX4 커맨더 로직(Commander.cpp)은 LTL의 속성을 소프트웨어적 if-then-else 가드(Guard) 트리와 타임스탬프(Timestamp) 비교 구문으로 구현하여 제어 루프를 방어한다.
GCS나 ROS 2 에이전트가 MAV_CMD_DO_SET_MODE 커맨드를 전송하면, 내부의 상태 판독기(State Evaluator)는 해당 모드 전환을 즉시 수용하지 않고, 현재 FSM 스택의 과거 이력(History) 및 센서 유효성 플래그(Validity Flags)들의 타임스탬프를 대조한다.

2.1 Ardupilot과의 LTL 모델링 차이

Ardupilot 역시 비행 모드 전환 시 set_mode() 함수 내부에서 상태 검사를 철저히 수행하지만, 센서 필터의 상태와 모드 전환 로직이 C++ 클래스 내에 절차적(Procedural)으로 결합되어 있다. 반면, PX4는 vehicle_status 및 vehicle_control_mode uORB 토픽을 통해 마이크로 모듈 간 이벤트를 퍼블리싱함으로써, 상태와 제어를 분리(Decoupling)하고 LTL 기반의 정형 검증(Formal Verification) 도구인 SPIN이나 NuSMV 등을 적용하여 시스템 안정성을 수학적으로 증명하기에 더 적합한 이산 사건 모델(Discrete Event Model)을 제공한다.

3. LTL 규칙 기반의 Offboard 모드 전환 검증 (Mermaid)

자율 에이전트(Companion Computer)가 개입하는 Offboard 모드 전환은 LTL 기반 시퀀스 검증의 가장 대표적인 사례이다. “목표 제어값(Setpoint)이 일정 시간(>0.5s) 동안 연속 퍼블리시되지 않으면 Offboard 천이는 반려된다“는 시제 논리 규칙이 강제된다.

    direction TB
    
    state "Offboard Command Received (k=t0)" as CmdRx
    state "Verify Temporal History (t0 - 0.5s <= t <= t0)" as VerifyHist
    state "Accept Transition (k=t0+1)" as Accept
    state "Reject Transition (MAV_RESULT_TEMPORARILY_REJECTED)" as Reject
    
    [*] --> CmdRx
    CmdRx --> VerifyHist
    
    state VerifyHist {
        direction LR
        Check1[Check Setpoint T_stamp] --> Check2{Continuously updated?}
        Check2 -- Yes --> LTL_Sat[LTL Condition Satisfied]
        Check2 -- No --> LTL_Unsat[LTL Condition Violated]
    }
    
    VerifyHist --> Accept : if LTL_Sat
    VerifyHist --> Reject : if LTL_Unsat
    
    Accept --> Offboard_Mode_Active
    Reject --> Maintain_Current_Mode

4. 모드 시퀀스 무결성 위반 시의 예외 복구 구조 (Failsafe)

만약 비행 중 센서 고장이나 통신 단절 등으로 인해 시스템 안전 불변성(\square) 규칙이 위배되는 사건(Event)이 발생하면 어떻게 되는가?
PX4는 해당 위반을 GCS로 단순히 에러 메시지를 통보하는 것에 그치지 않는다. 커맨더의 NavigationState 우선순위 스케줄러가 개입하여, 오퍼레이터의 현재 명령보다 상위 권한을 갖는 강제 모드 전환(\text{Mission} \rightarrow \text{Hold} \rightarrow \text{RTL})을 트리거한다.

즉, LTL의 불변성 수식이 거짓(False) 판정을 받는 즉시, 시스템은 위상적으로 가장 안전한 거리에 위치한 예외 복구(Failsafe) FSM 노드로 점프(Jump)하는 안전 장치(Safety Interlock)가 발동되는 것이다. 이러한 구조 덕분에 외부의 악의적 해킹 펄스나 조종 컨트롤러의 일시적 오류에도 PX4-Autopilot은 스스로의 상태 무결성을 수호(Defend)할 수 있다.
