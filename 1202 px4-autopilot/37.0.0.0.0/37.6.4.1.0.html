37.6.4.1. 어댑터 패턴(Adapter Pattern) 및 파사드 패턴(Facade Pattern)을 활용한 상태 추상화 계층(Abstraction Layer) 구축

이기종 플릿(Heterogeneous Fleet)의 통신 복잡도를 효과적으로 제어하기 위해 GCS 미들웨어 설계자들은 객체 지향 프로그래밍(OOP)의 고전적인 구조 패턴인 **어댑터 패턴(Adapter Pattern)**과 **파사드 패턴(Facade Pattern)**을 조합하여 기체 상태 추상화 계층(Vehicle State Abstraction Layer)을 구축한다. 이 아키텍처는 하위 MAVLink 프로토콜의 방언(Dialect) 종속성을 끊어내고 다형성(Polymorphism)을 극대화하는 중추적 역할을 수행한다.

1. 하위 복잡성 은닉을 위한 어댑터 패턴(Adapter Pattern)

어댑터 패턴의 핵심은 특정 펌웨어(PX4, Ardupilot)가 내뿜는 호환되지 않는 기본 상태 변수들을 중간에서 한 번 가로채어, 미들웨어가 이해할 수 있는 단일화된 공통 인터페이스(Common Interface)로 변환해 주는 것이다.

1.1 추상화된 IVehicleState 데이터 모델 정의

미들웨어 아키텍트는 가장 먼저 상위 에이전트나 UI 뷰 모델(ViewModel)이 바라보게 될 이상적인 단일 상태 객체 IVehicleState (또는 VehicleBase) 인터페이스를 정의한다. 여기에는 기종에 상관없이 통용되어야 할 거시적 비행 상태(예: Arming, Navigation Mode, Failsafe)가 열거형(Enum)으로 선언된다.

1.2 펌웨어별 구상 어댑터(Concrete Adapter) 구현

PX4와 Ardupilot을 위한 각각의 구상 어댑터 클래스를 생성하여, HEARTBEAT, EXTENDED_SYS_STATE 등 수신된 순정(Raw) MAVLink 구조체(Struct)를 받아들여 공통 속성으로 매핑(Mapping)한다.

// 어댑터 패턴 기반의 상태 추상화 인터페이스 예시 (C++)
enum class CommonFlightMode { MANUAL, HOLD, MISSION, RETURN };

class VehicleStateAdapter {
public:
    virtual CommonFlightMode get_flight_mode() const = 0;
    virtual bool is_armed() const = 0;
};

// PX4 전용 어댑터: 비트 시프트 및 다차원 매핑 로직 내장
class Px4StateAdapter : public VehicleStateAdapter {
    px4_custom_mode custom_mode; 
public:
    CommonFlightMode get_flight_mode() const override {
        if (custom_mode.main_mode == PX4_CUSTOM_MAIN_MODE_AUTO) {
            if (custom_mode.sub_mode == PX4_CUSTOM_SUB_MODE_AUTO_RTL) {
                return CommonFlightMode::RETURN;
            }
        }
        return CommonFlightMode::MANUAL;
    }
};

// Ardupilot 전용 어댑터: 모놀리식 열거형 직접 매핑
class ArduCopterStateAdapter : public VehicleStateAdapter {
    uint32_t custom_mode;
public:
    CommonFlightMode get_flight_mode() const override {
        switch (custom_mode) {
            case COPTER_MODE_RTL: return CommonFlightMode::RETURN;
            case COPTER_MODE_AUTO: return CommonFlightMode::MISSION;
            default: return CommonFlightMode::MANUAL;
        }
    }
};

이러한 패턴 도입으로, GCS 프론트엔드 개발자는 기체의 고유 프로토콜 규격을 몰라도 단순히 adapter->get_flight_mode()만을 호출하여 UI 상단 상태 바(Status Bar)를 업데이트할 수 있게 된다.

2. 상향식 API 도출을 위한 파사드 패턴(Facade Pattern)

어댑터 패턴이 데이터의 **‘수신 상태 관측(Observation)’**을 추상화한다면, 파사드 패턴(Facade Pattern)은 미들웨어가 기체에 내리는 **‘다단계 제어 명령(Command)’**의 복잡성을 은닉(Hiding)하는 서브시스템 전면부(Front-end Facade) 역할을 담당한다.

2.1 복합 제어 시퀀스(Complex Control Sequence)의 캡슐화

단순히 “이륙(Takeoff)하라“는 명령이라 할지라도, PX4와 Ardupilot 내부적으로 밟아야 하는 허용 오차 프로세스 및 MAVLink 트랜잭션 시퀀스가 다르다.

파사드 객체인 VehicleCommandManager는 상위 레이어에게 단일화된 takeoff(altitude) 형태의 퍼블릭(Public) API만을 노출한다.

2.2 제어 흐름 분기(Routing)

파사드 인터페이스는 내부적으로 현재 체결된 기체의 프로파일 속성에 따라, 하위 모듈들(Arming Checker, MAVLink Dispatcher, Timeout Watchdog)을 알맞은 순서와 타이밍으로 조립하여 호출한다. 앞서 “37.6.3 커맨드 수용성” 절에서 다룬 바와 같이, PX4라면 오프보드 스트림 예열 로직을 파사드가 암묵적으로 선행 실행하고, Ardupilot이라면 MAVLink MAV_CMD_NAV_TAKEOFF만을 즉시 발송 후 모니터링 루프로 돌입하는 식이다.

3. 요약: 아키텍처적 격리(Isolation)의 완성

이 두 패턴의 결합은 미들웨어 아키텍처에 있어 **의존성 역전 원칙(DIP, Dependency Inversion Principle)**을 실현하는 가장 우수한 기법이다.

상위 컴포넌트(UI, 자율 복합 추론 엔진)는 저수준의 MAVLink 비트마스크나 Ardupilot의 열거형에 조금도 의존하지 않으며, 오직 IVehicleState라는 추상 계층에만 의존한다. 반대로, 하위 스레드들은 MAVLink 패킷이 도착할 때마다 각 펌웨어 어댑터를 통해 파사드 뒤에 안전히 감춰져 있는 블랙박스의 속성들을 갱신하기만 하면 된다. 이를 통해 새로운 기종(예: 독자 개발한 국산 비행제어기)이 플릿 시스템에 추가되더라도, 시스템 전체를 뜯어고칠 필요 없이 새로운 어댑터 클래스 한두 개를 추가 주입(Injection)하는 것만으로 완벽한 확장이 보장된다.
