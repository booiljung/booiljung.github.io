37.6.2.2. 객체 지향 패러다임(Object-Oriented Paradigm) 관점의 상태 패턴(State Pattern) 구현 방식 차이

유한 상태 기계(FSM, Finite State Machine)를 소프트웨어 아키텍처로 구현하는 방법은 크게 제어 흐름(Control Flow) 지향적 방식과 객체 지향(Object-Oriented)의 상태 패턴(State Pattern) 활용 방식으로 나뉜다. 무인기(UAV) 비행 제어 및 관제 미들웨어 개발에 있어 이 두 아키텍처의 차이를 이해하는 것은 상태 전이의 안정성과 코드 확장성을 결정짓는 핵심 요소이다. 본 절에서는 PX4-Autopilot과 Ardupilot이 비행 모드라는 ’상태’를 코드 레벨에서 어떻게 상이하게 객체 지향 설계(OOP)에 접목하고 있는지 대조한다.

1. Ardupilot의 GoF ‘상태 패턴(State Pattern)’ 정석 구현

Ardupilot(특히 ArduCopter 펌웨어)은 객체 지향 설계의 고전인 GoF(Gang of Four) 디자인 패턴 중 **상태 패턴(State Pattern)**을 매우 충실하게 따르고 있다.

1.1 다형성(Polymorphism)을 이용한 상태의 객체화

Ardupilot은 모든 비행 모드를 단일 부모 클래스인 Mode에서 파생된 독립적인 자식 클래스(Concrete State)로 인스턴스화(Instantiation)한다. 기체의 최상위 제어 객체(Copter)는 현재 동작 중인 상태 객체의 포인터(Pointer)만을 소유하며, 외부 상태 전이 요청이 오면 해당 포인터를 전환(Swapping)하는 방식으로 모드를 변경한다.

// Ardupilot의 상태 패턴(State Pattern) 예시 구조 (C++)
class Mode {
public:
    virtual bool init(bool ignore_checks) = 0; // 진입(Entry) 조건 검사
    virtual void run() = 0;                    // 실행(Do) 로직
    virtual void exit();                       // 탈출(Exit) 정리 로직
};

class Mode_Loiter : public Mode {
    bool init(bool ignore_checks) override {
        // GPS 3D Fix 확인 등 Loiter 전용 진입 조건
        return gps.status() >= AP_GPS::GPS_OK_FIX_3D;
    }
    void run() override {
        // 위치 제어기 루프 구동
    }
};

class Copter {
    Mode *flightmode; // 현재 상태 객체 포인터
    
    void set_mode(Mode *new_mode) {
        if (new_mode->init(false)) { // 다형성을 이용한 진입 검증
            flightmode = new_mode;   // 포인터 교체를 통한 상태 전이
        }
    }
    void update() {
        flightmode->run(); // 현재 모드에 관계없이 동일한 인터페이스 호출
    }
};

1.2 아키텍처적 장단점

  • 장점: 캡슐화(Encapsulation)가 뛰어나다. 새로운 비행 로직을 추가하려면 Mode 클래스를 상속받는 객체를 하나 더 정의하기만 하면 되므로 개방-폐쇄 원칙(OCP, Open-Closed Principle)을 훌륭하게 만족한다. 각 모드의 생명주기(진입, 실행, 이탈) 코드가 하나의 파일과 클래스 내에 완벽하게 응집(Cohesion)된다.
  • 단점 (오버헤드): 실시간 제어 루프(통상 400Hz) 내에서 가상 함수(Virtual Function) 테이블 조회가 주기적으로 일어나는 오버헤드가 발생한다. 또한, A 상태에서 B 상태로 내부 변수나 컨텍스트를 전달하기 위해서는 부가적인 전역 객체 의존성이 불가피해진다.

2. PX4의 uORB 토픽 중심의 분산 상태 평가 기법

반면 PX4-Autopilot은 상태를 ’행위(Behavior)를 가진 객체’로 보지 않는다. PX4에서 상태는 오직 **순수한 데이터(Pure Data)**일 뿐이며, 행위는 별도의 컨트롤러 노드(Task)들에 위임(Delegation)된다.

2.1 커맨더(Commander) 독점형 진리표 검증 단계 분리

PX4는 비행 모드에 따른 ’제어 로직(Execution)’을 캡슐화하지 않고, ’상태 전이(Transition)의 타당성’만을 캡슐화한다. commander 모듈은 GCS 등으로부터 모드 변경 요청이 들어오면 모든 센서와 시스템 상태(uORB 토픽)를 수집하여 커다란 방향성 비순환 그래프(DAG) 형태의 로직 트리를 통과시킨다.

상태의 확정은 다형성을 이용한 포인터 교체가 아니라, vehicle_status uORB 메시지 내의 nav_state 열거형 상수를 변경하고 네트워크에 브로드캐스트(Broadcast)하는 연산으로 마무리된다.

// PX4의 데이터 결합 기반 상태 전이 예시 (C++)
bool set_nav_state(vehicle_status_s *status, uint8_t new_nav_state) {
    bool valid = false;
    switch (new_nav_state) {
        case vehicle_status_s::NAVIGATION_STATE_AUTO_LOITER:
            // 순수 데이터 스트림에서 상태 확인
            if (global_position_is_valid) {
                 valid = true;
            }
            break;
    }
    
    if (valid) {
        status->nav_state = new_nav_state;
        orb_publish(ORB_ID(vehicle_status), status_pub, status);
    }
    return valid;
}

2.2 상태와 제어의 철저한 디커플링(Decoupling)

이러한 설계 하에서는 PositionControl이나 AttitudeControl과 같은 제어 노드들이, 자신이 어떤 ’비행 모드’에 있는지 능동적으로 관할하지 않는다. 제어 노드는 단지 변경된 nav_state 토픽을 수신하면 그 모드에 합당한 셋포인트(Setpoint, 목표값) 소스를 바라보도록 스위치(Multiplexing)를 바꿀 뿐이다. 즉, 상태 관리 객체(Commander)와 추종 로직(FlightTasks)이 극한으로 분리되어 있다.

3. GCS 미들웨어 통합 아키텍트의 관점 차이 융합

QGroundControl이나 ROS2 DDS 상위 관제 에이전트를 개발하는 미들웨어 아키텍트(Architect)는 이 대조적인 두 구현 패턴을 명확히 인지하고 어댑터(Adapter)를 설계해야 한다.

  • Ardupilot 대상 통합 (객체 상태관): GCS는 모드 변경 시 해당 모드의 init() 메서드가 실패할 수 있음을 상정하고 즉각적인 MAVLink COMMAND_ACK 결과(수용, 거부)를 UI나 상위 에이전트에 통지해야 한다. 거절 시 재진입을 위한 추가 센서 데이터(예: GPS Fix)를 대기시키는 로직을 미들웨어가 조율하기 상대적으로 쉽다.
  • PX4 대상 통합 (데이터 상태관): PX4의 상태가 거부당할 경우, 이는 특정 모드 객체의 진입 실패라기보다 시스템 전체의 선결 조건(Precondition 토픽 스트림) 미충족을 의미한다. 따라서 미들웨어는 PX4에 거절당하면 단순히 재시도(Retry)하는 선을 넘어, 특정 토픽(예: estimator_status) 플래그의 상태가 무엇이 부족한지 스스로 관측(Observer)하여 에이전트 측 유한 상태 기계(FSM)에 복원(Recovery) 시나리오를 가동해야 한다.

결론적으로, Ardupilot은 고전적이고 직관적인 OOP 설계 템플릿(상태 패턴)을 비행 제어에 성공적으로 정착시켰으나 마이크로컨트롤러 부하 단점이 상존하며, PX4는 데이터 위주의 발행-구독(Publish-Subscribe) 패턴을 채용하여 코드 분산성과 실행 효율성을 무섭게 최적화하였음을 알 수 있다.
