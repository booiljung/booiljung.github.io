37.2.3.2 전제 조건(Pre-condition) 논리곱(AND) 연산 및 사후 조건(Post-condition) 검증 알고리즘

소프트웨어 공학의 계약에 의한 설계(Design by Contract) 원칙은 모듈 간의 인터페이스 유효성을 보장하기 위해 선행 조건(Pre-condition)과 후행 조건(Post-condition)의 엄격한 준수를 요구한다. PX4-Autopilot의 유한 상태 기계(FSM) 아키텍처 역시 기체의 치명적인 오작동을 방어하기 위해 이러한 계약적 검증 원리를 채택하고 있다. 본 절에서는 특정 비행 모드나 무장(Arming) 상태로 진입하기 전 수행되는 논리곱(AND) 기반의 전제 조건 평가와, 상태 전이가 완료된 후 시스템의 정합성을 확인하는 사후 조건 검증 알고리즘의 동작 메커니즘을 심도 있게 분석한다.

1. 전제 조건(Pre-condition)의 논리곱(AND) 연산

오토파일럿이 새로운 상태로 전이하기 위해 가장 먼저 거치는 관문은 전제 조건의 검증이다. PX4의 PreFlightCheck 모듈과 Commander 로직은 수십 가지 이상의 개별적인 안전 플래그(Safety Flag)들을 집계하여 하나의 통합된 불리언(Boolean) 결과를 도출한다.

1.1 직렬화된 논리곱(AND)의 방어적 특성

이륙을 위한 기체 시동(Arming) 시, PX4가 확인해야 할 전제 조건 P는 다음과 같은 하위 명제들의 집합으로 구성된다.

  • P_{ekf}: EKF(위치 추정기)의 혁신(Innovation) 오차가 허용치 이내인가?
  • P_{mag}: 나침반(Magnetometer)의 간섭(Interference)이 임계치 미만인가?
  • P_{rc}: 조종기(RC) 신호가 유효하게 수신되고 있는가?
  • P_{bat}: 배터리 전압이 설정된 최소 이륙 전압 이상인가?

전체 이륙 가능 여부를 판별하는 유효성 함수 f_{valid} 는 모든 조건이 참(True)일 때만 참이 되는 순수 논리곱(AND) 수식으로 매핑된다.
f_{valid} = P_{ekf} \land P_{mag} \land P_{rc} \land P_{bat} \dots

이러한 직렬화된 논리곱 구조는 단 하나의 센서라도 비정상적인 값을 보고할 경우(P_i = \text{False}), 전체 수식의 결과를 즉각 거짓(False)으로 붕괴시키는 강력한 페일 세이프(Fail-safe) 특성을 지닌다.

2. 사후 조건(Post-condition) 검증 및 콜백(Callback) 알고리즘

전제 조건을 통과하여 상태 변수(State Variable)의 덮어쓰기(Write) 연산이 성공적으로 수행되었다 하더라도, 물리적인 액추에이터(Actuator)나 다른 소프트웨어 모듈이 그 상태 변화를 정상적으로 추종하고 있는지 확인하는 사후 조건(Post-condition) 검증이 뒤따라야 한다.

2.1. 상태 천이 후속 행동(Side-effects)의 정합성 확인

PX4에서는 주 상태(Main State)가 변경되면, vehicle_status uORB 토픽이 퍼블리시될 뿐만 아니라 시스템 아키텍처 전체 요소들에 아래와 같은 연쇄 반응(Chain Reaction)이 예외 없이 일어나야 한다.

  • 모터 및 ESC 통제: ARMED 상태로 진입했다면, 모터 믹서(Mixer) 또는 할당기(Allocator)가 PWM 유효 신호를 PWM 출력 핀에 실제로 인가(Assert)했는지 점검.
  • LED 및 부저(Buzzer): 상태 전환을 조종자에게 알리기 위한 시각적/청각적 오디오 피드백(Tune)이 스케줄링되었는지 점검.
  • 데이터 로깅(ULog): 이륙(Takeoff) 등 중대한 상태 변이가 발생했을 때 로거 모듈이 고해상도 기록(High-rate Logging)을 정상적으로 시작(Start)했는지에 대한 묵시적 확인.

만약 상태 변수는 ARMED로 변경되었으나 하드웨어 PWM 타이머 설정이 실패하여 사후 조건 검증에서 불일치(Mismatch)가 발생한다면, 시스템은 즉시 안전 장치를 가동하여 강제로 DISARMED 상태로 롤백(Rollback)하거나 비상 정지(Kill)를 트리거한다.

3. PX4와 Ardupilot의 조건 검증 철학 비교

  • Ardupilot의 순차-분산(Sequential-Distributed) 평가: Ardupilot(Prearm.cpp)은 메인 루프에서 단계별로 여러 함수를 호출하여 조건을 검토하고, 실패 지점에서 메시지를 출력한 뒤 곧바로 리턴(Return)하는 절차적 프로그래밍 관습이 강하다. 런타임 분석 툴로 논리의 전체 그림을 한 번에 그려내기가 어렵다.
  • PX4의 매트릭스(Matrix) 및 트랜잭션(Transaction) 평가: PX4는 전제 조건을 논리 행렬(Logical Matrix)처럼 취급하여 모든 PreFlightCheck 결과를 하나의 비트마스크(Bitmask)에 논리곱 연산으로 누적한 후 단번에 결정(Decision)을 내린다. 이 방식은 코드의 테스트 용이성(Testability)을 극대화하며, 성공과 실패의 원인을 GCS 에 단일 MAVLink STATUSTEXT 메시지로 명확히 리포팅할 수 있게 해준다.

4. 검증 알고리즘 타이밍 다이어그램

커맨더 내부에서 전제 조건의 AND 연산과 사후 조건의 확인 절차를 시퀀스로 도식화하면 다음과 같다.

sequenceDiagram
    participant Ctrl as 외부 제어기 / GCS
    participant Cmd as PX4 Commander 모듈
    participant Eval as Pre/Post Condition 검증기
    participant HW as 액추에이터 및 센서 계층

    Ctrl->>Cmd: 상태 천이 요청 (예: ARM_COMMAND)
    
    Note right of Eval: 1. 전제 조건 논리곱 평가
    Cmd->>Eval: 모든 센서 및 상태 플래그 확인
    Eval-->>Cmd: 결과 반환 (True / False)
    
    alt Pre-condition == False
        Cmd-->>Ctrl: 명령어 거부 (NACK) 및 원인 보고
    else Pre-condition == True
        Note right of Cmd: 2. 단일 원자적 트랜잭션 (상태 변이)
        Cmd->>Cmd: 상태 변수 갱신 (STANDBY -> ARMED)
        
        Cmd->>HW: 액추에이터 제어 명령 하달
        
        Note right of Eval: 3. 사후 조건(Post-condition) 검증
        HW-->>Eval: PWM 활성화 피드백 / 에러 상태 보고
        
        alt Post-condition == Error
            Eval->>Cmd: 하드웨어 후속 조치 실패 알림
            Cmd->>Cmd: 강제 롤백 (Lockdown / DISARM)
            Cmd-->>Ctrl: Critical Error (Hardware Failure) 보고
        else Post-condition == OK
            Cmd-->>Ctrl: 명령어 수락 (ACK)
        end
    end

도해에서 핵심적인 부분은 상태 변수가 업데이트되었다 하더라도, 하위 하드웨어 계층의 사후 조건(Post-condition)이 성립되지 않으면 시스템 스스로 즉각적인 롤백 로직을 구동한다는 점이다.

5. 결론

무인기의 유한 상태 기계는 단순한 메모리 변수의 집합이 아니라, 기계적 에너지의 방출(예: 모터 회전)을 통제하는 물리적 자물쇠(Lock)이다. PX4-Autopilot은 이 자물쇠를 열기 위해 수많은 센서의 정상 동작을 증명하는 강력한 전제 조건(Pre-condition) 논리곱(AND) 알고리즘을 강제하며, 자물쇠가 열린 뒤에도 실제로 기어가 물리적으로 맞물렸는지 사후 조건(Post-condition) 피드백을 통해 이중적으로 검증한다. 관제 소프트웨어 시스템을 설계하는 엔지니어는 기체에 MAVLink 변경 명령을 내린 후 단순히 전송 성공(ACK)만으로 상태 변이를 확신해서는 안 되며, 기체로부터 지속적으로 발송되는 HEARTBEAT 및 하드웨어 오류 코드를 교차 검증하여 GCS 상의 상태와 관측기 상의 사후 조건을 완벽히 일치시켜야만 치명적 사고를 예방할 수 있다.
