Chapter 37. GCS 제어 로직: 상태 기반 제어(State-based Control)의 필요성

지상 관제 시스템(Ground Control Station, GCS)은 무인항공기(UAV)와 작업자 혹은 상위 자율 에이전트 간의 핵심 인터페이스로서, 비행 임무(Flight Mission)의 계획, 실행, 원격측정(Telemetry) 상태 모니터링을 포괄적으로 담당한다. 본 장에서는 최신 PX4-Autopilot(Firmware v1.14.0 기준) 및 QGroundControl(v4.3.0 기준) 아키텍처 환경에서 GCS 제어 로직이 왜 엄격한 상태 기반(State-based)으로 설계되어야 하는지, 그 당위성과 기술적, 동역학적 배경을 논의한다.

1. 개요: 상태 비저장(Stateless) 제어와 상태 기반(Stateful) 제어의 대조

무인항공기 제어 체계에서 상위 노드(GCS 또는 Companion Computer)가 하위 비행 제어기(Flight Controller, FC)에 명령을 하달하는 방식은 구조적으로 ‘상태 비저장(Stateless)’ 방식과 ‘상태 기반(Stateful)’ 방식으로 대별된다.

  • 상태 비저장 제어(Stateless Control):
    GCS가 명령(Command)을 일방향으로 전송(Fire-and-forget)하는 모델이다. 명령의 전송 행위 자체에 집중하며, 기체(Vehicle)의 현재 논리적 허용 상태를 사전에 조회하거나 명령 실행 후의 상태 전이 여부를 강하게 검증하지 않는다. 이러한 구조는 구현이 단순하지만 통신 불안정성과 시스템 지연이 발생하는 무선 환경에서는 예기치 않은 오작동이나 중복 명령(Over-control)을 유발할 위험이 높다.

  • 상태 기반 제어(State-based Control):
    본 장에서 핵심적으로 다루게 될 패러다임으로, 기체의 현재 내부 상태(Vehicle State, 예: Arming State, Flight Mode, Navigation State)와 GCS가 인지하는 프록시(Proxy) 상태 간의 주기적인 동기화(Synchronization)를 전제로 한다. 명령 인가 전 전제 조건(Precondition)을 시스템적으로 검증하며, 명령 수행 후 피드백 결과(Acknowledgment)를 바탕으로 GCS 측의 상태 머신 전이(State Transition)를 완료하는 추적(Tracking) 기반 제어 루프를 뜻한다.

2. PX4-Autopilot 제어 시스템 내 상태 기반 로직의 필연성

PX4는 실시간 운영체제(NuttX RTOS) 기반에서 비동기(Asynchronous) 이벤트 구동형 시스템으로 설계되었으며, Commander 모듈 주도하에 복잡하게 얽힌 계층적 상태 머신(Hierarchical State Machine)을 운영한다. 지상국에서 이 시스템을 안전하게 제어하기 위해 상태 기반 로직이 요구되는 주요 원인은 다음과 같다.

2.1 텔레메트리(Telemetry) 무선 네트워크의 확률론적 비결정성(Non-determinism)

MAVLink 프로토콜을 통하여 GCS와 기체 간 이루어지는 직렬 포트(UART) 및 UDP 기반 무선 채널 통신은 근본적으로 제한된 대역폭 하에서 작동한다. 외부 간섭, 통신 거리 증가, 또는 군집 비행 시 단말 간 경합으로 인하여 패킷의 손실(Packet Loss), 지연(Latency), 혹은 순서 역전(Out-of-order) 현상이 불가피하게 발생한다. 이륙(Takeoff), 시동(Arming), 임무 모드 전환(Mission Mode Transition) 등 비가역적 파급력을 가지는 치명적 제어 명령(Critical Commands)이 네트워크 손실로 누락될 경우, 기체는 명령 불이행 상태에 머물게 되며, GCS가 이를 인지하지 못한 채 후속 명령을 지속 전송하게 되면 비행 시퀀스의 붕괴를 초래한다. 따라서 GCS는 COMMAND_LONG 메시지에 대한 COMMAND_ACK 응답을 확인하고 타임아웃(Timeout) 발생 시 재전송(Retry) 스케줄링을 재귀적으로 수행하는 상태 기반 복구 메커니즘을 필수적으로 내장해야 한다.

2.2 로버스트 제어(Robust Control)와 GCS 상태 락(Lock) 기법

상태 기반 제어의 대표적인 응용 사례로 GCS UI 레벨 혹은 제어 알고리즘 층에서 수행되는 ‘상태 락(Lock)’ 기법을 들 수 있다. 비행 모드 변경(예: Position 모드에서 Auto-Mission 모드로의 전환)을 요청했을 때, 기체의 HEARTBEAT 메시지 상에서 custom_mode 필드의 갱신이 확인될 때까지 시스템은 과도기적 중첩 통제 상태(Transitional Control Lock)에 진입한다. 이는 작업자 혹은 상위 자율 에이전트(ROS2 경로 계획 노드 등)가 무분별하게 상충하는 명령을 동시 다발적으로 주입하는 것을 논리적으로 차단하는 방어적(Defensive) 설계이다.

2.3 전제 조건(Precondition)의 다단위(Multi-stage) 검증

PX4-Autopilot은 상태 전이 시 매우 엄격한 런타임 제약 조건을 요구한다. 시동을 전개하기 위해서는 IMU 및 나침반 센서의 유효성, GPS의 3D Fix 확보 유무(ekf2 Estimator의 로컬 위치 수렴 여부), 배터리 방전 임계치 초과 여부 등이 종합적으로 평가되어 사전 점검(Pre-flight Check)을 통과해야 한다. GCS는 상태 기반 로직 내에서 수신되는 SYS_STATUS 및 진단 텍스트(STATUSTEXT)를 분석하여, 불가지(Unknown) 상태에서의 무리한 명령 전송을 억제하고, 거부(Reject) 원인을 작업자에게 즉각적으로 표출하는 인터랙티브 구조를 취해야 한다.

3. PX4와 Ardupilot 제어 관철성의 철학적 비교

Ardupilot 유저 인터페이스(주로 Mission Planner)는 상대적으로 유연한 파라미터 접근 및 즉각적인 상태 변경 시도를 폭넓게 허용하는 편이다. 각 비행 컨텍스트(차박, Copter, Plane) 내부에서 자체 방어 기제가 동작하지만, 관제 소프트웨어 자체의 상태 동기화 결합도는 PX4 대비 강제되지 않는다.
이에 반해 PX4 생태계와 결합된 QGroundControl(QGC)은 소프트웨어 아키텍처 수준에서 ’차량 매니저(Vehicle Manager)’라는 강력한 싱글턴(Singleton) 프록시 객체를 두어 펌웨어의 실제 Commander 상태와 1:1 대응되도록 디자인되어 있다. PX4는 상태의 일관성이 보장되지 않은 오프보드(Offboard) 개입을 엄격히 차단하므로, GCS 개발자는 반드시 PX4의 내부 상태 전이 모델에 동기화되는 상태 머신을 외부(GCS/ROS2 에이전트)에도 미러링(Mirroring)하여 구현해야 한다.

4. 구조화된 GCS 통신 트랜잭션 도식(Sequence Analysis)

GCS의 상태 기반 제어가 어떠한 폐루프(Closed-loop) 트랜잭션 흐름을 가지는지 다음의 시퀀스 다이어그램으로 확인할 수 있다.

sequenceDiagram
    participant Agent as GCS (혹은 자율 관제 에이전트)
    participant Channel as 물리적 무선 계층 (MAVLink Channel)
    participant PX4 as PX4-Autopilot (Commander / uORB)

    %% Background State Sync
    Note over Agent, PX4: 백그라운드 상태 폴링 (Background Polling)
    loop 주기적 동기화 (1Hz)
        PX4->>Channel: HEARTBEAT (current_mode, system_status)
        Channel->>Agent: Vehicle State 객체 업데이트
    end

    %% State-validated Command Execution
    Note over Agent: 전제 조건 평가 (Is Arming Allowed?)
    Agent->>Agent: 검증 완료 (Condition Verified)
    Agent-->>Channel: COMMAND_LONG (MAV_CMD_COMPONENT_ARM_DISARM)
    Channel->>PX4: 명령 수신

    %% Command Processing
    PX4->>PX4: 내부 상태 유효성 재검증 및 액추에이터 활성
    PX4-->>Channel: COMMAND_ACK (MAV_RESULT_ACCEPTED)
    
    %% Post-condition Update
    Channel->>Agent: Ack 수신 처리 (명령 대기 큐 해소)
    PX4->>Channel: HEARTBEAT 표출 갱신 (Armed)
    Channel->>Agent: 최종 State Lock 해제 및 UI 반영

네트워크 지연 혹은 패킷 드롭으로 인해 COMMAND_ACK가 도착하지 않을 경우, Agent 측은 Timeout 후 내부 재전송 버퍼 큐에서 명령을 다시 발급하게 되며, 이는 무인항공기 신뢰성 운용의 척도가 된다. 결론적으로 GCS 제어 로직에서 설계된 ’상태 기반 제어’는 시스템의 맹목적인 실행을 배제하고, 수학적, 논리적 제어 안정성을 보장하기 위한 가장 필수적인 공학적 토대이다.
