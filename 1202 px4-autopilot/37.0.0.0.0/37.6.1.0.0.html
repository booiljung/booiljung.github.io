37.6.1. 상태 공간 매핑(State Space Mapping) 및 비행 모드(Flight Mode) 캡슐화(Encapsulation) 철학 대조

비행 제어 시스템이 외부(GCS 또는 자율 에이전트)로부터 명령을 수신하고 스스로의 거동을 결정하는 가장 핵심적인 요소는 “현재 상태를 어떻게 정의하고, 비행 모드를 어떻게 캡슐화(Encapsulation)할 것인가“이다. 이 질문에 대해 PX4-Autopilot과 Ardupilot은 소프트웨어 공학적으로 확연히 다른 철학을 제시한다. 본 절에서는 두 이기종 아키텍처가 “상태(State)“와 “모드(Mode)“를 수학 공간 및 메모리 상에 매핑(Mapping)하는 방식의 차이를 심층 분석한다.

1. Ardupilot의 비행 모드 캡슐화(Capsulation) 철학

Ardupilot은 객체 지향 프로그래밍(OOP, Object-Oriented Programming)의 다형성(Polymorphism)과 캡슐화 개념을 비행 제어 로직에 적극적으로 투영한다.

1.1 모노리식(Monolithic) 상태 머신과 다형성

Ardupilot 소스 코드(예: ArduCopter)를 분석해 보면, 비행 모드는 독립적인 클래스(Class)로 구현되어 있다. 최상위 Copter 클래스는 현재 활성화된 모드의 포인터(mode)를 소유하고, 메인 루프(Main Loop)의 매 이터레이션(Iteration, 통상 400Hz)마다 mode->run() 계열의 가상 함수(Virtual Function)를 지속적으로 호출한다.

// Ardupilot 내부 간략화된 구조 (C++)
class Mode {
public:
    virtual bool init(bool ignore_checks) = 0;
    virtual void run() = 0;
    // ...
};

class ModeLoiter : public Mode {
public:
    bool init(bool ignore_checks) override;
    void run() override; // 위치 제어 알고리즘 응집
};

이러한 설계의 강점은 응집도(Cohesion) 에 있다. 하나의 비행 모드가 해야 할 센서 퓨전(Sensor Fusion)의 일부 처리, 조종기 입력 맵핑, 제어 루프 진입, 그리고 가속도 명령 생성이 단일 클래스 내부로 캡슐화되어 있다.

1.2 한계점: 상태 공간(State Space)의 결합(Coupling)

그러나 이러한 설계는 전역적(Global)인 상태 모니터링이나 특정 하위 시스템(예: 회피 루프, Failsafe 로직)이 기체의 현재 맥락(Context)을 이해해야 할 때 구조적 결합도(Coupling)를 높이는 요인이 된다. 모드 자체에 비행 제어 로직과 상태 논리가 얽혀 있기 때문에, 특정 모드 외부에서 해당 모드의 세부 상태를 조작하거나 관측하기 위해서는 강제적인 캐스팅(Casting)이나 복잡한 예외 처리가 동반되어야 한다.

2. PX4의 상태 공간 매핑(State Space Mapping) 철학

PX4는 비행 모드라는 개념을 단일한 객체로 취급하는 대신, 다양한 컴포넌트 간의 수학적 상태 공간(State Space) 교집합으로 해석한다.

2.1 네비게이션 상태(Navigation State)와 전역 상태 분리

PX4의 소스 코드(src/modules/commander/)를 분석해보면, 기체의 모드는 여러 개의 직교하는(Orthogonal) 상태 플래그들의 조합으로 매핑된다. 대표적으로 vehicle_status uORB 토픽 안에는 다음과 같은 요소들이 독립적으로 존재한다.

  • arming_state: Armed, Standby, Init 등
  • nav_state: Manual, Altitude, Position, Mission, Offboard 등
  • hil_state: HITL 상태 여부

PX4의 캡슐화는 ’비행 모드 별 제어 로직’을 캡슐화하는 것이 아니다. 모드 검증 로직은 commander 모듈이 독점(Encapsulate)하여 수행하고, 실제 물리적 제어를 위한 계산은 FlightTasks 프레임워크나 PositionControl 모듈 등에 위임(Delegation)된다.

2.2 방향성 비순환 그래프(DAG) 형태의 직교(Orthogonal) 상태 매핑

PX4의 상태 전이는 다차원 상태 공간 내의 벡터 이동으로 간주될 수 있다. 오프보드(Offboard)와 같은 복합적인 nav_state를 점유하기 위해서는, 위치 추정치 유효성(local_position_valid), RC 신호 상태, 오프보드 제어 스트림의 존재 유무 등 다수의 직교하는 차원(차원 검증)들을 만족해야 한다.

graph TD
    A[GCS: Set Mode Mission] --> B(Commander Module)
    B --> C{State Space Validation}
    C -->|Global Position: Valid| D{Arming State}
    C -->|Global Position: Invalid| E[Reject Transition]
    D -->|Armed| F[Transition to Mission State]
    D -->|Disarmed| E

이 방식은 각 상태 조건들이 uORB를 통해 결합도 없이(Decoupled) 관리되므로, 시스템 전역에 걸쳐 높은 예측 가능성(Predictability)과 안전성(Safety)을 확보한다.

3. 미들웨어 및 ROS2 연동 관점의 시사점

이기종 관제 미들웨어를 설계할 때 이러한 근본적 차이는 중요한 의미를 가진다.

  1. Ardupilot 연동 시: GCS나 ROS2 관측기(Observer)는 기체로부터 오는 HEARTBEAT.custom_mode 하나만을 읽어도 현재 기체가 무슨 계산 루프 속에 진입했는지 완전히 파악할 수 있다. 직관성이 매우 우수하다.
  2. PX4 연동 시: GCS나 ROS2 관측기는 기체가 ‘Acro’ 모드인지 판단하기 위해 단순히 MAVLink 모드 매핑만 분석해서는 안 되며, 기체의 arming_state 및 센서의 Failsafe 플래그들이 nav_state인 Acro 모드를 정상적으로 지탱하고 있는지를 다차원적으로 검증하는 로직을 지상 통제 소프트웨어(GCS) 측에 반영해야 한다.

요약하자면, Ardupilot은 행위를 중심으로 상태를 종속시키는 객체 지향적 캡슐화 철학을 채택하였고, PX4-Autopilot은 종합적인 상태 벡터(State Vector)를 검증한 후 합의된 행위를 도출하는 데이터 흐름(Data Flow) 및 이벤트 기반(Event-driven) 상태 매핑 철학을 채택하였다.
