37.6.3. 커맨드 수용성(Command Acceptability) 평가 기준의 허용 오차(Tolerance Margin) 스펙트럼

지상 관제 시스템(GCS) 또는 자율 비행 미들웨어 시스템(예: ROS2 기반 에이전트)이 무인기(UAV)를 제어하기 위해 MAVLink나 uORB 메시지를 전송할 때, 비행 제어기(Flight Controller) 내부에서는 해당 명령을 즉각 수용할지 혹은 거부할지 결정하는 수용성 평가(Command Acceptability Evaluation) 과정이 수행된다. 이 평가 기준의 관대함 내지 엄격함, 즉 허용 오차(Tolerance Margin)의 스펙트럼은 오토파일럿 아키텍처마다 확연히 다르며, GCS-UAV 간의 통신 신뢰성 유지 설계에 결정적인 영향을 미친다.

1. 커맨드 수용성(Command Acceptability)의 개념적 스펙트럼

커맨드 수용성이란 “기체가 현재 완벽한 이상적 상태(Ideal State)가 아니더라도, 외부의 명령을 얼마나 유연하게 수용하여 실행을 시도할 것인가“를 나타내는 척도이다. 이 허용 오차의 스펙트럼은 양 극단으로 나눌 수 있다.

  1. 무관용 방어 철학 (Zero-Tolerance / Strict Bound): 기체의 센서, 위치 추정치, 시스템 타이밍 등이 사전에 정의된 임계치(Threshold)를 단 1%라도 벗어나거나, 사전 필수 명령이 누락되었다면 외부 명령을 안전 위협(Safety Threat)으로 간주하고 즉각 MAV_RESULT_DENIED 또는 MAV_RESULT_TEMPORARILY_REJECTED를 반환한다.
  2. 조건부 허용 및 자율 보상 (Conditional Acceptance / Heuristic Compensation): 기체의 일부 센서가 일시적으로 불안정하거나 파라미터가 최적 범위를 살짝 벗어났더라도, 내부의 휴리스틱(Heuristic) 루프나 백업 추정기를 통해 비행의 유효성을 보상(Compensation)할 수 있다면 일단 명령을 수용(MAV_RESULT_ACCEPTED)하고 자율적으로 제어를 시도한다.

2. 이기종 아키텍처 간의 허용 오차 스펙트럼 비교

PX4-Autopilot과 Ardupilot은 이 스펙트럼의 서로 다른 지점에 위치하고 있으며, 이는 미들웨어를 설계하는 데 있어 재시도(Retry) 로직과 상태 전이 로직을 근본적으로 다르게 짜도록 유도한다.

2.1 엄격한 사전 조건(Precondition) 록인(Lock-in): PX4-Autopilot

PX4는 기본적으로 무관용 방어 철학 쪽에 가깝다. PX4 커맨더(commander.cpp) 모듈과 비행 작업(FlightTasks) 모듈은 상태 천이 및 외부 명령 수용에 대해 극도로 보수적인 잣대를 들이댄다.

  • Set-point 데이터 스트림 강제: 외부에서 Offboard 모드로 전환하라는 명령(MAV_CMD_DO_SET_MODE)이 수신되기 이전에, 이미 기체로 초당 2회(2Hz) 이상의 타겟 셋포인트 스트림(시간 동기화된 위치/속도/자세 명령)이 들어오고 있어야만 모드 변경을 ’허가’한다.
  • Estimator 엄격성: EKF2 모듈에서 위치 추정 오차 분산(Position Variance)이 내부 보안 파라미터 상한선(예: COM_POS_FS_EPH)을 넘어서는 순간, 자동 비행 목적지로의 Goto 명령은 즉시 인터락(Interlock)에 걸려 거부된다.

2.2 유연한 수용과 자율적 보상: Ardupilot

상대적으로 Ardupilot(특히 Copter)은 명령을 최대한 수용하고 기체 스스로 예외를 핸들링하는 조건부 허용 철학 쪽에 더 가깝다.

  • 타임아웃(Timeout) 기반 유연한 Guided 모드: GCS나 에이전트가 Guided 모드를 활성화한 뒤 곧바로 데이터 스트림을 보내지 않더라도 일단 모드 진입이 허용된다. 기체는 이전에 유지하던 상태(예: 제자리에 정지하여 호버링)를 자율적으로 연장하며 새로운 명령을 느긋하게 기다린다.
  • 포용적 위치 추종: 센서 노이즈가 일시적으로 튀더라도, EKF3의 다중 코어 혁신(Innovation) 허용 오차폭 내부에 있으면 미세한 궤적 이탈을 자율적으로 억제해가며 전달된 웨이포인트(Waypoint) 이동 명령을 계속해서 추종(Follow)한다.

3. GCS 미들웨어 설계에 미치는 영향

이러한 수용성 평가 기준의 차이로 인해, 단일 GCS 코드로 이기종 플릿(Heterogeneous Fleet)을 관리하려는 미들웨어(예: QGroundControl의 백엔드 통신 매니저)는 다음과 같이 분기된 통신 전략(Communication Strategy)을 내장해야 한다.

3.1 동기화 및 재시도(Synchronization and Retry) 설계

PX4를 타겟으로 할 때, 미들웨어는 ‘상태 준비(State Preparation) ➔ 스트림 예열(Stream Pre-warming) ➔ 명령 송신(Command Dispatch) ➔ 결과 확인(ACK Check)’ 이라는 원자적 트랜잭션(Atomic Transaction) 파이프라인을 매우 정교하게 구축해야 한다. 단순히 COMMAND_LONG을 쐈다고 해서 동작할 것이라는 긍정적 가정(Optimistic Assumption)은 치명적인 연계 오류(Cascading Failure)를 발생시킨다. PX4가 명령을 거부할 확률(명령 폭주, 스트림 지연, 센서 딜레이 등)이 상시 존재하기 때문이다.

역으로 Ardupilot을 타겟으로 할 때, 미들웨어는 기체가 명령을 쉽게 수용한 뒤 내부적으로 자율 보상하는 동안 실제로 기체가 의도한 궤적에 완벽히 정렬되었는지를 지속적으로 ’모니터링(Monitoring)’하는 옵저버 패턴(Observer Pattern)의 주기를 더 촘촘히 짜야 한다. 명령 수용(ACK)이 곧 완벽한 물리적 상태의 달성을 보장하는 것은 아니기 때문이다.

정리하자면, 기체의 수용성 허용 오차폭(Tolerance Margin)을 어떻게 설정했는가는 곧 “GCS와 UAV 중 어느 쪽이 예외 처리(Exception Handling)의 지배권을 가져갈 것인가“를 보여주는 아키텍처 철학의 핵심 지표이다.
