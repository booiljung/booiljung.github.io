37.3.1. 시제 논리(Temporal Logic) 기반의 동적 명령어 검증(Dynamic Command Validation)

무인항공기(UAV)의 제어 시스템에서 수신되는 명령어는 단순한 정적 상태뿐만 아니라 시간에 따른 상태의 연속적인 변화(Sequence of States)를 고려하여 검증되어야 한다. PX4-Autopilot의 유한 상태 기계(FSM, Finite State Machine)와 안전 인터락(Safety Interlock) 아키텍처는 내부적으로 시제 논리(Temporal Logic)의 원형적 개념을 차용하여 동적 명령어 검증(Dynamic Command Validation)을 수행한다. 이 절에서는 이산 사건 시스템(Discrete Event System)에서의 명령어 무결성 확보를 위해 시제 논리가 어떻게 적용되는지 학술적으로 분석한다.

1. 선형 시제 논리(LTL, Linear Temporal Logic)의 제어 공학적 도입

시제 논리는 고전적 명제 논리(Propositional Logic)에 시간적 양상(Temporal Modality)을 도입하여, “미래의 특정 시점에 참이 된다(Eventually)”, “항상 참을 유지한다(Always)” 등의 명제를 수식화할 수 있는 정형 기법(Formal Methods)의 핵심 도구이다.

제어 공학 및 소프트웨어 검증 분야에서 선형 시제 논리(LTL)는 FSM의 천이 경로(Transition Path)가 합당한지 검증하는 수학적 근거를 제공한다. LTL의 핵심 오퍼레이터를 PX4의 명령어 검증 로직에 빗대어 설명하면 다음과 같다.

  • \square (Always/Globally): “현재부터 미래의 모든 시점에 명제 p가 참이다.”
  • PX4 적용 예: \square (\text{In-Air} \implies \text{Motor Armed})
  • 기체가 공중에 있는 상태(In-Air)라면, 모터는 항상 시동(Armed) 상태를 유지해야 한다. 이를 위배하는 조종자의 비상 정지(Kill Switch) 명령은 특수한 오버라이드 조건이 아니면 기각되어야 한다.
  • \lozenge (Eventually/Future): “미래의 어느 시점에는 반드시 명제 p가 참이 된다.”
  • PX4 적용 예: \text{Takeoff Command} \implies \lozenge (\text{Altitude} > h_{target})
  • 이륙 명령이 수용되었다면, 기체는 유한한 시간 안에 목표 고도에 도달해야 한다. 만약 제한 시간 내에 도달하지 못하면 시스템은 예외 처리(Failsafe)를 발동한다.
  • \bigcirc (Next): “바로 다음 시점(상태 천이)에 명제 p가 참이다.”
  • \mathcal{U} (Until): “명제 q가 참이 될 때까지 명제 p는 계속 참을 유지한다.”
  • PX4 적용 예: \text{Takeoff State} \ \mathcal{U} \ (\text{Altitude} == h_{target})

이러한 수식화된 명제들은 PX4 커맨더(Commander) 내부의 명령어 수용 가능성성(Acceptability) 판별 로직으로 하드코딩(Hard-coding)되어, 동기/비동기적으로 들어오는 MAVLink 명령을 필터링하는 데 사용된다.

2. PX4-Autopilot의 동적 명령어 검증 구조 분석

PX4는 GCS(QGroundControl 등)나 Companion Computer(ROS 2 연동)로부터 COMMAND_LONG 형태의 MAVLink 제어 패킷을 수신하면, 이를 vehicle_command uORB 토픽으로 변환하여 시스템 전역에 퍼블리시한다. 이때, 명령어 검증은 단순한 if-else 분기문을 넘어 현재 FSM 상태에 기반한 LTL 규칙 검사기로 동작한다.

2.1 맥락 의존적(Context-Dependent) 수용 판별

비행 모드 전환(예: Position \rightarrow Mission) 명령이 하달되었을 때, Commander는 단순 파라미터뿐만 아니라 시간적 맥락을 확인한다.
예를 들어 Mission 모드로의 전환은 **“반드시 이전에 유효한 로컬 포지션(Local Position estimate)이 확보되어 있어야 한다”**는 강한 동시성 제약을 갖는다. 글로벌 포지션 데이터가 간헐적으로 유실(Drop-out)되는 상황에서는 일시적으로(Temporarily) 명령을 지연시키거나 반려(Reject)한다.

2.2 Ardupilot 트랜잭션 처리 구조와의 비교

Ardupilot 역시 강력한 ARMING_CHECK 시스템과 모드 변경 사전 검사(Pre-flight Mode Check) 기능을 제공한다. 그러나 Ardupilot은 다소 경험적인(Heuristic) 분기 논리가 각 모듈별 C++ 클래스 내부에 강하게 결합되어(Tightly Coupled) 있는 반면, PX4는 Commander라는 별도의 중앙집중형 상태 전이 관리자가 비행 통제 로직을 관장한다.
이러한 구조적 차이 덕분에 PX4는 상태 변화를 비순환 그래프(DAG)나 LTL 시퀀스로 모델링하고 추적하기에 더 직관적인 아키텍처를 보유하고 있으며, 정형 검증(Formal Verification) 도구를 연동하기에 상대적으로 유리한 환경을 제공한다.

3. LTL 기반 비행 모드 시퀀스 무결성 검증 (Mermaid 모델링)

아래 명세는 오토파일럿 내부에서 수신되는 명령어의 타이밍 시퀀스와 상태 검증(Validation) 과정을 추상화한 시제 논리 흐름도이다.

stateDiagram-v2
    direction TB
    
    [*] --> Standby (Not Armed)
    
    Standby (Not Armed) --> Preflight_Evaluating : Arm Command Rx
    
    state Preflight_Evaluating {
        [*] --> CheckSensors
        CheckSensors --> CheckPosition : Sensors Valid (Globally)
        CheckPosition --> ValidatingTemporalConstraints : Pos Valid
        ValidatingTemporalConstraints --> Accepted : LTL Condition Met
        ValidatingTemporalConstraints --> Rejected : Fails Condition
    }
    
    Preflight_Evaluating --> Armed : Accepted
    Preflight_Evaluating --> Standby (Not Armed) : Rejected (MAV_RESULT_DENIED)
    
    Armed --> InAir : Takeoff Command Rx
    
    state InAir {
        direction LR
        ValidState --> Failsafe : Sensor Drop (Eventually fails)
        ValidState --> RejectKill : Disarm Rx (Violates Always In-Air rule)
    }
    
    InAir --> Landed : Land Command Rx (Next State)
    Landed --> Standby (Not Armed) : Disarm

위의 도표에서 InAir 상태 내에서의 RejectKill 경로는 \square (\text{In-Air} \implies \text{Motor Armed}) 명제를 통해 공중에서 모터가 임의로 꺼지는 치명적인 사고를 소프트웨어 구문 체계에서 차단하는 동적 명령어 검증 구조의 전형적인 예시를 보여준다.

4. GCS 및 자율 에이전트(ROS 2) 제어를 위한 시사점

시제 논리가 적용된 FSM은 자율 비행 군집이나 Companion Computer 기반의 제어기 설계자에게 중요한 가이드라인을 제공한다. ROS 2(Micro XRCE-DDS) 브리지를 통해 고빈도 Offboard 제어 명령을 하달할 경우, 에이전트는 PX4가 요구하는 시제 조건(\lozenge 및 \mathcal{U})을 만족하도록 루프 주기를 맞춰야 한다.
타임아웃(Timeout) 규정이나 명령어 시퀀싱(Sequencing) 절차(예: Offboard 모드 진입 전에 반드시 Setpoint를 연속적으로 먼저 퍼블리시하여야 함, \bigcirc)를 어길 경우 명령의 무결성이 훼손된 것으로 간주되어 PX4 내부 로직에 의해 즉각 반려 처리된다. 따라서 정교한 GCS 및 비행 제어 애플리케이션 개발자는 PX4의 상태 기계를 LTL 관점에서 역공학적으로 해석하고 그에 맞춘 관제 타이머(Timer) 로직을 설계해야 한다.
