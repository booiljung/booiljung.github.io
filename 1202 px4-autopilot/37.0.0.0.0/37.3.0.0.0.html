37.3. 제어 명령의 무결성(Integrity) 및 안전 인터락(Safety Interlock) 보증 논리

이 장에서는 PX4-Autopilot에서 지상 관제 시스템(GCS, Ground Control Station) 또는 반려 컴퓨터(Companion Computer)로부터 전달되는 제어 명령의 무결성(Integrity)을 어떻게 보증하고, 기체의 치명적인 오작동을 방지하기 위한 안전 인터락(Safety Interlock) 메커니즘이 어떠한 제어 공학적, 소프트웨어적 원리로 설계되어 있는지 심도 있게 분석한다.

1. 제어 명령 무결성(Command Integrity) 검증 아키텍처

무인항공기(UAV) 시스템에서 제어 명령의 무결성은 수신된 명령이 통신 과정에서 변조되지 않았으며, 현재 비행 상태(Flight State)에서 논리적으로 유효한지 확인하는 일련의 검증 과정을 의미한다. PX4는 외부 명령을 MAVLink 프로토콜을 통해 수신하며, 내부적으로는 uORB(Micro Object Request Broker) 통신 미들웨어를 통해 각 제어 모듈로 제어 플로우를 전달한다.

1.1 MAVLink 파싱 로직 및 유효성 검사

GCS나 외부 에이전트가 전송하는 제어 명령은 주로 COMMAND_LONG 전역 메시지 또는 SET_POSITION_TARGET_LOCAL_NED와 같은 로컬 제어 지시자로 전달된다. PX4의 mavlink_receiver 소스 코드 모듈은 패킷의 CRC(Cyclic Redundancy Check)를 통해 데이터 링크 계층(Data Link Layer)에서의 페이로드 손상을 1차적으로 확인하여 구조적 무결성을 보증한다.

1.2 명령 수용성(Command Acceptability) 평가 로직

PX4의 핵심 상태 제어기인 커맨더(Commander) 모듈(src/modules/commander/Commander.cpp)은 수신된 vehicle_command uORB 토픽을 구독(Subscribe)하여 현재 기계의 유한 상태 기계(FSM, Finite State Machine) 내에서 수행 가능성을 동적으로 평가한다.
예를 들어, 기체가 이미 이륙하여 공중에 체공 중인 상태(In-air)에서 치명적인 시동 끄기(Disarm) 명령이 수신될 경우, 시스템은 고도 및 하강 속도 에스티메이터(Estimator) 데이터를 참조하여 추락을 유발할 수 있는 비논리적 명령을 즉각 기각(Reject)한다. 이러한 상태 기반 명령 필터링 논리가 제어 명령 무결성의 핵심 중 하나이다.

2. 안전 인터락(Safety Interlock) 메커니즘과 상태 기계(State Machine) 모델링

안전 인터락(Safety Interlock)은 시스템이 정의되지 않은 불안정 상태나 위험 상태(Hazardous State)로 전이되는 것을 하드웨어적 또는 소프트웨어적으로 차단하는 보증 논리이다. PX4는 하위 레벨의 I/O 프로세서(PX4IO 코프로세서)와 상위 레벨의 비행 제어 관리부(FMU, Flight Management Unit) 간의 상호 상태 감시를 통해 이중 스위칭 보호 구조를 확립하였다.

2.1 하드웨어 및 소프트웨어 인터락의 융합

조종자가 조종기(Radio Control)의 시동 스위치(Arm Switch)를 물리적으로 조작하더라도, 비행 전 점검(Preflight Check)을 통과하여 비행 소프트웨어 인터락(Software Interlock) 상태가 해제되지 않으면 PWM 제어 신호가 모터 액추에이터(Actuator)로 전달되지 않는다. 이는 비행 상태 검증 상태 트리에서 인터락 트리거의 종속성을 부여하는 메커니즘이다.

    [*] --> Init
    Init --> Standby : Preflight Checks Passed
    Standby --> Armed : Arm Command (Interlock Released)
    Armed --> Flight : Takeoff Threshold Reached
    Flight --> Armed : Landed
    Armed --> Standby : Disarm Command
    
    state Standby {
        [*] --> SafetySwitchEngaged
        SafetySwitchEngaged --> ReadyToArm : Hardware Safety Button Pressed
    }
    
    Standby --> Failsafe : Sensor Failure / Loss of Datalink
    Armed --> CriticalFailsafe : Critical State Error

3. 이기종 플랫폼(PX4 vs Ardupilot)의 안전 제어 아키텍처 비교

PX4-Autopilot과 Ardupilot은 명령의 무결성 검증 및 안전 인터락 트리거 구조에서 설계 철학의 극명한 차이를 보인다.

  • PX4-Autopilot 아키텍처: 철저한 마이크로 서비스 기반 아키텍처(Microservice Architecture)와 퍼블리시-서브스크라이브(Publish-Subscribe) 패턴(uORB)을 채택하고 있다. Commander 데몬이 전역 상태(Global State)를 관리하며 락(Lock) 메커니즘을 부여한다. 파라미터 CBRK_*(Circuit Breaker) 계열 등을 설정하여 개발자가 명시적으로 인터락을 우회하지 않는 한, 설계된 FSM 제약 하에서 강제적인 안전 로직이 시스템 전역에 개입한다.
  • Ardupilot 아키텍처: 단일 모놀리식(Monolithic) 제어 루프를 기반으로 설계되어 비행 모드(Flight Mode) 클래스 내부의 분기 처리를 통해 무결성과 안전성을 검사한다. ARMING_CHECK 파라미터 비트마스크를 통해 검사 항목을 세밀하게 제어할 수 있는 유연성을 제공하지만, 상태 관리가 모듈별로 다소 분산되어 있어 수학적 예측 가능성이나 정형 기법(Formal Methods)을 통한 증명에는 PX4의 중앙 집중형 FSM 구조가 더 학술적 분석에 직관적인 이점이 있다.

4. GCS 연동 및 QGroundControl의 시각적 인터락 모니터링

지상 관제 시스템인 QGroundControl(QGC)은 MAVLink HEARTBEAT 및 SYS_STATUS 메시지를 기반으로 주기적인 폴링(Polling) 없이도 PX4의 현재 인터락 셋업 상태 및 센서 상태를 모니터링한다. 무결성 검증 로직에 의해 사용자 명령이 거부될 경우, PX4의 Commander는 COMMAND_ACK 메시지의 result 필드에 MAV_RESULT_DENIED 또는 MAV_RESULT_TEMPORARILY_REJECTED 열거형 상수를 반환한다.
QGC 프론트엔드는 이 응답 코드를 파싱하여, 조작자에게 UI 계층에서 즉각적인 텍스트 및 오디오 경고(Warning) 피드백을 발생시킨다. 이는 인간-기계 상호작용(HMI) 관점에서의 상황 인식(Situation Awareness)을 제고하는 중요 설계이다.

5. 센서 인프라 및 추정기(Estimator) 상태 의존성

명령 무결성을 검증하기 위한 핵심 전제 조건은 센서 관측 모델에 기반한 상태 평가의 무결성이다. PX4의 EKF2 (Extended Kalman Filter 2) 또는 ECL 추정기가 산출하는 위치, 속도, 자세 정보 혁신(Innovation) 파라미터에 오류 스파이크가 감지될 수 있다. 이때 Commander는 해당 텔레메트리를 불순물(Tainted) 데이터로 역학적으로 판별하고, 외부 에이전트 컴퓨팅 경로를 통해 요청되는 동적 오프보드(Offboard) 모드 추종 명령을 즉시 기각한다. 이와 관련된 대표 설정이 COM_ARM_EKF_HGT 와 같은 임계값 파라미터들이다.

6. ROS 2 Ми들웨어 연동 및 Offboard 자율 비행 안전 논리

ROS 2(Micro XRCE-DDS 프로토콜 기반) 브리지를 통해 반려 컴퓨터 에이전트에서 고빈도의 자율 제어 명령(Position, Velocity Setpoints)을 전송할 때, PX4는 통신 주기의 연속성을 매우 엄격하게 감시한다.
COM_OF_LOSS_T 파라미터로 설정된 제한 타임아웃(Timeout) 내에 유효한 OffboardControlMode 데이터 구조체가 갱신 및 서브스크라이빙되지 않으면, PX4의 인터락 논리는 즉각적으로 Offboard 모드를 내부적으로 해제시킨다. 연이어 NAV_DLL_ACT (Data Link Loss Action) 파라미터 정책에 따라 위치 유지(Hold) 모드나 복귀(Return/RTL) 모드로 강제 유한 천이(Transition)하여, 실시간 제어 명령 체계의 단절로 인한 모터 치명 트러스트(Runaway) 현상과 추락 위험을 하드웨어 레벨에 도달하기 이전에 차단한다.
