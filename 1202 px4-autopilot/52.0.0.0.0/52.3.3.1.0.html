52.3.3.1. 비행 관리 프로세서(FMU)와 입출력 보조 프로세서(IO) 간 범용 비동기 송수신(UART) 직렬 통신 데드록(Deadlock) 감시를 위한 워치독 타이머(Watchdog Timer)

본 절에서는 Pixhawk 다중 마이크로컨트롤러(MCU) 아키텍처 내부에서, 메인 비행 관리 프로세서(FMU)와 입출력 보조 프로세서(IO Coprocessor)를 연결하는 범용 비동기 송수신(UART) 직렬 통신이 다양한 원인에 의해 마비(Deadlock)되었을 때 이를 감지하고 시스템을 생존시키는 워치독 타이머(Watchdog Timer, WDT) 기반의 하드웨어/소프트웨어 하이브리드 안전 장치에 대해 다룬다. 본 문서는 PX4-Autopilot v1.14 및 최신 하드웨어 펌웨어 구조를 기준으로 작성된다.

1. FMU와 IO 간 직렬(Serial) 통신 아키텍처 개요

Pixhawk 아키텍처에서 FMU(Flight Management Unit)와 IO 코프로세서는 상호 의존적이지만 물리적으로 격리되어 있다. FMU는 고성능 제어 주기(루프)를 산출하고, IO는 이를 전달받아 서보/ESC로 최종 단계의 PWM 펄스를 출력한다.

이 두 칩셋 간의 커뮤니케이션은 고속의 범용 비동기 송수신(UART) 또는 보조 I2C 버스를 통해 이루어진다. 정상적인 동작 상태(Normal Operation)에서 FMU 내의 px4io 드라이버 로직은 수십 헤르츠(최대 400Hz) 주기로 상태 교환 패킷(Status Packet)과 액추에이터 제어 패킷(Control Packet)을 전송한다.

2. UART 데드록(Deadlock)의 발생 원인

임베디드 비행 시스템에서 내부 통신 마비(Deadlock 혹은 Halt)는 여러 복합적 요인에 의해 야기될 수 있다.

  1. 전자기 간섭 (EMI/RFI): 모터나 고전압 전원부에서 발생하는 강한 전자기파 노이즈가 보드 층을 타고 UART의 RX/TX 라인을 교란하여 패킷 오염(Corrupted Packets)을 유발하고 수신 버퍼에 가비지(Garbage) 데이터를 누적시킨다.
  2. 다이렉트 메모리 액세스(DMA) 충돌: NuttX RTOS 상에서 다른 고순위 인터럽트(High-Priority Interrupt)와의 자원 경합으로 인해 UART DMA 전송 큐(Queue)가 멈추는(Stall) 현상이 발생할 수 있다.
  3. FMU 커널 패닉 (Kernel Panic): 메모리 누수(Memory Leak) 또는 산술 오류(Arithmetic Exception)로 인해 FMU의 운영체제 스케줄러 자체의 동작이 중단된다.

위와 같은 경우 FMU는 제어 루프를 갱신하지 못하지만, 배터리 전원이 끊기기 전까지 IO는 물리적으로 살아있으므로 스스로의 판단이 절실히 요구된다.

3. IO 코프로세서의 워치독 타이머(Watchdog Timer) 설계

IO 프로세서는 FMU의 정상적인 동작 여부를 스스로 판별하기 위하여 내부적으로 워치독 타이머(Watchdog Timer, 하드웨어 또는 소프트웨어 타이머 기반)를 구동한다.

워치독 타이머의 동작 원리는 고전적인 ’데드맨 스위치(Dead Man’s Switch)’와 같다.

  • 타이머 갱신 (Heartbeat Kick): IO 프로세서는 FMU로부터 유효한 형식(Valid CRC 및 Header)의 UART 패킷을 수신할 때마다 WDT 카운터를 초기화(Reset/Kick)한다.
  • 타이머 만료 (Expiry): 만약 특정 허용 시간(예: 50ms) 동안 단 하나의 유효 패킷도 수신되지 않으면 타이머가 0에 도달하여 인터럽트(Interrupt)를 발생시킨다.

4. 타이머 만료 시 IO의 자율 제어 권한 확보(Failsafe)

워치독 타이머가 데드록을 선언(Timeout)하면, IO 프로세서는 즉시 수동적 전달자(Passive Relay)에서 자율 제어자(Autonomous Failsafe Controller)로 모드 전환(Mode Transition)을 수행한다.

  1. 안전 PWM 출력(Safe PWM Outputs): FMU가 마지막으로 지시한 출력을 파기하고, 파라미터로 사전에 정의된 페일세이프 기본값(예: Throttle = 900us (Zero Spin), Roll/Pitch/Yaw = 1500us (Center))으로 즉강 강제 변경하여 기체의 플라이어웨이(Flyaway)를 막는다.
  2. 조종기 직접 통과 (RC Passthrough): 만약 S.BUS / PPM 조종기 수신기가 IO 프로세서에 직접 체결되어 있다면, 조종기 채널 데이터를 직접 디코딩하여 FMU를 거치지 않고 일부 수동 회복(Manual Recovery) 제어를 획득할 수 있도록 허가(옵션 구성에 따름)하기도 한다.
  3. FMU의 재연결 거부: FMU가 일시적으로 복구되어 다시 패킷을 보내기 시작하더라도, 신뢰도를 상실했다고 판단하여 특수한 상태 갱신 교류(Handshake) 절차를 거치기 전까지는 FMU의 제어 출력을 무시(Ignore)한다.

5. C/C++ 소스코드 레벨 설계 예시

아래는 논리적으로 추상화된 IO 프로세서 내의 워치독 감시 인터럽트 서비스 루틴(ISR) 의사 코드이다.

// Pseudo code for IO Watchdog Timer on UART RX
void uart_rx_interrupt_handler() {
    Packet pkt = read_uart_buffer();
    if (validate_crc(pkt)) {
        // 유효한 패킷이 도착하면 워치독 타이머를 연장한다.
        reset_watchdog_timer(FAILSAFE_TIMEOUT_MS);
        apply_pwm_commands(pkt);
    }
}

// OS Timer Tick Handler (1ms 주기)
void systick_handler() {
    if (watchdog_timer_is_expired()) {
        // 타이머 데드록 발생 시
        enter_io_failsafe_mode();
    }
}

void enter_io_failsafe_mode() {
    disable_fmu_override();          // FMU 권한 박탈
    set_pwm_to_failsafe_defaults();  // 비상 정지 최소 출력 인가
    notify_led_buzzer_error();       // 지상 관제 피드백용 청각/시각 알림
}

6. 요약

다중 마이크로컨트롤러 환경에서 워치독 타이머(WDT)는 물리적 연결 레이어(Physical Layer)에서 발생하는 예측 불가능한 직렬 통신 결함을 방어하는 최종적인 안전 릴레이(Final Safety Relay)이다. GCS 개발자나 하드웨어 제조사는 커스텀 비행 제어기(Flight Controller)를 설계할 때, 단순 소프트웨어 락(Lock)에만 의존하지 않고 이와 같은 IO 레벨의 하드웨어 타이머 방어선을 반드시 구축해야만 Ardupilot 및 PX4와 완벽하게 관제 생태계를 호환할 수 있다.
