52.3.2 수동 및 자동 비행 종료 트리거(Trigger)의 다변수 제어 조건 연산 로직

최신 무인항공기 관제 시스템에 있어 비행 종료(Flight Termination) 판단은 기체의 완전한 소실이나 대규모 사고를 결정짓는 핵심 분기점이다. PX4-Autopilot(기준 펌웨어 버전: v1.14.0)은 단순한 단일 임계치 초과(예: 롤 각도 과도 이탈)로 비행 종료를 선언하지 않는다. 이를 방지하고 판단의 정확성을 높이기 위해, 커맨더(Commander) 모듈은 수동 개입(Manual Intervention)과 다수의 센서/소프트웨어 상태를 융합한 자동(Automatic) 다변수 제어 조건(Multivariable Control Conditions)을 연산한다. 본 절에서는 이러한 트리거 로직의 아키텍처와 실제 소스 코드 상의 연산 모델을 상술한다.

1. 개요: 트리거의 이원화 아키텍처

PX4의 비행 종료는 그 발생 원천(Source)에 따라 크게 두 가지로 분류된다:

  1. 수동 비행 종료(Manual Flight Termination): 현장의 안전 통제관(Safety Officer)이나 조종사가 조종기 스위치나 지상 관제 시스템(GCS) UI를 통해 명시적으로 인가하는 락다운(Lockdown) 명령.
  2. 자동 비행 종료(Automatic Flight Termination): 탑재된 비행 제어 컴퓨터(Flight Controller 컴패니언 컴퓨터 및 내부 MCU)가 다중 센서 결함 및 제어 발산을 자체적으로 진단하여 내리는 최후의 자율적 결단.

이 두 가지 트리거는 결국 중앙 커맨더 모듈 내의 파라미터 CBRK_FLIGHTTERM (회로 차단기 파라미터) 설정에 따라 억제되거나 혹은 인가되며, 논리합(OR) 조건으로 동작하여 어느 한쪽의 조건이라도 충족할 시 즉각 시스템 락다운(Lockdown) 노드로 전이시킨다.

2. 수동 비행 종료 연산 로직

수동 비행 종료는 인간의 직관적 판단이 기계의 센서 융합 판단보다 우선한다는 철학(Human-In-The-Loop)에 기반한다.

2.1 조종기(Radio Control) 기반 트리거

RC_MAP_FLT_TERM 파라미터를 통해 특정 PWM 채널을 비행 종료 스위치로 할당할 수 있다. RC 수신기(Receiver)에서 들어오는 신호는 Commander 모듈 내 FailureDetector 혹은 RC 처리 스레드에 의해 평가된다.

  • 연산 조건: 할당된 채널의 PWM 값이 사전 정의된 높은 임계치(High Threshold, 일반적 1750µs 이상)를 초과하고, 이것이 소프트웨어적인 디바운스(Debounce, 노이즈 필터링을 위한 일정 지연 시간 유지)를 통과할 경우 즉각 트리거된다.

2.2 GCS 기반 MAVLink 트리거

QGroundControl(v4.3 기준)과 같은 GCS에서는 화면 상의 비상 정지(Emergency Kill)나 비행 종료 버튼을 누름으로써 트리거할 수 있다.

  • 연산 조건: COMMAND_LONG 패킷을 통해 MAV_CMD_DO_FLIGHTTERMINATION (Command Enum: 185) 명령을 송신한다. PX4는 커맨더 내 mavlink_command_receiver.cpp 로직에서 해당 프로토콜을 수신 및 승인(ACK) 한 후 내부 상태 FSM을 락다운으로 갱신한다.

3. 자동 비행 종료의 다변수 조건 연산 (Multivariable Logic)

현대적 드론의 자동 비행 종료는 고도, 자세, 위치, 센서 헬스(Sensor Health) 등 여러 상태 변수를 독립적으로 평가한 후 복합적인 논리곱(AND)/논리합(OR) 매트릭스를 거쳐 최종 트리거된다.

주요 다변수 조건은 다음과 같다:

  1. 자세/위치 추정 실패 (Estimator Failure):
    EKF2(Extended Kalman Filter) 확장 칼만 필터가 내부 이노베이션(Innovation) 수치의 발산으로 인해 더 이상 수렴하지 못하거나 (estimator_status 토픽 발산), 위치 및 자세 공분산(Covariance)이 임계치를 초과한 경우.
  2. 비정상적 동역학 (Flight Dynamics Anomaly):
    멀티로터의 경우 피치(Pitch)/롤(Roll) 각도가 FD_FAIL_P 및 FD_FAIL_R 파라미터 제원을 침범하여 회복 불능으로 간주될 경우 (예: 후류 진입, 프롭 손상에 의한 물리적 뒤집힘).
  3. 지오펜스 이탈 불가항력 (Geofence Breach & RTL Failure):
    기체가 지오펜스(Geofence)를 위반하였으나, RTL(Return To Launch) 시도마저 실패 혹은 타임아웃(Timeout)된 경우.

4. 소스 코드 구조 분석 (Failure Detector)

PX4의 소스 코드 트리 src/modules/commander/FailureDetector/ 내부의 C++ 아키텍처를 분석하면 다변수 처리가 어떻게 구현되어 있는지 알 수 있다. FailureDetector::update() 루틴은 약 100Hz 이상의 주기로 현재 상태를 지속적으로 평가한다.

// [예시 코드] PX4 FailureDetector 복합 조건 검사 로직 단편
bool FailureDetector::update_status()
{
    bool is_failed = false;

    // 1. 동역학적 이탈 평가 (자세)
    if (_param_fd_fail_p.get() > 0 && fabsf(_vehicle_attitude.pitch) > _param_fd_fail_p.get()) {
        is_failed = true;
    }

    // 2. 외부 비행 종료 명령 OR 통신(RC) 조건 평가
    if (_vehicle_command_flightterm_received) {
        is_failed = true;
    }

    // 3. 필터링 및 디바운스 (히스테리시스 회피)
    if (is_failed) {
        _failure_time = hrt_absolute_time();
        if (_failure_time - _first_failure_time > _failure_timeout_param) {
             _status_changed = true;
             return true; // 최종 비행 종료 승인
        }
    } else {
        _first_failure_time = 0; // 조건 풀림 시 타이머 초기화
    }

    return false;
}

위 로직에서 핵심은 **디바운스 타임(Debounce Time)**이다. 난기류(Turbulence)나 아주 짧은 센서 통신 불량(Glitch)으로 인해 조건이 순간적으로 충족되었다고 해서 기체를 바로 추락시키지 않고, 설정된 시간(_failure_timeout_param) 동안 다변수 조건 충족 상태가 유지될 때만 최종적으로 트리거한다.

5. Ardupilot 대비 통제 철학 및 GCS 관제 차이

PX4-Autopilot과 Ardupilot 모두 수동 및 자동 비상정지 프로세스를 갖고 있으나, 처리 아키텍처에서 분명한 철학적 차이를 보인다.

  • Ardupilot의 Crash Check: Ardupilot은 “기체가 충돌했는가?“라는 관점에서 접근한다. Throttle, Attitude Error, 가속도계 데이터를 휴리스틱하게 조합하여 Crash Check 모듈이 작동을 중지시킨다. 다소 덜 엄격하므로, 경우에 따라 지면에 닿기도 전에 “충돌“로 인식하는 오탐지율 관련 이슈가 보고되기도 한다.
  • PX4-Autopilot의 Failure Detector: PX4는 “비행의 지속 가능성이 통계적/시스템적으로 유효한가?“의 관점으로, 확장 칼만 필터(EKF)의 이노베이션 상태와 여러 다변수(Multi-variable) 조건을 선형 논리로 결합하여 판단한다. 그로 인해 산업용 관제 시스템(Fleet Management) 설계 시 PX4의 아키텍처가 상태 추적성(Traceability) 면에서 압도적으로 유리하다.
  • QGroundControl 연동 관제: QGC는 PX4의 내부 상태 변조를 파악하여 HEARTBEAT와 SYS_STATUS가 비행 종료를 나타낼 경우 맵 UI 상에 에이전트를 적색 아이콘으로 점멸 표기하고 모든 하위 제어 세션을 단절시킨 후, 로그 수집(ULog Download) 안내 UI를 팝업하는 등의 능동적 사후 조치를 행한다.

6. 결론

수동 및 자동 비행 종료 트리거를 분리하고, 이를 다변수(Multivariable) 제어 연산으로 융합하는 PX4-Autopilot의 로직은 단순 비상 정지(Kill Switch)를 넘어 항공 공학 수준의 무결성(Integrity)을 제공하기 위한 구조이다.
QGroundControl, MAVLink 프로토콜 및 내장된 커맨더 모듈 센서 융합기(Estimator)가 상호 연동되어 오작동을 필터링(Debounce)함과 동시에 필요 시각에 전격적 조치를 취하는 이 논리 회로는 대형 자율 운항 시스템의 감항성(Airworthiness)을 담보하는 필수적인 컴포넌트이다.
