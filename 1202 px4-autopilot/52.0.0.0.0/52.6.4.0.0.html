52.6.4. 아두파일럿(Ardupilot) 예외 처리 아키텍처와의 소프트웨어 추상화 계층(Abstraction Layer) 심층 비교

무인항공기 시스템(UAS)의 생존성을 담보하는 예외 처리(Exception Handling) 및 페일세이프(Failsafe) 메커니즘은 오픈소스 비행 제어기(FC) 하드웨어와 소프트웨어가 발전함에 따라 비약적으로 진보해왔다. 현존하는 가장 강력한 두 개의 오픈소스 생태계인 아두파일럿(Ardupilot)과 PX4-Autopilot은 각기 다른 목적과 철학을 기반으로 진화했다. 본 섹션에서는 고수준의 예외 처리 로직이 두 시스템 내부에서 어떠한 소프트웨어 추상화 계층(Software Abstraction Layer)을 거쳐 구현되는지 심층적으로 비교 분석한다.

1. 예외 처리 설계 철학의 근본적 차이

두 비행 제어 시스템 아키텍처의 가장 결정적인 차이는 모듈의 결합도(Coupling) 에 있다.

  • 아두파일럿(Ardupilot): 객체 지향 모델(Object-Oriented Model) 기반의 단일 애플리케이션(Monolithic-like Application) 구조에 가깝게 설계되어 있다. 기종별 핵심 클래스(예: Copter, Plane, Rover) 내에 예외 처리 로직이 밀접하게 결합(Tightly Coupled)되어 있어 직관적이고 즉각적인 제어권 전환이 가능하다.
  • PX4-Autopilot: 마이크로 커널 관점의 분산형 다중 프로세스 아키텍처(Distributed Multi-process Architecture)를 취한다. 센서 수집, 상태 추정기(EKF2), 내비게이터(Navigator), 커맨더(Commander) 모듈들이 NuttX RTOS 상에서 독립된 태스크(Task)로 구동되며, uORB 메시지 버스를 통해 비동기적으로(Asynchronously) 통신한다. 즉, 예외 상황의 감지와 조치가 추상화 계층에 의해 완전히 분리(Loosely Coupled)되어 있다.

2. 아두파일럿의 상속 구조 기반 예외 처리 아키텍처

아두파일럿 소스 코드(예: ArduCopter)에서는 배터리 저전압(Low Battery), 통신 단절(RC/GCS Loss), GPS 글리치(Glitch) 등의 예외 상황을 각 모듈이 감지하면 AP_Vehicle 클래스나 메인 비행 제어 루프를 직접적으로 호출하여 플라이트 모드(Flight Mode)를 즉각 변경한다.

예를 들어 GCS 통신이 단절되었음을 통신 모듈이 파악하면, C++ 내부 객체 메서드 체인을 통해 set_mode(Mode::RTL)과 같이 명령을 직접 내린다. 이 방식은 코드 실행 흐름(Control Flow)이 매우 명확하고, 각 기체 유형별(멀티로터, 고정익) 특화된 예외 처리 로직을 구현하기 쉽다는 장점이 있다. 반면, 새로운 유형의 외부 하드웨어나 컴패니언 컴퓨터(Companion Computer)에서의 ROS2 기반 외부 명령 연동 시, 내부 코드를 깊숙이 수정해야 하는 한계점이 존재한다.

3. PX4-Autopilot의 uORB 통신 기반 예외 처리 추상화

PX4(v1.14 기준)는 커맨더(Commander) 모듈을 중앙 상태 머신(Central State Machine)으로 구성하여 예외 처리 로직을 완전히 추상화했다.

PX4에서는 배터리 데몬(Battery Monitor)이나 센서 평가 모듈(Sensor Check)이 기체 문제를 감지하더라도 직접 모드를 변경하거나 액추에이터를 정지시키지 못한다. 대신 vehicle_status 혹은 battery_status 와 같은 uORB 토픽(Topic) 을 발행(Publish)한다. 커맨더 모듈은 백그라운드에서 이 토픽을 구독(Subscribe)하다가, 사전 정의된 시스템 파라미터(예: COM_LOW_BAT_ACT)와 현재 기체 상태를 대조하여 최종적으로 페일세이프 트리거(Failsafe Trigger) 여부를 결정한다.

이러한 분리형 추상화 계층 구조는 다음과 같은 구조적 이점을 제공한다.

  1. 플러그 앤 플레이(Plug-and-play): 사용자 정의 앱(Custom Application)이나 ROS2 노드가 특정 센서를 흉내 내거나 임의의 결함 토픽을 주입(Injection)하여 시스템을 손쉽게 테스트(SITL/HITL)할 수 있다.
  2. 단일 책임 원칙(Single Responsibility Principle): 모터 제어 모듈, 센서 융합 모듈, 상태 전이 모듈의 코드가 섞이지 않으므로 학술적 연구 및 상용 제품 개발 시 소스 코드 가독성과 모듈 재사용성이 극대화된다.

5. 결론: 학술 및 상용 GCS/ROS2 연동 관점의 시사점

결론적으로, 아두파일럿의 예외 처리는 강력하게 결합된 모놀리식 구조 덕분에 전통적인 RC 조종기와 텔레메트리 환경에서 매우 보수적이고 신뢰도 높은 동작을 보여준다. 반면, PX4-Autopilot은 uORB라는 강력한 유저 스페이스(User Space) 추상화 계층을 거치기 때문에, 외부 제어 컴퓨터(Companion Computer)와의 연동성에서 우위를 점한다.

예를 들어, 자율 에이전트 개발 시 ROS2의 uXRCE-DDS 브리지를 사용하여 PX4의 vehicle_command 토픽에 접근하면, 오프보드(Offboard) 환경에서도 PX4 내부의 복잡한 예외 처리 체인을 그대로 상속받거나 오버라이드(Override)할 수 있다. 따라서 복잡한 임무 수행 및 타 시스템(통합 관제, VIO, 비전 딥러닝 등)과의 긴밀한 통합이 필요한 최신 R&D 및 상용 드론 소프트웨어 아키텍처에서는 모듈화된 추상화 계층을 제공하는 PX4의 구조가 장기적인 유지보수에 더 넓은 확장성을 부여한다고 평가된다.
