52.3.1.1 상태 전이 행렬(Transition Matrix) 상의 논리적 락다운(Lockdown) 조건절 만족 여부 검사 및 예외 처리 무효화

PX4-Autopilot(기준 펌웨어 버전: v1.14.0)의 비상 제어 시스템에서 가장 극단적이고 최종적인 방어 기제는 비행 종료(Flight Termination)에 따른 락다운(Lockdown)이다. 무인항공기 구조에 있어 락다운 상태는 일반적인 페일세이프(Failsafe)나 비상 복귀(RTL, Return To Launch)와는 근본적으로 다른 차원의 제어 단계이다. 본 절에서는 소프트웨어 유한 상태 기계(FSM, Finite State Machine) 내에서 어떻게 상태 전이 행렬(Transition Matrix)을 통해 락다운 조건절을 검사하고, 이 과정에서 발생하는 모든 하위 예외 처리를 무효화(Invalidation)하는지 상세하게 분석한다.

1. 개요 및 락다운(Lockdown)의 개념

무인항공기 제어에서 논리적 락다운은 치명적인 시스템 오작동(비행 제어 컴퓨터 하드웨어 이상, 모든 이중화 센서의 융합 실패, 회복 불가능한 기체 자세 이탈 등)이 발생했을 때, 기체의 추가적인 거동을 강제로 억눌러 2차 인명 피해 또는 재산 피해를 최소화하는 상태이다.
PX4의 커맨더(Commander) 모듈은 현재 기체의 상태와 발생한 이벤트를 기반으로 다음 상태를 결정하는 유한 상태 기계 기반의 **상태 전이 행렬(Transition Matrix)**을 내장하고 있다. 락다운 상태로의 전이가 결정되면, 시스템은 기존에 예약된 어떠한 복구 루틴이나 예외 처리(Exception Handling) 로직도 강제로 무효화(Override 및 Invalidation)하고 액추에이터 출력을 영구적으로 정지(Disarm)시킨다.

2. 상태 전이 행렬(Transition Matrix) 아키텍처

PX4 내비게이션 및 시스템 상태 전이 행렬은 복잡한 다변수 조건들을 논리적 함수로 환산하여 평가한다. 락다운을 트리거하는 조건문은 여러 센서 상태 플래그 및 예측기(Estimator, EKF2 등)의 발산 여부에 의해 계산된다.

락다운 상태 전이를 위한 핵심 논리적 조건절은 다음과 같다:

  1. 내비게이션 발산(Navigation Failure): 위치 제어기(Position Controller)나 자세 제어기(Attitude Controller)에서 navigation_state가 한계치 이상 발산하였으며, 이중화(Redundancy) 센서로의 페일오버(Failover)도 실패한 경우.
  2. 비행 종료 외부 트리거(External Flight Termination Trigger): 지상 관제 시스템(QGroundControl v4.3 이상)을 통해 송신된 명시적인 MAV_CMD_DO_FLIGHTTERMINATION MAVLink 명령 수신 시.
  3. 치명적인 하드웨어 패닉(Critical Hardware Panic): 배터리 전압의 한계치 이하 급강하(하드웨어 차단 임계치) 및 핵심 항공전자장비 간 버스 통신(I2C, SPI, CAN)의 전면 단절 현상 지속.

아래는 커맨더 모듈 내 락다운 상태로 전이하는 논리 조건을 모델링한 유한 상태 기계(FSM) 모식도이다.

    Standby --> Armed : 시동 인가 (Arming)
    Armed --> Failsafe_RTL : 부분적 예외 발생 (Partial Failure)

    Failsafe_RTL --> Lockdown : 오류 에스컬레이션 (Error Escalation)

3. 예외 처리 무효화(Exception Handling Invalidation) 메커니즘

상태 전이 행렬에서 락다운 조건절이 “참(True)“으로 평가되는 순간, PX4 아키텍처는 가장 높은 인터럽트 우선순위(Highest Interrupt Priority)를 부여하여 다른 모든 제어 경로를 폐쇄한다. 이를 **예외 처리 무효화(Invalidation)**라고 부른다.

3.1 무효화 대상 절차

  • 지형 추적(Terrain Following) 및 회피 단절: 장애물 회피 알고리즘 모듈(Obstacle Avoidance)이 퍼블리싱하는 제어 명령을 즉각 무시한다.
  • RTL 및 착륙 루틴 취소: 안전 궤적 생성기(Safe Trajectory Generator)가 계산하는 감속 후 착륙 알고리즘을 강제 폐기한다.
  • 모터 아이들링(Idling) 정지: 기체가 공중에 체공하고 있음을 나타내는 vehicle_land_detected uORB 메시지 상태와 무관하게 모든 PWM(펄스 폭 변조) 신호를 DISARMED 수치(일반적으로 900µs 또는 모터 정지 명령)로 고정한다.

3.2 C++ 소스 코드 관점의 분석

src/modules/commander/state_machine_helper.cpp 및 관련된 비행 종료 로직은 다음과 같은 구조적 흐름을 가진다.

// [예시 코드] PX4 Commander 모듈의 Lockdown 전이 검사 로직 단편
bool check_lockdown_conditions(const vehicle_status_s &status, const vehicle_local_position_s &local_pos) {
    bool is_critical_failure = false;

    // 1. 센서 융합 발산 여부 검사
    if (status.failsafe && status.failsafe_and_user_took_over == false) {
        if (local_pos.v_xy_valid == false && local_pos.v_z_valid == false) {
             is_critical_failure = true;
        }
    }

    // 2. 외부 마이크로컨트롤러(FMU) 통신 마비 검사 등 기타 조건
    // ...

    return is_critical_failure;
}

void transition_to_lockdown(vehicle_status_s &status) {
    // 예외 처리 무효화 단계
    status.nav_state = vehicle_status_s::NAVIGATION_STATE_TERMINATION;
    status.arming_state = vehicle_status_s::ARMING_STATE_STANDBY_ERROR;
    
    // 강제 액추에이터 비활성화 트리거(uORB 토픽 퍼블리시)
    actuator_armed_s armed{};
    armed.armed = false;
    armed.force_failsafe = true; // 일반적 제어로 복구 불가 상태 지정
    armed.lockdown = true;       // 하드웨어 타이머 수준의 차단 플래그
    
    orb_publish(ORB_ID(actuator_armed), actuator_armed_pub, &armed);
}

위의 예시에서 볼 수 있듯, armed.lockdown = true 변수는 액추에이터 제어 모듈(예: PWM_OUT, UAVCAN)이 어떠한 혼합(Mixer) 값도 허용하지 않도록 최상위 게이트 역할을 한다.

4. Ardupilot 대비 아키텍처 및 관제 차이점

Ardupilot은 Crash Check 모듈을 통해 기체의 충돌이나 극단적인 피치(Pitch)/롤(Roll) 각도 이탈을 감지하면 모터를 끄는 방식을 주로 사용한다. 하지만 PX4의 상태 전이 행렬 기반 락다운 로직은 보다 명시적인 **위상 공간 분리(Phase Space Separation)**를 꾀한다.

  1. 상태 설계 철학:
  • Ardupilot은 특정 조건(예: Throttle > 0 이나 고도가 낮아지는 상황에서 가속도 이상)을 휴리스틱(Heuristic)하게 판단하여 Crash를 선언한다.
  • PX4는 수학적으로 정의된 상태 전이 행렬을 구성하여, 락다운 상태 노드(Node)로 진입할 수 있는 엄격한 엣지(Edge) 제약 조건을 명확하게 정의한다.
  1. QGroundControl의 피드백:
    QGroundControl(v4.3)은 MAVLink의 HEARTBEAT 메시지에서 base_mode와 custom_mode를 해석한다. PX4가 락다운 상태에 진입하면, STATUSTEXT MAVLink 메시지를 통해 “CRITICAL: Flight Termination triggered. System Lockdown.“과 같은 형태의 경고를 붉은색 배너와 텍스트-음성 변환(TTS) 인터페이스를 통해 GCS 운영자에게 실시간으로 전송한다. 운영자는 이 상태에서 시동(Arm) 버튼을 눌러도 기체가 반응하지 않음을 확인하게 된다.

5. 자율 에이전트 비행 시의 락다운 제어 통제방안

ROS2 에코시스템(uXRCE-DDS) 기반의 자율 비행 군집 에이전트 운용 환경에서는 단일 기체의 락다운이 전체 시스템에 어떤 파급력을 미칠지가 중요하다. 다중 에이전트(Swarm) 통신 구성 시, 특정 에이전트 노드가 락다운을 선포하면 해당 로직은 즉각적으로 동료 노드나 중앙 관제 컴퓨터(Companion Computer)에 전파되어야 한다.

이를 위하여 락다운 상태 조건 검사 완료 시, 비행 제어기(Flight Controller) 단에서 뿐만 아니라 ROS2 토픽으로 노출되는 /fmu/out/vehicle_status 메시지를 통해 다음과 같은 로직을 동기화해야 한다.

  • 오프보드(Offboard) 노드에서의 제어 메시지(예: TrajectorySetpoint) 전송 즉시 중단.
  • 관제 시스템(Fleet Management System) 화면 상에서 해당 에이전트를 영구 추락(Terminal) 상태로 맵업(Mapping) 하여 후속 경로 계획(Path Planning)에서 회피 영역(Keep-out Zone)으로 간주하도록 재계산.

6. 결론

PX4-Autopilot의 상태 전이 행렬 상 락다운 조건절 검사는 기체의 완전 소실 및 외부 충돌 방지를 위한 최종 안전 보루이다. FSM 구조상 일단 이 조건절이 충족되면 기존의 모든 구조 궤적 및 제어 행위는 무효화되며(Exception Handle Invalidation), 시스템은 완전히 정지된다. 이는 복잡화된 센서와 다중 코어 처리 환경에서 확실하게 생명을 보호하고 피해 통제를 우선시하는 실시간 운영체제(RTOS)의 최상위 우선순위 소프트웨어적 결단으로 이해하여야 한다.
