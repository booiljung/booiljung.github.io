52.1.4. 실시간 운영체제(NuttX RTOS) 내 비상 모듈의 스레드 스케줄링 및 선점형(Preemptive) 우선순위 할당 아키텍처

무인항공기(UAV)의 시스템 구조에서 가장 핵심적인 부분은 센서의 데이터를 처리하고 엑추에이터에 출력을 전달하는 과정에서 발생하는 지연을 최소화하여 조종 안정성을 담보하는 것이다. 이를 위해 최신 PX4-Autopilot(펌웨어 버전 1.14 기준)은 POSIX 프로토콜을 준수하는 NuttX 실시간 운영체제(RTOS)를 기반으로 동작한다. 본 절에서는 NuttX RTOS 기반의 스레드 스케줄링(Thread Scheduling) 기법과 선점형(Preemptive) 최우선 순위가 할당되어 있는 비상 모듈(Emergency Module)의 동작 메커니즘을 상세히 다룬다. QGroundControl(버전 4.3 기준)과의 연동 관점에서의 비상 트리거링 시스템도 함께 분석한다.

1. 다중 스레드(Multi-threaded) 스케줄링 메커니즘의 공학적 배경

비행 시나리오 내에서 비상 컷오프(Kill Switch), 하드웨어 고장, 배터리 서지(Surge)와 같은 안전 임계(Safety-Critical) 상황이 발생할 경우 즉각적인 제어권 회수와 비행 종료(Flight Termination)가 필요하다. 일반적인 범용 운영체제(General Purpose OS)인 Linux 계열과 달리, PX4의 NuttX RTOS는 엄격한 의미의 실시간성(Hard Real-Time)을 보장함으로써 정해진 데드라인(Deadline) 내에 주어진 연산을 완료할 수 있는 확정적(Deterministic) 스케줄링을 구현한다.

PX4 아키텍처는 Commander, Navigator, Sensors, EKF2(상태 추정기), mc_att_control(자세 제어기) 등 다양한 백그라운드 데몬(Daemon)과 애플리케이션 태스크(Task)들이 독립된 스레드 형태로 구동된다. 특정 트리거(Trigger)가 발생하면 NuttX 커널은 레지스터(Register) 및 커널 스택(Kernel Stack)을 보존하는 콘텍스트 스위칭(Context Switching) 작업을 수행한다. 시스템의 신뢰성을 극대화하기 위하여, 치명적 예외 상황을 처리하는 비상 모듈에는 일반 모듈보다 압도적으로 높은 최상위 우선순위가 할당된다.

2. 우선순위 할당 모델 및 선점형(Preemptive) 스케줄링 아키텍처

NuttX는 라운드 로빈(Round-Robin) 방식과 선점형 FIFO(First-In, First-Out) 스케줄링 알고리즘을 혼합하여 지원한다. SCHED_FIFO 스케줄링 정책을 채택한 PX4 모듈들은 자신보다 더 높은 우선순위의 스레드가 나타나지 않는 이상 부여받은 CPU 할당 시간을 끝까지 사용하거나 대기 상태(Waiting/Sleep)로 진입할 때까지 실행을 보장받는다. 우선순위는 시스템 부팅 시 정적으로 선언되며, POSIX 규격에 맞추어 SCHED_PRIORITY_MAX(주로 255)에 근접할수록 높은 지위를 뜻한다.

PX4-Autopilot 내의 프로세스 우선순위 대역폭은 다음과 같이 설정되어 있다.

  1. 인터럽트 처리 및 비상 코어 모듈 (우선순위 대역: 240 ~ 255): 하드웨어 타이머(Timer), I2C/SPI 버스 인터럽트 처리를 위한 워크 큐(Work Queue), 센서 데이터 버스 취득을 통합하는 모듈이다. 비상 상황에서 외부 MAVLink 개입 없이 하드웨어적 직접 명령을 수행하는 Safety Button 스레드 등이 포함된다.
  2. 상태 추정 및 제어부 (우선순위 대역: 240 ~ 250): ekf2 기반 상태 추정기와 mc_att_control / mc_pos_control 등의 로우 레벨 자세 및 위치 제어 스레드이다. 밀리초 단위의 루프 타임을 확보하기 위해 최상위 실행 권한을 가진다.
  3. 네비게이션 및 시스템 관리기 (우선순위 대역: 100 ~ 150): Commander 및 Navigator 데몬으로 비행 모드 변경, MAVLink 파싱 커맨드 승인 및 웨이포인트(Waypoint) 생성 등을 담당한다.
  4. 통신 미들웨어 및 주변 기기 (우선순위 대역: 50 ~ 100): MAVLink 송수신(mavlink 모듈), ROS2 연동용 Micro XRCE-DDS 에이전트, SD 카드 로깅(Logger) 등의 비 동기적 태스크이다.

동기화 및 락(Lock)을 처리할 때 발생할 수 있는 우선순위 역전(Priority Inversion) 현상을 막고 비상 모듈이 무한 대기 상태에 빠지지 않도록 유도하기 위해 우선순위 상속(Priority Inheritance) 메커니즘이 커널 내에 활성화되어 있다.

2.1 스케줄링 아키텍처 다이어그램

비상 모듈의 선점형 스케줄링 프름을 파악하기 위해 다음의 Mermaid 시퀀스를 이해해야 한다.

    participant C as Commander (Medium Priority)
    participant E as EKF2/Control (High Priority)
    participant S as Safety/Emergency Module (Critical Priority)
    
    M->>C: MAVLink Heartbeat 지속 점검 (QGroundControl)
    Note over C,M: 일반 비행 (자율/수동 모드 실행 중)
    E->>E: 400Hz 루프 제어 (Sensor > EKF > Control)
    
    rect rgb(255, 200, 200)
    Note over S: 비상 킬 스위치(Kill Switch) 인터럽트 발생 트리거
    S->>S: 커널 스케줄러 인터럽트 서비스 루틴 (ISR) 진입
    S->>E: 제어 루프 선점 (Preemption) - CPU 즉시 점유
    S->>C: Failsafe 모드 즉시 전파 (uORB Publish)
    C-->>M: GCS로 CRITICAL 비상 메시지 통보
    end
    
    S->>E: 액추에이터 PWM 정지 혹은 안전 출력 하달

이와 같이 비상 시그널(예: RC 조종기의 하드웨어 Kill 스위치나 내부 Watchdog 타이머 에러)이 감지될 경우, 현재 실행 중이던 MAVLink 통신이나 오프보드 경로 계획(ROS2) 프로세스는 NuttX 스케줄러에 의해 강제 중단되고, S 모듈 (Emergency Module)이 시스템 제어권을 완전히 독점하여 모터를 정지시키는 하강 안전 로직을 수행한다.

3. Ardupilot (ChibiOS 통합 구조) 기반 스케줄링과의 아키텍처 비교

PX4-Autopilot 의 NuttX 기반 구조는 Ardupilot에서 채택한 범용적 ChibiOS의 런타임 스레드 관리 기법과 결을 달리한다.
Ardupilot 역시 실시간 동작이 가능하지만, 전통적인 메인 루프(Main Loop)를 도는 모놀리식(Monolithic) 아키텍처 성향을 다수 내포하고 있다. Ardupilot에서는 400Hz의 주 루프가 fast_loop 주기 안에서 순차적 함수 호출에 의존하며, 스레드를 극도로 잘게 분리하기보다는 타이머 스케줄러를 기반으로 하위 태스크를 일괄 처리한다. 비상 판별 로직 또한 메인 루프의 분기문(switch 또는 if-else 검증)에 의존하는 경향성을 갖는다.

반면, PX4-Autopilot 아키텍처는 마이크로커널(Microkernel) 개념을 적극 도입해 개별 기능을 POSIX 스레드로 완전 격리시켰다. 따라서 Sensors, EKF2 그리고 Commander(비상 판별 포함) 등이 완전히 독립된 메모리 컨텍스트와 우선순위를 가진 태스크로 활동한다. 이러한 분산 구조로 인해 PX4는 치명적인 비상 모듈 태스크가 생성되었을 때 주 제어 루프(mc_att_control) 자체를 선점해버리는 강력한 하드웨어 인터럽트 기반 대응에 유리하며, 코드의 결합도(Coupling)를 낮추어 시스템 유지보수와 모듈 별 단위 테스트 역량을 높이는 긍정적 효과를 창출한다.

4. uORB 미들웨어 및 QGroundControl 관제와의 우선순위 융합

오프보드 제어 중이거나 QGroundControl 로부터 자동 비임무(Auto Mission) 경로를 수행 중일 때, 비상 상황 판단은 센서 퓨전 에러, 배터리 임계선 도달, MAVLink 통신 두절 등 여러 요소에서 도출된다. 이러한 상태 진단 변수는 uORB (Micro Object Request Broker) 버스를 통해 시스템 전역으로 전파된다.

QGroundControl(GCS)에서 보내는 조종 명령 패킷은 통신용 모듈(mavlink 스레드)을 통해 들어오며, 이는 앞에서 살펴본 바와 같이 중간 정도의 우선순위를 갖는다. 만원 트래픽 상황 시 통신 병목이 생기더라도 비행 제어 및 비상 감지 센싱 스레드는 통신 수신보다 월등히 높은 240+ 구간의 SCHED_FIFO 우선순위를 선점하기에 기체의 추락이나 통제 불능 현상을 원천적으로 차단한다.

결론적으로, PX4-Autopilot은 다중 에이전트(Swarm) 제어와 ROS2를 통합하는 고도의 복잡한 연산 부하가 인가되더라도 실시간 운영체제의 선점형 우선순위 정책에 힘입어 가장 안전에 직결된 비상 모듈이 시스템 연산 자원의 지휘권을 절대적으로 확보할 수 있도록 설계되어 있다. 이러한 구조적 완결성은 시스템 전체의 결함 허용도(Fault-Tolerance)를 극대화하는 중추 역할을 담당한다.
