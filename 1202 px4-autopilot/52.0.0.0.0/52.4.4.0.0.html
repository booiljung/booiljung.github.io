52.4.4 사고 사후 조사(Post-crash Analysis)를 위한 비행 기록(ULog) 무결성 확보 아키텍처

본 절에서는 기체 추락 등 치명적 사고 발생 후, 정확한 원인 규명(Post-crash Analysis)을 위해 PX4-Autopilot(기준 펌웨어 버전: v1.14.0)이 채택하고 있는 비행 기록(ULog) 시스템의 데이터 무결성(Data Integrity) 확보 아키텍처를 면밀히 분석한다. 지면 충돌 순간의 물리적 충격이나 전원 차단으로 인해 발생할 수 있는 파일 시스템 손상(Corruption)과 데이터 소실을 방지하기 위한 소프트웨어 및 하드웨어 연계 메커니즘을 주로 다룬다.

1. 사고 사후 조사의 중요성과 데이터 소실 메커니즘

항공기 추락(Crash)이 발생하면 가장 먼저 전원 공급 장치(Battery)가 물리적으로 이탈하거나 단락(Short)될 확률이 매우 높다. 이 순간, 메모리 버퍼(Buffer)에 담겨 아직 SD 카드 등 비휘발성 저장 매체로 기록되지 못한 수백 밀리초 분량의 마지막 텔레메트리(Telemetry) 데이터는 영구적으로 소실된다. 이러한 현상은 ’마지막 순간(Last-millisecond)’의 센서 스파이크나 액추에이터 포화(Actuator Saturation) 상태를 분석해야 하는 사고 조사관에게 치명적인 블랙박스 공백을 초래한다.

이를 방지하기 위해 PX4는 대용량 RAM 버퍼링, 실시간 플러시(Flush) 강제화, 그리고 ULog 포맷 자체의 복구 탄력성(Resilience)을 결합한 통합 로깅 아키텍처를 구현하고 있다.

2. ULog 기록 프로세스 및 무결성 보호 아키텍처

PX4의 logger 모듈은 내부적으로 생산되는 수백 개의 uORB 토픽을 구독하여 이를 이진(Binary) 형태의 ULog 포맷으로 압축/직렬화(Serialization)한다. 이 과정에서 병목을 방지하고 전원 차단 시 손실을 최소화하기 위한 데이터 흐름은 다음과 같다.

    A[uORB 토픽\n(Sensor, Control_opt)] -->|Subscribe| B(Logger 모듈\n프론트엔드)
    B -->|직렬화\nSerialization| C[대용량 고리 버퍼\nRing Buffer in RAM]

    D -->|블록 단위 쓰기| E[(MicroSD 카드\nFAT32 파일 시스템)]
    
    F[Commander / Failure Detector] -->|비상 상태 트리거| G[fsync 강제 호출\nImmediate File Sync]
    G --> D

위의 아키텍처에 따르면, 일반적인 순항 비행 시에는 SD 카드의 수명(Wear-leveling)과 CPU 부하를 고려하여 일정량의 데이터가 버퍼에 쌓였을 때만 백그라운드 태스크를 통해 간헐적으로 기록(Write)이 수행된다. 그러나 파괴 검출기(Failure Detector)가 비정상 하강이나 추락 임계 조건을 감지할 경우, logger 모듈은 예약된 모든 버퍼를 즉시 SD 카드로 밀어내는 fsync() 시스템 콜(System Call)을 강제 실행한다.

3. 소스 코드 구현: Fallback 플러싱과 로깅 태스크

PX4 트리 구조 내 src/modules/logger/logger.cpp를 검토하면, 메인 로깅 스레드가 기체의 전원 차단 직전까지 데이터를 살려내기 위해 어떻게 설계되어 있는지 확인할 수 있다.

// 비상 상태 감지 시 동기화 강제 수행 (유사 코드)
void Logger::run() {
    while (!should_exit()) {
        // 일반적인 버퍼 기록 루틴
        int written = write_buffer_to_sd();
        
        // 치명적 오류 파악
        if (_vehicle_status.fd_critical_failure) {
            // 버퍼 내 잔여 데이터를 모두 밀어내고 (Flush) 파일 시스템 동기화 (fsync)
            fsync(_log_file_fd);
            
            // 충격 방지를 위한 로깅 주기 최소화 모드 돌입
            _log_interval = EMERGENCY_LOG_INTERVAL; 
        }
        
        // 잔여 버퍼 크기에 따른 휴면 시간 계산
        usleep(calculate_sleep_time());
    }
}

이러한 로직 덕분에 프랙션 초(Fractional second) 단위로 전원이 완전히 죽기 전 커패시터(Capacitor)에 남은 잔류 전압을 사용하여 핵심 블록을 플래시 메모리에 박아 넣는(Committing) 극적인 파일 복구 메커니즘이 완성된다.

4. Ardupilot의 DataFlash 대비 ULog 포맷의 이점

Ardupilot은 예로부터 .bin 포맷을 기반으로 하는 DataFlash 로깅 시스템을 오랜 기간 발전시켜 왔으나, 이는 매우 가볍고 직관적이지만 포맷 설계의 특성상 파일이 닫히지 않고 비정상 종료 시, 로그 파싱 툴(Mission Planner)에서 헤더/푸터 인식 오류로 인해 후반부 데이터가 절단되는 이슈가 간헐적으로 발생했다.

반면, PX4가 주도하여 개발한 ULog 포맷은 ‘자기 기술적(Self-describing)’ 설계 철학을 바탕으로 한다. 파일 헤더 이후에 매지 데이터 구조체의 포맷이 정의되어 있으며 데이터 청크 단위로 기록이 추가(Append)된다. 따라서 사고로 인해 파일 시스템의 EOF(End Of File) 마커가 제대로 닫히지 않더라도, FlightPlot이나 pyulog 라이브러리를 통해 강제로 깨진 지점 직전까지의 완벽한 복원 및 추출이 가능하다는 아키텍처적 우위를 뽐낸다.

5. 관제 연동: MAVLink 로그 스트리밍 (Log Streaming)

더 나아가 PX4는 기체가 형체를 알 수 없을 정도로 전소되거나 SD 카드가 유실되는 극단적인 사고에 대비하여 QGroundControl 환경으로 실시간 로그 전송을 지원한다.
MAVLink의 LOGGING_DATA 메시지를 통해 기체 내부 마이크로 SD 카드에 기록되는 것과 동일한 ULog 데이터 스트림을 무선 원격 측정(Telemetry Link) 대역폭이 허용하는 한도 내에서 지상국 PC 저장소에 미러링(Mirroring)할 수 있다. 이는 완전한 ‘오프보드 블랙박스(Off-board Blackbox)’ 환경을 구축함으로써, 어떤 형태의 하드웨어 소실에도 불구하고 소프트웨어적 추적(Traceability)과 사고 책임 규명을 보장하는 완벽한 무결성 아키텍처라 볼 수 있다.
