52.2.1. 무선 조종기(RC) 수신기 프로토콜의 하위 계층(Physical Layer) 신호 처리 및 인터럽트 라우팅

안전 임계(Safety-Critical) 시스템인 무인항공기의 제어권 확립을 논할 때, 무선 조종기(Radio Control, RC)로 유입되는 외부 조작 명령의 무결성과 하드웨어 응답 지연 최소화는 무엇보다 중요한 요소이다. 최신 PX4-Autopilot(펌웨어 버전 1.14 기준)은 다중 통신 프로토콜을 단일 직렬 인터페이스 노드로 수용하면서도, NuttX 실시간 운영체제(RTOS)의 하드웨어 인터럽트를 지능적으로 라우팅하여 최단 처리 경로 구조를 구현하고 있다. 이 장에서는 물리 계층(Physical Layer)에서의 직렬 파형 처리와 소프트웨어 단에서의 스케줄링 연계를 상세히 다룬다.

1. 물리 계층(Physical Layer) 및 통신 프로토콜 아키텍처

범용적인 아날로그 펄스 폭 변조 방식인 PWM이나 여러 채널을 시분할 다중화한 PPM 방식 등은 현대 비행 제어기(FC) 아키텍처 관점에서 볼 때 전송 지연 시간(Latency) 조절 및 노이즈 측면에서 극복해야 할 한계를 내포한다. 때문에 안정적인 디지털 직렬 통신(UART) 기반의 프로토콜이 주류 기술로 자리매김했다.

  1. SBUS (Futaba, FrSky 등)
  • SBUS는 전통적으로 100kbps의 비표준 보레이트(Baudrate)와 8 데이터 비트, 짝수 패리티(Even Parity), 2 스톱 비트(8E2) 규칙을 따른다.
  • 물리적 신호 파형이 논리 반전(Inverted)되어 전송되기 때문에, Pixhawk 등 하드웨어 보드의 설계 시 물리 트랜지스터로 신호를 재반전시키거나 고도의 STM32 호환 칩에서는 내부 레지스터를 변경해 소프트웨어 인버터(Inverter) 모드를 켜서 신호를 해독해야 한다.
  1. CRSF (Crossfire, Team BlackSheep 계열)
  • 400kbps 넘는 고속 보레이트와 양방향 송수신 특성(Telemetry 병행 지원)을 갖추어 PX4 내 rc_update 루프 병목을 해소하는 가장 이상적인 프로토콜로 평가받는다. CRC 기반 오류 검증을 포함해 데이터 단절 패킷의 물리적 추적이 훨씬 수월하다.

2. 하드웨어 인터럽트 및 DMA(Direct Memory Access) 라우팅

FC 하드웨어의 특정 입력단 구역(RC IN 혹은 UART RX 핀)으로 유입된 RC 프로토콜 신호는 즉시 CPU 인터럽트 컨트롤러(Nested Vectored Interrupt Controller, NVIC)에 외부 하드웨어 인터럽트(IRQ)를 촉발시킨다. 여기서 중요한 점은 CPU(ex: STM32F7 / H7 시리즈)가 수신된 바이트 스트림을 한 바이트씩 일일이 처리하지 않도록 DMA(Direct Memory Access) 컨트롤러가 개입한다는 것이다.

DMA는 CPU의 연산 자원과 독립적으로 UART RX 버퍼의 내용을 지정된 메모리 구역으로 누적하여 전송해 둔다. 지정된 프레임 종단 시간(Frame End Timeout) 타이머가 폭발하거나 지정된 패킷 길이의 데이터가 전송 완료되었을 때만 커널에 수신 완료 인터럽트(RX Complete Interrupt)를 발생시키기 때문에, Mc_att_control(자세 제어) 및 상태 추정 에어리어(EKF2)와 같은 고부하 태스크에 영향을 미치지 않고도 하위 계층의 노이즈 전파가 차단된다.

3. PX4 소스 코드 기반 모듈 추상화 및 파편화 해결 구조

초기 RC 패킷은 I/O 보조 프로세서(보통 px4io 펌웨어가 구동되는 작은 STM32F1 칩) 단 혹은 직접 FMU(비행 제어 주처리장치) 내부 직렬 통신 데몬에 의해 수합된다.

인터럽트가 처리된 후 PX4의 sensors 혹은 개별 프로토콜 파서(ex: sbus.cpp, crsf.cpp) 루틴은 해당 버퍼에서 16개의 채널 값을 압축 해제하고 디코딩한다. 패킷 검증(CRC 통과 및 오염 회피)이 성공한 유의미한 채널 변동값은 즉각적으로 시스템 전역 미들웨어인 uORB 토픽 버스 위로 input_rc 형태로 퍼블리싱된다.

이러한 모듈형(Modularity) 퍼블리시 메커니즘은 비행 제어기와 RC 수신 로직 간의 소스 코드 결합도(Coupling)를 분산시켜, 추가로 새로운 통신 프로토콜 플러그인을 개발하더라도 다른 제어부 루프 속도를 방해하지 않는 PX4-Autopilot 설계 철학의 정수로 평가된다.

4. MAVLink / ROS2 관제망과의 연동 및 오버라이드 제어 우위

최신의 자율 기술들은 더 이상 물리적 조종기에만 의존하지 않으며 QGroundControl과 같은 지상 관제 시스템이나 보조 컴퓨터 컴패니언의 ROS2 제어권에 많은 부분을 위임한다.

GCS는 MAVLink 메시지인 RC_CHANNELS_OVERRIDE를 보내어 기체의 RC 데이터를 덮어쓸 수 있으며, ROS2 환경에서는 ManualControlSetpoint 토픽을 uXRCE-DDS 브리지를 통해 구독하여 처리한다. 하지만 비행 통제 매니저(Commander) 시스템은 하드웨어 인터럽트로 발생한 input_rc 토픽 데이터를 소프트웨어적 가상 명령 토픽인 RC_CHANNELS_OVERRIDE 보다 항상 우선하거나, 킬 스위치 채널 변동과 같은 치명적(Critical) 변화 감지 시 자동으로 하드웨어 물리 신호로 페일오버(Fail-over) 결정을 내리는 계층형 거부 보호 메커니즘을 내장하고 있다.

5. Ardupilot 아키텍처 대비 구조적 해상도 차이 분석

Ardupilot은 AP_RCProtocol 하위 클래스를 이용하여 메인 스케줄링 타임라인 안에서 순차적 타이머 기법으로 비동기 스트림을 파싱하는 다소 모놀리식(Monolithic) 성향을 띠다. 이러한 통합 방식은 하드웨어와 컨트롤러 프레임워크 간의 메모리 복사 구간이 적어 오버헤드는 작으나, 고성능의 다중 스레드 연산 능력을 극대화하기에는 제한이 존재한다.

반면, PX4는 하드웨어 인터럽트의 DMA 종착 지점에서 데몬 형태의 독립 스레드가 해독을 주도하고 uORB로 배포하는 방식을 택하고 있다. uORB는 스레드 안전(Thread-Safe)하며 각기 다른 스레드가 동기화 병목 없이 읽어갈 수 있기 때문에, 멀티 코어 환경이나 POSIX 기반의 확장에 있어서 프로세스 생명 주기를 독립적으로 제어하기에 아키텍처적으로 유연하다는 근원적 차별성을 지닌다.
