52.5.3. 커맨더(Commander) 스레드의 비상 제어 전용 C++ 클래스 구조 및 스케줄링

1. 개요

PX4-Autopilot의 아키텍처에서 커맨더(Commander) 모듈은 기체의 두뇌 역할을 수행하는 최상위 상태 머신(State Machine)이다. 조종사의 조종기(Radio Control) 입력, 지상 관제 시스템(GCS)의 MAVLink 명령, 기내 배터리 상태, 센서 상태 등을 종합적으로 평가하여 기체의 무장 상태(Armed/Disarmed)와 비행 모드(Flight Mode)를 결정한다.

비상 상황(Emergency)이 발생했을 때, 커맨더는 단순히 “현재 상태“를 바꾸는 것에 그치지 않고, 비상 제어 전용 로직을 즉각적으로 스케줄링하여 하위 모듈(제어 할당기, 내비게이터 등)에 전파해야 한다. 본 장에서는 커맨더 스레드 내부에 구현된 비상 제어 전용 C++ 클래스 구조와 스케줄링 메커니즘을 심도 있게 분석한다.

2. 커맨더 스레드의 C++ 클래스 아키텍처

커맨더 모듈은 단일 통짜 소스 코드가 아니라, 상태 천이(State Transition)의 복잡성을 관리하기 위해 여러 개의 헬퍼 클래스(Helper Class)와 구조체(Struct)로 나뉘어 구성되어 있다. 그 중 비상 제어 및 하드웨어 안전과 직결된 주요 클래스는 다음과 같다.

  • Commander (메인 클래스): 스레드 실행 주체로 Run() 루프를 보유하고 있으며 uORB 업데이트를 폴링한다.
  • HealthAndArmingChecks: 센서, 배터리, CPU 부하 등 하드웨어 및 소프트웨어의 건전성(Health)을 매 주기 평가하여 비행 가능 여부 및 강제 비상 착륙 대상 여부를 판단한다.
  • FailureDetector: 비행 중 기체가 뒤집히거나(Roll/Pitch 한계 초과), 자유 낙하(Freefall) 중이거나, 장시간 제어 명령을 상실한 상태 등을 감지하는 전용 C++ 로직 블록이다.
  • SafetyButton: 하드웨어 안전 스위치 상태를 모니터링하여 강제 무장 해제(Disarm) 여부를 결정한다.

3. 비상 명령 스케줄링(Scheduling) 구조

NuttX RTOS에서 실행되는 커맨더는 WorkQueue 기반 스케줄링 방식 혹은 무한 루프(px4_task_spawn) 형태를 띤다. 비상 제어 관점에서 가장 중요한 것은 **응답 지연 시간(Latency)**의 최소화이다.

3.1 하드(Hard) 비상 처리: Kill Switch / 비행 종료(Flight Termination)

RC 조종기에서 Kill Switch가 활성화되거나 MAVLink를 통해 MAV_CMD_DO_FLIGHTTERMINATION 명령이 수신되면, 커맨더 내부의 handle_command 로직이 이를 최우선으로 인터셉트한다. 이 조건들은 메인 루프에서 일반적인 항법(Navigation) 연산보다 앞서 평가되며, 확인 즉시 actuator_armed.manual_lockdown 또는 actuator_armed.force_failsafe를 true로 설정하여 서브스크라이버들(제어 할당기 등)에게 강제로 브로드캐스트한다.

3.2 소프트(Soft) 비상 처리: 페일세이프(Failsafe) 상태 머신 천이

배터리 소진이나 통신 두절(Data Link Loss)과 같은 상황은 FailureDetector 혹은 HealthAndArmingChecks에서 감지된다. 이 경우 즉각적인 하드웨어 차단보다는 페일세이프 상태 머신 로직에 따라 행동 지침(Action)이 스케줄링된다.

  • RTL (Return To Launch): 복귀 명령 산출
  • Land: 즉각적인 수직 강하 및 착륙
  • Hold: 현재 위치 유지 및 대기

4. Ardupilot (AP_Vehicle) 구조와의 차이점

Ardupilot의 메인 스레드는 AP_Vehicle를 상속받은 Copter, Plane 등 기종별 클래스의 최상단 loop() 함수에 의해 구동되며, 100Hz 이상의 빠른 주기로 update_flight_mode(), failsafe_check() 등의 순차 함수를 호출한다. 비상 제어 로직 또한 기종마다 약간씩 다르게 정의된 메인 루프 콜백 내에서 절차적으로 실행된다.

반면 PX4-Autopilot은 기종 독립적인(Vehicle-Agnostic) 단일 커맨더 모듈을 지향한다. 비상 제어 로직 구조 역시 C++ 다형성에 의존하기보다, 별도로 분리된 FailureDetector 객체가 물리적 파라미터를 분석하고 메인 루프에 플래그를 넘겨주는 합성(Composition) 및 상태 오토마타 패턴을 사용한다. 나아가 RTOS의 태스크 우선순위(Priority) 조정을 통해 uORB 데이터 파이프라인 상에서 가장 높은 반응성을 가져가도록 설계되었다.

5. 결론

PX4의 커맨더 스레드는 단순히 함수들이 나열된 스크립트가 아니라, 비상 상황의 경중을 판단하는 여러 개의 전문화된 C++ 도메인 객체들(FailureDetector, HealthAndArmingChecks 등)로 분산 위임된 견고한 아키텍처를 가지고 있다. 이 설계는 치명적 에러 발생 시 최우선 순위로 스케줄링되어, 기체의 무장 상태를 강제로 차단하는 안전 임계 통제 권한을 단독으로 보장한다.
