52.5.2.1. 가상 3축 토크 및 추력 지시 명령을 물리적 구동기 신호 공간으로 투영(Projection)하는 믹싱(Mixing) 행렬 연산의 강제 정지 처리 블록(mixer_module.cpp)

1. 개요 및 역학적 배경

멀티로터나 고정익 등 다양한 형태의 무인항공기에서 자세 제어기(Attitude Controller)와 위치 제어기(Position Controller)는 순수한 물리적 물리량, 즉 “가상 3축 토크(Virtual 3-axis Torque: Roll, Pitch, Yaw)“와 “추력(Thrust)“으로 명령을 내린다. 하지만 이 명령은 실제 모비빌리티 기체(Vehicle)를 구동할 수 없다. 이를 실제 기체에 장착된 개별 모터와 서보 모터의 회전 속도나 각도 명령어(PWM 또는 DShot 신호 공간)로 변환하는 과정을 믹싱(Mixing) 또는 **제어 할당(Control Allocation)**이라 한다.

PX4-Autopilot의 아키텍처에서 이 과정은 기체의 기하학적 형태(모터 벡터, 모멘트 암)를 선형 행렬 연산(Linear Matrix Operation)으로 투영(Projection)하여 해결한다. 그러나 비정상 상태(Kill Switch 작동, 심각한 센서 오류 등)에서는 이 행렬 연산 자체가 기체를 통제 불능 상태로 만들 위험이 있으므로, mixer_module.cpp(과거 버전) 및 최신 ControlAllocator 스레드 내에는 행렬 연산을 강제 정지시키고 믹서를 우회하는 안전 블록(Safety Block)이 존재한다.

2. 믹싱 행렬 연산(Mixing Matrix Computation)의 원리

자세 제어기에서 출력되는 제어 지시자 벡터를 u = [\tau_x, \tau_y, \tau_z, T]^T로 정의하자. 여기서 \tau는 각 축의 토크, T는 추력을 의미한다. 그리고 실제 n개의 액추에이터 출력을 y = [y_1, y_2, \dots, y_n]^T라 할 때, 믹싱 행렬(Effectiveness Matrix) B는 다음과 같은 관계를 갖는다.

u = B y

제어 할당 과정에서는 목표 u를 달성하기 위한 y를 구하기 위해 의사 역행렬(Pseudo-inverse) 연산이나 최적화 기법을 사용한다. 즉, y = B^{+} u를 연산하는 부동소수점 수학적 파이프라인이 믹싱의 핵심이다.

3. 강제 정지(Force Failsafe/Lockdown) 처리 블록 아키텍처

정상적인 비행 상태에서는 매 주기(Loop)마다 위의 부동소수점 행렬 곱셈이 일어난다. 하지만 manual_lockdown 또는 비상 정지(Kill Switch)가 감지되었을 때는 이 복잡한 투영 파이프라인으로 진입하는 것조차 위험하다.

3.1 처리 메커니즘

최신 제어 할당 아키텍처(PX4 v1.14 이상)와 이전 mixer_module.cpp 체계 모두에서 믹서 모듈은 uORB를 통해 vehicle_status 및 actuator_armed 토픽을 지속적으로 폴링(Polling)한다.

  • 상태 평가 (Condition Evaluation): 믹싱 행렬 곱셈을 호출하기 직전, armed.manual_lockdown 속성을 검사한다.
  • 파이프라인 우회 (Pipeline Bypass): 해당 속성이 true일 경우, 투영 행렬 함수를 호출하지 않고 Disarmed 상숫값(보통 PWM 900이나 DShot 0 커맨드)으로 매핑 배열(Array)을 덮어쓴다(Overwrite).
  • 락 프리 처리 (Lock-free Execution): 이 과정은 커널 단위의 컨텍스트 스위칭 지연을 피하기 위해 동일한 스레드 스택 안에서 락 프리(Lock-free)로 작동하여 지연 시간을 나노초(ns) 단위로 통제한다.

4. Ardupilot (AP_Motors) 대비 차이점

Ardupilot 아키텍처는 AP_Motors 클래스의 다형성(Polymorphism)을 기반으로 동작한다. 예를 들어 AP_MotorsMatrix가 믹싱 행렬 곱셈을 직접 수행하고, E-Stop 상황 시에는 내부 output() 함수단에서 논리적으로 출력을 덮어쓴다.

PX4-Autopilot은 이와 달리 RTOS 시스템인 NuttX의 모듈성을 적극 활용하여 믹서 모듈 자체가 독립적으로 동작한다는 점에서 차이가 있다. PX4의 mixer_module.cpp 또는 최신의 ControlAllocator는 데이터 브로커(uORB)를 통해 명령을 수신받아 투영 연산을 수행한 뒤 다시 uORB(actuator_outputs)로 내보낸다. 따라서 PX4에서는 모듈의 제일 앞단에서 uORB 제어 플래그를 읽어 부하가 큰 투영 행렬 연산 단계를 선제적으로 “강제 정지(Cut-off)“시키는 아키텍처가 매우 명확하게 구현되어 있다. 이는 CPU 사이클 낭비를 줄이고 시스템 크래시 시 예측 가능성을 확보하는 데 유리하다.

5. 결론

PX4의 믹서 파이프라인에서 가상 명령을 물리적 신호로 투영하는 행렬 연산 블록은 시스템 자원을 가장 많이 사용하는 구간 중 하나이다. 비상 상황에서 이를 즉각적으로 우회하고 하드웨어 제어권으로 최소 명령을 내려보내는 강제 정지 처리 블록은, UAS 시스템의 “임계 안전(Safety-Critical)” 원칙을 달성하는 핵심 소프트웨어 장치라 할 수 있다.
