52.3.3. 메인-서브 다중 마이크로컨트롤러(MCU) 간의 상태 불일치(Inconsistency) 해소 및 명령 동기화

본 절에서는 Pixhawk와 같은 고신뢰성 다중 마이크로컨트롤러(MCU) 아키텍처 상에서, 비상 제어(Flight Termination, Kill Switch) 시스템이 활성화될 때 **메인 비행 관리 프로세서(FMU)**와 입출력 보조 프로세서(IO Coprocessor) 간의 발생 가능한 상태 불일치(State Inconsistency)를 해소하고 제어 권한을 동기화하는 메커니즘을 심도 있게 분석한다. 본 기술 문서는 PX4-Autopilot v1.14 및 NuttX RTOS 환경을 기준으로 기술된다.

1. 무인항공기 다중 MCU 아키텍처의 필요성

산업용 무인항공기(UAV)의 하드웨어(예: Pixhawk 4/6 시리즈)는 단일 장애점(Single Point of Failure)을 극복하고 실시간 I/O 타이밍을 보장하기 위해 역할이 엄격하게 분리된 이원화 아키텍처를 채택한다.

  • FMU (Flight Management Unit): 고성능 마이크로컨트롤러(예: STM32H7 등)가 탑재되며, EKF 상태 추정, 경로 계획(Path Planning), uORB 메시지 브로커, MAVLink 통신 등 고성능 연산을 전담한다.
  • IO (Input/Output Coprocessor): 비교적 단순한 마이크로컨트롤러(예: STM32F1 등)가 탑재되며, 조종기(RC/S.BUS) 시그널 파싱, 하드웨어 안전 버튼 감지, 그리고 모터/서보(ESC)로의 물리적 PWM 신호 출력을 전담한다.

2. 상태 불일치(Inconsistency) 및 제어 권한 경합의 문제

비상 제어 상황에서는 FMU와 IO 간의 상태 머신이 항상 동일해야(Synchronized) 한다. 그러나 하드웨어 직렬 통신 결함이나 소프트웨어 스케줄링 지연으로 인해 다음과 같은 불일치(Inconsistency) 오류가 발생할 수 있다.

  1. FMU Panic (소프트웨어 붕괴): FMU 커널 패닉(Kernel Panic)이나 하드폴트(HardFault) 발생 시, FMU는 PWM 중지 명령을 내릴 수 없다. 이때 IO 프로세서는 FMU의 통신 단절(Heartbeat Loss)을 스스로 인지하고 독립적인 페일세이프(Failsafe)나 모터 셧다운(Shutdown) 제어로 이행해야 한다.
  2. IO Overriding (하드웨어 인터럽트): 안전 스위치(Safety Switch)나 조종기의 Kill Switch가 IO에 직접 체결된 경우, IO는 FMU의 비행 명령을 무시하고 하드웨어 계층에서 즉각적으로 출력을 차단한다. 이때 FMU는 자신이 명령하는 제어 출력과 실제 IO를 통해 나가는 출력이 다르다는 것을 즉시 인지(Feedback)해야 한다.

3. 상태 동기화 및 셧다운(Shutdown) 레지스터 아키텍처

PX4는 FMU와 IO 간의 통신을 위해 전용 직렬 프로토콜(주로 UART 또는 I2C 기반 버스)을 사용하며, 상태 동기화를 위해 레지스터 기반의 패킷 교환(Register-based Packet Exchange) 모델을 활용한다.

3.1 통신 프로토콜 및 동기화 흐름

FMU에 위치한 px4io 드라이버는 초당 다수 회(예: 400Hz) IO 프로세서의 레지스터 메모리 맵(Memory Map)을 폴링(Polling) 및 업데이트한다. 비상 사태 시 동기화는 다음의 흐름을 지닌다.

sequenceDiagram
    participant RC as RC Receiver
    participant IO as IO Coprocessor (STM32F1)
    participant FMU as FMU (STM32H7 / Commander)
    participant ESC as PWM ESC

    RC->>IO: Kill Switch Trigger (S.BUS)
    IO->>IO: Set internally SAFETY_OFF/ARMED = false
    IO->>ESC: Force PWM to failsafe/zero (Hardware override)
    
    loop Every Tick (400Hz)
        FMU->>IO: Request Status Registers
        IO-->>FMU: Return ARM_STATUS, FAULT_STATUS
    end

    FMU->>FMU: Detect IO is Disarmed by Hardware
    FMU->>FMU: Trigger uORB (vehicle_status transition)
    FMU->>FMU: Abort Estimation & Mission Logic

4. 소스 코드 아키텍처 분석

FMU가 IO의 제어 권한을 명시적으로 회수하거나 셧다운 명령을 동기화하는 핵심 로직은 src/drivers/px4io/px4io.cpp에 구현되어 있다.

// PX4IO Driver - Sending Flight Termination (Force Disarm) to IO
bool PX4IO::force_flight_termination()
{
    // IO 측의 상태 레지스터에 명시적인 비상 정지 비트(Flag)를 쓴다(Write)
    uint16_t setup_reg = io_reg_get(PX4IO_PAGE_SETUP, PX4IO_P_SETUP_ARMING);
    setup_reg |= PX4IO_P_SETUP_ARMING_TERMINATE_FLIGHT; // 비상 정지 비트 할당
    setup_reg &= ~PX4IO_P_SETUP_ARMING_FMU_ARMED;       // FMU 시동 비트 제거
    
    int ret = io_reg_set(PX4IO_PAGE_SETUP, PX4IO_P_SETUP_ARMING, setup_reg);
    
    if (ret == OK) {
        PX4_WARN("IO Coprocessor synchronized: Flight Terminated");
        return true;
    }
    
    PX4_ERR("Failed to synchronize termination state with IO");
    return false;
}

위의 C++ 의사 코드에서 보이듯, FMU는 소프트웨어적으로 결정된 비행 종료(Flight Termination) 명령을 IO의 특정 레지스터(PX4IO_PAGE_SETUP)에 매핑된 비트마스크를 통해 물리적으로 전파한다. 만약 이 통신이 실패하더라도 IO 자체의 워치독 타이머(Watchdog Timer)가 개입하여 이중 안전(Dual Safety)을 도모한다.

5. Ardupilot 대비 아키텍처 비교

AP_IOMCU 드라이버를 사용하는 Ardupilot 역시 PX4와 매우 유사한 하드웨어 레지스터 제어 구조를 가진다.

그러나 PX4-Autopilot은 **uORB(Micro Object Request Broker)**라는 강력한 미들웨어를 통해 IO 드라이버와 상위 자율 관제 로직(commander, navigator) 간의 의존성을 완벽히 분리(Decoupling)한다. 즉, IO에서 불일치가 감지되면 px4io 드라이버가 즉각 uORB의 actuator_armed 및 vehicle_status 토픽을 갱신하며, 이 토픽을 구독하는 MAVLink 시스템이나 제어기 모듈이 1주기(Tick) 이내에 연쇄적으로 정지 로직을 수반하도록 설계되어 있어 RTOS 스케줄링 관점에서의 모듈화 밀집도가 돋보인다.

6. 요약

다중 마이크로컨트롤러 배포 환경에서의 비행 역학 기반의 비상 정지(Kill Switch) 및 비행 종료 연동은, 단순한 소프트웨어 플래그 뿐만 아니라 하드웨어 I/O 레지스터(Memory Map)의 즉각적인 검증 및 덮어쓰기(Overwriting)를 수반한다. 고신뢰성 무인항공기를 설계하는 공학자는 FMU와 IO 간 프로토콜 병목(Bottleneck) 현상을 파악하고, 단 한 프레임의 제어 권한 탈선(Derailment)도 허용치 않는 강건한 레지스터 인터페이스 동기화 전략을 수립해야 한다.
