52.5.3.2. 상태 전이 토폴로지(State Transition Topology)를 검증하여 허가되지 않은 비행 모드 변경을 차단하는 보호 분기문(Guard Condition) 최적화 로직

PX4-Autopilot 시스템에서 다중 에이전트 제어 및 GCS(Ground Control Station) 관제 환경을 운용할 때, 시스템 안전성의 핵심은 예기치 못한 비행 상태의 변경을 원천적으로 차단하는 것이다. 커맨더(Commander) 스레드 내부에서 비행 모드(Flight Mode)는 엄격한 상태 전이 토폴로지(State Transition Topology)에 따라 변화하며, 이 과정에서 허가되지 않은 모드 변경을 검증하기 위해 최적화된 보호 분기문(Guard Condition) 검사 파이프라인이 동작한다.

본 절에서는 PX4-Autopilot(Firmware v1.14 기준)이 채택하고 있는 상태 관리 구조를 이해하고, MAVLink, QGroundControl(v4.3 기준), 그리고 ROS 2 환경에서 잘못된 비행 모드 변경을 방지하기 위해 어떠한 검증 로직이 수행되는지 분석한다.

1. 상태 전이 토폴로지(State Transition Topology)의 개념과 안전성 검증

기체의 비행 상태와 모드는 커맨더 모듈 내부의 유한 상태 머신(Finite State Machine, FSM)에 의해 관리된다. 기체는 시동 해제(Disarmed), 시동(Armed), 이륙(Takeoff), 임무(Mission), 복귀(Return) 등 여러 상태를 가질 수 있으며, 임의의 상태에서 다른 상태로 전환될 수 있는 경로는 제한되어 있다.

보호 분기문(Guard Condition)은 상태 전이가 유효한지 검사하는 로직이다. 예를 들어, 기체가 GPS Lock을 획득하지 못한 상황에서 자율 비행 모드(Mission)로 전환을 요청하면 커맨더 모듈의 보호 분기문은 이를 차단해야 한다.

1.1 Mermaid를 통한 상태 전이 검증 모델 설계

다음은 PX4 커맨더 모듈이 비행 모드 변경 요청을 처리할 때 검증하는 상태 전이 네트워크와 보호 분기문을 단순화한 Mermaid 다이어그램이다.

stateDiagram-v2
    [*] --> Manual_Mode : 초기화(Init)
    Manual_Mode --> Altitude_Mode : 센서 유효성 검사 통과 (비행 제어기)
    Altitude_Mode --> Position_Mode : 로컬 위치 지표(Local Position) 유효
    Position_Mode --> Mission_Mode : 글로벌 위치 지표(Global Position) 및 임무 데이터 유효
    
    state 유효성검사 <<choice>>
    Mission_Mode --> 유효성검사 : GCS 또는 MAVSDK의 Offboard 전환 요청
    유효성검사 --> Offboard_Mode : [Guard: 외부 제어 신호(Offboard Signal) 유효]
    유효성검사 --> Mission_Mode : [Guard: 외부 제어 신호 시간 초과(Timeout) / 거부(Denied)]
    
    Offboard_Mode --> Return_Mode : 통신 단절(Data Link Loss) 감지
    Return_Mode --> Land_Mode : 목표 지점 도달

이러한 전이 토폴로지는 특정 센서의 고장이나 통신 단절 시 치명적인 모드로 강제 진입하는 것을 방지하는 중추 역할을 한다.

2. PX4 소스 코드 내 보호 분기문 최적화 로직 분석

비행 모드 변경은 MAVLink 명령(MAV_CMD_DO_SET_MODE)을 통해 수신되며, 이는 uORB 메시지인 vehicle_command를 통해 커맨더 스레드로 전달된다. 커맨더 스레드에서는 Commander::set_main_state()와 같은 함수 호출 체인을 거치며 보호 분기문을 수행한다.

2.1 최적화된 상태 플래그(Status Flag) 비트 연산 검증

PX4-Autopilot은 상태 전이의 유효성을 실시간 운영체제(NuttX)의 높은 스케줄링 빈도 내에서 부하 없이 처리하기 위해, 다중 if-else 체인을 피하고 비트 마스킹(Bit Masking)과 구조체 내 불리언 플래그 기반의 상태 해시(Hash) 또는 플래그 레지스터 검증 기법을 활용한다.

src/modules/commander/state_machine_helper.cpp에 구현된 전이 모듈 로직의 핵심 아이디어는 다음과 같다.

  • 사전 조건 검증(Pre-condition Check): vehicle_status_s 토픽과 vehicle_local_position_s 등의 토픽을 구독하여 현재 시스템 상태가 새로운 모드의 최소 요구 조건(예: EKF2 필터 수렴 완료 여부)을 만족하는지 즉시 검사한다.
  • Fail-Fast 방식의 조기 종료 분기: 조건을 만족하지 못할 경우 하위 함수 체인으로 내려가지 않고 즉각적으로 TRANSITION_DENIED를 반환하여 연산 리소스 낭비를 차단한다.
// O(1)에 가까운 상태 전이 보호 분기문 논리 예시 (PX4-Autopilot 개념적 코드)
transition_result_t set_main_state(vehicle_status_s &status, bool is_armed, main_state_t new_state) {
    // 1. 상태 전환이 필요 없는 경우 조기 반환 (Guard Condition 최적화)
    if (status.nav_state == new_state) {
        return TRANSITION_NOT_CHANGED;
    }

    // 2. 위험 모드 변경 시 필수 조건 유효성 검사 (Fail-Fast)
    if (new_state == navigation_state_t::OFFBOARD) {
        if (!status.offboard_control_signal_found) {
            return TRANSITION_DENIED_NO_OFFBOARD_SIGNAL; // 허가되지 않은 모드 변경 차단
        }
    }
    
    if (new_state == navigation_state_t::MISSION) {
        if (!status.is_valid_global_position) {
            return TRANSITION_DENIED_NO_GLOBAL_POS; // 글로벌 위치 픽스 없이 전환 불가
        }
    }

    // 3. 상태 변경 적용
    status.nav_state = new_state;
    return TRANSITION_CHANGED;
}

이와 같은 보호 분기문 구조는 상태 변경 시 발생할 수 있는 교착 상태(Deadlock)나 레이스 컨디션(Race Condition)을 방지하며, O(1)에 가까운 연산 복잡도를 확보하여 루프 수행 주기를 준수하도록 최적화되어 있다.

3. ArduPilot과의 소프트웨어 아키텍처 비교

ArduPilot의 경우 상태 전이 로직은 비행체 유형(Copter, Plane)별로 오버라이딩(Overriding)된 set_mode() 함수 내부에서 처리된다. ArduPilot은 mode.cpp와 같은 컨트롤러 구현체 내에서 하ード코딩된 다중 조건문(if-else 블록 위주)을 통해 상태 검증을 수행하는 경향이 강하다.

반면, PX4-Autopilot은 커맨더 스레드가 vehicle_status uORB 토픽을 독립적으로 구독해 중앙 집중적으로 상태 관리를 처리하며, 상태(State)와 제어(Control)의 역할 및 논리 계층이 명확히 분리된 이벤트 기반 통신(Message-oriented) 아키텍처를 따른다. 이 분리로 인해 PX4-Autopilot은 새로운 비행 모드를 시스템 토폴로지에 삽입하거나 테스트(SITL/HITL)할 때 상대적으로 부작용(Side Effect) 파악이 용이하다.

4. GCS (QGroundControl)와의 비대칭 통신 및 피드백 제어

상태 전이 모델을 바탕으로 GCS는 통신 단절이나 예외 상황 시 시스템의 인적-기계적 상호작용(Human-Machine Interaction) 안전망 역할을 한다. MAVLink 프로토콜을 통해 사용자가 QGroundControl 프론트엔드 버튼(예: “재배치(Reposition)” 또는 “임무(Mission)”)을 누를 경우 다음의 파이프라인이 실행된다.

  1. GCS 명령 전송: QGC에서 MAV_CMD_DO_SET_MODE 명령 전송.
  2. PX4 커맨더 보호 분기문 평가: 기체 내 센서 데이터 결함 등으로 보호 분기문이 작동하여 변경이 차단됨.
  3. 거부 응답 수신: PX4는 즉각적으로 MAV_RESULT_DENIED 패킷과 에러 텍스트 STATUSTEXT(“Rejecting mode switch”)를 GCS에 반환.
  4. QGC 롤백 메커니즘: QGC 내부의 차량(Vehicle) 관리 매니저는 명령이 실패했음을 인식하고 UI에서 전환 애니메이션을 취소(Rollback)시켜 사용자에게 상태 비정상을 시각/청각적 피드백으로 전달.

5. 자율비행 에이전트(ROS 2/MAVSDK) 연동 시의 고려 사항

드론 군집 비행(Swarm)이나 컴퓨터 비전 기반 객체 인식 비행 시에는 ROS 2 미들웨어(Micro XRCE-DDS)를 이용하거나 MAVSDK 에이전트를 통한 외부(Offboard) 제어를 광범위하게 사용한다.

이러한 외부 컴퓨팅 유닛은 기체의 실제 펌웨어 상태나 이중화 센서 결함을 완벽히 파악하기 어렵다. 따라서 PX4-Autopilot 내부의 상태 전이 토폴로지가 최종적인 ’안전 문지기(Gatekeeper)’로 작동해야 한다. ROS 2 자율 에이전트가 악의적이거나 소프트웨어 버그로 인해 잘못된 시그널을 통해 수만 번의 모드 전환 트래픽을 유발하더라도, PX4 내부 보호 로직은 무시(Drop) 처리 또는 에지 케이스(Edge Case) 방어 로직을 통해 비행 모드 오염(Pollution)을 완벽하게 차단한다.

결론적으로 PX4-Autopilot은 명확하게 설계된 상태 전이 토폴로지와 마스킹된 플래그 수준의 보호 분기문을 바탕으로, 허가되지 않은 모드 변경의 개입을 차단하여 신뢰할 수 있는 무인항공기 환경을 보장한다.
