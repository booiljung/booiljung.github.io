52.6.2. 의도치 않은 조작 방지를 위한 2단계 승인(Two-step Confirmation) 슬라이더의 터치 스크린 이벤트 리스너(Event Listener) 처리와 입력 타임아웃 롤백(Rollback) 알고리즘

지상 관제 시스템(GCS)에서 시동(Arming), 시동 해제(Disarming), 이륙(Takeoff), 복귀(RTL)와 같은 중요 비행 제어 명령은 기체의 생존을 좌우한다. 터치스크린 기반의 모바일 디바이스 환경에서는 주머니 속 마찰, 땀방울, 혹은 당황한 조종사의 미세한 손떨림으로 인하여 치명적인 “고스트 터치(Ghost Touch)” 및 오조작이 발생할 수 있다.

PX4-Autopilot의 기본 GCS인 QGroundControl(QGC)은 이를 원천적으로 방지하기 위해 일반적인 1차원 클릭 버튼 대신 조작의 의도성을 명확히 요구하는 2단계 승인(Two-step Confirmation) 슬라이더 UI를 핵심 제어 인터페이스로 채택하고 있다. 본 절에서는 이러한 슬라이더가 이벤트 리스너 레벨에서 어떻게 동작하며, 안전 강화를 위해 타임아웃 롤백 알고리즘이 어떻게 구현되어 있는지 분석한다.

1. 2단계 승인(Two-step Confirmation) 슬라이더의 개념

단순한 탭(Tap) 이벤트와 달리 2단계 승인 슬라이더는 사용자의 ’의도적 행위(Intentional Action)’를 검증하는 인지적 안전 장벽이다.

  1. 1단계 (인지 및 진입): 슬라이더 컴포넌트를 터치하고 활성화 상태로 진입.
  2. 2단계 (승인 및 실행): 손가락을 떼지 않은 상태로 슬라이더를 시작점에서 끝점까지 일정한 궤적을 그리며(Drag) 이동한 후, 최종 위치에서 릴리즈(Release).

이 구조는 오조작을 거의 0%에 수렴하게 만들어 사용자 과실로 인한 치명적 MAVLink 명령(MAV_CMD) 전송을 차단한다.

2. 터치 스크린 이벤트 리스너(Event Listener) 처리 아키텍처

QGC 프론트엔드는 QML과 Qt 프레임워크를 기반으로 구성된다. 슬라이더 컴포넌트(SliderButton.qml 또는 ConfirmationSlider.qml 유사 모듈) 내 이벤트 리스너는 일반적인 MouseArea 혹은 MultiPointTouchArea를 확장하여 복합 이벤트를 처리한다.

2.1 주요 이벤트 콜백(Callback) 파이프라인

  • onPressed: 사용자가 슬라이더 섬(Thumb) 컴포넌트를 최초로 터치한 순간. UI 시각적 피드백(하이라이트 또는 크기 확대)을 제공하며, 드래그 위치 변수 값을 스크린 좌표계로 기록(Capture)한다.
  • onPositionChanged (드래그 감지): 터치가 유지된 채 좌표가 이동하는 동안 수시로 호출. 이동 한계를 프레임(Frame) 내에서 제한(Clamp)하고 애니메이션 진행률(Progress)을 업데이트한다. 수직 방향의 이동(Y축 변화)이 과도하게 클 경우, 드래그 취소로 간주하는 방어 로직이 적용된다.
  • onReleased: 터치 스크린에서 손을 뗀 시점. 섬(Thumb) 컴포넌트가 슬라이더 전체 길이의 임계치(Threshold, 통상 95% 이상)를 초과했는지 판별한다.

3. 입력 타임아웃 롤백(Rollback) 알고리즘

실제 비행 환경에서는 조종사가 끝까지 슬라이더를 밀었다가 심리적 망설임으로 손을 수 초간 멈추거나, 터치 불량으로 인해 좌표 이벤트가 끊기는 현상이 빈발한다. 시스템은 이를 무한정 대기할 수 없으며, 롤백(Rollback) 알고리즘을 통해 UI의 무결성을 유지해야 한다.

3.1 롤백 작동 메커니즘

  1. 탄성 애니메이션(Spring Animation): 사용자가 임계치에 도달하지 않은 상태에서 손을 떼면(onReleased 발동), 슬라이더 버튼은 즉각적으로 부드러운 스프링 효과를 내며 좌표 x: 0 위치로 되돌아간다. 이는 상태가 승인되지 않았음을 사용자의 시각 및 신경에 반사적으로(Reflexively) 인지시킨다.
  2. MAVLink 커맨드 대기 상호작용(Timeout Rollback): 사용자가 슬라이더를 끝까지 밀어 CONFIRMED 이벤트를 발생시켰어도, QGC 백엔드(Vehicle Manager)가 기체로 C++ 기반 송신 큐를 통해 COMMAND_LONG을 전달한 후 기체로부터 COMMAND_ACK를 지정된 타임아웃(예: 500ms~1000ms) 내에 받지 못하면 명령이 실패한 것으로 판단한다. 이때 백엔드에서 신호 파이프라인(Signal Pipeline)을 통해 QML UI로 실패 상태를 전달하고, 끝에 도달했던 슬라이더 섬은 다시 초기 위치로 강제 롤백된다.

이 타임아웃 롤백은 드론 시스템과의 실제 상태 비동기성(State Asynchrony)을 방어하는 관제 소프트웨어의 필수 안전 장치다.

4. ArduPilot (Mission Planner) 아키텍처와의 UI 설계 노선 비교

전통적인 데스크톱 기반의 GCS인 Mission Planner(주로 ArduPilot 환경)는 비상 상황에서의 승인 방식으로 윈도우 네이티브의 **모달 다이얼로그(Modal Dialog Box)**를 자주 활성화한다.

  • Mission Planner 방식: 화면 중앙에 “정말로 이륙하시겠습니까(Are you sure)?“라는 팝업 창이 뜨고, 사용자가 [예(Yes)] / [아니오(No)] 버튼을 마우스로 클릭하는 구조다. 마우스 환경에서는 익숙하지만 태블릿 기반의 터치 환경에서는 무심코 [예]를 누르는 ‘팝업 피로도(Popup Fatigue)’ 현상을 유발할 위험이 있다.
  • QGroundControl / PX4 방식: 다이얼로그를 지양하고 액션(Action)과 승인(Confirmation)을 슬라이드 모션 1개로 통합하여 인지 부하(Cognitive Load)를 줄이고 촉각적 확인 절차를 거치는 공간 지향적(Spatial-oriented) UI 철학을 따른다.

5. 지상 관제 시스템(Custom GCS) 구축용 설계 지침

ROS 2(Micro XRCE-DDS) 기반 혹은 커스텀 에이전트를 위한 GCS(예: PyQt, Flutter 등으로 개발하는 모바일 컨트롤러 앱) 설계 시, QGC의 설계 메커니즘을 소프트웨어 추상화 형태로 차용해야 한다.

  • 이벤트 위임 패턴(Event Delegation): 멀티터치 시스템 하에서는 다른 손가락이 화면을 누르고 있을 때에도 UI 슬라이더가 정상 동작해야 하므로 캡처된 ID 이벤트 관리를 독립적으로 격리할 것.
  • 롤백 애니메이션 동기화 방어 로직: MAVLink 네트워크의 응답 지연(RTT, Round Trip Time)을 고려하여 애니메이션 타이머와 커맨드 타임아웃 워치독(Watchdog C++ Timer)을 서로 동기화하고 연쇄적인 UI 크래시(Crash)를 차단하는 옵서버 패턴(Observer Pattern) 논리를 도입해야 한다.

QGC 프론트엔드의 이러한 터치 스크린 리얼타임 이벤트 처리 구조는, 하드웨어 스위치를 조작하는 듯한 신뢰도 높은 인프라를 소프트웨어로 제공하여 PX4-Autopilot의 전반적인 유인-무인 복합 체계 안정성 구축에 이바지한다.
