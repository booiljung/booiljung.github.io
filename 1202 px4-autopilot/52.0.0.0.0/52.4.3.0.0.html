52.4.3 비상 상태 식별을 위한 다중 모달(Multi-modal) 피드백 시스템 스케줄링

본 절에서는 PX4-Autopilot(기준 펌웨어 버전: v1.14.0) 아키텍처 내에서 기체의 비정상적인 상태 및 비상 조치(Emergency Action) 발동을 운용자에게 직관적이고 즉각적으로 전달하기 위한 다중 모달(Multi-modal) 피드백 시스템의 스케줄링 메커니즘을 상세히 분석한다. 비상 상황에서 시스템은 청각적 수단(피에조 부저)과 시각적 수단(고휘도 RGB LED), 그리고 데이터 링크를 통한 관제 시스템 오류 메시지 출력을 동기화하여 다차원적인 경고를 발생시킨다.

1. 다중 모달 피드백의 필요성과 구조적 특징

비행체의 제어 상실(Loss of Control), 배터리 임계치 도달(Critical Battery), 비행 종료(Flight Termination) 등 치명적인 결함이 발생할 경우, 조종자가 시선을 기체에 두고 있는지, 아니면 관제 화면(GCS)을 모니터링하고 있는지 알 수 없다. 따라서 PX4는 다음과 같은 세 가지 독립적인 채널을 동시에, 혹은 우선순위에 따라 스케줄링하여 피드백을 발생시킨다.

  1. 시각적 채널 (Visual Channel): 기체 외부의 RGB LED 점멸.
  2. 청각적 채널 (Auditory Channel): 온보드 피에조 부저(Piezo Buzzer)의 경고음 송출.
  3. 디지털 텔레메트리 (Digital Telemetry): MAVLink STATUSTEXT 패킷을 활용한 QGroundControl(버전 v4.3.0 이상) 텍스트 및 TTS(Text-to-Speech) 피드백.

이러한 출력 형태는 센서 데이터를 담당하는 commander 모듈 및 events 모듈에 의해 조율(Orchestration)되며, 실시간 운영체제(NuttX RTOS)의 작업 스케줄러를 통해 지연 시간(Latency) 없이 하드웨어 핀으로 전달된다.

2. 피드백 시스템 내부 아키텍처 및 uORB 통신 흐름

PX4 내부에서 이벤트가 발생하면, events 모듈이 상태 변화를 평가하여 uORB 토픽을 매개로 하드웨어를 제어한다. 아래는 이를 표현한 다중 모달 피드백 데이터 흐름도이다.

sequenceDiagram
    participant C as Commander (상태 검출)
    participant E as Event Module (이벤트 큐)
    participant L as led_control (uORB)
    participant T as tune_control (uORB)
    participant M as mavlink 모듈
    participant G as QGroundControl

    C->>E: 비상 상태 트리거(Critical Failure)
    E->>L: 비상 LED 패턴 (적색 고속 점멸)
    E->>T: 비상 튠 (고주파 사이렌)
    E->>M: EVENT/STATUSTEXT 메시지 전송
    M->>G: 텍스트 알림 및 구두 경고(TTS)

위 도표에서 알 수 있듯, Commander 모듈이 _status.fd_critical_failure 등과 같은 치명적 오류를 감지하면 이벤트 인스턴스를 생성한다. 이벤트 모듈은 우선순위 큐(Priority Queue) 기반으로 이벤트를 관리하며, 기존에 재생 중이던 일반 상태 튠(Tune)이나 LED 패턴을 즉각적으로 선점(Preemption)하고 비상 패턴을 퍼블리싱(Publishing)한다.

3. Ardupilot 모델과의 스케줄링 메커니즘 비교

Ardupilot 환경에서는 경보(Buzzer) 및 표시 등(LED) 제어가 주로 메인 루프(Main Loop) 내 특정 스케줄링 주기에 얽매여 있거나, Notify 라이브러리를 통해 상대적으로 단일화된 클래스 인스턴스로 관리된다.

반면, PX4-Autopilot은 모듈화된 마이크로 아키텍처를 지향한다. tune_control 프로세스와 led_control 프로세스는 분리된 자체 워크 큐(Work Queue) 혹은 독립 태스크로 실행되며, uORB 메시지에 의존하여 비동기적으로(Asynchronously) 작동한다. 이는 비행 제어 루프(Flight Control Loop)가 연산 과부하에 빠지더라도, 별도의 워크 큐 할당을 통해 시각적/청각적 피드백 시스템의 실시간성을 보장할 수 있는 구조적 쾌거(Milestone)라 할 수 있다.

4. 소스 코드 분석: 이벤트 라우팅 및 하드웨어 제어

PX4의 소스 트리에서 src/modules/events 디렉토리는 시스템 전체의 이벤트를 수집하고 배포하는 역할을 한다. 비상 상태 발생 시, 시스템은 다음과 유사한 C++ API를 호출하여 이벤트를 통보한다.

// 비상 상태 선언 및 다중 모달 피드백 호출 (유사 코드)
events::send<events::px4::enums::emergency_action_t>(
    events::ID("commander_emergency_triggered"),
    events::Log::Emergency, 
    "Critical Failure Detected: Terminating Flight",
    action
);

// 하드웨어 드라이버로 uORB 퍼블리싱
tune_control_s tune{};
tune.tune_id = tune_control_s::TUNE_ID_NOTIFY_CRITICAL_ERROR;
tune.volume = tune_control_s::VOLUME_LEVEL_MAX;
_tune_control_pub.publish(tune);

led_control_s led{};
led.color = led_control_s::COLOR_RED;
led.mode = led_control_s::MODE_BLINK_FAST;
led.num_blinks = led_control_s::MAX_BLINKS;
_led_control_pub.publish(led);

이러한 분리형 제어 방식은, 개발자가 특정 주변기기(예: I2C 기반의 외부 스마트 LED 모듈 등)를 새롭게 장착하더라도 기존 코드의 대대적인 수정 없이 led_control 토픽을 구독하는 별도의 드라이버만 작성하면 확장(Plug-and-play)이 가능하다는 엄청난 장점을 제공한다.

5. MAVLink 기반 QGroundControl 통신 연동

QGroundControl(GCS) 쪽으로 전송되는 로직 또한 정교하다. PX4는 구버전 호환성을 위해 STATUSTEXT 메시지를 활용하는 동시에, 최신 규격의 EVENT 마이크로서비스를 활용하여 대역폭(Bandwidth) 조절과 신뢰를 향상시킨다.
관제 측면에서 GCS 운영자는 비상 상황 도달 시, 시각적으로 사용자 인터페이스(UI) 전체에 붉은색 경고 오버레이(Overlay)를 관측함과 동시에, QGC 내부 TTS(Text-to-Speech) 엔진이 “Critical Error“를 음성으로 송출하는 동적인 피드백을 경험하게 된다.

이러한 동시 다발적(Concurrent) 피드백 스케줄링은 돌발적인 비상 상태(Flight Termination, Kill Switch 활성화 등)가 발동되었을 때 인지 지연(Cognitive Delay)을 최소화하여 2차 사고를 방지하기 위한 핵심 항공 전자(Avionics) 설계 원칙이다.
