52.6.3.2. 심장박동(Heartbeat) 메시지 동기화 실패 시 GCS 차원의 링크 상태 시각화 추정 모델 및 자율 비행 종료 절차 제안 로직

무인항공기 시스템(UAS) 운영 환경에서 지상 관제 시스템(GCS, Ground Control Station)과 기체 간의 생존 여부를 확인하는 가장 본질적인 통신 단위는 MAVLink 프로토콜의 심장박동(Heartbeat, HEARTBEAT #0) 메시지다. 드론과 GCS 양측은 1Hz(초당 1회)의 주기로 심장박동 메시지를 브로드캐스트(Broadcast)하며 상호 간의 통신 연결 및 비행 모드(Flight Mode) 상태를 동기화한다. 본 섹션에서는 심장박동 수신 처리가 실패했을 때, QGroundControl(v4.3 기준)과 같은 GCS가 운영자에게 상황을 인지시키는 시각화 추정 모델(UI/UX Visualization)과 더불어, 통신 복구 불능 시 시스템이 취해야 할 자율 비행 종료(Autonomous Flight Termination) 절차의 제안 로직을 심층적으로 다룬다.

1. MAVLink 심장박동(Heartbeat) 메시지의 구조와 수신 대기(Timeout)

PX4-Autopilot(v1.14 기준) 펌웨어 내부의 mavlink_receiver 모듈과 QGC 백엔드 통신 모듈은 각각 상대방의 시스템 ID(System ID)와 컴포넌트 ID(Component ID)를 필터링하며 HEARTBEAT 메시지를 수신한다. GCS 입장에서 기체의 심장박동 메시지가 사전에 정의된 임계 시간(일반적으로 3~5초) 이상 수신되지 않으면 하트비트 타임아웃(Heartbeat Timeout) 이벤트가 발생한다. 이는 일시적인 텔레메트리(Telemetry) 무선 간섭일 수도 있고, 기체의 전원 상실이나 메인 비행 제어기(FC, Flight Controller)의 커널 패닉(Kernel Panic)과 같은 치명적 고장일 수도 있다.

이러한 모호성 때문에 GCS 단독으로 기체의 추락이나 파손을 확정 짓기 어려우며, 이를 합리적인 UI로 해석하여 운영자에게 시각화(Visualization)하는 추정 모델 로직이 필수적이다.

2. GCS 차원의 링크 상태 시각화 추정 모델 (Link State Visualization Estimator)

QGroundControl(QGC) 내부의 C++ 아키텍처인 Vehicle 클래스와 VehicleLinkManager 모듈은 통신 링크의 단절 상태를 프론트엔드(QML)에 제공하기 위해 3단계의 시각화 추정 모델(3-Stage State Estimator)을 채택하도록 설계할 수 있다.

  1. 주의 단계 (Degraded, 1~3초 누락):
  • 일시적인 방해 전파(Jamming)나 통신 지연으로 판단한다. UI 상에는 연결 신호 안테나 아이콘이 주황색으로 변하거나 점멸하며, 기체 아이콘의 궤적이 흐려지는(Ghosting effect) 효과를 준다. 아직 통신 단절(Link Lost) 알람을 명시적으로 발생시키지 않는다.
  1. 단절 경고 단계 (Critical, 3~10초 누락):
  • QGC 백엔드에서 VehicleSettings 파라미터로 지정된 MAVLink 타임아웃 임계치를 초과한 상태다. UI는 즉각적으로 빨간색 경고 배너로 “Communication Lost“를 표출하며, 스크린 상에 마지막으로 수신된 좌표(Last Known Location)를 반투명한 기호로 고정(Freeze)시킨다.
  1. 추락 또는 생존 추정 모드 단계 (Estimation, 10초 이상 누락):
  • GCS는 이전에 수신된 EKF(Extended Kalman Filter) 기반의 속도 벡터(Velocity Vector)와 고도 하강률을 바탕으로 기체의 현재 예상 활공 혹은 추락 궤적 원추(Estimation Cone) 영역을 지도 UI 상에 폴리곤 형상으로 블렌딩(Blending)하여 표기한다. 이를 통해 운영자가 수색(SAR, Search and Rescue) 절차를 신속히 수립하도록 돕는다.

3. Ardupilot 대비 PX4의 하트비트 타임아웃 설계 철학

Ardupilot 아키텍처는 Mission Planner나 QGC 연결 시 하트비트 유실 시점부터 GCS UI의 개입과 매뉴얼한 사용자 상호작용(Action) 요청을 비교적 강하게 우선시하는 경향이 있다.
반면 PX4 소프트웨어 추상화 계층(Abstraction Layer)은 기체의 자율적이고 독립적인 상태 머신(State Machine)인 커맨더(Commander) 모듈의 역할을 더 신뢰하는 아키텍처적 특성을 갖는다. 따라서 GCS의 연결이 유실되었을 때 GCS 측의 예외 처리(시각화 알림 등)와 독립적으로, PX4 FC 내부에서는 COM_DL_LOSS_T 파라미터(데이터 링크 상실 타임아웃) 조건이 충족되는 순간 비행 제어기 독자적으로 “RTL(Return to Launch)” 혹은 “Land(착륙)“와 같은 페일세이프(Failsafe) 시퀀스로 전환된다.

4. 통신 영구 단절 시 자율 비행 종료(Flight Termination) 로직 제안

안전 임계(Safety-Critical) 관점에서, 만약 기체의 데이터 링크가 소실됨(GCS의 Heartbeat 수신 실패)과 동시에 기체 내부의 GPS 측위 신호 상실 등 이중 장애(Dual Failure)가 발생한다면, 일반적인 자율 복귀(RTL)는 불가능하다. 이러한 환경에서 GCS 프론트엔드와 기체 내부 로직이 사전에 협의해야 할 자율 비행 종료 시스템(AFTS, Autonomous Flight Termination System) 의 상태 전이 로직은 다음과 같이 제안된다.

stateDiagram-v2
    [*] --> 정상통신
    정상통신 --> 링크열화_경고 : Heartbeat N회 누락 (1<T<3s)
    링크열화_경고 --> 정상통신 : Heartbeat 복구 (ACK 수신)
    링크열화_경고 --> 통신상실_페일세이프 : Timeout 임계점 초과 (T>3s)
    
    state 통신상실_페일세이프 {
        [*] --> 지능형_복귀_판단
        지능형_복귀_판단 --> 자율복귀_수행(RTL) : GPS/EKF 정상
        지능형_복귀_판단 --> 강제착륙(Land) : 배터리 부족
        지능형_복귀_판단 --> 비행종료(Flight Termination) : 이중장애_발생
    }

    비행종료(Flight Termination) --> 비행종료(Flight Termination) : 모터 인터록 해제(PWM=0)<br/>낙하산 사출(Deploy Parachute)
    자율복귀_수행(RTL) --> [*] : GCS 사후 복구 대기
  1. 상태 모니터링: 커맨더 메인 루프에서 GCS 시스템 ID에 해당하는 하트비트 갱신 주기를 지속 검사한다.
  2. 지능형 복귀 판단: 통신 단절 시 하드웨어 센서 상태 비트마스크(Sensor Status Bitmask)를 확인하여 생존 가능성을 수치 해석적으로 판정한다.
  3. 독자적 비행 종료 개입: GCS 통신이 없고 위치 센서 융합도 붕괴되어 Flyaway(경로 이탈 표류) 위험이 임계치를 초과할 시, PX4 내부의 CBRK_FLIGHTTERM 보호를 무시하고 즉시 PWM 출력을 차단(Kill)하며 보조 채널을 통해 낙하산 유닛(Parachute Unit)을 사출한다.

5. ROS2 미들웨어 환경에서의 확장성 (uXRCE-DDS)

최근 GCS 아키텍처가 ROS2를 도입하여 MAVLink 대신 uXRCE-DDS 관제 환경으로 이행함에 따라, 심장박동 동기화 확인 역할 역시 DDS 프로토콜 고유의 리브니스(Liveness) QoS(Quality of Service) 정책으로 추상화(Abstraction)되고 있다. LIVELINESS_ASSERTED 이벤트를 응용하면, 애플리케이션 계층에서 주기적인 하트비트 카운터를 폴링(Polling)하던 전통적인 방식에서 벗어나, 운영체제(Linux/NuttX) 및 미들웨어 수준에서 콜백(Callback) 형태로 실시간 단절을 통보받을 수 있어 관제 소프트웨어의 추정 모델 지연 성능(Latency)을 비약적으로 개선할 수 있다.
