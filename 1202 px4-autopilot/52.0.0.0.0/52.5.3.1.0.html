52.5.3.1. 메인 루프 프로세싱 주기(Main Loop Rate) 내에서 가장 높은 가중치를 갖는 비상 명령 폴링(Polling) 우선순위 할당 구조

1. 개요 및 스케줄링 철학

PX4-Autopilot의 아키텍처는 다수의 센서 데이터를 처리하고 제어 알고리즘을 수행하며, 동시에 수많은 보조 작업(Telemetry, Logging 등)을 병렬로 처리하는 복잡한 실시간 운영체제(NuttX RTOS) 기반 시스템이다. 이러한 시스템에서 메인 비행 제어 루프(Main Flight Control Loop)는 보통 400Hz (2.5ms 주기) 이상의 매우 빠른 주기로 실행된다.

특히 생명과 재산을 보호해야 하는 무인항공기에서, 비상 제어 명령(Kill Switch, Flight Termination 등)은 그 어떠한 알고리즘(자세 제어, 내비게이션 등)보다 선행되어 즉각적으로 처리되어야 한다. 이를 보장하기 위해 커맨더(Commander) 스레드와 비행 제어 스레드는 메인 루프 프로세싱 주기(Main Loop Rate) 내에서 가장 높은 가중치를 갖는 우선순위 기반 폴링(Priority-based Polling) 메커니즘을 갖추고 있다.

2. uORB 메시징 버스와 스레드 우선순위(Thread Priority)

PX4의 모듈 간 통신은 마이크로 객체 요청 브로커(uORB)를 통해 비동기식(Asynchronous)으로 이루어진다. 하지만 중요한 데이터를 단순히 비동기로 두면 메인 루프 내에서 처리 지연(Latency)이 생길 수 있다. 따라서 PX4는 다음과 같은 스케줄링 디자인을 강제한다.

  • 스레드 가중치(Task Priority): 하드웨어 I/O 드라이버 다음으로 커맨더 모듈과 비행 제어 모듈 스레드가 가장 높은 우선순위(예: SCHED_PRIORITY_MAX - 5 수준)를 배정받는다.
  • 선제적 폴링(Preemptive Polling): 매 루프의 시작점(Top of the Loop)에서 항상 가장 먼저 비상 상태 uORB 토픽(vehicle_command, actuator_armed, manual_control_setpoint 등)의 업데이트를 폴링(Polling)한다.

4. 커맨더의 우선순위 할당 구조 분석 (Commander.cpp)

커맨더 모듈 소스 코드 내에서 Run() 루프가 구동되면, poll() 함수나 .update() 함수 호출 순서가 철저하게 코드 레벨에서 가중치화(하드코딩된 실행 순서)되어 있다.

  1. RC 입력 및 보조 채널 즉각 확인: manual_control_setpoint 토픽을 확인하여, 조종사의 하드웨어 스위치 중 Kill Switch와 직결된 보조 채널(AUX) 값의 변화를 최우선으로 검사한다.
  2. MAVLink 비상 커맨드 병합: 지상 관제 시스템(GCS)에서 전송된 vehicle_command (예: MAV_CMD_DO_FLIGHTTERMINATION) 중 비상관련 명령 파싱이 먼저 수행된다.
  3. 이 두 가지가 확인되면 커맨더는 handle_command() 로직 내에서 arming_state를 우회하고(Bypass) 직접 actuator_armed 내 manual_lockdown을 세팅하여 즉시 publish 한다. 이 퍼블리시는 전체 시스템에서 가장 높은 가중치를 가지고 전파된다.

5. Ardupilot (AP_Scheduler) 대비 차이점

Ardupilot은 AP_Scheduler라는 중앙 집중식 루프 컨트롤러를 사용하여 각 서브루틴의 실행 시간과 주기를 통제한다(예: 400Hz Fast Loop 내 타임 슬라이스 분할). Ardupilot에서는 failsafe_check() 등의 함수가 Fast Loop 내의 스케줄러 테이블에 등록되어 순차적으로 콜백 형태로 호출된다.

반대로 PX4-Autopilot은 독립된 스레드들이 각자 px4_poll을 사용하여 uORB 파일 디스크립터에 블로킹(Blocking) 되어 있다가 깨어나는 이벤트 플리븐(Event-driven) 혼합 모델을 지향한다. PX4의 메인 루프 프로세싱 주기 내 우선순위 할당 구조는 실행 제어권 측면에서 “순차적 콜백“이 아니라 “서치 앤 처리(Search and Process)“의 파이프라인 형태를 취한다. 즉, 루프 상단에서 무조건적으로 1순위 비상 토픽을 서치하고, 플래그가 세워져 있으면 아래의 무거운 알고리즘 연산 블록들을 모두 무효화(Return 또는 Break) 시킨 후 루프를 즉시 종료시켜버림으로써 지연율(Latency) 수치를 마이크로초 단위까지 최적화한다.

6. 요약

비상 상황에서의 시스템 레이턴시(Latency)는 기체의 물리적 추락 충격량과 직결되므로 최소화되어야 한다. PX4-Autopilot은 메인 루프 프로세싱 아키텍처 설계 시부터 가장 높은 가중치를 가진 비상 폴링 블록을 루프의 극상단에 배치하였다. 이는 복잡한 센서/제어 연산이 완료되기까지 기다리지 않고 “즉각적 절단(Immediate Cut-off)“을 가능하게 하여, 생명과 직결되는 높은 수준의 시스템 신뢰성을 보장한다.
