52.3.1. 소프트웨어 유한 상태 기계(FSM) 내 비행 종료 상태 천이(State Transition) 메커니즘

비행 제어 소프트웨어에서 안전과 직결된 최상위 의사결정은 엄격하게 설계된 유한 상태 기계(FSM, Finite State Machine)를 통해 이루어진다. PX4-Autopilot의 FSM은 시스템의 현재 운영 상태를 명확하게 정의하고, 예측 불가능한 예외(Exception) 조건 하에서 시스템이 어떤 비상 상태로 진입할지 결정하는 핵심 아키텍처이다. 본 절에서는 기체의 치명적인 결함 감지 시 발동되는 비행 종료(Flight Termination) 동작이 PX4 커맨더(Commander) 모듈 내부의 상태 천이(State Transition) 메커니즘 내에서 어떻게 수리적으로 취급되고 전이되는지 심층 분석한다.

1. PX4 최상위 멀티-FSM 구조 이해

PX4-Autopilot(v1.14 기준)의 상태 머신은 단일 계층이 아닌, 여러 목적에 따라 분리된 겹침(Overlapping) 상태 머신을 유지함으로써 시스템의 복잡도를 관리한다. 비행 종료 로직은 주로 다음 두 가지 핵심 상태 열거형(Enum)과 깊게 연관되어 있다.

  1. ArmingState (무장 상태): 구동기(Actuator)로의 물리적인 신호 출력 인가 여부를 결정한다. (INIT, STANDBY, ARMED, SHUTDOWN 등)
  2. NavState (항법 상태): 현재 기체가 수행 중인 논리적인 비행 모드와 제어 권한을 의미한다. (MANUAL, AUTO_MISSION, OFFBOARD, TERMINATION 등)

비행 종료가 선언되는 순간, 이 두 가지 상태 머신은 서로 독립적이지 않으며 강한 종속성(Dependency)을 가지고 동기화되어 즉각적인 페일 세이프(Failsafe) 처리를 유도한다.

2. 비행 종료(Flight Termination) 상태 천이 경로

정상적으로 비행 중인 기체(ARMED 및 임의의 NavState)에서 치명적인 오류(예: 센서 데이터 완전 소실, 통신 붕괴 후 추락 등)가 감지되거나 GCS로부터 명시적인 비행 종료 COMMAND_LONG 메시지가 수신되면, 커맨더(Commander) 내부의 state_machine_helper.cpp는 다음의 천이 제어 로직을 순차적으로 강제한다.

2.1 NavState의 TERMINATION 전이

어떠한 비행 모드(Position, Mission, Offboard 등)에 상관없이, 비행 종료 이벤트가 발생하면 NavState 는 최고 우선순위(Highest Priority)를 가지고 즉시 NavState::TERMINATION 상태로 전이(Transition)된다.
이 전이는 일반적인 비행 모드 변경 검증 알고리즘(예: 센서 정상 여부 확인 등)을 모두 우회(Bypass)하며 수행된다.

2.2 ArmingState의 SHUTDOWN 전이 보장

NavState가 TERMINATION으로 전이됨과 동시 혹은 직후에, ArmingState는 강제로 ArmingState::SHUTDOWN 상태(또는 시스템 설정에 따라 STANDBY_ERROR에 준하는 강제 무장 해제 상태)로 전이된다.
기존의 일반적인 무장 해제(Disarming)가 지면 접촉(Landed) 상태를 엄격히 요구하는 것과 달리, 비행 종료에 의한 천이는 비행 중(In-air)임을 무시하고 시스템 전력을 논리적으로 차단한다.

3. ArduPilot의 내부 제어 루프와의 비교

ArduPilot의 상태 관리 방식은 전통적으로 메인 loop() 내에서 현재 비행 모드(FlightMode 라이브러리)의 run() 메서드를 다형성(Polymorphism)을 통해 호출하는 단일 루프 폴링(Polling) 구조에 가깝다. AFR(Advanced Failsafe Recovery)이 발동되면 현재 모드의 제어권을 박탈하고 강제로 Mode_Crash 또는 우회 로직으로 진입하여 모터 인터록(Interlock)을 해제한다.

반면, PX4는 commander 모듈이 독립적인 스레드/태스크 로서 동작하며, vehicle_status uORB 토픽을 통해 상태값을 발행(Publish)한다. 하위 제어기 모듈(mc_att_control, navigator 등)은 이 NavState 값이 TERMINATION으로 갱신된 것을 비동기 주기(Subscription callback)로 인지하고, 각자의 내부 타이머와 제어 루프를 스탠바이(Standby) 또는 셧다운(Shutdown) 상태로 자체 천이시킨다. 즉, PX4는 중앙 집중식 메시징 교환망을 활용한 분산형 로직 중단을 구현하고 구현하고 있다.

4. 커맨더(Commander) 소스 코드 구현 요론

비행 종료 천이의 핵심은 커맨더 모듈 내 로직의 무조건성(Unconditionality)에 있다. PX4 베이스라인 코드에서 비행 종료 플래그(termination_triggered)가 활성화된 경우의 로직은 대략 다음과 같은 의사 코드(Pseudo-code) 흐름을 따른다.

if (termination_triggered) {
    // 1. 상태 머신의 우회적 강제 갱신
    internal_state.nav_state = navigation_state_t::NAVIGATION_STATE_TERMINATION;
    internal_state.arming_state = arming_state_t::ARMING_STATE_SHUTDOWN;
    
    // 2. 다른 모든 모드 변경 요청 거부 로직 래핑
    lock_status_machine();
    
    // 3. uORB를 통해 전 시스템에 통보
    vehicle_status_pub.publish(internal_state);
}

이와 같은 FSM 아키텍처 설계는 예측할 수 없는 런타임 환경에서 제어 로직이 교착 상태(Deadlock)나 레이스 컨디션(Race Condition)에 빠지지 않고, 수학적으로 증명 가능한 단일 종착점(Single Evaluation Sink)으로 기체를 유도하여 인간의 생명 및 자산을 보호하는 가장 근본적인 소프트웨어 통제 기술이다.
