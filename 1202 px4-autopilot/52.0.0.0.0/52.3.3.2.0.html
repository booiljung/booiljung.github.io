52.3.3.2. 메인 프로세서 커널 패닉(Kernel Panic) 발생 시 보조 프로세서의 하드웨어 타이머 기반 자율적 펄스 폭 변조(PWM) 차단(IO Failsafe) 기작

1. 개요

안전 임계(Safety-Critical) 시스템인 무인항공기(UAV) 운영 환경에서는 비행 제어 소프트웨어나 운영체제(OS)의 예기치 않은 오류가 치명적인 사고로 이어질 수 있다. PX4-Autopilot(버전 1.14 기준)은 이러한 상황에 대비하여 다중 마이크로컨트롤러(MCU) 아키텍처를 기반으로 한 하드웨어 수준의 안전 시스템을 갖추고 있다. 본 절에서는 메인 프로세서(FMU, Flight Management Unit)에서 커널 패닉(Kernel Panic)과 같은 심각한 오류가 발생했을 때, 보조 프로세서(IO, Input/Output)가 이를 감지하고 자율적으로 펄스 폭 변조(PWM, Pulse Width Modulation) 출력을 차단하거나 안전 상태로 전환하는 IO Failsafe 기작에 대해 상세히 분석한다.

2. PX4-Autopilot 하드웨어 아키텍처: FMU와 IO 프로세서

Pixhawk 시리즈를 비롯한 표준 PX4 호환 하드웨어(예: Pixhawk 4, Pixhawk 6C 등)는 일반적으로 두 개의 마이크로컨트롤러를 탑재한다.

  • 비행 관리 프로세서 (FMU, Flight Management Unit): 1차 프로세서로, 고성능 처리가 요구되는 자세 추정(Estimator), 비행 제어(Position/Attitude Control), 네비게이션, MAVLink 통신 등을 담당한다. NuttX RTOS 상에서 구동되며 상대적으로 복잡한 소프트웨어 스택을 실행한다.
  • 입출력 보조 프로세서 (IO, Input/Output Processor): 2차 프로세서로, 주로 액추에이터(모터 및 서보)로 전달되는 PWM 신호의 생성과 RC(Radio Control) 입력 감지를 담당한다. 핵심 역할 중 하나는 FMU의 상태를 모니터링하고 FMU에 장애가 발생 시 독립적으로 기체를 안전 상태로 유도하는 것이다.

이 두 프로세서는 주로 범용 비동기 송수신(UART, Universal Asynchronous Receiver-Transmitter) 기반의 직렬 통신을 통해 상태, 제어 명령, 텔레메트리 데이터를 고주파수로 교환한다.

3. 커널 패닉 감지 및 하드웨어 타이머 워치독 시스템

FMU의 소프트웨어 버그, 메모리 누수, 혹은 하드웨어적 결함으로 인해 NuttX 커널 패닉이 발생하면, FMU는 정상적인 스케줄링을 멈추고 제어 루프를 상실한다. 이때 보조 프로세서인 IO는 다음의 기작을 통해 이를 즉각 감지한다.

  1. 동기화 메시지 송수신: 정상 상태에서 FMU는 일정 주기(예: 50Hz 이상)로 최신 액추에이터 제어 명령과 생존 신호(Heartbeat)를 IO 프로세서로 전송한다.
  2. 하드웨어 워치독 타이머 (Hardware Watchdog Timer): IO 프로세서는 FMU로부터 유효한 패킷을 수신할 때마다 내부 하드웨어 타이머를 리셋(Reset)한다.
  3. 타임아웃(Timeout) 발생: 커널 패닉으로 인해 FMU의 패킷 전송이 중단되면, IO 프로세서 내부의 워치독 타이머가 임계 시간(Timeout)을 초과하게 된다. 이를 통해 IO 프로세서는 FMU가 정상 범주를 벗어난 고장 상태(Fault State)에 진입했음을 독립적으로 인지한다.

4. IO Failsafe: 자율적 PWM 차단 및 전환 기작

FMU 다운 상태를 감지한 즉시, IO 프로세서는 자율적으로 사전에 정의된 안전 조치를 취한다(IO Failsafe). 이는 기체가 통제 불능 상태로 비행하는 것을 방지하기 위한 최후의 물리적 제어 수단이다.

  1. PWM 차단 (비행 종료 / Kill Switch): 가장 강력한 페일세이프 조치로, 모터와 서보로 향하는 모든 PWM 출력을 즉각적으로 중단하거나 최소 펄스(Disarmed PWM)로 강제 고정한다. 이는 멀티로터의 경우 즉각적인 추락을 유발하지만, 예측할 수 없는 방향으로 통제 없이 고속 비행하여 발생할 수 있는 2차 인명 피해 현상을 방지하는 비행 종료(Flight Termination)의 성격을 띤다.
  2. 수동 오버라이드 (Manual Override): 고정익 기체 등 일부 설정에서는 FMU가 다운되더라도 IO 프로세서에 직접 연결된 RC(Radio Control) 입력을 통해 조종사가 기체를 완전 수동 모드로 조종할 수 있는 경로를 열어둔다(Direct pass-through).

4.1 관련 소스 코드 분석

PX4 펌웨어(v1.14 기준)의 src/modules/px4iofirmware/ 디렉토리를 살펴보면 IO 프로세서의 핵심 로직을 확인할 수 있다.

  • px4io.c 및 registers.c 파일 내에 타이머 인터럽트와 상태 머신이 구현되어 있다.
  • FMU 통신 타임아웃이 발생하면 상태 레지스터(Status Register)의 PX4IO_P_STATUS_FLAGS_FMU_OK 비트가 해제되고, 시스템은 Failsafe 모드로 돌입한다.
  • 이후 mixer.cpp 등의 처리 과정에서 안전(Failsafe) 매개변수에 정의된 펄스 폭(PWM) 값으로 액추에이터 출력이 덮어씌워지거나(Override), 출력이 완전히 차단(Cutoff)된다.

5. Ardupilot의 Failsafe 구조와의 비교

PX4-Autopilot과 Ardupilot 모두 Pixhawk와 같은 동일한 하드웨어를 사용할 경우, 기본적으로 유사한 FMU-IO 듀얼 아키텍처 이점을 누린다.

  • Ardupilot: Ardupilot 역시 IO 보드(IOMCU) 워치독 기능을 제공한다. BRD_SAFETYENABLE 설정과 결합되어 있으며, 메인 루프 동결 시 IOMCU가 이를 감지하여 모터를 정지시키거나 사전에 설정된 안전 PWM 레벨(SBUS Failsafe 값 등)을 송출한다.
  • 접근 철학의 차이: PX4는 uORB와 px4_io 모듈 간의 명시적 토픽 분리 및 미들웨어 계층의 안전성을 강조하는 구조를 지닌다. PX4는 커널 패닉 시 즉각적인 PWM 완전 차단(Kill) 정책을 고수하는 경향이 강한 반면, Ardupilot은 고정익의 RC 수동 오버라이드 유연성에 약간 더 많은 커스터마이징 파라미터를 제공하는 다변화된 구조를 가지고 있다.

6. QGroundControl 설정 및 관련 파라미터

QGroundControl(버전 4.3 이상)을 통해 이러한 하드웨어 수준의 비상 정지 및 PWM 동작 매개변수를 세밀하게 조율할 수 있다.

  • CBRK_IO_SAFETY: IO 보드의 하드웨어 안전 스위치 동작을 우회(Bypass)할지를 결정하는 회로 차단기(Circuit Breaker) 매개변수이다.
  • PWM_MAIN_FAILx / PWM_AUX_FAILx: (과거 PX4 버전 및 파라미터 설정 환경 기준) 페일세이프 발생 시 각 채널별로 출력할 펄스 폭(마이크로초, \mu s) 값을 직접 정의한다. 이를 통해 특정 플라이트 컨트롤러(FC) 보드 하에서 PWM 차단 시의 출력을 스케일링할 수 있다.
  • COM_FLTMODEx: 비행 전환 및 안전 트리거의 소프트웨어 임계치를 설정하며, 이는 하드웨어 IO 차단 이전 단계의 소프트웨어 로직을 통제한다.

7. MAVLink 및 ROS2 기반 자율 에이전트 환경의 시사점

현대적인 무인기 관제 시스템에서 단일 기체 또는 다중 에이전트 군집 로봇을 운영할 때, 지상 관제 시스템(GCS)이나 ROS2 기반의 컴패니언 컴퓨터(Companion Computer)는 원격에서 MAVLink의 심장박동(HEARTBEAT) 메시지를 모니터링한다.

FMU 내 커널 패닉이 발생하면, MAVLink 통신 역시 즉시 중단된다. ROS2 환경(Micro XRCE-DDS)을 통한 데이터 스트림 또한 끊어지게 된다.

  1. GCS의 대응 로직: QGroundControl 측에서는 MAVLink 타임아웃(통상 수 초 이내)을 감지하고 모니터 상에 “통신 두절(Communication Lost)” 알람을 발생시킨다.
  2. 데이터 링크와 하드웨어 동작 간의 시간차: GCS가 MAVLink 타임아웃을 연산하기 훨씬 이전(수십~수백 밀리초 단위 이내)에 기체 내장 하드웨어 IO 보드의 타이머가 먼저 발동하여 물리적 PWM 차단을 선행한다. 즉, 자율 에이전트 제어 구조에서는 원격 시스템의 소프트웨어적 로직 개입보다 하드웨어적 최종 방어망이 우선순위로 작동하여 기체의 통제 상실(Fly-Away) 확산을 저지한다.

8. 결론

PX4-Autopilot의 메인 프로세서 커널 패닉 대응 체계는 시스템의 이중화(Redundancy)와 신뢰성 확보 원칙을 입증하는 핵심 기작이다. 소프트웨어 한계로 발생할 수 있는 치명적 불능 상태를 제어하기 위해, 보조 프로세서(IO)의 하드웨어 타이머 기반 자율적 펄스 폭 차단(IO Failsafe) 기전은 UAV 시스템의 물리적 런어웨이 현상을 원천 차단한다. 이러한 결정론적 하드웨어 개입은 돌발적인 위험으로부터 학술적, 상업적 임무 지대 내의 보안 요소를 충족시키는 강력한 메커니즘을 제공한다.
