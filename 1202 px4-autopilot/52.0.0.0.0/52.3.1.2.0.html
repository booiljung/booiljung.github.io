52.3.1.2 비휘발성 랜덤 액세스 메모리(FRAM) 및 EEPROM의 웨어 레벨링(Wear Leveling)을 고려한 영구 잠금 플래그(Flag) 기록 알고리즘

PX4-Autopilot(기준 펌웨어 버전: v1.14.0)의 비행 종료(Flight Termination) 시스템이 가동되면, 기체는 치명적인 결함이 있다고 판단되어 “락다운(Lockdown)” 상태로 진입한다. 락다운 상태의 핵심 요구사항 중 하나는 **“전원을 인가했다가 다시 켜더라도(Power Cycling), 관리자의 명시적인 해제 명령 전까지 기체의 시동(Arming)이 영구적으로 차단되어야 한다”**는 것이다. 본 절에서는 이러한 영구 잠금 플래그(Flag)가 비휘발성 메모리인 FRAM 또는 EEPROM에 기록되는 메커니즘과, 물리적 소자의 수명을 연장하기 위한 웨어 레벨링(Wear Leveling) 알고리즘의 동작 원리를 분석한다.

1. 개요 및 영구 잠금 기능의 목적

비행 종료가 트리거되면 PX4 커맨더(Commander) 모듈은 현재 기체의 생명을 완전히 중지하는 소프트웨어적 플래그를 설정한다. 그러나 소프트웨어 계층의 변수는 RAM에 상주하므로 전원이 꺼지면 휘발된다. 따라서 PX4-Autopilot은 치명적 오류 코드를 파라미터(Parameter) 서브시스템을 통해 비휘발성 랜덤 액세스 메모리(FRAM) 또는 특정 Pixhawk 보드의 EEPROM 영역에 강제 기록(Force Commit)한다.
이때 비행 중 고빈도로 발생하는 이벤트 로그나 파라미터 자동 저장이 동일한 메모리 셀(Cell)을 반복적으로 덮어쓸 경우, 해당 영역의 쓰기 수명(Endurance Cycle)이 급격하게 저하되어 메모리 배드 섹터(Bad Sector)가 발생할 위험이 있다. 이를 방지하기 위해 파일 시스템 및 드라이버 단에서 **웨어 레벨링(Wear Leveling)**이라는 논리적 기법이 필수적으로 적용된다.

2. 웨어 레벨링(Wear Leveling)의 개념과 적용

웨어 레벨링은 비휘발성 메모리의 칩 수명을 전체 영역에 걸쳐 고르게 분산시키는 플래시 메모리 관리 기법이다. 보통 EEPROM은 약 10만 회 ~ 100만 회의 지우기/쓰기(Erase/Write) 사이클을 보장하지만 미션 컴퓨터 관점에서는 이 사이클을 관리하지 않으면 매우 짧은 시간 안에 한계에 도달할 수 있다. 반면 FRAM(Ferroelectric RAM)은 10조 회 이상의 압도적인 쓰기 내구성을 가지나, 데이터 파편화를 막기 위해 동일한 아키텍처적 관리를 적용받는 경우가 많다.

PX4의 기반이 되는 NuttX RTOS 환경 내 Parameter System은 다음과 같은 원리로 웨어 레벨링을 간접적으로 구현한다:

  1. BSON 기반의 순차 후입 기록(Append-Only Logging): 파라미터(Parameter)의 값을 변경할 때마다 정해진 메모리 주소(고정 주소)를 덮어쓰는 것이 아니라, BSON(Binary JSON) 포맷으로 직렬화(Serialization)된 블록을 순차적으로 이어붙여(Append) 쓴다.
  2. 가비지 컬렉션(Garbage Collection 및 Compaction): 할당된 파라미터 저장 공간이 가득 차게 되면, 유효한 최신 플래그와 파라미터 세트만 읽어들여 새로운 메모리 청크(Chunk)에 처음부터 다시 기록하고 이전 블록을 일괄 소거(Erase Block)한다.
  3. 이 과정을 통해 영구 잠금 플래그와 통상적인 파라미터의 기록 위치가 점진적으로 순환하여 마모도를 분산(Wear Leveling)시킨다.

3. 영구 잠금 플래그 기록 아키텍처

락다운 발생 시 파라미터 기록 루틴은 다음과 같은 단계를 거쳐 비휘발성 메모리에 도달한다.

sequenceDiagram
    participant Commander as Commander (App)
    participant ParamSys as Parameter System
    participant FileSys as NuttX FAT / MTD
    participant Hardware as FRAM / EEPROM (I2C/SPI)

    Commander->>ParamSys: param_set(PARAM_ID, VAL) <br/> [락다운 플래그 강제 쓰기]
    ParamSys->>ParamSys: BSON 직렬화 (Serialization)
    ParamSys->>FileSys: BSON 객체 Append (웨어 레벨링 기반)
    activate FileSys
    FileSys->>Hardware: Memory Technology Device <br/> 블록 단위 Page Program
    Hardware-->>FileSys: Write Ack
    deactivate FileSys
    FileSys-->>ParamSys: File sync() 완료
    ParamSys-->>Commander: 플래그 영구 기록 성공
  1. 커맨더 로직에서 비행 종료가 승인되면, 연관 플래그(비공식 락다운 강제 차단, 회로 차단기인 CBRK_FLIGHTTERM 플래그의 하드-리셋 등)에 대한 param_set() API를 호출한다.
  2. 시스템은 param_export()를 즉각적으로 호출하여 RAM 트리가 아닌 비휘발성 MTD(Memory Technology Device) 스토리지 영역으로 직접 커밋을 강제한다 (fsync 효과).

4. 소스 코드 관점의 분석 (NuttX 및 PX4 MTD)

파라미터 저장의 물리적 매핑은 보드 종류(Pixhawk 4, Pixhawk 6C/X 등)에 따라 다르다.
소스 코드 트리 src/lib/parameters/ 하위의 로직을 살펴보면, 파라미터의 자동 저장(Autosave) 스레드가 존재하지만 락다운과 같은 극우선순위(Critical) 상황에서는 스케줄러 딜레이를 피하기 위해 스레드를 우회하여 직접 블록킹 I/O(Blocking I/O) 쓰기를 발생시키기도 한다.

  • MTD(Memory Technology Device) 드라이버 레이어:
    NuttX 커널은 drivers/mtd/에 EEPROM(예: 24xx 계열 I2C) 및 FRAM(예: FM25V 계열 SPI) 드라이버를 정의하고 있다.
  • MTD 위에는 BCH (Block to Character) 계층과 간단한 플래시 파일 시스템(예: NXFFS 또는 FAT)이 올라간다. PX4의 순환 BSON 파라미터 구조는 이 계층에서 웨어 레벨링 알고리즘을 델리게이트(Delegate) 받거나, 혹은 플래시 블록의 페이지 단위 기록을 통해 자체적인 수명을 보장받는다.

5. Ardupilot 대비 아키텍처 및 QGroundControl 관제 차이

PX4와 Ardupilot 모두 비휘발성 플래시 또는 FRAM을 사용하여 설정값을 영구 저장한다는 점은 동일하지만, 아키텍처적 차원에서의 접근법은 조금 다르다.

  1. 저장 구조의 차이:
  • Ardupilot: 역사적으로 16kB 혹은 32kB 단위의 정적 주소 할당 기반 파라미터 블록을 유지하였으나, 최신 펌웨어에서는 플래시 기반의 Wear Leveling 레이어를 더욱 강화하여 Storage API를 통해 맵핑한다.
  • PX4-Autopilot: 초기부터 BSON 포맷을 도입하여 파라미터 셋의 메타데이터 크기를 줄이고, 파일 시스템 레벨에서의 Append 방식(Log-structured)을 이용하여 락다운 플래그의 안전성을 도모하는 구조이다.
  1. QGroundControl(v4.3 기준) 관제 개입:
    기체가 영구 잠금 락다운 상태에 진입하고 전원을 껐다가 켜면, QGroundControl은 MAVLink 프로토콜을 통해 SYS_STATUS의 비정상 플래그를 수신하고 “Vehicle is actively locked down” 메시지를 표시한다.
    이 락다운 플래그를 해제하기 위해서는 GCS 상의 QML 프론트엔드 UI를 통해 파라미터 편집(Parameter Editor) 창으로 진입한 뒤, 관련된 초기화 파라미터를 강제로 재설정(Reset to Default 등)하거나 MAVLink의 관리자 권한 콘솔 접근(MAV_CMD)을 통하여 강제 해제 절차를 거쳐야만 비로소 모터 인가가 가능해진다.

6. 결론

영구 잠금 플래그(Permanent Lockdown Flag)의 목적은 기체의 하드웨어나 핵심 소프트웨어의 통제력이 완전히 상실되었을 경우, 임의 재부팅을 통한 우발적 시동을 막아 대형 사고를 원천 차단하는 데 있다.
이를 비휘발성 메모리(FRAM, EEPROM)에 성공적으로 저장하기 위해서는, 단순히 플래그 비트를 쓰고 지우는 과정을 넘어 칩의 수명을 보장하기 위한 웨어 레벨링(Wear Leveling) 메커니즘과 BSON 기반의 순차 기록 아키텍처가 긴밀하게 맞물려 작동한다. 이 정교한 로직은 PX4-Autopilot이 요구하는 엄격한 산업용(Industrial) 및 항공 우주 등급의 안전성을 하드웨어 제약 내에서 극대화하는 훌륭한 사례이다.
