35.2.3.3. 시뮬레이터 클럭 동기화(Clock Synchronization)를 위한 록스텝(Lock-step) 스케줄링 및 비동기 센서 틱(Tick) 제어

1. 개요 및 설계 철학

네트워크 기반 또는 동일 호스트(Host) 기반의 소프트웨어 인 더 루프(Software In The Loop, SITL) 가상 환경은 본질적으로 비결정론적(Non-deterministic)인 운영체제(OS)의 스케줄러에 종속되어 있다. 과도한 CPU 부하, 다른 백그라운드 프로세스의 간섭, 그리고 디버깅(Debugging) 수행 중 브레이크포인트(Breakpoint)를 통한 정지로 인해 논리적인 시스템 클럭(System Clock)과 가상 세계의 물리적 벽시계 시간(Wall-clock Time) 사이에는 필연적으로 오차가 발생한다.

PX4-Autopilot은 이와 같은 고질적인 타이밍 동기화 문제를 근본적으로 해결하고 반복 가능(Repeatable)하며 결정론적인 시뮬레이션 동작을 보장하기 위해 록스텝(Lock-step) 스케줄링 메커니즘을 적극적으로 도입하였다. 이는 가상 물리 엔진(예: Gazebo, JMavSim)의 1단계 연산(Tick)과 비행 제어기 내부의 1단계 제어 루프를 강제로 동기화(Ping-pong)시켜, 네트워크 지연에 상관없이 완벽하게 톱니바퀴처럼 맞물려 돌아가는 타임라인을 구성하는 진보된 아키텍처이다.

2. 록스텝(Lock-step) 스케줄링 아키텍처

록스텝(Lock-step) 모드에서 비행 제어기 펌웨어는 더 이상 컴퓨터의 메인보드 하드웨어 클럭(CLOCK_MONOTONIC 등)이나 운영체제의 커널 시간(Kernel Time)을 직접 참조하지 않는다. 대신 MAVLink를 통해 시뮬레이터 측에서 주입해 주는 타임스탬프를 자신이 경험하는 ’유일한 현실의 시간’으로 간주하여 시간 관념을 오버라이딩(Overriding)한다.

2.1 핑-퐁(Ping-pong) 생명주기 모델

이 메커니즘은 다음과 같은 엄격한 순환 구조를 가진다.

  1. 시뮬레이터 계층: 가상 환경의 1밀리초(ms) 시뮬레이션 스텝을 진행한 뒤, 가장 최신의 타임스탬프와 센서 데이터(HIL_SENSOR 메시지)를 TCP 소켓을 통해 PX4로 발송하고, PX4로부터 제어 명령이 도착할 때까지 대기(Block) 상태에 진입한다.
  2. PX4 펌웨어 계층: 수신된 MAVLink 메시지 내부의 센서 데이터를 uORB 버스에 퍼블리싱(Publishing)함과 동시에, 탑재된 시스템 시간(Low-level Timer)을 메시지의 타임스탬프로 즉시 점프시킨다. 이후 상태 추정(Estimator)과 자세 제어기(Attitude Controller) 알고리즘을 1틱(Tick) 수행한다.
  3. 액추에이터 피드백 회신: PX4의 믹서(Mixer) 또는 모터 할당(Allocation) 로직의 연산이 끝나면 그 결과물인 PWM 대우 데이터(HIL_ACTUATOR_CONTROLS)를 시뮬레이터로 보낸다.
  4. 시뮬레이터 진행: 액추에이터 정보를 수신한 물리 엔진 대기 상태를 풀고 다음 가상 물리 연산을 진행한다.

이러한 로직을 통해 시뮬레이션 환경 내의 시간이 아무리 빠르게(가속) 진행되거나 아무리 느리게(통신 지연) 흘러가더라도 비행 동역학 모방의 발산(Divergence)이 발생하지 않는다.

3. 비동기 센서 틱(Tick) 제어 로직

실제 비행 제어 하드웨어는 IMU 센서(250Hz~1kHz), 나침반 센서(10Hz~50Hz), GPS(5Hz~10Hz) 등 비동기적이고 파편화된 다중 주파수의 센서 신호를 처리한다. 록스텝 환경에서는 가장 빈도가 높은 관성 측정 센서(IMU) 주기에 해당하는 틱(Tick) 모델이 베이스 클럭(Base Clock)을 이끄는 메인 드라이버를 구성한다.

3.1 외부 인터럽트 에뮬레이션

SITL 환경에서는 하드웨어 데이터 송신 주기(DATA READY) 핀 인터럽트가 존재하지 않는다. 대신, 시뮬레이터 브릿지 모듈이 simulator_mavlink.cpp 내부에서 poll() 루프를 돌며 MAVLink 수신을 감지하고, 이 즉시 타임스탬프 델타(Delta) 연산을 마친 뒤 센서 uORB 쓰기를 실행하여 가상 하드웨어 인터럽트를 촉발(Emulation)시킨다.

위 시퀀스의 치밀함 덕분에, IDE 상에서 PX4 실행부에 GDB/LLDB 디버거 브레이크포인트를 걸어 몇 초 동안 멈추더라도 Gazebo 리눅스 프로세스 역시 동시에 일시정지 상태에 머무르게 된다. 이후 중단점을 해제하면 아무런 시간 왜곡 없이 안정적으로 궤도에 복귀한다.

5. 소스 코드 구현 이해: 클럭 오버라이딩(Clock Overriding) 추상화

PX4 소스 체인 하단의 보드 지원 패키지(BSP, Board Support Package) 급에 속하는 타임 모듈을 살펴보면 록스텝 지원을 위한 커널 단위의 가상 패치가 훌륭하게 설계되어 있다. PX4 시간(absolute time) 반환 함수 내부는 매크로 및 구조체 포인터를 기반으로 분기된다.

[운영체제 독립적 타임 함수 래퍼 (C 계층 추상화)]

// platforms/posix/src/px4/common/px4_posix_impl.cpp 및 hrt 구현부 (개념화)

static uint64_t simulated_time_us = 0; // 시뮬레이터가 주입한 최신 시간 변수
static bool lockstep_enabled = false;

hrt_abstime hrt_absolute_time() {
    if (lockstep_enabled) {
        // 록스텝 모드 시, 실제 시스템 시계 무시 및 주입된 가상 타임스탬프 반환
        return simulated_time_us; 
    } else {
        // 실 환경 하드웨어(Pixhawk 계열) 또는 타 모드에서는 monotonic 시간 즉시 연산
        struct timespec ts;
        clock_gettime(CLOCK_MONOTONIC, &ts);
        return ts_to_abstime(&ts);
    }
}

유입되는 IMU 메시지의 타임스탬프 파싱 스레드에서 simulated_time_us 변수를 동기적(Mutex 또는 Atomic 처리 후)으로 최신화하는 구조가 록스텝 연산의 근간 모듈이 된다. 이로 인해 모든 상위 단계 비행 제어 알고리즘은 타임스탬프 변조 사실을 알지 못한 채 투명하게(Transparent) 알고리즘을 이행한다.

6. ArduPilot 대비 클럭 관리 및 관제 차이점

ArduPilot 체계 역시 SITL 기능을 오랫동안 주력으로 발전시켜 왔다. ArduPilot의 SITL 시간 메커니즘(AP_HAL_SITL 레이어)도 외부 가상 타이머 수신 방식을 채택하고 있으며, 기본 개념상 록스텝 제어는 두 오픈소스 생태계 모두 표준 편제이다.
그러나 PX4의 경우 uORB 네트워크 토폴로지 특유의 토픽 기반 퍼블리시/서브스크라이브 패턴이 융합되면서 개별 스레드 락(Lock) 및 대기 동기화에 있어 좀 더 객체 지향적이고 모듈화된 계층으로 격리되어 있다. 가령, ArduPilot의 슬로우 루프(Slow Loop) 제어 체계는 메인 스케줄러 자체 루프 순서에 종속적으로 동작하지만, PX4 구조 하에서는 워크 큐(Work Queue) 백엔드가 uORB에 게시된 sensor_accel 데이터의 갱신 시점이나 폴링 이벤트(px4_poll)의 타임아웃 주기에 맞춰 유동적으로 재단되는 구조를 가진다.

7. 결론

록스텝(Lock-step) 스케줄링 기법의 근본적인 목적은 불확실성이 내제된 컴퓨팅 환경(운영체제 멀티스레드 지연 스파이크)을 통제 가능한 닫힌 시스템(Closed System) 모델로 탈바꿈 시키는 데 의의가 있다. 이는 대규모 심층 강화학습(Deep Reinforcement Learning) 훈련 속도 향상은 물론 통신 두절, 센서 이상, 예외적 상태 머신 테스트 등 하드웨어 파괴의 리스크를 감수해야 하는 실증 시험(Flight Test)을 소프트웨어 계층으로 한치의 오차 없이 옮겨오기 위한 현대 드론 공학 통신 및 제어 아키텍처의 찬란한 산물이다.
