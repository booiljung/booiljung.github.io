35.5. 유한 상태 머신(FSM) 기반 기체 동작 제어 및 비행 포락선(Flight Envelope) 제한 (Action API)

현대의 무인항공기는 단방향으로 주어지는 수동 RC 입력에만 맹목적으로 복종하던 과거의 RC 비행기와 궤를 달리한다. PX4-Autopilot은 기체의 현재 물리적 한계와 비행 모드 맥락(Context)을 스스로 인지하고, 지상 관제소(GCS)나 컴패니언 컴퓨터(Companion Computer)가 내리는 고차원적인 전술 명령(Action)을 지능적으로 필터링하는 두꺼운 소프트웨어 방어벽을 갖추고 있다.

이 방어벽의 중추가 바로 commander 모듈 내에 구현된 ’유한 상태 머신(Finite State Machine, FSM)’과 공기역학적 ‘비행 포락선(Flight Envelope)’ 검증 알고리즘이다. 구조적 안정성을 유지하며 복잡한 비행 시퀀스를 지휘하는 이 시스템의 작동 원리와 MAVSDK의 Action API가 이와 어떻게 교신하는지 살펴본다.

1. Commander 모듈의 FSM(유한 상태 머신) 아키텍처

PX4의 비행 제어 논리는 거대한 유한 상태 머신(FSM) 위에서 돌아간다. 기체는 오직 FSM에 미리 정의된 승인된 궤적(Valid Transitions)을 통해서만 상태(State)를 전환할 수 있다.

핵심 상태 전이(State Transition) 생명주기는 다음과 같다.

  • INIT (초기화) \rightarrow PRE_FLIGHT (비행 전 점검 대기)
  • PRE_FLIGHT \rightarrow ARMED (동력 인가)
  • ARMED \rightarrow TAKEOFF (이륙 중) \rightarrow IN_AIR (체공 중)
  • IN_AIR \rightarrow LANDING (착륙 중) \rightarrow LANDED (지상 접촉 완료) \rightarrow DISARMED (동력 차단)

MAVSDK의 Action 클래스(예: action.arm(), action.takeoff(), action.land())를 통해 API 호출이 발생하면, 이는 내부적으로 MAVLink의 COMMAND_LONG 메시지로 캡슐화되어 비행 제어기로 전송된다. PX4 내의 commander 모듈은 이 vehicle_command uORB 토픽을 수신하면, 즉시 무조건 액추에이터를 가동하는 대신 “현재 기체의 FSM 상태 트리(State Tree)에서 해당 명령으로의 노드 점프가 합법적인가?“를 교차 검증한다. 만약 체공 중(IN_AIR)인데 이륙(TAKEOFF) 명령이 들어오거나, DISARMED 상태에서 LAND 명령이 들어오면 FSM은 이 트랜잭션(Transaction)을 즉각 거절(Reject)하고 명령 거부 (NACK) 상태 코드를 응답한다.

2. 비행 포락선(Flight Envelope)의 동적 제한

FSM이 거시적인 논리적 행동 제약(Logical Constraint)을 담당한다면, 비행 포락선(Flight Envelope)은 제어 루프가 출력할 수 있는 물리적 기동의 한계(Physical Limit)를 족쇄(Shackle) 채우는 역할을 한다.

PX4 멀티로터 제어기(Multicopter Attitude Control)는 MPC_TILTMAX_AIR, MPC_Z_VEL_MAX_UP, MPC_XY_VEL_MAX 등의 파라미터 조합으로 기체의 3차원 운용 한계를 동적으로 규정한다.

  1. 자세각 포락선 (Attitude Envelope): 파일럿이나 외부 제어기(Offboard)가 피치(Pitch)를 급격하게 +90^\circ 로 기울이라는 명령을 내려도, 자율 제어기는 MPC_TILTMAX_AIR (예: 45^\circ) 한계를 초과하는 요구치를 클리핑(Clipping, 자름)시켜 추락이나 뒤집힘을 방지한다.
  2. 속도 및 가속도 포락선 (Kinematic Envelope): MAVSDK의 특정 좌표로 이동하라는 명령(goto_location) 수행 중 기체는 스스로 최대 허용 속도 및 가속도(Jerk 한계치 포함)를 역산해 최적화된 S자 곡선(S-curve) 속도 궤적을 제너레이트(Generate)한다.

이러한 포락선 제한 장치 덕분에 개발자는 MAVSDK로 추상화된 Action API를 호출할 때, “내가 코드를 잘못 짜서 드론 기체를 공중에서 뒤집어엎어버리면 어떡하지?“라는 물리 역학적 공포에서 해방되어 논리적 비행 궤적 흐름에만 온전히 집중할 수 있다.

3. 예외적인 비대칭 기동: FTS 시스템

모든 규칙에는 예외가 존재한다. 물리적인 하드웨어 파편화 폭발이나 프로펠러 분리 상황에서는 FSM의 우아한 착륙(LANDING) 절차나 포락선 제한 알고리즘이 오히려 기체를 치명적인 미사일로 만들어 버릴 수 있다.

이러한 최악의 임계 지점(Critical Point)을 돌파하기 위해 PX4는 비행 종료 시스템(FTS, Flight Termination System)이라는 우아한 자살 메커니즘을 별도로 보유하고 있다. MAVSDK의 action.kill()명령은 FSM의 현재 상태가 무엇이건간에(심지어 50미터 상공이라 할지라도), 즉각적으로 commander 단의 모든 보호 회로와 제어 루틴을 우회(Bypass)하여 모터 PWM 출력을 제로(0)로 셧다운시킨다.

결론적으로, 현대의 PX4는 Action API 계층을 통제하는 엄격한 상태 머신(FSM)의 규제, 거친 조종으로부터 기체 동역학을 사수하는 포락선(Envelope) 제약, 비대칭적 즉각 명령을 위한 Kill-switch(FTS) 라는 3중 비행 제어 안전 구조를 통해 최상의 임무 수행 신뢰성을 확보하고 연계 기술의 안정성을 배가시킨다.
