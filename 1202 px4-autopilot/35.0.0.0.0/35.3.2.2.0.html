35.3.2.2. 시스템 식별자(System ID) 및 컴포넌트 식별자(Component ID)의 충돌 방지를 위한 동적 호스트 구성 프로토콜(DHCP) 모방 라우팅

다수 무인기가 투입되는 대규모 다중 에이전트(Swarm Multi-Agent) 운용 환경에서, 운영자가 수십 대 기체의 파라미터(MAV_SYS_ID 및 MAV_COMP_ID)를 일일이 수동으로 설정하고 펌웨어에 기록하는 방식(Ardupilot의 고전적 SYSID_THISMAV 정적 할당 방식)은 치명적인 인적 오류(Human Error)를 유발할 수 있다. 특히 야전(Field)에서 예비 기체를 긴급하게 편대에 합류시켜야 할 경우, 식별자 중복 선언으로 인한 MAVLink 라우팅 충돌은 전체 군집의 궤적 제어를 붕괴시키는 원인이 된다.

이러한 식별자 충돌(Collision)의 원천적 방지 및 유연한 자산 추가 구조를 확보하기 위해, 최신 통신 아키텍처에서는 마치 컴퓨터 네트워크의 동적 호스트 구성 프로토콜(DHCP, Dynamic Host Configuration Protocol)과 유사하게 MAVLink 노드의 식별자를 중앙 관제 또는 마스터(Master) 에이전트가 런타임에 동적으로 할당(Dynamic Allocation)하는 모방 라우팅 체계를 설계한다. 본 절에서는 이 아키텍처의 이론과 구현 방식을 심도 있게 분석한다.

1. MAVLink 동적 할당 체계의 아키텍처 이해

네트워크 계층(Network Layer)의 DHCP 서버가 IP 주소 풀(Pool)을 관리하며 클라이언트에게 사용 가능한 IP를 임대(Lease)해 주듯, 다중 에이전트 관제망에서는 지상 관제 시스템(GCS) 또는 마스터 드론 탑재 PC가 ’식별자 할당 서버(ID Allocator)’의 역할을 수행한다.

이 체계에서 비행 제어기(PX4-Autopilot)나 보조 컴퓨터 클라이언트는 다음과 같은 일련의 핸드셰이크(Handshake) 절차를 거쳐 자신만의 고유 네트워크 주소를 취득한다.

1.1 초기 탐색 및 할당 요청(Discovery & Request)

PX4 기체가 처음 부팅되어 MAVLink 네트워크(예: 공통 WiFi 모듈 또는 Mesh RF망)에 접속할 때, 자신의 임시 식별자(Provisonal ID, 보통 MAV_SYS_ID = 0 또는 특정 브로드캐스트용 약속된 ID)와 하드웨어 전역 고유 식별자(UUID)를 담아 브로드캐스트 영역에 ‘ID 할당 요청(Allocation Request)’ 메시지를 송출한다.

1.2 서버의 고유 식별자 임대(Allocation & Lease)

할당 서버(GCS 백엔드 모듈)는 내부 메모리에 유지 중인 ’사용 중인 ID 맵(In-use ID Map)’을 조회하여 충돌하지 않는 가용 sysid(예: 11번부터 254번 사이의 값)를 선택한다. 이후, 클라이언트의 UUID를 키(Key) 값으로 삼아 해당 가용 sysid를 매칭시킨 후, 네트워크 상에 ‘ID 할당 응답(Allocation Response)’ 패킷을 퍼블리싱한다.

1.3 클라이언트 식별자 반영 및 재부팅(Application & Reboot)

클라이언트(PX4)는 수신된 응답 패킷의 대상 UUID가 자신의 하드웨어 UUID와 일치하는지 검증한다. 일치할 경우, 내부 메시징 시스템(uORB)을 통해 MAV_SYS_ID 파라미터를 수신된 값으로 동적 변경(Dynamic Update)하고 이를 비휘발성 메모리(EEPROM/FRAM)에 영구 저장(Save)하거나 런타임에 즉시 텔레메트리 송출 헤더에 반영한다.

2. 컴포넌트 식별자(Component ID)의 계층적 라우팅 분리

단일 기체 내부에서도 비행 제어기 보드(MAV_COMP_ID_AUTOPILOT1 = 1), 짐벌 제어기(MAV_COMP_ID_GIMBAL = 154), 보조 컴퓨터(MAV_COMP_ID_ONBOARD_COMPUTER = 191) 등 수많은 서브 시스템이 공존한다. 동적 식별 체계 하에서는 GCS가 이 하위 컴포넌트들까지 충돌 없이 라우팅할 수 있어야 한다.

PX4의 MAVLink 라우터 데몬(Daemon)은 브릿지(Bridge) 역할을 수행하며, 외곽망(GCS 방향)에서 들어오는 지시를 적절한 내부 직렬(Serial) 포트 또는 UDP 포트로 전달(Forwarding)한다. DHCP 기반 MAV_SYS_ID 동적 할당이 완료되어 기체가 15번 SYSID를 취득했다면, 기체 내부에 결선된 짐벌이나 카메라는 자신의 SYSID를 메인 비행 제어기의 15번으로 추종(Snooping)하여 일치시키되, COMPID만 자신의 고유 ID로 발송함으로써 완벽한 계층적 식별 분리도(Hierarchical Identification Isolation)를 달성한다.

3. ROS2 환경(uXRCE-DDS)에서의 클라이언트 키(Client Key) 동적 할당 차이

MAVLink 기반 군집 제어와 대조적으로, 최근 각광받는 PX4 펌웨어와 ROS2의 미들웨어 통합 아키텍처(uXRCE-DDS)에서는 식별자 충돌 제방 방식이 더욱 고도화되어 있다.

ROS2 런타임 환경 하에서 PX4 기체는 Micro ROS 에이전트 측과 통신하기 위해 전역 고유 클라이언트 키(Client Key)를 생성한다. 즉, MAVLink의 단순한 1바이트 크기 sysid (1~254) 한계를 벗어나, DDS(Data Distribution Service) 표준에 따른 복잡한 네임스페이스(Namespace) 기반의 토픽 격리(Topic Isolation) 방식을 적극 따른다. PX4 v1.14 이후 도입된 UXRCE_DDS_CFG 관련 파라미터는 노드 명명 규칙(Naming Convention)을 하드웨어 고유 ID나 동적 할당된 접두어(Prefix)로 자동 설정하도록 지원하여, GCS 및 ROS2 백엔드 단에서 충돌 없는 완벽한 동적 호스트 맵핑을 보장할 수 있도록 발전하였다.

결론적으로, 다중 에이전트 시스템에서 DHCP 철학을 차용한 동적 식별 라우팅 아키텍처는 운용 초기화(Setup) 에러 확률을 제로(0)에 가깝게 수렴시키며, 하드웨어 교체와 네트워크 확장이 빈번한 자율 군집 작전 프로그래머가 반드시 이해하고 구현해야 할 소프트웨어 인프라스트럭처의 핵심이다.
