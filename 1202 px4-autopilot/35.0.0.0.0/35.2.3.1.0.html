35.2.3.1. 루프백 네트워크 인터페이스(Loopback Network Interface) 기반 전송 제어 프로토콜(TCP) 및 사용자 데이터그램 프로토콜(UDP) 포트 포워딩

1. 개요 및 설계 철학

네트워크 기반 물리 엔진 시뮬레이션(SITL, Software In The Loop) 환경에서 **루프백 네트워크 인터페이스(Loopback Network Interface)**는 동일한 호스트(Host) 내부에서 구동되는 다수의 프로세스 간에 트래픽을 라우팅하기 위한 가상의 네트워크 인터페이스(127.0.0.1 또는 localhost)를 의미한다. 현대의 비행 제어기(Flight Controller) 시뮬레이션 환경은 PX4-Autopilot 인스턴스, 가상 물리 엔진(예: Gazebo, JMavSim), 그리고 지상 관제 시스템(QGroundControl)이 독립된 프로세스로 실행되며 네트워크 소켓(Socket) 통신을 통해 유기적으로 결합된 분산 시스템이다.

PX4-Autopilot(v1.14 기준)은 이러한 로컬 네트워크 통신에서 **전송 제어 프로토콜(Transmission Control Protocol, TCP)**과 **사용자 데이터그램 프로토콜(User Datagram Protocol, UDP)**의 특성을 목적에 맞게 혼용하여 사용한다. 무인기 시스템의 통신은 시시각각 빠르게 변하는 고빈도의 원격 측정(Telemetry) 데이터와, 단 한 번의 유실도 치명적인 결과를 초래할 수 있는 임무(Mission) 및 제어 명령 데이터로 분류할 수 있다. PX4는 효율적인 포트 포워딩(Port Forwarding) 구조를 설계하여 이 두 가지 프로토콜 간의 트래픽을 지능적으로 분리 및 관리한다.

2. PX4 SITL 환경의 네트워크 아키텍처 및 포트 정책

PX4-Autopilot은 다중 에이전트(Multi-agent) 시뮬레이션 환경에 대응하기 위해 체계적인 베이스 포트 할당 수식을 도입하였다. 단일 호스트 내부에서 패킷 충돌을 방지하기 위해 각 기체 인스턴스(instance)는 고유한 포트 오프셋(Offset)을 가지게 된다.

2.1 기본 MAVLink 커뮤니케이션 토폴로지

단일 기체 시뮬레이션 작동 시 주요 기본 포트는 다음과 같이 할당된다.

  • UDP 포트 14550: QGroundControl(QGC) 등 지상 관제 프로토콜 통신용 엔드포인트
  • UDP 포트 14540: 오프보드 제어 목적의 MAVSDK, ROS 2, MAVROS 등 외부 응용 프로그래밍 인터페이스(API) 통신용
  • TCP 포트 4560 (기본값): 시뮬레이터 엔진(Gazebo)과 PX4 간의 고속 인터록(Inter-lock) 및 직접 연동 통신 포트

다중 기체를 시뮬레이션 할 경우, 인스턴스 번호 $i (예: i=0, 1, 2, ...)에 따라 포트는 다음의 수식에 의해 동적으로 할당된다.

  • Mavlink API 포트 (외부 제어): 14540 + i
  • 시뮬레이터 포트 (물리 엔진 연결): 4560 + i

2.2 통신 브로커 역할의 포트 포워딩

mavlink_main 모듈은 내부적으로 여러 가지 MAVLink 인스턴스로 분기 실행되며, 브로드캐스트 패킷이나 지정된 IP/포트를 대상으로 하는 유니캐스트 패킷의 포워딩 역할을 담당한다. 특히 MAVLink Router 모듈을 도입하면 루프백 인터페이스 상의 트래픽을 다른 물리 인터페이스(이더넷, Wi-Fi)로 투명하게 브릿징(Bridging)하여 분산 시뮬레이션 환경을 구성할 수도 있다.

3. UDP 대 TCP 프로토콜의 채택 전략 및 특성 비교

시뮬레이터 환경에서 각각의 통신 프로토콜이 수행하는 역할은 다음과 같이 극명한 차이를 나타낸다.

3.1 사용자 데이터그램 프로토콜(UDP)의 이점 및 적용

UDP는 비연결지향형(Connectionless) 프로토콜로, 상태 설정과 해제 오버헤드가 없으며 최선형 네트워크(Best-effort Network) 전달을 수행한다.

  • 적용 대상: 센서 IMU 스트림, 고빈도 자세 데이터 수신(ATTITUDE), Heartbeat 메시지
  • 논리적 접근: PX4 알고리즘 특성상 과거 시점의 센서 데이터가 유실되더라도 최신 스트림 데이터를 즉각 반영(State Estimation)하는 것이 중요하므로, 핸드셰이크(Handshake)와 재전송 지연 스파이크(Latency Spike)가 발생할 수 있는 TCP보다 UDP가 절대적으로 유리하다.
  • 네트워크 모델: INADDR_ANY를 사용하여 루프백을 포함한 모든 인터페이스에서 패킷을 수신하도록 구조화된다.

3.2 전송 제어 프로토콜(TCP)의 이점 및 적용

TCP는 연결지향형(Connection-oriented) 프로토콜로, 패킷 무결성 점검, 순서 보장, 오류 수정(Error Recovery) 메커니즘을 내장한다.

  • 적용 대상: SITL 록스텝(Lock-step) 타임스탬프 동기화 및 가상 물리 센서 데이터 파이프라인. 파라미터 동기화(Parameter Transfer) 및 FTP(File Transfer Protocol).
  • 논리적 접근: 특히 록스텝(Lock-step) 스케줄링에서는 역학 엔진의 물리적 1스텝에 대해 펌웨어 로직이 정확하게 1틱(Tick)을 회신할 수 있는 신뢰성 있는 통로가 필요하다. 만약 루프백 환경에서 이 패킷이 엇갈리면 시뮬레이션 물리 시계와 펌웨어 시스템 시계 간에 괴리가 발생하여 발산(Divergence) 현상을 유발할 수 있다. 따라서 PX4 SITL 브릿지는 TCP 커넥션을 물리 엔진과 결합하는데 특화되어 있다.

5. 소스 코드(Source Code) 구조 분석

PX4의 소스 트리 내부, 특히 src/modules/mavlink 및 src/modules/simulator 공간에서 소켓 관련 API가 중개(Broker) 코어 역할을 수행한다. UDP 및 TCP 바인딩을 위한 리눅스 환경의 버클리 소켓(Berkeley Sockets) 계층이 핵심 인터페이스를 구성한다.

루프백 상에서 포원을 생성하고 바인딩하는 대표적인 추상 로직은 서버 소켓 할당 과정에서 이루어진다.

[예시: TCP 포트 설정 및 수신 대기 (C++ 코드 흐름)]

// src/modules/simulator/simulator_mavlink.cpp 내부 TCP 수신 포트 할당 흐름
struct sockaddr_in sim_addr {};
sim_addr.sin_family = AF_INET;
sim_addr.sin_port = htons(_tcp_port); // 다중 기체 구동 시 동적으로 계산된 포트 입력
sim_addr.sin_addr.s_addr = htonl(INADDR_ANY);

// TCP 소켓 생성
_simulator_tcp_fd = socket(AF_INET, SOCK_STREAM, 0);

// 서버 바인드(Bind) 및 수신 대기(Listen)
bind(_simulator_tcp_fd, (struct sockaddr *)&sim_addr, sizeof(sim_addr));
listen(_simulator_tcp_fd, 5); // 큐 사이즈 5로 물리엔진 접속 허용 대기

기본적으로 루프백 네트워크 내부에서는 통신 병목 현상이 디바이스 하드웨어에 의해 발생하는 경우가 드물며, 논블로킹(Non-blocking) 소켓과 poll() 또는 select() 함수 체계를 도입하여 다중 클라이언트 트래픽이 비동기적으로 스루풋(Throughput) 한계치까지 처리될 수 있도록 병행 제어(Concurrency Control)가 적용된다.

UDP의 경우 recvfrom 시스템 콜을 이용하여 가상 IP를 할당받은 MAVLink 클라이언트의 세션 유실 없이 트래픽을 처리하며, 목적지 포트를 기반으로 uORB 채널로 패킷을 발행(Publishing)한다.

6. ArduPilot과의 차이점 분석

ArduPilot의 SITL 환경의 경우 네트워크 프로토콜 구성에 있어 PX4와 철학적 차이가 존재한다.
ArduPilot은 전통적으로 MAVProxy라는 역동적인 파이썬 기반의 멀티플렉서(Multiplexer) 겸 커맨드 라인 제어 툴을 필연적으로 경유하는 구조를 장려했다. SITL 인스턴스는 디폴트로 특정 TCP 채널(일반적으로 5760번 포트)을 오픈하고, MAVProxy 클라이언트가 이 인터페이스로 접속하여 다시 무수한 UDP 경로(--out=127.0.0.1:14550 및 --out=127.0.0.1:14551 등)로 포트 포워딩을 파편화 시키는 방식을 채용한다.

반면, PX4-Autopilot은 독립된 런타임 내에서 MAVLink Main 모듈을 여러 인스턴스로 동시에 부팅시켜 내부적으로 포트를 독립 할당하는 모놀리식(Monolithic) 분산 구조 특성을 더 직접적으로 활용한다. 별도의 외부 유틸리티 브릿지 없이 다양한 채널 개통 스크립트(예: rcS 혹은 ROMFS 관련 시동 시퀀스 스크립트)를 통해 시작 단계에서 직접 포트를 선언하고 열람하는 방식은 지연(Latency)을 최소화하고 의존성을 탈피하려는 아키텍처 설계와 밀접하게 관련이 있다.

7. 결론

루프백 인터페이스 관점에서 분석한 PX4 프레임워크의 네트워킹 전략은 신뢰도가 최우선시 되는 물리적 록스텝 환경(TCP)과, 즉각적이고 손실 탄력적인 명령/상태 수신 환경(UDP) 간의 명확한 임무 분담 구조를 통해 완성된다. 이러한 기초적인 네트워크 포트 포워딩 지식은 대규모 다중 에이전트 군집 알고리즘 연구 및 ROS2-DDS 하이브리드 미들웨어를 구축하기 위한 핵심적인 토대가 된다.
