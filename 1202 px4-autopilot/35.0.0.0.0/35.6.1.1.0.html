35.6.1.1. 전역 좌표계 기반 목적지 벡터 설정 및 도달 상태(State Reached) 판별을 위한 수용 반경(Acceptance Radius)의 구면 기하학적 정의

고도화된 자율 비행 임무(Autonomous Mission)의 뼈대를 이루는 3차원 목적지 벡터는, 지상 관제소(GCS) 사용자 인터페이스 상에서는 인간 친화적인 위도(Latitude)와 경도(Longitude)라는 지리 데이터(Geodata)로 할당된다. 그러나 PX4 비행 제어기 내부의 코어 수학 엔진은 이 거대한 구형(Spherical) 지구본 위에서 이도 저도 아닌 방위각만으로 직접 미적분학의 칼날을 휘두르지 않는다.

MAVSDK를 통해 오프보드에서 인가된 전역 좌표계(WGS84) 기반의 목적지 벡터 배열이 백엔드 내부에서 어떻게 기계어로 변환되며, 3차원 공간상의 점에 투사된 수학적 ’수용 반경(Acceptance Radius)’이 무자비한 유한 상태 머신의 강제 천이 트리거(Transition Trigger)로 어떻게 작용하는지를 기하학적 관점에서 심층 분석한다.

1. WGS84 전역 좌표계의 국소 평면 NED (Local North-East-Down) 투영(Projection) 매핑

MAVLink MISSION_ITEM_INT 통신 패킷을 통해 기체의 텔레메트리로 밀려 들어오는 목적지의 위경도 좌표는 부동소수점 정밀도 손실 병목을 막기 위해 10^7 이 곱해진 고강도 정수형(Integer) 글로벌 데이터 포맷이다. 하지만 PX4의 가장 낮은 단계에 위치한 백엔드 위치 제어기(Position Controller)는 순수 데카르트 좌표계(Cartesian Coordinate System) 기반의 강체 물리 법칙(뉴턴 역학 미분 방정식)만을 절대적으로 신봉한다. 미세한 곡률을 가진 위경도 체계를 초당 250번 도는 제어 루프 안에서 직접 삼각 함수로 곡면 계산하는 것은 마이크로컨트롤러(MCU)에 압도적이고 치명적인 부동소수점(Floating-point) 연산 발열 및 레이턴시 부하를 일으킨다.

이를 타개하기 위해, PX4의 Navigator 임무 스케줄러 및 EKF2(Extended Kalman Filter) 추정 모듈은 기체의 최초 이륙 지점(Home)을 절대 원점 (0, 0, 0) 으로 삼는 Local NED (North-East-Down) 평면 마스킹 좌표계를 프레젠테이션 계층 아래에 은밀하게 생성해 둔다.
임무 배열 큐(Queue)에서 다음 차례의 웨이포인트 글로벌 WGS84 좌표를 꺼내 들면, 수학 라이브러리는 현재 Home 원점을 기준으로 구면 삼각법(Spherical Trigonometry) 기법 혹은 방위각 등거리 투영(Azimuthal Equidistant Projection) 공식을 적용하여, 지구의 곡률을 무시한 채 해당 목표점이 원점으로부터 반듯하게 몇 미터 북쪽(North), 몇 미터 동쪽(East), 몇 미터 지표면 뚫고 아래쪽(Down)에 위치해 있는지의 3차원 상대 평면 미터(Meter) 단위 벡터 \mathbf{p}_{\text{target}} = [x_t, y_t, z_t]^T 로 완벽히 선형 변환(Linear Transformation)해 낸다. 캐스케이드(Cascade) P-PID 위치 제어 루프는 위경도의 존재조차 모른 채 오직 이 변환된 가벼운 NED 미터 벡터만 맹목적으로 추종한다.

2. 도달 상태(State Reached) 판별과 수용 반경(Acceptance Radius)의 제논의 역설(Zeno’s Paradox)

차기 임무 노드가 NED 벡터로 캐스팅되면, 기체는 즉각 목표 벡터 \mathbf{p}_{\text{target}} 를 향해 날카로운 추진 각을 틀고 날아간다. 이때, 소프트웨어 아키텍처 상 가장 커다란 논리적 난관에 부딪히게 된다. “과연 언제(When) 기체가 이 공간상의 웨이포인트에 정확히 ’도착’했다고 프로그램 적으로 확정 짓고, 다음 임무 노드 코드 블록으로 포인터를 넘길 것인가?” 하는 근원적인 이산화 딜레마이다.

만약 코드가 수학적으로 위치 오차 벡터 \mathbf{e} = \mathbf{p}_{\text{target}} - \mathbf{p}_{\text{current}} 가 정확히 [0, 0, 0]^T 이 되는, 소수점 이하까지 완벽한 0의 수렴 질점을 요구하도록 깐깐하게 하드코딩 되어있다고 가정해 보자.
거친 현실의 물리 세계에서는 예측 밖의 측풍(Wind Disturbance), GPS 안테나 중심 위치의 대기권 멀티패스(Multipath) 수신 노이즈, 그리고 PID 제어기의 고주파 피드백 진동 등으로 인해 기체의 질량 중심은 영원토록 그 미립자만 한 목적지의 중앙 점을 스쳐 지나치고 빙글빙글 선회만 거듭할 뿐이다. 결국, 도착했다는 True 판정을 영원히 내리지 못하고 배터리를 모두 소진해 버리는 비극적인 제논의 역설(Zeno’s Paradox) 혹은 화장실 소용돌이(Toilet-bowling) 무한 루프 버그에 빠지게 된다.

이를 타파하기 위해 MAV_CMD_NAV_WAYPOINT 임무 아이템 파라미터는 Param 2 메모리 공간에 수용 반경(Acceptance Radius, R_{\text{accept}}) 이라는 스칼라 톨러런스(Tolerance) 값을 의무적으로 강제 할당하도록 표준 명세가 박혀 있다.

이는 단순한 통과 여부 검사가 아니라, 3차원 무중력 공간의 목적지 좌표 \mathbf{p}_{\text{target}} 를 둥근 중심점으로 하여 반지름이 R_{\text{accept}} 길이인 가상의 거대한 3차원 구면(Sphere) 혹은 실린더 침투 볼륨을 기하학적으로 프로그래밍하는 행위이다. Navigator 내부의 도달 판별 알고리즘(Arrival Condition Checker) 루프는 매 틱(Tick)마다 EKF 모듈에서 추정하는 현재 기체의 위치 \mathbf{p}_{\text{current}} 와 목적지 간의 공간 3차원 유클리디안 노름(Euclidean Norm) 연산을 맹렬히 수행하여 다음의 수학적 부등식을 검열한다.

||\mathbf{p}_{\text{target}} - \mathbf{p}_{\text{current}}||_2 \leq R_{\text{accept}}

이 파라미터 부등식 조건 로직이 참(‘True’) 으로 평가되는 찰나의 이산적 순간, 즉 육중한 기체 질량 중심의 EKF 추정 궤적이 이 가상의 수용 반경 구면 유리창을 부수고 내부 부피로 스치듯 침투하는 즉시! FSM의 상태 천이기는 가차 없이 ‘항로 도달 완료(Mission Item Reached)’ 비동기 이벤트를 발생시키고 스케줄러의 배열 인덱스 포인터를 +1 증가시켜 다음 임무 아이템 코드 구역으로 제어권을 단호하게 넘겨버린다.

결과적으로 PX4의 수용 반경(Acceptance Radius) 파라미터화 시스템은 백색 소음(White Noise)과 공기역학적 비선형 외란(Disturbance)이 난무하는 거친 아날로그 대자연의 변수를 조율 무마하여, 강박적이고 융통성 없는 디지털 상태 머신(State Machine) 코드가 물리적 무한 루프 교착에 빠지지 않고 정해진 궤적 시나리오를 끝까지 연속 연주할 수 있도록 이음새에 윤활유를 쳐주는 가장 똑똑하고 위대한 기하학적 타협 공학(Geometrical Compromise Engineering)의 산물이라 정의할 수 있다.
