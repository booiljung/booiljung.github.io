35.4.1.1. 세계 측지계(WGS84) 기준 타원체 모델 데이터 파싱 및 타원체 고도(Ellipsoid Height)와 평균 해수면 고도(AMSL) 상호 변환

위성 항법 시스템(GNSS) 기반 자율 비행의 근간은 지구를 단순화된 기하학적 모델로 정의하는 데서 출발한다. PX4-Autopilot은 글로벌 산업 표준인 세계 측지계 1984(WGS84, World Geodetic System 1984) 규격을 사용하여 3차원 공간 좌표계를 구축한다.

그러나 이 과정에서 많은 GCS 개발자와 자율 비행 프로그래머가 간과하여 치명적인 고도 제어 실패(예: 지면 충돌)를 유발하는 핵심 변수가 바로 고도의 기준면(Datum) 설정 문제이다. 본 절에서는 기체에 탑재된 GNSS 수신기가 WGS84 타원체 고도(Ellipsoid Height) 데이터를 어떻게 파싱(Parsing)하고, 이를 비행 역학에 실질적으로 사용되는 평균 해수면 고도(AMSL, Above Mean Sea Level)로 상호 변환하는지 그 내부 아키텍처를 상세히 구명한다.

1. WGS84 타원체 고도와 지오이드(Geoid) 기복의 기하학적 관계

지구는 완벽한 구형(Sphere)이 아니며 찌그러진 타원체이다. WGS84는 이 지구를 수학적으로 가장 잘 근사한 매끄러운 ‘기준 타원체(Reference Ellipsoid)’ 모델을 제공한다. GNSS 위성에서 계산되어 내려오는 순수 3차원 삼각측량 고도 데이터는 바로 이 매끄러운 **타원체 표면으로부터의 수직 거리(타원체 고도, h)**이다.

하지만 실제 인류의 항공 운용 기준은 중력 포텐셜이 동일한 표면인 지오이드(Geoid), 즉 수면이 파도 없이 잔잔할 때의 해수면 연장선인 **평균 해수면 고도(AMSL, H)**를 따른다. 이 두 기준면 사이의 간극을 지오이드 기복(Geoid Undulation 또는 Geoid Separation, N)이라 정의하며, 지역에 따라 타원체가 해수면보다 높을 수도 길 수(-100m ~ +85m)도 있다.

방정식으로 나타내면 다음과 같다.
H \text{ (AMSL)} = h \text{ (Ellipsoid)} - N \text{ (Geoid Separation)}

PX4 내부의 vehicle_gps_position uORB 토픽에는 이 세 가지 고도계 데이터가 모두 독립된 필드(alt, alt_ellipsoid, epv)로 엄격하게 분리되어 밀리미터(mm) 단위 정수형으로 보관된다.

2. GNSS 프로토콜 스트림 파싱 및 비동기 변환 아키텍처

PX4의 GPS 드라이버(예: gps 모듈 내 u-blox, NMEA, RTCM 파서)는 직렬 포트(UART)를 통해 유입되는 이진 바이트 스트림을 비동기적으로 해독(Decoding)한다.

특히 u-blox의 NAV-PVT (Navigation Position Velocity Time Solution) 메시지의 경우, 하드웨어 수신기가 지오이드 모델 룩업 테이블(Lookup Table)을 내장하고 있어 타원체 고도(h)와 AMSL(H)을 동시에 출력해 준다. 드라이버 코드는 인터럽트 기반으로 패킷 페이로드(Payload)를 검증한 직후, 다음과 같이 파싱 로직을 수행한다.

// PX4 u-blox 파서(Parser) 펌웨어 내부 모델링 의사코드
if (pvt_msg_valid) {
    sensor_gps_s gps_data{};
    
    // 1. 위도 및 경도 (WGS84 타원체 투영면 기준, 1e7 스케일링)
    gps_data.lat = pvt_msg.lat;
    gps_data.lon = pvt_msg.lon;
    
    // 2. 평균 해수면 고도 (AMSL) 파싱 및 mm 단위 정규화
    gps_data.alt = pvt_msg.hMSL; 
    
    // 3. 타원체 고도 (Ellipsoid Height) 파싱
    gps_data.alt_ellipsoid = pvt_msg.height;
    
    // uORB 토픽 발행을 통한 비동기 전파
    orb_publish(ORB_ID(sensor_gps), gps_pub_fd, &gps_data);
}

만약 NMEA 0183 표준 규격의 GGA 문장처럼 지오이드 모델링이 취약하거나 생략된 스트림을 파싱할 때에는, PX4 펌웨어 내부(또는 GCS 환경)에 구축된 EGM96(Earth Gravitational Model 1996) 15도/30도 해상도의 메모리 매핑 격자망(Grid-mesh)에서 삼차원 선형 보간법(Trilinear Interpolation)을 수행하여 MAVLink 텔레메트리로 넘어가기 전에 해당 N 값을 복원(Reconstruction)해야만 한다.

3. 타원체-해수면 고도 착각으로 인한 비행 제어기 발산 경고

과거 Ardupilot의 일부 레거시 버전이나 초보 GCS 프로그래머들이 MAVLink 스트림을 해독할 때, GLOBAL_POSITION_INT 메시지의 alt(AMSL 고도) 필드와 오프보드 명령용 SET_POSITION_TARGET_GLOBAL_INT의 고도 필드 간의 기준면(Datum)을 혼용하여 치명적인 추락을 야기하곤 했다.

예를 들어 대한민국의 평균 지오이드 기복은 약 20m~30m 내외이다. 타원체 고도 50m를 목적지로 명령을 하달했는데 비행 제어 루프(ECL)가 이를 AMSL 50m로 해석한다면, 기체는 목표 고도보다 20~30m 낮게 순항하게 되어 산악 지형이나 고층 빌딩과 충돌하게 된다.

이를 방지하기 위해 PX4의 확장 칼만 필터(EKF2) 모듈은 부팅 초기 EKF2_GPS_CTRL 매개변수 매스크(Mask)를 엄격히 감사(Audit)한다. GPS를 절대 고도 센서로 사용할지(주로 지형이 평탄한 해상 비행), 기압계(Barometer)를 메인 AMSL 에스티메이터로 삼되 GPS 타원체 고도로 드리프트(Drift)만 상쇄할 것(기본값)인지 통제(Gating)함으로써 WGS84 타원체 지오메트리의 맹점을 소프트웨어 아키텍처 다이나믹스 차원에서 완벽하게 은닉(Encapsulation)한다.

결론적으로, 다차원 좌표계 간의 추상화 계층을 분리하고 WGS84 규격의 태생적 물리 한계를 극복하는 고도의 융합 및 파싱 로직은 오차율 제로(Zero-tolerance) 공간 컴퓨팅을 위한 자율 비행 소프트웨어의 모세혈관과도 같은 존재이다.
