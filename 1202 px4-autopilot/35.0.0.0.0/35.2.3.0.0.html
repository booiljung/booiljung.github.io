35.2.3 가상 물리 엔진(Gazebo) 연동을 위한 하드웨어 인 더 루프(HITL) 및 소프트웨어 인 더 루프(SITL) 네트워크 토폴로지

1. 개요

현대의 무인항공기(UAV) 개발 파이프라인에서 시뮬레이션 환경은 필수불가결하다. 실제 비행 시험 이전에 제어 알고리즘의 안정성을 극대화하기 위해, 실제 센서 입력 및 엑추에이터 출력과 동일한 환경을 모사하는 가상 물리 엔진(Virtual Physics Engine) 연동이 요구된다. PX4-Autopilot 생태계에서는 전통적으로 Gazebo Classic(v11) 및 최신의 Gazebo Harmonic(이전 명칭: Ignition Gazebo)이 표준 물리 엔진으로 활용된다.

본 장에서는 제어 루프의 실행 주체와 하드웨어 개입 여부에 따라 구분되는 소프트웨어 인 더 루프(Software-In-The-Loop, SITL)와 하드웨어 인 더 루프(Hardware-In-The-Loop, HITL)의 네트워크 토폴로지 구조를 심층적으로 분석한다. 기준 펌웨어는 PX4 v1.14 및 QGroundControl v4.3 이상을 전제로 한다.

2. 소프트웨어 인 더 루프(SITL) 네트워크 토폴로지

SITL 환경은 드론의 비행 제어 스택(Flight Control Stack) 전체가 호스트 PC 환경에서 소프트웨어 프로세스로 실행되는 형태를 말한다. 이 경우 물리 센서와 액추에이터 대신 Gazebo 시뮬레이터 플러그인이 그 역할을 대신하게 되며, 이들 컴포넌트 간의 고속 데이터 교환 네트워크가 핵심이다.

2.1 SITL의 내부 통신 아키텍처

PX4-Autopilot의 SITL 환경은 일반적으로 로컬 루프백(Local Loopback, 127.0.0.1) 네트워크를 이용해 다수의 프로세스가 상호 운용되는 분산 시스템 구조를 띤다.

  1. Simulator Plugin 통신: Gazebo 내부의 px4_sitl_gazebo 플러그인과 PX4 SITL 펌웨어 바이너리 간에 UDP 소켓 브리지가 형성된다. 기본적으로 UDP 포트 4560을 사용하여 센서 데이터와 모터(PWM) 명령을 초당 수백 회(Hz) 주기로 교환한다.
  2. MAVLink 바인딩 토폴로지: PX4의 내장 mavlink 모듈은 복수의 포트를 순차적으로 개방하여 다중 접속 환경을 제공한다.
  • UDP 포트 14550: QGroundControl과 같은 지상 관제 시스템(GCS)과의 텔레메트리 스트림 연동.
  • UDP 포트 14540: MAVSDK나 ROS 2(MAVROS)와 같은 오프보드(Offboard) 비동기 제어기와의 연동.

2.2 Ardupilot SITL 시스템과의 구조적 차이점

Ardupilot의 SITL은 전통적으로 메인 펌웨어가 단일 스레드 사이클을 강력하게 통제하는 모놀리식(Monolithic) 성격이 짙으며, JSON이나 TCP 스트림 방식을 이용한 서드파티 시뮬레이터 연동(예: RealFlight 혹은 Webots 연동) 통신 프로토콜이 파편화되어 있다. 반면, PX4의 SITL 아키텍처는 uORB 메시징 계층의 모듈성을 그대로 상속받기 때문에, 네트워크 인터페이스 UDP 플러그인 모듈 하나만 교체하면 Gazebo, jMAVSim, AirSim 등으로의 물리 엔진 스위칭(Switching)이 매우 유연하고 일관성 있게 이루어진다.

3. 하드웨어 인 더 루프(HITL) 네트워크 토폴로지

HITL 환경은 호스트 PC에서 실행되는 Gazebo의 물리 엔진 및 렌더링 환경을 유지하되, 자율 비행 알고리즘과 내부 센서 처리 로직이 실제 하드웨어인 비행 제어기(Flight Controller, 예: Pixhawk 6C) 내에서 실시간(Real-time)으로 구동되는 방식이다. 이를 통해 실물 마이크로컨트롤러(MCU)의 부동소수점 연산 한계, 통신 타이밍 제약, 그리고 RTOS(NuttX) 정밀 스케줄링 간섭을 실제와 가장 유사하게 검증할 수 있다.

3.1 HITL의 혼합(Hybrid) 네트워크 토폴로지

SITL이 순수한 로컬 네트워킹(127.0.0.1)에 전적으로 의존한다면, HITL은 PC와 하드웨어 간의 시리얼(Serial) 기반 USB 또는 이더넷(Ethernet) 전송 계층을 필연적으로 통과해야 한다.

  1. 시뮬레이터-FC 데이터 흐름: 이 토폴로지에서 QGroundControl은 단순한 GCS가 아닌 강력한 중계자(Broker/Proxy) 역할을 함께 수행한다. Gazebo 플러그인에서 기인하는 가상 물리 센서 데이터(MAVLink HIL_SENSOR 패킷)가 네트워크 루프백으로 QGroundControl에 도달하고, 이를 다시 Pixhawk(FC) 내부로 USB 직렬 라인(Baudrate: 921600 이상)을 통해 주입(Injection)한다.
  2. 제어 출력 역전송 흐름: Pixhawk 디바이스에서 처리된 제어 결과(자세 추정 및 PID 제어 출력)는 HIL_ACTUATOR_CONTROLS 메시지 형태로 패키징되어, 역으로 QGroundControl을 경유하여 Gazebo에 도달하고 물리 엔진상의 모터 RPM을 결정한다.

3.2 통신 대역폭(Bandwidth) 제약 극복과 최적화

HITL 토폴로지의 가장 큰 기술적 한계는 로컬 UDP에 비해 상대적으로 대역폭이 비좁은 USB 시리얼 및 직렬 인터페이스를 이용한다는 점이다. 특히 고주파수(250Hz 이상)의 가상 IMU 센서 데이터를 동기화하는 과정에서 전송 병목이 발생하면, 실시간성(Real-time Constraints) 제약조건이 무너져 적분 오차(Integration Error)가 급격히 발산하고 결국 가상 기체가 추락하게 된다.

따라서, HITL 네트워크를 설정할 때에는 불필요한 원격 측정(Low Priority Telemetry) 스트림 전송률(Rate)을 대폭 하향 조정하고, HIL 계열 패킷의 제어 루프 우선순위를 강제로 승상(Elevate)시키는 튜닝 파라미터(SYS_HITL 기동 활성화 등) 작용 구조를 완벽히 이해해야 한다.

4. 결론

PX4-Autopilot 환경 하에서 SITL 및 HITL 토폴로지에 대한 완전한 통찰은 신뢰성 높은 R&D 워크플로우를 담보하는 선결 요건이다. 알고리즘 설계의 초기 단계에서는 통신 대역폭 병목이 원천 차단된 UDP 기반 SITL 구조를 집중 응용하여 로직 중심의 신속 검증을 신속히 완료해야 한다. 이후, 점진적으로 실제 타겟 보드의 처리 병목 현상과 통신 지연시간을 연동시키는 통신 병목 기반 HITL 토폴로지로 검증을 이행함으로써, 파괴적인 물리적 사고의 위험성 없이 최고 수준의 임베디드 검증 무결성(Verification Integrity)을 획득할 수 있다.
