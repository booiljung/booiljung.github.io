35.3.1.1. 범용 비동기 송수신(UART) 직렬 통신(Serial Communication)의 하드웨어 흐름 제어(Hardware Flow Control) 및 패리티(Parity) 규격 설정

무인항공기 시스템(UAS)의 분산 네트워크 환경에서 PX4-Autopilot과 보조 컴퓨터(Companion Computer), 지상 관제 시스템(GCS), 혹은 다양한 외부 탑재체(Payload) 간의 안정적인 데이터 교환은 필수적이다. 이러한 이기종 시스템 간의 물리적 계층 통신에서 가장 널리 사용되는 인터페이스 중 하나가 바로 범용 비동기 송수신(UART, Universal Asynchronous Receiver-Transmitter) 기반의 직렬 통신(Serial Communication)이다. 본 절에서는 PX4-Autopilot 관점에서 UART 직렬 통신의 신뢰성을 확보하기 위한 핵심 메커니즘인 하드웨어 흐름 제어(Hardware Flow Control)와 패리티(Parity) 규격 설정 방법에 대해 심층적으로 분석한다.

1. UART 직렬 통신과 하드웨어 흐름 제어의 필요성

UART 직렬 통신은 송신측(TX)과 수신측(RX)이 독립된 클럭을 사용하여 데이터를 비동기적으로 전송하는 방식이다. MAVLink와 같은 고빈도 원격 측정(Telemetry) 데이터나 센서 데이터 스트림이 전송될 때, 수신측 마이크로컨트롤러(MCU)나 보조 컴퓨터의 처리 속도가 송신측의 데이터 발생 속도를 따라가지 못하면 수신 버퍼 오버플로우(Buffer Overflow) 현상이 발생한다. 이는 곧 패킷 손실률(Packet Loss Rate) 증가와 시스템 제어 루프의 치명적인 지연(Latency)으로 이어진다.

이러한 병목 현상을 하드웨어 레벨에서 원천적으로 차단하기 위해 하드웨어 흐름 제어(Hardware Flow Control) 기법이 적용된다. 주로 RTS(Request To Send)와 CTS(Clear To Send) 핀을 추가로 활용하여, 수신측이 데이터를 받을 준비가 되었는지를 송신측에 물리적 전기 신호로 명확히 알린다.

1.1 하드웨어 흐름 제어(RTS/CTS)의 동작 원리

RTS/CTS 흐름 제어 메커니즘은 다음과 같은 순서로 작동한다.

  1. 송신측 데이터 전송 준비: 데이터를 보낼 준비가 된 노드(예: PX4 비행 제어기)는 자신의 RTS 핀을 논리적 하이(High) 상태로 전환하여 수신 노드(예: MAVSDK 기반 보조 컴퓨터)에게 통신 재개를 요청한다.
  2. 수신측 버퍼 검사 및 승인: 수신 노드는 내부 버퍼 여유 공간과 현재 중앙 처리 장치(CPU) 부하를 검사한다. 수신 준비가 완료되면 자신의 RTS 핀 상태를 변경하여, 결과적으로 송신측의 CTS 핀에 활성화 신호를 인가한다.
  3. 데이터 스트리밍: 송신측은 CTS 핀 레벨이 유효함을 확인하고 시리얼 버스를 통해 데이터 프레임을 전송 방출한다.
  4. 전송 일시 중단: 수신측 버퍼가 가득 차면 수신 노드는 RTS 신호를 해제하며, 이를 감지한 송신측은 CTS 핀 비활성화를 인식하여 즉각적으로 전송을 멈춘다(Throttle).

2. 패리티(Parity) 비트를 활용한 프레임 무결성(Integrity) 검증

고전압 모터나 변속기(ESC, Electronic Speed Controller)가 탑재된 무인항공기는 본질적으로 전자기 간섭(EMI, Electromagnetic Interference) 요인을 다수 내포하고 있다. 이러한 환경에서는 UART 선로 상에 일시적인 비트 반전(Bit-flip) 오류가 발생할 가능성이 상존한다. 비대칭 물리 계층의 오류를 감지하기 위해 데이터 프레임에 추가되는 규격이 패리티 비트(Parity Bit)이다.

패리티 검사는 송신 데이터의 1의 개수를 짝수(Even) 또는 홀수(Odd)로 맞추기 위해 1비트를 추가하는 단순하면서도 효과적인 기법이다. 무인항공기 직렬 통신에서는 흔히 ’8 데이터 비트, 비패리티(None Parity), 1 정지 비트(8N1)’가 기본값으로 널리 사용되지만, 통신 신뢰성이 극도로 요구되는 미션 크리티컬(Mission-Critical)한 센서(예: RTK GPS, 고정밀 라이다) 연동 시에는 ‘짝수 패리티(Even Parity)’ 구성을 채택하여 데이터 오염 여부를 1차적으로 필터링하도록 설계한다. MAVLink 프로토콜 자체적으로 순환 중복 검사(CRC, Cyclic Redundancy Check)를 제공하지만, 하드웨어 UART 페리페럴(Peripheral) 단에서 패리티 검사를 선행함으로써 유효하지 않은 패킷에 대한 소프트웨어 스택 오버헤드를 경감할 수 있다.

3. PX4-Autopilot 및 MAVSDK에서의 규격 설정 실무

최신 PX4-Autopilot 펌웨어(v1.14 기준) 및 QGroundControl 환경에서 UART 포트의 하드웨어 흐름 제어 및 패리티를 설정하는 방식은 시스템 파라미터(Parameter)를 통해 이루어진다. 이와 연동되는 보조 컴퓨터 측의 MAVSDK(C++ 기준) 역시 직렬 포트 초기화 시점의 연결 문자열(Connection String)에 해당 물리 규격 지시자를 명시해야 한다.

3.1 PX4 파라미터 환경 설정

PX4에서는 TELEM1, TELEM2 등 사전에 정의된 직렬 포트별로 속도(Baudrate)와 흐름 제어 여부를 결정할 수 있다. QGroundControl을 통해 다음과 같은 파라미터를 조작한다.

  • SER_TEL1_BAUD: TELEM1 포트의 통신 속도를 정의한다. 보조 컴퓨터와의 연결 시 대역폭 제약을 해소하기 위해 통상적으로 921600 bps를 인가한다.
  • SER_TEL1_CFG: TELEM1 포트의 라우팅 목적을 정의한다. MAVLink Companion 또는 MAVLink Onboard 프로필을 할당한다.
  • MAV_1_FLOW_CTRL: MAVLink 인스턴스 1번(보통 TELEM1에 바인딩됨)에 대한 하드웨어 흐름 제어 활성화 여부를 결정한다.
  • 0: 비활성화 (흐름 제어 없음, RX/TX 케이블만 사용하는 환경)
  • 1: 활성화 (RTS/CTS 하드웨어 흐름 제어 강제, 케이블 6가닥 완전 결선 필수)

Ardupilot 환경에서는 SERIAL1_OPTIONS 파라미터 비트마스크(Bitmask)를 조작하여 패리티 및 인버전(Inversion)을 구성하는 데 반해, PX4-Autopilot은 드라이버 단의 포트 재편성 로직을 통해 NuttX RTOS 상에 상대적으로 더 직관적인 파라미터 네이밍 인덱스를 제공하는 설계 철학을 보여준다. 센서별 통신 인터페이스 활성화 코드는 NuttX의 board_config.h 및 시작 스크립트(Start-up script) 상에 termios 구조체를 조작하는 형태로 심층 구현되어 있다.

3.2 MAVSDK C++ 연결 문자열(Connection String) 구성

보조 컴퓨터에서 MAVSDK를 구동할 때, 직렬 포트 객체를 인스턴스화하는 과정에서 하드웨어 흐름 제어 유무와 패리티 규격을 명시해야 양방향 통신 협상이 성립된다.

#include <mavsdk/mavsdk.h>
#include <iostream>

using namespace mavsdk;

int main(int argc, char** argv) {
    Mavsdk mavsdk_instance;
    ConnectionResult connection_result;

    // 통상적인 8N1 연결 및 흐름 제어 비활성화 (8 데이터 비트, None 패리티, 1 스탑 비트)
    // std::string connection_url = "serial:///dev/ttyTHS1:921600";
    
    // MAV_1_FLOW_CTRL=1 설정에 대응하는 하드웨어 흐름 제어 명시적 인가(Flow Control = On)
    // 연결 문자열의 쿼리(Query) 파라미터로 flow-control을 true로 활성화한다.
    std::string connection_url = "serial:///dev/ttyTHS1:921600?flow-control=true";

    connection_result = mavsdk_instance.add_any_connection(connection_url);

    if (connection_result != ConnectionResult::Success) {
        std::cerr << "PX4 직렬 연결 오류: " << connection_result << std::endl;
        return 1;
    }

    std::cout << "PX4 시스템 UART 포트 연결 수립 시도 중..." << std::endl;
    
    // 이후 비동기 오버 보드(Offboard) 및 텔레메트리 시스템 검색(System Discovery) 동적 할당 수행
    
    return 0;
}

위의 예제와 같이 보조 컴퓨터의 Linux 직렬 장치 노드(예: NVIDIA Jetson의 /dev/ttyTHS1)에 접근할 때, 쿼리 인자(Query Argument) flow-control=true 속성을 부여하여 드라이버 계층의 POSIX tcsetattr 시스템 콜(System Call) 시 하드웨어 제어 플래그(CRTSCTS)가 삽입되도록 구조화해야 한다. 이러한 일치성 검증 과정은 다중 에이전트 토폴로지 구성 단계에서 치명적인 동기화 오류를 방지하는 근본적인 토대가 된다.

4. 규격 불일치로 인한 시스템 치명적 결함 현상 분석

하드웨어 흐름 제어 및 패리티 설정이 PX4-Autopilot 비행 제어기와 보조 컴퓨터 간에 불일치(Mismatch)할 경우 다발하는 시스템 장애 유형은 다음과 같다.

  1. RTS/CTS 라인 단선 또는 미연결 상태에서의 무한 대기 (Deadlock): PX4 파라미터 MAV_1_FLOW_CTRL은 1로 설정되었으나 물리적 배선에서 RTS/CTS 케이블이 누락된 경우, 송신측은 CTS 신호를 영원히 대기하게 된다. 이로 인해 MAVLink 통신 채널이 완전히 교착 상태(Deadlock)에 빠져 시스템 인스턴스 탐색(System Discovery) 단계를 전면 통과하지 못하는 치명적 록업(Lock-up) 오류를 유발한다.
  2. 패리티 비트 전파 지연 (Propagation Delay) 및 폐루프 불안정성: 고속 제어 주기가 요구되는 오프보드(Offboard) 루프에서 일관성 없는 패리티 비티 규격 강제는 페이로드 역직렬화(Deserialization) 단계에서 쓰레기 값(Garbage Value)을 초래한다. 이는 가짜 MAVLink 메시지 해독으로 이어지며, 폐루프(Closed-loop) 제어기에서 예측 범위를 벗어난 이상 제어 명령 스파이크(Spike) 현상으로 변질되어 무인기 추락과 직결될 수 있다.

결론적으로 단위 시스템의 안정성 확보 차원에서 UART 하드웨어 흐름 제어와 패리티 설정은 PX4 파라미터 계층, 물리적 결선 계층, 그리고 MAVSDK 응용 프로그램 계층 등 3단계 통합 검증(Three-tier Integration Verification)이 완벽히 병렬로 성립되어야 한다.
