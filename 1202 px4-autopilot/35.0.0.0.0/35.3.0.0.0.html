35.3. 비동기 분산 시스템 탐색(System Discovery) 및 다중 에이전트 토폴로지 관리

1. 개요 및 설계 철학

군집 비행(Swarm Flight) 및 다중 무인기 협업(Multi-UAV Collaboration) 환경에서는 수십에서 수백 대의 변동 가능한 기체들이 단일 네트워크 상호 작용계에 진입하고 이탈하는 과정이 빈번하게 발생한다. 이처럼 참여 노드(Node)의 숫자가 가변적인 동맹 네트워크에서는 사전에 각 에이전트의 IP 주소나 포트, 혹은 통신 채널을 고정적으로 할당(Static Allocation)하는 방식이 심각한 제약으로 작용하게 된다.

PX4-Autopilot은 이러한 문제를 해결하기 위해 비동기 분산 시스템 탐색(Asynchronous System Discovery) 메커니즘을 전면적으로 채택하고 있다. 이는 중앙 집중형 디렉터리 서버나 브로커에 의존하지 않고, 각 에이전트(기체, 짐벌, 컴패니언 컴퓨터 등)가 네트워크 지형에 자신의 존재를 자율적으로 전파(Broadcast/Multicast)하면 관제 시스템(QGroundControl)이나 주변 노드들이 이를 동적으로 감지하여 통신 토폴로지(Topology)를 스스로 갱신해나가는 탈중앙화 철학을 반영한다.

2. 비동기 시스템 탐색(System Discovery) 메커니즘

PX4 시스템 탐색의 가장 핵심적인 중추는 MAVLink 프로토콜의 HEARTBEAT 메시지에 기반한다.

2.1 하트비트(HEARTBEAT) 기반 존재 알림 모델

전원이 인가되고 PX4 펌웨어가 부팅을 완료하면 mavlink_main 모듈은 통상적으로 1Hz 주기(초당 1회)로 네트워크 상의 모든 활성화된 채널(UART, UDP 멀티캐스트, TCP 등)에 자신을 식별할 수 있는 하트비트 메시지를 송출하기 시작한다.
이 메시지 내부에는 기체의 유형(Type), 비행 제어기 계열(Autopilot Class, 예: MAV_AUTOPILOT_PX4), 그리고 현재의 비행 모드(Mode)와 시스템 상태(State) 정보가 고밀도로 압축되어 있다.

2.2 지상 관제 시스템의 동적 바인딩(Dynamic Binding)

QGroundControl이나 MAVSDK 기반의 관제 애플리케이션은 네트워크 포트를 열고 수신 대기(Listen) 상태로 진입한다. 애플리케이션은 이전에 보지 못한 완전히 새로운 패킷 소스(IP/Port 결합판)나 식별자로부터 HEARTBEAT 메시지를 포착하는 즉시, 해당 노드를 새로운 에이전트로 간주하여 내부 레지스트리(Vehicle Manager)에 등록한다. 이 순간부터 양방향 통신 세션(Session)이 공식적으로 성립된다.

3. 시스템 식별자 및 다중 컴포넌트 아키텍처

한 네트워크 공간 안에서 에이전트들을 충돌 없이 격리하고 통신을 라우팅하기 위해 PX4는 두 가지 깊이의 식별자를 둔다.

3.1 시스템 식별자(System ID / SYSID_THISMAV)

드론 한 대(또는 전체 모빌리티 기체 한 대)를 논리적으로 묶어주는 고유 숫자이다. (1~255)
다중 에이전트 시스템에서 1번 기체는 SYSID_THISMAV = 1, 2번 기체는 SYSID_THISMAV = 2와 같이 고유하게 파라미터가 설정되어야 네트워크 혼선이 발생하지 않는다. 메시지 라우터들은 이 Target System ID를 읽고 패킷을 취사선택한다.

3.2 컴포넌트 식별자(Component ID)

단일 시스템(동일한 System ID) 내부에서도 다양한 두뇌들이 존재할 수 있다. 예를 들어 비행을 주관하는 PX4 펌웨어는 MAV_COMP_ID_AUTOPILOT1 컴포넌트 식별자를 가지며, 짐벌 제어기는 MAV_COMP_ID_GIMBAL, 동반 컴퓨터(Companion Computer, 라즈베리 파이 등)는 MAV_COMP_ID_ONBOARD_COMPUTER 아이디를 지닌다.
관제 시스템은 System ID : Component ID의 조합을 독립적인 엔티티(Entity)로 취급하여 하위 컴포넌트와의 직통 제어 명령(Command Long)을 수행할 수 있다.

void MultiVehicleManager::_vehicleHeartbeatInfo(LinkInterface* link, int vehicleId, int componentId)
{
    // 이미 등록된 기체인지 Hash Map 조회
    if (!_vehicles.contains(vehicleId)) {
        // 존재하지 않는다면 새로운 Vehicle 객체 동적 할당(Instantiation)
        Vehicle* vehicle = new Vehicle(link, vehicleId, componentId, this);
        _vehicles.insert(vehicleId, vehicle);
        
        // UI 컴포넌트로 새 기체 출현 이벤트 발송 (플러그 앤 플레이)
        emit vehicleAdded(vehicle);
    }
}

6. ArduPilot 대비 토폴로지 관리의 유연성

ArduPilot 생태계 역시 MAVLink를 공통 분모로 사용하므로 HEARTBEAT 기반의 디스커버리 철학을 고유하게 공유한다. 그러나 인프라스트럭처 측면에서 다중 기체를 운용할 때 ArduPilot은 종종 중앙 MAVProxy 노드를 통해 트래픽을 정적 라우팅(Static Routing) 하는 관행이 짙다. 반면, PX4 생태계는 ROS 2(Micro XRCE-DDS)와의 브릿징 통합을 코어 설계부터 염두에 두어, MAVLink 레벨의 디스커버리 뿐만 아니라 DDS (Data Distribution Service) 고유의 RTPS(Real-Time Publish Subscribe) 자동 탐색 기능까지 이중으로 활용할 수 있게 토폴로지 관리의 지평을 넓혔다. 결과적으로 PX4는 수천 개 단위의 토픽 지향적(Topic-oriented) 메시지들이 에이전트들 사이를 유영하는 복잡한 IoT 메쉬(Mesh) 네트워크 망에도 유연히 스며드는 구조를 지향한다.

7. 결론

비동기 분산 시스템 탐색 기법은 드론 군집 관제에서 시스템의 생존성과 확장성을 담보하는 최전선 방어막이다. PX4의 HEARTBEAT 기반 동적 바인딩 시스템은 어떠한 네트워크 변경이나 기체의 임의 추가/삭제 상황에서도 관제국 레지스트리의 정합성을 보장하며, 다중 에이전트 간의 유기적 토폴로지 구성을 “제로 컨피규레이션(Zero-Configuration)” 수준으로 끌어올리는 혁신적인 통신 아키텍처이다.
