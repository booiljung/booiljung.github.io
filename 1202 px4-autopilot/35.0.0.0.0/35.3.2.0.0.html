35.3.2. 다중 에이전트(Swarm Multi-Agent) 환경에서의 시스템 식별 및 레지스트리(Registry) 동적 할당

현대의 무인항공기 시스템(UAS)은 단일 기체 운용을 넘어 수십, 수백 대의 드론이 협력하는 다중 에이전트(Swarm Multi-Agent) 편대 비행 기술로 진화하고 있다. PX4-Autopilot 기반의 군집 비행 통신망에서는 다수의 비행 제어기와 지상 관제 시스템(GCS), 보조 컴퓨터(Companion Computer), 그리고 다양한 센서 탑재체(Payload)가 하나의 거대한 MAVLink 라우팅 네트워크를 형성한다. 이 거대한 네트워크 내에서 각 노드(Node)를 고유하게 식별하고 트래픽을 정확히 라우팅하기 위해서는 체계적인 시스템 식별자(System ID) 할당과 레지스트리(Registry) 동적 관리 기술이 필수적으로 수반되어야 한다.

본 절에서는 MAVSDK 및 PX4 환경에서 다중 에이전트를 구성할 때 직면하게 되는 시스템 식별의 원리와 동적 할당 메커니즘에 대해 심층적으로 논의한다.

1. MAVLink 라우팅 아키텍처와 식별자(ID) 체계

MAVLink 프로토콜(v2.0 기준)의 프레임 헤더(Header)는 모든 메시지에 대해 발신자의 출처를 명시하는 두 가지 핵심 식별자를 포함한다.

  • 시스템 식별자 (System ID, sysid): 네트워크 상의 독립된 시스템(예: 드론 1호기, 지상 관제 시스템, 별도의 안테나 트래커)을 구분한다. 허용 범위는 1부터 255까지이다.
  • 컴포넌트 식별자 (Component ID, compid): 단일 시스템(동일한 sysid) 내에 존재하는 하위 모듈(예: 비행 제어기 보드 본체, 짐벌 컨트롤러, 보조 컴퓨터, 충돌 회피용 카메라)을 구분한다.

PX4-Autopilot의 기본 시스템 식별자는 파라미터 MAV_SYS_ID (기본값: 1)로 정의되며, 비행 제어기 컴포넌트 식별자는 MAV_COMP_ID (기본값: 1, MAV_COMP_ID_AUTOPILOT1)로 부여된다. 반면 QGroundControl(QGC)과 같은 일반적인 GCS는 보통 sysid 255를 사용하여 라우팅 트리의 최상단에서 클라이언트를 모니터링한다.

단일 기체 운용 환경에서는 이 기본값을 그대로 사용할 수 있으나, 다중 에이전트 환경에서는 각 기체가 중복되지 않는 MAV_SYS_ID를 유지해야 한다. 식별자가 충돌할 경우, QGroundControl은 두 기체의 텔레메트리 값을 번갈아 가며 한 화면에 겹쳐서 표시하게 되어 치명적인 상태 오인식(State Hallucination) 및 제어 권한 탈취를 유발한다.

2. 정적 할당(Static Allocation)의 한계와 동적 식별의 필요성

과거 Ardupilot 기반의 군집 비행에서는 운영자가 이륙 전 지상 관제소 모듈(Mission Planner)을 통해 각 기체의 SYSID_THISMAV 파라미터를 1, 2, 3과 같이 순차적으로 수동 기입하는 정적 할당(Static Allocation)에 압도적으로 의존해 왔다.

하지만 군집의 규모가 수십 대 이상으로 확장됨에 따라 정적 할당이 지니는 다음과 같은 한계점이 대두되었다.

  1. 유지보수의 경직성: 기체 하나가 고장나 예비 기체(Spare Drone)로 교체해야 할 때, 하드웨어 교체와 더불어 파라미터 수정이라는 병목 공정이 추가된다.
  2. 플러그 앤 플레이(Plug-and-Play) 부재: 보조 컴퓨터나 이기종 센서가 동적으로 네트워크에 진입(Attach) 및 이탈(Detach)할 때마다 포트 포워딩 룰이나 ID를 재정의해야 하므로 유연성이 극도로 떨어진다.

이러한 문제를 해결하기 위해 최신 PX4 군집 통신 구현체 (예: MAVSDK의 System Discovery 모듈 및 ROS2의 uXRCE-DDS 에이전트)는 런타임 상에서 노드를 동적으로 탐색하고 백엔드 레지스트리(Registry)에 해시 맵핑하는 동적 시스템 식별 기법을 차용한다.

3. MAVSDK 기반의 동적 시스템 탐색(System Discovery) 아키텍처

MAVSDK (C++ 또는 Python 환경)는 UDP/TCP 또는 직렬(Serial) 포트를 바인딩한 후, 공중망이나 로컬 인터페이스를 통해 브로드캐스트(Broadcast)되는 심장박동(Heartbeat) 메시지를 지속적으로 모니터링한다. MAVSDK 내부의 상태 머신(Internal State Machine)은 새로운 sysid를 지닌 Heartbeat가 수신될 때마다 이를 새로운 자산으로 판단하고, 이를 내부 메모리에 할당된 ‘시스템 레지스트리(System Registry)’ 딕셔너리에 동적으로 등록한다.

3.1 동적 레지스트리 할당 파이프라인

다음은 MAVSDK 내부에서 처리되는 시스템 식별 및 콜백(Callback) 핸들러의 논리적 흐름이다.

sequenceDiagram
    participant D1 as Drone 1 (sysid: 11)
    participant D2 as Drone 2 (sysid: 12)
    participant SDK as MAVSDK (GCS/Server)
    
    D1->>SDK: Heartbeat (sysid=11, compid=1) 발송 (1Hz)
    SDK->>SDK: Registry 검색: sysid 11 없음
    SDK->>SDK: MAVSDK System 객체 동적 할당 및 UUID 매핑
    SDK-->>SDK: OnDiscover 콜백 트리거 (새 기체 인식)
    D2->>SDK: Heartbeat (sysid=12, compid=1) 발송
    SDK->>SDK: Registry 검색: sysid 12 없음
    SDK->>SDK: MAVSDK System 객체 동적 할당 및 UUID 매핑
    SDK-->>SDK: OnDiscover 콜백 트리거
    
    Note over D1, SDK: 일정 시간(예: 3초) Heartbeat 미수신 시
    SDK->>SDK: Timeout 감지기 활성화
    SDK->>SDK: Registry에서 sysid 11 임시 비활성화
    SDK-->>SDK: OnTimeout 콜백 트리거 (통신 단절 처리)

이러한 동적 탐색 패턴 비동기 제어 루프를 통해 개발자는 std::vector<std::shared_ptr<System>> (C++ 기준)와 같이 동적으로 확장되는 컨테이너를 사용하여 군집 에이전트를 배열(Array) 형태로 관리할 수 있다.

3.2 분산 환경에서의 하위 컴포넌트 라우팅 제어

분산 시스템에서는 보조 컴퓨터가 기체 내에 탑재되어 있고(예: 시스템 ID 동일, 컴포넌트 ID만 다름), 이를 관할하는 상위 관제 서버 시스템이 존재하는 3-Tier 아키텍처가 흔히 구성된다. MAVSDK의 텔레메트리 라우터(Telemetry Router)는 수신되는 토픽의 컴포넌트 ID 필드까지 해독하여, 특정 메시지가 짐벌(Gimbal) 제어기로부터 온 것인지 비행 제어기(Autopilot)로부터 온 것인지를 구분하는 레지스트리를 유지한다.

결론적으로, 다중 에이전트 환경에서 시스템 통제를 온전히 지탱하는 근본 지주는 “동적 객체 바인딩 체계“이다. PX4 네트워크 백본(Backbone) 설계 시 런타임 식별자 자동 발급 및 Heartbeat 기반 생존 주기 트래킹(Tracking) 메커니즘을 소프트웨어 스택의 최상단 관제 레이어(GCS Frontend)와 최하단 통신 미들웨어(Middleware) 양쪽에 유기적으로 구현하는 것이 성공의 핵심이라 할 수 있다.
