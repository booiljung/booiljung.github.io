43.2.1 MAV_CMD_COMPONENT_ARM_DISARM (Command ID 400) 명령의 페이로드 바이트 오더링(Byte Ordering) 및 엔디안(Endianness) 독립적 구조체 설계

지상 관제 시스템(GCS)과 비행 제어기(PX4-Autopilot) 간의 통신에서 시동(Arming) 제어는 기체의 모터 구동 등 치명적인 물리적 변화를 초래한다. 따라서 MAV_CMD_COMPONENT_ARM_DISARM (Command ID: 400) 명령을 전송할 때, 데이터의 직렬화(Serialization)는 하드웨어 아키텍처의 엔디안(Endianness) 차이로 인한 오류 결함이 없도록 완벽하게 독립적으로 설계되어야 한다. 이 절에서는 MAVLink 프로토콜이 이 치명적인 명령을 어떻게 바이트 레벨에서 정렬(Ordering)하고, 이종 아키텍처(예: x86 기반 GCS와 ARM Cortex-M 기반 PX4) 간 엔디안 차이를 극복하는 구조체를 선언하는지 시스템 공학적 관점에서 심도 있게 분석한다.

1. MAVLink 페이로드의 리틀 엔디안(Little-Endian) 표준화 철학

임베디드 시스템 네트워크에서 다중 바이트 데이터를 메모리에 배치하는 방식(Byte Ordering)은 프로세서 아키텍처에 따라 리틀 엔디안(Little-Endian)과 빅 엔디안(Big-Endian)으로 나뉜다.
MAVLink 프로토콜은 기체 탑재 컴퓨터(Companion Computer)와 비행 제어기(Flight Controller)의 주류를 이루는 ARM Cortex 계열 프로세서가 기본적으로 리틀 엔디안을 채택하고 있음에 착안하여, 모든 온더와이어(On-the-wire) 패킷 페이로드의 바이트 오더링을 **리틀 엔디안(Little-Endian)**으로 엄격하게 표준화하였다.

COMMAND_LONG 메시지를 통해 전송되는 MAV_CMD_COMPONENT_ARM_DISARM의 주요 구조는 다음과 같다.

block-beta
    title COMMAND_LONG Payload Byte Ordering (Little-Endian)
    columns 4
    space:4
    block:p1:4
        "Param 1 (Float 32)\nByte 0 : Byte 1 : Byte 2 : Byte 3"
    end
    block:p2:4
        "Param 2 (Float 32)\nByte 4 : Byte 5 : Byte 6 : Byte 7"
    end
    block:cmd:1
        "Command (UInt16)\nByte 28:29"
    end
    block:target:1
        "Target Sys(UInt8)\nByte 30"
    end
    block:targetcomp:1
        "Target Comp(UInt8)\nByte 31"
    end
    block:conf:1
        "Confirm (UInt8)\nByte 32"
    end

Command 필드는 16비트 부호없는 정수(UInt16)이며, 400 (0x0190) 값을 가진다. 리틀 엔디안 표준에 따라 메모리와 네트워크 패킷 상에는 0x90 (Low Byte)가 먼저 오고 0x01 (High Byte)이 뒤따르는 [0x90, 0x01] 순서로 직렬화된다.

2. 엔디안(Endianness) 독립적인 C/C++ 구조체 설계 기법

QGroundControl(GCS)이 구동되는 데스크톱 PC(주로 x86_64 리틀 엔디안)와 PX4-Autopilot이 구동되는 STM32(ARM 리틀 엔디안)는 현재 주류 환경에서는 엔디안이 동일하다. 그러나 이식성(Portability)을 극대화하기 위해 MAVLink 라이브러리(C/C++ 헤더)는 호스트 아키텍처의 엔디안을 컴파일 타임에 판별하여 독립적인 매핑을 수행한다.

mavlink_msg_command_long.h 헤더 파일 내부에 정의된 직렬화 로직은 다음과 같은 C 언어의 구조체 패킹(Packing) 기술을 사용한다.

2.1 #pragma pack(1)을 통한 패딩(Padding) 바이트 제거

컴파일러는 구조체 멤버 접근 속도를 높이기 위해 워드(Word) 단위로 메모리를 정렬(Alignment)하며 빈 공간에 패딩(Padding) 바이트를 삽입한다. 이 패딩이 포함된 채널로 메모리를 통째로 복사(memcpy)하면, 아키텍처마다 패딩 규칙이 달라 패킷 붕괴(Corruption)가 발생한다.

#pragma pack(push, 1) // 1바이트 단위 정렬 강제 (패딩 제거)
typedef struct __mavlink_command_long_t {
    float param1;      // Offset: 0
    float param2;      // Offset: 4
    float param3;      // Offset: 8
    float param4;      // Offset: 12
    float param5;      // Offset: 16
    float param6;      // Offset: 20
    float param7;      // Offset: 24
    uint16_t command;  // Offset: 28  => Command ID 400 (MAV_CMD_COMPONENT_ARM_DISARM)
    uint8_t target_system;    // Offset: 30
    uint8_t target_component; // Offset: 31
    uint8_t confirmation;     // Offset: 32
} mavlink_command_long_t;
#pragma pack(pop)

MAVLink는 #pragma pack(push, 1) 매크로를 활용하여 패딩 바이트를 원천적으로 제거함으로써, 어떠한 컴파일러 스펙에서도 정확히 33바이트의 페이로드가 연속되도록 보장한다.

2.2 바이트 정렬법 독립적 타입 캐스팅 연산

데이터를 버퍼에 쓸 때 MAVLINK_NEED_BYTE_SWAP 매크로를 평가한다. 만약 호스트 시스템이 빅 엔디안 시스템(예: 모토로라 68000 계열이나 특정 임베디드 네트워크 장비)인 경우, 컴파일러는 _mav_put_uint16_t()와 같은 매크로 치환 함수 내부에서 즉시 바이트 스왑(Swap) 연산(__builtin_bswap16)을 수행하여 리틀 엔디안으로 변환 적재한다.

이러한 고도의 엔디안 독립적 구조체 캐스팅 체계 덕분에, 응용 계층의 프로그래머는 QGroundControl이나 PX4 코드 내에서 아키텍처 의존성에 신경 쓰지 않고 cmd.command = MAV_CMD_COMPONENT_ARM_DISARM; 이라고 선언하기만 하면 된다.

3. PX4와 Ardupilot의 바이트 오더링 파싱(Parsing) 관제 차이

바이트 스트림이 비행 제어기로 인입되었을 때, 이를 역직렬화(Deserialization)하여 구조체 객체로 환원하는 파서는 PX4와 Ardupilot 모두 mavlink_parse_char() 함수를 기반으로 한다.

  • PX4-Autopilot 아키텍처:
    PX4(v1.14.0 기준)에서는 송신과 마찬가지로 MAVLink 파서 헤더를 통해 수신된 리틀 엔디안 페이로드를 단일 memcpy() 혹은 포인터 타입 캐스팅으로 mavlink_command_long_t 구조체로 환원한다. 이 변환된 구조체는 즉각적으로 uORB 메시지인 vehicle_command_s 포맷으로 복사(Copy)된다. uORB 메시지 구조체(NuttX 커널 메모리 기반) 역시 호스트의 네이티브 엔디안을 따르므로, 이 시점에서 안전하게 msg.command == 400 비교 검증이 수행된다.

  • Ardupilot 아키텍처:
    Ardupilot 역시 동일한 MAVLink C/C++ 파서를 공유하므로 직렬화/역직렬화의 엔디안 핸들링 레이어는 PX4와 거의 동일하다. 다만, Ardupilot은 역직렬화된 직후 MAVLink 객체 그대로 클래스 핸들러(예: GCS_MAVLINK::handle_command_long())로 포인팅하여 각 비행 모드에 직접 전달하는 패턴이 강하다. 반면 PX4는 MAVLink 객체를 즉시 uORB 네이티브 메시지로 “형변환(Translation)“함으로써 네트워크 프로토콜 명세(MAVLink)와 코어 운영체제 상태 버스(uORB)를 철저히 디커플링(Decoupling)하는 차별점을 보여준다.

4. QGroundControl 소스 코드를 통한 직렬화 처리 시점 이해

QGroundControl(C++ Qt 프레임워크 기반, v4.3.0 기준) 소스 코드에서 시동 조작이 발생할 때, 바이트 오더링 직렬화가 완료되는 시점은 통신 스레드로 데이터 버퍼가 인계되는 직전이다.

  1. 사용자가 화면 하단의 Slide to Arm 스와이프를 수행한다.
  2. FirmwarePlugin 클래스의 구현체(PX4FirmwarePlugin.cc 등)가 Vehicle::sendMavlinkCommand()를 호출한다.
  3. 이 함수는 백엔드 스레드의 MAVLinkProtocol::packMavlinkMessage()를 호출하여 MAV_CMD_COMPONENT_ARM_DISARM 열거형 상수를 위에서 설명한 패킹 구조체의 command 필드에 할당한다. 이때 16비트와 32비트(Float)의 모든 바이트 치환이 리틀 엔디안 룰로 처리된다.
  4. 직렬화된 Byte Array 버퍼가 시리얼 포트(또는 UDP 소켓) 객체의 write() API에 밀어 넣어짐으로써 GCS 측의 임무가 종료된다.

결과적으로, MAV_CMD_COMPONENT_ARM_DISARM 명령의 페이로드 바이트 오더링 및 독립적 구조체 설계는 드론 시스템의 극단적 이기종 하드웨어 환경(이기종 CPU, 이기종 OS, 다양한 컴파일러 파편화) 속에서도 시동 제어와 같은 크리티컬(Critical) 명령이 한 치의 비트 오차 없이 관제될 수 있도록 보장하는 가장 근본적인 프로토콜 보안 및 시스템 무결성 기술이다.
