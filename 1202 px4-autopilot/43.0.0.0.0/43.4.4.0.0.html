43.4.4 전제 조건 불만족 시 발생하는 상태 전이 거부 예외 처리 및 인터럽트 서비스 루틴(ISR) 기반 실시간 응답 지연 한계 평가

1. 개요

이 절에서는 PX4-Autopilot 환경에서 시동(Arming) 전제 조건(Pre-flight Checks) 검증이 불만족되었을 때, commander 모듈이 상태 전이를 어떻게 안전하게 거부(Refusal)하고 시스템 예외 처리를 수행하는지 공학적으로 분석한다. 또한 하드웨어 제어 루프의 최전선에 있는 인터럽트 서비스 루틴(ISR, Interrupt Service Routine)이 예외 처리에 개입할 경우 발생하는 실시간 응답 지연(Latency) 한계와 이를 극복하기 위한 설계 전략을 다룬다.

2. 전제 조건 불만족 시의 상태 전이 거부(State Transition Denial) 매커니즘

PX4의 시동 메커니즘은 단순한 ’스위치 온’이 아닌, 복합적인 센서 및 상태 추정기(EKF2)의 수렴도를 평가하는 관문이다.

2.1 ArmingStateMachine의 안전한 거부(Graceful Rejection)

조종자나 GCS로부터 시동 명령이 인가될 때, commander 내부의 PreFlightCheck 클래스는 GPS PDOP, 나침반(Compass) 간섭 여부, 자이로스코프 캘리브레이션 상태, 배터리 전압 조건 등을 순차적으로 검증한다.
이 중 하나라도 허용 오차(Threshold)를 벗어나 불만족(Fail) 판정을 받게 되면 ArmingStateMachine은 상태 전이 요청을 즉시 파기한다. 이때 단순히 시동을 걸지 않는 것에 그치는 것이 아니라, 상태 전이 거부 예외 처리를 다음과 같이 수행한다.

  1. 상태 레지스터 불변 유지: Armed = false 상태를 원자적(Atomic)으로 유지하여 액추에이터 출력 포트가 유휴 상태를 벗어나는 것을 하드웨어 레벨에서 봉쇄한다.
  2. 명시적 거부 코드 반환: MAVLink 인터페이스를 통해 GCS로 COMMAND_ACK (결과 코드: MAV_RESULT_DENIED 또는 MAV_RESULT_TEMPORARILY_REJECTED)를 전송하여 조종자에게 거부 사유를 명확히 피드백한다.
  3. 청각적 알람(Buzzer) 및 시각적 피드백(LED): Tune Control 모듈로 실패 에러 비프음을 송출하여 현장(Line-of-Sight)의 조종자가 직관적으로 인지하게 한다.

2.2 예외 처리 코드의 스레드 무결성

상태 전이가 거부되는 예외 처리 블록(Exception Block) 역시 POSIX 뮤텍스 락 구역 내에서 처리되어야 한다. 전제조건을 검증하다가 실패하여 에러 비프음을 울리기 위해 메모리 상의 알람 상태를 갱신하는 찰나의 순간에, 다른 스레드가 불완전한 시스템 상태를 읽어가 버리는 잠재적 경쟁 상태를 방지하기 위함이다.

3. 인터럽트 서비스 루틴(ISR) 기반 실시간 응답 지연(Latency) 한계 평가

임베디드 무인항공기에서 가장 중요한 제어 주기는 400Hz~1000Hz 범위에서 동작하는 IMU(관성 측정 장치)의 데이터 수집 인터럽트(ISR, Interrupt Service Routine)이다.

3.1 시동 예외 처리에 따른 타이밍 오버헤드

전제 조건에 실패하여 commander 모듈이 MAVLink 거부 메시지를 조립하고, 부저(Buzzer) 주파수를 갱신하며, SD 카드 로깅 버퍼에 에러 이벤트를 플러시(Flush)하는 일련의 작업은 백그라운드 태스크로서는 상당히 무거운 레지스터 복사(Copy) 연산을 동반한다.
만일 이러한 거부 예외 처리 루틴이 ISR과 충돌하게 된다면 어떻게 될까? ISR은 OS 커널 단에서 스케줄러의 통제를 무시하고 강제로 CPU 제어권을 탈취하는 특권 레벨 코루틴이다. 즉, commander 모듈이 상태 전이 거부를 처리하다 말고 IMU 데이터 수집 ISR에 의해 멈추게(Preempt) 된다.
이로 인해 찰나의 지연시간(Microseconds 수준)이 누적되며, commander의 응답 지연(Latency) 한계를 벗어날 위험이 제기된다.

3.2 응답 지연 방어 및 Bottom Half 처리 기법

PX4는 이러한 병목 현상을 타개하기 위해, ISR 스레드 공간 내에서는 최소한의 버퍼 읽기/쓰기(최소 레지스터 교체)만 빠르게 수행하고 바로 종료(Top Half)한다.
대신, 데이터 처리가 완료되었음을 알리는 이벤트 플래그 신호(Hardware Semaphore 또는 이벤트 플래그)만을 던져놓음으로써, 센서 데이터 가공이나 알람 송출, 시동 처리 루틴 같은 무겁고 지연 한계가 넉넉한 예외 처리 루틴들은 모두 일반 스레드 우선순위 공간(Bottom Half)으로 밀어내어 분할 연산한다.
따라서, 전제 조건 불만족으로 인해 막대한 양의 에러 텍스트 스트리밍(STATUSTEXT)이 발생하여 commander 태스크가 수십 밀리초(ms) 지연되더라도 하단부의 하드웨어 타이밍(Rate Control)이 밀리는 임계 응답 지연 붕괴 사태를 근본적으로 억제한다.

4. Ardupilot과 대비되는 로직 격리성

Ardupilot의 경우 과거부터 단일 루프에 상당히 많은 예외 처리 구문들이 혼재되어 왔기 때문에, 극단적인 에러 연산 폭주 시 메인 루프 실행 주기가 미세하게 흔들리는 오버런(Over-run) 현상이 보고되기도 하였다.
반면 PX4는 RTOS의 근본적인 파편화된 인터럽트 쪼개기 철학을 적용하여, 예외 상황이 발생하여 commander 스레드의 타임아웃 카운터가 흔들리더라도 가장 최상위 우선순위를 가진 좁은 구역의 ISR 구동부와 비행 제어 태스크(Attitude Controller)는 완벽히 고립되어 정시성을 보호받는 매우 현대적인 임베디드 코어 설계 방식을 보여준다.

5. 요약

PX4의 시동 메커니즘은 전제 조건 불만족 시 무리하게 상태 전이를 시도하지 않고 즉각적이고 명시적으로 거부하는 안전 지향적 예외 처리 설계를 채택하고 있다. 나아가 과정 중 발생할 수 있는 긴급 하드웨어 ISR의 선점(Preemption) 상황 속에서도, 무거운 예외 처리를 유저 레벨 백그라운드 태스크로 분리(Bottom Half 기법)함으로써 비행 코어의 실시간 응답 지연(Latency) 한계를 엄격히 보장해 낸다.
