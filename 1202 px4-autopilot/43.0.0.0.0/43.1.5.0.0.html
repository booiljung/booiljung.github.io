43.1.5. 하이브리드 오토마타(Hybrid Automata) 기반의 이산 이벤트 제어 모드와 연속 시간 시동/해제 조건 간의 비선형적 위계 및 선점(Preemption) 권한 모델링

무인항공기(UAS) 제어 시스템은 이산적인 논리 상태(Discrete Logical State)와 기체의 연속적인 물리 동역학(Continuous Physical Dynamics)이 결합된 복잡한 시스템이다. PX4-Autopilot(기준 펌웨어 버전: v1.14.0)의 시동(Arming) 및 시동 해제(Disarming) 로직은 단순한 스위치 기능을 뛰어넘어, 하이브리드 오토마타(Hybrid Automata)라는 시스템 공학적 틀 안에서 동작한다. 이 절에서는 이러한 하이브리드 시스템 모델 내에서 연속적인 물리량의 변화가 어떻게 이산 이벤트를 트리거하는지, 그리고 비상 상황이나 사용자 제어 입력에 따른 선점(Preemption) 권한이 비선형적 위계로 어떻게 모델링되어 있는지 매우 상세히 분석한다.

1. 하이브리드 오토마타(Hybrid Automata)의 시스템 공학적 기초

하이브리드 오토마타는 이산 상태(Discrete State)와 그 상태에 종속된 연속 시간 동역학(Continuous-Time Dynamics)을 동시에 모델링하기 위한 수학적 체계이다. 무인항공기의 비행 제어기(Flight Controller)에서는 운영체제(NuttX RTOS) 기반의 유한 상태 기계(Finite State Machine, FSM)가 이산 이벤트를 관리하고, 센서 융합 및 제어 루프(Control Loop) 알고리즘이 기체의 물리 역학(위치, 속도, 자세 등)을 연속적으로 계산한다.

PX4의 커맨더(Commander) 아키텍처는 전형적인 하이브리드 오토마타의 구현체이다.

  • 이산 상태(Discrete States): INIT, PREFLIGHT, ARMED, DISARMED, FAILSAFE 등.
  • 연속 변수(Continuous Variables): 기체 고도 z, 스로틀(Throttle) 입력량 u_{th}, 배터리 전압 V_{bat}, 센서 혁신 오차(Sensor Innovation Error).
  • 전이 조건(Guard Conditions): 연속 변수가 특정 입계값(Threshold)을 교차할 때 발생하는 이산 이벤트. (예: V_{bat} < 14.2V 지속 시 Failsafe 모드로 전이).

수학적으로 하이브리드 오토마타 H는 다음과 같은 튜플로 정의된다.
H = (Q, X, f, Init, Dom, E, G, R)
여기서 Q는 이산 모드의 집합, X는 연속 상태 변수 벡터, G는 각 엣지에서의 전이 가드(Guard Condition)이며, PX4의 시동 시퀀스는 바로 이러한 가드의 집합을 통과해야만 상태 전이가 이루어지는 구조로 설계되었다.

2. 이산 이벤트 제어 모드와 연속 시간 평가 조건의 상호 작용

시동(Arming)과 시동 해제(Disarming) 결정은 단일 시점의 사용자 명령(MAVLink 또는 조종기 스위치) 이산 이벤트(Discrete Event)에 의해서만 결정되지 않는다. PX4 시스템은 이 명령을 수용하기 이전에 기체의 다이나믹스를 연속 시간(Continuous Time) 동안 평가한다.

2.1 연속 시간적 전제 조건(Pre-conditions) 평가

PX4에서 시동을 허용하는 이산 상태 전이 가드 조건은 다음과 같은 연속 시스템 모니터링 결과를 필요로 한다.

  1. EKF(Extended Kalman Filter) 안정성: 자이로스코프, 가속도계, GPS 데이터의 혁신(Innovation) 오차가 특정 시간 윈도우(Time Window) 동안 허용 한계선 내에 존재해야 한다.
  2. 랜딩 감지(Land Detection): 기체가 지면에 접지해 있는지 판별하기 위한 가속도 및 고도 변화율의 연속 시간 적분 결과.
  3. 사용자 제어 입력 스위핑: 조종기의 조이스틱이 시동 위치(Yaw Right & Throttle Down)에 1초 이상의 연속 시간 동안 머물러야 한다.

이 과정에서 연속 시간 조건이 이산 이벤트로 매핑되며, 커맨더(Commander) 스레드가 이 이벤트를 소비하여 모드 전이 명령을 커널 수준에서 인가하게 된다.

3. 선점(Preemption) 권한 계층 및 비선형적 위계(Non-linear Hierarchy) 모델링

UAS 제어에서는 수많은 주체(조종기, GCS, 내장된 Companion Computer, 기체 Failsafe 로직)가 명령을 하달한다. 이들 명령이 동시다발적으로 유입될 때 중재하는 우선순위가 필요하며, PX4는 이를 비선형적 위계 구조의 선점(Preemption) 모델로 해결한다.

3.1 엄격한 비선형적 위계(Strict Non-linear Hierarchy)

일반적으로 시스템은 ’센서 수집 단계 -> 상태 추정 단계 -> 제어 단계 -> 구동 단계’의 선형적(Linear) 데이터 파이프라인을 가진다. 그러나 시동 해제와 관련된 권한 모델은 이 파이프라인을 비선형적으로 파괴하거나 선점(Bypass/Preempt)하여 즉시 실행되어야 한다.

PX4의 모드 선점 모델링에서 가장 절대적인 우선순위를 가지는 것은 **상태 무결성 침해에 따른 비상 해제(Emergency Disarming)**이다.

stateDiagram-v2
    [*] --> PREFLIGHT : 부팅 완료
    PREFLIGHT --> DISARMED : 캘리브레이션 및 프시 체크 통과
    
    state "정상 구동 상태 (Nominal Operation)" as Nominal {
        DISARMED --> ARMED : 시동 인가 조건 (연속 시간 가드) 만족
        ARMED --> DISARMED : 착륙 후 정상 해제, 사용자 해제 명령
    }
    
    state "선점 및 비상 제어 권한 (Preemptive Control)" as Preemptive {
        ARMED --> FAILSAFE_RTL : 데이터 링크 단절 이벤트 (우선순위 높음)
        ARMED --> KILL_SWITCH : 강제 구동 정지 (우선순위 매우 높음, 선점)
        ARMED --> FLIGHT_TERMINATION : 치명적 에러 감지 (최상위 우선순위 선점)
    }

    FAILSAFE_RTL --> DISARMED : 착륙 후 강제 해제
    KILL_SWITCH --> DISARMED : 모터 즉각 정지 후 전이
    FLIGHT_TERMINATION --> DISARMED : PWM 신호 완전 차단 및 시스템 록(Lock)

아키텍처 상향에서 킬 스위치(Kill Switch) 및 비행 종료(Flight Termination) 이벤트는 일반적인 경로점(Waypoint) 추종이나 이착륙 모드 계산 루프를 완전히 선점(Preempt)하며, 즉각적으로 commander 태스크의 최우선 이벤트 인터럽트로 삽입된다.

3.2 Ardupilot과의 선점 모델 아키텍처 비교

  • PX4-Autopilot: 커맨더(Commander) 모듈이 중앙 집중형 락(Lock) 매니저로 작동한다. 어떠한 모듈이 비행을 요구하든 vehicle_status uORB 메시지와 커맨더의 엄격한 유한 상태 기계를 거쳐야 시동 상태를 변경할 수 있다. 중앙 집중형 하이브리드 오토마타 설계에 매우 부합하며 수학적 검증이 상대적으로 용이하다.
  • Ardupilot: 각 비행 모드 클래스(예: ModeRTL, ModeAuto) 내부에 시동 및 해제에 관한 일부 제어 로직과 Failsafe 트리거가 분산되어 구현된 경향이 있다. 객체 지향적 확장은 용이하나, 시스템 레벨에서의 선점 권한 흐름을 하나의 도표로 증명하거나 DO-178C 등급의 항공 소프트웨어 검증 시 모든 경로를 추적하는 데 있어 분산된 상태 관리로 인한 복잡성이 발생할 수 있다.

4. 소스 코드 이해: 선점(Preemption) 권한의 구현 논리

PX4 소스 코드 트리 내의 src/modules/commander/Commander.cpp와 관련 상태 머신 코드는 이 비선형 권한 모델을 구현한다.

// Commander 모듈 내의 선점 권한 및 상태 전이 검증 (의사 코드 및 구조)
if (vehicle_status.failsafe || hit_kill_switch) {
    // 1. 최상위 선점(Preemption): 킬 스위치 또는 치명적 Failsafe 발생 여부 확인
    if (hit_kill_switch) {
        // 기존의 모든 이벤트 큐 및 연속 제어 모드를 무시하고 즉시 시동 해제 메인 로직 트리거
        transition_to_disarmed();
        publish_vehicle_command_ack(MAV_CMD_DO_FLIGHTTERMINATION, MAV_RESULT_ACCEPTED);
    }
} else {
    // 2. 일반 위계: 조종기나 GCS 명령 처리 
    if (handle_command(&cmd)) {
        if (cmd.command == MAV_CMD_COMPONENT_ARM_DISARM) {
            // 연속 시간 조건 검증 (센서, GPS Lock, 위치 등)
            if (preflight_check_passed() && continuous_conditions_met()) {
                execute_arm_disarm_transition(cmd.param1);
            } else {
                reject_arming(); // 조건 미달 시 전이 거부
            }
        }
    }
}

이 코드는 “하이브리드 오토마타에서의 가드 조건 검증“을 충실히 반영하고 있다. 안전을 위협하는 이산 이벤트(예: 킬 스위치 트리거)는 if 구조의 최상단에서 가장 높은 우선순위로 평가되어 시스템 동역학의 흐름을 강제로 차단하고 PWM 출력을 비활성화하는 데까지 선행한다.

5. MAVLink 및 QGroundControl 연동 시의 권한 충돌 해소

QGroundControl(v4.3.0 이상 기준)과 연결된 다중 GCS 환경 및 ROS2(uXRCE-DDS) 오프보드(Offboard) 노드가 동시에 연결되었을 때, 이 위계 모델은 네트워크 레벨의 권한 충돌도 중재해야 한다.

예를 들어, ROS2 노드가 임무 수행을 위해 VehicleCommand 퍼블리싱을 통해 지속적으로 활성화(Arm)를 유지하려 시도하는 와중에 조종자가 지상 텔레메트리를 통해 RC 킬 스위치를 입력하면 어떻게 될까. 하이브리드 오토마타 모델은 “하드웨어에 물리적으로 매핑된 수동 선점 이벤트“를 “소프트웨어로 입력되는 MAVLink/uORB 이산 이벤트“보다 고차원적 선점 권한으로 인지하도록 설계되었다.

결과적으로 PX4 커맨더는 오프보드 제어기의 시동 유지 명령을 억제(Reject)하고 기체를 강제로 해제 모드로 전환시킨 후, MAV_CMD_ACK 및 상태 텍스트(STATUSTEXT)를 통하여 GCS로 “Kill switch engaged, bypassing offboard commander” 라는 취지의 이벤트를 보고하게 된다. 이러한 모델링 설계는 자율 에이전트 비행 프로그래밍과 장애물 회피 제어 기술을 다루는 ROS2 개발자들이 오프보드 연결 중단 시의 행동 요령(Failsafe Timeout)을 필히 구현해야 하는 기저 원리가 된다.

이처럼 PX4의 시동 및 해제 시스템은 복잡한 연속 시간 기체 동역학의 제어기 상태를 다루면서도 이산적 이벤트 간의 수학적으로 명백한 선점/위계 모델을 갖추고 있어, 드론 시스템의 비행 안정성 검증과 자율 에이전트 설계에서 반드시 심도 있게 이해해야 하는 핵심 아키텍처이다.
