43.2 MAVLink 프로토콜 기반 시동 제어의 직렬화(Serialization) 및 패킷 아키텍처

무인항공기(UAS) 제어에서 지상 관제 시스템(GCS)과 비행 제어기(PX4-Autopilot) 간의 통신은 MAVLink 프로토콜에 의해 광범위하게 이루어진다. 그 중에서도 특히 시동(Arming) 및 시동 해제(Disarming) 제어는 기체의 안전과 직결된 치명적인명령이므로, 시스템 간 데이터의 직렬화(Serialization) 검증 및 패킷 송수신의 아키텍처적 신뢰성이 철저하게 요구된다. 이 절에서는 MAVLink 프로토콜 구조 위에서 시동 제어 명령이 어떻게 바이트 시퀀스(Byte Sequence)로 직렬화되고 네트워크 망을 거쳐 PX4 내부 시스템으로 전달되는지 아키텍처 관점에서 상세히 분석한다.

1. MAVLink 시동 제어명령 패킷 아키텍처 (Packet Architecture)

MAVLink 프로토콜은 원격 측정(Telemetry) 데이터 송수신 및 지휘 통제(Command and Control)를 위해 고안된 경량 통신 프로토콜이다. 시스템 간 상태를 제어하기 위해 MAVLink 2.0 버전을 기준으로 패킷 아키텍처가 어떻게 구성되는지 이해하는 것은 매우 중요하다.

시동 및 시동 해제 명령은 주로 COMMAND_LONG (Message ID: 76) 또는 COMMAND_INT (Message ID: 75) 메시지 포맷을 페이로드(Payload)로 사용하여 전송되며, 이 페이로드 내부의 command 필드에 MAV_CMD_COMPONENT_ARM_DISARM (Command ID: 400) 이라는 매직 넘버(Magic Number)를 담아 전송한다.

1.1 MAVLink 2.0 프레임(Frame) 구조

MAVLink 2.0 패킷은 직렬화를 거쳐 아래와 같은 바이트 스트림으로 변환된다.

packet-beta
title MAVLink v2.0 Packet Structure for Arming Command
0-7: "Magic (0xFD)"
8-15: "Len (Payload)"
16-23: "Incompat Flags"
24-31: "Compat Flags"
32-39: "Seq"
40-47: "Sys ID"
48-55: "Comp ID"
56-79: "Msg ID (76)"
80-343: "Payload (COMMAND_LONG)"
344-359: "Checksum"
360-463: "Signature (Optional)"

여기서 각 필드의 아키텍처적 역할은 다음과 같다.

  • SYS ID (System ID) 및 COMP ID (Component ID): 명령의 목적지 기체(예: Sys ID = 1)와 컴포넌트(예: Autopilot = 1)를 식별하는 라우팅 키(Routing Key) 역할을 한다.
  • MSG ID (Message ID): 24비트로 확장된 MAVLink 2.0의 메시지 식별자로, COMMAND_LONG의 경우 76 값을 가진다.
  • Payload: 실제 시동 명령을 담고 있는 부분이다.

2. 직렬화(Serialization) 및 역직렬화(Deserialization) 메커니즘

직렬화는 메모리 상에 존재하는 C/C++ 구조체(Struct) 객체를 통신 채널(UART, UDP 등)을 통해 전송할 수 있도록 일련의 연속된 바이트 배열로 변환하는 과정이다.

QGroundControl(GCS)에서 시동 스위치를 슬라이드할 때 일어나는 직렬화 파이프라인은 다음과 같다.

  1. 사용자 인터페이스(UI): QGroundControl의 QML 프론트엔드에서 Arm 스위치 트리거를 감지한다.
  2. 차량 매니저(Vehicle Manager): C++ 백엔드의 Vehicle 클래스가 MAV_CMD_COMPONENT_ARM_DISARM 명령 전역 변수와 파라미터를 세팅하여 mavlink_msg_command_long_pack() 함수를 호출한다.
  3. 바이트 변환(Byte Transformation): MAVLink 라이브러리의 C 헤더 내에 매크로로 정의된 직렬화 함수가 구조체 필드를 리틀 엔디안(Little-Endian) 바이트 스트림으로 패킹(Packing)한다.
  4. CRC 연산(Cyclic Redundancy Check): 패킷의 무결성을 보장하기 위해 Message ID마다 할당된 CRC EXTRA 바이트를 포함하여 체크섬(Checksum)을 생성하고 패킷 트레일러(Trailer)에 추가한다.

2.1 페이로드(Payload) 상세 매핑 트리 트리

COMMAND_LONG 페이로드는 7개의 단정밀도 부동소수점(Float) 32비트 파라미터와 Command 필드 등으로 구성된다.

  • Param 1: 1.0 (Arm) 또는 0.0 (Disarm)을 표기하는 Float 값
  • Param 2: 21196.0 (강제 해제 시 사용되는 Magic Number) 비상 시퀀스의 경우
  • Param 3 ~ Param 7: 예약됨(Reserved)

3. PX4와 Ardupilot의 MAVLink 수신 아키텍처 및 관제 차이점

수신된 MAVLink 메시지는 Flight Controller 내부에서 역직렬화되어 의미를 가진 데이터로 환원된다. 여기서 PX4-Autopilot과 Ardupilot 간 아키텍처적 차이가 두드러진다.

3.1 PX4-Autopilot의 수신 파이프라인

PX4(펌웨어 버전 v1.14.0 기준)는 NuttX 통합 미들웨어인 uORB(Micro Object Request Broker) 버스를 활용하여 모듈 간 결합도를 심하게 낮추었다(Decoupled).

  1. mavlink_receiver 스레드가 직렬 포트나 UDP에서 바이트를 수신하여 역직렬화를 수행한다.
  2. 수신된 메시지가 COMMAND_LONG인 경우, mavlink_receiver는 이를 즉시 처리하지 않고 vehicle_command라는 uORB 토픽(Topic)으로 발행(Publish)한다.
  3. PX4의 중앙 상태 머신 역할을 하는 commander 모듈이 vehicle_command를 구독(Subscribe)하고 있다가 이벤트를 획득하여 시동 조건을 검증하고 실행한다.
  4. 처리가 완료되면 commander는 vehicle_command_ack 토픽을 발행하고, 이를 다시 mavlink 모듈이 역구독하여 GCS로 COMMAND_ACK (Message ID: 77) 메시지를 전송(Closed-Loop)한다.

3.2 Ardupilot의 수신 파이프라인

Ardupilot은 GCS_MAVLink 클래스를 통해 메시지를 처리하며, uORB와 같은 강력한 비동기 발행/구독 미들웨어를 메인 데이터 버스로 사용하지 않고(AP_HAL의 내부 메시지 큐 등 제외) 상대적으로 직접 함수(Method) 호출이나 폴링(Polling) 루프 내에서 상태를 변경하는 객체 지향적 접근이 더 깊게 결합되어 있다.
시동 명령 수신 시, 해당 GCS 연결을 관리하는 스레드 구문 안에서 AP_Arming 클래스 인스턴스의 arm() 메소드를 호출하여 로직을 동기 혹은 세미 비동기적으로 분기한다.

결과적으로 PX4의 구조는 모듈이 메시지 구조체(uORB)를 매개로만 통신하는 분산형 실시간 시스템 아키텍처를 취하여 이종 모듈(예: ROS2 노드) 확장성이 매우 뛰어나지만 지연 시간(Latency) 추적이 다소 복잡해진다. Ardupilot의 구조는 코어 제어 로직과 GCS 파서가 타이트하게 결합되어 있어 직관적 디버깅이 용이한 차이를 보인다.

4. 자율 에이전트(ROS2) 연동 시의 직렬화 패러다임 변화

전통적 GCS(QGroundControl)와의 MAVLink 스택과는 별개로, 최근 PX4는 실시간 분산 퍼블리싱/서브스크라이빙 체계인 ROS2 환경과의 통합을 위해 Micro XRCE-DDS 프로토콜을 사용한다.

이 환경에서는 MAVLink 라이브러리의 C-Struct 매크로 기반 직렬화 대신, OMG(Object Management Group) 표준에 따른 CDR (Common Data Representation) 직렬화 방식을 따른다. 즉, ROS2 노드에서 출발한 VehicleCommand 메시지는 uXRCE-DDS 에이전트 브릿지(Agent Bridge)를 거치면서 CDR로 직렬화되고, PX4 내부의 uXRCE-DDS 클라이언트 데몬이 이를 역직렬화하여 직접 uORB vehicle_command 토픽으로 삽입한다.

결과적으로 PX4 내부의 커맨더(commander) 관점에서는 MAVLink 기반 QGC에서 진입하든, ROS2 CDR 기반 자율 비행 노드에서 진입하든 동일한 vehicle_command 이벤트를 수신하게 되므로, 통신 채널에 구애받지 않는 다원화된 제어권 위계 통제가 가능해진다.

이와 같은 MAVLink 패킷 아키텍처 체계와 내부 버스(uORB) 연계 구조를 명확히 이해하는 것은, 독자적인 GCS를 개발하거나 대규모 자율 군집 비행(Swarm) 환경에서 발생하는 제어권 충돌과 명령 손실(Drop) 문제를 트러블슈팅하는 데 필수적인 공학적 토대가 된다.
