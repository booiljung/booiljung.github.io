43.4.5 외부 컴패니언 컴퓨터(ROS2/uXRCE-DDS) 연동 시동 승인 대기열(Queue) 관리 및 응답 지연 타임아웃 롤백(Rollback) 로직

1. 개요

최신 PX4-Autopilot(펌웨어 버전 1.14 이상 기준)에서는 외부 컴패니언 컴퓨터(Companion Computer)와의 연동을 위해 Micro XRCE-DDS(uXRCE-DDS) 미들웨어를 기본적으로 채택하고 있다. ROS 2(Robot Operating System 2) 환경에서 구동되는 외부 제어 알고리즘은 uXRCE-DDS 브릿지를 통해 PX4 내부의 uORB 토픽(Topic)에 직접 접근하여 기체의 시동(Arming) 및 비행 모드 변경을 요청할 수 있다.

이러한 분산 제어 환경에서는 지상 관제 시스템(GCS), 조종기(RC), 그리고 컴패니언 컴퓨터 등 다중 소스에서 동시에 시동 요청이 발생할 가능성이 존재한다. 따라서 PX4의 커맨더(Commander) 모듈은 다중 시동 승인 요청을 안전하게 직렬화하기 위한 대기열(Queue) 관리 시스템과, 통신 지연이나 시스템 부하로 인해 시동 전제 조건(Pre-condition) 검증이 지연될 경우 이를 안전 상태로 되돌리는 타임아웃(Timeout) 및 롤백(Rollback) 로직을 필수적으로 포함하고 있다. 이 장에서는 ROS 2 환경에서 기체 시동을 제어할 때 발생하는 응답 지연의 처리 메커니즘을 상세히 다룬다.

2. 시동 승인 대기열(Queue) 관리 메커니즘

다양한 인터페이스(MAVLink, RC, uXRCE-DDS)를 통해 전달되는 시동 요청은 최종적으로 vehicle_command uORB 토픽 내의 VEHICLE_CMD_COMPONENT_ARM_DISARM 명령으로 변환된다. 커맨더 모듈은 이 인터페이스로부터 유입되는 명령을 안전하게 처리하기 위해 내부적인 명령 대기열(Command Queue)을 운용한다.

  1. 명령의 직렬화(Serialization): ROS 2의 발행자(Publisher)가 시동 명령을 비동기적으로 전송하면, uXRCE-DDS 클라이언트는 이를 PX4 내부 구조체로 디코딩하여 uORB에 발행한다. 커맨더 모듈은 폴링(Polling) 루프를 통해 vehicle_command 토픽의 업데이트를 감지하고, 우선순위 및 타임스탬프를 기반으로 명령을 링 버퍼(Ring Buffer) 형태의 대기열에 적재한다.
  2. 상호 배제 및 경쟁 상태 제어(Mutex and Race Condition Control): 동시에 여러 시동 명령이 수신되더라도, 유한 상태 머신(Finite State Machine)의 상태 전이는 단일 스레드 컨텍스트 내에서 순차적으로 처리된다. 이를 통해 이중 시동 혹은 시동 중 해제와 같은 논리적 충돌(Logical Collision)을 방지한다.
  3. 중복 명령 드롭(Command Drop): 이미 시동 상태로의 전이(Transition)가 진행 중인 상황에서 추가적인 시동 명령이 대기열에 진입하면, 시스템은 상태 확인을 거쳐 후속 명령을 ‘이미 완료됨(Already Executed)’ 혹은 ’처리 중(In Progress)’으로 간주하여 안전하게 폐기(Drop)한다.

3. 응답 지연 타임아웃(Timeout) 판별 알고리즘

컴패니언 컴퓨터 기반의 자율 비행에서는 통신 대역폭 한계나 ROS 2 운영체제 스케줄링 지연으로 인해 네트워크 패킷의 전파 지연(Propagation Delay)이 발생할 수 있다. PX4-Autopilot은 비결정론적(Nondeterministic) 통신 환경에서 기체의 안전을 보장하기 위해 엄격한 타임아웃 규칙을 적용한다.

  • 절대 시간 기반 만료(Absolute Time Expiration): 모든 제어 명령에는 발생 시점을 기록한 고정밀 타임스탬프(timestamp)가 포함된다. 커맨더 모듈은 hrt_absolute_time() 함수를 통해 현재 시스템 해상도 시간(High Resolution Time)을 측정하고, 수신된 토픽의 타임스탬프와 비교하여 임계 시간(일반적으로 0.5초 ~ 1.0초)을 초과한 명령은 만료(Expired)된 것으로 간주하여 거부(Reject)한다.
  • 전제 조건 검증 지연(Pre-condition Verification Delay): 시동 명령이 유효하더라도, 센서 캘리브레이션 불일치, EKF(Extended Kalman Filter) 위치 추정 오차 등 비행 전 점검(Pre-flight Check) 로직을 통과하는 데 과도한 시간이 소요될 수 있다. 이 경우 상태 머신은 외부 응답 대기 상태(Waiting for Response)로 잠시 머무르며, 일정 시간 내에 조건이 충족되지 않으면 즉시 타임아웃 예외(Timeout Exception)를 발생시킨다.

4. 시동 상태 롤백(Rollback) 및 예외 처리 아키텍처

타임아웃이 발생하거나 전제 조건 불만족으로 인해 시동 절차가 중단될 경우, 시스템은 불안정한 중간 상태에 머무르지 않고 즉각적으로 안정한 시동 해제(Disarmed) 상태로 복구되어야 한다. 이를 롤백(Rollback) 로직이라 한다.

    Commander->>Commander: 타임스탬프 유효성 및 타임아웃 검사

        Commander-->>ROS2: 상태 롤백 (ACK: FAILED)

        Commander->>Commander: 비행 전 점검(Pre-flight Check) 수행

            Commander-->>ROS2: 예외 발생 및 롤백 (Rollback to Disarmed)
        else 조건 충족
            Commander->>Actuator: actuator_armed 토픽 발행
            Actuator-->>Commander: 구동기 활성화 확인
            Commander-->>ROS2: 시동 완료 (ACK: ACCEPTED)
        end
    end
    deactivate Commander
  1. 상태 원상 복구(State Reversion): 커맨더 모듈은 시동 과정에서 변경된 내부 변수나 플래그를 이전 상태로 돌려놓는다. ArmingStateMachine 인스턴스는 실패 코드(Failure Code)를 생성하고, 시스템의 메인 상태를 명시적으로 ARMING_STATE_STANDBY로 되돌린다.
  2. 장치 잠금 해제(Device Unlock): 시동 시도 중에는 일시적으로 센서 설정이나 파라미터 변경이 잠금(Lock)되는데, 롤백 과정에서 이러한 잠금을 즉시 해제하여 지상 관제 시스템(QGroundControl 등)에서 사용자가 문제를 수정할 수 있도록 리소스를 반환한다.
  3. 실패 응답(NACK) 피드백: ROS 2 시스템은 vehicle_command_ack 토픽을 구독(Subscribe)하여 자신이 전송한 제어 명령의 성공 여부를 비동기적으로 확인한다. PX4는 롤백 수행 직후 즉각적으로 명령 실패 이유를 담은 NACK(Negative Acknowledgement) 메시지를 발행하여, 자율 에이전트가 예외 상황을 인지하고 우회 경로 탐색(Obstacle Avoidance / Mission Replanning)이나 비상 대기 모드로 전환할 수 있도록 한다.

5. 결론 및 실무적 고려사항

로봇 프레임워크와 비행 제어기 간의 결합도가 높아지는 현대의 시스템 아키텍처에서는, 비동기 통신으로 인한 시변(Time-variant) 지연 요소를 안정적으로 제어하는 것이 핵심 공학 과제이다. PX4-Autopilot은 uXRCE-DDS 브릿지를 통과하는 모든 외부 제어 명령에 대해 큐 설계와 타임아웃 검증, 명시적인 롤백 패턴을 적용하여 결정론적(Deterministic) 강건성을 유지한다. 개발자는 ROS 2 애플리케이션을 설계할 때 이러한 PX4의 내부 타임아웃 임계치와 ACK/NACK 피드백 루프를 정확히 이해하고, 네트워크 지연을 보상할 수 있는 논블로킹(Non-blocking) 제어 흐름을 구현해야 한다.
