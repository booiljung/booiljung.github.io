43.2.4. 비동기 메쉬 네트워크 토폴로지 내 시스템 식별자(System ID) 및 컴포넌트 식별자(Component ID) 기반 라우팅 경로 탐색 트리 및 타겟팅 일치성 검증

지상 관제 시스템(GCS), 비행 제어기(Flight Controller), 보조 컴퓨터(Companion Computer), 스마트 카메라 페이로드 등 다양한 노드들이 상호작용하는 현대의 무인항공기(UAV) 운용 환경은 단일 1:1 통신을 넘어 다중 노드 기반의 비동기 메쉬 네트워크(Asynchronous Mesh Network) 토폴로지 형태를 띤다. 본 절에서는 MAVLink 프로토콜의 라우팅(Routing) 코어인 시스템 식별자(System ID)와 컴포넌트 식별자(Component ID) 체계를 분석하고, PX4-Autopilot 내에서 MAVLink 메시지가 어떻게 타겟팅(Targeting)되고 경로 탐색 트리(Discovery Tree)를 형성하는지 살펴본다.

1. MAVLink 식별자 체계의 구조적 특징

MAVLink 네트워크 상의 모든 패킷 헤더에는 발신자(Sender)와 수신 대상(Target)을 명시하기 위한 식별자가 포함된다. 이를 통해 다수의 드론(군집 비행 등)과 GCS가 혼재된 통신 채널 레이어에서도 혼선 없이 메시지가 전달될 수 있다.

  • 시스템 식별자(System ID): 개별 드론 기체 자체 또는 독립된 GCS와 같은 최상위(Top-level) 시스템 단위에 부여되는 고유 번호이다(예: GCS는 통상 255, 첫 번째 기체는 1 등). 1~255 사이의 할당 범위를 가진다.
  • 컴포넌트 식별자(Component ID): 하나의 시스템(System ID) 내부를 구성하는 더 작은 단위의 모듈들을 논리적으로 분리하는 식별자이다. 예를 들어 System ID = 1인 드론 내부에는 비행 제어기(AUTOPILOT1 = 1), 짐벌(GIMBAL = 154), 카메라(CAMERA = 100) 등의 컴포넌트 식별자가 동적으로 할당되어 통신한다.

2. 라우팅 경로 탐색 트리(Routing Discovery Tree) 구성

비동기 네트워크 특성상, 각 노드는 네트워크망에 진입하기 전까지 어떠한 시스템이나 컴포넌트가 연결되어 있는지 전혀 알지 못한다. 따라서 라우팅 경로는 지속적으로 브로드캐스팅(Broadcasting)되는 HEARTBEAT 메시지를 기반으로 동적인 탐색 트리(Discovery Tree)를 구축하여 완성된다.

  1. 동적 테이블 생성: PX4의 mavlink 모듈 및 QGroundControl 통신 매니저는 망 내에서 발생하는 모든 HEARTBEAT 메시지를 수집하여 [System ID, Component ID] -> 인터페이스(포트, IP 등) 형태의 동적 라우팅 테이블(Routing Table)을 메모리에 구축한다.
  2. 경로 탐색 트리의 생존 시간(TTL): 일정 시간(타임아웃, Timeout) 동안 심박(Heartbeat) 신호가 갱신되지 않는 노드는 가비지 컬렉터(Garbage Collector)에 의해 라우팅 테이블 트리에서 즉각 삭제된다. 이로써 단절된 컴포넌트로의 불필요한 메시지 포워딩 낭비를 방지한다.

3. 타겟팅 일치성 검증(Targeting Validation) 알고리즘

수신된 MAVLink 메시지 중 제어 명령(예: COMMAND_LONG) 패킷의 페이로드 내부에는 반드시 target_system과 target_component 필드가 존재한다. PX4-Autopilot 펌웨어의 MAVLink 인터페이스 루틴에서는 이 타겟 정보의 일치성을 검증하는 엄격한 알고리즘을 수행한다.

// 수신된 MAVLink 명령 메시지의 타겟팅 검증 로직 예시 (C++)
bool MavlinkReceiver::evaluate_target_ok(uint8_t target_sys, uint8_t target_comp)
{
    // 브로드캐스트 시스템 ID(0)이거나 자신의 시스템 ID와 동일해야 함
    if (target_sys != 0 && target_sys != mavlink_system.sysid) {
        return false;
    }

    // 브로드캐스트 컴포넌트 ID(0)이거나 자신의 컴포넌트 ID와 동일해야 함
    // (보통 px4_instance의 컴포넌트는 MAV_COMP_ID_AUTOPILOT1)
    if (target_comp != 0 && target_comp != mavlink_system.compid) {
        // 컴포넌트 ID가 다를 경우 패킷을 수용하지 않고 다른 노드로 포워딩(Forwarding)
        return false; 
    }

    return true; // 타겟팅 일치
}

만약 target_system은 자신과 일치하지만 target_comp가 자신이 아닌 짐벌 컴포넌트(예: 154)를 가리킨다면, 비행 제어기(PX4)는 해당 메시지를 처리하지 않고 내부의 시리얼(Serial) 버스로 재전송(Forwarding)함으로써 적절한 대상에게 전달되도록 메쉬 라우터(Mesh Router) 역할을 수행한다.

5. 결론 및 Ardupilot과의 차이점

Ardupilot 통신 계층도 동일한 MAVLink의 라우팅 정책을 준수하지만, PX4-Autopilot의 경우 uORB 토픽 기반의 마이크로서비스(Microservices) 아키텍처에 의해 노드 간 명령 포워딩이 모듈 단위에서 논리적으로 보다 깊게 캡슐화(Encapsulation)되어 있다는 특징이 있다. 이와 같은 시스템/컴포넌트 식별 기반의 비동기 메쉬 네트워크 토폴로지는 복잡성 높은 다중 에이전트(Multi-agent) 시스템이나 군집 에이전트(Swarm) 환경에서 기체 간(Vehicle-to-Vehicle) 통신 확장을 위한 결정론적인 인프라스트럭처를 제공한다.
