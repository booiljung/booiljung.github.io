43.1.2 하드웨어 인터럽트(Hardware Interrupt) 구동 안전 스위치와 운영체제 커널 수준 소프트웨어 시동 논리의 이종(Heterogeneous) 결합 아키텍처

무인항공기 시스템(UAS)의 시동 과정은 소프트웨어의 논리적 판단만으로 이루어질 경우, 예상치 못한 프로세스 충돌(Crash)이나 버그로 인해 치명적인 오작동을 유발할 수 있다. 이를 방지하기 위해 PX4-Autopilot은 **하드웨어 인터럽트(Hardware Interrupt) 기반의 물리적 안전 스위치(Safety Switch)**와 운영체제(NuttX RTOS) 커널 수준의 소프트웨어 시동 논리를 결합한 이종(Heterogeneous) 아키텍처를 채택하고 있다. 본 장에서는 이러한 하드웨어-소프트웨어 결합 구조가 어떻게 시스템의 무결성을 보장하는지 심층적으로 분석한다.

1. 물리적 안전 스위치(Safety Switch)의 하드웨어적 역할

Pixhawk 시리즈와 같은 호환 하드웨어에는 보통 GPS 모듈 내장형 또는 기체에 독립적으로 부착 가능한 LED 내장형 안전 스위치가 제공된다. 이 스위치는 시스템 전원이 인가되었을 때(부팅 직후) 기본적으로 모터 및 액추에이터로 향하는 신호(PWM/DShot) 출력을 물리적으로 차단(Lock)하는 역할을 수행한다.

조종자가 이 스위치를 약 1초 이상 길게 누르면, 핀의 전압 변화가 비행 제어기(MCU)의 특정 GPIO 외부 인터럽트(External Interrupt) 핀으로 전달된다.

1.1 하드웨어 인터럽트 라우팅(Routing) 메커니즘

  • FMU(Flight Management Unit)와 IO 컴패니언 마이컴 분리 (FMUv2 ~ FMUv4 아키텍처): 과거의 Pixhawk 아키텍처에서는 강력한 코어 CPU(FMU) 외에 입출력을 전담하는 보조 마이크로컨트롤러(PX4IO)가 존재했다. 안전 스위치는 이 PX4IO 프로세서의 하드웨어 인터럽트를 트리거하여, 메인 OS(NuttX)가 완전히 다운되더라도 펄스 폭 변조(PWM) 출력을 독립적으로 차단할 수 있도록 설계되었다.
  • 단일 칩(FMUv5, FMUv6) 아키텍처: 최신 비행 제어기에서는 단일 칩에서 모든 것을 처리하지만, 안전 스위치 핀은 타이머(Timer) 자원의 하드웨어 활성화 기능(Hardware Enable/Disable)과 저수준으로 연결되어 있어, 메인 소프트웨어 스레드의 부하와 무관하게 즉각적으로 출력을 차단하거나 허용할 수 있다.

2. 운영체제(NuttX RTOS) 커널 수준 소프트웨어 시동 논리

하드웨어가 준비되었다 하더라도 모터가 바로 회전하는 것은 아니다. 소프트웨어 계층에서는 commander 태스크가 비행 전 점검(Pre-flight Check)을 수행한다.

  1. 상태 감시(State Monitoring): commander는 uORB 메시지 버스를 통해 safety 토픽을 구독(Subscribe)한다. 안전 스위치가 눌려 하드웨어 잠금이 해제되면, 이를 감지하는 드라이버가 타이머 인터럽트 서비스 루틴(ISR) 내에서 시스템 상태 플래그를 변경하고 uORB 토픽을 발행(Publish)한다.
  2. 소프트웨어 시동(Arming) 허가: 하드웨어 측의 준비 신호(safety_off)가 도달했을 때 비로소 조종기(RC) 명령이나 MAVLink의 COMMAND_LONG(명령 코드: 400, MAV_CMD_COMPONENT_ARM_DISARM)을 통한 소프트웨어적 시동(Arm)이 의미를 갖게 된다.
  3. 조건부 진입 로직: 만약 하드웨어 안전 스위치가 눌려있지 않은 상태(safety_on)에서 QGroundControl이나 ROS2를 통해 시동(Arm) 명령이 들어오면, 커맨더는 “Safety Switch Locked“라는 예외를 발생시키고 시동을 강제로 거부(Decline)한다.

3. 이종(Heterogeneous) 결합 아키텍처의 상태 머신 모델링

이러한 두 가지 서로 다른 레벨(Hardware & Software)의 안전 메커니즘은 기계적 스위치의 ’결정론성(Determinism)’과 커맨더 스레드의 ’인지적(Cognitive) 유연성’을 결합한다.

stateDiagram-v2
    state "하드웨어 레벨 (Hardware Level)" as HW {
        [*] --> HW_Locked : 전원 인가 (Power On)
        HW_Locked --> HW_Unlocked : 안전 스위치 1초 이상 누름
        HW_Unlocked --> HW_Locked : 안전 스위치 짧게 누름
    }
    
    state "소프트웨어 레벨 (Software Level)" as SW {
        [*] --> Disarmed
        Disarmed --> Arm_Ready : Pre-flight Check 통과
        Arm_Ready --> Armed : 시동 명령 인가 & HW_Unlocked 상태 확인
        Armed --> Disarmed : 시동 해제 명령 / 비상 종료
    }
    
    HW_Unlocked --> Arm_Ready : uORB 제어권 인계 (safety topic)

위의 상태 다이어그램에서 알 수 있듯이, 모터의 회전을 결정하는 최종 관문논리(Final Gating Logic)는 (HW_Unlocked == true) AND (SW_Armed == true) 라는 교집합을 형성한다.

4. 커널 패닉(Kernel Panic) 및 비상시의 극단적 조치

이 이종 결합 구조의 가장 큰 철학적 장점은 메인 MCU의 복잡한 소프트웨어(예: ekf2 추정기나 경로 계획 navigator)가 메모리 누수(Memory Leak) 등으로 인해 커널 패닉(Kernel Panic) 상태에 빠지는 극한 상황에 구현된다.

운영체제가 먹통이 되는 경우에도 독립적인 오실레이터(Oscillator)와 외부 인터럽트에 기반한 하드웨어 안전 로직은 살아남는다. 조종자가 신속하게 안전 스위치를 다시 누르거나, Watchdog 타이머가 칩을 리셋함과 동시에 디폴트(Default) 잠금 상태로 돌아가 출력을 0으로 만든다. 이는 거대한 크기와 무게를 지닌 자율 에이전트 다기능 드론(예: 방제용 대형 멀티로터 시스템)의 안전성을 확보하는 최후의 보루(Failsafe) 역할을 한다. Ardupilot 역시 유사한 안전 스위치 기능을 지원하지만, PX4는 NuttX RTOS의 마이크로커널(Microkernel) 구조적 이점을 살려 태스크 간 상호 간섭 없이 하드웨어 레벨 인터럽트를 최상위 우선순위로 매핑하여 처리하는 특성을 띤다.

5. 결론

하드웨어 인터럽트를 지렛대로 삼는 안전 스위치와 고수준의 운영체제 시동 논리를 교차(Cross-check)시킨 설계는 현대 PX4-Autopilot 아키텍처의 성숙도를 보여준다. 연구자나 체계 개발자는 QGroundControl의 커스텀 빌드 시 무리하게 파라미터(CBRK_IO_SAFETY)를 변경하여 하드웨어 스위치 기능을 비활성화하는 것을 지양해야 하며, ROS2를 이용한 Offboard 제어 상황에서도 항상 하드웨어 인터럽트로 직결되는 수동 개입 구조를 아키텍처 관점에서 존중하며 체계를 설계해야 한다.
