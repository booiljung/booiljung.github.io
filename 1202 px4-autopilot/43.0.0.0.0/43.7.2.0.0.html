43.7.2. 소프트웨어 인 더 루프(SITL) 및 하드웨어 인 더 루프(HITL) 테스트 벤치를 위한 시스템 검증 우회(Bypass) 매개변수화(Parameterization) 기법

무인항공기(UAV)의 제어 알고리즘 및 임무 프로파일을 실제 비행 전에 검증하기 위해 소프트웨어 인 더 루프(Software-In-The-Loop, SITL) 및 하드웨어 인 더 루프(Hardware-In-The-Loop, HITL) 시뮬레이션 환경이 널리 활용된다. 그러나 시뮬레이션 환경 또는 실내 테스트 벤치에서는 실제 GPS 신호가 잡히지 않거나, 배터리 전압/전류 센서의 물리적 피드백이 누락되는 등 센서 환경이 불완전한 특징이 있다. 이러한 제약 조건 하에서 PX4-Autopilot의 엄격한 비행 전 점검(Pre-flight Check) 논리를 그대로 적용할 경우, 시스템은 시동 거부(Arming Denial) 상태에 빠지게 된다.

본 절에서는 이러한 문제를 해결하기 위해 PX4-Autopilot 아키텍처 내부에 설계된 시스템 검증 우회(Bypass) 매개변수화(Parameterization) 기법을 심층적으로 분석한다.

1. 아키텍처 이해: Commander 모듈과 검증 우회 메커니즘

PX4-Autopilot에서 기체의 시동(Arming) 및 상태 전이를 총괄하는 핵심 모듈은 commander이다. commander 모듈은 내부적으로 PreflightCheck.cpp를 호출하여 센서 무결성, 추정기(Estimator) 상태, 데이터 링크 품질 등을 엄격하게 검사한다.

시뮬레이션(SITL/HITL) 환경에서는 특정 하드웨어 모듈이 존재하지 않거나 지속적인 오차를 발생시키므로, 검증 로직에 논리적 OR 조건을 추가하여 특정 오류를 무시(Bypass)하도록 아키텍처가 설계되어 있다. 이 우회 로직은 정적으로 컴파일되는 것이 아니라, 런타임에 동적으로 변경 가능한 파라미터(Parameter) 시스템을 통해 구성된다.

1.1 Ardupilot 대비 PX4-Autopilot 매개변수화 철학의 차이

  • Ardupilot: 시스템 점검 우회를 위해 주로 ARMING_CHECK 매개변수 하나를 비트마스크(Bitmask) 형태로 활용한다. 각 비트(Bit)가 특정 검증 항목(예: Barometer, Compass, GPS 등)을 켜고 끄는 역할을 수행한다. 또한 SIM_* 접두사가 붙은 시뮬레이션 전용 파라미터 집합이 분리되어 있다.
  • PX4-Autopilot: 보다 명시적이고 분산된 명명 규칙을 채택하고 있다. 안전과 직결된 검증 우회 파라미터는 COM_(Commander), 하드웨어 존재 여부는 SYS_HAS_, 그리고 치명적인 안전 보루를 강제로 해제하는 회로 차단기(Circuit Breaker)는 CBRK_ 접두사를 사용하여 기능적 경계를 명확히 구분한다.

2. 주요 검증 우회(Bypass) 매개변수 분류 및 적용 기법

SITL 및 실내 HITL 검증을 위해 개발자가 주로 조작해야 하는 매개변수는 크게 세 가지 범주로 나뉜다.

2.1 하드웨어 의존성 해제 파라미터 (SYS_HAS_*)

특정 하드웨어가 물리적으로 연결되어 있지 않음에도 불구하고, 시스템이 해당 센서를 찾기 위해 블로킹(Blocking)되거나 에러를 발생시키는 것을 방지한다.

  • SYS_HAS_MAG: 지자기 센서가 없는 SITL에서 0으로 설정하면 나침반 점검 로직이 비활성화된다.
  • SYS_HAS_BARO: 기압계가 없는 시스템에서 오류 발생을 억제한다.

2.2 커맨더 정책 우회 파라미터 (COM_*)

비행 전 점검(Pre-flight Check) 중 특정 조건을 충족하지 않아도 강제로 시동을 허용하도록 commander의 정책을 완화한다.

  • COM_ARM_WO_GPS: 기본값은 0 (비활성화)이나, 값을 1로 설정할 경우 유효한 글로벌 위치 판별 데이터(Global Position Fix)가 없어도 시동이 가능해진다. 실내 VIO 테스트나 단순 자세 제어(Attitude Control) 검증 시 필수적이다.
  • COM_ARM_MAG_STR: 지자기 센서의 강도(Strength) 검사를 우회한다. 실내에서 자기장 간섭이 너무 심할 때 검사를 완화하기 위해 사용된다.

2.3 안전장치 동작 변경 파라미터 (NAV_*)

데이터 링크 상실 또는 배터리 부족 시 발생하는 페일세이프(Failsafe) 메커니즘을 테스트 벤치 용도에 맞게 재구성한다.

  • NAV_DLL_ACT (Data Link Loss Action): GCS와의 MAVLink 통신이 끊어졌을 때의 동작을 정의한다. SITL 디버깅 중 GCS 연결을 잠시 끊어도 기체가 비상 착륙(RTL/Land) 모드로 전환되지 않도록 비활성화(Disabled) 상태로 매개변수화할 수 있다.

3. 검증 우회 알고리즘의 소스 코드 수준 이해

commander 모듈 내의 시동 전제조건 평가는 다음과 같은 논리 흐름으로 구현된다. (C++ 슈도코드 예시)

// src/modules/commander/Arming/PreFlightCheck/PreFlightCheck.cpp 내부 로직 예형
bool PreFlightCheck::isGpsCheckPassed()
{
    // 1. GPS가 시스템에 활성화(SYS_HAS_GPS)되어 있는지 확인
    if (_param_sys_has_gps.get() == 0) {
        return true; // GPS 요구안함: Bypass 성공
    }

    // 2. 실제 GPS 토픽(uORB) 수신 및 무결성 확인
    bool gps_valid = _gps_validality_check();

    // 3. COM_ARM_WO_GPS 정책 매개변수 확인
    if (!gps_valid && _param_com_arm_wo_gps.get() == 1) {
        mavlink_log_warning(&_mavlink_log_pub, "GPS is bad, but bypassed by COM_ARM_WO_GPS");
        return true; // 정책적 강제 Bypass 성공
    }

    return gps_valid;
}

위 구조에서 볼 수 있듯이, 펌웨어는 하드웨어 추상화(SYS_HAS_*)를 최우선으로 검사한 뒤, 센서의 물리적 상태를 확인하고, 마지막으로 정책적 백도어(COM_ARM_WO_GPS)를 통하여 유연한 시뮬레이션 환경 구축을 보장한다.

4. QGroundControl (GCS) 기반 테스트 벤치 구성 및 설정

개발자는 QGroundControl(v4.3.x 이상)을 활용하여 GUI 환경에서 직관적으로 검증 우회를 설정할 수 있다.

  1. SITL/HITL 접속: Gazebo 시뮬레이터 또는 USB 연결된 픽스호크(Pixhawk) 컨트롤러를 QGC에 연동한다.
  2. Parameters 메뉴 진입: Vehicle Setup 화면에서 “Parameters” 탭을 클릭한다.
  3. 검색 및 변경: 검색창에 COM_ARM_WO_GPS, SYS_HAS_MAG 등을 입력하여 값을 1 또는 0으로 수정 후 “Save“를 눌러 기체에 반영한다.
  4. 재부팅(Reboot): 일부 핵심 시스템 파라미터(특히 SYS_HAS_* 및 CBRK_* 계열)는 비행 제어 데몬(Flight Control Daemon)이 초기화될 때만 반영되므로 재시동이 요구된다.

6. 결론 및 공학적 주의사항

테스트 벤치 및 SITL/HITL 환경에서의 시스템 검증 우회(Bypass) 파라미터 설정은 개발 생산성을 비약적으로 높여주는 핵심 기능이다. ROS2 연동을 통한 Offboard 모드 개발이나 복잡한 비행 로직 튜닝 단계에서 잦은 에러 발생을 억제할 수 있다.

그러나 본 매개변수화 기법은 필연적으로 “거짓 참(False Positive)” 상태를 유발한다. 따라서 시뮬레이션 및 실내 벤치 검증이 완료된 기체의 펌웨어를 실제 자율 에이전트 비행(Real-world Autonomous Flight)에 투입할 경우, 모든 COM_*, SYS_HAS_* 및 회로 차단기 등 배포 환경에 부합하지 않는 우회 파라미터를 반드시 원복(Factory Reset 또는 Parameter 재적용)해야 한다. 이러한 형상 관리 철저가 GCS 기반 관제 기술 및 안전 사고 방지를 위한 첫 번째 원칙이다.
