43.6.5 하트비트(HEARTBEAT) 오실레이터 주파수 분석 기반 기체 실상태와 GCS 가상 상태 머신 간의 위상 불일치(Divergence) 감지 및 상태 강제 정정(Error Correction) 트리거 로직

지상 관제 시스템(QGroundControl 등)과 무인항공기(PX4-Autopilot) 간의 자율비행 제어 아키텍처는 근본적으로 분산 시스템(Distributed System)의 형태를 취한다. 이러한 구조에서는 필연적으로 통신 매체의 불안정성에 의해 기체 내부에서 유지되는 “물리적 실제 상태(Physical State)“와 관제 시스템이 인지하는 “가상 상태 머신(Virtual State Machine)” 간의 상태 위상 불일치(State Divergence, 현업에서는 ‘Split-brain’ 문제로도 통칭함)가 발생할 수 있다.

이러한 상태 불일치를 조기에 감지하고, 비행 신뢰성을 보장하기 위해 시스템은 MAVLink 프로토콜의 근간인 HEARTBEAT 메시지의 주파수(Frequency) 및 상태 변수를 분석하여 즉각적인 에러 정정(Error Correction)을 수행해야 한다.

1. 하트비트 오실레이터(Oscillator) 주파수 특성 분석

PX4-Autopilot에서 HEARTBEAT 메시지는 특정 이벤트(Event)에 의존하지 않고 독립적인 하드웨어 혹은 RTOS 타이머(Timer) 기반의 발진기(Oscillator)처럼 고정된 주기(일반적으로 1Hz, 즉 1초 간격)로 MAVLink 인터페이스를 통해 브로드캐스팅(Broadcasting)된다. 이 주기적 신호는 기체의 생존성(Liveliness)을 입증하는 데 그치지 않고, 그 방출 주파수(Emission Frequency)의 도달 지연 패턴을 분석함으로써 네트워크의 상태를 추정하는 지표로 활용된다.

수신 측인 GCS(QGroundControl v4.3 기준)에서는 다음과 같은 주파수 지터(Jitter) 분석 모델을 사용해 통신 링크의 건전성을 평가한다:

  • 주파수 위상 고정 손실 (Phase Lock Loss): T_{interval} (연속된 두 하트비트의 수신 시간 차이)가 예상되는 1초를 현저히 초과할 경우(예: 패킷 손실 혹은 대역폭 고갈), 기체의 상태 변환 시점이 왜곡되어 수신될 가능성이 높다고 판단한다.
  • Ardupilot의 경우 상태 스트림(Stream) 구성 시 다양한 파라미터를 빈번하게 푸시(Push)하는 경향이 있으나, PX4는 보다 엄격하게 분리된 HEARTBEAT 중심의 단방향 상태 동기화 기조를 유지하여 채널 부하에 대한 민감도를 낮게 관리한다.

2. 상태 머신 간의 위상 불일치(Divergence) 발생 매커니즘

GCS 사용자가 비행 모드 변경(예: Position에서 Mission으로 전환)을 지시하면 GCS는 MAVLink COMMAND_LONG (명령: MAV_CMD_DO_SET_MODE) 패킷을 발송하고, 자체적인 가상 상태 머신의 UI/UX 렌더링을 선반영(Optimistic Update)하거나 대기열에 둔다.

만일 COMMAND_ACK 패킷이 무선 링크의 병목으로 인해 소실(Drop)되었음에도 GCS가 가상 상태 머신에서 로컬 타이머 오작동으로 모드 강제 변경을 수용하는 상황이 벌어지거나, 기체 자체가 펜스(Geofence) 조건 위반 또는 센서 고장 등 복합적인 이유로 즉각적인 모드 거부(Fallback)를 내부적으로 단행한 경우가 발생하면, 기체 실상태 \neq GCS 가상 상태 인 위상 불일치 조건이 성립된다.

3. 위상 불일치 감지 및 에러 강제 정정(Error Correction) 로직

QGroundControl과 PX4-Autopilot 아키텍처는 이를 방어하기 위해 **“절대적 신뢰의 원천(Single Source of Truth)은 무조건 기체의 HEARTBEAT가 담고 있는 상태 변수로 규정한다”**는 공학적 철학을 준수한다.

다음 트리거 로직을 기반으로 강제 정정이 이루어진다.

  1. 상태 불일치 평가기(State Divergence Evaluator) 가동: GCS 내부의 Vehicle 클래스 인스턴스는 매 1Hz씩 도달하는 HEARTBEAT 메시지를 디코딩하여 현재 기체의 custom_mode 및 base_mode 값을 읽어 들인다.
  2. 소프트웨어 래치(Latch) 분석: GCS가 마지막으로 발행한 비행 제어 명령 타임스탬프 이후, 일정 지연 시간(Timeout, 보통 RTT를 고려하여 1.5 ~ 2.0초 이내) 내에 하트비트의 보고 상태가 기대 상태방정식의 결과와 다를 경우 불일치 플래그(Divergence Flag)를 발생시킨다.
  3. 강제 정정 트리거(Forced Correction Trigger): 불일치가 확인되면 GCS는 내부 가상 상태 머신의 스냅샷(Snapshot)을 철회(Roll-back)하고 기체의 HEARTBEAT 값을 강제로 오버라이드(Override)하여 UI 렌더링 요소 및 다음 통제 로직 결정 트리에 반영한다. 이는 사용자로 하여금 허위의 통제권한을 갖고 있다는 착각을 미연에 방지하기 위함이다.

4. 소스 코드 관점의 결론적 이해

PX4-Autopilot의 커맨더(src/modules/commander) 스레드는 hrt_call 타이머 콜백을 통하여 정밀한 타이밍으로 시스템 리소스 현황과 센서 데이터 퓨전 모듈(EKF)의 이상 유무를 결합한 후 HEARTBEAT를 주기적으로 발행한다.

결국 자율 비행 시스템에 있어 GCS의 관제 논리는 기체가 스스로 방어하고 평가하여 브로드캐스팅하는 HEARTBEAT 오실레이터 주파수 변수에 철저하게 종속적(Dependent)으로 동작해야 함을 뜻한다. 이를 통해 복잡도 높은 군집 비행(Swarm) 환경이나 제한적인 통신 거리에서도 개별 기체 단위의 결정론적(Deterministic) 실패 복구력과 견고한 자율성을 확보할 수 있다.
