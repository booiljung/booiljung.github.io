43.5.4. 하드웨어 결함에 대비한 비상 정지(Kill Switch) 물리적 인터럽트의 커널 패닉(Kernel Panic) 회피 및 우선순위 스케줄링 할당

비행 중 제어 표면(Control Surface)의 파손, 모터 탈락, 또는 심각한 통신 보드 결함 등 복구 불가능한 하드웨어 결함이 발생했을 때 기체가 제어 불능 상태로 비행을 지속하는 것은 매우 위험하다. 이를 방지하기 위해 PX4-Autopilot은 즉각적으로 동력을 차단하는 물리적 혹은 소프트웨어적 비상 정지(Kill Switch) 기능을 제공한다.

본 절에서는 비상 정지 명령 인가 시, PX4 운영체제(NuttX RTOS) 수준에서 발생할 수 있는 커널 패닉(Kernel Panic)을 생존성 관점에서 어떻게 회피하는지, 그리고 비상 스레드가 액추에이터 출력을 차단하기 위해 어떻게 최고 등급의 우선순위 스케줄링(Priority Scheduling)을 할당받는지 분석한다.

1. 비상 정지(Kill Switch) 및 물리적 인터럽트 구조

비상 정지 시스템은 조종기(Radio Control)의 특정 채널 스위치나 GCS(QGroundControl 등)의 긴급 MAVLink 명령으로 트리거링된다. 물리적인 RC 인터럽트가 발생하거나 MAVLink 패킷이 파싱되면, Commander 및 Flight_Termination 처리 태스크가 이를 감지한다.

이 명령이 활성화되면, PX4 제어기 스택(Position/Attitude Controller)을 거치지 않고 액추에이터 출력 단(Actuator Output, 예: PWM, DShot)으로 회로 차단 신호가 직결(Bypass)된다. 즉, 모터 ESC(Electronic Speed Controller)로 향하는 지시값을 최소값(Disarmed PWM)으로 강제 고정하여 즉각적으로 동력을 제거한다.

2. 커널 패닉(Kernel Panic) 회피 및 무결성 보장 아키텍처

RTOS 기반 제어 시스템에서 액추에이터나 센서를 관장하는 드라이버 스레드를 외부에서 강제 종료(Kill)시키거나, I/O 버스(Bus)를 무리하게 잠가버리면 데드락(Deadlock)에 빠지거나 메모리 오류로 인한 **커널 패닉(Kernel Panic)**이 발생할 수 있다. 커널 패닉이 발생하면 비행 데이터 로깅(Logging)이 중단되므로 추락 원인을 사후 분석(Troubleshooting)할 수 없게 된다.

이러한 문제를 회피하기 위해 PX4-Autopilot의 Kill Switch 설계는 아래의 철학을 따른다.

  1. 소프트웨어적 상태(State) 오버라이드(Override): 개별 태스크(IMU 읽기, EKF2 추정, 제어 루프 등)를 운영체제 단에서 강제 종료시키지 않는다. 대신 actuator_armed uORB 토픽의 force_failsafe 및 manual_lockdown 플래그를 true로 브로드캐스팅(Broadcasting)한다.
  2. 믹서(Mixer) 및 출력 드라이버 종단점 차단: 제어 루프는 평소처럼 CPU 사이클을 돌며 계산을 수행하지만, 최종 단계인 출력 드라이버(예: px4io, fmu, dshot)에서 해당 uORB 플래그를 확인하고 출력을 disarmed 상태로 강제 매핑(Mapping)한다.
  3. 로깅 생존성 보장: 시스템 핵심 스레드가 그대로 살아있으므로 ULog 파일 모듈은 추락하는 순간까지 기체의 자이로(Gyro) 변화, GPS 위치, 그리고 배터리 상태 등 핵심 원격측정(Telemetry) 데이터를 SD 카드에 끊임없이 기록(Logging)할 수 있다.

3. 우선순위 스케줄링(Priority Scheduling) 할당 메커니즘

Kill Switch 로직이 다른 어떤 고부하 연산 프로세스(예: 경로 계획, 비전 융합 연산 등)보다 무조건 먼저 실행되도록 보장하기 위해, NuttX RTOS 기반의 스케줄러는 비상 정지를 관장하는 인터럽트 서비스 루틴(ISR) 및 관련 태스크에 최고 수준의 우선순위(Highest Real-time Priority)를 할당한다.

POSIX 표준을 따르는 NuttX에서 SCHED_FIFO 스케줄링 정책을 사용할 때, 센서 드라이버와 Commander 모듈은 매우 높은 태스크 우선순위(보통 240~255 사이)를 갖는다. 조종기 수신기(RC Input) 드라이버 인터럽트가 Kill Switch 패킷을 인지하는 즉시 컨텍스트 스위칭(Context Switching)이 발생하여, 다른 하위 제어기들의 CPU 점유를 선점(Preempt)하고 동력 차단 로직을 최우선으로 실행한다.

4. 아키텍처 구동 다이어그램

비상 정지 신호가 입력되었을 때 시스템이 패닉 없이 안전하게 액추에이터를 셧다운하는 과정을 보여주는 시퀀스 다이어그램(Sequence Diagram)이다.

sequenceDiagram
    participant RC as RC Receiver (조종기)
    participant Cmd as Commander (최우선 순위 태스크)
    participant uORB as uORB Messaging
    participant Ctrl as Flight Controller (PID 등)
    participant Out as Actuator Driver (PWM/DShot)
    participant Log as Logger (ULog)

    RC->>Cmd: Kill Switch 인터럽트 발생 트리거
    activate Cmd
    note over Cmd, uORB: 시스템 태스크 종료 없이 플래그만 변경\n(OS Panic 회피)
    Cmd->>uORB: actuator_armed.manual_lockdown = true
    deactivate Cmd
    
    Ctrl->>uORB: (정상 제어 연산 수행 및 출력 지시)
    
    Out->>uORB: actuator_armed 상태 구독
    activate Out
    note over Out: manual_lockdown 확인됨
    Out-->>Out: 모든 제어 지시값 무시 (Bypass)
    Out->>Out: ESC로 출력되는 모든 PWM을 최소/Disarm 값으로 강제 고정
    deactivate Out
    
    Out->>Log: (기체 추락 동역학 데이터 지속 기록 보장)
    note right of Log: 추락 전후의 데이터 로깅 \n 정상 수행 완료

이와 같은 고도의 실시간(Real-time) 운영체제 스케줄링 최적화와 결함 허용(Fault Tolerant) 설계를 바탕으로, PX4-Autopilot은 치명적 비상 정지 상황에서도 운영체제의 무결성을 유지하며 비행 분석 데이터를 온전히 보존하는 강력한 신뢰성을 제공한다.
