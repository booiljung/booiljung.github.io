53.6.3.2 누락된 임무 아이템 도달(Mission Item Reached) 이벤트의 사후 로깅 파싱(Parsing) 및 GCS 임무 진행도(Progress Bar) 선형 보상

드론과 지상 관제 시스템(GCS, Ground Control Station) 간의 통신이 일시적으로 단절된 상태에서, PX4-Autopilot 기반 기체가 자동 임무(Auto Mission)를 계속 수행할 경우, 중요한 MAVLink 메시지 중 하나인 MISSION_ITEM_REACHED 메시지가 GCS에 도달하지 못해 유실될 수 있다. 통신이 복구된 직후, GCS는 기체가 통신 단절 기간 동안 달성한 임무 진행 상황을 파악하여 임무 진행도(Progress Bar) 및 관련 UI를 정확하게 갱신해야 한다. 본 절에서는 QGroundControl(QGC)과 PX4-Autopilot 환경(예: PX4 v1.14, QGroundControl v4.3 기준)에서 유실된 임무 도달 이벤트를 사후에 파싱(Parsing)하고, 임무 진행도를 선형 보상(Linear Compensation)하는 알고리즘과 아키텍처를 상세히 분석하라.

1. 통신 단절 시 임무 진행 이벤트 누락의 문제 정의

PX4-Autopilot은 자동 비행 모드(Mission Mode)에서 각 임무 항목(Waypoint 등)에 도달할 때마다 GCS로 MISSION_ITEM_REACHED (MAVLink 메시지 ID: 46) 메시지를 브로드캐스트한다. 통신 링크(Data Link)가 단절되면 기체는 계속해서 비행을 수행하며 웨이포인트를 통과하지만, 이 메시지들은 GCS로 전달되지 못한다. 이후 통신이 복원되었을 때, GCS(MAVLink 연결 관리자) 측에서는 다수의 웨이포인트가 누락된 상태로 나타난다.

Ardupilot의 경우에도 임무 진행 중 통신이 유실되면 유사한 문제가 발생하나, Mission Planner와 같은 GCS는 재연결 시 파라미터나 상태값을 기반으로 재동기화를 수행하는 고유의 내부 전략이 존재한다. PX4-Autopilot과 QGroundControl 조합에서는 재동기화 시퀀스에서 현재 활성화된 임무 시퀀스 번호(MISSION_CURRENT 메세지)를 강제로 폴링하거나 상태 벡터를 통해 위치를 파악한다. 하지만 통신 두절 구간의 ‘도달 이벤트’ 자체는 누락되므로 진행도의 비선형적 도약과 조작자 UI 상의 인지적 혼란을 야기할 수 있다.

2. 사후 파싱(Parsing) 및 가상 이벤트를 통한 누락 내역 재구성

통신 단절 동안 발생한 이벤트를 GCS가 완전히 파악하고 UI를 갱신하기 위해, 실시간 텔레메트리 복구 외에도 기체 내부의 진행 정보를 기반으로 누락된 이벤트를 사후 파싱하는 방식을 이해해야 한다.

2.1 이벤트 복원 메커니즘

통신 복구 즉시 비동기 메타데이터 동기화를 수행하는 과정에서, GCS는 누락된 구간의 이벤트를 재구성하기 위해 기체의 임무 진행 상태 메타데이터를 활용한다. 비행 로깅(ULog 파일 등)을 완전히 다운로드하지 않고도 MAVLink 기반으로 빠른 복구 파싱을 수행하라.

  1. 현재 활성 임무 확인: 통신 복구 후 기체가 전송하는 MISSION_CURRENT (메시지 ID: 42) 메시지의 seq 필드를 확인하여 현재 기체가 향하고 있는 임무 항목 번호를 식별하라.
  2. 이력 추정(History Estimation): 통신 단절 전 마지막으로 온전하게 수신한 MISSION_ITEM_REACHED의 seq 번호와 현재 seq 번호 사이의 차이를 계산하라.
  3. 가상 이벤트 발행(Virtual Event Emission): QGroundControl 내부의 MissionController 클래스 등에서 누락된 seq 번호들에 대해 순차적으로 내부 이벤트를 발생시켜, 누락된 항목들이 달성된 것으로 시스템 상태를 업데이트하라.

2.2 시스템 아키텍처 동작 흐름도

시스템 내 복구 시퀀스의 데이터 흐름과 이벤트 보상 로직 프로세스를 다음 아키텍처 다이어그램을 통해 구성하라.

3. 임무 진행도(Progress Bar)의 선형 보상(Linear Compensation) 알고리즘

통신이 끊어졌다가 다시 연결되었을 때, GCS UI의 임무 진행 바가 일시에 급격하게 점프(Step Jump)하는 현상은 조작자에게 큰 불안정성을 주며, 비정상적 기체 거동이 있거나 오류가 발생한 것으로 오인하게 만들 수 있다. 이를 방지하기 위해 진행도를 지속적이고 매끄럽게 업데이트하는 인지 공학적 선형 보상 기법이 적용되어야 한다.

3.1 선형 보상 수학적 모델링

총 임무 항목 수를 N, 단절 전 마지막 도달 항목을 S_{last}, 통신 복구 직후 확인된 현재 활성 항목을 S_{current} 라고 하자. 진행도 P (백분율)는 이상적으로 다음과 같이 계산된다.

P = \left( \frac{S_{current}}{N} \right) \times 100 \ (\%)

단절 구간에 해당하는 진행률 도약 폭을 한 번에 갱신하지 않고, 인터폴레이션(Interpolation)을 사용하여 시간에 따른 선형 진행률 P(t)를 산출하여 적용하라.

P(t) = P_{last} + (P_{current} - P_{last}) \times \frac{t - t_0}{\Delta t_{comp}}

여기서:

  • t_0: 통신 복구 후 보상 애니메이션이 시작되는 시점
  • \Delta t_{comp}: 보상 애니메이션에 할당된 지속 시간 (예: 1.0 \, \text{sec} 내외)
  • P_{last}: 단절 전 UI에 표출되던 최종 진행도
  • P_{current}: 통신 복원 수신을 통해 파악된 현재 실제 진행도

3.2 구현 아키텍처 (C++ 백엔드 및 QML 프론트엔드 연동)

QGroundControl과 같은 최신 GCS 애플리케이션에서는 통신 모듈을 관장하는 C++ 백엔드와 시각화를 관장하는 QML 간의 시그널-슬롯(Signal-Slot) 구조를 통해 선형 보상을 구현한다. 다음은 누락 이벤트를 식별하고 보상을 트리거하는 C++ 구현의 개념적 원리를 보여준다.

// QGroundControl 측 누락 이벤트 처리 로직 개념 설계 (C++)
void MissionController::_handleMissionCurrent(const mavlink_message_t& message) {
    mavlink_mission_current_t mission_current;
    mavlink_msg_mission_current_decode(&message, &mission_current);

    int current_seq = mission_current.seq;
    
    // 이전에 기록된 시퀀스 번호와 현재를 비교하여 누락 판단
    if (current_seq > _last_reached_seq + 1) {
        // 통신 단절로 인한 누락 이벤트 감지
        for (int i = _last_reached_seq + 1; i < current_seq; ++i) {
            _parseAndEmitVirtualReachedEvent(i);
        }
        
        // UI 선형 보상을 위한 Signal 발생 (프론트엔드로 전달)
        emit smoothProgressUpdateNeeded(_last_reached_seq, current_seq);
    }
    
    _last_reached_seq = current_seq - 1; 
}

이와 같이 C++ 백엔드에서 smoothProgressUpdateNeeded 시그널을 발행시키면, 프론트엔드 UI 컴포넌트(ProgressBar.qml 내부의 NumberAnimation 등)가 이를 수신하여 점진적인 선형 보상 애니메이션 처리를 수행하여 시각적 급변을 완화한다.

4. 이벤트 파싱과 상태 보정 로직 통합 검증 (SITL)

독자적인 GCS를 설계하거나 펌웨어 관제 로직을 수정할 때, 임무 재진입 과정과 진행도 보상 로직은 SITL(Software In The Loop) 환경을 거쳐 신뢰성을 검증해야 한다. 다음 절차를 토대로 검증 환경을 시뮬레이션하라.

  1. 테스트 시나리오 구성: Gazebo 기반 PX4 SITL 시뮬레이터에서 15개 이상의 임무 항목을 가진 미션 경로를 구성하고 업로드하라.
  2. 네트워크 장애 인위적 유도: 기체가 3번 웨이포인트를 통과한 직후, 네트워크 트래픽 제어 도구(Linux tc 명령어 등)를 이용해 14550 또는 통신 포트의 UDP 트래픽을 인위적으로 차단(Drop)하라.
  3. 복구 및 시스템 검증: 기체가 8번 웨이포인트 근처에 도달한 시점에서 통신 차단을 해제시키고 연결을 회복하라. GCS 터미널 로그 상에서 누락된 4~7번 항목이 내부 가상 이벤트로 정상 파싱되는지 점검하라.
  4. UI 선형성 분석: 임무 진행바가 보상 로직에 따라 단시간에 선형적으로(Linear Animation 적용) 잔여 완료 진행도를 자연스럽게 표출하는지 분석하라.

통신 단절 후 누락된 도달 이벤트를 사후에 식별하고, 사용자 인터페이스 상에서 이를 부드럽게 보상하는 과정은 PX4-Autopilot을 활용하는 무인항공기의 장거리 운용 신뢰성을 확립하기 위한 필수 기술이다. 이러한 체계는 운용자에게 높은 상황 인식(Situation Awareness) 능력을 부여하고 시스템 패닉을 원천적으로 차단한다.
