지상 관제 시스템(GCS) 또는 무인기 내부의 텔레메트리 로깅 큐는 물리적인 메모리의 제약으로 인해 무한정 데이터를 적재할 수 없다. 특히 통신 단절(Data Link Loss) 시간이 길어질 경우 버퍼에 적재 대기 중인 패킷은 급격히 증가하며, 결국 메모리 오버플로우(Overflow) 상황에 직면하게 된다. 이러한 한시적 혹은 장기적인 버퍼 포화 상태에서 시스템 붕괴를 막고 최신 상태의 관측성을 유지하기 위해 링 버퍼(Ring Buffer) 구조와 LRU(Least Recently Used, 가장 오래된 데이터 폐기) 정책을 통합적으로 설계하는 기술을 상세히 고찰한다.

1. 링 버퍼(Ring Buffer) 아키텍처의 필요성

일반적인 선형 큐(Linear Queue)나 가변 길이 리스트(Dynamic List)는 데이터 삽입과 삭제 시 지속적인 동적 메모리 할당(Dynamic Memory Allocation) 및 해제(Deallocation)를 유발하여 메모리 단편화(Memory Fragmentation)와 성능 저하를 일으킨다. 실시간 운영체제(RTOS) 환경 기반의 PX4 구동 보드(NuttX 등)나 모바일 QGroundControl(이하 QGC) 환경에서는 이러한 지연을 제어하기 위해 링 형식의 배열 기반 순환 큐(Circular Queue)를 이용한다.

1.1 순환 큐의 동작 메커니즘

미리 정해진 고정 크기(Fixed Size)의 메모리 블록을 할당하고, 읽기 포인터(Head)와 쓰기 포인터(Tail)를 두어 메모리 주소를 순환하며 사용한다. 이를 통해 삽입 연산 시 포인터 연산 하나만으로 O(1)의 성능을 달성하며 런타임오류의 원인인 힙(Heap) 고갈 현상을 방지할 수 있다.

2. 오버플로우(Overflow) 발생 시의 LRU 폐기 정책

통신 단절로 인해 텔레메트리 전송이 지연되면 링 버퍼의 Tail 포인터가 Head 포인터를 따라잡게 되는 큐 풀(Queue Full) 상황이 도래한다. 이때 새로운 MAVLink 메시지를 어떻게 처리할 것인지에 대한 정책이 요구된다.

제어 공학 관점에서 GCS 관제는 과거 10초 전의 기체 상태보다, 현재 1초 전의 최신 상태(Latest State)가 압도적으로 높은 정보 가치를 지닌다. 따라서 버퍼 포화 상황에서는 버퍼 진입 시점이 가장 오래된(Least Recently Used) 데이터를 강제로 밀어내고(Drop) 최신 데이터를 덮어쓰는(Overwrite) 정책이 타당하다. 이를 LRU 기반 정책이라 명명할 수 있다.

2.1 PX4 및 QGC의 소스 코드 수준 구현 사례

PX4의 내부 로거 모듈이나 uORB 메시지 큐 통신 망에서는 큐사이즈를 초과할 경우 명시적으로 타임이웃 혹은 Drop 플래그를 증가시키며 버퍼 포인터를 강제로 이동시킨다.

// 링 버퍼 Push 로직 기반 LRU Drop 메커니즘 간소화 예시
if (buffer->count == buffer->capacity) {
    // 큐가 가득 찼으므로 가장 오래된 요소(Head 위치)를 폐기
    buffer->head = (buffer->head + 1) % buffer->capacity;
    buffer->drop_count++; // 로깅 또는 모니터링을 위한 Drop 카운트 갱신
}
// Tail 포인터 위치에 새 데이터 덮어쓰기 (Overwrite)
buffer->data[buffer->tail] = new_mavlink_message;
buffer->tail = (buffer->tail + 1) % buffer->capacity;

if (buffer->count < buffer->capacity) {
    buffer->count++;
}
  • Ardupilot(Mission Planner)와의 비교: Ardupilot의 일부 클래식 로깅 루틴은 큐가 찼을 때 Wait 혹은 아예 새로운 데이터를 버리는 버리고(Drop New) 기존 데이터를 유지하려는 시도(Strict Queueing)를 하기도 하나, 현대적인 PX4의 uORB 메시징 구조는 센서 융합(Sensor Fusion)의 필요성 상 ‘최신 데이터 우선 확보’ 철학을 적극적으로 채택하여 오래된 데이터를 과감하게 덮어쓴다.

4. 결론 및 고찰 (Conclusion & Discussion)

복잡한 다중 에이전트(Swarm) 관제나 열악한 통신 환경에서 대량의 텔레메트리가 버퍼에 누적되는 현상은 불가피하다. 동적 배열(Vector) 등을 무한정 늘리는 방식은 지상 관제 시스템의 OOM(Out of Memory) 크래시(Crash)를 유발하므로 절대 지양해야 한다. 대신, 시스템 설계자는 고정 크기의 링 버퍼(Ring Buffer) 환경을 구축하고, 상태 모니터링에 있어 과거 데이터의 유효 기간(Expiration)이 짧은 기체 역학(Dynamics) 특성을 반영하여 과감한 LRU 폐기(Overwrite) 정책을 수립해야 한다. 이를 통해 예측 불가능한 통신 지연 상황 속에서도 시스템의 동작 안정선(Margin of Safety)과 런타임 메모리 견고성을 보장하라.
