53.1.4 원격 조종 제어기(RC)와 지상 관제 시스템(GCS) 간의 다중화 상태 공간(Multiplexed State Space) 상실 영향

1. 개요: 다중화 상태 공간(Multiplexed State Space)의 정의

무인항공기(UAV) 시스템 운용 시, 비행 제어기(FCS)는 전통적인 조종기를 통한 원격 제어(RC, Remote Controller) 링크와 지상 관제 시스템(GCS, Ground Control Station)을 통한 데이터 텔레메트리 링크라는 두 가지 독립적인 지휘 채널을 동시에 수용한다. 이 두 제어 주체는 기체 내부의 비행 모드(Flight Mode)와 제어 입력(Control Input)을 상호 공유 및 덮어쓰기(Override)할 수 있는 **다중화 상태 공간(Multiplexed State Space)**을 형성한다.

평시 운용 중에는 두 링크가 기체의 안전한 비행 궤적을 이중으로 보장하는 결함 허용(Fault Tolerance) 수단이 되지만, 이 중 하나 또는 전체가 상실될 경우 상태 기계(State Machine)는 상호 의존성에 의해 복잡한 논리적 교착 상태(Logical Deadlock)나 의도치 않은 비행 거동을 겪게 된다. 본 절에서는 PX4-Autopilot 아키텍처를 기반으로 이러한 다중화 상태 공간 상실 현상의 메커니즘과 그 영향을 분석한다.

2. 제어 권한(Control Authority)과 PX4 아키텍처의 다중화 로직

PX4-Autopilot은 uORB(Micro Object Request Broker) 메시징 버스를 통해 제어 주체를 추상화한다. RC 수신기로부터 들어오는 SBUS/PPM 신호는 input_rc 버스를 거쳐 manual_control_setpoint 토픽으로 발행되며, GCS로부터 수신되는 MAVLink 조이스틱 패킷 또는 자율 비행(ROS2 등) 오프보드(Offboard) 제어 명령은 trajectory_setpoint나 비행 모드 변경 커맨드로 퍼블리시된다.

2.1 Ardupilot과의 제어 관제 철학 차이

Ardupilot은 전통적으로 RC 신호를 최상위 제어 권한으로 취급(RC Override)하는 단선적 우선순위 모델을 강하게 따르는 경향이 있다. 반면 PX4-Autopilot은 Commander 모듈 내에서 **비행 모드(Nav State)**에 따라 GCS 제어와 RC 제어의 상대적 가중치와 상태 전이 우선순위를 동적으로 결정하는 다원적 계층 모델을 지향한다. (예: Offboard 모드 에서는 RC 스틱 입력이 데드밴드(Deadband)를 넘지 않는 한 자동 비행 명령이 우선권을 가짐).

3. 다중화 상태 공간 상실(Loss)의 케이스 분석 및 시스템 영향

상호 보완적인 제어 공간이 훼손되었을 때, PX4 내부의 상태 측정기(State Estimator)와 커맨더(Commander)는 상실된 채널을 식별하고 가용한 채널로 권한을 완전히 이양(Fallback)해야 한다.

  1. RC 링크 단절 (RC Loss): 조종기 전원이 꺼지거나 전파 도달 거리를 벗어나는 경우. PX4-Autopilot은 NAV_DLL_ACT(Data Link Loss Action) 파라미터와 달리, NAV_RCL_ACT(RC Loss Action) 파라터를 통해 홀드(Hold), 복귀(Return), 착륙(Land) 등의 단계를 평가한다. 이 때 GCS 텔레메트리 채널이 살아 있다면 GCS 조작자가 즉시 자율 비행 통제권이나 마우스 조작 기반의 상태 이동 능력을 승계하게 된다.
  2. GCS 링크 단절 (Data Link Loss): COM_DL_LOSS_T를 초과하여 GCS와의 Heartbeat가 소실되는 경우. RC가 연결되어 있다면 조종사가 수동 모드(Manual) 또는 위치 제어 모드(Position)로 개입할 수 있다.
  3. 이중 상실 (Dual Loss / Multiplexed Loss): RC와 GCS 채널 모두 끊어지는 상태. 이때 멀티플렉서에 의한 궤적 추종 명령 공간은 완전 소실되며, 시스템은 오직 온보드 센서(IMU, GPS, Baro) 기반의 Failsafe Return 모드로 강제 전이(Transitions)를 감행한다.

4. 커맨더(Commander) 소스 코드 레벨의 다중화 상태 보호 로직

PX4의 심장부인 commander 태스크 구조를 살펴보면, 두 통신 단절을 지속적으로 폴링(Polling)하고 평가하는 논리적 분기점이 나타난다.

// src/modules/commander/Commander.cpp 의 상태 평가 루프 예시
void Commander::checkForFailsafe()
{
    const bool rc_lost = _rc_loss_alarm_active;
    const bool datalink_lost = _datalink_loss_active;

    // 두 공간 중 하나라도 상실되었을 경우의 상태 천이 평가
    if (rc_lost && !datalink_lost) {
        // GCS가 존재하면 GCS 통제권(Offboard/Mission) 허용 여부 점검
        if (!is_gcs_control_active()) {
            set_failsafe_state(FailsafeTrigger::RC_LOSS);
        }
    } 
    else if (!rc_lost && datalink_lost) {
        // RC가 존재하면 조종사의 수동 제어로 권한 이양 대기
        set_failsafe_state(FailsafeTrigger::DL_LOSS);
    } 
    else if (rc_lost && datalink_lost) {
        // 다중 제어 공간 완전 상실
        set_failsafe_state(FailsafeTrigger::RC_AND_DL_LOSS);
    }
}

위 코드는 단일 채널 장애가 전체 시스템 장애로 확산되지 않도록 다중화 공간의 부분적 유효성(Partial Validity)을 평가하는 제어 공학의 핵심적인 방어 지점을 보여준다.

5. QGroundControl 및 ROS2 연동 시의 고려 사항

관제 UI 측면에서 QGroundControl(QGC)은 RC Loss 메시지를 수신했을 때 메인 화면에 적색 경고 알람을 즉시 점등하고, 클릭 투 이동(Click-and-Go) 모드 또는 임무(Mission) 전환을 유도하여 상태 공간 부재를 오퍼레이터의 인지 구조로 메운다.

또한, 자율 에이전트 개발을 위해 ROS2 및 Micro XRCE-DDS 구조로 오프보드 제어를 수행할 때, GCS 텔레메트리 포트와 ROS2 컴패니언 컴퓨터 포트는 서로 다른 상태 공간을 갖는다. 이때 COM_OBL_ACT(Offboard Loss Action) 파라미터가 추가로 개입하여 RC, GCS, 다중 에이전트 비행 제어기(Companion Computer)라는 3중 다중화 공간(Triplex State Space)을 형성하게 된다. 각 제어 링크의 상실이 타 제어 링크에 미치는 동적 의존성을 명확히 계산하고, Failsafe 트리거 도중 발생하는 궤적 이탈(Trajectory Excursion)을 최소화하도록 전체 시스템 아키텍처를 설계하라.
