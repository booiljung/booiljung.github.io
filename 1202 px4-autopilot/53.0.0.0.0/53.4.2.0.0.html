53.4.2 텔레메트리 스트림 중단 시 시계열 데이터베이스 버퍼링 및 데이터 정합성 유지

참고 버전 정보

  • PX4-Autopilot: v1.14.0 이상
  • QGroundControl: v4.3.0 이상

무인기 비행 중 지상 관제 시스템(GCS)과의 통신 링크(Data Link)가 예기치 않게 단절되면, QGroundControl(이하 QGC)로 향하던 MAVLink 텔레메트리(Telemetry) 스트림이 유실될 위기에 처한다. 비행 이후의 사고 조사나 정밀한 데이터 분석을 수행하기 위해서는 이러한 결측 구간(Missing Data Segment) 없이 데이터를 연속적으로 확보해야 한다. 본 절에서는 통신 단절 구간을 극복하기 위해 PX4-Autopilot 내부 및 GCS 측에서 어떻게 시계열 데이터를 버퍼링(Buffering)하고, 통신 복구 시 양측의 데이터 정합성(Data Integrity)을 유지하는지 심층적으로 논의한다.

1. 텔레메트리 단절 현상의 제어 공학적 의미

텔레메트리 스트림의 중단은 GCS 관점에서는 ’기체의 현재 상태에 대한 관측(Observation) 불능’상태를 의미한다. 이는 단순한 화면 멈춤을 넘어서, QGC가 기록 중이던 시계열 데이터베이스(Time-Series Database)에 거대한 공백(Gap)을 형성하게 한다. 추후 “Flight Review” 등을 이용해 전체 비행 로그를 재구성할 때, 이러한 공백은 제어기의 과도 응답(Transient Response)이나 외부 교란(Disturbance) 인가 시점을 은폐하는 치명적인 결함이 된다. 따라서 시스템은 네트워크 계층의 투명성(Transparency)을 포기하더라도 데이터 로깅의 영속성(Persistence)을 보장해야 한다.

2. PX4-Autopilot의 로거(Logger) 모듈 버퍼링 아키텍처

PX4에서는 GCS로 전송되는 데이터와 별개로, 내부의 logger 프로세스가 uORB 메시지 토픽을 실시간으로 구독하여 기체 내장 SD 카드에 ULog 포맷으로 데이터를 영구 저장한다.

2.1 고성능 버퍼링(High-Performance Buffering)

통신이 단절되더라도 기체의 제어 루프나 로깅 루프는 멈추지 않는다. logger 모듈은 내부적으로 대규모의 링 버퍼(Ring Buffer)와 두 개의 핑퐁 버퍼(Ping-pong Buffer)를 두어, SD 카드의 I/O 지연(Latency) 주기 동안 발생하는 데이터를 메모리 상에 캐싱(Caching)한다.

  • 동기화 원리: GCS 텔레메트리 전송 모듈(mavlink_receiver 및 mavlink_messages)의 큐는 통신 단절 시 오버플로(Overflow)되어 오래된 패킷을 폐기(Drop)할 수 있지만, logger의 로컬 스토리지 기록 작업은 독립적인 스레드와 파일 I/O 대역폭을 통해 MAVLink 스트림 단절에 영향을 받지 않고 데이터 정합성을 확보한다.

3. QGroundControl 시계열 데이터베이스 설계 및 처리

QGC는 비행 중 들어오는 MAVLink 메시지를 실시간으로 메모리 버퍼 파이프라인에 담은 뒤 텔레메트리 로그 파일(.tlog)로 영구 기록한다.

3.1 파일 I/O 및 데이터 무결성

QGC의 텔레메트리 데이터베이스는 단일 패킷 단위가 아니라 시계열 기반 트랜잭션 단위로 파일 쓰기를 커밋(Commit)한다. 따라서 통신이 단절된 순간(Timeout), QGC는 버퍼링 중이던 잔류 데이터를 .tlog 파일 시스템에 강제로 플러시(Flush)하여 부분적으로 손상된 파일 꼬리(File Tail)에 의한 오염(Corruption)을 방지한다.

3.2 Ardupilot(Mission Planner)와의 데이터 로깅 차이점

Ardupilot 아키텍처에서는 기체 내부에 저장되는 로그 포맷을 DataFlash(*.bin) 규격으로 유지하며, MAVLink 프로토콜을 통제하는 방식이 PX4와 미세하게 다르다.

  • Mission Planner의 경우 .tlog 저장 방식에 있어 수신된 바이트 스트림을 그대로 바이너리 덤프(Dump)하는 성형 스레드를 별도로 둔다.
  • 반면 PX4와 연동되는 QGC는 MAVLink 패킷 디코딩을 거친 후, 명시적인 MAVLink 객체 모델 스트림으로 보존하려 시도하므로, 통신 재개 시 타임스탬프(Timestamp) 동기화 오류에 좀 더 민감하게 반응할 수 있다. 이에 QGC는 호스트 PC의 RTC(Real-Time Clock)와 기체의 MAVLink 시스템 타임스탬프를 상시 비교 보정(Drift Compensation)한다.

5. 결론 및 고찰 (Conclusion & Discussion)

대역폭이 제한적이고 환경 의존성이 강한 RF 텔레메트리 환경에서 통신 단절 시 발생하는 MAVLink 스트림의 유실은 회피가 불가능하다. 따라서 PX4-Autopilot과 QGC 시스템 설계자는 이 결측 구간을 GCS 단독으로 메우기보다는, QGC의 tlog 파일 시스템 정합성을 방어(Flush)하는 선에서 처리하고, 영구적인 시계열 데이터베이스 무결성 확보는 기체 내부의 logger 파일 시스템(SD 카드 중심의 ULog 형식)을 통해 구축하는 투 트랙(Two-track) 방식을 적용한다.

시스템 통합 전문가는 로깅 빈도(Logging Rate) 설정 시 SD 카드의 I/O 병목 상태를 사전에 성능 역량 평가(Benchmarking)하여 텔레메트리 유실 시에도 내부 로깅 시스템이 견고하게 이중화(Redundancy) 역할을 수행할 수 있도록 설계해야 한다.
