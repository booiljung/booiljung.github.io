53.8.2.1 토큰 버킷(Token Bucket) 알고리즘 기반 대역폭 스로틀링(Bandwidth Throttling) 주입을 통한 페일세이프 트리거 임계점 측정

1. 개요 (Introduction)

무인항공기 시스템(UAS) 전용 통신망에서 데이터 링크(Data Link) 대역폭의 물리적 한계 도달 현상은 고화질 비디오 스트리밍(Video Streaming) 프레임과 다량의 텔레메트리(Telemetry), 커맨드 제어 데이터가 동일한 RF 대역를 공유할 때 흔히 발생하는 병목 현상이다. 리눅스 커널의 네트워크 트래픽 제어(tc) 유틸리티는 토큰 버킷 구조(Token Bucket Filter, TBF) 알고리즘을 사용해 최대 패킷 전송 속도와 버스트(Burst) 트래픽 허용치를 정밀하게 제어할 수 있다.
본 장에서는 TBF 알고리즘을 활용해 MAVLink 통신 채널에 대역폭 스로틀링(Bandwidth Throttling)을 강제로 주입하고, 이로 인해 유발되는 PX4-Autopilot의 통신 단절 감지(Data Link Loss Detection) 임계점(Threshold)과 페일세이프(Failsafe) 트리거 발동 메커니즘을 SITL(Software In The Loop) 시뮬레이션 환경 내에서 실험적으로 분석 및 수치화하는 공학적 방법론을 다룬다.

2. 토큰 버킷 필터(TBF) 알고리즘과 MAVLink 통제 적용 (TBF Algorithm and MAVLink Control Application)

토큰 버킷 알고리즘은 네트워크상의 벼락같은(Burst) 트래픽을 일시적으로 허용하면서도, 장기적 시계열에서는 트래픽 대역폭 수준을 규제(Policing)하는 대표적인 큐잉 디시플린(Queuing Discipline)이다. TBF는 개념적인 양동이(Bucket)에 일정한 속도(Rate)로 토큰(Token)을 채우고, 패킷 인터페이스 통과 시 패킷의 크기(Byte)만큼 토큰을 소모시킨다.

이는 QGroundControl(QGC)과 PX4 사이의 주기적 하트비트(Heartbeat), 자율비행 임무(Mission) 포인트 업/다운로드 트랜잭션 등 단발적이며 밀도 높은 트래픽과 지속적 관제 트래픽을 동시 제약해야 할 때 매우 현실적인 무선 대역폭 축소를 정밀히 반영한다.

시뮬레이션에서 대역폭 전송률(Rate)을 1kbps 단위로 점진적으로 스로틀링(Throttling)해 나가면서, MAVLink 패킷이 버킷 용량을 초과해 큐(Queue) 밖에서 폐기(Drop)되기 시작하는 병목 라인이 바로 관제 시스템과 기체의 통신 페일세이프 신뢰성을 검증할 수 있는 통지된(Controlled) 임계 영역이 된다.

3. 대역폭 스로틀링 주입 시나리오 검증 프레임워크 (Bandwidth Throttling Scenario Verification Framework)

PX4 SITL 포트(기본 UDP 통신망: 14550)를 경유하여 로컬호스트(Localhost) 루프백을 오고 가는 메시지 트래픽에 TBF를 강제로 주입하는 셸(Shell) 명령어 단위의 테스트 시나리오는 다음과 같다.

# PX4 SITL 환경 내 QGC와의 MAVLink 대역폭 체계를 극도로 제한하는 TBF 제어 데몬 스크립트
# (Daemon script to inject TBF and severely restrict MAVLink bandwidth in PX4 SITL environment)

# 1. 찌꺼기 큐잉 규칙 삭제 (Flush existing queuing disciplines safe)
sudo tc qdisc del dev lo root 2> /dev/null

# 2. 대역폭 9.6kbps (열악한 시리얼 텔레메트리 수준), 버스트 용량 2kb, 큐 대기 시간 100ms 파라미터로 TBF 할당
# (Allocate TBF with severe restrictions: 9.6kbps rate, 2kb burst limit, and 100ms latency queue limit)
sudo tc qdisc add dev lo root tbf rate 9.6kbit burst 2kb latency 100ms

위 규칙 할당 스크립트가 실행된 직후, QGroundControl 매니저에서는 한 번에 다운로드되는 파라미터 동기화(Parameter Synchronization) 시퀀스가 급격하게 침체되며, QGC 내부 타이머가 설정된 통신 임계 구간 안에 응답 패킷들을 온전히 수신하지 못하고 타임아웃 오류(Timeout Error)를 연쇄 발생시킨다.

4. PX4의 페일세이프 트리거 임계치 측정 분석 (Threshold Measurement Analysis of PX4 Failsafe Triggers)

분석 기술서에 따라 네트워크 대역폭을 256kbps에서 1kbps 단위로 순차적인 점감(Decrement)을 실시할 경우, 드론 체계에서 순차적으로 트리거되는 예외 발동 현황(Progressive Failure)은 아래와 같다.

  1. 상태 지연 발생 및 임계 도달 징후 (Initial Symptoms):
    TBF 대역폭 스로틀링이 15kbps 전후로 위협받게 되면 GCS 맵 화면의 지형 고도 모델 연산이나 대용량 ULog 비행 기록 스트림 우선순위가 크게 뒤처지며, 초당 1Hz로 송수신되기를 기대하는 HEARTBEAT 패킷의 도달 지터(Jitter)가 무작위로 증폭(Amplified)되기 시작한다.

  2. 트래픽 마비점 도달 (Threshold Paralyzation Reached):
    대역폭이 약 4kbps~8kbps 임계 한계점 이하로 진압당하면 프로토콜 헤더 교환을 감당할 수 없어 TBF 큐 공간이 포화(Saturated)되며, COMMAND_LONG 텔레메트리 조작 패킷이나 에이전트 오프보드 제어 명령(Offboard/Guided Payload)을 포함한 MAVLink 트래픽 파편들이 드롭(Drop)으로 손실(Lost)된다.

  3. 페일세이프 오버라이딩 실행 (Autonomous Failsafe Execution):
    이 시나리오하에서 QGroundControl 인터페이스는 즉각 화면 상단 UI에 “Communication Lost” 치명적 에러 아이콘을 현출하며, PX4-Autopilot은 COM_DL_LOSS_T 파라미터 값(예: 10초 패킷 무응답 시 자동 반환 조건)이 경과할 때 자율 통제하에 Return mode 혹은 Hold mode 상태 기계(State Machine)로 자율적 상태 전이(State Transition)를 단행한다.

5. 결론 및 향후 실무 제언 (Conclusion and Engineering Suggestions)

리눅스의 토큰 버킷(TBF) 기반 대역폭 통제 기법은 오픈소스 관제 체계인 PX4-Autopilot의 결함 허용(Fault Tolerance)과 통신 단절 감지 감수성을 수치화된 채널 스루풋(Channel Throughput) 한계선으로써 증명하는 가장 신뢰도 높은 정량적 벤치마킹 데이터 세트 도구이다.
초장거리 위성 기반 통신망(SATCOM)이나 초협대역(Ultra-Narrowband) RF 모뎀 적용을 목표로 삼는 국방 및 특수 목적용 UAS 소프트웨어는 Ardupilot, PX4 플랫폼의 구분 없이 사전에 이와 같은 TBF 모형으로 페일세이프 인지 알고리즘 임계점을 측정해야만 한다. 시스템 보안 학회 등의 **“Simulation of Bandwidth-Constrained MAVLink Communications for Unmanned Aerial Systems”**와 같이 검증된 학술 설계 논조에서도 요구되는 바와 같이, 안정적인 GCS 관제 무결성을 얻고자 구현 단계에서 극한 대역폭 스로틀링 하중 테스트는 어떤 소프트웨어 검증보다도 우선 실시해야 한다.
