53.1.3.2 섀넌-하틀리 정리(Shannon-Hartley Theorem) 기반 대역폭 감소에 따른 채널 용량(Channel Capacity) 한계 돌파 현상

1. 개요 및 섀넌-하틀리 정리의 기초

정보 이론(Information Theory)에서 섀넌-하틀리 정리(Shannon-Hartley Theorem)는 잡음이 존재하는 통신 채널을 통해 에러 없이 전송할 수 있는 최대 정보 전송률인 채널 용량(Channel Capacity)을 정의한다. 수식은 다음과 같다.

C = B \log_2\left(1 + \frac{S}{N}\right)

여기서 C는 채널 용량(bps), B는 대역폭(Hz), S는 수신 신호 전력, N은 잡음 전력(Noise Power)을 의미한다.
장거리 무인항공기(UAV) 운영 시 기체가 GCS로부터 멀어지거나 전파 방해 지역을 통과할 때, 수신 신호 전력 S가 급감하여 신호대잡음비(SNR, S/N)가 하락한다. 이 경우 물리적 대역폭 B가 고정되어 있더라도 실제 유효 채널 용량 C는 급격히 감소하게 되며, 결국 PX4-Autopilot이 GCS로 송출해야 하는 MAVLink 데이터 스트림의 총량(트래픽 요구량)이 채널 용량 C를 초과하는 채널 용량 한계 돌파(Capacity Bottleneck 또는 Saturation) 현상이 발생한다.

2. 채널 용량 한계 돌파 발생 시의 시스템적 증상

채널 용량이 트래픽 요구량을 충족하지 못하면 물리 및 데이터 링크 계층에서 패킷 충돌(Collision)과 병목 큐잉(Queueing Delay)이 발생한다. 이 현상은 비행 제어 및 관제 시스템에 다음과 같은 악영향을 미친다.

  1. 텔레메트리 지연 현상(Telemetry Lag): 자세 제어 모니터링 토픽(ATTITUDE)과 위치 토픽(GLOBAL_POSITION_INT)의 GCS 수신 간격이 간헐적으로 수 초 단위로 늘어진다.
  2. MAVLink 파서 버퍼 오버런(Buffer Overrun): 픽스호크(Pixhawk) 기반 PX4의 MAVLink UART 포트 수신 버퍼에 병목이 생겨, GCS가 전송한 중요한 제어 명령(COMMAND_LONG) 패킷이 UART 수신 단계에서 소실될 위험이 급증한다.
  3. 거짓 페일세이프(False Failsafe) 트리거: HEARTBEAT 패킷이 대용량 센서 데이터 큐(Queue) 연산에 밀려 제때 전송되지 못함으로써, GCS나 기체 양측에서 실제 연결이 물리적으로 끊어지지 않았음에도 통신 단절(Data Link Loss) 상태로 오판하게 된다.

3. PX4-Autopilot의 내부 대역폭 제어 메커니즘 분석

채널 용량의 물리적 한계를 인지하고 이를 우회하기 위해, PX4-Autopilot은 mavlink 모듈에서 직렬화(Serialization) 최적화와 스트림 속도 제한(Stream Rate Throttling)을 동적으로 처리한다.

특히, MAV_0_MODE 및 MAV_0_RATE 파라미터는 텔레메트리 대역폭을 시스템 하드웨어 한계치에 맞추어 전송하는 역할을 한다. PX4 내부의 MAVLink 송신 구조(mavlink_messages.cpp)는 동적 대역폭 요구량에 맞추어 필수적인 패킷과 부가적인 패킷의 우선순위를 분리하여 큐를 스케줄링한다.

// src/modules/mavlink/mavlink_main.cpp 참조
// 버퍼 여유가 없을 때 우선순위가 낮은 스트림은 퍼블리시를 연기함
if (mavlink_get_free_tx_space(_channel) < required_space) {
    // 채널 용량 포화(Saturation) 감지
    _rate_limiter.drop_packet();
    // HEARTBEAT 및 시스템 알람(STATUSTEXT) 등은 최우선 큐에 삽입
}

Ardupilot 대비 차이: Ardupilot(SRx_* 파라미터)은 GCS의 요청에 의해 스트림 파라미터가 명시적으로 통제되는 형태를 띠는 반면, PX4는 내부의 mavlink_main 태스크 루프 내에서 자체적인 여유 버퍼 공간(Free TX Space)을 산출하여 자율적인 동적 패킷 누락(Drop) 결정을 수행한다. 이는 링크 열화 시 PX4가 하드웨어 병목에 더 탄력적으로 대응하도록 만든다.

4. QGroundControl(GCS) 측면의 링크 부하 경감 로직

QGroundControl 측은 유효 채널 용량 저하를 인지할 경우 사용자 경험(UX)과 제어 신뢰성을 보장하기 위해 다음과 같은 알고리즘을 사용한다.

graph TD
    A[Monitor MAVLink Traffic] --> B{Packet Loss > 10% ?}
    B -- Yes --> C[LinkManager: Throttle Telemetry]
    C --> D[Send MAV_CMD_SET_MESSAGE_INTERVAL]
    D --> E[PX4: Reduce IMU/Pos Publish Rate]
    E --> F[Channel Capacity vs Traffic Balance Restored]
    B -- No --> G[Maintain Current Rate]
  1. 트래픽 모니터링: QGC의 LinkManager 클래스는 시간당 수신되는 바이트 통계를 계산하여, 채널 지연이 한계 상황에 도달했다고 판단하면 동적 스트림 요건을 하향 조정한다.
  2. 동적 메시지 인터벌 제어: QGC는 PX4에 MAV_CMD_SET_MESSAGE_INTERVAL 커맨드를 전송하여, 고속으로 퍼블리시되던 HIGHRES_IMU(예: 50Hz) 나 관측 지표 스트림 속도를 5Hz 이하로 억제시킨다. 이를 통해 감소한 채널 용량 한계점(C) 이내로 전체 트래픽 대역폭을 강제 축소하게 된다.

5. 결론

섀넌-하틀리 정리에 비추어 볼 때 송수신 채널의 전파 환경이 악화되어 SNR이 감소하면 채널 용량은 불가피하게 저하된다. GCS 응용 프로그램과 자율 에이전트 개발자는 주어진 하드웨어 채널(예: SiK Radio, WiFi, LTE)의 물리적 한계를 이해하고, ROS2 기반 제어나 MAVSDK 프로그래밍 시에 불필요한 고빈도 데이터 구독(Subscription)을 줄여야 한다.

임무 필수적인 데이터(비행 모드, 페일세이프 상태, 좌표)에만 제한된 대역폭 리소스를 할당하고, 영상이나 로그 데이터 다운로드는 별도의 대용량 링크망으로 분리(Offload)함으로써 통신 단절 한계 상황에서의 유효 채널 포화 문제를 억제하라. 이는 무인기 시스템 설계의 가장 기본적인 생존성 보장 원칙이다.
