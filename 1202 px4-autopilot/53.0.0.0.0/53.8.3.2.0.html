53.8.3.2. 지속적 통합/지속적 배포(CI/CD) 파이프라인 내 통신 단절 자동화 회귀 테스트(Regression Test) 스크립트 구축 및 안전 무결성 기준(SIL) 만족도 평가

안전 필수(Safety-Critical) 무인항공기 소프트웨어의 품질 보증에 있어, 단발성의 테스트나 수동 테스트는 코드의 변경(Commit)마다 발생할 수 있는 잠재적 결함(Bug)을 완벽히 걸러낼 수 없다. 특히 PX4-Autopilot의 Failsafe State Machine 모듈이나 통신 텔레메트리 라우터 모듈은 매 릴리즈(Release)마다 구조적 개선이 이루어지기 때문에, 기존에 잘 동작하던 통신 예외 처리 로직이 훼손되지 않았음을 보장하는 **회귀 테스트(Regression Test)**의 자동화가 절실히 요구된다.

여기서는 PX4 펌웨어 v1.14 및 QGroundControl(QGC) v4.3.0 이상 환경을 타깃으로 삼아, GitHub Actions 로 대표되는 지속적 통합/지속적 배포(CI/CD) 파이프라인에 통신 단절 모사(Network Drop Simulation) 스크립트를 통합하는 구체적 아키텍처와, 이를 통해 산업용 드론의 안전 무결성 기준(Safety Integrity Level, SIL) 만족도를 평가하는 방법론을 상술한다.

1. CI/CD 기반 자동화 회귀 테스트 아키텍처

PX4의 CI 아키텍처는 보통 GitHub Actions 러너(Runner) 안에서 컨테이너화된(Dockerized) 에뮬레이션 환경(SITL + Gazebo)을 구동한다. 커스텀 지상 관제 시스템(GCS) 또는 컴패니언 컴퓨터(Companion Computer)와 PX4 간의 MAVLink 혹은 Micro XRCE-DDS 통신 단절 테스트를 파이프라인의 Check Stage로 포함시키기 위해서는 다음과 같은 3단계 파이프라인 요소가 필요하다.

  1. 지상 관제 에뮬레이터(GCS Emulator): MAVSDK(Python/C++)를 활용하여 실제 QGC와 유사한 HEARTBEAT, COMMAND_LONG 트래픽을 송출하며 기체를 가상으로 비행(Offboard 모드 등)시키는 역할.
  2. 트래픽 인터셉터(Traffic Interceptor): CI 컨테이너 브리지 네트워크 내에서 패킷 룰(예: iptables, tc)을 조작해 특정 틱(Tick)에서 통신을 완전히 끊거나 노이즈를 주입.
  3. 상태 검증기(State Validator): PX4의 원격 측정 로그(ULog 파일) 혹은 실시간 HEARTBEAT 스트림에서 Failsafe 작동 시간, 복귀 궤적, 안전 착륙 여부 등을 판명하는 Assert 구문 포함 스크립트.

1.1 CI/CD YAML 및 Python 검증 스크립트 예시

Ardupilot의 autotest 툴킷과 유사하지만, PX4는 비동기 아키텍처(MAVSDK/ROS2)를 보다 유연하게 플러그인 할 수 있다는 장점이 있다. 아래는 GitHub Actions YAML 워크플로우 내에서 호출될 수 있는 자동화 테스트 런 스크립트 구성의 예시이다.

# .github/workflows/comms_failsafe_test.yml
name: Communication Failsafe Regression Test
on: [push, pull_request]

jobs:
  sitl-test:
    runs-on: ubuntu-latest
    container: px4io/px4-dev-simulation-jammy
    steps:
      - uses: actions/checkout@v3
      - name: Build PX4 SITL
        run: DONT_RUN=1 make px4_sitl_default gazebo_typhoon_h480
      - name: Run Datalink Loss Simulation Test
        run: python3 tests/failsafe/test_comms_loss.py

test_comms_loss.py 내부에서는 비행이 시작된 후 프로세스 서브콜을 통해 의도적으로 브리지 통신 포트를 차단한다.

# test_comms_loss.py 내부 로직 발췌
import asyncio
from mavsdk import System
import os

async def run_test():
    drone = System()
    await drone.connect()
    await drone.action.arm()
    await drone.action.takeoff()
    
    # 1. 포지션 홀드 상태 대기
    await asyncio.sleep(5) 
    
    # 2. 통신 단절 트리거 (로컬 방화벽 룰 적용)
    print("Simulating Data Link Loss...")
    os.system("sudo iptables -A INPUT -p udp --dport 14550 -j DROP")
    
    # 3. 비동기 로그 폴링을 통해 10초(COM_DL_LOSS_T) 전후로 Failsafe 작동(Return 모드 전환) 검증
    failsafe_triggered = False
    async for state in drone.telemetry.flight_mode():
        if str(state) == "RETURN_TO_LAUNCH":
            failsafe_triggered = True
            print("Failsafe properly triggered.")
            break
            
    assert failsafe_triggered, "Failsafe was NOT triggered within expected timeframe!"
    
    # 4. 방화벽 룰 해제 (통신 복구)
    os.system("sudo iptables -D INPUT -p udp --dport 14550 -j DROP")

if __name__ == "__main__":
    asyncio.run(run_test())

통계적으로 유의미한 회귀 테스트를 위해, 위 스크립트는 매 풀 리퀘스트(Pull Request)마다 수십 번 다중 인스턴스로 병렬 실행(Parallel Execution)되며, 우연에 의한 성공(Flaky Test)을 필터링한다.

2. 안전 무결성 기준(SIL) 만족도 평가

이와 같은 CI/CD 자동화 도구를 구축하는 근본적 이유는 국제 기능 안전 표준인 IEC 61508 또는 항공 도메 정인 표준 시스템인 DO-178C에서 규정하는 소프트웨어 보증 수준(Software Assurance Level) 또는 안전 무결성 기준(Safety Integrity Level, SIL)을 충족하기 위함이다.

  • SIL 1 ~ SIL 2 대응: 대부분의 상업용 드론(예: 방제용, 배달용)은 통신 두절 상황에서의 위험 완화(Risk Mitigation) 요구치가 높다. CI/CD 상에서 전체 소스 코드 변경의 100% 분기 커버리지(Branch Coverage) 하에서 통신 예외 테스트가 성공적으로 Pass될 시, 규제 기관에 제출할 수 있는 신뢰성 인증 리포트로 가공될 수 있다.
  • 아키텍처 관점 분석: QGroundControl 개발 과정에서도 동일한 방식의 CI 파이프라인을 구축할 수 있다. 예를 들어, 통신이 단절되었다가 복구되는 짧은 윈도우 시간 동안 QGC 화면에 상태 불일치(Inconsistent State)를 일으키는 UI 프리징(Freezing)이나 잘못된 알림(False Alarm)이 발생하는지를 확인하는 시각적 자동화 테스트(예: Qt Squish 활용)가 동반되어야 완벽하게 GCS의 예외 처리 로직에 대한 SIL 평가를 완료할 수 있다.

이처럼 견고하게 구축된 자동화 테스트 스크립트 생태계는 FMEA 및 FTA와 결합하여 PX4-Autopilot 및 GCS 관제 비행체 소프트웨어의 치명적인 결함을 배포 이전에 선제적으로 차단하는 가장 강력한 방어선이 된다.
