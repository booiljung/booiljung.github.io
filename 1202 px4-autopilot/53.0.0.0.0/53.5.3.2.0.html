## 0.1 백엔드 제어 스레드 단절 상황에서 UI 렌더링 스레드의 초당 프레임(FPS) 저하 및 프리징(Freezing) 현상을 막는 비동기 아키텍처

무인항공기 시스템(UAS)의 지상 관제 시스템(GCS) 운영 중 통신 단절(Data Link Loss)은 피할 수 없는 예외 상황이다. PX4-Autopilot(v1.14 기준)과 QGroundControl(v4.3 기준) 간의 MAVLink 통신이 두절되었을 때, 관제 소프트웨어 내부에서는 지속적인 재연결 시도와 타임아웃(Timeout) 처리가 발생한다. 이러한 백엔드(Backend) 제어 스레드의 예외 처리 과정이 사용자 인터페이스(UI) 렌더링 스레드(Rendering Thread)에 영향을 미칠 경우, 초당 프레임(FPS, Frames Per Second) 저하 및 화면 프리징(Freezing) 현상이 초래될 수 있다. 본 절에서는 이러한 현상을 방지하기 위한 비동기 아키텍처(Asynchronous Architecture)의 개념과 구현 원리를 상세히 분석한다.

0.1.1 동기식 통신 처리의 한계와 프리징(Freezing)의 원인

과거 일부 GCS 소프트웨어나 초기의 관제 시스템에서는 통신 스레드와 UI 스레드가 물리적 혹은 논리적으로 강하게 결합(Tightly Coupled)되어 있는 경우가 있었다. 단일 스레드(Single-threaded) 환경이거나 동기식 블로킹 호출(Synchronous Blocking Call)을 남용하는 구조에서는, 네트워크 소켓의 타임아웃이나 재시도 블로킹이 UI 이벤트 루프(UI Event Loop)를 직접적으로 중단시킨다.

특히 통신이 단절된 직후, 네트워크 스택(Network Stack)은 패킷을 재시도하거나 소켓 상태를 확인하기 위해 지연(Delay)을 유발할 수 있다. 이때 메인 스레드(Main Thread, 보통 UI 렌더링을 담당)가 통신 스레드의 응답을 기다리는(Wait) 상태에 빠진다면, 렌더링 엔진은 화면을 갱신하지 못하게 되며, 사용자는 최소 수백 밀리초에서 수 초에 이르는 화면 멈춤(Freezing)을 경험하게 된다. 이는 긴급한 조작이 필요한 관제 상황에서 치명적인 문제를 야기한다.

0.1.2 비동기 아키텍처와 스레드 분리(Thread Isolation)

초고도화된 현대의 GCS 시스템인 QGroundControl은 Qt 프레임워크 기반의 철저한 비동기 아키텍처를 채택하고 있다. 프론트엔드인 QML(Qt Meta-Language) 기반 UI 렌더링 스레드와, C++ 기반의 백엔드 통신 및 로직 스레드(MAVLink 수신, Vehicle 객체 관리 등)는 완전히 분리되어 독립적인 이벤트 루프를 갖는다.

  1. 메인 UI 스레드 (GUI Thread): 프론트엔드 UI(지도 렌더링, 계기판 갱신, 사용자 입력 이벤트 처리)를 전담한다. 어떠한 경우에도 블로킹 연산을 수행해서는 안 된다.
  2. 통신 스레드 (MAVLink Link Thread): 물리적 혹은 논리적 연결(UDP, 시리얼 등)로부터 MAVLink 패킷을 수신하고 파싱(Parsing)한다.
  3. 차량 관리 스레드 (Vehicle Manager Thread): 수신된 패킷을 바탕으로 기체의 논리적 상태를 갱신하고 심박수(Heartbeat)를 추적한다. 통신 단절 감지(Data Link Loss Detection)도 이곳에서 수행된다.

백엔드 스레드에서 통신 단절을 감지하여 재연결 로직이 블로킹 상태에 놓이더라도, 해당 스레드만 대기 상태에 들어갈 뿐 메인 UI 스레드의 이벤트 루프는 정상적으로 회전하며 60 FPS의 부드러운 화면 갱신을 유지할 수 있다.

0.1.3 QGroundControl의 Signal-Slot 기반 논블로킹(Non-blocking) 상태 동기화

백엔드 스레드의 상태 변경(예: 통신 단절 감지)을 프론트엔드 UI 스레드에 전달할 때는 직접적인 함수 호출(Direct Call)을 사용하지 않는다. 대신, Qt 프레임워크의 시그널과 슬롯(Signal and Slot) 메커니즘을 큐 방식(Queued Connection)으로 호출하여 스레드 간 동기화를 논블로킹 방식으로 수행한다.

    VM-->>UI: emit connectionLostChanged(true) (Queued Connection)
    Note over VM: 재연결 루틴 또는 예외 처리 (블로킹/지연 발생 가능)
    Note over UI: UI 스레드는 VM 스레드 연산과 무관하게 즉시 동작
    UI->>UI: 계기판 '연결 끊김' 상태로 렌더링 (60 FPS 유지)
    UI->>UI: 가상 조이스틱 이벤트 무효화 처리

[도표 설명] 비동기 아키텍처 기반의 통신 단절 이벤트 처리 흐름. 백엔드 스레드(VM)의 연산 지연이 프론트엔드(UI)의 렌더링 이벤트 루프에 영향을 주지 않는 구조를 보여준다.

이 아키텍처에서 백엔드의 Vehicle 객체는 내부적으로 타이머(QTimer)를 작동시켜 마지막 Heartbeat 수신 이후 일정 시간(일반적으로 3~5초) 이상 갱신이 없으면 connectionLost 프로퍼티(Property)를 갱신하고 시그널을 방출(Emitting)한다. 프론트엔드는 이 시그널을 구독(Subscribe)하고 있다가 UI의 데이터 바인딩(Data Binding)을 통해 화면을 즉각적으로 붉은색(경고) 톤으로 전환하거나 “연결 단절(Communication Lost)” 메시지를 매끄럽게 띄우게 된다.

0.1.4 데이터 버퍼링 및 렌더링 최적화 기법

프리징을 방지하기 위한 비동기 아키텍처 외에도, 과도한 이벤트로 인한 FPS 저하를 막기 위해 추가적인 기법이 사용된다. 통신이 단절되기 직전, 노이즈가 심한 네트워크 환경에서는 잘못된 패킷이나 대량의 패킷 손실 알림이 발생하여 백엔드에서 프론트엔드로 수많은 상태 변경 시그널이 폭주할 수 있다.

이를 제어하기 위해 스로틀링(Throttling) 및 디바운싱(Debouncing) 기법을 활용한다. UI 갱신은 매 시그널마다 발생하는 것이 아니라, 렌더링 주기(V-Sync, 보통 16.6ms)에 맞춰 마지막으로 갱신된 백엔드 상태만을 한 번에 폴링(Polling)하거나 묶어서(Batch) 처리한다.

설계 시 고려사항 및 예시:

  1. 프로퍼티 바인딩(Property Binding)의 지연 평가(Lazy Evaluation): QML 내부에서 상태 값을 직접 연산하지 않고, C++ 백엔드에서 계산이 끝난 최종 결과값(플래그)만 읽어오게 하여 UI 스레드의 부하를 최소화하라.
  2. 비동기 재연결 타이머: 통신 단절 시 TCP/UDP 소켓 재연결 시도는 반드시 별도의 백그라운드 스레드(또는 Qt의 비동기 소켓 API)에서 처리하라. 소켓의 connect() 함수가 타임아웃될 때까지 블로킹되는 현상이 UI 렌더링을 방해하지 않아야 한다.

결론적으로, PX4-Autopilot과 GCS 간의 연결에서 예기치 않은 통신 장애가 발생하더라도 프론트엔드의 부드러운 렌더링 상태를 보장하는 것은 인지 공학 측면에서 매우 중요하다. 조종자에게는 시스템이 멈춘 것(Crash)인지, 단순히 통신이 끊어진 것인지 구분할 수 있는 즉각적인 시각적(Visual) 피드백이 제공되어야 하기 때문이다. 이를 위해 이벤트 주도적(Event-driven) 비동기 통신 및 스레드 격리 설계는 필수적인 기반 아키텍처로 작용한다.
