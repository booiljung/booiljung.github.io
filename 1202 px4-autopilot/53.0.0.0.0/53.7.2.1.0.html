53.7.2.1 인터넷 프로토콜(IP) 주소 변경 시 로컬 호스트 NAT(Network Address Translation) 통과 및 MAVLink 라우팅 테이블 동적 갱신

무인항공기(UAV)가 광역 이동망(LTE/5G)을 통해 비행할 때 기지국 간의 핸드오버(Handover)나 네트워크 재연결이 발생하면, 이동통신 모뎀이 ISP(Internet Service Provider)로부터 새로운 동적 IP를 할당받는 현상이 빈번하게 일어난다. 이러한 환경에서 지상 관제 시스템(GCS)이 방화벽이나 내부망 공유기 하단에 존재할 경우 NAT(Network Address Translation) 환경을 통과하여 양방향 MAVLink 무중단 통신을 이어가는 것은 네트워크 아키텍처 상의 중대한 난제이다. 본 절에서는 IP 주소 변경 환경에서 로컬 호스트 NAT를 안전하게 통과하고, PX4-Autopilot 및 QGroundControl의 MAVLink 라우팅 테이블을 동적으로 갱신하는 실무적 메커니즘을 상세히 분석한다.

1. MAVLink 환경의 NAT 통과(NAT Traversal) 구조적 한계

Ardupilot이나 PX4의 기본 TCP/UDP 통신은 일반적으로 지정된 IP 주소와 포트 번호를 고정 스크립트로 바인딩하여 시작된다.

# 일반적인 동반 컴퓨터(Companion Computer) MAVLink 라우터 실행 예
mavlink-routerd -e 192.168.1.100:14550 /dev/ttyS1:921600

그러나 모바일 통신망(LTE)을 사용할 경우 기체의 공인 IP는 언제든지 바뀔 수 있으며, GCS 역시 LTE 라우터나 사내 방화벽 뒤에 있어 외부(기체)에서 직접 GCS의 IP로 패킷을 밀어 넣기(Inbound) 어렵다. 이러한 비대칭 구조에서 IP 주소가 갱신될 때마다 기존 MAVLink 세션이 끊어지고 재연결을 위해 긴 시간 방황하는 지상 관제소 셧다운 현상이 발생한다.

2. 홀 펀칭(Hole Punching) 및 클라우드 중계(Relay) 아키텍처

NAT를 투명하게 통과하고 IP 변경 시에도 세션을 유지하기 위하여 최신 관제 시스템은 피어 투 피어(P2P) 기술인 홀 펀칭(Hole Punching)이나 브로커 서버(Broker Server) 기반의 터널링 방식을 채택한다.

2.1 UDP 홀 펀칭(UDP Hole Punching) 원리

QGroundControl(v4.3 이상)은 UDP 통신 시 기체(PX4)가 먼저 GCS 측으로 패킷 데이터를 보내도록 구성할 수 있다.

  1. GCS의 환경을 고정 공인 IP를 가진 클라우드 서버로 상정하거나, STUN/TURN 서버를 중간에 둔다.
  2. 기체가 이동통신망(LTE)에 접속하여 ISP NAT 장비를 거쳐 GCS로 HEARTBEAT UDP 패킷을 발송(Outbound)한다.
  3. 이 Outbound 패킷이 통과하는 순간, ISP와 GCS의 NAT 라우터에는 해당 세션에 대한 역방향 개방 포트(Hole)가 생성된다.
  4. GCS는 수신된 패킷의 ’실제 출발지 공인 IP와 포트’를 추출하여, 해당 주소로 MAVLink 제어 패킷을 반환(Inbound)한다.
sequenceDiagram
    participant P as PX4 (Drone)
    participant N1 as ISP NAT
    participant N2 as GCS Router NAT
    participant G as QGroundControl

    P->>N1: 동적 IP 발급 후 Heartbeat 송신
    N1->>N2: 출발지 IP 변환 (NAT) 후 전송
    N2->>G: 포트 포워딩
    Note over G: 수신 UDP 소켓의 주소와 포트를 <br/>새로운 목적지 주소로 동적 캐시
    G->>N2: 동적 캐시된 IP로 응답(Command) 전송
    N2->>N1: 방화벽 Hall 통과
    N1-->>P: 패킷 도달 보장

2.2 클라우드 기반 MAVLink 중계 브로커

가장 안정적이고 상용 UAM(도심항공교통) 레벨에 적합한 방식은 공용망(AWS, Azure 등)에 고정 IP를 가진 **MAVLink 중계 브로커(Relay Broker)**나 자율 프록시 시스템(예: ZeroTier, Tailscale, MAVLink-Router 클라우드 인스턴스)을 배치하는 것이다.

  • PX4(동반 컴퓨터)와 QGC 모두 부팅 직후 인터넷망을 통해 브로커 서버로 연결을 개시(Outbound TCP/UDP)한다. 양측 모두 외부 서버로 연결하는 형태이므로 어떠한 복잡한 로컬 NAT라도 자동 통과된다.
  • 중간 기지국 이동으로 기체의 4G/5G 모뎀 IP 주소가 변경되면, 클라우드 브로커는 기체로부터 새로 들어오는 소켓 연결을 인식하고, 기존 GCS로 향하는 라우팅 파이프라인(Session)과 즉시 병합시킨다. 이 과정을 통해 IP 변경에 따른 MAVLink 지연 시간은 수십 밀리초 이내로 단축된다.

3. MAVLink 라우팅 테이블 동적 갱신(Dynamic Routing Table Update) 메커니즘

통신 채널이 변경될 때 물리적 연결뿐만 아니라, MAVLink 프로토콜 레이어 상의 라우팅 테이블(System ID / Component ID Mapping Table) 또한 신속하게 인지하고 업데이트해야 한다.

3.1 QGroundControl의 Vehicle 인스턴스 소스 코드 이해

QGC 내부의 소스 코드를 분석하면 MAVLinkProtocol::receiveBytes() 함수 블록에서 동적 갱신 로직이 수행된다. GCS는 패킷을 1바이트씩 파싱하여 완전한 MAVLink 패킷을 만들어낸 뒤, 해당 패킷의 출발지 IP/포트 헤더와 MAVLink 페이로드 내부의 System ID를 매핑하는 해시맵(Hash Map)을 관리한다.

// QGroundControl 내부 유사 로직의 개념적 설명
void LinkManager::_receiveMessage(LinkInterface* link, mavlink_message_t message) {
    int sysid = message.sysid;
    // 기존에 알고 있던 System ID인지 확인
    if (activeVehicles.contains(sysid)) {
        Vehicle* v = activeVehicles[sysid];
        // 만약 들어온 Link 객체가 기존 통신 IP와 다르다면, 라우팅 테이블 동적 갱신
        if (v->activeLink() != link) {
             v->updateActiveLink(link); // 새로운 NAT IP/Port로 응답 경로 갱신
        }
    }
}

기체의 이동통신 IP가 바뀌어 완전히 새로운 IP 경로로 패킷이 들어오더라도, GCS는 System ID가 이전과 동일하다면 인증된 개체로 간주하고 즉각적으로 반환(Return) IP 캐시 테이블을 덮어쓴다(Overwrite). 이러한 무상태(Stateless) 기반의 응답 라우팅 갱신은 PX4-QGC 생태계가 Ardupilot 초기 생태계 대비 복잡한 분산 네트워크 구축에 더 높은 유연성을 발휘하는 장점이다.

3.2 ROS2 연동 환경 고려사항 (uXRCE-DDS)

ROS2의 DDS 생태계에서는 px4_ros_com과 Micro XRCE-DDS 에이전트 간의 통신이 다소 둔감할 수 있다. DDS 프로토콜(RTPS)은 MAVLink 대비 엔드포인트 디스커버리(Discovery) 오버헤드가 크기 때문에 NAT IP 변경 시 파티시펀트(Participant) 소멸 후 재생성에 상당한 시간이 소요된다.
따라서 PX4-ROS2 아키텍처 환경에서는 순수한 DDS 환경에 의존하기보다, 제2장에서 다룬 VPN 터널링 계층 위에서 DDS가 가상 고정 대역(e.g. 10.x.x.x)을 사용하도록 설계하여 NAT 변경 시점을 은폐(Masking)하는 것이 자율 제어 안정성을 보장하는 핵심 지침이다.

4. 요약

이기종 통신망 환경에서 모바일 IP 주소 변경에 대응하는 무중단 라우팅 아키텍처의 설계 핵심 요소는 다음과 같다.

  • 수동적 포워딩 배제: 방화벽 직접 설정이 아닌 TCP/UDP 아웃바운드 기반 서버 홀 펀칭 메커니즘을 적극 응용하라.
  • QGC 동적 라우팅 활용: GCS 소스 코드 단에 얽혀있는 System ID 기반의 무상태형 목적지 IP 다이내믹 갱신(Dynamic Update) 특성을 파악하고 활용하라.
  • 클라우드 브로커 브릿징: 대규모 자율 군집 제어나 ROS2 Offboard 관제 시에는 P2P NAT 통과의 불안정성을 피하기 위해 클라우드 중간 브로커나 VPN 인프라를 우선 도입하도록 강력히 권고한다.

이러한 전송 및 프로토콜 라우팅 계층의 동적 갱신 기능이 통합될 때, 비로소 완전한 무중단 페일오버 스펙을 갖춘 최상위 권한의 무인기 제어망을 구축할 수 있다.
