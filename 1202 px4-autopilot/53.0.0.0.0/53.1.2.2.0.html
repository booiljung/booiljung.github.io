53.1.2.2 데이터 링크 계층(Data Link Layer)의 매체 접근 제어(MAC) 프레임 충돌 및 채널 점유 실패

OSI 7계층 아키텍처에서 데이터 링크 계층(Data Link Layer)은 물리 계층을 통해 전달되는 비트(Bit)스트림을 프레임(Frame)이라는 논리적 단위로 묶고, 에러 검출 및 매체 접근 제어(MAC, Media Access Control)를 수행한다. 무인항공기(UAV) 시스템에서 발생하는 통신 단절 중에서, 기체와 GCS(지상 관제 시스템) 간 전파 자체의 세기(RSSI)는 양호함에도 불구하고 패킷 손실률(Packet Loss Rate)이 급증하여 MAVLink 통신이 두절되는 현상의 대부분은 이 데이터 링크 계층의 프레임 충돌과 채널 점유 실패에서 기인한다.

1. 매체 접근 제어(MAC)의 개념과 무인기 통신 환경

텔레메트리 라디오(예: SiK Radio 계열, Microhard 등)와 Wi-Fi 모뎀은 근본적으로 하나의 무선 주파수 매체를 근거리에 위치한 여러 기기가 공유해야 하는 다중 접속(Multiple Access) 환경에서 동작한다. 이를 통제하기 위해 CSMA/CA(Carrier Sense Multiple Access with Collision Avoidance)와 같은 MAC 프로토콜이 사용되는데, 이는 송출 전 채널의 점유 상태를 청취(Listen)하여 비어 있을 때만 송신을 시작함으로써 충돌을 회피하는 메커니즘이다.

그러나 공중 3차원 영역(Airspace)과 여러 기대체가 동시다발적으로 동원되는 복잡한 관제 환경에서는 이러한 전통적인 확률적 MAC 알고리즘이 예측하지 못한 취약점들을 드러내며 치명적인 통신 단절을 유발한다.

2. MAC 프레임 충돌(Frame Collision) 메커니즘

프레임 충돌은 두 개 이상의 노드(예: GCS 송신기, 기체 1, 기체 2 등)가 물리적으로 정확히 동일한 마이크로초 단위의 시간대에 동일한 주파수 채널로 데이터를 송출할 때 발생한다. 물리적으로 전파의 겹침(Superposition)이 발생하면서 수신 측 라디오 칩셋은 어느 쪽의 신호도 정상적인 프레임으로 복조(Demodulate)해내지 못하게 된다.

2.1 충돌 발생의 주요 원인

  1. 군집 비행(Swarm) 환경의 트래픽 폭주 점유: 다수의 PX4-Autopilot 기체가 하나의 지상 관제 시스템(GCS)에 묶여 있을 때, 모든 무인기가 동시에 고주기(예: 10Hz, 50Hz)로 고해상도의 텔레메트리 데이터(ATTITUDE, LOCAL_POSITION_NED 등)를 브로드캐스트하면, 설정된 채널 대역폭의 물리적 한계를 초과하여 무작위 충돌 확률이 기하급수적으로 증가한다.
  2. Duty Cycle 강제 제약 우회에 의한 버스트 전송: 특정 국가의 전파법(예: 유렵 SRD 대역의 LBT 의무화 및 Duty Cycle 시간 제한)에 의해 의무적인 송출 휴지기가 강제될 경우, 모뎀 내부 버퍼에 쌓여있던 밀린 프레임들이 타이머 해제 시점에 동일 권역 내의 각 노드에서 일제히 방출(Burst)되어 동시다발적 브로드캐스트 충돌 트래픽을 만들어낸다.

2.2 통신 및 관제 시스템에서의 장애 판별

MAC 프레임 차원에서의 혼잡 및 충돌이 발생하면 수신 측 라디오 모뎀 하드웨어 레벨에서는 CRC(Cyclic Redundancy Check) 또는 프레임 체크 시퀀스(FCS) 에러 카운트가 급성장한다. QGroundControl 및 PX4 측면에서는 지속적인 MAVLink 로스 메시지 경고(Packet Loss Warning)가 나타나며, 시리얼 포트 송신 버퍼 큐(Queue)를 해소하지 못해 Resource Exhaustion 형태의 지연(Latency) 증폭 에러가 관찰된다.

3. 채널 점유 실패(Channel Acquisition Failure)

MAC 프로토콜의 특성상 누군가 채널을 지속해서 점유하여 바쁘게 사용하고 있으면, 다른 대기 노드는 규칙에 따라 임의의 시간만큼 송출을 강제로 지연(Backoff)시켜야 한다. 데이터 과포화 트래픽 환경에서 이 지연이 끊임없이 반복되어, 특정 노드(기체 혹은 GCS)가 영영 송출 기회 메달을 얻지 못하고 장시간 벙어리 상태를 유지하는 현상을 채널 점유 실패라 한다.

3.1 발생 원인 및 특징

  1. 타 비행체 및 GCS에 의한 지속적 통신 대역 점유: 내가 설정한 것과 동일한 주파수 대역을 사용하는 미상의 다른 운용자의 고출력 텔레메트리나 GCS 인프라가 지속적으로 무거운 데이터를 전송하고 있을 때, 내 기체의 모뎀은 충돌 회피 알고리즘(CSMA) 규정에 순응하여 스스로 송출 메커니즘을 억제·마비시킨다. 즉, 통신 모듈 하드웨어도 정상이고 전파도 충분히 도달하지만, “에티켓(프로토콜 준수)“을 지키느라 데이터를 보내지 않아 어플리케이션 계층에서 스스로 하트비트 타임아웃을 유발하는 자승자박 상황이 발생한다.
  2. 비대칭 트래픽 스트림 상의 백오프 무한 루프: GCS에서 기체로 쏟아내는 제어 명령(Uplink)과 기체가 GCS로 응답하는 센서 데이터(Downlink) 간의 대역폭(Bandwidth Rate) 점유 설정이 극단적으로 기형적일 경우, 데이터 양이 많은 스트림 쪽이 채널 송출 권한을 독점해버리는 이기적 구조가 형성된다. 이로 인해 반대쪽 방향의 필수 제어 링크가 장시간 지연되다 결국 단절(Link Loss) 예외 상태에 돌입하게 된다.

4. PX4-Autopilot 및 GCS 관점의 조치 시사점

하드웨어 파손이 아닌 순수한 데이터 링크 계층 포화(Saturation) 및 혼잡으로 인한 관제 장애는, 소프트웨어 레벨의 파라미터 튜닝과 트래픽 관리를 통해 예방 및 완화할 수 있다.

  • MAVLink 스트리밍 레이트 동적 최적화: 채널 혼잡을 방지하기 위해 PX4-Autopilot의 내부 텔레메트리 레이트 프로파일(예: MAV_0_MODE, MAV_0_RATE 등) 파라미터를 보수적으로 설정하여 전체 패킷량을 제한해야 한다. 특히 무리를 지어 나는 군집 비행 혹은 장거리 관제에서는 불필요한 고속 피드백 데이터(Raw Sensor Data 등) 전송을 철저히 차단하고, 필수 제어 상태(Heartbeat 및 최소 Telemetry Status)만 주고받는 형태(Low Bandwidth / Telemetry Rate Limiting Mode)로 능동적으로 전이시켜 매체 점유율을 대폭 낮추어야 한다.
  • TDMA 기반 텔레메트리 적용 검토: CSMA 방식이 갖는 확률적 충돌 빈도의 한계를 근본적으로 돌파하기 위해, 시간을 미세하게 분할하여 각 무인기 노드에 고정된 전용 송신 슬롯(Time Slot)을 독립적으로 할당하는 시분할 다중 접속(TDMA) 방식의 산업용 통신망(예: 커스텀 SiK 펌웨어, 마이크로하드 등 상용 모뎀) 아키텍처 적용을 기획해야 한다. 이는 다수 기체 운용 시 충돌 없이 로버스트한 대역을 보장한다.
  • 지연 재전송(Retransmission) 백오프(Backoff) 튜닝: 지상 관제 명령어 전달을 담당하는 GCS 로직 레벨에서, 커맨드에 대한 응답(ACK)이 돌아오지 않는다고 하여 즉각 짧은 주기로 무자비하게 재전송(Retry)을 쏘아대는 행위는 패킷 충돌이 발생중인 MAC 계층의 혼잡을 치명적으로 역가중시킨다. GCS 개발자는 명령어 대기 시간초과 처리에 있어 지수적 백오프(Exponential Backoff) 기반의 여유 있는 재전송 딜레이 메커니즘을 GCS 커맨드 큐(Command Queue) 엔진에 필수적으로 탑재하여 잔여 네트워크에 가해지는 트래픽 과부하를 순차적으로 풀어주어야 한다.

5. 결론

기체와 지상국 간 물리적인 전자기파가 단단하게 살아있음(높은 RSSI)에도 불구하고 통신 단절 타임아웃이 거듭 반복된다면, 데이터 링크 계층 내부에서의 CSMA MAC 프레임 충돌 및 채널 획득 실패 증후군을 제1순위 점검 대상으로 의심해야 한다. PX4-Autopilot 시스템 엔지니어는 데이터 링크의 한계 수용량과 MAVLink 프로토콜의 트래픽 점유 특성을 완벽히 관통하고 이를 지능적으로 깎아내야(분배 및 제약) 극한의 라디오 혼잡 환경에서도 굴하지 않는 강건한(Robust) 지상 관제 통신망을 구축할 수 있다.
