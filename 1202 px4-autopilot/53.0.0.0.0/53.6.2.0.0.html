53.6.2 기체 상태 벡터(State Vector) 강제 폴링(Forced Polling) 및 관측기 보정

기체와 지상 관제 시스템(GCS, Ground Control Station) 간의 통신이 일시적으로 단절(Data Link Loss)되었다가 복구되는 시점에는 필연적으로 양측 고유 데이터 모델 간의 상태 불일치(State Inconsistency)가 발생한다. 지상 관제 시스템은 통신 두절 이전의 마지막 수신 데이터를 바탕으로 추측 항법(Dead Reckoning) 혹은 선형 외삽(Linear Extrapolation) 등을 수행하지만, 이는 실제 기체 역학과 외부 바람 교란(Wind Disturbance) 등에 의해 급격한 오차를 유발한다. 따라서 통신이 회복되는 즉시 수행되어야 하는 최우선 작업은 기체 상태 벡터(State Vector)의 강제 폴링(Forced Polling)과 이를 기반으로 한 GCS 측 관측기(Observer)의 오차 보정이다.

1. 상태 벡터 강제 폴링의 개념과 필요성

상태 벡터 강제 폴링이란 통신 복구 세션 수립 과정에서 GCS가 기체(PX4-Autopilot)에 현재의 핵심 비행 상태, 자세, 위치, 속도 등의 결정론적 상태(Deterministic State)를 즉각적이고 명시적으로 요구하는 절차를 말한다. 일반적인 MAVLink 통신 스트림(예: HEARTBEAT, ATTITUDE, GLOBAL_POSITION_INT)은 정해진 주기에 따라 기체에서 GCS로 브로드캐스트(Broadcast)되지만, 통신 복구 직후에는 이 주기적 메시지 발행을 수동적으로 기다리는 것만으로는 사용자 인터페이스 업데이트나 비상 제어 로직의 지연을 초래할 수 있다.

이러한 지연을 방지하기 위해 GCS는 MAV_CMD_REQUEST_MESSAGE와 같은 MAVLink 커맨드를 활용하여 기체의 필수 관측 데이터를 강제로(Forced) 즉시 송출하도록 지시한다. 이는 특히 QGroundControl(이하 QGC)의 Vehicle Manager 객체 내에서 링크 재연결 이벤트 트리거 시 호출되는 주요 복구 시퀀스(Recovery Sequence) 중 하나이다.

2. PX4-Autopilot과 QGroundControl 간의 MAVLink 기반 폴링 아키텍처

통신 복구 시, GCS(특히 QGC v4.2 이상 기준)는 기체가 발행하는 메시지 스트림 중 누락된 이력(History) 데이터 전체를 전송받는 것이 아니라, 현재 시점의 안정적인 상태 벡터를 확보하는 데 집중한다. 주요 폴링 대상 MAVLink 메시지는 다음과 같다.

  • HEARTBEAT (Msg ID: 0): 기체의 기본 커맨더 상태, 비행 모드(Flight Mode), 시동 상태(Arming State) 확인.
  • SYS_STATUS (Msg ID: 1): 배터리 잔량, 센서 상태 및 통신 링크 단절 동안 발생했을 수 있는 센서 결함(Sensor Fault) 폴링.
  • ATTITUDE (Msg ID: 30) 및 GLOBAL_POSITION_INT (Msg ID: 33): 현재 기체의 3차원 자세와 글로벌 위치(WGS84) 보정.

3. GCS 관측기 보정(Observer Correction) 메커니즘

GCS 내부에는 기체의 동역학적 궤적을 화면에 매끄럽게 도시하기 위한 관측기(일종의 시스템 상태 추적기)가 존재한다. Ardupilot의 Mission Planner나 QGroundControl 모두 차량 궤적 보간(Trajectory Interpolation) 및 UI 렌더링을 위한 버퍼를 운용한다. 통신이 두절된 구간 동안 GCS 관측기는 다음과 같은 행동 양상을 보인다.

  1. 예측 단계(Prediction Phase): 마지막 수신된 속도 벡터와 방향(Heading) 추정치를 기반으로 선형 예측을 수행하여 화면상 기체 아이콘을 이동시킨다.
  2. 임계치 기반 동결(Threshold Freezing): 통신 두절 지속 시간이 특정 시스템 임계값을 초과하면 불확실성이 기하급수적으로 커지므로, 기체 궤적 추정을 중단하고 UI 표출을 고정한다(통상 연결 끊김 경고 아이콘 표출).

해당 시각적 보정과 논리적 보정의 차이는 매우 중요하다. 통신이 부활하고 강제 폴링을 통해 실제 위치(X_{actual})가 유입되면, 임의로 추정된 GCS 내 위치(X_{estimated})와 실제 위치 간에 스텝 점프(Step Jump), 즉 단차가 형성된다. 무분별하게 차트에 보정 좌표를 즉각 반영하면 UI 상에서 기체 아이콘의 순간 이동(Teleportation) 현상이 나타나 조종자에게 혼란을 초래할 수 있다.

이를 방지하기 위해 GCS 소프트웨어는 시각적 관측기(Visual Observer)단에 부드러운 스무딩 필터(Smoothing Filter)나 동적 보정 곡선을 적용해 0.5초에서 1초 내외로 기체 아이콘을 실제 궤적으로 동기화시킨다. 하지만 내부 논리 제어 모델(Logical Control Model) 레벨에서는 폴링된 데이터로 변환을 즉각 하드 리셋(Hard Reset)하여, GCS가 누적 오차가 섞인 상태에서 잘못된 횡단 거리나 Goto 명령 알고리즘을 계산하는 것을 엄격히 차단한다.

4. PX4 시스템 내부(uORB) 메커니즘 관점

강제 폴링 커맨드가 MAVLink 패킷으로 기체에 도달하면, 펌웨어 내(PX4 Firmware v1.13 ~ v1.14 기준) mavlink_receiver 모듈에서 이를 디코딩한다. 패킷 수신부는 즉각 uORB(Micro Object Request Broker) 메시지 버스에 접근하여 대응되는 핵심 데이터 토픽(vehicle_local_position, vehicle_global_position, vehicle_attitude 등)의 인스턴스를 즉각 복사(Fetch)한다. 그 후 이를 MAVLink 인코더를 거쳐 송출 큐(Tx Queue)의 최우선 순위로 스케줄링하여 지상으로 전송한다.

이 설계 방식은 단순히 주기적으로 브로드캐스트 전송을 기다리는 루틴보다 마이크로컨트롤러 자원 및 네트워크 대조 오버헤드를 유발하지만, 시간 축 상으로 가장 지연이 적고 확정적인 시스템 현황을 보장해야 하는 GCS의 관리를 용이하게 돕는 공학적 실리가 우선된다.

5. Ardupilot 관제 체계와의 구조적 차이

Ardupilot 관제 시스템 내에서는 사용자화된 설정에 따라 통신 복구 시, 운영자가 수동으로 파라미터나 상태 새로 고침(Refresh)을 요구해야 완벽히 동기화 되는 일부 상황이 존재할 수 있다. 반대로 PX4 기반의 QGroundControl 통합 플랫폼에서는 통신 링크 객체(LinkManager)의 상태 머신 전이를 감지하는 즉시, GCS가 백그라운드 스레드에서 비동기적(Asynchronous)으로 Vehicle 관리자 하위의 복구 시퀀스를 순차적으로 자동 호출한다. 운영자의 인지 부하 제어 측면이나 무인기의 신뢰도 향상 측면에서 이러한 PX4 와일드 체인의 관측기 보정 자동화 로직은 시스템의 안정성을 본질적으로 끌어올리는 효과를 발생시킨다.
