53.3.2 데이터 링크 단절 임계 파라미터(NAV_DLL_ACT, COM_DL_LOSS_T)의 민감도 및 강건성 분석

본 절에서는 PX4-Autopilot 시스템에서 데이터 링크 단절(Data Link Loss)을 감지하고 대응 방안을 결정하는 핵심 파라미터인 NAV_DLL_ACT와 COM_DL_LOSS_T의 동작 원리를 분석한다. 또한, 이들 파라미터의 설정값에 따른 시스템의 민감도(Sensitivity)와 강건성(Robustness) 간의 상충 관계(Trade-off)를 제어 공학적 관점 및 비행 안전성 측면에서 심도 있게 다룬다.

1. 데이터 링크 단절 관련 핵심 파라미터 개요

지상 관제 시스템(GCS, QGroundControl 등)과 기체 간의 MAVLink 통신은 무선 환경의 특성상 간섭(Interference)이나 장애물에 의해 일시적으로 지연되거나 끊길 수 있다. PX4-Autopilot은 이러한 상황을 예외 상태로 규정하고 페일세이프(Failsafe)를 가동하기 위해 다음 두 가지 주요 파라미터를 제공한다.

  • COM_DL_LOSS_T (Communication Data Link Loss Timeout):
  • 정의: 지상 관제 시스템으로부터 유효한 MAVLink Heartbeat 메시지가 수신되지 않는 시간을 초(Second) 단위로 설정하는 임계값(Threshold)이다.
  • 역할: 이 파라미터에 설정된 시간 이상 MAVLink 메시지 수신이 없으면, 펌웨어의 Commander 모듈은 공식적으로 ‘통신 단절(Data Link Loss)’ 상태를 선언한다.
  • NAV_DLL_ACT (Navigation Data Link Loss Action):
  • 정의: 제어기(Commander)가 통신 단절을 감지했을 때 취해야 할 페일세이프 강제 행동(Failsafe Action)을 정의한다.
  • 옵션 예시:
  • 0: Disabled (무시)
  • 1: Hold (현재 위치에서 호버링/대기)
  • 2: Return mode (RTL, 지정된 홈/이륙 위치나 랠리 포인트로 복귀) - 가장 보편적인 설정
  • 3: Land mode (현재 위치에서 착륙)
  • 6: Terminate (비행 종료 및 모터 정지 - 매우 위험하므로 특수 목적에만 사용)

2. 민감도(Sensitivity)와 강건성(Robustness)의 제어 공학적 분석

데이터 링크 단절의 판정 기준인 COM_DL_LOSS_T 값의 설정은 드론의 통신 제어 시스템의 강건성과 민감도를 결정짓는 가장 중요한 요소이다. 임계 파라미터를 어떻게 튜닝하느냐에 따라 비행체의 생존성 및 임무 수행 능력이 극명하게 달라진다.

  • 높은 민감도(High Sensitivity) 구간: (COM_DL_LOSS_T를 짧게 설정, 예: 1~3초)
  • 장점: 통지가 단절되었을 때 즉각적으로 페일세이프 동작(예: Return)을 트리거하므로, 기체가 통제 불능 상태로 멀리 벗어나는 것(Fly-away)을 신속하게 방지할 수 있다. 장애물이 많은 도심지 비행이나 정밀 제어가 필요한 환경에 적합하다.
  • 단점 (강건성 저하): 안테나 방향 전환이나 일시적인 전파 간섭으로 인한 짧은 통신 끊김(Ping drop)에도 시스템이 불필요하게 복귀 모드(RTL)로 전환된다. 이는 임무(Mission) 수행의 연속성을 심각하게 해치는 요인이 된다.
  • 높은 강건성(High Robustness) 구간: (COM_DL_LOSS_T를 길게 설정, 예: 10초 이상)
  • 장점: 일시적인 통신 노이즈나 음영 지역을 지나갈 때 발생하는 짧은 단절 구간을 무시하고(Tolerance) 원래의 임무를 묵묵히 수행한다. 노이즈 내성이 높아 장거리 자율 비행 임무(Waypoint Mission)에 적합 단하다.
  • 단점 (민감도 저하): 실제 하드웨어 고장이나 영구적인 통신 두절 상황이 발생했을 때, 페일세이프가 발동하기 전까지 10초 이상 현재 상태(예: 전진 비행)를 유지하므로, 기체가 예상치 못한 위치로 이탈하거나 2차 사고를 유발할 위험이 비례하여 증가한다.

3. Ardupilot의 통신 단절 제어 철학과의 비교

Ardupilot은 PX4의 COM_DL_LOSS_T, NAV_DLL_ACT에 대응하는 파라미터로 FS_GCS_ENABLE를 사용한다.

  • Ardupilot (FS_GCS_ENABLE): 단순히 GCS 통신 단절 기능의 On/Off뿐만 아니라, 연결이 끊겼을 때의 액션(RTL, SmartRTL, Continue with Mission 등)을 하나의 파라미터에 복합적으로 할당하는 방식을 취한다. 타임아웃은 기본적으로 1.5초 등 고정적이거나 내부 로직에 의해 강하게 결합되어 있다.
  • PX4-Autopilot: 상태(State, COM_DL_LOSS_T)의 평가와 행동(Action, NAV_DLL_ACT) 결정을 명확히 분리하여 구조화하였다. 이는 상태 머신(State Machine) 아키텍처 관점에서 모듈화가 더 잘 되어 있으며, QGroundControl과 같은 관제 소프트웨어에서 복합적인 안전 트리(Safety Tree)를 구성하기에 유리한 장점이 있다.

5. 소스 코드 레벨 분석 (Source Code Analysis)

PX4의 핵심 상태 제어 모듈인 Commander (src/modules/commander/Commander.cpp 및 관련된 state_machine_helper.cpp) 에서는 uORB 메시지 시스템을 통해 MAVLink 수신 노드(Telemetry)로부터의 telemetry_status 토픽을 구독(Subscribe)한다.

  1. 시간 델타 연산: 내부 루프에서는 주기적으로 현재 시스템 시간(hrt_absolute_time())과 telemetry_status에서 업데이트된 heartbeat_time 간의 차이를 계산한다.
  2. 임계 파라미터 로드: COM_DL_LOSS_T 파라미터 값을 마이크로초(Microseconds) 단위로 변환하여 두 시간의 차이와 비교한다.
  3. 상태 천이(State Transition): 차이값이 임계값을 초과하면, status.data_link_lost 플래그를 true로 설정한다.
  4. 페일세이프 핸들러 호출: failsafe.cpp 등에서 데이터 링크 상태를 평가하고, NAV_DLL_ACT 에 명시된 동작으로 비행 모드를 강제 변환(Flight Mode Switch) 시킨다.

설정의 강건성을 극대화하기 위해, 실제 산업용 에이전트 개발에서는 COM_DL_LOSS_T를 단순히 고정된 값으로 두지 않고 환경의 노이즈 밀도(Noise Density)나 기체의 속도 벡터를 기반으로 동적으로 조절하는 적응형 임계값(Adaptive Threshold) 연구도 다수 진행되고 있다. 이 경우 통계적 학습 기법을 도입하여 False Positive(오알람) 와 False Negative(미탐지)를 최소화한다.
