## 0.1 통신 복구 시 비동기 분산 시스템 간의 상태 동기화(State Synchronization) 메커니즘

무인항공기(UAV)와 지상 관제 시스템(GCS)은 물리적으로 분리된 두 개의 독립적인 컴퓨팅 노드로 구성된 비동기 분산 시스템(Asynchronous Distributed System)이다. MAVLink와 같은 무선 통신 링크를 통해 상태를 교환하는 이 아키텍처에서, 통신 단절(Data Link Loss) 후 통신이 복구되었을 때 가장 중요한 과제는 두 시스템 간의 상태 불일치(State Inconsistency)를 해소하는 조화(Reconciliation) 및 동기화(Synchronization) 작업이다. 본 절에서는 통신 복구 시 발생하는 PX4-Autopilot과 QGroundControl 간의 상태 동기화 메커니즘을 심도 있게 분석한다.

0.1.1 비동기 분산 시스템에서의 분리된 상태(Diverged State) 문제

통신이 단절된 동안 PX4-Autopilot 내부의 비행 제어기(비행 관리자 컴퓨터, Flight Management Unit)는 자체적인 자율 비행 로직(예: Return to Launch, Hold, Land 등)에 따라 기체의 상태(State)를 지속적으로 변경한다. 예를 들어, 통신이 끊어졌을 때 기체의 모드가 “Mission“에서 페일세이프(Failsafe)에 의해 “Return” 모드로 자동 변경되었을 수 있다.

하지만 QGroundControl(GCS)은 통신 단절 직전의 마지막 수신 상태만을 기억하고 있다. 따라서 통신이 복구되는 시점(Connection Restored)에 GCS 로컬 메모리에 캐싱(Caching)된 기체 상태 벡터와 실제 기체의 상태 벡터 간에는 차이가 발생한다. 이를 즉각적으로 동기화하지 않으면, 사용자가 잘못된 상황 인지(Situational Awareness)를 바탕으로 위험한 제어 명령을 내릴 우려가 있다.

0.1.2 MAVLink 데이터 스트림 기반의 상태 동기화 로직

통신이 물리적으로 복구(RF 링크 재연결 또는 TCP/UDP 소켓 재수립)되면, PX4-Autopilot은 주기적으로 브로드캐스팅(Broadcasting)하던 고빈도 및 저빈도 텔레메트리(Telemetry) 패킷을 다시 GCS로 전송하기 시작한다. QGroundControl은 이러한 수신 데이터 스트림을 통해 Vehicle 객체의 속성을 순차적으로 갱신하여 동기화를 수행한다.

  1. 최상위 상태 동기화 (HEARTBEAT): 기체의 현재 비행 모드(Flight Mode), 시동 상태(Armed/Disarmed), MAVLink 컴포넌트 시스템 ID 등의 최상위 상태는 1Hz 주기로 송신되는 HEARTBEAT 메시지를 통해 즉시 동기화된다. GCS는 첫 번째 HEARTBEAT를 수신하자마자 통신 복구를 선언하고 UI 비활성화(Invalidation)를 해제한다.
  2. 센서 및 전력 상태 동기화 (SYS_STATUS, BATTERY_STATUS): 배터리 잔량 소모도, CPU 부하, 센서 건전성(Health) 등은 해당 메시지가 수신되는 즉시 GCS의 상태 변수로 덮어쓰기(Overwrite)된다.
  3. 위치 및 자세 동기화 (GLOBAL_POSITION_INT, ATTITUDE): 고주파수(보통 10~50Hz)로 스트리밍되는 기체 동역학 데이터는 칼만 필터(EKF2) 추정치와 함께 GCS의 3D 뷰와 지도를 가장 빠르고 자연스럽게 갱신한다.

0.1.3 양방향 핸드셰이크가 필요한 지연 동기화(Lazy Synchronization)

센서 데이터와 같은 단방향 스트림(Unidirectional Stream)은 수신 즉시 상태를 덮어쓰면 되지만, 파라미터(Parameter), 임무(Mission), 지오펜스(Geofence)와 같이 크기가 크고 무결성이 보장되어야 하는 데이터 모델은 단순 브로드캐스트만으로는 동기화할 수 없다. 이를 위해 요청-응답(Request-Reply) 기반의 트랜잭션 모델이 활성화된다.

    Note over GCS, PX4: 통신 물리적 복구 (Link Restored)
    
    PX4->>GCS: HEARTBEAT (모드=Return, 상태=Armed)
    GCS->>GCS: 통신 복구 감지 및 UI 활성화
    GCS->>GCS: 기체 모드 불일치 해소 (Mission -> Return 갱신)
    
    PX4->>GCS: SYS_STATUS, ATTITUDE 등 연속 수신
    GCS->>GCS: 텔레메트리 데이터 UI 동기화
    
    GCS->>PX4: MISSION_REQUEST_LIST
    PX4-->>GCS: MISSION_COUNT
    GCS->>PX4: PARAM_REQUEST_LIST (선택적)
    PX4-->>GCS: PARAM_VALUE (다중 패킷)
    
    Note over GCS: 모든 객체 모델 동기화 완료

[도표 설명] 통신 복구 직후 브로드캐스트 기반의 “즉시 동기화“와 프로토콜 기반의 “지연(Lazy) 동기화“가 혼합된 프로세스 플로우. GCS가 주도적으로 누락되었을 가능성이 있는 상태량을 쿼리(Query)한다.

특히 임무 진행 상태 동기화에 주의해야 한다. 통신이 단절된 동안 기체가 몇 번째 웨이포인트(Waypoint)를 통과했는지 GCS는 알지 못한다. 통신이 복구되면 기체는 MISSION_CURRENT 메시지를 전송하여 현재 활성화된 웨이포인트 시퀀스 번호를 GCS에 알리고, GCS는 화면에 그려진 비행 궤적과 현재 활성 지점을 이에 맞춰 동기화한다.

0.1.4 설계 시 고려사항: 충돌 해결(Conflict Resolution) 메커니즘

비동기 분산 시스템에서 동기화 중 가장 큰 딜레마는 충돌 해결(Conflict Resolution)이다. 예를 들어 통신이 단절된 동안 GCS 운영자가 오프라인 상태에서 새로운 임무(Mission) 계획을 편집하고 있었고, 기체는 기존 임무를 계속 수행 중이었다면, 통신 복구 시 누구의 상태를 “정답(진실의 공급원, Source of Truth)“으로 삼을 것인가?

제어 공학적 안전성(Safety) 기준에 입각하여, 비행 중인 기체 제어기(PX4-Autopilot) 내부의 RAM에 적재된 상태가 최우선적인 진실의 공급원(Primary Source of Truth)으로 작용한다. GCS에 캐시된 데이터는 플러시(Flush)되고 기체로부터 다운로드된 데이터로 덮어씌워져야 한다. 만약 GCS 운영자가 로컬에서 편집한 사항을 기체에 반영하고자 한다면, 동기화가 완전히 끝난 직후 명시적 업데이트(Explicit Upload) 명령 트랜잭션을 다시 시작해야 한다.

이러한 단방향 마스터-슬레이브(Master-Slave) 동기화 철학은 통신 재결합 과정에서 레이스 컨디션(Race Condition)을 방지하고 상태 왜곡에 의한 기체 추락이나 비정상 제어를 예방하는 핵심 메커니즘이다.
