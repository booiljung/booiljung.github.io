53.7.3 다중 경로(Multipath) 수신 환경의 패킷 지연 편차 보정 및 중복 패킷 필터링

다중 경로(Multipath) 라우팅을 활용한 이중화(Redundancy) 제어 네트워크 구성은 무인항공기 통신 시스템의 돌발 상황 생존성을 비약적으로 높여 주지만, 소프트웨어 시스템 아키텍처 단에 패킷 분산 및 병합(Gathering) 처리에 대한 지대한 연산 복잡성을 부가하게 된다. 동일한 PX4 기체에서 듀얼 링크 설정으로 동시에 브로드캐스트 전송된 MAVLink 메시지가 LTE와 RF 텔레메트리를 각각 경유하여 GCS에 수신될 때, 두 이기종 네트워크 간의 극단적인 비대칭적 물리적 전파 속도 및 라우팅 병목 지연 시간(Latency)으로 인하여 애플리케이션 수신 단에는 극심한 패킷 지연 편차(Jitter) 현상이 발생한다.

이 절에서는 이러한 복잡한 다중 경로 수신 환경에서 QGroundControl과 PX4-Autopilot이 어떻게 비동기적 지연 편차 시간 이탈도를 시스템 레벨에서 보정하고, 불필요한 쓰레기성 중복 패킷(Duplicate Packet)을 최적화된 자료구조를 통해 O(1) 복잡도로 필터링하여 프론트엔드 UI 렌더링 성능 최적화 및 제어 무결성(Integrity)을 성공적으로 지켜내는지 그 핵심 동작 원리를 기술한다.

1. 이기종 망 간 패킷 지연 편차(Jitter)의 발생 원인과 파급력

PX4-Autopilot 비행 제어기 시스템은 MAVLink 라이브러리 인터페이스를 통하여 기체의 현재 3차원 자세각(Attitude), 고정밀 글로벌 위치(GPS Position), 비행 모드 시스템 상태(System Status) 등의 텔레메트리 정보를 GCS 소프트웨어로 실시간 브로드캐스트(Broadcast)한다. 연결 통신 신뢰성(Fault Tolerance)을 극대화하기 위해 다중 링크(Multi-link)를 구성하여 유지할 경우, PX4 내부의 백그라운드 구동 태스크인 mavlink_main 모듈은 서로 다른 물리 H/W 인터페이스 포트(예: ttyS1, ttyS2, 혹은 Companion Computer 기반 이더넷 UDP)를 통해 완벽히 동일한 데이터 구조체 페이로드(Payload)를 지닌 쌍둥이 메시지를 동시에 발송 처리한다.

그러나 이 메시지 패킷들이 지상의 GCS(QGroundControl) 컴퓨터 네트워크 카드에 최종 도달하는 절대적 시간은 물리적 인프라망의 상태에 따라 짧게는 수십 밀리초(ms)에서 길게는 무선 혼잡 시 수 초(Seconds) 단위까지 편차가 발생하게 된다.

  • 초고속 광대역망 (LTE/5G): 기본 지연 레이턴시가 50ms 미만으로 극도로 낮지만, 기체가 비행 중 기지국 핸드오버(Handover) 섹터 경계 과정을 거치거나 망 코어 부하가 생길 때 드물게 일시적인 레이턴시 스파이크(Spike)가 튈 수 있는 단점이 있다.
  • 저속 라디오망 (RF Telemetry): 전송 대역폭 속도는 매우 느리나 전파 직진성에 기반을 둔 연결 지연 속도의 주기성과 편차가 비교적 일관된 베이스라인(Baseline)을 지향한다.

결과적으로 GCS 애플리케이션인 QGroundControl 계층 라우팅 구조에서는, 각 패킷이 도달하는 시간(Time of Arrival, ToA)의 순서가 원래의 생성 순서와 완전히 뒤섞여 버리는 역전 현상(Out-of-Order Delivery) 및 동일 프레임이 중복하여 수시로 도달하는 현상을 필연적으로 겪게 된다. GCS 내부 스레드 로직에서 이 패킷 도달의 난립 현상과 지연 편차를 제어하여 필터링 및 순서를 통제하지 못하면, 과거의 비행 기체 위치 데이터 렌더링이 뒤늦게 도착해 이미 처리된 0.5초 전의 최신 위치 그래픽을 다시 과거로 덮어씌어버려 화면상 기체가 앞뒤로 진동하는 일명 ‘타임 워프(Time Warp)’ 부작용과 심각한 UI 티어링 및 프레임 드랍을 유발하게 된다.

2. QGroundControl 백엔드의 중복 패킷 필터링(Filtering) 구조

MAVLink v2 메시지 패킷 규격 헤더 영역에는 메시지를 보낸 고유의 시스템과 컴포넌트 아키텍처 식별을 위한 System ID, Component ID는 물론이고, 발송되는 메시지의 순차적 타임라인 순서를 특정할 수 있는 1바이트 크기(0~255 값 순환 방식)의 시퀀스 넘버(Sequence Number, 통상 seq 필드) 헤더 필드를 필수적으로 포함하고 있다. 이기종 다중 경로 환경에서 GCS가 상이한 두 개의 물리 통신 채널을 통해 본질적으로 동일한 데이터를 수신하였다면, 이들은 내부 내용물뿐만 아니라 완벽하게 일치하는 패킷 헤더와 동일한 seq를 가지게 된다.

QGroundControl(버전 4.x 베이스라인 기준)의 C++ 백엔드 코어 통신 모듈(MAVLinkProtocol.cc 클래스 및 링크 인터페이스 인스턴스)은 시스템 블로킹 없이 패킷을 즉각 분류하기 위해 아래 도식과 같은 방어적인 필터링 아키텍처 로직을 거쳐 중복 패킷 처리를 무산(Drop)시킨다.

    LTE-->>QGC: (고속 수신 링크) MAVLink MSG (seq:42) 도착
    QGC->>QGC: 상태 추적 메모리 테이블 기록: (Sys=1, Comp=1) -> 최근수신 seq=42 업데이트
    QGC->>QGC: QML UI 응용 계층으로 신규 패킷 데이터 큐잉 Forwarding
    Note over QGC, RF: (동일 패킷에 대한 수백 ms 지연 표출)
    RF-->>QGC: (저속 수신 링크) MAVLink MSG (seq:42) 뒤늦게 도착
    QGC->>QGC: 해시 테이블 고속 조회: 기존 기록된 최근 수신완료 seq(42이상)와 정합 비교
    QGC->>QGC: 폐기 판정 알고리즘 작동: 중복된 과거 패킷 즉시 폐기(Drop, Filtering) 적용

이 고속 필터링 알고리즘에서 QGroundControl은 동일 데이터를 재처리하는 막대한 이중 루프(Double Loop) CPU 오버헤드를 막기 위해, 각각 독립된 (SysID, CompID) 조합 혹은 특정 식별 스트림에 부여된 전역 ID에 조응하여 가장 최근에 수신 성공한 seq 상태값을 해시 테이블(Hash Table) 알고리즘 구조체나, 혹은 상수 시간 O(1) 복잡도를 지닌 고정 슬롯 원형 버퍼(Circular Buffer) 상태 레지스터에 기록해둔다.

이후 뒤늦게 속도가 느린 다른 경로채널(Multipath)에서 힘겹게 수신된 패킷 프레임 안의 seq 데이터를 메모리 테이블을 조회 비교하여 이미 기록 성공 처리된 seq 값과 일치하거나, 오히려 숫자가 역행하는 이전 낡은 시퀀스일 경우에는 이를 “지연 수신된 가치 없는 중복 플로(Flow)” 프레임으로 가차 없이 간주한다. 곧장 메모리에서 해제 해버리며 상위 응용 계층 처리기 이벤트 버스로 절대 올려보내지 않는다. 이러한 구조적 링 필터링 설계는 QGroundControl 시리얼 파서 단 스레드의 백그라운드 CPU 점유율 부하를 획기적으로 낮추어 주며 동시망 사용 시 발생 가능한 치명적인 UI 성능 악화 및 지연 요소를 뿌리째 근본적으로 차단하는 훌륭한 엔지니어링 사례이다.

3. 다중 채널 패킷 도달 편차 보상을 위한 큐잉(Queueing)과 타임아웃 전략

메시지 도달 시간의 지연 편차 불확실성이 높은 필드 전파 환경에서 극단적으로 앞서 발송된 시퀀스의 낡은 패킷(예: seq 42)이 RF 링크 병목 구간에 갇혀 지연되어 가고 있는데 속도가 번개 같은 LTE 링크로는 그다음 최신 시퀀스의 패킷(예: seq 43)이 지상국 안테나에 먼저 도착해 버릴 수 있다. 사실 MAVLink 통신 코어 라이브러리는 UDP/IP 기반 메커니즘과 유사하게 전송 순서를 강제하지 않으며, 손실된 패킷의 재조립 및 순서 보장을 하드웨어 단에서 전혀 보장하지 않는 데이터그램(Datagram) 고유의 성격을 갖는다.

물리적 전송 경로 지연 보정을 엄격하게 요구하는 프로토콜 기반의 상태 미션 동작(예: Waypoint 업로드/다운로드 Transaction Sequence 등)의 경우에 한해서, QGroundControl은 무작정 패킷을 유실 처리하는 대신 Vehicle 엔티티 객체의 내부 매니저 모델(MissionManager 모듈 등) 단에서 논리적인 타임아웃 추적 윈도우(Timeout Windowing) 공간 기법을 두어 응답 패킷의 전송 속도 분산 여부와 손실 간극을 시스템 레벨에서 보상해 주는 과정을 밟는다.

  1. 실시간 모니터링 버퍼 룰의 비적용 원칙 (Telemetry Data): 기본적으로 텔레메트리(ATTITUDE, GLOBAL_POSITION_INT 등 단순 정보 보고용 데이터) 메시지들에 대해서는 과거의 누락 패킷을 기다릴 이유가 전무하므로 그 최신성(Freshness Data) 확보가 절대적이다. 따라서 윈도우나 큐잉(Queueing) 적용 없이 낡은 데이터는 즉시 버리고 앞선 번호 위주로 선취하는 실시간 반응 처리가 지배적으로 적용된다.
  2. RTT(Round-Trip Time) 기반 동적 타임아웃 재연산 추론: 신뢰성 높은 미션 전송의 경우 PING 라우팅 테스트와 같은 네트워크 하트비트 메시지를 통해 LTE 데이터망과 RF 망의 각각 왕복 응답 지연시간(RTT)을 상시 측정 기록한다. 그런 다음, 이 두 링크 네트워크 간 도출되는 절대 델타(Delta) 시간 편차폭 만큼 사후 보정하여 이륙 및 모드 변경 명령어(COMMAND_LONG) 발송 후 대기하는 기체의 확인 응답(ACK) 기대 유효 시간(Timeout Limits)을 소프트웨어적으로 추가 연장 조절 설정해 준다. 이는 성급한 통신 실패 오판정과 잘못된 커맨드 과다 재전송(Redundant Retry) 폭발로 인해 발생하는 양측 통신 모듈 블로킹(Blocking)과 치명적 마비 상태를 능동적으로 예방하는 조치이다.

결론적으로 이와 같은 GCS 백엔드의 중첩 패킷 필터링 구조 철학과 패킷 다중 수신 편차(Delay Jitter) 보상 대응 전략은, 수많은 이종 대역 주파수와 물리 네트워크 토폴로지가 얽히고설켜 운용되는 현대 최첨단 무인가시권밖(BVLOS) 드론 군집 관제 솔루션에 있어 중복 데이터로부터 보호되는 결함 없는 쾌적한 비행 상태 인터페이스 UI를 지속 유지토록 하는 강력한 중핵 아키텍처 보호 역할을 전담하여 수행한다.
