53.3.3.2 ROS2 Micro XRCE-DDS 에이전트 브릿지 단절 시 심장박동 토픽(Heartbeat Topic) 시간 초과에 따른 제어 권한 반환(Fallback) 메커니즘

0.1 개요 및 시스템 배경

PX4-Autopilot 생태계가 점차 단일 마이크로컨트롤러(MCU) 중심의 제어에서 고성능 컴패니언 컴퓨터(Companion Computer) 기반 분산 제어 환경으로 이행함에 따라, ROS 2(Robot Operating System 2)와의 네이티브 연동을 위한 Micro XRCE-DDS(eXtremely Resource Constrained Environments DDS) 에이전트/클라이언트 아키텍처가 표준으로 자리잡았다.

이 브릿지(Bridge) 환경에서 ROS 2 오프보드(Offboard) 노드는 조종기를 대체하여 기체의 궤적(Trajectory), 자세(Attitude), 추력(Thrust)을 실시간으로 제어할 수 있는 막강한 권한을 가진다. 그러나 네트워크 인터페이스(UART, UDP, USB 등)의 결함, 컴패니언 컴퓨터의 과부하, 에이전트 데몬 프로세스의 비정상 종료 등으로 인해 브릿지 단절(Bridge Disconnection)이 발생할 수 있다. PX4는 이러한 치명적인 상황에서 자율 비행을 중단하고 기체의 시스템 안전을 도모하기 위해 심장박동(Heartbeat) 토픽 시간 초과 유무를 주기적으로 감시하여 제어 권한을 자체 Commander 모듈로 기꺼이 강제 반환(Fallback)하는 안전 메커니즘을 내장하고 있다.

0.2 Micro XRCE-DDS의 Heartbeat 기반 통신 감지 아키텍처

0.2.1 OffboardControlMode를 통한 심장박동 메커니즘

QGroundControl이 발행하는 전통적인 MAVLink HEARTBEAT 메시지와 차별화되어, ROS 2-PX4 브릿지 환경에서는 OffboardControlMode uORB 토픽이 오프보드 제어 상태를 알리는 실질적인 심장박동(Heartbeat) 역할을 수행한다.

ROS 2 상의 오프보드 제어 노드는 제어 명령(Position, Velocity, Attitude 등)을 전송하기에 앞서, 제어 루프의 동일한 주파수(최소 2Hz 이상 권장, 보통 10Hz~50Hz)로 주기적인 OffboardControlMode 메시지를 퍼블리시해야 한다. 이 토픽 내부에는 현재 요구하는 제어 타입 깃발(예: position, velocity, acceleration, attitude, body_rate 필드의 불리언(Boolean) 값 표시)이 포장(Packing)되어 있다.

0.2.2 uXRCE-DDS 클라이언트의 시계열 동기화

PX4 펌웨어 내부에 탑재된 uxrce_dds_client 로컬 데몬 모듈은 ROS 2 에이전트로부터 전달받은 DDS 패킷을 uORB 토픽으로 매핑(Mapping)시킨다. 브릿지가 끊어지는 순간, uORB 파이프라인에서 OffboardControlMode 주제(Topic) 갱신은 즉각적으로 멈춘다. 즉, uORB 토픽의 마지막 타임스탬프(Timestamp)와 PX4 시스템 시간 hrt_absolute_time() 간의 차이(delta_time)를 계산하여 시간 초과(Timeout)를 분석한다.

0.3 에이전트 단절에 따른 타임아웃 판별 및 Commander 모듈 작동

PX4 Commander 모듈은 다음과 같은 파라미터 및 시간 임계값을 통해 통신 단절을 판별한다.

  • 파라미터 COM_OF_LOSS_T (Offboard Loss Timeout): 오프보드 제어 메시지의 수신 단절을 유효한 것으로 판단할 때까지 대기하는 초(Second) 단위의 임계값 (기본값 설정은 기체 설정에 따라 상이함).
  • 파라미터 COM_OBL_RC_ACT (Offboard Loss Action with RC): 조종기(RC)가 연결된 상황에서 오프보드 통신 상실 시 행동 지침 (RC로 권한 이양, RTL, Land 등).
  • 파라미터 COM_OBL_ACT (Offboard Loss Action without RC): 조종기 연결 없이 순수 자율 비행 상태에서 오프보드 데이터 상실 시 행동 지침.
// PX4 Commander 코드의 Offboard 타임아웃 판정 논리 예시 (가상 의사 코드 구조)
hrt_abstime now = hrt_absolute_time();
if (now - offboard_control_mode_pub.timestamp > (COM_OF_LOSS_T * 1000000)) {
    // Heartbeat 타임아웃 발생
    offboard_lost = true;
    fallback_to_failsafe_mode();
}

오류 시간(delta_time)이 COM_OF_LOSS_T를 초과하면 상태 기계(State Machine)는 더 이상 외부 노드(ROS 2)의 실시간 제어 능력을 신뢰할 수 없다고 판단하고 오프보드 제어 상실(Offboard Control Loss) 이벤트를 발생시킨다.

0.4 제어 권한 반환(Fallback) 메커니즘과 동작 원리

이벤트가 트리거되면 컴패니언 컴퓨터가 보유하고 있던 제어 프레임 워크의 권한 지위가 무효화(Denial)되고 권한은 PX4 스택의 내부 논리로 즉시 이관된다. 이는 다중 체계 시스템에서 통제권을 일원화하여 충돌을 막는 필수적인 Fallback 아키텍처이다.

  1. 내부 추종 궤적(Setpoints)의 폐기: PX4의 멀티플렉서 모듈(FlightTaskOffboard)은 외부 ROS 2 노드가 보낸 더러워진(Stale) TrajectorySetpoint를 더 이상 참조하지 않으며, 위치 오차 적분기(Integrator) 값을 초기화 리셋하여 미분항 스파이크(Derivative Kick)를 능동적으로 억제한다.
  2. 비상 거동 모드(Failsafe Mode)로의 FSM 천이: 사전에 COM_OBL_ACT 파라미터에 정의된 규칙에 따라 Hold(제자리 체공), Return(이륙 지점 복귀), 혹은 Land(제자리 즉시 착륙) 상태로 시스템 모드(Nav State)를 안전하게 변경한다. 만약 활성화된 수동 조종기(Radio Control) 링크가 존재한다면 파라미터 셋에 맞추어 Position 모드 등으로 천이시켜 사용자 개입(Manual Override)을 허용한다.

0.6 결론 기반 통찰

Ardupilot 대비 PX4-Autopilot의 가장 큰 설계적 강점 중 하나는 uORB와 네이티브하게 통합된 uXRCE-DDS 브릿지를 탑재한다는 것이다. 위에서 살펴본 Heartbeat 토픽 시간 초과 연계 제어 권한 반환(Fallback) 메커니즘은 로보틱스 관점에서 컴패니언 컴퓨터가 비행 안정성을 책임질 수 없는 상태가 도달하더라도 드론이라는 물리적 에이전트 스스로가 시스템 생존성을 유지하는 탈중앙화적(Decentralized) 예외 처리 모범 사례(Best Practice)에 해당한다. 사용자는 COM_OF_LOSS_T 파라미터를 적절히 식별하고 운영 환경(Operating Environment)에 맞춰 튜닝하여 제어 상실 시나리오를 탄탄하게 설계해야 한다.
