53.3.3 외부 컴패니언 컴퓨터(Companion Computer) 통신 단절(COM_OF_LOSS_T) 감지 및 분산 제어 시스템 안전 정지

본 절에서는 현대 무인항공기(UAV) 아키텍처에서 고수준 자율 비행(High-level Autonomy)을 담당하는 외부 컴패니언 컴퓨터(Companion Computer)와 비행 제어기(PX4-Autopilot) 간의 데이터 링크 단절을 감지하고, 분산 제어 시스템(Distributed Control System) 환경에서 기체를 안전하게 정지(Safe Stop) 또는 복구시키는 메커니즘을 살펴본다. 핵심 파라미터인 COM_OF_LOSS_T의 동작 원리와 오프보드(Offboard) 제어 모드의 안전 철학을 중점적으로 분석한다.

1. 컴패니언 컴퓨터 분산 제어 아키텍처와 Offboard 모드

PX4-Autopilot 기반 드론에서 복잡한 컴퓨터 비전(Computer Vision) 인식, 라이다(LiDAR) 기반 장애물 회피(Obstacle Avoidance), ROS/ROS2 경로 계획(Path Planning) 알고리즘 등은 컴퓨팅 파워의 한계로 인해 비행 제어기(Flight Controller) 내부에서 직접 연산되지 않는다. 대신, 라즈베리 파이(Raspberry Pi)나 엔비디아 젯슨(NVIDIA Jetson)과 같은 컴패니언 컴퓨터(Companion Computer) 에서 연산된 후, MAVLink 통신(또는 uXRCE-DDS)을 통해 PX4로 목표 위치(Position Setpoint)나 속도(Velocity Setpoint) 명령이 전달된다.

이러한 분산 제어 환경에서 기체가 컴패니언 컴퓨터의 명령을 수동적으로 추종하는 비행 모드를 오프보드(Offboard) 모드라고 한다.

2. 컴패니언 컴퓨터 통신 단절 감지: COM_OF_LOSS_T

오프보드 모드 중에는 비행 제어기가 자체적인 경로 계획 비행(Mission)을 수행하지 않고 전적으로 외부 컴퓨터의 주기적인 명령(Setpoint)에 의존한다. 따라서 컴패니언 컴퓨터의 소프트웨어 크래시(Crash), 과부하(Overload)로 인한 멈춤, 또는 물리적 케이블(UART/Ethernet) 단절이 발생하면 기체는 제어 입력을 상실하여 예측 불가능한 궤적으로 이탈할 위험이 있다.

이를 방지하기 위해 PX4는 COM_OF_LOSS_T (Offboard Loss Timeout) 파라미터를 제공한다.

  • 정의: 컴패니언 컴퓨터로부터 유효한 오프보드 제어 세트포인트(Setpoint)가 수신되지 않는 최대 허용 시간(초 단위 임계값).
  • 통상적 설정: GCS 통신 단절 파라미터(COM_DL_LOSS_T가 보통 10초 이상)와 비교하여, 오프보드 제어는 높은 제어 대역폭을 요구하므로 COM_OF_LOSS_T는 상대적으로 매우 짧게 설정된다 (보통 0.5초 ~ 1.0초 미만).
  • 페일세이프 트리거: 설정된 시간(예: 0.5초) 내에 오프보드 명령 메시지가 도착하지 않으면, Commander 모듈은 즉각 오프보드 제어 권한을 박탈하고 페일세이프 상태로 강제 전환한다.

3. 오프보드 연결 단절 시의 분산 제어 안전 정지(Safe Stop) 로직

컴패니언 컴퓨터의 제어 신호가 끊겼을 때 PX4가 취할 수 있는 동작(Action)은 다른 페일세이프 파라미터(예: COM_OBL_ACT 또는 NAV_DLL_ACT 활용)에 의해 결정되며, 비행 안정성을 최우선으로 하여 다음과 같은 단계로 안전 정지 로직이 구성된다.

  1. 동적 브레이크(Dynamic Braking): 기체가 빠른 속도로 비행 중 통신이 끊어졌을 때 즉시 모터를 멈추거나 착륙하면 전복될 수 있다. 따라서 PX4는 현재 속도 벡터 역방향으로 브레이크 제어를 수행하여 기체를 공중에 완전히 호버링(Hovering/Hold) 상태로 안정화시킨다.
  2. 안전 모드 전환 (Fallback Flight Mode):
  • Hold (Loiter): 일시적인 프로세스 지연일 수 있으므로 현재 위치에서 호버링하며 컴패니언 컴퓨터의 재연결을 기다린다.
  • Return (RTL): 지정된 대기 시간을 초과하면, 더 이상 임무 수행이 불가능하다고 판단하고 안전하게 이륙 지점(Home Position)으로 자동 복귀한다.
  • Land: 물리적인 손상이나 극단적인 상황에서 현재 위치에 강제 착륙한다.
  1. 조종사 제어권 이양(Manual Takeover): GCS 연결이나 RC(조종기) 연결이 살아있다면, 오프보드 단절 즉시 조종사가 Position 모드나 Manual 모드로 전환하여 기체를 수동으로 안전하게 회수할 수 있다.

4. 컴패니언 컴퓨터 통신 흐름 및 검증 메커니즘 도식화 (Mermaid)

다음은 컴패니언 컴퓨터에서 ROS2 노드나 MAVSDK를 통해 주기적으로 Setpoint를 퍼블리싱(Publishing)할 때, PX4 내부 루프에서 이를 감지하고 안전 정지(Safe Stop) 로직을 구동하는 과정을 도식화한 다이어그램이다.

    PX4_CMD->>PX4_OB: Revoke Offboard Control Authority
    PX4_CMD->>Motors: Trigger HOLD Mode (Dynamic Braking)
    Motors->>Motors: Stabilize & Hover in Place
    
    Note over PX4_CMD: If timeout persists, execute RTL / Land

5. 소스 코드 레벨에서의 처리 방식 (Source Code Implementation)

펌웨어 소스 트리 상에서 src/modules/commander/ 디렉터리의 상태 머신(State Machine)은 GCS 데이터 링크(Link) 채널과 컴패니언 컴퓨터 명령 수신 채널을 논리적으로 분리하여 감시한다.

  • Commander.cpp 루프 내에서 수시로 불리는 타임아웃 검사 로직은 offboard_control_mode 토픽의 주기적인 타임스탬프를 확인한다.
  • 컴패니언 컴퓨터의 인터페이스(예: mavlink_receiver.cpp 또는 uxrce_dds_client)가 새로운 메시지를 uORB에 게시(Publish)할 때마다 시간 관련 변수가 갱신된다.
  • 만일 갱신 주기가 COM_OF_LOSS_T를 초과하면, 내부적으로 failsafe_flags.offboard_control_signal_lost 플래그가 true로 설정되며 즉시 비행 모드의 강제 변경을 수반하는 페일세이프 처리기(failsafe.cpp)로 제어 흐름이 넘어간다.

이처럼 PX4-Autopilot은 분산 제어 아키텍처가 지닌 이면의 취약점인 단일 장애점(SPOF, Single Point of Failure)을 극복하기 위해, 매우 짧은 임계 시간과 즉각적인 동력 제어 개입을 통해 신뢰할 수 있는 강건 시스템(Robust System)을 구축하고 있다.
