53.6.2.1 통신 두절 구간 동안 비동기적으로 누락된 비행 모드(Flight Mode) 및 시동 상태(Arming Status)의 최우 추정(Maximum Likelihood Estimation) 동기화

기체와 통신이 두절(Data Link Loss)된 구간 동안 기체의 내부 상태(Internal State)는 GCS 관점에서는 마치 블랙박스와도 같다. 특히 비행 모드(Flight Mode)와 시동 상태(Arming Status)의 변화는 기체의 자율 비행 로직(Auto Flight Logic)이나 물리적 안전 결함에 의해 비동기적(Asynchronously)으로 변경될 가능성이 매우 높다. 예를 들어 통신이 끊긴 동안 기체가 배터리 부족을 감지하여 자동으로 RTL(Return to Launch) 모드로 전환되었거나, 혹은 지면에 닿아 자동으로 시동이 꺼진(Auto-disarm) 상태일 수 있다. 통신이 복구되었을 때 GCS는 누락된 기간 동안 이들 핵심 상태가 어떻게 변해왔는지, 그리고 현재 상태가 정확히 무엇인지 최우 추정(Maximum Likelihood Estimation, MLE) 개념을 도입하여 가장 신뢰성 높은 결과값으로 동기화해야 한다.

1. 비행 모드와 시동 상태 누락의 위험성

GCS와 PX4-Autopilot 간의 상태 동기화가 실패할 경우, 시스템 운용상 치명적인 취약점이 발생한다.

  • 비행 모드 불일치: 통신이 복구되었을 때 GCS는 기체가 여전히 Mission 모드라고 인식하고 있으나, 실제 기체는 통신 단절 페일세이프(Failsafe) 발동으로 Hold 혹은 RTL 모드로 진입했을 수 있다. 이 상태에서 운영자가 GCS 화면만 믿고 잘못된 후속 명령(예: 고도 변경)을 지시하면 시스템 충돌이나 예기치 않은 기동을 야기할 수 있다.
  • 시동 상태 불일치: 기체가 이미 착륙 후 디스암(Disarm) 되었음에도 GCS 상에서 암(Arm) 상태로 남아있다면, 운영자의 오조작으로 즉각적인 프롭 회전이 발생하여 인명 피해로 직결될 위험이 존재한다.

2. 최우 추정(Maximum Likelihood Estimation) 도입의 개념적 근거

통신 복구와 동시에 기체는 현재의 HEARTBEAT 메시지를 주기적으로 다시 송출하기 시작한다. 그러나 MAVLink 통신 채널 상의 노이즈나 무선 채널 점유 문제로 인해 최초 복구 시점의 단일 패킷만으로는 수신된 상태 데이터의 신뢰성을 **초기 100%**로 장담하기 어렵다. 간헐적 패킷 드롭(Packet Drop)이나 손상된 패킷의 체크섬 통과 오류 가능성이 이론적으로 존재하기 때문이다.

따라서 GCS의 통신 제어 아키텍처는 통신 복구 후 일정 시점 동안 도달하는 HEARTBEAT 메시지 시퀀스들을 관측 데이터로 삼아, 비행 모드와 시동 상태에 대한 **최우 추정(MLE)**을 수행한다. 즉, “관측된 상태 시퀀스를 기반으로 가장 확률이 높은 실제 기체의 상태를 식별(Identification)하는 과정“을 거친다.

3. QGroundControl에서의 MAVLink 상태 동기화 파이프라인

QGroundControl(QGC) 내부의 C++ Vehicle 클래스 및 디스패처는 위와 같은 철학에 기반하여 동기화 파이프라인(Synchronization Pipeline)을 설계한다. HEARTBEAT 객체의 base_mode, custom_mode 필드가 추정의 기본 입력 베이스라인이 된다.

  1. 샘플링 버퍼(Sampling Buffer) 수집: 통신 복구(Link Restored) 이벤트가 발생하면 즉시 UI를 업데이트하지 않고, 짧은 윈도우(예: 0.5초~1초, 1~2Hz 통신 시 2~3개의 패킷) 동안 HEARTBEAT 패킷을 버퍼링한다.
  2. 모드 분류 메이저리티 보팅(Majority Voting): 수집된 버퍼에서 가장 시점 간 지배적으로 나타나는(단순화된 형태의 MLE) 비행 모드 및 시동 상태를 결괏값 도출 체인에 올린다.
  3. 내부 상태 전이(State Transition): 추정 값이 확정되면 GCS의 내부 논리적 모델(Logical Model)을 하드 리셋(Hard Reset)하고, FlightModeChanged 및 ArmedChanged 시그널(Qt Signal)을 방출한다.
  4. UI 엔진 통보: QML 프론트엔드 단에 변경된 상태를 푸시(Push)하여 시각적 불일치를 일소한다.

4. PX4 시스템의 적극적인 상태 통보(Active State Notification) 설계

기체 시스템(PX4-Autopilot, v1.13 이상) 관점에서도 GCS가 수동적으로 HEARTBEAT를 폴링(Polling)하거나 필터링하는 방식에만 의존하게 두지 않는다.
PX4의 커맨더(Commander) 프로세스는 통신 두절 구간 동안 비동기적으로 비행 모드 변경이나 시동 해제 이벤트가 발생하면, 시스템 이벤트를 vehicle_status 토픽(uORB)에 기록한다. 통신이 복구되고 텔레메트리 세션이 수립되면, PX4 내 mavlink 모듈은 단순히 주기적인 HEARTBEAT 전송 큐 이외에, 변경 사항에 대한 중요 알림(Information/Warning 패킷 형태)을 MAVLink 스트림에 비동기적(Asynchronous)으로 추가 발송하는 절차를 권장한다.

이러한 양방향 상태 동기화 설계 체계는 GCS의 최우 추정(MLE) 체계가 단 몇 밀리초 내로 확실한 상태 수렴성(Convergence)을 확보할 수 있도록 돕는다.

5. Ardupilot 생태계와의 철학적 차이점 비교

Ardupilot은 전통적으로 MAVLink 1.0 시절부터 비교적 보수적이고 단순 반복적인 브로드캐스트 메커니즘을 신뢰하는 편이다. GCS(Mission Planner 등)는 통신 재개 시 수신되는 단일 패킷에 즉각 반응하여 UI의 상태를 오버라이트(Overwrite)하는 경향이 있으며, 그로 인한 찰나의 UI 글리치(Glitch) 발생을 감수한다.
반면 PX4와 QGroundControl 아키텍처는 내부 C++ 레코드 객체 기반의 미들웨어를 두고 시그널-슬롯(Signal-Slot) 기반으로 작동하므로, 시스템 타임스탬프와 연속성을 교차 검증하는 수학적 진단 절차(MLE를 표방하는 필터링)를 거쳐 UI 갱신을 원자적(Atomic)으로 보장하려는 방향으로 발전해 왔다. 이 미세한 차이는 통신 환경이 매우 열악한 다채널 릴레이(Relay) 비행이나 장거리 비가시권(BVLOS) 임무 시 GCS 운영의 심리적 안정감과 조작의 안전성을 결정짓는 핵심적 요소가 된다.
