53.3.2.2 히스테리시스 곡선(Hysteresis Curve) 적용을 통한 통신 플래핑(Flapping) 현상(연결/단절 반복) 시의 페일세이프 오작동 방지

지상 관제 시스템(GCS)과 무인항공기(UAV) 간의 무선 통신 환경은 도심지의 다중 경로 페이딩(Multipath Fading)이나 안테나의 지향성 문제로 인해 통신 품질이 급격하게 변동할 수 있다. 본 절에서는 통신 연결과 단절이 매우 짧은 주기로 반복되는 통신 플래핑(Flapping) 현상 시 PX4-Autopilot의 상태 머신이 겪을 수 있는 페일세이프(Failsafe) 오작동(Chattering)과, 이를 제어하기 위해 적용되는 히스테리시스(Hysteresis) 설계 기법을 분석한다.

1. 통신 플래핑(Flapping) 현상과 상태 채터링(Chattering)의 위험성

통신 플래핑 현상은 MAVLink Heartbeat 메시지가 수초 간격으로 유실되었다가 복구되기를 반복하는 현상이다.

만약 통신 단절(Link Lost) 임계값과 통신 복구(Link Regained) 임계값이 동일한 시간(예: Heartbeat 1개 수신 즉시 복구)으로 설정된 단순 임계치(Single-Threshold) 시스템을 가정해보자.

  • t = 10s: 통신 단절 플래그 발동 -> 기체가 RTL(Return To Launch) 모드로 전환하여 기수를 돌림.
  • t = 11s: 통신 1회 복구 플래그 발동 -> 기체가 다시 Mission 모드로 복귀하여 원래 궤적으로 기수를 돌림.
  • t = 12s: 다시 통신 단절 플래그 발동 -> 기체가 다시 RTL 모드로 전환.

이러한 모드 간의 빈번한 전환을 제어 공학에서 **채터링(Chattering)**이라고 부른다. 이는 기체의 자세 제어루프에 막대한 외란(Disturbance)을 주며, 불필요한 배터리 소모와 기구학적 불안정성(Kinematic Instability)을 초래하여 추락으로 이어질 수 있다.

2. 히스테리시스(Hysteresis) 기반의 상태 보호 메커니즘

이를 방지하기 위해 PX4-Autopilot의 상태 머신 설계에는 히스테리시스(Hysteresis), 즉 이력 현상 기반의 로직이 내재되어 있다. 전기 회로의 슈미트 트리거(Schmitt Trigger)와 유사하게, 상태가 ’A’에서 ’B’로 갈 때의 경로와 ’B’에서 ’A’로 갈 때의 경로(조건)를 다르게 두는 것이다.

  • 통신 단절 진입(Lost Transition) 임계 조건:
    Heartbeat 메시지가 COM_DL_LOSS_T 파라미터(예: 10초) 동안 수신되지 않을 때 비로소 Lost 상태로 전이한다. (엄격한 단절 기준)
  • 통신 복구 진입(Regained Transition) 임계 조건 (디바운스/히스테리시스):
    Lost 상태에 빠진 이후 단 하나의 Heartbeat 메시지가 수신되었다고 해서 즉시 Connected 상태로 복귀하지 않는다. PX4의 내부 상태 머신은 복구 시점에서도 일정 시간(또는 연속된 N개의 보장된 메시지 수신) 동안 연결이 안정적으로 유지됨을 확인하는 **복구 대기 시간(Regain Timeout / Debounce Time)**을 적용한다.

3. 히스테리시스 곡선 및 MAVLink 통신 상태 전이 모델 적용 (Mermaid)

4. 소스 코드 구현: 디바운싱(Debouncing) 및 사용자 확인(User Confirmation)

PX4의 모듈 중 src/modules/commander/ 및 failsafe.cpp는 히스테리시스 특성을 코드로 구현해내고 있다.

  1. 시간 윈도우(Time Window) 필터링:
    최신 펌웨어에서는 uORB의 telemetry_status 토픽을 통해 단순한 시간 지연뿐만 아니라 통신 대역폭(Data Rate)의 이동 평균(Moving Average)을 산출하여, 통신 상태가 지터링(Jittering) 기준치를 벗어났는지 디바운스 로직으로 필터링한다.
  2. 모드 복구의 수동 개입 (Manual Override/Confirmation):
    강건성(Robustness) 향상을 위해, 일단 RTL 등의 페일세이프가 트리거된 이후 통신이 완전히 구조적으로 복구되었다 하더라도, 시스템이 자의적으로 이전 Mission 모드로 복귀하지 않도록 제한하는 경우가 많다.
    대신 조종사가 GCS를 통해 명시적인 모드 변경 명령(예: MAV_CMD_DO_SET_MODE)을 브로드캐스팅(Broadcasting)하여 “내가 통신을 온전히 회복했고 상황을 통제하고 있다“는 것을 수동으로 어서트(Assert)해야만 상태 채터링 루프를 완전히 끊어낼 수 있다.

5. 결론 및 Ardupilot 비교

Ardupilot은 FS_GCS_ENABLE 동작 후 연결이 복구되면 즉시 또는 자동으로 원래 임무로 돌아가도록 설정하는 파라미터 옵션(FS_OPTIONS의 Continue if recovered 플래그)을 유연하게 제공하기도 한다.

하지만 PX4-Autopilot의 관제 철학은 “페일세이프는 최후의 수단이며, 한 번 깨진 신뢰(통신 불량)는 충분한 히스테리시스 시간과 조종사의 능동적인 확인(Acknowledgement)이 있기 전까지는 복구된 것으로 간주하지 않는다“는 강건한 안전 중심(Safety-centric)의 패러다임을 따른다. 이로 인해 통신 플래핑으로 인한 기체의 물리적 요동(Oscillation)과 시스템적 혼란을 원천적으로 차단한다.
