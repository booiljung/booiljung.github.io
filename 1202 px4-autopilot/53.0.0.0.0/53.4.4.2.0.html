53.4.4.2 유한 상태 기계 기반의 연결 재시도 한계 횟수(Retry Limit) 도달 시 심각한 오류(Fatal Error) 예외 처리 스레드 이관

무인항공기 시스템(UAS; Unmanned Aerial System)의 지상 관제 시스템(GCS; Ground Control Station)과 비행 제어기(Flight Controller) 간의 통신은 다양한 외부 요인에 의해 단절될 수 있다. 자동 재연결(Auto-Reconnect) 스케줄링을 통해 연결 복구를 시도하지만, 유한 상태 기계(FSM; Finite State Machine) 구조 안에서 정의된 연결 재시도 한계 횟수(Retry Limit)에 도달하게 되면 시스템은 이를 단순한 네트워크 지연이 아닌 심각한 오류(Fatal Error)로 규정해야 한다. 본 절에서는 이러한 상황이 발생했을 때 시스템 안정성을 확보하기 위해 수행되는 예외 처리(Exception Handling)와 해당 권한을 별도의 예외 처리 스레드(Exception Handling Thread)로 이관(Delegation)하는 메커니즘을 상세히 분석한다. (참고 버전: PX4-Autopilot v1.14, QGroundControl v4.3)

1. 재시도 한계 도달과 심각한 오류(Fatal Error)의 정의

GCS 관제 채널이나 MAVLink 통신 데몬(mavlink daemon)은 일시적인 데이터 손실(Data Loss)과 영구적인 통신 두절(Communication Failure)을 텔레메트리 타임아웃(Telemetry Timeout) 메커니즘을 통해 판별한다. MAVLink 프로토콜 기반 통신에서 일정 주기마다 수신되어야 하는 하트비트(Heartbeat) 메시지가 수신되지 않을 경우, 시스템은 연결 재시도를 수행한다.

재시도 메커니즘이 한계 횟수(예: QGroundControl의 경우 기본적으로 수 회의 재연결 로직 적용)를 초과하게 되면 시스템은 심각한 오류(Fatal Error) 상태로 전이(Transition)한다. 이는 통신 링크를 담당하는 물리 계층(Physical Layer) 또는 데이터 링크 계층(Data Link Layer) 자체의 하드웨어적 결함이나 환경적 차단이 발생했음을 의미하므로, 더 이상의 단순한 재시도는 무의미하며 즉각적인 페일세이프(Failsafe) 로직 작동이 필요함을 시사한다.

2. 유한 상태 기계(FSM) 아키텍처 상의 상태 전이 설계

PX4-Autopilot 및 QGroundControl의 통신 관리 모듈은 내부적으로 FSM을 기반으로 동작하여 각 연결 상태를 독립적이고 결정론적(Deterministic)으로 관리한다. 통신 재시도 로직에서 상태 전이는 다음과 같은 흐름을 따른다.

stateDiagram-v2
    [*] --> CONNECTED : 초기 연결 성공
    CONNECTED --> LINK_LOST : 하트비트 수신 실패 (Timeout)
    LINK_LOST --> RECONNECTING : 재연결 시도
    RECONNECTING --> CONNECTED : 재연결 성공
    RECONNECTING --> RECONNECTING : 재시도 횟수 증가 (Count < Limit)
    RECONNECTING --> FATAL_ERROR : 재시도 한계 횟수 도달 (Count >= Limit)
    FATAL_ERROR --> EXCEPTION_HANDLING : 스레드 이관 및 페일세이프 트리거
    EXCEPTION_HANDLING --> [*] : 자원 회수 및 작동 종료/대기

위의 모식도에서 볼 수 있듯이, RECONNECTING 상태에서 반복적인 시도가 실패하여 한계치(Limit)에 도달하면 FATAL_ERROR 상태로 전이한다. 이 시점부터는 일반적인 통신 제어 로직이 수행되는 메인 스레드(Main Thread)나 통신 스레드(Communication Thread)에서 예외를 처리하는 것은 시스템의 응답성을 저하시키고 교착 상태(Deadlock)를 유발할 수 있으므로, 제어권을 예외 처리 전담 스레드로 이관하게 된다.

3. 예외 처리 스레드 이관(Delegation) 메커니즘

예외 처리 스레드로의 권한 이관은 비동기 소프트웨어 아키텍처에서 결함 허용(Fault Tolerance)과 격리(Isolation)를 보장하기 위한 핵심 설계 패턴이다.

3.1 스레드 분리의 필요성

통신을 담당하는 스레드는 데이터 송수신 및 버퍼(Buffer) 처리에 고도로 최적화되어 있다. 통신 스택이 재시도 한계에 도달했을 때, 파일 디스크립터(File Descriptor)를 닫고, 연결된 포트를 해제(Release)하며, 시스템 전역에 치명적 오류 이벤트를 전파(Broadcasting)하는 작업은 상대적으로 많은 연산 시간과 I/O 블로킹(Blocking)을 수반한다. 이를 메인 통신 스레드에서 수행할 경우 시스템 전체의 이벤트 루프(Event Loop)가 정지될 위험이 있다. 따라서 예외 상황 발생 시 **작업 큐(Work Queue)**를 활용하여 전담 스레드로 임무를 이관한다.

3.2 C++ 기반 스레드 이관 예시 코드 구조

QGroundControl(C++ 기반 Qt 프레임워크) 및 PX4의 내부 모듈에서 스레드 이관을 처리하는 추상적인 소프트웨어 아키텍처 흐름은 다음과 같다.

// 통신 클래스의 재연결 시도 함수 예시
void LinkManager::attemptReconnection() {
    if (this->_retryCount >= MAX_RETRY_LIMIT) {
        // 심각한 오류 상태 선언
        this->_currentState = ConnectionState::FATAL_ERROR;
        
        // 예외 처리 스레드로 작업 이관 (비동기 시그널-슬롯 또는 워커 스레드 사용)
        ExceptionPayload payload(this->_linkId, ErrorCode::RETRY_LIMIT_EXCEEDED);
        
        // Qt의 메타 오브젝트 시스템을 이용한 스레드 간 비동기 이벤트 큐 전송
        QMetaObject::invokeMethod(this->_exceptionHandlerThread, 
                                  "handleFatalError", 
                                  Qt::QueuedConnection, 
                                  Q_ARG(ExceptionPayload, payload));
                                  
        // 현재 통신 스레드의 해당 링크 자원 접근 중지
        this->haltLinkOperations();
    } else {
        this->_retryCount++;
        // 재연결 로직 수행
    }
}

위 코드에서 Qt::QueuedConnection을 사용함으로써 이벤트를 예외 처리 스레드의 큐에 적재(Enqueue)하고, 통신 스레드는 즉각 반환(Return)되어 블로킹 없이 자신의 루프를 계속 실행할 수 있다.

4. 예외 처리 스레드의 주요 역할 및 절차

예외 처리 스레드로 제어권이 이관된 후, 시스템 안정성을 유지하기 위해 스레드는 다음의 과업을 순차적으로 수행한다.

  1. 사용자 인터페이스(UI) 경고 송출: 관제 인원에게 복구 불가능한 통신 단절이 발생했음을 즉각적으로 시각 및 청각적 알람으로 고지한다.
  2. 소켓 및 하드웨어 자원 강제 수거(Garbage Collection): 운영 체제 레벨에서 열려 있는 포트(TCP/UDP 및 직렬 포트)를 강제로 닫고, 누수(Memory Leak)를 방지하기 위해 할당된 버퍼 메모리를 해제한다.
  3. 페일세이프(Failsafe) 로직 트리거: GCS의 경우 더 이상 기체로 제어 명령을 보낼 수 없으므로, 현재까지 수집된 데이터를 안전하게 로컬 데이터베이스 또는 로그 파일(예: ULog)로 플러시(Flush)하여 기록을 보존한다. 기체(PX4-Autopilot) 측의 경우 이러한 심각한 통신 오류 시 즉시 RTL(Return to Launch) 모드로 전환하거나, 착륙(Land)을 수행하도록 독립적인 커맨더 스택이 작동하게 된다.
  4. 결함 로깅 및 진단 정보 기록: 사후 분석(Post-flight Analysis)을 위해 오류 발생 직전의 통신 핑(Ping) 지연율, 손실률(Drop Rate) 등의 통계 정보를 진단 로그에 기록한다.

5. 결론 및 공학적 시사점

유한 상태 기계 체계 하에서 통신 재시도 한계 횟수 도달 시 예외 처리를 분리된 스레드로 이관하는 아키텍처는 고신뢰성 실시간 시스템(Hard Real-Time System) 구축의 필수 요구 조건이다. Ardupilot이나 PX4와 같은 오픈소스 복합 시스템에서 모듈 간 침범(Coupling)을 막고, 예측 불가능한 I/O 블로킹으로부터 코어 로직을 방어하기 위해 이러한 설계 패턴은 깊이 있게 연구되고 적용되어야 한다. 관련 아키텍처에 대한 보다 깊은 이해는 비동기 소프트웨어 공학의 동시성 제어(Concurrency Control in Asynchronous Software Engineering) 및 관련 모델 기반 설계 논문을 참고할 수 있다.
