53.4.2.2 통신 재개 시 데이터베이스 트랜잭션 무결성(Transaction Integrity) 확보를 위한 체크포인트(Checkpoint) 롤백 아키텍처

참고 버전 정보

  • PX4-Autopilot: v1.14.0 이상
  • QGroundControl: v4.3.0 이상

RF(초고주파)를 기반으로 하는 텔레메트리(Telemetry) 통신 링크는 심각한 노이즈 환경에 노출될 수 있으며, 간헐적으로 수 분에 걸친 통신 단절(Data Link Loss)을 경험하기도 한다. 링크가 물리적으로 다시 복구(Link Restored)되었을 때, 지상 관제 시스템(GCS)은 그동안 수신하지 못한 파라미터(Parameter), 미션(Mission Item), 텔레메트리 시계열 데이터의 파편들을 다시 동기화해야 한다. 이때 수신되는 비정상적인 스트림 쓰레기를 배제하고 시스템의 내부 상태 모델(State Model)을 오염시키지 않기 위해서는, 관계형 데이터베이스(RDBMS) 기술에서 차용한 체크포인트(Checkpoint) 기반의 롤백(Rollback) 아키텍처를 GCS 통신 트랜잭션 계층에 도입하여 데이터 무결성(Transaction Integrity)을 방어해야 한다.

1. MAVLink 데이터 트랜잭션의 취약성

MAVLink 프로토콜은 본질적으로 비연결성(Connectionless) 지향의 UDP 또는 시리얼(Serial) 스트림을 사용한다. 따라서 TCP와 같이지원 계층 수준의 세션(Session) 복구나 무결성 보장을 완벽히 제공하지 않는다.
통신이 끊어지기 직전에 전송되던 QGroundControl(이하 QGC)의 파라미터 업데이트 요청이나 미션 업로드 트랜잭션이 중단되었다면, 시스템의 상태는 “절반만 성공한 데이터(Half-baked Data)” 상태로 남을 위험이 있다. 이러한 데이터 상태(Dirty Read)에서 통신이 재개될 경우, 비행 제어기와 GCS 간의 상태 불일치(State Data Inconsistency)를 유발하여 치명적인 오작동(예: 잘못된 경유지로 비행)을 초래할 수 있다.

2. 체크포인트(Checkpoint)와 트랜잭션 무결성 방어 메커니즘

GCS는 단편적인 패킷 갱신이 아니라, 일련의 데이터베이스 상태 변경 논리를 단일 트랜잭션(Transaction)으로 묶어 처리한다. 트랜잭션은 원자성(Atomicity), 일관성(Consistency), 고립성(Isolation), 지속성(Durability)의 ACID 특성을 모방하여 설계된다.

2.1 체크포인트(Checkpoint) 스냅샷 생성

QGC의 내부 ParameterManager나 MissionManager는 비행 제어기(PX4)와 마이크로 트랜잭션(Micro Transaction)을 개시하기 전에, 현재까지 동기화가 확인된 가장 안정적인 메모리 상태나 로깅 파일의 오프셋을 체크포인트(Checkpoint) 로 저장한다. 이 체크포인트는 해시(Hash) 서명과 시퀀스 번호(Sequence Number)를 포함한다.

2.2 롤백(Rollback) 구조의 발동 조건

통신 타임아웃(Timeout)이 발생하여 단절 상태(is_link_connected == false)로 진입하면, 관리 모듈은 진행 중이던 트랜잭션 버퍼를 즉시 폐기(Drop)하고 상태를 체크포인트 지점으로 강제 롤백(Rollback) 한다.
통신이 다시 복구된 시점에서는 끊어진 패킷부터 다시 조립(Reassembly)하는 대신, 체크포인트 이후의 전체 데이터를 PX4-Autopilot에 명시적으로 다시 요청(Refetch)함으로써 일관성을 보장한다.

3. QGC 및 PX4 소스 코드 아키텍처 분석

QGC의 C++ 백엔드 코어에서는 트랜잭션을 제어하기 위해 QStateMachine 또는 명시적인 상태 전이 열거형(Enum)을 사용한다. 상태 머신 내에서 타임아웃 엣지(Timeout Edge) 트리거 시 데이터베이스나 모델의 롤백 로직이 발동된다.

  • Ardupilot(Mission Planner)와의 비교: Mission Planner의 경우 대용량 파라미터 동기화가 중단되면, 누락된 파라미터 인덱스 범위를 계산하여 특정 ID 리스트 단위로 재요청(Bulk Request)하는 단순 캐싱 동기화를 취하는 경향이 있다.
  • 반면 PX4와 연동되는 QGC 환경의 경우, MAVLink FTP(File Transfer Protocol) 확장을 이용하여 통신 데이터 전송 자체를 블록 단위 트랜잭션으로 취급한다. 통신 단절 시 QGC는 MAVLink FTP 세션을 과감히 리셋(Reset Session) 처리해 버리고, 통신 복구 후 처음부터 재개하는 식의 소프트 롤백 정책을 적용하여 GCS 로컬 데이터베이스의 오염을 극단적으로 회피한다.

5. 결론 및 고찰 (Conclusion & Discussion)

안전 최우선(Safety-Critical) 환경인 무인기 지상 관제 시스템에서 통신 인터럽트(Interrupt)가 로컬 변수나 화면에 잔여 오염 데이터를 남겨서는 안 된다. 통신 단절 후 시스템이 복구되었을 때, 낡고 훼손된 메시지 파편들로 인해 시스템이 거짓 상태(False State)에 진입하는 것을 방어하려면 강력한 트랜잭션 처리 구조가 필수적이다.

개발자는 중요 데이터(경로 맵, 파라미터, 센서 캘리브레이션 트리거)를 송수신할 때, 반드시 체크포인트 설계와 롤백 뷰(Rollback View) 를 선행하여 GCS가 항상 PX4-Autopilot과 1:1로 일치된 참 상태(True State)만을 화면과 데이터베이스에 출력 및 기록하도록 설계해야 한다.
