53.6.2.2 시스템 타임스탬프 불일치(Desynchronization)에 의한 GCS 측 궤적 시계열 차트의 불연속(Discontinuity) 현상 보간(Interpolation) 처리

무인기 시스템 운용에 있어 지상 관제 시스템(GCS)이 수신하는 기체의 텔레메트리(Telemetry) 데이터는 시간(Time)이라는 근본적인 축을 바탕으로 정렬된다. 통신이 정상적일 때 기체(PX4-Autopilot)가 발송하는 각종 MAVLink 메시지는 기체 내부의 부팅 이후 경과 시간(Boot Time) 또는 동기화된 시스템 타임스탬프(System Timestamp)를 포함한다. 그러나 통신이 장시간 두절되었다가 복원되는 시점에는 필연적으로 GCS 내부의 로컬 타임스탬프와 기체에서 전송받는 페이로드(Payload) 타임스탬프 간의 동기화 오차(Desynchronization)가 누적되어 있으며, 이는 GCS 화면의 시계열 차트나 비행 궤적(Trajectory) 상에 거친 불연속성(Discontinuity)을 초래한다.

1. 타임스탬프 불일치(Desynchronization)의 발생 원인

통신이 끊어지면 GCS는 새로운 데이터를 수신하지 못하므로 차트 렌더링 엔진이나 데이터 로거(Data Logger)는 일시적으로 정지되거나, 마지막 값을 유지(Hold)하며 현재의 GCS 로컬 시간을 계속 흘려보낸다. 반면 기체 내부의 비행 제어기(Flight Controller) 시계는 이와 무관하게 계속해서 증가하며 상태로그를 갱신한다.

통신이 다시 연결되었을 때, 기체는 현재 시점의 타임스탬프를 달고 새로운 데이터를 쏟아낸다. 이때 GCS의 수신 버퍼는 장시간 비어있다가 갑자기 거대한 시간적 점프(Time Jump)를 동반한 데이터를 맞이하게 된다. 만약 이를 시계열 차트(Time-Series Chart)나 3D 궤적 화면에 그대로 밀어 넣게 되면 다음과 같은 문제가 발생한다.

  • 그래프의 직각 스파이크(Spike) 현상: 통신이 끊긴 시간(T_{loss})부터 복구된 시간(T_{restore}) 사이의 데이터가 공백으로 남아 있다가, 최신 데이터가 유입되면서 그래프 상에서 비현실적인 수직선이나 계단형 불연속 단차가 그려진다.
  • 속도(Velocity) 및 가속도 미분치 폭증: GCS 내부에서 수신된 위치 데이터를 로컬 시간으로 미분하여 속도를 추정하는 로직이 있다면, 시간은 순간적인데 위치는 수십 미터를 이동한 것으로 계산되어 무한대에 가까운 비정상적 속도 값이 관측기(Observer) 필터를 교란시킨다.

2. QGroundControl 및 PX4의 시간 동기화(Timesync) 프로토콜

이러한 문제를 방지하기 위해 MAVLink 프로토콜은 TIMESYNC (Msg ID: 111) 메시지를 이용해 기체와 GCS 간의 시계를 맞추는 네트워크 타임 프로토콜(NTP)과 유사한 메커니즘을 제공한다. PX4 펌웨어 모듈인 mavlink_timesync는 GCS에서 보내는 TIMESYNC 핑(Ping) 메시지에 응답하여 통신 지연(Round Trip Time, RTT)과 오프셋(Offset)을 계산하도록 돕는다.

통신 복구 세션(Session Restored)이 수립되는 즉시, GCS(QGC)의 링크 매니저는 백그라운드에서 TIMESYNC 메시지를 최우선(High Priority) 큐로 전송하여 두 시스템 간의 타임 오프셋을 즉각 재계산(Re-evaluation)한다. 이를 통해 통신 두절 동안 틀어져 버린 양측의 시간 축을 우선적으로 일치시킨다.

3. 궤적 시계열 차트 보간(Interpolation) 메커니즘

시간 축이 교정되었다 하더라도, 통신이 끊어졌던 수십 초에서 수 분 동안의 기체 공간 이동 이력은 MAVLink를 통해 즉각 실시간으로 모두 복구되지 않는다(네트워크 대역폭의 한계). 따라서 화면상 기체가 텔레포트(Teleport)한 것으로 나타나는 물리적 단차를 시각적, 논리적으로 부드럽게 잇는 보간(Interpolation) 과정이 필수적이다.

  1. 시계열 차트 결측치 핸들링(Invalidation & Gap Handling): 통신 두절 구간에 해당하는 시간대의 내부 데이터 그래프는 억지로 선을 긋지 않고 명시적인 ‘결측치(NaN, Not a Number)’ 구간이나 점선, 반투명 음영 영역으로 처리하여 운영자에게 통신의 부재를 인지 공학적으로 명확히 알린다.
  2. 공간 궤적 스플라인(Spline) 보간: 2D 또는 3D 지도 상에 기체의 궤적 선을 표출할 때, 통신 단절 직전의 위치(Node A)와 복구 직후의 위치(Node B)를 단순 직선으로 잇는 대신, A 지점에서의 마지막 속도 벡터와 B 지점에서의 새로운 진입 속도 벡터를 활용하여 에르미트 스플라인(Hermite Spline) 곡선 형태로 부드럽게 추정 경로를 시각화한다.
  3. UI 동적 스무딩(Dynamic Smoothing): 실제 지도 위의 기체 아이콘이 A에서 B로 순간이동 할 때, QML 기반의 프론트엔드(UI)는 애니메이션(Animation) 엔진을 통해 0.5초 내외의 매우 짧은 시간 동안 아이콘이 시각적으로 미끄러지듯 평행 이동(Translate)하게 하여 조종자의 시각적 충격을 완화한다.

4. 시계열 데이터 처리 시 Ardupilot 대비 PX4 관제 특성

Ardupilot 환경을 주로 지원하는 전통적인 서드파티 GCS 애플리케이션들(예: 초기 Mission Planner 버전 등)은 데이터 로깅이나 UI 시계열 표현 시 주로 GCS를 구동하는 로컬 PC의 커널 타이머 의존도가 높다. 이로 인해 통신 재개 시 타임스탬프 불일치나 누락 현상에 대해 필터링 없는 날것(Raw)의 스파이크(Spike) 그래프 역전 현상을 그대로 표출하는 경우가 빈번했다.

반면에 현재 사용되는 QGroundControl 의 최신 아키텍처는 C++ 기반의 데이터 모델 코어(FactSystem 및 Vehicle 객체)와 QML 프론트엔드(View) 엔진이 명확히 분리 되어있다. 시스템은 이러한 결측 데이터 스트림이 유입될 경우, 수학적 필터링을 통해 선제적으로 낡은 데이터를 폐기(Invalidate)하거나 부드럽게 보간(Interpolate)하여 QML 단에 전달한다. 이는 성능이 제한적인 태블릿 환경 등에서도 렌더링 엔진 오버헤드를 막고 조종자에게 일관된 궤적 흐름을 보장하는 최적화된 설계 양식이라 판단할 수 있다.
