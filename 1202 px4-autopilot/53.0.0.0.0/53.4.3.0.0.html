53.4.3. 제어 명령 대기열(Command Queue) 내 교착 상태(Deadlock) 방지 및 송신 버퍼 제어

통신 단절 현상이 발생했을 때 GCS(지상 관제 시스템)에서 가장 빈번하게 겪는 아키텍처 수준의 문제는 제어 명령 대기열(Command Queue)의 교착 상태(Deadlock)와 송신 버퍼(Tx Buffer)의 누수 현상이다. PX4-Autopilot과 연동되는 GCS는 MAVLink 프로토콜을 통해 양방향 비동기 통신을 수행한다. 이때 네트워크 링크가 불안정하거나 완전히 소실되면, 발송을 기다리는 COMMAND_LONG 메시지나 임무(Mission) 관련 패킷들이 내부 큐에 무한정 적체되는 현상이 발생한다. 이러한 데이터 적체는 메모리 누수는 물론, 메인 이벤트 루프의 블로킹(Blocking)을 유발하여 애플리케이션 전체의 프리징(Freezing)으로 이어질 수 있다.

이와 같은 교착 상태를 방지하기 위해서는 비동기 큐 관리 메커니즘과 송신 버퍼의 능동적 생명주기(Lifecycle) 관리가 필수적이다. 특히, QGroundControl(QGC)과 같은 현대적인 GCS는 다중 스레드(Multi-thread) 환경에서 텔레메트리를 처리하므로, 스레드 간의 통신 병목을 해제하는 세마포어(Semaphore) 및 뮤텍스(Mutex)의 정밀한 예외 처리 로직이 요구된다.

1. 제어 명령 대기열의 교착 상태 메커니즘

GCS에서 기체로 특정 명령(예: MAV_CMD_DO_SET_MODE)을 전송할 때, 일반적인 처리 파이프라인은 다음과 같은 단계를 거친다.

  1. 사용자 입력 및 명령 인스턴스화: UI 이벤트가 백엔드로 전달되어 MAVLink 커맨드 객체가 생성된다.
  2. 명령 큐 적재(Enqueue): 생성된 명령이 텔레메트리 송신 스레드의 대기열에 삽입된다. 이때 송신 스레드와 UI 스레드 간의 경쟁 상태(Race Condition)를 막기 위해 락(Lock)이 걸린다.
  3. 네트워크 I/O 송신: 소켓 또는 직렬 포트를 통해 물리적 전송이 이루어진다.
  4. ACK 대기(Waiting for Acknowledgment): 수신측(PX4)으로부터 COMMAND_ACK 응답을 기다리는 비동기 타이머가 동작한다.

만약 3번과 4번 단계에서 통신 단절이 발생하면, 시스템은 설정된 최대 재전송 횟수(Max Retries)만큼 패킷을 재발송하려 시도한다. 그러나 이 재시도 루프 내에서 소켓 에러 핸들링이나 락 해제(Unlock)가 명확하게 이루어지지 않으면, 다음 명령들이 큐에 계속 쌓이게 된다. 큐가 가득 차면 버퍼 오버플로우가 발생하거나, 큐에 접근하려는 다른 스레드들이 무한정 대기 상태에 빠지는 교착 상태(Deadlock)가 형성된다.

2. Ardupilot과 PX4 GCS 환경의 대기열 처리 차이

Ardupilot 환경에 주로 사용되는 Mission Planner의 경우, 윈도우 폼(Windows Forms) 기반의 동기적(Synchronous) 특성을 띠는 경우가 많아 명령을 전송하고 응답을 받을 때까지 UI 스레드가 블로킹되는 현상이 쉽게 나타난다. 이를 우회하기 위해 Application.DoEvents()와 같은 방식으로 메시지 펌프를 강제 구동하지만, 근본적인 교착 상태 해결책이 되지 못한다.

반면, PX4-Autopilot 제어에 널리 사용되는 QGC나 MAVSDK 기반의 커스텀 GCS는 철저한 비동기식(Asynchronous) 이벤트 루프(예: Qt의 Signal/Slot 메커니즘 또는 C++17의 std::async)를 사용한다. 통신이 단절되더라도 이벤트 루프 자체는 차단되지 않으나, 비동기 작업 콜백(Callback)이 응답 불가 상태비트(예: 타임아웃)로 전이될 때 메모리 내 큐 리소스를 즉각적으로 해제하는 가비지 컬렉션(Garbage Collection) 로직이 구현되어 있어야만 한다.

3. 송신 버퍼 제어와 윈도우 사이즈 튜닝

통신 단절 시 교착 상태를 막는 또 다른 핵심 요소는 운영체제 수준의 송신 버퍼(Tx Buffer) 동작을 이해하는 것이다. 운영체제의 TCP/UDP 소켓 버퍼나 직렬 포트 출력 버퍼는 가용 공간이 0이 되면 쓰기 작업(write() 또는 send()) 시스템 콜을 블로킹 모드로 전환하거나 EAGAIN 에러를 반환한다.

GCS 소프트웨어는 통신 단절로 인해 하위 네트워크 스택의 전송 버퍼가 비워지지 않는 현상을 감지해야 한다. 데이터 링크 계층에서 반송파 감지(Carrier Sense)를 상실하거나, MAVLink 하트비트 타임아웃이 트리거될 경우, GCS 애플리케이션 계층에서는 송신 파이프라인을 일시 중단(Suspend)하고, 버퍼에 대기 중인 모든 저우선순위 메시지를 강제로 비우는(Flushing) 절차가 필요하다.

    B -- 정상(버퍼 여유) --> C[일반 동작 수행]
    B -- 적체(버퍼 포화) --> D[명령 대기열 락 획득 시도]
    D --> E[타임아웃 적용된 뮤텍스 락 획득]
    E --> F[유효 기간 만료된 종속 패킷 드롭]
    F --> G[송신 버퍼 능동적 플러싱]
    G --> H[제어 로직 상태 초기화 및 교착 상태 해제]

위 흐름도에서 보듯, 타임아웃이 적용된 락(Timed Mutex) 획득은 데드락을 원천적으로 회피하기 위한 가장 중요한 소프트웨어 공학적 장치다. 단순히 락을 얻을 때까지 무한 대기(std::mutex::lock())하는 것이 아니라, std::timed_mutex::try_lock_for() 등을 사용하여 일정 시간 내에 큐의 소유권을 획득하지 못하면 예외를 발생시켜 스레드를 복구해야 한다. 이러한 제어 공학적 방어 기제는 GCS가 극한의 통신 불량 조건에서도 최소한의 생존성과 UI 반응성을 유지하도록 보장한다.
