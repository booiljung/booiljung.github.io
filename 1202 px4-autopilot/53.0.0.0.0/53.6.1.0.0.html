## 0.1 재연결 세션 수립(Session Establishment) 및 네트워크 핸드셰이크 프로토콜

위성 통신, RF 텔레메트리, 혹은 LTE/5G와 같은 무선 네트워크를 통해 연결된 무인항공기(UAV)와 지상 관제 시스템(GCS) 간의 통신이 단절되었다가 복구될 때, 양측 노드는 단순히 패킷을 주고받는 것을 넘어 논리적 계층에서의 세션(Session)을 공식적으로 재수립해야 한다. PX4-Autopilot과 QGroundControl 간의 MAVLink 기반 연결 구조에서, 이러한 재연결 세션 수립(Session Establishment) 및 네트워크 핸드셰이크(Handshake) 프로토콜은 기체의 식별, 호환성 검증, 그리고 통신 상태의 동기화를 위해 필수적인 과정이다.

0.1.1 물리적 계층과 MAVLink 논리적 계층의 분리

통신 복구 과정은 물리적/전송 계층(Transport Layer) 수준의 복구와 응용 계층(Application Layer, 즉 MAVLink) 수준의 복구로 명확히 나뉜다.

  • 네트워크 소켓 복구(Transport Layer): QGroundControl의 백엔드 통신 매니저는 끊어진 TCP/UDP 소켓이나 직렬 포트(Serial Port) 인터페이스를 지속적인 폴링(Polling)이나 비동기 재시도 로직을 통해 복구한다. 이때 TCP의 경우 3방향 핸드셰이크(3-Way Handshake)가 내부적으로 수행되며, UDP의 경우 첫 유효 패킷이 도달함으로써 파이프가 열린다.
  • MAVLink 세션 수립(Application Layer): 소켓이 연결되었다고 해서 기체와의 논리적 세션이 즉각 복구된 것은 아니다. MAVLink 프로토콜은 기체 스푸핑(Spoofing)이나 다중 기체 간 혼선 방지를 위해 MAVLink 핸드셰이크 프로토콜을 요구한다.

0.1.2 MAVLink 핸드셰이크 및 상호 식별 과정

물리적 채널이 열린 직후, GCS는 기체로부터 주기적으로 송출되는 브로드캐스트 패킷을 기다린다. 가장 권위 있는 세션 개시자는 HEARTBEAT 메시지이다.

  1. HEARTBEAT 감지 및 타겟(Target) 바인딩: GCS는 수신된 HEARTBEAT 패킷의 헤더에서 System ID와 Component ID를 추출한다. 이 식별자가 이전에 연결되었던 기체(단절 전의 Session Context)와 일치하는 경우, 기존 Vehicle 객체와 매핑(Mapping)하는 재수립 절차를 밟는다.
  2. AUTOPILOT_VERSION 요청과 호환성 검증: GCS는 기체에 MAV_CMD_REQUEST_AUTOPILOT_CAPABILITIES 명령을 발송하여(또는 기체가 자발적으로 브로드캐스팅하는 AUTOPILOT_VERSION을 수신하여) 대상 펌웨어가 PX4-Autopilot임을 확인하고, 지원하는 기능(Capabilities, 예: 임무 업로드 방식, 파라미터 프로토콜 버전 등) 비트마스크(Bitmask)를 파악한다.
  3. PING/PONG 핸드셰이크: 통신 링크의 왕복 지연 시간(RTT)과 채널의 실제 응답성을 검증하기 위해 무작위 페이로드 시퀀스를 포함한 PING 메시지를 MAVLink 레벨에서 교환한다. 이를 통해 논리적인 연결 건전성(Link Health)을 담보한다.

0.1.3 세션 수립 유한 상태 기계(FSM) 아키텍처

QGroundControl 백엔드 스레드의 재연결 및 핸드셰이크 과정은 다음과 같은 유한 상태 기계(FSM, Finite State Machine) 모델을 따른다.

stateDiagram-v2
    [*] --> Disconnected : 통신 단절 감지
    
    Disconnected --> Transport_Connecting : 물리 채널(소켓/시리얼) 복구 시도
    Transport_Connecting --> MAVLink_Handshake : 소켓 열림 및 데이터 수신 대기
    Transport_Connecting --> Disconnected : 연결 실패 (Timeout)
    
    MAVLink_Handshake --> Identification : HEARTBEAT 최초 수신
    Identification --> Protocol_Verification : AUTOPILOT_VERSION 수신 및 호환성 확인
    Protocol_Verification --> MAVLink_Handshake : 식별 실패 (알 수 없는 기체)
    Protocol_Verification --> Connected : 핸드셰이크 완료 및 세션 활성화
    
    Connected --> [*] : 정상 통신 상태 (State Sync 시작)

[도표 설명] QGroundControl의 상태 전이 설계도. 단순히 소켓이 연결되었다고 ‘Connected’ 상태로 전이하지 않고, 엄격한 MAVLink 핸드셰이크와 프로토콜 검증 단계를 거쳐야만 최종적으로 세션이 재수립됨을 보여준다.

0.1.4 보안 및 혼선 방지 (Session Spoofing & Clash Prevention)

하나의 GCS가 여러 대의 드론 모델(Swarm)을 관리하고 있거나, 인접한 대역폭에 다른 시스템의 통신 패킷이 난입할 수 있는 환경(예: 433MHz / 915MHz 텔레메트리 혼선)에서는 세션 수립 단계에서의 보안성이 더욱 중요해진다.

재연결 과정에서 GCS는 이전 세션의 고유 식별자(UUID, QGC 내에서 하드웨어 시그니처로 관리됨)를 참조한다. 만약 재연결된 통신 채널에서 수신되는 드론의 하드웨어 UUID 시그니처가 이전과 다르다면, GCS는 기존 통신 세션을 이어서 복구하는 것이 아니라 아예 새로운 기체가 감지된 것으로 간주하고 새로운 Vehicle 객체 인스턴스를 메모리 상에 생성한다.

이와 같은 정교한 네트워크 핸드셰이크 프로토콜은 단절된 시간 동안 발생했을 수 있는 악의적인 하이재킹(Hijacking) 시도를 무력화하고, 관제 시스템이 본래 조종하던 바로 그 기체와의 컨텍스트(Context)를 안전하게 이어받을 수 있도록 보장하는 소프트웨어적 방벽 역할을 수행한다.
