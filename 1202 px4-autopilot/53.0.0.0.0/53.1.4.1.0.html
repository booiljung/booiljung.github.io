53.1.4.1 하드웨어 인터럽트(Hardware Interrupt) 기반 RC 단절과 소프트웨어 폴링(Software Polling) 기반 GCS 단절의 응답 지연 시간 차이

1. 개요 및 통신 단절 감지 매커니즘의 본질적 차이

무인항공기(UAV)의 비행 제어 시스템(FCS)은 조종기(RC, Remote Controller)와 지상 관제 시스템(GCS, Ground Control Station)이라는 서로 다른 두 가지 핵심 통신 채널에 의존한다. 이 두 채널이 통신 단절(Loss) 상태에 빠졌을 때, PX4-Autopilot이 이를 인지하고 페일세이프(Failsafe) 상태 기계를 트리거하는 방식은 운영체제(RTOS)의 스케줄링 관점에서 근본적으로 다르다.

RC 신호 단절은 물리적 수신기와의 연결선에서 발생하는 신호의 부재를 포착하는 하드웨어 인터럽트(Hardware Interrupt) 기반으로 처리되어 극히 짧은 응답 지연 시간(Low Latency)을 갖는다. 반면, GCS와의 데이터 링크 단절은 수신된 MAVLink 패킷을 스레드(Thread) 상에서 주기적으로 검사하는 소프트웨어 폴링(Software Polling) 기반으로 동작하여 수 초 단위의 긴 지연 시간(High Latency)을 수반한다. 본 절에서는 이 두 방식의 기술적 차이와 그로 인한 상태 기계 전이의 영향성을 분석한다.

2. 하드웨어 인터럽트 기반 RC 단절(RC Loss) 감지

수신기(Receiver)에서 픽스호크(Pixhawk)와 같은 하드웨어로 조종 입력(SBUS, PPM 등)이 들어오면, PX4 내부의 입출력 프로세서(I/O Processor, IO MCU)나 시리얼 포트 드라이버가 하드웨어 인터럽트를 발생시킨다. 정상적인 상황에서는 고정된 주기(예: 11ms ~ 22ms)마다 프레임이 수신되어 인터럽트를 트리거한다.

  • 장애 감지 시점: 수신기와의 링크가 단절되거나 물리적 선이 끊어지면 펄스 신호나 유효 프레임이 정지한다. 드라이버 레벨의 하드웨어 타이머 혹은 인터럽트 감지 타임아웃(일반적으로 수백 ms 이내)이 작동하여 즉시 input_rc 버스에 RC LOST 상태를 발행한다.
  • 응답 지연 시간: MAVLink 파싱을 거치지 않으므로 감지에서 페일세이프 트리거까지 걸리는 시간은 약 ~500ms(반 초) 이내로 매우 즉각적이다. (설정 파라미터 COM_RC_LOSS_T의 제약을 받는다.)

3. 소프트웨어 폴링 기반 GCS 단절(Data Link Loss) 감지

GCS 통신 단절 평가는 UART 포트 수신 단락이 아니라, MAVLink 프로토콜 계층에 대한 의존적 감시(Dependency Watch)로 이루어진다. PX4의 commander 태스크 모듈은 백그라운드 스레드를 통해 주기적으로(Polling) 시스템 상태를 확인한다.

3.1 타임아웃 폴링 구조와 지연 요소

  • 메시지 기반 평가: GCS에서 전송하는 HEARTBEAT 메시지(통상 1Hz 주기)의 마지막 수신 타임스탬프를 NuttX RTOS 상의 타이머 클럭(hrt_absolute_time)과 비교한다.
  • 파라미터 기반 지연: 허위 텔레메트리 단절 선언을 막기 위해 COM_DL_LOSS_T 파라미터(기본값 10초 등)가 개입하여 의도된 임계 시간 한계를 부여한다.
  • 응답 지연 시간: 설정된 타임아웃 시간(T_{DL})에 스레드 컨텍스트 스위칭 지연, 파서 큐 지연 시간 등을 합산하여 최소 10초 + a의 긴 응답 지연 시간을 갖게 된다.

4. PX4 소스 코드 및 Ardupilot 비교 분석

PX4 펌웨어에서 RC 단절과 GCS 단절의 평가는 서로 다른 계층에서 진입하여 최종적으로 Commander에서 병합(Merge)된다.

// src/modules/commander/Commander.cpp 의 타임아웃 폴링 지연
// GCS 통신 단절의 경우 폴링 기반 평가를 수행함
const hrt_abstime time_now = hrt_absolute_time();
if (time_now - _last_gcs_heartbeat_time > _param_com_dl_loss_t.get() * 1_s) {
    _datalink_loss_active = true; // 10초 이상의 긴 지연 후 플래그가 Set 됨
}

// src/modules/rc_update/rc_update.cpp
// 반면 RC 단절은 하위 모듈에서 즉각적인 타임스탬프 스파이크로 감지
if (time_now - _rc_last_valid_time > _param_com_rc_loss_t.get() * 1_s) {
    // 0.5초 경과 후 즉각적으로 RC Loss 발송
}

Ardupilot 대비 차이: Ardupilot 역시 RC Failsafe는 쓰로틀 채널(Throttle Channel)의 펄스 폭(PWM 값, 예: 975us 이하)이 하드웨어 수신기 범위를 벗어남을 즉시 감별하여 트리거(FS_THR_VALUE)하며, GCS Failsafe(FS_GCS_ENABL)는 MAVLink HEARTBEAT 타임아웃에 의존하는 유사한 이중 구조를 채택하고 있다. 두 시스템 모두 이러한 감지 방식의 다원화를 공유한다.

5. 지연 시간 차이가 시스템 동역학에 미치는 영향

두 채널(RC 주파수 및 텔레메트리 주파수)이 전파 방해 등으로 동시에 소실될 경우, 시스템은 다음과 같은 시간차를 겪게 된다.

  1. Phase 1 (0초 ~ 0.5초): 전파 교란. RC 신호와 텔레메트리 신호가 동시 차단됨.
  2. Phase 2 (0.5초 시점): 하드웨어 타이머 만료로 PX4가 우선적으로 “RC Loss“를 감지. PX4 FSM은 GCS가 아직 살아있다고 판단(폴링 타임아웃 전이므로)하여, 조종 권한을 자동 비행 또는 GCS 지시로 이양시키려 함.
  3. Phase 3 (1초 ~ 10초): PX4는 GCS 명령을 기다리며 대기 궤도(Hold)나 현재 모드를 유지하게 됨 (GCS가 연결되어 있다는 착각(Illusion)).
  4. Phase 4 (10초 경과 후): 비로소 COM_DL_LOSS_T가 만료되어 소프트웨어 기반의 GCS 단절(Data Link Loss)이 감지. 이 시점이 되어서야 PX4는 “이중 상실(Dual Loss)“을 인지하고 비상 귀환(RTL/Return)을 시작함.

자율 비행(ROS2 등)이나 GCS(QGC) 애플리케이션 개발자는 이러한 응답 지연 불일치(Latency Discrepancy)를 반드시 이해해야 한다. 조종기와 텔레메트리가 동시 상실되는 최악의 조건에서 기체가 귀환 스위치를 작동하기 전 최대 10여 초간 제어 사각지대(Hold)에 빠지게 됨을 구조적으로 예측하고, 위험 공역 운용 시 파라미터(COM_DL_LOSS_T)의 임계 시간을 축소하거나 자율 에이전트의 내부 보조 페일세이프 스크립트를 능동적으로 삽입하라.
