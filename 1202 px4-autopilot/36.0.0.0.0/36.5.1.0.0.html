36.5.1. NuttX RTOS 환경 내 mavlink 모듈의 비율 단조 스케줄링(Rate Monotonic Scheduling, RMS)과 스케줄링 가능성(Schedulability) 수학적 검증

1. 개요 및 이론적 배경 (Introduction and Theoretical Background)

무인항공기(UAV)의 비행 제어기(Flight Controller)는 센서 데이터 수집, 자세 및 위치 제어, 그리고 지상 관제 시스템(GCS)과의 통신을 동시에 수행하는 복합 하드 리얼타임(Hard Real-time) 시스템이다. PX4-Autopilot은 POSIX 표준을 준수하는 NuttX 실시간 운영체제(RTOS) 위에서 동작하며, 각 모듈을 독립적인 태스크(Task) 또는 스레드(Thread)로 분리하여 실행한다.

그중 mavlink 모듈은 GCS(예: QGroundControl)와 원격 측정 데이터(Telemetry), 제어 명령, 시스템 상태(Heartbeat)를 교환하는 중추적인 역할을 담당한다. MAVLink 통신은 상대적으로 주기가 길고 대역폭을 많이 차지하지만, 자세 제어 루프나 이스케이퍼(Estimator, EKF2) 모듈과 CPU 자원을 두고 경합하게 된다. 이러한 태스크 간의 경합을 조율하고 모든 하드 리얼타임 데드라인을 만족시키기 위해, 선점형(Preemptive) 스케줄링 기법인 비율 단조 스케줄링(Rate Monotonic Scheduling, RMS)이 아키텍처 베이스라인으로 활용된다.

RMS는 Liu와 Layland가 1973년에 발표한 기념비적인 논문 “Scheduling Algorithms for Multiprogramming in a Hard-Real-Time Environment“에서 제안된 정적(Static) 우선순위 할당 알고리즘이다. 이 알고리즘의 핵심 철학은 **“주기(Period)가 짧을수록 높은 우선순위(Priority)를 부여한다”**는 것이다. Ardupilot과 대비하여, PX4는 NuttX의 풍부한 POSIX 스레딩(pthreads) 인터페이스를 바탕으로 제어 로직과 통신 로직 간의 컨텍스트 스위칭(Context Switching) 및 스케줄링 결정권을 운영체제 수준에서 매우 수학적이고 명시적으로 분리해 낸다는 특징이 있다.

2. PX4 아키텍처 내 태스크 특성과 RMS 우선순위 할당 (Task Characteristics and RMS Priority Allocation)

PX4 시스템은 크게 세 가지 주요 태스크 그룹으로 분류할 수 있다. RMS 원칙에 따르면 가장 짧은 주기를 갖는 센서 획득 태스크가 가장 높은 우선순위를 가지며, mavlink 통신 태스크는 그보다 낮은 우선순위를 갖는다.

  1. High Priority (센서 및 액추에이터 계층): IMU 데이터 샘플링(최대 1kHz~8kHz), PWM 출력 갱신. px4io 또는 센서 드라이버가 포함된다.
  2. Medium Priority (추정 및 제어 계층): EKF2 필터링 및 자세/위치 제어 루프(약 250Hz~400Hz).
  3. Low Priority (통신 및 부가 기능 계층): mavlink 스트리밍, 로깅(Logging) 등. 주기가 주로 50Hz 이하로 설정된다.

이러한 우선순위 할당은 NuttX의 sched_setscheduler() 함수 등을 통해 우선순위로 매핑된다. 따라서 mavlink 모듈은 고속 비행 제어 태스크가 CPU를 점유하지 않는 유휴(Idle) 시간이나 블로킹(Blocking)된 틈을 활용하여 비동기 입출력(Asynchronous I/O)을 수행하게 된다.

3. 스케줄링 가능성(Schedulability)의 수학적 검증 분석

RMS 시스템이 데드라인을 절대 놓치지 않음(Hard Real-time Guarantee)을 보장하기 위해서는 수학적인 스케줄링 가능성 분석이 필수적이다. 이를 위해 각 프로세스의 주기(T_i)와 최악 실행 시간(Worst-Case Execution Time, WCET, C_i)을 통해 프로세서 이용률(Utilization, U)을 계산한다.

3.1 RMS의 충분 조건 (Sufficient Condition)

Liu와 Layland 정리에 따르면, n개의 독립적인 주기적 태스크가 주어졌을 때, 총 프로세서 이용률 U가 다음의 경계값(Bound) 이하이면 시스템은 RMS 하에서 무조건 스케줄링 가능(Schedulable)하다.

U = \sum_{i=1}^{n} \frac{C_i}{T_i} \le n(2^{1/n} - 1)

여기서 태스크의 개수 n \to \infty 일 때, 경계값은 \ln 2 \approx 0.693에 수렴한다. 즉, 시스템 설계 시 총 CPU 이용률 설계치가 69.3% 이하로 유지되도록 튜닝한다면, 어떤 최악의 태스크 위상(Phase) 조합에서도 데드라인 미스(Deadline Miss)가 발생하지 않음을 수학적으로 보증할 수 있다.

3.2 PX4 가상 태스크 모델링 및 하모닉(Harmonic) 주기 최적화

실제 PX4 코드에서는 CPU 이용률의 한계치인 69.3%를 넘어서더라도, 태스크의 주기가 서로 배수 관계(Harmonic)를 이룰 경우 이용률이 100%에 근접해도 스케줄링이 가능하다는 RMS의 필요충분조건(Exact Schedulability Test) 응답 시간 분석(Response Time Analysis)을 도입한다.

다음은 PX4 단위 테스트와 벤치마킹에서 종종 인용되는 내부 모듈의 가상 파라미터를 사용한 이용률 평가 모델의 예시다.

태스크(Task) i역할주기 T_i (ms)실행 시간 C_i (ms)개별 이용률 \frac{C_i}{T_i} \vert 비율
\tau_1 (센서)IMU Read10.10.100 (10%)
\tau_2 (제어기)Attitude 제어41.00.250 (25%)
\tau_3 (통신)mavlink 파싱202.00.100 (10%)

위 모델에서 총 이용률 U = 0.1 + 0.25 + 0.1 = 0.450 (45\%) 이다.
n=3일 때의 RMS 경계값은 3(2^{1/3} - 1) \approx 0.779 (77.9\%)이므로, 해당 시스템 아키텍처에서는 0.45 \le 0.779를 완벽히 만족한다. 이로써 mavlink 모듈이 다른 하드 리얼타임 루프를 방해하지 않고 스케줄링 가능함이 수학적으로 완벽하게 검증된다. 결과적으로 비행 중 QGroundControl의 대량 데이터 송수신 요청이 쇄도하더라도 제어권(Control Authority)이 상실되거나 추락하지 않는다.

4. QGroundControl과의 연동 및 텔레메트리 큐(Queue) 압력 제어

mavlink 스레드는 통신 대역폭을 초과하는 수준의 로깅(Logging) 또는 텔레메트리 전송 요청이 있을 경우, 큐(Queue)가 팽창하여 실행 시간(C_i)이 가변적으로 길어질 수 있는 확률적 요소를 지닌다.
이를 방어하기 위해서 PX4의 mavlink_main.cpp 모듈 내부에서는 다음과 같은 구조적 제어 로직을 채택하고 있다.

  1. 비동기 워크 큐(Async Work Queue) 위임: 과도한 I/O 블로킹 락(Lock)을 유발하는 작업은 지연된 워크 큐(Delayed Work Queue)로 옮겨, 태스크 자체의 최악 실행 시간(WCET)을 제한적 범위 내로 엄격하게 통제한다.
  2. 데이터 누락(Data Drop) 및 속도 조절 전략: UART 전송 버퍼가 임계 여유도(Threshold)를 초과하게 되면, 우선순위가 낮은 텔레메트리 패킷(예: GCS의 텍스트 로그, 비필수 센서 상태)을 의도적으로 누락시켜 C_i를 강제로 재조정한다. 이는 RTOS가 시스템 전체의 RMS 스케줄링 조건을 초과하지 않도록 하는 본연의 메커니즘이다. 반면 중요한 기체 제어 명령(Heartbeat, Arming Command 등)은 별도의 최상위 FIFO 큐에서 보호되어 절대 지연되지 않는다.

5. 결론 (Conclusion)

PX4-Autopilot의 mavlink 모듈은 지상 관제 시스템과의 양방향 통신이라는 막대한 백그라운드 작업을 수행하면서도, NuttX RTOS 기반의 비율 단조 스케줄링(RMS) 정책을 통해 핵심 비행 제어 루프의 실시간성을 침해하지 않도록 정교하게 설계되었다. Liu와 Layland의 수학적 정리와 응답 시간 분석 모델을 코어 운영체제 아키텍처에 긴밀히 통합함으로써, PX4는 트래픽 폭주나 돌발적인 인터럽트 산사태(Interrupt Avalanche)에 대해서도 유연하고 예측 가능한(Deterministic) 방어막을 구축하여 미션의 안전성을 절대적으로 담보한다.
