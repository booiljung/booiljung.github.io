36.1.6. 교착 상태(Deadlock) 회피를 위한 은행원 알고리즘(Banker’s Algorithm)의 다중 스레드 GCS 환경 적용

1. 개요 (Introduction)
지상 관제 시스템(GCS) 소프트웨어 아키텍처는 그 특성상 수십 개의 다중 스레드(워커 스레드, 로깅 스레드, 렌더링 스레드 등)가 메모리 파라미터, 그래픽 버퍼, 데이터베이스 연결 등 한정된 시스템 내부 자원을 놓고 이리저리 경합을 벌이는 고도의 병렬 컴퓨팅(Parallel Computing) 환경이다. 만일 두 개 이상의 스레드가 상호 배제(Mutual Exclusion)된 락(Lock) 자원을 얻기 위해 서로를 무한정 기다리는 교착 상태(Deadlock)에 진입하게 되면, GCS 데스크톱 애플리케이션 자체가 완전히 동결(Freeze)되어 비행 중인 무인기를 상실할 수 있는 치명적 사고로 직결된다. 이를 수학적, 체계적으로 원천 봉쇄하기 위해 운영체제 이론에서 창안된 은행원 알고리즘(Banker’s Algorithm)의 자원 할당 거부(Resource Allocation Denial) 철학을 GCS 코어 백엔드 설계에 추상적으로 차용하고 이식할 수 있다.

2. GCS 환경 내 교착 상태(Deadlock) 발생의 4가지 필요충분조건
컴퓨터 과학에서 정의된 데드락이 GCS 환경 내에서 실질적으로 성립하려면 다음의 네 가지 고전적 조건(Coffman Conditions)이 동시에 스레드 모델 내에 형성되어야만 한다.

  1. 상호 배제 (Mutual Exclusion): Vehicle 객체의 내부 상태 변수나 텔레메트리 파서 버퍼 등은 한 번에 오직 하나의 스레드만 락(Mutex)을 획득하여 사용할 수 있는 독점 자원이다.
  2. 점유 및 대기 (Hold and Wait): 임무 렌더링(Mission Rendering) 스레드가 2D 지도 버퍼(Map Buffer) 락을 쥔 상태로, 동시에 데이터 로거(Data Logger) 스레드가 쥐고 있는 기체 궤적 데이터 락을 추가로 요청하고 기다린다.
  3. 비선점 (No Preemption): 렌더링 스레드가 지도를 다 그리기 전까지는 다른 어떤 스레드나 운영체제 시스템도 그 렌더링 락을 강제로 빼앗을 수 없다.
  4. 환형 대기 (Circular Wait): 스레드 A는 스레드 B의 자원을 원하고, B는 C의 자원을, C는 다시 A의 자원을 원하여 원형 꼬리를 무는 무한 대기 순환 고리가 형성된다.

3. 은행원 알고리즘(Banker’s Algorithm) 메커니즘 개론
에츠허르 데이크스트라(Edsger W. Dijkstra)가 고안한 은행원 알고리즘은 교착 상태 검출(Detection)이나 복구(Recovery)가 아닌, 사전에 시스템이 데드락에 빠지지 않도록 불안전 상태(Unsafe State)로의 자원 할당 시도를 원천 기각해 버리는 회피(Avoidance) 기법이다. 마치 은행이 파산을 막기 위해 최소 지급 준비율 자금을 남겨두고 대출을 승인하는 방식과 동일한 원리를 지닌다.

  • 안전 상태 (Safe State): 시스템 내의 모든 활성 스레드가 자신이 요구하는 최대 자원(Max Claim)을 미래 시점에 모두 할당받고 무사히 실행을 완료(Terminate)하여 자원을 반납할 수 있는 최소한의 시퀀스, 이른바 ’안전 순서(Safe Sequence)’가 하나 이상 수학적으로 존재하는 상태를 일컫는다.
  • 불안전 상태 (Unsafe State): 어떠한 안전 순서도 존재하지 않아, 추후 특정 스레드들이 자원을 요청할 경우 논리적으로 환형 대기(Circular Wait)에 빠질 가능성이 높은 리스크 상태를 지칭한다. 은행원 알고리즘은 시스템이 할당 심사를 거쳐 이 불안전 상태로 넘어가는 요청(Request)을 즉각 불허시킨다.

4. 다중 스레드 GCS 환경 내 자원 관리기(Resource Manager)로서의 적용 설계
현대적인 QGroundControl(QGC)이나 C++ 기반 커스텀 백엔드 관제기 내에서 은행원 알고리즘의 원형(Prototype)을 그대로 탑재하는 것은 CPU 컨텍스트 스위칭 비용 관점에서 대단히 무겁고 비효율적이다. 따라서 GCS 개발자들은 알고리즘의 엄밀한 \mathcal{O}(m \times n^2) 복잡도 수학식을 직접 가동하기보다는, 그 기저 철학을 락 프론트(Lock Front) 매니저로 간략화하여 도입한다.

  • 할당 한계치 및 사전 선언 모델 (Max Claim Declaration): GCS 내 스레드 풀(Thread Pool)에 속한 MAVLink 워커 스레드나 UI 렌더링 스레드는 실행 초기(Initialization) 단계에서 자신이 생애 주기 동안 가용할 수 있는 최대 DB 커넥션 수, 텔레메트리 큐 크기, GPU 렌더링 락의 개수를 시스템(Resource Manager)에 사전 선언해야 한다.
  • 락 순서화(Lock Ordering) 트랜잭션 래핑: 은행원 알고리즘의 가장 중요한 예방적 우회구현례로서, 환형 대기(Circular Wait) 자체를 완전히 파괴하기 위한 락 계층(Lock Hierarchy) 규칙을 수립한다. 만약 스레드가 [A -> B -> C] 자원 순서로만 락을 획득하도록 글로벌 문법을 강제화(디자인 패턴)하면, 은행원 알고리즘의 안전 시퀀스(Safe Sequence)를 논리적으로 무조건 확보할 수 있어 할당 검사 연산 비용을 혁신적으로 제거한다.
  • 가상 할당 심사 (Simulated Allocation & Request Denial): C++ 환경에서 std::unique_lock이나 std::shared_mutex에 try_lock() 타임아웃 기법을 적용할 때, 중앙 컴포넌트 브로커(Broker)는 내부 상태 그래프 행렬식을 빠르게 연산하여 남은 자원이 “안전 상태“를 보장할 경우에만 락 소유를 허가(True)한다. 거부(False)될 경우 해당 스레드의 작업은 데드락 방어막에 의해 즉각 롤백(Rollback)되어 나중 틱(Tick)으로 미뤄진다.

5. Ardupilot 시스템과 PX4 관제(QGroundControl) 구조 내 교착 상태 회피 철학 비교

  • 모놀리식 순차 통제 (Legacy 시스템): 낡은 Ardupilot 기반의 파이썬 스트립트 관제 플러그인 등은 대개 단일 전역 락(Global Interpreter Lock, GIL)이나 거대한 한두 개의 뮤텍스로 전체 데이터베이스를 한 번에 묶어버리는 코스-그레인드(Coarse-grained) 락다운을 시행한다. 이 방식은 아예 다중 락 점유 상황을 만들지 않으므로 은행원 알고리즘의 심사가 불필요하지만, 현대의 다수 에이전트(Swarm) 관제에서는 대기 지연(Latency)이 끔찍하게 치솟는 최악의 설계 병목 구간이 된다.
  • 파인-그레인드(Fine-grained) 분산 락 및 회피 시스템 (PX4/QGC 중심): PX4 아키텍처의 비동기적 성향을 온전히 받아내는 QGroundControl 코어는 차량 상태(Vehicle State), 미션 아이템(Mission Item), 로컬 맵 타일 버퍼마다 락(Mutex) 권한을 고도로 쪼개어 놓았다. 권한이 잘게 분리된 만큼 데드락 확률 또한 역으로 폭증하는데, 이를 제어하고자 내부 객체 생명주기 관리자(Object Lifecycle Manager) 모델을 은행원 알고리즘 철학의 브로커 통제 방식으로 이식하여 ’락 획득 타임아웃 감시 및 강제 해제(Release)’라는 간접적 회피 규약을 철저히 준수함으로써 멀티코어 렌더링 처리량을 타협 없이 방어해 낸다.

6. 결론 (Conclusion)
하드 리얼타임 무인기 통제라는 극한의 제약 속에서, 지상 관제 시스템 프로그램 내부의 교착 상태(Deadlock)는 단순한 소프트웨어 크래시(Crash)를 넘어 막대한 금전적, 물리적 추락 손실 사고를 유발하게 된다. 은행원 알고리즘(Banker’s Algorithm)이 제시하는 ’사전 최대 요구 자원 선언’과 ’안전 상태(Safe State) 심사 후 할당’이라는 컴퓨터 운용체제의 보수적인 안전 보장 수학 철학을 GCS 멀티 스레딩 아키텍처 설계의 뼈대에 치밀하게 투영함으로써, 개발자는 소프트웨어의 예측 불가능한 스레드 멈춤 현상을 영구히 증명하여 잠재우면서도 최고의 병렬 처리 성능을 안정적으로 손에 쥘 수 있다.
