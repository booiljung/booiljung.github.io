36.6. MAVLink 미들웨어 트래픽 라우팅 및 결함 허용(Fault-Tolerant) 이중화

1. 개요 및 복합 네트워크 패러다임 (Introduction to Complex Network Paradigms)

전통적인 드론 관제 환경은 단일 무인항공기와 단일 지상 관제 시스템(GCS) 간의 Point-to-Point(1:1) 무선 링크 토폴로지로 시작되었다. 그러나 최신 PX4-Autopilot 기반의 운용 환경은 컴패니언 컴퓨터(Companion Computer), 스마트 배터리 시스템, 카메라 짐벌(Gimbal) 제어기, 나아가 군집 비행(Swarm) 제어를 위한 다중 GCS 인프라까지 포함하는 방대한 사이버-물리 시스템(CPS)의 로컬 영역 네트워크(LAN)로 진화하였다.

이러한 복잡한 다대다(N:M) 네트워크 토폴로지에서 데이터 충돌 없이 특정 센서나 특정 GCS로 정보가 올바르게 전달되기 위해서는 MAVLink 미들웨어 트래픽 라우팅(Traffic Routing) 기술이 필수적이다. 더불어, 핵심비행 중 주 통신망이 단절되는 치명적인 상황을 방어하기 위해 여러 물리적 네트워크 인터페이스(예: RF 텔레메트리 보드와 4G/LTE 컴패니언 링크)를 통합적으로 묶는 결함 허용(Fault-Tolerant) 이중화 아키텍처가 동반 설계되었다.

2. PX4 내 MAVLink 라우팅 및 인스턴스 분산 아키텍처

MAVLink 프로토콜 프레임 내부에는 각 장치를 식별하기 위한 **시스템 ID(System ID)**와 **컴포넌트 ID(Component ID)**가 존재한다. 라우팅 홉(Hop)을 경유하는 패킷은 네트워크 스택 내부에서 এই ID 쌍을 기반으로 필터링(Filtering) 및 포워딩(Forwarding) 처리된다.

2.1 분산형 MAVLink 인스턴스와 uORB 브릿징

Ardupilot의 경우 GCS_MAVLink 허브 모듈 내에서 모든 직렬 포트에 대한 라우팅 테이블(Routing Table)을 중앙 집중식으로 갱신하고 스레드를 순회(Round-robin)하는 구조를 지닌다. 이는 구현이 직관적이라는 장점이 있으나, 특정 버스에서 과부하가 발생하면 전체 라우터의 처리량이 저하되는 단점을 낳는다.

이에 반해 PX4-Autopilot 시스템은 각각의 물리적 UART 포트나 UDP 소켓마다 독립적인 mavlink_main 인스턴스를 메모리에 별도로 구동(Spawn)시킨다. 즉, TELEM1용 시스템 스레드와 컴패니언 컴퓨터용 TELEM2 스레드가 완전히 격리되어 동작한다.
동일한 드론 네트워크 내에서 이 이더넷 부품들 간의 트래픽 브릿징이 필요할 경우, PX4는 하단에 위치한 **uORB (micro Object Request Broker)**를 내부 연동 버스(Inter-connect Bus)로 활용하여 인스턴스 간 MAVLink 페이로드를 원자적으로 교환 배달하는 혁신적인 분산 아키텍처를 자랑한다.

3. 결함 허용(Fault-Tolerant) 이중화 경로 통제 스택 (Redundancy Framework)

야전 환경에서 단일 433MHz/915MHz RF 모뎀은 전파 방해(Jamming)나 통신 음영 지역 진입 시 완전한 기능 마비 상태를 맞이할 수 있다. PX4는 복수의 통신 채널을 동시에 운용하여 하나의 링크가 손실되었을 때 백업 링크를 통한 심리스(Seamless) 제어를 보장하는 상태 머신 로직을 구현했다.

  1. 동적 페일오버(Dynamic Failover): 비행을 감독하는 코어인 commander 모듈은 내부적으로 등록된 복수의 텔레메트리 링크들로부터 하트비트 토픽(telemetry_status uORB 토픽)의 신선도를 병렬적으로 관측한다. 주 링크(Primary Link)의 패킷 수신이 설정된 타임아웃(Timeout, 기본 3초)을 경과하게 되면 파라미터 로직에 따라 보조 링크(Secondary Link)와의 GCS 연결로 통제권을 즉각적으로 자동 스위칭한다.
  2. 패킷 순차 번호(Sequence Number) 기반의 중복 제거: 두 개의 링크가 동시에 데이터를 쏟아내는 이중 활성(Active-Active) 환경에서 GCS가 송신한 하나의 제어 명령이 두 채널을 통해 중복 유입될 경우가 있다. PX4의 커맨더 모듈은 MAVLink 패킷 내부의 순차 번호를 철저하게 검증해, 이미 승인된 동일한 시퀀스 번호의 명령어(Command)는 중복 실행하지 않고 암묵적으로 소멸(Drop)시킨다.

4. 라우팅 루프 방어 및 브로드캐스트 스톰(Broadcast Storm) 억제

서로 다른 MAVLink 인스턴스가 uORB 버스로 묶이게 되면, A 포트로 수신된 패킷이 uORB로 전환된 뒤 B 포트로 포워딩되고, 다시 컴패니언 컴퓨터의 브릿지를 거쳐 A 포트로 유입되어 무한 순환을 일으키는 브로드캐스트 스톰(Broadcast Storm) 위협이 도사리게 된다.

PX4 아키텍처 내에서는 이를 네트워크 레벨에서 방지하기 위해 각 MAVLink 인스턴스가 자신이 패킷을 uORB에 게시(Publish)할 때, 고유의 하위 인스턴스 ID 플래그를 메타데이터에 각인한다. 수신 측 MAVLink 인스턴스는 해당 데이터를 릴레이할지 말지 결정하기 전, 해당 트래픽이 origin_id = self 인 루프백(Loopback) 패킷인지를 식별하는 스톰 필터링(Storm Filtering) 상태기를 필수적으로 거친다.

6. 결론 (Conclusion)

초창기 단순 명령어 직렬 통신(Serial Handshake) 프로토콜로 탄생한 MAVLink는, 오늘날 PX4-Autopilot의 아키텍처 내에서 독자적인 라우팅 테이블과 필터링 시스템, 그리고 인스턴스 간 격리/융합(Isolation/Fusion) 개념을 탑재한 최신 분산형 마이크로서비스 미들웨어(Microservice Middleware)로 완벽히 격상되었다.
독립스런 프로세스 설계와 uORB 코어 버스의 시너지는 결함 허용(Fault-Tolerant) 이중화 통신망의 구현을 본질적으로 단순화하였으며, 시스템 엔지니어들이 물리적 링크 파괴의 위험성을 딛고 다중 인터페이스 기반의 전천후 극한 비행 임무(Extreme Swarm Mission)를 수행할 수 있게 하는 단단한 인프라를 제공한다.
