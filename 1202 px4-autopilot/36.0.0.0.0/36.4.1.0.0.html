36.4.1. TCP 혼잡 제어(Congestion Control) 알고리즘의 대역폭-지연 곱(Bandwidth-Delay Product, BDP) 환경 내 헤드 오브 라인(Head-of-Line) 블로킹 한계 분석

PX4-Autopilot과 보조 컴퓨터(Companion Computer)를 통한 차세대 관제 시스템은 비디오 스트리밍, 대용량 센서 포인트 클라우드, 그리고 MAVLink 데이터를 통합적으로 전송하기 위해 종종 IP 네트워크(Wi-Fi, LTE, 위성 등)를 활용한다. 이론상으로는 전송 계층에서 가장 무결성이 보장된 TCP(Transmission Control Protocol)를 사용하는 것이 안전해 보일 수 있으나, 실제 무인기(UAV) 원격 제어망에서는 대부분 TCP를 배제하고 UDP(User Datagram Protocol)를 표준으로 삼는다. 본 절에서는 대역폭-지연 곱(BDP)이 큰 비가시권(BVLOS) 환경에서 TCP 혼잡 제어 알고리즘이 유발하는 치명적인 헤드 오브 라인(Head-of-Line, HoL) 블로킹 문제를 학술적으로 규명한다.

1. 대역폭-지연 곱(Bandwidth-Delay Product, BDP)의 이해

네트워크에서 대역폭-지연 곱(BDP)은 통신 채널의 용량(Bandwidth)과 패킷이 목적지를 다녀오는 데 걸리는 왕복 지연 시간(Round-Trip Time, RTT)의 곱으로 정의된다.

BDP = \text{Bandwidth (bits/sec)} \times \text{RTT (sec)}

BDP는 ’네트워크 파이프 내에 물리적으로 머물러 있는(In-flight) 데이터의 총량’을 의미한다. 위성 통신(SATCOM)이나 저속의 장거리 LTE망을 사용하는 BVLOS 드론 환경은 대역폭은 제한적임에도 지연(RTT)이 수백 밀리초(ms)에 달하므로 전형적인 ‘Large BDP (Long Fat Network)’ 특성을 띈다.

2. TCP 혼잡 제어 알고리즘의 동작 방식과 한계

TCP는 네트워크 혼잡을 방지하기 위해 혼잡 윈도우(Congestion Window, cwnd) 크기를 동적으로 조절한다. TCP Reno, CUBIC 등의 알고리즘은 패킷 드롭(Packet Drop)이 발생하면 이를 ’네트워크 라우터의 버퍼 혼잡’으로 간주하고 cwnd를 즉각 절반으로 줄이거나(Multiplicative Decrease), 전송 속도를 초기화(Slow Start)한다.

2.1 무인기 환경에서의 TCP 혼잡 제어 오작동

하지만 PX4 비행 환경에서의 패킷 드롭은 네트워크 혼잡이 아니라, **물리 계층(PHY)의 다중 경로 페이딩이나 신호 감쇠(SnR 저하)**로 인해 발생하는 경우가 절대다수이다. TCP는 이러한 무선 링크의 물리적 손실마저도 ’혼잡’으로 잘못 판단(Misinterpret)하여 전송 속도를 스스로 급감시킨다. 결과적으로 Large BDP 환경에서 줄어든 cwnd가 다시 복구되기까지 긴 RTT 시간이 소요되므로, 실제 사용 가능한 물리적 대역폭을 심각하게 낭비하게 된다.

3. 헤드 오브 라인(Head-of-Line, HoL) 블로킹의 치명성

BDP 문제보다 더욱 치명적인 TCP의 구조적 결함은 데이터의 ‘엄격한 순서 보장(In-order Delivery)’ 메커니즘에서 기인하는 헤드 오브 라인(HoL) 블로킹이다.

TCP는 수신단(예: QGroundControl)의 소켓 애플리케이션에 데이터를 전달할 때 무조건 Sequence Number 순서대로만 전달해야 한다.

  1. 드론에서 GCS로 패킷 #1, #2, #3, #4 가 연속으로 전송된다고 가정하자.
  2. 이 중 패킷 #1 에는 상대적으로 중요도가 떨어지는 BATTERY_STATUS 메시지가 들어있고, 패킷 #4 에는 매우 중요한 ATTITUDE (고도/자세) 메시지나 충돌 회피용 경고가 들어있다.
  3. 무선 페이딩으로 인해 패킷 #2 가 드롭되었다.

이 상황에서 TCP 수신 버퍼는 정상적으로 수신된 패킷 #3 과 #4 를 GCS 애플리케이션(QGroundControl의 MAVLink 파서)에 전달하지 않고 메모리에 가두어(Block) 버린다. 수신단은 송신단에 패킷 #2 의 재전송(Retransmission)을 요청하며, 수백 ms의 RTT를 거쳐 패킷 #2 가 다시 도착할 때까지 대기한다.

sequenceDiagram
    participant PX4 as PX4 / Companion PC
    participant Air as Wireless Channel
    participant GCS as QGroundControl (TCP Socket)
    
    PX4->>Air: Pkt #1 (Battery)
    Air->>GCS: Received Pkt #1 -> Parced immediately
    PX4->>Air: Pkt #2 (GPS Info) -> DROPPED
    PX4->>Air: Pkt #3 (Log Data)
    Air->>GCS: Received Pkt #3 -> Buffered (HoL Block)
    PX4->>Air: Pkt #4 (ATTITUDE - CRITICAL)
    Air->>GCS: Received Pkt #4 -> Buffered (HoL Block)
    
    Note right of GCS: QGC interface freezes for 500ms!<br/>Waiting for Pkt #2 retransmission.
    
    GCS-->>PX4: NACK / Duplicate ACK for Pkt #2
    PX4->>Air: Retransmit Pkt #2
    Air->>GCS: Received Pkt #2
    Note right of GCS: Now Pkt #2, #3, #4<br/>are delivered to MAVLink all at once.

이러한 HoL 블로킹 현상 때문에 실시간 자세 제어가 생명인 조종석(GCS) UI의 인공 수평의가 수백 밀리초 동안 멈춰버리는(Freeze) 치명적인 렉(Latency Jitter)이 유발된다.

4. 결론 및 PX4 아키텍처 관점

TCP의 신뢰성 보장 메커니즘은 유선 인터넷 망의 파일 다운로드에는 적합하지만, 실시간성과 신선도(Freshness)가 절대적인 무인항공기 관제 스트림에는 최악의 시나리오를 초래한다.

따라서 PX4-Autopilot 생태계의 모든 기본 텔레메트리 연결(MAVLink over UDP) 및 최신 ROS2 Micro XRCE-DDS 미들웨어 설계는 HoL 블로킹을 원천적으로 피하기 위해 베스트 에포트(Best Effort) 기반의 비연결성 전송 프로토콜(UDP)을 강제한다. MAVLink 프로토콜은 순서가 뒤바뀌거나 누락된 패킷에 대해서 애플리케이션 계층 검증(MAVLink Sequence Number 확인)을 통해 오래된 데이터를 그냥 버리거나(Drop), 파일 전송(FTP)이나 미션(Mission) 업로드와 같이 신뢰성이 반드시 필요한 패킷만을 식별하여 애플리케이션 계층 내부 로직(ACK/Timeout 구현)으로 재전송을 통제하는 설계 철학을 준수하고 있다.
