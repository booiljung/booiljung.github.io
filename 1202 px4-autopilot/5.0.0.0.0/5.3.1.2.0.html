5.3.1.2. Ardupilot (EKF3): 다중 인스턴스(Multi-lane) 병렬 처리 기반 생존성 극대화

단일 코어(Single Core) 마이크로컨트롤러 환경에서 극한의 연산 최소화를 이룩했던 PX4(ECL/EKF2)의 단조(Monolithic) 인스턴스 철학과는 대조적으로, 최근 하드웨어의 발전(STM32 H7 등 고성능 칩셋 보급)과 함께 Ardupilot 진영은 상태 추정기의 기본 사상을 **‘생존성(Survivability)과 다중 이중화(Redundancy)’**로 완전히 전환했습니다.

그 결과물이 바로 현재 Ardupilot의 표준 상태 추정 엔진인 EKF3 (Extended Kalman Filter v3) 이며, 이 필터 엔진의 가장 두드러진 아키텍처적 특징이 바로 다중 인스턴스(Multi-lane) 병렬 처리입니다. 본 절에서는 EKF3가 어떻게 하나의 칩셋 안에서 여러 개의 필터를 동시에 굴려 기체의 추락을 방지하는지 분석합니다.

0.1 단일 EKF 인스턴스의 치명적 약점: “모든 계란을 한 바구니에 담기”

과거의 EKF 아키텍처(Ardupilot EKF2나 초기 PX4 EKF2)에서는 기체에 2개의 IMU(관성측정장치)와 2개의 GPS가 장착되어 있을 경우, 센서 계층에서 두 데이터의 평균을 내거나 양호한 신호를 ’선택(Selection)’하여 단일 EKF 엔진 안으로 집어넣었습니다.

하지만 이 방식에는 논리적인 구멍이 존재합니다.

  1. 만약 1번 IMU 센서가 하드웨어 결함에 의해 갑자기 비정상적인 진동 값(G-force spike)을 뿜어낸다고 가정해 봅시다.
  2. 단일 EKF 엔진은 이 오염된 데이터를 그대로 삼킨 뒤 자신의 24차원 내부 상태(State Vector) 공분산을 완전히 망가뜨려 버립니다.
  3. 뒤늦게 센서 브로커 모듈이 “1번 IMU가 미쳤으니 2번 IMU로 교체해!” 라고 명령해도, 이미 유일한 EKF 엔진 내부의 수학적 지평선(Horizon)은 오염되어 발산(Divergence)해버렸으므로 드론은 뒤집혀 추락하게 됩니다.

Ardupilot 통계상 상업용 드론 추락 사고의 절반 가까이가 이처럼 ’센서 글리치(Glitch)에 의한 EKF 공분산 오염’에서 비롯되었습니다.

0.2 Multi-lane 아키텍처: 완벽히 독립적인 가상 두뇌의 동시 구동

Ardupilot EKF3는 위와 같은 파국을 막기 위해 **“센서가 2개면 EKF 필터도 2개를 돌려버리자”**는 극단적이고 무식하지만 가장 확실한 해법을 채택했습니다. 이를 다중 인스턴스, 이른바 멀티 레인(Multi-lane) 아키텍처라고 부릅니다.

매개변수 풀(Parameter) 설정 옵션인 EK3_IMU_MASK 등에 의해, 펌웨어는 부팅 시점에 아예 완전히 독립적인 메모리 영역을 가진 여러 개의 EKF3 코어 객체(Core Object)를 생성합니다.

graph TD;
    subgraph Sensors
        IMU1[IMU 1 (Pixhawk 내장)]
        IMU2[IMU 2 (외부 댐핑)]

    GPS1 --> CORE1

위 다이어그램처럼 Core 0은 1번 IMU만 먹고 자라며, Core 1은 2번 IMU만 먹고 자랍니다. 두 행렬 엔진은 서로의 연산에 전혀 관여하지 않고 독립적으로 병렬 계산을 묵묵히 수행하여 각자의 정답(자세, 위치)을 제어기에 제출합니다.

0.3 무거운 연산의 댓가: CPU 부하와 생존성의 트레이드오프(Trade-off)

앞 절에서 살펴보았듯, 24차원 EKF 알고리즘은 단 1개만 돌려도 CPU를 헐떡이게 만드는 무거운 알고리즘입니다. 그런데 이를 2개, 3개씩(최대 3코어) 동시에 돌린다면 과거의 168MHz 급 비행 제어기(Pixhawk 1)에서는 스케줄러가 터져버릴 수밖에 없었습니다.

Ardupilot 진영이 이러한 무거운 멀티 레인 기반을 공식화할 수 있었던 결정적인 배경은 하드웨어의 상향 평준화입니다. 최근 출시되는 Pixhawk 4/5X/6X 급 (Cortex-H7 400MHz 이상 모델)에서는 EKF 3개를 동시 구동해도 CPU 점유율이 여유 있게 방어됩니다.

따라서 Ardupilot은 “연산 효율을 위해 코드를 극도로 쥐어짜기보다는, 넘쳐나는 CPU 클럭 자원을 온전히 시스템 다중화(Redundancy)에 태워버리겠다“는 명확한 설계 노선을 정한 셈입니다.

만약 앞선 시나리오처럼 IMU 1번이 미쳐 날뛴다면 어떻게 될까요?
EKF3 Core 0의 내부 수학 상태는 완전히 붕괴되어 발산해 버릴 것입니다. 하지만 셀렉터(Selector/Voter) 모듈은 이를 즉시 감지하고 조용히 추종 대상을 건강하게 살아있는 EKF3 Core 1 로 스위칭(Switching) 해버리기만 하면 끝납니다. 메인 모터 제어 입장에서는 아무런 오염(Contamination) 없이 평온하게 비행을 이어갈 수 있는, 소름 돋는 무결성(Integrity)이 실현된 것입니다.
