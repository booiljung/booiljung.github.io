5.2.2.2.1. 전역 변수와 객체 포인터를 통한 직관적이고 빠른 데이터 접근

드론의 자동 비행 소프트웨어라는 복잡한 세계에서, 개발자가 가장 골머리를 앓는 문제는 단연코 데이터를 주고받는 방식입니다. “어떻게 하면 센서A의 데이터를 제어기B 상에서 단 1밀리초의 지연도 없이 가장 빠르고 정확하게 가져올 수 있는가?“라는 이 단순한 질문에 대해, Ardupilot 커뮤니티가 도출해 낸 해답은 어쩌면 현대적 소프트웨어 공학의 ‘객체 지향 은닉(Encapsulation)’ 철학을 살짝 거스르는 듯하면서도 엄청나게 실용적인 해법입니다.

이 절에서는 PX4의 uORB 모델과 완전히 대비되는, Ardupilot의 ‘전역 변수 수준의 객체(Singleton) 및 포인터 직접 접근(Direct Pointer Access)’ 메커니즘이 소프트웨어의 동작 속도와 이식성에 어떠한 영향을 미치는지 살펴봅니다.

0.1 직관의 승리: “있는 곳으로 직접 가서 꺼내 먹어라”

PX4의 미들웨어인 uORB를 이용할 경우, 개발자가 IMU 가속도 값을 읽어오기 위해 거쳐야 하는 절차는 다소 번거롭습니다. 파일 디스크립터(File Descriptor)를 열고(orb_subscribe), 업데이트 여부를 확인하고(orb_check), 구조체로 데이터를 안전하게 복사(orb_copy)해야 합니다. 즉, “메시지를 택배로 받는 구조“입니다.

반면 Ardupilot의 개발 방식은 매우 원초적이고 직관적입니다. “창고(전역 객체)가 어디인지 다들 알고 있으니, 방문해서 꺼내 먹어라“라는 방식입니다.

// Ardupilot 내부에서 매우 빈번하게 볼 수 있는 코드 호출 패턴 (Direct Getter)
#include <AP_InertialSensor/AP_InertialSensor.h>

void MyAwesomeModule::update_logic() {
    // AP::ins()는 Ardupilot의 센서 싱글톤을 전역적으로 획득하는 마법의 포인터
    AP_InertialSensor &ins = AP::ins();
    
    // 미들웨어 큐(Queue)를 거치지 않고, 헤더파일의 메서드로 메모리에서 값을 직통으로 뽑아옴
    const Vector3f &accel = ins.get_accel(0); // 0번 인스턴스(첫 번째 IMU) 가속도
    const Vector3f &gyro = ins.get_gyro(0);   // 자이로 추출
    
    // 읽어온 값으로 나의 연산을 0 지연 시간으로 즉각 수행...
}

C++ 클래스를 다루는 초급 프로그래머조차 이해할 수 있는 매우 평이한 구조입니다. AP::라는 네임스페이스 하위에 위치한 전역 수준의 싱글톤 객체(예: AP::ahrs(), AP::gps(), AP::compass())들은 거의 모든 하위 모듈(libraries/)에서 헤더 인클루드만으로 100% 개방되어 있습니다.

0.2 속도(Velocity)와 성능 오버헤드 ’Zero’의 마법

이러한 직접 포인터 스니핑(Pointer Sniffing) 방식의 장점은 두말할 나위 없이 퍼포먼스(성능)에 있습니다.

데이터를 미들웨어로 밀어 넣거나 빼내는 과정에서 발생하는 시스템 콜(System Call)이 전혀 배제됩니다. 컴파일러는 이 get_accel() 함수를 단순 인라인(Inline) 어셈블리 코드로 취급하여 최적화해 버리기 때문에, 메모리 번지수 점프(Jump) 외에는 CPU 사이클 낭비가 사실상 ’0’입니다. 또한, 임베디드 장비에서 극도로 경계해야 하는 메모리 할당(malloc) 및 동적 복사본 남발이 일어나지 않고 가벼운 참조(Reference &)만으로 해결되므로 MCU의 SRAM과 스택(Stack)을 극도로 아낄 수 있습니다.

0.3 구조적 대가: 스파게티성 강결합(Tight Coupling)의 위협

하지만 이 실용주의는 소프트웨어 공학 측면에서 양날의 검입니다. 데이터를 전역 포인터로 사방에서 직접 찔러 읽기 시작하면, 모듈 간의 의존성이 폭발적으로 증가합니다.

  1. 빌드(Build) 의존성: 나의 새 모듈 하나를 컴파일하기 위해, Ardupilot의 거대한 코어 헤더파일 수십 개가 줄줄이 엮여 들어와야만 합니다.
  2. 독립성 훼손: PX4처럼 센서 칩셋을 내가 만든 앱으로 손쉽게 ’교체 비행(Hot-swapping)’하기 위해서는 Ardupilot의 라이브러리 내부 클래스 의존성을 일일이 뜯어고쳐야 합니다. 특정 기능을 완전히 격리하여 모듈화(Decoupling)하는 것이 사실상 불가능에 가깝습니다.

결과적으로, 이 전역 직접 접근 구조는 C++ 프로그래밍에 갓 입문한 초보 DIY 메이커가 기능을 파악하고 살을 붙여 넣기에는 너무나도 훌륭한 “낮은 진입 장벽“을 만들어 주었고, Ardupilot 생태계가 다양한 기체로 폭발적으로 번져나가는 원동력이 되었습니다. 그러나 거대 다국적 방산/드론 기업들이 시스템을 규격화하여 분할 개발하기에는 악몽 같은 구조를 탄생시키기도 했습니다. 극과 극의 공학 패러다임이 펌웨어 코어에 녹아있는 셈입니다.
