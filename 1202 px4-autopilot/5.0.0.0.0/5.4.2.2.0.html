5.4.2.2. Ardupilot: AP_Motors 라이브러리 내 하드코딩된 기체 프레임별 모터 매트릭스

PX4가 런타임(Runtime)에 행렬을 동적으로 연산하는 ’소프트웨어 엔지니어링적 우아함’을 택했다면, Ardupilot은 연산 속도의 극대화와 철저한 휴리스틱 방어막 구축을 위해 **‘미리 굳혀진(Hard-coded) C++ 라이브러리 구조’**를 택했습니다.

그 핵심에 있는 것이 바로 Ardupilot 생태계의 구동을 책임지는 거대한 심장, AP_Motors 라이브러리입니다.

0.1 프레임 종속적인 모터 클래스 설계

Ardupilot 소스 코드를 열어보면, 쿼드콥터(AP_MotorsQuad), 헥사콥터(AP_MotorsHexa), Y6 동축반전 프레임(AP_MotorsY6) 등 전 세계에서 상상할 수 있는 거의 모든 드론의 기체 형태가 C++ 자식 클래스(Child Class)로 낱낱이 상속 구현되어 있습니다.

각 클래스는 초기화 부팅 과정에서 add_motor()라는 함수를 호출하여 믹싱 매트릭스를 물리적인 변수 방(Array)에 하드코딩해 넣습니다.
예를 들어 표준 Quad-X 프레임의 우측 전방(Front-Right)에 위치한 1번 모터의 경우, 기체를 앞으로 숙이는 데(Pitch Down) 절반의 기여를 하고, 왼쪽으로 기울이는 데(Roll Left) 절반의 기여를 하며, 반시계방향(CCW)으로 돌며 반작용으로 기체를 오른쪽으로 비틀어버리는(Yaw Right) 힘을 낸다는 사실을 다음과 같은 형태의 인자로 박아 넣습니다.
add_motor(1, 45도 위치, Roll Factor:-0.5, Pitch Factor: 0.5, Yaw Factor: 1.0)

0.2 출력 연산: output_armed() 함수의 직관성

동적 역행렬(Pseudo-inverse) 계산을 거치는 PX4와 달리, Ardupilot의 output_armed() 메인 믹서 함수는 단 한 번의 배열 순회(For loop)로 모터 출력을 결정해버리는 무식하지만 압도적으로 빠른 퍼포먼스를 자랑합니다.

제어기에서 [Roll, Pitch, Yaw] 요구량이 들어오면, 앞서 선언된 각 모터의 Factor(상수)에 요구량을 곱해서 더하기만 하면 개별 모터의 증감분(PWM offset)이 튀어나옵니다. 여기에 스로틀(Thrust) 기본값을 더해주면 곧바로 모터 출력 연산이 끝납니다. 마이크로컨트롤러(MCU) 입장에서는 부동소수점 곱셈 몇 번으로 끝나는 극저지연(Ultra-low latency) 구조입니다.

0.3 Ardupilot의 포화 억제(Saturation Handling) 철학: 무자비한 자르기

이러한 직관적인 방정식 체계에서 Ardupilot의 진가가 발휘되는 지점은, 출력값이 한계점(예: 최대 출력 1.0)을 돌파하려 할 때 작용하는 우선순위 타협(Prioritization limits) 로직입니다.

  1. 모터 정지 절대 불가 (MOT_SPIN_MIN): 비행 중 어떤 급격한 조작이 들어오더라도 모터 출력값이 프로펠러가 멈춰버리는 0으로 떨어지는 것을 강력하게 방어합니다. 비행 중 프로펠러가 멈추면 재시동에 시간이 걸려 치명적인 추락으로 이어지기 때문입니다.
  2. Yaw의 무자비한 희생 (yaw_headroom): 기체가 풀 스로틀로 상승하면서 동시에 롤(Roll)/피치(Pitch) 조작이 들어오면, 모터 한두 개는 반드시 100% 한계치에 강하게 부딪힙니다(Saturation). 이때 Ardupilot은 자세(Attitude)를 유지하기 위해 전체 추력(Throttle)을 깎아내는 PX4의 사상을 일부분 공유하지만, 수평 뱅크각(Roll/Pitch) 유지를 위해 기수의 회전(Yaw) 명령을 가차 없이 죽여버리는 데에 훨씬 더 공격적입니다.
    조종사가 러더 스틱을 비틀더라도 기체 생존에 위협이 된다면 믹서가 스스로 판단하여 조종사의 Yaw 입력을 무시해 버립니다.

0.4 아키텍처적 장단점 평가

결론적으로 Ardupilot의 AP_Motors 믹싱 아키텍처는 수식 역산의 간결성과 수만 번의 시행착오를 거친 무수한 클리핑(Clipping) 리미터들이 합쳐진 **‘경험의 결정체’**입니다. 프레임이 약간 뒤틀려 있거나 모터 4개의 성능이 제각각이어도 절대로 추락하지 않고 기어코 비행을 만들어냅니다.

단, 치명적인 단점도 존재합니다. 이전에 없던 완전히 기형적이고 비대칭적인 로터 배치를 가진 신종 드론 프레임을 개발했다면, PX4에서는 설정창에 숫자 몇 개를 타이핑하여 5분 만에 세팅을 끝낼 수 있는 반면, **Ardupilot에서는 반드시 원본 C++ 소스 코드를 열어 AP_Motors 하위 클래스를 새로 C 코딩하고 펌웨어를 통째로 재빌드(Re-build)**해야만 비행이 가능합니다.

이러한 모터 제어 철학의 차이는 곧 “누구나 쉽게 개발하고 붙여 쓰는 범용 연구 플랫폼(PX4)“과 “어떤 악조건에서도 신뢰성을 보장해야 하는 실전 산업용 플랫폼(Ardupilot)“의 정체성을 가장 잘 대변하고 있습니다.
