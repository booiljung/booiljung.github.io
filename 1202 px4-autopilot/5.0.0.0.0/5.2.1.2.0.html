5.2.1.2. Ardupilot: ChibiOS 기반의 경량화된 통합 라이브러리(Monolithic) 아키텍처

PX4가 POSIX 다중 독립 스레드를 띄워 우아한 모듈성(Modularity)을 과시할 때, **Ardupilot(아두파일럿)**은 “무거운 프로세스 분할과 잦은 컨텍스트 스위칭(Context Switching)은 필연적으로 비행 제어 사이클의 타이밍 손실(Latency)을 유발한다“는 확고한 실용적 안티테제(Antithesis)를 바탕으로 설계되었다.

Ardupilot은 과거의 운영체제 없는 슈퍼루프(Super-loop) 방식의 한계를 극복하기 위해 RTOS를 도입하긴 했으나, 이는 PX4와는 전혀 다른 방식인 **‘ChibiOS 기반의 경량화된 통합 라이브러리 기반 모놀리식(Monolithic) 아키텍처’**로 진화했다.

0.1 RTOS(ChibiOS)의 도구화와 AP_HAL 인터페이스

Ardupilot에게 운영체제란, 아키텍처를 뒤흔드는 철학(Philosopy)이라기보다는 다양한 잡다한 칩셋의 하드웨어 입출력을 편하게 다루기 위한 소모적인 추상화 도구에 가깝다.

과거 NuttX나 FreeRTOS 등 다양한 RTOS 위에서 포팅되던 Ardupilot은 현재 메인스트림 OS로 ChibiOS/RT를 채택하고 있다. ChibiOS는 POSIX 규격의 이식성이니 파일 구조 모델(open/read)이니 하는 웅장함 대신, 오로지 MCU 위에서 바이트 속도 수준으로 매우 빠르게 스레드를 전환하고 인터럽트를 쳐내는 ’경량성과 초고속성’에 올인한 임베디드 코어다.

Ardupilot은 이 ChibiOS의 커널(Kernel) 바로 위에 AP_HAL(Hardware Abstraction Layer)이라는 두꺼운 자체 C++ 인터페이스 레이어를 덮어씌운다. 개발자들은 보드 OS가 ChibiOS인지, Linux 보드(Navio)인지 전혀 알 필요 없이, 오직 AP_HAL 라이브러리의 클래스 원형(hal.spi->, hal.gpio-> 등)만을 호출하여 전역적인 하드웨어 자원을 직접 할당받아 조작한다.

0.2 모놀리식(Monolithic) 아키텍처의 비대한 단일 몸통 구조

PX4 시스템 소스 코드가 수십 개의 독립된 폴더(modules) 안에서 모르게 격리되어 돌아간다면, Ardupilot은 아키텍처의 지평선 전체가 하나의 거대한 통합 C++ 객체들의 거미줄로 이어져 있는 모놀리식 덩어리를 취한다.

ArduCopter, ArduPlane 같은 기체별 메인 파일이 빌드될 때, 수백 개의 센서 통신 라이브러리(AP_RangeFinder), 칼만 필터 예측 모델(AP_NavEKF3), 모터 출력 로직(AP_Motors) 등 온갖 라이브러리 클래스들이 독립된 프로세스로 띄워지는 것이 아니다. 전역 객체화되거나 포인터로 얽힌 채 단일 실행 바이너리(Single Binary) 안에 꽉꽉 뭉쳐 들어가 강하게 결합(Tightly-coupled)된 형태로 덩치를 키운다.

이러한 구조 속에서는 내부 모듈 통신을 위해 PX4처럼 uORB(메시지 큐 방식)라는 거추장스럽고 무거운 퍼블리시-서브스크라이브 직렬 변환(Serialization) 과정마저 거부한다. Ardupilot에서는 AP_Vehicle이 현재 GPS 상태를 알고 싶으면, 직접 AP_GPS 클래스의 퍼블릭 멤버 변수 래퍼(Wrapper)나 포인터를 끌어와 실시간 램(RAM) 주소의 메모리를 바로 읽어 들여 리턴 값을 도출해 버린다.

이 다이어그램처럼, 센서가 인터럽트를 발생시키고 칼만 필터를 거쳐 모터에 믹싱된 PWM 각속도 제어 값을 뿜어내기까지, 데이터 직렬화 복사나 스레드 간 권한(Context Switch)의 양보를 기다리는 대기 시간이 수학적 최소치에 도달하는 압도적인 로우 레이턴시(Low Latency) 타겟팅이 성립된다.

그러나 대가가 따른다. 코드가 극도로 밀접하게 얽혀 있으므로, AP_Library 한 구석에 있는 사소한 외부 센서 드라이버 배열 참조 오류(Array Index Out of Bound) 하나가 전체 단일 몸통의 무결성을 깨뜨리며 메인 드론 추락이라는 거대한 버그(Cascading Failure)로 연쇄 작용할 위험에 항시 노출되어 있다. 따라서 Ardupilot은 이를 회피하기 위해 컴파일 단계 검증을 미친 듯이 까다롭게 끌어올리는 살인적인 방어 코딩의 성지가 될 수밖에 없다.
