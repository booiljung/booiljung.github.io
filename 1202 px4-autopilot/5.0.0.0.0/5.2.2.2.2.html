5.2.2.2.2. 다중 스레드 동기화(Mutex/Semaphore) 및 경쟁 조건(Race Condition) 회피 전략

앞선 절에서 Ardupilot이 전역 변수 수준의 객체(Singleton) 포인터에 직접 접근하여 미들웨어 오버헤드 없이 빛의 속도로 데이터를 읽어오는 체계를 가졌다고 설명했습니다. 여기서 한 가지 날카로운 의문이 생깁니다.

“만약 A라는 스레드(ex: 백그라운드 GPS 파싱 스레드)가 AP::gps() 객체 안에 새로운 위성 데이터를 한창 쓰고(Write) 있는 와중에, 메인 루프(Main Thread)가 갑자기 끼어들어 데이터를 읽어(Read)가려고 한다면, 데이터가 반쯤 찢어진 채로(Corrupted) 읽히지 않겠는가?”

이 현상을 전산학에서는 **경쟁 조건(Race Condition)**이라고 부르며, 비행 드론에서 이것이 발생하면 기체가 순식간에 통제 불능인 쓰레기 값(Garbage data)을 참조하여 추락하게 됩니다. 동기적 구조를 주무기로 하는 Ardupilot은 이 무서운 동시성(Concurrency) 파괴를 어떻게 회피하고 있을까요? 바로 **뮤텍스(Mutex), 세마포어(Semaphore)**의 엄격한 규율과 백그라운드 스레드의 철저한 고립 정책입니다.

0.1 세마포어(Semaphore) 방어벽: 동시 접근 절대 금지

Ardupilot은 치비OS(ChibiOS)와 같은 RTOS를 기저(Backend) 엔진으로 탑재하고 있습니다. 비록 비행 자체는 하나의 거대한 메인 메가 루프(Main Loop)에 의존하지만, I/O 통신(I2C, SPI 통신 등)의 경우 대기 시간(블로킹)을 방지하고자 여러 백그라운드 워커 스레드들을 반드시 돌려야만 합니다.

경쟁 조건 회피를 위한 가장 기본적인 방어 수단은 세마포어(Semaphore) 락(Lock) 구조입니다. 특정 데이터를 보유하고 있는 싱글톤 객체 내부에는 보통 HAL_Semaphore 변수가 잠금장치로 내장되어 있습니다.

  1. **백그라운드 스레드(I/O)**가 하드웨어 통신 칩에서 SPI 버스로 새로운 자이로/가속도 센서 값을 받아왔습니다. 배열 변수에 이걸 기록(Write) 하기 직전, 스레드는 _semaphore.take()를 호출해 금고문을 잠급니다. 쓰는 동안 아무도 못 들어옵니다. (Lock)
  2. 찰나의 순간, 그사이 우리의 400Hz짜리 메인 루프가 이 센서 값을 포인터로 제발 읽어오겠다(Read)고 접근하려 시도합니다.
  3. 메인 루프가 get_accel() 함수 안으로 들어서자마자 _semaphore.take_nonblocking()을 만나게 됩니다. 하지만 이미 I/O 스레드가 락을 걸었으므로 메인 루프는 진입(Read)에 실패합니다.
  4. 메인 루프는 가다리지 않고(Non-blocking) 포기한 뒤 ’가장 최근에 안전하게 읽어 두었던 과거의 복사본 캐시 값’으로 제어를 이어갑니다. 메인 루프가 절대 정지(Sleep)하지 않고 흘러가게 두는 것이 핵심입니다.

위와 같이 백그라운드 스레드와 메인 스레드가 직통으로 읽고 쓸 때, Ardupilot은 HAL(Hardware Abstraction Layer) 수준의 고속 세마포어 유틸리티를 투입하여 메모리 찢어짐을 완벽히 봉쇄합니다.

0.2 Lock-less 링 버퍼(Ring Buffer) 통신 채널

세마포어 방식을 너무 남발하면 어떻게 될까요? 서로가 서로의 자원을 기다리느라 교착 상태(Deadlock)에 빠지거나, 정작 필요한 제어 루프가 너무 빈번히 락(Lock)에 부딪혀 밀려버릴 수 있습니다. 이 교착 구멍을 해결하기 위해 Ardupilot은 스레드 간 대용량 데이터 전송에 있어 락프리(Lock-Free) 링 버퍼(Ring Buffer/Circular Buffer) 자료 구조를 극도로 애용합니다.

  • ObjectBuffer 혹은 RingBuffer 클래스로 명명된 이 구조는 배열 양 끝단에 헤드(Head)와 테일(Tail) 인덱스를 둡니다.
  • 백그라운드 스레드는 오직 배열에 값을 집어넣으면서 테일(Tail) 포인터만 전진시킵니다(Producer).
  • 메인 스레드는 배열에서 값을 빼가면서 오직 헤드(Head) 포인터만 쫓아갑니다(Consumer).
  • 두 포인터가 맞물리지 않는 한 서로 동일한 데이터를 절대 건드리지 않으므로, 뮤텍스 락(Mutex Lock)을 잡는 낭비 자체를 회피합니다.

로깅(Logging) 모듈과 MAVLink 통신 페이로드는 이러한 링 버퍼에 데이터를 무구히 쏴버리는 구조로 이루어져, 시스템 동기화의 오버헤드를 우아하게 빗겨 나갑니다.

0.3 결국 돌아온 분할, 비동기 스레드 구조 최적화의 난관

문제는, 펌웨어가 비대해지고 컴퓨터 비전과 회피 알고리즘이 점점 Ardupilot 코드 안으로 침투하면서 이 단순 명료했던 동기 시스템에 막대한 금이 가고 있다는 데에 있습니다. 수십 개의 센서 포인터마다 세마포어 락 구조가 치장되어야 하니 코드는 점점 복잡해지고, 이는 결국 이 서적에서 대비한 PX4 uORB와 같은 완전 분산 미들웨어 도입의 압박을 가중시킵니다.

정리하자면, 다중 스레드 환경하에서 고전적인 ’전역 변수 직접 접근’의 이점을 살리기 위해, Ardupilot은 세마포어와 락프리 링 버퍼라는 방패를 겹겹이 두르고 그 성능의 아슬아슬한 줄타기(Race condition avoidance)를 현재까지도 훌륭하게 완수해 내고 있습니다.
