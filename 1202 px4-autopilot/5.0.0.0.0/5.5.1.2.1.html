5.5.1.2.1. 각 비행 모드가 스스로의 진입 조건과 초기화 로직을 캡슐화(Encapsulation)

Ardupilot의 분산형 상태 머신 철학을 가장 잘 보여주는 기술적 특징은 객체 지향 프로그래밍(OOP)의 핵심 원리인 **‘캡슐화(Encapsulation)’**의 극단적인 적용입니다. PX4가 모드 진입 허가권을 중앙(Commander)에서 거머쥐고 있다면, Ardupilot은 각 비행 모드 클래스(Class) 내부로 그 권한과 책임을 깊숙이 밀어넣었습니다.

0.1 수문장 역할의 위임: init() 메서드

Ardupilot에서 조종사가 조종기 스위치를 튕겨 특정 비행 모드(예: Loiter)를 요청하면, 시스템의 메인 루프는 해당 비행 모드 객체의 init(bool ignore_checks) 메서드를 단 한 번 호출합니다.

이 init() 함수는 해당 비행 모드 클래스 내부에 정의된 **진입 조건 검사소(Self-Gatekeeper)**입니다.
예를 들어 ModeLoiter::init() 블록 내부를 들여다보면 다음과 같은 일련의 캡슐화된 로직이 실행됩니다.

  1. EKF 상태 직접 조회: position_ok() 함수를 호출하여 현재 상태 추정기(EKF3)가 자신의 3차원 위치를 오차 한계 내에서 확신하고 있는지 묻습니다.
  2. Home 위치 검증: 홈(Origin) 좌표가 설정되어 이탈 한계선(Geofence)을 계산할 수 있는지 점검합니다.
  3. 초기화 연산: 조건이 모두 참(True)이라면, 현재 공중에 떠 있는 자신의 고도(Altitude)와 XY 좌표 수치를 캡처하여 ‘목표 유지 지점(Target Setpoint)’ 변수에 덮어씁니다. 이는 모드가 시작되자마자 기체가 엉뚱한 곳으로 튀지 않게 막아줍니다.

만약 1번이나 2번 조건에서 하나라도 미달되면 init() 함수는 가차 없이 false를 반환합니다. 그러면 상위 스위칭 로직은 모드 전환을 즉각 거부하고 원래 있던 안전한 모드로 기체를 되돌려보냅니다.

0.2 고립된 제어 생태계: run() 메서드

init()의 깐깐한 심사를 통과하여 해당 모드로 진입하는 데 성공하면, 시스템은 초당 400번(400Hz)씩 그 모드 객체의 run() 메서드만을 맹목적으로 호출합니다.

ModeLoiter::run() 내부에는 또 다른 캡슐화된 소우주(Micro-cosmos)가 펼쳐져 있습니다.

  • 독자적인 스틱 해석: 이 블록 안에서는 조종기의 피치(Pitch) 스틱을 앞으로 민 행위를 단순히 기체를 숙이라는 각도 명령(Angle Command)으로 해석하지 않습니다. 스틱의 이동량을 “앞으로 초당 5m로 이동하라“는 전진 속도(Velocity) 명령으로 가공하여 하위 위치 제어기(Position Controller)에 넘겨줍니다.
  • 조종사가 스틱을 놓으면, 자체 로직에 따라 이동 속도를 부드럽게 감속시켜(Braking) 기체를 지정된 좌표에 멈춰 세웁니다.

이 모든 스틱 수신, 타겟 변환, 감속 로직이 오직 ModeLoiter.cpp라는 단일 파일 안에 똘똘 뭉쳐 있습니다. 다른 모드들(ModeStabilize, ModeAuto 등)은 Loiter 모드가 속에서 무슨 짓을 하는지 알지도 못하며 간섭할 수도 없습니다.

0.3 구조적 트레이드오프: 응집도(Cohesion) 상승과 중복(Duplication)의 딜레마

이러한 철저한 캡슐화 전략은 개발자에게 막대한 이점을 줍니다.
로이터 모드(Loiter)에 브레이킹(Braking, 제동) 기능 향상을 위한 새로운 수학 공식을 추가하고 싶을 때, 개발자는 그저 ModeLoiter.cpp 파일만 열어서 수정하고 빌드하면 됩니다. 다른 모듈에 부작용(Side-effect)을 일으킬 확률이 0에 수렴하기 때문에, 고도의 **응집도(High Cohesion)**를 자랑합니다.

그러나 빛이 있으면 그림자가 있듯, 캡슐화는 **코드의 중복(Code Duplication)**이라는 딜레마를 낳습니다.
Loiter, Auto, Guided, RTL 모드 모두가 공통적으로 GPS를 필요로 합니다. 따라서 이 4개의 개별 클래스 init() 함수 내부에는 EKF 상태를 점검하는 거의 비슷비슷한 코드 조각들이 중복해서 작성되어 있습니다. 만약 GPS 수용 기준(Tolerance)에 대한 정책이 펌웨어 레벨에서 변경된다면, 개발자는 이 흩어져 있는 모드 클래스들을 일일이 순회하며 코드를 통일성 있게 수정해 주어야 하는 관리의 맹점이 발생합니다.
