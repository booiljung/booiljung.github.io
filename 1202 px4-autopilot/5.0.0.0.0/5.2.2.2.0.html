5.2.2.2. Ardupilot: AP_Message 및 싱글톤(Singleton) 객체 기반 동기식 데이터 공유

PX4의 uORB 비동기 메시징 시스템은 모듈 간 결합도를 끊기 위한 ’분산형 미들웨어’의 극치입니다. 그러나 비행 제어 소프트웨어 아키텍처의 쌍벽을 이루는 Ardupilot은 전혀 다른 철학적, 기술적 경로를 걷습니다. Ardupilot 개발자들은 데이터를 주고받기 위해 굳이 한 단계(Broker)를 더 거쳐야 하는 중간 다리(Middleware) 자체를 불필요한 퍼포먼스 제약이자 오버헤드로 간주합니다.

단일 루프(Main Loop) 기반으로 동작하는 Ardupilot 환경에서, 각종 센서 제어기 모듈 간의 통신은 싱글톤(Singleton) 객체 디자인 패턴과 중앙 집중형 로거 연산을 결합한 형태로, 매우 동기적이고 직접적인 방식으로 이루어집니다. 이를 통칭하여 Ardupilot의 내부 메시징 체계(Internal Messaging System)로 부르며, 최근의 고도화된 스레드 환경에서는 AP_Message 등의 확장형 라이브러리로 보완되고 있습니다.

0.1 퍼블리시-서브스크라이브 패러다임의 거부와 동기적 아키텍처

Ardupilot 아키텍처 내에서 특정 기능을 담당하는 핵심 클래스 (예: AP_InertialSensor, AP_GPS, AP_AHRS 등)들은 보통 전역적(Global)으로 하나만 동적 할당되는 싱글톤 객체 형태를 유지합니다.

// Ardupilot의 극단적인 동기적 데이터 교환 예시 (개념적 C++ 코드)
void Copter::fast_loop()
{
    // 1. 센서 값 읽기 (직접 메서드 호출)
    ins.update(); 
    
    // 2. 가속도/자이로 값 빼오기 (포인터나 전역 참조 객체에서 복사 없이 바로 Read)
    Vector3f accel = ins.get_accel();
    
    // 3. EKF 필터에 값 때려넣고 위치 추정 (직접 메서드 호출)
    ahrs.update();
    
    // 4. 모터 출력 (직접 제어기 호출)
    motors->set_roll_pitch_yaw_thrust(roll_out, pitch_out, yaw_out, thrust_out);
}

위의 의사 코드에서 보이듯, Ardupilot은 PX4처럼 “데이터가 도착했으니 읽어가라“고 브로드캐스트 하는 uORB 큐가 존재하지 않습니다. 메인 컨트롤 루프(Copter, Plane)가 전체 지휘자 역할을 맡아, 센서를 직접 깨워 값을 최신화(update()) 시킨 직후, 그 결과물을 다음 연산 클래스의 파라미터로 곧바로 쑤셔 넣습니다. 데이터의 생산과 소비가 동일한 스레드의 클럭(Tick) 내에서 순차적이고 동기적(Synchronous)으로 발생하기 때문에, 아키텍처 간의 시간 지연(Latency)이 사실상 ’0’에 수렴합니다.

0.2 싱글톤 객체와 데이터 무결성

센서의 값은 어떻게 저장되고 다른 모듈에 전달될까요? Ardupilot의 라이브러리(AP_Math, AP_InertialNav 등)는 데이터를 보관하기 위한 중앙 집중형 저장소로 기능합니다.

ins(AP_InertialSensor) 객체가 업데이트되면 그 내부의 물리 변수에 자이로 값이 업데이트됩니다. 이후 텔레메트리(MAVLink) 모듈이나 데이터 로깅 모듈이 이 값을 원할 때 uORB를 들여다볼 필요가 없습니다. 그냥 외부에서 AP::ins().get_gyro() 함수를 호출하여 메모리 번지에 박혀있는 값을 직접 복사(Direct Copy) 하거나 참조(Reference) 해 버리면 끝납니다.
중간 매개체(Broker)용 메모리 할당도 없고, 브랜치 프로젝션 지연도 생략됩니다.

0.3 AP_Message: 멀티 스레드 시대의 비동기적 메시지 보완재

최근 드론 하드웨어가 멀티코어 H7 프로세서 등으로 넘어가고, Ardupilot 역시 치비OS(ChibiOS) 기반으로 I/O 인터럽트 등 일부 백그라운드 스레드를 독립시키기 시작하면서, 완전히 전역 객체로 동기화하는 방식에 병목이 생겼습니다.

이를 보완하기 위해 나온 것이 제한적 비동기 큐잉 시스템인 AP_Message 유틸리티입니다. 데이터 로깅(Data Logging)이나 시리얼 텍스트 데이터 같이 초실시간성이 요구되지 않는 대용량 텍스트/이벤트 로그들은 메인 루프를 방해하지 않도록, 이 메시지 프로토콜을 통하여 백그라운드 워커 스레드(Worker Thread) 쪽에 쌓아둔 뒤 나중에 한꺼번에 빼가게 함으로써, 기존 동기식 아키텍처의 한계를 영리하게 극복하고 있습니다.

결론적으로 Ardupilot의 메시징 모델은 **“개발자 입장에서 극도로 직관적이며 속도의 손실이 없는 실리주의”**를 따릅니다. 이는 PX4의 철저한 규격 기반 미들웨어 통신과 대비되어, 양대 오픈소스 프로젝트의 취향 차이를 보여주는 가장 뚜렷한 식별자가 됩니다.
