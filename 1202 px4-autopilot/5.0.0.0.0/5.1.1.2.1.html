5.1.1.2.1. 소스 코드 공개 의무가 가져온 방대한 파생 코드 생태계

오픈소스 기술 생태계에서 GPLv3 라이선스는 가장 강력하면서도 파괴적인 확산력을 지닌다. Ardupilot(아두파일럿) 비행 제어 소프트웨어는 이 라이선스를 아키텍처의 근본 정체성으로 삼음으로써, 비록 상업적 독립성은 양보했으나 대신 “지식의 강제 환원“을 매개체로 세계 최고 수준의 기하급수적 파생 코드 생태계를 구축했다.

0.1 파생 코드(Derivative Works)의 정의와 리버스 엔지니어링 억제

GPLv3의 핵심은 “Ardupilot의 코드를 일부분이라도 정적으로 링킹(Static Linking)하거나 복사하여 수정한 펌웨어를 탑재한 하드웨어를 판매/배포할 경우, 그 수정된 전체 소스 코드를 반드시 동일한 GPLv3 라이선스로 원 사용자들에게 제공해야 한다“는 의무 조항이다.

만약 한 중국의 소규모 제조사가 Ardupilot의 C++ 메인 루프에서 자세 제어 루프의 P-I-D 게인 스케줄링 로직 일부를 자사의 독자적 배터리 관리 시스템(BMS) 코드와 융합한 후 그 바이너리를 컴파일하여 농업용 드론을 출시했다고 가정하자. 이 제조사는 자사의 BMS 코드를 공개하고 싶지 않더라도, GPLv3 조항에 입각하여 오토파일럿 전체 코드를 GitHub 등에 퍼블릭 형태로 즉시 릴리즈해야만 한다. 이를 회피하여 소스코드를 숨긴 채 바이너리만 탑재할 경우, 자유 소프트웨어 재단(FSF)이나 기존 핵심 컨트리뷰터들로부터 즉각적인 지식재산권 소송을 당하게 된다.

이러한 숨막히는 강제성은 소프트웨어의 리버스 엔지니어링이나 비밀주의를 억제하는 강력한 무기로 작동했다.

0.2 Upstream 병합(Merge)을 통한 집단 지성의 완성

이러한 강제적 공개 의무는 기업이나 개인 개발자들로 하여금 **“어차피 코드를 숨기지 못할 바에야, 차라리 우리가 직접 코어나 파생 기능을 만들어 버려 본가(Upstream) 생태계에 편입되는 것이 유지보수 측면에서 압도적으로 유리하다”**라는 전략적 선회를 이끌어냈다.

어느 회사가 독자적 분기점(Fork)을 만들었다 해도, 이를 유지하기 위해 Ardupilot 본가 저장소(Master Branch)의 수백 명 단위의 일일 커밋(Commit)과 리베이스(Rebase) 충돌을 평생 감당할 수는 없다. 따라서 가장 이기적인 주체들조차도 자사의 파생 버전을 Ardupilot의 기본 분기점(Main Thread)에 기여(Pull Request)하게 된다.

결과적으로, 지구상에 존재하는 수많은 기이형 기체(Tilt-rotor, Tailsitter, Flapping-wing 등)들에 대한 커스텀 비행 역학과 수십 종의 LiDAR/Radar 센서 통신 드라이버 로직이 자연스럽게 모두 모여 C++의 방대한 라이브러리(Library) 풀을 형성하게 된 것이다.

0.3 Ardupilot의 라이브러리 기반(AP_*) 아키텍처 성립 배경

Ardupilot 소스 코드에 진입하는 개발자가 가장 먼저 목격하는 것은, 루트 디렉토리에 끝없이 펼쳐진 수백 개의 AP_ 계열 라이브러리 폴더(AP_NavEKF, AP_Motors, AP_Mount, AP_HAL 등)들이다.

PX4가 POSIX 기반의 독립 모듈(Module)과 uORB라는 메시지 큐 통신으로 매우 우아하게 격리되어 결합된 체계를 갖춘 것과 비교하면, Ardupilot은 방대한 C++ 클래스들이 서로 얽혀서 직접 멤버 변수나 포인터를 주고받는 모놀리식 지향의 뚱뚱한(Fat) 라이브러리 구조를 취한다.
