5.3.1.2.1. 서로 다른 센서 조합을 사용하는 독립적인 EKF 코어 동시 구동

Ardupilot의 EKF3 멀티 레인(Multi-lane) 아키텍처가 빛을 발하는 가장 핵심적인 원리는 단순히 똑같은 계산을 여러 번 반복하는 것이 아니라, 서로 다른 물리적 센서 조합을 각기 다른 EKF 코어(가상 두뇌)에 배정하여 완벽한 ’구조적 다양성(Structural Diversity)’을 확보한다는 데 있습니다.

단일 EKF 인스턴스 기반인 PX4 초기 펌웨어 아키텍처와 매우 대비되는 이 특징은, Ardupilot 파라미터 내 센서 친화도(Sensor Affinity) 매핑 기법을 통해서 구현됩니다. 이 절에서는 서로 다른 센서들이 어떻게 이종 교배되어 다중 EKF 코어에서 독립적으로 연산되는지 분석합니다.

0.1 센서 친화도 할당(Affinity Masking)의 아키텍처

Ardupilot의 최신 코어 라이브러리(AP_NavEKF3) 내부의 생성자를 살펴보면, 각 EKF3 코어 객체가 인스턴스화될 때 자신이 어떠한 번호의 하드웨어 센서를 신뢰할 것인지 결정하는 ’마스크(Mask)’를 상속받게 됩니다.

조종자는 기체 세팅 시 EK3_AFFINITY 혹은 개별 마스크 파라미터(EK3_IMU_MASK)를 설정하여 이 가상 두뇌들의 연결선을 물리적으로 하드와이어링(Hard-wiring) 할 수 있습니다.
가령, 비행기에 2개의 가속도/자이로스코프(IMU), 2개의 GPS, 2개의 기압계가 장착되어 있다고 가정해 보겠습니다.

[ Ardupilot EKF3 코어별 센서 배정(Affinity) 예시 ]

* EKF3 Core 0 (Lane 1): IMU_1 + GPS_1 + Baro_1
* EKF3 Core 1 (Lane 2): IMU_2 + GPS_2 + Baro_2
* EKF3 Core 2 (Lane 3): IMU_1 + GPS_2 + Baro_1 (교차 배정)
  1. Core 0은 자신의 계산을 오직 1번 하드웨어 세트에만 의존합니다.
  2. Core 1은 철저히 2번 하드웨어 세트만 바라보고 독자적인 24차원 상태 벡터 행렬 공분산을 키워나갑니다.
  3. 두 객체 간의 수학적 데이터 침범이나 상호 참조는 소스코드 상 철저히 은닉(Encapsulated)되어 있어 일어날 수 없습니다. (독립성 보장)

0.2 비대칭적 생존성(Asymmetric Redundancy)의 마법

“서로 다른 센서 조합“이라는 아키텍처가 뿜어내는 생존력의 진가는, 하드웨어가 전혀 예상치 못한 물리적, 외부적 결함에 직면했을 때 나타납니다.

상황 1: 조립 불량으로 인한 기구적 진동 이상 (Mechanical Loose Mount)
만약 드론 제조업체가 모듈을 조립하다가 2번 IMU가 결합된 보드의 나사를 덜 조였다고 가정해 봅시다. 모터가 고속 회전하자 2번 IMU가 미친 듯이 공진(Resonance)하여 가속도계 값이 쓰레기처럼 발산합니다.
이때 Ardupilot의 EKF3 Core 1 (Lane 2)의 오차 공분산 행렬은 터져버립니다. 하지만 1번 IMU를 참조하는 EKF3 Core 0과 Core 2는 아무런 오염 없이 깨끗한 자세 계산을 유지합니다. 메인 제어기는 건강한 Core 0의 데이터로 비행을 퍼펙트하게 지속합니다. (단일 EKF였다면 2개 데이터가 내부에서 섞여 추락했을 것입니다.)

상황 2: GPS 재밍 및 스푸핑 (GPS Jamming/Spoofing)
더욱 진보된 형태의 센서 조합도 가능합니다. 최근의 Ardupilot은 아예 완전히 이질적인 센서를 교차로 집어넣습니다.
EKF3 Core 0은 메인 GNSS(GPS)를 결합합니다. 반면, EKF3 Core 1은 GPS 대신 하단에 부착된 광학 흐름 센서(Optical Flow)와 레이저 고도계의 조합으로만 위치 공분산을 업데이트하도록 세팅해 둡니다.
만약 적진에서 강력한 전파 GPS 재밍(Jamming) 무기를 발사하여 GPS가 조작당하면, Core 0은 기체가 100km 밖으로 날려갔다고 착각(Fly-away)하고 미친 듯이 제어기를 비틀어댑니다. 하지만 광학 흐름 센서(바닥의 타일을 보고 이동을 자각함)만 바라보던 Core 1은 “GPS 따위는 모르겠고 내 바닥은 여전히 제자리다“라고 올바른 정답을 부르짖게 됩니다.

0.3 코드 계층의 고립 구조

소프트웨어 아키텍처적으로 이러한 EKF 인스턴스들은 AP_NavEKF3_core 라는 C++ 클래스로 구현됩니다.
Ardupilot 프레임워크는 부트 단계에서 메모리가 허용하는 한 이 코어 클래스 배열을 new 키워드로 복수 생성하며, 각 코어의 update() 메서드는 메인 루프에서 순차적으로 불리게 되지만 내부 멤버 변수(_state, _P)는 완전히 고립된 메모리 번지를 향유합니다.

이는 PX4-Autopilot이 “완전무결하게 보상된 단 1개의 거대 두뇌를 정교하게 유지하자“는 스위스 시계 톱니바퀴 같은 철학이라면, Ardupilot EKF3는 **“두뇌를 3개 만들어서 환경 파괴에 맞서 다수결로 살아남아라”**는 좀 더 거칠고 야생적이며 실전적인 철학을 관철하고 있음을 극명하게 시사합니다.
