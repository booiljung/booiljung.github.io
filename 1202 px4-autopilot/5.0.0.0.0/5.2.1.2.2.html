5.2.1.2.2. 스레드 오버헤드 최소화를 위한 루프(Loop) 타이밍 제어

Ardupilot(아두파일럿)이 단일 주 스레드(Main Thread) 기반의 모놀리식(Monolithic) 구성을 유지하면서도, 수많은 비행 알고리즘을 문제없이 실행할 수 있는 핵심 비결은 바로 **루프 타이밍 제어(Loop Timing Control)**에 있습니다. 운영체제(OS)의 커널 기반 스케줄링에 전적으로 의존하는 대신, 코드 내부에서 시간 자원을 마이크로초(us) 단위로 철저히 쪼개서 관리하는 이 방식은 “스레드 컨텍스트 스위칭(Context Switching)에 낭비되는 시간을 제로(Zero)에 가깝게 만들겠다“는 Ardupilot 개발자들의 강력한 의지를 보여줍니다.

이 섹션에서는 Ardupilot 코드가 메인 스케줄러를 통해 오버헤드를 어떻게 회피하고 실시간성을 담보하는지 상세히 분석해 보겠습니다.

0.1 스레드 오버헤드(Thread Overhead)란 무엇이며, 왜 치명적인가?

RTOS 환경에서 50개의 스레드를 띄웠다고 가정해 봅시다. 1번 스레드가 작업을 하다가 커널 틱(Tick)에 의해 2번 스레드로 CPU 제어권이 넘어가려면, CPU는 1번 스레드가 쓰던 수많은 레지스터(Register) 값들을 메모리(Stack)에 백업하고, 2번 스레드의 백업본을 다시 레지스터로 불러와야 합니다. 이를 ’컨텍스트 스위칭(Context Switching)’이라고 부릅니다.
고급 스마트폰 AP에서는 이 작업이 순식간이지만, 클럭이 168MHz~400MHz 수준에 불과한 드론용 마이크로컨트롤러(MCU)에서는 이것이 잦아질수록 기체가 자세를 제어할 수 있는 소중한 연산(Math) 사이클을 갉아먹는 치명적인 오버헤드가 됩니다. Ardupilot은 이 오버헤드를 아키텍처 레벨부터 거부한 것입니다.

0.2 고정 주파수(Fixed Frequency) 메인 루프의 동작 원리

Ardupilot 기체별 메인 코드(ArduCopter/Copter.cpp 등)는 항상 메인 루프가 동작하는 고유의 주파성(Rate)을 가지고 있습니다. 멀티로터인 콥터(Copter)의 경우 기본적으로 400Hz로 동작합니다. 즉, 1초에 400번, 약 2500 마이크로초(2.5ms)마다 전체 시스템의 가장 중요한 루프가 한 바퀴씩 회전한다는 의미입니다.

// Ardupilot 스케줄러 테이블의 개념적 예시 (Copter.cpp)
const AP_Scheduler::Task Copter::scheduler_tasks[] = {
    // 함수 이름,            실행 주기(Hz), 최대 허용 소요 시간(us)
    { fast_loop,            400,          900 },  // 매 프레임 실행, 최대 0.9ms 소모
    { update_navigation,     50,          500 },  // 초당 50번 실행, 최대 0.5ms 소모
    { read_battery,          10,          100 },  // 초당 10번 실행, 최대 0.1ms 소모
    { update_notify,         50,          100 },  // 초당 50번 실행, LED 등 조작
    { send_heartbeat,         1,          100 },  // 초당 1번 실행
};
  1. Fast Loop의 절대 보장: fast_loop()는 IMU 자이로/가속도를 읽고(Read) 비행 자세를 보정하여 모터 값을 출력하는 가장 핵심적인 함수입니다. 스케줄러는 2.5ms의 루프가 시작될 때마다 최우선적으로 이 fast_loop를 호출합니다.
  2. 슬립 타임(Slip Time) 측정: fast_loop가 끝난 뒤, 스케줄러는 현재 타이머를 읽어 2.5ms 전체 루프 시간 중 남은 여유 시간이 얼마나 되는지(Time Available)를 정확히 산출합니다.
  3. 여유 시간에 따른 퍼즐 맞추기: 만약 1.5ms의 여유가 남았다면, 위 테이블에서 실행기일이 도래한 하위 태스크(예: 배터리 읽기, GCS로 데이터 쏘기)에 명시된 ’최대 허용 소요 시간’들을 저울질합니다. 남은 시간 안에 넣을 수 있는 태스크들을 차례대로 호출(Call) 한 뒤 메인 루프를 종료하고 다음 400Hz 틱을 기다립니다.

0.3 타이밍 튜닝: 스파게티 코드 방어선의 한계와 진화

Ardupilot의 이 방식은 콜백 함수들이 하나의 스레드 문맥 안에서 동기화되어 순차 통과하므로 컨텍스트 스위칭 시간이 이론상 0에 수렴한다는 엄청난 장점이 있습니다. 그러나 이는 개발자들에게 가혹한 규칙을 강제합니다,

  • 블로킹(Blocking) 절대 엄금: 만약 누군가 추가한 update_notify 함수가 외부 LED 칩의 응답을 기다리느라 1ms 이상을 대기(Sleep/Block) 해 버리면, 남은 메인 루프의 타이밍이 모조리 밀려버리게 됩니다(Slip). 이를 방지하기 위해 Ardupilot의 모든 함수 인터페이스는 철저히 논블로킹(Non-blocking) 상태 머신으로 짜여 있어야만 합니다.
  • 루프 페이스트(Loop Rate) 튜닝의 예술: 만약 연산량이 너무 많아 메인 사이클이 2.5ms를 지속적으로 초과한다면 어떻게 될까요? 기체는 자신의 위치를 놓치고 제어계가 불안정해집니다. 이를 Loop Time Exceeded(루프 타임 초과) 현상이라고 하며, Ardupilot 사용자가 로그(Log) 분석 시 PM(Performance Monitor) 메시지를 통해 가장 눈여겨봐야 하는 아키텍처의 한계 지표입니다.

결론적으로, Ardupilot의 루프 타이밍 제어 구조는 “OS를 무시하고 칩셋의 물리적 시간을 직접 재단(Tailoring)하여 성능을 쥐어짜 내는 극한의 미니멀리즘“이라 요약할 수 있습니다.
