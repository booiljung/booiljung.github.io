5.7.1.1.1. uXRCE-DDS 브릿지를 통한 uORB \rightarrow ROS2 Topic 다이렉트 매핑

PX4 생태계가 컴패니언 컴퓨터와 소통하는 방식은 최근 몇 년 새 **Micro XRCE-DDS (uXRCE-DDS)**의 도입으로 혁명적인 패러다임 전환을 맞이했습니다. 이 혁신의 핵심은 오랫동안 병목이었던 ’MAVLink 번역 계층’을 완전히 걷어내고, 펌웨어 내부 데이터 버스와 ROS 2를 직결(Direct Mapping)했다는 점입니다.

0.1 레거시(Legacy) 통신의 이중 번역 구조 한계

과거 MAVROS 시절, 기체의 현재 상태(예: 센서 융합이 끝난 현재 위치)가 컴패니언 컴퓨터로 전달되는 과정은 험난했습니다.

  1. PX4 내부에서 VehicleLocalPosition 이라는 uORB 메시지가 생성됩니다.
  2. 이를 MAVLink 모듈이 가로채서 LOCAL_POSITION_NED라는 MAVLink 바이트 배열로 직렬화(Serialize)합니다.
  3. 시리얼 케이블을 타고 컴패니언 컴퓨터로 넘어갑니다.
  4. 컴패니언 컴퓨터의 MAVROS 노드가 이를 다시 해독(Deserialize)하여 geometry_msgs/PoseStamped라는 ROS 2 토픽으로 발행(Publish)합니다.

이 과정은 데이터 구조를 두 번이나 구부려 맞추는 오버헤드가 발생했고, 이는 100Hz 이상의 초고주파 비전 제어(Visual Control) 루프에서는 치명적인 지연 시간(Latency) 타격으로 다가왔습니다.

0.2 다이렉트 매핑(Direct Mapping) 구조의 탄생

PX4 개발진은 펌웨어 컴파일 단계에서 uORB 메시지 정의 파일(.msg)을 읽어들여, ROS 2가 이해할 수 있는 IDL(Interface Definition Language) 규격이나 C++ 헤더 파일로 똑같이 찍어내는(Code Generation) 자동화 툴을 만들어냈습니다.

이 빌드 시스템 혁신 덕분에 현재 PX4의 오토파일럿 보드 내부에는 **uXRCE-DDS 클라이언트(Client)**가 기본 탑재되어 구동됩니다.
이제 PX4 내부의 메모리에 존재하는 VehicleOdometry C++ 구조체는 uORB 버스에 실리는 순간, 어떠한 논리적 구조 변경이나 MAVLink로의 번역 과정 없이 DDS 페이로드에 고스란히 담겨 시리얼이나 이더넷(Ethernet) 포트를 타고 발사됩니다.

0.3 컴패니언 컴퓨터의 노드화(Node-ification)

컴패니언 컴퓨터 측에는 Micro XRCE-DDS 에이전트(Agent) 하나만 백그라운드에서 돌고 있으면 됩니다.

  • 로스(ROS 2) 개발자 입장에서는 더 이상 골치 아픈 MAVROS 문서나 MAVLink 메시지 ID 체계를 공부할 필요가 없어졌습니다.
  • 단순히 px4_msgs/msg/VehicleOdometry 라는 토픽을 Subscribe(구독)하기만 하면, 마치 한 보드 내에 있는 프로세스처럼 투명하고 생생하게 오토파일럿의 내부 uORB 데이터를 직관적으로 뽑아 쓸 수 있게 되었습니다.

0.4 고속 모션 제어의 새로운 지평

이러한 uORB \rightarrow ROS 2 다이렉트 매핑 아키텍처는 대역폭 낭비와 지연 시간을 싱글 디지트 밀리초(ms) 단위로 끌어내렸습니다. 특히 하드웨어적으로 이더넷(Ethernet) 포트를 지원하는 최신 픽스호크(Pixhawk 5X/6X 시리즈)와 맞물렸을 때, 초당 수백 번씩 오차 없이 오프보드 셋포인트(Setpoint)를 때려부어 대형 드론을 아크로바틱하게 회피 기동시키는 진정한 자율 로봇(Autonomous Robot)의 시대를 열어젖혔습니다.
