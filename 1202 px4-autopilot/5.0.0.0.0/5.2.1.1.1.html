5.2.1.1.1. 독립된 프로세스 형태의 모듈 실행 및 메모리 관리

PX4-Autopilot 아키텍처를 유닉스(UNIX) 데스크톱 환경과 가장 유사하게 만들어 주는 특성은 펌웨어 안에서 각 기능들이 단단하게 결합된 라이브러리 형태가 아니라, 각각 독자적인 진입점(Entry Point)과 스택(Stack)을 가진 독립된 프로세스 형태의 애플리케이션으로 동작한다는 점이다. 이는 시스템에 강력한 오류 격리성(Fault Isolation)과 리소스 통제력을 부여한다.

0.1 Main 함수의 파편화: 모듈별 독립 진입점

전통적인 임베디드 프로그래밍은 단 하나의 main() 함수에서 모든 컴포넌트들을 순차적으로 초기화하고 슈퍼루프 안으로 진입한다. 하지만 C++ 기반의 PX4 시스템 소스코드를 살펴보면 단일한 메인 함수를 찾아보기 어렵다.

src/modules/ 아래에 존재하는 commander, mc_pos_control, ekf2 등의 디렉토리 내부에는 컴포넌트별로 int commander_main(int argc, char *argv[]) 혹은 int ekf2_main(...)과 같은 고유의 C++ 메인 함수 인터페이스가 구현되어 있다. PX4 스타트업 스크립트(rcS)는 부팅 시 리눅스 커맨드라인에서 셸(Shell) 스크립트를 실행하듯 이 개별 모듈들을 commander start, ekf2 start 명령어 형태로 독립적으로 파생(Spawn)시킨다.

이렇게 띄워진 모듈은 다른 모듈의 프로세스 동작 여부와 무관하게, 자신만의 while(1) 스레드 루프 안에서 주어진 uORB 메시지 업데이트를 기다렸다가 연산 후 다시 슬립(Sleep)하는 철저한 독립 비동기 상태를 영위한다.

0.2 세밀하게 조율되는 스택 메모리(Stack Memory) 파티셔닝

모놀리식 환경에서 전역 힙(Heap)과 스택(Stack)을 뭉뚱그려 사용하는 것과 달리, PX4는 태스크(Process/Thread)를 생성(task_spawn)할 때 각 모듈이 사용할 극소량의 스택 메모리 한도를 하드코딩된 바이트 단위로 정확히 할당한다.

// 예시: px4_task_spawn 호출을 통한 모듈 생성 메커니즘
int task_id = px4_task_spawn_cmd(
    "mc_pos_control",          // 태스크 이름
    SCHED_DEFAULT,             // 스케줄링 정책
    SCHED_PRIORITY_POSITION_CONTROL, // 우선순위 매핑 (예: 250)
    1920,                      // 독립적으로 할당된 스택 메모리 사이즈 (Bytes)
    (px4_main_t)&mc_pos_control_main, 
    (char *const *)argv);

이러한 독립 스택 할당 체계(Stack Allocation)는 다음과 같은 중요한 아키텍처적 이점을 제공한다.

  • 메모리 최적화: 단순히 변수를 배달하는 uORB 라우터나 로거 태스크에는 아주 작은 스택(예: 1KB)만 주고, 백터 및 행렬 연산을 수행하는 무거운 EKF 추정기 모듈에는 큰 스택(예: 8KB)을 부여하는 형식을 취해 제한된 MCU의 SRAM 공간(보통 256KB~1MB 수준)을 한계점까지 착취(Optimization)할 수 있다.
  • 스택 오버플로우(Stack Overflow) 방어: 모듈 A가 자신의 스택 이상을 침범했을 때 커널 레벨에서 해당 태스크에 한정된 패닉(Panic)을 터뜨림으로써, 전체 시스템의 메모리가 조용히 오염되어 기체가 추락하는 좀비 현상을 방지하고 결함 위치를 명확히 디버깅할 수 있다.

0.3 공유 메모리(Shared Memory) 대신 uORB를 통한 객체 독립

프로세스가 분리되어 있다는 것은 곧, 모듈 간 메모리 포인터(Pointer)의 직접 참조를 금지한다는 것을 의미한다.
예를 들어 mc_att_control(자세 제어기) 모듈이 현재 드론의 피치(Pitch) 속도 에러 값을 구하기 위해 sensors 모듈이 가진 자이로(Gyro) 데이터 배열 메모리에 직접 포인터로 접근하는 식의 스파게티 얽힘은 PX4 아키텍처에서 엄격히 차단된다.

대신, 센서 모듈은 vehicle_angular_velocity라는 C 구조체(struct)를 uORB 스택 영역에 복사하여 퍼블리시(Publish)하고, 컨트롤러는 이를 자신의 메모리 영역으로 카피(Copy)해 서브스크라이브(Subscribe)한다. 비록 이 과정에서 소폭의 메모리 복사 오버헤드가 발생하지만, 그 대가로 한 모듈의 변수에 쓰기(Write) 연산 도중 다른 모듈이 이를 읽어(Read) 데이터 레이스(Data Race)가 발생하여 비행 역학이 붕괴되는 치명적인 멀티스레드 충돌 위협을 아키텍처 단위에서 원천적으로 소거해 버렸다.
