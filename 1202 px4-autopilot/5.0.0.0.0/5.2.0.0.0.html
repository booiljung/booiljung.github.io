5.2. 운영체제(OS) 의존성 및 태스크 스케줄링 패러다임

마이크로컨트롤러(MCU)라는 비좁은 실리콘 공간 안에서, 드론의 센서 데이터를 읽고(Read), 필터링하며(Filter), 모터 제어 신호를 방출하는(Write) 행위는 단순히 코드를 순차적으로 실행하는 것으로는 불가능하다. “언제, 어떤 코드를, 얼마의 시간 동안 독립적으로 실행시킬 것인가“를 정의하는 운영체제(OS)와 태스크 스케줄링(Task Scheduling) 아키텍처는 비행 제어기 소프트웨어의 척추와도 같다.

PX4와 Ardupilot은 이 OS 의존성 및 스케줄링 패러다임에서 라이선스와 거버넌스 차이 못지않은 거대한 공학적 철학의 양극단(Polarization)을 보여준다.

0.1 운영체제(OS)를 바라보는 관점의 진화

비행 제어 소프트웨어 초기 역사(예: 8비트 Arduino 시절)에서는 OS라는 개념 자체가 사치였다. 거대한 하나의 main() 함수 루프(while(1)) 안에서 모든 센서와 제어기가 숨 가쁘게 순차적으로 호출되는 슈퍼루프(Super-loop) 동기식 아키텍처가 지배적이었다.

그러나 하드웨어가 32비트 ARM Cortex-M 계열로 넘어오며 컴퓨팅 자원이 폭발적으로 증가하자, PX4와 Ardupilot은 각기 다른 스케줄링 패러다임을 선택하며 갈라섰다.

  • PX4의 시각: “운영체제(RTOS)의 절대적인 신뢰와 의존”
    PX4는 NuttX라는 하드 리얼타임(Hard Real-time) POSIX 호환 운영체제를 근간으로 아키텍처를 새롭게 설계했다. 비행 제어 논리(Flight Control Logic)를 여러 개의 완전히 독립된 ‘스레드(Thread)’ 혹은 ’태스크(Task)’로 쪼개고, 언제 누구를 재우고 깨울지(Context Switching)에 대한 권한을 커널(Kernel)의 스케줄러에게 전적으로 위임(Delegation)한다.
  • Ardupilot의 시각: “운영체제 위에서 작동하는 자체 스케줄러(Custom Scheduler)”
    Ardupilot은 오랜 기간 쌓아온 모놀리식(Monolithic) 아키텍처를 한 번에 갈아엎는 대신, 기존의 슈퍼루프와 타이머 방식을 진화시켰다. ChibiOS 등의 RTOS를 기저에 깔긴 하지만, 그것은 그저 하드웨어를 추상화하기 위한 도구(AP_HAL)일 뿐이다. 실제 비핵심 비행 논리는 Ardupilot 자체의 ‘협력적 태스크 스케줄러(Cooperative Task Scheduler)’ 테이블 안에서 타임 슬라이스(Time Slice)를 배분받으며 돌아간다.

0.2 패러다임 차이가 파생시킨 아키텍처 성능 지표

스케줄링 패러다임의 차이는 코드 베이스의 구조뿐만 아니라, 엔지니어가 시스템 한계(Limit)를 디버깅하는 방식마저 갈라놓았다.

  • 태스크 선점(Preemption)과 응답성(Responsiveness)
    PX4(NuttX) 환경 하에서 rate_ctrl(각속도 제어)과 같은 우선순위(Priority)가 극도로 높은 태스크는, 하위 로깅 태스크가 CPU를 아무리 길게 점유하고 있더라도 커널 틱(Tick) 스위칭에 의해 언제든 즉각적으로 개입(Preemption)하여 모터를 제어할 수 있다. 하드 리얼타임성 확보에 압도적으로 유리하다.
    반면, Ardupilot은 태스크 스스로가 양보(Yield)하지 않으면 다른 태스크가 지연(Jitter)될 가능성이 잔존하는 협력적 스케줄링의 비중이 높기 때문에, 함수마다 실행 시간(Execution Time) 초과를 방지하는 철저한 시간 체크 로직(Timing Check Logic) 튜닝에 극도로 집착하게 된다.
  • 모듈화와 격리성(Isolation)
    POSIX 시스템을 지향한 PX4 아키텍처에서는, 새로 작성한 제어 알고리즘 모듈 내부에서 Segmentation Fault나 무한 루프 에러가 터지더라도 해당 프로세스만 사멸될 뿐 핵심 비행 스레드는 생존할 확률이 존재한다. 모듈 간 메모리 분리 사상을 띄고 있기 때문이다. 반대로 Ardupilot은 방대한 스래드들이 단일 실행 흐름 속에서 객체 참조를 통해 얽혀 있으므로, 작은 버그의 연쇄 폭발(Cascading Failure) 리스크를 코드 리뷰 단위에서 선제적으로 막아내야만 하는 운명을 지닌다.

이렇듯 운영체제를 어떻게 다루고 개별 연산의 시간을 어떻게 배분할 것인지에 대한 설계 아키텍처는, 당장 비행은 엇비슷하게 이루어질지라도 “드론을 어떤 방식으로 극한의 무중단 상태(Zero-downtime) 환경으로 몰아갈 것인가“라는 심오한 시스템 공학적 궤적의 출발점이 된다.
