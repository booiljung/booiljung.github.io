5.5. 비행 모드(Flight Mode) 설계와 상태 관리 시스템

조종기(RC, Remote Controller) 스틱을 앞으로 쭉 밀었을 때, 기체가 어떻게 반응해야 할까요?
어떤 비행 모드(Flight Mode)에서는 스틱을 민 만큼 기체가 앞으로 ‘기울어지고(Pitch Angle)’, 어떤 모드에서는 초당 5m 의 속도로 ‘앞으로 날아가며(Velocity)’, 또 어떤 모드에서는 스틱 조작을 아예 무시한 채 GCS가 지정한 ‘웨이포인트(Waypoint)를 향해’ 알아서 날아갑니다.

이처럼 하드웨어 센서와 제어 루프가 아무리 완벽하게 구성되어 있더라도, 기체가 현재 어떤 논리적 상태(Logical State)에 놓여 있으며 사용자의 입력을 어느 제어 루프의 계층(Layer)으로 연결해 줄 것인가를 결정하는 두뇌가 필요합니다. 이것이 바로 비행 모드 설계와 상태 관리 시스템(State Management System)의 핵심 역할입니다.

0.1 비행 모드의 3대 분류 스펙트럼

현대의 오토파일럿 시스템은 조종사(또는 외부 컴퓨터)의 개입 수준에 따라 크게 세 가지 스펙트럼으로 비행 모드를 분류합니다.

  1. 수동/보조 제어 모드 (Manual / Attitude Stabilized Mode)
  • 인간 조종사가 조종기의 스틱을 통해 직접 기체의 자세(기울기)와 스로틀(출력)을 조작합니다.
  • 조종사가 스틱을 중앙에 놓으면 기체가 수평을 유지(Stabilized)하지만, 바람이 불면 기체는 속절없이 밀려갑니다. (GPS나 비전 센서를 사용하지 않습니다.)
  • 주요 모드: Stabilized, Acro (자이로 센서만 사용), Altitude (기압계 센서 융합).
  1. 반자동/위치 제어 모드 (Assisted / Position Control Mode)
  • 조종사의 스틱 입력이 직접적인 ’기울기’가 아니라 ’목표 속도(Velocity)’로 변환됩니다.
  • 스틱을 중앙에 놓으면 오토파일럿이 GPS와 IMU 센서를 활용해 현재 3차원 위치에 기체를 문자 그대로 못 박듯 고정(Position Hold)시킵니다.
  • 주요 모드: Position, Hold, Offboard/Guided (외부 컴퓨터의 속도 제어).
  1. 전자동/임무 제어 모드 (Autonomous / Auto Mode)
  • 조종사의 스틱 입력을 기본적으로 무시합니다.
  • 기체 내부의 네비게이션 모듈이 사전에 업로드된 임무 궤적(Mission Item)을 따라가거나, 비상사태 시 자동으로 이륙 지점으로 복귀(Return to Launch)합니다.
  • 주요 모드: Mission (Auto), Return (RTL), Land.

0.2 아키텍처적 난제: 무결성 검증과 모드 전환 (Mode Transition)

비행 모드를 구현하는 것 자체는 어렵지 않으나, **‘모드 간의 전환(Transition)’**을 안전하게 관리하는 것은 비행 제어기 소프트웨어 설계에서 가장 까다로운 난제 중 하나입니다.

예를 들어, 조종사가 상공 100m에서 Stabilized 모드에서 Position 모드로 스위치를 변경했다고 가정해 보겠습니다.
만약 이때 기체의 GPS 수신율이 떨어져 있거나 컴파스(나침반) 센서에 심각한 자기장 교란이 발생한 상태라면 어떻게 될까요? 기체는 자신의 현재 위치를 모르기 때문에 Position 모드로 전환되는 즉시 위치 제어 루프가 미쳐 날뛰며(Divergence) 플라이어웨이(Fly-away, 기체 가출 현상)가 발생할 것입니다.

따라서 오토파일럿은 사용자가 모드 전환을 요구할 때마다 현재 센서와 시스템의 전반적인 건강 상태(Health Status)를 단 1밀리초 내에 평가하고, 조건이 충족되지 않으면 전환을 단호히 거부(Reject)하거나 안전한 하위 모드(Fallback mode)로 강제 이행시켜야만 합니다.

0.3 상태 관리(State Management) 철학의 분기

이 거대한 논리적 판단의 미로(Maze)를 소프트웨어적으로 어떻게 구현할 것인가를 두고, PX4와 Ardupilot 진영은 소프트웨어 아키텍처 관점에서 완전히 다른 길을 선택했습니다.

  • 동기적 중앙집권(Centralized) 설계: 어떤 진영은 비행 모드의 변경과 센서 헬스 체크, 트러블슈팅 판단을 거대한 하나의 ’커맨더(Commander) 타워’에 몰아넣어 전제 군주처럼 시스템 전반을 엄격하게 통제하게 만들었습니다.
  • 비동기적 분산(Decentralized) 설계: 다른 진영은 각각의 비행 모드를 독립적인 클래스(Class) 객체로 만들고, 각 모드가 자신의 생존 조건과 에러 처리를 스스로 판단하도록 권한을 쪼개어 분산시켰습니다.

이어지는 절에서는 이 두 펌웨어가 채택한 ’상태 머신(State Machine)’의 시각 차이를 비교하고, 이것이 실제 비행의 안정성과 커스텀 모듈 개발자들에게 어떤 파급 효과를 미치는지 심층적으로 분석합니다.
