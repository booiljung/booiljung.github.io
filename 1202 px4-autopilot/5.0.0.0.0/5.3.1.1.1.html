5.3.1.1.1. 센서 데이터별 타임스탬프 기반 링 버퍼(Ring Buffer) 정렬 알고리즘

앞선 절에서 PX4의 EKF2 모듈이 이종 센서들 간의 관측 시간 지연(Observation Time Delay)을 완벽하게 동기화하기 위해 일종의 ’타임머신’과 같은 과거 궤적 보상 메커니즘을 사용한다고 설명했습니다. 본 절에서는 이러한 지연 보상 아키텍처를 메모리단에서 물리적으로 강제하고 지탱해 주는 핵심 자료 구조 체계, 즉 타임스탬프 기반 링 버퍼(Ring Buffer) 정렬 알고리즘을 소프트웨어 공학적 관점에서 해부합니다.

0.1 시계열 불일치의 늪과 타임스탬프(Timestamp)의 중요성

무인기 시스템 안착(Integration)에 있어서 가장 무서운 적은 개별 센서들이 각자의 고유한 마이크로컨트롤러(MCU)와 내부 클럭(Clock)을 돌리며 데이터를 내뿜는다는 점입니다. GPS 모듈 안에도 시계가 있고, Lidar 안에도 시계가 있습니다.
만약 메인 비행 제어기(Pixhawk 등)가 메시지가 ’도착한 시간’을 측정 시점으로 오해한다면, 통신망의 큐잉 지연(Queueing latency)으로 인해 시스템의 동역학 방정식을 무너뜨리게 됩니다.

따라서 PX4 펌웨어는 가장 기저율 수준의 커널 타이머 도구인 HRT(High Resolution Timer) 메커니즘을 제공합니다.
hrt_absolute_time()이라는 시스템 콜을 호출하면, 시스템 부팅 이후 경과된 절대 시간의 마이크로초(micro-second, us) 단위 단위 타임스탬프를 획득할 수 있습니다. uORB를 통해 배달되는 모든 센서 토픽 구조체(.msg)에는 uint64_t timestamp 필드가 의무적으로 포함되어 있으며, PX4의 EKF2는 메시지 도착 시간이 아닌 이 태생적 발행 스탬프(Publish Stamp) 시간만을 신뢰하여 정렬을 시작합니다.

0.2 EKF2 관측 버퍼링을 위한 링 버퍼(Ring Buffer) 구조

지연 시간이 제각각인 데이터들을 ’과거의 올바른 시점’에 정확히 꽂아 넣으려면, 센서들의 로깅 배열(History Array)을 구축해야 합니다. 하지만 한정된 SRAM 공간 안에서 계속해서 날아오는 센서 데이터를 무한정 동적 할당(malloc/new)하여 연결 리스트(Linked List)로 이어 붙이는 행위는 메모리 단편화(Fragmentation)를 유발하여 비행 제어기를 수 초 만에 마비시킵니다.

이에 대한 우아한 해답이 바로 **고정 길이 링 버퍼(Fixed-size Circular/Ring Buffer)**입니다.

// PX4 EKF2 내부 링 버퍼의 추상적인 구현 개념 (src/modules/ekf2/EKF/ring_buffer.h 참조)
template<typename T>
class RingBuffer {
private:
    T* _buffer;           // 고정 크기로 컴파일 시 할당된 연속 메모리 배열
    uint8_t _head;        // 가장 최근에 써진 데이터의 스롯 인덱스
    uint8_t _tail;        // 가장 오래된 데이터의 슬롯 인덱스
    uint8_t _size;        // 버퍼의 총 길이 (지연 보상 허용치에 비례)

public:
    void push(const T& sample) {
        _head++;
        if (_head >= _size) { _head = 0; } // 배열 끝에 닿으면 맨 앞으로 (Circular)
        _buffer[_head] = sample;
        if (_head == _tail) { _tail++; }   // 버퍼가 꽉 차면 꼬리(가장 오래된 데이터)를 덮어씀
    }
};
  1. 동적 할당 제로(Zero-malloc): EKF가 부팅될 때 버퍼의 크기(예: 최대 지연 250ms 분량을 담을 자이로 배열 100칸)가 한 번 확정되면, 비행 중 절대 메모리 주기가 변하지 않습니다.
  2. 연산량 O(1) 상수 시간: FIFO(First In First Out) 형태로 데이터를 밀어 넣고 뺄 때, 인덱스의 증감 연산만 수행되므로 CPU 사이클 소모가 사실상 없습니다. 배열의 복사(Shift)나 이동 연산이 전혀 발생하지 않습니다.

0.3 지연 데이터의 삽입과 시계열 정렬(Chronological Sorting) 알고리즘

링 버퍼에 데이터가 예쁘게 시간순 정렬되어 밀려들어 온다면 아무 문제가 없겠지만, 각 센서들은 지연 시간에 따라 EKF 모듈에 비동기적으로(순서가 마구 뒤섞여서) 도착하게 됩니다.

[t=10] IMU, [t=11] 기압계, [t=12] IMU, [t=13] IMU, …
이렇게 메인 제어 루틴이 돌아가며 IMU 버퍼와 상태 예측(State Prediction) 버퍼를 t=13 까지 써놓았습니다.
그런데 갑자기 위성에서 통신을 거쳐 늦게 도착한 GPS 데이터 한 덩어리가 도착했는데 까보니 타임스탬프가 t=7 이었습니다.

이때 링 버퍼 위에서 작동하는 EKF2의 정렬 보상 알고리즘이 가동됩니다.

  1. 검색(Search): 상태 링 버퍼의 Head(t=13)부터 역방향으로 탐색 배열을 거슬러 올라갑니다. t=7에 해당하거나 근접한 타임스탬프의 인덱스를 찾아냅니다. (바이너리 서치 또는 선형 탐색 적용)
  2. 과거 업데이트(Time-jump Update): 필터 방정식의 위성 포인터를 t=7 인덱스로 롤백시킵니다. 그 과거 시점에서의 ’예측 좌표’와 ’GPS t=7 계측 좌표’를 융합하여(Fusion) t=7 시점의 에러를 보정합니다.
  3. 예측의 재전파(Re-propagation): 과거의 값이 교정되었으므로, t=7 부터 t=13 까지 링 버퍼에 보존되어 있던 IMU 관성력 데이터를 활용해 정방향으로 초고속 재계산을 수행하여 현재 t=13 시점의 최종 융합 좌표를 다시 만들어냅니다.

이 알고리즘 덕분에 EKF는 “느리게 도착하는 신호“를 버리지도 않고, “가장 최근의 IMU 주파수“를 포기하지도 않으면서 두 체계의 신뢰성을 완벽하게 확률론적으로 합산할 수 있게 되었습니다. 이는 멀티 스레드의 맹목적인 경쟁을 추구하기보다는 데이터가 가진 시간의 고유성(Timestamp)을 하나의 지평(Horizon) 위에서 순차 지연(Delayed) 복원시키는 우아한 단일 코어의 소프트웨어 패턴이라 평가받습니다.
