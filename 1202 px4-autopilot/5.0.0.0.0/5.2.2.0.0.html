5.2.2. 내부 데이터 통신 및 미들웨어(Middleware) 아키텍처

로봇 공학과 무인항공기(UAS) 소프트웨어에서 모형 헬리콥터 시대를 벗어나 ’자율 시스템(Autonomous System)’의 단계에 도달하기 위해서는 필연적으로 소프트웨어의 규모가 폭발적으로 커지게 됩니다. GPS를 읽는 코드, 기압도를 읽는 코드, 이들을 융합해 위치를 추정하는 코드, 모터를 돌리는 코드 등 수십 개의 기능 모듈들 간에 심장 박동처럼 정확하게 데이터를 주고받게 만드는 통로가 절대적으로 필요합니다.

이러한 **데이터의 생산자(Producer)와 소비자(Consumer) 사이의 복잡한 통신을 조율하고 관리하는 보이지 않는 인프라를 백본 미들웨어(Backbone Middleware)**라고 칭합니다.

0.1 펌웨어 내 미들웨어의 존재 이유: 스파게티성 제거

만약 10개의 독립적인 프로그램 기능(모듈)이 존재하고, 이들이 데이터를 공유하기 위해 C++ 전역 변수(Global Variable)나 상호 클래스 포인터를 무작위로 참조(Reference)하게 놔둔다면 최악의 ’스파게티 아키텍처’가 생성됩니다.

graph TD;
    subgraph Without Middleware (Spaghetti)
        GPS --> EKF;
        IMU --> EKF;
        RC_Input --> Commander;
        EKF --> PositionController;
        Commander --> PositionController;
        PositionController --> MotorMixer;
        GPS -.-> Logger;
        EKF -.-> Logger;
    end

하나의 센서 배열 위치가 바뀌거나, 새로운 제어기를 추가하려고 코드를 수정하면, 전체 메모리 참조 구조에 영향을 주어 시스템이 붕괴할 수 있습니다. 현대 무인항공기 아키텍처는 이를 방지하기 위해 각 모듈 사이에 일종의 ‘메모리 우체국(Postal Service)’ 역할을 하는 규격화된 통신 미들웨어를 구축합니다.

0.2 패러다임의 두 극단: uORB (PX4) vs AP_Message (Ardupilot)

PX4와 Ardupilot은 동일한 문제를 풀기 위해 설계되었음에도 불구하고, 앞서 언급한 리얼타임 운영체제(RTOS)의 활용 철학(결정론적 다중 스레드 vs 스레드 오버헤드가 배제된 강력한 단일 루프) 차이로 인해 내부 데이터의 통신 아키텍처 역시 완전히 다른 모습을 갖게 되었습니다.

  1. PX4의 선택: 마이크로 객체 요청 브로커 (uORB)
    PX4-Autopilot은 진정한 다중 프로세스(Multi-process)/다중 스레드(Multi-thread) 환경이므로 절대 모듈 간 메모리를 직접 참조하게 두지 않습니다. 대신 비동기식 퍼블리시-서브스크라이브(Publish-Subscribe) 메커니즘인 uORB를 도입하여 완벽한 모듈의 디커플링(Decoupling)을 추구합니다. ROS(Robot Operating System)의 토픽(Topic) 통신과 철학이 매우 유사하며 확장성과 시스템 통합의 관점에서 탁월한 유연성을 제공합니다.

  2. Ardupilot의 선택: 동기식 메시징 및 객체 싱글톤 구조
    단일 거대 루프 안에서 움직이는 Ardupilot은, uORB처럼 무거운(?) 데이터를 직렬화하고 큐(Queue)에 넣었다 빼는 미들웨어의 중간 지연(Latency) 마저 극도로 기피합니다. Ardupilot의 모놀리식 아키텍처 속에서는 필요한 데이터를 생산하는 객체 지향 클래스가 전역적으로(Singleton 개념과 유사하게) 위치해 있으며, 타 클래스가 AP_ 라이브러리의 멤버 포인터에 찰나의 순간 직결하여 즉시 값을 취하는 직관적이고 동기적인 데이터 공유 및 내부 메시징을 베이스로 합니다.

이러한 두 시스템의 극명한 미들웨어 아키텍처의 철학 차이는, 향후 서드 파티 개발자(Third-party developers)들이 새로운 장애물 회피 소프트웨어를 짜 넣거나 완전히 다른 형태의 수중 드론 용 센서를 부착할 때, 코드를 기여(Contribution)하는 방법론 자체를 송두리째 바꿔놓게 됩니다. 이어지는 하위 절들에서는 uORB 시스템의 혁신성과 데이터 복사 최적화부터, Ardupilot의 결합형 데이터 통신 철학까지 한 걸음 더 깊이 마이크로 단위로 해부하여 설명할 것입니다.
