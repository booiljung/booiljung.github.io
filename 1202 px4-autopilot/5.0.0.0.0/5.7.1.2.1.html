5.7.1.2.1. 글로벌 좌표계(Global) 및 속도 벡터(Velocity Vector) 위주의 외부 제어 입력

데카르트 직교 좌표계(로컬 NED)를 극단적으로 선호하여 로봇공학적 정밀도를 추구하는 PX4와 달리, Ardupilot의 가이디드(Guided) 모드는 **글로벌 좌표계(WGS84: 위도, 경도, 고도)**를 외부 제어 입력으로 받아들이는 데 훨씬 관대하고 최적화되어 있습니다.

이는 Ardupilot이 태생적으로 장거리 비행(Long-range)과 범용적인 지리 정보 시스템(GIS)과의 통합을 중시하는 아키텍처 철학을 가졌기 때문입니다.

0.1 글로벌 좌표계(Global Coordinates)의 직접 수용

Ardupilot은 외부 컴패니언 컴퓨터나 클라우드 관제 서버에서 날아오는 SET_POSITION_TARGET_GLOBAL_INT MAVLink 메시지를 매우 자연스럽게 소화합니다.

  • 컴패니언 컴퓨터에 탑재된 LTE 모뎀이 클라우드 서버로부터 “위도 37.5, 경도 126.9로 이동하라“는 외부 API 응답을 받으면, ROS 2나 파이썬(DroneKit) 스크립트는 이를 굳이 기체의 시작점 기준 직교 좌표계(X, Y 미터 단위)로 번역할 필요가 없습니다.
  • 그저 수신한 GPS 좌표를 그대로 가이디드 모드의 목표점(Target)으로 펌웨어에 던져주면 됩니다.
  • 지구 곡률을 계산하고 구면 스플라인(Spherical Spline) 궤적을 생성하여 모터 추력으로 변환하는 그 끔찍하게 복잡한 계산은 Ardupilot 내부의 AC_WPNav (Waypoint Navigation) 코어 라이브러리가 전담하여 깔끔하게 처리해 줍니다.

0.2 속도 벡터(Velocity Vector) 중심의 스무딩(Smoothing) 제어

비전 센서를 이용해 움직이는 객체를 추적(Object Tracking)하는 것과 같이, 특정한 절대 좌표가 없는 동적인 시나리오에서는 어떻게 할까요? Ardupilot은 이때 속도 벡터(Velocity Vector) 입력에 크게 의존합니다.

가이디드 모드에서 컴패니언 컴퓨터가 “북쪽으로 2m/s, 동쪽으로 0m/s의 속도를 내라“라고 명령을 내리면, Ardupilot 내부의 위치/속도 제어기(AC_PosControl)가 이 벡터를 물고 늘어집니다.
이때 Ardupilot 아키텍처의 특징은 외부에서 거칠게 변하는 속도 명령이 들어오더라도, 파라미터로 설정된 기체의 최대 가속도(Acceleration Limit)와 저크(Jerk, 가속도의 변화량) 한계치를 엄격하게 적용하여 매우 부드러운(Smooth) 피치(Pitch)와 롤(Roll) 기동으로 필터링해 낸다는 점입니다.

0.3 편의성과 제어 해상도의 트레이드오프

이러한 두 가지 입력 방식(글로벌 좌표 및 필터링된 속도 벡터)은 컴패니언 컴퓨터 측의 코딩 100줄을 10줄로 줄여주는 강력한 ’추상화(Abstraction)’의 축복입니다. 대규모 배달 드론 네트워크나 원격 순찰 시스템을 구축할 때 이보다 편할 수는 없습니다.

하지만 반대급부로, AI 비전 알고리즘이 “카메라 프레임의 오차를 0으로 만들기 위해 지금 당장 우측 모터 두 개의 출력을 정확히 5% 올려라“와 같은 즉각적이고 날것(Raw)의 로우 레벨 모터 개입을 시도할 때는 Ardupilot의 이 두터운 추상화 벽(내부 내비게이션 및 스무딩 필터)이 오히려 제어 지연(Latency)과 반응성 저하를 유발하는 족쇄가 되기도 합니다.
