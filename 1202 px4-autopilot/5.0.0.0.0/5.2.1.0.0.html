5.2.1. RTOS(Real-Time Operating System) 활용 철학의 대조

임베디드 비행 제어기(Flight Controller) 상에서 실시간 운영체제(RTOS)의 핵심 목표는 결정론적 시간 보장(Deterministic Timing Guarantee)이다. 센서에서 들어온 데이터를 수백 마이크로초(us) 이내에 믹서(Mixer)를 거쳐 액추에이터로 방출하지 못하면 기체가 뒤집히는 물리적 데드라인을 갖는다.

이 극한의 환경에서 작동하지만, PX4의 NuttX POSIX 생태계와 Ardupilot의 ChibiOS(또는 AP_HAL) 생태계는 RTOS라는 근간 엔진을 사용하는 철학과 심도가 완전히 다르다.

0.1 PX4의 선택: NuttX - “POSIX 표준 기반의 진정한 운영체제 아키텍처”

PX4-Autopilot 아키텍처를 이해하는 첫 단추는 수백만 줄짜리 C++ 코드가 단순히 마이크로컨트롤러(MCU)에서 순차 실행되는 펌웨어가 아니라, **NuttX(넛엑스)라는 정교한 UNIX/Linux 스타일의 유저 스페이스(User Space) OS 위에서 도는 애플리케이션들의 군집(Swarm)**임을 깨닫는 것이다.

  • POSIX API의 투명한 차용: PX4 개발자는 센서 파일이나 디바이스 노드를 제어할 때 일반적인 리눅스에서 다루듯 open(), read(), write(), ioctl() 시스템 콜(System Call)을 하드코드하여 사용한다.
  • 완전한 개별 프로세스/태스크 구동: sensors, ekf2, mc_pos_control, commander 등 수십 개의 PX4 모듈들은 main()에서 하나로 묶여 돌아가지 않는다. 이들은 각기 고유의 스택 사이즈와 커널 스케줄러 내 우선순위(Priority)를 부여받아 완전히 분리된 스레드/태스크(Thread/Task) 형태로 런오버(Run Over)된다. 최상위 권력인 EKF 스레드는 언제든 하위 logger(SD 카드 기록) 태스크의 숨통을 끊어버리고 CPU 자원을 모험적 선점(Preemption)할 수 있다.
  • 리눅스형 콘솔(NuttShell): 이들의 가장 매력적인 아키텍처 철학은 사용성이 리눅스와 동일하다는 점이다. USB 시리얼로 콘솔에 접근하면(NSH), ls, top, free 명령어를 통해 마치 리눅스 배포판 서버를 모니터링하듯 시스템 틱(Tick) 단위로 CPU 할당량과 메모리 상태를 파고들 수 있다.

0.2 Ardupilot의 선택: ChibiOS - “최소한의 스레드 오버헤드와 AP_HAL 모놀리식 통합”

PX4의 우아한 OS 의존성과 반대로, Ardupilot은 **“잦은 컨텍스트 스위칭(Context Switching) 자체가 불필요한 레이턴시(Latency)를 유발하는 악”**이라고 바라본다.

Ardupilot은 NuttX 대신 극단적으로 가볍고 빠른 컨텍스트 스위칭 속도를 자랑하는 ChibiOS(과거에는 FreeRTOS 및 NuttX도 병행)를 기반 OS로 적극 채택했다. 그러나 이것은 아키텍처를 POSIX화 시키기 위한 것이 아니라, 오직 AP_HAL(Hardware Abstraction Layer)의 하드웨어를 추론하고 감싸기(Wrapping) 위한 백엔드(Backend)일 뿐이다.

  • 통제된 최소 스레드 사상: Ardupilot은 PX4처럼 수십 개의 세밀하게 분절된 태스크를 띄워 커널 스케줄러에게 맡겨버리지 않는다. 센서를 읽는 초고속 스레드(SPI/I2C 버스 폴링 전용), 로깅용 저주파 스레드, 모터 출력 제어용 스레드 등 몇 개의 굵직한 통뼈(Bone) 스레드만을 생성한다.
  • 명시적 슬케줄링 오버헤드(Overhead)의 봉쇄: 대부분의 주요 비행 제어 논리(드론 자세 제어, 임무 수행)는 여전히 단일 중심 스레드인 메인 루프(Main Loop, 과거의 슈퍼루프 흔적) 안에서 도는 커스텀 스케줄러(AP_Scheduler)의 지시를 받는다. 개발자는 각 함수를 10Hz, 50Hz, 400Hz 타임 슬롯(Time Slot) 배열에 꽉꽉 욱여넣어 컴파일한다. 즉, RTOS 커널이 멋대로 우선순위를 뒤죽박죽 선점하도록 내버려 두지 않고, 코드 테이블에 박힌 절대 시간을 준수하여 스레드 스위칭 페널티를 제로(0)에 수렴하도록 설계한 모놀리식(Monolithic) 생존 철학이다.

이 결정적인 태스크 스케줄링 패러다임의 양극화는 어떤 결론을 만들어냈는가?

PX4의 다중 모듈 아키텍처는 코드의 가독성이 뛰어나고, 대학원생 컨트리뷰터(Contributor)가 다른 구역을 더럽히지 않은 채 독자적인 제어기 블록만 깔끔하게 추가하기에 완벽한 생태계를 선사했다. 그러나 과도하게 쪼개진 태스크들은 구형 MCU 환경에서 심각한 오버헤드(Overhead) 빈곤을 유발하기도 했다.

반대로 Ardupilot은 방대한 스냅 함수들을 루프 타임 안에 단 1us의 양보 없이 욱여넣는 방식을 통해 구동되므로, 거대한 C++ 코드 베이스 안에서 새로운 기능을 끼워 넣으려면 메인 루프 시퀀스의 지연 가능성(Timing Issue)을 치밀하게 분석하고 모든 클래스의 전역 결합도(Coupling)를 고려해야만 하는 엄청난 진입 장벽의 스파게티 구조를 감내하게 되었다.
