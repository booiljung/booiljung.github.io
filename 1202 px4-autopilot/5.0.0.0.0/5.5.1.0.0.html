5.5.1. 상태 머신(State Machine)에 대한 시각 차이

비행 제어 소프트웨어에서 ’상태 머신(State Machine)’이란 기체가 현재 어떤 조건에 놓여 있는지(예: 부팅 중, 대기 중, 시동 걸림, 비행 중, 비상 추락 중 등)를 명확히 정의하고, 각 상태에서 허용되는 행동(Action)과 다음 상태로 넘어갈 수 있는 전이(Transition) 규칙을 규정한 수학적/논리적 모델을 말합니다.

그리고 조종사가 선택하는 ’비행 모드(Flight Mode)’는 이 거대한 상태 머신 톱니바퀴의 일부로 맞물려 돌아갑니다. 이 논리 구조를 어떻게 소스 코드로 빚어낼 것인지에 대해 PX4와 Ardupilot은 소프트웨어 공학의 양극단에 선 철학을 보여줍니다.

0.1 설계 사상의 분기점: 중앙집권형(Centralized) vs 분산형(Decentralized)

두 펌웨어의 아키텍처 차이는 흔히 **‘독재적인 관제탑(PX4)’**과 **‘자치적인 전문가 집단(Ardupilot)’**으로 비유할 수 있습니다.

  • PX4의 관점 (중앙 제어형 아키텍처):
    PX4는 시스템 전체의 무결성을 최우선으로 삼습니다. 이를 위해 Commander라는 거대하고 절대적인 권력을 가진 중앙 모듈을 두어, 세상의 모든 센서 상태, 배터리 잔량, 조종기 신호 등을 단독으로 감시합니다. 조종사가 비행 모드를 변경하려 하면 오직 Commander만이 이 요청을 심사하여 승인하거나 거부(Reject)합니다.
  • Ardupilot의 관점 (객체 지향적 분산 아키텍처):
    Ardupilot은 실용성과 확장성을 우선시합니다. 각 비행 모드(Stabilize, Loiter, Auto 등)는 소스 코드 상에서 철저히 독립된 C++ 클래스(Class) 객체로 구현됩니다. 조종사가 Loiter(위치 제어) 모드를 켜면, 해당 모드 객체 자체가 깨어나 스스로 GPS 상태를 확인하고 자신이 켜질 수 있는지 없는지 주도적으로 판단(Self-check)합니다.

0.2 PX4의 엄격한 분리: 상태(Status)와 모드(Mode)는 다르다

PX4의 상태 머신이 갖는 가장 큰 특징은 논리 값의 층위를 철저하게 분리해 놓았다는 점입니다.

  1. Arming State (구동 상태): 프로펠러가 돌고 있는지(Armed) 아닌지(Standby).
  2. Vehicle Status (기체 건강망): GPS가 멀쩡한지, 배터리가 충분한지 등 전역적인 하드웨어 상태.
  3. Nav State (내비게이션/비행 모드): 조종사가 요망하는 목표 제어 형태 (Position 모드, Mission 모드 등).

PX4의 Commander는 이 세 가지 축을 교차 검증합니다. 즉, 비행 모드라는 것은 단순히 기체가 취해야 할 여러 ‘내비게이션 상태(Nav State)’ 중 하나일 뿐이며, 이 상태로 넘어가기 위한 모든 사전 검열은 중앙의 복잡한 Tree 구조 논리 회로를 강제로 거쳐야만 합니다. 이로 인해 단 한 치의 논리적 모순도 발생하지 않는 완벽한 상태 트리가 완성됩니다.

0.3 Ardupilot의 코드 실용주의: 직관적인 모듈 격리

반면 Ardupilot의 방식은 오픈소스 기여자(Contributor)들에게 압도적인 편리함을 제공합니다.

어떤 개발자가 드론이 마치 취객처럼 비틀거리며 날게 하는 완전히 새로운 Drunk Mode를 만들고 싶다고 가정해 보겠습니다.

  • PX4 시스템에서는 거대한 Commander.cpp 파일을 열어 수천 줄의 전역 상태 머신 트리에 자신의 모드가 들어갈 나뭇가지(Node)를 결함 없이 엮어 넣어야 하는 끔찍한 검증 작업을 거쳐야 합니다.
  • 하지만 Ardupilot에서는 그저 ModeDrunk.cpp라는 파일 하나를 만들고, 해당 파일 안에 init() (모드 진입 시 조건 검사 로직)과 run() (모드 작동 시 제어 로직) 함수만 작성해서 붙여 넣으면 끝납니다.

이러한 Ardupilot의 국지적(Localized) 상태 판단 구조는 기상천외한 기능을 찍어내듯 추가할 수 있는 확장의 원동력이 되었습니다. 하지만 각 모드 프로그래머들이 자신의 모드 파일 안에 안전망(GPS 체크 등) 코딩을 실수로 누락하면, 해당 모드에서만 기체가 치명적인 오류를 일으키는 파편화(Fragmentation) 리스크를 내부적으로 떠안고 있습니다.
