5.2.2.1. PX4: uORB(Micro Object Request Broker) 비동기 퍼블리시-서브스크라이브 모델

PX4-Autopilot 아키텍처가 전 세계 연구자와 엔지니어들에게 가장 사랑받는(그리고 이식하기 편리한) 이유를 단 하나만 꼽으라면, 주저 없이 그 근본인 uORB (Micro Object Request Broker) 미들웨어를 들 수 있습니다. uORB는 PX4 펌웨어 내의 수많은 독립적인 다중 스레드 프로세스들이 서로의 존재를 알 필요 없이, 빠르고 안전하게 데이터를 교환할 수 있도록 설계된 핵심 IPC (Inter-Process Communication) 신경망입니다.

당신이 만약 로봇 운영 체제(ROS 1, ROS 2) 개발 경험이 있다면, uORB의 구조는 매우 친숙하고 직관적으로 느껴질 것입니다. uORB의 알파와 오메가는 바로 비동기식 퍼블리시-서브스크라이브(Asynchronous Publish-Subscribe) 모델에 있습니다.

0.1 퍼블리시-서브스크라이브(Publish-Subscribe) 모델의 원리

전화 통화(Point-to-Point)처럼 송신자와 수신자가 직접 연결되어야만 통신이 가능한 방식과 반대로, uORB의 퍼블리시-서브스크라이브 패턴은 방송국의 라디오 전파 브로드캐스팅과 신문 구독의 철학을 따릅니다.

  • 토픽 (Topic)과 페이로드 (Payload): 데이터의 주제를 의미합니다. 예를 들어 센서 모듈이 읽어들인 자이로/가속도 값은 sensor_combined라는 이름의 토픽(채널)을 통해 유통되며, 그 안에 담긴 구체적인 C++ 구조체(struct) 데이터 내역이 페이로드(Payload)가 됩니다.
  • 발행자 (Publisher): sensors 모듈과 같은 데이터의 생산자는 메모리에 데이터를 측정한 뒤, uORB라는 중앙 브로커(Broker) 사무국에 “방금 최신 sensor_combined 데이터가 들어왔어!” 라고 던지기만(Publish) 할 뿐, 이 데이터를 누가, 언제, 얼마나 가져갈지는 파악하지도 관여하지도 않습니다.
  • 구독자 (Subscriber): 드론의 자세를 계산하는 EKF2 추정기(Estimator) 모듈이나 logger 모듈과 같은 데이터의 소비자들은, 오직 uORB 브로커에게 “나는 sensor_combined 토픽 정보가 들어오면 알려줘“라고 등록(Subscribe/Listen)만 해둔 채 조용히 자신의 루프를 돌고 있습니다.

0.2 uORB 시스템의 비동기적(Asynchronous) 설계 철학

왜 다이렉트로 메모리의 값을 읽지 않고(Synchronous Reader), 굳이 uORB 브로커라는 미들웨어를 거치는 비동기 방식을 채택했을까요?

graph LR;
    subgraph PX4 Firmware Architecture
        A[GPS Driver Module] -->|Publish| B((uORB Broker))
        C[IMU Driver Module] -->|Publish| B
        
        B -.->|Subscribe| D[EKF2 Estimator Module]
        B -.->|Subscribe| E[SD Logger Module]
        B -.->|Subscribe| F[Telemetry Module]
    end

이 다이어그램이 uORB의 ’시간적/공간적 비동기성’의 미학을 극명히 보여줍니다.
IMU 센서는 초당 1,000번(1kHz) 데이터를 찍어내 퍼블리시합니다. 하지만 텔레메트리(Telemetry) 라디오 모듈은 속도가 느려서 초당 50번(50Hz) 밖에 송신할 능력이 없습니다. 만약 두 코드가 동기적인(Synchronous) 실행 루프에 결합되어 있었다면, IMU 드라이버는 텔레메트리 모듈이 데이터를 송신해 낼 때까지 꼼짝없이 병목(Blocking)에 멈춰있어야만 합니다.

하지만 uORB 생태계 안에서 이들은 독립된 스레드로 살아 있습니다. 발행자(Publisher)는 구독자의 상태와 무관하게 자신의 페이스(Pace)대로 독립적으로 메모리 버퍼에 데이터를 전송(Write)하고 자기 할 일을 이어갑니다. 텔레메트리 구독자(Subscriber) 역시 자신이 처리할 수 있는 속도(50Hz)로 가장 최신 버퍼에 갱신된 데이터를 비동기적으로(Asynchronously) 꺼내 읽어가면(Read) 그만입니다.

0.3 uORB의 내부 구현체: NuttX 및 파일 시스템 기반 추상화

기술적인 심연으로 조금 더 들어가면, 커널이 NuttX로 구동되는 PX4에서 uORB 객체 데이터들은 실제로 어디를 통해 교환되는 것일까요? PX4는 놀랍게도 유닉스의 가상 파일 시스템(VFS, Virtual File System) 구조를 그대로 응용합니다.

부팅 단계에서 uorb start 커맨드가 실행되면, PX4 내부에 /obj 라는 가상의 디렉토리 마운션이 생성됩니다. 만약 sensor_combined 토픽이 광고(Advertise)되기 시작하면 시스템은 /obj/sensor_combined라는 임시 구상 파일 노드(Node)를 동적으로 배정합니다. 구독자는 그 노드 주소를 열어(Open) 마치 리눅스의 파일을 읽고 쓰듯(read/write/ioctl 시스템 콜 매핑) 데이터를 빼갑니다.

이 완벽에 가까운 비동기화 통신 패턴으로 인하여 수십, 수백만 줄의 코드가 각자의 도메인 내에서 병렬적인 멱등성(Idempotence)을 침범당하지 않게 되었으며, PX4 코어 프레임워크가 어떤 하드웨어 환경(ARM, Snapdragon, x86 Simulator 등)으로 포팅되더라도 유연하게 작동하는 궁극적인 이식성의 근본이 되었습니다.
