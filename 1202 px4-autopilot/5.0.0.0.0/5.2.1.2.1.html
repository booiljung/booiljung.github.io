5.2.1.2.1. 단일 주(Main) 스레드 의존성 및 콜백(Callback) 스케줄러 동작

Ardupilot은 PX4의 ‘POSIX 멀티 프로세스’ 이념에 맞서, 초정밀하게 제어되는 중앙 집중형 메인 루프(Main Loop)에 의한 타임 슬라이스(Time Slice) 콜백(Callback) 스케줄러 시스템이라는 지극히 단일하고도 강압적인 스레딩 정책을 유지한다.

하위에 깔린 운영체제(ChibiOS 등)가 다중 스레드를 제공할지언정, Ardupilot은 비행 역학에 관여하는 중추적인 제어 코드를 수십 개의 스레드로 나누어 운영체제의 변덕스러운 간섭(Preemption)에 맡기는 것을 용납하지 않는다.

0.1 단일 주(Main) 스레드의 절대 권력과 구조

Ardupilot(ArduCopter 등)의 비행 알고리즘 심장부는 사실상 단 하나의 거대한 메인 루프 실행 흐름(Main Thread)에 의해 구동된다고 보아도 무방하다. (물론 SPI 센서의 I/O 통신, 로깅(Logging)처럼 지연이 길고 블로킹(Blocking)이 발생하는 주변부 기능만은 AP_HAL 레이어에서 매우 한정적인 별도 백그라운드 스레드로 격리하여 돌린다.)

기체의 위치, 자세 추정(EKF 계산), PID 제어기(Controller)의 작동, RC 통신 파싱, 비행 모드(Flight Mode) 판단 등, 기체가 떨어지느냐 마느냐를 결정하는 중대 논리 함수들은 모두 이 단일 메인 스레드의 동기화된 호출 트리(Call Tree) 내에 묶여 있다.

이와 같은 “주요 비행 로직의 1-스레드 몰빵 정책“은 대단히 중요한 목적을 가진다.
바로, 여러 함수가 동시에 같은 객체(안전자치 플래그 등)를 수정하려고 할 때 발생하는 락(Lock)이나 데드락(Deadlock)과 같은 멀티 스레드 프로그램 특유의 소모적이고 디버깅 불가능한 데이터 충돌 버그를 아키텍처 단위에서 원천 봉쇄해버리는 것이다. 단일 스레드에서는 모든 연산이 절차적으로 발생하므로 동기화(Mutex) 오버헤드 자체가 증발해 버린다.

0.2 핵심 사령탑: AP_Scheduler 기반의 콜백 실행 모델

단일 스레드 안에서 수십 개의 부가 기능(배터리 전압 리딩, 텔레메트리 전송, LED 깜빡임 등)을 “마치 동시에 실행되는 것처럼” 시분할 해야 하므로, 전적인 책임은 커널OS가 아닌 Ardupilot이 자체 구현한 커스텀 스케줄러인 AP_Scheduler 코어 라이브러리로 이관된다.

이 시스템은 선점형(Preemptive)이 아닌 철저한 협력적(Cooperative) 폴링형 콜백(Callback) 스케줄링 방식이다.

개발자는 소스 코드(예: Copter.cpp)의 스케줄러 테이블 공간에 함수 포인터(Function Pointer) 형식으로, “어떤 함수를 몇 헤르츠(Hz, 초당 실행 횟수)로 호출할 것인지, 그리고 그 함수가 실행될 때 잡아먹을 수 있는 최대 수행 시간(Max duration, 마이크로초 단위) 허용치는 얼마인지“를 스펙 시트처럼 명시적인 매크로 문맥(SCHED_TASK)으로 빽빽하게 등록해 둔다.

메인 틱(Tick, Copter의 경우 보통 400Hz)이 떨어질 때마다 AP_Scheduler는 테이블을 순회(Polling)하며, 현재 이 타임루프 슬롯에 남은 여유 잔여 마이크로초(us) 시간과 각 태스크에 할당된 시간을 저울질한 뒤, 가장 실행 가중치 우선순위가 높은 콜백 함수를 실행하고 그 즉시 권한을 리턴(Return) 받는다.

0.3 협력적(Cooperative) 스케줄링의 공포와 코드 한계 엣지(Edge)

이 기계적일 정도로 명확한 아키텍처가 순조롭게 돌기 위해서는 하나의 무시무시한 전제 조건이 붙는다.
메인 사이클 내에 불려 가는 모단일 컴포넌트 함수는 절대 내부 통신 대기 무한루프(While-blocking)나 느린 연산으로 반환(Return) 시점을 지연시켜서는 안 된다는 철칙이다.

만약, 어떤 사용자가 새롭게 추가한 초음파 거리 센서(Sonar)의 I2C 응답을 기다리는 드라이버 코드를 메인 스케줄러 테이블에 올려두었는데 이 코드가 응답 지연으로 루프 탈출을 1밀리초(ms) 지연시킨다고 가정해 보자.
PX4의 NuttX 환경이었다면, OS 커널이 “우선순위 역전, Sonar 스레드 죽어라” 하고 자세 제어 스레드를 강제로 다시 깨워서 모터를 쳐내 기체를 살렸을 것이다.
하지만 Ardupilot의 스케줄러 테이블에서는 그 지연된 단독 함수가 스스로 빠져나오지(Yield) 않으면, 그다음에 불리기를 애타게 기다리고 있던 모터 믹싱 콜백 함수가 제시간(Timing)에 호출되지 못한다. 결국 메인 루프 타임 슬립 파괴가 일어나 드론 전체의 자세 유지 타이밍이 박살 나며 추락하게 된다.

Ardupilot은 운영체제 레벨의 문맥 교환 부담(Context Overhead)을 날려버리는 대신, 그 짐과 타이밍 준수라는 치명적 스트레스를 수많은 AP_ 라이브러리를 코딩하는 개별 개발자의 등짝 위로 통째로 전가시킨 위태로우나 실용적인 외줄 타기 아키텍처를 자랑한다.
