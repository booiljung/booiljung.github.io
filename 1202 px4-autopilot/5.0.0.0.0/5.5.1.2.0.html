5.5.1.2. Ardupilot의 개별 모드 클래스(Mode Class) 기반 분산형 객체 지향 구조

단 하나의 거대한 Commander 파일이 천하를 통일한 PX4와 대조적으로, Ardupilot 소스 코드(예: ardupilot/ArduCopter/) 디렉토리를 열어보면 수십 개의 mode_*.cpp 파일들이 흩뿌려져 있는 것을 볼 수 있습니다.

Ardupilot은 중앙에서 모든 상태 전이를 관리하는 대신, 각각의 비행 모드를 독립적인 C++ 객체(Object)로 격리하여 권한을 위임하는 분산형 아키텍처를 채택했습니다. 본 절에서는 이러한 객체 지향적 모드 설계가 어떻게 작동하는지 분석합니다.

0.1 기반 클래스(Base Class) 상속을 통한 모듈화

Ardupilot의 모든 비행 모드(Stabilize, Loiter, Auto, RTL 등)는 하나의 가상 부모 클래스인 Mode 클래스를 상속받아 탄생한 자식 클래스(Child Class)들입니다.

부모인 Mode 클래스는 모든 모드가 반드시 지켜야 할 일종의 규격(인터페이스)을 정의해 둡니다.
가장 핵심이 되는 두 가지 규격은 다음과 같습니다.

  • init() (초기화 및 진입 장벽): 조종사가 이 모드로 전환하려고 파문(Request)을 던질 때 단 한 번 호출되는 함수입니다.
  • run() (제어 루프 실행): 기체가 해당 모드로 진입하는 데 성공했다면, 매 루프 사이클(예: 400Hz)마다 반복적으로 호출되어 실제 비행 제어 로직을 가동하는 함수입니다.

0.2 셀프 체크(Self-Check): 모드 스스로 결정하는 진입 조건

PX4에서는 커맨더가 중앙에서 GPS 상태를 보고 모드 진입을 허락할지 말지 결정했습니다. 반면 Ardupilot에서는 조종사가 Loiter (위치 유지) 모드의 스위치를 켤 때, 제어권이 곧바로 ModeLoiter 객체의 init() 함수로 통째로 넘어갑니다.

그러면 ModeLoiter 객체가 스스로 현재 EKF(상태 추정기)의 GPS 유효성 플래그를 확인합니다.

“나는 위치를 유지해야 하는 Loiter 모드인데, 지금 GPS 신호가 충분한가?”

만약 충분하다면 init() 함수는 true를 반환하여 기체를 성공적으로 Loiter 모드에 안착시킵니다. 하지만 GPS가 불량하다면 init()은 false를 내뱉으며 조종사의 모드 전환 요구를 스스로 튕겨버립니다(Reject). 이처럼 ’자신이 살 수 있는 환경인지 아닌지’를 모드 객체 본인이 직접 판사처럼 결정하는 자치권(Autonomy)을 부여받은 것입니다.

0.3 분산 구조의 명과 암: 확장성과 파편화

이러한 분산형 로직 캡슐화(Encapsulation)는 압도적인 속도의 **커스텀 모드 개발(Extensibility)**을 가능하게 만들었습니다.

전 세계의 연구진이나 기업이 자신들만의 독특한 배달용 비행 모드를 만들고 싶다면, 복잡한 중앙 상태 머신 코드를 단 한 줄도 건드릴 필요가 없습니다. 그저 깡통 ModeDeliver.cpp 파일을 하나 생성하고, 부모 클래스를 상속받아 init()과 run() 함수만 마음대로 채워 넣으면 그 즉시 완벽하게 작동하는 비행 모드가 탄생합니다. 이는 Ardupilot이 수많은 변종 상용 드론의 베이스 펌웨어로 폭넓게 사랑받는 결정적 이유입니다.

하지만 이 자치권은 파편화(Fragmentation)와 예외 처리 누락이라는 치명적인 대가를 치러야 합니다.
숙련되지 않은 개발자가 새로운 모드의 init() 함수에 안전장치(예: 배터리 체크)를 깜빡하고 코딩하지 않았다면, 기체는 배터리가 없는 상태에서도 그 모드로 강제 진입해버린 후 허무하게 추락할 수 있습니다. 각 모드에 책임을 떠넘긴 대가로, 전체 시스템을 관통하는 100% 무결점 헬스 체크를 수학적으로 보장하기가 구조적으로 매우 어려워진 것입니다.
