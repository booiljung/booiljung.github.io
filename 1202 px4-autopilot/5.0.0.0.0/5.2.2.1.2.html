5.2.2.1.2. 데이터 복사 최소화를 위한 공유 메모리 기반 메시징 최적화

퍼블리시-서브스크라이브 패턴을 이용해 완벽한 비동기 스레드 독립성을 취한 “마이크로 객체 요청 브로커(uORB)“는 아키텍처 설계상 우아하지만, 시스템 프로그래머의 눈에는 엄청난 비용 결함의 불씨를 안고 있는 구조이기도 합니다. 다중 프로세스(Multi-process) 간 통신을 위해 데이터를 한 모듈의 메모리에서 꺼내 커널 소켓이나 파이프로 복사(Copy)하는 행위 자체가 드론의 빈약한 마이크로컨트롤러(MCU)에게는 과도한 지연(Latency)과 캐시 미스(Cache Miss), 램 메모리 점유의 낭비를 강요하기 때문입니다.

이에 PX4의 핵심 창립자들은 이 무거운 uORB 개념을 임베디드 코어 환경에 끼워 넣기 위해, 공유 메모리(Shared Memory) 아키텍처와 제로 카피(Zero-Copy) 사상을 결합하여 메시징 파이프라인(Messaging Pipeline)의 속도와 병목을 기적적으로 최적화해 냈습니다.

0.1 직렬화(Serialization) 배제: 구조체(Struct)의 즉석 전달

일반적인 네트워크 미들웨어나 PC 기반 ROS 1/2 시스템에서는 구조체를 전송하기 위해 이진(Binary) 직렬화나 JSON 텍스트 파싱을 사용하여 바이트 배열로 바꾸는(Pack/Unpack) 값비싼 변환 작업을 필수적으로 거칩니다.

하지만 PX4 내부에서 uORB가 처리하는 토픽 메시지들은(예: .msg 파일로부터 컴파일 타임에 C/C++ 코드로 자동 생성된 헤더들) 직렬화 오버헤드가 단 1바이트도 발생하지 않도록 철저히 차단되어 있습니다.

송신자 모듈이 만든 데이터 구조체의 전체 크기(예: 32바이트)를 바이트 단위 재배치 없이 C++의 패킹된(Packed) 메모리 레이아웃 구조체 그 자체 통째로 포인터 블록 단위에 옮기며, 받는 리시버 모듈 측에서도 캐스트 연산자(Struct Cast) 한 방만으로 수신 버퍼 덩어리를 즉각 해석하여 본래의 데이터를 뽑아냅니다. 오직 최소한의 C언어 단위 memcpy()(메모리 메모리 고속 복사 명령) 만 일어날 뿐입니다.

0.2 가상 파일 시스템 내의 중앙집중형 메모리 풀(Memory Pool) 모델

더 무서운 최적화는 여러 구독자가 데이터를 요구할 때 발생합니다. 발행자(Publisher) 모듈 1곳에서 만든 센서 데이터를, 구독자(Subscriber) 모듈 5곳에서 uORB를 통해 달라고 요청했다고 가정합시다. 과거 일반적인 큐잉(Queueing) 아키텍처였다면, 구독자 5명을 위해 메모리를 5번이나 각각 할당하여 메세지를 복사해서 던져주었을 것입니다.

PX4 uORB는 이를 비웃듯 공유 메모리 링 버퍼(Shared Memory Ring Buffer) 아키텍처를 사용합니다.

graph TD;
    subgraph uORB Shared Memory Area
        Node[/dev/obj/sensor_gyro (Generation Count: 994)/]
        Buffer[Ring Buffer: Payload Data Slot]
    end
    
    Pub(Publisher Module:<br/>sensors) -- 1. Write Data (memcpy) &<br/> Increment Gen Count --> Buffer
    
    Sub1(Subscriber A:<br/>EKF2) -- 2. Check Gen Count<br/>Read (memcpy) --> Buffer
    Sub2(Subscriber B:<br/>Logger) -- 2. Check Gen Count<br/>Read (memcpy) --> Buffer

발행자는 독립된 본인의 메모리에서 uORB 커널이 관리하는 중앙 공유 노드 구조체(/obj/..) 공간으로 딱 1번만 memcpy() 하여 올려버립니다. 그 직후 uORB 시스템은 “이 메세지는 994번째 발행된 데이터다“라는 **세대 카운터(Generation Counter)**만 원자적 연산(Atomic)으로 1개 갱신합니다.
스레드 대기 모드에서 동시 다발적으로 깨어난 여러 독립된 구독자들은 자신의 고유 인덱스와 세대(Gen) 카운터 숫자를 대조해 본 후, “새 파일이 올라왔군” 하고 중앙 공유 영역 메모리로 직접 날아가 각각 딱 한 번씩만 memcpy()로 빼내 갑니다.

0.3 왜 완전한 Zero-Copy 포인터 참조를 쓰지 않았는가?

여기서 한 가지 컴퓨터 공학도의 질문이 생길 수 있습니다. “왜 공유 메모리 포인터(Pointer) 주소만 서로 주고받지 않고, 굳이 한 번의 복사 연산(1-Copy)인 memcpy를 남겨두어 오버헤드를 안고 있는가?”

이 타협의 지점이 바로 강결합 구조의 공포를 피해 아키텍처 철학을 보존한 결정적 설계입니다.
만약 모듈들이 중앙 메모리 포인터를 쥐고(Reference) 바로 수정 연산을 한다면, A모듈이 값 위에서 연산하고 있는데 0.1밀리초 만에 돌연 B모듈이나 인터럽트가 메모리 값을 수정하게 되는 **데이터 레이스(Data Race, 메모리 찢어짐)**에 봉착하여 결국 락(Mutex Lock)을 걸어야 하며, 스레드 병목으로 시스템은 멈추게 됩니다.
따라서 발행자와 구독자가 자신만의 로컬 스택 영역에 찰나의 캐시 카피(Copy)를 생성하도록 강제함으로써, **뮤텍스 락프리(Lock-free)**를 유지하여 스케줄러 간섭 없이 다중 CPU 코어 효율마저 극한으로 끌어올리는, 완벽한 타이밍 무결성을 확보한 것입니다.
