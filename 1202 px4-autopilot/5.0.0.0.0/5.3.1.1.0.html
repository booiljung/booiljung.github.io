5.3.1.1. PX4 (ECL/EKF2): 단일 인스턴스 기반의 시계열 지연(Time-delay) 보상 최적화

비행 제어 소프트웨어에서 상태 추정기(Estimator)가 직면하는 가장 치명적인 물리적 난제는 바로 ‘센서 데이터의 시간적 불일치(Time-delay)’ 현상입니다. 기압계, GPS, 비전 센서는 측정 시점부터 내부 연산을 거쳐 메인 프로세서로 데이터가 도달하기까지 각기 다른 통신 지연 시간(Latency)을 갖습니다.

PX4의 주력 상태 추정 알고리즘인 ECL(Estimation and Control Library) 기반의 EKF2는 이러한 센서 간의 도착 시간 오차를 어떻게 완벽하게 동기화하고 보상하는지에 아키텍처의 사활을 걸고 설계되었습니다. 본 절에서는 단일 인스턴스 필터를 유지하면서도 과거의 시간을 추적하여 정교하게 타임라인을 통일하는 PX4만의 시계열 지연 보상 최적화 기법을 심층적으로 분석합니다.

0.1 센서 스태그네이션(Stagnation)과 시간 지연의 위험성

확장 칼만 필터(EKF)는 근본적으로 “지금 이 순간(Current Time)“의 상태를 예측하고 보정하는 엔진입니다. 하지만 실세계를 측정하는 센서들은 다음의 예시와 같이 각자 다른 속도와 병목을 가집니다.

  • IMU (가속도/자이로): 측정부터 도착까지의 지연(Delay)이 사실상 0~2ms 수준으로 시스템의 가장 빠른 기준 시계(Base Clock) 역할을 합니다.
  • GPS (GNSS): 위성 신호를 수신하고 수신기 칩셋에서 NMEA 포맷으로 계산되어 UART 직렬 배선을 타고 넘어오는 데 보통 100ms ~ 200ms의 막대한 지연이 발생합니다.
  • 컴퓨터 비전 (VIO / 광학 흐름): 카메라 이미지를 캡처해 Companion Computer(ROS2)에서 특징점(Feature)을 추출하고 uXRCE-DDS를 거쳐 넘어올 때, 프레임 상황에 따라 지연 시간이 **가변적(Jitter)**으로 요동칩니다.

만약 EKF가 도착한 GPS 데이터를 무작정 ’현재 시점’의 위치라고 간주하고 필터를 업데이트(Update)해 버리면 어떻게 될까요? 기체는 이미 0.2초 전에 통과했던 위/경도 좌표로 끌려가게 되며, 이는 비행 제어에서 치명적인 진동(Oscillation)과 상태 발산(Divergence)을 촉발합니다.

0.2 타임머신 알고리즘: 시연 지연(Time-delay) 보상 메커니즘

PX4의 EKF2는 이 문제를 해결하기 위해 관측 시간 지연(Observation Time Delay) 보상이라는 마치 타임머신과도 같은 수학적/아키텍처적 기법을 기저율(Baseline)로 탑재하고 있습니다. 이 알고리즘의 핵심 사상은 매우 직관적입니다. “GPS 데이터가 200ms 전의 것이라면, EKF 필터 자체를 200ms 전의 과거 상태로 되돌린 다음 거기서부터 다시 계산하자“는 것입니다.

이를 구현하기 위하여 PX4 EKF2 모듈은 내부에 정밀한 시계열 메모리 구조체를 동적으로 관리합니다.

  1. 지연 지평선(Delay Horizon) 설정: EKF 모듈은 펌웨어 매개변수(Parameter)인 EKF2_GPS_DELAY (기본값 약 110ms), EKF2_BCOEF_DELAY 등의 값을 통해 각 이종 센서들의 평균 하드웨어 지연 시간을 시스템 부팅 시점에 숙지합니다.
  2. 과거 궤적의 버퍼링(Traj Buffering): 가장 빠른 IMU 데이터(400Hz)를 기반으로 상태(State)가 예측될 때마다, EKF2는 과거 일정 시간(통상 수백 밀리초) 동안의 ‘자세 쿼터니언, 속도, 위치, 센서 퓨전 공분산(Covariance)’ 정보 세트를 촘촘하게 배열(Array) 메모리에 임시 저장(History Logging) 해 둡니다.
  3. 지연된 데이터의 도착 및 과거 퓨전(Fusion in the Past): 늦게 출발한 GPS 좌표가 마침내 uORB 토픽으로 도착합니다. EKF2는 이 GPS 스탬프 시간과 동기화된(Matched) 자신의 ‘과거 버퍼링 상태’ 메모리 번지로 정확히 타임 점프(Time Jump) 합니다.
  4. 현실 복귀(Fast-forward): 과거로 돌아가서 GPS 측정값(Measurement)과 당시의 예측값(Prediction) 간의 오차(Innovation)를 구한 뒤, 필터 게인을 계산하여 **과거의 시점에서 상태 보정(Update)**을 단행합니다. 그 직후 IMU 버퍼를 이용해 다시 눈 깜짝할 새 재연산(Re-propagation)을 수행하여 ’현재 시간(Now)’까지 동기화를 끌어올립니다.

이 일련의 과정들은 펌웨어의 수학적 깊이를 보여주는 절정이라 할 수 있으며, 이로 인해 PX4의 비행체는 단 1개의 EKF 인스턴스만을 구동하면서도 고속/가변 지연이 혼재하는 멀티 센서 환경에서 완벽한 칼만 추정의 매끄러움을 획득하게 됩니다.

0.3 단일 인스턴스(Single Instance) 집중 아키텍처의 철학

후술할 Ardupilot(EKF3)이 병렬적으로 여러 개의 EKF 엔진을 동시 구동하는 방향성으로 발전한 것과 큰 대조를 이루게도, PX4의 메인 개발 그룹인 Auterion 및 Dronecode 재단은 오랜 기간 **“수학적으로 완벽히 보상된 단일 EKF 인스턴스”**를 선호해 왔습니다 (최근 EKF2 다중 인스턴스 기능이 추가되기는 하였으나, 설계의 심장부는 여전히 통합 버퍼링 모델에 뿌리를 둡니다).

이러한 단일 인스턴스 지연 보상 아키텍처가 PX4 환경에 부여하는 철학적 이점은 다음과 같습니다.

  • 메모리 및 CPU 풋프린트 절약: 마이크로컨트롤러(NuttX 환경)의 한정된 연산 능력하에서, 과거 궤적 버퍼 메모리 할당만으로 퓨전을 끝마치기 때문에 불필요한 행렬 역연산 쓰레드 스파이크(Spike)가 방지되어 시스템 전체 스케줄링이 극도로 예측 가능(Deterministic) 해집니다.
  • 튜닝의 직관성(Tuning Simplicity): 상태 추정 디버깅 시, 사용자는 여러 인스턴스의 경쟁을 고려할 필요 없이 시간 지연 파라미터(EKF2_*_DELAY)와 해당 센서의 노이즈 공분산 파라미터 계수 조정이라는 단일화된 수학 방정식의 틀 안에서 기체를 빠르게 안정화시킬 수 있습니다.

이러한 정밀한 시계열 메모리 동기화 체계는, 현대 R&D 환경에서 ROS2 기반의 고지연 네트워크(Wi-Fi, LTE 등)를 타고 들어오는 옵티컬 플로우 및 VIO 좌표를 결합하여 비전 기반 완전 자율 비행(Vision-based Autonomy) 시대를 여는 데 핵심적인 토대가 되었습니다.
