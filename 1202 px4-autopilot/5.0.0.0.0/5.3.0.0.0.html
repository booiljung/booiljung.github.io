5.3. 상태 추정기(Estimator) 아키텍처와 센서 퓨전 전략

멀티로터, 고정익, 잠수정 등 그 어떤 형태의 무인 이동체(Autonomous Vehicle)든 상관없이 자율 비행의 근간은 결국 하나의 형이상학적 질문으로 귀결됩니다. “나는 지금 우주 공간 상의 어디에, 어떤 자세(Attitude)로 위치하고 있는가?” 그리고 “나는 1초 뒤에 어디로 향할 것인가?”

이 철학적, 공학적 질문에 대한 수학적 해답을 내놓는 핵심 모듈을 무인항공기 아키텍처에서는 **상태 추정기(State Estimator)**라고 칭합니다. 아무리 모터 피드백 제어 루프가 1,000Hz로 촘촘하게 짜여 있다 한들, 기체가 인식하는 현재의 ’상태(자세, 위치, 속도)’가 거짓이라면 드론은 즉시 콘크리트 벽으로 곤두박질치게 될 것입니다. 본 절에서는 현대 드론 소프트웨어가 복수 개의 불완전한 센서들을 어떻게 결합하여(Fusion) 오차율을 기적적으로 상쇄하는지, 그 근원적 아키텍처 전략에 관해 탐구합니다.

0.1 센서의 태생적 불완전성: 퓨전(Fusion)이 필요한 이유

왜 단순히 하나의 센서 값만 읽어서 비행하지 않고 무겁고 복잡한 상태 추정기(Estimator) 아키텍처를 도입해야만 할까요? 그것은 자연계의 법칙을 측정하는 물리 센서들이 저마다 태생적인 한계점과 기만성(Deception)을 띠고 있기 때문입니다.

  1. 가속도계(Accelerometer)의 노이즈: 가속도계 데이터 하나만으로 진북(Gravity Down)을 참조한 수평자세를 계산할 수는 있습니다. 그러나 프로펠러 모터가 돌아가는 순간, 드론 뼈대에 심한 진동 노이즈(High-frequency vibration)가 생겨 센서값이 수백 G를 널뛰어 도저히 쌩 데이터를 신뢰할 수 없습니다.
  2. 자이로스코프(Gyroscope)의 적분 오차(Drift): 자이로는 각속도를 측정하므로 진동에 상대적으로 강합니다. 하지만 각속도를 계속 적분(+)시켜 현재의 각도(자세)를 구할 때, 초기 상태 센서 결함에 의해 초당 0.1도의 미세한 누적 오차(Drift)만 생겨도, 1분이 지나면 드론은 자신이 6도 기울어있다고 완전히 착각하게 됩니다.
  3. GPS의 지연(Latency) 통신망: GPS 위성에서 지구 좌표를 가져오지만 그 수신 속도는 5Hz~10Hz로 매우 느립니다 (약 100~200밀리초 딜레이). 바람에 의해 기체가 좌측으로 밀릴 때 GPS만 보고 모터를 제어하려 들면, 컨트롤 속도가 뒤처져 시계추처럼 진동하는(Toilet Bowling) 현상이 발생합니다.

해결책은 이 약점들을 상호보완할 수 있는 수학적 바구니에 전부 때려 넣고 확률 모델로 통일하는 것입니다. 자이로의 빠른 속도감(고주파 신뢰)과, 가속도/GPS의 장기적 안정성(저주파 신뢰)을 비빔밥처럼 비벼서 궁극의 깨끗한 포장육 ’State(상태)’를 생산해 내는 행위가 바로 센서 퓨전(Sensor Fusion) 전략의 핵심 가치입니다.

0.2 컴플리멘터리 필터(CF)에서 EKF로의 진화

PX4 및 Ardupilot 펌웨어의 초창기(8비트 AVR 시절)에는 연산력이 약해, 상태를 추정할 때 **상보 필터(Complementary Filter, CF)**라는 단순 가중치 합산기를 썼습니다. 예를 들어 ‘자이로스코프 값은 단기적으로 95% 신뢰하고, 가속도 필터는 저주파로 5% 범위 내에서 누적된 적분 오차만 슬쩍 당겨(Pull) 보완해라’ 같은 휴리스틱(Heuristics)한 수식에 의존했습니다. 계산이 매우 빠르고 단순했지만, 원심력이 생기는 선회 비행(Coordinated Turn)이나 GPS 신호 간섭이 발생할 때 오차 보상력을 상실한다는 치명적 단점이 존재했습니다.

시간이 흘러 32비트 마이크로컨트롤러 시대(Cortex-M 시리즈)에 돌입하면서, 현대 자율비행 아키텍처는 센서 퓨전의 궁극적 해답인 확률론적 예측 제어, **EKF (Extended Kalman Filter)**를 메인 상태 추정기로 장착하게 되었습니다.

0.3 상태 추정기 모듈의 소프트웨어 아키텍처 인터페이스 (PX4/uORB 기준)

거대한 EKF 모듈이 드론 펌웨어 속에서 구동될 때, 주변 하드웨어 모듈과 어떻게 격리되어 설계되는지 그 입출력 다이어그램(I/O Diagram)을 요약하면 다음과 같습니다.

graph TD;
    subgraph Data Input (Sensors)
        IMU(IMU: 자이로, 가속) --> EKF Module 
        MAG(지자기: 나침반) --> EKF Module
        BARO(기압계) --> EKF Module
        GPS(GPS GNSS) --> EKF Module
        Optical(비전 센서 등) -.-> EKF Module
    end
    
    subgraph State Estimator
        EKF Module[EKF/ECL 필터 엔진:<br>위상 공간(State Vector) 업데이트<br>공분산 행렬 예측 및 보정]
    end
    
    subgraph Data Output (Topics)
        EKF Module -->|Publish: vehicle_attitude| CTRL_ATT[자세 제어기]
        EKF Module -->|Publish: vehicle_local_position| CTRL_POS[위치 제어기]
        EKF Module -->|Publish: estimator_status| MONITOR[GCS/상태 모니터 등]
    end

여기서 상태 추정기 엔진(EKF)의 역할은 단지 데이터 변환기에 머물지 않습니다.
이 모듈은 “센서값이 이러하니 정답은 이거다“라고 확신(Deterministic)하는 대신 “내가 볼 때 드론의 위치는 여기일 확률이 99%이며, 현재 나침반 데이터는 자성 왜곡이 심하여 공분산 오차가 커졌으니 GPS 헤딩 데이터를 더 신뢰하여 퓨전 비율을 조정하겠다“라는 동적인(Dynamic) 알고리즘 체계를 스스로 구동합니다.

PX4의 경우 추정기 모듈 코드를 ECL (Estimator and Control Library) 이라는 수학 라이브러리로 외부에 분리하여 관리하며, 이 필터가 실시간으로 내뿜는 단 하나의 진실된 정답지, 즉 vehicle_attitude 토픽을 믿고 비행 제어기(Flight Controller)의 모터 출력이 결정되게 됩니다.

이어지는 절에서는 이 오케스트라의 지휘자인 EKF (Extended Kalman Filter) 자체의 수학적 운용 사상과 내부 아키텍처 구조를 심층적으로 뜯어 살펴보겠습니다.
