5.2.2.1.1. 모듈 간 강한 결합(Tight Coupling) 회피 및 플러그앤플레이(PnP) 확장성

무인기 비행 소프트웨어 코드를 작성해 보면, 가장 깊은 수렁에 빠지는 단계가 “새로운 센서 알고리즘을 추가할 때 기존 코드 수백 곳을 수정해야 하는 현상“입니다. 앞 절에서 살펴본 uORB 시스템의 가장 거대하고 실천적인 위업은 바로 전역 변수와 하드코딩된 호출을 무찌르고, 시스템 전체에 걸친 강한 결합(Tight Coupling)의 회피를 완벽에 가깝게 구현해 냈다는 사실입니다.

이 장에서는 PX4 아키텍처가 어떻게 모듈러(Modular) 개념을 극한으로 확장하여 하드웨어 부품을 조립하듯 소프트웨어를 조립하는 플러그앤플레이(Plug and Play) 확장이 가능한지 그 공학적 자율 구조의 매력을 살펴봅니다.

0.1 스파게티 코드 방어막: 토픽 중심 인터페이스

과거 혹은 타 아키텍처 관점에서, 드론이 현재의 고도(Altitude)를 알기 위해 메인 컨트롤러 클래스가 직접 기압계.읽기(), GPS.고도계(), Lidar.범위() 함수 포인터들을 인클루드(Include)하고 각각 호출했다면, 이것은 객체 간의 극심한 ‘강한 결합’ 상태입니다. 만약 새로운 벤더(Vendor)의 센서가 이식된다면 메인 컨트롤러 코드의 복장이 터질 수밖에 없습니다.

PX4의 uORB 아키텍처는 이를 완벽히 끊어냅니다. (디커플링; Decoupling)

어떤 센서 드라이버 모듈이든 기저에 깔린 하드웨어 통신과 무관하게, PX4 표준 uORB 토픽인 vehicle_air_data 등 약속된 형태의 C/C++ 구조체 메세지만 뿌려주면 그뿐입니다. 마찬가지로 고도를 해석해야 하는 EKF2 추정기(Estimator) 모듈은 어떤 제조사의 어떤 칩셋 드라이버가 살아있는지 단 1%의 관심도 없으며, 그저 미들웨어인 uORB 큐에 누군가 갱신해 준 vehicle_air_data 토픽의 포장지만 뜯어서 해석하면 끝입니다. 인터페이스의 독립성이 확립된 것입니다.

0.2 블록 조립식 소프트웨어: 플러그앤플레이(PnP) 확장 메커니즘

이러한 느슨한 결합(Loose Coupling)은 PX4의 코드를 마치 레고 블록 단위로 끼웠다가, 에러가 나면 특정 센서 블록 소스코트만 비활성화(Disable) 시키는 동적 확장의 마법을 실현합니다.

graph TD;
    subgraph Modular Hot-Swapping Capability
        A[Old: Lidar Driver] -.->|Disable Publish| C(uORB: distance_sensor)
        B[New: Radar Driver] -->|Publish| C
        C -->|Subscribe| D[Collision Avoidance Module]
    end

드론 스타트업의 개발자가 기본 펌웨어가 안고 있는 충돌 회피 알고리즘이 마음에 들지 않는다고 가정합시다. 이 연구원은 기존 펌웨어의 무거운 위치 관제 클래스 전체 아키텍처를 뒤집을 필요가 없습니다.

  1. 백그라운드 스레드로 실행 중이던 기존 collision_prevention 프로세스를 NuttX 쉘(nsh)에서 중지(stop) 시킵니다.
  2. 본인이 직접 빌드(Build)한 완전히 새롭고 복잡한 인공지능 모듈 my_ai_avoidance 앱을 실행(start)시킵니다.
  3. 이 새로운 앱은 오직 전역 미들웨어인 uORB에 접속해 distance_sensor 구조체를 구독하기만 하면 됩니다.
  4. 결과를 산출하여 vehicle_trajectory_setpoint 토픽을 발행(Publish)하면 모터 콘트롤러가 알아서 움직입니다.

기존 코어를 절대 훼손하지 않고(Non-destructive) 무한 확장이 가능합니다. PX4가 전 세계 모든 대학원의 R&D 테스트베드 1순위로 채택된 이유가 여기에 응축되어 있습니다.

0.3 멀티 인스턴스(Multi-instance)를 통한 이중화(Redundancy) 기본 탑재

더 나아가, 모듈 간 결합도를 끊어놨기에 가능해지는 결정적인 장점이 **다중 센서 무결성(Redundancy)**의 아키텍처적 편안함입니다.

만약 산업용 대형 쿼드콥터에 상호 충돌 감시를 위해 완전히 기종이 다른 3개의 GPS를 설치했다고 해봅시다.
PX4 개발자는 코드를 바꿀 필요가 없습니다. 각각 구동된 3개의 GPS 드라이버 스레드는 똑같이 sensor_gps 토픽을 uORB에 던지지만, 미들웨어 내부적으로 인스턴스(Instance) 속성(ID 0, ID 1, ID 2)만 채번(Sequence)할 뿐입니다.
구독자인 센서 퓨전(Sensor Fusion) 알고리즘은 3개의 인스턴스가 날아옴을 자연스럽게 감지하고, “가장 위성이 많이 잡힌” 놈의 uORB 토픽 배열만 골라서 읽어가는 투표(Voting) 알고리즘 메커니즘을 어떠한 충돌 없이 매끄럽게 수행할 수 있습니다.

이는 하드웨어 생존성이 직결된 드론 항공학 소프트웨어에서 플러그 앤 플레이 방식이 완성시킨 눈부신 견고함입니다.
