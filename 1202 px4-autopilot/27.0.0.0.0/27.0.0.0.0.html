Chapter 27. Estimator(ECL/EKF2) 아키텍처 및 소스 코드 분석

무인 항공기(UAV)가 역동적인 3차원 공간에서 자율 비행을 수행하기 위해서는 기체의 현재 위치, 속도, 자세(Orientation) 및 각종 센서의 오차 상태를 정밀하게 추정하는 과정이 필수적이다. 실세계의 센서 데이터는 본질적으로 노이즈(Noise)를 포함하고 있으며, 단일 센서만으로는 기체의 완전한 6자유도(6-DOF) 상태를 신뢰성 있게 파악할 수 없다.

PX4-Autopilot은 이러한 한계를 극복하기 위해 다중 센서 융합(Multisensor Fusion) 기반의 정교한 상태 추정기(Estimator)를 내장하고 있다. 현재 PX4의 주력 상태 추정기는 ECL(Estimation and Control Library) 이라는 독립된 C++ 라이브러리 형태로 관리되는 EKF2 (Extended Kalman Filter 2) 모듈이다.

본 장에서는 PX4 시스템의 두뇌 역할을 하는 EKF2의 전반적인 아키텍처와 ECL 라이브러리의 소스 코드 구조를 심층적으로 분석한다.

1. 상태 추정기(Estimator)의 역할과 EKF2의 위상

비행 제어 스택에서 상태 추정기는 하드웨어 드라이버 계층과 제어(Control) 계층을 잇는 중추적인 역할을 담당한다. 센서로부터 측정된 데이터(관측값)와 기체의 물리적 동역학 모델(예측값)을 수학적으로 결합하여, 불확실성이 최소화된 ’최상의 상태 추정치(Best Estimate)’를 산출해 낸다.

1.1 EKF2의 주요 입력 및 출력

PX4의 EKF2는 24개 이상의 상태(State) 변수를 추정하는 거대한 확장 칼만 필터(Extended Kalman Filter)이다.

  • 주요 입력(Inputs):
  • 고주파 IMU 데이터: 자이로스코프(각속도), 가속도계 데이터 (보통 250Hz ~ 1kHz)
  • 저주파 측정 데이터: GPS (위치, 속도), 기압계(고도), 나침반/지자기 센서(헤딩), 옵티컬 플로우(Optical Flow), 거리 측정 센서(Range Finder), 비전 기반 포지셔닝(VIO) 데이터 등
  • 주요 출력(Outputs):
  • 3차원 위치(Position: North, East, Down) 및 속도(Velocity)
  • 쿼터니언(Quaternion) 기반의 3차원 자세(Attitude)
  • 센서 바이어스(Bias): 자이로 바이어스, 가속도계 바이어스
  • 지자기장 상태(Earth Magnetic Field) 및 풍속(Wind Speed) 추정치

2. ECL (Estimation and Control Library) 아키텍처 설계 철학

과거 PX4는 LPE(Local Position Estimator)와 구형 EKF 등 여러 추정기를 펌웨어 소스 코드 내부에 강하게 결합하여 사용했으나, 유지보수성과 범용성을 높이기 위해 추정 로직을 PX4 펌웨어와 완전히 분리하는 결정을 내렸다. 그 결과물이 바로 ECL (Estimation and Control Library) 이다.

2.1 펌웨어 독립성 (Firmware Agnostic)

ECL은 일종의 독립적인 수학/물리 엔진이다. PX4뿐만 아니라 다른 운영체제나 보드에서도 쉽게 포팅(Porting)하여 사용할 수 있도록, C++ 표준 라이브러리(STL) 및 PX4의 고유 운영체제(NuttX 등) API에 대한 의존성을 극도로 배제(Decoupling)한 채 순수 C++ 수식 형태로 작성되었다.

graph TD;
    A[Sensors (IMU, GPS, Mag)] -->|uORB Messages| B(PX4 Wrapper Module: ekf2);
    B -->|데이터 구조체 변환 주입| C[ECL Core Library];
    C -->|수학적 필터링 연산| D{칼만 알고리즘 (Predict & Update)};
    D -->|추정 완료된 C++ Struct| C;
    C -->|Output 가져오기| B;
    B -->|vehicle_odometry, vehicle_attitude 토픽 발행| E[Position / Attitude Controllers];

위 아키텍처에서 볼 수 있듯, 펌웨어 소스 트리 내의 src/modules/ekf2 폴더는 단순히 uORB 메시지를 ECL이 이해할 수 있는 순수 C++ 구조체로 변환해 주고, ECL의 연산 결과를 다시 uORB로 포장해 주는 래퍼(Wrapper) 역할만을 수행한다. 실제 복잡한 편미분과 행렬 연산은 src/lib/ecl 디렉토리 내에 위치한 ECL 코어 코드에서 이루어진다.

3. ArduPilot EKF(EKF3)와의 아키텍처 비교

오픈소스 드론 생태계의 양대 산맥인 PX4와 ArduPilot은 모두 확장 칼만 필터(EKF)를 핵심 추정기로 사용하지만, 그 아키텍처와 발전 방향에는 약간의 차이가 존재한다.

  1. 플랫폼 종속성: ArduPilot의 EKF3(현재 기본값)는 ArduPilot의 코어 라이브러리 프레임워크(AP_NavEKF3)에 매우 타이트하게 결합되어 있다. 반면 PX4의 ECL은 git submodule 형태로 관리되는 독립된 모듈로, 타 프로젝트 이식이 더 수월하다는 장점을 지닌다.
  2. 지연 융합(Delayed Fusion) 기법: GPS(약 10Hz)와 IMU(약 1kHz) 등 센서마다 데이터가 도착하는 주파수와 지연 시간(Latency)이 다르다. PX4 EKF2와 ArduPilot EKF3 모두 지연 시간을 보상하기 위해 거대한 관측 버퍼(Observation Buffer) 연대기를 유지하여 과거 시점으로 롤백(Rollback)한 뒤 필터를 융합하는 고도의 지연 융합 기법을 공통적으로 채택하고 있다.
  3. 다중 인스턴스(Multi-Instance) 지원: 최근 양쪽 진영 모두 다중 센서의 신뢰성을 높이기 위해 EKF 인스턴스를 여러 개 띄우고(예: IMU 3개면 EKF 인스턴스를 3개 독립 가동), 각 상태를 평가하여 최적의 코어(Core) 상태를 메인 제어 루프에 제공하는 다중 인스턴스 기술을 지원하고 있다. PX4의 ekf2 모듈 역시 내부적으로 이 기능을 활발히 사용 중이다.

4. 본 장의 구성 체계

본 장은 EKF2의 방대한 행렬 연산과 소스 코드 라인들을 단순히 나열하는 것을 넘어, 제어 공학도와 로보틱스 개발자가 그 내면의 물리적 통찰(Physical Intuition)을 얻을 수 있도록 다음의 체계적인 흐름을 따라 구성되었다.

  1. 수학적 토대 (Mathematical Foundation): 확장 칼만 필터의 핵심인 이산 시간(Discrete-time) 기반의 예측(Predict)과 보정(Update) 방정식, 그리고 비선형성을 처리하기 위한 야코비안(Jacobian) 행렬 도출 원리를 다룬다.
  2. ECL 아키텍처 및 빌드 시스템: ECL 라이브러리가 어떻게 PX4 빌드 시스템과 통합되며, 코어 필터 알고리즘이 펌웨어 환경으로부터 어떻게 이격되어 있는지 코드 구조를 살펴본다.
  3. IMU 동역학 및 오차 모델링: 관성 항법 알고리즘(Strapdown Navigation)을 기반으로 고주파 IMU 데이터가 기체의 자세와 속도/위치로 적분되는 과정과 센서 오차(Bias) 추정 방식을 분석한다.
  4. 다중 센서 융합의 실제: GPS 측정치 결합, 지자기 융합, 옵티컬 플로우(Optical Flow) 및 외부 비전(VIO) 데이터의 지연 융합(Delayed Fusion) 파이프라인 등 센서별 측정 모델의 구현 스크립트를 깊이 있게 해부한다.

이제 이어지는 절들을 통해 PX4의 뇌수에 해당하는 ECL/EKF2의 복잡하지만 우아한 수학적, 소프트웨어적 아키텍처 속으로 들어가 보도록 하자.
