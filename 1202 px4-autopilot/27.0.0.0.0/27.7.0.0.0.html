27.7. 시스템 상태 진단 메시징 및 트러블슈팅

다중 인스턴스(Multi-Instance) EKF 구조를 통해 무결점 비행을 이어나가는 과정은, 내부적으로는 끊임없이 오차를 감시하고 페일오버 스위칭을 단행하는 치열한 사투의 연속이다. 하지만 이 모든 일들이 비행 제어기 내부의 블랙박스(Black-box) 영역에서만 은밀하게 벌어진다면 사용자(Pilot)나 지상 관제 시스템(GCS)은 드론이 언제 추락할지 모르는 깜깜이 상태에 놓이게 된다.

투명한 관제와 사후 트러블슈팅을 보장하기 위해, PX4 ECL 생태계는 자신이 겪고 있는 고통과 결단의 순간들을 uORB 메시지로 철저하게 브로드캐스팅(Broadcasting)한다. 본 절에서는 estimator_status 와 estimator_sensor_bias 토픽을 중심으로 EKF가 뱉어내는 진단 메시징 체계와, 이를 바탕으로 한 QGroundControl 연동 트러블슈팅 기법을 분석한다.

1. estimator_status.msg: EKF의 심전도 모니터

EKF 코어가 자신의 연산 상태를 외부에 알리는 가장 굵직한 통신 파이프라인은 estimator_status uORB 토픽이다. 이 메시지 구조체(Message Structure)는 필터가 얼마나 외부 관측치를 신뢰하고 있는지, 그리고 현재 어느 센서 조합에 결함이 발생했는지를 나타내는 ’건강 점수(Health Score)’들로 가득 차 있다.

1.1 Test Ratios (잔차 검증 비율)

앞서 센서 셀렉터 절에서 다루었듯이, 센서 측정값의 잔차(Innovation)가 허용 한계(Gate)에 얼마나 근접했는지를 [0.0 \sim 1.0] 사이의 비율로 표현한다.

  • vel_test_ratio: 속도 센서 잔차 비율
  • pos_test_ratio: 위치 센서 잔차 비율
  • mag_test_ratio: 나침반 방향 잔차 비율
  • 이 수치 중 하나라도 장시간 1.0을 초과하면 해당 센서는 “버림받은(Rejected)” 것으로 간주되며, QGC 화면에 “EKF GPS fusion timeout” 같은 위협적인 빨간색 경고 메시지가 송출된다.

1.2 Solution Status Flags (해석 상태 비트마스크)

solution_status_flags 필드는 현재 EKF가 어떤 센서의 조력을 받아 기체 상태를 산출하고 있는지를 나타내는 16비트 정수형 비트마스크(Bitmask)다.

  • Bit 0 (CS_TILT_ALIGN): 중력 가속도를 인지하여 롤/피치 수평 자세를 잡았는가?
  • Bit 1 (CS_YAW_ALIGN): 나침반 등을 통해 북쪽 방향(Yaw) 인식이 끝났는가?
  • Bit 2 (CS_GPS): GPS 데이터를 위치 추정에 능동적으로 사용 중인가?
  • 만약 드론이 건물 사이로 들어가 GPS 음영 지역에 놓이면 CS_GPS 비트가 일시적으로 거둬지며(Cleared), 이를 감지한 사령관(Commander) 모듈은 현재의 자동 비행 모드를 해제하고 고도 유지(Altitude/Position) 모드의 락(Lock)을 풀어버린다. 조종사에게 강제로 수동 조종의 책임이 전가되는 순간이다.

2. estimator_sensor_bias.msg: 하드웨어 노화와 결함의 거울

IMU 센서는 출하된 순간부터 칩셋 내부의 뒤틀림과 열팽창에 의해 영점(Zero-point) 오차가 서서히 틀어지기 시작한다. EKF는 이 오차(Bias)마저도 상태 변수 10~15번 인덱스에 넣고 실시간으로 추정(Estimation)해 낸다.

estimator_sensor_bias 토픽은 이렇게 필터가 백그라운드에서 추정해 낸 나침반, 가속도계, 자이로스코프의 바이어스 보정치를 uORB로 퍼블리시한다.

  • gyro_bias[3], accel_bias[3]: 자이로 및 가속도계의 x, y, z축 바이어스
  • 트러블슈팅 관점: 조종자가 이륙 준비를 위해 기체 파워를 켰으나, QGC에 “High Accelerometer Bias” 라는 경고가 뜨며 시동(Arming)이 거부되는 상황이 잦다. 이는 predictState 함수 내부에서 추정된 accel_bias 벡터의 스칼라 크기(Norm)가 허용 한계(EKF2_ABL_LIM)를 초과했기 때문이다.
  • 솔루션: 기체가 물리적으로 바닥에 엎어지거나 심한 충격을 받아 IMU 캘리브레이션 팩터 자체가 박살 난 경우다. 현장에서 즉시 평평한 바닥에 기체를 놓고 “Level Horizon (수평 캘리브레이션)” 을 다시 수행하여 영점을 잡아주어야 EKF가 평온을 되찾는다.

3. 로깅(Logging) 분석: ULog와 Flight Review를 통한 사후 부검

실제 비행 중 드론이 알 수 없는 이유로 롤러코스터 기동을 하거나 경로를 이탈해 추락(Crash)했다면, 기체 내부에 저장된 .ulg (ULog) 파일을 분석하는 사후 부검 과정이 필수적이다.
PX4 커뮤니티가 제공하는 웹 기반 ULog 분석 도구인 Flight Review (logs.px4.io) 에 .ulg 파일을 업로드하면 estimator_status 와 estimator_innovations 토픽들의 시간에 따른 파동을 시각화(Plot)해 준다.

3.1 대표적인 패턴 분석: 자기장 교란 (Magnetic Interference)

  1. 현상: 이륙 직후 기체가 스월링(Swirling)을 하거나 요(Yaw) 방향 제어권을 잃음.
  2. 로그 플롯(Plot) 확인: “EKF Magnetic Innovation” 그래프 관찰.
  3. 원인 진단: 드론의 스로틀(Throttle)을 올려 모터가 거센 전류를 소모할 때마다 X, Y, Z축의 지자기 잔차(Innovation) 수치가 기이하게 요동친다면, 배터리나 ESC 전원선에서 발생하는 거대한 전자기장이 나침반 센서를 교란하고 있는 것이다. EKF는 잔차 한계를 견디다 못해 결국 나침반 융합을 포기(Mag Rejected) 해버린다.
  4. 트러블슈팅: 하드웨어적으로 나침반(GPS 퍽)의 위치를 전원선에서 멀리 띄우거나 차폐(Shielding)해야 하며, 소프트웨어적으로는 “배터리 전류 기반 자기장 보상(Power based mag compensation)” 파라미터를 활성화하여 모터 토크에 비례하는 자기장 간섭을 미리 예상 관측치에서 빼주는 조터(Tuning)를 수행해야 한다.

4. 소결

ECL 기반의 24상태 EKF 아키텍처는 고도의 수학적 성채 안에서 완벽하게 기체를 연산해 내지만, 자신의 고통을 외부에 호소하는 데뷔 창구 역시 치밀하게 마련해 두었다.

비행 제어 엔지니어는 C++ 동역학 코드를 해부하는 해부학자의 역량뿐만 아니라, estimator_status 가 토해내는 로그 지표를 마치 환자의 혈압과 심전도를 읽듯이 해석해 내는 내과의사의 통찰력까지 겸비해야 한다.
