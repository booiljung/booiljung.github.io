38.6.2.1 C++ std::future 및 std::promise 동시성(Concurrency) 모델을 활용한 비동기 콜백(Callback) 이벤트 핸들러 구현 기법

이전 절에서 살펴본 수동적인 모니터링(Listening)을 넘어, 이번에는 GCS가 능동적으로 기체에 명령(Command)을 내리는 상황을 가정해 보자. 이륙 명령(Takeoff)을 내린 후, 기체로부터 COMMAND_ACK (승인) 패킷이 돌아올 때까지 기다렸다가 다음 임무(Waypoint 1)를 지시해야 한다.

불안정한 무선 환경에서 언제 올지 모르는 этот ACK 패킷을 과연 어떻게 기다릴 것인가?

1. 콜백 지옥(Callback Hell)의 늪

비동기 프로그래밍의 가장 원시적인 형태는 모든 명령의 응답 처리를 중첩된 람다(Lambda) 콜백으로 엮는 것이다.

“이륙하라 \rightarrow 성공하면 Waypoint 1로 가라 \rightarrow 성공하면 사진을 찍어라“라는 로직을 콜백으로 짜면, 코드는 오른쪽으로 끝없이 파고드는 소위 콜백 지옥(Callback Hell) 형태의 스파게티 코드가 된다. 가독성은 붕괴하고, 각 콜백 범위(Scope)에서 변수들의 생명 주기(Lifetime)를 관리하는 것은 메모리 누수(Memory Leak)의 온상이 된다.

2. 동시성의 구원자: std::promise와 std::future

MAVSDK 코어 개발진은 이 비동기 제어의 복잡성을 정돈하기 위해 C++11 표준에 도입된 **std::promise**와 **std::future**라는 현대적인 동시성(Concurrency) 디자인 패턴을 적극적으로 채용했다.

이는 하나의 값이 미래의 어느 시점에 생성될 것임을 약속(Promise)하고, 다른 스레드는 그 약속된 미래의 티켓(Future)을 쥐고 기다리는 통신 메커니즘이다.

  1. 계약 생성 (The Contract): MAVSDK의 메인 스레드에서 takeoff_async() 함수를 호출하면, 내부적으로 std::promise<Action::Result> 객체가 생성된다. 이는 “미래에 반드시 이륙 명령의 성공/실패 여부를 담아 주겠다“는 약속이다.
  2. 티켓 발급 및 발송 (Dispatch): 함수는 이 약속으로부터 파생된 std::future<Action::Result> 티켓을 즉각 호출자(메인 스레드)에게 반환하고, 실제 COMMAND_LONG 패킷은 백그라운드 스레드를 통해 드론으로 날려 보낸다.
  3. 선택적 대기 (Suspension): 티켓을 받아 든 메인 스레드는 future.get()을 호출한다. 이 순간 런타임은 이 메인 스레드만을 안전하게 블로킹(수면 상태) 시킨다. 코어 이벤트 루프나 UI 렌더링 스레드는 절대 멈추지 않고 계속 백그라운드에서 돌아가며 들어오는 RF 패킷들을 파싱한다.
  4. 약속 이행 및 각성 (Fulfillment & Awakening): 몇백 밀리초 뒤, 백그라운드 이벤트 루프가 드론으로부터 COMMAND_ACK 패킷을 수신하고 파싱에 성공한다. 백그라운드 스레드는 애타게 기다리던 앞서의 promise 객체에 접근하여 promise.set_value(Action::Result::Success)를 호출하여 값을 채워 넣는다.
  5. 동기적 반환: 값을 채우는 정확히 그 순간, 잠들어 있던 메인 스레드의 future.get()이 깨어나면서 Success 결과를 반환받고 다음 줄의 코드를 실행하기 시작한다.
// MAVSDK 비동기 명령 처리의 동기적(Synchronous-looking) 구현 예시
std::promise<Action::Result> prom;
std::future<Action::Result> fut = prom.get_future();

// 비동기 명령 발송. 람다 콜백 안에서 promise의 값을 세팅한다.
action->takeoff_async([&prom](Action::Result%20result) {
    prom.set_value(result);
});

// 메인 로직은 여기서 우아하게 대기한다. (UI나 백그라운드 파서는 멈추지 않음)
Action::Result takeoff_result = fut.get(); 

if (takeoff_result == Action::Result::Success) {
    std::cout << "이륙 성공! 다음 임무를 시작합니다." << std::endl;
}

3. 아키텍처적 의의

std::future/promise 패턴의 채용은 지저분하고 예측 불가능한 RF MAVLink 비동기 통신의 복잡성(Asynchrony)을, 완전히 동기적이고 깔끔해 보이는 비즈니스 로직(Synchronous-looking Business Logic)으로 추상화(Abstraction) 해내는 마법을 부린다.

이 구현 기법 덕분에 GCS 개발자는 수십 단계의 복잡한 비행 임무를 C++ 코드상에서 절차적(Sequential)으로 안전하게 나열할 수 있으며, 어떠한 통신 지연이 발생하더라도 CPU나 코어 파이프라인의 명줄(Thread)을 끊어먹는 일 없이 우아하게 스플릿 브레인 방어 기제를 동작시킬 수 있게 된 것이다.
