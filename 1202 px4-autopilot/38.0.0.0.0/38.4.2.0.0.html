38.4.2 적응형 타임아웃(Adaptive Timeout) 결정 알고리즘

무선 텔레메트리 환경에서 “통신이 끊어졌다“고 선언(Declare)하는 시점인 타임아웃(Timeout)을 어떻게 설정할 것인가는 무인기 제어 공학의 오랜 딜레마다.

과거의 단순한 GCS나 초창기 비행 제어기들은 정적 타임아웃(Static Timeout), 즉 파라미터로 하드코딩된 고정 상수(예: 3초)를 사용했다. 마지막 HEARTBEAT 수신 후 일률적으로 3초가 지나면 무조건 페일세이프(Failsafe)를 발동시키는 식이다.

1. 정적 타임아웃(Static Timeout)의 구조적 모순

정적 타임아웃은 RF 환경의 변동성(Volatility)을 전혀 반영하지 못하는 치명적인 약점을 지닌다.

  • False Positive (오탐지): 도심지 비행 중 간섭이 심해져 패킷 도달 간격이 평균 2.5초 이상으로 벌어지고 지터(Jitter)가 심한 상황을 가정해 보자. 통신 자체는 끊기지 않고 끈질기게 살아있음에도, 시스템은 3초라는 고정 임계치에 자꾸 걸려 불필요한 RTL(Return to Launch)을 반복하게 된다.
  • False Negative (미탐지/지연 탐지): 반대로 가시권 내 맑은 하늘에서 100ms 간격으로 칼같이 통신이 이루어지던 완벽한 상태에서 갑자기 전원이 나가 진짜 추락이 발생했다고 하자. 시스템은 통신망의 원래 상태가 얼마나 훌륭했는지는 망각한 채, 멍하니 3초를 채울 때까지 비상 복구 시퀀스 시작을 지연시킨다.

2. PX4 생태계의 적응형 타임아웃(Adaptive Timeout) 철학

이러한 모순을 해소하기 위해 현대적인 네트워크/관제 프로토콜은 기체의 현재 통신 ’날씨(Weather)’를 스스로 학습하여 임계치를 유동적으로 조절하는 적응형 타임아웃(Adaptive Timeout) 알고리즘을 도입한다.

PX4의 커맨더(commander.cpp) 내 Data Link Loss(DLL) 방어 로직이나 고도화된 GCS 통신 스레드는 HEARTBEAT 메시지가 도착할 때마다 단순히 “생존했다“고 체크하는 데 그치지 않고, 도착한 패킷 간의 시간 간격(Arrival Interval, \Delta t) 시계열 데이터를 수학적으로 분석한다.

  1. 시간 간격 측정: n번째 HEARTBEAT 도착 시간 T_n을 기록하고, 순간 도착 간격 I_n = T_n - T_{n-1}을 계산한다.
  2. 동적 지연 추정: 이전 데이터들의 가중 평균을 내어 ’현재 네트워크가 평균적으로 겪고 있는 지연 시간(Smoothed Interval)’을 추정한다. (38.4.2.1절 참고)
  3. 분산(Jitter) 평가: 도착 시간들이 얼마나 들쭉날쭉한지 그 분산(Variance)을 계산한다. (38.4.2.2절 참고)
  4. 유동적 임계치 산출: 최종 타임아웃 임계치 T_{threshold}는 다음과 같이 실시간으로 재계산된다.
    T_{threshold} = \text{Smoothed\_Interval} + k \times \text{Variance} + \text{Base\_Margin}

Ardupilot과의 정책적 차이

이 지점에서 두 펌웨어 간의 철학적 차이가 살짝 엿보인다. Ardupilot은 전통성의 관점에서 운용자의 파라미터 스팅을 매우 신뢰한다. FS_GCS_ENABL 파라미터 활성화 시, GCS 페일세이프 타임아웃은 FS_GCS_TIMEOUT 파라미터(기본값 5초 등)에 의해 대체로 명확하고 보수적인 정적 타임아웃 정책에 크게 의존하는 경향을 보인다. 이는 예상치 못한 알고리즘의 오작동보다는, 예측 가능하고 묵직한 하드코딩 임계치가 현장에서는 더 안전하다는 믿음에 기반한다.

반면, 컴퓨터 비전이나 ROS2 군집(Swarm) 비행 등 비가시권 자율 에이전트 환경을 강하게 의식하는 PX4와 그 MAVSDK/QGC 생태계는, 복잡하고 잡음이 많은 무선망을 극복하기 위해 제어 이론(Control Theory)을 차용한 관측기 기반의 순응적/통계적(Adaptive/Stochastic) 한계선 설정을 더 적극적으로 수용하는 아키텍처적 유연성을 보여준다.
