38.2 MAVLink HEARTBEAT 데이터 링크 계층의 통신 이론적 해석

MAVLink(Micro Air Vehicle Link) 프로토콜은 OSI 7계층 모델 관점에서 주로 응용 계층(Application Layer)에 속하는 경량화된 메시징 스펙이다. 그러나 실질적인 드론 관제 시스템(GCS)과 기체 간의 네트워크 토폴로지(Topology)를 구축하고 연결의 생존성(Liveliness)을 검증하는 맥락에서 볼 때, HEARTBEAT 메시지(Message ID #0)는 데이터 링크 계층(Data Link Layer)의 연결 관리 프로토콜(예: 이더넷의 Link Pulse)과 유사한 통신 이론적 지위를 갖는다.

본 절에서는 단순한 “살아있음“의 알림을 넘어, 이 단일 패킷이 드론 네트워크의 형상 관리와 논리적 상태 동기화를 어떻게 주도하는지 분석한다.

1. HEARTBEAT 메시지의 위상과 기능적 정의

PX4-Autopilot 기반의 모든 MAVLink 노드(기체, GCS, 보조 컴퓨터, 카메라 페이로드 등)는 네트워크에 진입하는 순간부터 통상 1Hz(초당 1회)의 주기로 HEARTBEAT 메시지를 브로드캐스트(Broadcast)한다. 이 패킷은 크기가 불과 9바이트(페이로드 기준)에 불과하지만, 통신망에 참여하는 개체의 본질적인 메타데이터(Metadata)를 모두 함축하고 있다.

HEARTBEAT 페이로드에 포함되는 핵심 정보는 다음과 같다.

  • 기체 유형 (Type): 멀티로터, 고정익, VTOL, 지상 차량(Rover) 등 하드웨어의 물리적 뼈대.
  • 오토파일럿 유형 (Autopilot): PX4, Ardupilot, 원격 조종기 등 플라이트 스택의 종류.
  • 기본 모드 플래그 (Base Mode): 시스템의 시동 상태(Armed/Disarmed) 및 하드웨어 인 더 루프(HIL) 모드, 커스텀 모드 활성화 여부를 나타내는 비트마스크.
  • 커스텀 모드 (Custom Mode): 각 오토파일럿 고유의 세부 비행 모드(예: PX4의 Mission, Position 등).
  • 시스템 상태 (System Status): 부팅 중, 보정(Calibration) 중, 활성(Active), 치명적 에러(Critical) 등의 전반적인 헬스(Health) 상태.

이러한 주기적 브로드캐스트는 통신 이론의 ‘상태 비저장(Stateless)’ 모델과 ‘연결 지향적(Connection-oriented)’ 모델 사이의 타협점이다. MAVLink 토폴로지 자체는 엄격한 핸드셰이크(e.g., TCP SYN/ACK)를 요구하지 않는 UDP/시리얼 기반의 비연결성(Connectionless) 망이지만, 수신 측(GCS) 서버는 주기적으로 도착하는 HEARTBEAT를 청취하여 가상의 논리적 세션(Session)을 구성하고 유지한다.

2. 신뢰성 없는 채널(Unreliable Channel) 생존성 모델 예측

통신 공학적 관점에서 HEARTBEAT 메시지 자체는 데이터 전송의 신뢰성(Reliability)을 전혀 보장하지 않는다. 즉, 기체가 송신한 패킷이 노이즈로 인해 유실되더라도 GCS는 재전송(Retransmission)을 요구(NACK)하지 않는다.

대신 상태 감시 로직은 ’추계학적 확률 모델(Stochastic Model)’에 의존하여 링크의 건전성을 평가한다. 만약 1Hz로 들어와야 할 패킷이 설정된 타임아웃(예: 3초, 5초) 동안 연속으로 누락된다면, 관제 시스템은 패킷 손실률(Packet Loss Rate)이 임계치를 초과했다고 판단하여 ‘통신 단절(Data Link Loss)’ 페일세이프(Failsafe) 시퀀스를 발동시킨다. 이는 복잡한 TCP/IP 스택을 구현할 여력이 없는 마이크로컨트롤러 환경에서 최소한의 오버헤드로 네트워크 단절을 조기에 감지하기 위한 최적화된 설계다.

3. PX4와 Ardupilot의 커스텀 모드(Custom Mode) 파싱(Parsing) 아키텍처 비교

두 오픈소스 시스템 모두 동일한 MAVLink HEARTBEAT 패킷 규격을 청취하지만, 그 안에 담긴 4바이트 길이의 custom_mode 필드를 해석(Decoding)하는 방식은 아키텍처적으로 큰 차이를 보인다.

  • Ardupilot의 평면형(Flat) 열거형 모델: Ardupilot(예: ArduCopter)은 기체 유형별로 독립적인 1차원 숫자 배열을 갖는다. (Stabilize = 0, Acro = 1, AltHold = 2, Auto = 3 등). GCS는 기체 유형(Type)을 확인한 후, 해당 숫자를 단순한 Switch-Case 문으로 1:1 매칭하여 비행 모드를 식별한다.
  • PX4-Autopilot의 계층형(Hierarchical) 비트마스크 모델: PX4는 4바이트 공간을 분할하여 메인 모드(Main Mode)와 서브 모드(Sub Mode)의 2차원 매트릭스로 활용한다. 상위 바이트에는 시스템의 포괄적 상태(예: 자동 비행, 수동 제어)를 담고, 하위 1바이트에는 세부 동작(예: 자동 비행 중 ‘이륙’, ‘임무’, ‘복귀’)을 담는 구조체 래핑(Struct Wrapping) 방식을 채택했다.

이러한 PX4의 계층적 디코딩 방식은 이어지는 38.2.3절의 ’비트 단위 논리 연산 메커니즘’에서 다루어질 데이터 병렬 처리 및 비트 마스킹(Bit Masking) 기법과 직접적으로 연관된다. 이어지는 하위 절들에서는 HEARTBEAT 패킷이 하드웨어 인터페이스를 거치며 겪는 메모리 정렬 문제와 무결성 검증, 그리고 대규모 군집 제어에서의 의미를 심도 있게 통신망 단위에서 해부한다.
