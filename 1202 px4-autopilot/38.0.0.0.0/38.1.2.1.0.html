38.1.2.1 실시간 운영체제(RTOS)의 선점형 스케줄링(Preemptive Scheduling) 환경 내 문맥 교환(Context Switching) 오버헤드 분석

PX4-Autopilot의 근간을 이루는 NuttX RTOS는 엄격한 실시간성(Real-time Constraints)을 만족시키기 위해 선점형 스케줄링(Preemptive Scheduling)을 채택하고 있다. 이는 현재 실행 중인 스레드보다 높은 우선순위를 가진 스레드가 대기열(Ready Queue)에 진입하거나 인터럽트가 발생하면, OS 커널이 즉시 현재 스레드의 실행을 강제 중단시키고 제어권을 넘기는 방식이다. 이러한 다중 스레드 환경에서 가장 주의해야 할 제어 공학적, 컴퓨터 구조적 취약점은 바로 문맥 교환(Context Switching)으로 인한 시간 지연 오버헤드(Overhead)다.

1. 문맥 교환(Context Switch)의 물리적 비용 구조

Pixhawk 시리즈와 같은 STM32 칩셋(ARM Cortex-M 기반) 환경에서 하나의 NuttX Task(스레드)가 선점 당하고 다른 Task로 제어권이 넘어갈 때, 하드웨어 계층에서는 다음과 같은 비용이 발생한다.

  1. 레지스터 스태킹(Register Stacking): 현재 실행 중인 스레드의 CPU 범용 레지스터, 프로그램 카운터(PC), 상태 레지스터(xPSR)를 해당 스레드의 메모리 스택에 대피(Push)시킨다.
  2. FPU 레지스터(부동 소수점) 저장 (선택적): 만약 해당 스레드가 부동 소수점 연산을 수행 중이었다면, FPU 레지스터 컨텍스트까지 추가로 백업해야 하므로 오버헤드는 배가된다.
  3. 스케줄러 연산: RTOS 커널이 Ready Queue를 탐색하여 최상위 우선순위 Task를 결정한다.
  4. 레지스터 언스태킹(Unstacking): 새롭게 스케줄링된 스레드의 저장된 컨텍스트를 스택에서 CPU 레지스터로 복원(Pop)한다.

이러한 일련의 과정은 통상 수 마이크로초(\mu s)밖에 걸리지 않지만, 초당 수천 번의 uORB 메시지가 발행되는 PX4 아키텍처 특성상 설계가 잘못될 경우 전체 CPU 리소스의 상당 부분을 이 ’스위칭 오버헤드’에 낭비하는 스래싱(Thrashing)이 발생할 수 있다.

2. Commander 모듈의 오버헤드 최소화 기법: 일괄 처리(Batching)와 폴링(Polling)

이론적으로, 특정 센서 값이 상태 전이의 임계치를 넘을 때마다 commander 스레드를 즉시 깨워(Wake-up) 처리하게 만든다면, 하루에도 수만 번의 불필요한 문맥 교환이 일어날 것이다. PX4는 상태 결정론뿐만 아니라 вычисли적(Computational) 효율성을 확보하기 위해 극도의 최적화 패턴을 도입했다.

1. 대기 상태(Sleep)로 CPU 이양(Yield)
commander는 50Hz(기본값)의 일정한 주기로 실행되는 메인 루프를 가지며, 자신의 턴(Turn)이 끝나면 즉시 px4_usleep()이나 poll() 대기 상태로 전환하여 하위 우선순위 스레드들에게 CPU 제어권을 자발적으로 양보한다. 이는 폴링(Polling) 오버헤드(Busy-waiting)를 제거하여 문맥 교환의 빈도를 극적으로 낮춘다.

2. uORB 메시지 큐의 일괄 처리(Batch Processing)
commander가 한 번의 문맥 교환을 통해 CPU를 할당받아 활성화(Wake-up)되면, 그동안 uORB 버스에 누적된 여러 통신 메시지(RC 상태, 배터리 상태, 시스템 파워 상태 등)를 큐에서 한 번에 묶어서(Batch) 모두 페치(Fetch) 및 평가(Evaluate)한다. 즉, 1번의 스위칭 비용을 지불하고 N개의 인터럽트 요인을 일괄 해소함으로써, 오버헤드를 O(N)에서 사실상 O(1)에 가깝게 상각(Amortization)한다.

3. Ardupilot(ChibiOS)과의 구조적 비교

Ardupilot은 과거 NuttX 기반 버전도 지원하였으나, 현재는 훨씬 가벼운 ChibiOS 기반으로 이주(Migration)하여 오버헤드를 최적화했다. Ardupilot의 스케줄러 메인 루프 모듈인 AP_Scheduler는 여러 기능 블록(태스크)을 등록해 두고, 마치 협력형 다중 작업(Cooperative Multitasking)처럼 하나의 큰 메인 스레드 안에서 여러 논리적 태스크들을 순차적으로 실행(Dispatch)하는 방식을 병행한다(물론 I/O 등은 별도 스레드를 사용). 이 방식은 프로세스 자체의 문맥 교환 횟수를 물리적으로 극소화하는 성능적 이점을 갖는다.

반면, PX4-Autopilot의 순수 마이크로 서비스 모델은 구조적 캡슐화와 시스템 투명성 면에서 탁월하지만, 각 모듈이 독립된 태스크이므로 Ardupilot 대비 OS 커널 단에서의 문맥 교환 빈도는 상대적으로 더 잦을 수밖에 없다. 따라서 PX4에서 commander 모듈의 개발 및 최적화(예: px4_poll 구조체의 최적 체인화)는 이러한 마이크로 서비스 아키텍처의 물리적 패널티를 제어 소프트웨어 설계 패턴으로 극복해낸 우수한 공학적 성과라 할 수 있다.
