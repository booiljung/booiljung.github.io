38.5.3 폐루프(Closed-loop) 상태 재동기화(Resynchronization) 프로토콜 설계

앞선 38.5.2절까지는 운용자가 직접 능동적인 제어(명령)를 시도할 때 일어나는 예외 처리와 핸드셰이크를 다루었다. 하지만 더 흔하고 치명적인 불일치(Mismatch)는 기체가 스스로(Autonomously) 상태를 변경했을 때 발생한다.

예를 들어, 통신이 10초간 두절된 사이 배터리가 임계치(Critical) 밑으로 떨어졌다고 가정하자. 비행 제어기(PX4 Commander)는 생존을 위해 스스로 배터리 페일세이프(RTL 또는 Land)를 발동시키고 모드를 변경한다. 10초 뒤 통신이 기적적으로 복구되었을 때, GCS는 여전히 과거의 환영인 Mission 모드를 화면에 띄우고 있을 것이며, 기체 내부의 절대적 진실(True State)과 심각하게 어긋나게 된다.

1. 개루프(Open-loop) 수동 관측의 한계

초창기 단순한 텔레메트리 뷰어 수준의 GCS 프로그램들은 이 상황을 개루프(Open-loop) 방식으로 처리했다. 단순히 수신 대기 (Listening)만 하다가, 통신 복구 후 처음 날아온 HEARTBEAT 패킷에 Return 모드로 적혀 있으면 “아, 모드가 바뀌었구나” 라며 수동적으로 텍스트만 업데이트하는 식이다.

이 방식은 전파 환경이 좋을 때는 문제가 없지만 극한의 노이즈 상황에서는 끔찍하게 위험하다. 통신망 복구 직후 수신된 단 한 개의 패킷은 CRC 에러를 간신히 통과한 비트 플립(Bit Flip) 점철된 쓰레기 데이터일 수도 있고, 하필 버퍼 깊숙이 10초 전 과거에 쌓여있던 지연 패킷(Stale Packet)이 튀어나온 것일 수도 있다. 단 하나의 텔레메트리를 맹목적으로 믿고 UI를 뜯어고치는 것은 시스템의 신뢰성을 근본에서부터 무너뜨린다.

2. 폐루프(Closed-loop) 재동기화 아키텍처

따라서 QGroundControl이나 튼튼하게 설계된 MAVSDK 백엔드는 이 불일치 감지 순간을 다루기 위해, 단순 관측을 넘어선 폐루프(Closed-loop) 재동기화(Resynchronization) 프로토콜을 적극 가동시킨다.

  1. 불일치 이벤트 트리거(Mismatch Detection): 백그라운드 데몬이 캐싱된 (Mode_GCS)와 수신된 패킷의 (Mode_UAV) 사이에서 불일치를 감지한다 (또는 MAVLink 시퀀스 번호의 거대한 끊어짐을 감지한다).
  2. UI 동결 및 제어 락(Freeze and Lock): GCS는 기존의 UI 상태를 유지하는 것을 포기하지만, 곧바로 새 상태를 화면에 확정 짓지도 않는다. 대신 화면을 “동기화 중(Re-syncing)” 또는 “추적 중(Acquiring)” 상태로 회색 처리(Disable)하여, 운용자가 오판으로 인한 긴급 명령을 충동적으로 클릭하는 것을 하드웨어적으로 차단한다.
  3. 능동적 질의(Active Interrogation): GCS는 수동적인 청취자(Listener)의 태도를 버리고, 기체를 향해 MAV_CMD_REQUEST_MESSAGE 기반의 질의 패킷을 상향링크로 쏘아 보낸다. “정규 1Hz 주기를 기다리지 말고, 당장 지금 너의 최신 EXTENDED_SYS_STATE를 강제로 내놓아라!” 라고 능동적으로 정보를 캐내는 것이다.
  4. 디바운싱 검증 (Debouncing Verification): 기체로부터 특별히 요청된 신선한 응답이 도착하고, 뒤이어 오는 2~3개의 후속 HEARTBEAT들이 일관되게 모두 똑같은 새로운 상태(예: Return)를 가리킬 때 비로소 진실성을 확인(Debounce)한다.
  5. 락 해제 및 반영: 이 견고한 질의-응답 검증의 루프가 성공적으로 닫히면(Closed), 그제야 GCS는 회색 동결을 해제하고 UI를 기체의 최신 진성 상태와 영구적으로 결속시킨다.

이 폐루프 프로토콜 모델을 구축함으로써, GCS는 단순히 ’들어오는 데이터를 보여주는 뷰어’의 한계를 벗어나, 무인기와 능동적으로 대화하며 혼돈(통신 단절) 속에서 질서(일관성)를 찾아내는 **지능형 관제 파트너(Intelligent Partner)**로 진화하게 된다.
