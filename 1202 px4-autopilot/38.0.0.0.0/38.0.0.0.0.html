Chapter 38. 기체 상태 머신(State Machine) 동기화 및 Heartbeat 모니터링

현대의 무인 항공기 시스템(UAS, Unmanned Aircraft System)은 단순히 공중을 비행하는 비행체를 넘어, 지상 관제 시스템(GCS, Ground Control Station)과 끊임없이 상태 정보를 교환하며 임무를 수행하는 복잡한 분산 제어 시스템(Distributed Control System)으로 발전하였다. 이러한 분산 시스템 모델에서 기체와 GCS 간의 상태 불일치(State Mismatch)는 치명적인 사고로 이어질 수 있으므로, 두 노드 간의 철저한 기체 상태 머신(State Machine) 동기화(Synchronization)는 전체 시스템의 안전성과 신뢰성을 보장하는 핵심 과제가 된다.

본 장에서는 PX4-Autopilot(기준 펌웨어 버전: v1.14 이상)과 QGroundControl(기준 버전: v4.3 이상) 환경을 바탕으로 기체 상태 머신의 동기화 메커니즘과 그 기반이 되는 HEARTBEAT 데이터 스트림의 모니터링 로직을 심층적으로 분석한다.

1. 분산 시스템 관점에서의 상태 동기화 개요

PX4-Autopilot의 내부 아키텍처는 비행 모드, 무장(Arming) 상태, 내비게이션 상태 등을 관리하는 커맨더(Commander) 모듈을 중심으로 거대한 유한 상태 머신(FSM, Finite State Machine)을 형성한다. 기체가 성공적으로 임무를 수행하기 위해서는 GCS가 기체의 내부 상태(Digital Twin)를 정확히 추정하고 유지해야 한다. 이를 위해 PX4는 MAVLink 통신 프로토콜을 활용하여 주기적으로 상태 정보를 브로드캐스팅(Broadcasting)한다.

1.1 상태 머신 동기화의 본질적인 어려움

비행 중인 기체와 지상 관제소 간의 통신은 무선 주파수(RF) 데이터 링크를 통해 이루어지므로, 필연적으로 대역폭의 제약, 지연 시간(Latency), 그리고 패킷 손실(Packet Loss)이 발생한다. 따라서 GCS는 항상 과거의 기체 상태를 관측하게 되며, 제어 명령을 내릴 때 기체의 현재 상태와 명령을 수신할 시점의 예상 상태를 모두 고려해야 하는 비동기 제어(Asynchronous Control)의 딜레마에 직면하게 된다.

이를 해결하기 위해 시스템은 다음과 같은 메커니즘을 요구한다.

  1. 주기적인 하트비트(Heartbeat) 발송: 통신 링크의 건전성(Connection Health)을 입증하고 시스템의 현재 기저 상태를 지속적으로 알림.
  2. 이벤트 기반 상태 변경 통보: 상태 변화가 발생한 즉시 최우선 순위로 이벤트를 전송하여 GCS 모델의 지연을 최소화함.
  3. 분산 합의 및 교정: 관측된 상태와 기대 상태 간의 불일치가 탐지되었을 때, 이를 자동으로 교정하거나 안전 장치(Failsafe)를 가동하는 메커니즘.

2. HEARTBEAT 메시지의 역할과 중요성

MAVLink 프로토콜의 첫 번째 메시지(#0 HEARTBEAT)는 네트워크상에 존재하는 모든 노드(기체, GCS, 페이로드 등)의 존재(Presence)와 기본 상태를 알리는 심장 박동과 같은 역할을 수행한다. 대체로 1Hz의 고정된 주기로 송출되는 이 메시지는 다음과 같은 핵심 정보를 포함한다.

  • 기체 유형(Type): 멀티로터, 고정익, VTOL, 지상 차량 등 시스템의 물리적 형태.
  • 오토파일럿(Autopilot): PX4, Ardupilot 등 비행 제어 소프트웨어의 종류.
  • 기본/커스텀 비행 모드(Base/Custom Mode): 시스템의 무장 여부 및 현재 활성화된 비행 제어 모드.
  • 시스템 상태(System Status): 부팅 중, 캘리브레이션 중, 활성화(Active), 치명적 에러(Critical) 등 기체의 건전성(Health) 상태.

HEARTBEAT 메시지는 단순히 기체가 “살아 있음(Alive)“을 증명하는 것을 넘어, 위와 같은 기저 상태 변수를 단일 패킷으로 압축하여 전달함으로써 GCS가 디지털 트윈(Digital Twin) 모델을 구축하고 유지할 수 있는 가장 신뢰성 있는 정보원(Source of Truth)을 제공한다.

3. PX4-Autopilot과 Ardupilot의 상태 파악 및 관제 철학 차이

관제 시스템 관점에서 PX4-Autopilot과 Ardupilot은 상태 전이와 모니터링 로직에서 상이한 철학을 지닌다.

  1. 상태 전이의 엄격성(Strictness of State Transitions)
  • PX4-Autopilot: 상태 전이에 매우 엄격하고 결정론적(Deterministic)이다. commander 모듈은 상태 전이의 유효성을 실시간으로 엄격하게 검사하며, 전제 조건(Pre-condition)이 완벽히 충족되지 않는 상태 변경 요청은 즉시 거부(Reject)한다. 관제 시스템(QGC) 역시 이러한 PX4의 성향에 맞추어, 기체가 특정 명령을 수행할 수 없는 상태라면 UI 상에서 아예 해당 명령 버튼을 비활성화(Disable)하는 등 방어적 디자인(Defensive Design)을 채택하고 있다.
  • Ardupilot: 상대적으로 상태 전이에 유연함을 보인다. 예외적인 상황이나 부분적인 센서 에러가 발생하더라도 가능한 범위 내에서 비행을 유지하거나 조종사의 명령을 수용하려는 경향(Conditional Acceptance)이 있다. 따라서 GCS(예: Mission Planner)는 기체의 상태를 모니터링하면서도 사용자의 의도를 최대한 반영할 수 있도록 설계되어 있다.
  1. 단일 진실 공급원(SSoT, Single Source of Truth)
  • PX4는 MAVLink 브로드캐스트 자체가 유일한 진실의 원천으로 취급되도록 강제한다. GCS는 기체로부터 명시적인 상태 변경(ACK 또는 HEARTBEAT 갱신)을 수신하기 전까지는 내부 상태 모델을 임의로 변경하지 못한다. 반면, 일부 커스텀 관제 시스템이나 연구용 구현체에서는 응답성(Responsiveness) 향상을 위해 명령 송신과 동시에 낙관적UI(Optimistic UI) 업데이트를 수행하기도 하는데, 이는 네트워크 지연 환경에서 심각한 상태 역전(State Reversal) 현상을 초래할 수 있으므로 지양해야 한다.

본 장의 이어지는 절에서는 이러한 상태 머신 동기화와 HEARTBEAT 모니터링이 실제 소스 코드 레벨에서 어떻게 구현되어 있으며, 통신 이론적, 제어 공학적인 관점에서 어떠한 의미를 갖는지 다단계로 깊이 있게 분석할 것이다.
