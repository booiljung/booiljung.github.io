38.6.4 QGroundControl 프론트엔드(Frontend) 상태 반영 최적화 기술

앞선 38.6.3절의 C++ 백엔드에서, Vehicle 객체는 MAVLink 패킷 파싱을 마치고 자신의 멤버 변수(예: _flightMode)를 “Hold“에서 “Return“으로 무사히 덮어쓰는 데 성공했다 (Layer 4).

이제 마지막 남은 과제는 이 깊은 곳의 C++ 메모리 변수값이 변경되었다는 사실을, 화면 가장 바깥에 있는 화려한 UI(프론트엔드, Layer 5)에 어떻게 가장 빠르면서도 안전하게 전달하여 글자를 “Return“으로 바꾸고 색상을 붉은색으로 칠할 것인가이다.

1. UI 폴링(Polling)의 전력 낭비와 렌더링 재앙

태블릿이나 스마트폰 같은 모바일 디바이스에서 가장 피해야 할 아키텍처는 프론트엔드(QML/JavaScript)가 주도권을 쥐고 백엔드 변수값을 초당 60프레임(60FPS)으로 계속해서 폴링(Polling)하며 묻는 방식이다.

만약 수백 개의 텔레메트리 변수(자세, 고도, 배터리 전류, GPS 위성 수 등)를 화면 렌더링 스레드가 매 프레임 C++ 함수(Vehicle->getFlightMode())를 호출해 가져온다면, CPU의 캐시 미스(Cache Miss)가 폭주하고 배터리는 순식간에 방전될 것이다. 더 큰 문제는 상태가 변하지 않았음에도 화면을 다시 그리는 불필요한 렌더링 비용이 발생한다는 점이다.

2. Qt Q_PROPERTY 기반의 옵저버(Observer) 패턴

QGroundControl은 프론트엔드와 백엔드를 완전히 분절(Decoupling)시키기 위해 소프트웨어 공학의 정수인 **옵저버 패턴(Observer Pattern)**을 Qt 프레임워크의 고유 기능인 Q_PROPERTY 시스템을 통해 우아하게 구현해 냈다.

백엔드의 Vehicle.h 헤더 파일을 열어보면 수많은 프로퍼티(Property) 선언을 볼 수 있다.

class Vehicle : public QObject {
    Q_OBJECT
    // ...
    Q_PROPERTY(QString flightMode READ flightMode WRITE setFlightMode NOTIFY flightModeChanged)
    // ...
};

가장 핵심적인 키워드는 맨 뒤의 **NOTIFY flightModeChanged**이다. 이는 C++ 영역에서 “비행 모드 값이 변경될 때마다, 내가 직접 flightModeChanged()라는 방송(Signal)을 송출하겠다“는 약속이다.

프론트엔드의 QML 파일에서는 백엔드를 귀찮게 폴링하지 않는다. 그저 이 프로퍼티에 선을 꼽아두기(Data Binding)만 하면 된다.

// QGC UI 프론트엔드 코드 예시
Text {
    text: _activeVehicle.flightMode  // 데이터 바인딩!
    color: _activeVehicle.flightMode === "Return" ? "red" : "white"
}

3. 진정한 이벤트 주도형(Event-Driven) 렌더링 메커니즘

  1. 백엔드 C++ 스레드가 텔레메트리를 감지하고 상태를 업데이트한다.
  2. 상태가 정말로 변경되었을 때만 C++ 본체에서 emit flightModeChanged(_flightMode); 라는 권총 방아쇠(Signal)를 당긴다.
  3. 이때 비로소 옵저버(구독자) 역할을 하던 QML 렌더링 엔진 내부의 깊은 곳이 깨어난다.
  4. QML 엔진은 “아, flightMode 변수를 바라보고 있던 UI 요소들을 전부 다시 그려야겠구나!“라고 판단하고, 아주 정밀하게 해당 텍스트 블록의 데이터만 다시 가져와 화면의 극히 일부분만 즉시 다시 칠한다.

이러한 단방향 데이터 플로우(Unidirectional Data Flow)와 시그널/데이터 바인딩 조합은, 백엔드가 수만 번의 연산을 하든 말든 상관없이 오직 “실제로 진실(State)이 바뀌는 단 한 순간“에만 프론트엔드 연산력을 소모하게 만드는 극강의 상태 반영 최적화 기술이다.

물론 백엔드(네트워크 스레드)와 프론트엔드(메인 GUI 스레드)가 전혀 다른 스레드인데 시그널이 어떻게 충돌 없이 넘어가는가에 대한 비밀인 ‘메타 객체 시스템(Meta-Object System)’, 이른바 병목 큐잉(Queuing)의 묘수는 다음 절에서 상세히 다룬다.
