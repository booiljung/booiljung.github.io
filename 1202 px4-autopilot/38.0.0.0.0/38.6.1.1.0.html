38.6.1.1 uORB 토픽(Topic) 구독 루프의 폴(Poll) 시스템 콜(System Call) 기반 논블로킹(Non-blocking) I/O 처리 메커니즘

이전 절에서 mavlink 모듈이 commander가 뱉어내는 vehicle_status uORB 토픽을 구독(Subscribe)한다고 설명했다. 그렇다면 C++ 코드 레벨에서 이 구독 행위는 정확히 어떻게 CPU 자원을 낭비하지 않고 비동기적으로 이루어질까?

1. 바쁜 대기(Busy-Waiting)의 치명적 결함

가장 하급의 프로그래밍 기법은 while(true) 루프 안에서 끊임없이 “새 데이터가 왔는가?“를 묻는 바쁜 대기(Busy-Waiting) 방식이다. 만약 MAVLink 스레드가 이 방식을 사용한다면, Pixhawk의 제한된 ARM Cortex-M 코어 CPU 점유율을 100% 갉아먹게 될 것이며, 이로 인해 수백 Hz로 핑핑 돌아가야 할 자세 제어(Attitude Control) 루프가 기아 상태(Starvation)에 빠져 기체가 추락할 것이다.

NuttX와 같이 엄격한 실시간 운영체제(RTOS) 환경에서는, 할 일이 없는 스레드는OS 스케줄러 레벨에서 즉시 잠들어(Sleep) CPU를 반납해야만 한다.

2. POSIX poll() 기반의 이벤트 주도형(Event-Driven) 대기

PX4의 uORB 아키텍처는 NuttX의 POSIX 호환 가상 파일 시스템(VFS, Virtual File System)과 깊숙하게 통합되어 있다. 놀랍게도 uORB 토픽을 구독(orb_subscribe)하는 행위는 리눅스에서 텍스트 파일을 여는 것과 동일하게 **파일 디스크립터(File Descriptor, FD)**를 반환한다.

MAVLink 모듈(또는 토픽을 기다리는 어떤 백그라운드 모듈이든)은 반환받은 구독 FD를 쥐고, POSIX 표준 시스템 콜인 poll() (PX4 래퍼 함수인 px4_poll())을 호출한다.

  1. OS 레벨의 블로킹(Blocking): px4_poll()이 호출되는 순간, MAVLink 스레드의 실행은 그 라인에서 멈추고(Blocking), 상태가 ’대기(Waiting)’로 전환되며 CPU 소모량은 0%가 된다.
  2. 인터럽트 웰이크업(Wake-up): commander 모듈이 orb_publish()를 호출하여 새로운 vehicle_status 데이터를 밀어 넣는 순간, uORB 코어는 해당 FD에 이벤트 시그널을 발생시킨다. OS 스케줄러는 즉각 잠들어 있던 MAVLink 스레드를 깨워 실행(Runnable) 상태로 되돌린다.

3. 논블로킹 패러다임과 타임아웃(Timeout)의 우아한 결합

여기서 동기화(Synchronization)의 천재성이 발휘된다. HEARTBEAT 패킷은 상태 변경 이벤트가 발생하지 않더라도 생존 증명을 위해 무조건 1Hz(1000ms) 주기로 발송되어야 한다.

따라서 mavlink 루프는 px4_poll()을 호출할 때 타임아웃 인자를 1000ms로 설정한다.

  • 상태 갱신 시 (Event-driven): 만약 1000ms 내에 사용자가 조종기로 모드를 바꿔 commander가 새 토픽을 발행하면, poll()은 즉시 반환(Return)을 때리고 MAVLink 모듈은 방금 바뀐 신선한 모드를 즉각 하트비트로 쏴버린다. (반응성 극대화)
  • 평시 상태 (Tick-driven): 아무런 모드 변경이 없어서 1000ms 동안 commander가 조용하다면, poll()은 아무 데이터도 수신하지 못했지만 ’타임아웃(Timeout)’을 이유로 스레드를 깨운다. 깨어난 스레드는 기존에 메모리에 캐싱하고 있던 마지막 상태 값을 그대로 다시 직렬화하여 “나는 아직 살아있다“고 하트비트를 발송한다.
// PX4 소스 코드 패러다임의 단순화된 의사 코드 (Pseudo Code)
px4_pollfd_struct_t fds[1];
fds[0].fd = orb_subscribe(ORB_ID(vehicle_status)); // 파일 디스크립터 획득
fds[0].events = POLLIN;

while (!should_exit) {
    // 1000ms 타임아웃을 걸고 OS 단위로 딥슬립 (CPU 0%)
    int poll_ret = px4_poll(fds, 1, 1000); 

    if (poll_ret > 0) {
        // 이벤트 발생! commander.cpp가 새 상태를 발행함. 지연 시간 없이 즉시 갱신.
        orb_copy(ORB_ID(vehicle_status), fds[0].fd, &status_obj);
        send_heartbeat(status_obj); // 즉각적인 UI 반응성 보장
    } else if (poll_ret == 0) {
        // 1000ms 타임아웃 도달. 변한 건 없지만, 의무적인 1Hz 생존 핑(Ping) 발송.
        send_heartbeat(status_obj); 
    }
}

이처럼 PX4의 파이프라인은 폴링(Poll) 시스템 콜을 통해 CPU 자원 낭비를 제로(0) 수준으로 단속하는 성능과, 모드가 변경된 지 불과 밀리초 만에 즉시 관제 화면을 갱신해 내는 극한의 반응성, 그리고 아무 일이 없어도 1초마다 무조건 전송하는 정시성이라는 세 마리 토끼를 C++ 네트워크 프로그래밍의 가장 우아한 형태로 잡아내고 있다.
