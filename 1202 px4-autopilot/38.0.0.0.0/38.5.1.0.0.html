38.5.1 제어 네트워크 내 경쟁 상태(Race Condition) 시나리오 기하학적 모델링

순수한 통신 두절(Loss) 이외에도, 비동기적(Asynchronous)으로 동작하는 GCS와 UAV 사이에는 필연적으로 시간 지연에 따른 **경쟁 상태(Race Condition)**라는 끔찍한 제어 공학적 함정이 도사리고 있다.

이 경쟁 상태를 가장 명확히 이해하기 위해, 우리는 시간을 X축으로, 물리적 공간(GCS와 비행 제어기간의 거리)을 Y축으로 둔 2차원 시공간(Spacetime) 다이어그램을 상상해 볼 수 있다.

  • 상향링크(Uplink) 명령은 GCS에서 출발해 우상향하는 기울기(\Delta t_{up})의 궤적을 그린다.
  • 하향링크(Downlink) 텔레메트리(HEARTBEAT)는 UAV에서 출발해 우하향하는 기울기(\Delta t_{down})의 궤적을 그린다.

1. 교차(Crossing) 현상과 선후 관계의 붕괴

문제는 이 두 궤적이 허공(RF 공간)에서 X자로 **교차(Cross)**할 때 발생한다. 다음의 치명적인 시나리오를 살펴보자.

  1. 시점 t_0: 기체는 현재 Hold(제자리 비행) 모드에 있다.
  2. 시점 t_1: 운용자가 GCS 화면에서 Return(복귀) 버튼을 누른다. GCS는 즉각 기체를 향해 상향링크로 모드 전환 명령(COMMAND_LONG)을 발사한다. 동시에 GCS 내부의 섀도우 상태 머신은 “곧 기체가 Return 모드가 될 것“이라 기대하며 전환 준비 상태(Transitioning)에 돌입한다.
  3. 동시점 t_1 (교차 발생): 명령이 아직 기체에 닿기 직전, 기체는 여전히 자신이 Hold 모드라 굳게 믿고 있으며, 마침 1Hz 주기의 HEARTBEAT 패킷을 하향링크로 정기 발송한다. (“나는 Hold 상태다!”)
  4. 시점 t_2: GCS가 방금 전 기체가 보낸 HEARTBEAT를 수신한다. 여기에는 선명하게 “현재 모드: Hold“라고 적혀있다.
  5. 시점 t_3: 기체가 GCS가 보낸 Return 명령을 드디어 수신하고 모드를 전환한다.

만약 GCS를 1차원적인 상태 동기화 로직으로만 코딩했다면(Ardupilot 기반의 구형 미션 플래너 등에서 가끔 목격되는 증상), 시점 t_2에서 끔찍한 UI 붕괴가 발생한다.

GCS는 조금 전 자신이 Return 명령을 보냈다는 역사적 맥락을 망각한 채, 막 도착한 신선한 HEARTBEAT 패킷을 맹목적으로 신뢰해 버린다. 그 결과 운용자 화면을 강제로 다시 Hold 모드로 롤백(Rollback)시켜 버리며, 운용자는 “분명히 Return을 눌렀는데 왜 다시 Hold로 튕겨 나가지?“라며 패닉에 빠져 버튼을 연타하게 된다. (버튼 연타는 네트워크 혼잡 윈도우를 다시 붕괴시키는 악순환을 낳는다.)

2. 상태 전환 보류(State Transition Hold) 메커니즘

이러한 기하학적 교차 상태에서 벌어지는 낡은 데이터(Stale Data)의 폭격을 막아내기 위해, PX4 아키텍처와 MAVSDK 기반의 GCS 백엔드는 명령 응답 확인(COMMAND_ACK) 기반의 상태 락(Locker) 메커니즘을 구현한다.

GCS가 능동적인 제어 명령을 상향링크로 쏘고 나면, 기체로부터 명시적인 COMMAND_ACK (승인) 패킷이 돌아오거나 설정된 타임아웃(Timeout)이 만료될 때까지 일시적인 제한 구역(Transition Window)을 설정한다.
이 윈도우 기간 동안, 과거의 유령처럼 날아 들어오는 구형 HEARTBEAT 패킷들의 모드 정보는 철저히 **무시(Ignoring / Masking)**된다. 오직 COMMAND_ACK를 받은 이후에 도착하는 새로운 HEARTBEAT만이 진실(True State)로 채택되어 분산 시스템 간의 핑퐁(Ping-Pong) 착각 현상을 원천 차단하는 것이다.
