38.2.1.1 이기종 시스템 간 엔디안(Endianness) 불일치 해소를 위한 네트워크 바이트 정렬(Network Byte Order) 최적화

무인항공기 제어 네트워크는 본질적으로 이기종 분산 컴퓨팅(Heterogeneous Distributed Computing) 환경이다. 지상 관제 시스템(GCS)은 주로 인텔 Core나 AMD 라이젠 기반의 x86/x64 아키텍처(Windows/Linux)에서 구동되고, 기체의 오토파일럿(Pixhawk)은 ARM Cortex-M 계열 마이크로컨트롤러에서, 영상 처리를 담당하는 보조 컴퓨터(Companion Computer)는 ARM Cortex-A 계열(Raspberry Pi, Jetson)에서 구동된다.

이러한 이기종 시스템 장치들이 통신할 때 가장 먼저 직면하는 하드웨어적 장벽이 바로 멀티바이트(Multi-byte) 데이터를 메모리에 저장하는 순서인 **엔디안(Endianness)**의 불일치 현상이다.

1. 빅 엔디안(Big-Endian) 네트워크 표준에 대한 MAVLink의 반항적 설계

전통적인 TCP/IP 인터넷 프로토콜 스택은 서로 다른 컴퓨터 간의 통신 호환성을 보장하기 위해, 네트워크 상으로 데이터를 전송할 때 반드시 빅 엔디안(Big-Endian) 방식을 따르도록 강제한다. 이를 **네트워크 바이트 정렬(Network Byte Order)**이라고 부르며, 송수신 측은 htonl()이나 ntohl() 같은 함수를 호출하여 자신의 호스트 바이트 정렬(Host Byte Order)을 네트워크 표준으로 상호 변환해야 한다.

그러나 MAVLink 프로토콜 설계자들은 이 전통적인 법칙을 과감히 폐기하고, MAVLink 통신망 위에서 전송되는 모든 패킷의 기본 엔디안을 **리틀 엔디안(Little-Endian)**으로 규정했다. 이는 다분히 드론 통신 환경의 제약을 고려한 극단적인 실용적 아키텍처 결단이었다.

1.1 오버헤드 비대칭성(Overhead Asymmetry)의 원리

그 이유는 시스템 생태계의 절대다수를 구성하는 칩셋들의 하드웨어 특성에 기인한다.

  1. 오토파일럿 MCU: STM32 등 ARM Cortex-M 칩셋은 네이티브 리틀 엔디안 모드로 동작한다.
  2. GCS 프로세서: 인텔/AMD x86_64 아키텍처 역시 철저한 네이티브 리틀 엔디안이다.
  3. 보조 컴퓨터: 최신 ARM 애플리케이션 프로세서들도 대부분 리틀 엔디안 모드로 구동된다.

즉, MAVLink 네트워크에 참여하는 노드의 99% 이상이 이미 리틀 엔디안 아키텍처로 통일되어 있는 셈이다. 만약 전통적인 TCP/IP 규칙을 따라 무선 링크의 표준을 빅 엔디안으로 강제했다면, 모든 노드가 데이터를 보낼 때마다 바이트 스와핑(Byte Swapping) 연산을 수행해야 하는 거대한 CPU 사이클 낭비가 매 초당 수천 번의 메시지마다 발생했을 것이다.

MAVLink는 전송 표준 자체를 리틀 엔디안으로 확정함으로써, 비행 제어기(PX4)가 메모리 배열(RAM)에 있는 구조체 버퍼를 아무런 CPU 연산(Zero-copy byte-swap) 없이 곧바로 DMA(Direct Memory Access)를 통해 UART 포트로 밀어 넣을 수 있도록 통신 오버헤드를 제로(0)에 가깝게 최적화했다.

2. HEARTBEAT 커스텀 모드(custom_mode)의 엔디안 파싱

HEARTBEAT 패킷의 최상단에 위치한 4바이트(32비트) custom_mode 필드는 이러한 리틀 엔디안 버스 정렬의 수혜를 직접적으로 받는다.

만약 PX4가 custom_mode 값을 0x12345678로 설정했다면, 송신 UART 버퍼에는 최하위 바이트(LSB)인 0x78부터 시작해 0x56, 0x34, 0x12 순서로 직렬화되어 무선 공간을 날아간다. 수신 측 GCS(QGroundControl 등) 역시 x86 프로세서이므로, 들어온 4바이트 블록을 별도의 시프트 연산 없이 그대로 32비트 포인터로 캐스팅(Casting)하여 읽어 들이면 완벽하게 0x12345678이 복원된다.

2.1 소수파(Minority) 노드를 위한 매크로 스위치

그럼에도 불구하고 과거의 PowerPC나 특정 MIPS 코어 같은 빅 엔디안 프로세서가 MAVLink 네트워크에 하드웨어 컴포넌트로 플러그인될 가능성도 배제할 수 없다.

이를 위해 MAVLink 헤더 생성기(mavgen)는 C 코드를 생성할 때 컴파일러 단에서 호스트의 엔디안을 판별하여 타겟 코드를 분기시킨다.

#if MAVLINK_NEED_BYTE_SWAP
    // 빅 엔디안 노드 전용: 송수신 시 CPU를 소모하여 바이트를 뒤집음
    _mav_put_uint32_t(buf, 0, custom_mode); 
#else
    // 리틀 엔디안 노드 전용: 메모리 복사 연산(Memcpy)만으로 즉시 처리
    buf[0] = custom_mode; 
#endif

결과적으로 Ardupilot과 PX4 공히, 이기종 간의 엔디안 불일치 패널티(Penalty)를 소수인 빅 엔디안 노드 쪽으로 전가(Offloading)하여, 비행에 가장 필수적인 메인 제어 루프의 통신 지연시간(Latency)을 마이크로초 단위까지 최적화하는 통신 공학적 승리를 거두었다.
