38.5 상태 불일치(State Mismatch) 탐지 및 분산 합의(Distributed Consensus) 기반 복구

앞선 38.4절까지 우리는 통신 링크 그 자체의 물리적/확률론적 건전성에 대해 논의했다. 그러나 통신망이 잠시 끊어졌다가(Data Link Loss) 기적적으로 다시 연결(Re-established)되는 그 찰나의 순간, 관제 시스템은 프로토콜 레이어를 넘어선 훨씬 끔찍한 논리적 재앙, 즉 상태 불일치(State Mismatch) 상황에 직면하게 된다.

1. 상태 불일치(State Mismatch)의 정의와 파국적 시나리오

상태 불일치란 GCS의 메모리 캐시 속에 아로새겨진 **섀도우 상태(Shadow State, 디지털 트윈)**와 수 킬로미터 밖 상공에 떠 있는 기체 내부(PX4 Commander)의 **물리적 진성 상태(True State)**가 네트워크 단절이나 패킷 유실로 인해 서로 다른 궤도로 분기해 버린 현상을 뜻한다.

전형적인 파국적 시나리오는 다음과 같다.

  1. 기체가 지상 관제의 통제하에 Mission 모드로 비행 중이다. (GCS 상태: Mission / UAV 상태: Mission)
  2. 산등성이를 넘으며 심각한 다중 경로 페이딩으로 통신이 30초간 완전히 단절된다.
  3. PX4의 Data Link Loss 페일세이프가 발동하여, 기체는 스스로 생존을 위해 모드를 Return(RTL)으로 전환하고 이륙장으로 기수를 돌린다. (GCS 상태: Mission / UAV 상태: Return)
  4. 30초 후 기체가 고도를 높이면서 LTE 통신이 극적으로 복구된다.

이 끊어졌던 선이 다시 이어진 직후, 아무런 합의 알고리즘이 없다면 GCS 화면에는 여전히 “임무 수행 중“이라는 초록색 불이 켜져 있는데, 실제 기체는 전속력으로 되돌아오고 있는 환각(Hallucination) 상태에 빠진다. 만약 당황한 운용자가 이 상태에서 “임무 일시 정지(Hold)” 명령을 누른다면, 기체 입장에서는 “Return 중인데 왜 Mission 규격의 Hold 명령이 날아오는가?“라며 커맨드를 거부(Command Denied)해 버리는, 통제권의 총체적 난국(스플릿 브레인)이 발생한다.

2. 고빈도 Heartbeat를 통한 멱등적(Idempotent) 상태 비교 탐지

이러한 분열을 막기 위해 PX4 아키텍처와 MAVSDK 기반의 현대적 GCS는 멱등성(Idempotency) 기반의 탐지 로직을 가동한다.

기체가 초당 1회씩 쏘아 보내는 MAVLink HEARTBEAT와 EXTENDED_SYS_STATE 메시지는 단순히 생존 신고의 의미를 넘어, 내부에 기체의 절대적인 진성 상태 벡터(Arming State, Nav Mode, Custom Mode 등)를 빼곡히 채워 넣고 있다. GCS의 통신 데몬은 이 HEARTBEAT가 도착할 때마다, 즉시 패킷 내부의 (Mode_UAV, Arm_UAV) 버전을 꺼내어 GCS 자신이 마지막으로 기억하고 있던 (Mode_GCS, Arm_GCS) 튜플과 바이트 단위로 대조(Diff) 연산을 수행한다.

만약 여기서 1비트라도 틀어짐이 감지되면, 아키텍처는 즉각 State Mismatch Exception 이벤트를 발생시켜 양측의 뇌가 분리되었음을 시스템에 경고한다.

3. 분산 합의(Distributed Consensus) 도메인에서의 단일 진실 공급원(SSOT)

불일치가 감지되었다면 누구의 상태가 맞는 것일까?
클라우드 데이터베이스의 Raft나 Paxos 같은 분산 합의 알고리즘에서는 노드 간 투표를 통해 진실을 결정하지만, 무인기 관제 아키텍처의 권력 구조는 철저히 수직적이다. 모든 제어 공학적 합의 과정에서 **단일 진실 공급원(SSOT, Single Source of Truth)은 오직 비행 제어기(PX4)**가 쥐고 있어야 한다.

통신이 복구되어 불일치가 탐지되는 즉시, GCS는 자신이 가지고 있던 낡은 섀도우 상태 벡터를 즉각 폐기(Flush)하고, 무조건적으로 UAV가 주장하는 현재 상태(위 시나리오의 경우 Return 모드)를 그대로 강제 덮어쓰기(Overwrite)하여 합의를 강제한다.

결코 GCS가 “내 메모리에는 Mission으로 되어 있으니, 너도 당장 Mission으로 돌아가라“고 기체의 모드를 기계적으로 끌어내리는(Revert) 오만함을 범해서는 안 된다. 오직 운용자(Human-in-the-loop)가 변경된 화면을 보고 새로운 판단을 내려 명시적인 전환 커맨드(COMMAND_LONG)를 내릴 때만 상태 전이가 이루어지도록 엄격하게 잠그는 것, 이것이 PX4 기반 분산 제어의 핵심적인 상태 복구 철학이다.
