40.5.4.1. 타임스탬프 단조 증가(Monotonically Increasing Timestamp) 성질을 이용한 스텔스 패킷(Stale Packet) 원자적 폐기(Atomic Discard)

드론 관제 통신 네트워크(예: MAVLink) 환경에서는 일시적인 전파 간섭, 다중 경로 페이딩(Multipath Fading) 또는 네트워크 라우팅 지연 등으로 인해 지상 관제 시스템(GCS)에서 송신한 제어 패킷이 순서가 뒤섞이거나(Out-of-order) 예상보다 훨씬 늦게 기체에 도달할 수 있다. QGroundControl(QGC)과 PX4-Autopilot 환경에서, 과거에 발송된 낡은 제어 명령이 뒤늦게 유효하게 해석되어 비행 기체의 물리적 거동을 방해하는 것을 방지하기 위해 타임스탬프 단조 증가(Monotonically Increasing Timestamp) 성질을 이용한 스텔스 패킷(Stale Packet) 원자적 폐기(Atomic Discard) 기법을 철저히 구현해야 한다.

1. 개요 및 스텔스 패킷(Stale Packet) 모델링

네트워크 혼잡(Congestion) 상황 하에서, GCS 개발 엔지니어는 MAVLink 패킷 전달의 순차성과 실시간성(Real-time Constraints)을 맹신해서는 안 된다.

  • 스텔스 패킷(Stale Packet): 수신자(PX4) 측에 도달한 패킷 중 이미 논리적 유효 기간(Validity Window)이 지났거나, 이전에 더 최신의 제어 명령이 수신되어 상태가 천이된 이후에 도착한 과거의 ‘낡은(Stale)’ 패킷을 의미한다.

예를 들어, GCS 사용자가 기체의 고도를 50m로 설정하는 COMMAND_INT 메시지 M_1을 t=0에 송신했으나, 네트워크 지연에 빠져 즉시 도달하지 못했다. 조종자는 기체가 반응하지 않자 고도를 20m로 재설정하는 메시지 M_2를 t=2에 송신했고 이는 즉시 도달하여 기체가 20m로 하강하기 시작했다. 이후 t=5에 네트워크에서 빠져나온 낡은 메시지 M_1이 수신된다면, 약 5초 전의 “50m로 상승하라“는 무효한 과거 명령이 실행되는 치명적인 상태 역행(State Reversion)이 발생한다.

2. 타임스탬프 단조 증가(Monotonically Increasing Timestamp) 성질

이러한 상태 역행을 막기 위한 가장 견고한 전산학적 해결책 중 하나가 패킷 헤더나 페이로드에 단조 증가(Monotonically Increasing)하는 타임스탬프(Timestamp)나 시퀀스 번호(Sequence Number)를 원자적으로 강제 결합(Atomic Binding)하는 것이다.

단조 증가란 시간이 흐름에 따라 값 T가 이전 값보다 결코 작아지지 않는(순 감소하지 않는) 수학적 속성이다.
\forall i < j, \quad T(i) < T(j)

2.1. QGroundControl 측 타임스탬프 주입 (Sender Side)

QGC 내부의 MavlinkCommandSender 모듈에서 제어 패킷을 발송할 때, 일반적인 시스템 Wall-Clock 시간이 아닌, 역행할 수 없는 시스템 업타임(예: QElapsedTimer를 통한 msec 또는 usec) 값을 명령 페이로드 내부에 타임스탬프로 삽입한다.

2.2. PX4-Autopilot 측 원자적 필터링 (Receiver Side)

PX4 내부의 Commander나 MAVLink 수신 처리 객체(mavlink_receiver.cpp)는 메모리 영역에 마지막으로 유효하게 수신·승인한 제어 명령의 타임스탬프(이를 T_{latest\_valid}라 칭함)를 std::atomic<uint64_t>로 기록하고 유지한다.

새로운 패킷 P_{new}가 도달하여 내재된 타임스탬프 T_{new}를 해석할 때, 다음의 검사가 원자적 블록(Atomic Block) 내에서 선형적으로 수행(Linearizability)되어야 한다.

\text{If } T_{new} \leq T_{latest\_valid} \text{, then Discard}(P_{new})
\text{Else, accept } P_{new} \text{ and set } T_{latest\_valid} \leftarrow T_{new}

4. 실무 적용 시 고려사항: Wrap-around 및 Clock Drift

단조 증가 타임스탬프 로직을 uint32_t나 uint64_t로 구현하여 펌웨어나 GCS 백엔드에 반영할 때, 반드시 다음의 분산 시스템 엔지니어링 한계를 극복해야 한다.

  1. Wrap-around (오버플로우) 대처: 만약 타임스탬프 자료형이 uint32_t (ms 단위)라면 약 49.7일 후 오버플로우되면서 0으로 회귀한다. 이때 단조 증가 속성이 파괴되므로, 두 타임스탬프의 차이 값을 부호 있는 정수연산(int32_t)으로 비교하여, 특정 범위(예: 절반 값)의 음수가 된 경우 오버플로우로 간주하여 수용하는 예외 로직을 반드시 포함해야 준다.
  2. Clock Drift (시계 지연 현상): 다중 GCS 시스템(예: 모바일 GCS와 메인 GCS가 분리된 스웜 관제)에서 각기 다른 컴퓨터 시스템 시계를 의존할 경우 타임스탬프 비교가 어긋날 수 있다. 독립적인 오프보드 제어 에이전트(ROS2 노드)가 접근할 경우, NTP(Network Time Protocol) 또는 타임싱크(Timesync) MAVLink 확장 메시지를 활용해 발신자(Sender)와 수신자(Vehicle) 간의 기준 시계를 마이크로초 단위로 엄밀하게 일치시켜야만 Stale Packet 필터가 오작동하지 않는다.

5. 결론

비행 제어 소프트웨어에서 제어 명령어의 실행 순서를 보장하는 것은 PID 제어 루프를 튜닝하는 것만큼이나 기체의 생존을 좌우하는 코어 통신 아키텍처이다.

개발자는 PX4와 교신하는 커스텀 GCS나 ROS2 자율 비행(Offboard) 노드를 설계할 때, 맹목적인 MAV_CMD 발송 루프를 지양해야 한다. 대신, 단조 증가 타임스탬프를 패킷에 주입하고, 비행 제어기 측에서 Stale 패킷을 식별·차단하는 원자적 폐기(Atomic Discard) 루틴이 활성화될 수 있도록 철저한 메타데이터 전송 규약을 수립해야 한다. 이를 통해 예측 불가능한 통신 망 지연 속에서도 PX4 상태 머신의 결정론적(Deterministic) 안결성을 강력하게 보수할 수 있다.
