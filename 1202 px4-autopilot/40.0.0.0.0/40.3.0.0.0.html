40.3 적응형 타임아웃(Adaptive Timeout)의 시계열 제어 이론 적용

본 문서에 사용된 기준 버전은 PX4-Autopilot v1.14.0 및 QGroundControl v4.3.0이다.

무인항공기(UAV)와 지상 관제 시스템(GCS) 사이의 MAVLink 기반 텔레메트리(Telemetry) 통신은 RF 서브시스템(RF Subsystems), 고도, 주변 지형지물, 전파 간섭 등 통신 환경에 따라 왕복 지연 시간(Round-Trip Time, RTT)이 시시각각 변동한다.
관제 명령(Command)을 GCS에서 기체로 전송할 때 고정된 타임아웃(Static Timeout)을 설정할 경우, 통신 지연이 일시적으로 커질 때마다 불필요한 패킷 재전송(Retransmission)이 폭주하여 제한된 대역폭(Bandwidth)을 마비시키거나, 반대로 지연 예외 처리가 너무 늦어지는 제어 부작용을 낳게 된다.

이를 근본적으로 해결하기 위해 QGroundControl(QGC) 백엔드와 PX4-Autopilot의 커맨더(Commander) 양단에서는 **적응형 타임아웃(Adaptive Timeout)**이라는 시계열 제어 이론(Time-Series Control Theory)을 적용하여 동적인 네트워크 응답성을 도모해야 한다.

1. 고정 타임아웃(Static Timeout)의 한계와 적응형 제어의 필요성

과거 GCS나 초기 Ardupilot 연동 관제 체계에서는 중요한 명령(예: COMMAND_LONG)을 보낸 뒤 COMMAND_ACK가 오기까지 기다리는 시간을 500ms 혹은 1초와 같이 하드코딩된 정적 변수로 취급했다. 이러한 접근 방식은 다음과 같은 맹점을 지닌다.

  • False Positive (오탐지): 일시적으로 통신 거리가 멀어져 RTT가 600ms로 늘어났는데 타임아웃 변수 설정값이 500ms라면, 패킷이 정상적으로 도착하고 있음에도 불구하고 GCS는 실패로 오판하여 동일한 패킷을 반복 전송함으로써 귀중한 무선 대역폭 자원을 낭비한다.
  • 통신 폭주(Congestion Collapse): 불필요한 재전송이 텔레메트리 모뎀의 큐 리스트를 꽉 채워버리면서 진짜 시급한 자세 제어 메시지(Attitude)나 위치 추적(Local Position) 텔레메트리의 수신을 막는 연쇄 셧다운 현상을 초래한다.

따라서 PX4-Autopilot 환경에서는 시계열 데이터를 면밀히 수집 및 분석하여 네트워크의 실시간 지연 상태를 능동적으로 추정하는 폐루프(Closed-loop) 제어 모델을 도입해야 한다.

2. 적응형 타임아웃(Adaptive Timeout) 제어 모델 및 수식화

QGC나 분산 브릿지 서버(ROS2 uXRCE-DDS 에이전트 등)에서 적정 타임아웃 타이머 한계인 T_{RTO} (Retransmission Timeout)를 동적으로 결정하기 위해서는, RTT의 과거 샘플들을 필터링하며 부드럽게 추종(Tracking)하는 알고리즘 패턴이 요구된다.
일반적으로 TCP/IP의 혼잡 제어(Congestion Control)에서 널리 검증된 Jacobson / Karels 평활화 알고리즘 구조를 차용하라.

2.1 왕복 시간(RTT) 연속 계측 (Real-time RTT Measurement)

MAVLink 프로토콜의 표준 패킷 기능인 PING 메시지나 내부 타임스탬프가 포함된 TIMESYNC 메시지 규격을 활용하여 이산 샘플 RTT_i 값을 주기적으로 샘플링(Sampling)한다.

2.2 지수 가중 이동 평균(EWMA)을 통한 평활화 (Smoothing)

단발적인 RF 노이즈나 일시적 채널 간섭으로 인한 RTT 튐(Spike) 현상이 타임아웃을 비합리적으로 요동치게 만드는 것을 방어하기 위해, 시계열 평활화 필터인 SRTT(Smoothed RTT)와 RTTVAR(RTT Variation) 상태 변수를 갱신한다.

SRTT_{i} = (1 - \alpha) \cdot SRTT_{i-1} + \alpha \cdot RTT_i
RTTVAR_{i} = (1 - \beta) \cdot RTTVAR_{i-1} + \beta \cdot \vert SRTT_{i-1} - RTT_i \vert

위 수식에서 \alpha(통상 0.125 적용)와 \beta(통상 0.25 적용)는 제어 필터의 극점(Pole)과 수렴 속도를 결정하는 수학적 가중치 팩터이다.
PX4의 uORB 워크 큐 혹은 QGroundControl의 텔레메트리 쓰레드 갱신 라인 내부에서, 이 수식을 소프트웨어 메인 타이머 주기와 연계하여 빈틈없이 계산하라.

2.3 T_{RTO} (Retransmission Timeout)의 최적 한계점 설정

최종적으로 어플리케이션이 다음 재전송을 트리거하기 위해 대기하는 한계 시간 T_{RTO}는, 평균 지연 시간에 지연 분산차(Variation)의 약 4배 가량을 덧붙여 타당성 있는 안전 마진(Safety Margin)을 이끌어낸다.

T_{RTO} = SRTT_{i} + \max(G, 4 \cdot RTTVAR_{i})
(단, G는 OS 타이머 스케줄러의 최소 클럭 분해능(Granularity)이다.)

이때 드론 전원이 아예 꺼지거나 통신망 권역을 심각하게 이탈하여 계산된 T_{RTO}가 무한정 커지는 예기치 않은 오류를 막기 위해, 상한치(Upper Bound, 예: 3000ms)와 하한치(Lower Bound, 예: 200ms) 클리핑(Clipping) 가드 코드를 필수로 덧붙여라.

4. 실무 구현 가이드 및 코딩 주의사항

QGroundControl 프레임워크나 커스텀 C++ MAVSDK 애플리케이션에 본 적응형 타임아웃 제어 공학을 이식할 실무 개발자는 다음의 사안을 엄격히 준수하라.

  1. 지수적 백오프(Exponential Backoff) 혼합 적용: 타임아웃 변수 T_{RTO}가 만료되어 패킷을 불가피하게 다시 쏠 때마다, 다음 틱의 억제 대기 시간은 이전 지연의 2배로 할당하여, 통신 채널의 전파 혼잡도를 의도적으로 완화시켜라.
  2. QGC 커스텀 Vehicle 클래스 및 링크 오버라이딩: QGroundControl 내부 코어에 위치한 Vehicle.cc 및 프로토콜 매니저 소스에서 재전송을 관장하고 있는 고정 상수형 QTimer 변수 체계를, 실시간 산출되는 SRTT 수식에 의해 업데이트가 허용되는 동적 타이머(Dynamic Timer) 클래스로 재설계(Refactoring)하라.
  3. 텔레메트리 하드웨어 모듈 특수성 반영: 마이크로하드(Microhard), 실리콘 랩(SiK) 계열 라디오, 또는 이리듐 위성 통신 모듈은 각각 고유의 TDMA 타임슬롯 지연 특성과 징크(Jitter) 패턴을 소유하므로, 최초 필드 테스트 과정을 통해 \alpha와 \beta 필터 가중치를 하드웨어 규격에 맞게 치밀하게 튜닝하라.
