40.3.4.1 컨텍스트 스위칭(Context Switching) 지연 시간의 타이머 인터럽트(Timer Interrupt) 간섭 효과

무인항공기(UAV) 자동비행장치(Autopilot)의 실시간성(Real-Time Property)을 보장하기 위해서는 운영체제의 컨텍스트 스위칭(Context Switching) 속도가 일정 수준 이하로 유지되어야 한다. PX4-Autopilot이 채택한 NuttX 실시간 운영체제(RTOS) 환경에서는 고주파수(High-frequency)로 동작하는 다양한 내부 타이머(Timer)들이 존재하며, 이들이 프로세스 전환 과정에 개입하여 의도치 않은 지연을 유발하는 현상을 타이머 인터럽트 간섭 효과(Timer Interrupt Interference Effect)라고 한다. 본 절에서는 이러한 간섭 효과가 스케줄링 지연 시간에 미치는 물리적 영향과 이를 최소화하기 위한 PX4 아키텍처의 대응 전략을 논한다.

1. 컨텍스트 스위칭 및 타이머 인터럽트 구조 이해

지상 관제 시스템(GCS)으로 데이터를 전송하는 MAVLink 송신 태스크와 비행 자세를 제어하는 제어 태스크 간의 제어권 전환 과정(Context Switching)은 필연적으로 중앙처리장치(CPU) 레지스터의 상태 저장(Push) 및 복구(Pop) 과정을 수반한다.

이때 Pixhawk 하드웨어 시리즈의 핵심이 되는 STM32 마이크로컨트롤러 환경에서는 NVIC(Nested Vectored Interrupt Controller)를 통해 다양한 인터럽트가 관리된다. 시스템 틱(System Tick)을 관장하는 OS 타이머 외에도 PX4 펌웨어(v1.14 기준)는 센서 데이터 동기화를 위해 고해상도 타이머(High-Resolution Timer, hrt_call API)를 1us 미만의 정밀도로 다수 운용한다. 컨텍스트 스위칭 코어 루틴이 실행되는 찰나의 순간에 이러한 하드웨어 타이머 인터럽트가 동시다발적으로 발생하면, 인터럽트 핸들러(Interrupt Handler)가 강제로 실행되어 본래의 스위칭 시간이 비정상적으로 연장된다.

아래 다이어그램은 타이머 인터럽트가 일반적인 컨텍스트 스위칭 과정에 개입하여 총 지연 시간을 늘리는 간섭 상태를 도식화한 것이다.

    participant T_Low as 하위 태스크 (MAVLink 통신)
    participant Kernel as NuttX 커널 (스케줄러)
    participant HRT as 고해상도 타이머 (HRT) 인터럽트
    participant T_High as 상위 태스크 (자세 제어)

    T_Low->>Kernel: 선점(Preemption) 신호 감지
    Kernel->>Kernel: 레지스터 상태 저장 (Push) 시작
    HRT-->>Kernel: 타이머 인터럽트 발생 (센서 동기화 등)
    Note over Kernel, HRT: 컨텍스트 스위칭 일시 중단 (간섭 발생)
    HRT->>HRT: 타이머 콜백 서비싱 (Interrupt Service Routine)
    HRT-->>Kernel: ISR 처리 완료 및 리턴
    Kernel->>Kernel: 남은 문맥 저장 및 복원 (Pop) 재개
    Kernel->>T_High: 제어권 양도
    Note right of T_High: 예상 시간보다 (ISR 소요 시간만큼) 지연되어 실행 시작

2. 간섭 효과의 수학적 모델링

컨텍스트 스위칭 오버헤드 L_{csw}는 순수한 OS 레벨의 문맥 교체 기본 베이스 시간 L_{base}와, 스위칭 도중 개입하는 k개의 타이머 인터럽트 처리 시간들의 총합으로 모델링된다.

L_{csw} = L_{base} + \sum_{i=1}^{k} I_{timer\_i}

여기서 I_{timer\_i}는 i번째 타이머의 인터럽트 서비싱에 소진된 시간이다. 이러한 타이머 간섭이 지속적으로 누적되면, Estimator(EKF2)의 시간 종속 모형 모델(Time-Dependent Kinematic Model)이나 센서 데이터 융합 루프의 주기성이 어긋나게 되어 물리적 제어 품질이 하락할 수 있다.

3. PX4와 Ardupilot의 타이머 인터럽트 관제 기법 차이

QGroundControl(v4.3)과 같은 GCS를 기반으로 개발을 진행할 때, 백엔드에서는 각 비행 제어 소프트웨어의 타이머 처리 아키텍처 결함을 보정하는 알고리즘을 갖추어야 한다.

  • Ardupilot의 타이머 아키텍처: 전통적으로 Ardupilot은 I/O를 전담하는 코프로세서(Co-processor)로 하드웨어 타이머와 센서 폴링(Polling) 로직을 최대한 이관하여, 메인 MCU의 루프 스케줄링 간섭을 원천적으로 차단하는 분리 하드웨어 아키텍처 접근법을 선호한다. 이로 인해 메인 컨텍스트의 인터럽트 간섭 확률이 상대적으로 평탄하게 유지된다.
  • PX4-Autopilot의 타이머 아키텍처: PX4는 단일화된 고해상도 타이머 큐(hrt Queue)에서 콜백 기반으로 이벤트를 트리거하는 방식을 강력하게 고수한다. 이 방식은 코드 모듈화 및 이식성(Portability)을 극대화시켜 새로운 임베디드 보드를 수용하는 데 매우 유리하지만, 다수의 타이머가 동시에 만료(Expiration)될 때 컨텍스트 스위칭 구간에서 병목 및 간섭 겹침 현상이 증폭될 수 있는 잠재적 취약점을 내포한다.

4. 커스텀 자동 에이전트 개발을 위한 소프트웨어 엔지니어링

독자적인 관제 애플리케이션 및 비행 스택을 커스터마이징할 경우, 개발자는 위와 같은 hrt_call 큐의 중첩(Nesting) 및 스위칭 간섭 효과를 회피하기 위해 STM32 칩셋 레벨의 다이렉트 메모리 엑세스(DMA, Direct Memory Access) 기능을 극한으로 활용해야 한다.

SPI 버스나 UART 통신을 철저히 비동기 DMA(Asynchronous DMA)로 이관하여 타이머 인터럽트 구동을 최소화하면, 컨텍스트 스위칭 지연 기본값 L_{base}만으로 태스크 전환을 보장할 수 있다. 이는 ROS2 uXRCE-DDS 퍼블리셔(Publisher) 태스크가 다른 핵심 센서/동역학 연산과 경합할 때 네트워크 메시지 발행 주파수(예: 50Hz, 100Hz)의 스파이크(Spike)를 억제하여 안정적인 관제 통신 및 응답성(Responsiveness) 향상에 결정적인 기여를 한다. 관련하여 PX4 시스템 내부의 SystemLoad 타스크 로그 및 ULog 스케줄러 기록 데이터를 QGroundControl의 분석 툴과 교차 검증함으로써, 개발자는 타이머 간섭 없는 견고한 선점형 다중 스레드 설계를 달성할 수 있다.
