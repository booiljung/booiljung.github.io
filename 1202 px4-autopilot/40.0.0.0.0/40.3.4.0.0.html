40.3.4 실시간 운영체제(RTOS)의 선점형 스케줄링(Preemptive Scheduling) 지터(Jitter) 모델링

무인항공기(UAV)의 비행 제어 및 관제 시스템을 설계함에 있어, 지상 관제 시스템(GCS)과 상호작용하는 자동비행장치(Autopilot) 내부의 타이밍 예측 가능성은 매우 중요하다. PX4-Autopilot은 POSIX 호환 실시간 운영체제(RTOS)인 NuttX(펌웨어 v1.14 기준) 위에서 동작하며, 다양한 소프트웨어 모듈이 다중 스레드(Multi-thread) 환경에서 독립적인 태스크(Task)로 실행된다. 본 절에서는 NuttX RTOS의 선점형 스케줄링(Preemptive Scheduling) 환경에서 비행 제어 루프와 MAVLink 통신 태스크 간에 발생하는 스케줄링 지터(Jitter)를 수학적으로 모델링하고, 통신 지연에 미치는 영향을 분석한다.

1. 선점형 스케줄링 체계에서의 지터 발생 원인

PX4 시스템 아키텍처는 센서 데이터 처리, 자세 제어(Attitude Control), MAVLink 메시지 처리, 센서 로깅 등 수십 개의 태스크들이 각각의 고정 우선순위(Fixed Priority)를 할당받아 주기적으로 실행된다. 기본적으로 선점형 스케줄러(Preemptive Scheduler)는 현재 실행 중인 태스크보다 높은 우선순위의 태스크가 실행 준비(Ready) 상태가 되면 즉각적으로 CPU 제어권을 전환(Context Switch)한다.

이러한 선점 연산 과정에서 이상적인 호출 주기와 실제 호출 주기 간의 시간적 편차가 발생하는데, 이를 지터(Jitter)라고 정의한다. 지터의 주요 원인은 다음과 같다.

  1. 인터럽트 지연(Interrupt Latency): 하드웨어 타이머 혹은 센서의 외부 인터럽트(EXTI) 발생 시 처리가 지연되는 현상.
  2. 비선점 구역(Non-preemptive Section): 임계 구역(Critical Section) 보호를 위해 인터럽트가 비활성화된 구간에서 발생하는 대기 시간.
  3. 컨텍스트 스위칭 오버헤드(Context Switching Overhead): 레지스터(Register) 상태 저장 및 복구에 소모되는 하드웨어 의존적 시간.

이를 수식화하면, n번째 주기의 실제 태스크 실행 시간 t_{actual}(n)과 예상 실행 시간 t_{expected}(n)의 차이인 지터 J(n)은 다음과 같이 모델링할 수 있다.

J(n) = t_{actual}(n) - t_{expected}(n) = L_{int} + L_{csw} + W_{block}

(여기서 L_{int}는 인터럽트 지연, L_{csw}는 컨텍스트 스위칭 커널 비용, W_{block}은 더 높은 우선순위 태스크에 의한 블로킹 대기 시간이며, 제약조건 \vert J(n) \vert \le J_{max} 를 충족해야 시스템이 실시간성을 띠게 된다.)

2. NuttX RTOS 스케줄링의 선점(Preemption) 모델링 다이어그램

다이어그램에서 확인되듯, T\_Ctrl 모듈의 실행 시간이 길어지거나 너무 잦은 빈도로 인터럽트가 발생하게 되면 T\_Mav 태스크의 실행이 지연된다. 관제 관점에서는 이러한 MAVLink 송신 루프의 비주기성이 네트워크 병목(Bottleneck)이나 타임아웃 오탐(False Positive)으로 이어질 확률을 증가시킨다.

3. PX4-Autopilot과 Ardupilot 간의 스케줄링 및 지터 제어 차이점

두 오픈소스 생태계는 지터 관리 측면에서 설계 철학의 미세한 차이를 보인다.

  • Ardupilot(ChibiOS 기반): Ardupilot은 전통적으로 중앙 집중식(Monolithic)이거나 정적인 루프(Static Loop) 스케줄링 패턴을 선호해 왔으며, ChibiOS로 이식된 이후 스레드 모델을 사용하지만 여전히 Fast Loop 메인 제어문이 전체 시스템의 주기를 리드하는 형태를 일부 띠고 있다. 이로 인해 결정론적인(Deterministic) 설계가 용이하나, 시스템 오버헤드가 단일 루프에 집중될 경우 하위 우선순위의 텔레메트리 태스크 역시 예측 가능한 수준 안에서 지연된다.
  • PX4-Autopilot(NuttX 기반): PX4는 모듈화가 매우 뛰어나며 철저한 멀티 스레드 퍼블리시-서브스크라이브(uORB) 패턴을 기반으로 한다. 각 모듈이 완전히 독립된 태스크로 스케줄링되므로, CPU 자원 활용 효율은 높으나 유기적인 모듈 간 선점형 지터가 예측하기 어려운 패턴으로 전파될 가능성이 존재한다. 따라서 QGroundControl(v4.3)과 같은 GCS 시스템은 텔레메트리 타임스탬프(Timestamp)를 직접적으로 신뢰하기보다 수신측에서 스무딩(Smoothing)이나 이동 평균 필터(Moving Average Filter)를 적용하여 도착 지터를 최소화한다.

4. GCS 개발 및 ROS2 연동 측면의 시사점

커스텀 GCS 또는 자율 비행을 위한 오프보드(Offboard) 에지 컴퓨터를 설계할 무렵, 개발자는 MAVLink 패킷이 이산적인 시점(Discrete Time)에 항상 정확하게 도착하지 않음을 인지해야 한다. 선점형 RTOS 지터 모델링의 결과값을 토대로, 제어 패킷 분실 방지를 위해 앞선 절의 파레토 프론티어 곡선에서 도출된 최적의 적응형 타임아웃 주기를 이 최대 보장 지터(J_{max})보다 항상 길게 설정해야 한다.

또한 ROS2 연동(uXRCE-DDS)을 통한 고도의 자율 관제를 수행할 경우, XRCE-DDS Micro 에이전트 소프트웨어 자체가 NuttX 위에서 동작하는 태스크 스레드로 실행되므로, 이 에이전트의 프로세스 우선순위 및 스택 사이즈를 어떻게 튜닝하느냐에 따라 오프보드 제어(Offboard Control) 응답성의 한계가 시스템 구조적으로 결정된다.
