40.5.3.1. 뱅커 알고리즘(Banker’s Algorithm) 기반의 비동기 폴링(Polling) 스레드 자원 할당 검증

본 절에서는 지상 관제 시스템(GCS) 소프트웨어 내에서 다수의 비동기 폴링(Polling) 스레드들이 자원을 요청하고 반환하는 과정에서 교착 상태(Deadlock)를 원천적으로 예방하기 위해, 에츠허르 데이크스트라(Edsger W. Dijkstra)가 고안한 은행원 알고리즘(Banker’s Algorithm)을 어떻게 적용하고 검증할 수 있는지 상세히 논한다.

1. 개요 및 뱅커 알고리즘의 개념적 이해

뱅커 알고리즘(Banker’s Algorithm)은 운영체제에서 교착 상태를 회피하기 위해 널리 사용되는 대표적인 리소스 할당 및 결함 허용(Fault Tolerance) 알고리즘이다. 은행(운영체제 메인 컨트롤러 혹은 GCS 리소스 매니저)이 대출자(비동기 폴링 스레드)에게 자본(메모리, 파일 디스크립터, 소켓 포트 등)을 대출할 때, 남은 자원이 모든 대출자의 요구를 충족시켜 파산(교착 상태)에 이르지 않는 이른바 **안정 상태(Safe State)**를 유지할 수 있는지를 사전에 검증하는 수학적 기법이다.

QGroundControl(QGC)과 같은 현대 GCS 설계에서, 비디오 스트리밍 디코딩 스레드, 차량 텔레메트리 폴링 스레드, 오프보드 제어(Offboard Control) 명령 송신 스레드 등 다수의 시스템 스레드는 런타임 내내 동적으로 MAVLink 큐 리소스나 직렬 통신 락(Lock)을 지속적으로 요청한다. 이 때 자원 락(Lock)을 단순 반복적으로 부여하는 대신, 할당 직전에 뱅커 알고리즘을 수행하여 교착 상태 유발 가능성을 차단해야 한다.

2. GCS 비동기 폴링 스레드 환경과 자원 요구 모델

PX4-Autopilot과 통신하는 GCS 내부 아키텍처는 주기적으로 버퍼 상태를 질의(Polling)하는 비동기 워커들로 구성되어 있다. 자원 검증을 수행하기 위해 시스템의 전체 상태를 행렬(Matrix) 구조로 모델링 하라. 다음의 4가지 주요 자료구조가 필요하다.

  • Available[m]: m개 유형의 드론 관제 자원에 대한 현재 가용(비어있는) 인스턴스의 개수를 나타내는 1차원 벡터.
  • Max[n][m]: n개의 스레드(Process)가 각각 최대로 요구할 m개 유형 자원의 최대 수요량을 나타내는 행렬.
  • Allocation[n][m]: 현재 각 스레드에 할당되어 있는 자원의 개수를 나타내는 행렬.
  • Need[n][m]: 각 스레드가 작업을 완료하기 위해 추가로 필요로 하는 자원의 개수. 즉, Need[i][j] = Max[i][j] - Allocation[i][j].

2.1 자원(m) 종류 예시

  1. MAVLink 직렬화 버퍼 (Serialization Buffer)
  2. UDP 통신 소켓 멀티플렉서 락 (Socket Multiplexer Lock)
  3. UI 렌더링 동기화 큐 (UI Rendering Sync Queue)

3. 안정 상태(Safe State) 판별 및 자원 할당 알고리즘

뱅커 알고리즘의 핵심은 특정 스레드의 자원 할당 요청이 들어왔을 때, 해당 자원을 할당해 주었다고 가정하고 시스템이 안정 상태(Safe State)에 머무를 수 있는지 시뮬레이션(안정성 검사 알고리즘)하는 것이다.

스레드 T_i가 자원을 요청하는 벡터를 Request_i 라고 할 때, GCS 자원 스케줄러(Resource Scheduler)는 다음 검증 절차를 수행하라.

  1. 초기 검증: 만약 Request_i[j] <= Need[i][j] 라면 다음 단계로 진행하고, 아니라면 스레드가 기존에 통과했던 최대 요구량을 초과했으므로 오류 예외 처리(Exception)를 트리거하라.
  2. 가용 자원 검사: 만약 Request_i[j] <= Available[j] 라면 자원이 현재 존재하므로 다음 단계로 진행하고, 부족할 경우 자원이 확보될 때까지 스레드 T_i를 폴링 큐에서 대기(Wait) 상태로 전환하라.
  3. 가상 할당(Virtual Allocation): 요청을 수락했다고 가정하고 상태 벡터를 다음과 같이 갱신하라.
  • Available = Available - Request_i
  • Allocation_i = Allocation_i + Request_i
  • Need_i = Need_i - Request_i

이 가상 할당 상태에서 **안정성 검사 서브 알고리즘(Safety Check Algorithm)**을 실행하여 시스템의 교착 가능성이 없다고 수학적으로 증명(안정 상태 판별)되면 비로소 실제 자원을 할당한다. 만약 불안정 상태(Unsafe State)로 빠진다면 롤백(Rollback) 후 T_i를 대기시킨다.

    B -- No --> C[요구량 초과 예외 발생]
    B -- Yes --> D{Request_i <= Available ?}
    D -- No --> E[스레드 블로킹 대기 상태]
    D -- Yes --> F[가상 할당 상태 계산]
    F --> G{가상 상태가 Safe State인가?}
    G -- No --> H[상태 롤백 및 스레드 대기]
    G -- Yes --> I[실제 자원 할당 승인 및 락 취득]

4. GCS 개발 응용 시 제약 조건 및 튜닝 전략

뱅커 알고리즘은 수학적으로 완벽하게 교착 상태를 회피하지만, QGC나 사용자 정의 GCS 애플리케이션에 적용할 때에는 다음과 같은 GCS의 특성을 고려하여 커스터마이징하라.

  1. 자원 수 및 스레드 수의 정적 고정:
    뱅커 알고리즘은 다중 프로그래밍 환경에서 작업(Job) 수와 총 자원 수가 일정하다는 전제가 필수적이다. 따라서 GCS 백엔드의 스레드 풀(Thread Pool) 크기와 MAVLink 메시지 버퍼 객체의 수를 디자인 타임에 고정(Static Allocation)하라.
  2. 최대 요구량 예측의 오버헤드 완화:
    각 센서 텔레메트리 스레드가 요청할 ’최대 필요한 락(Lock)의 수’를 추정하는 것은 매우 까다로울 수 있다. 따라서 복잡한 파일 전송이나 미션 업로드 태스크(Mission Upload Task) 등과 같이 장시간 I/O를 점유하는 오퍼레이션(Operation)에만 선택적으로 핵심 자원에 대한 뱅커 알고리즘을 적용하고, 일반적인 짧은 하트비트 폴링에는 예외를 두는 하이브리드 전략을 채택하라.

5. 결론

Px4-Autopilot를 통제하는 지상 관제 시스템의 안정성은 단일 기체의 조작 편의성을 넘어, 군집 비행(Swarm Flight)이나 BVLOS(비가시권) 비행에서 발생하는 임무 크리티컬(Mission-Critical)한 상황의 운영 지속성을 담보한다.

뱅커 알고리즘은 GCS 내부의 비동기 폴링 스레드 간 발생하는 자원 경쟁(Race Condition) 시 선제적 자원 시뮬레이션을 통해 시스템의 정지 대란(Deadlock Freeze)을 예방하는 훌륭한 이론적 프레임워크를 제공한다. 향후 고성능 GCS 개발자는 QGC 아키텍처 내부의 뮤텍스(Mutex) 및 소켓 매니저 설계 시 이 검증 로직을 C++ 백엔드 미들웨어에 이식하여, 다중 연결 및 대용량 I/O 상황 속에서도 견고하게 동작하는 무중단 소프트웨어를 구축해야 한다.
