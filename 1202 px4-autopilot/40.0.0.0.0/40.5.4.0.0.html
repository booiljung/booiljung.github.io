40.5.4. 에포크(Epoch) 카운터 기반 장기 통신 단절 복구 시멘틱(Semantics)

본 절에서는 무인항공기(UAV)와 지상 관제 시스템(GCS) 사이의 텔레메트리 데이터 링크가 일시적이 아닌 ‘장기적으로’ 단절되었다가 복구되는 시나리오에서, 양측의 상태 머신(State Machine) 불일치로 인해 발생할 수 있는 크리티컬 결함을 방지하는 에포크 카운터(Epoch Counter) 구조와 동기화 시멘틱(Synchronization Semantics)에 대해 깊이 있게 다룬다.

1. 개요 및 장기 통신 단절의 위험성

비가시권(BVLOS, Beyond Visual Line of Sight) 비행이나 도심 항공 모빌리티(UAM) 환경에서는 지형지물, 전파 혼선, 기지국 핸드오버(Hand-over) 지연 등으로 인해 수십 초에서 수 분에 이르는 장기 통신 단절(Long-term Data Link Loss) 현상이 흔히 발생한다.

통신이 단절된 동안, PX4-Autopilot 보드 내부의 네이티브 상태 머신은 자체적인 Failsafe 로직(예: Return to Launch, Hold, Land 등)에 따라 비행 모드를 자율적으로 여러 차례 전이(Transition)시켰을 가능성이 높다. 반면, QGroundControl(QGC)과 같은 GCS는 통신이 끊어진 마지막 시점의 상태(Vehicle::armingState)를 버퍼에 캐싱한 채 멈춰 있게 된다.

통신이 기적적으로 재개(Reconnect)되는 순간, GCS가 과거의 낡은 상태 버퍼를 기준으로 기체에 MAVLink 제어 명령(예: COMMAND_LONG)을 재전송하거나 동기화 로직을 잘못 수행하면 기체는 조종자의 의도와 전혀 다른 치명적인 급격한 기동을 일으킬 수 있다.

2. 에포크(Epoch) 카운터의 개념 및 동작 원리

통신 재연결 직후 발생할 수 있는 “과거와 현재의 상태 교차” 문제를 해결하기 위해, 분산 컴퓨팅 및 데이터베이스 이론에서 활용되는 에포크 카운터(Epoch Counter) 또는 세대 카운터(Generation Counter) 개념을 관제 프로토콜 아키텍처에 도입하라.

에포크(Epoch)란 시스템이 특정 중대 이벤트(예: 부팅, Failsafe 시동, 장기 통신 단절 후 복구)를 겪을 때마다 단방향으로 1씩 증가하는 논리적 시계(Logical Clock)의 일종이다.

2.1 에포크 카운터 도입 시퀀스 모델

  1. 초기화: 기체가 부팅되거나 GCS와 최초 연결 시, 에포크 카운터 E_v(Vehicle)와 E_g(GCS)는 동기화되어 동일한 값(예: 1)을 갖는다.
  2. 단절 및 이벤트 발생: 장기 통신 단절 상태에서 PX4가 자율적으로 Failsafe(RTL 모드 전환 등)를 트리거하면, 기체는 내부 상태 버퍼를 플러시(Flush)하고 자신의 에포크 카운터 E_v를 1 증가시켜 새로운 세대(Generation)로 진입한다.
  3. 재연결 및 검증: 통신이 재개되면 기체는 하트비트(HEARTBEAT)나 확장 상태 메시지 릴레이 시 새로운 에포크 E_v 값을 동봉한다.
  4. 시멘틱 예외 처리: GCS는 수신된 E_v가 자신이 기억하고 있는 E_g보다 크다는 것을 감지한다. GCS는 통신 단절 기간 동안 자신이 큐잉(Queuing) 해두었던 모든 재전송 대기 보류 명령(Pending Commands)을 즉각 완전 폐기(Drop)하고, 기체의 새로운 최신 상태를 강제로 완전히 내려받는다(Full Sync).

3. PX4 및 GCS 연동을 위한 복구 시멘틱(Semantics) 설계

위와 같은 카운터를 실제 구현할 때에는 MAVLink 마이크로 서비스 모델에 에포크 기반의 유효성 검사 계층을 추가 설계해야 한다.

3.1 Stale Action 폐기 (Drop Stale Commands)

GCS 내 Vehicle 관리 매니저는 내부의 커맨드 큐(Command Queue) 요소에 GCS 에포크 카운터 E_g를 메타데이터로 부착(Tagging)하라. GCS가 네트워크 스레드를 통해 명령을 팝(Pop)하여 기체로 패킷을 전송하기 직전, 최신 E_v와 큐에 붙어있는 E_g를 비교한다.
E_g < E_v 이면 이 명령은 통신 단절 이전의 낡은(Stale) 명령이므로, 큐에서 빼내어 암묵적으로 폐쇄(Discard)하고 사용자 UI에 ‘통신 단절로 인한 명령 취소’ 경고를 발생시킨다.

3.2 상태 강제 재동기화 (State Force Re-Sync)

통신이 복구된 후 에포크 불일치를 발견한 GCS는 과거 통신 단절 직전의 궤적(Trajectory)이나 미션 진행도를 맹신해서는 안 된다.

  1. 기체의 현재 글로벌 좌표구속(Lat, Lon, Alt, Heading)을 즉각 최우선으로 리프레시한다.
  2. 진행 중이던 Waypoint 시퀀스를 조회(MISSION_REQUEST)하여 PX4 내비게이터가 현재 몇 번째 항목(Item)을 향해 비행 중인지 최적화하여 다시 캐싱한다.
  3. 모든 비행 전 안전 확인(Pre-flight Check) 파라미터를 다시 폴링한다.

4. QGroundControl 아키텍처 관점의 실무 예시

QGC v4.3.0 이상의 소스코드에서 Vehicle 싱글톤 객체는 MAVLink의 통신 품질 상실을 모니터링하기 위해 내부적으로 커넥션 매니저를 둔다. 에포크 메커니즘을 심화 적용하려면 단순히 연결이 끊겼다(LinkManager::linkInactive) 판단하는 것에 그치지 않고, 기체가 발송하는 확장 시스템 상태 바이트에서 재부팅 카운트 타임스탬프와 결합한 ’상태 천이 카운터’를 정의하여 수신 필터(Receive Filter)로 활용하는 C++ 백엔드 미들웨어를 구축해야 향고도 자율 임무의 신뢰성을 담보할 수 있다.

5. 결론

에포크 카운터 기반의 분산 시멘틱 모델 적용은 GCS와 PX4 에이전트 간의 **시공간적 시간 지연(Spatio-temporal Latency)**에 따른 위험 요소를 소프트웨어 논리 차원에서 봉쇄하는 매우 강력한 패러다임이다.

항공기 관제에서 가장 위험한 순간 중 하나는 기체의 실제 상태와 오퍼레이터 앞 스크린에 표시되는 GCS 상태 버퍼가 묵시적으로 불일치할 때이다. 장기 통신 단절 이후 복구 과정에서 강제적인 에포크 동기화 및 이전 명령의 강제 롤백(Drop) 처리를 적용함으로써, 관제 인터페이스의 무결성을 수학적, 구조적으로 완벽히 보장해야 할 것이다.
