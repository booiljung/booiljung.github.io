40.2.2.1. 생성 다항식(Generator Polynomial) 최적화를 통한 해밍 거리(Hamming Distance) 극대화 방안

1. 개요(Overview)

무인항공기 시스템(UAS)의 지상 관제 시스템(GCS)과 PX4-Autopilot 간의 데이터 링크(Data Link)는 물리 계층의 노이즈, 무선 전파 간섭 등으로 인해 비트 오류(Bit Error)에 취약하다. MAVLink 프로토콜은 기체 제어 명령, 센서 데이터, 텔레메트리(Telemetry)를 실시간으로 송수신하는 핵심 규격이다. 이러한 MAVLink 통신에서 전송 데이터의 무결성(Data Integrity)을 보장하기 위해서는 순환 중복 검사(Cyclic Redundancy Check, CRC)가 필수적으로 적용된다.

MAVLink CRC 프로세스에서 오류 검출 성능을 결정짓는 핵심 요소는 바로 **생성 다항식(Generator Polynomial)**과 이로 인해 확보되는 **해밍 거리(Hamming Distance)**이다. 해밍 거리는 송신된 유효한 코드워드(Codeword)들 사이의 최소 비트 차이를 의미하며, 해밍 거리가 d일 때 d-1개의 단일 비트 오류를 100% 검출할 수 있다. PX4-Autopilot(v1.14.0 기준)과 QGroundControl(v4.3 기준)의 구현에서 사용되는 MAVLink 기반 CRC-16 최적화 기법을 심층적으로 분석한다.

2. 생성 다항식(Generator Polynomial)의 수학적 기초

갈루아 필드(Galois Field) GF(2) 상에서 데이터 메시지는 계수가 0 또는 1인 다항식 M(x)로 표현된다. 체크섬을 생성하기 위해 데이터를 사전에 합의된 생성 다항식 G(x)로 모듈러 2 나눗셈(Modulo-2 Division)을 수행하며, 이때 발생하는 나머지(Remainder)가 바로 CRC 코드가 된다.

2.1 X.25 CRC-16-CCITT 표준

MAVLink는 ITU-T V.41에 정의된 CRC-16-CCITT 다항식을 사용한다. 이 다항식은 다양한 생성 다항식 중에서도 임베디드 통신 환경에서의 연산 효율성과 해밍 거리 특성이 우수하여 널리 채택되었다.
다항식은 다음과 같이 정의된다.

G(x) = x^{16} + x^{12} + x^5 + 1
이를 이진수로 매핑하면 1 0001 0000 0010 0001 (16진수 0x1021)이 된다. MAVLink 프로토콜은 패킷 구조의 메시지 ID 속성에 따라 1바이트의 추가적인 시드(CRC Extra)를 덧붙여 다형적인(Polymorphic) 해싱을 수행하며, 수신 측에서 메시지의 구조적 불일치를 추가적으로 방지한다.

2.2 데이터 페이로드(Payload) 크기에 따른 해밍 거리(Hamming Distance)

MAVLink v2의 최대 데이터 페이로드 크기는 255바이트이다. G(x) = x^{16} + x^{12} + x^5 + 1 다항식은 데이터 비트 길이에 따라 보장하는 해밍 거리가 달라진다.

  • 데이터 길이 \leq 32,767 비트: HD = 4를 보장 (최대 3개의 에러 비트율 검출)
  • MAVLink v2 패킷(최대 255바이트 = 2,040비트): 여유롭게 해밍 거리 4의 성질을 갖추어 3개의 임의 비트가 변형되는 모든 오류(Single, Double, Triple Bit Errors)를 탐지한다.

3.1 바이트 단위 룩업 테이블(Lookup Table) 기반 최적화

PX4-Autopilot 내부에서는 GF(2) 연산을 비트 단위의 시프트(Shift) 및 XOR 연산으로 1비트씩 수행하는 대신, 256개의 미리 계산된 16비트 정수 배열을 룩업 테이블로 선언하여 계산 속도를 높인다. 이 공간-시간 트레이드오프(Space-Time Tradeoff) 전략은 MAVLink 프로토콜의 C 헤더 라이브러리(mavlink_crc.h 내의 _crc_accumulate 매크로 구현) 깊숙이 내재되어 있다.

이 설계 방식은 임베디드 보드(STM32 기반 Pixhawk 호환 하드웨어 등)의 CPU 캐시 미스(Cache Miss) 확률을 낮추고, 수천 Hz 속도로 쏟아지는 자이로(Gyro)와 가속도계 센서의 uORB 메시지를 패키징할 때 병목 현상을 획기적으로 낮추어 준다. 반면, QGroundControl은 PC 리소스를 사용하므로 동일한 룩업 테이블 로직을 C++ 백엔드에서 병렬 처리 쓰레드 레벨로 구현하여 QSerialPort 버퍼로부터 들어오는 대용량 바이트 스트림을 고속으로 검증한다.

3.2 Ardupilot 대비 구조적 차이 비교

Ardupilot과 PX4-Autopilot 모두 MAVLink 라이브러리 표준을 준수하기 때문에 핵심적인 G(x) = x^{16} + x^{12} + x^5 + 1 다항식 수식 및 해밍 거리 특성 자체는 동일하다.
하지만 이를 활용한 관제 시스템의 텔레메트리 파서(Telemetry Parser) 구조에서는 차이를 나타낸다. Ardupilot은 서브 루틴 블록 내에서 포트 파싱 시 수개의 에러가 잡힐 경우, 재전송(Retransmission) 혹은 통신 대기 시간을 조절하는 헬퍼 함수와의 응집도가 강하다. PX4-Autopilot의 경우 uORB 구조에 의존적이므로, MAVLink 수신 모듈(mavlink_receiver.cpp)이 CRC 오류 로깅을 전담하여 vehicle_command uORB 토픽 배포 자체를 원천 차단하는 방식으로 완전히 분리(Decoupling)되어 있다. 이러한 구조 차이는 데이터 드롭 시 시스템의 반응 지연 시간에 영향을 미치게 된다.

4. 해밍 거리(Hamming Distance)와 생성 다항식 평가 지표

통신 시스템 엔지니어들이 설계 초기 단계에서 생성 다항식의 적합성을 평가할 때는 다음과 같은 성능 비용 모델(Performance Cost Model)을 활용한다.

  1. 버스트 오류 검출 길이(Burst Error Detection Length): 현재 MAVLink는 차수 16의 다항식을 채택하여 최대 16비트 버스트 손상을 절대적으로 식별한다.
  2. 미검출 에러 확률(Probability of Undetected Error, P_u): BER(Bit Error Rate) 모델 p에 대하여, 다항식 G(x)의 웨이트 분포(Weight Distribution) A_w를 통해 잔류 에러를 모델링한다.
    P_u(p) \approx \sum_{w=HD}^{N} A_w p^w (1-p)^{N-w}
    HD=4 이므로, 수식은 p^4 항에 의해 지배된다. 이는 무전기가 신호 대 잡음비(SNR) 컷오프 한계를 초과하는 극히 열악한 간섭 환경을 제외하고는 수학적으로 실시간 텔레메트리 루프 내에서 결함 코드워드가 정상으로 인식될 확률이 무시할 수 있을 만큼 작음을 입증한다(“Performance Analysis of CRC Codes”, Koopman, P. & Chakravarty, T.).

5. 결론 및 실무적 권고사항

PX4 기반의 자율 에이전트 개발 시, 하드웨어 계층의 시리얼 통신(UART/USB) 케이블 길이가 길어지거나 고전력 BLDC 모터 근처로 라우팅되어 EMI(전자기 간섭)가 발생할 때 MAVLink 트래픽 손실 빈도를 관찰해야 한다.
QGroundControl 내 MAVLink Inspector 패널에서 loss rate가 빈번하게 증가한다면, 이는 단순히 송수신 버퍼 오버플로우 트러블뿐만이 아니라 16비트 다항식의 해밍 거리가 감지하고 걸러낸(Drop) 패킷들의 파편을 의미한다.
따라서 다항식 연산으로 무결성이 1차 방어막 역할을 수행한다 하더라도, 물리적 쉴딩(Shielding) 보강과 전송 주파수 하향 조정을 통해 통신 채널 자체의 BER을 낮추려는 노력이 시스템 통합(System Integration) 단계에서 반드시 선행되어야 한다.
