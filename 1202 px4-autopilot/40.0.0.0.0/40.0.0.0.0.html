Chapter 40. GCS 제어 명령 타임아웃(Timeout) 및 재전송(Retry) 처리 기법

지상 관제 시스템(Ground Control Station, GCS)과 시스템 간의 신뢰성 있는 통신은 무인기 운영에서 핵심적인 역할을 한다. 무선 통신 환경에서는 필연적으로 노이즈, 다중 경로 페이딩, 신호 간섭 등으로 인해 패킷 유실이 발생할 수 있다. 이러한 패킷 유실 상황에서도 무인기가 안전하고 지속적으로 임무를 수행할 수 있도록 MAVLink(Micro Air Vehicle Link) 프로토콜은 명령어(Command) 수준에서의 강건한 상태 점검 및 재전송(Retry) 메커니즘을 규정하고 있다.

본 장에서는 PX4-Autopilot(Firmware v1.14 기준) 및 QGroundControl(v4.3 기준) 관제 소프트웨어 환경을 바탕으로, GCS에서 전송된 제어 명령의 타임아웃(Timeout) 조건과 이를 극복하기 위한 재전송 처리 기법을 깊이 있게 다룬다.

1. MAVLink 명령 프로토콜과 타임아웃 개요

MAVLink 기반 통신에서는 GCS가 무인기로 명령을 전달할 때 주로 COMMAND_LONG 혹은 COMMAND_INT 메시지를 사용한다. 명령을 수신한 시스템(PX4-Autopilot)은 명령의 수용, 거부, 실행 중 등의 상태를 GCS에 COMMAND_ACK(Acknowledge) 메시지로 응답해야 한다.

만약 GCS가 명령 송신 후 일정 시간(Timeout) 내에 COMMAND_ACK를 수신하지 못하면, 통신 유실 또는 일시적인 시스템 과부하로 간주하고 동일한 명령을 다시 전송(Retry)하는 정책을 실행한다. 이를 통해 통신 채널의 불안정성을 보상할 수 있다.

1.1 타임아웃 파라미터 설계의 고려사항

타임아웃(Timeout) 간격은 통신 매체의 종류(텔레메트리, Wi-Fi, LTE 등) 및 기체의 동역학적 특성에 따라 상이하게 설정되어야 한다. 너무 짧은 타임아웃은 불필요한 네트워크 트래픽 증가와 기체 처리 부하를 유발하며, 반대로 너무 긴 타임아웃은 비상 상황에서의 대응 지연을 초래할 수 있다. QGroundControl은 이러한 통신 지연시간(Latency)을 동적으로 모니터링하여 최적의 재전송 주기를 결정하는 동적 타임아웃 조정 기능을 일부 내장하고 있다.

2. QGroundControl 및 PX4의 재전송(Retry) 로직 아키텍처

QGroundControl(이하 QGC)의 내부 아키텍처에서 명령 관리자는 타임아웃 타이머(Timer)를 운용하여 전송된 패킷의 응답 여부를 추적한다.

2.1 재전송 상태 머신(State Machine)

재전송 처리는 유한 상태 머신(FSM, Finite State Machine)으로 모델링된다. GCS의 명령어 전송 및 타임아웃 처리 과정은 다음과 같이 요약할 수 있다.

stateDiagram-v2
    [*] --> Command_Send : 사용자 명령 발생
    Command_Send --> Wait_ACK : COMMAND_LONG 전송\n(타이머 시작)
    Wait_ACK --> Acknowledged : COMMAND_ACK 성공 수신
    Wait_ACK --> Wait_ACK : COMMAND_ACK (진행 중, IN_PROGRESS) 수신
    Wait_ACK --> Retry : 타임아웃 발생 (Timer Expired)
    Retry --> Wait_ACK : 재전송 (최대 N회)
    Retry --> Failed : 최대 재전송 횟수(Max Retries) 도과
    Acknowledged --> [*]
    Failed --> [*] : 에러 알림 및 상태 복구

위 도표에서 나타나듯, 타이머 이벤트가 발생하면 QGC는 사전 정의된 최대 재전송 횟수(예: 3~5회) 내에서 패킷 재전송을 시도한다. 만약 재전송 횟수를 모두 소진할 경우 통신 단절(Link Loss) 또는 제어 불가 상태로 간주하여 사용자 UI를 통해 경고를 발생시킨다.

3. 타임아웃 처리 로직의 상세 구현

3.1 명령 큐(Command Queue) 및 타이머 관리

C++ 스레드로 동작하는 QGC 백엔드(Vehicle Management)는 비동기적으로 메시지를 전송하고 응답을 대기한다. 이를 위해 각 명령에 대해 Transaction ID 혹은 메시지 시퀀스 넘버(Sequence Number)를 기록해 둔 명령 대기열(Command Queue)을 관리한다.

명령을 송신하면서 스레드는 특정 시간(예: 500 ms) 파라미터를 갖는 QTimer를 구동한다. 응답이 도달하면 큐에서 해당 타이머를 해제(Stop)하고 명령 객체를 파기한다.

// QGroundControl 내부 코드 비유 예시
void Vehicle::sendCommand(MAVLinkCommand* cmd) {
    _commandQueue.enqueue(cmd);
    cmd->startTimer(_timeoutMs); // 타임아웃 설정 (예: 500ms)
    _mavlink->send(cmd);
}

void Vehicle::onTimeout(MAVLinkCommand* cmd) {
    if (cmd->retryCount() < MAX_RETRIES) {
        cmd->incrementRetry();
        _mavlink->send(cmd);
        cmd->startTimer(_timeoutMs); // 타이머 재시작
    } else {
        emit commandFailed(cmd->type(), "Target did not respond");
        _commandQueue.dequeue(cmd);
    }
}

3.2 PX4-Autopilot의 ACK 처리 아키텍처

PX4 커맨더(Commander) 모듈은 COMMAND_LONG을 수신할 때 이를 즉각적으로 검증하고 반환할 수 있는 경우 MAV_RESULT_ACCEPTED나 MAV_RESULT_DENIED를 전송한다. 그러나 복잡한 구동기 초기화나 센서 캘리브레이션과 같이 시간이 소요되는 명령의 경우 MAV_RESULT_IN_PROGRESS를 지속적으로 반환하여 GCS의 타임아웃 타이머를 갱신(Reset)시킨다.

이를 통해 긴 시간이 필요한명령 수행 중에도 GCS가 통신 유실로 오인하여 명령을 재전송하는 오류를 방지할 수 있다.

4. ArduPilot과의 재전송(Retry) 처리 철학 비교

Ardupilot은 전통적으로 파라미터 및 웨이포인트 업로드 트랜잭션 등에서 재전송 횟수를 통신 연결의 품질 지표에 따라 비교적 융통성 있게 가변시키는 미들웨어를 도입했다. 반면, PX4-Autopilot은 엄격한 모듈형 구조를 지향하며, 타임아웃 시 즉각적인 오류 반환 및 시스템 상태 전이(State Transition) 메커니즘을 보다 타이트(tight)하게 유지하려는 경향이 있다.

PX4의 ROS 2 기반 Micro XRCE-DDS 브릿지 통신에서는 이러한 MAVLink 기반 재전송 로직 외에도 DDS 수준의 QOS(Quality of Service) 정책인 Reliability(RELIABLE/BEST_EFFORT)를 통하여 하위 계층에서 패킷 재접수를 원천적으로 제어할 수 있는 점이 Ardupilot 아키텍처와의 또다른 차별성이다.

5. 요약 및 고찰

GCS 제어 명령 타임아웃과 재전송 처리 기법은 항공기의 물리적인 제어 지연과 통신망의 확률적인 패킷 유실을 연결하여, 논리적인 강건성을 확보하는 핵심 설계이다. QGroundControl에서의 QTimer 기반 큐 관리와 PX4에서의 COMMAND_ACK 응답 메커니즘 간의 유기적인 상호작용은 시스템 엔지니어가 통신 프로토콜을 분석하고 성능을 튜닝하는 데 필수적인 통찰을 제공한다.
