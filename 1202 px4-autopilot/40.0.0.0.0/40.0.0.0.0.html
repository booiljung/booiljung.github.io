Chapter 40. GCS 제어 명령 타임아웃(Timeout) 및 재전송(Retry) 처리 기법

1. 개요

무인항공기(UAV) 운영 환경에서 지상 관제 시스템(Ground Control Station, GCS)과 PX4-Autopilot(비행 제어 소프트웨어) 간의 통신은 주로 가시거리(Line of Sight, LOS) 및 비가시거리(Beyond Visual Line of Sight, BVLOS) 무선 데이터 링크를 통해 이루어진다. 이러한 무선 채널은 다중 경로 페이딩(Multipath Fading), 전파 간섭(Interference), 물리적 차폐 등으로 인해 본질적으로 패킷 유실률(Packet Loss Rate, PLR)이 높을 수밖에 없다. 따라서 MAVLink 프로토콜을 사용하여 중요한 제어 명령(Control Command)을 전송할 때, 통신의 신뢰성을 확보하기 위한 타임아웃(Timeout) 및 재전송(Retry) 처리 기법은 필수적인 관제 요소이다.

본 장에서는 PX4-Autopilot(v1.14 이상) 및 QGroundControl(v4.3 이상)의 환경을 기준으로, 신뢰성 있는 명령어 전달을 위한 타임아웃 관리 및 재전송 알고리즘의 구조와 동작 원리를 상세히 분석한다.

2. 타임아웃(Timeout) 메커니즘의 정의와 설계

타임아웃은 GCS가 기체로 제어 명령을 보낸 후, 기체로부터 정상 수신을 의미하는 응답 패킷(Acknowledge, ACK)을 기다리는 최대 대기 시간을 의미한다.

2.1 MAVLink 명령어 전송 모델

GCS에서 PX4-Autopilot으로 전달되는 지시 사항은 주로 COMMAND_LONG 혹은 COMMAND_INT MAVLink 메시지를 통해 전달된다. 기체는 명령을 수신하면 하위 제어 모듈 및 상태 머신에서 조건 부합 여부를 판단한 후, COMMAND_ACK 메시지 내에 MAV_RESULT 코드(예: MAV_RESULT_ACCEPTED, MAV_RESULT_DENIED 등)를 담아 회신한다.

2.2 고정 타임아웃(Static Timeout)과 적응형 타임아웃(Adaptive Timeout)

GCS 관제 기술에서 타임아웃은 데이터 링크의 상태와 지연 시간(Latency)에 따라 유동적이어야 하나, 많은 구현에서는 기초적인 고정 타임아웃을 적용한다.

  • 고정 타임아웃: 사전에 정의된 일정 시간(예: 500ms \sim 2000ms)을 대기하며, 초과 시 타임아웃으로 간주하라.
  • 적응형 타임아웃: 왕복 지연 시간(Round Trip Time, RTT)을 주기적으로 측정하여 동적으로 타임아웃 값을 조절하라. (네트워크 환경이 열악한 BVLOS 환경에서 유리)

3. 재전송(Retry) 알고리즘의 동작 원리

타임아웃이 발생하면 GCS는 패킷이 유실되었거나 ACK 메시지가 유실된 것으로 간주하고 전송 절차를 재시도(Retry)한다. 이는 무한히 시도할 수는 없으므로 최대 재전송 횟수(Maximum Retry Count) 규정이 수반된다.

3.1 백오프(Backoff) 전략

재전송 시 대역폭 혼잡을 방지하기 위해 백오프 알고리즘을 적용할 수 있다.

  • 고정 재전송 간격: 매 재전송마다 동일한 타임아웃 타이머를 재설정하라.
  • 지수 백오프(Exponential Backoff): 재전송 횟수가 증가할수록 대기 시간을 지수 함수적으로 늘려 시스템 부하를 감소시키고 동적 혼잡(Congestion) 상태에서 안정성을 제고하라.

3.2 QGroundControl의 C++ 백엔드 로직

QGroundControl의 소스코드(Vehicle 매니저 객체 및 VehicleCommandManager 관련 클래스)에서는 MAVLink 명령 송신 시 명령어를 대기열(Queue)에 삽입하고, 타이머 이벤트를 통해 ACK 수신 여부를 점검한다. 타임아웃이 발생하면 설정된 _maxRetryCount 내에서 명령어를 재포장하여 텔레메트리 포트로 재전송한다.

4. Ardupilot 대비 PX4-Autopilot의 관제 차이점

PX4-Autopilot과 Ardupilot은 모두 MAVLink를 상위 통신 프로토콜로 사용하지만, 내부 타임아웃 구조와 재전송 요구사항에서 다음과 같은 차이가 있다.

  • 상태 전이 타이밍: PX4는 Commander 모듈 내 엄격한 상태 머신(State Machine)을 보유하고 있어, 모드 변경이나 시동(Arming) 시 명령어가 조건에 부합하지 않으면 즉각적으로 거부(DENIED) 메시지를 회신한다. 반면, 일부 Ardupilot의 분산화된 구조에서는 모듈 간 메시지 전달 지연으로 인해 ACK 응답 시간이 일시적으로 지연될 수 있어, GCS 개발 시 Ardupilot 타겟 설계보다 PX4 타겟 설계에서 타임아웃 마진(Margin)을 보다 촘촘하게 설정할 수 있다.
  • 다중 링크 동기화: PX4는 멀티 데이터 링크 환경 시 여러 경로(MAVLink 채널)를 통해 동일 명령이 중복 도달했을 경우의 중복 처리(De-duplication) 알고리즘이 간결히 최적화되어 있다. 따라서 GCS의 과도한 재전송이 기체 상태를 훼손하지 않도록 보장한다.

5. 지상 관제 시스템(GCS) 개발을 위한 실무 지침

서드파티(Third-party) 관제 애플리케이션을 개발할 때는 다음 사항을 준수하라.

  1. 멱등성(Idempotency) 보장: 어떠한 제어 명령이든 여러 번 반복해서 받더라도 상태가 안전하게 동일한 결과에 수렴하도록 시스템 파라미터를 설계하라.
  2. 명령어 분류: 비상 정지(Kill Switch)와 같은 안전 직결 명령어(Safety Critical Command)는 타임아웃 시간을 매우 짧게 가져가고 재전송 우선순위를 높여라. 반면 파라미터 요청 등은 타임아웃 시간을 길게 부여하여 대역폭 점유를 낮추어라.
  3. UI/UX 피드백 관리: 재전송이 발생하는 즉시 사용자 인터페이스(UI)에 네트워크 지연 상태를 시각적으로 피드백하고, 최종적으로 타임아웃에 도달하여 재전송 횟수를 초과한 경우 시스템 경고(Alert)를 발송하여 운영자가 수동 대처할 수 있도록 하라.
