40.5.2.1 방향성 비순환 그래프(DAG)를 이용한 명령어 시퀀스 의존성(Dependency) 역전 방지

무인항공기(UAV) 지상 관제 시스템(GCS)에서 기체로 전송하는 일련의 자율 비행 제어 명령(Command Sequence)은 고유한 선후 관계를 갖는다. 예를 들어, 기체는 이륙(Takeoff)을 한 뒤에만 임무(Mission)를 수행할 수 있고, 시동(Arming)이 걸려야만 이륙할 수 있다. 만약 멀티 코어 환경의 비동기적(Asynchronous) 네트워크 처리 과정에서 스레드 경쟁 상태(Race Condition)나 네트워크 지연으로 인해 이 순서가 뒤바뀌어 기체에 도달한다면, 이를 의존성 역전(Dependency Inversion) 현상이라고 한다.

본 절에서는 이러한 치명적인 명령어 시퀀스 의존성 역전을 원천적으로 방지하기 위해 큐잉 시스템(Queuing System) 내부에 방향성 비순환 그래프(DAG, Directed Acyclic Graph) 모델을 적용하여 명령의 논리적 흐름을 완벽히 통제하는 기법을 학습하라.

1. 명령어 의존성 역전 현상의 발생 원인

PX4-Autopilot과 QGroundControl 간의 MAVLink 통신은 UDP 기반으로 이루어지는 경우가 많아 패킷의 도착 순서(Ordering)가 보장되지 않는다. GCS 백엔드의 서로 다른 스레드에서 거의 동시에 여러 제어 명령을 비동기식으로 전송할 때, 의존성 역전 현상이 가장 빈번하게 발생한다.

의존성 역전 사례:

  1. 사용자가 GCS UI에서 ‘Arm and Takeoff’ 버튼을 클릭한다.
  2. GCS의 UI 스레드가 비동기 통신 큐에 MAV_CMD_COMPONENT_ARM_DISARM 명령(A)과 MAV_CMD_NAV_TAKEOFF 명령(B)을 거의 동시에 적재한다.
  3. 통신 모뎀의 채널 스케줄러 현상이나 수신 측 멀티 코어 FC(Flight Controller)의 스레드 스와핑에 의해 B 명령이 A 명령보다 먼저 파싱되어 PX4 Commander 모듈에 도달한다.
  4. 기체는 Arming되지 않은 상태이므로 이륙(Takeoff) 명령 B를 거부(RESULT_DENIED)한다.
  5. 직후 A 명령이 도달하여 기체는 모터 시동(Arming)만 걸린 채 지상에서 대기하는 비상식적인 상태에 빠진다.

Ardupilot의 기본 Mission Planner GCS는 스크립트 기반의 순차적(Synchronous) 딜레이(Delay) 기법(예: Arm(), Sleep(1000), Takeoff())으로 이 문제를 임시방편적으로 회피하는 경향이 있으나, 이는 네트워크 환경 변화에 취약하고 시스템 가용성을 크게 떨어뜨린다. PX4 관제 시스템은 수학적 데이터 구조를 통해 이 문제를 우아하게 해결해야 한다.

2. 방향성 비순환 그래프(DAG) 기반 명령 큐 구조 설계

이를 해결하기 위해 GCS 내부의 명령 스케줄러(Command Scheduler)를 단순한 FIFO(First-In-First-Out) 큐가 아닌 DAG를 평가하는 **위상 정렬 가능 큐(Topologically Sortable Queue)**로 재설계하라.

2.1 DAG의 노드와 간선 정의

  • 노드(Node): MAVLink 명령 구조체(예: CommandLong 또는 PX4 사용자 정의 동작). 시스템 상태를 변경하는 단일 원자적 작업(Atomic Operation).
  • 간선(Edge, u \rightarrow v): 명령 u가 완료(성공적인 COMMAND_ACK 수신 및 상태 반영)되어야만 명령 v를 전송할 수 있음을 의미하는 강한 선행 의존성(Strict Precedence Dependency).

위 그래프에서 이륙(Takeoff)을 위한 노드 D는 Arming(B)과 Target Alt 설정(C)이 모두 성공적으로 완료되어야만 진입 차수(In-degree)가 0이 되어 네트워크상으로 송출될 수 있다.

2.2 Kahn의 위상 정렬 알고리즘을 응용한 디스패처(Dispatcher) 구현

커스텀 GCS C++ 백엔드를 개발할 때, DAG 기반 디스패처 데몬 스레드를 다음의 논리로 구현하라.

  1. 초기화: 새 명령이 스케줄러에 추가될 때 자신이 선행해야 할 명령 목록(간선)을 명시한다.
  2. 진입 차수 산출: 각 노드(명령)의 진입 차수(In-degree, 선행 노드의 개수)를 계산하여, InDegree == 0인 명령들만 **발행 가능 큐(Ready Queue)**에 적재한다.
  3. 비동기 전송 및 트랜잭션 대기: Ready Queue의 명령을 PX4로 전송하고, 해당 노드를 ‘대기(Pending)’ 상태로 전환한다. 이 명령들은 서로 의존성이 없으므로 멀티 스레드 네트워크 송신기를 통해 병렬 전송이 가능하다.
  4. 의존성 해소(Dependency Resolution): PX4로부터 특정 명령에 대한 성공 ACK와 Heartbeat 기반의 상태 갱신이 확인되면, 해당 노드를 DAG에서 완전히 제거한다.
  5. 차수 갱신: 완료된 노드가 가리키고 있던 후행 노드들의 진입 차수를 1씩 감소시킨다. 만약 진입 차수가 0이 된 새로운 노드가 생기면 Ready Queue에 푸시(Push)하고 3번 과정을 반복한다.

3. 예외 처리와 트랜잭션 롤백(Rollback) 시맨틱

현실의 무인기 제어에서는 선행 명령이 실패할 가능성(예: Arming 중 센서 에러로 인한 RESULT_DENIED)을 염두에 두어야 한다. DAG 의존성 트리에 예외 발생 시나리오 처리를 어떻게 반영할 것인가가 핵심이다.

  1. 하위 트리 파기(Sub-tree Pruning): 선행 노드(Arming) 실패가 감지되면, 해당 노드에 도달 가능성(Reachability)이 있는 모든 하위 의존 노드(Takeoff, Mission Start 등)를 스케줄러에서 즉각 파기(Cancel)하라.
  2. 안전 상태 롤백(Rollback to Safe State): 부분적으로 성공한 노드들이 기체를 불안정한 상태로 유도할 수 있다면, 역방향(Reverse) DAG를 따라 복구 명령(예: Disarm 또는 RTL)을 역순으로 자동 삽입하는 보상 트랜잭션(Compensating Transaction) 패턴을 구현하라.

결론적으로, 다수의 MAVLink 명령이 복합적으로 얽힌 고급 자동화 관제 환경에서는 멀티 코어 및 비동기 네트워크 I/O의 불확실성에 의존하지 마라. GCS 백엔드의 워크플로우 엔진 내부에 위상 정렬이 보장된 DAG 스케줄러를 이식함으로써 어떠한 통신 상황에서도 기체로 향하는 명령어 의존성의 역전(Inversion)을 원천 차단해야 한다.
