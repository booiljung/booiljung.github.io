40.2.3. 유한 상태 기계(FSM)의 상태 전이 위상 공간(Topological Space) 매핑

1. 개요(Overview)

무인항공기 시스템(UAS)과 지상 관제 시스템(GCS) 사이의 명령 및 제어(Command and Control, C2) 아키텍처는 근본적으로 복잡한 의사 결정 시스템이다. 기체의 비행 모드 전환, 시동(Arming), 임무 수행 및 비상 복구 절차는 모두 특정 조건과 시간의 흐름에 따라 변화하는 이산적인 시스템 상태(Discrete System State)의 연속적 변화로 규정된다.
이러한 시스템의 동작 행위를 수학적으로 명확하게 규명하기 위해 유한 상태 기계(Finite State Machine, FSM) 모델이 핵심적으로 사용된다. FSM 모델론에 위상 공간(Topological Space)의 개념을 결합하면 FSM의 수많은 상태 변수들이 형성하는 집합(Set) 모델 내부에서 상태들 간의 연결 관계(Connectivity)와 허용된 전이 궤적(Allowed Transition Trajectory)을 체계적으로 매핑(Mapping)할 수 있다. 본 문서에서는 PX4-Autopilot(v1.14.0 기준) 펌웨어 및 QGroundControl(v4.3 기준)의 소스 코드 수준에서, 이 위상 공간을 통제하는 MAVLink 명령어 처리 로직과 상태 매핑 함수들을 심층적으로 분석한다.

2. PX4 비행 제어기(Flight Controller)의 FSM 위상학적 구조

위상 공간 수학에서 상태(State) 노드들 간의 유효한 거리는, 물리적 거리가 아니라 ’도달 가능성(Reachability)’으로 정의된다.
PX4의 심장부인 커맨더(Commander) 모듈(src/modules/commander/Commander.cpp)은 수만 줄의 코드와 거대한 switch-case 블록으로 구성된 메가 FSM 팩토리이다. 이 FSM의 위상 공간은 단일 변수가 아닌, 수집된 센서 건강 상태(sensor_combined), 위치 추정기 상태(vehicle_local_position), 전원 관리(battery_status) 등의 여러 차원이 결합된 다차원 유향 그래프(Directed Graph)이다.

2.1 상태 차원의 직교성(Orthogonality)과 분리 설계

PX4는 비행 모드 공간(Navigation State)과 시동 공간(Arming State)을 위상적으로 분리하여 관리한다. 이는 특정 상태의 변화가 의도치 않게 다른 안전망을 무너뜨리는 현상을 방지하는 핵심 설계 철학이다.

  • 시동 상태(Arming State): Init, Standby, Armed, Standby_Error 등의 상태를 가지며 프로펠러 회전 및 동력 인가 여부를 결정한다.
  • 주행 모드(Navigation State): Manual, Altitude, Position, Mission, RTL 등으로 기체 제어 권한의 주체를 구분한다.

이 두 FSM 위상 차원의 결합으로 인해 “기체가 Armed 된 상태(차원 1)에서만 Auto-Mission 모드(차원 2)로 전이할 수 있다“는 식의 교차 제약 조건(Cross-constraint)이 Commander::set_main_state()와 Commander::arm() 루틴 내에 강력하게 명시되어 있다.

3. QGroundControl 동기화 FSM 위상 편이(Phase Shift) 관리

QGroundControl(GCS)은 내부에 독자적인 Qt 기반의 Vehicle 클래스를 C++ 인스턴스로 보유하며, 이 클래스는 원격 기체의 상태 FSM 거울상(Mirror Image) 프로토타입을 형성한다. 기체(PX4) 측의 FSM 상태가 전이되면 MAVLink의 HEARTBEAT 및 SYS_STATUS 메시지를 통해 GCS로 상태 변화 브로드캐스트가 날아간다.

문제는 무선 통신 레이어의 자체적인 레이턴시(Latency) 및 드롭아웃(Dropout)으로 인하여, 실제 하늘을 나는 PX4의 물리적 FSM 위상 위치와 지상 QGC 터미널에 렌더링 된 위상의 캐시(Cache) 위치 사이에 시공간적 불일치가 생긴다는 점이다. 이러한 위상 편이 현상을 해결하기 위해 QGC는 상태 명령 발송 후 응답(ACK)이 올 때까지 타임아웃 타이머 기반의 “가상(Virtual) 전이 대기 공간“이라는 중간자 모델을 적용한다. ACK를 수신하지 못하면 GCS의 해당 FSM 모델은 원래의 상태 노드로 되돌아오는 복원력(Resilience) 패턴을 보유하게 된다.

4. Ardupilot 대비 아키텍처 매핑 차이

Ardupilot 통제 모델과 기능 관점에서는 거의 유사하지만, 내재적 FSM 통신 파이프라인에서 차이가 있다.
Ardupilot의 경우 Copter/mode.cpp 파생 클래스 내에서 제어 루프가 실행될 때, 해당 모드의 진입(Enter) 및 탈출(Exit) 과정 자체를 함수 포인터 다형성(Polymorphism)으로 객체 지향적으로 처리하는 경향이 크다. 즉 클래스 인스턴스의 변환이 곧 상태 위상의 변환이다.
반면 PX4 펌웨어는 uORB 메시지 브로드캐스팅 아키텍처에 근간을 두고 있다. 커맨더 모듈이 모든 전제 조건(Pre-flight Check 등)을 평가하여 FSM의 다음 위상을 승인하면 vehicle_status uORB 토픽을 발행(Publish)한다. 이 토픽을 구독(Subscribe)하고 있던 각각의 FlightTask 및 Navigator 모듈이 메시지 수신 이벤트에 트리거되어 자신들의 로컬 제어기를 해당 위상에 맞게 스위칭하는 탈중앙화된(decentralized) 동기화 파급 방식을 취하고 있다.

5. MAVLink 매핑 프로세스와 도달성 검증(Reachability Check)

아래 도표는 시스템이 STANDBY에서 ARMED 상태로 매핑(전이)되기 위해 요구되는 위상 공간의 허들(Hurdle)들을 나타낸다.

stateDiagram-v2
    direction TB
    state "초기화 위상 (INIT)" as Init
    state "대기 위상 (STANDBY)" as Standby
    state "무장 위상 (ARMED)" as Armed
    state "거절 위상 (DENIED)" as Denied
    
    Init --> Standby : 센서 캘리브레이션 및 시스템 부팅 완료
    
    Standby --> Check_Conditions : ARM 커맨드 수신 (MAVLink / 조종기)
    
    state Check_Conditions <<choice>>
    Check_Conditions --> Armed : [Pre-flight Check 패스] AND [GPS Lock]
    Check_Conditions --> Denied : [조건 미달] 센서 에러, 배터리 부족
    
    Denied --> Standby : GCS 경고 표출 후 재대기
    
    Armed --> Standby : DISARM 커맨드 수신 또는 착륙 감지
    Armed --> Emergency : 심각한 하드웨어 결함 발생 시

위의 위상 궤적에서 보듯이 FSM 토폴로지는 특정 노드들로 매핑되는 단방향/양방향 엣지를 엄격히 제한한다. MAVLink MAV_CMD_COMPONENT_ARM_DISARM 명령 패킷이 기체 수신 단에 접수되면 Commander::handle_command() 루틴 내에서 TransitionResult 열거형을 반환하는 거대한 내부 매퍼 루틴 arming_state_transition()이 호출된다.
해당 함수 코드 블록 내부는 조건 검사(Condition Checking)의 긴 파이프라인이며, 단 하나의 평가 항목이라도 부정할 경우 거절(Denied) 상태를 리턴하여 위상 공간 내 목적지 노드로의 진입을 수학적으로 차단한다.

6. 결론

유한 상태 기계(FSM)의 위상 공간을 MAVLink 명령어 로직과 연동하여 분석함으로써, 수많은 조건 변수들이 얽혀있는 PX4 비상 상황 대처 로직의 무결성을 더욱 체계화할 수 있다. 고급 자율 에이전트를 개발하거나 QGroundControl과 완전히 다른 독자적인 GCS 백엔드를 구축할 코어 개발자라면, PX4 펌웨어 내부의 vehicle_status 차원과 시동 제어 차원의 분리형 설계 의도를 명확히 파악하고, 각 명령어 패킷이 두 위상 공간 사이의 어떠한 연결성을 촉발하는지를 검증(Validation)하는 작업에 매진해야 한다. (“Formal Methods in Systems Engineering”, Clarke, E. M. 외 참고). 이는 궁극적으로 시스템 내의 잔류 오작동 구역(Deadlock 또는 Unreachable State)을 제거하여 항공 안전성 및 신뢰성 인증을 획득하는 가장 근본적인 토대가 된다.
