40.2.3. 유한 상태 기계(FSM)의 상태 전이 위상 공간(Topological Space) 매핑

비행 제어 시스템(Flight Control System)의 코어 두뇌라 할 수 있는 PX4-Autopilot의 커맨더(Commander) 모듈은 수많은 비행 모드와 시동(Arming), 페일세이프(Failsafe) 상태를 관리하기 위해 유한 상태 기계(FSM, Finite State Machine) 아키텍처를 채택하고 있다. 복잡한 무인기 시스템에서 단순히 “현재 모드가 무엇인가?“를 정의하는 것을 넘어, “어떤 상태에서 어떤 상태로 전이(Transition)가 허용되는가?“를 수학적이고 구조적으로 통제하는 기법이 바로 상태 전이 위상 공간(Topological Space) 매핑이다. 본 절에서는 PX4의 상태 머신을 기하학적 위상 공간으로 모델링하고 제어 흐름의 무결성을 확보하는 방안을 고찰한다.

1. FSM 위상 공간(Topological Space)의 수학적 토대

위상 공간 매핑 모델에서 비행 제어기의 각 고유 상태(State: 예 - Standby, Armed, Takeoff, Mission, RTL)는 위상 공간 내의 하나의 노드(Node) 또는 정점(Vertex)으로 정의된다. 그리고 MAVLink 명령(QGroundControl의 제어 인가)이나 센서 기반의 이벤트(예: 배터리 소진, 데이터 링크 단절)는 이 노드 사이를 잇는 유향 간선(Directed Edge)으로 맵핑된다.

PX4-Autopilot 커맨더의 수학적 FSM 위상 공간 T는 (S, E, P)의 집합으로 정의할 수 있다.

  • S: 유효한 비행 및 시스템 상태의 집합 (Nodes)
  • E: 센서 조건 및 MAVLink 입력에 따른 전이 함수 방아쇠 (Edges)
  • P: 특정 노드에서 다른 노드로 넘어가기 위해 반드시 만족해야 하는 전제 조건 (Pre-conditions, 예를 들어 Pre-flight Check 통과 여부)

이러한 위상 모델링을 통하면, 소프트웨어 엔지니어는 시스템 내에 결코 도달해서는 안 되는 **절대 금지 경로(Forbidden Path)**를 사전에 차단할 수 있다. 예를 들어 기체가 공중에 있는가(condition_system_in_air_restore)를 무시하고 Standby 모드에서 곧바로 Mission 모드로 강제 워프(Warp)하는 비상식적 명령은 위상 공간 트리 단절 원리에 의해 원천적으로 무효화된다.

2. PX4-Autopilot의 커맨더(Commander) FSM 아키텍처

PX4 커맨더 모듈 소스 코드(예: commander.cpp 및 내부 상태 평가기)는 MAVLink로 들어오는 모든 제어 요청을 위상 공간상의 ’방향 벡터’로 취급한다.

2.1 Ardupilot과의 FSM 설계 철학 차이

Ardupilot 생태계(Copter, Plane, Rover 등)는 전통적으로 각 기체 형태(Vehicle Type)별 메인 스케줄러 내부에 set_mode() 함수를 두어, 다소 모놀리식(Monolithic)하게 각 비행 모드의 조건을 개별 검사하는 경향이 짙었다. 반면 PX4-Autopilot의 경우, 액추에이터 및 제어 믹서(Mixer)와는 철저히 분리된 거대한 독립 FSM 상태 기계 모듈(Commander)을 운용한다. 모든 내비게이션 상태(nav_state)와 시동 상태(arming_state)의 전이는 오직 이 단일 커맨더의 위상 검증 회로를 거쳐야만 uORB 토픽(vehicle_status)으로 퍼블리싱(Publishing)된다. 이로 인해 코드 추론이 기하급수적으로 명확해지는 장점을 지닌다.

3. QGroundControl 명령과 상태 전이 위상도 (Mermaid)

QGroundControl과 같은 지상 관제 시스템이 발송하는 MAV_CMD_DO_SET_MODE 또는 MAV_CMD_COMPONENT_ARM_DISARM 명령은 PX4 위상 공간 내에서 특정 방향으로의 이동을 촉구하는 외부 에너지원과 같다.
아래 다이어그램은 기본적인 비행 프로파일에 대한 PX4의 핵심 유한 상태 기계 위상 공간도를 나타낸다.

    direction LR

    %% 노드 정의
    state "초기화 (Booting)" as Boot
    state "대기 (Standby / Disarmed)" as Standby
    state "시동 (Armed / Ground)" as Armed
    state "이륙 모드 (Takeoff)" as Takeoff
    state "임무 모드 (Mission / Auto)" as Mission
    state "복귀 모드 (RTL)" as RTL
    state "착륙 모드 (Land)" as Land
    
    %% 트랜지션(Edges) 및 전제조건
    [*] --> Boot
    Boot --> Standby : 부팅 및 초기화 완료
    Standby --> Armed : MAV_CMD_ARM_DISARM (Pre-flight 검증 통과)
    Armed --> Standby : 시동 해제 명령 / Timeout
    
    Armed --> Takeoff : 이륙 명령 인가
    Takeoff --> Mission : 이륙 고도 도달 (Altitude Reached)
    
    Mission --> RTL : 임무 완료 (Mission Complete) / 데이터 단절 / 배터리 경고
    RTL --> Land : 홈 포인트 도달
    Takeoff --> Land : 비상 상황 (Failsafe 트리거)
    Land --> Standby : 지면 접촉 (Landed 감지 후 자동 Disarm)

위의 다이어그램에서 보듯, Standby 상태에 있는 기체가 Mission 스페이스로 진입하려면 반드시 Armed 정현점과 Takeoff 정현점을 논리적으로 거치거나 물리적 이륙 감지 트리거를 갱신해야만 한다. 위상학적으로 이어져 있지 않은 두 공간으로의 도약은 펌웨어 레지스터 단에서 치명적 결함(Fatal Error)으로 규정된다.

4. 커스텀 비행 모드 개발 시 FSM 맵핑 가이드라인

PX4-Autopilot 기반의 새로운 자율 비행 알고리즘을 추가하고자 하는 ROS2 개발자나 C++ 임베디드 코어 모듈 개발자는, 새로운 제어 노드를 위상 공간에 맵핑할 때 다음의 룰을 반드시 준수하라.

  1. 내비게이션 상태(Navigation State) 등록: vehicle_status_s uORB 구조체 내에 고유한 NAVIGATION_STATE_CUSTOM_XXX 상수를 할당하여 위상 공간에 새로운 물리적 점(Point)을 창조하라.
  2. 전이 벡터(Transition Vector) 유효성 평가: commander.cpp의 main_state_transition 함수 로직에 개입하여, 어떤 이전 모드(Previous Mode)에서 커스텀 모드로 진입이 가능한지 수학적 조건식 분기를 엄격하게 명세하라. 예를 들어, 새로운 ’로봇팔 포획 모드’를 만들었다면, 기체가 공중에 부양한 상태(In-air)에서만 이 모드로 진입할 수 있도록 락(Lock)을 걸어야 한다.
  3. 페일세이프 엣지(Failsafe Edge) 생성: 새로 만든 상태(Node)에서 통신이 두절되거나 센서 오류가 발생했을 때, 기체가 즉시 RTL이나 Land 공간으로 후퇴할 수 있도록 비상 탈출 간선(Escape Edge)을 반드시 라우팅하라.

5. 결론

유한 상태 기계(FSM)의 위상 공간 매핑은 단순한 if-else 조건문의 나열이 아니다. 이는 무인 항공기와 같은 복잡계(Complex System) 소프트웨어에서 오직 정의된 논리적 시퀀스로만 기체가 동작하도록 강제하는 수학적 차단벽(Mathematical Barrier)이다. PX4-Autopilot의 FSM 아키텍처를 정확히 이해하고 QGroundControl 백엔드 로직에 반영하는 것은, 예기치 않은 MAVLink 명령 폭주나 채널 오류 속에서도 기체의 거동 시퀀스를 100% 예측 가능하게 묶어두는 최고의 안전 확보 기술이다.
