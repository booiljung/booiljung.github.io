40.4.4.1.1. 로빈 후드 해싱(Robin Hood Hashing) 프로빙(Probing)을 통한 최악 시간 복잡도 상한 보장

무인 항공기(UAV) 지상 관제 시스템(GCS) 및 ROS2 기반 자율 에이전트(Autonomous Agent) 아키텍처에서 MAVLink 패킷을 처리할 때, 완벽 해싱(Perfect Hashing)을 입증하거나 컴파일 타임에 모든 시스템 토폴로지를 예측하는 것은 현실적으로 불가능한 경우가 많다. 특히 임무(Mission) 중 새로운 센서 페이로드나 동적 드론이 추가되는 환경에서는 충돌(Collision)을 허용하면서도 그 처리 시간의 엣지 케이스(Edge Case)를 엄격히 제어할 수 있는 개방 주소법(Open Addressing) 기반의 해시 테이블이 필수적이다. 본 절에서는 이런 환경에서 GCS 백엔드의 최악 시간 복잡도(Worst-case Time Complexity) 상한을 보장하기 위해 도입되는 로빈 후드 해싱(Robin Hood Hashing) 프로빙(Probing) 기법의 수학적 원리와 그 적용 방안을 다룬다.

1. GCS 환경에서의 전통적 선형 프로빙(Linear Probing)의 한계

동적 메모리 할당을 최소화해야 하는 임베디드 리눅스(Embedded Linux) 기반의 컴패니언 컴퓨터나 고성능 GCS 백엔드는 체이닝(Chaining) 대신 개방 주소법(Open Addressing)을 선호한다. 그중 가장 단순한 선형 프로빙(Linear Probing)은 충돌 발생 시 바로 다음 버킷(Bucket)을 확인하여 빈자리에 데이터를 삽입하는 방식이다.

하지만 군집 드론 환경에서 불특정 다수의 MAVLink 식별자(System ID, Component ID)가 한 버킷 근처로 군집화(Primary Clustering)되는 현상이 발생하면 문제가 심각해진다. 특정 기체의 로그 메시지를 처리하기 위해 해시 테이블을 순회할 때, 충돌 군집(Cluster)의 길이가 길어지면 최악의 경우 \mathcal{O}(N)의 시간 복잡도가 발생한다. GCS 입장에서는 평균 \mathcal{O}(1)의 처리 속도도 중요하지만, 단 한 번의 프레임 드롭(Frame Drop)이나 타이밍 위반 결함(Timing Violation Deadline)이 기체 제어 상실로 직결되므로 “가장 오래 걸리는 탐색 시간(Worst-case)“을 상한선 내로 억제하는 것이 훨씬 더 중요하다.

2. 로빈 후드 해싱(Robin Hood Hashing)의 수학적 원리

로빈 후드 해싱은 선형 프로빙의 변형으로, “부자에게서 빼앗아 빈자에게 준다(Take from the rich and give to the poor)“는 철학을 자료구조에 적용한 알고리즘이다. 여기서 ’부자’란 이상적인 해시 위치(Ideal Hash Position)에 가깝게 배치된(탐색 거리가 짧은) 데이터를 뜻하고, ’빈자’란 이상적인 위치에서 멀리 밀려난(탐색 거리가 긴) 데이터를 뜻한다.

각 버킷에 데이터 식별자와 함께 PSL (Probe Sequence Length), 즉 원래 자신이 있어야 할 이상적인 인덱스로부터 얼마나 멀어져 있는지를 기록한다.

새로운 MAVLink 핸들러 블록이나 기체 상태 레코드를 삽입할 때의 로직은 다음과 같다:

  1. 삽입할 요소의 초기 PSL은 0이다.
  2. 테이블을 선형 탐색하며 빈자리를 찾는다. 탐색할 때마다 삽입할 요소의 PSL을 1씩 증가시킨다.
  3. 만약 탐색 중 이미 채워진 버킷을 만났는데, 그 버킷에 있는 기존 요소의 PSL보다 현재 삽입하려는 요소의 PSL이 더 크다면(즉, 현재 요소가 더 ’빈자’라면), **두 요소의 자리를 교환(Swap)**한다.
  4. 이제 기존 요소(밀려난 요소)를 쥐고 다시 다음 버킷으로 탐색을 계속하며 빈자리를 찾을 때까지 이 과정을 반복한다.

이 교환(Swap) 매커니즘 덕분에, 해시 테이블 내 모든 요소들의 PSL 편차(Variance)가 극단적으로 작아진다. 특정 요소만 억울하게 탐색 거리가 길어지는 현상이 사라지며, 테이블의 적재율(Load Factor)이 90%에 달하더라도 최대 탐색 거리(Maximum PSL)가 \mathcal{O}(\log N) 수준 혹은 좁은 상한선(Upper Bound) 내로 억제되는 수학적 특성을 지닌다.

3. MAVLink 라우터 아키텍처에의 적용 및 성능 이점

QGroundControl(v4.3 기준)이나 고수준 C++ MAVLink 라우터(Router) 모듈에 이 로빈 후드 기반의 해시 맵을 적용하면 시스템 예측 안정성(Predictability)이 극대화된다.

graph TD
    A[MAVLink Packet Rx Thread] --> B{Calculate Hash Index h(k)}
    B --> C[Fetch Bucket at h(k)]
    C --> D{Is Bucket Empty or Match?}
    D -- Match/Empty --> E[O(1) Processing & Exit]
    D -- Collision --> F[Start Robin Hood Probing]
    F --> G{Current PSL > Target PSL?}
    G -- Yes --> H[Swap Local State & Continue Search]
    G -- No --> I[Move to Next Bucket (PSL++)]
    H --> I
    I --> D

이 알고리즘을 GCS 환경에 적용했을 때의 구체적 이점은 다음과 같다:

  1. 최악의 지연시간(Worst Latency)의 절대적 상한 보장: CPU 캐시 라인(Cache Line) 크기에 맞춰 버킷 배열을 정렬(Alignment)하면, 최대 PSL이 6~10 이내로 통제되므로 어떠한 충돌 상황에서도 L1/L2 메모리 캐시 미스(Cache Miss)가 1~2회로 제한된다. 결과적으로 패킷 처리 스레드가 절대 멈추지 않는다(Non-blocking & Predictable).
  2. 조기 탐색 종료(Early Exit): 요소를 검색할 때, 현재 버킷에 들어있는 요소의 PSL보다 내가 찾고자 하는 요소가 누적한 PSL이 더 작아지는 순간이 오면, “이 테이블 안에는 해당 요소가 절대 존재하지 않는다“고 확신하고 즉각 \mathcal{O}(1) 에 가깝게 탐색을 종료(Fail-fast)할 수 있다. 이는 존재하지 않는 시스템 ID 패킷(예: 해킹 시도 또는 잘못된 네트워크 라우팅)을 거를 때 막대한 연산력 절감을 가져온다.

4. 결론 및 실무 시사점

자율 주행 에이전트와 수십 대의 군집 드론을 단일 워크스테이션에서 통합 관제(Centralized Control)하기 위해서는, 단순한 ‘빠른’ 백엔드를 넘어 ‘지연의 최대치가 보장된(Bounded Latency)’ 아키텍처를 설계해야 한다.

완벽 해싱(Perfect Hashing)이 메모리 소모나 컴파일 타임의 제약으로 동적 드론 추가 시나리오에서 한계를 보인다면, 로빈 후드 해싱(Robin Hood Hashing) 프로빙은 실용적으로 가장 완벽한 대안이 된다. 이는 소프트웨어 엔지니어가 C++ 자료 구조 레벨에서 데이터 편향성(Variance)을 억제함으로써, GCS의 메인루프 및 ROS2 발행(Publish) 병목을 제어 이론 관점의 수렴 안정성처럼 확보할 수 있음을 증명한다.
