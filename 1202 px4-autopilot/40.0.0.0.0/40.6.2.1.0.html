40.6.2.1. 시그널-슬롯(Signal-Slot) 델리게이트 큐잉의 메시지 패싱(Message Passing) 오버헤드 추정

QGroundControl(QGC)과 같이 고빈도 통신과 복잡한 프론트엔드 UI를 동시에 처리하는 데스크톱/모바일 애플리케이션에서는 스레드 간 교착 상태(Deadlock)를 회피하기 위해 Qt::QueuedConnection을 통한 비동기 델리게이트(Delegate) 큐잉 기법이 필수적으로 사용된다. 즉, MAVLink 네트워크 스레드는 데이터를 파싱한 후 메인 스레드에 직접 접근하지 않고, 메인 스레드의 이벤트 큐(Event Queue)에 메시지를 “패싱(Passing)“하는 방식으로 데이터를 위임한다.

본 절에서는 이러한 다중 스레드 환경에서 시그널-슬롯을 이용한 메시지 패싱(Message Passing) 오버헤드를 정량적으로 추정하고, 아키텍처 관점에서 어떠한 성능 지연(Latency)이 발생하는지 상세히 규명한다.

1. 메시지 패싱(Message Passing) 메커니즘 개요

동일한 스레드 내에서의 시그널-슬롯 호출(Qt::DirectConnection)은 근본적으로 연속된 함수 호출에 불과하여 \mathcal{O}(1)의 상수 지연만 발생한다. 그러나 송신자와 수신자의 스레드가 다를 경우, Qt는 스레드 안전성(Thread Safety)을 보장하기 위해 자동으로 큐 연결(Queued Connection) 메커니즘으로 전환한다.

큐 연결이 동작할 때 발생하는 일련의 과정은 다음과 같다.

  1. 파라미터 복사 (Argument Deep Copy): 발송 스레드는 전달할 데이터(인자)를 스레드 안전한 새로운 힙(Heap) 메모리 할당 공간에 깊은 복사(Deep Copy)한다. (QMetaCallEvent 생성)
  2. 이벤트 큐 락(Lock) 획득 대기: 수신 스레드(보통 메인 UI 스레드)의 이벤트 큐에 메시지를 삽입하기 위해, 내부적으로 OS 레벨의 뮤텍스(Mutex) 또는 스핀락(Spinlock) 매커니즘을 가동하여 락을 획득한다.
  3. 이벤트 삽입 및 스레드 깨우기: 복사된 이벤트 객체를 큐에 편입시킨 뒤, 수신 스레드가 Sleep 상태라면 조건 변수(Condition Variable) 등을 통해 커널에 OS 문맥 교환(Context Switch)을 요청하여 스레드를 깨운다(Wake-up).
  4. 역직렬화 및 슬롯 실행: 수신 스레드는 자신의 이벤트 루프(QCoreApplication::exec()) 블록 내에서 큐를 비우면서(Dequeuing), 복사해 둔 파라미터를 역직렬화(Unpacking)하여 최종적으로 타겟 슬롯 함수를 실행한다.

2. 메시지 패싱 오버헤드(Overhead)의 정량적 추정

위의 4단계 라이프사이클을 통해 발생하는 시간적 지연(Latency)을 정량적 복잡도 모델로 추정해 보자.

2.1 파라미터 복사 비용 (Deep Copy Cost)

C++ 기본 자료형(Primitive Type)인 int나 double을 넘길 때는 오버헤드가 작지만, MAVLink 패킷 전체나 QVector 배열과 같은 커다란 컨테이너를 복사할 때는 동적 메모리 할당(malloc/new) 비용이 매우 크게 작용한다.
전달하는 데이터 크기가 S 바이트일 때 복사 비용 지연시간은 \mathcal{O}(S)에 비례한다. (단, Qt의 암시적 데이터 공유(Implicit Data Sharing / Copy-on-Write) 기능이 작동하는 QString 등은 예외적으로 포인터와 참조 카운트 복사 비용만 든다.)

2.2 컨텍스트 스위칭 및 뮤텍스 경합 비용

큐 락(Lock)을 획득하는 과정에서 메인 스레드가 이미 바쁘게 렌더링 작업을 중이거나 수백 개의 다른 시그널이 폭발적으로 발생해 큐를 점유하고 있다면 경합(Contention) 지연이 발생한다.
일반적으로 Linux 기반 커널(예: 라즈베리 파이 또는 오프보드 컴퓨터) 환경에서 하나의 스레드가 커널을 거쳐 다른 스레드를 깨우는 데 걸리는 문맥 교환 지연 최하한선은 수십에서 수백 마이크로초(μs) 범위이다.

L_{passing} = T_{alloc} + T_{lock} + T_{context\_switch} + T_{deserialize}

가장 고비용 구간은 OS의 스케줄러를 관통하는 T_{context\_switch} 이며, 이는 PX4-Autopilot의 FPU(Floating Point Unit) 연산 속도와 비교하면 영겁의 시간이다.

이 다이어그램이 시사하는 핵심은, MavLink I/O Thread는 자신만의 처리 속도(예: 0.1ms)로 큐에 작업을 밀어 넣지만, 정작 UI 쪽 Main UI Thread에서 해당 작업이 언제 인출되어 실행될지는 **절대적으로 비결정론적(Non-deterministic)**이라는 점이다. 이를 **큐 팽창(Queue Bloating)**이라고 부른다.

4. QGC-PX4 아키텍처에서의 방어 기술

이러한 메시지 패싱 오버헤드로 인해, 자율 주행 궤적(Trajectory)이나 초정밀 VIO(Visual Inertial Odometry) 데이터를 QGroundControl 백엔드에서 큐잉 방식으로 프론트엔드로 무분별하게 퍼부으면 안 된다.

엔지니어는 반드시 두 가지 타협 기법을 적용해야만 한다.

  1. 원시 포인터 매개변수 사용 (안전한 스레드 소유권 이전): 데이터 크기가 큰 이미지 프레임이나 라이다(LiDAR) 포인트 클라우드 배열을 QueuedConnection으로 넘기지 말라. 대신 힙(Heap)에 포인터를 생성한 뒤 QVariant나 void* 매개변수로 포인터만 던지고(Message Passing Overhead \mathcal{O}(1)), 수신 측 스레드에서 메모리 해제를 전담하게 하는 소유권 이전(Ownership Transfer) 방식을 고수하라.
  2. 데이터 취합(Aggregation) 패싱: 400Hz로 들어오는 데이터를 건건이 400번 메시지 패싱하지 마라. MAVLink I/O Thread 내부에 가변 배열 하나를 두고 60Hz 간격으로 타겟 대상(예: 렌더링 갱신)에만 한 번씩 묶음(Batch) 패싱을 수행함으로써 T_{context\_switch} 패널티를 백분율 단위로 떨어뜨려야 한다.

5. 결론

Qt의 비동기 시그널-슬롯 분리 모델은 프로그래머에게 완벽한 스레드 분리 추상화를 제공해 주지만, 이면에는 매 호출마다 동적 메모리 할당, 뮤텍스 락, 문맥 교환(Context Switching)이라는 육중한 비용 명세서가 뒤따른다.

PX4-Autopilot의 폭발적인 연속 상태 전이(High-Rate Telemetry) 환경에서, 단순한 콜백 호출로 오해하고 델리게이트 큐잉 시스템을 남용한다면 OS 커널 레벨의 스레드 기아(Thread Starvation)를 초래하게 된다. 진보된 커스텀 GCS를 창조하고자 하는 백엔드 아키텍트는, 메시지 패싱 오버헤드의 정량적 지연(Latency) 한계를 염두에 두고 가장 효율적이고 보수적인 데이터 전달 설계 패턴을 채택해야 할 것이다.
