40.2.4.1 외부 단편화(External Fragmentation) 억제를 위한 슬랩 할당기(Slab Allocator) 구조

본 문서에 사용된 기준 버전은 PX4-Autopilot v1.14.0 및 QGroundControl v4.3.0이다.

UAV(무인항공기) 비행 제어기 및 지상 관제 시스템(GCS) 통신에서, 트랜잭션 메모리를 할당할 때 겪는 가장 치명적인 문제는 외부 단편화(External Fragmentation)이다. 외부 단편화란 전체 가용 메모리가 충분함에도 불구하고 연속적인(Contiguous) 메모리 공간이 확보되지 않아 새 객체 할당에 실패하여 시스템 패닉(Panic) 또는 통신 메시지 유실(Drop)을 초래하는 현상을 말한다.

이를 해결하기 위해 운영체제(NuttX RTOS) 계층과 애플리케이션 계층(QGC 백엔드) 모두에서 슬랩 할당기(Slab Allocator) 아키텍처를 차용하여 메모리를 관리해야 한다.

1. 외부 단편화(External Fragmentation)의 원인 및 영향

MAVLink 메시지는 크기가 다양하다(예: HEARTBEAT는 매우 작지만, MISSION_ITEM_INT나 PARAM_VALUE는 상대적으로 크다).
일반적인 동적 할당 방식(malloc()/free())을 사용할 경우, 서로 다른 크기의 메모리 블록이 백엔드 힙(Heap) 공간에 무작위로 할당 및 해제되면서 힙 사이에 빈틈이 발생한다.

드론의 임베디드 환경(PX4) 혹은 GCS의 장시간 백그라운드 구동 환경(QGC)에서 이러한 파편화가 누적되면, 결국 “충분한 크기의 연속된 가용 메모리 블록이 없는” 상태인 메모리 고갈(Out of Memory, OOM) 현상에 도달한다. 반면 Ardupilot의 정적 배열(Static Array) 접근 방식은 파편화를 물리적으로 차단하지만 가변 크기의 메시지 처리에 있어 유연성이 현저히 떨어진다. PX4-Autopilot은 슬랩 할당 모델과 유사한 기법을 메모리 큐 운용에 간접 적용하여 유연성과 메모리 안정성을 동시에 도모한다.

2. 슬랩 할당기(Slab Allocator) 아키텍처 매핑

슬랩 할당기는 메모리 관리자 측에서 자주 사용되는 크기의 메모리 청크(Chunk)들을 미리 큰 덩어리(Slab) 단위로 할당한 다음 캐시(Cache)에 저장하여 재사용하는 최적화 기법이다. MAVLink 트랜잭션 관리와 uORB 메시징 브로커에서도 이와 유사하게 미리 정의된 메시지 규격 크기를 기준으로 프리 리스트를 구축하는 패턴을 확인할 수 있다.

2.1 슬랩(Slab) 구조와 객체 캐시(Object Cache)

메모리 구조는 다음과 같은 논리적 계층으로 나뉜다.

  1. 캐시(Cache): 특정 객체(예: MAVLink 트랜잭션 구성을 위한 객체 트레일, uORB Subscriber Payload) 전용 관리자
  2. 슬랩(Slab): 캐시 내부에 존재하는 페이지(Page) 단위의 대용량 연속 메모리 블록. 하나의 슬랩은 일정한 크기의 객체 청크(Chunk) 배열로 나뉜다.
  3. 객체 상태(Object State): 각 청크는 ‘사용 중(Allocated)’ 혹은 ‘비어 있음(Free)’ 상태 중 하나를 가진다.

2.2 Ardupilot의 정적 풀(Static Pool) 패턴과의 설계 철학 비교

Ardupilot은 전통적으로 컴파일 타임(Compile-time)에 AP_Vehicle 및 하위 라이브러리의 클래스 멤버로 큐(Queue) 버퍼 공간을 1차원 정적 배열 크기로 선언하여 단편화 원인 자체를 원천 봉쇄하는 방향성을 추구한다.

반면, PX4-Autopilot은 POSIX 표준 호환인 NuttX RTOS 환경 하에서 실행된다. PX4 시스템 내부의 메시지 핸들러들은 메시지를 처리할 때마다 일정한 크기를 가지는 범용 풀 포맷을 사용한다. 이때 다양한 페이로드 크기를 포섭하기 위해 메모리 구조체들을 특정 사이즈 클래스(Size Class) 단위로 그룹화하여 관리한다. 이는 다수의 이기종 기체를 통합 통신해야 하는 ROS2(uXRCE-DDS) 에이전트 브리지 환경 및 커스텀 분산 관제망에서 폭발적인 MAVLink 트래픽을 처리할 때 더욱 높은 메모리 밀도와 안정적인 확장성을 제공한다.

3. 슬랩 할당기의 메모리 동작 메커니즘

수명 주기에 따라 메인 런타임에서 할당 요청이 들어올 때의 처리 알고리즘 단계는 다음과 같다.

  1. 객체 요청: 프로세스가 새 트랜잭션 컨텍스트 데이터를 담기 위해 미리 지정된 크기의 슬랩 캐시에 요청을 발생시킨다.
  2. 부분 슬랩(Partial Slab) 우선 검색: 할당기는 먼저 partial_slabs 리스트(일부 청크는 사용 중이고, 일부는 비어 있는 덩어리)를 훑어 남은 공간을 \mathcal{O}(1) 스텝 내에 식별하여 할당한다.
  3. 빈 슬랩(Free Slab) 활성화: 만약 partial_slabs에 가용 객체가 고갈되어 없다면, free_slabs 카테고리에서 새로운 통 메모리를 하나 꺼내어 partial_slabs 리스트로 승격 및 이동시킨 후 첫 번째 청크 포인터를 호출자에게 즉각 반환한다.
  4. 해제 시 병합 프로세스: 커맨드 트랜잭션이 성공하거나 타임아웃되어 포기되어 객체가 free() 된 후, 해당 슬랩 공간의 free_objects가 초기 풀 사이즈와 같아지면, 해당 슬랩은 다시 온전히 깨끗해졌으므로 free_slabs 리스트로 환원된다.

E(Effciency) = \frac{M_{payload}}{M_{payload} + M_{fragment\_loss}} \approx 1

상기 수식에 따르면, 템퍼링된 슬랩 덩어리 내에서는 남는 바이트 버려짐에 의한 단편화 폭 M_{fragment\_loss} 가 슬랩 디자인 패턴에 의해 거의 0에 수렴하게 되므로, 시스템 콜(System Call) 횟수 감소뿐만 아니라 캐시 히트율(Cache Hit Ratio) 측면에서 매우 투명하고 예측 가능한 비행 제어기 동작을 설계할 수 있다.

4. QGroundControl 및 커스텀 데스크톱 GCS 구현에의 시사점

QGroundControl과 같은 다목적 관제 데스크톱 소프트웨어는 Qt의 이벤트 분배 계층에서 다발성 시그널-슬롯(Signal-Slot) 이벤트 컨텍스트 객체를 생성한다. 한꺼번에 군집(Swarm) 환경에서 여러 기체의 센서, 위치, 배터리 메시지를 지속 수신할 경우 C++의 힙 단편화 누수가 응용 프로그램 지연 시간 튜닝의 최대 적으로 부상한다.

실무 개발자는 QGC 내장 메모리 메커니즘을 수정하는 것에 앞서, 성능 지연 지점이 관찰될 때 커스텀 백엔드 네트워크 소켓 수신 파트에 모던 C++17 기반의 다형성 할당자(std::pmr::polymorphic_allocator)를 사용하여 메시지 페이로드용 메모리 자원을 슬랩 기반 풀 모델로 교체 체결하라. 메시지의 사이즈 분포를 32 Byte, 64 Byte, 128 Byte와 같이 이산적인 슬랩 사이즈 클래스로 나누고, 클래스 별 독립적인 캐싱을 구현하는 방법이 응답 지연 시간(Latency Jitter)을 현격히 감쇄시키는 전략임을 주지하라.
