40.2.4.1. 외부 단편화(External Fragmentation) 억제를 위한 슬랩 할당기(Slab Allocator) 구조

무인 항공기(UAV)의 비행 제어 및 관제 시스템은 끊임없이 생성되고 소멸하는 센서 데이터스트림과 MAVLink 커맨드 트랜잭션으로 구성된다. 이러한 런타임 환경에서 표준 malloc()을 남용할 경우, 힙(Heap) 공간 중간중간에 사용 불가능한 미세한 빈 공간들이 난립하는 외부 단편화(External Fragmentation)가 필연적으로 발생한다. 총 가용 메모리는 충분함에도 연속된 큰 블록을 할당하지 못해 시스템이 멈추는(OOM, Out of Memory) 치명적 결함을 막기 위해, PX4-Autopilot의 근간이 되는 NuttX RTOS 및 고성능 GCS 백엔드는 슬랩 할당기(Slab Allocator) 아키텍처를 도입하여 메모리 파편화를 원천적으로 차단한다.

1. 외부 단편화(External Fragmentation)의 발생 기제

동적 할당기(Dynamic Allocator)가 메모리를 관리할 때, 크기가 제각각인 객체(예: 32바이트 GPS 패킷, 255바이트 임무 패킷 등)들이 무작위 순서로 할당되고 해제되면, 메모리 공간은 ’스위스 치즈’처럼 작은 구멍(Hole)들로 가득 차게 된다.
이 작은 구멍들의 총합이 1MB에 달하더라도, 물리적으로 연속된(Contiguous) 1KB의 공간이 파괴되어 있다면 시스템은 새로운 1KB짜리 객체의 할당 요청을 거부(Allocation Failure)하게 된다. 이것이 바로 외부 단편화이며, 제어 주기가 생명인 임베디드 비행 제어기에서는 할당 실패가 곧 추락(Crash)을 의미한다.

2. 슬랩 할당기(Slab Allocator)의 아키텍처 철학

슬랩 할당기는 개발자 제프 본윅(Jeff Bonwick)이 SunOS 커널의 객체 캐싱을 위해 처음 고안한 기법이다. 무인기 미들웨어 생태계에서 적용되는 슬랩 할당기의 핵심 사상은 **“동일한 크기와 타입을 가지는 커널 객체 또는 프로토콜 데이터 단위(PDU) 전용의 고정 크기 메모리 풀단(Slab)을 미리 여러 종(Class)으로 분리하여 운용한다”**는 것이다.

2.1 슬랩(Slab) 구성 메커니즘

슬랩 할당 시스템의 위계 구조는 Cache \rightarrow Slab \rightarrow Object 로 나뉜다.

  • Cache (캐시): 특정 타입의 데이터(예: MAVLink 파라미터 구조체 전용)만을 다루는 논리적 관리소이다.
  • Slab (슬랩): 하나 혹은 다수의 연속된 물리적 페이지(Page) 덩어리로 구성되며, 그 내부는 동일한 크기의 Object들로 쪼개져 있다. 상태에 따라 Full(모두 사용 중), Partial(일부 사용 중), Empty(모두 비어있음)로 분류된다.
  • Object (객체): 슬랩 안에 미리 쪼개져 있는 단일 요소이다. 할당과 해제가 발생해도 이 Object의 물리적 경계는 절대 허물어지지 않는다.

이러한 고정 경계(Fixed Boundary) 원칙 덕분에, 슬랩 내부에서 객체가 아무리 빈번하게 생성되고 파괴되어도 인접한 객체 구역을 침범하거나 경계선을 무너뜨리지 않아 외부 단편화가 수학적으로 0에 수렴하게 된다. (단, 객체 크기보다 데이터가 작아서 남는 내부 단편화(Internal Fragmentation)는 미미하게 발생할 수 있으나 생존성에 위협이 되지 않는다.)

3. PX4-Autopilot (NuttX)과 Ardupilot의 메모리 단편화 억제 비교

  1. PX4-Autopilot의 uORB Slab 캐싱 로직: PX4의 기반이 되는 NuttX RTOS는 내부 커널 오브젝트(세마포어, 메시지 큐 등)뿐만 아니라, 가장 핵심인 uORB 퍼블리셔/서브스크라이버 통신 트리에 정적 크기 기반의 메모리 풀(Pool)과 슬랩 패턴을 적극 차용한다. 예를 들어 vehicle_local_position uORB 토픽 큐는 부트 시점에 정확히 구조체 크기의 배수에 맞게 메모리 파티션을 재단(Slicing)해두고 재사용하므로, 비행 시간이 아무리 길어져도 힙 메모리의 엔트로피(Entropy)가 증가하지 않는다.
  2. Ardupilot의 Memory Pool 기반 블록 관리: Ardupilot 역시 치명적인 단편화를 막기 위해 AP_Param이나 GCS_MAVLink의 내부 큐 구현 시 파라미터 크기에 맞게 미리 얼로케이션된 (Pre-allocated) 메모리 풀/링 버퍼(Ring Buffer)를 사용한다. 하지만 일부 C++ 컨테이너나 문자열 파싱 구역에서 산발적으로 동적 할당을 점유하는 유지가 존재하여, 장시간 동작 시 힙 공간의 가용선이 서서히 변동하는 메모리 누수성(Memory Leak-like) 단편화 추이가 간헐적으로 관측되기도 한다.

4. QGroundControl에서의 슬랩 아키텍처 토폴로지 (Mermaid)

QGroundControl과 같은 고성능 C++ (Qt 기반) 백엔드 스레드는 실시간 임무 트랜잭션 수명 주기를 위해 커스텀 슬랩 할당 메커니즘을 캐싱 팩토리(Caching Factory) 형태로 둘 수 있다.

5. 결론

무인 시스템의 제어 주기 스레드는 극도로 가혹한 하드 리얼타임(Hard Real-time) 구역이다. 한가로이 힙 파티션을 쪼개고 합치는 dlmalloc 계열의 파편화 늪에 시스템을 방치하는 것은 자율 비행의 신뢰성을 근본적으로 훼손하는 행위이다.
PX4 생태계를 아우르는 비행 제어 및 관제 소프트웨어 개발자는 메모리의 외부 단편화(External Fragmentation) 리스크를 완전히 박멸할 수 있도록, **‘데이터 구조 성격에 따른 사전 분할과 고정 객체 캐싱’**이라는 슬랩 할당기(Slab Allocator)의 핵심 철학을 반드시 I/O 통신 미들웨어 레이어에 내재화해야 한다.
