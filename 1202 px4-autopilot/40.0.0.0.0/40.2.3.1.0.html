40.2.3.1. MAV_RESULT 열거형 변수의 마르코프 결정 과정(Markov Decision Process) 기반 전이도

비행 제어 시스템(Flight Control System)과 지상 관제 시스템(GCS) 간의 명령어(Command) 트랜잭션은 단방향 통보가 아닌 비동기적인 피드백 루프(Feedback Loop)로 구성된다. PX4-Autopilot 및 QGroundControl 환경에서 이러한 명령어의 처리 결과는 MAVLink 프로토콜의 COMMAND_ACK 메시지에 포함된 MAV_RESULT 열거형(Enum) 변수를 통해 반환된다. 본 절에서는 이 MAV_RESULT 열거형 변수들이 단순한 결과 값을 넘어, 마르코프 결정 과정(MDP, Markov Decision Process)의 상태 공간(State Space) 내에서 어떻게 확률적 전이(Stochastic Transition)를 구성하는지 수학적 모델과 함께 고찰한다.

1. MAV_RESULT와 마르코프 결정 과정(MDP)의 맵핑

마르코프 결정 과정(MDP)은 의사결정 모델링을 위한 수학적 프레임워크로, 상태(State, S), 행동(Action, A), 전이 확률(Transition Probability, P), 그리고 보상(Reward, R)의 튜플 (S, A, P, R)로 정의된다. GCS가 비행 제어기에 MAVLink 명령어(예: MAV_CMD_NAV_TAKEOFF)를 발행하는 과정은 MDP 기반의 자율 에이전트(Autonomous Agent)가 행동 a \in A를 선택하는 과정으로 볼 수 있다.

이때, 기체의 응답인 MAV_RESULT 변수는 다음 상태 s'로의 전이를 확정 짓는 관측(Observation) 체계가 된다.
주요 MAV_RESULT 상태 풀(State Pool)은 다음과 같다.

  • ACCEPTED (0): 명령이 성공적으로 인가되고 실행 승인됨 (보상 R > 0).
  • TEMPORARILY_REJECTED (1): 시스템이 바쁘거나 일시적인 조건 미달로 보류됨. 재시도(Retry) 행동 유도.
  • DENIED (2): 논리적/물리적 제약으로 런타임에 명령이 거절됨 (보상 R < 0).
  • UNSUPPORTED (3): 해당 비행 제어기 펌웨어(예: PX4 v1.14)에서 지원하지 않는 명령어.
  • FAILED (4): 실행을 시도하였으나 치명적인 하드웨어/소프트웨어 결함으로 실패함.
  • IN_PROGRESS (5): 장기 실행 명령어(Long-running Command)의 진행률(Progress) 피드백.

2. 상태 전이도(State Transition Diagram)의 토폴로지 분석

MAVLink 기반 제어 로직에서 MAV_RESULT는 흡수 상태(Absorbing State)와 비흡수 상태(Non-absorbing State)로 나뉘는 마르코프 연쇄(Markov Chain) 위상 공간을 형성한다.

아래 다이어그램은 QGroundControl이 명령어를 하달한 후, PX4 커맨더(Commander) 내부 상태 및 통신 채널의 지연 환경에 따라 전이되는 MDP 기반의 위상 공간을 나타낸다.

stateDiagram-v2
    direction TB
    
    state "명령 전송 (Action: a)" as CMD_SEND
    state "IN_PROGRESS (5)" as IN_PROGRESS
    
    %% 흡수 상태군 (Absorbing States - 종료 상태)
    state "ACCEPTED (0)" as ACCEPTED
    state "DENIED (2)" as DENIED
    state "UNSUPPORTED (3)" as UNSUPPORTED
    state "FAILED (4)" as FAILED
    
    %% 일시적/순환 전이 상태
    state "TEMPORARILY_REJECTED (1)" as TEMP_REJ
    state "Time Out (패킷 유실)" as TIMEOUT

    [*] --> CMD_SEND
    
    CMD_SEND --> ACCEPTED : 즉시 승인 (P = p1)
    CMD_SEND --> UNSUPPORTED : 파서 에러/미구현 (P = p2)
    CMD_SEND --> DENIED : 조건 미달/권한 부족 (P = p3)
    
    CMD_SEND --> TEMP_REJ : EKF 미수렴 / 바쁨 (P = p4)
    TEMP_REJ --> CMD_SEND : 지수적 백오프(Exponential Backoff) 후 재요청
    
    CMD_SEND --> IN_PROGRESS : 센서 캘리브레이션 등 장기 명령
    IN_PROGRESS --> IN_PROGRESS : 진행률(Progress) 업데이트
    IN_PROGRESS --> ACCEPTED : 작업 완료
    IN_PROGRESS --> FAILED : 런타임 중단/하드웨어 결함
    
    CMD_SEND --> TIMEOUT : 통신 단절 (P = p5)
    TIMEOUT --> CMD_SEND : 타임아웃 재전송 로직 (Retry)
    TIMEOUT --> [*] : 최대 재전송 횟수 초과 (Fail)
    
    ACCEPTED --> [*]
    DENIED --> [*]
    UNSUPPORTED --> [*]
    FAILED --> [*]

이 모델에서 ACCEPTED, DENIED, UNSUPPORTED, FAILED는 상태가 한 번 도달하면 더 이상 상태 전이가 일어나지 않는 **흡수 상태(Absorbing State)**이다. MDP 프레임워크 하에서 GCS의 목표는 누적 보상(Expected Cumulative Reward)을 최대화하는 최적 정책(Optimal Policy) \pi^*를 찾아, 기체의 상태를 궁극적으로 ACCEPTED 흡수 상태로 이끄는 것이다.

3. PX4와 Ardupilot의 MAV_RESULT 처리 아키텍처 비교

PX4-Autopilot과 Ardupilot은 MAV_RESULT 열거형을 처리하고 상태 전이를 제어하는 철학에서 뚜렷한 아키텍처적 차이를 보인다.

  1. PX4-Autopilot (커맨더 중앙 집중형 모델): PX4는 commander.cpp 및 vehicle_command uORB 토픽을 통해 대부분의 명령어를 중앙 집중식 큐(Queue)로 관리한다. 따라서 센서 초기화나 EKF2 추정기(Estimator)의 상태(예: GPS 3D Fix 전)를 단일 상태 머신에서 일괄 평가하여, 수학적으로 유효하지 않으면 대기(Wait) 상태 없이 즉각적으로 DENIED 또는 TEMPORARILY_REJECTED를 반환해 버리는 특성이 강하다. (결정론적이고 빠른 루프 종료 지향).
  2. Ardupilot (비동기 분산형 스케줄러 모델): Ardupilot은 GCS_MAVLink 클래스 라우팅을 거쳐 개별 차량(Copter, Plane)의 모드 스케줄러로 제어권이 이관된다. 일부 명령어의 경우 내부적으로 조건 만족을 위해 일정 밀리초(ms) 스레드 타임아웃까지 펜딩(Pending)하며 상태 전이를 지연시켰다가 ACCEPTED를 응답하는 유연한(하지만 덜 결정론적인) 엣지 케이스들을 지니고 있다.

4. IN_PROGRESS 상태 전이와 장수명 트랜잭션(Long-Lived Transaction) 최적화

IN_PROGRESS (5) 상태는 주로 센서 캘리브레이션(Sensor Calibration)이나 대규모 임무(Mission) 웨이포인트 업로드와 같은 장수명 트랜잭션 과정에서 발생한다.
MDP 관점에서 IN_PROGRESS 노드는 자가 순환 루프(Self-looping Edge)를 가지며, 이때 0부터 100까지의 진행률(Progress percentage) 값이 부가적인 관측(Observation) 데이터로 제공된다.

자율 에이전트(ROS2 기반 Offboard 노드 등)를 설계하는 개발자는, COMMAND_ACK 응답으로 IN_PROGRESS가 반환되는 즉시 시스템을 동기식 블로킹(Synchronous Blocking) 상태에 두어서는 안 된다. 대신, 마르코프 연쇄의 순환 기대 횟수(Expected Number of Visits)를 기반으로 최적화된 비동기식 폴링(Asynchronous Polling) 스레드를 설계하거나 uORB 구독(Subscription) 이벤트 기반의 락-프리(Lock-Free) 이벤트를 트리거하여야, 메인 제어 루프의 프레임 저하(Frame Drop)를 차단할 수 있다.

5. 결론

지상 관제 인터페이스에서의 제어 명령과 그에 상응하는 MAV_RESULT 응답 튜플은 단순한 ’성공(True) / 실패(False)’의 이진 부울 로직(Boolean Logic)으로 해석되어서는 안 된다. 이는 텔레메트리 통신 채널의 혼잡도(Congestion)와 비행 제어기 센서 융합기(Sensor Fusion)의 확률적 수렴성을 반영하는 거대한 마르코프 결정 과정(MDP)의 확률장이다.

PX4 생태계를 커스터마이징하는 시스템 개발자는 GCS 단의 백엔드(예: QGroundControl C++ 로직)를 설계할 때, 이 확률적 전이 위상 공간의 흡수 궤적(Absorbing Trajectory)을 완벽하게 계산하여, TEMPORARILY_REJECTED 상황에서의 지수 회피(Exponential Backoff) 타이머를 정밀 지연시키고 DENIED 결괏값에서 즉각적인 우회 행동 최적화 정책(Fallback Policy)을 구현해야 한다. 이를 통해 다중 오류 변수 앞에서도 흔들림 없는 강건한 제어 링키지(Control Linkage)를 확립하라.
