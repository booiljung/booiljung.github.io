40.5. 분산 합의(Distributed Consensus) 및 동시성(Concurrency) 예외 처리 공간

과거의 단순한 원격 조종(RC) 패러다임을 넘어, 현대의 자율 에이전트(Autonomous Agent) 관제 시스템은 다수의 지상 관제 시스템(GCS), 보조 컴퓨터(Companion Computer), 그리고 백엔드 클라우드 서버가 동시에 기체의 제어권을 분할하고 공유하는 마이크로서비스(Microservices) 아키텍처로 진화하고 있다. PX4-Autopilot 및 ROS2 생태계가 융합된 복합 시스템에서는 필연적으로 여러 제어 주체가 동시에 상충되는 명령(예: 한 GCS는 귀환, 다른 보조 컴퓨터는 충돌 회피)을 하달하는 상황(Concurrency)이 발생한다. 이러한 분산 제어 환경에서 일관성 없는 상태(Inconsistent State)를 방지하고, 단일 진실 공급원(Single Source 단일 진실 공급원(Single Source of Truth)을 유지하기 위해 필수적인 분산 합의(Distributed Consensus) 이론과 예외 처리(Exception Handling) 아키텍처를 본 절에서 탐구한다.

1. MAVLink 네트워크에서의 동시성(Concurrency) 문제

동시성 문제는 두 개 이상의 노드(Node)가 공유 자원(Shared Resource)—여기서는 기체의 비행 모드나 웨이포인트(Waypoint) 리스트—을 동시에 변경하려고 시도할 때 발생한다.

예를 들어, GCS A와 GCS B가 동일한 PX4 기체(System ID 1)를 관제하고 있다고 가정하자.

  • GCS A는 기체에게 고도 50m로 상승하라는 임무(Mission)를 작성하여 업로드 중이다.
  • 이와 완벽히 동시에, 네트워크 지연으로 인해 상황 업데이트를 늦게 받은 GCS B가 기체에 비상 착륙(RTL) 명령을 전송한다.

Ardupilot이나 초창기 단순 MAVLink 파서(Parser)의 경우, 이러한 경쟁 상태(Race Condition)에서 수신되는 패킷의 순서에 따라 기체 상태가 엎치락뒤치락하는(Thrashing) 치명적인 불안정성을 겪을 수 있다. PX4는 이러한 분산 오류 공간(Distributed Error Space)을 통제하기 위해 MAVLink 마이크로 서비스 규칙(MAVLink Microservices)에 따른 엄격한 트랜잭션(Transaction) 및 프로토콜 락(Protocol Lock) 메커니즘을 적용한다.

2. 분산 합의(Distributed Consensus)의 적용: 상태 권한 분리

분산 시스템 이론에서 널리 쓰이는 합의 알고리즘(예: 팍소스(Paxos)나 래프트(Raft))처럼 무거운 마스터-슬레이브(Master-Slave) 투표를 오프보드 제어에 그대로 적용하기엔 임베디드 통신 채널(예: SiK Telemetry)의 대역폭이 턱없이 부족하다. 따라서 PX4-Autopilot은 **“상태 권한의 명시적 분리(Explicit Separation of Authority)”**라는 경량화된 합의 메커니즘을 사용한다.

2.1 제어 권한(Control Authority)의 위임 구조

PX4 시스템은 MAVLink 프로토콜의 SET_POSITION_TARGET_LOCAL_NED 나 COMMAND_LONG 메시지를 받을 때, 기체가 **어느 주체(System ID & Component ID)**로부터의 지시를 신뢰할 것인지 사전에 합의(Consensus)하는 과정을 거친다.

  • 오프보드 모드(Offboard Mode): ROS2 컴패니언 컴퓨터가 제어 권한을 획득하기 위해서는 1차적으로 GCS 또는 조종기의 명시적인 ‘모드 전환’ 허가가 필요하다. 일단 오프보드 모드에 진입하면, 기체는 특정 컴펀넌트 ID(예: MAV_COMP_ID_ONBOARD_COMPUTER)에서 오는 주기적인(최소 2Hz 이상) 타겟 메시지만을 수용한다. 다른 GCS 노드가 끼어들어 동시성 충돌을 일으키는 것을 프로토콜 설계 단에서 락(Lock)을 걸어 무시하게 된다.

2.2 임무(Mission) 프로토콜의 트랜잭션 분리

가장 심각한 동시성 문제는 임무(Waypoints)를 수정할 때 발생한다. MAVLink 미션 프로토콜은 기체와 GCS 간에 철저한 요청-응답(Request-Response) 트랜잭션 구조를 가진다.
MISSION_REQUEST_LIST를 보내면, 다른 노드는 해당 기체의 임무 편집 상태가 잠금(Lock) 형태의 유휴 대기 상태임을 알 수는 없으나, 기체의 commander 컴포넌트는 내부 상태 머신(State Machine)을 ’임무 수신 중(Receiving)’으로 전이시켜, 다른 GCS 프레임으로부터 들어오는 불법적인 병렬 쓰기(Write) 연산을 NACK(거절) 처리한다.

3. 예외 처리 공간(Exception Space)과 네트워크 분할(Network Partition)

분산 시스템의 CAP 정리(CAP Theorem)에 의하면 일관성(Consistency), 가용성(Availability), 분할 내성(Partition Tolerance) 중 두 가지만을 완벽히 만족시킬 수 있다. 드론과 GCS 네트워크에서는 빈번한 통신 단절(Network Partition)이 상수(Constant)로 취급되므로, GCS 개발자는 C(일관성)와 A(가용성) 사이에서 타협하는 예외 처리 공간을 설계해야 한다.

  • 비일관적 가용성 선호(AP): RC 조종기가 끊어졌음에도 컴패니언 컴퓨터(보조 컴퓨터)의 VIO 트래킹을 믿고 자율 주행을 계속하도록 허용하는 상태. 통신망 분할 상태에서도 기체의 “임무 가용성“을 극대화한다.
  • 강한 일관성 선호(CP): 통신망이 단절(Data Link Loss)되는 즉시 0.5초 안에 기체의 내부 모드를 Failsafe(RTL 또는 Land)로 락(Lock)시켜버린다. 이는 GCS의 지시(일관된 명령)와 기체 행동 간의 오차를 원천 차단하는 보수적인 철학으로 QGroundControl에서의 기본 Failsafe 설계 방침에 해당한다.

    [*] --> Connected: Handshake
    Connected --> Normal_Operation: Consensus Achieved
    Normal_Operation --> Concurrent_Conflict: Dual Cmd Received
    Concurrent_Conflict --> Normal_Operation: Priority Filter / NACK (Idempotent)
    
    Normal_Operation --> Network_Partition: Link Loss Triggered
    Network_Partition --> Exception_Handling: Enter CP Fallback
    Exception_Handling --> Failsafe_RTL: Safety Override
    Exception_Handling --> Autonomous_Hold: AP Fallback (Offboard)

4. QGroundControl 백엔드의 동시성 방어 철학

QGroundControl(v4.3 기준)의 백엔드는 C++ 기반으로 이러한 분산 환경의 비동기적 트랜잭션을 매우 철저하게 관리한다. Vehicle 객체 내부에 구현된 MissionManager, ParameterManager 등의 클래스들은 동시적인 UI 조작(예: 사용자가 파라미터 2개를 동시에 Refresh 누르는 상황)이나 다중 링크(Wi-Fi 라디오와 LTE 모뎀)를 통해 들어오는 혼재된 메시지 응답을 큐(Queue)와 뮤텍스 락(Mutex Lock)을 이용해 직렬화(Serialization)한다.

결과적으로 최신 PX4-Autopilot 기반의 군집 관제 솔루션을 설계할 때는 단일 기체-단일 조종기라는 스칼라 세계관을 탈피해야 한다. 동시성 제어 큐(Concurrency Control Queue)와 상태 시퀀스 락(State Sequence Lock) 계층을 아키텍처 다이어그램 중앙에 배치하고, 분산된 각 노드 간의 위임(Delegation)과 합의 메커니즘을 예외 처리 공간에 수용하는 것이 차세대 드론 제어 공학의 필수 요소이다.
