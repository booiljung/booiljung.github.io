40.6.1.1.1. 상태 패턴(State Pattern)을 적용한 다형성(Polymorphism) 기반 명령 처리기 분리

무인항공기(UAV)와 지상 관제 시스템(GCS) 통신에서 MAVLink 명령 송수신은 단순히 패킷을 보내고 응답을 기다리는 절차적(Procedural) 1회성 로직으로 끝나지 않는다.
QGroundControl(이하 QGC)의 코어에서 이루어지는 신뢰할 수 있는 명령 전달은 WaitAck(응답 대기), Retry(재전송), Timeout(시간 초과), Failed(명령 실패), Success(명령 성공) 등 매우 복잡하고 순차적인 생명기 주기(Lifecycle)를 거친다. 본 절에서는 QGC 내부에 설계된 복잡한 명령 재전송 및 검증 로직을 우아하게 제어하기 위해 고안된 상태 패턴(State Pattern) 및 다형성(Polymorphism) 기반 설계 아키텍처를 면밀히 해부한다.

1. 개요 및 스파게티 코드(Spaghetti Code)의 한계

네트워크 통신의 결함을 해소하기 위해 단순히 MavlinkCommandSender 내부의 handleAck() 함수 안에 거대한 switch-case 문이나 수 십 줄의 if-else 구조를 하드코딩(Hard Coding)한다고 가정해 보자.

  • “만약 현재 1번 명령 전송 후 대기 중인데, 2번 명령이 들어오면 대기 큐에 삽입하고, 만약 타임아웃이 발생하면 Retry 카운터를 1 올리되 3회를 초과하면 버린다.”

이와 같은 절차 지향적 제어 로직은 시스템 규모가 커질수록 유지 보수를 원천적으로 불가능하게 만든다(일명 상태 폭발 카오스: State Explosion Chaos). PX4-Autopilot의 Failsafe, EKF 오류, 센서 캘리브레이션 거부 등 명령 응답(ACK)의 예외 케이스가 수백 가지에 달하기 때문이다.

2. 상태 패턴(State Pattern)의 도입 설계

객체 지향 소프트웨어 디자인 패턴의 고전인 GoF(Gang of Four) 패턴 중 **상태 패턴(State Pattern)**은 객체의 내부 상태에 따라 완전히 동일한 함수 호출이 다르게 행동(Behavior)하도록 허용하는 유연한 아키텍처이다.

QGC 내부 명령 관리기에서는 복잡한 분기문(Branching)을 제거하고, 각각의 **“상태” 그 자체를 독립적인 클래스로 추상화(Abstraction)**하여 캡슐화한다.

명령어의 라이프사이클을 추상 클래스 CommandState로 정의하고, 이를 상속(Inheritance)받는 구체 클래스(Concrete Class)들을 분리하라.

  • StateIdle: 큐가 비어있고 새로운 명령 하달을 대기하는 상태.
  • StateWaitAck: MAVLink 패킷(COMMAND_LONG 등)을 기체로 송신한 뒤 COMMAND_ACK 수신 리스너를 열고 대기하는 상태.
  • StateRetry: ACK가 지연되어 재전송 메커니즘이 활성화된 상태.
// 다형성 기반 상태 패턴 가상 인터페이스 예시
class CommandState {
public:
    virtual ~CommandState() {}
    virtual void handleAck(MavlinkCommandSender* context, const MAVLinkMessage& msg) = 0;
    virtual void handleTimeout(MavlinkCommandSender* context) = 0;
};

위의 예시처럼 컨텍스트(Context) 클래스인 MavlinkCommandSender는 현재의 명령 진행 상태를 나타내는 CommandState 포인터만을 보유하며 직접 로직을 판단하지 않는다.

3. 다형성(Polymorphism)을 통한 제어 위임

타임아웃(Timeout) 이벤트가 이벤트 루프에 의해 트리거될 때, MavlinkCommandSender 내부 로직은 단 한 줄이다.

// 컨텍스트 클래스의 위임
void MavlinkCommandSender::onTimeoutEvent() {
    _currentState->handleTimeout(this);
}

이 코드는 **다형성(Polymorphism)**의 극치를 보여준다. 컴파일러가 정적으로 함수를 맵핑하지 않고 런타임에 동적 바인딩(Dynamic Binding)을 수행하기 때문이다.
만약 _currentState가 현재 StateIdle 객체라면 타임아웃을 안전하게 무시(Ignore)할 것이고, StateWaitAck 객체라면 재전송(Retry)을 시도한 뒤 스스로를 StateRetry 객체로 컨텍스트 전이(State Transition)시킬 것이다. 이런 구조는 if(state == WAIT_ACK) 와 같은 부도덕한 코딩을 아키텍처 수준에서 말소시킨다.

4. 구조적 아키텍처 다이어그램 (Mermaid)

QGC 명령 재전송 구조의 상태 패턴을 UML 다이어그램으로 가시화하면 아래와 같다.

classDiagram
    class MavlinkCommandSender {
        -CommandState* _currentState
        +changeState(CommandState* newState)
        +receiveAck(MAVLinkMessage)
        +triggerTimeout()
    }

    class CommandState {
        <<Abstract>>
        +handleAck(MavlinkCommandSender*, MAVLinkMessage)*
        +handleTimeout(MavlinkCommandSender*)*
    }

    class StateIdle {
        +handleAck(...)
        +handleTimeout(...)
    }

    class StateWaitAck {
        +handleAck(...)
        +handleTimeout(...) // Переход -> StateRetry или Failed
    }

    class StateRetry {
        +handleAck(...)
        +handleTimeout(...) // MaxRetry 초과 시 -> Failed
    }

    MavlinkCommandSender o--> CommandState : Delegates
    CommandState <|-- StateIdle
    CommandState <|-- StateWaitAck
    CommandState <|-- StateRetry

다이어그램을 통해 입증되듯, 만약 PX4-Autopilot의 펌웨어 업데이트로 인해 MAV_RESULT_IN_PROGRESS와 같은 비동기 지속 판별상태(In-Progress State)가 추가되더라도 기존 코드를 단절시키지 않고 StateInProgress 구체 클래스만 독립적으로 새로 상속받아 추가하면 된다 (개방-폐쇄의 원칙 준수).

5. 결론 및 GCS 아키텍처 확장성

항공 관제 소프트웨어에서 제어 명령 전송은 단순한 비동기 프로토콜 통신을 넘어, 무인기의 자율 결함 허용(Fault Tolerance)을 연대 책임지는 가장 결정론적인(Deterministic) 파이프라인이다.

명령 대기 큐 처리기에 **상태 패턴(State Pattern)**을 투입하여 분기문을 완전히 소거하고 객체 지향의 **다형성(Polymorphism)**으로 로직을 분리하는 구조 역량은, QGroundControl을 현존하는 오픈소스 통제 솔루션 중 가장 유지 보수성이 뛰어난 프레임워크로 자리 잡게 한 숨은 공신이다. 차세대 GCS 펌웨어 호환 시스템을 설계하고자 하는 엔지니어는 반드시 이러한 디자인 패턴을 습득하여 비즈니스 로직과 시스템 상태 전이 절차를 안전하게 캡슐화해야 할 것이다.
