40.6.3.1. 스마트 포인터 참조 카운트(Reference Count) 사이클 검출을 위한 가비지 컬렉션(Garbage Collection) 회피 설계

QGroundControl(QGC)과 같은 방대한 콜백(Callback) 연쇄와 비동기 이벤트 핸들러를 지닌 C++ 시스템에서 std::shared_ptr는 생명주기 관리에 필수적인 스마트 포인터이다. 그러나 스마트 포인터의 파멸적인 안티 패턴인 **순환 참조(Circular Reference)**에 빠질 경우, 메모리가 영원히 해제되지 않는 치명적인 좀비(Zombie) 객체 누수가 발생한다.

본 절에서는 자바(Java)나 C#과 같이 무거운 가비지 컬렉터(Garbage Collector, GC)를 사용할 수 없는 하드 리얼타임 C++ 환경에서, 참조 카운트 사이클을 구조적으로 회피하고 무결성 있는 메모리 트리 구조를 설계하는 기법을 해부한다.

1. std::shared_ptr의 강한 참조(Strong Reference)와 한계

std::shared_ptr은 내부에 원시 포인터와 함께 참조 카운트(Reference Count) 블록을 쌍으로 유지한다. 특정 객체를 셰어드 포인터로 복사하여 소유(Ownership)할 때마다 카운트가 1씩 증가하며, 소유권을 포기(블록 변수 소멸 등)할 때마다 카운트가 1씩 감소한다. 카운트가 0이 되는 순간 메모리는 즉시 소각(delete)된다.

그러나 서로를 강하게 참조하는 두 객체가 존재한다면, 이 완벽해 보이는 메커니즘은 붕괴된다.

1.1 순환 참조 사이클(Cycle)의 발생

QGC 데이터 구조 중 기체(Vehicle) 객체와 임무 관리자(MissionManager) 객체가 논리적으로 긴밀하게 연결되어 있다고 가정하자.

  • Vehicle은 하위 객체로 MissionManager를 소유해야 하므로 std::shared_ptr<MissionManager>를 멤버 변수로 가진다.
  • 반대로 MissionManager 역시 부모인 Vehicle의 텔레메트리 파라미터를 읽기 위해 부모 객체에 접근해야 하므로 역방향으로 std::shared_ptr<Vehicle>를 쥐고 있다.
// 순환 참조 안티 패턴 (Memory Leak 유발)
class MissionManager;

class Vehicle {
public:
    std::shared_ptr<MissionManager> _missionMgr;
    ~Vehicle() { qDebug() << "Vehicle Destroyed!"; }
};

class MissionManager {
public:
    std::shared_ptr<Vehicle> _parentVehicle;
    ~MissionManager() { qDebug() << "MissionMgr Destroyed!"; }
};

void createVehicleSession() {
    auto v = std::make_shared<Vehicle>();           // v ref_count: 1
    auto m = std::make_shared<MissionManager>();    // m ref_count: 1

    v->_missionMgr = m;         // m ref_count: 2
    m->_parentVehicle = v;      // v ref_count: 2
} // 함수 종료. v, m 로컬 변수가 소멸되면서 각각 ref_count가 1로 감소.
  // [치명적 오류] ref_count가 0이 아니므로 소멸자가 불리지 않고 힙에 영구 고립됨!

이 코드가 실행되면 함수 스코프를 벗어났음에도 어느 쪽도 카운트가 0이 되지 않아 영원히 메모리에 고아(Orphan) 상태로 부유하게 된다. 타 언어의 백그라운드 가비지 컬렉터(GC)는 “루크(Root) 객체에서 도달할 수 없는 섬(Island) 덩어리“를 런타임에 추적하여 파괴해 주지만, C++에는 이러한 무거운 런타임 추척 에이전트가 존재하지 않는다.

2. 가비지 컬렉션 회피 및 논리적 트리 설계 원칙

C++ 아키텍처에서 GC의 부재를 완벽히 극복하는 수학적 원리는 시스템 구조를 순환 그래프(Cycle Graph)가 아닌 단방향 화살표만을 허용하는 비순환 방향 그래프(Directed Acyclic Graph, DAG) 혹은 부모-자식 트리(Tree) 계층으로 강제 분리하는 것이다.

2.1 약한 참조 std::weak_ptr의 도입

순환 참조를 끊어내는 핵심 문법인 std::weak_ptr는 해당 객체의 주소를 쳐다볼 수는 있되, 참조 카운트를 증가시키지 않는(Non-owning) 관찰자 포인터이다. 언제든 자신이 가리키는 대상이 null인지 검사하고, 유효할 때만 일시적으로 락(lock())을 걸어 잠시 사용할 수 있다.

  • 소유의 법칙: 논리적으로 부모(상위 객체)가 자식(하위 객체)을 강하게 소유(Strong Ownership)한다. \rightarrow std::shared_ptr
  • 관찰의 법칙: 자식이 부모를 참조하거나, 동등한 형제 객체끼리 참조할 때는 절대 소유권(Ownership)을 주장해서는 안 된다. \rightarrow std::weak_ptr

QGC의 Vehicle 컴포넌트 아키텍처는 철저히 이 법칙에 따른다.

// 순환 참조를 끊은 안전한 아키텍처
class Vehicle {
public:
    std::shared_ptr<MissionManager> _missionMgr; // 강한 소유 보장
};

class MissionManager {
public:
    std::weak_ptr<Vehicle> _parentVehicle;       // [수정점] 약한 참조 (카운트 증가 안함)
    
    void doUpload() {
        // 부모 객체가 여전히 메모리에 살아있는지 원자적으로 검증 후 획득
        if (auto vehicleLocked = _parentVehicle.lock()) {
            vehicleLocked->sendMavlinkMessage(...); // 정상 통신 수행
        } else {
            qWarning() << "Parent vehicle already destroyed. Aborting.";
        }
    }
};

4. PX4 및 ROS2 오프보드(Offboard) 노드 개발자 가이드

PX4-Autopilot 생태계 내에서 MAVSDK나 ROS2 C++(rclcpp) 에이전트를 프로그래밍할 때, 센서 콜백 리스너(Listener)는 흔히 콜백 객체가 자신을 등록한 매니저 객체를 상호 캡처([this] lambda)하면서 깊은 순환 구렁텅이에 빠지곤 한다.

GCS 프레임워크나 자율 주행 경로 계획(Path Planning) 미들웨어 코어 팀 리더라면 정적 분석 도구(Static Analyzer)를 활용하여 참조 카운트 사이클을 컴파일 타임에 검출해야 한다. 절대로 가비지 컬렉터와 유사한 커스텀 쓰레기 수집 추적 객체를 직접 만들 생각을 버려라. 단지 부모-자식 간의 소유권 트리를 거시적으로 직관화하고, 상향식(Bottom-Up)으로 역주행하는 포인터는 무조건 std::weak_ptr로 격하(Demote)하는 것이 결정론적 메모리 해제를 이룩하는 유일무이한 정도(正道)이다.
