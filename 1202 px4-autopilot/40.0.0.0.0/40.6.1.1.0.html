40.6.1.1. MavlinkCommandSender 클래스의 의존성 주입(Dependency Injection) 및 결합도(Coupling) 최소화 구조

PX4-Autopilot과 같은 무인항공기 비행 제어기(Flight Controller)에 지상 관제 시스템(GCS)이 MAVLink 명령(예: COMMAND_LONG 패킷)을 발송할 때 재전송 타이머(Retry Timer), 성공/실패 여부 폴링(ACK/NACK), 명령 트랜잭션 관리 등의 지루하고 반복적인 검증 작업이 수반된다. QGroundControl(이하 QGC) 아키텍처는 이를 전담하는 별도의 단일 책임 객체(Single Responsibility Object)인 MavlinkCommandSender 클래스를 설계하여 운영한다. 본 절에서는 해당 클래스의 핵심 디자인 명세인 의존성 주입(Dependency Injection) 기법과 결합도(Coupling) 최소화 구조를 정밀하게 분석한다.

1. 개요 및 단일 책임 원칙(SRP)의 중요성

GCS 개발 초심자들이 흔히 범하는 안티 패턴(Anti-Pattern)은 통신 채널 클래스(LinkManager) 내부나 기체 관리 클래스(Vehicle) 내부에 MAV_CMD 패킷을 빌드하고 전송하며, 그 타이머를 폴링하는 코드를 무분별하게 혼재시키는 것이다.

QGC 백엔드는 SOLID 원칙 중 단일 책임 원칙(Single Responsibility Principle, SRP)을 엄격히 준수한다.

  • Vehicle 클래스: 기체의 상태 데이터(Telemetry) 상태 머신 관리 및 UI 컴포넌트 데이터 공급에 집중한다.
  • LinkManager 클래스: 하드웨어 통신 포트(UART, UDP Socket)의 생명 주기와 바이트 로우 데이터(Byte Raw Data) 스트리밍에 특화된다.
  • MavlinkCommandSender 클래스: 오직 제어 버퍼 내 MAVLink 명령어 포맷팅, ACK 응답 큐 관장 및 최적의 재전송 알고리즘 처리에 책임을 한정한다.

이러한 책임 분리를 통해 Vehicle 스레드 내부에 불필요한 while 루프 지연이나 타이머 블로킹 현상이 파생되는 것을 구조적으로 막을 수 있다.

2. 생성자 의존성 주입(Dependency Injection: DI) 아키텍처

소프트웨어 엔지니어링에서 의존성 주입(Dependency Injection, DI) 이란 객체가 작동하는 데 필요한 외부 종속 객체를 내부에서 직접 생성(New/Instantiate)하지 않고, 외부 컨테이너나 상위 팩토리 모듈로부터 매개변수(Parameter)로 주입받는 하향식 설계 기법이다.

MavlinkCommandSender는 독립적인 단위 테스트(Unit Test)를 보장하고, QGC 내부 구조 변경 시 결합도(Coupling)에 의한 파급 효과를 차단하기 위해 생성자 수준의 DI 패턴을 완벽히 소화하고 있다.

// 의존성 주입이 적용된 QGC 내부 소스 코드 추상화 구조 (예시)
class MavlinkCommandSender : public QObject {
public:
    MavlinkCommandSender(LinkInterface* link, 
                         Vehicle* context, 
                         QObject* parent = nullptr);

    void sendCommand(const MAVLinkMessage& msg);
    
private:
    LinkInterface* _link; // 주입받은 통신 인터페이스 (다형성 보장)
    Vehicle* _vehicle;    // 명령 컨텍스트
};

위의 뼈대 코드에서 보듯, MavlinkCommandSender 클래스는 UART 통신인지 TCP 통신인지에 대한 하위 구성을 알 필요가 없다.
상위의 Vehicle 클래스(의존성 주입자)가 초기화될 때, 단지 구체적인 LinkInterface 포인터를 생성자를 통해 주입(Inject)해 줄 뿐이다. 다형성을 띤 인터페이스만을 바라보고 프로그래밍(Programming to Interface)하게 함으로써 모듈 간의 강부하가 유발하는 결합도(Coupling)를 최하단 수준까지 떨어뜨린다.

3. 구조 다이어그램 및 결합도(Coupling) 최소화 증명

아래 Mermaid 다이어그램은 QGC 내부의 MAVLink 명령어 송신 파이프라인이 어떻게 느슨한 결합(Loose Coupling)을 달성하고 있는지 보여준다.

다이어그램에서 입증되듯, MavlinkCommandSender는 SerialLink 나 UDPLink 같은 구체 클래스(Concrete Class)의 구현 기술에 전혀 독립적이다. 만약 보안 강화를 위해 추후 BleLink (블루투스 제어)나 SecureLink(AES 암호화 통신 채널)를 개발하더라도 MavlinkCommandSender 소스코드는 단 한 줄도 오염시키지 않고 확장이 가능하다(Open-Closed Principle 위배 근절).

4. 제어의 역전(Inversion of Control, IoC) 관점의 실무 예시

제어가 어떻게 진행되는지 실무 트랜잭션을 따라가 보라.

  1. 조종자가 마우스 키를 클릭하여 [이륙 (Takeoff)] 버튼을 누른다.
  2. UI 계층은 기체를 식별해 해당 Vehicle 인스턴스에 sendCommand() 매서드를 호출한다.
  3. Vehicle은 직접 MAVLink 포맷을 조립하지 않고, 단순히 파라미터를 자신이 소유한 MavlinkCommandSender 에게 위임(Delegate)한다.
  4. MavlinkCommandSender는 MAVLink 규격(MAV_CMD_NAV_TAKEOFF)에 맞춰 패킷을 바이트로 시리얼라이즈(Serialize) 화한 뒤 통신이 필요한 시점에 주입받은 _link->writeBytes()를 호출하여 망에 패킷을 흘려보낸다.
  5. 이후 비동기 타이머를 동작 시켜 PX4-Autopilot으로부터 도달할 응답(COMMAND_ACK)을 대기 큐에서 모니터링한다.

5. 결론

지상 관제 시스템의 핵심 I/O 병목 구간에 위치한 MavlinkCommandSender 모듈이 구조적으로 독립적일 수 있는 이유는 전적으로 의존성 주입(Dependency Injection)과 이를 통한 결합도 최소화 패턴 덕분이다.

PX4-Autopilot를 위한 커스텀 GCS나 ROS2 자율 통제 프레임워크를 개발할 때도, 단일체 스크립트 방식의 코딩에서 탈피하라. 송신자(Sender) 로직을 추상화된 독자 클래스로 격리하고, 생성자를 통한 인터페이스 주입 설계 패턴을 습관적으로 작성함으로써, 테스트 코드 작성이 용이하고 신뢰성 높은 고성능 관제 미들웨어를 엔지니어링해야 할 것이다.
