40.4 피드백 제어 시스템(Feedback Control System) 관점의 재전송 알고리즘

무인항공기(UAV) 지상 관제 시스템(GCS)에서 기체(Vehicle)로 MAVLink 제어 명령어(COMMAND_LONG 등)를 송신하고 패킷 유실에 대비하여 타임아웃(Timeout) 및 재전송(Retransmission)을 처리하는 로직은 표면적으로는 단순한 IT 네트워크 프로토콜처럼 보인다. 그러나 제어 공학(Control Engineering)과 자동화 서적의 설계 철학을 적용하면, GCS와 PX4-Autopilot 간의 신뢰성 통신 구조는 본질적으로 ’네트워크로 연결된 닫힌 루프 피드백 제어 시스템(Networked Closed-Loop Feedback Control System, NCS)’으로 치환하여 해석할 수 있다. 본 절에서는 이러한 거시적 제어 관점에서 재전송 알고리즘이 어떻게 안정성(Stability)을 유지하는지 분석한다.

1. MAVLink 재전송 메커니즘의 피드백 루프 모델링

지상 통제소에서 자율 에이전트(기체)의 비행 모드(Flight Mode)를 ’Return(복귀)’으로 변경하는 명령을 하달했다고 가정해 보자. 이 행위는 피드백 제어기의 기준 입력(Reference Input, r(t))을 변경하는 것과 같다. 기체 내부의 Commander 모듈(플랜트, Plant)은 명령을 수신한 후 상태(State)를 변경하고 결과 출력(Output, y(t))을 생성한다. 그리고 MAVLink의 COMMAND_ACK 메시지는 GCS 측 센서 피드백(Sensor Feedback) 역할을 수행하여 제어기(Controller)로 전달된다.

무선 환경의 간섭이나 대역폭 제한으로 인한 패킷 증발은 제어기 관점에서는 통신 라인 상의 강력한 외란(Disturbance, d(t)) 혹은 데드 타임(Dead Time, e^{-s\tau})으로 작용한다. GCS 내부에 캡슐화된 ’타임아웃 감지 및 재전송 알고리즘’은 사실상 이 외란 d(t)로 인해 피드백 y(t)가 갱신되지 않는 에러 상태(Error, e(t) = r(t) - y(t) \ne 0)를 인지하고, 제어 입력(Control Input, u(t))을 다시 플랜트에 주입하는 이벤트 기반 비례 제어(Event-based P-Control) 루프라 할 수 있다.

2. 피드백 제어 루프 블록 다이어그램 (Mermaid)

아래의 다이어그램은 GCS의 MAVLink 재전송 알고리즘 체계를 전형적인 제어 공학의 피드백 루프 블록도로 투영한 모델이다.

graph LR
    R[기준 입력 $r(t)$<br>사용자 명령] --> Sum((+))
    Sum -->|오차 $e(t)$| C[재전송 제어기<br>QGC Command Queue]
    C -->|제어 입력 $u(t)$<br>MAVLink Transmit| Delay1[네트워크 지연/유실 $\tau_{tx}$]
    Delay1 --> P[PX4-Autopilot<br>Commander Plant]
    P --> |무선 외란 $d(t)$| P
    P -->|상태 출력 $y(t)$| Delay2[네트워크 지연/유실 $\tau_{rx}$]
    Delay2 -->|피드백 $y_m(t)$<br>COMMAND_ACK| Sum

타임아웃은 위 루프에서 y_m(t)가 제한된 시간 안에 도달하지 못해 오차 e(t)가 양(Positive)의 값으로 유지될 때 u(t)를 다시 펄스 스파이크(Pulse Spike) 형태로 인가하는 비선형 스위칭 메커니즘을 띤다.

3. QGroundControl(GCS) 및 PX4의 관제 로직 해석

QGroundControl(v4.3 기준) 소스 코드 내부의 Vehicle 객체와 Command 관리 매니저는 이러한 피드백 관점에서 고도로 최적화되어 있다.

  • Anti-Windup 메커니즘 차원의 최대 재전송 횟수 제한:
    서보 모터나 PID 제어기에서 누적 오차에 의한 적분 와인드업(Integral Windup) 발산을 막기 위해 한계값(Saturation)을 설정하듯, QGC의 재전송 제어기 역시 최대 3~5회의 재시도 상한선을 둔다. 이는 네트워크 단절 상황에서 무한히 패킷을 쏘아 대역폭을 소진시키는 통신 발산(Communication Oscillation)을 방지하기 위한 안티-와인드업 기법이다.
  • PX4 생태계와 Ardupilot 간의 상태 피드백 철학 비교:
    GCS 관제 양상에서 Ardupilot은 MAVLink 명령 전송과 응답(ACK)이라는 이벤트(Event) 중심의 선형적 트랜잭션 성공 여부에 큰 책임을 부여한다. 반면, PX4 생태계는 GCS 내부의 상태 머신 모델(State Machine Model)을 주기적 스트리밍 텔레메트리(예: 고주파수 HEARTBEAT 및 시스템 상태 메시지)를 통해 전역적으로 갱신(Global State Observation)하는 것에 중점을 둔다. 즉, PX4 구조에서는 특정 명령어 단위의 국소적(Local) 피드백 루프가 타임아웃으로 모조리 실패하더라도, MAVLink 시스템 전체의 상태 관측 주기 위에서 백그라운드 모델이 동기화되도록 설계된 강건함(Resilience)을 특징으로 한다.

4. 자율 에이전트 비행 개발 시의 기술적 시사점

학술적인 시스템 엔지니어링에서는 ROS2(이하 uXRCE-DDS 미들웨어 브리지 기반)로 복수 기체(Swarm) 관제 에디터(Director)를 구축할 시 이 피드백 루프의 안정성 보장을 최우선으로 해야 한다.

송신 주기 T_s와 통신 채널의 전송 데드 타임 \tau, 그리고 패킷 분실 확률에 의한 지연 성분이 제어 루프의 위상 여유(Phase Margin)를 갉아먹는다. 따라서 네트워크 제어 시스템(NCS) 이론에 입각하여, 재전송 간격을 고정된 타임아웃 값이 아니라 채널의 트래픽 상태를 반영하는 시변(Time-varying) 피드백 이득(Feedback Gain) 형태로 스케줄링하는 로직을 GCS 백엔드 통신 아키텍처나 ROS2 퍼블리싱 매니저에 구현해야 한다. 이를 통해 극심한 RF 혼잡(RF Congestion) 환경에서도 PX4-Autopilot 기반 에이전트들이 예외 상황에 추락하거나 비이상적인 진동 모드로 빠지지 않도록 높은 제어 품질을 이끌어낼 수 있다.
