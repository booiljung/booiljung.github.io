40.6.2.1.1. 컨텍스트 스위칭 간 캐시 미스(Cache Miss) 유발 최소화를 위한 스레드 친화도(Thread Affinity) 설정

멀티 코어(Multi-Core) 체제의 오프보드(Offboard) 미션 컴퓨터(예: Raspberry Pi, NVIDIA Jetson)나 지상 관제 시스템(GCS) 컴퓨터에서 QGroundControl(QGC) 백엔드를 구동할 때, 운영체제(OS)의 스케줄러는 부하 분산을 위해 수시로 스레드의 실행 위치를 코어 간에 이주(Migration)시킨다.
그러나 PX4-Autopilot과 초당 수천 번의 MAVLink 패킷을 교환하거나 고해상도 비디오 프레임을 디코딩하는 핵심 I/O 스레드가 이리저리 이동할 경우, 하드웨어 레벨의 **캐시 미스(Cache Miss)**가 폭증하여 시스템 전반의 지연 시간(Latency)이 기하급수적으로 악화된다.

본 절에서는 이러한 하드웨어 오버헤드의 근본 원인을 분석하고, 이를 방어하기 위한 스레드 친화도(Thread Affinity) 고정 기법을 다룬다.

1. CPU 코어 이주(Migration)와 캐시 미스(Cache Miss)의 역학

최신 CPU 아키텍처는 느린 메인 메모리(RAM) 접근 속도를 보완하기 위해 코어마다 독립적인 초고속 L1/L2 캐시(Cache)를 지니고 있으며, 여러 코어가 공유하는 L3 캐시를 갖춘다.

  1. 캐시 웜업(Cache Warm-up): MavlinkReceiver 스레드가 코어 1에서 실행 중일 때, 패킷 파싱에 필요한 데이터 변수들과 MAVLinkMessage 구조체들은 코어 1의 L1/L2 캐시에 적재(Warm-up)된다.
  2. OS의 스레드 이주(Thread Migration): 운영체제의 스케줄러가 코어 1의 부하가 높다고 판단하여, 선점형 컨텍스트 스위칭(Preemptive Context Switching)을 통해 해당 스레드를 유휴 상태인 코어 2로 강제 이주시킨다.
  3. 콜드 캐시 미스(Cold Cache Miss) 폭발: 코어 2의 L1/L2 캐시에는 해당 스레드가 쓰던 데이터가 전혀 없다. 결국 코어 2는 비싼 주기억장치(RAM) 또는 L3 캐시에서 모든 데이터를 다시 끌어와야(Fetch) 하므로 수백 CPU 사이클의 지연(Cache Penalty)이 발생한다.
  4. 캐시 버스 무효화(Cache Invalidation 트래픽): 코어 1에 남아있는 기존 캐시 라인들도 무효화(Invalidate) 처리되어야 하므로 CPU 내부 버스망 공간이 심하게 오염된다.

이러한 캐시 미스는 단순한 소프트웨어 알고리즘 최적화(\mathcal{O}(1) 등)를 무의미하게 만들 정도로 강력한 하드웨어 병목 현상이다.

2. 혼합 아키텍처(big.LITTLE)에서의 위협

동반 컴퓨터(Companion Computer)로 널리 쓰이는 ARM 기반 시스템(NVIDIA Jetson 등)은 코어 성능이 비대칭적인 big.LITTLE 아키텍처를 주로 사용한다.

이 환경에서 OS 스케줄러가 비디오 스트림 디코딩 스레드나 경로 계획(Path Planning) 스레드를 고성능 코어(Big)에서 저전력 코어(LITTLE)로 무책임하게 문맥 교환시킬 경우, 알고리즘의 실행 시간(Execution Time)이 순식간에 3배 이상 튀어 오르는 지터(Jitter) 경련이 발생한다. 자율 드론의 제어 루프에서 제어 주기가 \pm 10ms 이상 요동치는 것은 비행 추락을 유발할 수 있는 치명적 결함이다.

3. 스레드 친화도(Thread Affinity) 설정 기법

캐시 미스를 구조적으로 근절하는 해법이 바로 스레드 친화도(Thread Affinity), 즉 특정 스레드를 특정 개별 CPU 코어(또는 코어 그룹)에 완전히 묶어두는(Pinning) 운영체제 레벨의 튜닝이다.

3.1 C++/Qt 레벨에서의 pthread_setaffinity_np

표준 C++이나 Qt 프레임워크 자체에는 범용적인 스레드 피닝(Pinning) API가 추상화되어 있지 않다. 따라서 Linux 환경(Ubuntu, Yocto 등) 하단의 POSIX API인 pthread를 직접 호출해야 한다.

#include <pthread.h>
#include <sched.h>
#include <QThread>
#include <QDebug>

// 비디오 디코딩이나 MAVLink I/O 전용 스레드의 run() 메서드 루프 진입 전 호출
void setCoreAffinity(int coreId) {
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(coreId, &cpuset); // 특정 코어 ID 할당

    pthread_t currentThread = pthread_self();
    
    // 현재 스레드를 지정된 코어에 바인딩
    int rc = pthread_setaffinity_np(currentThread, sizeof(cpu_set_t), &cpuset);
    if (rc != 0) {
        qWarning() << "Error calling pthread_setaffinity_np:" << rc;
    } else {
        qDebug() << "Thread successfully pinned to Core:" << coreId;
    }
}

4. 결론

멀티 스레드 프로그래밍의 대가는 코어 간 컨텍스트 스위칭으로 인한 끔찍한 L1/L2 캐시 미스(Cache Miss) 폭발이다.

개발자는 PX4-Autopilot 생태계 내에서 고성능 보조 컴퓨터(Companion Computer)용 C++ 모듈을 빌드할 때, 자신이 생성한 스레드가 어느 코어의 캐시 공간을 차지할 것인지를 물리적(Physical)으로 통제해야 한다. 적재적소의 코어에 **스레드 친화도(Thread Affinity)**를 하드 코딩하여 할당하는 최적화 패스(Pass)를 가동할 때 비로소, OS의 스케줄링 변덕에도 흔들림 없는 결정론적(Deterministic) 자율 제어 루프를 보장받을 수 있을 것이다.
