40.5.1.1 캡 정리(CAP Theorem) 관점의 일관성(Consistency)과 가용성(Availability) 타협 모델

분산 컴퓨팅 및 네트워크 시스템 설계에 있어 핵심적인 원리 중 하나인 캡 정리(CAP Theorem)는 분산 데이터 저장소가 일관성(Consistency), 가용성(Availability), 분할 내성(Partition Tolerance)의 세 가지 특성 중 동시에 최대 두 가지만을 만족할 수 있음을 수학적으로 입증한 이론이다.
네트워크 기반의 다중 기체(Swarm) 제어나 지상 관제 시스템(GCS)과의 실시간 MAVLink 통신망 역시 네트워크 파티션(분할)이 빈번하게 일어나는 전형적인 분산 시스템 환경이다. 드론 시스템은 태생적으로 통신 단절(Data Link Loss)이라는 네트워크 분할 상황(Partition Tolerance, P)을 기본적으로 상정해야 한다. 따라서, PX4-Autopilot과 QGroundControl 간의 분산 제어 아키텍처에서는 본질적으로 일관성(C)과 가용성(A) 중 하나를 우선순위에 두고 타협(Trade-off)하는 설계적 결단이 요구된다.

본 절에서는 캡 정리(CAP Theorem)의 관점에서, 통신 지연이나 일시적 단절 상황 하에 PX4-Autopilot이 텔레메트리(Telemetry) 상태 동기화 및 명령(Command) 처리를 어떻게 타협하고 있는지, 그리고 Ardupilot 및 MAVLink 프로토콜과는 어떤 차별화된 일관성-가용성 밸런싱(Balancing) 모델을 갖는지 학습하라.

1. 드론 관제 시스템에서의 CAP 정리 정의

멀티로터나 고정익 등 무인기 기반 관제 시스템에서 CAP의 각 요소는 다음과 같이 매핑된다.

  • 일관성(Consistency, C): GCS의 화면에 표시되는 기체의 상태(위치, 배터리, 비행 모드 등)와 기체 내부의 실제(Actual) 상태가 한 치의 오차도 없이 동일한 순간을 보장하는 특성이다. 높은 일관성을 요구할수록 명령의 확정 응답(ACK)을 강제해야 한다.
  • 가용성(Availability, A): GCS가 기체에 내린 명령(Arm, Takeoff, 비행 모드 변경 등)이 언제나 즉시 수락 및 처리되어야 하며, 기체 역시 통신 상태와 무관하게 자율 비행의 흐름을 멈추지 않고 지속해야 하는 응답성 특성이다.
  • 분할 내성(Partition Tolerance, P): RF 모뎀 통신 장애, 장애물에 의한 전파 가림(Line-of-Sight 상실), 혹은 비대칭 라우팅(Asymmetric Routing) 등으로 인해 패킷이 유실되거나 지연되더라도 전체 시스템의 붕괴 없이 작동을 유지하는 능력이다. 무선 통신 기반 무인기에서는 필수 불가결한 전제조건이다.

따라서 PX4 관제 시스템 설계는 **CP 모델(Consistency 지향)**과 AP 모델(Availability 지향) 사이의 선택으로 귀결된다.

2. PX4-Autopilot의 가용성 중심(AP 모델) 지향적 텔레메트리 설계

상태 정보 모니터링 체계에서 PX4-Autopilot의 uORB 아키텍처 및 MAVLink 텔레메트리 스트리밍은 기본적으로 **가용성(AP 모델)**을 우선시하도록 설계되어 있다.

2.1 MAVLink 텔레메트리의 최종적 일관성(Eventual Consistency)

비행 중인 기체의 모든 센서 데이터(예: EKF2 추정 위치, IMU, 나침반 등)는 매 순간 변한다.
PX4의 mavlink_receiver 및 텔레메트리 송신 모듈은 이 데이터를 MAVLink 메시지(예: GLOBAL_POSITION_INT, ATTITUDE)로 변환하여 GCS로 전송한다. 이때, UDP나 Serial 채널의 패킷 유실을 복구하기 위한 무거운 재전송 메커 편집 메커니즘을 사용하지 않는다.
이는 GCS가 순간적으로 과거의 상태를 표시하더라도(일관성 희생), 스트리밍되는 다음 패킷을 통해 곧 최신 상태로 덮어씌워지며 종국에는 데이터가 동기화되는 최종적 일관성(Eventual Consistency) 방식을 취하기 때문이다. 가용성과 실시간성을 극대화한 결과이다.

이는 전통적인 Ardupilot 환경에서도 유사하게 적용되나, PX4의 경우 uORB의 퍼블리시-서브스크라이브 패턴이 네트워크 밖으로까지 확장된 무상태(Stateless) 브로드캐스트의 성격을 지녀 AP 모델에 더욱 부합하는 형태로 구현되어 있다.

3. 명령 트랜잭션에서의 CP 모델(일관성 중심) 타협

반면, 모드 변경(Flight Mode Switch), 이륙(Takeoff), 조종 권한 위임과 같은 **중요 제어 명령(Command Long/Int)**은 AP 모델로 처리할 수 없다. GCS가 ‘이륙’ 명령을 전송하고 이를 기체가 수신하지 못했는데도 GCS 화면상에서 ’이륙 중’으로 표시된다면 심각한 안전사고로 직결되기 때문이다.

명령 처리 채널에 대해서는 **CP 모델(네트워크 분할 시 지연이 발생하더라도 일관성을 확보하는 모델)**로 타협하여 설계되어 있다.

3.1 COMMAND_ACK 기반의 엄격한 상태 확인

PX4의 커맨더(Commander) 모듈은 외부 명령 수신 시 내부 상태 유효성 검사(Pre-flight Check 등)를 수행 후 그 결과를 COMMAND_ACK 트랜잭션으로 강제 반환한다. GCS는 이 ACK를 수신하기 전까지 화면 상의 UI 상태 토글을 대기 상태(Pending)로 유지하여 UI와 기체 간의 상태 불일치(Inconsistency)를 예방한다.
통신 환경이 열악(Partition)할 경우 ACK 응답 지연에 의해 GCS의 제어 버튼이 잠시 멈춘 것처럼 보일 수 있다. 이는 일관성(C)을 확보하기 위해 사용자 응답 가용성(A)을 희생한 대표적인 사례이다.

4. 커스텀 GCS 아키텍처 설계자를 위한 지침

QGroundControl을 수정하거나 자체적인 GCS(Ground Control Station) 백엔드를 개발할 때, CAP 정리의 관점에서 개발자가 통제해야 할 타협점(Trade-off)은 다음과 같다.

  1. 텔레메트리 큐의 비차단(Non-blocking) 처리: 텔레메트리 파싱 스레드가 네트워크 I/O 지연에 의해 블로킹되지 않도록 하라. 과거 상태에 얽매여 일관성을 맞추기 위해 지연 대기 큐(Wait Queue) 체계를 구축하면 가용성이 치명적으로 저하된다. 즉시 버리고 최신 패킷을 수용하는 링 버퍼(Ring Buffer) 형태를 채택하라.
  2. 트랜잭션 명령의 비동기 상태 머신 구축: 중요한(Critical) 제어 명령은 철저한 상태 기반 제어(State-based Control) 파이프라인으로 구축하라. 명령을 전송한 즉시 GCS의 상태 변수를 변경하지 말고, COMMAND_ACK와 이어지는 HEARTBEAT 내의 custom_mode 필드가 변경되었는지에 대한 의존성 플래그(Dependency Flag)를 검증한 후에만 GCS UI에 반영하도록 설계하라.
  3. 오프보드(Offboard) 제어의 타협점 이해: ROS 2 기반의 실시간 Offboard 제어(TrajectorySetpoint 메시지 등)는 초당 수십 회의 브로드캐스트를 전제로 한다. 이는 제어 명령임에도 불구하고 CP 모델의 락(Lock)을 걸면 통신 지연으로 인해 드론의 제어 루프가 붕괴한다. 따라서 이 영역은 예외적으로 지속적인 재전송 브로드캐스팅(AP 중심) 방식을 채택하여 기체의 반응성을 최대로 끌어올려야 한다.

결론적으로, PX4-Autopilot과 GCS 간 통신 시스템 설계는 모든 데이터에 동일한 CAP 제약을 가하는 것이 아니라, 관찰 데이터(Telemetry)에는 AP 모델을, 중요 상태 명령(Command)에는 CP 모델을 이중으로 분리 적용한 혼합(Hybrid) 타협 아키텍처를 구현해야 한다.
