40.7.1.1. epoll 기반 이벤트 디스패처(Event Dispatcher)의 상수 폴링 복잡도 달성

고성능 지상 관제 시스템(GCS, Ground Control Station)이나 ROS2 마이크로 XRCE-DDS(uXRCE-DDS) 브리지 에이전트는 기체 수백 대가 송신하는 MAVLink 데이터 스트림을 실시간으로 감당해야 한다. 본 절에서는 Linux 커널의 epoll 시스템 콜을 활용하여 GCS 서버 내 이벤트 디스패처(Event Dispatcher)가 어떻게 기존 방식(select, poll)의 선형 폴링 복잡도 \mathcal{O}(N)를 극복하고, 상수 시간에 가까운 \mathcal{O}(1) 폴링 복잡도를 달성하는지 수학적 구조 및 동작 원리와 함께 분석한다.

1. 기존 I/O 멀티플렉싱(Multiplexing)의 한계

초기의 네트워크 관제 시스템과 Ardupilot의 기본 통신 스트림 처리기 등에서 활용되던 POSIX select() 또는 poll() 함수식 멀티플렉싱 기법은 연결된 드론 소켓 개수 N에 비례하여 시스템 오버헤드가 증가한다.

  • 선형 탐색 병목 현상: select() 호출 시 커널은 파일 디스크립터(File Descriptor, FD) 세트 전체를 반복 탐색하여 수신 상태를 검사한다.
  • 메모리 복사 비용: 매 I/O 폴링 루프마다 전체 FD 세트를 유저 공간(User Space)에서 커널 공간(Kernel Space)으로 복사해야 하는 중대한 성능 저하 요인이 발생한다.

따라서 전체 검사 대상 N개 중 실제로 데이터가 유입된 활성 소켓이 유의미하게 적은 K개(K \ll N)라 하더라도, 연산 시간 복잡도는 여전히 \mathcal{O}(N)이 된다.

2. epoll의 구조적 특징과 \mathcal{O}(1) 복잡도 달성 원리

최신 맞춤형 GCS 서버 응용프로그램 및 C++ 기반 퍼포먼스 통신 모듈에서는 epoll(Linux) 또는 kqueue(BSD/macOS) 시스템 콜을 채택하여 I/O 비용을 비약적으로 낮춘다. epoll은 다음 두 가지 핵심 데이터 구조를 활용하여 폴링 복잡도를 \mathcal{O}(1) 근사치로 최적화한다.

  1. 레드-블랙 트리(Red-Black Tree): 커널 공간에서 관리되는 모니터링 대상 FD의 트리에 소켓을 등록(epoll_ctl)한다. 삽입/삭제 복잡도는 \mathcal{O}(\log N)이나, 실제 이벤트 탐지 로직과는 물리적으로 분리되어 관리된다.
  2. 준비 리스트(Ready List): 하드웨어 인터럽트(Interrupt)가 발생하여 네트워크 카드 버퍼에 데이터가 수신되면, 커널은 해당 FD만을 이 리스트의 이중 연결 리스트(Doubly Linked List)에 추가한다.

따라서 epoll_wait() 호출 시, 커널은 전체 대상을 N번 순회하지 않고 단순히 무조건 활성화된 소켓의 리스트(Ready List) 메모리 블록만을 반환하므로 이벤트 발생 개수 K에만 의존하는 \mathcal{O}(K) 혹은 단건 처리 시 \mathcal{O}(1) 복잡도를 달성한다.

3. 에지 트리거(Edge Trigger, ET) 대 레벨 트리거(Level Trigger, LT)

이벤트 디스패처 구현에 있어 epoll은 크게 두 가지 동작 방식을 제공하며, PX4-Autopilot 의 MAVLink처럼 트래픽 갱신 주기가 짧은 환경에서는 이를 면밀히 고려해야 한다.

  • 레벨 트리거(Level Trigger, LT): 단말 소켓의 입력 버퍼에 데이터가 1바이트라도 남아 있으면, epoll_wait()은 지속적으로 이벤트를 발생시킨다. QGroundControl 프레임워크와 유사한 Qt 기본 호환 루프 구조 구현이 쉬우나 중복 이벤트 비용이 발생할 수 있다.
  • 에지 트리거(Edge Trigger, ET): 상태가 완전히 변경되는 시점(즉, 데이터가 없다가 수신되는 그 짧은 찰나)에 단 한 번만 이벤트를 발생시킨다. 락-프리(Lock-Free) 큐와 비차단(Non-blocking) 소켓의 EAGAIN 판독 결합 시 매우 높은 Throughput 스루풋 방어가 가능하지만, 파싱 버그로 수신 버퍼 내 일부 메시지를 흘려보내는 메모리 릭(Memory Leak)이나 락-업(Lock-up) 위험을 내포한다.

3.1 수식적 성능 비교 모델

이벤트 디스패처의 전체 지연 시간(Total Latency, L_{Total})을 모델링하면 다음과 같다.

L_{Total} = T_{poll} + \sum_{i=1}^{K} (T_{dispatch_i} + T_{handle_i})

여기서 select 방식은 T_{poll} \propto N (모든 소켓 검사)이므로 L_{Total} \approx c_1 N + c_2 K \vert M \vert 의 형태를 띤다. (단, \vert M \vert 는 각 메시지 페이로드의 정보 크기 치환 함수). 반면 epoll 방식에서는 T_{poll} \propto 1 에 수렴하므로 지연 시간이 L_{Total} \approx c_3 + c_2 K \vert M \vert 로 대폭 단축되며, 관제할 PX4 장치 개수 N에 따른 오버헤드 변동성이 거의 제거된다.

4. QGroundControl 관제 아키텍처에의 적용 방안 고찰

상용 QGroundControl(v4.3 기준)의 모바일/크로스 플랫폼 최적화를 위한 Qt 이벤트 루프 엔진(QEventDispatcherUNIX 구현체 최하단부) 역시 플랫폼 종속 시스템 콜(epoll, glib, kqueue)로 위임되도록 추상화되어 있어, 일정 수준의 정수 복잡도 최적화가 적용되어 있다.

그러나 ROS2-PX4 브리지 시스템이나 완전한 서버리스(Serverless) 관제 백엔드를 설계할 경우엔, C++ 네이티브 epoll 기반 디스패처를 직접 작성하여 스레드 풀(Thread Pool) 계층으로 이벤트를 즉시 인계하는 “리액터 + 스레드 풀(Reactor-Thread Pool)” 패턴을 도입하라. 이는 GCS의 처리 용량을 한 대의 물리 머신당 수천 대의 MAVLink 채널 동시 통신 수준으로 격상시킬 수 있는 근간이 된다.
