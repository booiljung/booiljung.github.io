40.7.1.1.1. 스레드 풀(Thread Pool) 작업 훔치기(Work Stealing) 스케줄링을 통한 부하 분산(Load Balancing)

다수의 PX4-Autopilot 장치(Swarm)가 연결된 지상 관제 시스템(GCS, Ground Control Station)이나 ROS2 기반의 클라우드 백엔드에서는 특정 기체로부터 대량의 데이터(예: 고해상도 VIO 비전 데이터 융합 메시지)가 집중될 때, 이를 처리하는 단일 스레드에 병목 현상이 발생할 수 있다. 본 절에서는 이러한 불균형을 해소하기 위해 최신 고성능 GCS 아키텍처에 도입되는 작업 훔치기(Work Stealing) 기반의 스레드 풀(Thread Pool) 부하 분산(Load Balancing) 기법을 설명한다.

1. 정적 스레드 할당(Static Thread Allocation)의 병목 한계

대부분의 기본 관제 애플리케이션에서는 개별 드론(Vehicle) 객체가 생성될 때마다 할당된 특정 워커 스레드(Worker Thread)가 해당 드론의 MAVLink 메시지 해독 및 파라미터(Parameter) 동기화를 전담한다. 그러나 QGroundControl(v4.3 기준)과 같이 다수의 차량 스레드가 각 통신 포트를 관리하는 구조에서 다음과 같은 한계가 존재한다.

  • 부하 불균형(Load Imbalance): A 드론은 단순히 1Hz의 Heartbeat만을 송신하고, B 드론은 50Hz의 센서 원시 데이터 스트림과 MAVLink FTP를 통한 수 메가바이트의 임무(Mission) 마이크로로그를 업로드 중인 상황을 가정해 보라.
  • 자원 유휴 현상(Resource Starvation/Idling): B 드론을 담당하는 스레드는 처리 지연(Latency)이 기하급수적으로 늘어나 대기 큐(Wait Queue)가 폭주하는 반면, A 드론을 담당하는 스레드는 연산 자원이 남은 채로 유휴(Idle) 상태에 대기하게 된다.

2. 작업 훔치기(Work Stealing) 아키텍처의 이론과 원리

작업 훔치기 스케줄링 알고리즘은 분산 병렬 컴퓨팅 공간에서 대기 시간 복잡도를 최소화하기 위해 고안된 큐잉 모델(Queuing Model)이다.

동작 원리:

  1. 풀(Pool) 내의 모든 워커 스레드는 각자 고유한 로컬 큐(Local Deque: Double-ended Queue)를 하나씩 보유한다.
  2. 각각의 워커 스레드는 자신의 큐에서 가져온 작업(MAVLink 파싱 콜백, 제어기 상태 연산 등)을 LIFO(Last-In-First-Out) 방식으로 우선 처리한다. 이 방식은 메모리 캐시 지역성(Cache Locality)을 극대화한다.
  3. 자신의 로컬 큐가 비어 유휴 상태(Idle)가 된 워커 스레드(Thief)는 무작위로 또는 지정된 알고리즘에 따라 다른 바쁜 워커 스레드(Victim)의 로컬 큐를 검사한다.
  4. Victim 스레드 큐의 **반대쪽 끝(FIFO 위치)**에서 대기 중인 작업을 훔쳐 와(Steal) 대신 연산을 수행한다.

3. 부하 분산의 수학적 복잡도 증명

작업 훔치기 방식의 연산 스케줄링 대기열 시간 복잡도는 기존의 공유 글로벌 큐(Global Queue) 패러다임이 가진 병목을 우회한다.
단일 글로벌 큐 구조에서는 \mathcal{O}(P) 명의 프로세서가 락(Lock)을 획득하기 위해 경쟁(Contention)하므로 컨텍스트 스위칭 지연이 증가한다. 하지만 Work Stealing 기반의 덱(Deque) 모델에서는 최악의 시간 복잡도 기댓값이 T_P \le \frac{T_1}{P} + \mathcal{O}(T_{\infty}) 로 증명된다. (단, T_1은 단일 프로세서 실행 시간, T_{\infty}는 무한대 스레드 환경에서의 임계 경로 실행 시간, P는 코어 수).
즉, 글로벌 락 경합 없이 스레드 개수에 거의 선형적인(Linear) 성능 스케일링을 자랑한다.

4. 커스텀 GCS 및 백엔드 적용 아키텍처 실무

순수 C++로 개발되는 최상급 ROS2 관제 단말기 혹은 다중 기체 관제 GCS에 이 알고리즘을 도입하기 위해서 다음과 같은 설계가 요구된다.

  1. 디스패처와 결합된 I/O 스레드 분리: epoll 이벤트 디스패처는 소켓 이벤트 관리에만 집중하고, 디스패치된 콜백 함수(데이터 역직렬화, 암호화 복호화, 관제 UI 업데이트 등)를 Work Stealing 스레드 풀에 던지는(Submit) 아키텍처로 분리하라.
  2. C++17/20 및 Intel TBB 활용: 스레드 풀을 밑바닥부터 구현하는 대신, 이미 락-프리(Lock-Free) 덱 연산이 결합된 Intel TBB(Thread Building Blocks) 라이브러리의 tbb::task_group 이나 C++ std::async의 백엔드 스케줄러(OS 종속)를 채택하라.
  3. 가벼운 작업 단위 입도(Granularity): 훔치기 오버헤드보다 작업 자체의 연산 시간이 길 때 효율성이 높다. 따라서 1바이트씩 MAVLink 메시지를 조립하는 스트림 계층보다는, 완전한 메시지 패킷 1개가 파싱 완료된 직후인 콜백 단계(Callback Phase)를 워커 스레드가 훔칠 수 있는 독립적인 태스크(Task)로 정의하라.
