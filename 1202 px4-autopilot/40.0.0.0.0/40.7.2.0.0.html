40.7.2. 비차단(Non-blocking) 동시성을 위한 락-프리 자료구조 구현 수학

고성능 지상 관제 시스템(GCS, Ground Control Station)이나 무인기 탑재 미들웨어(예: ROS2 브리지, PX4-Autopilot uORB 통신단)에서 멀티스레딩(Multithreading) 환경을 구축할 때, 데이터 경합을 방지하기 위한 전통적인 상호 배제(Mutual Exclusion, Lock) 기법은 심각한 컨텍스트 스위칭 지연(Context Switching Latency)과 우선순위 역전(Priority Inversion) 현상을 유발한다. 본 절에서는 이러한 블로킹 방식을 회피하고 하드웨어 수준의 원자적 연산(Atomic Operation)을 기반으로 동작하는 락-프리(Lock-Free) 자료구조의 수학적 토대와 시스템적 구현 모델을 심도 있게 분석한다.

1. 다중 스레드 동시성 모델의 한계와 락-프리 알고리즘의 대두

PX4-Autopilot 펌웨어의 핵심인 uORB(Micro Object Request Broker) 메시징 버스 또는 QGroundControl의 고주파수 텔레메트리 파싱 파이프라인 수신부를 살펴보면, 다수의 생산자(Sensors/Networks)와 다수의 소비자(Controllers/UI)가 복잡하게 얽혀 있다.

만약 이 공유 버퍼(Shared Buffer)에 단순히 std::mutex를 적용할 경우 다음의 치명적인 문제들이 발생한다.

  • 우선순위 역전(Priority Inversion): 높은 우선순위 태스크(예: 자세 제어기)가 낮은 우선순위 태스크(예: 로그 저장)가 쥐고 있는 락(Lock)을 기다리느라 제어 실시간성(Real-time Deadline)을 놓치는(Miss) 현상. (NuttX RTOS 차원에서 Priority Inheritance Protocol 등을 지원하나 한계가 존재)
  • 락 경합 오버헤드(Lock Contention Overhead): 무거운 부하 환경에서 코어 간 배타적 소유권 다툼으로 인해 확장성(Scalability)이 급감.

이를 근본적으로 해결하기 위해 락 없이(Lock-Free) 데이터를 안정적으로 교환할 수 있는 알고리즘 설계가 필연적으로 요구된다.

2. 블로킹, 락-프리, 웨이트-프리의 수학적 엄밀성 정의

비차단(Non-blocking) 동시성 제어 속성은 다음 세 가지 수학적 진척 보장성(Progress Guarantee) 단계로 분류된다.

  1. 장애 면역(Obstruction-Free): 다른 모든 스레드가 일시 중지(Suspended)되었다고 가정할 때, 실행 중인 임의의 스레드는 반드시 유한한 단계(Bounded Steps) 내에 작업을 완료할 수 있다. (가장 약한 비차단 특성)
  2. 락-프리(Lock-Free): 시스템 전체 스레드 집합 S 중 최소한 하나 이상의 스레드가 연산을 진행 중이라면, 무한 대기(Deadlock)나 라이브락(Livelock)에 빠지지 않고 반드시 유한한 시점 안에 시스템 전체의 진척(Global Progress)을 이끌어낸다.
  3. 웨이트-프리(Wait-Free): 모든 스레드가 각자의 속도나 다른 스레드의 간섭에 관계없이, 반드시 유한한 연산 단계(\mathcal{O}(1) 또는 유한 다항 시간) 내에 개별 작업(Local Progress)을 마친다. 절대적인 강형 Hard Real-time 시스템에 요구된다.

수학적으로 락-프리는 구조 내에서 Compare-And-Swap(CAS) 연산의 원자성(Atomicity) 무결성에 의존한다. 시스템 상태 벡터를 \mathbf{x} \in \mathbb{R}^n라 할 때, CAS 연산은 이전 상태 추정치 x_{old}와 일치할 때만 새로운 상태 x_{new}로 전이하는 비선형 전이 함수 f(\mathbf{x}, t) 로 취급된다.

3. 원자적 메모리 기저: Compare-And-Swap 연산 구조

대부분의 락-프리 자료구조는 CPU 명령어 집합 수준(x86의 CMPXCHG, ARM의 LDREX/STREX)에서 제공하는 Compare-And-Swap (CAS)를 조립하여 구현된다.

graph TD
    A[Thread 1: Read Shared Variable 'V'] --> B[Calculate new value 'N']
    C[Thread 2: Read Shared Variable 'V'] --> D[Calculate new value 'M']
    B --> E{CAS(V, Old, N)}
    D --> E
    E -- Success (V=Old) --> F[Update V to N]
    E -- Fail (V!=Old) --> G[Retry: Go back to Read V]
    F --> H[Operation Completed Lock-Free]

3.1 C++ std::atomic을 활용한 구현 수학적 표현

PX4의 센서 버퍼나 GCS의 비동기 큐 등을 작성할 때, 단일 변수에 대한 원자적 갱신은 다음의 C++ 의사 코드와 동치이다.

std::atomic<int> shared_state;
int expected = shared_state.load(std::memory_order_relaxed);
int desired;

// CAS Loop (Lock-Free Retry Logic)
do {
    desired = expected + 1; // 연산 수행 (예: MAVLink 시퀀스 넘버 증가)
} while (!shared_state.compare_exchange_weak(expected, desired,
                                             std::memory_order_release, 
                                             std::memory_order_relaxed));

위 루프에서 compare_exchange_weak 함수는 내부적으로 하드웨어 CAS 명령어와 1:1 대응된다. 만약 여러 스레드가 동시에 shared_state를 증가시키려 시도한다 해도, 운영체제의 스케줄러 개입(Mutex Block) 없이 단지 하드웨어 버스 단위의 충돌 해소 메커니즘만으로 순차적 진척이 보장된다.

4. 커스텀 GCS/PX4 아키텍처 연동 및 활용 분야

  1. QGroundControl 수신 스트림: Ardupilot의 Mission Planner와 다르게 QGC는 Qt의 시그널-슬롯 계층이 복잡하게 얽혀 있다. 네이티브 C++ 백엔드(LinkManager)로부터 UI(QML) 컴포넌트로 초당 수백 회씩 수신되는 관성 측정 장치(IMU) 주파수를 락(Lock) 없이 큐잉하려면, Single-Producer Single-Consumer (SPSC) 락-프리 링 버퍼(Ring Buffer) 구조를 구현하여 병목을 완전히 제거해야 한다.
  2. uORB 메시징 버스 최적화: PX4-Autopilot의 uORB 발행/구독(Publish/Subscribe) 내부 구현 역시 락을 최소화하여 센서 데이터가 제어기(Attitude/Position Controller)로 넘어갈 때 시스템 딜레이가 나노초(ns) 단위에 머물 수 있도록 수학적으로 증명된 비차단 동시성 배열 구조를 지속적으로 고도화하고 있다.
