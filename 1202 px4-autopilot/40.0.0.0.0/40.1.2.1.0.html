40.1.2.1. 에르고딕 용량(Ergodic Capacity) 한계점과 텔레메트리 데이터그램 포화 상관관계

장거리 비행이나 도심지 비행과 같이 다중 경로(Multipath) 페이딩이 극심하고 채널 상태가 시변(Time-varying)하는 무인기 통신 환경에서는, 앞서 다룬 정적인 섀넌 채널 용량뿐만 아니라 시간에 따른 평균적인 채널 용량, 즉 에르고딕 용량(Ergodic Capacity)의 개념을 도입하여 분석해야 한다. 본 절에서는 PX4-Autopilot의 MAVLink 텔레메트리 통신 상에서 에르고딕 용량의 한계점이 텔레메트리 데이터그램(Datagram) 포화(Saturation) 현상과 어떻게 수학적으로 상관관계를 갖는지 규명한다.

1. 시변 페이딩 채널에서의 에르고딕 용량(Ergodic Capacity)

드론이 고속으로 이동하거나 주변 지형지물에 의해 전파 환경이 급변할 때, 수신 신호의 신호 대 잡음비(SNR)인 \gamma는 확률 변수(Random Variable)가 된다. 이때 채널 용량 역시 상수가 아닌 확률 변수 C(\gamma)로 취급되며, 장기간 비행하는 드론의 평균적인 통신 가능 용량을 나타내기 위해 에르고딕 용량 \langle C \rangle을 다음과 같이 수식화한다.

\langle C \rangle = \int_{0}^{\infty} B \log_{2}(1 + \gamma) p(\gamma) d\gamma

  • B : 채널의 대역폭 (Bandwidth, Hz)
  • \gamma : 순시 수신 신호 대 잡음비 (Instantaneous SNR)
  • p(\gamma) : \gamma의 확률 밀도 함수 (PDF, Probability Density Function), 예: 레일리(Rayleigh) 또는 라이시안(Rician) 분포

이 수식은 송신단(PX4 기체)에서 채널 상태 정보(Channel State Information, CSI)를 정확히 알지 못하여 송신 전력을 일정하게 유지할 때 달성 가능한 최대 평균 전송률을 의미한다. 만약 MAVLink 데이터 스트림의 발생률(Data Generation Rate)이 \langle C \rangle를 지속적으로 초과하게 된다면, 텔레메트리 링크는 필연적으로 데이터그램 포화 상태에 빠지게 된다.

텔레메트리 데이터그램 포화 메커니즘

텔레메트리 큐 대기 시간과 포화 경로

QGroundControl(GCS)과 PX4 간의 통신은 기본적으로 UDP 기반(IP 링크 시) 또는 라디오 모뎀(Serial 링크 시)을 거치며, 각 노드는 패킷을 송신하기 전 버퍼(큐, Queue)에 저장한다. 에르고딕 용량 한계점을 초과하는 순간 통신 병목 현상이 발생하여 큐 길이(Queue Length)가 무한대로 증가하려는 경향을 보인다.

PX4 펌웨어(v1.14.0 기준)에서는 송신 버퍼가 가득 찰 경우, 최신 비행 자세(ATTITUDE)나 HEARTBEAT와 같이 타임스탬프가 중요한 데이터를 위해 버퍼 내 잔류하는 오래된 데이터그램을 덮어쓰거나(Overwrite) 드랍(Drop)하는 전략을 사용한다. 반면, TCP/IP 연결 방식처럼 신뢰성을 강제하는 경우 버퍼가 막히면 Application Layer (MAVLink 모듈 측) 로직 전체가 블로킹(Blocking)되는 치명적인 결과가 생길 수 있다.

Ardupilot 대비 PX4의 포화 제어 관점

에르고딕 용량이 부족하여 데이터그램이 포화될 때, Ardupilot과 PX4는 서로 다른 설계 철학을 바탕으로 대처한다.

  • Ardupilot (Mission Planner): 통신 링크의 RTT(Round Trip Time)와 패킷 손실률을 분석하여 Mission Planner 측에서 능동적으로 기체에 REQUEST_DATA_STREAM 메시지를 보내 MAVLink 주기를 늦추는(Adaptive Rate Limit) 전략이 널리 사용된다.
  • PX4-Autopilot (QGroundControl): QGC 또한 저대역폭 링크를 감지할 수 있으나, 기본적으로 PX4 내의 MAVLink 스트리밍 인스턴스는 한 번 설정된 모드(Onboard, Normal 등)에 의해 정적(Static)으로 트래픽을 배분하려는 성향이 짙다. 큐 제한이나 포화가 감지되면, 무리한 재전송보다는 실시간성이 떨어진 패킷을 자체 드랍하고 최신 상태만 빠르게 전송하도록(Fire and Forget) 유도하여 비행 제어 지연율을 방어한다.

포화 방지 및 시스템 설계 고려사항

시변 채널 환경을 고려하는 로버스트(Robust) 관제 시스템을 설계할 경우, 개발자는 다음을 준수해야 한다.

  1. 마진(Margin) 설계: 필수적인 MAVLink 데이터 레이트(Required Data Rate, R_{req})가 통신 링크의 에르고딕 용량 \langle C \rangle 대비 50~60% 이하가 되도록 설정하라.
  2. 트래픽 쉐이핑(Traffic Shaping): 하드웨어 모뎀의 버퍼 사이즈를 무조건 키우는 것은 포화를 막지 못하며, 오히려 큐잉 딜레이(Queuing Delay)만 유발시킨다(버퍼블로트, Bufferbloat 현상). PX4 MAV_0_RATE와 같은 파라미터를 보수적으로 설정하여 텔레메트리 생성 단계에서부터 데이터 발생을 억제하라.
  3. 중요도(Severity) 기반 드랍: 포화 시 지상국에서 반드시 수신해야 하는 패킷(COMMAND_ACK, Failsafe 발생 경고 등)에 보장 대역을 따로 할당하라.

결론

단순한 순간 최대 속도가 아닌, 확률 모델링 기반의 에르고딕 용량(Ergodic Capacity)을 분석하는 것은 무인기 텔레메트리 채널의 데이터그램 포화를 예측하는 핵심 근거가 된다. 텔레메트리 설계 시, 데이터 발생률이 평균 용량을 초과하지 않도록 철저한 대역폭 할당 및 상태 공간(State Space) 모델링을 결합해야 치명적인 데이터 지연과 Failsafe 오작동을 예방할 수 있다.
