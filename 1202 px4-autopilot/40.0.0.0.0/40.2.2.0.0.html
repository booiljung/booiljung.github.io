40.2.2. 갈루아 필드(Galois Field) 연산 기반 순환 중복 검사(CRC) 무결성 증명

비행 제어 시스템(Flight Control System)과 지상 관제 시스템(GCS, 예: QGroundControl) 간의 무선 데이터 링크는 전자기 간섭(EMI), 다중 경로 페이딩(Multipath Fading), 또는 물리적 장애물로 인해 빈번한 비트 에러(Bit Error)에 노출된다. PX4-Autopilot 및 Ardupilot 생태계의 표준 통신 프로토콜인 MAVLink는 이러한 패킷 전송 오류를 하드웨어 계층이 아닌 소프트웨어 계층에서 엄격히 검출하기 위해 순환 중복 검사(CRC, Cyclic Redundancy Check)를 채택하고 있다. 이는 단순한 산술합 연산(Checksum)이 아닌 유한 체인 갈루아 필드(Galois Field), 특히 \text{GF}(2) 상의 다항식 나눗셈 연산을 기반으로 하여 수학적으로 철저한 무결성을 증명한다.

1. 갈루아 필드 \text{GF}(2) 연산과 순환 중복 검사의 원리

순환 중복 검사(CRC)의 코어 알고리즘은 0과 1로 구성된 비트스트림을 유한 공간 \text{GF}(2) 상의 다항식(Polynomial) 계수로 간주하여 나눗셈 연산을 수행하는 것에 착안한다. \text{GF}(2)는 집합 \{0, 1\} 내에서 덧셈 연산과 뺄셈 연산이 배타적 논리합(XOR)과 동일하게 취급되고, 캐리(Carry)나 빌림(Borrow)이 발생하지 않는다는 특성을 갖는다.

MAVLink 메시지의 직렬화된 바이트 배열을 M(x)로 두고, 시스템에 사전에 약속된 \text{GF}(2) 생성 다항식(Generator Polynomial)을 G(x)라고 할 때, CRC 값은 M(x)를 G(x)로 나눈 나머지(Remainder) R(x)로 정의된다.
이론적 검증 과정은 다음과 같다.

  1. 송신 측(PX4-Autopilot)은 메시지에 CRC 크기(예: 16비트)만큼의 0을 패딩한 M'(x)를 생성한다.
  2. M'(x)를 G(x)로 \text{GF}(2) 모듈로(Modulo) 나눗셈하여 나머지 R(x)를 산출한다.
  3. 패킷의 꼬리에 R(x)를 덧붙여 전송 프레임 T(x) = M'(x) + R(x)를 완성한다.
  4. 수신 측(QGroundControl)은 수신된 T(x)를 동일한 G(x)로 \text{GF}(2) 나눗셈을 수행하며, 이때 나머지가 정확히 0이 떨어지는 경우 비트 에러가 없다고 수학적으로 무결성을 증명(Integrity Proving)한다.

2. MAVLink 프로토콜의 CRC-16-CCITT 및 CRC Extra 규칙

MAVLink v2.0 프로토콜 규격은 위 연산 체계 중 CRC-16-CCITT 규격을 근간으로 한다. 이 때 사용되는 표준 생성 다항식은 x^{16} + x^{12} + x^5 + 1 (16진수로 0x1021)이다.
여기서 PX4-Autopilot의 MAVLink 구현체가 지니는 가장 독특한 특징은 단순 프레임 전송 오류뿐만 아니라, **버전 불일치 오류(Version Mismatch Error)**까지 검증하기 위해 CRC Extra라는 시드(Seed) 값 해싱 기법을 쓴다는 것이다.

2.1 CRC Extra 증명 매커니즘

수정된 비행 제어기 펌웨어나 각기 다른 버전의 QGroundControl 간에는 동일한 Message ID를 가짐에도 불구하고 내부의 필드(Field) 구조나 바이트 길이가 다를 수 있다. 만일 패킷 길이나 구조가 바뀌었는데도 일반 CRC만 수행한다면 수신 측은 이를 정상적인 패킷으로 오인식할 위험성이 높다.
따라서, MAVLink 파서(Parser)를 생성하는 Python 제너레이터(Generator) 스크립트는 XML 명세 파일을 컴파일할 때, 해당 메시지의 모든 필드명과 타입 문자열을 조합하고 이에 대해 SHA-256 기반의 해싱을 수행하여 고유의 1바이트 요약값, 즉 CRC Extra 바이트를 추출해낸다.
이러한 CRC Extra 바이트는 MAVLink 메시지 직렬화 과정의 가장 마지막에 가상의 데이터로 결합되어 \text{GF}(2) 나눗셈 누적기(Accumulator)를 통과하므로, 서로 다른 XML 명세를 가진 엔티티 간의 통신은 무조건 CRC 실패(CRC Fail)를 유발하게끔 수학적으로 설계되어 있다.

3. PX4와 Ardupilot의 CRC 소스 코드 구현 차이

두 플랫폼 모두 MAVLink C 라이브러리의 mavlink_update_checksum() 인라인 함수를 코어로 활용하지만, 수신된 CRC 무결성이 실패했을 때의 예외 처리 아키텍처는 뚜렷히 구별된다.

  1. PX4-Autopilot의 처리 방식: uORB 메시징 구조상 MAVLink Receiver 모듈 단일 스레드에서 통신 채널 파싱을 전담한다. 패킷 단위의 CRC 무결성 증명에 실패할 경우, 파싱 상태 머신(Parsing State Machine)은 해당 프레임을 즉각적으로 버리고(Drop), 패킷 드롭 카운터(Dropped Packet Counter)를 1 증가시킨 뒤 즉시 매직 바이트(Magic Byte: MAVLink v2 버전에선 0xFD)를 다시 스캔하는 단계로 롤백한다. (Fail-fast 패턴).
  2. Ardupilot의 처리 방식: AP_SerialManager와 GCS_MAVLink 클래스가 인스턴스별로 구성되며, 멀티 링크 환경(예: 텔레메트리 1, 2 공존 시)에서 라우팅(Routing) 오버헤드를 막기 위해 CRC 실패 이벤트를 시스템 로깅(Dataflash Log) 시스템에 적극적으로 푸시한다. 이는 실시간 비행 중 무결성 증명 실패의 패턴을 시각적으로 추적하기 위함이다.

4. QGroundControl 수신 파이프라인에서의 위상학적 검증 모델링

아래 다이어그램은 QGroundControl 백엔드의 C++ Vehicle Manager 객체가 드론으로부터 들어오는 바이트스트림의 무결성을 증명하는 \text{GF}(2) 누적 상태 파이프라인을 나타낸다.

stateDiagram-v2
    [*] --> WAIT_MAGIC_BYTE : 통신 초기화
    WAIT_MAGIC_BYTE --> READ_PAYLOAD_LENGTH : 0xFD 획득
    
    READ_PAYLOAD_LENGTH --> ACCUMULATE_CRC : 헤더 수신 (System ID / Component ID / Msg ID)
    
    ACCUMULATE_CRC --> READ_PAYLOAD : GF(2) 다항식 CRC 누적 시작
    
    READ_PAYLOAD --> APPEND_CRC_EXTRA : 패킷 Payload 순차 수신 및 CRC 누적
    
    APPEND_CRC_EXTRA --> COMPARE_CHECKSUM : Message ID 기반 XML 시그니처 1바이트 가상 누적
    
    COMPARE_CHECKSUM --> MESSAGE_ACCEPTED : 수신된 CRC 꼬리값 == 연산된 CRC 값
    COMPARE_CHECKSUM --> DROP_PACKET : 수신된 CRC 꼬리값 != 연산된 CRC 값
    
    MESSAGE_ACCEPTED --> WAIT_MAGIC_BYTE : uORB / QGC Signal 분배
    DROP_PACKET --> WAIT_MAGIC_BYTE : 상태 머신 오류 카운트 증가

위의 위상 공간 도식(Topology Graph)에서 가장 중요한 트랜지션(Transition)은 APPEND_CRC_EXTRA 상태이다. 이는 직렬화된 실제 물리적 바이트에는 존재하지 않지만 로컬 메모리 테이블 상에만 존재하는 시그니처를 \text{GF}(2) 연산기에 유령(Ghost) 바이트로 주입하여 무결성 필터를 이중막으로 만드는 결정적 핵심 역할을 담당한다.

5. 결론

갈루아 필드 \text{GF}(2) 상의 나눗셈을 이용한 CRC 무결성 증명 체계는 단순 데이터 변조 검출을 넘어서 비행 제어 소프트웨어 간의 호환성, 즉 XML 프로토콜 명세의 동기화 여부까지 수학적으로 엄격하게 보증한다. PX4-Autopilot 환경 하드웨어 포팅이나 사용자 지정 MAVLink 메시지(Custom MAVLink Message)를 등록하는 시스템 개발자는 반드시 GCS 빌드 체인과 펌웨어 빌드 체인이 참조하는 common.xml 등 프로토콜 정의 파일의 해시 동기화를 이루어야 하며, 이를 간과하면 올바른 물리선 맵핑에도 불구하고 끝없는 무결성 검사 실패(CRC Error) 루프에 빠지게 될 것임을 강력히 명심하라.
