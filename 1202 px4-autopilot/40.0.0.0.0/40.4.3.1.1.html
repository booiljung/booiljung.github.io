40.4.3.1.1. 논리적 시계(Logical Clock)를 이용한 부작용(Side-effect) 발현 억제 벡터(Vector) 제어

PX4-Autopilot(펌웨어 버전 1.14 이상 기준)과 QGroundControl(버전 4.3 이상) 환경에서 지상 관제 시스템(GCS, Ground Control Station)과 무인항공기(UAV, Unmanned Aerial Vehicle) 간의 비동기적 통신망은 메시지 전송 지연과 순서 역전 현상에 취약하다. 이를 해결하기 위해 논리적 시계(Logical Clock) 알고리즘, 특히 램포트 타임스탬프(Lamport Timestamp) 또는 벡터 시계(Vector Clock) 방식이 분산 제어 시스템 설계에서 점차 중요하게 다뤄지고 있다. 네트워크 통신에서는 이전 상태의 지연된 수신으로 인해 기체가 의도치 않은 동작(Side-effect)을 수행할 위험이 있으며, 이를 방지하기 위한 억제 벡터(Suppression Vector) 제어 메커니즘을 상세히 분석한다.

1. 분산 시스템에서의 논리적 시계(Logical Clock) 도입 배경

드론 제어 시스템은 GCS, 비행 제어기(FC, Flight Controller), 보조 컴퓨터(Companion Computer) 등이 상호 작용하는 전형적인 분산 시스템(Distributed System)이다. 물리적 시계(Physical Clock) 동기화 오류는 네트워크 지연이나 시스템 부하 발생 시 제어 명령의 인과적 순서를 왜곡시킬 수 있다.

따라서 상태 업데이트 필터링을 위해 각 메시지에 단조 증가(Monotonically Increasing)하는 논리적 시계 카운터를 부여하여, 제어 명령의 시공간적 인과 관계(Causality)를 엄격히 보존하라. 램포트의 시간 이론에 따라 논리적 시간 L(e)가 주어진 이벤트 e에 대해, a \rightarrow b (이벤트 a가 b보다 선행함)라면 L(a) < L(b)가 항상 성립하도록 설계해야 한다.

2. 부작용(Side-effect) 발현 억제를 위한 상태 필터링

PX4-Autopilot은 GCS로부터 수신된 설정, 경로, 제어 명령을 처리하는 과정에서 멱등성(Idempotency)을 부분적으로 지원한다. 하지만 특정 연속 동작(예: 이륙 후 웨이포인트 변경, 오프보드 기동 상태 업데이트)의 경우 동일한 패킷이 지연 전송(Delayed Transmission)되어 늦게 도달하거나 순서가 뒤섞이면 내부 상태 머신(State Machine)에 치명적인 부작용이 발현된다.

이러한 부작용을 원천적으로 억제하기 위해 수신단은 **억제 벡터(Suppression Vector)**를 유지하라. 내부에 저장된 이전 벡터 상태와 비교하여, 타임스탬프가 낮거나 이미 처리 완료된 논리적 시계 값을 가진 MAVLink 명령을 즉각 식별하여 폐기(Drop)하는 상수 시간(O(1)) 구조의 정책을 구비해야 한다.

3. 벡터 제어(Vector Control) 메커니즘과 통신 다이어그램

각 서브시스템(예: GCS, PX4 커맨더(Commander), ROS2 에이전트)은 자신만의 독립적인 논리적 시계 카운터를 관리한다. 상태 업데이트 발생 시, 각 모듈의 벡터값을 전파하고 동기화하여 글로벌 상태 합의(Global State Consensus)를 달성하라.

sequenceDiagram
    participant GCS as GCS (Lamport Clock: C)
    participant Companion as Companion Computer (ROS2)
    participant PX4 as PX4-Autopilot (FC)

    GCS->>PX4: MAVLink COMMAND_LONG [C=10]<br>Mode: Offboard
    PX4-->>GCS: COMMAND_ACK [C=10]<br>Vector Updated: {G:10, P:1}
    
    GCS->>PX4: MAVLink SET_POSITION_TARGET [C=11]
    Note over GCS, PX4: Network Delay Occurs
    
    Companion->>PX4: uXRCE-DDS Trajectory Setpoint [C=12]
    PX4-->>Companion: ACK [C=12]<br>Vector Updated: {G:10, P:2, C:12}
    
    Note over PX4: Delayed GCS Packet Arrives
    PX4->>PX4: Check Logical Clock<br>GCS [C=11] vs Internal State (C >= 12)
    PX4->>PX4: Suppress Side-effect (Drop Packet)

위 다이어그램은 ROS2(Companion Computer) 기반 에이전트와 GCS가 동시에 제어 권한을 행사하려 할 때, 네트워크 정체로 늦게 도착한 과거의 GCS 제어 정보가 현재의 기동에 오개입(Side-effect 발현)하는 것을 내부 벡터 시계를 통해 억제 및 폐기결정(Drop)하는 과정을 명확히 보여준다.

4. Ardupilot과 PX4-Autopilot의 통신 동기화 구조 차이

Ardupilot의 통신 스택은 상대적으로 강력한 물리적 동기화(Time Synchronization) 체계를 주로 활용하며, 연속적인 heartbeat 주기 및 타임아웃 윈도우에 엄격하게 의존하여 통신 손실과 패킷 역전을 판단한다.
반면 PX4-Autopilot 아키텍처는 내부 uORB(Micro Object Request Broker) 버스와 단단히 결합되어 있으며, 서브모듈 간 고도의 비동기 메시지 패싱(Asynchronous Message Passing)을 근간으로 작동한다.

이러한 설계 차이에 기인하여 PX4 환경에서 복수의 지휘 제어 체계(Multi-GCS 운영 또는 GCS + 다수 ROS2 노드 구조)를 구성할 때는 MAVLink의 단순 타임스탬프 의존성을 넘어, 애플리케이션(QGroundControl 및 사용자 ROS2 노드) 계층에서 명령의 논리적 흐름 및 인과율을 자체적으로 제어 및 보정하는 논리 벡터 식별 규약(Vector Clock Identification Protocol)을 필히 구현해야 한다.

5. 아키텍처 설계 시 실무 고려 사항

  • 카운터 오버플로우 방지 및 방어: 논리적 시계 구현 시 uint64_t와 같은 충분한 크기의 자료형을 활용하거나 순환형 윈도우(Circular Window) 제어 코드를 적용하여 단조 증가 시계의 롤오버(Rollover) 문제로 촉발되는 순서 역전 오류를 철저히 방지하라.
  • 통신 대기시간(Latency) 평가와의 결합: 지연된 메시지를 억제하는 것은 시스템 안전(Safety)을 확보하는 방어적 수단이다. 따라서 무인기 제어 정확도(Precision)는 QGroundControl과 기체 간 네트워크 지연 튜닝 및 MAVLink 최적화로 본질적인 문제를 선형적으로 해소해 나가야 한다.
  • MAVLink 확장 적용: MAVLink v2.0 프로토콜 규격은 데이터 무결성을 위해 메시지 서명(Signature) 메커니즘을 지원하지만, 인과성 제약을 직접 보장하지는 않는다. 고수준의 제어 앱에서는 COMMAND_LONG 전송 시 페이로드 또는 시퀀스 필드에 논리적 시계 해시값을 삽입하여 비행 제어기 측 억제 로직과 수직으로 결합하라.

본 접근법은 소프트웨어 공학과 분산 컴퓨팅(Distributed Computing)의 클래식한 동시성 제어(Timestamp-based Concurrency Control) 이론을 적용한 핵심적 설계 메커니즘이다. 이를 자율 비행 응용 소프트웨어에 반영하여, 예측 불허한 무선 네트워크 환경에서도 에이전트의 안정성을 견고히 보장하라.
