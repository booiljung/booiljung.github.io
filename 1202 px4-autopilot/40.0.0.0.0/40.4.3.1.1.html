40.4.3.1.1. 논리적 시계(Logical Clock)를 이용한 부작용(Side-effect) 발현 억제 벡터(Vector) 제어

여러 대의 지상 관제 시스템(GCS)이 단일 기체를 제어하거나, 군집 비행(Swarm Flight) 중인 다수의 드론이 메쉬(Mesh) 네트워크 망을 통해 통신할 때, 분산 시스템(Distributed System)의 고질적인 문제인 패킷 도달 순서 역전(Out-of-order Delivery) 현상이 빈번하게 발생한다. 특히 반송파가 끊기거나(Drop) 다중 경로 페이딩(Multipath Fading)으로 인해 재전송(Retransmission)이 일어날 경우, 과거에 내린 제어 명령이 최신 명령보다 늦게 기체에 도착할 수 있다. 본 절에서는 이러한 비동기적 통신 환경에서 구식(Stale) 명령으로 인한 의도치 않은 역주행 및 상태 붕괴(Side-effect)를 원천 차단하기 위해, PX4-Autopilot과 QGroundControl 간 통신에서 논리적 시계(Logical Clock)와 벡터(Vector) 제어를 어떻게 활용하는지 탐구한다.

1. 램포트의 논리적 시계(Lamport’s Logical Clock) 기반 인과율 제어

분산 시스템 이론에서 레슬리 램포트(Leslie Lamport)가 창안한 논리적 시계 모델은, 노드 간의 물리적 시간(Physical Time, 동기화가 불완전한 GPS 시간이나 RTC)에 전적으로 의존하는 대신 이벤트의 선후 인과 관계(Causality)를 수학적 단조 증가(Monotonically Increasing) 카운터로 추적하는 기법이다. MAVLink 프로토콜의 패킷 헤더(Header)에 존재하는 시퀀스 번호(Sequence Number) 필드가 이 논리적 시계의 가장 훌륭한 물리적 실체다.

PX4-Autopilot 수신단에서는 특정 시스템 및 컴포넌트(target_system, target_component)별로 가장 마지막에 성공적으로 수신 및 처리한 논리적 시계 값(벡터 V_{sys})을 유지한다. GCS로부터 새로운 제어 명령 패킷 p가 도달하면 수신된 패킷의 논리 시계 V(p)와 로컬 벡터를 다음과 같이 비교한다.

V(p) \le V_{sys}

만약 V(p) 가 작거나 같다면, 이는 네트워크 상에서 지연되어 뒤늦게 도착한 “과거의 이벤트“로 간주된다. 즉, 이미 더 미래의 인과율(V_{sys})이 시스템 상태에 반영되었으므로, 해당 패킷의 명령어(Command)나 제어 집합점(Setpoint) 업데이트는 시스템의 불변성을 깨뜨릴 부작용(Side-effect) 인자로서 판별되어 즉각 폐기(Discard)된다.

2. 벡터 제어(Vector Control)를 통한 다차원 부작용 억제

단일 GCS와 단일 드론 간에는 스칼라(Scalar) 형태의 시퀀스 넘버만으로도 인과율 검증이 가능하지만, 다중 GCS(Multi-GCS) 운영이나 ROS2 기반 다중 노드(Multi-node) 연결체계에서는 각 명령 주체별 독립적인 시계선(Clock-line)을 다변량 벡터 형태로 파악해야 한다.

\mathbf{V}_{sys} = [v_{GCS1}, v_{GCS2}, v_{ROS2\_Companion}]

PX4-Autopilot(v1.14 기준) 내부의 MAVLink 수신기(mavlink_receiver.cpp)나 uXRCE-DDS 클라이언트(Client)는 수신되는 메시지의 송신자 식별자(System ID 및 Component ID)를 인덱스로 삼아 다차원 배열(Vector 배열) 형태로 타임스탬프와 시퀀스 상태를 기록한다.

  • 예제 시나리오: QGroundControl(System ID 255)이 기체에 이동(Goto) 명령(v_{GCS1}=5)을 내린 직후, 부착된 라즈베리 파이(Companion Computer, System ID 1)가 컴퓨터 비전 기반 충돌 회피를 위해 긴급 비행 정지(Hold) 명령(v_{ROS2\_Companion}=12)을 발행했다고 가정하라.
  • 부작용 방어: 만약 네트워크 지연으로 인해 QGC의 Goto 명령(v_{GCS1}=5)이 늦게 도달하여 Hold 상태를 덮어쓰려고 시도한다면, 각 제어기 주체 간의 우선순위 가중치 텐서(Tensor)와 결합된 논리적 시계 검증 로직이 과거의 임무 재개를 거부함으로써 기체 충돌이라는 치명적 부작용(Side-effect)을 억제한다.

2. Ardupilot 대비 PX4의 오프보드(Offboard) 벡터 제어 우위

Ardupilot은 전통적으로 RC(Remote Controller) 입력이나 단일 GCS 통제에 최적화되어 있어, 시퀀스 유실이나 지연 명령에 대해 큐(Queue)를 버리거나 단순히 타임아웃 시간(Time-to-Live)에 의존하는 경향이 강했다.

반면, 컴퓨터 비전 기반 자율 주행 연구에 깊이 뿌리를 둔 PX4-Autopilot 아키텍처는 수많은 외부 제어 스크립트(MAVSDK, ROS2)가 동시다발적으로 개입하는 상황을 상정하여 설계되었다. 특히 Offboard 모드 제어에서는 초당 50Hz 이상으로 쏟아지는 제어 위치/속도 벡터(Vector) 입력에 대하여 데이터 노후화(Staleness)를 엄격히 감시한다. MAVLink 데이터나 uORB 토픽의 수신 타임스탬프(물리적 누적 시계 hrt_absolute_time)와 시퀀셜 플래그를 결합 처리하여, 시스템 상태가 과거의 제어 벡터로 퇴행(Regression)하는 것을 방지하는 아키텍처가 매우 훌륭하게 구현되어 있다.

3. 논리적 시계 및 상태 전이 다이어그램

아래의 다이어그램은 네트워크 지연 환경에서 논리적 시계가 구식 명령(Stale Command)을 어떻게 걸러내어 기체 상태 오염을 방지하는지 시각적으로 보여준다.

sequenceDiagram
    participant GCS1 as QGroundControl
    participant GCS2 as Secondary GCS (Observer)
    participant Net as High-Latency Network
    participant PX4 as PX4-Autopilot<br>(Vector Clock State)

    GCS1->>Net: Cmd A: Waypoint 1 (Seq: 10)
    Note over Net: Packet A Delayed (Congestion)
    
    GCS1->>Net: Cmd B: Hold Position (Seq: 11)
    Net->>PX4: Receive Cmd B (Seq: 11)
    Note over PX4: Vector Updated: {GCS1: 11}<br>State -> Hold
    PX4->>GCS2: MAVLink Telemetry (Status: Hold)
    
    Net->>PX4: Receive Cmd A (Seq: 10) [Delayed]
    Note over PX4: Check Logical Clock:<br>10 < 11 (V(p) < V_sys)
    Note over PX4: Stale Command Detected!<br>Side-effect Suppressed (Drop)
    
    PX4->>GCS1: COMMAND_ACK (Result: DENIED / IGN_STALE)

이 시퀀스 다이어그램에서 통신 지연으로 인해 뒤늦게 도착한 Cmd A (Seq: 10) 패킷은 PX4의 논리적 시계 필터에 의해 부작용 인자로 차단된다. 이를 통해 PX4-Autopilot은 이기종 다중 연결 환경에서도 시스템 내부 변수나 비행 궤적이 손상되지 않고, 가장 최신의 의도(Intention)만을 추종하는 강건한(Robust) 제어 안정성을 달성한다.
