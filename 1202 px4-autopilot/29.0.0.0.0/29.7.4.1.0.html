29.7.4.1. 상태 머신 매니저의 팩토리 패턴(Factory Pattern)을 수정하여 사용자 정의 특수 비행 모드 클래스를 의존성 주입(Dependency Injection)하는 펌웨어 수정 방법

PX4-Autopilot의 최신 코드 베이스(v1.10 이후)는 수동 및 자동 비행 제어 로직을 스파게티 코드에서 분리해내어, 객체 지향(OOP) 소프트웨어 공학의 정수라 불리는 디자인 패턴들을 대거 채용했다. 그 중 핵심이 비행 모드를 생성하고 관리하는 FlightModeManager 모듈의 **팩토리 패턴(Factory Pattern)**과 의존성 주입(Dependency Injection) 아키텍처이다.

본 절에서는 개발자가 자신만의 수학적 모델을 담아 구현한 FlightTaskCustom 클래스를 메인 펌웨어의 루프(Loop) 안에 물리적으로 하드코딩하지 않고, 객체 지향 팩토리를 통해 우아하게 주입(Injection)하는 메커니즘을 소스 코드 예제와 함께 분석한다.

1. FlightModeManager와 팩토리 패턴(Factory Pattern) 아키텍처

PX4 비행 제어의 중심인 커맨더(Commander) 프로세스가 현재 상태(State)에서 새로운 비행 모드(예: Position \rightarrow Custom Acro)로의 진입 허가를 내리면, vehicle_status 메시지를 통해 FlightModeManager에 모드 전환 이벤트가 통보된다.

이때 FlightModeManager는 기존 태스크(Task)의 메모리를 소멸(Destruction)시키고, 팩토리 인터페이스(FlightTaskFactory)를 호출하여 새 모드의 인스턴스를 동적 메모리 힙(Heap) 공간에 인스턴스화한다.

graph TD
    A[조종기 스위치 조작 또는 GCS 커맨드] --> B[Commander 모듈 수신]
    B -->|모드 전환 결의| C[vehicle_status 발행]
    C --> D[FlightModeManager 이벤트 검출]
    D --> E{FlightTaskFactory 호출}
    E -->|case: TASK_POSITION| F[FlightTaskAuto 객체 주입]

    G --> H[제어 루프 (mc_pos_control / mc_att_control) 인수]

2. 팩토리 소스 코드 내 사용자 정의 모드 의존성 주입(Injection)

개발자가 작성한 FlightTaskCustom.cpp 로직을 팩토리에 등록하기 위해서는 src/modules/flight_mode_manager/FlightTaskFactory.cpp 내의 switch 분기문 인터페이스를 수정해야 한다.

PX4의 팩토리는 정적 메모리 최적화를 위해 무분별한 new 연산자를 지양하고, 시스템이 사전 할당한(pre-allocated) 메모리 밴드 안에서 객체를 초기화(Placement New)하는 고급 메모리 주입 방식을 사용한다.

// src/modules/flight_mode_manager/FlightTaskFactory.cpp 의사 코드 라우팅
FlightTask *FlightTaskFactory::getTask(FlightTaskIndex index)
{
    switch (index) {
        case FlightTaskIndex::Position:
            // 순정 비행 모드 주입
            return _instantiateTask<FlightTaskPosition>();

        case FlightTaskIndex::ManualAngle:
            return _instantiateTask<FlightTaskManualAngle>();

        // [추가된 커스텀 모드]
        case FlightTaskIndex::CustomMode:
            // Custom 클래스를 Factory의 포인터 컨테이너에 의존성 주입
            return _instantiateTask<FlightTaskCustom>();

        default:
            return nullptr;
    }
}

위의 주입 인터페이스(_instantiateTask)를 통해 주입된 FlightTaskCustom 객체는 이제 최상위 메인 루프에서 다형성(Polymorphism)을 띠는 베이스 포인터(FlightTask *)로 다뤄지며, 매 틱(Tick) 마다 update() 함수가 자동으로 래핑되어 호출된다.

3. MAVLink 커스텀 모드 대응 및 QGC 트리거 확장 제어

팩토리 레포지토리에 커스텀 객체를 심어 넣었어도, 외부(조종사 스위치 또는 MAVLink)에서 이 객체를 트리거(Trigger)할 수 있는 채널이 마련되지 않으면 무용지물이다. Ardupilot의 고정된 비스트럭처형 비행 모드 비트마스킹과 달리, PX4-Autopilot은 vehicle_command uORB 토픽을 통해 모드 진입 명령을 파이프라이닝한다.

  1. 내부 ENUM 정의: src/include/uORB/topics/vehicle_status.h 내부의 헤더에 커스텀 모드를 위한 메타 플래그 식별자(예: NAVIGATION_STATE_CUSTOM)를 추가한다.
  2. 커맨더(Commander) 트랜지션 로직 승인: src/modules/commander/state_machine_helper.cpp 내부의 유효성 검사 분기에 개발자가 정의한 모드로 진입하기 전 필요한 조건을 명시한다 (예: GPS Fix가 필수인지 여부).
  3. MAVLink Custom Sub-mode 매핑: mavlink_receiver.cpp 내에서 GCS로부터 날아온 VEHICLE_CMD_DO_SET_MODE 커맨드 패킷의 custom_mode 필드를 파싱하여, 내부 uORB NAVIGATION_STATE_CUSTOM 신호로 변환하여 뿌리도록 브릿지 코드를 개설한다.

결과적으로 최신 PX4의 확장 가능성(Extensibility)은 펌웨어 전체를 갈아엎지 않고도 FlightModeManager의 팩토리 인터페이스 단 한 곳에서 클래스 포인터 스위칭만 지원하면, 남은 모든 제어와 안전 루프는 엔진의 다형성을 통해 자동으로 상속된다는 가장 강력한 소프트웨어 아키텍처적 이점을 선사한다.
