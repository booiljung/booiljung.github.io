29.1.1.1. FlightModeManager.cpp의 메인 실행 루프 및 이벤트 폴링(Event Polling) 주기 최적화 코드 분석

본 소절에서는 FlightModeManager의 심장부라 할 수 있는 FlightModeManager.cpp 소스 코드를 해부하여, 비행 상태 머신의 메인 실행 루프(Main Execution Loop)가 어떠한 방식으로 구동되며, uORB 토픽의 이벤트 폴링(Event Polling)을 통해 CPU 점유율을 어떻게 극단적으로 최적화하는지 분석한다.

1. 워크 큐(Work Queue) 기반의 비동기적 Run() 메서드 구조

과거 병렬 처리 아키텍처에서 흔히 사용되던 무한 루프 블로킹(Blocking) 방식은 실시간 운영체제(RTOS) 환경에서 타겟 스레드가 무한 자원을 점유하여 다른 중요한 태스크의 실행 타이밍을 앗아갈 위험을 내포한다. 이를 원천 차단하기 위해 현재의 PX4 FlightModeManager 모듈은 px4::WorkItem 클래스에서 상속받은 Run() 메서드를 단일 진입점으로 활용하는 비동기 틱(Asynchronous Tick) 연산 구조를 취하고 있다.

void FlightModeManager::Run()
{
    if (should_exit()) {
        _local_pos_sub.unregisterCallback();
        return;
    }

    // 연산 시간 및 타이밍 성능 캐싱(Caching) 성능 프로파일링
    perf_begin(_loop_perf);

    // 1. 핵심 uORB 토픽들의 구독 검사 및 상태 업데이트
    CheckForUpdates();

    // 2. 현재 활성화된 비행 모드의 비행 태스크(Flight Task) 실행 루프
    RunCurrentMode();

    perf_end(_loop_perf);
}

제시된 핵심 뼈대 코드에서 입증되듯, Run() 함수는 스케줄러에 의해 호출될 때마다 독립된 단일 연산 프레임을 처리하고 지체 없이 워크 큐 스케줄러로 스레드 제어권을 반환(Yield)한다. 이는 전형적인 이벤트 드리븐(Event-driven) 상태 머신의 구조형이다. 코드 프로파일링을 위한 perf_begin()과 perf_end() 매크로는 해당 블록에서 FlightModeManager가 실제로 소비하는 마이크로초(us) 단위의 CPU 체공 시간을 실시간으로 로깅(Logging)하여, 임베디드 리눅스 또는 NuttX 시스템 하에서 비정상적인 지연(Latency) 현상을 개발자가 탐지할 수 있도록 보조한다.

2. uORB 콜백(Callback) 연동을 통한 스마트 폴링(Smart Polling)

FlightModeManager 비행 제어 루프가 구동되어야 하는 당위성은 명확하다. 외부 환경 요인(설정된 조종기 짐벌 스틱 값의 변경이나 EKF 추정기의 위치 파라미터 갱신 등)이 변화했을 때, 이를 평가하고 기체의 기구학적 움직임 목표치(Setpoint)로 환산해 내기 위함이다. 만약 데이터의 근본적인 갱신이 없음에도 무의미하게 높은 고정 주파수(예: 1000Hz)로 본 루프를 강제 폴링(Polling)한다면, 시스템 전반에 치명적인 CPU 오버헤드(Overhead)가 유발된다.

PX4-Autopilot 아키텍처는 이 교착 문제를 극복하기 위해 uORB 통신 미들웨어의 콜백 등록(Callback Registration) 소프트웨어 패턴을 적극 차용한다. FlightModeManager의 초기화(init()) 단계 소스 코드를 면밀히 살펴보면 다음과 같은 구문이 배포되어 있다.

// EKF2 등에서 발행되는 로컬 위치 토픽에 실행 콜백을 거는 로직
if (!_local_pos_sub.registerCallback()) {
    PX4_ERR("vehicle_local_position callback registration failed!");
}

여기서 _local_pos_sub는 시스템 통신 버스 상의 vehicle_local_position 토픽을 독점 구독하는 uORB 서브스크립션(Subscription) 객체이다. registerCallback() 연산을 호출하게 되면, 해당 uORB 버스 토픽에 새로운 데이터 패킷이 발행(Publish)되는 정확히 그 찰나의 순간에, 운영체제 하위 스케줄러 인터럽트가 FlightModeManager가 종속된 워크 큐를 스레드 대기 상태에서 깨워(Wake-up) Run() 함수를 1회 점화(Trigger)시킨다.

즉, FlightModeManager의 연산 사이클은 내부적인 하드웨어 고정 타이머(Timer) 클럭에 제한적으로 의존하는 것이 아니라, 비행체의 실시간 동역학 상태를 가장 정밀하게 대변하는 EKF 추정기의 필터 갱신 주기(일반적으로 50Hz ~ 250Hz)에 완벽히 위상 동기화(Phase Synchronization) 된 상태로 구동된다. 이는 운영체제의 CPU 자원 점유율을 최적화할 뿐만 아니라, 물리적 센서 데이터 취득 시점부터 액추에이터 제어기 커맨드 산출까지 직선으로 이어지는 파이프라인의 이산 타임 딜레이(Phase Delay)를 소프트웨어 아키텍처 상으로 최소화하는 공학적 쾌거이다.

3. vehicle_control_mode 폴링과 예외 처리(Edge Case Handling)

방어 로직의 일환으로 CheckForUpdates() 프로세스 내부에는 조종 명령의 핵심 지표 격인 vehicle_control_mode 토픽에 대한 정기 폴링(Polling) 및 폴백(Fallback) 로직이 추가로 내장되어 있다. 로컬 위치 토픽 콜백이 이 모듈 루프 전체의 박자(Metronome) 주기를 주도한다면, 모드 제어 토픽은 스위칭 밸브 역할을 수행하게 된다.

만약, 텔레메트리(Telemetry) 링크나 무선 조종기(RC) 링크의 이상 소실로 인해 vehicle_control_mode 토픽 버스의 갱신이 비정상적으로 장기화되어 멈추는 데이터 기아(Data Starvation) 상황이 발생할 경우를 대비하여 시스템 커널에는 타임아웃(Timeout) 데이터 한계 방어선이 구축되어 있다. 일정 누적 시간(timeout_us)이 초과하도록 모드 명령이 유출되지 않으면, 메인 루프는 동적으로 활성화된 현재의 하위 FlightTask를 비활성(Inactive) 상태로 강제 전환하거나, 하드 리셋된 페일세이프(Failsafe) 비행 태스크를 강제 할당하여 무인항공기의 무동력 추락이나 시스템 락(Lock) 현상을 치명적으로 거부하는 고도의 안전망(Safety Net)을 가동하게 된다.
