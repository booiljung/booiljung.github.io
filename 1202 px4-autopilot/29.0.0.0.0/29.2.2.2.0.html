29.2.2.2. 캘리브레이션 트림(Trim) 값 오차에 따른 비대칭(Asymmetric) 스케일링 보상 수학적 모델 소스 레벨 검증

본 소절에서는 ManualControl 모듈의 스틱 정규화(Normalization) 과정 중, 물리적 조종기 짐벌(Gimbal)의 스프링 장력이 완벽한 대칭을 이루지 못해 필연적으로 발생하는 비대칭 트림(Asymmetric Trim) 오차를 수학적으로 어떻게 보상(Compensation)하고 선형 스케일링(Linear Scaling)하는지 그 심층적인 C++ 커널 구현 모델을 검증한다. 앞선 29.2.2.1절의 기본 선형 보간 로직을 확장하여, 기구학적 오작동을 유발하는 불균형 요소를 소프트웨어 알고리즘으로 극복하는 공학적 사상을 다룬다.

1. 물리적 조종기 짐벌의 비대칭성 한계와 제어 오차의 발현

현존하는 RC 송신기의 가변 저항(Potentiometer)이나 홀 센서(Hall Effect Sensor) 기반 조종 스틱은 기계적 한계를 지닌다. 이상적인(Ideal) 조종기는 스틱을 완전히 내렸을 때(Min) 1000\mu s, 완전히 올렸을 때(Max) 2000\mu s를 출력하며 손을 놓아 스프링에 의해 중앙으로 복귀했을 때(Trim) 정확히 1500\mu s를 출력해야 한다. 즉, 하단 행정 거리(1500 - 1000 = 500\mu s)와 상단 행정 거리(2000 - 1500 = 500\mu s)가 완벽한 대칭(Symmetric)을 이루어야 한다.

그러나 양산형 조종기는 스프링의 장력 편차, 센서의 조립 오차, 플라스틱 기구물의 마모 등으로 인해 Min 988\mu s, Max 2015\mu s, Trim 1520\mu s 와 같이 극심한 비대칭(Asymmetric) 상태로 영점이 틀어지는 현상이 빈번하게 발생한다. 만약 PX4 커널이 \frac{\text{raw} - \text{min}}{\text{max} - \text{min}} \times 2.0 - 1.0 과 같은 단순 비례 1차 함수로 스케일링을 수행할 경우, 스틱에서 손을 뗀 영점(1520\mu s) 상태에서 정규화된 제어값(Set-point)은 0.0f가 아닌 +0.04f 따위의 오프셋(Offset)을 띠게 된다. 이 미세한 잔류 오프셋은 드론이 호버링(Hovering) 중에도 끊임없이 한쪽 방향으로 흐르는(Drifting) 치명적인 제어 왜곡을 초래한다.

2. 구간 분리형 비대칭 스케일링(Piecewise Asymmetric Scaling) 보상 모델 레이어

PX4의 apply_calibration() 함수 및 캘리브레이션 모듈은 이러한 기계적 불균형을 원천적으로 흡수하기 위해, 전체 행정을 트림(Trim)을 기준으로 확실히 반분(Bisection)하여 상단과 하단에 각각 독립적인 일차 함수 기울기(Gradient)를 적용하는 ’구간 분할형 비대칭 모델’을 가동한다.

2.1 하단부(Bottom Half) 스케일링 수학적 수식 모델

원시 입력 값(R_{raw})이 중앙 트림 값(R_{trim})보다 작을 때, 즉 조종기를 50% 아래로 조작했을 때는 오직 R_{min} 과 R_{trim} 매개변수만이 계산에 관여한다.
이 구간에서의 정규화된 값 N_{out} \in [-1.0, 0.0] 의 산출 공식은 다음과 같다.

N_{out} = -1.0 \times \frac{R_{trim} - R_{raw}}{R_{trim} - R_{min}}

이 수식에서는 상단 한계값(R_{max})이 완전히 배제된다. 오직 하단의 실제 물리적 가동 범위 격차 대비 현재 스틱이 얼마나 아래로 내려갔는가 하는 비율만을 추출하기 때문에, 스프링 장력이 비대칭이더라도 스틱을 최하단으로 내렸을 때는 예외 없이 정확하게 -1.0f 에 수렴하게 된다. (데드존이 포함된 더욱 정교한 모델은 29.2.2.1절 수식 참조)

2.2 상단부(Top Half) 스케일링 수학적 수식 모델

반대로 R_{raw} 가 R_{trim} 보다 클 때, 조종기를 50% 위로 조작한 구간에서는 오직 R_{trim} 과 R_{max} 만이 연산 변수로 작용한다.
정규화 값 N_{out} \in [0.0, 1.0] 의 산출 공식은 다음과 같다.

N_{out} = \frac{R_{raw} - R_{trim}}{R_{max} - R_{trim}}

위의 상하단 분리 분기(Branching) 아키텍처는 수학적으로 트림(Trim) 지점에서 완벽한 연속성(Continuity)을 만족하면서도, 기하학적 관점에서 보면 트림을 기점으로 꺾인 V자 형태(또는 두 1차 함수의 접합)의 기울기 특성을 띠게 된다.

3. C++ 런타임 구현의 함정과 0 나누기(Divide-by-Zero) 예외 처리 방어기제 코드 분석

이러한 비대칭 구간 모델을 C/C++ 커널 코드로 이식할 때, 가장 치명적인 잠재 결함은 바로 ‘0 나누기(Divide-by-Zero)’ 예외(Exception)와 부동 소수점 NaN (Not a Number) 발산이다. 사용자 실수 혹은 QGroundControl의 캘리브레이션 툴 오작동으로 인해 시스템 파라미터가 잘못 기입되어 R_{trim} = R_{min} 또는 R_{trim} = R_{max} 로 동일하게 셋팅되는 우발적 사태가 벌어질 수 있기 때문이다.

// 비대칭 캘리브레이션 적용 시 런타임 예외 방비 처리 매크로 및 구문 발췌본
float apply_calibration(int16_t raw, int16_t min, int16_t center, int16_t max, uint16_t dz) 
{
    // ... (상단/하단 분기 판단 로직) ...

    if (raw > center) {
        // [경고 방어] 분모가 0 이하가 되어 무한대(Infinity)로 발산하는 치명적 런타임 재앙 사전 차단
        int16_t travel_limit = max - (center + dz);
        if (travel_limit <= 0) {
            return 0.0f; // 잘못된 캘리브레이션 파라미터인 경우, 제어권을 잃더라도 시스템 충돌은 피하기 위해 0 리턴
        }
        return (float)(raw - (center + dz)) / (float)travel_limit; // 안전한 부동 소수점 연산

    } else if (raw < center) {
        int16_t travel_limit = (center - dz) - min;
        if (travel_limit <= 0) {
            return 0.0f; // 마이너스 범위 역전 및 0 나누기 동일 방어
        }
        return -(float)((center - dz) - raw) / (float)travel_limit;
    }
    
    // ...
}

PX4 코어 개발진은 위 코드와 같이 분수 연산 직전에 분모인 travel_limit 변수의 부호와 0 여부를 엄격하게 조건문 탐색하여(<= 0), 제어기가 나누기 불능 오류 상태에 진입하여 시스템 전체가 다운(Watchdog Timeout 등)되는 참사를 커널 초입에서 결사 저지한다. 만약 잘못된 파라미터가 감지되면, 과감히 해당 축의 조종 권한을 0.0f 정지 상태로 강제 포기(Fallback)시킴으로써 나머지 모듈들이 무의미한 쓰레기 데이터에 휘둘리는 것을 방어하는 다중 안전망(Failsafe) 철학을 소스 코드에 녹여내었다. 이로써 PX4 시스템은 아무리 심하게 비틀린 하드웨어 조종기 물리 엔진조차도 오류 없이 우아하고 강건하게(Robustly) 포용해 낸다.
