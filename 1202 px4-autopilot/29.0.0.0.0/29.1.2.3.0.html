29.1.2.3. 출력단: 목표 자세, 목표 각속도, 최종 액추에이터 제어 명령으로 이어지는 제어 권한 위임(Delegation) 소스 흐름

본 소절에서는 FlightModeManager의 스위칭 밸브를 빠져나와 물리적 의미를 부여받은 궤적 스칼라가 기체 동역학(Rigid Body Dynamics) 피드백 제어 루프의 폭포수를 차례로 통과하며, 목표 자세(Attitude), 목표 각속도(Rates), 그리고 비로소 모터 펄스로 치환되는 최종 액추에이터(Actuator) 제어 명령으로 계층적으로 연산 및 변환되는 ‘제어 권한 위임(Control Delegation)’ 아키텍처의 C++ 메인스트림(Mainstream) 동력학 흐름을 소스 코드 수준에서 해부한다.

1. 자세 제어기(mc_att_control): 선형 스칼라에서 방향 쿼터니언(Quaternion)으로의 차원 전이

조종 안정화(Stabilized) 모드를 비행 중일 때, 모드 매니저가 조립해 낸 무차원 스칼라 인풋의 최우선적인 위임 수신 시스템은 자세 제어 루프 서브모듈인 mc_att_control 데몬이다. 직전 파이프라인에서 정제된 조종기 스틱의 [-1.0, 1.0] 구간 선형 스칼라 값 배열(vehicle_attitude_setpoint 토픽 송수신부)은 이 캡슐 모듈 내부에서 기체의 한계 비행 역학 파라미터(예컨대 최대 허용 기울기 MPC_MAN_TILT_MAX 등) 인자와 곱해져 환산 결합되며, 드론이 좌표계 공간 상에서 기하학적으로 즉각 지향해야 할 3차원 타깃 자세 쿼터니언(Target Attitude Quaternion) 세트로 차원 전이 변환된다.

이 런타임 제어에서 가장 중요한 역학은 ‘수학적 회전 오차(Error)의 산출’ 기법에 속한다.
자세 제어 루프 코어는 앞서 상위 궤적 모듈에서 하달 위임받은 ’목표 쿼터니언 행렬’과 현재 EKF2 센서 융합 모듈이 실시간으로 연속 토해내는 ‘실제 현존 기체 추정 쿼터니언 행렬’ 포맷을 구면 선형 보간 비례(Slerp) 방식으로 역행렬 연산하여, X, Y, Z 3개 물리 축별 위상 오차 각도 배열(\theta_{err})을 도출해 낸다. 소프트웨어는 이렇게 도출된 각 롤/피치/요 축 오차 라디안 값에 비례 이득값 루프(P-Gain, 파라미터 명 튜닝 MC_ROLL_P 등) 상수 행렬을 스칼라 곱셈한 결과물을 산출하며, 이 1차 출력 스펙이 바로 다음 서브 계층으로 통행 위임될 타깃 토픽인 vehicle_rates_setpoint(목표 3축 각속도 지령) 데이터 덩어리가 된다.

2. 각속도 제어기(mc_rate_control): P-PID 코어 역학 루프와 3축 토크(Torque) 물리 위임

상위 통제 모듈인 자세 제어기로부터 vehicle_rates_setpoint 토픽을 위임 넘겨받은(또는 Acro 모드 상태이기에 자세 제어를 우회하여 조종 스틱 스칼라를 직접 바이패스로 넘겨받은) 각속도 제어기 스레드 모듈(mc_rate_control)은 전체 PX4 비행 제어 스택에서 제일 극도로 빠른 초고주파 주기(통상 IMU 대역인 250Hz ~ 1000Hz)로 백그라운드 구동되는 최하단 피드백 멀티채널 PID 코어 연산의 요새이다.

// src/modules/mc_rate_control/RateControl.cpp (개념적 이해를 위한 PID 커널 코어 연산 단순화)
Vector3f RateControl::update(const Vector3f &rate, const Vector3f &rate_sp, const float dt)
{
    // 1. 현재 자이로스코프 측정 각속도(rate)와 상위 모듈 목표 지령 각속도(rate_sp)의 잔차 벡터 오차 계산
    Vector3f rate_error = rate_sp - rate;

    // 2. 비례(P), 적분(I), 미분(D) 이득(Gain) 제어기 연산 이터레이션
    _rate_int += rate_error * dt; // I 적분항 오차 누적 (Anti-Windup 적분기 포화 방지 리미트 로직 내장)
    Vector3f rate_d = (_rate_prev - rate) / dt; // D 미분 킥(Derivative Kick) 파동 방지를 위한 오차가 아닌 실제 센서 각속도 독립 미분 채용
    
    // 3. 기체 운동 역학을 지배할 3축 토크 제어 지시 명령(Torque Setpoint) 스칼라 가제트 생성
    Vector3f torque_sp = (rate_error * _gain_p) + (_rate_int * _gain_i) + (rate_d * _gain_d);
    
    // 4. 지연 억제 및 반응률 극대화를 위한 선제 피드포워드(Feed-forward) 제분 추가 인가
    torque_sp += rate_sp * _gain_ff;

    return torque_sp;
}

제시된 코어 C++ 로직 뼈대에서 입증되듯, 이 각속도 제어 종단 인터페이스 모듈은 현재의 원시 자이로스코프(Gyroscope) 데이터와 타깃 각속도의 극한의 수학적 오차 미적분 연산을 통과하여 최종적으로 에어프레임(기체 하드웨어) 본체가 대기 중에 뿜어내어야 할 공력학적 모서리 3축 회전 토크량(Torque Setpoint, 논리 스케일 수치 [-1.0, 1.0]) 벡터를 최종 병합 산출해 내어 하단의 모터 믹서 계층에 위임(Delegation)하는 막중한 엔진룸 역할을 관장한다.

3. 컨트롤 얼로케이터(Control Allocator / Mixer): 기하 행렬 역변환과 액추에이터 펄스 출력

소프트웨어 제어 런루프 파이프라인의 최종 종착 관문이자 기계 하드웨어와의 물리적 접점 매개체는 바로 전통적 믹서(Mixer) 또는 그 차세대 현대적 리팩터링 형태인 Control Allocator 할당 모듈이다. 상위 PID 통제 루프인 각속도 제어기는 오직 기구학적으로 “기체 전체를 피치(Pitch) 방향의 회전 물리 축으로 40% 정도의 토크 에너지를 생성하라“는 극히 1차원적이고 추상적인 논리 명령 배열(vehicle_torque_setpoint 벡터 및 vehicle_thrust_setpoint 벡터) 토픽만을 하달 발행할 뿐, 본인이 올라탄 현재 기체의 프로펠러 달린 모터 하드웨어가 4개인지 좌우 비대칭의 8개인지(Quad, Hexa, Octo) 같은 비행 기체 하드웨어 기하 형상(Airframe Physical Geometry)에 대해서는 완벽하게 백지 무지(Blind) 상태로 격리 설계된다.

PX4-Autopilot은 이 결합 단절을 영리하게 해결하기 위해 믹서 시스템(Control Allocator) 아키텍처 조립체의 구조체 배열 내부에 장착 기체의 기하학적 로터 배치도를 수학적으로 환산한 토크-추력 행렬 분배 함수(Mixer Matrix, Allocation Method)를 내부적으로 정적재 보유하고 있다. 하위 계층은 상위로부터 넘겨 하달받은 3차원 토크 회전 벡터 배열과 1차원의 지면 수직 추력 절대 스칼라 값을 바로 이 기체 형상별 믹서 역행렬(Inverse Matrix Function)에 매 틱 통과시킴으로써, 개별 드론 프레임 암(Arm) 끄트머리에 장착된 개별적인 자율 ESC/모터 부품들이 구동을 위해 실제로 내야 할 PWM 전기 신호 펄스 명령(액추에이터 제어 페이로드, Motor \mu s 출력 혹은 고속 DShot 디지털 명령어 프레임)으로 개별 연산해 완전히 최종 확정 분해해버린다. 비행 도중 배터리 잔량 전압 강하에 맞추어 PWM 출력을 알아서 증폭 보상해 추락을 면케 해 주는 전압 동적 스케일링(Dynamic Voltage Scaling) 로직 역시 이 하드웨어 출력단 권한 위임 물리 과정의 최후 관문 필터에서 개발자의 눈을 피해 아주 투명하게(Transparent) 인티그레이션 연산 처리된다.
