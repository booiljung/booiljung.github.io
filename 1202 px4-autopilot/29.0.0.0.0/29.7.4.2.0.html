29.7.4.2. 빌드 시스템 환경 설정 파일(CMakeLists.txt) 내 컴파일 타겟 추가 및 Kconfig 연동을 통한 신규 제어 파라미터 등록 절차

독자적인 비행 제어기(Custom Controller)나 특수 제어 모듈을 C++ 소스 코드로 작성을 완료했다 하더라도, 이것이 실제 비행 제어 보드(Pixhawk 등)의 .px4 펌웨어 바이너리 이미지에 포함되어 동작하려면 빌드 시스템(Build System) 아키텍처 규칙에 맞게 타겟으로 등록되어야 한다.

PX4-Autopilot은 CMake 기반의 메타 빌드 시스템과 Kconfig 및 파이썬 파싱 스크립트를 활용하여, 수백 개의 독립된 모듈 중에서 특정 하드웨어에 탑재할 모듈만 정밀하게 취사선택하는 모듈형(Modular) 컴파일 체계를 갖추고 있다. 본 절에서는 새롭게 작성된 수동 제어 커스텀 모듈을 빌드 시스템에 통합하는 절차를 분석한다.

1. 컴파일 타겟 추가 아키텍처: CMakeLists.txt

사용자 정의 로직을 담은 디렉토리를 src/modules/custom_control 이라 가정할 때, 이 디렉토리 내부에 CMakeLists.txt를 생성하고 PX4 전용 매크로인 px4_add_module()을 명시해야 한다. 이 매크로는 일반적인 CMake의 add_executable() 기능을 넘어서, NuttX RTOS 기반의 스레드 속도, 스택(Stack) 사이즈 크기 및 포직스(POSIX) 워커 매핑을 총괄적으로 처리한다.

# src/modules/custom_control/CMakeLists.txt
px4_add_module(
    MODULE modules__custom_control
    MAIN custom_control
    STACK_MAIN 1200          # RTOS 구동 시 할당될 메모리 스택 바이트
    COMPILE_FLAGS
        -Os                  # 최적화 플래그 (공간 우선)
    SRCS
        custom_control.cpp
        CustomAttitudeControl.cpp
    DEPENDS
        px4_work_queue       # 스케줄러 워크 큐 의존성
        mathlib
)

이 매크로가 선언되면, 컴파일러는 이 모듈을 독립된 실행 단위 앱(App)으로 빌드 아키텍처 타겟 트리에 편입시킨다.

2. 보드 프로파일(Board Profile) 활성화

컴파일 타겟이 존재하더라도 PX4는 타겟 하드웨어의 플래시(Flash) 메모리 용량을 보존하기 위해 오직 식별된(Enabled) 모듈만 링킹(Linking)한다. 예를 들어 Pixhawk 4(FMUv5) 보드에 이 모듈을 삽입하려면, 해당하는 default.px4board (Kconfig 기반) 파일에 타겟 모듈을 명시해야 한다.

# boards/px4/fmu-v5/default.px4board
CONFIG_MODULES_MC_ATT_CONTROL=y     # (기존 순정 제어기 비활성화 시 주석 처리)
CONFIG_MODULES_CUSTOM_CONTROL=y     # 새롭게 추가한 커스텀 모듈 활성화

3. Kconfig 연동 및 파라미터(Parameter) 메타데이터 등록 절차

기체 제어 파라미터(예: 새로 도입한 CUS_ROLL_P 게인 등)는 코드 베이스 내에서 PARAM_DEFINE_FLOAT() 매크로로 소스 상단에 선언된다. PX4 빌드 툴체인 파이프라인에는 구문 분석(Parser) 스크립트가 포함되어 있어, 컴파일 과정 중 이 매크로 블록 안의 주석(Doxygen-style) 텍스트를 파싱하여 QGroundControl(QGC)이 소비할 수 있는 XML, JSON 메타데이터 구문을 동반 생성한다.

또한 새로운 모듈은 module.yaml 파일을 통해 파라미터 그룹 식별자와 명령행(CLI) 실행 환경을 Kconfig 시스템에 연동한다.

# src/modules/custom_control/module.yaml
module_name: Custom Control
command_name: custom_control
parameters:
  - group: CustomControl
    name: CUS_ROLL_P
    type: float
    default: 6.5
    min: 1.0
    max: 15.0
    decimal: 1
    short_desc: "Roll Axis Custom Proportional Gain"

위 메타데이터 파이프라인이 정상적으로 동작하면, 개발자는 QGroundControl 소프트웨어를 켜고 “Parameters” 탭에 진입했을 때 CustomControl이라는 새로운 트리거(Category)와, 그 아래 미려한 슬라이더 인터페이스가 얹혀진 CUS_ROLL_P 파라미터를 즉시 관측 및 튜닝할 수 있다.

4. Ardupilot Waf 빌드 시스템과의 아키텍처 비교

Ardupilot은 전통적으로 Waf(와프) 빌드 시스템을 사용하여, 모든 코드베이스를 하나의 거대한 단일 스레드 모놀리식(Monolithic) 바이너리로 직렬 링킹한다. 이는 실행 오버헤드를 줄여주지만, 특정 디렉토리나 커스텀 앱 하나만 분리해내어 ON/OFF를 토글(Toggle)하기엔 소스 코드 의존성이 강하게 결합되어 있어 다소 번거롭다.

반면, PX4-Autopilot은 CMake와 파일 시스템 기반의 POSIX 호환 구조를 활용하여 비행 제어조차 하나의 “앱(Application)” 레이어로 추상화했다. 따라서 CMakeLists.txt에 신규 모듈을 추가하고 보드 컨피그에서 y/n을 조작하는 일련의 절차 단 몇 줄만으로 거대한 펌웨어 아키텍처를 조금도 훼손하지 않고 자신의 커스텀 로직을 플러그인(Plug-in)할 수 있다. 이는 상업용 벤더나 학계 연구실에서 특수 기체를 개발할 때 가장 강력하게 체감되는 아키텍처적 유연성(Flexibility)이다.
