29.7.1.2. 시동 전제 조건 검사: 메인 추진체 스로틀 입력이 최하단 기저점에 근접하지 않았을 때 인터락(Interlock) 해제를 거부하는 물리적 안전 확보 분기문

무인항공기(UAV) 제어 시스템에서 시동(Arming)은 기체의 모든 추진체(Propulsion Area)가 전기적, 기계적으로 활성화됨을 의미하는 가장 크리티컬(Critical)한 상태 전이(State Transition)다. 만약 조종기(Radio Control)의 스로틀 스틱이 인위적으로 높게 올라가 있는 상태에서 시스템이 시동을 허용한다면, 모터는 즉각적으로 높은 RPM으로 회전하여 기체가 통제 불능 상태로 튀어 오르는 이른바 런어웨이(Runaway) 또는 콜드 스타트 급발진 참사를 초래하게 된다.

이를 방지하기 위해 PX4-Autopilot 시스템은 시동 명령이 인가되는 그 찰나의 순간에 **“스로틀 입력이 최하단 기저점(Base Point)에 위치하고 있는가?”**를 엄격히 검증하는 인터락(Interlock) 방어 기제를 동작시킨다. 본 장에서는 PX4의 커맨더(src/modules/commander) 모듈 내부에서 수행되는 물리적 안전 확보 분기문과 소프트웨어 아키텍처를 심도 있게 분석한다.

1. 아키텍처 수준의 인터락(Interlock) 해제 방어 트리

조종기 스틱 제스처(Stick Gesture) 혹은 지상 관제 시스템(GCS)의 MAVLink(COMMAND_LONG: VEHICLE_CMD_COMPONENT_ARM_DISARM) 메시지를 통해 시동 명령이 커맨더 모듈에 도달하면, 최우선적으로 arming_state_transition 프로세스가 활성화된다.

이 트랜지션 로직은 기체 상태(Vehicle Status)의 무결성을 검사하는 체크리스트를 순차적으로 실행하며, 그 중 가장 상위에 랭크된 물리적 조건이 바로 유효 하단 스로틀(Valid Low Throttle) 판별이다.

    B -- No --> C[Arming Denied: Safety Switch]
    B -- Yes --> D{스로틀이 하단 기저점(Low Threshold) 이하인가?}

    E --> I[MAVLink 에러 메시지 GCS 로 송출]

2. 소스 코드 레벨의 스로틀 기저점 검사 분기문

PX4 최신 펌웨어 아키텍처에서 스로틀 상태의 무결성을 검증하는 흐름은 단순히 PWM 레지스터의 값을 읽어오는 것이 아니라, 추상화된 uORB 데이터인 manual_control_setpoint 토픽을 활용하여 물리적 조종기의 Z축 변위(스로틀)를 판독하는 구조를 갖는다.

2.1 제어 입력 검열 로직 (Commander.cpp 및 Arming 라이브러리)

시동 허가를 결정하기 전, 스크립트는 bool prearm_check() 체인을 통해 다양한 조건을 평가한다. 스로틀 포지션 평가는 논리합(OR)이나 논리곱(AND)으로 하드코딩된 물리적 바운드 점검을 통과해야 한다.

// src/modules/commander/ArmAuth/arming_checks.cpp 의사 코드 예시
bool is_throttle_low_for_arming(const manual_control_setpoint_s &manual_sp) {
    // manual_sp.z 는 스로틀 축이며 [0.0, 1.0] 또는 [-1.0, 1.0] 범위로 정규화됨 (기체 유형별 상이)
    // 멀티로터의 경우 z축이 주로 0.0 ~ 1.0의 범위를 가짐.
    
    // 파라미터 기반 마진(Margin) 접근
    float throttle_base_margin = 0.1f; 
    
    // 스로틀이 임계값 이상에 머물고 있다면
    if (manual_sp.z > throttle_base_margin) {
        mavlink_log_critical("Arming denied: Throttle is not at minimum");
        return false;
    }
    
    return true; // 안전 마진 내부에 존재함
}

조종 스틱의 포텐셔미터(Potentiometer) 노이즈나 하드웨어 캘리브레이션 오차 범위를 흡수하기 위해 엄격한 영점(0.0)이 아닌 0.1(약 10% 이내) 수준의 톨러런스(Tolerance) 밴드를 적용하는 것이 특징이다.

3. Ardupilot 대비 아키텍처 및 철학 차이

Ardupilot(Mission Planner 기반) 역시 THR_MIN 파라미터와 Pre-Arm Checks를 통해 동일한 형태의 방어 로직을 수행하지만, 내부 메커니즘에서 차이가 발생한다.

  • Ardupilot의 수동적 폴링 의존: Ardupilot은 메인 스케줄러 루프에서 전역 변수에 마운트된 RC 채널 값에 직접 접근하여 시동 조건을 검증하는 모놀리식(Monolithic) 성향이 짙다.
  • PX4-Autopilot의 독립적 모듈화 (uORB Pub/Sub): PX4의 경우, RC 입력 모듈(rc_input)이 퍼블리시한 메시지를 manual_control 뎁스에서 1차 정제 및 필터링 한 뒤 이를 구독(Subscribe)한 commander 프로세스에서 판독한다. 이는 다중 센서 결함(Sensor Failsafe)이 발생하더라도 커맨더 스레드가 병목 없이 독립적인 메모리 공간에서 안전하게 Arming Deny 정책을 수행할 수 있게 하는 아키텍처적 우위를 제공한다.

4. GCS 관제(QGroundControl)와의 상호작용 및 예외 처리

해당 분기문에서 false가 반환되어 시동이 거부되면, PX4는 내부 상태 머신을 유지함과 동시에 QGroundControl(QGC) 등 MAVLink 호환 GCS로 에러 원인을 텍스트 텔레메트리로 퍼블리시(STATUSTEXT MAVLink Message)한다.

  • QGC의 사용자 인터페이스는 **“Arming denied: throttle not minimum”**이라는 스토퍼 경고 알림을 스크린의 팝업 컴포넌트로 즉각 렌더링하며 음성(TTS) 피드백을 제공한다.
  • 또한, ROS2(uXRCE-DDS) 환경에서 동작하는 자율 에이전트(Offboard Mode) 제어 시에도, 자율 에이전트 노드가 시동 직전 발행하는 가상 스로틀 추종 값(Virtual Setpoint)이 최하단 기저점(예: Hovering Thruster Baseline 미만)을 만족하지 못한다면 Commander 모듈은 여지없이 인터락 해제를 거부한다.

결과적으로, PX4의 시동 전 스로틀 하단 검사 로직은 단순히 사용자 실수를 방어하는 수준을 떠나, 인간이든 인공지능 자율형 에이전트이든 간섭 없이 시스템 내부의 절대적인 물리적 안전 계통을 보증하는 최후의 **하드-소프트웨어 인터락(Hardware-Software Interlock)**이라 할 수 있다.
