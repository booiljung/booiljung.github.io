29.2.1.1. 하드웨어 타이머 인터럽트 기반 통신 프로토콜 프레임 디코딩 C/C++ 소스 분석

본 소절에서는 PX4-Autopilot의 하위 운영체제 계층(NuttX RTOS)과 직렬 통신 드라이버가 맞닿는 경계에서, 마이크로컨트롤러(MCU)의 하드웨어 타이머 및 직렬 인터럽트(Serial Interrupt) 메커니즘이 어떻게 C/C++ 커널 레벨 소스 코드로 구현되어 무선 통신 프로토콜(SBUS, CRSF 등)의 데이터 프레임을 고정밀도로 디코딩(Decoding)해 내는지 분석한다.

1. 비동기 프레임 수신을 위한 하드웨어 인터럽트(ISR)의 커널 결합

외부 조종기에서 송신되는 통신 프로토콜은 기체 내장 플라이트 컨트롤러(FC)와 클럭(Clock)을 동기화하지 않는 완벽한 비동기(Asynchronous) 스트림 형태를 띤다. 따라서 수신 측에서는 프레임(Frame) 패킷의 물리적인 시작과 끝을 오직 바이트 간의 시간 간격(Inter-byte Timeout)과 패킷 간의 휴지기(Idle Time)를 계측함으로써 유추해야만 한다. 이를 달성하기 위해 rc_input 디바이스 드라이버 백엔드의 하부 요소(예: NuttX의 UART 드라이버 하드웨어 추상화 계층)는 UART 인터럽트와 고해상도 하드웨어 타이머 인터럽트(e.g., HRT - High Resolution Timer)를 융합하여 사용한다.

수신 핀(RX)에 전위 변화가 일어날 때마다 MCU의 UART 주변기기는 RXNE(Receive Not Empty) 하드웨어 인터럽트를 즉각적으로 발생시킨다. 이 인터럽트 서비스 루틴(ISR)은 애플리케이션 계층으로 제어권을 즉각 넘기는 대신, 수신된 바이트를 링 버퍼에 넣음과 동시에 그 수신된 찰나의 절대 시간 타임스탬프(Absolute Timestamp, 마이크로초 단위)를 메타데이터로 함께 기록한다. 이러한 극한의 타이밍(Timing) 스탬핑은 이후 프로토콜 파서(Parser)가 프레임 경계 지점을 논리적으로 획정하는 데 가장 결정적인 단서로 작용한다.

2. 타이밍 타임아웃(Timeout) 판별과 상태 머신(FSM) 리셋 소스 로직

통신 잡음(Noise)이나 프레임 누락(Drop) 상황에서 파서 엔진이 손상된 프레임 포맷에 갇히는 교착 상태를 방지하기 위해, 파서 내부는 타임아웃 룰 셋팅에 의존하는 거대한 유한 상태 기계(FSM, Finite State Machine) 패턴으로 짜여 있다. 대표적으로 Futaba SBUS 프로토콜 디코딩을 담당하는 소스 트리(src/lib/rc/sbus.cpp)를 조명할 수 있다.

// src/lib/rc/sbus.cpp (하드웨어 인터럽트 타임스탬프를 소화하는 파서 상태 머신 개념 소스)
bool sbus_parse(uint64_t now, uint8_t *frame, unsigned len, uint16_t *values,
                uint16_t *num_values, bool *sbus_failsafe,
                bool *sbus_frame_drop, uint16_t max_channels)
{
    // 1. 프레임 간 휴지기(Idle Time) 초과 감지 및 파서 초기화
    // 이전 바이트가 수신되었던 시점(now_prev) 대비 현재 시점(now)의 시간차가
    // SBUS 규격의 인터 프레임 타임아웃 임계치(예: 약 3ms)를 상회하는가?
    if (now - sbus_frame_time > SBUS_FRAME_GAP_TIMEOUT) {
        // 인터럽트 공백기가 길었다면 이는 새로운 패킷 프레임의 시작점을 의미함.
        // 따라서 상태 머신 파서 진행 상태를 강제 초기화(Reset)하여 이전 가비지 데이터를 무효화함.
        sbus_decode_state = SBUS_DECODE_STATE_DESYNC;
    }

    sbus_frame_time = now; // 최신 바이트 캡처 시간 갱신

    // 2. 상태 머신 구동에 따른 바이트 시프트 로직
    switch (sbus_decode_state) {
        case SBUS_DECODE_STATE_DESYNC:
            if (frame[0] == SBUS_START_BYTE) { // 0x0F 동기화 바이트 감지
                sbus_decode_state = SBUS_DECODE_STATE_SYNC;
                sbus_partial_frame_count = 1;
            }
            break;
            
        case SBUS_DECODE_STATE_SYNC:
            // ... (동기화된 버퍼에 패킷 길이가 25바이트 도달할 때까지 연속 축적 처리)
            break;
            
        // ...
    }

    // ... (최종 프레임 파싱 성공 반환 여부)
}

위 C++ 코드 로직에서 명확히 증명되듯, 하드웨어 타이머 시스템(HRT)이 제공하는 절대 시간 값 now 매개변수는 단순한 정보성 메타데이터가 아니다. 이 값은 인터럽트 발생의 역학적 간격을 도출하여, 이전 프레임의 잔재가 현재 프레임의 시작을 오염시키는 동기화 이탈(Desync) 상태를 하드 리셋(Hard Reset)해 버리는 강력한 필터링 판별의 잣대로 동작한다. 파서 로직은 단순히 바이트의 나열을 맹신하지 않으며, 이들 바이트가 도달한 ’시간적 간격의 연속성’을 엄정히 수학적으로 따져 물음으로써 혹독한 전파 환경 속에서도 무결점을 보장한다.

3. 비트 마스킹(Bit-masking) 연산을 통한 11-Bit 채널 디코딩 메커니즘

SBUS와 같이 대역폭 대비 전송량을 쥐어짜는 구세대 직렬 프로토콜들은 칩셋 대역폭 절약을 위해 채널 데이터를 범용 8비트/16비트 경계선에 정렬(Byte-alignment)시켜 보내지 않는다. SBUS는 하나의 채널 값을 오직 11비트(Bit) 길이에 욱여넣어, 가변 바이트 경계를 횡단하도록 패킹(Packing)하는 기괴한 명세를 지닌다. 이를 해체해 내는 C/C++ 커널 레벨 디코딩 연산은 비트 시프트(Bit Shift)와 논리곱(AND) 마스킹 연산의 집약체이다.

// SBUS 11비트 채널 페이로드 언패킹(Unpacking) 아키텍처 발췌
// 25바이트 프레임 중 1번 인덱스부터 시작되는 22바이트 페이로드(Payload) 구획 해석

values[0] = ((frame[1]    | frame[2] << 8)                     & 0x07FF);
values[1] = ((frame[2] >> 3 | frame[3] << 5)                     & 0x07FF);
values[2] = ((frame[3] >> 6 | frame[4] << 2 | frame[5] << 10)    & 0x07FF);
// ... 총 16개 채널 누적 연산 진행 ...

위 런타임 언패킹(Unpacking) 구문은 메모리에 저장된 원시 바이트 단위(frame[])를 물리적으로 결합한 후, 0x07FF (이진수 11111111111, 정확히 11개의 비트 온) 마스킹(Masking)을 씌워 불필요한 바이트 간 경계 침범 쓰레기 값들을 기계적으로 잘라낸다(Trimming). 이러한 극도로 최적화된 저수준(Low-level) 비트 조작(Bitwise Manipulation)은 MCU의 ALU(Arithmetic Logic Unit) 병목을 전혀 유발하지 않은 채 매 100Hz 주기마다 수십 개의 채널 값을 실수 없이 번역해 냄으로써, 가장 가볍고 가장 빠른 커널 전처리 디코딩 엔진의 공학적 완성도를 증명한다. 이처럼 물리적 하드웨어의 클럭에 맞춘 정밀 타이머 조건문과 고전적인 비트 병렬 연산이 유기적으로 직조된 C++ 아키텍처가 존재하기에, PX4는 복잡한 하드웨어 수신기 종류 트리에 구애받지 않고 언제나 균일한 16비트 정수 배열 포맷을 윗단 uORB 시스템으로 안심하고 위임할 수 있게 된다.
