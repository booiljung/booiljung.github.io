29.1.1.3. 상태 머신(State Machine) 전환 논리에 따른 하위 비행 태스크(Flight Task) 동적 인스턴스화 및 메모리 해제 메커니즘

본 소절에서는 FlightModeManager 모듈 내부에서 파일럿(Pilot)의 조종기 스위치 변환이나 지상 관제 시스템(GCS)의 자율 비행 커맨드 하달 시 발생하는 런타임 모드 전이(State Transition) 메커니즘을 C++ 소스 코드 레벨에서 역추적한다. 특히 PX4 아키텍처 특유의 ‘비행 태스크(Flight Task)’ 팩토리 디자인 패턴(Factory Pattern)을 매개로 한 힙(Heap) 메모리 동적 할당(Dynamic Instantiation) 기법과 메모리 릭(Memory Leak) 방지를 위한 안전한 해제(Deallocation) 과정을 핵심적으로 해부(Dissection)한다.

1. 다형성(Polymorphism) 기반의 FlightTask 커널 아키텍처

PX4 비행 제어 스택에서 모든 수동 조종 가공 및 자동 궤적 생성 로직은 최상위 기본 추상 클래스(Abstract Base Class)인 FlightTask를 상속(Inherit)받아 구체화된 자식 클래스로 구현된다. 예를 들어, FlightTaskManualPosition, FlightTaskAcro, FlightTaskAutoMapper 등 수십여 개의 태스크 모듈들이 자신의 비행 물리 특성에 맞는 각각의 고유한 조종 맵핑 수식 연산 및 타깃 궤적 생성 알고리즘을 캡슐화(Encapsulate)하여 지니고 있다.

FlightModeManager 컨트롤러 모듈 자체는 이러한 개별 자식 클래스들의 난해한 세부 역학 수학식이나 알고리즘 로직을 인지할 필요가 없도록 설계되었다. 매니저는 오로지 _current_task라는 명칭을 부여받은 부모 클래스형 스마트 포인터(Smart Pointer)만을 소유하며, 객체 지향의 다형성을 활용하여 현재 활성화된 객체의 가상 함수(Virtual Function)인 update() 인터페이스만을 메인 폴링(Polling) 사이클마다 기계적으로 호출할 뿐이다. 이는 전형적인 소프트웨어 공학의 개방-폐쇄 원칙(Open-Closed Principle)을 극적으로 구현해 낸 강력한 추상화 아키텍처이다.

2. 런타임 상태 전환 시의 동적 인스턴스화(Dynamic Instantiation) 시퀀스

시스템 버스의 vehicle_control_mode 토픽이나 네비게이터 상태(Navigator State)가 변경 인지되어 비행 모드 전환 스위칭(State Transition)이 요구되면, FlightModeManager의 내장 방어형 메서드인 switchTask() 인터럽트 루틴이 점화(Trigger)된다. 이 시퀀스에서 PX4는 구시대적인 대규모 정적 클래스 배열(Static Array) 선언 방식을 탈피하여 팩토리 메서드(Factory Method) 패턴을 활용한 동적 할당 로직을 전개한다.

// src/modules/flight_mode_manager/FlightModeManager.cpp의 모드 전환 제어 분기 뼈대
FlightTaskError FlightModeManager::switchTask(FlightTaskIndex new_task_index)
{
    // ... (모드 전환 사전 무결성 검사 로직 보안 생략) ...

    // 1. 기존에 연산 점유 중이던 낡은 태스크 객체의 메모리 안전 해제 파괴자 호출
    if (_current_task != nullptr) {
        _current_task->~FlightTask();
        // 메모리 정리 체계 이행
    }

    // 2. Task Factory 패턴을 매개로 한 신규 요구 태스크의 런타임 동적 할당 생성
    _current_task = _task_factory.instantiate(new_task_index);

    if (_current_task == nullptr) {
        // 메모리 할당 제한 초과(OOM) 등 커널 에러 발생 시 즉각적인 백업(Fallback) 예외 처리
        return FlightTaskError::ActivationFailed;
    }

    // 3. 신규 태스크 모듈의 변수 초기화 및 동역학 상태 단차(Bumpless) 전이 루틴 점화
    if (_current_task->activate(previous_state)) {
        _current_task_index = new_task_index;
    } else {
        // activate() 실행 실패 (예: GPS 솔루션 오류로 포지션 모드 불가 시) 롤백 전환
    }
}

이 고해상도 루틴에서 공학도의 시선이 집중되어야 할 부문은 바로 비행 태스크의 적시 생성(JIT: Just-In-Time Instantiation) 구조이다. 무겁고 방대한 수십 개의 FlightTask 서브 객체 인스턴스들을 RTOS 부팅 시점의 메모리에 한꺼번에 올려 묵어두는(Pre-allocation) 낭비 대신, 오직 편재 비행 모드에 관여하는 단 1개의 태스크 객체만이 임베디드 코어 힙(Heap) 메모리의 실행 점유를 차지하게 된다. 이는 1MB 단위의 치열한 가용 RAM 공간 하에서 임무를 영위해야 하는 Flight Controller(MCU)의 극도로 타이트한 한계를 타개하기 위한 NuttX 운영 체제 수준의 정교한 프로파일링(Profiling) 설계이다.

3. 메모리 파편화(Fragmentation) 방지 기법과 안전 무결성(Safety Integrity)

엄격한 가이드라인을 요구하는 항공 C/C++ 시스템 단에서, 고주파 메인 루프 가동 중 비행 모드 전환 시마다 힙 구조를 뒤엎으며 동적 메모리를 빈번히 할당 및 해제(new / delete)하는 것은 최악의 성능 파편화(Fragmentation) 및 영구적 메모리 누수(Memory Leak)라는 끔찍한 실질적 추락 위험을 유발한다. PX4 펌웨어는 이 치명적 단점을 하드웨어 아키텍처 레벨에서 제어하기 위해 시스템 코어에 자체적인 블록 메모리 풀(Memory Pool) 관리 기법을 병용하거나, 포지셔닝 제어 모듈 내에 사전 할당 및 고정된 공용 스택 덩어리(Shared Task Block) 영역 위에 placement new 특수 연산자를 활용하여 신규 태스크 객체를 오버라이트(Overwrite)하는 고급 기법을 구사함으로써 힙 파편화를 원천적으로 파훼한다.

부수적으로, 모드 스위칭 시 발현될 수 있는 기능 무결성 실패(예: 조종자가 포지션 위치 제어 모드 스위치를 켰으나, 현재 EKF의 추정 신뢰도 스코어 부족으로 activate() 콜백이 즉각 false를 뱉으며 기각(Reject)하는 상황)를 커버하기 위해, PX4 소스 트리 메인보드에는 언제나 즉각 하위 가용 안전 상태(예컨대 GPS 제약이 불필요한 고도 유지 Altitude 모드나 최하위 Stabilized 모드)로 상태 머신의 포인터를 되돌려버리는 가드레일 형 폴백(Fallback) 라우팅 트리가 하드코딩되어 있다.
결과적으로 지상 오퍼레이터나 초보 조종사가 공중 기동 중 아무리 부적절한 타이밍에 스위치를 조작하더라도, FlightModeManager는 코어 덤프(Core Dump)를 발생시키며 제어 권한을 상실하지 않고, 발생한 시스템 및 환경 오류를 기체 내부에서 집어삼킨 뒤(Error Swallowing), 비행 로보틱스를 시스템 파국으로부터 즉시 스위칭 구출해 내는 극상의 체계적 소프트웨어 유연성을 완성해 낸다.
