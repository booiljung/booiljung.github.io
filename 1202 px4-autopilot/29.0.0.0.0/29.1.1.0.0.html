29.1.1. FlightModeManager 스레드 구조 및 모드 라우팅 로직 (src/modules/flight_mode_manager)

본 소절에서는 PX4-Autopilot의 수동 및 자동 비행을 총괄하는 관문이자 라우팅 스위치 역할을 수행하는 FlightModeManager 모듈 시스템의 내부 스레드(Thread) 구조와 상태 머신(State Machine) 지향적 메시지 분배(Message Routing) 메커니즘을 소스 코드 수준에서 상세히 분석한다.

1. FlightModeManager의 아키텍처적 위상과 설계 근거

과거 PX4 구버전(v1.10 이전) 아키텍처에서는 수동/자동 비행 모드에 따른 목표 궤적(Setpoint) 산출 로직이 mc_pos_control(멀티로터 위치 제어기)이나 fw_pos_control_l1(고정익 제어기)과 같은 하위 제어 모듈 내부에 얽혀(Tightly Coupled) 있었다. 이는 새로운 커스텀 비행 모드(Custom Flight Mode)를 이식할 때마다 코어 제어기 소스를 대폭 수정해야 하는 소프트웨어 공학적 부채가 누적되는 결과를 초래했다.

이러한 모놀리식(Monolithic) 제어기 구조를 극복하기 위해, PX4 커뮤니티는 비행 제어기 루프 이전 단계에 위치하는 독립적인 FlightModeManager 데몬(Daemon)을 신설하였다. 본 모듈의 핵심 존재 의의는 다형성(Polymorphism)을 활용한 비행 태스크(Flight Task) 격리와 단일화된 목표치 인터페이스 제공이다. FlightModeManager는 오직 현재 조종자의 스위치 입력이나 GCS 임무 명령에 따른 ’논리적 모드 상태’만을 평가하고 평가 결과에 맞추어 적절한 하위 태스크 클래스(예: FlightTaskManual, FlightTaskOrbit 등)에 스레드 연산 제어권을 위임(Delegation)한다.

2. 스레드 초기화 및 워크 큐(Work Queue) 등록 메커니즘

FlightModeManager는 NuttX 상에서 독립적인 POSIX 스레드(POSIX Thread)로 직접 생성되지 않고, 최신 PX4의 고효율 스케줄링 패러다임인 워크 큐(Work Queue) 아키텍처 위에서 스케줄링된다.

모듈이 부팅 시 초기화되는 진입점(Entry Point) 함수인 FlightModeManager::task_spawn()을 분석해 보면, 이 모듈이 px4::WorkItem 클래스를 상속받아 구현되어 있음을 확인할 수 있다.

  1. 큐 지정: 모듈은 비행 제어 루프의 지연(Latency) 허용성에 맞게 wq_configurations::nav_and_controllers 워크 큐에 자체 루프를 등록한다.
  2. 구독 폴링(Subscription Polling): Run() 메서드는 독자적인 무한 while 루프 방식이 아닌, 특정 uORB 센서나 위치 추정 토픽(예: vehicle_local_position)이 새롭게 업데이트되었을 때 스케줄러로부터 이벤트 콜백(Callback)을 수신하여 1회 연산을 수행하고 즉시 큐 양보(Yield)를 단행하는 비동기 반응형(Reactive) 구조를 띈다.

이러한 구조적 특성 덕분에 CPU 컨텍스트 스위칭(Context Switching) 비용이 획기적으로 감축되며, 타이트하게(100Hz ~ 250Hz 영역) 맞물려 구동되어야 하는 추정기(Estimator)와 제어기(Controller) 사이에서 유휴 병목(Idle Bottleneck) 현상을 유발하지 않는다.

3. 모드 라우팅 로직(Mode Routing Logic)의 구현 실체

FlightModeManager의 핵심 로직인 Run() 메서드 내부에서는 매 틱(Tick)마다 시스템의 포괄적 모드 지시자인 vehicle_control_mode 토픽을 가장 우선적으로 비동기 검사(Check)한다.

3.1 vehicle_control_mode 비트 검사 및 라우팅 분기

입력단에서 조종기 스위치 변환이 발생하면 Commander 모듈은 vehicle_control_mode 구조체의 불리언(Boolean) 플래그들을 갱신한다. FlightModeManager는 다음과 같은 분기(Switch/If-else) 로직을 통해 진입 대상(Target) 태스크를 결정한다.

  • flag_control_manual_enabled == true이고 flag_control_position_enabled == false:
    시스템은 기구학적 위치 고정 의무가 없는 완전 수동(Stabilized, Acro) 형태임을 인지하고 해당 바이패스(Bypass) 로직으로 라우팅한다.
  • flag_control_manual_enabled == true이고 flag_control_position_enabled == true:
    드론이 조종 스틱의 입력을 받아들이되 공간 위치를 유지하는 ’Position 모드(수동 위치 제어)’임을 파악하여 FlightTaskManualPosition이라는 객체를 호출한다.
  • flag_control_auto_enabled == true:
    조종기 스틱 데이터 기각 후, 자율 임무 궤적 생성 태스크(예: FlightTaskAutoMapper)로 연산 권한을 완전히 이양한다.

3.2 상태 기계(State Machine)에 의한 단차 없는(Bumpless) 전환

드론이 시속 40km로 전진 비행(Forward Flight) 중 조종자가 스위치를 튕겨 ‘자율 복귀(Return to Launch)’ 모드로 급격히 전환할 경우, 모드 매니저는 즉시 타겟 태스크를 FlightTaskAutoReturn 객체로 교체한다.
이 교체 과정은 C++ 스마트 포인터와 팩토리 패턴(Factory Pattern)을 통해 이전 태스크를 메모리 상에서 안전하게 소멸(Destruct)시키고 새 태스크를 로딩하는 방식이다. 전환 시점의 운동에너지 역학 단차(Jump)를 무마하기 위해 신규 호출된 비행 태스크의 activate() 초기화 메서드가 호출되며, 이 시점에 EKF2의 현재 물리적 위치 및 속도 벡터 상태를 파라미터로 주입받아 모터 제어의 연속성을 수학적으로 보증하는 스무딩(Smoothing) 작업이 수행된다.
