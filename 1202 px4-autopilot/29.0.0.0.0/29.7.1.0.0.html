29.7.1. 스틱 제스처(Stick Gestures) 기반 하드웨어 제어 커맨드 분석

PX4-Autopilot 시스템에서 스틱 제스처(Stick Gestures)는 단순히 비행체에 동역학적 모멘트를 제공하는 것을 넘어, 하드웨어의 특정 상태를 직접 제어하거나 비행 모드 변경을 트리거하는 강력한 시그널 메커니즘이다. 조종기(Radio Control)의 스틱을 특정 조합의 극단점(Extreme)으로 유지함으로써 시동(Arming), 시동 해제(Disarming), 자이로 캘리브레이션 등 GCS(QGroundControl 등) 없이 기체 제어기(Flight Controller)에 직접 개입할 수 있다.

이 장에서는 PX4-Autopilot의 최신 펌웨어 아키텍처(주로 v1.14 이상)를 기준으로 스틱 제스처가 어떻게 소프트웨어적으로 파싱되고, 커맨더(Commander) 모듈을 통해 어떻게 하드웨어 제어 이벤트로 변환되는지 분석한다.

1. 스틱 제스처 인식 아키텍처 및 메커니즘

수동 비행 하위 시스템에서 조종기의 좌우 스틱 값(Yaw, Pitch, Roll, Throttle)은 input_rc uORB 토픽을 통해 정규화([-1.0, 1.0])되어 퍼블리시된다. PX4의 manual_control 모듈 내에서 스틱 제스처를 인식하는 코어 로직은 순수 매뉴얼 제어 트래픽이 manual_control_setpoint로 라우팅되기 전에 평가된다.

스틱 제스처를 인식하기 위해서는 크게 두 가지 파라미터가 요구된다.

  1. 극단 임계값 (Extreme Threshold): 스틱이 물리적으로 끝까지 밀렸는지를 판별하는 영역(기본적으로 스케일된 값의 90% 이상).
  2. 지속 시간 (Dwell Time): 조종사가 의도적으로 해당 제스처를 취하고 있는지를 판단하기 위해 필요한 누적 논리 타이머. 통상적으로 시동(Arming)을 위해서는 0.5초에서 1.0초 이상의 유지가 필요하다.

2. 상태 파라미터 기반 제스처 추출 로직 (Source Code Analysis)

PX4 스택에서 스틱 제스처를 처리하는 핵심 함수는 src/modules/commander/Commander.cpp 내부 혹은 src/modules/manual_control 내의 입력 전처리 체인에 위치한다.

예를 들어, 스틱을 이용한 시동(Arming) 제스처(스로틀 최소, 요 스틱 최대 우측 이동)는 다음과 같은 논리 흐름으로 평가된다.

// src/modules/manual_control/ManualControl.cpp의 의사 코드 예시
void evaluate_stick_gestures(const manual_control_setpoint_s &manual_sp) {
    bool is_throttle_min = (manual_sp.z < THRESHOLD_MIN);
    bool is_yaw_max      = (manual_sp.r > THRESHOLD_MAX);
    
    if (is_throttle_min && is_yaw_max) {
        // 제스처 활성화 및 타이머 누적 시작
        _stick_gesture_time += _dt;
        if (_stick_gesture_time > ARMING_DWELL_TIME) {
            send_vehicle_command(vehicle_command_s::VEHICLE_CMD_COMPONENT_ARM_DISARM, 1.0f);
            _stick_gesture_time = 0; // 리셋
        }
    } else {
        _stick_gesture_time = 0; // 조건 이탈 시 즉시 리셋 (오작동 방지)
    }
}

이러한 타이머 기반 평가는 일시적인 오작동(Glitch)이나 우발적인 스틱 타격을 제스처로 오인하는 것을 철저하게 방지한다.

3. Ardupilot 대비 스틱 제스처 철학의 차이점

Ardupilot 환경(Mission Planner 등)에서는 스틱 제스처 또한 PX4와 유사한 방식으로 동작하나, Ardupilot은 전통적으로 채널 값 매핑 기반의 RC_OPTIONS 설정과 트리거를 좀 더 광범위하게 허용하는 반면, PX4-Autopilot은 명시적인 상태 머신(State Machine)의 안전성을 최우선으로 둔다.

  • 안전 검증 강도: PX4는 제스처 논리가 통과하더라도(예: Arming 이벤트 발생), Commander 모듈 내부의 방대한 비행 전 점검(Pre-flight Check) 리스트(센서 통합성 평가, EKF 수렴 등)가 하나라도 실패하면 제스처 커맨드를 즉시 NACK 처리(반려)한다.
  • 상태의 독립성: 제스처는 vehicle_command uORB 메시지로 변환되어 MAVLink로 들어온 외부 GCS 명령과 완벽히 동일한 트랜잭션 큐를 탄다. 이는 내부 하드웨어(RC)와 외부 소프트웨어(ROS2, GCS) 간의 명령 우선순위를 일원화하는 PX4 아키텍처의 큰 특징이다.

4. 커스텀 스틱 제스처 확장을 위한 디버깅 및 개발 팁

기체에 서보(Servo) 모터를 추가하거나 자율 에이전트 개발 과정에서 특정한 로직(예: 강제 EKF 리셋, 페이로드 드롭)을 스틱 제스처로 구현해야 할 때, 개발자는 MAVLink 메시지 없이도 독자적인 vehicle_command를 발생시킬 수 있다.

  1. uORB 토픽 주입: ManualControl.cpp의 제스처 평가 함수에 본인만의 분기문(Switch Case)을 추가한다.
  2. ULog 디버깅: 타겟 하드웨어에 배포하기 전, 개발자는 SITL 환경에서 ULog 및 QGroundControl의 MAVLink Console을 활용해 listener manual_control_setpoint 커맨드를 날려, 특정 제스처가 유효한 임계값을 넘나들고 있는지 논리적 상태를 실시간으로 점검해야 한다.

스틱 제스처는 직관적이고 즉각적인 물리적 제어를 가능케 하지만, 예외 상황에 노출되지 않기 위해 매우 보수적인 타임아웃(Timeout) 및 임계값 튜닝(Parameter Tuning)이 요구되는 중요한 하위 시스템이다.
