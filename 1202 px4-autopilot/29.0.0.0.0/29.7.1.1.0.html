29.7.1.1. 스틱 시동 논리: 특정 조종 축의 극단점 배치를 논리적 시간 카운터로 검증하여 시동 활성화 플래그를 변경하는 로직

PX4-Autopilot의 스틱 제스처 중 가장 빈번하게 사용되며, 가장 높은 수준의 안전성이 요구되는 기능이 바로 스틱 시동(Stick Arming) 및 시동 해제(Disarming) 메커니즘이다. 지상 관제 시스템(GCS)의 개입 없이 조종사(Pilot)가 기체에 직접 물리적 동력을 부여하는 절차이므로, 일시적인 노이즈나 조종사의 실수로 인해 시동이 걸리는 참사(Catastrophe)를 막기 위해 엄격한 논리적 시간 카운터(Logical Time Counter) 기반의 검증 아키텍처가 적용되어 있다.

본 절에서는 src/modules/manual_control 내에서 조종 축(Throttle, Yaw)의 극단점(Extreme) 배치가 어떻게 파싱되고, 펌웨어 타이머와 연동되어 시동 활성화 플래그(Arming Flag)를 전이시키는지 심층 분석한다.

1. 조종 축의 극단점(Extremes) 배치 및 임계값 검증

가장 보편적인 조종기 모드(Mode 2 기준)에서 스틱 시동 제스처는 **“스로틀(Throttle) 최하단, 요(Yaw) 최대 우측”**으로 정의된다. 반대로 시동 해제는 **“스로틀 최하단, 요 최대 좌측”**이다.

PX4-Autopilot은 각 수신된 RC 채널의 PWM 값을 선형 보간 처리하여 [-1.0, 1.0]의 정규화된 실수(Float) 범위로 변환한 뒤 manual_control_setpoint uORB 토픽으로 발행한다. 펌웨어는 파라미터를 통해 이 “극단점“을 수학적 임계값(Threshold)으로 규정한다.

  • 스로틀 기저점: Z \le 0.1 (일반적으로 COM_RC_IN_MODE 등에 종속되나, 내부 하드코딩된 최소 스태틱 바운드가 존재)
  • 요 극단점: R \ge 0.9 (Arming), R \le -0.9 (Disarming)
graph TD
    A[manual_control_setpoint 갱신] --> B{Z (스로틀) < 0.1?}
    B -- No --> C[제스처 타이머 초기화 (Reset)]
    B -- Yes --> D{R (요) > 0.9?}
    D -- No --> C
    D -- Yes --> E[Arming 제스처 활성화 및 타이머 누적 시작]

이와 같이 단일 축에 대한 평가가 아니라 다변수(Multi-variable) 조건을 AND 논리로 묶어 첫 번째 관문을 설정한다. 이 방식은 Ardupilot의 스틱 파싱 로직과 개념적으로 유사하지만, PX4는 ManualControl 클래스 내에서 제스처 판독과 비행 동역학 연산을 더욱 철저히 브랜칭(Branching)하여 모듈 캡슐화를 유지한다.

2. 논리적 시간 카운터(Logical Time Counter) 적용

스틱이 지정된 극단점에 배치되었다고 해서 즉시 모터가 회전한다면 매우 위험하다. 기체가 추락하는 과정이나 스틱을 강하게 튕길 때 우발적으로 해당 임계값을 스쳐 지나갈 수 있기 때문이다. 이를 방어하기 위해 PX4는 **지속 시간 검증(Dwell Time Validation)**을 도입하였다.

src/modules/manual_control/ManualControl.cpp 혹은 Commander 스레드 내 타이머 로직은 루프 주기(\Delta t)마다 누적 카운터를 증가시킨다.

// 스틱 제스처 판별 루프의 추상화된 소스 코드 아키텍처
if (is_throttle_min && is_yaw_max_right) {
    if (_stick_gesture_start_time == 0) {
        _stick_gesture_start_time = hrt_absolute_time(); // 카운터 시작
    } else {
        // 현재 시간과 시작 시간의 델타값 검증
        hrt_abstime elapsed = hrt_absolute_time() - _stick_gesture_start_time;
        
        if (elapsed > COM_RC_ARM_HYST) {  // 예: 파라미터로 정의된 1,000,000 us (1초)
            trigger_arming();            // 커맨드 전송 함수 호출
            _stick_gesture_start_time = 0; // 중복 호출 방지를 위한 리셋
        }
    }
} else {
    _stick_gesture_start_time = 0; // 플로팅(Floating) 노이즈 등으로 조건 이탈 시 즉각 리셋
}

이 고해상도 타이머(hrt, High-Resolution Timer) 카운터는 로직의 핵심이다. RTOS 환경하에서 스케줄러 지연이나 일시적 인터럽트 핸들링으로 인해 주기가 흔들리더라도, 절대 시간(Absolute Time)을 스냅샷으로 비교하므로 1초라는 물리적 지연값을 정확하게 보장한다.

3. 시동 활성화 플래그(Arming Flag) 전이 아키텍처

논리적 시간 카운터를 통과하면 trigger_arming() 트랜잭션이 발생한다. 이때 직접 하드웨어 I/O로 접근하여 모터 핀에 PWM을 출력하는 것이 아니라, 아키텍처의 의도에 따라 vehicle_command uORB 메시지를 발행한다.

발행되는 메시지의 필드 구조는 다음과 같다:

  • command: VEHICLE_CMD_COMPONENT_ARM_DISARM (MAVLink 표준과 매핑 됨)
  • param1: 1.0 (Arming을 의미, Disarm은 0.0)
  • source_system: 내부 발생임을 증명하는 로컬 시스템 ID.

이 명령이 발행되면 src/modules/commander 블록의 백그라운드 스레드가 이를 취득한다. 커맨더 모듈은 스틱 제스처에서 기원한 시동 명령일지라도 다음의 안정성 트리(Safety Tree)를 통과시킨다.

  1. 인터락(Interlock) 상태: 물리적 안전 스위치(Safety Button)가 해제되었는가?
  2. 비행 전 점검(Pre-flight Check): EKF2가 수렴했고 센서 에러 플래그가 없는가?
  3. 착륙 상태(Landed State): 공중 시동(In-air arming) 허용 파라미터(CBRK_AIRSPD_CHK 등)를 위반하지 않았는가?

모든 조건이 충족되었을 때 비로소 actuator_armed 토픽의 armed 플래그 변수가 true로 트랜지션 되며, 믹서(Mixer) 또는 ControlAllocation 모듈이 액추에이터의 출력값을 유휴 스로틀(Idle Throttle) 레벨 이상으로 끌어올린다.

4. QGroundControl 및 MAVLink 관제와의 관계

조종기 스틱을 통한 시동은 GCS 기반의 시동 메커니즘과 충돌하지 않는다. 만약 조종사가 스틱으로 시동(Arm) 커맨드를 트리거하는 동시에, 관제사가 QGroundControl의 대시보드에서 시동 해제(Disarm) MAVLink 커맨드를 날린다면 어떻게 될까?

두 명령 모두 동일한 vehicle_command 버퍼에 인큐(Enqueue)되며 순차적으로 처리된다. PX4의 상태 머신(State Machine)은 순방향 논리 전이(Transitions)만을 허용하므로, 각 이벤트의 도달 시점에 따라 기체의 최종 무장 상태 플래그가 전이된다. 결국 스틱이라는 레거시 하드웨어 인터페이스를 디지털 노드의 하나로 추상화한 이 아키텍처 디자인은, 로스 2(ROS2) 에이전트 다중 편대 비행 시나리오에서도 높은 일관성과 안전성을 유지할 수 있는 토대가 된다.
