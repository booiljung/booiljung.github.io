29.3.2.3. 믹서(Mixer) 모듈로 직행하는 액추에이터 제어 배열 할당 및 제어값 클리핑(Clipping) 로직

본 소절에서는 FlightTaskManual에서 생성된 원시(Raw) RC 사상 데이터가 fw_att_control 모듈의 종단부를 거쳐 최종적으로 uORB 토픽(actuator_controls_0)으로 발행(Publish)되기 직전 수행되는 마지막 방어선(Last Line of Defense) 인 배열 할당(Array Allocation) 및 클리핑(Clipping) 알고리즘을 분석한다. 다이렉트 패스스루의 철학은 ’무개입’이지만, 수학적 범위를 벗어난 이상(Anomaly) 데이터가 하드웨어를 파괴하는 것까지 방관하지는 않는다.

1. actuator_controls_s 구조체의 배열 할당 역학

PX4의 제어 파이프라인에서 자세 제어기(Attitude Controller)와 믹서(Mixer) 또는 제어 할당기(Control Allocator)를 잇는 범용적인 다리 역할은 actuator_controls_s 구조체가 담당한다. 이 구조체 내부는 8개의 float 타입 원소를 갖는 control[8] 배열로 정의되어 있다.

수동 모드 패스스루 과정에서 할당되는 인덱스 매핑(Index Mapping) 프로토콜은 다음과 같이 하드코딩되어 있다.

  • control[0] (INDEX_ROLL): 에일러론(Aileron) 등 롤 기동을 유발하는 제어 성분
  • control[1] (INDEX_PITCH): 엘리베이터(Elevator) 등 피치 기동을 유발하는 제어 성분
  • control[2] (INDEX_YAW): 러더(Rudder) 등 요 기동을 유발하는 제어 성분
  • control[3] (INDEX_THROTTLE): 메인 추진 모터(Propulsion Motor) 제어 성분
  • control[4] ~ control[7]: 플랩(Flaps), 스포일러(Spoilers), 랜딩 기어(Landing Gear) 등 보조 채널 포트

이 배열 구조는 기체의 물리적 형상(예: 비행기, 플라잉 윙, v-tail 등)과 완전히 무관한 ‘추상화된 역학적 모멘트(Abstracted Mechanical Moment)’ 성분이다. 즉, 수동 모드 조종자는 특정 서보 모터 번호(PWM 1번, 2번 등)를 직접 구동하는 것이 아니라, “나는 롤을 50\% 생성하고 싶다“는 추상화된 의지(Intention)를 control[0] 버퍼에 할당하는 셈이다. 이 의지를 받아 어느 서보 모터를 몇 도 꺾을지 번역하는 책임은 전적으로 다음 스테이지인 믹서 레이어로 이양(Delegation)된다.

2. 하드웨어 보호를 위한 제어값 수학적 클리핑(Clipping) 방어벽

수동 모드에서는 중간 연산 루프가 구동되지 않으므로 조종기의 짐벌(Gimbal) 캘리브레이션이 약간 틀어져 있거나, MAVLink에서 악의적인(혹은 비정상적인) 오프셋 데이터가 유입될 경우 수학적 범위를 초과하는 인풋이 그대로 믹서로 직행할 위험이 존재한다.

믹서 엔진은 내부적으로 행렬 덧셈 및 곱셈을 수행하므로, \pm 1.0 범위를 벗어난 값이 유입되면 서보 모터의 물리적 기계 단점(Mechanical End-stop)을 강하게 때리거나(Binding), ESC 역기전력으로 보드가 타버리는 치명적 부하를 유발할 수 있다.

따라서 토픽 퍼블리싱 직전에 아주 가벼운 연산량의 수학적 제약(Mathematical Constraint) 체인이 가동된다.

// 액추에이터 제어 배열의 종단 클리핑 연산 구조 (개념적 코드)
void FixedwingAttitudeControl::Run()
{
    // ... [다이렉트 패스스루 배열 복사 완료 구간] ...

    // [최종 데이터 클리핑 (Clipping)]
    // 조종면(Control Surfaces): 역학적 대칭성을 가지므로 -1.0 ~ +1.0 구간 강제 구속
    _actuator_controls.control[0] = math::constrain(_actuator_controls.control[0], -1.0f, 1.0f);
    _actuator_controls.control[1] = math::constrain(_actuator_controls.control[1], -1.0f, 1.0f);
    _actuator_controls.control[2] = math::constrain(_actuator_controls.control[2], -1.0f, 1.0f);
    
    // 스로틀(Throttle): 단방향 모터(Unidirectional)이므로 0.0 ~ 1.0 구간 구속
    // (단, V-TOL이나 특정 가변 피치 로터의 경우 별도의 파라미터 매핑을 따름)
    _actuator_controls.control[3] = math::constrain(_actuator_controls.control[3],  0.0f, 1.0f);

    // 타임스탬프 스탬핑 (데이터 유효성 확인 목적)
    _actuator_controls.timestamp = hrt_absolute_time();

    // 믹서/할당기 스레드로 토픽 릴리즈(Release)
    _actuator_controls_pub.publish(_actuator_controls);
}

여기서 사용되는 math::constrain(값, 최소값, 최대값) 템플릿 함수는 내부적으로 if-else나 ternary 삼항 연산자로 최적화되어 동작하는, 연산 비용이 거의 0에 수렴하는 매크로적 루틴이다.

이 최후의 안전 파이프(Safe-pipe)를 통과한 데이터는 메모리를 떠나 uORB 네트워크로 방출(publish)되며, 이것으로 fw_att_control 내에서의 수동 제어(Manual Passthrough) 처리 스레드는 자신의 역할적 생명 주기(Lifecycle)를 모두 완수하게 된다.
