29.3.2.1. FlightTaskManual 기반 스틱 입력 데이터의 직접 메모리 복사 연산 과정

본 소절에서는 고정익 자세 제어기(fw_att_control)가 다이렉트 패스스루(Direct Passthrough)를 결정했을 때, 이 원시(Raw) 데이터를 최종 액추에이터 포맷으로 이관하는 전담 실무 객체인 FlightTaskManual 알고리즘의 최하단 메모리 복사 로직을 분석한다.

1. 비행 태스크(Flight Task) 다형성(Polymorphism)의 예외적 활용

PX4 시스템의 FlightTask 인터페이스는 본래 자율 주행 경로(Trajectory) 생성이나 부드러운 스틱 가속도 제한(Jerk-limited) 등을 수행하기 위해 고안된 객체지향적 추상화 계층이다. 예를 들어 FlightTaskManualAltitude 나 FlightTaskManualPosition 같은 파생 클래스들은 조종자의 스틱을 ’속도’나 ‘고도’ 목표치로 치환하기 위한 수많은 운동학 방정식을 내포하고 있다.

하지만 FlightTaskManual (가장 기저에 있는 순수 매뉴얼 태스크)은 이러한 객체지향적 설계 철학을 정면으로 배반(Betrayal)한다. 이 클래스의 update() 오버라이딩(Overriding) 메서드는 궤적 생성 연산을 완전히 포기하고, 단지 하나의 메모리 버퍼에서 다른 메모리 버퍼로 데이터를 이관하는 단순한 컨베이어 벨트(Conveyor Belt) 역할만을 자처한다.

2. 직접 메모리 맵핑(Direct Memory Mapping) 연산 해부

다이렉트 패스스루 시, FlightTaskManual 은 구독해 둔 정규화 \pm 1.0 데이터 토픽인 manual_control_setpoint 를 읽어들여 믹서(Mixer)가 바로 삼킬 수 있는 vehicle_torque_setpoint (과거 actuator_controls_0 배열) 토픽으로 원-투-원(One-to-one) 복사한다.

// src/lib/FlightTasks/tasks/Manual/FlightTaskManual.cpp 의 패스스루 연산 발췌
bool FlightTaskManual::update()
{
    // ... 상태 유효성 검사 생략 ...
    
    // 조종기 입력 데이터 캐싱 (기본적인 데드존/EXPO 처리는 29.2.3절의 전처리단에서 이미 완료됨)
    const float roll_in  = _manual_control_setpoint.roll;
    const float pitch_in = _manual_control_setpoint.pitch;
    const float yaw_in   = _manual_control_setpoint.yaw;
    const float thr_in   = _manual_control_setpoint.throttle;

    // [직접 메모리 복사 (Direct Copy) 로직]
    // 어떠한 튜닝 게인(Gain)이나 삼각함수, 미적분 개입 없이 순수 아날로그 값을 그대로 
    // 하위 액추에이터 목표 배열 구조체로 대입(Assignment)

    _actuator_controls.control[actuator_controls_s::INDEX_ROLL]  = roll_in;  // 서보 1 (Aileron)
    _actuator_controls.control[actuator_controls_s::INDEX_PITCH] = pitch_in; // 서보 2 (Elevator)
    _actuator_controls.control[actuator_controls_s::INDEX_YAW]   = yaw_in;   // 서보 3 (Rudder)
    
    // 스로틀의 경우 기체 설정에 따라 -1.0~1.0 구간을 0.0~1.0(단방향 모터)으로 리스케일링(Rescaling)할 수 있음
    _actuator_controls.control[actuator_controls_s::INDEX_THROTTLE] = 
                        math::max(0.0f, thr_in);

    // ... 플래그 마킹 후 리턴 ...
    return true;
}

이 C++ 코드 블록은 너무나 간결하여 학술적 분석의 산력을 떨어뜨리는 것처럼 보이나, 항공 소프트웨어 공학의 관점에서는 이 ‘아무 연산도 하지 않음(No-operation)’ 자체가 지니는 가치가 절대적이다.

  • O(1)의 상수 시간(Constant Time) 시간 복잡도를 가지므로 제어 루프 레이턴시(Latency) 지터(Jitter)를 거의 0에 수렴시킨다.
  • 부동소수점 오버플로(Overflow)나 NaN (Not a Number) 전파 등 수학적 엣지 케이스(Edge Case)가 발생할 논리적 여지를 원천적으로 제거한다.

3. 원시 맵핑 구조가 믹서(Mixer) 시스템에 미치는 영향

이처럼 철저하게 정규화된 직결 변수(roll_in, pitch_in 등)가 actuator_controls 배열에 탑재되면, 이 배열은 PWM 하드웨어 드라이버 바로 윗단에 자리 잡은 믹서(Mixer) 모듈로 하달된다.

만약 전통적인 테일-로터 방식의 비행기가 아니라 V-Tail(V 형 꼬리날개) 기체라 할지라도, 조종자는 단순히 피치(Pitch)와 요(Yaw) 입력을 FlightTaskManual을 통해 정해진 인덱스(INDEX_PITCH, INDEX_YAW)로 쑤셔 넣기만 하면 된다. V-Tail의 복잡한 기하학적 믹싱(예: 왼쪽 테일 서보는 Pitch + Yaw, 오른쪽은 Pitch - Yaw) 행렬 연산은 전적으로 믹서의 영역이며, 패스스루 파이프라인은 이 분업화(Decoupling) 경계를 철저히 준수함으로써 코드의 유지보수성(Maintainability)과 확장성을 동시에 담보한다.
