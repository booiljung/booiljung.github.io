29.2.1.2. 직접 메모리 접근(DMA) 기술을 활용한 수신 버퍼 처리 및 CPU 점유율 최소화 기법

본 소절에서는 고대역폭 직렬 수신기 프로토콜(예: TBS Crossfire/Tracer 400k+ baudrate, 텔레메트리 양방향 송수신 등) 환경에서 메인 비행 제어기(FC)의 코어 CPU 자원을 보존하기 위해 도입된, 직접 메모리 접근(DMA, Direct Memory Access) 기술 기반의 rc_input 스트림 수신 레이어 아키텍처와 버퍼 운영 커널 최적화(Optimization) 기법을 심층 분석한다.

1. 레거시(Legacy) 인터럽트 방식의 CPU 한계와 DMA 도입의 필연성

과거 SBUS(100Kbps) 등 상대적으로 저속의 수신 프로토콜 시대에는 MCU의 UART 주변기기가 바이트 단위를 수신할 때마다 매번 RX 인터럽트를 트리거(Trigger)하더라도 전체 CPU 리소스의 점유율이 여유로웠다. 그러나 현대의 고성능 자율 비행 펌웨어가 요구하는 막대한 EKF 센서 융합 처리량과 맞물려, CRSF나 MAVLink 텔레메트리 포트와 같이 고속 이중(Duplex) 직렬 통신을 쏟아내는 환경에서 매 바이트마다 인터럽트 문맥 교환(Context Switching)에 진입하는 기존 방식은 가혹한 오버헤드(Overhead) 캐스케이딩을 초래한다.

이러한 병목(Bottleneck) 한계를 해소하기 위해 PX4-Autopilot의 하부 소프트웨어 계층인 NuttX RTOS 기반의 시리얼 드라이버 아키텍처는 하드웨어 자체의 직접 메모리 접근(DMA) 컨트롤러를 적극적으로 채택하였다. 통신 데몬의 UART 포트 설정 시 특정 컴파일 파래미터(예: HW_UARTS 보드 타겟의 DMA 록인)를 활성화하면, 수신기 핀(RX)을 통해 인입되는 전기적 비트 열들은 더 이상 MCU 연산 코어의 ALU(산술논리연산장치)를 직접 거칠 필요가 없어진다. MCU 내부의 전담 DMA 브릿지 컨트롤러가 외부에서 유입되는 수신 바이트 데이터를 메인 시스템 RAM 내부의 순환 링 버퍼(Circular Buffer) 특정 블록에 어떠한 CPU 클럭 낭비 개입 없이 완전히 묵시적으로 백그라운드 복사(Background Copy) 투사해 버리기 때문이다.

2. 하프(Half) 및 풀(Full) 전송 완료 인터럽트와 대용량 링 버퍼 운용 역학

DMA를 차용하게 되면 1바이트마다 발생하던 개별 인터럽트 융단폭격은 소멸된다. 대신, CPU(메인 펌웨어 루프 스레드)에게 데이터의 무더기 갱신을 통지하기 위해 거시적인 버퍼 청크(Chunk) 단위의 통보(Notification) 체계로 패러다임이 전이(Shift)된다.

// NuttX UART DMA 수신 인터럽트 백엔드 처리에 대한 추상화 개념 로직
void uart_dma_rx_callback(DMA_HANDLE handle, uint8_t status)
{
    // DMA 하드웨어 컨트롤러가 설정된 DMA 버퍼 크기의 정확히 50%를 채웠거나(Half Complete)
    // 혹은 100% 버퍼 공간을 모두 소진했을 때(Full Complete)만 간헐적으로 호출됨
    
    if (status & DMA_STATUS_HALF_COMPLETE) {
        // 하위 절반 버퍼(0 ~ 사이즈/2) 영역 데이터 가용 통지
        notify_upper_layer(buffer_ptr, buffer_size / 2);
    } else if (status & DMA_STATUS_FULL_COMPLETE) {
        // 상위 절반 버퍼(사이즈/2 ~ 버퍼 끝) 영역 데이터 가용 통지
        notify_upper_layer(buffer_ptr + (buffer_size / 2), buffer_size / 2);
    }
}

위의 커널 내부 메커니즘에서 보듯, DMA 기반 rc_input 드라이버는 커다란 순환 링 버퍼를 메인 메모리에 할당해두고 DMA 채널 컨트롤러에게 목적지 주소로 위임한다. DMA 장치 회로가 쉴 새 없이 조종기 수신 프레임을 링 버퍼에 백그라운드로 밀어 넣는 동안, MCU 코어 CPU는 오로지 HT(Half Transfer, 절반 전송) 및 TC(Transfer Complete, 전송 완료) 두 번의 단속적인 인터럽트 시점에만 개입하여, 묵직해진 수십 바이트 분량의 통신 패킷 뭉텅이 데이터(Chunk) 전체를 한 번에 상위 객체 폴링(Polling) 대기열 큐로 일괄 보고 파싱 포워드(Forwarding) 처리한다.

더 나아가, 통신이 잠시 쉬어가는 아이들(Idle Time) 라인 현상을 식별하기 위해 UART 하드웨어 고유의 IDLE Line Interrupt 메커니즘이 접합된다. 만약 패킷 중간에 전송이 완료되어 버퍼 50% 분기점조차 도달하지 않은 어정쩡한 상태에서 통신이 정지할 경우, 이 추가 식별 인터럽트가 곧바로 남은 잔상 바이트 청크를 위로 통지(Flush)하여, RC 데이터의 조작 지연성(Latency)을 마이크로초 단위로 엄격히 극한 방어해 낸다. 이토록 영리하게 컴비네이션 조립된 하드웨어 앙상블 덕분에 고속 통신 환경에서도 CPU 점유율 대역은 1% 미만으로 억제되면서 제어 응답성은 손실 없이 최대로 담보된다.

3. DMA 캐시 일관성(Cache Coherency) 확보 및 데이터 부패 방지 소프트웨어 기법

고성능 Cortex-M7 혹은 최신 H7 아키텍처 기반의 픽스호크(Pixhawk) 컨트롤러 보드에서는 L1 데이터 캐시(D-Cache) 기능이 활성화되어 있어 시스템 속도를 배가시킨다. 그러나 오직 MCU 관점에서의 캐싱일 뿐, 독립적인 하드웨어 장치인 DMA 컨트롤러가 직접 메인 SRAM의 링 버퍼 내용을 변경해 버리면 CPU 코어 내부의 캐시 메모리 라인과 실제 메인 메모리(RAM) 간에 심각한 데이터 불일치(Data Incoherence) 부패 현상이 유발된다. CPU는 이미 쓰레기가 된 과거의 낡은 캐시 메모리 값을 꺼내어 읽는 치명적인 재앙이 발생하게 된다.

PX4와 기저 NuttX 운영체제는 이 민감한 메모리 물리 대역폭 침범 아키텍처 결함을 막기 위하여 커널 깊숙한 곳에서 정교한 캐시 무효화(Cache Invalidation) 소프트웨어 방어선을 가동한다. 상위 rc_input 혹은 디코더 파서 인스턴스가 링 버퍼에서 수신된 프레임을 꺼내어 열람(Read-out)하기 직전, 하부 드라이버는 명시적으로 데이터 캐시를 메인 시스템 메모리 블록과 강제 무효 동기화(Clean & Invalidate D-Cache 연산)시키는 arch_invalidate_dcache(address, size) 코어 명령을 선제 하드 선언한다.

오직 이러한 메모리 펜스(Memory Fence) 아키텍처 계층과 정밀 보정 과정이 무결하게 보장된 기반 위에, 고속 CRSF 양방향 텔레메트리 프레임이 깨어짐 없이 순결하게 유지되어 다중 채널을 타고 상위 파서 엔진(Protocol Parser) 입구로 안착할 수 있게 된다. 이것이 바로 극한의 신뢰성과 경량화된 퍼포먼스율(Performance Rate) 사이에서 아슬아슬하게 줄타기하는 고등 비행 제어기(FC) 항공 커널 레벨 데이터 처리 런타임의 미학적 공학이다.
