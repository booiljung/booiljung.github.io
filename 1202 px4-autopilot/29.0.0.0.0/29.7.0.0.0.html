29.7. 수동 제어 예외 상황(Edge Case) 처리 및 소프트웨어 디버깅 기법

무인항공기(UAV)의 제어 소프트웨어, 특히 PX4-Autopilot과 같은 고도화된 비행 제어 스택에서 수동 제어(Manual Control) 루프는 단순히 조종기(Radio Control, RC) 스틱의 변위를 액추에이터(Actuator) 출력으로 양도하는 1차원적 매핑(Mapping) 메커니즘이 아니다. 조종사의 의도가 기체의 동력학적 한계를 초과하거나, 하드웨어적 결함, 혹은 외부 환경적 요인으로 인해 센서가 발산하는 예외 상황(Edge Case)에 직면했을 때, 시스템은 즉각적이고 결정론적인(Deterministic) 안전 조치를 취해야 한다. 본 장에서는 PX4-Autopilot 최신 펌웨어(주로 v1.14 이상)를 기준으로 수동 비행 중 발생 가능한 극단적 예외 상황을 소프트웨어적으로 어떻게 식별하고 매니징하는지, 아울러 발생한 버그를 추적하기 위한 소프트웨어 디버깅(Software Debugging) 기법을 심층적으로 분석한다.

1. 수동 제어 예외 상황(Edge Case)의 정의와 아키텍처적 대응

수동 제어에서 발생하는 예외 상황은 크게 세 가지 카테고리로 분류할 수 있다.

  1. 입력 신호 결함(Input Signal Faults): 조종기 스틱의 하드웨어적 노이즈(Jitter), 단선으로 인한 프레임 손실(Frame Drop), 그리고 통신 지연(Latency).
  2. 동력학적 특이점(Dynamical Singularities): 짐벌 락(Gimbal Lock) 근접 시 쿼터니언 미분 발산 계산 오류, 그리고 자유 낙하(Freefall) 상황.
  3. 가속도 및 각속도 센서 포화(Sensor Saturation): 극단적 기동(Acroabatic Maneuvers)으로 인해 IMU 센서가 측정 한계를 벗어나는(Clipping) 현상.

이러한 예외 상황 발생 시, PX4-Autopilot의 커맨더(Commander) 모듈은 vehicle_status 및 manual_control_setpoint uORB 메시지의 변화를 실시간으로 폴링(Polling)하여 최상위 상태 머신(State Machine) 레벨에서의 펄백(Fallback) 방어 로직을 수행한다.

graph TD
    A[조종기 입력 (RC Input)] --> B{프레임 드롭/노이즈 검출?}
    B -- Yes --> C[RC Loss Failsafe (RTL/Land)]
    B -- No --> D[디지털 필터링 (Deadzone & LPF)]
    D --> E{자세 한계 임계값 초과?}
    E -- Yes --> F[목표 자세 클리핑 (Attitude Setpoint Clipping)]
    E -- No --> G[표준 제어 루프 진행 (Normal Control Loop)]
    F --> G
    G --> H{IMU 포화 감지 (Saturation)?}
    H -- Yes --> I[적분항 누적 정지 (Anti-windup)]
    H -- No --> J[액추에이터 출력 (Actuator Outputs)]
    I --> J

2. 동력학 및 센서 특이점(Singularity) 회피 로직

수동으로 극단적인 피치(Pitch)나 롤(Roll)을 입력할 경우, 오일러 각(Euler Angles) 기반의 제어는 90도 부근에서 수학적 특이점에 빠지는 코사인 분모 영점 문제(Gimbal Lock)를 수반한다. 이를 회피하기 위해 PX4-Autopilot은 수동 제어(Stabilized 등) 목표 생성 단계에서 쿼터니언(Quaternion) 대수학을 전면적으로 채용한다.

2.1 쿼터니언 기반 목표 자세 클리핑(Setpoint Clipping)

비행 모드가 수동 제어일지라도, 자세 제어기(Attitude Controller)는 조종사가 기체를 완전히 뒤집는(Inverted) 기동을 방지하기 위해 최대 경사각(예: MPC_MAN_TILT_MAX 파라미터)을 강제한다. 만약 스틱 입력이 이 값을 초과하면, PX4 소프트웨어 라이브러리(src/lib/mathlib) 내의 벡터 제한 함수가 동작한다.

// src/modules/mc_att_control/AttitudeControl.cpp 내의 클리핑 응용 예시
float tilt_max = math::radians(_param_mpc_man_tilt_max.get());
math::Quaternion q_target = generate_attitude_setpoint(stick_roll, stick_pitch);

// 목표 자세의 틸트 각도가 임계값을 초과하는지 스칼라 투영으로 검증
if (calculate_tilt(q_target) > tilt_max) {
    q_target = limit_tilt(q_target, tilt_max);
}

이와 같은 소프트웨어적 클리핑은 입력 배열의 무결성을 유지하고 제어기가 발산하지 않도록 보호한다. Ardupilot의 경우 ANGLE_MAX 파라미터를 통해 유사한 방어 로직을 구성하지만, PX4는 내부적으로 쿼터니언 행렬 연산을 더욱 심층적으로 추상화하여 행렬 곱셈 시 발생하는 부동소수점 오차 누적을 정규화(Normalization)하는 백그라운드 스레드를 독립적으로 둔다는 점에서 소프트웨어 아키텍처적 차이가 있다.

2.2 역동적 비행 시의 안티 와인드업(Anti-Windup) 기법

조종사가 스로틀을 급격히 낮추어 자유 낙하 상태에 돌입하거나 극단적인 루프(Loop)를 시도할 때, 액추에이터는 물리적 한계(PWM 최대/최소값)에 도달한다. 이 상황에서 위치 및 속도 PID 제어기의 적분항(Integral Term)이 계속 누적되면, 반대 방향으로 제어 입력이 들어갔을 때 반응 지연(Overshoot)이 발생하는 와인드업(Windup) 현상이 나타난다.

수동 제어 중 이러한 액추에이터 포화가 감지되면, src/lib/pid/PID.cpp의 제어 알고리즘은 즉시 적분 누적을 동결(Freeze)한다.

I[k] = \begin{cases} I[k-1] + e[k] \cdot \Delta t, & \text{if Actuator is NOT saturated} \\ I[k-1], & \text{if Actuator IS saturated} \end{cases}

테이블 셀에서의 행렬식 비교는 다음과 같이 표현될 수 있다.

제어 상태액추에이터 상태적분항 동작안전 조치 로직의 특징
정상 수동 비행$u< U_{max}$
극한 스틱 입력$u\ge U_{max}$

이 수동 비행의 예외적 동력학 상태를 방어하는 것은 결국 소프트웨어 제어기의 유연성에 달려있다.

3. 소프트웨어 디버깅 기법 (Software Debugging Techniques)

수동 제어 모드에서 기체가 비정상적인 진동(Damping)을 일으키거나, 사용자 스틱 입력에 지연(Latency)이 발생하는 이슈는 디버깅하기 매우 까다롭다. 기체의 충돌(Crash) 시, 원인이 하드웨어적 결함(모터 손상 등)인지, 펌웨어 내 수동 제어 매핑 로직의 결함인지 판단하기 위해서는 체계적인 소프트웨어 디버깅 기법이 동원되어야 한다.

3.1 ULog를 활용한 지연 시간 및 데이터 누락 분석

PX4의 비행 데이터 로깅 포맷인 ULog는 내부 uORB 메시지 트랜잭션을 실시간으로 기록한다. 수동 제어의 예외 상황 분석 시에는 QGroundControl(v4.3 이상 권장) 혹은 Flight Review 웹 분석 도구를 사용하여 입력 경로의 병목을 진단해야 한다.

  • 조사 토픽(Topics): input_rc \rightarrow manual_control_setpoint \rightarrow vehicle_local_position_setpoint \rightarrow actuator_outputs.
  • 디버깅 절차: input_rc에 타임스탬프가 찍힌 시점부터 actuator_outputs 레지스터가 갱신되는 시점 간의 델타 타임(\Delta t)을 측정하여 RTOS 스케줄러의 기아(Starvation) 현상이 발생하지 않았는지 추적한다.

3.2 SITL(Software In The Loop) 및 GDB 연동 디버깅

수동 제어 스택을 커스터마이징하거나 새로운 예외 처리 로직(예: 특정 스틱 제스처 입력 시 낙하산 강제 전개 등)을 추가할 때, 실제 기체를 바로 띄우는 것은 리스크가 크다. 따라서 Gazebo 기반의 SITL 시뮬레이션 환경에 GDB(GNU Debugger)를 연결하여 코드 레벨에서의 중단(Break) 및 메모리 추적을 실시해야 한다.

# PX4 펌웨어 소스 디렉토리에서 SITL 디버깅 모드로 빌드 및 실행
make px4_sitl gazebo-classic_iris_ide

위 런처를 통해 프로세스를 띄우면 호스트 OS(Linux 환경 등)의 GDB 패널을 활용해 src/modules/manual_control/ManualControl.cpp의 특정 분기문에 브레이크포인트(Breakpoint)를 설정할 수 있다. 조종기 조이스틱 장치 패스스루(Passthrough)를 사용하여 스틱을 극한으로 이동시킬 때, apply_calibration() 함수가 변수 범위를 [ -1.0, 1.0 ] 사이에서 올바르게 클리핑(Clipping) 하는지 메모리 주소를 직접 읽어 실시간으로 검증하는 것이 필수적이다.

이와 같은 하위 모듈 수준의 실시간 동적 코드 분석은, 복잡화된 Ardupilot의 단일 모놀리식(Monolithic) 아키텍처 대비 PX4-Autopilot의 독립적 모듈(uORB 기반) 아키텍처가 제공하는 막강한 검증 이점이 된다.

4. 결론

최신 PX4-Autopilot에서 수동 제어 기능은 단순히 사용자 입력을 전달하는 파이프가 아니라, 인간의 불완전한 제어 입력을 동력학적으로 안전하고 유효한 신호로 정제(Sanitization)하는 고도의 안전망(Safeguard)이다. 특이점, 입력 포화 및 통신 결함 등 예측 가능한 모든 엣지 케이스는 소프트웨어적으로 처리되어야 하며, 시스템 개발자는 ULog 아키텍처와 RTOS 레벨의 디버깅 기술(GDB 및 SITL)을 융합하여 제어 시스템의 무결성을 엄격하게 입증해야 한다.
