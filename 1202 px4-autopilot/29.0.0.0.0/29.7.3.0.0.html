29.7.3. 비행 데이터 로깅(ULog) 기반 수동 모드 제어 성능 심층 분석법

무인항공기(UAV) 개발에 있어 조종사의 체감 비행성(Handling Qualities)과 제어 시스템의 응답성(Responsiveness)을 정량적으로 평가하고 튜닝하기 위해서는, 직관적인 감각에 의존하는 맹목적 테스트(Blind Flight Testing)를 지양해야 한다. 대신, 기체 내부에서 발생하는 고주파(High-frequency) 제어 트래픽을 정밀하게 추출하고 보드 플롯(Bode Plot)이나 응답 지연(Latency) 그래프로 렌더링하는 기법이 필수적이다.

PX4-Autopilot은 독자적이고 강력한 바이너리 직렬화 포맷인 ULog를 사용하여 마이크로초(Microsecond, \mu s) 단위의 타임스탬프를 보존한 채 uORB 메시지 트래픽을 저널링(Journaling)한다. 본 장에서는 ULog 시스템 아키텍처를 개괄하고, 수동 모드 제어루프 검증을 위해 어떤 메시지 토픽을 계측하고 교차 검증해야 하는지 심층 분석한다.

1. PX4 Logger 스레드 및 ULog 아키텍처 (Source Code Analysis)

ULog 파일은 단순한 비행 텔레메트리 덤프가 아니다. src/modules/logger 스레드에서 돌아가는 이 모듈은 사전에 정의된 로깅 프로파일(SDLOG_PROFILE 파라미터)에 따라 uORB 토픽 큐를 폴링(Polling)하고, 이를 비동기적(Asynchronous)으로 SD 카드 캐시 버퍼에 기록한다.

graph TD
    A[uORB Topics (센서/제어/추정)] --> B[Logger 모듈 (배치 버퍼링)]
    B --> C{SD 카드 쓰기 속도 병목 발생?}
    C -- Yes --> D[데이터 드롭(Drop) 카운터 증가 및 패킷 폐기]
    C -- No --> E[ULog 바이너리 패킷 직렬화]
    E --> F[SD 카드 파일 I/O 기록]

수동 제어 성능 분석처럼 고해상도(예: 250Hz 이상의 actuator_outputs) 분석이 필요할 경우, SD 카드의 클래스가 낮아 병목 현상이 발생하면 중요한 샘플들이 드롭되어 분석 무결성이 훼손된다. 따라서 고성능 튜닝을 위해서는 SDLOG_DIRS_MAX 및 로깅 버퍼(SDLOG_MISSION 등) 설정을 최적화하여 ड्रॉप아웃을 원천 차단해야 한다.

2. 수동 제어 성능 튜닝을 위한 필수 추적 토픽 (Required Topics)

수동 모드(Acro, Stabilized, Manual)의 제어 경로 최적화를 분석하기 위해서는 입력(Input), 추정(Estimation), 그리고 출력(Output) 단계를 나타내는 대표 토픽들을 시간 축으로 정렬(Time-Alignment)하여 보아야 한다.

2.1 사용자 제어 의도 인지 단계

  • input_rc: 하드웨어 수신기가 포착한 원시 PWM 채널 값 (대개 50Hz ~ 150Hz).
  • manual_control_setpoint: 노이즈 데드존(Deadzone) 및 스케일링이 끝난 정규화 [-1.0, 1.0] 명령 객체. 이 두 토픽 간의 신호 지연 커브는 필터(Low Pass Filter 등)에 의한 오버슈트나 지연을 나타낸다.

2.2 제어기(Controller)의 추종(Tracking) 검증

  • vehicle_attitude_setpoint vs vehicle_attitude: 안정화(Stabilized) 모드에서 기체가 조종사의 타겟 각도(예: 전방 30^\circ 틸트)를 얼마나 빨리 쫓아가는지(Rise Time)와 진동(Oscillation) 여부를 파악하는 핵심 지표.
  • vehicle_rates_setpoint vs vehicle_angular_velocity: 아크로(Acro) 모드의 순수 각속도 제어 루프 대역폭 분석에 사용된다.

2.3 물리 계층(Physical Layer) 응답 및 포화 판별

  • actuator_outputs: 제어기가 모터(ESC)로 산출해낸 최종 제어 인가 값(PWM, DShot 등).
  • 이 값을 살펴보면 수동 비행 중 특정 기동 시 하나의 모터가 100\%로 치솟는 포화(Saturation) 현상이 언제, 왜 발생했는지 추론할 수 있다.

3. ULog 렌더링 및 교차 분석 도구(Toolchain) 활용 기법

개발자가 리눅스(Linux) 환경에서 확보한 .ulg 확장자의 바이너리 포맷을 분석하기 위해 PX4 생태계는 다음과 같은 파이프라인을 지원한다.

  1. Flight Review (웹 기반 플랫폼): 가장 접근성이 좋은 1차원적 분석 툴이다. 로그를 업로드하면 서버가 이 데이터를 파싱(Parsing)하여 PID 추종 그래프, 3D 비행 궤적, FFT 기반의 노이즈 주파수 분석 그래프를 자동 생성해 준다.
  2. PlotJuggler: ROS2 환경 시스템 분석 시 주로 사용하는 오프라인 C++ 기반 렌더링 엔진이다. ULog 플러그인을 활성화하면 1GB 이상의 방대한 로그도 실시간 줌/팬(Zoom/Pan) 연산을 통해 빠르게 디코딩(Decoding) 가능하며, 여러 토픽 간의 커스텀 파이썬 스크립트를 통한 델타(Delta) 렌더링을 지원한다.

4. Ardupilot DataFlash Log와의 로깅 시스템 철학 비교

Ardupilot은 전통적으로 DataFlash Log(.bin 포맷)를 통해 비행 정보를 저장한다. DataFlash는 로깅 구조가 정적(Static Table)으로 할당되는 경향이 강해 고해상도 로그를 위해 LOG_BITMASK 값을 비트 연산으로 활성화해야 하는 번거로움이 있다.

반면 PX4의 ULog 포맷은 페이로드(Payload) 크기가 런타임에 동적으로 컴파일되는 메타데이터 패킷 시그니처를 내장하고 있다. 이는 사용자가 src/modules/logger/logged_topics.cpp에 특정 커스텀 uORB 메시지(예: 독자적인 짐벌 락 회피 트리거 변수 등) 단 한 줄만 추가하더라도 어떠한 외부 파서(Parser) 수정 없이 모든 툴체인(QGC, Flight Review 등)에서 즉시 시각화할 수 있는 강력한 유연성을 제공한다.

결과적으로 PX4의 ULog 파이프라인은 인간의 개입이 가장 두드러지는 수동 모드의 성능 튜닝에 있어, 비행체의 미세한 지연과 모터 포화 병목을 개발자의 시야에 가장 선명하게 투영해 주는 X-Ray 역할을 수행한다.
