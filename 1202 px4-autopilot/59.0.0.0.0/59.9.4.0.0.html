59.9.4. 비행 로그 데이터와 GCS 시스템 클럭 간의 시계열 위상 오차 보정을 위한 선형 회귀(Linear Regression) 기반 동기화 위상 맞춤

1. 개요

지상 관제 시스템(Ground Control Station, GCS)과 무인기(UAV)는 물리적으로 분리된 별개의 컴퓨팅 환경을 가지며, 각각 독립적인 하드웨어 클럭(Hardware Clock)에 의해 구동된다. 드론 내부의 실시간 운영체제(RTOS, 예: NuttX)는 부팅 시점부터 경과한 마이크로초 단위의 모노토닉 시계(Monotonic Time, hrt_absolute_time)를 사용하는 반면, GCS는 범용 운영체제(OS, 예: Windows, Linux)의 시스템 클럭(유닉스 시간, Epoch Time)을 사용한다.
두 시스템에서 생성되는 시계열 데이터(Time-Series Data)를 사후 분석하거나 실시간 센서 융합(Sensor Fusion)을 수행하기 위해서는 반드시 동일한 시간축(Time Axis) 상에 놓아야 한다. 이때 무선 데이터 링크를 통해 전달되는 MAVLink 메시지는 전파 지연(Propagation Delay), 큐잉 지연(Queueing Delay), 그리고 운영체제의 스케줄링 변화에 따라 비결정론적(Non-deterministic) 오차를 수반한다.

본 장에서는 MAVLink의 TIMESYNC 메시지를 활용한 왕복 지연 시간(Round-Trip Time, RTT) 측정 원리를 분석하고, 누적되는 위상 오차(Phase Offset) 및 클럭 드리프트(Clock Drift)를 보정(Compensation)하기 위해 컴파일타임 및 런타임에 선형 회귀(Linear Regression) 알고리즘을 적용하는 소프트웨어 시스템 공학적 기법을 기술한다.

기준 시스템 버전 정보

  • PX4-Autopilot: v1.14.x
  • QGroundControl: v4.3.x 이상

2. MAVLink TIMESYNC 프로토콜 메커니즘

PX4는 시계열 오차 보정을 위해 MAVLink 메시지인 TIMESYNC (#111) 프로토콜을 구현하여 GCS와 기체 간 시간 위상 맞춤을 수행한다.

과정은 동기화 요청(Request)과 응답(Response)으로 이루어진다:

  1. GCS \rightarrow PX4 (요청): GCS가 현재의 시스템 시간 t_1을 패킷(tc1 보관)에 담아 전송.
  2. PX4 수신 및 회신: 기체가 패킷을 수신한 즉시 기체의 내부 모노토닉 시간 ts_1을 기록하고 응답.
  3. GCS 수신: 응답 패킷이 도착한 순간 GCS 시스템 시간 t_2를 기록.

RTT(왕복 지연)는 \Delta t = t_2 - t_1 이며, 단방향 전송 지연(One-way Delay)은 대칭적(Symmetric)이라고 가정 시 \Delta t / 2 가 된다.
따라서, 특정 순간의 오프셋(Offset) O는 수학적으로 다음과 같이 추정된다:
O = t_1 + \frac{\Delta t}{2} - ts_1

단편적인 RTT 측정만으로는 순간적인 네트워크 지터(Jitter)에 의해 심각한 오류가 발생할 수 있다. QGroundControl 및 심화 GCS 시스템에서는 연속된 TIMESYNC 표본 데이터를 수집하여 통계적 보정을 수행한다.

3. 선형 회귀(Linear Regression) 알고리즘 기반 타임스탬프 동기화

3.1 클럭 드리프트(Clock Drift)의 수학적 모델

하드웨어 오실레이터(Oscillator)의 주파수 변동이나 온도 차이로 인해 GCS와 PX4의 시간은 흐르는 “속도” 조차 미세하게 다르다. 이를 클럭 스큐(Clock Skew) 또는 클럭 드리프트라고 부른다.
GCS 시간 T_{GCS} 와 PX4 시간 T_{PX4} 간의 관계는 1차 선형 방정식으로 근사할 수 있다:

T_{GCS} = \beta_1 T_{PX4} + \beta_0 + \epsilon

  • \beta_1: 클럭 스큐 요소 (이론적으로 1.0에 매우 가깝지만 정확히 1은 아님)
  • \beta_0: 초기 위상 오버셋(Phase Offset)
  • \epsilon: 측정 노이즈 (통신 지연의 변동성)

2.1 최소 자승법(Ordinary Least Squares) 적용

연속적으로 수집된 n 개의 (PX4 시간, GCS 추정 시간) 쌍 (x_i, y_i) 에 대하여 최소 자승법을 적용하여 잔차 제곱합(Residual Sum of Squares, RSS)을 최소화하는 매개변수 \beta_1 과 \beta_0 를 도출한다:

매개변수산출 공식
Skew (\beta_1)\beta_1 = \frac{\sum_{i=1}^{n} (x_i - \bar{x})(y_i - \bar{y})}{\sum_{i=1}^{n} (x_i - \bar{x})^2}
Offset (\beta_0)\beta_0 = \bar{y} - \beta_1\bar{x}

이때, QGroundControl 아키텍처에서는 단순 선형 회귀의 민감도를 낮추기 위해, RTT가 너무 긴 표본(아웃라이어, Outlier)은 이동 평균(Moving Average) 혹은 중앙값 필터링(Median Filtering)을 거쳐 제거한 뒤 회귀선(Regression Line)을 업데이트한다. 이를 통해 시계열 로그가 완벽하게 시간 정렬(Time Aligned)된 상태로 로컬 데이터베이스에 기록된다.

3. GCS 시스템 공학적 적용: 로컬 데이터베이스와 사후 분석

시계열 데이터베이스(Time-Series DBMS)나 SQLite에 기록될 때, GCS 수신 시간(Received Time)만으로는 실시간 제어 스텝(Control Step)을 엄밀하게 분석할 수 없다. 기체가 바람의 영향을 받아 제어 개입을 시작한 시점과 GCS 관제화면에서 시각적 반응을 보인 시점 사이의 지연 오차를 식별하려면 **동기화된 기체 생성 시간(Synchronized Vehicle Generation Time)**이 절대적인 키열(Primary Timeline)이 되어야 한다.

독자적인 GCS 파이프라인(Pipeline)을 개발할 경우, mavlink_msg_timesync_decode() 함수를 주기적 백그라운드 스레드로 가동하고 회귀 모델을 실시간으로 업데이트 하는 모듈을 VehicleDataManager와 같은 코어 싱글톤(Singleton) 컨텍스트에서 관리하는 소프트웨어 디자인 패턴이 권장된다.

QGroundControl(QGC) v4.3 소스 코드의 src/Vehicle/Vehicle.cc 나 통신 매니저 부분을 살펴보면 이러한 TIMESYNC 교환이 주파력을 갖고 오프셋 값을 보정하여 내부 QDateTime과 맵핑하는 로직이 내재되어 있음을 알 수 있다. 이 로직은 QML 프론트엔드 단에 “드론 연결 시작 시차” 등으로 전달되어 영상(Video Streaming)과의 싱크를 맞출 때에도 핵심적인 역할을 담당한다.

4. 결론

PX4-Autopilot의 내부 시간 체계와 GCS 시스템 클럭 간의 불일치는 선형 회귀(Linear Regression)를 통한 위상 오차 보정 메커니즘에 의해 완벽하게 극복된다. 단순 오프셋 추가 방식에서 벗어나, 클럭 드리프트를 모델링하고 통신 지연 분포를 필터링하는 접근법은 고도로 정밀한 임무 비행 로그 사후 분석과 멀티 에이전트 다기체 동기화(Multi-agent Synchronization)의 기반 시스템으로 작동한다.
