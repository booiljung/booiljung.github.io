59.2.4 다중 스레드(Multi-threading) 환경의 자원 경합(Race Condition) 방지를 위한 뮤텍스(Mutex) 및 세마포어(Semaphore) 기반 동기화 제어

지상 관제 시스템(GCS, Ground Control Station)은 비행 중인 무인항공기(UAV)로부터 수신되는 방대한 양의 MAVLink 텔레메트리(Telemetry) 데이터를 실시간으로 파싱하고, 사용자 인터페이스(UI)를 업데이트하며, 동시에 비디오 스트리밍을 디코딩하고 로그를 기록하는 등 고도의 동시성(Concurrency)을 요구한다. 이러한 복잡한 병렬 처리 구조를 구현하기 위해 다중 스레드(Multi-threading) 아키텍처는 필수적이다. 그러나 여러 스레드가 동시에 동일한 메모리 공간이나 공유 자원(Shared Resource)에 접근하고 이를 수정하려 할 때, 실행 순서나 타이밍에 따라 결과값이 달라지거나 데이터가 손상되는 자원 경합(Race Condition) 현상이 발생할 수 있다. 이러한 데이터 무결성(Data Integrity) 훼손은 GCS의 오작동을 유발하여 치명적인 비행 사고로 직결될 수 있으므로, 철저한 동기화 제어(Synchronization Control)가 요구된다.

본 절에서는 자체적인 GCS를 설계할 때 자원 경합을 방지하기 위한 핵심 기법인 뮤텍스(Mutex)와 세마포어(Semaphore)의 원리를 이해하고, 이를 코드 레벨에서 어떻게 안전하고 효율적으로 적용할 수 있는지 살펴본다. (참고로 QGroundControl 및 자체 개발 GCS는 주로 C++17 이상의 표준이나 Qt 프레임워크를 기반으로 개발되므로, 이를 기준으로 설명한다.)

1. 자원 경합(Race Condition)과 임계 구역(Critical Section)

1.1 자원 경합의 치명적 위험성

두 개 이상의 스레드가 공유 데이터에 동시 접근하여 변경을 시도할 때 발생하는 데이터 불일치 문제를 자원 경합이라고 한다. 예를 들어, 네트워크 수신 스레드가 MAVLink 패킷 버퍼에 데이터를 기록하는 도중에, UI 렌더링 스레드가 해당 버퍼의 절반만 업데이트된 상태의 값을 읽어 화면에 표시한다면, 화면에는 유효하지 않은(비정상적으로 튀는) 고도나 위치 값이 출력될 것이다. 조종자는 이 잘못된 정보를 바탕으로 위험한 수동 개입을 할 수 있으며, 이는 자율 비행 시스템에서 절대적으로 피해야 하는 상황이다.

1.2 임계 구역(Critical Section)의 정의

자원 경합이 발생할 수 있는 코드의 영역을 임계 구역(Critical Section)이라고 한다. 즉, 전역 변수, 정적 변수, 공유 자료구조(예: 큐, 맵, 리스트 등)에 접근하고 수정하는 코드 블록이다. 이 구역에는 오직 하나의 스레드만 진입하여 작업을 마칠 수 있도록 상호 배제(Mutual Exclusion) 조건을 만족시켜야 한다.

2. 뮤텍스(Mutex, Mutual Exclusion) 기반의 동기화

2.1 뮤텍스의 원리

뮤텍스는 ’상호 배제’의 약자로, 오직 하나의 스레드만이 자원에 접근할 수 있도록 하는 락(Lock) 메커니즘이다. 공유 자원에 접근하려는 스레드는 반드시 뮤텍스를 획득(Lock)해야 하며, 사용을 마친 후에는 반환(Unlock)해야 한다. 이미 다른 스레드가 뮤텍스를 점유하고 있다면, 새로 접근하려는 스레드는 뮤텍스가 반환될 때까지 블로킹(Blocking) 상태로 대기(Sleep)하게 된다.

2.2 GCS 환경에서의 뮤텍스 활용 예시

GCS 내부에서 PX4-Autopilot으로부터 수신한 기체 상태 정보(VehicleState 구조체)를 여러 스레드(네트워크 스레드, 로깅 스레드, 화면 갱신 스레드)가 공유하는 상황을 가정해 보자.

#include <mutex>

class VehicleManager {
private:
    VehicleState _vehicleState;
    std::mutex _stateMutex;

public:
    // 네트워크 스레드에서 주기적으로 호출하여 상태 업데이트
    void updateVehicleState(const VehicleState& newState) {
        std::lock_guard<std::mutex> lock(_stateMutex); // 락 획득 (스코프를 벗어나면 자동 해제)
        _vehicleState = newState;
    }

    // UI 스레드에서 화면을 갱신하기 위해 현재 상태 조회
    VehicleState getVehicleState() {
        std::lock_guard<std::mutex> lock(_stateMutex); // 락 획득
        return _vehicleState;
    }
};

위 예시에서 std::lock_guard를 활용하여 RAII(Resource Acquisition Is Initialization) 패턴을 구현함으로써, 예외 발생 시에도 락이 안전하게 해제되도록 구성하였다. QMutex 및 QMutexLocker를 활용하는 방법 또한 Qt 기반 GCS에서 널리 쓰이는 표준적인 접근 방식이다.

2.3 교착 상태(Deadlock) 회피

뮤텍스를 사용할 때 가장 주의해야 할 함정은 ’교착 상태(Deadlock)’이다. 두 개 이상의 스레드가 서로가 점유하고 있는 뮤텍스의 락 해제를 무한정 기다리는 상태를 의미한다. GCS 시스템이 교착 상태에 빠지면, UI가 프리징(Freezing)되거나 조종 명령 전달이 중단되는 최악의 상황이 발생한다.
교착 상태를 방지하기 위해서는 다음의 규칙을 준수해야 한다.

  • 락의 순서 강제: 여러 개의 락을 획득해야 할 경우, 항상 동일한 순서로 점유되도록 설계한다.
  • 락 유지 시간 최소화: 뮤텍스를 점유한 상태에서 파일 I/O나 네트워크 블로킹 호출 등 시간이 오래 걸리는 작업을 수행하지 말아야 한다.

3. 세마포어(Semaphore) 기반의 동기화

3.1 세마포어의 원리

세마포어는 뮤텍스와 유사하지만, 단순히 상태를 Lock/Unlock 하는 것을 넘어 공유 자원에 접근할 수 있는 스레드의 ’최대 허용 개수’를 정수 카운터로 관리하는 형태의 동기화 기법이다. 상태값(카운트)이 양수일 경우 접근이 허용되며(카운트 감소), 0일 경우 대기하게 된다(블로킹).
이진 세마포어(Binary Semaphore)는 카운트가 1인 경우로, 뮤텍스와 비슷하게 작동하지만 소유권(Ownership) 개념이 없다는 차이가 있다(즉, 락을 건 스레드와 해제하는 스레드가 달라도 무방하다). 카운팅 세마포어(Counting Semaphore)는 여러 스레드가 제한된 크기의 리소스 풀에 동시 접근할 때 유용하다.

3.2 생산자-소비자 패턴 (Producer-Consumer Pattern) 설계

GCS에서 세마포어의 가장 이상적인 적용 사례는 생산자-소비자 아키텍처이다. MAVLink 데이터 파이프라인이나 로그 기록 시스템을 예로 들 수 있다. 수신 스레드(생산자)가 버퍼에 데이터를 채우고, 파싱 혹은 로깅 스레드(소비자)가 이를 가져다가 처리하는 구조이다.

sequenceDiagram
    participant P as 수신 스레드 (생산자)
    participant Q as MAVLink 버퍼 큐
    participant S as 세마포어 (데이터 수)
    participant C as 파서 스레드 (소비자)

    P->>Q: 수신된 MAVLink 패킷 Enqueue()
    P->>S: 세마포어 Signal() (카운트 +1)
    
    loop 파서 스레드 루프
        C->>S: 세마포어 Wait() 대기...
        Note over C,S: 세마포어 카운트가 1 이상이면 즉시 진입,<br/>0이면 데이터가 올 때까지 Blocked 상태
        S-->>C: 진입 허용 (카운트 -1)
        C->>Q: 패킷 Dequeue() 및 처리
    end

세마포어는 데이터가 없을 때 파서 스레드를 유휴(Idle) 상태로 완전히 잠재우므로, while(true) 형태의 폴링(Polling) 루프로 인한 불필요한 CPU 사이클(Busy-waiting) 소모와 발열을 방지할 수 있다.

4. 효율적인 동기화를 위한 GCS 아키텍처 설계 지침

뮤텍스와 세마포어는 확실한 동기화를 보장하지만, 무분별하게 사용할 경우 스레드의 병목 현상(Bottleneck) 및 우선순위 역전(Priority Inversion)을 유발하여 GCS의 실시간성을 해칠 수 있다. 따라서 GCS 시스템 레벨에서 다음과 같은 최적화 설계가 동반되어야 한다.

  1. 상태 동기화 최소화 (Message Passing 권장)
  • 스레드 간 메모리를 공유하고 뮤텍스를 거는 방식보다는, 메시지나 이벤트를 큐를 통해 비동기적으로 전달받는 발행-구독(Pub-Sub) 혹은 Actor 모델을 채용하는 것이 좋다.
  • Qt 프레임워크를 기반으로 한다면, 스레드 간 Signal-Slot 메커니즘(Qt::QueuedConnection)을 활용하면 내부적으로 이벤트 루프 큐를 통해 안전하게 데이터를 복사 전달하므로, 개발자가 직접 뮤텍스를 관리할 위험도를 대폭 낮출 수 있다.
  1. Wait-Free 버퍼 및 Lock-Free 자료구조 활용 검토
  • 실시간 제어 루프나 MAVLink 인터페이스 레이어 등 1ms의 딜레이도 아쉬운 고빈도 처리 병목 지점에는, 뮤텍스를 배제하고 아토믹 연산(std::atomic)과 CAS(Compare-and-Swap) 명령어 기반의 Lock-Free 원형 버퍼(Circular Buffer) 구성을 권장한다.
  1. 우선순위 역전(Priority Inversion)에 대한 대비
  • 낮은 우선순위의 스레드가 락을 선점하여, 높은 우선순위(예: 조종 명령 전송 스레드) 스레드가 락이 풀릴 때까지 대기하게 되어 시스템 전체의 실시간성이 무너지는 현상이다. 이를 피하기 위해 실시간 운영체제(RTOS) 환경에서는 ‘우선순위 상속(Priority Inheritance)’ 프로토콜을 사용하지만, GCS가 구동되는 일반적인 범용 데스크톱 OS(Linux, Windows 등)에서는 설계 단계에서 실시간 스레드가 접근하는 락과 비실시간 스레드가 접근하는 락의 영역을 철저하게 분리하는 아키텍처적 격리가 필수적이다.
