59.3.2 이기종 하드웨어 간의 엔디안(Endianness) 불일치 해소를 위한 바이트 순서 변환 및 비트 필드 마스킹(Bit-field Masking) 기법

1. 개요: 이기종 통신 환경에서의 데이터 정렬 문제

지상 관제 시스템(GCS, Ground Control Station)과 비행 제어 장치(FCU, Flight Control Unit)인 PX4-Autopilot은 종종 서로 다른 아키텍처의 프로세서를 기반으로 구동된다. 일반적인 데스크톱 기반의 GCS(Intel x86/x64)와 드론에 널리 탑재되는 마이크로컨트롤러(ARM Cortex-M 직군 등) 사이에 데이터를 송수신할 때, 가장 먼저 직면하는 호환성 문제는 엔디안(Endianness, 바이트 저장 순서) 불일치와 구조체 데이터 정렬(Data Alignment) 문제이다.

네트워크를 통해 혹은 비동기 직렬 포트(Asynchronous Serial)를 통해 전송되는 바이너리(Binary) 데이터 스트림은 바이트 단위로 쪼개져 전달되므로, 이를 수신부 코어에서 다중 바이트 자료형(예: uint32_t, float)으로 재조립할 때 송신부와 동일한 기준을 적용해야 한다. 본 항목에서는 하드웨어 종속적인 바이트 배치 특성을 극복하고 안정적인 MAVLink 미들웨어를 구축하기 위한 바이트 순서 변환(Byte Swapping) 및 비트 필드 마스킹(Bit-field Masking) 기술을 깊이 있게 다룬다.

참고: 본 항목은 PX4 펌웨어 v1.14 이상 및 QGroundControl v4.3.0 이상을 기준으로 하며, MAVLink v2.0 프로토콜 명세에 입각하여 설명한다.

2. MAVLink 프로토콜의 바이트 순서 정책

MAVLink 통신 규격에서는 성능 최적화와 구현의 단순화를 위해 **리틀 엔디안(Little-Endian)**을 유선 프로토콜(Wire Protocol)의 표준 바이트 순서로 강제한다. 이는 인터넷 프로토콜(IP)의 표준 네트워크 바이트 순서(Network Byte Order)가 빅 엔디안(Big-Endian)인 것과는 대조적인데, 현대의 주요 호스트 및 임베디드 코어(x86, x64, ARM)들이 기본적으로 리틀 엔디안을 채택하고 있음을 반영한 설계적 결정이다.

  • Little-Endian: 데이터의 하위 바이트(Least Significant Byte, LSB)가 메모리의 낮은 주소에 먼저 저장된다.
  • Big-Endian: 데이터의 상위 바이트(Most Significant Byte, MSB)가 메모리의 낮은 주소에 먼저 저장된다.

만일 개발 중인 GCS가 PowerPC나 특정 네트워크 칩 등 빅 엔디안 호스트 기반에서 돌아간다면, MAVLink 패킷을 수신(Deserialization)하고 송신(Serialization) 시에 반드시 엔디안 변환을 거쳐야 한다.

3. 이식성 있는 바이트 순서 변환 기법 (Byte Swapping)

빅 엔디안 시스템 환경에서 MAVLink 헤더 및 페이로드 데이터를 추출하기 위해서는, 리틀 엔디안 형태의 바이트 배열 스트림을 호스트의 엔디안 형식으로 변환해야 한다. C/C++ 표준에서 이러한 변환은 다음과 같은 논리 시프트 연산과 비트 OR(|) 연산을 통해 안전하고 이식성 높게(Portable) 구현된다.

#include <stdint.h>

// 16비트 정수바이트 스와핑 상수(Macro)
#define MINIMAL_BSWAP_16(x) \
    ((((x) & 0xff00) >> 8) | \
     (((x) & 0x00ff) << 8))

// 32비트 정수 바이트 스와핑 로직 구현
static inline uint32_t byte_swap_32(uint32_t val) {
    return ((val & 0xFF000000) >> 24) |
           ((val & 0x00FF0000) >>  8) |
           ((val & 0x0000FF00) <<  8) |
           ((val & 0x000000FF) << 24);
}

POSIX 호환 시스템 패밀리에서는 <endian.h> 헤더의 le32toh(), htole32()와 같은 시스템 내장 매크로를 제공한다. 따라서, 효율적인 GCS 개발 시에는 컴파일러에 내장된 __builtin_bswap32 단위 지시어를 우선적으로 사용하되, 이를 지원하지 않는 컴파일러일 경우 위와 같은 시프트 연산 기반의 폴백(Fallback) 해법을 사용토록 설계해야 한다. MAVLink C Library 내의 mavlink_types.h 파일은 플랫폼 감지(Platform Detection)를 통해 이런 스와핑 로직을 동적으로 매핑한다.

4. 구조체 패딩(Padding)과 비트 필드 마스킹(Bit-field Masking)

이기종 환경 간 데이터 직렬화 시 맞이하는 또 다른 핵심 위험 요소는 컴파일러에 따른 구조체 패딩(Struct Padding) 및 **C언어 비트 필드(Bit-fields)**의 모호한 동작이다.

4.1 컴파일러 비트 필드(C Bit-field)의 위험성

C/C++의 struct에서 제공하는 비트 필드(예: uint8_t flag : 3;)는 선언과 사용이 간편하지만, 메모리에 바이트가 쌓이는 방향성을 C 표준(ISO/IEC 9899)이 엄격하게 규정하지 않는다. 따라서, gcc는 비트 필드를 LSB부터 팩킹(Packing)하는 반면, MSVC 혹은 일부 임베디드 컴파일러는 MSB부터 채워 넣을 가능성이 존재하며, 패딩(Padding) 규칙 또한 다르다.

이러한 이분화 현상을 방지하고자 기체와 GCS가 소통하는 직렬화 페이로드 구조체 내에서는 비트 필드(: 연산자) 사용이 엄격히 금지되며, 순수하게 1바이트 및 다중 바이트 형식(uint8_t, uint16_t 등)만으로 팩킹(__attribute__((packed))) 선언을 하여 관리한다.

4.2 논리 연산을 통한 비트 필드 마스킹 처리 (Bit-field Masking)

구조체 레벨에서 비트 필드 선언을 회피하는 대신, 단일 정수형 변수 내에 여러 옵션과 플래그를 저장한 후 수동으로 마스킹 로직(Bitwise AND &, OR |)을 구현하는 ’비트 필드 마스킹’을 사용하라.

MAVLink 메시지의 HEARTBEAT 내 base_mode 혹은 custom_mode 필드의 플래그 해제 기법이 이를 활용하는 가장 대표적인 사례이다.

// 비트 위치(Bit offset) 및 마스크(Mask) 정의 (예: 시스템 상태 플래그)
#define FLAG_ARMED_MASK     (0x01 << 0) // 0b00000001 (1)
#define FLAG_MANUAL_MASK    (0x01 << 1) // 0b00000010 (2)
#define FLAG_HIL_MASK       (0x01 << 2) // 0b00000100 (4)

uint8_t system_status = 0; // 플래그를 보관할 단일 변수

// 1. 상태 설정 (Set)
system_status |= FLAG_ARMED_MASK;    // Armed 상태로 설정
system_status |= FLAG_MANUAL_MASK;   // 매뉴얼 모드 동시 활성화

// 2. 상태 해제 (Clear)
system_status &= ~FLAG_MANUAL_MASK;  // 매뉴얼 모드 해제

// 3. 상태 검사 (Check)
if ((system_status & FLAG_ARMED_MASK) != 0) {
    // 기체가 Arming 된 상태임을 확인 (비트 마스킹으로 안전한 추출)
    trigger_armed_ui_event();
}

이와 같은 비트 마스킹 체계는 엔디안과 무관하게 바이트 배열 내 레지스터의 수학적 논리 연산자로 성립되므로, 플랫폼 독립성을 강력하게 보장하며 시프팅(<<, >>)시 발생하는 컴파일러 종속적 패딩을 완전 배제한다.

5. 결론 및 GCS-Autopilot 연동 시의 공학적 지침

고도로 신뢰성 있는 자체 GCS를 개발하고자 할 때, 직렬화 계층(Serialization Layer) 설계에서의 불명확한 포인터 캐스팅(*(float*)byte_array) 및 C언어 비트 필드 구조체의 사용은 소프트웨어 취약성(치명적 NaN 발생 혹은 시스템 폴트)의 가장 큰 주범이 된다.

따라서 이기종간 아키텍처를 교차 연결하는 프로토콜 파서는, 다음 3가지의 엄격한 원칙을 준수하여 작성해야 한다.

  1. 기본 직렬화 스키마(Schema)는 리틀 엔디안으로 가정하되, 수신 플랫폼의 엔디안 특성을 매크로 혹은 동적 탐지로 파악하여 명시적 bswap을 거칠 것.
  2. 모든 통신 구조체는 #pragma pack(1) 혹은 __attribute__((packed)) 구문을 이용하여 컴파일러 패딩 삽입을 무효화 시킬 것.
  3. 여러 개의 True/False 상태 플래그들은 1바이트의 단일 변수에 통합하되 명시적인 비트 마스킹(&, |, ~)과 매크로 정의를 사용하여 값의 추출 및 수정을 수행할 것.

이는 MAVLink 라이브러리가 기본적으로 C 레벨에서 보장해 주지만, 독자 개발 과정 중 Java, C#, Python 등 강타입 언어나 구조체 직렬화 룰이 다른 생태계로 바인딩(Binding) 시 각 언어의 직렬화 모듈(예: Python의 struct.unpack("<f", data))에 리틀 엔디안 심볼(<)을 필수 기입함과 맥락을 같이 한다.
