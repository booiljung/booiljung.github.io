59.2.1 프레젠테이션-비즈니스 로직-데이터 접근 계층의 완전한 결합도 분리(Decoupling) 모델링

1. 개요

현대화된 지상 관제 시스템(GCS, Ground Control Station)은 초당 수백 개의 텔레메트리 패킷을 렌더링하고, 조종자의 즉각적인 지휘 통제 명령(Command)을 MAVLink로 전송하며, 방대한 지리 공간(GIS) 데이타베이스 및 비행 로그(Log)를 로컬 시스템에 동기적으로 기록해야 한다. 성격이 전혀 다른 이 세 가지 워크로드가 소이어(Coupling)된 채로 한 데 얽혀 구조화될 경우, 데이터 접근 시 발생하는 미세한 지연(Latency) 현상이 화면 주사율 한계(UI Freezing)나 조종 명령 누락이라는 치명적인 오작동으로 직결된다.

엔터프라이즈급 GCS의 코어 아키텍처를 설계할 때는 철저한 소프트웨어 공학적 분리 원칙(Separation of Concerns)을 바탕으로 프레젠테이션 계층, 비즈니스 로직 계층, 데이터 접근 계층 간의 완전한 결합도 분리(Decoupling) 모델링을 구성해야 한다.

2. 3계층(3-Tier) 아키텍처의 GCS 도메인 모델링

각 계층은 인접한 계층과만 추상화된 인터페이스(Interface)를 통해 소통하며, 구체적인 내부 구현 로직을 은닉한다.

2.1 데이터 접근 계층 (Data Access Layer)

역할 및 특징:
지속성(Persistence)을 갖는 데이터를 다루는 최하위 계층이다. 오프라인 지형(Terrain) 데이터베이스 캐싱 작업, 임무(Mission) Waypoint 파일 스토리지 I/O, ULog 기반 데이터 스트리밍 기록 및 설정 파라미터(Parameter)의 영구 저장을 담당한다.
PX4-Autopilot 기반의 임무 운용에서는 대용량 맵 데이터를 다루거나, 비행 후 데이터 사후 분석을 위해 수십 GB에 달하는 시계열 데이터가 쿼리 될 수 있다. 이 계층의 함수형 및 프로시저 호출은 철저히 비동기 I/O 함수로 감싸여져, 상위 비즈니스 로직 계층의 메인 이벤트 루프를 블로킹하지 않아야 한다.

2.2 비즈니스 로직 계층 (Business Logic Layer)

역할 및 특징:
관제 애플리케이션의 뇌(Brain) 역할을 담당하는 가장 핵심적인 도메인 계층이다. QGroundControl에서의 Vehicle 클래스 로직 그룹이 이 영역에 해당하며, PX4-Autopilot의 “가상 기체(Shadow Vehicle)” 속성을 메모리에 모델링하여 유지한다.

  • MAVLink 파싱 및 인코딩
  • 안전망(Failsafe) 규칙 평가 엔진 (예: MAVLink 연결 끊김 시 팝업 경고 발생 조건 판단 로직)
  • 비행 모드(Flight Mode) 유효성 검증
    이러한 핵심 로직은 UI의 존재 여부와 무관하게 메모리상에서 독자적이고 연속적으로 동작해야 한다. 이 계층은 데이터 접근 계층에 인터페이스로 쿼리를 요청하고, 프레젠테이션 계층에는 시스템 상태 변경 이벤트를 발행(Publish)하는 통신 허브 역할을 병행한다.

2.3 프레젠테이션 계층 (Presentation Layer)

역할 및 특징:
사용자 인터페이스(UI)를 화면에 렌더링하고 조종자의 터치 및 마우스 명령을 캡처하는 최상위 계층이다.
HUD(Head-Up Display), 인공 수평의(Artificial Horizon), 계기판(Instruments Dashboard), 맵 위젯 등으로 구성된다. 핵심 분리 원칙에 따라 “프레젠테이션 계층은 스스로 MAVLink 소켓을 절대 생성하지 않는다“라는 절대적인 설계 룰이 적용된다.
모든 UI 요소는 비즈니스 로직 계층이 메모리에 갱신해둔 상태 값만을 읽기 전용형(Read-only) 데이터 바인딩 혹은 관찰자 모델(Observer Pattern)을 통해 가져와 초당 60 프레임(FPS) 이상으로 부드럽게 화면을 그리는 데만 컴퓨팅 자원을 투입한다.

3. 계층 분리 아키텍처 다이어그램 및 흐름

UML 다이어그램을 통해 GCS 시스템이 의존성 역전 원칙(DIP, Dependency Inversion Principle) 하에 어떻게 계층적 분리를 이룩하는지 살펴본다.

4. 완전한 결합도 분리 모델링의 검증적 이점

4.1 텔레메트리 스톰(Telemetry Storm) 면역력 체계

PX4 기반의 기체가 밀집되거나(Swarm 환경), 파라미터 동기화(Parameter Syncing) 시, GCS 네트워크 브릿지로는 초당 수천 개의 메시지 트래픽 폭풍이 들이닥칠 수 있다. 이때 프레젠테이션 계층-비즈니스 로직 계층 간의 철저한 관심사 분리가 이루어져 있다면, 백엔드 비즈니스 로직은 메모리에 들어오는 속성값만 최고 속도로 치환(Overwriting)하고, 프레젠테이션 계층은 본인의 렌더링 틱(Tick)에 맞추어 마지막 값만을 가져가 화면에 그리기 때문에, 화면 멈춤 현상을 근본적으로 차단할 수 있다.

4.2 플랫폼 종속성 탈피 및 테스트의 용이성 달성

비즈니스 로직 계층은 UI 프레임워크 전용 라이브러리(예: import QtQuick, import UnityEngine)를 전혀 가져다 쓰지 않은 순수한 네이티브 코드(Native Code) 구조여야 한다. 이렇게 구현함으로써 UI 엔진 없이도 GCS의 핵심 동작(Headless Mode)을 구동시킬 수 있다.
결과적으로 가상 기체 클래스와 MAVLink 라우팅 모듈 전체를 대상으로 하는 CI/CD 환경에서의 무작위 결함 주입(Fault Injection) 및 자동화 단위 테스트(Unit Testing) 구축이 극적으로 용이해진다.

5. 결론

GCS 아키텍처의 프레젠테이션, 비즈니스 로직, 데이터 접근 계층의 완전한 결합도 분리(Decoupling)는 선택적인 리팩터링 작업이 아니라 시스템 신뢰성을 보장하는 핵심 토대이다. 개발자는 상위 계층이 하위 계층에 직접 의존하지 않고 오직 약속된 인터페이스(Interface)와 이벤트(Event)에만 의존하도록 시스템을 훈련시켜야 한다. 이와 같이 수립된 견고한 3계층 아키텍처는 코드의 모듈화를 실현하고, 궁극적으로 새로운 PX4 제어 알고리즘이나 차세대 임무 체계가 도입되었을 때 UI와 통신부의 수정을 최소화하여 유지보수 비용을 획기적으로 낮추는 근간이 된다.
