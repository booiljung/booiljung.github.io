59.10.1. 무선 데이터 링크 침투(Spoofing/Replay Attack) 방어를 위한 고급 암호화 표준-갈루아 카운터 모드(AES-GCM) 대칭키 암호화 및 타원 곡선 디피-헬만(ECDH) 키 교환 메커니즘

1. 개요

지상 관제 시스템(Ground Control Station, GCS)과 무인항공기(UAV)를 연결하는 데이터 링크는 433MHz, 900MHz 등 개방형 주파수를 주로 사용한다. 악의적인 공격자는 전파 도청 및 분석을 통해 제어 신호를 탈취하고, 기체에 조작된 명령을 주입하는 **스푸핑(Spoofing)**이나, 기존의 명령 패킷을 캡처해 두었다가 나중에 그대로 다시 송출하여 시스템을 붕괴시키는 **재전송 공격(Replay Attack)**을 시도할 수 있다.

기본적인 MAVLink 2.0 프로토콜 자체는 메시지 서명(Message Signing)을 제공하여 비인가 제어자의 접근을 차단하고 재전송 공격을 막아낼 수 있으나, 원격 측정 로그가 평문으로 노출되는 기밀성(Confidentiality) 문제까지 해결해 주지는 않는다. 이를 완벽히 방어하기 위해서는 GCS와 기체 간 터널링(Tunneling) 계층에서 **AES-GCM(Advanced Encryption Standard - Galois/Counter Mode)**을 활용한 고속 인증 암호화(Authenticated Encryption) 데이터 링크와, 세션 성립 시 안전한 비밀키 수립을 위한 타원 곡선 디피-헬만(Elliptic Curve Diffie-Hellman, ECDH) 키 교환의 시스템 공학적 융합이 요구된다.

기준 시스템 버전 정보

  • PX4-Autopilot: v1.14.x
  • QGroundControl: v4.3.x 이상

2. 보안 취약점과 암호학적 공격 벡터 분석

  • 재전송 공격(Replay Attack): 공격자가 합법적인 GCS에서 무인기로 전송된 “이륙(Takeoff)” 또는 “무장 해제(Disarm)” 패킷을 중간에서 녹음(Record)한 후, 시간이 지난 뒤 임의로 전송해 드론 행위를 강제하는 방식.
  • 스푸핑(Spoofing): GCS의 MAC 주소 또는 Node ID를 복제(Clone)하여, 스스로 합법적인 관제 노드인 것처럼 위장하고 악성 웨이포인트(Waypoint)를 주입하는 기법.

이러한 공격을 차단하려면 패킷마다 **고유한 순서 번호(Sequence Number)**나 **타임스탬프(Timestamp)**가 논스(Nonce)로 암호화 시스템에 편입되어야 한다.

3. AES-GCM 대칭키 인증 암호화 메커니즘

3.1 갈루아 카운터 모드(GCM)의 시스템 공학적 특성

일반적인 블록 암호화(예: AES-CBC)는 기밀성은 보장하지만 무결성(Integrity) 검증을 위해서는 추가적인 HMAC 연산이 필요해 오버헤드가 발생한다. 이에 반해 AES-GCM은 기밀성과 무결성, 그리고 인증(Authentication)을 한 번의 연산으로 동시에 제공(AEAD: Authenticated Encryption with Associated Data)하는 알고리즘이다.

  • 카운터(Counter) 모드 블록 암호화: 블록 단위가 아닌 스트림 암호처럼 동작하여 실시간 드론 Telemetry와 같이 패킷 길이가 유동적이고 잦은 데이터 송신에 매우 병렬적(Parallelizable)이고 빠르다.
  • 갈루아 해시 함수(GHASH) 인증: 암호화된 데이터의 끝부분에 생성되는 인증 태그(Authentication Tag)를 거쳐, 평문이 1비트라도 위조될 경우 칩셋 레벨에서 무결성 검증을 즉각 취소(Fail) 처리한다.

3.2 논스(Nonce) 관리를 통한 재전송 공격 방어

AES-GCM에서 사용하는 논스(Nonce, 보통 96비트)는 절대 재사용되어서는 안 된다. 드론 보안 모듈은 논스를 구성할 때 [기체 고유 ID + MAVLink 패킷 시퀀스 번호 + 타임스탬프]의 조합을 사용한다.
공격자가 옛날 패킷을 탈취해 전송하더라도, GCS나 드론의 파서 엔진이 이전 시퀀스나 옛 타임스탬프로 유도된 논스임을 감지하여 AES-GCM 복호화 단계에서 태그 불일치(Tag Mismatch) 오류를 발생시키며 즉시 드롭(Drop) 시킨다.

4. 타원 곡선 디피-헬만(ECDH) 키 교환

안전한 대칭키(AES-GCM Key)를 만들기 위해, 무장(Arming) 전 GCS와 PX4 간 초기 핸드셰이크(Handshake) 과정에서 ECDH 기반 비대칭 키 교환이 이루어진다.

4.1 ECDH 수학적 통신 원리

타원 곡선 암호학(Elliptic Curve Cryptography, ECC)은 짧은 키 길이(예: 256비트)만으로도 RSA 3072비트와 동등한 보안성을 제공하여 마이크로컨트롤러(NuttX 하드웨어)의 배터리 및 연산 부하를 최소화한다.

  1. 매개변수 준비: 공통의 타원 곡선 매개변수 집합(예: NIST P-256 또는 secp256r1)을 GCS와 드론 모두 사전 정의.
  2. 비밀키-공개키 생성:
  • GCS는 무작위 개인 비밀키 d_G를 선택하고 공개키 Q_G = d_G \times G 전송.
  • 드론은 무작위 개인 비밀키 d_D를 선택하고 공개키 Q_D = d_D \times G 전송.
  1. 공통 비밀키(Shared Secret) 연산:
  • GCS: S = d_G \times Q_D
  • 드론: S = d_D \times Q_G
  • 수학적 산술에 의해 양 플랫폼이 동일한 비밀키 S 를 공유하게 되며, 이 공유키 S 가 다시 KDF(Key Derivation Function)를 거쳐 최종 AES-GCM 256비트 대칭키로 파생된다.

5. 설계 적용 및 QGroundControl 확장성

QGroundControl 내지 커스텀 GCS의 네이티브 C++ 코어(src/CommInfo)에서 ECDH 핸드셰이크 모듈 오픈소스(예: OpenSSL 라이브러리)를 연동시켜, 시리얼 링크 또는 TCP/UDP 포맷팅 전·후 처리에 훅(Hook)을 거는 설계 패턴이 가장 합리적이다.

PX4-Autopilot의 경우 Micro XRCE-DDS(ROS 2 연동)나 동반 컴퓨터(Companion Computer, 예: Raspberry Pi)와 연계될 시, 동반 컴퓨터 레벨의 VPN(예: WireGuard 기반 암호화)을 통해 AES-GCM 암호화 스택을 위임(Offloading)하면, Pixhawk(비행 제어기)의 CPU 부하를 분산시키고 강력한 제로 트러스트(Zero-Trust) 보안 망을 실현할 수 있다.

6. 결론

고급 무인기 통신 환경에서 공격을 방어하려면 패킷 서명(Signing)만으로는 불충분하다. ECDH를 통한 동적 키 분배 시스템은 암호화 키를 안전하게 초기화하며, AES-GCM 암호화 엔진은 최저 지연 시간(Low Latency)으로 실시간 텔레메트리와 제어 명령의 기밀성, 무결성, 그리고 재전송 오프 방어 메커니즘을 견고하게 지원한다. 시스템 공학자와 보안 아키텍트는 독자 GCS 설계 시 이 하이브리드 암호화 모델을 통신 계층 최하단에 반드시 내재화해야 한다.
