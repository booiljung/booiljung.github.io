59.3.1 OSI 참조 모델 기반의 GCS-Autopilot 통신 스택 토폴로지 매핑 및 추상화

지상 관제 시스템(GCS)과 PX4-Autopilot은 단순한 직렬 데이터 케이블(Serial Data Cable)로만 연결되는 것이 아니다. 복잡한 유무선 IP 네트워크 라우팅, 위성 통신망(SATCOM), 장거리 커스텀 무선 RF 망(예: RFD900 텔레메트리 라디오) 등 수많은 물리적 스펙 환경 속에서 제어 명령과 방대한 텔레메트리(Telemetry)를 흔들림 없이 교환해야 한다. 이처럼 급변하는 다매체 통신 환경 하에서 높은 확장성과 무중단(Seamless) 실시간 신뢰성을 보장하는 통신 프로토콜 엔진을 개발하려면, 국제 표준인 OSI 7계층 참조 모델(Open Systems Interconnection Reference Model)을 기반으로 GCS 통신 아키텍처를 추상화(Abstraction)하고 모듈화하는 시스템 공학적 설계 과정이 필수적이다.

본 절에서는 PX4-Autopilot의 핵심인 MAVLink 프로토콜 생태계가 OSI 7계층 토폴로지상에 어떻게 맵핑되는지 분석하고, 이를 통해 자체 GCS 소프트웨어를 설계할 때 모듈 간 결합도(Coupling)를 낮추어 유지보수성과 유연성을 극대화하는 통신 스택 설계 방법론에 대해 상세히 살펴본다.

1. GCS-Autopilot 통신 환경의 OSI 7계층(L1~L7) 매핑 분석

1.1 물리 계층 (L1) 및 데이터 링크 계층 (L2)

GCS와 기체, 가장 밑단의 통신 하드웨어 제어를 담당하는 계층이다.

  • 물리 계층(L1, Physical Layer): 무인기 내부의 UART/USART 직렬 와이어 핀, 외부 900MHz/2.4GHz RF 안테나의 전자기파 전송 매체, Wi-Fi(802.11) 등 전기적/광학적 물리 매체가 이 계층에 속한다.
  • 데이터 링크 계층(L2, Data Link Layer): 직렬 통신의 경우 하드웨어 오류를 정정하는 비트스트림 기반 프레이밍(Framing) 및 흐름 제어(Flow Control), 그리고 이더넷의 경우 802.3 MAC 프레임 구조와 스위칭 로직이 이 계층에 속한다. GCS 엔진 설계 시에는 OS가 관장하는 디바이스 드라이버나 소켓 API를 감싸는 파일 디스크립터(File Descriptor) 레벨의 추상화 인터페이스를 구현하게 된다.

1.2 네트워크 계층 (L3) 및 전송 계층 (L4)

노드에서 다른 노드로의 논리적 패킷 전달과 신뢰성을 보장하는 핵심 인프라 계층이다.

  • 네트워크 라우팅 계층(L3, Network Layer): IPv4/IPv6 IP 주소를 기반으로 한 라우팅과 함께, MAVLink 프로토콜 자체가 내장하고 있는 System ID(SysID)와 Component ID(CompID) 필드가 서브네트워크 계층의 라우터 역할을 수행한다. 즉 MAVLink 스택 자체가 드론 내부 Companion Computer, 짐벌, 스웜(Swarm) 환경 하에서 L3 스위칭 허브(Hub)의 기능을 독자적으로 수행한다.
  • 전송 계층(L4, Transport Layer): IP 데이터 무결성을 보존하고 통신 포트(Port) 역할을 담당한다. MAVLink 데이터 패킷은 일반적으로 TCP 또는 UDP 위에 캡슐화(Encapsulate)되어 전송되며, 고빈도 텔레메트리 스트림 전송 시에는 재전송 오버헤드가 적은 UDP 브로드캐스트/멀티캐스트 기법이 전체 병목을 줄이기 때문에 GCS에서 가장 선호된다. 한편 대용량 미션 로그 파일 등 FTP 전송이 필요한 경우에는 TCP/IP 스택이 활용된다.

1.3 세션 계층(L5), 표현 계층(L6) 및 응용 계층 (L7)

GCS 소프트웨어의 비즈니스 로직과 데이터 검증이 집중되는 최상단 계층이다.

  • 세션(L5) 및 표현 계층(L6): MAVLink v2 프레임의 헤더 암호화 규격(Message Signing, AES 기반 256비트 서명 적용), 그리고 통신 커넥션 단절을 모니터링하기 위한 Heartbeat 메시지 타이머. 나아가 네트워크 유실 방지를 위한 패킷 시퀀스 번호(Packet Sequence Number) 관리 로직이 이곳에 구성된다.
  • 응용 계층(L7, Application Layer): 실질적인 관제 UI 비즈니스 로직 도메인이다. FTP(File Transfer Protocol) 파일 송수신, 마이크로 서비스 단위의 MAVLink 파라미터(Parameter) 동기화, 사용자 정의 웨이포인트(Waypoints) 궤적 업로드 인프라, 카메라/페이로드(Gimbal) 제어 명령(COMMAND_LONG) 프로토콜 모델이 위치한다. 조종자가 GCS 화면에서 터치한 시각적 UI 행동은 이 L7 응용 API 함수를 호출함으로써 구동된다.

2. GCS 통신 스택 추상화(Abstraction) 아키텍처 설계 지침

자체 GCS 아키텍처를 견고하게 설계하기 위해서는 반드시 위에서 분석한 OSI 모델 계층의 책임을 바탕으로, 소스 코드 내 패키지(Package) 및 C++ 클래스 단위를 엄격하게 분리(Decoupling)해야 한다. 최악의 안티 패턴(Anti-Pattern)은 통신 매체를 담당하는 시리얼 드라이버 매니저 객체가 내부적으로 비행 상태를 나타내는 MAV_STATE까지 동시에 다루도록 코드를 혼재시키는 God Object 패턴이다. 이러한 구조는 추후 통신 매체가 LTE 모뎀(L4 소켓 기반)으로 바뀌는 하드웨어 업그레이드 시 전체 GCS 앱을 전부 재작성(Rewrite)해야 하는 부채로 돌아온다. 올바른 인터페이스 추상화 전략은 아래와 같다.

  1. 물리 네트워크 은닉 (ICommunicationLink 인터페이스 (L1~L4))
  • 시스템 전역에 SerialLink, UdpLink, TcpLink 와 같은 구체 클래스들을 ICommunicationLink 단일 다형성 부모 인터페이스로 묶어 제공한다.
  • 이로 인해 상위 미들웨어는 매체가 무엇으로 변하든 간섭 없이 readBytes() 및 writeBytes() 가상 함수와 연결 단절을 통보하는 linkStateChanged 이벤트 신호(Signal)에만 의존하며 통신할 수 있다.
  1. MAVLink 파이프라인 은닉 (MavlinkProtocolHandler 인터페이스 (L5~L6))
  • L1~L4에서 단순히 0100101 바이트 단위로 쏟아져 들어온 버퍼를 인계받아 mavlink_message_t 객체로 디코딩하고 프레임을 추출하는 전담 객체를 구성한다.
  • 메시지 위변조, CRC 오류, 중복 수신 패킷 드롭과 같은 악의적, 오류성 예외는 오직 본 핸들러 층에서만 차단 및 필터링되어, 상위 L7 계층으로 쓰레기 데이터가 올라가지 않도록 방호벽을 세운다.
  1. 응용 비즈니스 로직 격리 (Vehicle Manager (L7))
  • 디코딩과 검증이 끝난 무결점 MAVLink 통신 메시지를 발행-구독(Pub/Sub) 버스 패턴을 거쳐 넘겨받는다.
  • 여기서는 오로지 현재 드론의 배터리 퍼센트가 몇 프로인지, 고도는 몇 미터인지, UI 렌더링 계기판 뷰에 어떻게 상태값을 동기화할지에만 집중한다. 통신 연결이 Wi-Fi인지 시리얼인지 등에 대한 하위 정보는 일절 모르는 상태가 되며 모듈 간 완벽한 독립성이 구축된다.

이러한 OSI 기반 관심사 분리(Separation of Concerns, SoC) 아키텍처는 PX4 펌웨어 버전의 메이저 업데이트나 관제 부대의 통신 하드웨어 장비(Data Link)의 대규모 업그레이드시에도, 복잡도 높은 UI 코드 수정 및 버그 없이 ‘하위 물리 링크 플러그인 컴포넌트 추가’ 하나만으로 신속한 통합 및 유연한 대응을 보장한다.
