59.10.3 프로세스 교착 상태(Deadlock) 및 스레드 기아(Starvation) 감지를 위한 하드웨어 독립적 소프트웨어 워치독 타이머(Watchdog Timer) 인터럽트 서비스 루틴 설계

복잡한 멀티스레드(Multi-thread) 환경으로 구동되는 지상 관제 시스템(GCS) 및 PX4-Autopilot의 실시간 운영체제(RTOS) 환경에서는 예기치 못한 동시성(Concurrency) 오류로 인해 시스템이 응답 불가 상태에 빠질 위험이 존재한다. 대표적인 현상으로 스레드 간 자원 점유 대기로 인해 실행이 영구적으로 멈추는 **교착 상태(Deadlock)**와, 낮은 우선순위의 프로세스가 CPU 할당을 받지 못해 지연되는 **기아 상태(Starvation)**가 있다. 이를 감지하고 복구하기 위해, 특정 마이크로컨트롤러 아키텍처(MCU)에 종속되지 않는 하드웨어 독립적 소프트웨어 워치독 타이머(Software Watchdog Timer, SWDT) 메커니즘 설계가 필수적이다.

1. 교착 상태와 스레드 기아 현상의 메커니즘

비행 제어 소프트웨어나 대규모 GCS는 통신 스레드, UI 렌더링 스레드, 데이터베이스 기록 스레드, 제어 알고리즘 스레드 등 수많은 태스크가 병렬로 실행된다.

  • 교착 상태(Deadlock): 두 개 이상의 스레드가 서로가 점유하고 있는 자원(Mutex, Semaphore 등)을 무한정 기다리는 상태다. 예를 들어, GCS의 MAVLink 파싱 스레드가 Mutex A를 점유하고 Mutex B를 대기하는 동안, UI 스레드가 Mutex B를 점유하고 Mutex A를 대기하면 양쪽 모두 영원히 블록(Block)된다.
  • 스레드 기아(Starvation): 높은 우선순위 태스크(예: PX4의 rate_ctrl 스레드나 GCS의 타이머 콜백)가 CPU 자원을 독점할 경우, 상대적으로 낮은 우선순위의 태스크(예: SD 카드 로깅 태스크)가 영구적으로 실행 사이클을 할당받지 못하여 시스템 전반의 메시지 큐(Queue) 폭실 현상을 유발하는 상태다.

2. 하드웨어 독립적 소프트웨어 워치독 설계

순수 하드웨어 워치독(Hardware WDT)은 지정된 시간 내에 카운터가 초기화(Kick 또는 Feed)되지 않으면 시스템 전원을 물리적으로 리셋하는 원시적인 구조를 제한적으로 지닌다. 그러나 시스템이 즉각 재부팅될 경우 GCS 조종 권한 일시 상실, 비행 중인 기체의 예기치 못한 동역학적 불안정 및 오류 추적 컨텍스트(Context) 유실 등의 부작용이 크다.

소프트웨어 워치독은 운영체제 수준의 타이머 인터럽트(Timer Interrupt) 또는 최고 우선순위의 감시 전용 독립 스레드 형태로 구동되며, 세부 스레드별 프로세스 상태를 정밀하게 모니터링한다. 특정 제조사의 하드웨어 칩 레지스터 제어에 얽매이지 않고 (하드웨어 독립성), POSIX 스레드 인터페이스(pthread)나 NuttX RTOS의 API를 통해 유연하게 이식될 수 있도록 설계된다.

2.1 SWDT 구조적 설계 원리

핵심 로직 아키텍처는 다음과 같다.

  1. 등록(Registration): 감시대상(Critical Threads)이 워치독 관리자 내부의 감시 리스트에 자신을 등록한다.
  2. 보고(Heartbeat/Feed): 각 스레드는 정상적으로 자신이 맡은 루프 스케줄을 순회할 때마다 공유 메모리(Shared Memory) 상의 타임스탬프(Last Activity Time)를 갱신한다.
  3. 감시(Monitor): 최고 우선순위를 부여받은 SWDT 인터럽트 서비스 루틴(ISR) 또는 스레드가 설정된 주기(예: 100ms)마다 기상하여 등록된 모든 스레드의 타임스탬프를 횡단 검사한다.
  4. 대응(Action): 특정 스레드의 갱신 시점이 임계치(Timeout Threshold)를 초과하면 해당 스레드가 교착 혹은 기아 상태에 빠졌다고 수학적으로 판단하고, 예외 처리 파이프라인(Exception Handling)을 발동시킨다.

3. PX4-Autopilot 및 GCS 관점의 SWDT 로직 구현 비교

3.1 PX4-Autopilot 비행 제어기 적용 로직

PX4 펌웨어(NuttX RTOS 기반) 내부에는 px4_work_queue 및 내부 모듈 마이크로아키텍처 스케줄링이 활발하게 일어난다. PX4는 워치독 관리 메커니즘을 두루 활용하여 주요 백그라운드 태스크(센서 버스 폴링, 자세 제어기 역학 연산 등)가 데드락에 빠지지 않았는지 지속해서 감리한다. FC(비행 제어기) 임베디드 환경에서는 동적 메모리 복구에 제약이 많으므로, SWDT는 복잡한 스레드 살리기보다 페일세이프(Failsafe: RTL 또는 고도 낙하산 자동 전개) 즉각 진입을 명령 통제 시스템으로 이관하여 기체 무결성을 보호하는 데 집중한다.

3.2 GCS (QGroundControl 및 사용자 정의 관제 소프트웨어) 적용 로직

통상 데스크톱이나 모바일 환경을 타겟팅하는 GCS (예: QGroundControl v4.3.x 이상)에서는 UI(사용자 인터페이스) 엔진의 주 이벤트 루프(Main Event Loop)가 블로킹되지 않게 담보하는 것이 최우선이다. 메인 코어(프론트엔드 UI 렌더링)와 MAVLink 메시지 파싱 스레드(백엔드 비동기 통신) 간 데이터 경합(Race Condition)으로 교착이 일어나면 화면이 일시 정지되어 치명적인 수동 조종 상실 위기를 초래한다.

  • 방어 설계 기법 방안: 백그라운드에 WatchdogQThread 등 관찰자 스레드를 독자 생성하여, 메인 이벤트 루프에 더미 비동기 콜백 이벤트를 지속적으로 핑(Ping) 테스트한다. 응답 타이머가 오버플로우되면 백그라운드 관찰자는 메인 스레드에 데드락이 발생한 것으로 정밀 판정하고, 시스템 콜 수준에서 프로세스를 클린업하거나 미니덤프(Crash Dump)를 하드디스크에 남기도록 리다이렉션한다.

4. 인터럽트 서비스 루틴(ISR) 레벨 설계 시 주의 및 제약 사항

SWDT가 하드웨어 타이머 혹은 OS 타이머 인터럽트(ISR; Interrupt Service Routine) 레벨에서 실행되도록 아키텍처를 잡을 경우, 운영체제의 컨텍스트 스위칭 지연을 완벽히 피할 수 있다는 장점이 있으나, 설계 시 심각한 공학적 제약을 반드시 준수해야 한다.

  • 논-블로킹(Non-Blocking) 실행 원칙 준수: ISR 내부에서는 파일 I/O 시스템 호출(예: 장치에 대한 printf 출력), 과도한 메모리 할당(malloc), 또는 상호배제 잠금(Mutex Acquisition) 획득 대기와 같은 블로킹(Blocking) 동작을 행해서는 절대 안 된다. ISR이 블로킹되는 순간, 시스템 전체 커널의 인터럽트 응답 구조가 마비된다.
  • 원자적 변수 접근(Atomic Memory Access): 검사자와 작업 대상 스레드가 공유 테이블 변수를 수시로 수정/참조하므로 메모리 가시성 및 동기화 무결성 난제가 유발된다. C++ 기준 std::atomic 구문 등을 배제 없이 활용, 하드웨어 명령어 수준의 원자적 연산 규약을 적용하여 데이터 읽기/쓰기 동시 충돌(Data Race/Tearing)을 구조적으로 봉쇄해야 한다.

5. 결론

하드웨어 독립적 소프트웨어 워치독 타이머 감지 시스템 아키텍처의 설계는 무인항공기 소프트웨어와 GCS 관제 프로그램의 고가용성(High Availability) 및 미션 완수 신뢰성을 판가름하는 방패다. 단순히 하드웨어 보드의 전력을 강제로 단락시켜 리셋하는 원시적 방어 매뉴얼에서 벗어나, 시스템 스레드 각각의 논리 체계적 결함(Deadlock, Starvation 현상)을 정밀 진단하고 자가 복구하며 그 증거 로그를 채증하는 지능적인 결함 허용 구조(Fault Tolerance) 공학은 최신 모던 플라이트 컨트롤 인프라스트럭처의 필수 요건이라 하겠다.
