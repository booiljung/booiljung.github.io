59.8.4 이기종 타겟 컴파일을 위한 CMake 기반 빌드 시스템 구성 및 MISRA-C++ 코딩 가이드라인 기반 정적 분석(Static Code Analysis) CI/CD 파이프라인 통합

1. 개요

지상 관제 시스템(GCS; Ground Control Station)은 안드로이드(Android), iOS 모바일 기기부터 윈도우(Windows), 리눅스(Linux), 맥OS(macOS) 데스크톱 환경에 이르기까지 매우 다양한 이기종 플랫폼(Heterogeneous Platforms)에서 실행되어야 한다. 이를 소스 코드 하나로 대응하기 위해서는 크로스 플랫폼 빌드 도구인 CMake를 활용한 체계적인 빌드 시스템 구성이 필수적이다.
또한, 무인항공기 관제 소프트웨어는 생명과 재산에 직접적인 영향을 미치는 안전 임계 시스템(Safety-Critical System)이다. 따라서 기능 구현뿐만 아니라 C++ 언어의 잠재적 오류를 예방하는 MISRA-C++(Motor Industry Software Reliability Association) 규격을 준수하고, 이를 지속적으로 검증하는 정적 분석(Static Code Analysis) 기반의 CI/CD(Continuous Integration/Continuous Deployment) 파이프라인 통합이 요구된다.

이기종 타겟 빌드와 코드 품질 검증을 동시에 자동화하기 위한 전체 파이프라인은 다음과 같다.

3. 이기종 타겟을 위한 CMake 빌드 시스템 구성

크로스 플랫폼 빌드를 성공적으로 수행하기 위해서는, 컴파일러, 링커, 시스템 루트(sysroot)의 경로를 정의하는 툴체인(Toolchain) 파일과 CMake의 플랫폼 판별 매크로를 활용해야 한다.

3.1 툴체인(Toolchain) 파일 활용

안드로이드나 iOS와 같은 모바일 환경으로 데스크톱 코드를 컴파일(크로스 컴파일)할 때는, 호스트(Host)의 기본 컴파일러가 아닌 타겟 기반의 컴파일러가 필요하다.

  • Android NDK: 안드로이드 빌드 시 NDK가 제공하는 android.toolchain.cmake를 로드하여 아키텍처(예: arm64-v8a)와 Android API 레벨을 지정한다.
  • iOS: 최신 Xcode 빌드 도구와 호환되는 오픈소스 ios.toolchain.cmake를 사용하여, 빌드 타겟을 시뮬레이터(Simulator) 또는 실제 기기(Device)로 분기하고 서명(Code Signing) 속성을 제어한다.

3.2 플랫폼 종속적(Platform-Dependent) 모듈 분리

네트워크 소켓, 파일 시스템, 스레드 초기화 등은 플랫폼마다 API가 다르다. CMake 수준에서 이를 우아하게 처리하기 위해 매크로 분기를 적용한다.

# CMakeLists.txt 예시
if(ANDROID)
    add_definitions(-DOS_ANDROID)
    target_link_libraries(MyGCSCore log android)
elseif(IOS)
    add_definitions(-DOS_IOS)
    # iOS 프레임워크 링킹
    find_library(FOUNDATION Foundation)
    target_link_libraries(MyGCSCore ${FOUNDATION})
elseif(WIN32)
    add_definitions(-DOS_WINDOWS)
    target_link_libraries(MyGCSCore wsock32 ws2_32)
endif()

이러한 구성은 MAVLink 프로토콜 파싱 코드(Core)를 하나로 유지하면서, 시리얼 포트나 블루투스(BLE) 같은 하드웨어 계층 제어 인터페이스만 플랫폼별 파일로 교체하여 컴파일할 수 있게 해준다.

4. MISRA-C++ 가이드라인 기반 정적 분석

C++은 매우 강력하지만 포인터 오용, 언바운드 배열 참조, 암시적 타입 변환 등 휴먼 에러를 유발하기 쉬운 한계를 갖는다. MISRA-C++ 가이드라인은 이러한 제약 사항을 규칙으로 정의한다. QGroundControl(v4.3)과 PX4-Autopilot(v1.14) 소스 코드 역시 이러한 코드 신뢰성 확보에 많은 노력을 기울이고 있다.

4.1 핵심 MISRA-C++ 규칙 적용 예시

  1. 동적 메모리 할당 제한: 런타임 중 new/delete나 malloc/free 사용을 최소화하고, 정적 메모리나 스마트 포인터를 사용하여 메모리 단편화 및 누수를 방지한다.
  2. 명시적 형 변환(Explicit Type Casting): 데이터 손실이 발생할 수 있는 암시적 캐스팅을 금지하며, static_cast 등을 명시적으로 사용하여 의도를 분명히 한다.
  3. 포인터 연산 금지: 배열의 인덱싱 이외에서의 무분별한 포인터 산술(Pointer Arithmetic)을 제한한다.

4.2 정적 분석 도구 연동 (Clang-Tidy & Cppcheck)

현대의 GCS 프로젝트들은 상용 도구인 SonarQube 외에도 오픈소스 도구를 널리 활용한다.

  • Clang-Tidy: clang-tidy는 컴파일러 프론트엔드 기반으로 동작하여 매우 정밀한 분석을 제공하며, .clang-tidy 설정 파일을 통해 특정 MISRA 범주(cppcoreguidelines-*, bugprone-*, cert-*)의 규칙을 강제할 수 있다. CMake에서 CXX_CLANG_TIDY 속성을 설정하면 빌드 과정에 정적 분석이 투명하게 통합된다.
  • Cppcheck: 수행 속도가 빠르고 널 포인터 역참조(Null Pointer Dereference) 검출에 특화되어 있어, CI 파이프라인의 가장 앞단에서 1차적인 오류 필터링 역할을 수행한다.

5. CI/CD (Continuous Integration & Delivery) 파이프라인 자동화

GitHub Actions를 활용한 GCS 자동화 파이프라인 설계는 코드의 무결성을 유지하면서 릴리스(Release) 주기를 단축하는 핵심 요소이다.

5.1 파이프라인 구성의 핵심 원칙

  1. Pull Request(PR) 게이트키핑: 개발자가 코드를 커밋하여 베이스 브랜치(Base Branch)로 병합을 시도할 때, CI 서버가 코드 포맷 컨벤션(예: clang-format) 검사와 정적 분석(Clang-Tidy)을 자동으로 실행한다. 오류나 경고가 검출되면 병합을 기술적으로 차단한다.
  2. 행렬 빌드(Matrix Build) 실행: PR 검증이 끝난 후에는 macOS, Ubuntu, Windows 환경의 러너(Runner)를 동시에 구동하여(Parallel Execution), 데스크톱 바이너리와 Android/iOS 패키지를 동시에 크로스 컴파일한다. 이를 통해 특정 OS에서 머리말(Header) 누락이나 컴파일러 차이로 인해 발생하는 오류(예: MSVC와 GCC의 C++ 표준 해석 차이)를 조기에 잡아낸다.
  3. 결과물 아카이빙(Artifact Archiving): 빌드 성공 시 .apk, .dmg, .exe 파일을 아티팩트(Artifact)로 보존하거나 OTA(Over-The-Air) 테스트 플랫폼으로 배포하여 즉각적인 검증이 가능하도록 한다.

6. 결론

이기종 타겟을 단일 코드베이스로 지원하는 관제 시스템은 CMake 툴체인의 정교한 제어를 통해 컴파일 시간과 복잡도를 획기적으로 줄일 수 있다. 아울러 MISRA-C++ 규격과 정적 오류 분석을 강제하는 CI/CD 자동화 파이프라인은 런타임 크래시나 치명적 오류가 허용되지 않는 무인기 도메인에서 최고 수준의 소프트웨어 신뢰성과 안전성(Safety)을 담보하는 필수적인 공학적 토대가 된다.
