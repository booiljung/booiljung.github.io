59.9.2 관계형 데이터베이스(SQLite) 인덱싱 트리(B-Tree) 구조 최적화 및 트랜잭션 저널링(Journaling)을 통한 전원 차단 시 데이터 무결성 보장

1. 개요

지상 관제 시스템(GCS)이 모바일 디바이스나 휴대용 콘솔에서 운용될 때 가장 주의해야 할 엣지 케이스(Edge Case) 중 하나는 디바이스의 갑작스러운 전원 차단(Power Loss)이나 운영체제에 의한 앱의 강제 종료(OOM Kill) 현상이다.
PX4-Autopilot의 고주파수 시계열(Time-series) 데이터가 로컬 관계형 데이터베이스(주로 SQLite)에 끊임없이 기록되고 있는 상황에서 이러한 중단이 발생하면, 데이터베이스 파일 캐시가 디스크에 완전히 기록(Flush)되지 않아 데이터베이스(Database) 전체가 복구 불가능한 식별(Corruption) 상태에 빠질 수 있다. 본 장에서는 B-Tree 기반의 SQLite 구조에서 성능 저하를 최소화하면서도 완벽한 비트 단위의 데이터 무결성(Data Integrity)을 보장하기 위한 트랜잭션 저널링(Journaling) 기법과 구조적 최적화 방안을 논의한다.

2. 트랜잭션 저널링(Journaling)을 통한 데이터 무결성 보장

디스크 파일에 직접 의존하는 SQLite에서 전원 차단 대비책의 핵심은 저널링 모드의 최적화이다. 기존의 롤백 저널(Rollback Journal) 방식은 쓸 때마다 원본을 백업하므로 I/O 디스크 비용이 매우 비싸다.

2.1 WAL (Write-Ahead Logging) 모드의 도입

GCS 로깅 애플리케이션에서는 SQLite의 PRAGMA journal_mode = WAL; 선언을 통해 로깅 성능을 극대화해야 한다.

  • 동작 원리: 데이터를 데이터베이스 파일에 직접 덮어쓰지 않고, 원본 대신 로그 형태의 별도 WAL 파일의 끝에 데이터 변경 사항을 추가(Append)한다. 일정량이 쌓이거나 체크포인트(Checkpoint)가 발생할 때 백그라운드에서 메인 DB 파일에 병합(Sync)된다.
  • 무결성 복구: WAL에 배치가 기록되던 도중 GCS가 종료되더라도, 재시동 시 SQLite 엔진이 WAL의 트랜잭션을 읽어 정상적인 데이터만 본 파일로 롤포워드(Roll-forward)하므로 파일 깨짐(Corruption)이 원천 봉쇄된다.

2.2 디스크 동기화(Synchronous) 제어

SQLite의 PRAGMA synchronous = NORMAL; 파라미터는 WAL 모드와 결합할 때 쓰기 성능과 안전성 사이의 최적의 타협점을 제공한다. 운영체제의 파일 시스템 캐시를 신뢰하면서 디스크 동기화 횟수를 획기적으로 줄이므로, 텔레메트리 데이터를 고주파수로 인서트(Insert)하는 상황에서도 시스템 프리징(Freezing)을 막아준다.

3. B-Tree 구조 최적화 및 인덱싱(Indexing) 전략

로컬 SQL 데이터베이스를 구성하는 자료 구조인 B-Tree는 검색에는 매우 빠르지만, 삽입이 발생할 때마다 트리의 균형(Balance)을 맞추고 페이지(Page)를 분할(Split)해야 하므로 쓰기 오버헤드가 크다.

3.1 단조 증가(Monotonically Increasing) 기본 키(Primary Key)의 사용

B-Tree에 임의의 UUID나 무작위 해시(Hash) 값을 기본 키로 사용하면 트리의 중간 페이지들에 지속적으로 분할이 발생하여 I/O 성능이 폭락한다.

  • 시계열 최적화: 텔레메트리 데이터 기록 시 GCS 도착 시간(Unix Epoch Milliseconds)이나 자동 순차 증가(Auto-Increment ID) 등 단조 증가하는 값을 기본 키(Primary Key)이자 클러스터드 인덱스(Clustered Index)로 지정한다. 이 경우 언제나 가장 우측 하단의 트리에만 새로운 데이터가 Append 되므로, 페이지 재구성(Page Reorganization) 비용이 거의 발생하지 않는다.

3.2 과도한 보조 인덱스(Secondary Index) 배제

MAVLink 특성상 Message ID, Component ID, Sequence 등 검색할 만한 칼럼이 많다. 하지만 인덱스를 하나 추가할 때마다 쓰기 시의 B-Tree 페이지 업데이트 비용은 제곱으로 증가한다.
사후 분석(Post-Flight Analysis) 목적의 검색 기능이 필요하더라도, 삽입 시점(비행 중)에는 최소한의 타임스탬프 인덱스만 유지하고, 비행 종료 후(비행 대기 상태)에 보조 인덱스들을 일괄 생성(Create Index)하는 설계 패턴이 자원 제약이 있는 GCS 설계에서 적극적으로 권장된다.

4. QGroundControl 및 자체 GCS 적용 시사점

오픈소스인 QGC의 C++ 기반 로깅 엔진 역시 파일 무결성을 위한 나름의 버퍼링(QFile) 기법을 사용하지만, 확장된 메타데이터 관리를 위해 SQLite를 융합할 경우 위와 같은 파라미터 튜닝이 필수적이다.
특히 모바일 하드웨어(eMMC나 UFS 기반의 플래시 메모리)는 임의 쓰기(Random Write) 속도가 현격히 떨어지므로, WAL 모드를 통한 연속 쓰기(Sequential Write)로의 패턴 전환은 플래시 메모리의 수명(Wear Leveling) 연장이라는 부가적인 이득도 함께 가져다 준다.

5. 결론

고빈도의 자율주행 상태 데이터를 기록하는 GCS 내부 데이터베이스는, I/O 부하와 전원 차단 시나리오를 설계 단에서 원천적으로 통제해야 한다. B-Tree 트리의 페이지 단편화를 막기 위한 단조 증가 인덱스 활용, 그리고 WAL 모드를 기반으로 한 트랜잭션 저널링 구조 최적화는 어떠한 운용 환경의 붕괴 속에서도 “기록된 텔레메트리의 원본 무결성“을 극한으로 보장하는 핵심 공학 설계 원칙이다.
