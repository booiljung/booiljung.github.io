59.2.5 실시간성 보장을 위한 스레드 풀(Thread Pool) 할당 알고리즘 및 선점형(Preemptive) 스케줄링 우선순위 설계

지상 관제 시스템(GCS, Ground Control Station)은 무인항공기(UAV)와의 다채널 MAVLink 통신, 실시간 페이로드 비디오 스트리밍 처리, 고해상도 지도 렌더링, 방대한 로그 파일 시스템 I/O 등 이질적인 작업을 동시에 지연 없이 처리해야 하는 고부하 시스템이다. 특히 무인기의 안전과 직결되는 수동 조종 명령(Joystick/RC) 전달과 텔레메트리 상태 갱신의 ‘실시간성(Real-Time Performance)’ 보장은 전체 GCS 소프트웨어 아키텍처 설계에서 가장 중요한 목표이다.

본 절에서는 작업이 발생할 때마다 무분별하게 스레드를 생성하여 발생하는 성능 저하를 방지하기 위한 스레드 풀(Thread Pool) 할당 알고리즘과, 생명 유지에 치명적인 필수 작업이 CPU 자원을 먼저 차지할 수 있도록 강제하는 선점형(Preemptive) 스케줄링 우선순위 설계 기법을 심도 있게 다룬다.

1. 컨텍스트 스위칭(Context Switching) 오버헤드와 스레드 풀의 필요성

일반적으로 네크워크 패킷이 수신될 때마다 새로운 std::thread나 QThread를 생성하여 처리하게 되면 치명적인 문제가 발생한다. 스레드의 생성과 소멸 자체로 메모리 할당 및 OS의 커널 모드/유저 모드 전환 등 상당한 시스템 자원이 소모된다.
더욱 심각한 것은 생성된 스레드의 수가 하드웨어 프로세서(CPU)의 물리적 코어 수를 크게 초과할 때 발생하는 컨텍스트 스위칭(Context Switching, 문맥 교환) 오버헤드이다. OS 스케줄러가 여러 스레드에게 공평하게 CPU 시간을 나누어주기 위해 현재 실행 중인 스레드의 상태 레지스터를 저장하고 다른 스레드의 상태를 복원하는 과정을 쉴 새 없이 반복하게 되며, 이 과정에서 CPU 캐시 메모리의 무효화(Cache Invalid)가 발생하여 전체 시스템의 지연 속도를 증폭시킨다.

스레드 풀(Thread Pool)은 시스템 시작 시점에 하드웨어 코어 수에 최적화된 일정 개수의 스레드를 미리 생성해 두고(Warm-up), 들어오는 작업을 대기 큐(Task Queue)에 넣은 후 유휴 상태의 스레드를 할당하여 처리하게 하는 기법이다. 이를 통해 스레드 생성 한계 비용을 0으로 만들고, 불필요한 문맥 교환을 억제하여 예측 가능한 타이밍 내에 작업을 마칠 수 있게 한다.

2. GCS 맞춤형 스레드 풀(Thread Pool) 할당 알고리즘 설계

GCS 환경에서는 단순히 단일 스레드 풀을 사용하는 것보다, 작업의 성격에 따라 다중 스레드 풀(Multi-Tier Thread Pool)을 운영하는 큐잉 네트워크(Queueing Network) 설계가 권장된다.

2.1 I/O Bound 작업과 CPU Bound 작업의 분리

  • CPU Bound Pool (계산 집약적 작업 풀): MAVLink 데이터 패킷 파싱, CRC 체크섬 연산, 좌표계 변환, 상태 머신 로직 검증 등 CPU 사이클을 적극적으로 소비하는 작업. CPU 코어 개수에 맞추어 고정된 수(std::thread::hardware_concurrency())의 풀을 할당하는 것이 캐시 히트율을 극대화한다.
  • I/O Bound Pool (입출력 집약적 작업 풀): 로컬 데이터베이스의 비동기 기록, 인터넷 타일 맵 다운로드, TCP/UDP 소켓 대기 등 I/O 병목으로 인해 빈번하게 블로킹(Blocking)되는 작업. 이 풀은 블로킹으로 인한 대기 시간이 길기 때문에, 물리 코어 수보다 2~3배 많은 개수로 동적 확장(Dynamic Expand)할 수 있도록 알고리즘을 설정하는 것이 유리하다.

3. 선점형(Preemptive) 스케줄링과 우선순위(Priority) 설계

GCS를 구동하는 일반적인 범용 OS(Windows, macOS, Desktop Linux)는 기본적으로 모든 응용 프로그램의 스레드에 공평한 처리 시간을 분배하는 라운드 로빈(Round-Robin) 또는 CFS(Completely Fair Scheduler) 정책을 사용한다. 그러나 GCS에서는 일반적인 데스크톱 애플리케이션과는 달리 “가장 중요한 작업이 하위 작업을 밀어내고 즉각 실행되어야 하는” 선점형 방식이 묘사되어야 한다.

이를 위해 OS나 프레임워크가 제공하는 스레드 우선순위 API(예: POSIX의 pthread_setschedparam, Qt의 QThread::Priority)를 통제하여 계층적인 우선순위 트리를 구축해야 한다.

3.1 GCS 아키텍처 내 스레드별 이상적인 우선순위 배치

우선순위 계층대상 스레드 및 작업 기능설명 및 설정 가이드
Highest (Time Critical)조종 스틱(Joystick) 명령 수집, MAVLink Heartbeat 전송, 비상 정지(Kill Switch) 송신PX4-Autopilot의 Failsafe 발동을 막고 수동 제어를 유지하기 위한 생명선. 어떠한 시스템 부하에서도 10~50ms의 주기를 지켜야 함.
High텔레메트리 패킷 수신 및 구문 분석, 알람 처리기체의 현재 상태(고도, 배터리)를 신속하게 판단하기 위해 OS 소켓 버퍼에서 패킷을 즉시 꺼내야 함.
Normal (Default)QML 기반 UI 렌더링 스레드 (Main Thread), 지도 타일 렌더링조종자 인터페이스의 반응성을 유지하는 일반적인 애플리케이션 우선순위. 초당 30~60 프레임을 목표로 함.
Low / Background대용량 백그라운드 파일 로깅, 페이로드 영상 분석(AI 추론 등), 펌웨어 다운로드CPU 코어를 오랫동안 점유하는 작업으로, 백그라운드 우선순위를 주어 조종 조작에 지연을 주지 않도록 격리 시스템 적용.

4. 프레임워크 기반 구현 예시 (C++ / Qt 환경)

QGroundControl을 비롯한 최신 멀티플랫폼 GCS들은 내부적으로 Qt의 스레딩 시스템을 적극 채용한다. 백그라운드 I/O 작업이나 단발성 계산 작업은 QThreadPool과 QRunnable 클래스를 상속받아 구현함으로써 스레드 풀 할당 알고리즘을 쉽게 적용할 수 있다.

class TelemetryParserTask : public QRunnable {
    void run() override {
        // MAVLink 페이로드 복호화 알고리즘
    }
};

// I/O 대기가 적은 CPU Bound 작업을 위한 전역 스레드 풀 사용
QThreadPool::globalInstance()->start(new TelemetryParserTask());

주기적으로 평생 동작해야 하는 심박태스크(Heartbeat)나 조이스틱 인터페이스는, 별도의 백그라운드 인스턴스 전용 스레드를 생성한 뒤 최상위 우선순위를 부여(QThread::TimeCriticalPriority)하여 선점형 모델을 모방한다.

QThread* joystickThread = new QThread();
JoystickManager* manager = new JoystickManager();
manager->moveToThread(joystickThread);

connect(joystickThread, &QThread::started, manager, &JoystickManager::processLoop);

// 조종 명령 스레드에 최상위 실시간 우선순위 설정
joystickThread->start(QThread::TimeCriticalPriority);

이와 같이 스레드 풀 할당 최적화 및 엄격한 우선순위 스케줄링이 결합되면, 코어 간 통신 효율성이 극대화되어 PX4-Autopilot과 같은 고성능 비행 제어 컴퓨터와 통신할 때 GCS 측의 병목으로 인한 제어권 상실 위험(Data Link Loss Failsafe)을 최소화할 수 있다.
