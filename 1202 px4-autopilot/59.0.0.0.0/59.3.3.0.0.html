59.3.3 다중 채널 인터페이스(비동기 직렬, UDP 다중 송광, TCP 스트림) 병렬 I/O 다중화(Multiplexing) 및 논블로킹(Non-blocking) 소켓 프로그래밍

1. 개요: 지상 관제 시스템의 다중 채널 통신 필요성

현대의 무인항공기(UAV) 지상 관제 시스템(GCS, Ground Control Station)은 한 번에 단일 기체하고만 통신하는 과거의 1:1 구조에서 벗어나, 군집 비행(Swarm Flight)을 관제하거나 단일 기체라 하더라도 텔레메트리(Telemetry), 페이로드 영상(Payload Video), RTK 보정 데이터(RTCM) 등 이종의 데이터 스트림을 동시에 처리해야 하는 복합 환경으로 발전하였다.

PX4-Autopilot 기반의 드론 시스템에서는 이러한 이종의 데이터를 비동기 직렬(Asynchronous Serial), UDP 송방향/다중 송광(Multicast/Broadcast), 그리고 안정성이 요구되는 TCP 스트림 등을 섞어 사용한다. 이와 같이 다층적인 네트워크를 하나의 애플리케이션 내에서 스레드(Thread) 병목 없이 매끄럽게 수신 및 처리하기 위해서는 **I/O 다중화(Multiplexing)**와 논블로킹(Non-blocking) 프로그래밍 기법 도입이 필수적이다.

참고: 본 기술 설명은 QGroundControl v4.3.0 통신 아키텍처(LinkManager 클래스 관점) 및 PX4 v1.14 버전을 기준으로 서술된다.

2. 매체별 통신 인터페이스의 특성

GCS 개발자는 자신이 다루는 인터페이스의 블로킹 양상(Blocking behavior)과 데이터 전송 단위를 명확히 인지하고 그에 맞는 소켓 속성을 부여해야 한다.

  • 비동기 직렬 포트 (Asynchronous Serial/UART): 주로 433MHz/915MHz 텔레메트리 모뎀과 통신할 때 사용된다. 바이트(Byte) 단위의 스트림 통신이므로, MAVLink 패킷이 중간에 쪼개져서(Fragmented) 들어올 수 있음을 전제해야 한다.
  • UDP 네트워크 (User Datagram Protocol): 인터넷 프로토콜 통신(Wi-Fi, LTE 등)을 사용할 때 주로 활용되는 방식으로, QGroundControl은 기본적으로 포트 기반의 UDP 수신을 사용한다. 패킷(Datagram) 단위로 통신이 이루어지며, 전송 신뢰성이 떨어지지만 지연 시간(Latency)이 짧다.
  • TCP 스트림 (Transmission Control Protocol): 설정 파라미터(Parameter)의 대량 업로드/다운로드, 펌웨어 업데이트, 그리고 RTK 보정 데이터와 같이 단방향의 높은 무결성이 요구될 때 선별적으로 활용된다. 연결 지향성(Connection-Oriented) 특징으로 인해 연결 수립(Handshake)간 블로킹 현상에 유의해야 한다.

3. 병렬 I/O 다중화(Multiplexing) 아키텍처

여러 개의 소켓과 파일 디스크립터(File Descriptor, FD)를 보유한 상태에서 각각의 입력 채널로부터 언제 데이터가 도달할 지 알 수 없다면, 개별 채널마다 무한 대기를 하는 수신 대기 스레드(Blocking thread)를 여러 개 생성할 수밖에 없다. 이는 시스템 리소스를 낭비하고 컨텍스트 스위칭(Context Switching)에 따른 지연을 초래한다.

이를 극복하기 위해 운영체제(OS)가 제공하는 select(), poll(), 혹은 Linux의 epoll(), MacOS의 kqueue()와 같은 I/O 다중화 기법을 활용한다.

graph TD
    A[GCS Main Reactor Thread] --> B{epoll / select 다중화기}
    
    B -->|Event Ready| C(Serial /dev/ttyUSB0)
    B -->|Event Ready| D(UDP Socket :14550)
    B -->|Event Ready| E(TCP Socket :5760)
    
    C -.->|읽기 발생| M[MAVLink Stream Parser]
    D -.->|읽기 발생| M
    E -.->|읽기 발생| M
    
    M --> F[MAVLink Message Router]

위 도표와 같이 메인 통신 스레드(Reactor)는 다중화 함수에서 모든 소켓의 소켓 상태 변화(데이터 도달, 오류 발생, 쓰기 버퍼 비워짐)를 감지하다가, 이벤트가 발생한 소켓의 FD(File Descriptor)만 추출하여 읽기(Read) 작업을 수행한다.

4. 논블로킹(Non-blocking) 소켓 파싱 기법

I/O 다중화로 이벤트 도달을 통보받았더라도, 해당 소켓의 read() 혹은 recv() 시스템 콜을 호출할 때 남은 데이터양이 예상보다 모자라 블로킹이 일어나면 전체 스레드가 멈추게(Hang) 된다. 이에 대응하기 위해 POSIX 시스템에서는 fcntl() 호출을 통해 모든 소켓을 논블로킹 모드(O_NONBLOCK)로 설정해야 한다.

다음은 다중 채널을 다루는 C++ 네이티브 환경에서의 뼈대가 되는 논블로킹 소켓 파싱의 예시이다.

#include <fcntl.h>
#include <sys/socket.h>

// 논블로킹 모드 활성화 (FD: File Descriptor)
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

// 데이터 수신 버퍼
uint8_t rx_buffer[1024];

while (true) {
    // 논블로킹 읽기 시도
    ssize_t bytes_read = recv(fd, rx_buffer, sizeof(rx_buffer), 0);
    
    if (bytes_read > 0) {
        // 성공적으로 데이터를 읽음, MAVLink 파서로 전달
        for (ssize_t i = 0; i < bytes_read; i++) {
            mavlink_message_t msg;
            mavlink_status_t status;
            
            // 바이트 단위 파싱 (내부적으로 상태 머신 작동)
            if (mavlink_parse_char(MAVLINK_COMM_0, rx_buffer[i], &msg, &status)) {
                handle_mavlink_message(&msg);
            }
        }
    } else if (bytes_read == -1 && (errno == EAGAIN || errno == EWOULDBLOCK)) {
        // 더 이상 수신할 데이터가 없이 깨끗하게 비워짐
        break; 
    } else {
        // 통신 오류 혹은 채널 끊어짐 처리
        trigger_link_disconnected(fd);
        break;
    }
}

이 코드는 한 번의 이벤트 통보에 대하여 해당 채널 수신 버퍼 내의 모든 잔류 데이터를 EAGAIN 오류 코드가 반환될 때까지 모두 읽어(drain) 들임으로써, MAVLink 파싱의 처리 지연(Processing Latency)을 전면적으로 억제한다.

5. QGroundControl 내 실제 구현 구조

QGroundControl 프레임워크는 내부적으로 Qt 네트워크 모듈(QTcpSocket, QUdpSocket, QSerialPort)을 사용하여 위에서 언급한 다중화와 논블로킹 아키텍처를 구현하고 있다.

  • LinkManager 계층: GCS로 들어오는 모든 TCP, UDP, Serial 연결 인스턴스를 관리한다. 이 계층은 드론 시스템 한 대(Vehicle)에 대해 여러 통신 링크를 가질 수 있으며 반대로 하나의 통신 링크가 다수 대의 드론 메시지를 실어 나를 수 있는 M:N 다중화 처리를 내재하고 있다.
  • Qt 이벤트 루프(Event Loop): Qt의 readyRead() 시그널을 통해 운영체제 레벨의 이벤트 알림(epoll/kqueue/select) 메커니즘을 캡슐화한다.
  • 통로 추상화(Abstraction): 각각의 링크 채널은 LinkInterface 부모 클래스를 상속받아 _readBytes(QByteArray)와 같은 가상 함수를 비동기로 구현하도록 강제된다.

QGroundControl과 같은 성숙한 환경을 모방하여 커스텀 GCS를 고안할 때, 다중 채널 환경에서 가장 치명적인 문제인 데이터 버퍼 잠금(Buffer deadlock)이나 좀비 소켓(Zombie socket) 문제를 방지하기 위해서는 앞서 설명한 I/O 다중화 리액터(Reactor) 패턴과 논블로킹 바이트 스트림 파싱 구조를 시스템 뇌관의 깊숙한 곳에서부터 철저하게 구현해야 함을 명심해야 한다.
