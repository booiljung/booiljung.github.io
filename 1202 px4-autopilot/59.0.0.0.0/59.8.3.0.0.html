59.8.3 모바일 관제(Flutter/React Native)를 위한 직렬 포트 프로파일(SPP) 및 저전력 블루투스(BLE) GATT(Generic Attribute Profile) 특성(Characteristic) 매핑 브릿지 설계

1. 개요

최근 무인항공기(UAV) 운영 환경에서는 PC 기반의 무거운 지상 관제 시스템(GCS; Ground Control Station)에서 벗어나, 스마트폰이나 태블릿을 활용한 모바일 관제의 수요가 급증하고 있다. QGroundControl(QGC, v4.3 기준)이나 별도의 커스텀 GCS를 모바일 프레임워크인 플러터(Flutter) 또는 리액트 네이티브(React Native)로 개발할 때, 기체와의 통신은 주로 와이파이(Wi-Fi)나 블루투스(Bluetooth)를 통해 이루어진다.
블루투스 통신은 전력 소모가 적고 근거리에서 안정적인 연결을 제공하므로 모바일 관제에 적합하다. 이 장에서는 클래식 블루투스의 직렬 포트 프로파일(SPP; Serial Port Profile)과 저전력 블루투스(BLE; Bluetooth Low Energy)의 일반 속성 프로파일(GATT; Generic Attribute Profile)을 기반으로, PX4-Autopilot(v1.14 기준)의 MAVLink 메시지를 안정적으로 송수신하기 위한 매핑 브릿지(Mapping Bridge) 설계 기법에 대해 서술한다.

2. 직렬 포트 프로파일(SPP) 및 BLE GATT의 이해

모바일 디바이스와 PX4 텔레메트리(Telemetry) 간의 원활한 MAVLink 송수신을 위해서는 두 가지 블루투스 규격을 명확히 이해해야 한다.

2.1 직렬 포트 프로파일(SPP; Serial Port Profile)

SPP는 클래식 블루투스(Classic Bluetooth) 규격의 하나로, 기존의 유선 RS-232 직렬 통신을 무선으로 에뮬레이션(Emulation)하는 프로토콜이다.

  • 작동 방식: 데이터 스트림을 순차적으로 바이트(Byte) 단위로 전송하여, UART 기반의 텔레메트리 모듈과 매우 훌륭한 호환성을 보여준다.
  • 장점: 한 번 연결되면 연속적인 데이터 스트림을 지속적으로 밀어낼 수 있어, MAVLink와 같이 페이로드 크기가 가변적이고 연속적으로 발생하는 데이터 전송에 유리하다.
  • 단점: iOS(Apple) 기기에서는 MFi(Made for iPhone) 인증을 받지 않은 블루투스 기기의 SPP 연결을 엄격하게 제한하므로, 크로스 플랫폼(Cross-Platform) 모바일 GCS 개발 시 심각한 제약 사항이 된다.

2.2 저전력 블루투스(BLE) 및 GATT(Generic Attribute Profile)

BLE는 저전력 통신을 목적으로 설계되었으며, 상태 및 데이터를 속성(Attribute) 형태로 관리하는 GATT(Generic Attribute Profile) 기반 통신을 사용한다.

  • 서비스(Service) 및 특성(Characteristic): GATT 통신은 서비스(Service)라는 큰 묶음 안에 여러 개의 특성(Characteristic)을 포함하는 계층적 트리 구조를 갖는다. 각 특성은 고유한 UUID(Universally Unique Identifier)를 가지며, 읽기(Read), 쓰기(Write), 알림(Notify), 지시(Indicate) 등의 속성을 통해 데이터를 교환한다.
  • MTU (Maximum Transmission Unit) 한계: BLE는 패킷 기반 전송을 하므로 한 번에 보낼 수 있는 데이터 크기(MTU)가 20바이트 내외(기본값)에서 최대 512바이트로 제한된다. 긴 MAVLink 메시지를 전송하려면 필연적으로 패킷을 분할(Fragmentation)하고 재조립(Reassembly)하는 과정이 요구된다.
  • 장점: iOS와 Android 모두 표준 API를 통해 특별한 인증 없이 주변기기와 통신할 수 있어, Flutter/React Native를 활용한 크로스 플랫폼 앱 개발에 필수적이다.

3. 매핑 브릿지(Mapping Bridge) 설계 아키텍처

블루투스 기반 매핑 브릿지의 핵심 역할은, 연속적인 바이트 스트림을 전제하는 MAVLink 메시지를 BLE의 이벤트 및 속성(Attribute) 기반 패킷 구조로 원활하게 넘겨주는 추상화 계층(Abstraction Layer)을 제공하는 것이다.

graph TD
    sub향 [PX4-Autopilot / 텔레메트리]
        A(UART 송수신) --> B(Bluetooth SoC - ESP32, HM-10 등)
    end

    sub향 [모바일 기기 내부 브릿지 구조]
        B -.-> |BLE / SPP 무선 링크| C{Bluetooth API 브릿지}
        C --> |SPP 스트림| D[Data Stream Parser]
        C --> |BLE Notify / Write| E[GATT Fragmenter / Reassembler]
        
        D --> F(MAVLink 프레이머 및 검증)
        E --> F
        
        F --> G(Flutter / React Native UI 계층)
    end

3.1 GATT 특성(Characteristic) 매핑 전략

BLE 환경에서 UART 통신을 흉내 내기 위해 이른바 “BLE UART” 서비스(예: Nordic UART Service, NUS) 패턴을 주로 활용한다. 브릿지 상에서 다음과 같이 GATT 특성을 매핑한다.

  1. RX Characteristic (수신 특성):
  • 속성: Write, Write Without Response
  • 역할: 모바일 GCS에서 PX4로 MAVLink 명령(예: MAV_CMD_NAV_TAKEOFF)을 전송할 때 사용한다.
  1. TX Characteristic (송신 특성):
  • 속성: Notify
  • 역할: PX4에서 모바일 GCS로 상태 데이터(예: HEARTBEAT, ATTITUDE, GLOBAL_POSITION_INT)를 전송할 때 사용한다.

3.2 패킷 분할(Fragmentation) 및 재조립(Reassembly)

MAVLink 2.0 프로토콜의 패킷 길이는 헤더 및 체크섬(Checksum)을 포함하여 최대 280바이트에 달할 수 있다. 반면, 안드로이드나 iOS의 기본 BLE MTU 협상(Negotiation)이 실패율을 감안하여 보수적으로 설정될 경우 20바이트씩 데이터를 쪼개어(Chunking) 보내야 한다.

  • 송신(GCS -> PX4): 모바일 앱의 펌웨어 브릿지에서는 MAVLink 메시지를 직렬화(Serialization)한 후, 현재 연결된 BLE의 가용 MTU 크기 단위로 잘라서(Payload Chunk) 여러 번의 Write 요청을 통해 순차적으로 전송한다.
  • 수신(PX4 -> GCS): Notify 이벤트로 들어오는 페이로드 파편들을 링 버퍼(Ring Buffer)에 적재하고, MAVLink 파서(Parser)가 슬라이딩 윈도우(Sliding Window) 방식으로 페이로드의 STX(시작 바이트, MAVLink v2 버전에선 0xFD), 길이, CRC 체킹을 수행하여 유효한 메시지만을 추출한다.

4. Flutter / React Native 구현 시 고려사항 및 예시

모바일 프레임워크 환경에서 MAVLink 관제를 구현할 때 가장 주의해야 할 요소는 가비지 컬렉터(Garbage Collector)의 부하와 이벤트 루프 블로킹(Event Loop Blocking)이다.

4.1 오버헤드 최적화

초당 десятки 헤르츠(Hz)로 쏟아지는 센서 데이터(예: HIGHRES_IMU)를 BLE 알림(Notify)으로 직접 수신하면 네이티브-Dart(혹은 JS) 브릿지의 JNI/ObjC 오버헤드로 인해 UI 프레임 드랍(Frame Drop)이 발생한다.
이를 완화하기 위해 네이티브 계층(안드로이드의 경우 Kotlin/C, iOS의 경우 Swift/C)에 MAVLink 파서를 내장하여 패킷 조립을 네이티브 영역에서 수행하게 한다. 그런 다음, 완성된 특정 상태 모델(State Model) 단위로 묶어(Batching) 비동기 채널(MethodChannel, EventChannel)을 통해 Flutter/React Native 측으로 이벤트 알림을 보내는 방식이 매우 효과적이다.

4.2 매핑 브릿지 상태 관리 예시 (Flutter)

// Flutter에서의 BLE MAVLink 브릿지 버퍼 수신부 로직 예시
StreamSubscription? _bleNotifySub;
final RingBuffer _mavlinkBuffer = RingBuffer(capacity: 1024);

void setupBLEStream(BluetoothCharacteristic txChar) {
  _bleNotifySub = txChar.value.listen((List<int> chunk) {
    // 1. 단편화된 데이터를 링 버퍼에 적재
    _mavlinkBuffer.add(chunk);
    
    // 2. MAVLink 스트림 파싱 (STX 및 Checksum 검증)
    while (_mavlinkBuffer.hasCompleteMessage()) {
      MavlinkMessage msg = _mavlinkBuffer.extractMessage();
      
      // 3. 메시지 ID에 따른 상태 업데이트
      _processMavlinkMessage(msg);
    }
  });
}

5. Ardupilot 대비 관제 시스템 매핑 차이

PX4-Autopilot과 Ardupilot 모두 MAVLink를 표준 텔레메트리 프로토콜로 사용하므로 통신 계층의 프레이밍(Framing) 자체는 동일하다. 하지만, 모바일 환경에서 SPP 및 BLE 브릿지를 통해 제어할 때 다음과 같은 설계 철학의 차이를 반영해야 한다.

  • 메시지 스트리밍 주기 구성: Ardupilot은 Mission Planner에서 볼 수 있듯, GCS가 REQUEST_DATA_STREAM 명령을 적극적으로 호출하여 대역폭을 관리하는 패턴이 주를 이뤘다. 반면, PX4는 최신 버전으로 올수록 MAVLink의 SET_MESSAGE_INTERVAL 커맨드를 선호하며 개별 이벤트 스트림 주기를 세밀하게 매핑하도록 유도된다. 대역폭이 제한된 BLE 환경에서는 PX4의 방식을 이용하여 실제로 UI에 표출되는 데이터(예: 배터리 상태, 고도)의 스트림 주기만 선별적으로 요청(Request)하여 블루투스 스택의 과부하를 막는 것이 필수적이다.
  • 컴포넌트 식별자: PX4는 시스템 및 컴포넌트 ID를 보다 세분화하여 다루는 경향이 있어 모바일 브릿지가 멀티 드론(Swarm) 환경에서 UUID-컴포넌트 식별 매핑을 보다 유연하게 가져가야 한다.

6. 결론

모바일 프레임워크 기반의 GCS 개발 시 안정성을 결정짓는 핵심은 BLE 패킷의 단편화를 처리하고 직렬(Serial) 기반의 MAVLink 스트림을 온전하게 복원하는 매핑 브릿지의 견고함에 있다. MTU 기반의 동적 분할 알고리즘을 구현하고, 모바일 플랫폼 네이티브 수준에서 데이터 전처리를 수행한다면 오버헤드를 줄이면서 안정적인 무인항공기 관제 앱을 설계할 수 있다.
