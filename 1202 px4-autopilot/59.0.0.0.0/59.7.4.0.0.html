59.7.4. 고빈도 텔레메트리 업데이트로 인한 화면 깜빡임(Flicker) 억제를 위한 더블 버퍼링(Double Buffering) 및 하드웨어 수직 동기화(V-Sync) 활용

1. 개요 및 문제 정의 (Introduction and Problem Statement)

무인항공기(UAV)의 비행 중에는 막대한 양의 자세(Attitude), 위치(Position), 배터리(Battery) 등의 비행 데이터가 GCS(Ground Control Station)로 전송된다. 특히 PX4-Autopilot(펌웨어 버전 v1.14 기준) 기반의 시스템에서 MAVLink 프로토콜을 사용해 고빈도 텔레메트리 스트림(Telemetry Stream)을 수신할 때, 초당 수십에서 수백 회의 데이터 업데이트가 발생할 수 있다. QGroundControl(QGC, v4.3 및 v4.4 기준)과 같은 현대적인 관제 소프트웨어는 수신된 데이터를 바탕으로 즉각적으로 사용자 인터페이스(UI)를 갱신해야 한다.

하지만, 단일 버퍼링(Single Buffering) 방식 또는 비동기화된 렌더링 루프를 거치게 되면 렌더링 주기와 디스플레이 화면의 주사율(Refresh Rate)이 일치하지 않아 화면 깜빡임(Screen Flicker) 현상이나 화면이 찢어지는 테어링(Screen Tearing) 현상이 빈번하게 발생한다. 이는 임무 수행 중인 조종사에게 시각적 피로감을 유발하고, 치명적인 오판(Misjudgment)을 초래하는 핵심적인 원인이 된다. 본 절에서는 이러한 깜빡임을 억제하고 데이터의 시각적 지연을 최소화하기 위한 그래픽 처리 기법으로, 더블 버퍼링(Double Buffering) 및 하드웨어 수직 동기화(V-Sync) 기술의 공학적 원리와 GCS 내에서의 활용 방법을 깊이 있게 논의한다.

2. 화면 깜빡임(Flicker) 및 테어링(Tearing)의 발생 메커니즘

디스플레이 장치는 위에서 아래로(Top-to-Bottom), 그리고 좌에서 우로 픽셀(Pixel)을 갱신하며 화면을 그려나간다. 응용 프로그램(GCS)이 모니터 화면에 직접 픽셀 데이터를 쓰는 상황을 가정해 보자.

  1. 단일 버퍼링(Single Buffering)의 문제점: 그래픽 데이터를 디스플레이의 프레임 버퍼에 직접 기록할 경우, 렌더링 속도와 모니터 주사율이 불일치하게 된다. 만약 렌더링이 화면 중앙을 그리는 도중에 MAVLink로부터 새로운 고도(Altitude)나 인공 수평의 자세(Roll/Pitch) 데이터가 들어와 픽셀을 덮어쓰게 되면, 모니터 상단부는 이전 프레임, 하단부는 새로운 프레임이 표시되어 수평으로 잘린 듯한 ’어긋남(Tearing)’이 관측된다.
  2. 높은 빈도의 갱신(High-Frequency Updates): GCS가 HIGHRES_IMU나 ATTITUDE 메시지와 같은 고빈도 데이터를 UI 스레드에 그대로 반영할 경우 UI 자체의 무효화(Invalidation)와 리페인트(Repaint) 작업이 연속적으로 발생한다. 이로 인해 불연속적인 렌더링과 버퍼 언더런(Buffer Underrun)이 일어나며 극심한 화면 깜빡임(Flickering)이 유발된다.

3. 더블 버퍼링(Double Buffering) 아키텍처

이를 근본적으로 해결하기 위한 표준 컴퓨터 그래픽스 기법이 **더블 버퍼링(Double Buffering)**이다.

더블 버퍼링은 메모리 내부에 두 개의 디스플레이 영역을 가지는 원리이다. 하나는 현재 사용자에게 보여지고 있는 **프런트 버퍼(Front Buffer)**이며, 다른 하나는 다음 프레임을 준비하기 위해 백그라운드 렌더링이 이루어지는 **백 버퍼(Back Buffer)**이다. 고빈도 텔레메트리 데이터가 유입되면, GCS의 렌더링 엔진(예: Qt Scene Graph)은 모든 그리기 작업(UI 갱신, 계기판 계측 등)을 백 버퍼에서 수행한다.

백 버퍼에서 새로운 프레임 구성이 완전히 끝나게 되면 단일 명령으로 프런트 버퍼와 백 버퍼의 포인터(Pointer)를 교환하는 버퍼 스왑(Buffer Swap) 또는 플립(Flip) 작업을 수행한다. 사용자는 온전히 완성된 프레임만을 확인하게 되므로 화면을 그리는 과정이 시각적으로 노출되지 않고, 깜빡임을 완벽하게 억제할 수 있다.

4. 하드웨어 수직 동기화(Hardware V-Sync) 활용

더블 버퍼링만으로는 테어링(Tearing) 현상을 완전히 억제할 수 없다. 버퍼 스왑(Buffer Swap)이 모니터가 화면을 그리고 있는 도중(Scanout)에 발생하면 여전히 화면이 찢어지게 된다.

이를 방지하기 위해 디스플레이 컨트롤러가 화면 갱신을 끝내고 다음 화면을 그리기 위해 커서를 맨 위로 옮기는 짧은 대기 시간, 즉 **수직 귀선 기간(Vertical Blanking Interval, VBlank)**에 맞춰 버퍼 스왑을 실행하는 기술이 **수직 동기화(Vertical Synchronization, V-Sync)**이다.

  • V-Sync의 논리적 동작 메커니즘:
    GCS 애플리케이션 프레임워크는 GPU 드라이버에게 SwapInterval = 1 과 같은 플래그(Flag)를 전달하여, 다음 VBlank 신호가 발생할 때까지 버퍼 스왑 명령을 대기(Block)하도록 지시한다.
  • 텔레메트리 갱신 속도의 최적화:
    예를 들어 디스플레이 주사율이 60Hz라면, V-Sync가 켜진 상태에서는 최대한 60FPS에 맞추어 렌더링 파이프라인이 구동된다. 만약 텔레메트리 데이터 스트림이 초당 200번 들어온다고 하더라도, UI 쪽에서는 200번 그릴 필요가 없다. 가장 최신의 텔레메트리 데이터를 캐시(Cache)에 저장해 두고, V-Sync 주기에 동기화되는 프레임 렌더링 타이밍에 단 한 번 샘플링하여 렌더링함으로써 CPU/GPU 리소스 낭비를 막고 극도로 부드러운 화면 전환을 성취한다.

5. QGroundControl 아키텍처 내 적용 및 차별점

QGroundControl(이하 QGC)을 포함한 독자 개발 GCS 애플리케이션은 이러한 그래픽 최적화 아키텍처를 적극적으로 도입해야 한다.

  • Qt/QML 프레임워크 기반 렌더링 적용 (QGC 관점):
    QGC(v4.x 버전)는 Qt 그래픽 프레임워크 위에서 개발된다. Qt Quick 및 QML의 핵심 렌더링 백엔드인 **씬 그래프(Scene Graph, SG)**는 내부적으로 하드웨어 가속(OpenGL, Vulkan, Metal)과 스레드 렌더링(Threaded Rendering), 그리고 더블 버퍼링 및 V-Sync가 기본 적용(Enable)되어 동작하도록 설계되었다.
    따라서 수평 인디케이터(Attitude Indicator) 컴포넌트를 설계할 때, 개별 텔레메트리 도달 이벤트(Event)에 종속적으로 즉각 화면 갱신 API를 호출해서는 안 되며, property binding과 V-Sync에 종속적인 Qt 애니메이션 루프에 의존하도록 설계하여야 깜빡임 없는 견고한 계기판 표시가 가능하다.

  • 임베디드 및 제한된 디바이스 환경에서의 활용:
    현장(Field)에서 타블렛 또는 팬리스(Fanless) 기반 스마트 조종기에 GCS가 탑재되는 경우, V-Sync 오버헤드가 발생할 수 있다. 따라서 프레임 생성 시간이 모니터 주사율 임계치(16.6ms at 60Hz)를 초과하게 되면 프레임 드롭(Frame Drop) 현상으로 인한 화면 멈춤(Stuttering)이 유발되므로, 백 버퍼에서 처리되는 데이터 계측 알고리즘의 최적화가 병행되어야 한다.
