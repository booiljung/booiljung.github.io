59.1.2 항공기 탑재 소프트웨어 표준(DO-178C)에 준하는 지상 관제 시스템의 신뢰성 요구사항 도출

무인항공기(UAV)가 도심 항공 모빌리티(UAM)나 고위험 산업 현장(예: 원자력 발전소 정찰, 인구 밀집 지역 비행 등)에서 가시권 밖(BVLOS) 임무를 일상적으로 수행하게 됨에 따라, 지상 관제 시스템(GCS)은 단순한 보조 모니터링 장치가 아닌 기체의 안전 운항을 직접적으로 책임지는 “절대적 무결성 검증 노드(Mission-Critical Node)“로 격상되었다. 특히 비행 제어 및 안전과 직결된 소프트웨어의 경우, 국제 항공 라디오 기술 위원회(RTCA)에서 제정한 전통적인 유인 항공기 탑재 소프트웨어 인가 표준인 **DO-178C (Software Considerations in Airborne Systems and Equipment Certification)**의 엄격한 가이드라인 요구사항을 점진적으로 준용하는 추세이다.

비록 DO-178C가 엄밀히 말하여 기체 탑재(Airborne) 장비만을 대상으로 제정된 표준이나, DO-278A(지상 관제용 및 통신 시스템 감항 관리 표준)와 연동하여 해석할 때 PX4-Autopilot 생태계 기반 관제망의 신뢰성을 공학적으로 보증하는 가장 강력하고 체계적인 기준점이 된다. 본 절에서는 DO-178C의 철학을 준용하여 고신뢰성 도메인 특화 GCS를 설계하기 위한 필수 소프트웨어 품질 요구사항들을 도출한다.

1. 소프트웨어 안전성 보증 수준(DAL) 기반 아키텍처 파티셔닝

DO-178C는 시스템 고장 상태(Failure Condition)가 항공기 기체 그 자체와 지상 인프라에 미치는 심각하고 치명적인 영향도에 따라 감항 인증 레벨인 DAL(Design Assurance Level)을 A(Catastrophic, 파국적 고장)부터 E(No Effect, 영향 없음)까지 다섯 단계로 엄격하게 분류한다.

독자적인 GCS 애플리케이션 개발 시 전체 소스 코드 100%를 최고 등급(DAL A 또는 B)의 빡빡한 기준으로 관리하는 것은 개발 예산과 스케줄을 고려할 때 거의 불가능에 가깝다. 따라서 GCS 내부 시스템 모듈을 해당 모듈이 내포한 위험도와 제어 권한에 따라 물리적, 논리적으로 파티셔닝(Partitioning)하여 신뢰성 요구사항을 다음과 같이 분리 도출해야 한다.

1.1 -1. DAL A/B 대응 모듈 (생명 위협 및 기체 완전 파괴 직결)

  • 제어 도메인: 비상 제어 명령 발동(Emergency Flight Termination), 모터 즉각 차단(Kill Switch Command).
  • 신뢰성 요구사항: 해당 치명적 명령은 운용 체제(OS)의 메인 UI 렌더링 스레드가 다운(Hang)되거나 백엔드 비즈니스 로직 연산이 과부하에 걸린 먹통 상태에서도 즉각 발동될 수 있어야 한다. 이를 위해 프로세스를 완전히 분리한 상태 모니터링 스레드를 운용하거나, 시스템 아키텍처 상으로 MAVLink 데이터 링크 통로를 이원화(예: 별도의 텔레메트리 하드웨어 채널 활용)하는 하드웨어-소프트웨어 이중화 조치를 강구해야 한다.

1.2 -2. DAL C 대응 모듈 (심각한 부상 위험 또는 기체 인프라 손상)

  • 제어 도메인: 주요 비행 모드 전환(Mission, RTL, Position 제어), 이착륙 명령 트랜잭션 수립, 정밀 웨이포인트(Waypoint) 업로드.
  • 신뢰성 요구사항: GCS 소프트웨어는 MAVSDK 등을 통해 하달한 COMMAND_LONG 데이터 패킷 전송에 대하여 PX4 펌웨어로부터 수신되는 COMMAND_ACK 결과값을 지연 없이 절대적으로 파싱해야 한다. 만약 타임아웃 창(Window) 내에 응답이 없거나 RESULT_FAILED가 수신될 경우, 즉시 무한 대기 상태를 빠져나오고 시스템 로그에 기록함과 동시에 조종자에게 즉각적인 시청각 알람(Audio-Visual Alert)을 제공하여 인지시켜야 한다.

1.3 -3. DAL D/E 대응 모듈 (경미한 운항 불편 정도의 상황)

  • 제어 도메인: 오프라인 지도 타일 렌더링(GIS), 페이로드 카메라 영상 스트리밍 디코딩, 보조 비행 로그 표시.
  • 신뢰성 요구사항: 영상 처리나 지도 렌더링 과정에서 발생하는 메모리 누수(Memory Leak) 현상이나 메모리 폴트(Segmentation Fault)가 DAL A/B/C를 담당하는 메인 제어 모듈 메모리 전반에 파급(Propagation)되지 않도록 철저히 차단해야 한다. 이를 위해서는 기능별로 마이크로서비스 아키텍처(MSA)를 도입하거나, 내부 프로세스 간 분리(IPC 통신 구조) 샌드박스 위에서 이들 보조 기능을 안전하게 격리(Isolation)시켜 띄우는 것이 바람직하다.

2. 양방향 시스템 추적성(Bi-directional Traceability) 정립

DO-178C에서 소프트웨어 신뢰성 검증의 가장 핵심 코어로 요구하는 것은 소스 코드 한 줄 한 줄이 과연 어떠한 고수준 통제 요구사항(High-level Requirements)으로부터 유래하여 설계되었는지를 증명하는 **‘양방향 추적성’**이다.

2.1 -1. MAVLink 통신 무결성(Communication Integrity) 검증의 추적화

  • GCS 통신 계층에서 하부 네트워크 버퍼(UDP/TCP, Serial)로부터 스트림을 읽어올 때, MAVLink 패킷 시퀀스 번호(Sequence Number)의 누락을 100% 감시하고, CRC(Cyclic Redundancy Check) 검증 오류 빈도를 실시간으로 통계화하는 로직 코드는, ’네트워크 패킷 유실률 감시’라는 고수준 요구사항과 1:1로 정확하게 문서상 대응되어야 한다.
  • 예를 들어, 3초 이상 PX4-Autopilot의 HEARTBEAT 토픽 수신이 완전히 실패할 경우, GCS 내부 상태 머신은 명시적으로 “통신 단절(Data Link Loss)” 상태 열거형(Enum)으로 강제 전이되어야 한다. DO-178C A/B 레벨에 준하기 위해, 이 핵심 분기 로직은 반드시 자동화된 단위 회귀 테스트 코드 베이스에 의해 구조적 커버리지(MC/DC, Modified Condition/Decision Coverage) 100%를 달성하여 수학적 증명에 가깝게 보증되어야 한다.

2.2 -2. 정적 분석 및 예외 처리의 캡슐화 방어

  • MAVSDK 기반 C++ 또는 ROS2 호환 백엔드 노드(Node)를 설계할 때, 동적 메모리 할당 실패나 포인터 참조 오류(Null Pointer Exception) 등의 예외 상태가 치명적인 크래시로 이어지는 것을 시스템적으로 원천 차단해야 한다. DO-178C 철학을 준수하기 위해서는 MISRA C++ 등의 코딩 가이드라인을 강제하고, CI/CD 파이프라인에서 정적 분석(Static Analysis) 도구 통과를 소프트웨어 머지(Merge)의 필수 조건으로 편입시켜 런타임 이전의 신뢰성을 극한으로 끌어올려야 한다.

3. 구조적 비결정성(Non-determinism) 타파 설계

DO-178C는 예상치 못한 동작을 유발하는 비결정론적 소프트웨어 동작 알고리즘의 사용을 극도로 경계한다.
Windows나 범용 Linux와 같은 일반 운영체제(General Purpose OS) 위에서 구동되는 GCS는 필연적으로 스레드 스케줄링의 지터(Jitter)로 인한 비결정성에 노출된다.

이를 타파하기 위해, 가장 크리티컬한 실시간 제어 로직 패킷의 발송 간격(예: Offboard 모드의 경로 추종을 위한 SET_POSITION_TARGET_LOCAL_NED의 10Hz 보장 전송 등)은 기저 OS의 불규칙한 타임 틱(Tick) 스케줄링 타이머에만 의존해서는 안 된다. 시스템 구조적으로 내부 타이머 전송의 편차(Deviation)를 1ms 단위로 실시간 모니터링하여, 통신 대기 임계를 넘을 경우 스스로를 안전 대기 모드(Safe-guard)로 폴백(Fallback)시킬 수 있는 ‘자기 진단형(Self-Monitoring) 상태 감시 워치독(Watchdog)’ 아키텍처를 도입하여야 한다.

이와 같이 DO-178C의 철학적 인가 표준을 준용하여 GCS의 주요 모듈을 파티셔닝하고 예외 처리 커버리지를 투명하게 입증하는 일련의 과정은, 추후 군용 납품 및 대규모 B2B 산업 인프라 구축의 신뢰도를 확고히 담보할 수 있는 가장 고도화된 시스템 소프트웨어 엔지니어링 접근법이다.
