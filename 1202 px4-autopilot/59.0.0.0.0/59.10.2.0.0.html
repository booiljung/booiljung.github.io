59.10.2 해시 기반 메시지 인증 코드(HMAC-SHA256)를 적용한 파라미터 쓰기 및 비행 제어 명령의 위변조 방지 무결성 검증

지상 관제 시스템(GCS; Ground Control Station)과 PX4-Autopilot 간의 통신은 무인항공기(UAV; Unmanned Aerial Vehicle)의 안전한 운용을 위해 필수적이다. 통신이 탈취되거나 위변조될 경우, 악의적인 비행 제어 명령이 주입되거나 핵심 파라미터가 임의로 변경되어 치명적인 사고로 이어질 수 있다. 이러한 공격(Spoofing 또는 Replay Attack)을 방어하기 위해 MAVLink 프로토콜은 해시 기반 메시지 인증 코드(HMAC-SHA256; Hash-based Message Authentication Code with SHA-256)를 이용한 위변조 방지 무결성 검증 메커니즘을 지원한다. 본 절에서는 PX4-Autopilot 플랫폼에서 해시 기반 인증 메커니즘을 파라미터 쓰기 및 비행 명령에 어떻게 적용하는지 상세히 분석한다.

1. MAVLink 통신 환경에서의 보안 위협 및 HMAC-SHA256 개요

초기 MAVLink 1.0 프로토콜은 데이터의 신뢰성 검증(CRC 패리티 체크)만을 수행할 뿐, 데이터의 출처(데이터가 인가된 GCS로부터 작성되었는지 여부) 및 위변조 여부는 검증하지 못했다. 이는 악의적인 공격자가 무선 링크에 침투하여 COMMAND_LONG (비행 모드 변경, 시동 온/오프 등) 메시지나 PARAM_SET (파라미터 변경) 메시지를 무단으로 송신할 수 있는 심각한 보안 취약점이다.

MAVLink 2.0 규격부터는 이를 극복하기 위해 메시지 서명(Message Signing) 기능이 도입되었다. 메시지 서명 기능의 핵심은 암호학적 해시 함수인 HMAC-SHA256 알고리즘을 사용하여 메시지의 무결성(Integrity)과 인증(Authentication)을 확보하는 것이다.

HMAC은 송신자와 수신자가 사전에 공유한 비밀 키(Secret Key)와 송신할 원본 메시지를 연산하여 고정된 길이의 서명(Signature) 값을 생성한다. 수신 측(PX4-Autopilot)은 메시지를 수신한 후 동일한 비밀 키로 서명을 계산하여 서명 값이 일치하는지 비교하는 방식으로 위변조 여부를 실시간으로 감지한다.

2. 메시지 서명 아키텍처 및 무결성 검증 메커니즘

MAVLink 2.0의 메시지 서명 구조는 기존 패킷 뒷단에 13바이트의 서명 필드를 추가하는 형태(Appended Signature)로 구현된다. (참조: PX4 Firmware v1.14.0 및 QGroundControl v4.3.x 환경)

2.1 서명 데이터 구조(Signature Data Structure)

서명 필드는 다음과 같이 구성된다.

  • Link ID (1 byte): 통신 링크의 식별자
  • Timestamp (6 bytes): 10 마이크로초 단위의 타임스탬프 (재전송 공격 방어용)
  • Signature (6 bytes): 완전한 HMAC-SHA256 출력(32바이트) 결과에서 추출한 첫 48비트(6바이트)의 값

서명 계산식은 다음과 같이 정의된다.
Signature = \text{HMAC-SHA256}( \text{Secret Key}, \text{MAVLink Header} + \text{Payload} + \text{Link ID} + \text{Timestamp} )

(참고: 수식 내 기호 \vert 는 수식 구조상 생략되었으나, 엄격한 수학적 검증 시 준수한다.)

타임스탬프(Timestamp) 필드는 송신 시점의 시간을 기록하며, 이는 네트워크 스니핑을 통해 패킷을 가로챈 공격자가 나중에 동일한 패킷을 재전송(Replay Attack)하는 것을 방지하기 위해 사용된다. 수신된 메시지의 타임스탬프가 이전에 수신된 패킷의 타임스탬프보다 과거인 경우, 시스템은 해당 패킷을 즉시 폐기(Drop)한다.

    Crypto->>Crypto: 6바이트 Signature 추출 및 패킷 후방 압입
    Crypto-->>GCS: MAVLink 2.0 서명된 패킷 완성
    
    GCS->>FC: 서명된 MAVLink 패킷 무선 송신
    
    Note over FC: 패킷 수신 및 파싱
    FC->>FC: Timestamp 무결성 검증 (Replay Attack 판별)
    alt Timestamp 불량 (과거 시간)
        FC-->>GCS: 패킷 폐기 (보안 정책에 따른 무시)
    else Timestamp 정상
        FC->>FC: 내부 비밀 키와 수신된 데이터를 연산하여 자체 HMAC 도출
        FC->>FC: 자체 계산된 HMAC과 수신된 Signature 필드 비교
        alt Signature 불일치 (위변조 감지)
            FC-->>GCS: 패킷 폐기 분기 처리
        else Signature 일치 (정상 인가 인가)
            FC->>FC: 파라미터 값(SYS_AUTOSTART) 내부 스토리지에 반영
            FC-->>GCS: PARAM_ACK 확인 메세지 반환
        end
    end

3. 파라미터 쓰기 및 제어 명령에 대한 서명 강제(Enforcement) 설정

위변조 공격은 주로 드론을 탈취하거나 오작동을 유발하는 데 목적이 있으므로, 중요도가 낮은 원격 측정(Telemetry) 데이터는 서명이 없어도 허용하되, 기체 상태에 직접적인 영향을 미치는 명령(예: COMMAND_INT, COMMAND_LONG, PARAM_SET, PARAM_EXT_SET)에 대해서만 서명을 강제하도록 설정하는 것이 일반적인 보안 관행이다.

QGroundControl 측의 보안 채널 설정

QGroundControl에서 Setup_Signing (Message ID 256) 프로세스를 통해 비행 제어기 측에 초기 관제용 키(Key)를 설정할 수 있다. QGroundControl은 MAVLink v2 인프라가 활성화되어 있을 때, 사전에 정의된 Passphrase 알고리즘을 바탕으로 암호화 키를 생성하여 PX4 시스템에 배포한다. 이 키가 할당되면 이후 해당 GCS와 PX4간 송수신되는 모든 제어 메시지에 서명이 자동으로 활성화된다.

PX4 소스 코드 수준의 검증 로직 분석

PX4 펌웨어 내의 mavlink_receiver.cpp 모듈은 패킷이 수신되면 MAVLink Parser를 통해 서명 존재 여부를 파악한다. 파라미터 변경이나 비행 제어 명령과 관련된 패킷 수신 시, 서명이 존재하지 않거나 검증에 실패하면 PX4는 즉각적으로 오류를 계측하고 해당 명령을 무시한다. 키의 무결성 저장을 위해 MAVLink::handle_message_setup_signing 함수 영역 내에서 비밀 키와 타임스탬프 이력을 비휘발성 저장소(EEPROM 또는 FRAM)에 기록하여 기체 재부팅 시에도 일관된 세션 보안을 유지한다.

4. 보안 적용 시 공학적 고려 사항 및 Ardupilot 대비 아키텍처 차이

4.1. 보안 연산 리소스와 채널 대역폭 오버헤드

해시 기반 서명은 필연적으로 추가적인 연산 부하를 동반한다. PX4 펌웨어를 구동하는 임베디드 코어(일반적으로 STM32F7, H7 등)에서는 패킷마다 SHA-256 연산이 수반되므로 시스템 CPU 클럭 자원을 소모하게 된다. 또한, 패킷 당 13바이트의 오버헤드가 발생하므로 저대역폭 링크 운용 환경에서는 치명적일 수 있다. 따라서 모든 텔레메트리 데이터에 서명을 적용하기보다는 고위험 제어 명령 패킷에만 선택적으로 강제 검증을 구성하여 CPU 부하와 통신 가용성 간의 적절한 엔지니어링 타협을 도출하는 것이 핵심적이다.

4.2. Ardupilot 대비 통신 보안 아키텍처 차이

PX4-Autopilot과 Ardupilot 모두 MAVLink v2 기반의 암호화 서명 보안 기술을 제공하나, 세부적인 보안 정책 강제 방식에 기술적 차이가 있다.

  • PX4-Autopilot: 최근 버전(v1.14.0 이상)에서는 MAVLink 단독 보안뿐만 아니라 ROS2 미들웨어 브릿지(uXRCE-DDS)를 광범위하게 적용하는 추세이므로, DDS Security(예: SROS2)의 상위 수준 보안 계층과 병목 없이 혼용될 수 있는 하이브리드시스템 아키텍처를 지향한다. 이를 위해 MAVLink 모듈 자체는 최소한의 프로토콜 오버헤드 검증에 초점을 맞춘다.
  • Ardupilot: 전통적으로 취미 기체부터 무인기 산업계 전반에 걸친 넓은 호환성을 중시하여 SEC_ 형태의 전용 파라미터 그룹을 운영한다. 예를 들어 Ardupilot은 서명이 일치하지 않아도 허용할지 등을 결정하는 광범위한 세부 정책(Accept_Unsigned 설정 등)을 펌웨어 단비에서 자체적으로 세밀하게 프로그래밍할 수 있도록 MAVLink 종속적 보안 제어를 상대적으로 다채롭게 제공한다.

5. 결론

다양한 무선 전파 간섭 환경 및 공개된 프로토콜 규격으로 인해 MAVLink 기반 드론 제안 시스템은 항상 데이터 침투 및 스푸핑의 위험에 노출된다. 해시 기반 메시지 인증 코드(HMAC-SHA256)를 적용한 위변조 방지 설계는 악의적인 파라미터 덮어쓰기 행위 및 인가되지 않은 비행 제어권 강탈 공격으로부터 PX4-Autopilot 운용 시스템을 철벽같이 방어하는 가장 근원적이고 필수적인 소프트웨어 방파제이다. 드론 시스템의 자율 운용성이 극대화되고 네트워크 연결성이 점진 확장됨에 따라 제어 명령 권한의 암호학적 검증은 더욱 정교한 통신 신뢰 체계의 기준으로 부상할 것이다.
