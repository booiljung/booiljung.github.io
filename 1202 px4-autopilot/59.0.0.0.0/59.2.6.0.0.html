59.2.6 동적 런타임 확장을 위한 추상 팩토리(Abstract Factory) 패턴 기반의 플러그인 모듈 적재 아키텍처

지상 관제 시스템(GCS, Ground Control Station)은 고정된 형태의 일반 애플리케이션과 달리, 운영 목적(수색, 정찰, 농업 등)이나 탑재된 센서(LiDAR, 열화상 카메라), 신형 기체(VTOL, 고정익 등) 도입 시마다 지속적으로 새로운 기능의 추가가 요구된다. 기존의 모든 코드가 하나의 거대한 실행 파일(Executable)로 묶이는 모놀리식(Monolithic) 아키텍처는 신규 하드웨어 프로토콜을 추가할 때마다 GCS 전체를 재컴파일하고 재배포해야 하는 치명적인 유지보수 문제를 일으킨다. 이를 극복하고 유연성을 확보하기 위해서는 런타임(Runtime) 환경에서 필요한 모듈만 동적 라이브러리(.dll, .so) 형태로 읽어 들이는 ‘플러그인 기반 아키텍처(Plugin-based Architecture)’ 도입이 필수적이다.

본 절에서는 자체 GCS 및 차세대 관제 인프라를 설계할 때 핵심이 되는 ’추상 팩토리(Abstract Factory) 패턴’의 원리를 살펴보고, 이를 응용하여 시스템 재시작 없이도 동적으로 새로운 기능을 확장할 수 있는 플러그인 모듈 적재 메커니즘을 상세히 다룬다.

1. 모놀리식 GCS 아키텍처의 한계점과 동적 확장의 필연성

모놀리식 구조는 초창기 GCS 프로토타입 개발에서는 단순성 때문에 선호되지만, 프로젝트 규모가 커짐에 따라 다음과 같은 한계를 명확하게 드러낸다.

  • 높은 결합도(Tight Coupling): 커스텀 MAVLink 메시지 파서나, 제3자(3rd-party) 업체의 특정 짐벌(Gimbal) 제어 코드가 GCS 코어 비즈니스 로직과 강하게 엮이게 된다.
  • 재배포의 부담: 단 한 줄의 센서 코드를 수정하더라도 수백 메가바이트 크기의 전체 통합 빌드를 클라이언트 장비에 다시 배포해야 한다.
  • 라이선스 문제 격리: GPL 베이스의 오픈소스 모듈과 상용 라이선스를 지닌 독점 기능 모듈이 하나의 프로세스에서 충돌하는 법적 문제를 우회하고 분리 관리하는 데 플러그인 시스템이 활용된다.

따라서 플랫폼으로서의 GCS 코어(Core) 시스템은 최소한의 뼈대와 공통 인터페이스만을 정의하고, 나머지 비즈니스 로직, 페이로드 렌더러, 파서 등은 플러그인 모듈 구동 파일로 완전히 분리해야 한다.

2. 추상 팩토리(Abstract Factory) 패턴을 응용한 객체 생성

2.1 추상 팩토리 패턴 정의

추상 팩토리는 “구체적인 클래스를 명시하지 않고도 서로 연관되거나 의존적인 객체들의 조합을 생성하기 위한 인터페이스를 제공“하는 생성(Creational) 디자인 패턴이다. GCS 플랫폼의 코어는 새로운 장비가 런타임에 로드되었을 때 시스템 모듈이 A사 짐벌 인터페이스인지 B사 짐벌 인터페이스인지 미리 알 필요가 없다. 단지 AbstractPayloadFactory가 제공하는 인터페이스만을 통해 필요한 객체를 넘겨받으면 된다.

2.2 GCS 코어와 플러그인의 인터페이스 추상화

C++ 기반으로 작성된 GCS(QGroundControl의 Plugin 아키텍처 등 참조)에서는 코어와 플러그인 제조사가 반드시 합의해야 할 ‘순수 가상 함수(Pure Virtual Function)’ 기반의 추상 인터페이스 헤더 구조를 선행 설계해야 한다.

// CoreSystemInterfaces.h (GCS Core 엔진과 모듈 개발자가 서로 공유)
class IGCSPlugin {
public:
    virtual ~IGCSPlugin() = default;
    virtual const char* getPluginName() const = 0;
    virtual const char* getPluginVersion() const = 0;
    virtual bool initialize() = 0;
};

// 추상 팩토리: 플러그인 내부의 다양한 하위 통신 요소들을 일관된 형태로 생성 반환
class IComponentFactory {
public:
    virtual ~IComponentFactory() = default;
    virtual IVideoDecoder* createVideoDecoder() = 0;
    virtual IMavlinkParser* createMavlinkParser() = 0;
    virtual ITelemetryWidget* createCustomUI() = 0;
};

의존성 역전 원칙(DIP, Dependency Inversion Principle)에 따라 메인 코어 GCS는 위 추상 인터페이스 헤더에만 의존하며 컴파일되고, 구체적인 .cpp 모듈은 분리된 개발 환경에서 독립적으로 각각 빌드된다.

3. 동적 런타임 모듈 적재(Dynamic Loading) 메커니즘 설계

3.1 동적 라이브러리 심볼 로딩

아키텍처 관점에서 플러그인 모듈은 운영체제가 이해하는 동적 링킹(Dynamic Linking) 방식으로 읽어진다. POSIX 호환 시스템(Linux 환경)에서는 dlfcn.h의 dlopen, dlsym, dlclose를 활용하며, 다중 플랫폼(Windows, macOS)을 지원하는 Qt 기반의 GCS라면 QPluginLoader 객체를 통해 안전하고 일관되게 플랫폼 바이너리를 파싱한다.

플러그인 모듈(.so, .dll)의 내부에서는 컴파일러의 네임 맹글링(Name Mangling) 처리를 피하기 위해 반드시 extern "C" 매크로를 사용하여 팩토리 생성 함수의 심볼을 평문으로 외부에 노출해야 한다.

// 플러그인 바이너리 소스 측 구현체 (ex: CustomPayloadPlugin.so)
extern "C" {
    // 코어 시스템이 dlsym을 통해 이 함수의 메모리 포인터를 찾아 호출
    IComponentFactory* CreateFactoryInstance() {
        return new CustomPayloadComponentFactory();
    }
}

3.2 플러그인 레지스트리(Plugin Registry)와 매니저 구현

비행 관제 애플리케이션 시작 단계이거나 조종자의 ‘모듈 리로드’ 요청이 들어오면, PluginManager 클래스는 특정 디렉토리(예: /usr/lib/gcs/plugins/ 혹은 실행 파일 하위의 plugins 폴더)를 스캔한다.
스캔된 각각의 동적 라이브러리 파일을 dlopen()으로 메모리에 적재한 뒤, CreateFactoryInstance 함수 포인터를 찾아내 팩토리 객체를 인스턴스화하여 중앙 레지스트리 목록(Registry List)에 해시 맵 형태로 보관한다. 이후 시스템은 이 팩토리로부터 필요한 하위 렌더링 뷰나 프로토콜 객체를 생성하여 라우팅 버스에 연결한다.

4. 확장 아키텍처의 안정성(Safety) 제약 규칙 제어 가이드

아주 작은 플러그인의 결함(Null 참조 에러나 메모리 누수)이 추락과 직결된 메인 GCS 코어 프로세스를 Segmentation Fault(코어 덤프) 시키거나 멈춰버리게 할 수 있다.

  1. 메모리 할당-비할당 책임 규칙 (ABI 호환성)
  • 플러그인 내부에서 new로 동적 할당된 객체는 반드시 해당 플러그인의 문맥 안에서 delete되어야 한다. C++ 런타임 라이브러리(CRT)가 다를 수 있으므로, 팩토리 인터페이스에 virtual void destroyInstance(IGCSPlugin* plugin) = 0; 와 같은 삭제 위임 함수를 정의하여 메모리 해제 책임을 플러그인에 명확히 명시해야 한다.
  1. 샌드박싱(Sandboxing)과 안전망 로직
  • 써드파티 플러그인이 로드되었을 때 중요한 코어 객체(예: 조종 스틱 포인터나 MAVLink 송신 버퍼)에 무분별하게 접근하는 것을 막아야 한다. 객체 포인터를 직접 전달하기보다, 59.2.3절에서 설명한 비동기 이벤트 주도(Pub-Sub) 시스템을 통해 메시지 단위로만 통신하도록 인터페이스를 차단(Isolating)하는 패턴이 선호된다.
  1. 가상 소멸자(Virtual Destructor)
  • 자식 클래스의 객체가 부모 인터페이스의 포인터인 상태로 소멸될 때, 메모리 누수 방지를 위해 부모 클래스는 반드시 소멸자를 virtual로 선언해야 정상적으로 메모리가 회수된다.

이러한 추상 팩토리 및 플러그인 아키텍처의 철저한 구축은 차세대 GCS 인프라 플랫폼이 하나의 단일 소프트웨어를 넘어 거대한 ’서드파티 생태계’를 품을 수 있게 하는 가장 결정적인 시스템 공학적 모멘텀이 된다.
