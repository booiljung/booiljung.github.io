59.5.6 임무 항목 배열의 무결성 검증을 위한 상태 머신(State Machine) 기반 트랜잭션 마이크로버스트(Microburst) 송수신 제어 및 교착 상태(Deadlock) 타임아웃 처리

무인 항공기(UAV) 지상 관제 시스템(GCS)과 탑재 비행 제어기(Flight Controller, FCU) 간의 통신은 주로 반이중(Half-duplex) 혹은 패킷 로스(Packet Loss)가 흔히 발생하는 RF 텔레메트리(Telemetry) 무선 네트워크를 통해 이루어진다. 이로 인해 다수의 웨이포인트(Waypoint)로 구성된 임무(Mission) 항목 배열을 기체에 전송하거나, 반대로 기체로부터 배열을 다운로드받을 때 하나라도 데이터가 누락되면 전체 자율 비행의 무결성(Integrity)이 크게 훼손될 수 있다.

본 장에서는 MAVLink 기반 통신망에서 임무 항목 배열의 송수신 신뢰성을 강제하기 위한 상태 머신(State Machine) 설계 기법과, 전송 효율 극대화 및 병목 해소를 위한 트랜잭션 마이크로버스트(Microburst) 제어, 그리고 무한 대기(Deadlock)를 방지하는 타임아웃(Timeout) 아키텍처에 대해 상세히 다룬다.

1. MAVLink 미션(Mission) 프로토콜의 한계와 트랜잭션 구조

MAVLink 프로토콜 상에서 GCS가 기체로 임무를 업로드하는 기본 트랜잭션(Transaction)은 “1-Way Request, 1-Way Reply” 형태의 핑퐁(Ping-pong) 구조를 가진다.

  1. GCS: MISSION_COUNT (전송할 총 웨이포인트 수 N개 전송)
  2. PX4: MISSION_REQUEST_INT (0번 인덱스 요구)
  3. GCS: MISSION_ITEM_INT (0번 데이터 송신)
  4. PX4: MISSION_REQUEST_INT (1번 인덱스 요구)
  5. GCS: MISSION_ITEM_INT (1번 데이터 송신)
    … (반복) …
  6. PX4: MISSION_ACK (수신 성공 메시지 발송)

이러한 구조는 확실한 동기화를 보장하지만, 무선 링크의 지연(Latency)이 100ms만 되어도 100개의 웨이포인트를 올리는데 수십 초의 막대한 지연 릴레이가 발생한다는 치명적인 단점이 있다.

2. 마이크로버스트(Microburst) 송수신 제어를 통한 고속화 플래닝

이러한 송수신 지연을 회피하기 위해, MAVLink 프로토콜의 표준 규격을 위반하지 않으면서도 전송 효율성을 높이는 방법 중 하나가 마이크로버스트 기법이다. (단, 펌웨어 버전 및 GCS 종류에 따라 지원 여부가 달라진다.)

마이크로버스트 송신의 핵심은 GCS 측에서 기체의 상태를 예측하여 MISSION_REQUEST_INT를 기다리지 않고, 허용된 윈도우(Window) 사이즈 내의 여러 개의 MISSION_ITEM_INT 패킷을 짧은 시간 단위에 폭발적으로 연달아 쏟아붓는 것이다. PX4-Autopilot 수신 버퍼는 순서가 어긋나더라도 이를 캐싱(Caching)해 두고 누락된 시퀀스 번호만을 선별하여 다시 MISSION_REQUEST_INT를 통해 요청(NACK 역할)할 수 있도록 설계된다.

3. 유한 상태 머신(FSM) 설계와 로직 흐름

독자적인 GCS를 개발할 때, 이러한 복잡한 비동기(Asynchronous) 송수신 이벤트들을 안전하게 관리하려면 백엔드(C++) 코드에 엄격한 상태 머신(State Machine)을 배치해야 한다.

stateDiagram-v2
    [*] --> IDLE
    
    IDLE --> SENDING_COUNT : Upload 명령 실행
    
    SENDING_COUNT --> WAITING_REQUEST : MISSION_COUNT 송신
    WAITING_REQUEST --> SENDING_ITEM : MISSION_REQUEST 수신
    
    SENDING_ITEM --> WAITING_REQUEST : 패킷 송신 (또는 Microburst)
    SENDING_ITEM --> TIMEOUT_RETRY : 응답 지연 발생 (Timer Triggered)
    
    WAITING_REQUEST --> UPLOAD_COMPLETE : MISSION_ACK 수신
    UPLOAD_COMPLETE --> IDLE
    
    TIMEOUT_RETRY --> WAITING_REQUEST : 재요청 발송
    TIMEOUT_RETRY --> FAILED : 최대 재시도(Max Retry) 횟수 초과
    FAILED --> IDLE : 에러 팝업 발생

위 상태 머신 모델에서는 어떠한 상태(State)에서라도 패킷 유실로 인해 멍청하게 기다리는 상황(Deadlock)이 없도록 각 상태로 전이할 때 타이머(Timer)를 시동시킨다.

4. 교착 상태(Deadlock) 해소 및 타임아웃(Timeout) 롤백 아키텍처

임무 송수신 중 GCS와 PX4 둘 중 한쪽이라도 다음 시퀀스를 망각하거나 텔레메트리 연결이 단절되면 “가짜 완료(False Complete)“나 “무한 대기(Deadlock)“에 빠진다.

  • 타임아웃(Timeout) 파라미터: MAVLink 네트워크의 특성을 고려하여 약 250ms ~ 500ms의 타임아웃을 적용한다.
  • 재전송(Retry) 백오프 메커니즘: 만약 타임아웃이 발생하면, 현재 상태에 해당하는 패킷을 다시 날린다. 이 짓을 N회(보통 3~5회) 초과 시도하게 되면 전체 트랜잭션을 강제 취소(Abort) 처리하고 무결성 실패(Integrity Failed) 알람을 사용자에게 노출시킨다.
  • QGroundControl의 구현 사례: QGC C++ 구조체 안의 PlanManager 혹은 MissionManager 코드를 열어보면 _ackTimeoutTimer 타이머 객체를 사용하여, 특정 기간 내 응답이 없으면 Retry 후 연결 자체를 리셋하는 코드가 촘촘하게 배열되어 있다.

5. Ardupilot 대비 PX4-Autopilot의 트랜잭션 핸들링 차이

  • Ardupilot의 경우 MISSION_ITEM 기반의 예전 MAVLink 프로토콜과 MISSION_ITEM_INT 기반 최신 프로토콜 모두 유연하게 대응하지만, 경우에 따라 GCS가 쏘는 Microburst 속도를 따라가지 못해 MISSION_REQUEST 루프가 심하게 꼬이는 현상(Mission Upload Timeout)이 자주 발생한다.
  • 반면 PX4-Autopilot은 uORB 메시징 구조상 내부 버퍼의 분리가 확실히 되어 있어 고속의 플로우 컨트롤(Flow Control)에 상대적으로 덜 취약하다. 그러나 PX4 역시 기동 중(Armed)에 수백 개의 웨이포인트를 덮어씌우는 동적 할당 과정에서는 CPU 병목 현상이 생길 수 있으므로, 반드시 교착 상태 브레이커(Breaker)를 구현해 두는 것이 권장된다.

6. 결론

무인 시스템 개발자는 자율 비행 알고리즘의 우수함 이상으로 GCS와 FCU 사이의 MAVLink 데이터 신뢰성 확보에 집착해야 한다. 단절과 유실을 기본 전제로 깔고 진입하는 “낙관적이지 않은 상태 마신(Pessimistic State Machine)“의 도입은 GCS 품질 제고의 핵심이다. 송수신의 마이크로버스트 튜닝과 정밀한 타임아웃 로직은 무결성이 무너지는 치명적 사고(Flyaway 등)를 예방하는 가장 훌륭한 시스템 방패가 된다.
