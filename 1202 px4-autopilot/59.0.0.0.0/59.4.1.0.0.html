59.4.1. 웹 맵 타일 서비스(WMTS) 캐싱 계층 아키텍처 및 오프라인 공간 데이터베이스(Spatial Database) 쿼리 최적화

지상 관제 시스템(GCS)은 무인항공기(UAV) 운영 환경의 특성상 LTE/5G 통신망이 도달하지 않는 오지(Remote Area)나 전시 상황 등 오프라인(Offline) 환경에서의 관제를 보장해야 한다. 이를 위해서는 외부 인터넷망(API)에 의존하지 않고 사전에 구축된 지도 데이터를 GCS 내부에 저장하여 실시간으로 렌더링할 수 있는 독립적인 웹 맵 타일 서비스(Web Map Tile Service, WMTS) 캐싱 아키텍처와 오프라인 공간 데이터베이스 쿼리의 고도화가 필수적이다. 본 장에서는 QGroundControl(이하 QGC)의 지도 캐싱 메커니즘을 벤치마킹하여 성능을 극대화할 수 있는 커스텀 공간 데이터베이스 최적화 기법을 다룬다.

1. WMTS(Web Map Tile Service) 규격과 타일 피라미드 시스템

지도 엔진은 지구 전체를 하나의 거대한 해상도로 렌더링하는 대신, 사용자 시야(View Port)와 줌(Zoom) 레벨에 따라 지도를 잘게 쪼갠 타일(Tile) 단위로 호출하여 메모리 점유율을 줄인다. 이는 WMTS 또는 TMS(Tile Map Service)의 핵심 원리이다.

  • Zoom 레벨에 따른 피라미드 구조: 지구의 전체 지도는 Zoom Level 0에서 256 \times 256 픽셀짜리 타일 1장으로 표현된다. Zoom Level Z에서의 타일 개수는 4^Z개가 된다.
  • 투영(Projection): 대부분의 상용 지도(Google, Mapbox, OSM) 및 PX4의 표준 타일 서비스는 Spherical Mercator(EPSG:3857) 투영법을 사용하며, 이는 위도와 경도를 타일의 X (동서 방향) 및 Y (남북 방향) 인덱스로 변환하는 기반이 된다.

2. GCS 캐싱 계층(Caching Layer) 아키텍처

독자적인 GCS는 지도 타일 데이터를 메모리 레벨, 로컬 디스크 레벨, 네트워크 패치(Fetch) 레벨의 다중 계층 아키텍처(Multi-tier Architecture) 로 설계해야 한다.

  1. L1 캐시 (Memory Cache): 가장 최근에 조회했거나 현재 화면에 렌더링 중인 타일을 저장한다. LRU(Least Recently Used) 알고리즘을 통해 큐(Queue)를 관리하며, 화면 스크롤 시 지연 시간(Latency) 없이 매끄러운 패닝(Panning) 효과를 제공한다.
  2. L2 캐시 (Offline Spatial Database): 이전 비행에서 로드했거나 GCS 운영자가 비행 전(Pre-flight) 오프라인 다운로드 도구를 통해 저장해 둔 정적 SQLite 데이터베이스이다.
  3. L3 네트워킹 플러그인 (Tile Fetcher): L1과 L2 캐시에 조회 대상 타일이 없는 경우(Cache Miss)에 한정하여, 비동기(Asynchronous) HTTP/HTTPS 요청을 통해 온라인 서버에서 타일을 내려받고 이를 L1과 L2에 저장한다.

2.1 QGroundControl의 SQLite 기반 타일 구조 타산지석

QGC는 내부적으로 QGeoTileFetcher 클래스를 상속받아 타일을 다운로드하며, QGCCacheWorker를 통해 오프라인 데이터베이스 파일(QGCMapCache.db)에 타일을 BLOB 타입으로 저장한다. 이와 동일하게 독자 GCS에서도 경량 파일 기반 RDBMS인 SQLite를 활용하는 것이 시스템 이식성과 성능 면에서 유리하다.

3. 오프라인 공간 데이터베이스(Spatial Database) 모델링

단순한 Key-Value 저장소가 아닌 관계형 데이터베이스(RDBMS)를 통해 오프라인 타일을 관리할 때 구조적 최적화가 필수적이다.

3.1 DDL 구조 설계

성능 집약적인 타일 조회를 위해 지도의 제공자(Provider), 맵의 유형(Map Type), 그리고 타일 좌표(Zoom, X, Y)를 결합한 복합 인덱스(Composite Index) 기반의 테이블 스키마가 요구된다.

CREATE TABLE IF NOT EXISTS MapTiles (
    tile_id INTEGER PRIMARY KEY AUTOINCREMENT,
    hash TEXT UNIQUE NOT NULL,       -- Provider + MapType + Zoom + X + Y의 HASH
    provider_id INTEGER NOT NULL,    -- 예: 카카오맵, 구글맵, Vworld 등
    map_type INTEGER NOT NULL,       -- 예: 위성(Satellite), 일반(Street), 지형(Terrain)
    zoom_level INTEGER NOT NULL,
    tile_x INTEGER NOT NULL,
    tile_y INTEGER NOT NULL,
    format TEXT NOT NULL,            -- 'png', 'jpg' 등
    tile_data BLOB NOT NULL,         -- 바이너리 이미지 데이터
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- B-Tree 방식의 다중 컬럼 인덱스를 통해 O(log N) 조회 속도 최적화
CREATE INDEX idx_tile_lookup ON MapTiles(provider_id, map_type, zoom_level, tile_x, tile_y);
CREATE INDEX idx_timestamp ON MapTiles(timestamp);

3.2 타일 해싱(Hashing) 최적화 기법

지속적인 INSERT와 SELECT 연산에 의한 SQLite 단편화를 막기 위해 (provider, type, z, x, y) 값을 결합한 후, FNV-1a와 같은 속도가 매우 빠르고 충돌율(Collision Rate)이 낮은 비암호학적 해시(Non-cryptographic Hash) 알고리즘을 돌려 64비트 정수형 기반의 hash 컬럼을 생성해 Primary Key 대용으로 활용하면 서치 성능이 급격히 향상된다.

4. 데이터베이스 쿼리 쓰기/읽기 최적화 및 동시성(Concurrency) 관리

오프라인의 Z=18 이상의 고해상도 타일을 수천 장 단위로 긁어오면(Tile Scraping), GCS의 I/O 리소스가 고갈되고 관제 스레드(차량 텔레메트리 파싱)의 프리징(Freezing) 현상을 야기할 수 있다.

4.1 WAL(Write-Ahead Logging) 저널링 모드 활성화

SQLite의 디폴트 저널 모드인 DELETE 방식은 트랜잭션 도중 DB 전체에 락(Lock)을 건다. MAVLink 데이터 처리가 병렬로 수행되는 GCS 환경에서는 읽기가 쓰기를 블록(Block)하지 못하게 SQLite의 연결 설정에서 쿼리 최적화를 위해 WAL 모드를 선언해야 한다.

PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA cache_size = -64000; -- 메모 메모리에 64MB 캐시 유지

4.2 대량 청크 단위 트랜잭션(Bulk Transaction) 삽입

사용자가 오프라인 맵 패키지를 구성하여 다운로드할 때 단일 INSERT 문을 매번 호출하면 파일시스템 I/O 트러블이 발생한다. 쿼리 병목 지연을 줄이기 위해 C++ 백엔드 코드에서 트랜잭션을 일괄 처리해야 한다.

db.exec("BEGIN TRANSACTION;");
for (const auto& tile : tile_download_queue) {
    // BLOB 바인딩 및 INSERT
    stmt.bind(1, tile.hash);
    // ... data ... //
    stmt.execute();
}
db.exec("COMMIT;");

4.3 공간 부족에 대비한 LRU 가비지 컬렉션(Garbage Collection)

고해상도 위성 지도의 타일(Tile) 크기는 장당 20KB~50KB이다. 이 DB가 수 GB를 넘어가면 하드웨어 포터빌리티(태블릿 등) 제약이 생기므로, 스토리지 한계(예: 2GB) 도달 시 가장 오래된(ORDER BY timestamp ASC) 타일 레코드를 백그라운드에서 삭제하는 SQL 배치 쿼리가 실행되도록 백그라운드 Worker 스레드를 구성하여야 한다.

5. 결론

PX4-Autopilot 기반의 무인기 미션을 안정적으로 관제하기 위한 오프라인 지도 캐싱 시스템은 단순한 네트워크 호출의 대체가 아니며, 파일시스템 I/O, SQLite DB 최적화 기술, 병렬 트랜잭션 구조가 융합된 데이터 엔지니어링의 결과물이다. WAL 저널링 방식과 다중 복합 인덱싱을 이용한 WMTS 쿼리 최적화는 어떠한 폐쇄망 환경에서도 지도 랜더링의 무결성과 퍼포먼스를 유지할 수 있도록 돕는다.
