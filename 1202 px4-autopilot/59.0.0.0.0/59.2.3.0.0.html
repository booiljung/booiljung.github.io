59.2.3 발행-구독(Publish-Subscribe) 모델 기반의 비동기 이벤트 주도 아키텍처(Event-Driven Architecture) 통신 버스 설계

1. 개요

하나의 통합 지상 관제 시스템(GCS) 내부에는 수십 가지 이상의 이질적인 목적을 띤 컴포넌트들이 맞물려 구동된다. 고도와 속도를 파싱하는 MAVLink 스레드, 화면 상에 3D 기체 형상을 렌더링하는 UI 엔진, 배터리가 일정 수준 미만으로 떨어졌을 때 음성 합성(TTS) 경고를 내보내는 알람 데몬, 임무(Mission) 진행 현황을 로컬 디스크에 기록하는 I/O 모듈 등 그 종류가 방대하다.

이러한 모듈들이 상호 작용을 위해 서로를 직접 호출(Direct Method Call)하는 방식으로 결합되어 있다면 의존성 그래프(Dependency Graph)는 최악의 거미줄 구조(Spaghetti Code)가 된다. 어느 한 컴포넌트를 교체하거나 수정하려면 얽혀있는 전체 코드베이스를 뜯어고쳐야 하므로 고신뢰성 달성이 불가능해진다. 이 치명적 한계를 극복하기 위해 다수의 백엔드 모듈 및 UI 렌더러 간의 의존성을 완전히 제거하고 메시지 단위로 결합도를 낮추는 **이벤트 주도 아키텍처(EDA, Event-Driven Architecture)**의 통신 버스(Bus) 설계가 필수적이다.

2. 이벤트 주도 아키텍처(EDA)의 핵심 원리

이벤트 주도 아키텍처는 시스템 내에서 상태 변화, 즉 **‘유의미한 사건(Event)’**이 발생했을 때 이를 중앙의 메시지 버스 채널에 던져놓고(발행, Publish), 해당 이벤트에 사전에 관심이 있다고 등록한 로직들만 이를 가져가(구독, Subscribe) 알아서 비동기적으로 처리하도록 내버려 두는 철학이다.

  1. 익명성 보장: 이벤트를 발행하는 송신자는 이 메시지를 누가, 몇 개의 객체가 받아 처리할지 알 필요가 없으며 심지어 수신자가 하나도 존재하지 않아도 정상 작동한다.
  2. 비동기 격리 (Asynchronous Isolation): 스레드 블로킹으로부터 서로를 격리한다. MAVLink 파서가 텔레메트리 업데이트 이벤트를 발행하고 즉시 다음 패킷 처리를 위해 즉시 돌아가는 동안, 지도 렌더러는 별도의 스레드에서 해당 이벤트를 구독하여 자기 페이스에 맞춰 지도를 움직인다.

3. 중앙 통신 이벤트 버스(Event Bus)의 물리적 설계

독자적인 GCS는 핵심 인프라로서 싱글톤(Singleton) 혹은 의존성 주입(DI, Dependency Injection) 패턴으로 관리되는 중앙 이벤트 버스(Event Bus) 객체를 내장해야 한다.

3.1 토픽(Topic) 기반 이벤트 식별 채널링

이벤트 객체들은 명확한 식별자(Topic String 또는 Enum)를 가져야 채널링이 가능하다.

  • TOPIC_VEHICLE_ARMED: 기체 시동 활성화 상태 전환 통보 (구독자: 비행 시간 타이머, 비디오 레코딩 시작 데몬, UI 상태 표시기).
  • TOPIC_CRITICAL_BATTERY: 임계 배터리 전압 도달 (구독자: RTL 자동 귀환 커맨더, TTS 경고 알람러).
  • TOPIC_DATA_LINK_LOST: GCS와 무인기 간 하트비트 타임아웃 발생 (구독자: 연결 해제 UI 팝업 시스템, 재접속 시도 스레드).

3.2 락 프리 큐(Lock-Free Queue)와 이벤트 디스패처 (Event Dispatcher)

다중 스레드(Multi-Thread) 환경의 GCS에서 이벤트 처리는 자원 경합(Race Condition)을 유발할 수 있다. 발행자가 무수히 쏟아내는 MAVLink 기반 이벤트를 중앙 버스가 소화하기 위해서는 내부적으로 락 프리 큐(Lock-free Concurrent Queue) 자료구조를 적용하여 지연(Locking Latency) 인자를 제거해야 한다.
스레드 풀(Thread Pool)의 워커(Worker) 스레드들은 이 큐를 지속적으로 소비하여, 등록된 구독자의 콜백 함수를 비동기식으로 디스패치시킨다. 이는 Node.js의 이벤트 루프(Event Loop) 철학과 완전히 궤를 같이한다.

4. 아키텍처 흐름 및 시퀀스 다이어그램

이 설계 패러다임은 놀랍게도 PX4 내부의 실시간 운영체제(RTOS) 구동 기반인 uORB(Micro Object Request Broker) 미들웨어 구조와 철학적으로 대칭을 이룬다. 기체 내부 컴포넌트 간 비동기 소통이 uORB로 이룩되듯, GCS 내부 컴포넌트 간 통신은 Event Bus로 이루어지는 구조다.

5. 결론

소프트웨어 아키텍처가 거대해질수록, 모듈 간의 유연함을 담보하는 일방향적이고 명시적인 통신 규칙의 부재는 필연적으로 유지보수성을 파괴한다. GCS 내부에 발행-구독(Publish-Subscribe) 기반의 비동기 이벤트 주도 로직을 설계함으로써, 우리는 새로운 UI 위젯, 새로운 센서 모니터링 데몬, 심지어 외부 연동 애플리케이션 프레임워크가 추가되더라도 기존 GCS 데이터 수신 코드를 단 한 줄도 수정하지 않고 시스템을 끝없이 확장시킬 수 있는 엔터프라이즈 통합 플랫폼을 손에 넣을 수 있게 된다. 이는 QGroundControl과 Ardupilot용 Mission Planner를 뛰어넘는 신뢰성을 확보하기 위한 백엔드 설계의 정수이다.
