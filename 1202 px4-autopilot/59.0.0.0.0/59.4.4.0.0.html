59.4.4 고도 맵핑(DEM) 데이터를 활용한 3차원 지형 표고 재구성 및 이중 선형 보간법(Bilinear Interpolation) 기반 표고 평활화

1. 개요 및 필요성

현대의 무인항공기(UAV) 지상 관제 시스템(GCS) 및 비행 제어기(Flight Controller)는 임무 수행의 안전성을 확보하기 위해 지형 추적(Terrain Following) 및 지형 유지(Terrain Hold) 비행 기능을 필수적으로 제공해야 한다. 이를 안정적으로 구현하기 위해서는 고도 맵핑(Digital Elevation Model, DEM) 데이터를 기반으로 한 연속적이고 정밀한 3차원 지형 표고 재구성(3D Terrain Elevation Reconstruction) 과정이 선행되어야 한다.

PX4-Autopilot(펌웨어 v1.14 이상 권장)과 QGroundControl(QGC v4.3 이상) 환경에서는 다양한 위성 및 항공 측량 소스(SRTM, ASTER 등)로부터 획득한 이산적인(Discrete) 그리드형 DEM 데이터를 활용한다. 데이터가 이산적인 특성을 갖기 때문에 비행체가 이동할 때 인접한 격자점 사이의 불연속적인 타일 경계와 낮은 공간 해상도(Spatial Resolution)로 인해 고도 데이터에 단차(Step) 현상이 발생하게 된다. 이를 완화하기 위해 본 장에서는 **이중 선형 보간법(Bilinear Interpolation)**을 적용하여 표고 평활화(Elevation Smoothing)를 수행하는 수학적 원리와 소프트웨어 아키텍처를 심도 있게 분석한다. 본 장은 독자적인 GCS 개발과 자율 에이전트 비행 제어 기술 구현을 목표로 하는 GCS 개발자와 펌웨어 엔지니어에게 최적화된 자료를 제공한다.

2. 고도 맵핑(DEM) 데이터 아키텍처의 이해

디지털 표고 모델(DEM)은 특정 지리적 좌표(WGS84 좌표계 기준 위도와 경도)에 대한 해발 고도(AMSL, Above Mean Sea Level) 값을 저장하는 래스터(Raster) 기반 그리드 맵이다.

2.1 Ardupilot과 PX4-Autopilot의 지형 데이터 처리 구조 비교

전통적인 Ardupilot 생태계의 GCS(주로 Mission Planner) 환경에서는 지형 데이터 패키지 파일 형태를 통째로 비행 제어기와 연결된 하드웨어의 SD 카드에 직접 옮기거나, 지상관제소가 지속적으로 지형 정보를 푸시(Push)하는 직관적인 구조가 오랜 기간 채택되어 왔다.

반면, 현대적인 PX4-Autopilot 환경 및 QGC 생태계에서는 지형 서버(Terrain Server) 모델을 도입한 MAVLink 기반 실시간 타일(Tile) 캐싱 시스템 아키텍처를 사용한다. GCS 엔진부는 Mapbox나 AGI 등의 온라인 맵 공급자로부터 지형 높이 패치(Height Patch) 데이터를 동적으로 다운로드 받아 로컬 SQLite 캐시 데이터베이스에 공간 데이터를 효율적으로 저장한다. 이후 비행 중인 PX4 기반 무인기가 TERRAIN_REQUEST (MAVLink MSG ID 133) 메시지를 필요 시 브로드캐스트하면, GCS의 지형 서비스 매니저가 이를 구독(Subscribe)하고 TERRAIN_DATA (MSG ID 134) 메시지를 통하여 요구된 영역의 이산적 표고 격자 배열만을 비행체로 비동기 송신하게 된다. 이러한 비동기형 MAVLink 클라이언트-서버 구조를 통해 텔레메트리 모뎀의 좁은 데이터 통신 대역폭 환경에서도 지연율(Latency) 오버헤드를 줄이면서 거대한 3차원 지형 표고를 안정적으로 보급받을 수 있다.

3. 이중 선형 보간법(Bilinear Interpolation) 알고리즘 수학적 설계

수신된 이산적인 2차원 DEM 타일 격자 데이터를 기반으로, 비행체 내부 또는 실시간 경로 계획(Path Planning) 알고리즘 구동부에서 임의의 WGS84 연속 좌표 위치에 대한 기준 표고를 얻기 위해서는 공간적 보간 계산이 핵심적인 역할을 수행한다. 비행체의 궤적이 두 고도 타일 격자를 자연스럽게 횡단할 때 고도 제어기가 튀는 현상을 수학적으로 억제하기 위하여, 수학적으로 X축(국소 좌표계의 Easting 방향)과 Y축(국소 좌표계의 Northing 방향)의 2차원 평면에서 연속성을 부여하는 이중 선형 보간법이 널리 채택된다.

3.1 수식적 정식화(Mathematical Formulation)

지도 위를 덮고 있는 네 개의 인접한 DEM 이산 격자점을 Q_{11}=(x_1, y_1), Q_{12}=(x_1, y_2), Q_{21}=(x_2, y_1), Q_{22}=(x_2, y_2)라 하고 각 격자 위치에 사상된 해발 고도 스칼라 값을 h(Q_{ij})로 정의한다. 이 데이터 평면 내부에서 비행체가 점유하고 있는 임의의 점 P=(x,y)의 연속적인 평활화 고도 h(x,y)는 X축에 대한 두 번의 독립적 1차 선형 보간 과정을 거친 후, 도출된 두 결과값을 Y축을 기준으로 다시 1차 선형 보간함으로써 유도된다.

X방향(경도 또는 Easting 최적 축) 선형 보간 모델:

  • h(x, y_1) \approx \frac{x_2 - x}{x_2 - x_1} h(Q_{11}) + \frac{x - x_1}{x_2 - x_1} h(Q_{21})
  • h(x, y_2) \approx \frac{x_2 - x}{x_2 - x_1} h(Q_{12}) + \frac{x - x_1}{x_2 - x_1} h(Q_{22})

마지막으로 Y방향(위도 또는 Northing 최적 축) 선형 보간으로 최종 평활화된(Smooth) 고도 모델을 산출한다:

  • h(x,y) \approx \frac{y_2 - y}{y_2 - y_1} h(x,y_1) + \frac{y - y_1}{y_2 - y_1} h(x,y_2)

이러한 이중 선형 보간 알고리즘의 시간 복잡도는 O(1)으로 제어 루프 진입에 필요한 비용이 매우 낮다. 따라서, 자원 소모와 인터럽트 응답성에 민감한 PX4의 실시간 운영 체제(NuttX RTOS) 기반 Task 스케줄링 환경과 독자적 GCS의 오프라인 맵핑 파이프라인(Mapping Pipeline)에서 모두 CPU 오버헤드 징후 없이 50Hz ~ 250Hz 이상의 고주파수 지형 평활화 역기능학적 연산을 충분히 실시간 수행할 수 있다는 구조적 타당성을 갖는다. 관련 연구 문헌인 “Real-time 3D Terrain Reconstruction and Smoothing for Autonomous UAV Navigation” 에서도 계산 비용 효율성과 고도 추종 성능 궤적 사이의 탁월한 균형점을 긍정적으로 평가하고 있다.

4. 소스 코드 구현 및 센서 데이터 융합 아키텍처

4.1 C++ 기반 표고 평활화 파이프라인 개념

독자 GCS의 TerrainQuery 처리부를 설계하거나, PX4 펌웨어 환경(예: src/modules/ekf2 내 지형 추정기(Terrain Estimator))을 세밀하게 수정할 때 앞서 정의한 이중 선형 보간 알고리즘이 C++로 변환되어 실장된다. 다음은 GCS 측면에서 요구되는 이중 선형 표고 평활화 C++ 인터페이스의 개념적 작성 예시이다.

// GCS/PX4 환경에서의 고도 데이터 이중 보간 파이프라인 예시 (C++17)

struct DemGridInfo {
    double lat_0, lon_0;      // 앵커 기준점 좌표계 (Q11)
    double grid_spacing_deg;  // 격자 공간 분해능(간격)
    float elevations[2][2];   // Q11, Q12, Q21, Q22의 표고 높이 정보 매트릭스 (미터 단위)
};

float calculateBilinearElevation(const DemGridInfo& grid, double target_lat, double target_lon) {
    // 1. 목표 좌표계가 격자 내부에서 차지하는 상대적 위치 비율(0.0 ~ 1.0) 산출
    double dx = (target_lon - grid.lon_0) / grid.grid_spacing_deg;
    double dy = (target_lat - grid.lat_0) / grid.grid_spacing_deg;

    // 2. 부동소수점 오차나 외부 요인에 대비한 Boundary Check 루틴 방어 프로그래밍
    dx = std::clamp(dx, 0.0, 1.0);
    dy = std::clamp(dy, 0.0, 1.0);

    // 3. X 축(가로) 방향 지형 보간 적용
    float h1 = (1.0f - dx) * grid.elevations[0][0] + dx * grid.elevations[1][0];
    float h2 = (1.0f - dx) * grid.elevations[0][1] + dx * grid.elevations[1][1];

    // 4. 추출된 두 평면에 대하여 Y 축(세로) 방향 최종 선형 보간 후 반환
    return static_cast<float>((1.0 - dy) * h1 + dy * h2);
}

예제에서 다룬 보간 로직은 PX4-Autopilot 펌웨어, MAVLink 네트워크 브리지, 그리고 맞춤형 사용자 GCS까지 데이터가 흐르는 파이프라인 상의 모든 에이전트 노드에 수학적 기반으로 통일되게 설계해야 고도 오차를 극소화 가능하다.

4.2 ROS2(Micro XRCE-DDS) 연동과 자율 경로 계획 설계 제안

최신 PX4 환경과 같이 ROS2 프레임워크와 결합된 구조 하에서는 Micro XRCE-DDS 브리지를 통해 기체의 동역학적 좌표 스냅샷인 VehicleLocalPosition 등의 토픽 아키텍처가 동기화된다.

독자적으로 구축된 GCS 프레임워크에서 오프라인 DEM 메타데이터를 기반으로 이중 선형 보간을 활용하여 넓은 3차원 표고 지형 맵(Map)을 계산해내면, ROS2 환경의 글로벌 최적화 목적 경로 생성 노드(Global Path Planner)에서는 해당 지도를 참조하여 궤적을 계획한다. 또한, 비행 중인 PX4 펌웨어 측에서는 전방 혹은 하향 거리 측정 센서(LiDAR) 데이터를 추가로 수집한 후 센서 퓨전(Sensor Fusion) 알고리즘인 EKF2 스택과 연계 처리한다. 이를 통해 예측된 DEM의 수학적 평활화 고도 모델값과 실시간 거리 레이저/레이더 센서의 반사 물리 표고 데이터를 함께 융합 분석함으로써, 더욱 정밀하고 신뢰성 높은 3차원 장애물 회피(Obstacle Avoidance) 루틴 및 지형 제어 추종 신호를 도출하는 모던(Modern) 기술 구조를 갖출 수 있게 된다.
