20.2 로그 및 진단 데이터 분석

시스템이 병들었을 때 내지르는 가장 정직하고 직접적인 비명은 로그(Log)다. 하지만 분산 네트워크 환경에서 로그를 다루는 것은 완전히 다른 차원의 난제다. 수십 개의 공장 존(Zone)에 흩어진 10,000대의 마이크로컨트롤러, 그 데이터를 모으는 100대의 안개(Fog) 라우터, 그리고 클라우드의 관제 서버들이 초당 수 메가바이트의 텍스트를 파편적으로 배출하기 때문이다.

단일 개발 환경(IDE)의 터미널 창에 뜨는 에러 메시지 한 줄을 보고 디버깅하던 습관을 버려라. 분산 시스템에서는 에러가 발생한 지점(A 로봇)과 그 에러로 인해 치명적 타격을 받는 지점(B 클라우드 서버)이 수백 킬로미터 떨어져 있을 수 있다. 거대한 건초 더미에서 바늘을 찾기 위해, 우리는 로그가 내뿜는 ‘노이즈(Noise)’ 속에서 진짜 ’신호(Signal)’를 걸러내는 투시력을 가져야 한다.

이 절에서는 Zenoh 생태계가 쏟아내는 수많은 진단 데이터를 통제하는 법을 명세한다. 라우터와 클라이언트의 tracing 레벨을 동적으로 튜닝하여 평시에는 시스템 부하를 줄이고 장애 시에는 현미경처럼 들여다보는 기술을 익혀라. Zenoh 데몬이 뱉어내는 암호문 같은 에러 코드들을 기계적으로 해석하는 사전(Dictionary)을 제공하며, 파편화된 로그를 Elasticsearch나 Grafana Loki 같은 중앙 로그 저장소로 한데 묶어(Aggregation) 시계열 메트릭(Metric)과 교차 검증하는 인프라 수준의 관제 전략을 완성할 것이다.

1. Zenoh 라우터 및 클라이언트 로그 레벨 제어: Trace, Debug, Info, Warn, Error

Zenoh 네트워크 데몬(zenohd)과 공식 Rust 클라이언트 라이브러리는 내부적으로 Rust의 tracing 프레임워크를 사용하여 고도로 구조화된(Structured) 로그를 방출한다. 로그 레벨은 단순한 필터가 아니라, 라우터의 CPU 성능과 디스크 I/O를 깎아먹는 양날의 검이다.

1.0.1 운영 및 디버깅 런타임 로그 레벨 설정

zenohd 기동 시 RUST_LOG 환경 변수는 시스템의 가시성을 완전히 통제한다.

  • RUST_LOG=error: 치명적인 크래시 부위나 스토리지의 하드 디스크 I/O 강제 차단 등 시스템 생존에 직결된 알람만 배출한다. 평상시 극단적인 성능 최적화가 필요한 라즈베리 파이 제로(Zero) 등 에지 기기에서 사용하라.
  • RUST_LOG=info: (권장 기본값) 세션의 연결(Session Open) 및 종료(Close), 플러그인 로드 상태, 주요 스플릿 브레인 복구 등 라이프사이클 이벤트를 기록한다. 전역 트래픽 흐름을 추적할 수 있다.
  • RUST_LOG=debug: 라우팅 테이블이 변경되는 찰나의 전파 과정, 로컬 가십(Gossip) 스카우팅 스냅샷, 복잡한 인증 절차(ACL) 내부 판정 결괏값 등을 시각화한다.
  • RUST_LOG=trace: 모든 패킷의 이동, 큐(Queue)의 밀리초 단위 버퍼링 상태, 프레임 파편화(Fragmentation) 조각까지 쏟아진다. 이 모드를 운영 서버에서 켜면 1분 안에 수 기가바이트의 디스크가 소진된다. 오직 로컬 격리망에서 재현 테스트를 할 때만 켜라.

1.0.2 모듈(컴포넌트) 타겟팅 로깅

“라우터 네트워크 코어는 정상인데 웹소켓(WebSocket) 수신부만 타임아웃이 난다“고 하자. 전체를 trace로 올리면 텍스트 홍수에 휩쓸린다. tracing 프레임워크의 문법을 활용하여 컴포넌트별로 로그망을 좁혀라.

## 기본은 info로 유지하되, 통신 전송(transport) 모듈만 trace, 스토리지 플러그인만 debug로 타겟팅
export RUST_LOG="zenoh=info,zenoh_transport=trace,zenoh_plugin_storage_manager=debug"
./zenohd

1.0.3 클라이언트 애플리케이션 로그 통합 (Rust & TypeScript)

라우터뿐만 아니라 여러분이 짠 클라이언트의 로그 타이밍이 완벽히 라우터의 현상과 일치해야 한다.

  • Rust 클라이언트: tracing-subscriber 크레이트를 엮어 애플리케이션의 비즈니스 로직(예: info!("Motor stopped due to Zenoh error");)과 Zenoh 통신 오류가 같은 시간 축, 같은 타임스탬프로 출력되게 아키텍처를 설계하라.
  • TypeScript (Node.js) 백엔드: @eclipse-zenoh/zenoh-ts 패키지에서 발생하는 연결 에러를 catch 블록에서 가로채고, Winston이나 Pino 같은 로거를 통해 JSON 포맷 구조체로 출력해야 향후 중앙 로그 서버에서 정규식 파싱(Parsing)이 가능하다.

2. 로그 포맷 분석과 핵심 에러 코드(Error Code) 해석

Zenoh 데몬과 API 에러 콘솔에 노출되는 문자열은 단순한 영단어가 아니라 프로토콜의 상태 기계(State Machine) 고장을 의미하는 신호 체계다. 빈번한 에러 지표를 외우고 현장에서 단박에 원인을 짚어내라.

3. 분산 네트워크 환경에서의 중앙 집중형 로그 수집 전략

로봇 수백 대와 에지 서버 수집 대가 공장 전역에 퍼져 있을 때, 문제 발생 시마다 해당 장치의 리눅스 셸에 SSH로 로그인해서 grep ERROR /var/log/zenohd.log를 치는 것은 전형적인 구시대적(Old-school) 막노동이다.

분산 인프라 시스템 트러블슈팅의 전제 조건은 분리되고 파편화된 모든 텍스트 로그를 한곳(클라우드 저장소)으로 끌어모으고 실시간으로 검색할 수 있는 중앙 집중형 로깅(Centralized Logging) 인프라를 구축하는 것이다.

3.0.1 경량 에이전트 (Fluent Bit) 아키텍처

IoT나 에지(Edge) 기기에 메모리를 수백 메가바이트씩 잡아먹는 무거운 자바 기반 Logstash 에이전트를 까는 것은 자살 행위다. 초경량 Fluent Bit 포워더(Forwarder)를 각 라즈베리 파이나 로봇의 데몬으로 상주시켜라.

  • 기본 설정의 핵심: 라우터와 여러분의 클라이언트 앱은 로그 파일(/var/log/zenoh/*.log)에 텍스트를 내뱉고, Fluent Bit은 이 경로를 꼬리 물기(Tail-ing)하여 읽은 뒤, JSON 포맷화하여 HTTP 또는 TCP 스트림으로 클라우드로 쏴 올린다.
  • 태그(Tagging) 주입: 전송 전에 [노드 ID(AMR-01)], [공장 존(Zone-B)], [장비 스펙(Ubuntu 22.04)]과 같은 정적 식별용 메타데이터 라벨 구조를 JSON 헤더에 반드시 박아 넣어라. 이 메타데이터가 없으면 클라우드 저장소에서 어떤 기기가 뿜어낸 에러인지 찾을 방법이 없다.

3.0.2 저장과 시계열 검색: Elasticsearch (ELK) 또는 Grafana Loki

전 세계에서 쏘아 올린 로그가 도달할 클라우드 수집 베이스캠프다.

  • ELK 스택 (Elasticsearch, Logstash, Kibana): 가장 전통적이고 텍스트 본문 풀 텍스트(Full-text) 인덱싱이 강력한 구조다. 특정 단어(Queue full)를 포함한 로그를 1년 치 아카이브에서 순식간에 골라낸다. 단, 스토리지 비용이 높다.
  • Grafana Loki: 최근 마이크로서비스 및 K8s 환경에서 각광받는다. 로그의 본문을 인덱싱하지 않고 라벨(Label)만 인덱싱하므로 메모리를 훨씬 덜 먹고, 프로메테우스 메트릭 대시보드 화면(Grafana)과 동일한 창에서 밀리초 단위로 엮어서(Side-by-side) 조회하기에 압도적으로 유리하다.

3.0.3 디버깅 사례 (Incident Mockup)

“서버 대시보드에서 14시 31분에 로봇팀 전체 통신이 10초간 완전 증발했다“는 타임아웃 이벤트가 발생했다.
엔지니어는 즉각 중앙 관제소의 Log Explorer 화면 창을 연다.

  1. 질의어 검색: {system="zenoh", severity="ERROR", time="14:31:00" ~ "14:31:10"}
  2. 결과값 확인: 10초 사이에, 공장 네트워크의 스위치에 연결된 ‘안개(Fog) 라우터-X’ 한 대가 뱉어낸 무수한 [WARN] Liveliness token dropped 로그와, 그 즉시 부활하면서 생성한 [INFO] Session Open 복구 로그가 쏟아져 있는 것을 시계열로 확인.
  3. 결론 격리: 로봇 자체나 클라우드 서버의 버그가 아니라, Fog 라우터 머신의 일시적인 재부팅(정전이나 네트워크 포트 재시작 등)이 이 10초짜리 마비 사태의 근원임을 클릭 3번 만에 증명해 낼수 있다.

4. 시스템 메트릭과 로그의 교차 검증 방법

로그(Log)는 프로그램이 겪은 특정 사건(Event)에 대한 서술형 일기장일 뿐, 시스템 전체의 체력을 보여주는 X-Ray가 아니다. 반대로 프로메테우스 기반의 메트릭(Metric) 지표는 하드웨어와 트래픽의 거시적인 굴곡(Trend)을 파형으로 그리지만, 대체 왜 파형이 솟구쳤는지에 대한 구체적 범인의 이름은 모른다.

최상의 트러블슈팅 스킬은 “무슨 일(What)“이 일어났는지 말해주는 단일 로그 텍스트를 한쪽 창에 띄우고, “언제 물리적 한계점(Why)“을 넘어섰는지 보여주는 그래프 메트릭을 다른 쪽 창에 띄워 완벽히 시간을 일치시켜 교차 검증(Cross-validation)하는 데서 완성된다.

4.0.1 교차 검증 (Cross-validation) 시나리오 기법

  1. 시간 동기화(Time Synchronization)의 확보
    모든 교차 검증은 각 종단 단말간의 시계가 일치한다는 환상에서 착각을 일으킨다. 로봇의 커널 시간이 5분 느리고 라우터 시간이 빠르면, 로그 창에서 범인을 찾고 그래프에서 엉뚱한 파형을 바라보는 촌극이 빚어진다. 반드시 모든 노드의 OS 시간을 NTP(Network Time Protocol) 서비스나 Chrony 로 밀리초 단위까지 하드-싱크해 두어라.

  2. 사례 1: 메모리 점유율 차트 + Out of Memory 커널 로그

  • 그래프 (Prometheus): 그라파나의 인프라 스크린을 띄운다. zenohd_memory_bytes 그래프가 어제 밤 11시부터 계단식 우상향을 그리며 미친 듯이 치솟다가, 11시 45분에 차트가 끊기고 0으로 바닥을 친다.
