20.2.4 시스템 메트릭과 로그의 교차 검증 방법

로그(Log)는 프로그램이 겪은 특정 사건(Event)에 대한 서술형 일기장일 뿐, 시스템 전체의 체력을 보여주는 X-Ray가 아니다. 반대로 프로메테우스 기반의 메트릭(Metric) 지표는 하드웨어와 트래픽의 거시적인 굴곡(Trend)을 파형으로 그리지만, 대체 왜 파형이 솟구쳤는지에 대한 구체적 범인의 이름은 모른다.

최상의 트러블슈팅 스킬은 “무슨 일(What)“이 일어났는지 말해주는 단일 로그 텍스트를 한쪽 창에 띄우고, “언제 물리적 한계점(Why)“을 넘어섰는지 보여주는 그래프 메트릭을 다른 쪽 창에 띄워 완벽히 시간을 일치시켜 교차 검증(Cross-validation)하는 데서 완성된다.

0.1 교차 검증 (Cross-validation) 시나리오 기법

  1. 시간 동기화(Time Synchronization)의 확보
    모든 교차 검증은 각 종단 단말간의 시계가 일치한다는 환상에서 착각을 일으킨다. 로봇의 커널 시간이 5분 느리고 라우터 시간이 빠르면, 로그 창에서 범인을 찾고 그래프에서 엉뚱한 파형을 바라보는 촌극이 빚어진다. 반드시 모든 노드의 OS 시간을 NTP(Network Time Protocol) 서비스나 Chrony 로 밀리초 단위까지 하드-싱크해 두어라.

  2. 사례 1: 메모리 점유율 차트 + Out of Memory 커널 로그

  • 그래프 (Prometheus): 그라파나의 인프라 스크린을 띄운다. zenohd_memory_bytes 그래프가 어제 밤 11시부터 계단식 우상향을 그리며 미친 듯이 치솟다가, 11시 45분에 차트가 끊기고 0으로 바닥을 친다.
  • 로그 창 (Loki/Kibana): 정확히 11시 45분 03초 부근에 애플리케이션 로그는 침묵하지만, 리눅스 커널의 무시무시한 시스템 로그 dmesg -> Out of memory: Killed process 3144 (zenohd)가 남아 있다.
  • 교차 결론: 이는 C 라이브러리 포인터 오류로 인한 크래시(Crash)가 아님이 100% 명확하다. 서브스크라이버의 읽기 지연으로 인한 페이로드 캐싱 큐(Queue Full) 버퍼 오버플로우 메모리 릭 형태이며, 이 지점을 디버깅 타겟으로 삼아야 한다.
  1. 사례 2: 세션 카운트 붕괴 + 네트워크 오류 트레이스
  • 그래프 (Prometheus): zenoh_session_count 게이지(Gauge) 그래프가 10,000에서 안정적으로 유지되다 14:10에 수직으로 추락하여 5,000으로 떨어졌다(세션 반타작).
  • 로그 창 (Loki): 해당 분초에 클라우드 라우터에서 엄청난 양의 [WARN] Connection Reset by Peer 로그와 Session closed 이력이 쇄도한다.
  • 교차 결론: 라우터나 Zenoh 데몬 코어 자체는 멀쩡하게 동작하고 있었다는 증거다. 절반의 에러가 한 번에 터졌다면 통신망 허리를 구성하는 ISP(인터넷 제공자) 구간 단절이나 보안 스위치의 재부팅에 의한 외부 인프라 타격 사건이다. 즉각 Zenoh 파이프라인 수리가 아니라 네트워크 장비 엔지니어링 파트로 티켓 통보를 던져야 할 사안이다.

메트릭은 징후를 조기에 감지하여 알람을 울리는 센서이고, 로그는 장애 후 해부학적 해답을 내놓는 수술칼이다. 이 두 가지를 양손에 쥐고 지휘하는 관제사(SRE)는 어떤 모호한 장애의 안개 앞에서도 침착하게 근본 원인(Root Cause)의 살을 발라낼 수 있다.
