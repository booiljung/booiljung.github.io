20.1.3 Zenoh 네트워크 토폴로지에 따른 문제 격리(Isolation) 전략

분산 시스템의 크기가 거대해질수록, 메시지가 출발지(Source)에서 목적지(Destination)까지 도달하는 사이에 거쳐 가는 라우터 홉(Hop) 카운트는 계속 증가한다. 데이터 유실이 보고되었을 때, 이 긴 파이프라인 중 정확히 어느 철창에서 쥐가 새고 있는지 이진 탐색(Binary Search) 논법을 적용하여 문제를 격리하라.

graph LR
    Sensor[C++ 에지 센서] -- 1구간 --> Fog[공장 Fog 라우터]
    Fog -- 2구간 --> Firewall[방화벽/NAT]
    Firewall -- 3구간 --> Cloud[클라우드 메인 라우터]
    Cloud -- 4구간 --> Backend[TypeScript 백엔드]

0.1 전제 1. 토폴로지 계층의 중간 허브 타격 (Fog 라우터 기준)

1번 구간의 센서부터 4번 구간의 관제 모니터까지 데이터가 전혀 도달하지 않는 상황이다. 위 다이어그램에서 정중앙에 위치한 Fog 라우터 주변을 우선 디버깅 타겟으로 삼는다.

공장 네트워크에 랩탑을 들고 들어가 노트북 CLI 환경에서 Fog 라우터 IP에 다이렉트로 z_sub "sensor/**" 구독 커맨드를 날린다.

  • 데이터가 나온다면: 1구간(에지->Fog)의 소스 코드와 연결은 100% 무죄다! 문제는 외부 인터넷으로 나가는 2번 구간 방화벽 차단이나 클라우드 라우터(3, 4번)의 설정 오류다. 디버깅 전선을 외부망으로 일거에 좁힐 수 있다.
  • 데이터가 나오지 않는다면: 문제는 완전히 센서 단말 자체이거나(센서 다운), 로컬 네트워크(AP 공유기)의 멀티캐스트 실패다. 클라우드 쪽에 접속해 로그를 뒤적거리느라 낭비하는 스위칭 코스트를 제거할 수 있다.

0.2 전제 2. 통신 모드(Mode) 교차 점검

Zenoh 클라이언트는 구성 시 Peer 모드와 Client 모드를 명시할 수 있다.
어느 순간부터 자율주행 로봇 A와 로봇 B 사이의 로컬 통신(Peer to Peer)이 단절됐다면 무엇을 의심할 것인가?
해당 로봇 터미널에 접속하여 CLI를 Client 모드로 켜서 클라우드로는 데이터가 닿는지 찔러보라.
z_put -m client -e tcp/cloud-ip:7447 "test" "hello"
클라우드 라우터와는 통신이 멀쩡한데 자기들끼리(Peer)만 소통을 못한다면? 이는 무선 AP 스위치 장비 단에서 UDP 가십 디스커버리 혹은 브로드캐스트 패킷을 강제로 스눕(IGMP Snooping) 차단했거나, 방화벽 방침 변경 때문임이 명백하게 증명된다.

0.3 전제 3. 플러그인(Plugin) 바이패스를 통한 논리적 계층 격리

데이터는 라우터를 잘 통과하여 백엔드로 도달하는데, 이상하게 특정 과거 이력(History) 데이터를 쿼리(z_get)할 때만 타임아웃이 나고 지연된다고 하자. 이 현상의 병목이 네트워크 라우팅 코어 엔진인지, 아니면 라우터에 달아둔 백엔드 DB 연동부(Storage Plugin) 때문인지 격리해야 한다.

이때는 라우터 설정 파일 zenohd.json5에서 plugins.storage_manager 섹션을 과감히 주석 처리(Disable)하고 라우터를 단독 기동하라. 데이터를 조회했을 때 당연히 데이터 쿼리 값은 읽어오지 못하겠지만, 타임아웃 지연이나 라우터 멈춤 현상(Hang)이 말끔히 사라진다면? 코어 네트워크는 완전히 결백하며, 스토리지 어댑터 I/O 병목으로 범인을 100% 한정할 수 있다.
