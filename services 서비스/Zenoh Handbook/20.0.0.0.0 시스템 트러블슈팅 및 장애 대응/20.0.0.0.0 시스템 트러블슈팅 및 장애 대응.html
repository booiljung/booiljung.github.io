20.0 시스템 트러블슈팅 및 장애 대응

거대한 분산 시스템에서 완벽한 무결성이란 환상에 불과하다. 단일 서버 아키텍처에서 발생하는 장애는 서버의 로그를 확인하고 프로세스를 재시작하는 것으로 대부분 해결될 수 있지만, 수천 대의 마이크로컨트롤러, 수백 대의 자율주행 모바일 로봇(AMR), 그리고 전 세계에 분산된 클라우드 라우터로 구성된 Zenoh 토폴로지에서의 장애는 전혀 다른 차원의 복잡성을 띤다.

데이터가 중간 지점에서 유실되었을 때, 이것이 무선 간섭으로 인한 물리 계층의 패킷 강하(Drop)인지, 방화벽 단의 멀티캐스트 차단인지, 아니면 수신 측 애플리케이션의 메모리 부족으로 인한 소켓 버퍼 오버플로우인지 즉각적으로 판별하는 것은 불가능에 가깝다. 트러블슈팅(Troubleshooting)은 막연한 감각이나 운에 의존하는 행위가 아니라, 시스템의 동작 메커니즘에 대한 깊은 이해를 바탕으로 논리적으로 원인을 소거해 나가는 엄밀한 귀납적 과학이어야 한다.

이 장의 목적은 개발자와 시스템 운영자가 패닉에 빠지지 않고, 마치 숙련된 외과의사처럼 시스템의 어느 부분을 도려내고 어느 부분을 디버깅할지 판단하는 역량을 길러주는 데 있다. 네트워크 트래픽 분석, 로그 레벨의 정밀한 타겟팅 방식을 다루며, 나아가 C, C++, Rust, TypeScript, Python 등 Zenoh 클라이언트가 지원하는 다양한 프로그래밍 언어의 런타임 특성에서 기인하는 특수한 장애 유형들을 해부한다.

1. 분산 네트워크 트러블슈팅의 핵심 사상 (Core Philosophy)

  1. 로그 기반의 교차 검증 (Log-based Cross Validation):
    하나의 기기에서 발생한 에러 로그만으로 전체 그림을 그리지 마라. 발신자(Publisher)의 발송 성공 로그와 수신자(Subscriber)의 수신 실패 이력, 그리고 중간 라우터 단의 라우팅 상태 로그를 반드시 교차 검증하여 트래픽이 소멸된 정확한 지점(Choke point)을 특정해야 한다.
  2. 이분 탐색을 통한 격리 (Isolation by Binary Search):
    전체 네트워크 경로를 반으로 쪼개어 문제가 어느 구간에서 발생하는지 좁혀 나가라. 클라우드 관제 대시보드 스크린에 로봇 위치가 보이지 않는다면, 당장 로봇의 C++ 코드나 관제 서버의 TypeScript 코드를 뜯어보기 전에 공장 현장의 안개(Fog) 라우터에서 로컬 z_sub 커맨드로 데이터가 올라오고 있는지부터 점검하라.
  3. 재현 불가능한 장애는 해결할 수 없다 (Reproducibility):
    운영 환경에서 간헐적으로 발생하는 문제는 코드를 들여다본다고 빈틈이 보이지 않는다. 반드시 Docker와 네트워크 시뮬레이션(카오스 엔지니어링)을 통해 문제를 개발자의 로컬 환경으로 끌어내려 재현(Reproduce) 가능한 상태로 만들어야만 근본 원인을 타격할 수 있다.

이 장을 통해 독자는 단순한 API 사용자 수준을 벗어나, 거대한 데이터의 격류 속에서 벌어지는 병목과 단절, 그리고 데이터의 훼손을 본질적으로 이해하고 완벽히 통제하는 시스템 아키텍트(System Architect)이자 진정한 엔지니어로 거듭나게 될 것이다.
