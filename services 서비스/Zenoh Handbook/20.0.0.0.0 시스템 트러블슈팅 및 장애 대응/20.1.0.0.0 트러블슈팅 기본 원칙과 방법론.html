20.1 트러블슈팅 기본 원칙과 방법론

장애가 발생했을 때 가장 경계해야 할 태도는 “일단 서버를 재부팅해 보자“거나 “가장 최근에 작업한 코드를 롤백(Rollback)하자“며 직관적이고 무계획적인 조치를 취하는 것이다. 이러한 대증 요법은 일시적으로 증상을 완화할 수는 있으나, 문제의 근본 원인(Root Cause)을 영원히 미궁 속으로 빠뜨리며, 동일한 상황이 발생했을 때 시스템을 또다시 마비시키는 결과를 초래한다.

분산 시스템, 특히 Zenoh와 같이 네트워크 상태의 위상(Topology)이 실시간으로 변하고 멀티캐스트 기반의 가십(Gossip) 프로토콜로 움직이는 생태계에서는 장애 해결을 위한 프로세스가 잘 훈련된 기계처럼 작동해야 한다. 트러블슈팅은 정해진 절차를 엄격하게 따르는 일련의 체계적 방법론(Methodology)이다.

1. 방법론의 3대 원칙 (Three Principles of Troubleshooting)

  1. 옵저버빌리티(Observability)의 확보:
    문제가 발생했다면, 시스템 내부의 상태를 외부에서 짐작하지 말고 명확한 데이터로 들여다볼 수 있는 상태로 만들어야 한다. 추측은 버려라. Zenoh 라우터의 tracing 로그, 클라이언트의 메모리 덤프, 포트 패킷 캡처 도구(Wireshark 등)를 동원하여 시스템의 현재 상태를 측정 가능한 수치와 텍스트로 치환하라.
  2. 가설의 수립과 과학적 검증 (Hypothesis-Driven Debugging):
    증상을 바탕으로 가능성 있는 가설 목록을 작성하라. 예를 들어 “에구동 모터의 제어 명령이 누락된다“는 증상에 대해,
  • 가설 A: C++ 클라이언트의 퍼블리셔 버퍼가 가득 차 패킷이 폐기되었다.
  • 가설 B: Wi-Fi 음영 지역을 지나며 물리적 단절이 발생했다.
  • 가설 C: 라우터 단에서 ACL 쓰기 권한이 거부되었다.
    이 가설들을 하나씩 증명하거나 기각하기 위한 검증 스크립트와 격리 테스트를 수행하며 폭을 좁혀 나가라.
  1. 단일 변인 통제 (Single Variable Control):
    문제를 수정하기 위해 한 번에 두 개 이상의 설정을 변경하지 마라. zenohd.json5 파일에서 포트를 바꾸고, 동시에 큐(Queue) 사이즈를 늘리고 라우터를 재시작했는데 시스템이 고쳐졌다면, 당신은 무엇 때문에 시스템이 고쳐졌는지 알 수 없다. 한 번에 오직 하나의 변수만을 변경하고 결과를 확인하는 끈기를 유지하라.

이 절에서는 이러한 철학을 바탕으로 실제 사건이 터졌을 때 어떤 순서와 절차로 움직여야 하는지, 라이프사이클 모델과 격리 기법을 학습하여 혼돈 속에서도 흔들림 없는 기준을 수립한다.

2. 장애 대응 라이프사이클 이해: 식별, 격리, 원인 규명, 해결, 검증

분산 네트워크 장애 대응은 군사 작전에서의 교전 수칙과 같다. 혼란과 비즈니스 손실의 압박 속에서 개발자가 길을 잃지 않도록 표준화된 5단계의 라이프사이클(Lifecycle)을 몸에 익혀라.

graph TD
    A[장애 발생 발생/알람] --> B(식별 Identification)
    B --> C(위치 격리 Isolation)
    C --> D(원인 규명 Root Cause Analysis)
    D --> E(해결 Resolution)
    E --> F(검증 Verification)
    F -->|이상이 발견되면| D
    F -->|검증 완료| G[재발 방지 대책 수립 Post-mortem]

2.0.1 단계: 식별 (Identification - “무엇이 문제인가?”)

관제 대시보드(TypeScript)의 화면에 로봇 아이콘이 멈추거나, 센서의 차트가 업데이트되지 않을 때 장애가 인지된다. 이때 가장 중요한 것은 막연한 문제 제기를 정량적이고 구체적인 상황 정의로 치환하는 것이다.

  • 나쁜 식별: “시스템이 멈췄다. 로봇 제어가 안 된다.”
  • 좋은 식별: “오후 3시 15분부터, 제 2공장 구역의 10대 AMR에서 발행되는 /telemetry/odom 토픽 업데이트가 전면 중단되었으나, 서버 라우터의 프로세스는 생존해 있으며 CPU 사용량은 평상시(10%) 수준이다.”

2.0.2 단계: 격리 (Isolation - “피해 확산을 무작정 막아라”)

분산 시스템에서는 하나의 오작동하는 노드(예: 1초에 10만 번 연산을 쏟아내는 버그 걸린 코드)가 네트워크 대역폭 전체를 마비시켜 정상 노드들의 통신마저 박살 내는 전파 장애(Cascading Failure)가 잦다.
원인을 찾아 헤매기 전에 물리적 또는 논리적 방화벽을 쳐서 시스템 출혈을 막아라. 버그를 뿜어내는 공장 구역의 라우터 데몬을 클러스터망에서 강제 연결 해제하거나, 방화벽(iptables)을 통해 해당 트래픽을 임시 차단하여 멀쩡한 중앙 클라우드와 다른 서버 노드들의 2차 붕괴를 사수하라.

2.0.3 단계: 원인 규명 (Root Cause Analysis - “블랙박스를 열어라”)

본격적인 디버깅의 꽃이다. 각 엔드포인트에 남겨진 로그(Log), 시계열 모니터링 메트릭, 라우터 설정 파일(zenohd.json5)을 수집하라.
최근에 이루어진 변경 사항(최근 펌웨어 업데이트, 네트워크 인프라 스위치 교체 등)부터 추적한다. z_sub 커맨드라인 유틸리티를 허브 중간중간에 삽입하여 트래픽의 스냅샷을 찍고, 가설 검증 기법을 동원하여 문제가 되는 코드 부위나 설정값을 찾아낸다.

2.0.4 단계: 해결 (Resolution - “정밀한 수정과 배포”)

원인이 규명되었다면 코드를 리팩토링 커밋(Commit) 하거나 라우터의 설정을 수정한다.

  • 큐(Queue) 사이즈 오버랩이 문제였다면 transport/shared_memory 버퍼 한계를 기존 10MB에서 50MB로 증설한다.
  • C++ 클라이언트의 소켓 누수가 문제였다면 생명주기에 맞게 명시적 z_close(session) 코드를 추가한 바이너리를 다시 컴파일해 배포(OTA)한다.

2.0.5 단계: 검증 (Verification - “장애의 완전한 소멸 확인”)

적용된 핫픽스가 실제로 장애를 소멸시켰는지 보수적인 시각으로 검증한다. 문제가 해결된 것처럼 보여도 단순히 네트워크가 일시적으로 숨을 돌린 것일 수 있다. 수정 버전 스크립트를 올리고 최소 1시간 동안의 트래픽 모니터링 추이를 관찰하여, 더 이상 에러 카운터 메트릭이 상승하지 않고 Liveliness 토큰이 유지됨을 수치 데이터로 완전히 입증한 뒤에 롤백 대기 조치를 해제하라.

3. 재현 가능한 테스트 및 디버깅 환경 구축

간헐적인 네트워크 끊김이나 특수한 트래픽 폭주 시에만 발생하는 엣지 케이스를 운영(Production) 서버에서 직접 디버깅(Print/Log 감시)하려 드는 것은 무모한 행위다.
모든 엔지니어링 병목 타파는 내 데스크톱 안으로 시스템의 축소판을 완벽하게 가져올 수 있느냐에 달려 있다. 로컬에서 문제를 재현(Reproduce)하지 못한다면 디버깅의 절차는 한 발짝도 나아갈 수 없다.

3.0.1 Docker Compose를 이용한 토폴로지 축소 복제

클라우드 라우터 1대, 안개(Fog) 라우터 2대, 그리고 이기종 ROS2 브릿지 연결 구성을 내 PC 메모리 상에 그대로 띄워야 한다. 다음 docker-compose.yml 예제는 복잡한 라우터 Mesh 환경을 로컬 포트를 분리하여 즉석에서 복제하는 방법이다.

version: '3.8'
services:
  cloud-router:
    image: eclipse/zenoh:latest
    command: ["--listen", "tcp/0.0.0.0:7447"]
    ports:
      - "7447:7447"

  fog-router-a:
    image: eclipse/zenoh:latest
    # 연결(connect)을 클라우드 라우터로 직접 지정하여 계층망 구현
    command: ["--connect", "tcp/cloud-router:7447", "--listen", "tcp/0.0.0.0:7448"]
    depends_on:
      - cloud-router

  dummy-sensor:
    build: ./sensor-app-rust
    environment:
      - ZENOH_ROUTER=tcp/fog-router-a:7448

위와 같이 논리적 격리망을 구성해두면 실제 클라우드로 접속할 필요 없이, 로컬 단에서 라우터 버전을 교체해 보거나 설정을 부수고 새로 올리며 디버깅 가설을 마음껏 소모할 수 있다.

3.0.2 가혹한 네트워크 시뮬레이션 (Traffic Control 활용)

“공장 지하에서 로봇이 와이파이를 읽지 못할 때 연결 타임아웃 붕괴 현상이 잦다“는 제보를 받았다. 당신의 강남역 기가비트 사무실 랜선 환경에서는 이 장애가 죽었다 깨어나도 재현되지 않는다.
Linux 커널 트래픽 컨트롤 레이어 도구인 tc (Traffic Control) 유틸리티나 toxiproxy 같은 네트워크 열화(Degradation) 도구를 로컬 도커 브릿지에 강제로 주입하라.

## eth0 인터페이스에 200ms 지연 및 30% 패킷 로스, 지터 적용
sudo tc qdisc add dev eth0 root netem delay 200ms 50ms loss 30%

네트워크를 만신창이로 찢어버린 상태에서 Zenoh 클라이언트의 Rust 에러 백오프(Backoff) 로직이 무한 루프에 빠지지 않고 인내를 유지하며 복원(Reconnection)에 성공하는지 검증 보완하라.

3.0.3 Bash 스트레스 분사 스크립트 작성

1만 개의 센서가 0.1초 만에 쏟아지는 OOM(Out of Memory) 트래픽 병목을 재현하기 위해 커스텀 클라이언트를 C++로 작성할 수도 있으나, 즉각적인 재현 도구로는 배시(Bash) 루프와 Zenoh CLI 툴(z_put)을 연동한 스크립가 최고다.

#!/bin/bash
echo "Initiating DdoS Chaos Put..."
for i in {1..20000}; do
  # &를 백그라운드 구동하여 병렬성을 극대화 (스레드 블로킹 우회)
  z_put -p "stress_payload_no_$i" "test/perf/chaos_topic" -m client -e tcp/localhost:7447 &
done
wait
echo "Stress test finished."

위 스크립트를 구동 후 라우터의 컨테이너 docker stats 메모리 메트릭이 지수적으로 솟구치다가 버퍼를 버리거나 크래시 나는 정확한 임계 지점(Breakpoint)을 찾아내어 원인 분석의 지렛대로 삼아라.

4. Zenoh 네트워크 토폴로지에 따른 문제 격리(Isolation) 전략

분산 시스템의 크기가 거대해질수록, 메시지가 출발지(Source)에서 목적지(Destination)까지 도달하는 사이에 거쳐 가는 라우터 홉(Hop) 카운트는 계속 증가한다. 데이터 유실이 보고되었을 때, 이 긴 파이프라인 중 정확히 어느 철창에서 쥐가 새고 있는지 이진 탐색(Binary Search) 논법을 적용하여 문제를 격리하라.

graph LR
    Sensor[C++ 에지 센서] -- 1구간 --> Fog[공장 Fog 라우터]
    Fog -- 2구간 --> Firewall[방화벽/NAT]
    Firewall -- 3구간 --> Cloud[클라우드 메인 라우터]
    Cloud -- 4구간 --> Backend[TypeScript 백엔드]

4.0.1 전제 1. 토폴로지 계층의 중간 허브 타격 (Fog 라우터 기준)

1번 구간의 센서부터 4번 구간의 관제 모니터까지 데이터가 전혀 도달하지 않는 상황이다. 위 다이어그램에서 정중앙에 위치한 Fog 라우터 주변을 우선 디버깅 타겟으로 삼는다.

공장 네트워크에 랩탑을 들고 들어가 노트북 CLI 환경에서 Fog 라우터 IP에 다이렉트로 z_sub "sensor/**" 구독 커맨드를 날린다.

  • 데이터가 나온다면: 1구간(에지->Fog)의 소스 코드와 연결은 100% 무죄다! 문제는 외부 인터넷으로 나가는 2번 구간 방화벽 차단이나 클라우드 라우터(3, 4번)의 설정 오류다. 디버깅 전선을 외부망으로 일거에 좁힐 수 있다.
  • 데이터가 나오지 않는다면: 문제는 완전히 센서 단말 자체이거나(센서 다운), 로컬 네트워크(AP 공유기)의 멀티캐스트 실패다. 클라우드 쪽에 접속해 로그를 뒤적거리느라 낭비하는 스위칭 코스트를 제거할 수 있다.

4.0.2 전제 2. 통신 모드(Mode) 교차 점검

Zenoh 클라이언트는 구성 시 Peer 모드와 Client 모드를 명시할 수 있다.
어느 순간부터 자율주행 로봇 A와 로봇 B 사이의 로컬 통신(Peer to Peer)이 단절됐다면 무엇을 의심할 것인가?
해당 로봇 터미널에 접속하여 CLI를 Client 모드로 켜서 클라우드로는 데이터가 닿는지 찔러보라.
z_put -m client -e tcp/cloud-ip:7447 "test" "hello"
클라우드 라우터와는 통신이 멀쩡한데 자기들끼리(Peer)만 소통을 못한다면? 이는 무선 AP 스위치 장비 단에서 UDP 가십 디스커버리 혹은 브로드캐스트 패킷을 강제로 스눕(IGMP Snooping) 차단했거나, 방화벽 방침 변경 때문임이 명백하게 증명된다.

4.0.3 전제 3. 플러그인(Plugin) 바이패스를 통한 논리적 계층 격리

데이터는 라우터를 잘 통과하여 백엔드로 도달하는데, 이상하게 특정 과거 이력(History) 데이터를 쿼리(z_get)할 때만 타임아웃이 나고 지연된다고 하자. 이 현상의 병목이 네트워크 라우팅 코어 엔진인지, 아니면 라우터에 달아둔 백엔드 DB 연동부(Storage Plugin) 때문인지 격리해야 한다.

이때는 라우터 설정 파일 zenohd.json5에서 plugins.storage_manager 섹션을 과감히 주석 처리(Disable)하고 라우터를 단독 기동하라. 데이터를 조회했을 때 당연히 데이터 쿼리 값은 읽어오지 못하겠지만, 타임아웃 지연이나 라우터 멈춤 현상(Hang)이 말끔히 사라진다면? 코어 네트워크는 완전히 결백하며, 스토리지 어댑터 I/O 병목으로 범인을 100% 한정할 수 있다.
