2.8.4 데이터 타당성(Validity) 보장 및 메시지 생존 시간(TTL/Drop) 정책 관리

가끔은 목적지에 기어이 도달했음에도 불구하고, 이미 도착한 데이터가 ‘상해버린 쓰레기’ 취급을 받을 때가 있다. 네트워크 구석구석을 라우팅하며 1초가 넘게 헤매다 도착한, 이미 옛날이 되어버린 “자율주행 차량의 1초 전 핸들 조향각 수치“는 현재 시점에서 도리어 차량을 탈선하게 만들 대재앙의 씨앗이 되기 때문이다.

이렇듯 데이터가 생명체처럼 유통기한(Lifespan)을 가지고 있다는 사상은, 고전 통신 프로토콜에서는 크게 신경 쓰지 않았던 분야다. 하지만 Zenoh가 창안한 데이터 타당성(Validity) 메커니즘을 만나게 된다면, 개발자는 이 모든 것을 라우터 엔진에게 위임할 수 있다.

QoS 마법 주문 예시:
TTL(Time-to-Live) = 50ms, Policy = Drop-if-Expired

메시지가 출발할 때 자신의 가방에 TTL(생존 한계 시간) 옵션값을 담아서 네트워크로 투신한다. 라우터 A, B를 거쳐 수많은 큐(Queue) 대기열에서 고생을 격다가, 어느 라우터 엔진이 이 메시지의 남은 잔여 수명을 점검해 보고 “앗, 이 녀석 이미 50ms를 초과해 버렸군“이라고 판단하는 순간, 이 라우팅 허브는 남은 수신자까지의 경로를 무자비하게 폐기(Drop)해 버린다!

네트워크 전반에 걸쳐 유효기한을 지났거나 앞으로 지날 확률이 명백한 데이터 찌꺼기들을 조기에 색출하여 아예 포워딩조차 막아버리는(Early Drop) 이 막강한 살균 작용 덕택에, 시스템 종착역에 앉아 있는 제어 컴퓨터는 무척 신선하고 싱싱한(Valid) 실시간 통제 권한 데이터만을 섭취하게 됨을 100% 신뢰할 수 있다.
