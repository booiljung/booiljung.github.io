2.8.3 메시지 우선순위(Priority) 부여 시스템 및 큐잉(Queueing) 모델

네트워크 혼잡 제어를 통해 라우터가 자생적으로 과부하의 죽음을 간신히 막았다고 하더라도, 고립된 수많은 트래픽 큐(Queue, 대기열) 안에서는 또 다른 살벌한 약육강식 생태계가 벌어진다. 자율주행 차량이 시속 100km 단위로 전진할 때, 그 내부 통신망에는 “음악 볼륨업 스트리밍”, “에어컨 온도 표기 데이터” 같은 한가한 패킷들부터, “전방의 라이다 센서 장애물 발견에 따른 즉각 긴급 브레이크(Crash Action!)” 패킷이 동시에 라우터 좁은 통로에 수만 개씩 몰리게 된다.

어떤 멍청한 미들웨어 엔진이 만일, 브레이크 데이터 명령이 먼저 줄을 서 있는 에어컨 데이터 백만 개가 다 나갈 때까지 대기(FIFO: First-In, First-Out)하라고 명령한다면 그 차량은 영락없이 고철로 전락하고 만다. 그래서 Zenoh 프레임워크는 이 ‘우선순위(Priority) 부여’ 시스템을 절대적으로 프로토콜 아키텍처 깊숙한 코어 디자인에 이식해 두었다.

journey
    title 큐잉(Queue) 통행료 최적화 마법
    section (하위 등급 큐) - 백그라운드 (음악 전송중)
      큐1대기열: 1: 일반
      큐2대기열: 1: 일반
    section (VIP 우선등위 큐) - 즉각 브레이크! (새치기!)
      큐0대기열(최상급): 5: 즉각 발사!
    section 라우터 와이어 전송
      통신선 최적 병합 점유: 5: 송신

개발자는 패킷(키 표현식)을 전송할 때, 그 패킷 블록에 “이 메시지가 시스템에서 차지하는 권력의 척도(1~8등급 등)“를 메타 헤더에 짧게 서명하여 넘긴다. 그럼 Zenoh 라우터 속의 큐(Queue) 관리 엔진은 일반적인 FIFO 스택 버퍼가 아니라, 실시간으로 패킷을 등급별 여러 개 큐 바구니로 분류하여 가장 계급이 높은 녀석에게 모든 전송 스루풋(Throughput, 발사 통로)을 배정하거나, 찰나의 순간에 하위 등급 패킷들을 강압적으로 밀어내고 이른바 “VIP 새치기 우선 통과(Preemption Mode)“를 허용하여 무자비하게 날려버린다. 이로 인해 생사를 오가는 중요한 미션 크리티컬 시스템에서 Zenoh의 채택은 필수가 되었다.
