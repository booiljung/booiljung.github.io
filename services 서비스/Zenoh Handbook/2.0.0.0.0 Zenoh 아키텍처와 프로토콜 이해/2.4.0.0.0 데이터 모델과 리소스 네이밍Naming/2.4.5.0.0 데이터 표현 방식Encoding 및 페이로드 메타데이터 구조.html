2.4.5 데이터 표현 방식(Encoding) 및 페이로드 메타데이터 구조

DDS와 같은 군사용 프로토콜이 컴파일 단계에서 Message.idl로 메모리 배치를 꽉 묶어둔 타입 종속형(Strongly Typed)이라면, Zenoh는 반대로 데이터에 일관적인 사상과 구조를 강요하지 않는 무타입(Untyped) 바이트 스트림 지향점을 갖고 있다. “네안에 뭐가 들었든 나는 알 바 아님, 그냥 우체부 역할만 깔끔하게 하겠소.” 라는 선언이다.

0.1 Zenoh 페이로드(Payload)의 본질: “ZBytes”

Zenoh로 흐르는 모든 데이터 본문 공간은 사실상 Rust의 제로 카피(Zero-copy)를 응용한 덩어리형 바이트 버퍼, 일명 ZBytes 구조로 패키징된다.
따라서 개발자는 다음과 같은 이기종 포맷을 같은 Put API 함수 안에 자유자재로 짬뽕시킬 수 있다.

  • 문자열("Hello")
  • 플롯 바이너리 배열([f32; 1024])
  • H.264 인코딩 비디오 프레임 바이너리 덩어리

0.2 [Runbook] 데이터 오염을 막는 명시적 인코딩 (Encoding Metadata) 지정 매뉴얼

데이터를 그냥 0과 1의 쓰레기 같은 뭉치(ZBytes)로 던지면, 받는 쪽의 다국어(Python, C, Rust 혼용) 환경에서는 도대체 이 뭉치가 JSON인지, CBOR인지, 아니면 통짜 이미지(JPG) 문자열인지 파싱 하다 크래시(Crash)가 터진다.

이를 규합하기 위해 Zenoh는 패킷을 보낼 때 와이어 헤더에 딱 1바이트짜리 **‘인코딩(Encoding) 라벨 식별자’**를 붙일 수 있도록 규약 해두었다. (마치 HTTP의 Content-Type 헤더와 유사하다)

# [Python 클라이언트 런북: 데이터 오염 방지 표준 발사법]
import zenoh
session = zenoh.open()

# 단순 bytes가 아니라, 이것이 'application/json' 이라는 메타데이터 도장을 
# 패킷 헤더 구조체에 박아넣어서 전파한다!
session.put(
    "sensor/temp", 
    '{"val": 25.4}', 
    encoding=zenoh.Encoding.APP_JSON()
)

[인코딩 스니핑 필터 전략]
데이터를 수집하는 클라우드의 ElasticSearch 백엔드 브릿지 파이프라인에서, 저 헤더의 인코딩을 체크하는 IF 문을 구성하라.
Zenoh 패킷의 reply.sample.encoding 값이 APP_JSON인 놈들만 파싱 로직으로 넘기고, APP_OCTET_STREAM(비디오 등 raw 데이터)이 날아오면 파싱하지 않고 S3 디스크로 곧장 덤프 시켜 버리는(Bypass) 스마트한 데이터 스위칭 요강을 정립할 수 있다.
