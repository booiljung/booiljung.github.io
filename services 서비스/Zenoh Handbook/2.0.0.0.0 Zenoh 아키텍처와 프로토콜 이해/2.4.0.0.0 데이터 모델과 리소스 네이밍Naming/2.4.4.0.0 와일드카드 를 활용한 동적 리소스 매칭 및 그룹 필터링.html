2.4.4 와일드카드(*)를 활용한 동적 리소스 매칭 및 그룹 필터링

Zenoh 네이밍의 진정한 피날레는 와일드카드 필터링 엔진이다. 통신에 구독이나 질의를 할 때, 특정 센서 이름 1개만을 지정하는 것이 아니라 와일드카드(*, **)를 섞어 거대한 그룹(Group) 지시어를 만들어낸다.

0.1 문법 런북 지침

Zenoh의 트라이 엔진은 오직 2개의 와일드카드 규칙만 엄격하게 따른다.

  1. 싱글 애스터리스크 (*): 정확히 슬래시(/)와 슬래시 사이의 “1개 뎁스(Depth)“만 무차별 허용한다.
  • 예: factory/*/temp는 factory/A/temp에는 응답하지만, factory/A/B/temp에는 매칭되지 않는다.
  1. 더블 애스터리스크 (**): 뎁스의 수와 관계없이 그 밑에 떨어진 모든 하위 디렉터리와 파일을 싹쓸이(Recursive) 매칭한다.
  • 예: factory/**는 factory 밑에 있는 수십만 개의 뎁스 토픽을 전부 뒤집어 가져온다.

0.2 [Runbook] 와일드카드 오남용 수습 - 라우팅 루프 및 OOM 방지

session.get("**") 이라는 악의적인 쿼리를 날려버리면, 네트워크 망의 모든 디바이스가 자기 캐시를 전부 뱉어내려 하면서 대역폭 버퍼 붕괴 스톰(Storm)이 일어난다. 라우팅 트래픽을 관제하는 시니어 아키텍트는 반드시 라우터 데몬에서 강력한 펀치(Filter) 옵션을 먹여야만 한다.

// zenohd.json5 내 플러그인 접근 제어(ACL) 예시 방벽
// "관제실 유저가 무식하게 와일드카드를 갈겨도, 비디오(영상) 스트리밍 채널만큼은 
// 절대 와일드카드 그룹 질의에 도매급으로 휩쓸려가지 않게 강제 분리하라!"
plugins: {
  access_control: {
    // Subscriber 차단 정책 부여
    sub_deny: ["**/video_frames/**"]
  }
}

보안과 대역폭의 90%는 코드가 아니라, 어떤 문법의 와일드카드를 열어둘 것인가에 대한 “이름 기반 통제(Named Authorization)“에서 갈린다는 사실을 통감하라.
