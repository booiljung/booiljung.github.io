2.4.2 계층적 URI 스타일 리소스 식별자(Path-like ID) 설계 규칙

Zenoh의 Key는 유닉스 셸(UNIX Shell)의 파일 시스템 경로 패턴을 완벽히 모방하도록 스펙이 잡혀있다. (슬래시 / 로 구분)
factory/zoneA/robot1 이라는 키는 결국 factory라는 큰 폴더 밑에 zoneA라는 중간 폴더, 그 밑에 robot1이라는 데이터 파일이 있다는 철저한 ‘디렉터리 트리’ 사상을 내포한다.

0.1 [Runbook] 안티 패턴: 평면적 식별자(Flat ID)의 파국

아키텍트가 팀원들에게 키 네이밍 원칙(Rule)을 공표하지 않으면, 초보자들은 보통 이렇게 짠다.

  • robot1_temp
  • robot2_temp
  • camera_robot1

이런 평면적 구조(Flat naming)는 나중에 “모든 로봇의 온도만 가져와라” 혹은 “로봇 1번의 모든 센서를 제어해라“라는 쿼리 필터링을 절대 불가능하게 만든다.

0.2 베스트 프랙티스 (조직적 멘탈 모델 네이밍 규칙)

반드시 다음 조합 순서를 지켜 트리 깊이(Depth)를 제약사처럼 정해야만 추후 라우터 ACL(접근 제어) 필터가 원활하게 먹힌다.
[글로벌 영역] / [지리적 위치] / [플릿 혹은 시스템군] / [개별 장비 ID] / [모듈] / [속성]

모범 답안 세팅:

  • corp/seoul/agv_fleet_alpha/amr_01/hardware/battery_volt
  • corp/seoul/agv_fleet_alpha/amr_01/software/nav_status

이렇게 깊이(Depth)가 예측 가능해지면, 인프라 팀장은 라우터 설정(ACL 방화벽)에서 "corp/seoul/agv_fleet_alpha/**" 통신망에 대해서는 5G 망 바깥으로 나가는 것을 트래픽 차단해라!" 라고 단 한 줄로 거대한 통제를 가할 수 있게 된다.
