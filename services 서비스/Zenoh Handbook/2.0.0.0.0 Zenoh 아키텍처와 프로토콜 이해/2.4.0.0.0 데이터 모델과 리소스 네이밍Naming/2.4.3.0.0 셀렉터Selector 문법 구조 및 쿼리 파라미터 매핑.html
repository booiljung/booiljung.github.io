2.4.3 셀렉터(Selector) 문법 구조 및 쿼리 파라미터 매핑

Zenoh 통신망에서 데이터를 끌어오는 Get 함수는 단순한 Key 매칭 조회 기능에 그치지 않고, 마치 http://api.com/users?age=20&sort=desc 같은 HTTP 파라미터를 붙여 서버로 우아하게 넘길 수 있는 완벽한 인터페이스를 갖추고 있다. 이 매칭 규칙 덩어리를 셀렉터(Selector) 라 부른다.

0.1 셀렉터(Selector) = Key Expression + Parameters

Zenoh가 쿼리를 쏠 때는 오직 경로(factory/robot1)만 보내는 것이 아니라, 꼬리표에 달린 물음표(?) 쿼리 매개변수까지 백엔드 노드 안으로 고스란히 배달해 준다.

// [관제 대시보드 - 클라이언트 질의]
// "부산 지사의 로봇들 중, 배터리 20% 이하인 놈들만 필터링해서 대답해라"
session.get("corp/busan/robots/*/battery?max_volt=20").res().await.unwrap();

위 요청이 발사되면, corp/busan/robots/ 경로 밑에서 쿼리에 응답하겠다고 대기 중추인 에지 장비의 연산 함수(Queryable Callback)에게 저 max_volt=20 이라는 参数 텍스트가 정확히 파싱(Parsing)되어 전달된다.

0.2 [Runbook] 파라미터 기반 인메모리 필터링 방어

모든 배터리 값을 클라우드로 풀(Pull)로 당겨온 뒤에 클라우드 서버 CPU에서 if (volt < 20)로 걸러내는 것은 대역폭 낭비의 극치다.
Zenoh의 쿼리 파라미터 파이프를 이용하면, 필터링 연산을 데이터를 쥐고 있는 에지 단말기 쪽으로 밀어내는(Push-down) 아키텍처가 성립된다. 데이터가 공중을 날아가는 양 자체를 줄이기 위해 셀렉터 파라미터 컨벤션을 API 스펙(Swagger 표기처럼)으로 문서화하여 팀원들에게 배포하라.
