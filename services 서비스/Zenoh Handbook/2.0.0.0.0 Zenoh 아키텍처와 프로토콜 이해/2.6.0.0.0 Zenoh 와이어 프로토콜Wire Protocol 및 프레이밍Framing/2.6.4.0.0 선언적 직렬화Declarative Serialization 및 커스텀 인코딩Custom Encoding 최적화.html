2.6.4 선언적 직렬화(Declarative Serialization) 및 커스텀 인코딩(Custom Encoding) 최적화

데이터가 프로그램 메모리에서 네트워크의 바이트(Byte) 배열, 이른바 와이어(Wire) 위로 올라탈 때, 반드시 데이터 구조를 깨지지 않게 납작한 1차원 데이터로 나열시키는 직렬화(Serialization) 작업이 수반된다.

만일 당신이 ROS2(로봇 운영체제) 생태계에 발을 들여놓았다면, 지루하고 복잡한 IDL 파싱기(IDL Parser)를 거친 후 거쳐 무겁고 느려 터진 CDR(Common Data Representation) 직렬화 오버헤드에 치를 떨었을 것이다. 또한 gRPC 개발자라면 시스템에 Protobuf 스키마를 강제 주입해 매번 시스템 코드를 새로 컴파일(Code Generation)해야 하는 지옥을 겪어봤을 것이다.

Zenoh가 가져온 이 파격적인 와이어 세계에서의 혁명은, 바로 이 직렬화 체계(Serialization)를 통신 코어 엔진과 분리한 이른바 **선언적(Declarative)**이고 투명한 취급 체계다!

Zenoh 라우터는 그 어떤 종교적(독점적) 직렬화 스키마(gRPC, CDR 등)를 강요하지 않고 개발자의 취향을 모두 묵인해 준다!

개발자는 오직 데이터를 밀어 넣을 때 그것이 무엇이라는(예컨대 encoding=application/my-custom-robot-algo) 단 몇 바이트짜리 커스텀 인코딩(Custom Encoding) 메타데이터 플래그만 덧붙이면 끝이다. Zenoh 네트워크의 코어 엔진은 그 내용물이 JSON 덩어리인지 파이썬(Python)의 Pickle 덤프인지 압축된 Base64 그림 파일인지 전혀 묻지도 따지지도 않고 가장 빠른 최적 길(Latency)을 거쳐 목적지로 배달해 준다.

그 압도적인 선언적 자율성 덕분에, 데이터 형식과 구조에 얽매였던 개발자들은 코드 제약 없이 자신만의 커스텀 압축 알고리즘(Zstandard, LZ4 등)을 씌워 자유자재의 최적화를 달성하며, 오직 양쪽 끝 점의 단말기 애플리케이션끼리만 그 데이터를 해석하는 아름다운 ’코어-페이로드 분리 최적화 생태계’를 완수하게 된 것이다.
