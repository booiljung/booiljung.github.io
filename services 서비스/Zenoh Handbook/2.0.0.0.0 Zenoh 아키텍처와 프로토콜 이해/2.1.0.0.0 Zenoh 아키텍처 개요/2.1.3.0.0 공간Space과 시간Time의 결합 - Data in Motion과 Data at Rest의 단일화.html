2.1.3 공간(Space)과 시간(Time)의 결합 - Data in Motion과 Data at Rest의 단일화

기존 IT 씬(Scene)에서는 데이터가 움직일 때(Data In Motion - 통신망) 다루는 전공자와, 데이터가 멈춰있을 때(Data At Rest - 데이터베이스) 다루는 전공자가 분리되어 있었다. RabbitMQ는 통신만 하고 끝이고, PostgreSQL은 저장만 하고 끝이다.
그러나 시속 100km로 달리는 커넥티드 카 통신에서는 “방금 보낸 데이터“와 “1분 전에 저장된 데이터” 사이의 구분을 허무는 극단적인 패러다임이 요구된다.

0.1 공간적(위치) 디커플링의 완성

퍼블리셔 A가 /vehicle/speed 데이터를 던진다. 구독자 B는 A가 지구 반대편 데이터센터에 있는지, 수 미터 앞의 단말기인지, 그것조차 아니면 “내가 질문할 때마다 실시간으로 RPM을 계산해서 던져주는 RPC 함수 형태(Queryable)” 인지 알 필요조차 없다. 이 거대한 은닉성(Transparency)이 바로 공간 기반의 분리다.

0.2 시간적결합(Time Decoupling)의 기적적 성취

이것이 Zenoh가 이룩한 가장 이질적인 발명품이다. 전통적인 Pub/Sub 환경에서는 데이터를 쏠 때 구독자가 ‘살아있지 않으면’ 기회비용은 증발한다 (망각).

로봇 관제 웹 브라우저가 클라우드 라우터에 3분 늦게 접속했다. 옛날 상태값을 가져오려 할 때 Zenoh를 어떻게 요리해야 하는가?
해답은 Zenoh 라우터에 스토리지(Storage) 플러그인을 결합하여 데이터를 ’기억(Memories)’하게 만드는 데 있다. 통신사(Carrier) 주제에 스스로를 DB 서버로 작동하도록 유전자(Mutation) 조작을 가한 것이다.

  1. 우선 Zenoh 운영 서버용 워커 데몬에 영속성 플러그인(e.g., RocksDB Backend)을 부착한다. 
    # 백업 플러그인이 탑재된 라우터 구동
zenohd --plugin storage_manager

2. 라우터 설정(API)을 통해, `/vehicle/**` 토픽 패킷이 지나갈 때마다 디스크에 한 장씩 베껴두라고 명령한다 (Transparent Interception).
3. 3분 지각한 관제 브라우저는 일반적인 Sub 대신 "데이터베이스 조회하듯" `Get` 쿼리를 때린다.
   ```typescript
   // [TypeScript 관제 대시보드]
   // 발행자가 죽었든 살았든, 라우터가 과거 데이터를 품고 대리 응답(Proxy Reply) 해준다.
   let replies = await session.get("/vehicle/speed");

통신 패킷(In Motion)과 저장된 데이터셋(At Rest)을 문법 상 차별 없이 “단일화된 질의 인터페이스(API)” 속으로 녹여낸 이 폭력적인 추상화 아키텍처. 이것이 분산 시스템의 코드를 비정상적일 정도로 간결하게 압축시키는 마법이다.
