2.1.2 클라우드-투-마이크로컨트롤러(Cloud-to-Microcontroller) 통합의 확장성

일반적으로 기업들은 거대한 생태계를 구축할 때 하드웨어 계급에 따라 철저한 카스트 제도를 도입한다.

  • 최하위 8비트 센서칩 (MCU) -> I2C/SPI 나 CoAP 사용.
  • 게이트웨이 및 미들급 서버 -> MQTT 또는 AMQP 사용.
  • 최상위 클라우드 백본 -> gRPC 나 Kafka 사용.

이 세 계급이 교류하려면 중간마다 “CoAP에서 MQTT로 번역하는 서버(Proxy)“와 “MQTT를 풀어서 Kafka에 넣는 다리(Bridge)“가 필요하다. 에러 발생 시 도대체 이 3단 파이프라인 중 어디에서 병목이 났는지 추적(Tracing)하는 것은 엔지니어에게 지옥도(Hell)를 선사한다.

0.1 Zenoh의 척추 통일 (Continuum 아키텍처)

Zenoh는 “프로토콜의 스펙 자체를 고무줄처럼 늘렸다 줄였다(Scaling)” 할 수 있도록 와이어 프로토콜(Wire Protocol)을 가변으로 설계했다.

[Runbook: 체급별 프로토콜 가변 배치 훈련]

  1. 극 저사양 MCU (RAM 30KB 미만): 메모리 동적 할당을 금지시킨 순수 C 언어 기반의 zenoh-pico를 굽는다. Zenoh 헤더조차 1~2바이트로 축소(Minimal header)하여 넘기며, 운영체제가 없는 Bare-metal이나 FreeRTOS 위에서 완벽히 돌아간다. 
    // [MCU 디바이스용 zenoh-pico 코드 예시]
z_put(session, z_keyexpr("sensor/temp"), (const uint8_t *)"20", 2, NULL);

2. **에지 게이트웨이 (라즈베리 파이):** 완전한 기능을 갖춘 `zenoh-c` 혹은 `zenoh-rust` 로컬 데몬을 띄워 MCU들의 데이터를 스펀지처럼 빨아들이고 결합(Aggregation)한다.
3. **클라우드 데이터 센터 (128 Cores CPU):** `zenohd` 라우터 클러스터스(Cluster)를 배포해 초당 수천만 건의 트래픽을 처리하는 글로벌 교차로 역할을 부여한다.

중요한 것은 이 세 계층이 서로 대화할 때 **'어떠한 번역기나 변환기(Translator)도 중간에 끼어들지 않는다'**는 점이다. 15바이트의 메모리를 소모하는 온도 센서 칩과 구글 클라우드 기반의 쿠버네티스 파드가 오직 한 가지 언어(Zenoh)로 직접 대면 통신(End-to-End)을 벌이는 경이로운 연속성, 이것이 유지보수 비용을 1/10 수준으로 압살시키는 비결이다.
