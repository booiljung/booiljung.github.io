2.1.1 데이터 중심(Data-Centric) 네트워킹 철학과 설계 목표

전통적인 TCP/IP 통신망은 철저하게 ‘호스트 중심(Host-centric)’ 이었다. 데이터를 보내려면 무조건 상대방 서버의 ’IP 주소’와 ’Port 번호’를 런타임에 알아내어 소켓을 뚫어야 했다(Coupling). DNS, L4 로드밸런서, 서비스 메쉬(Service Mesh) 등은 모두 이 ’IP 찾기’라는 짐을 덜어주기 위해 발명된 거대한 땜질(Workaround)에 불과하다.

0.1 호스트 네트워킹의 참사 (런북 시나리오)

구축해 놓은 스마트 팩토리의 서버 인프라에 장애가 발생해 K8s(Kubernetes) 파드가 재기동되었다. 이때 내부 IP가 할당(DHCP) 변경되면서, 공장 바닥을 기어 다니는 로봇들의 MQTT/REST 클라이언트 모듈이 예외(Exception)를 뿜어내고 영광스러운 ’502 Bad Gateway’를 맞으며 통신 셧다운에 빠진다. 애플리케이션 개발자는 소켓 재연결(Reconnect)과 백오프 지연(Exponential Backoff) 코드를 빙빙 꼬아가며 수천 줄씩 작성해야만 한다.

0.2 Zenoh의 데이터 중심(Data-Centric) 혁명

Zenoh는 호스트(IP)라는 개념을 애플리케이션 코드단에서 완전히 증발시켰다.
로봇 애플리케이션은 대상이 누구든, 어디에 있든 관심이 없다. 오직 허공의 캔버스(Global Data Space)에 “센서 1번의 온도 값”(factory/sensor1/temp) 이라는 데이터 자체의 명찰(Name/Key)만 바라본다.

[코드 레벨의 패러다임 시프트 증명]
개발자의 코드는 IP 주소를 몰라도 동작한다.

// [Zenoh의 Data-Centric 접근 방식 (Rust)]
let session = zenoh::open(config).res().await.unwrap();

// IP 주소 123.x.x.x:8080 따위는 없다. "이름(Key)"을 외치며 던질 뿐이다.
session.put("factory/sensor1/temp", "26.5").res().await.unwrap();

// 마찬가지로 데이터를 구독하는 쪽도 상대의 IP를 묻지 않는다.
let subscriber = session.declare_subscriber("factory/sensor1/temp").res().await.unwrap();

0.3 아키텍트의 런북 행동 지침 (Action Item)

  • 절대 하드코딩 금지: 애플리케이션 로직을 작성할 때 zenohd.json5나 소스 코드 어디에도 특정 구독자/발행자의 고정 IP를 맵핑하려 들지 마라.
  • 의존성 주입 분리: 인프라망의 물리적 접속(TCP 연결, QUIC 세션 등) 세팅은 철저히 Zenoh 데몬 실행 매개변수나 환경설정 계층으로 밀어내고, 앱 코드는 순수하게 ’어떤 주제(Topic/Key)를 다룰 것인가’에만 집중하도록 계층을 분절시켜라. 이 설계가 망가지는 순간 당신의 시스템은 특정 서버실 환경에 갇힌 노예(Vendor lock-in)가 된다.
