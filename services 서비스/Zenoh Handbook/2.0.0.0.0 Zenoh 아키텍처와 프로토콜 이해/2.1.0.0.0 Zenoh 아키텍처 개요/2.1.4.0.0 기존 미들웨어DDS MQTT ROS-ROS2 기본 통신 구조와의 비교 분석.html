2.1.4 기존 미들웨어(DDS, MQTT, ROS/ROS2) 통신 구조와의 비교 분석

시스템 통합 설계(SI) 미팅 현장에서 엔지니어는 반드시 외부 레거시 진영의 거구들, 즉 “왜 MQTT나 DDS를 안 쓰고 생소한 Zenoh를 써야 하죠?” 라는 공격에 직면하게 된다. 이때 구차하게 변명할 것이 아니라, 다음과 같은 기존의 통신 구조가 짊어진 맹점(Blind spots)을 적나라하게 난도질하고 선을 그어야 한다.

0.1 MQTT의 맹점: 단일 두뇌(Single Brain)의 치명적 병목

  • 구조적 한계: MQTT는 완벽한 스타(Star)형 토폴로지다. 모든 데이터는 반드시 브로커(Broker)라는 통행소를 한 번 찍고 와야 한다.
  • 격파 런북 지식: 1미터 옆에 있는 두 로봇이 교신하기 위해 수 킬로미터 떨어져 있는 클라우드를 다녀와야 한다는 설계는 어불성설이다(레이턴시 100배 증가).
    반면 Zenoh는 Peer 모드를 통해 브로커를 거치지 않고 피어 간 자동 다이렉트 소켓을 활성화해 밀리초(ms) 단위의 직통로를 연다. 극심한 대역폭 지연 시대에 MQTT의 중앙집권적 설계는 이미 사형 선고를 받았다.

0.2 DDS의 맹점: 인터넷의 벽을 넘지 못하는 P2P 멀티캐스트의 저주

  • 구조적 한계: 미 국방성 요구사항인 DDS는 단일 폐쇄망(LAN) 안에서는 타의 추종을 불허하는 폭발적 스피드를 낸다. 하지만 IP 서브넷(Subnet) 하나를 넘어가거나, 사내 방화벽 인바운드 문턱에만 닿아도 수만 개의 디스커버리 멀티캐스트 패킷이 차단당하며 바보로 전락한다. 게다가 C++ 컴파일 타임에 막대한 IDL 구조체를 생성해야 하는 경직성은 덤이다.
  • 격파 런북 지식: Zenoh는 광역망(WAN, Internet) 라우팅 프로토콜(QUIC/TCP)과 트리 구조의 영리한 라우팅 엔진을 탑재해 로컬망의 속도감을 고스란히 글로벌 인터넷 망으로 배달한다. zenoh-bridge-dds 한 줄 실행으로 꽉 막힌 DDS 망을 글로벌 클라우드에 숨결처럼 연동시킨다.

0.3 ROS / ROS 2의 맹점: 거대한 직렬화(Serialization) 식탐

  • 구조적 한계: ROS2의 DDS 기반 통신(Message)은 구조상 방대한 데이터를 직렬화/역직렬화(Marshaling) 하는 데에 CPU 코어 연산을 끔찍하게 소비한다. Lidar 포인트클라우드나 비디오 텐서(Tensor)라도 전송할라치면 ROS2 엔진 자체가 프레임 드랍(Frame-drop)을 시전한다.
  • 격파 런북 지식: Zenoh는 데이터 포맷 자체를 강제(Typed)하지 않고 파이프처럼 날 것의 바이트 덩어리들을(Untyped) 제로 카피(Zero-copy) 효율성으로 수송하는 원초주의를 지녔다. 최근 등장한 공식 rmw_zenoh RMW 구현체를 ROS2 환경 변수에 박아 넣는 행위 하나만으로, 기존 DDS 엔진이 먹어 치우던 과도한 ROS2 메모리 사용량과 통신 렉(Lag)을 무자비하게 평정(Flatout)할 수 있다.

당신의 네트워크 위로 흘려야 할 데이터가 단순히 텍스트를 넘어선 메가바이트 급 괴물이라면, 낡은 미들웨어들은 결국 메모리 누수와 처리량 한계점 앞에서 무릎 꿇고 OOM(Out of Memory) 비명을 지를 것임을 현장에 인식시켜라.
