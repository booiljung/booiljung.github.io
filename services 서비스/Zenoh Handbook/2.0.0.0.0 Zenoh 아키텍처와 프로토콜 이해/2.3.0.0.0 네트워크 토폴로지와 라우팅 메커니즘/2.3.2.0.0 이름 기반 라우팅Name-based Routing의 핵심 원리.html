2.3.2 이름 기반 라우팅(Name-based Routing)의 핵심 원리

네트워크 스위치가 길을 찾을 때, 일반적인 패킷은 L3 계층에서 목적지의 IP 주소(192.168.1.10)를 보고 방향을 정한다. 그런데 Zenoh 라우터는 들어오는 패킷을 볼 때 IP 헤더 따위는 거들떠보지도 않는다. 오직 데이터의 ’이름(Name)’만을 쳐다본다.

0.1 트라이 트리(Trie-tree)의 가공할 속도

Zenoh 라우터 메모리 안에는 현재 네트워크 망에서 구독 중이거나 저장되어 있는 모든 토픽(이름)들의 거대한 계층구조 트리(Trie)가 펼쳐져 있다.

퍼블리셔가 [Key: /robot1/temp] 패킷을 던지면, 라우터는 메모리의 트리 뿌리부터 r -> o -> b -> ... 순서로 일치하는 경로가 있는지 나노초(ns) 단위로 타고 내려간다. 만약 이 트리에 자신과 연결된 K8s 파드의 구독자 노드 링크가 열려있다면, 그 파이프(소켓)를 향해 패킷을 냅다 던지는(Forwarding) 식이다.

0.2 [Runbook] 네임스페이스 격리를 통한 글로벌 브로드캐스트 스톰 방어

이름 자체가 길잡이가 되기 때문에 생기는 치명적 단점이 있다. 만약 “최상단(Root)“을 통째로 쿼리해 버리면 모든 글로벌 라우터가 자기가 가진 모든 답을 뱉어내느라 뻗어버릴 수 있다.

와일드카드 남용 금지(Anti-pattern):

// [오답] 이렇게 구독하면 전 세계 모든 데이터가 내 노트북 1대로 몰려와 1초 만에 OOM이 발생한다.
session.declare_subscriber("**")

[아키텍처 대응책] 라우팅 필터링(Routing Firewall):
글로벌 클라우드 라우터를 세팅할 때, 특정 지사(서울, 부산)에서 올라오는 쓸데없는 잔챙이 로그는 클라우드 망으로 아예 넘어오지 못하게 차단해야 한다. 이것을 IP 포트 블로킹이 아닌, **‘Name ACL(이름 기반 필터링)’**으로 막아낸다.

// [클라우드 K8s 라우터의 zenohd.json5]
routing: {
  forwarding: {
    // 오직 /seoul/alert/** 로 시작하는 비상 알림 이름에 대해서만 
    // 서울 지사망에서 클라우드 백본으로 포워딩(라우팅)을 허락한다!
    allow: ["/seoul/alert/**", "/busan/alert/**"],
    deny: ["/seoul/debug_logs/**"]
  }
}

“이름이 곧 목적지이자 보안 정책의 전부다.” 이 사상을 깨닫지 못하면 당신의 글로벌 라우터는 쓰레기 로그 트래픽을 나르느라 정작 중요한 알람 패킷을 타임아웃 낼 것이다.
