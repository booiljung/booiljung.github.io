2.3.3 최단 경로 탐색 알고리즘과 링크 상태(Link State) 관리

Zenoh가 Mesh(거미줄) 토폴로지로 여러 피어(Peer)와 라우터(Router)들이 무작위로 뒤엉켜 있을 때, 데이터는 어떻게 길을 잃지 않고 목적지까지 당도할까? Zenoh는 BGP 통신 프로토콜 등에서 쓰이는 Link-State 알고리즘의 유전자를 차용하여 스스로 완벽한 지도를 그린다.

0.1 가십(Gossip) 프로토콜을 이용한 실시간 지도(Map) 완성

  1. 노드들이 연결되는 순간 서로에게 자신이 가진 연결 리스트(“나는 A, C랑 연결되어 있어”)를 속삭인다(Gossip).
  2. 이 속삭임이 릴레이로 퍼지면서, 네트워크상에 있는 모든 라우터는 머릿속에 ’전체 망의 구조도(Graph)’를 동일하게 복제해 낸다.
  3. 이를 바탕으로 다익스트라(Dijkstra)와 유사한 최단 경로 연산을 수행하여, 목적지까지 가기 위해 “다음 홉번(Next Hop)“으로 어떤 포트를 열어야 할지 결정한다.

0.2 [Runbook] 링크 단절 시의 레이아웃 재계산 딜레이 방어 (Link State Flapping)

에지 환경의 무선망(Wi-Fi, 5G) 특성상 노드 C가 계속 1초마다 끊겼다 붙었다(Flapping)를 반복하면 어떻게 될까?
전체 망의 모든 라우터가 매초마다 “지도를 다시 그려야 해!“라며 미친 듯이 연산을 반복(Route Thrashing)하다 CPU가 마비된다.

링크 플래핑 방어를 위한 Liveliness 튜닝:
zenohd.json5 및 클라이언트 프로필에서, 무선 환경 특화 튜닝으로 ’노드가 죽었다’고 판단하는 유예 기간(Timeout)을 넉넉하게 늘려주어야 한다.

// [불안정한 에지 통신망 라우터의 런북 설정]
transport: {
  link: {
    // Keep-alive 핑 주기를 늦추고,
    keep_alive: 5000, 
  },
  scouting: {
    // "연결이 끊긴 것 같다"고 확정 짓는 데 10초의 유예를 준다.
    // 10초 안의 일시적 끊김에는 지도를 갈아엎는 라우팅 Thrashing 연산을 참아낸다.
    gossip_timeout: 10000 
  }
}

최적 경로를 빠르게 찾는 것보다 중요한 것은, 불안정한 노드가 전체 네트워크의 연산을 무너뜨리는 기생충이 되지 않도록 인내심(Timeout Tolerance)을 부여하는 것이다.
