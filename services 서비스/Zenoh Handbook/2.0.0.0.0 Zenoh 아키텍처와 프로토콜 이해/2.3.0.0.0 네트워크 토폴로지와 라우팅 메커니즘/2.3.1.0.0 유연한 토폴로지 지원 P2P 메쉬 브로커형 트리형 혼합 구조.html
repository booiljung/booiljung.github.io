2.3.1 유연한 토폴로지 지원 (P2P 메쉬, 브로커형, 트리형 혼합 구조)

네크워크 아키텍처 설계에는 정답이 없다. AWS에 데이터를 모을 때는 중앙 집중형(Star/Broker)이 최고고, 같은 공장 바닥의 로봇끼리 통신할 때는 메쉬(Mesh) 탈중앙화가 압도적이다. Zenoh는 이 이분법적 논란을 종식시킨 **하이브리드 카멜레온(Chameleon)**이다.

0.1 단계: 브로커형 토폴로지 (Cloud 관제) 결성

AWS나 K8s 안에 거대한 오케스트레이터 zenohd 데몬(Router) 하나 띄운다. 나머지 전국의 수만 대 로봇들은 이 라우터만 바라보는 Client 모드로 편성한다.
관리가 깔끔하고 보안 정책을 단일 허브(Hub)에서 필터링할 수 있는 익숙한 중앙 집권 형태가 탄생했다.

0.2 단계: 에지 P2P 메쉬형 토폴로지 자가 융합 (Self-healing Edge)

중앙 라우터가 타버리면 로봇들은 전멸하는 대신 이렇게 살아난다.
에지의 로봇 몇 대를 Peer 모드로 격상(Upgrade)시키도록 재구동 스크립트를 밀어 넣으라.
자신들끼리 P2P 거미줄 통신망을 형성하여, 클라우드가 죽어 있는 재난 상황에서도 “로봇 간의 회피 제동 신호 데이터(Pub/Sub)“는 끊김없이 무제한 교환되는 불사조의 생태계가 잉태된다.

0.3 [Runbook] 다중 라우터(Multi-router) 데이지 체인 연결 구축법

지역이 서울 센터, 부산 팩토리, 제주 엣지로 나뉘어 있을 때의 최적 세팅이다.
서울, 부산, 제주의 각각의 포그 라우터를 서로 이어지게(Connect 매개변수 사용) 붙여버려라.

# 부산 엣지 라우터. 클라이언트들을 받아줌과 동시에 서울 중앙망으로 트래픽을 도매 넘기듯 던져준다.
zenohd -e "tcp/seoul.router.cloud:7447"
