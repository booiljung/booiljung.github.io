2.2.3 Zenoh 클라이언트(Client) - 자원 제약 환경(에지 디바이스)을 위한 경량 노드

피어(Peer)가 수다쟁이라면, 클라이언트(Client)는 철저한 은둔형 개인주의자다.
자기 데이터를 남에게 퍼블리시하고 원하는 걸 받아보기만 할 뿐, **“남의 데이터를 대신 중계해 주거나(No Relay)”, “자신이 접속할 수 있는 엔드포인트라고 허공에 광고(No Scouting Multicast)”**하는 오지랖을 부리지 않는다.

0.1 클라이언트 모드의 압도적 무기: 초저전력 배터리 보존

배터리로 돌아가는 드론이나 로봇 센서라면, 무조건 Mode: Client로 세팅하라.
남의 멀티캐스트를 걸러내고 자기 할 일만 하므로 CPU 슬립(Sleep) 전환률이 비약적으로 치솟아 배터리가 서너 배 이상 연장된다.

0.2 [Runbook] 방화벽 NAT 한계의 돌파 (Hole Punching)

방화벽(공유기 IP NAT) 안에서 밖으로 데이터를 쏘기란 험난하다. 하지만 클라이언트는 외부의 라우터 IP를 명시적으로 지목(Connect)하여 단방향 아웃바운드 링크(Outbound Link)를 연다. 한번 열린 연결을 통해 외부 클라우드에서 내부 로봇으로 진입하는 인바운드 트래픽도 이 문을 역방향으로 통과할 수 있다. (기존 웹 소켓의 구조와 동일하다)

클라이언트 기동 필수 커맨드라인:

# 로봇의 센서 노드. 외부의 클라우드 라우터를 철저히 수동 지칭하고 조용히 복종한다.
./my_robot_app --mode client -e tcp/aws-cloud-router.com:7447

만약 당신의 로봇 프로세스를 -e 엔드포인트 지시자 없이 단순 client 로만 띄웠다면 아무한테도 데이터가 안 간다. 피어(Peer) 모드와 달리 자율 탐색 멀티캐스트 스카우팅에 참여하지 않기 때문에 수동으로 목적지 라우터를 박아주지 않으면 이 노드는 영원한 고아(Orphan) 상태로 전락함을 명심하라.
