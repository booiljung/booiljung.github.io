2.2.4 동적 노드 탐색(Discovery) 및 피어 간 세션 협상(Negotiation) 절차

A 노드(IP 10.0.0.2)와 B 노드(IP 10.0.0.5)가 서로 IP를 하드코딩하지 않았음에도 어떻게 귀신같이 달라붙어서 데이터를 쏘는지, 그 물밑 협상(Negotiation) 프로세스를 런타임 틱(Tick) 레벨로 분해한다.

0.1 부팅부터 소켓 연결까지의 생애 주기 (Lifecycle)

  1. Phase 1 (UDP Multicast 발사): A가 부팅되자마자 이더넷 네트워크 전체에 UDP 포트 7447번을 향해 멀티캐스트 Hello 비콘(Beacon) 패킷을 낭독한다. “내 이름은 A고, 내 유니캐스트 TCP 소켓 주소는 이거야.”
  2. Phase 2 (Greeting 수신 및 검증): 허브에 물려있던 B가 우연히 멀티캐스트를 잡아챈다. B는 A가 보낸 유니캐스트 TCP 주소를 보고 자신이 직접 TCP 프로토콜 핸드셰이크를 건다 (SYN).
  3. Phase 3 (Protocol Negotiation): 그냥 소켓만 열림 끝나는 게 아니다. 양측 노드는 서로의 Zenoh 프로토콜 버전 넘버(예: v0.11), 암호화(TLS) 지원 여부, 큐(Queue) 버퍼 최대 한계치 등의 스펙 문서를 밀리초 단위로 교환한다. 한쪽이 너무 낡았거나 규격이 맞지 않으면 이 협상은 파투(Reject) 난다.
  4. Phase 4 (Session Established): 세션 협의가 끝나면 라우팅 테이블 트리 엔진이 상호 교환되며, 본격적인 Pub/Sub/Get 패킷 와이어가 열린다.

0.2 [Runbook] 프로토콜 버전 비대칭 충돌 회피

시스템의 가장 흔한 파국은 이것이다. 작년에 배포된 파이썬 애플리케이션(A)과 오늘 막 배포된 K8s 클라우드 라우터(B) 간에 연결이 죽어버린다. Wireshark로 떠버리면 TCP SYN-ACK은 되는데 직후에 FIN을 때리며 자폭한다.

  • 원인: Zenoh는 0.x.x 개발 단계다. B가 버전 v0.10.x 이고 A가 v0.7.x 일 때 상호 협상(Negotiation) 과정에서 프로토콜 하위 호환성 단절(Wire Protocol Mismatch)이 일어나 튕겨낸 것이다.
  • 대응 매뉴얼: 기업망 내의 인프라 형상 관리(apt, pip)를 스크립팅하여 메이저 버전 업데이트 시 전체 클러스터 데몬과 애플리케이션 클라이언트 SDK를 원샷 오케스트레이션 배포로 동시에 이주(Migration) 시켜라. 프로토콜 파편화가 발생하는 조직에서 통신은 숨을 쉴 수가 없다.
