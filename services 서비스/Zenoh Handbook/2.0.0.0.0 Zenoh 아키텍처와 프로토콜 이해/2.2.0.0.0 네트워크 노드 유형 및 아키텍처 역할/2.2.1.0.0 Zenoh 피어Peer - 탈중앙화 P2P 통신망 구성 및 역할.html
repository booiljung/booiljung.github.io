2.2.1 Zenoh 피어(Peer) - 탈중앙화 P2P 통신망 구성 및 역할

피어(Peer)는 Zenoh 네트워크에서 가장 역동적이고 진취적인 플레이어다. 자신에게 생성되는 패킷뿐만 아니라, 망 내에서 발생하는 “다른 녀석들의 짐“을 대신 날라주는 ‘릴레이(Relay)’ 역할까지 겸임하는 중계 단말기다. (마치 토렌트 망의 시드(Seed)처럼 작동한다)

0.1 피어의 핵심 특성

  1. 스카우팅(Scouting)의 주체: Peer는 네트워크에 부팅되는 즉시 “여기 나 살아있소!“라는 멀티캐스트 비콘을 쏘며 주변의 다른 Peer와 Router를 적극적으로 탐험한다.
  2. 동적 라우팅 트리 편입: 주변 망 구성원이 확인되면 스스로 토폴로지(Mesh/Tree)의 일원 중 하나로 참여 테이블을 만들고 쿼리 경로를 중계한다.
  3. 용도: 자원 계산 능력이 넉넉한 엣지(Edge) 컴퓨터 (예: NVIDIA Jetson, 강력한 IPC)

0.2 [Runbook] P2P(멀티-피어) 망 구성 및 트러블슈팅

단순히 두 대의 노트북에서 zenoh 기반 애플리케이션을 켰더니 데이터가 안 간다? 백이면 백 ‘모드 설정’ 결여다.

정상 구성 명령어:

# 디바이스 A (로봇 제어용 파이썬 스크립트) - 피어 모드로 기동
python3 app_robot.py --mode peer

# 디바이스 B (로봇 관제용 파이썬 스크립트) - 피어 모드로 기동
python3 app_dash.py --mode peer

--mode peer 옵션이 켜지는 순간, 두 기기는 로컬 공유기 내에서 서로를 멀티캐스트로 인지하고 다이렉트 소켓을 연다. (Broker 불필요)

[트러블슈팅 가이드: Multicast Storm 방어]
만약 사내 Wi-Fi에 노트북 10,000대를 가져다 놓고 다 저 --mode peer로 돌리면 어떤 일이 벌어지는가?
1만 대가 동시에 멀티캐스트 핑을 주고받으며(O(N^2) 교배) 무선 공유기 스위치가 10초 만에 뻗어버린다.
조치 방점: 로컬 망 내에 단말기가 10대가 넘어가는 시점부터는 Peer 모드의 자율 중계 기능을 과감히 포기(Shutdown)하고, 철저한 허브(Hub) 방식인 Client-Router 관계로 전환(Migration)해야 네트워크가 생존한다. “뭐든 제일 빠른 P2P“라는 생각은 초보자의 가장 큰 아키텍처 설계 오류다.
