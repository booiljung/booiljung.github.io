2.2.5 스크립트 탐색(Scouting) 메커니즘과 멀티캐스트-유니캐스트 비콘(Beacon)

Zenoh 통신망 디커플링의 심장은 바로 주변 기기를 수색하는 ‘스카우팅(Scouting)’ 기술에 박동을 빚고 있다.

0.1 스카우팅의 2대 기술적 구현 (Multicast vs Unicast)

수색망을 좁히는 데에는 두 가지 빔(Beam) 발사가 존재한다.

  1. Multicast Scouting (은망치기):
    로컬 서브넷 공유기에 연결된 모든 장비에게 방송(Broadcasting)을 한다. IP를 아예 몰라도 된다.
    다만 스위치 장비의 옵션(예: Cisco IGMP Snooping)이 활성화되어 있거나 도커 네트워크 브릿지가 막혀있으면 멀티캐스트 패킷은 허공에서 드랍(Drop)된다.
  2. Unicast Scouting (유도 미사일):
    AWS 클라우드에 떠 있는 공인 IP는 당연히 우리 집 공유기 멀티캐스트를 받지 못한다. 방화벽 너머의 라우터를 찾기 위해, 명시적으로 URL(.e tcp/124.5.x.x)을 쪼아서 유도 미사일처럼 단일 TCP 세션을 조준하는 탐색이다.

0.2 [Runbook] 컨테이너 인프라 멀티캐스트 단절 트러블슈팅

도커-컴포즈(Docker-Compose) 환경 안에서 A, B 두 개의 Zenoh 피어 컨테이너를 돌리는데 통신이 붙지 않는 에러가 터진다.
원인은 Docker 기본 Bridge 네트워크가 UDP 멀티캐스트 224.0.0.225 포트를 차단하거나 드롭하기 십상이기 때문이다.

인프라 해결 처방전:
컨테이너 네트워크 모드를 Host로 파괴하여 런타임을 구동시키거나 커스텀 브릿지에서 브로드캐스트를 허용시켜야 한다.

# docker-compose.yml 런북 수정 (Host 네트워크 브릿징 승격)
services:
  zenoh-peer-1:
    image: eclipse-zenoh
    command: ["-m", "peer"]
    network_mode: "host"  # 컨테이너 가상 격리를 부수고 물리 이더넷 계층을 공유

이 세팅이 먹히는 순간 격리된 컨테이너의 답답했던 멀티캐스트가 한 큐에 터져 피어 다이렉트 교신이 확립된다. “IP를 몰라도 통신시켜 주겠다“는 약속의 대가는, 언더레이(Underlay) 물리망 스위치의 브로드캐스트 포용력에 전적으로 의존한다는 무거운 설계적 진실임을 간과하지 마라.
