2.7.1 멀티 트랜스포트(Multi-transport) 아키텍처 개요

기존 네트워크 프레임워크들은 너무도 태만했다. 그들은 대부분 TCP/IP 하나에만 강하게 종속되어 개발되거나, 혹여 UDP를 지원하더라도 각기 다른 통신 API를 다시 호출해야 하는 파편화를 유발했다. 개발자가 만일 이더넷망을 위성 통신망으로 교체하고 싶다면, 소스 코드 전체를 전부 재설계하거나 어댑터를 새로 만들어야만 했다.

Zenoh가 선보인 이 ’멀티 트랜스포트 아키텍처’는, 개발자의 비즈니스 로직(Pub/Sub, Query)과 가장 하단의 시스템 소켓 파이프라인(TCP, UDP, Serial 등) 사이의 완전한 절연(Isolation) 상태를 선언한다. 이 혁신의 핵심은 플러그인(Plug-in) 메커니즘을 통한 확장성이다.

    end

      C -.->|런타임 동적 선택| T1[TCP/IPv4 플러그인]
      C -.-> T2[QUIC 플러그인]
      C -.-> T3[시리얼(Serial) 모듈]
      C -.-> T4[공유 메모리(SHM)]
    end
    
    T1 --> Web(인터넷 망)
    T2 --> Web
    T3 --> Local(센서 하드웨어)

이 구조 덕분에, Zenoh 라우터는 시작될 때 설정 파일(Config) 한두 줄만 바꿔 끼우는 것으로, 어제는 인터넷 웹소켓 라우터로 굴렀다가 오늘은 블루투스(BLE) 중계기로 즉각 탈바꿈할 수 있다. 소스 코드 단에서는 바뀐 게 단 하나도 없음에도, 하부의 물리적 매체에 완벽하게 대응해 주는 이것이 바로 네트워킹 세계의 진정한 다형성(Polymorphism) 구현체라 할 수 있다.
