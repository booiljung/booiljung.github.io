2.7.4 QUIC 및 WebSockets를 활용한 클라우드 및 브라우저 네이티브 통합 방안

IoT와 로보틱스의 데이터를 결국 사람이 직접 모니터링하고 제어하기 위해서는 현대의 기술의 종착점, 바로 화려한 대시보드가 그려지는 웹 브라우저(Web Browser) 화면으로 모든 데이터가 올라가야 한다. 하지만 브라우저는 극도로 보안이 강화된 독특한 모래상자(Sandbox) 환경 안에 격리되어 있어, 앞서 말한 UDP나 구식 TCP 소켓을 마음대로 뚫고 허공의 로봇들과 통신할 권한을 절대 부여받지 못한다.

Zenoh 개발진은 프론트엔드/백엔드 브릿지를 또 만들기 위한 미들웨어 지옥을 완벽히 분쇄하기 위해, 웹과 클라우드의 전용 통신 엔진으로 군림한 WebSockets 모듈과 차세대 표준 패러다임인 구글(Google)의 QUIC 트랜스포트 계층(Transport Layer)을 완벽하게 자체 플러그인화 해내었다.

특히 HTTP/3 표준의 기틀이 된 QUIC의 채택은, 와이파이가 자주 단절되거나 셀룰러망 안테나 기지국을 이리저리 바꿔가며 이동하는 5G 자율주행 차량 환경(단절과 IP 스위칭이 잦은 환경)에서 TCP가 필연적으로 갖는 ’세션 재연결 대기시간 10초의 악몽(Head of Line Blocking, HOL)’을 근본적으로 잘라낸다. 단 한 번의 TLS 보안 악수(Handshake)만 거치고 나면 IP가 아무리 차량 이동으로 인해 변환된다 한들 연결된 세션 아이디가 투명하게 이어나가져 지연 현상 자체를 마사지해 주는 기적의 백본 역할을 한다.

여기에 웹 개발자들이 가장 접근하기 쉬운 ‘WebSockets(ws://)’ 트랜스포트 계층을 병행 채택함에 따라, 개발자는 React나 Vue.js 같은 일반적인 웹 기술로 짠 클라이언트 스크립트에 그저 타겟 URL 하나를 박아넣는 것만으로, 단 한 줄의 프록시 변환 자바스크립트 코드 없이 수천 킬로미터 떨어진 로봇의 Zenoh P2P 피어망 엔진과 완벽한 브라우저 네이티브 다이렉트 교신을 해낼 수 있게 된 것이다.
