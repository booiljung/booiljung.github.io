2.5.4 Liveliness 토큰 아키텍처 - 시스템 상태 모니터링 및 결함 감지

분산 아키텍처의 영원한 숙제 하나. “공장 구석의 로봇 3번이 방금 죽었다는(Offline) 사실을 중앙 대시보드는 어떻게 알아차릴 것인가?”
일반적으로는 로봇 3번에서 초당 1번씩 Heartbeat(하트비트) 패킷을 중앙에 무지성으로 전송하는 무식한 방법을 쓴다. 로봇이 1만 대라면, 단지 “살아있음“을 증명하기 위해 1초에 1만 개의 쓸데없는 네트워크 핑 패킷 낭비(Overhead)가 발생한다.

Zenoh는 이 멍청한 폴링 방식을 폐기하고, 유언장(Liveliness Token) 이라는 예술적인 비동기 이벤트를 창조했다.

0.1 Liveliness (유언장) 토큰의 동작 원리

  1. 유언장 작성: 로봇 3번(클라이언트)이 처음 Zenoh에 부팅될 때, 라우터에게 "로봇 3번 살아있음"이라는 명찰(Token)을 하나 맡긴다(Declare).
  2. 침묵: 그 이후 로봇 3번은 하트비트를 날리지 않는다. 그냥 조용히 자기 할 일(데이터 푸시 등)만 한다. 통신 대역폭 낭비는 제로(0)다.
  3. 사망 감지 및 유언장 발송: 만약 로봇 3번이 와이파이를 벗어나 연결이 툭 끊겼다. 클라우드 라우터 데몬은 연결소켓(TCP/QUIC)이 물리적으로 끊어졌음을 감지(Drop)한다.
  4. 대리 브로드캐스트: 라우터는 로봇 3번이 맡겨놨던 유언장 토큰을 꺼내어 전체 망을 향해 “로봇 3번이 방금 죽었습니다(Drop Event)!” 라고 대리방송(Proxy Broadcast) 을 때려준다.

0.2 [Runbook] 실전 관제망: 사망 알림(Death Alert) 인터셉트

대시보드 관리자(관제 서버)는 로봇들에게 하나하나 핑을 날리는(Ping-Pong) 코드를 지우고, 오직 이 ’유언장 파열음’만 귀 기울여 모니터링하면 된다.

# [Python 상태 관제 서버 런북]

# 시스템 망 안에서 발생하는 노드들의 "가입(Alive)"과 "사망(Drop)" 이벤트만 구독한다.
def on_liveliness_event(sample):
    if sample.kind == zenoh.SampleKind.PUT:
        print(f"✅ [ONLINE] 장비가 부팅되었습니다: {sample.key_expr}")
    elif sample.kind == zenoh.SampleKind.DELETE:
        print(f"❌ [OFFLINE/DEAD] 장비가 사망했습니다!: {sample.key_expr}")

# Liveliness 전용 메타 경로 구독
session.declare_subscriber(
    "robot/*/status", 
    on_liveliness_event, 
    # 일반 데이터가 아닌 '상태 토큰'만을 감시하겠다는 강력한 선언
    reliability=zenoh.Reliability.Reliable() # 이런 알람은 유실되면 안되므로 TCP 수준 신뢰성 강제
)

# 로봇 단말기 측 선언 예시 (부팅될 때 딱 1번만 실행됨)
# token = session.liveliness().declare_token("robot/3/status")

이 Liveliness 토큰 디자인을 도입하는 순간 클라우드 데이터센터 인바운드 트래픽의 20~30%를 차지하던 지저분한 헬스체크(Health Check Ping) 트래픽이 완전히 박멸된다. “통신하지 않음으로써 살아있음을 증명하는” 역설적 통신 공학의 극치다.
