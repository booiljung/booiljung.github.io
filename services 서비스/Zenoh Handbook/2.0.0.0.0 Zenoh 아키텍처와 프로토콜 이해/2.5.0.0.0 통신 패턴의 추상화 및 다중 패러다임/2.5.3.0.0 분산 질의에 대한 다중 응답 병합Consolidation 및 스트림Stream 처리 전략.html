2.5.3 분산 질의에 대한 다중 응답 병합(Consolidation) 및 스트림(Stream) 처리 전략

앞서 Get 쿼리를 던지면 10대의 로봇이 다발적으로 응답을 쏟아낸다고 했다.
그런데 만약 라우터 백본에 스토리지(DB) 플러그인이 켜져 있다면 어떻게 될까?
로봇 1번이 직접 자신의 최신 배터리 값을 응답함과 동시에, 라우터 스토리지도 자신이 과거에 훔쳐보았던 로봇 1번의 과거 배터리 캐시 값을 꺼내서 대답해 버린다. 즉 질문자에게 **중복된 두 개의 대답(과거 캐시 vs 최신 라이브)**이 스트림으로 날아오는 혼돈이 빚어진다.

이 혼돈을 교통정리 해주는 엔진이 바로 라우터 코어에 내장된 Consolidation(응답 병합) 메커니즘이다.

0.1 [Runbook] 쿼리 병합(Consolidation) 파라미터 조작술

데이터의 목적과 정합성 수준에 따라 쿼리문을 날릴 때 반드시 아래의 파라미터 방패를 덧대어야 한다.

  1. Consolidation.None (병합 없음 - 최고속도 스트림)
  • 라우터보고 “생각하지 말고 오는 대로 다 나에게 토스해라!” 라고 시키는 것이다.
  • 응답이 가장 빠르지만, 앱(App) 단말기에서 직접 과거 캐시 데이터와 최신 데이터를 구별해 중복을 버리는 코드를 짜야 한다.
  1. Consolidation.Latest (최신 데이터 1건 병합 - 추천)
  • 라우터가 응답들을 0.1초 정도 손에 쥐고 기다려 본다.
  • 스토리지에서 올라온 5분 전 데이터와, 로봇에서 직접 올라온 실시간 응답 이 두 개를 비교(HLC 타임스탬프 기반)하여 옛날 데이터는 라우터 선에서 기각(Drop)해 버린다.
  • 최신값 단 하나만 질문자에게 전달되므로 가장 깔끔한 데이터 파이프라인이 보장된다.
// [Rust 클라이언트 병합 옵션 통제 런북]
use zenoh::query::ConsolidationMode;

// 라우터 레이어에서 중복 응답을 가지치기(Latest) 하라고 강력히 지시한다.
let replies = session.get("factory/robot1/battery")
    .consolidation(ConsolidationMode::Latest)
    .res().await.unwrap();

“내 대시보드가 자꾸 똑같은 배터리 데이터를 두 번씩 깜빡거려요“라고 신입 개발자가 묻는다면, 망설임 없이 Consolidation 설정을 누락한 초보적 실수를 지적하라. 라우터에게 연산(병합)을 위임하는 것이 에지 통신의 대역폭 절약의 핵심 전술이다.
