2.9.2 인네트워크 스토리지(In-Network Storage) 아키텍처의 동작 원리

캐싱(Caching)이 짧은 찰나의 순간을 버티기 위한 일회성 램(RAM)의 휘발성 저장소 역할을 뛰어넘어, 우리가 “1년 전의 로봇 센서 배터리 소모량 내역 10GB 전체 다 가져와!“라는 묵직한 이력(History) 데이터 조회까지 이뤄내려면 아키텍처는 무엇이 되어야 하는가?

기존 통신 미들웨어 생태계(DDS, MQTT 등)는 이런 막대한 저장 능력의 의무를 어플리케이션 개발자에게 철저하게 떠넘겼다. 개발자는 서버 뒤편에 InfluxDB, MongoDB 같은 엔진을 따로 설치해 놓고, 통신망에서 날아온 데이터를 하나하나 손으로 직접 변환해 DB 소켓을 열어 Insert 쿼리를 날렸어야 했다(이른바 데이터브릿지의 비극). 하지만 Zenoh 파이프라인의 종착역에는, 통신망 자체가 곧 거대한 분산 데이터베이스로 둔갑해 버리는 이른바 **인네트워크 스토리지(In-Network Storage)**라는 무지막지한 철학이 도사리고 있다.

이 개념은 Zenoh 노드(라우터나 클라이언트)의 몸통 안에 자체적으로 스토리지 저장 보관소 모듈(Storage Engine) 플러그인이 떡하니 꽂혀 있다는 얘기다! 데이터 생성자가 “이 키로 보낼 테니 넌 영원히 기억하고 있어라” 라며 네트워크에 무언가를 발행하면, 인터넷 통신을 담당하는 Zenoh 인프라가 그걸 그저 다음 라우터로 넘기고 버려버리는 대신, 자기 배 안에다가 차곡차곡 하드 디스크 블록으로 직접 영구 압축 기록(Write)해 놓는다.

이 압도적인 아키텍처는 놀랍지 않은가? 데이터를 저장하는 서버 데몬(Daemon)과 통신을 중계하는 인터넷 미들웨어 데몬이 하나의 프로그램(Process)으로 병합되어 버림으로써, 수만 대의 에지 장비들이 퍼내는 실시간 바이트들을 별도의 “DB 저장 연결 오버헤드” 없이 라우터가 직접 집어삼키게 만들어준다. 이 덕에 시스템 설계자는 시스템 연산 부하(TCO)를 극적으로 격감시킬 수 있다.
