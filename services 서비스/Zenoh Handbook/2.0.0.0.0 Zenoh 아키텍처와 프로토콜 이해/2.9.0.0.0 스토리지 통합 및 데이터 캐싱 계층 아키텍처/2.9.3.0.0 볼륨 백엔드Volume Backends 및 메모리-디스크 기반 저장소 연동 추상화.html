2.9.3 볼륨 백엔드(Volume Backends) 및 메모리/디스크 기반 저장소 연동 추상화

방금 앞 절에서 인네트워크 스토리지의 우아함을 극찬했지만, 개발자는 아직 목이 마르다. 내가 이 라우터 안에 꽂을 하드 디스크가 초당 수십만 번의 Insert를 버티는 고성능 비휘발성 플래시 메모리(RocksDB) 일지, 아니면 엄청나게 느리지만 가장 범용적인 시계열 데이터베이스 플러그인(InfluxDB) 일지, 아니면 단지 테스트 환경이라 컴퓨터 전원 코드를 빼면 1초 만에 깔끔하게 사라지는 휘발성 램(RAM, Memory Backend) 공간일지 입맛대로 고르고 싶기 때문이다.

이러한 밑바닥 기저 시스템의 철저한 선택의 자유를 위해, Zenoh 창시자들은 ’볼륨 백엔드(Volume Backend) 체계’라는 우아하고 독립된 플러그인(Plug-in) 아키텍처를 도입했다.

라우터에 꽂히는 이 볼륨 드라이버들 덕분에 시스템 설계자는 무겁고 복잡한 인프라스트럭처 세팅을 할 필요가 없다.

가령 어플리케이션이 1년 치 공장 센서를 보관해야 한다면 설정 파일 열 줄을 조작해 RocksDB나 InfluxDB 플러그인을 활성화하고 하드디스크를 박아넣으면 끝이다. 만약, 찰나의 순간에 지나가는 고화질 영상 스트림의 1분 치 프레임만 백스페이스(되감기) 용도로 간직했다 지울 것이라면 그저 Memory 백엔드를 채택하기만 하면 그만이다.

개발자의 코드(쿼리를 날리는 클라이언트 언어)는 이 하부에 꽂힌 물리적 디스크가 무엇인지, 코어가 RocksDB로 돌아가는지 램(RAM)인지 단 1바이트조차 눈치챌 수 없다. 인프라와 소프트웨어, 그 철저한 이중화 분리(Decoupling) 마이그레이션이 Zenoh 아키텍처 세계관이 선물하는 아름다운 저장소 추상화의 끝판왕이다.
