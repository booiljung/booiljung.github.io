2.9.1 데이터 캐싱(Caching)의 역할과 지연 시간 최소화 전략

세상에서 가장 빠른 네트워킹은 “물리적으로 데이터를 요청하지도 않는 것“임에 이견이 없다. 만일 한국 지사에 있는 1만 명의 클라이언트 모바일이 미국 아마존 클라우드에 떠 있는 유일무이한 발행자(Publisher) 애플리케이션에게 “지금 온도 뭐야?“라는 쿼리를 각기 다른 타이밍에 1초 간격으로 10,000번 반복해서 날린다면 어떨까?

서버의 CPU는 대폭발을 면치 못할 것이고 태평양을 건너는 해저 케이블 비용은 눈덩이처럼 불어나 개발 팀을 해고해 버릴 것이다. 이때 중간의 완충지대 역할을 해내는 구조가 이른바 데이터 캐싱(Caching) 메커니즘이다.

Zenoh가 그리는 캐싱의 사상은 아주 직관적이고 공격적이다.

Zenoh 엔진의 라우터망은 지나쳐가는 트래픽에 설정된 ’보관 유효 규칙’이 있다면, 자신의 메모리 램(RAM) 일부를 덩그러니 비워내어 그 키 값(예: factory/temp)의 가장 갓 배달된 따끈따끈한 결과를 ’최근값(Latest Value)’으로 보관해 둔다. 지구를 한 바퀴 돌아야 했던 수많은 쿼리들은 출발하자마자 만나는 동네 첫 번째 라우터 단에서 “어차피 아까 거랑 값 똑같아, 내가 줄게!” 라며 강제로 회차(Return) 당하여 엄청나게 단축된 1밀리초(ms) 단위의 응답으로 둔갑해 돌아온다.

이 영리한 중간 가로채기(Mid-way Interception) 캐싱 아키텍처는 데이터 출판자(서버)가 받는 통신 피로도를 거의 “최초 1인치 방어” 수준으로 격감시키며, 모든 데이터 조회자에게 “마치 세상의 모든 데이터가 내 모니터 바로 앞(Localhost)에 들어있는 듯한” 환각 수준의 극초저지연(Zero Latency Illusion) 혜택을 흩뿌려주는 것이다.
