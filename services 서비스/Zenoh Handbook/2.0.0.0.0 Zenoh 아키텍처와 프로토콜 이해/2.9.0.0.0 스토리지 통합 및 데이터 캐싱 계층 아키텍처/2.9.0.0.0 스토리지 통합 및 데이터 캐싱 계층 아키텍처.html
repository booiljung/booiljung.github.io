2.9 스토리지 통합 및 데이터 캐싱 계층 아키텍처

우리가 1장에서 논했던 “공간과 시간의 완벽한 결합(Data in Motion과 Data at Rest의 단일화)“이라는 위대한 선언이 가장 물리적으로 찬란하게 피어나는 곳이 바로 여기, Zenoh 스토리지와 캐싱 계층이다.

과거에는 퍼블리셔(센서)가 발생시킨 살아 있는 데이터 스트림과, 이를 쌓아두었다가 과거를 캐묻는 데이터베이스 질의(Query) 서버는 물과 기름처럼 완벽하게 코드가 다르게 돌아가는 두 개의 세계관이었다. 개발자는 실시간 감시 시스템용 MQTT 브릿지도 만들고, 동시에 과거 이력을 조회하기 위해 무거운 InfluxDB 연동 웹 서버도 별도로 돌려야 하는 이중고를 겪었다.

Zenoh는 이 두 단절된 시스템을 ’네트워크 인프라의 안방’으로 끌고 들어와 직결시켜 버렸다. 이 장에서는 라우터와 네트워크 장비가 단순히 데이터를 통과(Pass-through)시켜 스쳐 보내는 파이프라인의 임무를 넘어, 스스로 데이터를 꿀꺽 삼키며 임시로 기억해 두는 ‘캐싱(Caching)’ 기능의 지연율 혁신을 먼저 파헤친다. 뒤이어 이 네트워크망 깊숙이 꽂혀 들어간 플러그인 형태의 ’인네트워크 스토리지(In-Network Storage)’가 얼마나 우아하게 백엔드 DB(메모리, 파일, DBMS)를 흡수해 버리는지, 그리고 쿼리 한 줄로 “방금 전송된 내역“과 “3개월 전의 백업 데이터“가 완벽하게 한 몸으로 버무려져 튀어나오는 그 통쾌한 투명성 응답 원리를 설계 단계부터 심층적으로 마스터해 볼 것이다.
