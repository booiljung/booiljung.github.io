16.1 성능 최적화의 기본 원칙과 방법론

눈감고 튜닝 옵션들을 이리저리 바꿔보는 것은 서버를 가장 빨리 파괴하는 기도 메타(Praying Meta) 다.
성능 최적화는 철저하게 지표(Metrics) 라는 해부학적 데이터에 근거한 메스(Scalpel)질이어야 한다. OS, 네트워크 레이어, 그리고 Zenoh 라우터 내부에 존재하는 보이지 않는 장벽(Bottleneck)들을 수치화하여 포착하는 방법론을 마스터하라.

1. 지연 시간(Latency)과 처리량(Throughput)의 상관관계 이해

가장 멍청한 질문은 “빨라요?” 이다. 스포츠카(낮은 지연 시간)가 필요한지, 덤프트럭(높은 처리량)이 필요한지 명확히 정의해라. 둘 다 가질 수는 없다. 물리학의 법칙이다.

1.0.1 [인스펙션] 제로섬(Zero-Sum) 트레이드오프 파악 전술

1. 로보틱스 제어 파이프라인 (초저지연, Ultra-Low Latency)
자율주행차의 브레이크 페달 신호는 무조건 1ms(밀리초) 안에 타이어에 닿아야 한다. 데이터 크기는 10바이트에 불과하다.

  • 어떻게 타협하는가? 라우터의 모음(Batch) 옵션을 전부 끈다. CPU 가 미친 듯이 연산을 하더라도, 패킷이 1개 들어오는 즉시 네트워크 밖으로 내던진다(Nagle’s Algorithm 활성화 해제).
  • 이 경우 CPU 사용률이 치솟고, 대역폭 효율은 최악(오버헤드가 더 큼)이 되지만, 무조건 가장 빨리 도달한다.

2. 고해상도 영상 및 센서 덤프 (초대용량 처리량, High Throughput)
4K 카메라 프레임이나 라이다(LiDAR) 포인트 클라우드 데이터를 클라우드에 쌓아야 한다. 이건 1초 늦게 가도 된다. 대신 초당 10Gbps 가 막힘없이(Stall-free) 밀어내져야 한다.

  • 어떻게 타협하는가? 라우터 설정에서 버퍼와 프레이밍 사이즈를 수 메가바이트 단위로 미친 듯이 키운다. 데이터를 1000개 모아서 한 번의 암호화(TLS) 와 단일 TCP 패킷으로 쏴버린다.
  • 가장 늦게 출발하지만(버퍼링), 1초 동안 실어 나른 짐의 총량(Throughput) 은 스포츠카보다 100배 많아진다.

2. 종단 간(End-to-End) 시스템 병목 현상 식별 전략

센서 \rightarrow 로컬 라우터 \rightarrow 퍼블릭 인터넷 망 \rightarrow 클라우드 라우터 \rightarrow 웹 모니터 화면.
어디서 지연이 발생하는지 찍기 게임을 멈추고 과학적으로 잘라내라(Isolate).

2.0.1 [Runbook] 데이터 혈관 조영술(Angiography)

1. Node-to-Node(구간별) 분절 테스트
전체 파이프라인이 100ms 가 걸린다면, 중간 관절들을 끊어보고 테스트한다.

  • 로봇 앱 \leftrightarrow 로봇 라우터 (로컬 루프백): 유도된 지연시간 0.1ms 이내. 여기서 10ms 가 찍힌다면 Zenoh 설정이 잘못된 게 아니라 로봇의 C++ / Python 로직 스레드(Thread) 가 잠자는 것이다(CPU 스케줄링 병목).
  • 로봇 라우터 \leftrightarrow 클라우드 라우터 (WAN): ping 명령어로는 모른다. 프로토콜 단위에서의 통신 지연을 봐야 한다. 대역폭 지연 곱(BDP) 이 터지고 있거나, 기지국 문제다.
  • 클라우드 라우터 \leftrightarrow 스토리지 (Disk I/O): I/O 대기를 봐라(iostat). 데이터베이스가 락(Lock)에 걸려 라우팅이 막혀있다.

2. 블로킹(Blocking) vs 비블로킹(Non-blocking)
네트워크 계층(레이어 3, 4) 은 미친 듯이 빠른데 애플리케이션 계층(레이어 7) 이 미치도록 느린 경우가 90%다.
수신단 콜백(Callback) 함수 안에서 SQL 커밋이나 로그(printf) 를 적지 마라. 콜백 함수가 막혀있는 동안, 소켓 버퍼에 차있는 후속 데이터 패킷들은 영원히 대기 상태에 빠지고 큐가 붕괴한다.

3. Zenoh 성능 측정을 위한 벤치마킹 도구(zenoh-bench) 활용

내가 직접 만든 파이썬 수신 스크립트로 성능을 재지 마라. 파이썬의 태생적 한계(GIL 블로킹 이슈) 때문에, 네트워크가 늦는 건지 파이썬이 늦는 건지 알 길이 없다. 무조건 프로토콜 한계점 테스트는 C/Rust 기반의 순정 네이티브 도구로 찍어라.

3.0.1 [Runbook] z_perf 및 zenoh-bench를 통한 한계 돌파 측정

우선 터미널 2개를 연다.

1. 대역폭(Throughput) 테스트: 우주 방어 측정
“내 망이 1초에 기가바이트를 견딜 수 있는가?”

## 터미널 1 (수신단)
z_sub_perf -e "tcp/127.0.0.1:7447"

## 터미널 2 (송신단: 1024바이트짜리 더미 데이터를 초당 죽을만큼 보냄)
z_pub_perf -e "tcp/127.0.0.1:7447" -s 1024

마지막에 화면에 [Throughput] 1.2 GB/s, 1,200,000 msgs/s 라는 스펙이 찍힌다. 이것이 당신이 구축한 인프라와 OS 환경의 “이론상 우주 최강 한계값“이다.

2. 지연 시간(Latency / Ping-Pong) 테스트: 날 끝의 예리함 측정
“왕복에 몇 마이크로(Micro)초가 걸리는가?”

## 터미널 1 (응답자)
z_ping -e "tcp/MY_ROUTER_IP:7447"

## 터미널 2 (질의자)
z_pong -e "tcp/MY_ROUTER_IP:7447"

나오는 Ping: 124us (마이크로초) 라는 값. 이것보다 느린 서비스 코드가 있다면, 네트워크 문제가 아니라 전적으로 100% 로봇 애플리케이션 코드를 잘못 짠 것이다. 이 벤치마킹 값을 면죄부(Baseline) 로 활용해라.

4. 성능 지표(Metrics) 선정 및 베이스라인(Baseline) 설정

최적화를 시작하기 전, 당신의 시스템이 원래 어느 정도의 체력을 가졌는지 벤치마크 점수를 박제해 둬야 한다. 이 점수표(Baseline) 없이 커널 설정을 건드리는 것은 미친 짓이다.

4.0.1 [인스펙션] 4대 황금 지표(Golden Signals) 추적

구글 SRE가 주창한 4대 지표를 Zenoh 네트워크 로봇 버전으로 변환하라.

1. 대기 시간 (Latency)

  • 추적값: Payload의 헤더에 박힌 발생 시간(Timestamp) - 관제 센터 도착 시간.
  • 베이스라인: “동일 공유기 내에서는 평균 2ms, P99(상위 1% 최악의 경우) 5ms 이하.” 이 선을 넘으면 라우팅을 의심한다.

2. 트래픽 (Traffic / Throughput)

  • 추적값: 1초당 통과하는 ZBytes 의 총합.
  • 베이스라인: “초당 10MB 이상의 LiDAR 데이터가 10초 이상 끊기지 않고 유지되어야 함.”

3. 에러 (Errors)

  • 추적값: z_put 혹은 z_get 시 떨어지는 Timeout 에러, 혹은 TLS Handshake 실패율.
  • 베이스라인: “1시간 동안 패킷 드롭(Drop) 률 0.01% 미만.”

4. 포화도 (Saturation)

  • 추적값: CPU 0번 코어의 사용률(%, Soft-IRQ), 그리고 메모리(RAM) 로드율. 네트워크 카드(NIC) 의 Rx/Tx 버퍼 오버플로우 카운트 (ifconfig의 dropped 지표).
  • 베이스라인: “최대 스ループ트 상황에서도 CPU 0번 코어가 80%를 넘지 않아야 함.” (넘는 순간 패킷이 물리적으로 소거되기 시작한다).

5. 테스트 환경과 실제 운영 환경의 차이점 보정

실험실(개발팀 책상) 의 맥북과 초고속 듀얼 밴드 공유기 환경에서는 지연 시간 1ms 를 찍으며 완벽하게 돌아가던 코드가, 공장(프로덕션) 에 가져가는 순간 지연 시간 1,000ms 로 폭발하는 현상이다.

5.0.1 [Runbook] 프로덕션 네트워크 에뮬레이션 전술

실험실 환경을 강제로 지옥으로 떨어뜨려야 한다. 리눅스 커널의 네트워크 통제 도구인 tc (Traffic Control) 와 NetEm (Network Emulator) 을 동원하여 프로덕션의 악조건을 모사한다.

1. 패킷 로스(Packet Loss) 및 딜레이 강제 주입
드론이 지하차도에 들어갔을 때를 실험실에서 모사하라.

## eth0 랜 카드를 거치는 패킷에 강제로 50ms 지연을 주고, 5% 확률로 패킷을 찢어버린다.
sudo tc qdisc add dev eth0 root netem delay 50ms loss 5%

이 세팅을 걸고 Zenoh 벤치마클(z_perf) 을 돌려보라.
만약 TCP 기반이었다면 패킷 로스 5% 구간에서 TCP 재전송의 늪에 빠져 스루풋이 1/100 로 박살나는 끔찍한 광경을 보게 될 것이다. (이것이 16.3장에서 QUIC 으로 갈아타야 하는 명백한 수치적 증거가 된다).

2. 대역폭(Bandwidth) 옥죄기
LTE 환경처럼 초당 10Mbps 밖에 안 나오는 파이프를 강제로 만든다.

sudo tc qdisc add dev eth0 root tbf rate 10mbit burst 32kbits latency 400ms

이 비좁은 파이프로 4K 비디오를 쏴보라. 라우터 데몬(zenohd) 이 OOM 으로 터지는지, 아니면 알아서 오래된 패킷을 버리며(Backpressure) 연명하는지 관찰할 수 있다. 실험실에서 인프라를 한계치까지 고문(Torture) 해 봐야, 야전에서 로봇이 죽지 않는다.
