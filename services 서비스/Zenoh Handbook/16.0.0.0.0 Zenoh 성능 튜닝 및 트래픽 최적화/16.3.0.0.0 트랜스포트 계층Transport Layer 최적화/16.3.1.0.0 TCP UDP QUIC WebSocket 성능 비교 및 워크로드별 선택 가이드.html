16.3.1 TCP, UDP, QUIC, WebSocket 성능 비교 및 워크로드별 선택 가이드

데이터의 종류(Workload) 에 따라 무조건 갈아입어야(Swap) 한다. 이것은 철칙이다.

0.1 [Runbook] 프로토콜 타겟팅 (Targeting) 결단 가이드

1. 로보틱스 원격 주행 제어 (Joystic Control)

  • 가장 중요한 것: “방금 꺾은 핸들의 각도가 즉시 도달해야 한다. 어차피 0.1초 전의 핸들 각도 따위는 중간에 사라져버려도 알 바 아니다(Drop).”
  • 선택: udp/192.168.1.1
  • UDP 는 패킷이 손실되더라도 절대 뒤돌아보지 않고(No Retransmission) 다시 달린다. 초저지연(실시간성 1ms 이내) 게임 / 제어 환경의 원툴이다.

2. 고화질 비디오 영상 및 클라우드 통신망 (WAN)

  • 가장 중요한 것: 5G 망에서는 터널에만 들어가도 무선 신호가 끊겨 패킷이 잃어버린다(Loss). 그런데 화질은 깨지면 안 된다. 빠른 재전송이 필요하다.
  • 선택: quic/server.com (또는 tls/)
  • QUIC 은 패킷 로스가 생겨도, 그 유실된 조각(Stream) 만 핀셋으로 다시 뽑아오지, 다른 정상 프레임(Stream) 의 진행을 절대로 막지 않는다(Head-of-Line Blocking 해결). TCP 대비 혼잡한 모바일 무선 전장에서 압도적인 성능 깡패다.

3. 브라우저/웹 대시보드 강제 통신 (Web UI)

  • 어쩔 수 없이 방화벽 80, 443 만 뚫려있는 악몽 같은 사내 보안 환경이거나, TypeScript/React 로 짠 웹 브라우저에서 관제탑을 돌릴 때의 유일한 피난처다.
  • 선택: ws/server.com (WebSocket). 가장 느리지만 가장 범용적인 타협.
