16.3.4 QUIC 프로토콜의 스트림 다중화(Multiplexing) 및 혼잡 제어 알고리즘 튜닝

TCP 위에서는 데이터를 쏘다가 하나의 패킷이 길을 잃으면, 뒷단의 모든 정상 패킷들이 앞 패킷을 재전송받을 때까지 줄을 서서 대기하는 참사(Head-of-line Blocking) 가 벌어진다. 이 병목을 피하는 마법의 런북이다.

0.1 [인스펙션] 독립 스트림 타겟팅 전술

QUIC 소켓은 내부적으로 고속도로의 차선(Stream) 을 여러 개 만들어낸다.

1. 멀티플렉싱(Multiplexing) 최적화
Zenoh는 QUIC 을 쓸 때 토픽(Key Expression) 에 따라 내부 스트림을 쪼개는 능력이 있다.
카메라 영상 스트림 차선 한쪽에서 패킷 로스가 나서 차가 막혀도(Blocking), 라이다(LiDAR) 스트림의 차선은 아무 영향 없이 최고 속도로 클라우드로 날아간다. 개발자는 이 기능을 최대한 활성화해야 망이 죽지 않는다.

2. QUIC 파라미터 튜닝
QUIC 프로토콜의 윈도우 사이즈와 혼잡 제어 설정은 OS 가 아닌 유저 레벨(Zenoh 플러그인) 차원에서 설정이 가능하다.

// zenohd.json5
transport: {
  link: {
    quic: {
      // BBR 혼잡 제어 명시 (모바일 망 전용)
      congestion_control: "bbr",
      // 한 번에 날아다닐 수 있는 스트림의 무한 펌핑
      max_streams: 1024
    }
  }
}

LTE 급 모바일 망(무선 환경) 에서 테스트해보면, TCP 대비 드론의 영상 끊김 횟수가 완벽하게 1/10 로 줄어드는 것을 확인할 수 있다.
