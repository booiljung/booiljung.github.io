16.3.2 TCP Nagle 알고리즘 비활성화(TCP_NODELAY)와 지연 시간 최소화

Zenoh 라우터를 TCP 로 켰는데 지연 시간이 자꾸 40ms 씩 튀는 구간이 생긴다. 알고 보면 대역폭을 아끼기 위해 “우편물을 조금 모았다가 발송(Batching) 하는” Nagle 알고리즘의 저주에 걸려 있는 현상이다.

0.1 [Runbook] TCP 모래주머니 해체 전술

우편을 모으는 것 자체가, 로봇 제어(Teleoperation) 에 있어서 지연 시간(Latency) 그 자체다.
Zenoh는 기본적으로 끄는 편장(Default-Off) 을 택하기도 하지만, 명시적으로 확정 지어야 한다.

1. 라우터 강제 노딜레이 스위칭
“1바이트짜리 우편물이 들어와도, 뒤에 우편물이 올걸 기다리지 말고 그 즉시 우체부(TCP) 를 출발시켜라.”
이 한 줄의 설정이 Nagle을 박살 낸다. CPU 연산 코스트(트럭 출수 횟수) 는 늘어나지만, 핸들의 브레이크 제어 명령이 관성을 돌파하게 만든다.

// zenohd.json5
transport: {
  link: {
    tcp: {
      // 1밀리초의 대기도 참지 않는 즉발(Instant) 포워딩
      nodelay: true, 
      keep_alive: true 
    }
  }
}

이 옵션을 켜는 순간, 센서에서 쏜 4바이트짜리 데이터 하나가 라우터에 도달하자마자 지연 없이(Zero-delay buffering) 다음 통신 소켓으로 실시간 연계 복사(Forward) 되는, 스포츠카급 레이턴시 코스가 뚫린다.
