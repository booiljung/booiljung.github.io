16.3 트랜스포트 계층(Transport Layer) 최적화

OS 라는 거대한 항구를 정비했다면, 이제 배를 수리할 차례다. 어떻게 실어나를 것인가? (TCP, UDP, QUIC)

Zenoh 는 세상에 존재하는 거의 모든 트랜스포트(Transport) 프로토콜을 설정 한 줄(tcp/, udp/, quic/) 로 바꿀 수 있는 오버스펙(Over-spec) 의 결정체다.
“어차피 다 된다면 무조건 TCP(안전함)를 쓰면 되는 것 아닌가?”
바로 이 안일한 아키텍처적 타협이 지연 시간을 폭발시키고 기지국을 과부하에 빠뜨리는 주범이 된다. 당신의 애플리케이션 데이터가 가진 질량(크기) 과 속성에 맞는 완벽한 로켓(Transport Scheme) 을 매칭하는 런북을 완성하라.

1. TCP, UDP, QUIC, WebSocket 성능 비교 및 워크로드별 선택 가이드

데이터의 종류(Workload) 에 따라 무조건 갈아입어야(Swap) 한다. 이것은 철칙이다.

1.0.1 [Runbook] 프로토콜 타겟팅 (Targeting) 결단 가이드

1. 로보틱스 원격 주행 제어 (Joystic Control)

  • 가장 중요한 것: “방금 꺾은 핸들의 각도가 즉시 도달해야 한다. 어차피 0.1초 전의 핸들 각도 따위는 중간에 사라져버려도 알 바 아니다(Drop).”
  • 선택: udp/192.168.1.1
  • UDP 는 패킷이 손실되더라도 절대 뒤돌아보지 않고(No Retransmission) 다시 달린다. 초저지연(실시간성 1ms 이내) 게임 / 제어 환경의 원툴이다.

2. 고화질 비디오 영상 및 클라우드 통신망 (WAN)

  • 가장 중요한 것: 5G 망에서는 터널에만 들어가도 무선 신호가 끊겨 패킷이 잃어버린다(Loss). 그런데 화질은 깨지면 안 된다. 빠른 재전송이 필요하다.
  • 선택: quic/server.com (또는 tls/)
  • QUIC 은 패킷 로스가 생겨도, 그 유실된 조각(Stream) 만 핀셋으로 다시 뽑아오지, 다른 정상 프레임(Stream) 의 진행을 절대로 막지 않는다(Head-of-Line Blocking 해결). TCP 대비 혼잡한 모바일 무선 전장에서 압도적인 성능 깡패다.

3. 브라우저/웹 대시보드 강제 통신 (Web UI)

  • 어쩔 수 없이 방화벽 80, 443 만 뚫려있는 악몽 같은 사내 보안 환경이거나, TypeScript/React 로 짠 웹 브라우저에서 관제탑을 돌릴 때의 유일한 피난처다.
  • 선택: ws/server.com (WebSocket). 가장 느리지만 가장 범용적인 타협.

2. TCP Nagle 알고리즘 비활성화(TCP_NODELAY)와 지연 시간 최소화

Zenoh 라우터를 TCP 로 켰는데 지연 시간이 자꾸 40ms 씩 튀는 구간이 생긴다. 알고 보면 대역폭을 아끼기 위해 “우편물을 조금 모았다가 발송(Batching) 하는” Nagle 알고리즘의 저주에 걸려 있는 현상이다.

2.0.1 [Runbook] TCP 모래주머니 해체 전술

우편을 모으는 것 자체가, 로봇 제어(Teleoperation) 에 있어서 지연 시간(Latency) 그 자체다.
Zenoh는 기본적으로 끄는 편장(Default-Off) 을 택하기도 하지만, 명시적으로 확정 지어야 한다.

1. 라우터 강제 노딜레이 스위칭
“1바이트짜리 우편물이 들어와도, 뒤에 우편물이 올걸 기다리지 말고 그 즉시 우체부(TCP) 를 출발시켜라.”
이 한 줄의 설정이 Nagle을 박살 낸다. CPU 연산 코스트(트럭 출수 횟수) 는 늘어나지만, 핸들의 브레이크 제어 명령이 관성을 돌파하게 만든다.

// zenohd.json5
transport: {
  link: {
    tcp: {
      // 1밀리초의 대기도 참지 않는 즉발(Instant) 포워딩
      nodelay: true, 
      keep_alive: true 
    }
  }
}

이 옵션을 켜는 순간, 센서에서 쏜 4바이트짜리 데이터 하나가 라우터에 도달하자마자 지연 없이(Zero-delay buffering) 다음 통신 소켓으로 실시간 연계 복사(Forward) 되는, 스포츠카급 레이턴시 코스가 뚫린다.

3. UDP 기반 멀티캐스트(Multicast) 트래픽의 효율성과 패킷 유실 대처

하나의 마스터 로봇(관제) 이, 내 주변에 돌아다니는 100대의 서브 무인기(Swarm) 에게 “전진해!” 라는 명령을 쏠 스코프(Scope)를 상상해 보라.
TCP 나 Unicast 로 통신하면 똑같은 “전진(Go)” 이라는 10바이트 명령을 100대의 로봇에게 일일이(1,000바이트), IP 주소를 돌아가며 루프문(For-loop) 으로 쏴야 한다.

3.0.1 [인스펙션] 스웜 멀티캐스트(Swarm Multicast) 증폭 전술

단 한 번의 외침(Broadcast) 만으로, 나와 같은 무선 주파수(Subnet) 에 있는 100명이 전부 듣게 만드는 물리적 치트키다.

1. 멀티캐스트 핑 활성화와 TTL 조절
이 기능은 스위칭 허브 단에서 하드웨어 레벨 패킷 복사를 지원하기 때문에, 네트워크 점유 대역폭(Bandwidth) 이 사실상 1/100 로 감소한다. 로컬 군집 에지를 짤 때는 선택이 아닌 필수다.

transport: {
  multicast: {
    enabled: true,
    // 패킷이 다른 서브넷(바깥 세계 망) 으로 새어나가는 걸 물리적으로 자단(Time to Live 고정)
    ttl: 1 
  }
}

2. 패킷 드롭(Drop)과 최적화의 역설
멀티캐스트는 사실상 거대한 UDP 기반이다. 한 놈이 구석에 기둥 뒤에 숨어 있어서 명령 패킷을 잃어버렸다고 치자(Drop). 보내는 놈(관제탑) 도 걔가 못 들은 걸 모르고, 남들 다 아는 걸 걔 혼자만 모르고 낙오된다.
그렇기 때문에 멀티캐스트 도메인 안에 있는 100대의 로봇은 “초당 10번씩 주기적으로 자신의 위치를 보내는(Heartbeat)” 식으로, 과거의 명령을 유실해도 다음 명령 포대(Burst) 에서 다시 상태를 끼워 맞추는 멱등성(Idempotency) 로직으로 로봇을 제어해야 애플리케이션 버그(Bug) 없이 살아남는다.

4. QUIC 프로토콜의 스트림 다중화(Multiplexing) 및 혼잡 제어 알고리즘 튜닝

TCP 위에서는 데이터를 쏘다가 하나의 패킷이 길을 잃으면, 뒷단의 모든 정상 패킷들이 앞 패킷을 재전송받을 때까지 줄을 서서 대기하는 참사(Head-of-line Blocking) 가 벌어진다. 이 병목을 피하는 마법의 런북이다.

4.0.1 [인스펙션] 독립 스트림 타겟팅 전술

QUIC 소켓은 내부적으로 고속도로의 차선(Stream) 을 여러 개 만들어낸다.

1. 멀티플렉싱(Multiplexing) 최적화
Zenoh는 QUIC 을 쓸 때 토픽(Key Expression) 에 따라 내부 스트림을 쪼개는 능력이 있다.
카메라 영상 스트림 차선 한쪽에서 패킷 로스가 나서 차가 막혀도(Blocking), 라이다(LiDAR) 스트림의 차선은 아무 영향 없이 최고 속도로 클라우드로 날아간다. 개발자는 이 기능을 최대한 활성화해야 망이 죽지 않는다.

2. QUIC 파라미터 튜닝
QUIC 프로토콜의 윈도우 사이즈와 혼잡 제어 설정은 OS 가 아닌 유저 레벨(Zenoh 플러그인) 차원에서 설정이 가능하다.

// zenohd.json5
transport: {
  link: {
    quic: {
      // BBR 혼잡 제어 명시 (모바일 망 전용)
      congestion_control: "bbr",
      // 한 번에 날아다닐 수 있는 스트림의 무한 펌핑
      max_streams: 1024
    }
  }
}

LTE 급 모바일 망(무선 환경) 에서 테스트해보면, TCP 대비 드론의 영상 끊김 횟수가 완벽하게 1/10 로 줄어드는 것을 확인할 수 있다.

5. MTU(Maximum Transmission Unit) 파편화(Fragmentation) 회피 및 점보 프레임(Jumbo Frame) 활용

10MB짜리 카메라 영상을 보낼 때, 네트워크 장비(스위치) 들은 이것을 보통 1500바이트 크기의 택배 상자(MTU) 에 수만 개로 쪼개어 담는다. 이것을 쪼개고(Fragmentation) 합치는(Reassembly) 행위 자체가 CPU 를 찢어발기는 오버헤드원이다.

5.0.1 [Runbook] 점보 프레임 해방 전술

스위치와 서버의 한계를 깨고 상자 크기 자체를 키워버린다. (인트라넷 전용)

1. OS 랜카드 MTU 하드코딩
클라우드 센터 내부, 혹은 기가비트 이더넷으로 직결된 장비들(로봇 본체 내부 스위치) 이라면 MTU 크기를 점보 프레임 크기(9000 바이트) 로 단번에 찢을 수 있다.

## 리눅스 eth0 포트의 MTU 사이즈를 1500 -> 9000 으로 증폭
sudo ip link set dev eth0 mtu 9000

2. Zenoh 코어 단편화 최적화 (Zenoh Fragmentation)
만약 공용 망(인터넷) 을 탄다면 MTU 를 1500 이상으로 키울 수 없다(통신사 장비가 1501 바이트부터는 다 찢어버리거나 드랍시킨다).
이때는 Zenoh 라우터 단에서 네트워크 스택(OS) 보다 먼저(Application Layer 에서) 영리하게 패킷을 잘라서 OS 에 넘겨주게 세팅하면 패킷 로스 추적과 메모리 낭비를 극단적으로 줄일 수 있다.

// zenohd.json5
transport: {
  // OS 가 파편화를 하기 전에, Zenoh가 먼저 예쁘게 잘라서 내려주어라!
  // MTU 보다는 살짝 작은 값으로 설정 (헤더 크기 고려)
  fragmentation_size: 1400 
}

CPU가 랜 카드 로직(OS Kernel) 보다 똑똑하다면, 유저 레벨(Zenoh) 에서 패킷의 덩어리를 썰어주는 것이 훨씬 우아한 레이턴시를 가져다줄 것이다.
