16.2 시스템 및 운영체제(OS) 레벨 튜닝

Zenoh는 유저 스페이스(User Space)에서 동작하는 애플리케이션 데몬이다. 아무리 Zenoh 코드가 녹슬지 않고 완벽하게 짜여 있어도, 그 밑단을 받치는 거대한 대지인 리눅스 운영체제(Linux OS) 커널이 굼뜨면 10Gbps 랜 카드의 대역폭을 10%도 활용하지 못한다.

이 장에서는 Zenoh 데몬을 운영체제의 제약에서 해방시키기 위한 **커널 해킹(Kernel Hacking)**과 하드웨어 레벨의 저수준 런북에 돌입한다.

1. 리눅스 커널 네트워크 스택 최적화 (sysctl 파라미터)

리눅스의 기본 네트워크 커널 설정(sysctl.conf) 은 1990년대의 웹 서버 호스팅을 위해 타협된, 너무나도 보수적인 스펙이다. 이 족쇄를 풀어야 한다.

1.0.1 [Runbook] 리눅스 커널 강제 확장 전술

1. 좀비 소켓 학살 (TIME_WAIT 단축)
초당 수백 개의 연결이 맺어지고 끊어지는 환경에서 서버를 죽게 만드는 것은 버려진 소켓(TIME_WAIT) 수만 개가 포트를 고갈(Port Exhaustion) 시키는 현상이다. 커널이 이 시체들을 1초 만에 바로바로 소각하도록 수정하라.

sysctl -w net.ipv4.tcp_tw_reuse=1
sysctl -w net.ipv4.tcp_fin_timeout=15

2. 대기열(Backlog) 과대 확장
해커의 핑거프린팅이나 순간 튀는 트래픽 폭주(Burst)에 의해 리눅스의 커널 수신 큐가 터져 패킷이 바닥에 떨어지는(Drop) 일을 방지한다.

## 큐의 크기를 디폴트 1,000에서 100,000으로 극한 팽창
sysctl -w net.core.somaxconn=100000
sysctl -w net.ipv4.tcp_max_syn_backlog=100000
sysctl -w net.core.netdev_max_backlog=100000

이 세팅을 마친 서버는 순간적인 폭우(Traffic Burst) 가 쏟아져도 끄떡없이 데이터를 쥐고 있다가 어플리케이션으로 넘긴다.

2. 소켓 버퍼 크기(SO_RCVBUF, SO_SNDBUF)의 동적 조절

BDP(Bandwidth-Delay Product), 즉 데이터가 공중에 떠 있는 짐의 양을 수용하려면 그 짐을 받아낼 거대한 주머니가 필요하다.

2.0.1 [인스펙션] BDP 한계 돌파 시스템 전술

클라우드처럼 대륙 간(지연 시간 100ms) 통신인데 데이터 크기는 기가바이트(1Gbps) 라면?
디폴트 리눅스 버퍼(256KB) 로는 택도 없다. 커널이 소켓에 “그만 보내!” 라고 외치며 스루풋이 증발한다.

1. OS 레벨 소켓 창고 크기 강제 증설
커널이 프로세스에게 허락하는 최대 소켓 버퍼 한계점을 16MB 로 찢어버린다.

sysctl -w net.core.rmem_max=16777216
sysctl -w net.core.wmem_max=16777216

2. BBR 혼잡 제어(Congestion Control) 주입
구글이 만든 현대 전장의 지배자. 패킷이 잃어버리는 현상(Loss) 자체를 혼잡으로 치부하지 않고, 오직 지연시간(RTT) 이 길어지는 순간에만 살짝 브레이크를 밟는 최강의 알고리즘 BBR 로 디폴트 CUBIC을 대체하라.

sysctl -w net.ipv4.tcp_congestion_control=bbr

이 튜닝 하나만으로 라이다 포인트 클라우드 전송 속도가 30% 이상 상승하는 기적을 목격할 수 있다.

3. 스레드 컨텍스트 스위칭 최소화 및 인터럽트 병합 (Interrupt Coalescing)

초당 100만 개의 패킷이 쏟아져 들어온다. 네트워크 카드(NIC) 가 CPU 에게 “물건이 왔어요” 라고 100만 번 전화(인터럽트) 를 건다면, CPU 는 전화를 받느라 일(데이터 파싱) 을 아예 하지 못하는 Soft-IRQ Storm 에 휩싸인다.

3.0.1 [Runbook] 인터럽트 병합(Coalescing) 방어 전술

1. 전화기 묶음 처리 옵션 켜기
ethtool 로 랜카드의 레지스터에 하드웨어 레벨 명령을 내린다.

## 이더넷 카드 eth0 에 대해 인터럽트 병합 실시
sudo ethtool -C eth0 rx-usecs 50 rx-frames 128

“패킷이 한 개 왔다고 바로 CPU 한테 전화하지 말고, 50마이크로초 동안 기다리거나 128개 패킷이 찰 때까지 입 꾹 다물고 있다가 한꺼번에 보고해라.”
이 옵션을 켜는 순간, 지연시간(Latency) 은 수십 마이크로초 늦어지지만, CPU 부하율은 절반으로 뚝 떨어져 시스템 전체가 안정권에 접어든다. 지대한 처리량(Throughput) 이 필요할 때 쓰는 흑마법이다.

4. CPU 선호도(Affinity) 및 NUMA(Non-Uniform Memory Access) 아키텍처 인지 배치

서버 하나에 CPU 가 2개 꽂혀있고, 램(RAM) 도 두 바구니에 나뉘어(NUMA) 있다치자.
네트워크 카드(NIC) 는 0번 CPU 쪽에 꽂혀 있는데, 정작 Zenoh 라우터 프로세스는 1번 CPU 위에서 돌고 있다면? 패킷 하나를 처리할 때마다 데이터가 메인보드 중앙(QPI 통신망) 을 건너가며 치명적인 병목 지연(NUMA Miss) 이 발생한다.

4.0.1 [Runbook] 물리적 코어 바인딩(Pinning) 전술

“일하는 자의 발 밑에 자재를 두라.”

1. 인터럽트 핀셋팅 (SMP Affinity)
랜 카드가 외치는 통보망을 오직 0번, 1번 물리 코어로 몰아넣는다.

echo 3 > /proc/irq/<NIC_IRQ_NUMBER>/smp_affinity

2. 데몬 프로세스 격리 수용 (Taskset / NUMA Control)
Zenoh 라우터 바이너리(zenohd) 를 기동할 때, 랜 카드가 꽂혀 있는 바로 그 NUMA 노드에만 독박으로 가둬버린다.

## NUMA 0번 존에서만 zenohd 를 실행하고 램을 당겨써라!
numactl --cpunodebind=0 --membind=0 zenohd -c config.json5

이렇게 물리적으로 CPU 와 패킷과 메모리 버퍼를 한 구역에 밀어 넣고 문을 잠가버리면, 캐시 적중률(Cache Hit Ratio) 이 99%에 육박하게 되어 마이크로초 단위의 성능 최적화가 완성된다.

5. 제로 카피(Zero-Copy) 및 공유 메모리(Shared Memory) 통신의 이해

Zenoh 의 철학에서 데이터를 “복사“하는 행위는 유죄다. 같은 컴퓨터 안에서 프로세스 A 가 프로세스 B 에게 데이터를 줄 때, 커널 버퍼로 데이터를 집어넣고 빼는 것만으로도 수백 마이크로초가 날아간다.

5.0.1 [인스펙션] 메모리 포인터 직결 전술 (Zero-Copy)

Zenoh 는 데이터를 보낼 때, 바이트 배열(Byte Array) 단위가 아닌 ZBytes (혹은 ShmBuf) 라는 자기장(포인터 컨테이너) 을 쓴다.

1. 공유 메모리(Shared Memory) 릴레이
동일한 리눅스 서버 안에서 카메라 프로세스(C++) 가 Zenoh 라우터에게 4K 프레임(12MB) 을 보낼 때의 마법이다.

  • C++ 프로세스는 하드디스크(/dev/shm) 의 특수 구역에 12MB짜리 덩어리를 올린다.
  • 그리고 Zenoh 네트워크로는 이 덩어리를 통째로 TCP 로 쏘는 게 아니라, “공유 메모리 150번지에 4K 데이터 있다!” 라는 “작은 쪽지(Metadata)” 만 핑 날린다.
  • 라우터는 그 쪽지를 받고 메인 메모리의 똑같은 150번지를 쳐다보기만 하면 된다(읽음).

2. SHM (Shared Memory) 활성화 런북

// Rust 코드 예시 (수신단)
let mut config = zenoh::Config::default();
// 동일 머신 내에서는 무조건 SHM 모드를 켜서 복사를 없애버려라!
config.insert_json5("transport/shared_memory/enabled", "true");

이 기능이 켜지는 순간, 한 대의 컴퓨터 안에서 도는 수십 개의 마이크로서비스(ROS2 노드들 등) 간의 데이터 전송 대역폭은 메모리 램(RAM) 의 물리적 한계속도인 초당 50~100 기가바이트(GB/s) 에 도달하게 된다.
