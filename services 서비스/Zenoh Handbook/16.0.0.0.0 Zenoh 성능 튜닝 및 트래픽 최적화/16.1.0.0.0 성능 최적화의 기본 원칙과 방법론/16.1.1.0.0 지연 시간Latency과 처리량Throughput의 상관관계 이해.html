16.1.1 지연 시간(Latency)과 처리량(Throughput)의 상관관계 이해

가장 멍청한 질문은 “빨라요?” 이다. 스포츠카(낮은 지연 시간)가 필요한지, 덤프트럭(높은 처리량)이 필요한지 명확히 정의해라. 둘 다 가질 수는 없다. 물리학의 법칙이다.

0.1 [인스펙션] 제로섬(Zero-Sum) 트레이드오프 파악 전술

1. 로보틱스 제어 파이프라인 (초저지연, Ultra-Low Latency)
자율주행차의 브레이크 페달 신호는 무조건 1ms(밀리초) 안에 타이어에 닿아야 한다. 데이터 크기는 10바이트에 불과하다.

  • 어떻게 타협하는가? 라우터의 모음(Batch) 옵션을 전부 끈다. CPU 가 미친 듯이 연산을 하더라도, 패킷이 1개 들어오는 즉시 네트워크 밖으로 내던진다(Nagle’s Algorithm 활성화 해제).
  • 이 경우 CPU 사용률이 치솟고, 대역폭 효율은 최악(오버헤드가 더 큼)이 되지만, 무조건 가장 빨리 도달한다.

2. 고해상도 영상 및 센서 덤프 (초대용량 처리량, High Throughput)
4K 카메라 프레임이나 라이다(LiDAR) 포인트 클라우드 데이터를 클라우드에 쌓아야 한다. 이건 1초 늦게 가도 된다. 대신 초당 10Gbps 가 막힘없이(Stall-free) 밀어내져야 한다.

  • 어떻게 타협하는가? 라우터 설정에서 버퍼와 프레이밍 사이즈를 수 메가바이트 단위로 미친 듯이 키운다. 데이터를 1000개 모아서 한 번의 암호화(TLS) 와 단일 TCP 패킷으로 쏴버린다.
  • 가장 늦게 출발하지만(버퍼링), 1초 동안 실어 나른 짐의 총량(Throughput) 은 스포츠카보다 100배 많아진다.
