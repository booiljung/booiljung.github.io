16.1.5 테스트 환경과 실제 운영 환경의 차이점 보정

실험실(개발팀 책상) 의 맥북과 초고속 듀얼 밴드 공유기 환경에서는 지연 시간 1ms 를 찍으며 완벽하게 돌아가던 코드가, 공장(프로덕션) 에 가져가는 순간 지연 시간 1,000ms 로 폭발하는 현상이다.

0.1 [Runbook] 프로덕션 네트워크 에뮬레이션 전술

실험실 환경을 강제로 지옥으로 떨어뜨려야 한다. 리눅스 커널의 네트워크 통제 도구인 tc (Traffic Control) 와 NetEm (Network Emulator) 을 동원하여 프로덕션의 악조건을 모사한다.

1. 패킷 로스(Packet Loss) 및 딜레이 강제 주입
드론이 지하차도에 들어갔을 때를 실험실에서 모사하라.

# eth0 랜 카드를 거치는 패킷에 강제로 50ms 지연을 주고, 5% 확률로 패킷을 찢어버린다.
sudo tc qdisc add dev eth0 root netem delay 50ms loss 5%

이 세팅을 걸고 Zenoh 벤치마클(z_perf) 을 돌려보라.
만약 TCP 기반이었다면 패킷 로스 5% 구간에서 TCP 재전송의 늪에 빠져 스루풋이 1/100 로 박살나는 끔찍한 광경을 보게 될 것이다. (이것이 16.3장에서 QUIC 으로 갈아타야 하는 명백한 수치적 증거가 된다).

2. 대역폭(Bandwidth) 옥죄기
LTE 환경처럼 초당 10Mbps 밖에 안 나오는 파이프를 강제로 만든다.

sudo tc qdisc add dev eth0 root tbf rate 10mbit burst 32kbits latency 400ms

이 비좁은 파이프로 4K 비디오를 쏴보라. 라우터 데몬(zenohd) 이 OOM 으로 터지는지, 아니면 알아서 오래된 패킷을 버리며(Backpressure) 연명하는지 관찰할 수 있다. 실험실에서 인프라를 한계치까지 고문(Torture) 해 봐야, 야전에서 로봇이 죽지 않는다.
