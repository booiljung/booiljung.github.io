9.2.2.3 에러 처리(Error Handling) 구조 및 디버깅 기법

C 언어는 함수가 실패했을 때 보통 NULL 이나 -1 혹은 0 을 뱉는다.
하지만 “왜 실패했는가?” 를 C 레벨에서 알기는 매우 까다롭다. Zenoh 엔진 깊숙한 곳에서 “포트가 충돌했다“거나 “원격 라우터가 응답이 없다“라고 알려줘도 C가 알아듣게 전달(Pass) 하는 체계가 필요하다.

0.1 [Runbook] 체크(_check()) 매크로와 Rust 로그 역추출 기법

zenoh-c 는 리턴 값으로 에러 코드를 포장해 주는 대신, 만들어진 구조체 자체의 ’유효성’을 묻는 z_xxx_check() 함수 체계를 쓴다.

1. 기본 검증 패턴

z_owned_session_t session = z_open(z_move(config));
if (!z_check(session)) {
    // 실패했다는 것은 알았다.
    printf("[크리티컬] Zenoh 세션을 열지 못했습니다.\n");
    // 근데 대체 왜 실패했지? C 언어에선 에러 메세지가 안보인다!
    exit(1);
}

2. 에러 텍스트 역추출 (디버그 콘솔 환경 변수 강제 주입)
C 단에서 에러 원인을 파악하려면 프로그램 부팅 전에 Rust 코어 엔진단의 로그 스피커(RUST_LOG) 자체를 풀가동시켜 버려야 한다.

# 리눅스 터미널에서 C 프로그램을 실행하기 직전에 환경변수를 때려 넣는다.
export RUST_LOG=zenoh=debug,zenoh_c=debug

# 이제 내 C 프로그램을 실행한다
./my_robot_node

# [출력 결과] C 코드에는 없던, Rust 내부 엔진의 상세 에러가 C stdout에 폭포수처럼 쏟아진다!
# [DEBUG] zenoh::net: Attempting to connect to TCP 192.168.0.5:7447 ...
# [WARN]  zenoh::net: Connection refused! (OS error 111)

3. 콜백 지옥에서의 에러 봉쇄
Subscriber 콜백 안에서 C 포인터를 잘못 다루어 Segmentation fault 가 나면 Zenoh 세션 통신망 전체가 파괴된다.

void on_sample(const z_sample_t *sample, void *arg) {
    // [보안 주의] sample->payload.start 는 NULL 로 끝나는 '문자열(String)' 이 아니다!
    // 순수 바이너리 바이트 덩어리다!!!
    
    // ❌ 절대 금지 (코어 덤프 유발)
    // printf("받은 값: %s\n", sample->payload.start); 

    // ✅ 베스트 프랙티스 (길이 지정 출력)
    printf("받은 값: %.*s\n", (int)sample->payload.len, sample->payload.start);
}

Zenoh C 프로그래밍에서 String 과 Buffer(Bytes) 를 헷갈리는 순간, 당신의 코드는 우주비행사(메모리 런타임 에러)가 되어 우주 미아가 될 것이다.
