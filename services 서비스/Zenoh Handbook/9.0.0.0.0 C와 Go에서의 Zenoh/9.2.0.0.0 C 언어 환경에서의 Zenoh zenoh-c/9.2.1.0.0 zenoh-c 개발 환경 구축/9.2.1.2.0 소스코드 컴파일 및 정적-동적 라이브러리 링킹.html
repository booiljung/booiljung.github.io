9.2.1.2 소스코드 컴파일 및 정적-동적 라이브러리 링킹

개발망(x86 PC)에서는 잘 돌아가던 C 코드가, 실제 운영서버(Docker)로 파일을 복사해 실행하면 ./robot_node: error while loading shared libraries: libzenohc.so: cannot open shared object file 이라는 에러를 뿜으며 즉사한다.

이는 동적 링킹(Dynamic Linking) 과 정적 링킹(Static Linking) 의 근본적인 차이를 이해하지 못한 대가다.

0.1 [인스펙션] .so (동적) vs .a (정적) 의 전쟁

1. 동적 라이브러리 링킹 (libzenohc.so / .dylib)

  • 빌드 옵션 타겟: target_link_libraries(robot_node zenohc)
  • 장점: 실행 파일(.elf) 크기가 수십 KB로 매우 작다.
  • 단점: 프로그램을 실행하는 리눅스 장비의 /usr/lib 나 LD_LIBRARY_PATH 경로에 반드시 libzenohc.so 파일이 미리 복사되어 있어야만 켜진다. 도커 빌드를 할 때 파일 빼먹으면 무조건 죽는다.

2. 정적 라이브러리 링킹 (libzenohc.a)

  • 빌드 옵션 타겟: 
    # 정적 덩어리를 찾아서 아예 컴파일 할때 내 비즈니스 로직에 합쳐버려라
target_link_libraries(robot_node ${CMAKE_SOURCE_DIR}/libs/zenoh/lib/libzenohc.a)
# (주의: 정적 링킹 시에는 하위 의존성 라이브러리의 링커 에러가 발생할 수 있다. 
#  Ubuntu의 경우 -lpthread, -ldl, -lm 옵션을 꼬리에 반드시 붙여줘야 함)

- **장점:** 만들어진 `robot_node` 실행 파일 단 1개만 상용 깡통 리눅스 칩에 던져 넣어도 외부 종속성 없이 **그냥 켜진다 (Stand-alone).**
- **단점:** 실행 파일 용량이 10MB 가까이 부풀어 오른다 (임베디드 제약 시 주의).

**[권장 아키텍처 결론]**
자율주행 자동차 내부에 들어가는 ECU(단일 컴퓨터)에 코드를 배포할 때는, "환경(Dependencies) 누락에 의한 런타임 크래시"를 원천봉쇄하기 위해 무조건 **정적 링킹(Static Linking)** 을 해서 하나의 거대한 단일 실행 덩어리를 만들어 꽂아넣어라.
