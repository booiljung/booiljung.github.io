9.2.4.1 멀티스레드 환경에서의 동시성 제어 및 스레드 안전성 확보

C 언어의 전역 변수(Global Variable)는 멀티 스레드 콜백 환경에서 쥐약이다. Zenoh의 라우팅 스레드는 언제 어느 때 당신의 Subscriber 콜백 함수를 호출할지 모른다. 두 대의 로봇이 동시에 put 을 때리면 당신의 콜백 함수는 정확히 2개의 스레드에서 동시에 난입(Reentrancy)된다.

0.1 [Runbook] 뮤텍스 볼트(Mutex Vault) 락 전술

데이터 무결성을 지키기 위해 콜백 함수 안에서 pthread 뮤텍스를 건다.

#include <zenoh.h>
#include <pthread.h>
#include <stdio.h>

// 공용 금고 (공유 자원)
long long total_robot_battery = 0;
// 자물쇠
pthread_mutex_t battery_lock = PTHREAD_MUTEX_INITIALIZER;

// 이 콜백함수는 백그라운드의 불특정 Zenoh 워커 스레드에서 무작위로 호출됨
void thread_safe_callback(const z_sample_t* sample, void* context) {
    // 1. 임계 구역 진입 (자물쇠 잠금)
    // 여기서 다른 스레드는 대기(Block)하게 된다.
    pthread_mutex_lock(&battery_lock);
    
    // [보안 구역]
    // 페이로드가 '숫자 문자열' 이라 가정하고 누적한다.
    char buf[32] = {0};
    int copy_len = sample->payload.len > 31 ? 31 : sample->payload.len;
    memcpy(buf, sample->payload.start, copy_len);
    
    total_robot_battery += atoll(buf);
    printf("현재 누적 배터리 용량: %lld\n", total_robot_battery);
    
    // 2. 임계 구역 탈출 (자물쇠 해제)
    // 락(Lock)을 너무 오래 쥐고 있으면 전체 통신망이 병목(Bottleneck) 현상으로 죽어버린다.
    // printf 류의 I/O 함수도 사실 락 구역 밖으로 빼는게 베스트 프랙티스다.
    pthread_mutex_unlock(&battery_lock);
}

이 고전적인 Mutex 방식조차도 로보틱스에서는 “느리다”. 초당 10만 번 데이터를 받아야 한다면 원자적 연산(__atomic_fetch_add)을 사용하는 Lock-Free 아키텍처나 stdatomic.h (C11) 을 투입하는 것을 강하게 권장한다.
