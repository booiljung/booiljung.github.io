9.2.4.4 Shared Memory(SHM) 기반의 프로세스 간 통신 최적화

9.2.4.3에서 배운 Zero-Copy조차 TCP/IP 네트워크나 로컬 Loopback(127.0.0.1) 소켓를 타야 한다.
만약 한 컴퓨터 안에 센서 노드(C), AI 분석 노드(Python), 모터 제어 노드(C) 3개의 프로그램(프로세스)이 떠 있다면? 내 컴퓨터 안에서 소켓을 타는 것조차 레이턴시 낭비다.

OS 레벨의 “공유 메모리(Shared Memory)“에 데이터를 던져넣어 세 프로그램이 동시에 한 주소를 쓰다듬게 만드는 최강의 인프라 튜닝을 가동한다.

0.1 [Runbook] 리눅스 커널 SHM 매핑 전술

코드 변경? 단 0줄이다. 이게 Zenoh 아키텍처의 미친 점이다. 설정만 비틀면 코드는 그대로 둔 채 하부 인프라가 SHM으로 전환된다.

1. 설정 객체에서 SHM 플래그 점화

int boot_shm_session() {
    z_owned_config_t config = z_config_default();
    
    // [비기] "나는 SHM 백엔드를 무조건 쓴다!"
    // 이 명령어 한방에, 이 C 프로그램은 /dev/shm 구역에 거대한 캐시를 뚫는다.
    zc_config_insert_json(z_loan(config), "shared_memory/enabled", "true");
    
    // SHM으로 쏠 임계치 (예: 1024 바이트 이상일 때 무조건 공유메모리에 던짐)
    zc_config_insert_json(z_loan(config), "shared_memory/rx_threshold", "1024");
    
    z_owned_session_t session = z_open(z_move(config));
    
    // 이제 이 이후로 똑같이 z_publisher_put 을 쏘면, 
    // 동일한 컴퓨터 내에 떠있는 Python 스크립트는 소켓이 아니라
    // 이 C 프로그램이 연 /dev/shm 주소값을 훔쳐서 빛의 속도로 읽어간다!!
    return 0;
}

로봇 공학 미들웨어의 절대 강자였던 ROS 2 (DDS)가 수십 기가바이트의 복잡한 패키지를 깔아 치열하게 이룩했던 Shared Memory 아키텍처를, zenoh-c는 단 한 방울의 z_config 설정 문자열로 종결시킨다.
개발자는 z_custom_free 와 Shared Memory 옵션만 켜고, 나머지 데이터 무결성과 크로스 랭귀지 폴백(Python -> C) 통신은 전적으로 엔진에 맡기면 된다.
