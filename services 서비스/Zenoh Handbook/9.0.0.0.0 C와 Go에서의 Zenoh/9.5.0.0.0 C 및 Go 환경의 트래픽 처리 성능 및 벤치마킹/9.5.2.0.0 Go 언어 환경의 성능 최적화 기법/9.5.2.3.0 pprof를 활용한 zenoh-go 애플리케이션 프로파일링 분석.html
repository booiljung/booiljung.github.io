9.5.2.3 pprof를 활용한 zenoh-go 애플리케이션 프로파일링 분석

“어디서 느려지는지 짐작하지 마라. 측정해라.”
Go 엔진이 제공하는 우주 최고의 무기, pprof 를 백엔드 서버에 꽂아 넣는다. 무려 웹 대시보드로 CGO 와 GC 가 언제 얼마나 불탔는지를 증명해 준다.

0.1 [Runbook] 라이브 시스템 프로파일링 파이프라인 개통 전술

package main

import (
	"log"
	"net/http"
	_ "net/http/pprof" // 이거 한줄이면 자체적인 진단 모니터링 데몬이 기동된다!
	"github.com/eclipse-zenoh/zenoh-go"
)

func main() {
	// 1. 운영 서버의 백그라운드에 6060 포트로 프로파일러(X-Ray)를 튼다.
	go func() {
		log.Println("PPROF 진단 서버 개방: localhost:6060")
		log.Println(http.ListenAndServe("localhost:6060", nil))
	}()

	// 2. Zenoh 서버 메인 프로그램 기동
	session, _ := zenoh.Open(zenoh.DefaultConfig())
	defer session.Close()
	
	// ...(어마어마한 트래픽 통신 시뮬레이션)...
	select {}
}

진단(Diagnosis) 방법
서버가 미친 듯이 돌고 있을 때 당신의 노트북 터미널을 열고 다음 명령어를 쏜다.

# 30초 동안 서버의 CPU 점유율 내역을 훔친다 (웹 그래프 표출)
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30

브라우저가 열리며 빨간 네모 칸들(Flame Graph)이 쏟아져 나온다!

  • 만약 runtime.cgocall 박스가 전체의 60%를 차지하고 있다면 -> 9.5.2.1 대로 패킷 병합(Batching)을 수행해라.
  • 만약 runtime.mallocgc (가비지컬렉터 쓰레기수거 작업) 박스가 압도적이라면 -> 빙고! 9.5.2.2 의 sync.Pool 전략을 어기고 매번 make([]byte) 를 생성하고 있다는 명백한 증거다!
