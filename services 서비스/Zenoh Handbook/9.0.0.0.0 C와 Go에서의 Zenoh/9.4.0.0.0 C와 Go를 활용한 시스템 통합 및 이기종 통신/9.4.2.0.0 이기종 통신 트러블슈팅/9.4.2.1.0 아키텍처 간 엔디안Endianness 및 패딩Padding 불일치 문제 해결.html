9.4.2.1 아키텍처 간 엔디안(Endianness) 및 패딩(Padding) 불일치 문제 해결

C 구조체를 날것으로 넘겼을 때 데이터가 파괴되는 두 가지 대원칙이다.

1. 엔디안(Endianness) 함정
숫자 0x11223344 가 메모리에 쓰일 때, Intel 백엔드 서버(리틀 엔디안)는 44, 33, 22, 11 로 기록하지만 다수의 구형 임베디드 칩이나 네트워크 카드 펌웨어(빅 엔디안)는 11, 22, 33, 44 로 보낸다.

0.1 [Runbook] C Network Byte Order 강제 통일 전술

가장 좋은 것은 아예 Protobuf 를 쓰는 것이지만, Raw 바이트를 쏴야 할 땐 무조건 htonl, ntohl 매크로 구역을 통과시켜야 한다.

#include <arpa/inet.h>
// ...
uint32_t my_battery = 100;
// 망(Network) 규격인 빅 엔디안(Big-Endian)으로 강제 변환 후 쏜다!
uint32_t net_battery = htonl(my_battery); 

z_publisher_put(z_loan(pub), (const uint8_t*)&net_battery, sizeof(uint32_t), NULL);

그리고 Go 서버에서는 해당 규약을 알고 있으므로,

// Go 언어는 무조건 BigEndian 으로 읽어들인다.
err := binary.Read(bytes.NewReader(sample.Payload()), binary.BigEndian, &val)

2. 패딩(Size Alignment) 함정
C 언어 컴파일러는 CPU 가 메모리를 쉽게 읽도록, 빈자리에 더미 바이트(Padding)를 집어넣어 구조체를 부풀린다.

struct {
    char a;      // 1 byte
    // --- 3바이트의 보이지 않는 쓰레기(Padding)가 삽입됨 ---
    int32_t b;   // 4 byte
} msg; // 이 구조체의 크기(sizeof)는 5가 아니라 8바이트가 된다!

이걸 무시하고 sizeof(msg) 만큼 통째로 백엔드에 보내면, Go 서버는 값이 3칸씩 밀려버려서 쓰레기값을 DB에 쌓게 된다.
이를 방지하려면 방패를 깔아야 한다. 즉 컴파일러에게 **“메모리 압축 좀 억지로 해달라”**고 지시해야 한다. (GCC 기준 __attribute__((packed))). 이 규칙을 철저히 어기지 않는 하드코어 C 개발자만이 Go 백엔드 팀과 원수(Enemy)가 되지 않는다.
