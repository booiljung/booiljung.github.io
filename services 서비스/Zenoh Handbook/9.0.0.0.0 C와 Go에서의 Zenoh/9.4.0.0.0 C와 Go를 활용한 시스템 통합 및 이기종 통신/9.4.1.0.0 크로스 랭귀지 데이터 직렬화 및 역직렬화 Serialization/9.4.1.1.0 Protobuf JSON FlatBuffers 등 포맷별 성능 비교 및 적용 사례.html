9.4.1.1 Protobuf, JSON, FlatBuffers 등 포맷별 성능 비교 및 적용 사례

JSON 은 사람 눈에 보기 좋지만 기계에게는 폭력이자 자원 낭비의 극치다.
로봇의 X 좌표값 1.234567 을 보내기 위해 {"x": 1.234567} 처럼 17바이트 텍스트를 만들어서 쏘고, 심지어 파싱 과정에서 String-to-Double 변환 연산까지 시키는 것은 배터리 광탈의 주범이다.

0.1 [인스펙션] 로보틱스를 위한 직렬화 무기 선택 가이드

1. JSON (학습용/디버깅용)

  • 특징: C에서는 cJSON 라이브러리로 굽고, Go에서는 encoding/json 으로 파싱한다.
  • 단점: 압도적으로 느리며, 메모리 단편화(Fragmentation)를 유발한다. “1초에 1번씩” 설정값을 내리는 용도로만 허용된다.

2. Protobuf (Protocol Buffers) (일반적인 상용 표준)

  • 특징: .proto 파일 하나를 컴파일하면 C 에서는 Message_pb.h 가, Go 에서는 message.pb.go 가 생성되어 서로의 구조체를 완벽히 결합한다.
  • 성능: 패킷 크기가 JSON 대비 절반 이하로 줄어든다. 파싱 속도도 엄청나게 빠르다. API Gateway 나 K8s 클라우드 백엔드로 데이터를 모을 때 가장 많이 쓰인다.

3. FlatBuffers (극한의 로보틱스 성능 마스터피스)

  • 원리: 아예 “파싱(Unmarshal)” 이라는 과정 자체가 없다. 메모리를 복사하지 않고 바이너리의 특정 오프셋(Offset)에 있는 포인터를 읽어오는 진정한 Zero-copy 포맷이다.
  • 적용처: 1초에 10만 장 쌓이는 고주파 카메라 이미지 스트림이나, 자율주행 차량 내 DDS/ROS 2 토픽 통신 브릿지에 무조건 들어가야 하는 종결자.

[Runbook] C 프로토버프(Protobuf-C) 발사 예제
C에서 패킹하여 쏜다.

RobotStatus msg = ROBOT_STATUS__INIT;
msg.battery = 85; 
msg.x = 1.25;

// 메모리를 계산해서 압축
size_t len = robot_status__get_packed_size(&msg);
uint8_t *buf = malloc(len);
robot_status__pack(&msg, buf);

// Zenoh 로 발사! Payload 는 이 압축된 바이너리다.
z_publisher_put(z_loan(pub), buf, len, NULL);
free(buf);

[Runbook] Go 측 수신(Unmarshal) 예제

sub, _ := session.DeclareSubscriber("robot/status", func(s zenoh.Sample) {
	var msg proto.RobotStatus
	// Go 프로토버프 라이브러리가 바이너리를 구조체에 완벽히 매핑시킴
	err := protobuf.Unmarshal(s.Payload(), &msg)
	
	fmt.Printf("받은 배터리: %d, X좌표: %f\n", msg.Battery, msg.X)
})
