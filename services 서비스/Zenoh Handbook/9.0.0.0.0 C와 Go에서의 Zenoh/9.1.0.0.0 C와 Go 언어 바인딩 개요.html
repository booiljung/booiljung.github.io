9.1 C와 Go 언어 바인딩 개요

Zenoh의 심장인 Rust(zenoh-rs)에서 뻗어 나온 C, Go 바인딩은 단순한 “언어 번역판“이 아니다.
이것은 Rust의 메모리 안전성과 P2P 멀티캐스트 성능을 C 포인터와 Go 채널로 투영시키는 고도의 FFI (Foreign Function Interface) 아키텍처 다리다.

기존 데이터 라우팅(MQTT/DDS) 도구들이 언어마다 패키지 엔진을 새로 깎아서 만들었다면, Zenoh는 “하나의 심장(Rust Core)에 여러 언어의 껍데기(Binding)“를 씌우는 구조를 택했다. 이로 인해 C, Go, Python, TypeScript 간의 통신 속도는 사실상 100% 동일한 네이티브 C 속도에 수렴한다.

1. Zenoh 다국어 지원 아키텍처 이해

Zenoh의 다국어 지원 아키텍처는 거대한 “우산(Umbrella)” 모델이다.

1.0.1 [아키텍처 인스펙션] Core-Binding 스택 구조

  1. zenoh-rs (코어 엔진)
    모든 패킷 해석, 암호화, 라우팅 테이블 트리 순회는 100% Rust 로 작성되어 있다. 메모리 릭(Leak)은 컴파일 타임에 원천 차단되며, 멀티코어를 100% 쥐어짠다.

  2. zenoh-c (중계 허브)
    Rust 코어의 기능들을 C 명명 규칙(C-ABI) 표준을 따르는 함수 포인터로 깎아낸다. z_session_t, z_publisher_t 같은 순수 C 구조체가 이때 만들어진다.
    사실상 세상 모든 언어가 접근할 수 있는 물리적 접점(Socket) 역할을 수행한다.

  3. zenoh-go / zenoh-python (래핑 계층)
    Go 언어의 CGO, Python 의 ctypes, Node.js 의 N-API 가 이 zenoh-c 의 헤더 파일을 물고 들어간다.
    Go 개발자는 void* 를 만질 필요가 없다. 래핑 계층이 C++ 수준의 메모리 포인터를 Go 고루틴 채널(Channel)이나 Python 딕셔너리로 예쁘게 변환해 준다.

결과적으로 Go 에서 session.Put() 을 때리면, 내부적으로는 0.0001초 만에 zenoh-c 의 z_put() 을 거쳐 Rust 엔진으로 직행한다. 이것이 타 브로커 시스템 대비 패킷 지연(Latency)이 150배 가까이 빠른 근본적인 이유다.

2. C API(zenoh-c)와 Go API(zenoh-go)의 위상과 역할

C API 와 Go API 는 서로 경쟁하는 도구가 아니다. 완전히 배치되는 하드웨어 환경을 지배하는 양대 축이다.

2.0.1 C API (zenoh-c) 의 임무: 엣지(Edge)의 칼끝

  • 주 사용처: 자율주행 차량의 실시간 OS (RTOS, QNX), 드론의 임베디드 칩, 로봇의 모터 제어 보드.
  • 특징: 극단적인 저사양. 가비지 콜렉터(GC) 자체가 없다. 개발자가 z_check() 를 통해 메모리를 무조건 해제(Free)해야 한다.
  • 무게: 수십 킬로바이트 메모리만으로도 멀티캐스트 소켓을 뚫어내는 말도 안 되는 경량성.

2.0.2 Go API (zenoh-go) 의 임무: 클라우드 백엔드의 사령탑

  • 주 사용처: 수만 대의 로봇으로부터 날아오는 원격 관제 로그를 ElasticSearch 로 밀어 단체 모니터링을 하는 백엔드 서버(K8s Pod).
  • 특징: Go의 자랑인 고루틴(Goroutine) 과 완벽히 결합한다. session.DeclareSubscriber() 콜백 안에서 패킷이 쏟아져도, Go 채널(Channel)을 통해 초당 백만 개의 데이터를 락(Lock) 없이 분배할 수 있다.
  • 개발 환경: go get github.com/eclipse-zenoh/zenoh-go 로 1초 만에 설치하고 비즈니스 로직에 집중한다.

만약 당신의 팀이 공장 전체를 자동화한다면, 컨베이어 벨트 센서에는 C를 박고 서버실에는 Go를 박는 것이 가장 완벽한 Zenoh 아키텍처다.

3. 시스템 프로그래밍에서 C와 Go 언어의 선택 기준 및 상호 보완성

리눅스 배포판(Ubuntu 등) 위에 시스템 데몬(Daemon)을 작성하려 할 때, 어떤 언어를 골라야 할까? 두 언어의 트레이드오프(Tradoff)를 파헤친다.

3.0.1 [Runbook] 아키텍처 의사결정 나무 (Decision Tree)

전제 1: “메모리가 2MB 도 안 되는 ARM/RISC-V 칩이다.”

무조건 C (zenoh-c / zenoh-pico)
Go 루틴은 초기 부팅 시 상당한 런타임 힙 메모리(수십 MB)를 선점한다. 최소한의 RTOS 환경에서 통신하려면 C 이외의 대안은 존재하지 않는다.

전제 2: “하위 레이어(Kernel C 코드, OpenCV 등) 과 영거리 통신을 해야 한다.”

주로 C 계열 권장
Go 에서 CGO 를 이용해 C++ 라이브러리(OpenCV 등)로 접근할 때는 컨텍스트 스위칭(Context Switching) 오버헤드가 발생한다. 스트림 라이다(LiDAR) 데이터 같은 것들은 C로 순수 메모리 연산을 하는 것이 압도적으로 빠르다.

전제 3: “API Gateway를 짜거나, 동시에 1,000개의 로봇에게 쿼리를 뿌려야 한다.”

압도적으로 Go (zenoh-go)
C 로 멀티 스레드 소켓 1,000개를 관리(Epoll/Select)하는 코드를 짜면 소스 코드가 3,000줄이 넘고 메모리 누수로 고통받는다. Go 언어는 for 루프 안에서 go FetchRobotInfo(i) 를 치는 순간 고루틴 1,000개가 알아서 네트워크 비동기 블로킹을 처리해 준다.

[상호 보완성 플로우]
공장 바닥의 로봇(C 환경)에서 FlatBuffers 로 된 바이트 배열 덩어리를 쏘면,
라우터망을 거쳐 클라우드의 집계 서버(Go 환경)가 이 덩어리를 받아 Go Struct 로 순식간에 파싱하여 DB에 처넣는다.
언어의 런타임 제약에 종속되지 않는 것, 이것이 진정한 Zenoh 아키텍처 결속이다.
