8.1 TypeScript와 Zenoh의 통합 개요

1990년대의 브라우저 장난감 언어였던 JavaScript는, 이제 Node.js와 TypeScript라는 거대한 갑옷을 입고 클라우드 백엔드와 프론트엔드를 모두 지배하는 황제로 군림하고 있다.
기존의 웹 생태계는 클라이언트가 리퀘스트(Request)를 보내면 서버가 리스폰스(Response)를 주는 수동적인 HTTP(REST) 구조나, 기껏해야 중앙 집중형 브로커에 매달린 WebSocket 통신에 갇혀 있었다.

하지만 만약 웹 브라우저 자체가 거대한 분산 네트워크망의 P2P 노드 중 하나로 직접 편입될 수 있다면 어떨까? 중간에 서버나 데이터베이스를 거치지 않고, 스마트폰의 크롬 브라우저가 공장에 있는 로봇의 휠 베어링 온도를 마이크로초 단위로 직접 뽑아올 수 있다면?

이 장에서는 왜 TypeScript 생태계가 Zenoh를 만나야만 했는지, 이들의 만남이 기존의 REST / GraphQL / Socket.io 생태계를 어떻게 파괴하고 혁신하는지 그 아키텍처의 포문을 연다. 단순한 라이브러리 소개를 넘어, 진정한 ’분산형 풀스택(Full-Stack) 엔지니어’로 거듭나기 위한 런북(Runbook)의 시작이다.

1. 웹 및 JS/TS 생태계에서의 분산 통신 패러다임 변화

웹 개발자에게 로보틱스 통신(DDS, ROS)은 외계어와 같았다.
웹 개발자가 로봇의 데이터를 화면(React/Vue)에 띄우기 위해서는 다음과 같은 비참한 구조를 짜야만 했다.

1.0.1 [인스펙션] 레거시(Legacy) 통신 아키텍처의 한계

1. 폴링(Polling)의 지옥 (REST API 시대)
로봇(C++) ➔ HTTP/POST ➔ 웹 서버(Node.js) ➔ DB 저장.
사용자 웹 브라우저는 1초에 한 번씩 서버로 HTTP/GET을 날려 “데이터 들어온 거 있어?“를 묻는다. (트래픽의 99%가 HTTP 헤더와 빈 껍데기 응답으로 버려짐)

2. 소켓/브로커 병목 (WebSocket / MQTT 시대)
로봇 ➔ MQTT Broker ➔ 웹 서버(Node.js) ➔ WebSocket ➔ 브라우저.
연속성이 확보되어 조금 나아졌지만, 결국 중간의 브로커(Broker) 서버가 죽으면 모든 화면이 멈춘다. 게다가 웹 개발자는 서버쪽 라우팅 로직과 클라이언트쪽 소켓 리스너 로직을 이중으로 관리해야 한다.

1.0.2 [패러다임 시프트] Zenoh의 도입 (Data-Centric Web)

Zenoh가 TypeScript 생태계에 들어오면서 중간 다리(Middleware Server)들이 모조리 박살난다.

  • P2P 직결 통신: 브라우저(TypeScript)가 Zenoh Router/Peer 인프라의 일부로 작동한다.
  • 서버리스(Server-less) 로보틱스: Node.js 기반의 중계 백엔드가 필요 없다. 브라우저에서 session.get("factory/robot/1/cpu") 를 치는 순간, 이 쿼리는 통신망을 타고 C++ 로봇 기체에 직접 꽂힌 뒤 브라우저 화면으로 밀리초 단위로 되돌아온다.
  • 완벽한 타입 안정성 보장: TypeScript의 정적 타이핑(Interface/Type)을 활용하면, 로봇이 쏘는 바이너리 데이터를 프론트엔드 컴포넌트까지 런타임 에러 0% 로 안전하게 캐스팅할 수 있다.

이제 웹 프론트엔드 개발자는 단순히 “화면을 그리는 사람“이 아니라, “데이터 투사망(Data Mesh)의 최종 수신 터미널을 구축하는 네트워크 오퍼레이터“로 진화한다.

2. Zenoh TypeScript API의 아키텍처와 내부 동작 원리

npm install @eclipse-zenoh/zenoh 를 쳤을 때, 과연 내 프로젝트에는 순수 자바스크립트가 깔릴까?
반은 맞고 반은 틀리다. Zenoh TypeScript 바인딩은 구동되는 환경(Environment)에 따라 밑바닥 엔진(Engine)이 카멜레온처럼 변이하는 투 트랙(Two-Track) 아키텍처다.

2.0.1 [아키텍처 인스펙션] 이중 엔진 분기 시스템

Zenoh TS 코어는 자신이 현재 구동되는 곳이 ’브라우저’인지, 아니면 ‘Node.js’ 서버인지 스스로 감지하고 다른 심장을 꺼낸다.

1. Node.js (서버 백엔드) 환경 탑재 시: N-API (Rust Native Binding)

  • Node.js나 Deno 같은 백엔드 런타임에서 동작할 때는 C/Rust 기반의 운영체제 네이티브 엔진(zenoh-rs)이 Node의 N-API를 통해 직접 물린다.
  • 성격: 극강의 처리 속도(Throughput), 진정한 Zero-Copy 구현, 멀티캐스트 스카우팅(Scouting)을 통한 네트워크 자동 탐색 허용.
  • 패킷은 TCP/UDP 소켓을 직접 두드리며 날아간다.

2. 웹 브라우저 (React, Vue 프론트엔드) 탑재 시: WebAssembly (WASM) + WebSocket

  • 구글 크롬이나 사파리 브라우저는 보안(Sandbox) 특성상 개발자가 맘대로 TCP 포트를 열고 멀티캐스트 UDP를 쏠 수 없다. 스카우팅(Scouting) 기능이 원천 차단된다.
  • 성격: Zenoh Rust 코어는 WASM(웹어셈블리) 로 컴파일되어 브라우저 메모리 안으로 들어온다.
  • 통신 경로: 브라우저 런타임은 TCP 대신 브라우저의 유일한 양방향 통로인 WebSocket (WS/WSS) 포트를 뚫어서 클라우드에 떠 있는 Zenoh Router에 달라붙는다 (client 모드 강제).

결론 (단일화된 API)
놀랍게도 TypeScript 개발자 앞에는 이 두 개의 복잡한 엔진이 100% 똑같은 interface 의 API 하나로 추상화되어 제공된다.
개발자는 await zenoh.open(config) 단 한 줄만 치면 끝이다. 서버에서 짠 Pub/Sub 비즈니스 로직(TS 파일)을 복사해서 프론트엔드 React 프로젝트에 붙여 넣어도 한 줄의 코드 수정 없이 작동하는 기적이 발생한다.

3. 실행 환경별 차이점 이해 (Node.js vs Deno vs 웹 브라우저)

TypeScript 코드를 짰다고 해서 어디서나 똑같이 굴러갈 것이라 믿는 건 초보자의 오만이다. 이 엔진들이 메모리와 네트워크를 대하는 태도(Sandbox)가 완전히 다르기 때문이다.

3.0.1 [Runbook] 실행 환경별 전술 가이드라인

1. Node.js (V8 기반 운영체제 장악자)

  • 용도: 데이터 분석 마이크로서비스, 영상 트랜스코딩 백엔드, 로그 수집기.
  • 통신 제원: OS의 TCP/UDP 소켓을 마음대로 연다. Zenoh의 Peer 모드나 Router 모드까지 모두 자력으로 구동할 수 있다.
  • 특징: 멀티캐스트 스카우팅(Scouting) 레이더를 통해 주변 네트워크의 로봇이나 다른 서버를 알아서 찾아내어 그물망을 짠다.

2. Deno / Bun (모던 시큐리티 런타임)

  • 용도: 엣지(Edge) 함수, 보안이 극도로 중요한 클라우드 함수리스(Serverless).
  • 특징: Deno는 태생적으로 네트워크 접근 권한을 거부한다.
  • 조치: 스크립트 실행 시 반드시 deno run --allow-net --allow-env my_zenoh_app.ts 처럼 네트워크 커스텀 권한을 뚫어주어야만 Rust 엔진이 소켓을 열 수 있다. 그렇지 않으면 부팅 즉시 권한 에러(Permission Denied)를 뱉고 폭발한다.

3. Web Browser (가장 억압된 클라이언트)

  • 용도: 관리자 대시보드, 차량 모니터링 HUD 패널.
  • 통신 제원: 오직 WebAssembly(WASM) 엔진으로만 구동되며, 통신 채널은 오직 WebSocket 뿐이다.
  • 치명적 제약:
  1. Multicast Scouting 불가능. 무조건 타겟 Zenoh Router의 IP 주소(예: ws://10.0.0.5:7447)를 하드코딩하거나 환경변수로 주입해서 open() 해야 한다.
  2. 노드 모드는 무조건 Client 모드만 허용된다. (브라우저가 다른 사람의 데이터를 중계해 주는 라우터가 될 순 없다).
  3. TCP/UDP 터널을 쓰는 로봇들과 직접 맞닿을 수 없으므로, 로컬망에 반드시 브릿지 역할을 해줄 Zenoh Router 프로세스를 하나 띄워두어야 한다.

아키텍트의 룰(Rule)
당신이 진정한 분산 로보틱스망을 설계한다면, 화면(Browser)에서 뷰(View)를 조작하고(Client), 그 뒷단(Node backend)에서 라우팅 핑을 조율하는(Peer) 이중 트랙 아키텍처를 정확히 설계해야 한다. 성능 최적화의 첫걸음은 실행 환경(Runtime)의 한계를 인정하는 것에서 출발한다.

4. 로보틱스 및 IoT 대시보드에서 TypeScript의 역할

자율주행 차량 엔지니어링 생태계를 생각해보자. 센서를 달고 통신을 뚫어주는 일은 C++ / Rust 팀이 한다. 하지만 최종적으로 투자자에게 데모를 시연하거나, 공장 관리자가 모니터링할 “관제 화면 (Control Tower HUD)” 은 무조건 React, Vue.js 같은 TypeScript 생태계에서 탄생한다.

4.0.1 [아키텍처 인스펙션] TypeScript가 맡아야 할 3가지 특명 전술

웹 프론트엔드가 단순히 “가벼운 뷰어“에서 “핵심 관제 사령탑“으로 격상되는 포인트다.

1. 실시간 텔레메트리 렌더링 (Real-time Rendering)
로봇이 1초에 60번씩 쏘는 배터리, 모터 RPM, 좌표 정보(Pub/Sub)를 받아서 차트(Chart.js)나 3D 캔버스(Three.js)에 뿌려준다.
REST 폴링(Polling)의 “깜빡거림” 없이, WASM 소켓을 통해 들어온 바이너리를 프론트엔드 메모리 상에서 TypeScript의 DataView 나 TypedArray 로 받아(Zero-copy) 60FPS의 화면 주사율에 완벽히 동기화시킨다.

2. 분산 명령 하달 체계 (Action Executor)
관리자가 웹 브라우저 화면의 빨간색 “EMERGENCY STOP” 버튼을 누른다.
이때 더 이상 중앙 웹 서버로 HTTP POST /api/stop 을 호출하지 않는다.
브라우저가 그 즉시 P2P Query (session.get("robot/*/cmd/stop")) 를 발동시켜 로컬 네트워크의 모든 로봇 브레이크를 밀리초(ms) 단위로 직접 잠가버린다. 웹 브라우저 자체가 가장 강력한 무기가 된다.

3. 비정형 데이터(JSON)와의 찰떡 궁합
로봇의 설정 값이나 메타데이터들은 엄청나게 복잡한 계층(Tree) 구조를 가진 JSON 텍스트로 오간다.
C++에서는 파서를 만들어야 하지만, TypeScript는 그 자체가 JSON의 고향(JavaScript Object Notation)이다. 네트워크에서 건져 올린 바이트에 JSON.parse(payload) 를 걸고, 선행 정의해둔 TypeScript 컴파일러 인터페이스 interface RobotStatus { ... } 로 캐스팅(강제 주입)하면 완벽한 타입-추론(Auto-completion)의 가호 아래 무결점 대시보드를 10배 빨리 개발할 수 있다.

이것이 C++ 통신 제국이 TypeScript/Web 생태계를 절대로 무시할 수 없는, 오히려 완전히 종속 수용해야 하는 아키텍처적 당위성이다.
