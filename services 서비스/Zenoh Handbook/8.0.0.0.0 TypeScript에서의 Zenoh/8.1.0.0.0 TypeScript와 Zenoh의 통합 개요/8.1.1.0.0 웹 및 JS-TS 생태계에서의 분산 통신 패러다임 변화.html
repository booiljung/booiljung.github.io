8.1.1 웹 및 JS/TS 생태계에서의 분산 통신 패러다임 변화

웹 개발자에게 로보틱스 통신(DDS, ROS)은 외계어와 같았다.
웹 개발자가 로봇의 데이터를 화면(React/Vue)에 띄우기 위해서는 다음과 같은 비참한 구조를 짜야만 했다.

0.1 [인스펙션] 레거시(Legacy) 통신 아키텍처의 한계

1. 폴링(Polling)의 지옥 (REST API 시대)
로봇(C++) ➔ HTTP/POST ➔ 웹 서버(Node.js) ➔ DB 저장.
사용자 웹 브라우저는 1초에 한 번씩 서버로 HTTP/GET을 날려 “데이터 들어온 거 있어?“를 묻는다. (트래픽의 99%가 HTTP 헤더와 빈 껍데기 응답으로 버려짐)

2. 소켓/브로커 병목 (WebSocket / MQTT 시대)
로봇 ➔ MQTT Broker ➔ 웹 서버(Node.js) ➔ WebSocket ➔ 브라우저.
연속성이 확보되어 조금 나아졌지만, 결국 중간의 브로커(Broker) 서버가 죽으면 모든 화면이 멈춘다. 게다가 웹 개발자는 서버쪽 라우팅 로직과 클라이언트쪽 소켓 리스너 로직을 이중으로 관리해야 한다.

0.2 [패러다임 시프트] Zenoh의 도입 (Data-Centric Web)

Zenoh가 TypeScript 생태계에 들어오면서 중간 다리(Middleware Server)들이 모조리 박살난다.

  • P2P 직결 통신: 브라우저(TypeScript)가 Zenoh Router/Peer 인프라의 일부로 작동한다.
  • 서버리스(Server-less) 로보틱스: Node.js 기반의 중계 백엔드가 필요 없다. 브라우저에서 session.get("factory/robot/1/cpu") 를 치는 순간, 이 쿼리는 통신망을 타고 C++ 로봇 기체에 직접 꽂힌 뒤 브라우저 화면으로 밀리초 단위로 되돌아온다.
  • 완벽한 타입 안정성 보장: TypeScript의 정적 타이핑(Interface/Type)을 활용하면, 로봇이 쏘는 바이너리 데이터를 프론트엔드 컴포넌트까지 런타임 에러 0% 로 안전하게 캐스팅할 수 있다.

이제 웹 프론트엔드 개발자는 단순히 “화면을 그리는 사람“이 아니라, “데이터 투사망(Data Mesh)의 최종 수신 터미널을 구축하는 네트워크 오퍼레이터“로 진화한다.
