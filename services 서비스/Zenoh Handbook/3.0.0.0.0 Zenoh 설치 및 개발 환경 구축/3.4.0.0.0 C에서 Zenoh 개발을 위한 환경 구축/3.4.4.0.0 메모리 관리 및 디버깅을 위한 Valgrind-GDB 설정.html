3.4.4 메모리 관리 및 디버깅을 위한 Valgrind/GDB 설정

C언어로 Zenoh 애플리케이션을 개발할 때 버그의 90%는 통신망에서 오지 않는다. 어설픈 포인터 접근(Segfault)과 C 구조체 동적 할당 직후 해제 누락에 의한 끔찍한 **메모리 릭(Memory Leak)**에서 온다.
로봇을 켰을 때는 멀쩡히 잘 통신하다가 정확히 12시간 뒤에 C 프로세스가 메모리를 다 갉아먹고 로봇을 정지시킨다면, 이보다 끔찍한 현장 사고는 없다.

0.1 Zenoh-C의 메모리 철학 (수동 해제 원칙)

바인딩 라이브러리는 수많은 z_xxx_t 구조체들을 던져준다. 이 녀석들은 개발자가 다 쓰고 나면 z_drop()이나 구조체별 해제 함수를 통해 반드시 관을 짜서 묻어줘야 한다.

z_target_t target = Z_TARGET_BEST_MATCHING;
z_owned_reply_channel_t channel = z_default_reply_channel();
// ... Get 통신 처리 로직 ...

// [안티 패턴] 다 쓰고 그냥 함수 반환. 
// 이 순간 채널 버퍼 메모리는 영구 미아(Leak)가 된다.

// [정답 런북] C언어의 운명이자 저주 - 내가 연 것은 무조건 내가 부순다.
z_drop(&channel);

0.2 [Runbook] 메모리 누수 방탄 조끼 - Valgrind 강제 검진

사내 배포 파이프라인에서 “개발자가 짠 코드를 Valgrind에 통과시키지 않고 현장에 깔면 즉시 시말서“라는 룰을 세워야 한다.

  1. 디버그 심볼 주입(컴파일):
    CMake 시 최적화를 풀고 -g -O0 옵션으로 뇌 해부도(디버그 심볼)를 온전히 남긴 채 애플리케이션을 빌드한다.
  2. 독극물(Valgrind) 검사 루틴 가동: 
    # Valgrind로 앱을 포박한 채 실행한다.
# 내부에서 1 바이트라도 새나가면(leak) 정확한 코드 라인수를 색출하여 사형선고를 내린다.
valgrind --leak-check=full --show-leak-kinds=all --track-origins=yes ./my_robot_app

이 명령어를 거쳤을 때 `definitely lost: 0 bytes` 라는 무결한 성적표가 나오기 전까지, 당신의 로봇은 시한폭탄이다. 비동기로 휘몰아치는 ZBytes 패킷들을 C 레벨 무주공산으로 풀어놓는 행위가 얼마나 가혹한 책임감을 수반하는지 C 엔지니어의 뼛속에 각인시켜야 한다.
