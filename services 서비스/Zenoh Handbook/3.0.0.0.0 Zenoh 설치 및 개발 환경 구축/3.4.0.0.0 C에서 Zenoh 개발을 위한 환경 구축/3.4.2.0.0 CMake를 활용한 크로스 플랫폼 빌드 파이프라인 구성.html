3.4.2 CMake를 활용한 크로스 플랫폼 빌드 파이프라인 구성

어느 미친 인프라 팀도 “개발자가 수동으로 헤더 복사해서 gcc -lzenohc main.c 치고 앉아있기“를 용납하지 않는다.
수만 대의 로봇 펌웨어를 뽑아내야(Cross Compile) 하는 실무 환경에서는 오직 CMake의 시스템 인스펙션 기능과 빌드 파이프라인 자동화만이 살길이다.

0.1 [Runbook] zenoh-c를 품은 CMakeLists.txt 정석

현장 센서 앱을 위한 가장 견고한 CMakeLists.txt 스켈레톤 코드는 다음과 같다.

cmake_minimum_required(VERSION 3.12)
project(RobotSensorApp C)

# 최신 표준 강제
set(CMAKE_C_STANDARD 99)
set(CMAKE_C_STANDARD_REQUIRED ON)

# 1. Zenoh-C가 설치된 (혹은 빌드된) 디렉토리 경로 주입
# CMake 실행 시 -DZENOHC_DIR=/my/zenohc/path 옵션으로 외부에서 밀어넣을 수 있게 변수화
set(ZENOHC_DIR "" CACHE PATH "Path to zenoh-c installation")

# 2. 헤더(.h) 파일 주소 알려주기
include_directories(${ZENOHC_DIR}/include)

# 3. 내 소스코드 컴파일 선언
add_executable(robot_sensor src/main.c)

# 4. OS별 라이브러리 파일(.so 또는 .dylib) 찾아서 갖다 붙이기 (Linking)
find_library(ZENOHC_LIB NAMES zenohc PATHS ${ZENOHC_DIR}/lib)

if(NOT ZENOHC_LIB)
    message(FATAL_ERROR "Zenoh-C 링킹 라이브러리를 찾지 못했습니다! 경로를 확인하세요.")
endif()

# 5. 최종 융합!
target_link_libraries(robot_sensor ${ZENOHC_LIB})

0.2 [트러블슈팅] 동적 링킹(so 파일) 못 찾음 에러 (Runtime Crash)

빌드는 기가 막히게 성공시켰는데, 에지 기기에서 ./robot_sensor 실행 파일을 치자마자 error while loading shared libraries: libzenohc.so 라며 데드 크래시를 띄운다?
이건 컴파일은 성공했지만 프로그램 구동 시점에 동적 라이브러리(LD_LIBRARY_PATH) 메모리 참조 주소가 끊긴 것이다.

처방전 1: 실행 전에 환경변수를 밀어 넣는다.

export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/경로/zenoh-c/lib
./robot_sensor

처방전 2 (영구 해결책): 서버의 /usr/local/lib에 libzenohc.so를 진짜 복사하고 sudo ldconfig 명령어를 치면 시스템 글로벌 라이브러리로 봉쇄되어 어느 폴더에서 실행하든 정상 기동한다.
