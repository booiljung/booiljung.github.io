3.4.3 zenoh-c 바인딩 라이브러리 빌드 및 프로젝트 연동

바이너리를 남이 구워놓은걸(.zip) 다운받는 건 쉽다. 하지만 보안 통제 구역 내이거나 특수 임베디드 장비(ARM32 등)에 포팅하기 위해선, 오픈소스를 바닥부터 끌어와 C 바이너리로 뽑아내는 일련의 제련공정을 내재화해야 한다.

0.1 [Runbook] zenoh-c 맨손 컴파일링 파이프라인

위에서 배운 Rust 환경과 CMake 도구가 융합되는 지점이다.

# 1. 깃허브에서 바인딩 스켈레톤 추출
git clone https://github.com/eclipse-zenoh/zenoh-c.git
cd zenoh-c

# 2. 빌드 공장지대 생성 (OOP 원칙)
mkdir build && cd build

# 3. 릴리즈 모드(속도) 환경 셋업
# 이 과정 안에서 CMake가 자동으로 Rust의 'cargo build'를 배후 호출하여 
# Rust 엔진을 C 라이브러리(.so)로 감싸는 놀라운 짓을 수행한다.
cmake -DCMAKE_BUILD_TYPE=Release ..

# 4. 물리적 컴파일 집행!
make -j $(nproc) # 서버 CPU 코어 갯수만큼 풀 배기통 열어라!

0.2 링킹의 두 가지 치명적 선택로

빌드가 완료되면 폴더 안에 2개의 황금 열쇠가 떨어진다. libzenohc.so (동적) 와 libzenohc.a (정적). 당신의 제품 특성에 맞춰 런북을 쪼개라.

  • 방법 A (동적 링킹 .so 활용):
  • 특징: 당신이 짠 C 실행 파일 용량이 매우 작아진다. (몇 KB 수준) 대신, 실행될 로봇 장비에 반드시 libzenohc.so 파일을 동봉해서 배포해야 하는 파이프라인의 짜증남이 존재한다. 범용 리눅스 환경에 적합하다.
  • 방법 B (정적 링킹 .a 활용 - 권장):
  • 특징: Zenoh 엔진을 통째로 당신의 main.c 컴파일 바이너리 안에다 흡수 결합시켜(Fat Binary) 버린다. 파일 용량은 10~20MB로 비대해지지만, 이 실행 파일 하나만 로봇에 띡 던져 넣으면 추가 의존성 없이 완벽하게 독립 실행(Standalone)되는 축복을 누릴 수 있다. 도커 최소 경량 컨테이너(Scratch/Alpine) 배포 시 필수 전략이다.
