3.3.4 IDE 환경(rust-analyzer) 및 디버깅 도구 연동 설정

Rust로 거대한 분산 시스템을 개발할 때 VIM에 문법 하이라이터만 깔고 코딩하는 것은 “난 맨눈으로 런타임 버그의 원인을 잡겠다“는 오만이다. Zenoh는 수백 개의 제네릭 인자와 비동기 트레이트로 엮여있어, 강력한 코드 분석기와 메모리 디버거의 보조 없이 다루기 불가능에 가깝다.

0.1 단계: Rust-Analyzer (두뇌) 주입

VSCode나 CLion에 rust-analyzer 익스텐션을 설치하라.
Zenoh 특유의 session.declare_subscriber(..).res().await.unwrap() 같은 미친 듯이 연쇄(Chaining)되는 비동기 함수의 중간 리턴 타입(Type Inference)을 눈으로 확인할 수 있게 해 준다.

[IntelliSense 튜닝 런북]
VSCode settings.json을 수정하여 컴파일 매크로 분석 기능을 최상으로 끌어올린다.

{
  // 비동기 매크로(tokio)나 proc-macro 확장을 정확히 파싱해 빨간 줄을 제거
  "rust-analyzer.cargo.buildScripts.enable": true,
  "rust-analyzer.procMacro.enable": true,
  "rust-analyzer.lens.references": true
}

0.2 단계: 크래시 발생 지점 추출 (Panic Backtrace)

공장 돌리다가 뜬금없이 unwrap() 때문에 패닉이 터졌다. 그냥 thread 'main' panicked... 만 뜨면 수만 줄의 코드 중 어디서 네트워크가 찢어졌는지 찾을 길이 없다.
배포 런북 스크립트 작성 시 무조건 아래 환경 변수를 강제 주입하라.

# 운영 배포 시 반드시 적용해야 할 백트레이스 치트키
export RUST_BACKTRACE=1
# 더 변태적인 메모리 추적을 원한다면
export RUST_BACKTRACE=full

이 환경 변수가 켜져 있어야, 에러가 터졌을 때 “zenoh-core의 34번 라인 -> 내 소스코드 12번 라인 client.rs” 식의 스택 콜(Call Stack)이 예쁘게 출력된다.

0.3 단계: 런타임 디버깅 (LLDB / GDB) 타겟팅

비동기 문맥 전환 때문에 일반 디버거로는 태스크를 쫓아다니기 힘들지만, VSCode CodeLLDB 익스텐션을 결합하면 특정 Zenoh 메시지 송신 시점에 브레이크포인트(Breakpoint)를 박고 통신망의 페이로드가 정확히 파싱되어 메모리에 박혔는지 실시간 적출이 가능하다. 이는 JSON이 아닌 바이너리 ZBytes를 개발할 때 필수적인 관측 수단이다.
