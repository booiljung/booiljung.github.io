3.3.3 비동기 프로그래밍(Tokio)과의 호환성 설정

Zenoh는 1초에 수백만 개의 패킷이 오가는 스트림을 다루기 위해 Rust 생태계의 비동기 제왕인 Tokio 런타임 위에서 작동하도록 심장이 설계되었다. 당신이 이 사실을 간과하고 옛날 방식의 동기식(Sync) 스레딩 블로킹 코드를 짜는 순간 네크워크 스루풋(Throughput)은 나락으로 떨어진다.

0.1 [Runbook] Tokio 스레드 풀 오케스트레이션

Zenoh API 함수 대부분이 .await를 요구한다. 메인 함수에 #[tokio::main] 매크로 하나 붙이고 “끝났다“고 방관하면, 코어가 64개인 서버에서도 단 몇 개의 워커 스레드만 맴돌며 병목 현상이 발생한다.

하이엔드(High-end) 디바이스를 위한 튜닝 (ex. 자율주행 관제 서버)

// [오답] 범용 무지성 설정
// #[tokio::main] 

// [정답] Zenoh 네트워크 I/O 전용 스레드 극한 할당
// "로봇에서 날아오는 Pub/Sub 이벤트를 처리하기 위해 Tokio 워커 쓰레드를 강제 16개로 박아넣겠다!"
#[tokio::main(flavor = "multi_thread", worker_threads = 16)]
async fn main() {
    let config = zenoh::Config::default();
    let session = zenoh::open(config).res().await.unwrap();
    // ...
}

[트러블슈팅] 비동기 함수 내에서의 Blocking 파국
이벤트 구독 콜백(.declare_subscriber()) 안에서 파일을 디스크에 쓰거나(File I/O) 데이터베이스 쿼리를 동기적으로 돌려버리면(Blocking), 해당 Tokio 스레드 전체가 잠시 기절(Freeze)한다. 이 찰나의 순간, 라우터에서 밀려오는 다음 번 Zenoh 패킷들은 메모리 큐에 대기하다가 큐가 꽉 차면 Drop 되어버린다(데이터 유실).
구독 콜백 안에서는 반드시 tokio::task::spawn_blocking을 이용해 연산 부하를 Zenoh 네트워크 I/O 스레드 풀에서 타 스레드 풀로 분리(Offloading)시켜 버리는 비동기 위생 철칙을 준수해야 한다.
