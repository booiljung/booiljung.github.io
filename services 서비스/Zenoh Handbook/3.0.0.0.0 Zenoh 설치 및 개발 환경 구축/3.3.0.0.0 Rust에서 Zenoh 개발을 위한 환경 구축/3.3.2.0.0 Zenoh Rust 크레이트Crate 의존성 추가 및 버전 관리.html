3.3.2 Zenoh Rust 크레이트(Crate) 의존성 추가 및 버전 관리

Cargo.toml 파일에 패키지 한 줄 추가하는 게 뭐 대수냐고 생각할 수 있겠지만, 마이크로서비스 생태계에서 ’프로토콜 버전 불일치’는 레이더에 잡히지도 않는 가장 끔찍한 사일런트 킬러(Silent Killer)다.

0.1 [Runbook] 프로토콜 락-인(Lock-in)과 Cargo 튜닝

Zenoh는 현재 안정화되지 않은 0.10.x 메이저 버전을 통과하고 있다. 0.8 버전의 네트워크 데몬과 0.10 버전의 Rust 클라이언트 라이브러리는 통신 와이어 단에서 호환되지 않는다(Handshake Reject). 따라서 와일드카드나 카로트(^)를 쓴 의존성 선언은 미친 짓이다.

[Cargo.toml 모범 답안]

[dependencies]
# 1. 절대적 핀포인트 버전 고정 (안티 패턴: zenoh = "0.10" -> x)
# 무조건 패치 버전까지 완전체로 명시하여 의도치 않은 자동 업데이트를 막아라.
zenoh = "=0.10.1-rc"

# 2. 불필요한 기능(Feature) 가지치기
# "나는 라우터나 P2P 모드 통신은 안 하고, 에지에서 단순 'Client' 로만 접속할 거다"
# 그러면 무거운 default_features를 폭파시키고 핵심 기능만 살려 컴파일 타임을 3배 압축시켜라.
zenoh = { version = "=0.10.1-rc", default-features = false, features = ["transport_tcp"] }

0.2 사내 패키지 파이프라인 단절 대처 (Offline Build)

보안이 생명인 국방/의료 망에서는 외부 인터넷망인 crates.io에 접근하지 못해 빌드가 박살난다. 인프라 팀은 런북에 따라 로컬망 내륙에 Nexus Repository나 git submodule 기반으로 Zenoh Cargo 소스를 벤더링(Vendoring)하여, 폐쇄망 내부망에서만 의존성을 당겨오도록 .cargo/config.toml 거울(Mirror) 설정을 우회시켜야 한다.
