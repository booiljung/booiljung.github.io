3.7.1 상태 모니터링을 위한 zenoh-plugin-metrics 컴파일 및 로드

기본적인 순정(Vanilla) Zenoh 라우터 바이너리는 철저히 데이터의 전달 그 자체에만 미친 속도로 몰입하도록 다이어트되어 있다. 즉, 스스로 얼마나 CPU를 쓰고 있고 RAM 캐시를 비우는지 밖으로 동네방네 떠들도록 허용하지 않는다. 이 입 무거운 라우터에게 “너의 건강 상태 수치(Metrics)를 외부로 말하게 하라!“는 권력을 부여하려면, 개발자는 반드시 위대한 플러그인(Plug-in) 메커니즘을 소환하여 감시관 모듈, 이른바 **zenoh-plugin-metrics**를 코어에 동적으로 접착(Load)해 주어야 한다.

만일 여러분이 소스 기반의 Rust 컴파일러 통제권(Cargo)을 지니고 있다면, 가장 파괴적이고 확실한 모니터링 체계를 갖추는 수술은 의외로 간단하다.

  1. 플러그인 모듈 컴파일 (공장 가동)
    Zenoh 공식 깃허브 바다에서 메인 소스코드를 건져 올려 당신 컴파일 구역(~/zenoh)으로 가져온 뒤, 수많은 플러그인 묶음 중 plugins/zenoh-plugin-metrics 폴더로 향하라. 그리고 터미널에서 cargo build --release 망치질 하나만 찍어내면 수분의 엔진 가열과 함께 당신만의 우분투(Ubuntu) 환경에 딱 맞는 libzenoh_plugin_metrics.so (혹은 윈도우의 .dll, 맥의 .dylib) 단일 모듈 부품이 탄생한다.

  2. 라우터 설정 파일(zenoh.json) 수술과 런타임 동적 합체(Dynamic Load)
    이제 여러분이 실행할 순정 zenohd는 저 부품의 존재를 알아야만 한다. 라우터를 구동할 때 곁들일 당신의 설정(Config) 도화지 파일 속에 단 몇 줄만 지시를 기입하라.

{
  "plugins": {
    "metrics": {
      "__required__": true,
      "address": "0.0.0.0:8000"
    }
  }
}

터미널에서 zenohd --config my_config.json 명령으로 우렁차게 라우터 엔진을 시동 거는 순간, 백그라운드 코어 엔진은 방금 전 컴파일했던 저 .so 동적 라이브러리 부품을 메모리로 스스로 빨아들여 흡수결합한다.

그 결과, 아무도 접근 못 하던 라우터가 갑자기 8000번 포트 웹페이지(HTTP) 샛문을 활짝 열어젖히며, “내 포워딩 대기 패킷 큐 수치가 50개이고 초당 스루풋 파이프 수용량은 3GB다!” 라고 전 세계 모니터 관제 서버들이 언제든 읽어갈 수 있도록 적나라한 숫자 텍스트를 끝없이 토해내는, 완벽한 투시경 상태계로 거듭나게 된다.
