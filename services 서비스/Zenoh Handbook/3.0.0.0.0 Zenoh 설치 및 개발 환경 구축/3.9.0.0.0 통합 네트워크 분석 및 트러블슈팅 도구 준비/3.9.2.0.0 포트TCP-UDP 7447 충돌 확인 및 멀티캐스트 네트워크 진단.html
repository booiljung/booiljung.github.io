3.9.2 포트(TCP/UDP 7447) 충돌 확인 및 멀티캐스트 네트워크 진단

이 엄청난 라우터와 여러분이 작성한 어플리케이션 코어 사이를 연결하느라 쉴 새 없이 몰아칠 때, 분명 터미널에는 에러 메시지가 없는데(실행이 잘되었다고 속삭이는데), 그 어떤 데이터 통신망도 붙지 않는 초보적인 저주의 가장 큰 범인은 바로 시스템 자원 점유의 오만함과 무선 네트워크 인프라의 차단벽, 즉 2가지 물리적 트러블슈팅의 구멍 탓이다.

첫 번째 저주: 7447 포트 빙의 충돌 현상 (netstat / lsof)
만약 여러분이 아까 전 zenohd(라우터)를 백그라운드로 띄워놓고 잊어버린 채, 또 하나의 파이썬(Python)의 Zenoh 피어 애플리케이션 안심하고 같은 컴퓨터 위에서 켠다면? 둘은 7447이라는 이 세상 좁고 유일무이한 황금 번호 포트를 둘러싸고 은밀하게 충돌하게 된다. (Zenoh는 포트 선점의 배타성이 있다!) 터미널에서 lsof -i :7447 이나 리눅스 ss -tunlp 같은 명령어 망치를 치는 순간 깜짝 놀라게 된다. 수많은 node.js 앱이나 타 프로세스가 이미 7447 포트를 알박기하고 선점 중이라면 그 뒤에 오는 녀석은 묵묵부답으로 패킷을 허공에 버리고 영원한 먹통 호구에 머무른다. 해결책은 오직 앞선 유령의 목을 쳐(kill -9) 소켓 통로의 대동맥을 다시 숨통 열어주는 것뿐이다.

두 번째 저주: 7446 동적 스카우팅 포트의 Wi-Fi 매장
자, 포트는 둘 다 열어주었다 치자. 내 로봇과 모니터링 드론 둘 다 멀쩡하게 7447 포트를 들고 수다를 떨려고 준비하는 데 대체 почему(왜) 둘은 서로가 이 방안에 있다는 사실(Discovery)조차 모를까?
우리는 3.1절에서 강조했던 “멀티캐스트(7446)“라는 단어를 되찾아야 한다.

만일 당신의 공업용, 가정용 무선 공유기가 (특히 Cisco 매니저 등) 스스로 똑똑한 척하며 ‘네트워크 대역폭 아낀답시고’ IGMP Snooping 이라는 기능을 멋대로 켜두어, 공중으로 뿌려지는 Zenoh의 7446 UDP 스카우트 비콘을 모조리 바닥에 파묻어버린다면? 둘은 죽어도 만나지 못한다!
이 저주를 풀기 가장 빠른 현장 대처법은, 당신의 cargo run 테스트 실행 스크립트 실행 시 굳이 헛발길 치는 스카우트를 끄고 아예 7447 유니캐스트(직통 IP 지명 통화) 모드로 상대방 주소(Config/Locators)를 강제 지명 타격하는 정적 라우팅 설정(Static Peer Locator)(-e tcp/192.168.x.x:7447)으로 이 위기를 즉각 우회해 빠져나와 증명하는 대안적 유연함을 갖추는 일이다.
