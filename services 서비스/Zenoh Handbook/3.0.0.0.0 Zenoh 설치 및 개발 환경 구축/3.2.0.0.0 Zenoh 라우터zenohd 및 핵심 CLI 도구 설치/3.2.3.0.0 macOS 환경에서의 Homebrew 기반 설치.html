3.2.3 macOS 환경에서의 Homebrew 기반 설치

마이크로서비스나 애플리케이션 프론트엔드/백엔드를 짜는 대다수의 개발자들 손에는 M1/M2 Apple Silicon을 탑재한 macOS가 쥐여져 있다. 이 로컬 노트북은 운영계(Production)의 라우터가 아니라, 순수한 개발용 피어(Peer) 노드나 테스트 브로커로서 기능해야 한다.

macOS 환경에서 손수 아카이브(zip)를 풀어 경로를 잡는 짓은 Apple의 혹독한 Gatekeeper(애플 실리콘 보안 격리) 정책에 의해 서명되지 않은 바이너리 검열에 걸려 터미널 팝업 경고의 지옥에 빠진다. 따라서 철저히 Homebrew 생태계의 패스포트(Passport)를 이용해야 한다.

0.1 [Runbook] Homebrew Tap 추가 및 설치 파이프라인

공식 Homebrew 메인 저장소가 아닌 자체 Eclipse Tap에서 끌어와야 한다.

# 1. Zenoh 공식 브루 탭(Tap) 인덱스 매핑
brew tap eclipse-zenoh/zenoh

# 2. 라우터 데몬 및 CLI 도구 완전 설치
brew install zenoh

0.2 [트러블슈팅] macOS 백그라운드 서비스 종속성 (brew services)

노트북을 열자마자 zenohd가 백그라운드 포트 7447번을 강제 점유하고 돌게 하고 싶다. 터미널의 탭 하나를 zenohd에 내주는 짓은 미개하다.
Homebrew의 services 명령어로 macOS의 메인 시스템( launchd )의 깊숙한 곳에 데몬을 박아버린다.

# 단순 구동 후 종료가 아니라 백그라운드에서 영구 구동되도록 Launch 데몬에 위임
brew services start zenoh

[아키텍처 경고] 포트 충돌(Port Collision)에 대비하라:
당신이 TypeScript 코드 내에서 포트를 지정하여 임베디드 라우터를 코드로 띄우는 순간(런타임 통합), 방금 저 brew services가 선점한 7447 포트와 충돌하여 자바스크립트 쪽 ADDRINUSE (Address already in use) 에러가 튀어 오를 것이다.
맥북 로컬에서 코드 짤 때 “왜 소켓 바인딩이 안되지?” 벽에 막힌다면 가장 먼저 백그라운드 브루 서비스의 멱살부터 잡아라.

# 포트를 뱉어내게 강제종료 후, 다시 내 앱 코드에서 라우터 띄우기
brew services stop zenoh
