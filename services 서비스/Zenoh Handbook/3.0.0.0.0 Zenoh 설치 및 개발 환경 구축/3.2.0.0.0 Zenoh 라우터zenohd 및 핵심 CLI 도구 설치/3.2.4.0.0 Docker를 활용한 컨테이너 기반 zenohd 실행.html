3.2.4 Docker를 활용한 컨테이너 기반 zenohd 실행

클라우드의 워커 노드(Node)나 서버 팜(Server Farm)에 커스텀 apt 명령어를 갈기는 행위는 이뮨 인프라(Immutable Infrastructure) 철학에 정면으로 위배된다. 서버가 타버리면 또 apt를 치고 있을 것인가?
운영계(Production) 라우터는 단 한 장의 불변하는 ‘이미지(Image)’ 파일로 영속적으로 배포되고 소멸될 수 있어야 한다.

0.1 단계: 기본 컨테이너 원샷(One-Shot) 실행 런북

아무 패키지 설치 없이 우분투 서버에서 가장 빠르고 깨끗하게 라우터를 돌리는 마법이다.

# --network host: 컨테이너의 가상 브릿지를 박살내고 서버 자체의 물리망에 포트를 구멍낸다. 
# 이를 통해 스카우팅(Multicast) 패킷이 도커 밖으로 원활하게 빠져나가게 만든다.
docker run -d --name my-zenohd \
    --network host \
    eclipse/zenoh:latest \
    --mode router

0.2 단계: [아키텍처 격상] 영속적(Persistent) 환경 주입과 볼륨 마운트

당신이 만약 Storage 플러그인을 붙여 zenohd를 InfluxDB 뺨치는 캐싱 서버로 쓴다면, 컨테이너가 폭발(Restart)할 때 캐시된 메모리 파일이 다 날아가 버리면 재앙이다.
반드시 “설정 파일“과 “데이터 보관 폴더(RocksDB용)“를 서버의 호스트 디렉터리와 결속(Volume Mount) 시켜야 한다.

[운영계 docker-compose.yml 마스터 피스]

version: "3.8"
services:
  zenoh-router-primary:
    image: eclipse/zenoh:latest
    container_name: global_backbone_1
    restart: always # OOM 터지면 좀비처럼 살아돌아오게
    network_mode: "host" 
    volumes:
      # 호스트의 설정 파일을 도커 내부로 밀어넣기 (Read Only)
      - /opt/zenoh/zenohd.json5:/zenohd.json5:ro
      # RocksDB 캐시 내용물 보존용 영구 스토리지 맵핑 (이게 없으면 재부팅시 기억상실)
      - /opt/zenoh/data:/data
    # 이미지 내부의 기본 Entrypoint 옵션 오버라이드
    command: ["-c", "/zenohd.json5"]

이렇게 인프라 애즈 코드(IaC)로 라우터 망을 조각해 두면, “도커 배포판이 다 좋은데 데이터가 자꾸 날아간다“는 아마추어 팀원들의 아우성을 단숨에 종식시킬 수 있다.
