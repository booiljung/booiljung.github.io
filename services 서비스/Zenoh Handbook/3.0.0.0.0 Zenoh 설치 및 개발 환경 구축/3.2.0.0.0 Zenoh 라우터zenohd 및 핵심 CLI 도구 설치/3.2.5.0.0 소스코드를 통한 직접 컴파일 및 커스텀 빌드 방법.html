3.2.5 소스코드를 통한 직접 컴파일 및 커스텀 빌드 방법

Eclipse에서 제공하는 컴파일된 apt 패키지나 Docker 이미지는 범용 장비를 타겟팅(Targeting) 한 탓에, “가장 최신의 불안정한 실험적 기능(Nightly)“이 빠져 있거나 라이다 센서의 속도를 극한으로 쥐어짜기 위한 “특수 하드웨어 컴파일 옵션“이 누락되어 있다.
로보틱스 딥테크(Deep-tech) 기업의 코어 엔진 조직이라면, 벤더사의 바이너리를 받아먹는 수동적 태도를 버리고 Rust 소스를 직접 뜯어고쳐 스스로의 zenohd를 주조해 낼 줄 알아야 한다.

0.1 [Runbook] Rust 툴체인을 활용한 극단적 최적화 빌드 파이프라인

1단계: 전장 준비 (Rust & Git Clone)
컴파일을 주도하는 것은 호스트 PC의 RAM을 미친 듯이 소모하는 Rust의 Cargo 툴체인이다.

# Rust 환경 준비
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env

# 최신 릴리즈 브랜치가 아닌, 날 것의 Main 브랜치(출혈판) 심장 탈취
git clone https://github.com/eclipse-zenoh/zenoh.git
cd zenoh

2단계: 릴리즈(Release) 모드를 넘어서는 커스텀 스나이핑 (Targeted Build)
생산성을 위해 단순 cargo build를 치면 디버그(Debug) 모드의 뚱뚱하고 느릿한 바이너리가 태어나 당신의 클라이언트 패킷을 막아버릴 것이다. 속도를 갈취하려면 릴리즈 모드는 기본이고, CPU의 고급 벡터 연산(SIMD)까지 켜야 한다.

# [최강 성능 쥐어짜기 런북 커맨드]
# 환경변수 'RUSTFLAGS'로 현재 서버 장비의 CPU 네이티브(native) 아키텍처를 
# 완벽하게 감지해 AVX2 등의 SIMD 가속 구조까지 파고들게 강제한다!
RUSTFLAGS="-C target-cpu=native" cargo build --release --all-features

주의사항: 이렇게 target-cpu=native로 주조된 라우터 바이너리(target/release/zenohd)는 옆자리 인텔 서버에서 AMD 서버로 복사(Copy)해가면 CPU 명령어가 틀려 Illegal Instruction(SIGILL) 커널 데스 크래시를 맞게 된다. 따라서 이 런북은 어플리케이션이 구동될 그 하드웨어 위에서 현지 조달 빌드를 수행해야 하는 제약 사항이 뒤따름을 인프라 구성도에 새겨두어라.
