3.5.3 TSConfig 환경 설정 및 모듈 호환성(ESM, CommonJS) 해결

TypeScript(TS)로 멋지게 짜놓은 젠오 코드는 결국 JS로 변환(Transpiling) 되는데, 이때 tsconfig.json을 잘못 섞어놓으면 런타임에 “require is not defined” 나 “import assertion failed” 와 같은 모듈 충돌 폭탄이 터진다.
Zenoh-TS 라이브러리는 최신 동향인 ESM(ECMAScript Modules) 을 메인 파이프라인으로 강력히 권장한다.

0.1 [Runbook] Zenoh 방탄 TSConfig 구성

아래 요건은 타협 불가능한 최적의 런타임 스펙이다. 이 파일 복사해서 여러분의 백엔드 루트에 박아라.

// [tsconfig.json]
{
  "compilerOptions": {
    /* 타겟팅: 가장 쾌적한 네이티브 비동기(top-level await) 허용을 위함 */
    "target": "ES2022",
    /* 핵심: 오래된 CommonJS(require)를 버리고 최신 ESM(import) 방식으로 통일결합 */
    "module": "Node16",
    
    "moduleResolution": "node16",
    "esModuleInterop": true,
    "strict": true,           /* Zenoh의 까다로운 Callback Type 추론을 강제 검증토록 허용 */
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "outDir": "./dist"
  },
  "include": ["src/**/*"]
}

0.2 package.json 타입 록인(Lock-in)

루트 package.json 에도 내가 이 프로젝트를 진정한 모던 ESM으로 굴리겠다는 서약서를 박아 넣어야 Node.js 실행기가 젠오 라이브러리를 정상적으로 물어온다.

// package.json 파일에 반드시 추가
{
  "name": "zenoh-control-backend",
  "type": "module" // 이 한 줄이 없으면 상단의 Node16 모듈화 셋팅이 꼬인다!
}

이제 개발자는 구차하게 const zenoh = require("...") 대신, 깔끔한 Top-level Await와 함께 import * as zenoh from '@eclipse-zenoh/zenoh-ts'; 의 문맥 방풍망 안에서 코딩을 이어나갈 수 있다.
