3.5.4 비동기 이벤트 루프 처리를 위한 최적화 설정

JS/TS 백엔드(Node.js)의 치명적인 한계는 모든 유저의 로직이 단일 쓰레드(Single-Thread) 메인 이벤트 루프를 빙빙 돈다는 점이다.
초당 1만 번의 라이더 센서 값을 Zenoh에서 구독(Subscribe)으로 뽑아오는데, 그 루프 중간에 무거운 JSON 암호화 파싱 연산을 박아 넣었다 치자.
Node.js의 메인 루프가 막혀버리고(Blocked), 제때 수거되지 못한 메모리 상의 C++(Zenoh 코어) 네트워크 콜백 버퍼들이 수직 폭발하여 서버가 터진다(V8 Heap Out Of Memory).

0.1 [Runbook] 데이터 폭포수 강제 병렬화 오프로딩 (Worker Threads)

네트워크 통신망이 아무리 빨라도 JS 인터프리터가 느리면 망(Net) 전체의 속도를 하향 평준화 시킨다.

[안티 패턴: 무지성 콜백 랙(Lag)]

// 이렇게 짜면 로봇 센서가 폭주할 때 Node.js가 버벅이며 전체 웹서버 응답을 멈춰버린다.
session.declare_subscriber("heavy/data", async (sample) => {
    // [경고] 수백 ms가 걸리는 CPU 차단 블로킹 연산
    await heavyImageProcessing(sample.payload); 
});

[아키텍처 해결책: Worker 기반의 비동기 분리(Offload)]
Node.js의 Worker Threads를 가동해 Zenoh 통신(I/O 수신) 스레드와 데이터 렌더링(CPU 연산) 스레드를 물리적으로 격리하라.

  1. Zenoh의 수신 콜백은 오로지 데이터를 Worker 로 전달(Message Passing)만 하고 1ns 만에 퇴각한다.
  2. 뒷단의 멀티코어 Worker들이 데이터를 병렬로 파싱하여 데이터베이스에 밀어 넣는다.

[GC(가비지 컬렉터) 파열 방어 런북]
Node.js V8 엔진의 기본 메모리 한계(약 1.5GB)를 강제로 해제해야 한다. 컨테이너 띄울 때 도커 실행 변수에 아래를 추가하라.

# 로봇 트래픽을 견디기 위해 V8 JS 엔진의 Heap 메모리 한계치를 4GB(4096) 이상으로 뚫어버린다.
node --max-old-space-size=4096 dist/main.js

TypeScript 생태계에 발을 들였다면, “Zenoh 엔진 속도“에 도취되지 마라. 그 위를 굴러가는 인터프리터 V8 엔진의 좁은 아스팔트 길(이벤트 루프)을 4차선으로 늘리지(Worker & Heap 튜닝) 않으면 결국 적체되어 질식사한다.
