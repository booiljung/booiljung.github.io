3.6.3 네이티브 방식 통신 - rmw_zenoh 미들웨어 컴파일 및 설치

zenoh-bridge-dds 는 기존 망을 건드리지 않는 훌륭한 타협안이지만, “번역“을 수행하므로 직렬화 과정에서 CPU 파이프라인 지연(Latency overhead)이 필연적으로 발생한다.
진정으로 초급진적인 속도와 배터리 한계 극복을 원한다면, ROS2의 허파인 DDS 자체를 도려내고 rmw_zenoh_cpp 를 이식하는 ‘네이티브(Native)’ 절제 수술을 단행해야 한다.

0.1 [Runbook] rmw_zenoh 빌드 전술

이는 철저히 ROS2 작업 공간(colcon) 내에서 수행되어야 한다. 최신 ROS2 배포판(Jazzy 등)에서는 apt로 제공되기도 하지만, 여기서는 커스텀 라우터 최적화를 위해 심장부 소스 빌드를 감행한다.

# 1. 아까 파놓은 ROS2 작업 공간 진입
cd ~/zenoh_ws/src

# 2. 공식 rmw_zenoh 레포지토리 클론 
# 반드시 자신이 쓰고 있는 ROS2 배포판 이름(humble, iron) 브랜치를 명확히 타겟팅할 것!
git clone --branch humble https://github.com/ros2/rmw_zenoh.git

# 3. 디펜던시 이빨 맞추기
cd ~/zenoh_ws
rosdep update
rosdep install --from-paths src --ignore-src --rosdistro humble -y

# 4. RMW 엔진 컴파일 투하!
# RMW 계층은 핵심 코어이므로 반드시 Release 모드로 최적화 컴파일을 박아넣어야 한다.
colcon build --cmake-args -DCMAKE_BUILD_TYPE=Release

0.2 [트러블슈팅] C/Rust 헬퍼 바인딩 에러

colcon build 도중 zenoh-c 혹은 Rust 패키지와 바인딩하는 과정에서 <zenoh.h> 를 못 찾는 에러가 빈번하다.
이는 RMW 컴파일러가 CMake 단에서 빌드 툴체인을 얽을 때 cargo 경로를 유실했거나, OS 벤더 환경이 rustc를 인지하지 못한 파국이다.
어제 켜둔 터미널이 아니라, “반드시 source ~/.cargo/env 가 먹인 상태의 터미널” 에서만 colcon build를 날려야 이 치명적 링킹 단절을 예방할 수 있다.
