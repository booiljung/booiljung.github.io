3.6.2 브릿지 방식 통신 - zenoh-bridge-dds 패키지 설치 및 빌드

사내 로봇 군집(Fleet) 시스템에 이미 수십 개의 노드가 FastDDS 로 잘 세팅되어 돌고 있다 치자. 갑자기 “이제부터 전면 Zenoh를 도입한다“며 로봇 내부의 미들웨어를 싹 다 갈아엎는 것은 위험천만한 미친 짓(Big Bang Rewrite)이다.

이때 쓰는 우회 전술이 바로 zenoh-bridge-dds 다. 로봇 내부 컴포넌트 간의 통신은 기존 DDS를 그대로 놔두고, 로봇 밖(클라우드나 타 로봇)으로 나갈 트래픽만 브릿지 데몬이 가로채서(Intercept) Zenoh 와이어 패킷으로 번역해 날려주는 투 트랙 전략이다.

0.1 [Runbook] 브릿지 데몬 소스 빌드 파이프라인

이 브릿지 데몬은 ROS2 노드가 아닌 별개의 독립 실행형(Rust) 프로세스다. (Colcon 워크스페이스가 아닌 Cargo로 빌드한다)

# 1. Rust 툴체인 최신화 (앞선 3.3.1 장의 지침 필수)
source $HOME/.cargo/env

# 2. 브릿지 소스코드 격리 공간 가져오기
git clone https://github.com/eclipse-zenoh/zenoh-plugin-dds.git
cd zenoh-plugin-dds

# 3. 릴리즈 성능 쥐어짜기 컴파일 (10분 이상 소요될 수 있음)
# 여기서 CycloneDDS C 헤더 파일들을 추적하기 때문에 
# 빌드 머신에 3.6.1의 ROS2 환경(rosdep)이 깔려있어야 에러가 안 난다.
cargo build --release --bin zenoh-bridge-dds

0.2 운영 아키텍처 및 기동 지침

빌드가 성공하면 target/release/zenoh-bridge-dds 바이너리가 떨어진다.
로봇 한 대당 이 바이너리를 ‘단 1개만 백그라운드로 실행’ 시킨다.

# 로봇의 DDS 로컬망 데이터를 모두 빨아들여서, AWS 클라우드의 라우터로 송출한다!
./zenoh-bridge-dds -m client -e tcp/aws-cloud-router.com:7447 &

브릿지가 켜지는 순간, 로봇 내에서 발생하는 수많은 DDS 메시지(/cmd_vel, /odom)들은 더 이상 와이파이 망을 타고 멀티캐스트 폭풍을 일으키는 대신, 브릿지에 빨려 들어가 조용한 단방향(Unicast) Zenoh 터널을 타고 클라우드 서버에만 조용히(Silent) 배달된다. 이것이 대역폭 절약의 시작이다.
