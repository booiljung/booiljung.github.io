3.1.2 로컬 개발 환경 vs 에지 및 클라우드 배포 환경 비교

“로컬에서는 잘 돌던 코드가 배포만 하면 죽습니다.” 이건 100% 모드(Mode)와 네트워크 격리(Isolation)의 차이를 이해하지 못한 초보의 실수다.

0.1 개발 환경 (Local Environment)

  • 목적: 내 노트북 단일 생태계 안에서 로직만 빠르게 짜고 테스트한다.
  • 네트워크 속성: localhost(127.0.0.1) 이나 Docker Bridge 내부.
  • 모드 런북: 모든 개발자 단말기는 별도의 라우터 세팅 없이 오직 Peer 모드 멀티캐스트 에 의존하여 스크립트들을 결속시킨다. 
    # 로컬 테스트용 Pub 
zenoh-plugin-pub --mode peer
# 로컬 테스트용 Sub (즉각적으로 붙음)
zenoh-plugin-sub --mode peer


### 0.2 운영 배포 환경 (Edge-to-Cloud Production)

- **파국:** 공장 현장에 저 방식(Peer) 그대로 1,000대의 라즈베리 파이를 켜서 멀티캐스트 폭풍(Storm)이 공유기를 터트리는 즉시 해고감이다.
- **아키텍처 변환 지침:**
- 개발 완료 후 배포 스크립트를 올릴 때는 모든 에지 디바이스의 모드를 철저한 고립 모드인 **`Client`** 로 강제 전환해야 한다.
- AWS/GCP 클라우드 또는 로컬 사내 K8s 마스터 노드에 `Router(zenohd)` 하나를 정적(Static) 공인 IP로 띄운다.
- 클라이언트들은 `-e tcp/라우터공인IP:7447` 파라미터를 들고 오직 1개의 화살표(Unicast)만을 조준하여 중앙으로 빨려 올라가게 도면을 뜯어고쳐야 한다.

이 "개발은 Peer로, 배포는 Client-Router로"라는 대원칙이 당신 인프라의 확장성(Scalability) 한계점을 결정짓는다.
