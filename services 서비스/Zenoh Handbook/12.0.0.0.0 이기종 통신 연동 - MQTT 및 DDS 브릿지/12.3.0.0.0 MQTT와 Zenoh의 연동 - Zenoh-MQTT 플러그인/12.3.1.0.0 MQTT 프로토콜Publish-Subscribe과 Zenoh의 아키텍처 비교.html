12.3.1 MQTT 프로토콜(Publish-Subscribe)과 Zenoh의 아키텍처 비교

브릿지를 켜기 전에, 두 프로토콜의 유전자적 한계와 공통점을 직시해야만 설계 파열을 막을 수 있다.

0.1 [인스펙션] 브로커 패턴 vs 탈중앙화 패턴

1. 중앙집중형 독재자 (MQTT)
MQTT 는 브로커(Broker) 가 죽으면 전체 공장 통신이 100% 마비된다. 모든 센서(Pub) 와 화면(Sub) 은 무조건 백엔드 브로커 서버를 바라봐야 한다. (스타형 토폴로지). 센서끼리 직접 대화하는 로컬 무선통신(P2P) 같은 건 존재하지 않는다.

2. 탈중앙화 게릴라 (Zenoh)
Zenoh 는 단말(Client) 이 라우터 없이도 허공(UDP Multicast) 을 향해 서로를 찾고 직결(P2P Peer) 할 수 있다. 라우터가 있으면 그저 길을 좀 더 넓게 쓸 수 있을 뿐, 통신이 완전히 박살나지는 않는다.

[아키텍처의 교차점]
공통점은 둘 다 “주제(Topic)” 를 기반으로 한 Pub/Sub 모델을 쓴다는 것이다. 브릿지는 이 공통점을 파고든다.
Zenoh 입장에서는 MQTT 브로커 전체를 커다란 “Zenoh 센서 1개” 로 취급한다. 즉, MQTT 세상 전체가 Zenoh 의 거대한 트리 맵핑 아래의 일개 포크(Fork) 로 종속되는 형국을 띤다.
