12.3.5 MQTT QoS(0, 1, 2)와 Zenoh 신뢰성(Reliability) 수준 매핑 전략

MQTT 서버에서 강력한 보장 모델(QoS 2: Exactly Once)로 데이터가 날아왔는데, 브릿지가 이걸 Zenoh 쪽으로 대충(Best Effort) 넘겨버리면 시스템의 신뢰성에 치명적인 구멍이 생긴다.

0.1 [Runbook] 프로토콜 타협점(Trade-off) 튜닝

Zenoh 는 QoS 를 철저하게 이항 분리한다.

1. 번역의 디폴트 규칙

  • MQTT QoS 0 (Fire and Forget): Zenoh 망으로 넘어올 때 제일 싼 데이터 칩인 Best-effort 로 포장된다. 유실되어도 책임지지 않는다.
  • MQTT QoS 1 (At least once), QoS 2 (Exactly once): Zenoh 망으로 넘어올 때 모두 가장 값비싼 Reliable 칩으로 포장된다. TCP ACK 급의 완벽한 보장을 받는다.

2. 브릿지의 강제 하향(Downgrade) 정책
만약 공장 IoT 센서 10,000개가 QoS 2 (가장 무거운 완벽보장 프로토콜) 로 쏴대는 악성 세팅을 해놨다면? Zenoh 브릿지까지 QoS 2 인 채로 클라우드로 올리면 브릿지가 CPU 한도를 넘어서 터진다.
이때 브릿지 옵션에서 강제로 트래픽 질을 깎아내려야(Throttling) 한다.

{
  plugins: {
    mqtt: {
      broker: "tcp/...:1883",
      // 들어오는 모든 데이터를 무자비하게 QoS 0 / Best-effort 로 격하해버린다!
      qos: "0" 
    }
  }
}
