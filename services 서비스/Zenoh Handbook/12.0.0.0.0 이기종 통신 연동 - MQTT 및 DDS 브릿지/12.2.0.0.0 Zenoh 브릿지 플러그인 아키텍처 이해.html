12.2 Zenoh 브릿지 플러그인 아키텍처 이해

단순히 두 개의 통신 소켓을 열어놓고 데이터를 퍼나르는 싸구려 파이썬 스크립트 릴레이 서버를 생각했다면 오산이다.

Zenoh 는 라우터 엔진 그 자체를 뜯어고치지 않고도, 런타임에 동적으로 라이브러리를 갈아 끼워 통역 기능을 추가하는 “핫 플러그인(Hot-Plugin)” 아키텍처를 채택했다. 이 장에서는 zenoh-bridge-dds 나 zenoh-bridge-mqtt 가 어떤 구조로 Zenoh 라우터의 척수에 달라붙어 기생(스니핑)하는지, 그 기계공학적 해부도를 살펴본다.

1. Zenoh 라우터의 확장 가능한 플러그인 시스템 구조

Zenoh 라우터(zenohd) 는 그 자체로는 매우 단순한 형태의 라우팅 허브에 불과하다. 이 라우터가 다른 프로토콜과 대화를 나누기 위해서는 확장 모듈이 필요하다.

1.0.1 [인스펙션] 동적 적재(Dynamic Loading) 기반의 브릿징

Zenoh 인프라는 철저하게 코어(Core) 와 플러그인(Plugins) 으로 분리되어 있다.

1. 라이브러리 후킹 (DLL / SO injection)

  • zenoh-bridge-dds 같은 브릿지는 사실 독립적인 프로그램으로 켤 수도 있지만, 본질적으로는 라우터 데몬 안에 주입되는 ‘공유 라이브러리(.so, .dll)’ 규격의 플러그인으로 동작하도록 설계되어 있다.
  • 라우터를 구동할 때 환경 설정(zenoh.json5) 의 plugins 항목을 켜면, 라우터는 기동 즉시 운영체제의 동적 링커를 호출하여 메모리에 브릿지 엔진을 직결시켜 버린다.

2. 통신 오버헤드 0 의 영역 (Zero-Copy Bridge)

  • 외부 릴레이 서버를 띄울 때 가장 큰 문제는, 릴레이 프로세스와 라우터 프로세스 간에 다시 한번 ‘로컬 소켓(Localhost Socket)’ 통신을 해야 한다는 것이다.
  • 하지만 Zenoh 의 플러그인 몽골로이드(Mongoloid) 아키텍처 하에서는, 브릿지 코드와 라우터 코드가 하나의 메모리 주소(프로세스) 를 공유한다. DDS 로 들어온 1MB 짜리 패킷이 Zenoh 망으로 넘어갈 때 단 1 바이트의 복사(Copy) 도 일어나지 않는다. 포인터 주소만 넘겨주면 통역이 끝난다.

2. 브릿지 플러그인의 생명주기 및 메모리 관리

수백 개의 MQTT 장치가 붙었다 떨어지고, 수천 개의 ROS2 토픽이 날아다니는 환경에서 브릿지가 죽어버린다면 전체 인프라가 마비된다.

2.0.1 [인스펙션] Rust 기반의 불면(Sleeveless) 안정성

Zenoh 브릿지들이 며칠, 몇 달을 틀어놔도 뻗지 않는 이유는 개발 언어가 C++ 이 아니라 Rust 로 강제되어 있기 때문이다.

1. 세션의 생성과 파괴 (Lifecycle)

  • 브릿지는 외부 망(MQTT/DDS)에 접속을 시도할 때 백그라운드 스레드(Tokio async 런타임)를 띄운다.
  • 네트워크가 잠시 끊기면, 해당 백그라운드 스레드는 패닉(Panic) 상태에 빠지지 않고 대기열(Sleep) 로 조용히 물러났다가, 연결 트리거 신호가 오면 즉각 부활(Revive) 한다. 이때 기존에 물려 있던 메모리 자원은 Rust 의 소유권(Ownership) 모델에 의해 완벽하게 수거된다. (Memory Leak 0%)

2. 스나이퍼식 메모리 풀 튜닝

  • 브릿지 코어는 외부 프로토콜(예: DDS FastRTPS) 의 라이브러리가 할당하는 무식한 힙 메모리(Heap Memory) 마저도 제어하려 든다.
  • 브릿지는 내부적으로 고정된 사이즈의 버퍼 풀(메모리 수영장)을 미리 만들어 두고, 외부 패킷이 들어올 때마다 매번 malloc()(동적 할당) 을 치는 것이 아니라 이 풀에서 양동이를 빌려와 데이터를 담은 뒤, Zenoh 망으로 던진 후 양동이를 반납한다.

가비지 컬렉터(GC) 가 도는 Java 나 Python 으로 만든 싸구려 릴레이 서버라면 하루에 한 번씩 “멈칫” 하는 가비지 컬렉션 타임아웃이 터지겠지만, 이 아키텍처에서는 지연 시간(Latency) 이 항상 0.01ms 선으로 송곳처럼 플랫(Flat)하게 유지된다.

3. 이기종 간의 프로토콜 변환 및 페이로드(Payload) 매핑 메커니즘

언어의 본질은 “문법(Protocol Header)” 과 “의미(Payload)” 의 분리다. 브릿지는 이 두 가지를 외과 수술 수준으로 도려내어 치환한다.

3.0.1 [인스펙션] 아키텍처 번역 엔진 (Translation Engine)

1. 토픽 이름(Namespace) 의 이식 수술

  • MQTT: /sensor/temp
  • DDS(ROS2): rt/sensor/temp
  • 브릿지는 이 규격들을 Zenoh 의 범용 트리 구조인 Key Expression 으로 강제 스왑한다. 이 변환 테이블은 브릿지 설정 파일(allow, deny) 에 의해 철저히 통제된다.

2. 페이로드 (본문 덩어리) 의 투명성
가장 아름다운 부분이다. Zenoh 브릿지는 안에 들어있는 내용물을 이해하려 들지 않는다.

  • MQTT 에서 보낸 { "temp" : 25.4 } 라는 UTF-8 바이트 배열 덩어리
  • ROS2 에서 보낸 Float32 형식의 리틀엔디안 이진(Binary) 덩어리
    Zenoh 브릿지는 이 덩어리를 뜯기지 않게 래핑(Wrapping)만 한 뒤, 앞단에 “이건 DDS 규격 덩어리야”, “이건 MQTT 텍스트 덩어리야” 라는 태그표(MIME Type/Encoding) 만 하나 딱 붙여서 전송한다.

목적지 브릿지는 이 태그표를 보고, 원래의 프로토콜 규격으로 다시 포장지(Header)만 씌워서 밀어버린다.
이것을 단순 페이로드 스루트(Opaque Payload Passthrough) 라고 부르며, 이로 인해 브릿지가 CPU 파워를 소모하지 않고 초고속 톨게이트 렌더링이 가능해진다.

4. 데이터 직렬화 및 역직렬화 오버헤드 분석

페이로드는 내용물 그대로 날아간다고 쳤을 때, 과연 통신 오버헤드의 범인은 누구인가?
바로 “포장지를 벗기고 다시 입히는” 과정이다. 이것이 브릿지 지연 시간(Latency)의 유일한 함정이다.

4.0.1 [인스펙션] 직렬화(Serialization) 마이크로 오버헤드 부검

1. MQTT 브릿지의 오버헤드 (JSON, String)

  • MQTT 는 구조가 워낙 가벼워서 포장지를 벗기는 데 걸리는 시간이 거의 측정 불가능할 정도로 짧다 (sub-microsecond).
  • 하지만 MQTT 의 토픽 이름 스펙 (/ 로 이어진 문자열 매칭) 과 Zenoh 의 Key Expression 의 규칙이 살짝 달라, 문자열 파싱 구문을 처리하느라 약간의 정규식 매칭 오버헤드가 발생한다 (수십 us).

2. DDS 브릿지의 오버헤드 (CDR 변환 지연)

  • DDS 패킷을 뜯어낼 때, 브릿지는 ROS2 가 쓰는 RTPS 헤더 정보를 파헤쳐야 한다. 이게 생각보다 구조가 더럽다 (지터, Liveliness 타이머 등).
  • DDS 의 복잡한 QoS 메타데이터를 Zenoh 의 Reliability 와 Priority 로 매핑하는 과정에서 약간의 연산 딜레이 패널티(대략 0.1 ~ 0.3ms)가 추가된다.

아키텍트의 결론:
시스템 설계자는 이 “브릿지 당 0.3ms 가량의 포장 비용” 을 반드시 시스템 레이턴시(Latency) 예산에 포함해야 한다.
만약 로봇과 클라우드 사이에 브릿지가 총 2단(로봇 쪽 1단, 클라우드 수신 쪽 브릿지로 DDS 다시 변환 시 1단) 으로 깔려있다면, 기본 핑(Ping) 에 무조건 1ms 단위의 직렬화/패킹 비용이 추가된다는 것을 수학적으로 감내해야만 한다.
