12.4 DDS와 Zenoh의 연동: Zenoh-DDS 플러그인

11장의 ROS2 생태계에서 다뤘던 zenoh-bridge-dds 의 심화론이다.
ROS2 라는 로봇 운영체제 껍데기를 벗겨내고, 그 밑바닥에서 돌고 있는 순수 DDS (Data Distribution Service) 규격 그 자체를 어떻게 Zenoh 가 구워삶아 전 세계망으로 쏘아 올리는지, 그리고 그 과정에서 필연적으로 발생하는 DDS 특유의 이질적인 QoS 충돌을 어떻게 제어할 것인지에 대한 딥 다이브(Deep-Dive) 런북이다.

1. DDS(Data Distribution Service) 글로벌 데이터 공간 및 도메인 이해

Zenoh 에 꽂기 전, DDS 라는 고집스러운 시스템이 세상을 분할하는 방식을 이해해야 한다.

1.0.1 [인스펙션] DDS 의 방(Room) 만들기 철학

1. 도메인 ID (Domain ID)
DDS 는 같은 네트워크 망(Subnet) 에 있더라도, Domain ID (0~232) 가 다르면 서로가 투명인간 취급을 한다. 물리적으로 같은 랜선에 꽂혀 있어도 도메인 ID가 0 인 무리들과 1 인 무리들은 절대 섞이지(Discovery) 않는다. 이것은 공장 안의 로봇들을 A구역, B구역 등으로 안전하게 분리하는 가장 기초적인 논리적 격벽이다.

2. GDS (Global Data Space)
DDS 의 철학은 “각자가 데이터를 갖고 있는 게 아니라, 공중에 데이터 공간(GDS)이 둥둥 떠 있고 거길 다 같이 바라본다” 는 식이다. 이들은 중앙 관리자(Broker) 없이, UDP 멀티캐스트를 향해 자신이 어떤 토픽을 찾고 있는지 끊임없이 소리쳐서(Simple Discovery Protocol) 가상의 거리를 맺는다.

망을 넘어갈 수 없다는 이 태생적 고립성을 부수고 이 “방(Domain)” 에 바깥세상(WAN)으로 나갈 웜홀(Bridge) 을 하나 파버리는 것이 이 섹션의 목표다.

2. DDS 데이터 중심 패러다임과 Zenoh의 라우팅 구조 비교

왜 굳이 DDS 를 Zenoh 로 바꾸려 하는 건가? 두 철학의 극명한 한계차이 때문이다.

2.0.1 [인스펙션] 분산 캐싱 vs 라우팅 트리의 격돌

[DDS: 분산 데이터베이스의 흉내]
로봇 A가 /map (지도) 토픽을 수신하면, DDS 엔진은 노드 스레드 안에 이 10MB 짜리 지도를 통째로 캐싱한다. 만약 로봇 프로그램 안에 지도 보는 노드가 5개가 띄워져 있다면, 똑같은 데이터가 메모리 공간을 5번 장악한다. “데이터 중심(Data-centric)” 의 무서운 메모리 폭식이다.

[Zenoh: 제로에 가까운 터미널링]
Zenoh 통신망에서 데이터는 머물지 않는다(Non-blocking & Zero-copy). 그저 파이프 위를 빠르게 지나갈 뿐이다. 라우터만이 데이터를 잠시 쥐고 있다가, 달라고(Sub) 요청한 20개의 클라이언트에게 포인터(소켓)로 데이터를 부채살처럼 흩뿌린다. 메모리를 극도로 아끼면서도 망의 도달 범위를 무한정 넓히는(WAN Scaling) 현대적 아키텍처다. 브릿지는 DDS 의 무거운 “상태 보존” 정책을 이 “실시간 흐름” 으로 강제 역치환하는 파이프의 역할을 맡는다.

3. Zenoh-DDS 브릿지 설치 (CycloneDDS 및 FastDDS 호환성 체크)

Zenoh 는 오만하지 않다. 세상의 메이저 DDS 벤더 두 곳(Eclipse CycloneDDS, eProsima FastDDS) 의 통신 규격(RTPS/CDR) 과 완벽한 호환을 이뤄냈다.

3.0.1 [Runbook] 브릿징 엔진 설치 및 구동 확인 전술

1. 브릿지 단독 설치
11장에서 보았던 것과 동일한 데비안 패키지(혹은 Cargo 빌드)를 이용해 브릿지를 꽂는다.

sudo apt install zenoh-bridge-dds

2. DDS 벤더 중독성(Vendor-Locking) 탈피
당신이 속한 회사가 CycloneDDS(ROS2 Galactic 이후 기본)를 쓰든, FastDDS(ROS2 Foxy 기본)를 쓰든 상관없다.
zenoh-bridge-dds 는 기저의 네트워크 와이어 프로토콜(DDSI-RTPS) 을 완벽히 스니핑하여 두 진영 모두에 호환된다.

## 그냥 켜기만 하면 현재 망에 떠다니는 모든 RTPS 프레임을 낚아챈다.
zenoh-bridge-dds -l tcp/0.0.0.0:7447

켜는 순간 화면에 [INFO] Discovered Writer... 라는 로그가 미친 듯이 쇄도한다면 성공적으로 DDS 코어에 빨대를 꽂은 것이다.

4. DDS 도메인 ID(Domain ID) 및 파티션(Partition) 설정 가이드

로봇 공장에 두 대의 로봇이 똑같이 Domain ID: 0 을 쓰고 있다면 조이스틱을 꺾을 때 두 대가 같이 움직이는 대참사가 벌어진다. 이를 막기 위해 로봇별로 도메인 번호를 나눈다. 브릿지도 이 룰을 준수해야 한다.

4.0.1 [Runbook] 도메인 타게팅(Targeting) 훔치기 전술

1. 도메인 조준경 설정 옵션
로봇 A가 Domain ID: 41 안에서 돌고 있다면, 브릿지를 켤 때 이 방(Room) 에 맞춰서 들어가야 한다. 안 그러면 엉뚱한 로봇의 데이터를 퍼 올리게 된다.

## `-d` 옵션을 주어 41번 도메인의 트래픽만 잡아먹어라!
zenoh-bridge-dds -d 41 -e tcp/aws-router.com:7447

2. 멀티 도메인(Multi-Domain) 동시 연결은 불가능하다!

  • 경고: 브릿지 프로세스 하나당 오직 한 개의 도메인 아이디만 락온(Lock-on) 할 수 있다.
  • 만약 한 컴퓨터에서 41번과 42번 도메인을 모두 클라우드로 올리고 싶다면, zenoh-bridge-dds -d 41 프로세스와 -d 42 프로세스를 따로 2개 띄우는 것 만이 유일한 해법이다. (포트 충돌 주의).

5. DDS 토픽 이름 및 Zenoh 키 공간의 계층적 매핑 메커니즘

ROS2 가 개입하지 않은 “순수 DDS” 어플리케이션은 어떻게 Zenoh 로 번역되는가?

5.0.1 [인스펙션] 와일드카드 및 네임스페이스 트랜슬레이션

  1. 기본 매핑 원칙
    ROS2 일 때는 브릿지가 알아서 ros2/rt/... 접두사를 붙여줬지만, 순수 DDS(C++ FastDDS 앱)의 경우 토픽 이름이 있는 그대로 1:1 전송된다.
  • DDS Topic: HelloWorldTopic
  • Zenoh Key: HelloWorldTopic
  1. 스코프(Scope) 격리로 인한 아키텍처 보호
    도메인 아이디가 41번인 로봇A 와 42번인 로봇B 의 데이터가 동일한 HelloWorldTopic 이라면 클라우드에서 박치기를 한다.
    반드시 브릿지를 켤 때 덮어쓰기(Scope)를 해줘야 한다.
## 로봇 A (Domain 41) 
zenoh-bridge-dds -d 41 --scope "robotA" 
## -> Zenoh 에는 "robotA/HelloWorldTopic" 으로 올라감.

## 로봇 B (Domain 42)
zenoh-bridge-dds -d 42 --scope "robotB"
## -> Zenoh 에는 "robotB/HelloWorldTopic" 으로 올라감.

클라우드의 서버 프로그램은 robot*/HelloWorldTopic 이라는 단 한 줄의 정규식 구독 설정만으로 전 세계의 수만 대 로봇을 모니터링할 수 있는 확장성을 얻는다.

6. 데이터 타입 변환: CDR(Common Data Representation) 동적 처리

DDS는 본인의 데이터가 몇 바이트짜리 정수(Int)인지 어디서 끝나는지 메타데이터를 담은 .idl (혹은 .msg) 스키마 파일을 달고 다닌다. 반면 Zenoh 는 그런 거 신경 안 쓰고 그냥 이진 덩어리(Byte Array) 를 던져버린다.

6.0.1 [인스펙션] 스키마(Schema) 파괴와 재생성

1. Opaque Payload (불투명한 데이터) 전략
DDS 에서 브릿지를 거쳐 Zenoh 로 넘어갈 때, 브릿지는 DDS 의 CDR(빅엔디안/리틀엔디안이 섞여 있는 C++ 메모리 덤프 형태) 규격으로 구워진 바이트 배열을 해독하려 들지 않는다.
대신 이 바이트 배열 앞에 작은 편지(Header)를 붙인다.

  • “이 데이터의 이름은 std_msgs::msg::dds_::String_ 형식이야!”

2. 수신 측 브릿지의 부활
Zenoh 망을 타고 날아온 이 데이터가 저 멀리 클라우드의 또 다른 DDS 로 들어갈 차례다. 클라우드 측 브릿지는 저 편지를 뜯어보고는, DDS 망에 외친다 “어이구, 이 데이터는 std_msgs::msg... 타입이네.” 하고 DDS 의 형식(Type) 시스템에 맞춰서 완벽하게 재생산(Re-publish) 해준다.

결과적으로, 이기종 간의 브릿징 시스템을 구현할 때 양쪽 시스템은 IDL 파일을 브릿지에게 일일이 알려주거나 컴파일할 필요 없이 “100% 동적 타입 호환(Dynamic Type Resolution)” 이 이루어진다. 이것이 브릿지를 켤 때 아무런 코딩도 필요 없는 기적의 원리다.

7. 주요 DDS QoS 정책(Reliability, Durability, History)과 Zenoh 동작 매핑

DDS 에는 수만 가지 종류의 변태적인 QoS 조합이 존재한다. 브릿지가 이 복잡한 정책을 잘못 번역하면 통신이 얼어붙는다.

7.0.1 [Runbook] QoS 타협과 가지치기 (Pruning) 전술

가장 치명적인 3대 QoS 매핑 룰을 점검한다.

1. Reliability (신뢰성)

  • DDS 의 RELIABLE -> Zenoh 브릿지는 이를 감지하고 Zenoh 내부 터널의 Reliable 옵션을 켠다.
  • DDS 의 BEST_EFFORT -> Zenoh 패킷 유실을 허용하고 무조건 최신 데이터를 밀어 넣는다. (카메라나 라이다 토픽에 무조건 이 QoS 를 써야 시스템이 살 수 있다).

2. Durability (내구성 - 과거 보존)

  • DDS TRANSIENT_LOCAL (나중에 접속한 놈에게 과거 데이터를 보여줌) -> 브릿지는 Zenoh 엔진에 내장된 인-메모리 캐시(Cache) 를 켜서 일시적으로 데이터를 쥐고 있는 상태로 변환된다.
  • [위험 경고] 대용량 카메라 영상에 이 옵션이 켜져 있으면, 브릿지를 켜자마자 컴퓨터의 RAM 이 폭발한다! 반드시 VOLATILE (휘발성) 로 설정하라.

3. Liveliness (생존 확인)
DDS 의 피곤한 심장 박동(Heartbeat) 정책. 브릿지는 이것을 적당선에서 무시하거나 Zenoh 세션이 살아있는 한 계속 살아있는 것으로 퉁(?) 치고 넘어가도록 유연하게 설계되어 있어, WAN 통신에서의 트래픽 오버헤드를 극적으로 줄여준다.

8. Allow/Deny 리스트를 활용한 브릿지 트래픽 필터링 및 범위 제어

DDS 망 전체를 Zenoh 다리(Bridge)로 퍼 올리는 짓은 곧 네트워크 자살이다.
/rosout 에러 로그나 거대한 Map 데이터를 막아내고 오직 센서 통계값만 외부로 빼내야 한다.

8.0.1 [Runbook] 데이터 방화벽(Tollgate) 설정 전술

설정 파일 config.json5 의 위력을 보여주는 화이트리스트(Whitelist) 런북.

// dds_bridge.json5
{
  // 1. DDS망에서 Zenoh (외부 클라우드)로 나가는 데이터 통제
  dds_to_zenoh: {
    // 디폴트값: 아무것도 밖으로 나갈 수 없다. 방화벽 셧다운!
    forwarding: "deny",
    
    // 예외 조항 (허락된 놈들만 나간다)
    allow: {
      topics: [
        "/robot_status", 
        "/battery_level"
      ],
      // 특정 이상한 타입(Type) 이 들어오면 토픽명 상관없이 다 드랍!
      // (예: 대용량 이미지 타입 외부 유출 1차 방화벽)
      types: [ "std_msgs::msg::dds_::Float32_" ]
    }
  },

  // 2. 외부(클라우드)에서 내 로봇장비(DDS) 로 들어오는 데이터 통제
  zenoh_to_dds: {
    forwarding: "deny",
    allow: {
      topics: [
        "/cmd_vel" // 오직 주행 명령만 외부 해킹(?) 허용
      ]
    }
  }
}

실행: zenoh-bridge-dds -c dds_bridge.json5
이 스크립트는 거대한 공장 안에서 오직 허가된 텔레메트리(원격 통계) 데이터만 LTE 망의 좁은 구멍을 통과하게 만드는 기업형 엣지 톨게이트 세팅의 정수다.
