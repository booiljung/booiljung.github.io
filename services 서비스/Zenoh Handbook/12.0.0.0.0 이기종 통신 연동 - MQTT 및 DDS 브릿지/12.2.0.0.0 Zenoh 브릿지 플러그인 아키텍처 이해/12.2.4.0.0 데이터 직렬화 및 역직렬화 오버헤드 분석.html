12.2.4 데이터 직렬화 및 역직렬화 오버헤드 분석

페이로드는 내용물 그대로 날아간다고 쳤을 때, 과연 통신 오버헤드의 범인은 누구인가?
바로 “포장지를 벗기고 다시 입히는” 과정이다. 이것이 브릿지 지연 시간(Latency)의 유일한 함정이다.

0.1 [인스펙션] 직렬화(Serialization) 마이크로 오버헤드 부검

1. MQTT 브릿지의 오버헤드 (JSON, String)

  • MQTT 는 구조가 워낙 가벼워서 포장지를 벗기는 데 걸리는 시간이 거의 측정 불가능할 정도로 짧다 (sub-microsecond).
  • 하지만 MQTT 의 토픽 이름 스펙 (/ 로 이어진 문자열 매칭) 과 Zenoh 의 Key Expression 의 규칙이 살짝 달라, 문자열 파싱 구문을 처리하느라 약간의 정규식 매칭 오버헤드가 발생한다 (수십 us).

2. DDS 브릿지의 오버헤드 (CDR 변환 지연)

  • DDS 패킷을 뜯어낼 때, 브릿지는 ROS2 가 쓰는 RTPS 헤더 정보를 파헤쳐야 한다. 이게 생각보다 구조가 더럽다 (지터, Liveliness 타이머 등).
  • DDS 의 복잡한 QoS 메타데이터를 Zenoh 의 Reliability 와 Priority 로 매핑하는 과정에서 약간의 연산 딜레이 패널티(대략 0.1 ~ 0.3ms)가 추가된다.

아키텍트의 결론:
시스템 설계자는 이 “브릿지 당 0.3ms 가량의 포장 비용” 을 반드시 시스템 레이턴시(Latency) 예산에 포함해야 한다.
만약 로봇과 클라우드 사이에 브릿지가 총 2단(로봇 쪽 1단, 클라우드 수신 쪽 브릿지로 DDS 다시 변환 시 1단) 으로 깔려있다면, 기본 핑(Ping) 에 무조건 1ms 단위의 직렬화/패킹 비용이 추가된다는 것을 수학적으로 감내해야만 한다.
