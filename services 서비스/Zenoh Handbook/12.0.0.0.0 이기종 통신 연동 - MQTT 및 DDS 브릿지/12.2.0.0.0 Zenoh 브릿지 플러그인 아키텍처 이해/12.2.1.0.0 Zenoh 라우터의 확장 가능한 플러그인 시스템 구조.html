12.2.1 Zenoh 라우터의 확장 가능한 플러그인 시스템 구조

Zenoh 라우터(zenohd) 는 그 자체로는 매우 단순한 형태의 라우팅 허브에 불과하다. 이 라우터가 다른 프로토콜과 대화를 나누기 위해서는 확장 모듈이 필요하다.

0.1 [인스펙션] 동적 적재(Dynamic Loading) 기반의 브릿징

Zenoh 인프라는 철저하게 코어(Core) 와 플러그인(Plugins) 으로 분리되어 있다.

1. 라이브러리 후킹 (DLL / SO injection)

  • zenoh-bridge-dds 같은 브릿지는 사실 독립적인 프로그램으로 켤 수도 있지만, 본질적으로는 라우터 데몬 안에 주입되는 ‘공유 라이브러리(.so, .dll)’ 규격의 플러그인으로 동작하도록 설계되어 있다.
  • 라우터를 구동할 때 환경 설정(zenoh.json5) 의 plugins 항목을 켜면, 라우터는 기동 즉시 운영체제의 동적 링커를 호출하여 메모리에 브릿지 엔진을 직결시켜 버린다.

2. 통신 오버헤드 0 의 영역 (Zero-Copy Bridge)

  • 외부 릴레이 서버를 띄울 때 가장 큰 문제는, 릴레이 프로세스와 라우터 프로세스 간에 다시 한번 ‘로컬 소켓(Localhost Socket)’ 통신을 해야 한다는 것이다.
  • 하지만 Zenoh 의 플러그인 몽골로이드(Mongoloid) 아키텍처 하에서는, 브릿지 코드와 라우터 코드가 하나의 메모리 주소(프로세스) 를 공유한다. DDS 로 들어온 1MB 짜리 패킷이 Zenoh 망으로 넘어갈 때 단 1 바이트의 복사(Copy) 도 일어나지 않는다. 포인터 주소만 넘겨주면 통역이 끝난다.
