12.5.3 글로벌 키 공간(Global Key Space)을 활용한 이기종 네임스페이스 격리

MQTT 센서도 /temperature 를 쏘고, ROS2 로봇 내부의 배터리 센서도 /temperature 를 쏜다. 클라우드 관제사는 두 토픽이 똑같아서 구별할 수 없어 패닉에 빠진다.

0.1 [Runbook] 글로벌 수목 트리(Global Tree) 설계 전술

수백 대의 기종이 섞이는 환경에서는 무조건 클라우드 단에서 이 모든 것을 묶는 “폴더 트리의 꼬리표(Prefix)” 체계를 당신이 설계해야만 한다.

[모범 사례: 도메인 격리 네임스페이스]
모든 브릿지 서버(DDS, MQTT) 가 돌아가는 스크립트에 다음과 같은 규칙으로 --scope 나 json 설정을 강제 주입해라.

  • MQTT 센서망: zenoh-bridge-mqtt ... --scope "facility/building_A"
    -> 클라우드 도달명: facility/building_A/temperature

  • ROS2 물류 로봇: zenoh-bridge-dds ... --scope "fleet/agv_001"
    -> 클라우드 도달명: fleet/agv_001/rt/temperature

  • CCTV 카메라망 (REST API):
    -> 클라우드 도달명: surveillance/cam_04/video_stream

이제 클라우드의 Python 관제 프로그램은
session.subscribe("fleet/*/rt/temperature") 한 방으로 공장 내 모든 로봇의 온도만 깔끔하게 빼낼 수 있다. 브릿지의 세계에서 네임스페이스 설계 실패는 곧 데이터 오염(Data Corruption) 사태로 이어진다.
