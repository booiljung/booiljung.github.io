12.1 이기종 통신 연동의 필요성 및 개요

하나의 언어만 사용하는 완벽한 유토피아는 현실에 존재하지 않는다.
어떤 공장에는 5년 전에 도입된 MQTT 기반의 IoT 센서들이 있고, 옆 라인에는 최신 DDS 기반의 무인운반차(AGV)가 돌아다니며, 천장에는 REST API 로만 통신하는 레거시 CCTV 가 달려 있다.

이 모든 기기들은 서로의 언어를 이해하지 못한 채 거대한 데이터의 사일로(Silo)에 갇혀 있다. 이 장에서는 Zenoh 가 어떻게 중간에 서서 수십 개의 각기 다른 프로토콜의 통역사(Bridge) 역할을 수행하는지, 그 철학과 구조를 파헤친다.

1. 현대 분산 및 에지 컴퓨팅 환경의 통신 프로토콜 파편화

엔지니어가 새로운 시스템을 설계할 때 가장 먼저 빠지는 함정이 바로 **“모든 것을 최신 기술 통일하자”**는 몽상이다.

1.0.1 [인스펙션] 레거시(Legacy) 통신망의 콘크리트 장벽

1. 프로토콜 파편화의 현실

  • IoT 단말기 (ESP8266 등): MQTT 프로토콜이 절대적이다. 브로커(Broker)가 필수적이며, 메모리 풀이 한없이 작다.
  • 클라우드 백엔드 (AWS/GCP): REST API (HTTP), gRPC, WebSockets 가 메인스트림이다.
  • 국방/로보틱스 에지 시스템: DDS (Data Distribution Service) 가 장악하고 있다. 극한의 실시간성과 UDP 멀티캐스트의 변태적인(?) 복잡성을 자랑한다.
  • 차량 전장 시스템 (Automotive): CAN, SOME/IP 와 같은 완전한 이질적 통신 규격을 쓴다.

2. 통일을 막는 본질적 한계
“모두 뜯어내고 최신 시스템으로 교체하면 안 되나?” 불가능하다. 각각의 프로토콜은 자신만의 태생적 제약 위에서 가장 최적화된 형태로 발전해 왔다. 로봇의 라이다(LiDAR) 50MB 영상을 MQTT 로 보내면 브로커 서버가 숨통이 끊어지고, 반대로 온도 센서 데이터 1 Byte 를 보내는데 DDS 의 수 메가바이트짜리 RTPS 라이브러리를 올릴 수는 없다.

현장의 엔지니어는 시스템을 폐기(Replace)하는 것이 아니라, 서로 피를 섞어내는 융합(Integration) 전술을 택해야만 살아남을 수 있다.

2. 브릿지(Bridge) 패턴의 개념과 시스템 통합에서의 역할

시스템의 콘크리트 장벽을 뚫기 위해 우리에게 필요한 것은 서로의 언어를 이해하는 통역사, 즉 브릿지 패턴(Bridge Pattern)이다.

2.0.1 [인스펙션] 브릿지 아키텍처의 설계 대원칙

브릿지 프로그램의 가장 원초적이고 중요한 설계 철학은 바로 비-침투성(Non-intrusive) 이다.

1. 쌍방 몽둥이 (Double-Blind) 원칙

  • A 라는 MQTT 센서와 B 라는 DDS 로봇이 있다고 치자.
  • 브릿지가 도입되었을 때, A 와 B 는 절대로 자신이 다른 언어를 쓰는 상대와 소통하고 있다는 사실을 알아채서는 안 된다.
  • MQTT 센서는 오직 MQTT 브로커와 이야기하고 있다고 믿어야 하고, DDS 로봇은 ROS2 망의 다른 DDS 노드들과 이야기하고 있다고 믿어야 한다.
  • 브릿지는 양쪽 망에 동시에 발을 걸치고, A 가 내뱉은 패킷을 조용히 훔쳐서(Sniffing), B 의 규격으로 분해/조립(Serialization) 한 뒤 찔러 넣는 “스파이” 다.

2. O(N²) 통역의 비효율 타파
만약 프로토콜이 5개(MQTT, DDS, REST, ROS1, CAN) 라면, 서로 1:1 로 연결하는 브릿지를 짜려면 5 * 4 = 20 가지 의 통역 프로그램이 필요하다.
이것은 아키텍처의 재앙이다. 모든 프로토콜은 오직 가운데에 솟아오른 “바벨탑(공통 허브)” 하나하고만 브릿지를 맺어야 한다. 그러면 브릿지의 종류는 단 5개 로 줄어든다. 이 바벨탑의 역할을 수행하기 위해 벼려진 창이 바로 Zenoh 이다.

3. Zenoh를 이기종 통신의 허브로 사용할 때의 기술적 이점

수많은 브로커 엔진 중 왜 하필 “Zenoh” 가 이기종 클라우드를 통합하는 궁극의 바벨탑이 되었는가?

3.0.1 [Runbook] Zenoh Hub 아키텍처 채택 당위성 보고서

1. 극단적인 규격의 유연성 (Schema-less & Key Expression)
Zenoh 는 데이터의 “내용물(Payload)” 이 어떤 구조로 생겼는지 신경 쓰지 않는다. MQTT 의 텍스트든, DDS 의 이진(CDR) 압축 데이터든, REST 의 JSON 이든 그저 바이트(Byte) 덩어리 로 취급한다.
그리고 통신의 주소를 정의하는 방식이 factory/machine/01 과 같은 거대한 디렉토리 구조(Key Expression) 로 되어 있어, 타 프로토콜의 주소 트리(MQTT 의 Topic 이나 HTTP 의 URI)와 1:1 완벽한 호환 매핑이 가능하다.

2. 통신 패러다임의 초월 (Pub/Sub & RPC 대통합)

  • MQTT 는 발송(Pub/Sub)만 할 줄 안다.
  • REST 는 요청(Req/Rep)만 할 줄 안다.
    하지만 Zenoh 는 Pub/Sub 와 Queryable(RPC) 이라는 2가지 무기를 모두 내재하고 있다.
    어떤 종류의 레거시(Legacy) 앱이 연결되더라도, 브릿지가 그 행동 패턴을 Zenoh 의 2가지 스펙 중 하나에 억지로 우겨넣어(Mapping) 완벽히 동작하게 만들 수 있다.

3. 스케일 관통력 (Micro-to-Cloud)
만일 통역 허브(Hub) 가 너무 무겁다면 로봇 단말에서는 브릿지를 돌리지 못하고 서버단으로 통역 부하가 전가된다. 하지만 Zenoh 플러그인(Router/Bridge)은 Rust 로 컴파일되어 램(RAM) 점유율 10MB 대에 불과하다.
Raspberry Pi Zero 부터 데이터센터 서버까지, 원하는 위치 어디든 스파이(Bridge) 를 낙하산으로 침투시켜 가동할 수 있는 독보적인 수평선(Horizon) 을 자랑한다.
