12.5 다중 프로토콜 융합 및 복합 토폴로지 구성

MQTT 와 DDS 를 각각 따로 Zenoh 에 붙이는 법을 배웠다면, 이제는 세 개의 우주를 하나로 융합할 차례다.

ESP32 온습도 센서(MQTT) 의 데이터가 AWS 클라우드(Zenoh) 를 거쳐 ROS2 자율주행 로봇(DDS) 의 브레인으로 들어가 회피 기동을 일으키게 만드는, “이기종 대통합의 아키텍처” 를 설계한다. 이 파트에서는 그 과정에서 맞닥뜨리게 될 심각한 브릿지 루프 현상과 스코프 충돌을 해결하는 전술을 공개한다.

1. MQTT -> Zenoh -> DDS를 잇는 엔드투엔드(End-to-End) 파이프라인 설계

“온습도 센서 10,000개가 건물에 깔려있습니다. 빌딩 안의 방역 로봇(ROS2) 이 이 센서값들을 읽고 가장 뜨거운 방부터 찾아가서 소독하게 해주세요.”

1.0.1 [Runbook] 다중 릴레이 체인 구축 전술

어느 프로토콜에 어느 브릿지를 꽂아야 하는지 명확히 해야 한다. 절대 로봇 안에서 MQTT 와 DDS 브릿지를 동시에 돌리지 마라. 시스템이 녹아내린다.

1. AWS 관제 서버 (중앙 허브 구축)

  • 역할: 모든 센서와 로봇이 바라보는 거대한 트래픽 중계소.
  • 구성: zenohd (라우터 본체) 와 zenoh-bridge-mqtt 플러그인을 활성화하여 1층에 있는 거대한 건물 관제용 MQTT 브로커와 연결시킨다.
  • 효과: 건물 내 모든 센서 데이터가 AWS 의 Zenoh 망으로 실시간 흡수(Subscribe) 된다.

2. 방역 로봇 (종단 노드 세팅)

  • 역할: 데이터 소비 및 주행 명령(DDS) 수행.
  • 구성: 로봇 안에는 zenoh-bridge-dds 만 단독으로 켠다. 그리고 엔드포인트(-e) 를 AWS 의 공유기(zenohd) IP 로 맞춘다.
  • 흐름: AWS 라우터가 MQTT 브릿지로부터 넘겨받은 열화상 센서 데이터를, 즉시 로봇 안의 DDS 브릿지로 쏴버린다. 그 순간 로봇의 Rviz(ROS2) 화면에 건물의 실시간 열 분포도가 오버레이(Overlay) 되어 펼쳐진다!

이 파이프라인의 레이턴시(망 지연 시간)는 인터넷망 상태에 따라 다르지만 보통 30ms 이내로 주파된다. 3개의 완전히 다른 미들웨어를 거쳤음에도 믿을 수 없는 스피드다.

2. 다중 브릿지 환경에서의 데이터 무한 루프(Data Loop) 현상 방지 전략

“로봇 2대에 각각 브릿지를 켰더니 갑자기 초당 트래픽이 10Gbps 를 찍으며 공장 와이파이가 펑 터졌습니다.”

초보 아키텍트들이 가장 많이 저지르는 브릿지 반사(Reflection) 현상이다.

2.0.1 [인스펙션] 거울방(Hall of Mirrors) 타파 전술

현상 부검:

  1. 로봇 A가 DDS로 “Hello” 를 쏜다.
  2. 로봇 A의 브릿지가 이를 주워다 Zenoh 로 넘긴다.
  3. 로봇 B의 브릿지가 Zenoh 망에서 “Hello” 를 주워다 자기 쪽 DDS 망에 푼다.
  4. [재앙의 시작] 로봇 B의 브릿지는 “어? 우리 DDS 망에 누가 Hello 라고 썼네?” 하고 자기가 방금 뿌린 걸 자기가 다시 주워서 Zenoh 망으로 역류(Publish) 시킨다!
    이 사이클이 빛의 속도로 반복되며 1초 만에 수백만 개의 패킷 복제가 일어난다. 폭풍(Storm)이다.

1. GUID 스니핑 방어 기작 (내장형)
다행히 최신 zenoh-bridge-dds 는 똑똑하다. DDS 패킷에 박혀있는 발신자의 고유 ID(GUID) 를 기억해 뒀다가, 이 패킷이 바다 건너 브릿지로 들어갈 때 “아, 이건 내가 보냈던 놈의 흔적이잖아?” 하고 쳐낸다.

2. Scope 명시적 강제 분리 (아키텍트의 안전벨트)
기계의 자율적 방어에만 의존하지 마라.
로봇 A의 브릿지를 켤 때 --scope robotA 로 강제하고, 로봇 B 브릿지를 --scope robotB 로 강제하라.
이렇게 하면 A 가 퍼블리시한 글은 robotA/Hello 가 되고, B는 오직 robotB/Hello 만 발송한다. 무한 리디렉션 메아리가 근원적으로 100% 차단된다.

3. 글로벌 키 공간(Global Key Space)을 활용한 이기종 네임스페이스 격리

MQTT 센서도 /temperature 를 쏘고, ROS2 로봇 내부의 배터리 센서도 /temperature 를 쏜다. 클라우드 관제사는 두 토픽이 똑같아서 구별할 수 없어 패닉에 빠진다.

3.0.1 [Runbook] 글로벌 수목 트리(Global Tree) 설계 전술

수백 대의 기종이 섞이는 환경에서는 무조건 클라우드 단에서 이 모든 것을 묶는 “폴더 트리의 꼬리표(Prefix)” 체계를 당신이 설계해야만 한다.

[모범 사례: 도메인 격리 네임스페이스]
모든 브릿지 서버(DDS, MQTT) 가 돌아가는 스크립트에 다음과 같은 규칙으로 --scope 나 json 설정을 강제 주입해라.

  • MQTT 센서망: zenoh-bridge-mqtt ... --scope "facility/building_A"
    -> 클라우드 도달명: facility/building_A/temperature

  • ROS2 물류 로봇: zenoh-bridge-dds ... --scope "fleet/agv_001"
    -> 클라우드 도달명: fleet/agv_001/rt/temperature

  • CCTV 카메라망 (REST API):
    -> 클라우드 도달명: surveillance/cam_04/video_stream

이제 클라우드의 Python 관제 프로그램은
session.subscribe("fleet/*/rt/temperature") 한 방으로 공장 내 모든 로봇의 온도만 깔끔하게 빼낼 수 있다. 브릿지의 세계에서 네임스페이스 설계 실패는 곧 데이터 오염(Data Corruption) 사태로 이어진다.

4. 에지(DDS/로컬)와 클라우드(MQTT/원격) 간의 데이터 병목 해소 토폴로지

에지단에서 쏟아지는 초당 기가바이트(GB) 비전 데이터와, MQTT 밖에 이해 못 하는 외주 협력사의 클라우드 대시보드를 연결해야 하는 지옥불 시나리오다.

4.0.1 [Runbook] 데이터 칼질 (Edge Filtration) 및 QoS 압박 전술

초당 1GB 가 넘는 DDS 카메라 데이터를 아무리 브릿지를 타고 넘긴다 한들, 종단에 있는 MQTT 브로커(Mosquitto 등)는 그 트래픽을 받는 즉시 숨이 막혀 죽어버린다.

1. 에지 브릿지 필터링 (최전방 방어막)
로봇 안의 DDS 가 폭주하더라도 밖으로 나가는 관문에서 사살한다.
로봇 zenoh-bridge-dds 의 허용(Allow) 목록에 절대 대용량 비전 데이터(/image_raw) 를 적지 마라. 오로지 협력사가 필요로 하는 /robot_status 상태 데이터만 허용 범위에 적어 넣는다.

2. MQTT 포맷의 이질성 극복 (JSON 변환기 투입)
ROS2 로봇이 보낸 데이터는 Float32 형식의 깨진 이진 덤프 바이트(CDR) 다. AWS 에 꽂혀있는 zenoh-bridge-mqtt 가 이를 MQTT 브로커로 밀어넣어 봤자 클라우드의 Node.js 는 해독 불가능한 쓰레기로 인식한다.

  • 해결책: 이 거대한 아키텍처 중간(에지 라우터나 클라우드 입구)에 작은 Zenoh-Python 스크립트 를 띄운다.
  • Python 의 임무: Zenoh 망에서 도는 CDR 이진 바이트를 z_sub 로 받아서, struct.unpack() 으로 파싱한 뒤, 깔끔한 { "status": "running", "battery": 94.2 } 형태의 UTF-8 JSON 문자열로 바꿔서 다시 Zenoh 에 Publish 한다.
  • 그러면 MQTT 브릿지는 이 JSON 데이터를 예쁘게 MQTT 브로커로 넘겨줄 수 있다!

결론: 브릿지는 그저 언어 형태만 바꿔주는 파이프다. 복잡한 페이로드 직렬화 포맷(CDR -> JSON) 을 풀어헤치는 임무는, 가볍고 빠른 Python/Rust 마이크로 워커(Worker) 를 Zenoh 망 사이에 “플러그(Plug)” 처럼 꽂아서 해결해야만 완벽한 에지-클라우드 융합 인프라가 완성된다.
