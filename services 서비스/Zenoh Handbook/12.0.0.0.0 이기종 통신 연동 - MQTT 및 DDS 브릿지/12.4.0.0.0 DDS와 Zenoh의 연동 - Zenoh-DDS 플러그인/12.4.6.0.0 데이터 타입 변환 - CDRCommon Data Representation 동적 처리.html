12.4.6 데이터 타입 변환: CDR(Common Data Representation) 동적 처리

DDS는 본인의 데이터가 몇 바이트짜리 정수(Int)인지 어디서 끝나는지 메타데이터를 담은 .idl (혹은 .msg) 스키마 파일을 달고 다닌다. 반면 Zenoh 는 그런 거 신경 안 쓰고 그냥 이진 덩어리(Byte Array) 를 던져버린다.

0.1 [인스펙션] 스키마(Schema) 파괴와 재생성

1. Opaque Payload (불투명한 데이터) 전략
DDS 에서 브릿지를 거쳐 Zenoh 로 넘어갈 때, 브릿지는 DDS 의 CDR(빅엔디안/리틀엔디안이 섞여 있는 C++ 메모리 덤프 형태) 규격으로 구워진 바이트 배열을 해독하려 들지 않는다.
대신 이 바이트 배열 앞에 작은 편지(Header)를 붙인다.

  • “이 데이터의 이름은 std_msgs::msg::dds_::String_ 형식이야!”

2. 수신 측 브릿지의 부활
Zenoh 망을 타고 날아온 이 데이터가 저 멀리 클라우드의 또 다른 DDS 로 들어갈 차례다. 클라우드 측 브릿지는 저 편지를 뜯어보고는, DDS 망에 외친다 “어이구, 이 데이터는 std_msgs::msg... 타입이네.” 하고 DDS 의 형식(Type) 시스템에 맞춰서 완벽하게 재생산(Re-publish) 해준다.

결과적으로, 이기종 간의 브릿징 시스템을 구현할 때 양쪽 시스템은 IDL 파일을 브릿지에게 일일이 알려주거나 컴파일할 필요 없이 “100% 동적 타입 호환(Dynamic Type Resolution)” 이 이루어진다. 이것이 브릿지를 켤 때 아무런 코딩도 필요 없는 기적의 원리다.
