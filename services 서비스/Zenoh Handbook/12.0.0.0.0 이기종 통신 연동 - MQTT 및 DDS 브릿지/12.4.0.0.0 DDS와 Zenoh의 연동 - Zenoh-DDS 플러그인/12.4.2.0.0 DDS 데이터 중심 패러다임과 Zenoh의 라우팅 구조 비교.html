12.4.2 DDS 데이터 중심 패러다임과 Zenoh의 라우팅 구조 비교

왜 굳이 DDS 를 Zenoh 로 바꾸려 하는 건가? 두 철학의 극명한 한계차이 때문이다.

0.1 [인스펙션] 분산 캐싱 vs 라우팅 트리의 격돌

[DDS: 분산 데이터베이스의 흉내]
로봇 A가 /map (지도) 토픽을 수신하면, DDS 엔진은 노드 스레드 안에 이 10MB 짜리 지도를 통째로 캐싱한다. 만약 로봇 프로그램 안에 지도 보는 노드가 5개가 띄워져 있다면, 똑같은 데이터가 메모리 공간을 5번 장악한다. “데이터 중심(Data-centric)” 의 무서운 메모리 폭식이다.

[Zenoh: 제로에 가까운 터미널링]
Zenoh 통신망에서 데이터는 머물지 않는다(Non-blocking & Zero-copy). 그저 파이프 위를 빠르게 지나갈 뿐이다. 라우터만이 데이터를 잠시 쥐고 있다가, 달라고(Sub) 요청한 20개의 클라이언트에게 포인터(소켓)로 데이터를 부채살처럼 흩뿌린다. 메모리를 극도로 아끼면서도 망의 도달 범위를 무한정 넓히는(WAN Scaling) 현대적 아키텍처다. 브릿지는 DDS 의 무거운 “상태 보존” 정책을 이 “실시간 흐름” 으로 강제 역치환하는 파이프의 역할을 맡는다.
