12.6.1 프로토콜 변환 지연 시간(Latency) 및 처리량(Throughput) 측정

가장 흔한 컴플레인은 “브릿지를 달았더니 통신이 너무 느려요!” 다. 변명하지 말고 숫자로 반박하라.

0.1 [Runbook] 마이크로 벤치마킹 타격 전술

브릿지가 진짜 병목인지, 네트워크 망이 밀린 것인지 검증해야 한다.

1. RTT (Round Trip Time) 테스트를 통한 순수 변환 딜레이 추출
로컬 호스트(동일 컴퓨터) 안에 DDS 노드, 브릿지, Zenoh Ping 도구를 다 띄워놓고 네트워크 변수를 배제한 순수 “통역 시간” 을 잰다.

  • 보통 MQTT 브릿지의 경우 1바이트 변환에 0.05ms (50 microseconds), DDS 의 경우 복잡한 CDR 직렬화로 인해 0.2ms ~ 0.5ms 의 딜레이가 측정된다.
  • 이 숫자를 넘어서 수 밀리초(ms) 단위의 지연이 일어난다면? 브릿지가 할당받은 CPU 코어가 모자라서 운영체제 수준에서 컨텍스트 스위칭(Context Switching) 이 일어나고 있는 것이다.

2. Throughput 최대 한계치 확인
대용량 데이터(비디오 스트림 등)를 브릿지로 넘길 때 측정을 한다.

  • DDS 카메라 노드에서 브릿지로 1Gbps 로 때려 박아본다.
  • 브릿지의 CPU 사용률이 100% 를 찍으며 뻗는다면 그것이 그 하드웨어(Raspberry Pi 등) 에서 돌릴 수 있는 브릿지 통역의 절대적 물리 한계다. 브릿지를 2개를 켜서 토픽을 반씩 나누거나 (부하 분산), 12.5.4 장의 Opaque 매핑으로 직렬화를 피하는 꼼수를 써야 한다.
