12.6.3 로깅 시스템을 활용한 이기종 네트워크 상태 진단

“통신이 안 됩니다.” 엔지니어에게는 최악의 불친절한 보고다. 로깅 레벨을 조절하여 패킷이 어디서 증발했는지 정확히 타격점을 찾아낸다.

0.1 [Runbook] 트레이스(Trace) 추적 심문 기법

브릿지는 평소에는 입을 꾹 닫고 있지만(INFO 레벨), 환경변수 주사기를 놓는 순간 엄청난 양의 기밀을 쏟아낸다.

1. DDS 단절 의심 시
브릿지가 ROS2 노드를 도무지 찾지 못하는 상황이다.

RUST_LOG="zenoh_bridge_dds=trace,dds=debug" zenoh-bridge-dds
  • 화면에 Discovered Sub..., Discovered Pub... 로그가 안 뜬다? 브릿지가 DDS 멀티캐스트 네트워크(UDP) 를 못 잡고 있는 것이다. 리눅스 방화벽(ufw) 이 포트를 막고 있는지 확인해라.

2. MQTT 권한 거부(Permission) 의심 시

RUST_LOG="zenoh_bridge_mqtt=debug" zenoh-bridge-mqtt -b tcp/aws.com:1883
  • 화면에 [WARN] Connection closed by broker 혹은 Reason: Not Authorized 가 뜬다? 당신의 AWS 보안 그룹(Security Group) 이 막혔거나, 12.3.7 장에서 세팅한 ID/PW 나 인증서가 만료된 것이다. 백날 Zenoh 포럼 가서 불평해 봤자 소용없다. 클라우드 관리자에게 전화하라.

3. 트래픽 무한 루프 감지기
갑자기 화면에 [WARN] Dropping packet to prevent routing loop 로그가 미친 듯이 쇄도한다면, 12.5.2 장에서 경고했던 “거울방(Reflection)” 버그가 터진 것이다. 즉각 브릿지의 스코프(–scope) 설정을 분리해야 한다.
