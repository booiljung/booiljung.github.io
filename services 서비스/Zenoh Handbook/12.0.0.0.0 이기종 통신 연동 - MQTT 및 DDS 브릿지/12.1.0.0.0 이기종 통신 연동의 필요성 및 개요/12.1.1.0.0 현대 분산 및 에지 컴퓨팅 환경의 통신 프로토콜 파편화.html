12.1.1 현대 분산 및 에지 컴퓨팅 환경의 통신 프로토콜 파편화

엔지니어가 새로운 시스템을 설계할 때 가장 먼저 빠지는 함정이 바로 **“모든 것을 최신 기술 통일하자”**는 몽상이다.

0.1 [인스펙션] 레거시(Legacy) 통신망의 콘크리트 장벽

1. 프로토콜 파편화의 현실

  • IoT 단말기 (ESP8266 등): MQTT 프로토콜이 절대적이다. 브로커(Broker)가 필수적이며, 메모리 풀이 한없이 작다.
  • 클라우드 백엔드 (AWS/GCP): REST API (HTTP), gRPC, WebSockets 가 메인스트림이다.
  • 국방/로보틱스 에지 시스템: DDS (Data Distribution Service) 가 장악하고 있다. 극한의 실시간성과 UDP 멀티캐스트의 변태적인(?) 복잡성을 자랑한다.
  • 차량 전장 시스템 (Automotive): CAN, SOME/IP 와 같은 완전한 이질적 통신 규격을 쓴다.

2. 통일을 막는 본질적 한계
“모두 뜯어내고 최신 시스템으로 교체하면 안 되나?” 불가능하다. 각각의 프로토콜은 자신만의 태생적 제약 위에서 가장 최적화된 형태로 발전해 왔다. 로봇의 라이다(LiDAR) 50MB 영상을 MQTT 로 보내면 브로커 서버가 숨통이 끊어지고, 반대로 온도 센서 데이터 1 Byte 를 보내는데 DDS 의 수 메가바이트짜리 RTPS 라이브러리를 올릴 수는 없다.

현장의 엔지니어는 시스템을 폐기(Replace)하는 것이 아니라, 서로 피를 섞어내는 융합(Integration) 전술을 택해야만 살아남을 수 있다.
