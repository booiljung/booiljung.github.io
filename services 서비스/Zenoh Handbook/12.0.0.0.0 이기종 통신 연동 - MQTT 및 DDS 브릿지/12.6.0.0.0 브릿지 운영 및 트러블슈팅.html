12.6 브릿지 운영 및 트러블슈팅

언어가 다른 두 집단을 묶어 놓으면 사건 사고가 끊이질 않는다.
이기종 브릿징 환경에서 시스템이 마비되었을 때, 이것이 MQTT 브로커가 뻗은 것인지, 센서가 죽은 것인지, 아니면 당신이 켠 브릿지 코드에서 무한 루프가 터져 메모리가 터진 것인지를 1분 만에 가려내는 긴급 장애 대응 런북이다.

1. 프로토콜 변환 지연 시간(Latency) 및 처리량(Throughput) 측정

가장 흔한 컴플레인은 “브릿지를 달았더니 통신이 너무 느려요!” 다. 변명하지 말고 숫자로 반박하라.

1.0.1 [Runbook] 마이크로 벤치마킹 타격 전술

브릿지가 진짜 병목인지, 네트워크 망이 밀린 것인지 검증해야 한다.

1. RTT (Round Trip Time) 테스트를 통한 순수 변환 딜레이 추출
로컬 호스트(동일 컴퓨터) 안에 DDS 노드, 브릿지, Zenoh Ping 도구를 다 띄워놓고 네트워크 변수를 배제한 순수 “통역 시간” 을 잰다.

  • 보통 MQTT 브릿지의 경우 1바이트 변환에 0.05ms (50 microseconds), DDS 의 경우 복잡한 CDR 직렬화로 인해 0.2ms ~ 0.5ms 의 딜레이가 측정된다.
  • 이 숫자를 넘어서 수 밀리초(ms) 단위의 지연이 일어난다면? 브릿지가 할당받은 CPU 코어가 모자라서 운영체제 수준에서 컨텍스트 스위칭(Context Switching) 이 일어나고 있는 것이다.

2. Throughput 최대 한계치 확인
대용량 데이터(비디오 스트림 등)를 브릿지로 넘길 때 측정을 한다.

  • DDS 카메라 노드에서 브릿지로 1Gbps 로 때려 박아본다.
  • 브릿지의 CPU 사용률이 100% 를 찍으며 뻗는다면 그것이 그 하드웨어(Raspberry Pi 등) 에서 돌릴 수 있는 브릿지 통역의 절대적 물리 한계다. 브릿지를 2개를 켜서 토픽을 반씩 나누거나 (부하 분산), 12.5.4 장의 Opaque 매핑으로 직렬화를 피하는 꼼수를 써야 한다.

2. 브릿지 프로세스의 CPU 및 메모리 사용률 최적화 기법

브릿지는 통신 트래픽의 모든 칼바람을 온몸으로 맞는 최전방 방어막이다. 세팅이 틀어지면 브릿지가 로봇 컴퓨터의 RAM 의 절반을 집어삼키는 괴물이 된다.

2.0.1 [Runbook] 메모리 다이어트 및 시스템 자원 결발 전술

1. DDS 브릿지 메모리 폭주 (OOM) 억제

  • 증상: ROS2 가 떠 있는 로봇 안에 켠 브릿지가 점점 메모리를 갉아먹다가 OOM(Out of Memory) 으로 강제 종료된다.
  • 주원인: 로봇이 대용량 데이터(카메라, 레이저 등) 를 TRANSIENT_LOCAL (내구성 옵션 켜짐) 로 쏠 경우 브릿지가 이걸 Zenoh 캐시에 우겨넣으려다 램이 터진다.
  • 타개책: 브릿지 옵션 중 QoS 매핑을 무조건 VOLATILE (버림) 로 강제 강등시켜라.

2. CPU 코어 락온 (Taskset)
브릿지의 패킷 해독 속률을 최상으로 끌어올리려면 운영체제가 브릿지 스레드를 이 코어 저 코어로 넘기며 일어나는 캐시 미스(Cache Miss) 를 차단해야 한다.

## 리눅스의 1번과 2번 CPU 코어에만 브릿지 데몬을 강제 결속시킨다!
sudo taskset -c 1,2 zenoh-bridge-dds -c my_config.json5

이 한 줄로 브릿지의 Jitter(지연 시간 흔들림 폭) 가 30% 이상 극적으로 안정되는 튜닝의 마법을 겪게 될 것이다.

3. 로깅 시스템을 활용한 이기종 네트워크 상태 진단

“통신이 안 됩니다.” 엔지니어에게는 최악의 불친절한 보고다. 로깅 레벨을 조절하여 패킷이 어디서 증발했는지 정확히 타격점을 찾아낸다.

3.0.1 [Runbook] 트레이스(Trace) 추적 심문 기법

브릿지는 평소에는 입을 꾹 닫고 있지만(INFO 레벨), 환경변수 주사기를 놓는 순간 엄청난 양의 기밀을 쏟아낸다.

1. DDS 단절 의심 시
브릿지가 ROS2 노드를 도무지 찾지 못하는 상황이다.

RUST_LOG="zenoh_bridge_dds=trace,dds=debug" zenoh-bridge-dds
  • 화면에 Discovered Sub..., Discovered Pub... 로그가 안 뜬다? 브릿지가 DDS 멀티캐스트 네트워크(UDP) 를 못 잡고 있는 것이다. 리눅스 방화벽(ufw) 이 포트를 막고 있는지 확인해라.

2. MQTT 권한 거부(Permission) 의심 시

RUST_LOG="zenoh_bridge_mqtt=debug" zenoh-bridge-mqtt -b tcp/aws.com:1883
  • 화면에 [WARN] Connection closed by broker 혹은 Reason: Not Authorized 가 뜬다? 당신의 AWS 보안 그룹(Security Group) 이 막혔거나, 12.3.7 장에서 세팅한 ID/PW 나 인증서가 만료된 것이다. 백날 Zenoh 포럼 가서 불평해 봤자 소용없다. 클라우드 관리자에게 전화하라.

3. 트래픽 무한 루프 감지기
갑자기 화면에 [WARN] Dropping packet to prevent routing loop 로그가 미친 듯이 쇄도한다면, 12.5.2 장에서 경고했던 “거울방(Reflection)” 버그가 터진 것이다. 즉각 브릿지의 스코프(–scope) 설정을 분리해야 한다.

4. MQTT 및 DDS 연동 시 발생하는 대표적 오류 코드 및 해결 방안

경험 많은 아키텍트들의 눈물과 밤샘이 누적된 장애 해결의 핵심 바이블이다.

4.0.1 [인스펙션] 크로스 매핑 오류 부검 리스트

1. [ERROR] Unable to convert payload from MQTT to DDS

  • 증상: MQTT 에서 보낸 센서값이 ROS2 화면에 안 뜬다.
  • 원인: MQTT 는 문자열("{temp:20}") 을 보냈는데, 수신 측 ROS2 앱은 C++ 의 Float32 메모리 이진 덤프를 기다리고 있다. 데이터 타입(Type) 불일치로 브릿지가 번역을 포기한 것이다.
  • 해결책: 12.5.4 장의 런북을 참고하여, 중간에 JSON 을 CDR 규격으로 파싱해 주는 마이크로 파이썬 스크립트를 하나 띄워서 포맷을 일치시켜라. 브릿지는 마법사가 아니다.

2. [FATAL] Failed to initialize DDS participant

  • 증상: 로봇 안에서 브릿지를 켜는 순간 바로 꺼져버린다.
  • 원인: 브릿지와 ROS2 코어가 리눅스 커널의 공유 메모리(SHM) 세그먼트를 서로 잡아먹으려다 충돌한 것이다. (보통 CycloneDDS 환경의 버그).
  • 해결책: export CYCLONEDDS_URI=... 설정을 주어 브릿지의 DDS 통신을 공유메모리(SHM) 방식이 아닌 순수 UDP 네트워크 모드로 강제 격하시켜라.

3. 브릿지는 죽지 않았는데 데이터가 한 방향으로만 안 흐른다!

  • 증상: 로봇 상태는 클라우드로 잘 올라가는데, 클라우드 관제 대시보드에서 내리는 조작 명령이 로봇으로 들어가지 않는다. (절반의 성공)
  • 원인: 십중팔구 브릿지의 Allow / Deny (화이트리스트) 설정을 한쪽 방향만 열어 두었기 때문이다. zenoh_to_dds 정책과 dds_to_zenoh 정책이 서로 독립적으로 구성되어 있다는 기초를 망각한 실수다. 설정 파일의 forwarding 항목을 다시 읽어라.
