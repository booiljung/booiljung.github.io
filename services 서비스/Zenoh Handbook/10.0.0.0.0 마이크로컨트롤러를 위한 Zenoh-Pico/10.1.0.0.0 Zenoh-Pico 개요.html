10.1 Zenoh-Pico 개요

당신의 화분 습도 센서에 리눅스(Linux)가 깔려있는가? 스마트 전구에 도커(Docker) 컨테이너가 도는가? 아니다.
전 지구 센서의 99%는 리눅스를 돌릴 힘조차 없는 초소형 마이크로컨트롤러(MCU)다.

Zenoh 프로토콜을 이들에게 주입하기 위해 만들어진 Zenoh-Pico 에 대한 본질적이고 철학적인 고찰을 시작한다. 이 챕터는 어떤 칩(Chip)을 골라야 이 경이로운 통신망에 합류할 수 있는지, 그리고 왜 기존의 zenoh-c 라이브러리를 갖다 쓰면 칩이 불타서 뻗어버리는지를 명확히 규명한다.

1. 마이크로컨트롤러와 제한된 리소스 환경의 이해

당신이 만약 웹 개발자 출신이라면, 여기서부터 상식을 완전히 뒤집어야 한다. C/C++ 시스템 프로그래머라 할지라도 리눅스가 주는 혜택에 취해 있었다면 여기서부터 다시 일어서야 한다.

1.0.1 [인스펙션] 제약이라는 이름의 교도소 (Constraint Sandbox)

  1. 메모리의 빈곤 (RAM < 50KB)
    일반 라즈베리 파이가 최소 1GB 의 램을 갖는 반면, STM32G0 계열 칩은 단 8KB 의 램(SRAM)을 가진다. Node.js 가 부팅될 때 쓰는 메모리의 1 / 10,000 도 안 된다. 여기서 JSON 파서를 돌리거나 복잡한 프로토콜 헤더를 다는 것은 자살 행위다.

  2. 메모리 단편화 (Memory Fragmentation)의 공포
    MCU 환경에서는 malloc 을 함부로 쓰지 못한다. 가비지 콜렉터가 없기 때문에 할당과 해제를 반복하다 보면 램에 아주 작은 구멍(Hole)들이 생긴다. 그러다 어느 날 “1KB 연속 공간 할당” 에 실패하면 로봇이 주행 중 멈춰버린다. (Hard Fault).

  3. 운영체제의 부재 (No OS, Bare-metal)
    보안(TLS, SSL)을 처리하려면 난수 생성 시스템(/dev/random)이나 멀티 스레딩이 필요한데, 리눅스가 없는 칩에서는 이 모든 게 사치다. 네트워크 통신을 한 번 하기 위해 메인 프로그램 루프(while(1))가 대기(Block)를 타버리면 LED 깜빡임조차 멈춘다.

이 압도적인 결핍의 공간 속으로 네트워크를 밀어 넣기 위해, MQTT나 CoAP 같은 경량 프로토콜들이 등장했다. 하지만 그들은 ’속도’나 ‘멀티캐스트’ 능력을 타협해 버렸다. Zenoh-Pico는 타협하지 않고 어떻게 이 8KB 램 환경 속에 멀티캐스트 라우팅 기능을 우겨넣었을까?

2. Zenoh-Pico의 설계 철학과 경량화 전략

Zenoh-Pico 의 코드를 열어보면 “이게 통신 라이브러리라고?” 할 정도로 아무것도 없다.
엔지니어들이 뼈를 깎아낸 경량화의 진수 3가지를 파헤친다.

2.0.1 [아키텍처 인스펙션] 스파르탄 경량화 전술

1. 동적 메모리 할당(malloc)의 영구 추방 - Zero Dynamic Allocation
Zenoh-Pico 는 작동 시 무조건 “사용자의 펌웨어 안에서 미리 선언한 전역 배열(Static Array)” 만을 빌려 쓴다. 즉 부팅 이후에는 죽을 때까지 단 1바이트의 힙(Heap) 메모리도 사용하지 않으며, 이로 인해 OOM(Out of Memory)이나 메모리 파편화가 원천 불가능하다.

2. 99% 의 지능을 라우터(Router)로 떠넘기기 - Dumb Client
로봇(MCU) 스스로가 망의 위상(Topology)이나 최단 경로를 계산하는 기능을 다 뺏어버렸다. Zenoh-Pico 는 그저 눈앞에 연결된 단 하나의 라우터(Zenoh-Rust)에게 맹목적으로 패킷을 던질 뿐이다. “어디로 갈 건지” 에 대한 고민 조차 칩 안에서 지워버렸다.

3. 완전한 포팅 자율성 - Bring Your Own Network (BYON)
Zenoh-Pico 안에는 TCP/IP 코드가 없다! 심지어 시리얼(UART) 코드도 없다.
이 녀석은 그냥 “OS 독립적인 바이트 조작 기계” 일 뿐이다. 당신의 칩이 블루투스를 쓰든, RS-485 유선을 쓰든, TCP 소켓을 쓰든 간에, 당신이 “바이트를 하나 써라(Write), 하나 읽어라(Read)” 라는 빈 껍데기 함수(Porting Layer) 두 개만 구현해주면, 그 위에서 마법처럼 전 세계 통신망이 가동된다.

3. 일반 Zenoh (C/Rust)와 Zenoh-Pico의 차이점 및 한계

“왜 그냥 zenoh-c 를 쓰면 안 되나요? 똑같은 C 언어 잖아요.”
zenoh-c 는 내부적으로 가장 뚱뚱한 Rust 엔진(zenoh-rs) 을 백그라운드 스레드로 몰래 돌리고 있는 “거대 괴수” 인 반면, Zenoh-pico 는 진정한 순수(Pure) C 덩어리다.

3.0.1 [Runbook] 엔진 선택의 데스매치 (Decision Tree)

이 두 가지를 잘못 선택하면 칩이 불타버린다.

[한계점의 명확한 인지]
zenoh-pico 를 박은 당신의 로봇은 시스템 라우터(Zenoh Router) 역할을 절대로 할 수 없다. 그리고 스스로 백그라운드 핑(Ping)을 쳐주거나, 연결이 끊겼을 때 내부에서 큐를 보관하는(Store) 똑똑한 짓을 하지 못한다. 메인 보드에 붙은 똑똑한 컴퓨터(라우터)에게 보내는 “바이너리 주사기“일 뿐이다.

4. 지원하는 하드웨어 플랫폼 및 최소 요구 제원

당신의 팀이 단가 1천 원짜리 칩 1만 개를 사서 스마트 기기를 만들려 한다. 과연 Zenoh 가 이 칩에 들어갈 수 있을까? 하드웨어 발주 직전 반드시 점검해야 할 “생존 요구 조건(Survival Spec)“이다.

4.0.1 [인스펙션] Zenoh-Pico 아사(구굶어 죽기) 방지 스펙

1. 물리적 메모리 요구량 (절대적)

  • Flash Memory (코드 저장 공간): 최소 30KB ~ 50KB
  • zenoh-pico 의 순수 C 기능 함수들이 자리를 차지하는 용량이다.
  • SRAM (실행 메모리): 최소 2KB ~ 5KB
  • 패킷을 쏘거나 받기 위해 메모리에 올릴 버퍼가 필요하다. 만약 쏘려고 하는 센서 데이터가 1KB 라면 램은 넉넉히 4KB 정도만 있어도 구동된다.

2. 통신 인프라 요구량 (필수 포팅 조건)

  • 칩 자체가 인터넷이 되는 모듈(WiFi/Ethernet)을 가졌거나 (예: ESP32)
  • 아니면 인터넷이 안되어도 시리얼통신(UART)나 블루투스 칩을 가지고 있으며, 이 하드웨어 드라이버를 poll() / read() / write() 구조로 개발자가 우회 연결(Porting)해 줄 수 있어야 한다.

3. 공식 지원 및 널리 쓰이는 플랫폼

  • ESP32 & ESP8266 (FreeRTOS) : 현존 최고의 호환성. WiFi가 알아서 잡히며 전 세계 IoT 튜토리얼 1순위.
  • STM32 구형~신형 칩셋 (Bare-metal 및 RTOS) : 산업용 모터 제어의 표준.
  • Microchip (PIC32, AVR), Arduino 등 C99 표준 컴파일러(GCC) 가 돌아가는 세상의 거진 100% MCU 칩셋들.

결론적으로 당신의 칩이 void main() 을 돌릴 수 있고, 바깥으로 바이트를 하나 쏠 수 있는 구멍(TX/RX, WiFi)만 있다면 Zenoh-Pico 는 그 안에 완벽하게 기생하여 전 세계로 뻗어나갈 수 있다.
이 엄청난 범용성이 Zenoh가 차세대 디팩토 표준(De-facto standard)으로 꼽히는 이유다.
