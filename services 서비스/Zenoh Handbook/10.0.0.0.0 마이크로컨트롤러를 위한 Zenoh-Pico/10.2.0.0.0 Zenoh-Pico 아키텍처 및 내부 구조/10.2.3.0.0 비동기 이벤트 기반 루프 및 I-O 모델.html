10.2.3 비동기 이벤트 기반 루프 및 I/O 모델

Zenoh-Pico 안에는 스레드(Thread)를 분파해서 영원히 네트워크를 돌려주는 고마운 데몬 프로세스가 없다.
데이터가 왔는지 안 왔는지, 주기적으로 칩의 메인 루프 안에 들어가서 “문 좀 열어달라고(Poll)” 노크를 해 줘야 한다.

0.1 [Runbook] 협력적(Cooperative) 스웦(Sweep) 폴링 전술

당신의 기기 펌웨어 최하단 while(1) 루프(또는 FreeRTOS 의 메인 태스크) 안에 반드시 Zenoh-Pico 에게 “숨 쉴 기회” 를 지속적으로 공급해줘야 한다.

#include <zenoh-pico.h>

void my_robot_firmware_main() {
    // ... [1] 하드웨어 초기화 (GPIO, UART 설정 등)
    // ... [2] Zenoh-Pico 세션 개화 (z_open)

    while (1) {
        // [3] 로봇 펌웨어 고유의 일 (센서 읽기, 모터 PWM 제어 등)
        read_sensor();
        drive_motor();

        // [4] 핵심! Zenoh 엔진 숨 쉬게 만들기 (10ms 대기 한도)
        // 이 함수를 호출해야만 비로소 네트워크 칩에 쌓인 데이터가 
        // 내 Subscriber 콜백 함수들로 라우팅되어 불타게 된다!
        z_session_yield(&session, 10); 
        
        // 주의: 이 z_session_yield() 가 불리는 주기가 불규칙하거나 너무 느리면,
        // (예: drive_motor 에서 3초 지연됨)
        // TCP 버퍼가 가득 차서 연결된 Zenoh Router 가 당신을 "죽은 세션" 으로 간주하고
        // 망에서 강제 퇴출(Drop)시켜버린다.
    }
}

MCU 네트워크 프로그램이 뒤지는 원인의 80%는 이 yield 와 하드웨어 I/O 로직 사이의 “시간 분배 실패” 에서 비롯된다. 메인 함수 안에서 delay() 나 sleep() 을 남발하는 주니어 코더는 결코 이 이기종 멀티캐스트 네트워크를 통제할 수 없다.
