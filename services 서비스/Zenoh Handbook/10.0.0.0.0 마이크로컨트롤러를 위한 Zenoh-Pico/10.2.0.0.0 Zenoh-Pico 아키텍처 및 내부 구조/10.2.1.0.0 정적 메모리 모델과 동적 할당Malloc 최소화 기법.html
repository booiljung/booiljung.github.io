10.2.1 정적 메모리 모델과 동적 할당(Malloc) 최소화 기법

zenoh-c 였다면 z_publisher_declare() 를 부르는 순간 백그라운드 환경에서 수 킬로바이트의 힙(Heap) 메모리가 malloc 을 통해 동떨어지며 할당되었을 것이다.
Zenoh-Pico 는 이것을 철저히 금지한다.

0.1 [Runbook] 정적(Static) 구조체 선언 전술

Pico 세계에서는 당신이 직접 램(RAM) 지도를 그려서 Zenoh 에게 “이 공간을 써라” 고 갖다 바쳐야 한다. 이것이 유일한 규칙이다.

#include <zenoh-pico.h>
#include <stdio.h>

// 1. [핵심] 힙(Heap)이 아니라 데이터 포맷(Data Segment)이나 BSS 에 박히는 전역 변수들
// 운영체제가 프로그램 시작 전에 이미 메모리 한 켠에 확정해 놓는(Static Allocation) 공간이다.
uint8_t rx_buffer[1024]; 
uint8_t tx_buffer[1024];

z_session_t session; 
z_publisher_t publisher;

void boot_pico() {
    // 2. Pico 용 설정 깎기
    z_config_t config = z_config_default();
    
    // 3. 내가 만든 메모리 창고(전역 변수)의 주소값을 Pico 에게 먹인다!
    // "이 버퍼 안에서만 물놀이를 하렴. 절대 malloc 을 쓰지마!"
    zp_config_insert_json(&config, ZP_RX_BUF_ADDR_KEY, (char*)&rx_buffer);
    zp_config_insert_json(&config, ZP_RX_BUF_SIZE_KEY, "1024");
    
    zp_config_insert_json(&config, ZP_TX_BUF_ADDR_KEY, (char*)&tx_buffer);
    zp_config_insert_json(&config, ZP_TX_BUF_SIZE_KEY, "1024");

    // 4. 세션 기동 (이 안에서 malloc 은 1도 호출되지 않는다)
    z_open(&session, &config, NULL);
}

이 패턴의 파괴적인 장점은, 아무리 통신을 수십만 번을 때려도 Fragmentation(단편화) 에 의한 메모리 고갈 사태가 확률적으로 “0%” 라는데 있다. 우주선에 탑재되는 시스템들이 이 모델을 쓰는 이유다.
