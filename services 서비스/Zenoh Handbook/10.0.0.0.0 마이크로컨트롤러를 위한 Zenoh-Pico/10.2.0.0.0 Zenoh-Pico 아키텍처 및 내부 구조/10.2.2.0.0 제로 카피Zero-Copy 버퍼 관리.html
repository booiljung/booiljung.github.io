10.2.2 제로 카피(Zero-Copy) 버퍼 관리

1024 바이트의 버퍼를 복사(memcpy)하는 것조차 8Mhz 짜리 초저전력 칩에서는 사치다. Zenoh-Pico 는 라우터로 데이터를 쏘거나 받을 때, 버퍼 자체를 복사하는 짓을 피하기 위해 “물리적 주소의 끈(Pointer)” 만을 던져대는 예술적인 경량화를 선보인다.

0.1 [Runbook] 주소 오프셋 조작 전술

보통 데이터를 쏠 때 우리는 z_publisher_put(pub, bytes, len) 을 생각한다.
하지만 당신이 만든 하드웨어(카메라 모듈 등)가 특정 메모리 주소(예: 0x40020000)에 직접 영상을 밀어 넣고 있다면 굳이 그 메모리를 tx_buffer 로 퍼나를 필요가 없다.

#include <zenoh-pico.h>

void zero_copy_tx(z_publisher_t* pub) {
    // 1. 카메라 DMA 가 직접 물고 있는 물리 메모리 번지수
    uint8_t* hardware_image_buffer = (uint8_t*)0x40020000;
    size_t image_size = 153600; // 320x240 RGB565 사이즈

    // 2. 단지 그 주소값만 이쁘게 포장해서 날린다!
    // (Pico 는 절대 내부적으로 malloc 을 해서 150KB 를 옮겨담지 않는다)
    z_publisher_put(
        pub, 
        hardware_image_buffer, 
        image_size, 
        NULL
    );
    
    // 이 패킷은 TCP/IP (또는 시리얼) 드라이버에 의해 
    // 저 메모리 주소 그대로에서 바이트 바이트 끊어져서 곧장 네트워크 망으로 빨려 들어간다.
}

만약 Pico 에서 Zero-Copy 를 적용하지 않고 습관적으로 버퍼 2개를 만들고 memcpy 를 남발하는 코딩을 한다면, 당신의 칩은 영원히 CPU 점유율 100% 에 갇혀 정작 로봇의 바퀴는 굴리지도 못할 것이다.
