10.2.4 상태 머신(State Machine)과 프로토콜 스택 경량화

패킷이 1바이트 1바이트씩 더럽게 찢어져서 날아오는 시리얼 통신(UART) 환경을 상상해 보라.
이 찢어진 데이터를 “하나의 멀쩡한 JSON“이나 “명령어 덩어리“로 조립하는 것은 악몽이다.

그래서 Zenoh-Pico는 내부에 파괴적인 디코더 상태 머신(Decoder State Machine) 을 갈아 넣었다.

0.1 [아키텍처 인스펙션] 바이트 집적(Accumulation) 사이클

Pico 의 코어는 들어오는 바이트(Byte) 조각들을 버퍼(rx_buffer)에 밀어 넣다가, 완벽한 “Zenoh 규격 패킷 1개” 가 완성되는 찰나(State Shift) 에만 당신의 콜백 함수를 우아하게 깨워준다.

1. 상태 A (대기 및 헤더 스캐닝)

  • 네트워크 케이블(또는 안테나)로부터 1바이트가 떨어지길 개처럼 기다린다.
  • 매직 코드(Zenoh 프로토콜의 시작을 알리는 특수 문자열)인지 확인한다.

2. 상태 B (페이로드 조립)

  • 헤더를 통해 이 메시지가 사이즈 50바이트 짜리 PUT 명령임을 획득했다.
  • 앞으로 튈 50바이트를 rx_buffer 의 특정 틈새로 착착 끼워 맞춘다(Pointer Offset ++).

3. 상태 C (격발 및 대리 처리)

  • 50개가 다 모였다!
  • 드디어 z_session_yield 가 막고 있던 루프를 풀고 당신이 선언해둔 Subscriber Callback 에 모아둔 버퍼의 포인터 주소만 넘겨버린다. (Zero Copy)
  • 처리가 끝나는 순간, 상태는 귀신같이 다시 A로 회귀한다.

이 피말리는 바이트 머신(Byte Machine) 메커니즘이, 단 3KB 의 빈 공간만으로도 초당 수백 개의 불규칙한 네트워크 패킷을 무결성 테스트(CRC 검증)까지 완벽하게 소화해 내는 Zenoh-Pico 의 진정한 흑마술이다.
