10.2 Zenoh-Pico 아키텍처 및 내부 구조

어떻게 8KB 짜리 램에서 거대한 멀티캐스트 라우팅 엔진이 돌아가는가? 비결은 “아무것도 하지 않는 것” 이다.

Zenoh-Pico 의 소스코드를 열어보면, 화려한 스레드 스케줄러(Thread Scheduler)도, 데이터 베이스도, 캐시(Cache) 시스템도 없다. 오직 단일 스레드(Single Thread)에서 돌아가는 원시적인 상태 머신(State Machine)과, 바이트를 오른쪽 왼쪽으로 옮기는 연속된 Memcpy 의 향연만이 존재한다.

이 챕터는 극한의 환경에서 동작하는 Zenoh-Pico 내부 톱니바퀴의 구동 원리를 낱낱이 분해하여 보여준다. 이 원리를 제대로 꿰뚫지 못한 채 API만 대충 호출한다면, 당신의 로봇은 통신을 시작하자마자 커널 크래시(Kernel Crash)를 내며 멈춰 설 것이다.

1. 정적 메모리 모델과 동적 할당(Malloc) 최소화 기법

zenoh-c 였다면 z_publisher_declare() 를 부르는 순간 백그라운드 환경에서 수 킬로바이트의 힙(Heap) 메모리가 malloc 을 통해 동떨어지며 할당되었을 것이다.
Zenoh-Pico 는 이것을 철저히 금지한다.

1.0.1 [Runbook] 정적(Static) 구조체 선언 전술

Pico 세계에서는 당신이 직접 램(RAM) 지도를 그려서 Zenoh 에게 “이 공간을 써라” 고 갖다 바쳐야 한다. 이것이 유일한 규칙이다.

#include <zenoh-pico.h>
#include <stdio.h>

// 1. [핵심] 힙(Heap)이 아니라 데이터 포맷(Data Segment)이나 BSS 에 박히는 전역 변수들
// 운영체제가 프로그램 시작 전에 이미 메모리 한 켠에 확정해 놓는(Static Allocation) 공간이다.
uint8_t rx_buffer[1024]; 
uint8_t tx_buffer[1024];

z_session_t session; 
z_publisher_t publisher;

void boot_pico() {
    // 2. Pico 용 설정 깎기
    z_config_t config = z_config_default();
    
    // 3. 내가 만든 메모리 창고(전역 변수)의 주소값을 Pico 에게 먹인다!
    // "이 버퍼 안에서만 물놀이를 하렴. 절대 malloc 을 쓰지마!"
    zp_config_insert_json(&config, ZP_RX_BUF_ADDR_KEY, (char*)&rx_buffer);
    zp_config_insert_json(&config, ZP_RX_BUF_SIZE_KEY, "1024");
    
    zp_config_insert_json(&config, ZP_TX_BUF_ADDR_KEY, (char*)&tx_buffer);
    zp_config_insert_json(&config, ZP_TX_BUF_SIZE_KEY, "1024");

    // 4. 세션 기동 (이 안에서 malloc 은 1도 호출되지 않는다)
    z_open(&session, &config, NULL);
}

이 패턴의 파괴적인 장점은, 아무리 통신을 수십만 번을 때려도 Fragmentation(단편화) 에 의한 메모리 고갈 사태가 확률적으로 “0%” 라는데 있다. 우주선에 탑재되는 시스템들이 이 모델을 쓰는 이유다.

2. 제로 카피(Zero-Copy) 버퍼 관리

1024 바이트의 버퍼를 복사(memcpy)하는 것조차 8Mhz 짜리 초저전력 칩에서는 사치다. Zenoh-Pico 는 라우터로 데이터를 쏘거나 받을 때, 버퍼 자체를 복사하는 짓을 피하기 위해 “물리적 주소의 끈(Pointer)” 만을 던져대는 예술적인 경량화를 선보인다.

2.0.1 [Runbook] 주소 오프셋 조작 전술

보통 데이터를 쏠 때 우리는 z_publisher_put(pub, bytes, len) 을 생각한다.
하지만 당신이 만든 하드웨어(카메라 모듈 등)가 특정 메모리 주소(예: 0x40020000)에 직접 영상을 밀어 넣고 있다면 굳이 그 메모리를 tx_buffer 로 퍼나를 필요가 없다.

#include <zenoh-pico.h>

void zero_copy_tx(z_publisher_t* pub) {
    // 1. 카메라 DMA 가 직접 물고 있는 물리 메모리 번지수
    uint8_t* hardware_image_buffer = (uint8_t*)0x40020000;
    size_t image_size = 153600; // 320x240 RGB565 사이즈

    // 2. 단지 그 주소값만 이쁘게 포장해서 날린다!
    // (Pico 는 절대 내부적으로 malloc 을 해서 150KB 를 옮겨담지 않는다)
    z_publisher_put(
        pub, 
        hardware_image_buffer, 
        image_size, 
        NULL
    );
    
    // 이 패킷은 TCP/IP (또는 시리얼) 드라이버에 의해 
    // 저 메모리 주소 그대로에서 바이트 바이트 끊어져서 곧장 네트워크 망으로 빨려 들어간다.
}

만약 Pico 에서 Zero-Copy 를 적용하지 않고 습관적으로 버퍼 2개를 만들고 memcpy 를 남발하는 코딩을 한다면, 당신의 칩은 영원히 CPU 점유율 100% 에 갇혀 정작 로봇의 바퀴는 굴리지도 못할 것이다.

3. 비동기 이벤트 기반 루프 및 I/O 모델

Zenoh-Pico 안에는 스레드(Thread)를 분파해서 영원히 네트워크를 돌려주는 고마운 데몬 프로세스가 없다.
데이터가 왔는지 안 왔는지, 주기적으로 칩의 메인 루프 안에 들어가서 “문 좀 열어달라고(Poll)” 노크를 해 줘야 한다.

3.0.1 [Runbook] 협력적(Cooperative) 스웦(Sweep) 폴링 전술

당신의 기기 펌웨어 최하단 while(1) 루프(또는 FreeRTOS 의 메인 태스크) 안에 반드시 Zenoh-Pico 에게 “숨 쉴 기회” 를 지속적으로 공급해줘야 한다.

#include <zenoh-pico.h>

void my_robot_firmware_main() {
    // ... [1] 하드웨어 초기화 (GPIO, UART 설정 등)
    // ... [2] Zenoh-Pico 세션 개화 (z_open)

    while (1) {
        // [3] 로봇 펌웨어 고유의 일 (센서 읽기, 모터 PWM 제어 등)
        read_sensor();
        drive_motor();

        // [4] 핵심! Zenoh 엔진 숨 쉬게 만들기 (10ms 대기 한도)
        // 이 함수를 호출해야만 비로소 네트워크 칩에 쌓인 데이터가 
        // 내 Subscriber 콜백 함수들로 라우팅되어 불타게 된다!
        z_session_yield(&session, 10); 
        
        // 주의: 이 z_session_yield() 가 불리는 주기가 불규칙하거나 너무 느리면,
        // (예: drive_motor 에서 3초 지연됨)
        // TCP 버퍼가 가득 차서 연결된 Zenoh Router 가 당신을 "죽은 세션" 으로 간주하고
        // 망에서 강제 퇴출(Drop)시켜버린다.
    }
}

MCU 네트워크 프로그램이 뒤지는 원인의 80%는 이 yield 와 하드웨어 I/O 로직 사이의 “시간 분배 실패” 에서 비롯된다. 메인 함수 안에서 delay() 나 sleep() 을 남발하는 주니어 코더는 결코 이 이기종 멀티캐스트 네트워크를 통제할 수 없다.

4. 상태 머신(State Machine)과 프로토콜 스택 경량화

패킷이 1바이트 1바이트씩 더럽게 찢어져서 날아오는 시리얼 통신(UART) 환경을 상상해 보라.
이 찢어진 데이터를 “하나의 멀쩡한 JSON“이나 “명령어 덩어리“로 조립하는 것은 악몽이다.

그래서 Zenoh-Pico는 내부에 파괴적인 디코더 상태 머신(Decoder State Machine) 을 갈아 넣었다.

4.0.1 [아키텍처 인스펙션] 바이트 집적(Accumulation) 사이클

Pico 의 코어는 들어오는 바이트(Byte) 조각들을 버퍼(rx_buffer)에 밀어 넣다가, 완벽한 “Zenoh 규격 패킷 1개” 가 완성되는 찰나(State Shift) 에만 당신의 콜백 함수를 우아하게 깨워준다.

1. 상태 A (대기 및 헤더 스캐닝)

  • 네트워크 케이블(또는 안테나)로부터 1바이트가 떨어지길 개처럼 기다린다.
  • 매직 코드(Zenoh 프로토콜의 시작을 알리는 특수 문자열)인지 확인한다.

2. 상태 B (페이로드 조립)

  • 헤더를 통해 이 메시지가 사이즈 50바이트 짜리 PUT 명령임을 획득했다.
  • 앞으로 튈 50바이트를 rx_buffer 의 특정 틈새로 착착 끼워 맞춘다(Pointer Offset ++).

3. 상태 C (격발 및 대리 처리)

  • 50개가 다 모였다!
  • 드디어 z_session_yield 가 막고 있던 루프를 풀고 당신이 선언해둔 Subscriber Callback 에 모아둔 버퍼의 포인터 주소만 넘겨버린다. (Zero Copy)
  • 처리가 끝나는 순간, 상태는 귀신같이 다시 A로 회귀한다.

이 피말리는 바이트 머신(Byte Machine) 메커니즘이, 단 3KB 의 빈 공간만으로도 초당 수백 개의 불규칙한 네트워크 패킷을 무결성 테스트(CRC 검증)까지 완벽하게 소화해 내는 Zenoh-Pico 의 진정한 흑마술이다.
