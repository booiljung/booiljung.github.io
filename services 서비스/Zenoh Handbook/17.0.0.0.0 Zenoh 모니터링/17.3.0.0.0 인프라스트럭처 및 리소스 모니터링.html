17.3 인프라스트럭처 및 리소스 모니터링

Zenoh 가 아무리 날고 기어도, 그 발밑을 받치는 리눅스(Linux) 서버의 CPU 가 타버리거나 하드디스크가 꽉 찼다면 모니터링 데몬조차 비명을 지르지 못하고 즉사한다.

이 장에서는 프로토콜을 보기 전, “이 서버(Hardware) 자체가 과연 살아서 숨 쉬고 있는가?” 에 대한 노드 중심(Node-centric) 적 기초 체력 모니터링 구성 런북을 완성한다.

1. Zenoh 노드(라우터, 피어, 클라이언트)의 CPU 및 메모리 프로파일링

데몬 프로세스(zenohd) 가 서버의 피(RAM) 를 얼마나 빨아먹고 있는지 감시하라.

1.0.1 [Runbook] OS 레벨 시스템 매트릭 추출 전술

1. node_exporter 탑재
서버(에지/클라우드) 의 기본 상태를 Prometheus (17.4장) 로 쏘기 위해 node_exporter 를 데몬으로 올린다.

## 백그라운드에서 CPU, Memory 수치를 수집하여 9100 포트로 열어두는 표준 툴
./node_exporter &

2. 프로세스(PID) 타겟팅
전체 서버의 CPU 도 중요하지만, 우리는 zenohd 가 범인인지 아닌지 명확히 알고 싶다. process-exporter 를 추가로 연동하여 zenohd 이름을 가진 프로세스만의 CPU 커널(Kernel) 모드와 유저(User) 모드 점유율을 찢어서 발라봐야 한다. 만약 CPU 커널 타임 리소스가 유저 타임 리소스를 압도한다면, 그것은 I/O(소켓 버퍼 튜닝) 나 폴링(Polling) 에서 병목 코드가 돌고 있다는 물리적인 적신호다.

2. 네트워크 대역폭 사용량 및 패킷 드롭률 측정

단순히 “인터넷이 켜져 있다” 로는 부족하다. 파이프가 얼마나 꽉 찼고, 몇 개의 자재가 바닥으로 떨어졌는지(Drop) 봐야 한다.

2.0.1 [인스펙션] 파이프라인 누수(Leak) 탐지 전술

1. 인터페이스(NIC) 단위 대역폭 모니터링
ifstat 이나 sar -n DEV 1 같은 OS 명령어를 통해, 1초마다 랜 카드(eth0) 에 꽂히는 RX/TX 단위 바이트를 수집하라. 이 지표가 인프라가 설계한 Max 대역폭 (예: 1Gbps) 의 80% 를 넘어가는 순간, 지연 시간(Latency) 은 기하급수적으로 폭발하게 되어있다.

2. 커널 스택 패킷 드롭 감시
가장 무서운 건 애플리케이션(Zenoh) 단까지 데이터가 오지도 못하고 리눅스 커널이 데이터를 찢어버렸을 때다.

## 리눅스 커널이 큐 오버플로우나 에러로 버린 패킷의 숫자
netstat -s | grep "packet receive errors"

여기가 0 이 아니면 당신의 서버가 DDoS 를 맞고 있거나 서버의 뇌(버퍼 사이즈) 가 너무 작은 것이다. 16.2장(OS 레벨 튜닝) 으로 다시 돌아가라.

3. 파일 디스크립터(File Descriptor) 및 소켓 상태 모니터링

“갑자기 새로운 로봇이 라우터에 연결을 못 합니다! 라우터 로그엔 Too many open files 라고 뜹니다!”
리눅스에서 소켓(네트워크 연결) 은 곧 파일(File) 이다.

3.0.1 [Runbook] 연결 고갈(Port Exhaustion) 방어 런북

최대 연결이 1024개인데 로봇이 2000대가 접속을 시도하면 포트 고갈(Exhaustion) 로 서버 셧다운이 일어난다.

1. FD 쿼터 한계 모니터링
라우터 데몬이 열고 있는 파일(소켓) 의 개수를 실시간으로 추적하라.

## zenohd 프로세스(PID) 가 열고 있는 소켓의 개수
ls -l /proc/<PID>/fd | wc -l

이 숫자가 ulimit -n (운영체제가 허락한 최대 한도, 보통 1024) 의 80% 를 넘어가면 서버 마비 직전의 응급 상황(Critical) 알람을 슬랙(Slack) 에 쏴야 한다.

2. TIME_WAIT 상태의 좀비 색출
연결이 끊어졌는데 깔끔하게 안 죽고 대기 중인 소켓(TIME_WAIT) 이 수천 개 쌓여있는지 모니터링해야 한다.
ss -ant | grep TIME_WAIT | wc -l
이 수치가 비정상적으로 높다면, 당신이 구축한 Zenoh 세션 타임아웃 주기가 잘못되었거나 개발자가 의도치 않게 커넥션을 무한 생성하고 있다는 범죄의 강력한 증거다.

4. 컨테이너 및 Kubernetes 환경에서의 리소스 추적

Zenoh 를 도커(Docker) 나 쿠버네티스(K8s) Pod 으로 감싸서 배포했다면, 기존 리눅스 커널 모니터링은 장님이 된다. 컨테이너는 격리된 환상 속 공간(cgroups) 에 살고 있기 때문이다.

4.0.1 [인스펙션] Cgroups 포위(Siege) 전술

Kubernetes 환경에서의 관제탑 모범 설계다.

1. cAdvisor 탑재
호스트 서버에 node_exporter 대신 (또는 함께) cAdvisor(Container Advisor) 를 띄워라.
이 놈은 도커 컨테이너들의 격리된 벽(namespace) 을 뚫고 들어가, “현재 K8s 1번 Pod(Zenoh 라우터) 가 RAM 한계치(Limit) 인 2GB 중 1.9GB 를 썼음” 이라는 초정밀 지표를 뽑아낸다.

2. OOMKilled 추적
K8s 에서는 프로세스가 죽으면 자기가 알아서 다시 살려낸다(Restart). 개발자는 그것도 모르고 “잘 도네” 라고 착각한다.
kube-state-metrics 를 달아서, 라우터 Pod 의 restarts 횟수와 OOMKilled (메모리 초과로 K8s 가 강제 사살함) 이벤트를 반드시 덤프 해야 한다.
라우터가 재시작(Restart) 한다는 것은 그동안 날아가던 모든 패킷(Data Payload) 이 메모리 공중분해 되었다는 재앙의 묵시록이기 때문이다.
