17.9 대규모 에지-클라우드 환경에서의 모니터링 전략

로봇이 10대일 때는 프로메테우스 하나로 관제탑이 돌아갔다. 하지만 그 숫자가 10,000대로 늘어나면 어떤 일이 벌어질까? 관제탑의 모니터링 서버 자신이 수집 트래픽의 스나미를 이기지 못하고 가장 먼저 붕괴(OOM) 하여 시스템全体の 눈먼 장로(Blind Spot) 가 되어버린다.

이 장에서는 수만 대의 에지 단말과 거친 야전 네트워크 단절 환경에서도 모니터링 지표만큼은 절대 유실되지 않고 안전하게 클라우드로 합쳐지는 거대한 관제 연합(Observability Federation) 과 고가용성(HA) 아키텍처 구축 런북을 완성한다.

1. 수천 개 이상의 에지 디바이스 모니터링을 위한 데이터 페더레이션(Federation)

클라우드의 메인 프로메테우스가 전 세계 10,000 대의 로봇 IP 를 알고 일일이 찾아가서 Scrape(당겨오기) 하는 건 물리적으로 불가능하다.

1.0.1 [인스펙션] 제국주의 페더레이션 전술

가장 말단(Edge) 부터 중간 지휘소(Fog) 를 거쳐 황제(Cloud) 에게 도달하는 피라미드식 지표 압축 전술이다.

1. 로컬 에지 스크래핑
로봇 안에는 작은 프로메테우스(혹은 OTel Collector) 가 돌아가고 있다. 이 녀석은 자기 기기 내부의 Docker 와 Zenoh 세션 상태만 초 단위로 수집하여 자기가 임시 보관한다.

2. 중간 기지국의 축약 보고 (Thanos / Prometheus Federation)
공장마다 있는 Fog Router 옆에 공장용 프로메테우스를 하나 둔다. 이 녀석이 공장 안의 로봇 100 대의 데이터를 긁어모아 1분 단위 평균값(Average) 과 합계(Sum) 로 요약(Recording Rule) 한다.
최종적으로 클라우드의 메인 프로메테우스는 개별 로봇을 안 쳐다보고 “공장별 프로메테우스” 들만 긁어모은다. 이렇게 계층을 타고 올라가며 데이터 해상도(Resolution) 를 고의로 떨어뜨리는 것이 관제탑 폭파를 막는 유일한 페더레이션 방법론이다.

2. 네트워크 단절(Network Partition) 상황에서의 모니터링 데이터 보존 및 동기화

로봇이 지하 주차장에 들어갔다. 10분 동안 5G 망이 끊어졌다.
일반적인 관제탑 도구들은 “로봇이 보이지 않음(Down)” 이라고만 표기하고 그 시간에 무슨 일이 있었는지 데이터를 그냥 허공에 날려버린다(Loss).

2.0.1 [Runbook] 데이터 블랙박스(Store and Forward) 전술

사각지대(Offline) 에서의 데이터를 살려내는 에지 옵저버빌리티 기법이다.

1. Remote Write 기반의 밀어내기 (Push)
표준 프로메테우스 규격인 Pull(가져오기) 은 망이 끊기면 무용지물이다.
로봇(Edge) 에 설치된 수집기를 Remote Write 모드로 강제 설정하라.
망이 끊긴 동안 로봇은 수집된 CPU 사용량과 Zenoh 메트릭을 자기 디스크 백업 폴더(WAL - Write Ahead Log) 에 미친 듯이 박아둔다.

2. 재연결 시 역류(Backfill) 전송
10분 뒤, 로봇이 지상으로 올라와 5G 에 재연결된다.
이 순간 로봇의 OTel Collector 나 vmagent (VictoriaMetrics Agent) 가 디스크에 쌓아둔 “끊긴 10분 치의 모니터링 파일” 을 통째로 클라우드로 발사한다. 11분 뒤 관제탑의 그라파나(Grafana) 차트는, 끊겼던 10분 구간의 하얀 이빨(공백) 을 과거 시점에서 완벽하게 메워버리는 사후 복원(Backfilling) 을 달성하게 된다.

3. 고가용성(HA) 모니터링 클러스터 아키텍처 구성

관제탑(Cloud Prometheus/Grafana) 이 죽으면 전 세계 수만 대의 로봇을 눈 뜬 장님 상태로 놔두게 된다. 모니터링 시스템 스스로가 결코 죽지 않는 좀비(HA) 구성이 되어야 한다.

3.0.1 [인스펙션] 쌍둥이 관제탑(Twin Tower) 미러링

모니터링 서버는 절대 1대만 운영하지 않는다.

1. 프로메테우스 듀얼(Dual) 인스턴스
클라우드 센터에 똑같이 생긴 프로메테우스 서버 A 와 B 를 만든다.
둘 다 동시에 동일한 타겟(라우터들) 을 긁어온다. 데이터베이스 용량은 2배로 들지만, A 서버가 전기 공급이 끊겨 타버리더라도 B 서버의 디스크에는 똑같은 메트릭이 생체 복사된 채로 회전하고 있다.

2. Thanos (타노스) 를 활용한 글로벌 통합 스토어
A서버와 B서버는 서로 쌍둥이지만 각각 독립되어 있다. 관리자가 Grafana 를 켤 때 A 와 B 에 든 데이터를 알아서 섞어서 중복을 없애고(Deduplication) 1개의 예쁜 차트로 보여주는 마법의 레이어가 필요하다.
Thanos Querier 컴포넌트를 이 두 프로메테우스 위에 군림시키면, 하위 노드 10개가 터져 나가도 관제탑 화면에는 1픽셀의 흔들림도 없는 무적의 로보틱스 관제 인프라스트럭처가 완성된다.
