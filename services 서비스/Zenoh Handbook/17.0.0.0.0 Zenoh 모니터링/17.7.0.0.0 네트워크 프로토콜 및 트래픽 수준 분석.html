17.7 네트워크 프로토콜 및 트래픽 수준 분석

모니터링 대시보드(Prometheus, Grafana) 와 분산 트레이서(Jaeger) 는 의사가 병의 증상과 위치를 진단하는 청진기다. 그러나 때로는 환자의 배를 가르고 직접 장기를 꺼내봐야만 하는 처참한 사고가 터진다.

이 장은 네트워크 케이블을 자르고 들어가, 1과 0으로 이루어진 생(Raw) 전기 신호를 캡처(PCAP) 하여, Zenoh 프로토콜이 약속대로 패킷을 조립하고 있는지, 혼잡 제어가 정상적으로 브레이크를 밟고 있는지를 1바이트 단위로 해부하는 “네트워크 포렌식(Network Forensics)” 런북이다.

1. Wireshark를 이용한 Zenoh 프로토콜 패킷 캡처 (PCAP)

공장에서 “로봇이 미쳤어요, 데이터를 가끔 빼먹습니다!” 라고 울부짖을 때 애플리케이션 로그에는 정상이 찍혀있다면? 답은 공기 중으로 쏘아 올린 전파(Wire) 에 있다.

1.0.1 [인스펙션] 덤프 오디오(Dump Audio) 채취 전술

에지 컴퓨터(로봇) 에 Wireshark UI 를 깔 수는 없다. CLI 환경에서 소리소문없이 패킷을 훔쳐야 한다.

1. 대규모 덤프(PCAP) 커맨드
로봇의 터미널(SSH) 에 들어가서 tcpdump 로 7447 포트의 통신을 낚아챈다.

## eth0 인터페이스에서 라우터(7447 포트) 로 오가는 패킷 1만 개를 잡아 파일로 저장
sudo tcpdump -i eth0 port 7447 -w robot_zenoh_drop_investigation.pcap -c 10000

2. PCAP 파일의 사후 부검
저장된 .pcap 파일을 개발자의 노트북으로 복사(scp) 해 온 뒤 Wireshark 로 연다.
이 단계에서는 단순히 “패킷이 오가느냐” 와 “TCP 레벨의 재전송(Retransmission) 이 있느냐” 만을 확인한다. 만약 검정색과 빨간색으로 범벅된 TCP Retrasmission 바다가 펼쳐진다면, Zenoh 코드가 아니라 로봇의 와이파이(Wi-Fi) 안테나가 고장난 것이다. (하드웨어 교체).

2. Zenoh Wire Protocol 구조와 디섹터(Dissector) 활용법

17.7.1 에서 잡은 PCAP 파일은 Data 부분에 그냥 Payload: 1a 2f 4c... 같은 알아볼 수 없는 쓰레기값만 보여준다. Wireshark 는 HTTP 같은 건 잘 읽지만, 커스텀 규격인 Zenoh 패킷은 해독하지 못하기 때문이다.

2.0.1 [Runbook] 프로토콜 해독기(Dissector) 장착 전술

1. LUA 디섹터 주입
Zenoh 공식 GitHub 에서 제공하는 zenoh.lua Wireshark 플러그인 파일을 다운로드하여 Wireshark 의 플러그인 폴더에 심는다. (Wireshark 재시작 필요).

2. 1바이트 뼈대(Wire Protocol) 해부
해독기를 키포 Wireshark 에서 아무 패킷이나 하나 눌러보면, Payload 항목이 마법처럼 펼쳐지며 뼈대가 드러난다.

  • Message Type: 이게 Put (데이터 쏘기) 인지 Declare (나 구독할래) 인지 식별된다.
  • Key Expression: 누가 어떤 토픽(예: robot/video) 으로 쏘았는지 평문으로 보인다.
  • Attachment/Timestamp: 이 데이터가 정확히 언제 발생했는지 증거가 나온다.

이 해부학을 통해 특정 토픽(robot/useless/log) 이 1초에 100만 번 보내지면서 전체 7447 대역폭의 90%를 갉아먹는 트래픽의 주범(Hog) 임을 정확하게 색출해 낼 수 있다.

3. TLS 및 보안 설정 구간에서의 패킷 모니터링 한계와 우회 전략

문제는, 앞서 15장 규정대로 이 통신에 TLS(암호화) 를 발라버렸다면?
Wireshark 의 디섹터(lua) 따위는 통하지 않는다. 눈앞에 보이는 건 완전한 노이즈 덩어리(Encrypted Application Data) 뿐이다.

3.0.1 [인스펙션] 크립토 락픽(Lock-pick) 전술

개발 환경에 한해, 이 암호를 까기 위해 마스터 키를 잠시 Wireshark 에 넘겨주는 우회로를 뚫는다.

1. 세션 키 덤프(SSLKEYLOGFILE) 유출
클라이언트 코드나 라우터가 TLS 핸드셰이크를 할 때 생성하는 “세션 비밀키” 를 텍스트 파일로 몰래 빼돌리도록 OS 환경변수를 설정한다.

## Zenoh 구동 시, TLS 암호 해독용 마스터 키를 저 파일에 계속 기록해라!
export SSLKEYLOGFILE=/tmp/zenoh_keys.log
./zenohd

2. Wireshark 복호화 엔진 연동
나중에 PCAP 파일을 Wireshark 에서 열 때, Edit -> Preferences -> Protocols -> TLS 메뉴에서 (Pre)-Master-Secret log filename 항목에 아까 저장한 /tmp/zenoh_keys.log 파일을 물려준다.
적용 즉시 암호화되어 있던 패킷들의 봉인이 풀리면서, 평문이었을 때처럼 Put 명령어와 토픽 문자열이 화면에 쏟아지는 기적을 볼 수 있다. (프로덕션 환경에서는 이 행위 자체가 치명적 보안 위반이므로 즉각 삭제해야 한다).

4. 대규모 트래픽 발생 시 혼잡 제어(Congestion Control) 모니터링

16장에서 배운 Drop-Oldest 큐 삭제 정책이나 QUIC BBR 혼잡 제어가 과연 실전에서 진짜 동작하고 있는지 눈으로 확인해야 한다. 당신이 코딩한 설정이 뇌피셜이 아니라는 증명이 필요하다.

4.0.1 [Runbook] 쓰나미(Tsunami) 방어 스캔 레이더

1. Zenoh 내부 수위표(Admin Space) 스크래핑
가장 우아한 방법. 네트워크 캡처 없이 라우터에게 직접 묻는다.
z_get "@/router/local/metrics/dropped" 를 호출하여 숫자가 벼락같이 올라가는 시점을 시계열 DB(Prometheus) 에 박아둔다. 이 그래프가 치솟는 순간, 로봇의 영상 스트림이 버벅거리기 시작하는 시간대와 1초도 오차 없이 일치한다면 범인은 버퍼 부족(Congestion) 이다.

2. Wireshark TCP 윈도우 스로틀링(Throttling) 관측
더 깊은 레벨의 증명이다. PCAP 덤프를 떠서 TCP Window Size 가 0 으로 곤두박질치는(TCP Zero Window) 핑퐁 현상이 있는지 확인하라.
수신 라우터가 “제발 그만 보내, 나 지금 버퍼 터질 거 같아!” 라고 로봇에게 구조 요청(혼잡 제어) 을 정상적으로 보내고 있다는 증거다. 만약 Window Size 가 여유 넘치는데도 Zenoh 가 패킷을 버리고(Drop) 있다면, OS 커널은 널널한데 당신이 로봇 앱을 잘못 짜서 콜백(Callback) 연산이 멈춰 선 애플리케이션 데드락 사태다.
