17.1 모니터링 개요와 필요성

라우터 하나가 동작을 멈췄다. 로봇 10대의 통신이 마비되었는데, 원인은 로봇의 버그인가, 5G 기지국의 문제인가, 아니면 클라우드 라우터 메모리가 터진 것인가?

모니터링이 없다면 이 참사의 책임을 묻고 원인을 찾는 데 1주일이 걸린다. 모니터링이 있다면 1초 만에 대시보드 그래프가 파인(Drop) 구간을 찾을 수 있다. 시스템을 “관찰 가능한 상태(Observable)” 로 만드는 것은 선택이 아니라, 당신이 집에 퇴근할 수 있느냐를 결정하는 아키텍트의 생존 런북이다.

1. 분산 데이터 네트워크에서 모니터링의 역할

HTTP 웹 서버(단일 경로) 와 달리, Zenoh 는 노드들이 다대다(N:M), 메시, 트리 구조로 미친 듯이 복잡하게 얽혀서 대화한다. 특정 포인트 관제만으로는 파도(Traffic) 의 근원지를 절대 찾을 수 없다.

1.0.1 [인스펙션] 전체주의 가시성(Holistic Observability) 전술

에지, 중간 라우터, 클라우드 백엔드 등 모든 노드는 각자의 혈압(지표) 을 하나의 컨트롤 타워로 일제히 전송해야 한다.

1. 침묵의 살인마 식별
데이터가 안 오면 에러가 나는 게 아니다. 그저 끊길(Drop) 뿐이다.
모니터링이 없다면 당신의 시스템은 조용히 아무 일도 없었던 것처럼 데이터를 흘리고 있을 것이다. 오직 초당 패킷 건수(msg/s) 그래프만이, 어제와 비교하여 30%의 구멍이 발생했다는 사실을 폭로할 수 있다.

2. OOM (Out of Memory) 스나이퍼 회피
메모리가 서서히 새고 있다면(Leak), 서버가 죽기 직전에 “디스크 정리“나 “서비스 재시작“을 걸 골든 타임이 필요하다. 모니터링 알람(Alert) 이야말로 수작업 재시작을 자동화(Auto-healing) 로 연결할 수 있는 유일한 방아쇠(Trigger) 다.

2. Zenoh 시스템 모니터링의 핵심 지표 (KPI)

수천 개의 지표 중에 단 4가지만 살아서 퇴근길에 당신의 눈을 따라와야 한다.

2.0.1 [Runbook] 골든 시그널 4대 천왕 추출

1. 라우팅 테이블(Routing Table) 엔트리 수

  • Zenoh 는 구독(Sub) 이 생길 때마다 지도를 그린다.
  • 이 지도가 갑자기 10만 개, 100만 개로 튀어오르는 순간(Spike), 시스템은 룩업(Look-up) 딜레이에 빠져 죽는다. 와일드카드 구독의 남발을 감시하는 핵심이다.

2. Dropped Messages (학살당한 패킷)

  • 가장 치명적인 지표. 라우터가 자기가 보낼 능력이 없어서 바닥에 찢어버린 패킷의 숫자다.
  • 이 숫자가 0 이 아니라면, 당신이 설계한 파이프 두께(Bandwidth) 나 큐 리미트(Queue Limit) 가 명백히 실패했다는 소리다. (16.5장 혼잡 제어 튜닝 필요).

3. In/Out Throughput (대역폭 절대량)

  • “이 라우터가 지금 초당 몇 메가바이트(MB/s) 를 들이마시고, 몇 번의 세션(Session) 을 안고 있는가?”
  • 인프라의 한계 용량(Capacity) 을 계산하여, 라우터를 2대로 스케일 아웃(Scale-out) 할 시점을 알려주는 지표다.

4. 락 경합(Lock Contention) / CPU 스파이크

  • OS 레벨의 지표다. 트래픽은 저조한데 CPU가 100% 라면, 이것은 무조건 암호화(TLS) 버그나 코딩 실수(Lock Deadlock)다.

3. 가시성(Observability)의 3요소: 메트릭, 로그, 트레이스

장님 코끼리 만지기를 끝내기 위해 센서 3개를 병렬로 장착해야 한다. 성능 최적화 진단을 위해서는 모니터링의 3대 인프라를 동시에 가동해야 입체적인 범인 색출이 가능하다.

3.0.1 [인스펙션] 삼위일체 가시성 전술

1. 메트릭 (Metrics): 시스템의 심장 박동수

  • 형태: cpu_usage: 45.2%, msgs_dropped_total: 1204
  • 역할: “지금 문제가 생겼나?(Yes/No)” 와 “문제가 얼마나 심각한가?(Size)” 를 1초 단위로 알려주는 거대한 대시보드 역할이다 (Prometheus / Grafana).

2. 로그 (Logs): 범인의 육성 녹음

  • 형태: [WARN] 2026-10-15 14:02:00 "Too many sessions from IP 192.168.1.5"
  • 역할: 메트릭이 “문제 생김!” 하고 빨간불을 울렸을 때, 담당자가 스크롤을 올려 “대체 어떤 놈이 뭘 하다가 죽은 건지” 서술형 스토리텔링으로 읽게 만들어주는 기록이다 (ELK Stack).

3. 트레이스 (Traces): 탄환의 궤적 추적

  • 형태: 드론 $\rightarrow$ 5G망 (12ms) $\rightarrow$ 라우터A (3ms) $\rightarrow$ DB 쓰기 (45ms)
  • 역할: “왜 느린 것입니까?” 라는 상사의 질문에 명확한 그래프 폭포수(Waterfall) 를 보여준다. 병목이 네트워크 라인인지 시스템 저장부인지 노드 간의 릴레이 구간을 초 단위로 끊어서 보여주는 엑스레이(X-Ray) 다. (OpenTelemetry 연동 필요).
