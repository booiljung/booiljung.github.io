17.2.3 zenoh-cli를 이용한 네트워크 토폴로지 점검

로봇 A가 보낸 데이터가 라우터 B를 거쳐 관제탑 C 로 들어왔다. 그런데 이 데이터가 D 로 안 간다.
선이 어디서 끊겼는지 알려면 “현재 이 네트워크의 별자리(Topology)” 가 어떻게 그려져 있는지 투시경이 필요하다.

0.1 [인스펙션] 노드 연결망 엑스레이 전술

1. 이웃(Neighbors) 탐색기 가동
마우스로 라우팅 지도를 클릭하는 상상은 버려라.
에지 로봇에 SSH 로 접속한 뒤, 터미널에서 당장 자신이 연결된 홉(Hop) 의 타겟을 찢어 발겨라.

# 이 로봇이 바라보고 있는 모든 라우터 형님들의 IP와 세션 ID를 출력하라
z_get "@/router/local/sessions/link/**"

2. 통신 단절(Split-Brain) 증거 수집
만약 공장 1구역의 라우터들끼리 뭉친 망과, 공장 2구역의 라우터들끼리 뭉친 망이 중간 다리가 끊기면, 각 구역은 자기들만 정상인 줄 안다.
이럴 때는 관제탑(Core) 데몬의 세션을 나열했을 때 공장 1구역을 대변하는 Peer_ID 가 sessions 리스트에 존재하는지만 문자열 매칭(grep) 해보면 1초 만에 브릿지 끊김 사태를 단정 지을 수 있다.
