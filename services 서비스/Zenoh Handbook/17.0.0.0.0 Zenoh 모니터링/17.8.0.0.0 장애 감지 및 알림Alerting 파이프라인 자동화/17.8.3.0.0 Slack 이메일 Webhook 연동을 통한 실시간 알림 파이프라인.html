17.8.3 Slack, 이메일, Webhook 연동을 통한 실시간 알림 파이프라인

사냥개(Alertmanager) 가 범인을 물었으면, 이제 주인의 스마트폰이 진동할 차례다.

0.1 [인스펙션] 에스컬레이션(Escalation) 전술

이벤트의 등급(Severity) 에 따라 깨워야 할 직급의 타겟이 달라야 한다.

1. Alertmanager Routing 설정

# alertmanager.yml
route:
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 10s
  receiver: 'slack-warning'
  routes:
    # Disaster 급 (라우터 죽음) 은 최우선 슬랙 채널과 SMS로 동시에 쏜다.
    - match:
        severity: disaster
      receiver: 'sms-pagerduty'
    # Warning 급 (메모리 80% 참) 은 인프라팀 슬랙에만 조용히 알린다.
    - match:
        severity: warning
      receiver: 'slack-warning'

2. Webhook 을 이용한 자가 치유(Auto-Healing)
알람을 단순히 사람에게만 보내면 하수다. 최상위 아키텍트는 알람을 시스템으로 우회시킨다(Webhook 연동).
만약 HighMemoryUsage 알람이 터지면, 슬랙으로 메시지를 보냄과 동시에 쿠버네티스(K8s) API 로 Webhook 트리거를 쏴서 “야, OOM 터지기 전에 라우터 Pod 을 하나 더 복제해서 달아라(Scale-out)” 고 기계 대 기계통신(M2M) 보상을 명시하는 것이 궁극의 SRE 전략이다.
