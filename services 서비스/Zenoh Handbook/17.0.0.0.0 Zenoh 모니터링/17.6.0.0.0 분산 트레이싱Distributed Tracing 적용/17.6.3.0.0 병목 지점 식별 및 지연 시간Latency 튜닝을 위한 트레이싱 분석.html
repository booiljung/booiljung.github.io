17.6.3 병목 지점 식별 및 지연 시간(Latency) 튜닝을 위한 트레이싱 분석

Jaeger 그래프에서 빨간불이 들어온 Span 을 찾았다. 이제 그 구간을 현미경으로 파헤쳐서 튜닝(16장 적용) 으로 연결해야 한다.

0.1 [인스펙션] 지연시간 핀셋 튜닝 전술

1. 네트워크 오버헤드 vs 애플리케이션 수리기
트레이싱 그래프가 알려주는 진실은 딱 2가지로 나뉜다.

  • 간격 딜레이 (Gap): 라우터 A 에서 출발한 시간이 10ms 인데, 라우터 B에 도착한 시간이 50ms 라면 중간에 빈 공간(Gap) 이 생긴다. 이것은 100% 네트워크 회선 자체의 물리적 지연(WAN Latency) 또는 OS 커널의 큐잉 지연(Buffer Bloat) 이다. TCP BBR 이나 MTU 튜닝에 들어가야 한다.
  • 실행 딜레이 (Width): 도착은 50ms 에 했는데, 라우터 B 의 routing_lookup 이라는 네모 박스 자체가 20ms 나 길게 뻗어있다면? 라우터 B 의 CPU 성능이 낮거나 구독자(Sub) 가 너무 많아서 룩업 타임이 걸린 것이다. (16.4장 정적 라우팅으로 해결 가능).

2. 에러 폭포(Error Cascade) 추적
중간에 DB_Write Span 이 에러가 나서 붉게 물들면, 부모 Span 들도 줄줄이 에러 로그를 뱉는다. 트레이싱이 없으면 라우터 에러 로그만 보고 라우터를 고치려 드는 바보짓을 하게 되지만, 트레이싱은 가장 깊은 골짜기에 있는 “최초 원인(Root Cause)” Span 을 콕 집어내어 오폭을 막아준다.
