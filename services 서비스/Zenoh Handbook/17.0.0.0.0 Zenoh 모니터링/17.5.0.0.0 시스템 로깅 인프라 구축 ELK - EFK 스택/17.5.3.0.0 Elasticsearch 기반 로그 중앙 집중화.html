17.5.3 Elasticsearch 기반 로그 중앙 집중화

Fluent-bit 가 쏘아 올린 수만 건의 로그가 쏟아져 들어오는 종착역, Elasticsearch(ES) 가 버티고 있다. 엘라스틱서치는 텍스트를 인덱싱(Indexing) 하여 기가바이트의 도서관에서 원하는 로그를 0.1초 만에 찾아주는 검색의 신이다.

0.1 [인스펙션] 빅 컨트롤 타워 구축 전술

1. Kibana(키바나) 대시보드 시각화
ES 에 저장된 로그는 Kibana 라는 웹 UI 를 통해 시각화해야 비로소 가치가 있다.
키바나에 접속해 level: "ERROR" AND message: "Dropping" 이라고 검색어를 치는 순간, 전 세계 어디에 있는 라우터든 패킷을 버린 호스트 IP 와 시간대 목록이 쫙 뽑혀 나온다.

2. 인덱스 수명 주기(ILM) 관리
하루에 수 기가바이트씩 쌓이는 라우터 로그를 영원히 모아둘 수는 없다.
Elasticsearch 에 Index Lifecycle Management 규칙을 걸어라.

  • Hot 단계: 최근 3일 치 로그는 최고급 NVMe SSD 에 올려서 번개처럼 검색되게 한다.
  • Warm 단계: 3일이 지난 로그는 싼 HDD 스토리지로 내린다.
  • Delete 단계: 30일이 지난 로그는 흔적도 없이 삭제(Delete) 하여 디스크 파산을 방어한다.
