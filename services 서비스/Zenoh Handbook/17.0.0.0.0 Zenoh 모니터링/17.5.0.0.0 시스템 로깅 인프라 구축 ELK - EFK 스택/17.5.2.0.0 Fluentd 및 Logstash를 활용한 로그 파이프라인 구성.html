17.5.2 Fluentd 및 Logstash를 활용한 로그 파이프라인 구성

Zenoh 가 뱉어내는 로그는 터미널 화면(stdout) 구석에서 조용히 사라질 운명이다. 이 운명을 꺾고 로그를 트럭에 실어 거대한 물류 창고(Elasticsearch) 로 보내는 배달부가 바로 Fluentd(또는 Logstash) 다.

0.1 [Runbook] 데이터 운송망(Shipper) 부착 전술

1. JSON 포맷팅 (Log Forwarding 의 대원칙)
텍스트(Text) 로 떨어지는 로그는 컴퓨터(검색 엔진) 가 읽기 너무 힘들다.
Zenoh가 로그를 애초에 JSON 형태로 뱉게 가공하거나, Fluentd 에 정규식(Regex) 필터를 먹여 시간(time), 경고등급(level), 내용(message) 을 칼같이 쪼개야 한다.

2. Fluentbit (경량화 배달부) 에지 탑재
로봇(Edge) 환경 같은 가벼운 기기에 무거운 Logstash(자바 기반) 를 달면 로봇이 느려진다.
따라서 로봇에는 C코드로 작성된 3MB 짜리 초경량 Fluent-bit 를 데몬으로 띄워, zenohd 의 표준 출력(stdout) 을 잡아물게 한다.

# fluent-bit.conf (로봇 측)
[INPUT]
    Name   tail
    Path   /var/log/zenohd.log
[OUTPUT]
    Name   es
    Host   13.4.5.6 # 클라우드 쪽 Elasticsearch 주소
    Port   9200

이 세팅이 완료되면, 100대의 로봇이 뱉어내는 모든 빨간줄(Error) 이 클라우드로 일제히 발사되기 시작한다.
