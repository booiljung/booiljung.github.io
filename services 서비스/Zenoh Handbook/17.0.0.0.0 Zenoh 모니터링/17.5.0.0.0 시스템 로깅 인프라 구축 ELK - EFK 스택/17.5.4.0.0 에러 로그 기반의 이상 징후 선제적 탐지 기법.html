17.5.4 에러 로그 기반의 이상 징후 선제적 탐지 기법

로그를 모아두고 언제 쳐다볼 것인가? 서버가 고장 나서 상사에게 불려 간 뒤에야 키바나(Kibana) 창을 열어본다면 이미 늦었다.
가장 진보된 엘라스틱 인프라는 로그가 스스로 울부짖으며(Alerting) 당신의 슬랙 멱살을 잡고 깨우는 시스템 이다.

0.1 [Runbook] 엘라스틱 옵저버(Elastic Observer) 감시 런북

ElastAlert 나 Kibana Alerting 기능을 이용해 파수꾼을 세워라.

1. 임계치(Threshold) 강제 스파이크 룰
“어라? 지난 1달 동안 WARN: Auth Failed 텍스트가 하루에 2~3회 찍혔는데, 지금 10분 동안 500개가 찍혔네요? 해킹 방어 타공망이 뚫린 것 같습니다!”
로그의 특정 키워드가 단위 시간당 발생하는 ‘빈도수(Frequency)’ 가 임계치를 돌파 했을 때 즉각 알람을 쏜다.

2. 희귀(Rare) 이벤트 룰
PANIC 이나 OOM_KILL, Zenoh core crashed 와 같은 치명적 단어는 빈도수가 문제가 아니다. 1년에 딱 한 번 발생하더라도 발견 즉시 무조건 담당자 전화벨(PagerDuty 연동) 을 울려야 하는 절대 규칙(Absolute Rule) 이다.

에러가 터질 때마다 이 감시 룰(Rule) 딕셔너리를 두껍게 쌓아 올려, 똑같은 에러 로그가 다음번 장애를 일으키기 전에 선제적으로 통제하는 것이 AIOps 를 향한 아키텍트의 궁극적 목표다.
