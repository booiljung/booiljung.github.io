17.4 메트릭 수집 시스템 통합 (Prometheus)

CLI(명령어) 로 쳐서 보는 지표는 1회용이다. 엔지니어가 퇴근한 새벽 3시에 발생한 장애의 원인을 다음 날 아침에 분석하려면, 밤새 변동했던 지표들이 1초 단위 시계열(Time-series) 데이터베이스에 차곡차곡 쌓여 있어야만 한다.

클라우드 네이티브의 황제 툴인 Prometheus(프로메테우스) 를 Zenoh 에 빨대로 꽂아 넣어, 무한히 팽창하는 에지-클라우드 네트워크의 숨결을 영구적인 데이터로 박제하는 데이터 파이프라인 아키텍처를 구현한다.

1. Zenoh-Prometheus 익스포터(Exporter) 아키텍처

프로메테우스는 자기가 직접 데이터를 수집하지 않는다. 누군가 웹 포트(HTTP) 로 “/metrics” 라는 페이지를 열어주면, 거길 정기적으로 긁어가는(Scraping) 구조(Pull 기반) 다.

1.0.1 [인스펙션] 지표 적출구 개방 전술

Zenoh 코어 라우터 데몬은 기본적으로 친절하게도 프로메테우스 익스포터 엔진을 내장하고 있다. 켤 줄만 알면 된다.

1. 라우터 설정(zenohd.json5) 에 웹 서버 점화
라우터의 설정 뇌관에 플러그인을 하나 꽂는다.

// zenohd.json5
plugins: {
  rest: { // REST API 서버 플러그인 기동
    listen: ["http/0.0.0.0:8000"]
  }
}

2. 프로메테우스 노출 확인 (Scrape Target)
라우터를 켜고 웹 브라우저나 curl 로 http://localhost:8000/@/router/local/metrics 에 들어가 보라.
화면에 zenoh_session_count 12, zenoh_messages_dropped_total 0 같은 형태의 프로메테우스 규격 텍스트가 쏟아진다면, Zenoh 가 자신의 심장 박동(지표) 을 외부인에게 HTTP 포트로 무방비하게 뱉어낼 준비를 마친 것이다.

2. 퍼블리셔-서브스크라이버 통신 메트릭 수집 설정

라우터뿐만 아니라, 당신이 직접 짠 C++/Rust 로봇 프로그램에서도 성능 지표가 나와야 한다. “이 퍼블리셔 객체 1개가 초당 몇 번의 Put 을 호출하고 있는가?”

2.0.1 [Runbook] 어플리케이션 계층 지표 삽입 런북

1. Zenoh 세션 내부 측정 옵션(Metrics) 켬
개발자가 Zenoh(Rust) 클라이언트를 연결할 때, “내 기록(Metrics)을 남겨줘” 라고 선언해야 한다.

let mut config = zenoh::Config::default();
// 클라이언트 객체 스스로 자기 자신의 Rx/Tx 지표를 수집하게 만든다.
config.insert_json5("metrics/enabled", "true"); 
let session = zenoh::open(config).res().await.unwrap();

2. 커스텀 지표(Custom Metrics) 강제 발행
기본 제공되는 rx_bytes 외에도, 당신의 프로그램의 비즈니스 로직 지표(예: ai_inference_delay) 도 프로메테우스에 태워야 한다. 가장 쉬운 꼼수는 Zenoh 애플리케이션 내에 별도의 prometheus 크레이트(Crate) 와 hyper (웹 서버) 를 하나 파서 /metrics 포트를 임의로 열어버린 뒤 이 값들을 병렬로 노출하는 것이다. (망 전체가 한곳을 가리키는 Pull 기반 철학 유지).

3. 브릿지(ROS2, MQTT, DDS) 및 백엔드 플러그인 전용 지표 수집

이기종 통신의 병목은 항상 변환기(Bridge) 와 스토리지(Backend DB) 에서 발생한다. 이 녀석들의 지표를 놓치면 모니터링은 반쪽짜리가 된다.

3.0.1 [인스펙션] 이단아들의 심장박동 스크래핑

1. ROS2 Bridge (zenoh-bridge-dds) 텔레메트리
브릿지 역시 독립적인 데몬이므로 자체 REST 포트를 열게 패러미터를 준다.

## 브릿지의 관리용 포트를 8080 으로 뚫어버린다.
zenoh-bridge-dds --rest-http-port 8080

이후 http://localhost:8080/@/router/local/metrics 로 접속하면, DDS 진영에서 받아온 트래픽과 Zenoh 로 내보낸 트래픽의 오차가 얼마나 되는지(Loss) 증명하는 지표들이 쏟아진다.

2. 스토리지 백엔드(RocksDB/InfluxDB) 블로킹 추적
라우터 본체의 프로메테우스 로깅 포트(8000) 를 유심히 살펴보면 zenoh_storage_insert_time_seconds 같은 지표가 있다. 당신의 백엔드(DB) 가 하드디스크 I/O 병목으로 인해 쿼리를 버벅거리는지 실시간으로 고발하는 치명적 팩트 데이터다.

4. Prometheus 설정 및 시계열 데이터 저장 주기 최적화

수집 대상을 다 뚫어놨으면, 중앙의 관제탑에 설치된 Prometheus 데몬의 머리채를 잡아 이 포트들로 돌려세워야 한다.

4.0.1 [Runbook] 프로메테우스 강제 흡입(Scraping) 런북

1. 대상 주입 (prometheus.yml)
prometheus.yml 에 가서 “내가 지어볼 사냥감 목록(Targets)” 을 적는다.

scrape_configs:
  - job_name: 'zenoh-cloud-router'
    scrape_interval: 5s # 5초에 한 번씩 라우터를 긁어온다.
    metrics_path: '/@/router/local/metrics' # Zenoh 의 Admin URL 스펙
    static_configs:
      - targets: ['10.0.0.5:8000'] # 클라우드 라우터 IP

  - job_name: 'zenoh-edge-bridge'
    scrape_interval: 15s # 에지 기기는 대역폭을 아끼기 위해 15초 단위로 긁는다.
    metrics_path: '/@/router/local/metrics'
    static_configs:
      - targets: ['192.168.1.100:8080'] # 공장 쪽 DDS 브릿지 IP

2. 스크레이핑 간격(Interval) 의 공포
너무 짧게(1초) 잡으면 프로메테우스가 라우터에게 1초 초마다 HTTP 요청을 날려 오히려 라우터를 디도스(DDoS) 공격하는 오버헤드가 발생한다. 에지(Edge) 구간은 15~30초로 늘리고, 클라우드 코어는 5초로 당기는 하이브리드 간격 조절(Scrape Interval Tuning) 이 프로메테우스 연동의 핵심 철칙이다.
