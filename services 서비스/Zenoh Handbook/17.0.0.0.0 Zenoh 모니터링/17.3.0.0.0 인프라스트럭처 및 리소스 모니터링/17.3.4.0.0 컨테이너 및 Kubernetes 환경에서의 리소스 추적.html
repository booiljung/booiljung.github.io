17.3.4 컨테이너 및 Kubernetes 환경에서의 리소스 추적

Zenoh 를 도커(Docker) 나 쿠버네티스(K8s) Pod 으로 감싸서 배포했다면, 기존 리눅스 커널 모니터링은 장님이 된다. 컨테이너는 격리된 환상 속 공간(cgroups) 에 살고 있기 때문이다.

0.1 [인스펙션] Cgroups 포위(Siege) 전술

Kubernetes 환경에서의 관제탑 모범 설계다.

1. cAdvisor 탑재
호스트 서버에 node_exporter 대신 (또는 함께) cAdvisor(Container Advisor) 를 띄워라.
이 놈은 도커 컨테이너들의 격리된 벽(namespace) 을 뚫고 들어가, “현재 K8s 1번 Pod(Zenoh 라우터) 가 RAM 한계치(Limit) 인 2GB 중 1.9GB 를 썼음” 이라는 초정밀 지표를 뽑아낸다.

2. OOMKilled 추적
K8s 에서는 프로세스가 죽으면 자기가 알아서 다시 살려낸다(Restart). 개발자는 그것도 모르고 “잘 도네” 라고 착각한다.
kube-state-metrics 를 달아서, 라우터 Pod 의 restarts 횟수와 OOMKilled (메모리 초과로 K8s 가 강제 사살함) 이벤트를 반드시 덤프 해야 한다.
라우터가 재시작(Restart) 한다는 것은 그동안 날아가던 모든 패킷(Data Payload) 이 메모리 공중분해 되었다는 재앙의 묵시록이기 때문이다.
