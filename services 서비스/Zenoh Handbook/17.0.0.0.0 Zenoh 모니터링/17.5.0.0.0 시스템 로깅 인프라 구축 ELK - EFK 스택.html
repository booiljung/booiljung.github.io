17.5 시스템 로깅 인프라 구축 (ELK / EFK 스택)

메트릭(Metric) 이 “서버가 죽었다” 라는 사실을 알려준다면, 로그(Log) 는 “어떤 로봇이 잘못된 데이터를 쏴서 서버를 죽였는가” 라는 범인의 지문을 담고 있다.

하지만 로봇 100대가 각자 자기 디스크에 텍스트 파일로 범인의 지문을 남겨둔다면, 관리자는 100대의 로봇에 SSH 로 일일이 접속하여 grep 을 쳐야 하는 지옥에 빠진다. 이 장에서는 흩어진 모든 Zenoh 데몬의 로그를 하나로 모아 빛의 속도로 검색하게 해주는 ELK/EFK 스택 기반 중앙 집중화(Centralized Logging) 런북을 설파한다.

1. Zenoh 로깅 레벨(RUST_LOG) 설정 가이드

Zenoh 코어는 Rust 로 짜여 있다. 그리고 Rust 진영의 가장 강력하고 무지비한 로깅 엔진인 tracing 과 env_logger 규격을 따르고 있다.

1.0.1 [인스펙션] 노이즈 캔슬링(Noise-canceling) 런북

아무 설정 없이 켜면 화면에 쓸데없는 정보(INFO) 가 너무 많이 나와 디스크를 꽉 채워버린다.

1. RUST_LOG 의 기본 타겟팅
운영체제의 환경 변수(RUST_LOG) 하나만으로 Zenoh 의 입을 열고 닫을 수 있다.

## 프로덕션(Production) 표준 세팅: 전체는 경고(WARN)만 띄우되, zenoh 본체 모듈만 INFO를 띄워라.
export RUST_LOG="warn,zenoh=info"
./zenohd

2. 트러블슈팅을 위한 극단적 엑스레이 (Trace)
개발 중 패킷 하나가 어디서 드랍되는지 찾고 싶을 때는 코어 깊숙한 곳의 네트워크 모듈만 미친 듯이 파고들어야 한다.

## 디버깅 세팅: zenoh의 네트워크(net) 모듈 중 프레이밍(framing) 파트의 모든 바이트 흐름(trace)을 까발려라.
export RUST_LOG="warn,zenoh::net::framing=trace"

이렇게 범위를 좁히지 않고 무작정 RUST_LOG=trace 를 주면 1초 만에 기가바이트(GB) 단위의 로그가 터져 나와 터미널이 마비되고 컴퓨터가 죽어버리니 주의하라.

2. Fluentd 및 Logstash를 활용한 로그 파이프라인 구성

Zenoh 가 뱉어내는 로그는 터미널 화면(stdout) 구석에서 조용히 사라질 운명이다. 이 운명을 꺾고 로그를 트럭에 실어 거대한 물류 창고(Elasticsearch) 로 보내는 배달부가 바로 Fluentd(또는 Logstash) 다.

2.0.1 [Runbook] 데이터 운송망(Shipper) 부착 전술

1. JSON 포맷팅 (Log Forwarding 의 대원칙)
텍스트(Text) 로 떨어지는 로그는 컴퓨터(검색 엔진) 가 읽기 너무 힘들다.
Zenoh가 로그를 애초에 JSON 형태로 뱉게 가공하거나, Fluentd 에 정규식(Regex) 필터를 먹여 시간(time), 경고등급(level), 내용(message) 을 칼같이 쪼개야 한다.

2. Fluentbit (경량화 배달부) 에지 탑재
로봇(Edge) 환경 같은 가벼운 기기에 무거운 Logstash(자바 기반) 를 달면 로봇이 느려진다.
따라서 로봇에는 C코드로 작성된 3MB 짜리 초경량 Fluent-bit 를 데몬으로 띄워, zenohd 의 표준 출력(stdout) 을 잡아물게 한다.

## fluent-bit.conf (로봇 측)
[INPUT]
    Name   tail
    Path   /var/log/zenohd.log
[OUTPUT]
    Name   es
    Host   13.4.5.6 # 클라우드 쪽 Elasticsearch 주소
    Port   9200

이 세팅이 완료되면, 100대의 로봇이 뱉어내는 모든 빨간줄(Error) 이 클라우드로 일제히 발사되기 시작한다.

3. Elasticsearch 기반 로그 중앙 집중화

Fluent-bit 가 쏘아 올린 수만 건의 로그가 쏟아져 들어오는 종착역, Elasticsearch(ES) 가 버티고 있다. 엘라스틱서치는 텍스트를 인덱싱(Indexing) 하여 기가바이트의 도서관에서 원하는 로그를 0.1초 만에 찾아주는 검색의 신이다.

3.0.1 [인스펙션] 빅 컨트롤 타워 구축 전술

1. Kibana(키바나) 대시보드 시각화
ES 에 저장된 로그는 Kibana 라는 웹 UI 를 통해 시각화해야 비로소 가치가 있다.
키바나에 접속해 level: "ERROR" AND message: "Dropping" 이라고 검색어를 치는 순간, 전 세계 어디에 있는 라우터든 패킷을 버린 호스트 IP 와 시간대 목록이 쫙 뽑혀 나온다.

2. 인덱스 수명 주기(ILM) 관리
하루에 수 기가바이트씩 쌓이는 라우터 로그를 영원히 모아둘 수는 없다.
Elasticsearch 에 Index Lifecycle Management 규칙을 걸어라.

  • Hot 단계: 최근 3일 치 로그는 최고급 NVMe SSD 에 올려서 번개처럼 검색되게 한다.
  • Warm 단계: 3일이 지난 로그는 싼 HDD 스토리지로 내린다.
  • Delete 단계: 30일이 지난 로그는 흔적도 없이 삭제(Delete) 하여 디스크 파산을 방어한다.

4. 에러 로그 기반의 이상 징후 선제적 탐지 기법

로그를 모아두고 언제 쳐다볼 것인가? 서버가 고장 나서 상사에게 불려 간 뒤에야 키바나(Kibana) 창을 열어본다면 이미 늦었다.
가장 진보된 엘라스틱 인프라는 로그가 스스로 울부짖으며(Alerting) 당신의 슬랙 멱살을 잡고 깨우는 시스템 이다.

4.0.1 [Runbook] 엘라스틱 옵저버(Elastic Observer) 감시 런북

ElastAlert 나 Kibana Alerting 기능을 이용해 파수꾼을 세워라.

1. 임계치(Threshold) 강제 스파이크 룰
“어라? 지난 1달 동안 WARN: Auth Failed 텍스트가 하루에 2~3회 찍혔는데, 지금 10분 동안 500개가 찍혔네요? 해킹 방어 타공망이 뚫린 것 같습니다!”
로그의 특정 키워드가 단위 시간당 발생하는 ‘빈도수(Frequency)’ 가 임계치를 돌파 했을 때 즉각 알람을 쏜다.

2. 희귀(Rare) 이벤트 룰
PANIC 이나 OOM_KILL, Zenoh core crashed 와 같은 치명적 단어는 빈도수가 문제가 아니다. 1년에 딱 한 번 발생하더라도 발견 즉시 무조건 담당자 전화벨(PagerDuty 연동) 을 울려야 하는 절대 규칙(Absolute Rule) 이다.

에러가 터질 때마다 이 감시 룰(Rule) 딕셔너리를 두껍게 쌓아 올려, 똑같은 에러 로그가 다음번 장애를 일으키기 전에 선제적으로 통제하는 것이 AIOps 를 향한 아키텍트의 궁극적 목표다.
