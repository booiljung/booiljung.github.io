17.2 Zenoh 내장 도구를 활용한 기본 상태 점검

복잡한 모니터링 시스템(Prometheus, Elasticsearch) 을 구축하기 전, 우리가 가진 가장 빠르고 날카로운 메스는 Zenoh 가 자체적으로 내장하고 있는 관리망(Admin Space) 과 CLI 도구다.

이 도구들은 추가적인 설치 없이, 터미널 명령 한 줄과 REST API 만으로 시스템의 핏줄(Topology) 과 맥박(Session) 을 실시간으로 투시할 수 있게 해 준다. 외부의 무거운 감시 장비를 들이기 전, 네이티브 도구를 이용해 Zenoh 의 뼈대를 살펴보는 런북을 전개한다.

1. Admin Space(@/router/)의 이해와 활용

Zenoh 라우터는 일반 사용자의 데이터 토픽(robot/sensor) 외에도, 자기 자신의 내장 장기(상태) 를 숨겨진 은밀한 백도어 토픽 공간에 실시간으로 업데이트하고 있다. 이 공간이 바로 Admin Space (@/*) 다.

1.0.1 [인스펙션] 라우터 뇌파 투시 전술

라우터의 설정과 상태를 보려면 환경 설정 파일이나 남이 만든 문서를 찾을 게 아니라, 라우터에게 직접 물어보면 된다.

1. @/router 공간의 강제 리딩(Sub)
이 숨겨진 토픽을 구독(Subscribe) 하는 순간, 라우터 안에 어떤 사용자가 접속해 있고 버전이 뭔지 쏟아진다.

## 내 컴퓨터의 로컬 라우터에게 "너의 모든 설정과 상태를 내놔라" 고 명령한다.
z_get "@/**"

2. 런타임 설정의 핫-리로드(Hot-Reload) 주입
이 Admin Space 가 무서운 점은, 읽기(Get) 만 되는 게 아니라 쓰기(Put) 도 지원한다는 점이다. 라우터를 끄지 않고도 런타임에 설정을 바꿀 수 있다.

## 라우터를 재부팅하지 않고, 즉시 백엔드 스토리지 로깅 레벨을 debug로 변경!
z_put "@/router/local/config/logging/level" "debug"

에지(Edge) 인프라에서 라우터를 껐다 켜는 것 자체가 서비스 마비(Outage) 다. Admin Space 의 제어권을 장악하는 자가 무중단(Zero-downtime) 관제의 승리자가 된다.

2. Admin Space REST API를 통한 실시간 상태 조회

Zenoh CLI(z_get) 가 없거나 설치할 수 없는 원격지의 웹 브라우저, 나아가서 중앙 관제탑의 TypeScript 백엔드 서버에서 라우터 상태를 긁어오려면 무거운 Zenoh 라이브러리 연동보다 HTTP 가 훨씬 가볍다.

2.0.1 [Runbook] HTTP 브리지(REST API) 관제 전술

Zenoh 라우터를 기동할 때 터져 나오는 기본 관리용 REST API 포트(8000번) 를 적극적으로 악용하라.

1. 라우터 설정(Config) 덤프 추출
웹 브라우저 창이나 curl 로 라우터에 질문을 던진다.

## 라우터의 핵심 설정(Admin) 들을 JSON 포맷으로 떨어뜨려라.
curl http://localhost:8000/@/router/local/config

2. 구독 맵(Routing Table) 훔쳐보기
사내 메신저 슬랙(Slack) 의 봇(Bot) 에게 “지금 라우터에 물려있는 토픽 리스트 좀 가져와“라고 명령하기 가장 좋은 아키텍처다.

## 현재 라우터가 중계하고 있는 모든 세션들의 목록을 JSON 으로 뽑아낸다.
curl http://localhost:8000/@/router/local/sessions

이 REST API 자체를 사내 인트라넷 모니터링 페이지(React/Vue) 와 다이렉트로 연동해 버리면, 그것 자체가 훌륭한 1차 경고(Alert) 대시보드로 작동한다.

3. zenoh-cli를 이용한 네트워크 토폴로지 점검

로봇 A가 보낸 데이터가 라우터 B를 거쳐 관제탑 C 로 들어왔다. 그런데 이 데이터가 D 로 안 간다.
선이 어디서 끊겼는지 알려면 “현재 이 네트워크의 별자리(Topology)” 가 어떻게 그려져 있는지 투시경이 필요하다.

3.0.1 [인스펙션] 노드 연결망 엑스레이 전술

1. 이웃(Neighbors) 탐색기 가동
마우스로 라우팅 지도를 클릭하는 상상은 버려라.
에지 로봇에 SSH 로 접속한 뒤, 터미널에서 당장 자신이 연결된 홉(Hop) 의 타겟을 찢어 발겨라.

## 이 로봇이 바라보고 있는 모든 라우터 형님들의 IP와 세션 ID를 출력하라
z_get "@/router/local/sessions/link/**"

2. 통신 단절(Split-Brain) 증거 수집
만약 공장 1구역의 라우터들끼리 뭉친 망과, 공장 2구역의 라우터들끼리 뭉친 망이 중간 다리가 끊기면, 각 구역은 자기들만 정상인 줄 안다.
이럴 때는 관제탑(Core) 데몬의 세션을 나열했을 때 공장 1구역을 대변하는 Peer_ID 가 sessions 리스트에 존재하는지만 문자열 매칭(grep) 해보면 1초 만에 브릿지 끊김 사태를 단정 지을 수 있다.

4. 라우터 및 피어 연결 상태와 세션 확인

연결된 놈 중 누가 피어(Peer) 이고 누가 클라이언트(Client) 인지, 그리고 이놈이 지난 1분 동안 무슨 개짓거리(Traffic) 를 했는지 낱낱이 파헤친다.

4.0.1 [Runbook] 세션 텔레메트리(Telemetry) 추출 전술

Admin Space 에는 “현재 이 세션이 얼마나 많은 패킷을 쐈는지“에 대한 엑셀 표가 실시간으로 돌아가고 있다.

1. 세션의 정체와 권한 파악법
라우터에 접속된 세션 목록을 까보면 그놈의 신분이 튀어나온다.

z_get "@/router/local/sessions/*"
## 출력 예시: [Session: xxx, WhatAmI: Client, Locators: tcp/10.0.0.5:45677]

2. 네트워크 도둑(Traffic Hog) 색출
특정 로봇 하나가 라이다 센서의 배칭(Batching) 을 풀어서, 라우터 대역폭 전체를 갉아먹는 트래픽 빌런이 됐다고 치자.

## 각 세션(접속자) 들의 RX(수신), TX(송신) 바이트 카운터를 실시간 덤프
curl http://localhost:8000/@/router/local/sessions/*/metrics

여기서 다른 놈들은 rx_bytes: 1540 인데 혼자 rx_bytes: 9999999999 를 찍고 있는 세션 ID 를 찾아라. 그 ID 에 물린 IP 범인이 바로 당신의 시스템을 지금 마비시키고 있는 좀비(Zombie) 프로세스다.
