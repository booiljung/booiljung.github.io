14.10.4 장애 발생 시 스토리지 백업(Backup) 및 스냅샷(Snapshot) 복구 절차

건물이 무너져 클라우드 DB가 메인보드째로 불타버렸다. 하지만 우리에겐 백업이 있다. 복구 타임(Recovery Time)을 최소화하는 런북이다.

0.1 [Runbook] RocksDB 콜드 백업 및 주입 전술

가장 많이 쓰는 내장형 엔진, RocksDB 의 생존 복원술이다.

1. 무정지 스냅샷(Snapshot) 생성 주기 지정
로보틱스 엣지에서는 파일 압축 툴(tar)로 실행 중인 DB 폴더를 그냥 떠버리면 파일이 깨져버린다(Corruption). 반드시 RocksDB 자체 백업 API 를 사용하거나, 볼륨을 내린 후 떠야 한다.

  • 가장 원시적이고 확실한 방법: 마운트-아웃 후 이중화 복제.
# REST API로 볼륨 잠시 분리! (디스크 I/O 강제 정지)
curl -X DELETE http://admin:admin@localhost:8000/@/router/local/plugin/storages/backend/rocksdb/volume/robot_vol
# 조각난 .sst 파일들 안전하게 압축
tar -czvf robot_db_backup.tar.gz /var/lib/zenoh/rocksdb
# 다시 볼륨 마운트!
curl -X PUT http://admin...

2. 콜드 부팅(Cold Boot)과 복원
새로 사온 깡통 서버에 Zenoh 데몬을 깐다. 데몬을 켜기 전에 백업 파일(tar.gz)의 압축을 풀어 지정된 디렉토리(/var/lib/zenoh/rocksdb)에 정확히 복원해둔다.
그리고 라우터를 켜면, 데몬 스토리지 플러그인이 기동되는 순간 하드디스크에 깔려있는 이전 시대의 인덱스와 멤테이블 구조를 통째로 읽어 들여 1초 만에 폭발 전의 세계를 완전히 복원해 낸다.
