14.10.3 스토리지 데드락(Deadlock) 및 메모리 누수 식별 및 해결 방법

라우터는 멀쩡히 돌아가고 다른 토픽은 다 잘 지나가는데, 유독 DB 에 저장하는 쿼리(Get)나 플러시 통로가 먹통이 되는 현상이다.

0.1 [Runbook] 비동기 블로킹(Async Blocking) 색출 전술

Zenoh 코어는 코루틴(Tokio) 기반으로 비동기 스레드를 돌린다.

1. 독성 스토리지 플러그인에 의한 데드락

  • 사용자 정의(Custom) 백엔드를 짜면서, 디스크에서 데이터를 읽어오는 함수(Storage::get) 단에 시간이 오래 걸리는 무거운 코드를 await 없이 블로킹(Blocking) 함수로 짜버렸다.
  • 결과: Tokio의 코어 스레드 한 개가 그 함수에서 멈춰버리고(Stuck), 뒤이어 들어오는 수천 개의 Put 데이터가 스레드를 할당받지 못해 라우팅 전체가 터져버린다.
  • 해결책: 플러그인 개발 시 절대 std::fs 블로킹 함수를 쓰지 말고, 무조건 tokio::fs 나 비동기 클라이언트풀을 사용해야 한다.

2. 멤테이블(MemTable) 해방 누수 (Memory Leak)

  • InfluxDB 서버가 터졌는데도 라우터의 InfluxDB 백엔드 플러그인이 이를 감지하지 못하고 버퍼에 데이터만 계속 쌓아서 램(RAM)을 다 잡아먹는 현상.
  • RUST_LOG=trace zenohd 로 데몬을 켜서 스플릿(Split) 로그를 관찰하라. “Timeout Flush” 경고 로그가 연속으로 뜨면, 스토리지 엔진과 뒷단 실제 DB 의 연결선이 끊어진 것이다. 즉각 플러그인을 언마운트(Unmount)시켜 라우터가 같이 죽는 것을 막아야 한다.
