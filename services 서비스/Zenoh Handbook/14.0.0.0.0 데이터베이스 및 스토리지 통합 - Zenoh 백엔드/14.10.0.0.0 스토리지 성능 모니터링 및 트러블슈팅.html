14.10 스토리지 성능 모니터링 및 트러블슈팅

Zenoh의 통신 오버헤드가 제로(0)에 가깝다 한들, 데이터를 받아먹는 디스크(DB)가 체하면 전체 파이프라인이 즉격으로 박살 난다.

모니터링 없는 스토리지 운영은 눈을 가리고 시속 200km로 달리는 것과 같다. RocksDB가 디스크를 얼마나 긁고 있는지, InfluxDB에 데이터를 밀어 넣는 큐(Queue)가 꽉 차서 터지기 직전인지 알아야 한다.
이 장에서는 백엔드 플러그인이 내뿜는 비명 소리(Metrics)를 수집하여, 엣지(Edge)와 클라우드(Cloud)의 디스크 병목을 조기에 색출하고 멸망을 막아내는 트러블슈팅 런북(Troubleshooting Runbook)을 마스터한다.

1. 디스크 I/O 병목 분석 및 성능 한계 측정

데이터 저장의 종착역은 결국 물리학, 즉 하드디스크의 자성 패턴을 바꾸는 시간이다.

1.0.1 [Runbook] 디스크 스톨(Write Stall) 감지 전술

로봇 센서가 초당 1만 번씩 데이터를 쏘는데, 디스크가 초당 5천 번밖에 못 적으면 벌어지는 참사다.

1. RAM 버퍼의 한계 돌파 증상

  • 증상: z_put을 쏠 때는 멀쩡한데 메모리(RAM) 사용량이 미친 듯이 치솟다가 어느 순간 라우터 데몬이 OOM(Out Of Memory)으로 폭사한다.
  • 원인: 스토리지 엔진(RocksDB 등)이 디스크 쓰기 버퍼(MemTable)에 데이터를 채웠으나, 하드디스크가 너무 느려서(IOPS 부족) 비워내질 못해 버퍼가 무한정 팽창한 것이다.

2. iostat 기반 병목(Bottleneck) 추적
리눅스 서버에서 병목을 색출하라.

## 초당 디스크 I/O 지표 모니터링
iostat -x 1
  • %util (디스크 활용률): 이 값이 99%에 달해 있다면, 당신이 짠 Zenoh 네트워크 스피드가 빠르건 느리건 상관없이 물리적 하드디스크가 한계점인 것이다.
  • 해결책: 볼륨 마운트 설정에서 compression: lz4 압축을 켜서 디스크 쓰기 용랑을 줄이거나, 센서의 발송 주기를 낮춰라.

2. 백엔드 플러그인 전용 지표(Metrics) 수집 및 시각화 연동

라우터 겉면에서 아무리 패킷 수를 세어봤자, 스토리지 내부의 상태는 보이지 않는다.

2.0.1 [인스펙션] Prometheus 심장 박동 모니터링 전술

Zenoh 라우터에 내장된 원격 측정(Telemetry) 포트를 개방하여 관제탑으로 모든 스탯을 전송한다.

1. 프로메테우스(Prometheus) 종단점(Endpoint) 해제

// zenohd.json5
plugins: {
  // 모니터링 플러그인 켬!
  prometheus: {
    port: 8001
  }
}

2. 스토리지 심층 지표(Deep Metric) 가시화
기본 라우팅 스탯뿐 아니라 스토리지의 입출력 건수까지 찍혀 나온다.

  • zenohd_storage_put_count_total{volume="robot_rocks"}: 이 볼륨에 몇 건이나 Insert 되었는가?
  • zenohd_storage_get_latency_ms{volume="robot_rocks"}: 쿼리를 때렸을 때 디스크에서 꺼내오는 데 걸린 내부 소요 밀리초 단위 시간.

이 지표를 Grafana 대시보드에 연동해 두면, “어느 순간 쿼리 리턴 속도(Latency)가 1밀리초에서 갑자기 500밀리초로 역주행하기 시작하는 시점“을 포착할 수 있다. 그것이 바로 디스크 조각 모음 기능(Compaction)이 발동하여 시스템 I/O가 정체되는 치명적인 순간이다.

3. 스토리지 데드락(Deadlock) 및 메모리 누수 식별 및 해결 방법

라우터는 멀쩡히 돌아가고 다른 토픽은 다 잘 지나가는데, 유독 DB 에 저장하는 쿼리(Get)나 플러시 통로가 먹통이 되는 현상이다.

3.0.1 [Runbook] 비동기 블로킹(Async Blocking) 색출 전술

Zenoh 코어는 코루틴(Tokio) 기반으로 비동기 스레드를 돌린다.

1. 독성 스토리지 플러그인에 의한 데드락

  • 사용자 정의(Custom) 백엔드를 짜면서, 디스크에서 데이터를 읽어오는 함수(Storage::get) 단에 시간이 오래 걸리는 무거운 코드를 await 없이 블로킹(Blocking) 함수로 짜버렸다.
  • 결과: Tokio의 코어 스레드 한 개가 그 함수에서 멈춰버리고(Stuck), 뒤이어 들어오는 수천 개의 Put 데이터가 스레드를 할당받지 못해 라우팅 전체가 터져버린다.
  • 해결책: 플러그인 개발 시 절대 std::fs 블로킹 함수를 쓰지 말고, 무조건 tokio::fs 나 비동기 클라이언트풀을 사용해야 한다.

2. 멤테이블(MemTable) 해방 누수 (Memory Leak)

  • InfluxDB 서버가 터졌는데도 라우터의 InfluxDB 백엔드 플러그인이 이를 감지하지 못하고 버퍼에 데이터만 계속 쌓아서 램(RAM)을 다 잡아먹는 현상.
  • RUST_LOG=trace zenohd 로 데몬을 켜서 스플릿(Split) 로그를 관찰하라. “Timeout Flush” 경고 로그가 연속으로 뜨면, 스토리지 엔진과 뒷단 실제 DB 의 연결선이 끊어진 것이다. 즉각 플러그인을 언마운트(Unmount)시켜 라우터가 같이 죽는 것을 막아야 한다.

4. 장애 발생 시 스토리지 백업(Backup) 및 스냅샷(Snapshot) 복구 절차

건물이 무너져 클라우드 DB가 메인보드째로 불타버렸다. 하지만 우리에겐 백업이 있다. 복구 타임(Recovery Time)을 최소화하는 런북이다.

4.0.1 [Runbook] RocksDB 콜드 백업 및 주입 전술

가장 많이 쓰는 내장형 엔진, RocksDB 의 생존 복원술이다.

1. 무정지 스냅샷(Snapshot) 생성 주기 지정
로보틱스 엣지에서는 파일 압축 툴(tar)로 실행 중인 DB 폴더를 그냥 떠버리면 파일이 깨져버린다(Corruption). 반드시 RocksDB 자체 백업 API 를 사용하거나, 볼륨을 내린 후 떠야 한다.

  • 가장 원시적이고 확실한 방법: 마운트-아웃 후 이중화 복제.
## REST API로 볼륨 잠시 분리! (디스크 I/O 강제 정지)
curl -X DELETE http://admin:admin@localhost:8000/@/router/local/plugin/storages/backend/rocksdb/volume/robot_vol
## 조각난 .sst 파일들 안전하게 압축
tar -czvf robot_db_backup.tar.gz /var/lib/zenoh/rocksdb
## 다시 볼륨 마운트!
curl -X PUT http://admin...

2. 콜드 부팅(Cold Boot)과 복원
새로 사온 깡통 서버에 Zenoh 데몬을 깐다. 데몬을 켜기 전에 백업 파일(tar.gz)의 압축을 풀어 지정된 디렉토리(/var/lib/zenoh/rocksdb)에 정확히 복원해둔다.
그리고 라우터를 켜면, 데몬 스토리지 플러그인이 기동되는 순간 하드디스크에 깔려있는 이전 시대의 인덱스와 멤테이블 구조를 통째로 읽어 들여 1초 만에 폭발 전의 세계를 완전히 복원해 낸다.
