14.9.2 데이터 암호화(Data-at-Rest) 기법 및 암호화 키 관리

도둑이 라우터를 해킹하지 못하자, 엣지 기기(라즈베리 파이)의 전원을 끄고 SD 카드를 훔쳐갔다. 그 안에 담긴 RocksDB 폴더나 SQLite 파일을 열어보면 회사 기밀이 고스란히 담겨 있다.

0.1 [인스펙션] 저장소 절대 암호화 (Encryption-at-Rest) 전술

Zenoh 코어는 네트워크 위를 날아다니는 데이터(Data-in-Transit)를 TLS로 암호화할 수는 있지만, 디스크에 누워있는 데이터(Data-at-Rest)를 암호화해 주지는 않는다.

1. OS 레벨 암호화 (fscrypt / LUKS 활용)
가장 강력하고 빠른 런북이다.
Zenoh 백엔드가 마운트될 디렉토리(/var/lib/zenoh/rocksdb) 자체를 리눅스의 하드웨어 가속 암호화 파일시스템(LUKS) 파티션으로 묶어라.
로봇 부팅 시 보안 모듈(TPM/HSM)이 잠금을 풀기 전에는 디스크 자체가 쓰레기 값(Ciphertext)으로 보인다.

2. 애플리케이션 레벨 암호화 (Publisher-side Encryption)

  • 로봇(발신자)이 Zenoh 라우터로 데이터를 Put 쏘기 직전에, 센서 코드 단에서 페이로드(Payload)를 AES-256GCM 으로 구워버린다.
  • Zenoh 라우터 스토리지 백엔드는 이 데이터가 영상인지 암호문인지 모르고 일단 디스크에 거대한 바이너리 덩어리(BLOB)로 박는다(Persist).
  • 관제 센터의 Get 쿼리 결과로 그 바이너리 덩어리가 날아오면 관제 센터의 마스터 키(Master Key)로 즉석 복호화한다.
  • 이 전술의 장점: 스토리지 서버가 통째로 털려도, 파일 시스템 전문가가 와도 암호 단어를 풀 방법이 절대 존재하지 않는다. 진정한 제로-트러스트(Zero-Trust) 저장소다.
