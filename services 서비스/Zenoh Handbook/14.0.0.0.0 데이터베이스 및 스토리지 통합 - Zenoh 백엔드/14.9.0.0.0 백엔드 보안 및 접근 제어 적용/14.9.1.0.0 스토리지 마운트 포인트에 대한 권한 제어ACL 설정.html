14.9.1 스토리지 마운트 포인트에 대한 권한 제어(ACL) 설정

스토리지 엔진(Backend)은 눈이 멀어있다. 누군가 인증서(Certificate) 없이 들어와서 Delete 패킷을 날려도 망에 보이면 그냥 지워버린다. 따라서 라우터 본체 단에서 “이 볼륨에는 누가 접근할 수 있는가“를 검문해야 한다.

0.1 [Runbook] 라우터 레벨의 절대적 권한 통제 (ACL)

Zenoh 라우터 기동 설정 파일에서, 백엔드 마운트 이전에 미리 권한(Access Control List) 계층을 덮어 씌운다.

1. 읽기-쓰기 분할 통제
일반 센서는 쓰기(Put)만 가능해야 하고, 모니터링 데몬은 읽기(Get)만 가능해야 한다. 절대 Delete 권한을 모두에게 주지 마라.

// zenohd.json5
plugins: {
  access_control: {
    // 1. 센서 권한: 오직 넣을 수만 있다(Put)
    roles: {
      "sensors": {
        allow: ["Put:robot/sensor/**"]
      },
      // 2. 관제탑 권한: 볼 수만 있다(Get)
      "monitor": {
        allow: ["Query:robot/sensor/**"]  
      },
      // 3. 관리자 권한: 삭제 가능
      "admin": {
        allow: ["Put:**", "Query:**", "Delete:**"]
      }
    }
  }
}

2. 스토리지 방패막의 완성
이 ACL 이 켜지는 순간, 라우터는 악성 클라이언트가 보낸 z_delete "robot/sensor/**" 패킷 자체를 스토리지 플러그인에게 전달하지 않고 네트워크 앞단에서 즉시 폐기(Drop) 해버린다. 백엔드(DB) 엔진은 자신이 공격받았다는 사실조차 모르게 하는 가장 우아한 아키텍처다.
