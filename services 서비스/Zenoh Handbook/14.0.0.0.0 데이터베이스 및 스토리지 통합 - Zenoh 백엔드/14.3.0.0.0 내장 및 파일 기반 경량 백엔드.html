14.3 내장 및 파일 기반 경량 백엔드

외부 데이터베이스(InfluxDB, MySQL 등) 를 별도로 설치할 자원도, 시간도 없는 초소형 엣지(라즈베리 파이, 드론) 디바이스를 위한 구원의 런북.

Zenoh 는 오직 자기 자신만으로도 RAM 과 하드디스크를 완벽한 스토리지 서버로 탈바꿈시키는 2종의 내장형(Built-in) 백엔드를 기본 제공한다. Memory 와 FS 백엔드를 통해 최소 비용으로 데이터댐을 건설한다.

1. Memory 백엔드: 초고속 인메모리 캐싱 전략

“하드디스크가 느려서 못 참겠다. 휘발되더라도 RAM 속도로 무조건 꽂아 넣어라!”

1.0.1 [Runbook] 무중단 RAM 레디스(Redis) 대체 전술

별도의 Redis 캐시 서버를 단 한 줄의 설정만으로 대체해 버린다.

1. 초고속 캐시 레이어 구성

// zenohd.json5
storages: {
  fast_cache: {
    key_expr: "cam/100fps/**",
    backend: "memory" // 다른 건 필요 없다!
  }
}

2. 아키텍트의 무기: 0.1ms 응답속도

  • 자율주행 자동차가 카메라 프레임(Blob) 을 Put 하면, 메모리 백엔드는 이를 운영체제의 RAM 에 곧바로 포인터 참조 형태로 구겨 넣는다(Zero-copy 지향).
  • 분석 서버(AI 노드) 가 Get "cam/100fps/**" 을 치면, 디스크 I/O 나 소켓 오버헤드 없이 데몬의 RAM 속 포인터가 즉각 반환된다.
  • 명심할 것: 라우터 데몬 프로세스가 죽으면(Kill), 저 memory 백엔드 안에 들어있던 데이터도 지구상에서 완벽히 소멸한다. “가장 빠르고, 가장 가벼우며, 가장 무책임한” 스토리지다.

2. FS(File System) 백엔드: 로컬 파일 시스템과의 데이터 연동

디스크에 남겨야 하지만, 데이터베이스를 설치할 권한이 없거나 오버스펙일 때 쓴다.

2.0.1 [인스펙션] OS 파일 트리를 DB로 승격시키기

운영체제의 mkdir, echo, 파일 쓰기 시스템 콜 자체를 훌륭한 Key-Value 저장소로 우려먹는 런북이다.

1. 파일시스템 백엔드 마운트

storages: {
  local_robot_storage: {
    key_expr: "robot/config/**",
    backend: "fs",
    // 컴퓨터의 실제 하드디스크 폴더 경로를 지정한다!
    dir: "/var/lib/zenoh/my_storage" 
  }
}

2. 자동화된 파일 로깅
이 마운트가 활성화된 이후부터, 누군가가 z_put "robot/config/speed" "100" 을 때리면?

  • FS 플러그인은 리눅스 명령어로 번역한다: mkdir -p /var/lib/zenoh/my_storage/robot/config && echo "100" > speed
  • 파일 브라우저로 열어보면 데이터가 진짜 텍스트 파일과 폴더들 무리로 예쁘게 정리되어 저장되는 미친 직관성을 자랑한다.

3. FS 백엔드의 디렉터리 구조 및 파일 저장 규칙

와일드카드 질의가 들어오면 스토리지 엔진은 리눅스 하드디스크를 어떤 형식으로 긁어오는가?

3.0.1 [인스펙션] 하드디스크 I/O 매핑 룰

1. URI 기반 폴더 트리 생성기
z_put "factory/zone_1/sensor_a" "24.5"
이 명령 하나는 디스크 상에 factory/zone_1/ 이라는 중첩 폴더를 찍어낸다. 만약 트리가 100 뎁스라면 리눅스 폴더도 100 뎁스가 만들어진다.

2. 덤프 타겟으로서의 텍스트 파일
Zenoh 프로토콜의 Payload 값("24.5") 은 sensor_a 라는 이름의 파일 본문으로 저장된다. 이때 확장자는 따로 붙지 않는다.

3. 와일드카드 폭격 대응 (Tree Traversal)
만약 관제사가 z_get "factory/**" 권한으로 쿼리를 날린다면?
FS 플러그인은 리눅스의 find /var/lib/zenoh/my_storage/factory -type f 명령어를 돌려 모든 파일시스템 트리를 긁어온 뒤, 그걸 몽땅 JSON 배열로 직렬화하여 클라이언트에 던진다.

  • 경고: 매우 직관적이지만, 파일 갯수가 10만 개가 넘어가는 순간 운영체제의 하드디스크 I/O 블로킹 병목에 걸려 DB 로서의 성능을 완전히 상실한다. (이후 설명할 RocksDB 가 필요한 이유다).

4. 경량 엣지 디바이스를 위한 휘발성 데이터 관리 실무

드론이나 배터리 구동 로봇에 SD 카드(하드디스크) 쓰기 작업을 남발하면?
SD 카드의 플래시 셀이 갈려나가 몇 달 만에 컨트롤러가 부서진다(Wear-leveling 파괴).

4.0.1 [Runbook] I/O 완전 억제 하이브리드 전술

1. 로봇 내부 State 의 전면 무효화 (RAM 화)
로봇 내부의 속도, 배터리, 모터 상태 토픽(robot/telemetry/**) 은 무조건 Memory 백엔드에만 마운트하라.

  • 로터가 돌 때마다 초당 10번씩 갱신(Update) 이 일어나더라도, RAM 위에서만 덮어쓰기가 벌어지므로 SD 카드의 수명은 마이너스 제로다.

2. 영구 보존 데이터(Config) 와의 교차 마운트
하지만 로봇의 “무선랜 비밀번호” 나 “PID 제어 상수값” 은 로봇 전원이 꺼졌다 켜져도 무조건 보존되어야 한다. 이 녀석들은 robot/config/** 라는 키 스페이스(Key Space) 로 분리시킨 뒤 억지로 FS (File System) 백엔드에 마운트(접착) 한다.

같은 데몬, 같은 시스템 안에 돌아가더라도 아키텍트가 키 배분을 어떻게 쪼개고 각각을 Memory 냐 FS 냐로 배치하느냐에 따라 엣지 디바이스 하드웨어의 물리적 파손 시기조차 몇 년 단위로 뒤바꿀 수 있다.
