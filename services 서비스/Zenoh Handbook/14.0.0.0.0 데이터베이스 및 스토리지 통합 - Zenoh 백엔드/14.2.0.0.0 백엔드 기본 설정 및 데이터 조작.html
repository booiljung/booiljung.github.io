14.2 백엔드 기본 설정 및 데이터 조작

스토리지 플러그인을 데몬에 주입했다면, 이제 그 저장소에 밥(Data) 을 줄 차례다.
이 장에서는 가장 기초적인 JSON 설정 파일 조작법부터, 외부 관제망에서 REST API 로 DB 를 찔러보는 CRUD 의 런북을 완성한다.

1. JSON 설정 파일을 이용한 스토리지 마운트 및 언마운트

데몬이 켜질 때, 어떤 DB 를 잡고 어떤 키(Key) 를 훔쳐볼지를 설정 파일(zenohd.json5) 로 박아넣는다.

1.0.1 [Runbook] 마운트(Mount) 선언 전술

1. 영구 마운트 (정적 설정)

{
  listen: ["tcp/0.0.0.0:7447"],
  storages: {
    // 임의의 스토리지 아이디
    my_memory_db: {
      // 1. 어떤 토픽을 훔쳐볼 것인가?
      key_expr: "robot/sensors/**",
      // 2. 어떤 플러그인 워커를 쓸 것인가? (memory 백엔드)
      backend: "memory"
    }
  }
}

데몬을 zenohd -c zenohd.json5 로 기동하는 순간, 저 플러그인이 로드되어 메모리상에 독립적인 DB 인스턴스를 하나 켜고 앉아서 데이터를 쓸어 담는다.

2. 핫-언마운트 및 관제
파이프라인이 돌고 있는데 스토리지만 조용히 내리거나 올리고 싶다면 관리자 REST API (admin 플러그인) 를 치면 된다. 하지만 로보틱스 보안과 시스템 안정성 측면에서, 마운트는 “재부팅 시 무조건 정적으로 묶이는 파일 기반 설정” 을 마스터 런북으로 삼는 것이 생존형 아키텍트의 철칙이다.

2. 스토리지 생성, 읽기, 갱신, 삭제 (CRUD) 기본 연산

스토리지라고 해서 SQL 을 치는 게 아니다. Zenoh 스토리지에 대한 CRUD 는 철저하게 기반 프로토콜(Pub/Sub, Query) 의 함수를 그대로 재활용한다.

2.0.1 [인스펙션] 오버헤드 제로의 데이터 매니지먼트

스토리지가 robot/sensors/** 에 마운트되어 있다고 가정한다.

1. Create & Update (생성 및 갱신)

z_put "robot/sensors/temp" "25.0"

Zenoh 는 Put 과 Update 를 구분하지 않는다. 그냥 Put 을 치면 (원래는 다른 로봇에게 날아가는 명령이지만) 스토리지 데몬도 이걸 주워듣고 DB 에 집어넣는다. 데이터가 없으면 만들고(Create), 있으면 덮어쓴다(Update).

2. Read (읽기)

z_get "robot/sensors/temp"

Get 을 치는 순간, 스토리지 플러그인은 자기가 가진 하드디스크를 뒤져서 방금 집어넣었던 25.0 이란 텍스트를 Reply 로 쏘아 보낸다.

3. Delete (삭제)

z_delete "robot/sensors/temp"

자연 상태의 데이터는 날아다니다 사라지지만, 스토리지에 영구결박된 데이터는 명시적으로 지우지 않으면 하드디스크가 터질 때까지 남는다. Delete 패킷을 전송하면 스토리지 플러그인이 DB 단에서 Drop 쿼리를 때려 디스크 용량을 비워준다.

3. REST API를 활용한 백엔드 데이터 직접 접근 및 검증

웹 모니터링 대시보드(React/Vue) 를 만드는 개발자에게 “Zenoh C++ SDK 깔아서 z_get 치세요” 라고 할 수는 없다.
Zenoh 라우터는 훌륭한 HTTP 브릿지(REST API) 플러그인을 기본 보유하고 있다.

3.0.1 [Runbook] 웹 관제망과의 접합 전술

라우터 세팅에서 rest 플러그인을 활성화(zenohd --rest-http-port 8000) 시켰다. 이제 모든 CRUD 는 포스트맨(Postman) 이나 웹 브라우저(curl) 의 HTTP 메서드로 치환된다.

1. Read (HTTP GET)
웹 프론트엔드에서 스토리지에 데이터를 물어본다.

curl -X GET http://<router_ip>:8000/robot/sensors/temp

라우터가 이 HTTP GET 요청을 순수 Zenoh 의 z_get 패킷으로 번역하여 백엔드 DB 에 쿼리하고, 그 응답을 다시 브라우저에 JSON 으로 돌려준다.

2. Create/Update (HTTP PUT)

curl -X PUT http://<router_ip>:8000/robot/sensors/temp -d '27.5'

이 REST API 백도어가 존재하기 때문에, 아키텍트는 분산 DB 의 검증(Validation) 을 위해 굳이 클라이언트 컴파일 코드를 짤 필요 없이, 브라우저 주소창에 치는 것만으로도 수천 킬로미터 떨어져 있는 로봇의 DB 단 무결성을 즉석에서 디버깅해 낼 수 있다.

4. 데이터 직렬화(Serialization) 및 역직렬화 설정 (JSON, MessagePack 등)

카메라 영상(Vec) 을 InfluxDB(텍스트형 시계열 DB) 에 그냥 들이부으면 데이터가 박살 나거나 스토리지 플러그인이 패닉을 일으킨다.

4.0.1 [인스펙션] 스토리지 타입 정렬(Alignment) 룰

1. Payload 맹목성의 대가
Zenoh는 기본적으로 Payload (본문 데이터) 에 형(Type) 검사를 하지 않는 “바이트 스트림 버스” 다. 그러나 DB 로 들어갈 때는 엄격한 형 변환(Cast) 이 필수다.

2. 아키텍트의 헤더 명시화 작전
데이터를 쏠 때(Put), 페이로드의 “Encoding” 옵션을 반드시 설정해야 스토리지 플러그인이 올바르게 역직렬화(Deserialize) 를 쳐서 DB 에 구겨 넣는다.

  • 텍스트/JSON 데이터: Encoding: application/json 로 쏘면 백엔드는 이것을 문자열-키 기반의 Document 형태로 예쁘게 파싱해 DB(예: Mongo) 에 저장한다.
  • 이진 바이너리 데이터: 구글 프로토콜 버퍼(Protobuf) 나 MessagePack, 이미지 데이터인 경우 application/octet-stream 으로 쏴라. 백엔드 플러그인은 파싱을 포기하고 BLOB (Binary Large Object) 통짜 덩어리로 디스크에 박아버린다.

데이터베이스라는 까다로운 포식자에게 밥을 줄 땐, 이게 물인지 고기인지를 네트워크 계층(Encoding Header) 에 단단히 적시해야 시스템 크래시를 피할 수 있다.

5. 스토리지 쿼리를 위한 셀렉터(Selector) 작성 규칙

스토리지가 집어먹은 하위 센서값 수백 개를 한 방에 쓸어 올려야(Read) 할 때, 코드를 어떻게 짜야 하는가?

5.0.1 [Runbook] 와일드카드 전광석화 추출 전술

Zenoh 의 쿼리 셀렉터는 SQL 의 SELECT * FROM tbl 을 대체하는 고속 계층형 슬라이서다.

1. 단일 노드 정밀 타격

z_get "factory/robot_1/sensor/temp"

DB 안에서 정확히 저 키에 해당하는 값 하나를 가져온다. (로컬 데이터 캐싱).

2. 하위 브랜치(자식) 전체 폭격 (*)

z_get "factory/robot_1/sensor/*"

스토리지 백엔드는 temp, humidity, pressure 등 sensor/ 바로 아래에 매달린 1계층 자식들만 뽑아서 복수 개의 Reply 배열로 던져준다.

3. 섭-트리 글로벌 폭격 (**)

z_get "factory/robot_1/**"

스토리지는 robot_1 하위에 숨겨진 센서, 모터, 카메라, 상태 로그 등 수천 개의 계층구조 트리 전체를 풀 스캔(Full-Scan) 쳐서 융단폭격으로 반환한다.
이 기능을 웹 관제(REST API) 에 열어둘 때에는 반드시 디스크 I/O 과부하(DB 성능 마비) 로 이어질 위험이 있으므로, 권한 설정(ACL) 으로 와일드카드 사용을 철저히 막아야 하는 생존 런북이다.
