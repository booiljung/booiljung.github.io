14.9 백엔드 보안 및 접근 제어 적용

데이터를 모으는 것만큼 중요한 것이, 그 모아둔 금고를 지키는 일이다.
누군가 악의적으로 라우터에 침투하여 z_delete "**" 명령을 한 줄 치는 순간 페타바이트급 데이터가 먼지로 돌아가고, z_get "finance/**" 를 쳐서 회사 기밀을 10초 만에 빼내 가는 참사가 발생할 수 있다.

이 장에서는 스토리지 레벨(Storage Level)에서 이루어져야 하는 통제선, 암호화, 그리고 악의적인 와일드카드 질의를 무참히 박살 내는 보안 런북(Runbook)을 전개한다. 이 보안선(Line of Defense)을 구축하지 않는다면 시작조차 하지 마라.

1. 스토리지 마운트 포인트에 대한 권한 제어(ACL) 설정

스토리지 엔진(Backend)은 눈이 멀어있다. 누군가 인증서(Certificate) 없이 들어와서 Delete 패킷을 날려도 망에 보이면 그냥 지워버린다. 따라서 라우터 본체 단에서 “이 볼륨에는 누가 접근할 수 있는가“를 검문해야 한다.

1.0.1 [Runbook] 라우터 레벨의 절대적 권한 통제 (ACL)

Zenoh 라우터 기동 설정 파일에서, 백엔드 마운트 이전에 미리 권한(Access Control List) 계층을 덮어 씌운다.

1. 읽기-쓰기 분할 통제
일반 센서는 쓰기(Put)만 가능해야 하고, 모니터링 데몬은 읽기(Get)만 가능해야 한다. 절대 Delete 권한을 모두에게 주지 마라.

// zenohd.json5
plugins: {
  access_control: {
    // 1. 센서 권한: 오직 넣을 수만 있다(Put)
    roles: {
      "sensors": {
        allow: ["Put:robot/sensor/**"]
      },
      // 2. 관제탑 권한: 볼 수만 있다(Get)
      "monitor": {
        allow: ["Query:robot/sensor/**"]  
      },
      // 3. 관리자 권한: 삭제 가능
      "admin": {
        allow: ["Put:**", "Query:**", "Delete:**"]
      }
    }
  }
}

2. 스토리지 방패막의 완성
이 ACL 이 켜지는 순간, 라우터는 악성 클라이언트가 보낸 z_delete "robot/sensor/**" 패킷 자체를 스토리지 플러그인에게 전달하지 않고 네트워크 앞단에서 즉시 폐기(Drop) 해버린다. 백엔드(DB) 엔진은 자신이 공격받았다는 사실조차 모르게 하는 가장 우아한 아키텍처다.

2. 데이터 암호화(Data-at-Rest) 기법 및 암호화 키 관리

도둑이 라우터를 해킹하지 못하자, 엣지 기기(라즈베리 파이)의 전원을 끄고 SD 카드를 훔쳐갔다. 그 안에 담긴 RocksDB 폴더나 SQLite 파일을 열어보면 회사 기밀이 고스란히 담겨 있다.

2.0.1 [인스펙션] 저장소 절대 암호화 (Encryption-at-Rest) 전술

Zenoh 코어는 네트워크 위를 날아다니는 데이터(Data-in-Transit)를 TLS로 암호화할 수는 있지만, 디스크에 누워있는 데이터(Data-at-Rest)를 암호화해 주지는 않는다.

1. OS 레벨 암호화 (fscrypt / LUKS 활용)
가장 강력하고 빠른 런북이다.
Zenoh 백엔드가 마운트될 디렉토리(/var/lib/zenoh/rocksdb) 자체를 리눅스의 하드웨어 가속 암호화 파일시스템(LUKS) 파티션으로 묶어라.
로봇 부팅 시 보안 모듈(TPM/HSM)이 잠금을 풀기 전에는 디스크 자체가 쓰레기 값(Ciphertext)으로 보인다.

2. 애플리케이션 레벨 암호화 (Publisher-side Encryption)

  • 로봇(발신자)이 Zenoh 라우터로 데이터를 Put 쏘기 직전에, 센서 코드 단에서 페이로드(Payload)를 AES-256GCM 으로 구워버린다.
  • Zenoh 라우터 스토리지 백엔드는 이 데이터가 영상인지 암호문인지 모르고 일단 디스크에 거대한 바이너리 덩어리(BLOB)로 박는다(Persist).
  • 관제 센터의 Get 쿼리 결과로 그 바이너리 덩어리가 날아오면 관제 센터의 마스터 키(Master Key)로 즉석 복호화한다.
  • 이 전술의 장점: 스토리지 서버가 통째로 털려도, 파일 시스템 전문가가 와도 암호 단어를 풀 방법이 절대 존재하지 않는다. 진정한 제로-트러스트(Zero-Trust) 저장소다.

3. 악의적 쿼리(Query Injection) 방어 및 데이터 유출 방지 전략

관제탑에 접속 권한을 얻은 직원이 z_get "**" 이라는, 단 두 글자의 글로벌 와일드카드 쿼리를 날렸다.
전 세계의 Zenoh 라우터와 100대의 스토리지 서버는 동시에 이 쿼리에 응답하기 위해 디스크를 미친 듯이 긁기 시작하고, 100기가바이트의 배열이 망을 타며 시스템 전체가 마비(DDoS)되어 뻗어버렸다.

3.0.1 [Runbook] 심도 제한(Depth Limitation) 방어 전술

단순한 데이터 조회의 영역을 넘어, 스토리지에 가해지는 로드(Load) 자체를 시스템 권한으로 찍어눌러야 한다.

1. 와일드카드 격리 및 제한 조치
ACL 시스템 안에서 와일드카드의 깊이나 범위를 강제로 박탈하라.

// zenohd.json5
plugins: {
  access_control: {
    roles: {
      "junior_operator": {
        // 특정 로봇 단일값 조회는 허용하되,
        allow: ["Query:factory/robot_*/*"],
        // 전체 조회를 뜻하는 ** 는 접근 제어 레벨에서 하드 블로킹 해버린다!
        deny: ["Query:**", "Query:factory/robot_*/**"]
      }
    }
  }
}

2. 스토리지의 방어적 Query 제한(Pagination 도입)
커스텀 스토리지 플러그인을 개발할 경우, Get 트레이트(Trait) 내부 구현 시 한 번에 100개 이상의 Reply 가 잡힐 경우 앞의 100개만 던지고 쿼리를 강제 종료(Truncate) 해버리는 방어 로직을 하드코딩하라.
아키텍처 선에서 “인간의 실수“와 “악의적 자원 고갈 공격“을 물리적으로 막아낼 수 있는 유일한 마지노선이다.
