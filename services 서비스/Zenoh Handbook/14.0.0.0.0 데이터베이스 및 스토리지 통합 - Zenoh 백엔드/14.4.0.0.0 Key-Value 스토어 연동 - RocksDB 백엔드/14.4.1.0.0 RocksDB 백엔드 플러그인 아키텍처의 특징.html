14.4.1 RocksDB 백엔드 플러그인 아키텍처의 특징

왜 수많은 DB 를 두고 하필 RocksDB 인가? Zenoh 철학과 완벽하게 맞아떨어지는 이빨 때문이다.

0.1 [인스펙션] 임베디드(Embedded) 극한 튜닝의 이유

1. 네트워크 오버헤드의 증발
Redis 나 PostgreSQL 은 내 컴퓨터에 깔려 있어도 “TCP/IP 루프백 채널(127.0.0.1)” 을 타야 한다. 포트를 열고 소켓을 붙여야 데이터가 들어간다.
RocksDB 는 로컬 라이브러리(임베디드 엔진) 다. Zenoh 라우터 데몬 프로세스 안에서 동일한 메모리 포인터를 공유하며 돌아간다. TCP 소켓의 벽이 없기 때문에 오직 디스크 I/O 속도만이 한계다.

2. Key-Value 구조와 Zenoh 토픽의 1:1 완벽 매핑

  • Zenoh 통신: 토픽(robot/sensor1), 페이로드([0x10, 0x11])
  • RocksDB 구조: Key(robot/sensor1), Value([0x10, 0x11])
    스키마(테이블) 변환 같은 쓸데없는 오버헤드 없이, 백엔드로 넘어온 바이트 블록이 그 모양 그대로 하드디스크의 트리(LSM Tree) 에 박혀버린다.
