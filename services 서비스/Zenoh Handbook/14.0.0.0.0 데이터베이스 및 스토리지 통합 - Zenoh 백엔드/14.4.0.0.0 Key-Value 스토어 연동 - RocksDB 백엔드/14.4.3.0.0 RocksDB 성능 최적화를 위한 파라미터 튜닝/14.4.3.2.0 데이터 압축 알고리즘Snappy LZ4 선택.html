14.4.3.2 데이터 압축 알고리즘(Snappy, LZ4) 선택

10일간 주행한 블랙박스 데이터가 로봇의 64GB SD 카드를 꽉 채웠다. 우린 디스크 용량과 CPU 연산력 사이의 악마적 거래를 승인해야 한다.

0.1 [인스펙션] 실시간 압축 전술

RocksDB 는 디스크에 파일을 구워버릴 때(.sst 생성 시점), 즉석에서 압축 엔진을 돌릴 수 있다.

1. LZ4 / Snappy 의 강제 옵션

storages: {
  robot_rocks: {
    // ...
    compression: "lz4" // 무적의 실시간 압축기
  }
}
  • ZSTD: 압축률은 최고지만, CPU 코어가 미친 듯이 돌아간다. 전력이 중요한 로봇에선 자살 행위다.
  • LZ4: 압축률은 조금 낮아도, 압축/해제 속도가 RAM 복사 속도와 비견될 만큼 무식하게 빠르다. 로보틱스와 가장 궁합이 좋은 엔진이다.

2. 아키텍트의 승리 선언
LZ4 를 켜는 순간, 카메라 영상이나 JSON 배열의 디스크 쓰기 용량이 30% 이하로 쪼그라든다. 하드디스크 쓰기 량(I/O) 이 30% 로 줄었다는 건, 곧 쓰기 딜레이(Latency) 마저 압도적으로 단축되었음을 의미한다.
CPU 코어를 조금 더 태워서 극한의 디스크 성능 이득을 챙기는, 엣지 컴퓨팅의 교과서적인 자원 교환 법칙이다.
