14.8 사용자 정의(Custom) 백엔드 플러그인 개발

세상에는 RocksDB나 InfluxDB만 존재하는 것이 아니다. 여러분의 회사에는 국장에서만 쓰이는 기괴한 형태의 사내 자체 파일 시스템이 있을 수도 있고, 데이터를 무조건 AWS S3(클라우드 객체 저장소)에 밀어 넣어야 하는 특수한 보안 규정이 있을 수도 있다.

Zenoh가 기본으로 제공하지 않는 데이터베이스나 스토리지를 연동하고 싶다면?
고개를 숙이고 C++로 이기종 프록시 브릿지를 짜는 어리석은 짓을 멈춰라. Zenoh는 당신이 짠 Rust 클래스 뭉치를 데몬(Daemon)의 심장에 플러그인 형태로 수술해 넣을 수 있는 **궁극의 확장성 인터페이스(Extensibility Interface)**를 제공한다. 본 장에서는 나만의 커스텀 백엔드(Custom Backend)를 개발하여 Zenoh의 한계를 부수는 런북(Runbook)을 전개한다.

1. 커스텀 백엔드 개발 환경 세팅 (Rust 중심)

백엔드 플러그인은 파이썬(Python) 같은 스크립트 언어로 짤 수 없다. 라우터와 1밀리초의 오차도 없이 동일한 메모리 포인터를 주고받아야 하므로 오직 Rust로만 빚어내야 한다.

1.0.1 [Runbook] 플러그인 뼈대(Skeleton) 강제 주입

당신의 코드는 독립된 실행 파일(Binary)이 아니다. 라우터가 삼키기 좋은 “동적 라이브러리(Dynamic Library)” 형태로 조각해야 한다.

1. Cargo.toml 파괴적 설정
보통의 애플리케이션 컴파일 규칙을 버려라. cdylib 패러다임으로 전환하라.

[package]
name = "zenoh-backend-mycustom"
version = "0.1.0"
edition = "2021"

[lib]
## 데몬이 로드할 수 있는 C-호환 동적 라이브러리(.so)로 강제 빌드!
crate-type = ["cdylib"]

[dependencies]
zenoh-plugin-trait = "0.1"       # 필수: 플러그인 등록 매크로
zenoh-backend-traits = "0.1"     # 필수: 스토리지 뼈대 트레이트
tokio = { version = "1", features = ["full"] } # 비동기 런타임

2. 생명 불어넣기: 매크로(Macro) 등록
당신의 코드가 데몬에게 “나 스토리지 플러그인이야!” 라고 소리치려면, Rust 코드 최상단에 마법의 매크로를 찍어야 한다.

use zenoh_plugin_trait::{export_plugin, Plugin};

// 백엔드 구조체 선언
struct MyCustomBackend;

// 데몬이 기동될 때 이 매크로를 보고 코드를 낚아챈다!
export_plugin!(MyCustomBackend);

여기까지 세팅했다면, 데몬이라는 숙주의 몸에 기생할 기생수(Plugin)의 완벽한 유전자(DNA) 뼈대가 완성된 것이다.

2. Backend Trait 및 Storage Trait의 구조와 구현

뼈대를 만들었다면 이제 라우터 데몬이 요구하는 잔혹한 인터페이스 계약(Contract)을 이행해야 한다.
커스텀 스토리지를 짜려면 당신은 무조건 **두 개의 인터페이스(Trait)**를 강제로 구현해야 한다.

2.0.1 [인스펙션] 투 트랙(Two-Track) 지휘 체계의 구조

1. Backend Trait: 공장장 (Factory)

  • 이 클래스는 데이터를 저장하지 않는다. 단지 JSON 설정 파일(zenohd.json5)을 읽어내서, “자신 밑에서 일할 일꾼(Storage)“을 생성하는 역할만 수행한다.
  • 기능: new() (백엔드 기동), start() (라우터 런타임 진입), new_storage() (볼륨 마운팅 시점의 일꾼 창조).

2. Storage Trait: 실무자 (Worker)

  • 이 클래스가 진짜 피 터지게 데이터를 읽고 쓰는 놈이다.
  • 기능: put(), delete(), get() 등.
// 1. 공장장(Backend) 구현
#[async_trait]
impl Backend for MyCustomBackend {
    async fn new_storage(&self, admin_node: AdminNode) -> Arc<dyn Storage> {
        // 라우터가 "DB 좀 뚫어줘" 라고 명령하면, 
        // 실무자(MyStorage) 구조체를 만들어서 던져준다!
        Arc::new(MyStorage { ... })
    }
}

// 2. 일꾼(Storage) 구현
#[async_trait]
impl Storage for MyStorage {
    // 라우터가 Put 메시지를 받으면 이 함수를 때린다!
    async fn put(&self, key: &KeyExpr, payload: &ZBytes) -> ZResult<()> {
        // DB에 진짜로 쓰는 척 하는 코드
        println!("디스크에 저장 중... Key: {}", key);
        Ok(())
    }

    // 누군가 Get을 때리면 라우터가 이 함수를 호출한다!
    async fn get(&self, key: &KeyExpr) -> ZResult<Vec<Sample>> {
        // 하드디스크를 뒤져서 찾아주는 척 하는 코드
        Ok(vec![])
    }
}

이 두 개의 트레이트가 톱니바퀴처럼 맞물려(Interlocking), “데몬 로딩” \rightarrow “하위 볼륨(Volume) 생성” \rightarrow “데이터 입출력” 의 거대한 라이프사이클을 돌린다.

3. 플러그인 컴파일 및 공유 라이브러리(.so, .dll) 동적 로딩 매커니즘

Rust 코드를 다 짰다면, 이제 이것을 쇳덩어리(Binary)로 구워서 라우터에 먹여야 한다(Feeding).

3.0.1 [Runbook] 핫-로드(Hot-Load) 외과 수술 전술

1. 릴리즈(Release) 빌드 사수
로보틱스 런타임에서 디버그(Debug) 모드 플러그인은 성능을 50배 하락시키는 바이러스다. 무조건 릴리즈 모드로 강제 컴파일하라.

cargo build --release

빌드가 끝나면 target/release/ 폴더 아래에 libzenoh_backend_mycustom.so (윈도우는 .dll) 라는 파일이 떨어진다.

2. 라우터 주입 규칙 (Injection)
Zenoh 라우터는 기동될 때 딱 2가지 장소에서만 플러그인을 찾는다.

  • 자신(zenohd)이 켜진 바로 그 폴더.
  • 사용자가 지정한 특별한 라이브러리 경로 폴더(예: ~/.zenoh/lib).

만든 .so 파일을 폴더에 던져놓고 라우터를 켜라.

zenohd -c my_custom_config.json5

라우터가 부팅되면서 “Oh, I found mycustom backend!” 하는 로그를 띄우면 수술 성공이다. 시스템 본체 코드(Zenoh Core)를 단 한 줄도 재컴파일하지 않고도, 커스텀 DB 드라이버를 런타임(Runtime)에 강제 주입해 버린 것이다.

4. 클라우드 스토리지(AWS S3, Azure Blob)를 연동하는 가상 스토리지 구축 실습

지구 반대편에 있는 가장 거대하며 가장 느린 스토리지, 버킷(Bucket, AWS S3)을 엣지의 하드디스크인 것처럼 속여보자.

4.0.1 [인스펙션] 추상화 브릿징(Bridging) 모의 런북

Zenoh 의 커스텀 Storage Trait 에 AWS S3 SDK 를 녹여 넣는다.

1. Storage::put() 오버라이딩: 구름 속으로 던지기
강력한 비동기 스로틀링(Throttling)을 적용해야 한다.
데이터가 100건 들어왔다고 S3 API 를 100번 때리면 당장 아마존에서 요금 폭탄과 API 블로킹(HTTP 429)이 날아온다. 커스텀 스토리지 구현 시, 데이터를 RAM 버퍼에 머금고 있다가 10초에 한 번씩 S3 put_object 로 거대한 덤프를 쏘는 배치(Batch Processing) 로직을 짜야 한다.

// 개념적 구현(Conceptual Implementation)
async fn put(&self, key: &KeyExpr, payload: &ZBytes) {
    let s3_path = format!("s3://my-bucket/{}", key);
    // 비동기 S3 업로드 (직접 짜 넣은 S3 SDK 코드)
    s3_client.put_object().key(s3_path).body(payload).send().await;
}

2. Storage::get() 오버라이딩: 구름 낚시질
관제 센터가 z_get "AWS_S3/**" 를 호출하면, 이 커스텀 스토리지 플러그인은 즉각 S3에 list_objects 및 get_object API 를 날려 데이터를 파일로 당겨온다(Pull). 그러고는 이 데이터를 Zenoh 의 Reply 양식에 맞게 슬쩍 포장해서 관제 센터에 반환한다.

사용자는 S3 토큰이 뭔지도, API가 뭔지도 모른다. 그저 z_get "robot/s3_storage_area/*" 라고 치면 S3에서 데이터가 쏟아져 나온다. 스토리지 백엔드 플러그인 런북이 가져다주는 극강의 망 추상화(Network Abstraction)가 이룩어낸 걸작이다.
