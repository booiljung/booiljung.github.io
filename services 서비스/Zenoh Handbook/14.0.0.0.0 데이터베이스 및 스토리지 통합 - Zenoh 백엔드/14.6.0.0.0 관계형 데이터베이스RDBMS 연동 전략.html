14.6 관계형 데이터베이스(RDBMS) 연동 전략

빅데이터 시대라지만, 공장의 깐깐한 설비 정보, 로봇의 소유주(Owner) 식별자, 결제 정보 등은 여전히 엄격한 스키마(Schema) 구조를 갖춘 “안전한 금고(RDBMS)” 에 보관되어야 한다.

아무 형태나 때려 넣는 Zenoh 의 자유분방한 Pub/Sub 을, 딱딱한 테이블(Table) 로 조각하는 융합 아키텍처 런북을 살펴본다.

1. SQLite 백엔드: 엣지 컴퓨팅을 위한 독립형 로컬 DB 구축

오프라인 무인 자판기 안에 PostgreSQL 을 깔 수는 없다. 하지만 텍스트 파일(FS) 로 결제를 관리하기엔 무결성이 아쉽다.

1.0.1 [Runbook] 무정지 인-프로세스 SQL 엔진 구축

네트워크 서버 없이, Zenoh 라우터 하나로 로컬 SQL 파일 엔진을 굴린다.

1. SQLite 플러그인 로드

storages: {
  vending_db: {
    key_expr: "vending/001/payment/**",
    backend: "sqlite",
    // 엣지 하드디스크의 단일 파일로 모든 테이블을 묶어버린다!
    file: "/var/lib/zenoh/vending.sqlite"
  }
}

2. 텍스트 스키마의 묵시적 치환

  • z_put "vending/001/payment/card_id" "1234"
  • SQLite 백엔드는 .sqlite 파일의 기본 zenoh_storage 테이블 안에, key=card_id, value="1234" 형태로 레코드를 Insert/Update 친다.
    SQL 쿼리문 하나 작성하지 않고도 엣지 디바이스 위에 완벽한 트랜잭션 보장(ACID) 과 동시성 락(Lock) 기능을 갖춘 미니 금고를 구축하는 패턴이다.

2. PostgreSQL / MariaDB 연동을 위한 브릿지 및 백엔드 구성 방안

클라우드 중앙 관제 센터의 으리으리한 거대 SQL 데이터베이스. 프라이머리 키(PK) 와 외래 키(FK) 로 엮인 철창 안에 어떻게 데이터를 집어넣을 것인가?

2.0.1 [인스펙션] 맞춤형 SQL 브릿징 통제 역학

Zenoh 생태계는 MySQL 이나 PostgreSQL 전용 공식 C/Rust 플러그인을 기본으로 지원하지는 않는다. RDBMS 는 스키마 설계가 회사마다 천차만별이어서 “공통 모듈” 로 묶는 시도 자체가 실패하기 때문이다.

1. 파이썬 마이크로-브릿지 (Micro-Bridge) 구축
가장 고전적이고 안정적인 방식은, 관제용 DB 서버 옆에 조그만 파이썬 스크립트 기반의 Zenoh Subscriber 를 데몬으로 띄우는 것이다.

import zenoh
import psycopg2 # PostgreSQL 드라이버

z = zenoh.open()
conn = psycopg2.connect("dbname=robot user=admin")

def on_robot_status(sample):
    # 도착한 JSON 쪼개기
    data = json.loads(sample.payload.decode('utf-8'))
    # SQL 쿼리 매핑 및 주입!
    cur = conn.cursor()
    cur.execute("INSERT INTO status (robot_id, battery) VALUES (%s, %s)", 
                (data['id'], data['battery']))
    conn.commit()

z.declare_subscriber("robots/status/**", on_robot_status)

이 방법은 단순한 브릿지를 넘어서(Decoupling), 로봇 측은 무지성으로 JSON 만 던지게 두고 중앙 서버에서 스키마에 맞게 마음대로 파싱을 지휘할 수 있는 아키텍트의 재량권을 보장해 준다.

3. NoSQL 형태의 Zenoh 데이터를 RDBMS 스키마에 매핑하는 기법

“계층형 키(Key) 와 덩어리 데이터(Value)”
이 두 개뿐인 NoSQL 형 트리 구조를 억지로 2차원 표(RDBMS) 로 납작하게 눌렀을 때 벌어지는 참상과 극복법이다.

3.0.1 [Runbook] RDBMS 스키마 설계 전략

1. 맹목형(Blind) EAV 패턴

  • 테이블 컬럼을 단 3개만 판다: [ID, Topic(Key), Payload(Value)]
  • 모든 브릿지는 데이터를 받으면 무조건 이 테이블의 행(Row) 으로 추가한다.
  • 아키텍트 경고: 쓰기는 엄청나게 빠르지만, 나중에 “속도가 10 이상인 로봇” 을 쿼리할 때 Full-Table 스캔이 벌어져 DB 가 죽는다. 임시 보관용으로만 써라.

2. JSON/B-Tree 컬럼 주입 전술
현대적인 PostgreSQL 이나 최신 MariaDB 의 필살기다.

  • 테이블 컬럼: [robot_id], [timestamp], [data_jsonb]
  • 브릿지는 토픽 경로(robot_1) 를 갈라서 robot_id 에 넣고, Payload (JSON 통짜 덩어리) 를 SQL jsonb 컬럼에 밀어 넣는다.
  • 이렇게 해두면, 수십 개의 센서 속성을 일일이 컬럼으로 빼지 않아도, PostgreSQL 서버 내부 인덱스를 태워 초고속으로 WHERE data_jsonb->>'temp' > 30 같은 쿼리를 때려 박을 수 있다. RDBMS 의 무결성과 NoSQL 의 유연함을 한 몸에 가져가는 모범 답안이다.

4. 트랜잭션(Transaction) 처리 및 데이터 무결성 확보 방안

분산 네트워크 망에서는 로봇 A 의 결제 신호가 두 번 날아올 수도 있고, 아예 안 올 수도 있다. 이것을 SQL 에 그냥 박으면 회사 계좌에 빵꾸가 난다.

4.0.1 [인스펙션] 분산 트랜잭션 극복 런북

Zenoh 자체는 Pub/Sub 메신저이기에 2PC(Two-Phase Commit) 같은 분산 트랜잭션 롤백(Rollback) 기능이 없다.

1. 데이터의 멱등성(Idempotency) 사수
브릿지(Python 코드) 단에서 데이터를 SQL 에 넣을 때, INSERT 만 쳐서는 안 된다. 항상 UPSERT (있으면 덮어쓰고 없으면 추가) 를 쳐라.
방금 들어온 패킷이 네트워크 혼잡 때문에 어제 들어온 패킷이랑 순서가 뒤섞였더라도, 무작정 더하지 말고 “센서에 찍힌 타임스탬프를 기준으로, 더 최신 데이터면 치환하고 과거 값이면 버려라” 라는 로직을 브릿지 프로그래머가 강제로 넣어야만 한다.

2. 덤프 타이밍 동기화
초당 1만 개의 트랜잭션을 RDBMS 가 받아줄 리 없다.
브릿지는 Zenoh 로 들어오는 1만 개의 메시지를 RAM 에 1초 동안 구겨 넣고(Buffer), 정확히 1초마다 SQL DB 에 향해 한 방의 거대한 BULK INSERT (Transaction Commit) 를 때려야 한다. 무결성은 지키되 속도는 NoSQL 에 비견되도록 백엔드 레이어를 지휘하는 최고의 예술이다.
