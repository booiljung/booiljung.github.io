14.7 분산 스토리지 및 고가용성(High Availability, HA) 구성

하나의 데이터베이스(Database)에 모든 데이터를 밀어 넣는 행위는 시스템 파멸의 지름길이다. 하드 디스크(Hard Disk)가 고장 나거나 라우터(Router)가 죽는 순간, 로봇과 서버가 만들어낸 수천 시간 분량의 피 같은 데이터가 완전히 증발해 버린다.

Zenoh 생태계에서 스토리지를 단일 장애점(Single Point of Failure, SPOF)으로 두는 것은 끔찍한 설계다. 본 장에서는 데이터를 여러 대의 컴퓨터, 엣지(Edge), 클라우드(Cloud)에 갈기갈기 찢어서 복제(Replication)하고, 서버 한 대에 불이 나도 서비스가 1밀리초(ms)조차 멈추지 않는 **궁극의 분산 스토리지(Distributed Storage) 고가용성 아키텍처(Highly Available Architecture)**를 구축하는 방법을 런북(Runbook) 형태로 전개한다.

이 장의 설계를 마스터하면, 당신의 데이터 댐은 지구상의 어떤 네트워크 절단(Network Partition)에도 붕괴하지 않는 불사신(Immortal)으로 격상된다.

1. 다중 Zenoh 라우터 환경에서의 백엔드 데이터 일관성(Consistency)

여러 대의 Zenoh 라우터(Router) 각각에 스토리지를 물려 놓았다 치자.
로봇 A가 ’온도=25도’를 기록했는데, 라우터 1에는 도달하고 라우터 2에는 도달하지 않는다면? 데이터를 읽는 사람마다 다른 온도를 보게 되는 스플릿 브레인(Split-Brain) 지옥이 시작된다.

1.0.1 [인스펙션] 분산 시스템의 정합성(Consistency) 사수

Zenoh 백엔드는 기본적으로 최종 일관성(Eventual Consistency) 모델을 지향한다. 데이터 불일치(Inconsistency)를 방어하기 위한 근본적인 아키텍처 규칙을 인지하라.

1. 타임스탬프(Timestamp) 우선순위 병합 (LWW: Last-Write-Wins)

  • 로보틱스 환경에서 데이터가 순서를 뒤집어 도착하는 것은 일상이다.
  • Zenoh 백엔드 스토리지 플러그인들은 들어온 패킷(Packet)의 순서(Sequence)를 믿지 않는다. 패킷에 박혀있는 HLC (Hybrid Logical Clock) 기반의 메타데이터(Metadata) 타임스탬프를 확인한다.
  • 라우터 1과 라우터 2가 각각 다른 시점에 데이터를 받더라도, “가장 늦게 생성된(Latest Timestamp)” 데이터를 진실(Truth)로 받아들여 과거 데이터를 덮어쓴다. (이를 LWW 전략이라 부른다.)

    Note over Router2: T=10 < T=12 이므로<br/>강제 폐기(Drop)!
    Note over Router2: 데이터 일관성 유지됨

2. 아키텍트의 타임 락(Time-sync Lock)

  • 어떤 백엔드 저장소(예: InfluxDB)를 달더라도 z_put으로 데이터를 밀어넣을 때 발행 시간(timestamp: true)을 옵션에 박지 않으면, 이 LWW 매커니즘이 완전히 망가진다.
  • 라우터나 서버의 로컬 수신 시간을 기준으로 적재(Write)하도록 방치하는 순간, 네트워크 지연(Lag)이 곧 데이터 오염(Corruption)으로 직결된다. 반드시 발신 노드(Source)의 이벤트 타임(Event Time)을 기준으로 데이터가 적재되도록 강제하라.

2. 스토리지 간 데이터 복제(Replication) 및 미러링(Mirroring) 파이프라인

데이터를 한국 서버에도 남기고 미국 서버에도 남겨야 백업(Backup)이 완벽해진다.
Zenoh를 쓰면 Kafka 수준의 무거운 미러 메이커(MirrorMaker) 프로세스 없이도 데이터 라우팅(Routing) 자체만으로 무한 복제가 가능하다.

2.0.1 [Runbook] 허공을 가르는 멀티스토어 미러링 전술

이것은 스토리지 플러그인 수준의 세팅이 아니다. 네트워크 토폴로지(Topology)를 꼬아서 만드는 본질적 데이터 플러드(Data Flood) 기법이다.

1. 쌍둥이 스토리지 동시 마운트

  • 가장 원시적이면서도, 가장 강력한 복제 전술.
  • 한국 AWS 서버와 미국 AWS 서버에 떠 있는 Zenoh 라우터 2대에 **“완전히 동일한 Key Expression”**을 관찰하는 별개의 스토리지를 마운트한다.
// 한국 서버 라우터 (zenohd.json5)
storages: {
  seoul_backup: {
    key_expr: "global/telemetry/**",
    backend: "rocksdb",
    dir: "/data/seoul_rocks"
  }
}

// 미국 서버 라우터 (zenohd.json5)
storages: {
  us_backup: {
    key_expr: "global/telemetry/**",
    backend: "influxdb",  // 백엔드 종류마저 달라도 상관없다!
    // ...
  }
}

2. 복제의 발동 기전
로봇 1대가 일본에서 z_put "global/telemetry/gps" "LAT/LON" 을 치는 순간, 기적이 일어난다.

  • Zenoh 코어 라우팅 엔진은 전 세계를 뒤져서 global/telemetry/**를 듣고 있는(Sub) 모든 노드를 타겟으로 삼는다.
  • 이 Put 패킷은 네트워크 망의 최단 경로를 타고 두 갈래로 나뉘어 한국 라우터와 미국 라우터로 동시에 날아간다.
  • 한국 라우터는 RocksDB에 이진 바이너리로 박아넣고, 미국 라우터는 InfluxDB에 시계열 데이터로 박아넣는다.

단 하나의 데이터 베이스 미러링 설정 코드 없이, 그저 “같은 주제를 들으라“고 선언한 것만으로 이기종(Heterogeneous) 데이터베이스 간의 글로벌 데이터 복제(Global Data Replication)가 0.1ms의 오버헤드로 완성되어 버린다. Zenoh 생태계의 무서움이다.

3. 데이터 샤딩(Sharding)을 통한 대용량 데이터 분산 저장 기법

미러링(Replication)이 “똑같은 걸 여러 곳에 복사“하는 것이라면, 샤딩(Sharding)은 “너무 커서 한 곳에 다크 담을 수 없으니 쪼개서 담는” 것이다.
드론 1만 대의 라이브 비디오가 클라우드로 쏟아지면 아무리 좋은 SSD를 단 서버라도 한 대로는 버틸 수 없다.

3.0.1 [Runbook] URL 라우팅 기반의 데이터 분단(Cleaving) 전술

Zenoh에서는 데이터 샤딩 역시 복잡한 해시 링(Hash Ring) 알고리즘이나 프록시(Proxy) 서버 따위가 필요 없다. 오직 URL 기반의 Key Expression을 정밀 타격하여 파티셔닝(Partitioning)한다.

1. 저장소별 수비 범위(Coverage) 강제 분리
서버 3대(A, B, C)를 준비하고 각 라우터 스토리지에 다음과 같이 수비 범위를 제한한다.

// 서버 A 라우터 (ID 0~3000 전담)
storages: {
  shard_1: {
    key_expr: "fleet/drone/[0-3]**",  // 0~3으로 시작하는 ID만 수신
    backend: "rocksdb",
    dir: "/data/shard1"
  }
}

// 서버 B 라우터 (ID 4~6000 전담)
storages: {
  shard_2: {
    key_expr: "fleet/drone/[4-6]**",  // 4~6으로 시작하는 ID만 수신
    backend: "rocksdb",
    dir: "/data/shard2"
  }
}

// 서버 C 라우터 (ID 7~9999 전담)
storages: {
  shard_3: {
    key_expr: "fleet/drone/[7-9]**",  // 7~9로 시작하는 ID만 수신
    backend: "rocksdb",
    dir: "/data/shard3"
  }
}

2. 우아한 트래픽의 조각화 (Decentralized Load Balancing)

  • 드론 5500번이 접속하여 z_put "fleet/drone/5500/video" 를 쏘면, Zenoh 라우터는 첫 글자가 5인 것을 보고 정확히 “서버 B” 로만 트래픽을 던트린다. 서버 A와 C는 이 데이터를 받지도 않으므로 CPU를 1%도 소모하지 않는다.
  • 디스크 쓰기 부하(Write I/O)가 정확히 3대의 서버로 1/3씩 완벽하게 분산된다.

3. 스케일 아웃(Scale-out) 시의 분산 질의(Scatter-Gather)
샤딩이 되어있어도 데이터를 읽을(Get) 때는 고민할 필요가 없다.
관제탑에서 z_get "fleet/drone/**" 하나만 치면, Zenoh 코어 네트워킹 엔진이 자동으로 서버 A, B, C 3곳에 병렬로 Query를 날린 후, 그 결과를 0.01초 만에 하나의 배열(Array)로 합쳐주기(Reduce) 때문이다. 사용자는 데이터가 3곳에 찢어져 있다는 사실조차 알지 못한다.

4. 네트워크 단절(Network Partition) 시 데이터 큐잉 및 오프라인 복구 매커니즘

자율주행 자동차가 터널 안으로 진입했다. LTE 기지국과 클라우드 백엔드의 연결이 완전히 끊어졌다(Network Partition). 수집된 주행 데이터는 하늘로 증발하는가?

4.0.1 [인스펙션] 오프라인 좀비 모드 (Store-and-Forward) 극복 런북

Zenoh 의 파멸적인 스토리지 아키텍처는 이런 상황을 극복하기 위해 차량 내부(Edge)와 클라우드(Cloud)의 이중 스토리지 결합을 강제한다. 망이 끊겨 폭발하기 직전의 압력을 자체 큐(Queue)로 견뎌내는 댐을 건설한다.

1. 엣지 선행 임시 저장 (Local Edge Storage)

  • 차량(로봇) 내부의 Zenoh 라우터에 초경량 FS 혹은 RocksDB 스토리지를 “임시 보관소“로 마운트한다.
  • 로봇 센서는 무조건 자기 내부망 라우터 허공에 Put "telemetry/**" 를 쏜다.
  • 로컬 스토리지가 이 데이터를 모조리 차량용 하드디스크에 집어넣어 버린다.

2. 지연-안내 라우팅 (Delayed Forwarding / Sync)
터널을 빠져나와 5G 통신망이 재접속되면 벌어지는 일이다.

  • 단순 MQTT 라면 연결 직후 센서가 다시 실시간 값을 찍어 보낼 뿐, 터널 안에서 날아간 5분 치 데이터는 절대 복구할 수 없다.
  • Zenoh의 동기화 플러그인(zenoh-plugin-storage-manager 등) 또는 어플리케이션 계층은 네트워크 복구를 감지하는 즉시, 차량 하드디스크에 임시 보관된(큐잉된) 데이터들의 타임스탬프(HLC)를 조사한다.
  • “클라우드에 아직 안 올라간 데이터 블록“을 통째로 압축 덤프 떠서 클라우드 백엔드로 벌크 푸시(Bulk Push)해버린다.
graph TD

    B -->|Persist| C[(Local RocksDB)]
    C -.->|Network Disconnected| C
    B -->|Network Restored| D(Cloud Zenoh Router)
    C -->|Batch Sync| D
    D --> E[(Cloud InfluxDB)]

네트워크가 살아있을 때는 클라우드가 실시간으로 데이터를 주워 먹고, 네트워크가 끊기면 엣지가 로컬 DB에 데이터를 머금고 버틴다. 통신이 재개되면 잠자던 데이터들이 타임스탬프를 달고 좀비처럼 기어나와 클라우드 DB의 타임라인 빈 공간을 정확히 메꿔버리는, 로보틱스 오프라인 생존성의 극치다.
