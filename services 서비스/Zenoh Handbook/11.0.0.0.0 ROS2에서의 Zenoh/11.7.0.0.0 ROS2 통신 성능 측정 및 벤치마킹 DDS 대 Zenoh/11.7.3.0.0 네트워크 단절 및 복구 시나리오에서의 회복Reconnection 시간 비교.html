11.7.3 네트워크 단절 및 복구 시나리오에서의 회복(Reconnection) 시간 비교

공장 바닥은 전파의 암흑기다. 로봇이 철제 터널을 지나는 2초 동안 와이파이는 완벽하게 죽는다.
엔지니어가 통신 미들웨어를 평가할 때 가장 소름 끼치게 확인해야 할 지표는 속도가 아니다. “끊어진 뒤, 전파를 되찾았을 때 얼마나 빨리 제정신을 차리는가?” 다!

[잔혹한 시나리오: 강제 인터페이스 셧다운]

  1. 스크립트로 로봇의 wlan0 인터페이스를 ifconfig wlan0 down 때린다.
  2. 5초 뒤 강제로 ifconfig wlan0 up 시킨 뒤, ROS2 첫 번째 패킷이 도달할 때까지의 초시계를 잰다!

결과: DDS의 조울증 vs Zenoh의 무호흡 부활

  • DDS (FastDDS/Cyclone): 와이파이가 돌아오면 DDS는 처음부터 다시 지옥의 멀티캐스트 폭풍(Discovery)을 시작해야 한다. IP를 할당받고, 서로를 찾고, 리스트를 교환하고, 트리를 다시 짜는 데 최소 3초~10초의 경악스러운 연결 복구(Reconnection Penalty) 시간이 발생한다. 이 10초면 로봇은 이미 벽에 처박히고 남을 시간이다.
  • Zenoh (rmw_zenoh): 로컬 스카우팅이나 단일 클라우드 라우터(TCP 연결)로 묶인 상태다. 와이파이가 붙고 IP가 확인되는 즉시, 불과 몇백 밀리초(ms) 만에 TCP 세션을 꽂아버리며 곧바로 밀려있던 토픽을 빨아들인다!
    심지어 Zenoh의 Storage 옵션을 켜뒀다면, 터널 안에서 단절되었을 때 놓쳤던 서버의 제어 커맨드까지 라우터에서 캐시로 쭉 뽑아오며 “시공간의 공백“을 메워버린다!!
