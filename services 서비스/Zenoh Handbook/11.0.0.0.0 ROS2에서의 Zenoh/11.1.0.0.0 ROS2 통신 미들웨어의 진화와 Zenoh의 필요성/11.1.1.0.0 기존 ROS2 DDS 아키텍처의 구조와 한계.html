11.1.1 기존 ROS2 DDS 아키텍처의 구조와 한계

로보틱스의 표준으로 자리 잡은 ROS2는 네트워크 계층을 RMW(ROS Middleware)라는 껍데기로 감싼 뒤, 그 아래에 DDS(FastDDS, CycloneDDS 등)를 탑재하여 구동된다. 하지만 이 아키텍처는 유선 기가비트 이더넷(Gigabit Ethernet) 환경에서 만들어진 레거시(Legacy) 설계의 한계를 여실히 드러냈다.

0.1 [인스펙션] DDS의 태생적 족쇄 (The Architectural Fetters)

1. 막대한 디스커버리 오버헤드 (Discovery Storm)
DDS는 노드(Node)가 하나 켜질 때마다 네트워크 내의 모든 노드와 통신하며 악수(Handshake)를 한다. 로봇이 10대만 넘어가も, 백서라운드 네트워크 트래픽의 80%가 실제 센서 데이터가 아닌 그들만의 “인사이드 트래픽(Discovery XML)” 으로 채워지는 미친 오버헤드(Discovery Storm)가 발생한다.

2. 멀티캐스트(Multicast) 의존증
DDS는 로봇과 로봇, 서버와 로봇 간에 상대를 찾기 위해 무조건 UDP 멀티캐스트를 발사한다. 하지만 현대의 클라우드(AWS, Azure)망 구석 어디에도 멀티캐스트를 허용하는 스위치 장비는 없다. 결국 클라우드와 로봇 통신을 위해 VPN이나 더러운 릴레이(Relay) 서버를 구축해야만 했다.

이러한 DDS의 태생적 한계점을 뚫어내기 위해서라도, 차세대 ROS2 프로젝트들은 뼈대를 뜯어고쳐야 하는 필연적인 상황에 직면해 있었다.
