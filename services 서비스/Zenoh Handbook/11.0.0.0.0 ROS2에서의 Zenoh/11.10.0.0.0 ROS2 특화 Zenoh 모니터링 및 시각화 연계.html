11.10 ROS2 특화 Zenoh 모니터링 및 시각화 연계

눈먼 비행은 자살행위다. 통신 미들웨어를 교체했을 때 기존 ROS2 디버깅 툴들이 무용지물이 되는 현상을 어떻게 타개할 것인가?
이 장에서는 rqt_graph 를 대체하는 Zenoh 라우터 시각화 툴 체인 구축과, 차세대 관제 앱인 Foxglove Studio 와 Zenoh 를 직결시켜 초저지연 모니터링 대시보드를 구축하는 런북을 안내한다.

1. ROS2 노드 토폴로지 및 Zenoh 라우터 연결 상태 실시간 모니터링

DDS 환경에서는 ros2 node list 나 rqt_graph 로 로봇 내부망 토픽 관계도를 스캔했다. 그러나 Zenoh 브릿지나 rmw_zenoh 를 씌우면, 모든 패킷이 중앙의 라우터 하나로 모여버리기 때문에 기존 툴에서는 단순히 “브릿지 하나가 모든 걸 구독하는” 이상한 별 모양으로 압착되어 보인다. 이를 뚫어보려면 라우터 내부를 직접 까뒤집어야 한다.

1.0.1 [Runbook] 라우터 내시경 (Router Endoscopy) 전술

Zenoh 코어는 라우터 안에서 무슨 짓이 벌어지고 있는지 자체적인 제어 API (@/router/...) 를 통해 전부 불어댄다.

1. 터미널 스캐너 기동
가장 무식하게 라우터의 세션 테이블을 훔쳐보는 방법이다.

## 로봇의 Zenoh 라우터(localhost)에 접속해 있는 모든 세션 열람!
z_get -m get "tcp/localhost:7447" "@/router/*/session/*"

어떤 IP 의 노드가 붙어있고, 지연율이 몇인지 JSON 텍스트로 폭포수처럼 쏟아진다.

2. Zenoh Admin Web UI 가동
도무지 검은 화면을 보고 살 수 없다면 zenohd 기동 시 웹 대시보드 플러그인을 활성화하라.

zenohd --admin-http-port 8080

웹 브라우저로 http://로봇IP:8080 에 들어가면, 당신의 ROS2 노드들이 어떤 Key Expression (ros2/rt/...) 을 Publish 하고 Subscribe 하는지 실시간으로 그려진 지형도(Topology) 를 제공한다.

2. Zenoh 트래픽 시각화 도구를 활용한 ROS2 병목 구간 진단

관제 화면의 라이다가 점차 뚝뚝 끊기다가 서버가 뻗는다. ROS2 패킷 어딘가에서 병목이 생기면 Zenoh 터널 안에 패킷이 쌓이면서 메모리 누수가 터지는 현상이다.

2.0.1 [Runbook] 병목 부검(Autopsy) 전술

네트워크가 숨 막혀 죽기 직전의 징후를 알아채야 한다.

1. Zenoh Metrics 활성화
라우터는 사실 자신의 혈압과 맥박 값을 Prometheus 규격의 HTTP 엔드포인트로 노출할 수 있다.

## 시스템 모니터링 전용 노출
zenohd --rest-http-port 8000

http://localhost:8000/@/router/local/metrics 로 찌르면 전송 바이트 수, 드랍(Drop)된 프레임 수가 출력된다.

2. Prometheus + Grafana 시각화 통합
위의 주소를 Prometheus 타겟으로 물려 Grafana 대시보드에 띄운다.

  • 주시해야 할 핵심 지표: zenoh_router_drop_count
  • 이 값이 0 이 아니라 상승 중이라면, 어플리케이션(Ex: 클라우드 파이썬 스크립트) 쪽에서 Zenoh 가 쏴주는 데이터를 빨리 꺼내먹지 않아서 내부 큐(Queue)가 미어터져 “어쩔 수 없이 최신 데이터를 바닥에 버리고 있는” 중재(Throttling) 사태를 의미한다. CPU 클럭을 높이거나 전송 주파수(Hz)를 낮춰야만 한다!

3. RViz 및 Foxglove Studio와 Zenoh 네트워크의 원활한 통합 설정

Rviz2 나 최신 Web 관제툴 Foxglove Studio 는 어떻게 저 멀리 떨어진 로봇과 “지연 없이” 통신을 붙일 것인가.

3.0.1 [Runbook] 관제 모니터링 초결합(Hyper-link) 전술

1. 전통적인 Rviz2 아키텍처

  • [로봇] —(Zenoh 브릿지)— [AWS 터널] —(Zenoh 브릿지)— [당신의 노트북]
  • 당신의 노트북은 로컬 브릿지를 켜둔 채, 평소처럼 무심하게 터미널에서 rviz2 만 치면 끝난다. 그러면 Rviz 는 노트북 내부에 떠 있는 브릿지를 로봇으로 착각하고 빨대를 꽂아 그림을 그린다.

2. 차세대 Foxglove Studio 아키텍처 (브릿지 폭파 전술)
Foxglove Studio 는 웹 브라우저 기반이거나 일렉트론(Electron) 앱이다. 이 녀석에게 굳이 ROS2(Rviz2) 를 띄워 줄 필요 자체가 없다!
앞선 11.9.3 장에서 썼던 웹 브라우저 다이렉트 통신(Zenoh REST 또는 WebSocket 플러그인)을 이용한다.

  • AWS 에 떠 있는 라우터에 플러그인(WebSocket)을 결합해둔다.
  • Foxglove 의 “데이터 소스 커넥션” 세팅에서, 거추장스러운 rosbridge 가 아닌 Zenoh 규격을 바로 선택 후 AWS IP를 찍어준다.
    (Foxglove 는 최근 Zenoh 와의 파트너십을 통해 네이티브 커넥터를 정식 지원 중이다)

이 순간 노트북 메모리 1GB 를 거뜬히 먹던 Rviz2 와 ROS2 설치팩 전체를 삭제해도 된다.
오직 구글 크롬 브라우저 하나만 열면 TCP 소켓을 통해 우주정거장에 있는 로봇의 3D 공간을 실시간으로 렌더링하게 될 것이다.
