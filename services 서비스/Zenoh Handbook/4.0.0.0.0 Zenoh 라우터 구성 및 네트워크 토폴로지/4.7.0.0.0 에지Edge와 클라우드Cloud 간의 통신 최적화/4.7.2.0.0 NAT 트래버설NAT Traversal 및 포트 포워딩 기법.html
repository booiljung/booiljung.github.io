4.7.2 NAT 트래버설(NAT Traversal) 및 포트 포워딩 기법

로봇이 LTE/5G 모뎀이나 공장 와이파이에 연결되면 192.168.x.x 와 같은 사설(Private) IP를 받게 되며, 외부 인터넷(클라우드)으로 나갈 때 수많은 단말기들이 하나의 공인 IP(Public IP)로 뭉뚱그려져 나가는 기법을 NAT(Network Address Translation)라고 한다.

0.1 [트러블슈팅] NAT 테이블 만료 시 연결 두절 (Silent Drop)

로봇이 클라우드에 한 번 연결을 맺고, 10분 동안 아무런 온도 데이터를 보내지 않았다고 하자. 공유기의 NAT 테이블은 가혹해서 빈 깡통 소켓이라고 판단하여 중간에서 그 연결 튜브를 가위로 잘라버린다.
이후 클라우드 관제 대시보드 관리자가 로봇 조종 명령을 쐈는데 로봇이 반응을 안 하는 “침묵의 연결 장애“가 발생한다.

0.2 [Runbook] Keep-Alive 심폐 소생술

중간의 공유기(NAT)가 소켓을 회수하지 못하게 하려면 논리적으로 계속 헛기침(Keep-Alive)을 시켜야 한다.

  1. OS 커널 단 TCP KeepAlive (전역적 방어):
    Linux 서버의 내부 설정을 바꾼다. (도커 컨테이너에서도 Sysctl 권한 필요)
# 7200초나 되는 리눅스 기본 KeepAlive를 60초로 파격 단축 
sysctl -w net.ipv4.tcp_keepalive_time=60
sysctl -w net.ipv4.tcp_keepalive_intvl=10
sysctl -w net.ipv4.tcp_keepalive_probes=6
  1. Zenoh L4 설정 단 방어 (앱 종속 적용):
    zenohd.json5 및 클라이언트 Config에 세션 유지 핑(Ping) 간격을 수동으로 강제한다.
// 클라우드 라우터의 수신 타임아웃 방어막 설정
{
  transport: {
    // 10초 동안 핑이 없으면 파이프를 끊고 재연결 로직으로 유도
    link_rx_keepalive: 10000 
  }
}

NAT는 분산 시스템의 영원한 적이다. 에지는 쉴 틈 없이 자신의 생존을 증명하게(Ping) 만들거나 방금 서술한 KeepAlive 파라미터 간격을 네트워크 장비의 NAT TTL(Time-to-Live)보다 짧게 설정하는 것이 인프라의 알파이자 오메가다.
