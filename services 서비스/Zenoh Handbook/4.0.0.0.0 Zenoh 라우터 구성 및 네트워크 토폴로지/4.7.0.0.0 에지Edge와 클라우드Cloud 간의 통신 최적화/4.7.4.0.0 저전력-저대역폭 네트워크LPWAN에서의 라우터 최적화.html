4.7.4 저전력-저대역폭 네트워크(LPWAN)에서의 라우터 최적화

일반적인 5G 망이 아니라 스마트 농점의 LoRa, 선박의 위성 통신, 비닐하우스의 BLE 메시 망처럼 초당 몇 KB밖에 전송할 수 없는 열악한 대역폭(Constraints Network)에서는 어떻게 해야 할까?
라우터가 기본값으로 남발하는 제어 평면(Control Plane)의 패킷 낭비를 극단적으로 다이어트(Diet)시켜야만 망이 멈추지 않는다.

0.1 [Runbook] 저대역폭 생존을 위한 극단적 라우터 다이어트 전술

zenohd.json5 파일에서 기본으로 켜져 있는 편의성 기능(오토 디스커버리, 빠른 상태 동기화)을 모두 깎아(Drop) 버린다.

1. 주기적인 구독/발행 경로(Routing info) 동기화 시간 지연
새로운 로봇이 들어올 때마다 망 구조를 즉시 업데이트하려 들면 그게 전부 트래픽 비용(요금)이 된다.

{
  // 라우터 간에 서로의 라우팅 상태를 교환하는 주기(기본 1초)를 30초로 대폭 늘린다.
  // 응답성은 느려지지만 오버헤드 메타 트래픽이 90% 이상 절감된다!
  routing: {
    gossip_interval: 30000 
  }
}

2. 통신 버퍼(Batch) 압축 영입
데이터를 보낼 때 1바이트짜리를 100번 보내지 말고, 11바이트가 찰 때까지 라우터 메모리에 쥐고 있다가 한 덩어리로 묶어서 MTU(Maximum Transmission Unit) 낭비를 최소화한다.

{
  transport: {
    // 패킷 송신 전 버퍼링을 대기하는 강제 플러시(Flush) 대기 시간을 튜닝
    // 초지연성(Low-Latency)을 약간 희생하여 데이터 압축(Throughput) 한계돌파.
    batch_size: 16384 // 16KB가 찰 때까지 참았다가 발사
  }
}

이렇게 세팅된 라우터는 위성 통신비가 1MB당 수만 원씩 청구되는 해양 선박 로보틱스 관제에서, 비용 부서를 미소 짓게 만드는 진정한 “돈을 버는 아키텍처“로 군림하게 된다.
