4.7.1 방화벽(Firewall) 환경에서의 라우터 포트 정책

제아무리 훌륭한 zenohd.json5 파일도 클라우드 벤더(AWS EC2, GCP Compute Engine)의 Security Group 앞에서는 한낱 텍스트에 불과하다.

0.1 [Runbook] 클라우드 인바운드/아웃바운드 개방 원칙

클라우드 라우터는 전 세계 에지 장비가 뚫고 들어와야 하는 요새(Fortress)다.

  1. TCP 7447 (필수): 모든 연결의 근간이다. Inbound 0.0.0.0/0 개방.
  2. UDP 7447 (필수-QUIC 가속용): QUIC을 쓰려면 UDP를 열어야 한다. 클라우드 해킹 공격(UDP Amplification)이 두렵다면, 아웃바운드 포트를 특정 공장 IP 대역으로만 제한(Whitelist)하는 하이브리드 방화벽 정책을 인프라 팀에 요청하라.
  3. [안티 패턴] 멀티캐스트 방화벽 요청: AWS 기술 지원팀에 “UDP 224.0.0.225 열어주세요” 하지 마라. 거절당할 뿐만 아니라 네트워크 아키텍처 기본 지식을 의심받는다. 멀티캐스트 스카우팅은 물리 스위치가 통제되는 온프레미스(사내) 망의 전유물이다.

0.2 방화벽 타공의 비대칭성 (단방향 Connect 전술)

에지 공장 안의 방화벽(Inbound)은 절대 열 수 없다. (외부에서 들어오는 연결 불가)
따라서 연결의 화살표(Connect)는 반드시 “에지 ➔ 클라우드” 의 단방향(Outbound)으로 쏘아올려져야 한다.
일단 이렇게 한 방향으로 세션 튜브만 연결해 두면, 클라우드가 에지에게 역(Reverse)으로 데이터를 밀어내거나(Pull), 명령(RPC)을 내리는 데는 아무런 지장이 없다. TCP의 Full-Duplex 성질을 100% 활용하는 라우팅 메커니즘 덕분이다.
