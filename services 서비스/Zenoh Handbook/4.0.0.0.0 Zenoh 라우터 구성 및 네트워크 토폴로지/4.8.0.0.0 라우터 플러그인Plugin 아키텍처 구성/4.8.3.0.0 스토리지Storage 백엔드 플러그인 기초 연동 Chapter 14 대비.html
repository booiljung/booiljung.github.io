4.8.3 스토리지(Storage) 백엔드 플러그인 기초 연동 (Chapter 14 대비)

“통신이 안 되고 있을 때 발생한 데이터는 어디로 가나요?”
MQTT의 QoS 1/2를 묻는 단골 질문이다. Zenoh는 이를 통신 프로토콜 수준에서 하드코딩하지 않고, 저장소(Storage) 라는 개념으로 아키텍처를 추상화해버렸다.

0.1 [Runbook] 지능형 캐싱 공간 활성화

라우터가 지나가는 패킷을 흘려보내는 우체국이 아니라, 임시 창고(DB) 역할을 하도록 Storage Manager 플러그인을 로드해야 한다.

// zenohd.json5
{
  plugins: {
    storage_manager: {
      volumes: {
        // 백엔드로 메모리 공간(RAM)을 할당. 인스턴스가 뻗으면 휘발된다.
        memory: {}
      },
      storages: {
        // "로봇들의 배터리 데이터"가 오가면 무조건 메모리에 붙잡아(Cache) 둔다!
        my_battery_cache: {
          key_expr: "robot/*/battery",
          volume: "memory"
        }
      }
    }
  }
}

[아키텍트 시연: 캐싱의 위력]
위 설정을 켠 후 로봇(Publisher) 전원을 확 뽑아버려라.
그다음 클라우드에서 zenoh-cli --mode client -e tcp/라우터 GET "robot/1/battery" 를 쳐보라. 로봇은 죽었지만 라우터가 “아까 내가 캐시해 둔 55%가 마지막 값이야” 라며 1ms 만에 과거의 데이터를 꺼내준다.

이 플러그인의 한계(메모리 볼륨은 라우터 재부팅 시 삭제됨)를 돌파하고 영구 스토리지(InfluxDB, RocksDB)로 결착하는 초고도화 런북은 뒤이은 [Chapter 14] 에서 집중 포화한다.
