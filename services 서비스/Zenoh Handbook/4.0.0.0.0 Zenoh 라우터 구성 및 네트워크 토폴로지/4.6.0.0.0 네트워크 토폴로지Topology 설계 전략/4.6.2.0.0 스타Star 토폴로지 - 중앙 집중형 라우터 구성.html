4.6.2 스타(Star) 토폴로지 - 중앙 집중형 라우터 구성

MQTT 브로커나 REST API 서버 구조와 가장 유사한 멘탈 모델. 클라우드나 온프레미스(사내) 서버실에 하나의 거대한 zenohd 마스터 라우터를 띄우고, 모든 에지 로봇들이 오직 이 라우터만 조준(.connect([ 마스터 IP ]))하는 형태다.

0.1 [Runbook] 스타 토폴로지 운영 팁

장점과 시나리오
관리자가 단 1대의 서버 방화벽만 관리하면 되고, 로그(Log)나 트래픽 메트릭 수집이 라우터 하나에 집중되므로 디버깅이 가장 쉽다. 스마트 홈(Smart Home)의 IoT 전구들이나 센서들을 통제할 때 적합하다.

[트러블슈팅] OOM(Out of Memory) 폭우 방어막
1만 대의 클라이언트(Client)가 1대의 라우터(Router)에 붙는다. 라우터가 아무리 Rust로 짜였다지만 수만 개의 커넥션을 비동기로 버틸 수 있을까?
zenohd.json5 파일에서 동시성(Concurrency) 보호 차단막을 내려야 한다.

// [운영계 라우터 런북 방어벽]
{
  listen: {
    endpoints: ["tcp/0.0.0.0:7447"]
  },
  transport: {
    tcp: {
      // 10,000개의 소켓을 동시에 물기 위해 OS 커널 한계 해제 확인 必
      max_sessions: 20000,
      // 한 놈이 대역폭을 독식하여 백프레셔(Backpressure)를 막는 소켓 버퍼 튜닝
      recv_buffer_size: 1048576, 
      send_buffer_size: 1048576
    }
  }
}

주의사항: 단일 장애점(SPOF)이므로, K8s 환경에서라면 반드시 L4 로드밸런서 밑에 라우터 팟(Pod)을 Replica 2대 이상으로 띄우고 망을 병렬 분산시킬 방안(4.9장 참조)을 모색해야 한다.
