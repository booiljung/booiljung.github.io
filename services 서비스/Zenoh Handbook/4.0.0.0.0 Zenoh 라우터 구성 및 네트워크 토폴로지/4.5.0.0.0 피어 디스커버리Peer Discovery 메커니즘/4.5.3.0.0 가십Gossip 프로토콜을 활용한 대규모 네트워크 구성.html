4.5.3 가십(Gossip) 프로토콜을 활용한 대규모 네트워크 구성

멀티캐스트 탐색의 저주는 노드가 100대를 넘어가는 순간 10,000개가 넘는 탐색 패킷(N^2)이 공간을 지배한다는 점이다. 이를 해결하기 위해 거대 클러스터 망(ex. 데이터센터 간 연결)에서는 레이더 방사 대신 Gossip (소문 내기) 프로토콜을 백그라운드로 돌린다.

0.1 Gossip 프로토콜의 우아함

  1. A 노드가 C 노드의 존재를 알게 된다.
  2. A 노드가 B 노드와 주기적 통신(Heartbeat)을 할 때, 슬쩍 C의 명함을 담아 넘긴다. (“야, 저기 C라는 애도 새로 왔대.”)
  3. B 노드는 스스로 스카우트 레이더를 쏘지 않고도 C의 주소를 확보하게 된다. 이 소문은 바이러스처럼 N대의 서버로 삽시간에 전파된다.

0.2 [Runbook] 라우팅 테이블 파열 방어

이 Gossip 메커니즘 덕분에, 10대의 라우터가 서로 뒤엉켜 있어도 최단 거리 경로표(Spanning Tree)가 눈 깜짝할 사이에 동기화된다. 아키텍트는 그저 다음만 확인하면 된다.

  • 거대 망에서는 scouting.multicast.enabled = false 로 박아두고, 초기 시드 핑을 날릴 타겟(connect.endpoints) 1~2개만 잡아줘라. 나머지는 Gossip 스레드가 알아서 1,000대 규모의 망 지도를 클러스터 내부에 매핑시켜 줄 것이다.
