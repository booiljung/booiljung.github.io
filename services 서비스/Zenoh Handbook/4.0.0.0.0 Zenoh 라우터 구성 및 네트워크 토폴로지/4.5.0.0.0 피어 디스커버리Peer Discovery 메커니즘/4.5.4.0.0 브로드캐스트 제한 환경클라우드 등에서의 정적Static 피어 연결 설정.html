4.5.4 브로드캐스트 제한 환경(클라우드 등)에서의 정적(Static) 피어 연결 설정

AWS VPC, GCP, Azure 같은 퍼블릭 클라우드 인스턴스끼리는 무조건 L3(IP) 라우팅만 지원할 뿐, 태생적으로 L2 멀티캐스트/브로드캐스트 패킷을 통과시키지 않는다. (AWS 보안 정책 1원칙이다).
컨테이너화된 Kubernetes 브릿지 네트워크 환경도 마찬가지다.

여기서 자꾸 Peer 모드로 멀티캐스트를 발사하면서 “왜 클라우드 로봇 관제가 안 되냐“고 울부짖으면 곤란하다.

0.1 [Runbook] 클라우드망에서의 하드코딩(Static) 조준선 정렬

클라우드 척추(Backbone)를 세울 때는 자동 탐험(Discovery)이라는 마법을 깨끗이 잊고, 명확하고 폭력적인 1:1 조준(Unicast Connect)을 수행해야 한다.

1. 마스터 라우터 (AWS 도쿄 리전)

  • 가만히 앉아서 들어오는 요청만 받도록 무장한다. (스카우트 완전 차단)
// aws_master.json5
{
  listen: { endpoints: ["tcp/0.0.0.0:7447"] },
  scouting: { multicast: { enabled: false } }
}

2. 에지 라우터 (부산, 서울 공장)

  • 일꾼들은 오직 마스터의 고정 IP 도메인을 향해 1직선으로 파이프를 박아 넣는다.
// busan_edge.json5
{
  connect: {
    // VPC의 Elastic IP나 도메인(DNS)을 명확하게 타겟팅
    endpoints: ["tcp/aws-tokyo-router.myapp.com:7447"]
  },
  scouting: { multicast: { enabled: false } }
}

“운영계의 안정성은 설정 파일의 고정성(Hard-coded endpoints)에서 온다“는 인프라 금언을 잊지 마라.
