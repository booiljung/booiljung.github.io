4.1.1 기존 브로커(Broker) 방식과 Zenoh 라우터(Router) 방식의 차이점

엔터프라이즈 설계자들은 “Zenoh 라우터가 결국 MQTT 브로커랑 똑같은 중앙 서버 아니냐?“는 날 선 질문을 던진다. 아키텍트는 이 질문에 단호하게 선을 그을 수 있어야 한다.

0.1 브로커(Broker)의 독재와 병목 (MQTT, Kafka)

  • 독재성: 기존의 브로커 통신 방식에서는 나와 1미터 옆에 있는 장비와 통신할 때도 무조건 클라우드에 있는 브로커를 다녀와야(Hairpinning) 한다.
  • 블라인드 포워딩: 브로커 자신은 데이터 내용에 관심이 없다. 그저 A가 보내면 B에게 맹목적으로 밀어낼 뿐이다. 데이터의 중복이나 로컬 캐싱에 대한 지능(Intelligence)이 결여되어 있다.

0.2 Zenoh 라우터(Router)의 지능적인 분배 (Routing & Consolidation)

Zenoh 라우터 데몬(zenohd)은 단순한 우체국이 아니라 지능적인 물류 허브다.

  • 다중 홉(Multi-hop) 탈중앙화: 라우터 1번과 라우터 2번을 서로 엮어 거대한 분산 망으로 팽창시킬 수 있다. AWS 라우터가 죽어도, 로컬 공장에 띄워둔 라우터 망은 오프라인 상태로 로봇끼리의 통신을 계속 보장한다.
  • 가까운 곳에서 가져오기 (Data Locality): 내가 온도 데이터를 물어봤을 때, 10,000km 밖의 로봇까지 가서 물어보는 게 아니라 로컬 라우터가 자기가 캐시해 둔 5초 전 응답 값으로 방어(Consolidation)해 버린다.

0.3 [Runbook] 안티 패턴: 브로커식 중앙 집중의 배제

[위험한 아키텍처] : 1,000대의 로봇에 mode: client를 박고 AWS에 있는 단 1대의 zenohd로 쏘게 만들지 마라.
[모범 런북 아키텍처] : 로봇 100대 단위로 공장 내부에 에지(Edge) 라우터를 1대씩 배치하여 1차 데이터 융합(Aggregation)을 수행하고, 에지 라우터들끼리 클라우드 글로벌 라우터와 백본망(Backbone)으로 묶이는 Tree 구조를 강제해야 트래픽 버퍼링 크래시를 면할 수 있다.
