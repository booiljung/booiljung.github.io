4.1.4 분산 해시 테이블(DHT) 기반의 데이터 라우팅 개념

어떻게 수천 대의 라우터가 “누가 어떤 데이터를 필요로 하는지” 귀신같이 알아내 불필요한 데이터(Broadcast)를 타 채널에 낭비하지 않는 걸까? 이것이 가능케 하는 내장 엔진이 바로 **분산 해시 테이블(DHT)**과 연계된 구독 트리(Subscription Tree) 최적화다.

0.1 라우팅의 본질: 똑똑한 라우팅 테이블(Routing Table)

라우터는 멍청하게 모든 데이타를 돌리지 않는다. 자신이 거느린 클라이언트들이나 이웃 라우터들이 “나 robot/1/battery 데이터 받을게!“라고 구독(Subscribe) 을 선언하는 순간, 라우터 내부의 해시 트리 테이블에 거대한 지도가 그려진다.

  1. AWS의 라우터 A는 로컬망 라우터 B에게 안테나를 꽂고 있다.
  2. 당신이 스마트폰으로 A 라우터에 접속하여 구독 버튼("robot/busan/**")을 누른다.
  3. 라우터 A는 라우터 B에게 귓속말로 속삭인다. “나 부산 로봇 데이터 필요하니까 보내줘.”
  4. 부산 라우터 B에서 발생하는 데이터 중, 그 외의 쓰레기 데이터("robot/seoul/**")는 AWS 해저 케이블을 단 한 방울도 타지 못하고 라우터 B 선에서 튕겨버린다 (Drop).
  5. 오직 부산 데이터 5바이트만 네트워크를 건너뛴다.

0.2 [Runbook] 라우터 포워딩 루프(Routing Loop) 방지법

만약 인프라 엔지니어가 라우터 A, B, C를 삼각형으로 빙빙 돌게 엮어버린다면? 패킷 하나가 영원히 A->B->C->A 로 뺑뺑이를 도는 폭풍이 일어나는가?

Zenoh 데몬은 기본적으로 위상 정렬 기반의 다중-경로 검증(Spanning Tree & Loop Avoidance) 코어가 내장되어 있어, 어느 쪽 노드에 어떤 ID로 패킷이 발사되었는지 UUID 트랙킹을 통해 지능적으로 루프를 방어한다.
개발자가 직접 iptables로 타임 투 리브(TTL)를 만지작거릴 필요가 없다는 뜻이다. 그러나 불필요하게 망을 거미줄처럼 다중(Mesh) 링크로 도배하는 행위는 이 라우팅 계산의 메모리 부하(Overhead)를 야기하므로, 철저한 계층적 트리 구조(Edge -> Fog -> Cloud) 로 라우터 연결선을 정립하는 것이 아키텍트의 사명이다.
