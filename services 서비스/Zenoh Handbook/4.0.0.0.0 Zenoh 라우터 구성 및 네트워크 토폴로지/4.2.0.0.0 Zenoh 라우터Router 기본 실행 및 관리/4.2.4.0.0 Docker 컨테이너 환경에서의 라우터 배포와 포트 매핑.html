4.2.4 Docker 컨테이너 환경에서의 라우터 배포와 포트 매핑

최근에는 베어메탈 서버(Bare-metal)에 직접 데몬을 까는 짓(3.2장 방식)보다, Docker 구명정판 위에 zenohd를 띄워 k8s 클러스터로 편입시키는 클라우드 네이티브(Cloud Native) 방식이 절대 표준이다.
이때 주의해야 할 것은 Docker의 격리망(NAT/Bridge)이 Zenoh의 멀티캐스트 스카우팅 능력을 질식시킨다는 점이다.

0.1 [Runbook] 도커 네트워크 파괴 전술 (Network Host vs Port Forwarding)

방법 A: --network host (에지 장비/단일 서버용 권장)
가장 강력하고 빠른 치트키다. 도커의 가상 네트워크 벽을 박살 내고, 호스트 서버가 가진 랜카드(물리 NIC)를 컨테이너가 100% 직수령하게 만든다. P2P 탐색을 위한 UDP 스카우트 패킷이 단 한 톨의 필터링 없이 공유기로 방사된다.

# 로컬 공장 망에 붙어있는 스카우팅 로봇들을 빨아들일 수 있다.
docker run -d --name zenoh-router \
    --network host \
    -v /my/conf/zenohd.json5:/zenohd.json5:ro \
    eclipse/zenoh:latest -c /zenohd.json5

方法 B: --publish 명시적 포트 타공 (클라우드/K8s 용)
AWS, GCP 인스턴스 위에서는 어차피 인터넷 밖으로 멀티캐스트 방사(Scouting)가 통신사 망에서 막혀버리므로, 깔끔하게 클라이언트들이 쳐들어올 TCP 고속도로 구멍만 뚫어주면 된다.

docker run -d --name zenoh-router \
    -p 7447:7447/tcp \
    -p 7447:7447/udp \
    -v /my/conf/zenohd.json5:/zenohd.json5:ro \
    eclipse/zenoh:latest -c /zenohd.json5

0.2 [Docker Compose 런북] 볼륨 무결성 확보

스토리지 캐시 플러그인을 붙여 라우터를 DB처럼 쓰려 한다면, 컨테이너가 죽었다 살아나도 메모리가 유지되어야 한다. /data를 호스트 파일 시스템과 엮어야 피눈물을 흘리지 않는다.

version: "3.8"
services:
  zenoh-router:
    image: eclipse/zenoh:latest
    ports:
      - "7447:7447/tcp"
    volumes:
      - ./config.json5:/etc/zenohd.json5:ro
      - ./zenoh-data:/data # 여기서 데이터의 영속성(Persistence)이 보장된다!
    command: ["-c", "/etc/zenohd.json5"]
