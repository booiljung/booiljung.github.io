4.10.4 패킷 덤프 및 네트워크 병목 구간 기초 분석

“로그 상으로는 보냈다고 되어있는데 실제로는 데이터가 안 와요!”
모든 네트워크 계층 구조에서 절대 거짓말을 하지 않는 것은 오직 랜카드(NIC)에 꽂히고 빠지는 와이어 패킷(Wire Packet) 원본뿐이다.

0.1 [Runbook] Wireshark 및 tcpdump 포렌식 기법

Zenoh는 자체적인 통신 프로토콜 규격(Zenoh Protocol)을 따른다. 패킷을 까봤을 때 이게 쓰레기 문자(Encrypted)인지 진짜 Zenoh 메시지인지 기초 식별(Identify)을 할 줄 알아야 병목을 증명할 수 있다.

1. 서버 터미널에서의 생존 덤프 (tcpdump)

# 7447 포트로 오가는 모든 패킷을 헥사 덤프로 출력 (로봇과 라우터 통신 캡처)
sudo tcpdump -i any port 7447 -n -vvv -X

출력물(Hexadecimal) 안에서, ASCII 코드로 당신이 Publish 한 ZBytes 데이터(ex: 85%)나 Key 표현식(robot/1/battery) 평문이 흐르고 있다면 네트워크 라우팅 자체에는 전혀 문제가 없는 것이다. 받는 쪽(Subscriber) 애플리케이션의 메모리 콜백 스레드가 블로킹되어 데이터를 처리하지 못하고 있을 확률이 99%다.

2. Wireshark Dissector 연동 (고급)
더 전문적인 디버깅을 위해 이클립스 파운데이션(Eclipse Foundation)은 Wireshark에 얹어 쓸 수 있는 zenoh-lua-dissector 를 제공한다. 이를 Wireshark 플러그인 폴더에 들이부으면, Data, Declare, Scout 같은 Zenoh 프로토콜의 오프코드(Opcode)가 아름답게 시각화되어 분석 속도를 10배 끌어올려 준다.
