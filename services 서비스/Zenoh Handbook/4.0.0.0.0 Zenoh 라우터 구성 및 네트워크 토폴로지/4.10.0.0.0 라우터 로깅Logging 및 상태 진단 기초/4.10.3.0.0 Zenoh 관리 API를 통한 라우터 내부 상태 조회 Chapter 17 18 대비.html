4.10.3 Zenoh 관리 API를 통한 라우터 내부 상태 조회 (Chapter 17, 18 대비)

터미널의 스크롤 로그(Stdout)를 사람 눈으로 읽는 것은 1990년대 방식이다.
현대 클라우드 데브옵스 생태계(Prometheus, Grafana 등)는 라우터에게 “너 지금 세션 몇 개 물고 있어? 초당 전송량은 어때?” 하고 HTTP/REST로 찔러서 JSON 대답을 얻을 수 있는 포트(Admin API) 를 요구한다.

0.1 [Runbook] 마이크로커널 Admin API 인스펙션

라우터(zenohd)는 자기 내부의 해소된 스키마, 맺어진 세션 수, 토폴로지 맵 등을 @ (골뱅이) 문법을 통해 실시간 ZBytes 정보로 노출하고 있다.
CLI 툴이나 REST 플러그인(8000번 오픈 가정)을 통해 라우터의 민낯을 털어보자.

1. “현재 맺어진 모든 네트워크 이웃(이웃 라우터, Peer)의 맵을 그려내라.”

# 로컬 라우터의 물리적/논리적 경로 테이블 덤프 (디버깅의 끝판왕)
curl -X GET http://localhost:8000/@/router/local/network/links

2. “현재 로컬 라우터에 연결된 클라이언트(로봇) 세션 리스트 전체 확인”

# 지금 어떤 단말기가 내 포트에 붙어있는가?
curl -X GET http://localhost:8000/@/router/local/sessions

3. “라우터 자체의 메모리/CPU 생태 지표 파악”

# 라우터 버전, 가동 시간(Uptime), 빌드 아키텍처 추출
curl -X GET http://localhost:8000/@/router/local/info

이 백도어(@) API에서 추출된 JSON 데이터들을 가공해 Prometheus 익스포터 플러그인으로 쏴서, 관제실 모니터에 초현대적인 Grafana 네트워크 그래프를 그리는 기법은 뒤쪽의 모니터링 심화 [Chapter 17] 에서 다룬다.
