4.4.6 리스너(Listener) 바인딩 및 멀티플렉싱(Multiplexing) 구성

수백 개의 커넥션을 받는 라우터에서, 각 프로토콜(TCP, UDP, WebSocket)을 포트 1개(7447)에 싹 다 우겨넣어 묶을(Multiplexing) 것인가, 아니면 프로토콜별로 방을 나눌 것인가?

0.1 [Runbook] 프로토콜 멀티플렉싱(Multiplexing) 대 혼잡분리 전술

전술 A. 단일 포트 멀티플렉싱 (방화벽 관리자 친화형)
인프라 팀에서 포트 구멍을 딱 1개만 열어준다고 횡포를 부린다면 어쩔 수 없이 포트를 통일시킨다. Zenoh 엔진이 패킷 헤더를 분석해 TCP/QUIC을 분할 매핑해 준다.

{
  listen: {
    endpoints: [
      "tcp/0.0.0.0:7447",
      "quic/0.0.0.0:7447"
    ]
  }
}

(장점: 보안 스위치 정책 1줄 / 단점: Linux 커널 단에서 TCP/UDP 분배 병목 부하 확률)

전술 B. 포트 아이솔레이션 (성능 튜닝형 - 권장)
백본 트래픽과 말단 디바이스 트래픽이 섞이는 혼잡 붕괴를 막으려면 철저히 게이트웨이를 나눠야 한다.

{
  listen: {
    endpoints: [
      "quic/0.0.0.0:8000", // 서버-to-서버 백본 전용 채널 (초고속)
      "tcp/0.0.0.0:7447",  // 로봇-to-서버 텔레메트리 
      "ws/0.0.0.0:8080"    // 웹 프론트엔드 관제 대시보드
    ]
  }
}

이렇게 포트를 나누면 사내 모니터링 툴(Prometheus/Grafana)에서 프로토콜별 네트워크 트래픽 인바운드/아웃바운드 밴드위스를 시각적으로 분리하여 감시할 수 있는 “모니터링의 특권“이 생긴다.
