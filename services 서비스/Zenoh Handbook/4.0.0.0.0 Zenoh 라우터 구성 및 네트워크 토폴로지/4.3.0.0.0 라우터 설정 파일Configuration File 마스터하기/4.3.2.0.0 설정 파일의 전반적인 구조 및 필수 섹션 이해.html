4.3.2 설정 파일의 전반적인 구조 및 필수 섹션 이해

라우터의 뼈대를 이루는 json5 텍스트 안에는 백본 네트워크의 생사를 가르는 몇 개의 거대 블록(Sector)이 존재한다.

0.1 [Runbook] Production JSON5 템플릿 해부

// [zenohd_production.json5]
{
  // 1. 아이덴티티: 네트워크 안에서 이 라우터의 고유 지문. 
  // (생략 시 자동 생성되지만, 관제를 위해 K8s 팟(Pod) 이름이나 장비 번호로 박아두는 것이 디버깅에 편하다)
  id: "EDGE_ROUTER_A01",
  
  // 2. 물리적 네트워크 구멍 뚫기 (가장 흔한 수정 지점)
  listen: {
    // 0.0.0.0 은 외부 핑을 받겠다는 선언, TCP/QUIC 포트를 다 연다.
    endpoints: ["tcp/0.0.0.0:7447", "quic/0.0.0.0:7447"]
  },

  // 3. 상위 클라우드에 뻗어 나갈 동아줄
  connect: {
    // 켜지는 즉시 도쿄에 있는 글로벌 라우터를 찾아가 악수(Handshake)를 청한다.
    endpoints: ["tcp/aws-tokyo-backbone.com:7447"]
  },

  // 4. 스카우팅 통제 (멀티캐스트)
  scouting: {
    multicast: {
      // 로컬망에 로봇이 100대가 넘으면 false로 끄는 것이 네트워크 폭파를 막는 길이다.
      enabled: true
    }
  },

  // 5. 플러그인 로드 모듈
  plugins: {
    // REST API 프록시 구동 (http: 8000 포트로 붙을 수 있게 됨)
    rest: {
      http_port: 8000
    },
    // 저장소 DB 활성화 (Data-at-rest) => 14장에서 심화 학습
    storage_manager: {
      volumes: {
        memory: {}
      }
    }
  }
}

사내 데브옵스 설계자는 이 뼈대 코드를 복사해서, Edge용(스카우팅 On, DB Off), Cloud용(스카우팅 Off, DB On) 등의 json5 프로파일을 만들어 K8s ConfigMap에 배포해야 한다.
