4.3.4 설정 유효성 검사 및 문법 오류 디버깅

인프라 엔지니어가 zenohd.json5 파일에 콤마 하나를 실수로 빼먹었거나 키(endpoints를 endpoint로 오타 냄)를 잘못 적어서 서버를 켰다고 해보자.
일반적인 저급 데몬들은 일단 켜지긴 하지만 해당 포트가 닫혀있거나 조용히 트래픽을 허공으로 날려 보낸다(Silent Bug). 하지만 강력한 Rust 문법 기반으로 짜여진 Zenoh는 켜짐과 동시에 Panic을 일으키며 굉음을 낸다.

0.1 [Runbook] 크래시 로그 분석과 파싱 에러 색출

에러 케이스 1: 배열 안의 콤마 누락 식별 (JSON Parsing Error)
터미널에서 켰을 때 다음과 같은 에러가 튀어 오르면, 방화벽이나 딴 데서 헤매지 말고 당장 에디터로 설정 파일을 열어야 한다.

Error: Failed to parse configuration file '/etc/zenohd.json5'
Caused by: Expected ',' or ']' at line 14 column 5

이 메시지는 정확히 파일 14번 째 줄 주변에 당신의 오타가 있다는 것을 지적해 준다.

에러 케이스 2: 존재하지 않는 키(Key) 값 주입 (Schema Validation Error)

Error: unknown field `endpoint`, expected one of `endpoints` at line 6

이 강력한 타입 매칭 기능(Serde) 덕분에 오타를 친 설정값이 무시 고 기동되는 재앙은 존재하지 않는다.

0.2 [사전 검증 런북] CI/CD 파이프라인의 안전핀

서버에 배포하기 전에 내가 짠 설정 파일이 똥인지 된장인지 먼저 검사하는 방법은 없을까? CLI에 --check 파라미터가 이를 수행한다.

# 실제로 라우터를 가동하지 않고 메모리에 설정파일만 돌려보며 문법과 스키마를 테스트한다!
# (CI 빌드 파이프라인에 반드시 넣어야 할 커맨드)
zenohd -c ./test_config.json5 --check

# 출력: [OK] Configuration is valid.

사내 K8s CD 파이프라인(ArgoCD 등)에서 위 --check 명령어가 실패할 경우 배포(Rolling Update) 릴리즈 락을 걸어서 파이프라인을 중지시키는 훅(Hook)을 장착하라. 이것이 현장에서의 24시간 가용성을 지켜주는 마지막 방패다.
