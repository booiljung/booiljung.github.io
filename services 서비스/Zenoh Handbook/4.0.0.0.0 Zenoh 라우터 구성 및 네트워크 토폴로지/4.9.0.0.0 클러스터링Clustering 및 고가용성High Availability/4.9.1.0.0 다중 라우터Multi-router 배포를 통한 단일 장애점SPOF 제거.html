4.9.1 다중 라우터(Multi-router) 배포를 통한 단일 장애점(SPOF) 제거

클라이언트 장비가 100대가 넘어가면, 클라우드에 띄워 둔 라우터 1대는 곧 “이것만 죽이면 우리 회사의 통신망이 마비된다“는 단일 장애점(SPOF)이 된다. 이를 막으려면 무조건 라우터를 이중화해야 한다.

0.1 [Runbook] 라우터 레플리카(Replica) 배포 및 Mesh 결착 전술

1. [클라우드/인프라 측면] Active-Active 클러스터 링(Ring)
AWS 인스턴스 A와 B에 각각 라우터를 띄우고, 서로가 서로를 물도록 connect 설정을 교차시킨다.

  • 라우터 A의 zenohd.json5: connect: { endpoints: ["tcp/Router_B_IP:7447"] }
  • 라우터 B의 zenohd.json5: connect: { endpoints: ["tcp/Router_A_IP:7447"] }
    이렇게 두 라우터가 악수(Handshake)를 마치면 두 개의 라우팅 테이블(DHT)이 완벽하게 거울처럼 동기화(Mirroring)된다.

2. [클라이언트(에지) 측면] 이중 조준 파이프라인
로봇(클라이언트)은 두 라우터의 IP를 모두 알고 있어야 한다.

let mut config = Config::default();
config.insert_json5(
    "connect/endpoints", 
    "[\"tcp/Router_A_IP:7447\", \"tcp/Router_B_IP:7447\"]"
).unwrap();
let session = zenoh::open(config).res().await.unwrap();

여기서 마법이 일어난다. Router A가 서버 크래시로 죽어버리면?
Zenoh 클라이언트 엔진은 개발자가 예외 처리 소켓 스크립트 코드를 짜지 않았음에도, 소켓 단절을 감지하는 즉시 0.1초 만에 내부 버퍼에 쌓여있던 패킷을 Router B 경로를 통해 재발사한다. 어떠한 데이터 유실도 발생하지 않는 완벽한 핫스탠바이(Hot-Standby) + 액티브(Active) 결합 체계다.
