4.9.3 노드 연결 끊김 시 자동 재접속(Auto-reconnect) 및 백오프(Backoff) 전략

엘리베이터를 탄 배달 로봇이 LTE 망 음영 지대에 들어가서 AWS 클라우드 라우터와의 소켓이 터져버렸다.
엘리베이터에서 내렸을 때 로봇(클라이언트)이 한 행동은 무엇이어야 할까? 수동으로 프로세스를 껐다 켜줘야(Restart) 하는가?

0.1 [Runbook] 백오프(Background Backoff) 재연결 튜닝

Zenoh가 찬양받는 또 다른 이유는, 네트워크 장애 시 클라이언트 엔진 단벽 내부에서 미친 듯이 살기 위해 발버둥 치는 Exponential Backoff 재연결 로직이 하드코딩 되어 있기 때문이다.

[연결 시도 주기(Interval) 통제]
1초 - 2초 - 4초 - 8초 식으로 재접속 시도 간격을 기하급수적으로 늘려나감으로써 통신사의 기지국 트래픽을 배려하고 과부하를 막는 설정이다.

어그레시브(Aggressive)하게 당장 붙어야 하는 환경이라면 클라이언트/라우터 json5의 이 부분을 손보아라.

{
  connect: {
    endpoints: ["tcp/cloud.com:7447"],
    // 기본 백오프 주기(최초 100ms 다이렉트 핑, 최대 10초 대기)
    backoff: {
      initial_delay: 200,   // 끊기자마자 0.2초만에 1차 시도
      max_delay: 5000,      // 아무리 포기 상태여도 5초마다 무한으로 살리려 시도
      multiplier: 1.5       // 시도 주기가 1.5배씩 증가
    }
  }
}

주의: 엘리베이터에 낀 로봇 1만 대가 동시에 initial_delay: 1ms 로 클라우드를 치게 만들면 스스로 서버를 폭파하는 DDOS 공격이 된다. 망의 대수를 고려하여 Backoff에 지터(Jitter, 난수 시간)를 섞을 수 있게 최소 1000ms 이상으로 이완시켜라.
