4.9.4 라우터 간 로드 밸런싱(Load Balancing) 및 트래픽 분산 메커니즘

1대의 라우터가 1Gbps 속도를 감당할 수 있는데 우리 로봇 떼가 초당 5Gbps의 데이터를 퍼붓고 있다면 어떻게 해야 하는가? (자율주행 비전 등).
이때는 L4/L7 로드밸런서(AWS ALB, HAProxy, Nginx)를 스택 앞에 끼워 넣어 트래픽을 가위로 썰어(Sharding) 나눠줘야 한다.

0.1 [Runbook] 라우터 스케일 아웃(Scale-out) 부하 분산 아키텍처

안티 패턴: Stateful 한 Zenoh 브릿지 채널을 일반 HTTP 로드밸런서(Round-Robin)로 묶으면 세션이 끊길 때마다 다른 라우터로 날아가 구독(Sub) 트리가 엉망이 된다.

[정답 인프라 아키텍처: HAProxy L4 매핑]
오직 TCP 바이트 스트림 자체를 분산하는 L4 스위치(NLB) 레벨에서 처리하고, 로봇의 커넥션이 유지되는 한 라우터를 고정시켜야(Session Affinity) 라우팅 트리 연산 오버헤드가 줄어든다.

# docker-compose.yml 런북 배포 예시
version: '3'
services:
  zenoh-router-1: { image: eclipse/zenoh }
  zenoh-router-2: { image: eclipse/zenoh }
  zenoh-router-3: { image: eclipse/zenoh }

  # L4 트래픽 밸런싱 터널 
  haproxy:
    image: haproxy:latest
    ports: ["7447:7447"]
    volumes:
      - ./haproxy.cfg:/usr/local/etc/haproxy/haproxy.cfg

# haproxy.cfg
frontend zenoh_front
    bind *:7447
    mode tcp
    default_backend zenoh_routers

backend zenoh_routers
    mode tcp
    # Source IP 해싱! 같은 로봇은 무조건 같은 라우터에게 보내도록 강제
    balance source 
    server r1 zenoh-router-1:7447 check
    server r2 zenoh-router-2:7447 check
    server r3 zenoh-router-3:7447 check

운영 팁: 로봇(클라이언트) 단말기 입장에서는 이 복잡한 L4 HAProxy의 존재를 모른 채, 그저 tcp/load_balancer_ip:7447 하나만 바라보고 쏘면 끝이다. 뒷단의 3대의 라우터는 백그라운드로(Gossip) 또 서로를 물어뜯으며 완벽한 동기화 망(Clique)을 형성해야 한다.
