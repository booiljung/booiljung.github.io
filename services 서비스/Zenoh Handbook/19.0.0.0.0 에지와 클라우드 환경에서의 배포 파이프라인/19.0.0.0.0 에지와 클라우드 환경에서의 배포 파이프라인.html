19.0 에지와 클라우드 환경에서의 배포 파이프라인

단일 서버나 단일 클라우드 영역 내에서 코드를 배포하는 과정은 현대 소프트웨어 공학에서 이미 정복된 영역이다. 그러나 “로컬의 우분투(Ubuntu) 서버 환경에서 완벽하게 동작하는 코드“라는 수식어는 거대한 사물인터넷(IoT) 및 로보틱스 생태계로 진입하는 순간 아무런 보증 수표가 되지 못한다.

개발자의 노트북에서 컴파일된 코드는 쾌적한 온도, 무한한 전력, 터질 듯한 연산 능력을 갖춘 아마존(AWS) 서버실로 향하는 동시에, 지하 광산에서 간신히 1분에 한 번씩 기지국과 통신하는 초저전력 센서와 진흙투성이의 공사판을 뒹구는 자율주행 모바일 로봇(AMR)의 두뇌 속으로 스며들어야 한다. 이처럼 물리적 제약이 극단적으로 대비되는 양극단의 환경 속으로 단일한 비즈니스 로직을 분배하고 조율하는 과정을 “에지-클라우드 연속성(Edge-Cloud Continuum) 배포“라 부른다.

본 장에서는 천 대의 로봇과 백 대의 클라우드 서버에 코드를 동시에 밀어 넣고, 장애 상황 발생 시 단 1초의 서비스 중단 없이 완벽하게 이전 버전으로 롤백(Rollback) 시킬 수 있는 엔터프라이즈급 배포 파이프라인의 구축 사상과 전술을 해부한다.

1. 분산 환경 배포의 핵심 철학

Zenoh 기반의 분산 시스템을 완전히 통제하기 위해서는 다음의 배포 철학을 파이프라인 내부에 반드시 이식해야 한다.

  1. 상태의 선언적 통제 (Declarative State Management): “엔지니어가 서버에 접속하여 설치 스크립트를 실행한다“는 인프라 관리의 낭만은 폐기해야 한다. 변경을 원하는 목표 상태(Desired State)를 코드로 작성하고, 시스템이 그 코드를 읽어 스스로 물리 세계의 상태를 코드와 일치(Reconciliation)시키게 만들어라.
  2. 단일한 진실의 원천 (Single Source of Truth): 에지 기기에 직접 접속하여 설정 파일을 임의로 타협하고 수정하는 행위를 완전하게 봉쇄하라. 모든 배포, 설정 파일, 그리고 암호화된 키는 오버 더 에어(OTA, Over-The-Air) 파이프라인을 통해서 중앙 Git 리포지토리로부터 흘러나와야 한다.
  3. 네트워크 단절의 일상화 수용 (Embrace Intermittency): 무선 환경의 로봇 통신은 완벽하지 않다. 다운로드 도중 끊어진 네트워크 연결, 배터리 방전으로 인한 전원 차단 같은 가혹한 물리 환경을 예외 상황(Exception)이 아닌 일상적인 업무 흐름(Routine)으로 감내하고 복구해 낼 수 있는 멱등성(Idempotency) 구조를 설계하라.
