1.2.4 인터넷 규모(Internet-scale) 환경에서의 기존 미들웨어 제약 사항

로컬 공장을 넘어 대륙과 대륙의 로봇을 관제하려는 거대 연결 시대가 도래했다. 유럽에 있는 로봇을 한국의 관제 클라우드에서 다루는 ‘인터넷 스케일(Internet-scale)’ 세계관에서 기존 브로커와 로컬 통신 방식은 산산조각 난다.

0.1 제약 1: 지리적 거리로 인한 TCP 혼잡 윈도우 붕괴

물리적인 거리(태평양 해저 케이블)가 멀어지면 핑(RTT)이 200ms 이상으로 증가한다. 기존 MQTT나 HTTP REST API 기반 시스템은 TCP 혼잡 제어(Congestion Control) 알고리즘에 묶여 있다. 패킷 유실이 발생할 때마다 송신 윈도우 사이즈를 절반으로 깎아버리며(TCP Reno, CUBIC 등), 대역폭 쓰루풋(Throughput)이 10Mbps에서 100Kbps 수준으로 곤두박질친다.

0.2 제약 2: IP 주소 체계의 NAT-Traversal 한계

공인 IP를 모든 에지 장비에 나눠주는 것은 불가능하다. 모든 로봇과 센서는 사설망(Nat/Firewall) 뒤에 숨어있다. 기존 P2P 방식이나 UDP 통신 기반 프로토콜(DDS 등)은 이 이중고의 인터넷 NAT 펀치홀링(Punch-holing)을 뚫어내는 데 극도로 취약하다.
결국 엔지니어들은 중간에 비싼 VPN 터널망이나 전용선(Leased Line)을 임대해야만 한다.

0.3 런북 대응책 (Zenoh의 와이드 에어리어 최적화)

이 문제를 근본적으로 분쇄하기 위해 Zenoh는 두 가지 파괴적 아키텍처를 프로토콜 코어에 이식했다.

  1. 차세대 트랜스포트 레이어 QUIC 도입: Google이 만든 인터넷 스케일용 QUIC 알고리즘을 기본 탑재하여, 패킷 로스가 나도 전체 파이프라인이 멈추지 않게(Head-of-Line Blocking 해소) 조련했다.
  2. Scouting 및 자동 터널링: 인트라넷 방화벽 뒤의 에지 Zenoh 라우터가 자발적으로 외부 인터넷의 공인 글로벌 라우터를 찾아가(Scouting) 연결의 끈을 유지함으로써, 방화벽 인바운드를 열지 않고도 역방향 통신 트래픽의 수문을 개방한다.
