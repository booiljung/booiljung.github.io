1.2.3 DDS(Data Distribution Service)의 뛰어난 성능과 그에 반하는 복잡성

ROS2(Robot Operating System 2)의 뼈대로 채택되며 로보틱스 엔지니어들의 영혼을 갈아 넣게 만든 주범, 데이터 분배 서비스(DDS)다. 중앙 서버(Broker) 없이 완벽한 P2P 로컬 분산 통신을 이뤄내며 통신 속도의 극한을 자랑하지만, 실전 환경에서 이를 운영하는 일은 저주받은 십자가를 짊어지는 것과 같다.

0.1 절망적 난이도 1: 디스커버리(Discovery)의 악재와 방화벽 지연 지옥

DDS 머신들이 네트워크 상에서 서로의 존재를 인지하기 위해 수천, 수만 개의 멀티캐스트(Multicast) 스카우트 패킷을 폭풍처럼 날려댄다. 이는 사내망 보안 관리자들의 가장 큰 두통거리다. 공유기/L3 라우터 너머로는 아예 데이터가 나가지 못해, 한 사무실에 배포된 로봇 패킷을 다른 오피스에서는 영원히 조회할 수 없다. 외부 클라우드와의 접점은 아예 봉쇄된 폐쇄망 특화 엔진일 뿐이다.

0.2 절망적 난이도 2: 지독한 진입 장벽, IDL 파싱 컴파일 타임

MQTT가 JSON을 그대로 pub/sub하는 우아함을 지녔다면, DDS로 문자열 하나를 보내려거든 이 기괴한 절차를 밟아야 한다.

  1. Message.idl 파일을 C 언어 스타일로 엄격하게 타이핑한다.
  2. rtiddsgen이나 Cyclone 제네레이터 스크립트를 돌려 수백 줄짜리 C++ Boilerplate 자동 코드를 생성한다.
  3. 이 생성된 헤더와 소스를 본 프로젝트의 CMake에 복잡하게 엮어 컴파일한다.

이 파이프라인 덕분에 데이터 변수명을 하나 추가하려고 하면, 팀 전체가 도커 컨테이너를 부수고 다시 클린 빌드(Clean Build)를 30분간 수행해야 하는 악몽의 루프에 갇힌다.

0.3 실전 테스트 극복 런북: Zenoh 브릿지를 통한 해방

DDS의 그물망에서 고통받고 싶지 않은 개발자는, 하위 ROS2 계층은 DDS에 맡겨두되 클라우드로 나가는 통문(Tether) 위치에 단 하나의 Zenoh 명령어를 투입해 탈출구를 개척하라.

# 로컬 DDS 장비들을 글로벌 인터넷 공간으로 견인하는 단일 런처 코맨드
# 더 이상의 복잡한 XML 설정도, NAT 방화벽 펀치홀도 필요 없다.
zenoh-bridge-dds -e tcp/aws-cloud-router.com:7447

Zenoh는 DDS의 통신망을 깔끔하게 흡수(Bridge)하여 와이어(Wire) 위로 가볍에 띄워 올린다. 무거운 IDL 페이로드를 동적인 데이터 객체로 유연하게 캐스팅(Dynamic Casting)하여 프론트엔드 TypeScript나 Python 데이터 사이언티스트들이 구조체 재컴파일 없이 즉답으로 JSON처럼 다루게 만들어주는 구원자다.
