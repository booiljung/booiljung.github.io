1.2.2 MQTT의 태생적 한계와 중앙 집중형 브로커의 병목 현상

“개발자들에게 IoT 통신 아키텍처를 짜라고 하면 99%는 MQTT를 선택한다. 그리고 1년 뒤, 99% 확률로 서버가 터져서 후회한다.”

0.1 한계 1. 완전한 중앙 집중화 구조 (Single Point of Bottleneck)

MQTT는 반드시 퍼블리셔와 서브스크라이버 사이에 우체국(Broker)을 둬야만 한다. 스마트 팩토리에서 자율주행 셔틀 A가 1미터 옆에 있는 셔틀 B에게 서로 부딪치지 않기 위해 정지 신호를 보낸다고 치자.
이 데이터는 무조건 수 킬로미터 떨어져 있는 아마존 AWS 클라우드의 MQTT 브로커를 다녀와야 한다.

실전 터미널 체감 (Ping Test):

# 로컬에서 두 장비가 직접 P2P 통신할 때의 핑(Ping)
ping 192.168.1.10 -c 1
# -> Result: 1.2 ms

# MQTT를 태워 클라우드를 왕복(RTT)하는 레이턴시
# 1. 로봇 A -> AWS Broker -> 로봇 B
# -> Result: 125.0 ms

이 124ms의 구조적인 지연 시간은 소프트웨어 최적화로 절대 메꿀 수 없는 ’설계적 죄악’이다. 즉시성을 다투는 Lidar 데이터나 조향 데이터가 모조리 이 중앙 우체국의 트래픽 병목 톨게이트에 막혀 폐기된다.

0.2 한계 2. TCP 종속성 및 모바일 네트워크에서의 치명적 약점

MQTT는 무조건 연결 지향형 포트인 TCP/IP 환경 위에서만 작동한다. 5G/LTE처럼 연결이 빈번히 단절되었다가 복구되는 ‘핸드오버(Hand-over)’ 상황에서 TCP의 3-Way Handshake 규칙은 무자비할 정도로 느리다.
연결이 끊기면 다시 3번에 걸쳐 SYN -> SYN ACK -> ACK를 교환하고 나서야 데이터를 보낼 수 있다.
수천 대의 모바일 셔틀이 와이파이 음영 지역을 지날 때마다 이 연결/해제 폭풍(Connection Storm)을 발생시키면 MQTT 브로커 데몬 서버는 단시간에 CPU 포화 상태(Socket Exhaustion)를 버티지 못하고 크래시(Crash)난다.

0.3 선언적 대안: Zenoh의 해법

Zenoh에 중앙 브로커는 필수가 아니다. 로봇 A와 B가 동일 네트워크에 있다면, 굳이 AWS로 올리지 않고 발견(Scouting) 즉시 직접 다이렉트 P2P 소켓을 열어 1ms 만에 통신을 마친다. TCP가 불안하면 즉각 그 밑단을 QUIC이나 UDP 트랜스포트 레이어로 갈아 끼우는 카멜레온적 위상이야말로 MQTT 구조가 넘보려야 넘볼 수 없는 혁신이다.
