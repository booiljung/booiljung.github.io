1.5.1 최소한의 오버헤드(Minimal Overhead)와 초저지연(Ultra-low Latency) 지향

에지(Edge) 네트워크는 배터리, CPU, 대역폭이라는 세 가지 혹독한 제약 속에서 생존해야 한다. 무거운 헤더와 복잡한 핸드셰이크는 이 환경에서 사치이자 시스템 마비의 주범이다. Zenoh는 와이어 상에서 오가는 바이트의 단 1비트조차 낭비하지 않는 결벽증적 설계를 채택했다.

0.1 Bytes 와이어 프로토콜의 기적

전통적인 HTTP/REST 파이프라인에서 “Hello“라는 5바이트 데이터를 전송하기 위해서는 최소 150바이트의 TCP 헤더와 HTTP 텍스트 헤더가 덧붙여진다(오버헤드 3000%).
반면 Zenoh가 채택한 가변 길이 인코딩(Variable-length Encoding, varint)과 최소 헤더(Minimal Headers) 포맷은 놀랍다. 데이터 본문을 포장하는 Zenoh의 순수 프로토콜 프로필 두께는 최소 4~5 Bytes에 불과하다.

0.2 지연(Latency)의 두 가지 족쇄 끊기

  1. Serialization(직렬화) 오버헤드 제거:
    기존 기술은 데이터를 쏠 때 JSON, Protobuf 등으로 ‘반드시’ 직렬화를 강제한다. 이는 수신 측에서의 정규화된 파싱을 위해 CPU 시간을 잡아먹는다. Zenoh는 이를 강제하지 않고 바이너리 페이로드 자체를 그대로 통신관표(Wire) 위로 미끄러뜨린다. 데이터 파싱 여부는 비즈니스 로직(최종 목적지)의 몫이다.
  2. 동적 라우팅 최적 경로 탐색:
    데이터가 물리적으로 최단 경로를 우회(Short-circuit)하지 못하면 지연이 생긴다. Zenoh는 BGP(Border Gateway Protocol)처럼 거대한 망을 맴돌게 하지 않고, 세션 개방 즉시 로컬 망 내의 P2P 피어들을 스카우팅하여 직접 다이렉트 소켓을 체결해 1ms의 홉(Hop) 레이턴시를 쟁취한다.

0.3 [Runbook] 오버헤드 프로파일링 실증

당신의 산업용 네트워크가 Zenoh 도입 후 얼마나 가벼워졌는지 증명하고 싶다면, Wireshark를 켜고 다음 커맨드를 통해 eth0의 트래픽을 정량화하라.

# 1초에 1,000건의 작은 센서 데이터를 쏘는 퍼블리셔 기동
z_put -k "sensor/temp" -v "25" -p 1000

# ifconfig나 ip 명령어의 인터페이스 RX/TX 바이트로 대역폭 소모량 적출
# MQTT 사용 시 초당 150KB를 먹던 트래픽이, Zenoh에선 10KB 미만으로 측정되는가?
watch -n 1 'ifconfig eth0 | grep "TX packets"'

네트워크 부하가 1/10 수준으로 압축된 현장을 눈으로 확인했다면, 남는 대역폭은 4K 로봇 관제 비디오 스트리밍을 위한 쾌속 도로로 재활용할 수 있게 된다.
