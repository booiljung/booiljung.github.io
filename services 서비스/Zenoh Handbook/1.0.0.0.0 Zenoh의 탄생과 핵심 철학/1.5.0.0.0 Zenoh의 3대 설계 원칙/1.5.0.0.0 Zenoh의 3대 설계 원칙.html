1.5 Zenoh의 3대 설계 원칙

어떤 오픈소스 소프트웨어든 10년이 지나면 기능(Feature)이 비대해져서 괴물이 되고, 결국 가벼움을 무기로 삼던 초창기의 장점은 거대한 설정 파일(Config XML)에 묻혀 증발한다. DDS가 그러했고, AMQP가 그러했다.

Zenoh 개발 태스크포스를 이끄는 Eclipse Foundation 팀은 이러한 ’비대화의 저주’를 막기 위해 기둥 벽에 세 가지 타협 불가능한(Non-negotiable) 절대 설계 원칙을 박아 넣었다. 기능이 이 원칙에 위배된다면 커밋(Commit)은 과감히 기각당한다.

이 절에서는 로봇 엔지니어와 시스템 아키텍트가 프로메테우스 벤치마킹이나 구조설계를 그릴 때, 왜 Zenoh가 특정 기능에서 이상하리만치 결벽에 가까운 가벼움을 보여주는지 납득할 수 있는 뼈대, 즉 “1. 최소 오버헤드 최우선, 2. 완벽한 체급별 스케일링 동기화, 3. 토폴로지 지옥으로부터의 완전 해방“이라는 3대 성역 원칙을 실제 런타임 결과물에 비추어 파해친다.
