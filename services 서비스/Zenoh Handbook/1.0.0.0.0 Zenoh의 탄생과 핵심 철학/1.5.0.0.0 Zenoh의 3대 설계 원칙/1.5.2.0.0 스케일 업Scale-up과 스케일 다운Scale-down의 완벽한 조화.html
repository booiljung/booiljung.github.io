1.5.2 스케일 업(Scale-up)과 스케일 다운(Scale-down)의 완벽한 조화

프로토콜의 우수성은 얼마나 무거운 데이터를 실어나를 수 있는가(스케일 업)뿐만 아니라, 얼마나 좁고 숨 막히는 공간(MCU, 저사양 칩)까지 구겨 들어갈 수 있는가(스케일 다운)로 평가된다.
Zenoh는 하나의 거대한 모놀리식 바이너리가 아니라, 노드의 역할과 하드웨어 스펙에 따라 뇌의 크기를 능동적으로 조절하는 플래그십(Flagship)과 마이크로(Micro) 이원화 전략을 취한다.

0.1 스케일 다운 (Scale-down): Zenoh-Pico의 한계 돌파

메모리가 64KB밖에 없는 저전력 ESP32나 펌웨어 기반의 산업용 PLC 칩셋은 리눅스 커널(OS)조차 올라가지 않아 전통적인 MQTT TCP 소켓 라이브러리마저 힘겨워한다.
Zenoh는 철저하게 POSIX 스탠다드 의존성을 걷어내고 순수 C언어로 작성된 초경량 구현체 **zenoh-pico**를 제공한다.

  • 메모리 풋프린트: 구동 시 OS 동적 할당(malloc)을 아예 봉쇄(Static Memory Allocation)하도록 컴파일하여 단 15KB~30KB의 램만으로도 Put과 Sub 통신 스레드를 영구 구동할 수 있다.
  • 프로토콜 스위치: 무거운 TCP 대신 시리얼(Serial), 블루투스(BLE), 심지어 캔 버스(CAN Bus) 위로 와이어 포맷을 흘려보낸다.

0.2 스케일 업 (Scale-up): Rust 기반 Zenohd 라우터의 포효

반대편 끝엔 구글 클라우드(GCP)에 떠 있는 64코어 256GB 메모리의 괴물 서버가 있다.
여기엔 Rust로 완벽히 무장된 zenohd 라우터 데몬이 위치한다. 이 라우터는 1만 개의 에지 노드(zenoh-pico)가 쏘아대는 초당 1천만 건의 텔레메트리 트래픽을 멀티스레드 비동기 런타임인 Tokio를 통해 논블로킹(Non-blocking)으로 받아낸다.

0.3 [Runbook] 아키텍처 다이어트 설계안

“어떤 장비에 무엇을 올릴 것인가?” 이 원칙이 무너지면 시스템은 불탄다.

  1. Level 0 (IoT 센서, 8비트/16비트 MCU): 무조건 zenoh-pico (C 라이브러리) 배포.
  2. Level 1 (라즈베리 파이, Jetson Nano 에지 컴퓨터): 일반 퍼블리셔/서브스크라이버 바이너리 또는 Client 모드로 연결 확립.
  3. Level 2 (공장 스위치망 Gateway, K8s 클러스터): 완벽한 zenohd 라우터를 띄운 뒤, 라우팅 스위치 및 Storage 플러그인 탑재.

단일한 통신 규칙(Zenoh Protocol) 하에서 15KB의 칩셋과 150GB의 서버가 변환 프록시의 간섭 없이 곧바로 다이렉트(Get/Put) 대화를 나누는 모습, 이것이 진정한 에지-투-클라우드 컨티뉴엄(Continuum)이다.
