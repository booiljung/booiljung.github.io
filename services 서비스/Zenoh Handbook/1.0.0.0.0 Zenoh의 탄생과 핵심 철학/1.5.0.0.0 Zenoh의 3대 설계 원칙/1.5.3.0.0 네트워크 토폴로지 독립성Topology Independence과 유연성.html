1.5.3 네트워크 토폴로지 독립성(Topology Independence)과 유연성

기존의 마이크로서비스 설계자들은 인프라 단계에서 큰 딜레마를 마주했다.

  • Client-Server(Broker) 모델: 관리가 쉽고 클라우드에 유리하지만 단일 고장점(SPOF)이 되고 에지 간 통신이 지연된다.
  • Peer-to-Peer(P2P) 다이렉트 망 구성(DDS 등): 속도는 미친 듯이 빠르지만, 망이 거대해질수록 디스커버리 오버헤드가 기하급수적으로 폭발하고 방화벽을 뚫지 못한다.

Zenoh 개발진은 “어떤 토폴로지가 이길까?“를 놓고 싸우는 대신, **“네트워크의 형태(Topology)를 애플리케이션 코드와 완벽히 격리하자”**는 제3의 신의 한 수를 꺼내 들었다.

0.1 토폴로지 블라인드(Topology Blind) 아키텍처

퍼블리셔가 session.put("topic", "data")을 호출할 때, 이 장비가 중앙 라우터를 바라보고 있는지(Client 모드), 아니면 옆 장비를 향해 P2P로 연결되었는지(Peer 모드), 자신 스스로가 망의 중심인 라우터 데몬인지(Router 모드)를 코드단에서 전혀 명시하지 않는다.

Zenoh의 3대 운영 모드 (런타임 주입)
개발된 바이너리를 실행하는 순간 인프라 엔지니어가 런타임 파라미터로 이 애플리케이션의 위상(Topology)을 결정짓는다.

  1. Router 모드 (zenohd): 허브 앤 스포크(Hub-and-spoke)의 중심. 멀티 라우터들이 맞붙어 거대한 Mesh 글로벌 관문망 형성.
  2. Peer 모드 (-m peer): 다른 Peer들을 적극적으로 스카우팅(멀티캐스트)하고, 동적으로 트리를 구축하며 중간 경유자(릴레이) 역할을 자처하는 P2P 모바일 게이트웨이.
  3. Client 모드 (-m client): 배터리를 극단적으로 아끼기 위해 자기 자신을 남에게 뽐내지 않고, 스카우트에 반응 조차 하지 않은 채 얌전히 상위 Router/Peer에게만 한 가닥 선을 연결하는 최소 전력 모드.

0.2 [Runbook] 무정지 망 변형(Mutation) 대응 훈련

당신의 클라우드 중앙 라우터 A가 디도스(DDoS) 공격으로 죽는 재난이 벌어졌다고 치자. 클라이언트 100대는 연결선을 잃고 떠돈다.
기존 MQTT라면 망이 전멸했겠지만, 당신의 런북 매뉴얼엔 이런 지침이 하달된다.

# 긴급 조치: 클라이언트 중 사양이 가장 빵빵한 Edge 1번과 2번 컴퓨터의 데몬을 일시 중지 후, 
# '자율 결합 P2P Peer 모드' 로 모드 체인징 후 재부팅한다.
killall edge_app
./edge_app -m peer &
# 다른 98대의 Client들은 클라우드 라우터가 죽어있음을 알고, 
# 옆에 새롭게 켜진 Peer 단말기들에게 자동으로 스카우팅되어 연결을 "우회 결속" 시킨다.

애플리케이션 코드를 1바이트도 수정하지 않고, 그저 실행 커맨드라인 매개변수 조절만으로 거대한 공장의 중앙 집권형(Star) 망을 무정부의 탈중앙 P2P(Mesh) 망으로 트랜스포메이션 시켜 살려냈다. 이것이 토폴로지 독립성이 선사하는 공학적 ‘방탄’ 아키텍처다.
