1.1.1 차세대 데이터 분산 네트워크(Data Distribution Network)란 무엇인가

기존의 네트워크는 ’메시지를 어떻게 전달할 것인가(Message-centric)’에 집착했다. 목적지의 IP 주소와 포트 번호를 찾고 세션을 맺는 과정 자체가 막대한 오버헤드였다. 그러나 Zenoh가 제시하는 **데이터 분산 네트워크(DDN)**는 발상을 뒤집는다. “어디(Where)에 있는지가 아니라 무엇(What)인지“에 집중하는 네임드 데이터 네트워킹(NDN)의 사상을 실용화한 것이다.

0.1 단계: 기존 네트워크의 한계 (IP 매핑의 지옥)

마이크로서비스 A에서 B로 데이터를 보낼 때, 기존 환경에서는 개발자가 IP를 하드코딩하거나 DNS, K8s Service Discovery에 전적으로 의존해야만 했다.

  • 네트워크 단절이 잦은 모바일 로봇(AMR) 환경에선 IP 대역이 스위칭될 때마다 소켓 연결 에러 리트라이 스니펫(Retry Snippet)을 수백 줄씩 짜넣어야 한다.
  • MQTT조차 단일 브로커(Broker) IP 주소라는 ’SPOF(단일 고장 지점)’에 족쇄가 묶인다.

0.2 단계: DDN(Data Distribution Network)의 선언적 통신 해결책

Zenoh는 개발자로부터 IP 계층의 고통을 완벽히 추상화(Abstraction)한다. 사용자는 오직 데이터의 ’키(Key)’만 바라보고 통신한다.
가장 직관적인 DDN의 위력을 zenoh-cli 로컬 명령어로 테스트하라.

# [스텝 1] 로봇 단말(IP: 동적 변동)에서 온도 섭씨 데이터를 퍼블리시
# IP 주소나 대상 서버 설정이 전혀 필요 없다. 오직 Key(로봇1/온도)만 존재한다.
z_put -k "factory/robot1/temp" -v "25.4"

# [스텝 2] 클라우드 관제 대시보드(IP: 동적 변동)에서 데이터 수신 대기
# Zenoh 데몬이 언더레이(Underlay)망을 스카우팅하여 알아서 최적 경로를 라우팅한다.
z_sub -k "factory/robot1/temp"

위 명령어에서 보듯 DDN 환경 하에서는 인프라(IP, Port, TCP/UDP)가 변하는 물리적 혼란 속에서도 애플리케이션의 코드는 단 한 줄도 수정할 필요가 없다.
데이터를 그물망(Mesh Topology) 안의 공간에 흩뿌려 놓고, 필요한 자가 키(Key)를 통해 알아서 길어 먹는(Pull) 진정한 의미의 **위치 투명성(Location Transparency)**이 바로 차세대 네트워크의 정수다.
