1.1.3 데이터 폭증 시대의 네트워크 대역폭 한계와 지연 문제

현대 클라우드 시대의 가장 큰 기만은 ’망은 항상 빠르고 넉넉하다’는 환상이다. 서울 한복판의 AWS 리전과 5G망은 쾌적하지만, 해상 시추선, 지하 40미터의 스마트 마이닝(광산) 설비, 농기계 트랙터 단말기에서 통신하는 순간 대역폭은 킬로바이트(KB) 단위로 쪼그라들고 통신 지연(Latency)은 500ms(0.5초)를 우습게 돌파한다.

0.1 대역폭 낭비의 근원: 비효율적 헤더 오버헤드

JSON을 REST API(HTTP/1.1)로 전송한다고 치자. 센서 정보는 달랑 “온도 20” ({"t":20} = 8바이트) 이지만, 하부 TCP/IP와 HTTP 헤더, TLS 암호화 오버헤드가 따라붙으면 실제 전파를 타는 패킷 크기는 한 방에 300바이트로 뻥튀기(Overhead 4,000%)된다. 이러한 비효율은 수만 대의 센서가 물릴 때 게이트웨이 스위치의 목을 조른다.

0.2 와이어 포맷 최적화: Zenoh의 극단적 다이어트

이에 대응하여 Zenoh 프로토콜 위원회는 ’바이트 단위의 쥐어짜기’를 핵심 스펙으로 구현했다.
Zenoh의 와이어 프로토콜(Wire Protocol)은 데이터 인코딩, 포맷, 식별자 전부를 극한의 비트마스킹(Bitmasking) 기술로 압축시켰다.

0.3 홉 지연(Hop Latency) 파괴

퍼블릭 클라우드 데이터베이스에 직접 데이터를 밀어 넣으면, 왕복 응답시간(RTT)이 150ms 이상 걸린다. 로봇 팔이 장애물을 피하기에는 너무도 긴 영겁의 시간이다.
Zenoh는 중앙 서버로 데이터를 굳이 보내지 않고, 최단 거리 통신(P2P 다이렉트 소켓)이나 로컬망 내의 1단계 에지 라우터를 스스로 탐색하여 먼저 붙는다. 따라서 물리적 빛의 속도를 갉아먹는 중앙 노드 경유(Hair-pinning) 현상을 구조적으로 타파하여 5ms 이내의 홉 지연율(Hop Latency)을 달성한다. 이것이 병목 시대를 정면 돌파하는 철학적 기조다.
