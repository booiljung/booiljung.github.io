1.1.2 IoT 및 로보틱스 생태계의 급격한 변화와 진화

단순히 스위치를 켜고 끄거나 1초에 한 번씩 온도를 측정하던 고전적인 IoT(사물인터넷) 시대는 끝났다. 최신 하드웨어의 보급과 에지 AI(Edge AI)의 결합은 로보틱스의 데이터 패러다임을 폭발적으로 진화시켰다.

0.1 실무 시나리오: 자율주행 셔틀(AMR) 트래픽의 모순

현재 공항이나 대형 물류 창고에서 운영되는 차세대 로봇(AMR) 1대는 다음과 같은 데이터를 뿜어낸다.

  1. 제어/제어 피드백 (1ms 단위, 크기 100Bytes): 모터 조향명령 (초저지연, 데이터 유실 금지)
  2. 비전/카메라 (30FPS, 초당 20MB): 자율주행 회피를 위한 원격 스트리밍 (지연 허용안됨, 유실 일부 허용)
  3. 상태 로그 DB 백업 (크기 500MB): 어제 주행한 라이다 포인트클라우드 누적 맵 (백그라운드 비동기 대용량 전송)

하나의 바퀴 달린 로봇 안에서, **’모기처럼 가볍고 잽싼 스레드’와 ‘코끼리처럼 무거운 스레드’**가 동시에 네트워크 파이프를 통과해야 한다.

0.2 레거시 시스템의 처참한 붕괴 현장 (Anti-Pattern)

과거의 개발 조직은 이 세 가지 요구사항을 맞추기 위해 끔찍한 잡탕(Frankenstein) 아키텍처를 건설했다.

  • 제어용 통신으로는 파이프가 거대해질 수 없는 MQTT 브로커를 설치한다.
  • 실시간 카메라는 WebRTC 전용 미디어 서버 컨테이너를 하나 더 올린다.
  • 지도 백업은 무식하게 FTP나 HTTP REST API 데몬을 추가로 열어 크론(Cron) 작업으로 밀어 넣는다.

하나의 로봇에 세 종류의 소켓과 세 종류의 보안 방화벽이 구멍 뚫린다. 디버깅 포인트가 세 배로 늘고, 서버가 재부팅되면 이 세 파이프가 엉키며 포트 충돌이 터진다.

0.3 Zenoh가 제안하는 단일 프로토콜 천하 통일

Zenoh는 이러한 생태계 변화, 즉 **‘이기종 프로필 데이터의 병합’**이라는 통증을 치유하기 위해 등장했다.
개발자는 단 하나의 분산 런처(zenohd)만 켜고, 위 세 가지 극단적인 QoS 특성을 프로토콜 하나로 제압한다.

// [Zenoh를 통한 3단 트래픽 통합 관리 (Rust)]
let session = zenoh::open(config).res().await.unwrap();

// 1. 제어 신호 퍼블리시 (Reliability 최우선)
session.put("robot/cmd", cmd_payload).res().await.unwrap();

// 2. 고화질 비디오 스트림 전송 (단편화 Fragmentation 지원, Best effort)
session.put("robot/video", huge_video_frame)
       .congestion_control(CongestionControl::Drop) // 막히면 옛 프레임 폐기
       .res().await.unwrap();
       
// 3. 누적 DB 원격 조회 (Query 아키텍처 결합)
let replies = session.get("robot/map_db").res().await.unwrap();

각기 다른 성질의 생태계 트래픽을 단일 라이브러리로 봉합하는 추상화. 이것이 로보틱스 개발자가 수십 개의 포트를 관리하는 악몽에서 벗어나게 해 줄 진화의 산물이다.
