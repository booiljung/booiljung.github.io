1.4.3 위치 투명성(Location Transparency)의 구현 원리

데이터베이스 테이블에서 SELECT * FROM users; 쿼리를 칠 때, 우리는 그 데이터가 물리적으로 디스크 섹터 1번 트랙에 있는지, 캐시 램 메모리 0x88에 할당되어 있는지 묻지 않는다.
Zenoh는 바로 이 데이터베이스의 편리한 철학을 ‘전 지구적인 분산 엣지 네트워크’ 위로 고스란히 끌어올렸다.

0.1 위치 투명성의 정의

클라우드 대시보드에서 /factory/machine-A/rpm 값을 달라고 요청(Get)할 때, 이 RPM 수치가 다음 중 어디에 위치하든 호출하는 개발자의 코드(.get())는 완벽히 똑같아야 한다는 원칙이다.

  1. 물리적으로 바로 옆 단말기의 로컬 캐시에 있을 수도 있고,
  2. 지구 반대편의 AWS 라우터 스토리지 플러그인의 록스DB(RocksDB)에 들어있을 수도 있으며,
  3. 아예 저장된 적 없이, 이 값을 계산할 줄 아는 Lidar 센서 내부에 수학적 연산 함수 공간에 존재할 수도 있다.

어디에 있건 간에, 패킷을 찾는 엔진(Zenoh Routing Algorithm)이 알아서 최단 경로를 탐색하여 당신의 변수에 값을 넣어준다.

0.2 Queryable을 통한 ‘컴퓨팅의 위치 투명성’

데이터 저장본뿐 아니라, “데이터를 생성해 내는 컴퓨팅(RPC 연산)“조차 위치 투명성에 포함된다. 이를 Zenoh에서는 Queryable 이라 부른다.

// [에지 단말기 로봇 코드]
// 로봇이 자신에게 "/robot/battery/compute_time_left" 라는 쿼리가 들어오면
// 그 즉시 '계산'해서 답을 던져주도록 함수(Queryable)를 등록해 둔다.
session.declare_queryable("robot/battery/compute_time_left").callback(|query| {
    let left_time = perform_heavy_calculation();
    query.reply(Sample::new(query.key_expr().clone(), left_time));
}).res().await.unwrap();

관제 서버에서는 데이터가 “어떤 IP 데이터베이스“에 있는지, 그리고 “어떤 로봇의 어떤 RPC 함수“인지 구별할 필요 없다. 그저 일관되게 session.get("robot/battery/compute_time_left") 한 줄을 호출하면 된다. Zenoh 라우터는 그 키(Key)가 평범한 데이터 조각인지, 아니면 동적으로 실행되어야 할 에지 안의 연산 함수(Queryable) 체계인지 투명하게 분석하여 요청을 조준(Routing)해 낸다. 이 경이적인 추상화 덕분에 백엔드 프레임워크 크기가 1/10로 축소된다.
