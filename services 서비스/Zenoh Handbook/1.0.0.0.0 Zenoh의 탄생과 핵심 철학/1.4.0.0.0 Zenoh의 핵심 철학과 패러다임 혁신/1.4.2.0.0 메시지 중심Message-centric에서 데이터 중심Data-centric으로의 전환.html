1.4.2 메시지 중심(Message-centric)에서 데이터 중심(Data-centric)으로의 전환

0.1 Message-centric Network의 소모성

“가스 밸브를 닫아라!“라는 명령은 메시지 중심적(Message-centric)이다. 이 명령어는 수신자에게 도착하는 순간 휘발되며, 재차 명령하지 않는 이상 그 패킷 자체가 생명력을 갖지 않는다. 기존 TCP, MQTT는 모두 이 휘발성 메시지 전송관에 불과하다.

0.2 Data-centric Network의 영속성

반면 Zenoh가 지향하는 데이터 중심(Data-centric) 아키텍처는 “밸브가 닫힌 상태(State)” 자체를 네트워크의 전역 객체처럼 보존하고 다루는 세계관이다.
네트워크 그 자체가 거대한 ’분산 데이터베이스(Distributed DB)’처럼 동작하게 만든다.

이 개념을 개발 코드(API)로 입증하는 비교 런북:

## 일반적인 메세징 시스템(MQTT)의 패러다임
mqtt_client.publish("/sensor/temp", "24.5")
# (1초 뒤 누군가 듣지 않으면 이 "24.5"라는 데이터는 우주에서 소멸한다)

## Zenoh의 Data-centric 패러다임 (상태 질의)
session.put("sensor/temp", "24.5")

# 1초 뒤, 혹은 10분 뒤, 관제 서버는 "수신 대기(Listen)"를 하지 않고
# 데이터베이스에 쿼리하듯(SQL Select) 네트워크에 대고 상태를 '물어본다'.
replies = session.get("sensor/temp")
for reply in replies:
    print(f"최근 온도 상태는: {reply.data.payload.decode('utf-8')}")

Zenoh 망 위에 올라간 데이터는 허공에 쏘아진 ’소리’가 아니라, 거대한 화이트보드에 기록된 ’문자’처럼 취급된다. 개발자는 다른 노드가 언제 메시지를 쐈는지 감시하는 ’이벤트 폴링 노예’에서 벗어나, 자신이 원하는 시점에 망 전체를 향해 “지금 /robot/status가 뭐지?“라고 질문(Query/Pull)하면 끝나는 위대한 주도권을 획득하게 되었다.
