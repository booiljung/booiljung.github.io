1.4.1 공간과 시간의 분리 (Decoupling in Space and Time)

마이크로서비스 A(퍼블리셔)가 마이크로서비스 B(서브스크라이버)에게 데이터를 전달해야 한다고 가정해 보자.

0.1 고전적인 결합(Coupling)의 고통

  • 공간적 결합 (Space Coupling): A는 데이터를 보내기 위해 B의 ’IP 주소’를 완벽히 알아야 한다. B가 다운되어 K8s 파드가 재시작되며 IP가 변경되면 A는 통신 송출을 멈추고 에러를 뿜는다.
  • 시간적 결합 (Time Coupling): A가 생방송을 쏘고 있을 때, B가 반드시 그 순간에 접속해 ’가동 중(Online)’이어야만 데이터를 받을 수 있다. B가 10분 뒤에 부팅하여 켜진다면, 지나간 10분간의 데이터는 허공으로 증발해버린다. (Late-joiner 문제)

0.2 Zenoh의 진정한 디커플링 아키텍처 구현 (런북 예제)

Zenoh는 이 공간과 시간의 제약이라는 두꺼운 쇠사슬을 끊어버렸다.

1. 공간의 분리 구현 (Location Transparency):
Zenoh 디바이스들은 session.put("robot/status", data) 식으로 ’이름표(Key)’가 붙은 허공의 메일박스를 향해 던질 뿐이다. 라우팅 프로토콜이 백그라운드에서 동적으로 수신자를 매칭시킨다. B가 IP를 10번 바꾸든, 클라우드에서 에지로 이사를 가든 A의 코드는 아무 타격이 없다.

2. 시간의 분리 구현 (Temporal Decoupling):
구독자 B가 전원이 꺼져있는 사이 발행된 데이터 유실을 막기 위해, Zenoh는 라우터 플러그인 생태계에 StorageManager(스토리지 매니저) 기능을 고안해 넣었다.

# 클라우드 중앙 라우터에 RocksDB 백엔드를 붙여 영속성 공간 확보
zenohd --plugin storage_manager

# curl을 통해 라우터에 스토리지 할당 (CLI 구성)
curl -X PUT http://localhost:8000/config/plugins/storage_manager/volumes/my_rocksdb \
    -d '{ "dir": "/var/lib/zenoh/db" }'
curl -X PUT http://localhost:8000/config/plugins/storage_manager/storages/sensor_storage \
    -d '{ "key_expr": "robot/**", "volume": "my_rocksdb" }'

이제 퍼블리셔 A가 데이터를 쏘면, 이 스토리지 라우터가 구독자 B 대신 패킷을 메모리와 디스크에 층층이 저장(Caching)해 둔다.
서브스크라이버 B가 한 시간 뒤 지각 부팅하여 session.get("robot/**") 쿼리를 날리면, 라우터가 자판기처럼 과거의 데이터를 내어준다.
송신자와 수신자가 같은 ’시간’에 살아있지 않아도 통신이 성립되는 기적, 바로 시간적 디커플링의 완벽한 실현이다.
