1.7.3 서드파티 통합과 모듈형 플러그인 생태계 확장성

어떤 훌륭한 시스템이라 할지라도, 기존 인프라에 뿌리를 둔 채 돌아가고 있는 수많은 자산(Legacy Systems)과 타 언어들, 외부 플랫폼들을 배척한다면 시장의 환영을 받을 수 없다. 시스템 아키텍트가 전사에 Zenoh를 도입하기로 결정했다 한들, 공장에 깔린 기존의 MQTT 수집 모뎀이나 수만 대의 InfluxDB 데이터베이스까지 전부 폐기해야 한다면 경제적으로(TCO) 엄청난 재앙이기 때문이다.

Zenoh의 아키텍트들은 이러한 현실을 정확히 파악하고, 처음부터 완전한 모듈형 코어와 플러그인-아키텍처(Plug-in Architecture)로 프레임워크를 빚었다.

Zenoh 라우터(Router) 중앙 코어 주변부에는 손쉽게 부착할 수 있는 수십 가지의 플러그인(Plugins)들이 생태계를 이루고 있다. 가령 Zenoh-MQTT 플러그인을 라우터에 살짝 꽂아 탑재하면, Zenoh 망은 즉시 자신이 MQTT 브로커(Broker)인 척을 하며 수만 대의 MQTT 구형 센서 단말기들과 원래부터 한 가족이었던 것처럼 완벽히 데이터를 중개하고 수집한다. 또는 Zenoh-DDS 플러그인을 붙임으로써 기존 차량용 DDS 데이터 망과 클라우드 인터넷망 사이를 뚫어주는 단일한 매직 브릿지가 성립된다.

graph TD
    subgraph 기존 레거시 인프라 생태계 결합
      A(구형 MQTT 단말) -.-> Z_MQTT[Zenoh-MQTT 플러그인]
      B(차량용 DDS 버스) -.-> Z_DDS[Zenoh-DDS 플러그인]
      C(InfluxDB 스토리지) -.-> Z_Storage[스토리지 플러그인]
    end
    Z_MQTT --> Core((Zenoh 코어 라우터 엔진))
    Z_DDS --> Core
    Z_Storage --> Core
    Core == "단일 통일 Zenoh 망!!" ==> Internet((글로벌 클라우드 통신))

이 엄청난 플러그인 아키텍처는 과거 무수한 프로토콜이 혼재되어 각각 다른 라이브러리와 지식으로 찢어졌던 시스템 복잡도를 혁파하며, 그 어떤 타사(Third-party) 오픈소스나 레거시 데이터망이 버티고 있더라도 완벽하게 하나로 끌어안아 버리는 거대한 블랙홀 같은 생태계 포용력을 완성했다.
