1.6.3 분산 컴퓨팅을 위한 쿼리-응답(Query-Reply) 모델

발행/구독(Pub/Sub) 하나만으로 모든 시스템을 짤 수는 없다. 로봇 관제 대시보드에서 “현재 3번 로봇의 배터리가 몇 퍼센트지?” 궁금할 때마다, 로봇이 스스로 배터리 통보를 Push 해줄 때까지 대시보드가 하염없이 기다리는(Subscribe) 폴링(Polling) 대기 상태에 빠지는 건 우스꽝스럽다.
필요할 때 명시적으로 끌어오는(Pull/Query) 동기식 질의 응답 패턴이 시스템 파이프의 두 번째 핵심 뼈대다. 일반적인 HTTP REST의 GET 요청 방식과 철학이 동일하다.

0.1 분산 Queryable의 우월성 (Multi-Reply)

기존 REST의 HTTP GET은 ’나’와 ‘특정 서버(IP) 1대’ 간의 1:1 폐쇄적인 응답이다.
하지만 Zenoh의 session.get("factory/robot_*/battery") 쿼리는, 내가 IP를 지목하는 것이 아니라 네트워크 백본 전체를 향해 “로봇들의 배터리 아는 놈은 다 대답해라!“라고 소리치는 1:N 분산 질의다.

네트워크상에 자신을 쿼리 가능 상태(Queryable)로 열어둔 10대의 로봇이 이 목소리를 듣고 저마다 “나는 80%”, “나는 15%“라고 대답(Reply)을 강물처럼 클라이언트를 향해 비동기 스트림(Stream)으로 흘려보낸다.

0.2 [Runbook] 비동기 다중 응답(Stream) 타임아웃 붕괴 대처법

클라우드 서버(TypeScript)에서 저 쿼리를 던지고 10대의 응답을 기다리고 있다 켜자. 하필 로봇 7번의 네트워크 칩셋이 타버려서 영원히 응답이 없다.
대시보드 코드를 허술하게 await get(...) 으로 직렬 대기 락(Lock)을 걸어두면 웹 화면 자체가 타임아웃 지옥에 빠지며 하얀 화면(Freeze)에서 멈춰버린다. 백엔드 스레드의 무덤이 열린 것이다!

모범 타개책 (BestMatching과 비동기 순회 탈출):

  1. 목표 라우팅 세팅 시 무조건 Target.BestMatching 플래그를 건다. “완벽하게 10대를 다 채울 필요 없고, 망 내에서 구할 수 있는 빠릿빠릿한 놈들 것 위주로 모이는 대로 줘!“라는 여유를 허락해야 한다.
  2. 타임아웃 시간제한 강제.
// [TypeScript 기반 다중 쿼리 스트림 방어 파이프라인]
import { Target, QueryConsolidation } from '@eclipse-zenoh/zenoh-ts';

try {
    // Timeout 3초 방어벽 설정. 3초 지나면 무자비하게 연결을 절단하고 넘어감!
    const replies = await session.get("factory/robot_*/battery", {
        target: Target.BestMatching,
        consolidation: QueryConsolidation.None // 중복 응답도 필터링 없이 쾌속으로 던져라
    });
    
    for await (let reply of replies) {
        // 응답이 오는 족족 1대씩 화면에 비동기로 도장 찍기
        renderBattery(reply.keyExpr.toString(), reply.sample.value);
    }
} catch (e) {
    // 7번 로봇의 물리적 단절 에러가 튀어나오더라도 메인 스레드는 생존한다.
    console.warn("Some robots are offline. Continuing with arrived caches...", e);
}

분산 네트워크 세상에서는 ’절대 완벽한 100% 응답 데이터 모임’을 기대해서는 안 된다. 지연되는 연결은 과감하게 절단하고 현재 도달한 결과만으로 애플리케이션 라이프사이클을 매끄럽게 보존하는 타임아웃 관용 설계가 필수다.
