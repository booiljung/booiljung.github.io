1.6.1 리소스 표현과 계층적 키 경로(Key Path) 기반 라우팅

IP 어드레스를 버리고 ’이름(Name)’으로 통신하는 NDN(Named Data Networking)의 세계관에서 가장 중요한 설계 자산은 ’키(Key, 또는 Key Expression)’를 어떻게 지을 것인가이다. 이 이름 짓기가 엉망이면 라우터의 람다 식과 필터링이 박살 나 트래픽 혼잡(Broadcast Storm)이 유발된다.

0.1 URI 스타일의 리소스 추상화

디바이스가 내뱉는 각 데이터는 운영체제의 디렉토리 트리나 REST API의 URL 경로처럼 철저한 계층 구조를 갖추어야 한다.
예시: factory/zoneA/robot1/sensors/lidar

만약 어떤 로직이 factory/*/robot1/** 이라는 표현식으로 구독(Subscribe)을 신청했다 치자. Zenoh의 라우팅 엔진 트리(Trie)는 이 와일드카드 문자열의 위력을 십분 발휘하여 다음과 같이 즉시 길을 연다.

  • * (애스터리스크 1개): 해당 뎁스(Depth)의 단 1단계 경로만 아무거나 매칭. (예: zoneA 통과, zoneB 통과)
  • ** (애스터리스크 2개): 깊이에 상관없이 하위 모든 디렉토리를 무제한 매칭. (예: /sensors/lidar, /battery 모조리 끌어옴)

0.2 [Runbook] 불량 Key 네이밍에 의한 보안 붕괴 및 병목 차단

공장 온도를 /temp/factory1처럼 분류 기준을 소분류(temp)부터 대분류(factory)로 뒤집어 작성하는 아마추어 같은 실수는 치명적이다. 나중에 /temp/** 로 구독을 걸면, 엉뚱한 사무실(office)과 서버룸의 온도 패킷까지 글로벌 라우터 백본으로 모조리 빨려 올라와 데이터 병목이 파열된다.
