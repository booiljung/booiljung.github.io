1.6.4 데이터 저장과 캐싱을 통합하는 분산 스토리지 개념

Zenoh의 가장 이질적이면서도 파괴적인 철학은 네트워크 통신사(Carrier) 주제에 스스로를 거대한 데이터베이스 서버(Storage)처럼 작동토록 아키텍처를 뒤섞어 놓았다는 것이다.
클라우드 라우터(zenohd) 데몬 백그라운드에 RocksDB, InfluxDB 등의 플러그인을 레고 블록 끼우듯 톡 붙이는 순간, 그 라우터는 단순한 스위칭 허브(Hub)가 아니라 지나가는 트래픽을 모두 암기해 버리는 **블랙박스(캐시 스토리지)**로 각성한다.

0.1 Late-Joiner (지각생 합류) 문제의 완벽 봉합

시스템 구동의 절대 순서 종속성인 “서브스크라이버가 무조건 퍼블리셔보다 먼저 켜져 있어야 데이터를 놓치지 않는다“는 미들웨어계의 공식을 박살 냈다.
에지의 로봇이 오전 9시에 [위치: 강남, 속도: 20] 이라고 한 번 Pub을 치고 통신을 껐다.
오후 1시에 늦잠에서 깨어난 클라우드 관제 대시보드가 Get 쿼리를 때리면, 로봇은 이미 없지만 라우터 백업 스토리지가 대리 만족(Proxy Reply)하여 과거 오전 9시의 데이터를 꺼내 던져준다.

0.2 [Runbook] 데이터 오염 및 무한 팽창(OOM) 방어 - TTL 캐싱 전략

포그(Fog) 라우터에 스토리지를 구성해 뒀더니, 센서가 초당 1만 번씩 때리는 Lidar 더미 데이터를 스토리지에 무한정 디스크 라이팅(Disk Writing)하다 라우터의 하드디스크가 터지고 Write-lock에 걸려 라우터 프로세스가 마비(Freeze) 되는 지옥이 열린다.
보안 카메라나 로봇 조향 값 같은 데이터는 과거 1시간 치를 보관할 이유가 전혀 없는 극휘발성 데이터임에도 기본 무한 보존(Keep-alive) 룰이 적용된 끔찍한 사고다.

회피 전술: 인메모리(Memory) 기반의 고속 캐시 backend_memory 플러그인을 활용하고 데이터 패킷의 생명주기(Time to Live)를 강제로 옥죄어라.

// 포그 라우터 zenohd.json5 내 초고속 메모리 백업 아키텍처
plugins: {
  storage_manager: {
    volumes: {
      // 디스크를 쓰지 않는 초고속 Ramdisk 캐싱 공간
      memory: { } 
    },
    storages: {
      latest_sensor_cache: {
        key_expr: "sensor/fast_data/**",
        volume: "memory"
        // 최신 데이터 1건의 복사본만 기억하고, 밀려오는 센서 데이터는 옛날 걸 과감히 갈아치운다 (OOM 방어 절대 요새 방벽)
        // (세부 옵션은 RocksDB 백엔드의 rolling time 등을 백엔드 특화 형태로 튜닝)
      }
    }
  }
}

캐시 스토리지는 만병통치약이 아니다. 이 기능을 방만하게 운영하면 데이터베이스 I/O 병목이 전체 네트워크 통신 속도를 기어 다니게 만드는 질식의 목줄이 된다. 철저히 휘발성 메모리(RAM)와 영속성 디스크(DB)를 이원화(Sharding) 분리하여 트래픽의 본질에 맞게 격리해야 승리한다.
