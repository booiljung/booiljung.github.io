1.6.2 고성능 퍼블리시-서브스크라이브(Publish-Subscribe) 모델

1초에도 수백 번 수치가 변하는 속도 센서(RPM), 충돌 방지 라이다 스캔 데이터의 본질은 “내가 상태를 바꿨으니 이걸 필요한 놈들은 즉시 알아서 다 가져가라“는 방송(Broadcast) 메커니즘이다. 이를 위해 Zenoh는 Pub/Sub을 엔진 코어의 1 기통으로 품고 있다.

0.1 밀어내기(Push) 통신의 극단적 장점

Publisher(발행자) 측은 Subscriber(구독자)가 지구상에 존재하는지조차 모른다. 누가 듣든 안 듣든 자신의 Key가 매핑된 포트(Wire)에 페이로드를 우겨넣고 돌아선다. 반대로 구독자들은 미리 라우터에게 “이 키(Key)가 오면 내게 복사본을 쏴줘“라고 가입(Subscription)된 상태다. 양쪽 시스템이 비동기적으로(Asynchronous) 독립 가동되기에 마이크로서비스 확장에 거침이 없다.

0.2 [Runbook] 실전 Pub/Sub QoS 혼잡 제어 및 튜닝 대응

자율주행 셔틀이 와이파이가 시원찮은 음영 창고로 들어갔다 치자. 클라우드로 올리는 /robot1/video_stream 퍼블리셔가 패킷을 쏘는데 인터넷이 막혀버렸다.
로컬 네트워크 버퍼에 1초, 2초 치 영상 프레임 큐(Queue)가 산더미처럼 쌓여 터지려 한다! (Out of Memory 파열 조짐)

Zenoh 개발자가 반드시 통제해야 할 Pub 옵션이 바로 Congestion Control (혼잡 제어) 다. 퍼블리시를 선언할 때 이 파라미터 방패를 들지 않으면 시스템은 좀비가 된다.

// [오답] 혼잡 제어 부재 - 네트워크가 막히면 큐가 폭발할 때까지 TCP 단에서 무한 대기(Block) 
let publisher = session.declare_publisher("robot1/video_stream").res().await.unwrap();

// [정답] 로보틱스 생존술 (Drop)
// "네트워크 줄이 막혔냐? 그럼 무조건 지금 막 찍은 카메라 프레임 1개만 살리고 
// 버퍼링 된 나머지 옛날 영상은 큐에서 싹 다 버려라(DROP)!"
let publisher = session.declare_publisher("robot1/video_stream")
    .congestion_control(zenoh::publication::CongestionControl::Drop) // 핵무기급 방어책
    .res().await.unwrap();

publisher.put(latest_video_frame).res().await.unwrap();

비디오 스트리밍이나 로봇 조이스틱 컨트롤의 딜레이(지터, Jitter)는 치명적이다. 3초 늦게 수신된 옛날 데이터 10개보다, 앞에 것 다 버리더라도 방금 발생한 “최신 프레임 1개“를 살려내는 Drop 파라미터 세팅은 당신의 로봇이 벽에 때려 박는 참사를 막아줄 유일한 브레이크다.
