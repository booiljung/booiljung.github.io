1.8.1 이기종 시스템 통합에 따른 아키텍처 복잡성 감소

과거의 IT 환경에서는 하나의 완벽한 시스템을 만들기 위해 각각의 영역에서 검증받은 수십 개의 솔루션들을 모듈처럼 기워 붙이는(Patchwork) 방식을 택해야만 했다. 로보틱스의 내부 통신에는 DDS 프레임워크가 필요했고, 그 데이터들을 로컬 컨트롤 타워망으로 전송할 때는 MQTT 브로커(Broker) 인프라를 구축해야 했다. 또 이를 데이터 베이스에 쌓을 때는 AMQP나 HTTP REST API를 별도로 설치하여 운용해야만 했으며, 결과적으로 엔지니어는 3개 내지 4개의 상호 호환되지 않는 미들웨어를 익히고 관리해야 하는 지옥과 같은 아키텍처 스파게티(Architecture Spaghetti)에 갇혀버렸다.

이러한 수많은 이기종(Heterogeneous) 통신망들은 각기 완벽하게 다른 방식의 보안, 인증서 관리, 네이밍 체계, 그리고 튜닝 전략을 개발자에게 강제한다. 통신 문제가 하나 터질 때마다 어느 구간의 프로토콜에서 생긴 병목인지 추적하기조차 여간 까다로운 일이 아니다.

Zenoh가 이 기형적 아키텍처 복잡성을 어떻게 획기적으로 줄여줄까?
Zenoh 단 하나로, 개발자는 로보틱스와 IoT, 클라우드 그리고 저장소의 백엔드까지 모든 구간의 프로토콜 매커니즘을 오직 ‘Zenoh C/C++/Rust 클라이언트’ 기반의 단일 네이밍 트리 하나로 통일할 수 있다.

설령 레거시 시스템 때문에 MQTT와 DDS 모듈을 완전히 버리지 못한다 하더라도, Zenoh의 플러그인 아키텍처가 중간에서 이들을 자사망으로 포용하고 매끄러운 단일 통신망처럼 에뮬레이션(Emulation) 처리해 준다. 결국 유지보수를 담당할 시스템 아키텍트의 두뇌에서는 “A 프로토콜에서 B 프로토콜로 매핑(Mapping)하는 규칙“과 지식들이 모조리 생략되며, 이 장엄한 추상화는 실무진의 개발 피로도를 제로(Zero)에 수렴하게 만들어 준다.
