1.3.3 마이크로컨트롤러부터 데이터센터까지 아우르는 연속성(Continuum)의 필요성

일반적으로 시스템을 짤 때 배터리가 제한된 마이크로컨트롤러(MCU) 칩셋 위에는 C 기반의 초경량 zenoh-pico를 얹고, 클라우드 거대 서버군에서는 Rust 기반의 거대한 zenohd 라우터를 운영한다. 여기서 **연속성(Continuum)**이란, 이 두 극단의 체급 차이에도 불구하고 데이터 교환 언어(API)가 완벽히 동일하다는 철학을 뜻한다.

0.1 레거시 분절화(Fragmentation) 아키텍처의 악몽

과거엔 이렇다.

  1. MCU는 C언어로 CoAP 프로토콜을 사용해 온도 전송.
  2. 공장 포그(Fog) 엣지 서버의 파이썬 데몬이 CoAP를 받아 JSON으로 파싱 한 후 MQTT로 변환.
  3. 클라우드의 Kafka 백엔드가 MQTT를 받아다 다시 gRPC로 마이크로서비스에 뿌림.

이 통신 사슬의 중간 번역 서버 프로세스 코드만 3배로 생성된다! 디버깅은 거의 불가능하다.

0.2 Zenoh Continuum 아키텍처의 우아한 구현

Zenoh는 변환기(Translator)를 없앴다. 프로토콜 하나가 사이즈 조절(Scale up & down)만 될 뿐이다.

  • 초소형 엣지 노드 (C API - zenoh-pico): z_put("sensor/temp", 25);
  • 클라우드 데이터센터 통합 관제 (Rust API): let sample = session.get("sensor/temp").await?;

중간 포그(Fog)망의 라우터조차 “데이터 내용물을 뜯어보고 헤더를 변환하는 로직” 없이 단순히 데이터 바이트의 스위칭 역할만 묵묵히 수행한다(End-to-End 원리). 1바이트짜리 램덤 메모리를 쓰는 MCU의 코드 구조와 구글 클라우드 기반 쿠버네티스 파드의 코드 아키텍처가 “단순한 Key 조회 Put/Get/Sub 구조” 문법 아래 한 몸으로 묶이는 것. 이 연속성이야말로 Zenoh가 설계해낸 가장 우아한 패러다임의 산물이다.
