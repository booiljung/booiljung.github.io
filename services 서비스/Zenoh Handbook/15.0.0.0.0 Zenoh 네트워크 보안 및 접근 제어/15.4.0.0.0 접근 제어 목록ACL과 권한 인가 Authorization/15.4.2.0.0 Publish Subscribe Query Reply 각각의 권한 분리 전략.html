15.4.2 Publish, Subscribe, Query, Reply 각각의 권한 분리 전략

Zenoh 통신망을 지나는 메시지는 4개의 계급(Operation Type)으로 철저히 나뉘어 통제된다. “읽기 권한“을 줬다고 해서 “구독 권한”, “응답 권한“까지 한꺼번에 열리는 어리석음은 없다.

0.1 [Runbook] 프로토콜 행위 통제(Operation Sandbox) 전술

1. 생산자(Sensor)의 모래상자
온도 센서는 데이터를 허공에 쏘기만 하면 된다(Put).

  • 인가 권한: 오직 publisher (Put 통과).
  • 센서가 해킹당해 “현재 온도가 뭐지?” 하고 역으로 Get 쿼리를 때리거나 몰래 남의 온도 데이터를 빼가려고 Sub (구독) 소켓을 여는 순간, 라우터가 에지 단말의 접속을 강제로 찢어버린다.

2. 소비자(Dashboard)의 모래상자
관제 웹 모니터는 데이터를 듣기만 해야 한다.

  • 인가 권한: 오직 subscriber (Sub 통과), querier (Get 통과).
  • 대시보드를 보는 인턴이 브라우저 콘솔창을 켜고 z_put "robot/motor/speed" "200" 을 날려 로봇을 급발진시키려 해도, 쓰기 권한이 없으므로 그 패킷은 라우터를 향해 날아가는 즉시 소멸한다.

3. 스토리지 엔진(Backend)의 모래상자
DB 저장용 스토리지 플러그인조차 이 권한의 영향을 받는다.

  • 인가 권한: 남의 쿼리에 대답해야 하므로 replier (Reply 통과) 가 반드시 필요하다. 데이터 주인의 자리표를 점유하는 오직 허락된 자(Storage)만이 대답할 수 있다.
