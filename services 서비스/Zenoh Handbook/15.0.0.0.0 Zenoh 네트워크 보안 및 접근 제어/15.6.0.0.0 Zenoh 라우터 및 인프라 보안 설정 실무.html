15.6 Zenoh 라우터 및 인프라 보안 설정 실무

지금까지 배운 암호화(TLS) 와 권한 제어(ACL) 가 논리적인 자물쇠였다면, 이 장에서는 클라우드 엔지니어가 직접 스패너를 들고 서버 인프라(OS, 네트워크 카드) 의 밸브를 물리적으로 잠가버리는 디바이스 하드닝(Device Hardening) 런북을 전개한다.

개발자가 아무리 훌륭한 암호화 코드를 짰어도, 인프라 관리자가 라우터 설정 파일을 잘못 여는 순간 모든 데이터는 공중으로 흩어진다. 클라우드와 에지(Edge) 에서 라우터 프로세스를 띄울 때 반드시 지켜야 할 목숨과도 같은 설정 원칙들을 해부한다.

1. 라우터 설정 파일(zenoh.json) 내 보안 파라미터 최적화

zenohd.json5 파일은 라우터의 뇌 구조다. 디폴트(Default) 설정으로 라우터를 프로덕션(Production) 에 돌리는 것은 자살 행위다.

1.0.1 [Runbook] 데몬 프로덕션 하드닝 전술

1. 멀티캐스트(Multicast) 스캐닝 물리적 차단
Zenoh는 친절하게도 근처의 라우터를 자동으로 찾아주는 스카우팅(Scouting) 기능이 기본적으로 켜져 있다. 이는 해커의 망 탐색(Reconnaissance) 에 완벽한 타겟팅 시스템을 제공하는 꼴이다.

// zenohd.json5
scouting: {
  // 프로덕션 에지 환경에서는 주변 스캐닝을 무조건 꺼라(false)!
  multicast: { enabled: false }
}

2. 고정 피어(Fixed Peer) 토폴로지 강제
누군지 모를 우연한 연결(Scouting) 을 죽였다. 이제 데몬은 내가 명시한 좌표(관제탑 IP) 로만 정확히 레이저 통신을 연결해야 한다.

// 오직 인증된 마스터 서버의 주소만 박아 넣는다.
connect: [
  "tls/198.51.100.5:7447"
]

3. 메모리 제한 (Max Memory) 방어
해커가 10기가바이트짜리 더미 페이로드를 Put 으로 던졌을 때, 라우터가 이것을 다 버퍼에 담으려다 OOM(Out of Memory) 으로 폭사하는 것을 막아야 한다.

transport: {
  link: {
    scout: {
      // 10MB 이상의 패킷이 통째로 들어오면 소켓을 즉시 찢어버려라!
      max_message_size: 10485760 
    }
  }
}

이 3줄의 설정만으로도, 라우터는 외부의 무작위 스캐닝에 반응하지 않는 “스텔스(Stealth) 요새” 로 탈바꿈한다.

2. 리스너(Listener) 바인딩 제한 및 네트워크 인터페이스 격리

컴퓨터에 랜 카드가 2개(인터넷망 하나, 로봇 내부 내부망 하나) 꽂혀 있다. 라우터를 켰을 때 어느 랜 카드로 통신을 열어둘 것인가?

2.0.1 [인스펙션] 0.0.0.0의 대학살과 인페이스(Interface) 고립 전술

가장 초보적인 실수는 listen: ["tcp/0.0.0.0:7447"] 이라고 적는 것이다.
0.0.0.0 은 “이 컴퓨터에 달린 모른 랜 카드의 7447 포트를 전부 다 열어라” 라는 뜻이다. 로봇 내부 부품끼리만 은밀하게 대화하려던 통신선이, 바깥 인터넷까지 싸그리 노출되는 참상이 벌어진다.

1. 로컬 호스트(Localhost) 감금
관제 센터의 백엔드용 라우터나, 로봇 내부 컴포넌트 간의 브릿지 용도로만 라우터를 띄운다면, 타겟 반경을 하드디스크 밖으로 벗어나지 못하게 묶어라.

// 오직 내 컴퓨터 안에 있는 프로세스들끼리만 접속 가능! 
// 바깥 세상(네트워크) 에서는 이 포트가 아예 닫힌 것으로 보인다.
listen: ["tcp/127.0.0.1:7447"]

2. 특정 랜 카드(NIC) 강제 타겟팅 지정
에지 컴퓨터에 5G 모뎀(외부망, 10.0.x.x) 과 라이다 센서 직결망(내부 랜, 192.168.1.1) 이 동시에 존재한다면?
외부에서 해커가 5G망을 뚫고 들어와 라이다 제어 포트를 조작하지 못하도록, 제어 라우터는 오직 격리된 내부 랜 카드의 IP 로만 소켓을 뚫어라.

// 라이다망(내부망) 만 이 라우터를 볼 수 있다.
listen: ["tcp/192.168.1.1:7447"]

OS 의 방화벽(iptables)조차 건드릴 필요 없이, Zenoh 레벨에서 인터페이스 바인딩을 잘라버리는 가장 우아하고 무자비한 네트워크 격리법이다.

3. 로컬 API 및 관리자 포트 접근 통제

Zenoh 라우터는 기동될 때 자신을 제어할 수 있는 막강한 REST API 관리자 포트를 포트 8000번에 생성한다. 이 포트가 뚫리면 라우터는 사실상 하이재킹(Hijacking) 당한 것이다.

3.0.1 [Runbook] 마스터 스위치(Master Switch) 암호화 런북

1. 관리자 포트 전면 폐쇄
가장 극단적이고 안전한 방식. 프로덕션 라우터에서는 아예 관리용 API 를 켜지도 마라.

// zenohd.json5
plugins: {
  // admin 블록 자체를 통째로 지우거나, 포트 바인딩 배열을 비운다.
  admin: {
    listen: [] // 관리자 접근 차단!
  }
}

2. 로컬 브릿지 및 HTTP Basic Auth 강제
그래도 핫-리로드(Hot-Reload) 등을 위해 관리자 포트를 열어야 한다면?
절대 외부망(0.0.0.0) 으로 열지 말고, 라우터가 깔린 방구석(Localhost) 으로만 포트를 열고 암호를 거둬드려라.

plugins: {
  admin: {
    listen: ["http/127.0.0.1:8000"], // 로컬 접속만 허용
    // admin 플러그인 자체에 접근 암호를 강제한다 (htpasswd 형식과 구조 동일 가능)
    auth: {
      password: "SuperSecretAdminPassword" 
    }
  }
}

이제 개발팀이 라우터를 원격 제어하려면 서버 컴퓨터에 SSH 로 터미널 접속을 뚫은 다음, 거기서 curl 명령어로 로컬 루프백(127.0.0.1) 을 질의하는 방식, 즉 이중 배스천(Bastion) 터널링을 통해서만 이 마스터 스위치에 손댈 수 있게 된다.
