15.2.3 상호 TLS (mTLS) 설정 및 노드 간 신뢰 구축

일반적인 TLS(웹사이트 HTTPS)는 클라이언트(브라우저)가 서버가 진짜인지 인증명단(CA)만 확인하고 끝난다. 클라이언트가 누군인지는 묻지 않는다.
**mTLS (Mutual TLS)**는 서버도 클라이언트에게 “네 신분증(인증서)도 내놔라“고 요구하는 양방향 멱살잡이다.

0.1 [Runbook] 제로 트러스트(Zero Trust) 노드 인증망 사수

해커가 tls/ROUTER_IP 로 연결해서 암호화 채널을 뚫었다 해도, 해커의 로트북에 우리가 발급한 “클라이언트 인증서“가 없으면 데몬은 그 암호화 채널을 곧바로 부숴버린다(Tear-down).

1. 라우터(서버) mTLS 강제 선언

transport: {
  link: {
    tls: {
      server_cert: "/etc/zenoh/server_cert.pem",
      server_key: "/etc/zenoh/server_key.pem",
      // 핵심: 네 신분증도 내놔라! 내가 만든 서명(CA)이 아니면 패죽인다!
      client_auth: true, 
      root_ca_certificate: "/etc/zenoh/my_company_root_ca.pem"
    }
  }
}

2. 클라이언트(센서) 기동 런북
로봇 안의 프로그래머가 코드를 짤 때도, 단순히 주소만 넣어서는 연결이 거부당한다. 클라이언트 인증키를 심어서 구동해야 한다.

// Rust 클라이언트
let mut config = zenoh::Config::default();
config.insert_json5("connect", r#"["tls/10.0.0.5:7447"]"#);
config.insert_json5("transport/link/tls/client_cert", r#""/robot1/cert.pem""#);
config.insert_json5("transport/link/tls/client_key", r#""/robot1/key.pem""#);

let session = zenoh::open(config).res().await.unwrap(); // 여기서 비로소 통과!

mTLS를 켜는 순간, 방화벽(IP) 따위는 전혀 필요 없어진다. 전 세계 어디서 공유기를 타고 구글 클라우드에 핑을 때리든 간에, 당신 회사의 Root CA 로 도장 찍힌 인증서 파일 2개가 없으면 Zenoh 망에 발 끝 하나 들일 수 없는 철옹성이 완성된다.
