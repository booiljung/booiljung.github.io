15.9 보안 감사(Audit) 및 컴플라이언스

해커의 접속을 끊는 방어(Defense) 만으로는 부족하다. 회사의 자산 관리팀(혹은 국가 규제 기간) 은 “작년 11월 3일에 1번 로봇의 카메라 영상(z_get) 을 조회한 직원이 누구인가?” 라는 무서운 질문을 던질 것이다.

보안 모델의 마지막 퍼즐은 튕겨나간 패킷들과 성공한 패킷들의 자국(Footprint) 을 지워지지 않는 돌판에 모조리 새겨 넣는 것이다. 라우터가 뱉어내는 파편화된 로그를 분석하고, 거대한 보안 감사 트레일(Audit Trail) 시스템을 엮어내며, 동시에 전 세계의 개인정보 보호법(GDPR) 을 우회하는 치명적인 데이터 분칠(Masking) 전술을 마스터한다.

1. 인증 실패 및 비정상 접근 시도 로깅

시스템이 누군가의 접근을 차단했다면 잘한 일이지만, 차단 로그를 남기지 않았다면 당신은 눈을 감고 맞고 있는 것과 같다.

1.0.1 [Runbook] 침투 지표(IoC, Indicator of Compromise) 확보 전술

Zenoh 코어 라우터를 구동할 때, 로깅(Logging) 레벨의 정밀한 조준이 필요하다.

1. 보안 레벨(Security Level) 로깅의 필터링
단순히 RUST_LOG=info 로 켜면 에지 로봇 1만 대가 1초마다 보내는 “연결됨/종료됨” 로그에 하드디스크가 3시간 만에 타버린다. 오직 “인증 모듈“과 “ACL” 에 대한 필터만을 핀셋으로 집어내야 한다.

## Auth(인증) 와 ACL 모듈에서 발생하는 WARN 급 이상의 에러만 로그 파일로 떨어뜨려라!
export RUST_LOG="zenoh::net::auth=warn,zenoh::plugin::acl=warn"
zenohd -c config.json5 > /var/log/zenoh_security.log 2>&1

2. 로그 분석 (Fail2Ban 연동)
이 로그 파일에는 [WARN] Auth failed for user 'admin_fake' from IP 198.51.100.99 라는 텍스트가 쌓인다.
이 텍스트 파일을 리눅스의 Fail2Ban 데몬이 실시간으로 읽게 하여, 1분 안에 로그인 실패가 5번 이상 찍히는 즉시 OS 단위(iptables) 에서 그 IP 를 1시간 동안 완전 블록(Drop) 시켜버리는 2중 방어망을 구축해야 한다.

2. 보안 감사 트레일(Audit Trail) 구축 및 중앙 집중화

1,000대의 로봇과 100대의 로컬 라우터에 분산된 텍스트 로그 파일. 해커가 침투한 뒤 rm -rf /var/log/zenoh 를 치고 나가버리면 증거는 영원히 종적을 감춘다.

2.0.1 [인스펙션] 로그의 스텔스(Stealth) 적출 전술

Zenoh 의 통신 패러다임을 이용해, 보안 로그 자체를 Zenoh 망에 던져버린다. (Eat your own dog food!)

1. 로깅 플러그인 텔레메트리 연동
라우터에서 발생한 접근통제 로그를 텍스트로 적는 대신, 라우터 내부에서 곧바로 sys.log.auth 라는 시스템 전용 토픽으로 퍼블리싱(Put) 하게 만든다.

  • 공격받는 즉시, 그 흔적이 에지(Edge) 의 디스크에 저장되기도 전에 클라우드 블랙박스(Elasticsearch / Splunk 서버) 로 빛의 속도로 복사된다.

2. 중앙 백엔드(RocksDB/InfluxDB) 의 불변 저장소(WORM) 구성
클라우드 관제탑에서 무한 용량의 시계열 스토리지 엔진(InfluxDB) 에 이 로그 전용 토픽(sys.log.**) 을 구독(Subscribe) 시켜 바닥에 때려 박는다.
이 스토리지 볼륨 영역은 최고 관리자조차 수정/삭제(Delete) 가 하드웨어적으로 불가능한 WORM (Write Once, Read Many) 설정으로 깡통을 막아 버린다. 이것이 국가 감사반이 와도 절대 훼손되지 않는 영원한 증거물의 완성이다.

3. GDPR 등 데이터 보호 규정 준수를 위한 페이로드 마스킹 전략

자율주행 자동차가 쏘는 영상(카메라) 속에는 신호등뿐만 아니라, 길을 걷는 행인들의 얼굴(Privacy) 이 초당 30프레임으로 찍혀 클라우드로 들어온다.
이를 그대로 Zenoh 망의 스토리지에 모아뒀다가 해킹당하거나 외부에 열람되면, 회사는 GDPR 위반으로 수천 억 원의 벌금을 맞고 파산한다.

3.0.1 [Runbook] 데이터 즉결 검열(Masking) 파이프라인 전술

로봇 단에서 암호화조차 하지 못한 평문 데이터라 하더라도, 클라우드의 일반 작업자(관제 화면) 와 스토리지 볼륨에 닿기 전에 “검열관” 을 통과시켜야 한다.

1. Zenoh-Flow 를 이용한 절단 파이프라인 (Data Cleansing)

  • 로봇은 원본 영상을 put "raw/camera/front" 에 쏜다.
  • 이 토픽은 오직 숨겨진 백그라운드 노드인 Zenoh-Flow 마스킹 워커(Worker) 만이 읽을 수(Sub) 있게 ACL 로 가둬버린다.
  • 마스킹 워커가 영상을 받아 AI 로 사람 얼굴에 모자이크를 씌우고, 차량 번호판을 지워버린 다음 put "safe/camera/front" 에 다시 퍼블리싱 한다.
  • 일반 관제사들이나 모니터링 데몬은 오직 "safe/**" 에만 접근할 수 있다.

2. 투-트랙 스토리지 (Two-Track Storage)
AI 팀의 훈련을 위해 언젠가 원본(raw/**) 이 필요할 수도 있다.
이때는 원본 데이터(raw/**) 전용 스토리지 볼륨만을 핀셋으로 따서, 인터넷이 끊어진 오프라인 AWS S3 글래시어(Glacier) 이중 암호화 금고 로 바로 직결시켜 파묻는다. 망에 실시간으로 남겨두고 돌아다닐 데이터(Safe) 와, 법적인 근거로 파묻어두는 데이터(Raw) 를 네트워크 라우팅 계층에서부터 영구히 격리(Isolation) 시키는 컴플라이언스(Compliance) 아키텍처다.
