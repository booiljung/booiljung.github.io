15.2 전송 계층 보안 (Transport Layer Security)

공용 인터넷과 5G망을 넘나드는 Zenoh의 데이터들은 누군가 마음만 먹으면 안테나로 떠서 다 들여다볼 수 있는 엽서(Postcard)와 같다. 기밀성을 확보하려면 이 엽서를 뜯어볼 수 없는 티타늄 금고에 봉인하여 배송해야 한다.

이 장에서는 어플리케이션(Application) 계층을 건드리지 않고, 오직 네트워크 소켓 연결(Socket Connection) 자체를 도청 불가능한 암흑의 터널로 탈바꿈시키는 **전송 계층 보안(TLS와 QUIC)**의 실전 구축 런북을 파헤친다.

1. TLS 기반 통신 암호화 메커니즘

Zenoh 라우터에 TCP로 연결한다는 것은 벌거벗은 채로 광장에 서는 것과 같다. tls 스킴(Scheme)을 강제하여 철갑을 둘러야 한다.

1.0.1 [Runbook] 평문 통신의 전면 금지(Deprecation) 전술

이 세팅 이후로 라우터는 인증서가 없는, 혹은 암호화되지 않은 모든 TCP 연결 자체를 소켓 핸드쉐이크 단계에서 걷어차 버린다(Kick-out).

1. 라우터의 TLS 인증서 장착
라우터가 자신을 증명할 서버 인증서(cert.pem)와 비밀 키(key.pem)를 준비하고 로드한다.

// zenohd.json5
listen: [
  // 1. 기존 tcp 포트는 주석 처리하여 완전히 버린다!
  // "tcp/0.0.0.0:7447", 
  // 2. 오직 tls 포트만 개방한다.
  "tls/0.0.0.0:7447"
],
// TLS 플러그인 설정
transport: {
  link: {
    tls: {
      server_cert: "/etc/zenoh/cert.pem",
      server_key: "/etc/zenoh/key.pem",
    }
  }
}

2. 도청의 물리적 무력화
이제 클라이언트(센서)는 tls/ROUTER_IP:7447 로 연결을 시도한다. 연결이 수립되면 해커가 스니핑(Wireshark)을 하더라도 보이는 것은 의미 없는 노이즈(Noise)뿐이다.
단, tls를 켜는 순간 라우터의 CPU 연산 부하는 암·복호화로 인해 평문 통신 대비 최소 10~20% 상승한다. 전기가 부족한 드론(Edge) 배터리를 깎아 먹으면서 기밀성을 얻어내는, 아키텍트의 치명적인 자원 교환 법칙이다.

2. QUIC 프로토콜의 내장 보안 기능과 성능 이점

TCP 위에 TLS를 올리는 건 구시대의 유물이다.
Google이 만든 차세대 프로토콜 QUIC 은 UDP 기반이면서도 “암호화가 선택이 아닌 강제(Built-in)“인 기형적이면서 완벽한 아키텍처를 자랑한다.

2.0.1 [인스펙션] QUIC의 극강 암호화 퍼포먼스

Zenoh는 네트워크 스펙 최상단에서 완벽하게 QUIC 을 네이티브 지원한다.

1. 핸드쉐이크(Handshake) 딜레이 소멸

  • TCP + TLS: 소켓 연결(1-RTT) 후 다시 암호화 키 교환(1~2-RTT)에 총 3번의 왕복 지연이 발생한다.
  • QUIC: 연결 수립과 동시에 암호화 키 교환을 끝내버린다(1-RTT, 혹은 0-RTT). 로봇이 터널에서 나와 통신망을 복구할 때, QUIC 은 TCP 보다 3배 빠르게 암호화 채널을 뚫고 데이터를 쏘기 시작한다.

2. QUIC 바인딩 런북

// zenohd.json5
listen: [
  // tcp나 tls가 아닌 quic 스킴 사용!
  "quic/0.0.0.0:7447"
],
transport: {
  link: {
    // quic 은 내부적으로 무조건 tls 환경변수를 공유해서 쓴다.
    // 인증서가 없으면 아예 quic 소켓 자체가 에러를 뿜으며 안 켜진다.
    quic: {
      server_cert: "/etc/zenoh/cert.pem",
      server_key: "/etc/zenoh/key.pem"
    }
  }
}

보안을 지키면 속도가 느려진다는 IT의 대원칙을 박살 낸다. TCP+TLS보다 빠르면서도 완벽한 기밀성을 보장하는 가장 폭력적인 성능의 스윔라인(Swimlane)이 바로 QUIC 통신이다.

3. 상호 TLS (mTLS) 설정 및 노드 간 신뢰 구축

일반적인 TLS(웹사이트 HTTPS)는 클라이언트(브라우저)가 서버가 진짜인지 인증명단(CA)만 확인하고 끝난다. 클라이언트가 누군인지는 묻지 않는다.
**mTLS (Mutual TLS)**는 서버도 클라이언트에게 “네 신분증(인증서)도 내놔라“고 요구하는 양방향 멱살잡이다.

3.0.1 [Runbook] 제로 트러스트(Zero Trust) 노드 인증망 사수

해커가 tls/ROUTER_IP 로 연결해서 암호화 채널을 뚫었다 해도, 해커의 로트북에 우리가 발급한 “클라이언트 인증서“가 없으면 데몬은 그 암호화 채널을 곧바로 부숴버린다(Tear-down).

1. 라우터(서버) mTLS 강제 선언

transport: {
  link: {
    tls: {
      server_cert: "/etc/zenoh/server_cert.pem",
      server_key: "/etc/zenoh/server_key.pem",
      // 핵심: 네 신분증도 내놔라! 내가 만든 서명(CA)이 아니면 패죽인다!
      client_auth: true, 
      root_ca_certificate: "/etc/zenoh/my_company_root_ca.pem"
    }
  }
}

2. 클라이언트(센서) 기동 런북
로봇 안의 프로그래머가 코드를 짤 때도, 단순히 주소만 넣어서는 연결이 거부당한다. 클라이언트 인증키를 심어서 구동해야 한다.

// Rust 클라이언트
let mut config = zenoh::Config::default();
config.insert_json5("connect", r#"["tls/10.0.0.5:7447"]"#);
config.insert_json5("transport/link/tls/client_cert", r#""/robot1/cert.pem""#);
config.insert_json5("transport/link/tls/client_key", r#""/robot1/key.pem""#);

let session = zenoh::open(config).res().await.unwrap(); // 여기서 비로소 통과!

mTLS를 켜는 순간, 방화벽(IP) 따위는 전혀 필요 없어진다. 전 세계 어디서 공유기를 타고 구글 클라우드에 핑을 때리든 간에, 당신 회사의 Root CA 로 도장 찍힌 인증서 파일 2개가 없으면 Zenoh 망에 발 끝 하나 들일 수 없는 철옹성이 완성된다.

4. 요구 보안 수준에 따른 암호화 스위트(Cipher Suites) 선택 가이드

TLS를 켰다고 방심하지 마라. 당신의 라우터가 “아유, 구형 기기도 받아줄게~” 라며 RSA 기반의 허접한 옛날 암호 알고리즘을 허용해버리면, 양자 컴퓨터나 강력한 최신 GPU 에 의해 데이터가 실시간으로 뚫린다.

4.0.1 [인스펙션] 알고리즘 군단의 강제 절단 전술

Zenoh 네트워크의 타협 없는 기밀성 표준(Standard)을 수립하라.

1. TLS 1.3 강제화 런북
과거의 찌꺼기(TLS 1.1, 1.2)는 버린다.

  • 가끔 에지(Edge)에 10년 전 모델 칩셋이 있어 TLS 1.2를 써야 한다는 엔지니어가 있다면 그 에지 디바이스 전체를 버려야 한다. 보안 아키텍처의 하향 평준화는 곧 망 전체의 궤멸이다.
  • Zenoh의 내부 옵션을 수정하거나 프록시(Envoy/Nginx)를 덧대어 무조건 TLS 1.3 Only 정책을 고정한다.

2. AES-256-GCM / ChaCha20-Poly1305 교차 전략

  • 엔비디아 Jetson / 인텔 서버: AES-NI (하드웨어 암호화 가속기) 가 칩에 박혀있다. 무조건 AES-256-GCM 알고리즘 세트를 쓴다. CPU의 0.1% 도 쓰지 않고 100Gbps 암호화를 수행한다.
  • 초저전력 ARM 코어 (가속기 없음): AES 연산을 소프트웨어로 돌리면 CPU가 타버린다. 이때는 모바일 환경을 위해 설계된 ChaCha20-Poly1305 가 훨씬 빠르고 가볍다.

에지 컴퓨팅에 있어 암호화 스위트(Cipher Suite)의 결정은 단순한 보안의 영역을 넘어, 로봇의 배터리 방전 속도와 디바이스 CPU 온도를 결정짓는 피지컬(Physical) 엔지니어링의 영역이다.

5. 인증서 만료 및 갱신 파이프라인 자동화

TLS 환경 구축 후 가장 많은 시스템 크래시 원인은 해킹이 아니라 “인증서 만료(Certificate Expiration)“다. 인증서가 죽는 순간, 해커보다 더 잔혹하게 Zenoh 데몬이 에지의 접속을 칼같이 잘라버리고 오프라인 미아로 만들어버린다.

5.0.1 [Runbook] 데스 클럭(Death Clock) 무력화 전술

클라우드와 달리 물리적으로 회수할 수 없는 수천 대의 로봇 인증서를 자동으로 살려내는(Renewal) 파이프라인이다.

1. SCEP(Simple Certificate Enrollment Protocol) 에이전트 결합

  • 로봇에는 만기 1년짜리 인증서가 있다. 만료 30일 전, 로봇 내부의 백그라운드 크론(Cron) 스크립트가 깨어난다.
  • 에이전트는 독자적인 암호화 터널(VPN 등) 을 뚫고 본사의 PKI 서버에 접속해 새 인증서를 발급받아 로컬 디스크 리눅스 권한 영역에 덮어쓴다(cert_new.pem).

2. 데몬 무정지 핫-리로드 (Hot-Reload) 설계
가장 치명적인 부분이다. 새 인증서를 다운받았어도 Zenoh 데몬을 껐다 켜면(Restart) 통신이 1초간 끊긴다.

  • Zenoh 는 향후 혹은 플러그인을 통해 설정 파일 변경을 감지하는 핫-리로드(Hot-Reload)를 적용해야 한다.
  • 또 다른 전술로, 라우터단 앞단에 Nginx / Envoy 기반의 리버스 프록시(Reverse Proxy)를 덧대어, 이 프록시가 TLS 종단(Termination)과 인증서 리로드를 무정지로 전담하고, 프록시와 Zenoh 라우터 간에는 127.0.0.1 평문으로 넘겨버리는 강제 분리(Decoupling) 아키텍처를 구사한다. 관제탑 서버에서는 이 방식이 트러블슈팅의 복잡도를 무한히 낮추는 최고의 패턴이다.
