15.3 노드 및 사용자 인증 (Authentication)

문을 잠그는 것(TLS)과, 문을 두드리는 사람이 누구인지 얼굴을 확인하는 것(인증)은 전혀 다른 차원의 문제다.

Zenoh 네트워크는 무수한 익명의 패킷들이 날아다니는 난장판이다. 누군가는 “접니다” 하고 나타났을 때, 라우터 데몬(Router Daemon)이 그자의 신분을 완벽히 까발리고 입장장을 끊어주는 인증(Authentication) 메커니즘을 해부한다. 이 런북(Runbook)을 통해 허가받지 않은 모든 좀비 노드(Zombie Node)들을 네트워크 입구에서 모조리 사살(Kill)시켜라.

1. 인증(Authentication)과 인가(Authorization)의 구조적 분리

신분증을 확인하는 데스크와, “비밀금고” 출입 열쇠를 주는 데스크는 완벽하게 쪼개져야 한다.

1.0.1 [인스펙션] 쌍두마차 격리 통제 전술

Zenoh 코어는 이 두 가지를 단 하나의 시스템 플러그인에서 처리하지 않고 2개의 계층(Layer)으로 찢어 놓았다.

1. 1차 관문: 인증 (AuthN, Authentication)

  • “너는 누구인가?”
  • 가장 밖에서 작동한다. 접속자가 보낸 패킷이 암호(Password) 건 인증서(TLS Certificate) 건 간에, 그 값이 1비트라도 틀리면 즉각 TCP/QUIC 소켓 연결 자체를 드랍(Tear-down)시킨다.
  • 연결 단계(Session Establish)에서만 1회성으로 벌어지는 검문이다.

2. 2차 관문: 인가 (AuthZ, Authorization / ACL)

  • “들어온 네 녀석이 어느 서랍을 열 수 있는가?”
  • 1차 관문을 통과한 클라이언트에게 “사용자 이름(Username)“이라는 딱지표(Token)를 부여한다.
  • 이후부터 이 클라이언트가 z_put 혹은 z_get 패킷을 날릴 때마다, 라우터 내부의 ACL 플러그인이 패킷 1개 1개 단위로 딱지표와 토픽 이름을 대조해 낸다(Access Control).

인증이 무너지면 시스템 전체가 뚫리지만, 인가가 무너지면 특정 데이터 베이스 한 개만 뚫리는 수준의 방어선 설계다. 아키텍트는 인증을 가장 무자비하게, 인가를 가장 촘촘하게 짜야 한다.

2. 비밀번호 및 사전 공유 키(PSK) 기반 인증

10년 된 구형 마이크로컨트롤러(MCU)에는 인증서를 파싱할 메모리 자원(RAM)조차 없다. 이들을 망에 들일 가장 원시적인 무기, 평문 비밀번호(Password) 인증법이다.

2.0.1 [Runbook] 레거시 호환 로그인 전술

가장 가볍지만 가장 멍청한 방법이다. 절대 평문 통신(15.2장 TCP) 위에서 이 방식을 켜서는 안 된다. 만드시 TLS 위에 올리되, 인증 수단만 Password로 열어두는 하이브리드(Hybrid) 패턴을 고수하라.

1. 라우터 유저 테이블 구축 (htpasswd)
리눅스 서버의 아파치 모듈(.htpasswd)처럼 암호화된 유저 명단 파일을 생성한다.

## 아파치 유저 테이블(Basic Auth) 생성기
htpasswd -B -c /var/lib/zenoh/users.passwd robot_alpha
## 패스워드 입력: "SuperSecret123!"

2. 데몬 Auth 플러그인 마운트

// zenohd.json5
plugins: {
  // 인증 플러그인
  auth_password: {
    // 만들어 둔 패스워드 테이블을 뒤져서 검사하라!
    passwd_file: "/var/lib/zenoh/users.passwd"
  }
}

이제 에지 로봇은 zenoh::Config 에 사용자 이름(robot_alpha) 과 암호(SuperSecret123!) 를 담아서 망에 접속해야만 입장권을 얻을 수 있다.
가장 원시적이지만 100만 대의 에지에서 코드를 한 줄 고치지 않고 즉각 도입할 수 있는 무적의 편의성(Convenience)을 보장한다.

3. X.509 인증서 기반의 강력한 엔드포인트 인증

비밀번호는 결국 사람(개발자)이 만든 깃허브 코드 조각 어딘가에 평문으로 남기 마련이다.
비밀번호를 아예 버리고 하드웨어(TPM) 에 박혀있는 수학적 인증서(X.509) 로만 신분을 증명하는 런북이다.

3.0.1 [인스펙션] 패스워드리스(Passwordless) 극강 방어망 전술

15.2 장에서 수행한 mTLS 구축이 곧 인증(Authentication) 의 궁극적 완성 형태다.

1. TLS Subject Name 파싱 (인증에서 인가로의 도약)
mTLS 로 연결이 통과되는 순간, Zenoh 인증 메커니즘은 클라이언트가 들고 온 509 인증서를 뒤져서 Common Name (CN) 문자열을 뜯어낸다.

  • 로봇이 가져온 인증서의 Subject 에 CN=drone_007 이라고 적혀있다면?
  • 라우터는 즉시 이놈의 아이디(Username) 를 drone_007 로 탕탕 못 박고(Authentication), 곧바로 2차 관문인 ACL(Authorization) 모듈에게 “drone_007 이니까 그에 맞는 서랍(Topic) 만 열어줘라” 라고 인가 지시를 던진다.

2. 무결성의 사슬 사수 (Chain of Trust)
누군가 자기 맘대로 인증서를 만들어서 CN=admin 이라 적고 들고 오면?
라우터는 가장 먼저 그 인증서의 발행소(Issuer) 가 우리 회사(Root CA) 의 도장으로 찍힌 게 맞는지를 수학적으로 암호화 검증한다. 우리 회사의 루트 서명이 없는 인증서는 그 즉시 TLS 계층(Socket Layer) 에서 물리적으로 터져(Cut) 버린다.

4. 커스텀 인증 플러그인 개발 및 외부 IdP(Identity Provider) 연동

회사가 이미 아마존 AWS 의 IAM, 구글 클라우드의 OAuth, 또는 화이트리스트 IP 서버스 등 자체 인증 시스템(IDP) 을 들고 있는 경우가 90% 이상이다.
여기에 억지로 .htpasswd 파일 계정이나 자체 인증서(CA) 를 이중으로 굴리는 것은 운영의 악몽(Nightmare) 이다.

4.0.1 [Runbook] 백도어 OIDC 로그인 전술

커스텀 백엔드(14.8 장) 를 짰던 것처럼, 이번엔 커스텀 인증 플러그인(Auth Plugin) 을 Rust 로 수술해서 라우터에 주입한다.

1. zenoh-plugin-trait 의 Auth_check 오버라이딩
Rust 로 플러그인을 짤 때, 외부 시스템을 찌르도록 코드를 바꾼다.

// 거대 IdP (카카오/아마존/사내 시스템) 연동
async fn check(&self, username: &str, password: &str) -> ZResult<bool> {
    // 1. 로봇이 보낸 패스워드를 "AWS Cognito Token" 취급한다!
    let my_auth_server = "https://my-company-sso.com/api/verify";
    // 2. 비동기 HTTP 요청을 때려 사내 시스템에 계정이 진짜인지 물어본다.
    let res = reqwest::Client::new()
        .post(my_auth_server)
        .json(&serde_json::json!({ "token": password }))
        .send().await;

    // 3. 사내 서버가 "OK(200)" 하면 라우터 문을 열어준다(true).
    if res.status().is_success() { Ok(true) } else { Ok(false) }
}

이 플러그인 하나로 수천 대의 로봇과 수백 명의 관제 개발자는 별도의 비밀번호를 발급받을 필요 없이, 평소 사내망에서 쓰던 사원증(SSO Token) 만으로 전 세계의 분산 Zenoh 네트워크를 마음껏 드나드는 완전 통합 인프라망을 이룩하게 된다.
