15.8 네트워크 위협 방어 및 완화 (Mitigation)

보안의 가장 무서운 적은 똑똑한 해커가 아니다. 무식하게 힘으로 우주 방어선을 찢고 들어오는 힘의 논리, 즉 자원 고갈 공격(DDoS)과 좀비 노드의 인해전술이다.

서버가 암호화를 푸느라 CPU 100%를 찍고 폭주하거나, 1년 전의 데이터를 계속 다시 쏴서 로봇 시스템을 과거로 되돌려버리는 물리적 공격들에 직면해야 한다. 이 장에서는 라우터가 압도적인 불법 트래픽을 만났을 때 시스템을 어떻게 우아하게 마비시키지 않고 좀비들을 도륙해 낼 것인가에 대한 네트워크 방어 및 완화(Mitigation) 런북을 완성한다.

1. DoS/DDoS 공격 방어를 위한 트래픽 속도 제한(Rate Limiting)

인가된 계정을 탈취한 해커가, 합법적인 Put 패킷을 1초에 1,000만 번 쏜다. ACL 도 통과하고 TLS 도 통과한 합법적 트래픽이지만, 백엔드 DB 는 그 압력에 터져버린다.

1.0.1 [Runbook] 트래픽 스로틀링(Throttling) 방어 런북

Zenoh 의 통신 계층에서 대역폭을 물리적으로 잘라서 옥죄어야 한다. 방어를 외주(Proxy) 주지 말고 라우터 본체에서 썰어버린다.

1. 소켓 버퍼(Socket Buffer) 및 QoS 브레이크

// zenohd.json5
transport: {
  link: {
    tls: {
      // 1. 읽기/쓰기 버퍼를 최소한으로 고립시킨다.
      rx_buffer_size: 65536,
      tx_buffer_size: 65536
    }
  },
  // 2. 백프레셔(Backpressure) 시스템 가동
  multicast: {
    // 큐가 가득차면 무조건 오래된/넘치는 패킷을 Drop 하라!
    queue_length: 1000 
  }
}

2. 에지 퍼블리셔(Publisher) 단의 흐름 제어
zenoh::Config 에서 퍼블리셔를 뚫을 때 메인 함수를 비동기로 짜서 혼잡(Congestion) 제어를 시도하라. 해커의 맹점은 혼잡 회피 메커니즘을 끄고 패킷을 융단 폭격한다는 점이다. 라우터가 “잠깐만! 수신 큐 꽉 찼어!” 라고 백프레셔 레이블(QoS) 을 던지면 그 즉시 해당 IP 의 전송 채널(Pipe) 전체에 강제 휴식 스톨(Stall) 을 먹여 디스크 연산이 따라갈 숨구멍을 틔워내야 한다.

2. 메시지 타임스탬프와 Nonce를 활용한 재생 공격(Replay Attack) 방어

관제탑이 “엔진 정지(STOP)” 명령을 내렸다. 해커는 이 명령이 암호화(TLS) 되어 있어 내용은 못 보지만, 암호화된 Payload 통뼈 자체를 복사해서 저장해 뒀다.
다음 날, 로봇이 잘 날아가고 있을 때 해커가 그 암호화된 통뼈를 그대로 Zenoh 망에 다시 투척한다(Replay). 로봇은 뜬금없이 “어제 서버가 서명한 진짜 정지 명령이네?” 하고 추락한다.

2.0.1 [인스펙션] 시간 고정(Time-Locked) 일회용 패킷 전술

메시지가 두 번 소비되는 것을 시스템적으로 거세해야 한다.

1. Timestamp 와 Nonce(1회용 난수) 의 결합

  • 발신자 (관제탑): 명령을 내릴 때 페이로드 헤더에 현재 시간(NTP 시간표 기준, 1 밀리초 단위) 과 랜덤한 문자열(Nonce) 을 박아 넣고 “합쳐서” 서명(Sign) 한다.
  • 수신자 (로봇/라우터): 패킷을 뜯었을 때, NTP 생존 시간 (Time-to-Live, TTL) 규칙을 적용한다.
  • “이 명령서에 찍힌 타임스탬프가 내 현재 시간보다 5초 이상 과거인가? 어제 만든 걸 다시 가져온 거네? 즉시 기각(Drop)!”
  • “최근 5초 안에 들어온 명령인가? 그럼 메모리 맵(HashMap) 을 뒤져본다. 이 Nonce 값을 가진 명령을 방금 1초 전에 실행한 적이 있는가? 똑같은 거 또 가져왔네? 기각!”

E2E 암호화(15.5장) 를 할 때, 암호문 안에 반드시 시간 정보와 난수를 박아라. 데이터를 암호화만 하고 시간에 묶어두지 않으면, 데이터는 영원히 당신을 공격할 좀비 무기(Zombie Weapon) 로 재활용된다.

3. 스푸핑(Spoofing) 및 중간자 공격(MitM) 원천 차단 아키텍처

로봇이 접속하려는 타겟 라우터의 IP(10.0.0.5) 를 해커가 로컬 네트워크의 ARP 스푸핑을 통해 자기 컴퓨터(10.0.0.99) 로 낚아챘다. 로봇은 가짜 라우터에 데이터를 밀어 넣고 있다.

3.0.1 [Runbook] X.509 화이트리스팅 방어 전술

이것을 막아내는 것은 IP 도, 설정 파일도 아니다. 오직 무결성 증명 체계뿐이다.

1. Server Certificate 검증의 절대 원칙
로봇(클라이언트) 이 라우터에 접속할 때, 가장 먼저 라우터가 보내온 인증서를 뜯어야 한다.
해커가 가짜 서버를 띄워 인증서를 보냈다면? 해커의 인증서는 우리 회사(Root CA) 가 도장을 찍어준 인증서가 아니다. 클라이언트의 TLS 라이브러리는 즉시 연결을 부러뜨린다.

2. SAN (Subject Alternative Name) 검증의 잔혹함
만약 해커가 어떻게든 우리 회사의 인증서를 하나 훔쳐서(예: 일반 작업자용 인증서) 라우터 행세를 한다면?

  • 로봇은 인증서 발급자만 보지 않고, 인증서 안에 텍스트로 적힌 타겟 노드 이름 (SAN) 을 검사한다.
  • “내가 연결하려 한 서버는 CN=main-router 여야 하는데, 이 신분증을 까보니 CN=worker-03 이구먼? 스푸핑이네!”
  • Zenoh 연결 설정 단계에서 타겟 주소(tls/10.0.0.5:7447) 뿐 아니라 그 서버가 제시해야 하는 명백한 호스트 네임 검증(Host Name Verification) 을 라이브러리 레벨에서 하드코딩하라. 이것만이 중간에서 가로채는 신분 위장(Spoofing) 을 끊어내는 물리적 끊임점이다.
