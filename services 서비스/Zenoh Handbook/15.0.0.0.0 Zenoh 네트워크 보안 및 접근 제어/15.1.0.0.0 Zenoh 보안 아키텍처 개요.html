15.1 Zenoh 보안 아키텍처 개요

서버실(On-premise) 안의 컴퓨터 10대가 통신하는 환경에선 보안을 방화벽 1대에 맡기면 끝이었다. 그러나 Zenoh가 뛰노는 전장은 다르다. 달리는 자동차, 날아다니는 드론, 방탄 유리에 덮인 외부 CCTV, 그리고 퍼블릭 클라우드까지 뻗쳐있다.

물리적 통제가 100% 붕괴한 개방형 오프로드 전장에서, 오로지 논리적인 암호학(Cryptography)과 프로토콜 아키텍처만으로 나의 데이터를 방어해 내는 Zenoh 보안의 메타-프레임워크를 해체한다.
당신은 방화벽의 수호자가 아니라, 각 노드 자체가 흉기이자 방패인 제로-트러스트(Zero-Trust) 지휘관이 되어야 한다.

1. 에지 및 분산 컴퓨팅 환경의 보안 위협 모델

해커는 VPN과 IP 주소를 찾는 해킹을 멈췄다. 드론을 훔쳐서 안에 깔린 라우터를 갈취하는 물리적 해킹의 시대다.

1.0.1 [인스펙션] 분산 네트워크의 치명적 3대 맹점

1. 물리적 탈취 (Physical Compromise)
서버실과 달리, 라즈베리 파이가 탑재된 로봇은 발로 차면 쓰러진다.

  • 가장 큰 위협: 해커가 로봇 안에 있는 zenohd.json5 파일이나 z_put 스크립트를 조작해, 클라우드로 가짜 화재 경보("fire_alarm": true)를 무한 전송하여 관제 센터를 마비시키는 행위다. 물리적 디바이스가 뚫리면 내부망 프리패스 권한이 열린다.

2. 도청과 중간자 공격 (Man-in-the-Middle)
자동차 간 통신(V2V)은 공중의 전파(LTE, 5G, Wi-Fi)를 타고 흐른다. Zenoh가 만약 일반 TCP/UDP 평문으로 데이터를 쏜다면, 길가에 안테나를 세운 누구나 패킷 스니퍼(Wireshark)로 “이 차의 브레이크 오일이 터졌다“는 정보를 평문으로 읽어낼 수 있다.

3. 스푸핑 및 쿼리 폭격 (Query Bomb)
나쁜 의도를 가진 노드가 나타나 관제 센터 행세를 하며 라우터에 z_get "**" 이라는, 세상의 모든 데이터를 빨아들이는 쿼리를 0.01초 단위로 날린다. 시스템 전체의 CPU 와 대역폭이 비명을 지르며 질식(DDoS) 해버린다.

“망 자체가 안전하다” 는 환상을 완전히 버려라. 모든 메시지는 적이 쏘는 독화살이라는 전제하에 아키텍처를 설계해야 한다.

2. Zenoh 보안의 3대 핵심 원칙: 기밀성, 무결성, 가용성

보안의 삼위일체(CIA Triad)가 Zenoh 스택 안에서 어떻게 물리적인 함수로 변환되는지 살펴본다.

2.0.1 [Runbook] CIA 방어술의 논리적 뼈대

1. 기밀성 (Confidentiality): “아무도 들여다보지 못하게 하라”

  • 위협: 공중망 도청.
  • Zenoh의 무기: 오직 TLS와 QUIC를 통한 전송 구간 터널링.
  • 행동 규칙: tcp를 버려라. 모든 데몬과 라우터 연결 플러그인은 무조건 tls 스킴(Scheme)으로 시작해야 한다. tls/MY_IP:7447.

2. 무결성 (Integrity): “누구도 내용을 변조하지 못하게 하라”

  • 위협: “온도=25” 라는 패킷의 숫자를 허공에서 누군가 99로 바꿔친다.
  • Zenoh의 무기: 인증서(Certificate) 기반의 핸드쉐이크, 혹은 유저 애플리케이션 단에서의 Payload Hashing (디지털 서명). 라우터는 중간에 패킷이 단 1비트라도 훼손되면, TLS 레이어 단에서 검증에 실패하여 해당 패킷을 강제 드랍(Drop) 시킨다.

3. 가용성 (Availability): “서비스가 멈추지 않게 방어하라”

  • 위협: 악의적 쿼리 플러딩(Flooding), 무한 Put 융단폭격 (DDoS).
  • Zenoh의 무기: 권한 인가 모듈(ACL Plugin)과 라우팅 백프레셔(Backpressure).
  • 행동 규칙: 토픽별로 Put과 Get이 가능한 유저 롤(Role)을 분리하라. 쓸데없이 폭격을 가하는 IP 는 ACL 모듈이 네트워크 제일 앞단 소켓에서 끊어버려 백엔드 디스크의 생명을 연장시킨다.

3. Zero-Trust 네트워크 설계와 Zenoh의 역할

“Zero-Trust (아무 것도 믿지 마라)”
기존 방화벽 개념은 성벽 스위치와 같다. “사내 IP 10.0.X.X 대역은 착한 놈들이니 묻지도 따지지도 말고 포트를 다 열어줘라”
이것이 클라우드와 엣지가 뒤섞인 현대 로보틱스 망을 가장 빨리 망하게 하는 지름길이다.

3.0.1 [인스펙션] 기만없는 파편화 인증의 역학

1. IP의 무가치성 선언
로봇은 5G를 타고 터널 안에 들어가면 IP가 동적으로 계속 바뀐다. IP 주소가 바뀌는 상황에서 “IP 기반 방화벽“은 아무런 통제 장치가 되지 못한다.
Zenoh는 접속하는 자의 IP를 믿지 않는다. 오직 “그 자가 가져온 암호화폐(인증서 키 Pair)” 만을 보고 판단(Authentication)한다.

2. 노드 레벨의 미시적(Micro) 방화벽
중앙(클라우드 입구)에 거대한 방화벽 1개를 두는 짓을 버려라.
Zenoh에서는 1,000대의 라즈베리 파이 단말기마다 떠 있는 1,000개의 데몬 스레드 각각이 자체적인 ACL 인증 엔진을 소유하도록 설계되어 있다. 이웃한 로봇 2대가 직통(P2P)으로 연결될 때조차 상호 간의 TLS 인증서 검문을 수행한다. 거대한 방화벽 1개가 아니라, 1,000개의 미세 바리케이드가 망 전체에 깔려 있는 궁극의 분산 격리(Distributed Isolation) 망이다.

4. Zenoh 보안 도메인 및 격리(Isolation) 개념

회사에 A 부서(공장 제어) 와 B 부서(인사부 화상 회의) 가 섞여 있다 치자.
망이 연결되어 있다는 이유만으로 A 부서 직원이 실수로 B 부서의 패킷을 들여다볼 가능성을 아예 0%로 만들 수는 없는가?

4.0.1 [Runbook] 도메인 아이디(Domain ID) 파티셔닝 전술

Zenoh는 이것을 물리적으로 가위질해 버릴 수 있다. 바로 도메인 아이디(Domain ID) 메타데이터를 통해서다.

1. 완전한 데이터 절연(Galvanic Isolation)
Zenoh 프로토콜을 설정할 때, domain 이라는 숫자를 박아 넣을 수 있다.

// 부서 A 라우터
{
  domain: 100 // 로보틱스 전용 도메인
}

// 부서 B 라우터
{
  domain: 200 // 비디오 스트리밍 전용 도메인
}

이 규칙이 켜지는 순간, 도메인 100 에 속한 라우터와 클라이언트가 날린 Put 혹은 Get 패킷은 다른 도메인에게 절대 전달되지 않는다.
토픽 이름(Key Expression)이 똑같다 하더라도 상관없다. 밑단의 프로토콜 엔진 자체가 도메인 번호가 다르면 그 통신 소켓을 마치 “다른 물리적 네트워크 랜 케이블“이 꽂힌 것처럼 완벽하게 격리(Isolate)하고 폐기한다.

도메인 레벨 파티셔닝은 복잡한 접근 제어(ACL) 설정 없이도 가장 원시적이고 철저하게 데이터 군락을 독립시키는, 보안 아키텍트의 무책임하지만 가장 강력한 칼날이다.
