15.7.2 Zenoh-Pico(마이크로컨트롤러 환경)의 경량 암호화 적용 한계와 대안

수십 킬로바이트(KB) 메모리밖에 없는 아두이노(Arduino) 나 ESP32 에서 TLS 1.3 을 돌린다는 것은 환상이다. 암호화 연산을 하느라 프로세서가 타버린다.

0.1 [Runbook] 베어메탈(Bare-Metal) 차폐 브릿징 전술

Zenoh-Pico(C 언어 초경량 에지) 에는 TLS 터널을 뚫을 권한 자체가 없다. 이 물리적 한계를 네트워크 토폴로지로 극복한다.

1. P2P 평문(Plaintext) 허용과 물리적 고립
초소형 센서 단말 \rightarrow 엣지 게이트웨이(라즈베리 파이) 구간, 즉 1미터 이내의 짧은 I2C 나 BLE(블루투스), 시리얼 통신 구간에서는 눈물을 머금고 암호화를 포기한다. 이 구간을 오직 평문(Serial/TCP) 기반으로 타겟팅 한다.

# 센서 근처의 에지 라우터 (zenohd)
# 1. Pico 가 접속할 로컬 평문 소켓 개방 (가장 위험한 포트)
listen: ["tcp/192.168.1.100:7447"] 

# 2. 이 라우터가 클라우드(형님)에게 데이터 보낼 때는 무조건 TLS!
connect: ["tls/cloud-server.com:7447"]

2. 에지 라우터의 “보안 대리인(Proxy)” 승격
Pico 센서는 오직 라즈베리 파이까지만 데이터를 보내면 역할이 끝난다 (Fire and Forget).
패킷을 넘겨받은 라즈베리 파이의 Zenoh 라우터 본체(zenohd) 가 강력한 연산력을 바탕으로 받은 평문 데이터를 순식간에 TLS 터널에 담아 클라우드로 쏴버린다(Encryption Offloading). 물리적 로컬 리스크만 감당하고 인터넷망의 리스크는 완벽히 차단하는 현실계의 생존 협상 런북이다.
