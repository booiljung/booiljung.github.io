15.5.1 라우터를 신뢰하지 않는 종단간(End-to-End) 애플리케이션 암호화

“클라우드 프로바이더를 믿지 마라.”
구글이나 아마존에 설치한 Zenoh 라우터조차, 그 안에서 무슨 로그를 남기고 패킷을 복사할지 아무도 보장할 수 없다.

0.1 [인스펙션] Payload 눈가림 전술

1. 라우터의 맹목성(Blindness)의 역이용
Zenoh는 바이트 스트림 프로토콜이다. 라우터는 자기가 쏘는 Payload (ZBytes) 가 JSON(문자열) 인지, 동영상(H.264) 인지 알 바 없고 알지도 못한다. 오직 토픽명(factory/secret) 만 보고 길을 찾아줄 뿐이다.

2. 에지 퍼블리셔(Publisher) 단의 봉인 (Sealing)
로봇 엔지니어가 C++ 나 Rust 코드에서 put 함수를 부르기 직전, 로컬 RAM 위에서 직접 데이터를 암호화(AES-256) 한 후 발송한다.

  • 센서 측 (발송): z_put "top_secret", AES256_Encrypt(data, my_key)
  • 클라우드 측 라우터: “오, 뭔가 쓰레기 값이 지나가네. 저장해야지~” (내부의 데이터 원본은 절대 알 수 없다).
  • 관제탑 측 (수신): z_get "top_secret" \rightarrow 암호문 도착 \rightarrow AES256_Decrypt(cipher, my_key) \rightarrow 복호화 성공.

TLS 가 터널의 안팎을 지켰다면, 이 전술은 편지 봉투 안에 들어있는 “편지지” 지 자체를 외계어로 써버리는 기법이다. 서버를 통째로 탈취한 해커 앞에서도 아키텍트가 조소할 수 있는 유일한 무기다.
