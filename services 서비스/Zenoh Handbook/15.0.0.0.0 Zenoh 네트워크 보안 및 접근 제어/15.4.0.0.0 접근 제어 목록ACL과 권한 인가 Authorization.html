15.4 접근 제어 목록(ACL)과 권한 인가 (Authorization)

1차 관문(인증)을 통과한 해커가 회사망에 발을 들였다. 이제 그가 z_get "**" (모든 데이터 내놔) 를 치거나 z_delete "factory/**" (공장 데이터 다 지워) 라는 파멸적인 패킷을 날리는 것을 막아야 한다.

Zenoh 는 시스템에 흐르는 “모든 단일 패킷” 의 멱살을 잡아채 그 패킷이 향하는 토픽(Key Expression) 과 패킷의 종류(Put/Get) 를 판독하여 즉결 심판을 내리는 ACL (Access Control List) 플러그인을 기본 내장하고 있다. 단 한 줄의 접근 불가 패킷이 잡히면 그 즉시 허공에서 증발(Drop) 시켜 백엔드 디스크가 자신의 공격받음조차 모르게 하는 치밀한 방어벽 전술이다.

1. Zenoh 키 표현식(Key Expression) 기반의 정밀한 권한 제어

기존의 방화벽이 IP 주소를 필터링했다면, 데이터 중심(Data-Centric) 네트워크인 Zenoh 에서는 “그 이름표(URL) 에는 접근할 수 없다” 고 명시한다.

1.0.1 [인스펙션] 문자열 검열주의 기반 ACL

1. 와일드카드 블랙리스트 전술
가장 위험한 패턴은 ** (모든 데이터) 와일드카드 남용이다.
라우터 설정에서 특정 유저(권한 그룹) 에게 하위 자식 노드의 데이터를 뽑아내는 범위 쿼리 자체를 부분적으로 절단시킬 수 있다.

  • 허용: factory/sensor/temp
  • 금지: factory/sensor/**
    이렇게 설정하면 악의적인 유저가 센서 데이터를 일일이 뽑아가는 것(temp, humidity) 은 통과시키되, 단 한 번의 쿼리로 전체 망을 싹쓸이하는 스니핑(Sniffing) 시도는 라우팅 자체에서 블로킹된다.

2. 논리적 망분리와 교차점 차단
admin 계정은 모든 것을 볼 수 있지만, robot_1 계정은 오직 "robot/1/**" 영역에만 데이터를 쓸 수 있게 묶어라.
이를 통해 로봇 하드웨어가 탈취당하더라도, 범죄자가 다른 로봇(robot_2) 의 제어 채널로 스파이 패킷을 전송하는(Lateral Movement) 시도를 라우터 단에서 완벽하게 격리해 낸다.

2. Publish, Subscribe, Query, Reply 각각의 권한 분리 전략

Zenoh 통신망을 지나는 메시지는 4개의 계급(Operation Type)으로 철저히 나뉘어 통제된다. “읽기 권한“을 줬다고 해서 “구독 권한”, “응답 권한“까지 한꺼번에 열리는 어리석음은 없다.

2.0.1 [Runbook] 프로토콜 행위 통제(Operation Sandbox) 전술

1. 생산자(Sensor)의 모래상자
온도 센서는 데이터를 허공에 쏘기만 하면 된다(Put).

  • 인가 권한: 오직 publisher (Put 통과).
  • 센서가 해킹당해 “현재 온도가 뭐지?” 하고 역으로 Get 쿼리를 때리거나 몰래 남의 온도 데이터를 빼가려고 Sub (구독) 소켓을 여는 순간, 라우터가 에지 단말의 접속을 강제로 찢어버린다.

2. 소비자(Dashboard)의 모래상자
관제 웹 모니터는 데이터를 듣기만 해야 한다.

  • 인가 권한: 오직 subscriber (Sub 통과), querier (Get 통과).
  • 대시보드를 보는 인턴이 브라우저 콘솔창을 켜고 z_put "robot/motor/speed" "200" 을 날려 로봇을 급발진시키려 해도, 쓰기 권한이 없으므로 그 패킷은 라우터를 향해 날아가는 즉시 소멸한다.

3. 스토리지 엔진(Backend)의 모래상자
DB 저장용 스토리지 플러그인조차 이 권한의 영향을 받는다.

  • 인가 권한: 남의 쿼리에 대답해야 하므로 replier (Reply 통과) 가 반드시 필요하다. 데이터 주인의 자리표를 점유하는 오직 허락된 자(Storage)만이 대답할 수 있다.

3. 역할 기반 접근 제어(RBAC) 모델의 Zenoh 적용

로봇이 10,000 대인데 robot_1부터 robot_10000까지 1만 줄의 권한 코드를 짤 것인가?
아키텍처 설계의 기본은, 개별 유저를 통제하는 것이 아니라 **역할(Role)**이라는 가상의 바구니를 통제하는 것이다.

3.0.1 [인스펙션] 유저-롤-퍼미션 대통합 전술

1. 롤(Role) 바구니 생성
zenohd.json5 설정 파일 안에, 다음과 같은 바구니 3개를 파놓는다.

  • Role: Worker
  • Role: Inspector
  • Role: Master

2. 유저(User) 투하
인증(Authentication) 단계에서 계정이 drone_05 인지 operator_1 인지 파악되면? 즉시 그에 맞는 롤 바구니에 던져 넣는다.

  • drone_05 \rightarrow Worker 바구니 소속
  • operator_1 \rightarrow Inspector 바구니 소속

3. 퍼미션(Permission) 재정의
이제 ACL 검역소는 특정 유저 이름이 아니라, 그가 속한 **[바구니 이름]**만 본다.

  • Worker 바구니 소속자? \rightarrow 무조건 Put: telemetry/** 만 허용.
  • Inspector 바구니 소속자? \rightarrow 무조건 Get: ** 만 허용.

수만 대의 기기가 생겼다 사라져도, 시스템 엔지니어의 ACL 설정 파일은 단 10줄을 넘지 않는다.
이것이 진정한 제로-트러스트 시대의 자동화 스케일링(Scalable Access Control) 패턴이다.

4. JSON/JSON5 기반 ACL 설정 파일 작성 문법 및 예제

말로만 백날 설명하는 것은 끝났다. 이제 데몬(Routers)을 틀어막는 가장 기초적이며 파괴적인 JSON 검열 룰북을 완성하라.

4.0.1 [Runbook] 실전 차단(Kill-Switch) 코드 구현

// zenohd.json5
plugins: {
  access_control: {
    // 1. 유저를 역할(Role)에 매핑한다. (인증 플러그인에서 넘어온 유저 대상)
    users: {
      "drone_01": "worker",
      "drone_02": "worker",
      "monitoring_server": "inspector",
      "admin_kim": "god_mode"
    },

    // 2. 역할별 권한(ACL)을 강제한다.
    roles: {
      "worker": {
        // [허용 리스트] 센서값을 쓰고(Put), 다른 놈들 데이터를 구독(Sub)할 수 있다.
        allow: [
          "Put:drone/telemetry/**",
          "Sub:drone/telemetry/**"
        ],
        // [금지 리스트] 무슨 일이 있어도 루트 패킷 전체를 빨아먹거나 지우지 못하게 죽여라.
        deny: [
          "Query:**",
          "Delete:**",
          "Put:system/config/**"
        ]
      },

      "inspector": {
        // 대시보드 관제탑은 오직 과거 기록(쿼리)과 구독(Sub)만 된다. 쓰는 건 불가능하다.
        allow: ["Query:**", "Sub:**"]
      },

      "god_mode": {
        // 모든 권한 다 열어줌
        allow: ["**"]
      }
    }
  }
}

이 플러그인이 켜지는 순간, Zenoh 라우터는 데이터가 목적지에 도달하기도 전에, 혹은 백엔드 DB 에 닿기도 전에 소켓 계층에서 저 deny 블랙리스트 룰에 걸린 모든 패킷을 빛의 속도로 묵살(Drop) 해버린다.

5. 시스템 재시작 없는 동적 ACL 업데이트 및 핫 리로드(Hot Reload)

직원 한 명이 퇴사했다. 그의 권한(deny) 을 막으려고 라우터를 껐다 켜면(Restart), 그 라우터 밑에 붙어있던 드론 1,000 대의 조종 스틱이 1초간 마비되어 추락할 것이다. 보안을 위해 물리적 시스템 가용성을 버티면 안 된다.

5.0.1 [Runbook] 무정지 ACL 업데이트 전술

Zenoh 라우터의 관리자(Admin) 공간을 통과하는 라이브 업데이트 런북이다.

1. 관리자 토픽에 수술용 메스 꽂기
Zenoh는 자기 자신의 설정을 자기 자신의 토픽(Key Expression) 으로 관리하는 기묘한 재귀적(Recursive) 프로토콜이다.
특정 라우터(예: ID가 1234 인 놈) 의 ACL 설정은 그 라우터의 @/router/1234/plugin/access_control/config 라는 특수한 관리자용 토픽으로 매핑되어 있다.

2. 핫-리로드 API 호출 (z_put)
아래와 같이 새로운 JSON 설정 문자열을 만들어서, 다른 컴퓨터에 앉은 관리자가 Put 을 쏴서 라우터의 머릿속에 던져 넣는다.

## REST API (HTTP PUT) 를 사용하는 런북
curl -X PUT http://admin:admin_password@<ROUTER_IP>:8000/@/router/local/plugin/access_control/config \
     -H "Content-Type: application/json" \
     -d '{
           "users": { "drone_01": "worker", "bad_guy_01": "hacker" },
           "roles": { "hacker": { "deny": ["**"] } }
         }'

이 패킷이 라우터에 꽂히는 즉시, 데몬 프로세스는 재부팅 없이(No Restart) 메모리 위의 ACL 트리를 갈아 끼운다(Hot-Swap). 통신선이 1나노초도 끊기지 않은 채, bad_guy_01 유저가 쏜 다음번 패킷부터는 즉시 소켓에서 필터링당하는 진정한 로보틱스-네이티브 방어 시스템이 구현된다.
