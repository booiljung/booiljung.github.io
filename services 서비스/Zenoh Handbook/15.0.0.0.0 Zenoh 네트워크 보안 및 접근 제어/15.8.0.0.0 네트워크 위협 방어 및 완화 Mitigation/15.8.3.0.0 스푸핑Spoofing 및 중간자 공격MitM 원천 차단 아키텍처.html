15.8.3 스푸핑(Spoofing) 및 중간자 공격(MitM) 원천 차단 아키텍처

로봇이 접속하려는 타겟 라우터의 IP(10.0.0.5) 를 해커가 로컬 네트워크의 ARP 스푸핑을 통해 자기 컴퓨터(10.0.0.99) 로 낚아챘다. 로봇은 가짜 라우터에 데이터를 밀어 넣고 있다.

이것을 막아내는 것은 IP 도, 설정 파일도 아니다. 오직 무결성 증명 체계뿐이다.

1. Server Certificate 검증의 절대 원칙
로봇(클라이언트) 이 라우터에 접속할 때, 가장 먼저 라우터가 보내온 인증서를 뜯어야 한다.
해커가 가짜 서버를 띄워 인증서를 보냈다면? 해커의 인증서는 우리 회사(Root CA) 가 도장을 찍어준 인증서가 아니다. 클라이언트의 TLS 라이브러리는 즉시 연결을 부러뜨린다.

2. SAN (Subject Alternative Name) 검증의 잔혹함
만약 해커가 어떻게든 우리 회사의 인증서를 하나 훔쳐서(예: 일반 작업자용 인증서) 라우터 행세를 한다면?

  • 로봇은 인증서 발급자만 보지 않고, 인증서 안에 텍스트로 적힌 타겟 노드 이름 (SAN) 을 검사한다.
  • “내가 연결하려 한 서버는 CN=main-router 여야 하는데, 이 신분증을 까보니 CN=worker-03 이구먼? 스푸핑이네!”
  • Zenoh 연결 설정 단계에서 타겟 주소(tls/10.0.0.5:7447) 뿐 아니라 그 서버가 제시해야 하는 명백한 호스트 네임 검증(Host Name Verification) 을 라이브러리 레벨에서 하드코딩하라. 이것만이 중간에서 가로채는 신분 위장(Spoofing) 을 끊어내는 물리적 끊임점이다.
