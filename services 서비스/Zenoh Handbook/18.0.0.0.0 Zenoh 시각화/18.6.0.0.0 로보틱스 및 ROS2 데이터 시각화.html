18.6 로보틱스 및 ROS2 데이터 시각화

거짓말을 하지 않는 숫자(네트워크 핑, CPU) 만 바라보던 인프라를 넘어, 이제 “현장(Reality)” 을 마주해야 한다.
로보틱스에서의 시각화란 라이다(LiDAR) 가 쏘는 3차원 점들이며, 카메라이고, 나침반이다. 이를 시각화하지 못하면 로봇은 눈 뜬 장님이자 폭주 기관차에 불과하다.

이 장에서는 기존 ROS2의 닫힌 로컬 환경(Local Area) 을 벗어나, 11장에서 구축했던 Zenoh 브릿지 망을 타고 대륙 반대편의 RViz2 와 Foxglove Studio 에 현장의 물리적 데이터를 끊김 없이 렌더링하는 진정한 “원격 운전(Teleoperation)” 시각화 런북을 설파한다.

1. zenoh-bridge-dds 환경에서의 ROS2 데이터 시각화 워크플로우

관제 센터에 앉아있는 당신의 ROS2 시각화 툴(RViz2) 은, 캘리포니아 공장에 있는 로봇의 존재를 모른다. DDS 멀티캐스트(UDP) 는 바다를 건널 수 없기 때문이다.

1.0.1 [인스펙션] 시각적 웜홀(Wormhole) 개통 전술

zenoh-bridge-dds 는 이 단절된 시야를 이어주는 투명한 유리창이다.

1. 대륙 간 브릿지(Bridge) 파이프라인 결속

  • 공장(로봇 쪽): 로봇이 내뿜는 100개의 DDS 토픽을 zenoh-bridge-dds 가 집어삼킨 뒤, 단 하나의 TCP 소켓으로 압축해 클라우드 라우터로 쏜다.
  • 관제소(나의 PC 쪽): 내 PC 에도 zenoh-bridge-dds 를 켠다. 클라우드 라우터에서 내려온 TCP 데이터를 받아, 다시 내 PC 내부망에 DDS 멀티캐스트로 흩뿌린다. (로컬 호스트 디스커버리)

2. 구독 트리거 (On-Demand Visualizing)
Zenoh 브릿지의 마법은 “게으름(Laziness)” 에 있다.
내가 PC 에서 RViz2 를 켜지 않으면 데이터는 바다를 건너오지 않는다.
하지만 내가 RViz2 를 켜고 /robot/lidar 화면 표시 체크박스를 누르는(Subscribe) 그 순간! ROS2 통신 규약에 의한 Discovery 가 트리거되고, Zenoh 브릿지가 대륙 너머 로봇에게 “야 지금 당장 라이다 주시 시작해!” 라고 명령하여 비로소 대용량 그래픽 데이터가 흐르기 시작하는 완벽한 On-Demand 워크플로우를 보장한다.

2. RViz2를 활용한 원격/분산 Zenoh 노드 제어 및 시각화

관제탑 노트북에서 RViz2 (ROS2 표준 3D 시각화 도구) 가 무사히 캘리포니아 로봇의 라이다를 잡았다. 그런데 로봇을 원격 조종하려니 지연 시간(Lag) 때문에 차가 벽에 처박힌다.

2.0.1 [Runbook] RViz2 텔레오퍼레이션(Teleop) 튜닝 런북

단방향 시각화와 양방향 제어(Control) 는 차원이 다른 오버헤드다.

1. QoS 프로필 불일치 해결
로봇은 카메라 영상을 Best-Effort (UDP, 빠르지만 손실 가능) 로 쏘는데, 관제탑 RViz2 의 영상 표시 위젯이 기본값인 Reliable (TCP, 느리지만 완벽함) 로 설정되어 있다면 화면이 영원히 나오지 않는다.
원격 제어 RViz 의 모든 렌더링 패널(Topic) 설정 창에 들어가, QoS 정책을 반드시 Best-Effort 와 Volatile (과거 데이터 안 받음) 로 튜닝하여 브릿지가 에러 없이 패킷을 즉시 버릴 수(Drop) 있도록 허락하라.

2. Teleop 제어 토픽(cmd_vel) 역전송 분리
보는 것(영상) 은 지연되어도 되지만, 조이스틱 제동(Stop) 명령은 0.1초 만에 로봇에 도달해야 한다.
zenoh-bridge-dds 설정(allow 룰) 에서 /cmd_vel 토픽만큼은 반드시 Reliable 브릿지 채널을 거치거나, 우선순위가 높은 프록시 대역(Priority 1) 을 태워 시각화 렌더링 찌꺼기 패킷들에 조향 명령이 파묻히는 참사를 원천 세팅으로 틀어막아야 한다.

3. Foxglove Studio와 Zenoh의 통합 연동 아키텍처

로봇 엔지니어들은 C++ 기반의 RViz2 를 좋아하지만, 웹 브라우저 하나로 접속해야 하는 클라우드 관제 플랫폼에서는 차세대 로보틱스 시각화 표준, 즉 Foxglove Studio 가 절대 권력이다.

3.0.1 [인스펙션] 진화된 브라우저 관제 센터 가동

1. Foxglove WebSocket 브릿지 통신망
Foxglove 는 기본적으로 웹 기술 베이스다. 로컬 PC 의 DDS 통신에 기생하는 대신, rosbridge_server (또는 foxglove_bridge) 를 로봇단에 켜둔다.
Zenoh 는 이 로봇 내부의 Foxglove 웹소켓 포트(8765) 트래픽을 통째로 릴레이(Relay) 하여, 관제탑 브라우저에 접속한 Foxglove UI 화면으로 토스해버리는 완벽한 TCP 터널 역할을 수행한다.

2. 메세지 규격의 투명성 (Schema Resolution)
Foxglove 의 최대 강점은 데이터를 해석하는 IDL(스키마) 파일이 없어도 동적으로 그래프를 그린다는 것이다.
Zenoh 망을 통해 넘긴 ROS2 메시지 바이너리에는 그 구조(Type) 가 꼬리표처럼 달려 있다. Foxglove 가 이 꼬리표를 읽어 자동으로 “온도 꺾은선그래프” 나 “카메라 이미지” 위젯을 마법처럼 스냅(Snap) 시키는 놀라운 연동성을 체감하라. 더 이상 개발용 PC 에 무거운 ROS2 전체 패키지를 깔 필요가 없다. 브라우저면 충분하다.

4. 로봇 센서 데이터 (LiDAR, 포인트 클라우드, 카메라 영상, IMU)의 실시간 렌더링 기법

센서는 무자비하다. 16채널 LiDAR 는 1초에 30만 개의 3D 좌표 스파이크를 토해낸다. 이것을 태평양 건너 관제탑으로 다 보내서 그리면 화면이 뻗고 망이 터진다.

4.0.1 [Runbook] 데이터의 중력(Gravity) 감쇄 렌더링 런북

고대역폭 시각화는 “얼마나 빨리 버리느냐” 에 달렸다.

1. Point Cloud (수십만 개 좌표) 다운샘플링
로봇 안에서 Zenoh 로 쏘기 전, Voxel Grid Filter (공간 깍두기 압축) 를 거친다.
“반경 5cm 안에 있는 점들은 1개로 퉁쳐서 합친다.”
이 연산 하나로 데이터 백만 개가 1만 개로 압축된다. 관제탑 시각화 도구(Foxglove/RViz) 가 부드러운 초당 30 프레임의 부드러운 회전(Orbit) 을 하려면 원본 데이터 전송의 만행을 원천적으로 쳐내야 한다.

2. H.264 / H.265 비디오 스트림 트래픽 태우기
카메라 이미지(sensor_msgs/Image) 1장은 무손실 일 때 몇 메가(MB) 에 달한다. 이걸 Zenoh 퍼블리셔로 쏘면 로봇의 목숨이 날아간다.
반드시 카메라 프레임을 로봇 단에서 하드웨어 코덱(예: NVIDIA V4L2) 을 통해 H.264/WebRTC 스트림으로 압축한 뒤, 그 압축된 바이너리 자체를 통째로 Zenoh Value 로 실어 보내야 한다. 브라우저 쪽에서는 이 바이너리를 HTML <video> 태그 소스로 다이렉트로 넘겨 압축을 풀게 하는 것이 유일한 저지연(Low-latency) 스트리밍 생존법이다.

5. 다중 로봇(Multi-Robot) 관제 화면 설계 및 구현

로봇이 100대다. RViz2 화면을 창 100개 띄워놓고 볼 것인가?
다중 로봇 관제탑 (Fleet Management System, FMS) 의 화면은 개별 로봇의 현미경이 아니라, 100대의 로봇 전체를 포괄하는 거시적 위성 지도여야 한다.

5.0.1 [인스펙션] 전지적 작전 지도(Bird’s Eye View) 전술

우리의 시각화 패러다임이 3D 공간(Room) 레벨에서 국토(Map) 레벨로 확산한다.

1. 네임스페이스(Namespace) 기반 클러스터링 매핑
각 로봇의 좌표 토픽 구조가 fleet/seoul/robot_001/gps 와 같이 네임스페이스로 완벽히 격리되어 있어야 한다.
웹 대시보드의 메인 화면(지도 패널) 은 fleet/**/gps 와일드카드로 100대 로봇의 좌표를 한 번에 다 구독한다. 지도상에는 개별 로봇이 점 하나(Dot) 로 표시되거나 너무 뭉쳐있다면 집단 핀(Cluster Marker) 1개로 합쳐져 큰 물결처럼 움직임을 표현한다.

2. 트래픽의 스파셜(Spatial) 멀티플렉싱 뷰
관제사가 지도에서 에러가 난 “특정 로봇 아이콘(robot_042)” 을 마우스로 클릭하는 순간!
대시보드 프론트엔드는 그 즉시 다른 99대 로봇의 모든 데이터 구독(Subscription) 을 중지하고, fleet/*/robot_042/** 와일드카드의 고해상도 정보(Vitals, LiDAR, Camera) 만을 집중적으로 빨아당기는 “저격용 구독 전환(Targeted Sub Switch)” 아키텍처를 가동해야 한다.
보지 않는 것은 구독하지 않는 우아한 토픽 통제. 이것이 1만 대 로봇 시각화를 브라우저 하나로 통제하는 FMS(Multi-Robot) 관제 프론트엔드의 정통 코어 철학이다.
