18.4 엔터프라이즈 대시보드 도구와의 연동 (Grafana 중심)

현대의 인프라 관제는 더 이상 개발자가 한 땀 한 땀 차트 라이브러리를 바닥부터 짜는 삽질을 용납하지 않는다. 전 세계 서버 엔지니어들의 90% 가 매일 아침 출근해서 켜놓는 글로벌 스탠다드 시각화 도구, 바로 그라파나(Grafana) 가 있다.

이 장에서는 17장에서 애써 모아둔 Prometheus 의 숫자들을 그라파나의 화려한 UI 와 결합하여, 경영진의 보고 화면부터 DevOps 팀의 철야 모니터링 화면까지 자유자재로 구성하는 대시보드 엔지니어링 런북을 전개한다.

1. Prometheus 및 Grafana를 활용한 시각화 아키텍처 설계

그라파나 자신은 데이터베이스가 아니다. 오로지 남의 데이터(Data Source) 를 예쁘게 포장해서 보여주는(View) 프론트엔드 액자일 뿐이다.

1.0.1 [인스펙션] 전시관(Gallery) 조립 전술

1. 데이터 소스(Data Source) 연동
그라파나를 켜고 Connections -> Data Sources 메뉴에서 17장에서 세팅한 Prometheus 서버의 URL (http://prometheus:9090) 을 꽂아 넣는다. 이 한 번의 연결로 그라파나는 Zenoh 라우터가 뱉어내는 모든 지표를 자신의 붓과 물감으로 쓸 권한을 얻는다.

2. 대시보드 아키텍처 (Layered Design)
대시보드를 막 만들지 마라. 화면을 3개의 층(Layer) 으로 나누어야 한다.

  • Top 층 (경영진 뷰): 현재 네트워크 전체의 총 대역폭, 총 연결된 로봇 수. (거대한 숫자 타일)
  • Middle 층 (운영팀 뷰): 라우터별 CPU 사용량, 패킷 전송 수 꺾은선그래프. (비교 차트)
  • Bottom 층 (엔지니어 뷰): 각 세션별 패킷 Drop 발생량, 메모리 프로파일링 (에러 텍스트).

2. Zenoh 메트릭 플러그인을 활용한 데이터 익스포트

만약 Zenoh 가 기본적으로 내뿜는 메트릭 외에 다른 커스텀 데이터가 보고 싶다면?

2.0.1 [Runbook] 데이터 착즙기(Juicer) 확장 전술

1. StatsD / Telegraf 브릿지 우회
Zenoh 가 내장 프로메테우스 익스포터를 지원하지만, 사내 인프라 규정상 InfluxDB 나 Datadog 으로 데이터를 밀어 넣어야(Push) 한다면 Telegraf 데몬을 중간에 끼워 넣어야 한다.
Zenoh 가 쓰는 포트(8000번) 를 Telegraf 가 긁어간(Scrape) 뒤, 뒷단에서 Influx 양식으로 변환하여 쏘아주는 데이터 번역기(Translator) 파이프라인을 구축하라.

2. 변수 템플릿(Variable Templating) 필터
라우터가 100 대인데 대시보드를 100 개 그릴 것인가?
그라파나의 상단에 Variable 을 추가하고, PromQL 함수 label_values(zenoh_session_count, instance) 를 적어 넣는다. 그러면 그라파나 화면 위쪽에 100대의 라우터 IP 가 쭉 나오는 드롭다운 박스가 자동 생성된다. 사용자가 IP 를 바꿀 때마다 대시보드 전체의 그래프가 그 라우터의 데이터로 한 번에 착착 바뀌는 “동적 대시보드(Dynamic Dashboard)” 야말로 엔터프라이즈 시각화의 꽃이다.

3. Zenoh 전용 Grafana 커스텀 대시보드 구축 가이드

“차트 만드는 거야 드래그 앤 드롭 아닌가요?” 아니다.
시각화 패널마다 가장 효과적인 그래프의 형태(Chart Type) 와 PromQL(쿼리 언어) 이 정해져 있다.

3.0.1 [인스펙션] 패널(Panel) 레시피 전술

1. 현재 살아 숨 쉬는 세션 수 (Stat Panel)

## 쿼리: sum(zenoh_session_count)

이 숫자가 1,000에서 900으로 갑자기 뚝 떨어지면 100대의 로봇이 지하로 들어간 것이다. Sparkline(배경 꺾은선) 을 추가하여 숫자의 흐름을 박스 배경으로 깔아준다.

2. 라우터별 수신 트래픽 랭킹 (Bar Gauge)

## 쿼리: topk(5, rate(zenoh_router_rx_bytes_total[1m]))

그라파나의 Bar Gauge 를 쓰면 가장 트래픽을 많이 먹는 라우터 1등부터 5등까지가 가로 막대그래프로 레이싱하듯 출렁인다. 네트워크 밸런싱이 한쪽으로 쏠렸는지(Hot-spot) 감시하는 최고의 무기다.

4. 주요 성능 지표(Throughput, Latency, CPU/Memory) 패널 구성

가장 크고 화려하게 대시보드 정중앙에 배치해야 할 생명 유지 장치들이다.

4.0.1 [Runbook] 심박수(Vitals) 모니터 전술

1. 대역폭(Throughput) 차트: 들숨과 날숨

  • 패널: Time series (꺾은선)
  • 초록색 선은 rate(rx_bytes) (수신), 빨간색 선은 rate(tx_bytes) (송신) 으로 그린다. 정상적인 라우터라면 초록색이 들어온 만큼 빨간색이 나가야 한다. 만약 초록색은 치솟는데 빨간색이 바닥을 기고 있다면 그 라우터는 지금 먹기만 하고 내보내지 못하는 변비(Deadlock/Buffer full) 에 걸린 것이다.

2. 지연 시간(Latency) 차트: 아우토반 스피드 건

  • 패널: Heatmap 또는 Time series
  • histogram_quantile(0.99, ...) 쿼리를 이용해 하위 99% 의 속도가 얼마나 걸리는지 방어선을 친다. 평균(0.50) 속도는 아무 쓸모가 없다. 가장 느린 쓰레기 같은 상위 1% 패킷이 얼마나 쳐지는지를 감시하는 것이 엔터프라이즈 관제의 핵심이다.

3. 하드웨어(CPU/Memory) 퓨즈 차트

  • 라우터가 메모리를 80% 이상 먹었을 때 노란색, 95% 이상 먹었을 때 빨간색으로 게이지(Gauge) 색깔이 자동으로 바뀌게 세팅하여 시각적 직관성을 극대화하라.

5. 임계치 초과에 따른 시각적 경고(Alert) 및 알림 인터페이스 구현

17.8장의 알람이 “스마트폰을 울리는 기능” 이라면, 시각화 단계에서의 알람은 “전광판을 빨갛게 물들이는 기능” 이다.

5.0.1 [인스펙션] 시각적 사이렌(Siren) 전술

사령실에 걸려있는 큰 모니터의 화면이 스스로 위험을 알려야 한다.

1. Threshold (한계선) 렌더링
CPU 꺾은선그래프 패널의 속성창에 들어가 Thresholds 를 추가한다. Value=80 (노란색), Value=90 (빨간색).
이 세팅을 마치면 차트에 가로줄이 하나 그어지며, CPU 선이 이 가로줄(90%) 을 돌파하는 순간 차트 영역 전체가 핏빛으로 물들어서 저 멀리 커피를 마시던 엔지니어도 차를 엎고 뛰어오게 만든다.

2. Alert State Map (신호등 행렬)
Stat 패널 하나에 라우터 100대의 상태를 조그만 네모 100개로 압축해버린다. 수수밭처럼 펼쳐진 초록색 네모 가운데, 단 하나 붉은색 네모가 깜빡인다면 그게 터진 라우터의 IP 다. 한눈에 1000대의 서버 상태를 브리핑할 수 있는 궁극의 단일 뷰(Single Pane of Glass)다.
