18.8.2 데이터 샘플링(Sampling) 및 다운샘플링(Downsampling) 전략 적용

브라우저는 잘못이 없다. 백엔드에서 폭탄(수백만 개의 행) 을 던진 것이 죄악이다. 1년 치 과거 데이터를 1920x1080 모니터 화면에 전부 다 표시하는 것은 수학적으로 불가능하다.

0.1 [Runbook] 데이터의 수학적 도끼질(Downsampling) 런북

백엔드 스토리지(InfluxDB, Prometheus) 나 중계 프록시에서 반드시 처리한 뒤 화면에 보내야 한다.

1. 주기 기반 압축 (Time-based Smoothing)
1분 동안 들어온 데이터가 6,000개 라면, 화면에 보낼 때는 이를 1개의 데이터로 뭉쳐서(Average) 보낸다.

  • Grafana의 Max Data Points 설정: 차트의 가로 폭이 1000픽셀이라면, 백엔드 쿼리에 Max Data Points=1000 파라미터를 강제로 주입하라. 데이터가 아무리 수억 개라도 무조건 1천 개로 쪼그라들어서 날아온다.

2. 변곡점 유지 압축 알고리즘 (LTTB)
단순히 평균(Average) 을 내면, 치명적으로 치솟아 올랐던 “1초짜리 바늘 모양 에러 스파이크(Spike)” 도 뭉개져 사라진다.
Largest Triangle Three Buckets (LTTB) 이라는 시계열 다운샘플링 알고리즘을 백경(Backend) 에 적용하라. 데이터 양은 1/1000 로 부서지지만, 그래프의 산봉우리(Spike) 와 계곡(Dip) 의 뾰족한 시각적 특징은 소름 돋게 원본 그대로 브라우저 화면에 유지된다.
