18.8 시각화 시스템 성능 최적화 및 트러블슈팅

로봇이 멈춘 것이 아니다. 단지 당신의 크롬 브라우저 탭이 OOM(Out of Memory) 으로 뻗었을 뿐이다.
Zenoh 처럼 1초에 1,000만 건의 메시지를 소화하는 괴물을 일반적인 웹 프론트엔드 아키텍처로 받아내려 하면, 열에 아홉은 시각화 시스템 자체가 병목이 되어 인프라 전체의 응답 속도를 거짓으로 보고하게 된다.

이 장에서는 수백만 픽셀의 데이터를 다루는 프론트엔드 한계를 돌파하기 위해, 메모리 누수(Leak) 를 막아내고 쓰레드를 찢어발기며 관제탑 화면의 끊김 현상(Jitter) 을 제거하는 극한의 브라우저 성능 해킹 런북을 설파한다.

1. 대규모 데이터 유입 시 브라우저 및 대시보드 과부하 방지 기법

브라우저의 가비지 컬렉터(Garbage Collector) 는 빅데이터를 혐오한다. 1초마다 수천 개의 객체가 생기고 버려진다면 화면 전체가 1초 단위로 뚝뚝 끊기는 프리징(Freezing) 이 온다.

1.0.1 [인스펙션] 메모리 누수 방탄(Bulletproof) 전술

이벤트 버블링과 쓰레기 객체 생성의 악순환을 끊어야 한다.

1. 변수 풀링 (Object Pooling)
new 키워드로 배열이나 객체를 계속 생성하지 마라.
Zenoh 에 들어온 패킷 배열을 다룰 때, 길이가 10만인 Array 를 딱 하나만 화면 로딩 시 처음 만들어두어라(미리 할당). 그리고 새 데이터가 들어올 때 무식하게 덮어쓰기(Overwrite) 만 수행해야 가비지 컬렉터가 잠에서 깨지 않고 프레임을 칼같이 지킬 수 있다.

2. 보이지 않으면 렌더링하지 않는다 (DOM Virtualization)
Grafana 가 그려주는 패널이 50개인데, 한 화면에 보이는 건 5개다.
웹 UI 가 화면 밖(Off-screen) 에 스크롤 되어 숨겨진 45개의 차트를 여전히 CPU 로 갱신하고 있다면 당장 옵션을 켜라. (React 의 react-window, Intersection Observer). 눈에 보이지 않는 패널은 Zenoh 의 Subscribe 자체를 Pause 시키거나 차트 업데이트를 동결하는 기능이 반드시 구현되어야 브라우저 과부하를 막을 수 있다.

2. 데이터 샘플링(Sampling) 및 다운샘플링(Downsampling) 전략 적용

브라우저는 잘못이 없다. 백엔드에서 폭탄(수백만 개의 행) 을 던진 것이 죄악이다. 1년 치 과거 데이터를 1920x1080 모니터 화면에 전부 다 표시하는 것은 수학적으로 불가능하다.

2.0.1 [Runbook] 데이터의 수학적 도끼질(Downsampling) 런북

백엔드 스토리지(InfluxDB, Prometheus) 나 중계 프록시에서 반드시 처리한 뒤 화면에 보내야 한다.

1. 주기 기반 압축 (Time-based Smoothing)
1분 동안 들어온 데이터가 6,000개 라면, 화면에 보낼 때는 이를 1개의 데이터로 뭉쳐서(Average) 보낸다.

  • Grafana의 Max Data Points 설정: 차트의 가로 폭이 1000픽셀이라면, 백엔드 쿼리에 Max Data Points=1000 파라미터를 강제로 주입하라. 데이터가 아무리 수억 개라도 무조건 1천 개로 쪼그라들어서 날아온다.

2. 변곡점 유지 압축 알고리즘 (LTTB)
단순히 평균(Average) 을 내면, 치명적으로 치솟아 올랐던 “1초짜리 바늘 모양 에러 스파이크(Spike)” 도 뭉개져 사라진다.
Largest Triangle Three Buckets (LTTB) 이라는 시계열 다운샘플링 알고리즘을 백경(Backend) 에 적용하라. 데이터 양은 1/1000 로 부서지지만, 그래프의 산봉우리(Spike) 와 계곡(Dip) 의 뾰족한 시각적 특징은 소름 돋게 원본 그대로 브라우저 화면에 유지된다.

3. 백그라운드 워커(Web Worker)를 활용한 시각화 연산 분산 처리

자바스크립트는 태생적으로 싱글 스레드(Single-Thread) 다. 데이터 파싱 연산이 UI 쓰레드를 막아버리면, 사용자가 마우스 클릭을 해도 패널이 열리지 않는 저주에 걸린다.

3.0.1 [인스펙션] 브라우저 내부의 코어 찢기 전술

HTML UI 의 부드러운 반응 속도를 수호하기 위한 병렬 아키텍처.

1. Zenoh Payload 파싱의 감금
Zenoh 로 날아온 바이너리 페이로드(MsgPack 등) 를 푸는 행위는 극도로 무거운 연산이다.
이 파싱 로직 전체를 worker.js (Web Worker) 안으로 유배 보낸다. 메인 쓰레드는 아무것도 고민하지 않고, 그저 Web Worker 가 예쁘게 해독해서 넘겨준 좌표 평면의 배열(postMessage()) 만 받아다가 즉시 그리기만 하면 마우스는 폭풍 속에서도 끊기지 않는다.

2. 차트 레이아웃 연산 스레드 오프로딩
노드 1,000 대짜리 네트워크 토폴로지를 Cytoscape.js 로 켤 때마다 화면이 3초간 얼어붙는다? 레이아웃 정돈 연산을 워커 안에 가둬야 한다. (예: fcose 의 Web Worker 지원 모드 활성화). 화면은 빙글빙글(Loading Spin) 계속 부드럽게 돌아가면서, 워커가 연산을 마쳤다고 신호를 주면 그때 1ms 만에 노드를 착, 하고 펼치면 된다.

4. 모바일 및 에지 디바이스를 위한 경량화된 UI/UX 설계

관제탑은 8코어 i7 PC 지만, 현장에 있는 공장 직원은 덜떨어진 안드로이드 태블릿이나 구형 스마트폰으로 이 대시보드를 켜야 한다. PC 와 똑같은 데이터를 밀어 넣으면 브라우저 앱 자체가 강제 종료(Crash) 된다.

4.0.1 [Runbook] 타겟 디바이스 맞춤형(Adaptive) 다이어트

기기 스펙에 따라 렌더링의 질량 자체를 바꿔버려라.

1. 해상도 감지식 스트리밍 제어 (Progressive Degradation)
클라이언트 브라우저가 접속할 때 User-Agent 와 해상도가 모바일(Mobile) 라면, Zenoh 에서 쏘는 토픽부터 달라야 한다.

  • PC: subscribe("robot/video/4k")
  • 모바일: subscribe("robot/video/360p") 혹은 차트 내의 점의 개수를 제한하는 다운샘플링 쿼리로 강제 스위칭한다.

2. 에지용 CSS/SVG 오버헤드 박살 내기
모바일 브라우저의 적은 “그림자 폭탄(Box-shadow)” 과 “투명도 연산(Alpha-blending)” 이다.
모바일 기기에 띄운 관제 다이어그램에서는 box-shadow 와 backdrop-filter (Blur) 효과를 강제로 display: none 으로 끄거나 제거해야 하며, 토폴로지를 그릴 때 SVG 노드에 입혀놓은 그라데이션 필터들을 전부 원색(Solid pattern) 으로 갈아 끼워버려라. 이 단순한 CSS 제거만으로 모바일 배터리 광탈과 브라우저 버벅거림을 50% 이상 치료할 수 있다.

5. 시각화 파이프라인 상의 지연 시간 병목 지점 분석 및 해결

그래프가 안 나온다. 로봇 탓인가, 네트워크 탓인가, 그라파나 탓인가, 당신 브라우저 탓인가.

5.0.1 [인스펙션] 시각화 망막 파열 분석 전술

이 병목을 찾아내려면 프론트엔드 개발자도 브라우저를 까고 F12 를 눌러야 한다.

1. 크롬 개발자 도구 (Performance Tab) 해부
브라우저 F12 의 Performance 탭에서 녹화를 누르고 5초간 대시보드를 관찰하라.
노란색 막대(Scripting) 가 화면 전체를 덮고 있으면 백엔드 잘못이 아니라 자바스크립트를 거지같이 짠 것이다. (웹 워커 도입).
반대로 보라색 막대(Rendering / Painting) 가 미친 듯이 치솟아 있다면, 한 번에 너무 많은 점들을 DOM 에 쑤셔 넣느라 브라우저 그래픽 엔진이 터진 것이다. (Canvas 엔진으로 교체 필요).

2. Time-to-Glass (유리 도달 시간) 분석
센서에서 빛이 출발해 내 모니터 유리(Glass) 표면에 닿기까지의 파이프라인 구간을 해부한다.

  • A: 센서 \rightarrow DB (Zenoh Ingestion Time)
  • B: DB \rightarrow 시각화 툴 (Grafana Query Time)
  • C: 화면에 그리기 (Browser Render Time)
    그라파나 하단의 쿼리 인스펙터(Query Inspector) 기능을 켜고 쿼리 시간을 까봐라.
    만약 B구간 쿼리(Query) 속도가 5초가 걸린다면, 아무리 브라우저 프론트엔드 최적화를 해봤자 소용없다. 인프라 DB 로 당장 달려가 타임시리즈 인덱싱(Indexing) 과 하드 디스크 I/O 속도 폭발을 막는 근원적 튜닝(14장 참고) 에 착수해야 한다.
