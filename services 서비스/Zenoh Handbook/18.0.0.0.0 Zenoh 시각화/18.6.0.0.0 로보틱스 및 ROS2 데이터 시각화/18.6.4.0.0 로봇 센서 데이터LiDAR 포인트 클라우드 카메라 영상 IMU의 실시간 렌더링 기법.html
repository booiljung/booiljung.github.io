18.6.4 로봇 센서 데이터 (LiDAR, 포인트 클라우드, 카메라 영상, IMU)의 실시간 렌더링 기법

센서는 무자비하다. 16채널 LiDAR 는 1초에 30만 개의 3D 좌표 스파이크를 토해낸다. 이것을 태평양 건너 관제탑으로 다 보내서 그리면 화면이 뻗고 망이 터진다.

0.1 [Runbook] 데이터의 중력(Gravity) 감쇄 렌더링 런북

고대역폭 시각화는 “얼마나 빨리 버리느냐” 에 달렸다.

1. Point Cloud (수십만 개 좌표) 다운샘플링
로봇 안에서 Zenoh 로 쏘기 전, Voxel Grid Filter (공간 깍두기 압축) 를 거친다.
“반경 5cm 안에 있는 점들은 1개로 퉁쳐서 합친다.”
이 연산 하나로 데이터 백만 개가 1만 개로 압축된다. 관제탑 시각화 도구(Foxglove/RViz) 가 부드러운 초당 30 프레임의 부드러운 회전(Orbit) 을 하려면 원본 데이터 전송의 만행을 원천적으로 쳐내야 한다.

2. H.264 / H.265 비디오 스트림 트래픽 태우기
카메라 이미지(sensor_msgs/Image) 1장은 무손실 일 때 몇 메가(MB) 에 달한다. 이걸 Zenoh 퍼블리셔로 쏘면 로봇의 목숨이 날아간다.
반드시 카메라 프레임을 로봇 단에서 하드웨어 코덱(예: NVIDIA V4L2) 을 통해 H.264/WebRTC 스트림으로 압축한 뒤, 그 압축된 바이너리 자체를 통째로 Zenoh Value 로 실어 보내야 한다. 브라우저 쪽에서는 이 바이너리를 HTML <video> 태그 소스로 다이렉트로 넘겨 압축을 풀게 하는 것이 유일한 저지연(Low-latency) 스트리밍 생존법이다.
