21.1 실전 프로젝트 개요 및 아키텍처 설계

분산 시스템 설계, 데이터 로우팅, QoS(Quality of Service), 보안, 통신 최적화 등 지금까지 다루어 왔던 Zenoh의 핵심 기술과 개념들을 실제 산업 현장의 문제에 적용할 시간이다.

본 장에서는 스마트 팩토리(Smart Factory) 환경에서 다수의 자율주행 물류 로봇(Autonomous Mobile Robot, AMR)과 사물인터넷(IoT) 센서 노드가 유기적으로 동작하는 아키텍처를 그린다. 이러한 환경에서는 수만 개의 센서가 뿜어내는 데이터의 실시간 처리, 이동하는 로봇의 끊김 없는 통신 유지, 클라우드와 에지(Edge) 인프라 간의 동적 데이터 연동이라는 과제가 주어진다. 단일 프로토콜로 에지 센서부터 클라우드 빅데이터 플랫폼까지 이어지는 통신 체계를 Zenoh를 기반으로 구성하라.

이 프로젝트를 완수함으로써 다음과 같은 기술적 목표를 달성한다. 첫째, 마이크로컨트롤러부터 클라우드 백엔드 구축까지 다양한 언어와 플랫폼 간의 상호 운용성을 확인한다. 둘째, ROS2(Robot Operating System 2)의 한계를 극복하는 원거리 오버레이(Overlay) 통신을 구현한다. 셋째, 분산 쿼리(Distributed Query)를 통해 중앙 집중형 데이터베이스의 부하를 줄이고 데이터 중심의 관제 플랫폼(Dashboard)을 완성한다.

1. 스마트 팩토리 및 자율주행 물류 로봇(AMR) 시나리오 정의

현대적인 스마트 팩토리는 정적인 제조 설비뿐만 아니라, 생산 라인을 따라 동적으로 이동하며 물류를 운반하는 자율주행 물류 로봇(AMR)들이 필수적으로 도입된다. 이러한 환경에서는 매우 복잡한 데이터 스트림들이 발생하며 통신적 과제를 만들어 낸다. 이 프로젝트의 핵심 시나리오를 정의하여 설계의 방향성을 명확히 하라.

첫째, 고정된 환경에서 수집되는 초정밀 산업용 IoT(IIoT) 센서 네트워크다. 공장 내 생산 설비들(예: 모터, 프레스기)의 진동, 온도, 전력 소모량 등은 고주파수 단위로 측정되어 기계적 이상이나 예지 보전을 위해 수집된다. 이 데이터들은 밀리초(ms) 단위의 높은 처리량(Throughput)으로 상위 계층으로 라우팅되어야 한다. 이를 위해 C와 구동 환경에 제약이 적은 초소형 디바이스들이 사용된다.

둘째, 동적 환경에서 작동하는 자율주행 물류 로봇(AMR)의 제어 및 관제다. 이 로봇들은 공장 내부를 이동하며 무선 네트워크를 통해 제어된다. 음영 지역을 스칠 때 통신 끊김이나 대역폭 제한이 일어날 수 있다. 로봇에 내장된 LiDAR, 카메라 등 고용량 데이터 센서들을 원격으로 모니터링하면서도 충돌 방지나 이동 경로 업데이트와 같은 중요한 제어 명령들을 지연 없이 주고받기 위해 신뢰성 높은 통신 연결 계층이 필수적이다. 따라서 ROS2에서 Zenoh 프로토콜을 통하여 클라우드 서버와 Seamless한 통신이 유지되어야 한다.

셋째, 글로벌 통합 로봇 관리/관제시스템이다. 여러 대의 로봇과 공장 내 각종 설비들의 센서값을 종합하여 공장 운영자에게 시각적으로 보여주는 대시보드 애플리케이션이다. TypeScript에서 Zenoh를 활용하여, 복잡한 REST API 기반의 폴링(Polling) 방식을 벗어나 실시간 Pub/Sub 통신과 분산 Query 메커니즘을 웹 백엔드 및 모니터링 플랫폼에서 수용해야 한다.

요약하자면, 각 디바이스(센서 노드 중심의 마이크로컨트롤러, ROS2 기반 AMR 모바일 컴퓨팅 유닛, 중앙 클라우드 및 에지 서버)는 독립적인 역할을 지니며, 끊김 없는 정보의 흐름을 보장하기 위해 Zenoh를 통신망의 척추(Backbone)로 삼게 될 것이다.

2. 에지, 안개(Fog), 클라우드 계층을 포함한 전체 시스템 아키텍처 설계

스마트 팩토리 시스템의 안정성과 확장성을 확보하기 위해 컴퓨팅 자원을 에지(Edge), 안개(Fog), 클라우드(Cloud) 3계층으로 분리하여 배치하라. Zenoh 라우터망은 이 세 계층을 관통하는 통신의 연결점 역할을 수행한다. 세 계층 간의 물리적 및 네트워킹 아키텍처를 면밀히 설계해야만 지연 없는 제어가 가능하다.

    subgraph Cloud Layer
        C_Router[Zenoh Router - Cloud]
        Backend[TypeScript 관제 웹/백엔드]
        DB[(InfluxDB / TimescaleDB)]

        C_Router <--> DB
    end

    subgraph Fog Layer / Factory Server
        F_Router[Zenoh Router - Fog Server]
        Monitor[Zenoh 시각화 / 모니터링 노드]
        F_Router <--> Monitor
        F_Router <--> C_Router
    end

    subgraph Edge Layer
        E1_Client[AMR 1: ROS2 Bridge]
        E2_Client[AMR 2: ROS2 Bridge]
        S1_Client[IIoT Edge Sensor: Rust/C]
        S2_Client[Microcontroller: Zenoh-Pico]
        
        E1_Client <--> F_Router
        E2_Client <--> F_Router
        S1_Client <--> F_Router
        S2_Client <--> F_Router
    end

첫째, 에지 계층(Edge Layer)이다. 센서 데이터를 수집하는 소형 디바이스와 자율주행 로봇들이 위치하는 공간이다. 리소스가 극히 제한된 하드웨어는 Zenoh-Pico(C 언어 기반)를 사용하며, C++에서 Zenoh를 통해 구동되는 IIoT 장비들이 데이터를 퍼블리시한다. ROS2에서 Zenoh 브릿지를 구동하는 내비게이션 자율주행 모듈은 주기적인 배터리 및 위치(Odometry) 정보를 상위 라우터로 전달한다.

둘째, 안개 계층(Fog Layer)이다. 공장 내 서버급 장비 또는 소형 게이트웨이들이 이에 해당한다. 각 구역(Zone)별로 Zenoh 라우터 모듈을 띄워 하위 에지 노드들로부터 쏟아지는 통신 트래픽을 지역적으로 1차 종합(Aggregation)하라. 데이터가 클라우드로 무조건 넘어가기 전에 로컬 제어를 담당하거나, 단기적인 통신 단절을 보상할 수 있는 중간 캐싱 역할을 부여한다.

셋째, 클라우드 계층(Cloud Layer)이다. 가장 성능이 좋은 백엔드 서버와 영속성 데이터베이스가 있다. 다수의 공장(Fog) 라우터들이 글로벌 클라우드에 위치한 마스터 다중 라우터 네트워크(Mesh)로 연결된다. 퍼블리시된 시계열 데이터가 스토리지 백엔드에 통합 기록되며, TypeScript에서 Zenoh 및 GraphQL, 웹소켓 등과 인터페이스하여 최종 사용자(관리자)에게 대시보드로 요약된 정보를 제공한다. 이를 통해 에지와 클라우드 환경에서 끊김 없는 통합 시스템 아키텍처가 전개된다.

3. 데이터 흐름(Data Flow) 및 Pub/Sub/Query 토픽 구조 설계

Zenoh 통신망을 원활히 운영하기 위한 가장 기본적이면서도 중요한 단계는 바로 네임스페이스(Namespace) 기반의 토픽(Topic) 설계다. 단순히 계층적 트리 구조를 만드는 것을 넘어서 시스템 규모가 커지고 노드가 추가될 것에 대비하여 URI 형식의 논리적인 와일드카드 매핑을 지원하도록 계층 구조를 직관적으로 만들어라.

스마트 팩토리 시스템의 전체 데이터 흐름(Data Flow)에서 토픽 트리는 크게 세 가지 도메인: 센서(Sensor), 로보틱스(Robotics), 관제(Control/Mgmt)로 나뉜다. 이를 바탕으로 토픽 패스를 다음과 같이 설계하라.

첫째, 고주파수 스트리밍 영역인 IoT 센서 도메인이다.

  • /factory/zone_A/sensor/temp/<sensor_id> : 각 구역 내 설비의 구체적인 온도 모니터링 Pub/Sub 토픽
  • /factory/zone_B/sensor/vibration/<sensor_id> : 진동 데이터. 이 데이터들은 밀리초 단위로 발행되므로 네트워크 최적화 시 라우터 브로커 단에서 필터링이 요구된다.

둘째, 자율주행 물류 로봇(AMR) 영역이다. ROS2 브릿지를 통해 이뤄지는 통신의 토픽 매핑 구조를 고려하라.

  • /amr/<robot_id>/telemetry/odom : 로봇의 실시간 위치 정보. 높은 주기로 클라우드 관제 대시보드로 전달된다.
  • /amr/<robot_id>/cmd_vel : 사용자로부터 로봇으로 전달되는 실시간 수동 조작 제어 명령 패스. 이 패스는 QoS에서 높은 우선순위를 지녀야 한다.
  • /amr/<robot_id>/status/battery : 배터리 등의 상태값.

셋째, 데이터베이스 조회 및 Query-able 도메인이다.
단순 퍼블리시(Publish)를 넘어서 스토리지 백엔드에 저장된 과거 데이터를 조회하거나 로컬 애플리케이션 상태를 분산적으로 물어볼 때(Query) 활용된다.

  • /factory/history/sensor/temp/<sensor_id>?time_range=... : 특정 센서의 과거 이력(TSDB 연계) 조회용
  • /amr/+/status/battery : 와일드카드(+ 및 **)를 적극적으로 사용하여, 관제 시스템이 구동될 때 전체 로봇의 배터리 상태 목록을 한 번에 Pull 형태로 당겨올 수 있도록 설계하라.

결과적으로 센서 단의 C 언어 퍼블리셔, 제어 노드의 Rust 퍼블리셔, 관제단의 TypeScript 기반 Zenoh 쿼리러블(Queryable)이 동일한 토픽 구조를 바탕으로 상호 통신할 수 있으며, 시스템의 모든 데이터는 논리적 주소의 단일화(Single Logical Address Space) 아래 통합된다.

4. 요구 하드웨어(라즈베리 파이, MCU, 클라우드 서버) 및 소프트웨어 스택 명세

실전 프로젝트를 원활하게 시작하기 위해 필요한 하드웨어 및 소프트웨어 스택을 정의하고 준비하라. 개발 환경은 단일 노트북으로 가상화하여 테스트할 수 있지만, 물리적 인프라 환경에서 검증하는 것을 목표로 한다. 분산 시스템이므로 각 하드웨어 노드별 스택을 체계화한다.

첫째, 초소형 마이크로컨트롤러(MCU) 및 임베디드 장비(Edge Device)이다.

  • 하드웨어: ESP32, STM32, 혹은 간단한 아두이노(Arduino) 보드.
  • 소프트웨어 스택: FreeRTOS나 베어메탈(Bare-metal) 환경. C 환경에서의 Zenoh-Pico를 탑재하라. 이들은 저전력으로 온도 센서나 진동 센서 측정값을 일방향으로 송출(Publish)하는 역할을 수행한다.

둘째, 안개(Fog) 계층을 담당하는 중간 게이트웨이 및 소형 컴퓨터다.

  • 하드웨어: Raspberry Pi 4 모델(혹은 젯슨 나노), ARM 기반 64비트 싱글 보드 컴퓨터. 스마트 팩토리의 섹션 단위 중간 공유기로 동작한다.
  • 소프트웨어 스택: Ubuntu Linux 22.04 LTS. 이곳엔 C++에서 Zenoh 혹은 Rust에서 Zenoh 개발을 통한 1차적 복잡한 예지 보전 모델 및 데이터 필터링 기능이 동작하며, 별도 데몬(Daemon)으로 Zenoh 라우터(zenohd)가 실행된다. ROS2와 인프라의 브릿지 허브 역할도 겸한다.

셋째, 로보틱스 측면의 컴퓨팅 노드(AMR)이다.

  • 하드웨어: 모터 컨트롤러, 라이다 센서 장착형 이동 플랫폼 보드 및 이를 제어할 호스트 PC(Raspberry Pi 혹은 Intel NUC).
  • 소프트웨어 스택: Ubuntu 22.04와 ROS2 Humble 분산 프레임워크 기반. ROS2에서 Zenoh를 활용하기 위해 zenoh-bridge-dds 패키지를 설치하라. Navigation2 및 로컬라이제이션 스택이 동작한다.

넷째, 클라우드 서버 및 관제 계층이다.

  • 하드웨어: AWS EC2, GCP Compute Engine 등의 리눅스 기반 가상 머신이나 데스크탑 서버급 파워. x86_64 아키텍처.
  • 소프트웨어 스택: Docker 컨테이너 오케스트레이션을 활용하라. zenohd의 루트 라우터 역할, 시계열 데이터 저장을 위한 InfluxDB, PostgreSQL, 그리고 TypeScript에서 Zenoh 기반으로 구동되는 Node.js(혹은 Next.js) 백엔드를 구동한다. 클라우드의 자원은 무한대라 가정하고, 트래픽 유입에 따라 확장되는 수평적 설계(Scale-out)를 구비하라.

이 다채로운 소프트웨어 스택 위에서 오직 Zenoh만이 모든 언어와 프로토콜 틈새를 투명하게 메워 데이터의 신경망을 완성해 줄 것이다.
