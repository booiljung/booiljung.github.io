21.7.1 Zenoh ACL(Access Control List)을 적용한 토픽별 읽기/쓰기 권한 제어

공장 내외부망이 무결점으로 분리되어 있다는 망상은 버려라. 악의적인 직원이 태블릿을 통해 amr/robot_1/cmd_vel 토픽을 임의로 퍼블리시하면 자율주행 물류 로봇(AMR)이 급발진할 수 있다. 이를 막기 위해 라우터 단에서 ACL 규칙을 강력히 시행하라.

첫째, 모든 클라우드 및 안개(Fog) 라우터 모델의 구성 파일(zenohd.json5)에 접근 제어 항목을 활성화한다.
특정 인증서나 사용자 식별자를 획득한 주체에 한해 특정 네임스페이스 경로만 허가한다.

# zenohd.json5
access_control: {
  enabled: true,
  rules: {
    # 대시보드 관리자 계정: 모든 곳 조회 가능, 제어 권한 있음
    admin_user: {
      "**": "rw"
    },
    # C 언어로 구동되는 에지 센서 노드: 오직 /factory/zone_A/sensor/ 의 쓰기권한(Publish)만 존재
    edge_sensor_cert: {
      "/factory/zone_A/sensor/**": "w"
    },
    # 공장 방문자용 모니터링 앱: 센서 리딩 기능만 가능(Read)
    guest: {
      "/factory/**/sensor/**": "r"
    }
  }
}

둘째, 텔레오퍼레이션 노드(Teleoperation)의 격리다. Rust 기반의 긴급 제어 노드가 작동할 때, 해당 단말에서 생성된 세션(Session)만이 /amr/+/cmd_vel 키 표현식에 w (Write) 권한을 지니도록 엄격히 통제하라.

셋째, 동적 ACL 업데이트다. Zenoh는 운영 도중에도 라우터를 재시작하지 않고 관리자 공간(admin_space)을 통해 동적으로 규칙을 제어할 수 있다. 문제가 발생한 노드의 인증서를 식별하는 즉시 해당 서브젝트의 접근 권한을 0으로 만들어 격리(Isolating)시켜라.
