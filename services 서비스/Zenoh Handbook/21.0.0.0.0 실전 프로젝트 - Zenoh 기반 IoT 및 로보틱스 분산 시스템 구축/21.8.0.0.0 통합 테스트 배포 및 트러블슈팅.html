21.8 통합 테스트, 배포 및 트러블슈팅

아키텍처 설계, 노드 개발, 백엔드 연동, 시각화, 통신 최적화와 보안 적용까지의 긴 여정을 거쳐 스마트 팩토리의 완벽한 모습을 그렸다. 하지만 스크립트 상에서 유려하게 동작하는 코드가 먼지와 진동, 불규칙한 네트워크가 난무하는 실제 공장 현장에서 바로 성공할 리 없다.

본 장의 마지막 절에서는 이렇게 촘촘히 엮인 분산 퍼즐들이 극단적인 장애 조건 속에서도 올바르게 동작하는지 가혹하게 검증하는 과정을 기록한다. 클라우드의 메인 라우터들을 산업 표준인 Docker 환경으로 배포하고 CI/CD로 펌웨어를 쏘며, 강제로 네트워크 선을 뽑아내는 카오스 엔지니어링(Chaos Engineering)을 수행하라. 마지막으로 운영 이후 쏟아지는 트러블들을 모니터링 지표로 분석하여 Zenoh 네트워크를 완벽하게 정복한다.

1. Docker 및 Kubernetes를 이용한 클라우드 라우터 클러스터 배포

스마트 팩토리의 중심 척추인 Zenoh 클라우드 라우터(zenohd)가 죽으면 공장의 데이터 흐름은 완전히 마비된다. 따라서 라우터의 고가용성(High Availability, HA)을 확보하기 위해 인프라를 프로비저닝하는 컨테이너 오케스트레이션 도구(Kubernetes)를 활용하라.

첫째, 라우터 이미지의 컨테이너화다. Eclipse 재단에서 제공하는 공식 Zenoh Docker 이미지(eclipse/zenoh:latest)를 베이스로 삼고, 앞서 작성한 zenohd.json5 설정 파일(ACL, InfluxDB 스토리지 플러그인 포함)을 마운트하여 기동하라.

## docker-compose.yml 예시
version: '3'
services:
  zenoh-router-a:
    image: eclipse/zenoh:latest
    ports:
      - "7447:7447/tcp"
      - "7447:7447/udp"
    volumes:
      - ./zenohd.json5:/zenohd.json5
    command: ["--config", "/zenohd.json5"]

둘째, Kubernetes 환경에서의 클러스터링이다. 클라우드에 3개 이상의 라우터 파드(Pod)를 띄워 Mesh 구조를 만든다. Kubernetes의 Service 혹은 Ingress 모듈에서 포트를 노출하고, 노드 간 연결을 위해 클러스터 내부 DNS 이름 기반으로 zenohd -e tcp/zenoh-router-b:7447 형식으로 코드를 연결시킨다. 한 파드가 죽더라도 K8s가 자동으로 새 파드를 살려내어 통신 백본이 영구적으로 복원되는 인프라를 달성하라.

셋째, TypeScript 관제 백엔드의 동시 배포다. Node.js 백엔드 역시 Docker로 묶어 같은 내부 망에서 라우터 파드들과 붙인다. 이를 K8s 헬름(Helm) 차트로 묶어 공장이 신설될 때마다 단일 명령(helm install smart-factory ./chart)으로 클라우드 제어반 인프라 전체가 수 분 내로 배포되게 만들어야 한다.

2. 다중 노드 환경에서의 시나리오 기반 통합 시스템 테스트

책상 위 노트북 1대에서 도커 스웜으로 돌리는 테스트는 실전에서 아무 의미가 없다.

[가상의 1,000노드 스트레스 테스트 망각]

  1. Mock 센서 군단 배포: 클라우드의 K8s나 저렴한 EC2 스팟 인스턴스를 수백 대 띄워, 컴파일된 C++ 센서 노드 모의(Mock)용 바이너리를 뿌려놓고 초당 100건씩 Publish 하도록 만든다.
  2. 모바일 네트워크 시뮬레이터: AMR 단말 머신에 리눅스 커널 도구(tc qdisc)를 적용해 의도적인 지터(Jitter: 네트워크 속도 균일성 붕괴)와 최대 대역폭 제한(10Mbps) 등 가상의 악조건을 씌운다.
  3. 검증 로직: 관제 대시보드에서 getRobotsStatus() API를 불렀을 때 응답 시간이 1초를 넘기는지 살피고, InfluxDB에 “유실율(Drop Rate) 0.1% 이하“로 데이터가 온전히 꽂히는지 검증해라.

3. 대규모 센서 데이터 발생 시의 병목 구간 분석 및 성능 튜닝

스트레스 로버트가 가동되자 Grafana 대시보드의 에지 라우터 CPU가 100%를 치고 뻗기 시작한다. 어디가 막힌 것인가?

[진단 프로세스와 튜닝 포인트]

  1. 프로파일링: dmesg나 htop으로만 보지 말고, 네트워크 계층 문제인지 디스크 I/O 문제인지 확인해야 한다. InfluxDB 라우터 플러그인이 달린 곳에서 I/O Wait이 90%라면 범인은 네트워크가 아니라 디스크 I/O(쓰기 한계)다.
  2. RocksDB 튜닝: 만약 스토리지 플러그인 I/O 병목이 입증되었다면, batch_size 를 1에서 5000으로 극단적으로 늘려라. 영속화 주기를 1초 단위로 미루더라도 밀려드는 초고속 데이터를 메모리 큐에서 방어해야 한다.
  3. 라우터 다중화 (Scale-out): 에지 라우터 1대가 커버할 수 있는 세션이 초과했다면 구역 라우터를 2대로 분할(Zone A-1, Zone A-2)하고, 클라이언트를 로드 밸런싱 시켜 부하를 낮춰라.

4. 에지 디바이스 펌웨어 무선 업데이트(OTA) 및 CI/CD 파이프라인

수천 대의 로봇과 센서가 공장 곳곳에 설치된 상태에서 버그를 수정하기 위해 엔지니어가 직접 USB 포트를 꽂고 다니는 것은 불가능하다. Zenoh 통신망을 활용하여 코드 개발(Git)부터 단말 기기까지 펌웨어를 무선 업데이트(Over-The-Air, OTA)하는 파이프라인(CI/CD)을 구축하라.

첫째, GitHub Actions 기반 CI 파이프라인 설계다.
C 엔지니어나 Rust 엔지니어가 에지 코드를 커밋(Commit)하면, CI 빌드 서버가 이를 크로스 컴파일(Cross-compile)하여 ARM 아키텍처용 패키지 혹은 바이너리를 생성한다.

둘째, Zenoh를 통한 바이너리(Firmware) 전파다.
생성된 펌웨어 .bin 파일을 일반 REST 서버가 아닌 클라우드 라우터의 특정 토픽(예: /factory/ota/firmware/v1.2)에 거대한 청크 블록 단위로 퍼블리시하라.

셋째, 에지 노드의 OTA 구독(Subscribe) 및 자가 패치(Self-patch) 로직이다.
각 디바이스의 부트로더 보조 시스템(혹은 운영체제 데몬)은 OTA 토픽을 항시 구독한다. 새로운 펌웨어 메시지가 들어오면 버전을 확인한 뒤 무중단으로 로컬 디스크에 내려받는다. 다운로드가 100% 완료되면 Zenoh 브릿지 연결을 안전하게 종료(drop(session))한 뒤, 시스템을 재부팅(Reboot)하여 새로운 C/Rust 버전 노드로 깨어나게 만든다. 이 모든 과정이 인간의 개입 없이 물 흐르듯 일어나야 한다.

5. 네트워크 단절 및 노드 크래시를 가정한 카오스 엔지니어링 테스트

시스템이 정상 상태에서 잘 작동하는 것은 당연하다. 진정한 분산 시스템의 가치는 극한의 장애 상황에서 드러난다. 넷플릭스(Netflix)가 주창한 카오스 엔지니어링(Chaos Engineering) 기법을 도입하여, 공장 네트워크의 인위적인 재앙 속에서 Zenoh가 어떻게 트래픽을 지켜내는지 테스트하라.

첫째, 라우터 크래시 테스트다.
Kubernetes 클러스터 내에서 메인 zenoh-router-a 파드를 강제로 삭제(Kill)하라.

kubectl delete pod zenoh-router-a-5f9c...

이 순간, 수만 개의 센서 패킷과 로봇의 위치 데이터가 일시적으로 미아가 된다. 하지만 에지 노드들은 zenoh-router-b의 엔드포인트를 캐시하고 있거나, 클러스터 IP 덕분에 1초 내로 세션 재접속(Reconnect)을 수행해야 한다. 프론트엔드 대시보드의 차트가 1초의 끊김 후 유려하게 복구되는지 확인하라.

둘째, 패킷 로스(Packet Loss) 및 딜레이 트래픽(Delay Traffic) 시뮬레이션이다.
단일 로봇인 AMR-1의 리눅스 커널에 tc (Traffic Control) 명령어를 주입하여 50%의 패킷 드롭과 200ms의 네트워크 딜레이를 인위적으로 생성하라.

sudo tc qdisc add dev wlan0 root netem loss 50% delay 200ms

ROS2 로컬 DDS 환경이었다면 이러한 네트워크 악화 시 로보틱스 통신이 완전히 박살 났겠지만, Zenoh 브릿지가 QUIC 프로토콜의 패킷 복구 능력과 함께 신뢰성 전송(Reliable) 큐를 버퍼링하며 명령어를 끝끝내 보장해 내는지 점검하라. 이 테스트를 통과해야만 진정한 인터넷 오버레이 망의 자격이 있다.

6. 네트워크 단절 및 복구(Failover) 시나리오에서의 로봇 안전성 검증

AMR이 공장 한가운데 철제 랙 뒤(Dead Zone)로 숨어 클라우드 기반 관제망과의 접속이 끊겼다.

[단절 상황의 로터스(Lotus) 생존 검증]

  1. P2P 대체경로 작동: AMR_01의 코어 연결망이 잘렸을 때, 옆을 지나가던 AMR_02가 스카우팅(Scouting) 레이더에 잡히면 자동으로 AMR_02의 노드를 경유(Routing)하여 관제망에 상태 패킷을 뱉어내는지 확인하라. 이것이 분산 데이터 패브릭의 꽃이다.
  2. 하드 타임아웃 락(Lock): 5초간 모든 통신이 두절(Ping Timeout)된 최악의 상황. 관제탑의 E-Stop 제어 신호마저 못 받는다면 AMR은 그 즉시 자율주행 모터 전원을 차단(Safe-Mode) 해야 한다.
  3. 지능적 재연결(Backoff): 통신망이 복구될 기미가 보일 때 너무 많은 노드가 동시에 connect 를 시도해 라우터가 “연결 폭풍(Connection Storm)“에 질려 뻗어버리는 현상을 막기 위해, 랜덤한 대기 시간(Exponential Backoff) 기반으로 연결을 재시도하도록 튜닝하라.

7. 실전 프로젝트 회고 및 프로덕션 환경 도입을 위한 체크리스트

모든 테스트가 완료되고 시스템이 견고함을 증명했다면 운영 환경(Production)으로 올릴 자격이 주어졌다.

[프로덕션 체크리스트]
도입 직전, 개발팀은 이 물음표들에 철저히 대답(YES)해야 한다.

  • 보안 (Security): 모든 라우팅 토폴로지는 TLS 로 구성되었는가? 임베디드 디바이스부터 관제 서버까지 ACL 규칙 하에 최소 권한만 부여되었는가?
  • 가용성 (Availability): 클라우드 마스터 라우터가 죽었을 때 이중화(HA)된 복제 라우터로 전환되는가? 로컬 에지망이 끊겨도 P2P 생존망으로 데이터 교환이 진행되는가?
  • 모니터링 (Monitoring): 메모리/네트워크 메트릭 수집 파이프라인(Prometheus-Grafana)은 구축되었는가? 새벽 3시에 OOM 에러가 터지면 슬랙(Slack)으로 알람 봇이 정상적으로 호출되는가?
  • 버전닝 (Versioning): 개발에 사용된 Zenoh 라우터 스택의 버전(예: v0.10.x)이 전 노드에 동일하게 통일되어 ABI 및 Protocol 호환성이 파편화되지 않았는가?

이 문항들을 뚫고 릴리즈 된 환경은 로보틱스와 IoT의 물리적 경계를 완전히 무너뜨리는 21세기 최첨단 분산 시스템 그 자체로 거듭날 것이다. 세상의 모든 데이터가 물 흐르듯 유연하게 움직이는 ’Zenoh Data Fabric’의 세계에 온 것을 영광으로 여겨라.

8. 실제 운영 환경에서의 트러블슈팅 사례 및 성능 모니터링 지표 분석

가혹한 테스트를 거쳐도 실제 365일 무정지 운영(Always-on) 체제에 들어가면 예상치 못한 병목(Bottleneck) 현상이나 장애가 발생한다. 이때 장님처럼 헤매지 않으려면 모니터링 메트릭(Metric)을 해석하는 분석 능력을 갖추어야 한다.

첫째, 프로메테우스(Prometheus)를 연동한 라우터 메트릭 수집이다.
Zenoh 라우터는 --rest-http-port를 통해 스스로의 성능 지표 데이터를 내보낼 수 있다. 프로메테우스를 연결하여 zenohd_messages_routed_total, zenohd_session_count, zenohd_dropped_messages 등의 지표를 그라파나(Grafana) 대시보드에 띄워라.

둘째, 빈번한 트러블슈팅 사례: “특정 로봇의 데이터가 관제 시스템에 보이지 않는다.”
이럴 땐 z_get 커맨드라인 툴을 디버거로 사용하라.

  1. 라우터 단에서 해당 로봇 데이터가 스토리지에 박히고 있는지 확인한다: z_get "amr/robot_1/odom"
  2. 제대로 데이터가 나온다면, 관제 웹 백엔드의 TypeScript Zenoh 클라이언트 연결(세션)이 닫혀 있지 않은지 확인한다.
  3. 만약 라우터에서도 나오지 않는다면, 로봇 기체 내 접속하여 zenoh-bridge-dds 데몬이 살아있는지, 아니면 ROS2 내부의 DDS 브로드캐스트가 도메인 ID 충돌로 묵음 상태인지 역추적하라.

셋째, 고주파수 데이터로 인한 라우터 OOM(Out of Memory) 현상이다.
수천 개의 센서가 1kHz로 퍼블리시될 때 대역폭을 감당하지 못하면 라우터 메모리가 급증한다. 이 경우 에지(C, Rust) 단말에서 앞서 배운 데드밴드(Deadband) 필터링이나 다운샘플링 수치를 높이고, 라우터 측 Zenoh 큐(Queue) 사이즈를 제한하여 OOM 크래시를 방지하는 튜닝을 가하라.

트러블슈팅은 마법이 아니다. 통신 패스 전 구간에 대한 이해가 깊어질 때, 디버깅은 예술의 경지에 이른다.

지금까지의 과정으로 독자는 마이크로컨트롤러의 C부터 클라우드의 TypeScript 관제 시스템까지 섭렵하며, 단일 신경망처럼 작동하는 거대한 분산 시스템을 구축하는 완벽한 역량을 증명했다. Zenoh는 그 여정에서 가장 강력한 척추가 될 것이다.
