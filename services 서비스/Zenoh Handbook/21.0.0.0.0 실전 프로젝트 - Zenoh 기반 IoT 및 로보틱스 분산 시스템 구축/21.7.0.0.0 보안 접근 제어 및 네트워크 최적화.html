21.7 보안, 접근 제어 및 네트워크 최적화

수만 개의 노드가 연결된 완벽한 스마트 팩토리 인프라와 관제 대시보드도 단 한 번의 외부 침입이나 악의적인 제어 패킷 탈취에 의해 치명적인 흉기로 변할 수 있다. 분산 네트워크에서 라우터는 데이터를 신뢰성 있게 전달하지만, 그 ’신뢰’가 해커의 데이터에까지 작용해서는 안 된다.

이 절에서는 Zenoh의 네이티브 ACL(Access Control List) 기능과 TLS(Transport Layer Security) 암호화를 통해 누가, 어떤 토픽에, 어떻게 접근할지 엄격히 통제하는 인프라 보호 방법을 구성한다. 아울러 보안 설정으로 인해 발생할 수 있는 네트워크 오버헤드를 극복하고, 대단위 데이터가 몰릴 때 성능을 최적화하는 튜닝 및 라이브니스 모니터링 기술까지 망라하여 견고한 무결점 네트워크를 증명하라.

1. TLS를 이용한 엔드투엔드 데이터 암호화 적용

클라우드의 마스터 라우터와 공장의 에지 라우터 간 통신은 공용 인터넷을 탈 확률이 높다. TLS 적용은 선택이 아닌 필수다.

[TLS 인증서 마운트 및 라우터 설정]
사설 CA 혹은 Let’s Encrypt로 발급받은 server.crt 와 server.key 를 라우터에 주입한다.

// cloud_config.json5
{
  listen: {
    endpoints: ["tls/0.0.0.0:7447"] // tcp에서 tls로 업그레이드
  },
  transport: {
    link: {
      tls: {
        certificate: "/cert/server.crt",
        private_key: "/cert/server.key",
        // 자체 서명 인증서일 경우 에지 라우터에서 도메인 검증 완화 필요
        verify_hostname: false 
      }
    }
  }
}

이제 에지 데몬들은 connect: ["tls/클라우드IP:7447"] 형태로 접속하게 되며, 패킷 스니핑 장비가 중간에 개입해도 데이터는 복호화할 수 없는 바이너리 쓰레기 값으로만 보이게 된다.

2. Zenoh ACL(Access Control List)을 적용한 토픽별 읽기/쓰기 권한 제어

공장 내외부망이 무결점으로 분리되어 있다는 망상은 버려라. 악의적인 직원이 태블릿을 통해 amr/robot_1/cmd_vel 토픽을 임의로 퍼블리시하면 자율주행 물류 로봇(AMR)이 급발진할 수 있다. 이를 막기 위해 라우터 단에서 ACL 규칙을 강력히 시행하라.

첫째, 모든 클라우드 및 안개(Fog) 라우터 모델의 구성 파일(zenohd.json5)에 접근 제어 항목을 활성화한다.
특정 인증서나 사용자 식별자를 획득한 주체에 한해 특정 네임스페이스 경로만 허가한다.

## zenohd.json5
access_control: {
  enabled: true,
  rules: {
    # 대시보드 관리자 계정: 모든 곳 조회 가능, 제어 권한 있음
    admin_user: {
      "**": "rw"
    },
    # C 언어로 구동되는 에지 센서 노드: 오직 /factory/zone_A/sensor/ 의 쓰기권한(Publish)만 존재
    edge_sensor_cert: {
      "/factory/zone_A/sensor/**": "w"
    },
    # 공장 방문자용 모니터링 앱: 센서 리딩 기능만 가능(Read)
    guest: {
      "/factory/**/sensor/**": "r"
    }
  }
}

둘째, 텔레오퍼레이션 노드(Teleoperation)의 격리다. Rust 기반의 긴급 제어 노드가 작동할 때, 해당 단말에서 생성된 세션(Session)만이 /amr/+/cmd_vel 키 표현식에 w (Write) 권한을 지니도록 엄격히 통제하라.

셋째, 동적 ACL 업데이트다. Zenoh는 운영 도중에도 라우터를 재시작하지 않고 관리자 공간(admin_space)을 통해 동적으로 규칙을 제어할 수 있다. 문제가 발생한 노드의 인증서를 식별하는 즉시 해당 서브젝트의 접근 권한을 0으로 만들어 격리(Isolating)시켜라.

3. TLS 인증서를 이용한 노드 간 암호화 통신 구현

데이터를 아무리 토픽별로 통제한다 한들, 공장 외부 인터넷 구간이나 5G망을 지나는 중간에 패킷 스니핑(Packet Sniffing)을 당하면 아무 소용이 없다. Zenoh는 QUIC 및 TCP/TLS 프로토콜을 네이티브로 지원하여, 엔드투엔드(End-to-End) 암호화를 완벽히 보장한다.

클라우드 라우터와 에지 노드 간의 통신에 인증서를 발급하고 암호화 채널을 구축하라.

첫째, 인증기관(CA)과 라우터용 TLS 인증서를 생성한다. OpenSSL 등을 사용하여 ca.crt, server.crt, server.key를 생성한 뒤, 메인 클라우드 라우터(zenohd)가 이 인증서를 물고 실행되게 설정하라.

## 클라우드 라우터 tls 설정
listen: ["tls/0.0.0.0:7447"]
tls: {
  server_cert: "/path/to/server.crt",
  server_key: "/path/to/server.key",
  ca_cert: "/path/to/ca.crt"
}

둘째, C 또는 Rust 에지 클라이언트의 TLS 접속 구성이다.
에지 노드는 신버전 Zenoh API의 설정 빌더를 이용해 동일한 라우터 CA 루트 인증서를 신뢰하도록 세팅한 후 tls/<cloud-router-ip>:7447로 붙어야 한다.

// Rust 클라이언트 예시
let mut config = Config::default();
config.insert_json5("connect/endpoints", `["tls/10.0.0.1:7447"]`).unwrap();
config.insert_json5("tls/root_ca_certificate", `"/path/to/ca.crt"`).unwrap();
let session = zenoh::open(config).res().await.unwrap();

셋째, 고성능을 위한 QUIC의 도입이다. TLS가 보안을 챙기지만 TCP의 헤드 오브 라인 블로킹(Head-of-Line Blocking) 문제 때문에 이동하는 로봇(AMR) 환경에서는 잦은 끊김이 발생할 수 있다. UDP 기반의 QUIC 프로토콜은 기본적으로 TLS 1.3 암호화를 포함하며 핸드셰이크가 신속하다. 로봇에서 클라우드로 붙는 브릿지의 엔드포인트를 quic/10.0.0.1:7447로 변경하는 것만으로 보안과 로밍 연동성을 동시에 확보할 수 있다.

4. 역할 기반 접근 제어(RBAC) 및 Zenoh 인증서(Certificate)를 통한 노드 권한 분리

온도 센서(MCU)가 해킹당해서 로봇 자율주행 정지 명령(command/robot/amr_1/estop)을 쏘는 대참사를 막아야 한다.

[ACL(Access Control List) 매니페스트 적용]
Zenoh 라우터 단에서 각 클라이언트의 인증서 CN(Common Name) 혹은 연결 토큰을 파악하여 권한을 발라낸다.

// acl_config.json5 (클라우드/에지 공통 적용)
{
  access_control: {
    enabled: true,
    rules: [
      {
        // 룰 1: 센서류는 오직 자신의 구역(Zone) 채널에 publish만 가능 (읽기 불가)
        subject: "CN=factory_sensor_*",
        resource: "factory/**",
        permissions: ["pub"] 
      },
      {
        // 룰 2: 관제 백엔드 서버는 모든 것을 읽고 쿼리할 수 있음
        subject: "CN=admin_server",
        resource: "**",
        permissions: ["pub", "sub", "query"]
      }
    ]
  }
}

가장 신뢰할 수 없는 엔드포인트(MCU 등)에는 가장 좁은 권한(Least Privilege) 원칙을 부여하여망의 횡적 전이 공격(Lateral Movement)을 구조적으로 셧다운 시킨다.

5. Liveliness 토큰을 활용한 이상 노드 감지 및 네트워크 자기 복구

디지털 보안만큼이나 중요한 것이 시스템의 운영 안정성 보장(Resilience)이다. 공장 한구석에서 지게차와 충돌한 센서 노드가 박살 나 통신이 두절되거나, 해킹으로 인해 AMR의 ROS2 브릿지 연결이 다운되었을 때 관제 대시보드는 이를 1~2초 내로 즉각 인지해야 한다. 데이터가 오지 않는 것을 “잠시 안 오는 것“으로 착각하면 공정 전체가 꼬이게 된다.

Zenoh 모니터링 생태계가 자랑하는 Liveliness(생존) 토큰 메커니즘을 적용하여 시스템 자가 치유 및 알람 체계를 확립하라.

첫째, 에지 노드의 Liveliness 토큰 선언이다.
Rust 기반 배터리 모니터링 노드나 C 기반 진동 센서 퍼블리셔가 처음 Zenoh 세션을 맺을 때 다음과 같이 생존 토큰을 함께 선언한다.

let token = session.liveliness()
    .declare_token("factory/zone_A/sensor/motor_1/alive")
    .res()
    .unwrap();

이 토큰은 세션이 유지되는 한 네트워크 전역에 “나는 살아있다“는 메타데이터를 유지한다. 단, 폴링오버헤드 없이 TCP/QUIC 연결 상태와 연계되므로 대역폭 낭비가 전혀 없다.

둘째, 클라우드 관제 백엔드(TypeScript)의 Liveliness 구독(Subscribe)이다.
백엔드 서버는 factory/**/alive 및 amr/**/alive 토픽의 생존 여부만을 전용으로 구독하는 옵저버를 띄워라.
어떤 노드가 물리적 크래시나 무선 렌(LAN) 범위 이탈로 연결이 끊기면, 라우터가 즉각 하트비트(Heartbeat) 유실을 감지하고 관제 백엔드로 Drop 이벤트를 퍼블리시한다.

셋째, 자기 복구(Self-healing) 로직 연계다. Drop 이벤트를 받은 백엔드는 관리자에게 긴급 알람(Siren)을 발생시킴과 동시에, 해당 로봇에 할당되어 있던 “물품 이송 작업“을 다른 예비 로봇(Stand-by AMR)의 큐(Queue)로 재할당(Re-assign)하는 비즈니스 로직을 즉각 실행해야 한다. 이를 통해 개별 노드는 실패(Failure)할지언정 시스템 전체는 절대 죽지 않는 진정한 무결함 분산 시스템이 탄생한다.

6. QoS(Quality of Service) 설정을 통한 로봇 제어 명령의 네트워크 지연(Latency) 최소화

TLS 암호화가 도입되면 지연율이 높아지기 마련이다. 이때 네트워크 큐가 꽉 차 있더라도, 긴급 정지 명령만큼은 모세가 바다를 가르듯 가장 1순위로 빠져나가야 한다.

[Zenoh Priority 및 Congestion Control 활용]
클라이언트 코드(Node.js 측 관제 서버)에서 E-Stop 명령을 퍼블리시할 때 옵션을 극단적으로 튜닝한다.

// 긴급 정지 메시지 발행 설정
session.put(`command/robot/${robotId}/estop`, "STOP")
  .priority(1) // 1~8 중 가장 높은 최상위 네트워크 우선순위 부여
  .congestionControl(zenoh.CongestionControl.Drop) // 정체가 심하면 과거 데이터 버리고 뚫고 나감
  .reliability(zenoh.Reliability.Reliable); // 상대의 수신 ACK는 받을 때까지 보장

Priority(1) 마킹을 타고 날아가는 패킷은 중간 라우터 단의 송/수신 버퍼 큐에서 다른 모든 메시지(LiDAR 영상, 온도 데이터)를 새치기하여 제일 먼저 발송 소켓으로 옮겨지는 VIP 대우를 받는다.
