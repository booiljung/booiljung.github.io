21.5 클라우드 관제 백엔드 서버 구축: TypeScript 활용

수많은 에지 데이터가 클라우드로 집결했다면 이제 브라우저 기반의 대시보드 애플리케이션에 렌더링하고, 사용자의 마우스 클릭이 역으로 거대한 공장을 제어할 수 있게 만들 차례다. 최신 웹 백엔드 생태계의 표준인 Node.js 환경에서 작동하는 TypeScript에서 Zenoh 패키지를 활용하라.

지금까지의 전통적인 사물인터넷(IoT) 백엔드는 HTTP REST 폴링(Polling)이나 중앙 집중식 MQTT 브로커를 섞어 쓰는 스파게티 구조였다면, Zenoh 기반의 생태계에서는 백엔드 서버마저도 하나의 거대한 분산 애플리케이션 노드(Node)에 불과해진다.

본 절에서는 @eclipse-zenoh/zenoh-ts 라이브러리를 통해 공장 라우터로 직결된 클라우드 프로세스를 만들고, Express나 NestJS와 결합된 아키텍처를 설계한다. 분산 쿼리를 적극 활용해 무거운 DB의 역할 중 일부를 에지 라우터들에게 위임(Offloading)하여 클라우드 백엔드의 아키텍처를 혁신하라.

1. TypeScript와 Node.js 기반의 관제 웹 서버 아키텍처 설계

스마트 팩토리의 수많은 장비 및 로봇 상태를 모니터링하기 위해 백엔드 구조 역시 비동기 I/O를 적극적으로 관리해야 한다. TypeScript에서 Zenoh 및 Express 프레임워크를 결합하여 웹 서버 아키텍처를 설계하라.

첫째, 웹 서버(Node.js 인스턴스) 자체가 Zenoh 클라이언트 모드로 클라우드 메인 라우터에 세션을 맺도록 구성하라.

import { open, Config, Session } from "@eclipse-zenoh/zenoh-ts";
import express from 'express';

const app = express();
let zenohSession: Session;

async function initZenoh() {
    const config = new Config();
    // 클라우드 라우터(localhost에 데몬으로 띄운 zenohd)에 연결
    config.insertJson5("mode", `"client"`);
    zenohSession = await open(config);
    console.log("Connected to Zenoh Cloud Router");
}

app.listen(3000, async () => {
    await initZenoh();
    console.log("TypeScript Dashboard Backend Running");
});

둘째, 서버리스(Serverless) 트래픽 회피다. 웹 브라우저 클라이언트가 1초에 한 번씩 수만 개의 센서 데이터를 REST API로 묻는 구조를 탈피하라. 그 대신 백엔드는 Zenoh 라우터로부터 Subscribe 콜백을 등록해 놓고, React 기반의 웹 프론트엔드와는 단일 WebSocket으로 연결을 묶어 데이터가 도착할 때마다 즉시 푸시(Push) 해주는 반응형 구조를 채택해야 한다.

셋째, 메모리 리크(Leak) 방지다. TypeScript에서 Zenoh 워커를 생성할 때 동적으로 수많은 로봇들의 토픽 구독(Subscriber) 인스턴스를 만들고 삭제하게 되는데, 로봇 로그오프 시 subscriber.close()를 호출하여 이벤트 루프에 누수가 발생하지 않도록 코드를 철저하게 생명주기(Lifecycle)에 바인딩하라.

2. @eclipse-zenoh/zenoh-ts 라이브러리를 활용한 분산 데이터 쿼리(Query) 구현

REST API에서 GET /sensors/temp를 호출하면 백엔드가 DB를 긁어오는 방식이 일반적이지만, Zenoh에서는 session.get() 이라는 단 한 줄의 Query로 백엔드, 로컬 라우터의 DB 플러그인, 저 멀리 에지의 C 기반 노드까지 모든 저장 매체를 횡단하며 데이터를 당겨올 수 있다.

이것이 TypeScript에서 Zenoh를 사용할 때 경험하는 가장 강력한 무기 중 하나다.

첫째, 백엔드 라디우팅(DB 대신 Zenoh 쿼리 활용)을 구현하라.

import { Session, QueryTarget } from "@eclipse-zenoh/zenoh-ts";

async function fetchRobotBatteries(session: Session) {
    // 모든 amr 로봇의 battery_state 토픽을 일괄 쿼리
    // 이 쿼리는 라우터를 통해 실제 동작 중인 로봇의 Queryable 혹은 InfluxDB 백엔드로 직행한다.
    const replies = await session.get("amr/*/battery_state", { target: QueryTarget.ALL });
    
    let batteryList = [];
    for await (const reply of replies) {
        if (reply.keyExpr) {
            batteryList.push({
                robotId: reply.keyExpr.toString(),
                level: parseFloat(Buffer.from(reply.payload).toString())
            });
        }
    }
    return batteryList;
}

둘째, Queryable 노드의 작성이다. 웹 백엔드 측도 정보의 수집자만이 아니다. 만약 레거시 관제 시스템의 ERP DB에서 ’오늘의 작업 지시서’를 관리한다면, 백엔드는 해당 데이터를 쥐고 있는 Queryable이 되어야 한다.
로봇(Rust)이 아침에 켜지자마자 session.get("erp/today_task/amr_1") 을 날리면, 클라우드의 TypeScript 웹 백엔드의 session.declareQueryable() 콜백이 훅(Hook) 되어 ERP DB를 읽어 역으로 로봇에게 응답(Reply)을 돌려주는 구조를 만들어라.

3. REST API 및 WebSocket 브릿지를 통한 기존 레거시 웹 시스템과의 통합

모든 기업이 하루아침에 전면적인 Zenoh 네트워크로 시스템을 뜯어고칠 수는 없다. 기존의 레거시(Legacy) ERP, MES, 혹은 오래된 관제 모니터링 시스템은 여전히 HTTP REST API나 일반적인 WebSocket을 통해서만 데이터를 받을 수 있는 경우가 잦다. 우리의 클라우드 백엔드는 이 둘 사이를 매끄럽게 연결하는 브릿지가 되어야 한다.

첫째, Zenoh REST 플러그인의 한계를 넘어서라. zenohd 라우터 자체에도 REST API 플러그인이 제공되지만, 인증 로직과 복잡한 비즈니스 파싱 로직을 덧붙이기엔 무리가 있다. 따라서 TypeScript 기반의 Express 백엔드에서 Zenoh 세션을 쥐고 직접 REST 컨트롤러를 작성하라.

app.post('/api/robot/:id/move', async (req, res) => {
    const robotId = req.params.id;
    const { x, y } = req.body;
    
    try {
        const publisher = zenohSession.declarePublisher(`amr/${robotId}/goal_pose`);
        await publisher.put(JSON.stringify({ x, y }));
        res.status(200).send({ status: "success", msg: "Command published to Zenoh network" });
    } catch(err) {
        res.status(500).send({ status: "error" });
    }
});

둘째, 레거시 시스템을 위한 WebSocket 푸시(Push) 서버 구성이다. 기존의 React/Vue 프론트엔드와 백엔드 간에 Socket.IO 등을 연결해 두고, 백엔드가 Zenoh 망에서 amr/*/odom 을 지속적으로 Subscribe 하다가 데이터가 들어올 때마다 WebSocket을 통해 브라우저로 중계(Broadcast)하라. 이 브릿징을 통해 구형 프론트엔드를 한 줄도 고치지 않고서 현장의 공장 데이터를 실시간으로 밀어 넣어 줄 수 있다.

4. 로봇 작업 할당 및 IoT 센서 원격 설정 제어 로직 구현

수집과 조회를 끝마쳤다면, 이번에는 클라우드 관제 백엔드가 능동적으로 공장 시스템에 개입하는 양방향(Bi-directional) 제어 로직을 작성하라. 중앙 서버는 현장의 상황을 파악한 뒤 개별 로봇에게 목적지를 하달하고, 오류가 발생한 센서 노드의 민감도를 원격으로 변경할 수 있어야 한다.

첫째, 자율주행 물류 로봇(AMR)으로의 작업(Task) 할당 로직이다.
백엔드 서버 내부의 스케줄러가 대기 중인 로봇의 목록(Zenoh 쿼리 이용)을 확인하고 유휴 로봇에게 업무를 푸시(Publish)한다.

async function assignTaskToRobot(robotId: string, taskData: any) {
    const topic = `/amr/${robotId}/task/assign`;
    // 작업 데이터를 JSON으로 변환하여 퍼블리시
    await zenohSession.put(topic, JSON.stringify(taskData));
    console.log(`Task dispatched to ${robotId}`);
}

둘째, 원거리 IoT 디바이스(C/Rust 기반) 설정 제어다.
현장에 있는 진동 센서가 너무 많은 노이즈 데이터를 보내고 있다면, 관리자는 대시보드 버튼을 눌러 센싱 주기를 10ms에서 50ms로 늦출 수 있어야 한다.
이 경우 /factory/zone_A/sensor/vibration/motor_1/config/interval 토픽에 50 이라는 값을 퍼블리시한다. 에지 단의 C 퍼블리서는 이 config 토픽을 동시에 Subscribe하고 있으므로 콜백이 호출되는 즉시 내부의 usleep() 지연값을 동적으로 변경하게 된다.

결국 클라우드의 백엔드 코드 단 몇 줄의 퍼블리시 명령이, Zenoh의 최단 경로 라우팅을 타고 수천 킬로미터 밖 공장 모터 옆에 붙은 초소형 칩의 동작 사이클을 즉각적으로 제어하는 짜릿한 엔드 투 엔드(End-to-End) 제어를 완성한다.
