21.2.2 에지 디바이스 간의 P2P(Peer-to-Peer) 통신 브릿지 설정

중앙 클라우드 라우터가 모든 트래픽을 처리하는 것은 원거리 모니터링이나 글로벌 관제에 적합하지만, 짧게 움직이며 장애물을 회피해야 하는 자율주행 로봇(AMR) 간의 긴급 충돌 방지 통신에는 어울리지 않는다. 이런 초저지연 시나리오는 공장 내 에지 노드 간 P2P 통신망으로 풀어야 한다.

Zenoh 클라이언트를 Peer 모드로 기동시켜 에지 계층의 자율망을 구축하라. Peer 노드들은 동일 로컬 네트워크(LAN 또는 지정된 무선망) 상에 있을 때 멀티캐스트(Multicast)나 가십 프로토콜(Gossip)을 이용해 서로의 존재를 자동으로 발견한다.

다음과 같이 Rust 기반 AMR 긴급 제동 모듈을 Peer 모드로 실행하라.

use zenoh::config::Config;

fn main() {
    let mut config = Config::default();
    // 클라이언트나 라우터가 아닌 피어 모드로 지정
    config.set_mode(Some(zenoh::config::Mode::Peer)).unwrap();

    let session = zenoh::open(config).res().unwrap();
    // 이 노드는 동일망의 로봇들과 라우터 없이 직접 데이터를 교환한다.
}

첫째, ROS2를 구동하는 각 로봇에 zenoh-bridge-dds를 피어 모드로 배치하여 서로 통신하게 하라. 이를 통해 A 로봇이 B 로봇의 /scan 위치를 인지할 때 인터넷 구간을 거치지 않아 지연 시간이 밀리초(ms) 단위 이하로 단축된다.

둘째, 스카우팅(Scouting) 기능의 세밀한 조정이다. 만약 로봇이 위치한 무선 AP 트래픽이나 NAT 환경 제약으로 멀티캐스트 발견이 불가능하다면, 특정 안개(Fog) 라우터를 코디네이터로 사용하여 연결될 수 있게 한다. 이 과정에서 C++에서 Zenoh 또는 Rust에서 Zenoh 피어를 띄우며, 고주파수(High frequency) 로보틱스 데이터가 밖으로 새어나가지 않도록 도메인을 한정 짓는 것이 성능의 핵심이다.
