21.4 로보틱스 제어 시스템 연동: ROS2 및 C/Rust 활용

스마트 팩토리의 주인공은 단연 자율주행 물류 로봇(AMR, Autonomous Mobile Robot)이다. 기존에는 개별 로봇이 독립적으로 움직이거나 제한된 와이파이(Wi-Fi) 내에서 중앙 서버의 조종을 받았다면, 이제는 모든 로봇이 Zenoh라는 거대한 신경망 위에서 클라우드 관제 지시를 받고 이웃 로봇들과 충돌을 회피하며 협업해야 한다.

이 절에서는 로보틱스의 사실상 표준이 된 ROS2(Robot Operating System 2) 환경을 Zenoh와 통합하는 과정을 그린다. ROS2의 기저 통신인 DDS(Data Distribution Service)는 로컬 네트워크 환경에 특화되어 있어서, 복잡한 공장 라우팅 환경이나 인터넷 너머의 클라우드로 제어 명령을 보낼 때 치명적인 단점을 드러낸다. 이를 해결하기 위해 zenoh-bridge-dds를 활용하여 원거리 오버레이 망을 구성하라. 또한 C와 Rust 언어로 작성된 경량 텔레오퍼레이션(Teleoperation) 노드를 결합하여, 통신 음영 지대에서도 안전이 담보된 로보틱스 분산 제어 시스템을 구축하라.

1. ROS2 네비게이션 스택 및 LiDAR/카메라 센서 노드 구성

ROS2 환경에서 작동하는 자율주행 물류 로봇(AMR)의 제어 기반을 다지는 것으로 시작하라. 이 단계에서는 아직 인터넷 외부 클라우드를 신경 쓰지 않고, 로봇 기체 내부에 존재하는 로컬 통신만을 설정한다.

첫째, Navigation2 패키지를 기반으로 자율주행 스택을 올린다. 로봇은 /scan (LiDAR 센서 데이터), /image_raw (전방 카메라 영상), /odom (오도메트리, 바퀴 회전량에 따른 위치 추정치) 세 가지 필수 토픽을 퍼블리시하고 있어야 한다. 이 토픽들은 초당 수십 메가바이트의 트래픽을 발생시킨다.

둘째, ROS2의 도메인 ID 검증이다. 여러 대의 로봇이 구동될 때, 각각의 내부 로컬 DDS 통신이 엉키지 않도록 각 기체별로 고유한 ROS_DOMAIN_ID를 할당하라. 예를 들어 AMR-1은 1번을, AMR-2는 2번을 사용하게 지정한다. 이는 나중에 Zenoh 브릿지가 네임스페이스를 관리할 때 충돌을 방지하는 기반이 된다.

셋째, 로컬 QoS의 조정이다. 로컬 하드웨어(카메라)에서 /image_raw를 퍼블리시할 때 DDS의 QoS 프로필을 ‘Best Effort (신뢰성 없음)’ 수준으로 맞춰라. 카메라 영상 프레임 하나를 놓쳤다고 해서 재전송을 시도하는 것은 로컬망 트래픽 체증을 유발하고 이어진 Zenoh 브릿지에 과도한 압력을 행사한다. 오직 명령 하달 토픽(/cmd_vel 등)만 ’Reliable’하게 관리하라.

2. Zenoh-bridge-dds를 이용한 ROS2와 Zenoh 원거리 네트워크 브릿징

로컬에 갇힌 ROS2의 DDS 메시지를 클라우드나 다른 공장의 로봇들에게 넘기기 위해, 그리고 외부에서 들어오는 제어 명령수신을 위해 zenoh-bridge-dds를 배치해야 한다. 브릿지의 역할을 단순한 포워더(Forwarder)로 얕보지 말고, 트래픽을 정교하게 걸러내는 밸브로 사용하라.

첫째, 플러그인 또는 스탠드얼론 실행이다. 각 로봇 내장 PC에서 zenoh-bridge-dds -e tcp/<fog-router-ip>:7447 명령어를 통해 공장 상단 안개(Fog) 라우터로 연결 통로를 열어라. 브릿지가 실행되면 로봇 내부의 모든 DDS 토픽 검색을 시작하여 이를 Zenoh 네임스페이스로 자동 변환(Mapping)한다.

둘째, 네임스페이스 격리(Namespace Isolation) 설정이다. 만약 브릿지에 옵션을 주지 않으면 모든 로봇이 /odom 이라는 동일한 이름으로 Zenoh 망에 데이터를 쏟아낸다. 반드시 --scope /amr/<robot_id> 옵션을 부여하라. 이를 통해 내부 DDS의 /odom 토픽은 글로벌 Zenoh 망에서 /amr/robot_1/odom 으로 예쁘게 포장된다.

셋째, 트래픽 필터링이다. 카메라의 원시 데이터 등 클라우드로 보낼 필요 없는 헤비급 데이터 토픽들을 배제시켜라. zenoh-bridge-dds의 구성 파일에 allow 혹은 deny 규칙 정규식을 활용하여, 오직 시스템 대시보드에서 필요로 하는 /odom, /battery_state, /cmd_vel 토픽들만이 브릿지를 통과하여 거대한 Zenoh 백본을 타도록 제어하라. 이렇게 해야만 무선 공장 네트워크 폭주를 차단할 수 있다.

3. 로봇 상태(Odometry, 배터리)의 클라우드 실시간 스트리밍

ROS2 망과 Zenoh 백본을 브릿지로 연결했다면, 이제 로봇의 핵심 상태 정보가 거침없이 클라우드까지 스트리밍될 시간이다. 이 과정은 공장 현장을 멀리 떨어진 본사 지휘통제실에서 실시간으로 감시할 수 있게 만들어준다.

로봇이 발산하는 위치 정보(Odometry)와 배터리 상태는 Zenoh 망을 타고 중간 공장 안개(Fog) 라우터를 거쳐, 클라우드 글로벌 라우터에 집결한다. 이 데이터들을 저장하고 시각화하기 유리한 구조로 최적화하라.

첫째, 클라우드 측에서 로봇 상태 데이터를 영구 보존(Database)하라. 앞선 절에서 구축한 InfluxDB를 통해 토픽 amr/+/odom 및 amr/+/battery_state의 모든 과거를 시계열 데이터베이스로 빨아들인다.

둘째, 텔레메트리 스트리밍(Telemetry Streaming) 최적화다. 오도메트리는 초당 수십 번 퍼블리시되는데, 실시간 대시보드(TypeScript)에서 이를 전부 그릴 필요는 없다. 애플리케이션 단인 TypeScript에서 Zenoh를 통해 구독(Subscribe)할 때, 일정 주기로 도착하는 최신 데이터만을 샘플링하여 UI에 렌더링함으로써 브라우저의 부하를 막아라.

셋째, 분산 시스템의 신뢰 확인이다. 로봇의 연결이 끊긴 경우를 즉각 판단하기 위해 Zenoh 라이브니스(Liveness) 토큰 메커니즘을 적용하라. 로봇 측 Zenoh 브릿지가 접속을 맺을 때 amr/robot_1/status/liveness 토큰을 선언하고 유지하게 하면, 네트워크 단선 등으로 라우터 연결이 유실되는 순간 해당 토큰이 삭제되어 관제 시스템이 “로봇 1호기 오프라인 연동 실패” 경고를 즉발(Instant)할 수 있게 된다.

4. Rust 및 C를 활용한 원격 로봇 긴급 제어(Teleoperation) 노드 작성

완전한 자율주행이 실패하거나 예상치 못한 장애물이 탐지될 경우, 원격 관리자가 즉시 개입(Intervention)하여 조이스틱 컨트롤러로 AMR을 조종해야 한다. 클라우드에서 조종석(Teleoperation)이 실행된다고 가정할 때 밀리초 단위의 지연만 생겨도 로봇 운용에 큰 사고가 날 수 있다.

이 통신 링크는 가볍고, 멈춤이 없어야 하므로 Garbage Collection 지연이 없는 C 또는 Rust에서 Zenoh 클라이언트 노드를 구축하여 해결하라.

첫째, Rust에서 Zenoh를 이용해 /amr/robot_1/cmd_vel 로 제어 명령을 쏘는 단안 노드를 만들라.

use zenoh::prelude::r#async::*;
use serde_json::json;

#[tokio::main]
async fn main() {
    let session = zenoh::open(zenoh::config::default()).res().await.unwrap();
    let publisher = session.declare_publisher("amr/robot_1/cmd_vel")
        .res().await.unwrap();

    // 조이스틱 입력 루프 가상화
    loop {
        let (linear_x, angular_z) = get_joystick_input();
        
        // ROS2의 geometry_msgs/msg/Twist와 호환되는 JSON 직렬화
        let twist_msg = json!({
            "linear": { "x": linear_x, "y": 0.0, "z": 0.0 },
            "angular": { "x": 0.0, "y": 0.0, "z": angular_z }
        });

        // 초단기 지연 퍼블리시
        publisher.put(twist_msg.to_string()).res().await.unwrap();
        tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
    }
}

둘째, 타임아웃과 셧다운 로직이다. 로봇 본체(C에서 Zenoh 혹은 ROS2) 단에서는 긴급 제어 메시지가 500ms 이상 끊기면 자동으로 cmd_vel을 0으로 만들어 브레이크를 잡는 Fail-Safe 로직을 심어야 한다.

이렇게 클라우드(혹은 원격 통제실)에서 Rust 제어 노드가 쏘아 올린 cmd_vel 메시지는 Zenoh 트리 망을 타고 초고속으로 공장 라우터로 하강한 뒤, zenoh-bridge-dds를 통해 ROS2의 내부 토픽으로 주입되어 즉시 모터를 구동한다.
