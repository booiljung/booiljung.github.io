21.2 Zenoh 인프라 및 네트워크 토폴로지 구축

복잡한 시스템 하드웨어 및 소프트웨어 아키텍처를 정의했으니, 이제 피를 돌게 하는 혈관인 Zenoh 통신망(인프라)을 실제로 배포(Deploy)하고 구축할 단계이다.

물리적으로 떨어진 공장 층(Shop Floor)의 여러 로봇 및 센서들과 클라우드 인프라가 어떻게 안전하고 지연 없이, 데이터 충돌 없이 연결될 것인가를 결정짓는 핵심 토폴로지를 구축하라. 이 토폴로지는 단순한 1대N 서버-클라이언트 모델을 벗어나, 클라우드 중심의 트리(Tree)형 구조와 에지 단의 피어 투 피어(P2P) 구조가 혼합된 하이브리드(Hybrid) 메쉬망으로 설계된다.

이 절에서는 zenohd 라우터 바이너리를 Docker 컨테이너 형태로 안정적으로 호스팅하는 방법부터 스토리지 기능 추가(Storage Backend 연동), 그리고 각 디바이스가 라우터를 경유하지 않고 직접 통신하는 P2P 모드 활성화까지, Zenoh 네트워크 망을 물리적으로 엮어내는 작업을 실습한다. 올바른 네트워크 연결 없이는 상위 애플리케이션의 분산 쿼리나 고성능 Pub/Sub은 모래성에 불과함을 명심하라.

1. 클라우드 중심의 Zenoh 라우터(Router) 인스턴스 배포 및 구성

가장 먼저 모든 통신의 구심점이 될 클라우드 상의 메인 Zenoh 라우터를 배포하라. 시스템 규모에 맞춰 수백, 수천 개의 노드가 동시 접속할 수 있도록 서버 파라미터를 조율하고, 백업 및 다중화(HA)를 위해 2대 이상의 라우터를 클러스터링(Clustering)한다.

## zenoh_cloud_router_1.json5 (클라우드 환경 설정 예시)
{
  mode: "router",
  listen: [
    "tcp/0.0.0.0:7447",
    "quic/0.0.0.0:7447"
  ],
  admin_space: {
    "permissions": "r"  # 외부 접근 시 관리형 API(Read Only) 보호 권한
  }
}

첫째, zenohd 바이너리를 컨테이너화된 환경(Docker)이나 프로세스 매니저(systemd)를 통해 상시 기동하라. 통신 지연을 최소화하기 위해 클라우드 방화벽(AWS Security Group 등)에서 TCP/UDP 7447 포트를 개방한다.

둘째, 멀티 라우터 구성이다. 단일 장애점(SPOF, Single Point of Failure)을 없애기 위해 클라우드 상에 Router-A와 Router-B를 띄운 후, zenohd --connect tcp/<router-a-ip>:7447 명령으로 연결한다. Zenoh 라우터들은 서로 연결되는 즉시 내부 라우팅 테이블 및 현재 네트워크 내의 퍼블리셔, 서브스크라이버 목록을 효율적으로 교환한다.

셋째, 안개(Fog) 계층을 담당할 하위 라우터를 배포하라. 이들은 각 공장(Zone)의 마스터 노드로 작동하며, zenohd -e tcp/<cloud-router-ip>:7447 옵션을 사용해 클라우드 라우터들의 주소를 하드코딩된 엔드포인트(Endpoint)로 물고 올라가야 한다. 이를 통해 공장 현장의 데이터는 하위 라우터에 취합된 후, 고속의 클라우드 메인 라우터로 전달되는 완벽한 트리형(Tree) 통신 토폴로지가 생성된다.

2. 에지 디바이스 간의 P2P(Peer-to-Peer) 통신 브릿지 설정

중앙 클라우드 라우터가 모든 트래픽을 처리하는 것은 원거리 모니터링이나 글로벌 관제에 적합하지만, 짧게 움직이며 장애물을 회피해야 하는 자율주행 로봇(AMR) 간의 긴급 충돌 방지 통신에는 어울리지 않는다. 이런 초저지연 시나리오는 공장 내 에지 노드 간 P2P 통신망으로 풀어야 한다.

Zenoh 클라이언트를 Peer 모드로 기동시켜 에지 계층의 자율망을 구축하라. Peer 노드들은 동일 로컬 네트워크(LAN 또는 지정된 무선망) 상에 있을 때 멀티캐스트(Multicast)나 가십 프로토콜(Gossip)을 이용해 서로의 존재를 자동으로 발견한다.

다음과 같이 Rust 기반 AMR 긴급 제동 모듈을 Peer 모드로 실행하라.

use zenoh::config::Config;

fn main() {
    let mut config = Config::default();
    // 클라이언트나 라우터가 아닌 피어 모드로 지정
    config.set_mode(Some(zenoh::config::Mode::Peer)).unwrap();

    let session = zenoh::open(config).res().unwrap();
    // 이 노드는 동일망의 로봇들과 라우터 없이 직접 데이터를 교환한다.
}

첫째, ROS2를 구동하는 각 로봇에 zenoh-bridge-dds를 피어 모드로 배치하여 서로 통신하게 하라. 이를 통해 A 로봇이 B 로봇의 /scan 위치를 인지할 때 인터넷 구간을 거치지 않아 지연 시간이 밀리초(ms) 단위 이하로 단축된다.

둘째, 스카우팅(Scouting) 기능의 세밀한 조정이다. 만약 로봇이 위치한 무선 AP 트래픽이나 NAT 환경 제약으로 멀티캐스트 발견이 불가능하다면, 특정 안개(Fog) 라우터를 코디네이터로 사용하여 연결될 수 있게 한다. 이 과정에서 C++에서 Zenoh 또는 Rust에서 Zenoh 피어를 띄우며, 고주파수(High frequency) 로보틱스 데이터가 밖으로 새어나가지 않도록 도메인을 한정 짓는 것이 성능의 핵심이다.

3. 스토리지 백엔드(InfluxDB 또는 RocksDB) 연동을 통한 데이터 영속성 확보

네트워크로 흘러 다니는 데이터는 저장되지 않으면 휘발성 메시지에 불과하다. 스마트 팩토리의 예지 보전(Predictive Maintenance) 모델링이나 로봇의 과거 경로 조회를 가능하게 하려면 데이터 영속성을 책임지는 스토리지 백엔드를 연결하라. Zenoh의 분산 쿼리 철학은 데이터가 별도의 외부 애플리케이션 없이 라우터 자체에서 데이터베이스로 직접 흐르도록(Plugin) 지원한다.

클라우드 라우터 노드에 시계열 데이터 저장에 최적화된 InfluxDB 플러그인을 활성화하라.

첫째, InfluxDB를 설치하고 auth_token, org, bucket 정보를 마련한다. 센서들의 주 토픽인 /factory/+/sensor/** 와 로봇 상태 토픽인 /amr/+/status/**를 해당 버킷으로 적재할 타깃으로 설정한다.

둘째, Zenoh 라우터 설정 파일(zenohd.json5)에 스토리지 영역을 다음과 같이 선언하라.

plugins: {
  storage_manager: {
    volumes: {
      influxdb: {
        url: "http://localhost:8086",
        token: "your-influxdb-token",
        org: "factory-org",
        bucket: "sensor-data"
      }
    },
    storages: {
      ts_storage: {
        key_expr: "/factory/+/sensor/**",
        volume: "influxdb",
        append_only: true
      }
    }
  }
}

셋째, 고성능 로컬 파일 시스템 스토리지가 필요하다면 RocksDB 플러그인을 활용하라. 클라우드까지 오가기 벅찬 안개(Fog) 라우터 단기 캐싱 목적이라면, 로컬 라우터의 RocksDB 백엔드에 /factory/+/latest_status 같은 키를 잡아 즉각 로컬 조회가 가능토록 유지하라.

이와 같이 백엔드와 Zenoh 라우터를 통합해두면, 나중에 TypeScript에서 Zenoh 쿼리러블(Queryable)을 만들 필요 없이 클라이언트가 session.get("/factory/zone_A/sensor/temp")만 호출해도 라우터가 InfluxDB에 들어가서 데이터를 꺼내오는 투명한 데이터 페칭(Data Fetching)의 마법이 실현된다.
