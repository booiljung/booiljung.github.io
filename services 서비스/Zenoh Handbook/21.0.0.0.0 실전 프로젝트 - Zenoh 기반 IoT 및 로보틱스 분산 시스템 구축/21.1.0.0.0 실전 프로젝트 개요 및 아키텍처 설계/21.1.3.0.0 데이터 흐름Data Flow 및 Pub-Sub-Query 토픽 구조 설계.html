21.1.3 데이터 흐름(Data Flow) 및 Pub/Sub/Query 토픽 구조 설계

Zenoh 통신망을 원활히 운영하기 위한 가장 기본적이면서도 중요한 단계는 바로 네임스페이스(Namespace) 기반의 토픽(Topic) 설계다. 단순히 계층적 트리 구조를 만드는 것을 넘어서 시스템 규모가 커지고 노드가 추가될 것에 대비하여 URI 형식의 논리적인 와일드카드 매핑을 지원하도록 계층 구조를 직관적으로 만들어라.

스마트 팩토리 시스템의 전체 데이터 흐름(Data Flow)에서 토픽 트리는 크게 세 가지 도메인: 센서(Sensor), 로보틱스(Robotics), 관제(Control/Mgmt)로 나뉜다. 이를 바탕으로 토픽 패스를 다음과 같이 설계하라.

첫째, 고주파수 스트리밍 영역인 IoT 센서 도메인이다.

  • /factory/zone_A/sensor/temp/<sensor_id> : 각 구역 내 설비의 구체적인 온도 모니터링 Pub/Sub 토픽
  • /factory/zone_B/sensor/vibration/<sensor_id> : 진동 데이터. 이 데이터들은 밀리초 단위로 발행되므로 네트워크 최적화 시 라우터 브로커 단에서 필터링이 요구된다.

둘째, 자율주행 물류 로봇(AMR) 영역이다. ROS2 브릿지를 통해 이뤄지는 통신의 토픽 매핑 구조를 고려하라.

  • /amr/<robot_id>/telemetry/odom : 로봇의 실시간 위치 정보. 높은 주기로 클라우드 관제 대시보드로 전달된다.
  • /amr/<robot_id>/cmd_vel : 사용자로부터 로봇으로 전달되는 실시간 수동 조작 제어 명령 패스. 이 패스는 QoS에서 높은 우선순위를 지녀야 한다.
  • /amr/<robot_id>/status/battery : 배터리 등의 상태값.

셋째, 데이터베이스 조회 및 Query-able 도메인이다.
단순 퍼블리시(Publish)를 넘어서 스토리지 백엔드에 저장된 과거 데이터를 조회하거나 로컬 애플리케이션 상태를 분산적으로 물어볼 때(Query) 활용된다.

  • /factory/history/sensor/temp/<sensor_id>?time_range=... : 특정 센서의 과거 이력(TSDB 연계) 조회용
  • /amr/+/status/battery : 와일드카드(+ 및 **)를 적극적으로 사용하여, 관제 시스템이 구동될 때 전체 로봇의 배터리 상태 목록을 한 번에 Pull 형태로 당겨올 수 있도록 설계하라.

결과적으로 센서 단의 C 언어 퍼블리셔, 제어 노드의 Rust 퍼블리셔, 관제단의 TypeScript 기반 Zenoh 쿼리러블(Queryable)이 동일한 토픽 구조를 바탕으로 상호 통신할 수 있으며, 시스템의 모든 데이터는 논리적 주소의 단일화(Single Logical Address Space) 아래 통합된다.
