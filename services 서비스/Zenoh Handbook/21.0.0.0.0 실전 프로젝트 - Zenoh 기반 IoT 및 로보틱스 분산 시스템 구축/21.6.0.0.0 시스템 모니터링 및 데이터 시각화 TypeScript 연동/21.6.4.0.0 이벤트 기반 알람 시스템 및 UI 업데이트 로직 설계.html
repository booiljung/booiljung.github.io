21.6.4 이벤트 기반 알람 시스템 및 UI 업데이트 로직 설계

대시보드 운영자에게 가장 유용한 기능은 아름다운 차트 자체보다, 비정상적인 징후를 알아서 포착하고 경고음을 발생시켜주는 알람(Alarm) 시스템이다. 분산 통신 환경의 철학에 맞춰 각 계단위로 알람 로직을 분배하라.

첫째, 발생 지점의 분산이다. 클라우드 백엔드가 모든 로봇의 배터리를 검사하는 것은 무겁다. 이전 장에서 구성했듯 Rust를 활용한 배터리 제어 제어 노드나 C 에지 디바이스 자체에서 임계치(Threshold)를 초과할 경우 /factory/alarm/battery_low/amr_1 와 같이 별도의 고우선순위 알람 전용 토픽으로 퍼블리시하게 하라.

둘째, 클라이언트 UI 업데이트 로직이다.
웹 프론트엔드의 React 컴포넌트는 모든 일반 데이터를 수신하지만, 별도로 alarm/# 토픽에 바인딩된 글로벌 Context를 유지한다.

// 알람 수신 시 글로벌 토스트 메시지 발생
if (topic.startsWith('factory/alarm/')) {
    toast.error(`긴급 알람 징후: ${payload.message}`);
    // 도면상에 해당 로봇 주위로 붉은색 파동 애니메이션을 추가
    highlightNodeOnMap(payload.source_id);
}

셋째, 알람 상황의 Liveliness 유실 연동이다. 로봇의 하드웨어나 네트워크 자체가 파괴되어 데이터를 전혀 보내지 못하는 ‘조용한 장애’ 상황은 토픽 데이터로 렌더링할 수 없다. Zenoh Liveliness 토큰이 끊겨 백엔드에서 Drop 이벤트를 인지하면, 이를 강제로 인위적 이벤트 스트림으로 생성하여 “연결 유실” 알람을 팝업 시켜라.
