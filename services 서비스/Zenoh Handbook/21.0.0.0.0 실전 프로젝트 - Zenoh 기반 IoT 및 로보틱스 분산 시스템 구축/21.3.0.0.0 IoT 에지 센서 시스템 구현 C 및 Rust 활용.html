21.3 IoT 에지 센서 시스템 구현: C 및 Rust 활용

공장 내 수많은 설비들이 건강하게 작동하고 있는지를 모니터링하기 위해서는 진동, 온도, 전압, 습도 등 방대한 데이터를 지속적으로 수집해야 한다. 스마트 팩토리의 말단(Edge)에서 직접 센서 하드웨어와 맞닿아 작동하는 코드는 하드웨어 리소스 효율성이 극대화되어야 한다. 이를 위해 C와 Rust 언어가 주력으로 사용된다.

본 절에서는 C 언어를 활용해 밀리초 단위의 고주파수 센서 데이터를 처리하고, Rust를 통해 메모리 안전성이 보장된 저전력 제어 노드를 개발하는 과정을 다룬다. 또한 초소형 마이크로컨트롤러(MCU)에서 구동되는 Zenoh-Pico를 통해 단순 데이터 수집뿐만 아니라 직접적인 액추에이터 제어까지 포괄하는 에지 컴퓨팅 생태계를 완성하라.

네트워크 단절과 불안정한 외부 요인으로부터 데이터를 보호하고, 필터링된 유의미한 데이터만을 클라우드로 퍼블리싱(Publishing)하기 위한 최적의 전략이 여기에서 완성된다.

1. C 기반 고주파수 산업용 센서 데이터(진동, 온도) 수집 노드 개발

제조 라인의 핵심 모터나 프레스기에서 발생하는 미세한 이상 징후를 포착하려면 통상 1kHz(초당 1000회) 이상의 빈도로 진동 센서 등에서 데이터를 수집해야 한다. 리소스 오버헤드를 줄이기 위해 C 언어 환경에서 Zenoh C API를 활용하여 경량 데이터 퍼블리셔(Publisher)를 구축하라.

첫째, C에서 Zenoh 세션을 초기화하고 빠른 퍼블리셔를 설정하라.

#include <zenoh.h>
#include <stdio.h>
#include <unistd.h>

int main(int argc, char **argv) {
    z_owned_config_t config = z_config_default();
    z_owned_session_t session = z_open(z_move(config));

    if (!z_check(session)) {
        printf("Zenoh session initialization failed\\n");
        return -1;
    }

    z_owned_publisher_t pub = z_declare_publisher(z_loan(session), z_keyexpr("factory/zone_A/sensor/vibration/motor_1"));
    
    // 센서 데이터 고속 Loop
    for (int i=0; i<10000; i++) {
        char buf[64];
        sprintf(buf, "{\\"vib_x\\": 0.5, \\"vib_y\\": 0.2, \\"seq\\": %d}", i);
        z_publisher_put(z_loan(pub), (const uint8_t *)buf, strlen(buf), NULL);
        usleep(1000); // 1ms
    }

    z_undeclare_publisher(z_move(pub));
    z_close(z_move(session));
    return 0;
}

둘째, 패킷 최적화다. JSON 문자열은 가독성이 좋지만 매 밀리초마다 전송하기엔 네트워크 대역폭 낭비가 심하다. 현업에서는 CBOR이나 FlatBuffers 같은 바이너리 직렬화 포맷을 사용하여 페이로드(Payload) 크기를 최소화하라.

셋째, 메모리 배포를 최적화하라. C 언어에서 포인터를 활용해 센서 드라이버가 읽은 메모리 블록을 그대로 Zenoh 페이로드 주소로 맵핑(Zero-copy) 전송을 시도하여 CPU 점유율을 대폭 낮춰야 한다. 이것이 C에서 Zenoh를 활용할 때의 핵심 강점이 된다.

2. Rust를 활용한 저전력 배터리 모니터링 및 안전 제어 노드 개발

자율주행 환경(AMR)이나 외부 무선 센서 노드에서는 하드웨어 자원이 제한될 뿐만 아니라 구동 안전성(Safety)이 보장되어야 한다. Rust 언어의 소유권(Ownership) 모델과 동시성 제어 능력을 활용하면, 데이터 페칭 쓰레드와 Zenoh 통신 쓰레드를 락(Lock) 없이 안전하게 구현할 수 있다.

Rust에서 Zenoh를 활용하여, 로봇 배터리 전압이 임계치 이하로 떨어질 때 즉각 상위 계층으로 긴급 메시지를 퍼블리시하고 로컬 액추에이터를 통해 전원을 차단하는 제어 노드를 구현하라.

첫째, Rust의 비동기 프레임워크인 tokio와 zenoh-rs를 결합하라.

use zenoh::prelude::r#async::*;
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let session = zenoh::open(zenoh::config::default()).res().await.unwrap();
    let publisher = session.declare_publisher("amr/robot_1/status/battery")
                           .res().await.unwrap();

    loop {
        let voltage = read_battery_voltage(); // 하드웨어 I2C/SPI 가상 함수
        
        // 데이터 전송
        publisher.put(voltage.to_string()).res().await.unwrap();
        
        // 안전 임계치 검사
        if voltage < 11.5 {
            println!("Low voltage detected! Shutting down non-essential modules.");
            trigger_local_shutdown();
        }
        
        sleep(Duration::from_secs(5)).await;
    }
}

둘째, 멀티스레딩 데이터 수집이다. Rust의 채널(Channel) 메커니즘을 사용해 GPIO에서 값을 읽는 쓰레드와 Zenoh로 데이터를 Publish하는 비동기 쓰레드를 깔끔하게 분리하라. 이를 통해 통신 지연(Network Latency)으로 인해 하드웨어 센싱 주기가 밀려버리는 현상을 방지할 수 있다. 전력 제한이 있는 에지 기기에서는 CPU wake-up 주기를 타이트하게 제어하여 배터리 수명을 극대화하라.

3. Zenoh-Pico를 이용한 초소형 마이크로컨트롤러(MCU) 액추에이터 연동

운영체제(OS)가 없는 베어메탈(Bare-metal)이나 FreeRTOS 기반 마이크로컨트롤러(MCU, 예: ESP32, STM32)에서는 표준 IP 스택이 무겁기 짝이 없다. 이런 한계 상황에서 클라우드 관제 지시를 받고 모터를 구동하거나 밸브를 제어하는 액추에이터 노드를 만들기 위해서는 Zenoh-Pico가 필수적이다.

Zenoh-Pico는 아주 적은 수 킬로바이트(KB)의 RAM만으로도 시리얼통신, Bluetooth, Thread/Zigbee 및 최소한의 UDP/TCP 스택 위에서 Zenoh 프로토콜을 온전히 구현해낸다.

첫째, ESP32 기반 MCU에서 Zenoh-Pico의 Sub 기능을 활성화하여 클라우드에서 내려오는 명령을 기다리는 코드를 작성하라.

#include "z_pico.h"

void command_callback(const z_sample_t *sample, void *arg) {
    printf("Received command: %.*s\\n", sample->payload.len, sample->payload.start);
    // 예: "VALVE_OPEN" 문자열 수신 시 GPIO 핀 High 설정
    // turn_on_relay();
}

void main() {
    z_session_t session;
    z_open(&session, NULL);

    z_owned_closure_sample_t callback = z_closure_sample(command_callback, NULL);
    z_subscriber_t sub;
    z_declare_subscriber(&sub, &session, z_keyexpr("factory/zone_A/actuator/valve_1"), z_move(callback), NULL);

    // FreeRTOS Task Loop
    while (1) {
        z_v_task_delay(10); // 양보 처리
    }
}

둘째, Zenoh-Pico 디바이스들은 직접 클라우드 라우터로 붙는 것보다 직렬 통신(Serial)을 통해 안개 계층(Fog) 라우터(Raspberry Pi 등)에 붙게 설정하라. 공장에 수만 개의 노드가 퍼져 나갈 때, Zenoh-Pico는 상위 라우터라는 플러그(Plug)에 의존하는 스포크(Spoke) 역할을 한다.

이렇게 구현된 마이크로컨트롤러 액추에이터는 TypeScript에서 Zenoh를 사용하는 웹 대시보드 관리자가 클릭 한 번으로 토픽 /factory/zone_A/actuator/valve_1에 “ON” 페이로드를 Put(Publish) 하는 즉시, 밀리초 단위의 지연만으로 즉시 물리적 장비를 제어(Control)하게 만든다.

4. 에지 노드의 데이터 전처리 및 Zenoh 네트워크로의 퍼블리싱(Publishing)

현장에서 수집되는 로(Raw) 데이터를 가공 없이 곧바로 클라우드나 라우터로 밀어 넣는 것은 최악의 설계다. 모든 노이즈 섞인 데이터가 네트워크를 막아서 정작 중요한 긴급 제어 패킷이 누락될 수 있다. 퍼블리싱 직전에 에지 단말에서 수행해야 할 데이터 전처리(Pre-processing) 전략을 구현하라.

첫째, 데드밴드(Deadband) 필터링이다. 온도 센서와 같이 천천히 변하는 값의 경우, 이전 송신값 대비 지정된 임계치(예: 0.5도) 이상 변했을 때만 퍼블리싱하도록 하라. 센싱은 밀리초 단위로 하되, 전송은 이벤트 기반으로 축소하는 것이 대역폭 절감의 기본이다.

둘째, 타임스탬프 동기화 및 메타데이터 주입이다. 클라우드에서 여러 공장의 데이터를 합쳐서 볼 때 기준 시간이 틀어지면 시계열 분석(TSDB)이 망가진다. 에지 노드는 NTP(Network Time Protocol) 등을 활용해 정확한 시간을 유지하고, 메시지 페이로드 내부에 반드시 수집 시점의 Unix Timestamp를 주입하라.

// C 언어에서의 페이로드 구성 예시
sprintf(buf, "{\\"value\\": %f, \\"ts\\": %ld}", current_temp, get_current_timestamp());
z_publisher_put(z_loan(pub), (const uint8_t *)buf, strlen(buf), NULL);

셋째, QoS(Quality of Service) 기반의 차등 퍼블리싱이다. 일반적인 모니터링 데이터들은 베스트 에포트(Best Effort) 수준으로 처리하여 간혹 누락되더라도 재전송을 피하라. 반면 배터리가 폭발 온도에 도달했다는 이상 신호는 신뢰성(Reliability) 우선 모드로 z_publisher_put 시 옵션을 주어 전송하여, 네트워크의 무의미한 재전송 폭주를 막고 진정한 가치 단위의 데이터 라우팅을 설계하라.
