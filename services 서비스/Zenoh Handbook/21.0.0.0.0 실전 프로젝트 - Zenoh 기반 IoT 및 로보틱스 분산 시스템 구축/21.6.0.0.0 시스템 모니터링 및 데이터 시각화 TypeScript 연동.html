21.6 시스템 모니터링 및 데이터 시각화: TypeScript 연동

보이지 않는 백그라운드 데이터의 홍수는 데이터 자체만으로는 인간 관리자에게 아무런 영감을 주지 못한다. 공장에 분산된 수천 개의 센서와 숨 막히게 움직이는 수십 대의 자율주행 물류 로봇(AMR)의 상태 데이터를 단일 스크린에 응집력 있게 그려내는 일은 스마트 팩토리의 정점이다.

웹 생태계와의 고성능 결합, 이것이 이 절에서 추구하는 목표다. Zenoh 모니터링 토픽을 백엔드에서 낚아채고, WebSocket과 React 혹은 Vue.js 프레임워크를 묶어 완전히 살아 숨 쉬는 대시보드를 구축하라. 로봇들의 x, y 좌표를 바탕으로 2D/3D 지도 위에 점을 찍고, 밀리초 단위로 쏟아지는 진동 센서의 궤적을 실시간 차트로 렌더링(Rendering)하는 과정에서 TypeScript 생태계가 진가를 발휘하게 된다. Zenoh 시각화의 정수를 경험하라.

1. React 또는 Vue.js 기반의 실시간 데이터 대시보드 구축

수천 개의 디바이스 데이터를 인간이 해석할 수 있도록 UI 창으로 가져오기 위해, 최신의 SPA(Single Page Application) 프레임워크인 React 또는 Vue.js를 활용하여 전체 대시보드의 골격을 세운다. 웹 환경에서 분산 네트워크의 메시지가 밀려들어올 때, 브라우저가 버벅이거나 메모리가 고갈되지 않게 렌더링 성능을 관리하는 것이 핵심이다.

첫째, 상태 관리(State Management) 라이브러리와 WebSocket의 결합이다. 브라우저는 직접 Zenoh 포트에 붙기 불안정할 수 있으므로 앞선 절에서 만든 TypeScript 백엔드의 WebSocket을 경유하여 데이터를 수신하라.

// React 예시
useEffect(() => {
    const socket = new WebSocket('ws://localhost:3000/telemetry');
    socket.onmessage = (event) => {
        const data = JSON.parse(event.data);
        // 상태 전파
        dispatch(updateRobotState(data));
    };
    return () => socket.close();
}, []);

둘째, 렌더링의 격리와 최적화다. 1초에 열 번씩 들어오는 온도 데이터가 앱 전체 트리(Tree)를 리렌더링(Re-rendering)시키게 두지 마라. 상태 관리를 분리하거나 React.memo 등 메모이제이션 도구를 적극 활용하여 트래픽이 집중되는 차트나 텍스트 하위 컴포넌트만 별도로 다시 그려지게 조치해야 한다.

셋째, 초기 데이터 로딩(Cold Start) 시나리오 설계다. 관리자가 브라우저를 처음 열었을 때, 시스템은 백엔드를 통해 Zenoh Query(session.get)를 날려 전체 로봇의 마지막 상태와 스토리지의 어제자 이력을 끌어와 초기 화면 세팅을 마쳐야 한다. 그 이후에 스트리밍된 Pub/Sub 구독 데이터를 덧칠(Patching)하는 방식의 하이브리드 로직을 짜라.

2. Zenoh 시스템 모니터링 플러그인 활성화 및 트래픽 메트릭 수집

네트워크 자체가 얼마나 건강한지 파악하는 것이 우선이다. Zenoh는 라우터 헬스 체크를 위한 내장 REST API를 제공한다.

[Admin Space 활용]
클라우드 라우터의 설정에서 rest 플러그인을 활성화("http_port": 8000) 한 뒤, 브라우저에서 http://localhost:8000/@/router/local/** 로 접속해 보라.
현재 활성화된 모든 세션 식별자(Session ID), 연결된 엣지 라우터의 개수, 초당 처리 바이트 수가 JSON 형태로 표출된다.

이 데이터를 Prometheus가 주기적으로 긁어갈 수 있도록 Node.js 서버에 가벼운 “Metrics Exporter” 엔드포인트를 노출해 두면 인프라 계층의 대시보드(Grafana) 준비가 마무리된다.

3. 2D/3D Canvas를 이용한 공장 내 AMR 실시간 위치 매핑

ROS2와 Zenoh를 거쳐 올라온 로봇 오도메트리(Odometry - amr/+/odom) 데이터 속 기체의 x, y 좌표 및 방향각(Yaw) 정보를 화면에 그리는 기능은 대시보드에서 가장 시각적 몰입도가 높은 요소다.

정적인 HTML DOM으로 수십 대의 로봇 아이콘을 움직이면 퍼포먼스 제약이 심하다. 이를 회피하기 위해 HTML5 Canvas 2D 혹은 Three.js 기반의 WebGL 3D 캔버스를 활용하라.

첫째, Canvas 2D 기반 매핑이다. 배경에는 공장 레이아웃 지도를 이미지로 깔아두고, 그 위에 로봇의 현재 좌표를 뷰포트(Viewport) 픽셀 좌표로 변환(Transformation Matrix)하여 렌더링해라.

function drawRobots(ctx, robotsList) {
    ctx.clearRect(0, 0, canvas.width, canvas.height);
    // (옵션) 배경 지도 렌더링
    
    robotsList.forEach(robot => {
        const px = mapXtoPixel(robot.x);
        const py = mapYtoPixel(robot.y);
        
        ctx.save();
        ctx.translate(px, py);
        ctx.rotate(robot.theta); // 방향 회전
        
        // 화살표 또는 로봇 아이콘 드로잉
        ctx.fillStyle = robot.battery < 20 ? 'red' : 'green';
        ctx.fillRect(-10, -10, 20, 20);
        
        ctx.restore();
    });
}

둘째, 렌더링 루프 분리다. 데이터가 도착할 때마다 그리지 말고 브라우저의 requestAnimationFrame을 사용하여 모니터 주사율(주로 60fps)에 맞춰 부드럽게 화면을 갱신해야 시각화 안정성이 높다. 이 과정은 Zenoh 시각화의 정점이며 ROS의 Rviz2 대신 웹에서 클라우드 관제를 대체하는 강력한 효과를 발휘한다.

4. TypeScript 기반 프론트엔드(React/Vue) 웹 대시보드 구축

관제 모니터에는 프론트엔드 프레임워크인 React(또는 Vue)를 사용한다.

[UI 컴포넌트 아키텍처]

  1. Map Component: 로봇의 현재 x, y 좌표를 찍어줄 2D 캔버스.
  2. Chart Component: 센서의 진동/온도 값을 보여줄 Chart.js 또는 Recharts 모듈.
  3. Control Panel: 관리자가 특정 로봇에게 긴급 정지(E-Stop)나 목적지 하달을 버튼 클릭으로 지시하는 패널.

[상태 관리 포인트]
프론트엔드 앱이 마운트(Mount) 될 때 WebSocket을 통해 Node.js 서버와 직결 채널을 열고, 초당 10번씩 들어오는 수십 개의 로봇 좌표 상태를 React의 Redux 나 Zustand 같은 전역 상태 저장소에 쉼 없이 밀어 넣는다 (Push 방식).

5. 센서 데이터의 실시간 시계열 그래프 시각화

AMR의 현재 배터리 소모량과 컨베이어 벨트의 진동 센서 데이터를 실시간 선 그래프로 렌더링한다.

[WebSocket 기반 실시간 스트리밍]
Node.js 백엔드가 Zenoh로부터 telemetry/zone_A/sensor/vib 를 수신할 때마다, 즉시 WebSocket 통로로 직렬화하여 뿌린다.

// React 프론트엔드 측 수신 코드 (Chart.js 연동 예시)
useEffect(() => {
  const ws = new WebSocket('ws://cloud-server:3000/stream');
  
  ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data.topic.includes('vib')) {
      // 차트의 데이터 배열 끝에 새 값을 밀어넣고 랜더링 트리거
      setChartData(prev => [...prev.slice(-50), data.value]); 
    }
  };
  return () => ws.close();
}, []);

배열(Array)의 크기를 항상 최근 50개(또는 100개)로 제한(slice(-50)) 하는 윈도우잉(Windowing) 기법을 적용해야, 며칠 동안 켜진 브라우저의 메모리가 터지는 OOM 장애를 예방할 수 있다.

6. 차트 라이브러리를 활용한 센서 시계열 데이터 시각화

AMR의 위치가 2D 캔버스 공간의 점이라면, 공장 에지(Edge)의 수많은 모터와 프레스기가 뿜어내는 데이터는 시간의 축을 따라 흐르는 시계열(Time Series) 데이터로 시각화해야 한다. C 기반의 고주파수 노드에서 수집되어 InfluxDB에 저장된 데이터, 그리고 현재 실시간으로 스트리밍되는 데이터를 Chart.js나 Apache ECharts 같은 라이브러리에 렌더링하라.

첫째, 과거 데이터와 실시간 데이터의 봉합이다. 화면에 차트를 그리기 시작할 때, 클라우드의 TypeScript 웹 백엔드가 Zenoh의 session.get('/factory/zone_A/sensor/temp') 분산 쿼리를 영속성 백엔드(InfluxDB)로 날려 최근 1시간의 과거 이력(History) 배열을 반환받는다. 이를 차트에 일차적으로 그린 후, 이후 들어오는 실시간 데이터는 배열의 뒤에 push() 하고 가장 오래된 데이터를 shift() 하며 애니메이션 효과를 부여하라.

둘째, 주파수 다운샘플링(Down-sampling)이다. 산업용 진동 센서가 1kHz로 퍼블리시한 모든 데이터를 브라우저에 들이부으면 탭이 멈춘다.
TypeScript 백엔드가 데이터를 수신할 때 LTTB(Largest Triangle Three Buckets) 알고리즘이나 단순 평균법을 적용하여 1000개의 점을 10개로 묶어 프론트엔드로 전달하라. 이를 통해 브라우저 차트 라이브러리의 DOM 렌더링 병목 현상을 원천적으로 차단할 수 있다.

7. 로봇의 현재 위치(Map) 및 라우팅 경로의 웹 브라우저 시각화 구현

AMR에서 생산되는 ROS2의 nav_msgs/Odometry 메시지는 x, y, z 좌표와 쿼터니언(Quaternion) 회전각을 포함한다. 브리지(Bridge)를 통과하여 Node.js를 거쳐 브라우저로 들어온 이 숫자를 2D 평면도에 찍어라.

[Canvas API를 활용한 로봇 드로잉]
공장의 도면 레이아웃 이미지를 백그라운드에 깔고, 그 위에 HTML5 Canvas를 오버레이(Overlay) 한다.

// 렌더 타겟 함수 예시 (React Canvas)
function drawRobot(ctx: CanvasRenderingContext2D, x: number, y: number, yaw: number) {
    ctx.save();
    // 공장 맵의 배율에 맞게 픽셀(Pixel) 좌표로 스케일링 변환
    ctx.translate(x * SCALE_FACTOR, y * SCALE_FACTOR); 
    // 로봇의 회전각(Heading) 반영
    ctx.rotate(yaw);
    
    // 로봇을 나타내는 화살표나 원 그리기
    ctx.fillStyle = "blue";
    ctx.fillRect(-10, -10, 20, 20); 
    ctx.restore();
}

웹 브라우저는 60FPS로 draw_robot 을 무한 호출하며, 전 세계 어디서 접속하든 공장 바닥의 로봇 수십 대가 꿀벌처럼 움직이는 관경을 끊김 없이 감시할 수 있게 된다.

8. 이벤트 기반 알람 시스템 및 UI 업데이트 로직 설계

대시보드 운영자에게 가장 유용한 기능은 아름다운 차트 자체보다, 비정상적인 징후를 알아서 포착하고 경고음을 발생시켜주는 알람(Alarm) 시스템이다. 분산 통신 환경의 철학에 맞춰 각 계단위로 알람 로직을 분배하라.

첫째, 발생 지점의 분산이다. 클라우드 백엔드가 모든 로봇의 배터리를 검사하는 것은 무겁다. 이전 장에서 구성했듯 Rust를 활용한 배터리 제어 제어 노드나 C 에지 디바이스 자체에서 임계치(Threshold)를 초과할 경우 /factory/alarm/battery_low/amr_1 와 같이 별도의 고우선순위 알람 전용 토픽으로 퍼블리시하게 하라.

둘째, 클라이언트 UI 업데이트 로직이다.
웹 프론트엔드의 React 컴포넌트는 모든 일반 데이터를 수신하지만, 별도로 alarm/# 토픽에 바인딩된 글로벌 Context를 유지한다.

// 알람 수신 시 글로벌 토스트 메시지 발생
if (topic.startsWith('factory/alarm/')) {
    toast.error(`긴급 알람 징후: ${payload.message}`);
    // 도면상에 해당 로봇 주위로 붉은색 파동 애니메이션을 추가
    highlightNodeOnMap(payload.source_id);
}

셋째, 알람 상황의 Liveliness 유실 연동이다. 로봇의 하드웨어나 네트워크 자체가 파괴되어 데이터를 전혀 보내지 못하는 ‘조용한 장애’ 상황은 토픽 데이터로 렌더링할 수 없다. Zenoh Liveliness 토큰이 끊겨 백엔드에서 Drop 이벤트를 인지하면, 이를 강제로 인위적 이벤트 스트림으로 생성하여 “연결 유실” 알람을 팝업 시켜라.
