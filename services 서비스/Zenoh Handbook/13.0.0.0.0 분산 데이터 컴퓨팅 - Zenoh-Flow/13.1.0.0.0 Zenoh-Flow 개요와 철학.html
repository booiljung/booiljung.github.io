13.1 Zenoh-Flow 개요와 철학

“데이터를 A에서 B로 옮기는 법(Zenoh)” 을 마스터했다면, 이제는 “그 중간에 흘러가는 데이터를 공중에서 어떻게 쪼개고, 합치고, 계산할 것인가” 에 대한 패러다임 전환이 필요하다.

이 챕터에서는 데이터가 한 곳에 멈춰서 처리되는 낡은 폰 노이만 구조를 벗어나, 파이프파인 그 자체가 살아 움직이는 컨베이어 벨트가 되는 Zenoh-Flow (분산 데이터플로우 프레임워크) 의 기계론적 지배 철학을 해부한다.

1. 데이터플로우 프로그래밍(Dataflow Programming) 패러다임 이해

로봇의 눈(카메라)이 영상을 찍어 자율주행 알고리즘(추론) 으로 보내고, 또 모터 제어기로 보내는 긴 파이프라인(Pipeline)을 상상해 보라.

1.0.1 [인스펙션] 제어 흐름(Control Flow) 의 종말

1. 전통적인 명령형 프로그래밍의 한계

  • “카메라 스레드야, 영상을 찍어라 -> 찍었니? -> 그럼 추론 함수를 호출해라 -> 그 리턴값을 모터 함수로 줘라”
  • 이 방식은 “프로그램의 논리(Logic)” 가 데이터를 직접 끌고 다닌다. 데이터가 커지고 컴퓨터가 2대, 3대로 늘어나면 로직 자체가 스파게티처럼 엉키고 소켓(Socket) 코딩을 하다가 무너진다.

2. 데이터플로우 (Data-driven) 의 지배

  • 데이터플로우에서 “메인 프로그램(main 함수)” 따위는 존재하지 않는다.
  • 오직 각자의 방진복을 입고 컨베이어 벨트(Belt) 앞에 서 있는 독립적인 노동자(Node) 들만 존재할 뿐이다.
  • 노드 A는 눈앞에 카메라 데이터가 도착하면 즉시 가공해서 옆 벨트로 던진다. 노드 B는 그 데이터가 자기 벨트에 떨어질 때까지 잠을 잔다.
  • 데이터가 흐르는(Flow) 현상 그 자체가 프로그램의 실행(Execution) 을 강제(Run) 하는 궁극의 물류센터 아키텍처다.

2. Zenoh 기반 분산 데이터 컴퓨팅의 필요성 및 장점

세상에는 텐서플로우나 ROS2 Node 같은 훌륭한 파이프라인 프레임워크가 이미 널렸다. 그런데 왜 굳이 Zenoh-Flow 여야 하는가?

2.0.1 [Runbook] 물리적 분산(Geo-Distributed) 한계계측

1. 네트워크 위상의 완벽한 은닉 (Location Transparency)
ROS2 나 파이썬 멀티프로세싱은 “내 컴퓨터 안” 에서는 완벽하게 데이터를 퍼나르지만, 컨베이어 벨트가 바다 건너 AWS 에 있는 서버로 이어져야 한다면? 개발자는 하던 일을 집어치우고 IP 주소와 포트번호를 하드코딩해야 한다.
Zenoh-Flow 는 기저 계층을 100% 순혈 Zenoh 프로토콜로 채웠다. 노드 A가 노트북에 있든, 노드 B가 클라우드에 있든, 이 둘을 연결해 놓으면 데이터를 쏘는 순간 전 지구적 라우팅 망(Zenoh Router) 이 알아서 바다 건너 컨베이어 벨트로 딜리버리해버린다.

2. 통신 비용(Zero-copy) 의 극단화

  • 로컬 연결 시: 노드 A와 B가 우연히 같은 칩셋 안에 있다면, Zenoh-Flow 는 무식하게 네트워크 소켓을 타지 않는다. 공유 메모리(Shared Memory) 의 포인터 주소 스왑만으로 기가바이트(GB) 의 데이터를 마이크로초 만에 건네준다.
  • 원격 연결 시: 노드 A방과 B방이 다른 망에 있다면, 그때만 살얼음처럼 얇은 Zenoh 바이너리 터널(TCP/QUIC) 을 개통하여 던진다.

아키텍트는 그저 “그래프(Graph)” 만 한 장 그려서 배포하면, 프레임워크가 네트워크의 물리적 족쇄를 알아서 쪼개버린다.

3. 마이크로서비스 아키텍처(MSA) 및 액터 모델(Actor Model)과의 패러다임 비교

“이거 그냥 백엔드에서 쓰던 마이크로서비스(MSA) 를 로봇에 붙인 거 아닌가?”
비슷하지만, 데이터 처리 밀도(Density) 가 완전히 다르다.

3.0.1 [인스펙션] 소프트웨어 아키텍처 삼국지

1. 마이크로서비스 (gRPC / REST 기반)

  • 철학: “네가 필요할 때 내 API를 호출(Request) 해라. 그럼 내가 계산해서(Response) 주마.”
  • 한계: 로봇의 초당 60 프레임 카메라 데이터를 마이크로서비스에 HTTP 로 밀어 넣으면 서버가 숨이 막혀 터진다. 서비스 간의 결합도가 너무 무겁다.

2. 액터 모델 (Erlang / Akka)

  • 철학: “서로에게 우편함(Mailbox) 을 만들어 놓고, 상태값(메시지) 을 던져주자.”
  • 한계: 액터들은 상태 기계(State Machine) 를 관리하는 데 특화되어 있지, 1GB 짜리 행렬 텐서(Tensor) 파이프라인을 쪼개서 병렬 계산하는 데스크톱 공장장으로는 부족하다.

3. 데이터플로우 (Zenoh-Flow)

  • 철학: “우편함이고 API 고 다 필요 없고, 그냥 내 앞의 파이프(Link) 에서 물이 쏟아지면 파싱해서 다음 파이프로 밀어라.”
  • 결론: 마이크로서비스가 “은행원” 이라면, 데이터플로우는 “용광로(Furnace) 파이프 연쇄” 다. 인공지능 추론(AI Inference), 신호 처리(Signal Processing) 등 무자비한 컴퓨팅 파워가 쉬지 않고 흐르는 곳에서는 오직 데이터플로우 만이 물리 엔진의 한계선 위에서 생존할 수 있다.

4. 기존 스트림 처리 엔진(Apache Kafka, Flink 등)과의 아키텍처 차이점

“빅데이터 팀에서는 스파크(Spark) 랑 카프카(Kafka) 로 스트리밍 파이프라인 다 짜던데, 그냥 그거 쓰면 안 됩니까?”

4.0.1 [Runbook] 데이터센터 도구 vs 엣지(Edge) 무기의 스펙 분기점

만약 당신의 로봇이나 드론(Edge) 에 카프카를 깔려 한다면 사표를 써야 한다.

1. 덩치의 재앙 (JVM vs Rust)

  • Kafka / Flink: 자바(Java / Scala) 가상 머신(JVM) 이 멱살을 잡고 있다. 기동하는 데만 RAM 1GB 를 거뜬히 먹으며, 가비지 컬렉터(GC) 가 도는 순간 실시간 제어 스레드는 마비된다.
  • Zenoh-Flow: 순혈 Rust 와 C 로 빌드되어, 초소형 마이크로컨트롤러부터 자율주행 보드까지 10MB 대 램만 먹고 네이티브 속도로 폭주한다.

2. 브로커(Broker) 의 존폐 유무

  • Kafka: 모든 데이터는 무조건 거대한 중앙 “토픽 클러스터(카프카 서버)” 를 찍고 디스크에 기록(Log Append) 된 후 다른 노동자에게 전달된다.
  • Zenoh-Flow: 브로커 따위는 없다. 로봇의 눈(센서)에서 나온 데이터가 로봇의 두뇌(CPU) 로 직행하는 다이렉트 컨베이어 벨트(Peer-to-Peer) 다.

아키텍트의 타협안:
Zenoh-Flow 는 공장 현장(Edge) 에서 초당 수백 회의 기동-회피 계산을 끝내버리는 전술 타격 무기다. 그리고 그 계산의 최종 “통계 결과물” 1 바이트만을 Kafka 로 쏴서 데이터센터의 거대 스토리지에 붓는 것이 유일한 상생 아키텍처다.

5. 클라우드에서 에지(Edge-to-Cloud Continuum)까지의 주요 활용 분야

그렇다면 이 괴물 같은 분산 파이프라인(Zenoh-Flow) 은 대관절 어느 전장에 투입되어야 하는가?

5.0.1 [인스펙션] 분산 다중 연산(Federated Compute) 의 타겟 영점

이 툴은 단일 컴퓨터 안에서 돌릴 거면 쓸 이유가 없다. 여러 기계의 자원을 빨아먹는 “하이브리드 파이프라인” 에 목숨을 걸어라.

1. 분할 인공지능 (Split AI Inference)

  • 로봇(프론트) 에서는 가벼운 얼굴 인식만 하고, 크기가 큰 특징점(Feature Array) 만 뽑아 파이프에 던진다.
  • 파이프 끝에 연결된 지하실의 거대한 RTX 서버 (백엔드) 가 그걸 받아서 범죄자 데이터베이스와 매칭 후 결과를 반환한다.
  • 이 모든 과정이 소스 코드 수준에서 [Node A] -> [Node B] 로 선언되어, 네트워크 레이어의 존재감이 희석된다.

2. 자율주행 차량 군집의 센서 퓨전 (Sensor Fusion)

  • 선두 차량이 도로 상황 포인트 클라우드를 추출해 허공에 쏜다.
  • 후위 차량의 Zenoh-Flow 노드가 이 데이터를 낚아채서, 내 차의 센서값과 융합(Fusion) 하는 필터(Node) 로 밀어 넣는다.
  • 군집 전체가 하나의 거대한 컴퓨팅 보드가 된 것처럼 작동한다.

3. 공장 지능형 비전(Vision) 불량 검출기

  • 1라인부터 10라인까지 있는 100대의 카메라 데이터(Source) 가, 1대의 고성능 GPU 판독 랙 서버(Operator) 로 쫙- 빨려 들어가, 통과/불량(Sink) 신호로 갈라져 나오는 Y자, 사다리타기 모양의 초대형 플로우(Flow) 차트를 구축할 때 진가를 발휘한다.

엔지니어는 파이프(Link) 설계 도면(YAML) 만 구워내면 된다.
