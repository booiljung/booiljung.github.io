13.5.3.2 하드웨어 가속기(GPU/NPU) 연동을 위한 C++ 오퍼레이터 작성

C++를 쓰는 궁극적인 이유는 바로 **하드웨어 가속(Hardware Acceleration)**이다. NVIDIA의 CUDA나 드론에 달린 NPU 하드웨어 I/O 통로를 열어젖히려면 시스템 가장 밑바닥 언어인 C++가 필수적이다.

[1. 비동기 I/O가 아닌 메모리 매핑 연산]
하드웨어 가속기 노드는 CPU 스레드가 노는 꼴을 보지 못한다.
데이터가 포트로 들어오자마자, 여러분의 C++ 코드는 이 바이트 배열(Byte Array) 주소를 GPU의 VRAM으로 올려버려야(MemCpy H2D) 한다.

void operator()() override {
    auto msg = port_in.recv();
    
    // 호스트의 메모리를 GPU로 폭격
    cudaMemcpy(d_buffer, msg->data, size, cudaMemcpyHostToDevice);
    
    // 쿠다 커널(Thread 1024 격자) 발사!
    my_cuda_kernel<<<blocks, threads>>>(d_buffer, d_result);
    cudaDeviceSynchronize();
    
    // 결과를 다시 CPU로 긁어온 뒤 Zenoh 버퍼 구조체로 패키징
    cudaMemcpy(h_result, d_result, size, cudaMemcpyDeviceToHost);
    port_out.send(h_result);
}

[2. 병목 분리: H2D와 D2H의 비동기 파이프라인 분리]
만약 GPU 연산이 30ms가 걸리는데 하나의 노드 안에서 cudaDeviceSynchronize 로 멍때리고 있으면, 이 파이프라인의 최대 FPS는 33으로 제한된다.
초극강의 튜닝을 원한다면 C++ 노드를 두 개로 분리하라.

  • Node 1 (H2D Node): 데이터가 오면 GPU(VRAM)에 밀어 넣고 “커널 기동” 명령만 비동기로 날린 채 곧바로 리턴한다. (Zenoh-Flow 스레드 점유율 복구)
  • Node 2 (D2H Node): CUDA Event나 큐를 모니터링하다가 GPU 연산이 떨어지면 그제야 VRAM을 긁어 포트 밖으로 쏜다.

Zenoh-Flow 런타임에 이런 “CUDA-Aware(쿠다 인지)” 구조를 매핑하는 순간, 소프트웨어의 극한인 에지 AI 병렬화가 완성된다!
