13.5.3.1 C API 및 기존 레거시 시스템 통합 가이드

CMakeLists.txt 로 떡칠 된 기존의 무거운 수학 라이브러리를 건드리지 말고, 빈 C 파일을 하나 더 파서 “파이프라인 통신용 어댑터(Adapter)” 를 만들어라.

0.1 [Runbook] 레거시 바인딩(Binding) 외과 수술

1. Zenoh-Flow C 헤더 인젝션

#include <zenoh-flow.h>

// 당신 회사의 20년 된 고전 레거시 함수라고 가정
extern void legacy_math_algo(const uint8_t* in, uint8_t* out);

// Zenoh-Flow 가 콜해 줄 프로세스 델리게이트
void my_operator_process(void* state, const struct zf_message* inputs, struct zf_message* outputs) {
    // 1. 인풋 포트에서 데이터 빼오기
    zf_bytes_t in_bytes = zf_message_get_bytes(inputs, "legacy_in");
    
    // 2. 아웃풋 버퍼 할당
    uint8_t out_buf[1024]; 
    
    // 3. 레거시 코드 호출 (여기서 블록킹이 일어난다!)
    legacy_math_algo(in_bytes.data, out_buf);
    
    // 4. 다음 파이프로 전송
    zf_message_send(outputs, "legacy_out", out_buf, 1024);
}

2. 메모리 라이프사이클 붕괴 방지

  • C 언어 통합의 가장 큰 위험은 “데몬 본체(Rust)” 와 “플러그인(C)” 간의 메모리 주도권 싸움이다.
  • 당신이 C 코드 안에서 malloc() 한 포인터를 zf_message_send 에 실어서 보낸 뒤 당신이 직접 free() 해버리면, 저쪽 파이프에서 쓰레기값을 읽다가 Rust 데몬 전체가 세그폴트(Segfault) 로 폭파된다.
  • 아키텍트 룰: Zenoh API 에 데이터를 넘겼다면(Send), 그 데이터를 해제할 권한(Drop) 도 프레임워크에 위임해라. C 언어의 수동 메모리 관리에 얽매이는 순간 분산 시스템은 무너진다.
