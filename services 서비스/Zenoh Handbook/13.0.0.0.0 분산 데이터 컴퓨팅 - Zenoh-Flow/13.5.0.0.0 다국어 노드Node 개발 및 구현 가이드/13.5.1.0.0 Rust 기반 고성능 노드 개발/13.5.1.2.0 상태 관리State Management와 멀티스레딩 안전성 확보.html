13.5.1.2 상태 관리(State Management)와 멀티스레딩 안전성 확보

카메라의 “어제 프레임” 과 “오늘 프레임” 의 차이를 추종하는 델타-필터를 만든다고 가정하자. 오퍼레이터는 1초 전의 사진을 메모리 뱃속에 움켜쥐고(State) 있어야 한다.

0.1 [인스펙션] 메모리 결함의 근원적 축출 구조

Rust 토키오(Tokio) 스케줄러 위에서 당신의 코드는 어떤 스레드로 불려갈지 모른다. 그냥 변수를 박아두면 Rust 컴파일러가 “Thread-Safe 하지 않아!” 라며 빌드를 뱉어버린다.

1. Mutex의 오남용 금지

  • 뱃속에 상태(State) 를 저장하기 위해 std::sync::Mutex 를 쓰면, Zenoh-Flow 의 거대한 파이프 시스템 전체 성능이 당신의 노드 하나 때문에 블러킹(Blocking) 에 걸려 마비될 수도 있다.
  • 아키텍트 룰: 비동기(Async) 런타임 내에서는 반드시 tokio::sync::Mutex 나 RwLock (읽기 비대칭 락), 또는 동시성 채널(Channel) 기반의 액터 구조를 차용하여 상태를 지켜라.

2. 불변성(Immutability) 의 강제
Zenoh-Flow 개발진은 개발자들을 100% 믿지 못해, 오퍼레이터의 계산(process) 함수 파라미터에 들어오는 자기 자신(&self) 을 가변(&mut self) 으로 제공하지 않는다.
즉, 내부 변수를 바꾸고 싶으면 당신이 “내부 가변성(Interior Mutability)” 기법을 써서 알아서 동시성을 통제하라는 거룩한 족쇄다. 이것 덕분에 수만 대의 로봇이 Zenoh-Flow 환경 하에서 메모리 누수 없이 동작한다.
