13.1.1 데이터플로우 프로그래밍(Dataflow Programming) 패러다임 이해

로봇의 눈(카메라)이 영상을 찍어 자율주행 알고리즘(추론) 으로 보내고, 또 모터 제어기로 보내는 긴 파이프라인(Pipeline)을 상상해 보라.

0.1 [인스펙션] 제어 흐름(Control Flow) 의 종말

1. 전통적인 명령형 프로그래밍의 한계

  • “카메라 스레드야, 영상을 찍어라 -> 찍었니? -> 그럼 추론 함수를 호출해라 -> 그 리턴값을 모터 함수로 줘라”
  • 이 방식은 “프로그램의 논리(Logic)” 가 데이터를 직접 끌고 다닌다. 데이터가 커지고 컴퓨터가 2대, 3대로 늘어나면 로직 자체가 스파게티처럼 엉키고 소켓(Socket) 코딩을 하다가 무너진다.

2. 데이터플로우 (Data-driven) 의 지배

  • 데이터플로우에서 “메인 프로그램(main 함수)” 따위는 존재하지 않는다.
  • 오직 각자의 방진복을 입고 컨베이어 벨트(Belt) 앞에 서 있는 독립적인 노동자(Node) 들만 존재할 뿐이다.
  • 노드 A는 눈앞에 카메라 데이터가 도착하면 즉시 가공해서 옆 벨트로 던진다. 노드 B는 그 데이터가 자기 벨트에 떨어질 때까지 잠을 잔다.
  • 데이터가 흐르는(Flow) 현상 그 자체가 프로그램의 실행(Execution) 을 강제(Run) 하는 궁극의 물류센터 아키텍처다.
