13.2 Zenoh-Flow 코어 아키텍처 및 구성 요소

파이프라인을 설계하려면 배관공학의 기초 부품들을 먼저 눈에 익혀야 한다.

Zenoh-Flow 의 세상은 매우 단순하다. 물을 끌어오는 펌프(Source), 물에 색소를 타는 정수기(Operator), 그리고 물을 모아두는 저수지(Sink). 이 3개의 블록만 있으면 로봇의 자율주행 알고리즘부터 글로벌 스케일의 스마트시티 관제망까지 모든 것을 직관적으로 조립해 낼 수 있다.

1. Zenoh-Flow 전역 및 로컬 아키텍처 모델

Zenoh-Flow 의 아키텍처는 “현미경으로 보느냐, 망원경으로 보느냐” 에 따라 구조가 완전히 다르게 작동한다.

1.0.1 [인스펙션] 하이브리드 토폴로지의 이중성

1. 로컬 배관망 (단일 머신 내에서의 폭주)
만약 Source, Operator, Sink 가 “기초 C 컴파일 랩” 의 하나의 PC 나 라즈베리 파이 화면 안에서 돈다면, Zenoh-Flow 런타임은 네트워크 계층(TCP 소켓)을 아예 깨우지 않는다. 대신 메모리 스레드 간의 락-프리 큐(Lock-free Queue) 와 링 버퍼(Ring Buffer) 만으로 데이터를 초당 수만 개 단위로 넘긴다. 외부에서 스니핑조차 불가능한 0 오버헤드의 블랙박스 화가 진행된다.

2. 전역 배관망 (클러스터 단위의 스케일 아웃)
당신이 YAML 파이프라인 파일에서 node: Operator 의 위치 속성(mapping) 을 AWS 서버로 바꾸는 순간 마법이 일어난다.
로컬에서 잘 돌던 컨베이어 벨트에 투명한 게이트웨이(Zenoh Router) 가 생성되고, 데이터는 UDP 멀티캐스트 혹은 TCP 를 타고 바다를 건너가 AWS 의 수신 컨베이어 벨트에 안착한다.
개발자가 소켓 프로그래밍, 직렬화, 커넥션 풀을 1줄도 짤 필요 없이, 그저 프레임워크가 아키텍처 크기에 맞춰 “가장 최적화된 통신 모드” 로 자동 기어 변속을 해준다.

2. 소스(Source) - 외부 환경 데이터 유입 노드의 역할과 구조

모든 데이터의 강은 여기서 발원한다. 소스는 파이프라인 안쪽에 있는 노드이지만, 그 눈은 바깥세상(External World)을 향해 있는 가장 중요한 외교관이다.

2.0.1 [인스펙션] 발원지(Spring) 설계 원칙

1. 폴링(Polling) 대 이벤트-드리븐 (Event-driven)

  • 소스 노드는 크게 두 가지 양태로 동작한다.
  • 타이머를 시곗바늘처럼 달아놓고 1초마다 카메라 드라이버 베어메탈 계층에서 프레임을 긁어오는 주기적 소스(Periodic Source).
  • MQTT 나 외부 디버거 핑이 떨어질 때만 인터럽트처럼 깨어나 벨트에 데이터를 올리는 이벤트 소스(Reactive Source).

2. 백프레셔(Backpressure) 와의 싸움
만들어낸 소스 노드가 초당 100장의 이미지를 컨베이어 벨트에 밀어 넣는데, 다음 처리 노드(Operator) 가 1지능당 10장밖에 처리 못한다면? 벨트는 넘치고 터진다.
훌륭한 Zenoh-Flow 소스 엔진은 뒤에서 “잠깐, 큐 꽉 찼어!” 라는 신호(Backpressure) 가 오면 알아서 데이터를 생성 단계에서 바닥에 버리는 드롭 로직(Drop Strategy) 을 반드시 프로그래밍해야 한다.

3. 오퍼레이터(Operator) - 데이터 변환, 필터링, 연산 노드의 핵심 로직

이 녀석이 바로 파이프라인의 두뇌, 즉 CPU 코어를 활활 태우는 포식자다. 소스에서 들어온 데이터를 가공하여 또 다른 파이프로 내뱉는다.

3.0.1 [Runbook] 다중 입력/출력 (MIMO) 용광로 설계

오퍼레이터는 단순히 1:1 파이프가 아니다.

1. 조인 (Join / Fusion)
자율주행의 꽃이다. 오퍼레이터는 파이프라인 입력 단자(Input Port) 를 2개, 3개 뚫어놓을 수 있다.

  • 포트 1: 카메라 이미지 도착!
  • 포트 2: 라이다 3D 데이터 도착!
    두 개가 같은시간(Timestamp) 에 맺히는 순간 오퍼레이터가 이빨을 맞물려 “센서 퓨전 결과” 라는 1개의 출력 포크(Output Port) 로 뿜어낸다.

2. 상태 유지 (Stateful) vs 상태 없음 (Stateless)

  • 안면 추출 필터처럼 들어온 사진 1장에 박스 하나만 칙 긋고 버리는 건 무상태(Stateless) 다. 언제 죽고 다시 살아나도 문제가 없다.
  • 하지만 “최근 10초간의 온도 평균” 을 내는 필터(Windowing) 라면? 이 녀석은 속 안에 메모리 지갑을 쥐고 있어야 하는 상태 의존적(Stateful) 오퍼레이터다. Zenoh-Flow 에서는 오퍼레이터 코드 안에서 로컬 변수를 살려둘 수 있도록 라이프사이클 컨텍스트(Context) 를 철저히 보호해 준다.

4. 싱크(Sink) - 외부 시스템 및 스토리지로의 데이터 출력 노드

데이터의 무덤이자 종착역. 이 컨베이어 벨트가 내뱉는 결과물이 외부 세상으로 버려지는 하수구다.

4.0.1 [인스펙션] 엔드포인트 터미널의 역할

싱크 노드는 출구 파이프(Output Port) 가 아예 존재하지 않는 유일한 노드 계층이다.

1. 액추에이터 제어 (로보틱스)
추론된 자율주행 회피각 데이터를 받아, 모터 드라이버의 CAN 통신으로 밀어 넣고 스레드는 종료된다.

2. 영구 보존 스토리지 덤프 (클라우드)
수만 개의 필터링을 거친 순수한 인메모리(RAM) 데이터가 최종 싱크 노드에서 비로소 파일 시스템 구조체에 씌워져 MySQL, InfluxDB 또는 AWS S3 버킷 에 하드 디스크의 영역으로 응고(Persist) 된다.

3. 비동기 블로킹 방어
데이터베이스에 값을 쓰는 행위는 네트워크 I/O 병목을 유발한다. 싱크 작업이 지연되면 앞선 파이프 전체가 막혀 로봇이 뻗어버릴 수 있다. 따라서 훌륭한 싱크 노드는 반드시 비동기 워커 풀(Async Workers) 에 던져놓고 즉각 다음 데이터를 받아야 한다.

5. 링크(Link) - 노드 간의 통신 채널과 Zenoh 라우팅의 결합

Source 와 Operator 의 손과 손을 잡게 만드는 접착제다.
수동 코딩을 할 수 없고, 오로지 마스터 설계도 (YAML) 에 의해서만 선언(Declare) 되는 영역이다.

5.0.1 [Runbook] 데이터의 투명한 혈관 투영

1. 추상화의 정점
당신은 Python 이나 Rust 로 오퍼레이터를 짤 때 아무리 눈을 씻고 찾아봐도 이 데이터가 TCP 로 왔는지 통신 메모리로 왔는지 알 수 없을 거다. 링크는 물리망을 완벽히 은닉(Abstraction) 한다.

2. Zenoh Key Expression 과의 동기화
Zenoh-Flow 가 내부적으로 통신할 때 그냥 파이프를 잇는 게 아니다.
NodeA -> NodeB 로 링크를 묶었을 뿐인데, Zenoh 런타임은 백그라운드에서 이 파이프에 임의의 난수형 Zenoh 토픽 이름(UUID Key) 을 파버리고 한쪽은 임시 퍼블리셔, 다른 한쪽은 임시 구독자로 맺어버린다.
기존 통신망을 지저분하게 오염시키지 않으면서도, Zenoh 코어 라우팅 엔진의 가속 효과를 100% 빼먹는 천재적인 기만전술이다.

6. 페이로드 처리: 커스텀 데이터 타입(Custom Data Types)과 직렬화(Serialization)

파이프 A(Python 카메라 로직) 에서 뿜어낸 “영상 메모리 포인터” 를 어떻게 파이프 B(Rust 딥러닝 추론 로직) 에 구겨 넣을 것인가?

6.0.1 [인스펙션] 이종 언어 간의 혈액 수혈 (Transfusion)

만약 같은 메모리 레이아웃 내(동일 PC, 동일 언어 환경) 이라면 데이터는 C 포인터 형식의 메모리 주소값 하나만 링크를 넘나들며(Zero-Copy) 끝난다. 하지만 언어가 다르거나 클라우드의 장벽(네트워크) 을 넘는 순간 무자비한 이진화(Binary Serialization) 작업이 들어간다.

1. 커스텀 스키마 방어 정책

  • Zenoh-Flow 는 데이터 타입을 당신이 마음대로 정하게 내버려두지만, 언어 독립성을 확보하기 위해 내부적으로 Protobuf, FlatBuffers, 혹은 Bincode 같은 “바이트 직렬화” 컨트랙트를 마련해야 함을 경고한다.
  • 즉 파이프라인 단자대에 데이터를 던질 때 my_struct.serialize() 로 바이트 버퍼 형태로 밀어 넣고, 받는 측에서 역으로 풀게 만드는 것이 안전한 네트워크 횡단 규약이다.

2. 투명성(Opaque) 오버헤드 주의
바이트 뭉치를 넘기는 건 가장 빠르다. 하지만 Python 에서 Pickle 로 구운 바이트를 Rust 필터가 푸는 짓은 불가능하다. 언어 독립적인 표준 페이로드 구조체를 설계하는 것이 시스템 아키텍트의 피할 수 없는 철칙이다.
