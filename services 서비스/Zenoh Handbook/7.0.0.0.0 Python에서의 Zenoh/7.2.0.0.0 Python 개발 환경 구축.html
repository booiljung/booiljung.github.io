7.2 Python 개발 환경 구축

파이썬 개발의 시작은 거대한 시스템 아키텍처 구상보다 가벼운 pip install 한 줄에서 출발한다. 하지만 Zenoh는 순수 파이썬 라이브러리가 아니라 컴파일된 Rust 네이티브 코어를 품고 있는 강력한 미들웨어다. 따라서 OS 아키텍처 제원을 무시하고 무작정 패키지를 당기면 기괴한 DLL/SO 링크 에러를 마주하게 된다.

이 챕터에서는 일반적인 클라우드 프로덕션 서버부터, ARM64가 탑재된 모바일 에지(Edge) 로봇 보드에 이르기까지 Zenoh를 가장 안정적으로 안착시키는 설치 런북(Runbook)을 다룬다. 기본 패키지 설치부터 극한의 최적화를 위한 소스코드 직결(Build-from-source) 방법, 그리고 설치 성공을 알리는 Hello World 타점 통과 시뮬레이션까지 전개한다.

1. 시스템 요구 사항 및 Python 버전 호환성 확인

Zenoh Python 바인딩은 파이썬 인터프리터 안에 침투하는 C-ABI 확장이기 때문에, 호환 가능한 파이썬 버전의 커트라인(Cut-line)이 매우 엄격하다.

1.0.1 [Runbook] 프로덕션 서버 기초 감정(Survey)

1. Python 생태계 마지노선
Zenoh는 더 이상 수명이 끝난 Python 3.6 이하의 언데드 시스템을 지원하지 않는다.
최소 Python 3.8 이상이 강제된다. 성능을 위해서는 dict 연산이 극적으로 빨라진 Python 3.11 을 강력히 추천한다.

python3 --version
## Expected: Python 3.8.x 이상

2. OS 플랫폼 및 CPU 아키텍처

  • x86_64 기반 리눅스/윈도우/맥: 파이프라인에서 이미 빌드된 Wheel (.whl) 파일(Binary)이 통째로 내려오므로 컴파일러(GCC)가 없어도 즉시 설치된다.
  • ARM64 (Jetson Nano, Raspberry Pi 4/5): 최근 버전에선 ARM64 Wheel 도 제공하기 시작했다. 만약 구형 32비트 라즈비안을 쓰고 있다면 Wheel 이 없어서 자동으로 “Source 빌드“를 시도하다가 Rust 컴파일러가 없어 죽어버리는 에러가 발생한다. 반드시 64Bit OS로 마이그레이션(Migration)하라.

3. 가상 환경(Virtualenv) 격리
AI 생태계에서는 라이브러리 충돌이 일상이다. zenoh 를 전역(global)에 설치하는 건 데브옵스 철학에 위배된다.

python3 -m venv zenoh_env
source zenoh_env/bin/activate
## 이 격리된 환경에서만 Zenoh와 NumPy 등을 섞어 쓴다.

2. PyPI를 통한 기본 패키지 설치 (pip install eclipse-zenoh)

환경 격리가 끝났다면 이 한 줄로 끝난다. 가장 안전하고 권장되는 방법이다.

2.0.1 [Runbook] 표준 인스톨 절차

## 최신 안정화 버전(Stable API) 설치
pip install eclipse-zenoh

만약 터미널에서 Building wheel for eclipse-zenoh (setup.py) ... \ 같은 메시지가 뜨면서 5분 이상 멈춰 있다면 위험 신호다.
이는 당신의 OS나 Python 버전에 맞는 미리 구워진 바이너리(whl) 릴리즈가 없어서, Pip가 소스코드를 처음부터 Rust로 컴파일하려고 시도하는 중압감 상태 임을 뜻한다. 리눅스 보드에 Rust/Cargo 가 설치되어 있지 않다면 곧 빨간 글씨와 함께 폭발하게 될 것이다.
(이 현상이 발생하면 OS 커널 64비트 여부와 Python 버전이 3.8+ 인지 재점검하라.)

설치가 0.5초 만에 완료되었다면 성공이다.

## 파이썬 콘솔에서 타임 스탬프 및 버전 출력 스모크 테스트
import zenoh
print(zenoh.__version__)

버전 문자열(예: 1.0.0)이 깨끗하게 떨어지면, 하부 Rust 코어가 성공적으로 파이썬 C-API 에 안착한 것이다.

3. 소스 코드 기반 빌드 및 커스텀 설치 방법

“내 로봇은 RISC-V 칩셋을 씁니다.” 혹은 “Zenoh의 최신 Master 브랜치에 있는 실험적 기능을 땡겨 오고 싶습니다.”
이런 상황에서 pip install 은 통하지 않는다. 당신이 직접 대장장이가 되어 Rust 코어를 파이썬 바인딩으로 녹여내야(Build) 한다.

3.0.1 [Runbook] 하드코어 컴파일 (From Source)

이 파이프라인의 필수 무기는 Rust/Cargo 와 파이썬의 빌더인 maturin 이다.

1. 무기고(Dependencies) 세팅

pip install setuptools maturin

2. 소스코드 직접 추출 및 정밀 연성

## 1. Eclipse 재단의 Zenoh Python 레포지토리를 복제(Clone)
git clone https://github.com/eclipse-zenoh/zenoh-python.git
cd zenoh-python

## (선택) 최신버전 말고 특정 구버전 브랜치로 체크아웃
## git checkout 0.10.x 

## 2. Maturin을 돌려 Rust 코드를 컴파일하고 파이썬 환경에 꽂아넣는다 (Release 옵션 필수!)
maturin develop --release

--release 옵션을 주지 않으면 디버그 모드로 컴파일되어 1/10 속도로 느려터진 괴물이 탄생하니 절대 잊지 마라.
컴파일이 끝나면 pip install 로 깐 것과 100% 동일하게 import zenoh 가 가능해지며, 벤치마크 테스트에서 극한의 성능을 뽐낼 수 있게 된다.

4. 개발 환경 테스트: Hello Zenoh (퍼블리셔와 서브스크라이버 작성)

환경 구축을 마쳤으니, 이 거대한 파이프라인이 정상적으로 개통되었는지 타격 테스트를 가한다.
C++ 에서는 이 테스트를 위해 CMake 를 짜고 빌드 폴더를 만들어야 했지만, 파이썬에서는 터미널 창(Shell) 2개만 띄우면 10초 안에 끝난다.

4.0.1 [Runbook] 망 개통 파괴 시뮬레이션

가장 짧은 단 10줄의 코드로 통신망의 생사를 가늠해본다.

1. [수신탑] Subscriber 구축 (터미널 A)
파일 생성: sub_test.py

import zenoh
import time

## 단 한 줄로 기본 네트워크 세션을 가동 (라우터와 붙거나 P2P 연결 수립)
session = zenoh.open()

print("망(Network) 수신 안테나 가동. [demo/hello] 대기 중...")

## 람다 콜백 (패킷이 들어오는 즉시 실행됨)
def listener(sample):
    print(f">> 수신 폭격! 출처: {sample.key_expr} / 본문: {sample.payload.decode('utf-8')}")

## 구독 파이프 개통
sub = session.declare_subscriber("demo/hello", listener)

## 파이썬 메인 스레드 유지 (백그라운드에서 C/Rust 코어가 통신을 전담함)
try:
    while True:
        time.sleep(1)
except KeyboardInterrupt:
    print("\n수신 안테나 종료")

python3 sub_test.py 를 입력하고 터미널을 대기 상태로 둔다.

2. [발사대] Publisher 타격 (터미널 B)
파일 생성: pub_test.py

import zenoh
import time

session = zenoh.open()

print("목표물 조준: demo/hello")
pub = session.declare_publisher("demo/hello")

## 5발 연사
for i in range(5):
    msg = f"Zenoh Python API Test - Strike {i+1}"
    pub.put(msg)
    print(f"발사: {msg}")
    time.sleep(0.5)

print("작전 완료.")

python3 pub_test.py 를 치는 순간, 단 1의 지연(Delay)도 없이 즉각 터미널 A의 화면에 텍스트가 후두둑 떨어져야 정상이다. 만약 작동하지 않는다면 방화벽(Port 7447 멀티캐스트)이 막혀있는지 데브옵스 환경을 점검해야 한다.
