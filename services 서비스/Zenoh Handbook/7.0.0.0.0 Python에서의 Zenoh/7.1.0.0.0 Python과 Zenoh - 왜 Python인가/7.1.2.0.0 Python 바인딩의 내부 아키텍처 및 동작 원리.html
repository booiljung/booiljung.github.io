7.1.2 Python 바인딩의 내부 아키텍처 및 동작 원리

당신이 import zenoh 를 입력했을 때, Python 인터프리터(CPython)는 순수 파이썬으로 작성된 네트워크 코드를 실행하는 것이 아니다.
Zenoh Python API는 철저하게 껍데기(Wrapper)일 뿐, 그 밑바닥에는 시스템 프로그래밍의 끝판왕인 Rust 라이브러리(.so / .dll / .dylib) 가 심장처럼 고동치고 있다.

0.1 [아키텍처 인스펙션] PyO3 기반의 C-ABI 연동 메커니즘

Zenoh 재단은 Python 바인딩을 만들기 위해 초기엔 C-FFI 방식을 시도했으나, 현재는 PyO3 라는 Rust-to-Python 라이브러리에 완전히 정착했다.

동작 레이어 구조 (Top-Down):

  1. Python User Layer: 사용자가 zenoh.open() 을 호출한다.
  2. PyO3 Binding Layer: Python 런타임의 동적 타입(Dynamic Typing) 인자를 받아서 Rust의 정적 타입(Static) C-API 구조체로 번역(Marshalling)한다.
  3. Zenoh Rust Core (zenoh-rs): 실제 소켓을 열고, B-Tree 라우팅 테이블을 뒤지고, 비동기(Tokio) 이벤트 루프를 돌리는 진짜 주역이다.

GIL (Global Interpreter Lock) 해방 전술
파이썬 언어의 최악의 단점인 GIL(멀티코어를 1개 스레드만 쓰게 만드는 자물쇠)을 Zenoh는 어떻게 우회할까?
네트워크 I/O를 대기하거나 패킷을 직렬화/라우팅하는 무거운 작업 내내, Rust 바인딩 레이어는 가차없이 Python GIL을 스스로 해제(Release) 해버린다!
즉, 데이터를 다운로드받고 소켓을 지키는 99%의 시간 동안 당신의 파이썬 메인 스크립트는 다른 AI 모델 텐서 연산 등에 CPU 100%를 때려부을 수 있다.
콜백(Callback)을 수신해서 파이썬 함수를 때릴(Trigger) 때에만 찰나의 순간 동안 GIL을 다시 획득하여 코드를 실행한다.
