9.5.2 Go 언어 환경의 성능 최적화 기법

Go 언어 환경의 제일 큰 적은 “CGO 장벽” 과 “가비지 컬렉터(GC)” 다.
Zenoh 엔진 자체는 0.01ms 만에 처리를 끝마쳐도, 이 결과물이 CGO 런타임을 타고 메모리를 할당(malloc -> Go 힙)받는 순간 속도가 반토막 나버린다. 이 장은 서버(Server) 아키텍트가 숙지해야 할 Go 엔진 회피 기동 전술이다.

1. CGO 호출 오버헤드 측정 및 최소화 설계 원칙

CGO 는 공짜가 아니다. Go 메모리 포인터 세계와 C 포인터 물리 세계가 한번 교차할 때마다 runtime.cgocall 이라는 어마어마한 컨텍스트 스위칭 비용을 지불해야 한다. (약 50나노초~수 마이크로초).

1.0.1 [Runbook] CGO 횟수 압축 전술 (Batching)

1초에 10만 번 put 을 때리는 로직이 필요하다고 치자.

// ❌ 안티 패턴: CGO 방어벽을 10만 번 때림 (서버 CPU 폭발)
for i := 0; i < 100000; i++ {
    pub.Put(tinyPayload[i], nil) // 이 안에서 CGO 콜이 각각 발생!
}

Go 서버를 작성할 때 Zenoh 로 통신을 한다면, 데이터 생성 부하가 CGO 호출 오버헤드를 추월하는지 반드시 체크해야 한다. 만약 극한의 속도가 필요하다면, 여러 개의 텔레메트리 데이터를 하나의 큰 바이트 덩어리(Batch Array)로 뭉친 뒤 한 방에 Push 하는 식으로 CGO 횟수를 극적으로 줄여내야 한다.
“통신 횟수” 자체를 줄이는 것, 이것이 CGO 기반 라이브러리인 zenoh-go 를 다루는 첫 번째 원칙이다.

2. 메모리 풀(sync.Pool)을 이용한 메시지 객체 재사용 전략

고빈도 메시지(1초에 10만 개 송수신) 환경에서, 무심코 적은 payload := make([]byte, 1024) 코드 한 줄은 끔찍한 재앙을 불러온다. 1초에 10만 개의 쓰레기 껍데기(Garbage)가 생성되며, 10초만 지나도 Go의 GC 데몬이 깨어나 서버를 강제 연산 중지(Stop The World) 시킨 뒤 쓰레기통을 비우느라 트래픽을 먹통으로 만든다.

2.0.1 [Runbook] 영구적인 버퍼 재활용 창고(Pool) 전술

package main

import (
	"fmt"
	"github.com/eclipse-zenoh/zenoh-go"
	"sync"
)

// 1. 단 한 번만! 1KB 짜리 빈 박스를 무한정 뿜어내는 공장(Pool)을 설계한다.
var payloadPool = sync.Pool{
	New: func() interface{} {
		return make([]byte, 1024)
	},
}

func startOptimizedTelemetry(pub *zenoh.Publisher) {
	for i := 0; i < 100000; i++ {
		// 2. 새 메모리를 만들지(make) 말고, 공장에서 헌 바이트 배열을 "빌려" 온다.
		buffer := payloadPool.Get().([]byte)

		// 3. 실제 데이터 조작 (예: 앞부분에만 데이터를 덮어씌움)
		copy(buffer, []byte("Hello Robot!"))

		// 4. 발송 (필요한 만큼의 길이만 슬라이싱해서 쏜다)
		pub.Put(buffer[:12], nil)

		// 5. [핵심] 다 쓴 바이트 배열을 다시 공장에 "반납" 한다!
		// 이거 하나로 이 애플리케이션의 메모리 할당량은 10만 번이 아니라 '0번'에 수렴한다.
		payloadPool.Put(buffer)
	}

	fmt.Println("🚀 10만방 락프리/GC프리 폭격 시뮬레이션 완료")
}

이 패턴을 도입하면 pprof 로 메모리 측정을 했을 때, Heap 할당 그래프가 상향 곡선을 그리는 대신 바닥을 기어가게 될 것이다.

3. pprof를 활용한 zenoh-go 애플리케이션 프로파일링 분석

“어디서 느려지는지 짐작하지 마라. 측정해라.”
Go 엔진이 제공하는 우주 최고의 무기, pprof 를 백엔드 서버에 꽂아 넣는다. 무려 웹 대시보드로 CGO 와 GC 가 언제 얼마나 불탔는지를 증명해 준다.

3.0.1 [Runbook] 라이브 시스템 프로파일링 파이프라인 개통 전술

package main

import (
	"log"
	"net/http"
	_ "net/http/pprof" // 이거 한줄이면 자체적인 진단 모니터링 데몬이 기동된다!
	"github.com/eclipse-zenoh/zenoh-go"
)

func main() {
	// 1. 운영 서버의 백그라운드에 6060 포트로 프로파일러(X-Ray)를 튼다.
	go func() {
		log.Println("PPROF 진단 서버 개방: localhost:6060")
		log.Println(http.ListenAndServe("localhost:6060", nil))
	}()

	// 2. Zenoh 서버 메인 프로그램 기동
	session, _ := zenoh.Open(zenoh.DefaultConfig())
	defer session.Close()
	
	// ...(어마어마한 트래픽 통신 시뮬레이션)...
	select {}
}
