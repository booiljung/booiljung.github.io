4.85.2.3 기존 아키텍처에 대한 안하무인적 재작성(Rewrite) 요구: 레거시(Legacy) 코드가 품고 있는 수년간의 비즈니스 히스토리와 엣지 케이스 방어 로직을 이해하지 못한 채, 무작정 리팩토링(Refactoring)부터 시도하여 발생하는 치명적 호환성 단절 사고

소프트웨어 시스템이 오랜 기간 운영되며 성장할 때, 필연적으로 기존 코드베이스(Codebase)는 복잡해지고 기술 부채(Technical Debt)를 안게 된다. 이렇게 누적된 코드를 흔히 레거시 엔지니어링 시스템(Legacy Engineering System) 또는 단순히 ’레거시(Legacy)’라고 부른다. 신규 입사한 엔지니어, 특히 기술적 과신(Overestimation)에 빠지기 쉬운 주니어 엔지니어들은 종종 기존 코드의 가독성이 떨어진다거나 구조가 최신 트렌드를 따르지 않는다는 이유로 코드의 전면적인 재작성(Rewrite)을 요구하곤 한다. 그러나 시스템의 역사적 맥락과 도메인의 엣지 케이스(Edge Case)를 충분히 고려하지 않은 안하무인적인 재작성 시도는 기업의 비즈니스 연속성에 치명적인 타격을 입힐 수 있다. 본 장에서는 이러한 무분별한 재작성 요구가 야기하는 리스크와, 이를 방어하고 올바른 리팩토링(Refactoring)으로 유도하기 위한 방법론을 상세히 다룬다.

1. 레거시 코드에 내재된 숨겨진 가치 (Hidden Value of Legacy Code)

많은 주니어 엔지니어들이 레거시 코드를 ’쓸모없고 지저분한 쓰레기’로 치부하지만, 이는 소프트웨어 공학의 역사적 발전 과정을 간과한 커다란 착각이다. 미국의 유명 소프트웨어 엔지니어인 조엘 스폴스키(Joel Spolsky)는 자신의 아티클에서 “코드를 처음부터 다시 짜는 것(Things You Should Never Do)“을 소프트웨어 회사가 저지를 수 있는 최악의 전략적 실수 중 하나로 규정했다.

레거시 코드는 단순히 오래된 텍스트 파일이 아니다. 그 안에는 기업이 수년간 시장에서 겪었던 무수한 비즈니스 규칙(Business Rules), 고객의 기상천외한 요구사항, 그리고 예기치 못한 치명적 버그에 대처하기 위해 겹겹이 덧댄 방어적 프로그래밍(Defensive Programming)의 흔적이 고스란히 남아 있다.

새로운 엔지니어의 눈에는 의미론적으로 불필요해 보이거나 기이하게 작성된 조건문(if-else)들이, 사실은 특정 시점에 터졌던 중대한 장애(Failure)를 막기 위한 피땀 어린 ’핫픽스(Hotfix)’일 가능성이 크다. 따라서 겉보기에 코드가 우아하지(Elegant) 않다는 이유만으로 시스템을 처음부터 재작성하려 들 경우, 과거에 이미 해결했던 결함(Defect)들을 새로운 코드베이스에서 다시 마주하는 회귀 버그(Regression Bug)의 늪에 빠지게 된다.

2. 무작정 시도하는 재작성(Rewrite)의 치명적 리스크

레거시 시스템을 섣불리 갈아엎고 완전히 새로운 아키텍처로 넘어가려는 시도는 다음과 같은 심각한 비즈니스 및 기술적 리스크를 초래한다.

2.1 치명적 호환성 단절 (Fatal Backward Incompatibility)

레거시 코드는 기존의 클라이언트, 연동된 서드파티(Third-party) 시스템, 그리고 하위 시스템들과 깊게 결합(Coupled)되어 있다. 전면 재작성을 수행한 시스템(Greenfield System)이 기존 레거시 시스템의 입력(Input) 및 출력(Output) 명세와 정확히 100% 동일하게 동작하는 것은 실질적으로 불가능에 가깝다. 이로 인해 마이그레이션(Migration) 당일, 기존에 잘 동작하던 API 단말(Endpoint)에서 수많은 오류가 발생하며 하위 호환성 단절(Backward Incompatibility) 사태가 벌어진다.

2.2 추정 시간 및 비용의 극단적 과소평가 (Underestimation of Time and Cost)

주니어 엔지니어들은 기존 코드의 겉모습만 보고 “이 정도면 한 달이면 새로 다 짤 수 있습니다“라고 호언장담한다. 그러나 시스템을 재구축하는 과정에서 문서화되지 않은(Undocumented) 암묵적 요구사항들을 끝없이 발견하게 된다. 결국 추정(Estimation)했던 일정은 수 개월에서 수 년으로 늘어나며, 기업은 새로운 기능 개발은 멈춘 채 비용(ROI 타당성)만 갉아먹는 ’빅뱅 마이그레이션(Big Bang Migration)’의 수렁에 빠진다.

2.3 ’새로운 레거시’의 탄생

몇 년의 고생 끝에 새로운 시스템을 런칭하더라도, 그 시스템 역시 시간이 흐르면 필연적으로 변하는 비즈니스 환경에 의해 다시 덕테이프(Duct Tape)가 발라진 ’새로운 레거시’로 전락하게 된다. 결국 근본적인 아키텍처 개선 역량 없이 버리고 새로 짜는 행위만 반복하는 것은 엔지니어링 조직의 성숙도를 저해한다.

3. 리팩토링 및 마이그레이션 연착륙 전략

경영진(CEO) 및 R&D 책임자(CTO)는 엔지니어들의 전면 개편 요구에 대해 무조건적인 거부보다는, 학술적이고 점진적인 마이그레이션 방법론을 제시하여 리딩해야 한다.

3.1 스트랭글러 피그 패턴 (Strangler Fig Pattern) 활용

전면 재작성(Big-bang Rewrite)의 위험을 회피하기 위한 가장 대표적인 아키텍처 패턴은 마틴 파울러(Martin Fowler)가 제안한 ’스트랭글러 피그 패턴(Strangler Fig Pattern)’이다. 이는 특정 시스템을 한 번에 대체하는 대신, 구 시스템(Legacy)의 외곽에 API 게이트웨이(API Gateway) 층을 두고 신규 기능부터 통제된 마이크로서비스(Microservices) 환경으로 서서히 대체해 나가는 방식이다. 점진적으로 구 시스템의 기능이 새 시스템으로 이관되며, 최종적으로 구 시스템은 고사(Strangled)하여 자연스럽게 폐기된다.

        API --> NewService3[신규 마이크로서비스 B]:::new

    end

3.2 견고한 회귀 테스트(Regression Test) 스위트 구축

코드를 수정하거나 다시 작성하기 전, 반드시 기존 시스템의 동작을 보장하는 테스트 코드(Test Code)를 작성하도록 강력히 요구해야 한다. 특히, 유닛 테스트(Unit Test)뿐만 아니라 엔드 투 엔드(End-to-End, E2E) 테스트나 블랙박스 테스트(Black-box Test)를 통해 현재 레거시가 출력하는 결괏값을 철저히 캡처해야 한다. 이 기준선(Baseline)이 없다면, 새롭게 작성한 코드가 발생시킬 엣지 케이스 결함을 잡아낼 방도가 없다.

3.3 RFC (Request for Comments) 기반의 아키텍처 제안 거버넌스

엔지니어가 단순히 구두나 슬랙(Slack)으로 구형 아키텍처를 불평하지 못하게 하고, 공식적인 RFC(Request for Comments) 문서나 ADR(Architecture Decision Record)를 작성하도록 문화를 조성해야 한다. 문서에는 다음 사항들이 포함되어야 한다.

  • Why: 왜 이 재작성이 기술적으로, 그리고 비즈니스적으로 필요한가?
  • Cost: 예상되는 공수와 자원은 얼마나 되는가?
  • Risk: 기존 데이터 및 인터페이스와의 호환성은 어떻게 검증할 것인가?
  • Plan: 롤백(Rollback) 계획이나 무중단 배포(Zero-downtime Deployment) 전략은 무엇인가?

4. 결론 (Conclusion)

안하무인식의 시스템 재작성 요구는 주로 주니어 엔지니어들이 기존 시스템의 역사적 타당성을 무시하고 눈앞에 보이는 문법적, 구조적 단점에만 집착할 때 발생한다. 연구 개발 책임자(CTO)와 시니어 엔지니어링 그룹은 레거시 코드가 품고 있는 깊은 비즈니스 로직과 예외 처리의 가치를 교육해야 한다. 더 나아가 단번에 시스템을 갈아엎으려는 무모한 도전을 통제하고, 검증된 학술적 엔지니어링 기법(스트랭글러 패턴, 테스트 주도 마이그레이션 등)을 도입하여 시스템의 안정성과 기업의 런웨이(Runway)를 함께 지켜내는 리더십을 발휘해야 한다.
