4.86.3 WBS 리뷰를 활용한 엔지니어링 매니저(EM)의 방어적 설계 리더십

엔지니어링 매니저(Engineering Manager, EM)의 핵심 역할 중 하나는 주니어 엔지니어의 경험 부족으로 인해 발생하는 기술적 사각지대(Blind Spot)를 식별하고, 프로젝트의 잠재적 리스크를 사전에 차단하는 것이다. 특히 주니어 엔지니어가 자신의 역량을 과대평가하는 더닝-크루거 효과(Dunning-Kruger Effect)에 빠져 있을 때, 작업 분업 구조(Work Breakdown Structure, WBS) 리뷰는 오버 엔지니어링(Over-engineering)과 기술 부채(Technical Debt)를 예방하는 가장 효과적인 방어적 설계(Defensive Design) 리더십 도구로 작동한다.

1. 방어적 설계 리더십(Defensive Design Leadership)의 개념

방어적 설계 리더십이란 시스템이 정상적인 조건뿐만 아니라 비정상적이고 예측 불가능한 예외(Exception)와 장애 상황에서도 안정적으로 동작하도록 설계 단계를 통제하는 리더십 접근법이다.
인지적 편향(Cognitive Bias)이나 경험 부족으로 인해 낙관적 경로(Happy Path) 위주로 사고하는 주니어 엔지니어들을 이끌어, 실패 가능성을 기본 전제로 하는 방어적 프로그래밍(Defensive Programming) 패러다임을 WBS 수립 단계에서부터 내재화시키는 과정이다.

이는 단순히 코드를 꼼꼼히 리뷰하는 것(Code Review)을 넘어, 작업이 시작되기 전 단계인 WBS 리뷰에서부터 “이 컴포넌트가 실패할 경우 시스템은 어떻게 복구되는가?(How does the system recover if this component fails?)“와 같은 질문을 던져 설계의 무결성을 검증하는 행위를 의미한다.

2. WBS 리뷰를 통한 인지적 사각지대(Cognitive Blind Spots) 제거

주니어 엔지니어가 작성한 WBS는 흔히 핵심 기능 구현에만 매몰되어 비기능적 요구사항(Non-functional Requirements)이나 오류 처리(Error Handling)가 누락되는 ’비대해진 태스크(Fat Task)’의 형태를 띤다. EM은 WBS 리뷰를 통해 이러한 태스크를 해체하고 숨겨진 복잡성을 수면 위로 끌어올려야 한다.

2.1 예외 처리 및 엣지 케이스(Edge Cases)의 가시화

엔지니어는 종종 정상적인 데이터 흐름만 가정하여 일정을 추정한다. EM은 WBS 항목에 예외 처리 로직(예: 네트워크 타임아웃, 트랜잭션 롤백, 서드파티 API 장애 시의 Fallback 매커니즘)이 명시적으로 분리되어 있는지 검증해야 한다.

2.2 로깅, 모니터링 및 테스트(Observability and Testing)

기능 구현이 완료되었다 하더라도, 프로덕션 환경에서 추적 가능성(Observability)이 확보되지 않으면 운영 단계에서 치명적 장애로 이어진다. EM은 단위 테스트(Unit Test) 작성, 통합 테스트(Integration Test) 시나리오, 그리고 로그 적재 및 모니터링 지표 설정이 개별 WBS 단위로 할당되어 있는지 확인해야 한다.

2.3 시스템 한계와 자원 제약(Resource Constraints) 인지

메모리 누수(Memory Leak)나 동시성(Concurrency) 문제 등은 개발 환경에서는 쉽게 발현되지 않는다. WBS 분할 과정에서 스레드 풀(Thread Pool) 크기, 데이터베이스 커넥션(Database Connection) 관리, 멱등성(Idempotency) 보장과 같은 심도 깊은 설계 요소가 태스크에 반영되도록 유도해야 한다.

5. 결론 및 EM의 리더십 원칙

EM의 WBS 리뷰는 팀원의 코드를 감시하기 위함이 아니라, 팀원이 안전하게 실패하고 학습할 수 있는 환경을 조성하는 적극적인 리스크 매니지먼트(Risk Management)이다.

성공적인 방어적 설계 리더십을 위한 3대 원칙은 다음과 같다.

  1. 해결책이 아닌 질문을 제공하라: “예외 처리를 추가해라“라고 지시하기보다 “결제사 서버가 5초 이상 응답하지 않으면 우리 서버의 커넥션 풀(Connection Pool)은 어떻게 되는가?“라고 질문하여 구성원 스스로 결함을 깨닫게 유도하라.
  2. 모든 가정(Assumption)을 의심하라: ’당연히 정상 동작할 것’이라는 암묵적 가정을 명시적 검증 대상으로 전환하라.
  3. 작게 실패하도록 설계하라(Design for Small Failures): 위험도(Severity)가 높은 작업일수록 WBS를 가장 작은 단위의 검증 가능한 마일스톤으로 쪼개어, 전체 시스템에 미치는 영향을 격리(Isolation)하라.

이러한 WBS 리뷰 과정을 일상화하면, 주니어 엔지니어는 자연스럽게 시스템 설계의 거시적 관점(Macro Perspective)과 인프라스트럭처의 한계를 학습하게 되며, 결과적으로 조직 전체의 엔지니어링 성숙도(Engineering Maturity)가 크게 향상된다.
