4.86.3.2 블로커 예측과 플랜 B 수립 훈련: 방어적 엔지니어링

소프트웨어 개발에서 완벽한 계획이란 존재할 수 없으며, 통제 범위를 벗어난 외부 요인으로 인한 지연이나 장애는 필연적으로 발생한다. 특히 주니어 엔지니어는 자신이 담당하는 로직이 완벽하게 동작할 것이라는 ’낙관적 편향(Optimism Bias)’을 기반으로 WBS(Work Breakdown Structure)를 수립하는 경향이 있다.

이러한 인지적 오류를 교정하기 위해 엔지니어링 매니저(EM)는 기획 단계에서부터 발생 가능한 블로커(Blocker, 작업 진행을 가로막는 장애물)를 사전에 도출하고, 플랜 B(우회 로직 및 대안)를 WBS의 필수 태스크로 편입시키는 방어적 엔지니어링(Defensive Engineering) 마인드셋을 팀 내에 주입해야 한다.

1. WBS에서의 블로커(Blocker) 예측 훈련

블로커 예측 훈련은 주니어 엔지니어가 자신의 작업이 시스템의 전체 아키텍처 및 외부 의존성(Dependencies)과 어떻게 연결되어 있는지 거시적 관점에서 파악하도록 돕는다.

WBS를 작성할 때, 각 컴포넌트나 API 통신 단계에 대해 다음과 같은 질문을 의무화하여 병목(Bottleneck) 리스크를 사전에 식별하도록 유도해야 한다.

  • 외부 서비스 의존도 검토: 서드파티(3rd-party) API가 다운되거나 응답성이 크게 저하되었을 때 우리 시스템의 UI/UX가 마비되지 않는가?
  • 리소스 한계 도달: 트래픽 폭증으로 인해 데이터베이스 커넥션이 모두 고갈되었을 때, 신규 요청을 대기시킬 것인가, 아니면 즉각 실패(Fail-fast) 처리할 것인가?
  • 유관 부서 스케줄 불일치: 인프라팀의 서버 세팅이나 타 개발팀의 API 배포 파이프라인이 지연될 경우, 현재 태스크의 개발과 테스트를 어떻게 독립적으로 진행할 것인가? (예: Mock 객체 활용)

2. 외부 장애에 대비한 안전 우회 로직(Circuit Breaker) 선행 반영

방어적 엔지니어링의 핵심은 예외 상황을 코드 작성 도중에 임기응변으로 처리하는 것이 아니라, 설계 및 WBS 산정 단계에서 아키텍처 패턴으로 확정 짓는 데 있다. 그 대표적인 사례가 회로 차단기(Circuit Breaker) 패턴의 태스크화이다.

2.1 회로 차단기 적용의 WBS 예시

주니어 엔지니어가 외부 인증 서버(OAuth)와 연동하는 작업을 WBS로 산정할 때, EM은 단순한 ‘연동 및 테스트’ 태스크를 넘어 장애 격리(Fault Isolation) 태스크가 포함되도록 지도해야 한다.

  • [기본 태스크] 외부 인증 API 연동 로직 작성 (2일)
  • [방어적 태스크 추가 1] 타임아웃(Timeout) 지속 발생 시 인증 요청을 차단하고 로컬 세션 풀백(Fallback)을 제공하는 Circuit Breaker 설정 (1일)
  • [방어적 태스크 추가 2] 회로 차단기 상태(Closed, Open, Half-Open) 변이를 외부 모니터링 대시보드로 전송하는 로깅 태스크 (0.5일)

이처럼 방어적 우회 로직을 WBS에 선행 반영함으로써, 개발자는 장애 발생 시 서비스가 멈추는 대신 기능이 제한적으로 축소되는 ’우아한 성능 저하(Graceful Degradation)’를 기본 스펙으로 인지하게 된다.

3. 낙관적 WBS와 방어적 WBS(Plan B 포함)의 아키텍처 비교

아래의 시스템 다이어그램은 블로커에 대비한 플랜 B(회로 차단기 및 캐시)가 WBS에 반영된 방어적 설계가 어떻게 시스템의 생존력을 높이는지 직관적으로 보여준다.

4. 플랜 B 수립 훈련의 리더십 전략

EM이 방어적 엔지니어링 마인드셋을 성공적으로 주입하기 위해서는 실무 지침과 리뷰 프로세스를 제도화해야 한다.

  1. 의존성 지도(Dependency Map) 작성 의무화: WBS를 제출할 때, 자신이 개발하는 모듈이 의존하고 있는 외부 API, 라이브러리, 데이터베이스 등의 리스트를 함께 명시하도록 한다.
  2. “What If” 시나리오 프리모텀(Pre-mortem): 프로젝트 시작 전 미팅에서 “만약 내일 오픈인데 결제 API 제공사가 서버 점검에 들어가면 어떻게 할 것인가?“와 같은 극단적인 프리모텀 토론을 진행하여 플랜 B 도출을 게임처럼 훈련시킨다.
  3. WBS 리뷰 템플릿의 체크리스트화: WBS 승인 조건에 ‘의존 시스템 장애 시의 Fallback 정책 유무’ 항목을 추가하여, 시스템적 강제성을 부여한다.

5. 결론

블로커를 예측하고 플랜 B를 수립하는 과정은 단순히 방어 장치를 하나 더 추가하는 작업이 아니다. 이는 주니어 엔지니어의 시야를 코드라는 미시적 세계에서 전체 시스템 생태계와 안정성이라는 거시적 세계로 확장시키는 근본적인 메타인지 강화 훈련이다.

시스템 장애는 피할 수 없다 하더라도, 계획 단계에서의 무방비 상태는 교육과 프로세스를 통해 확실하게 예방할 수 있다.
