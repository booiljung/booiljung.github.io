4.78.3.3 서드파티(3rd Party) 솔루션 연동 및 인프라 프로비저닝(IaC) 코드를 특정 인물의 로컬 환경에서 클라우드로 강제 이관

딥테크 및 IT 스타트업이 겪는 가장 아찔한 위기 중 하나는 핵심 인프라 스크립트나 외부 연동 환경 설정이 퇴사한 직원의 ‘개인 노트북’ 또는 ’개인 계정’에만 남아있어 시스템 복구나 확장이 불가능해지는 상황이다. 연구 개발 담당자(CTO)와 경영진(CEO)은 인프라 구성 요소와 서드파티 연동 자산이 조직의 소유가 아닌 특정 개인의 사유물이 전락하는 현상을 극도로 경계해야 하며, 이를 막기 위한 강제적인 클라우드 이관 프로세스를 수립해야 한다.

1. 서드파티(3rd Party) 연동 자산의 사유화 리스크

결제 대행사(PG), 알림 서비스(SMS, 카카오톡 알림톡), 외부 AI API(OpenAI 등), 소셜 로그인 등의 서드파티 서비스를 연동할 때는 반드시 API 키(Key), 시크릿(Secret), 인증서 등이 발급된다. 초기 스타트업이나 기술 관리 체계가 잡히지 않은 조직에서는 특정 개발자가 자신의 이름으로 된 계정을 생성하고, 인증 키를 자신의 로컬 PC의 하드코딩된 환경 변수 .env 파일이나 개인 클라우드 드라이브에 보관하는 경우가 잦다.

이러한 상태는 버스 지수(Bus Factor)를 1로 만드는 가장 대표적인 실패 사례이다. 해당 개발자가 부재중일 때 인증서가 만료되거나 API 한도 초과(Rate Limit)가 발생하면, 남은 인원은 계정 접근 권한조차 없어 서비스 장애를 속수무책으로 바라볼 수밖에 없다.

해결책:
모든 서드파티 계정은 ’회사 소유의 마스터 계정(또는 그룹 메일)’으로 강제 등록되어야 하며, 발급된 기밀 정보(Secrets)는 보안이 유지되는 클라우드의 중앙 집중식 저장소(예: AWS Secrets Manager, Azure Key Vault, HashiCorp Vault)에 안전하게 이관 및 보관되어야 한다.

2. 인프라 프로비저닝(IaC)의 로컬 실행 금지와 클라우드 파이프라인 강제화

인프라스트럭처로서의 코드(Infrastructure as Code, IaC)는 서버, 네트워크, 데이터베이스 등의 인프라 구성을 테라폼(Terraform), 앤서블(Ansible), AWS CloudFormation 등의 코드로 작성하고 관리하는 시스템이다. IaC 자체는 훌륭한 탈중앙화 기법이지만, 이를 실행하는 환경이 개인 개발자의 ’로컬 터미널’이라면 여전히 단일 장애점(SPOF)으로 작용한다.

A 개발자의 로컬 PC에서만 terraform apply 명령어가 작동하고, 다른 팀원이 동일한 코드를 내려받아 실행했을 때 권한 오류나 로컬 종속성(Dependency) 문제로 실패한다면, 그 코드는 형식지(Explicit Knowledge)로 기능하지 못하는 ’죽은 코드’나 다름없다.

해결책:
어떠한 개발자도 자신의 로컬 랩탑에서 운영 인프라 변경 명령을 즉각적으로 실행하지 못하도록 권한을 회수해야 한다. 모든 인프라 프로비저닝 코드는 중앙 컨트롤 타워 역할을 하는 클라우드 기반 CI/CD 파이프라인(예: GitHub Actions, Terraform Cloud, GitLab CI) 위에서만 실행되도록(Execution in the Cloud) 시스템적으로 강제되어야 한다.

4. 구조 개편 시 경영진 및 기술 리더의 점검 사항(Checklist)

  1. 전수 조사와 중앙 계정 편입: 조직 내에서 사용 중인 모든 SaaS 및 서드파티 솔루션 목록을 도출하고, 개인 이메일이나 개인 신용카드로 결제/등록된 계정을 법인 계정 및 법인 카드로 즉시 이관시킨다.
  2. 보안 감사(Audit) 트레일 확보: 로컬 실행을 막고 클라우드로 프로비저닝 권한을 이관하면, “누가(Who), 언제(When), 무엇을(What) 변경했는지“에 대한 추적성(Traceability)이 확보된다. 이는 장기적으로 기술 부채를 통제하는 강력한 감시망이 된다.
  3. 최소 권한의 원칙(Principle of Least Privilege): 인프라 파이프라인 계정에는 배포를 위한 최소한의 권한(IAM Roles)만을 부여하고, 개발자는 해당 파이프라인을 트리거할 수 있는 권한만 가지도록 역할을 분리해야 사고 발생 시 타격 반경을 좁힐 수 있다.

이처럼 자산과 실행 환경 모두를 개인의 로컬 영역에서 공용 클라우드로 강제 이관하는 조치는, 단순히 보안을 강화하는 행위가 아니라 소프트웨어 팀의 자율성을 보장하고 조직 전체의 운영 지식을 영속적인 자산으로 만드는 완성 단계라고 할 수 있다.
