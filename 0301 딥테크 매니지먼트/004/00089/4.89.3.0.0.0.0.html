4.89.3 매니저의 지시(Telling)를 코칭(Coaching)으로 진화시키는 질문(Questioning) 설계

R&D 조직의 매니저는 주니어 엔지니어나 팀원의 기술적 난제를 해결할 때, 일방적인 해답 지시(Telling)가 아닌 코칭(Coaching) 기반의 접근법을 취해야 한다. 코칭의 핵심은 정답을 ’알려주는 것’이 아니라, 적절한 질문(Questioning)을 통해 엔지니어 스스로 ’발견하게 하는 것’이다. 이는 엔지니어의 메타인지(Meta-cognition) 능력을 극대화하고 문제 해결 능력의 자생력을 키우는 가장 효과적인 리더십 도구이다.

1. 지시(Telling)의 한계와 코칭(Coaching)의 실효성

과거의 기계적인 하향식(Top-down) 조직에서는 매니저의 명확한 지시가 효율적일 수 있었다. 그러나 고도의 불확실성을 다루는 딥테크(Deep Tech) 분야에서는 지시형 리더십이 심각한 한계를 드러낸다.

  • 지시(Telling)의 한계: 매니저가 항상 시스템 전체의 최적 해답을 알 수는 없으며, 매니저 자체가 모든 기술적 결정의 병목(Bottleneck)이 된다. 또한 지시받은 대로 코딩만 수행하는 엔지니어는 시스템에 대한 책임감(Ownership)을 갖지 않고 수동적인 실행기(Executor)로 전락한다.
  • 코칭(Coaching)의 장점: 열린 질문(Open-ended Question)을 통해 구성원이 스스로 아키텍처를 설계하고 사고하도록 유도한다. 이는 암묵적(Tacit) 형태의 고민을 형식지(Explicit Knowledge)로 끄집어내는 과정이며, 지식 사일로(Silo) 현상을 예방하는 조직의 근본적인 체질 개선 작업이다.

2. 효과적인 질문 설계를 위한 GROW 모델

코칭 대화를 체계적이고 객관적으로 이끌기 위해 비즈니스 코칭의 표준인 **GROW 모델(GROW Model)**을 R&D 실무 회의나 1on1에 변형하여 적용하라.

  • G (Goal, 목표): “우리가 이 거대한 모놀리스(Monolith) 애플리케이션을 여러 마이크로서비스(Microservices)로 분리함으로써 궁극적으로 달성하려는 비즈니스 가치(Business Value)는 구체적으로 무엇입니까?”
  • R (Reality, 현실): “현재의 레거시(Legacy) 코드 구조에서 데이터베이스 트랜잭션 병목을 일으키는 가장 큰 원인은 어떠한 아키텍처적 한계 때문입니까?”
  • O (Options, 대안): “비동기 메시지 큐(Message Queue, 예: Kafka)를 도입하는 것 외에, 트래픽 폭증 시 데이터 일관성(Data Consistency) 문제를 해결할 수 있는 다른 아키텍처 패턴(Architecture Pattern)에는 어떤 것들이 있습니까?”
  • W (Will/Way Forward, 실행 의지 및 계획): “그렇다면, 방금 도출된 세 가지 대안 중 다음 스프린트(Sprint)에서 신속하게 PoC(Proof of Concept) 빌드를 통해 검증해 볼 방안은 무엇으로 결정하겠습니까?”

3. 지시형 대화와 질문형 대화의 실무 비교

답을 떠먹여 주는 매니저의 단편적 화법은 엔지니어의 비판적 인지 능력을 퇴화시킨다. 아래의 비교를 참조하여 대화를 재설계하라.

[나쁜 예: 지시와 정답 제공]

“이번 메인 API 응답 속도가 너무 느립니다. Redis를 붙여서 DB 조회 결과를 캐싱(Caching)하도록 코드를 당장 수정하세요.”

[좋은 예: 코칭과 문제 탐색 질문]

“최근 메인 API의 응답 지연 시간(Latency) 데이터가 P99 기준 500ms를 초과하고 있습니다. 데이터베이스의 I/O 병목을 우회하면서 응답 속도를 개선하기 위해, 현재 시스템 구조에서 우리가 시도해 볼 수 있는 인메모리(In-memory) 최적화 전략에는 어떤 것들이 있을까요?”

이러한 개방된 질문은 엔지니어가 단순히 Redis 라이브러리 코드를 붙여넣어 단기 땜질을 하는 대신, 캐시 무효화 전략(Cache Invalidation Strategy)과 동시성 문제까지 종합적으로 설계(Design)하도록 스스로를 압박하게 만든다.

5. 지시에서 코칭으로 확실히 전환하기 위한 리더십 수칙

  1. 소크라테스식 문답법(Socratic Questioning) 채택: 팀원이 논리적 비약을 범하거나 잘못된 대안을 제시했을 때 즉시 반박하기보다, “그 옵션을 선택했을 때 보안(Security)이나 유지보수(Maintainability) 측면에서 우려되는 트레이드오프(Trade-off)는 무엇인지 설명해 주시겠습니까?“라고 되물어라.
  2. 침묵(Silence)의 기술 전략적 활용: 질문을 던진 후, 구성원이 답을 찾기 위해 사고하는 3~10초의 어색한 공백을 참지 못하고 매니저가 먼저 답을 제시하는 실수를 결코 범하지 마라. 인지적 고통과 사유의 시간 없이는 성장이 없다.
  3. 겸손한 탐구(Humble Inquiry)의 자세 확보: ’내가 이미 모든 정답을 알고 있다’는 가정 하에 던지는 확인성 질문(Leading Question, 유도 질문)을 철저하게 버려라. 매니저 스스로도 진심으로 엔지니어의 로직(Logic) 전개 방식을 이해하고 새로운 접근을 탐구하고자 하는 진짜 호기심에 기반하라.

질문 중심의 리딩(Leading by Questioning)은 엔지니어링 조직 내에 극한의 심리적 역동성과 자기 주도적 책임감(Ownership)의 문화적 토대를 견고하게 세운다. 따라서 R&D 부문 최고 경영진(CTO, VP of Engineering)은 중간 기술 관리자들의 1on1 미팅 품질을 평가할 때, 그들이 얼마나 지식을 명확히 ’전달’하는가가 아니라, 얼마나 날카롭고 깊이 있게 업무의 본질을 ’질문’하는가를 핵심 리더십의 척도로 삼아야 한다.
