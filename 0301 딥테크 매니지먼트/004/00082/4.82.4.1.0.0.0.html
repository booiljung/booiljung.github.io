4.82.4.1 거대 규모 PR(Big Bang PR)이 유발하는 무비판적 ‘LGTM(Looks Good To Me)’ 통과 방지: PR당 최대 변경 라인 수(LoC) 제한 및 원자적 커밋(Atomic Commit) 강제 분할 정책

소프트웨어 연구 개발 조직에서 수백, 수천 줄의 코드가 포함된 거대 규모의 풀 리퀘스트(Pull Request, 이하 PR)가 올라올 경우, 리뷰어가 코드를 면밀히 검토하지 않고 관행적으로 ’LGTM(Looks Good To Me)’을 외치며 승인(Approve) 처리하는 현상이 잦다. 이는 기술 부채(Technical Debt)를 급증시키고, 치명적인 결함(Defect)을 운영 환경(Production Environment)에 유입시키는 주된 원인이 된다. 본 절에서는 거대 규모 PR, 이른바 ’빅뱅 PR(Big Bang PR)’이 야기하는 병폐를 분석하고, 이를 방지하기 위한 정량적 라인 수(Lines of Code, LoC) 제한 및 원자적 커밋(Atomic Commit) 강제 분할 정책에 대해 상세히 다룬다.

1. 거대 규모 PR이 유발하는 구조적 문제와 인지적 한계

개발자가 여러 장의 요구사항을 하나의 PR에 모두 담아 리뷰를 요청할 때, 리뷰를 수행하는 동료 엔지니어는 심각한 인지 과부하(Cognitive Overload)를 겪게 된다.

초기에는 꼼꼼하게 함수의 로직과 변수의 생명주기를 파악하려 노력하지만, 리뷰해야 할 코드의 양이 인간의 단기 기억(Short-term Memory) 범위를 초과하게 되면 뇌는 피로도를 낮추기 위해 점차 대충 넘기는 ‘스캐닝(Scanning)’ 모드로 전환한다. 시스코 시스템즈(Cisco Systems)의 코드 리뷰 연구에 따르면, 한 번에 리뷰하는 코드가 200~400 라인을 초과할 경우 리뷰어의 결함 발견율(Defect Density)이 급격히 저하된다 구조적 한계점이 존재함이 입증된 바 있다(“Best Kept Secrets of Peer Code Review”, SmartBear).

또한, 거대한 PR은 다수의 기능 변경과 리팩토링, 버그 수정이 혼재되어 있어, 리뷰어가 코드를 원래의 비즈니스 로직과 대조하며 검증하는 것을 불가능하게 만든다. 특히 주니어 엔지니어의 경우, 복잡한 거대 PR을 지적했다가 양산될 추가 논의가 부담스러워 단순히 묵인하는 현상도 빈번하다.

2. PR당 최대 변경 라인 수(LoC) 제한 정책

무비판적 LGTM을 근본적으로 차단하기 위해서는 PR의 크기를 정량적으로 제한해야 한다. 리뷰어가 한 번의 세션(약 60분 이내)에서 집중력을 잃지 않고 검토할 수 있는 최대 코드 라인 수(Lines of Code, LoC)를 조직의 표준으로 설정하고, 지속적 통합(Continuous Integration, CI) 파이프라인에 이를 강제하는 메커니즘을 부여하는 것이 효과적이다.

2.1 최적의 LoC 설정 및 예외 처리

일반적으로 단일 PR의 변경 라인 수는 250~300줄 내외로 제한하는 것이 권장된다. 단, 주석 추가, 자동 생성된 코드(Generated Code), 순수 패키지 의존성 업데이트(예: package-lock.json 등)는 리뷰의 실질적 부하를 높이지 않으므로 LoC 산정에서 제외하는 예외 처리 규칙(Exclusion Rule)을 설정해야 한다.

2.2 CI/CD 파이프라인을 통한 자동 거절(Auto-Reject)

조직의 코드 저장소(예: GitHub, GitLab) 에 훅(Webhook) 또는 정적 분석 도구(예: Danger JS, SonarQube)를 연동하여, 임계치를 초과하는 PR이 생성될 경우 즉각적으로 빌드 실패(Build Fail)를 발생시키고 리뷰를 반려(Reject)하는 정책을 도입한다.

[예시] GitHub Actions와 Danger JS를 이용한 LoC 제한 룰

import { danger, warn, fail } from "danger"

const MAX_LOC = 300;
const changes = danger.github.pr.additions + danger.github.pr.deletions;

if (changes > MAX_LOC) {
  fail(`현재 PR의 변경 라인 수가 ${changes}줄입니다. PR 크기가 ${MAX_LOC}줄을 초과할 수 없습니다. 기능을 모듈 단위로 분리하여 여러 개의 PR로 나누어 주십시오.`);
}

이러한 자동화는 리뷰어가 직접 동료에게 PR을 줄여달라고 요구하는 인간적인 껄끄러움을 배제하고, 규칙 기반(Rule-based)의 객관적 거절을 가능하게 하므로 조직 내 감정 소모를 줄인다.

3. 원자적 커밋(Atomic Commit) 강제 분할 정책

PR 크기를 줄이기 위한 근본적인 선행 조건은 개발 과정에서부터 커밋(Commit)이 ’원자성(Atomicity)’을 가지도록 작게 쪼개어 기록하는 습관이다. 원자적 커밋이란 “단 하나의 논리적 변경 사항(기능 추가, 버그 수정, 리팩토링 등)만을 포함하며, 해당 커밋만으로도 프로젝트가 정상적으로 빌드 및 실행되어야 함“을 의미한다.

3.1 원자적 커밋의 필수 조건

  1. 단일 목적성(Single Purpose): 하나의 커밋은 하나의 명확한 이유로만 이루어져야 한다. 버그를 수정하면서 동시에 다른 메소드를 리스타일링(Restyling)해서는 안 된다.
  2. 독립적 실행 가능성(Testable): 특정 커밋으로 프로젝트를 체크아웃(Checkout) 했을 때, 컴파일 에러가 발생하거나 테스트가 실패해서는 안 된다.
  3. 가독성 높은 메시지(Readable Message): 커밋 메시지만 보고도 변경 사항의 의도를 정확히 파악할 수 있어야 한다(예: Conventional Commits 규격 활용).

3.2 관심사의 분리 적용

기능 개발(Feature Development)과 코드 구조 개선(Refactoring)은 절대로 동일한 PR이나 커밋에 섞여서는 안 된다. 리팩토링으로 인해 수많은 파일이 변경되면, 정작 핵심 비즈니스 로직에 추가된 몇 줄의 기능 코드가 눈에 띄지 않게 되어 치명적인 논리 결함이 LGTM으로 통과할 확률이 극대화된다.

  • Bad Practice: 하나의 커밋에 로그인 인증 로직 추가 및 사용자 서비스 클래스 코드 정리
  • Good Practice:
  • 커밋 1: Refactor: 사용자 서비스 클래스 메소드 추출 및 정리
  • 커밋 2: Feat: 로그인 인증 예외 처리 로직 추가

4. 정책 도입을 위한 경영진 및 기술 리더의 역할

이러한 정책을 도입할 때 주니어 또는 기존 관행에 익숙한 시니어 개발자조차 “개발 속도가 느려지고 귀찮아진다“며 반발할 수 있다. 기술 책임자(CTO) 및 테크 리드(Tech Lead)는 이 정책이 개발자를 통제하기 위한 마이크로 매니지먼트(Micro-management)가 아니라, 시스템의 안정성을 높이고 리뷰어의 권리를 보호하며 결함으로 인한 후속 수정 비용을 최소화하기 위한 품질 보증(Quality Assurance)의 핵심 프로토콜임을 지속적으로 설득해야 한다.

조직은 작은 PR의 잦은 병합(Merge)을 뒷받침할 수 있도록 배포 파이프라인의 속도를 최적화하고, 자동화 테스트 커버리지(Test Coverage)를 높여 세분화된 변경 사항이 안정적으로 운영에 반영될 수 있는 물리적 토대를 반드시 함께 제공해야 한다.
