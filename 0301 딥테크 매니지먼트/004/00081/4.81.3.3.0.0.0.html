4.81.3.3 문서 간 정보 중복(DRY 위반) 방지 지침: 특정 페이지에 시스템 속성을 중복 복사하지 않고, 단일 진실 공급원(SSOT) 원칙 하에 Include 매크로 및 동적 하이퍼링크 참조로만 연결

1. 개요 및 근본적 문제 정의

기술 조직이 성장하고 연구개발(R&D) 인력이 증가함에 따라, 사내 위키(Wiki) 공간에는 방대한 양의 기획 문서, 시스템 아키텍처 문서, API 명세서, 배포 가이드라인 등이 누적된다. 이때 가장 빈번하게 발생하는 치명적인 안티패턴(Anti-pattern)은 특정 시스템 속성, 네트워크 주소, 비즈니스 룰, 도메인 용어 등의 정보가 여러 문서에 중복으로 복제되어 기재되는 현상이다.

이러한 정보의 파편화와 중복은 정보 공학에서 금기시하는 DRY(Don’t Repeat Yourself) 원칙의 명백한 위반이다. 시스템 사양이 변경되었을 때, 이를 기술한 일부 문서만 수정되고 나머지 복제된 문서들이 방치된다면, 시스템 구성원들은 낡은 정보를 기반으로 시스템을 구현하거나 테스트하게 된다. 이는 치명적인 장애(Defect) 발생, 롤백(Rollback), 그리고 팀 간의 커뮤니케이션 비용 급증이라는 심각한 부작용을 초래한다. 따라서 조직 내 R&D 문서화(Documentation) 체계는 ‘단일 진실 공급원(Single Source of Truth, SSOT)’ 원칙을 확립하고, 정적인 텍스트 복사를 금지하며, 동적 참조 메커니즘을 전면 도입해야 한다.

2. DRY(Don’t Repeat Yourself) 원칙의 문서화 적용

DRY 원칙은 “모든 형태의 지식은 시스템 내에서 단일하고, 명확하며, 신뢰할 수 있는 구체적인 표현을 가져야 한다“고 천명한다(Hunt & Thomas, The Pragmatic Programmer).

  • 정적 복제 금지: 소프트웨어 코드 작성 시 동일한 비즈니스 로직을 여러 함수에 반복해서 복사-붙여넣기(Copy & Paste) 하지 않듯이, 문서화 과정에서도 시스템 식별자, 데이터베이스 스키마 속성, 서버 IP, 혹은 릴리스 일정 등의 핵심 요소를 여러 문서에 반복 서술하는 행위를 철저히 배제하라.
  • 문서의 모듈화(Modularization): 공통적으로 사용되는 지식 컴포넌트는 독립적인 페이지나 특정 데이터 블록으로 모듈화시켜야 한다. 특정 서비스의 상태 코드(Status Code) 목록이나 에러 메시지 정의가 필요하다면, 이를 각 서비스의 기획서에 개별 표기하는 대신 중앙 집중화된 ’공용 상태 코드 명세서’에서 관리해야 한다.

3. 단일 진실 공급원(SSOT) 기반의 참조 지침

단일 진실 공급원(SSOT)은 시스템 내의 모든 컴포넌트나 문서 모델이 오직 하나의 근본 데이터를 바라보도록 강제하는 아키텍처 원칙이다. 이를 문서 시스템에 도입하기 위해서는 다음과 같은 기술적 수단을 활용하라.

3.1 Include 매크로(Macro)의 적극적 도입

Confluence나 Notion 등 현대적인 엔터프라이즈 위키 플랫폼은 특정 문서의 내용이나 데이터 블록을 다른 문서에 실시간으로 불러와서 렌더링하는 Include 기능을 제공한다(예: Confluence의 ‘Include Page’ 또는 ‘Excerpt Include’ 매크로).

  • 컴포넌트 뷰 구성: 복잡한 시스템 아키텍처 다이어그램이나 핵심 지표(KPI) 표 등을 각 하위 시스템의 설계 문서 맨 앞에 고정적으로 표시해야 한다면, 페이지마다 이미지를 각각 삽입하지 마라. 하나의 공통 마스터 페이지에 다이어그램을 위치시키고, 나머지 문서에서는 Include 매크로를 사용하여 해당 영역을 호출하라.
  • 실시간 정합성 보장: ма스터 문서가 수정되면, 해당 문서를 Include하는 수십 개의 연관 문서들 역시 페이지 로드 시점에 즉시 최신 상태로 반영된다. 이는 업데이트 누락에 따른 장애 발생 가능성을 0으로 수렴하게 만든다.

3.2 동적 하이퍼링크(Dynamic Hyperlink)를 통한 문맥적 연결

부득이하게 다른 문서의 세부 내용을 설명해야 할 때, 그 내용을 풀어서 다시 설명하는 과오를 범하지 마라. 대신 강력한 동적 하이퍼링크 체계를 활용하라.

  • 요약과 링크의 분리: 서술형 문장 내에 구체적인 스펙을 하드코딩(Hardcoding)하지 마라. 예를 들어, “결제 시스템의 재시도 임계값은 3회, 타임아웃은 5,000ms로 설정한다“라고 적는 대신, “결제 로직의 예외 처리 정책은 [결제 시스템 리질리언스 정책 설정 항목]을 엄격히 준수한다“라고 명시하고 SSOT 페이지로 링크하라.
  • 링크 상태 추적: 동적 하이퍼링크는 원본 페이지의 제목이 변경되거나 위치가 이동되더라도 연결이 끊어지지 않아야 한다. 이를 위해 문서 도구가 제공하는 시스템 내부 고유 ID 기반의 링크 삽입 기능을 표준화하라.

4. 코드 애즈 파일스(Docs-as-Code)와 소스코드 연동

조직 내에서 가장 완벽한 형태의 SSOT는 ‘실제 동작하는 소스코드(Working Code)’ 그 자체이다. 엔지니어 조직에서는 다음 기법을 활용하여 문서와 코드의 정합성을 최고 수준으로 끌어올려라.

  • Swagger/OpenAPI 동적 연동: API 명세는 결코 위키에 수동으로 표기하지 마라. 소스코드 내의 어노테이션(Annotation)에서 자동 생성된 OpenAPI 스키마 데이터를 위키 시스템이 동적으로 파싱(Parsing)하여 보여주도록 통합 플러그인을 구축하라.
  • 설정 파일 리포지토리 직결 연동: 운영 환경의 중요한 변수(Timeout, Rate Limit, 분산 캐시 만료 시간 등)는 Git 저장소의 YAML 혹은 JSON 환경 설정 파일을 직접 참조하여 문서에 표출하라.
  • 이를 통해 개발자가 코드베이스의 속성을 변경하고 머지(Merge)하는 순간, 별도의 문서화 작업 없이 사내 위키의 정보도 최신 상태로 자동 동기화된다.

5. 리더십의 역할과 기술 부채 관리

소프트웨어 자산으로서 위키를 운영하는 리더, 프러덕트 오너(PO), 그리고 스크럼 마스터는 구성원들의 문서 작성 행태를 주기적으로 리뷰(Review)해야 한다.

  • 문서 리뷰 프로세스 확립: 코드 리뷰(Code Review)와 마찬가지로 기획서 및 설계서에 대한 리뷰를 거쳐라. 리뷰 과정에서 정보의 무단 복제 정황이 포착되면, 즉시 DRY 위반이자 문서적 기술 부채(Technical Debt in Documentation)로 규정하고 리팩터링(Refactoring)을 요구하라.
  • 스파게티 문서 의존성 방지: 무분별한 Include와 링크 연동은 문서 간의 순환 참조(Circular Reference)를 야기할 수 있다. 도메인 주도 설계(DDD)의 바운디드 컨텍스트(Bounded Context) 패턴을 준용하여, 상위 도메인 문서가 하위 컴포넌트 속성을 단방향으로 참조하는 계층형 위키 구조를 제도화하라.

문서 간 고립과 불일치를 막는 것은 단순한 관리 편의성의 문제가 아니라, 제품(Product)의 신뢰성과 직결되는 무결성 경영(Integrity Management)의 핵심 기반임을 한시도 망각해서는 안 된다.
