4.88.1.3 ’작동하는 레거시(Working Trash)’의 무비판적 양산: 출력 결과(Output)는 정상이지만, 시간/공간 복잡도(Complexity), 트랜잭션 안전성, 예외 처리(Exception) 관점에서 낙제점인 코드가 주니어의 로컬 터미널 내에서 자가 승인되는 위험성

소프트웨어 연구 개발(R&D) 조직에서 단편적 목표만 부여받은 주니어 엔지니어들이 가장 빈번하게 빠지는 치명적 함정은 기계적인 인풋(Input) 대비 기대하는 아웃풋(Output)이 반환된다는 사실 하나만으로 해당 모듈의 구현이 완료되었다고 단정 짓는 데 있다. 이처럼 화면상에는 정상적으로 나타나지만, 내부의 설계 구조와 최적화 관점에서는 결코 프로덕션(Production) 레벨에 적합하지 않은 코드를 업계에서는 이른바 ’작동하는 레거시(Working Trash)’라고 명명한다. 이는 주니어 엔지니어의 로컬 터미널 환경에서 자가 승인(Self-Approval) 과정을 거쳐 메인 저장소로 유입되는 핵심 기술 부채(Technical Debt)의 진원지가 된다.

1. ’작동하는 레거시’의 본질과 자가 승인 편향

코드를 작성하는 주니어 엔지니어의 시야는 대게 자신의 노트북 머신인 로컬 터미널(Local Terminal) 안에 국한되어 있다. 이 제한된 환경에서는 데이터베이스의 레코드 수가 수십 개 이하이며, 동시 접속자는 오직 테스트를 수행하는 본인 한 명뿐이다. 이러한 환경 변수 통제 하에서 작성된 코드는 어떠한 비효율적인 로직으로 작성되어도 지연 시간(Latency)이나 장애를 겉으로 드러내지 않는다.

결과적으로, 포스트맨(Postman)이나 로컬 브라우저로 API를 호출했을 때 200 OK 상태 코드와 함께 JSON 응답이 반환되면, 주니어 엔지니어는 자신의 코드 품질을 시스템에 통합해도 안전한 단계인 것으로 착각(Dunning-Kruger Effect)하게 되며 무비판적으로 버전 관리 시스템에 소스를 밀어 넣는 병리적 현상을 반복한다.

2. 프로덕션 낙제점을 판가름하는 3대 아키텍처 요소

아웃풋만 정상인 코드가 왜 시스템 차원의 재난을 초래하는지, 핵심 소프트웨어 공학 지표 세 가지를 들어 설명한다.

2.1 시간/공간 복잡도(Time/Space Complexity)의 간과

구글링이나 AI의 스니펫을 수정한 코드는 대개 우아한 알고리즘 설계보다는 직관적이고 1차원적인 다중 루프(Nested Loop)를 채택하는 경우가 많다.

  • 시간 복잡도: 배열 내에서 특정 아이템을 찾기 위해 O(1)의 해시 테이블(Hash Table) 대신, O(N^2)의 이중 반복문을 사용하는 경우를 예로 들 수 있다. 로컬 환경에서 N=10일 때는 100번의 연산에 불과하여 0.01초 내에 끝나지만, 프로덕션에서 N=100,000이 되는 순간 100억 번의 연산이 수행되며 서버의 CPU 자원을 독점(Starvation)하게 된다.
  • 공간 복잡도: 방대한 데이터베이스를 순차적으로 스트리밍(Streaming) 방식으로 처리해야 함에도 불구하고, 한 번의 쿼리로 전체 데이터를 메모리(RAM)에 일괄 로드(Load)하는 코드를 작성할 경우 메모리 고갈 즉, OOM(Out of Memory) 크래시를 유발한다.

2.2 트랜잭션 안전성(Transactional Safety)의 위배

비즈니스 로직은 단일 작업 단위로써 원자성(Atomicity), 일관성(Consistency), 고립성(Isolation), 지속성(Durability) 즉, ACID 원칙을 준수해야 한다.
주니어 엔지니어가 작성한 구매 로직이 ’재고 차감’과 ’결제 기록 생성’이라는 두 가지 쿼리로 분리되어 있다고 가정해 보자. 순수하게 기능 관점에서는 잘 동작하지만, 첫 번째 쿼리 실행 후 두 번째 쿼리를 실행하기 직전에 네트워크 오류나 서버 재시작이 발생하면 재고는 줄어들었으나 결제 기록은 없는 치명적 데이터 오염(Data Corruption) 상태가 발생한다. 트랜잭션 경계(Transaction Boundary)를 설정하고 실패 시 롤백(Rollback)하는 안전장치가 빠진 코드는 언제 폭발할지 모르는 시한폭탄과 같다.

2.3 극단적 해피 패스(Happy Path) 중심의 예외 처리(Exception Handling)

’작동하는 레거시’의 가장 큰 특징은 외부 시스템 의존성이 100% 정상 작동한다는 비현실적 대전제하에 작성된다는 것이다.

  • 네트워크 단절 및 지연: 외부 결제 API의 응답 속도가 30초 이상 지연될 때 타임아웃(Timeout)을 발생시키고 연결을 끊어 스레드를 반환해야 하지만, 이러한 예외 처리가 누락되면 모든 서버 스레드가 좀비 상태로 대기하게 된다.
  • Null 참조 예외(Null Pointer Exception): 예상치 못한 빈(Empty) 데이터가 유입되었을 때 방어적 프로그래밍(Defensive Programming)이 부재하여 런타임 시스템이 강제 종료되는 치명적 버그가 빈발한다.

4. R&D 책임자를 위한 매니지먼트 전략

최고 기술 책임자(CTO) 및 R&D 매니저는 이러한 하급 코드가 무비판적으로 양산되지 않도록 강력한 아키텍처 가드레일(Guardrail)을 세워야 한다.

  1. 정적 분석(Static Analysis) 및 CI/CD 파이프라인 의무화: 소나큐브(SonarQube) 등의 정적 분석 도구를 통합하여 순환 복잡도(Cyclomatic Complexity)가 높은 코드나 인덱스를 타지 않는 비효율적 쿼리, 예외 없는 블록킹(Blocking) I/O가 감지되면 자동으로 PR 커밋을 반려(Reject)하는 정책을 시행하라.
  2. 데이터 모의 확장 테스트(Mock Scale Test): 로컬 환경의 착각을 깨기 위해, 개발 DB 환경에 최소 수만 건 이상의 가짜(Mock) 데이터를 기본적으로 주입해 놓아야 한다. 이렇게 하면 O(N^2) 알고리즘을 작성한 주니어 엔지니어는 로컬 브라우저에서 화면을 볼 수조차 없을 것이며, 최적화가 필수적임을 스스로 체감하게 된다.
  3. 혼돈 공학(Chaos Engineering) 요건 훈련: 주니어에게 과제를 브리핑할 때 단순히 명세서(Specification)만 제공하는 것이 아니라, “외부 통신이 3초 내에 응답하지 않을 때 시스템이 어떻게 반응해야 하는가?” 혹은 “동시에 5명의 사용자가 수정 버튼을 누르면 어떤 락(Lock) 매커니즘이 방어하는가?“를 요구사항에 직접 명시해야 한다.

엔지니어링의 본질은 무에서 화면상의 결과물을 조립해 내는 단편적 기술이 아니라, 시스템의 라이프사이클 안착과 극단적 성능 통제 능력에 있다. 겉보기 등급만 훌륭한 레거시 코드를 조기에 식별하여 반려하지 않으면, 언젠가 치러야 할 유지보수 비용과 인프라 요금은 눈덩이처럼 조직의 자본을 소모시킬 것이다.
