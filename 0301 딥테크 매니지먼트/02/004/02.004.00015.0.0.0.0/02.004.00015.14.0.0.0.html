15.14 프로젝트 범위 변경(Scope Change) 및 변경 통제 위원회(CCB) 운영

1. 범위 변경 관리의 필요성

프로젝트 범위 변경(Scope Change)은 프로젝트 실행 중 초기에 합의된 범위에 대한 추가, 삭제, 또는 수정을 의미한다. PMBOK Guide에 따르면, 범위 변경은 프로젝트의 일정, 비용, 품질, 그리고 리스크에 연쇄적 영향을 미치므로 체계적 관리가 필수적이다. 통제되지 않은 범위 확대(Scope Creep)는 프로젝트 실패의 주요 원인 가운데 하나이다.

딥테크 프로젝트에서 범위 변경의 빈도는 기술적 불확실성에 비례하여 증가한다. 기술 개발 과정에서의 예기치 못한 발견, 고객 요구의 진화, 규제 환경의 변화, 그리고 경쟁 환경의 변동이 범위 변경의 주요 촉발 요인이다.

2. 변경 통제 프로세스

변경 통제 프로세스(Change Control Process)는 다음 단계로 구성된다. 변경 요청의 접수(Change Request Submission)이다. 모든 범위 변경은 공식적 변경 요청서(Change Request Form)를 통하여 접수된다. 영향 분석(Impact Analysis)이다. 기술 담당이 기술적 영향(구현 노력, 아키텍처 영향, 기술 리스크)을, 과제 관리자가 프로젝트적 영향(일정, 비용, 자원)을 각각 분석한다. 변경 통제 위원회 심의(CCB Review)이다. 변경 통제 위원회가 영향 분석 결과에 기반하여 변경의 승인, 반려, 또는 보류를 결정한다. 변경의 실행(Change Implementation)이다. 승인된 변경이 프로젝트 계획, 기술 사양, 그리고 관련 문서에 반영되고 실행된다.

3. 변경 통제 위원회(CCB)의 구성과 운영

변경 통제 위원회(Change Control Board, CCB)는 범위 변경에 대한 승인 권한을 가지는 의사결정 기구이다. 딥테크 프로젝트의 CCB 구성은 과제 관리자(의장), 기술 담당 대표(기술적 영향 평가), 영업 담당 대표(고객 영향 평가), 프러덕트 오너(제품 전략 관점), 그리고 필요시 품질 보증 담당과 규제 담당을 포함한다.

CCB의 의사결정 기준은 변경의 기술적 타당성, 일정·비용에의 영향, 고객 가치에의 기여, 리스크 수준, 그리고 제품 전략과의 정합성이다. CCB의 운영 원칙은 신속한 의사결정(긴급 변경에 대한 신속 심의 절차), 투명한 기록(모든 변경 결정의 문서화), 그리고 이해관계자에의 적시 전달이다.
