16.24 칸반(Kanban)과 스크럼의 혼합(Scrumban)에서의 스크럼 마스터 역할 변화

1. 칸반의 핵심 원칙과 스크럼과의 차이

칸반(Kanban)은 Anderson(2010)이 소프트웨어 개발에 체계화한 흐름 기반(Flow-based) 작업 관리 방법론이다. 칸반의 핵심 원칙은 작업의 시각화(Visualize Work), 진행 중 작업 제한(Limit Work in Progress, WIP), 흐름의 관리(Manage Flow), 프로세스 정책의 명시화(Make Process Policies Explicit), 피드백 루프의 구현(Implement Feedback Loops), 그리고 협력적이고 실험적인 개선(Improve Collaboratively, Evolve Experimentally)이다.

스크럼은 시간 제한적 반복 주기(Sprint)를 중심으로 구조화되는 반면, 칸반은 연속적 흐름(Continuous Flow)을 중심으로 운영된다. 스크럼은 역할(PO, SM, Developers), 이벤트(Sprint Planning, Daily Scrum, Sprint Review, Sprint Retrospective), 그리고 산출물(Product Backlog, Sprint Backlog, Increment)을 명시적으로 정의하는 반면, 칸반은 기존 프로세스 위에 점진적으로 적용되는 경량적 접근이다.

2. 스크럼반(Scrumban)의 개념

스크럼반(Scrumban)은 Ladas(2009)가 제시한 스크럼과 칸반의 하이브리드 접근이다. 스크럼의 구조(스프린트, 역할, 이벤트)를 유지하면서 칸반의 원칙(WIP 제한, 흐름 시각화, 리드 타임 측정)을 통합하는 방식이다.

3. 스크럼 마스터의 역할 변화

스크럼반 환경에서 스크럼 마스터의 역할은 다음과 같이 변화한다.

흐름 관리의 촉진이다. 스프린트 단위의 속도(Velocity) 관리에 더하여, 개별 작업 항목의 리드 타임(Lead Time)과 주기 시간(Cycle Time)의 측정과 최적화를 촉진한다. 누적 흐름 다이어그램(Cumulative Flow Diagram, CFD)을 활용하여 작업 흐름의 건강 상태를 모니터링한다.

WIP 제한의 관리이다. 팀이 동시에 진행하는 작업의 수를 제한하는 WIP 정책을 수립하고, 이 정책의 준수를 촉진한다. Little의 법칙(Little’s Law)에 따르면, 리드 타임은 WIP와 처리량(Throughput)의 비율로 결정되므로, WIP의 제한은 리드 타임의 단축으로 이어진다.

병목(Bottleneck)의 식별과 해소이다. 칸반 보드의 시각적 분석을 통하여 작업 흐름의 병목 지점을 식별하고, 병목의 해소를 팀과 함께 추진한다.

이벤트의 유연한 운영이다. 스크럼반에서는 스프린트 경계의 엄격성이 완화될 수 있으며, 스크럼 마스터는 팀의 흐름에 맞추어 이벤트의 빈도와 형식을 유연하게 조정한다.

딥테크 기업에서 스크럼반은 유지보수 활동, 고객 지원 대응, 그리고 하드웨어 팀의 비반복적 작업 등 고정된 스프린트 주기에 맞추기 어려운 활동에 특히 적합하다.
