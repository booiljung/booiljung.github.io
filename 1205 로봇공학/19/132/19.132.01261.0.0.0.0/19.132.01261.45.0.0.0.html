1261.45 실행자(Executor)와 콜백 그룹의 관계

1. 실행자의 정의와 역할

ROS2에서 실행자(Executor)는 노드에 등록된 콜백(callback) 함수들을 스케줄링하고 실행하는 핵심 런타임 구성요소이다. 토픽 구독 콜백, 서비스 요청 처리 콜백, 타이머 콜백, 액션 관련 콜백 등 모든 이벤트 기반 처리가 실행자를 통해 디스패치(dispatch)된다.

ROS1에서는 ros::spin() 또는 ros::spinOnce()라는 단일 전역 함수에 의해 모든 콜백이 처리되었다. 이러한 구조는 콜백 실행의 병렬성 제어나 우선순위 관리가 불가능하다는 한계를 지니고 있었다. ROS2는 실행자라는 추상화 계층을 도입하여 콜백 스케줄링에 대한 세밀한 제어를 가능하게 하였다.

2. 실행자의 동작 원리

실행자는 내부적으로 대기 집합(wait set)을 관리한다. 대기 집합에는 구독자(subscription), 타이머(timer), 서비스 서버(service server), 액션 서버(action server), 클라이언트(client) 등의 핸들이 등록된다. 실행자는 반복적으로 다음의 과정을 수행한다.

  1. 대기 집합에 등록된 모든 엔티티에 대해 준비 상태를 확인한다.
  2. 새로운 데이터가 도착하였거나 타이머가 만료된 등 준비 완료된 엔티티를 식별한다.
  3. 해당 엔티티에 연결된 콜백 함수를 실행한다.
  4. 위 과정을 반복한다.

이 과정은 DDS 미들웨어의 조건 변수(condition variable) 및 대기 집합 메커니즘에 기반하며, 불필요한 폴링(polling)을 최소화하여 CPU 자원을 효율적으로 사용한다.

3. 실행자의 유형

ROS2는 다음의 실행자 유형을 제공한다.

실행자 유형클래스명특성
단일 스레드 실행자SingleThreadedExecutor하나의 스레드에서 모든 콜백을 순차적으로 실행한다.
멀티 스레드 실행자MultiThreadedExecutor다수의 스레드를 사용하여 콜백을 병렬로 실행한다.
정적 단일 스레드 실행자StaticSingleThreadedExecutor노드 추가 시 대기 집합을 한 번만 구성하여 오버헤드를 줄인다.

단일 스레드 실행자는 콜백 간 경쟁 조건(race condition)이 발생하지 않으므로 동기화 문제를 회피할 수 있다는 장점이 있으나, 하나의 콜백이 장시간 차단(blocking)될 경우 다른 콜백의 실행이 지연된다는 단점이 존재한다. 멀티 스레드 실행자는 이러한 지연 문제를 완화하지만, 공유 자원에 대한 동기화 메커니즘이 필요하다.

4. 콜백 그룹의 개념

콜백 그룹(Callback Group)은 콜백 함수들의 실행 정책을 정의하는 논리적 묶음이다. 멀티 스레드 실행자를 사용할 때, 콜백 그룹은 어떤 콜백들이 병렬로 실행될 수 있는지, 어떤 콜백들이 상호 배타적으로 실행되어야 하는지를 결정한다.

ROS2는 두 가지 유형의 콜백 그룹을 제공한다.

4.1 상호 배타적 콜백 그룹 (Mutually Exclusive Callback Group)

MutuallyExclusiveCallbackGroup에 속하는 콜백들은 동시에 실행되지 않는다. 하나의 콜백이 실행 중이면 같은 그룹에 속하는 다른 콜백들은 대기 상태에 놓인다. 이는 암묵적인 뮤텍스(mutex)와 유사한 효과를 제공하며, 공유 자원에 대한 접근을 보호하는 데 적합하다.

4.2 재진입 콜백 그룹 (Reentrant Callback Group)

ReentrantCallbackGroup에 속하는 콜백들은 동시에 병렬 실행이 허용된다. 동일한 콜백 함수의 복수 인스턴스가 동시에 실행될 수도 있다. 이 그룹을 사용할 때에는 개발자가 명시적으로 스레드 안전성(thread safety)을 보장하여야 한다.

5. 실행자와 콜백 그룹의 상호 작용

실행자와 콜백 그룹의 관계는 다음과 같이 정리된다.

Executor
├── Node A
│   ├── MutuallyExclusiveCallbackGroup (기본)
│   │   ├── Subscription Callback 1
│   │   └── Timer Callback 1
│   └── ReentrantCallbackGroup
│       ├── Subscription Callback 2
│       └── Service Callback 1
└── Node B
    └── MutuallyExclusiveCallbackGroup (기본)
        ├── Subscription Callback 3
        └── Timer Callback 2

노드를 생성할 때 명시적으로 콜백 그룹을 지정하지 않으면 기본 콜백 그룹(default callback group)이 사용된다. 기본 콜백 그룹은 상호 배타적 유형이다.

멀티 스레드 실행자에서 위의 구성이 적용되면, Subscription Callback 1과 Timer Callback 1은 상호 배타적이므로 동시에 실행되지 않는다. 반면 Subscription Callback 2와 Service Callback 1은 재진입 그룹에 속하므로 병렬로 동시에 실행될 수 있다. 서로 다른 콜백 그룹에 속하는 콜백들은 항상 병렬 실행이 가능하다.

6. 콜백 그룹의 설정 방법

C++에서 콜백 그룹을 생성하고 구독자, 타이머, 서비스에 할당하는 방법은 다음과 같다.

#include "rclcpp/rclcpp.hpp"

class MyNode : public rclcpp::Node
{
public:
  MyNode() : Node("my_node")
  {
    // 콜백 그룹 생성
    cb_group_mutex_ = this->create_callback_group(
      rclcpp::CallbackGroupType::MutuallyExclusive);
    cb_group_reentrant_ = this->create_callback_group(
      rclcpp::CallbackGroupType::Reentrant);

    // 구독 옵션에 콜백 그룹 지정
    rclcpp::SubscriptionOptions sub_options;
    sub_options.callback_group = cb_group_reentrant_;

    subscription_ = this->create_subscription<std_msgs::msg::String>(
      "topic", 10,
      std::bind(&MyNode::topic_callback, this, std::placeholders::_1),
      sub_options);

    // 타이머에 콜백 그룹 지정
    timer_ = this->create_wall_timer(
      std::chrono::milliseconds(100),
      std::bind(&MyNode::timer_callback, this),
      cb_group_mutex_);
  }

private:
  void topic_callback(const std_msgs::msg::String::SharedPtr msg) { /* ... */ }
  void timer_callback() { /* ... */ }

  rclcpp::CallbackGroup::SharedPtr cb_group_mutex_;
  rclcpp::CallbackGroup::SharedPtr cb_group_reentrant_;
  rclcpp::Subscription<std_msgs::msg::String>::SharedPtr subscription_;
  rclcpp::TimerBase::SharedPtr timer_;
};

7. 콜백 그룹 설계 시 고려사항

콜백 그룹의 설계는 시스템의 동시성(concurrency)과 안전성(safety) 사이의 균형을 결정한다. 다음의 원칙을 준수하는 것이 권장된다.

설계 원칙설명
공유 자원 보호동일한 자원에 접근하는 콜백들은 상호 배타적 그룹에 배치한다.
독립 연산 분리상호 의존성이 없는 콜백들은 재진입 그룹 또는 별도의 상호 배타적 그룹에 배치하여 병렬성을 확보한다.
교착 상태 방지상호 배타적 그룹 내에서 동기적 서비스 호출을 수행하면 교착 상태(deadlock)가 발생할 수 있으므로 주의하여야 한다.
실시간성 보장실시간 제약이 있는 콜백은 장시간 차단이 발생할 수 있는 콜백과 분리된 그룹에 배치한다.

특히 교착 상태 문제는 빈번하게 발생하는 설계 오류이다. 상호 배타적 콜백 그룹 내에서 동기적 서비스 클라이언트 호출을 수행하면, 서비스 응답을 처리하는 콜백이 같은 그룹의 잠금에 의해 실행되지 못하여 무한 대기 상태에 빠지게 된다. 이를 방지하기 위하여 비동기적 서비스 호출을 사용하거나, 서비스 클라이언트를 별도의 콜백 그룹에 할당하여야 한다.

8. 실행자와 노드 조합 패턴

하나의 실행자에 복수의 노드를 등록하여 단일 프로세스 내에서 여러 노드를 실행할 수 있다. 이를 컴포넌트 기반 구성(component composition)이라 한다.

rclcpp::executors::MultiThreadedExecutor executor;
auto node_a = std::make_shared<NodeA>();
auto node_b = std::make_shared<NodeB>();
executor.add_node(node_a);
executor.add_node(node_b);
executor.spin();

이 패턴은 프로세스 간 통신 오버헤드를 제거하고, 동일 프로세스 내에서 제로 카피(zero-copy) 메시지 전달을 가능하게 하여 통신 지연을 최소화한다. 로봇 행동 제어 시스템에서는 센서 처리 노드, 행동 결정 노드, 액추에이터 명령 노드를 단일 실행자에 구성하여 제어 루프의 지연을 줄이는 전략이 널리 사용된다.
