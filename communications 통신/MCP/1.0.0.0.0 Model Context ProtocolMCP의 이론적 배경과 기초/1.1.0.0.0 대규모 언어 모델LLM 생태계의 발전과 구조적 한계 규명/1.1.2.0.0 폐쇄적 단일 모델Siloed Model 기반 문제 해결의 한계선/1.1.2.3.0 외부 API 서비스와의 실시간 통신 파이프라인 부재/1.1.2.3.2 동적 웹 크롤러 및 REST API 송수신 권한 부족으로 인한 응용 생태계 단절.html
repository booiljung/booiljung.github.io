1.1.2.3.2 동적 웹 크롤러 및 REST API 송수신 권한 부족으로 인한 응용 생태계 단절

폐쇄적 단일 모델(Siloed Model)이 기업의 복잡한 요구사항을 소화하지 못하고 지식의 갈라파고스(Galapagos)로 고립되는 현상은 텍스트 생성 엔진의 ‘송수신 권한(Network Autonomy)’ 부재에서 그 절정에 달한다. 현대의 거의 모든 유의미한 비즈니스 팩트(Fact)는 실시간으로 요동치는 주식 호가창, 초 단위로 업데이트되는 기상 네트워크, 다이나믹하게 렌더링(Rendering)되는 경쟁사의 웹페이지 구조 등 동적(Dynamic)인 HTTP 트래픽 위를 흐른다. 그러나 순수 트랜스포머(Transformer) 디코더 모델은 자신에게 던져진 프롬프트(Prompt)라는 단일 입력 채널과 생성된 텍스트라는 단일 출력 채널에만 영구적으로 결박되어 있다.

사용자가 “현재 A사의 웹사이트 메인 배너 이벤트를 분석하고, B사의 공개된 REST API를 호출하여 최신 상품 단가와 비교해 보라“고 질의했을 때, 언어 모델 아키텍처 내부에는 이러한 다형적인(Polymorphic) 통신을 수행할 수 있는 어떠한 톱니바퀴도 존재하지 않는다. 언어 모델은 스스로 브라우저를 띄워 자바스크립트(JavaScript)로 구동되는 동적 웹페이지를 렌더링하고 DOM(Document Object Model) 트리를 크롤링(Crawling)할 수 없다. 또한, 특정 REST API의 엔드포인트 헬스 체크(Health Check)를 수행하거나, 지수 백오프(Exponential Backoff) 알고리즘을 적용하여 실패한 패킷을 재시도하는 복종적인 웹 클라이언트의 역할도 결코 해낼 수 없다.

이 모델 중심적인 ’소켓(Socket)의 부재’는 곧 무수한 응용 생태계와의 철저한 단절을 의미한다. 언어 모델을 감싸고 있는 호스트(Host) 애플리케이션이 외부 세계와의 통신을 전담하는 프록시(Proxy) 역할을 대신해주지 않는다면, 모델의 추론 능력은 그저 화려한 언어유희에 그치고 만다. 과거에는 이것을 극복하기 위해 플러그인(Plugin)이라는 미봉책이 난립했으나, 이는 각 언어 모델 벤더(Vendor)마다 각기 다른 포맷과 인증 체계를 요구함으로써 또 다른 형태의 무질서와 보안 취약점을 초래했다는 비판을 면하기 어렵다. 통신 인터페이스가 파편화됨에 따라 개발자들은 A 모델용 웹 크롤러, B 모델용 REST 클라이언트를 중복으로 개발해야 하는 이른바 ’플러그인 지옥(Plugin Hell)’에 빠지게 되었다.

이 거대한 소통의 장벽을 허물고 언어 모델에게 외부 세상의 맥박을 전달하기 위해서는, 모델과 외부 클라이언트(브라우저, API 서버 등) 사이에서 일어나는 비동기적(Asynchronous)이고 다형적인 통신을 중재할 수 있는, 언어 모델과 호스트 간의 철저하게 독립되고 범용적인 통신 프로토콜이 요구된다. 모델에게 ’행위(Doing)’의 책임을 지우는 대신, 오직 ’의사결정(Reasoning)’에만 집중하게 만들고, 실제 HTTP/REST 송수신과 크롤링은 보안 통제하에 놓인 외부의 클라이언트 엔진이 전담하여 결과를 모델의 컨텍스트에 주입하는 방식. 바로 이것이 동률의 응용 생태계를 언어 모델의 뇌 속에 이식하려는 Model Context Protocol (MCP) 아키텍처 체계가 직면한 필수적 전제 조건이다.
