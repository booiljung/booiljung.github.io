1.1.2.1.1 기업 인트라넷 방화벽 내부의 레거시 DB 접근 및 질의 권한 부재

엔터프라이즈 환경에서 언어 모델의 실질적 가치를 급감시키는 가장 거대한 물리적, 아키텍처적 장벽은 바로 ’기업 방화벽(Corporate Firewall)’과 수십 년간 축적된 ’레거시 데이터베이스(Legacy Database)’의 존재에서 비롯된다. 대부분의 핵심 비즈니스 로직과 원천 데이터(Source of Truth)는 외부 인터넷 망과 철저히 격리된 안전한 인트라넷(Intranet) 내부의 오라클(Oracle) RDBMS, 온프레미스(On-premise) 하둡(Hadoop) 클러스터, 혹은 독자적인 ERP 시스템 내부에 깊숙이 저장되어 있다. 폐쇄적 단일 모델(Siloed Model), 특히 퍼블릭 클라우드(Public Cloud) 상에서 SaaS(Software as a Service) 형태로 구동되는 상용 LLM들은 이러한 사내 네트워크 경계(Network Boundary)를 뚫고 들어갈 물리적 경로와 논리적 권한 자체를 구조적으로 결여하고 있다.

단순하게 언어 모델을 웹 검색(Web Search) 기능과 연동한다고 해서 이 문제가 해결되는 것이 아니다. 회사의 작년도 분기별 영업 이익 상세 내역이나 특정 부서의 인사 평가 기준표는 구글 커스텀 검색 인덱스(Index)에 존재하지 않기 때문이다. 사용자가 업무 맥락을 파악하기 위해 모델에게 질문을 던지더라도, 언어 모델은 방화벽 내부의 데이터 저장소들을 향해 SQL을 역추론하여 날리거나(Text-to-SQL) REST API를 독자적으로 호출할 수 있는 접근 제어 권한(Access Control List, ACL), VPN 연결, 내부망 인증 자격 증명(Credentials) 그 어느 하나도 가지고 있지 않다.

더욱 치명적인 것은 질의의 논리적 번역 권한이 부재하다는 점이다. 사내 레거시 DB의 테이블 스키마(Schema)와 컬럼 매핑 구조는 개발자들조차도 사내 매뉴얼을 보아야 이해할 수 있을 만큼 암호화되고 축약되어 있는 경우가 빈번하다(예: 회원의 결제 내역을 조회하기 위해 TB_USR_PAY_HIST_A02 테이블을 조인해야 하는 상황). 언어 모델이 기업의 독자적인 메타데이터(Metadata) 사전에 접근할 수 없다면, 설령 네트워크가 뚫려있다 한들 사용자의 “최신 결제 내역“이라는 자연어 질의를 정확한 DB 쿼리로 변환해내는 것은 완벽하게 불가능하다.

결국, 언어 모델 시스템이 엔터프라이즈의 레거시 DB와 호흡하기 위해서는 모델 스스로가 방화벽을 뚫고 쿼리를 던지도록 내버려 두는 무책임한 아키텍처를 버려야 한다. 대신, 인트라넷 내부망에 안전하게 상주하면서 사내 데이터베이스 시스템들과 직접 통신 가능한 신뢰할 수 있는 에이전트 서비스(데이터 프록시)를 배치해야 한다. 그리고 클라우드의 언어 모델은 이 내부 데이터 프록시에게 “이러한 정보가 필요하니 검색해서 결과를 안전하게 반환해 달라“고 요청하는 이원화된 시스템 설계가 요구된다.

방화벽 내부의 시스템과 통신 채널을 확립하고, 어떠한 데이터를 어떤 툴(Tool) 함수로 노출시킬 것인지 엄격히 통제하는 이 복잡한 프로토콜 계층이 부재했던 것이 과거 LLM 도입의 최대 실패 원인이었다. 바로 이 지점에서 클라이언트(LLM 도구)와 방화벽 내부의 서버(데이터 커넥터) 간에 표준화되고 안전한 양방향 통신 규격을 정의함으로써, 모델의 추론 엔진에 레거시 DB의 숨결을 불어넣으려 하는 시도성 프로토콜로서 Model Context Protocol (MCP)의 핵심적 존재 이유가 또렷하게 부각된다.
