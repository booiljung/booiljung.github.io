1.1.2.2.2 메모리(Memory) 비존속성에 따른 단발성 대화형 에이전트의 한계점

고차원적 복합 추론이 실패하는 또 다른 결정적 이유는 언어 모델(LLM) 아키텍처가 지닌 태생적인 ‘메모리 비존속성(Statelessness)’ 때문이다. 컴퓨터 논리 구조의 관점에서 볼 때, 거대한 가중치 파일 덩어리인 딥러닝 모델에는 런타임(Runtime) 상태를 영구적으로 기록하고 업데이트할 수 있는 RAM(Random Access Memory)이나 하드 드라이브가 존재하지 않는다. LLM은 본질적으로 입력된 프롬프트(Prompt) 텍스트를 통계적으로 변환하여 출력 텍스트를 반환하는 거대한 순수 함수(Pure Function)에 불과하며, 매 호출은 이전 호출에 대한 어떠한 기억도 공유하지 않는 단발성(Stateless) 트랜잭션이다.

우리가 흔히 챗봇 인터페이스에서 모델이 맥락을 기억하는 것처럼 느끼는 이유는, 클라이언트(프론트엔드 앱)가 사용자와 모델이 나눈 이전 대화 기록 전체를 복사하여 매번 새로운 프롬프트에 투박하게 욱여넣고(Context Stuffing) 있기 때문이다. 그러나 이러한 얄팍한 트릭은 엔터프라이즈 환경의 장기적이고 복합적인 워크플로우를 소화하기에는 턱없이 부족하다.

프로젝트 기간이 수개월에 달하는 소프트웨어 개발 보조 에이전트를 상상해 보라. 이 에이전트는 사용자가 선호하는 코딩 스타일(예: “항상 타입스트립트 엄격 모드를 사용할 것”), 과거에 실패했던 접근 방식(예: “A 라이브러리는 버전 충돌이 있었으니 B 라이브러리를 쓸 것”), 현재 진행 중인 마일스톤의 목표 등 무수한 메타데이터를 영구적으로 기억하고(Persistent Memory) 상황에 맞게 꺼내 써야 한다. 컨텍스트 윈도우(Context Window)가 아무리 늘어난다 한들, 수개월 치의 상호작용 로그를 매번 API 호출마다 모델에 전송하는 것은 연산 토큰 낭비이자 통신 병목을 유발하는 지극히 비효율적인 설계이다. 더욱이 컨텍스트 길이에 한계가 도달하면 과거의 핵심 기억부터 밀려나 잘려버리는(Truncation) 현상을 피할 수 없다.

따라서 ‘기억’ 능력이 없는 단일 언어 모델만으로는 과거의 경험을 누적하여 점점 더 스마트해지는 연속적인(Continuous) 문제 해결 파트너를 구축할 수 없다. 진정한 복합 추론을 위해서는 인지 엔진(Model)과 기억 저장소(Memory Layer)가 물리적으로 분리된 아키텍처가 필수적이다. 모델 바깥에 벡터 데이터베이스(Vector Database)나 그래프 데이터베이스(Graph Database)를 구축하여 단기, 중기, 장기 기억을 체계적으로 쓰고(Write) 읽어(Read) 오는 명시적인 메모리 관리 컨트롤러가 도입되어야 한다.

언어 모델이 스스로 과거의 기억을 조회(Retrieve)하거나 중요한 컨텍스트를 메모리에 저장(Store)하도록 도구(Tool)를 부여하는 구조, 즉 비존속적 함수 단위의 모델을 존속적(Stateful) 지능 시스템으로 승격시키기 위해서는 이 외부 저장소들과의 매끄러운 통신이 전제되어야 한다. 이를 위해 언어 모델과 스파스(Sparse)한 로컬/원격 메모리 자원 간의 읽기/쓰기 규격을 획기적으로 통일하고 간소화하려는 목표 아래, Model Context Protocol (MCP)와 같은 보편화된 리소스(Resource) 접근 규약의 필요성이 대두되는 것이다.
