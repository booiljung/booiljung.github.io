1.1.2.2.1 다중 뎁스(Multi-Depth) 질의 시 중간 과정(Intermediate) 정보 소실 및 추론 붕괴

엔터프라이즈 환경에서 마주하는 복잡한 문제들은 대부분 얕은 1차원적 질의응답으로 해결되지 않으며, 여러 계층(Depth)의 원인 분석과 데이터 교차 검증을 요구하는 다중 뎁스(Multi-Depth) 질의의 성격을 띤다. 예를 들어, “지난달 발생한 클라우드 서버 장애의 근본 원인을 추적하고, 이를 해결하기 위해 수정해야 할 쿠버네티스(Kubernetes) 설정 파일을 작성하라“는 요청은 최소 3~4개의 논리적 뎁스(장애 로그 분석 \to 원인 컴포넌트 식별 \to 연관 설정 파일 검색 \to 수정안 도출)를 거쳐야만 비로소 완수될 수 있다.

그러나 단일 대규모 언어 모델(LLM)은 이러한 다중 뎁스 과제를 수행할 때, 각 뎁스에서 생성된 ’중간 과정(Intermediate)’의 정보를 컨텍스트(Context) 내에 온전히 유지하고 구조화하는 데 극도로 취약한 모습을 보인다. 트랜스포머(Transformer) 디코더의 어텐션(Attention) 메커니즘은 본질적으로 컨텍스트 윈도우(Context Window) 내의 모든 텍스트에 확률적 가중치를 분산시키기 때문이다. 논리의 사슬이 길어지고 중간 단계에서 생성된 텍스트의 양이 방대해질수록, 정작 다음 단계 추론에 핵심이 되는 정보의 어텐션 가중치는 희석(Dilution)되는 현상이 필연적으로 발생한다.

결과적으로 모델은 뎁스 1과 2를 성공적으로 수행하여 “DB 커넥션 풀(Connection Pool) 고갈“이라는 정확한 중간 결론을 도출해놓고도, 뎁스 3과 4에 진입하는 순간 그 중요한 중간 결론을 망각(Lost in the middle)해버리곤 한다. 그리고는 엉뚱하게 네트워크 대역폭 부족을 원인으로 상정한 쿠버네티스 수정안을 늘어놓는 어처구니없는 ‘추론 붕괴(Reasoning Collapse)’ 현상을 일으킨다. 인간 개발자가 화이트보드에 중간 도출값을 명확하게 적어두고 다음 스텝으로 넘어가는 것과는 전혀 다른, 단일 생성 행위에 모든 것을 의존하는 모델의 구조적 한계이다.

이러한 다중 뎁스 추론 붕괴를 막기 위해서는 텍스트 생성 버퍼에만 의존하는 아키텍처를 탈피해야 한다. 각 논리적 뎁스가 완료될 때마다 그 결과물(Intermediate State)을 명시적인 데이터 구조(예: JSON 객체 혹은 외부 변수)로 캡슐화(Encapsulation)하여 안전하게 보관하고, 다음 뎁스의 추론을 시작할 때 이 캡슐화된 상태값만을 프롬프트(Prompt)에 명확하게 강제 주입하는 제어 흐름(Control Flow)이 필요하다.

이는 결국 언어 모델을 단일 함수가 아닌, 상태를 유지하며(Stateful) 단계별로 외부 컴포넌트와 통신하는 ‘프로그램’ 코드처럼 오케스트레이션(Orchestration)해야 함을 의미한다. 각 추론 뎁스마다 필요한 도구(Tool)를 호출하여 확인된 사실(Fact)만을 메모리에 적재하고 다음 단계로 나아가는 이 에이전틱(Agentic)한 패턴을 표준화하기 위한 통신 규약, 그것이 Model Context Protocol (MCP)가 해결하고자 하는 다중 뎁스 추론 시스템의 아키텍처적 기반이다.
