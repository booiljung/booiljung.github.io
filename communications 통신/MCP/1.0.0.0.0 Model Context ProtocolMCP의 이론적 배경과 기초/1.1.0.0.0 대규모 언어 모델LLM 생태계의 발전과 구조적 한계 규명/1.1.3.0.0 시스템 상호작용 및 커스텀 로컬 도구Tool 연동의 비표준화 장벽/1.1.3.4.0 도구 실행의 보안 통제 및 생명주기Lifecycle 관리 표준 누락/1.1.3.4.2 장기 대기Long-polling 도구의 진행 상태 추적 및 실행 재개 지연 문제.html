1.1.3.4.2 장기 대기(Long-polling) 도구의 진행 상태 추적 및 실행 재개 지연 문제

엔터프라이즈 환경에서 지능형 에이전트가 호출하는 도구(Tool)들은 항상 수 밀리초(ms) 이내에 계산 결과를 반환하는 단순 수학 연산이나 날씨 조회에 국한되지 않는다. 수 기가바이트(GB)에 달하는 로그 파일의 압축을 해제하고 파싱(Parsing)하거나, 머신러닝 모델의 재학습(Retraining)을 트리거하는 배치 작업(Batch Job) 등은 짧게는 수 분에서 길게는 수 시간이 소요되는 전형적인 ’장기 실행 태스크(Long-running Task)’이다. 그러나 현재의 표준적인 API 기반 함수 호출(Function Calling) 체계는 본질적으로 ‘동기식(Synchronous)’ HTTP 요청-응답(Request-Response) 모델에 얽매여 있어, 이러한 장기 대기 도구의 생명주기(Lifecycle)를 다루는 데 치명적인 결함을 노출한다.

대규모 언어 모델(LLM)이 클라이언트에게 장기 실행 도구의 호출을 지시한 후 10분이 넘어가는 대기 상태(Long-polling)에 진입하게 되면, 웹 기반의 아키텍처에서는 필연적으로 HTTP 타임아웃(Timeout)이 발생하게 된다. 네트워크 연결이 끊어지는 순간, 모델은 자신이 지시한 작업이 현재 50% 정도 진행 중인지, 아니면 메모리 부족으로 중간에 실패(Fail)했는지 파악할 수 있는 모든 연결고리(Context)를 상실한다. 클라이언트 시스템에서 백그라운드 프로세스가 무사히 종료되어 결과값이 도출되었다 하더라도, 이미 연결 창구(Session)가 닫혀버린 모델에게 그 성공 메시지와 페이로드(Payload)를 콜백(Callback) 형태로 전달할 표준화된 역방향 라우팅(Reverse Routing) 방법론이 전무한 실정이다.

상태 추적(State Tracking)의 부재는 에이전트의 워크플로우를 심각하게 마비시킨다. 인간 관리자는 진척도를 보여주는 프로그레스 바(Progress Bar)를 통해 시스템의 상태를 인지하고 인내심을 유지하지만, 상태에 대한 런타임 가시성(Visibility)이 차단된 언어 모델은 조급하게 동일한 도구를 중복해서 재호출(Retry)하는 환각(Hallucination) 발작을 일으키기 십상이다. 이는 결과적으로 호스트 시스템에 동일한 무거운 배치 데이터 작업을 여러 개 띄우는 이른바 ‘도구 호출 폭풍(Tool Call Storm)’ 스파이크(Spike)를 유발하여 인프라 전체를 다운시키는 재앙으로 직결된다.

이러한 지연(Latency)과 단절의 문제를 회피하기 위해 개발자들은 개별 애플리케이션 프레임워크마다 자체적인 비동기(Asynchronous) 태스크 큐(Task Queue, 예: Celery, Redis 기반 메지시 브로커)를 덕테이프(Duct-tape)처럼 덕지덕지 이어 붙이고 있다. 도구를 실행할 때 즉시 작업 ID(Job ID)를 반환하고, 모델이 수시로 폴링을 통해 상태를 조회하게 만드는 이 고비용의 우회로(Workaround)는 에이전트 아키텍처의 복잡도를 극단적으로 팽창시킨다. 도구의 실행(Execution), 상태의 진행(Progress), 그리고 완료 후의 재개(Resume)라는 비동기 생명주기 흐름을 벤더의 파편화된 규격이 아닌, 일원화된 펍섭(Pub/Sub)이나 세션 교환 프로토콜로 보장하는 Model Context Protocol (MCP) 규약의 전면적인 도입이 장기 실행 에이전시 구현을 위한 필수 선결 과제임을 시사한다.
