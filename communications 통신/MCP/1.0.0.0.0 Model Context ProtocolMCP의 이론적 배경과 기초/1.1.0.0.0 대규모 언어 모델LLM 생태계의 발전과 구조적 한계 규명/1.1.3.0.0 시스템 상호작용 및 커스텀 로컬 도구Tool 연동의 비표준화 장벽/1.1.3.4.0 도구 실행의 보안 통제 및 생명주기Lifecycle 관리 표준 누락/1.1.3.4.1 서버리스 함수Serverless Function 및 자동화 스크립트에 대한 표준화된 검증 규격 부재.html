1.1.3.4.1 서버리스 함수(Serverless Function) 및 자동화 스크립트에 대한 표준화된 검증 규격 부재

현대 엔터프라이즈 에이전트 시스템에서 도구(Tool)는 점차 정적인 라이브러리 형태를 벗어나 AWS Lambda, 사용자 정의 파이썬 스크립트(Python Script), 사분면의 컨테이너 환경 등 동적인 서버리스(Serverless) 함수 및 자동화 스크립트로 파편화되어 급증하고 있다. 개발자들은 사내의 잡다한 비즈니스 로직(예: 슬랙 메시지 파싱 후 JIRA 티켓 생성, AWS S3 버킷의 정산 덤프 파일 클렌징)을 마이크로스크립트 조각들로 만들어 대규모 언어 모델(LLM)에 던져주고 있다. 그러나 이처럼 급조되어 모델에게 쥐어지는 동적 스크립트들에 대해, 현재의 함수 호출(Function Calling) 체계는 “그것이 무엇을 하는가“에 대한 메타데이터(Description)만 강제할 뿐, **“그 코드가 검증된(Verified) 안전한 코드인가?”**에 대한 시스템 차원의 검증 규격을 완전히 누락하고 있다.

표준화된 검증 규격의 부재는 가장 먼저 무분별한 의존성(Dependencies) 투입과 코드 품질의 붕괴를 초래한다. 개발 팀 A가 작성한 데이터 변형 스크립트와 팀 B가 작성한 검색 스크립트가 호스트 에이전트의 단일 런타임 내에 로드될 때, 해당 스크립트들이 참조하는 외부 라이브러리(NPM 패키지, Pip 모듈 등)가 충돌하거나 악의적인 패키지로 오염(Supply Chain Attack)되어 있을 가능성을 모델은 물론이고 클라이언트 시스템 자체도 판단할 재간이 없다. 각 스크립트가 실행될 때 네트워크 아웃바운드(Outbound)를 어디까지 허용할 것인지, 로컬 디스크의 어느 경로까지 읽기/쓰기를 허가할 것인지에 대한 정적 분석(Static Analysis) 기반의 명세서(Manifest) 규약이 전무하기 때문이다.

또한 런타임에 모델이 직접 코드를 작성하고(Synthesized Code) 스스로 실행하는 코드 인터프리터(Code Interpreter) 모드나 동적 서버리스 함수 호출 체제 아래에서는 위험이 기하급수적으로 폭발한다. 모델이 작성한 임의의 자동화 스크립트는 컴파일 타임의 보안 스캔이나 인간 개발자의 코드 리뷰(Code Review) 파이프라인(CI/CD)을 완전히 우회하여 프로덕션 환경에서 즉각 구동된다. 만약 그 생성된 스크립트 내부에 할당 해제되지 않는 무한 루프 로직이 섞여 있거나, 인자 값 조작을 통해 SQL 인젝션(SQL Injection)을 유도하는 취약점이 뚫려 있다면, 이는 호스트 인프라 전체의 메모리를 스래싱(Thrashing)하거나 고객 데이터를 유출시키는 재앙으로 이어진다.

이러한 사태를 막기 위해 현재의 개발 플랫폼들은 각자의 입맛에 맞는 컨테이너 격리나 실행 시간 턱(Threshold)을 대충 버무려 놓은 반쪽짜리 방어망에 의존하고 있다. 도구 생태계가 진정으로 투명하고 안전하게 연동되려면, 도구의 패키징 구조, 서명(Signature)을 통한 위변조 검증, 그리고 도구가 리거시 환경에서 소모하는 최대 자원량(CPU/Memory Quota)과 권한(Capabilities)을 사전 선언(Declarative)하고 검증하는 인프라 레벨의 단일한 표준 프로토콜이 절대적으로 시급하다. 어떤 환경에서 실행되든 이 도구 스크립트가 ’안전하게 검증된 규격’에 부합함을 보증(Attestation)할 수 있는 Model Context Protocol (MCP) 기반의 인증 인터페이스가 보안 통제망의 핵심 기둥으로 자리매김해야 하는 이유이다.
