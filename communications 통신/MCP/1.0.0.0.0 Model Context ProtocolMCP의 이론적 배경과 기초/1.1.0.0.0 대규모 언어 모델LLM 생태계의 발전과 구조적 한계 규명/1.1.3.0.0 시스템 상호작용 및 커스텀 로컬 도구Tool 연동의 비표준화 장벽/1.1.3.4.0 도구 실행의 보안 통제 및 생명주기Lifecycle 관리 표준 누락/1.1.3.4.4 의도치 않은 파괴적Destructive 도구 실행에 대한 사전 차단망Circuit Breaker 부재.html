1.1.3.4.4 의도치 않은 파괴적(Destructive) 도구 실행에 대한 사전 차단망(Circuit Breaker) 부재

대규모 언어 모델(LLM)에 기반한 에이전트는 본질적으로 확률적(Probabilistic) 결정 기구이다. 아무리 정교하게 프롬프트 엔지니어링(Prompt Engineering)을 수행하고 시스템 인스트럭션을 강화하더라도, 어텐션 스파이크(Attention Spike)나 컨텍스트 누락으로 인해 모델이 ’비합리적이고 파괴적인 명령어’를 도출할 확률을 수학적 0으로 수렴시킬 수는 없다. 엔터프라이즈 환경에서 이러한 0.1%의 확률적 이상 행동(Anomaly)은 데이터베이스 테이블 삭제(DROP TABLE), 클라우드 컴퓨팅 인스턴스 일괄 종료, 혹은 수십만 명의 고객 대상 자산 매각과 같은 돌이킬 수 없는 재난으로 직결될 수 있다.

마이크로서비스 아키텍처(MSA)나 전통적인 금융 통신 시스템에서는 이러한 비정상적인 파괴적 명령이 하위 시스템으로 전파되는 것을 막기 위해 ‘서킷 브레이커(Circuit Breaker)’ 패턴을 표준적으로 도입한다. 시스템의 오류율이 임계치를 넘거나, 실행 결과의 외연(Side-effect)이 너무 크다고 판단될 경우, 즉각적으로 통신 회로를 끊어(Trip) 더 이상의 피해 전파를 차단하는 릴레이(Relay) 메커니즘이다. 그러나 현재 LLM과 도구 사이를 잇는 단방향의 파편화된 함수 호출(Function Calling) 규격에는 이토록 필수적인 사전 차단망의 개념 설계가 완전히 부재하다.

가장 시급한 결함은 ‘상태 변경(State-mutating)’ 도구와 ‘상태 조회(State-querying)’ 도구를 인프라 차원에서 격리하고 통제하는 시맨틱(Semantic) 프로토콜의 부재이다. 모델이 단순히 날씨 정보를 조회(GET)하는 것과 사용자의 결제 정보를 승인(POST/DELETE)하는 행위는 호스트 시스템 입장에서 궤를 명확히 달리하는 리스크 층위(Risk Tier)를 가진다. 그럼에도 불구하고 모델은 두 도구를 호출할 때 완벽히 동일한 JSON 페이로드 구조를 사용하며, 클라이언트 프레임워크는 이 명령이 읽기 전용(Read-only)인지, 파괴적(Destructive)인지 프로토콜 수준에서 판별(Introspection)할 수 있는 표준화된 태그나 헤더(Header) 명세서를 전달받지 못한다.

결과적으로, 파괴적인 도구 호출이 발생했을 때 이를 사전에 가로채어(Intercept) 인간 관리자에게 최종 승인(Human-in-the-loop)을 요청하거나, 시스템의 일일 트랜잭션 한도를 초과했는지 기계적으로 확인하기 위해서는 각각의 애플리케이션 개발자가 또다시 지저분한 하드코딩(Hardcoding) 필터를 덧대어야만 한다. 도구의 메타데이터 명세서 단계에서부터 해당 도구가 유발할 수 있는 파괴적 영향력(Impact Radius)과 이에 따른 승인 레벨(Authorization Level)을 벤더 독립적으로, 그리고 기계가 판독 가능하게(Machine-readable) 선언하는 프로토콜 계층의 개입 없이는 에이전트의 자율성을 프로덕션 레벨로 격상시키는 것은 불가능하다. 이것이 Model Context Protocol (MCP)이 도구 호출 규격에 엄격한 권한 템플릿과 상태 제어 메커니즘을 이식하려는 핵심적인 배경이다.
