1.1.3.4 도구 실행의 보안 통제 및 생명주기(Lifecycle) 관리 표준 누락

대규모 언어 모델(LLM)이 추론을 넘어 도구(Tool)를 직접 실행하는 ’행동하는 에이전트(Acting Agent)’로 진화함에 따라, 시스템 통합의 초점은 어떻게 모델과 도구를 연결할 것인가에서 ’어떻게 모델의 도구 사용을 통제하고 감독할 것인가’로 급격히 이동하고 있다. 엔터프라이즈 환경에서의 도구 실행은 단순한 함수 원격 호출(RPC)을 넘어 비즈니스 데이터의 변형, 외부 자산의 과금, 그리고 핵심 인프라의 상태 변화를 수반한다. 그러나 현재 LLM 생태계를 지배하는 독자적인 함수 호출(Function Calling) 규격들은 도구의 선언과 입출력 스파이크(Spike)에만 집착할 뿐, 정작 기업 심장부에 에이전트를 배치하기 위해 가장 중요한 ’보안 통제(Security Governance)’와 ‘생명주기(Lifecycle) 관리’ 메커니즘을 완전히 누락하고 있다.

도구의 생명주기 관리란 시스템에 등록된 도구가 언제 인스턴스화되고, 어떠한 권한 컨텍스트(Context) 하에서 실행되며, 타임아웃(Timeout)이나 장애 발생 시 어떻게 안전하게 소멸(Teardown)될 것인지에 대한 엔드투엔드(End-to-End) 파이프라인의 정의를 의미한다. 현재의 느슨하게 결합된 HTTP API 방식이나 벤더 종속적인 툴링 프레임워크들은 도구를 단순히 텍스트를 던지면 응답이 돌아오는 블랙박스(Blackbox)로 취급한다. 모델은 자신이 호출한 도구가 현재 실행 중인지, 무한 루프에 빠져 서버 자원(Memory Leak 등)을 갉아먹고 있는지, 아니면 비정상 종료되었는지 상태(State)를 추적할 수 있는 표준화된 채널을 전혀 제공받지 못한다.

더욱 심각한 것은 도구 실행에 따른 보안 통제(Security Control)의 성긴 그물망이다. 모델이 생성하는 텍스트 페이로드(Payload)는 본질적으로 비결정적(Non-deterministic)이며, 환각(Hallucination) 오류나 프롬프트 인젝션(Prompt Injection) 공격에 의해 언제든 ’의도치 않은 파괴적인 명령어’로 돌변할 위험을 내포하고 있다. 그럼에도 불구하고 현재의 체계는 모델이 제안한 도구 호출 스크립트가 로컬 호스트나 사내망에서 구동되기 전, 이 코드가 안전한지, 권한 범위를 벗어나지 않았는지를 기계적으로 검증(Validation)하는 규율된 샌드박싱(Sandboxing) 표준이 부재하다.

결과적으로 개발자들은 각각의 도구를 구현할 때마다 임시방편적인 권한 검사(Auth Check) 로직을 하드코딩(Hardcoding)해야 하고, 긴 실행 시간을 요구하는 도구의 타임아웃을 앱 레벨에서 수동으로 제어하는 ’스파게티 코드(Spaghetti Code)’를 양산하게 된다. 이는 도구가 10개에서 100개, 1000개로 스케일 아웃(Scale-out)될 때 엔터프라이즈 전체의 보안 아키텍처를 치명적인 위험에 빠뜨리는 뇌관이 된다.

본 절에서는 도구 통합 생태계 내에 보안 및 생명주기 관리의 표준이 부재함으로써 발생하는 핵심적인 기술적 위협 요소를 구체적으로 살펴결명고자 한다.

첫째, 무분별하게 양산되는 서드파티 코드, 즉 서버리스 함수(Serverless Function) 및 자동화 스크립트에 대한 표준화된 검증 규격 부재 현상이다.
둘째, 물리적으로 긴 시간이 소요되는 장기 대기(Long-polling) 도구의 진행 상태 추적 및 실행 재개 지연 문제이다.
셋째, 악의적인 명령어 주입으로 시스템 권한을 앗아가는 권한 탈취(Privilege Escalation) 공격을 방어할 범용 샌드박싱(Sandboxing) 가이드라인 결여 현상이다.
넷째, 데이터베이스 삭제나 트랜잭션 역행 등 의도치 않은 파괴적(Destructive) 도구 실행에 대한 사전 차단망(Circuit Breaker) 부재 문제이다.

이 네 가지 통제망의 결핍은 지능형 에이전트를 통제 불능의 시한폭탄으로 방치하는 결과를 낳는다. 도구를 선언하는 문법을 넘어, 그 도구가 호스트 환경에서 어떻게 탄생하고 동작하며 사멸할 것인지, 그리고 그 과정에서 어떠한 권한의 경계를 준수해야 하는지를 엄격한 프로토콜 명세서에 바인딩(Binding)하는 새로운 패러다임, 곧 Model Context Protocol (MCP)이 요구되는 핵심적인 보안 공학적 관성이 바로 이곳에서 기인한다.
