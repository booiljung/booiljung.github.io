1.1.3.2.2 복잡한 시스템 호출 권한 위임 및 결과값에 대한 모델의 사후 복원(Rollback) 불능

도구를 실행하는 주체가 대규모 언어 모델(LLM) 자신이 아니라 호스트 클라이언트 애플리케이션에 고정되어 있는 단방향 실행 구조는, 엔터프라이즈 환경에서 필수적인 ’트랜잭션(Transaction) 관리’와 ‘권한 위임(Delegation of Privilege)’ 체계를 극도로 취약하게 만든다. LLM은 본질적으로 상태가 없는(Stateless) 아키텍처이므로, 특정 도구를 실행하기 위해 구성한 인자(Arguments) 묶음을 클라이언트에게 넘기는 순간 자신이 내린 결정에 대한 모든 맥락적 통제권을 상실해 버린다.

비즈니스 로직에서 복잡한 시스템 호출은 대부분 순차적인 워크플로우를 따른다. 예를 들어, 사용자의 요청에 따라 “1. 클라우드 스토리지 리소스 할당 \rightarrow 2. 관리자 데이터베이스에 기록 \rightarrow 3. 결제 API 승인“이라는 3단계의 도구 실행 시나리오를 생각해 보자. 모델은 1번 도구의 호출 명세서를 클라이언트에게 전달하고, 클라이언트가 이를 실행한 후 결과를 돌려주면, 그제야 2번 도구를 호출하는 식의 Ping-Pong 통신을 이어간다.

문제는 이 도구 연쇄 호출(Chain of Tool Calls) 과정 중 3번 결제 API 단계에서 네트워크 오류나 권한 부족으로 실패(Failure)가 발생했을 때 두드러진다. 데이터베이스와 시스템 상태를 다루는 강력한 에이전트라면, 즉시 1번과 2번 단계에서 발생한 시스템의 외연(Side-effect)을 철회하고 이전 상태로 되돌리는 사후 복원, 즉 롤백(Rollback)을 수행해야 마땅하다. 하지만 단방향 도구 실행 생태계에서 모델은 클라이언트가 실행한 도구가 구체적으로 호스트 시스템의 물리적 상태를 어떻게 변경했는지에 대한 런타임 가시성(Visibility)을 전혀 갖지 못한다. 모델에게 반환되는 것은 클라이언트가 요약해서 보내주는 에러 텍스트 한 줄뿐이다. 이로 인해 모델은 스스로 트랜잭션을 묶거나(Commit) 복원(Rollback)하는 고차원적인 시스템 통제를 수행할 능력을 원천적으로 박탈당한 상태로 남겨진다.

또한, ’권한 위임’의 관점에서도 심각한 모순이 발생한다. 클라이언트는 모델이 보내온 도구 실행 지시를 무비판적으로 신뢰하여 그대로 호스트 OS에서 실행할 것인지(보안 리스크 극대화), 아니면 매 실행마다 사용자(Human)의 승인(Human-in-the-loop)을 강제할 것인지(자율성 훼손)를 두고 양자택일의 딜레마에 빠진다. 모델이 도구의 실행 결과는 물론 실행 과정 상의 에이전시(Agency)를 온전히 책임질 수 없는 단방향 구조 속에서, 복잡한 시스템의 제어 권한을 AI에게 온전히 이양하는 것은 자살 행위와도 같다.

결국, 에이전트가 단편적인 조회(Read)를 넘어 시스템의 상태를 변형하는(Write/Execute) 진정한 책임 주체로 거듭나기 위해서는, 모델과 자원(Resource) 간에 트랜잭션의 컨텍스트를 유지하고 상태를 동기화할 수 있는 강력한 프로토콜 계층이 삽입되어야 한다. 도구의 실행과 그 결과가 파편화된 HTTP 스트림 사이로 흩어지는 것을 막고, 도구의 생명주기를 프로토콜 규격서로 명시하여 보장하는 MCP(Model Context Protocol) 패러다임이 요구되는 절박한 이유다.
