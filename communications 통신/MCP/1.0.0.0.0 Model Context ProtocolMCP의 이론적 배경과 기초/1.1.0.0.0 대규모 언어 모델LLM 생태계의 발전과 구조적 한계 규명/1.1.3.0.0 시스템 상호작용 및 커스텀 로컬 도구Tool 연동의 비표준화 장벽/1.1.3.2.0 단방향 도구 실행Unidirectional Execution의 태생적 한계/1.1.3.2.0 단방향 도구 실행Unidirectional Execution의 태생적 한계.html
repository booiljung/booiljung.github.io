1.1.3.2 단방향 도구 실행(Unidirectional Execution)의 태생적 한계

함수 호출(Function Calling) 체계를 통해 대규모 언어 모델(LLM)이 외부 세계에 개입할 수 있는 통로가 열렸지만, 현재의 지배적인 구현 방식은 ’단방향(Unidirectional)’이라는 치명적인 구조적 결함을 안고 있다. 이 단방향성이란, 호스트 애플리케이션(Client)이 모델(Server)에게 프롬프트를 보내고, 모델이 그에 대한 응답으로 실행할 도구의 이름과 인자(Arguments)를 반환하는 지극히 수동적이고 순차적인 Request-Response 사이클에 갇혀 있음을 의미한다.

이러한 단방향 도구 실행 모델에서 LLM은 결코 스스로 먼저 움직이는 능동적 주체가 될 수 없다. 모델은 “이 도구를 실행해 달라“는 텍스트 페이로드(Payload)를 만들어 클라이언트에게 던져줄 뿐, 실제로 호스트 운영체제의 프로세스를 띄우거나 데이터베이스에 TCP 커넥션을 맺고 쿼리를 전송하는 물리적인 스위치는 전적으로 클라이언트가 쥐고 있다. 클라이언트가 도구를 대신 실행한 후 그 결과값을 다시 프롬프트에 담아 모델에게 두 번째 Request를 보내기 전까지, 모델은 자신의 인지적 흐름을 강제로 멈추고 멍하니 대기(Blocking)해야 한다.

이러한 메커니즘은 단순한 날씨 조회나 주식 가격 확인 같은 1회성 도구 실행에는 무리가 없으나, 엔터프라이즈 환경에서 요구되는 복잡한 ‘에이전시(Agency)’ 앞에서는 극명한 태생적 한계를 노출한다. 복잡한 시스템 장애를 복구하기 위해 수십 번의 셸(Shell) 커맨드를 입력하고 에러 코드를 분석하며 파라미터를 수정해 나가는 연속적인 탐색(Sequential Exploration) 과정에서 단방향 구조는 엄청난 네트워크 왕복 지연(Round-trip Latency)과 컨텍스트 단절을 유발한다.

더욱 치명적인 것은, 이 모델이 지능형 에이전트가 현실 세계와 맺고 있는 권한과 책임의 불일치를 극대화한다는 점이다. 모델은 자신이 내린 지시(도구 호출 명세)가 호스트 환경에서 어떤 사이드 이펙트(Side Effect)를 일으킬지 실시간으로 추적하거나 통제할 수 없다. 이는 마치 눈과 귀를 가린 뇌가 신경망을 통해 근육에 명령만 시달할 뿐, 그 명령으로 인해 손끝이 불에 닿았는지 여부는 한참 뒤에 보고서 텍스트로 전달받는 것과 같은 기형적인 인지 구조를 낳는다.

본 절에서는 이러한 단방향 도구 실행 패러다임이 만들어내는 구조적 병목과 한계를 다음과 같은 측면에서 심층 분석하고자 한다.

첫째, 모델이 먼저 클라이언트에게 연결을 시도하거나 통제를 가할 수 없는 서버-클라이언트 아키텍처 상의 모델 주도형 역방향(Reverse) I/O 통제의 어려움이다.
둘째, 모델이 직접 시스템을 제어하지 않는 구조에서 발생하는 복잡한 시스템 호출 권한 위임 및 결과값에 대한 모델의 사후 복원(Rollback) 불능 상태이다.
셋째, 물리 세계의 변화를 모델이 스스로 감지하지 못하고 클라이언트의 전달에만 의존해야 함으로써 발생하는 이벤트 주도형(Event-driven) 푸시 알림 수신 불가로 인한 폴링(Polling) 오버헤드 양산 문제이다.

이러한 단방향성의 장벽을 허물고, 모델과 클라이언트, 그리고 로컬 자원 간의 안전한 양방향 통신 채널(Bi-directional Communication Channel)을 열어주기 위해서는 단순한 API 호출(HTTP REST) 구조를 넘어서는 새로운 세션 기반(Session-based)의 표준 연결 규약이 불가피하다. 실시간 양방향 협력 네트워크의 필요성, 이것이 MCP (Model Context Protocol)가 대안으로 부상하는 근본적인 아키텍처적 당위성이다.
