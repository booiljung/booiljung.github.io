1.1.3.2.3 이벤트 주도형(Event-driven) 푸시 알림 수신 불가로 인한 폴링(Polling) 오버헤드 양산

대규모 언어 모델(LLM)을 기반으로 한 자율 에이전트가 현실 세계의 다이내믹한 흐름 속으로 편입되기 위해 극복해야 할 가장 폭력적인 시스템 장벽은 ’시간의 비동기성(Asynchrony of Time)’을 다루지 못한다는 데 있다. 현실의 비즈니스 이벤트—새로운 이메일의 무작위적인 수신, 주식 가격의 급락, 혹은 서버 CPU 사용률의 임계치 돌파 등—는 모델이 프롬프트를 쿼리(Query)하는 시점에 얌전하게 맞춰서 발생해주지 않는다. 이러한 이벤트 주도형(Event-driven) 상황을 포착하기 위해 모델은 즉각적인 ’푸시 알림(Push Notification)’을 수신할 수 있어야 하나, 클라이언트 중심의 단방향 통신 구조는 이 역방향 스트림의 채널을 철저히 틀어막고 있다.

서버에 갇힌 언어 모델은 클라이언트 시스템에서 언제 이벤트가 발생할지 알 도리가 없다. 따라서 클라이언트 애플리케이션 개발자는 모델을 실시간 에이전트처럼 ‘보이게’ 탈바꿈시키기 위해, 백그라운드 스레드를 돌려 일정한 주기(예: 1초, 5초)마다 모델에게 “지금 새로운 로그가 있나 도구로 확인해 봐“라는 강제 틱(Tick) 프롬프트를 무한히 말아올려야 한다. 이를 ’폴링(Polling) 기반 상황 인지’라 일컫는다.

이 무지성 폴링 방식은 클라우드 인프라 전반에 걸쳐 끔찍한 연산 리소스의 낭비, 즉 오버헤드(Overhead)를 양산한다. 이벤트가 발생하지 않은 평온한 상태(Idle State)에서도, 클라이언트는 지속적으로 HTTP 커넥션을 열어 모델 API를 호출해야 하고, 모델은 수천 개의 토큰으로 구성된 시스템 프롬프트와 함수 명세서를 무의미하게 재파싱(Re-parsing)하며 빈 응답을 뱉어내는 헛바퀴를 돌려야 한다. 이는 과부하를 초래할 뿐만 아니라, API 사용료의 기하급수적 폭발을 불러와 엔터프라이즈의 비용 구조를 붕괴시킨다.

더 나아가 폴링은 태생적으로 ’지연(Latency)’을 내포한다. 모델이 이벤트를 확인하는 5초의 간격 사이에 시스템이 다운되어 버리는 골든 타임의 유실은 초단타 거래나 자율주행, 실시간 관제 시스템에서는 결코 용납될 수 없는 치명타이다. 상황이 변했을 때 호스트 클라이언트 측에서 모델 측으로 즉각 이벤트를 쏘아줄 수 있는(Emit) 진정한 의미의 서버 푸시(Server Push) 아키텍처—웹소켓(WebSocket)이나 Server-Sent Events (SSE) 기반의 지속적 연결망—가 부재한 탓이다.

요컨대, 모델이 수동적인 질의응답 자판기에서 능동적인 관제탑으로 변모하기 위해서는 지능망의 인프라가 Request-Response 패러다임을 혁파해야 한다. 데이터나 자원(Resource)의 상태가 갱신(Update)됨과 동시에, 그 알림이 프로토콜의 파이프라인을 타고 모델 혹은 호스트 엔진에 비동기적으로 도달할 수 있는 권장된 스펙(Spec). 이 양방향 이벤트 구독 메커니즘을 표준화하여 폴링의 낭비를 소거하고 극단적 실시간성을 담보하는 통신 인터페이스가 바로 Model Context Protocol (MCP) 아키텍처의 필연적 지향점인 것이다.
