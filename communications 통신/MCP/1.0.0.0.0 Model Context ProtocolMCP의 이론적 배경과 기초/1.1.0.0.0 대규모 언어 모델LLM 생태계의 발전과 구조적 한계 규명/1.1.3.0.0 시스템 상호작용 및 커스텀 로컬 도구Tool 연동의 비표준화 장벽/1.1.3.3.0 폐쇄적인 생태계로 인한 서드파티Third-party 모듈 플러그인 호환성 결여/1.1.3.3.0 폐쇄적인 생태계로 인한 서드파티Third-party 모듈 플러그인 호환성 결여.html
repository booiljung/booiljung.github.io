1.1.3.3 폐쇄적인 생태계로 인한 서드파티(Third-party) 모듈 플러그인 호환성 결여

엔터프라이즈 환경에서 지능형 에이전트 서비스가 진정한 가치를 발휘하기 위해서는, 대규모 언어 모델(LLM) 제조사가 자체적으로 제공하는 기본 도구(예: 웹 검색, 코드 실행기)를 넘어, 외부의 독립적인 개발자나 기업들이 만들어낸 수많은 맞춤형 도구들을 매끄럽게 흡수할 수 있는 ’서드파티(Third-party) 확장 생태계’의 조성이 필수적이다. 이는 과거 스마트폰 산업 폭발의 기폭제가 되었던 앱 스토어(App Store)의 혁명과 정확히 일치하는 메커니즘이다. 그러나 현재 LLM 연동 도구의 생태계는 지독히도 폐쇄적인 모놀리식(Monolithic) 아키텍처에 갇혀, 플러그인(Plug-in) 호환성이 궤멸적인 수준에 머무르고 있다.

이 비극은 1.1.3.1절에서 논의된 ’파편화된 함수 호출 체계’와 궤를 같이하지만, 그 피해의 범위가 개별 클라이언트 애플리케이션 개발자를 넘어 글로벌 오픈소스 커뮤니티와 B2B SaaS 기업 등 광범위한 서드파티 생태계 전체로 전이된다는 점에서 훨씬 치명적이다.

특정 벤더(Vendor), 예를 들어 OpenAI의 GPT 플러그인 규격이나 OpenAI Actions 명세에 맞춰 심혈을 기울여 개발된 훌륭한 사내 데이터베이스 연동 모듈이나 외부 API 호출 애플리케이션 프레임워크는 Google의 Gemini 생태계나 Anthropic의 로컬 컨텍스트에서는 한 줄의 코드도 재사용될 수 없다. 거대 AI 제조사들은 각자의 플랫폼을 ’정원 딸린 감옥(Walled Garden)’으로 설계하여 서드파티 개발자들의 진입 스펙을 통제하고 있으며, 이는 필연적으로 수많은 ’바퀴의 재발명(Reinventing the wheel)’을 강제한다.

이러한 모바일 OS 초기 시절을 방불케 하는 폐쇄성은 서드파티 도구를 개발하려는 의지 자체를 꺾어버린다. 개발자들은 자신의 컴포넌트(Component)가 특정 펌웨어나 모델 버전에만 종속되는 위험을 감수하기를 꺼리며, 결과적으로 엔터프라이즈 기업들은 시장에 이미 나와 있는 훌륭한 솔루션을 두고도 자사 프롬프트 규격에 맞추기 위해 내부 개발(In-house)을 고집해야 하는 깊은 딜레마에 빠진다.

본 절에서는 이러한 폐쇄적인 생태계가 에이전트 확장에 미치는 악영향을 세 가지 측면에서 해부하고자 한다.

첫째, 하나의 모델 제공 플랫폼이 도구 스펙을 독점함으로써 발생하는 단일 플랫폼에 종속된 도구 생태계가 초래하는 소프트웨어 유연성(Flexibility) 저해이다.
둘째, 하루가 다르게 변하는 벤더의 API 변경 스펙에 짓눌려 발생하는 글로벌 오픈소스 개발자 커뮤니티의 오픈 도구 개발을 억제하는 스키마 변동성 문제이다.
셋째, 최신 LLM 시스템이 기존 사내 시스템에 스며들지 못하고 겉돌게 만드는 레거시(Legacy) 코드베이스 통합을 가로막는 폐쇄적 SDK(Software Development Kit) 의존성 현상이다.

도구(Tool)와 모델(Model)을 강하게 결합하려는 이 시대착오적 패러다임을 혁파하고, 마치 웹 브라우저가 어떤 백엔드 서버 언어(Java, Python)에 상관없이 HTTP 표준 하나로 데이터를 주고받듯, 다양한 제조사의 모델과 다국적인 서드파티 도구들을 하나의 표준 언어로 중재할 범용 프로토콜의 필요성, 즉 Model Context Protocol (MCP)이 이 생태계의 구원자로 소환된 배경이 바로 여기에 있다.
