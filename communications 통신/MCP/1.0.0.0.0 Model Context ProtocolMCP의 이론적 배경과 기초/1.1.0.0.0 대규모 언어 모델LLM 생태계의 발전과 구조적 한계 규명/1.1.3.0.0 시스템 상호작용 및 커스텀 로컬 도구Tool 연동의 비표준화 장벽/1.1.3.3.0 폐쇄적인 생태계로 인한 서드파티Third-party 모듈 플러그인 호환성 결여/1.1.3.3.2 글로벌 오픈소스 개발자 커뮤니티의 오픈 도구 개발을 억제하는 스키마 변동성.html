1.1.3.3.2 글로벌 오픈소스 개발자 커뮤니티의 오픈 도구 개발을 억제하는 스키마 변동성

지능형 에이전트 생태계가 진정한 폭발력을 갖기 위해서는 글로벌 오픈소스 커뮤니티의 참여가 절대적이다. 수많은 개발자들이 참여하여 다양한 API, 데이터베이스, 심지어 물리적 IoT 기기까지 제어할 수 있는 수만 개의 ’오픈 도구(Open Tools)’를 만들어내야만, 대규모 언어 모델(LLM)은 비로소 일상의 모든 영역에 침투할 수 있다. 그러나 메이저 AI 제조사들이 주도하는 현재의 폐쇄적이고 독자적인 함수 호출(Function Calling) 체계는 이러한 오픈소스 커뮤니티의 자발적인 기여 의지를 뿌리째 꺾어버리는 거대한 장애물로 작용하고 있다.

가장 치명적인 문제는 제조사들이 일방적으로 주도하는 ’스키마(Schema)의 극심한 변동성(Volatility)’이다. 통상적인 REST API나 POSIX 표준과 같이 수십 년간 안정적으로 유지되는 규약과 달리, 현재의 LLM 도구 명세 스펙은 모델의 마이너 버전 업그레이드에도 수시로 구조가 뒤바뀌고 파라미터 제약 조건이 변경된다. 오픈소스 개발자가 자신의 여가 시간을 쪼개어 특정 벤더의 최신 tools 배열 구조에 완벽히 호환되는 훌륭한 날씨 데이터 연동 플러그인을 GitHub에 공개했다고 하자. 불과 몇 달 뒤 벤더가 컨텍스트 윈도우(Context Window) 최적화를 이유로 스키마 파싱(Parsing) 규칙을 미묘하게 변경해버리면, 이 플러그인은 전 세계 사용자의 로컬 환경에서 일제히 에러를 뿜어내는 ’좀비 코드(Zombie Code)’로 전락하고 만다.

오픈소스 메인테이너(Maintainer)들은 새로운 로직을 개발하는 대신, 각기 다른 벤더(OpenAI, Anthropic, Google 등)의 끝없이 변하는 API 문서를 추적하며 하위 호환성(Backward Compatibility)을 맞추기 위한 지루한 패치(Patch) 버전을 무한정 배포해야 하는 고통에 시달린다. 이는 지속 가능하지 않은 노동 집약적 구조이다. 소프트웨어 생태계에서 ’안정적인 컨트랙트(Stable Contract)’의 부재는 가장 유능한 개발자들조차 해당 플랫폼을 외면하게 만드는 확실한 독약이다.

더 나아가, 이러한 변동성은 오픈소스 도구를 레고 블록처럼 결합하여 더 큰 워크플로우를 만들어내는 ’합성성(Composability)’을 원천 차단한다. 개발자 커뮤니티는 특정 기업의 중앙 집중형 마켓플레이스나 플러그인 스토어의 엄격한 심사(Review) 프로세스와 불투명한 노출 알고리즘에 자신의 지적 재산권을 저당 잡히는 것을 본능적으로 거부한다.

결국, LLM과 도구 사이의 인터페이스가 특정 영리 기업의 사유물로 남아 있는 한, 커뮤니티 주도의 진정한 도구 폭발(Tool Explosion)은 요원하다. 글로벌 개발자들이 단 한 번의 작성(Write once)으로 모든 모델에서 영구히 구동(Run anywhere)할 수 있다는 믿음을 줄 수 있는 권위 있고 중립적인 개방형 표준, 즉 Model Context Protocol (MCP)과 같은 보편적 생태계의 출현이 기술 공동체 내부에서 강력하게 대두되는 것은 필연적인 수순이다.
