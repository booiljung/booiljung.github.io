1.1.3.3.3 레거시(Legacy) 코드베이스 통합을 가로막는 폐쇄적 SDK(Software Development Kit) 의존성

엔터프라이즈 환경에서 지능형 에이전트 시스템의 성패를 가르는 가장 결정적인 요인은, 모델이 얼마나 이성적인가(Reasoning capacity)에 있는 것이 아니라, 그 모델이 수십 년간 축적된 기업의 구형 인프라, 즉 레거시(Legacy) 코드베이스와 얼마나 자연스럽게 융화(Integration)될 수 있는가에 달려 있다. 대다수의 전통적인 B2B 기업들은 이미 Java, C++, Python 기반의 방대한 비즈니스 로직과 사내 프레임워크를 운영하고 있으며, AI 프록시를 도입하기 위해 이 거대한 레거시를 처음부터 다시 작성할 여력은 전무하다.

그러나 현재 주요 LLM 제조사들이 자사의 생태계 확장을 위해 배포하는 도구 통합 방식은 대부분 심각할 정도로 비대한, 이른바 ’폐쇄적 SDK(Software Development Kit) 의존성’을 강제한다. LLM에게 사내 온프레미스(On-premise) 데이터베이스의 CRUD 기능을 인가하기 위해 개발자는 벤더가 제공하는 수백 메가바이트짜리 특정 언어(주로 Python/Node.js)의 SDK 패키지를 레거시 서버에 설치(pip/npm install)하고, 해당 벤더 특유의 데코레이터(Decorator)나 매크로를 기존 함수 위에 주입(Injection)해야만 한다.

이러한 침투적인(Invasive) 통합 방식은 레거시 시스템의 결합도(Coupling)를 기형적으로 높이는 부작용을 낳는다. 수십 년간 독립적으로 무결성을 유지해오던 핵심 비즈니스 모듈이, 갑자기 외부 AI 벤더의 통신 프로토콜과 오류 처리 로직을 직결로 상속받아야 하는 상황에 처하게 된다. 만약 해당 벤더의 SDK에 심각한 보안 취약점 패치가 적용되거나 메이저 버전 업데이트가 발생하면, AI와 전혀 무관한 코어 뱅킹(Core Banking)이나 스마트 팩토리 제어 모듈까지 전체 프로젝트의 의존성을 재해결(Dependency Resolution)하고 재컴파일해야 하는 끔찍한 연쇄 효과(Ripple Effect)를 맞닥뜨리게 된다.

또한, 벤더 SDK에 강하게 결합된 시스템 아키텍처는 기업의 기술 스택을 제한한다. Rust나 Go 컴파일된 고성능 바이너리로 툴을 작성하고 싶어도 벤더가 공식 SDK를 제공하지 않으면 어설픈 API 래퍼(Wrapper) 층을 겹겹이 쌓아 올려야 하며, 이 과정에서 발생하는 런타임 오버헤드는 실시간성을 요구하는 시스템에서 치명적인 병목으로 작용한다.

요컨대, 기존의 훌륭한 사내 코드베이스를 LLM이 사용할 수 있는 도구(Tool)로 자유롭게 노출시키기 위해서는 특정한 프로그래밍 언어나 영리 기업의 무거운 SDK 패키지에 의존해서는 안 된다. 레거시 코드에 어떠한 변형도 가하지 않은 채, 표준화된 텍스트 프로토콜만 입출력 파이프(Stdio 혹은 HTTP SSE)에 살짝 연결하는 것만으로 전 세계의 모든 모델과 즉시 상호작용할 수 있는 가벼운(Lightweight) 브릿지(Bridge), 즉 언어와 플랫폼에 완전히 종속되지 않는 Model Context Protocol (MCP) 규약의 등장이 불가피했던 핵심적인 아키텍처적 배경이다.
