1.1.3.1 언어 모델 개발사(Vendor) 별 파편화된 함수 호출(Function Calling) 체계

대규모 언어 모델(LLM)이 단순한 텍스트 생성기를 넘어 비즈니스 로직을 제어하는 능동적 에이전트(Agent)로 진화하는 데 있어 가장 중추적인 역할을 한 기술은 단연 ’함수 호출(Function Calling)’이다. 함수 호출은 모델이 사용자 프롬프트를 분석하여, 외부 도구(Tool)를 거쳐야만 답변할 수 있다고 판단할 경우, 직접 JSON 형태의 인자(Arguments)를 구성하여 호스트 애플리케이션에 반환하는 메커니즘이다. 이를 통해 LLM은 브라우저 검색, 사내 데이터베이스 쿼리, 캘린더 일정 추가 등 무한한 확장성을 확보하게 되었다.

그러나 이 혁신적인 패러다임은 역설적으로 AI 생태계에 전례 없는 수준의 파편화(Fragmentation)라는 깊은 상흔을 남겼다. 단일 벤더(Vendor)의 모델만 사용할 것이라면 문제가 없지만, 현대의 엔터프라이즈 환경은 특정 모델에 대한 종속성(Lock-in)을 거부하며 비용과 성능의 균형을 위해 여러 모델을 혼용하는 멀티 모델(Multi-Model) 전략을 채택하고 있다.

비극의 시작은 함수 호출이라는 개념이 개방형 표준 기구나 프로토콜 위원회를 거쳐 탄생한 것이 아니라, OpenAI를 필두로 한 거대 AI 기업들이 각자의 API 주도권을 쥐기 위해 경쟁적으로 내놓은 독자 규격에서 출발했다는 데 있다. 이들 메이저 제조사들은 ’어떻게 외부 도구의 스펙을 모델에게 설명할 것인가(Tool Description Schematic)’와 ’설명된 도구를 모델이 어떤 구조로 응답할 것인가(Response Payload)’에 대해 서로 양립할 수 없는 제각각의 체계를 강제했다.

결과적으로 개발자들은 동일한 사내 비즈니스 도구(예: “현재 재고를 확인하는 함수”)를 LLM에게 인식시키기 위해, OpenAI용 JSON 스키마를 짜고, Anthropic용 XML 형식을 흉내 내며, Google Gemini를 위한 또 다른 형태의 딕셔너리 구조를 삼중으로 유지보수해야 하는 처지에 놓였다. 이는 단순한 문법의 차이를 넘어, 각 모델이 도구를 추론하는 방식, 필수 인자(Required Parameter)를 처리하는 로직, 그리고 예외 상황에서 반환하는 에러 메시지 형식 등 아키텍처 전반의 깊은 불일치를 의미한다.

이러한 벤더 주도형의 파편화된 함수 호출 체계는 AI 에이전트의 확장성을 가로막는 가장 거대한 족쇄이다. 도구 연동 로직이 특정 모델의 API 버전에 단단하게 결합(Tightly-coupled)되면서, 애플리케이션의 유연성이 무너지고 기술 부채가 산더미처럼 쌓이게 된다. 이는 마우스나 키보드 같은 범용 주변기기를 PC에 연결하기 위해 컴퓨터 제조사마다 각기 다른 물리적 포트를 고집하던 과거 IT 산업의 암흑기를 떠올리게 한다. 이 소모적인 규격의 바벨탑을 다루기 위한 보편적인 ‘번역 레이어’ 혹은 ’통합 규약’의 필요성, 즉 Model Context Protocol (MCP)가 필연적으로 태동할 수밖에 없었던 역사적 배경이 바로 여기에 있다.
