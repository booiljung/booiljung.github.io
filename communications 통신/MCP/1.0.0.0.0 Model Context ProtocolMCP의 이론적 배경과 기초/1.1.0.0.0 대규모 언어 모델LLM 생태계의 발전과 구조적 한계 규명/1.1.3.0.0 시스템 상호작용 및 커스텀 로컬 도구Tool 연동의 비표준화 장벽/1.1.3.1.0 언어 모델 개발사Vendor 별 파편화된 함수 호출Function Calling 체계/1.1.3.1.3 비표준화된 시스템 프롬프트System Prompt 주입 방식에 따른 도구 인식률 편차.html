1.1.3.1.3 비표준화된 시스템 프롬프트(System Prompt) 주입 방식에 따른 도구 인식률 편차

단순히 도구의 스키마(Schema) 형식을 맞추는 것 이상으로 모형의 성능을 극심하게 요동치게 만드는 보이지 않는 장벽은, 모델에게 “이 도구들을 언제, 어떻게 사용할 것인가“를 지시하는 메타-지침, 즉 ’시스템 프롬프트(System Prompt)’의 주입 및 해석 방식에 얽힌 파편화이다. 대규모 언어 모델(LLM)이 개발자가 의도한 대로 외부 도구(Tool)를 정확한 타이밍에 호출하게 하려면 단순히 함수의 인자 명세서를 나열하는 것만으로는 부족하다. 도구의 선택 맥락, 예외 처리 규칙, 그리고 다중 도구의 순차적 워크플로우를 시스템 차원의 맥락(Context)으로 심어주어야 한다.

그러나 메이저 모델 제조사들은 이 시스템 프롬프트를 다루는 철학과 아키텍처에서 근본적인 차이를 보인다. OpenAI의 GPT 모델들은 messages 배열 내에서 role: "system"을 통해 인스트럭션을 강하게 전역으로 주입(Global Injection)받으며, 학습 데이터 풀 자체가 이러한 시스템-어시스턴트 역할을 충실히 따르도록 미세 조정(Fine-tuning)되어 있다. 따라서 상대적으로 간결한 지시만으로도 제공된 JSON 도구 목록을 민감하게 탐지하고 실행을 결정한다.

반면, Anthropic의 Claude 생태계는 전통적으로 시스템 프롬프트를 별도의 파라미터로 격리하여 다루는 방식을 선호했으며, 안전성(Safety)과 정렬성(Alignment)을 중시하는 아키텍처 특성상 외부 도구의 호출을 보수적으로 판단하려는 경향이 짙다. Claude에게 도구를 적극적으로 사용하게 하려면, 도구의 명세(Description) 내에 “You MUST use this tool when…“과 같이 매우 상세하고 방어적인 추가 프롬프토리아(Prompt engineering)를 명시적으로 삽입해야만 인식률(Hit Rate)의 저하를 막을 수 있다.

더 나아가, Google Gemini나 일부 로컬 오픈소스 모델(예: Llama 3)의 경우, 시스템 프롬프트와 사용자 프롬프트(User Prompt) 간의 어텐션(Attention) 가중치 배분 방식이 전혀 다르다. 이들 모델에서는 프롬프트의 길이가 길어질수록 상단에 선언된 시스템 지침이나 도구의 존재 자체를 망각(Lost in the middle)하는 현상이 훨씬 뚜렷하게 나타난다. 특정 모델에서는 잘 작동하던 워크플로우 지시문이, 다른 모델로 스위칭(Switching)하는 순간 도구를 허구로 지어내어 호출(Hallucinated Tool Call)하거나, 명백히 도구가 필요한 시점임에도 평문 텍스트로 추측성 답변을 내뱉는 치명적인 일탈 행위로 이어진다.

이처럼 비표준화된 시스템 프롬프트 주입 메커니즘은 ’도구 기반 에이전트(Tool-based Agent)’의 행동 예측성(Predictability)을 철저히 무너뜨린다. 개발자는 모델 교체 시마다 스키마를 다시 짜는 고통에 더해, 각 모델의 ’성향(Behavior)’에 맞춰 지시어의 뉘앙스, 길이, 위치를 샤머니즘에 가깝게 깎고 다듬는 끝없는 휴리스틱(Heuristic) 튜닝의 늪에 빠지게 된다. 이는 프롬프트와 도구(Tool), 그리고 자원(Resource)을 명확한 계층으로 분리하고, 각 제조사의 변덕스러운 해석 방식에 의존하지 않는 프로토콜 기반의 인터페이스 명세화 작업(MCP)이 왜 필수 불가결한지를 역설적으로 증명하는 대목이다.
