1.1.1.2.3 긴 문맥(Long-context) 내 정보 압축 과정에서 발생하는 핵심 사실 누락 현상

허위 정보 생성(Hallucination)을 유발하는 세 번째 기술적 기제는 트랜스포머(Transformer) 아키텍처의 핵심인 어텐션(Attention) 메커니즘이 방대한 길이의 문맥(Context)을 처리할 때 노출하는 본질적인 정보 압축 및 누락 현상, 이른바 ‘중간 소실(Lost in the Middle)’ 문제에 기인한다. 최근 대규모 언어 모델(LLM) 생태계는 수십만 방향의 토큰을 한 번에 입력받을 수 있는 롱 컨텍스트(Long-context) 윈도우 지원을 통해 정적 지식의 한계를 극복하려는 양적 확장 경쟁을 벌이고 있다. 그러나 이러한 컨텍스트 윈도우의 물리적 연장은 역설적으로 정보 추출의 정밀도를 훼손하는 부작용을 동반한다.

트랜스포머의 어텐션 메커니즘은 입력된 모든 토큰 쌍(Token Pair) 간의 연관성을 계산하기 위해 연산량이 시퀀스 길이의 제곱(O(N^2))에 비례하여 폭발적으로 증가하는 태생적 구조를 갖는다. 이를 완화하기 위해 다양한 희소 어텐션(Sparse Attention)이나 근사 기법들이 도입되었으나, 본질적으로 모델이 길고 복잡한 문맥을 한정된 크기의 고정된 은닉 상태(Hidden State) 벡터로 정보 압축(Information Compression)하는 과정에서 치명적인 데이터 손실이 발생한다.

수많은 실증 연구와 벤치마크 테스트 결과에 따르면, 언어 모델은 프롬프트(Prompt)로 제공된 긴 문서의 ’최상단(시작 부분)’과 ’최하단(끝 부분)’에 위치한 정보는 비교적 정확하게 주의(Attention)를 기울여 추출해내지만, 수만 토큰의 문서 한가운데(Middle)에 파묻혀 있는 핵심 명제나 수치 데이터는 제대로 인식하지 못하고 지나치거나 완전히 다른 정보로 왜곡하여 출력하는 극명한 ‘중간 소실’ 현상을 보여준다.

사용자의 질의를 해결하기 위해 수십 페이지 분량의 기업 매뉴얼이나 방대한 API 로그(Log) 파일 전체를 언어 모델의 컨텍스트 윈도우에 통째로 쑤셔 넣는 방식(Stuffing)은 극도로 비효율적일 뿐만 아니라, 치명적인 허위 정보 생성을 촉발하는 방아쇠가 된다. 모델은 거대한 정보의 늪에서 길을 잃고, 누락된 핵심 사실 자리를 자신의 확률적 상상력으로 대체하여 그럴듯한 오류성 결론을 도출한다.

이러한 물리적 컨텍스트 처리의 한계는 단순 컨텍스트 윈도우의 기계적 확장이 결코 지식 처리 아키텍처의 궁극적 해답이 될 수 없음을 강력히 시사한다. 오히려 방대한 데이터를 무리하게 한 번에 주입하는 대신, 모델이 질의를 해석하고 현재 단계에서 반드시 필요한 특정 정보 조각(Chunk)만을 외부 시스템으로부터 동적이고 정밀하게 호출(Call)하여 가져올 수 있도록 하는 세밀한 컨트롤 플레인(Control Plane)의 도입이 요구된다. 이것이 바로 로컬 도구(Tool) 연동 체계이자 표준화된 인터페이스인 Model Context Protocol (MCP)가 달성하고자 하는 분산 지식 처리의 핵심 철학이다.
