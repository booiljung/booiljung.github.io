1.2.2.1 검색 증강 생성(RAG) 파이프라인의 보편화와 한계

대규모 언어 모델(LLM) 산업 내에서 외부 컨텍스트를 주입하는 가장 대표적이고 상용화된 체계는 ‘검색 증강 생성(Retrieval-Augmented Generation, 이하 RAG)’ 아키텍처다. 논문 “Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks” 등을 통해 정립된 이 방법론은, 사용자의 질의에 앞서 방대한 전사적 문서망에서 시맨틱(Semantic) 유사도가 높은 정보 블록을 선제적으로 찾아내어 프롬프트에 제공함으로써 모델의 할루시네이션(Hallucination) 리스크를 수학적으로 차단하는 것을 목표로 한다.

0.1 RAG 파이프라인의 보편적 아키텍처 구조

RAG 패러다임은 모델이 스스로 정보를 창조하는 부담을 전적으로 해소하고, 대신 독해(Reading Comprehension)라는 모델의 가장 강력한 장기만을 활용하게끔 강제한다. 현대의 보편화된 프레임워크는 텍스트를 고차원 텐서로 변환하는 임베딩(Embedding) 모델과 빠르고 정밀한 인덱싱을 지원하는 벡터 데이터베이스(Vector Store)의 삼각 편대로 구성된다.

RAG 파이프라인의 물리적 작동 논리는 다음과 같이 요약된다:

  1. 오프라인 색인화(Offline Indexing): 방대한 사내 PDF, 위키피디아, 내부 로그 등을 청크(Chunk) 단위로 분할한 뒤, 임베딩 모델을 통해 Dense 벡터 매트릭스로 변환하여 데이터베이스에 적재한다.
  2. 실시간 검색(Online Retrieval): 사용자가 프롬프트를 전송하면 쿼리 또한 동일한 임베딩 차원으로 투영되며, 코사인 유사도(Cosine Similarity) 혹은 L2 거리 연산 등을 통해 가장 기하학적으로 근접한 Top-K 개의 문서 조각들을 소환(Recall)한다.
  3. 증강 생성(Augmented Generation): 필터링된 K개의 문서 원문을 사용자 쿼리와 병합하여 거대한 프롬프트 덩어리를 생성한 뒤, LLM 엔드포인트에 전송하여 철저하게 참고 형태소에 기반한 최종 답변을 생성한다.

0.2 코사인 유사도 기반 시맨틱 검색의 인지적 맹점(Blind Spot)

그러나 RAG 시스템은 기업과 학계에서 ’만능키’로 보급되었음에도 불구하고, 시스템 설계상 극복할 수 없는 분명한 아키텍처 한계선에 부딪히며 서서히 모순을 노출하고 있다. 가장 뼈아픈 한계는 시맨틱 유사도를 평가하는 임베딩 알고리즘 자체의 구조적 맹점에서 기원한다.

벡터 데이터베이스는 두 문장이 ’의미적으로 얼마나 유사한가’를 측정할 뿐, 논리적으로 상호 배타적인가, 혹은 인과관계를 성립하는가를 판단할 능력이 없다. 예를 들어, 사용자가 2024년 1분기 서버 크래시(Crash) 원인을 제외한 나머지 성공 운영일지를 검색해줘. 라는 배제적(Negative) 질문을 던졌을 때, RAG 시스템의 임베딩 엔진은 오히려 서버 크래시라는 단어의 벡터 공간성에 압도적으로 끌려가 크래시 원인이 담긴 장애 보고서를 최우선(Top-1)으로 검색하여 반환해 버리는 오류를 범한다. 즉, 부정어(Negation)나 조건부 지시(Conditional Logic)를 수학적 벡터의 유클리드 거리가 인지하지 못하면서 검색의 순방향 역설을 생산하는 것이다.

0.3 정적 파일 중심 한계와 Model Context Protocol(MCP)을 향한 도약의 필요성

기존 RAG 아키텍처의 한계 중 산업 파이프라인 도입에 있어 가장 치명적인 제약은 이 시스템이 결국 ’사전에 수동으로 인덱싱 된 정적 데이터(Static Chunked Data)’에 국한되어 있다는 점이다.

    A --> C(단순 문자열 벡터화로 인한 RDBMS/SQL 연계성 단절)
    A --> D(Top-k 조각 파편화 반환으로 인한 전역적 컨텍스트 조망 실패)
    
    B & C & D --> E[한계 돌파를 위한 차세대 통합 모형 요구 대두]
    E --> F((Model Context Protocol <br> 동적 Tool Execution 및 자원 통합 구조망 설계))

다이어그램의 분기점처럼, 마이크로서비스(Microservices)의 실시간 트래픽, 주식 시장의 초 단위 호가창, 상시 변경되는 사용자 접근 권한(ACL) 테이블과 같은 형태의 리소스는 한 번 청킹(Chunking)하여 저장해두는 전통적 RAG 구조 안에 절대 담을 수 없다. 따라서 시스템 엔지니어계는 텍스트 조각을 검색하는 단계를 넘어서서, 직접 SQL 쿼리를 실행해 결과를 들고 오거나 외부 REST API에 통신하여 라이브 상태값을 획득하는 광대한 개념, 즉 도구(Tool) 중심의 외부 체계 편입을 구상하게 된다. 이는 구시대적 RAG 파이프라인의 보편적 한계를 종식하고 범용성 높은 Model Context Protocol(MCP)로 아키텍처 진화를 이룩하는 결정적 원동력이 되었다.
