1.2.2.4 플러그인(Plugin) 기반 도메인 지식 캡슐화 모형

대규모 언어 모델(LLM)의 폭발적인 발전 단계에서, 모델 하나가 세상의 모든 전문 지식을 완벽히 외워내고 실시간으로 모든 동작을 통합 제어하리라는 ’순정형 단일 모델(Siloed Model)’의 환상은 점차 붕괴되었다. 세상의 지식은 도메인(Domain) 별로 파편화되어 있고 각자의 고유한 문법적, 시스템적 보안 규칙이 존재하기 때문이다. 이를 해결하고자 태동한 것이, 특정 전문 영역의 지식과 행동(Action) 수행 능력을 독립적인 모듈 형태로 분리하여 주입하는 ’플러그인(Plugin) 기반 도메인 지식 캡슐화 모형’이다.

0.1 지식의 격리와 캡슐화(Encapsulation) 패러다임

객체 지향 프로그래밍(OOP)에서 모듈의 내부 복잡성을 감추고 외부에 명확한 인터페이스만을 제공하여 의존성을 줄이는 ‘캡슐화(Encapsulation)’ 철학이 LLM 프롬프트 생태계에도 이식되었다.

거대 언어 모델이 수학적 계산(Math Computation), 항공권 예약(Flight Booking), 사내 데이터베이스 SQL 쿼리(DB Querying) 같은 고도로 복잡하고 이질적인 명령들을 그 자체 파라미터 내부에 구겨 넣으려 할 때, 할루시네이션(Hallucination)과 치명적 권한 오남용 에러가 동시다발적으로 촉발된다. 플러그인 모형은 이러한 전문 엔진들을 LLM 외부 공간으로 완벽히 격리(Isolation)시킨다.

  • 수학 특화 플러그인: 울프람알파(WolframAlpha)와 같이 기호 연산을 전문적으로 다루는 수식 엔진. LLM은 “수학 문제 도구 사용설명서” 만을 인지하며, 계산식 문자열만을 던질 뿐 실제 복소수 연산 논증은 전적으로 해당 플러그인의 캡슐 안에서 실행된다.
  • 예약 특화 플러그인: 항공사 예약 시스템과 연결된 모듈. 신용카드 보안 처리와 시간대 조율 로직은 이 플러그인 내에 캡슐화되어 있어 언어 모델은 오직 사용자 의사만 묻고 통과(Pass-through)시키는 역할만을 담당한다.

이를 통해 LLM은 스스로 모르는 지식에 대해 허위 정보를 만들어내지 않고 “이것은 내 권한 밖이나, 저기 연결된 ’날씨 전문 플러그인’의 도구 상자를 열어 물어보겠다“는 식의 고차원적 책임 위임 체계를 구현하게 된 것이다.

0.2 선언적 개방 생태계(Open Ecosystem) 구조의 안착

이 캡슐화 모형의 눈부신 점은, 외부 개발자들이 거대한 LLM 서버 코드를 단 한 줄도 수정할 필요 없이 표준화된 규격 문서(OpenAPI 매니페스트 형식 등)만 스크립트로 정의하여 허브(Hub)에 올리면 즉시 시스템의 두뇌와 결합(Plug-in) 가능하다는 뛰어난 확장성에 있다.

이 선언적 명세서에는 “이 플러그인의 이름은 무엇인가”, “어떤 형태의 입력(Input) 문장을 받아들이는가”, “어떤 결괏값(Response)을 토해내는가“라는 형틀만이 존재한다. LLM은 채팅을 이어 나가던 중 사용자 발화 속에 나타난 의도(Intent)가 특정 플러그인의 스펙 설명서의 묘사와 강력한 유사성을 보인다고 (수학적 가중치 연산을 통해) 수렴될 때 비로소 해당 툴(Tool)을 발동시키는 논리적 라우터(Router)로서 기능한다. 결국 이는 모델 중심의 산업 구조를 서드파티(Third-party) 개발자가 주도하는 수만 종의 플러그인 앱스토어(App Store) 체계로 전환시켜, 초단기 시간에 이질적인 전 세계의 도메인 자원들을 융합할 수 있도록 폭발성을 불어넣었다.

0.3 Model Context Protocol(MCP)를 향한 아키텍처 진화의 역사적 거점

그러나 무분별한 플러그인의 폭발은 곧 통제 불가능한 프랑켄슈타인 괴물의 탄생을 예고했다. 사용자의 입력란 하나에 10개의 플러그인이 경쟁적으로 도구를 호출하려고 마스크(Mask)를 벗고 충돌하며, 심지어 어떤 플러그인은 “내 대답을 최고의 정답이라 선언하라“는 등의 프롬프트 인젝션(Prompt Injection)을 가해 다른 플러그인의 행동을 덮어쓰려 하는 시스템 모순이 발견되었다.

초기 플러그인 모형이 시도했던 ’도메인 지식의 캡슐화’라는 찬란한 철학은 그대로 계승하되, 각각의 플러그인들이 중구난방으로 컨텍스트 메모리를 잠식하고 보안의 허점을 노출하던 규격의 미비함을 극복하기 위해 기술의 주체들은 결국 단 하나의 보편화된 인터페이스를 찾게 되었다. 그것이 곧 Model Context Protocol(MCP)이다. 플러그인 모형이 인공지능이 무기와 팔다리를 가져야 함을 주창한 1세대 실험작이었다면, MCP는 그 무수한 팔다리를 단일 중추신경계 안에서 질서 정연하게 컨트롤할 수 있게 만들어 주는 2세대 진화의 완벽한 매뉴얼이자 차세대 시스템 그 자체로 평가받게 된 것이다.
