1.2.2.4.1 전문가 도메인 지식을 함수 호출(Function Call) 형태로 분리하는 아키텍처

대규모 언어 모델(LLM)이 범용적인 언어 능력의 정점에 도달했음에도 불구하고, 복잡한 수학적 연산이나 특정 사내 시스템의 최신 예약 현황 조회와 같이 정밀성이 요구되는 폐쇄적 전문가 도메인(Expert Domain) 지식 앞에서는 필연적인 한계를 노출한다. 이러한 한계를 극복하기 위해 모델의 신경망 내부 파라미터(Neural Network Parameters)에 모든 도메인 지식을 무리하게 욱여넣어 미세 조정(Fine-tuning)하는 대신, 지식과 연산의 논리를 함수 호출(Function Call) 형태의 인터페이스로 완전히 분리해 내는 아키텍처가 시스템 공학의 핵심 대안으로 자리 잡았다.

0.1 지식의 결합 단절(Decoupling)과 범용 언어 뇌의 독립

언어 모델은 확률 기반의 다음 토큰(Token) 예측 머신이지, 레거시 데이터베이스(RDBMS)의 검색 엔진이나 심벌릭 연산(Symbolic Computation)을 수행하는 계산기가 아니다. 만약 항공권 예매 발권 로직이라는 전문가 도메인을 LLM 내부에 직접 코딩하려고 들면, 항공사의 정책이 바뀔 때마다 파운데이션 모델(Foundation Model) 전체를 재학습(Retraining)해야 하는 끔찍한 비용 구조적 모순에 빠진다.

함수 호출 아키텍처는 이를 완벽히 분리(Decoupling)한다.

  • LLM의 역할 (언어 및 논리 제어): 사용자의 복잡하고 모호한 자연어를 파싱(Parsing)하고, 현재 어떤 의도가 필요한지를 파악하여 적합한 ’도구의 이름’과 그 도구를 작동시키기 위한 ’매개변수(Arguments)’를 JSON 형태로 조립해 내는 추론 라우터(Reasoning Router) 역할에 집중한다.
  • 전문가 도메인의 역할 (독립된 연산 및 실행): 외부 플러그인 서버에 위치한 전문 도메인 API가 모델이 건네준 인자 배열을 받아, 실제 항공권 발권 트랜잭션을 실행하고 그 결정론적(Deterministic) 결괏값을 문자열 형태로 다시 LLM에게 반환(Return)한다.

0.2 선언적 스키마(Declarative Schema) 기반의 인터페이스 규약

이러한 분리 아키텍처가 작동하기 위해서는 언어 모델이 현실 세계에 어떠한 ’함수(외장형 두뇌)’들이 존재하는지 사전에 인지해야 한다. 이를 위해 시스템 엔지니어는 OpenAPI 명세서(Specification) 등과 같이 엄격하게 규격화된 선언적 스키마를 모델의 시스템 프롬프트(System Prompt) 측면에 주입한다.

예를 들어, 환율 계산기라는 전문가 도메인은 모델에게 다음과 같은 스키마로 선언된다.

{
  "name": "calculate_exchange_rate",
  "description": "두 국가 간의 실시간 환율을 반영하여 금액을 계산한다.",
  "parameters": {
    "type": "object",
    "properties": {
      "base_currency": { "type": "string", "description": "기준 통화 (예: USD)" },
      "target_currency": { "type": "string", "description": "변환할 통화 (예: KRW)" },
      "amount": { "type": "number", "description": "변환할 현금 규모" }
    },
    "required": ["base_currency", "target_currency", "amount"]
  }
}

LLM은 내부 지식으로 환율을 예단하는 대신, 위 스키마를 독해한 뒤 대화 문맥 속에서 변수들(USD, KRW, 1000)을 정확히 맵핑하여 추출해 낸다. 이는 자연어 처리(NLP) 스택 위에 가장 프로그래밍 친화적이고 결정론적인 API 통신망을 결속시키는 완벽한 연결 고리로 작용한다.

0.3 Model Context Protocol(MCP) 환경의 동적 함수 위상 기하학(Topology)

과거 단위 플러그인 생태계에서는 모델이 직접 개별 서드파티 서비스의 함수 스키마를 일일이 관리해야 했기에 연결망이 기하급수적으로 복잡해지는 문제가 있었다.

graph TD
    A[사용자 자연어 입력] --> B(LLM 클라이언트)
    B -->|문맥 분석 후 Tool 선택| C{Model Context Protocol 미들웨어}
    C -->|함수 라우팅 A| D[통계 데이터 쿼리 시스템 SQL Domain]
    C -->|함수 라우팅 B| E[기상 관측 위성 데이터 API Domain]
    C -->|함수 라우팅 C| F[사내 인사평가 HR 조회망 Domain]
    
    D & E & F -->|결정론적 Deterministic 연산 결괏값 Return| C
    C -->|통합된 Context 주입| B
    B --> G[고신뢰성 최종 병합 응답 생성]

Model Context Protocol(MCP)은 위 도식처럼 이 무수한 선언적 함수 스키마들을 다루는 클라이언트-서버 간 통합 위상 기하학(Topology)을 제공한다. 수평적으로 끝없이 연결되는 세상의 모든 전문가 지식들을 함수 호출(Function Calling)이라는 단일한 규약 체계 안으로 밀어 넣음으로써, LLM은 백과사전의 형태를 벗어나 전 세계의 분산된 시스템 모듈들을 손가락 하나로 지휘하는 오케스트라의 거대한 마에스트로(Maestro)로 그 진화 방향을 확고히 하게 되었다.
