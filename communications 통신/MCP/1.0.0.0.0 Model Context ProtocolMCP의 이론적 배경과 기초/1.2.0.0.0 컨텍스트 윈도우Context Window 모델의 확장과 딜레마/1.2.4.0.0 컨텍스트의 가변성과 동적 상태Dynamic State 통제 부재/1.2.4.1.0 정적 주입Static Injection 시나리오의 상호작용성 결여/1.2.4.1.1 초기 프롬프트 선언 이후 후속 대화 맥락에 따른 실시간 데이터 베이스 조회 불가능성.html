1.2.4.1.1 초기 프롬프트 선언 이후 후속 대화 맥락에 따른 실시간 데이터 베이스 조회 불가능성

대규모 언어 모델(LLM)을 사내 데이터베이스(DB)와 연결하는 RAG(검색 증강 생성) 아키텍처의 가장 치명적인 약점은, 시스템이 오직 첫 번째 사용자 질의(Initial Query)에만 반응하여 외부 지식을 끌어오는 단발성 사고 구조에 갇혀 있다는 점이다. 즉, 초기 프롬프트 선언 이후 대화의 맥락(Context)이 고도화되거나 전환될 때, LLM이 자율적으로 백엔드 DB에 재접속(Re-connect)하여 부족한 단서를 파헤칠 수 있는 실시간 조회 파이프라인의 부재 현상을 겪게 된다.

0.1 턴(Turn) 기반 통신의 구조적 단절과 “장님 코끼리 만지기”

기존 애플리케이션 프레임워크(예: LangChain의 단순 Retrieval 체인) 하에서 정보의 획득은 철저히 1회성(Single-turn)으로 규정된다.

사용자가 “2023년도 영업팀 인사 평가 결과 요약해 줘“라고 질문하면, 시스템은 그 즉시 DB에서 ‘영업팀’, ’인사 평가’라는 문자열(String)이나 벡터(Vector)를 기반으로 관련 문서 청크(Chunk)들을 1차로 싹쓸이하여 LLM의 프롬프트 상단에 주입(Injection)한다. 모델은 그 제한된 문서를 바탕으로 “A 직원이 가장 높은 최고 등급(S)을 받았습니다“라고 답변한다.

문제는 그다음 턴(Turn)이다. 사용자가 흥미를 느끼고 연이어 “그럼 A 직원이 주로 맡았던 프로젝트의 수익률은 어떻게 돼?“라고 2차 질의를 던질 때, 비극이 발생한다.
이 시점에서 LLM의 컨텍스트 윈도우(Context Window) 속에는 1차 질의 때 주입된 <인사 평가 표> 외에는 아무런 정보가 없다. 이상적인 인공지능이라면 이 질문을 받은 즉시 “아, 이젠 ‘A 직원 프로젝트 수익률’ 이라는 새로운 데이터가 필요하구나“라고 깨닫고 실시간으로 사내 재무 DB에 접속해 새로운 표를 가져와야 한다. 하지만 정적 주입형 아키텍처에서는 모델에게 DB 조회 기능이 위임되어 있지 않으므로, 정보 검색의 파이프라인은 굳게 닫혀버린다. 결국 모델은 “제공된 문서에는 A 직원의 프로젝트 수익률이 나와 있지 않습니다“라고 답변을 포기하거나, 최악의 경우 과거의 확률적 패턴에 의거해 거짓 데이터(Hallucination)를 태연하게 지어내는 치명적 에러를 뿜어낸다.

0.2 추론의 파편화와 쿼리 엔지니어링(Query Engineering)의 과부하

후속 대화 중 실시간 조회가 불가능하다는 구조적 장벽은 엔지니어와 사용자 모두에게 막대한 부하를 떠넘긴다.
시스템이 알아서 부족한 정보를 캐오지 못하므로, 사용자는 매 턴마다 AI에게 ’완결된 형태의 완벽한 쿼리’를 입력해야 한다는 쿼리 엔지니어링(Query Engineering)의 압박에 시달리게 된다.

“A 직원이 맡았던 B 프로젝트의 1분기 수익률을 사내 재무 DB에서 검색한 후, 그가 S등급을 받은 2023년 인사 평가서의 항목과 대조해서 다시 설명해 줘“라고 구구절절하게 명령하지 않으면 챗봇은 말귀를 끊어버린다. 이는 LLM이 도입되었음에도 불구하고, 과거 SQL 쿼리를 직접 작성하여 DB를 뒤지던 시절의 노동과 별반 다르지 않은 경험을 제공하며, 인간과 AI가 티키타카(Ping-pong)를 통해 진리를 탐구하는 대화형 상호작용성(Conversational Interactivity)의 본질을 완벽하게 파괴한다.

0.3 Model Context Protocol(MCP)의 다중 턴(Multi-turn) 리소스 프로비저닝

초기 주입에만 의존하던 기존의 죽은 RAG 구조에 숨통을 트이게 하고, 대화의 매 순간마다 실시간으로 지식의 혈관을 열어주는 것이 바로 Model Context Protocol(MCP) 기반의 동적 도구 연동(Dynamic Tool Integration)이다.

    MCP->>ERP: Fetch Data
    ERP-->>MCP: [Response] 인사 평가서 반환
    MCP-->>LLM: 주입 (Context Injection)
    LLM->>User: "A 직원이 1등입니다."

MCP 환경에서는 LLM 스레드(Thread) 자체가 계속 살아 숨 쉬는 유기체처럼 행동한다. 모델은 사용자와의 긴 대화 속에서 특정 데이터가 부족하다고 인지하는 순간, 즉시 대화 생성을 일시 정지(Pause)하고 MCP 서버에게 tool_calls 요청을 타전한다. 위 스퀀스 다이어그램처럼 1차가 되었든, 100차가 되었든, 후속 대화의 맥락이 바뀔 때마다 MCP 라우터는 실시간으로 엔터프라이즈 DB를 두드려 필요한 정보를 가져와 모델의 컨텍스트 창문을 보수해 준다. 이처럼 지속적이고 제한 없는 실시간 지식 탐사의 자유(Freedom of Exploration)를 부여하는 MCP의 아키텍처적 위엄은, 단발성 검색 엔진의 초라한 한계를 넘어 모델이 진정한 의미의 독립적 지능(Autonomous Intelligence)으로 도약할 수 있도록 하는 가장 강력한 백본(Backbone) 네트워크로 작용한다.
