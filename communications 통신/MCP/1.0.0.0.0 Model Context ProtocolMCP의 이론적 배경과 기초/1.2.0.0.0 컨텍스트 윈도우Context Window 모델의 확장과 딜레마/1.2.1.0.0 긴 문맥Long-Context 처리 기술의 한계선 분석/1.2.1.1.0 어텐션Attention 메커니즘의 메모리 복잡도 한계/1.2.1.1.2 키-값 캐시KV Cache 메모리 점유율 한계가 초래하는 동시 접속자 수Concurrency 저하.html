1.2.1.1.2 키-값 캐시(KV Cache) 메모리 점유율 한계가 초래하는 동시 접속자 수(Concurrency) 저하

대규모 언어 모델(LLM, Large Language Model)의 추론(Inference) 과정에서 어텐션(Attention) 메커니즘은 이전 토큰들의 정보를 기억하기 위해 키-값 캐시(KV Cache, Key-Value Cache)를 활용한다. 이 자동 회귀(Auto-regressive) 생성 방식의 캐싱은 이전 시점들의 벡터 재연산(Recomputation)을 방지하여 추론 속도를 비약적으로 향상시키지만, 컨텍스트 윈도우(Context Window)가 길어질수록 메모리 점유율이 시퀀스 길이에 비례해 선형적으로 증가하는 치명적인 한계를 동반한다. 모델 파라미터(Weight Parameter)가 차지하는 정적 공간 외에, 팽창하는 KV 캐시의 VRAM(Video RAM) 요구량은 GPU 서버가 감당할 수 있는 클라이언트 요청의 수, 즉 동시 접속자 수(Concurrency)를 직접적으로 제한하는 핵심 시스템 병목(System Bottleneck)으로 작용한다.

0.1 KV 캐시의 구조와 메모리 요구량의 수학적 모델링

트랜스포머(Transformer) 아키텍처에서 구동되는 디코딩(Decoding) 단계는 이전 시점까지 생성된 모든 토큰들의 키(Key) 및 값(Value) 벡터를 메모리에 적재하여 쿼리(Query) 벡터와의 어텐션 연산을 수행한다. 시스템에 주어진 레이어의 층수를 L, 한 번에 처리되는 배치 크기(Batch Size)를 B, 토큰 시퀀스의 길이를 S, 어텐션 헤드(Attention Head)의 수를 H, 그리고 각 헤드의 벡터 차원을 D_h라고 할 때, FP16(16-bit Floating Point) 정밀도를 기준으로 요구되는 병렬 KV 캐시 메모리 용량은 다음과 같이 수식화된다.

Mem_{KVCache} = 2 \times 2 \times B \times S \times L \times H \times D_h \text{ (Bytes)}

여기서 첫 번째 배수 2는 각각 상호 독립적인 Key와 Value 벡터의 쌍이 존재함을 의미하며, 두 번째 배수 2는 FP16 자료형이 2 바이트(Bytes) 크기를 갖는다는 공간적 제약을 나타낸다.

예를 들어 현대적인 거대 대규모 언어 모델 시스템에서 100명의 동시 접속자(B=100)가 각각 긴 문서 수준인 약 8,192 토큰(S=8192)의 문맥을 유지하며 질의한다고 가정하라. GQA(Grouped-Query Attention)나 MQA(Multi-Query Attention) 같은 최적화 기법이 적용되지 않았을 때, 상기 수식에 의한 VRAM 요구량은 단일 모델 추론 환경에서도 수백 기가바이트(GB)를 훌쩍 상회하게 되어 상용 GPU의 스케일업 파티션 물리적 한계를 쉽게 초과한다.

2. 메모리 병목으로 인한 동시 접속자 수(Concurrency) 처리 성능 한계

LLM 서빙(Serving) 인프라에서 하드웨어 스토리지의 자원은 크게 세 가지, 즉 가중치 모델 파라미터(Model Weights), 활성화 공간(Activations), 그리고 KV 캐시에 의해 삼분할(Trisection) 된다. 파라미터가 차지하는 시스템 메모리는 고정적 상수 자원(Static Resource)인 반면에, KV 캐시는 사용자 요청의 유입량인 배치 크기와 질문의 길이에 따라 지속적으로 가변 증대되는 동적 자원(Dynamic Resource)의 성격을 지닌다. 컨텍스트 윈도우가 극단적으로 확장될 때 발생하는 주요 딜레마는, 긴 문맥을 단절 없이 처리할수록 한정된 GPU 가용 버퍼 내에서 KV 캐시 블록이 차지하는 비중이 폭증하여 시스템이 다룰 수 있는 동시 처리 배치 크기(Batch Size)가 급격하게 축소된다는 점이다.

배치 크기 B가 줄어들게 되면 하드웨어의 메모리 대역폭(Bandwidth) 활용도가 시스템 설계 한계치 이하로 떨어지고 텐서 연산기(Tensor Core)의 효율성이 낮아져, 단위 시간당 연산 처리량(Throughput)이 매우 감소하며 결과적으로 다수의 동시 접속자 당 평균 대기 시간(Latency)이 기하급수적으로 악화된다. 이와 관련하여 논문 “vLLM: Easy, Fast, and Cheap LLM Serving with PagedAttention“에서는 기존 LLM 서빙 백엔드 소프트웨어에서 KV 캐시가 물리 메모리의 지엽적 단편화(Memory Fragmentation) 상태를 유발하고, 실제 활용 가능한 VRAM 가용 구획 공간을 비효율적으로 낭비시킨다는 구조적 맹점을 명확히 지적하였다.

3. 컨텍스트 길이와 동시 접속자 트래픽 용량의 트레이드오프(Trade-off)

아래 머메이드(Mermaid) 도표는 제한된 서버 가용 자원을 제어하는 서빙 프레임워크 아키텍처가, 문맥 길이의 연속적 증가 상황을 마주했을 때 겪게 되는 자율적 상충 관계의 흐름도(Flowchart)를 체계적으로 나타낸다.

    D --> E[메모리 파편화 및 힙 영역 가용 공간 고갈 확산]
    E -->|시스템 OOM 방지 기제 작동| F[허용 가능한 최대 멀티-배치 크기 B 강제 축소]
    F --> G[동시 접속자 수 Concurrency의 치명적 처리율 저하]

위의 위상 수학적 다이어그램에서 증명하듯이, 사용자가 더 방대한 논문 전문, 코드 베이스, 로그 기록을 모델에 입력함으로써 어텐션 연산을 수행하고자 할 때 시스템은 단일 사용자의 세션 보장을 위하여 전체 노드의 접속 트래픽 수용 한계선을 필연적으로 양보해야만 한다.

여기에 다중 통신 인터페이스인 Model Context Protocol(MCP)의 운용 조건이 결합되면 시스템 파괴적 성향은 더욱 증폭된다. MCP의 분산 시스템 아키텍처 상에서는 LLM 추론 코어가 지리적으로 분리된 다수의 외부 자원 서버(Resource Servers)로부터 쉴 새 없이 실시간 동적 컨텍스트를 주입(Injection) 받는다. 이런 이종 통합 지연 시간까지 겹칠 때 클라이언트 세션 간 KV 캐시 블록의 변동폭(Volatility)과 불규칙성은 제어 불가능에 가까워진다. 결국 논리 크기가 제한된 공유 메모리 풀 체계 내에서 페이지 세그먼트를 안정화하는 구조를 고안하지 않는다면, 이 한계는 대규모 기업용 생태계 지향의 MCP 프레임워크 서비스의 확장성 자체를 근본적으로 부정하는 병목 현상을 유발한다.
