1.4.2.2.1. 스트리밍 자원(Resources)과 동적 파라미터 제어기(Tools)의 기능적 이원화

언어 모델(LLM) 기반의 에이전트(Agent)가 외부 시스템과 안전하게 결합하기 위해서는 데이터의 단순한 관찰 행위와 상태를 변이시키는 조작 행위를 엄격하게 분리(Decoupling)해야 한다. Model Context Protocol(MCP)은 이 시스템 공학적 과제를 자원(Resources)과 도구(Tools)라는 극명하게 분리된 두 가지 핵심 규격을 통해 완벽하게 해결하며, 양자 간의 기능적 이원화(Functional Dichotomy)를 구축한다.

0.1 상태 불변성(Immutability)을 지향하는 자원(Resources) 스트리밍

MCP에서 자원(Resources)은 전적으로 ’읽기(Read)’에 특화된 데이터 공급망이다. 이는 REST API의 GET 메서드가 가지는 멱등성(Idempotency)의 철학을 계승하지만, 단발성 응답을 넘어선 스트리밍(Streaming) 메커니즘을 내포한다는 점에서 훨씬 진보적이다.

자원은 본질적으로 호스트 시스템이나 서버의 상태를 변경조차 할 수 없도록 규정된 컴포넌트이다. 글로벌 URI(Uniform Resource Identifier)를 통해 식별되며, 클라이언트는 서버가 노출한 자원의 목록을 나열(resources/list)하거나 특정 데이터를 읽어(resources/read) LLM의 상황 인지(Context Grasping)를 돕는 맥락 정보로 타격 없이 활용한다.

더욱이 자원은 정적 텍스트 반환에만 머물지 않고 능동적인 ‘구독(Subscription)’ 모델을 지원한다. 파일이 수정되거나 데이터베이스 레코드가 갱신되는 등 자원의 상태가 변이하면 서버는 이를 SSE 혹은 Stdio 채널을 통해 이벤트(Event)로 즉각 푸시(Push)할 수 있다. 이로써 외부 세계의 변화가 에이전트의 상황 인지에 실시간으로 반영되는 투명한 지식의 창이 완성된다.

0.2 능동적 부작용(Side-effect)을 통제하는 조향장치: 도구(Tools)

반면 도구(Tools)는 에이전트의 결정(Decision)을 시스템의 조작(Operation)으로 치환하는 액추에이터(Actuator)의 역할을 수행한다. 시스템 백엔드의 상태를 변화시킬 수 있는 어떠한 형태의 ’부작용(Side-effect)’이라도 수반된다면, 그 캡슐화된 로직은 반드시 ’자원’이 아닌 ’도구’로 포진되어야만 한다.

도구의 가장 핵심적인 특징은 실행 전 클라이언트와 서버 간의 엄격한 규약 교환이다. 서버는 각 도구가 수행하는 기능의 설명을 담은 명세와 함께 입력 인자값의 형태를 JSON Schema 포맷으로 직렬화(tools/list)하여 클라이언트에게 제출한다. 언어 모델(LLM)은 이를 분석하여 의도(Intent)를 파악하고, 자신의 추론 과정을 거친 뒤 JSON Schema에 정확히 들어맞는 동적 파라미터(Dynamic Parameters) 객체를 조립하게 된다.

결과적으로 시스템에서 호출(tools/call)되는 것은 단순한 텍스트 덩어리가 아니라, 보안 유효성 검사(Validation)를 거칠 수 있게 엄밀히 정제된 구조체(Structure)가 된다.

0.3 기능적 이원화를 통한 권한 격리망 설계

자원과 도구를 이분법적으로 쪼갠 아키텍처는 에이전트에 통제권을 위임할 때 막강한 보안 제어력을 발휘한다.

관리자는 에이전트에게 특정 깃허브(GitHub) 폴더의 로그나 파일들을 자유롭게 관찰할 무제한적 자원 읽기(Resources) 권한을 허가하면서도, 동시에 코드 변경을 유발할 수 있는 커밋(Commit) 생성 도구(Tools)의 호출권은 거절하거나 사용자 승인(Human-in-the-loop) 윈도우로 라우팅하는 식의 정밀한 심층 방어(Defense-in-Depth)를 수립할 수 있다.
