1.4.2.2.3. 클라이언트/서버 상호 간 지원 기능을 선언하는 핑거프린트 교환(Handshake) 로직

Model Context Protocol(MCP) 생태계는 본질적으로 고도로 파편화된 다기종(Heterogeneous) 환경을 전제로 한다. 다양한 언어(Python, TypeScript, Rust 등)로 각기 다르게 구현된 수많은 클라이언트(Client)와 서버(Server)들이 한 치의 오차 없이 상호작용하기 위해서는 통신 극초기에 서로의 역량을 즉각적으로 파악하고 조율하는 치밀한 동기화 메커니즘이 필수적이다.

이를 위해 MCP는 세션 연결이 시작되는 가장 첫 단계에 핑거프린트 교환(Handshake) 로직을 배치하여, 양방향(Bi-directional)으로 각자의 능력값(Capabilities)과 지원 컴포넌트를 협상(Negotiation)하는 프로토콜 명세 체계를 수립하였다.

0.1 프로토콜 버전 및 메타데이터 상호 검증(Initialization)

클라이언트가 Stdio 파이프나 SSE 연결을 여는 즉시, 가장 먼저 발송하는 JSON-RPC 메시지는 initialize 요청이다. 이 요청 핑거프린트에는 클라이언트 애플리케이션의 이름과 버전(예: Claude Desktop v1.2.0) 및 자신이 이해할 수 있는 MCP 프로토콜의 스펙 버전(Protocol Version)이 포함된다.

서버는 이 요청을 수신하여 자신이 지원 가능한 프로토콜 최신/하위 버전과 대조한다. 호환성이 검증되면, 서버 역시 자신의 시스템 메타데이터(예: GitHub-MCP-Server v2.0)와 프로토콜 버전을 응답 페이로드(Payload)에 담아 반환(InitializeResult)한다. 이 초기 메타데이터의 교환은 통신 암호화에서의 핸드셰이크처럼, 앞으로 전개될 복잡한 RPC 메시지 간의 호환성(Interoperability)을 담보하는 신뢰의 기반이 된다.

0.2 능력값(Capabilities)의 선언과 컴포넌트 동적 바인딩

프로토콜 버전의 일치 여부 확인을 넘어, 생태계적 관점에서 핸드셰이크의 진정한 꽃은 능력값(Capabilities) 객체의 상호 교환이다.

MCP의 3대 핵심 컴포넌트인 **자원(Resources), 프롬프트(Prompts), 도구(Tools)**는 모든 서버가 의무적으로 전부 구현해야 하는 강제 사양이 아니다. 예를 들어 단순한 사내망 읽기 전용 서버라면 자원(Resources) 기능만 활성화하고 도구(Tools) 지원은 비활성화(Disable)시킬 수 있다.

  • 서버 측 능력 선언: 초기화 응답 메시지에 capabilities: { resources: {}, tools: {} }와 같이 자신이 제공할 수 있는 피처(Feature) 목록을 나열한다.
  • 클라이언트 측 능력 선언: 클라이언트 역시 roots: { listChanged: true }와 같이 서버에게 특정 기능(예: 사용자의 작업 디렉터리 경로 제공)을 지원할 수 있음을 능동적으로 알린다.

클라이언트 에이전트는 서버가 반환한 이 핑거프린트를 바탕으로 자신의 런타임 UI나 추론 알고리즘을 즉석에서 재구성(Reconfigure)한다. 서버가 프롬프트를 지원하지 않는다고 선언했다면 해당 메뉴를 렌더링하지 않으며, 도구를 지원한다면 다중 도구 호출(Parallel Tool Use) 분기를 활성화하는 식의 스마트 라우팅이 이뤄진다.

0.3 통지 및 이벤트 구독(Notification & Subscription) 합의망

핑거프린트 교환 단계에서는 정적인 기능의 유무뿐만 아니라, 향후 벌어질 비동기(Asynchronous) 이벤트 스트림에 대한 동의 규약도 설정된다.

서버가 향후 특정 자원의 변경 사항을 서버-주도(Server-driven)로 알림 전송할 수 있도록, 초기화 과정에서 listChanged 혹은 subscribe 능력값을 선언한다. 클라이언트가 이 협상에 동의(initialized 알림 발송)하고 나면, 양측 간에는 성공적인 양방향 상태 전이 머신(State Machine)이 확립되며 본 궤도의 통신, 즉 LLM 프롬프트 조립 및 도구 실행이 이행될 수 있는 안정된 투명망(Transparent Grid)이 완성된다.

다음 다이어그램은 클라이언트와 이기종 서버 간에 이루어지는 MCP 초기 핸드셰이크와 능력값 협상 시퀀스를 나타낸다.

sequenceDiagram
    participant C as MCP Client (LLM App)
    participant S as MCP Server (Data Provider)

    Note over C, S: Session Connection Established (Stdio/SSE)
    
    C->>S: 1. initialize 요청<br/>(Client Info, Protocol Version, Client Capabilities)
    
    alt 호환성 검증 실패 시
        S-->>C: JSON-RPC Error (버전 불일치 거부)
    else 호환성 검증 성공 시
        S-->>C: 2. InitializeResult 응답<br/>(Server Info, Protocol Version, Server Capabilities)
    end
    
    Note over C: 수신된 Server Capabilities(Tools, Resources 등) 분석 및 UI/라우팅 동적 바인딩
    
    C->>S: 3. initialized 알림<br/>(핸드셰이크 완료 및 비동기 이벤트 수신 동의)
    
    Note over C, S: 정상 세션 확립 (정상적인 자원 탐색 및 도구 호출 가능)
    
    C->>S: 4. tools/list 등 본 궤도 통신 시작

이와 같이 MCP의 핑거프린트 교환(Handshake) 로직은 각자 다른 벤더(Vendor)의 개발 철학으로 구동되는 파편화된 분산 개체들이 서로 충돌이나 에러 없이 원활하게 톱니바퀴를 맞물리게 하는, 범용 시스템 통합 아키텍처의 탁월한 플러그 앤 플레이(Plug-and-Play) 설계의 정수이다.
