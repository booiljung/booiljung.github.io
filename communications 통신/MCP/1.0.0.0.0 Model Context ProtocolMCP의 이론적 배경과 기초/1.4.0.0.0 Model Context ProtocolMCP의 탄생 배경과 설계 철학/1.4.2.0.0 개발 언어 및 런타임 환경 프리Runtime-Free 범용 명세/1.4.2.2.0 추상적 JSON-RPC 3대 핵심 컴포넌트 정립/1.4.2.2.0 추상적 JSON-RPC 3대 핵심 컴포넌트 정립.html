1.4.2.2. 추상적 JSON-RPC 3대 핵심 컴포넌트 정립

지능형 에이전트(Agent)가 외부 세계(World)와 소통하기 위해 정의된 Model Context Protocol(MCP)의 설계 철학은, 무수히 다양하고 복잡한 세상의 모든 데이터와 기능을 단 세 가지의 추상화된 핵심 컴포넌트로 응축하는 데서 출발한다.

이러한 고도의 추상화 체계는 클라이언트와 서버가 서로 어떤 언어로 작성되었는지, 배후의 인프라가 몽고DB(MongoDB)인지 깃허브(GitHub) API인지를 묻지도 따지지도 않게 만들어주는 ‘런타임 프리(Runtime-Free)’ 사상의 결정체이다. MCP는 모든 통신 페이로드(Payload)를 표준화된 JSON-RPC 2.0 규격에 담아, **자원(Resources), 프롬프트(Prompts), 도구(Tools)**라는 세 개의 큰 기둥으로 분류(Taxonomy)하여 실현한다.

0.1 세계를 묘사하는 정적-동적 데이터의 뿌리: 자원(Resources)

자원(Resources) 컴포넌트는 에이전트가 읽고 해석할 수 있는 세상의 모든 ’데이터(Data)’를 추상화한 개념이다. 로컬 파일 시스템의 마크다운 문서, 데이터베이스의 테이블 스키마, 시스템의 실시간 로그 파일 스트림, 혹은 서드파티 API가 반환하는 거대한 JSON 응답 등 그 형태가 정적이든 동적이든 상관없이 모두 하나의 ’자원’으로 취급된다.

MCP 서버는 자신이 가진 자원을 고유한 URI(Uniform Resource Identifier, 예: mcp://database/schema) 형태로 노출하고, 클라이언트는 해당 URI를 향해 JSON-RPC resources/read 메서드를 발송하여 텍스트나 이진(Binary) 데이터를 스트리밍받는다. 이 컴포넌트는 본질적으로 ‘읽기 전용(Read-only)’ 속성을 지니며, 어플리케이션의 상태(State)를 변경시키지 않는 안전한 컨텍스트 주입망으로 기능한다.

0.2 컨텍스트의 조립과 상호작용의 지휘 템플릿: 프롬프트(Prompts)

프롬프트(Prompts) 컴포넌트는 사용자와 언어 모델(LLM) 간의 대화의 창(Window)을 조율하는 재사용 가능한 템플릿(Template)이다.

기존 개발 환경에서는, 호스트(Host) 측 코드가 일일이 ’서버 에러 로그를 분석해 줘’와 같은 인스트럭션(Instruction) 텍스트를 구성해야 했다. 그러나 MCP는 이를 서버(Server) 단으로 이관하여 추상화한다. 서버가 미리 ["error-analysis"]와 같은 프롬프트 템플릿을 정의하여 클라이언트에게 통지하면, 클라이언트는 JSON-RPC prompts/get 메서드를 통해 해당 템플릿과 실행 시 결정될 변수(Arguments)들을 서버에 건네어 완성된 거대 프롬프트 메시지를 반환받는다.

이를 통해 프롬프트 엔지니어링의 로직이 클라이언트 런타임에서 완벽히 분리(Decoupling)되어 서버 측에 중앙 집중되는 아키텍처적 유연성을 확보한다.

0.3 세계에 작용하는 액추에이터(Actuator): 도구(Tools)

세상을 읽고(Resources) 지시문(Prompts)을 얻었다면, 에이전트가 실세계에 자신의 의도를 직접 실행(Action)할 차례이다. 도구(Tools) 컴포넌트는 LLM이 호출할 수 있는 ’실행 가능한 함수(Callable Functions)’를 의미한다.

날씨 API 조회, 사내 이메일 전송, 데이터베이스 내 테이블 레코드 삭제(DELETE) 등, 애플리케이션의 상태를 동적으로 변화시킬 수 있는 **‘부작용(Side-effect)’**을 지닌 모든 연산이 도구로 분류된다. 서버는 JSON 스키마(JSON Schema)를 통해 자신이 지닌 도구의 이름과 목적, 매개변수(Parameters)의 데이터 타입을 엄격하게 정의하여 클라이언트에게 브로드캐스팅한다. LLM은 이를 전제로 인텐트(Intent)를 직렬화하고, 호스트 클라이언트는 JSON-RPC tools/call 레이어를 통과해 서버의 실제 기능(Function)을 격발시킨다.
