1.5.2.1.2. 서버 주도(Server-driven) 자원 갱신 알림 푸시(Push Notification) 비동기 수신망 구축

지능형 에이전트가 외부 세계와 의미 있는 상호작용을 하려면, 세계의 상태 변화를 실시간으로 인지해야 한다. 구형 통신망에서 이를 해결하는 방식은 앞서 비판한 웹훅(Webhook)처럼 “에이전트가 서버용 문을 열어두고 기다리거나”, 반대로 에이전트가 1초마다 서버에 “데이터가 변했느냐?“라고 묻는 악명 높은 무한 폴링(Polling) 뿐이었다.

Model Context Protocol(MCP)은 영구적으로 연결된 상태 보전(Stateful) 파이프라인의 이점을 극대화하여, 무거운 오버헤드 없이 서버 측(Server)이 클라이언트(Agent)에게 즉각 알림을 쏘아댈 수 있는 서버 주도형 비동기 푸시(Server-driven Push Notification) 수신망을 프로토콜 내부에 아키텍처적으로 내재화(Internalization)했다.

0.1 자원(Resource) 구독(Subscribe)과 푸시의 원리

MCP 환경에서 에이전트는 특정 자원(예: 깃허브의 특정 레포지토리, 혹은 실시간 로그 파일)이 변동될 때 그 내용을 알고 싶으면, 서버에 resources/subscribe라는 JSON-RPC 메서드를 호출하여 구독 의사를 밝힌다.

일단 구독이 맺어지면, 에이전트는 더 이상 폴링을 하지 않고 본연의 다른 사고(Reasoning) 워크플로우에 집중한다. 이후 서버 측 파일 시스템이나 데이터베이스에 실제 변동(Mutation)이 발생하면, 서버는 쥐고 있던 현재의 연결 스트림(Stdio 파이프나 SSE)을 활용해 notifications/resources/updated라는 알림 이벤트를 에이전트 측으로 즉각 밀어 넣는다(Push).

이 모델이 무상태 웹훅 통신과 대조되는 압도적인 강점은 다음과 같다.

  1. 방화벽 및 인바운드 제약의 완전한 돌파: 에이전트가 외부에 포트를 개방(Port Forwarding)할 필요가 없다. 에이전트가 서버로 향해 먼저 뚫어놓은 아웃바운드(Outbound) 채널을 역으로 타고 서버의 이벤트가 흘러 내려오기 방화벽의 철벽 방어를 유연하게 통과한다.
  2. 컨텍스트(Context) 역추적 비용 제로: 이벤트가 별도의 낯선 HTTP 엔드포인트를 타고 날아오는 것이 아니다. 이미 에이전트가 쥐고 있는 동일한 TCP 세션 파이프로 데이터가 도착하므로, 에이전트 시스템은 난수화된 콜백 ID를 DB에서 역추적하는 등의 뼈아픈 매핑(Mapping) 연산을 수행할 필요가 없다. 즉시 현재 LLM의 컨텍스트 윈도우 스냅샷에 해당 변동 사항을 병합(Merge)하면 끝난다.

0.2 이벤트 루프(Event Loop)의 비동기적 상생

이러한 알림 시스템은 에이전트의 사고 흐름(멀티 스텝 추론)과 완벽한 비동기적(Asynchronous) 상생을 이룬다.
LLM이 긴 코드를 짜고 있는 동안에도, 바탕화면(Background)에서는 통신 스트림을 통해 로그 서버의 에러 발생 알림이나 다른 작업자의 코드 갱신 알림이 끊임없이 버퍼로 흘러들어온다.

에이전트 클라이언트 프레임워크는 이 푸시 알림들을 큐(Queue)에 고이 모아두었다가, LLM의 현재 추론 단계가 끝나는 즉시(Next Tick) 새로운 프롬프트 정보로 주입하여 LLM이 “방금 새로운 에러가 떴군요. 이 부분을 수정해서 다시 코드를 작성하겠습니다“라고 기민하게 반응할 수 있게 만든다.

결국 MCP가 구현한 서버 주도의 비동기 푸시망은, 무상태 통신의 맹목적인 단절감과 폴링의 자원 낭비를 동시에 종식시키며 진정한 실시간 대화형 AI(Real-time Conversational AI)를 구동하는 강력한 심장이 된 것이다.
