1.5.1.2.1. 프론트엔드 최적화 목적의 GraphQL 오버페칭(Over-fetching) 방지 설계와 LLM 탐색 패턴의 불일치

페이스북(Facebook, 현 Meta) 내부의 데이터 페칭(Fetching) 문제를 해결하기 위해 탄생한 GraphQL은 단일 엔드포인트(Single Endpoint)와 클라이언트 주도의 선언적 쿼리(Declarative Query)라는 혁신을 통해, REST API가 지니던 고질적인 매개변수 파편화 현상을 훌륭하게 봉합했다. 클라이언트는 자신이 원하는 필드(Field)만을 정확히 선택하여 요청할 수 있으며, 이를 통해 불필요한 메타데이터가 전송되는 오버페칭(Over-fetching)을 완벽하게 차단함으로써 모바일 환경과 프론트엔드 네트워크 최적화에 지대한 공헌을 하였다.

그러나 인간의 눈으로 볼 화면(View)을 렌더링(Rendering)하기 위해 극도로 계산된 이 ’오버페칭 방지 설계’는, 미지의 데이터 공간을 무작위로 탐색하며 스스로 논증을 구축해 나가는 대규모 언어 모델(LLM) 에이전트의 ‘우회 추론(Heuristic Exploration)’ 패턴과 정면으로 충돌한다.

0.1 엄격한 구조화(Strict Structuring) 모델이 거세한 에이전트의 우연한 발견(Serendipity)

프론트엔드 개발자는 특정 UI 화면에 데이터베이스의 어떤 칼럼(Column)들이 노출될지 사전에 100% 확신할 수 있으므로, 단일한 GraphQL 쿼리 문서를 사전에 작성해 둘 수 있다.
반면 LLM 에이전트가 수행하는 프롬프트 기반의 데이터 분석 지시는 대부분 목표조차 모호한 상태에서 출발한다. 사용자가 “최근 매출이 급락한 원인을 분석해 봐“라고 지시했을 때, 에이전트는 매출 테이블뿐만 아니라 부서별 예산, 마케팅 캠페인 로그, 기상 악화 데이터 등 관련된 모든 레코드를 수평적으로 ‘넓게’ 스캐닝(Scanning)할 필요가 있다.

문제는 GraphQL이 오버페칭을 막기 위해 클라이언트가 명시적으로 요청한 필드 이외의 모든 데이터를 묵살(Silencing)하도록 강제한다는 점이다. LLM은 “매출 급락의 원인“을 찾기 위해 다음과 같은 중첩 쿼리(Nested Query)를 동적으로 작성해야 하지만, GraphQL 스키마에 어떤 숨겨진 연관 필드(예: campaign_delay_days)가 존재하는지 알지 못하기 때문에 이를 쿼리 문자열에 정확히 타이핑해 낼 수가 없다.

# LLM은 미리 스키마 전체를 읽어 구조를 암기하지 않는 이상, 
# 아래와 같은 깊은 중첩(Deep Nested) 구조와 필요한 정확한 필드명을 스스로 조합해 내기 어렵다.
query AnalyzeRevenueLoss {
  company(id: "1") {
    departments {
      sales(period: "last_quarter") {
        total_revenue
        marketing_campaigns {
          budget_spent
          expected_roi
          # 여기에 원인을 밝힐 핵심 필드가 있을지 없을지, 에이전트는 요청 전에 알 수 없다!
        }
      }
    }
  }
}

이처럼 REST에서는 우연히 딸려 온 쓰레기 데이터(Over-fetched Meta-data) 통틀미리 속에 에이전트가 미처 예상치 못했던 핵심 단서가 포함되어 문제 해결의 우연한 발견(Serendipity)을 돕는 경우가 생존했다. 하지만 GraphQL의 지나치게 정교한 **“가져올(Fetch) 필드 목록 제출 의무”**는 데이터 통찰의 지평을 에이전트가 사전에 상상할 수 있는 어휘력 수준으로 좁혀버리는 시야 결손(Blind Spot)을 야기한다.

0.2 동적 쿼리 신서시스(Query Synthesis) 과정의 문법적 오류(Syntax Error) 마찰

더불어 GraphQL은 매우 엄격한 타입 체계(Type System)와 전용 질의어(DSL, Domain-Specific Language) 문법을 갖추고 있다. LLM이 GraphQL 서버와 대화하기 위해서는 단순한 JSON 파라미터를 넘기는 수준이 아니라, 인자(Arguments), 지시어(Directives), 프래그먼트(Fragments) 등이 버무려진 완전무결한 구조적 문자열을 새롭게 합성(Synthesis)하여 제출해야 한다.

토큰 생성 과정에서 단 한 개의 중괄호({})가 누락되거나, 오타가 발생하여 스키마에 존재하지 않는 필드명을 요구할 경우, 서버는 즉시 무자비한 SyntaxError나 ValidationError를 반환하며 응답을 거부한다. 인간 코더라면 IDE의 플러그인이나 자동 완성(IntelliSense)을 통해 쿼리 작성 시점(Build-time)에 이를 방지할 수 있지만, 런타임(Runtime)에 텍스트를 즉석에서 비봉합 해내는 통계적 모델인 LLM에게 이는 치명적인 오류 전파(Error Propagation)의 온상이 된다.

0.3 목적이 어긋난 최적화: UI 페칭(UI-Fetching)과 추론(Reasoning)의 분리

결과적으로 아키텍처 관점에서 보았을 때, GraphQL은 화면에 그릴 데이터를 가장 경제적으로 네트워크를 통해 전달하기 위한 **렌더링 네트워크 브릿지(Rendering Network Bridge)**이지, 이기종 데이터를 자율적으로 관측하고 사고를 확장해야 하는 **추론 에이전트의 지능형 파이프라인(Intelligent Pipeline)**으로는 부적합하다.

Model Context Protocol(MCP)은 바로 이러한 오류를 극복하기 위해, 에이전트가 복잡한 쿼리 문법을 퍼즐처럼 조립하게 두는 대신 서버에 존재하는 특정 형태의 자원(Resources)을 단순히 “읽기명령(Read)“만으로 통째로 가져올 수 있도록 직관적이고 납작한(Flat) JSON-RPC 스펙을 제공함으로써, 에이전트의 귀중한 토큰 예산을 복잡한 통신 문법 작성이 아닌 본연의 논리 추론(Logical Reasoning) 과정에 쏟을 수 있도록 아키텍처적 배려를 구현하고 있다.
