1.5.1.1. REST API 아키텍처 기반 통합 연동의 단점

로이 필딩(Roy Fielding)이 2000년대 초 제안한 REST(Representational State Transfer) 아키텍처는 지난 20여 년간 웹 생태계를 정의하는 절대적인 프로토콜 표준(De facto standard)으로 독점적 지위를 누려왔다. 자원(Resource)을 고유한 URI로 식별하고, HTTP의 기본 메서드(GET, POST, PUT, DELETE)를 통해 해당 자원의 상태(State) 형상을 조작하는 이 직관적인 방식은 인간이 설계하는 분산 시스템에는 더없이 훌륭한 해답이었다.

하지만 자기 참조적(Self-referential)으로 사고하고 비정형화된 태스크를 수행해야 하는 대규모 언어 모델(LLM) 에이전트의 관점에서, REST API 기반의 통합 연동은 다음과 같은 근본적인 형태적 제약과 아키텍처적 결함을 드러낸다.

0.1 고정된 명사 중심(Noun-centric) 엔드포인트의 경직성

REST 아키텍처의 핵심 철학 중 하나는 모든 엔드포인트(Endpoint)가 동사(Verb)가 아닌 명사(Noun), 즉 ’자원(Resource)’을 가리켜야 한다는 점이다. (예: GET /users, POST /orders)
이러한 명사 중심 설계는 텍스트를 요약하거나, 수학적 연산을 수행하거나, 복잡한 파이썬 스크립트를 즉석에서 실행해야 하는 AI 통신망의 ‘도구 호출(Tool Calling)’ 패러다임과 극렬하게 충돌한다.

LLM 에이전트가 수행하고자 하는 행위는 종종 특정 데이터 레코드를 수정하는 것이 아니라, “주어진 데이터 프레임을 분석하여 특정 조건에 맞는 통계 값을 계산해 달라“와 같은 절차적이고 행위 중심적(Behavior-oriented)인 연산이다. 이를 억지로 RESTful 패턴에 맞추어 설계하려면 개발자는 POST /computations/statistics 와 같은 어색한 명사의 우회로(Workaround)를 만들어야 하며, 이러한 디자인의 경직성은 API의 가독성과 확장성을 심각하게 훼손시킨다.

0.2 오버페칭(Over-fetching)과 언더페칭(Under-fetching)의 한계

LLM의 컨텍스트 윈도우(Context Window)는 한정된 자원이며 토큰 하나하나가 곧 비용(Cost)이다. 따라서 외부 구동부로부터 전달받는 데이터는 정밀하게 재단된 순수한 지식의 정수여야 한다. 그러나 REST API는 클라이언트가 응답 페이로드(Payload)의 형태나 깊이를 자유롭게 지정할 수 없는 구조적 한계를 띤다.

  1. 오버페칭(Over-fetching): 사용자가 “A 프로젝트의 최신 커밋 메시지만 알려줘” 라고 지시했을 때, REST API는 커밋 메시지뿐만 아니라 작성자의 아바타 URL, 해시(Hash) 값, 날짜 타임스탬프, 그리고 관계된 수많은 메타데이터를 통째로 반환한다. 이 거대한 JSON 덩어리는 LLM의 컨텍스트를 오염시키고 환각(Hallucination)을 유발하는 주요 원인이 된다.
  2. 언더페칭(Under-fetching): 반대로 “A 프로젝트의 커밋 작성자가 소속된 주요 부서를 알려줘“라는 복합적인 추론 지시가 내려졌을 때, LLM은 GET /commits를 호출한 뒤, 파싱된 작성자 ID를 기반으로 다시 GET /users/{id}/department를 호출해야 하는 지루한 연쇄 호출(Chain Calling)을 겪어야 한다.

이러한 페칭(Fetching) 한계는 LLM 에이전트로 하여금 한 번에 알아낼 수 있는 사실을 수차례의 네트워크 왕복(Network Round-trip)에 걸쳐 알아내게 함으로써 추론 속도 지연과 비용 낭비를 극대화한다.

0.3 무상태성(Statelessness)에 따른 컨텍스트 유지의 어려움

REST 아키텍처의 또 다른 대원칙은 완벽한 무상태성(Statelessness)이다. 서버는 클라이언트의 과거 요청 기록이나 현재 상태를 일절 메모리에 저장하지 않으며, 각각의 HTTP 요청은 그 자체로 완결된 독립적 컨텍스트를 지녀야 한다. 따라서 기존 REST API 생태계에서 에이전트가 “방금 조회했던 그 데이터에 이어서…” 와 같은 맥락 의존적(Context-dependent)인 후속 요청을 보내려면, 이전 단계에서 획득한 모든 조건값과 세션 토큰을 매번 헤더(Header)나 본문(Body)에 욱여넣어 재전송해야 한다.

반면 인간과 AI가 교감을 나누는 채팅 인터페이스 혹은 에이전트 워크플로우는 철저한 상태 유지형(Stateful) 통신이다. 대화의 맥락이 점진적으로 쌓이고(Incremental), 중간에 생성된 중간 연산 산출물이 다음 단계의 입력으로 자연스럽게 흘러가야(Fluid) 한다.

REST API는 매 요청마다 세션을 초기화하고 전체 맥락 파라미터를 강제하기 때문에, LLM 모델이 복잡한 다중 단계 추론(Multi-step reasoning)을 이어나가는 도중 심각한 인지적 과부하(Cognitive Overload)를 일으키는 아키텍처적 장벽으로 작용하게 된다. Model Context Protocol(MCP)은 HTTP의 매 순간 끊어지는 무상태성을 타파하기 위해 Stdio 파이프라인이나 SSE(Server-Sent Events)를 활용항 영속적(Persistent)이고 상태 유지적인 생태계로 진화하게 되었다.
