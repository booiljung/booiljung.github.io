1.5.1.1.2. 엄격한 스웨거(Swagger/OpenAPI) 명세 사전 배포 요구로 인한 동적 도구 생성 지연

기업형 REST API 생태계에서 서비스 제공자와 클라이언트 간의 통합을 매개하는 가장 핵심적인 컴포넌트는 스웨거(Swagger)나 OpenAPI 스펙(OpenAPI Specification, OAS)과 같은 API 명세서(API Documentation)이다. 이 문서들은 어떤 엔드포인트(Endpoint)가 열려 있는지, 파라미터의 타입은 정수형(Integer)인지 문자열(String)인지를 규정하는 엄격한 계약(Contract) 역할을 담당한다. 과거 OpenAI 플러그인 등 초기 LLM 통합 환경 역시 에이전트에게 OpenAPI yaml이나 json 파일을 텍스트로 읽혀 도구(Tool)를 파악하게 하는 방식을 채택했다.

하지만 이 정적 명세 사전 배포(Static Specification Pre-distribution) 구조는 살아 움직이는 유기체와 같은 대규모 언어 모델(LLM) 에이전트 시스템에 치명적인 동적 확장성 저해를 가져왔다.

0.1 수정-배포 파이프라인(Modify-Deploy Pipeline)에 묶인 에이전트 지능

OpenAPI 기반의 REST 환경에서 새로운 기능을 에이전트에게 가르치거나, 기존 도구의 파라미터를 수정하고자 할 때 발생하는 타임라인의 지연 현상을 살펴보자.

  1. 백엔드 수정: 서버 개발자가 백엔드 로직에 새로운 매개변수나 API 경로를 추가한다.
  2. 명세서 빌드(Build): 백엔드 빌드 파이프라인을 돌려 새로운 OpenAPI json 파일을 스캐닝(Scanning)하고 생성한다.
  3. 명세 배포(Deployment): 수정된 OpenAPI 스펙 파일을 정적 웹 서버나 외부 레지스트리(Registry)에 배포한다.
  4. 에이전트 재초기화(Re-initialization): 클라이언트 애플리케이션은 모델 세션을 강제로 종료한 후, 새로운 스펙 파일을 다운로드 받아 파싱(Parsing)하고, 다시 LLM의 시스템 프롬프트(System Prompt) 구조로 변환하여 에이전트를 재기동한다.

이러한 무겁고 지루한 연쇄 파이프라인은, 인간 개발자 조차 새로운 기능 테스트를 꺼리게 만드는 강력한 마찰력(Friction)으로 작용한다. 인공지능이 시스템의 상태를 통제하는 최신 워크플로우에서는 시스템 내부에 새로운 파일 하나가 생성되거나, 사용자 권한이 바뀌었을 때 그 즉시 “이 파일을 읽을 수 있는 새로운 도구(Tool)“가 에이전트 시야에 팝업처럼 떠올라야(Dynamically Generated) 한다. 하지만 사전 배포된 정적(Continuous Static) OpenAPI 스키마에 의존하는 REST 아키텍처는 이 유연성을 원천적으로 박탈한다.

0.2 컨텍스트 윈도우(Context Window) 내부의 스키마 관리 실패

추가적으로, 거대한 OpenAPI 명세 파일을 통째로 에이전트에게 주입하는 방식은 근본적인 토큰 예산 관리의 실패를 야기한다.

범용적인 기업 API 서버는 종종 수십, 수백 개의 엔드포인트를 가진 거대한 OpenAPI 스키마를 발행한다. 이 중 에이전트가 현재 사용자와의 대화 맥락(Context)에서 실제로 필요로 하는 도구는 단 1~2개에 불과할 수 있다. 그러나 OpenAPI 스키마는 덩어리째(Bulk) 제공되므로, 에이전트는 불필요한 모든 API의 리소스 경로, 스키마 제약 사항, 응답 코드(200, 404, 500)에 대한 부가적인 텍스트를 인지해야만 한다.

이렇게 강제 주입된 수만 단위의 토큰 낭비는, 실제 대화의 질을 높이기 위한 인지 컨텍스트(Cognitive Context) 공간을 심각하게 잠식하여 환각(Hallucination) 현상을 극대화한다.

0.3 상태 전이형(State-transitional) 도구 생태계의 패배

결합점(Integration Point)을 미리, 그리고 강력하게 묶어버리는 Swagger/OpenAPI의 ’명세 사전 배포 요구’는 인간과 소프트웨어 간의 신뢰도를 높이기 위한 공학의 산물이었다. 그러나 시시각각 변화하는 디렉터리를 감시하고, 즉석에서 스크립트를 컴파일하여 자신만의 도구로 활용해야 하는 상태 전이형(State-transitional) 에이전트 통신망에서는 치명적인 걸림돌이 된다.

Model Context Protocol(MCP)은 이 경직된 정적 명세 파일을 전부 폐기하고, 연결 과정 속에서 즉시 JSON 스키마를 동적으로 협상(Dynamic Negotiation)하는 방법을 채택함으로써, LLM 에이전트가 단 1초의 지연도 없이 세상의 변화된 도구를 손에 쥘 수 있는 구조적 지위를 획득하게 되었다.
