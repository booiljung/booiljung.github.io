1.5.1.1.3. 무상태(Stateless) 구조로 인해 발생하는 매 요청 간 페이로드(Payload) 비대화 한계

REST(Representational State Transfer) 아키텍처가 월드 와이드 웹(WWW)의 대규모 확장을 이끌어낼 수 있었던 가장 결정적인 철학은 역설적이게도 ’서버가 클라이언트의 상태를 기억하지 않는다’는 무상태성(Statelessness) 원칙이었다. 이 원칙에 따라 웹 서버는 클라이언트와의 통신 상태나 세션(Session) 데이터를 저장할 의무에서 해방되었고, 로드 밸런싱(Load Balancing)과 수평적 확장(Horizontal Scaling)을 극도로 효율화할 수 있었다.

그러나 서버 측의 자원 관리를 최적화하기 위해 고안된 이 무상태성 원칙은, 대화형(Conversational) 구조를 띠고 점진적인 기억(Incremental Memory)을 요하는 대규모 언어 모델(LLM) 에이전트의 통신 생태계로 이식될 때 최악의 아키텍처적 마찰(Architectural Friction)을 빚어낸다.

0.1 컨텍스트(Context) 강제 복원에 따른 페이로드(Payload) 비대화

REST API 서버는 클라이언트가 방금 전에 어떤 요청을 보냈는지 단 1초 전의 기억조차 보존하지 않는다.
따라서 LLM 에이전트가 “A사의 지난 분기 재무제표를 요약해 줘“라고 요청한 뒤, 곧이어 “거기서 마케팅 비용만 다시 상세하게 분류해 봐“라고 문맥을 이어갈 때, 에이전트가 호출해야 하는 후속 API 페이로드(Payload)는 기하급수적으로 비대화된다.

서버가 “거기서(이전 재무제표의 응답 결과)“가 무엇을 지칭하는지 모르기 때문에, 클라이언트(에이전트 구동 환경)는 다음과 같은 엄청난 양의 데이터를 매 HTTP 요청마다 본문(Body)이나 헤더(Header)에 강제로 욱여넣어 다시 전송(Re-transmit)해야만 한다.

  1. 세션 및 인증 지표: 수백 바이트에 달하는 JWT 구조의 무상태 접근 토큰(Access Token)
  2. 과거 대화 내역: 현재의 지시가 어떤 맥락에서 도출되었는지 증명하기 위한 과거의 질의 및 응답 캐시(Cache) 전체
  3. 복합 파라미터: 직전 요청에서 필터링했던 company=A_Corp, period=last_quarter 등의 모든 쿼리 파라미터(Query Parameters)

결과적으로 에이전트와 백엔드 간에 오가는 데이터의 크기는 눈덩이처럼 불어나며, 이는 매 요청마다 불필요한 직렬화(Serialization)/역직렬화(Deserialization) 연산을 유발하여 네트워크 대역폭(Bandwidth)을 심각하게 고갈시킨다.

0.2 토큰 예산(Token Budget) 낭비와 다단계 추론(Multi-step Reasoning)의 붕괴

이러한 매 요청 간 페이로드의 비대화는 에이전트 시스템에 국한된 지연(Latency) 문제를 넘어, 인공지능 자체의 아킬레스건을 직격한다. LLM이 사용할 수 있는 컨텍스트 윈도우(Context Window)는 엄격히 제한된 토큰 예산(Token Budget)에 구속받기 때문이다.

REST의 무상태성을 보완하기 위해 클라이언트-서버 간에 과거 정보를 계속해서 핑퐁(Ping-pong) 치듯 주고받다 보면, 에이전트는 진작에 폐기했어야 할 수십 턴 전의 쓰레기 API 응답 값까지 토큰 공간에 계속 짊어지고 가야 하는 상황에 직면한다. 이로 인한 ’인지적 과부하(Cognitive Overload)’는 결국 에이전트가 제일 중요한 직전 단계의 프롬프트를 망각하게 만들고 엉뚱한 결론을 도출하는 환각(Hallucination) 현상을 촉발한다. 다중 단계 논리 전개(Multi-step Logic Reasoning)를 요하는 복잡한 태스크일수록 시스템은 더욱 처참하게 붕괴한다.

다음 다이어그램은 무상태 기반 REST API 통신 환경에서 매 요청마다 페이로드가 누적(Accumulation)되어 토큰 자원을 고갈시키는 현상을 시각화한 것이다.

결론적으로, 다단계 추론(Multi-step Reasoning)이라는 지능적 워크플로우를 소화하기 위해선 시스템 내부에 세션의 지속성과 상태 존속성(Stateful Persistence)이 절대적으로 요구된다.

Model Context Protocol(MCP)은 단발성 HTTP 요청/응답 구조를 지양하고, Stdio 혹은 SSE(Server-Sent Events) 기반의 영속적인(Persistent) 연결 계층을 수립함으로써 이러한 REST의 무상태성에 마침표를 찍었다. 이를 통해 클라이언트와 서버는 한 번 수립된 세션 위에서 중복된 패키지의 강제 송신 없이, 오직 새로 갱신된(Delta) 상태나 도구(Tool) 지시만을 순수하게 주고받으며 인공지능의 토큰 시야를 극한으로 아껴낼 수 있는 것이다.
