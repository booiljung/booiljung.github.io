1.5.1.3.1. 무상태 비동기 콜백(Callback) 구조상 발생하는 컨텍스트 역추적 및 매핑의 난이도

웹훅(Webhook)은 클라이언트 요청에 대해 즉각적으로 응답을 대기하는 동기식(Synchronous) API 통신과 달리, 작업이 완료된 후 전혀 다른 시간대에 전혀 다른 세션(Session)으로 응답을 푸시(Push)하는 완벽한 유리형 비동기 콜백(Detached Asynchronous Callback) 구조를 채택하고 있다. 이는 서버 입장에서 데이터베이스 인덱싱(Indexing)이나 긴 렌더링(Rendering) 시간 동안 연결을 잡아두지 않아도 된다는 이점을 제공하지만, 대화의 맥락(Context)을 끊임없이 추적해야 하는 대규모 언어 모델(LLM) 에이전트 환경에서는 악몽과도 같은 매핑 파편화(Mapping Fragmentation)를 초래한다.

0.1 끊어진 세션(Session)과 콜백 식별자(Correlation ID) 의존성

사용자가 에이전트 인터페이스(가령 Claude Desktop)를 통해 “이번 프로젝트의 대규모 데이터셋 생성 작업을 AWS에 지시해 줘“라고 명령했다고 가정해 보자.

  1. 에이전트는 이 명령을 파싱(Parsing)하여 외부 시스템(AWS 등)으로 작업 지시를 날린다. 이때 원래의 HTTP 요청은 즉시 성공(HTTP 202 Accepted)으로 끝나며 세션이 닫힌다.
  2. 에이전트는 사용자와의 활성 대화창(Active Chat Session)을 유지하며 다른 대화를 이어간다.
  3. 30분 뒤, 외부 시스템이 작업을 끝마치고 웹훅 경로(/webhook/job-completed)로 완료 레코드를 쏘아 보낸다(POST).

이때 에이전트 시스템에 도착한 웹훅 페이로드(Payload) 자체는 무상태(Stateless) 단일 메시지에 불과하다. 이 “작업 완료” 알림이 사용자와의 ‘어떤 대화 세션’, ‘어떤 직전 프롬프트’, 그리고 ’어느 질문 맥락’에서 파생된 것인지 엮어내기 위해서는 오로지 난수화된 콜백 식별자(Correlation ID) 하나에 의존해야만 한다.

개발자는 웹훅 수신 백엔드 서버에 1) 수신받은 Correlation ID를 레디스(Redis)나 데이터베이스에 저장된 활성 세션 정보와 대조하고, 2) 어떤 클라이언트(WebSocket이나 SSE 채널)로 이 메시지를 포워딩(Forwarding)할지 라우팅 구조를 짜며, 3) 이를 다시 LLM의 컨텍스트 윈도우(Context Window)에 이전 메시지의 속편(Sequel)처럼 주입하는 방대한 상태 관리 로직(State Management Logic)을 이중 삼중으로 구현해야 한다.

0.2 컨텍스트 역추적(Context Traceback)의 한계와 환각(Hallucination)

이 빈약한 끈(Correlation ID)을 통한 매핑은 에러나 네트워크 지연이 생겼을 때 쉽게 끊어진다.

에이전트는 30분 전의 상황을 이미 토큰 예산(Token Budget) 낭비 방지를 위해 윈도우 밖으로 밀어냈을(Eviction) 수 있다. 갑자기 날아온 웹훅의 결과를 넘겨받은 LLM은 “이 데이터가 대체 내가 언제 명령한 작업의 결과물인가?“를 역추적(Traceback)하는 과정에서 심각한 혼선(Confusion)을 겪는다.
문맥을 잃어버린 에이전트는 사용자가 묻지도 않은 30분 전 작업의 결과를 뜬금없이 화면에 출력하거나, 현재 오가고 있던 엉뚱한 업무 논리에 웹훅의 데이터를 섞어버리는(Merge) 논리적 파탄(Logical Breakdown) 즉 환각(Hallucination) 현상을 극대화한다.

다음 모식도는 웹훅의 비동기성(Asynchrony)이 에이전트의 대화 맥락을 물리적으로 분리시켜 시스템의 복잡도를 얼마나 상승시키는지 보여준다.

0.3 상태 존속(Statefulness)의 내재화 실패

결론적으로, 웹훅 아키텍처는 통신 규약 자체가 ’상태 존속성(Statefulness)’을 인프라 밖으로 내동댕이쳤기 때문에 에이전트와 같은 상태 의존적(State-dependent) 시스템에 접목하기 가장 까다로운 방식이다. 개발자가 에이전트 코더(LLM Prompting) 역할인지, 아니면 수많은 비동기 콜백 큐(Queue)를 관리하는 인프라 관리자인지 혼동하게 만들 지경이다.

Model Context Protocol(MCP)은 이 파편화를 방지하기 위해 로컬의 Stdio나 실시간 전이중(Full-duplex) HTTP 스트림망(SSE)을 활용해 어플리케이션 생명주기(Lifecycle)와 통신 생명주기를 동일선상에 묶어놓았다. 에이전트의 사고 흐름(Reasoning Flow) 내에서 직접 데이터를 기다리고 폴링(Polling)하거나 이벤트를 즉각적으로 받아들임으로써, 콜백 ID를 DB에 욱여넣고 역추적하는 등의 부차적이고 소모적인 인프라 복잡성을 완벽에 가깝게 제거해 낸 것이다.
