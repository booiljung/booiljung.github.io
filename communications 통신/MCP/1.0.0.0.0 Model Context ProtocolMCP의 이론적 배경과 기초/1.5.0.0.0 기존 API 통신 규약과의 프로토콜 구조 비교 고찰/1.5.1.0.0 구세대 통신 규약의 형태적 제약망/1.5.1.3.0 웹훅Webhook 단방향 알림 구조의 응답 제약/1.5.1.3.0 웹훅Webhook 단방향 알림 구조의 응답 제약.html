1.5.1.3. 웹훅(Webhook) 단방향 알림 구조의 응답 제약

분산 시스템 간의 이벤트 연동을 구현할 때 가장 보편적이고 손쉽게 채택되는 아키텍처 중 하나가 바로 웹훅(Webhook)이다. 웹훅은 특정 시스템에서 데이터베이스의 레코드가 변경되거나 특정 이벤트(예: 깃허브 저장소의 푸시, 결제 승인 완료 등)가 발생했을 때, 서버 측에서 사전에 등록된 클라이언트의 HTTP 엔드포인트(Endpoint)로 페이로드(Payload)를 능동적으로 발송(Push)하는 역방향 API 호출 방식이다.

폴링(Polling) 방식의 네트워크 자원 낭비를 줄이기 위해 고안된 웹훅은 “이벤트 발생 시 즉각적인 알림“이라는 알림(Notification) 본연의 목적에는 완벽히 부합한다. 하지만 이 기술을 대규모 언어 모델(LLM)을 장착한 에이전트와 백엔드 간의 ‘교섭적이고 상호작용적인(Interactive)’ 통신망으로 사용하려 할 때, 우리는 가장 근본적인 아키텍처적 장벽에 부딪히게 된다.

바로 웹훅이 그 태생부터 철저히 **단방향(Uni-directional)**으로 흐르게끔 설계된 ‘통보(Notify)’ 시스템에 불과하다는 점이다.

0.1 대화형(Conversational) 에이전트 워크플로우와의 구조적 괴리

지능형 에이전트가 세계와 상호작용하는 방식은 인간의 대화 흐름과 유사하다. 에이전트는 A라는 질문을 던지고, 그에 대한 즉각적인 피드백(Feedback)인 B를 받아, 이를 바탕으로 다시 판단을 수정하여 C라는 후속 조치를 실행하는 정교한 연속 피드백 루프(Continuous Feedback Loop) 구조를 따른다.

하지만 전통적인 웹훅 아키텍처에 에이전트를 연동하려 하면, 이 자연스러운 피드백 루프가 다음과 같이 물리적으로 단절된다.

  1. 에이전트(혹은 외부 시스템)가 특정 작업을 트리거(Trigger)한다.
  2. 외부 시스템이 작업을 완료한 후, 에이전트가 열어둔 웹훅 URL로 결과 데이터를 POST 전송한다.
  3. 웹훅은 본질적으로 비동기 단방향 통신이므로, 에이전트는 수신 성공을 알리는 단순한 HTTP 200 OK 이외에 그 어떤 복합적인 후속 질의(Query)나 제어 데이터(Control Data)를 실어서 해당 TCP 연결 세션 위로 되돌려 보낼 수 없다.

만약 에이전트가 웹훅 알림을 받은 직후 “수신된 데이터 포맷에 맞춰 추가적인 데이터 N건을 더 보내달라“고 요청해야 한다면, 웹훅 응답 객체를 통해서는 불가능하다. 에이전트는 별도의 REST API나 다른 통신 채널을 열어 외부 시스템에 ’요청(Request)’을 보내는 로직을 새로 짜야 하며, 이는 데이터의 인바운드(Inbound) 파이프라인과 아웃바운드(Outbound) 파이프라인이 기형적으로 찢어지는 아키텍처적 파편화(Fragmentation)를 유발한다.

0.2 Fire-and-Forget 에러 처리의 맹점

또한 웹훅은 쏘고 잊어버리는 이른바 ‘Fire-and-Forget’ 철학을 빈번하게 택한다.
외부 서버가 에이전트에게 데이터를 밀어 넣는(Push) 순간 에이전트 측 서버가 일시적인 트래픽 폭주로 죽어 있거나 컨텍스트 윈도우가 가득 차 처리를 거부(HTTP 503 등)하게 될 경우, 지능형 시스템 간 필수적인 ‘에러 컨텍스트(Error Context)’ 보존 메커니즘이 무너진다. 외부 서버는 기계적으로 몇 번의 재시도(Retry) 알고리즘을 수행할 뿐, LLM이 도대체 “어떤 논리적 근거로 이 알림을 거부했는지“에 대한 자연어 기반의 이유(Reasoning)를 피드백받을 재간이 없다.

결국 단방향 알림에 최적화된 웹훅 기술은, 스스로 판단하고 양방향으로 타 시스템의 상태를 조회 및 조작(Mutate)해야 하는 전천후 AI 에이전트 결합 모델로는 명백한 뼈대의 한계를 지닌다. 이후 이어지는 세부 절들에서는 이 단방향의 콜백(Callback) 구조가 어째서 컨텍스트 관리의 난이도를 폭증시키고, 또 보안이 중요시되는 프라이빗(Private) 방화벽 환경에서 얼마나 치명적인 네트워크 보안 위협을 초래하게 되는지 기술적으로 더 깊이 파헤쳐 볼 것이다.
