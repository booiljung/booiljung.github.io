1.5.1.4.3. 방화벽 내 L7 필터링(DPI)에 걸려 조기 차단될 가능성이 높은 보안 프로파일 약점

웹소켓(WebSocket)이 직면하는 또 다른 치명적인 장벽은, 기업 환경이나 제로 트러스트(Zero Trust) 보안망 내부에서 네트워크 관문에 의해 ’비정상 트래픽’으로 간주되어 강제 차단될 확률이 매우 높다는 점이다.

웹소켓 프로토콜은 최초 연결 시 표준 HTTP 형태의 핸드셰이크(Handshake)를 거치며 Upgrade: websocket 헤더를 통해 TCP 터널로의 전환을 시도한다. 그러나 이 전환이 성공한 직후부터 해당 소켓 위로 흐르는 페이로드(Payload)는 더 이상 전통적인 HTTP 규격을 따르지 않는 순수한 이진(Binary) 또는 텍스트 프레임(Frame)의 연속체가 된다.

0.1 심층 패킷 분석(DPI) 시스템과의 충돌

현대의 엄격한 기업용 네트워크 방화벽(Firewall)과 차세대 침입 탐지 시스템(IDS)은 애플리케이션 계층(OSI 7계층)까지 패킷의 내용을 뜯어보는 심층 패킷 분석(Deep Packet Inspection, DPI) 기술을 기본으로 탑재하고 있다.

DPI 장비들은 자신들이 해독하거나 패턴을 예측할 수 없는 ‘불투명한(Opaque)’ 영속적 TCP 터널링을 본능적으로 보안 위협(예: 악성코드의 C&C 서버 통신, 데이터 탈취용 리버스 쉘)으로 간주한다. 웹소켓 페이로드는 명확한 HTTP 헤더나 메서드(Method)가 명시되어 있지 않으므로, DPI 필터가 이를 “식별할 수 없는 장기 지속 트래픽“으로 분류해 연결을 조기에 강제 종료(Session Kill)해 버리거나 연결 자체를 거절하는 상황이 빈번하게 발생한다.

0.2 프록시(Proxy) 게이트웨이 환경에서의 취약성

또한 HTTP 요청-응답(Request-Response) 주기에 최적화된 리버스 프록시(Reverse Proxy)나 클라우드 기반 로드 밸런서(Load Balancer)들은 웹소켓의 ‘무한에 가까운 지속적 연결(Long-lived Connection)’ 스펙을 처리하는 데 큰 어려움을 겪는다.
이러한 중간 장비들은 통상적으로 특정 유휴 시간(Idle Timeout, 예: 60초) 이상 트래픽이 오가지 않는 연결을 가차 없이 끊어버린다. 에이전트와 서버가 잠시 추론을 위해 대기하는 사이, 웹소켓 터널은 외부 프록시 장비에 의해 무수히 난도질당하며 결국 앞서 다룬 복잡한 ‘재연결(Reconnection) 로직’ 구축의 부담을 가중시킨다.

0.3 SSE(Server-Sent Events)의 구조적 우위와 MCP의 선택

이처럼 7계층 프로토콜 인스펙터 앞에서는 무용지물이 되기 쉬운 웹소켓의 단점을 타개하기 위해, Model Context Protocol(MCP)은 원격 클라우드 통신의 기본 뼈대로 웹소켓 대신 **SSE(Server-Sent Events)**와 HTTP POST의 이원화 구조를 전격 채택했다.

SSE는 처음부터 끝까지 변형 없는 순수 HTTP/1.1 혹은 HTTP/2 스트리밍 통신 규약을 철저히 준수한다. 방화벽이나 프록시 서버 입장에서는 이것이 그저 ’내용이 조금 길게 이어지는 일반적인 HTTP 웹 페이지 요청’과 완벽히 동일하게 보이므로, 어떠한 L7 필터링이나 차단 룰셋에도 걸리지 않고 방화벽을 유연하게 통과(Bypass)할 수 있다.

즉, 폐쇄망이나 깐깐한 인트라넷을 뚫고 들어가 에이전트에게 데이터 통로를 열어주어야 하는 현대적 통합 프로토콜에게 있어, 웹소켓은 보안 장비와의 이질감 탓에 결코 채택될 수 없는 치명적인 구조적 약점을 지니고 있는 것이다.
