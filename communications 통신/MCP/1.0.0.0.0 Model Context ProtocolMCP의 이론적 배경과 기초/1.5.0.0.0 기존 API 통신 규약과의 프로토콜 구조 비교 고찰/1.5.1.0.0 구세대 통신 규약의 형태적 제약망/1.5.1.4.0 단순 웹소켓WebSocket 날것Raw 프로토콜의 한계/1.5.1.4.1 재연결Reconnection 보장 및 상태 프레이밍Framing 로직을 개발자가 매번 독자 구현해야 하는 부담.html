1.5.1.4.1. 재연결(Reconnection) 보장 및 상태 프레이밍(Framing) 로직을 개발자가 매번 독자 구현해야 하는 부담

웹소켓(WebSocket) 프로토콜은 단방향 통신의 한계를 극복하고 실시간 양방향 채널을 열어준다는 점에서는 뛰어나지만, 통신망의 신뢰성을 담보하기 위한 필수 전제 조건들을 전혀 제공하지 않는다. 가장 대표적인 결함은 네트워크 단절 및 오류 복구(Fault Tolerance)에 대한 내장 시스템 부재와, 전송 데이터의 논리적 경계를 짓는 추상적 맥락 프레이밍(Semantic Framing)의 결여이다.

0.1 무방비한 연결 유실과 재연결(Reconnection)의 책임

모바일 접속망의 전환(Wi-Fi에서 LTE로), NAT 라우터의 일시적 끊김, 혹은 서버 배포로 인한 일시적 커넥션 드롭 등 불규칙한 네트워크 장애는 현실 모바일 및 에지(Edge) 통신에서 상시로 발생한다.

REST API나 SSE(Server-Sent Events)의 경우, 이러한 오류 발생 시 HTTP 스펙 수준에서의 캐싱, 혹은 내장된 재연결 메커니즘(SSE의 Last-Event-ID 등)을 통해 유실된 데이터를 복구하기 쉬운 구조를 갖춘 반면, 날것의 웹소켓은 단순히 TCP 소켓을 가차 없이 닫아버리는(Close) 것으로 끝난다.

에이전트 개발자가 이를 운용하려면 결국 밑바닥부터 파이프라인 수리공 역할을 떠맡아야 한다.

  1. 일정 주기로 핑/퐁(Ping/Pong) 하트비트를 날려 좀비 세션을 감별하는 생명 주기 감시기를 달아야 하고,
  2. 소켓이 끊어지면 무작정 재요청하여 서버를 죽이는 상황을 피하기 위해 지수 백오프(Exponential Backoff) 기반의 재시도(Retry) 알고리즘을 짜야 하며,
  3. 연결이 끊어진 수 초 동안 유실된(Dropped) ’에이전트 제어 메시지’들을 캐싱해 두었다가 연결 복구 시 중복 없이 순서대로 밀어 넣는 데드레터 큐(Dead-Letter Queue)를 독자적으로 구축해야 한다.

0.2 데이터 조각화와 상태 프레이밍(Framing)의 굴레

웹소켓은 메시지를 전송할 때 데이터를 작은 프레임(Frame) 단위로 쪼개어 보낸다. 하지만 이는 철저히 이진 데이터의 전송 효율을 위한 TCP 단위의 껍데기일 뿐, **“이 프레임들이 모여 어떤 하나의 논리적인 의미(의도, 인텐트)를 구성하는가?”**에 대한 응용 계층의 상태 프레이밍 규칙은 전무하다.

LLM 에이전트가 처리하는 컨텍스트는 수천 토큰 이상의 마크다운 텍스트, 바이너리 이미지 데이터, JSON 도구 호출 구조 등 크기와 형태가 매우 방대하고 가변적이다. 이 거대한 컨텍스트를 웹소켓 파이프 안으로 집어넣을 때, 수신 측은 이 여러 번 나뉘어 도착하는 데이터 조각들이 ’도구 호출에 대한 응답’인지 ’에러 메시지’인지, 아니면 ’단순한 시스템 알림’인지 스스로 파악해야 한다.

개발자는 다음과 같은 기초적인 골조 공사를 매 서버마다 억지로 반복해야 한다.

  • “모든 메시지의 첫 4바이트는 식별자 규약으로 쓰자.”
  • “완전한 메시지의 끝을 알리는 종단자(Delimiter) 기호를 정하자.”
  • “수신 순서가 꼬일 때를 대비해 타임스탬프와 시퀀스 번호(Sequence ID) 프로퍼티를 메시지의 JSON 껍데기로 매번 싸서 보내자.”

0.3 애드혹(Ad-hoc) 바벨탑의 붕괴

이러한 재연결 및 상태 구조화 로직을 개발자가 매번 독자적으로 구현하게 되면, 전 세계의 수만 개 도구 생태계는 각기 다른 방식의 커스텀(Custom) 프로토콜을 탑재한 채로 파편화될 수밖에 없다. 누구는 핑/퐁을 3초마다 보내고, 누구는 10초마다 보내며, 장애 발생 시 메시지를 보존하는 순서와 식별자 규칙이 철저히 다 다르다.

Model Context Protocol(MCP)은 이 파괴적인 ‘애드혹(Ad-hoc) 바벨탑’ 구조를 지양하기 위해, 날것의 웹소켓 대신 철저하게 규격화된 **JSON-RPC 2.0 기반 메타데이터 래핑(Wrapping)**과 SSE를 활용한 인프라 종속적 재연결 보장 스펙을 프로토콜 전면에 차용했다. 어플리케이션 개발자가 인프라 유지보수라는 바닥 공사에 시간을 쏟지 않게 함이 범용 통합 인터페이스의 기본 승리 조건이기 때문이다.
