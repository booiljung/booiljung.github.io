1.5.1.4.2. 표준화된 프로시저(RPC) 호출 규격 없이 메시지 페이로드가 파편화되는 구조적 모순

웹소켓(WebSocket)이 열어둔 빈 도화지(Blank Canvas) 위에서 발생하는 가장 파괴적인 결과는 바로 데이터 페이로드(Payload) 포맷의 극심한 파편화(Fragmentation)이다. 전송 계층의 통로가 뚫렸을 뿐, 그 위로 오고 가는 메시지들이 “무슨 목적을 가진 어떤 구조의 언어인지” 규정하는 시스템, 즉 원격 프로시저 호출(Remote Procedure Call, RPC)에 대한 표준 규격이 내재되어 있지 않기 때문이다.

0.1 각기 다른 사투리를 구사하는 도구 생태계

에이전트가 데이터베이스 조회(Query) 도구와 운영체제 파일 읽기(Read) 도구에 동시 접속해야 하는 상황을 가정해 보자. 만약 이 도구들이 날것의 웹소켓 기반으로 독립적으로 개발되었다면, 에이전트는 동일한 ‘도구 실행’ 명령을 내리기 위해 각 서버가 자체 발명한(Ad-hoc) 페이로드 구조에 억지로 맞추어 통신해야 한다.

예를 들어 A 서버는 {"action": "read_file", "data": {"path": "/"}}라는 구조를 요구할 수 있고, B 서버는 {"type": "execute", "payload": {"method": "query", "args": [...]}}라는 전혀 다른 계층적 구조를 강제할 수 있다. 심지어 에러가 발생했을 때 A 서버는 {"error": "Not Found"}를, B 서버는 {"status": 500, "message": "Failed"}를 반환하는 등 에러 코드와 컨텍스트 응답 스펙마저 제각각이다.

0.2 M:N 연결 복잡도와 어댑터(Adapter) 설계의 한계

이처럼 표준화된 RPC 규격이 부재한 상황에서는 에이전트(Client) 구조 안에 접속하려는 서버(Server) 갯수만큼의 전용 파서(Parser)와 어댑터(Adapter)를 덕지덕지 붙여야 하는 끔찍한 코드 침식(Code Erosion) 현상이 유발된다.
새로운 도구가 생태계에 추가될 때마다 개발자는 해당 도구의 독자적인 웹소켓 메시지 명세서를 읽고 LLM이 이를 이해할 수 있도록 번역기 로직을 하드코딩(Hard-coding) 해야 한다. 이는 플러그 앤 플레이(Plug-and-Play)를 궁극적 목표로 삼는 자율형 에이전트 연합망 구축을 사실상 불가능하게 만든다.

반면, GraphQL이나 gRPC 기반의 통신망은 적어도 이 ‘메시지 규격’ 만큼은 강제하여 생태계를 통일시켰으나, 앞선 1.5.1.2 챕터들에서 살펴본 바와 같이 스키마에 대한 정적 컴파일 의존성(Static Schema Dependency)이라는 너무 과도한 경직성을 지녀 LLM의 동적 구조와 맞지 않았다.

0.3 JSON-RPC 2.0 도입의 당위성

결국 웹소켓과 같이 유연한 통신관을 사용하되, 무질서한 페이로드 파편화를 막고 확장성을 유지하기 위해서는 유연하면서도 예측 가능한 경량 RPC 규격이 절실하게 필요했다.

Model Context Protocol(MCP)은 이 딜레마를 해결하기 위해, 전송 계층(Transport Layer) 위에서 돌아가는 응용 계층의 메시지 포맷으로 JSON-RPC 2.0 스펙을 범용 표준으로 채택하였다. 이를 통해 method, params, id 등 실행과 역추적에 필요한 최소한의 구조적 합의(Consensus)를 강제함으로써, 에이전트는 세상에 존재하는 어떤 MCP 서버와 연결하더라도 별도의 어댑터 개발 없이 동일한 방식의 도구 호출 및 에러 처리를 일관되게 수행할 수 있게 되었다. 즉, 날것의 웹소켓이 초래하던 구조적 모순을 명세(Specification)의 분리 설계로 완벽히 극복한 것이다.
