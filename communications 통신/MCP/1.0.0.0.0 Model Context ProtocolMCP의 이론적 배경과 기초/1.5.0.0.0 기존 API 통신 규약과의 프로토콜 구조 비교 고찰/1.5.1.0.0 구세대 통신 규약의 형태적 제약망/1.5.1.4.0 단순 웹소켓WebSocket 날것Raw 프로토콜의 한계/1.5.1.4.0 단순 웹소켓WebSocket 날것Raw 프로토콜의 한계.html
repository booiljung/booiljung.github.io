1.5.1.4. 단순 웹소켓(WebSocket) 날것(Raw) 프로토콜의 한계

지금까지 논의한 무상태(Stateless) REST API 구조와 비동기 단방향 체계인 웹훅(Webhook)의 근본적인 한계점들을 타개하기 위해, 많은 시스템 설계자들은 자연스럽게 웹소켓(WebSocket) 기술로 눈을 돌려왔다. 웹소켓은 단일 TCP 연결 위에서 클라이언트와 서버 통신이 완벽하게 전이중 양방향(Full-duplex Duplex)으로 이루어지는 것을 가능하게 한다. 이론적으로는 이러한 실시간 양방향 스트리밍 프로토콜이야말로 지속적으로 대화 컨텍스트(Context)를 주고받아야 하는 대규모 언어 모델(LLM) 에이전트와 서버 간의 완벽한 짝(Pair)으로 보인다.

하지만 시스템 연동 아키텍처 관점에서, 부가적인 규칙이 덧씌워지지 않은 이른바 ‘날것의 웹소켓(Raw WebSocket)’ 프로토콜로 에이전트 연동망을 구축하는 것은 오히려 개발 측면의 구조적 늪(Swamp)에 빠지는 결과를 초래한다.

0.1 비어 있는 캔버스(Blank Canvas)가 초래하는 모호성

웹소켓은 통신 계층의 관점에서 그저 “문자열이나 바이너리 데이터를 실시간으로 보낼 수 있는 파이프(Pipe)” 그 이상도 이하도 아니다. 이 파이프를 통해 어떤 형태의 데이터(JSON, XML, Custom Binary)를 보낼지, 특정 메시지가 ’요청(Request)’인지 아니면 이전 요청에 대한 ’응답(Response)’인지 식별할 수 있는 내재적 문법이 전혀 존재하지 않는다.

마치 전화선은 뚫어놓았으나, 서로 어떤 언어로 대화할지 합의하지 않은 상태와 같다. 이로 인해 에이전트 인프라 개발자는 날것의 웹소켓을 도입하는 순간 단순히 통신을 연결하는 것을 넘어 다음과 같은 거대한 기능 계층들을 원점에서 응용 계층(Application Layer) 상에 직접 설계해야 하는 가혹한 책임을 떠안게 된다.

  1. 메시지 프레이밍과 규격화: 메시지 구조의 표준이 없으므로 자체적인 라우팅 스키마(Schema) 규칙을 발명해야 한다.
  2. 연결 지속성 보장: 네트워크 유실에 대비하여 직접 재연결(Reconnection) 및 세션 복원 시스템을 짜야 한다.
  3. 보안 및 검사망 우회: 기업 내의 방화벽이나 프록시(Proxy) 시스템과의 심각한 충돌 가능성을 감수해야 한다.

0.2 프로토콜 설계 짐의 전가(Delegation of Burden)

결과적으로 단순 웹소켓 프로토콜은 “실시간 전송“이라는 하나의 이점을 위해, 나머지 모든 통신 신뢰성과 규격화의 책임을 에이전트 클라이언트(Client)와 외부 도구 서버(Server)의 개발자에게 무책임하게 전가해 버린다. 이는 다양한 벤더(Vendor)나 개인 개발자가 만든 수많은 도구 생태계를 마찰 없이 플러그 앤 플레이(Plug-and-Play) 방식으로 연결해야 하는 에이전트 기반 생태계의 철학과는 양립하기 힘든 치명적 결함이다.

이어지는 세부 문서들에서는 양방향 통신의 껍데기만 제공하는 날것의 웹소켓 프로토콜이 어째서 상태 구조화(Structuring), RPC 표준화(Standardization), 그리고 망 분리 보안망(Security Boundary) 관점에서 구세대 통신의 형태적 제약망을 벗어나지 못하는지를 심층적으로 규명할 것이다. 외부 규약 개입 없이 작동하는 이 ‘하얀 도화지’ 같은 구조가 복잡한 에이전트 오케스트레이션 환경에서 어떻게 무너져 내리는지 단계별로 분석한다.
