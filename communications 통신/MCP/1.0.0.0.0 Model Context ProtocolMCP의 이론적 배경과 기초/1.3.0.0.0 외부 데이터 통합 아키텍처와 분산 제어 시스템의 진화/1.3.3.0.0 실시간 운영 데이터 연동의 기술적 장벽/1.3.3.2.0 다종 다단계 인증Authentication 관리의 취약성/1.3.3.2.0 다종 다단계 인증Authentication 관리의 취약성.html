1.3.3.2 다종 다단계 인증(Authentication) 관리의 취약성

대규모 언어 모델(LLM)을 기업의 핵심 운영망이나 사용자의 개인 데이터가 보관된 프라이빗 에코시스템(Private Ecosystem)에 연동할 때, 기술적 구현의 절반 이상은 ‘인증(Authentication) 및 인가(Authorization)’ 문제를 지옥처럼 방어해 내는 데 소모된다. 기존 플러그인(Plugin) 체제는 단순히 외부 서버로 HTTP 요청을 던지는 구조에 불과했기에, 보안 주체(Security Principal)가 여러 계층으로 찢어지고 복잡하게 얽히는 엔터프라이즈의 ‘다종 다단계 인증’ 환경을 감당할 아키텍처적 기반이 전혀 마련되어 있지 않았다. 이는 곧 데이터 유출과 세션 하이재킹(Session Hijacking)이라는 치명적인 보안 취약성으로 직결되었다.

0.1 보안 주체의 분열: ‘대리인(Agent)’ 딜레마

엔터프라이즈 시스템에서는 명확한 보안 주체의 식별이 필수적이다. 사용자가 사내 시스템에 접근할 때, 시스템은 그 ’사용자 개인’의 권한(Role)을 확인한다. 그러나 LLM 에이전트가 도입되면서 이 1:1 대응 관계가 무너지고 ’대리인 딜레마(Agent Dilemma)’가 발생했다.

에이전트가 서드파티 플러그인에 데이터를 요청할 때 호출의 주체는 ’인간 사용자’인가, 아니면 ’LLM 서버’인가?
기존 플러그인 시스템은 대부분 LLM 서버(예: OpenAI 서버)가 단일한 서비스 차원의 API 키(Service-level API Key)를 들고 서드파티 서버를 타격하는 형태를 띠었다. 이는 심각한 보안 결함이다. 사내 시스템(서드파티 서버) 입장에서는 자신을 찌르고 있는 것이 ’영업팀의 홍길동 대리’인지 ’인사팀의 김철수 과장’인지 분간할 수 없게 된다.
오직 ’LLM 봇’이라는 거대한 단일 주체만이 잡히기 때문에, 김철수 과장은 접근해서는 안 될 영업팀의 기밀문서를 LLM 챗봇을 통해 우회(Bypass)하여 열람할 수 있는 권한 상승(Privilege Escalation)의 구멍이 발생하게 된다. 개별 사용자의 컨텍스트(User Context)가 LLM 서버 계층에서 뭉개져 버리는 이 분열 현상은, 무결성(Integrity)을 생명으로 여기는 기업 보안의 제1원칙을 정면으로 위반했다.

0.2 하드코딩된 자격 증명(Hard-coded Credentials)과 난립하는 토큰 갱신

이 대리인 딜레마를 막기 위해, 초기 플러그인 개발자들은 프론트엔드 환경이나 시스템 프롬프트(System Prompt) 어딘가에 사용자 개인의 OAuth 토큰(Token)이나 API 키를 질끈 박아 넣는(Hard-coding) 위험천만한 편법을 동원했다.

문제는 여러 개의 플러그인을 결합하여 사용할 때 발생한다. A 시스템은 JWT를 요구하고, B 시스템은 OAuth 2.0 기반의 짧은 생명주기(Short-lived) 엑세스 토큰(Access Token)과 리프레시 토큰(Refresh Token)을 요구하며, C 시스템은 커스텀 헤더를 요구한다.
상태 유지(Stateless) 기반의 플러그인 환경에서 LLM은 이 수많은 다종 스펙의 토큰 만료 주기를 추적하고 스스로 갱신(Token Renewal)할 능력이 전무하다. 작업 중간에 B 시스템의 토큰이 만료되면, 에이전트는 무기력하게 401 Unauthorized 에러를 맞고 쓰러진다. 이 난립하는 인증의 파도 속에서 파이프라인의 보안과 유지보수성은 완전히 붕괴하였다.

0.3 Model Context Protocol(MCP)의 로컬 호스트(Local Host) 기반 인증 위임(Delegation)

다종 다단계 인증이 유발하는 이 끔찍한 취약성을 소진시키기 위해, Model Context Protocol(MCP)은 위험한 ’클라우드 LLM 계층’에서 인증을 처리하던 기존의 모순을 버리고, 사용자의 가장 안전한 울타리인 ’로컬 호스트(Local Host) 클라이언트’에게 모든 인증 권한을 위임(Delegation)하는 아키텍처를 창안했다.

MCP 아키텍처에서 클라우드의 LLM은 외부 서버와 직접 통신하지 않는 거대한 계산기로 전락한다. 실질적인 통신은 사용자의 IDE(예: Cursor)나 데스크톱 애플리케이션(MCP Client)이 로컬 백그라운드에서 구동 중인 MCP 서버와 맺는다.

이 클라이언트(Host) 런타임은 이미 로그인한 사용자의 환경 변수, OS의 파일 시스템 권한, 그리고 브라우저 내부에 저장된 사내 Single Sign-On(SSO) 세션을 매우 안전하고 합법적으로 물려받아(Inherited) 구동된다.
MCP 클라이언트가 사내망 데이터베이스를 조회할 때, 이미 사용자 개인의 명백한 자격 증명(Identity)이 적용된 상태로 통신(stdio)이 이루어지기 때문에 권한이 격상되거나 토큰 관리가 꼬일 우려가 완전히 배제된다. 외부로 빠져나갈 때 노출될 수 있는 값비싼 프록시나 외부 토큰 스토리지 없이도, 기존 엔터프라이즈의 다단계 인가(Authorization) 체제를 손끝 하나 건드리지 않고 LLM 생태계와 조화롭게 맞물리게 하는 MCP만의 독보적인 보안 추상화(Security Abstraction)다.
