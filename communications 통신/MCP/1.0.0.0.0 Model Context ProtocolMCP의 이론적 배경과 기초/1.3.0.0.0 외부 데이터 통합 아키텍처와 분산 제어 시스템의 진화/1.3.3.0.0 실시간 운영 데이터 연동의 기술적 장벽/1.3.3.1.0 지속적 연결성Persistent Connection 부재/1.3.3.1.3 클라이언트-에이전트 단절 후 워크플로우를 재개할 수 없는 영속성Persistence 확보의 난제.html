1.3.3.1.3 클라이언트-에이전트 단절 후 워크플로우를 재개할 수 없는 영속성(Persistence) 확보의 난제

자율형 에이전트(Autonomous Agent)가 고도화되면서 수 시간에 걸쳐 방대한 데이터베이스를 마이그레이션(Migration)하거나, 심층적인 코드를 분석하여 리팩토링(Refactoring)하는 장기 워크플로우(Long-term Workflow)의 수요가 급증했다. 그러나 단발성 HTTP 플러그인 체제는 ’일회성 문답’만을 상정하고 설계되었기 때문에, 클라이언트(사용자 브라우저 또는 IDE)와 에이전트 간의 네트워크 연결이 단 1초라도 끊기면 그동안 진행되던 수백 단계의 작업 문맥(Context)이 완전히 증발해 버리는 치명적인 ‘영속성(Persistence)의 모래성’ 구조를 띠고 있었다.

0.1 세션(Session) 증발과 롱 러닝(Long-running) 작업의 비극

기존 플러그인 기반 아키텍처에서 롱 러닝(Long-running) 작업을 수행하는 것은 살얼음판을 걷는 것과 같았다.

에이전트에게 “사내 위키(Wiki) 문서 1만 개를 읽고 요약하라“는 임무를 부여했다고 가정하자. 에이전트는 플러그인 API를 수천 번 호출하며 열심히 문서를 읽어 들이고 프롬프트 버퍼(Prompt Buffer)에 지식을 쌓아간다. 그런데 사용자가 실수로 브라우저 탭을 닫거나, 와이파이(Wi-Fi) 연결이 잠시 끊겨 웹소켓이 끊기면 어떻게 될까?
중앙 서버에 종속된 기존 언어 모델(LLM) 채팅 인터페이스는 클라이언트와의 TCP 세션(Session)이 끊어지는 즉시, 할당되어 있던 인메모리(In-memory) 상의 임시 워크플로우 스레드를 ’종료된 작업’으로 간주하고 가차 없이 파기(Garbage Collection)해 버린다. 사용자가 다시 접속하더라도 “어디까지 읽었더라?“라는 중간 상태(State)를 추적할 수 있는 토큰이나 세션 포인터(Session Pointer)가 서드파티 플러그인에는 존재하지 않으므로, 에이전트는 문서 1번부터 다시 읽기 시작하는 끔찍한 연산 비용 낭비와 타임아웃(Timeout)의 비극을 맞이하게 된다.

0.2 단절된 상태(Stateless) 아키텍처의 책임 전가

이 영속성의 난제는 본질적으로 ’도구의 상태(State) 보존 책임을 누가 질 것인가’에 대한 아키텍처적 합의가 결여되어 발생했다.

서드파티 플러그인(서버)은 “우리는 HTTP 요청에 답만 줄 뿐, 네가 몇 번째 문서를 읽고 있었는지와 같은 상태(State)를 저장하지 않는다“라며 무상태(Stateless) 원칙을 고수했다. 반면 LLM 플랫폼 벤더 측은 “우리는 범용적인 채팅창일 뿐, 너의 특정 플러그인 작업 진척도를 매번 DB에 백업해주지 않는다“라고 책임을 회피했다.
양측의 아키텍처가 서로에게 상태 보존 책임을 전가하는 동안, 워크플로우가 중단되었을 때 이를 재개(Resume)할 수 있는 ‘체크포인트(Checkpoint)’ 메커니즘은 시스템 어디에도 존재하지 않게 되었고, 결과적으로 플러그인은 절대 중간에 끊어지면 안 되는 짧고 가벼운 장난감 태스크에만 국한되어 사용될 수밖에 없었다.

0.3 Model Context Protocol(MCP)의 호스트 주도형 맥락 유지(Context Persistence)와 재연결(Reconnection) 회복력

Model Context Protocol(MCP) 생태계는 이 영속성 문제를 해결하기 위해, MCP 서버(도구)는 완전한 무상태를 유지하게 놔두되 ‘클라이언트 호스트(MCP Client Host)’ 런타임이 무거운 세션(Session) 매니저이자 워크플로우 오케스트레이터 역할을 전담하는 스마트 클라이언트(Smart Client) 아키텍처를 확립했다.

MCP 환경에서는 로컬 PC에 설치된 ’Cursor IDE’나 ‘Claude Desktop’ 같은 강력한 클라이언트 어플리케이션(Host)이 통신망의 중심축을 잡는다.

만약 클라이언트가 장기 워크플로우를 지시받고 MCP 서버와 교신하며 작업을 진행하던 도중 외부 네트워크 오류나 일시적 서버 크래시(Crash)로 인해 연결이 단절되더라도, 호스트 애플리케이션 자체가 구동 중이므로 LLM과 나눈 현재까지의 전체 대화 컨텍스트(Context)와 이전에 실행한 도구 결괏값 목록은 로컬 메모리(또는 로컬 DB)에 안전하게 영속화(Persisted)되어 있다.

통신이 정상화되고 MCP 서버가 재기동되면, 호스트(Client)는 즉각적으로 재연결(Reconnection) 핸드셰이크를 맺고, 이어서 서버 측에 “아까 실패했던 105번째 파일부터 다시 가져와라(Resource Read)“라고 정확히 지시할 수 있다.
이처럼 MCP 생태계는 외부 도구의 빈번한 단절(Disconnection)을 자연스러운 현상으로 가정하고, 호스트 중심의 상태 보존(State Preservation)을 통해 외부 장애가 전체 에이전트의 워크플로우 붕괴로 전이되는 것을 막아내는 고도의 회복 탄력성(Resilience)을 확보하였다.
