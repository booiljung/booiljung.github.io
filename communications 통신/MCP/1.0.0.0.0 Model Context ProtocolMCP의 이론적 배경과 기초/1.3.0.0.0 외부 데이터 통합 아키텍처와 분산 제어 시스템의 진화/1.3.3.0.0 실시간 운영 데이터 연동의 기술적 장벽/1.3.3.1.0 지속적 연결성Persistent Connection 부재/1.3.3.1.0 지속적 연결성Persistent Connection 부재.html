1.3.3.1 지속적 연결성(Persistent Connection) 부재

네트워크 아키텍처에서 ’지속적 연결성(Persistent Connection)’이란 클라이언트와 서버가 최초의 핸드셰이크(Handshake)를 맺은 이후, 연결의 파이프를 끊지 않고 유지하며 언제든 데이터를 자유롭게 주고받을 수 있는 상태를 의미한다. 실시간 주식 틱(Tick) 데이터나 분산 시스템의 라이브 로그(Live Log)를 받아오기 위해 현대 애플리케이션들은 당연하게 웹소켓(WebSocket)이나 gRPC와 같은 지속적 연결망을 채택한다. 그러나 대규모 언어 모델(LLM)과 도구를 연결하던 초기 플러그인 생태계는 이 필수적인 ’지속성’의 철학을 완전히 결여한 채 시작되었고, 이는 실시간 운영 데이터 연동에 있어 극복 불가능한 태생적 한계를 노출했다.

0.1 단발성(Stateless) HTTP 프로토콜의 구조적 한계

플러그인 생태계의 태동기는 웹의 가장 기초적인 통신 규약인 HTTP/1.1 기반의 단건(Single-shot) REST API 아키텍처에 깊게 뿌리내리고 있었다.

사용자가 “서버 상태를 확인해줘“라고 LLM에게 요청하면, 플러그인 서버로 단 한 번의 HTTP GET 요청이 발사되고 응답(Response)을 수신한 즉시 연결(TCP Connection)은 매정하게 끊어졌다(Stateless).
이렇게 연결이 즉각 단절되는 구조적 결함은, 서버 측에서 1초 뒤에 발생한 ’치명적 에러 발생’이라는 새로운 이벤트를 모델 측으로 능동적으로 밀어 넣어줄(Push) 경로가 완전히 차단되어 있다는 것을 의미한다. 끊어진 라인을 다시 잇기 위해서는 무조건 LLM(클라이언트 측)에서 먼저 요청(Request)을 다시 보내야만 했으며, 이는 이벤트가 언제 발생할지 알 수 없는 ’실시간 운영 시스템’의 불규칙한 데이터 스트림과는 근본적으로 상극(Incompatibility)인 통신 역학(Dynamics)이었다.

0.2 가짜 실시간성: 폴링(Polling)의 부하와 상태 유실

지속적 연결성의 부재를 소프트웨어 계층에서 강제로 우회하기 위해, 엔지니어들은 폴링(Polling)이라는 낡고 비효율적인 우회술을 동원했다.

이는 LLM 어플리케이션(에이전트)이 외부 시스템을 향해 “지금 서버 온도 어때?”, 1초 뒤 다시 “지금은 어때?“라며 의미 없는 비동기 반복 루프를 수없이 호출하는 방식이다. 그러나 이 가짜 실시간(Mock Real-time) 구현 방식은 막대한 네트워크 오버헤드(Overhead)를 발생시켰다.
수십 대의 에이전트가 단일 데이터베이스(DB)를 향해 동시에 무한 폴링을 시전하면, 외부의 운영 시스템은 디도스(DDoS) 공격에 준하는 I/O 마비 상태를 겪게 된다. 반대로 폴링 주기를 10초로 길게 늘리면 그사이(Blind Spot)에 발생한 치명적인 순간 트래픽 폭주 로그를 LLM이 완전히 놓쳐버리는(Data Loss) 치명타(Fatal Impact)를 맞게 되었다. 지속적 연결이 단절된 아키텍처에서는 ’비용 절감’과 ’실시간성 확보’라는 두 마리 토끼가 서로를 파괴하는 트레이드오프(Trade-off)의 늪에 빠질 수밖에 없었다.

0.3 Model Context Protocol(MCP)의 영구 파이프라인(Persistent Pipeline) 복원

실시간 운영 데이터와의 마찰을 종식시키기 위해, Model Context Protocol(MCP)은 HTTP의 단건 처리 사상을 버리고 클라이언트와 서버 간의 영구적인 파이프라인(Persistent Pipeline)을 프로토콜의 표준 계층(Standard Layer)으로 포섭했다.

MCP는 서버와 클라이언트가 stdio(표준 입출력 스트림)나 SSE(Server-Sent Events)를 통해 최초 핸드셰이크를 맺으면, 명시적 종료 프레임워크가 작동하기 전까지 그 통신 채널을 영구적으로 열어둔다.

특히 stdio 전송 계층을 사용할 경우, LLM의 클라이언트 호스트(예: IDE 애플리케이션)와 MCP 서버는 동일한 머신 내에서 프로세스 간 통신(IPC) 파이프를 공유하게 되므로, 어떠한 네트워크 오버헤드나 폴링 지연(Latency) 없이 초당 수만 건의 상태 변경 패킷을 원활하게 흘려보낼 수 있다.

이러한 지속적 파이프라인이 확보되면서, MCP 서버는 외부 사내망 데이터베이스에 레코드가 추가될 때마다 즉각 호스트를 향해 notifications/resources/updated 이벤트를 발사하여 클라이언트 측의 캐시(Cache) 상태를 능동적으로 갱신(Invalidation)시킨다. MCP는 모델의 눈을 가리고 있던 낡은 통신 프로토콜의 안대를 벗겨냄으로써, LLM이 끊김 없이 흐르는 현실의 타임라인 위에서 살아 숨 쉴 수 있는 완벽한 생태적 ’연결망(Connection Web)’을 구축한 것이다.
