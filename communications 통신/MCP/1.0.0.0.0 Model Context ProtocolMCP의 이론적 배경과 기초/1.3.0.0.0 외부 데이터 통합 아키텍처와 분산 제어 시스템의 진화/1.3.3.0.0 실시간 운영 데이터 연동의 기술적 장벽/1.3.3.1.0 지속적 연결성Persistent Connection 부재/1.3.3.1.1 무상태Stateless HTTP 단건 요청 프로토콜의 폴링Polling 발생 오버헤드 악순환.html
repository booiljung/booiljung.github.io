1.3.3.1.1 무상태(Stateless) HTTP 단건 요청 프로토콜의 폴링(Polling) 발생 오버헤드 악순환

초기 대규모 언어 모델(LLM) 플러그인 생태계를 지배했던 통신 철학은, 웹의 가장 원시적인 규약인 무상태(Stateless) HTTP/1.1 단건 요청(Single-shot Request) 아키텍처였다. 이는 본질적으로 “클라이언트가 질문할 때만 서버가 대답한다“는 철저한 수동성(Passivity)을 전제로 한다. 그러나 이 수동적 프로토콜을 시시각각 상태가 변동하는 ’실시간 운영(Live Operational) 서버’에 억지로 접목시키는 순간, 시스템은 무의미한 질문을 끊임없이 반복해야 하는 ’폴링(Polling) 오버헤드(Overhead)의 악순환’이라는 깊은 수렁에 빠지고 말았다.

0.1 실시간성 확보를 위한 맹목적 폴링(Blind Polling)의 굴레

무상태(Stateless) HTTP 환경에서 서버는 자발적으로 클라이언트(LLM 에이전트)에게 새로운 데이터를 밀어 넣어줄(Push) 수 있는 권한과 통로가 없다. 따라서, “주식 가격이 10% 이상 하락하면 즉각 대처하라“는 임무를 부여받은 에이전트가 선택할 수 있는 구조적 대안은 오직 하나뿐이다. 주기적으로 서버의 문을 두드려 “지금은 떨어졌습니까?“라고 묻는 ‘폴링(Polling)’ 로직을 무한히 가동하는 것이다.

만약 목표하는 실시간성의 해상도가 1초라면, 에이전트는 1초마다 무거운 HTTP 헤더(Header)와 암호화된 TLS 핸드셰이크(Handshake)를 포함한 완전한 네트워크 세션을 새로 맺고 끊어야 한다. 실제 주식 가격이 변동하지 않은 평온한 99%의 시간 동안, 에이전트와 서버는 “변경 사항 없음(HTTP 304 Not Modified)“이라는 텅 빈 응답을 주고받기 위해 막대한 트래픽과 컴퓨팅 파워를 허공에 태우는 맹목적 루프(Blind Loop)에 갇히게 된다.

0.2 레이턴시(Latency)와 확장성(Scalability)의 파괴적 트레이드오프

이 폴링 주기는 시스템 아키텍트에게 해결 불가능한 파괴적 트레이드오프(Trade-off)를 강요한다.

실시간성을 보장하기 위해 폴링 주기를 0.1초로 극단적으로 좁히면, 수백 개의 에이전트가 동시에 API를 타격할 경우 대상 운영 서버는 디도스(DDoS) 공격에 준하는 커넥션 풀(Connection Pool) 고갈과 I/O 마비 상태를 겪게 된다.
반대로 서버를 보호하기 위해 폴링 주기를 10초, 혹은 1분 단위의 롱 폴링(Long-polling)으로 완화하면, 그 지연 시간(Latency Blind Spot) 사이에 골든 타임을 놓쳐버리는 치명적 비즈니스 실패가 발생한다. 플러그인 생태계는 ’서버를 죽이거나, 아니면 한발 늦거나’라는 구조적 딜레마 속에서 실시간 엔터프라이즈 환경으로의 확장을 스스로 가로막았다.

0.3 Model Context Protocol(MCP)의 이벤트 기반 갱신(Event-driven Update)을 통한 오버헤드 소거

Model Context Protocol(MCP)은 이 소모적인 단건 HTTP 폴링 루프를 프로토콜의 설계 단계에서부터 원천적으로 소거해 버렸다. MCP의 통신망은 클라이언트가 묻기 전에 서버가 먼저 알려주는 ’이벤트 기반(Event-driven)’의 능동형 아키텍처를 채택했다.

MCP 클라이언트는 초기 연결이 이루어질 때 특정 데이터(Resource)에 대해 명시적인 ‘구독(Subscription)’ 의사를 밝히고, 그 후에는 완전히 침묵(Idle) 상태에 돌입한다. 더 이상 1초마다 서버의 문을 두드리며 네트워크 대역폭을 낭비하지 않는다.

서버 내부의 데이터베이스나 센서에 실제 변경(Mutation)이 발생한 그 찰나의 순간에만, MCP 서버 시스템이 영구적으로 열려 있는 표준 입출력 스트림(stdio)이나 SSE 채널을 타고 역방향으로 알림 알람(notifications/resources/updated)을 일발 장전하여 클라이언트에게 주입한다.
이로써 수만 건의 상태 변경도 네트워크 오버헤드 없이 실시간 스트리밍이 가능해졌으며, 무상태 폴링이 유발하던 트래픽 낭비와 레이턴시의 지연이라는 오랜 악순환의 고리가 MCP 호스트 환경 내에서 완전히 끊어지게 되었다.
