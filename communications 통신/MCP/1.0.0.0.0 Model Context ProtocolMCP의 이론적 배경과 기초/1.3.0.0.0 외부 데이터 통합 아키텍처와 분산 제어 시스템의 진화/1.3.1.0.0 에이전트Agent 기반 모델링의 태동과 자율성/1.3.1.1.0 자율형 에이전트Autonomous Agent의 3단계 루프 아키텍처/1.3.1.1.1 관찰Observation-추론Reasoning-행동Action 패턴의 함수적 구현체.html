1.3.1.1.1 관찰(Observation)-추론(Reasoning)-행동(Action) 패턴의 함수적 구현체

자율형 에이전트(Autonomous Agent)가 철학적 개념을 넘어 공학적 실체로 현현(Manifestation)할 수 있었던 결정적 계기는 인간의 인지 과정을 코드로 사상시킨 3단계 루프의 ‘함수적 구현체(Functional Implementation)’ 확립에 있다. 에이전트 아키텍처는 관찰(Observation), 추론(Reasoning), 행동(Action)이라는 추상적 행위 모형을 대규모 언어 모델(LLM)의 생성 텍스트와 외부 시스템의 API 서명(Signature)으로 엄격하게 치환하여, 단순한 문자열의 나열을 물리적 세계를 조작하는 컴퓨팅 명령어의 연쇄배열로 진화시켰다.

0.1 추론(Reasoning)의 가시성 확보: 생각의 사슬(Chain-of-Thought)과 프롬프트 할당

인간은 의사결정을 내릴 때 두뇌의 신경망에서 비가시적인 사고 과정을 거친다. 하지만 LLM을 에이전트로 활용하기 위해서는 이 내밀한 사고 과정을 컨텍스트 윈도우 위로 투명하게 끄집어내는 공학적 트릭이 필요하다.

에이전트의 ‘추론(Reasoning)’ 단계는 시스템 프롬프트를 통해 “반드시 행동을 취하기 전에 네가 왜 그런 행동을 해야 하는지 텍스트로 먼저 작성하라“는 규약(Contract)을 강제함으로써 구현된다.
예를 들어, “사용자의 메일을 읽고 답장해라“라는 미션이 주어졌을 때, 에이전트는 곧각 이메일 발송 API를 때리지 않는다. 대신 컨텍스트 윈도우에 Thought: 사용자는 내일 미팅 시간을 묻고 있다. 캘린더를 우선 확인해야 시간 충돌을 피할 수 있다.라는 자기 성찰적(Self-reflective) 문자열을 먼저 생성(Reasoning)한다. 이 짧은 ‘생각의 사슬(Chain-of-Thought)’ 토큰들은 이어지는 다음 단어의 확률 분포를 계산의 영역으로 강력하게 편향(Bias)시키는 훌륭한 닻(Anchor)의 역할을 수행하며, 에이전트가 충동적 판단을 내리지 않도록 제어하는 핵심 함수적 규약이 된다.

0.2 행동(Action)의 함수화: 자연어의 기계어 파싱(Parsing)

추론이 끝나고 결단을 내리는 ‘행동(Action)’ 단계는 자연어(Natural Language)를 구동 가능한 기계어(Machine Code)로 강제 변형하는 극단의 파싱(Parsing) 공간이다.

LLM은 손발이 없으므로, 에이전트는 자신이 쓸모 있는 외부 도구를 호출하길 원할 때 텍스트 생성의 형태를 비틀어 함수 서명(Function Signature)의 형태로 출력한다.
Action: CheckCalendar(date="2024-10-31")
이 문자열이 스트리밍 되는 순간, 에이전트를 감싸고 있는 런타임 프레임워크(예: LangChain, LlamaIndex 등)는 텍스트 생성을 즉각 정지시킨다(Suspend). 그리고 해당 문자열의 정규표현식(Regex)을 추출하여 실제 로컬 환경이나 네트워크 백엔드에 존재하는 파이썬(Python) 함수 혹은 REST API를 거세게 호출해버린다. 언어 모델링의 산물인 추상적 단어가 물리적 서버의 엔드포인트를 타격하는 진정한 ’행동의 실체화’가 이루어지는 지점이다.

0.3 관찰(Observation) 버퍼와 MCP 관점의 맥락 병합

행동(Action)의 파장이 외부 세계에서 어떤 결과를 가져왔든, 그 결괏값 뭉치를 에이전트의 망막 안에 다시 집어넣어 순환 궤도를 완성하는 것이 ‘관찰(Observation)’ 버스(Bus)의 임무이다.

graph TD
    A[Agent 런타임] -->|Thought 생성| B(Reasoning)
    B -->|도구 선택 / 인자 할당| C(Action)
    
    C -->|Action: Search_DB| D{Environment/Middleware}
    D -->|결과 1: "데이터 없음"| E[Observation Buffer]
    D -->|결과 2: JSON Payload| E
    D -->|결과 3: Error Code 500| E
    
    E -->|결과를 Text로 Serialize| A
    A -.->|Observation을 읽고 다시 Thought 수정| B

외부 API가 반환한 수천 줄의 JSON 덩어리나 가차 없는 500 Internal Server Error 메시지 같은 원시적 타격은, 미들웨어에 의해 다시 단순한 텍스트 문자열로 직렬화(Serialization) 된다.
Observation: 해당 날짜에는 이미 오전 10시에 마케팅 회의가 잡혀 있음.
이 관찰 데이터가 에이전트의 다음 프롬프트 후미에 달라붙어 재주입(Re-injection)됨으로써, 에이전트는 “예상과 달리 10시는 안 되겠군. 오후 2시를 제안해야겠다“라며 끊어진 생각을 이어 붙일 수 있게 된다.

하지만 수백 개의 파편화된 외부 툴들이 각자 어지러운 규격으로 Observation 결과값을 반환하던 혼란의 시기를 지나, Model Context Protocol(MCP)은 이 관찰 결과값을 포맷팅하고 에이전트에게 전달하는 통로를 단일화(Standardization)했다. MCP 자원(Resource) 규약을 통해 정제된 데이터만을 공급받게 된 에이전트는, 더 이상 관찰 데이터의 비정형적 쓰레기 문자열들을 파싱 하는 데 컨텍스트 토큰을 낭비할 필요 없이 완벽하게 통제된 규격의 순환 루프 위에서 비행할 수 있게 되었다.
