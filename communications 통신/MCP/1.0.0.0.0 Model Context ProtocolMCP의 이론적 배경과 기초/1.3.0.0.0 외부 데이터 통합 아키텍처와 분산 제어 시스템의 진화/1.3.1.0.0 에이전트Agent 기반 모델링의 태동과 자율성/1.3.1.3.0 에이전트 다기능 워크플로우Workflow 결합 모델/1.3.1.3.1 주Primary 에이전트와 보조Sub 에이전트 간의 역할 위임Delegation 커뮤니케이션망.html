1.3.1.3.1 주(Primary) 에이전트와 보조(Sub) 에이전트 간의 역할 위임(Delegation) 커뮤니케이션망

단일 대규모 언어 모델(LLM)이 모든 지식 추론과 도구 호출(Tool Calling)을 전담하던 모놀리식(Monolithic) 에이전트 시대는 엔터프라이즈의 거대한 복잡성 앞에서 빠르게 붕괴되었다. 하나의 뇌 구조로 수십 장의 API 명세서를 암기하고 수백 개의 컨텍스트 분기를 감당하려다 보니 심각한 인지 과부하(Cognitive Overload)가 발생했기 때문이다. 이 한계를 극복하기 위해 시스템 공학계는 군대나 기업의 조직도 메커니즘을 모사하여, 통제권을 쥔 ’주(Primary) 에이전트’와 특정 도메인의 작업만을 철저히 수행하는 ’보조(Sub) 에이전트’로 분리하는 계층적 역할 위임(Delegation) 커뮤니케이션망을 창조해 냈다.

0.1 모놀리식(Monolithic) 에이전트의 인지 과부하와 문맥 출혈

어떤 소프트웨어 프로젝트에 “데이터베이스에서 고객 코드를 찾아 결제 상태를 확인한 후, 이메일로 송장을 발송하고 Slack에 보고해“라는 복합 명령이 주어졌다고 치자.

초창기의 설계는 단일 에이전트의 프롬프트 창에 DB 조회 도구, 이메일 발송 도구, Slack 연동 도구의 방대한 스키마(Schema)와 각 API의 에러 처리 매뉴얼을 통째로 욱여넣는 방식을 취했다.
이는 에이전트의 컨텍스트 윈도우(Context Window) 예산을 극심하게 갉아먹는 ’맥락 출혈(Context Bleeding)’을 야기할 뿐만 아니라, 모델의 어텐션(Attention)을 극도로 혼란스럽게 만든다. 결과적으로 에이전트는 결제 코드를 찾기도 전에 송장부터 빈 이메일로 쏘아버리거나, Slack 접속 시크릿 키를 이메일 본문에 유출하는 등 치명적인 제어 상실(Control Loss) 환각에 빠져들게 된다. 한 번에 너무 많은 도구를 쥐여준 대가로 에이전트의 추론 정확도가 0에 수렴하는 이른바 ’능력 임계점(Capability Threshold)’에 부딪힌 것이다.

0.2 분업의 정립: 주(Primary) 에이전트와 보조(Sub) 에이전트 계층망

이 임계점을 뚫기 위해 도입된 아키텍처는 마이크로서비스(Microservices)의 철학을 에이전트 두뇌에 이식한 ’다중 에이전트 오케스트레이션(Multi-Agent Orchestration)’이다.

이 체계 안에서 리더 역할을 하는 주(Primary) 에이전트는 외부 도구(API)를 직접 만지는 등 육체노동을 하지 않는다. 주 에이전트는 뛰어난 통찰력을 갖춘 거대 모델(예: GPT-4, Claude 3 Opus)로 구성되며, 오직 사용자의 거시적인 의도를 분석(Reasoning)하고 전체 워크플로우(Workflow)를 설계하는 컨트롤 타워 역할에만 집중한다.
반면, 리더의 명령을 하달받는 보조(Sub) 에이전트들은 덩치가 작은 대신 특정 도메인에 훈련이 집중된 특화 모델들로 이루어진다. “DB Search Agent”, “Email Agent”, “Slack Agent” 처럼 각자의 영역이 나뉘어 있으며, 이들에게는 자신이 맡은 단 1개의 도구 명세서만 컨텍스트에 할당되므로 환각이나 실수 없이 완벽하게 백업 임무를 완수할 수 있다.

이처럼 역할을 철저히 분산시킴으로써 시스템은 거대한 복잡성을 쪼개어 정복(Divide and Conquer)하는 고도의 인지 최적화망을 성취한다.

0.3 Model Context Protocol(MCP) 기반의 위임(Delegation) 통신 프로토콜

그러나 에이전트들을 수십 개로 쪼개 놓으면 필연적으로 마주하는 새로운 골칫거리는 “이 지능형 부품들끼리 도대체 무슨 언어로 소통할 것인가?“라는 M2M(Machine-to-Machine) 통신 불협화음 문제다. 서로 다른 회사가 만든 언어 모델 간에 프롬프트와 JSON을 주고받는 규격이 통일되지 않으면, 시스템은 곧장 통합의 지옥(Integration Hell)에 함몰된다. 이 절망적인 소통의 단절을 강제로 이어 붙이는 우주 통용어(Lingua Franca)가 바로 Model Context Protocol(MCP)이다.

sequenceDiagram
    participant User as Human User
    participant P_Agent as Primary Agent <br>(Orchestrator)
    participant MCP as MCP Middleware <br>(Router)
    participant S_Agent1 as Sub Agent 1 <br>(DB Searcher)
    participant S_Agent2 as Sub Agent 2 <br>(Email Sender)

    User->>P_Agent: 사용자 지시 방출
    Note over P_Agent: 목표 추론 및 작업 분해
    
    P_Agent->>MCP: 1. 위임 통신: [Call: Sub_Agent_1(task="조회")]
    MCP->>S_Agent1: 2. MCP 규격으로 Sub-task 할당
    
    Note over S_Agent1: 국소적 도구 실행 (DB 접근)
    
    S_Agent1-->>MCP: 3. 결과 반환 (JSON Payload)
    MCP-->>P_Agent: 4. 정규화된 Observation 전달
    
    Note over P_Agent: 데이터 확인 및 2단계 돌입 
    
    P_Agent->>MCP: 5. 위임 통신: [Call: Sub_Agent_2(data="송장")]
    MCP->>S_Agent2: 6. 이메일 템플릿 작성 명령
    S_Agent2-->>MCP: 7. 발송 완료 Code 반환
    MCP-->>P_Agent: 8. 종합 상황 보고
    
    P_Agent->>User: 최종 임무 완료 보고

MCP의 생태계 안에서는 보조(Sub) 에이전트 그 자체가 하나의 ’MCP 도구(Tool)’로써 취급된다. 즉, 주(Primary) 에이전트의 시야에는 DB 서치 모델이나 이메일 발송 모델들이 거대한 자율성을 띤 생명체가 아니라 Call_DB_Agent(), Call_Email_Agent() 라는 깔끔한 형태의 MCP 함수 서명(Function Signature)으로 노출(Expose)되는 것이다.

주 에이전트는 필요할 때 해당 함수를 호출하며 하위 에이전트에게 위임장(Delegation payload)을 던지고, 결과를 대기한다. 중앙의 MCP 미들웨어는 이 에이전트 간의 호출(Agent-to-Agent Call)이 오갈 때마다 중구난방인 파라미터 규격을 JSON-RPC 2.0 표준 규격으로 억지로 욱여넣어 깔끔하게 포장(Formatting)하고 전달하는 거대한 통신 교환기(Switchboard) 역할을 완수한다.
그 결과, 에이전트 간의 의사소통은 번역의 파편화를 극복하고, 언제나 예측 가능한 표준화된 망 위에서 자유로이 역할 위임 제국을 세울 수 있게 되었다.
