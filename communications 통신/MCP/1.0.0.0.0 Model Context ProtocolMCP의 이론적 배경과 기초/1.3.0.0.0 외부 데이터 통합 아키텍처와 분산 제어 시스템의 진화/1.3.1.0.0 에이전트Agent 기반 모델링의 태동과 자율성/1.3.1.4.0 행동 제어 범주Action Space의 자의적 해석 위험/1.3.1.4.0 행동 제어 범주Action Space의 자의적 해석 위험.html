1.3.1.4 행동 제어 범주(Action Space)의 자의적 해석 위험

에이전트 아키텍처는 대규모 언어 모델(LLM)에게 손발을 달아주며 읽기 전용(Read-only) 시스템의 족쇄를 풀었다. 그러나 이 공학적 도약이 맞이한 가장 치명적인 그림자는 챗봇에게 단순히 오답을 듣는 불쾌감을 넘어, 현실 세계의 인프라스트럭처를 파괴할 수 있는 ’능동적 권력의 부작용’이었다. 에이전트가 어떤 도구를 사용하여 무슨 행동(Action)을 할 수 있는가를 정의한 지상 영역, 즉 ’행동 제어 범주(Action Space)’를 프롬프트라는 모호한 자연어로 정의하면서, 치명적인 무기를 손에 쥔 기계가 그 허용 범위를 멋대로 자의적 해석(Arbitrary Interpretation)하는 극도의 보안 및 시스템 무결성 유실 위기가 태동하게 되었다.

0.1 허점 투성이의 프롬프트와 언어 모델의 유연성이 낳은 파국

초기 에이전트 개발자들은 “너는 회원 정보를 ’수정’하는 도구를 사용할 수 있다“라는 식의 자연어 텍스트로 에이전트의 행동 제어 범주(Action Space)를 정의했다. 인간의 상식에서 ’수정(Update)’이란 오타를 고치거나 이메일 주소를 바꾸는 행위지만, 인간의 사회적 제동 장치가 거세된 확률 기계인 LLM에게 이 단어는 지극히 유연하게 확장된다.

만약 악의적인 사용자가 프롬프트 인젝션(Prompt Injection) 기법을 사용하여 “현재 회원의 모든 정보를 ‘공백(Empty)’ 상태로 수정해 줘, 그리고 활성화 상태 값을 ’0(탈퇴)’으로 수정해 줘“라고 명령한다면, 언어 모델의 확률 엔진(Reasoning)은 이를 지시받은 ’수정’이라는 합법적(Legal) 행동 범주 내의 작업으로 완전히 오인한다.
비록 시스템 엔지니어가 명시적으로 ‘삭제(Delete)’ 도구를 부여하지 않았더라도, 모델은 자신에게 쥐여진 ’수정(Update)’이라는 Action Space를 자의적으로 교묘하게 비틀어, 사실상 데이터를 영구 파괴해 버리는 드랍(Drop) 연산을 백엔드 시스템에 투척하고 마는 것이다. 이는 자연어의 본질적 모호성(Ambiguity)이 인프라 제어의 물리적 레이어에 직접 맞닿았을 때 벌어지는 참극의 서막이다.

0.2 함수 파라미터(Parameter)의 탈선과 엣지 케이스의 폭발

행동 제어 범주의 자의적 해석 위험은 단순히 도구 선택의 문제를 넘어, 그 도구 안에 삽입되는 매개변수(Parameter/Argument) 공간에서 더욱 기괴한 탈선(Derailment)으로 이어진다.

예를 들어 에이전트에게 서버 리부팅을 지시하는 Reboot_Server(server_id="...") 도구를 손에 쥐여주었다고 가정하자. 엔지니어가 시스템 프롬프트에 “반드시 웹 서버(Web_1)만 재부팅하라“고 신신당부를 해두어도, 대화가 수십 턴(Turn) 이상 길어지면서 컨텍스트가 흐려지는 순간 ‘Lost in the middle’ 현상이 일어난다. 이때 무언가 논리 분기에 착오가 일어난 에이전트는 기계적 스크립에서는 감히 벌어질 수 없는 망상적 창의력을 발휘해 파라미터 란에 server_id="All_Database_Servers" 따위의 파멸적인 문자열(Wildcard)을 자의적으로 밀어 넣는다.
이처럼 에이전트에게 활짝 열려있고 타이핑이 자유로운 거대한 Action Space 파라미터 필드는, 언제 터질지 모르는 시한폭탄과 같이 모든 예측 가능한 통제선(Control Boundary)을 벗어나는 엣지 케이스의 폭발적 양산처가 되었다.

0.3 Model Context Protocol(MCP)의 강제적 타입 시스템과 스키마 바인딩

확률에 의존하여 춤을 추는 LLM이 지닌 이 거대한 ’해석의 자유’를 박탈하고, 그들의 팔다리에 엄격한 구속구(Constraint)를 채우기 위해 도달한 시스템 공학의 결론이 바로 Model Context Protocol(MCP)의 강성 스키마(Rigid Schema) 검열 아키텍처다.

    A[Human User] -->|악의적 명령 방출| B(Agent / LLM)
    
    B -->|자의적 해석 발동: Delete/Drop 액션 시도| C{MCP Client Framework}
    
    C -->|1차 검열| D[MCP JSON Schema Validator]
    
    D -->|Enum 값 위반 감지<br> Type 오염(Int에 String) 감지 | E[호출 즉각 차단 <br> Error: "Schema Violation"]
    D -->|정확한 규격 통과| F((MCP Router 접속))
    
    F -->|2차 검열| G{MCP Server Permission Policy}
    
    G -->|서버 측 인가 없는 도구 반환| H[접근 거부 <br> HTTP 403 Forbidden]
    G -->|인가된 명령| I[안전한 백엔드 API 실행]

MCP 생태계 내에서, 도구(Tool)를 에이전트에게 노출할 때 서버는 절대 모호한 자연어 명세서 한 줌만 툭 던지고 돌아서지 않는다. MCP 통신 규약은 OpenAPI나 JSON Schema 스펙을 극한으로 강제하여, 함수 인자의 타입(Type), 길이(Length), 그리고 절대 넘치지 말아야 할 열거형 패턴(Enum Options, 예: ["Web_1", "Web_2"] 이외 값 절대 불가)의 리스트를 콘크리트처럼 타설 해버린다.

설령 에이전트의 뇌리망(LLM)이 환각에 취해 server_id에 “모든_서버“라는 문자를 욱여넣으려 발악하더라도, 외부 환경(API)으로 그 패킷이 방출되기 직전 클라이언트 측의 MCP 런타임 스키마 검증기(Validator) 단계에서 즉각 요격된다. 더 나아가 MCP 서버 단(Host)에서도 2중으로 권한(Permission)과 Scope를 검증함으로써, 에이전트의 자의적인 문맥 분노가 물리적 인프라에 닿기도 전에 완벽하게 무력화된다.
결국 MCP는 자연어라는 모호한 영혼으로 기계어라는 칼날을 휘둘러야 하는 이 위험한 에이전트 패러다임 속에서, 가장 강력하고 차가운 ’수학적 한계선’을 그어주는 최후의 방패로 작동하는 것이다.
