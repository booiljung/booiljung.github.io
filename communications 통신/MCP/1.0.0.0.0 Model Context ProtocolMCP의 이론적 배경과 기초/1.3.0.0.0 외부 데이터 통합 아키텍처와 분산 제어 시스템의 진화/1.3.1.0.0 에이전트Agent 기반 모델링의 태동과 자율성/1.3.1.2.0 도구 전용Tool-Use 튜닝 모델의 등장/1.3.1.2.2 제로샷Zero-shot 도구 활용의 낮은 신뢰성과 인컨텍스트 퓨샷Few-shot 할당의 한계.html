1.3.1.2.2 제로샷(Zero-shot) 도구 활용의 낮은 신뢰성과 인컨텍스트 퓨샷(Few-shot) 할당의 한계

대규모 언어 모델(LLM)을 외부 시스템의 톱니바퀴로 연동하기 위해 업계가 맞닥뜨린 가장 잔혹한 현실은, 언어 모델의 ‘제로샷(Zero-shot) 함수 호출’ 능력이 실무에 투입하기에는 절망적일 정도로 신뢰성이 낮다는 사실이다. 시스템 엔지니어들은 이 거친 지능망을 길들이기 위해 엄청난 양의 예제를 프롬프트 안에 밀어 넣는 퓨샷(Few-shot) 전략에 매달렸으나, 이는 곧장 컨텍스트 윈도우 폭발이라는 또 다른 거대한 구조적 한계와 충돌하고 말았다.

0.1 제로샷(Zero-shot) 도구 호출의 헛발질과 파괴적 환각

제로샷 도구 활용(Zero-shot Tool Use)이란, 에이전트에게 도구의 이름과 매개변수(Parameter) 스키마만 텍스트로 던져주고 “이 규칙만 보고 완벽한 JSON을 알아서 생성해 보아라“고 명령하는 극단적인 위임(Delegation) 방식이다.

이 방식은 학술적 데모에서는 그럴싸하게 작동하지만, 수백 개의 엣지 케이스(Edge Case)가 난무하는 엔터프라이즈 환경에서는 철저히 붕괴한다.
예를 들어 BookTicket(destination, date, class_type) 이라는 도구를 주면, 범용 모델은 “도착지는 파리, 날짜는… 음 오늘이 무슨 요일이지? 그냥 ’내일’이라고 쓰면 알아듣겠지?” 라며 {"destination": "파리", "date": "내일", "class_type": "퍼스트"}처럼 시스템이 절대 파싱할 수 없는 자연어 구조의 인자(Argument)를 제멋대로 꽂아 넣는 환각(Hallucination)을 일으킨다.
시스템 파서는 당연히 “날짜 형식이 YYYY-MM-DD가 아닙니다“라며 서버 에러(HTTP 400)를 반환하고, 에이전트는 혼란에 빠져 동일한 오류를 무한 반복하는 ‘통제권 상실’ 상태로 추락한다. 제로샷의 뇌 구조는 오직 문자열의 통계적 확률에 의존하므로, 정밀한 기계적 타이핑(Typing) 규약을 단 한 번의 지시문만으로 준수할 수 있는 구조적 뼈대가 원천 결여되어 있다.

0.2 인컨텍스트 퓨샷(In-context Few-shot)의 딜레마와 맥락 출혈

이러한 제로샷의 멍청함을 극복하기 위해 프롬프트 엔지니어링 학계가 내놓은 처방전이 바로 ’인컨텍스트 퓨샷(In-context Few-shot) 할당’이다. 이는 모델이 잘못된 길로 빠지지 않도록, “네가 어떻게 대답해야 하는지 모범 답안 5개를 미리 보여줄게“라며 프롬프트의 머리맡에 거대한 예제 스키마를 박아 넣는 행위다.

System Prompt: 너는 티켓 예약 에이전트야. 아래의 예시를 반드시 따르라.
[Example 1]
User: "내일 파리 가는 1등석 부탁해"
Action: {"tool": "BookTicket", "args": {"destination": "Paris", "date": "2024-11-01", "class_type": "First"}}

[Example 2]
User: "모레 런던 가는 이코노미"
Action: {"tool": "BookTicket", "args": {"destination": "London", "date": "2024-11-02", "class_type": "Economy"}}
...
(이러한 예시가 10개 이상 주입됨)

확실히 퓨샷(Few-shot)은 LLM이 규격을 엄수하게 만드는 가장 직관적이고 훌륭한 교정기(Brace) 역할을 수행한다. 그러나 비극은 시스템이 거대해질 때 발생한다.
기업용 에이전트가 다뤄야 할 도구의 개수가 1개가 아니라 인사, 재무, 일정, 메일 등 50개가 넘어가는 순간, 각 도구의 에러 케이스를 커버하기 위한 퓨샷 예제 토큰(Token)의 양은 기하급수적으로 폭발한다. 본 질문을 하기도 전에, 에이전트의 컨텍스트 윈도우(Context Window)는 수만 개의 퓨샷 예시 JSON 텍스트 덩어리로 꽉 차버리는 이른바 ‘맥락 출혈(Context Bleeding)’ 사태가 발생한다.
이는 모델의 추론(Inference) 비용을 폭증시킬 뿐만 아니라, 어텐션 메커니즘을 교란해 정작 중요한 사용자의 진짜 질문(User Query)을 모델이 망각해 버리는 치명적인 성능 저하의 주범으로 전락하고 만다.

0.3 Model Context Protocol(MCP)을 통한 동적 명세 주입과 규격 강제화

퓨샷(Few-shot)을 무한정 넣을 수 없는 컨텍스트 예산 창의 한계선 앞에서, 인프라스트럭처가 도출해 낸 최상위 추상화 아키텍처가 바로 Model Context Protocol(MCP)이다.

MCP 생태계는 에이전트의 프롬프트 안에 미리 50개 도구의 거대한 퓨샷 매뉴얼을 전부 쑤셔 박아놓는 구시대적 야만성을 거부한다.
대신, MCP 라우터는 대화가 진행되는 동안 ‘현재 상황에 가장 필요할 것 같은’ 1~2개의 도구 명세(Resources/Tools Schema)만을 실시간으로 핀셋처럼 집어내어 에이전트의 단기 컨텍스트(Short-term context)에 동적으로 할당(Dynamic Context Injection)한다.

더욱 중요한 것은, MCP의 통신 규약 자체가 클라이언트(LLM)와 호스트 서버 간의 견고한 JSON-RPC 통신 방어막을 친다는 점이다. 에이전트가 제로샷의 한계로 인해 날짜 포맷을 틀리게 호출하더라도, MCP 미들웨어가 이를 가로채어 InvalidParams JSON-RPC 에러와 함께 명확한 교정 예시를 덧붙여 반환하는 런타임 자체 교정(Runtime Self-Correction) 루프를 지원한다.
결국 에이전트는 무식하게 부풀어 오른 퓨샷 프롬프트의 짐을 짊어질 필요 없이, 가벼운 제로샷(Zero-shot) 혹은 단일 샷(One-shot)만으로도 가벼운 몸싸움을 시작하고, 틀릴 경우 MCP 서버가 찰나의 피드백을 주어 궤도를 수정하게 만드는 ‘가장 가볍고, 가장 정밀하게 피드백되는’ 이상적인 자율 제어 통신망을 성취하게 된 것이다.
