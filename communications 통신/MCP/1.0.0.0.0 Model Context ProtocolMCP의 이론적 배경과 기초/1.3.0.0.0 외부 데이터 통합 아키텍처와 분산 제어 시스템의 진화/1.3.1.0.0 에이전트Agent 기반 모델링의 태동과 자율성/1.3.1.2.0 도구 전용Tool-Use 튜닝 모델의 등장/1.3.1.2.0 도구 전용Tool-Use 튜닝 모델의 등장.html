1.3.1.2 도구 전용(Tool-Use) 튜닝 모델의 등장

에이전트(Agent) 아키텍처의 초기 단계에서 시스템 엔지니어들은 엄청난 장벽에 부딪혔다. 철학적 시/소설이나 대화형 텍스트에 강점을 가진 범용 대규모 언어 모델(General-purpose LLM)들은 인간과 대화하는 데는 탁월했지만, 기계(Machine)와 대화하기 위한 엄격한 문법, 즉 ‘API 호출(JSON 형식 등)’ 구조를 생성하는 데는 처참한 실패율을 보여주었기 때문이다. 이에 따라 산업계는 오직 도구를 자유자재로 다루고 외부 환경 코드를 완벽하게 호출하는 데에만 특화된 ’도구 전용(Tool-Use/Tool-Calling) 파인 튜닝 모델’이라는새로운 하위 분기가 탄생시켰다.

0.1 범용 언어 모델의 ’자연어 본능’과 기계어 생성의 충돌

기존의 범용 언어 모델은 거대한 코퍼스(Corpus)에서 자연어의 확률 변동성을 극대화하도록 훈련(Training)되었다. 그 결과, 에이전트 파이프라인 안에서 모델에게 “이제 너는 검색 API를 호출해야 해. 반드시 {“query”:“날씨”, “location”:“서울”} 형태의 JSON만 반환해.“라고 수십 번 프롬프트로 협박하더라도, 모델은 걸핏하면 본능에 이끌려 “알겠습니다! 제가 검색을 해드릴게요. {“query”:“날씨”…} 어때요? 도움이 되셨나요?“라는 식의 다정한 자연어 꼬리말을 억지로 이어붙이는 치명적 일탈을 감행하곤 했다.

시스템이 요구하는 것은 파싱(Parsing)이 가능한 100% 무결점의 기계어 코드 덩어리였지만, 모델은 자꾸 인간과 수다를 떨려고 했다. 따옴표 하나, 쉼표 하나의 누락이나, 여분의 마크다운(Markdown) 백틱(`) 삽입만으로도 외부 API 서버는 즉각 400 Bad Request 에러를 내뿜으며 파이프라인 전체를 무너뜨렸다. 결국 범용 텍스트 생성이라는 설계 사상 자체가 에이전트의 견고한 도구 사용(Tool Use) 신뢰성을 저해하는 근본 원인이었음이 입증된 것이다.

0.2 함수 호출(Function Calling)에 강제 편향된 학습 구조의 도래

이 결함을 부수기 위해, 언어 모델 제조사들은 아예 ‘기계와의 통신(Function Calling)’ 텍스트 구조만을 수억 번 반복 학습시킨 전용 특화 모델(예: Gorilla, ToolLLaMA 등)을 구워내기 시작했다.

도구 전용(Tool-Use) 튜닝 모델의 학습(Fine-tuning) 데이터셋은 셰익스피어의 희곡이나 위키백과가 아니라, GitHub의 방대한 코딩 저장소와 웹상의 수백만 개가 넘는 REST API Swagger 명세서, OpenAPI JSON 스키마 구조로 꽉꽉 채워져 있다. 이 모델들은 ’사용자의 의도’를 이해하는 추론 코어는 살려두되, 답변을 생성하는 출력부(Output Layer)의 확률 분포를 철저하게 JSON이나 파이썬(Python) 함수 서명 구조로 쏠리게(Biased) 강제 조율해 둔 변종들이다.

이 도구 특화 모델들은 사용자 프롬프트 안에 수십 개의 도구(Tool) 명세서가 동시에 쑤셔 넣어져도 혼돈에 빠지지 않으며, 적절한 도구 하나를 귀신같이 픽킹(Picking)해내어 단 1비트의 자연어 쓰레기 지저귐도 섞지 않은 순수한 기계어 문맥만을 송출하는 궁극의 ’도우미 부품(Proxy Component)’으로 자리 잡게 되었다.

0.3 Model Context Protocol(MCP) 생태계 핵심 엔진으로서의 위상

이 도구 전용 모델들의 도래는, 외부 세상의 모든 도구와 데이터를 표준화된 명세로 엮어버리겠다는 야심을 가진 Model Context Protocol(MCP) 설계 사상과 완벽한 톱니바퀴를 이루며 결합한다.

MCP 생태계 내에서, MCP 클라이언트는 여러 대의 개별 컴퓨터가 연결된 시스템과 수백 개의 다양한 MCP 도구(Tool) 스키마를 클라이언트(LLM) 쪽으로 쏟아낸다. 이때 이 막강한 도구의 명세서(Schema)들을 일말의 환각(Hallucination) 없이 정확히 읽어내고 그 구멍 모양에 딱 맞는 인자(Argument) 블록을 찍어내는 금형틀의 역할은, 오직 뼈를 깎는 수련을 거친 Tool-Use 튜닝 모델만이 해낼 수 있다.

현대의 첨단 MCP 아키텍처는 이제 단일 모델에 의존하지 않고, 전체 상황 대화 흐름(Chat)을 주관하는 ’범용 모델’의 메인 스레드 위에, 순수하게 MCP 도구의 함수 호출 명령만을 파싱하고 조립하는 ’Tool-Use 특화 모델(Sub-agent)’을 부품처럼 결합해 쓰는 다중 모델-오케스트레이션(Multi-model Orchestration)의 화려한 시대를 꽃피우게 되었다.
