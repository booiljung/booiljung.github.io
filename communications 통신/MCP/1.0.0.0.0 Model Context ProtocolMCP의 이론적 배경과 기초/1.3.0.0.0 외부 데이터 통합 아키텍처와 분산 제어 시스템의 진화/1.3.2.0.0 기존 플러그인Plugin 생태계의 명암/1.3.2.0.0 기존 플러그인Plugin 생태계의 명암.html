1.3.2 기존 플러그인(Plugin) 생태계의 명암

대규모 언어 모델(LLM)이 밀폐된 파라미터 텍스트의 감옥을 부수고 웹 검색과 API 호출이라는 외부 세계와의 연결 고리를 처음 획득했던 그 역사적 단초는 ‘플러그인(Plugin)’ 시스템에서 비롯되었다. OpenAPI 스펙의 선언적 명세서를 들고 시스템의 도구 상자에 플러그를 꽂듯 LLM과 외부 서비스를 연결하려 했던 이 거대한 사상적 실험은, 초기 에이전트(Agent) 아키텍처의 폭발적 부흥을 이끌었으나 동시에 극복할 수 없는 기술적 파편화와 본질적 폐쇄성의 그림자를 짙게 드리웠다.

0.1 연결의 혁명: 매니페스토(Manifest) 기반의 지능형 API 라우팅

모델에게 손발을 처음으로 달아준 플러그인 아키텍처의 혁명성은, 그 ’은밀한 선언주의’에 있었다.
과거에는 외부 날씨 정보를 LLM과 연결하려면 자연어 처리 파이프라인(NLP)과 날씨 API 통신 로직을 백엔드에서 일일이 수작업으로 매핑(Hard-coding Binding)해야 했다. 그러나 ChatGPT 플러그인 등으로 대변되는 플러그인 생태계는 ai-plugin.json과 같은 단순한 매니페스토(Manifesto) 파일 하나와, RESTful 규격을 서술한 OpenAPI.yaml 스펙 문서만 던져주면, 언어 모델이 문서의 내용을 스스로 읽고 이해하여 정확한 API 호출(Function Calling) 체계를 알아서 조립해 버리는 기적을 연출했다.

이 선언적 모델링의 도입은 소프트웨어 벤더(Vendor)들에게 가히 충격적인 경험이었다. 익스피디아(Expedia), 클라르나(Klarna) 같은 기업들은 자신들의 기존 레거시(Legacy) 백엔드 API를 전혀 뜯어고치지 않고, 단지 “우리의 도구는 비행기 표를 검색합니다“라는 자연어 설명서(Description) 한 장만 모델의 프롬프트 윈도우에 밀어 넣음으로써 자사의 서비스 생태계를 전 세계의 챗봇 인터페이스와 결합시킬 수 있었다. 플러그인은 언어 모델을 파라미터 덩어리에서 ’의도를 해석하는 거대한 분배기(Intelligent Router)’로 격상시켰다.

0.2 고착된 락인(Lock-in)과 승자 독식 생태계의 파편화

그러나 플러그인 붐의 찬란함 이면에는 치명적인 프로토콜 파편화(Fragmentation)와 승자 독식의 플랫폼 락인(Lock-in)이라는 어두운 마수가 숨어 있었다.

업계를 주도하던 OpenAI가 ChatGPT 플러그인의 규격을 독단적으로 정의함에 따라, 세상의 모든 데이터 프로바이더와 도구 제작자들은 오로지 OpenAI 모델의 구미에만 맞는 전용 JSON 포맷과 인증 규격을 하드코딩해야 했다.
만약 동일한 익스피디아 플러그인을 Anthropic의 Claude 모델이나, Google의 Gemini 시스템에 붙이고 싶다면? 작동하지 않았다. 각기 다른 LLM 개발사들은 서로를 배척하며 고유의 함수 호출(Function Calling) 문법과 플러그인 레지스트리(Registry)를 따로 구축했기 때문이다.
결국 이 벤더 종속적인(Vendor-dependent) 생태계에서, 애플리케이션 개발자들은 ‘OpenAI용 플러그인 껍데기’, ‘Claude용 껍데기’, ’Llama용 어댑터’를 각기 다르게 코딩하여 유지 보수해야 하는 통합의 지옥(Integration Hell)에 빠지고 말았다. 이는 진정한 의미의 자율 생태계가 아니라, 단지 소수의 독점적 AI 기업이 거대한 깔때기를 꽂아 전 세계의 데이터를 빨아들이는 종속적 배관망에 불과했다.

0.3 클라이언트 종속성과 단방향 연결의 한계, 그리고 MCP의 태동

더욱 심각한 한계는 이 초기 플러그인 모델이 가진 구조적 ’단방향성(Unidirectionality)’에 있었다.
기존의 플러그인은 철저히 “LLM(클라이언트)이 질문을 던지면 -> 외부 서버 앱이 답을 준다“는 Pull 방식의 단발성 Rest API 호출에 얽매여 있었다.

만약 사용자의 로컬 IDE에서 코드가 실시간으로 수정되는 이벤트를 감지하여 챗봇에게 즉시 푸시(Push) 알림을 주거나, 매우 복잡한 데이터베이스 백엔드 서버가 먼저 챗봇에게 “지금 DB 락(Lock)이 걸렸으니 너의 이전 지시를 철회하라“고 상황을 양방향으로 동기화(Synchronization)하는 것은 애초에 시스템 명세서(Schema) 설계를 벗어난 불가능한 행위였다.

이 고립되고 경직된 플러그인 모델의 명암을 뼈저리게 목도한 시스템 엔지니어들은, 더 이상 특정 AI 회사의 입맛에 맞는 플러그인을 각출하는 종속적 생태계를 거부하기 시작했다.
언어 모델 개발사에 상관없이(Agnostic), TCP/IP나 HTTP처럼 세상의 모든 호스트 프로그램(클라이언트)과 데이터 제공자(서버)가 완전히 독립적이고 양방향으로 자유롭게 소통할 수 있는 우주 공통의 ’자율형 연결망’이 절실히 요구되었다. 그 참혹한 기술적 파편화의 폐허를 딛고, 분산형 인지 통합망의 차세대 만국 공통어로서 제안된 거대한 패러다임이 바로 **Model Context Protocol(MCP)**인 것이다.
