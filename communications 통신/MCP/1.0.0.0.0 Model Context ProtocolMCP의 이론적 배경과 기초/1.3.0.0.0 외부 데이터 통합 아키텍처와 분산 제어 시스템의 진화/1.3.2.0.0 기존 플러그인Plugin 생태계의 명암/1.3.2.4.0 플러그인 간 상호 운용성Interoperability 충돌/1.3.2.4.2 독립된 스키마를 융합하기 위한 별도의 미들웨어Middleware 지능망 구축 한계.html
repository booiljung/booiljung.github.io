1.3.2.4.2 독립된 스키마를 융합하기 위한 별도의 미들웨어(Middleware) 지능망 구축 한계

기존 플러그인(Plugin) 생태계에서 발생하는 치명적인 매개변수 충돌과 스키마(Schema) 불일치를 극복하고자, 엔터프라이즈 엔지니어들은 플러그인과 대규모 언어 모델(LLM) 사이에 위치하는 거대한 ’미들웨어 지능망(Middleware Intelligence Layer)’을 강제적으로 구축하기 시작했다. 이는 서로 말이 통하지 않는 두 서드파티 플러그인 사이에서 데이터를 변환(Parsing & Formatting)해 주는 랭체인(LangChain) 같은 파이프라인(Pipeline) 코드를 일일이 하드코딩(Hard-coding)하는 사후약방문식 해결책이었으나, 이내 아키텍처의 복잡성을 기하급수적으로 폭발시키며 자체적인 한계에 다다랐다.

0.1 하드코딩된 접착 코드(Glue Code)와 유지보수성의 종말

두 개의 독립된 플러그인, 즉 사내 ‘고객 데이터베이스(Custom DB)’ 플러그인과 외부 ‘메일 발송(Mail API)’ 플러그인이 있다고 가정하자.
고객 DB 플러그인은 고객 정보를 {"user_identifier": "A123", "email_addr": "test@test.com"}이라는 커스텀 JSON 규격으로 반환한다. 반면 메일 발송 플러그인은 받는 사람의 정보를 오직 {"recipient": "test@test.com"} 형태로만 수신할 수 있다.

이 스키마 간의 틈을 메우기 위해, 개발자는 시스템 중간에 “DB 플러그인의 결괏값 중 email_addr 필드의 값을 읽어서, 다시 메일 플러그인의 recipient 파라미터로 매핑(Mapping)하라“는 접착 코드(Glue Code)를 짜야만 했다.
플러그인의 개수가 10개로 늘어나면 접착망의 경우의 수는 10 \times 9, 즉 90개의 파이프라인(경우의 수)이 필요해진다. 만약 외부 메일 API 스키마가 업데이트라도 되는 날에는 이 미들웨어 파이프라인 전체가 도미노처럼 붕괴하며, 개발팀은 끊임없이 중개 코드를 수정해야 하는 ’결합의 지옥(Coupling Hell)’에 빠져버리고 말았다.

0.2 미들웨어 계층의 컨텍스트(Context) 병목과 환각 전이

하드코딩된 접착 코드의 부하를 줄이기 위해, 일각에서는 또 다른 초경량 LLM을 중개자(Broker LLM)로 배치하여 “네가 알아서 A 플러그인의 출력을 보고 B 플러그인의 입력 양식으로 변환해서 쏴라“고 지시하는 자율 라우팅(Autonomous Routing) 지능망을 구상했다.

그러나 이 미들웨어 지능망은 오히려 시스템 네트워크에 치명적인 ’구조적 지연(Latency)’과 ‘환각 전이(Hallucination Cascading)’ 현상을 몰고 왔다.
A 도구의 결괏값을 B 도구에 온전히 전달하기 위해 중간 라우터 LLM이 개입하는 순간, 매번 수천 토큰(Token)의 API 호출 스레드가 추가로 낭비된다(추론 비용의 2배 증가). 더 큰 문제는, 중간 LLM이 A 로데이터(Raw Data)를 B 형식으로 변환하는 과정에서 데이터를 누락하거나 제멋대로 요약해 버리는 환각(Hallucination)이 발생하면, 결과적으로 제일 마지막 단계의 도구가 ’오염된 파라미터’를 전달받아 파괴적인 행위(예: 잘못된 이메일 주소로 기밀 문서를 발송)를 저지르는 원인이 되었다.

0.3 Model Context Protocol(MCP)의 통합 런타임 호스트(Unified Runtime Host) 패러다임

이러한 미들웨어의 한계점, 즉 통제할 수 없는 스파게티 접착망(Spaghetti Glue Code)과 변환 중개 LLM의 환각 문제를 근원적으로 소거하기 위해 Model Context Protocol(MCP)은 도구 간의 변환을 외부 지능 미들웨어에 위임하지 않고, ’통합 런타임 클라이언트 호스트(Unified Runtime Client Host)’가 그 직접적인 통제 권한을 단일화하는 아키텍처를 제시했다.

MCP 환경에서는 A 도구와 B 도구가 서로의 존재를 억지로 알 필요도, 중간에 어설프게 데이터를 포맷팅(Formatting) 할 별도의 중개 LLM(Broker)을 둘 필요도 없다.

사용자가 구동하는 거대한 ‘MCP Client(호스트 앱)’ 자체가 모든 MCP 서버들의 스키마(Schema)를 JSON-RPC 2.0이라는 단일한 국제 표준으로 통일성 있게 읽어 들이고 통합 관리한다. 호스트 앱은 A 도구에서 반환된 원본(Raw Data)을 단 한 바이트의 손실 없이 캡슐화하여 단일 메인 LLM에게 전달하며, LLM은 전체 시스템의 규격화된 맥락(Context)을 완벽히 인지한 상태에서 B 도구를 호출할 정확한 구조체(Structure)를 호스트에게 지시한다.

결국 MCP는 “A의 스키마를 B에 맞게 어거지로 변환하는 외부의 지능망“을 폐기하고, 수십 개의 독립된 도구를 마치 하나의 운영체제(OS) 안에 탑재된 로컬 함수(Local Function)처럼 단일한 표준망 위에서 매끄럽게 교차 호출할 수 있는 극강의 ’상호 운용성(Interoperability)’을 이룩해 낸 것이다.
