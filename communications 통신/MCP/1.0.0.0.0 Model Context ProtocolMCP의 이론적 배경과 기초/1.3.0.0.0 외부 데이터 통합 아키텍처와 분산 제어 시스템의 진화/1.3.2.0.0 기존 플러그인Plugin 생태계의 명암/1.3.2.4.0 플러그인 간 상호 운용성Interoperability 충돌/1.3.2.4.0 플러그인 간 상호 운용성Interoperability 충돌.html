1.3.2.4 플러그인 간 상호 운용성(Interoperability) 충돌

에이전트(Agent)의 지능이 단조로운 1차원적 문답을 넘어, 여러 도구를 연쇄적으로 호출(Chain-of-Tools)하여 복합적인 과업을 수행하는 다기능 워크플로우(Workflow) 시대로 접어들면서, 기존 플러그인(Plugin) 생태계는 치명적인 기술적 한계에 부딪혔다. 서로 다른 서드파티 개발자들이 고립된 환경에서 만들어낸 플러그인들은, 언어 모델(LLM)의 컨텍스트 윈도우(Context Window) 속에서 함께 구동될 때 마치 좁은 방 안에 여러 야생동물을 몰아넣은 것처럼 예측 불가능한 ’상호 운용성(Interoperability) 충돌’을 일으키며 시스템을 붕괴시켰다.

0.1 스키마(Schema) 고립과 데이터 파이프라인의 단절

두 개 이상의 플러그인이 성공적으로 협업하기 위해서는, A 플러그인의 결괏값(Output)이 B 플러그인의 입력값(Input)으로 매끄럽게 연결되는 파이프라인 형태의 상호 운용성이 보장되어야 한다.

그러나 중앙 스토어에 등록된 기존 플러그인들은 애초에 타 기관의 도구와 데이터를 주고받을 것을 염두에 두고 설계되지 않았다. 가령, 익스피디아(Expedia)의 항공권 검색 플러그인은 공항 코드를 ["JFK", "LAX"] 형태의 문자열 배열로 반환하지만, 뒤이어 이 공항의 날씨를 조회해야 하는 다른 날씨 플러그인은 위도와 경도({"lat": 33.94, "lon": -118.40}) 형태의 JSON 객체를 요구한다.
언어 모델은 이 두 플러그인 사이에서 고군분투하며 데이터를 강제로 변환(Data Munging)해 보려 하지만, 프롬프트 공간의 한계와 두 도구 간의 본질적인 도메인 스키마(Domain Schema) 단절로 인해 환각(Hallucination)을 일으키거나 “잘못된 파라미터가 입력되었습니다“라는 에러와 함께 파이프라인이 붕괴하는 현상이 고질적으로 발생했다. 각각의 플러그인은 거대한 탑의 벽돌이 되지 못하고, 모래알처럼 흩어져 고립된 사일로(Silo)로 남게 된 것이다.

0.2 인지 부하(Cognitive Load) 폭발과 도구 선택의 경합(Race Condition)

에이전트에게 쥐여주는 도구의 개수가 늘어날수록, 충돌의 양상은 시스템 연산 자체를 마비시키는 ’인지 부하(Cognitive Load)’의 문제로 악화된다.

언어 모델의 컨텍스트 윈도우에 50개의 서로 다른 플러그인 명세를 밀어 넣었다고 가정해 보자. 사용자가 “내일 제주도 출장 갈 거야“라고 말하면, 모델은 ‘호텔 예약 플러그인’, ‘항공권 플러그인’, ‘날씨 플러그인’, 심지어 ‘캘린더 플러그인’ 중 어느 것을 제일 먼저, 어떤 순서로 호출해야 할지 결정해야 한다.
각 플러그인의 제조사가 자신들의 도구가 가장 중요하다고 부풀려서 매니페스트(Manifest)를 작성해 두었기 때문에, 프롬프트 내부에서는 수십 개의 도구가 서로 호출(Trigger)해 달라며 매개변수 우선순위 경합(Priority Race)을 벌인다. 이로 인해 모델의 추론(Reasoning) 성능은 급격히 저하되며, 호텔을 먼저 예약하고도 항공권을 찾지 않아 일정이 박살 나거나, 똑같은 API를 무한정 재호출하는 치명적인 오작동(Malfunction) 루프에 빠지게 된다.

0.3 Model Context Protocol(MCP)의 리소스-프롬프트-툴(RPT) 융합 생태계

이 무질서한 독립 플러그인들의 충돌을 제어하기 위해, Model Context Protocol(MCP)은 도구(Tool)라는 단일 개념에 집착하던 구시대적 발상을 버리고, 에이전트 인터페이스를 구성하는 우주를 자원(Resource), 도구(Tool), 프롬프트(Prompt)라는 세 가지 일원화된 체계로 구조화(Categorization)하여 상호 운용성을 확보했다.

MCP 생태계 내부에서는 모든 것이 조화롭고 예측 가능하다.
에이전트가 어떤 데이터를 ’조회’해야 할 때는 위험하게 툴(Tool)을 호출하는 대신, MCP의 Resource 계층에서 표준화된 URI(ex: file:///data/flight_info.json) 포맷으로 안전하게 읽어 들이며 다른 도구와의 데이터 스키마 충돌을 원천 차단한다.

또한, 벤더들이 매니페스트에 억지로 숨겨두었던 편법적 지시어들은 MCP 서버가 명시적으로 제공하는 Prompt 인프라를 통해 표준화된 템플릿(Template)으로 클라이언트에게 공급된다. 마지막으로 외부 시스템의 상태를 변경(Mutate)하거나 복잡한 연산이 필요한 경우에 한해서만 строго하게 격리된 Tool 기능을 호출하게 된다.

이처럼 MCP는 플러그인 간의 무질서한 밥그릇 싸움을 멈추고, 읽기(Resource), 쓰기(Tool), 사고(Prompt)라는 소프트웨어 공학의 본질적 행위들을 분리(Separation of Concerns)함으로써, 수백 개의 서버가 연결되어도 에이전트의 워크플로우 안에서 오작동 없이 한 몸처럼 맞물려 돌아가는 ’마스터 퍼즐(Master Puzzle)’의 청사진을 완성해 낸 것이다.
