1.3.2.1 중앙집중형 플러그인 생태계의 폐쇄성 진단

플러그인(Plugin) 아키텍처는 대규모 언어 모델(LLM)이 외부 시스템과 소통하는 최초의 문을 열었지만, 그 생태계의 구조적 형태는 특정 소수 기업이 통제권을 독점하는 형태인 ‘중앙집중형(Centralized)’ 모델로 굳어졌다. 애플(Apple)의 앱스토어(App Store) 방식을 모방한 이 하향식(Top-down) 배포 구조는 초기 생태계를 통제하는 데는 유용했으나, 엔터프라이즈의 보안성과 확장성을 억압하는 치명적인 폐쇄성(Closedness)의 문제를 야기했다.

0.1 벤더 종속적 카르텔(Vendor-locked Cartel)의 형성

플러그인 기반 아키텍처의 가장 근본적인 문제는 특정 LLM 거대 기업이 스스로 생태계의 문지기(Gatekeeper) 역할을 자처했다는 점이다.

서드파티(Third-party) 개발자가 자신이 만든 사내 데이터베이스 연동 도구를 챗봇(예: ChatGPT)에 붙이고 싶다면, 단순히 코드를 짜는 것을 넘어 해당 벤더(Vendor)가 제시하는 독점적 규격을 엄수해야 했다. JSON 기반의 manifest 파일을 특정 기업의 양식에 맞게 깎아내야 하며, 심지어 코드가 완벽하더라도 해당 기업의 ’플러그인 스토어 심사’를 통과하지 못하면 세상의 빛을 볼 수조차 없었다.
이러한 구조는 필연적으로 카르텔(Cartel)을 형성한다. 개발자들은 범용적인 웹 표준을 따르는 것이 아니라 OpenAI나 Anthropic 등 특정 기업의 입맛에 맞는 플러그인을 중복으로 개발해야 하는 이른바 ’벤더 종속성(Vendor Lock-in)’의 함정에 빠졌다. A 회사의 스마트 비서에서 작동하는 플러그인을 B 회사의 스마트 비서로 이식(Porting)하는 것은 불가능에 가까웠고, 기술의 발전은 플랫폼의 장벽에 갇혀 산산조각이 났다.

0.2 물리적 라우팅의 종속과 로컬 인프라의 배제

더욱 심각한 문제는 구조적으로 모든 데이터가 플랫폼 사업자의 중앙 서버를 반드시 관통해야 한다는 아키텍처적 중앙집중성이다.

초기 플러그인 아키텍처의 구동 흐름을 보면, 사용자가 “내 회사 이메일 요약해 줘“라고 명령할 경우 이메일을 읽어오는 플러그인 연산이 내 PC나 우리 회사의 프라이빗 로컬 망에서 구동되는 것이 아니다. 모든 플러그인의 함수 호출(Function Calling) 패킷은 LLM 개발사의 중앙 클라우드를 거쳐서, 클라우드 서버가 직접 서드파티의 이메일 API 서버를 타격하는 방식으로 이루어졌다.

이러한 중앙집중형 라우팅 구조에서는 망 분리(Network Isolation)가 필수적인 대기업, 금융권, 그리고 국방/의료 시스템의 로컬 환경(On-premise)에서는 에이전트 기술을 도입하는 것이 구조적으로 불가능해진다. 민감한 내부 데이터베이스(DB)를 조회하기 위해서는 우리 회사의 DB 방화벽을 구글이나 OpenAI의 서버 IP를 향해 활짝 열어주어야 한다는 해괴망측한 보안 결론에 도달하기 때문이다. 플러그인 생태계는 태생부터 퍼블릭 클라우드(Public Cloud)의 지배만을 염두에 둔, 극도로 편협하고 폐쇄적인 망상(Network Topology)이었다.

0.3 Model Context Protocol(MCP)을 통한 분산형 직결(Decentralized P2P) 아키텍처로의 해방

이러한 중앙 벤더의 심사관적 통제망과 보안 허혈을 근본적으로 폭파시켜 버린 구원자가 바로 Model Context Protocol(MCP)의 분산형 클라이언트-서버(Client-Server) 아키텍처다.

MCP의 핵심 철학은 “어떤 특정 기업의 스토어 심사도, 클라우드 경유(Routing)도 필요 없다“는 완벽한 분산형 직결성(Decentralization)이다.

MCP 생태계에서는 플러그인을 퍼블릭 스토어에 등록할 필요가 없다.
엔터프라이즈는 스스로 MCP 서버를 로컬에 구축하여 사내 방화벽 안쪽에 숨겨두고, 사용자의 PC 위에서 구동되는 MCP 클라이언트(예: Cursor, Claude Desktop)만을 이 서버에 다이렉트로 연결(stdio 등)시킨다.

이때 저 멀리 퍼블릭 클라우드에 존재하는 거대 LLM(Claude, GPT)은 우리의 데이터를 훔쳐보는 중개자 역할을 하지 않는다. LLM은 단지 클라이언트가 던져준 스키마를 보고 어떻게 추론해야 할지만 알려주는 ’인지적 렌더링 엔진’으로만 동작할 뿐이며, 물리적인 도구(Tool) 호출과 데이터(Resource)의 송수신 트래픽은 오직 사용자 클라이언트와 사내 MCP 서버 간의 P2P(Peer-to-Peer) 채널 안에서만 은밀하게 교환된다.

결국 MCP 프로토콜은 기형적인 벤더 종속성과 중앙집중형 배포의 병폐를 완전히 치료하며, 이 세상 모든 개발자가 10분 만에 자신의 도구를 세상의 모든 LLM과 직결시킬 수 있는 ’개방형 자율 제어 생태계’를 무혈입성시킨 것이다.
