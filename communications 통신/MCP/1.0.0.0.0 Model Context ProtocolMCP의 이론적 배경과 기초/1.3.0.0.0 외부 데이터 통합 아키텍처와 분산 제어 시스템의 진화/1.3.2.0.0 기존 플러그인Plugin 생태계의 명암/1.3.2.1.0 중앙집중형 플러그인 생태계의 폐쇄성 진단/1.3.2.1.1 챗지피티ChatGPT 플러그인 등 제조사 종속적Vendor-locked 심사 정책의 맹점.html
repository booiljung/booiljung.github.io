1.3.2.1.1 챗지피티(ChatGPT) 플러그인 등 제조사 종속적(Vendor-locked) 심사 정책의 맹점

대규모 언어 모델(LLM) 생태계의 패러다임이 단방향 챗봇에서 외부 시스템과 연동하는 플러그인(Plugin) 체제로 전환되던 초기, 가장 큰 스포트라이트를 받았던 것은 OpenAI의 ‘ChatGPT 플러그인 스토어’ 모델이었다. 이는 애플(Apple)의 앱스토어(App Store)가 이룩했던 찬란한 중앙 통제식 배포 구조의 영광을 AI 생태계에 재현하려는 시도였다. 하지만 앱 생태계와 확연히 다른 특성을 지닌 엔터프라이즈 AI 연동망에서, 제조사 종속적(Vendor-locked)인 중앙 심사 정책은 혁신의 가장 강력한 걸림돌로 전락하였으며, 생태계 확장을 질식시키는 모순적 맹점을 적나라하게 드러냈다.

0.1 비정형적 보안 심사와 타임투마켓(Time-to-Market)의 상실

AI 플러그인을 개발하여 중앙 스토어에 배포하기 위해서는 해당 LLM 벤더(Vendor)가 제시하는 까다로운 심사 가이드라인을 통과해야 한다. 문제는 이 심사 과정이 전통적인 소프트웨어 공학의 ’정형 검증’처럼 일관된 테스트 코드로 통과 여부를 결정할 수 있는 성질의 것이 아니라는 점이다.

플러그인이 언어 모델에게 제공하는 자연어 지시문(Prompt Description)이 과연 안전한지, 모델의 프롬프트 인젝션(Prompt Injection)을 유도하지는 않는지, 경쟁사의 이익을 침해하지 않는지를 심사하는 주체는 결국 거대 벤더의 불투명한 내부 정책(Policy)과 휴먼 리뷰어(Human Reviewer)다.
어느 날 갑자기 벤더의 보안 정책이 업데이트되면, 잘 작동하던 플러그인 시스템이 “부적절한 자연어 지시 모델 내포“라는 모호한 이유로 스토어에서 강제 퇴출당(De-listed)한다. 서드파티 개발자들은 이러한 블랙박스 심사(Black-box Review) 결과를 하염없이 기다리며 수주일의 귀중한 타임투마켓(Time-to-Market)을 낭비하게 되고, 혁신적인 도구들이 대중에게 신속히 배포되는 길목이 벤더의 좁은 심사관 파이프라인에 의해 치명적인 병목(Bottleneck) 사태에 직면하게 되었다.

0.2 제조사 종속(Vendor Lock-in) 규격과 다중 플랫폼 유지 비용의 폭발

더욱 심각한 맹점은, 벤더마다 플러그인을 구성하는 데이터 교환 규격(Manifest format)이 극단적으로 파편화되어 있다는 사실이다.

개발자가 하나의 완성된 사내 API를 외부로 노출하고 싶어도, OpenAI의 심사용 ai-plugin.json 포맷을 짜야 하고, Anthropic의 Claude 모델에 붙이려면 그들만의 독자적인 툴 연동 규격을 다시 타이핑해야 하며, 구글(Google)의 에코시스템에 붙이려면 또 다른 구글 전용 Extension 사양을 연구해야 한다.
이는 과거 웹 브라우저 전쟁(Browser War) 시절, 인터넷 익스플로러와 넷스케이프 양측에 맞게 똑같은 웹사이트를 두 번씩 코딩해야 했던 야만적인 시대를 연상시킨다. 영리적인 벤더들이 자사의 생태계 성벽(Walled Garden)을 높이기 위해 의도적으로 프로토콜 간의 상호 호환성(Interoperability)을 틀어막았으며, 결과적으로 서드파티 개발사들은 배보다 배꼽이 큰 유지보수 비용(Maintenance Cost)의 함정에 빠져 플러그인 참여 자체를 포기하는 생태계 공동화(Hollowing-out) 현상을 불러왔다.

0.3 Model Context Protocol(MCP)의 스토어리스(Store-less) 선언주의

이러한 중앙집중형 벤더 통제 시스템의 불합리함을 근원적으로 타파하기 위해 등장한 것이 바로 Model Context Protocol(MCP)의 ‘스토어리스(Store-less) 아키텍처’ 사상이다.

MCP 생태계 내부에는 플러그인을 업로드하거나 심사받기 위한 중앙 스토어가 애초에 존재하지 않는다.
도구를 제공하고 싶은 주체(Data Provider)는 개발 언어가 무엇이든 MCP 표준 명세(JSON-RPC 2.0)를 따르는 MCP 서버를 스스로 구축하여 자신이 원하는 공간(사내 서버, Localhost 등)에 띄워두기만 하면 된다.
사용자 측(Client)에서는 그 서버의 주소(Command 혹은 URL)만 환경 변수에 등록하는 순간, 어떠한 벤더의 심사도, 승인 대기 시간도 없이 그 즉시 도구와 자원을 끌어다 쓸 수 있는 완전한 로컬(Local) 자율 배포망이 성취된다.

그뿐만 아니라, MCP라는 단일 표준 규격만 준수했다면 사용자는 오늘 OpenAI의 모델을 쓰다가, 내일은 Claude로 메인 브레인을 통째로 교체(Swapping)하더라도 기존에 연결해 둔 수백 개의 사내 도구(MCP 서버들) 코드를 단 한 줄도 수정할 필요가 없다. 벤더 종속형 스토어가 지녔던 독점과 검열의 족쇄를 끊어내고, 진정한 의미의 ‘Write Once, Run With Any LLM’ 시대를 개막한 사상적 중심에 MCP의 존재 이유가 자리 잡고 있는 것이다.
