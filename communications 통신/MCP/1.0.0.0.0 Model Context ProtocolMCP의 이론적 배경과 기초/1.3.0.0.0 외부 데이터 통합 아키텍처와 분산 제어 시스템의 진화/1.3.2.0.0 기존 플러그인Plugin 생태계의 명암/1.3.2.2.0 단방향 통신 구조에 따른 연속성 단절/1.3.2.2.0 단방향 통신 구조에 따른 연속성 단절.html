1.3.2.2 단방향 통신 구조에 따른 연속성 단절

기존 플러그인(Plugin) 생태계가 대규모 언어 모델(LLM)에게 도구(Tool)를 쥐여주는 데는 성공했지만, 본질적으로 그것은 1990년대 웹 초창기의 통신 패러다임에 갇혀 있었다. 플러그인의 근간을 이루는 HTTP REST(Representational State Transfer) 아키텍처는 철저히 클라이언트(LLM)가 요청(Request)하고 서버가 응답(Response)하는 단기적이고 일회성인 ’단방향(Unidirectional) 통신 구조’를 따른다. 이러한 구조적 편향성은 실시간으로 변동하는 외부 세계의 리소스를 LLM과 자연스럽게 동기화해야 하는 현대적 에이전트(Agent) 환경에서 치명적인 대화 맥락(Context)의 단절과 지능의 일시적 정지 사태를 불러왔다.

0.1 REST API의 무상태성(Statelessness)과 요청 주도권의 편향

전통적인 플러그인은 언어 모델이 오직 “나에게 정보를 달라“고 요청(Pull)할 때만 작동한다.
플러그인 매니페스토(Manifest)에 명세된 모든 함수들은 철저히 LLM 클라이언트 쪽에 ’주도권(Initiative)’을 부여한다. 사용자가 챗봇에게 “현재 AWS 서버 상태 어때?“라고 묻기 전까지, 플러그인은 결코 LLM에게 먼저 서버의 다운(Down) 상태나 경고 로그를 알려줄 수 없다.

이러한 무상태(Stateless) 기반의 응답/요청 사이클 모델은 단순한 일기예보 검색이나 주식 티커(Ticker) 조회 같은 가벼운 연산에는 적합할지 모른다. 하지만 엔터프라이즈 환경에서는 백그라운드에서 비동기적(Asynchronous)으로 실행되는 데이터 전처리, 데이터베이스의 잠금(Lock) 해제, 또는 CI/CD 파이프라인의 배포 완료 같은 수많은 이벤트가 수 분에서 수 시간에 걸쳐 발생한다. 단방향 통신망에서는 외부 API 서버가 “기존에 요청한 데이터 전처리가 지금 막 끝났다“라는 사실을 LLM 모델 측으로 밀어 넣을(Push) 수단이 원천적으로 존재하지 않는다.

0.2 폴링(Polling)의 비효율성과 대화 컨텍스트(Context)의 노이즈 폭발

단방향 통신의 한계를 메우기 위해 개발자들은 과거의 웹 어플리케이션이 취했던 우회로인 ‘폴링(Polling)’ 기법을 LLM 프롬프트 상에 강제로 이식하려 시도했다.

이는 에이전트에게 “작업이 끝났는지 10초마다 API를 다시 호출하여 확인해 보라“는 반복적 지시(System Prompt)를 내리는 방식이다. 그러나 언어 모델의 추론(Reasoning) 사이클에 폴링을 결합하는 것은 재앙에 가깝다.
에이전트가 Check_Job_Status() API를 연속으로 호출하면, 그 수많은 “아직 안 끝났음“이라는 응답 텍스트가 모델의 컨텍스트 윈도우(Context Window) 안에 차곡차곡 쌓이게 된다. 귀중한 토큰(Token) 예산이 무의미한 상태 확인 응답으로 심각하게 고갈될 뿐만 아니라, 어텐션(Attention) 기법을 사용하는 모델 특성상 무가치한 로깅 텍스트의 반복은 실제 해결해야 할 논리적 주제를 망각하게 만드는 심각한 집중력 분산(Lost-in-the-middle) 현상을 유발한다.

0.3 Model Context Protocol(MCP)의 양방향 완전 이중(Full-Duplex) 통신 전환

초등적인 단방향 호출-응답 구조의 단절을 완전히 봉합하고, LLM과 외부 인프라를 하나의 생명체처럼 유기적으로 결합하기 위해 Model Context Protocol(MCP)은 웹소켓(WebSockets) 및 서버 전송 이벤트(SSE, Server-Sent Events)에 기반한 ‘단일 양방향 파이프(Full-Duplex Pipeline)’ 사상을 프로토콜의 코어로 채택했다.

sequenceDiagram
    participant LLM as Cloud LLM / Client
    participant P_Server as 기존 Plugin Server (REST)
    participant MCP as MCP Server (SSE/WebSockets)

    Note over LLM, P_Server: [기존 플러그인: 단방향 Polling의 참극]
    LLM->>P_Server: 1. 데이터 분석 요청 (Pull)
    P_Server-->>LLM: 2. 202 Accepted (작업 시작)
    LLM->>P_Server: 3. 끝났는가? (Polling)
    P_Server-->>LLM: 4. 아직 안 됨
    LLM->>P_Server: 5. 끝났는가? (Polling - 토큰 낭비)
    P_Server-->>LLM: 6. 아직 안 됨
    
    Note over LLM, MCP: [MCP 아키텍처: 상태 구독(Subscription) 기반 양방향 푸시]
    LLM->>MCP: 1. 도구 호출 및 상태 변경 "구독(Subscribe)"
    MCP-->>LLM: 2. 요청 수신 즉시 Acknowledge 
    
    Note over MCP: (백그라운드에서 수 분간 데이터 처리)
    
    MCP->>LLM: 3. [Push Event] 서버 측 능동 푸시: <br> "ResourceUpdated" 또는 "Tool Execution Complete"
    
    Note over LLM: 폴링 없이 즉각 콜백 처럼 이벤트 수신 및 대화 재개

MCP의 연결망 하에서 클라이언트(LLM 호스트 어플리케이션)는 서버에게 단순한 일회성 API 패킷을 쏘고 연결을 끊는 자폐적 행위를 하지 않는다. 대신 연결 초기부터 서버와 영구적인 수명 주기를 갖는 세션(Session)을 맺고 커넥션을 열어둔다.

이를 통해 MCP 서버 측 로컬 데이터베이스의 값이 외부 요인에 의해 변동되거나 백그라운드에서 돌던 딥러닝 연산이 종료되는 그 즉시, 서버는 클라이언트(LLM)를 향해 능동적으로 알림 이벤트(Notification Event, 예: ResourceUpdated)를 쏘아 올릴(Push) 수 있다.
결과적으로 언어 모델은 대화의 맥락(Context)을 소진하는 멍청한 반복 질문(Polling)을 멈추고 사색에 잠겨 있다가, 외부 환경(Environment)이 어깨를 두드려주는 명확한 신호에 의해서만 즉각적으로 깨어나 사용자에게 “데이터 분석이 방금 모두 완료되었습니다“라고 보고하는 완벽한 동기화(Synchronization)와 지능의 연속성을 획득하게 되었다.
