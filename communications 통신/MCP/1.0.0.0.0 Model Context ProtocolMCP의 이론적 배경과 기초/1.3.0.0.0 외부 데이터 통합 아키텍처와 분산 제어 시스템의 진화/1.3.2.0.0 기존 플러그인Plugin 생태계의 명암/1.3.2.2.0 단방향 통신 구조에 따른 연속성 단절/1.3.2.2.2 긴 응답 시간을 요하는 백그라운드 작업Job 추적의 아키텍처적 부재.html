1.3.2.2.2 긴 응답 시간을 요하는 백그라운드 작업(Job) 추적의 아키텍처적 부재

대규모 언어 모델(LLM) 기반 에이전트가 처리해야 하는 엔터프라이즈급 워크로드는 종종 수 초 내에 끝나는 단순 정보 검색이 아니라, 수 분에서 때로는 수 시간이 걸리는 비동기적(Asynchronous) ’백그라운드 작업(Background Job)’을 수반한다. 예를 들어 대용량 머신러닝 모델의 파인튜닝(Fine-tuning) 지시, 테라바이트급 데이터 파이프라인의 배치(Batch) 처리 구동 등이 이에 해당한다. 그러나 기존 플러그인(Plugin) 체계의 근간인 REST API 단방향 HTTP 통신은 태생적으로 이렇게 긴 응답 시간을 요하는 비동기 작업을 추적하고 조율할 수 있는 아키텍처적 그릇을 전혀 갖추지 못했다.

0.1 HTTP 타임아웃(Timeout)의 벽과 블로킹(Blocking) 통신의 한계

기존 플러그인 환경에서 LLM이 “과거 10년간의 판매 데이터를 분석해 달라“며 외부 서버의 연산 도구를 호출했다고 가정해 보자. 이 요청을 받은 외부 서버는 즉각 데이터 크런칭(Crunching)에 돌입하지만, 연산에는 최소 30분이 소요된다.

이때 구조적인 파국이 발생한다. LLM 클라이언트와 플러그인 서버를 연결하는 HTTP 웹 커넥션은 무한정 열려있지 않다. 통상적으로 브라우저 네크워크나 클라우드 게이트웨이(Gateway)는 30초에서 60초 안에 응답 패킷(Response)이 돌아오지 않으면 강제로 HTTP 504 (Gateway Timeout) 에러를 발생시키고 연결 회선을 끊어버린다.
에이전트 입장에서는 백그라운드 서버가 열심히 일을 하고 있는지, 아니면 서버가 죽어버렸는지 도무지 알 길이 없는 ’침묵의 벽’에 가로막힌다. 더욱 절망적인 것은, 이 응답을 기다리는 수 초 내내 에이전트의 대화형 스레드는 완전히 블로킹(Blocking)되어 사용자의 추가적인 질문이나 다른 도구의 병렬적 사용을 전혀 처리하지 못하고 화면만 멈춰버리는 끔찍한 사용자 경험(UX)을 유발한다는 점이다.

0.2 가짜 비동기(Pseudo-Asynchronous) 폴링 설계의 복잡도

이 통신 레이어의 원초적 결함을 피해가기 위해, 개발자들은 플러그인을 ‘가짜 비동기(Pseudo-Asynchronous)’ 방식으로 억지로 비틀기 시작했다. 외부 서버는 1차 작업 요청을 받자마자 즉시 HTTP 202 (Accepted) 코드와 함께 {"job_id": "12345", "status": "pending"} 이라는 대기표(Ticket)만 LLM에게 반환하고 커넥션을 닫는다.

이후 에이전트는 프롬프트에 하드코딩된 규칙에 따라 Check_Job_Status(job_id=12345)라는 상태 확인용 API를 10초나 1분 간격으로 계속해서 재호출해야만 한다(Polling 방식).
이는 개발자와 모델 양측 모두에게 엄청난 논리적 부하(Logical Overhead)를 안겨준다. 플러그인 개발자는 실제 데이터 처리 로직 외에도 상태값을 관리할 Redis 같은 별도의 캐시를 구축하고 Job ID 발급 로직을 추가로 짜야 한다. LLM은 수십 번에 걸친 무의미한 상태 확인 반환값을 컨텍스트 윈도우(Context Window)에 허비하며 “작업이 아직 완료되지 않았습니다. 다시 확인하겠습니다“라는 강박적 반복 발화의 수렁에 빠진다. 이는 시스템 아키텍처의 빈틈을 프롬프트 엔지니어링과 억지 로직으로 땜질한 근본 없는 미봉책일 뿐이다.

0.3 Model Context Protocol(MCP)의 프로세스 간 통신 방식

이러한 비동기 작업 통제 불능 사태를 종식시키기 위해, Model Context Protocol(MCP)은 HTTP 연동이라는 허약한 단발성 고리를 버리고, 운영체제 단위의 영구적 연결망이나 클라이언트-서버 간 양방향 이벤트 채널(SSE, Server-Sent Events)에 기반한 통신망을 채택했다.

MCP 생태계 내부에서는, 에이전트가 2시간짜리 비동기 데이터 분석 도구를 호출할 경우, MCP 서버는 즉시 실행 커맨드만 밀어 넣고 즉각적인 ‘Acknowledge(인지)’ 신호를 클라이언트에 반환한다.
에이전트는 이 즉각적인 응답을 바탕으로 UI 스레드를 대기 상태(Blocking)로 두지 않고, 사용자와 다른 주제의 대화를 이어나가거나 다른 MCP 도구를 병행하여 자유자재로 다룰 수 있다.

마침내 백그라운드의 거대한 연산 Job이 종료되어 결괏값이 도출되는 순간, MCP 서버는 클라이언트를 향해 능동적인 알림 스트림(Push Event)을 쏘게 된다. 이 알림을 수신한 어플리케이션 단의 런타임은 잠들어 있던, 혹은 다른 일을 하던 LLM의 워크플로우에 해당 완료 신호와 결과 페이로드를 조용히 인터럽트(Interrupt) 형식으로 밀어 넣는다.
LLM은 길었던 대화 컨텍스트의 낭비(Polling) 없이 완벽한 순간에 깨어나, 2시간 전에 던졌던 작업의 종결을 인지하고 사용자에게 매끄럽게 결과를 브리핑할 수 있게 된다. 플러그인 시절 붕괴 직전이었던 장시간 비동기 작업(Long-running Async Job)의 아키텍처적 공백이, MCP의 양방향 완전 제어(Full-Control) 연결망을 통해 비로소 완숙한 엔터프라이즈의 품격으로 완성된 것이다.
