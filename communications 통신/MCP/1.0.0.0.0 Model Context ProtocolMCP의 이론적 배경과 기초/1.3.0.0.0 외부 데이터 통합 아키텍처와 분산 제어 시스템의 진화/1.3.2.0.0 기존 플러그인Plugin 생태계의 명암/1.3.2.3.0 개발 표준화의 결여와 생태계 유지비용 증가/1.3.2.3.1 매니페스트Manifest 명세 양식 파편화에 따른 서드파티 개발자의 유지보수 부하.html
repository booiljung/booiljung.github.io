1.3.2.3.1 매니페스트(Manifest) 명세 양식 파편화에 따른 서드파티 개발자의 유지보수 부하

플러그인(Plugin) 생태계가 최초로 약속했던 장밋빛 미래는 “API를 단 한 번만 정의하면, 세상의 모든 언어 모델이 그 글을 읽고 스스로 도구를 사용할 수 있다“라는 이른바 ’선언적 통합(Declarative Integration)’의 마법이었다. 하지만 이 마법의 주문서인 매니페스트(Manifest) 파일의 규격이 통일되지 못하고 각 벤더별로 산산조각 나는 순간, 서드파티 개발자들은 혁신의 마법사가 아니라 종이봉투를 접어 납품하는 영세 공장의 단순 노동자로 전락하고 말았다.

0.1 선언적 통합의 이상과 파편화된 문서의 충돌

매니페스트(Manifest)란, 외부 API 서비스가 무엇을 할 수 있는지 언어 모델(LLM)에게 서술하는 메타데이터(Metadata)와 JSON/YAML 양식의 스펙 문서 묶음을 의미한다.
엔지니어링 관점에서 개발자가 작성한 순수 비즈니스 로직(예: 날씨 데이터를 JSON으로 내려주는 Python 백엔드)은 완벽하게 동일하다. 문제는 이 로직을 덮고 있는 껍데기, 곧 매니페스트의 규격이었다.

OpenAI의 에코시스템에 진입하기 위해서는 반드시 OpenAPI 3.0 스펙을 기반으로 ai-plugin.json이라는 그들만의 포맷을 작성하고, 모델이 읽기 좋은 자연어 설명(Description for Model)을 특정 길이와 문법에 맞춰 정밀하게 깎아내야 했다.
그런데 이 똑같은 날씨 API를 Google이나 Anthropic 같은 타사 플랫폼에 연계하려 할 때 파국이 발생한다. 경쟁 플랫폼들은 OpenAI의 매니페스트를 호환해주지 않고, 자체적인 파라미터 구조와 Tool Calling JSON 규격(혹은 GraphQL 기반의 명세)을 내밀었다. 결국 단 하나의 날씨 조회 도구를 세상에 유통하기 위해, 서드파티 개발자는 본질적인 비즈니스 로직 수정 없이 오직 ‘A사 제출용 문서’, ‘B사 제출용 문서’, ’C사 양식’을 복사하고 붙여넣어 관리하는 극악무도한 문서화 늪(Documentation Swamps)에 빠져버린 것이다.

0.2 동기화 장벽과 눈덩이처럼 불어나는 기술 부채(Technical Debt)

매니페스트 양식의 파편화가 초래하는 진정한 재앙은 ’유지보수의 비동기화 현상’이다.

사내 백엔드 시스템에 테이블 스키마가 변경되어 날씨 API의 temperature 파라미터가 temp_celsius로 단 한 글자 바뀌었다고 가정하자. 기존 웹 개발 환경이었다면 프론트엔드 코드 한 곳만 수정하면 그만이었다. 하지만 파편화된 플러그인 생태계에서는 A사, B사, C사의 플랫폼 개발자 포털(Developer Portal)에 일일이 접속하여 서로 다른 구조의 JSON 매니페스트 파일을 수정하고, 심지어 벤더의 승인 검수(Review) 절차를 다시 처음부터 밟아야 하는 최악의 파이프라인이 열린다.

이 과정에서 인간 엔지니어의 실수는 필연적이다. C사의 매니페스트만 갱신하고 A사의 갱신을 까먹는 순간, A 플랫폼을 쓰는 언어 모델은 존재하지도 않는 과거의 temperature 파라미터를 계속 타격하다가 500 Internal Error를 뿜어내며 실신한다.
플랫폼 개수(N)가 늘어날수록 서드파티 개발진이 감당해야 하는 유지보수 공수와 잠재적 버그의 뇌관은 O(N) 수준을 넘어 기하급수적(Exponential)으로 폭증하게 되었고, 이 무거운 기술 부채(Technical Debt)는 자금력이 부족한 스타트업들이 플러그인 생태계에서 완전히 손을 떼게 만드는 거대한 절망벽으로 굳어졌다.

0.3 Model Context Protocol(MCP)의 스키마 동적 변환(Dynamic Schema Conversion)을 통한 공수 O(1) 달성

이 처참한 매니페스트 파일 관리의 지옥을 종식시키기 위해, Model Context Protocol(MCP)은 ’정적 파일 기반의 사후 검수’라는 낡은 패러다임을 폐기하고 ‘표준 프로토콜 기반의 동적 질의(Dynamic Querying)’ 사상을 도입했다.

MCP 생태계에는 개발자가 플랫폼별로 깎아 바쳐야 할 json이나 yaml 매니페스트 파일의 묶음이 존재하지 않는다.
MCP 구조 하에서, 도구를 소유한 서버(MCP Server)는 오직 표준화된 JSON-RPC 2.0으로 “네가 가진 도구의 카탈로그(Schema)를 읊어보라(tools/list)“는 요청을 수신할 뿐이다.
그러면 MCP 서버는 자신이 가진 단 1개의 표준 스펙을 클라이언트(MCP Client)를 향해 능동적으로 내뿜는다. 그 패킷을 수신한 클라이언트(Ex: Cursor, Claude Desktop 등)가 백그라운드에서 동작 중인 각기 다른 언어 모델(Claude, GPT-4 등)의 고유한 Function Calling 규격에 맞게 런타임 단계에서 즉석 번역(Just-In-Time Translation)을 수행하여 모델의 뇌리에 꽂아 넣는다.

이 아름다운 ‘어댑터 패턴(Adapter Pattern)’ 덕분에 서드파티 데이터 제공자들은 세상에 존재하는 수십 종의 LLM을 위해 수십 종류의 껍데기를 코딩할 필요가 완전히 사라졌다. 자신의 MCP 서버 코드 내부에 도구의 스펙을 단 한 번만(O(1)) 작성해 두면, 클라이언트 계층에서 프로토콜 사양이 동적으로 변환되어 모든 이기종 LLM 호스트와 자동으로 매핑되는 ’단일 진실의 공급원(Single Source of Truth)’이 역사상 처음으로 구축된 것이다.
