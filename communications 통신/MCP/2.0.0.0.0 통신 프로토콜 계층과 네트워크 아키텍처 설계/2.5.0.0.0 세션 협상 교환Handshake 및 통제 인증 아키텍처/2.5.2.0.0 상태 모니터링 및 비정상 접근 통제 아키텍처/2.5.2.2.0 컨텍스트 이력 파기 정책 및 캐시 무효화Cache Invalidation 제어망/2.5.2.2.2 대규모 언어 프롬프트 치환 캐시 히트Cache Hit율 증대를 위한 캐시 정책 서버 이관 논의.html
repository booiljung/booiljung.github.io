2.5.2.2.2. 대규모 언어 프롬프트 치환 캐시 히트(Cache Hit)율 증대를 위한 캐시 정책 서버 이관 논의

컨텍스트 창(Context Window)이 거대한(예: 200K 토큰 이상) 대규모 언어 모델을 활용하는 MCP 시나리오에서는, 에이전트가 “동일한 코드 10만 줄” 위에 “질문 한 줄“만 바꿔서 서버의 프롬프트 도구로 계속 던져댈 경우 엄청난 연산 및 통신 과부하가 걸린다.

오버헤드를 막기 위해, 단순한 클라이언트 사이드 메모이제이션(Memoization)을 넘어서 서버 측 프롬프트 렌더링 캐싱(Server-side Prompt Cache) 이관 전략이 도입된다.

  1. 해시 기반(Hash-based) 프롬프트 치환 캐시: 클라이언트가 prompts/get을 요청할 때, 서버가 전달받은 인자값(Arguments)의 전체 페이로드를 SHA-256 형태의 지문 값으로 해싱한다.
  2. 캐시 히트 조건(Cache Hit Criteria) 성립: 이전과 동일한 해시값이 들어오고 캐시 TTL(Time-To-Live) 내라면 방대한 치환 렌더링 프로세스를 생략하고 인메모리에 저장된 완성형 프롬프트 문자열을 0 밀리초만에 내뿜는다.
  3. 종속성 트리 무효화(Dependency Invalidations): 만약 프롬프트 렌더링에 참조된 원본 파일(Resource)이나 외부 도구 값이 파일 감시 시스템 감지에 의해 변경(list_changed)되었다면, 해당 프롬프트 해시와 묶여 있는 모든 트리형 데이터 캐시 노드들을 즉시 파괴(Invalidate)시켜 더럽혀진(Stale) 과거 데이터를 읽는 환각(Hallucination) 현상을 차단한다.
