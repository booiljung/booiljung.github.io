2.5.2.4.2. 재전송 공격(Replay Attack) 방어용 서명 검증을 통한 HTTP 헤더 인증 통제 아키텍처

로컬 프로세스(Stdio) 환경과 달리 원격 네트워크(SSE/HTTP) 레이어를 지나는 MCP 통신은 중간자(Man-in-the-Middle) 공격에 취약하다. 공격자가 에이전트와 서버 간에 오간 성공적인 인증 페이로드(예: 토큰이 발급된 HTTP 헤더)를 스니핑(Sniffing)하여 복사해 두고, 나중에 그대로 쏘아대는 **재전송 공격(Replay Attack)**은 가장 흔한 탈취 수법 중 하나다.

이를 분쇄하기 위해 단순한 세션 키 검증과 더불어 시변(Time-variant) 서명 아키텍처를 HTTP 헤더 단에 이중으로 적용한다.

  1. Timestamp와 논스(Nonce) 주입: 성공적인 매 요청마다 클라이언트는 현재의 밀리초(ms) 단위 X-MCP-Timestamp와 무작위 난수 문자열인 X-MCP-Nonce를 생성하여 HTTP 헤더에 담는다.
  2. 단방향 해시(HMAC) 검증: 서버는 수신된 타임스탬프와 논스, 그리고 요청 본문을 결합하여 사전에 교환한 시크릿 키(Secret Key)로 서버 측 HMAC 유효성(Validity)을 재계산한다. 이 값이 클라이언트가 보낸 X-MCP-Signature와 한 글자라도 틀리면 즉각 거부당한다.
  3. 만료 시간과 캐시 히스토리 통제: 유효한 서명이라 하더라도 타임스탬프가 현재 서버 시간보다 수 분을 초과했거나(Stale Request), Redis 캐시에 이미 사용된 적이 있다고 기록된 Nonce라면 이는 100% 탈취된 과거 데이터를 쏘는 재전송 공격이므로 인증 세션을 즉시 파괴한다.
