2.5.2.4.3. 역할 기반(RBAC) 권한 축소 규칙에 따라 오염된 세션을 격리시키는 샌드박스 폴백(Sandbox Fallback) 망

명백한 해킹 시도는 아니지만 클라이언트의 행동이 다소 의심스러울 때(예: 평소에 쓰지 않던 Github 삭제 도구를 세 번 연달아 호출) 즉각적으로 연결을 박살 내는 것은 사용자 경험(UX) 관점에서 위험할 수 있다. 오작동일 확률을 고려해야 하기 때문이다.

이때 서버는 세션을 유지하되 기능의 범위를 강력하게 축소시켜 범위를 가두는 샌드박스 폴백(Sandbox Fallback) 전략을 전개한다.

  1. 동적 역할 기반 통제(Dynamic RBAC): 에이전트는 평소 ‘Admin’ 권한을 가지고 로컬의 파일 시스템을 자유롭게 넘나들었지만, 의심스러운 징후(Anomaly) 스코어가 누적 임계치를 넘는 순간 그 즉시 ’Guest(읽기 전용)’으로 역할(Role)이 강등(Demotion)당한다.
  2. 격리 구간(Quarantine Zone) 이동: 강등된 세션이 쓰기 관련된 tools/call이나 resources 삭제를 요망하면 서버는 내부 파일망과 연결을 끊고, 순수한 메모리 내의 가짜 파일 시스템(Mock VFS)만을 보여주는 허니팟(Honeypot) 샌드박스로 라우팅을 우회시킨다.
  3. 오염 통제(Containment): 이후 클라이언트가 무슨 행위를 하던지 오직 그 격리 구역 안에서만 적용되며, 실 데이터베이스에는 단 하나의 바이트도 영향을 주지 못한 채 활동 패턴만 로깅 분석기로 전송되어 시스템 관리자의 사후 감사(Audit) 자료로 채택된다.
