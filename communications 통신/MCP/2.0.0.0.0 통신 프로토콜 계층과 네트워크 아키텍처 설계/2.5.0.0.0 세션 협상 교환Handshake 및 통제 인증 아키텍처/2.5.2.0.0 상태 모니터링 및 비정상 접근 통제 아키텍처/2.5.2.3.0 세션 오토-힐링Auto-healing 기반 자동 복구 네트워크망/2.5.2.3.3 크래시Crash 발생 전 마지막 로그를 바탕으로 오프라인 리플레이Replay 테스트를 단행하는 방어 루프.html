2.5.2.3.3. 크래시(Crash) 발생 전 마지막 로그를 바탕으로 오프라인 리플레이(Replay) 테스트를 단행하는 방어 루프

서버 측 엔진 내에서 발생한 버그(Logic Bug)로 인해 서버가 크래시되고 재부팅(Restart)된 상황이라면, 단순히 클라이언트가 재접속한다고 문제가 해결되지 않는다. 똑같은 파멸적 페이로드(Poison Pill)가 여전히 클라이언트의 재시도 큐프레이임(Retry Queue)에 맺혀 있어, 부활한 서버를 향해 발사될 즉시 서버를 다시 죽이는 **무한 크래시 루프(Crash Loop)**를 유발하기 때문이다.

  1. 포이즌 필(Poison Pill) 감별망: 서버는 예비 동작 과정이나 재부팅 직후, 직전에 크래시를 유발한 마지막 JSON-RPC 패킷의 흔적을 로컬 스토리지 또는 OS 로거에서 찾아낸다.
  2. 오프라인 샌드박스 리플레이(Replay): 이 악의적(혹은 버그가 담긴) 요청 구조를 실 환경 라우터에 통과시키기 전, 메모리가 철저히 분리된 오프라인 샌드박스(Offline Sandbox) 환경 내에서 재실행(Replay) 테스트를 단행한다.
  3. 영구 차단 및 블랙리스트 등재: 리플레이 결과 해당 페이로드가 지속적으로 서버 메모리를 찢거나 패닉을 유발하는 원인으로 지목되면, 해당 요청을 생성해 낸 클라이언트의 메타데이터 서명(2.5.1.2)이나 특정 파라미터 패턴을 블랙리스트 규칙 엔진에 주입한다. 이후 복구된 정상 세션에서 클라이언트가 동일한 요청을 다시 쏘면, 서버는 이를 실행 엔진에 도달시키기 전에 방어 라우터 단에서 즉시 절단(Drop) 처리한다.
