2.5.2.3.2. 상태 불일치(Desync)를 교정하기 위해 서버-클라이언트 간의 능력값을 재비교하는 체크포인트 구조

어렵사리 죽었던 세션이 오토 힐링에 성공하여 소켓이 부활했다 하더라도, 그 찰나의 시간축 사이에서 서버 측에 코드 배포(Deployment)가 일어나 도구의 종류(capabilities)가 변동되는, 소위 상태 불일치(Desync) 현상이 일어났을 가능성을 철저히 염두에 두어야 한다.

단순히 소켓 연결을 재활성(Resume)하는 것을 넘어, 부활 직후의 안전한 정합성(Integrity) 검증을 통과해야 한다.

  1. 소리 없는 마이크로 initialize 교환: 클라이언트가 세션을 복원하자마자, 백그라운드에서는 매우 가벼운 형태의 capabilities 해시(Hash) 지문만을 교환하여 재비교 라우팅 체계를 가동한다.
  2. 데싱크(Desync) 발견 시 강제 무효화: 서버가 기억하는 “죽기 전 클라이언트의 형태“와, 지금 막 “부활하면서 서버가 강제 배포된 결과” 간의 해시 충돌이 감지되면, 서버는 안전한 샌드박스 유지를 위해 기존 유예(Grace)되고 있던 캐시값들을 파괴(Invalidate)하고 State Desync Error를 내려보내 클라이언트가 정상적인 전체 initialize 핸드셰이크를 새롭게 밟도록 강제한다.
    이는 클라이언트가 “과거에 존재했으나 지금은 사라진” 유령 환경으로 도구 실행을 찔러넣으려다 발생하는 파국을 방어하는 초석이다.
