2.5.2.3.1. 프로세스 종료 이후 과거 상태를 완전히 복구시켜 클라이언트에게 제공하는 영속성 마이그레이션 전략

에이전트 클라이언트가 네트워크 지연이나 메모리 부족으로 크래시(Crash)되어 완전히 화면에서 소멸해 버리는 경우가 있다. 사용자가 화를 내며 1분 뒤 앱을 다시 켤 때, 이전에 실행 중이던 데이터베이스 스캔 도구(tools/call)의 결과가 날아가지 않고 그대로 전송(Delivery)되게 만드는 것이 영속성 마이그레이션 전략의 목표다.

  1. 상태 머신 체크포인팅(Checkpointing): 클라이언트는 initialize 시점부터 부여받은 강력한 암호화 기반의 Session-ID를 로컬 디스크나 보안 키체인(Keychain)에 가장 먼저 캐싱해둔다.
  2. 이어서 접속하기(Resume) 핸드셰이크: 부활한 클라이언트가 방금 구워낸 새로운 껍데기로 접속할 때, 초기화 페이로드에 기존의 Session-ID를 복원용 토큰(Resume Token)으로 끼워 넣는다.
  3. 가비지 컬렉션의 유예(Grace Period) 및 재결합: 서버 측의 세션 감시망(2.5.2.1)은 세션이 죽었다고 바로 파쇄기(2.5.2.2.3)에 넣지 않고 예비 유예 시간(예: 3분)을 둔다. 이 시간 내에 동일한 Session-ID가 진입하면, 서버는 찰나에 버퍼링해 두었던 도구 실행 결과값(Response)들과 중간에 놓친 notifications들을 재전송(Re-play) 큐에 밀어넣어 중단 없는 마이그레이션을 완성한다.
