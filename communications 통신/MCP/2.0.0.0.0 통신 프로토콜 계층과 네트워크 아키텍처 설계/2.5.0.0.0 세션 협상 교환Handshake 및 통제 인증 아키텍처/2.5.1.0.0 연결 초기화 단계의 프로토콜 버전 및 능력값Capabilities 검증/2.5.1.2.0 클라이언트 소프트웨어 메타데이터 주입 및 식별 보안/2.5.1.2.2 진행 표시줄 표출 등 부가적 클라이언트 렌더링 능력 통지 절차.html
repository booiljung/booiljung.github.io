2.5.1.2.2. 진행 표시줄 표출 등 부가적 클라이언트 렌더링 능력 통지 절차

MCP 통신의 묘미는 백엔드 서버가 클라이언트의 화면(UI)에 시각적인 변화를 지시할 수 있다는 점에 있다. 그러나 터미널 기반의 원시적인 CLI 에이전트에게 “거대한 원형 프로그레스 바(Circular Progress Indicator)를 그려라“라고 지시하는 것은 치명적인 렌더링 오류를 낳을 수 있다.

따라서 클라이언트는 initialize 핸드셰이크 단계에서 스스로의 제한적인 렌더링 능력을 서버에게 자수(Declaration)해야 한다.

  • capabilities.progress 선언: GUI 기반의 고도로 발전된 IDE 에이전트는 자신의 페이로드에 capabilities: { progress: {} } 객체를 삽입하여, “나는 서버가 보내는 장기 실행 작업(Long-running Process)의 % 단위 진행 상황 알림(notifications/progress)을 네이티브 UI로 예쁘게 그려낼 수 있다“고 통지한다.
  • 서버의 점진적 향상(Progressive Enhancement): 서버는 이 플래그를 확인한 후에만 무거운 AI 모델 추론 과정에서 progressToken을 발행하여 중간중간 퍼센테이지를 보고한다. 만약 클라이언트가 이 능력이 없다고 선언했다면, 서버는 불필요한 네트워크 트래픽(진행 알림 패킷)을 발송하지 않고 완전히 연산이 끝난 뒤에 단발성 결과물만 내려받게 하는 조용한 모드(Silent Mode)로 동적 전환된다.
