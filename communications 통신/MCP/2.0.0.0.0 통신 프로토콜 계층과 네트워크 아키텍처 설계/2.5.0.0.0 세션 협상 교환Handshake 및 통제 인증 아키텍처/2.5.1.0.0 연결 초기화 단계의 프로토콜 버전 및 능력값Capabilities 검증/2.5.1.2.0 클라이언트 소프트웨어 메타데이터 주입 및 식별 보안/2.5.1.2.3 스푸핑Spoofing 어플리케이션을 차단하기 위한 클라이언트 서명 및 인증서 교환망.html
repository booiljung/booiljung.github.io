2.5.1.2.3. 스푸핑(Spoofing) 어플리케이션을 차단하기 위한 클라이언트 서명 및 인증서 교환망

클라이언트가 스스로를 Claude-Desktop v1.0이라고 자칭(clientInfo.name)한다고 해서 서버가 이를 100% 맹신하는 것은 최악의 보안 취약점이다. 해커가 파이썬 스크립트를 짜서 이름 필드만 위조하여(Spoofing) 보내면 그만이기 때문이다.

이러한 신원 위장 공격을 분쇄하기 위해 엔터프라이즈급 MCP 백엔드는 단순 문자열 검증을 넘어, 핸드셰이크 과정에 고도화된 암호학적 서명(Cryptographic Signature) 교환망을 덧씌운다.

  1. 상호 TLS(mTLS) 인증 기반 접근 통제: 전송 계층(Transport Layer) 단계에서부터 클라이언트는 사전에 프로비저닝된 클라이언트 인증서(x.509)를 첨부하여야 한다. 서버는 이 인증서가 기업의 내부 CA(Certificate Authority)에서 적법하게 서명되었는지를 1차적으로 뜯어본다.
  2. 페이로드 자체 서명(JWS) 검증: 네트워크가 아닌 애플리케이션 계층에서는, 클라이언트가 initialize JSON 페이로드 본문 전체를 자신의 비공개 키(Private Key)로 서명하여 HTTP 헤더(예: X-MCP-Signature)나 페이로드 외벽에 첨부한다.
  3. 거부의 결정적 근거 확보: 서버는 공개 키(Public Key) 디렉토리에서 해당 에이전트 벤더의 키를 가져와 서명을 대조한다. 서명이 조작되었거나 매칭되지 않는다면, 즉각 신분 도용(Spoofing) 봇단으로 간주하고 AuthToken 발급을 영구 거절하며 칩입자의 IP를 관제망에 블랙리스트로 등재한다. 이 철벽같은 검문 메커니즘이 MCP 프로토콜의 신뢰 기반(Root of Trust)을 완성한다.
