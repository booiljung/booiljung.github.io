2.5.1.4.3. 파일 감시자(File Watcher) 변동 콜백과 핸드셰이크(Handshake) 파이프라인의 논리적 결합도

서버가 스스로의 상태가 변했음을 인식하고 list_changed 이벤트를 발사하기 위해서는, 외부의 환경적 변화를 감지하는 ’파일 감시망’과 MCP 프로토콜의 통신 파이프라인(Handshake/Notification) 간의 끈끈한 기계적 결합이 선행되어야 한다.

0.1 이벤트 버스(Event Bus)를 통과하는 결합 아키텍처

  1. 감시망(File Watcher) 전진 배치: 운영체제의 고성능 파일 감시 API(Linux의 inotify, macOS의 fsevents)가 MCP 백엔드의 디렉터리를 감시한다. 특히 capabilities를 정의한 설정 파일이나 도구 스크립트 폴더에 변경(MODIFY, CREATE, DELETE) 서명이 감지되면 코어 이벤트 버스에 트리거를 태운다.
  2. 디바운스(Debounce) 완충기: 개발자가 스크립트를 작성하며 Ctrl+S를 수십 번 연타하는 환경에서 서버가 수십 번의 list_changed를 발사해 클라이언트를 마비시키지 않도록, 이벤트 버스에는 300ms 디바운스와 같은 완충 지대가 필수적으로 자리 잡아야 한다.
  3. 파이프라인 주입(Injection): 정제된 단 한 번의 “상태 변동 이벤트“는 현재 활성화된 세션 관리자의 핸드셰이크 파이프라인에 꽂혀, 조건부로 동결(Freeze) 메커니즘을 발동시키고 최종적으로 알림(Notification) 패킷을 생성해 내보낸다.
    결국 단순한 로컬 파일의 바이트 변경이 OS 커널망을 타고 올라와 실시간 RPC 프로토콜망으로 번역되는, 놀랍도록 강력한 유기적 결합도가 완성된다.
