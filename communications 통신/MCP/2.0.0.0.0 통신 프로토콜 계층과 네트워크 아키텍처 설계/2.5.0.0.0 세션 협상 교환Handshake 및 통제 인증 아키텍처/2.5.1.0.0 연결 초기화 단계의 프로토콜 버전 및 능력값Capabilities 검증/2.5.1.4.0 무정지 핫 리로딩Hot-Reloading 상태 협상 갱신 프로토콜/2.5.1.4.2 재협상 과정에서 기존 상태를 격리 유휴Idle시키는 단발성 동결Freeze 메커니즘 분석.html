2.5.1.4.2. 재협상 과정에서 기존 상태를 격리 유휴(Idle)시키는 단발성 동결(Freeze) 메커니즘 분석

능력값의 거대한 변화(새로운 권한 규정, 주요 모듈의 폐기 등)를 수반하는 극단적인 재협상(Re-negotiation) 이벤트가 발생했을 경우, 알림이 전파되고 클라이언트가 새 리스트를 가져오기까지 발생하는 미세한 시간 차(Race Condition) 동안 거대한 장애가 터질 수 있다.
클라이언트는 아직 삭제된 도구를 호출하려 하고, 서버는 이미 그 도구를 메모리에서 지워버린 상태 불일치(Incoherent State) 구간이다.

0.1 임시 동결(Freeze)과 상태 버퍼링

이를 막기 위해 라우터 계층에는 핫 리로딩이 진행되는 수백 밀리초 동안만 가동되는 단발성 동결(Freeze) 메커니즘이 삽입된다.

  1. 인바운드 큐 락(Inbound Queue Lock): 서버 측의 코어가 리로딩 되는 순간, 라우터는 클라이언트로부터 날아오는 도구 실행 요청(tools/call)을 즉시 거절하지 않고, 일시적으로 인메모리 버퍼 큐(Buffer Queue)에 묶어둔다(Lock).
  2. 논리적 유휴(Idle) 상태 진입: 해당 순간 클라이언트는 요청이 지연되는 것을 경험하지만 에러를 만나지는 않는다.
  3. 상태 결빙 해제(Thaw): 리로딩이 완료되고 새로운 명세가 파싱 완료되면 버퍼링된 요청들은 새 정책 기반 위에 재평가(Re-evaluated)된다. 만약 그 도구가 여전히 존재한다면 실행되어 결과를 반환하고, 폐기되었다면 그제서야 명확한 MethodNotFound 에러를 반환하여 우아한 상태 천이를 마무리한다.
