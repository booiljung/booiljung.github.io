2.5.1.3.2. 실험적 사양(Experimental Features) 활성화를 강제하는 옵트인(Opt-in) 환경설정 협상 스텝

MCP 명세는 급격히 발전 중이므로, 아직 정식 인증을 거치지 않은 실험적(Experimental) 기능들(예: 비공개 AI 모델 연동, 초안 단계의 실시간 오디오 스트리밍 프로토콜 등)이 지속적으로 추가된다. 이러한 불안정한 기능들을 구형 에이전트 환경에 함부로 밀어 넣으면 시스템 붕괴가 일어난다.

이러한 사양의 확장을 위해 MCP 핸드셰이크는 엄격한 옵트인(Opt-in) 방식의 실험체 협상망을 가동한다.

  1. experimental 전용 구역의 활용: 양측은 능력값(capabilities) 객체 내부의 정식 명세 구역이 아닌, 오직 experimental 필드 구역 안에 커스텀 플래그(예: capabilities.experimental.voiceStream: true)들을 구겨 넣는다.
  2. 쌍방 합의(Mutual Agree)의 원칙: 서버가 해당 실험적 기능을 가지고 있더라도, 클라이언트의 experimental 목록에 해당 플래그가 정확한 이름으로 기재되어 있지 않다면 서버는 그 능력을 절대 사용하지 않고 잠가 둔다. 오직 양쪽 교환 페이로드에 공통적으로 합집합 매칭이 일어났을 때만 기능의 봉인이 해제된다.
  3. 비공식 벤더 확장(Vendor Extensions): 이 옵트인 협상 스텝은 서로 다른 클라이언트-서버 벤더 간에 공식 스키마에 없는 독자적 기능(예: experimental.myCompanyCustomSync)을 남몰래 주고받는 유연한 도어(Backdoor) 역할도 동시에 수행한다.
