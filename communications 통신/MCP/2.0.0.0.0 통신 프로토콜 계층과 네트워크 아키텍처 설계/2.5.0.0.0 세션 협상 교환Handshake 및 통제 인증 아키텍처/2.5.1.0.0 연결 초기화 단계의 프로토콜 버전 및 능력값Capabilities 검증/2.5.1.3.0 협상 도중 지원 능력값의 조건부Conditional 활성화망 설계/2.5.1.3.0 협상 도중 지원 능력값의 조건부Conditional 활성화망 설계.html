2.5.1.3. 협상 도중 지원 능력값의 조건부(Conditional) 활성화망 설계

클라이언트와 백엔드 서버가 무조건적으로 모든 기능을 ’On’으로 두고 통신하는 것은 자원 낭비일 뿐만 아니라 심각한 보안 문제를 야기할 수 있다. 예를 들어, IDE 에이전트가 단지 날씨 정보만 묻고 싶은데, 서버가 무단으로 하드디스크의 루트(Root) 접근 권한 망을 활성화해 버리는 것은 과도한 프로비저닝(Over-provisioning)이다.

MCP의 핸드셰이크는 철저하게 조건부 활성화(Conditional Activation) 철학을 따른다. 클라이언트가 특정 능력(Capabilities)을 명시적으로 요구(Opt-in)할 때만, 서버는 해당 기능의 스위치를 켜고 권한을 승인한다. 반대로 서버의 자원 여력이 부족할 경우, 클라이언트의 요구 중 일부 기능(예: 무거운 자원 구독)만을 선별적으로 거절하는 부분 활성화(Partial Activation) 전략 또한 핸드셰이크에 내포되어 있다. 본 절에서는 이 정교한 조건부 타협망을 분석한다.
