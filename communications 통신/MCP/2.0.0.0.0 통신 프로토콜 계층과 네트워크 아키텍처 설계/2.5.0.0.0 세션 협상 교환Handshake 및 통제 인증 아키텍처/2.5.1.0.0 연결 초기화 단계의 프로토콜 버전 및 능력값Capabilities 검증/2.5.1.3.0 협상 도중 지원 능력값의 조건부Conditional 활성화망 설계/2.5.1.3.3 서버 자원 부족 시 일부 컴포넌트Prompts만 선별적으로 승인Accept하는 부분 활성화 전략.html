2.5.1.3.3. 서버 자원 부족 시 일부 컴포넌트(Prompts)만 선별적으로 승인(Accept)하는 부분 활성화 전략

수천 대의 클라이언트가 동시에 달라붙는 엣지(Edge) 서버나 대규모 클러스터링 환경의 백엔드에서는, 연결을 요청하는 모든 에이전트의 거대한 자원 구독(Subscription) 요구를 전부 들어줄 시스템적 여유가 없을 때가 존재한다.

이러한 한계 상황 속에서 서버는 아예 접속을 차단(Reject)해 버리는 거칠고 단순한 폴리시 대신, 연결의 생명선은 유지하되 요구사항의 일부만 잘라내어 승인하는 부분 활성화(Partial Activation) 전략을 취한다.

  1. 협상 페이로드의 가지치기(Pruning): 클라이언트가 initialize 요건으로 복잡한 자원 구독능력(resources.subscribe)과 도구 실행능력을 무겁게 요구하며 진입한다. 서버의 부하율이 90%를 초과한 상태라면, 서버의 핸드셰이크 응답 페이로드에는 tools: {}와 prompts: {} 능력만 반환되고 resources 객체는 의도적으로 누락된다.
  2. 클라이언트 측 기능 열화(Feature Degradation) 대비: 이 부분 승인 응답을 받은 클라이언트는 에러를 뿜는 것이 아니라, UI의 ‘실시간 자원 동기화’ 버튼을 비활성화(Dimmed) 처리하는 식으로 우아하게 소프트웨어의 기능을 열화판(Degraded) 모드로 렌더링한다.
  3. 핵심 기능 우선주의망: 서버는 도달 가능한 대역폭이나 메모리가 부족할 때, 데이터 전송량이 무거운 자원의 폴링 기능부터 차단하고 에이전트의 사고 증폭을 위한 프롬프트나 가벼운 단발성 도구 연산(Tools)에 자원을 집중적으로 몰아주는 우선순위(Priority) 스케줄링을 핸드셰이크 단계에서 미리 재단(Tailoring)한다.
