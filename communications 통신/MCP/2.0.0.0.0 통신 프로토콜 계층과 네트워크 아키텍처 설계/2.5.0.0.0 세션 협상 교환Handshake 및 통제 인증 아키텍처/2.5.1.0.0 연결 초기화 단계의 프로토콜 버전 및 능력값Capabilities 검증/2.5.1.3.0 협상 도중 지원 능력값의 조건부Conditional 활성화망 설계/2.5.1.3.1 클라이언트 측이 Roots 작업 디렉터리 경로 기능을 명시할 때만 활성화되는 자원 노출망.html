2.5.1.3.1. 클라이언트 측이 Roots (작업 디렉터리 경로) 기능을 명시할 때만 활성화되는 자원 노출망

클라이언트 IDE(예: Cursor, VSCode)가 코드베이스 전체의 맥락을 파악하고 파일을 조작하기 위해서는, 에이전트가 어떤 폴더들(Workspaces) 위에서 활동 가능한지에 대한 경계선 정보인 Roots 명세가 제공되어야 한다.

하지만 이 Roots 정보의 노출 및 조작 권한은 철저히 클라이언트 유도형(Client-driven) 조건부 활성화망에 의해 통제된다.

  1. 에이전트의 명시적 capabilities.roots 선언: 서버는 절대로 먼저 에이전트의 로컬 폴더 시스템에 대한 권한을 요구하거나 스캔 목록을 달라고 하지 않는다. 클라이언트 에이전트 측에서 핸드셰이크 페이로드에 capabilities: { roots: { listChanged: true } } 와 같이 노출 의지를 선언했을 때만 루트 제어망이 가동된다.
  2. 샌드박싱(Sandboxing) 기반의 자원 방벽: 이 기능이 합의되면, 서버 측의 도구(Tools)나 프롬프트(Prompts) 엔진은 클라이언트가 명시한 Roots 디렉터리 외부 경로(예: ../../../etc/passwd 같은 디렉터리 트래버설 공격)로 접근하는 것을 원천 차단하는 가상 환경 규제를 활성화한다.
  3. 루트 미선언 시의 백색소음(Zero-trust) 처리: 에이전트가 Roots 능력을 생략했다면, 서버의 파일 시스템 접근 도구 모듈은 아예 비활성화(Disable)되거나, “파일 접근 권한이 없음“이라는 에러만을 도출하도록 자원 노출망 자체가 완전히 닫히게 된다.
