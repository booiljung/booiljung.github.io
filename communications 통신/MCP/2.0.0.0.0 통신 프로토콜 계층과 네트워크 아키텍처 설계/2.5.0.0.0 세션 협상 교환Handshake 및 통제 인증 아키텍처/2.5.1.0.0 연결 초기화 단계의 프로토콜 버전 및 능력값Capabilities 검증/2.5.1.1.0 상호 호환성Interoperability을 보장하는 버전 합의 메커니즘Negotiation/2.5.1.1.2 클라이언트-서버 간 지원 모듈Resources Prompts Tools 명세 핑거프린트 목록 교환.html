2.5.1.1.2. 클라이언트-서버 간 지원 모듈(Resources, Prompts, Tools) 명세 핑거프린트 목록 교환

단순한 전역 프로토콜 버전을 합의하는 것만으로는 부족하다. MCP는 자원(Resource), 도구(Tool), 프롬프트(Prompt)라는 3대 모듈이 결합된 집합체이므로, 양측은 핸드셰이크 단계에서 어떤 모듈을 사용할 수 있는지에 대한 세밀한 능력값(Capabilities) 객체를 서로 맞교환한다.

0.1 Capabilities 핑거프린트 교차 선언

  1. 클라이언트의 렌더링 능력(Client Capabilities) 선언: 클라이언트는 initialize 요청 내의 capabilities 객체를 통해, 자신이 리소스를 구독(Subscribe)할 능력이 있는지, 혹은 서버에서 보내는 진행 표시줄(Progress bar) 이벤트 등을 화면에 렌더링할 UI 능력 유무를 핑거프린트처럼 남겨 서버에 보고한다.
  2. 서버의 가용 엔진 명시: 요청을 받은 서버도 InitializeResult 응답의 capabilities 필드 속에 답을 보낸다. 서버가 도구(Tools)의 형태만 제공하고 프롬프트(Prompts) 엔진은 꺼져 있다면, 명확히 tools: {}만을 표기하고 프롬프트는 제외한다.

이 명세 핑거프린트(Fingerprint)의 교환이 완료되면, 양측은 상대방의 “지원 불가” 기능을 억지로 호출하여 헛된 런타임 오류(RPC Method Not Found)가 터져 나오는 현상을 선제적으로 완벽히 타파하게 된다.
