2.3.2.3. 프로세스 격리형 리소스 쿼터(Quota) 제어 한계 지점

Stdio를 통해 구동되는 자식 서버 프로세스는 동일한 물리 호스트의 자원을 공유한다. 만약 오픈소스로 받아온 MCP 데이터 분석 툴이 무한 루프 버그에 빠져 CPU를 100% 점유하거나, 악의적인 메모리 리크(Leak)를 일으켜 32GB의 RAM을 포화 상태로 만들어 버린다면 어떻게 될까? 이 경우 부모 에이전트는 물론이거니와 사용자 데스크톱 OS 전체가 셧다운(OOM 킬러 발동)되는 처참한 결과로 이어진다.

로컬 기반 IPC 환경의 치명적인 **한계 지점(Pain Point)**은 바로 자식 프로세스가 빨아들이는 하드웨어 리소스 쿼터(Quota)를 프로토콜 자체의 언어로 강제할 방법이 전무하다는 점이다. HTTP API나 클라우드 FaaS(Function as a Service) 람다는 인프라 단에서 Timeout과 512MB RAM 등의 한계를 명확히 씌울 수 있지만, spawn() 된 생 프로세스는 거대한 호스트의 자원을 게걸스럽게 탐식하는 데 아무런 논리적 브레이크가 없다.

0.1 Cgroups 및 OS 수준 쿼터 적용의 당위성

이 한계 지점을 극복하기 위해, 최전선 MCP 런타임 플랫폼들은 단순한 자식 프로세스 기동(fork)을 넘어서 커널 수준의 리소스 제한(Resource Limiting) 조끼를 강제로 채운다.
Linux 환경에서는 자식 서버 트리를 리눅스 제어 그룹(Cgroups v2) 영역으로 묶어, 해당 트리가 가질 수 있는 최대 RAM 용량(예: memory.max=2GB)을 고정해 버린다.
Windows 계열에서는 앞서 언급한 워커 **작업 객체(Job Object)**에 JOBOBJECT_CPU_RATE_CONTROL_INFORMATION 할당 속성을 부여하여 스케줄러가 데스크톱의 메인 CPU 사이클을 15% 이상 뺏기지 않도록 강력한 브로커(Broker) 역할을 수행하게 만든다.

결론적으로, Stdio 채널 프로토콜의 격리 수준은 로컬 OS가 지원하는 외부의 샌드박스 역량과 필연적으로 의존관계를 맺어야만 호스트 공멸의 한계선을 피할 수 있다.
