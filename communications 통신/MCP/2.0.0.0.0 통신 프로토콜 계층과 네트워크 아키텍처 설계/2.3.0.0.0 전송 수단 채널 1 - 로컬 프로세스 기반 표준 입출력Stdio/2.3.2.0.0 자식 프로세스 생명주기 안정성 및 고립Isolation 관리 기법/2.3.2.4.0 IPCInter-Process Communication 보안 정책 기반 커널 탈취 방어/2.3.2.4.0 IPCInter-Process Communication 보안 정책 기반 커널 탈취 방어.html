2.3.2.4. IPC(Inter-Process Communication) 보안 정책 기반 커널 탈취 방어

개발자의 워크스테이션에는 SSH 프라이빗 키(Private Key), 지갑 비밀번호, 클라이언트 소스코드 등 상상할 수 있는 모든 고가치 지적 재산이 널려 있다. MCP 스펙 하에서, 클라이언트 에이전트가 서드파티 서버 툴을 Stdio 파이프라인으로 연결하기 위해 자식 프로세스를 파생(Spawn)하는 행위는 결국 나의 가장 내밀한 안방 열쇠를 낯선 프로세스의 손에 쥐어주는 무서운 백도어(Backdoor) 개방과 진배없다.

이 자식 프로세스가 프로토콜 트래픽을 처리하는 척하다가 몰래 ~/.ssh/id_rsa 파일을 읽어 외부 네트워크 소켓으로 유출하는 등, 호스트 IPC 권한을 휘둘러 커널 데이터를 탈취하는 스파이 시나리오를 막기 위해 엄격한 기계적 방어망이 필수적으로 요구된다.

0.1 능동적 권한 소거(Privilege Dropping) 정책

이 공격 벡터를 진압하기 위해 클라이언트 로컬 라우터는 자식 프로세스를 생성하면서 부모(개발자 권한)가 지닌 능력을 무지성으로 상속(Inheritance)해 주는 대신, 적극적인 권한 소거(Privilege Dropping) 정책을 전개한다.

  • 네트워크 차단 (Network Blackholing): 리눅스 네임스페이스(Network Namespaces)를 조작하여 자식 서버 프로세스가 외부망 인터넷으로 IP 패킷을 쏘는 것을 물리적으로 차단한다. 서버 프로세스의 세상은 오직 Stdio 파이프와 127.0.0.1 만으로 제한된다.
  • 파일시스템 마운트 격리 (chroot/jail): 서버가 구동될 때 chroot 환경을 만들어, 허용된 타겟 프로젝트 디렉터리(/workspace/task_A) 외에 호스트의 어떤 민감한 파일 시스템 경로(/etc, ~/.ssh)로도 절대 접근할 수 없는 가상 감옥(Jail)을 구축한다.

이러한 IPC 보안 샌드박싱은 아무리 파괴적인 프롬프트 인젝션(Prompt Injection)을 받아 타락해 버린 MCP 서버 도구라 한들, 에이전트의 한정된 모래 상자 안에서 발버둥 치게 만들 뿐, 호스트의 커널이나 사용자의 근원적인 권한을 영원히 탈취할 수 없게 만드는 최종의 보안 방패막이다.
