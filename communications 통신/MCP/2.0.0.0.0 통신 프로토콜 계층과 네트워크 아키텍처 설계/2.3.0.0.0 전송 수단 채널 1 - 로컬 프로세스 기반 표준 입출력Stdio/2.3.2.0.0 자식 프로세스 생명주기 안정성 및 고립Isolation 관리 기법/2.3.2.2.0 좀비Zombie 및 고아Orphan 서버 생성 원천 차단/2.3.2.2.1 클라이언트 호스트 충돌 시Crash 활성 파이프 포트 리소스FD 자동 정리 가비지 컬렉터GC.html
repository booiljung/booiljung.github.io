2.3.2.2.1. 클라이언트 호스트 충돌 시(Crash) 활성 파이프 포트 리소스(FD) 자동 정리 가비지 컬렉터(GC)

좀비 서버가 창궐하는 것을 가만히 지켜볼 수 없는 OS 커널은 프로세스가 갑작스럽게 죽었을 때 나름대로 자신이 관리하던 자원을 강제로 쓸어 담는 최후의 파수꾼인 **가비지 컬렉터(OS Kernel GC)**를 가동한다.

MCP 통신의 절대적 근간은 부모 클라이언트의 Stdout과 자식 서버의 Stdin을 이어주는 유-닉스 커널 메모리 기반 파일 디스크립터(File Descriptor, FD) 링 버퍼이다.
클라이언트 메인 에이전트 프로세스가 허용 불가능한 에러(Crash)를 일으켜 즉사(Die)하는 찰나, 프로세스의 유저 랜드 런타임은 시그널을 발송할 기회조차 얻지 못하지만 해당 런타임을 감싸고 있는 OS 커널 그 자체는 자신이 해당 런타임에 빌려주었던 유효한 **활성 파이프 포트 핸들(Active Pipe Handles)**들을 인지하고 있다.

0.1 FD 강제 회수(Revocation)와 버퍼 단절

부모 프로세스가 소멸함과 동시에, 리눅스의 커널 스케줄러(또는 Windows의 커널 엑스큐티브)는 죽은 클라이언트가 꽉 쥐고 있던 이 IPC 파일 파이프 양단 중 자신의 쪽 파이프 핸들을 전부 close() 시켜 강제로 회수해 버린다(FD Revocation).

이는 파이프라는 가상의 ‘대화 파이프관’ 한 쪽 끝을 물리적으로 잘라내 버리는 행위와 같다. 물리적 네트워크(HTTP 소켓)는 TCP 재전송 타임아웃 등으로 인해 끊김을 인지하는 데 수초에서 수십 초가 소요될 수 있지만, 로컬 커널 메모리에 종속된 Stdio 파이프는 포트가 날아가는 밀리초(ms) 단위의 그 찰나에 버퍼가 터지는 즉각적인 충격을 동반한다.
이렇게 파이프가 뜯겨 나가는 현상 자체가 부모 없는 서버들에게 자신들의 운명을 알려주는 가장 빠르고 원시적인 ’죽음의 전보’가 되며, 이는 후술할 서버들의 자가 종료 로직(Self-Termination)을 촉발하는 방아쇠로 작용한다.
