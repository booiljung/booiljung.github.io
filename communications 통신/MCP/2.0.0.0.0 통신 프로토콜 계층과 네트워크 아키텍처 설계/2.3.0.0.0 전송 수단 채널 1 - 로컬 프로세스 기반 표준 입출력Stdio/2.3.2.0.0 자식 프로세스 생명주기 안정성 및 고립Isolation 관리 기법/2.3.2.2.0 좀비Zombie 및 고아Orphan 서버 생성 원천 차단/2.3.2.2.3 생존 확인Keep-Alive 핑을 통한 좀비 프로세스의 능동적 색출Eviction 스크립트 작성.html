2.3.2.2.3. 생존 확인(Keep-Alive) 핑을 통한 좀비 프로세스의 능동적 색출(Eviction) 스크립트 작성

파이프 닫힘 이벤트를 통한 자가 종료(Self-Termination)는 매우 강력하지만, 가끔 OS 커널 계층의 버그나 다층 가상화 환경(Docker in Docker, WSL2) 같은 기형적인 라우팅 상태에서는 파이프 소멸 시그널(EOF)이 자식 노드까지 전파되지 않고 먹혀버리는(Swallowed) 코너 케이스(Corner Case)가 존재한다.
이러한 한계 상황에서 고아 프로세스가 된 서버를 솎아내기 위해, 고급 MCP 엔터프라이즈 환경은 **양방향 생존 확인 핑(Keep-Alive Ping/Pong)**이라는 능동적인 최후의 색출망(Eviction Net)을 보조재로 가동한다.

0.1 세션 계층의 핑-퐁 감시 타이머(Watchdog)

이 방법은 전송 계층(Transport) 하단의 파이프 단절에 의지하지 않고, 프로토콜 애플리케이션 계층(Layer 7) 자체에서 상대방의 맥박을 감시하는 논리적 스크립트 체계이다.

  1. 클라이언트의 주기적 Ping: 에이전트는 정상적인 상황에서 매 30초마다 MCP 통신망에 아무 의미 없는 백그라운드 핑(ping) JSON 메시지를 부유시킨다.
  2. 서버의 독자적 워치독(Watchdog): 서버 프로세스 내부에는 독자적인 타이머 스레드가 돌고 있다. 이 워치독은 만약 “최근 120초 동안 메인 클라이언트로부터 그 어떠한 핑 메시지나 JSON-RPC 함수 호출 명령도 수신하지 못했다면,” 부모 프로세스가 정상적인 경로로 파이프를 닫지 못한 채 데드락(Deadlock)에 걸려 죽어버렸다고 확신(Suspect)한다.

워치독 타이머가 타임아웃 뇌관을 건드리면, 시스템 트래픽의 마비(Hung)나 커널 시그널 증발과 무관하게, 서버 프로세스는 스스로 메모리를 해제하고 데이터베이스 락(Lock)을 푼 뒤 자결(Kill oneself)한다. 스튜디오 파이프라인(Stdio)이라는 수동적인 망에서 애플리케이션 계층 능동 색출 스크립트가 이처럼 이중으로 결합될 때 좀비 서버의 생존 확률은 문자 그대로 0%에 수렴한다.
