2.3.2.2.2. 표준 입력 스트림(stdin) 닫힘 감지를 통한 서버 프로세스의 즉각적 자가 자살(Self-Termination)

클라이언트 애플리케이션의 비정상 종료(Crash) 시 커널에 의해 Stdio 파이프가 물리적으로 뜯겨 나간 뒤, 이제 공은 백그라운드에 홀로 남겨진 MCP 자식 서버의 코트로 넘어온다.
결론부터 말하자면, MCP 스펙을 준수하는 모든 로컬 서버 애플리케이션은 **“부모 클라이언트 프로세스로부터 뻗어 나온 표준 입력(Stdin) 스트림이 닫힌(Closed) 것을 감지하는 즉시, 어떠한 유예도 없이 자가 자살(Self-Termination) 절차에 돌입해야 한다”**는 엄격한 무명(Implicit)의 설계 의존성을 지닌다.

0.1 EOF(End Of File) 이벤트 구독 메커니즘

서버 런타임은 구동과 동시에 입력 파이프에 대한 읽기(Read) 스트림을 논블로킹(Non-blocking) 형태로 열어둔다.
만약 부모 클라이언트 프로세스가 폭파되면서 커널이 파이프를 회수하면, 자식 서버의 Stdin 읽기 스트림에는 물리적 데이터 끝을 알리는 강력한 EOF(End Of File) 시그널이 강제로 투입된다.

  • Node.js 생태계: process.stdin.on('close', () => { process.exit(0); }) 와 같은 단 한 줄의 이벤트 리스너 설정만으로 완벽한 자가 폭파 스크립트가 성립된다.
  • Python 생태계: sys.stdin.read() 루프가 갑자기 빈 문자열("")을 반환하거나 파이프 파손(BrokenPipeError) 예외를 던질 때, 즉시 sys.exit(0)을 호출하여 실행 트리를 부숴버린다.

이 기법은 클라이언트가 자식을 죽일 방법을 고민하는 것이 아니라, 자식이 부모의 맥박(Stdin)을 짚고 있다가 부모가 심정지를 일으키면 스스로 순장(殉葬)되는 극도의 **의존적 생명주기 결합 패턴(Dependent Lifecycle Coupling)**이다. 이를 통해 좀비 프로세스 문제를 완벽에 가깝게 근절할 수 있다.
