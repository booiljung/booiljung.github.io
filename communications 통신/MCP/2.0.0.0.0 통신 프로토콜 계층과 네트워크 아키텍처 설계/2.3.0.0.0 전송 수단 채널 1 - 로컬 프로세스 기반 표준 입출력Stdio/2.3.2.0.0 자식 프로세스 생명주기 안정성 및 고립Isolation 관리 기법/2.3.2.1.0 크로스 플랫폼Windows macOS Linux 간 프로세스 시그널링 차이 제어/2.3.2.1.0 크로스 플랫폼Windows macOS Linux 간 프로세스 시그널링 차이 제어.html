2.3.2.1. 크로스 플랫폼(Windows, macOS, Linux) 간 프로세스 시그널링 차이 제어

MCP 클라이언트가 자식 프로세스인 서버를 안전하게 다루기 위한 제1 관문은 각기 다른 진화 트리를 거쳐온 이기종 데스크톱 운영체제들의 시그널링(Signaling) 언어를 통일하는 것이다.
클라이언트는 에이전트 종료 시점에 하위 MCP 서버 프로세스의 ID(PID)를 향해 커널 중단 명령을 하달해야 하지만, 그 구현의 층위는 OS별로 잔인할 만큼 다르다.

  • Linux/macOS (POSIX) 생태계: SIGINT(키보드 중단), SIGTERM(소프트 종료 권고), SIGKILL(묻지마 강제 사살) 등 풍부한 인터럽트 구조체를 갖추고 있다.
  • Windows 생태계: POSIX 시그널의 개념이 희박하며, 대신 GenerateConsoleCtrlEvent를 통한 가짜 Ctr+C 전달망이나, TerminateProcess Windows API 계층을 거치는 상이한 강제 종료 트리를 밟아야 한다.

만약 클라이언트 런타임이 이 차이를 우아하게 추상화(Abstraction)하지 못하고 플랫폼별 하드코딩된 시스템 종속 루틴을 노출한다면, 개발자가 작성한 MCP 구동기 컴포넌트는 오직 특정 OS 편식증에 걸리게 되며 이는 곧 범용성을 잃은 고철이 된다.
이러한 파편화를 제어하기 위해 Stdio 전송 계층은 클라이언트 래퍼(Wrapper) 계층에 크로스 플랫폼 시그널 브릿지(Cross-platform Signal Bridge) 패턴을 의무적으로 탑재하여, 어떠한 OS 커널을 마주하더라도 일관성 있는 생명주기 중단 이벤트 파이프라인을 관철시킨다.
