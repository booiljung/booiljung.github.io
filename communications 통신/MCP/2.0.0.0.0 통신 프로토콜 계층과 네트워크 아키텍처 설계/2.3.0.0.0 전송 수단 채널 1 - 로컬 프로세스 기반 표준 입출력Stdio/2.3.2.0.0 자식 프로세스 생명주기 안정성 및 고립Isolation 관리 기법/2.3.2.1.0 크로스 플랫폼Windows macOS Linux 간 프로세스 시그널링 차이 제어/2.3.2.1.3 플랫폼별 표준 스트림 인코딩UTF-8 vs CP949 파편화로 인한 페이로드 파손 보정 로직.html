2.3.2.1.3. 플랫폼별 표준 스트림 인코딩(UTF-8 vs CP949) 파편화로 인한 페이로드 파손 보정 로직

Stdio 파이프는 그 바탕이 바이트 배열(Byte Array) 기반의 스트림일 뿐이다. 이 바이트의 무리들을 의미 있는 JSON 문자열 텍스트로 전환하려면 일치된 디코딩(Decoding) 문자셋이 절대적으로 요구되는데, 바로 이 지점에서 크로스 플랫폼의 저주가 쏟아진다.

글로벌 표준을 지향하는 macOS나 최신 Linux의 터미널 커널은 자식 프로세스를 파생시킬 때 파이프 입출력의 인코딩을 암묵적으로 UTF-8로 맵핑한다.
그러나 한글 Windows 환경(특히 구형 CMD 셸이나 특정 PowerShell 계층)은 아직도 레거시 지역 인코딩인 **CP949(EUC-KR)**를 시스템의 Stdout 기본 로케일 문자셋으로 질기게 물고 늘어지는 참사가 발생한다.

파이썬 기반의 서버 트리가 코드 컨텍스트에서 한글로 된 주석이나 파일명(“분석보고서.txt”)을 긁어모아 JSON 페이로드로 쐈는데, 클라이언트 Node.js 기반 수신기가 इसे 무작정 UTF-8 스트림으로 잘라 파싱하려 든다면 파이프라인 상의 문자열은 즉각 이가 빠진 **물음표(Replacement Character, ``)**로 찢겨 나가며(Mojibake), 프로토콜 JSON-RPC 구조체 전체가 무결성 위반으로 심각하게 파손(Corruption)된다.

0.1 강제 UTF-8 모드 주입과 버퍼 보정(Buffer Compensation)

로컬 Stdio 환경에서 일어나는 인코딩 파편성 결함을 보정하기 위해, MCP 런타임 클라이언트는 자식 프로세스를 띄울 때 호스트 OS의 기본 동작을 일절 신뢰하지 않는 강제 인코딩 주입(Forced Encoding Injection) 원칙을 따른다.

가령 Python 서버를 띄운다면 환경 변수(Environment Variables)에 PYTHONIOENCODING=utf-8을 쐐기 박듯 세팅하여 파이썬 인터프리터가 Windows 셸의 권고를 묵살하고 파이프에 오직 UTF-8 바이트만 배출하도록 통제한다.
만약 상대측 런타임을 임의로 조작할 수 없는 제3자 모듈 바이너리를 구동할 경우, 클라이언트 수신단(Rx)은 날 것의 텍스트가 아닌 원시 바이너리 청크 버퍼(Buffer Raw Chunk) 상태로 파이프에서 데이터를 건져 올린 뒤, BOM(Byte Order Mark)과 헤더를 스스로 탐지하여 프로토콜의 스펙인 순수 UTF-8 규격으로 역디코딩(Inverse Decoding)하는 디코딩 보정 레이어를 반드시 탑재해야만 한글이나 다국어 토큰이 난도질당하는 사태를 면할 수 있다.
