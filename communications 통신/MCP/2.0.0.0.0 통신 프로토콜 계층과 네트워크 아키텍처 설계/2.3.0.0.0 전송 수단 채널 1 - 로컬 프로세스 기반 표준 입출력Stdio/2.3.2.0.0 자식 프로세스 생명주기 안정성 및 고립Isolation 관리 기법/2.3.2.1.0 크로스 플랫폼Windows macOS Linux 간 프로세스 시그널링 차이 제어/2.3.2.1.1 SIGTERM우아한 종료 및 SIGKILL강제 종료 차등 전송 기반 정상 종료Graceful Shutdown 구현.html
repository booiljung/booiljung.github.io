에이전트 클라이언트가 사용자의 요청 혹은 세션 만료에 의해 MCP 서버를 종료해야 할 때, 단순히 운영체제에 사살(Kill) 명령을 내리는 것은 최하책이다. 만약 서버 프로세스가 파일 시스템에 분석 결과를 기록(Write)하고 있거나, 무거운 벡터 스토어(Vector Store)의 인덱스를 갱신하는 찰나에 하드 킬(Hard Kill)을 당한다면 해당 파일은 영구적인 데이터 오염(Corruption) 상태에 빠지게 된다.

0.1 단계 프로세스 종료 파이프라인(Two-Stage Termination Pipeline)

데이터 무결성을 보존하기 위해 MCP Stdio 래퍼 구현체들은 권장된 **2단계 차등 파이프라인(Two-Stage Pipeline)**을 적용하여 정상 종료(Graceful Shutdown)를 유도한다.

  1. 소프트 종료(Soft Termination) - SIGTERM 전송: 클라이언트는 먼저 자식 프로세스에 SIGTERM 시그널(Windows의 경우 CTRL_C_EVENT)을 송출한다. 이는 “너를 곧 종료할 테니 하던 작업을 정리하고 자원(메모리 락, 파일 핸들)을 반환하라“는 정중한 권고이다. 잘 작성된 MCP 서버는 내부의 이벤트 루프에서 이 시그널을 갈취(Intercept)하여, 진행 중인 I/O를 비우고 스스로 exit(0)을 호출하며 평화롭게 소멸한다.
  2. 타임아웃 감시와 하드 종료(Hard Termination) - SIGKILL 전송: 클라이언트는 첫 번째 시그널을 보낸 후 즉시 끝내지 않고 유예 시간(Grace Period, 보통 3~5초)을 두고 자식의 PID 상태를 감시(Polling)한다. 이 유예 기간이 지나도록 서버가 응답 불능(Hang) 상태로 버티며 죽지 않는다면, 클라이언트는 그제야 최후의 수단인 무조건적 사살 시그널(SIGKILL 또는 Windows의 TerminateProcess)을 투척하여 커널이 강제로 메모리를 회수하도록 다그친다. 이 우아한 이중 통제망이 데이터베이스 손상을 막고 시스템을 일관된 상태로 유지하는 핵심 열쇠이다.
