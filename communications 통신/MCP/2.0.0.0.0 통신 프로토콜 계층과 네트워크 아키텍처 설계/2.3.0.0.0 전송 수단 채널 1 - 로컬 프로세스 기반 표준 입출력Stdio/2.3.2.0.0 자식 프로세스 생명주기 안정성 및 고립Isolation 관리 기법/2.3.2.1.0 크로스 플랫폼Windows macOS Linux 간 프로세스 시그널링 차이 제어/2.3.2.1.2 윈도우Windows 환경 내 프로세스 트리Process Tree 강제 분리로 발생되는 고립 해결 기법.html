2.3.2.1.2. 윈도우(Windows) 환경 내 프로세스 트리(Process Tree) 강제 분리로 발생되는 고립 해결 기법

POSIX 규격을 따르는 Unix(Linux/macOS) 계열은 부모 프로세스가 죽을 경우 특정 조건(예: SIGHUP 전파 등) 하에 자식 프로세스 트리를 일괄 정리하는 커널 매커니즘이 비교적 명확하다. 그러나 Windows 운영체제의 프로세스 관리 모델은 이와 판이하게 달라서 악명 높은 ’고아 프로세스(Orphan Process) 대량 발생’이라는 늪을 만든다.

Windows에서 Node.js/Python 기반의 클라이언트가 spawn() 된 패키지 매니저(예: npm.cmd 또는 uvx.exe)를 통해 타겟 서버를 띄울 경우, 패키지 래퍼(Wrapper) 프로세스가 메인 서버 자식(Grandchild)을 파생시킨 후 자기 자신만 쏙 빠져나가 버리는 프로세스 트리 분리(Process Tree Detachment) 현상이 빈번하게 터진다.
이렇게 되면 클라이언트가 자신이 낳은 첫 번째 자식 PID를 타겟으로 taskkill을 날려도 껍데기만 죽을 뿐, 실제 램(RAM)을 점유하고 있는 서브 에이전트는 부모와 연결 고리가 끊긴 채 백그라운드에서 영원히 살아 숨 쉬며 고립(Isolated)된다.

0.1 Job Object를 통한 그룹 생명주기 완전 통제

이 처참한 크로스 플랫폼 누수 결함을 잡기 위해, Windows 기반의 MCP 클라이언트 코어는 커널 특수 객체인 **작업 객체(Job Object)**를 사용하여 자식들의 생살여탈권을 독점한다.

클라이언트는 spawn으로 새로운 프로세스를 구조하기 직전에 빈 Windows Job Object를 생성하고 이를 JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE 속성으로 태깅(Tagging)한다. 이후 방금 띄운 최상위 자식 프로세스를 이 Job 경계판 안에 강제로 밀어 넣는다.
이 속성의 마법은 자식(Child)이 낳은 손자(Grandchild)와 증손자 프로세스까지 모조리 이 단일 Job Object 컨테이너 안에 자동으로 귀속시킨다는 점이다. 클라이언트 앱이 스스로 종료되거나 충돌하여 Job 핸들(Handle)이 OS에 의해 닫히는 그 즉시, 커널은 이 캡슐 안에 든 모든 프로세스 트리를 연쇄 폭파(Massacre)시켜 단 한 마리의 고아도 남기지 않고 Windows 백그라운드 생명주기를 완벽하게 정화(Purge)해 낸다.
