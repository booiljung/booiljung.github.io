2.3.1.3.1. MCP 메시지 스트림(Stdout)과 시스템 디버깅 스트림(Stderr)의 완전 파티셔닝 구조

MCP의 Stdio 환경에서 프로세스의 입출력 소자는 완전히 목적이 다른 두 개의 평행 우주로 쪼개져야(Partitioned) 한다.

  1. Stdout (통신 파이프라인): 이 채널은 오직 MCP(또는 JSON-RPC 2.0) 규격을 완벽하게 준수하는 직렬화된 JSON 문자열 바이트(Bytes)만이 한 치의 오차도 없이 통과할 수 있는 청정 구역(Clean Zone)이다.
  2. Stderr (관측 파이프라인): 시스템 레벨의 에지 케이스, 경고, 상태 변화 등을 인간이 읽을 수 있는 평문(Plaintext) 혹은 구조화된 로그(Structured Log)로 자유롭게 배출하는 오수관(Sewage Pipe)의 역할을 담당한다.

0.1 프레임워크 레벨의 스트림 재지정(Redirection)

이 두 채널이 혼선(Crossover)되는 사고를 막기 위해, 올바르게 작성된 MCP 서버 스택(예: Python의 mcp 라이브러리, Node.js의 @modelcontextprotocol/sdk)은 서버가 런타임에 진입하는 맨 첫 번째 마이크로초(Milisecond)에 언어 자체의 기본 시스템 출력(예: sys.stdout, console.log) 객체를 해킹하여 가로챈다.

프레임워크는 내부적인 Monkey-patching 기법을 동원하여 개발자가 무심코 호출한 범용 print() 함수의 트래픽을 모조리 강제로 **Stderr 파일 디스크립터(FD 2)로 재지정(Redirection)**시켜 버린다.
그리고 오로지 프레임워크 내부의 프라이빗(Private) 통신 송출 함수만이 날 것의 Stdout (FD 1) 디스크립터를 점유하도록 단단히 방화벽을 친다. 이 물리적 파티셔닝 체계는 초보 백엔드 엔지니어의 단순한 실수(개발 중 남겨둔 콘솔 로그)로 인해 전체 에이전트 인프라가 무너지는 대참사를 프레임워크단에서 원천 거세(Castration)하는 신의 한 수이다.
