2.3.1.3. 표준 에러(Stderr) 채널 분리를 통한 독립적 로깅망

Stdio 전송망의 아킬레스건은 바로 단일 파이프(Stdout)에 모든 프로토콜 데이터의 운명을 걸어야 한다는 점에 있다. 서버 애플리케이션은 필연적으로 수많은 디버깅 로그, 경고, 크래시 덤프(Crash Dump)를 외부로 표출해야 하는 속성을 지닌다. 하지만 만끽할 수 있는 터미널 창도 없는 백그라운드 프로세스 환경에서 이 로그 덩어리들이 엉뚱하게 Stdout 채널로 흘러들어간다면, 클라이언트의 JSON 파서는 이를 불법 패킷으로 간주하여 그 즉시 연결을 참수(Kill)하게 될 것이다.

이 치명적인 취약점을 보완하기 위해 MCP 로컬 전송망은 OS가 제공하는 3번째 파일 디스크립터, 즉 표준 에러(Standard Error, Stderr) 채널을 구조적으로 완벽히 격리된 **독립적 아웃바운드 로깅망(Out-of-band Logging Network)**으로 격상시켰다.

본 절에서는 Stdout과 Stderr 간의 엄격한 파티셔닝(Partitioning) 정책을 통해 프로토콜 페이로드의 무결성을 어떻게 방어하는지, 그리고 백그라운드에서 돌고 있는 서버의 맹분(Blind) 상태를 해소하기 위해 클라이언트가 이 Stderr 채널을 가로채어(Intercept) 어떻게 개발자 친화적인 실시간 모니터링망으로 포워딩(Forwarding)해 내는지 그 기저 아키텍처를 면밀히 조명한다.
