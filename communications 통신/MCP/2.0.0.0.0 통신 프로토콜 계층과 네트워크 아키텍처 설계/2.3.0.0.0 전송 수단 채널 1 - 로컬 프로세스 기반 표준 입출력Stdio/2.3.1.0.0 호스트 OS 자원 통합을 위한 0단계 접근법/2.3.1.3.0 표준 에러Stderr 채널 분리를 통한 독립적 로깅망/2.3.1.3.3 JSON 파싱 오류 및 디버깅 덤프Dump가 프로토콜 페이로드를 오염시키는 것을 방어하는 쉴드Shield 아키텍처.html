2.3.1.3.3. JSON 파싱 오류 및 디버깅 덤프(Dump)가 프로토콜 페이로드를 오염시키는 것을 방어하는 쉴드(Shield) 아키텍처

최악의 시나리오를 가정해 보자. MCP 프레임워크 엔지니어가 Stdout과 Stderr의 파티셔닝(Partitioning) 경계를 제대로 치지 못했다. 혹은 서드파티 C 라이브러리가 프레임워크의 통제를 뚫고 하드코딩된 C언어의 printf 명령을 써서 파이프라인의 핵심인 Stdout 채널로 세그멘테이션 폴트(Segmentation Fault) 덤프 텍스트를 무자비하게 뿜어버렸다.

{ "jsonrpc": "2.0", "resulSegmentation fault (core dumped)t": "..." }

위와 같이 합법적인 JSON 페이로드 중간에 시스템 텍스트가 융단 폭격을 맞듯 끼어드는 **오염(Contamination)**이 발생하면, 클라이언트 측 JSON 파서는 불량 객체(Invalid Token)를 뱉고 즉시 크래시(Crash)할 위기에 처한다.

0.1 정규식 가드레일(Guardrail)과 클라이언트 측 선제적 정화(Sanitization)

단순한 JSON 파싱 에러(예: -32700) 자체로 클라이언트 전체가 다운되는 연쇄 참사를 막아내기 위해, MCP 클라이언트 수신단(Receiver)은 버퍼를 JSON.parse에 넘기기 직전 한 단계의 강력한 **정화 쉴드(Sanitization Shield)**를 가동한다.

클라이언트의 디스패처는 스트림에서 들어오는 원시 텍스트 버퍼 단위에서, 최상위 스코프의 중괄호 { 와 } 사이의 토큰 정합성을 검증하는 그리디(Greedy) 구조체 사전 스캐너를 돌린다. 만약 완전히 망가진 형태의 문자열 조각이나 덤프 찌꺼기 텍스트 단위가 파이프에서 발견될 경우, 클라이언트 쉴드는 이를 JSON 파서로 넘기지 않는다. 대신 이를 유해한 노이즈로 처리하여 프로토콜 트랜잭션에서 무시(Drop)하거나, 임의로 분리하여 호스트 측 디버깅 로그 패널로 은밀히 격리 전송한다.
이 방어선 덕분에, 하위 자식 프로세스가 아무리 발작하며 오염물을 Stdout에 뿌려대도 클라이언트 에이전트 시스템 자체는 강제 종료의 사신으로부터 결코 파괴당하지 않는 불사(Undead)의 신뢰성을 얻게 된다.
