2.3.1.1.1. 파일 시스템 마운트 제한 등 샌드박싱(Sandboxing)을 결합한 격리 실행 환경 설계 지침

로컬 OS의 통합 권한을 지닌 MCP 서버 자식 프로세스를 띄울 때, 클라이언트가 자신의 권한을 아무런 제약 없이 전부 상속(Inheritance)해 주는 것은 잠재적으로 거대한 백도어(Backdoor)를 개방하는 행위이다. 만약 오픈소스로 다운로드된 타사(Third-party) MCP 도구가 악의적인 목적을 띠고 있거나 프롬프트 인젝션(Prompt Injection)에 속아 넘어간다면, 이 도구는 rm -rf / 와 같은 치명적인 OS 내부 조작을 서슴지 않을 것이다.

0.1 컨테이너(Container) 환경과 마운트 패핑(Mount Mapping)

이러한 내부자 위협을 격리하기 위해, 상용 에이전트 클라이언트들은 서버를 단순 OS 프로세스로 띄우는 것이 아니라, 도커(Docker) 혹은 WebAssembly(Wasm) 기반의 샌드박싱(Sandboxing) 컨테이너 내부에서 자식 프로세스를 구동하는 형태를 취한다.

이때 클라이언트는 표준 입출력(Stdio) 파이프를 컨테이너 내부로 바인딩(Binding)하면서, 서버가 접근할 수 있는 호스트의 파일 시스템 마운트(Mount) 볼륨을 철저히 제한(Jail)한다.
예를 들어, “이 MCP 서버는 오직 /home/user/project_a 디렉터리 내부에만 읽기(Read-only) 접근이 가능하다“라는 지침을 적용하여 서명된 환경 변수로 런타임에 주입한다. 이로써 외부 생태계의 복잡한 도구들을 끌어안으면서도, 커널망의 피해를 원천 봉쇄하는 높은 신뢰 트러스트 존(Trusted Zone)을 로컬 환경에 성공적으로 구현할 수 있게 된다.
