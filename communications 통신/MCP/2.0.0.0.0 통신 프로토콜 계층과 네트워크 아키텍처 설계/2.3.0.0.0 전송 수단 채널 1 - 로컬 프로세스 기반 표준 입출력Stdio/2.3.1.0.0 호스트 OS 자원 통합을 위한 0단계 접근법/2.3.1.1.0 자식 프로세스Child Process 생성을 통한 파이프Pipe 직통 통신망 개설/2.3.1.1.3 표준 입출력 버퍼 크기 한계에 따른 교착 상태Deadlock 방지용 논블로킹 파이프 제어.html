2.3.1.1.3. 표준 입출력 버퍼 크기 한계에 따른 교착 상태(Deadlock) 방지용 논블로킹 파이프 제어

로컬 OS에서 호스트와 자식 프로세스 간에 뚫려 있는 Stdio 파이프라인은 무한한 우주 공간이 아니다. 이 파이프라인은 본질적으로 커널(Kernel) 메모리 공간에 할당된 아주 작은 고정 크기의 링 버퍼(Ring Buffer, 보통 Linux 기준 64KB 수준)에 불과하다.

서버가 1GB짜리 로그 데이터를 스트리밍하면서 데이터를 파이프에 맹렬히 집어넣고(Write), 반대로 클라이언트는 다른 무거운 렌더링 작업을 하느라 파이프에서 데이터를 빼내어 읽어가는(Read) 속도가 느려졌다고 가정해 보자.
파이프 버퍼가 제한치(64KB)에 도달하여 꽉(Full) 차게 되면, OS 커널은 기본적으로 블로킹(Blocking) I/O 정책에 따라 서버 측 스레드의 쓰기 연산을 그 즉시 중단(Suspend)시키고 대기 상태로 만든다.

하지만 만약 이때, 잠들어버린 서버를 향해 클라이언트가 “작업 취소(Cancel)“를 지시하는 패킷을 Stdin 채널로 보냈으나, 서버 스레드가 Stdout 채널의 빈 공간을 기다리며 OS 락(Lock)에 결박된 상태라면 어떻게 될까?
서버는 클라이언트의 명령을 읽지(Read) 못하고, 클라이언트 역시 서버가 응답할 때까지 대기하는 양방향 무한 대기 상태, 즉 **치명적인 교착 상태(Deadlock)**에 돌입하게 된다.

0.1 비동기 논블로킹(Non-blocking) I/O와 이벤트 루프

이러한 IPC 교착 상태를 회피하기 위해 MCP 클라이언트와 서버의 통신 구현체는 Stdio 파일 디스크립터(예: FD 0, 1, 2)를 반드시 **상태 기반 논블로킹 모드(Non-blocking Mode, O_NONBLOCK)**로 강제 전환하여 사용한다.

이 모드에서 쓰기(Write) 연산이 꽉 찬 버퍼를 찌를 경우, 커널은 스레드를 잠재우는 대신 EAGAIN 에러(다시 시도하라)를 반환하며 제어권을 돌려준다. 서버 프로세스는 제어권을 뺏기지 않으므로, Node.js의 epoll 기반 이벤트 루프나 Rust의 Tokio 비동기 런타임으로 돌아가 클라이언트가 Stdin을 통해 보낸 취소 명령 같은 통제 신호를 정상적으로 읽어내고 최우선으로 처리할 수 있게 된다.
이 논블로킹 파이프 제어 시스템이야말로 대용량 스트리밍 시대에 에이전트와 서버가 공멸하는 데드락(Deadlock)을 방어하는 가장 깊은 기저의 방어선이다.
