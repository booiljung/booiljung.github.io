2.3.1.1.2. VSCode 등 개발자 도구(CLI) 연동을 위한 무중단 백엔드 데몬(Daemon) 상주 실행 원리

개발자가 로컬 IDE(예: VSCode)에서 제공하는 코파일럿(Copilot) 확장 프로그램을 켤 때마다, 에이전트 클라이언트는 백엔드의 MCP 서버 프로세스를 매번 새로 띄워야만 할까?
만약 서버가 딥러닝 모델 가중치(Weight)를 메모리에 얹어야 하거나 대규모 코드베이스의 인덱스 벡터DB를 웜업(Warm-up)해야 하는 무거운 서버라면, 프로세스를 띄울 때마다 수십 초의 지연 시간(Cold Start)이 소모될 것이다. 이는 즉각적인 코드 완성을 원하는 개발자의 런타임 경험을 치명적으로 저하시킨다.

0.1 데몬(Daemon) 상주 패턴과 프로세스 어태칭(Attaching)

이런 문제를 해소하기 위해 로컬 MCP 환경은 자식 프로세스를 백엔드 데몬(Backend Daemon) 형태로 OS 메모리에 무중단 상주(Residency)시키는 패턴을 지원한다.

클라이언트 앱은 실행 시, 자신이 요구하는 타겟 MCP 서버 데몬이 이미 백그라운드에서 돌고 있는지를 OS의 프로세스 리스트(Process List)나 기명 파이프(Named Pipe), Unix 도메인 소켓을 통해 탐지한다.
만약 데몬이 이미 돌고 있다면 클라이언트는 비싼 fork() 연산을 수행하여 새로운 인스턴스를 창조하는 대신, 이미 열려 있는 기존 프로세스의 Stdio 스트림이나 시스템 파이프에 자신의 입출력을 찰칵 소리가 나도록 조립망에 **밀착 연결(Attaching)**하기만 하면 된다.

VSCode의 창을 닫더라도 해당 데몬은 OS 백그라운드에서 살아 숨 쉬며 파일 시스템의 변화 이벤트를 인덱싱하는 임무를 계속 수행(Watchdog)할 수 있으며, 추후 사용자가 다시 IDE를 켜는 즉시 웜 스타트(Warm Start) 상태로 초고속 0초대 응답을 재개할 수 있는 것이다.
