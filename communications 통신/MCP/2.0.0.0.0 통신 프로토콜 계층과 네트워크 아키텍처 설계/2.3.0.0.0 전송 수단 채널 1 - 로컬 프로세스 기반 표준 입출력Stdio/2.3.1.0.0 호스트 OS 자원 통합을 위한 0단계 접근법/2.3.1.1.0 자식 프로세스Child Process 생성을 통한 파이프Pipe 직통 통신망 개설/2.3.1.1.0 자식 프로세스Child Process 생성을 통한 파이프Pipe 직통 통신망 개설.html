2.3.1.1. 자식 프로세스(Child Process) 생성을 통한 파이프(Pipe) 직통 통신망 개설

MCP의 Stdio 환경에서 통신의 시작은 클라이언트가 서버 프로세스를 직접 잉태하는 것에서부터 출발한다. 클라이언트(메인 프로세스)는 OS의 커널 함수(예: Linux의 fork() 및 exec(), Windows의 CreateProcess())를 직접 호출하여 MCP 서버 실행 파일(바이너리 또는 스크립트)을 백그라운드의 **자식 프로세스(Child Process)**로 스폰(Spawn)한다.

이때 클라이언트와 서버 사이에는 네트워크 소켓이 아닌, 운영체제 메모리 공간에 맵핑된 무형의 단방향 파이프(Unidirectional Pipe) 쌍이 연결된다.

  1. 클라이언트의 Stdin \rightarrow 서버의 Stdout: 클라이언트가 서버에게 지시를 내리는 파이프.
  2. 클라이언트의 Stdout \leftarrow 서버의 Stdin: 서버가 클라이언트에게 응답을 반환하는 파이프.

서버 측 백엔드 코드는 어떠한 경우에도 print(로그) 나 console.log(디버깅) 와 같이 사람이 읽기 위한 임의의 평문 텍스트를 Stdout 채널로 내보내서는 안 된다는 점이다.
서버의 Stdout 채널은 오직 약속된 JSON-RPC 2.0 포맷의 페이로드만이 지나가는 **“데이터 전용 정밀 레일”**로 간주되며, 이 원칙이 깨져 단 한 줄의 잡음(Garbage Text)이라도 섞여 들어가는 순간, 클라이언트의 JSON 파서(Parser)는 즉각 에러를 내뿜으며 파이프라인 전체를 셧다운시켜 버린다.
