2.2.1. 상태 보전 파이프라인(Stateful Pipeline)과 RPC 메시징 구조

일반적인 HTTP REST API 통신망에서는 모든 요청이 완벽하게 독립적이며(Stateless), 한 번의 요청과 응답이 끝나면 서버는 클라이언트를 미련 없이 잊어버린다. 그러나 MCP가 지향하는 거대 언어 모델(LLM)과 분산 노드 간의 통신은 이러한 일회성 규격에 갇히는 순간 그 효용성을 급격히 상실한다. 클라이언트는 파일 디렉토리 트리를 열어두고 자원의 업데이트 스트림을 무한히 감시(Watch)해야 하며, 다단계 추론(Multi-step Reasoning)의 맥락을 길게 이어 나가야 하기 때문이다.

이에 대응하기 위해 MCP의 전송 계층은 JSON-RPC 2.0 패킷 포맷을 차용하면서도, 그 아래에 깔리는 네트워크 파이프라인을 철저한 상태 보전(Stateful) 기반의 지속적 연결(Persistent Connection) 위주로 이끌어간다.

0.1 상태 전이와 생명주기의 바인딩

이 파이프라인 내부에서는 메시지의 일회성 전송만으로 일이 종결되지 않는다. 클라이언트가 자원에 대한 ‘구독(Subscribe)’ 메시지를 발송하는 순간, 그 접속 소켓(Socket) 내부에는 클라이언트의 식별값과 이벤트 수신 권한을 래핑(Wrapping)한 세션 객체가 적재된다. 이 영속적 파이프들이 유지되는 동안 내부에서는 단순 데이터를 묻고 답하는(Request-Response) 패턴부터, 서버 측 인트라넷에서 일방적으로 변동 점을 통보하는(Notification) 푸시 패턴, 그리고 방대한 양의 텍스트가 조각나서 밀려 들어오는 스트리밍(Streaming) 패턴에 이르기까지 복잡 다단한 통신들이 뒤엉킨 채 흐른다.

본 절에서는 이러한 상태 보전형 파이프라인 위에서 각각의 RPC 메시지들이 어떻게 자신의 정체성을 유지한 채 오가는지, 논블로킹(Non-blocking)을 위한 ID 생성과 재전송 멱등성 체계, 나아가 비동기화의 최상위 제어 기법인 인터리빙(Interleaving) 처리 원칙에 이르기까지 구체적인 메시징 구조망을 면밀히 해부한다.
