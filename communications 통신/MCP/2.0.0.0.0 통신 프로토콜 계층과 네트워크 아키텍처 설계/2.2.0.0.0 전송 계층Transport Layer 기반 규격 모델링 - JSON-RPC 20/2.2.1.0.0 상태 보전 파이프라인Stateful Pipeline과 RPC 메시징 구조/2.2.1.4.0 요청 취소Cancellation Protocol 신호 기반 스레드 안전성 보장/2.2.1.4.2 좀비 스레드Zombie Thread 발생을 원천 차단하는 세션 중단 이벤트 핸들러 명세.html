2.2.1.4.2. 좀비 스레드(Zombie Thread) 발생을 원천 차단하는 세션 중단 이벤트 핸들러 명세

클라이언트가 발송한 notifications/cancelled 패킷이 서버의 통신 모듈(Dispatcher)에 무사히 도달했다 하더라도, 이미 운영체제의 커널(Kernel)이나 파일 시스템(I/O), 또는 GPU의 쿠다(CUDA) 연산으로 깊숙이 파고들어간 백그라운드 스레드를 멈춰 세우는 것은 프로토콜의 영역을 벗어난 시스템 공학의 지옥에 가깝다.
만약 이 런타임 제어권 회수에 실패한다면, 해당 연산은 결과를 반환할 대상이 사라졌음에도 무한히 실행되는 이른바 **좀비 스레드(Zombie Thread)**로 전락하여 서버 가용성(Availability)을 점진적으로 잠식해 들어간다.

이를 방어하기 위해 MCP 기반 서버 런타임은 철저한 중단 이벤트 기반의 컨텍스트 훅(Context Hook) 명세를 강제 적용한다.

0.1 AbortController와 트랜잭션 컨텍스트 주입

MCP 서버 내에서 개발자가 도구(Tool)를 구현할 때, 핸들러 함수의 인자(Arguments)로 단순 파라미터만 넘어오게 설계해서는 안 된다. 모든 핸들러는 반드시 메인 라우터로부터 [CancellationToken] 혹은 [Context.Context] (Go 언어 기준) 와 같은 생명주기 관리 객체를 주입(Injection)받아야만 한다.

클라이언트의 취소 알림이 도달하면 메인 라우터는 이 트리거를 당겨 CancellationToken.cancel()을 호출한다.

0.2 폴링(Polling)과 래퍼(Wrapper)를 통한 안전한 파괴

도구 내부의 무거운 반복문(While, For)이나 긴 지연시간(Sleep, 외부 HTTP 요청)을 포함한 연산을 수행할 때마다, 해당 로직은 매 마이크로 사이클마다 Token.isCancellationRequested() 상태를 확인(Polling)하는 방어 밸브를 필수적으로 이식해야 한다.
만약 플래그가 True로 뒤집힌 것을 스레드가 감지하면, 스레드는 자신이 쓰던 임시 파일 핸들을 닫고, 열어둔 DB 트랜잭션을 롤백(Rollback)한 뒤 AbortException을 던져(Throw) 스스로 목숨을 끊는다(Suicide).
이 엄격한 중앙 생명주기 핸들러 명세 덕분에, 에이전트의 충동적인 취소 폭격(Cancellation Spamming) 하에서도 MCP 서버 인프라는 결코 피를 흘리거나 메모리 파편을 남기지 않는다.
