2.2.1.4. 요청 취소(Cancellation Protocol) 신호 기반 스레드 안전성 보장

단일 요청과 응답으로 끝나는 전통적 마이크로 서비스와 달리, 거대 언어 모델(LLM)을 근간으로 하는 에이전트 트랜잭션은 때로 끔찍할 만큼 긴 체류 시간(Dwell Time)을 요구한다.
에이전트가 “로컬 코드베이스 전체의 심볼릭 링크를 분석하라“는 도구(Tool)를 호출했을 때, 이 연산은 경우에 따라 5분 이상의 무거운 I/O 대기 상태를 유발할 수 있다. 이때 인간 사용자(User)가 이 작업이 너무 길다고 판단하여 인터페이스의 ‘정지(Stop)’ 버튼을 누르거나, 클라이언트 에이전트 스스로 연산 방향이 틀렸음을 인지하고 백트래킹(Backtracking)을 결정하는 상황이 빈번하게 발생한다.

클라이언트 측에서 이미 발송한 트랜잭션이라 할지라도, 서버 측 연산을 중간에 강제 파괴(Abort)할 수 있는 매커니즘이 존재하지 않는다면 분산 네트워크는 곧 풀리지 않는 좀비 프로세스와 자원 독점이라는 파국을 맞이하게 된다.

이를 통제하기 위해 JSON-RPC 2.0의 본래 스펙을 상회하는 Model Context Protocol(MCP)만의 독자적 확장 규격인 요청 취소(Cancellation Protocol) 매커니즘이 설계되었다.
이는 단순한 네트워크 단절(Socket Close)이 아닌, 명시적으로 특정 Request ID를 지목하여 원격지 서버 인스턴스의 스레드 포크(Fork)를 안전하게 살해(Kill)하고 관련된 메모리 자원들을 운영체제에 말끔히 반환하는 고도의 우아한 셧다운(Graceful Shutdown) 절차이다.

이어지는 절에서는 취소 명령 전송 시나리오의 통신 규격부터, 백엔드 서버가 좀비 스레드를 방어하기 위해 어떤 이벤트 루프를 운용해야 하는지, 그리고 연쇄적으로 물려있는 서브 태스크(Sub-tasks)들을 어떻게 파괴하는지 그 구조적 명세를 상세히 해부한다.
