2.2.1.1.2. 반올림(Rounding) 원인 타임아웃 오류 및 메시지 유실에 대비한 멱등성(Idempotency) 재전송 제어

논블로킹 비동기 프로토콜망에서 고유 식별자(Request ID)가 혼선없는 교통정리를 수행한다지만, 불안정한 에이전트 분산 네트워크 환경은 이보다 더욱 파상적인 공격을 가해온다. 바로 **메시지 유실(Packet Loss)**과 미세한 시계 스큐(Clock Skew)로 야기되는 반올림 원인 타임아웃(Rounding Timeout) 에러이다.

클라우드망 인프라에서 수신측 MCP 서버의 과부하나, NAT 라우터의 세션 파괴로 인하여 패킷이 증발했을 때 클라이언트의 Promise 콜백 큐는 영원히 Request ID를 오매불망 기다리며 메모리 누수(Memory Leak)를 일으킨다. 이를 방어하기 위해 클라이언트는 엄격한 TTL(Time-To-Live) 타임아웃 타이머를 트리거해야만 한다.

0.1 찰나의 오판, 반올림 원인 타임아웃 현상극복

하지만 단순히 ’10초 후에 응답이 안오면 버린다’는 단편적 타임아웃 논리 기저에는 함정이 있다. MCP 로컬 노드의 서버가 9.99초 시점에 연산을 간신히 완료하고 결과값을 반환 프레이밍(Framing)할 때, 커널에서 마이크로 타이머를 스레드로 컨텍스트 스위치하는 도중 CPU 반올림 오차가 더해져 클라이언트가 10.01초에 수신하는 일이 빈번히 일어난다.
타임아웃은 이 패킷을 에러로 간주해 폐기해 버리고, 클라이언트는 서버 측이 아무 작업도 하지 않았다고 오판해 버리는 치명적 부조화가 발생한다.

0.2 멱등성(Idempotency) 위임 및 트랜잭션 재전송(Retry)

이러한 부조화와 메시지 유실 피해를 능동 치유하기 위해, MCP 트랜스포트 레이어는 실패로 낙인찍힌 Request ID 패킷을 백오프 지연(Exponential Back-off) 후 재전송(Retry)하는 방어 시스템을 가동한다.
그러나 같은 지시를 두 번 날렸을 때 서버가 앞선 트랜잭션의 잔해를 기억하지 못하고 데이터베이스에 행(Row)을 2번 삽입(Insert)하는 만행을 저지르면 안 된다.

따라서 MCP 프로토콜 하에서 민감한 데이터 갱신형 도구 호출(Mutation Tool Call)은 서버 노드 쪽에 설계 레벨의 철저한 멱등성(Idempotency) 강제 원칙을 요구한다. 클라이언트가 똑같은 Request ID나 idempotency_key 파라미터를 실어 재요청 패킷을 송신하면, 서버는 수신단 캐시 레이어에서 “아, 이건 아까 내가 9.99초에 처리 완료했던 이전 식별자의 쌍둥이 요청이구나“라고 인지해야 한다. 그리고 부작용(Side Effect) 일으키는 로직을 통과(Bypass)시킨 뒤, 이전의 성공 결과를 그대로 재반환함으로써 파괴된 타임아웃의 악순환 고리를 매끄럽게 접합해낸다.
