2.2.1.1. 비동기 메시지 송수신(Request-Response) 패턴 구현 기술

분산 컴퓨팅 및 에이전트 파이프라인망에서 성능과 동시성을 좌우하는 가장 치명적인 인프라는 메시지의 송수신 구조 그 자체이다. 코파일럿(Copilot)과 같은 LLM 클라이언트가 로컬 개발 환경과 무수히 얽혀 구동될 때, 클라이언트가 단 하나의 명령(Request)을 서버로 발송하고 그 응답(Response)이 올 때까지 스레드를 동기식(Synchronous)으로 정지(Block)시킨다면, 사용자는 IDE 화면의 지독한 프리징(Freezing) 현상을 겪게 될 것이다.

이를 타파하기 위해 MCP는 JSON-RPC 2.0이 제시하는 고유의 속성을 활용하여, 통신의 근간을 완벽한 비동기 메시지 송수신(Asynchronous Request-Response) 패턴으로 철저히 개조하여 모델링하였다.

0.1 논블로킹(Non-blocking) 파이프라인의 완성

이 비동기 체계 하에서는 클라이언트가 파일 읽기 작업 같은 무거운 도구 호출을 서버에 지시(Dispatch)한 직후, 결과를 기다리기 위해 메인 이벤트 루프(Event Loop)를 공전시키지 않는다. 클라이언트 측 트랜스포터 모듈은 그저 JSON 형태의 페이로드를 Stdio 파이프나 TCP 소켓 버퍼에 밀어 넣은(Write) 후 곧장 제어권을 다음 로직으로 반환(Return)해 버린다.
이후 서버가 연산을 마치고 독자적인 시그널과 함께 결과 텍스트를 돌려주기 전까지, 클라이언트는 화면 UI 이벤트를 처리하거나 다른 MCP 노드에게 병렬로 수십 개의 추가 쿼리를 방사할 수 있는 무한의 자유도를 획득한다.

본 절에서는 이러한 논블로킹 세계관이 실전에서 겪게 되는 무서운 부작용인 통신 간 꼬임(Interleaving) 현상을 어떻게 식별자(ID) 기반으로 방어하는지, 핑거프린트 분실에 따른 재시도(Idempotent Retry) 규정은 무엇인지 세부 구현 기술을 하나씩 파고들 것이다.
