2.2.1.1.1. 고유 식별자(Request ID)를 활용한 논블로킹(Non-blocking) 동시성 요청 충돌 방지 체계

비동기 메시지 시스템이 병렬적 성능의 비약적 향상을 약속하지만, 이를 통제 없이 풀어버릴 경우 지독한 교착 상태나 결과값 오염에 시달리게 된다.
클라이언트(Host)가 거의 찰나의 순간에 서버를 향해 “A 시스템 로그 조회”, “B 데이터베이스 쿼리”, “C 스크립트 실행“이라는 세 가지의 무거운 요청(Request)을 하나의 파이프라인(Socket)에 연달아 던졌다고 상상해 보라. 이들의 처리 시간은 모두 다르기에, 서버가 이 세 가지 작업에 대한 처리 결과를 뒤죽박죽된 순서로 돌려보낼(Response) 가능성이 농후하다.

만약 클라이언트가 “가장 먼저 보낸 것이 가장 먼저 돌아올 것이다(FIFO)“라는 순차적 믿음을 가지고 동기식 큐를 매핑 하려 든다면, C 스크립트의 결과가 A 로그 변수에 오염되어 흘러 들어가는 참극이 빚어진다.
이를 원천적으로 차단하기 위해 JSON-RPC 통신의 심장부에는 **고유 식별자(Request ID)**를 강제하는 논블로킹 충돌 방지 체계가 도입되었다.

0.1 퓨처/프로미스(Promise)의 맵핑 레지스트리

모든 MCP 도구 호출 쿼리는 내부 전송 모듈을 타기 전 반드시 런타임 내에서 전역(Global) 범위로 유일무이한 숫자형 혹은 고유 문자열(UUID) 형태의 id 속성을 강제 할당받는다({ "jsonrpc": "2.0", "method": "...", "id": "4b9a-..." }).

이 id는 단지 전송망으로 흩뿌려지는 데 그치지 않는다. 클라이언트는 이 문자를 발송함과 동시에 자신의 인메모리 딕셔너리에 Promise<Response> (혹은 타 언어의 Future 객체) 상태와 이 식별자 id를 1:1로 록킹(Locking)한 채 보류망(Pending Request Map)에 저장한다.

0.2 역순 귀환(Out-of-Order Response)의 안전한 해석

시간이 흘러 비동기망으로부터 서버의 응답 덩어리가 마구잡이 순서로 날아온다 하더라도, 클라이언트의 페이로드 라우터 모듈은 전혀 당황하지 않는다. 수신된 패킷에서 { "jsonrpc": "2.0", "result": "...", "id": "4b9a-..." }라는 심볼을 추출하여, 아까 보류망에 잠재워두었던 식별자 키(Key) 값과 정확히 대조한다. 일치하는 Promise 컨텍스트 트리를 깨워 단독으로 resolve() 처리하고, 보류망 캐시에서 해당 키를 즉각 삭제해버리는 방식이다.

이 메커니즘을 통해 MCP 클라이언트는 통신 채널을 단 하나만 열고서도 동시에 수백 개의 교차되는 콜아웃(Callout) 트랜잭션을 아무런 간섭(Collision) 없이 논블로킹 병렬 파이프라인 안쪽에 수납하는 경이로운 다중 통신 안정성을 실현하게 된다.
