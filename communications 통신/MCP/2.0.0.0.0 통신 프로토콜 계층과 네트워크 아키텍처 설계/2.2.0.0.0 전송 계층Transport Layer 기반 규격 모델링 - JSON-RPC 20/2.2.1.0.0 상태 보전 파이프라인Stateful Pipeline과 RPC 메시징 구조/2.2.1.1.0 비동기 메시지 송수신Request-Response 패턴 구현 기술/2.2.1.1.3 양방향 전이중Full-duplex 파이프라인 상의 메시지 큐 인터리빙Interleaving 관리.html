2.2.1.1.3. 양방향 전이중(Full-duplex) 파이프라인 상의 메시지 큐 인터리빙(Interleaving) 관리

JSON-RPC 통신망 내에서 식별자와 멱등성 재전송 룰이 비동기 트랜잭션의 안전장치 역할을 해주지만, 데이터 스트림의 물리적 흐름 관점에서는 또 다른 무시무시한 장벽이 존재한다. MCP가 기반으로 하는 통신 채널(Stdio, WebSocket 등)은 본질적으로 클라이언트와 서버가 동시에 서로를 향해 데이터를 뿜어낼 수 있는 양방향 전이중(Full-duplex) 파이프라인이다.

이 파이프라인을 타고 클라이언트가 서버로 질의 A, B, C를 보냄과 동시에, 서버는 과거에 받았던 질의 Z에 대한 스트림 응답 조각들(Chunks)과 백그라운드의 자원 알림(Notification) 이벤트를 클라이언트 쪽으로 무차별적으로 발사할 수 있다.

0.1 메시지 인터리빙(Interleaving)의 카오스

이처럼 파이프라인 양방향에서 각기 다른 맥락의 패킷들이 믹서기처럼 섞여서 들어오는 현상을 **인터리빙(Interleaving)**이라 부른다. 만약 트랜스포터(Transporter) 계층에 정교한 격벽(Isolation) 장치가 없다면, 클라이언트의 JSON 파서(Parser)는 A 콜아웃의 응답값을 조립하는 도중 뜬금없이 날아온 Z 응답의 꼬리표나 Notification 패킷을 이어붙여 치명적인 구문(Syntax) 파괴 에러(Unexpected token { in JSON at position...)를 폭발시키게 된다.

0.2 다중 버퍼링(Multi-buffering)과 메시지 큐 분배망

이 카오스를 통제하기 위해 MCP 전송 계층은 단일 수신 버퍼 통(Bucket)을 사용하지 않는다. 바이트(Byte) 단위의 스트림이 소켓으로 수신되는 즉시, 선행 구문 분석기(Lexer)가 패킷 최상단의 개행 문자(Newline)나 프레임 헤더를 인식해 완전한 단위의 1개의 JSON 객체로 도려낸다(Slice).

도려내진 각각의 원시 JSON 객체들은 그 내부의 필드 구조(method 속성의 유무, id 속성의 유무)에 따라 **목적지 메시지 큐(Message Queue)**가 갈라진다. 응답(Response) 큐, 알림(Notification) 큐, 신규 요청(Request) 큐로 각각 컨텍스트가 라우팅(Routing)되어 적재되므로, 각 큐를 소비하는 워커 스레드들은 다른 성격의 트래픽이 파이프라인 안에서 아무리 난도질되어 섞여 들어오더라도 완벽하게 본연의 패킷만을 온전히 조립해 낼 수 있는 위대한 구조적 안정성을 획득한다.
