2.2.1.3.1. 부분(Partial) 응답을 여러 차례 분할 송신하여 메모리 사용량을 절감하는 청크형(Chunked) 데이터 교환망

MCP 디스패처가 10GB 크기의 시스템 로그 복사 명령을 수신했다고 가정하자. 기존의 동기식 통신에서는 서버가 디스크에서 이 10GB를 모두 메모리로 끌어올린 뒤에야 단 하나의 클라이언트 응답 체인으로 만들 수 있었다. 이는 서버와 클라이언트 양측의 메모리 팽창(Memory Bloat)을 초래하는 파멸적 구조이다.

0.1 스트리밍 분할 전송 매커니즘

이를 해결하기 위해 MCP는 특정 도구(Tool) 호출에 대해, 단일 결과(Result) 발송을 약속하는 대신 청크형(Chunked) 이벤트 스트림 브로드캐스팅으로 우회하는 라우팅 기법을 허용한다.
서버는 10GB 파일을 디스크로부터 아주 작은 윈도우 크기(예: 64KB)만큼씩만 읽어 들인다. 그리고 한 번 읽은 64KB의 데이터 덩어리를 즉각 JSON-RPC 메시지에 래핑하여 파이프라인에 푸시(Push)해 버린 후, 로컬 메모리 버퍼를 곧바로 비워낸다(Flush).

에이전트 클라이언트 측에서는 이 파편화된 조각들을 수신할 때마다 내부의 전체 컨텍스트 윈도우 한구석에 ’이어붙이기(Append)’만 수행하면 된다.
이 과정에서 서버와 클라이언트 그 어느 쪽도 전체 10GB 파일에 상응하는 거대한 힙(Heap) 메모리를 보유할 필요가 없어지는 극단적인 이득이 발생한다. 더불어 클라이언트는 데이터 집계가 완전히 끝날 때까지 멍하니 대기할 필요 없이, 가장 처음 수신된 64KB 청크 텍스트를 단 1밀리초(ms) 만에 넘겨받아 즉시 화면에 렌더링하거나 LLM의 프롬프트 전처리 파이프라인에 사전에 태울 수 있는(Pipelining) 비동기적 자유도를 획득한다.
