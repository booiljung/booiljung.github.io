2.2.1.3.2. 멀티파트 포맷(Multipart) 매핑을 통한 이진 파일(이미지/PDF) 컨텍스트 전송 로직

비전(Vision) 기능을 갖춘 최신 에이전트들이 등장하면서, 텍스트 일변도였던 도구 호출 결과물에 PNG 이미지, PDF 스캔 본, 오디오 스펙트로그램 등 **순수 이진 트래픽(Binary Payload)**이 범람하기 시작했다.
그러나 태생적으로 JSON 문자열 규칙체계인 범용 JSON-RPC 2.0 포맷은 이진 데이터를 그대로 전송 버퍼에 싣는 것을 허용하지 않았다. 바이트 배열을 강제로 스트링 변환하려다간 제어 문자(Control Character) 파싱 오류가 발생하기 십상이다.

0.1 Base64 인코딩과 텍스트 페이로드 확장의 한계

초기에 이 문제를 우회했던 유일한 방법은 이진 데이터를 Base64로 강제 인코딩(Encoding)하여 JSON의 문자열 밸류(Value)로 밀어 넣는 방식이었다. 하지만 이 경우 33% 이상의 엄청난 페이로드 증폭 오버헤드가 발생하며 파싱을 위한 무거운 CPU 연산 비용이 지출된다는 명백한 결함이 있었다.

0.2 멀티파트 스트림(Multipart Stream) 바인딩

이 난관을 파훼하기 위해 MCP의 스트리밍 계층은 HTTP/1.1이 증명했던 위대한 발명품인 멀티파트 스트림(Multipart/form-data)의 바운더리(Boundary) 개념을 RPC 전송 계층의 섀도우 채널로 맵핑하는 설계를 취한다.

클라이언트가 이미지 해독을 명령하면, 서버는 응답 패킷의 메인 JSON 바디 안에 이미지를 담지 않는다. 대신 {"type": "image", "content_type": "image/png", "stream_id": "vid-839"}와 같은 가벼운 메타데이터 포인터만을 반환한다. 그리고 그 직후, 전송 파이프라인 최하단을 관장하는 바이너리 펌프(Binary Pump)가 별도의 제어 채널이나 특수한 바운더리 마커를 선두로 세운 뒤 순수하게 압축된 바이트 배열 덩어리를 송출해버린다.

클라이언트의 바이너리 인터셉터 모듈은 파이프라인으로 들어오는 이 멀티파트 스트림이 발견되면 즉시 JSON 파싱을 멈추고 바이트를 래핑한 뒤 LLM의 비전 해석기 측으로 곧장 우회(Bypass) 투사한다. 이러한 이진 컨텍스트 주입 로직을 통해 에이전트망은 비정형 미디어 트래픽을 처리함에 있어 그 어떤 오버헤드나 포맷 단절도 겪지 않는 궁극의 투명성(Transparency)을 유지한다.
