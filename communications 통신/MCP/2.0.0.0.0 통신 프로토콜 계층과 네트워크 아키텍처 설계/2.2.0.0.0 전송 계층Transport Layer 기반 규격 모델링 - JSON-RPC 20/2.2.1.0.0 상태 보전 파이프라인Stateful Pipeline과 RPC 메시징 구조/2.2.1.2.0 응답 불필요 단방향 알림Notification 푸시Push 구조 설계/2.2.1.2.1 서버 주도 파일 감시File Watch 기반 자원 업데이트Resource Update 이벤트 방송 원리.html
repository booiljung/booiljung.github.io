2.2.1.2.1. 서버 주도 파일 감시(File Watch) 기반 자원 업데이트(Resource Update) 이벤트 방송 원리

MCP 생태계 내에서 가장 대표적으로 ‘알림(Notification)’ 시그널이 폭발하는 구역은 이른바 **자원 주입망(Resource Injection Network)**이다.
특히 IDE 환경과 결합된 코파일럿 에이전트는 사용자가 실시간으로 편집하는 수십 개의 소스 코드 파일을 자신의 두뇌(Context Window) 안에 항시 동기화해 두어야 한다. 사용자가 파일에 console.log("hello")라는 코드 한 줄을 추가하고 저장하는 그 찰나의 순간, 에이전트 측은 이 변경 사실을 득각 인지해야 한다.

0.1 커널 수준의 인덱스 파일 감시망(inotify / FSEvents) 포팅

이를 실현하기 위해 로컬 호스트(PC)에 상주하는 MCP 서버 노드는 운영체제의 커널(Kernel) API와 극도로 밀착된다.
서버가 부트스트래핑(Bootstrapping)될 때 리눅스의 inotify, 맥 OS의 FSEvents, 혹은 윈도우의 ReadDirectoryChangesW와 같은 극도로 로우 레벨(Low-level)인 파일 시스템 이벤트 감시기(File Watcher)를 메모리 백그라운드 상에 띄운다.

클라이언트로부터 특정 자원(예: mcp://local/src/main.js)에 대한 감시 권한(Subscribe)을 인계받는 순간, 이 워처 데몬은 해당 파일의 IN_MODIFY(수정됨), IN_DELETE(삭제됨) 등의 시스템 I/O 인터럽트를 가로채게(Intercept) 된다.

0.2 이벤트를 단방향 질의(JSON-RPC Notification)로 캐스팅

감시기가 마이크로초(ms) 단위의 트리거를 울리면, MCP 서버는 즉각 이것을 JSON-RPC의 notifications/resources/updated 메소드로 변환(Casting)한다.
앞서 논의한 단방향 알림(Notification) 규격에 따라 id 값은 포함하지 않으며, 단지 변경된 자원의 URI 주소만을 페이로드에 욱여넣은 채, 유지되고 있는 양방향 파이프라인(Stdio 또는 SSE)을 타고 에이전트 클라이언트를 향해 통보 패킷을 허공에 브로드캐스팅(Broadcasting)한다.

이러한 서버 주도 메커니즘을 통해서만 클라이언트는 아무런 리소스를 소모하지 않고도 외부 세계 물리 엔진의 변화를 실시간 촉수로 느끼고, 스스로 환각(Hallucination)에 빠지는 치명상을 원천 예방할 수 있게 된다.
