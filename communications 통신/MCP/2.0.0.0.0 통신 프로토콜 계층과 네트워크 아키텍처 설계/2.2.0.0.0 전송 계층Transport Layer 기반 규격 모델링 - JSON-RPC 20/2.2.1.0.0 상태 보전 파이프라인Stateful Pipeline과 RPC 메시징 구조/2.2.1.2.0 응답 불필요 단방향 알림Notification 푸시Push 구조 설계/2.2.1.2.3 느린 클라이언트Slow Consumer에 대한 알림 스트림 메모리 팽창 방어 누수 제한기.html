2.2.1.2.3. 느린 클라이언트(Slow Consumer)에 대한 알림 스트림 메모리 팽창 방어 누수 제한기

단방향 푸시(Push) 아키텍처는 필연적으로 생산자(Producer)와 소비자(Consumer) 간의 속도 비대칭성이라는 치명적인 아킬레스건을 잉태한다. MCP 서버망(생산자)은 초당 3만 건의 시스템 변동 알림(Notification)을 날릴 수 있는 괴력을 지녔지만, 정작 이를 수신하는 IDE 환경 기반 LLM 클라이언트(소비자) 프로그램은 메인 스레드가 UI 렌더링에 처절하게 매달려 있어 초당 500건밖에 처리하지 못하는 사태가 벌어질 수 있다.

이처럼 클라이언트가 서버의 발송 속도를 따라가지 못하는 현상을 일컬어 일명 느린 클라이언트(Slow Consumer) 병목 사태라 규정한다.
단방향 푸시 규격은 클라이언트가 “잠깐 멈춰, 소화하게 시간 좀 줘!(Backpressure)“라고 응답할 의무(Response)마저 제거했기 때문에, 서버의 소켓 전송 버퍼와 클라이언트의 수신 OS 커널 메모리는 미처 파싱되지 못한 JSON 알림 스트림 텍스트 덩어리로 무한 팽창(Memory Bloat)하다 종국엔 OOM(Out-Of-Memory)으로 산화하게 된다.

0.1 백업 큐 제한기(Bounded Queue Leaky Bucket) 가동

강제 배칭(Batching)으로도 해결되지 않는 이 파국적 팽창을 원천 통제하기 위해, 전송 계층 최말단 버퍼(Buffer) 앞단에는 고정된 크기의 한계치 큐(Bounded Queue) 알고리즘이 누수 제한기(Leaky Bucket) 형태로 매립되어 있다.

클라이언트 인메모리에 파싱 대기 중인 JSON-RPC 알림 패킷의 숫자가 임계치인 Max_Pending_Notifications (예: 10,000건)를 돌파하는 순간 파이프라인 방어막이 발동한다.
방어막이 쳐지면, 큐 디스패처 엔진은 이후 서버로부터 비 쏟아지듯 밀려 들어오는 “과거 파일 A 수정”, “과거 파일 B 삭제” 알림 등의 **미래 꼬리 패킷들을 자비 없이 커널 레벨에서 폐기(Drop, Tail-drop)**해버린다.

0.2 상태 오염 방주(Ark) 복원 정책

하지만 알림을 폐기해 버리면 클라이언트는 변경된 외부 세상의 최신 상태에서 도태되어 치명적인 환각 상태에 빠진다. 따라서 큐 제한기는 패킷을 허공에 버림과 동시에 즉각적으로 “자원 일관성이 파괴되었다(State Desynced)“라는 내부 적색경보(Red Flare) 플래그를 세운다.

UI 스레드의 부하가 해소되고 클라이언트 버퍼가 0으로 비워졌을 때, 이 적색경보 플래그가 세워져 있음을 인지한 트랜스포트 레이어는 즉각 최상위 복구 절차에 돌입한다. 이전까지 구독 중이던 파일 목록의 **최신 컨텍스트 스냅샷의 전체 본문(Full Snapshot)을 한 번에 재조회(Re-fetch)**하는 강력한 동기식 풀링(Polling) 수습 망을 가동하는 것이다.
이를 통해 메모리 폭파라는 시스템 전체의 파멸은 회피하면서도, LLM 에이전트의 상황 인지 연속성을 완벽히 재동기화하는 아슬아슬한 극한의 줄타기가 실현된다.
