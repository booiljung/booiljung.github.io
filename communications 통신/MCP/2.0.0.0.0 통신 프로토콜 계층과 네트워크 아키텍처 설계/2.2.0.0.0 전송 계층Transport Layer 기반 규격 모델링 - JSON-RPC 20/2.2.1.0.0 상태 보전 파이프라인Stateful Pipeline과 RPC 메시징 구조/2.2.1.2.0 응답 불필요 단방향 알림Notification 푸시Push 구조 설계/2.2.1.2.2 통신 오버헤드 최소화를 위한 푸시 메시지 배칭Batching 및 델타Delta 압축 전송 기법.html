2.2.1.2.2. 통신 오버헤드 최소화를 위한 푸시 메시지 배칭(Batching) 및 델타(Delta) 압축 전송 기법

커널 레벨의 파일 감시 시스템이 런타임 컨텍스트 자동 동기화의 축복을 내렸지만, 소프트웨어 공학의 세계는 언제나 가혹한 트레이드오프(Trade-off)를 동반한다.
개발자가 터미널에서 git checkout branch-new를 입력하거나, npm install을 기동하여 프로젝트 내의 파일 수천 개가 단 1초 만에 갱신되었다고 가정해 보자. 감시망(File Watcher)은 순식간에 5,000번의 IN_MODIFY 인터럽트를 울릴 것이며, 서버는 5,000번의 JSON-RPC notifications/resources/updated 패킷을 생성해 작은 Stdio 파이프라인에 소방호스 발사하듯(Fire-hose) 박아 넣으려 들 것이다. 이는 전송망 버퍼 범람과 JSON 파서의 자원 고갈(CPU Spiking)을 동반하는 통신 오버헤드 재앙이다.

0.1 시간 윈도우 기반 이벤트 배칭(Time-Window Batching)

이를 원천적으로 차단하기 위해 MCP 서버 내부의 이벤트 디스패처는 마이크로 지연 타이머 기반의 배칭(Batching) 파이프를 중간에 설립한다.
시스템 인터럽트가 발생하면 디스패처는 이 알림을 즉시 발송하지 않고 약 50ms에서 200ms 사이의 인위적인 대기소(Holding Pen)에 가둔다. 이 찰나의 시간 동안 특정 폴더에서 수백 개의 파일 수정 이벤트가 동시다발로 도달한다 하더라도, 디스패처는 그것들을 잘게 썰어 발송하지 않고 하나의 거대한 JSON Array 객체 안에 배열 인자로 줄줄이 래핑(Wrapping)한다.
그 결과 수천 번 발생할 뻔했던 TCP/IP 전송 혹은 IPC 컨텍스트 스위칭의 부하가 단 한 번의 단일 대형 트랜잭션 전송으로 병합(Squashed) 처리되는 기적이 발생한다.

0.2 컨텍스트 이중화 방어망: 델타(Delta) 압축

설령 이벤트 통지망을 뚫고 지나갔더라도 페이로드 본문의 비대화라는 난관이 도사린다. 100MB짜리 데이터 로그 파일의 끝에 단 1줄이 업데이트되었다고 해서 전체 100MB의 스냅샷 텍스트를 응답 불필요 푸시 통로로 밀어 넣는 것은 최악의 설계이다.

MCP의 진보된 통신망은 이처럼 크기가 큰 텍스트 덩어리를 취급할 때, 변경 이전 텍스트와 변경 이후 텍스트의 차이점만을 도출해내는 바이트 델타 압축(Byte-Delta Compression) 알고리즘을 차용한다. {"start_line": 9999, "new_content": "[error] db timeout\n" } 와 같이, 수정된 좌표계와 짧은 청크만을 JSON 페이로드에 얹어 발송함으로써, 백그라운드 푸시망은 네트워크 패킷 한도(MTU) 안에서 영원하고 가벼운 춤을 이어나갈 수 있게 된다.
