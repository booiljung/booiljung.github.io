2.2.2.1. 추상화된 에러 송출(Fault Tolerance) 코딩 표준 체계

거대 언어 모델(LLM) 기반의 분산 시스템 아키텍처에서 에러(Error)의 처리는 단순한 시스템 로그 백업의 차원을 한참 넘어선다. 에이전트 환경에서 발현된 에러 객체 자체는 더 이상 버그의 잔재망이 아니라, **“LLM이 스스로 방향을 수정(Self-Correction)하고 다음 행동을 교정하도록 돕는 강력한 프롬프트 컨텍스트(Prompt Context)”**로 기능하기 때문이다.

따라서 MCP 서버를 개발하는 백엔드 엔지니어가 NullPointerException이나 파일 접근 권한 거부(EACCES)와 같은 스택 트레이스(Stack Trace)의 파편을 클라이언트 측으로 여과 없이 직사(Direct Hit)하는 행위는 절대로 용납되지 않는다. 무분별하게 날아온 날것의 에러 텍스트는 클라이언트 LLM의 프롬프트를 심각하게 오염시켜 컨텍스트 윈도우의 주의력 집중(Attention Mechanism)을 교란시키는 주범이 된다.

결과적으로, MCP 생태계의 모든 서드파티 통합(Integration) 모듈은 로컬 서버 코드베이스 내부의 특정 런타임 예외(Exception) 객체들을 JSON-RPC 범용 에러 코드북과 **사용자 친화적인 심각도 레벨(Severity Levels)**로 극도로 정제하여 번역해 내는 추상화(Abstraction) 래퍼를 전송 계층 최전방에 고정적으로 덧씌우는 표준 체계를 따르도록 설계되었다.
