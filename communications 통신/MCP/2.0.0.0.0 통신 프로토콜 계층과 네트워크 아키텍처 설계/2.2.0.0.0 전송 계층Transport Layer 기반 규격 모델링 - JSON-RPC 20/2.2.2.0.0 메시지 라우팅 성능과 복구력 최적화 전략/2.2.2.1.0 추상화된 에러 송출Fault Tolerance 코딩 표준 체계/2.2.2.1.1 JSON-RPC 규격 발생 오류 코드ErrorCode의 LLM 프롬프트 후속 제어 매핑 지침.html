2.2.2.1.1. JSON-RPC 규격 발생 오류 코드(ErrorCode)의 LLM 프롬프트 후속 제어 매핑 지침

MCP 프로토콜은 기저 전송망으로 JSON-RPC 2.0 스펙을 온전히 흡수하였다. 따라서 MCP 서버 프레임워크는 내부 로직의 수행 결과에 따라 JSON-RPC 표준 규칙인 -32000부터 -32768 대역폭 사이의 에러 코드(Error Code) 제어 규약을 완벽하게 구현할 의무가 있다.
클라이언트 쪽에 내장된 LLM은 이러한 표준 Error Code 규격을 역으로 파싱하여, 자신이 과거에 던졌던 프롬프트가 무엇 때문에 튕겨져 나왔는지를 메타인지(Meta-cognition)하고 즉각 우회 기동을 실시한다.

0.1 표준 에러 코드와 오토 에이전트(Auto-Agent)의 후행 반응

  1. -32600 (Invalid Request): LLM이 도구(Tool)를 호출할 때 문법에 맞지 않는 JSON 객체를 던졌거나 타입을 틀리게 보낸 경우. 클라이언트 디스패처는 이 코드를 수신하면 LLM에게 “JSON 형태가 깨졌으니 Schema를 다시 읽고 제출 포맷을 교정할 것” 이라는 시스템 프롬프트를 자동 사출(Injection)한다.
  2. -32601 (Method Not Found): LLM이 환각(Hallucination)에 빠져 현재 연결된 MCP 서버에 존재하지도 않는 가상의 함수명(예: reboot_system)을 호명했을 때 발생한다. 에이전트는 이를 수신하여 “능력값 검증(Capabilities Discovery)을 재수행하여 사용 가능한 툴 리스트를 갱신하라“는 판단을 내린다.
  3. -32602 (Invalid Params): read_file 도구를 호출했는데 필수 인자인 filepath가 누락된 경우. 에이전트는 누락된 파라미터를 다시 추론하여 호출 체인을 자가 치유(Self-heal)한다.

0.2 애플리케이션 정의 확정 오류 대역 (-32000 ~ -32099)

만약 파일 서버의 디스크가 꽉 찼거나(Disk Full), 데이터베이스 연결이 일시적으로 거부되었다면 위 표준 코드에는 해당되지 않는다.
이때 서버 엔지니어는 -32000 대역폭인 서버 에러(Server Error) 포맷을 채빙하여, {"code": -32001, "message": "Database Timeout. Retry after 5 secs"} 와 같이 message 속성에 인간 지향적인 명확한 지침을 텍스트로 적어 반환해야 한다. LLM은 이 message를 단순한 텍스트로 파싱해 의미를 읽어낸 후, ’5초 대기 후 재전송’이라는 백오프(Backoff) 전략을 스스로 실행하게 된다.
