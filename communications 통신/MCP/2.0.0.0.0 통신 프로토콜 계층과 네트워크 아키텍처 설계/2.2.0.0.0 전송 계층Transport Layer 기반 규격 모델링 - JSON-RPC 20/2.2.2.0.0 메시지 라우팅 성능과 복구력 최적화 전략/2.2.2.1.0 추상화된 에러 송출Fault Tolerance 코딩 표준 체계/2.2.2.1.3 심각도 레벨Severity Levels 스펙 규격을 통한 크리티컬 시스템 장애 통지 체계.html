2.2.2.1.3. 심각도 레벨(Severity Levels) 스펙 규격을 통한 크리티컬 시스템 장애 통지 체계

서버와 에이전트 클라이언트 사이를 가로막는 장애 상황은 단순한 API 응답 실패에서 그치지 않는다. 가령 서버 애플리케이션이 묶여 있는 백그라운드 DB 컨테이너가 폭파(Down)되었거나, 로컬 폴더에 대한 파일 시스템 모니터링 데몬 스레드가 OS 권한 부족으로 비정상 종료되는 이른바 ‘서버의 호흡 정지’ 수준의 장애가 도래할 수 있다.

이때 서버는 더 이상 클라이언트의 도구(Tool) 호출을 기다리며 Error Code로 응답할 여유가 없다. 클라이언트의 질의가 없더라도 즉각 자신의 생명 유지 불가를 세상에 브로드캐스팅해야 한다.
이를 위해 MCP 전송 계층은 시스템 백그라운드 장애를 일방 푸시(Push) 형태로 전송하는 크리티컬 로그 알림(Logging Notification) 스펙을 독자적으로 이식해 두었다.

0.1 Logging 메시지의 추상화 포맷과 심각도(Severity)

이 통지 체계는 순수하게 관측 및 디버깅을 목적으로 하며, JSON-RPC 알림(Notification) 포맷을 빌려 notifications/message 메소드로 발송된다.
이때 서버는 단순한 텍스트 덤프(Dump)를 보내는 대신, 시스로그(Syslog)의 표준과 유사한 형태의 정밀한 정수 **심각도 구조체(Severity Levels)**를 통해 에이전트 클라이언트가 현재 상황의 치명률을 기계적으로 해석하게 만든다.

  • level: 1 (Error): 런타임 중 특정 쿼리의 처리 실패. 에이전트는 재시도 전략을 펼친다.
  • level: 2 (Critical): 내부 데몬이나 인프라망의 단절. 특정 툴들의 결함 예상. 에이전트는 해당 툴을 배제하고 다른 노드로 라우팅한다.
  • level: 3 (Alert): 인프라 전체의 심각한 파손.
  • level: 4 (Emergency): 서버 자신의 크래시 임박. 즉시 연결 해제 요망.

서버가 송출한 Emergency 심각도 페이로드가 클라이언트에 닿는 순간, 에이전트 라우터는 0.1초 만에 해당 서버 파이프라인을 완전히 차단(Banish)시키고 헬스 체크(Health Check) 복구망이 회복될 때까지 어떠한 패킷 오염도 허락하지 않는 무중단(Zero-downtime) 우회로 격리를 단행한다.
