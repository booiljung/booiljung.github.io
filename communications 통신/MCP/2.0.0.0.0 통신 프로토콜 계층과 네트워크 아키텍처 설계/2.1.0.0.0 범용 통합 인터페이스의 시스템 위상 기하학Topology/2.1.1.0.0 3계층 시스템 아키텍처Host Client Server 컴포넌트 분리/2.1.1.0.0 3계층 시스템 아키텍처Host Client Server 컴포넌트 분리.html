2.1.1. 3계층 시스템 아키텍처(Host, Client, Server) 컴포넌트 분리

소프트웨어 시스템의 복잡도를 제어하는 최선의 공학적 접근법(Engineering Practice)은 단연코 ’관심사의 분리(Separation of Concerns)’이다. 하나의 프로그램이 렌더링, 네트워크 패킷 라우팅, 데이터 연산, 그리고 인공지능 추론까지 모두 감당하게 되면 코드는 필연적으로 결합도(Coupling) 높은 스파게티 덩어리형 모노리스(Monolith)로 전락하게 된다.

Model Context Protocol(MCP)은 이 거대한 인지 아키텍처망의 데이터 흐름을 명확하게 규제하기 위해 기존의 2계층 클라이언트-서버(Client-Server) 구도를 파괴하고, **호스트(Host) - 클라이언트(Client) - 서버(Server)**라는 명맥형 3계층 아키텍처(3-Tier Architecture) 모델을 프로토콜의 표준 규격으로 확립하였다.

이 3계층 시스템 컴포넌트 분리 모델은 다음의 세 가지 명확한 경계선(Boundary)을 구축한다.

  1. 호스트(Host): 사용자의 눈과 맞닿아 있는 애플리케이션의 껍데기(Shell)이다. 통합 개발 환경(IDE, 예: Cursor, VSCode) 시스템이나 챗봇(Chatbot)의 메인 윈도우 UI를 뜻한다. 호스트는 데이터와 인공지능이 내부에서 어떻게 돌아가는지는 알 필요도, 알아서도 안 된다. 오직 정보를 보기 좋게 렌더링(Rendering)하고, 시스템에 심각한 변경(Destructive Action)을 가하려 할 때 사용자에게 “승인하시겠습니까?“라는 팝업 창을 띄우는 보안 뷰(Security View) 역할에만 철저히 집중한다.
  2. 클라이언트(Client): 사용자의 눈을 떠난 그 뒤편 백그라운드 프로세스에서 묵묵히 돌아가는 인지망의 ’통신 심장’이다. 클라이언트는 실제 대규모 언어 모델(LLM API)과 통신하며 의도를 해석하고, 그 의도에 맞추어 여러 개의 MCP 서버 연결 풀(Connection Pool)을 조율하고 패킷을 전송하는 헤비웨이트(Heavy-weight) 중개 역할을 전담한다.
  3. 서버(Server): 가장 바깥쪽의 에지(Edge) 노드에 위치한 정보 공급자다. 서버 컴포넌트는 오직 자신이 보유한 파일 시스템, 깃허브(GitHub) API, 혹은 로컬 데이터베이스 쿼리에만 집중하며 데이터 청크(Chunk)를 제공할 뿐, 이 데이터가 어떤 LLM 모델에서 어떻게 인간 언어로 조립되는지에 대한 컨텍스트는 일체 보유하지 않는 무상태(Stateless) 엔진으로 남게 된다.

강력한 규격을 통해 분리된 이 3개의 컴포넌트는 단방향이 아닌 양방향 비동기 메시지 버스(Message Bus) 위에서 끊임없이 핑퐁(Ping-Pong)을 교환하며 교향악 같은 상호작용을 유도해 낸다. 이어지는 절들에서는 각 계층 컴포넌트가 담당하는 독자적인 로직과 생명주기(Lifecycle) 은닉 기술을 낱낱이 분해하여 설명한다.
