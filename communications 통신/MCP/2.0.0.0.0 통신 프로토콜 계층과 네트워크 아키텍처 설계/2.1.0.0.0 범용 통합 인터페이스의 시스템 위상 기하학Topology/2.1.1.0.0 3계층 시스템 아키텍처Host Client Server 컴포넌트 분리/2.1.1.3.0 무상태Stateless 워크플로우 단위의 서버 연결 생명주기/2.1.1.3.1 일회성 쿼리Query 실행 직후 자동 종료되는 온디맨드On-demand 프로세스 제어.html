2.1.1.3.1. 일회성 쿼리(Query) 실행 직후 자동 종료되는 온디맨드(On-demand) 프로세스 제어

MCP 서버망을 구성하는 주된 전송 계층 중 하나는 호스트 시스템과 직접적인 파이프망을 수립하는 표준 입출력(Stdio) 방식이다. 이때 MCP 서버는 Node.js 혹은 Python 등의 독립적인 자식 프로세스(Child Process) 형태로 메모리에 상주하게 된다. 그러나 도구 사용 빈도가 상대적으로 낮은 로컬망에서 이러한 프로세스들을 무기한으로 대기 상태로 두는 것은 전력 및 자원 소모의 낭비를 불러온다.

이 한계를 타파하기 위해 적용된 것이 바로 온디맨드(On-demand) 기반의 프로세스 제어 기법이다.

0.1 npx, uvx 등 패키지 런너 통제

에이전트가 특정 도구(예: 타 로컬 저장소 내부의 PDF 텍스트 추출 도구)를 실행할 목표가 생겼을 때, MCP 클라이언트는 전역으로 서버 프로세스를 미리 구동 시켜 놓지 않는다. 그 대신 npx (Node Package Execute) 혹은 uvx (Python Package Execute) 와 같은 일회성 실행 환경 런너를 통해 쿼리가 발동하는 그 찰나의 순간에 패키지를 다운로드하고 구동시킨다.

에이전트가 단 일회성 쿼리(“test.pdf 파일 내용 요약해 줘”)를 발송하여 JSON-RPC 직렬화된 응답을 도출받고 파싱을 끝내면, 클라이언트 백그라운드 워커는 해당 서버 자식 프로세스에 전원이 꺼지듯 SIGTERM 혹은 부모 프로세스 파이프 단절 시그널을 즉각 전송한다.

결과적으로 이 서버 프로세스는 파일의 내용을 클라이언트에게 돌려준 그 즉시, 운영체제(OS)의 가상 메모리상에서 자동 포기되고 소멸한다. 어떠한 전역 연결 풀에 잔존하여 쓸데없는 CPU 리소스를 갉아먹는 좀비(Zombie) 프로세스나 고아(Orphan) 프로세스로 남지 않는 것이다. 이 극단적인 메모리 반환 구조 덕분에 사용자의 호스트 에디터(IDE)는 무수히 많은 도구 플러그인을 설치하더라도 램(RAM) 용량 초과(OOM Killer 발동)라는 재앙 없이 쾌적한 런타임을 영위할 수 있다.
