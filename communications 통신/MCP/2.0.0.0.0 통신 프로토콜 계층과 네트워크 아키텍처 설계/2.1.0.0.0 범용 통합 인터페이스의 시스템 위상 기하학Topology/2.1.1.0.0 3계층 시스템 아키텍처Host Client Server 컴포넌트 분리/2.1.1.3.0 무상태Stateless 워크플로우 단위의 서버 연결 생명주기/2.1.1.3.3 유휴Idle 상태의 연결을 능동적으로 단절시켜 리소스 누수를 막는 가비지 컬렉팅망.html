2.1.1.3.3. 유휴(Idle) 상태의 연결을 능동적으로 단절시켜 리소스 누수를 막는 가비지 컬렉팅망

상태 유지(Statefulness)를 최대한 지양한다 하더라도, 1:N 토폴로지 체제하에서 빈번히 호출되는 핵심 연결망(예: 사내 핵심 데이터베이스 등)의 경우 일시적인 연결 유지가 필요할 때가 존재한다. 하지만 사용자가 자리를 비우거나, 다른 작업(도메인)으로 포커스가 전환된 시간 동안 여전히 클라이언트가 이 연결 파이프라인을 점유하고 있다면 치명적인 리소스 누수(Resource Leak)가 발생한다.

이러한 문제를 제어하기 위해 클라이언트 계층에는 사용되지 않는 세션을 추적하고 자동 소거하는 능동형 가비지 컬렉팅(Garbage Collecting) 알고리즘이 상주한다.

0.1 마이크로 단위 타임아웃(Timeout) 해제 타이머

클라이언트의 백그라운드 이벤트 데몬(Daemon)은 각 자원(Resource) 호출 및 도구(Tool) 실행 트랜잭션의 마지막 발생 시점 타임스탬프(Timestamp)를 기록한다. 만약 어떠한 추가적인 JSON-RPC 푸시(Push)나 클라이언트 핑(Ping) 작업이 규정된 임계 임시 대기시간(예: 5분~10분)을 돌파하게 되면, 파이프라인은 즉각 유휴(Idle) 상태로 평가된다.

유휴 판정 즉시 가비지 컬렉터는 서버 측에 우아한 세션 종료(Graceful Teardown) 시그널링을 던지고 네트워크 I/O 할당을 포기시킨다.
만약 서버 측에 비동기적으로 대기 걸린 긴 큐가 존재했더라도, 클라이언트 측은 가차없이 연결망 스레드를 수거해 감으로써 호스트 환경의 좀비(Zombie) 메모리 누수를 원천 봉쇄한다.

0.2 투명한(Transparent) 생명주기 재생망

이 무자비한 연결 스윕(Sweep) 과정이 자율주행 에이전트의 작업 수행에 타격을 주지 않는 이유는, MCP 프로토콜이 지원하는 투명한 재연결(Transparent Re-connection) 프레임워크에 기인한다.
연결이 컬렉팅되어 소멸된 직후 사용자가 동일한 서버에 재요청을 인계할 경우, 클라이언트는 서버가 처음부터 계속 켜져 있던 척 0초에 가까운 속도로 새로운 온디맨드 샌드박스를 띄워 쿼리를 마법처럼 스무스하게 이어가게 된다.

개발자나 사용자는 언제 연결이 파괴되고 언제 소켓이 재생되었는지 결코 눈치챌 수 없다. 오직 시스템 내부망만이 불필요한 트래픽의 중단과 재생을 능동적으로 통치함으로써 시스템 전체 아키텍처의 부하 임계치를 안전하게 조율해 낼 뿐이다.
