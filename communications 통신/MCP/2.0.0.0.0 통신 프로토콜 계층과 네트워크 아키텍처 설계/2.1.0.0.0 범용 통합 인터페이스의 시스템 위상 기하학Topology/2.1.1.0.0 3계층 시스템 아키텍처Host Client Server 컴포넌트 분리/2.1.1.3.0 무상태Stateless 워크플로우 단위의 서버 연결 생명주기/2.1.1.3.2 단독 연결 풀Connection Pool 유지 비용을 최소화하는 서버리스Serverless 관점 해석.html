2.1.1.3.2. 단독 연결 풀(Connection Pool) 유지 비용을 최소화하는 서버리스(Serverless) 관점 해석

MCP 프로토콜 상에서 클라이언트가 원격(Remote)에 위치한 수십 개의 데이터 제공 서버들과 통신망을 꾸릴 때, 단방향 SSE(Server-Sent Events) 및 양방향 HTTP 통신 기술이 주로 활용된다. 하지만 클라우드 내에 상주하는 이 서버들이 각 클라이언트망의 접속 소켓 상태 정보를 장기적으로 저장하고 있는 스테이트풀(Stateful) 모델이라면, 가용성 초과 상황에서 서버의 로드 밸런서(Load Balancer)는 임계점(Bottleneck)에 달하게 될 것이다.

이를 우회하고자 MCP 아키텍처는 서버리스(Serverless) 컴퓨팅 패러다임의 특성을 네트워크 전송 계층 설계로 체화시켰다.

0.1 연결의 무상태성(Statelessness)

MCP 클라이언트가 HTTP POST를 이용하여 JSON-RPC 도구 호출 쿼리를 전송할 때, 그 요청을 처리하는 서버가 꼭 AWS EC2처럼 상시 구동되어 있는 덩치 큰 인스턴스일 필요는 없다. AWS Lambda, Google Cloud Functions, Cloudflare Workers와 같이 요청이 들어오는 그 순간에 컨테이너가 스핀업(Spin-up) 되어 찰나의 연산만 수행하고 파괴되는 완벽한 온디맨드 서버리스 함수들이 단독 연결의 훌륭한 엔드포인트(Endpoint)가 될 수 있다.

이 모델에서 서버는 특정 클라이언트가 앞선 트랜잭션에서 어떤 질의를 남겼는지 ’기억’할 필요가 전혀 없다. 클라이언트는 각각의 콜백(Callback) 호출마다 프롬프트 해석에 필요한 전체적인 컨텍스트 토큰을 메타데이터와 패키지화하여 전달하고, 서버리스 백엔드는 단순히 인풋 대비 결과값(Output)만을 조립해 반환한다.

이로 인해 서버 관리자는 동시 접속자 수천 명이 유입되어도, 소켓 자원 풀(Pool) 고갈을 걱정할 필요 없이 클라우드 벤더의 본원적 자동 수평 확장(Auto-scaling)만으로 모든 트래픽 부하를 분산 제거할 수 있게 된다.
즉, 시스템 유지 보수 인프라의 고비용 구조를 탈피하여 완전 관리형(Managed) 비용 최소화 모델을 가능하게 하는 것이 MCP 프로토콜이 투사한 서버리스 상태 파괴 패러다임의 이면이다.
