2.1.1.1.2. 샌드박싱(Sandboxing)된 서버 구동 환경 프로비저닝 및 승인(Approval) 팝업 통제망

MCP 호스트(Host) 시스템의 시각적 렌더링만큼이나, 혹은 그 이상으로 핵심적인 역할은 에이전트 하위 프로세스들의 무분별한 팽창과 자원 침해를 차단하는 **운영체제 레벨의 샌드박싱(Sandboxing)**과 승인(Approval) 팝업 통제망 체계이다.

MCP 아키텍처 환경에서는 에이전트 시스템이 사용자를 대신하여 무수히 많은 자식 프로세스(Child Process) 기반의 로컬 서버를 구동(Provisioning)시킨다. 이때 호스트 시스템은 무작정 이들의 실행을 허가하는 것이 아니라, 보안 컨테이너 혹은 파일 권한 제어가 결합된 강력한 샌드박스 장벽 안에서만 프로세스를 구동하도록 강제하는 지휘관 역할을 수행한다.

0.1 샌드박싱(Sandboxing) 기반 프로비저닝

호스트 시스템은 새로운 MCP 서버 인스턴스가 생성될 때, 해당 서버가 호스트 운영체제의 핵심 시스템 콜(System Call)을 터치하지 못하도록 철저하게 권한을 제한한다. 예를 들어, node나 python 명령어를 통해 실행되는 서버 프로세스들은 오직 특정 워크스페이스(Workspace) 경로 하위에만 파일 읽기와 쓰기가 허용되며, 사용자의 데스크톱이나 루트(Root) 디렉터리로의 디렉터리 탐색(Path Traversal) 시도는 호스트가 사전에 마운트한 가상 환경벽에 막혀 즉각 소멸된다. 이는 악의적으로 조작된 서버 패킷이나 프롬프트 인젝션(Prompt Injection)으로 인한 로컬 시스템 감염을 원천 차단하는 제 1차 방어선이다.

0.2 파괴적 행동에 대한 승인(Approval) 통제망

그럼에도 불구하고 지정된 워크스페이스 내에서 파일을 삭제(Delete)하거나, 외부 데이터베이스 스키마를 떨어뜨리는(Drop) 등 ’파괴적 성격(Destructive)’의 도구(Tool) 실행 시도가 발생할 수 있다.
호스트 시스템은 에이전트가 통신망으로 쏘아올린 이러한 의도적 부작용(Side-effect)성 쿼리 패킷을 가로챈 뒤 통신을 일시 동결(Freeze)시킨다. 그리고는 사용자 화면 정중앙에 “도구 승인 대기 중(Tool Approval Pending)“이라는 강력한 모달(Modal) 팝업망을 강제로 띄운다.

  • 패킷이 전달하고자 하는 도구의 원본 이름(Name)
  • 실행될 파라미터(Arguments) 값
  • 변경 전/후 위험도 시각화

호스트 시스템은 위 정보들을 사용자에게 정교하게 직렬화하여 표출한 후, 인간 작업자가 “승인(Accept)” 버튼을 클릭해야만 비로소 클라이언트 하위 계층에 실행 허가 시그널을 전개한다. 즉, 기계의 자율적 쾌속 주행을 유지하면서도 가장 비가역적인 행위 앞에서는 반드시 인간의 명시적 서명(Signature)을 요구하는 에스크로(Escrow) 관제 구조가 바로 호스트 시스템에 내장된 통제망의 정수이다.
