2.1.1.1. 사용자 인터페이스 로직을 담당하는 MCP 호스트(Host) 시스템

3계층(3-Tier) 아키텍처의 맨 위 꼭대기, 즉 인간 사용자와 가장 최전선에서 호흡하는 관문을 MCP 명세 체계에서는 ‘호스트(Host)’ 컴포넌트로 명명한다. 호스트 시스템은 에이전트 소프트웨어가 런타임에 기동하는 근원적인 ‘실행 모체’ 역할을 겸하며, 통상적으로 우리가 눈으로 바라보는 독립적인 애플리케이션(통합 개발 환경 IDE 창, 터미널 구동 셸, 혹은 웹 브라우저 기반의 AI 챗봇 인터페이스 등) 자체를 의미한다.

소프트웨어 공학의 단일 책임 원칙(SRP, Single Responsibility Principle)의 관점에서, MCP의 호스트 시스템은 백엔드 통신(Client)이나 외부 지식 처리(Server)의 책임 비중을 극단적으로 낮춘(Lean) 대신, 두 가지 고유한 본질적 책임에 모든 자원을 투과시킨다.

첫째는 시각 정보의 재구성(Visual Re-rendering)이며, 둘째는 뚫려서는 안 될 최후의 권한 통제선(Authorization Boundary) 역할이다.

MCP 파이프라인 깊숙한 곳에서 기계 친화적(Machine-centric)인 JSON-RPC 메시지들이 아무리 요동치며 교환되더라도, 사용자는 결코 중괄호({, })로 뒤덮인 날것(Raw)의 데이터를 마주쳐서는 안 된다. 호스트 시스템은 이 백그라운드의 숨 가쁜 데이터 통신망 위에 유려한 프레젠테이션 계층(Presentation Layer)의 장막을 덮어씌운다.

더불어, 하위 계층에서 구동되는 에이전트나 통신 클라이언트(Client) 소프트웨어가 백그라운드 환경에서 폭주(Runaway)하여 의도치 않게 운영체제의 핵심 폴더를 날려버리거나, 사내 깃허브 망에 악의적인 푸시(Push)를 발생시킬 위험이 상존한다. 이때 호스트 시스템은 이 모든 도구 호출 명령 패킷을 렌더링 계층에서 강제로 가로채어, 인간 작업자(Human-in-the-loop)의 최종 시각적 서명(승인 팝업 클릭)을 인가받아 내는 문지기 책임을 수행해야만 한다.

이어지는 세부 항목에서는 이 호스트(Host) 시스템 컴포넌트가 대화형 UI를 어떻게 동적으로 구축하고 프레임워크화 하는지, 그리고 운영체제 환경의 샌드박스 밖으로 탈출하려는 에이전트 인프라의 위협을 어떤 팝업(Approval) 메커니즘을 통해 능동적으로 차단하고 파싱하는지 해부할 것이다.
