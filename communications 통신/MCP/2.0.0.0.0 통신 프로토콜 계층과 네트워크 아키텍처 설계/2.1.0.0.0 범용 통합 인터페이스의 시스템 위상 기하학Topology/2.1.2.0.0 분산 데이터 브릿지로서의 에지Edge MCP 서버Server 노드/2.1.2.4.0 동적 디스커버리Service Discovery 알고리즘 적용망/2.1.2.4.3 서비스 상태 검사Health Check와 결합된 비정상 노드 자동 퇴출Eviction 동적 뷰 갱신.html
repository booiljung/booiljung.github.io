2.1.2.4.3. 서비스 상태 검사(Health Check)와 결합된 비정상 노드 자동 퇴출(Eviction) 동적 뷰 갱신

네트워크 토폴로지의 생명 주기를 논함에 있어 ’추가(Discovery)’만큼이나, 어쩌면 그 이상으로 더욱 치명적인 기술적 도전은 ’사멸(Death)된 자원의 파괴’를 인지하고 대처하는 논리 회로이다.
OOM(Out of Memory)으로 인해 내부 스레드가 정지된 좀비 컨테이너 노드, 혹은 물리적 스위치 장애로 끊어진 블랙홀 상태의 서버 노드로 MCP 프록시가 계속해서 LLM의 도구 호출 인텐트(Intent) 패킷을 밀어 넣는다면 연쇄적인 시스템 타임아웃 붕괴 현상이 필연적으로 뒤따른다.

이러한 인프라 흑사병의 전염을 막아내기 위해 MCP 트래픽 라우터는 고강도의 능동적, 수동적 서비스 상태 검사(Health Check)와 연계된 비정상 노드 자동 퇴출(Auto-Eviction) 알고리즘을 상시 가동한다.

0.1 능동형 프로빙(Active Probing)과 수동형 래치(Passive Latch)

가장 최전선에 서 있는 프록시는 각 엔드포인트 노드를 향해 주기적으로 초경량 핑(Ping) 패킷 혹은 JSON-RPC의 mcp.health 전용 규격 통신을 발사(Active Probing)한다. 만일 서버가 3연속(Threshold) 침묵하거나 500 계열의 내부 결함 에러 코드를 토해내면 프록시는 해당 엔드포인트에 즉각 ‘오염(Tainted)’ 낙인을 찍는다.

동시에 프록시는 실제 도구 호출 과정 중에 발생하는 스트림 응답 메타데이터의 레이턴시(Latency) 분포까지 백그라운드에서 실시간 집계(Passive Latch)한다. 핑은 정상적으로 통과하더라도 특정 노드만 트래픽 반환이 5초 이상 늘어진다면, 이 또한 좀비화 전조 현상으로 간주하여 동일한 논리적 살처분 절차를 밟는다.

0.2 카탈로그 동적 뷰 갱신(Dynamic View Refresh)과 라우팅 차단

특정 서버 노드가 최종적으로 오염 판촉 기준을 초과하면 (Eviction 도달), 에그리게이터나 클라이언트의 중앙 연결 관리자는 그 즉시 메모리 상의 라우팅 배열에서 이 IP를 도려낸다(Slicing).
이러한 퇴출 절차는 단지 해당 노드로의 트래픽을 단절하는 방어적 역할에만 그치지 않는다. MCP 관리자는 해당 노드가 제공한다고 선언했었던 수많은 기능들(Tools/Resources)의 목록마저 글로벌 추상 카탈로그에서 실시간으로 박탈, 갱신(Dynamic View Refresh)시켜버린다.

결과적으로, 연결되어 있던 LLM 에이전트 측은 자신이 구사할 수 있었던 특정 분석 도구 컨텍스트가 런타임에 갑자기 소멸하는 현상을 이벤트 알람으로 즉시 통보받게 되며, 죽어있는 서버 노드에게 파멸적인 헛스윙(Dead Call)을 날리는 비극적 상황에서 완벽하게 구조받게 된다.
