2.1.2.4.2. 클라우드 오케스트레이터(K8s) 내에서 신규 Pod 단위로 스케일아웃되는 서버의 엔드포인트 동기화

퍼블릭 클라우드의 무한한 자원 풀로 눈을 돌리면 멀티캐스트 기반의 DNS-SD 방식은 더 이상 생존할 수 없다. 분산된 서브넷 구조와 엄격한 라우팅 정책이 지배하는 세계에서, 멀티캐스트 패킷은 첫 번째 발견된 라우터를 관통하지 못하고 즉각 폐기되기 때문이다. 더군다나 초당만 번의 도구 호출 부하를 견디기 위해 하나의 MCP 서버 컨테이너가 수십 개의 클론(Clone) 인스턴스로 자동 스케일 아웃(Scale-out)되는 상황에서는 보다 거시적인 체인지스트림(Change-Stream) 구독 시스템이 요청된다.

이를 해결하기 위해 MCP 에그리게이터(프록시)는 대규모 클라우드 환경에서 쿠버네티스(Kubernetes, K8s)와 같은 어플리케이션 오케스트레이터(Orchestrator)의 컨트롤 플레인(Control Plane) API와 직접 결합되는 엔드포인트 동기화 매커니즘을 가동한다.

0.1 컨트롤 플레인(Control Plane) API 감시 구독

에그리게이터 노드는 K8s API 서버를 향해, 특정 라벨(Label Selector, 예: app=mcp-vector-db-server)이 부착된 파드(Pod) 집합체들의 라이프사이클 이벤트를 영구 구독(Watch)하는 훅(Hook) 소켓을 열어둔다.
수만 명의 에이전트 동시 접속으로 CPU 임계치가 폭발하여 K8s 오토스케이러(HPA)가 신규 Pod 5개를 클러스터망에 추가로 프로비저닝(Provisioning)하는 즉시, API 서버는 에그리게이터에게 이 새로운 5개의 내부 아이피(Private IP) 명단을 실시간 델타 푸시(Delta Push)로 전송한다.

0.2 라우팅 테이블 무중단 열반사 갱신

통지를 접수한 에그리게이터의 디스패처 메모리망 상에서는 어떠한 프록시 재기동(Restart) 절차나 트래픽 손실 없이도 즉시 가용한 백엔드 라우팅 테이블 배열(Array) 슬롯이 확장된다.
기존에는 2대의 백엔드를 향해 돌고 있던 라운드 로빈(Round-Robin) 로드 밸런서 지침서가 순식간에 7대로 재편성되며, 에이전트 클라이언트들은 어떠한 설정 변경이나 찰나의 순단(Downtime)도 눈치채지 못한 채 스케일아웃된 광활한 파벌 연산망의 혜택을 온전히 빨아들이게 되는 것이다.
