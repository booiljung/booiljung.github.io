2.1.2.3.1. 클라이언트가 다수의 서버와 직접 통신하는 대신, 중간 프록시 서버에서 인터페이스를 단일화

MCP 프로토콜을 도입하는 생태계 내에서 거대 언어 모델(LLM)을 포함한 호스트 클라이언트는 런타임에 최대한 가벼운(Lean) 상태를 유지해야 할 의무가 있다. 그러나 클라이언트가 50개의 개별 도구 제공 서버와 직접 HTTP/SSE 연결을 맺고 유지(Keep-Alive)하려면, 세션 관리 모듈과 네트워크 할당 자원(Socket Descriptor)이 폭발적으로 소모된다.

0.1 단일 진입점(Single Entry Point) 구조의 형상화

이를 방어하기 위해 도입된 중간 프록시(Proxy) 서버는 이 50개의 개별 서버 리스트(Tools, Resources, Prompts)를 백그라운드망에서 미리 모두 긁어모아(Fetch), 하나의 거대한 종합 카탈로그(Catalog) 객체로 평탄화(Flattening)한다.

클라이언트 입장에서 보자면, 외부 세상에는 단 한 대의 **‘만능 MCP 서버(프록시)’**만이 존재하는 것처럼 착각(Illusion)을 일으킨다. 클라이언트는 이 프록시 엔드포인트와 단 하나의 직통 파이프라인(소켓)만을 개설하여 최초 1회의 핸드셰이크(Handshake) 프로토콜을 수행한다.

결과적으로 “슬랙 메시지 전송(Server A)“과 “깃허브 PR 병합(Server B)“이라는 이질적인 목적의 두 도구 통신이 모두 동일한 단일 소켓 터널을 타고 흘러가게 된다.

0.2 이름 충돌(Naming Collision) 방지 및 네임스페이스 격리망

단일 프록시가 다중 서버의 스펙을 병합할 때 필연적으로 겪는 문제는 도구 이름의 충돌이다. 예를 들어, 로컬 파일 서버 노드와 원격 클라우드 구글 드라이브 서버 노드가 모두 read_document 라는 기능 명세표를 보고했다면 이를 병합할 때 충돌(Collision)이 발생한다.

프록시 라우터는 이러한 충돌을 막기 위해 각 타겟 서버 노드별로 고유한 접두사(Prefix) 기반의 가상 네임스페이스(Namespace) 매핑 룰셋을 강제한다.
클라이언트에는 local_fs__read_document와 gdrive__read_document라는 고유한 구분 식별자로 통합 노출되며, 클라이언트가 특정 도구를 호출할 시 프록시 디스패처(Dispatcher)는 이 식별자의 접두사를 해석(Routing Regex)하여 원래의 목적지 백엔드 노드로 패킷을 정확히 유도(Tear-off)하는 브릿지 지휘관 역할을 수행한다.
