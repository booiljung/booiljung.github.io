2.1.2.3. 에그리게이터(Aggregator) 기반 프록시(Proxy) 병합 라우팅 체계

에이전트 시스템이 엔터프라이즈급으로 고도화될수록, 하나의 에이전트 클라이언트가 참조해야 할 MCP 서버 노드의 숫자는 십여 개에서 수백 개 단위로 폭증하게 된다. 로컬 시스템 파일 탐색용 서버, 클라우드 DB 연동 서버, 슬랙(Slack) 메시지 발송 서버, 코드 컴파일 샌드박스 서버 등 이질적인 수많은 기능들이 각기 독립된 서버 인스턴스로 파편화되어 구동되기 때문이다.

이러한 1:N(클라이언트:다중 서버) 토폴로지 환경에서 클라이언트가 직접 모든 서버와 커넥션을 맺고 핸드셰이킹(Handshake)을 유지하는 구조는 치명적인 병목 현상(Bottleneck)과 클라이언트의 메모리 초과(OOM Killer)를 초래한다.

이 한계를 타파하기 위해 고안된 것이 바로 에그리게이터(Aggregator) 기반 프록시(Proxy) 병합 라우팅 체계이다.
이는 클라이언트와 무수한 백엔드 서버들 사이에 강력한 미들웨어 계층(Middleware Layer)을 삽입하여 통신의 복잡도를 중간에서 완전히 은닉시키는 아키텍처 패턴이다.

에그리게이터는 클라이언트에게 단일화된 거대 진입점(Single Entry Point)을 제공하고, 뒤편에 숨은 이기종 백엔드망들을 향해 요청을 분산(Scatter)시키며 다시 응답을 병합(Gather)하는 허브 앤 스포크(Hub-and-Spoke) 구조의 심장부로 기능한다.
본 장에서는 이 프록시 계층이 가져다주는 인터페이스 단일화의 추상화 매커니즘, 횡단 관심사(Cross-cutting Concerns)인 캐싱(Caching) 모델, 그리고 응답 압축 및 배칭(Batching)을 통한 대역폭 보존망 설계 원칙을 심도 있게 탐구한다.
