2.1.2.2.3. 세션 종료 시나리오 대비 안전한 자원 해제(Teardown)를 규정하는 상태 전이 머신 설계

무상태(Stateless) 도구 실행과 달리, 상태를 물고 있는 자원(Resource) 구독이나 SSE(Server-Sent Events) 이벤트 펌핑 과정 중 클라이언트 시스템의 전원이 끊기거나 네트워크가 크래시(Crash)되는 단절 사고가 발생할 수 있다. 만일 분산 서버 노드들이 클라이언트의 단절을 영구히 인지하지 못하고 이벤트를 무한정 발송하거나 소켓 커서를 물고 대기한다면, 서버 인프라는 곧장 좀비(Zombie) 콜백 지옥에 빠져 리소스가 고갈될 것이다.

이를 근본적으로 통제하기 위해 MCP 서버 내부에는, 어떤 치명적인 시스템 단절 상황 하에서도 완벽하고 깨끗한 셧다운을 담보하는 **상태 전이 머신(State Transition Machine)**이 이식되어 있다.

0.1 핑-퐁(Ping-Pong) 하트비트를 통한 고아 상태 인지

서버망은 클라이언트의 생존 여부를 주기적으로 검증하기 위해 하트비트(Heartbeat) 핑(Ping) 패킷을 능동적으로 투사한다. 만약 N번 연속으로 퐁(Pong) 응답이 손실되거나 파이프(Stdout)에 브로큰 파이프(Broken Pipe) 시그널이 접수되면, 데몬 워커는 해당 세션을 즉시 ORPHANED(고아)라는 상태 트랜지션으로 전이시킨다.

0.2 유예 윈도우와 안전한 티어다운(Graceful Teardown)

ORPHANED 상태 전이 직후, 서버는 자원을 즉각 분쇄시키지 않고 미리 설정된 타임 TTL 유예 기간(Grace Period) 동안 보류망(Pending)에 가둔다. 모바일 통신처럼 잠시 터널을 통과하는 순간의 단절일 가능성을 배제할 수 없기 때문이다.
유예 기간 안에 재접속(Reconnect) 명령이 타진되면 서버는 상태를 다시 ACTIVE로 복권시킨다. 하지만 유예 시간의 카운트다운 초시계가 0에 수렴하여 만료되는 즉시 상태 엔진은 TEARDOWN(강제 파쇄) 단계를 호출한다.

이 파쇄 단계에서는 인메모리에 잔존했던 가상 시스템 읽기 커서 해제, 임시 스풀(Spool) 파일들 삭제, 자원 변경을 전파하기 위해 캐싱해 둔 델타 브로드캐스트 이벤트 구독자 명단에서 해당 세션 리스트를 영구 삭제하는 등, 클라이언트가 남겨둔 오염된 그림자를 완벽히 진공 청소하는 백룸 가비지 컬렉터의 우아한 마법이 실현된다.
