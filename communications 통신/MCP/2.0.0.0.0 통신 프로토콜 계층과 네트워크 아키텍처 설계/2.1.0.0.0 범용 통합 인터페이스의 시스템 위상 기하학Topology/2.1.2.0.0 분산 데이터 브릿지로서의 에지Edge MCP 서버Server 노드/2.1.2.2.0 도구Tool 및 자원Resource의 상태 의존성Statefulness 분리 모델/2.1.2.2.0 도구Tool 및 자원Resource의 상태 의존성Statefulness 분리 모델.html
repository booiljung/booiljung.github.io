2.1.2.2. 도구(Tool) 및 자원(Resource)의 상태 의존성(Statefulness) 분리 모델

서버 노드를 논할 때 가장 깊이 고민해야 하는 아키텍처적 주제 중 하나는, 분산된 프로세스들이 사용자와의 상호작용 속에서 발생한 ’기억(Memory, State)’을 어떻게 처리할 것인가에 대한 철학이다.
Model Context Protocol(MCP)은 시스템의 안정성과 확장성을 극대화하기 위해 제공 객체인 **도구(Tool)와 자원(Resource)**의 상태 의존성(Statefulness)을 명확하게 분리(Decoupling)하는 설계를 강제한다.

기본적으로 MCP에서 ’도구(Tool)’는 LLM이 외부 세계를 조작하거나 연산을 수행하기 위해 호출하는 일회성 액추에이터(Actuator)이며 철저한 무상태(Stateless)를 지향한다. 반면 ’자원(Resource)’은 데이터의 흐름과 업데이트 이력을 구독하는 장기적 센서(Sensor)로써 필연적으로 상대적인 상태성(Stateful)을 띠게 마련이다.

만약 서버 노드가 이 두 가지 객체의 상태 이력을 뒤섞어 관리하게 되면, 일시적인 네트워크 단절 시 도구의 실행 결과는 증발하고 자원의 업데이트 스트림은 꼬여버려 치명적인 컨텍스트 오염을 유발하게 된다. 따라서 서버 인스턴스 내부는 도구 호출을 담당하는 무상태 휘발성 파이프라인과, 자원 제공을 보장하는 영속적 상태 머신(State Machine) 영역으로 완전히 반반 쪼개어(Split) 진화하게 되었다.

이러한 상태 의존성의 분리 모델이 실전 네트워크 환경, 즉 가상 파일 시스템(VFS) 커서 유지, 단발성 부하 병렬 분산, 불안정한 세션 해제(Teardown) 대비와 같은 가혹한 조건에서 어떻게 작동하는지 본 절의 하위 문서들에서 체계적으로 접근하도록 한다.
