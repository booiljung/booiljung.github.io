2.1.2.2.1. 영속적 소켓(Socket) 연결을 통한 가상 파일 시스템(VFS) 읽기 커서 상태 유지 메커니즘

자원(Resource) 계층에서 다루어야 하는 가장 까다로운 데이터 형태 중 하나는, LLM 컨텍스트 한계를 훌쩍 초과하는 테라바이트(TB) 급의 서버 로그 파일이나 대규모 데이터베이스 덤프 파일이다.
이러한 거대 자원을 한 번의 JSON-RPC 통신으로 송수신하는 것은 불가능하다. 결국 클라이언트는 데이터를 잘게 쪼개어 점진적으로 읽어들이는 청크(Chunk) 단위의 순회 렌더링이 필수적인데, 이때 핵심이 되는 기술이 바로 가상 파일 시스템(Virtual File System, VFS) 읽기 커서(Cursor)의 상태 유지이다.

0.1 커서(Cursor) 기반의 분절적 스트리밍

MCP 서버는 특정 거대 파일 URI(mcp://server/logs/sys.log)에 대한 구독(Subscription) 요청을 수신하면 내부 메모리 공간 혹은 Redis와 같은 상태 맵핑 스토리지에 클라이언트 세션별 일련번호(Session ID)와 함께 ‘읽기 커서(Read Cursor)’ 객체를 인스턴스화한다.
이 커서 객체는 파일 시스템 내에서 클라이언트가 현재 몇 번째 바이트 풀(Byte pool)까지 읽어갔는지를 나타내는 정수 오프셋(Offset) 값을 생명줄처럼 쥐고 있는 심장부이다.

클라이언트가 다음 페이지네이션(Pagination) 요청을 전송하면, 서버는 유지되고 있는 소켓 인스턴스를 통해 이 읽기 커서를 전진시킨 뒤, 그 구간부터의 데이터만 추출하여 델타 페이로드(Delta Payload)로 반환한다.

0.2 영속적 소켓 핀홀링(Pin-holing) 유지 장치

커서 값이 증발하면 클라이언트는 이전에 읽은 데이터를 중복해서 전달받게 되어 트랜잭션의 치명적인 모순(Inconsistency)에 봉착한다. 이를 위해 클라이언트와 서버 간의 백그라운드망은 영속성이 보장되는 **HTTP Keep-Alive 통신이나 단방향 SSE 소켓을 끊기지 않도록 핀홀링(Pin-holing)**한다.

만일 TCP/IP 불안정성으로 세션이 끊어질지라도, 서버 측의 타임아웃 만료(TTL) 윈도우가 허용하는 유예 시간 내부라면 클라이언트는 기존 Session-ID를 제시하여 끊어진 파이프라인의 커서 오프셋부터 재할당받고, 아무 일도 없었다는 듯 거대 데이터의 스트리밍을 이어나가는 고탄성(High-resilience) 상태 복원력을 과시한다.
