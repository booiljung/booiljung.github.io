2.1.2.2.2. 무상태(Stateless) 도구 호출 시 발생하는 단발성 부하(Load) 병렬 분산 및 메모리 반환 정책

엄격한 세션 관리가 요구되는 자원(Resource)과 정반대로, 도구(Tool)의 실행 스펙트럼은 철저한 **무상태(Stateless)**를 추구해야만 한다. GitHub 레포지토리에 특정 커밋(Commit)을 푸시(Push)하거나, 원격 리눅스 시스템에 npm install 명령을 던지는 도구 호출은 그 트랜잭션이 끝남과 동시에 서버나 클라이언트 메모리에 어떠한 논리적 부채(Debt)도 남기지 않아야 한다.

0.1 병렬 작업자(Workers)를 통한 수평 연산 분산

LLM이 10개의 소스 코드 파일을 각각 요약하기 위해 10개의 도구를 일제히 동시에 호출(Parallel Tool Call)했다고 가정해보자. 도구 실행이 무상태(Stateless)적 특성을 지니기 때문에, 서버 노드 중앙 관리자(Dispatcher)는 이 10개의 요청이 순차적으로 이전 상태를 열람할 필요가 없음을 확신할 수 있다.

그 결과, 서버는 각 요청을 독립된 10개의 파이썬/노드 자식 프로세스 버퍼나 10개의 클라우드 람다(Lambda) 컨테이너로 아무런 컨텍스트 공유 락(Lock) 없이 통째로 흩뿌려버릴 수 있다. 무상태성은 곧 완벽한 **수평적 스케일 아웃(Horizontal Scale-out)**을 달성하게 해주는 황금열쇠이며, 시스템의 연산 단발성 부하(Spike Load)를 무한의 가용성 안에서 병렬적으로 소화하도록 허락한다.

0.2 제로 코스트(Zero-Cost) 메모리 반환 가비지 룰셋

단발성 부하 분산 처리 도중, 도구 호출 함수가 성공적으로 결과 문자열을 반환하거나, 혹은 타임아웃(Timeout) 붕괴로 크래시를 발생시켰다면, MCP 서버 계층은 그 즉시 해당 구동 런타임(컨테이너 혹은 서브 스레드)을 운영체제로부터 박탈시켜버린다.

무상태 구조 하에서는 “이전 도구 호출에서 할당된 특정 환경 변수“를 보존해야 할 의무조차 존재하지 않으므로, 복잡하고 지연시간이 심한 GC(가비지 컬렉팅) 루틴을 돌릴 이유조차 없다. 단순히 워커 프로세스의 SIGKILL 시그널 전파를 통해 운영체제가 통째로 메모리 주소를 포기(Drop)하게 만드는 ‘제로 코스트 메모리 파쇄망’ 체제가 가동되는 것이다.

이는 결국 방대한 플러그인을 무한대로 조작하는 에이전트 시스템이 메모리 오버플로우라는 파멸을 평생 겪지 않고 영구 런타임을 유지할 수 있게 만들어주는 궁극의 방어기제로 작동한다.
