2.4.2.4.1. 클라우드 벤더 제약으로 SSE 수신이 불가능할 때 순수 폴링(Polling) 방식으로 우회하는 분기점

클라이언트 에이전트가 GET /mcp/events 요청을 날린 직후 응답 헤더를 대기하지만, 5초~10초를 넘어도 스트림 껍데기는커녕 아무런 바이트조차 내려오지 않는다면, 에이전트 커널은 즉각 ’SSE 호환 불가망(Non-Streaming Network)’에 갇혔음을 선언해야 한다.

0.1 스트리밍에서 동기식 큐 추출로의 패러다임 전환

이 분기점을 넘어선 에이전트는 기존의 스트리밍 철학을 전면 백지화하고, 철저히 클라이언트 주도형인 순수 폴링(Polling) 아키텍처로 채널을 격하(Downgrade)시킨다.

  1. 폴링 엔드포인트 개시: 서버는 스트리밍이 막힌 클라이언트를 위해 별도의 대안 경로(GET /mcp/messages?sessionId=xxx)를 열어둔다. 클라이언트는 이제 이 주소를 향해 무뚝뚝한 단발성 GET 요청을 지속적으로 날린다.
  2. 이벤트 배열 반환: 서버의 폴링 핸들러 엔진은 자신이 들고 있는 해당 세션의 데드레터 큐(DLQ)나 인메모리 버퍼에 쌓여있던 모든 비동기 이벤트(JSON-RPC 알림들)를 쓸어 담아, 하나의 묵직한 JSON 배열([Message, Message, ...]) 덩어리로 직렬화하여 클라이언트에게 평범한 HTTP 200 응답으로 뱉어낸다.

이 우회로를 통해 서버 측은 스트리밍을 위한 영구 소켓을 유지하지 않아도 되고, 클라이언트는 방화벽의 스트리밍 차단 정책을 완전히 농락하며 필요한 JSON-RPC 이벤트를 동기적으로 모조리 수거(Fetch)해 나갈 수 있다.
