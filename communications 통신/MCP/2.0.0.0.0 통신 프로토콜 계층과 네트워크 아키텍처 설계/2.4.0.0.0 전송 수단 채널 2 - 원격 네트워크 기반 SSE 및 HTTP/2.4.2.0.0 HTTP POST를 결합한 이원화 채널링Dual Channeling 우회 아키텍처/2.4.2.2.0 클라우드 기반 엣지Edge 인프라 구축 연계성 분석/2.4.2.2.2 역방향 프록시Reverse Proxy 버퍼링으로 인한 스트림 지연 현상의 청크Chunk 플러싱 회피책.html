2.4.2.2.2. 역방향 프록시(Reverse Proxy) 버퍼링으로 인한 스트림 지연 현상의 청크(Chunk) 플러싱 회피책

클라우드 엣지망(예: Cloudflare, AWS CloudFront)을 경유하여 서버리스(Serverless) 워커에서 SSE 이벤트를 방출하려 할 때, 가장 악명 높은 장애물은 역방향 프록시의 강제 버퍼링(Buffering) 정책이다. 엣지 프록시들은 네트워크 최적화를 명목으로 람다(Lambda)나 워커가 뱉어내는 응답 조각들을 최소 4KB ~ 8MB가 쌓일 때까지 절대 클라이언트에게 넘겨주지 않고 메모리에 움켜쥐고 있다.

0.1 강제 플러싱(Flushing)과 특수 널(Null) 패딩

이 잔인한 지연 현상(Latency Spike)을 타파하기 위해, 서버측 프로그래밍 논리에는 청크(Chunk) 단위를 강제로 짓이기는 전투적 회피책이 삽입되어야 한다.

  1. X-Accel-Buffering: no 무력화 시대: Nginx 시절 유효했던 저 헤더는 현대의 매니지드 클라우드 프록시에선 보안상의 이유로 종종 묵살된다. 의존해서는 안 된다.
  2. 트랜스퍼 청킹(Transfer-Encoding: chunked)의 활용: 서버는 응답을 한 덩어리로 보내지 않고, 이벤트가 발생할 때마다 독립된 바이트 청크로 직렬화해야 한다.
  3. 가비지(Garbage) 바이트 패딩의 투입: 가장 확실한 물리적 돌파구는 가짜 트래픽(Padding)의 투하다. 실질적인 이벤트 페이로드가 100바이트에 불과하더라도, 서버는 프록시의 버퍼 강제 탈락 한계선(예: 4096 바이트)을 맞추기 위해 무의미한 SSE 코멘트 주석(: padding padding...)을 수 킬로바이트씩 덧붙여서 전송한다.

결국 프록시는 버퍼가 넘쳐나는 것을 감지하고 억지로 청크를 클라이언트에게 토해내며(Flushing), 이 거친 인위적 플러싱 공작을 통해 엣지 인프라에서도 극저지연(Ultra-low latency) 실시간 알림망이 사수된다.
