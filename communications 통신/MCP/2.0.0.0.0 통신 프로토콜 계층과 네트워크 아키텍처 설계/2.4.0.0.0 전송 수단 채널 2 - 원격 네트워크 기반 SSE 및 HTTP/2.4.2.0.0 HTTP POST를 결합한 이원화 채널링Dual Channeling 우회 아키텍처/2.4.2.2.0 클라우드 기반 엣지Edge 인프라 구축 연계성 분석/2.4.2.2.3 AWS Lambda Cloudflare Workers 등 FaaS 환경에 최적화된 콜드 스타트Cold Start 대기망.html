2.4.2.2.3. AWS Lambda, Cloudflare Workers 등 FaaS 환경에 최적화된 콜드 스타트(Cold Start) 대기망

에이전트가 아무도 사용하지 않던 휑한 심야 시간에 원격 서버로 HTTP POST 명령어(도구 호출 요청)를 찔러 넣는 순간, 잠들어 있던 AWS Lambda나 Cloudflare Workers 컨테이너가 새롭게 메모리에 적재되며 깨어나는 콜드 스타트(Cold Start) 현상이 발생한다.

이 콜드 스타트로 인해 서버의 첫 응답은 길게는 3~5초까지 끔찍하게 지연된다. 사용자 UI 관점에서는 프로토콜이 다운(Down)된 것 같은 착각을 불러일으킬 수 있는 매우 민감한 허들이다.

0.1 폴백 UI와 옵티미스틱 대기(Optimistic Waiting) 윈도우

이 인프라적 딜레이를 클라이언트 애플리케이션 계층에서 부드럽게 윤활(Lubricate)시키기 위해 에이전트 라우터는 다음과 같은 대기망 로직을 수반한다.

  1. 사전 준비망(Warm-up Ping): 에이전트는 사용자가 프롬프트 입력창에 텍스트를 입력하기 시작하는(Typing) 그 찰나의 순간, 실제로 명령을 던지지도 않았으면서 원격 FaaS 노드를 향해 아주 가벼운 옵션(OPTIONS) 요청이나 더미 Ping을 쏜다. 사용자가 엔터(Enter)를 누르기 전에 컨테이너를 미리 데워버리는(Warm-up) 선제적 은폐 타격이다.
  2. 낙관적 UI 표출(Optimistic UI): 실제 명령(POST)이 발사되었고 그것이 콜드 스타트에 걸렸음을 대비하여, 클라이언트는 서버의 ‘202 Accepted’ 응답을 받기도 전에 “서버 깨우는 중(Waking up remote server…)“이라는 인지 심리학적 방어 UI를 띄운다.
  3. 탄력적 타임아웃(Elastic Timeout): 단발성 POST 명령의 HTTP 응답 타임아웃을 통상적인 5초가 아닌 15초 이상의 ’콜드 스타트 전용 윈도우’로 대폭 늘려 잡는다. 이를 통해 서버리스 인프라가 부팅되는 그 거친 파동 구간 내에서 프로토콜이 허무하게 예외(Exception) 처리되어 죽어버리는 급사를 방지한다.
