2.4.2.2.1. 서버리스(Serverless) 함수 호출 제한 시간(Timeout) 극복을 위한 콜백(Callback) 지연 결합 구조

AWS Lambda와 같은 전형적인 FaaS(Function-as-a-Service) 환경은 하나의 API Gateway 요청당 최대 30초 혹은 15분의 엄격한 **실행 제한 시간(Timeout)**을 강제한다. 만약 에이전트가 POST /mcp/messages를 통해 넘긴 데이터 분석 작업이 40분이 걸리는 초거대 모델 추론이라면, 함수를 붙잡고 동기적으로 기다리는 방식은 필연적으로 504 Gateway Timeout 파멸을 맞이하게 된다.

0.1 백그라운드 워커 이양과 비동기 콜백 체인

이 제한선 속에서 이원화 채널링이 가진 유연함이 빛을 발휘한다. 상향(POST)과 하향(SSE) 채널이 물리적으로 끊어져(Decoupled) 있기 때문에, 엣지 함수는 본질적으로 작업을 “끝까지” 수행할 의무가 없다.

  1. 큐잉(Queuing)과 즉각 응답: 클라이언트가 POST 명령을 날리면, 타겟을 맞고 깨어난 엣지 함수는 요청 파라미터를 SQS(Simple Queue Service)나 Kafka 같은 메시지 브로커로 조용히 토스(Toss)하고, 즉시 HTTP 202 Accepted 응답을 쿨하게 반환하며 자신의 수명을 종료한다.
  2. 워커 실행 및 콜백: 실제 무거운 연산은 타임아웃 제약이 없는 별도의 백그라운드 워커 컨테이너(예: AWS ECS)가 큐를 스캐닝하여 수행한다.
  3. 지연 결합(Lazy Coupling) 응답: 워커는 연산이 끝나는 시점에, 앞서 파싱해 둔 클라이언트의 Session ID를 활용하여 원격 SSE 스트림 관리 노드(또는 Redis Pub/Sub)에 결과값을 콜백(Callback) 형태로 쏘아 보낸다.

결국 1초 만에 종료되는 단발성 엣지 함수 트리거와 무한정 열려있는 SSE 스트림이라는 비대칭 구조가 매끄럽게 맞물리며, 타임아웃이라는 인프라의 족쇄를 완전히 무력화시킨다.
