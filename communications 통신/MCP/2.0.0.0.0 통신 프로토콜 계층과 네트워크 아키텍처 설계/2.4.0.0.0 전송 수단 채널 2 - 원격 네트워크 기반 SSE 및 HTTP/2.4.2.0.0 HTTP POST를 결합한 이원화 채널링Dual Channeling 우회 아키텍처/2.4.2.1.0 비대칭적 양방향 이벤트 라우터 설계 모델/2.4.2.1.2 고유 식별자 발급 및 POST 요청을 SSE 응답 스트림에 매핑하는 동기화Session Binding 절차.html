2.4.2.1.2. 고유 식별자 발급 및 POST 요청을 SSE 응답 스트림에 매핑하는 동기화(Session Binding) 절차

이원화 구조의 치명적인 모순은 채널이 2개(POST와 SSE)로 갈라져 있다는 점이다. 에이전트가 POST /mcp/messages를 통해 도구 실행 명령을 내렸는데, 서버가 그 연산 결과를 도대체 수천 개의 열려있는 SSE 파이프 중 누구의 스트림으로 돌려보내야 하는가?

이 미아(Orphaned) 패킷의 혼돈을 막기 위해, 접속 극초기 단계에 기계적인 세션 바인딩(Session Binding) 동기화 의식이 거행된다.

  1. 스트림 발급 및 ID 획득: 에이전트가 최초에 GET /mcp/events를 찔러 SSE 파이프를 열 때, 서버는 빈 스트림 껍데기만 주는 것이 아니라 가장 먼저 환영 인사(Welcome Event)와 함께 endpoint URL(예: /mcp/messages?sessionId=x7a9-b3k2)이라는 절대적인 세션 고유 식별자 꼬리표(Token)를 던져준다.
  2. 명령어 스탬핑(Stamping): 이제 클라이언트는 앞으로 발원할 모든 상향 POST 요청의 URL이나 헤더에 반드시 이 sessionId=x7a9-b3k2를 각인하여 보낸다.
  3. 메모리 맵 라우팅(Memory Map Routing): POST 요청을 넘겨받은 백엔드 라우터 엔진은 해시 맵(Hash Map)을 뒤져 x7a9-b3k2 ID를 소유한 활성 SSE 소켓 객체를 신속히 찾아낸다. 워커 노드의 연산이 끝나면 그 결과물은 오차 없이 정확히 해당 클라이언트의 앞마당으로 이어지는 SSE 파이프에 밀려 들어간다.

이 바인딩 절차야말로 물리적으로 동떨어진 두 개의 HTTP 트래픽을 논리적인 하나의 양방향 RPC 파이프라인으로 봉합하는 핵심 연금술이다.
