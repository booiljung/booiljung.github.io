2.4.2.3.3. 민감한 인증 헤더 교차 전송(Cross-site) 방어용 SameSite 쿠키 결합 보안 설계망

MCP 백엔드가 브라우저를 벗어난 외부 에이전트(예: 독립형 데스크톱 앱)로부터는 단순히 Authorization: Bearer <token> 헤더를 믿고 통과시키더라도, 크로스 오리진(Cross-Origin) 환경에서 진입하는 웹 브라우저 기반 에이전트의 경우에는 이 인증 헤더를 자바스크립트에 고스란히 쥐여주는 것은 매우 위험한 탈취 표적이다 (XSS 우려).

웹 에이전트를 위한 가장 방탄형 체계는 인증 정보를 브라우저 깊숙이 은닉 가능한 HTTP-Only 쿠키로 교체하는 것이다. 그러나 쿠키 기반 인증 구조를 무심코 외부 도메인에 방치할 경우 서드파티 크로스 사이트 요청 위조(CSRF) 공격에 즉시 격침당하게 된다.

0.1 SameSite: None과 Secure 플래그의 이중갑옷

CORS 통제망과 완벽히 융합되는 가장 엄격한 브라우저 인증 세션 정책은 3중 방어막을 친 SameSite 토큰 설계망에 귀결된다.

  1. Secure 필수 조건: 브라우저와 서버 간 체결되는 모든 세션 쿠키는 프로토콜 계층이 무조건 HTTPS로 암호화된 통신망(TLS) 위에서만 전송되도록 오직 Secure 마킹을 씌운다.
  2. SameSite=None 개방 허용 (양날의 검): 타 도메인(클라이언트 웹사이트)에서 내 도메인(MCP 서버)을 API처럼 호출해야 하므로, 기존 브라우저가 차단하는 Lax 정책을 허물고 SameSite=None을 명시하여 통행증을 개방한다.
  3. Access-Control-Allow-Credentials: true 검문 강화: 와일드카드 오리진(*) 룰과 절대 양립할 수 없는 이 강력한 플래그를 통해, “명확하게 지정된 단일 오리진에서만 이 쿠키를 품고 본진으로 날아오는 것을 허락한다“는 논리적 도장을 완성한다.

이 치밀한 교차 전송 설계망 하에서는 스크립트 탈취범이 프론트엔드 환경을 장악하더라도 서버의 핵심 쿠키(Credential)에는 손가락조차 댈 수 없게 되며, 강력한 CSRF 원천 차단 우산 아래서 온전한 프로토콜 교류가 이루어진다.
