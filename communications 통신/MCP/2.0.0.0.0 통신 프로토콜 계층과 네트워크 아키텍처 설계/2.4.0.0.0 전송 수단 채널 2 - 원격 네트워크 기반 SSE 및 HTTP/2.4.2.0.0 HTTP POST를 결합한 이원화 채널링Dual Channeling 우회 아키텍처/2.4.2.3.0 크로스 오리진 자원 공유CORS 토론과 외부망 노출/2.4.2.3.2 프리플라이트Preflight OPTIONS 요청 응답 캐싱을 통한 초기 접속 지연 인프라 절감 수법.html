2.4.2.3.2. 프리플라이트(Preflight) OPTIONS 요청 응답 캐싱을 통한 초기 접속 지연 인프라 절감 수법

이원화 채널링 구조에서 에이전트가 도구(Tool)를 거침없이 호출할 때마다, 브라우저는 무조건 본 요청(POST /mcp/messages, Content-Type: application/json)을 날리기 직전, 1차 검문소 격인 OPTIONS /mcp/messages (프리플라이트) 요청을 대기시킨 서버에 먼저 쏘아 올려 CORS 권한 판별을 의뢰한다.

이중 통신망으로 인해 도구 하나를 사용할 때마다 네트워크 왕복 시간(RTT)이 2배로 폭증하며, 대서양을 횡단하는 글로벌 라우팅 환경에서는 치명적인 200~500밀리초 이상의 체감 지연(Latency)이 사용자 UI 응답성을 갉아먹게 된다.

0.1 Max-Age를 통한 거대한 브라우저 캐싱 방파제

이 무의미한 프리플라이트 난사를 평정하기 위한 가장 강력한 인프라 절감 무기는 바로 브라우저 내부의 CORS 캐싱 보일러플레이트를 자극하는 것이다.

  • 서버는 OPTIONS 요청에 대한 성공 응답 시 반드시 Access-Control-Max-Age: 86400 헤더를 하사한다.
  • 이는 브라우저 커널(Engine)에게 “앞으로 24시간 동안은, 이 똑같은 엔드포인트(URL)로 똑같은 메서드(POST)와 헤더 구조를 들고 갈 때는 나한테 묻지 말고 그냥 바로 본 요청을 발사하라“는 통행증 도장을 이마에 찍어주는 것과 같다.

이 한 줄의 규약만으로도, MCP 프로토콜 통신 오버헤드의 절반(Preflight Layer)이 통째로 소거되며 수십만 에이전트의 타격으로부터 백엔드 게이트웨이의 인프라 과부하(Overkill)를 방패처럼 지켜내는 파괴적 지연 절감 효익이 발현된다.
