2.4.2.3.1. 웹 브라우저 기반 에이전트 사용자를 위한 세밀한 Origin 검증 기반 허용 화이트리스트 규정

가장 초보적이면서도 치명적인 실수 중 하나는 CORS 설정에 무심코 Access-Control-Allow-Origin: * (와일드카드)를 열어 두는 짓이다. 이 한 줄의 코드로 인해, 사용자가 우연히 악성 웹사이트(evil.com)에 접속하는 순간 브라우저 백그라운드 스크립트가 사용자의 로컬 환경이나 사내망에 띄워진 MCP 서버에 제멋대로 접근하여 모든 데이터를 유출해 갈 수 있다.

0.1 정규식 화이트리스트와 다이내믹 오리진(Dynamic Origin) 주입

웹 브라우저를 클라이언트로 삼는 MCP 아키텍처는 반드시 엄격한 화이트리스트(Whitelist) 오리진 검증망을 거쳐야만 한다.

  1. 정적 오리진 배열 선언: 서버의 부트스트랩(Bootstrap) 단계에서, 접근을 허가할 인가된 호스트 네임 목록(예: https://chat.openai.com, https://claude.ai, 사내 도메인 등)을 환경 변수 배열로 하드코딩하여 로드한다.
  2. 동적 리플렉션(Reflection): 브라우저가 자비심 없이 날려대는 Origin 헤더를 검열하여, 화이트리스트 내포 검사(Sub-string Match나 정규식 패턴 엔진)를 통과한 경우에만 해당 오리진 도메인 이름을 그대로 복사하여 Access-Control-Allow-Origin: {인가된_도메인} 형태로 미러링(Mirroring) 반환한다.
  3. 와일드카드 및 널 오리진 차단 거부: 일치하지 않거나, 심지어 Origin: null (iframe 내 샌드박스 등)로 접근을 찔러보는 불분명한 횡적인 요청은 즉각 403 Forbidden 네트워크 패킷 폐기 대상으로 간주하고 CORS 헤더 자체를 아예 생략하여 브라우저 엔진이 스크립트를 파괴하도록 방치한다.
