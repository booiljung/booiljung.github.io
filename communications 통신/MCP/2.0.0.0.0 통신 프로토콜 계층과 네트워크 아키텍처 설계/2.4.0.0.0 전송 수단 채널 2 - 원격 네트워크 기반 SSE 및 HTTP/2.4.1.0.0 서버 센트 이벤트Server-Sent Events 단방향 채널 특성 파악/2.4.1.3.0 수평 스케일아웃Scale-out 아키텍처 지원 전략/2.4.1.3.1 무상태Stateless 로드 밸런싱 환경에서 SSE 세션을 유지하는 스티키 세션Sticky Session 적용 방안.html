2.4.1.3.1. 무상태(Stateless) 로드 밸런싱 환경에서 SSE 세션을 유지하는 스티키 세션(Sticky Session) 적용 방안

현대 클라우드의 트래픽 분산처리는 본질적으로 무상태(Stateless) 라우팅을 지향한다. ALB(Application Load Balancer) 서버는 클라이언트의 POST 명령을 라운드 로빈(Round-Robin) 방식으로 각기 다른 워커 노드(Node A, Node B, Node C)에 뿌려버린다.

그러나 이원화 채널링을 수행하는 MCP 프로토콜에서 SSE 스트림은 지극히 상태 존속적(Stateful)인 단일 TCP 연결이다. 만약 클라이언트가 Node A와 SSE 연결을 맺고 있는데, 후속 POST 명령어(도구 실행 등)가 로드 밸런서에 의해 엉뚱한 Node C로 라우팅된다면, Node C는 그 명령의 결과(진행 상태 알림 등)를 도대체 어느 파이프라인으로 돌려보내야 할지 몰라 이벤트를 허공에 날리게 된다.

0.1 채널 고정을 위한 스티키 세션(Sticky Session) 아키텍처

이러한 세션 단절 병리 현상을 돌파하기 위해 클라우드 인프라 계층에는 **스티키 세션(Sticky Session)**이 강제 투입된다.

  1. 세션 쿠키(Cookie) 주입: 클라이언트가 최초로 SSE 연결 맺기(GET /mcp/events)를 시도할 때, 로드 밸런서는 도착한 워커 노드의 고유 식별자 자국(Hash)이 남겨진 Set-Cookie 헤더(예: AWSALB, route=node-A)를 클라이언트에게 강제로 쥐어준다.
  2. 후속 POST 타겟팅 고정(Pinning): 이후 클라이언트가 발송하는 모든 명령어(POST /mcp/messages)는 이 쿠키를 품고 전송된다. 로드 밸런서는 쿠키의 해시를 해석하여, 무상태 기반 라운드 로빈 정책을 즉시 폐기하고 모든 트래픽을 오차 없이 기존에 SSE 파이프가 꽂혀 있는 정확히 그 노드(Node A)로만 강제 전송한다.

이 메커니즘을 통해 MCP 클러스터는 무상태 라우팅의 수평 스케일아웃 이점을 온전히 누리면서도, SSE 기반의 상태형 비동기 스트림을 파괴하지 않는 투명한 세션 존속 환경을 유지하게 된다.
