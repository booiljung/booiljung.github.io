2.4.1.3.2. 레디스 펍섭(Redis Pub/Sub)을 매개로 인스턴스 간 전파형 이벤트 브로드캐스팅 라우팅 설계

클라이언트가 스티키 세션(Sticky Session) 방어막마저 뚫지 못하는 엄격한 모바일 접속망이나 다중 인그레스(Ingress) 환경에 놓여 있다면, 앞서 논했던 단일 노드 고정(Pinning) 전략은 맥없이 스위칭 단절을 맞이하게 된다. POST 요청이 Node B로 넘어가고 SSE 연결은 Node A에 나뉘어 물려 있는 최악의 분단(Partition) 사태 하에서는, 이원화된 노드 간에 이벤트 페이로드를 조달할 수 있는 광역 백플레인(Global Backplane) 버스가 긴급히 요구된다.

0.1 중앙 집중형 Pub/Sub 아키텍처의 개입

이 고차원의 메시지 라우팅을 위해 MCP 백엔드 클러스터는 Redis Pub/Sub 혹은 Apache Kafka와 같은 초고속 인메모리 브로드캐스팅 버스망을 아키텍처 정중앙에 투입(Inject)시킨다.

  1. 동적 채널 구독(Subscribe): 연결 초단, Node A가 특정 클라이언트와 SSE 파이프를 수립하면, Node A는 즉시 레디스 서버에 해당 클라이언트 고유 식별자(Session ID)를 바탕으로 전용 Pub/Sub 채널(mcp:events:session-123)을 구독(Subscribe)하기 시작한다.
  2. 크로스-노드 브로드캐스팅(Publishing): 후속 POST 명령을 라운드 로빈으로 접수한 엉뚱한 Node B가 연산 결과 이벤트를 도출하면, Node B는 이를 클라이언트에게 직접 밀어 넣는 대신 (연결이 없으므로), 앞서 정의된 레디스 채널(mcp:events:session-123)로 그 페이로드를 발행(Publish)해 버린다.
  3. 메시지 릴레이(Relay): 레디스 버스의 전파를 감지한 Node A는 이를 낚아채어 자신이 쥐고 있는 물리적 SSE TCP 채널을 통해 최종적으로 클라이언트 에이전트에 발사한다.

이 전파형 이벤트 브로드캐스팅 라우팅 망이 성립됨으로써, MCP 클러스터는 어떤 로드 밸런싱의 변덕 앞에서도 백엔드 노드 간 수평 결합력(Horizontal Cohesion)을 유지하며 극도의 탄력적 이벤트 릴레이를 성취하게 된다.
