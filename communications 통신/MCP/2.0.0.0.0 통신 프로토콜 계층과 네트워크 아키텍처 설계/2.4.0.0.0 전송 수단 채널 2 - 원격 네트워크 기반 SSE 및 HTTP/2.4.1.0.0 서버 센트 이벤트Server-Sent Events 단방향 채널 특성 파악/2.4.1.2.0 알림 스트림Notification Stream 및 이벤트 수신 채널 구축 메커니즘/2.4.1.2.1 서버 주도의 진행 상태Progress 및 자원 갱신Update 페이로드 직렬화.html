2.4.1.2.1. 서버 주도의 진행 상태(Progress) 및 자원 갱신(Update) 페이로드 직렬화

에이전트가 100GB 로그 파일 분석과 같은 장기 실행 도구(Long-running Tool)를 원격 서버에 요청했을 때, 서버가 분석을 끝낼 때까지 침묵한다면 클라이언트 사용자는 시스템이 멈춘 것인지 정상 작동 중인지 알 길이 없다.
이러한 인지적 사각지대(Blind Spot)를 해소하기 위해, 서버는 SSE 채널을 통해 주기적으로 현재의 진행 상태(Progress) 토큰을 발산해야 한다.

0.1 스트림 페이로드의 다차원 직렬화 구조

SSE 프로토콜 상에서 서버가 내려보내는 데이터 블록은 단순히 data: 접두어가 붙은 평문이 아니라, 엄격하게 규격화된 JSON-RPC 알림(Notification) 객체로 직렬화(Serialization)되어야 한다.

id: 1042
event: message
data: {
data:   "jsonrpc": "2.0",
data:   "method": "notifications/progress",
data:   "params": {
data:     "progressToken": "task-abc-123",
data:     "progress": 45,
data:     "total": 100
data:   }
data: }
\n\n

위의 직렬화 페이로드에서 보듯, 서버는 진행 토큰(progressToken)과 현재 작업량 퍼센티지를 JSON으로 단단하게 패키징하여 스트림 파이프에 밀어 넣는다. 더불어 파일 스토리지나 메모리 자원의 상태가 변했을 때는 notifications/resources/updated 메서드를 사용하여 즉각적인 자원 갱신(Resource Update) 페이로드를 살포(Broadcast)한다.
이처럼 규격화된 JSON 직렬화 구조 덕분에 클라이언트의 라우터는 끊임없이 밀려오는 바이트 스트림 파편 속에서도, 정확한 이벤트를 적출하여 에이전트 UI의 프로그레스 바(Progress Bar)를 실시간으로 전진시킬 수 있는 동력을 얻게 된다.
