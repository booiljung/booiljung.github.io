2.4.1.2.2. 일시적 네트워크 단절 극복을 위한 Last-Event-ID 기반의 투명한 이벤트 재연결(Reconnection) 전략

아무리 HTTP/2 기반의 연결이 견고하다 한들, 와이파이 재접속, 모바일망 핸드오버, 혹은 로드밸런서의 순단 현상 앞에서는 물리적 TCP 소켓이 끊어지는 상황(Drop)을 결코 피해 갈 수 없다.
만약 SSE 스트림이 찰나의 순간 끊어졌고, 하필 그 단절된 1초 사이에 서버가 “치명적인 보안 경고(Security Alert)” 이벤트 알림을 발송했다면 이 결측된 이벤트는 영원히 우주 미아로 사라져 시스템의 정합성을 붕괴시킬 것이다.

0.1 이벤트 커밋 식별자(ID)를 통한 상태의 재동기화

MCP 아키텍처는 이 끔찍한 스트림 결손 사태를 방어하기 위해 SSE 스펙의 고유 기능인 Last-Event-ID 복구 메커니즘을 투명하게 활용한다.

  1. 순차적 ID의 발급: 서버는 이벤트를 송출할 때마다 페이로드 앞에 반드시 순차적으로 증가하는(Monotonically increasing) 고유 식별자인 id: 필드를 조립하여 전송한다 (예: id: 844). 클라이언트는 이벤트를 파싱함과 동시에 이 마지막 ID를 커밋(Commit)하여 자신의 메모리에 각인한다.
  2. 재연결 시 헤더 주입: 네트워크가 끊어지고 클라이언트 에이전트가 재연결(Reconnection)을 위해 다시 GET /mcp/events 요청을 시도할 때, 라우터는 브라우저 엔진(또는 Fetch API)의 통제망을 빌려 HTTP Request 헤더에 은밀하게 **Last-Event-ID: 844**를 끼워 넣는다.
  3. 서버 측 캐시 리플레이(Replay): 헤더를 넘겨받은 서버는 “클라이언트가 844번 이벤트까지만 읽고 기절했군“이라고 정확히 인지한다. 서버는 자신의 짧은 메모리 링 버퍼(Ring Buffer)를 뒤져 클라이언트가 놓쳤던 845번, 846번 이벤트를 소켓이 열리자마자 즉각적으로 몰아서 리플레이(Replay) 투척한다.

단 한 줄의 커스텀 폴링(Polling) 로직 없이, 오직 HTTP 표준 헤더의 재전송만으로 이루어지는 이 놀라운 재동기화 전략은 클라이언트 개발자에게 네트워크의 변덕을 완전히 추상화(Abstraction)하여, 마치 영원히 마르지 않는 무결점 파이프를 가진 것 같은 투명한 환상을 제공한다.
