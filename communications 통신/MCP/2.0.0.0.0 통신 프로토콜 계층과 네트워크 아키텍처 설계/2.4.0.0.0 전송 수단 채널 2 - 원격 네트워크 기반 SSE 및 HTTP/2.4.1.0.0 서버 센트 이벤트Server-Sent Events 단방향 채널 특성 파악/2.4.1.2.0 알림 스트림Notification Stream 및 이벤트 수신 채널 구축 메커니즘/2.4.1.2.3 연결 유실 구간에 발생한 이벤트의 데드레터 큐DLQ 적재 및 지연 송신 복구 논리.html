2.4.1.2.3. 연결 유실 구간에 발생한 이벤트의 데드레터 큐(DLQ) 적재 및 지연 송신 복구 논리

Last-Event-ID 기반의 이벤트 리플레이(Replay) 아키텍처는 찰나의 순단(Blink) 현상을 커버하는 훌륭한 방패이지만, 만약 클라이언트가 장시간 오프라인 상태(예: 노트북 덮개 닫힘 등)에 빠진다면 상황은 180도 역전된다.
수만 명의 에이전트를 동시 다발로 obsług(Serving)하는 중앙 집중형 MCP 서버의 입장에서, 연락이 끊긴 특정 클라이언트 하나를 위해 메모리(RAM) 상의 링 버퍼에 과거 이벤트를 무한정 쌓아두는 행위는 필연적으로 서버 메모리 고갈(Out of Memory)과 극단적인 장애(Crash)를 초래한다.

0.1 핫 버퍼(Hot Buffer) 초과분 탈락과 DLQ의 콜드 스토리지 전환

이 극한의 모순을 방어하기 위해 도메인 주도의 엔터프라이즈 서버 라우터는 이중화된 지연 송신 복구망을 구축해야 한다.

  1. 버퍼 윈도우 스로틀링(Throttling): 서버는 각 클라이언트 커넥션마다 허용된 인메모리 링 버퍼의 크기(예: 최근 500개 이벤트)를 넘기는 순간, 오래된 이벤트를 자비 없이 RAM에서 잘라낸다(Eviction).
  2. 데드레터 큐(Dead Letter Queue) 영속화: 이때 증발하기 아까운, 특히 결코 유실되어선 안 되는 ’치명적 상태 갱신’과 같은 중요 이벤트들은 메모리에서 탈락되는 순간 Redis 스트림이나 카프카(Kafka) 기반의 **물리적 데드레터 큐(DLQ)**로 떨구어져 영구 저장(Cold-Storage)된다.

0.2 지연 복원(Lazy Restore) 시점의 컨텍스트 리빌딩

훗날 클라이언트가 2시간 만에 다시 깨어나 Last-Event-ID: 700을 들고 재연결을 요청해 왔을 때, 서버의 핫 버퍼에는 이미 1500번 이후의 최신 이벤트들만 남아있다.
이 갭(Gap)을 발견한 서버 측 세션 매니저는 클라이언트에게 HTTP 409 Conflict나 페이로드 초기화 신호(Reset)를 발송하는 대신, 비동기 워커를 가동해 DLQ 디스크에서 701번부터 1500번까지의 데이터를 허겁지겁 퍼들여 스트림을 통해 리플레이(Rebuild)해준다.
결국 이 투트랙(Two-Track) 지연 송신 복구 논리 덕분에 서버는 소중한 RAM 리소스를 안전하게 수호하면서도, 클라이언트 입장에서는 단 하나의 비트도 잃어버리지 않는 극도의 영속적 견고함을 동시에 획득하게 된다.
