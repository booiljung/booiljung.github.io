2.4.1.1.1. 양방향 WebSocket 대신 단방향 SSE를 주력 모델로 채택한 아키텍처적 근거 및 폴백(Fallback) 우위성

현대의 실시간 웹 애플리케이션에서 양방향 통신의 대명사인 WebSocket(웹소켓)을 배제하고, 굳이 단방향인 SSE(Server-Sent Events)와 HTTP POST를 조합한 이원화 아키텍처를 MCP 프로토콜의 표준 원격 모델로 격상시킨 데에는 매우 차갑고 실용적인 공학적 타산이 존재한다.

WebSocket은 최초 HTTP 핸드셰이크 이후 Upgrade: websocket 헤더를 통해 TCP의 날 것(Raw)에 가까운 커스텀 프레이밍 프로토콜로 채널을 탈바꿈시킨다. 이는 양방향 저지연 통신에는 탁월하지만, 인프라의 관점에서는 HTTP가 제공하던 캐싱, 압축(GZIP/Brotli), HTTP 보안 헤더, 라우팅 컨텍스트를 모조리 잃어버리는 치명적 형태의 블랙박스(Blackbox)가 된다.

0.1 백엔드 아키텍처의 단순성과 HTTP 폴백 우위성

반면 SSE 기술의 본질은 순도 100%의 단방향 표준 HTTP 응답 스트림이다.

  1. 단순화된 연결 관리: 서버는 세션마다 생명주기가 무한정에 가까운 양방향 WebSocket 소켓 객체를 메모리에 고상하게 들고 있을 필요가 없다.
  2. 이원화의 강점: 클라이언트가 보내는 명령어(요청)는 HTTP POST 채널을 타고 무상태(Stateless)로 서버 트래픽 밸런서를 자유자재로 통과하고, 서버로부터 수신하는 실시간 알림은 오직 단일한 SSE 창구를 통해 내려받는 역할 분담(Separation of Concerns) 아키텍처가 확립된다.
  3. 폴백(Fallback) 생태계: SSE 파이프가 끊기거나 특수한 구형 환경에서 차단될 경우, 이 아키텍처는 즉시 전통적인 Long-Polling(HTTP POST/GET의 반복)만으로도 프로토콜의 기능성을 100% 재현(Fallback)해낼 수 있는 막강한 우위성을 발휘한다. WebSocket처럼 프로토콜이 근본적으로 차단되는 환경을 마주했을 때 전체 통신망이 무너져 내리는 파멸적 실패(Catastrophic Failure)를 원천 차단하는 것이다.
