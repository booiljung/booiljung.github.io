2.4.1.1.3. HTTP 스트리밍 지원 게이트웨이(Gateway)를 통과할 때 발생하는 버퍼링 타임아웃 회피망

SSE 통신이 표준 HTTP 프로토콜을 사용한다는 점은 방화벽 관통에 막대한 이득을 주지만, 동시에 치명적인 미들웨어 응답 버퍼링(Response Buffering) 문제를 유발한다. Nginx, AWS API Gateway, 혹은 Cloudflare 와 같은 현대의 L7 게이트웨이들은 본능적으로 백엔드 서버의 조각난 HTTP 응답(Chunks)을 메모리에 전부 모았다가 (Buffering), 일정 덩어리가 완성되거나 연결이 닫힐 때만 클라이언트에게 한 번에 쏘아 보내는 최적화 습성을 지니고 있다.

실시간으로 초당 수십 개의 상태 갱신을 푸시(Push)해야 하는 SSE 이벤트 스트림이 이 버퍼링의 덫에 걸리면, 클라이언트 에이전트는 서버가 이벤트를 쏴도 게이트웨이 버퍼가 가득 찰 때까지 최대 수 분간 아무런 메시지도 받지 못하는 **가짜 타임아웃(Fake Timeout)**과 극단적인 지연(Latency Spikes)을 경험하게 된다.

0.1 X-Accel-Buffering 제한과 인위적 여백(Padding) 회피술

이 기계적 족쇄를 풀기 위해 MCP 원격 통신망은 두 가지 층위의 하이브리드 회피망을 가동한다.

  1. 동적 헤더 주입(Dynamic Header Injection): 서버는 SSE 채널을 최초 개방할 때, Cache-Control: no-cache 및 X-Accel-Buffering: no (Nginx 전용) 같은 특수 제어 헤더를 강제 삽입하여, 중간에 위치한 모든 리버스 프록시 노드들에게 “이 스트림은 절대로 버퍼링하지 말고 들어오는 즉시 송출하라“고 강력히 지시한다.
  2. 페이로드 볼륨 펌핑(Payload Padding): 완고한 클라우드 게이트웨이(예: AWS API Gateway의 8MB 버퍼 강제 정책)가 헤더 지시를 무시할 최악의 경우를 대비하여, 서버는 보낼 데이터가 당장 없더라도 주기적으로 1KB 이상의 의미 없는 공백(Whitespace) 혹은 더미 주석 코멘트(: ping\n\n)를 스트림에 인위적으로 퍼붓는다. 이 지속적인 볼륨 펌핑은 프록시의 버퍼 용량 임계치를 강제로 초과하게(Overflow) 만들어, 게이트웨이가 억지로라도 패킷을 클라이언트 측으로 밀어내게 멱살을 잡는 극단적인 스트리밍 회피 기술이다.
