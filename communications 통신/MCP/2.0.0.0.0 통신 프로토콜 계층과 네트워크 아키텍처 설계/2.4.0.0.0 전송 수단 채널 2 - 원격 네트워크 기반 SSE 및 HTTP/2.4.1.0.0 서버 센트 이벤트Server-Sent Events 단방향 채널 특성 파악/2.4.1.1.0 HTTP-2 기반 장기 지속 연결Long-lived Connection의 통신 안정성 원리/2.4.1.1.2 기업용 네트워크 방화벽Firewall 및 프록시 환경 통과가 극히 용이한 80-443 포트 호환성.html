2.4.1.1.2. 기업용 네트워크 방화벽(Firewall) 및 프록시 환경 통과가 극히 용이한 80/443 포트 호환성

현대의 B2B/엔터프라이즈 환경에서 클라이언트 에이전트 애플리케이션이 외부의 MCP 서버(또는 사내망 깊숙이 숨겨진 백엔드)와 연동하려 할 때 직면하는 가장 무자비한 장벽은 바로 사내 IT 보안팀이 치밀하게 구축해 놓은 거대한 L7 딥 패킷 방화벽(Firewall)과 강제 프록시(Proxy) 게이트웨이이다.

사내 방화벽 규정은 대개 비표준 포트의 In/Outbound 트래픽을 모조리 막아버리며, 심지어 80(HTTP), 443(HTTPS) 포트라 하더라도 패킷 내부 구조가 전통적인 HTTP 규격과 다르면(예: WebSocket Frame) 악성 우회 트래픽으로 간주하여 무참하게 잘라(Drop) 버린다.

0.1 표준 HTTP 프로토콜의 백과사전적 투명성

MCP가 원격 채널로 SSE(Server-Sent Events)를 채택한 결정타가 바로 여기에 있다. 앞서 기술했듯 SSE 스트림은 본질적으로 그 어떠한 특수 프로토콜의 마스킹(Masking)도 없는 순수한 HTTP/1.1 커넥션 또는 HTTP/2 데이터 프레임에 불과하다.

SSE 통신은 일반적인 웹 브라우징과 완벽히 동일한 표준 80번 혹은 443번 포트를 사용하며, 그 응답 헤더에는 그저 Content-Type: text/event-stream 이라는 유효무해한 텍스트 선언이 적혀 있을 뿐이다.
그 결과, 기업의 엄격한 포워드/리버스 프록시(Reverse Proxy)나 DPI(Deep Packet Inspection) 방화벽은 이 스트림을 대용량 평문 텍스트가 다운로드 창구를 통해 흘러나가는 정상적인 HTTP 통신으로 매끄럽게 인지(Interpret)한다. 에이전트 운영팀은 IT 부서에 별도의 포트 개방 요청이나 보안 예외 등록(Whitelisting)을 호소할 필요 없이, 가장 폐쇄적인 코포레이트 네트워크 환경에서도 에이전트-서버 간의 척수 신경망을 조용하고 매끄럽게 관통(Traverse)시킬 수 있다.
