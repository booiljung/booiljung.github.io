2.4.1.4.3. 백그라운드 전환 시 일시 중지(Suspend) 리소스 상태를 능동 보존하는 전력 최적화 로직

사용자가 에이전트 앱에 복잡한 검색 요청을 날려놓고 결과를 기다리다 지쳐 홈 화면으로 나가거나 다른 앱으로 스와이프(Background Transition)해 버릴 때, 모바일 OS(iOS 및 Android)는 자원 보존을 위해 몇 초 내로 해당 앱의 CPU 스레드와 백그라운드 소켓을 가차 없이 동결(Suspend)시켜 버린다.
OS 레벨에서 강제 동결된 프로세스는 서버 관점에서는 핑퐁이 멈춘 ’갑작스러운 데드 커넥션’으로 비치며, 에이전트 입장에서는 작업의 연속성이 붕괴되는 파열 기점이 된다.

0.1 포그라운드-백그라운드 통신 상태 전이망

이 혹독한 OS 생태계에서 에이전트가 살아남으려면 애플리케이션의 렌더링 생명주기와 전송 계층 소켓 상태를 완전히 결합하는 전방위적 생태 적응(Adaptation) 로직을 짜넣어야 한다.

  1. 우아한 절전 모드 진입(Graceful Suspend): 앱이 백그라운드로 밀려나는(Background Activity) OS 이벤트를 감지한 즉시, 클라이언트 라우터는 SSE 스트림 소켓을 비정상적으로 터뜨리거나 유지하려 애쓰지 않는다. 대신 서버를 향해 명시적인 Disconnect(Suspend) 시그널을 꽂아주고 조용히 소켓을 스스로 닫는다.
  2. 서버 측 작업 고아화 방지: 서버는 연결이 끊어졌다고 하여 돌고 있던 무거운 LLM 추론 연산을 폐기하지 않는다. 해당 세션을 ‘동결-연산 중(Suspended-Running)’ 상태로 백그라운드 파크(Part)하고, 결과 도출 시 이를 캐시(Cache)에 고이 모셔둔다.
  3. 포그라운드 부활(Foreground Revival): 사용자가 다시 앱을 켜면(Restore), 클라이언트 라우터는 Last-Event-ID 복원 절차와 함께 활기차게 재연결을 찌르며 들어온다. 서버는 캐시해 두었던 연산 완료 이벤트를 리플레이(Replay)하여 내려주고, 클라이언트는 마치 아무 일도 없었다는 듯 정지된 진행 바를 100%로 채우며 화면을 갱신해 낸다.
