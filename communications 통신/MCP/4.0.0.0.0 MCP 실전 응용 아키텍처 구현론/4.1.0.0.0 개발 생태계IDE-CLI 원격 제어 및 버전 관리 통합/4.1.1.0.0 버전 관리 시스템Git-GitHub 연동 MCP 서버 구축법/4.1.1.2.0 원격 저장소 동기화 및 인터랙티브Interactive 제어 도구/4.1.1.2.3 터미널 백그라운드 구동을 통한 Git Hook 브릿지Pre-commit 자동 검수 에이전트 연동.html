4.1.1.2.3. 터미널 백그라운드 구동을 통한 Git Hook 브릿지(Pre-commit 자동 검수 에이전트 연동)

에이전트가 코드를 다 짜고 당당하게 git commit 도구를 호출했는데, 사내 규정상 설치된 pre-commit 훅(예: Lint, Black)이 실패하여 터미널이 붉은색 에러를 내뿜으며 프로세스가 죽어버린다면?

  1. Pty(단말기 가상화) 할당: 브로커는 자식 프로세스로 Git 명령을 실행할 때, 단순 스트림 버퍼가 아니라 가상의 TTY(Pseudo-terminal) 환경을 할당하여 pre-commit 훅이 정상 작동할 수 있는 생태계를 제공한다.
  2. 표준 출력 인터셉트 및 피드백: pre-commit이 내뱉은 에러(예: “Line 25 is too long”)를 브로커가 고스란히 캡처하여 도구 실행 실패(ToolExecutionError)의 메타데이터로 에이전트에게 돌려준다.
  3. 루프백(Loopback) 자가 수정: 훌륭하게 설계된 에이전트는 이 에러를 받자마자 “아, 내가 짠 코드가 Lint를 통과 못 했네“라고 인지한 뒤, 코드를 다시 아름답게 다듬고 2차 git commit을 시도하는 완벽한 자가 교정(Self-correction) 루프를 달성한다.
