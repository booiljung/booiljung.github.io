4.1.1.2.1. 무인증(No-Auth) 로컬 액세스와 원격 쓰기(Push/Pull) 간의 자격 증명(Credential) 격리 방안

내 PC의 에이전트가 다른 사람의 깃허브 레포지토리에 악성 코드를 푸시(Push)할 수 있다면 끔찍한 일이다.

  1. 권한의 비대칭성: 로컬 파일을 읽고 쓰는 데는 사용자 OS의 권한을 통과하면 그만이지만, git push를 쏘는 순간에는 외부 서버(GitHub)의 인증 장벽을 만나게 된다.
  2. 키체인(Keychain) 위임 브릿지: MCP 브로커는 절대 에이전트(LLM)에게 깃허브 PAT(Personal Access Token) 문자열을 직접 주지 않는다. 대신 에이전트가 tool: git_push를 호출하면, 브로커가 로컬 OS의 자격 증명 관리자(예: Mac Keychain, SSH Agent)를 대리 호출하여 1회성 서명(Signing)만 받아다가 푸시를 수행한다.
  3. 블라인드 푸시: LLM은 어떻게 인증이 통과되는지 전혀 모르는 블라인드(Blind) 상태로 오직 “명령의 성공/실패 여부“만 응답받는 완벽한 자격 증명 격리 환경이 구축된다.
