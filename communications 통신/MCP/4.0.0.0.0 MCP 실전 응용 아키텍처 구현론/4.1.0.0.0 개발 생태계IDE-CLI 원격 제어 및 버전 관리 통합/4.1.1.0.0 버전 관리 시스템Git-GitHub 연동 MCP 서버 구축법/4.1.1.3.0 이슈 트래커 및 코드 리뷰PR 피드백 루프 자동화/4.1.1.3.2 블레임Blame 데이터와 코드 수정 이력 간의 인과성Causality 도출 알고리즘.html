4.1.1.3.2. 블레임(Blame) 데이터와 코드 수정 이력 간의 인과성(Causality) 도출 알고리즘

“이 이상한 코드를 누가, 도대체 왜 짰는지 알아내서 고쳐.” 이 고난도의 추리 임무를 해결하는 과정이다.

  1. Deep Blame 스캐닝: 에이전트는 해당 함수의 줄 번호를 넣고 git_blame 도구를 호출한다. 도구는 “이 줄은 3달 전 커밋 abc1234에서 Alice가 작성함” 이라는 응답을 준다.
  2. 커밋 메시지 및 이슈 티켓 역추적: 여기서 멈추지 않고, 에이전트는 search_commit(hash="abc1234") 도구를 다시 호출해 커밋 메시지 "Fix memory leak in issue #42"를 찾아낸다. 그 다음 GitHub 이슈 도구인 read_issue(42)를 호출해, 당시 어떤 컨텍스트에서 이 코드가 도입되었는지 “사건의 지평선” 전체를 빨아들인다.
  3. 컨텍스트 인식형 리팩토링: 3달 전의 비즈니스 맥락(이슈 42번)을 온전히 이해한 LLM은, “당시엔 메모리 릭 수정을 위해 이렇게 칠 수밖에 없었으나, 이제는 캐시 시스템이 도입되었으니 이렇게 고치는 것이 맞다“라며 원작자(Alice)의 의도를 파괴하지 않는 고차원적인 리팩토링 제안을 도출해 낸다.
