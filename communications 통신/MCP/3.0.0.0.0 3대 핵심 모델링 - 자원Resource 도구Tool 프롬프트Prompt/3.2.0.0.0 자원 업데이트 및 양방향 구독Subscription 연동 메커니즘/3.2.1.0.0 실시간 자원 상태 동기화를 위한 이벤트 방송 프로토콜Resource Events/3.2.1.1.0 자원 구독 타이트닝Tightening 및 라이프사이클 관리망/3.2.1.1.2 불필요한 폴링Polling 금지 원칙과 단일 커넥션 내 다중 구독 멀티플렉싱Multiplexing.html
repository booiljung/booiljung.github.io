3.2.1.1.2. 불필요한 폴링(Polling) 금지 원칙과 단일 커넥션 내 다중 구독 멀티플렉싱(Multiplexing)

웹 초창기에는 클라이언트가 데이터의 변경 여부를 알기 위해 1초마다 서버에 새로고침(Refresh)을 요청하는 폴링(Polling) 기법이 성행했다. 하지만 수많은 에이전트가 동시에 동작하는 MCP 환경에서 파일 100개를 감시하기 위해 1초마다 100건의 HTTP 요청을 쏘는 것은 네트워크 스택의 마비를 의미한다.

이를 차단하기 위해 MCP는 프로토콜 레이어 레벨에서 설계적 제어 장치를 도입했다.

  1. 폴링 금지 프로토콜 원칙: 서버는 클라이언트로부터 들어오는 일률적인 반복 조회(Rate-limited Polling) 패턴을 감지하면, 이를 자원 고갈(Rate Limit) 공격으로 간주하고 해당 에이전트에게 경고(Warning)를 보낸 뒤 일시적으로 연결을 끊어버린다. 동기화가 필요하다면 오직 subscribe 스펙을 따르도록 강제하는 것이다.
  2. 다중 채널 멀티플렉싱(Multiplexing): 에이전트가 README.md, src/app.py, package.json 세 개의 리소스를 구독한다고 해서 3개의 감시 스레드나 3개의 SSE(Server-Sent Events) 커넥션을 맺을 필요는 없다. MCP는 이미 확립된 단 하나의 양방향 채널(JSON-RPC over SSE/Stdio) 위에서 모든 자원의 이벤트 푸시(Push) 스트림을 논리적으로 멀티플렉싱(Multiplexing)하여 전송한다.
  3. 토픽 기반 라우팅(Topic-based Routing): 단일 커넥션 위로 우르르 쏟아져 들어오는 이벤트 알림들은 {"uri": "mcp://...", "event": "updated"} 포맷에 따라 클라이언트 내부의 토픽 라우터에서 갈래갈래 찢어져(Demultiplexing) 각각의 파일 렌더링 뷰를 독립적으로 갱신한다.
