3.2.1.1.3. 네트워크 단절로 인한 구독 손실을 감시하고 재연결 시 일괄 복원하는 상태 유지기

로컬 프로세스(Stdio)가 아닌 원격 네트워크(SSE) 기반으로 동작하는 에이전트의 경우, 필연적으로 라우터 재부팅이나 와이파이 단절 등으로 인한 찰나의 연결 끊김 현상(Drop-out)을 겪게 된다. 연결이 끊어지면 서버는 더 이상 알림을 보낼 곳이 없다고 판단해 구독 명단에서 해당 세션을 파기(Destroy)한다.

다시 네트워크가 복원되었을 때, 에이전트가 이전에 구독 중이던 수십 개의 URI를 일일이 기억해 내어 다시 subscribe 핑을 날리는 것은 로직상 비효율적이며 누락의 위험이 따른다.

  1. 클라이언트 측 인메모리 구독 명부(Registry): 클라이언트의 MCP SDK에는 현재 성공적으로 모니터링 중인 자원 URI 목록들의 집합(Set)을 들고 있는 명부 저장소가 활성화되어 있다.
  2. 커넥션 힐링 훅(Healing Hook): 네트워크 소켓 재연결이 감지(Reconnected Event)되는 순간, 클라이언트의 컨트롤러는 찰나에 놓친 데이터 변경분을 동기화하기 위해 백그라운드에서 명부에 있는 모든 자원을 재구독(Multi-subscribe)하는 일괄 복원 트랜잭션을 날린다.
  3. 단절 구간의 오염 회복: 재연결 직전 수 초 동안 서버 측에서 리소스가 변경되었을 수 있다. 따라서 클라이언트는 재구독 요청과 동시에, 명부에 있는 자원들에 대해 투명한 1회성 백그라운드 갱신 쿼리(resources/read)를 날려 캐시(Cache) 상태를 현실 세계와 강제 정렬시키는 자동 치유 알고리즘을 수행한다.
