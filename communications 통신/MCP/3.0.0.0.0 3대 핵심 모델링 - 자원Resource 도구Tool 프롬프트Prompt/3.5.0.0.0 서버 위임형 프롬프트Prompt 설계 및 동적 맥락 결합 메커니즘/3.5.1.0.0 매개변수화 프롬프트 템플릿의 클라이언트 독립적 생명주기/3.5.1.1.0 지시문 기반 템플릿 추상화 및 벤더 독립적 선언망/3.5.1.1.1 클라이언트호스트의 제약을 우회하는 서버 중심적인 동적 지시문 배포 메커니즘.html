어제까지 잘 작동하던 “사내 코드 리뷰 프롬프트“가 오늘부터 적용된 새로운 보안 규정 때문에 낡은 지시문이 되었다면, 기존 클라이언트-서버 구조에서는 전 직원의 로컬 클라이언트 메타데이터를 강제로 업데이트해야 하는 악몽이 펼쳐진다.

  1. 상시 최적화된 서빙(Always-Optimized Serving): MCP 템플릿 엔진은 요청 시점(Just-In-Time)에 서버 데이터베이스나 사내 위키에서 최신 가이드라인을 긁어모아 텍스트 트리를 렌더링한다.
  2. 클라이언트 배포면(Deployment Surface) 최소화: 클라이언트는 템플릿의 이름을 지칭하는 단순한 RPC 호출(prompts/get(name: "code_review", args: {"file": "main.py"}))만 쏠 뿐이다. 본문이 10바이트든 10만 바이트든, 클라이언트는 전혀 신경 쓰지 않은 채 그저 배달받아 LLM에게 넘겨주는 파이프스루(Pass-through) 역할로 전락한다.
  3. 지시문의 민첩성: 이를 통해 프롬프트 엔지니어는 클라이언트 애플리케이션의 배포(App Store Review 등) 족쇄를 완전히 끊어내고, 서버 측에서 초당 10번씩 지시문을 바꾸고 테스트하는 극단적 민첩성(Agility)을 획득한다.
