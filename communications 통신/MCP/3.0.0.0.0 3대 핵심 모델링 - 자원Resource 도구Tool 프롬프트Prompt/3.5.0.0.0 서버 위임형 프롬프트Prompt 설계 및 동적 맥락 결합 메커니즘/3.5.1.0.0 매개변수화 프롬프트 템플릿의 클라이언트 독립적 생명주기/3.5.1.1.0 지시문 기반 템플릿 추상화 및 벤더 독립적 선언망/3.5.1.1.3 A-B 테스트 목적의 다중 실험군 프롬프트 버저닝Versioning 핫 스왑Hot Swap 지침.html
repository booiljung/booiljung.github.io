3.5.1.1.3. A/B 테스트 목적의 다중 실험군 프롬프트 버저닝(Versioning) 핫 스왑(Hot Swap) 지침

어떤 프롬프트가 (예: “결론을 맨 아래 배치하라” vs “결론을 맨 위로 배치하라”) 더 좋은 추론 엔진(LLM)의 결과를 도출하는지는 직접 굴려보기 전까지는 프롬프트 엔지니어도 모른다.

  1. 가중치 기반 라우팅(Weighted Routing): MCP 서버의 prompts/get(name="debug") 엔드포인트 뒷단에는 A/B 테스트 스플리터(Splitter)가 배치되어 있다. 요청의 70%는 안정적인 v1.2 프롬프트를 내려주고, 나머지 30%의 세션에는 실험적인 v2.0 프롬프트를 핫 스왑(Hot Swap)하여 내려준다.
  2. 섀도우 버저닝 태그(Shadow Version Tag): 컨텍스트에 엮여 나가는 프롬프트 메시지의 메타데이터에는 몰래 헤더 리플렉터("prompt_version_applied": "2.0-beta")가 박혀 나간다.
  3. 품질 피드백 루프: 향후 에이전트가 생성한 결과물을 보고 사용자가 불만족(“다시 해”) 버튼을 누르면, 이 피드백 지표가 방금 배포한 v2.0 프롬프트의 실패율 벡터로 서버 측에 집계되어, 덜떨어진 프롬프트 실험군을 즉시 자동으로 도태(Kill)시키는 거대한 ML 파이프라인의 촉수로 기능한다.
