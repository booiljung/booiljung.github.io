3.5.1.4.3. 모델 종류별 맞춤형(Role/System) 맵핑 차이 마크업 언어 최적 변환 구역

에이전트를 구동하는 기저 LLM 모델이 GPT-4인지, Claude 3.5인지, LLaMA 3인지에 따라 최적의 지시문 형상(Syntax)은 완전히 달라진다. 어떤 모델은 마크다운 ### 헤더를 인지하지만, 어떤 모델은 <prompt> 형태를 좋아한다.

  1. 클라이언트 컨텍스트 협상(Context Negotiation): MCP 클라이언트가 프롬프트를 호출할 때 HTTP 헤더나 파라미터에 {"llm_client": "claude-3-opus"} 라는 부가 메타데이터를 넌지시 심어보낸다.
  2. 트랜스파일러(Transpiler) 개입: 브로커 내부에 위치한 AST 기반 마크업 언어 트랜스파일러가 원본 서버 템플릿(보통 표준 MD로 작성됨)을 읽어 들여 당해 LLM 벤더가 가장 좋아하는 최적의 토큰 배열 구조로 문법 번역(Syntax Translation)을 갈겨버린다.
  3. 폴리모픽(Polymorphic) 템플릿 결과물: 클라이언트(Claude Desktop 윈도우 앱, Cursor IDE, 혹은 터미널 CLI)가 어떤 모양새와 어떤 LLM을 들고 있든, 이 구역을 통과하며 융통성 있게 형상이 변이되므로, 서버의 템플릿 설계자는 단방향 다이렉트(Single-source-of-truth) 설계만 유지하면 되는 극도의 유지보수성을 뽐낸다.
