3.5.2.3.1. 특정 복잡한 프롬프트가 응답 실패/타임아웃을 야기할 시 단순화(Simplify)된 템플릿으로 우회하는 패턴

수천 줄의 코딩 컨벤션과 10개의 XML 태그가 결합된 v1_complex_review 프롬프트를 LLM에 던졌더니, 30초가 지나도록 504 Gateway Timeout이나 빈 응답이 돌아온다면 어떻게 해야 할까?

  1. 에러 후킹(Error Hooking): 실패를 감지한 MCP 브로커는 클라이언트에게 “에러 났으니 다시 해보라“고 말하지 않고 인터셉트한다.
  2. 단순화 템플릿(Simplified Template) 스왑: 브로커는 자체적으로 v1_complex_review 대신, 모든 부가적인 마크다운 표기와 잔가지를 쳐내고 오직 핵심만 남긴 v1_simple_review (Fallback Level 1)로 프롬프트의 살을 다이어트(Diet)시켜 재시도(Retry)를 날린다.
  3. 무정지 연속성(Seamless Continuity): 클라이언트 입장에서는 원래 3초 걸릴 응답이 10초 걸려 도착했을 뿐, 내부적으로 프롬프트가 가벼운 버전으로 교체되어 자동 복구되었다는 사실조차 모른 채 워크플로우를 이어간다.
