3.3.1.3.3. 병렬 호출 간 자원 락(Lock) 경합 방지를 위한 상태 격리(Isolation) 파이프라인

에이전트가 “A파일의 1번 라인을 지우는 도구“와 “A파일의 10번 라인을 지우는 도구“를 동시에 쏘아 보냈을 때, 동일 프로세스 내의 두 스레드가 하나의 물리 파일(A)을 잡고 아귀다툼을 벌인다면 어떻게 될까? 운영체제는 데드락(Deadlock)에 빠지거나 파일이 처참하게 깨질 것이다.

이를 방지하기 위한 것이 자원 락 기반 상태 격리(State Isolation) 파이프라인이다.

  1. 글로벌 리소스 레지스트리 뮤텍스(Mutex): 서버는 도구 실행이 건드리는 타겟(URI, DB Table 등)을 추출하여 글로벌 해시맵에 등록한다. 만약 두 도구가 동일한 타겟에 쓰기(Write) 권한을 요구한다면, 뒷순위의 도구가 슬립 스핀(Sleep Spin) 상태로 빠져 암묵적인 직렬화(Serialization)를 강제한다.
  2. 트랜잭션 격리 수준(Isolation Level): 데이터베이스와 연동하는 다중 도구 호출은, 단 하나의 묶음 트랜잭션 블록(BEGIN … COMMIT)으로 래핑된다. 하나라도 실패하면 앞서 실행된 병렬 도구의 변경 사항까지 몽땅 롤백(Rollback)시켜 데이터 무결성을 지켜낸다.
  3. 낙관적 동시성 제어(Optimistic Concurrency Control): 굳이 OS 레벨의 무거운 락을 걸지 않고, 각 도구가 작업 완료 후 파일의 버전 해시(Version Hash)를 제출하게 하여 병합 충돌이 나면 클라이언트 측에 ConflictError를 던져 스스로 병합을 해결하게 만드는 Git 방식의 우회론도 활용된다.
