3.3.1.2.3. 에이전트의 이전 버전 스키마 호출을 실시간 최신 버전으로 맵핑하는 가비지 필터링

서버 측에서 도구의 파라미터 스펙을 V1에서 V2로 변경했다 하더라도, 에이전트의 내부 컨텍스트 윈도우에는 이전에 오고 갔던 대화나 V1 시절의 툴 스키마 스냅샷이 그대로 남아 있을 수 있다. 이로 인해 에이전트가 고집스럽게 V1 규격의 페이로드를 조립해 날려대는 현상이 종종 발생한다.

이러한 버전 엇갈림(Version Desync) 현상을 매끄럽게 흡수하기 위한 가비지 필터링(Garbage Filtering) 아키텍처가 존재한다.

  1. 사용되지 않는 레거시 필드 소거(Pruning): 에이전트가 V1에 존재하던 legacy_flag: true 매개변수를 보내왔다면, V2 서버 패키지는 이 알 수 없는 필드 때문에 400 에러를 뿜는 엄격함(additionalProperties: false) 대신, 내부적인 툴 업그레이드 미들웨어를 통해 조용히 이 가비지 필드를 삭제(Prune)해 버린다.
  2. 동적 역 호환성 브릿지(Backport Bridge): V1의 user_id가 V2에서 userIdx로 카멜케이스(CamelCase) 네이밍 변경이 일어난 경우, 검사 계층은 JSON Payload를 넘겨받은 즉시 user_id 캐치를 확인하여 새로운 userIdx 구문으로 인메모리 스왑(Swap)을 치고 뒤로 흘려보낸다.
  3. 경고 시스템과 학습 유도: 에러를 일으키진 않았지만, 서버는 처리 결과 응답에 "warning": "Parameter 'user_id' is deprecated. Please update your tool schema usage."라는 시스템 시그널을 동봉하여, LLM이 자신의 내부 캐시를 파인 튜닝(Fine-tuning/Self-correction) 하도록 부드럽게 재교육시킨다.
