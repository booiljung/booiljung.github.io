3.3.2.1.3. 중복 실행(Idempotency) 방지를 위한 캐시된 식별자 해시 체킹망

에이전트가 “A 서버 재부팅“이라는 막중한 도구를 호출했는데, 네트워크 랙(Lag)으로 응답을 늦게 받자 “실패한 줄 알고 한 번 더 버튼을 누르는(Retry)” 사고가 날 수 있다. 이 경우 재부팅 스크립트가 2번 연속 맞물려 돌아가며 시스템 클러스터가 완전히 붕괴될 수 있다.

이를 방어하기 위해 MCP 도구 라우터에는 멱등성(Idempotency) 키 캐싱망이 존재한다.

  1. 지문(Fingerprint) 추출: 도구 호출 시 넘어온 매개변수들의 JSON 트리(예: {"server": "A", "action": "reboot"})를 정렬(Sort)한 뒤 SHA-256 해시 함수로 돌려 고유한 지문을 추출한다.
  2. 단기 인메모리 락(Short-lived Lock): 이 지문이 인메모리 캐시(Redis 등)에 존재하지 않으면 새로운 작업으로 간주하여 실행하고, 지문과 함께 Tracking ID를 30초간 캐싱한다.
  3. 재시도 쳐내기(Bounce Back): 만일 10초 뒤 에이전트가 똑같은 매개변수로 도구를 재호출하면 지문이 일치한다. 이때 서버는 스크립트를 재실행하는 대신, 캐싱해 두었던 기발급 Tracking ID를 그대로 반환한다. 즉, “이미 진행 중이니 기다려라“라며 무거운 백그라운드 작업의 중복 격발을 완벽하게 틀어막는다.
