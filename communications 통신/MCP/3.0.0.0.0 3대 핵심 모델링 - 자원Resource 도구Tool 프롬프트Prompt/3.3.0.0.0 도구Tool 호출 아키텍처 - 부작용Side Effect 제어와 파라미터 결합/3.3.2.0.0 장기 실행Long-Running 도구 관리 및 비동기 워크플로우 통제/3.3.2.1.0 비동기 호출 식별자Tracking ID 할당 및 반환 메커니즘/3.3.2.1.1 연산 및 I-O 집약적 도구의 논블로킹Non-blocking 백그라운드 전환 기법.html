3.3.2.1.1. 연산 및 I/O 집약적 도구의 논블로킹(Non-blocking) 백그라운드 전환 기법

모든 도구를 비동기로 돌릴 필요는 없다. 날씨 조회처럼 100ms 이내에 끝나는 가벼운 도구마저 이벤트 루프에 넣었다 꺼내면 통신 오버헤드가 더 커진다.

  1. 자동화된 비용 임계치(Cost Threshold) 판별: 서버는 tools/call 요청이 들어오면 도구의 메타데이터(예: @LongRunning)를 확인하거나, 혹은 역사적 평균 실행 시간(Historical Avg. Latency) 데이터베이스를 조회하여 해당 도구가 블로킹(Blocking)을 유발할 작업인지 논블로킹(Non-blocking) 백그라운드로 전환해야 할 작업인지 동적으로 분류한다.
  2. 이벤트 루프의 오프로딩(Off-loading): 백그라운드 대상을 식별한 즉시 서버 측 Node.js나 Python asyncio 메인 이벤트 루프는 이 작업을 워커 스레드(Worker Thread)나 외부 Celery/RabbitMQ 메세지 브로커로 조용히 위임(Delegate)하여 I/O 블로킹을 칼같이 차단한다.
  3. LLM 사고(Reasoning) 스레드의 단절 부활: 메인 루프를 점유하지 않으므로 에이전트는 즉각적으로 “작업이 백그라운드에서 시작되었다“는 응답을 받고, 기다리는 동안 다른 추론 작업(예: 사용자에게 현황 설명)을 병행할 수 있는 다중 스레딩적 사고 능력을 확보하게 된다.
