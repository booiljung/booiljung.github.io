3.4.1.4.2. 실행 후 데이터 파괴 시 원복할 수 있는 임시 스냅샷(Shadow Copy) 체제 구축법

“A 파일을 B의 내용으로 바꿔줘.” 에이전트가 이 작업을 충실히 이행했는데, 막상 인간 사용자가 보니 코드가 완전히 박살 나(Broken) 구동되지 않는다. 이럴 때 단순한 Auditing만으로는 부족하다. 망가진 파일을 즉각 복구할 Undo 버튼이 필요하다.

  1. Before-Mutation 트리거: 도구 내부에 fs.writeFile이나 git commit 처럼 상태를 파괴적으로 덮어쓰는(Destructive Write) 트리거가 당겨지면, 시스템은 I/O를 잠시 멈춘다.
  2. 단기 섀도우 카피(Short-term Shadow Copy): 시스템은 원본 텍스트 파일이나 DB 레코드의 상태를 슬쩍 떠서, 버전 관리 시스템(.mcp_history)이나 S3 임시 버킷에 타임스탬프와 함께 직렬화 보관한다.
  3. 타임 투 리브(TTL) 기반 증발: 이러한 섀도우 카피가 무한정 쌓이면 디스크를 마비시키므로, 파일 용량 대비 중요도를 해싱하여 보통 24시간~72시간 내의 TTL(Time-To-Live) 타이머를 걸어두고, 사용자가 명시적인 복원(Restore) 액션을 취하지 않으면 자연스레 소멸하도록 배압을 제어한다.
