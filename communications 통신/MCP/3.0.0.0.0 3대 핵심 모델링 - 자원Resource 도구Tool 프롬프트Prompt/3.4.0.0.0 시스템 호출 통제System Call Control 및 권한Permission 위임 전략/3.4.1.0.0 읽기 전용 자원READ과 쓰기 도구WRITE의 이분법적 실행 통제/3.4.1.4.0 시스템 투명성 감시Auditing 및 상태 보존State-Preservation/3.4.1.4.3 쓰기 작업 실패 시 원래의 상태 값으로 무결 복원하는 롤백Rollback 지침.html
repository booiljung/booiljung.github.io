3.4.1.4.3. 쓰기 작업 실패 시 원래의 상태 값으로 무결 복원하는 롤백(Rollback) 지침

만일 에이전트가 세 개의 파일 A, B, C를 연쇄적으로 수정하는 도구를 호출했는데, A와 B를 고친 직후 서버에 일시적 네트워크 장애가 발생하여 C 파일을 고치지 못한 채 워커가 뻗어버렸다면(Crashing) 어떻게 될까? 시스템은 완전히 일관성(Consistency)을 잃게 된다.

  1. 원자적 단위(Atomic Unit) 격상: 단일 도구 호출 내에서 벌어지는 모든 변경 사항은 아무리 여러 폴더를 건드리더라도 하나의 논리적, 원자적 트랜잭션으로 취급되어야 한다.
  2. 더티 스테이트 캐칭(Dirty State Catching): 에러가 내뿜어져서 스레드가 죽을 때, 서버의 전역 Exception 데몬은 현재 도구가 COMMIT을 날렸는지 확인한다. 만약 커밋되지 않았다면 즉시 “더티 스테이트(Dirty State)” 경보를 울린다.
  3. 리버스 오퍼레이션(Reverse Operation) 컴파일: 위 절에서 언급한 스냅샷을 기반으로 역연산(예: 추가한 텍스트는 지우고, 변경한 파일은 원본으로 되돌리는 스크립트)을 자동 렌더링 하여 즉각 롤백(Rollback)을 수행한다. 결국 세계는 에이전트가 도구를 호출하기 전 매끈한 평상 상태(Zero State)로 100% 무결 회귀한다.
