3.4.1.1.3. 쓰기 동작으로 위장된 파괴적 읽기(e.g., Lock 획득 후 읽기) 쿼리를 분별하는 패킷 정적 분석기

데이터베이스의 SELECT 문은 전형적인 ‘읽기(Read)’ 요청이다. 그렇다면 에이전트가 SELECT * FROM users FOR UPDATE 라는 쿼리를 던지면 1번 파이프라인(가속/무제한)으로 흘려보내도 안전한 걸까?

답은 ’아니오’다. 저 쿼리는 테이블에 베타적 락(Exclusive Lock)을 거는 파괴적 부작용이 동반된 읽기다.

  1. AST / Query Parser 주입: 고급 MCP 어댑터는 인텐트를 단어(String)의 패턴 매칭에만 기대지 않는다. 내부적으로 SQL이나 쉘 텍스트를 추상 구문 트리(AST)로 파싱해 내는 정적 분석기를 돌린다.
  2. 히든 리스크 탐지(Hidden Risk Detection): 그 결과 단순 READ처럼 보일지라도 내부에 타임스탬프를 갱신하거나, 세션 로그를 남기거나, 락(FOR UPDATE, LOCK IN SHARE MODE)을 획득하는 부작용(Side Effect)이 단 한 톨이라도 감지되면, 이를 무자비하게 2번 파이프라인(쓰기/검열) 궤도로 강제 승격(Promote)시켜 버린다.
  3. 스머프(Smurf) 공격 차단: 에이전트가 고의든 아니든 읽기 핑자체로 시스템 자원을 무한 탈취하는 것을 사전에 방지하여, 단순 정보 탐색이라는 미명 하에 자행되는 구조적 파탄을 걸러내는 마지막 그물 역할을 수행한다.
