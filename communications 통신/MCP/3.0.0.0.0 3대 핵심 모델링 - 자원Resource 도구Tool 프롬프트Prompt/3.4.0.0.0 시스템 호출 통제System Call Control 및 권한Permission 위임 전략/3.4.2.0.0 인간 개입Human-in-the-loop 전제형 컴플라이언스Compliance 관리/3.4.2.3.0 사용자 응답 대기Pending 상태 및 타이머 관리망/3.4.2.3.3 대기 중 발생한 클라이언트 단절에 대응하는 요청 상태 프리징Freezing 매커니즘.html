3.4.2.3.3. 대기 중 발생한 클라이언트 단절에 대응하는 요청 상태 프리징(Freezing) 매커니즘

모바일 기기에서 MCP 클라이언트를 켜두고 결제 승인을 누르려는 찰나, 지하철 터널에 들어가 네트워크(TCP)가 뚝 끊어졌다. 서버의 펜딩 큐에 떠있는 요청은 어떻게 될까?

  1. 하트비트 연결 상실 감지: 서버 측 웹소켓(WebSocket) 감시자는 클라이언트와의 하트비트(Ping-Pong)가 3회 이상 단절되었다면 이를 ’물리적 연결 소실(Lost Connection)’로 판단한다.
  2. 세션 프리징(Session Freezing): 이 순간, 기존의 TTL 타이머(예: 15분)는 작동을 멈추고 서버는 해당 펜딩 요청을 반영구적인 상태 프리징 볼트(Freezing Vault)로 이동시켜 얼려버린다(Freeze). 네트워크 단절 때문에 억울하게 오퍼레이션이 날아가는 것을 방지하기 위함이다.
  3. 재개(Resume) 및 재동기화(Resync): 10분 뒤 터널을 빠져나와 클라이언트가 다시 연결을 맺고 Session ID를 인증하면, 곧시 프리징 볼트에 얼어있던 승인 창 다이얼로그 패킷이 클라이언트로 다시 푸시(Re-Push)되어 워크플로우를 완벽한 단절 전 상태로 복구(Graceful Recovery)시킨다.
