3.1.2.3.2. 방대한 쿼리 결과 반환 방지를 위한 자동 페이지네이션(Pagination) 및 레코드 리미트 부착

에이전트가 단순히 “모든 상품 목록을 가져와“라고 SELECT * FROM products를 날렸고 상품이 100만 개라면, 데이터베이스는 100만 행을 메모리에 퍼올리고 서버는 파멸적인 1GB 크기의 JSON을 만들어 MCP 채널을 향해 쏘아 보낼 것이다.

이를 원천적으로 틀어막기 위해, 쿼리 파이프라인에는 무조건적 자동 리미트 부착 장치가 탑재되어 있다.

  1. 상시적 후행 제어 구문(Trailing Clause) 패치: 에이전트의 쿼리가 AST 상에서 무결하다고 판명되어도, 서버의 쿼리 조작기(Query Mutator) 계층은 그 쿼리의 제일 꼬리에 강압적으로 LIMIT 100을 끼워 넣는다.
  2. 에이전트의 컨텍스트 창(Context Window) 방어: 이 100개의 레코드는 LLM이 단일 추론 턴(Turn) 내에서 읽어내기에 부족함이 없으며, 비용 초과 현상을 철저히 봉쇄한다.
  3. Offset 주입을 통한 페이징(Paging): 에이전트는 리미트가 걸려 잘려나간 다음 100개의 데이터셋이 필요할 경우, page: 2 파라미터를 던진다. 그러면 내부적으로는 LIMIT 100 OFFSET 100 문법이 전개되어 방대한 데이터베이스를 “LLM이 한입에 씹어 삼킬 수 있는 조각“으로 지속적으로 떠먹여 주는 페이지네이션이 실현된다.
