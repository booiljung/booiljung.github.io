3.1.2.3.3. 인메모리 커서(Cursor)를 유지하여 대규모 결과셋을 스트림하는 반복 읽기(Iterative Fetch) 도구화

단순한 LIMIT/OFFSET 페이지네이션은 무거운 RDBMS에서 수만 번째 행을 읽기 위해 앞의 데이터를 모두 스캔해야 하는 치명적인 ’Offset 성능 저하(Offset Degradation)’를 야기한다.

이를 극복하기 위해, 대규모 로그 분석이나 데이터 마이그레이션을 돕는 에이전트에게는 인메모리 커서 기반 반복 읽기(Iterative Fetch from Cursor) 아키텍처가 제공된다.

  1. 서버 사이드 커서 선언(Declare Cursor): 에이전트가 최초 쿼리를 지시하면, 서버는 데이터를 가져오지 않고 데이터베이스 엔진 내에 DECLARE agent_cursor CURSOR FOR SELECT... 구문으로 결과를 홀드(Hold)할 가상의 책갈피(Cursor)만을 생성한다.
  2. 세션 바인딩(Session Bound) 커서 징표 보관: 서버는 이 커서의 식별자 ID를 MCP 세션 메모리에 바인딩하고, 클라이언트에게 "cursorId": "a1b2", "hasMore": true 형태의 징표를 건넨다.
  3. 결정론적 반복 호출(Deterministic Fetch): 에이전트는 데이터가 더 필요할 때마다 새로운 쿼리를 치는 것이 아니라, 제공받은 도구(예: fetch_next_batch)에 cursorId를 첨부하여 호출한다. 데이터베이스는 Offset 연산 없이 직구로 다음 100행을 빠르게 토해내며, 통신 오버헤드와 DB 부하를 동시에 극한으로 압축시키는 이점을 얻는다.
