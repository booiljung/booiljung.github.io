3.1.1.1.2. 다중 워크스페이스(Workspace) 환경에서 명명 충돌(Naming Collision)을 제어하는 격리 모델

단일 서버가 여러 개의 서로 다른 프로젝트(워크스페이스)를 동시에 서비스할 때, 각각의 워크스페이스에는 동일한 이름의 파일(예: config.json, main.py)이 존재할 가능성이 높다. 단순히 파일 이름이나 상대 경로만으로 URI를 매핑하면, 에이전트가 “어떤 워크스페이스의 main.py인지” 알 수 없는 심각한 명명 충돌(Naming Collision)이 발생한다.

MCP 설계는 이를 방지하기 위해 URI의 호스트(Host) 속성이나 첫 번째 경로(Path) 블록을 철저한 격리된 네임스페이스 영역으로 강제한다.

  1. 테넌트(Tenant) 및 워크스페이스 키의 해시화: 각 워크스페이스가 서버에 등록될 때, 고유한 UUID 빌드(예: ws-8f92a)를 부여받는다.
  2. 복합적 URI 라우팅 공간: 파일의 URI는 자연스럽게 mcp://ws-8f92a/src/main.py와 같은 방식으로 할당되며, 디렉터리 구조 이전에 워크스페이스 키가 라우팅을 선행(Pre-routing)하여 파일 간의 논리적 충돌을 차단한다.
  3. 컨텍스트 분리 장벽(Context Barrier): 이는 데이터 결합 오류를 막아줄 뿐만 아니라, 특정 세션 토큰을 가진 에이전트가 자신의 워크스페이스 바운더리를 넘어서 다른 워크스페이스의 URI를 직접 조합하여 호출(Path Traversal)하는 해킹 시도까지 원천적으로 잘라내는 다중 테넌트(Multi-tenant) 보호 체계의 척추 역할을 수행한다.
