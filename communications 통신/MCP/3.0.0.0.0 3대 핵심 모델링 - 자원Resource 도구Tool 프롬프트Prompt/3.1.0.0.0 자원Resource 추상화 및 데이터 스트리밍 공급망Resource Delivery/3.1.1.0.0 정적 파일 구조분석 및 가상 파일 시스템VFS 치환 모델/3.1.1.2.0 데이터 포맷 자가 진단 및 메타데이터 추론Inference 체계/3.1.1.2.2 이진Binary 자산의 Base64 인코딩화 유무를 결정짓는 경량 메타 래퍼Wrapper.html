3.1.1.2.2. 이진(Binary) 자산의 Base64 인코딩화 유무를 결정짓는 경량 메타 래퍼(Wrapper)

에이전트는 단순히 텍스트만 읽는 것이 아니라 다이어그램, 스크린샷 등 이미지 데이터도 읽어 들일 수 있어야 한다. 하지만 JSON-RPC라는 순수한 문자기반(String-based) 프로토콜 골격 위에서는 0과 1로 이루어진 날것의 바이너리(Raw Binary) 바이트를 직접 전송할 수 없다. JSON 파서가 붕괴되기 때문이다.

MCP 서버는 앞선 3.1.1.2.1 절에서 판별된 MIME-Type을 바탕으로, 해당 리소스에 **인코딩 래핑(Encoding Wrapping)**을 씌워야 할지 결정한다.

  1. 바이너리 갈림길(Forking) 판별: 추론된 MIME-Type이 text/* 계열이거나 application/json 등 명확한 아스키(ASCII)/UTF-8 텍스트 영역이라면 내용물을 그대로 text 필드에 담는다. 그러나 image/*, application/octet-stream 등 이진 계열이라면 전송 채널을 우회시킨다.
  2. Base64 치환 래핑: 바이너리 데이터는 안전한 전송을 위해 Base64 인코딩이라는 값비싼 연산을 거쳐 순수한 영어 알파벳 배열로 변환된 후, text 필드가 아닌 가상 자산의 전용 공간인 blob 필드에 담긴다.
  3. 클라이언트의 무결절 디코딩: 에이전트는 응답 패킷에서 text 필드가 비어있고 blob 필드에 값이 들어있는 구조(Polymorphic Type)를 확인하면, 이를 다시 역(逆) Base64 디코딩하여 자신의 비전(Vision) 분석 모델이나 이미지 렌더러로 투명하게 넘기는 메타 래퍼 아키텍처가 구동된다.
