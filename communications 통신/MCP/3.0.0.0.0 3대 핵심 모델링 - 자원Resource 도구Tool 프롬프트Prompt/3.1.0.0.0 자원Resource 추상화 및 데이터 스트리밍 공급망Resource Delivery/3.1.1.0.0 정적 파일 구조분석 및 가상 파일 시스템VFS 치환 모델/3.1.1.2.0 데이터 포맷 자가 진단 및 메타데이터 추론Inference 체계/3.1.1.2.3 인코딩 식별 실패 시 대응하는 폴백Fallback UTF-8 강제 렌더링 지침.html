3.1.1.2.3. 인코딩 식별 실패 시 대응하는 폴백(Fallback) UTF-8 강제 렌더링 지침

매직 넘버를 뒤져도, 확장자를 확인해도 도저히 알 수 없는 정체불명의 파일(예: 확장자 없는 덤프 파일, 특이한 포맷의 개인 로그)이 존재할 수 있다.

데이터를 함부로 폐기하거나 무조건 바이너리로 묶어버리면 유용한 텍스트를 LLM에 전달할 소중한 기회를 영영 잃게 되므로, MCP 서버는 최후의 수단으로 UTF-8 강제 렌더링 폴백(Fallback) 지침을 수행한다.

  1. 휴리스틱(Heuristics) 텍스트 스캔: 서버는 파일 내용 중 첫 512바이트 구간을 순회하여, 널(Null, \0) 바이트 비율이나 인쇄 불가능한(Non-printable) 제어 문자의 빈도가 임계치를 넘는지 휴리스틱하게 판별한다.
  2. UTF-8 강제 우겨넣기 시도: 만약 텍스트일 가능성이 ’조금’이라도 있다고 판단되면, 서버는 용감하게 content-type을 text/plain으로 선언하고 이 바이트들을 UTF-8 인코딩 스키마를 통해 문자열로 치환한다.
  3. 치환 문자(Replacement Character) 정제 방어선: 이 강제 변환 과정에서 실제 UTF-8 규격에 맞지 않는 불량 바이트가 등장하면, 파서가 멈추지 않도록 해당 바이트를 무시어(Ignore) 처리하거나 유니코드 치환 문자(``, U+FFFD)로 덮어씌운다. 비록 일부 글자가 깨질지언정 에이전트에 주변 컨텍스트 문자들은 전달하여 의미론적 추론 기회를 조금이라도 살려내는 강전성(Resilience)을 확보한다.
