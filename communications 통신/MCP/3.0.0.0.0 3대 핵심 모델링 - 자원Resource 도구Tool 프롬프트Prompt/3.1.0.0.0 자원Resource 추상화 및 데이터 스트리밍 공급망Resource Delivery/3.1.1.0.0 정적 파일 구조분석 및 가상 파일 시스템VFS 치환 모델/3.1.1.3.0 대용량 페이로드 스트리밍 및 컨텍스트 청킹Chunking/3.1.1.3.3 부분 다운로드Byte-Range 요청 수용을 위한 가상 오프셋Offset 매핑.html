3.1.1.3.3. 부분 다운로드(Byte-Range) 요청 수용을 위한 가상 오프셋(Offset) 매핑

경우에 따라서 에이전트는 파일의 앞부분 10만 줄은 무시하고, 에러가 발생했다고 판단되는 “정확히 15만 번째 줄“부터만 데이터를 꺼내보고 싶어 할 수 있다. 이때 서버가 무식하게 1번째 바이트부터 전송하는 것은 엄청난 컴퓨팅 낭비다.

마치 HTTP의 Range 헤더 요청처럼, MCP 자원 열람 프로토콜 역시 커스텀 페이로드를 통해 부분 바이트/라인 오프셋 수용 기법을 처리한다.

  1. 파라미터 기반 가상 오프셋 요청: 클라이언트는 도구를 쿼리하듯 resources/read를 호출할 때 start_offset: 154000, end_offset: 156000 (혹은 라인 넘버 기반) 와 같은 부분 명세(Range Spec)를 함께 발송할 수 있다.
  2. 커널 레벨의 seek() 및 추출: 요청을 받은 서버는 거대한 파일을 램으로 퍼올리는 불상사를 피하기 위해, 운영체제의 저수준 파일 seek() 시스템 콜을 통해 디스크 포인터를 정확히 start_offset 위치로 날아간다.
  3. 가상화된 상대 좌표계 반환: 추출된 데이터 청크를 클라이언트에게 내어줄 때, 서버는 이 데이터가 “전체 인덱스 상 몇 번째 좌표에서 떨어져 나온 부분 데이터“인지에 대한 메타데이터 맵을 함께 반환한다. 이를 통해 클라이언트는 자신이 이 파일의 허리를 들여다보고 있는지, 꼬리를 들여다보고 있는지 정확한 상대적 컨텍스트를 지각할 수 있다.
