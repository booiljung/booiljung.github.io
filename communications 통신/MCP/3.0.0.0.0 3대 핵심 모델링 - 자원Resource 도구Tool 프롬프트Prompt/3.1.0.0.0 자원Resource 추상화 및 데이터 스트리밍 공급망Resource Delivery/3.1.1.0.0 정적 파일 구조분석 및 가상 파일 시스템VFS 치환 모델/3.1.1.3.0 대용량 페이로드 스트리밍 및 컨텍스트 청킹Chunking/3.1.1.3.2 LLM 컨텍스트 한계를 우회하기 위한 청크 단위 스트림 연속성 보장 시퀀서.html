3.1.1.3.2. LLM 컨텍스트 한계를 우회하기 위한 청크 단위 스트림 연속성 보장 시퀀서

서버 측에서 아무리 데이터를 예쁘게 잘라서 주더라도, 클라이언트 측에서 청크들이 순서가 뒤죽박죽된 채로 LLM에게 던져지면 대재앙이 발생한다. 특히 비동기 다중 전송 모델을 활용하는 고성능 에이전트라면 패킷 순서의 역전 징후가 언제든 도사린다.

MCP는 쪼개진 자원 스트림을 완벽한 순서로 재조립하기 위한 연속성 보장 시퀀서(Sequencer) 스펙을 정의한다.

  1. 엄격한 시퀀스 넘버(Sequence ID) 부여: 거대한 resources/read 응답이 여러 차례의 알림이나 푸시로 나뉘어 전송될 때, 각 페이로드에는 반드시 chunk_id 또는 sequence_number가 (0, 1, 2…) 순차적으로 기입된다.
  2. 클라이언트 사이드 재조립 버퍼: 클라이언트 에이전트는 곧장 들어오는 청크를 LLM에 쑤셔 넣지 않는다. id=0을 처리하는 동안, 네트워크 지연으로 먼저 도착한 id=2번 청크는 인메모리 대기 버퍼 큐에 잠시 홀드(Hold)시켜 둔다.
  3. 종결자(Terminator) 패킷 선언: 마지막 패킷에는 is_last=true 혹은 eof(End Of File) 마커를 삽입하여, 클라이언트의 시퀀서 라우터가 최종 합본 렌더링을 끝마치고 “원문 데이터 독해를 완료했다“라고 LLM 프롬프트에게 상태 전이를 보고할 수 있도록 돕는다.
