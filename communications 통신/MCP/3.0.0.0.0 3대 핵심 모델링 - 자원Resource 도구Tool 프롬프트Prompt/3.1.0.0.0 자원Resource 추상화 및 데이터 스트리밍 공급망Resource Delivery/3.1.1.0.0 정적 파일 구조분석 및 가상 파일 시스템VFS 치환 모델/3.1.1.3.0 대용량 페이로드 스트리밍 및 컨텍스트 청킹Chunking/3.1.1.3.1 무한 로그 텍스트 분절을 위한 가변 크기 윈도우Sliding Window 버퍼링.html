3.1.1.3.1. 무한 로그 텍스트 분절을 위한 가변 크기 윈도우(Sliding Window) 버퍼링

서버의 실행 로그 파일처럼 끝을 알 수 없고 계속해서 자라나는 성질의 데이터를 에이전트에게 공급할 때는, 데이터를 고정된 크기(예: 무조건 1MB씩)로 기계적으로 썰어버리는 방식이 위험하다. 단어 한가운데나 핵심 에러 트레이스(Trace) 스택이 중간에서 반동강이 나면, LLM은 앞뒤 문맥을 이어붙이지 못하고 환각을 일으킨다.

이를 방지하기 위해 텍스트 리소스 스트링 엔진에는 가변 크기의 슬라이딩 윈도우(Sliding Window) 버퍼링 알고리즘이 적용된다.

  1. 문맥 탐색 기반의 유연한 자르기: 서버는 1차적으로 1MB 단위의 청크(Chunk)를 잡되, 절단면 근처에서 개행 문자(\n)나 스페이스 등 의미론적 단락이 나뉘는 델리미터(Delimiter)를 역으로 탐색(Look-behind)한다.
  2. 청크 사이즈의 탄력성: 만약 정확히 1MB 위치에서 단어가 끊긴다면, 윈도우 스팬(Span)을 약간 줄이거나 늘려 온전한 문장이 완성되는 지점에서 정확히 분절을 일으킨다.
  3. 중첩 윈도우 보장(Overlapping Spans): 어떤 경우에는 이전 청크의 맨 마지막 몇 줄을 다음 청크의 시작 부분에 의도적으로 중첩(Overlap)시켜 전송한다. 이를 통해 에이전트는 분절된 두 텍스트 덩어리를 받아도 문맥의 끈을 놓치지 않고 매끄러운 코드 분석을 수행할 수 있게 된다.
