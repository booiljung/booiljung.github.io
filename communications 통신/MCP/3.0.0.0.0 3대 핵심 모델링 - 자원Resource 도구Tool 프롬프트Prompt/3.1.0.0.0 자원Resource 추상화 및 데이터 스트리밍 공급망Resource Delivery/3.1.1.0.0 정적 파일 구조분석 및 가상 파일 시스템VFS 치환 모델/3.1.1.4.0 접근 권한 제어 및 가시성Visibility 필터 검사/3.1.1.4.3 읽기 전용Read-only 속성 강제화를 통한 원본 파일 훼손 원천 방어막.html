3.1.1.4.3. 읽기 전용(Read-only) 속성 강제화를 통한 원본 파일 훼손 원천 방어막

자원(Resource)이라는 명명(Naming) 자체가 시사하듯, MCP 아키텍처 내에서 Resource 모델은 오직 데이터를 소비(Consume)하는 대상일 뿐 수정을 가하는 대상이 아니다. (파일 수정 행위는 Tool 모델을 통해 별도로 통제된다.)

그러나 서버 로직의 버그나 악의적인 조작으로 인해 에이전트가 파일 스트림 소켓에 역으로 쓰기 버퍼(Write-buffer)를 밀어 넣으려 시도할 수 있다. 이를 막는 것이 읽기 전용 강제화 방어막이다.

  1. 커널 레벨 핸들 제어: 서버가 물리적 디스크에서 파일을 열어(Open) 클라이언트 스트림과 결합시킬 때, 파일 디스크립터(FD)는 반드시 O_RDONLY(Read-Only) 플래그로만 인스턴스화된다.
  2. 추상화 계층에서의 Setter 제거: 객체 지향적으로 구성된 Resource 클래스나 VFS 인터페이스에는 데이터 쓰기, 삭제, 이동과 관련된 어떠한 메소드(Method/Setter)도 구현되어 있지 않다. 컴파일 레벨에서부터 수정 행위를 원천 배제한다.
  3. 불변(Immutable) 프로토콜 스펙: JSON-RPC 인터페이스 명세 상 resources/read, resources/list, 알림 목적인 notifications/resources/updated만이 존재할 뿐, resources/write 또는 resources/update 같은 엔드포인트 자체를 스펙상에서 영구적으로 소거함으로써 프로토콜의 의미론적 무결성(Semantic Integrity)을 지켜낸다.
