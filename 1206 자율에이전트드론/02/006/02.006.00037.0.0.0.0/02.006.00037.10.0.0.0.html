37.10 자율 주행 궤적 생성 주기 런타임 동적 조정을 위한 파라미터 이벤트 트리거 아키텍처

자율 주행 드론의 로컬 플래너(Local Planner)는 장애물 회피와 목표점 도달을 위해 초당 수회에서 수십 회에 걸쳐 다항식 궤적(Polynomial Trajectory)을 샘플링하고 최적화한다. 이 궤적 생성 주기(Trajectory Generation Period)는 드론의 순항 속도(Cruising Speed), 컴퓨팅 전력 제한, 그리고 주변 장애물의 밀도에 따라 런타임(Runtime)에 동적으로 가변해야 하는 핵심 파라미터이다. 기존 ROS1 시스템에서는 이러한 주기 제어 타이머(Timer)를 재설정하기 위해 노드 라이프사이클(Lifecycle) 전체를 재기동해야 했으나, ROS2는 파라미터 이벤트 콜백 구조를 활용하여 연산 주기를 밀리초 단위로 핫스왑(Hot-swap)할 수 있는 아키텍처를 제공한다. 본 절에서는 WallTimer 객체와 파라미터 이벤트 콜백을 융합하여 제어 주기를 동적 리볼빙(Dynamic Revolving)하는 타이머 트리거(Timer Trigger) 아키텍처를 분석한다.

1. 상태 의존적 주파수 지배 방정식(State-dependent Frequency Governing Equation)

드론의 경로 최적화 연산 주기(T_{opt})는 배터리 소모량과 제어 추종성 간의 물리적 트레이드오프(Trade-off)를 갖는다.

1.1 고정 주파수 제어의 한계

만약 드론이 아무런 장애물이 없는 개활지(Open Area)를 비행함에도 불구하고, 조밀한 숲속 비행을 가정하여 설정된 50\text{Hz}(T_{opt} = 20\text{ms})로 궤적을 갱신한다면, 이는 엣지(Edge) 컴퓨터의 불필요한 발열과 전력 낭비를 초래한다.
따라서 최적 궤적 생성 임무를 수행하는 컴포지션 노드는 optimization_frequency 파라미터를 개방해 두고, 관제소나 행동 트리(Behavior Tree) 노드가 외부에서 현재 환경의 복잡도 맵(Complexity Map)을 기반으로 주파수를 5\text{Hz}에서 50\text{Hz} 사이로 유연하게 제어할 수 있는 API를 확보해야 한다.

2. rclcpp::TimerBase 객체의 인-메모리 재설정(In-Memory Reset) 메커니즘

ROS2에서 주기적인 제어 루프는 create_wall_timer() 팩토리 메서드로 발현되는 TimerBase 스마트 포인터에 의해 구동된다. 이 타이머의 주기를 런타임에 동적으로 변경하기 위해서는 객체 지향적 접근이 요구된다.

2.1 파라미터 콜백과의 유기적 연결

파라미터 이벤트 콜백 add_on_set_parameters_callback 내부에 optimization_frequency 변형이 감지되면, 콜백 핸들러는 다음과 같은 위상 로직(Topological Logic)을 전개한다.

if (param.get_name() == "optimization_frequency") {
    double new_freq = param.as_double();
    // 1. 기존 타이머 작동 일시 중지
    if (this->optimization_timer_) {
        this->optimization_timer_->cancel();
    }
    // 2. 주파수 역산 및 신규 타이머 클럭 할당
    auto new_period = std::chrono::duration<double>(1.0 / new_freq);
    
    // 3. 타이머 코어 재설정 트리거 (Timer Reset)
    this->optimization_timer_ = this->create_wall_timer(
        std::chrono::duration_cast<std::chrono::nanoseconds>(new_period),
        std::bind(&PathPlannerNode::optimize_trajectory_loop, this)
    );
    // ...
}

여기서 핵심은 타이머 객체(optimization_timer_)가 파괴되었다가 재생성되는 과정이 파라미터 이벤트 콜백의 컨텍스트(Context) 내부에서 원자적으로 처리된다는 점이다. cancel() 호출 이후 미들웨어 큐(Queue)에 잔존할 초과 인터럽트를 완전히 비우고, 새로운 주기(\frac{1}{f}) 단위의 폴링 에포크(Epoch)를 덮어씌운다. 이러한 메커니즘은 로컬 플래너 노드를 셧다운(Shutdown)하지 않고도 실시간 스케줄러(Scheduler)의 활성 인터벌을 재봉합하는 ROS2만의 진보된 상태 기계 재진입(State Machine Reentry) 패턴이다.

3. 스케줄러 오버런(Overrun)과 데드라인(Deadline) 방어

주파수가 외부로부터 매우 높게(예: 100\text{Hz}) 주입될 경우, 드론 컴퓨터의 실제 연산 런타임 능력을 상회하게 되어 타이머 콜백이 이전 주기의 완료 전에 재귀적으로 호출을 시도하는 오버런(Overrun) 지연 상태가 촉발될 수 있다.

이를 제어하기 위해, add_on_set_parameters_callback 상단의 방어 계층에서는 단순한 [5.0, 100.0] 범위 검사뿐만 아니라, 가장 최근 10회의 궤적 연산에 소요된 평균 수행 시간(t_{mean\_execution}) 데이터를 실시간 모니터링해야 한다.
만약 t_{mean\_execution} = 15\text{ms} 인 상태에서 외부 클라이언트가 optimization_frequency = 100Hz (요구 주기 10\text{ms})로 파라미터를 변경하려 시도한다면, 콜백 로직은 하드웨어 연산 능력의 임계 한계를 감지하고 successful = false를 리턴하며 이 무모한 갱신 요동(Fluctuation)을 전면 거부(Veto)해야 한다.
이처럼 타이머 동적 재구성에 결합된 파라미터 이벤트 트리거 아키텍처는, 단순히 변수값을 교체하는 수동적인 통로가 아니라 시스템의 컴파일된 퍼포먼스 마진(Performance Margin)을 스스로 측량하고 물리적 다운타임(Downtime)을 예방하는 능동적 자기 방어 방화벽(Active Self-Defense Firewall)으로서 그 가치가 입증된다.
