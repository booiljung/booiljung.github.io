Chapter 36. 런치 파일 내 파라미터(Parameter) 선언 및 YAML 기반 외부 파일 동적 로딩

로보틱스 미들웨어 인프라에서 하드웨어 드라이버나 복잡한 SLAM, 제어 알고리즘은 본질적으로 다양한 튜닝 계수(Tuning Coefficient)와 시스템 식별자(Identifier)를 요구한다. 코드베이스(Codebase)에 이러한 변수들을 하드코딩(Hard-coding)하는 방식은 재사용성을 심각하게 파괴하며, 다양한 기체 모델이나 환경 조건에 대응하기 어렵게 만든다. ROS2는 이러한 시스템 의존적 변수들을 런타임에 동적으로 주입하고 관리할 수 있도록 고도화된 파라미터(Parameter) 프레임워크를 제공한다. 본 장에서는 시스템의 제어 흐름을 결정짓는 설정 파일들의 구조적 특징과, 런치 파이프라인(Launch Pipeline) 단계에서 이를 선언적으로 로딩하는 메커니즘을 상세히 규명한다.

1. ROS2 파라미터 프레임워크의 계층적 구조

ROS1의 글로벌 파라미터 서버(Global Parameter Server) 방식은 노드 간 결합도(Coupling)를 높이고 분산 통신 환경에서의 상태 불일치(State Inconsistency)를 유발하는 안티패턴(Anti-Pattern)으로 지적받았다. 이를 극복하고자 ROS2는 파라미터 소유권을 개별 노드 인스턴스(Node Instance)로 이관하는 노드-중심(Node-Centric) 분산 파라미터 아키텍처를 채택하였다.

1.1 분산형 키-값(Key-Value) 트리 구조

각 노드는 고유한 프라이빗(Private) 메모리 공간에 자신만의 키-값 해시 트리(Hash Tree)를 구축하고 방어한다. 파라미터의 타입은 엄격히 통제되며, 정수(Integer), 부동소수점(Float), 불리언(Boolean), 문자열(String) 및 이들의 동차 배열(Array)만을 허용한다.
특히 다차원 계층 구조(Hierarchical Structure)를 도메인으로 표현하기 위해 마침표(.)를 구분자(Delimiter)로 사용하는 네임스페이스 규칙을 따른다(예: pid_controller.gains.kp). 이는 외부 관찰자에게 직관적인 접근 인터페이스를 보장하면서도, 노드 내부적으로는 C++의 std::map 또는 이에 준하는 \mathcal{O}(\log N) 탐색 복잡도를 가진 구조체로 런타임 매핑(Mapping)된다.

2. 파이썬 런치 파일 내 파라미터 인라인(In-line) 주입

가장 기초적인 런타임 변수 주입 방식은 런치 파이프라인 내의 Node 액션 선언부에서 파라미터를 딕셔너리(Dictionary) 형태로 직접 던지는(Injecting) 인라인 모델이다.

2.1 parameters 배열 매개변수

파이썬 런치 스크립트에서 타겟 노드를 포크할 때, parameters 인자를 활용하여 튜플(Tuple)의 리스트를 전달한다.

Node(
    package='drone_controller',
    executable='attitude_control_node',
    name='roll_pitch_controller',
    parameters=[
        {'max_tilt_angle': 35.0},
        {'use_ekf_feedback': True}
    ]
)

이 방식은 코드를 읽는 개발자 관점에서 직관적이라는 장점이 있으나, 알고리즘이 100개가 넘는 칼만 필터(EKF) 공분산 행렬(Covariance Matrix) 등을 요구할 경우에는 런치 스크립트 자체가 비대해져 I/O 병목 및 유지보수 한계를 초래한다. 따라서 이 패러다임은 디버깅 모드 토글과 같은 최상위 메타 플래그(Meta Flag) 선언에 주로 제한적으로 사용된다.

3. YAML 백엔드를 활용한 객체 기반 동적 로딩

대규모 매트릭스나 복잡한 중첩 딕셔너리 파라미터의 군집을 관리하기 위해, 분산 소프트웨어 스택의 표준 설정 포맷인 YAML(YAML Ain’t Markup Language) 파일 포맷이 ROS2 런치 로딩의 백엔드로 공식 채택되었다.

3.1 노드 네임스페이스와 YAML 스키마 매핑

외부 YAML 파일은 글로벌 구성을 지양하고, 루트 최상단에 대상 노드의 명칭(Node Name)을 반드시 기재하여 런타임 스코프(Runtime Scope)를 엄격히 한정한다.

/**:
  ros__parameters:
    global_timezone: "UTC"
roll_pitch_controller:
  ros__parameters:
    gains:
      kp: 1.5
      ki: 0.1
      kd: 0.05

이 스키마가 앞선 Node 액션의 parameters 배열에 파일 경로 객체(os.path.join()) 형태로 삽입되면, 런치 데몬 발진 시 파서(Parser)가 이를 AST(Abstract Syntax Tree) 형태로 해석한다. 흥미로운 점은 /** 와일드카드(Wildcard)를 사용하여 런치 파일 내에 포크되는 다수의 서브 노드 전체에 글로벌 속성을 부분적으로 유포(Broadcasting)할 수 있는 계층적 상속성을 제공한다는 것이다.

3.2 경로 해석의 지연 평가(Late Evaluation) 모델

자율 에이전트 아키텍처에서는 Ament Index 구조와 연계하여 대상 .yaml 파일의 물리적 절대 경로를 런치 런타임 단계까지 확정하지 않는 지연 평가(Late Evaluation) 방식을 취한다. launch_ros.substitutions.FindPackageShare 클래스는 시스템 부팅 직전 환경 변수 트리를 스캔하여 올바른 구성 파일의 위치를 동적으로 결합(Resolution)함으로써, 도커(Docker)나 복수 작업공간(Workspace Overlay) 환경에서도 경로 파괴 없이 견고하게 대용량 튜닝 값들을 노드 메모리로 흘려넣는 핵심 혈맥 역할을 수행한다.
