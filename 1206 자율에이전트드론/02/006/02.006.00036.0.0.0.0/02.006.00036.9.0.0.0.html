36.9 CLI 명령줄 인자와 YAML 로딩 순서 간 충돌 제어 및 결정론적 매개변수 초기화

ROS2 미들웨어 환경에서 노드의 파라미터(Parameter)는 소스 코드의 기본값, 런치 파일의 인라인 딕셔너리, 외부 구성(YAML) 파일, 그리고 프로세스 발진 시점의 터미널 명령줄 인터페이스(CLI) 인자 등 다중 채널을 통해 동시다발적으로 주입될 수 있다. 이처럼 주입 소스(Source)가 다각화될 경우, 특정 파라미터 키(Key)에 대해 서로 다른 채널이 상충하는 값을 요구하는 충돌(Collision) 시나리오가 필연적으로 발생한다. 본 절에서는 런치 시스템 내부의 로딩 순서(Loading Sequence)가 이러한 충돌을 어떻게 결정론적(Deterministic)으로 제어하는지, 그리고 CLI 인자가 절대적 우선순위를 점유하는 아키텍처의 평가 대수(Evaluation Algebra)를 규명한다.

1. 다중 주입 소스 간의 파라미터 트랜잭션 큐(Transaction Queue)

노드가 메모리에 스폰(Spawn)되어 rclcpp::Node 생성자를 통과하는 런타임 극초기 단계에서, 미들웨어의 파라미터 백엔드는 대기 중인 모든 설정 값들을 하나의 전역 트랜잭션 큐로 집결시킨다.

1.1 주입 소스의 계층적 평가(Hierarchical Evaluation) 모델

시스템은 이 트랜잭션 큐를 임의의 순서로 해소하지 않고, 구조적으로 정의된 강건한 해석 우선순위(Priority Hierarchy)에 따라 순차적 덮어쓰기(Sequential Overwrite)를 시행한다.

  1. 소스 기반 묵시적 할당 (우선순위 최하위): C++ 코드 내 this->declare_parameter("gain", 1.0)에 의해 부여된 값.
  2. YAML 파일 매핑: launch_ros의 Node 파라미터 리스트를 통해 전달된 .yaml 구성 파일 내부의 값.
  3. 인라인(In-line) 딕셔너리: 파이썬 런치 파일의 parameters=[{'gain': 2.0}] 구문 등에 의해 명시적으로 선언된 로컬 스코프 값.
  4. CLI 인자 오버라이드 (우선순위 최상위): $ ros2 run pkg executable --ros-args -p gain:=3.0과 같이 사용자가 터미널에서 강제 주입한 런타임 아규먼트.

이 스레드-세이프(Thread-safe)한 평가 사슬에 따르면, YAML 파일에 gain: 5.0이 적혀 있더라도 런치 액션 내부 딕셔너리가 이를 덮어쓰며, 궁극적으로 시스템 부팅 최후 방어선인 CLI 인자가 이전의 모든 캐재를 무효화(Invalidate)하고 제어권을 독점하게 된다.

2. 로딩 순서 충돌의 결정론적 제어와 지연 캡처(Late Capture)

YAML 구성 파일과 런치 인라인 딕셔너리가 동시에 런치 블록에 존재할 때, parameters 배열 내부에서의 기술 순서(Writing Order) 조차도 오버라이드 결과를 지배한다.

2.1 parameters 리스트 요소의 평가 위상(Topological Phase)

Node(
    ...
    parameters=[
        yaml_config_path,            # Step A
        {'flight_mode': 'manual'},     # Step B
        another_yaml_config_path     # Step C
    ]
)

파이썬 런치 데몬은 위의 리스트를 위에서 아래로 순차 평가(Sequential Evaluation)한다.
만약 첫 번째 yaml_config_path 문서 내부에 flight_mode: 'auto'가 정의되어 있었더라도, Step B를 통과하며 'manual'로 덮어씌워진다. 만약 세 번째 인자인 another_yaml_config_path 내부에 다시금 flight_mode: 'hold'가 기재되어 있다면 최종적으로 노드는 'hold' 상태로 부팅된다.
따라서 대규모 ROS2 시스템 엔지니어링에서는 글로벌 군집 파라미터가 담긴 공통 YAML을 배열의 최상단(Index 0)에 배치하고, 해당 드론 기체만의 특별한 튜닝 값을 담은 인라인 딕셔너리를 배열의 하단에 배치하는 수리적 스택 오버라이드(Stack Overwrite) 디자인 패턴이 강제된다.

3. --ros-args 기반 CLI 오버라이드와 런타임 긴급 개입

모든 런치 스크립트 시맨틱스가 확정되었더라도, 최상위 제어권은 이그제큐터(Executor)를 기동시키는 OS 터미널의 커맨드라인 파서(Command Line Parser)가 쥐고 있다.

3.1 CLI 주입의 무결성(Integrity) 침강

디버깅이나 긴급 제어 탈취 상황에서, 엔진 코드를 재컴파일하거나 YAML 파일을 열어 수정할 시간적 여유가 없을 때 --ros-args -p 커맨드는 유일한 돌파구를 제공한다. ROS2 미들웨어는 커널 레벨에서 argv 벡터를 파싱하여, --ros-args 토큰 뒤에 이어지는 파라미터 선언들을 노드 초기화 객체 속 initial_parameters 해시 맵의 가장 높은 권한 노드(Root Node)로 강제 이식(Grafting)한다.
이러한 결정론적 매개변수 초기화 시스템은, 소프트웨어 구조의 모듈화(Modularity)를 위해 로직과 파라미터 변수를 고도로 분리(Separation)하면서도, 인간 감독관(Human Supervisor)의 긴급 개입 권한을 시스템 아키텍처의 최정점에 위상수학적으로 고정시키는 강력한 Fail-Safe 인프라로 작용한다.
