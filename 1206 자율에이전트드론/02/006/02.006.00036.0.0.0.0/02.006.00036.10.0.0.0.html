36.10 ROS_DOMAIN_ID 논리 네트워크 격리 환경 내 글로벌 파라미터 브로드캐스트 스코프 규제

ROS2 미들웨어는 DDS(Data Distribution Service) 통신 규약을 기저에 깔고 있으며, 물리적인 로컬 에리어 네트워크(LAN) 상에서 다수의 로봇 시스템이 동시에 기동할 때 발생할 수 있는 데이터 혼선(Crosstalk)을 방지하기 위해 ROS_DOMAIN_ID 환경 변수 기반의 논리적 네트워크 격리(Logical Network Isolation) 기법을 사용한다. 본 절에서는 이러한 도메인 격리 체계가 파라미터(Parameter)의 브로드캐스트 스코프(Broadcast Scope) 및 글로벌 파라미터 이벤트 바인딩(Global Parameter Event Binding)에 어떠한 영향을 미치는지 상호작용적 관점에서 분석하고 규제안을 설계한다.

1. ROS_DOMAIN_ID의 DDS 포트 해싱(Hashing)과 멀티캐스트 단절

ROS2의 ROS_DOMAIN_ID는 단순한 소프트웨어적 필터링 태그가 아니라, 하부 네트워킹 스택(UDP/IP) 레벨에서 멀티캐스트(Multicast) 포트 번호를 가름하는 강제적 해싱 인자(Hashing Factor)로 작용한다.

1.1 파라미터 서비스 탐색(Discovery) 그래프의 분리

파라미터 조회 및 갱신을 수행하는 ~/get_parameters, ~/set_parameters 등의 내부 은닉 서비스들은 본질적으로 DDS의 RPC 토폴로지를 따른다. 만약 A 드론 인스턴스가 ROS_DOMAIN_ID=10에서 기동하고 B 드론 인스턴스가 ROS_DOMAIN_ID=20에서 기동한다면, 두 에이전트는 서로의 파라미터 서비스 포트를 물리적으로 청취할 수 없다.
따라서 지상 관제소(GCS)에서 A 드론의 /ekf_filter/gain 값을 동적으로 덮어쓰기 위해 파라미터 클라이언트 노드를 켤 때, 해당 클라이언트의 도메인 ID 역시 정확히 10으로 일치해야만 파라미터 데몬(Daemon) 간의 핸드셰이크(Handshake)와 그래프 탐색(Graph Discovery)이 성립한다. 이는 군집 에이전트 간 악의적이거나 우발적인 파라미터 오염을 네트워크 계층에서 원천 방어하는 하드웨어 친화적(Hardware-Affinity) 보안 경계를 형성한다.

2. 글로벌 파라미터 브로드캐스트 스코프의 통제

ROS2는 특정 파라미터가 변경될 때마다 그 이벤트를 글로벌 토픽인 /parameter_events 로 퍼블리싱(Publishing)하여, 생태계 내의 로거(Logger)나 모니터링 노드(예: rqt_reconfigure)가 그 사실을 인지할 수 있도록 브로드캐스트 아키텍처를 도입했다.

2.1 이벤트 에코(Echo)의 도메인 국소성(Locality)

/parameter_events 토픽은 시스템 전역(Global) 토픽처럼 보이지만, 실제로는 해당 ROS_DOMAIN_ID 내부로 그 브로드캐스트 스코프(Broadcast Scope)가 엄격하게 제한(Confined)된다.
이는 대규모 군집 제어 스택에서 매우 중요한 성능 최적화 지점(Performance Optimization Point)이다. 만약 N대의 드론이 모두 ROS_DOMAIN_ID=0으로 동일한 논리망에 묶여 있다면, 임의의 하나의 드론 내부에서 빈번한 센서 보정 등으로 파라미터가 갱신될 때마다 그 이벤트 메시지가 다른 모든 N-1대의 드론 서브네트워크로 무차별 살포(Flooding)된다.
따라서 대역폭(Bandwidth) 포화로 인한 제어 주기 지연을 방어하기 위해, 엔지니어는 드론 개별 개체마다 독립된 ROS_DOMAIN_ID (예: 1번 드론은 11, 2번 드론은 12 등)를 부여하여 글로벌 파라미터 브로드캐스트 스톰(Broadcast Storm)을 L4 계층에서 기하학적으로 절단(Truncation)해야만 한다.

3. 멀티 도메인 환경에서의 런치 파이프라인 매개변수 주입

론치 파이프라인(Launch Pipeline)에서 복수의 프로세스를 기동시킬 때, 어떠한 노드가 어느 도메인 환경 변수를 쥐고 생성(Spawn)될 것인지를 런타임 제어하는 것은 난해한 작업이다.

파이썬 런치 스크립에서는 launch.actions.SetEnvironmentVariable 액션을 GroupAction과 결합하여, 특정 노드 집합이 포크(Fork)될 때만 ROS_DOMAIN_ID를 오버라이딩(Overriding)할 수 있다.
이를 통해 단일 마스터 PC 안에서도 서로 완벽히 격리된 복수의 가상 드론 에이전트를 시뮬레이션하고 독립적인 파라미터 구성을 주입할 수 있게 된다. 이 때 생성된 개별 SetEnvironmentVariable 컨텍스트는 오직 자신의 GroupAction 내부에서 기동되는 노드의 파라미터 탐색 스코프에만 반영되며, 전체 호스트 리눅스 환경 변수(Host OS Environment Variable)의 오염이나 런타임 충돌(Runtime Collision)을 유발하지 않는 수학적 스코핑 룰(Scoping Rule)을 따른다.
