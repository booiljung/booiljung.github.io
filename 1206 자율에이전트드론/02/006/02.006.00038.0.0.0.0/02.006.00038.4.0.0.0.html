38.4 콘솔 스트림(Console Stream) 계층 및 파일 로거 기반 이원적 출력 제어 모델

제어 시뮬레이션(HIL) 환경을 벗어나 실제 비행 테스트(Flight Test)가 진행되는 실외 필드에서는 드론에 SSH(Secure Shell) 터미널을 지속적으로 개방해두는 것이 네트워크 가시거리 제약상 불가능하다. 필연적으로 드론 내부의 엣지(Edge) 컴퓨터는 개발자의 모니터링 없이도 블랙박스(Black Box)처럼 스스로 로그 텍스트 스트림을 물리 디스크에 영구화(Persisting)해야 한다. 이를 지원하기 위해 ROS2 rcl_logging 미들웨어는 터미널을 향하는 콘솔 스트림(Console Stream) 외에, 백엔드 파일 입출력(File I/O) 계층을 비동기적으로 구동하는 이원적 출력(Dual Output) 제어 모델을 채택하고 있다. 본 절에서는 운영체제 종속적인 파일 핸들링과 콘솔 렌더링이 어떻게 상호 간섭 없이 독립적인 라이프사이클을 영위하는지 분석한다.

1. 출력 스트림(Output Stream)의 분기(Bifurcation) 메커니즘

ROS2 노드가 기동되며 rcl_init()이 호출되는 순간, 시스템은 프로세스가 가질 수 있는 최종 로그 목적지를 두 갈래의 파이프라인으로 분기시킨다.

하나는 rcutils가 운영체제의 커널 스트림(Kernel Stream) 포인터인 stdout과 stderr로 문자열을 인계하는 가시적 통로이며, 다른 하나는 ~/.ros/log/ 디렉터리 하위에 프로세스마다 고유한 타임스탬프 기반 파일 핸들(File Handle)을 개통하는 영속적 통로이다.
RCLCPP_INFO 매크로가 호출되어 필터링(Filtering)을 무사히 통과한 직렬화 데이터 블록은, 이 분기점에 도달하여 메모리 상에서 두 번 복제(Clone)된다. 하나의 패킷은 콘솔 포매터(Formatter) 큐로, 다른 하나는 디스크 I/O를 전담하는 백그라운드 스레드의 파일 출력 큐로 삽입된다. 이 두 파이프라인은 논리적으로 완전히 결속력(Coupling)이 분리되어 있으므로, 쉘(Shell) 스크립트 실행 환경에서 터미널 출력(stdout)을 /dev/null로 강제 리다이렉션(Redirection)시켜 콘솔 백프레셔(Backpressure) 현상을 소거하더라도, 디스크의 로그 파일 덤핑 작용에는 전혀 간섭을 미치지 않는다.

2. spdlog 백엔드 엔진의 구조와 파일 회전(Log Rotation) 설계

과거 초기 ROS2 배포판에서는 단순한 FILE* 기반의 C 라이브러리로 디스크 로깅을 처리했으나, I/O 스톨(Stall)이 전체 제어 루프를 지연시킬 위험이 대두되었다. 따라서 최신 설계에서는 고성능 비동기 C++ 로깅 라이브러리인 spdlog를 공식 백엔드 파일 로거 플러그인 모듈(rcl_logging_spdlog)로 채택하였다.

2.1 메모리 버퍼 플러시(Flush) 및 디스크 I/O 최적화

spdlog 모듈은 메인 제어 스레드가 디스크 쓰기 완료(Disk Write Completion)를 기다리지 않고 \mathcal{O}(1) 시간에 큐에 메시지만 넣고 이탈할 수 있도록 락-프리(Lock-free) 기반의 멀티 생산자-단일 소비자(MPSC, Multi-Producer Single-Consumer) 비동기 버퍼를 운영한다.
특히 드론의 임베디드 리눅스 eMMC나 SD 카드의 가용 공간 한계(Storage Limit)를 고려하여, 덤프되는 파일 공간이 수백 메가바이트(MB)를 단일 파일로 초과하지 않게 파일 단위 회전(Log Rotation) 아키텍처를 적용한다. 운영체제가 예기치 않은 전력 손실(Power Loss)을 겪더라도, 플러시 주기에 따라 분할 보존된 파일 세그먼트들은 파일 시스템의 저널링(Journaling)을 통해 안전하게 사후 수거(Retrieve)될 수 있는 무결성(Integrity)을 확보하게 된다.

3. 런타임 이원적 제어 스위치(Switch)와 성능 마진의 회수

컴퓨팅 자원이 극도로 제한된 자율 에이전트는 특정 임무(Mission) 구간에서 파일 로거나 콘솔 로거 중 하나를 명시적으로 차단하여 렌더링에 소요되는 CPU 사이클 점유율을 회수해야만 할 때가 있다.

ROS2 론칭 시스템은 이를 제어하기 위해 런타임 인자(Runtime Argument) 레벨에서 백엔드 핸들러의 스위치를 지원한다. 예를 들어 CLI 상에서 --ros-args --disable-rosout-logs 매개변수를 주입하면, 미들웨어는 프로세스 부팅 단계에서 파일 I/O 파이프라인과 글로벌 토픽 발행기 자체의 힙(Heap) 할당을 전면 거부한다. 동시에 --ros-args --disable-stdout-logs 옵션을 인가하면 터미널 렌더링 스트림마저 구조적으로 단전(Cut-off)시킬 수 있다.
결과적으로, 이 이원적 출력 제어 모델은 최적 제어 궤적을 연산하는 에이전트 노드들이 소프트웨어 개발 단계의 가시명료성(Observability)과 프로덕션 운용 단계의 스파르타식(Spartan) 성능 최적화라는 극단적 두 마리 토끼 사이를 유연하게 스위칭할 수 있는 엔지니어링 론의 핵심 중추가 된다.
