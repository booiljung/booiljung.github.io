38.6 RCUTILS_CONSOLE_OUTPUT_FORMAT 환경 변수를 활용한 로그 형식 구성

플라이트 컨트롤러(FC)의 시계열 데이터를 분석하거나 여러 ROS2 노드 간의 통신 지연(Latency) 논거를 확보하기 위해, 엔지니어는 종종 터미널에 스크롤되는 로그 메시지에서 페이로드(Payload) 내용뿐만 아니라 그 발화 시점(Timestamp)과 소스 코드의 진원지(Origin)를 정밀하게 역추적해야 한다. ROS2 로깅 프레임워크는 이러한 메타데이터를 기본적으로 메시지에 결합(Concatenate)하여 출력하지만, 텍스트 파싱(Parsing) 스크립트를 운용하거나 가독성을 극대화하기 위해서는 출력 형식을 사용자의 의도에 맞게 재구성할 필요가 있다. 본 절에서는 운영체제 환경 변수 인젝션(Injection)을 통해 rcutils 콘솔 스트림의 토큰 치환 템플릿(Token Substitution Template)을 런타임에 커스터마이징하는 기법을 고찰한다.

1. 포맷 문자열 템플릿의 토큰(Token) 구조

rcutils 콘솔 출력 포매터(Formatter)는 미리 정의된 토큰 집합을 제공하며, 이 토큰들은 백엔드 렌더링 과정에서 실제 프로세스의 컨텍스트로 \mathcal{O}(N) 시간에 문자열 치환(String Substitution)된다. 핵심 토큰의 명세는 다음과 같다.

  • [{severity}]: 해당 로그의 심각도(INFO, WARN, FATAL 등)를 문자열로 출력한다.
  • [{name}]: 로그를 발화한 로거(Logger) 인스턴스의 이름(일반적으로 노드 이름 또는 네임스페이스 경로)을 출력한다.
  • [{time}]: rcutils 코어 클럭 기준의 시스템 에포크(Epoch) 초 단위 타임스탬프를 소수점 이하 7자리 수준의 분해능으로 출력한다. 다중 노드 시계열 분석의 기준점이 된다.
  • [{message}]: RCLCPP_INFO 매크로에 인자로 주입된 개발자의 실제 원시 텍스트(Raw Text) 페이로드를 출력한다.
  • [{file_name}], [{function_name}], [{line_number}]: 전처리기가 컴파일 타임에 캡처한 호출 스코프의 물리적 파일명, 함수 서명(Signature), 그리고 라인 번호를 명시한다.
  • [{thread_id}]: 해당 로그가 실행된 운영체제의 네이티브 스레드 ID 번호를 표기하여, 멀티 스레딩 환경 내 실행 문맥(Execution Context)을 증명한다.

2. RCUTILS_CONSOLE_OUTPUT_FORMAT 환경 변수의 주입과 구문론

기본적으로 ROS2(Humble 기준)는 [{severity}] [{name}]: {message} 라는 가장 간결하고 수수한 디폴트 포맷을 적용하고 있다. 이 기본 레이아웃을 덮어쓰기 위해, 엔지니어는 노드가 기동하기 전에 쉘(Shell)의 프레임워크 환경 변수인 RCUTILS_CONSOLE_OUTPUT_FORMAT 공간에 사용자 정의 템플릿을 노출(Export)시킨다.

2.1 커스텀 템플릿 설계의 시나리오

예를 들어, 로그 텍스트를 grep과 awk 기반의 정규 표현식으로 파싱하여 비행 궤적 생성기의 응답 속도 매트릭스(Matrix)를 추출하려는 경우, 기본 포맷은 소수점 기반 시계열 정보가 누락되어 있어 분석에 실패하게 된다.
이러한 경우 터미널에 다음과 같이 환경 변수를 인가한다.

export RCUTILS_CONSOLE_OUTPUT_FORMAT="[{time}] [{severity}] [{name}] ({file_name}:{line_number}): {message}"

이후 런치 스크립트를 스폰(Spawn)하면, rcutils 포매터는 스트림 버퍼링 중 해당 환경 변수를 감지(Detect)하고 토큰 치환기를 새로 정의된 정규식 구조에 따라 오버라이드(Override)한다. 그 결과 모든 터미널 출력은 다음과 같이 고밀도 정보 지향적 포맷으로 탈바꿈한다.
[1678901234.5678912] [INFO] [path_planner] (planner_core.cpp:142): Trajectory optimized locally.

3. 시스템 쉘 의존성과 디버깅의 격리(Isolation) 효과

환경 변수를 통한 포맷의 재구성은 소스 코드 재입력이나 CMakeLists.txt 수정, 그리고 바이너리 재빌드를 일체 동반하지 않는다는 점에서 아키텍처적 우수성을 갖는다.

만약 C++ 소스 코드 내부에 문자열 포맷팅을 하드코딩(Hard-coding)한다면, 자율 에이전트의 온보드 컴퓨터(Raspberry Pi, Jetson 등)와 관제소 개발 머신 사이의 이기종 모니터링 규격을 통일할 수 없게 된다. 반면 환경 변수 치환 기법은 노드 바이너리를 철저하게 격리(Isolation)시킨 채, 각자가 배치된 운영체제 쉘(Shell)의 로컬 프로파일(~/.bashrc)에 전적으로 프레젠테이션 계층의 권한을 위임(Delegation)한다. 이는 ROS2 시스템 설계 원칙 중 관심사의 분리(Separation of Concerns)를 터미널 시각화 계층에까지 투영한 우아한 로깅 론(Logging Methodology)이라 할 수 있다.
