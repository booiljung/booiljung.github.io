38.8 실시간(Real-Time) 태스크 내 로깅 함수 호출이 유발하는 지터(Jitter) 분석 및 최소화 요건

로보틱스 엣지 컴퓨팅에서 제어 주기(Control Loop) 내부의 소프트웨어 아키텍처는 데드라인 미스(Deadline Miss)를 허용하지 않는 하드 실시간(Hard Real-Time) 제약 조건을 갖는다. 특히 기하학적 비행 제어나 쿼드로터(Quadrotor)의 모터 회전수를 결정하는 자세 제어(Attitude Control) 루프가 400\text{Hz} \sim 1\text{kHz} 의 주파수로 구동되는 상황에서, 제어 블록 내부 무심코 삽입된 RCLCPP_INFO 호출 단 한 줄은 밀리초(ms) 단위의 무작위적인 지연, 즉 커널 지터(Kernel Jitter)를 유발하는 치명적인 병목 코드가 될 수 있다. 본 절에서는 운영체체의 시스템 콜(System Call) 관점에서 왜 로깅 함수가 리얼타임 태스크를 파괴하는지 분석하고, 이를 설계적으로 우회하는 무중단(Lock-free) 로깅 요건을 고찰한다.

1. 동기식(Synchronous) 로깅 오버헤드의 시스템 코어 분석

디폴트 ROS2 로깅 매크로는 내부적으로 콘솔 출력(Console Output)과 /rosout 패킷 발행이라는 두 개의 무거운 파이프라인과 결합되어 있다.

  1. 동적 문자열 할당(Heap Allocation): 가변 인자(Variadic Arguments)를 파싱하여 메시지 오브젝트를 조립하는 과정에서 std::string의 힙(Heap) 메모리 동적 할당과 해제가 반복 발생한다. 커널의 메모리 매니저 동작은 예측 불가능한 지연을 산출한다.
  2. 블로킹 시스템 콜(Blocking System Call): rcutils 코어가 문자열 포맷을 완성하고 나면, 터미널 출력을 위해 운영체제(OS)의 컨텍스트 스위칭(Context Switching)을 야기하는 write() 시스템 콜을 호출한다. 이 때 다른 노드가 터미널 파이프라인을 선점하고 있다면, 뮤텍스(Mutex) 대기 큐에 갇히게 된다(Thread Blocking).
  3. DDS 직렬화 및 소켓 통신: /rosout 글로벌 라우팅은 송신 큐에 패킷을 락(Lock)을 걸어 밀어 넣는 프로토콜의 영역이다. 네트워크 버퍼가 포화 상태일 경우 미들웨어 아키텍처 상 수 밀리초의 트래픽 전송 지연이 발생할 수 있다.

이와 같은 연쇄 작용의 결과로 정상적인 상황에서는 1\text{ms} 내에 통과해야 할 단일 제어 틱(Tick)이 로깅 트랜잭션 도중 15\text{ms} 이상으로 길어지게 되며, 이는 제어기의 적분항(Integral Term) 와인드업(Windup) 발산 및 물리적 진동(Oscillation)을 유발하는 시스템적 재앙으로 이어진다.

2. 리얼타임 루프에서의 로깅 함수 원천 배제 이론

이에 대응하는 가장 강력하고 수리적인 원칙은, 하드 리얼타임 임계 구역(Critical Path) 내부에서는 그 어떠한 표준 로깅 매크로도 호출하지 않는 것이다.

2.1 공유 메모리 및 링 버퍼(Ring Buffer) 기반 비동기 위임(Delegation)

만약 제어기가 연산한 모터 명령값(Motor Command)의 발산 여부 등 디버깅이 절실한 시점이 발생한다면, 제어 루프 내부에서 즉각 텍스트를 포맷팅하여 RCLCPP_WARN을 호출해서는 안 된다.
대신 플라이트 컨트롤러(Flight Controller) 설계자는 메인 RT 스레드와 완전히 분리된 하위 로깅 전담 백그라운드 스레드(Logging Daemon Thread)를 스폰(Spawn)하고, 두 스레드 사이에 락-프리(Lock-free) 구조의 생산자-소비자 링 버퍼(MPSC Ring Buffer) 채널을 개통해야 한다.
제어 스레드(생산자)는 지터(Jitter) 유발의 원흉인 디버깅 스칼라 수치만을 \mathcal{O}(1) 시간에 링 버퍼 메모리에 밀어 넣은 뒤(Push) 즉시 제어 사이클로 복귀한다. 이후 리얼타임 제약이 없는 로깅 스레드(소비자)가 느긋하게 버퍼를 폴링(Polling)하여 문자열 조립과 DDS 오버헤드를 온전히 감당하며 /rosout으로 방출하는 지연 평가(Lazy Evaluation) 모델을 채택해야만 한다.

3. 실시간 확장(Real-time Extension) 커널 환경에서의 로거 우선순위(Priority) 격하

리눅스 커널의 PREEMPT_RT 패치 등을 통해 에이전트 시스템이 경직된 선점(Preemption) 환경을 지니고 있을 경우, 로거 스레드의 CPU 스케줄러 정책(Scheduler Policy) 편성은 매우 정밀해야 한다.

로깅 백그라운드 데몬 역시 시스템 콜에 의해 락을 획득하는 과정에서 우선순위 역전(Priority Inversion)을 유발할 수 있다. 따라서 로깅 서브시스템 및 spdlog의 비동기 싱크 스레드는 제어, 비전 프로세싱 컴포넌트의 일반 스레드보다 항상 운영체제의 스케줄링 우선순위(Nice/Priority)가 더 낮게(예: SCHED_OTHER 공간으로) 편성되어야 한다. 이를 통해 로깅 I/O 이벤트가 절대로 자율 주행의 조향 액추에이터 스케줄링을 침범하지 못하도록 운영체제 레퍼런스 단에서부터 물리적 통제선(Containment Line)을 획정해야 한다.
