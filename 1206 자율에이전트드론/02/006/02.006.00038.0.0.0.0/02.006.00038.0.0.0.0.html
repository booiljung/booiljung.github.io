Chapter 38. ROS2 로깅(Logging) 구조(RCL_INFO 등) 및 Severity 레벨 관리 체계

자율 에이전트 드론을 구성하는 수십 개의 분산 노드(Node)들은 초당 수백 회의 센서 콜백과 제어 연산을 동시다발적으로 수행한다. 이러한 고밀도 데이터 스트림 속에서, 시스템의 결함을 디버깅(Debugging)하거나 특정 제어 틱(Tick)에서의 편차(Deviation)를 사후에 증명하기 위해서는 표준화되고 일관된 기록(Record) 체계가 요구된다. ROS2 미들웨어는 단순한 터미널 텍스트 출력을 넘어, 로깅(Logging) 자체를 시스템 전역의 토픽(Topic)으로 발행하고 파일 시스템에 영구적으로 보존하는 통합 로깅 아키텍처를 제공한다. 본 장에서는 rclcpp 로깅 매크로 체계의 내부 구조와, 시스템의 긴급도에 따라 스펙트럼을 분리하는 심각도(Severity) 레벨 관리 메커니즘을 탐구한다.

1. 터미널 I/O를 대체하는 로깅 아키텍처의 필연성

C++ 표준 입출력 라이브러리(std::cout, printf)를 이용한 디버깅 기법은 단일 스레드 애플리케이션에서는 유효할 수 있으나, 분산 로보틱스 생태계에서는 치명적인 한계를 드러낸다.

전통적 입출력은 표준 출력 버퍼(Stdout Buffer)를 락킹(Locking)하여 메인 제어 루프의 실행 스레드를 블로킹(Blocking)할 위험이 크며, 다수의 노드가 동시에 터미널을 점유하려 할 때 텍스트 레이스 컨디션(Text Race Condition)을 유발하여 글자가 섞이는 현상이 발생한다. 무엇보다도 런치 파일(Launch File)을 통해 백그라운드 데몬으로 스폰(Spawn)된 노드들의 콘솔 출력은 런타임에 직접 모니터링하거나 필터링하기 불가능하다.
이러한 한계를 극복하고자 ROS2 로깅 시스템은 로컬 터미널 렌더링, /rosout 글로벌 토픽 발행, 그리고 디스크 로그 덤프(Log Dump)라는 3방향 비동기 출력(Three-way Asynchronous Output) 파이프라인을 구축하였다.

2. RCLCPP_INFO 계열 매크로의 메타데이터 직렬화

ROS2 노드 소스 코드 내에서 호출되는 RCLCPP_INFO(this->get_logger(), "...");와 같은 매크로 지시자는 단순히 문자열을 화면에 띄우는 함수가 아니다.

컴파일 타임에 이 매크로 지시자는 해당 구문이 호출된 소스 파일명(File name), 함수명(Function name), 그리고 소스 코드 내부의 라인 번호(Line number)를 캡처(Capture)한다. 런타임에는 이 정적 메타데이터에 시스템 프로세스 기동 경과 시간(System Up-Time), 운영체제의 에포크 타임스탬프(Epoch Timestamp), 노드의 고유 네임스페이스 및 심각도(Severity) 비트 플래그를 결합하여 하나의 거대한 커스텀 구조체(rcl_interfaces/msg/Log) 인스턴스로 조립(Assembly)한다.
이렇게 직렬화(Serialization)된 로그 메시지 페이로드는 미들웨어의 비블로킹(Non-blocking) 링 버퍼(Ring Buffer)에 즉시 양도되며, 제어 스레드는 문맥 전환(Context Switch)의 오버헤드 없이 다음 제어 방정식 연산으로 복귀할 수 있다.

3. 로그 심각도(Severity) 레벨의 체계적 위상

시스템 프레임워크가 발생시키는 모든 로그는 정보의 위급성과 목적에 따라 5단계의 심각도(Severity) 레벨로 계층화된다.

  1. DEBUG: 알고리즘 내부의 스칼라 중간값 연산 이력 등 릴리즈(Release) 환경에서는 억제(Suppress)되어야 할 개발자용 추적 데이터.
  2. INFO: 센서 연결 성공, 이륙(Take-off) 시퀀스 진입 등 시스템의 정상적인 상태 전이(State Transition)를 증명하는 정보.
  3. WARN: 통신 레이턴시 지연 등 시스템이 아직 복구 가능(Recoverable)하지만 결함으로 발전할 가능성이 농후한 징후의 포착.
  4. ERROR: 특정 노드의 콜백 크래시(Crash)나 센서 패킷 유실 등 명백한 예외 상태. 그러나 다른 노드의 기능 기동에는 영향을 주지 않는 격리된 국소 결함.
  5. FATAL: 비행 제어기 하드웨어와의 직렬 통신(UART) 단절 등 시스템 전체의 파국적 다운타임이나 긴급 착륙(RTL)을 유발하는 치명적 붕괴.

4. 학습 주안점

본 장 전반에 걸쳐 학습자는 ROS2의 거대한 텍스트 트래픽을 효율적으로 제어하고 모니터링하는 감시자(Supervisor)로서의 엔지니어링 론을 정립하게 될 것이다. 이 목표를 완수하기 위해 본 장은 다음과 같은 세부 논제를 체계적으로 전개한다.

  • rclcpp 로깅 매크로의 문법적 구조와 포맷팅(Formatting) 프로토콜
  • 조건부(Conditional), 주기적(Throttled), 상태 의존적 커스텀 로그 트리거링 기법
  • 런치 스크립트와 CLI(Command Line Interface)를 활용한 런타임 노드별 심각도 임계값 동적 재구성(Dynamic Reconfiguration)
  • /rosout 전역 토픽의 구조 해석 및 파일 스토리지 I/O 메커니즘

결론적으로, 견고한 로깅 설계는 자율 시스템 블랙박스(Black Box)의 해상도를 결정짓는다. 본 장의 지식을 통해 프레임워크 내부의 은닉된 물리/소프트웨어 전이 과정을 명시적 시계열(Chronological) 데이터로 영구화하는 방법을 터득할 수 있을 것이다.
