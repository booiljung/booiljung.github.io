38.3 시간 지연(Throttle) 및 조건부(Skip First, Once) 로깅 매크로의 상태 유지 메커니즘

자율 에이전트의 메인 제어 루프나 고해상도 카메라 센서의 이미지 취득 콜백은 초당 수십에서 수백 회(50\text{Hz} \sim 400\text{Hz})에 달하는 고주파수(High-frequency)로 발화된다. 만약 이러한 임계 경로(Critical Path) 내부에서 발생한 사소한 예외 상황이나 센서 편차를 일반적인 RCLCPP_WARN 매크로로 분출(Emit)한다면, 불과 수 초 만에 수만 줄의 로그 플러딩(Log Flooding)이 발생하여 디스크 I/O 인프라가 붕괴되고 네트워크 링 버퍼가 포화(Saturation)되는 파국을 맞이한다. 이를 공간적, 시간적으로 제어하기 위해 ROS2 rclcpp는 매크로 호출 지점에 정적(Static) 상태 머신을 은닉하여 로그의 발행 빈도를 조절하는 조건부(Conditional) 로깅 매크로 군을 제공한다. 본 절에서는 이러한 매크로들이 내부적으로 어떻게 상태(State)를 보존하고 타이머(Timer)를 평가하는지 컴파일러 치환 메커니즘을 분석한다.

1. 제어 루프 압사 방지를 위한 THROTTLE 매크로 지터 분석

특정 경고 메시지가 일정 시간 주기를 두고 한 번식만 터미널에 렌더링되게 하려면 RCLCPP_WARN_THROTTLE 매크로를 사용한다.

1.1 타이머 임계값과 스레드 로컬 정적 변수 구조

RCLCPP_WARN_THROTTLE(
  this->get_logger(),
  *this->get_clock(),
  1000, // 1000ms (1초) 대기
  "센서 오차 누적 중: %f", error_val);

위 코드가 100\text{Hz}로 구동되는 제어 루프에 삽입되었을 때, C++ 전처리기는 이 매크로를 단일 함수 호출이 아니라 내부적인 블록 스코프(Block Scope)를 가지는 복합 파이프라인으로 확장(Expansion)시킨다.
핵심 메커니즘은 매크로 내부에 은둔형으로 선언되는 static double last_logged_time 변수에 있다. 이 변수는 함수의 스택 프레임(Stack Frame)이 소멸되어도 데이터 구역(Data Segment)에 영구적으로 보존(Persisted)된다. 루프가 100번 반복되며 매크로에 진입할 때마다, 매크로는 현재 클럭(now())에서 last_logged_time을 감산하여 그 차이가 지정된 throttle_duration (1000\text{ms})을 초과했는지 수학적으로 평가한다. 초과한 단 1회의 틱(Tick)에서만 실제 rcutils_log 백엔드 함수가 호출되고 last_logged_time이 갱신되며, 나머지 99회의 호출은 \mathcal{O}(1) 의 시계열 분기 검사만 수행한 채 즉각 반환(Early Return)되어 시스템 콜 오버헤드를 억제한다.

2. 점화(Ignition) 억제를 위한 ONCE와 SKIPFIRST 메커니즘

빈도 제어(Throttle)가 시간 종속적이라면, 실행 횟수에 종속적인 상태 제어 매크로도 존재한다.

  1. RCLCPP_INFO_ONCE: 시스템 초기화 모듈에서, 특정 서브 노드의 부팅 완료를 알리는 로그는 루프 내부라 할지라도 노드의 전체 수명 주기(Lifecycle) 동안 단 한 번만 출력되어야 한다. 이 매크로는 내부에 static bool has_been_logged = false; 라는 플래그를 생성한다. 첫 실행 시 이 플래그가 원자적(Atomic)으로 true로 반전(Toggle)되며, 이후의 모든 진입 시도는 하드웨어 수준의 명령어 분기 예측(Branch Prediction)에 의해 거의 0에 가까운 비용으로 무시(Ignored)된다.
  2. RCLCPP_INFO_SKIPFIRST: 초기 시스템 스폰 직후의 센서 데이터는 칼만 필터(EKF)의 과도 응답(Transient Response)으로 인해 신뢰할 수 없는 쓰레기 값(Garbage Value)일 가능성이 높다. 이 매크로는 첫 번째 호출을 의도적으로 묵살하여 시스템 안정화 이전의 노이즈 로그를 디스크 I/O에서 배제시키는 역할을 수행한다.

3. 표현식(Expression) 기반의 논리적 스트림 필터링 (EXPRESSION)

타이머나 횟수가 아닌, 특정 연산 수학식의 진리값(Truth Value)에 의해 로깅 여부를 결정짓는 RCLCPP_ERROR_EXPRESSION 패밀리는 자율 에이전트의 예외 처리(Exception Handling)에 주로 쓰인다.

RCLCPP_ERROR_EXPRESSION(
  this->get_logger(),
  (battery_voltage < 10.5),
  "배터리 저전압 컷오프 발동 임박!");

이 매크로의 강력한 이점은 지연 평가(Lazy Evaluation) 설계에 있다. 전처리기는 인자로 들어온 문자열 포맷팅 연산과 rcutils 백엔드 호출 지시문을 내부의 if (battery_voltage < 10.5) 조건문 블록 안으로 밀어넣는다(Encapsulation). 따라서 조건식이 거짓(false)으로 평가되는 한, 문자열 복사, 힙 할당, 그리고 포맷팅 오버헤드 자체가 아예 트리거되지 않는다. 이는 가비지 컬렉터(Garbage Collector)의 부하를 원천 차단하고 C++의 정적 타입 효율을 백분 활용하는 ROS2 철학의 정수이다.
