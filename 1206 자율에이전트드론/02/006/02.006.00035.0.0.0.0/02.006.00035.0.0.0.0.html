Chapter 35. 컴포지션(Composition) 노드 통합 체계 및 메모리 공유 이점

전통적인 ROS1 아키텍처에서는 개별 노드(Node)가 독립적인 운영체제 프로세스(Process)로 인스턴스화되었다. 이러한 프로세스 기반의 토폴로지는 결함 격리(Fault Isolation) 측면에서는 유리할 수 있으나, 고빈도(High-Frequency)로 센서 데이터를 교환해야 하는 자율 주행 및 무인기 제어 시스템에서는 프로세스 간 통신(Inter-Process Communication, IPC) 레이턴시와 직렬화/역직렬화(Serialization/Deserialization) 오버헤드를 야기하여 전체 시스템의 실시간성(Real-time Performance)을 심각하게 저해하였다. ROS2는 이러한 구조적 한계를 극복하기 위해 Nodelets 개념을 발전시킨 ‘컴포지션(Composition)’ 노드 통합 체계를 핵심 아키텍처로 도입하였다. 본 장에서는 컴포지션의 수학적 제어 구조와 런타임 메모리 공유 모델이 분산 제어 시스템의 통신 대역폭 최적화에 미치는 이점을 분석한다.

1. 컴포넌트(Component) 지향 아키텍처와 생명주기

ROS2 컴포지션 모델의 본질은 노드를 더 이상 main() 함수를 소유한 ’실행 파일(Executable)’로 한정 짓지 않고, 런타임에 동적으로 컴포넌트 매니저(Component Manager)가 적재(Load)할 수 있는 ‘공유 라이브러리(Shared Library)’ 객체로 추상화하는 데 있다.

1.1 rclcpp::Node 상속을 통한 동적 로딩(Dynamic Loading)

컴포지션 생태계에 편입되기 위해 C++ 노드는 독립된 프로세서 데몬 스레드를 갖는 대신, 클래스 생성자(Constructor)에서 모든 하부 자원(퍼블리셔, 서브스크라이버, 타이مر 등)을 초기화하는 구조로 재작성된다. 이 컴파일된 .so (또는 .dll) 라이브러리는 rclcpp_components::register_node_macro를 통해 플러그인 레지스트리에 등록된다.

런타임 시(Launch phase), 운영체제 상에는 단일 컨테이너 프로세스(Component Container)만이 기동된다. 이 매니저 프로세스는 외부에서 제공된 런치 파이프라인의 명세에 따라, 여러 개의 컴포넌트 라이브러리 포인터를 자신의 단일 가상 메모리 주소 공간(Virtual Memory Address Space)으로 순차 포크(Fork) 없이 인월(Injected)시켜 인스턴스 체인을 조립한다.

2. 인트라 프로세스 통신(Intra-Process Communication)의 제로 카피(Zero-Copy) 매커니즘

단일 프로세스 컨테이너 안방에 다수의 노드가 동거(Co-location)하게 됨에 따라 얻어지는 가장 강력한 토폴로지적 이득은, 네트워크 스택 파이프라인 우회를 통한 인트라 프로세스 통신(Intra-Process Communication)이다.

2.1 데이터 포인터 연산 및 네트워크 계층 생략

서로 다른 프로세스 간 통신 시, 퍼블리셔 시스템은 C++ 메시지 객체를 바이트 배열로 직렬화(Serialization)하고, 이를 커널 공간(Kernel Space) 네트워크 소켓(e.g., UDP/lo)을 통해 전송한 후, 수신 서브스크라이버가 다시 메모리로 역직렬화(Deserialization)하는 무거운 연산 경로를 거쳐야만 한다. 메시지 페이로드가 수십 MB에 달하는 다채널 LiDAR 포인트 클라우드 등에서는 이 \mathcal{O}(N) 바이트 복사 시간 비용이 치명적인 병목점(Bottleneck)이 된다.

반면 컴포지션 환경에서 노드 A 와 노드 B가 같은 프로세스 내에 적재되어 있을 때, ROS2 미들웨어 하부의 rcl 계층은 발송 대상이 로컬 프로세스임을 판별한다. 이때 직렬화 루틴을 우회하고, 메시지 객체의 std::unique_ptr 또는 std::shared_ptr 스마트 포인터 정보만을 \mathcal{O}(1) 시간 복잡도로 스레드 큐(Thread Queue) 간에 교환한다(Zero-Copy). 이는 센서-제어기 루프의 추론 지연(Inference Latency)을 나노초(Nanosecond) 단위로 삭감하는 데 결정적인 역할을 수행한다.

3. 런치 시스템을 통한 컴포지션 컨테이너 통제 및 스케줄링 이점

런치(Launch) 시스템은 이러한 컴포지션 노드들을 효과적으로 스케줄링하고 런타임 제어권을 행사하는 도메인 특화 인터페이스를 탑재하고 있다.

3.1 ComposableNodeContainer와 ComposableNode 트리

개발자는 파이썬 런치 파일 내부에서 launch_ros.actions.ComposableNodeContainer 객체를 루트로 선언하고, 그 배열 인자로 다수의 ComposableNode 인스턴스들을 나열한다. 시스템 구동 시 런치 데몬은 단일 컨테이너 프로세스를 발진시킨 직후, ROS2 서비스 리퀘스트 패킷을 내부 통신으로 쏘아보내어 명세된 .so 라이브러리들을 일제히 dlopen()(동적 로드)하도록 오케스트레이션(Orchestration)한다.

3.2 스레드 풀(Thread Pool) 공유와 컨텍스트 스위칭 최소화

다수의 독립 프로세스는 각자의 OS 커널 스레드를 무분별하게 선점하여 컨텍스트 스위칭(Context Switching) 캐시 미스(Cache Miss)를 양산한다. 컴포지션 노드는 MultiThreadedExecutor 하나에 묶여 병렬성을 획득할 수 있으며, CPU 코어 밀도가 높은 프로세서에서 스레드 워커(Thread Worker) 풀(Pool)을 전사적으로 공유하게 되므로 연산 부하 모델링이 훨씬 결정론적(Deterministic)으로 변화한다.

결론적으로 컴포지션 노드 통합 아키텍처는 고사양 자율 에이전트 시스템이 극한의 샘플링 레이트(Sampling Rate)에 대응하면서도 대규모 노드 그래프의 유연성을 잃지 않게 하는 강력한 \mathcal{O}(1) 메모리 참조 브릿지(Memory Reference Bridge) 인프라로 정의된다.
