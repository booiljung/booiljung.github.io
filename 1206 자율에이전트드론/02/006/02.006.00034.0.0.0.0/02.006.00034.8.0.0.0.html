34.8 런치 런타임 단계 표준 출력(stdout) 및 표준 에러(stderr) 리다이렉션 파이프라인

분산 로보틱스 시스템에서 수십 개의 독립된 서브프로세스(Sub-Process)가 터미널이라는 단일 입출력 버스(I/O Bus)로 로그 스크림을 일제히 방출할 때, 디버깅 정보의 시각적 포화(Visual Congestion) 현상이 필연적으로 발생한다. 개별 노드에서 발생하는 치명적인 세그멘테이션 오류(Segmentation Fault)나 행오버(Hang-over) 현상의 초기 징후가 정상적인 센서 페이로드의 INFO 로그 스트림에 파묻히게 되면, 오퍼레이터의 상황 인식(Situational Awareness) 속도가 저하된다. ROS2 런치 프레임워크는 이러한 I/O 멀티플렉싱(Multiplexing)의 한계를 극복하기 위해, 운영체제(OS) 수준의 파일 디스크립터(File Descriptor)를 제어하여 표준 출력(stdout)과 표준 에러(stderr)의 데이터 패스를 가로채고 재분배(Redirection)하는 로그 라우팅 파이프라인을 체계적으로 지원한다. 본 절에서는 런타임 단계에서의 스트림 분리 메커니즘과 로깅 인프라 연동망을 규명한다.

1. Node 액션 클래스의 출력(Output) 파기 및 스크린 포맷팅 패턴

파이썬 런치 스크립트에서 Node 인스턴스를 포크(Fork)할 때, 개발자는 output 매개변수를 통해 해당 프로세스가 생성할 아웃풋 스트림의 물리적 목적지를 선언적으로 통제할 수 있다.

1.1 출력 목적지(Output Destination) 할당자

  • 'screen': 서브프로세스의 모든 스트림이 현재 런치 시스템을 띄운 부모 터미널의 표준 출력으로 병합되어 렌더링된다. 이 과정에서 런치 시스템의 launch.logging 모듈은 단순히 원문 스트림을 바이패스(Bypass)하는 것에 그치지 않고, 각 라인의 앞단에 [node_name-PID] 형태의 식별자 접두사(Prefix)를 덧붙여 다중 프로세스 환경에서도 출력의 발원지를 추적할 수 있는 텍스트 포맷팅을 런타임에 삽입한다.
  • 'log': 터미널 UI의 혼잡을 방지하기 위해 스트림을 화면에 표출하지 않고, 현재 세션의 ROS 로그 디렉토리(대개 ~/.ros/log/)에 생성된 특정 파일 스트림으로 직결시킨다.
  • 'both': 터미널 출력과 백그라운드 파일 저장을 동시에 수행하는 디버깅 채널 이중화 분기점(Dual-bifurcation point)이다.

2. stderr 격리와 이기종 로거(Heterogeneous Logger) 연동 모델

복합 노드 네트워크에서는 정상 루틴은 무시하되, 이상 징후(예: TF 패스 스파이크, 센서 Timeout)만을 선별적으로 모니터링 체계로 송출하는 비대칭 리다이렉션 구조가 요구된다. 런치 시스템은 운영체제의 stderr (파일 디스크립터 2)만을 독립적인 채널로 캡처하여 처리할 수 있다.

2.1 에러 출력(Error Output) 스트림 제어

Node 구문 내에 추가되는 arguments=['--ros-args', '--log-level', 'WARN']와 같은 동적 매개변수 주입은 미들웨어 백엔드로 하여금 하위 단계인 DEBUG 및 INFO 패킷의 스트리밍을 원천 차단하도록 유도한다. 그러나 이러한 rclcpp 레벨의 논리적 필터망 우회하여 OS 단에서 발생하는 C 메모리 충돌 덤프(Core Dump)나 C++ 예외(Exception) 추적 로그는 통제할 수 없으므로, 런치 데몬은 stderr를 강압적으로 스크랩(Scraping)하여 중앙 에러 트래커(Central Error Tracker)로 플러싱(Flushing)하는 파이프라이닝을 전개한다.

3. 이벤트 핸들러 기반의 온디맨드(On-demand) I/O 리다이렉션

보다 고도화된 스크립트 작성 시, 특정한 라이프사이클 이벤트(예를 들어 자율 드론이 ‘이륙 가능’ 모드로 도달한 직후)가 발현되었을 때에만 디버그 출력 패스를 스크린으로 여는(Open) 온디맨드 스위칭(Switching)이 활용된다.

이벤트 루프는 register_event_handler 함수 콜과 더불어 대상 노드의 I/O 정책 객체 속성을 런타임 중에 오버라이트(Overwrite)하는 메타 액션을 트리거할 수 있다. 이는 지상 관제소 모니터링 시스템에서 초기 부트스트래핑 과정의 백색소음(White Noise) 로그를 완벽히 소거하고, 기체가 실제로 공중에서 미션에 돌입하는 임계 시점(t_{mission})부터 비로소 핵심 텔레메트리 관측망을 활성화시키는 동적 가시성(Dynamic Visibility) 통제 아키텍처의 근간이 된다.
