34.3 XML 및 YAML 프론트엔드 포맷 비교 분석 및 DOM 트리 파싱 메커니즘

ROS2 런치 아키텍처의 백엔드가 파이썬(Python) 기반의 비동기 이벤트 루프 구동기(Executor)로 통일되었음에도 불구하고, ROS 생태계는 하위 호환성 유지와 시스템 통합 엔지니어의 작성 편의성을 도모하기 위해 XML과 YAML 형식의 선언적 프론트엔드(Frontend) 포맷을 공식적으로 병행 지원한다. 튜링 완전한 파이썬 스크립트가 모든 운영체제 제어권을 쥐고 있는 반면, 마크업 언어(XML) 및 데이터 직렬화 포맷(YAML)은 로직과 설정이 철저히 분리된 정적 토폴로지(Static Topology) 설계에 최적화되어 있다. 본 절에서는 이 두 텍스트 포맷 간의 문법적, 구조적 특성을 비교 분석하고, 런치 인터프리터(Launch Interpreter) 내부에서 이질적인 프론트엔드 포맷들이 동일한 LaunchDescription DOM(Document Object Model) 트리로 변환되는 파싱(Parsing) 메커니즘을 규명한다.

1. XML 대 YAML 구조의 선언론적 특성 비교

동일한 노드 실행 지시를 내리더라도, XML과 YAML은 기계 판독성(Machine Readability)과 인간 가독성(Human Readability) 측면에서 상이한 문법적 스키마를 따른다.

1.1 XML (eXtensible Markup Language) 프론트엔드

ROS1 roslaunch의 직접적인 후계 포맷인 XML은 태그(Tag)와 어트리뷰트(Attribute)의 엄격한 계층 트리를 통해 런치 엔티티를 명세한다. <launch> 루트 노드 하부에 <node>, <include>, <arg> 등의 예약어 태그를 배치하며, 파라미터나 설정값은 대개 태그의 내장 속성(pkg="demo_nodes_cpp" exec="talker")으로 정의된다. XML은 구조적 엄밀성이 뛰어나고 XSD 스키마를 통한 사전 유효성 검증(Validation)이 용이하다는 강점을 가지나, 괄호 기반의 장황한 보일러플레이트(Boilerplate) 문법으로 인해 직관적인 가독성이 다소 저하된다.

1.2 YAML (YAML Ain’t Markup Language) 프론트엔드

최신 클라우드 네이티브(Cloud-Native) 아키텍처나 컨테이너 오케스트레이션(예: Kubernetes)의 표준 문법으로 자리 잡은 YAML은, ROS2 런치 시스템에서도 구조적 단순함의 극치를 제공한다. 태그 대신 스페이스바 들여쓰기(Indentation)를 통해 스코프(Scope)를 계층화하며, launch:, node:, pkg: demo_nodes_cpp 등 키-값(Key-Value) 쌍의 리스트 형태로 런치 트리를 구성한다. YAML 포맷은 문서 라인 수를 대폭 감축시키고 인간의 시각적 인지 부하를 줄여주어, 초대규모 환경 파라미터 딕셔너리(Parameter Dictionary)를 정의하는 데 압도적인 유리함을 지닌다.

2. 런치 인터프리터의 DOM 트리 생성 역학

개발자가 XML이나 YAML 파일 중 어떤 것을 사용해 런치 스크립트를 작성하더라도, ROS2 런타임 시스템 콜 백엔드는 C++ 또는 순수 Python 코어 언어이므로 이 텍스트 파일을 처리하기 위한 추상 구문 트리(AST, Abstract Syntax Tree) 해석기가 필연적으로 개입해야 한다.

2.1 확장형 파서(Pluggable Parser) 시스템 아키텍처

launch 패키지 내부에는 Frontend 클래스 인터페이스를 상속받는 전용 파서 플러그인 모듈들(launch_xml.Parser, launch_yaml.Parser)이 서브모듈로 존재한다. 커맨드라인에서 ros2 launch package_name file.xml이 호출될 때 프레임워크는 파일의 확장자명을 정규식으로 판별하여 적합한 파서 엔진을 동적 바인딩(Dynamic Binding)한다.

2.2 문자열 속성의 파이썬 Action 객체 런타임 치환 과정

XML이든 YAML이든 이를 관통하는 변환 메커니즘의 본질은 문자열 토큰의 Python 객체 인스턴스화(Instantiation)이다.

  1. 어휘 분석 및 딕셔너리 매핑: 파서는 텍스트 파일의 DOM 텍스트 트리를 읽어 들여, 메모리 상에 거대한 중첩 딕셔너리 혹은 엘리먼트 노드 트리로 1차 적재한다.
  2. 팩토리 함수(Factory Function) 호출 제어: 맵핑된 키워드(예: <node> 또는 node:)를 포착하면, 프론트엔드 변환기는 launch_ros.actions.Node Python 클래스의 생성자(Constructor)를 반사적(Reflective)으로 팩토리 호출한다. 이때 속성으로 부여된 패키지명, 파라미터 경로는 치환(Substitution) 객체 래퍼(Wrapper)로 포장되어 주입된다.
  3. 최종 LaunchDescription 컴파일: 텍스트 상의 모든 중첩 계층이 적절한 Action 객체와 LaunchConfiguration 객체로 컨버팅(Converting) 완료되면, 메모리 단일 객체인 역직렬화된 LaunchDescription 이 도출체로 반환되며 백엔드의 멀티프로세싱 이벤트 루프에 제어권이 이관된다.

이러한 DOM 트리 변환 아키텍처는 프론트엔드 언어의 차이를 미들웨어 백엔드로부터 완벽하게 디커플링(Decoupling)시켜, 사용자가 임무의 복잡도와 취향에 따라 파이썬, XML, YAML이라는 ‘다중 전술 언어’ 중 최적의 도구를 선택할 수 있는 유연한 생태계를 성립시킨다.
