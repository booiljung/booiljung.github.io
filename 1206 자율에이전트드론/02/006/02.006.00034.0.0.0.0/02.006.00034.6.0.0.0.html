34.6 서브 런치 파일(IncludeLaunchDescription) 계층적 스코프 분할 전략

다차원적인 센서 스위트(Sensor Suite)와 다중 자율 에이전트가 복잡하게 얽힌 현대 로보틱스 시스템을 단일 런치 스크립트 파일 내에서 모놀리식(Monolithic)으로 서술하는 것은 엔지니어링 유지보수성 측면에서 치명적이다. 수백 줄에 달하는 선언적 코드는 변수 충돌, 네임스페이스 오염(Namespace Pollution), 반복 코드의 증식을 초래하며 협업 개발의 병목(Bottleneck) 요인이 된다. ROS2 런치 프레임워크는 이러한 거대 토폴로지를 마이크로 수준으로 모듈화(Modularization)할 수 있도록 IncludeLaunchDescription 액션 클래스를 제공한다. 이를 통해 개발자는 독립적으로 검증된 타 패키지의 런치 파일을 재사용 함수처럼 호출할 수 있다. 본 절에서는 서브 런치 파일을 통해 시스템을 계층적으로 조립하는 역학과, 부모-자식 간의 스코프(Scope) 분할 전략을 수학적 의존성 그래프 관점에서 고찰한다.

1. IncludeLaunchDescription의 위임(Delegation) 매커니즘

IncludeLaunchDescription은 부모 인스턴스(Parent Launch File)의 트리 가지 내에 자식 인스턴스(Child Launch File) 전체의 실행 위상(Topology)을 통째로 이식(Grafting)하는 메타-액션(Meta-Action)이다.

1.1 런치 소스 로케이터(LaunchSource Locator)

서브 런치를 호출하기 위해서는 파이썬 모듈의 동적 임포트(Import) 방식과 유사하게 해당 파일의 위치를 런타임에 추적해야 한다. 이때 PythonLaunchDescriptionSource 또는 XMLLaunchDescriptionSource 래퍼(Wrapper)가 사용되며, 내부 인자로 FindPackageShare 치환 토큰을 결합하여 파일 시스템 경로의 하드코딩을 방지한다. 이 과정은 ROS2 패키지 인덱스 시스템을 통해 대상 파일의 컴파일된 공유 디렉토리(Share Directory)를 안전하게 역추적하는 지연 바인딩(Late Binding) 패턴에 해당한다.

2. 부모-자식 스코프 간의 파라미터(Argument) 전달 체계

하나의 자율비행 드론 Swarm 런치 파일이 개별 드론의 동작을 관장하는 Single_Drone 서브 런치 파일을 N번 다중 호출할 때, 동일한 자식 스크립트가 각기 다른 드론 ID와 초기 좌표를 문맥으로 전달받아 다형성(Polymorphism)을 획득해야 한다.

2.1 launch_arguments 딕셔너리 포워딩(Forwarding)

부모 런치 파일은 IncludeLaunchDescription의 launch_arguments 매개변수에 튜플(Tuple)의 리스트 형태로 인자 값들을 명시적으로 주입할 수 있다.

  • 자식(Sub) 파일의 수신: 서브 런치 파일 내부에서는 DeclareLaunchArgument 액션을 선언하여 외부로부터 주입되는 해당 변수명(예: drone_id)의 진입 인터페이스(Interface)를 개방한다.
    해당 인자는 LaunchConfiguration('drone_id') 토큰을 통해 자식 스코프 내의 노드 속성이나 조건 분기문에 동적으로 치환된다. 이 명시적 인수 전달(Explicit Argument Passing) 메커니즘은 프로그래밍 언어의 함수 입출력 프로토콜과 완전히 동치되어, 서브 런치를 순수 함수(Pure Function)처럼 부작용(Side-effect) 없이 재사용할 수 있게 한다.

3. PushRosNamespace를 연계한 런타임 스코프 고립화(Isolation)

다중 에이전트 런치 파일 병합 시 발생하는 가장 흔한 충돌계 오류는 동일한 이름의 토픽(예: /cmd_vel)이나 동일한 노드명(예: imu_filter)이 전역 메모리 공간에서 경합(Race)하는 현상이다.

이를 논리적으로 격리하기 위해 IncludeLaunchDescription 액션 자체를 GroupAction 블록으로 감싸고, 그 상단에 PushRosNamespace(LaunchConfiguration('namespace')) 변이 체인을 삽입하는 설계 패턴이 널리 차용된다.
이러한 토폴로지 구성 하에서는 서브 런치 내부 객체들이 자신들의 위치를 / 루트 네임스페이스로 착각하여 부트스트래핑하더라도, ROS2 런타임 엔진이 백그라운드 환경 변수를 덮어씌워 이들을 지정된 서브-도메인(예: /drone_1/cmd_vel) 속으로 무결하게 리매핑(Remapping)시킨다.

요컨대, 계층적 스코프 분할 전략은 거대 로보틱스 시스템 코드를 유지보수 가능한 트리 단위의 컴포넌트 조합(Component Composition)으로 환원시킨다. 부모 런치 스크립트는 중앙 오케스트레이터로서 하위 서브 파이프라인의 생애주기를 관장하고, 서브 론치 파일은 추상화된 입력(Argument)만을 소비하여 예측 불가능한 변수 오염 없는 철저한 캡슐화(Encapsulation)를 달성한다.
