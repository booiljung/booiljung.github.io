52.4 Blackboard 전역 데이터 저장소 구조 및 포즈(Pose), 경로(Path) 컨텍스트 런타임 공유 매커니즘

1. 무상태성(Statelessness) 패러다임과 전역 컨텍스트 공유의 필요성

행동 트리(Behavior Tree; BT)의 코어 철학은 단일 노드가 외부 시스템의 상태에 구속되지 않는 철저한 무상태성(Statelessness)을 유지하는 데 있다. 그러나 로봇 내비게이션이라는 본질적인 임무는 “A 지점에서 계산된 경로(Path)를 B 노드가 추종(Follow)해야 한다“는 강한 순차적 데이터 의존성(Data Dependency)을 내포한다.

예를 들어, ComputePathToPose 액션 노드가 전역 플래너(Global Planner) 알고리즘을 구동하여 도출한 A^* 궤적을 FollowPath 제어기(Controller) 노드가 건네받아 모터를 회전시켜야 한다. 이 두 노드는 서로 완전히 격리된 C++ 플러그인 클래스이므로, 멤버 변수를 통한 직접적인 메모리 참조가 불가능하다. 이러한 무상태성 노드 간의 브리지(Bridge) 역할을 수행하기 위해 Nav2는 프로세스 전역(Global)에 상주하며 런타임 레이블(Label) 단위로 데이터를 교환하는 해시 맵(Hash Map) 기반의 인메모리 저장소, Blackboard 시스템을 도입하였다.

2. Blackboard 데이터 저장소의 추상화 해시 맵(Hash Map) 아키텍처

Blackboard는 C++의 std::any (또는 boost::any)와 유사한 다형성 매커니즘을 기반으로 구축된 동적 타입 해시 컨테이너(Dynamic Typed Hash Container)이다. bt_navigator 프로세스가 시작되어 BT XML 파일이 메모리에 파싱되는 순간, 해당 트리 인스턴스를 위한 유일한 전역 블랙보드 객체가 힙(Heap) 메모리에 인스턴스화된다.

각 노드는 블랙보드라는 하나의 칠판에 Key-Value 쌍으로 데이터를 기록(Write)하거나 읽어 들인다(Read). 데이터의 Key는 XML 트리 상에서 정의된 고유한 문자열 변수명(예: "{path}", "{goal}")이며, Value는 ROS 2 메시지 포맷 객체 트리의 포인터이다. 특정 플래너 노드가 연산을 끝내면 결과물인 nav_msgs/Path 객체를 블랙보드의 "{path}" 키 공간에 덮어씌운다(Overwrite). 이후 트리 순회가 진행되어 로컬 제어기 노드에 틱(Tick)이 도달하면, 제어기 노드는 자신의 입력 인자 공간에서 "{path}" 키를 해시 레퍼런스로 조회하여 블랙보드로부터 동일한 오프셋의 경로 객체를 추출(Extract)해 낸다.

3. 포트(Port) 바인딩 및 데이터 타입 형변환(Type-casting) 매커니즘

블랙보드와 개별 오퍼레이터 노드 간의 안전한 메모리 연결을 강제하는 규격화 모델이 포트(Port) 바인딩 체계이다. 모든 커스텀 BT 노드는 자신이 설계될 당시 C++ 소스 코드 내에 providedPorts() 제어자 함수를 통해 자신이 블랙보드에 쓸 데이터의 종류인 OutputPort와 읽어올 수 있는 InputPort를 정적 선언(Static Declaration)해야 한다.

InputPort<geometry_msgs::msg::PoseStamped>("goal")와 같이 포트가 개방되면, BT 프레임워크는 XML 런타임 타임에서 두 노드의 포트 간 자료형 불일치(Type Mismatch)가 있는지를 사전에 컴파일러 수준으로 교차 검증(Cross-validation)한다. 만약 정수형 int를 기대하는 입력 포트에 nav_msgs/Path 타입의 블랙보드 Key가 매핑될 경우, 런타임 캐스팅 에러(Bad Any Cast)를 발생시켜 시스템이 구동 자체를 거부한다. 이 포트 매핑 스키마 덕분에 노드 개발자는 블랙보드의 메모리 포인터를 직접 조작하는 위험에서 벗어나, 추상화된 포트 심볼로만 노드 간 통신 논리를 전개할 수 있다.

4. 포즈(Pose) 및 경로(Path) 공유 컨텍스트의 스레드 안전성(Thread Safety)

다중 에이전트 드론이 고속 이동을 하는 병렬 노드(Parallel Node) 모델 하에서는 블랙보드 자원에 대한 동시다발적 레이스 컨디션(Race Condition)이 발현될 수 있다. 예를 들어 회피 궤적 계산 노드와 궤적 추종 노드가 "{path}" 블랙보드 공간에 Read/Write를 겹쳐서 수행할 경우, 세그멘테이션 폴트(Segmentation Fault) 구조 계층 파손이 일어난다.

이러한 메모리 경합을 봉쇄하기 위해, Nav2 블랙보드 시스템 아키텍처 내부의 set() 및 get() 오퍼레이터는 std::mutex 기반의 상호 배제(Mutual Exclusion) 락(Lock)으로 단단히 래핑(Wrapping)되어 있다. 특히 용량이 거대한 수백 개의 포인트 배열인 전역 경로 Path나 에이전트의 현재 Pose를 추출할 때는, 원시 C++ Call-by-Value 복사를 회피하여 CPU L2 캐시 부하를 최소화하는 반면, 포인터 소유권 이전 구조에 기반한 읽기 전용 스레드 동기화(Thread Synchronization)를 보장함으로써 고주파 틱(Tick) 환경에서의 데이터 무결성을 보존한다.
