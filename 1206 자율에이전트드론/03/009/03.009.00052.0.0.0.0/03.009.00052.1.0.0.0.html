52.1 ROS2 Navigation2 (Nav2) 메타 아키텍처 설계 철학 및 분산 컴포넌트 생명주기(Lifecycle) 제어 모델

1. 분산형 마이크로서비스(Microservice) 아키텍처 기반의 설계 철학

전통적인 ROS 1 시스템의 내비게이션 엔진인 move_base는 전역 경로 탐색기(Global Planner)와 지역 궤적 제어기(Local Planner)가 단일 프로세스 내에서 스레드를 공유하는 모놀리식 아키텍처(Monolithic Architecture)를 띠고 있었다. 이러한 구조는 데이터 복사 최소화 측면에서는 유리할 수 있으나, 단일 컴포넌트의 치명적 오류(예: 메모리 누수로 인한 세그멘테이션 폴트)가 전체 시스템의 런타임을 동반 붕괴시키는 한계를 노출했다.

ROS 2(Navigation 2; Nav2)의 메타 아키텍처 설계 철학은 이를 원천적으로 분리하는 마이크로서비스 지향 분산 프레임워크(Microservice-Oriented Distributed Framework)에 근간을 둔다. Nav2 하에서는 글로벌 경로를 생성하는 planner_server, 로컬 궤적과 속도 명령을 도출하는 controller_server, 비상 이탈 역학을 실행하는 recoveries_server 등 모든 핵심 컴포넌트가 독립된 프로세스 컨텍스트 위에서 ROS 2 액션(Action) 인터페이스를 매개로 느슨하게 결합(Loosely Coupled)된다.

2. 모듈 관장 격리(Isolation)와 내결함성(Fault Tolerance)의 역학

이러한 분산 컴포넌트 지향성은 에이전트 드론에 두 가지 강력한 공학적 이점 즉, 결함 격리(Fault Isolation) 및 플러그인 확장 가능성(Plugin Extensibility)을 부여한다. 물리적/논리적 결함으로 인해 특정 플래너 서버가 비정상적으로 스톨(Stall) 상태에 진입하더라도, 상위 수준의 제어 통제 노드인 bt_navigator는 해당 서버와의 통신 타임아웃(Timeout)을 감지하고 즉시 연결을 해제한 후 회복 행동 서버로 제어 권한을 라우팅(Routing)할 수 있다.

더불어 이는 시스템 운영 중단 없이 pluginlib 기반의 새로운 C++ 알고리즘 노드를 주입하거나 탈착(Hot-swappable)할 수 있는 아키텍처적 관대함을 제공한다. 따라서 맵의 해상도가 변동되거나 드론의 센서 체계가 갱신되었을 때, 전체 Nav2 스택을 재컴파일하지 않고 특정 제어 서버 노드만을 유연하게 재배치하는 메타 아키텍처의 기조가 완성된다.

3. Managed Node 위상과 생명주기(Lifecycle)의 상태 전이 기계(FSM) 모형

다수의 독립된 컴포넌트 서버 계층이 상호 의존성을 띤 채 구동되는 분산 시스템에서는 부팅(Booting) 과정에서 막대한 레이스 컨디션(Race Condition)이 발생한다. 예를 들어 TF 트리가 발행되지 않았거나 코스트맵 생성 노드가 시작되지 않은 상태에서 컨트롤러가 구동되면 인덱스 예외(Exception)와 함께 파이프라인이 붕괴된다.

Nav2는 이를 타개하기 위해 핵심 서버 노드들을 일반적인 ROS 2 Node 클래스가 아닌 rclcpp_lifecycle::LifecycleNode 클래스(일명 Managed Node)로 격상 구현하였다. 라이프사이클 노드는 Unconfigured, Inactive, Active, Finalized의 엄밀하게 정형화된 유한 상태 기계(FSM; Finite State Machine)를 내재하고 있다. 각각의 노드는 초기화 시점에 Unconfigured 위상에서 대기하며 파라미터가 모두 적재되어야 메모리를 할당(on_configure)받고, 퍼블리셔(Publisher)와 액션 서버 앵커(Anchor)가 개방(on_activate)되어 비로소 Active 상태에 도달한다. 즉, 네트워크 상의 제어 패킷은 오직 목적지 서버가 Active 위상에 진입했을 때만 수용되는 구조적 안전판이 성립한다.

4. nav2_lifecycle_manager 주도의 결정론적 관제 및 동기화 시퀀스 파서

이러한 수십 개의 라이프사이클 노드 위상 전이를 사람의 개입 없이 프로그래밍적으로 총지휘하는 중앙 통제 타워가 바로 nav2_lifecycle_manager 노드이다. 이 매니저는 시스템 파라미터(.yaml)로 주입된 노드 부팅 위계(Topology Dependency Array)를 참조한다.

예를 들어, 관리자는 가장 먼저 공간 맵을 배포하는 map_server의 Activate 전이를 강제 투사하고 반환값을 기다린다. 맵이 확인되면 amcl을 부팅하고, 이후 planner_server와 controller_server를 차례로 순차 부팅(Sequential Booting)시키는 결정론적 관제 스크립트를 백그라운드 스레드 상에서 전개한다. 반대로 시스템 강제 종료(Shutdown) 명령이 하달될 경우, 부팅의 정확한 역순(Reverse Order)으로 Deactivate 및 Cleanup 시그널을 각 노드에 직렬화 전송(Serialized Transmission)하여, 좀비 프로세스(Zombie Process)나 메모리 고아 할당 누수를 완벽히 억제하는 고정밀 시스템 클린업(System Cleanup)을 보장한다.
