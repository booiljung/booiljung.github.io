Chapter 52. Nav2 (Navigation2) 메타 아키텍처 구성 및 행동 트리(Behavior Tree) 플로우 체인

1. ROS 1 Navigation의 모놀리식 모형에서 ROS 2 Nav2 미시적 분산 아키텍처로의 전환

과거 ROS 1 생태계의 내비게이션 스택(Navigation Stack)은 글로벌 플래너와 로컬 플래너가 하나의 거대한 상태 기계(Monolithic State Machine)인 move_base 노드 안에서 강하게 결합(Tightly Coupled)되어 작동했다. 이러한 구조는 특정 플래너가 교착 상태(Deadlock)에 빠지거나 메모리를 점유할 경우 전체 내비게이션 파이프라인이 붕괴되는 치명적인 결함을 내포하고 있었으며, 자율 에이전트 드론과 같은 3차원 동적 환경에서의 확장이 극도로 제한적이었다.

ROS 2의 Nav2(Navigation 2) 프레임워크는 이러한 단일 장애점(Single Point of Failure)을 극복하고자 기능 단위의 마이크로서비스 아키텍처(Microservice Architecture)로 철저히 분산 및 모듈화되었다. Nav2 체계 내에서 글로벌 경로 탐색, 로컬 궤적 제어, 코스트맵 생성, 회복 행동(Recovery) 등은 각각 분리된 독립 ROS 2 액션 서버(Action Server) 노드로 격리 구동된다. 이는 특정 컴포넌트의 실패 현상이 격리(Isolation)되고 대체 알고리즘 컴포넌트로의 동적 스왑(Dynamic Swap)을 보장함으로써 메타 아키텍처의 내결함성(Fault Tolerance)을 극대화한다.

2. Nav2 생명 주기(Lifecycle) 시스템 및 컴포넌트 메타 서빙(Meta-Serving) 구조

분산된 다중 노드들이 동시다발적으로 기동될 때 발생하는 초기화 시퀀스의 불확실성을 통제하기 위해, Nav2의 모든 핵심 액션 서버들은 규격화된 상태 머신 기반의 생명 주기 노드(Lifecycle Node) 클래스를 상속받는다.

nav2_lifecycle_manager는 Unconfigured, Inactive, Active, Finalized의 엄격한 상태 그래프(State Graph)를 정의하고, 플래너 서버(planner_server), 제어기 서버(controller_server), 회복 서버(recoveries_server) 등을 정해진 위상 배열(Topological Sequence)에 따라 동기화(Synchronize) 부팅한다. 예를 들어 TF 트리(Transform Tree)와 센서 데이터 파이프라인이 완벽히 인가된 이후에야 코스트맵 노드가 Active로 전이되며, 이어서 경로 생성 및 추종 로직이 활성화된다. 이러한 메타 서빙 및 관제 체계는 통신 레이턴시나 하드웨어 부팅 지연으로 인한 초기 구동 시의 널 포인터 역참조(Null Pointer Dereference) 에러를 원천적으로 차단한다.

3. FSM의 상태 경직성과 행동 트리(Behavior Tree)의 수학적 논리 전개

자율 비행 중인 드론은 갑작스러운 동적 장애물 출현, 텔레메트리 단절, GPS 음영 진입 등 비선형적인 우발 변수에 직면한다. 기존 move_base의 유한 상태 기계(FSM) 모형은 이러한 복합 분기(Branching)를 처리하기 위해 기하급수적으로 팽창하는 상태 천이도(State Transition Matrix) 코딩을 강제하여 사실상 유지보수가 불가능한 스파게티 로직을 야기했다.

이러한 제어 유연성 부재를 타파하기 위해 Nav2는 행동 트리(BT; Behavior Tree) 엔진을 도입하여 메타 의사결정 체계의 수학적 논리 전개 방식을 전환했다. BT는 제어의 흐름을 트리(Tree) 모양의 그래프 자료구조로 시각화 및 직렬화한다. 행동 노드(Action Node)와 조건 노드(Condition Node)가 리프(Leaf)를 형성하고, 셀렉터(Selector, Fallback) 노드와 시퀀스(Sequence) 노드 등의 계층적 오퍼레이터 연산자가 스케줄링 흐름 틱(Tick)을 좌에서 우로, 상에서 하로 전파한다. 이러한 트리 벡터 공간 하에서는 특정 노드의 실패(Failure) 틱트(Tick) 반환이 부모 노드의 우회 경로(Fallback Branch)로 자연스럽게 라우팅되므로, 코드의 전면 재작성 없이 노드 연결의 재배학만으로 무한한 복구(Recovery) 전략 그래프를 증식시킬 수 있다.

4. bt_navigator 중심의 제어 플로우 체인(Flow Chain) 및 플러그인 확장론

Nav2 메타 아키텍처의 정점에는 BT 연산 엔진인 bt_navigator 노드가 군림한다. 외부 애플리케이션이나 임무 관제 API로부터 NavigateToPose 액션 골(Action Goal)이 수신되면, bt_navigator는 미리 정의된 XML 다큐먼트를 파싱하여 인메모리 트리 트래버설(Tree Traversal)을 시작한다.

bt_navigator는 자신이 직접 경로를 연산하거나 모터를 구동하지 않는다. BT의 리프 노드(예: ComputePathToPose, FollowPath)에 도달할 때마다 ROS 2 액션 클라이언트 인터페이스를 통해 백그라운드에 대기 중인 독립된 planner_server나 controller_server에게 임무 파라미터를 하달계 위임(Delegate)한다. 위임받은 서버가 연산 결과를 도출 및 응답하면, 다시 트리 구조로 성패(Success/Failure)가 반환되어 다음 시퀀스로 플로우 체인(Flow Chain)이 전이된다.
더욱이 이 Nav2 생태계 안의 플래너, 컨트롤러, 심지어 BT 노드들 자체도 C++ pluginlib 기반의 플러그인(Plugin) 인터페이스 스펙을 준수하므로, DWB, TEB, MPPI 등 서로 다른 제어 수학 모델을 런타임 상에서 파라미터 매핑(.yaml) 수정만으로 탈착(Hot-swappable)할 수 있는 가공할 확장성을 획득한다.
