52.5 트리 순회(Tree Traversal) 알고리즘 내 틱(Tick) 타이머 주파수 최적화 및 연산 부하 계측

1. BT 실행 주기(Tick Frequency)와 노드 활성화의 상관 역학

Nav2 행동 트리(BT) 엔진인 bt_navigator는 내부적으로 고정된 타이머 루프에 의해 지휘되는 트리거-동기식(Trigger-Synchronous) 메커니즘을 따른다. 개발자가 .yaml 파라미터 계층에서 bt_loop_duration (예: 10ms, 즉 100Hz)을 정의하면, 메인 제어 루프는 정확히 이 주기성을 바탕으로 트리의 최상단 루트(Root) 노드에 틱(Tick) 신호를 주가한다.

틱 시그널은 Sequence나 Fallback 등의 분기 레이어 제어 노드 연산자를 통과하며 깊이 우선 탐색(DFS, Depth-First Search) 방식으로 말단 리프 노드까지 전파(Propagation)된다. 만약 트리 깊성(Tree Depth)이 과도하게 복잡하여 1회의 DFS 순회 연산 시간이 bt_loop_duration 물리적 타이머 제약을 침범할 경우, 틱 미스(Tick Miss) 현상이 유발되어 드론의 궤적 제어 루프(Control Loop) 상에 치명적인 응답 지연(Latency)이 중첩 누적된다. 따라서 시스템 아키텍트는 틱 전파의 시간 복잡도(Time Complexity)가 허용 대역폭을 초과하지 않도록 컴포넌트를 설계해야 압박을 받는다.

2. RateController 기반 서브 트리(Sub-tree) 틱 레이팅 독립 통제론

Nav2 메타 프레임워크 산하에서 가장 많은 연산 리소스를 강탈하는 주범은 역동적 장애물을 평가하는 글로벌 차원의 A^* 인스턴스 경로 재탐색(Replanning) 오퍼레이션이다. 만약 메인 BT 루프 주파수(100Hz)에 맞춰 ComputePathToPose 리프 노드가 1초에 100번 연속 호출된다면, 임베디드 엣지 컴퓨터의 CPU 스케줄러는 즉각적으로 마비(Stall)된다.

이러한 국소적 오버헤드를 제압하기 위해 BT 문법은 RateController 라는 데코레이터 노드(Decorator Node)를 개입시킨다. 이 델리게이트 랩퍼(Delegate Wrapper) 노드는 상단에서 100Hz의 틱이 쏟아져 내려오더라도, 자신이 포용하고 있는 직할 자식(Child) 서브 트리로는 자신이 설정된 임계 주파수(예: hz="1.0") 내에서만 틱을 하달한다. 이 시간적 댐핑(Temporal Damping) 메커니즘을 통해 빈번하게 반응해야 하는 국지적 컨트롤러 노드(FollowPath)는 100Hz 스펙을 투명하게 수여받으면서도, 고사양 레이턴시를 지불해야 하는 전역 플래닝 브랜치는 1Hz의 느린 호흡으로 서브 파이프라인을 구동하는 비대칭 속도 제어 동시성(Asymmetric Speed Control Concurrency)을 이룩한다.

3. 전역 트리 순회(Global BT Traversal) 알고리즘의 CPU 연산 부하 모델링

BT의 모든 틱 사이클이 매 순간 무조건 루트(Root)에서 출발하여 말단까지 왕복한다면 이는 \mathcal{O}(N) 회차의 순회 비용 반복을 의미한다. bt_navigator의 코어인 BehaviorTree.CPP v3/v4 엔진은 상태 회복 추적(State Retrieval Tracking) 메커니즘을 통해 이러한 브루트 포스(Brute-Force) 부하를 상쇄시킨다.

시스템 내부적으로, 이전 사이클 타임에서 RUNNING 상태를 피드백하며 장동(Long-action) 코루틴의 점유 권한을 래치(Latch)하고 있는 노드들의 스택 포인터 경로(Pointer Path)가 BT 캐시 메모리에 유지된다. 따라서 새로운 글로벌 틱이 루트에서 개시될 때, 스케줄러 알고리즘은 완성된 노드나 관련 없는 분기를 스킵(Skipping)하고 \mathcal{O}(\log N) 또는 심지어 \mathcal{O}(1) 의 액세스 타임으로 활성화된 RUNNING 브랜치 계층에 즉각 다이브(Dive)하여 틱을 정밀 주입한다. 이 캐싱 튜닝(Caching Tuning)이 없다면 드론의 배터리나 센싱 레이블을 상호 모니터링하는 복합 트리의 점유율은 비행 시간이 경과할수록 누적 팽창할 수밖에 없다.

4. 스핀 락(Spin Lock) 회피와 트리거(Trigger) 반응형 최적화

연산 부하의 마지막 변수는 특정 노드가 ROS 2의 메시지 도착을 오로지 대기(Wait)하기만 할 때 발생하는 CPU 스핀 락(Spin Lock) 현상이다. 예를 들어 Wait 행동 노드나 특정 토픽 데이터의 도착 여부를 확인하는 커스텀 조건(Condition) 노드가 구현될 때, 빈번하게 유입되는 틱 요청 속에서 단어 while(!msg_received) 구조로 스레드를 차단해버리면 전체 BT 스케줄러가 데드락(Deadlock)에 동반 귀속된다.

Nav2 BT 스키마 최적화론에서는 이러한 블로킹 통신을 엄격히 배격한다. 노드들은 틱 주기 안에서 단순히 인터럽트 플래그나 콜백 핸들러 계층의 최신 버퍼링 포인터 상태만을 비동기 확인(Async Polling)한 뒤, 데이터가 없다면 즉시 RUNNING (또는 로직에 따라 FAILURE)을 반환하여 제어 권한을 상위 시프터에 양도(Yield)해야 한다. 이러한 이변 기반 트리거 반응형(Trigger-Reactive) 비동기 순회 체계가 정립될 때 비로소 노드 간 무섭게 얽힌 제어 의존성 그래프가 상호 병목(Bottleneck) 현상 없이 순수 마이크로초 단위의 평가 레이턴시만으로 최적화 구동될 수 있다.
