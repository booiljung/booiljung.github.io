57.7 런타임 하드웨어 탈락(Drop-out) 발생 시 경유지 추종 실패 지점(Failure Point) 상태 저장 및 복구(Resume) 프로토콜

1. 개요 및 학술적 배경

무인 항공기(UAV)의 전역 경로 추종 과정에서 통신 단절, 센서 오류(IMU 가속도계 포화, GPS 멀티패스 등), 또는 일시적인 페이로드 전원 차단과 같은 하드웨어 탈락(Hardware Drop-out) 현상은 빈번하게 발생할 수 있는 교란 요소이다. 이러한 불확실성 하에서 드론이 초기 출격 지점(Home)으로 복귀하거나 전체 상태 기계(State Machine)를 리셋(Reset)하는 것은 막대한 에너지 낭비와 임무 지연을 초래한다. 따라서 ROS2 Nav2 기반 자율 제어 시스템은 에러 발생 지점, 즉 추종 실패 지점(Failure Point)에서의 컨텍스트를 비휘발성 혹은 영속성 구조체에 저장하고, 하드웨어 계층이 안정화되었을 때 중단된 경유지부터 동작을 재개(Resume)할 수 있는 강건한 복구 프로토콜을 필수적으로 요구한다.

2. 상태 저장(State Preservation) 매니폴드

런타임 하드웨어 탈락이 감지되면, nav2_waypoint_follower 계층과 상위 임무 제어기(Mission Controller)는 즉각적으로 현재 진행 중인 액션을 선점(Preempt)하고 안전 모드로 진입하며, 이하의 상태 정보를 직렬화(Serialization)하여 저장한다.

2.1 인덱스 기반 포인터 메모리 할당

추종할 경유지 목록(std::vector<geometry_msgs::msg::PoseStamped>) 중에서 실패 시점까지 성공적으로 순회한 노드의 인덱스(Index) 카운트를 기억하는 것이 상태 저장의 핵심이다.

  • 루프 불변성(Loop Invariant) 보존: 작업 대기열 루프 상에서 처리 중이던 인덱스 k를 전역 상태 서버(QoS Transient Local 정책 적용)나 데이터베이스에 저장한다.
  • 실패 지점의 공간적 오프셋: k번째 목표 지점까지의 L2-Norm 잔여 거리 혹은 타임스탬프 등 정밀한 동적 상태(Kinematic State)를 추출하여 nav2_msgs::action::FollowWaypoints_Feedback 형식으로 보존한다.

2.2 하드웨어 트리거 복원 컨텍스트

단순한 위치 좌표뿐만 아니라, Task Executor 플러그인 상의 런타임 문맥(Context)도 동기화되어야 한다. 촬영용 짐벌(Gimbal)의 피치 각도, 릴리즈(Release) 트리거의 래치(Latch) 상태 등은 ROS2 Lifecycle 노드의 on_deactivate 트랜지션 콜백 내부에서 파라미터 덤프(Parameter Dump)로 캐싱(Caching)된다.

3. Resume 프로토콜 아키텍처

하드웨어의 Heartbeat(생존 신호)가 복구되고, EKF/UKF 필터 상의 변위 공분산(Covariance) 수렴이 확인되면 복구 프로토콜이 승인된다.

3.1 동적 Waypoint List 재구성

복구를 위해 전체 웨이포인트 목록을 재송신하는 대신, 저장된 인덱스 k를 기준으로 리스트를 슬라이싱(Slicing)한다.

  1. 목록 가지치기(Pruning): 1부터 k-1까지의 이미 완료된 경유지 데이터는 벡터 컨테이너에서 삭제된다.
  2. 보정된 루트(Route) 생성: 현재 재개 지점(로봇의 현재 위치)에서 k번째 미완수 경유지까지의 동적 로컬 경로(Local Path)를 새로이 확보하여 Action Server에 FollowWaypoints_Goal로 인가한다.

3.2 Behavior Tree의 복구 분기(Recovery Branch) 연동

Nav2의 행동 트리(Behavior Tree) 엔진은 일반적인 내비게이션 플로우와 복구 플로우를 노드로 분리하여 관리한다.

  • Resume-Aware Fallback Node: 일반적인 Sequence 노드 대신 예외를 포착하는 Fallback 트리거를 배치하여 하드웨어 탈락 트리거 시 지정된 Wait 상태나 Hovering 동작으로 전환한다.
  • 상태 체인 회복: 센서 상태 복구가 Condition 노드를 통과하면, 캐시 메모리에 적재된 잔여 경유지(Remaining Waypoints)가 인자(Argument)로 투입되며 Sub-tree 체인이 정상 궤도로 복원된다.

4. 데이터 무결성 검증 및 재개(Resume) 오버헤드 통제

복구 연산이 진행될 때는 시공간적 데이터 무결성을 검증해야 한다.

  • 유효 시간(TTL, Time-To-Live) 모니터링: 탈락 시점으로부터 복구 시점까지의 경과 시간이 특정 임계치(Threshold)를 넘는 경우, 주변 동적 장애물이나 풍향(Wind Disturbance) 환경이 급변했을 가능성이 높으므로 무조건적인 복구를 거부하고 Global Planner로 하여금 경로 전면 재탐색(Re-planning)을 강제해야 한다.
  • 시간 복잡도 제어: 액션 서버로 재요청(Re-request)하는 오버헤드는 대기열 크기에 독립적인 상수 시간 \mathcal{O}(1)에 수렴하도록, 재개에 필요한 메모리 포인터 접근 및 교체 비용을 최소화하는 비동기식 상태 머신 이행 매커니즘이 운용된다.
