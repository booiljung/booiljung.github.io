57.4 경유지 도달 시점 기준 Task Executor 플러그인(촬영, 투하 등) 동기화 트리거(Trigger) 인터페이스 메커니즘

1. 개요 및 학술적 배경

ROS2 Nav2(Navigation2) 스택의 nav2_waypoint_follower 패키지는 단순한 다목적 경로 추종을 넘어, 특정한 지점(Waypoint)에 도달했을 때 물리적 또는 논리적 임무(Task)를 수행할 수 있는 플러그인 기반 아키텍처를 제공한다. 드론 시스템에서 경유지 노드는 단순히 이동의 경계점이 아니라, 항공 촬영, 페이로드 투하(Payload Delivery), 통신 중계 네트워크 형성 등 핵심 임무가 발동(Trigger)되는 공간적 제어 지점이다. 본 장에서는 경유지에 도달하는 시점을 기준점으로 삼아 Task Executor 플러그인이 시스템 내 어떻게 동기화되어 동작하는지, 그리고 이 과정에서 이벤트 트리거 인터페이스(Event Trigger Interface) 메커니즘이 어떠한 구조를 띠는지 학술적으로 분석한다.

2. Nav2 Task Executor 플러그인 아키텍처

nav2_waypoint_follower 내에 존재하는 Task Executor는 nav2_core::WaypointTaskExecutor 인터페이스를 상속받아 구현된다. 이를 통해 동적 라이브러리 형태로 로드(Load)되며, 임무 수행 도구(예: 카메라, 투하 장치 트리거)를 독립된 컴포넌트로 플러그인화한다.

2.1 플러그인 생명주기(Lifecycle) 및 동적 로딩

ROS2의 pluginlib 기반으로 설계된 Task Executor 플러그인은 런타임에서 파라미터 서버(Parameter Server)를 통해 인스턴스화된다.

  • 초기화(Initialization): initialize() 메서드를 통해 초기 구성 요소, ROS2 퍼블리셔(Publisher), 서브스크라이버(Subscriber), 그리고 액션 클라이언트(Action Client)가 파싱 및 할당된다.
  • 동적 바인딩(Dynamic Binding): 드론이 목표 웨이포인트(Goal Pose)의 위치 및 방위(Orientation) 오차 허용 범위(Tolerance) 내에 진입했을 때 processAtWaypoint() 메서드가 활성화된다.

2.2 동기화 및 비동기화 실행 모델 정립

드론의 비행 제어기(Flight Controller)와 Task Executor 간의 실행 흐름 제어는 동기식 방식(Synchronous)과 비동기식 방식(Asynchronous)으로 나뉜다.

  • 동기식 블로킹(Synchronous Blocking): 임무(예: 투하 과정의 완료)가 끝날 때까지 경로 추종 프로세스가 대기(Wait) 상태로 머무르는 구조이다. 플러그인이 true 또는 false를 반환하기 전까지 다음 웨이포인트 명령을 액션 서버에 전송하지 않는다.
  • 비동기식 논블로킹(Asynchronous Non-blocking): 영상 촬영과 같이 일시적인 트리거만 발동시킨 후 곧바로 다음 경유지로 이동해야 할 때 적용된다. Task Executor는 백그라운드 스레드(Background Thread) 또는 비동기 액션 요청을 통해 작업을 위임하고 내부 메인 루프 시퀀스는 즉각 진행된다.

3. 동기화 트리거(Trigger) 인터페이스 메커니즘

경유지 도달을 시스템 내 통합 임무 시스템에 동기적으로 알리기 위해서는 신뢰성 있는 트리거 인터페이스가 요구된다. ROS2 생태계에서는 크게 Service, Action, Topic 기반 트리거 메커니즘을 지원한다.

3.1 ROS2 Service 기반 트리거 연동

가장 직관적인 동기화 모델로, std_srvs/srv/Trigger 또는 사용자 정의(Custom) 서비스 타입을 활용한다.

  1. Waypoint 도달 감지: nav2_waypoint_follower 내부의 제어 루프가 base_link와 목표 좌표 간의 L2-Norm 거리를 지속 산출하여 도달 상태를 판별한다.
  2. Service Request 발송: 도달 즉시 해당 웨이포인트에 매핑된 플러그인 내에서 trigger_client_->async_send_request(request) 큐가 호출된다.
  3. Response 상태 확인: 응답 시퀀스를 대기(std::future 기반)하여 촬영 소자 버퍼 할당 및 셔터 제어 등의 완료 유무를 동기화하고 제어권을 반환한다.

3.2 Action 기반의 상태 피드백 트리거 체인

페이로드 투하나 정밀 짐벌 포지셔닝(Gimbal Positioning)처럼 작업 수행의 점진적 상태 피드백과 취소(Cancel) 매커니즘이 필요한 장기 임무의 경우 rclcpp_action::Client를 플러그인 내부에 구현한다.

  • Goal 상태 모니터링 구조: 골(Goal) 상태를 모니터링하여 임무의 ACCEPTED, EXECUTING, SUCCEEDED, ABORTED 등의 상태(State) 전이를 추적한다.
  • 동기화 피드백: 서보 모터의 회전율 또는 투하 고도 피드백 데이터를 실시간으로 모니터링하여 Result 콜백이 반환될 때 드론의 정지 상태 체공(Hovering) 모드를 해제한다.

3.3 이벤트 발생(Timestamp) 및 시스템 동기화

멀티 센서 데이터를 수집하는 환경에서 작업 수행 시점은 매우 정밀하게 동기화되어야 한다.
플러그인은 processAtWaypoint()가 트리거되는 시각의 rclcpp::Time 시스템 클럭 정보를 획득하여 메시지 헤더(Header)에 각인한다. 이를 기반으로 VIO(Visual-Inertial Odometry) 데이터 패킷 또는 센서 계측 데이터와 임무 데이터베이스의 타임스탬프를 동기화(Time-Synchronization)하여 데이터 추론 시 지연(Latency) 보상 알고리즘이 정상 작동하도록 구성한다.

4. 임무 실패 시 예외 처리 메커니즘

플러그인 내에서 할당된 임무가 실패할 경우의 예외 처리는 자율 에이전트의 연속성 보장을 위해 필수적이다.

  • 타임아웃(Timeout) 판정: 지정된 임계 시간(Threshold time)을 초과하여 응답이 없으면 플러그인은 로컬 에러 상태를 발행하고 false를 반환하여 상태 트리에 도달 실패를 통지한다.
  • Retry Policy 기반 재요청: 플러그인은 제한된 횟수만큼 동일한 하드웨어 인터페이스로 재연결(Reconnection) 및 트리거를 시도하도록 설계되어야 하며, 이를 초과할 시 nav2_behavior_tree 상의 Recovery 플로우를 호출하도록 연계된다.
