57.6 무인기 GPS 워프(GPS Waypoint) 제어 패키지 기반 대규모 옥외 전역 경로 명세 모델(YAML) 파싱 아키텍처

1. 개요 및 학술적 배경

무인 항공기(UAV)가 광활한 옥외(Outdoor) 환경에서 정밀 농업, 측량, 광역 정찰과 같은 임무를 수행할 때, 전역 경로(Global Path)는 수십에서 수천 개에 이르는 글로벌 측위 시스템(GPS) 경유지(Waypoint) 노드로 구성된다. 이러한 대규모 경유지 시퀀스는 하드코딩(Hard-coding)할 수 없으므로, 인간이 가독하기 쉽고 시스템이 구조적으로 처리 가능한 YAML(YAML Ain’t Markup Language) 명세 모델로 작성된다. 본 장에서는 ROS2 Nav2 기반의 무인기 GPS 워프(GPS Waypoint) 제어 패키지가 대규모 YAML 경로 파일을 런타임에 어떻게 로드(Load)하고 검증하며 메모리에 적재하는지, 그 파싱(Parsing) 아키텍처를 학술적으로 분석한다.

2. 전역 경로 YAML 명세 스키마(Schema) 설계

대규모 옥외 환경의 경유지는 단순히 2차원 평면 좌표가 아니라, 위도(Latitude), 경도(Longitude), 고도(Altitude), 방위각(Yaw/Heading) 및 Task 플러그인 할당 정보를 포함하는 다차원 메타데이터이다.

2.1 계층적 데이터 구조

표준적인 ROS2 네비게이션용 GPS 경유지 파일은 리스트(List) 구조와 키-값(Key-Value) 페어의 조합으로 구성된다.

  • 루트 파라미터(Root Parameter): 파라미터 서버에 매핑될 네임스페이스와 최상위 키(예: waypoints)를 정의한다.
  • 좌표 요소(Coordinate Elements): 하위 리스트 내에 lat, lon, alt의 부동소수점(Float64, double) 포맷을 할당하여 지구 타원체 상의 특정 지점을 명시한다.
  • 태스크 할당자(Task Allocator): 선택적으로 task_id를 할당하여 특정 경유지 도달 시 활성화시킬 Task Executor 프레임워크 플러그인을 지정한다.

3. ROS2 Parameter Server 기반 파싱 아키텍처

ROS2의 rclcpp::Node 파라미터 선언(Declaration) 방식을 활용한 파싱은 동기적 로드와 타입 검증을 동시에 지원한다.

3.1 YAML 토큰화(Tokenization) 및 파라미터 트리 로드

런타임 환경에서 launch 파이썬 스크립트나 터미널 인자를 통해 YAML 스펙이 인가되면, ROS2 파라미터 서버는 이를 트리(Tree) 계층으로 파싱한다.

  1. 동적 리스트 파싱: 노드는 declare_parameter()를 통해 배열(Array) 기반의 문자열 파라미터를 읽어오거나 rclcpp::Parameter API를 통해 std::vector<std::string> 형태로 개별 웨이포인트 하위 트리(Sub-tree) 이름을 획득한다.
  2. 타입 프로모션(Type Promotion): 각 노드명에 종속된 lat, lon 값을 get_parameter()로 호출하여 형 변환(Type Conversion)을 실시한다. 이 과정에서 유효하지 않은 포맷 문자열이나 NaN(Not a Number) 값이 인입(Input)되는지 확인하여 데이터 무결성 검증을 수행한다.

3.2 yaml-cpp 라이브러리를 활용한 File-IO 직접 파싱

경유지가 수천 개 이상 대규모로 주어져 파라미터 서버 통신 오버헤드나 메모리 한계치를 소모할 우려가 높은 경우, 어플리케이션(Application) 계층의 래퍼(Wrapper) 런칭 노드는 C++ 표준 생태계 라이브러리인 yaml-cpp를 이용해 파일 I/O 시스템에서 명세서를 직접 파싱한다.

  • DOM(Document Object Model) 생성: YAML::LoadFile()을 통해 직렬화된 문서 데이터를 메모리 상의 객체 트리로 로드한다.
  • 동적 할당과 시간 복잡도: 노드 조회를 위한 O(N) 크기의 반복 탐색(Iteration)을 실시하며, 포인터 재할당 부하를 억제하기 위해 사전에 노드 시퀀스 총합(node.size())을 계산하여 std::vector의 reserve() 메서드로 힙(Heap) 메모리를 연속 할당한다.

4. 파싱 데이터의 직렬화 및 Action Server 컴포넌트 바인딩

시스템 트리거로부터 반환된 단일 파싱 데이터 구조체는 통신 미들웨어 상에서 Nav2 프레임워크가 이해할 수 있는 메시지 타입으로 컨테이너화(Containerization) 되어 직렬화(Serialization)를 거쳐야 한다.

4.1 지리 좌표 메시지 컨버팅

추출 및 파싱된 lat, lon, alt 값들은 각각 geographic_msgs::msg::GeoPoseStamped 객체 내 메시지 인터페이스에 정합된다. 리스트 구조로 정렬된 이 객체의 배열은 ROS2 Action 통신 체계에서의 골(Goal) 명세 메시지인 nav2_msgs::action::FollowGPSWaypoints_Goal 헤더 내부의 벡터 필드에 순차적으로 결합된다.

4.2 참조 프레임 정합(Frame Alignment)

파싱된 모든 글로벌 지형 좌표 객체 요소들의 header.frame_id는 글로벌 기준 투영법칙인 wgs84 계열이나 표준 earth 프레임으로 통일되어 선언된다. 이후 Action Server 클라이언트 노드가 요청 송출을 마치고 통신 서버 측이 골을 수리(ACCEPTED) 했을 때 이를 navsat_transform 변환 방정식이 로컬 map 환경상의 nav_msgs::msg::Path 미터법 행렬로 직교 변환함으로써 컨트롤 버스로 전달하게 된다.
