57.9 지상 관제 시스템(GCS) 텔레메트리 통신을 위한 nav_msgs::msg::Path 직렬화 변환 및 피드백(Feedback) 스트림 오버헤드

1. 개요 및 학술적 배경

로봇 운영 체제(ROS2)를 기반으로 구동되는 자율 드론이 옥외 비행 임무를 수행할 때, 지상 관제 시스템(GCS, Ground Control Station)과의 실시간 텔레메트리(Telemetry) 통신은 시스템의 가시성(Observability) 확보를 위해 필수적이다. Nav2 스택 내부에서는 전역 경로 및 로컬 경로가 nav_msgs::msg::Path 메시지 구조로 빈번하게 갱신되며, 경유지 추종(Waypoint Following) 상태 또한 고주파수의 액션 피드백(Action Feedback)으로 발생한다. 본 장에서는 이러한 방대한 공간 제어 상태 메시지를 GCS로 전송할 때 발생하는 직렬화(Serialization) 오버헤드 구조를 분석하고, 대역폭(Bandwidth)의 물리적 선형 한계를 극복하기 위한 무선 스트림 최적화 기법을 학술적으로 다룬다.

2. nav_msgs::msg::Path 메시지 직렬화 병목 현상

nav_msgs::msg::Path 객체는 단일 메시지 내에 수백에서 수천 개의 geometry_msgs::msg::PoseStamped 트리를 포함하는 가변 배열(Array) 형태를 취한다. 이를 저대역폭 무선 링크로 송신할 때 직렬화 페이로드(Payload) 크기는 심각한 통신 체증(Congestion) 병목을 유발한다.

2.1 ROS2 DDS 기반 데이터 페이로드(Payload) 분석

  • 포즈 데이터 단위 크기: PoseStamped 객체 단일 인스턴스는 3차원 위치(Float64 \times 3) 및 쿼터니언 방위(Float64 \times 4), 그리고 타임스탬프와 프레임 등의 헤더 메타데이터를 포함해 약 80바이트 내외의 공간을 차지한다.
  • 스트림 대역폭의 지수적 증가: 글로벌 플래너(Global Planner)가 \text{A}^* 알고리즘이나 동역학적 확장을 통해 1,000개의 기하학적 노드 배열을 발행할 경우, 단일 Path 메시지는 DDS 자체의 직렬화 헤더 오버헤드를 포함하여 최소 80KB에 육박한다.
  • 이를 10Hz의 제어 주기로 GCS에 지속 브로드캐스트할 경우 텔레메트리 링크에만 800KB/s 이상의 네트워크 트래픽 부하가 발생하며, 이는 수십 kbps 수준의 장거리 RF(예: SiK Radio) 텔레메트리 망에서 심각한 패킷 드롭(Packet Drop)과 전송 레이턴시(Latency)를 유발한다.

2.2 직렬화(Serialization) 역학 및 시스템 부하

ROS2의 RMW(ROS Middleware) 계층은 RTPS/DDS 표준에 따라 CDR(Common Data Representation) 규격으로 메시지를 바이트 스트림(Byte Stream)화하여 캐싱한다. 동적 배열 기반의 대규모 메시지를 CDR 바이트 배열로 변환하는 마샬링(Marshaling) 프로세스에서 \mathcal{O}(N)의 메모리 복사와 CPU 사이클이 지속 소모되며, 이는 임무 컴퓨터의 실시간 스케줄링 퍼포먼스를 저하시킨다.

3. Action Feedback 스트림 오버헤드 제어 전략

nav2_waypoint_follower의 액션 서버는 현재 도달 중인 웨이포인트 번호와 현재 로봇 상태의 동적 변화를 피드백 스트림으로 GCS 시스템 및 내부 클라이언트에 지속 전달한다. 이 상태 머신의 고주파 피드백은 통신 오버헤드를 제어하기 위해 반드시 필터링 및 압축되어야 한다.

3.1 토픽 솎아내기(Decimation) 및 다운 샘플링(Down-sampling) 통제

기체의 순항 제어를 결정하는 시스템 내부의 로컬 업데이트 주기(Update Rate)는 통상 10Hz 이상으로 유지하되, 제한된 네트워크를 거쳐 외부 GCS로 송출되는 피드백 스트림은 별도의 브릿지 프록시(Bridge Proxy) 계층을 두어 필터링한다.

  • Decimation 주파수 필터: 큐에 생성된 메시지들을 시간축에서 버리거나(Drop), 특정 샘플링 간격(예: 1Hz)이 경과한 패킷만 추출하여 송출 인터페이스로 릴레이한다.
  • 이벤트 트리거 전환: 연속적인 타이머(Timer) 기반 송출 대신, 경유지 인덱스가 실제로 변경되거나 특정 Task Executor가 트리거되는 시점에만 상태를 갱신하는 상태 주도(Event-driven) 푸시(Push) 아키텍처로 개편한다.

3.2 분해능 압축 및 공간 솎아내기(Spatial Thinning) 기반 경로 경량화

Path 메시지에 대한 가장 원천적인 크기 축소 전략은, 글로벌 경로의 형태학적 정보 손실 오차율을 통제 범위 내로 억제하면서 불필요한 직교 노드들을 제거(Thinning)하는 공간 압축 매니폴드 구현에 있다.

  1. 더글래스-포이커(Douglas-Peucker) 알고리즘 응용: 선형으로 연속되는 구간(직선 비행 구간)의 중간 궤적 노드들을 소거하여, 주어진 오차 허용 한계(Tolerance) 내에서 전체 N개의 배열 크기를 1/10 수준으로 압축 구성한다.
  2. 부동 소수점 경량화 및 프로토콜 차분: 과도한 정밀도를 지닌 내부 64비트 정밀 객체를 Float32로 경량 프로모션(Lightweight Promotion)하거나 시작점 기준의 차분(Delta) 인코딩 방식으로 변환해 MAVLink 등 초경량 마이크로 프로토콜 브릿지로 이관한다.

4. GCS 연동을 위한 QoS(Quality of Service) 정교화

통신이 불안정한 옥외 무선 환경에서는 ROS2 DDS 계층의 강건한 전송을 보장하기 위해 rmw_qos_profile_t 파라미터 수준의 극단적 최적화가 요구된다.

  • History 및 Reliability 설계: Path 객체나 상태 피드백과 같이 언제나 가장 최신 정보 하나만 유효한 멱등성(Idempotent) 데이터 스트림에 대해서는 Reliability 파라미터를 반드시 BEST_EFFORT로 설정한다. 또한 History 뎁스(Depth)를 1(KEEP_LAST=1)로 제한하여, 소실된 과거 센서 패킷의 수동 재전송 기작으로 인한 전체 네트워크 대역의 연쇄적 채증 현상 방지를 최우선 목표로 간주한다.
