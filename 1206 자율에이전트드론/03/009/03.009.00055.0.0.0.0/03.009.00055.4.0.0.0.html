55.4 로컬(Local) 코스트맵의 롤링 윈도우(Rolling Window) 구조 및 런타임 갱신 주파수(Update Frequency) 동역학

1. 개요

전역 내비게이션 경로(Global Path)를 따라 자율 에이전트가 주행할 때, nav2_controller가 실시간으로 소모하는 제어의 기저는 전역 맵 전체가 아니라 기체 주변의 좁은 시야 범위를 담당하는 로컬 코스트맵(Local Costmap)이다. 무한대로 발산할 수 있는 광역 좌표계를 매 틱(Tick)마다 연산하는 것은 시스템 자원을 급격히 고갈시키므로, 로컬 코스트맵은 기체의 이동 반경을 중심으로 한시적으로 \text{width} \times \text{height} 범위를 추종하는 롤링 윈도우(Rolling Window) 메모리 아키텍처를 도입하였다. 본 장에서는 슬라이딩 버퍼로서의 롤링 윈도우 역학과, 동역학적 속도 한계에 따른 최적 런타임 갱신 주파수(Update Frequency) 조율 논리를 서술한다.

2. 롤링 윈도우(Rolling Window) 메모리 이동 아키텍처

로컬 코스트맵의 rolling_window 파라미터가 비활성화(False)되어 있다면 맵은 고정된 원점을 갖지만, 활성화(True)될 경우 맵의 기하학적 중심축은 항상 로봇의 base_link 혹은 추정 odom 좌표계 원점을 실시간으로 추종한다.

2.1 순환 배열(Circular Buffer) 기반의 메모리 시프트(Shift)

로봇이 전진함에 따라 새롭게 관측해야 할 전방 지역은 코스트맵 행렬 내부에 새롭게 할당되어야 하고 후방 지역은 삭제되어야 한다. 이 과정에서 O(N^2)의 픽셀 전면 복사(Copy)를 방지하기 위해 2D 배열에 순환 버퍼(Circular Buffer) 개념을 차용한다.
단위 픽셀 길이 이상의 물리적 기체 변위 \Delta d가 발생할 때, 코스트맵은 전체 메모리를 재할당하지 않고 기준 인덱스 오프셋(Offset)만을 시프트(Shift) 연산한다. 새롭게 시야에 들어온 경계(Border) 셀들에 대해서만 센서 투영 갱신과 레이어 병합 루프(Update Costs)를 트리거(Trigger)함으로써 메모리 대역폭을 획기적으로 절약한다.

2.2 폭(width, height) 파라미터 제원 결정

윈도우의 크기를 어느 정도로 설정할 것인가는 드론의 런타임 최고 속도 관성에 직결된다. 윈도우 크기가 4.0\text{m} \times 4.0\text{m} 일 때 기체가 전방에 장애물이 위치함을 인식할 수 있는 최대 시야는 자가 중심으로부터 단 2.0m에 불과하다. 속력 V와 제어기 반응 속도 딜레이 \tau를 고려할 때, 제동 안전거리 D_{stop}이 \text{width}/2 한계 범위를 초과하게 된다면 로컬 플래너는 장애물을 발견한 시점에 회피 혹은 제동 기동을 수행할 물리적 공간을 이미 상실한 상태가 된다. 안전 상수 \gamma를 고려할 때 \text{width}_{min} > 2 \times \gamma (D_{stop}) 의 하한선을 통제받는다.

3. 런타임 갱신 주파수(Update Frequency)와 동역학

로컬 코스트맵은 로봇과 동적 장애물들이 모두 예측을 벗어나 이동하는 초 단기(Short-term) 미래 연산을 위한 관측 영역이다. 따라서 맵에 그려진 확률 정보의 신선도(Freshness)를 통제하는 update_frequency (초당 업데이트 횟수) 파라미터는 프론트엔드 센서 및 제어기 사이클과 상호 결합되어야만 한다.

3.1 갱신 주파기 및 역학 스케줄링 병목

일반적으로 update_frequency는 5.0 \sim 15.0 \text{Hz} 수준으로 지정된다.

  • 제어기(Controller)와의 인과관계: DWB 혹은 TEB 플래너 모듈의 제어 생성 주기(controller_frequency)가 아무리 빠르더라도(예: 20\text{Hz}), 비용 맵의 갱신 주파수가 5\text{Hz}로 지연되어 있다면 모터는 4번의 제어 틱 동안 과거의 동일한 허구적 장애물 맵을 반복 참조하게 되며, 이는 제어 루프의 진동(Oscillation)과 물리 궤적의 헌팅(Hunting) 현상을 유발한다. 따라서 기구학적으로 f_{controller} \approx f_{update} 로 록스텝(Lock-step) 동기화시키는 것이 이상적이다.
  • CPU 인터럽트 스로틀링: 이 주파수를 무제한 상향시키는 것은 불가능하다. 글로벌 마스터 맵 대비 극히 작은 셀 배열이라 할지라도 레이 캐스팅, 인플레이션 커널 확장 등의 무거운 루프가 강제됨에 따라 엣지 컴퓨터의 잔여 CPU 슬롯이 소모되고 패치(Fetch) 병목을 유발하게 된다.

결과적으로 롤링 윈도우 인프라 내에서 윈도우 너비(해상도 부하)와 갱신 주파수(시간 부하)는 모두 로봇의 최고 동적 속도(V_{max})와 안전 정지 가속도 역학(a_{max\_decel})에 종속되어 수식이 결정되는 철저한 페널티 함수 트레이드오프 관계로 기능한다.
