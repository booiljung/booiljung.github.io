50.9 포인트 클라우드 전처리 지연 시간(Latency)을 최소화하기 위한 멀티스레딩 및 컴포지트 노드(Composite Node) 파이프라인

1. 자율 주행 제어 루프 내 전처리 지연 시간의 치명성

자율 에이전트 드론 시스템에서 3D LiDAR 원시 데이터 취득부터 하위 Nav2 모듈의 회피 벡터 산출까지 소요되는 총 지연 시간(Total System Latency)은 동적 장애물 회피 성공률을 결정하는 절대적 지표이다. 특히 수십만 개의 포인트로 구성된 sensor_msgs/PointCloud2 메시지가 로컬 플래너로 즉각 전달되지 못하고, 중간 변환 단계인 pointcloud_to_laserscan 노드에서 대기 상태(Queueing)에 머무르게 되면, 제어기는 과거의 시공간 정보(Stale Data)를 바탕으로 현재의 위치 제어를 수행하는 시간적 비동기화(Temporal Asynchronization)에 직면하게 된다. 이러한 인식과 반응의 시간차이는 고속 비행 환경에서 드론이 충돌 궤적을 제때 이탈하지 못하는 통제 불능 상태를 유발하므로, 전처리 지연 시간의 단일 밀리초(ms) 단위 감축이 요구된다.

2. 메시지 직렬화 병목과 ROS 2 프로세스 간 통신(IPC) 비용

ROS 2의 분산형 아키텍처 관점에서 볼 때, 센서 드라이버 노드(LiDAR Driver Node)가 생성한 대용량 점군 데이터가 독립된 프로세스로 존재하는 pointcloud_to_laserscan 노드로 전달되기 위해서는 네트워크 인터페이스 계층(DDS/RTPS)을 거쳐야 한다. 이 시점에서 원시 메모리 배열을 바이트 스트림(Byte Stream)으로 패키징하는 직렬화(Serialization) 연산과, 수신측에서 이를 다시 프로그래밍 언어의 구조체로 복원하는 역직렬화(Deserialization) 연산이 강제된다.

메모리 복사(Memory Copy) 및 페이로드 디코딩에 소모되는 CPU 사이클은, 실제 데이터 필터링 및 2D 스캔 투영 알고리즘 연산량보다 수 십 배 더 큰 병목(Bottleneck) 현상을 시스템에 유발한다. 따라서 전처리 레이턴시 단축의 핵심은 차원 축소 알고리즘 자체의 개선보다 미들웨어 레벨에서의 데이터 교환 오버헤드 소거에 집중되어야 한다.

3. 컴포지트 노드(Composite Node) 및 IPC 제로 카피(Zero-Copy) 최적화

레이턴시 최적화를 위한 가장 강력한 디자인 패턴은 ROS 2의 컴포넌트 라이브러리 프레임워크 기반의 컴포지트 노드(Composite Node) 조립 체계이다. pointcloud_to_laserscan 패키지의 실행 파일은 단일 스탠드얼론(Standalone) 프로세스뿐만 아니라 동적 링킹(Dynamic Linking)이 가능한 컴포넌트 플러그인 형태로 빌드된다.

시스템 통합자는 LiDAR 드라이버 노드와 pointcloud_to_laserscan 컴포넌트, 그리고 하위 플래너 모듈을 하나의 동일한 단일 프로세스(Single Process) 내의 컴포넌트 컨테이너(Component Container)에 논리적으로 적재할 수 있다. 동일 프로세스 환경 하에서 ROS 2의 인트라 프로세스 통신(Intra-Process Communication, IPC) 최적화를 활성화하면, 노드 간 이종 메모리 데이터 교환은 값의 복사(Pass by Value)가 아닌 고유 포인터 주소의 이전(Pass by Unique Pointer, std::unique_ptr) 계층인 제로 카피(Zero-Copy) 전송으로 전환된다. 이는 수 메가바이트(MB)에 달하는 점군 메시지의 발행과 수신 간 레이턴시를 \mathcal{O}(1)의 상수 시간으로 급격히 수렴시킨다.

4. 멀티스레딩(Multi-Threading) 콜백 그룹핑 및 병렬 연산 체계 보장

pointcloud_to_laserscan 노드는 sensor_msgs/PointCloud2 구독(Subscription) 및 tf2 브로드캐스터 기반의 역투영 등 복수의 비동기 이벤트를 처리한다. 단일 스레드 실행기(Single-Threaded Executor) 위에서 이러한 노드가 구동될 경우, TF 캐시 동기화 대기 시간이나 메시지 콜백 큐에서 헤드 오브 라인 블로킹(Head-of-Line Blocking)이 발생하여 투영 주기가 지연될 소지가 다분하다.

이를 억제하기 위해 pointcloud_to_laserscan 시스템은 다중 스레드 실행기(Multi-Threaded Executor) 환경 상에 배치되어야 하며, 상호 배타성이 요구되는 콜백(예: TF 수신과 포인트 클라우드 수신)을 상호 독립적인 콜백 그룹(Mutually Exclusive Callback Group)으로 분배하는 로직이 적용되어야 한다. 이러한 하단 프레임워크 층에서의 스레드 자원 할당 최적화는 멀티코어 내장형 엣지 컴퓨터 자원(CPU Cores)을 병렬적으로 한계점까지 점유하도록 유도함으로써, 레이저 스캔 배열 생성 주기를 센서의 베이스 주파수(예: LiDAR 10~20Hz)에 대해 지연율 스파이크(Jitter) 없이 밀착 추종하게 만드는 결정적 통신 파이프라인(Pipeline)을 보증한다.
