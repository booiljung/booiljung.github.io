51.1 rosbag2 스토리지 백엔드(Storage Backend: SQLite3, MCAP) 아키텍처 간 런타임 기록/참조 성능 모델 비교

1. rosbag2 스토리지 추상화 계층 및 백엔드 파이프라인 개요

ROS 2의 레코딩 유틸리티인 rosbag2는 전통적인 ROS 1의 정적 .bag 파일 형식을 탈피하고, 스토리지 플러그인(Storage Plugin) 기반의 동적 추상화 아키텍처를 도입하였다. 이 아키텍처에서 rosbag2_storage 라이브러리는 수신된 데이터 스트림의 미들웨어 계층(RMW) 메시지 형식과 디스크에 기록되는 물리적 형식을 분리한다. 이를 통해 시스템 통합자는 드론의 비행 환경, 데이터 발생 빈도, 그리고 저장 매체의 I/O 특성에 부합하는 구체적인 스토리지 백엔드(Storage Backend)를 런타임에 유연하게 선택 및 주입(Inject)할 수 있다. 현재 표준 ROS 2 생태계에서 가장 지배적으로 채택되는 두 가지 백엔드 엔진은 관계형 데이터베이스 기반의 SQLite3와 로봇 애플리케이션 시계열(Time-series) 데이터 전용 특화 포맷인 MCAP(Message Capture)이다.

2. SQLite3 백엔드의 트랜잭션 모델 및 관계형 스토리지 한계치

SQLite3는 rosbag2의 기본(Default) 스토리지 플러그인으로 동작하며, 토픽 트리, 메시지 타입 정의, 그리고 페이로드(Payload)를 단일 관계형 데이터베이스(*.db3) 구조에 결속한다. SQLite3 아키텍처 하에서 메시지 기록 로직은 SQL의 INSERT 구문 질의(Query)와 ACID(Atomicity, Consistency, Isolation, Durability) 무결성 체크에 의존한다.

저대역폭 센서(예: 50Hz의 IMU, 시스템 상태 메시지)를 로깅할 때는 데이터베이스 인덱스의 순차적 트랜잭션이 안정성을 담보하지만, 자율 드론의 카메라 쌍(Stereo Camera)이나 고밀도 포인트 클라우드와 같이 수 기가비트(Gbps) 스케일의 광대역 스트림이 유입될 경우 상황은 역전된다. SQLite3의 트랜잭션 오버헤드, 파일 시스템의 Write-Ahead Logging(WAL) 메커니즘, B-Tree 인덱스 갱신 시 발생하는 CPU 병목은 디스크 I/O 레이턴시를 급격히 상승시켜, 결국 rosbag2 인메모리 버퍼 버퍼링(Buffering) 한계를 초과하여 패킷 드롭(Packet Drop) 현상과 런타임 성능 저하(Performance Degradation)를 필연적으로 유발한다.

3. MCAP 백엔드의 청크(Chunk) 기반 직렬화 최적화 아키텍처

SQLite3의 I/O 병목을 타개하기 위해 개발된 MCAP 포맷(플러그인명: mcap)은 관계형 매핑 스키마를 과감하게 버리고 로봇 공학의 시계열 로깅 목적에 정확히 부합하는 블록 기반 마이크로 아키텍처를 취한다. MCAP 백엔드는 유입되는 SerializedMessage (ROS 2 미들웨어 계층에서 이미 바이트 단위로 직렬화된 데이터)를 SQL 파싱(Parsing)이나 역직렬화 없이 그대로 메모리 내의 청크 버퍼(Chunk Buffer)에 선형적으로 밀어 넣는다(Append-only).

일정 용량의 청크가 조립되면, 내부에 결합된 메시지 인덱스와 스키마 아이디를 헤더에 붙인 후 백그라운드 스레드를 통해 파일 시스템으로 비동기 플러시(Asynchronous Flush)한다. 이 과정은 CPU 점유율을 극단적으로 낮추고, 런타임 시간 복잡도를 O(1)에 수렴하게 함으로써, 임베디드 엣지 컴퓨터가 장착된 드론이 과도한 로깅 페널티 없이 자율 주행 연산에 집중할 수 있도록 시스템 전 주기 성능을 복원시킨다.

4. 런타임 기록/참조 성능(Record/Playback Performance) 정량해석 모델

두 백엔드 아키텍처의 런타임 기록(Record) 성능은 디스크 쓰기 대역폭(Disk Write Bandwidth) 측면에서 뚜렷한 차이를 보인다. 고속 데이터 로깅 환경에서 SQLite3는 잦은 트랜잭션 커밋(Commit)과 프래그먼테이션(Fragmentation)으로 인해 SSD의 최대 쓰기 속도의 약 30 \sim 40\% 수준을 상한(Upper Bound)으로 갖는 성능 곡선을 보이나, MCAP은 연속적인 대형 블록 순차 쓰기(Sequential Write)를 수행하므로 디스크 컨트롤러가 허용하는 최대 하드웨어 속도의 80 \sim 90\%에 근접하는 스루풋(Throughput) 이득을 달성한다.

반면, 참조 파일 읽기(Playback) 모델에서의 성능은 접근 패턴에 따라 교차 역학을 형성한다. 단순 순차 재생의 경우 MCAP이 앞서지만, 사후 데이터 애널리틱스 시점(Post-Analytics)에서 수백만 개의 분산된 메시지 중 특정 시간대(T_1 \sim T_2)의 한정된 특정 토픽 데이터만 단일 추출(Random Access Query)해야 할 경우, 강력한 쿼리 엔진과 복합 인덱스 트리를 갖춘 SQLite3의 속도가 MCAP의 범용 인덱스 탐색 속도를 일시적으로 능가할 수 있다. 구조적으로 MCAP은 통매체 환경의 ’기록(Record) 효율성’과 ’순차 배포(Sequential Playback) 최적화’에 편향(Bias)되어 있으며, SQLite3는 ’사후 부분 분석 쿼리(Analytics Query)’에 특화된 스토리지 모델로 정량 해석된다.
