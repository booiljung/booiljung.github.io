51.4 다중 분할 저장(File Splitting)을 위한 상태 조건(크기, 시간) 파라미터화 및 시계열 파일 명명(Naming) 규약

1. 장기 자율 비행 시스템에서 단일 거대(Monolithic) 파일 스토리지의 취약성

드론의 장기 자율 비행(Long-term Autonomous Flight) 임무 시, 고대역폭 다중 센서 텔레메트리를 수십 기가바이트(GB) 규모의 단일 파일(Monolithic File)로 영구 누적하는 행위는 파일 시스템 및 데이터 무결성 측면에서 치명적인 취약성(Vulnerability)을 내포한다. 운영체제(OS)의 I/O 블로킹 또는 드론 전력 공급의 순간적 붕괴(예: 하드웨어 결함에 의한 재부팅)가 발생할 경우, 아직 커밋(Commit)되지 않은 인덱스 트리나 파일 종단부의 헤더가 손상되어 단일 거대 파일 전체를 읽을 수 없게 되는 치명적 데이터 오염 현상이 지배적이다. 또한 분석 관점에서도 기가바이트급 파일을 메모리에 적재하여 특정 시점(예: 충돌 시점)의 메시지를 역직렬화하는 것은 극심한 쿼리(Query) 레이턴시를 촉발한다. 이를 회피하기 위해 rosbag2 생태계는 조건 기반 다중 분할(Conditional Splitting) 저장 아키텍처를 도입하여 위험 배분(Risk Distribution)을 강제한다.

2. 디스크 볼륨 크기(Size) 임계값 기반의 분할 파라미터화

첫 번째 분할 기준은 바이트 볼륨스펙에 기반한 max-bag-size 파라미터화이다. rosbag2 노드를 기동할 때 --max-bag-size <bytes> 매개변수를 주입하면, 내부 레코딩 컨트롤러는 활성화된 스토리지 파일(예: SQLite3의 .db3 단위 파일)에 기록된 총 물리적 스트림 크기 추적기(Byte Tracker)를 가동한다.

이 파라미터 기반 분할기(Splitter)는 유입 메시지를 직렬화하여 파일 버퍼에 플러시 덤프를 수행하기 직전, Current File Size + Target Message Size > max-bag-size 부등식이 성립하는지를 매 사이클마다 연산한다. 이 한계점을 돌파하는 순간, 파일 I/O 스트림(File Stream) 핸들은 안전하게 fclose() 또는 DBMS 트랜잭션 종료를 통해 닫히며(Sealed), 곧바로 새로운 인스턴스의 빈 볼륨 파일을 열어 다음 메시지의 기록을 이어간다. 이는 파일 시스템 특유의 4GB 할당 제한(예: FAT32)을 회피하거나, 클라우드 스토리지 전송을 위한 균일한 크기의 청크(Chunk) 조각을 생성하는 가장 직관적인 제어 파라미터이다.

3. 누적 런타임 구간(Duration) 기반의 분할 파라미터화

두 번째 분할 기준은 ROS 2 도메인 클럭(Clock)의 경과 시간을 기준으로 트리거되는 --max-bag-duration <seconds> 파라미터화이다. 이 기법은 센서의 데이터 유입량이 동적으로 가변(예: 이벤트 캡처 전용 카메라 트리거)하여 데이터 볼륨만으로는 특정 임무 시점을 역추적하기 어려울 때 채택된다.

레코더는 최초 기록 시작 시간 t_{start} 를 기준으로, 현재 유입되는 메시지의 ROS 시간 t_{current} 와의 차이분인 델타 파라미터 \Delta t = t_{current} - t_{start} 가 설정된 max-bag-duration 상수를 능가하는지를 지속적으로 평가한다. 부등식이 성립하면 크기 기반 분할과 동일한 메커니즘으로 현재 로깅 세션을 밀봉하고 초기 t_{start} 값을 최신화하여 새로운 파일을 연다. 이 방식은 분할된 개별 백(Bag) 파일 하나가 ‘정확히 60초’ 또는 ’정확히 10분’의 비행 단위 기록을 내포하도록 강제하므로, 개발자가 비행 타임라인 로그북(Logbook)과 대조하여 특정 고장 시점의 파일을 발취(Extraction)하는 인간-시스템(Human-System) 해석성을 극대화한다.

4. 메타데이터(Metadata) 연동을 위한 시계열 파일 명명(Naming) 규약 구조

rosbag2는 크기나 시간 파라미터에 의해 단일 기록 세션이 다수의 하위 파일로 분산될 것을 전제로 하므로, 분할된 파일 묶음이 논리적으로 하나의 세션임을 보장하기 위해 엄격한 디렉토리 구조 및 시계열 파일 명명(Time-Series Naming Protocol) 규약을 채택한다.

레코드가 시작되면 사용자가 지정한(혹은 타임스탬프 기반 자동 생성된) 글로벌 디렉토리(Parent Directory) 하나가 생성된다. (예: flight_log_2026_01_01_15_00).
그리고 이 최상위 폴더 내부에 분할 생성되는 실제 스토리지 파일들은 글로벌 디렉토리 이름에 O_n 순차 식별자(Sequential Identifier)를 결합한 어휘적 명명법(Lexical Naming)을 따른다. (예: flight_log_2026_01_01_15_00_0.db3, ..._1.db3, ..._2.db3).
무작위로 분개된 파일들을 역순으로 재조립(Reassembly) 또는 연속 재생할 수 있는 근거는 동일 디렉토리 계층에 하나 유일하게 기록되는 metadata.yaml 파일에서 제공된다. 메타데이터는 분할된 하위 파일들의 상대 경로 배열, 묶음 전체의 최초/최종 시간 범위, 누적된 총 메시지 및 토픽 스키마 식별 정보를 정적 보존함으로써, 파편화된 다중 분할 파일들이 ROS 2 플레이백 체인 안에서 끊김 없는 연속형(Contiguous) 스트림 객체로 다루어지도록 하는 관제자 역할을 수행한다.
