51.8 기록 인스턴스 손실 복구(Recovery)를 위한 데이터 인덱싱 트리 파싱 및 파손 페이로드(Bag) 복구 프로토콜

1. 하드웨어 전력 단절에 의한 메타데이터 및 인덱스 파손 메커니즘

자율 비행 시스템에서 동역학적 한계치 돌파나 기계적 결함으로 인해 배터리 전력이 급작스럽게 차단(Hard Shutdown)되거나, OS 커널 패닉(Kernel Panic)이 발생할 경우 운영체제의 디스크 I/O 버퍼는 스토리지 매체로 동기화(Sync)되지 못한 채 증발한다.

이러한 물리적 단절(Disruption) 순간, rosbag2 엔진이 물리 디스크에 플러시 덤프 중이던 파일들은 EOF(End-of-File) 헤더 처리가 생략된 ‘언클린 클로즈(Unclean Close)’ 상태에 귀속된다. 백 파일 시스템에서 언클린 클로즈는 단순히 마지막 수 밀리초의 텔레메트리 바이트 상실을 의미하지 않는다. 재생(Playback) 시 시간 스펙트럼과 오프셋(Offset)을 결정짓는 최상위 포인터 테이블, 즉 인덱싱 트리(Indexing Tree)가 디스크에 물리적으로 기입되지 않는 다면(Write-failure), 파손 지점 이전의 수 기가바이트(GB) 분량의 완벽한 텔레메트리 구조체조차 ROS 2 미들웨어 상에서 읽을 수 없는 미할당 페이로드 구조로 전락하게 된다.

2. MCAP 포맷의 청크 스캐닝 기반 인덱싱 트리 재생성 체계

rosbag2의 시계열 최적화 백엔드인 MCAP(Message Capture)은 자체 복구(Self-Recovery) 내성을 태생적인 아키텍처 스펙으로 선언한다. MCAP 파일의 내부는 헤더, 메시지 레코드 블록, 그리고 인덱스 청크로 선형 분할되어 있다. 정상 종료 시 파일 종단부에 DataEnd와 청크 인덱스(Chunk Index) 테이블이 순서대로 기재되어 \mathcal{O}(1) 탐색을 보장하나, 비정상 종료 시 이 테이블들이 소실된다.

이러한 파손 인스턴스를 복구하는 핵심 프로토콜은 순차적 바이트 스트림 파싱(Sequential Byte Stream Parsing) 기반의 인덱싱 재생성이다. MCAP 복구 유틸리티 도구는 파일의 처음부터 끝까지 개별 청크의 매직 바이트(Magic Byte) 시그니처를 \mathcal{O}(N) 선형 탐색 스캔하여 손상되지 않은 독립 청크(Chunk) 단위의 메시지를 읽어들이고, 누락된 마스터 인덱스 트리를 백그라운드 메모리에서 새로 컴파일한 후 정상 포맷의 새 .mcap 파일로 재구성(Re-indexing)한다. 이 과정은 바이트 배열 레벨에서의 헥스(Hex) 스캔을 강제하므로 복구 시간은 파일 사이즈에 선형 비례하나, 파손 프레임을 제외한 모든 데이터를 완벽히 복원하는 결정론적 회복(Deterministic Recovery)을 보장한다.

3. SQLite3 백엔드의 WAL(Write-Ahead Logging) 복구 역학

관계형 테이블 아키텍처를 고수하는 SQLite3 백엔드(.db3) 데이터베이스 파손은 트랜잭션 단위의 저널링 체계를 통해 복구를 시도한다. rosbag2의 기본 세팅은 I/O 오버헤드 완화를 위해 Write-Ahead Logging (WAL) 모드를 활성화한다. WAL 모드에서 메시지 변경 사항은 즉각 데이터베이스 메인 트리 공간에 기입되는 대신, 인접 디렉토리의 .wal 파일에 선행 부착(Append)된다.

전력 단절이 발생하면 주 .db3 파일은 커밋되지 않은 상태의 낡은 트리에 머물고, 최신 로깅 패킷은 메모리 동기화가 이루어진 구간까지 .wal 인스턴스에 살아남는다. 이 인스턴스를 복구하기 위한 프로토콜은 ROS 2 툴 체인이 아닌 SQL 엔진 자체의 무결성 검증 체계(Integrity Check Mechanism)를 호출(Invoke)하는 것이다. 복구 데몬은 다음 부팅 시점에 해당 커넥션을 점유하여 .wal 저널에 기록된 선행 로그의 트랜잭션 페이지를 메인 파일로 롤 포워드(Roll-forward) 병합 체인 처리하며 단절 직전까지의 인스턴스를 살려낸다.

4. 파손 페이로드를 위한 사후 파싱 프레임워크 한계

인덱스 트리 재구축을 완수하더라도, 직렬화 바이트 구조에 패리티 비트(Parity Bit) 손상이나 메모리 스크래치(Scratch)가 발생한 최하단 페이로드의 자체 복원은 원천적으로 불가능하다. 만약 복구 과정 중 바이트 시그니처의 정합성 경계를 침범한 특정 청크 영역이 발견되면, rosbag2 수리 알고리즘은 해당 손상 블록만을 우회적 논리 폐기(Logical Discard)하고 다음 정상 청크 인스턴스와 맞물리는(Splice) 형태의 부분 복구(Partial Recovery) 전략을 취하게 된다.

결과적으로 복구된 텔레메트리 묶음 파일은 시계열 스펙트럼 내에 일시적 블랙아웃(Blackout) 간격을 남기게 되며, 이는 시뮬레이터(Gazebo)를 기반으로 한 로컬 궤적 플래너(Local Trajectory Planner) 검증 시스템에서 상태 추이 모델의 도약 오류(State Jump Error)를 일으킬 소지가 있으므로 사후 재생 모델 구동 시 결측치 보간(Missing Data Interpolation) 필터 매개 변수가 고려되어야 한다.
