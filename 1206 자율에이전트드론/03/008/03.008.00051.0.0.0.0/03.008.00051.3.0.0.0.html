51.3 연속 기록(Continuous Recording) 환경 하 메모리 버퍼 오버플로우 한계점 및 디스크 I/O 병목 추적 기법

1. 비동기 큐 포화(Queue Saturation)와 런타임 오버플로우 메커니즘

자율 에이전트 드론이 수 시간 이상 비행하며 rosbag2를 통해 광대역 텔레메트리(고해상도 이미지, 3D LiDAR 점군 등)를 연속 기록(Continuous Recording)할 때, 가장 치명적인 실패 모드는 비동기 큐(Asynchronous Queue)의 포화로 인한 메모리 버퍼 오버플로우(Memory Buffer Overflow) 현상이다. rosbag2 노드는 RMW(ROS Middleware)로부터 유입되는 메시의 직렬화 패킷(Serialized Packet)을 즉시 디스크에 쓰지 않고, 힙(Heap) 메모리 영역에 할당된 링 버퍼(Ring Buffer) 캐시에 임시 저장한다.

메시지가 버퍼로 유입되는 속도를 \lambda_{in} (단위 시간당 생성되는 바이트 크기)라 하고, 백그라운드 스레드가 이를 추출하여 물리 디스크에 기록하는 디스크 I/O 속도를 \mu_{out} 이라 정의하자. 이상적인 무중단 로깅(Uninterrupted Logging) 조건은 항상 \lambda_{in} < \mu_{out} 을 만족해야 하나, 복합 센서 스트림의 버스트(Burst) 유입이나 OS 파일 시스템의 페이지 캐시(Page Cache) 동기화 지연으로 인해 일시적 역전 현상(\lambda_{in} > \mu_{out})이 발생한다. 이 수급 불균형 주기가 지속되어 적재된 미처리 데이터 볼륨이 설정된 max_cache_size 임계값을 타격할 때, 큐 포화에 의한 데이터 유실 궤적이 개시된다.

2. 메모리 버퍼 오버플로우 한계점(Limit Point)의 대수적 모델링

오버플로우 도달 시간(Time-To-Overflow, T_{OOM})은 시스템이 설정한 캐시 크기 한계치 C_{max}와 그 시점까지의 누적 잉여 데이터 전송률의 적분으로 대수적 모델링이 가능하다. 특정 순간 t에 큐 내부에 대기 중인 잔여 페이로드 크기 Q(t)는 아래와 같이 정의된다.

Q(t) = \int_{0}^{t} (\lambda_{in}(\tau) - \mu_{out}(\tau)) d\tau

만약 연속적인 고부하 상태가 지속되어 \lambda_{in}과 \mu_{out}이 각각 평균 상수 \overline{\lambda} 와 \overline{\mu} 로 수렴하고 \overline{\lambda} > \overline{\mu} 상태일 때, 버퍼 오버플로우 선언 지점 T_{OOM}은 대략적으로 T_{OOM} = C_{max} / (\overline{\lambda} - \overline{\mu}) 의 선형 관계식으로 도출된다.

시스템 메모리의 파국(Kernel OOM Killer 개입)을 막기 위해 rosbag2는 이 T_{OOM} 한계점에 도달하는 순간 큐에 추가 인입되는 가장 최신 패킷을 드롭(Drop-newest)하거나, 반대로 큐의 가장 앞단에 적체된 오래된 패킷을 강제 폐기(Drop-oldest)하는 캐시 오버라이드(Cache Override) 인터페이스를 발동시켜 억지 균형을 유지하게 되며, 이는 필연적으로 재생(Playback) 시 시계열 결손 갭(Time-series Gap)을 발생시킨다.

3. 디스크 I/O 병목의 물리적 원인 및 정량적 드롭률 추정

\mu_{out} 의 가용 한계를 결정하는 디스크 I/O 병목 현상은 단순히 저장 매체(SD Card, eMMC, NVMe SSD)의 벤치마크상 순차 쓰기 스루풋(Sequential Write Throughput)에 의해서만 결정되지 않는다.

기록 중인 운영체제(OS) 아키텍처 상 파일 시스템 동기화 함수(예: fsync() 시스템 콜)가 주기적으로 블로킹 I/O(Blocking I/O)를 유발할 때, 드론 탑재 컴퓨터의 PCIe 버스 대역폭이 비전 파이프라인의 V4L2 드라이버 DMA 전송과 경합(Contention)을 벌이게 된다. 이러한 경합 역학(Contention Dynamics) 하에서 디스크 컨트롤러의 응답이 막히면(Stall), \mu_{out} 곡선은 0으로 일시적 추락(Plunge)을 겪는다. 이 정지 시간 동안 큐 내 누설량 L(t) 에 의한 드롭률(Drop Rate) \gamma 는 패킷 주입 빈도 계수와 직결되어 전체 데이터의 5 \sim 15\% 에 이르는 고정 손실을 팩터(Factor) 화한다.

4. 병목 현상 추적(Tracking) 및 시스템 스레드 모니터링 기법

지속 가능한 (Sustainable) 연속 로깅 파이프라인을 설계하기 위해서는 앞서 도출된 Q(t) 와 \gamma 현상을 조기에 식별하고 추적하는 진단 메커니즘을 구비해야 한다.

첫 번째 추적 기법은 ROS 2 런타임 이벤트 모니터링이다. rosbag2 노드의 내부 로거(Logger) 체계는 정해진 max_cache_size 타격 빈도와 누적 드롭 카운트(Dropped Message Count)를 진단 토픽(예: /rosout 및 터미널 stdout)으로 경고(Warn) 브로드캐스팅한다.
두 번째는 운영체제 레벨의 I/O 블로킹 추적이다. iostat, vmstat, 그리고 bcc 툴킷 등을 동원하여 스토리지 블록 디바이스(Block Device)의 await (평균 I/O 대기 시간) 및 디스크 활용률(Util %) 계측값을 교차 감시할 수 있다. 디스크 활용률이 100\%에 고정된 상태에서 드롭 경고가 동반될 경우 이는 완벽한 디스크 I/O 병목으로, max_cache_size를 기동 램(RAM)의 한계치까지 상향하거나 대역폭을 축소하는 압축 플러그인(Zstd 등) 계층으로 전환할 것을 확정하는 정량적 지표가 된다.
