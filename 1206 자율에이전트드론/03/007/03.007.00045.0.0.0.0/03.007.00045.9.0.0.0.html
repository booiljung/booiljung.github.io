45.9 보안 프로토콜(SROS2) 적용 네트워크 환경 내 암호화 토폴로지 탐색 제약 조건 분석 및 감사(Audit) 기법

1. 개요

대규모 자율 에이전트 드론 시스템이 사이버 물리 시스템(Cyber-Physical Systems, CPS)으로 고도화됨에 따라, 제어 및 센서 데이터의 기밀성(Confidentiality), 무결성(Integrity), 가용성(Availability)을 보장하기 위한 보안 프로토콜의 학술적, 기술적 도입이 필수로 자리 잡고 있다. ROS2는 이러한 요구사항을 충족하기 위해 Data Distribution Service(DDS) 보안 규격을 래핑(Wrapping)하는 형식의 SROS2(Secure ROS2) 패키지를 제공한다.

SROS2가 적용된 네트워크 환경에서는 공개 키 기반 구조(Public Key Infrastructure, PKI)를 통해 노드간 상호 인증(Authentication) 및 암호화(Encryption)가 수행되는데, 이 과정에서 발생하는 심각한 부작용 중 하나가 그래프 탐색 가시성의 상실이다. 본 장에서는 SROS2 적용 시 발생하는 암호화된 토폴로지 탐색 제한(Discovery Constraint)의 원인을 DDS 계층과 ROS2 애플리케이션 계층 사이의 인증 절차 관점에서 분석하고, 이를 관측, 진단, 평가할 수 있는 통계적 및 프로토콜 수준의 시스템 감사(Audit) 기법을 논의한다.

2. 암호화 토폴로지 탐색의 제약 메커니즘

전통적인 ROS2 환경에서 rqt_graph나 ros2 node list 같은 도구는 RTPS의 SPDP(Simple Participant Discovery Protocol) 통신을 수신하여 활성 노드 목록을 별도의 인증 절차 없이 캐싱한다. 반면, SROS2가 환성화된 DDS Participant는 토폴로지 탐색 절차가 분리되고 강화된다.

2.1 인증 및 접근 제어 정책(Enclaves) 매핑

SROS2에서는 시스템을 안전 영역(Enclave) 단위로 분할하여, 각 노드에 x.509 인증서와 접근 제어 문서를 매핑시킨다. 이때, DDS Security Plugin(Authentication Plugin, Access Control Plugin)은 검색 패킷(Discovery Packet) 자체를 서명 및 암호화하여 브로드캐스팅(Broadcasting)한다.

접근 제어 정책 문서(policy.xml)에 Allow Unauthenticated Discovery 옵션이 명시적으로 비활성화된 경우, 허가되지 않은 노드(예: 개발자의 디버깅 툴)는 해당 패킷의 payload를 복호화할 수 없으므로 Participant UID나 활성 토픽명 등 메타데이터 자체를 해독할 수 없다. 즉, 네트워크 상에 패킷이 전송됨에도 불구하고 ROS2 시스템 상에서는 존재하지 않는 내부 노드(Hidden Node)로 취급되는 가시성 단절 현상이 발생한다.

2.2 비대칭 키 교환에 따른 탐색 지연 시간(Latency)

암호화된 탐색 프로토콜은 SPDP 교환 직후 핸드셰이크(Handshake) 및 세션 키(Session Key) 교환 절차를 수반한다. 타원 곡선 암호(ECC)나 RSA 기반의 비대칭 키 연산은 드론의 내장형(Embedded) 온보드 컴퓨터(SBC)에서 밀리초(ms) 단위 이상의 지연을 유발할 수 있다. 이로 인해 단기적으로 동적 생성되는 노드(Short-lived Node)의 경우, 토폴로지가 탐색되어 rqt_graph에 반영되기 전에 프로세스가 소멸하여 그래프 관측의 일관성(Consistency)이 훼손될 수 있다.

3. 네트워크 가시성 회복 및 감사(Audit) 기법

탐색이 제한된 SROS2 환경에서 시스템 전체의 건전성(Health)을 평가하기 위해서는 보안을 타협하지 않으면서도 합법적인 모니터링 체계를 구축하는 감사 기법이 필수적이다.

3.1 감사 전용 Enclave 권한 모델 설계

디버깅 도구(rqt_graph, tf2_tools 등) 전용으로 특권 계층의 Enclave를 발급하는 정책 스키마를 설계한다. 이 감사 전용 인증서는 전체 토픽 영역에 대한 Subscribe 권한을 부여받지 않고, 오직 활성화된 Participant의 메타데이터(예: /rosout, /parameter_events 등 시스템 레벨 구독)와 노드 그래프 상태 정보만을 해독할 수 있는 최소 권한의 원칙(Principle of Least Privilege) 하에 구성된다.

3.2 허니팟(Honeypot) 및 패턴 기반 트래픽 감사 계량 모델

보안상 직접 복호화가 불가능한 상황에서는 네트워크 인터페이스(예: Wireshark의 RTPS 필터 사용) 레벨에서 암호화된 트래픽 패킷의 메타 정보를 통계적으로 분석하는 기법을 도입할 수 있다.
패킷의 크기 플로우, 생성 주기 빈도, Multicast 주소와 UDP 포트(예: domain_id에 종속된 7400 포트 대역)로 송신되는 시계열 데이터의 변동성을 계량화하여 토폴로지의 은닉된 변화 상태를 추론한다.

예를 들어 특정 암호화 패킷 시퀀스가 10Hz의 주기로 일정하게 브로드캐스트되고 있다면, 이는 센서(Lidar 또는 IMU)의 토픽 퍼블리셔(Publisher)가 정상적으로 생존하여 네트워크에 참여하고 있음을 나타내는 귀납적 증거로 활용될 수 있다.

3.3 로컬 데몬(Local Daemon) 프록시를 통한 상태 수집

각 드론의 내부 네트워크 인터페이스(localhost, 127.0.0.1) 내에서는 SROS2 암호화 강도를 낮추거나 별도의 로컬 IPC(Inter-Process Communication) 터널링을 구성하여, 노드 생존 상태 메타데이터를 수집하는 경량 프록시(Proxy) 데몬을 이중으로 운용한다. 이 프록시 에이전트는 주기적으로 자신의 상태를 중앙 GCS(Ground Control Station) 서버의 특화된 보안 채널을 거쳐 보고함으로써, SROS2의 보안 규칙을 훼손하지 않고도 실시간 시스템 그래프 감사 체계를 완비하게 된다.

결론적으로 SROS2는 자율 드론 체계의 필수적 방어막이나, 토폴로지 탐색 제약에 대한 근원적 한계를 지닌다. 따라서 엔지니어는 권한 위임(Delegation), 암호화 트래픽의 메타 분석, 투 트랙 모니터링 설계를 통해 신뢰 가능한 통신망 무결성 진단 기준을 확립해야 한다.
