58.3 객체 지향 언어 C++ 다형성(Polymorphism) 기반 hardware_interface::SystemInterface 플러그인 서브클래싱 구현론

ros2_control 아키텍처는 객체 지향 프로그래밍(OOP)의 핵심 원리인 다형성(Polymorphism)을 기반으로 하드웨어 드라이버를 추상화한다(사용 버전: ROS 2 Humble/Iron 기준). 개발자가 커스텀 드론의 모터 컨트롤러나 로봇 매니퓰레이터 인터페이스를 작성할 때, hardware_interface::SystemInterface와 같은 추상 기저 클래스(Abstract Base Class)를 상속(Subclassing)하여 가상 함수를 오버라이딩(Overriding)하는 방식으로 C++ 플러그인을 구현한다.

1. 시스템 인터페이스(SystemInterface)의 상속 구조

hardware_interface::SystemInterface는 C++의 순수 가상 함수(Pure Virtual Function)를 포함하는 인터페이스 명세서 역할을 수행한다. 이 클래스를 상속받은 사용자의 커스텀 클래스(예: MyDroneHardwareSystem)는 ros2_control의 리소스 관리자가 기대하는 필수 생명주기 제어 인터페이스와 데이터 교환 인터페이스를 모두 구체화해야 한다. C++의 상속에 의한 동적 바인딩(Dynamic Binding) 특성에 의하여, 상위 컴포넌트인 controller_manager는 실제파생 클래스 구현체의 메모리 레이아웃 등 세부 사항을 알 필요 없이 기저 클래스 포인터를 통해 read()와 write() 함수를 안전하게 다형적으로 호출할 수 있다.

다음과 같은 주요 멤버 함수들이 다형성 모델 하에서 반드시 파생 클래스 공간에 재정의(Override)되어야 한다.

  1. 초기화 및 생명주기 콜백 (Lifecycle Callbacks)
  • on_init(const hardware_interface::HardwareInfo & info): 외부 URDF 파일에서 파싱된 하드웨어 파라미터(관절 이름, 지원 인터페이스 종류 등)를 매개변수로 주입받아 내부 데이터 컨테이너를 초기화한다.
  • on_configure(), on_activate(), on_deactivate(), on_cleanup(): 로드된 플러그인이 Lifecycle State Machine의 각 전이 요구 조건에 적합하게 통신 소켓 관리 및 하드웨어 안전 제어를 수행하도록 구현을 제공한다.
  1. 인터페이스 바인딩 콜백 (Interface Exporting)
  • export_state_interfaces(): 하드웨어 컴포넌트에서 읽어들일 상태 값(Position, Velocity, Effort 등)을 가리키는 포인터 버퍼를 std::vector<hardware_interface::StateInterface> 형태로 배열화하여 반환한다.
  • export_command_interfaces(): 상위 제어기들이 최종 연산 결과를 기록할 명령 지령 버퍼의 주소를 std::vector<hardware_interface::CommandInterface> 구조로 구성하여 상위 런타임에 노출시킨다.
  1. 실시간 제어 루프 함수 (Real-Time Control Loop)
  • read(const rclcpp::Time & time, const rclcpp::Duration & period): 실제 하드웨어 통신 버스를 경유하여 가장 최신의 센서 피드백을 시스템 내부 변수로 이관한다.
  • write(const rclcpp::Time & time, const rclcpp::Duration & period): 계산 완료된 제어 입력 배열을 이기종 하드웨어 디바이스 노드로 물리적 전송한다. 런타임의 실시간 스레드 내부에서 호출되므로, 힙 메모리 할당(malloc 등)을 철저히 배제한 논블로킹(Non-Blocking) 실행 시간이 보장되어야 한다.

2. pluginlib 매크로를 통한 플러그인 동적 로딩 노출

작성 완료된 서브클래스 모듈은 컴파일 타임에 ros2_control의 핵심 바이너리에 직접 정적으로 링크(Static Link)되지 않는다. 대신, 프레임워크는 런타임 시점에 동적 공유 라이브러리(.so) 객체를 주소 공간으로 로드하는 pluginlib 개방형 아키텍처 패턴을 따른다.

개발자는 구현 파일 소스 코드의 최하단에 다음과 같은 의존성 팩토리 매크로를 선언하여 완성된 C++ 클래스를 패키지 플러그인 레지스트리에 등록한다.

#include "pluginlib/class_list_macros.hpp"

PLUGINLIB_EXPORT_CLASS(
  my_robot_namespace::MyDroneHardwareSystem,
  hardware_interface::SystemInterface)

상기 매크로는 기저 클래스인 hardware_interface::SystemInterface를 베이스로 삼아, 사용자의 파생 클래스 MyDroneHardwareSystem의 힙 메모리 인스턴스를 동적으로 생성할 수 있도록 지원하는 C-style 팩토리 함수 심볼을 OS 로더에 노출한다. 최종적으로 package.xml과 플러그인 명세 plugin.xml 내의 정의를 바탕으로, controller_manager는 C++의 다형성을 활용하여 로봇 메인 프로세스 재시동 없이 하드웨어 통합 제어 트리를 투명하게 직조할 수 있다.
