Chapter 58. ros2_control 프레임워크 기초 및 하드웨어 인터페이스(Hardware Interface) C++ 플러그인 작성법

1. 개요 및 학술적 배경

로봇 제어 시스템에서 고수준의 궤적 생성 알고리즘과 저수준의 모터 구동기(Actuator) 간의 물리적 괴리를 해소하는 것은 이식성(Portability)과 시스템 확장성을 결정짓는 핵심 과제이다. ros2_control 프레임워크는 이러한 실시간 의존성을 논리적으로 추상화(Abstraction)하는 ROS2의 표준 제어 미들웨어 계층이다. 본 장에서는 자율 드론 및 로봇 매니퓰레이터 제어의 근간이 되는 ros2_control의 아키텍처를 학술적으로 분석하고, 센서 및 기구학 벤더(Vendor) 종속적인 모터 드라이버를 ROS2 범용 생태계로 통합하기 위한 Hardware Interface C++ 동적 라이브러리(Plugin)의 설계 및 구현 방법론을 다룬다.

2. ros2_control 프레임워크 통합 아키텍처

ros2_control은 크게 컨트롤러 매니저(Controller Manager), 고수준 컨트롤러(Controllers), 그리고 하드웨어 추상화 계층(Hardware Abstraction Layer)의 3단 구조로 이루어져 있다.

2.1 컨트롤러 매니저(Controller Manager, CM)

컨트롤러 매니저는 ros2_control 생태계의 핵심 진입점(Entry Point)으로, 실시간 제어 루프(Real-Time Control Loop)의 스케줄링을 관장한다.

  • 라이프사이클(Lifecycle) 관리: 로드(Load), 설정(Configure), 시작(Start), 중지(Stop), 언로드(Unload) 등의 메타 상태 전이(State Transition)를 통해 복수의 컨트롤러 알고리즘 플러그인을 동적으로 메모리에 적재 및 소거한다.
  • 리소스 충돌 제어: 특정 조인트(Joint)나 액추에이터 노드에 대해 복수의 컨트롤러가 동시에 접근 권한을 상호 요구할 때, 리소스 락(Resource Lock) 매커니즘을 활성화하여 기구학적 파손을 방지한다.

2.2 하드웨어 리소스 매니저(Hardware Resource Manager)

URDF(Unified Robot Description Format) XML 파일 내의 <ros2_control> 태그로 명시된 기구학적 종속 트리를 파싱하여, 논리적 조인트의 상태 벡터(State Interface)와 지령 벡터(Command Interface)를 C++ 연속 메모리 풀(Memory Pool) 상에 포인터로 매핑한다.

3. Hardware Interface C++ 플러그인 설계 방법론

플러그인 확장을 통해 인테그레이터(Integrator)는 CAN, EtherCAT, UART 등 독립적인 마이크로컨트롤러(MCU) 프로토콜을 ros2_control 스택에 통합할 수 있다. 하드웨어 위상에 따라 System(포괄형), Sensor(입력 전용), Actuator(출력 전용)의 3가지 추상화 모델 중 하나를 상속받아 구현한다.

3.1 C++ 생명주기 역학 및 가상 함수(Virtual Function) 오버라이딩

pluginlib 아키텍처 기반 하에 런타임 동적 라이브러리(.so)로 적재되는 클래스를 구축하기 위해서는 다음 4가지 핵심 생명주기 가상 함수를 필수적으로 오버라이딩(Overriding)해야 한다.

  1. on_init(): 클래스 인스턴스화 직후 단 1회 호출되며, URDF <hardware> 트리에 파싱된 내부 <param> 매개변수(예: 직렬 통신 포트 번호 \dev\ttyACM0, 보드레이트 등)를 파싱하여 포트를 바인딩한다. 에러 체크 로직 성공 시 CallbackReturn::SUCCESS를 반환해야 한다.
  2. export_state_interfaces() / export_command_interfaces(): 해당 하드웨어가 제공 가능한 제어 척도(예: position, velocity, effort 등) 각각에 대해 클래스 내부 멤버 변수(double)의 주소값 벡터(std::vector<hardware_interface::StateInterface>)를 매니저 메모리로 반환하여 심볼릭 연결(Symbolic Link)을 수행한다.
  3. on_activate() / on_deactivate(): 물리 계층과의 소켓을 개방(Open)하고 모터 레지스터 파라미터(PID 이득 등)를 초기화(Initialize)하며 통신 종료 시 안전 정지(Halt) 루틴을 수행한다.

3.2 실시간 RT 제어 루프: read()와 write() 구현

가장 높은 주파수(예: 1kHz ~ 4kHz)로 RT 스레드에서 무한 호출되는 구동 블록이다.

  • 동적 할당 제약: RT(Real-Time) 제약조건(예: PREEMPT_RT 커널)을 만족하기 위해, 해당 함수 내에서는 페이지 폴트(Page Fault)를 유발할 수 있는 동적 힙 메모리 할당(new, std::vector::push_back, 동적 형변환)이나 블로킹(Blocking) 커널 시스템 콜이 철저히 배제되어야 한다.
  • read(): 드라이버 계층으로부터 엔코더 펄스(Pulse) 및 션트 저항의 전류 피드백 데이터를 비동기 역직렬화하여 export_state_interfaces에 등록된 내부 캐시 변수를 갱신한다.
  • write(): 매니저 컨트롤러가 산출 하달한 상태 지령(Command)을 변수에서 추출하여 버스 프로토콜 패킷으로 직렬화한 후 피지컬 모터 계층으로 전송한다.

4. CMake 및 Pluginlib 매크로 익스포트 연동 체계

작성된 C++ 추상화 드라이버가 ROS2 생태계에서 동적 플러그인(Dynamic Plugin)으로 합법적 소환이 가능하기 위해서는 매크로 기반의 링킹(Linking) 프로토콜을 준수해야 한다.

  • PLUGINLIB_EXPORT_CLASS 매크로: 소스 파일 최하단에 PLUGINLIB_EXPORT_CLASS(custom_namespace::CustomHardwareInterface, hardware_interface::SystemInterface)의 상속 관계를 명시하여 Base-Derived 관계의 RTTI(Run-Time Type Information) 플러그인 종속 등록 테이블을 생성한다.
  • 메타데이터 선언: 패키지 내 custom_hardware.xml 상세 라이브러리 기술서를 작성한 뒤, CMakeLists.txt의 ament_target_dependencies 및 pluginlib_export_plugin_description_file 구문을 선언하여 의존성 충돌 및 버전 데드락이 없는 교차 컴파일(Cross-Compilation) 튜닝 체제를 완료한다.
