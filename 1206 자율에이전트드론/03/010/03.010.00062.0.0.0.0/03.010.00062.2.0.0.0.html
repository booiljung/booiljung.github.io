62.2 객체 지향 다형성(Polymorphism) 기반 기반 클래스(Base Class) 및 파생 클래스(Derived Class) 간 인터페이스 상속 규약

1. 다형성(Polymorphism)을 통한 소프트웨어 아키텍처 결합도 최소화

ROS 2의 pluginlib 확장은 C++의 객체 지향 패러다임(Object-Oriented Paradigm) 중 다형성 메커니즘을 핵심 기반으로 삼는다. 자율 에이전트 드론이 운용하는 센서, 구동기, 혹은 인지 알고리즘 모듈은 제각기 다른 내부 상태와 제어 로직을 갖는다. 코어 시스템이 이러한 이기종 모듈들을 일관되게 통제하기 위해서는 개별 모듈의 구체적인 구현(Implementation)에 강하게 의존(Tight-coupling)해서는 안 된다.

따라서 아키텍처 설계자는 오직 추상화된 기반 클래스(Base Class)만을 시스템의 프론트엔드에 노출시키며, pluginlib의 런타임 로더(Loader)는 파생 클래스(Derived Class)를 메모리에 인스턴스화하더라도 이를 부모 클래스의 포인터 속성으로 업캐스팅(Upcasting)하여 반환한다. 이러한 인터페이스 상속 규약을 통해 시스템은 컴파일된 바이너리의 교체 없이, 메시지를 송수신하는 규약만 합의된 형태라면 어떠한 파생 모듈이든 논리적 충돌 없이 운용할 수 있다.

2. 기반 클래스(Base Class) 설계의 학술적 제약

플러그인 아키텍처의 기준점이 되는 기반 클래스는 엄격한 C++ 상속 문법과 자원 해제 규칙을 준수하여 설계되어야 한다.

2.1 순수 가상 함수(Pure Virtual Function) 명세

기반 클래스는 파생 클래스가 준수해야 할 계약(Contract) 역할을 한다. 드론 페이로드의 다이어그노스틱 획득이나 행동 명령 하달과 같은 필수 동작은 반환 타입, 함수 이름, 매개변수 리스트가 정의된 가상 함수(Virtual Function)로 선언된다. 이때, 특정 동작 방식이 정의될 수 없는 완전한 추상형을 강제하기 위하여 C++의 virtual DataType MethodName() = 0; 형태인 순수 가상 함수를 활용한다. 이렇게 구성된 기반 클래스는 하나 이상의 순수 가상 함수를 지닌 추상 클래스(Abstract Class)가 되며, 그 자체로는 객체 인스턴스를 직접 생성할 수 없게 되어 논리적 오류를 컴파일 타임에 차단한다.

2.2 가상 소멸자(Virtual Destructor) 선언의 필수성

다이나믹 로딩된 파생 객체는 소멸(Destruction) 단계에서 기반 클래스 포인터를 통해 삭제(delete) 연산에 노출된다. 만약 기반 클래스의 소멸자가 가상(Virtual)으로 지정되지 않을 경우, C++ 컴파일러는 객체의 실제 메모리 타입과 무관하게 포인터의 타입인 기반 클래스의 소멸자만을 호출(Early Binding)한다. 이는 파생 클래스가 고유하게 할당한 동적 자원 또는 통신 소켓의 논리적 결착을 해제하지 못하게 만들어 심각한 메모리 누수(Memory Leak) 단위로 발전한다. 따라서 모든 종류의 pluginlib 기반 클래스는 반드시 명시적인 가상 소멸자 선언(virtual ~BaseClass() = default;)을 포함해야 한다.

3. 파생 클래스(Derived Class)의 구현 및 오버라이딩 규약

서드파티 모듈, 즉 파생 플러그인 생산자는 기반 클래스로부터 상속을 선언하고 구체화된(Concrete) 제어 알고리즘을 구현할 시스템 구축 책임이 있다.

3.1 명시적 오버라이드(override) 키워드 활용

파생 클래스는 기반 클래스가 정의한 순수 가상 함수의 시그니처(Signature)를 단 하나의 예외(Type, Argument, Const 속성 등) 없이 정확하게 일치시켜 재정의하여야 한다. 이때 컴파일러 수준에서 해당 함수가 부모 클래스의 가상 함수를 올바르게 대체하고 있는지 검증하기 위해, override 키워드를 기술하는 것이 현대(Modern) C++의 표준 지침이다. 이는 부주의한 인터페이스 불일치로 인해 완전히 새로운 함수가 하이딩(Hiding)되는 아키텍처 결함 생성을 원천 방지한다.

3.2 의존성 관리 영역의 은닉화 (Encapsulation)

파생 클래스는 기반 클래스의 가시성(Visibility) 인터페이스에만 국한되지 않는다. 실제 물리적 하드웨어 센서 드라이버와의 시리얼(Serial)/I2C 통신 핸들, 또는 자체적인 스레드 공간 배정 등 파생 클래스가 독립적으로 요구하는 고유 자원들은 외부로 노출되지 않는 private 영역 안에서 독립된 형태로 은닉되어야(Encapsulated) 한다. 이를 통하여 pluginlib이 어떠한 파생 모듈로 상호 호환 대체를 시도하더라도, 스코프(Scope) 외부에 미치는 사이드 이펙트(Side Effect)의 확산을 철저히 억제할 수 있다.

이처럼 C++ 언어계층에서 강제되는 형태의 객체 구조화 메커니즘을 명백히 파악하고 준수하는 것은, 무인 비행 모델 등과 같이 미션-크리티컬(Mission-Critical)한 실시간 환경 위에서 동작하는 ROS 2 (Humble Hawksbill 버전) 확장 프레임워크 설계의 근간이 된다.
