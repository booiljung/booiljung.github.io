14.1.2 실시간 운영체제(RTOS)와 범용 OS 간의 타이밍 비대칭성

ROS2 기반 오프보드(Offboard) 클러스터와 PX4 오토파일럿이 융합된 현대적 자율 드론 아키텍처는 필연적으로 상이한 스케줄링 철학을 가진 두 운영체제가 직렬(Serial) 버스로 결합되는 구조를 낳는다. 즉, 기체의 고도와 역학적 안정성을 직접 제어하는 픽스호크(Pixhawk) 기반의 비행 제어기는 넛엑스(NuttX)와 같은 하드 실시간 운영체제(Hard RTOS) 위에서 동작하는 반면, 복잡한 신경망 처리 및 광역망(WAN) 통신을 전담하는 컴패니언 컴퓨터(Companion Computer)는 리눅스(Linux) 기반의 범용 운영체제(General-purpose OS) 위에서 구동된다. 이 두 운영체제는 작업(Task)의 수용, 자원 할당, 그리고 인터럽트(Interrupt) 처리 방식에서 근본적인 비대칭성(Asymmetry)을 띄며, 이 비대칭성은 프레임워크 간 데이터를 매개하는 브리지 계층에서 치명적인 타이밍 병목 및 통신 지터를 유발한다.

1. 하드 실시간 운영체제(RTOS)의 결정론적(Deterministic) 아키텍처

비행 제어기에 탑재되는 RTOS의 최우선 목적은 시스템 처리량(Throughput)의 극대화가 아니라, 최고 우선순위 작업에 대한 응답 시간의 결정론적 보장(Deterministic Guarantee)이다.

  • O(1) 스케줄링 및 절대적 선점(Strict Preemption): 넛엑스 커널은 스레드 큐(Thread Queue)를 탐색할 때 소요되는 시간이 작업의 개수에 무관하게 상수 시간 O(1)으로 고정되어 있다. 하드웨어 인터럽트(예: IMU 센서 데이터 수신)가 발생하면, 현재 실행 중인 어떠한 백그라운드 연산이라도 즉각 멈추고 제어 루프 컨텍스트로 абсолю적인 선점(Preemption)이 이루어진다.
  • 예측 가능한 최악 실행 시간(WCET): 메모리 할당을 가상 메모리 스와핑 없이 물리적 힙(Heap) 공간에 정적으로 수행하여 페이지 폴트(Page Fault)를 배제한다. 결과적으로 내부 상태 추정 및 모터 PWM 출력에 소요되는 최악 실행 시간(Worst-Case Execution Time, WCET)은 1 마이크로초 단위까지 수학적으로 예측 가능해진다.

2. 범용 OS의 타이밍 비결정성(Non-Determinism)과 오버헤드

반면, 컴퓨터 비전 기반의 목표 추적이나 전역 경로 계획을 구동하는 범용 리눅스 커널은 다중 사용자와 다중 응용 프로그램 간의 공정한 자원 분배 및 평균 성능(Average Performance) 향상에 최적화되어 있다. 이로 인해 리눅스 커널 위에서 동작하는 ROS2 제어 노드는 필연적인 타이밍 비결정성을 내포하게 된다.

  1. 커널 락(Kernel Lock)과 비선점형 임계 영역: 일반적인 리눅스 커널에서 시스템 콜(System Call)이나 드라이버 인터럽트가 처리되는 임계 영역(Critical Section) 내부에서는 외부 스케줄러에 의한 선점이 제한된다. 이는 사용자 공간(User Space)에 존재하는 최고 우선순위 ROS2 제어 스레드마저도, 다른 프로세스의 파일 시스템 I/O 작업이나 네트워크 수신 처리가 끝날 때까지 블로킹(Blocking)되게 만든다.
  2. 동적 메모리 관리 및 데몬 간섭: 가상 메모리(Virtual Memory) 아키텍처는 페이지 스와핑(Swapping)을 유발하며, 이는 수십에서 수십 밀리초(ms)에 이르는 막대한 대기 시간을 생성한다. 또한 커널 자체의 타이머 데몬, 네트워크 스택 처리 데몬들이 주기적으로 CPU 자원을 점유하면서 응용 계층의 제어 주기를 불규칙하게 밀어내는 지터(Jitter)를 양산한다.

3. 타이밍 비대칭성이 매개 계층(Bridge)에 미치는 파급 효과

비행 제어기(RTOS)는 250Hz 이상의 고정된 주기로 오프보드 컴퓨터의 제어 설정점(Setpoint)을 기대하지만, 오프보드 컴퓨터(범용 OS)는 스케줄링 지터로 인해 어떤 주기에는 2개의 데이터를 연속으로, 어떤 주기에는 10ms 이상 침묵하게 되는 버스트(Burst) 특성을 출력한다.

이러한 타이밍 비대칭성은 두 시스템을 연결하는 Micro XRCE-DDS 또는 MAVLink 브리지 버퍼 구조에 직격타를 가한다. 수신단인 비행 제어기 입장에서는 정해진 처리 사이클에 제어 데이터가 부재할 경우 기체 역학 모델의 외삽(Extrapolation)을 오버헤드로 수행해야 하며, 데이터가 밀려 들어올 경우 최신 데이터를 식별하기 위해 수신 큐(Buffer Queue)를 비워내는(Flush) 과정에서 추가 연산 주파수가 낭비된다. 따라서 실시간 운영체제와 범용 OS 간의 브리징 아키텍처는 양단의 비대칭적인 시간 흐름을 논리적 데드라인 범위 내에서 동기화하고 흡수하는 지능적 버퍼(Intelligent Buffer) 및 타이머 동기화 전략을 필수적으로 요구한다.
