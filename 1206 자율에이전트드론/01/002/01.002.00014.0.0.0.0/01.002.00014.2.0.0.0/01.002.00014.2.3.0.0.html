14.2.3 IPC 메시지 큐 병목에 의한 통신 지연 인과성

자율 드론 시스템에서 브릿지 통신 계층의 지연 시간은 오프보드 임무 컴퓨터 내부의 프로세스 간 통신(IPC, Inter-Process Communication) 아키텍처와 미들웨어의 메시지 큐(Message Queue) 병목 현상에 깊이 뿌리내리고 있다. 비행 제어기에서 생성된 고주파수 텔레메트리 데이터가 컴패니언 컴퓨터의 ROS2 데이터 버스로 진입하거나 역방향으로 제어 명령이 전달될 때, 이 데이터는 단일 메모리 접근으로 처리되는 것이 아니라 복잡한 IPC 파이프라인과 버퍼 큐를 거치며 물리적인 시간 지연(Latency)을 누적시킨다.

1. ROS2 로컬 통신 오버헤드와 IPC 메커니즘

ROS2 노드와 브릿지 에이전트(예: Micro XRCE-DDS Agent) 간의 데이터 교환 방식은 근본적인 지터의 발원지이다.

  • 루프백(Loopback) 네트워크 스택의 한계: 동일한 리눅스 머신 내부에 위치한 노드 간 통신이라 할지라도, 대다수의 기본 DDS 구현체 구현은 로컬 영역 네트워크(LAN)와 동일하게 UDP 루프백(127.0.0.1) 인터페이스를 경유한다. 이는 데이터가 유저 공간에서 커널 공간의 소켓 버퍼로 복사되고, 네트워크 스택(TCP/IP 계층)을 순회한 후 다시 수신 프로세스의 유저 공간으로 복사되는 막대한 컨텍스트 스위칭 및 메모리 복사 오버헤드를 동반한다.
  • 공유 메모리(Shared Memory) 전송의 병목: Iceoryx와 같은 제로 카피(Zero-Copy) 공유 메모리 기반 IPC를 활용하더라도, 다수의 퍼블리셔(Publisher) 스레드가 브릿지 노드의 단일 수신 큐(Receive Queue)에 동시다발적으로 접근할 때 발생하는 뮤텍스(Mutex) 락 경합은 대규모 트래픽 하에서 대기 지연(Blocking Delay)을 유발한다.

2. 미들웨어 메시지 큐의 버퍼블로트(Bufferbloat) 현상

브릿지 노드 내부에는 직렬 버스(UART)로 내보내기 전 데이터를 임시 보관하는 송신 큐(Transmit Queue)가 존재한다. UART의 직렬화 및 전송 속도(보드레이트, Baudrate)가 ROS2 상위 애플리케이션의 데이터 발행 속도를 따라가지 못할 때, 메시지 큐 내부에 발생하는 병목은 제어망의 파국을 초래한다.

  1. 헤드 오브 라인 블로킹(Head-of-Line Blocking, HoL): 큐 내부가 우선순위 구분이 없는 선입선출(FIFO) 구조일 때, 페이로드가 큰 오도메트리(Odometry)나 상태 메시지 수십 개가 송신 큐를 채우게 되면, 기체의 생존에 직결되는 고주파수 짧은 제어 명령(예: 충돌 회피용 TrajectorySetpoint)이 이들에 가로막혀 적시에 전송되지 못하는 HoL 블로킹이 발생한다.
  2. QoS KEEP_ALL 정책의 치명성: ROS2 QoS 설정에서 메시지 유실을 막기 위해 KEEP_ALL 이력(History) 정책과 RELIABLE 신뢰성 정책을 결합할 경우, 병목 상황에서 송신 큐의 뎁스(Depth)가 끝없이 팽창한다. 이는 동적 메모리 할당 오버헤드를 극한으로 끌어올리며, 결과적으로 비행 제어기가 현재 시점과 수백 밀리초 단위로 괴리된 과거의 “묵은(Stale)” 제어 데이터를 순차적으로 수신하여 적분기 발산(Integral Windup)을 무조건적으로 겪게 만드는 버퍼블로트(Bufferbloat) 지연을 촉발한다.

따라서 실시간 브릿지 아키텍처에서는 IPC 지연을 상수화하기 위해 엄격한 대역폭 제어와 함께 데이터 수명(Lifespan) 및 마감 시간(Deadline) QoS 정책을 융합하여, 큐 내부에 누적된 시계열 데이터가 물리적 효용성을 잃은 즉시 폐기(Drop)되도록 설계하는 시스템 통합 최적화가 필수적이다.
