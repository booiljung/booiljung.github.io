14.3.3 NIC 인터럽트 빈도 제어 및 폴링 모드 전환

자율 드론 시스템에서 컴패니언 컴퓨터와 비행 제어기(Flight Controller) 간의 필수 텔레메트리 통신 방식이 직렬(Serial) 인터페이스 기반에서 이더넷(Ethernet) 기반의 고속 네트워크망으로 진화함에 따라, 네트워크 인터페이스 컨트롤러(NIC, Network Interface Controller)의 하드웨어 통제 방식이 전체 에이전트의 실시간 성능에 직접적인 물리적 영향을 미치기 시작했다. Micro XRCE-DDS 또는 향후 도입될 RTPS 브릿지 통신 환경에서, 리눅스 커널의 기본적인 물리 NIC 인터럽트 처리 방식은 대량의 데이터 처리량(Throughput)만을 달성하기 위해 설계되었으므로 드론 통신에 미세한 밀리초 단위의 통신 지터(Jitter)를 필연적으로 유발한다. 이를 근본적으로 해소하기 위해 NIC 하드웨어의 물리적 인터럽트 빈도를 직접 제어하고 커널의 폴링(Polling) 모드 구조를 역이용한 네트워크 튜닝을 수행한다.

1. 하드웨어 인터럽트 병합(Interrupt Coalescing) 기능 비활성화

표준 리눅스의 범용 네트워크 스택 및 현대적인 대다수의 상용 NIC 하드웨어 칩셋은, CPU로 유입되는 하드웨어 인터럽트의 잦은 발생 빈도를 억제하여 물리적 에너지 효율성을 증대시키고 내부 시스템 버스(Bus) 점유율을 개선하기 위해 인터럽트 병합(Interrupt Coalescing)이라는 통신 대기 기능을 기본적으로 활성화 상태로 둔다. 이는 NIC 내부 버퍼 메모리 공간에 수신된 개별 패킷이 일정 개수 이상 누적되거나 설정된 운영 타이머(예: 최소 수십에서 수백 마이크로초)가 최종적으로 만료될 때까지 커널로 향하는 송신 신호를 인위적으로 대기시키는 지연 기술이다.

일반적인 대용량 파일 다중 전송 서버 환경에서는 이 지연 기능이 필수적이지만, 1 밀리초 단위로 수십 바이트 이하 크기의 기하학적 위치 및 자세 제어 명령을 실시간으로 고속 송수신해야 하는 드론 제어 루프 계층에서는 이 병합 기능이 오히려 치명적인 인위적 통신 대기 시간(Artificial Latency)을 양산한다. 이를 관리적 차원에서 억제하기 위해 리눅스 커널 수준의 ethtool 터미널 명령어 유틸리티를 활용하여 해당 물리 NIC 이더넷 장치의 rx-usecs(수신 인터럽트 타이머 지연 시간) 및 rx-frames(수신 인터럽트 발동 누적 프레임 단위) 내부 매개변수 값을 0에 수렴하는 물리적 최소치로 강제 변경해야 한다. 이 조치를 통해 네트워크 컨트롤러 하드웨어 칩에 이더넷 데이터가 당도하는 즉각적인 시점에 곧바로 커널 인터럽트가 호출되도록 통신 계층을 재구성하여, 패킷 도착 지터 변화폭을 안정적이고 이상적으로 축소한다.

2. NAPI(New API) 기반 하이브리드 수신 아키텍처와 루프 모드 제어

단일 하드웨어 인터럽트를 즉시 발생시키는 환경을 조성했더라도, 드론의 센서 통신 루프 주기가 1kHz 이상으로 고도화될 경우, 역방향으로 NIC 하드웨어가 1초 당 수 천 번 이상의 과도한 하드웨어 인터럽트를 촉발시키면서 시스템을 무력화시킬 위험성이 있다. 이는 CPU 명령어 루틴 구조가 끝없는 물리 인터럽트 폭풍(Interrupt Storm)에 매몰되어 프로세스가 전혀 갱신되지 못하는 라이브락(Livelock) 시스템 마비 현상을 초래한다.

이러한 연산 마비의 부작용을 기술적으로 방어하고 실시간 패킷 처리를 균형적으로 수행하기 위해 가장 최근 배포되는 리눅스의 커널 네트워크 스택은 NAPI(New API)라는 명칭의 하이브리드 수신 아키텍처를 도입하여 탑재하였다. NAPI 메커니즘은 최초의 수신 네트워크 패킷이 도착한 시점에는 단 한 번만의 하드웨어 인터럽트 발생만을 허용하여 커널의 즉각적 주의를 환기한 뒤, 이어지는 후속 하드웨어 인터럽트 자체를 NIC 단말에서 물리적으로 일시 중지시키고 커널 수준의 소프트웨어적 순환 검사 모드인 소프트웨어 폴링(Software Polling) 루틴 체제로 전환해 네트워크 수신 버퍼 공간을 빠르게 연속 검수한다.

자율 드론 통신망을 위한 실시간 PREEMPT_RT 구동 커널 환경에서는 이러한 진보된 NAPI 메커니즘의 운영 체제 커널 제어 매개변수인 netdev_budget(단일 소프트웨어 폴링 연속 루프에서 NIC에서 꺼내올 수 있는 최대 수신 프레임 수치)과 netdev_budget_usecs(커널이 다른 작업 스위칭 없이 이 폴링 상태만을 지속적으로 유지할 허용 최대 마이크로초 시간) 값을 시스템 제어 고속 통신 주기에 맞추어 세밀하게 조정 튜닝한다. 관련 매개변수를 적절한 수준으로 축소 강제하면, 네트워크 폴링 스레드가 장시간 특정 코어의 시간 자원을 불필요하게 독점하는 비정상 행위를 스케줄러단에서 제한하여 기타 ROS 2 메인 통제 노드들의 실시간 CFS 스케줄링 과정을 침해하지 않으면서도 수 밀리초 오차 이내로 통신 패킷 응답성을 일관되게 유지할 수 있다.
