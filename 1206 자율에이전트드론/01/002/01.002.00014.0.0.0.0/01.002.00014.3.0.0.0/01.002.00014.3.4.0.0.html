14.3.4 동적 메모리 할당 제한 및 페이지 폴트 방지 메커니즘

자율 에이전트 드론을 위한 실시간 운영체제(RTOS) 및 PREEMPT_RT 패치가 적용된 리눅스 환경에서, 비결정적(Non-deterministic)인 지연을 유발하는 주요 원인 중 하나는 동적 메모리 할당과 페이지 폴트(Page Fault)의 발생이다. 시스템 실행 중 발생하는 메모리 관리는 커널 스케줄러의 개입을 초래하며, 최악 실행 시간(Worst-Case Execution Time, WCET) 예측을 불가능하게 만든다. 따라서 하드웨어 타이밍 보장과 브리지 스레드의 실시간성 확보를 위해 메모리 관리 체계는 초기화 단계에서 정적 할당을 완료하고, 런타임 중에는 페이지 폴트가 발생하지 않도록 강제하는 메커니즘이 필수적으로 적용되어야 한다.

1. 런타임 동적 메모리 할당의 비결정성

C/C++ 표준 라이브러리의 malloc() 또는 new 연산자는 힙(Heap) 영역에서 적절한 크기의 가용 메모리 블록을 탐색한다. 이 과정은 메모리 단편화(Fragmentation) 상태에 따라 탐색 시간이 가변적이다.

  1. 커널 락(Lock) 경합: 동적 메모리 할당은 멀티스레드 환경에서 힙 영역의 데이터 구조에 접근하기 위해 내부적인 뮤텍스(Mutex) 또는 스핀락(Spinlock)을 사용한다. 이는 스레드 간의 락 경합을 유발하여 우선순위 역전(Priority Inversion) 문제로 이어질 수 있다.
  2. 시스템 콜 오버헤드: 가용한 힙 영역이 부족할 경우, 시스템 콜인 sbrk() 또는 mmap()이 호출되어 사용자 공간으로 전환(Context Switch)이 발생하며, 이는 수십에서 수백 마이크로초 단위의 지연을 초래한다.

이러한 문제를 방지하기 위해 실시간 브리지 제어 루프 내부에서는 표준 동적 메모리 할당 함수의 사용을 엄격히 금지한다. 통신 메시지 객체 및 버퍼는 사전에 할당된 정적 메모리 풀(Static Memory Pool)을 사용하여 관리되어야 한다.

2. 메모리 풀(Memory Pool)과 객체 캐싱(Object Caching) 패턴

런타임 동적 메모리 제한으로 인한 데이터 처리의 유연성 저하는 메모리 풀 패턴을 통해 보완한다.

  1. 고정 크기 블록 할당: ROS2 메시지 패키지와 Micro XRCE-DDS 브리지 통신에 사용되는 데이터 구조체의 최대 크기를 계산하여 시스템 초기화 시 미리 배열 형태로 할당한다.
  2. TLSF(Two-Level Segregated Fit) 알고리즘 적용: 다양한 크기의 메모리 블록 요청이 필요한 경우, 실시간 환경을 위해 설계된 TLSF와 같은 결정론적 메모리 할당기를 제한적으로 사용한다. TLSF는 단편화를 억제하고 할당 및 해제 작업을 O(1)의 시간 복잡도로 보장한다.
  3. 가비지 컬렉션(Garbage Collection) 배제: 자율 에이전트 드론 구현에 있어 비결정성 중단(Stop-the-world)을 유발하는 가비지 컬렉터 기반 언어 환경은 비행 제어 루프 영역에서 배제되며, 명시적인 수명 주기(Lifetime) 관리가 요구된다.

3. 페이지 폴트 제어와 메모리 잠금(Memory Locking)

가상 메모리(Virtual Memory) 시스템에서 애플리케이션에 할당된 메모리는 물리 메모리(RAM)에 즉시 매핑되지 않고, 최초 접근 시 메인 메모리로 적재되는 요구 페이징(Demand Paging) 방식으로 동작한다. 이 과정에서 발생하는 마이너 페이지 폴트(Minor Page Fault)와 스왑 공간으로 내보내진 메모리를 다시 읽어오는 메이저 페이지 폴트(Major Page Fault)는 수 밀리초(ms) 이상의 디스크 I/O 대기 시간을 발생시킨다. 이는 마이크로초 단위의 제어 주기를 갖는 비행 제어 시스템의 치명적인 실패 원인이 된다.

이를 방지하기 위해 브리지 프로세스는 mlockall() POSIX 시스템 콜을 사용하여 프로세스의 가상 주소 공간 전체를 물리 메모리에 고정(Pinning)한다.

  • MCL_CURRENT 및 MCL_FUTURE 플래그: mlockall(MCL_CURRENT | MCL_FUTURE)를 호출하면 현재 프로세스에 할당된 모든 페이지뿐만 아니라, 향후 확장될 스택(Stack)이나 힙(Heap) 영역까지 물리 메모리에 고정된다.
  • 스택 사전 할당(Stack Prefaulting): 스레드 생성 시 스택 메모리 역전에 의한 지연을 방지하기 위해, 스택 크기를 pthread_attr_setstacksize()로 명시하고, 임의의 바이트를 기록하여 스택 페이지를 물리 메모리에 사전 매핑한다. 메모리 잠금 이후 이루어지는 이 과정은 스레드가 실시간 스케줄링으로 진입하기 전에 완료되어야 한다.

4. OOM(Out-Of-Memory) 킬러 메커니즘 우회 및 시스템 격리

리눅스 커널은 시스템 전체의 물리 메모리가 부족해질 경우 OOM Killer를 호출하여 가장 많은 메모리를 점유하거나 우선순위가 낮은 프로세스를 강제 종료(SIGKILL)시킨다.

  1. oom_score_adj 설정: 실시간 통신 브리지 및 비행 제어 노드가 OOM Killer에 의해 예기치 않게 종료되는 것을 방지하기 위해, /proc/[pid]/oom_score_adj 파일 시스템 노드의 값을 최소치인 -1000으로 설정한다. 이를 통해 커널이 해당 프로세스를 종료 대상에서 완전히 제외하도록 구성한다.
  2. 과할당(Overcommit) 정책 제어: 리눅스의 커널 메모리 과할당 방식을 제어하는 vm.overcommit_memory 커널 파라미터를 수정하여 시스템이 물리적으로 가용한 범위를 초과하는 메모리 할당 요청을 원천적으로 차단한다.
