18.5.1 Launch 파이프라인에서의 계층형 설정 파일 병합 오버라이드 규칙

ROS 2(Robot Operating System 2)의 런타임 환경에서 시스템의 상태와 행위를 결정하는 파라미터(Parameter)는 Launch 파이프라인을 통해 초기화된다. 복잡한 자율 에이전트 드론 시스템은 다수의 센서, 제어기, 통신 모듈 노드로 구성되며, 각 노드는 개별적인 설정 파일(Configuration file)을 요구한다. 대규모 분산 시스템에서는 단일 설정 파일로 모든 파라미터를 관리하는 것이 비효율적이므로, 계층적으로 분리된 다중 YAML(YAML Ain’t Markup Language) 설정 파일을 병합하고 오버라이드(Override)하는 규칙이 필수적으로 요구된다.

1. 계층형 설정 파일의 구조적 필요성

자율 비행 로봇 시스템에서는 하드웨어 제약, 운영 환경, 임무 목적에 따라 파라미터를 유연하게 재구성해야 한다. 계층형 설정 파일 구조는 기본 파라미터를 정의하는 베이스라인(Baseline) 설정, 특정 기체에 종속적인 하드웨어 추상화(Hardware Abstraction) 설정, 그리고 런타임 임무에 특화된 임무(Mission) 설정으로 구분된다.

Launch 파이프라인에서 이러한 여러 구성 파일을 로드할 때, 동일한 키(Key)를 가진 파라미터가 중복 선언될 수 있다. 시스템은 런타임의 예측 가능성(Predictability)과 안전 필수 설계(Safety-Critical Design)를 보장하기 위해, 명시적으로 정의된 우선순위에 따라 파라미터 병합(Merge) 및 오버라이드 연산을 수행한다.

2. Launch 시스템의 파라미터 로드 규칙

ROS 2의 launch_ros 프레임워크는 파라미터를 노드에 주입하기 위해 parameters 인자를 활용한다. 이 인자는 파라미터 딕셔너리, YAML 파일 경로의 리스트, 혹은 이들의 조합을 허용한다.

계층형 병합에서의 기본 오버라이드 규칙은 다음과 같이 전개된다.

  1. 순차적 평가(Sequential Evaluation): Launch 파일 내 parameters 인자에 전달된 리스트는 인덱스가 낮은 순서부터 순차적으로 평가된다.
  2. 후순위 오버라이드(Late-Override): 리스트 내의 요소 중 나중에 선언된 파일 또는 딕셔너리의 파라미터가 이전에 로드된 동일 키의 파라미터 값을 덮어쓴다(Override).
  3. 네임스페이스 결합(Namespace Resolution): 파라미터는 노드의 완전 정규화된 이름(Fully Qualified Name)을 기준으로 병합된다. YAML 파일 내부에서 선언된 네임스페이스 경로와 노드의 네임스페이스가 일치해야 파라미터가 올바르게 주입된다.

이를 수학적 집합으로 표현하면, n개의 파라미터 집합 P_1, P_2, \dots, P_n이 순서대로 주입될 때, 최종 노드 파라미터 상태 공간 S_{final}는 후위 연산이 우선하는 비대칭 병합 함수 f_{merge}에 의해 결정되며, 특정 키 k에 대한 값 v의 최종 상태는 다음과 같이 정의된다.

v_{final} = \begin{cases} v_n, & \text{if } k \in P_n \\ v_{n-1}, & \text{if } k \notin P_n \text{ and } k \in P_{n-1} \\ \dots & \\ v_1, & \text{if } k \in P_1 \text{ and } \forall j > 1, k \notin P_j \end{cases}

3. 계층형 병합의 실제 적용 메커니즘

자율 에이전트 드론의 제어 노드(flight_controller_node)를 예로 하여 병합 프로세스를 분석한다. 해당 시스템은 다음 세 가지 소스에서 파라미터가 제공됨을 원칙으로 한다.

  • base_config.yaml: 기본 제어 이득(Control Gains) 속성 지정
  • drone_v2_config.yaml: V2 기체 특성을 반영하는 제어 이득 오버라이드
  • launch 라인 인라인 딕셔너리: 런타임 디버그 레벨 강제 지정

Launch 스크립트 실행 시, 파이프라인의 처리 순서는 위의 리스트 구성에 의존한다. 최상위에 명시된 베이스 파일이 먼저 적재되어 기본값을 할당한 뒤, 후속 파일이나 인라인 딕셔너리에 의해 중복 정의된 동일 키의 데이터는 지속적으로 최종값으로 대체된다. 예를 들어 기본 파일에 특정 한계 속도가 정해져 있더라도, 마지막 인라인 딕셔너리에서 다른 값으로 선언되면 파이프라인은 최종 정의를 시스템에 주입한다.

4. 안전 검증 및 디버깅 고려사항

계층적 오버라이드 규칙은 유연성을 제공하나, 설정의 불투명성을 낳아 의도하지 않은 파라미터 덮어쓰기를 통해 비행 제어 루프의 불안정을 유발할 위험이 있다.

이를 방어하기 위해 ROS 2 프레임워크에서는 파라미터 로드 시 타겟 노드에서 명시적 파라미터 선언(Parameter Declaration)을 강제하는 정책을 엄격히 권고한다. 대상 노드 내에 정의되지 않은 파라미터가 외부 주입될 경우, 시스템 레벨에서 예외를 발생시키도록 파라미터 정책을 구성하여 파이프라인 안전성을 도모해야 한다. 더 나아가 CLI 명령어를 활용하여 최종 병합된 파라미터 트리를 주기적으로 진단하고 오버라이드 누락이나 오염 상태를 인스펙션(Inspection)하는 과정을 반드시 거쳐야 한다.
