19.3.1 컴포넌트 컨테이너 기반 노드 통합 구조의 메모리 효율

ROS(Robot Operating System) 1에서 노드(Node) 간 분산 처리는 본질적으로 개별적인 운영체제(OS) 프로세스를 생성하는 방식으로 이루어졌다. 그러나 자원을 극도로 제약받는 엣지 디바이스(Edge Device) 기반의 자율 드론 환경에서는 다중 프로세스 아키텍처가 유발하는 메모리 파편화 및 프로세스 간 통신(IPC: Inter-Process Communication) 오버헤드가 한계치에 다다르게 된다. ROS2는 이러한 단점을 극복하기 위해 동적 라이브러리(Shared Library) 로딩 방식을 활용한 컴포넌트 컨테이너(Component Container) 기반의 노드 통합 아키텍처를 도입하여 극단적인 메모리 효율성과 스케줄링 통합을 달성하였다.

1. 다중 프로세스 아키텍처의 한계와 메모리 낭비 기작

기존의 노드 실행 모델에서 N개의 노드를 독립적으로 실행할 경우, 운영체제는 N개의 개별적인 가상 메모리 공간(Virtual Memory Space)을 할당해야 한다. 이 과정에서 각 프로세스마다 rclcpp, DDS(Data Distribution Service) 미들웨어, 운영체제 종속적인 표준 라이브러리(libc 등)가 중복으로 메모리에 적재되어 베이스라인 점유율(Baseline Footprint)을 불필요하게 증폭시킨다.

더욱 치명적인 것은 프로세스 간 통신 방식이다. 로컬 호스트(Localhost) 내에서 실행되는 노드라 할지라도, 서로 다른 프로세스 경계를 넘기 위해서는 송신 노드에서의 직렬화(Serialization), OS 커널 공간으로의 데이터 복사, 수신 노드 커널 공간에서의 데이터 복사 체인, 수신 노드에서의 역직렬화(Deserialization)라는 4단계 페널티를 거쳐야 한다. 고주파 영상 데이터나 고용량 포인트 클라우드(Point Cloud) 데이터를 다루는 비전 시스템에서는 이러한 다단계 메모리 복사가 RAM 대역폭의 한계를 즉각적으로 초과하게 만든다.

2. 컴포넌트 컨테이너 아키텍처와 제로 카피(Zero-Copy) 전송

ROS2의 rclcpp_components::ComponentManager는 다수의 컴포넌트(노드 역할을 수행하는 C++ 클래스)를 단일 프로세스 컨테이너 내부로 동적 런타임 로드(Dynamic Runtime Load)하여 실행하는 프레임워크를 제공한다.

단일 프로세스 메모리 공간 지도로 통합된 컴포넌트들은 상호 간 데이터를 송수신할 때, 네트워크 스택이나 직렬화 계층을 거치지 않는다. 대신 ROS2의 Intra-Process Communication (IPC) 메커니즘이 활성화되어, 메모리에 저장된 데이터 객체의 스마트 포인터(std::unique_ptr 또는 std::shared_ptr)만을 교환하는 제로 카피(Zero-Copy) 버퍼 풀 기법이 동작한다. 이 메커니즘은 메시지 크기가 수 메가바이트(MB)에 이르는 센서 데이터조차도 O(1)의 상수 시간 복잡도로 전달할 수 있게 하며, 페이로드 페칭에 따른 메모리 복사를 원천적으로 소거한다.

3. Executor 공유를 통한 스레드 풀 압축 최적화

컴포넌트 컨테이너는 단순히 메모리 공간을 공유하는 것을 넘어 제어 스케줄링 자원을 통합 압축하는 부수적인 효과를 제공한다. 개별 프로세스로 구동되던 노드들은 자체적인 통신 WaitSet 스레드와 Executor 워커 스레드를 별도로 스핀(Spin)해야 했지만, 컨테이너 환경에서는 모든 하위 컴포넌트들이 단일 Multi-Threaded Executor 인스턴스에 안전하게 등록(Register)될 수 있다.

이는 시스템 내에서 유지되어야 하는 백그라운드 스레드의 총량을 획기적으로 감축시키며, 운영체제 스케줄러가 관장해야 하는 컨텍스트 스위칭 타겟을 줄여준다. 결과적으로, 컴포넌트 통합 구조는 엣지 보드의 잔여 RAM 여유 공간을 딥러닝 인퍼런스(Inference) 모델 용무나 맵핑(Mapping) 데이터 저장 등의 고부가가치 작업으로 전환 시키는 핵심적인 시스템 공학 기술 베이스가 된다.
