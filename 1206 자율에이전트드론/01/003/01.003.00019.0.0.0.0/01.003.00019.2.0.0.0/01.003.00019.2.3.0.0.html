19.2.3 스레드 풀 크기에 따른 컨텍스트 스위칭 오버헤드 상관관계

ROS2(Robot Operating System 2)의 실시간 비행 제어 및 장기 임무 환경에서 다중 스레드 기반의 노드(Node) 병렬화는 필수적인 구조이나, Multi-Threaded Executor의 워커 스레드 풀(Thread Pool) 크기를 물리적인 하드웨어 자원의 특성과 무관하게 과도하게 설정할 경우 치명적인 성능 저하를 초래한다. 특히 스레드 풀 크기와 운영체제의 컨텍스트 스위칭(Context Switching) 오버헤드 간의 비선형적(Non-linear) 상관관계는 자율 드론의 제어 루프 지연성(Latency)을 결정짓는 핵심 분석 대상이다.

1. 과다 가입(Oversubscription)과 스케줄러 오버헤드 기작

기본적으로 rclcpp::executors::MultiThreadedExecutor는 스레드 풀의 크기를 명시적으로 지정하지 않을 경우, 시스템이 제공하는 논리적 프로세서(Logical Cores)의 수와 동일한 개수의 스레드를 생성한다. 그러나 사용자가 콜백의 병렬 처리량을 늘리기 위해 스레드 풀 크기를 물리적/논리적 코어 수 이상으로 인위적으로 확장하는 과다 가입(Oversubscription)을 수행할 경우, 운영체제(OS)의 스케줄러는 제한된 CPU 자원을 다수의 스레드에 시분할(Time-sharing)하여 할당해야 한다.

이 과정에서 스케줄러가 실행 중인 스레드의 상태(레지스터 값, 프로그램 카운터, 스택 포인터 등)를 프로세스 제어 블록(PCB)이나 스레드 제어 블록(TCB)에 저장하고(Save), 다음으로 실행될 스레드의 상태를 복원하는(Restore) 컨텍스트 스위칭 작업이 기하급수적으로 빈발하게 된다. 컨텍스트 스위칭 자체는 순수한 CPU 연산 사이클을 소모하는 순수 오버헤드(Pure Overhead)이며, ROS2 응용 계층의 유효한 데이터 처리량을 급감시킨다.

2. 캐시 무효화(Cache Invalidation)와 TLB 미스 페널티

컨텍스트 스위칭이 드론의 실시간 제어 시스템에 미치는 더 심각한 악영향은 메모리 계층 구조에서의 부작용이다.

  1. L1/L2 캐시 오염 (Cache Pollution): 새로운 스레드가 CPU 코어에 적재되어 실행을 시작하면, 해당 코어의 L1/L2 데이터 및 인스트럭션 캐시에는 이전 스레드의 작업 데이터가 남아있다. 새로운 스레드는 자신의 메모리 공간에 접근하기 위해 기존 캐시 라인을 축출(Eviction)하고 새로운 데이터를 메인 메모리(RAM)로부터 페치(Fetch)해야 하므로 극심한 캐시 미스(Cache Miss)가 발생한다.
  2. TLB(Translation Lookaside Buffer) 플러시 (TLB Flush): 스레드가 서로 다른 프로세스 경계를 넘나들거나 심한 메모리 단편화를 겪을 경우, 가상 주소(Virtual Address)를 물리 주소(Physical Address)로 변환하는 캐시인 TLB가 플러시되어 메모리 접근 지연이 수백 사이클 이상 증폭된다.

수치적으로 스레드 풀 크기(N_{threads})가 코어 수(N_{cores})를 초과하여 임계점을 넘어서면, 캐시 무효화와 TLB 미스 오버헤드의 누적으로 인해 시스템 응답 지연 시간(T_{delay})은 선형 비례를 넘어 지수적으로 팽창(Exponential Growth)하는 경향을 보인다.

3. 제어 공학적 시사점과 스레드 풀 최적화

초당 수백 프레임을 처리해야 하는 비전 오도메트리(VIO)나 고주파 IMU 피드백 데이터는 이러한 컨텍스트 스위칭 지연에 극도로 취약하다. 수 ms의 스위칭 지연조차도 PID 또는 MPC(Model Predictive Control) 제어기에서는 역학적 발산을 초래하는 데드타임(Dead Time)으로 작용한다.

따라서 자율 드론 아키텍처에서 스레드 풀의 크기를 튜닝할 때는 “다다익선“의 패러다임을 배제해야 한다. 학술적인 성능 최적화 모델에 따르면, ROS2 워커 스레드의 개수는 가용한 물리적 CPU 코어의 수(또는 시스템 데몬 운영을 위해 N-1)와 정확히 일치시키는 것이 이상적이다. 더 나아가, pthread_setaffinity_np 등을 활용하여 특정 워커 스레드를 특정 코어에 고정(Core Pinning/Affinity)함으로써, 스레드의 코어 마이그레이션(Core Migration)으로 유발되는 캐시 동기화 오버헤드를 원천적으로 제거하는 격리 설계가 권장된다.
