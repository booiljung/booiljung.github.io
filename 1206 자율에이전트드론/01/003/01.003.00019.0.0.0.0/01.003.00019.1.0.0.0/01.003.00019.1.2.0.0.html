19.1.2 ROS1 Spin 루프와 ROS2 이벤트 루프 WaitSet 구조 비교

자율 에이전트 드론 컴포넌트들을 구형 ROS1 아키텍처에서 최신 ROS2 생태계로 마이그레이션(Migration)할 때, 개발자들이 겪는 가장 큰 논리적 장벽은 네트워크 통신 계층과 애플리케이션 계층 사이의 이벤트 스케줄링 메커니즘 차이이다. 겉보기에는 두 프레임워크 모두 spin()이라는 동일한 함수 명세로 노드의 실행 주기를 통제하는 듯 보이나, 그 하부에서 동작하는 콜백 디스패칭(Dispatching)의 철학은 근본적으로 다르다.

1. ROS1의 비동기 네트워크 스레드와 전역 콜백 큐 모델

ROS1은 내부적으로 TCPROS/UDPROS 프로토콜 레이어에 종속된 자체적인 통신 아키텍처를 가졌다. 노드가 생성되면 ROS1 코어 라이브러리(roscpp)는 보이지 않는 백그라운드 네트워크 스레드를 자동으로 스폰(Spawn)한다. 이 네트워크 스레드는 운영체제의 소켓을 폴링(Polling)하며 패킷이 들어오면 이를 식별하여 해당 노드의 전역 콜백 큐(Global Callback Queue)에 메시지를 무조건적으로 적재한다.

결과적으로 ROS1 개발자가 호출하는 ros::spin()은 단순히 이 전역 콜백 큐를 무한히 순회(Loop)하며, 큐에 데이터가 존재할 경우 이를 인출(Pop)하여 등록된 콜백 함수를 맹목적으로 실행하는 구조이다. 이러한 아키텍처는 네트워크 수신과 콜백 실행이 물리적으로 완전히 분리된 비동기(Asynchronous) 생산자-소비자(Producer-Consumer) 패턴을 따르므로, 콜백의 실행 속도보다 수신 속도가 압도적으로 빠를 경우 큐에 데이터가 무한정 팽창하는 논리적 지연(Latency)이 빈번하게 발생하였다.

2. ROS2의 WaitSet 기반 수동적 브릿징(Passive Bridging) 모델

반면, Data Distribution Service (DDS) 표준 미들웨어에 기반을 둔 ROS2는 데이터 중심(Data-Centric) 철학에 따라 노드의 스레드 자원을 RMW(ROS Middleware) 계층의 동기화 프리미티브와 직접적으로 결합한다. ROS2에는 ROS1과 같은 숨겨진 비동기 네트워크 스레드나 중앙 집중식 전역 콜백 큐가 존재하지 않는다.

ROS2의 rclcpp::spin()은 이그제큐터(Executor)를 호출하며, 이그제큐터는 앞서 언급한 ‘대기 체합(WaitSet)’ 자료구조를 생성하여 이를 RMW 층으로 곧바로 내려보낸다. DDS 구현체는 이를 운영체제 커널의 이벤트 다중화기(Multiplexer, 예: epoll, select)에 묶어놓고 호출 스레드를 수면(Sleep) 상태로 진입시킨다. 즉, 네트워크 소켓에 물리적인 데이터 버퍼가 도달하여 인터럽트(Interrupt)가 발생하기 전까지 이그제큐터 스레드 자체는 어떠한 CPU 사이클도 소모하지 않고 완벽하게 정지된 상태방어 계층을 이룬다.

3. 구조적 차이가 비행 제어에 미치는 학술적 시사점

ROS2의 WaitSet 모델은 ROS1의 큐 폴링 구조에 비해 몇 가지 강력한 제어 공학적 이점을 제공한다.

첫째, 이벤트 주도형(Event-Driven) 블로킹을 통해 유휴 상태의 CPU 점유율을 0%에 수렴하게 함으로써, 다중 코어 환경에서 복잡한 센서 융합이나 경로 계획 알고리즘에 코어 자원을 온전히 집중시킬 수 있다.
둘째, 대기 체합에서 깨어난 이그제큐터가 현재 활성화된 이벤트들의 스냅샷(Snapshot)을 한 번에 확인하고 우선순위를 재평가할 수 있는 기회를 제공한다. ROS1이 큐의 선입선출(FIFO) 논리에 속박되어 있었던 반면, ROS2는 어떤 토픽을 먼저 디스패치할 것인지 결정할 수 있는 스케줄링 역량을 이그제큐터 층으로 끌어올렸다는 점에서 실시간(Real-Time) 자율 제어 시스템이 갖춰야 할 결정론적 요구사항을 충족한다.
