19.4.1 네이티브 이그제큐터 WaitSet 폴링 대기 지연 한계 분석

ROS2(Robot Operating System 2)의 실시간성(Real-time Performance)을 결정짓는 가장 핵심적인 병목 지점(Bottleneck)은 네이티브(Native) 이그제큐터가 콜백(Callback) 이벤트의 도달을 감지하기 위해 사용하는 WaitSet 폴링(Polling) 메커니즘에 있다. 자율 드론의 고주파 비행 제어 루프를 정밀하게 설계하기 위해서는, 이 네이티브 구조가 내포하고 있는 근본적인 대기 지연(Wait Latency) 한계를 정량적, 구조적으로 분석해야 한다.

1. rcl_wait() 함수의 동작 원리와 WaitSet 구성 오버헤드

네이티브 이그제큐터 루프는 하부 C 라이브러리 계층인 rcl의 rcl_wait() 함수를 호출하여 운영체제의 이벤트 감지 API(예: 리눅스의 epoll 또는 select)를 통해 DDS(Data Distribution Service) 미들웨어 계층의 네트워크 이벤트를 지연 대기한다.

이때 이그제큐터는 rcl_wait()을 호출하기 앞서, 시스템 내의 모든 활성화된 노드로부터 구독(Subscription), 타이머(Timer), 서비스(Service), 클라이언트(Client), 웨이터블(Waitable) 엔티티(Entity)들의 핸들을 수집하여 하나의 거대한 WaitSet 구조체를 매 스핀(Spin) 사이클마다 동적으로 구성(Re-construction)한다.

이러한 WaitSet 갱신 작업의 시간 복잡도(Time Complexity)는 시스템에 등록된 엔티티의 총 개수 N에 대해 O(N)으로 비례한다. 센서 노드, 제어기, 진단 모듈 등 드론의 서브시스템이 복잡해질수록 N의 크기가 급증하며, 이에 따라 콜백 실행 자체가 아닌 순수한 스케줄링 준비 단계에서 소모되는 지연 시간이 선형적으로 증가하는 결정적인 한계가 발생한다.

2. O(N) 순회 시의 메모리 동적 할당과 캐시 무효화

네이티브 이그제큐터의 WaitSet 갱신 과정은 단순한 배열 순회로 끝나지 않는다.

  1. 동적 할당 페널티 (Dynamic Allocation Penalty): 노드의 추가 및 삭제에 대응하기 위해 이그제큐터는 WaitSet 컨테이너의 크기를 매번 재계산하고, 필요시 힙(Heap) 영역을 재할당(malloc/free)한다. 실시간 시스템에서 통제되지 않은 빈번한 동적 할당은 운영체제 메모리 관리자의 개입을 부르고, 비결정론적(Non-deterministic)인 지터(Jitter)를 생성하는 주범이다.
  2. 데이터 지역성 붕괴 (Poor Data Locality): WaitSet을 구성하기 위해 분산된 노드들의 메모리 주소(Pointer)를 참조(Dereferencing)하는 과정에서 잦은 L1/L2 캐시 미스(Cache Miss)가 유발된다. 엣지 컴퓨팅 기반의 드론 비행 제어기(예: NPU 통합형 ARM SoC 등)에서는 이러한 캐시 플러시(Flush)가 CPU 사이클을 극도로 소모하게 만든다.

3. 고주파 제어 루프에서의 한계 지적

일반적으로 자율 에이전트 드론의 관성측정장치(IMU) 기반 자세 제어 루프는 최소 250Hz에서 최대 1000Hz 이상의 주파수를 요구한다. 500Hz 제어 루프의 경우 1 사이클에 허용되는 하드 타임 마진(Hard Time Margin)은 정확히 2ms(2000\mu s)에 불과하다.

그러나 노드 수가 100개를 넘어가는 복합 시스템 환경에서 네이티브 이그제큐터의 WaitSet 재구성 및 역매핑(콜백 매칭) 연산 오버헤드는 엣지 디바이스 기준 통상적으로 1ms 내외를 손실하게 만든다. 가용 시간의 50% 이상이 애플리케이션 연산이 아닌 단순 폴링 오버헤드로 증발하게 되면, 비행 제어 알고리즘의 계산할당량이 삭감되고, 최악의 경우 데드라인 미스(Deadline Miss)가 누적되어 물리적인 제어 불안정성(Control Instability)으로 직결된다.

기술적으로, 이러한 폴링 기반 WaitSet의 비효율성은 ROS2 프로세스가 노드 개수 확장성(Scalability)과 실시간성 보장을 동시에 달성해야 하는 제어 공학의 파라독스를 나타내며, 이벤트 구동형(Event-Driven)의 최적화된 스케줄라 아키텍처 전환을 필연적으로 강제한다.
