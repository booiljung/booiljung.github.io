# 1. 전이 상태 파이프라인: Configuring, CleaningUp, ShuttingDown, Activating, Deactivating

자율 드론 아키텍처에서 노드의 생명주기를 결정론적으로 제어하기 위해서는 일차 상태(Primary State) 간의 이동을 관장하는 전이 상태(Transition State)의 설계와 이해가 필수적이다. ROS2의 rclcpp_lifecycle 프레임워크는 상태 변경 중 발생할 수 있는 교착 상태(Deadlock)나 자원 누수(Resource Leak)를 방지하기 위해 엄격히 통제되는 5가지의 전이 상태 파이프라인을 규정한다. 이러한 전이 상태들은 비동기 시스템 내에서 노드가 새로운 일차 상태로 진입하기 전에 요구되는 연산을 수행하고 그 결과를 검증하는 버퍼 역할을 수행한다. 비행 제어기와 같은 안전 필수(Safety-Critical) 환경에서는 이 전이 구간에서의 행위가 전체 시스템의 실시간성과 무결성에 직접적인 영향을 미치므로, 각 파이프라인의 내부 메커니즘을 명확히 정의하여야 한다.

1.1 Configuring 파이프라인의 자원 할당 메커니즘

Configuring 전이 상태는 노드가 Unconfigured 상태에서 Inactive 상태로 진입하기 위해 거치는 파이프라인이다. 이 구간에서는 노드가 임무 수행에 필요한 제반 하드웨어 리소스와 통신 미들웨어 계층의 네트워크 자원을 초기 할당한다. 드론의 로컬라이제이션 센서(예: IMU, LiDAR) 초기화, QoS(Quality of Service) 프로필이 적용된 퍼블리셔 및 서브스크라이버의 생성, 그리고 파라미터 서버로부터의 초기 구성값 로드 등의 작업이 이 파이프라인 내에서 동기적 또는 비동기적으로 수행된다.

Configuring 파이프라인의 성공적인 완료는 시스템이 모든 필수 자원을 확보하였음을 의미하며, 실패 시에는 메모리 단편화나 좀비 리소스가 남지 않도록 엄격한 롤백(Rollback) 정책이 적용되어야 한다. 실패된 노드는 즉각적으로 Unconfigured 파이프라인으로 복귀하거나 설정에 따라 ShuttingDown으로 전환되어 시스템 안정성을 확보한다. 자율 드론의 다중 노드 초기화 과정에서 이 파이프라인의 시간 복잡도를 최소화하는 것은 전체 시스템의 부팅 속도(Boot-up latency)와 직결된다.

1.2 Activating 파이프라인의 데이터 스트림 개방

Activating 파이프라인은 노드가 Inactive 상태에서 실제 데이터를 발행 및 수신하고 콜백 루프를 가동하는 Active 상태로 전이하는 구간이다. Configuring 파이프라인에서 할당된 메모리와 네트워크 핸들이 이 시점에서 실질적인 데이터 흐름(Data Flow)을 허용하도록 개방된다. ROS2 미들웨어 수준에서는 Network Flow Endpoint의 동작이 스케줄러에 인가되어 노드의 메시지 퍼블리싱이 네트워크로 전송 가능한 상태가 된다.

안전한 비행을 위해 Activating 단계는 의존성을 가진 다른 노드의 활성화 상태와 동기화되어야 할 필요가 있다. 예를 들어 비행 제어 명령을 하달하는 정책 노드가 활성화되기 전에는 모터 제어 노드의 데이터 파이프라인을 완전히 개방하지 않는 락스텝(Lockstep) 활성화 방식이 보편적으로 적용된다. 이 전이 상태 중 오류가 발생할 경우 시스템은 퍼블리셔의 동작을 즉각 중지하고 노드의 상태를 Inactive로 복원하거나 치명적 결함으로 판단 시 ErrorProcessing 체인으로 제어권을 넘긴다.

1.3 Deactivating 파이프라인의 안전 차단 설계

Deactivating 파이프라인은 임무 완료 혹은 일시적 비행 중단 시 Active 상태의 노드를 다시 Inactive 상태로 회귀시키는 전이 구간이다. 이 파이프라인의 핵심 학술적 목표는 데이터 스트림의 단절로 인한 하위 노드의 상태 불안정을 방지하는 것이다. 퍼블리셔는 네트워크 큐에 적재된 메시지의 전송을 점진적으로 락킹(Locking)하며, 진행 중인 서비스 콜백이나 액션 프리엠션(Action Preemption)을 안전하게 마무리한다.

자율 드론의 비상 정지(Emergency Stop) 또는 모드 전환 상황에서 Deactivating 파이프라인은 최소 지연 시간(Minimal Latency) 내에 완료되어야 한다. 이를 위해 잔여 데이터를 강제 폐기(Drop)할지, 큐 버퍼가 비워질 때까지 대기(Flush)할지에 대한 명시적인 QoS 스펙트럼 설계가 요구된다. 정상적인 비활성화가 이루어지면 노드는 메모리와 통신 객체를 그대로 유지한 채 연산 주기를 최소화하여 대기 전력 및 컴퓨팅 리소스를 보존한다.

1.4 CleaningUp 파이프라인과 자원 해제

CleaningUp 파이프라인은 Inactive 상태의 노드의 통신 객체를 회수하고 메모리 할당을 전면 해제하여 Unconfigured 상태로 복귀시키는 전이 생명주기이다. 이 파이프라인은 C++ 기반의 rclcpp 계층에서 생성된 동적 메모리를 반환하고 운영체제 레벨의 파일 서술자(File Descriptor) 결합을 해제하는 등 깊은 수준의 가비지 컬렉션(Garbage Collection) 및 자원 반환 연산을 수반한다.

드론의 핫스왑(Hot-Swap) 모듈이나 장기간 동작하는 시스템에서 자원 누수(Memory Leak)를 방지하기 위해 이 파이프라인은 예외 없이 보장된 순서로 리소스를 소멸시켜야 한다. 센서 드라이버와 같은 하드웨어 추상화 계층(HAL)과 연결된 노드는 해당 하드웨어를 안전한 슬립 모드로 전환하는 로직 또한 CleaningUp 단계에 포함하여야 하드웨어와 소프트웨어의 수명 주기를 동기화할 수 있다.

1.5 ShuttingDown 파이프라인과 파이널라이징 절차

ShuttingDown 전이 상태는 노드의 라이프사이클을 영구적으로 종료하고 Finalized 상태로 전환하기 위한 마지막 파이프라인이다. 이 상태 전이는 Unconfigured부터 Active까지 시스템의 어떠한 일차 상태에서도 발생할 수 있는 비동기적 강제 중단 시퀀스를 포함한다. 노드가 물리적 전원 차단 신호를 수신하거나 심각한 소프트웨어 패닉 에러에 직면했을 때 호출된다.

이 파이프라인의 주된 로직은 운영체제의 프로세스 종료 시그널(SIGINT, SIGTERM) 발생 시 노드가 소멸되기 직전에 시스템 로그를 플러시(Flush)하고, 글로벌 트리 구조를 관리하는 매니저 노드에게 자신의 완전한 소멸을 브로드캐스트하는 것이다. Finalized 상태로 진입한 노드는 메모리 공간에서 완전히 소멸될 준비를 갖추게 되며, 생명주기 상태 머신의 감독 영역에서 영구적으로 이탈함으로써 전체 드론 아키텍처의 상태 모델을 갱신한다.
