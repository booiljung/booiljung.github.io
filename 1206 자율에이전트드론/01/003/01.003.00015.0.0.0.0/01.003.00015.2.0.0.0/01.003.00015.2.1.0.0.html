15.2.1 기본 상태 정의: Unconfigured, Inactive, Active, Finalized

ROS2 Lifecycle 아키텍처는 개별 소프트웨어 컴포넌트(Node)의 생애 주기를 4개의 고정된 기본 상태(Primary State)로 계층화하여 정의한다. 각 기본 상태는 시스템 리소스(동적 메모리, 하드웨어 소켓)의 점유 수준과 통신 엔드포인트(DDS Publisher/Subscriber)의 활성화 여부를 물리적이고 논리적인 기준에 따라 엄격하게 통제한다. 자율 드론의 비행 제어 및 임무 스택은 이 4가지 휴지(Resting) 상태 궤도상에 지속적으로 안착하며, 시스템 통제기(Orchestrator)의 명시적인 외부 트리거 이벤트를 통해서만 다른 상태로 천이된다.

1. Unconfigured: 인스턴스 초기화 및 메모리 격리

Unconfigured 상태는 Lifecycle 노드가 C++ 클래스 생성자(Constructor)를 통해 운영체제의 메모리 힙(Heap) 공간에 객체로 인스턴스화된 직후 머무는 최초의 기본 상태이다. 이 상태에서의 노드 객체는 자신을 식별할 메타데이터(예: 노드 이름, 네임스페이스)만을 보유할 뿐, 외부와 상호 작용하기 위한 어떠한 물리적 행동도 취하지 않는다.

이 국면에서 시스템은 센서 디바이스를 초기화하거나, 파라미터(Parameter) 파일을 파싱하거나, 대규모 포인트 클라우드를 위한 동적 메모리 풀(Pool)을 할당하는 행위를 원천적으로 금지한다. DDS 라우팅 인터페이스도 생성되지 않으므로, 프로세서 연산 주기나 네트워크 대역폭에 가하는 부하(Overhead)는 완벽한 제로(0)에 수렴한다. 다수의 시스템 노드가 Unconfigured 상태로 배포됨으로써, 드론 통제기는 전체 노드를 안전하게 대기 버퍼에 올려놓고 런타임 간섭 없이 유휴 자원을 안전하게 계산할 수 있는 이점을 누린다.

2. Inactive: 콜드 스탠바이 및 시간적 동기화

Configuring 전이 과정을 성공적으로 통과한 노드는 파라미터 로딩, 내부 메모리 할당, 그리고 DDS 엔드포인트를 확립한 뒤 Inactive 상태에 도달한다. 이 상태는 시스템 자원을 온전히 점유하고 동작할 준비가 끝난 대기 모드(Standby Mode)이다.

Inactive 궤도상에 머무르는 노드는 외부의 센서 데이터를 수신(Subscribe)하고 내부 상태 파라미터를 버퍼링하는 것은 허용되나, 외부를 향해 비행 제어 지시값(Setpoints)을 퍼블리싱(Publishing)하거나 제어 주기 타이머(Timer) 콜백을 물리적으로 실행하는 행위는 강제적으로 차단된다. 이로 인해 임무 제어 컴퓨터 상의 모든 노드들이 락스텝(Lockstep) 동기화 구역에 도달할 때까지 네트워크의 트래픽 포화와 초기화 단계에서 유발되는 가비지(Garbage) 데이터의 전파를 하부 플라이트 컨트롤러(PX4, ArduPilot) 영역으로부터 철저하게 보호한다.

3. Active: 실시간 제어 루프 구동 및 임무 수행

Inactive 상태에서 활성화 트리거를 받은 노드는 Active 상태에 편입되며, 이는 노드가 완전히 가동되어 의도된 비행 알고리즘 로직을 전개하는 유일한 운영 상태(Operational State)이다.

이 상태에서 제어 노드는 VIO 혹은 SLAM 알고리즘이 산출한 상태 추정치를 입력받아 고도화된 MPC(Model Predictive Control) 궤적 제어 루프를 구동하고, 타겟 각속도 및 추력 모터 명령을 실시간으로 퍼블리싱한다. Active 궤도로의 진입 이전에 Inactive 상태에서 이미 모든 동적 메모리가 선할당(Pre-allocated)되고 네트워크 소켓이 개방되어 있었기 때문에, Active로의 전이는 지터(Jitter) 없이 1밀리초 이내의 즉각적 전환 속도를 보증한다. 이는 곧 하드 리얼타임(Hard Real-time) 비행 제어 애플리케이션이 요구하는 극단적인 결정성 단서를 수학적으로 증명하는 조건이 된다.

4. Finalized: 안전 파괴 트리거 및 셧다운 격리

노드의 임무가 완전하게 종료되었거나 하드웨어 센서 결함 등으로 인해 복구가 불가능한 시스템 예외 상황(Error) 도출 시 최종적으로 도달하는 무기동 공간이 Finalized 상태이다.

이 상태에 진입한 노드는 외부와의 연결을 완전히 차단당하며, 이 상태에 체류하는 시간은 곧 운영체제에 의한 프로세스 메모리 파괴(Destruction) 명령이 하달되기 전의 찰나로 직결된다. 이 영역을 통해 시스템 오케스트레이터(Orchestrator)는 고장 난 메모리 누수 프로세스를 글로벌 관리 트리에 잔존시키지 않고 안전하게 시스템 공간에서 축출(Eviction) 및 제거하게 된다. 결론적으로 이 4가지의 기본 상태 메커니즘은 컴포넌트 간의 투명한 상태 관찰성을 제공하며, 다중 프로세스의 생애 주기를 일사불란한 단일 지휘 체계로 묶는 강력한 관리 아키텍처의 표본을 제시한다.
