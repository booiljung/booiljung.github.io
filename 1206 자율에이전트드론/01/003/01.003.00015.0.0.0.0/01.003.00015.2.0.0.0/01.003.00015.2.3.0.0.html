# 1. 상태 전이 콜백 체인 및 에러 전처리 프로토콜

자율 에이전트 드론 아키텍처에 구현된 ROS2 생명주기(Lifecycle) 노드는 단순히 상태를 보관하는 정적 컨테이너가 아니다. 상태 간 전이 프로세스는 콜백 체인(Callback Chain)이라는 프로그래밍 레이어를 통해 실행 단위와 논리적 연산을 동기화한다. 이 콜백 계층은 상태 전이가 요청될 때 프레임워크에 의해 자동 호출되며, 응용 계층의 로직이 실제 하드웨어 제어나 미들웨어 통신을 조율할 수 있도록 인터페이스를 제공한다. 특히 실패가 치명적인 결과를 초래하는 비행 환경에서는 전이 중 발생하는 예외 사항을 포착하고 처리하는 에러 전처리 프로토콜(Error Pre-processing Protocol)이 상태 머신의 강건성(Robustness)을 결정하는 핵심 요소가 된다.

1.1 상태 전이 콜백 인터페이스 계층

rclcpp_lifecycle 프레임워크는 각 전이 파이프라인(Configuring, Activating, Deactivating, CleaningUp, ShuttingDown)에 대응하는 가상 함수(Virtual Function) 형태의 콜백 인터페이스를 제공한다. 사용자는 on_configure(), on_activate(), on_deactivate(), on_cleanup(), on_shutdown() 등의 콜백 메서드를 오버라이딩(Overriding)하여 상태 변환 시점에서 실행될 구체적인 로직을 주입한다.

콜백 함수는 호출 시 프레임워크로부터 전이 상태에 대한 컨텍스트(Context) 객체를 전달받으며, 연산이 종료되면 반드시 정해진 반환값(Return Value)을 반환해야 한다. 반환 타입은 CallbackReturn이라는 열거형(Enum)이며, 통상적으로 SUCCESS, FAILURE, ERROR 세 가지의 표준 응답 프로토콜을 따른다. SUCCESS는 요구된 모든 초기화 또는 해제 로직이 에러 없이 완수되었음을 미들웨어에 통지하여 다음 일차 상태로의 성공적 전이를 유도한다.

1.2 FAILURE 반환의 롤백 및 재시도 체계

콜백 함수 내에서 요구된 자원 할당이나 통신 포트 개방이 실패한 경우, 노드는 FAILURE 메시지를 프레임워크로 반환할 수 있다. 이 응답은 시스템에서 물리적 혹은 구조적 페이탈 에러(Fatal Error)가 발생한 것이 아니라 재시도 가능성 또는 논리적 실패가 발생했음을 인지시킨다. FAILURE 반환 시 프레임워크는 내부적인 전이를 중단하고 Unconfigured 상태 복귀 혹은 이전 단계의 일차 상태로 안전하게 회귀하는 롤백(Rollback) 메커니즘을 트리거한다.

비행 제어 소프트웨어에서 센서 노드의 on_configure() 콜백이 장치 타임아웃으로 실패한 경우, 시스템 전체를 종료시키기보다 FAILURE를 반환하여 상태를 유지하고 시스템 매니저에게 재설정을 요청하는 것이 권장된다. 이러한 FAILURE 체인은 개발자가 명시적으로 실패 경로를 예측하고 부분적인 서비스 중단만을 허용하도록 설계된 방어적 상태 제어기(Defensive State Controller)의 일환이다.

1.3 패닉 생명주기와 ERROR 반환의 전처리

비가역적인 메모리 오류, 하드웨어 소실 등 드론의 안전한 비행을 보장할 수 없는 중대한 결함이 발생할 경우, 상태 전이 콜백은 ERROR를 반환해야 한다. 이는 프레임워크가 상태 제어의 주도권을 비상 복구 혹은 영구 종료 프로토콜로 이관하도록 강제한다. ERROR 반환은 즉각적으로 ErrorProcessing 파이프라인으로 진입하는 진입점(Entry Point)이 되며, 이 과정에서 실행되는 특별한 콜백이 on_error()이다.

on_error() 체인은 에러 전처리 프로토콜의 핵심으로, 시스템이 완전한 Finalized 상태로 셧다운 되기 전 마지막으로 문제를 수습할 기회를 제공한다. 이 전처리 과정에서는 모터 정지 명령 방송, 마지막 블랙박스 로그 덤프, 하드웨어 릴레이 차단 등이 수행될 수 있다. 여기서 문제가 충분히 해결되었다고 판단하여 예외적으로 SUCCESS를 반환하면 노드는 Unconfigured 상태로 복구되기도 하나, 항공 안전 필수 규정에 따라 일반적으로는 심각한 에러 발생 시 오류 처리 후 Finalized 상태로 전환하여 독립적인 감시자 프로세스(Watchdog)에 의해 재시동되는 구조를 취한다.

1.4 상태 전이의 비동기적 타임아웃과 락(Lock) 제어

단일 쓰레도로 구동되는 콜백 로직 내부에서 블로킹(Blocking) 동작이 포함되면 락(Lock) 지연이나 시스템 멈춤 상태에 빠질 수 있다. 특히 액추에이터의 응답을 무한정 대기하는 로프(Loop)나, 네트워크 재연결을 위해 과도한 슬립(Sleep)을 허용하는 형태는 생명주기의 전이 시간 한계치를 초과하게 만들며 전체 비행 제어 시스템의 상태 동기화 그래프(Global State Synchronization Graph)를 교착에 빠뜨릴 위험이 있다.

이를 방지하기 위해 에러 전처리 프로토콜은 상태 전이 콜백 체인에 대한 명시적인 타임아웃 모델을 동반해야 한다. 타임아웃 발생 시, 시스템 하위 계층에서는 해당 콜백의 실행 컨텍스트를 강제 분리하고 내부 에러 상태를 주입하여 ErrorProcessing 체인으로 강제 점프시킨다. 따라서 모든 전이 콜백 로직은 비동기성을 보장하는 짧은 실행 주기 내에 완료되어야 하며, 불확실성이 큰 I/O 작업은 독립 쓰레드에 위임하는 등 사이드 이펙트를 차단하는 아키텍처적 제약을 준수하여야 한다.
