15.5.1 핵심 노드 장애 시 ErrorProcessing 상태 전이 설계

자율 에이전트 드론 아키텍처에서 비행 역학 제어기 인터페이스(Flight Dynamics Controller Interface)나 통합 상태 추정기(Integrated State Estimator)와 같은 최상위 핵심 노드(Core Node)의 런타임 기능 상실은 즉각적이고 치명적인 물리적 관성 사고를 유발한다. 따라서 이러한 핵심 노드에서 예상 범위를 벗어난 예외 상황(Exception)이 발생했을 때, 프로세스가 통제 불능의 좀비 상태나 세그먼테이션 폴트(Segmentation Fault) 구조로 붕괴(Crash)하는 것을 원천 방지하고, 구조적으로 프로그래밍된 안전 예외 처리 공간인 ErrorProcessing 상태로 신속하게 전이(Transition)시키는 프레임워크 설계가 필수적으로 요구된다.

1. ErrorProcessing 상태의 구조적 목적과 진입 트리거 조건

ROS2 Lifecycle 프레임워크 표준에서 ErrorProcessing 상태는 일반적인 데이터 처리 흐름 구역과 격리된 일종의 안전 결함 분석 지대(Safe Diagnostic Zone) 역할을 병행 수행한다. 핵심 노드는 시스템 파괴로 이어질 수 있는 아래와 같은 치명적 오류 조건이 런타임에 감지될 때, 즉시 진행 중이던 제어 흐름을 중단하고 ErrorProcessing으로 강제 전이되도록 설계되어야 한다.

  • 상태 전이 콜백 내부의 명시적 오류 보고: 대상 노드가 on_configure(), on_activate(), on_deactivate()와 같은 라이프사이클 과도기(Transitioning) 콜백 함수를 실행하던 중, 센서 버스 초기화 락(Lock)이나 주요 동적 메모리 할당 실패 등 복구 알고리즘이 부재한 결함에 직면하게 되면, 프레임워크에 CallbackReturn::ERROR 상태 반환 상수를 명시적으로 반환함으로써 ErrorProcessing으로의 궤도 전이를 트리거한다.
  • 런타임 예외(Runtime Exceptions)의 비동기 포착: 다량의 센서 퍼블리셔와 액추에이터 서브스크라이버가 교류하는 메인 제어 루프 영역(Active 상태 유지 중)에서 사전 처리되지 않은 C++ std::exception 도출 또는 rclcpp::exceptions 기반 아키텍처 예외가 발생할 경우, 노드를 포괄하는 생명주기 워치독(Watchdog) 데몬이 이를 포획(Catch)하여 즉시 ErrorProcessing 상태 방향으로 강제 진입시킨다. 이는 프로세스가 패닉에 빠져 연산 찌꺼기를 방출하는 것을 막는 소프트웨어적 1차 방어선 역할을 수행한다.

2. on_error() 콜백의 물리 자원 정지 및 진단 로직 설계 파이프라인

핵심 노드가 ErrorProcessing 논리 영역에 진입하게 되면, 노드 클래스 내부에 오버라이딩(Overriding)되어 구현된 on_error(const rclcpp_lifecycle::State & previous_state) 콜백 인터페이스가 프레임워크에 의해 최우선순위로 호출당한다. 이 콜백은 단순히 단문 에러를 통보하는 로그 체계를 뛰어넘어, 기체의 물리적 손상 확장을 제어하기 위한 안전 정지(Safe-Halt) 조치와 정밀 원인 진단을 수행하도록 밀도 있게 구현되어야만 한다.

  • 하드웨어 포트 및 네트워크 자원의 즉각적 강제 해제: 해당 핵심 노드가 프로펠러 모터의 ESC(Electronic Speed Controller) 등으로 직접적인 속도 제어 명령을 출력하고 있던 중 장애가 발생했다면, on_error() 콜백 최상단 진입 즉시 모든 제어 출력을 0으로 강제 하강(Zero-output Pull-down)시켜야 한다. 직후 점유하고 있던 직렬 통신 인터페이스(UART/SPI 포트)나 I/O DMA 메모리 버퍼 등 독점적 커널 자원을 즉각 운영체제로 반환(Release)하여 물리적 드론의 불규칙한 행동 폭주나 액추에이터 소손을 방지한다.
  • 원인 규명을 위한 메모리 덤프 및 진단 브로드캐스팅: 치명적 오류 파동을 일으킨 직전의 previous_state 파라미터 컨텍스트 및 센서 시스템의 주요 레지스터 변수값 세트를 미니 코어 스냅샷(Mini-core Snapshot) 형태로 메모리에 덤프한다. 포장된 이 장애 덤프 구조체 데이터는 ROS2 Diagnostics 표준 토픽 버스를 경유하여 전역으로 브로드캐스팅(Broadcasting)되며, 이는 상위에 존재하는 글로벌 Lifecycle 관리자 노드가 현재 상황의 재난 치명도를 정량적으로 판독할 수 있는 가장 핵심적인 평가 근거 도메인이 된다.

3. 진단 결과에 의거한 후속 궤도 상태 결정 메커니즘

on_error() 콜백 스코프 내부에서의 진단 판독 및 위험 자원 초기화 로직 파이프라인이 모두 안착하고 나면, 최종적인 콜백의 반환(Return) 상태 상수를 통해 결함 노드의 다음 목적지 상태 궤도를 프레임워크 계층에 통지한다.

  • 일시적 통신 오류 판정 시의 재초기화 궤도 복귀: 내부 진단 결과 센서 인터페이스의 단순 지연 모델링이나 무선 패킷 손실에 의한 일과성 오류로 판별되어 부분적인 소프트 리셋(Soft Reset) 조치만으로 구동 해결이 가능성이 높다고 판단될 경우, 노드는 CallbackReturn::SUCCESS를 반환한다. 이는 노드를 안전하게 완전 초기화 전 단계인 Unconfigured 상태로 강등 변위시키며, 외부 글로벌 관리 노드가 이 상태를 추적하여 다시금 재초기화 및 활성화 시퀀스를 안전하게 시도할 수 있는 기회 구간을 제공한다.
  • 하드웨어 영구 결함 판정 시의 폐기(Finalized) 궤도 진입: 영상 처리용 GPU 소자의 물리적 발열 결함이나 메인 RAM 소켓의 탈거 현상 등, 물리적 임베디드 장비의 웜 부트(Warm Boot) 재구동 없이는 소프트웨어적 해결이 절대 불가능한 영구 결함(Persistent Fault)으로 심층 진단될 경우, 노드는 CallbackReturn::FAILURE 상수를 반환해야 한다. 이를 통해 해당 핵심 노드는 라이프사이클의 영구적 종착역인 Finalized 상태로 추락하여 메모리 객체 파괴(Destruction) 대기 큐에 돌입하게 되며, 시스템 관리 구조 전체는 이중화 장비로의 전환을 기동하는 페일세이프(Fail-Safe) 긴급 복구 메인 절차를 전개하기 시작한다.
