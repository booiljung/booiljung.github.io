15.1.3 비행 제어 탑재 컴퓨터 부팅 시퀀스와 ROS2 Lifecycle 통합 아키텍처

자율 에이전트 드론의 전원 리셉터클(Receptacle)에 배터리가 인가되는 순간부터 기체가 지상 통제소(GCS, Ground Control Station)에 ‘이륙 준비 완료(Ready to Arm)’ 신호를 전송하기까지의 과정은 수많은 하드웨어적, 소프트웨어적 초기화 단계를 내포한다. 기존 체계에서는 리눅스 운영체제(OS)의 부팅 시퀀스와 비행 제어 애플리케이션의 구동 스크립트(예: rc.local 또는 단순 쉘 스크립트)가 느슨하게 결합하여 작동했다. 그러나 이러한 방식은 특정 하드웨어 인터페이스(예: USB 버스의 카메라 펌웨어 마운트)가 지연될 경우 전체 비행 소프트웨어 스택이 치명적인 교착 상태(Deadlock)에 빠질 수 있는 취약점을 내포한다. 이를 원천 차단하기 위해, 드론 임베디드 리눅스의 systemd 데몬과 ROS2 Lifecycle 상태 머신을 유기적으로 연동한 하향식(Top-Down) 부트 시퀀스 통합 아키텍처가 필수적으로 요구된다.

1. Systemd와 ROS2 Lifecycle Manager의 계층적 브리지 설계

효율적인 부팅 시퀀스 제어를 위해 컴패니언 컴퓨터(탑재 컴퓨터)의 운영체제 부팅 관리자인 systemd는 ROS2 환경의 글로벌 상태 제어기인 Lifecycle Manager 단 하나만을 최상위 데몬(Daemon) 서비스로 직접 관할한다. 이 통합 아키텍처에서 개별 센서 노드나 궤적 제어 노드는 systemd 프로비저닝 대상에서 제외되며, 오직 Lifecycle Manager 노드의 수명 주기 트리(Tree) 아래로 편입된다.

systemd 유닛 파일 구성에서는 네트워크 스택(DDS 멀티캐스트 라우팅)과 로컬 시스템 버스가 완전히 마운트된 직후(After=network-online.target)에만 Lifecycle Manager가 실행되도록 의존성을 부여한다. Lifecycle Manager 노드가 운영체제 위에서 성공적으로 런치(Launch)되면, 이 관리자 노드는 즉시 분산된 모든 비행 제어 관리형 노드(Managed Nodes)를 읽어 들여 Unconfigured 상태로 메모리에 상주시킨다. 이는 리눅스 커널 수준의 부팅 시퀀스를 미들웨어 수준의 동기화 가능한 객체(Lifecycle Object)로 매끄럽게 전환(Handover)시키는 핵심 브리지 역할을 수행한다.

2. 결정론적 단계별 기동 시나리오 (Deterministic Startup Sequence)

통합 아키텍처 내에서 컴패니언 컴퓨터가 전원을 인가받은 후 전개하는 시동 사이클은 엄격하게 직렬화된 하향식 상태 전이를 따른다.

  1. 하드웨어 및 네트워크 디스커버리 (Pre-Configuration): OS 부팅 직후 Lifecycle Manager가 구동되며, VIO 추정 노드, 물리 센서(LiDAR) 드라이버, 오프보드(Offboard) 플라이트 컨트롤 액추에이터 노드 등을 순차적으로 인스턴스화한다. 이 단계에서 모든 노드는 Unconfigured 상태 버퍼에 대기한다.
  2. 글로벌 Configure 글로벌 트리거: Lifecycle Manager는 하위 노드들에게 일제히 Configure 상태 천이를 명령한다. 각 노드는 동적 메모리 풀(Pool)을 할당하고, 직렬 포트(UART)를 개방하여 비행 제어기(PX4/ArduPilot)와의 초기 매개변수 동기화 및 캘리브레이션 검증을 수행한다. 모든 노드가 에러 없이 작업을 완료하고 Inactive 상태로 안착해야만 다음 단계로의 진입이 승인된다.
  3. 일괄 Activating 및 임무 준비: 모든 종의 센서 데이터 파이프라인과 액추에이터 연결성이 결함 없이 보증된 락스텝(Lockstep) 동기화 순간, 관리자는 전역 Activate 명령을 방송(Broadcasting)한다. 이 찰나의 순간에 모든 통신 소켓의 제어 밸브가 열리며(Publishing 활성화), 비행 제어 스택은 비로소 인지-판단-제어 알고리즘의 본 연산 루프를 수행하는 Active 상태로 진입하게 된다.

3. 워치독(Watchdog) 기반 비상 롤백 및 아키텍처 요약 결론

위와 같은 엄격한 순차 검증 시퀀스 중, 특정 노드의 Configuring 단계가 사전에 정의된 임계 시간(예: 5초)을 초과하여 타임아웃을 발생시킬 수 있다. 통합 아키텍처 하의 Lifecycle Manager는 자체적인 상태 워치독(Watchdog Timer) 메커니즘을 가동하여, 응답이 없는 노드를 강제로 ErrorProcessing 단계로 이행시키고 나머지 모든 노드의 부팅 절차를 연쇄적으로 롤백(Rollback)하여 Unconfigured 초기 상태로 원복(Revert)시킨다. 이는 이륙 전의 하드웨어 오결선이나 치명적인 파일 시스템 파손에 의한 비인가 이륙 시도를 사전에 차단하는 강력한 시스템 레벨 페일세이프(System-level Fail-safe)를 달성한다.

결과적으로 탑재 컴퓨터의 부트(Boot) 스크립트와 미들웨어 Lifecycle 제어를 논리적으로 융합한 이 아키텍처는, 전력 인가라는 전기적 시점을 “안전한 임무 개시 확인“이라는 비행 공학적이고 결정론적인 상태 보증으로 완벽히 통제하는 지상 준비 과정(Ground Preparation)의 무결성을 수리적으로 담보한다.
