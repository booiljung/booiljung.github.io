17.2.3 힙 파편화 방지를 위한 동적 할당 배제 및 정적 풀 활용 원칙

1. 개요

안전 필수(Safety-Critical) 하드 리얼타임 항공 소프트웨어 규격(예: DO-178C)은 자율 임무 시스템의 비결정론적 실패를 유발할 수 있는 잠재적 요인들을 소스코드 아키텍처 수준에서 원천 차단할 것을 강제한다. 이 중 임베디드 실시간 운영체제(RTOS) 제어기 환경에서 가장 치명적인 언해피 패스(Unhappy Path) 결함 원인으로 지목되는 것이 바로 힙 파편화(Heap Fragmentation)로 인한 메모리 고갈과 타이밍 락(Timing Lock) 현상이다. 본 절에서는 비행 제어기 내 실행 컨텍스트(Execution Context)가 시스템 자원과 데이터를 통신할 때 런타임 상의 동적 할당을 전면 배제해야 하는 구조적 이유와, 이를 보완하기 위한 대안으로 필수 채택되는 정적 역순 객체 풀(Static Object Pool) 설계 원칙을 심층 분석한다.

2. 힙 파편화(Heap Fragmentation)의 물리적 위협

자율 에이전트 드론의 마이크로컨트롤러(MCU) 주 기억장치인 수백 킬로바이트 한계의 정적 램(SRAM) 통제 공간 안에서, 수신되는 페이로드(Payload) 크기가 제각각인 하드웨어 이벤트 객체를 타이머 틱마다 빈번히 생성시키고(new) 스코프 밖에서 소멸시키면(delete), 커널이 관리하는 힙(Heap) 메모리 블록들 사이에 사용이 불가능한 미세한 틈새 영역들이 무수하게 방치된다. 이를 외부 파편화(External Fragmentation)라고 명명한다.

자율 비행 제어기는 단발성 운영으로 종료되지 않고 한 번 이륙하면 수십 분에서 수 시간 이상 연속 가동되어야 하는 장기 실행 제어 특성을 지닌다. 가동 시간이 누적됨에 따라 SRAM의 총 잉여 가용 메모리 블록 량은 충분함에도 불구하고, 그 파편들이 자잘하게 나뉘어져 있어 큰 덩어리의 연속된 메모리 공간을 찾지 못하게 된다. 이 시점에서 특정 센서 필터링 태스크가 동적 메모리 할당 공간을 운영체제에 요청할 경우 공간 부족 패닉(Out-of-Memory, OOM) 에러를 일으키며 시스템이 즉시 소프트 리셋(Soft Reset) 되거나, RTOS의 가비지 탐색기가 가용 블록을 찾기 위해 순회하는 데 인터럽트 보장 시간을 상회하는 비결정론적인 딜레이(Delay) 루프 사이클을 소모하게 만들어 결국 드론의 회복 불능한 물리적 추락을 촉발하게 된다.

3. 런타임 동적 할당(Dynamic Allocation) 배제 원칙

위와 같은 메모리 기반 크래시(Crash) 현상을 물리적으로 방어하기 위해 임베디드 자율 비행 제어기 아키텍처는 RTOS 부트 및 하드웨어 초기화(Initialization) 단계 과정만을 제한적으로 허용하고, 메인 런타임(Runtime) 제어 루프의 실행 진입 이후부터는 커널의 malloc(), free(), C++의 new, delete 연산자의 직접 호출을 아키텍처 룰로 직행 금지한다. 더 나아가, 내부 길이 가변 확장을 위해 메모리 재할당을 은닉 수행하는 STL의 동적 템플릿 컨테이너(std::vector, std::string, std::map 등) 사용 역시 철저히 통제된다.

실행 컨텍스트 주기마다 구조체를 동적으로 복사해 내거나, 조건 분기에서 스레드를 런타임에 새로 파생(Spawn)시키는 행위는 실시간 오버헤드 유발 및 MISRA-C++ 규격 설계 지침 위반으로 간주되며, 시스템 구동에 상시 필요한 코어 자원은 반드시 전역 변수(Global Variable) 영역(.bss 또는 .data)이나 이미 태스크 컨텍스트 생성 시 고정 할당된 스레드별 스택(Stack) 영역 프레임 내부 내부에서 정적 식별자로 소화되어야 한다.

4. 정적 메모리 풀(Static Memory Pool) 기반 연산 아키텍처

스택 오버플로우 한계와 가변 동적 할당 전면 배제 제약으로 인해 필연적으로 조우하게 되는 런타임 객체 수명 관리의 경직성을 논리적으로 타개하기 위해, 최신 프레임워크는 실행 컨텍스트 하단에 ‘고정 크기 정적 메모리 풀(Fixed-size Static Memory Pool)’ 아키텍처 패턴을 구축하여 해답을 구한다.

  • 오프라인 사전 할당(Pre-allocation): 시스템은 구동 초기화 단계 첫 시퀀스에서 고대역 텔레메트리 패킷, MAVLink 라우팅 메시지 버퍼, 하드웨어 인터럽트 핸들러 간의 비동기 전달 큐(Queue) 구조체 등이 런타임에 동시 최대로 유지될 수 있는 피크 수량(Worst Condition Peak Limit)을 산정하고, 이를 배열(Array) 형태의 정적 풀 연속 공간 힙 파생 영역에 미리 일괄 생성해 고정시킨다.
  • 시스템 콜 배제 차용 및 락 프리 반납: 실제 런타임 루프 중 센서나 이벤트 데이터 송신을 위해 메모리 컨테이너 생성이 필요할 때는, RTOS 단으로 진입하여 메모리를 ’신규 할당’받는 것이 결코 아니라, 이미 할당 고정된 풀 뱅크 내부의 단일 미사용 여분 빈 노드 주소를 상수 시간(O(1)) 내에 단지 ‘인덱스 갱신을 통해 빌려(Borrow)’ 온다. 해당 데이터 생명 주기가 완전히 만료된 페이로드 객체는 운영체제 메모리 해제 시스템 콜(호출 지연) 없이 곧바로 해당 구조체의 관리 클래스 내부 가용 명단 포인터로 논리적으로 ’즉각 반납(Return)’되므로 힙 할당 파편화 누적 곡선이 완벽하게 0(Zero)으로 수렴하는 물리적 무결성을 영구 보존하게 된다.
