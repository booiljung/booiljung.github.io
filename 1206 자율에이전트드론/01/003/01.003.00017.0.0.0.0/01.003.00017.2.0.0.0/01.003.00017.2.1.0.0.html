17.2.1 인터럽트 서비스 루틴(ISR)의 함수 호출 및 지연 제약

1. 개요

자율 비행 제어 시스템이 비동기적인 외부 물리 환경(예: IMU 센서 통신 버스의 데이터 도달, ESC 모터 제어 주기의 만료)과 즉각적으로 상호작용하는 1차적인 관문은 실시간 운영체제(RTOS)의 인터럽트 서비스 루틴(Interrupt Service Routine, ISR)이다. ISR은 프로세서 코어의 최우선 하드웨어 실행 권한을 선점(Preemption)하여 구동되므로, 이 좁은 문맥 구간 내에서의 연산 지연이나 부적절한 함수 호출은 코어 시스템의 실시간 결정론(Deterministic Real-time)을 치명적으로 훼손한다. 본 절에서는 비행 제어 RTOS 환경 하에서 ISR 아키텍처 설계 시 반드시 준수해야 하는 함수 호출 스펙과 고수준의 구조적 지연 제약을 학술적으로 규명한다.

2. 인터럽트 서비스 루틴(ISR)의 실행 컨텍스트 특수성

표준 태스크 스레드(Task Thread) 로직과 달리, ISR은 RTOS 스케줄러(Scheduler)의 알고리즘적 통제권을 우회하여 현재 실행 중인 백그라운드 태스크나 여타 스레드의 문맥(Context)을 강제로 동결시키고 즉시 물리적 실행(Immediate Execution)에 돌입한다.

이는 곧 ISR이 활성화되어 점유하는 수 마이크로초(µs) 동안 운영체제의 정상적인 멀티태스킹 스케줄링 메커니즘이 부분적으로 록(Lock) 상태에 놓이게 됨을 시사한다. 따라서 임베디드 드론 제어 이론에서 인터럽트 핸들러 내의 실행 컨텍스트는 매우 희소하고 값비싼 코어 자원으로 취급되며, 엔지니어는 이곳에서 어떠한 형태의 데이터 파싱이나 장시간의 산술 루프를 유발해서는 안 된다는 절대적인 설계 제약을 분담한다.

3. 블로킹(Blocking) API 호출 금지 원칙

ISR 컨텍스트 공간 내에서 RTOS 프레임워크가 부과하는 가장 엄격한 제약은 스레드의 상태 전이(State Transition)를 유발하는 시스템 API의 호출 원천 차단이다.

  • 동기화 원시 타입(Synchronization Primitive) 제한: ISR 내에서 상호 배제(Mutex) 획득을 위해 대기하거나, 일반적인 세마포어(Semaphore) 블로킹 함수(예: Take(), Wait())를 호출하는 것은 치명적인 안티 패턴(Anti-pattern)이다. 만약 ISR 핸들러가 이미 락(Lock)을 소유한 하위 스레드 자원을 대기하며 블로킹될 경우, 시스템 전체를 정지시키는 교착 상태(Deadlock)에 빠진다. 따라서 ISR 통신 구간에서는 시스템 제어권을 양보하지 않고 상태만 변경하는 전용 비차단(Non-blocking) API(예: FreeRTOS의 xSemaphoreGiveFromISR())만을 제한적으로 사용해야 한다.
  • 동적 메모리 및 직렬 I/O 연산 배제: malloc(), new 또는 printf()와 같은 C/C++ 표준 런타임 라이브러리 함수나 파일 입출력(File I/O) 계층은 내부적으로 스레드 블로킹 구조와 방대한 사이클의 실행 시간을 내포하므로 ISR에서의 호출 자체가 아키텍처 결함으로 직결된다.

4. 상단부(Top-half) 메커니즘과 연산 최소화 제약

고주파수 ISR 구조에서 절대적인 타이밍 오버헤드를 억제하기 위해 로보틱스 커널은 상단부(Top-half)와 하단부(Bottom-half) 분할 컴퓨팅 모델을 강제 수용한다.

물리적 ISR 영역에 직접 해당하는 상단부는 센서 핀(Pin)의 로우 레벨 인터럽트 플래그(Flag) 상태를 클리어(Clear)하고 SPI/I2C 하드웨어 버퍼로부터 원시값(Raw Data)을 단숨에 복사하여 적재하는 기구적 행위만을 완수한다. 이 상단부 공간 안에서 노이즈 필터링 연산(예: 보완 필터, 확장 칼만 필터)이나 부동소수점 수학 행렬(Floating Point Math Matrix) 연산을 시도하는 행위는 물리적 인터럽트 지연 시간(Interrupt Latency)을 초과시켜 기체의 추락 또는 후속 텔레메트리 인터럽트의 처리 누락(Missed Interrupts)을 유발하므로 정책적으로 철저히 차단된다.

5. 하단부(Bottom-half) 지연 처리(Deferred Processing) 아키텍처

상단부의 지연 허용 한계 제약을 해소하기 위해, 실제 센서 데이터의 고도화된 스케일링, 좌표 변환, 및 ROS2 토픽 발행(Publishing) 절차는 우선순위가 높은 독립적 후속 태스크 스레드로 위임된다.

상단부 ISR은 레지스터 데이터를 안전한 뮤텍스를 회피하는 락 프리 원형 버퍼(Lock-free Ring Buffer)에 적재한 뒤, 대상 하단부 센서 퍼블리셔 스레드나 자세 추정 제어기 스레드를 단순히 깨우는(Wake-up) 논-블로킹 알림(Task Notification) 신호만을 송출하고 최소 사이클 관문을 통과하여 즉시 종료된다. 인터럽트 프레임워크에서 빠져나온 프로세서 코어는 RTOS 스케줄러에 의해 선대기 중이던 해당 하단부 데이터 제어 태스크를 즉각 선점적으로 재개(Resumption)시킨다. 이는 하드웨어 인터럽트 지연과 블라인드 스팟(Blind Spot)을 최소화함과 동시에 로직 연산의 무결성을 시스템적으로 확보하는 지연 처리(Deferred Processing) 스레드 아키텍처를 완성한다.
