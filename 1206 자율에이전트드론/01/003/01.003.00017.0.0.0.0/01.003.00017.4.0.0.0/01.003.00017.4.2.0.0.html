17.4.2 작업 큐 아키텍처를 통한 비제어 데이터 연산의 권한 이관

실시간 자율 비행 제어 소프트웨어 아키텍처는 고주파수로 드론의 역학적 상태를 갱신하고 모터 제어 신호를 산출하는 ‘제어 연산(Control Computation)’, 그리고 시스템의 장기적인 운영을 보조하는 ’비제어 연산(Non-control Computation)’으로 대별된다. 비제어 연산에는 센서 데이터의 플래시 메모리 로깅(Logging), 진단 파라미터 데이터 직렬화(Serialization), 텔레메트리 상태 디버깅 및 네트워크 패킷 포매팅 등이 포함된다. 본 절에서는 이러한 비결정적이고 이질적인 연산 부하가 제어 주기를 침범하는 것을 방어하기 위해 시스템 코어에 적용되는 작업 큐(Work Queue) 아키텍처와 연산 권한 이관(Delegation) 모델을 학술적으로 분석한다.

1. 제어 컨텍스트 내 비제어 연산의 스케줄링 병목 선형화

고주파수 제어 스레드(예: 400Hz 이상의 자세 제어 루프) 내부에 비제어 연산이 통합되어 순차적으로 실행될 경우, 제어 루프의 최악례 실행 시간(Worst-Case Execution Time, WCET)은 본질적으로 통제 불가능한 상태에 이른다.

특히 SD 카드와 같은 비휘발성 대용량 저장 매체에 고해상도 비행 기체 로그를 기록하는 파일 입출력(File I/O) 하위 연산은 페이지 버퍼 관리 쓰기 지연, 가비지 컬렉션(Garbage Collection), 또는 메모리 통신 버스 대역폭 포화 상태에 따라 수 밀리초(ms)에서 수십 밀리초까지의 비결정적 지연(Non-deterministic Latency)을 임의로 발생시킨다. 만약 이러한 지연이 비행 제어 스레드의 선형적 연산 파이프라인 내부 계층에서 발생한다면, 제어기 신호 출력 주기의 편차(Jitter) 현상으로 즉각적으로 직결되어 기체의 고주파 진동, 시스템 위상 여유(Phase Margin)의 물리적 훼손, 최악의 경우 시스템 페이탈(System Fatal) 붕괴의 원인이 된다. 따라서 비제어 연산 서브 도메인의 실행 컨텍스트(Execution Context)를 물리적 제어 시간 범주에서 논리적 비동기 시간 범주로 격리하는 구조적 접근이 요구된다.

2. 작업 큐 아키텍처 및 권한 이관(Delegation) 모델 구체화

이러한 스케줄링 스레드 간섭을 원천적으로 배제하기 위해, 실시간 운영체제(RTOS) 환경에서는 임계 제어 스레드와 백그라운드 스레드를 메모리적으로 분리하는 작업 큐(Work Queue) 아키텍처를 필수로 적용한다. 메인 비행 제어 스레드는 비제어 직렬화 연산을 직접 수행하지 않으며, 단지 로깅 데이터 객체의 복사본 구조나 처리해야 할 연산의 함수 포인터 식별 정보(Job Descriptor)만을 비간섭 메모리 버퍼인 작업 큐에 비블로킹(Non-blocking) 방식으로 삽입(Enqueue)하는 책임 역할에 국한된다.

  1. 상태 데이터의 불변성 스냅샷(Immutable Snapshot): 고우선순위의 임계 제어 스레드는 현재 스케줄링 시점의 센서 공간 및 제어기 모델 파라미터 상태를 원자적 스냅샷 구조체 형태로 복제하여 작업 큐의 여유 엔트리에 데이터 링크를 할당한다.
  2. 연산 권한의 큐잉 및 도메인 이관: 이 불변성 스냅샷 메모리에 대한 직렬화, 암호화 패키징, 또는 하드웨어 I/O 인터페이스 버퍼로의 전송 계층 연산 등 모든 후속 과정에 대한 CPU 실행 점유 권한은 작업 큐(Work Queue) 스레드 시스템 관리 코어로 완전히 논리적 이관(Delegation)된다.
  3. 워커 스레드(Worker Thread)의 비동기 실행 메커니즘: 시스템 내 최소 중요도 범주(저우선순위 스케줄러 계층 수준)로 배정되어 대기 중인 다수의 워커 스레드 풀(Thread Pool)은 제어 스레드가 스케줄러 휴면(Sleep) 상태에 진입하여 CPU 코어 연산이 유휴(Idle) 상태가 된 시점에 RTOS 문맥 교환에 의해 활성화된다. 이후 작업 큐에서 데이터를 단일 인입하여 백그라운드 분리 환경 계층에서 비결정적 연산 소모 분량을 통제 가능하게 소진한다.

3. 큐 오버플로우 방어 및 버스트(Burst) 체인 제어

제어기 작업 큐를 통제하여 비제어 로깅 기반 데이터 이관 모델을 시스템 환경 내에서 설계할 시 시스템 통합 단계에서 반드시 고려해야 할 결함 매커니즘은 큐 할당 메모리의 데이터 오버플로우(Overflow) 고갈 문제이다. 고주파 비행 루프 태스크의 데이터 삽입 누적 속도가 워커 스레드의 시스템 I/O 한계 처리 속도를 연속적으로 상회할 경우(예: 저속 로깅 디스크 오류나 쓰기 속도 급감 시), 작업 큐 체인 공간에 무한히 쌓인 엔트리 블록 데이터는 시스템의 잔여 메모리 힙 자원을 지속적으로 포화시키는 런어웨이(Runaway) 현상을 일으킨다.

이를 방어하기 위하여 비행 제어기 시스템 코어 아키텍트는 작업 큐의 컨테이너 길이에 논리적/물리적 상한선(Bounded Size Queue Limits)을 정적 제약으로 설정하고, 큐 포화 데드라인 한계 도달 시 새로운 지연 가능 백그라운드 데이터 작업을 강제적으로 메모리 할당 없이 폐기(Drop)하는 선제적 자원 모니터링 관리 로직을 구현해야 한다. 어떠한 포화 상황에서도 제어 스레드의 타이밍 시간결정성을 견고히 보장하기 위해서는 비행 로그의 일시적 스킵(Truncation)이나 데이터 손실을 감수하더라도 메인 제어 루프 스레드 단에서 로직 블로킹 연산 대기(예: 큐가 비워질 때까지 대기하도록 하는 Spinlock 접근 제어)를 유발하는 구현 패턴 설계 관행을 시스템 소프트웨어 생명주기 레벨에서 엄격히 금지하는 것이 비행 제어 소프트웨어 무결성 유지 및 시스템 안전 페일 세이프(Fail-Safe) 프레임워크 구축에 있어 가장 핵심적인 학술적 설계 기반 지침이 된다.
