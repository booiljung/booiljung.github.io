16.3.3 데드라인 위반 회피를 위한 장기 실행 백그라운드 콜백 스레딩 모델

자율 드론 시스템에서 비행 제어 및 임무 관리는 본질적으로 장기 실행(Long-running) 특성을 가진다. 특정 경유지로 이동하거나 특정 구역을 정찰하는 단일 액션 목표라도 짧게는 수 초에서 길게는 수 분에 이르는 물리적 시간이 소요된다. 이러한 조건에서 ROS2 기반 시스템의 실시간성과 응답성을 확보하기 위해서는 데드라인 위반(Deadline Violation)을 시스템 구조적으로 회피할 수 있는 백그라운드 콜백 스레딩 모델(Background Callback Threading Model)의 설계가 필수적이다.

1. 장기 실행 임무에서의 콜백 블로킹 문제

ROS2 통신 아키텍처는 각 노드의 이그제큐터(Executor)가 큐(Queue)에 적재된 콜백(Callback) 함수를 스레드 풀(Thread Pool)에 할당하여 실행하는 방식으로 동작한다. 액션 서버가 목표 처리(Handle Goal) 또는 목표 수락(Handle Accepted) 과정에서 장기적인 물리 비행 로직을 기본 콜백 스레드 내에서 동기식(Synchronous) 루프로 처리할 경우, 치명적인 블로킹(Blocking) 현상이 발생한다.

수 분 동안 종료되지 않는 콜백 루프는 이그제큐터의 스레드 자원을 계속해서 점유하게 되며, 이는 드론의 자세 제어나 장애물 회피를 위한 고주파 센서 데이터 수신 콜백(예: IMU, Lidar)이 실행될 기회를 박탈한다. 결과적으로 콜백 대기 지연(Callback Latency)이 기하급수적으로 증가하여 실시간 제어 스케줄러의 데드라인을 위반하게 되며, 물리적 시스템의 불안정 혹은 추락으로 직결될 수 있다.

2. 분리된 백그라운드 작업 스레드(Background Task Thread)의 도입

블로킹 문제를 해결하기 위해, ROS2 액션 서버 아키텍처는 비즈니스 로직(행동 처리)을 통신 계층의 콜백과 물리적으로 격리하는 백그라운드 작업 스레드 패턴을 사용한다.

2.1 스레드 분리 메커니즘

목표 수락 콜백(handle_accepted_callback)이 호출되면, 타이머나 std::thread와 같은 별도의 비동기 실행 컨텍스트(Execution Context)를 즉시 생성하여 비행 관제 루프(Control Loop)를 오프로드(Offload)한다. 원래의 콜백 함수는 즉각 반환(Return)되어 이그제큐터에 스레드를 반납함으로써, 전체 시스템의 ROS2 메시지 처리 능력이 중단 없이 유지되도록 설계된다.

2.2 콜백 그룹(Callback Group)의 활용

멀티스레디드 이그제큐터(Multi-threaded Executor) 환경에서는 스레드 풀을 더욱 정교하게 분할하기 위해 상호 배타적 콜백 그룹(Mutually Exclusive Callback Group)과 재진입 가능 콜백 그룹(Reentrant Callback Group)을 조합하여 활용한다.

  • 장기 실행 목표를 감독하는 타이머 콜백은 독립된 재진입 가능 콜백 그룹에 배치하여 데드라인 위반의 영향을 시스템 전역으로 확산시키지 않는다.
  • 고주파 상태 피드백 노드는 배타적 콜백 그룹에 할당하여 작업 스레드의 무거운 연산이 제어 루프를 방해하지 못하도록 락(Lock) 경합 지점을 최소화한다.

3. 컨텍스트 전환 및 실행 상태 관리 메커니즘

비동기 백그라운드 스레드로 물리적 비행 제어 루프를 위임한 이후에는, 메인 ROS2 노드와 백그라운드 스레드 간의 안전한 상태 동기화 및 생명주기 관리가 요구된다.

3.1 동시성 제어 및 락-프리(Lock-free) 설계

백그라운드 스레드에서 지속적인 상태 평가(State Evaluation)를 수행하는 동안, 메인 노드에서는 사용자의 임무 취소(Cancel)나 우선순위 기반 선점(Preemption) 신호가 언제든지 도달할 수 있다. 두 스레드 간의 데이터 교환 시 상호 배제(Mutex)에 의한 병목 현상을 막기 위해, 최신 상태 플래그 확인 기법에는 std::atomic과 같은 락-프리(Lock-free) 메모리 접근 모델을 우선적으로 적용해야 한다.

3.2 안전한 종료 및 자원 반환

비행 목표 달성에 성공하거나 취소/선점 이벤트가 발생할 경우, 백그라운드 루프는 하드웨어 제어권을 즉각적으로 해제하고 스레드를 종료(Join/Detach)해야 한다. ROS2 액션 인터페이스 규격에 따라, 최종 물리적 비행이 정지한 사실을 확인한 직후 상태값(succeed, canceled, aborted)을 클라이언트에게 반환하고, 할당된 동적 자원들을 반납하는 과정이 지연 없이 수행되도록 스레딩 생명주기를 엄격하게 제어한다.

이러한 장기 실행 백그라운드 스레딩 모델은 단순히 프로그래밍의 용이성을 넘어, 드론 제어 시스템의 응답성과 결정론적 제어 한계(Deterministic Control Limits)를 지키기 위한 핵심 시스템 역량을 형성한다.
