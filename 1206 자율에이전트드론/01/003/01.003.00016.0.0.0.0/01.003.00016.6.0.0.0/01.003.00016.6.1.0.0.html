16.6.1 BT 액션 노드와 ROS2 클라이언트 계층 간 시맨틱 매핑

1. 개요

행동 트리(Behavior Tree, BT) 아키텍처가 자율 에이전트 시스템에서 실질적인 제어 권한을 행사하기 위해서는 BT의 추상적 제어 흐름 노드(Control Flow Node)들이 하위 로보틱스 미들웨어, 특히 ROS2 액션 인터페이스와 논리적으로 연결되어야 한다. 이를 ’시맨틱 매핑(Semantic Mapping)’이라 정의한다. 본 절에서는 단순한 API 래핑(Wrapping)을 넘어, BT의 실행 리프 노드(Execution Leaf Node)가 ROS2 클라이언트 계층(Client Layer)과 어떻게 상태 및 데이터를 동기화하고 의미론적인 무결성을 유지하는지 서술한다.

2. 시맨틱 매핑(Semantic Mapping)의 개념

시맨틱 매핑은 서로 다른 두 소프트웨어 아키텍처 패러다임 간의 상태, 데이터, 그리고 제어 의미(Semantics)를 변환하는 과정을 의미한다. BT는 본질적으로 ‘틱(Tick)’ 기반의 동기적 또는 비동기적 폴링(Polling) 모델을 따르는 반면, ROS2 액션은 요청-수락-피드백-결과로 이어지는 분산 이벤트 구동(Event-driven) 모델을 따른다.

따라서 BT 액션 노드는 이러한 두 체계 사이에서 어댑터(Adapter) 역할을 수행한다. 이 매핑 계층은 단순히 목표 전송(SendGoal)을 호출하는 것에서 그치지 않고, 네트워크 지연(Network Latency), 서버 거부(Server Rejection), 또는 로컬 선점(Local Preemption) 상황을 BT 구조가 이해할 수 있는 이산적 상태 변수(SUCCESS, FAILURE, RUNNING)로 번역한다.

3. BT 노드 생명주기(Lifecycle)와 ROS2 액션 상태의 동기화

시맨틱 매핑의 핵심은 두 컴포넌트 간의 생명주기를 일치시키는 데 있다. 일반적인 매핑 구조는 다음과 같은 위상 전이(Topological Transition)를 거친다.

  • 진입점(Tick/OnTick): BT 엔진에서 틱 신호가 특정 액션 노드에 처음 유입되면, 노드의 상태는 대기(IDLE)에서 실행(RUNNING)으로 전환된다. 이와 동시에 내장된 ROS2 액션 클라이언트는 액션 서버로 목표(Goal)를 비동기 전송한다.
  • 실행 루프(Running/Feedback): 서버가 목표를 수락(Accepted)하면, ROS2 프레임워크는 백그라운드 스레드에서 주기적인 피드백(Feedback) 콜백을 수신한다. BT 노드는 후속 틱 신호가 인가될 때마다 액션 클라이언트의 상태 비트를 조회하며, 액션이 연산 중일 경우 상위 트리 구조에 계속해서 RUNNING 상태를 반환한다.
  • 종료점(Halt/Cancel): 상위 트리 제어 노드에서 선점(Preemption) 조건이 만족되어 BT 엔진이 현재 노드에 중단(Halt) 신호를 하달하면, 시맨틱 매핑 계층은 이를 즉각 가로채어 ROS2 클라이언트의 목표 취소(CancelGoal) 서비스 요청으로 변환하여 통신망에 송출한다.

4. 블랙보드(Blackboard) 및 데이터 페이로드(Payload) 연동

데이터 계층의 시맨틱 매핑은 전역 저장소인 BT 블랙보드(Blackboard)와 ROS2 액션 메시지 통신 구조(.action) 간의 직렬화(Serialization) 및 역직렬화 메커니즘을 관리한다.

  • 입력 포트(Input Port) 매핑: BT 액션 노드의 입력 포트에 정의된 데이터(예: 이동을 위한 3차원 좌표 데이터)는 런타임 틱 주기 내에서 동적으로 읽혀 ROS2 Goal 통신 메시지 페이로드 구조체로 패킹(Packing)된다.
  • 출력 포트(Output Port) 매핑: 서버 측 연산이 완수되어 ROS2 계층으로부터 Result 메시지를 최종 수신하면, 해당 결과 데이터의 정합성을 파싱한다. 추출된 유효값은 노드의 출력 포트를 통해 다시 블랙보드 변수로 재할당되며, 이는 트리를 구성하는 후속 노드들의 연산 컨텍스트로 기능한다.
