16.6.2 BT 리프 노드의 Running, Success, Failure 상태 판정 기준 정규화

1. 개요

행동 트리(Behavior Tree, BT) 아키텍처가 자율 에이전트 드론의 복잡한 임무를 오동작 없이 수행하기 위해서는 트리의 종단점인 리프 노드(Leaf Node)에서 반환하는 상태값의 의미가 전 시스템에 걸쳐 보편적이어야 한다. 서로 다른 개발자가 작성한 액션 노드나 조건 노드들이 각기 다른 모호한 기준으로 상태를 반환할 경우, 상위 제어 흐름(Control Flow)이 예측 불가능해진다. 본 절에서는 BT 리프 노드가 부모 노드로 반환하는 세 가지 핵심 상태(RUNNING, SUCCESS, FAILURE)에 대한 엄격한 판정 기준을 정규화(Normalization)하는 엔지니어링 지침을 분석한다.

2. 행동 트리(BT) 리프 노드의 3상(Three-State) 모델

ROS2와 연동되는 BT 리프 노드는 틱(Tick) 신호를 받을 때마다 다음 세 가지 이산적 상태 중 하나를 반드시 반환해야 한다.

  • SUCCESS (성공): 노드에 부여된 논리적 혹은 물리적 목표가 설정된 허용 오차 내에서 완전히 달성되었음을 의미한다.
  • FAILURE (실패): 노드에 부여된 목표를 달성할 수 없는 물리적/논리적 결함 상태에 직면했거나, 검사 체계 산출 명제에서 거짓(False) 판정이 도출되었음을 의미한다.
  • RUNNING (실행 중): 목표를 달성하기 위한 구동 연산이나 비동기 작업 통신이 현재 순조롭게 진행 중이며, 즉각적인 종결 판정이 불가능함을 의미한다.

3. SUCCESS 및 FAILURE 판정의 정규화 기준

상태 판정의 정규화는 상태의 경계 조건을 수학적 오차 범위(Geometric Tolerance) 및 논리적 명제 형태로 명확히 규정하는 과정이다.

  • SUCCESS 정규화: 모션 액션 노드의 SUCCESS는 단순히 수치적 등식 성립이 아니라, 시스템의 추정 상태 공간(Estimated State Space)과 목표 공간 간의 유클리드 오차(Euclidean Error)가 사전에 정의된 유계(Bounded Tolerance) 이내로 수렴하고, 그 수렴 시점이 모델링된 안정화 시간(Settling Time) 이상 유지될 때만 선언되어야 한다. 허위 성공(False Success)을 방지하기 위해 노이즈 필터링이 선행되어야 한다.
  • FAILURE 정규화: FAILURE 상태는 일시적인 제어 지연만을 의미해서는 안 되며, 센서 입력의 완전 소실, 기구학적 임계점 초과, 또는 하위 ROS2 액션 서버로부터 명시적인 Aborted 상태 코드를 반환받은 경우에만 선언되도록 정규화해야 한다. 이는 Fallback 제어 노드가 불필요하게 트리거되어 임무가 진동(Thrashing)하는 현상을 억제한다.

4. RUNNING 상태의 연장 조건과 타임아웃(Timeout) 처리

가장 주의 깊게 설계해야 하는 판정 기준은 RUNNING 상태의 영속성이다. 통신이 단절된 비동기 작업 노드가 끝없이 RUNNING 상태를 유지하면 프레임워크 전체가 교착 상태(Deadlock)에 빠질 수 있다.

  • RUNNING 상태는 센서 피드백 스트림이 유효 기간 단위로 갱신될 때, 그리고 상위 노드의 틱(Tick) 주기와 하위 ROS2 피드백 통신 반응 주기가 허용 오차(Jitter Margin) 속에서 일치하는 구간 내에서만 허용된다.
  • 타임아웃(Timeout) 정규화: 모든 리프 노드는 모듈 자체적인 하드 타임아웃 조건식(T_{current} - T_{start} > T_{max})을 명시적으로 내장해야 한다. 이 임계치를 초과하여 유효한 피드백 없이 RUNNING 상태가 연장될 경우 즉시 하위 프로세스를 Cancel 처리하고 상위에 FAILURE를 강제 반환(Forced Transition)시키는 엄격한 보호 메커니즘이 강제되어야 한다.
