16.4.3 응답 타임아웃 방어 및 통신 단절 시 복구 트리거 로직

무선 통신 링크(RF, LTE, 5G 등)에 절대적으로 의존해야 하는 자율 에이전트 드론 아키텍처는 데이터 패킷 손실(Packet Loss), 비정상적인 대역폭 축소, 그리고 예측 불가능한 장기 통신 단절(Link Disconnection)이라는 통신 매체의 내재적 결함을 수반한다. ROS 2 환경 내에서 액션 클라이언트(Action Client) 기반 협력 통신을 통해 장기 실행(Long-running) 임무를 스케줄링할 때, 이러한 네트워크 인프라 결함은 통신 미들웨어(DDS 혹은 Zenoh) 계층을 넘어 애플리케이션 이그제큐터(Executor) 스레드의 무한 유휴 대기(Starvation) 상태를 유발하며, 종국에는 자율 비행 상태 머신 통제권 상실이라는 치명적 교착 상태(Deadlock)를 초래할 수 있다. 이를 소프트웨어 아키텍처 수준에서 방어하기 위해 결정론적 응답 타임아웃(Response Timeout) 기제와 물리적 통신 단절 상황의 복구 트리거(Recovery Trigger) 로직이 결합되어야 한다.

방어 체계의 1차 계층은 액션 통신 호출 시 하드 리얼타임(Hard Real-time) 제약에 준하는 워치독 타이머(Watchdog Timer)를 인스턴스화하여 병렬 구동하는 것이다. rclcpp_action 라이브러리를 통해 목표(Goal Request)가 하달되거나, 동기화된 시스템 내부에서 주기적 피드백(Feedback)을 청취할 때 액션 클라이언트의 콜백 그룹(Callback Group) 내부에는 타이머 노드가 주입된다. 시스템 아키텍처 설계 시 사전 정의된 최대 허용 지연 시간(Maximum Allowable Latency) 이내에 서버로부터 목표 승인/거부 응답이 당도하지 않거나, 진행 중인 액션의 생존 신호(Heartbeat) 혹은 피드백 데이터 갱신이 누락될 경우, 인터럽트 핸들러가 타이머 초과(Timeout Exception) 예외를 발생시킨다. 발생한 타임아웃 이벤트는 시스템 내부에서 대상 액션 서버의 컴퓨팅 크래시(Crash) 또는 미들웨어 토폴로지 붕괴로 해석된다.

타임아웃 감지가 확정되면 2차 계층인 통신 단절 감별(Isolation) 및 복구 트리거 파이프라인이 즉각 가동된다. 일반적인 임무 중단 상황과 달리, 네트워크 두절 상태에서는 async_cancel_goal과 같은 정상적인 비동기 취소 요청의 전달과 수신마저 보장할 수 없다. 따라서 중앙 임무 클라이언트 로직은 응답이 결여된 액션의 잔존 GoalHandle 객체와 퓨처(Future) 반환 대기열을 강제로 메모리에서 무효화(Invalidate) 및 해제(Teardown)시키고, 해당 서브시스템의 노드 상태를 오프라인(Offline) 모드로 강제 전이해야 한다. 이는 존재하지 않는 서버 응답을 대기하며 발생하는 좀비 스레드(Zombie Thread)의 자원 점유를 방지하는 필수 조치이다.

강제 해제 직후, 임무 스케줄러 복구 트리거는 자가 치유(Self-healing) 알고리즘 혹은 하드웨어 직결 페일세이프(Hardware Fail-safe) 계층을 순차적으로 격발(Trigger)한다. 논리적 복구 시도로는 QoS(Quality of Service) 데드라인 파라미터의 조정된 재협상 시도, 혹은 라이프사이클(Lifecycle) 매니저를 통한 대상 액션 서버의 재가동(deactivate 후 activate) 명령 브로드캐스팅(Broadcasting)이 포함된다. 만약 설정된 최대 재시도 횟수(Retry Constraint)를 초과하여 소프트웨어 구동 계층의 복구가 불가능함이 입증되면, 제어 패러다임은 응용 계층에서 즉각 이탈된다. 이후 비행 제어 유닛(Flight Control Unit, FCU) 내부의 저수준(Low-level) 오프라인 안전 논리가 발동되어, 드론은 공중 제자리 정호버링(Loitering) 모드에 진입하게 되며, 물리적 통신 두절 임계치(Critical Loss Threshold)를 돌파하는 순간 강제 홈 포인트 비행(Return-to-Launch, RTL)을 개시하여 기체의 물리적 보존을 완수하게 된다.
