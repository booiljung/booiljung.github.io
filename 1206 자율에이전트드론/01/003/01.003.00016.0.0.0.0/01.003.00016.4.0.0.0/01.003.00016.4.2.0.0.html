16.4.2 임무 강제 중단 인터럽트 처리 및 비상 복귀(RTL) 루틴 연결

자율 에이전트 드론이 수행하는 장기 지상 관측, 궤적 추종 등의 복합 임무는 동적 환경의 불확실성에 노출되어 있다. 따라서 예측 불가능한 치명적 이상 징후 (예: 배터리 전압 급감, 다중 모달 센서 결함, 국지적 통신 음영 발생 등)가 감지되었을 때, 현재 실행 중인 비행 액션을 즉각적으로 무효화하고 시스템을 안전 상태(Safe State)로 투사(Projection)하는 방어적 구동 체계가 요구된다. ROS 2 아키텍처 환경에서는 액션 클라이언트(Action Client) 주도의 선점형(Pre-emptive) 임무 중단 인터럽트(Interrupt) 처리 및 비상 복귀(Return-to-Launch, RTL) 루틴 파이프라인 설계를 통해 이러한 무결성(Integrity)을 확보한다.

임무 강제 중단 체계의 논리적 출발점은 실행 중인 액션 서버 내러티브에 취소 플래그(Cancel Flag)를 비동기 메커니즘으로 하달하는 과정이다. 클라이언트 애플리케이션 내부에 존재하는 진단 노드(Diagnostic Node)나 안전 감시(Safety Monitor) 스레드는 시스템 전역 상태를 고주파수(High-frequency)로 평가한다. 허용된 안전 임계치(Safety Threshold)를 위반하는 이상 상태 벡터가 식별될 경우, 임무 스케줄러는 지체 없이 현재 바인딩된 액션의 GoalHandle을 참조하여 async_cancel_goal 인터페이스를 호출한다. 이 호출은 ROS 2의 내부 액션 프로토콜 스택을 경유하여 서버에 도달하며, 서버 측 실행 컨텍스트(Execution Context)는 현재 연산 루프를 안전하게 종료하고 하드웨어 구동을 정지한 후 최종적으로 CANCELED 상태 코드를 반환한다.

단순한 목표 취소를 넘어서, 인터럽트 발생 직후 연속적인 재난을 방지하기 위해 비상 복귀(RTL) 루틴이 하드 리얼타임(Hard Real-time) 우선순위로 트리거(Trigger)되어야 한다. 이를 위해 중앙 임무 스케줄러 서브시스템(Subsystem) 내의 임무 큐(Mission Queue)는 완전 선점형 스케줄링 구조로 설계되어야 한다. 서버로부터 취소 수락 시그널과 함께 Result Callback이 CANCELED 상태로 반환되면, 큐에 적재되어 있던 잔여 일반 임무 항목들은 즉각 무효화(Invalidate)되거나 지연 메모리로 오프로딩(Off-loading)된다. 동시에 최상위 제어 권한(Supreme Override Authority)을 획득한 RTL 액션 객체가 큐의 선단으로 인입되어 디스패치(Dispatch) 준비를 마친다. RTL 액션 클라이언트는 기존의 복잡한 궤적 스케줄링 로직을 우회하여, 지정된 안전 고도 상승, 최단 원점(Home-point) 최적화 궤적 도출, 비상 착륙 절차만을 전담하는 독립된 RTL 전용 액션 서버로 새로운 비상 과업을 하달한다.

강제 중단 메커니즘과 RTL 체인 구성 시 소프트웨어 공학적으로 가장 주의해야 할 요소는 다중 노드 간의 제어 권한 교착(Deadlock) 및 교환 지연이다. 만약 기존 로컬 액션 서버의 스레드가 무한 루프에 빠져 있거나 과부하로 인해 async_cancel_goal 요청을 즉시 연산하지 못한다면, 액션 클라이언트는 CANCELED 응답을 무한정 대기(Starvation)하는 치명적 락(Lock) 상태에 직면한다. 이를 방지하기 위해 인터럽트 발행 시점에 맞춰 독립된 워치독 타이머(Watchdog Timer)가 가동되어야 한다. 사전에 협상된 데드라인(Deadline)을 초과하여 서버가 상태 전이 응답을 반환하지 않을 경우, 상위 스케줄러는 해당 액션 클라이언트 채널을 강제로 파괴(Teardown)하고, 비행 제어 유닛(Flight Control Unit, FCU) 수준의 저수준(Low-level) 명령 세트를 통해 즉각적인 개입을 강제하는 하위 페일세이프(Fail-safe) 레이어로 즉각 전환해야 한다.
