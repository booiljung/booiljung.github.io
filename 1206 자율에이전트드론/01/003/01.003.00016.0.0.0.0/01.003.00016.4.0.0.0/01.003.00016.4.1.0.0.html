16.4.1 상태 전이 추적 및 비동기 이벤트 콜백 체인 구성

자율 에이전트 드론 시스템에서 장기 실행(Long-running) 임무를 관리하기 위해서는 액션 클라이언트(Action Client)를 통한 정밀한 상태 전이(State Transition) 추적과 비동기 이벤트 제어 관리가 필수적이다. rclcpp_action 라이브러리를 기반으로 하는 ROS 2의 액션 통신 구조는 본질적으로 비차단(Non-blocking) 특성을 지니며, 이는 실행 과정에서 단일 스레드가 유휴 대기 상태에 빠져 발생할 수 있는 교착 상태(Deadlock)나 실시간성 제약(Real-time Constraints) 위반을 예방한다. 이러한 비동기적 아키텍처 내에서 연속적인 상태의 평가와 논리적 분기 판단을 이어가기 위해 이벤트 구동형(Event-Driven) 콜백 체인 디자인 패턴이 적용되어야 한다.

콜백 체인은 목표 요청(Goal Request) 전송 시점부터 반환되는 퓨처(Future) 객체에 연속적인 핸들러를 바인딩(Binding)하는 과정으로 시작된다. 액션 클라이언트는 async_send_goal 메서드를 호출하여 액션 서버로 비행 목표를 하달하고, SendGoalOptions 구조체 내에 구체적인 콜백 함수들을 등록함으로써 논리적인 콜백 체인을 구축한다.

첫 번째 연결 고리는 목표 응답 콜백(Goal Response Callback)이다. 액션 서버는 하달된 목표의 수용성(Acceptability)을 즉각 판단하여 응답을 반환한다. 콜백 함수 내부에서는 전달받은 GoalHandle 객체의 유효성을 검사하여 목표 승인 여부를 판별한다. 서버 측 컴퓨팅 리소스의 부족, 동역학적 범위를 벗어난 이상 목표치 하달, 혹은 사전 조건(Pre-condition) 불만족 등으로 인해 거부(Reject) 상태가 반환되는 경우, 클라이언트 애플리케이션의 상태 머신은 즉각 상위 스케줄러 노드로 실패 응답 코드를 전파하여 대체 비행 계획(Alternative Flight Plan) 수립을 트리거해야 한다.

두 번째 연결 고리는 피드백 수신 콜백(Feedback Callback)이다. 피드백 콜백은 장기 실행 액션 모델에서 웨이포인트(Waypoint) 간 궤적 추종 단계 등 연속적인 갱신이 필요할 때 주기적으로 호출된다. 이 수신 콜백은 액션 클라이언트가 서버에 능동적인 목표 취소 명령(Cancel Goal)을 발행할 시점을 결정하는 판단의 기준점(Criteria)을 제공한다. 예를 들어, 측풍(Crosswind) 등 심한 외란에 의한 궤적 이탈 오차가 허용 임계치(Threshold) 범위를 지속적으로 초과하거나 배터리 누수율이 급증하는 상태 지표가 피드백 데이터 패킷을 통해 수신되면, 콜백 체인 내부 논리 회로에서 인터럽트(Interrupt) 루틴을 가동하여 비동기 취소 요청(async_cancel_goal)을 호출한다.

세 번째 연결 고리는 액션 결과 콜백(Result Callback)이다. 해당 구간은 실행 목표의 최종 종결 상태(GoalTerminalState)를 수신하고 구조체를 분석하는 단계이다. 종결 상태 플래그는 크게 SUCCEEDED(임무 수행 성공), ABORTED(서버의 강제 중단에 의한 실패), CANCELED(클라이언트의 취소 요청 수용) 등으로 분류된다. 결과 콜백은 궤적 오차, 소요 시간, 소모 전력 등 해당 액션의 누적 통계 데이터를 해석 및 로깅(Logging)하며, 대기열에 순차적으로 존재하는 다음 스케줄러 큐(Scheduler Queue)의 임무 요소를 트리거(Trigger)하는 중추 역할을 담당한다. 결과 처리가 안전하게 완료된 후 연관된 퓨처 객체가 해제되며, 이그제큐터(Executor) 내 스레드 풀(Thread Pool) 자원이 시스템으로 반환된다.

비동기 이벤트 콜백 체인 구성 시 요구되는 가장 핵심적인 아키텍처 검토 사항은 동시성 제어(Concurrency Control) 메커니즘이다. 상태 전이 데이터 및 목표 핸들(Goal Handle) 추적 정보는 빈번하게 공유 자원(Shared Resource) 데이터 구조에 적재되므로, 다수의 콜백 그룹(Callback Group)이 별도의 스레드에서 병렬 교차 수행될 경우 심각한 메모리 레이스 컨디션(Memory Race Condition)에 직면할 수 있다. 이에 데이터 접근 블록을 뮤텍스(Mutex) 잠금 구조를 통해 보호하거나, 혹은 액션 통신과 관련된 모든 콜백 자원을 상호 배타적 이그제큐터(Mutually Exclusive Executor) 환경의 단일 콜백 그룹에 귀속시킴으로써 시스템 전반의 결정론적(Deterministic) 동시성 동기화를 보장해야 한다.
