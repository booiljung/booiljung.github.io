16.2.2 목표, 피드백, 결과 토픽 채널 매핑 체계

ROS2 시스템 프레임워크 단에서 애플리케이션 코드에 의해 선언되는 단일 모듈 형태의 액션(Action) 엔티티(Entity)는, 소스코드의 외부 인터페이스 상으로는 마치 하나의 통일되고 캡슐화된 단순 명령 인터페이스 구조체로 관측된다. 그러나 백그라운드에서 구동되는 ROS2 그래프 커널 브로커(Graph Broker)의 위상을 분해하여 추적해 보면, 액션을 구성하는 핵심 3요소인 목표(Goal), 상태 피드백(Feedback), 최종 결과(Result) 트랜잭션을 병렬로 파이프라이닝 처리하기 위해 총 5개의 상호 독립적인 특수 하위 토픽 및 서비스 채널들로 정교하게 매핑(Mapping) 분산되어 형성됨을 확인할 수 있다. 이러한 고도의 네트워크 채널 계층 분할 설계 사상을 통하여, 드론 시스템 내의 다수 이더넷 분산 노드들이 단 하나의 핵심 액션 서버 궤도를 동시에 상호간섭 없이 관찰 및 통합 제어할 수 있는 무결성 통신 토폴로지(Topology)가 전단에 확립 전개된다.

1. 액션 루트 네임스페이스(Namespace) 기반의 5단 채널 확장망

드론 비행 소프트웨어 개발자가 navigate_to_pose라는 명칭의 액션 서버 인스턴스를 메모리 상에 생성하여 띄울 경우, ROS2 코어 DDS 프레임워크는 내부적으로 이 명칭을 최상단 루트(Root) 네임스페이스로 삼아 다음과 같이 자동화된 5개의 하위 물리 통신 채널망을 동적으로 시스템에 맵핑(Mapping) 생성 및 런타임 개방시킨다.

  • ~/navigate_to_pose/_action/send_goal (Service): 최상위 클라이언트가 하부 제어기(Server) 스레드로 1차 계산된 목표 파라미터 텐서를 기계 전송하고, 하부의 하드웨어 한계 검증에 따른 승인(Accept)/기각(Reject) 상태 데이터를 가장 먼저 동기적(Synchronous)으로 응답받아오는 핵심 채널이다.
  • ~/navigate_to_pose/_action/cancel_goal (Service): 제어기가 한참 이행 구동 중인 특정 시계열 목표, 혹은 돌발 비상 상황 발생 시 서버가 엔진에서 처리 중인 모든 비행 목표 궤적 배열을 최상위 권한 스레드로 강제 선점 및 파괴 취소(Preemptive Cancel)할 때 호출 활성화되는 비정형 비동기 서비스망이다.
  • ~/navigate_to_pose/_action/get_result (Service): 장기 모터 제어 임무의 완전한 물리적 실행 종료 직후, 해당 기동의 종단 평가 지표인 Succeeded, Aborted 등의 영구 터미널 상태 플래그 핀과 센서 결과 페이로드를 클라이언트 측으로 반환 회수시키는 서브 스레드 콜백 채널이다.
  • ~/navigate_to_pose/_action/feedback (Topic): 타겟 좌표를 향해 험난한 공간을 뚫고 비행 중인 기체의 텔레메트리(Telemetry), 자세 오차율 행렬 등을 클라이언트로 실시간 브로드캐스트하는 퍼블리시/서브스크라이브 독립 무제어 토픽 채널이다.
  • ~/navigate_to_pose/_action/status (Topic): 액션 서버가 현재 캐시 풀에서 관리, 구동 중인 모든 활성 목표(Goal)들의 128비트 내부 상태 머신 식별자(ID) 배열 구조와, 각각의 최신 상태 전이 코드(예: ACCEPTED, EXECUTING, CANCELED) 맵을 매우 빠른 주기로 전체 네트워크 공용망에 지속 알림 송출하는 마트료시카(Matryoshka)형 디버깅 및 전체 전장 통제 채널이다.

2. 해시 암호화 UUID(Universally Unique Identifier) 기반의 궤적 트랜잭션 매핑

고성능 비행 메인 컨트롤 단일 액션 서버는, 네트워크 내에 흩어져 있는 다중 분산 클라이언트(예: 지상 통제소 클라이언트, 다중 자율 에이전트 플래너 클라이언트 등)로부터 동시다발적으로 비동기 목표 궤적들을 전송받아 시스템 큐에 병렬로 적재 처리할 가능성이 다분하다. 따라서 이 5단 융합 채널망에서 발생하는 방대한 패킷 핑퐁 통신의 상호 혼선 충돌을 물리적으로 완벽히 막아내기 위한 가장 엄밀한 해시 매핑 메커니즘이 강제 수반되어야 한다.

  • 128비트 무결성 해시 발급 체계: 특정 클라이언트가 임무 개시를 위해 send_goal 서비스를 아웃바운드 호출하는 즉각적인 순간, 프레임워크 코어는 128비트 길이의 암호화 역산이 불가능한 기계적 고유 UUID 텍스트를 생성하여 해당 목표 트랜잭션의 영구 고유 식별자로 발급 바인드(Bind)한다.
  • 패킷 섀도잉(Shadowing) 동기화 파이프라인: 1단계 승인 이후 파생 연산되는 후반부의 모든 feedback 메시지 텐서 조각들과 get_result 종단 콜백 반환 패킷 인터페이스 계층에는, 시스템이 발급한 이 UUID 해더 스탬프가 꼬리표처럼 강력하게 결합되어 섀도잉 트래킹(Tracking)된다. 클라이언트의 다원화 모델 멀티스레드 구조, 혹은 외부의 다수 로깅 노드가 동일한 하나의 피드백 토픽 기본 채널망(~/feedback)을 멀티캐스트로 동시에 도청 수신하고 있다 하더라도, 패킷 깊은 곳에 캡슐화되어 수신된 UUID 값을 로컬 필터링 파라미터 키(Key)로 역대조함으로써 여러 궤적 난무 속에서 오로지 자신이 요청했던 특정 단일 비행 목표(Goal)만의 진행 상황 및 결과물 텐서를 완벽하게 논리적으로 스레드 분리 추출해 낼 수 있다.

3. 분산 클라이언트 스웜(Multi-Client Swarm)의 상태 도청(Eavesdropping) 통신 트리

이와 같이 단일 엔티티를 5개의 분리된 계층망 채널로 해체하고 해시 UUID 매핑 토폴로지를 구축함으로써 얻는 가장 최상위의 강력한 아키텍처적 무기는, 물리적으로 이격된 다수의 이기종 분산 노드들이 단 하나의 제어 액션망을 클라이언트와 제어권 충돌 없이 수평적으로 전방위 모니터링할 수 있는 확장성을 확보한다는 점이다.

  • 설계 구조적으로 은닉된 모니터링 브리지(Monitoring Bridge)의 활용: 이 구조적 맹점을 역이용한 토폴로지 전술 하에서, 로봇 메인 임무 오케스트레이터가 비행 목표를 다이렉트로 주입하고 제어 스틱을 장악하는 1순위 오리지널 메인 클라이언트 역할을 독점 수행한다 하더라도 네트워크 제약이 발생하지 않는다. 가령 메인 코어와 분리배치된 오프보드(Off-board) 마이크로 컴퓨터 계층에 탑재되어 돌아가고 있는 검은 상자(Black-box) 데이터 로거 전용 노드나, 고정 IP를 보유한 지상 GCS 장거리 텔레메트리 스트림 중계 노드 모델은 비행 조종 및 서페이스 제어권 다툼에 일절 락(Lock)을 걸어 간섭 개입하지 않는다. 이 보조 노드들은 그저 백그라운드 환경에 조용히 은닉하여 오직 열려있는 토픽형 ~/feedback 채널망과 전역 추적용 ~/status 채널망의 스트림만을 비동기 서브스크라이브 패턴으로 한 발짝 물러나 일방향 도청(Eavesdropping) 수집할 수 있다. 이 토폴로지적 강점 패턴은 타겟 기체상의 메인 액션 서버 코어 CPU 스레드의 연산 오버헤드나 네트워크 캐시 부담률을 단 1%도 가중시키지 않고도, 대규모 멀티스레드 이더넷 네트워크 상에 완벽하게 결함 독립적이고 데이터 보존이 무결한 수평적 다중 분산 로깅(Distributed Logging) 감시망을 손쉽게 전개할 수 있는 전술적 기틀을 완성 부여한다.
