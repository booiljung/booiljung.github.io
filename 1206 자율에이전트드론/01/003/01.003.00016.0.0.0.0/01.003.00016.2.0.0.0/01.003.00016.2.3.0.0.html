16.2.3 액션 인터페이스 파일의 메시지 데이터 구조 및 IDL 직렬화

ROS2 액션 프레임워크가 5개의 하위 채널망 토폴로지를 횡단하며 복잡한 비동기 트래픽 통신을 무결하게 수행하기 위해서는, 목표(Goal), 최종 결과(Result), 그리고 실시간 피드백(Feedback) 페이로드들을 오차 없이 담아 운반할 수 있는 엄격히 규격화된 메시지 텐서(Tensor) 데이터 구조가 선행 확립되어야 한다. 이를 달성하기 위해 ROS2 생태계는 .action 이라는 특별한 확장자를 갖는 전용 인터페이스 정의 언어(IDL, Interface Definition Language) 규격을 사용하여 액션 메시지의 거시적 레이아웃을 선언 설계하고, 이를 하부 컴파일 계층에서 DDS 미들웨어 전용 바이트(Byte) 배열로 분해 해체하여 고속 직렬화(Serialization)해 내는 고도의 아키텍처 파이프라인을 작동시킨다.

1. 3단 횡단 분기 구조의 .action 인터페이스 텍스트 명세 및 설계

플러그인 패키지 단축에 선언되는 .action 파일은, 단 하나의 텍스트 문서 블록 내부에 세 개의 상호 독립적인 통신 구조체 구역을 --- (하이픈 3개) 문자 연산자로 엄격하게 수직 분리 구획하여 정의한다. 이러한 3단 분기 텍스트 구조는 액션 런타임의 시계열적 생애 주기(Lifecycle Time-series) 발현 순서에 따른 통신 페이로드의 물리적 진화 형태를 명세한다.

  • 상위 1구역 (Goal Definition): send_goal 상태 서비스 요청 개시 시점에, 최상위 클라이언트가 하위 역학 서버로 최초 1회 주입 전송하는 기하학적 데이터 텐서를 정의한다. 자율 드론의 3차원 공간 경로 추종 전술 액션(NavigateToPose.action)을 예로 들면, 이 구역에 geometry_msgs/PoseStamped pose 필드를 포함 명세하여 목표 3차원 GPS/로컬 위치와 쿼터니언(Quaternion) 목표 방위 제어 속성을 주입한다. 필요 시 비행 최대 속도 한계나 장애물 회피 마진율 등의 파라미터도 함께 이 구조체 필드 내에 정형화하여 포함 적재할 수 있다.
  • 중위 2구역 (Result Definition): 장비의 비행 기동이 끝나고 호출되는 get_result 서비스의 역응답으로, 수학적 터미널(Terminal) 상태 코드와 함께 클라이언트 루프로 반환 회수되는 최종 결과물 객체의 레이아웃을 정의한다. 단기 임무 완료 시의 단순한 긍정 표식을 위한 std_msgs/Empty부터, 복합 임무 완료 후의 텔레메트리 덤프인 성공 통과 웨이포인트 갯수 배열(int16 waypoint_passed), 착륙 패드와의 최종 엑츄에이터 오프셋 도달 에러(float32 final_error) 등이 포괄적으로 정의된다.
  • 하위 3구역 (Feedback Definition): 액션 제어기 루프가 맹렬히 가동되는 수십 분의 시간 구간 동안, 하위의 feedback 토픽망을 통해 백그라운드 위에서 끝없이 쏟아져 퍼블리싱되는 상태 모니터링 파라미터 변수를 정의한다. 수시로 갱신되는 현재의 텔레메트리 궤환 좌표(geometry_msgs/PoseStamped current_pose), 목표지점까지의 추정 잔여 이격 유클리드 거리(float32 distance_remaining), 예측 도달 시간(builtin_interfaces/Duration ETA) 등 시시각각 동적으로 불규칙 변화하는 역학 텔레메트리 배열 값을 고밀도 명세한다.

2. CMake 빌드 파이프라인과 이기종 브리지 IDL 직렬화(Serialization) 변환 메커니즘

작성된 .action 파일은 그 자체로는 C++나 파이썬 소스코드 런타임 단에서 직접 객체화되어 읽히지 못한다. 미들웨어 데이터망의 고주파 데이터 직렬화(Serialization)와 크로스 랭귀지(Cross-language) 네이티브 호환을 위한 심층적인 메타 컴파일 파이프라인 매핑과 빌드 시퀀스가 이면에 강제 요구된다.

  • rosidl_action_interfaces 커널 매크로의 메타 생성 분해: 메인 소스 코드 패키지의 CMakeLists.txt 내에서 rosidl_generate_interfaces 빌드업 매크로를 호출 트리거하면, ROS2 코어의 IDL 메타 컴파일러는 작성된 1개의 .action 파일을 3개의 잘게 쪼개진 하위 내장 .msg 파일(예: NavigateToPose_Goal, NavigateToPose_Result, NavigateToPose_Feedback)과 부가적인 백그라운드 서비스 인터페이스 호출 모듈 헤더로 산산이 해체(Decomposition) 분할 생성한다.
  • 이기종 보드 돌파를 위한 CDR(Common Data Representation) 바이너리 직렬화: 컴파일된 메시지 헤더 파일 소켓은, 하부 뼈대인 DDS(Data Distribution Service) 미들웨어 규격인 CDR 바이너리 바이트 스트림(Byte Stream) 포맷으로 클래스 데이터를 납작하게 직렬화(Serialize)하고 포인터 역직렬화(Deserialization)하는 인라인(Inline) 포인터 함수를 내부 시스템 코드적으로 자동 생성해 낸다. 이를 통해 메모리 주소 체계인 빅 엔디안(Big-Endian)이나 리틀 엔디안(Little-Endian) 메모리 정렬 구조가 각기 완전히 상이한 이기종 분산 보드(예: ARM 아키텍처 기반의 드론 비행 제어기와 x86 아키텍처 기반의 지상 관제 센터 컴퓨터) 사이에서도, 복잡한 3차원 액션 텐서 구조체가 단 1비트의 해석 오류나 찌그러짐 손실 오해 없이 완벽히 맵핑 전송 복원될 수 있는 우주선 수준의 범용성을 획득한다.

3. 프레임워크 내장형(Built-in) 헤더 및 암묵적 식별자 구조 포장(Wrapping)

개발자가 수동으로 직접 작성하는 3단 사용자 파이프라인 변수 외에도, 컴파일 타임(Compile Time) 환경에 ROS2 시스템 프레임워크 단에서 강제로 자동 주입 포장되는 암묵적(Implicit) 식별자 계층이 존재한다.

  • 128비트 Goal ID 암호화 헤더의 시스템 락(Lock) 자동 병합: C++ 직렬화 과정 초기에서 사용자가 선언한 Goal 페이로드 앞부분 헤더에는, 시스템이 난수 발급하는 128비트 unique_identifier_msgs/UUID 구조 타입의 해시 텐서 배열이 자동 병합(Merge/Wrap)된다. 이 헤더 포장지는 클라이언트 노드가 액션 서버로 목표를 발사 격발하는 그 찰나의 순간에 프레임워크가 자동 생성 개입하며, 이후 잇따라 방출생산되는 피드백(Feedback) 텐서와 결과물(Result) 페이로드 패킷 헤더 최상단에도 거머리처럼 완벽 동일하게 각인 복제되어 거대한 트랜잭션 추적망 연결고리를 물리적으로 구성한다.
  • 마스터 상태 코드(Status Code) 인덱싱 자동화 결합: 최종 종단 결과(Result) 반환 패킷 구조체에는 개발자가 직접 코딩 명세한 에러 변수 외에도, 액션 서페이스 내장 열거형(Enum)으로 엄격하게 하드코딩 포맷팅된 목표 상태값 변수(Status) 바이트가 추가 결합 적재되어 클라이언트에게 무조건 하달된다. 이는 0(Unknown), 3(Executing), 4(Succeeded), 5(Canceled) 등 시스템 코어 미들웨어에 불변 하드코딩된 규칙의 수를 따르며, 통신 데이터 검증 시 상위 클라이언트 노드가 거대한 페이로드 구조체를 전체 파싱(Parsing)하기 전에, 이 단일 상태 번호만을 우선 인터셉트 스캔하여 터미널 인터페이스의 무결성을 초고속으로 1차적 스위치 판독 확인해 내는 검문소 역할을 수행한다.
