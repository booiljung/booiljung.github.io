16.2.1 클라이언트-서버 간 3단계 양방향 상태 전이 통신 프로토콜

ROS2 액션 인터페이스 생태계를 구동하는 아키텍처의 절대적인 근간은 최상위 액션 클라이언트 관제 노드와 최하위 액션 서버 비행 엣지(Edge) 제어기 사이에서 엄밀하고 엄격하게 규정된 3단계(3-Phase) 양방향 상태 전이(State Transition) 프로토콜이다. 이 프로토콜은 거친 야외 환경에서 자율 비행 시스템이 운용 중 겪을 수 있는 수많은 엣지 케이스(Edge-case) 예외 상황(Exception)과 통신망의 물리적 데드락(Deadlock) 파급을 근원적으로 방어하기 위해 수학적으로 설계된 매우 정교한 비동기 핸드셰이킹(Asynchronous Handshaking) 메커니즘이다.

1. 제1단계: 목표 조율 및 하드웨어 승인(Goal Negotiation and Acceptance) 단계

이 단계는 클라이언트가 전송 하달하려는 장기 임무의 논리적, 물리적 유효성을 가장 말단의 하드웨어와 맞닿아 있는 서버가 단숨에 1차적으로 스캔 검증(Validation)하는 극초단(Micro-second) 동기화 단계이다.

  • 논블로킹 비동기적 목표 전송(Asynchronous Send Goal): 최상위 클라이언트 애플리케이션 플래너는 기하학적 비행 목표(예: action/NavigateToPose 메시지 타입 표장) 파라미터를 계산 텐서로 설정하고, 프레임워크가 제공하는 send_goal_async API를 통해 액션 서버로 비동기 송출 발사한다. 이때 클라이언트의 메인 제어 스레드는 즉시 락을 풀고 대기하지 않으며, 단지 미래에 회신될 퓨처(Future) 객체의 반환 이벤트만을 수동적으로 기다리는 논블로킹 상태의 콜백 리스너 큐 단을 편성해 둔다.
  • 서버 종단에서의 생존성 검증 콜백(Goal Callback Validation): 액션 서버 프로세스는 미들웨어 브리지 망에서 목표 페이로드를 꺼내 포장을 풀고, 즉각 등록된 on_goal 콜백 핸들러 인터럽트를 트리거한다. 이 찰나의 단계에서 서버는 수 밀리초(ms) 단위의 매우 얕은(Shallow) 물리적 한계선 스캔 검사를 수행한다. 즉, 수신된 목표 좌표가 지표면 아래(Under-ground) 구역인지, 현재 기체 배터리 BMS(Battery Management System) 컨디션으로 해당 거리까지 비행 가능한 물리적 도달 한계망 내에 있는지, 하드웨어 킬 스위치 페일세이프(Fail-Safe) 락(Lock)이 활성화되어 로터 제어가 불가능한 상태는 아닌지 등을 1차적으로 판독한다.
  • 결정론적 단기 응답 반환(Accept/Reject): 즉각적인 논스톱 검증 결과에 따라, 상태 머신은 목표를 공식 승인(Accept)하거나 즉각 기각(Reject)하는 분기를 확정 짓는다. 기각될 경우 퓨처 반환 객체는 에러 코드를 동반하며 클라이언트의 BT(Behavior Tree) 노드를 강제 Fail 시키고 예외 처리 폴백(Fallback) 로직으로 스위칭 시킨다. 반대로 승인될 경우 해당 목표 객체 포인터 수명은 공식 연장되어 액션 시스템 큐에 등록되고 제2단계 실행 궤도로 이행할 자격을 최종 부여받는다.

2. 제2단계: 물리적 활성화 및 스케줄링 모니터(Execute and Monitor) 전이 단계

목표가 시스템망에서 1차 승인 조율되었다고 해서 기체의 모터 변속기가 즉시 구동되어 반응하는 것은 아니다. 서버 내부 커널의 스케줄러가 메모리 자원을 할당 확보하고 물리 연산 스레드에 최종 제어권을 넘겨 전권을 위임하는 활성화 이행 단계가 필수적이다.

  • 실행 핸들러(Execution Handler) 연산 블록으로의 컨텍스트 스위칭: 대기열 큐에 등록된 임무는 액션 서버 타이머 인터페이스에 의해 Accepted 상태에서 Executing 런타임 상태 공간으로 밀려나는 상태 전이(Transition) 인터럽트를 겪는다. 이는 대상 시스템의 워커 스레드(Worker Thread) 풀의 on_execute 블로킹 콜백 내부를 트리거하며, 바로 이 블록 스코프 안에서부터 실제적인 드론의 비행 역학(Flight Dynamics) 궤도 연산이나 카메라 짐벌 락(Gimbal Lock) PID 제어 루프가 루프 타임을 돌며 맹렬히 물리 동작하기 시작한다.
  • 교차 관측 피드백(Cross-Feedback)의 개시와 방출: 해당 Executing 상태에 놓인 스레드 뎁스는 물리 연산 루프를 돌며 자발적으로 산출된 피드백 텐서를 뽑아내어 미들웨어로 브로드캐스트 송출한다. 클라이언트는 이 단계에서 실시간으로 끊임없이 갱신되는 3차원 위치 오차 행렬이나 최신 ETA(예상 도착 시간) 텔레메트리 스트림 데모를, 메인 스레드와 차단된 별도의 비동기 이벤트 핸들러를 통해 수신 가시화한다. 이 거시적인 2단계 공간은 목표가 물리적으로 기하 종단점에 도달 완료되어 부수 부러지거나, 외부 상위 로직의 인터럽트 명령에 의해 인위적으로 취소 격추당할 때까지 멈추지 않고 하드웨어를 지배하며 유지되는 가장 긴 장기 런타임 구간이다.

3. 제3단계: 물리적 터미널 전이 및 종단 결과 확정(Terminal Transition and Result) 단계

임무의 물리적 공간 기동이 어떠한 형태로든 최종 종료 처분되면, 액션 통신망 프로토콜은 시스템 할당 자원을 거두어들여 회수하고 완전한 시스템 락 상태 종료(Terminal State)를 확정 짓는 최종 마무리 단계를 엄격히 수행 집행한다.

  • 액션 상태 머신의 이탈 없는 3대 터미널(Terminal) 분기 벽: 비행 기동이 종료된 수학적, 혹은 물리적 조건에 따라 액션 서버 로컬 스케줄러는 현재 이행 중이던 목표 객체의 영구 상태성 값을 반드시 다음 세 가지 중 단 한 가지로 빈틈없이 확정 지어야만 한다. 제어 오차율 내에 만족하며 성공적으로 목표 물리 지점에 도착했다면 Succeeded, 비행 중 하드웨어 나침반 오류나 난기류 타격으로 인하여 궤적을 잃고 드롭 연산 한계를 초과했다면 즉각 Aborted를 각인한다. 만약 외부 클라이언트나 지상 운영자의 강제 취소 개입에 의해 안전하게 브레이킹 멈추어 섰다면 당당히 Canceled로 표기되어 영구 기록된다.
  • 최종 모음 결과(Result) 페이로드 송출 및 핸드셰이크 청산: 서버의 터미널 상태 핀이 락(Lock) 다운되어 버리면, 마지막으로 당해 역학 루프 연산의 관련 잔여 센서 배열 메트릭 데이터 집합(예: 최종 통과 성공한 웨이포인트 노드 갯수, 남은 오프셋 에러 통계율치)을 단 한 번의 종합 결과 패킷(Result Packet)으로 강력하게 압축 캡슐화한다. 그리고 상단에서 멍하니 잠들어 파동을 기다리던 클라이언트의 get_result 대기 퓨처 채널망 시스템 역방향으로 이를 강하게 송출 격발시킨다. 클라이언트 스레드는 이 마지막 종단 신호를 수신해 적재함으로써, 비로소 해당 기동 액션의 전체적인 라이프사이클 횡단망(Traversal Pipeline)을 완전히 소멸 청산하고 OS 커널에 쓰레기 수집 객체 반환(Garbage Collection)을 명령하게 된다.
