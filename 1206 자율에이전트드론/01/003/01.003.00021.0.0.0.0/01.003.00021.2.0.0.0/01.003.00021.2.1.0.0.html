21.2.1 리눅스 정밀 타이머(High-Resolution Timer) 프레임워크 및 hrtimer API의 동작 한계율

전통적인 저해상도 범용 타이머 구조가 지닌 시스템 틱(jiffies)의 본질적 한계를 극복하기 위해, 리눅스 커널 2.6 기반부터 나노초(ns) 단위의 기저 타이밍을 지원하는 고해상도 타이머(High-Resolution Timer, hrtimer) 프레임워크가 도입되었다. 현대의 ROS2 미들웨어 및 어플리케이션 계층은 clock_nanosleep, timerfd 등의 유저 스페이스(User Space) API를 통해 내부적으로 이 hrtimer 서브시스템을 활용하여 고주파수 제어 타이밍을 확보하려 시도한다. 그러나 하드웨어 클럭 소스를 직접 참조하는 구조에도 불구하고, 유저 스페이스에서 드론 비행 제어기가 체감하는 실효적 정밀도는 hrtimer의 수학적 명세를 심각하게 밑도는 한계율(Limit Rate)을 보인다.

1. hrtimer 프레임워크의 구조적 구현체 및 동작 원리

hrtimer는 O(N)의 비효율적인 리스트 검색 대신 시간 복잡도 O(log N)을 보장하는 레드-블랙 트리(Red-Black Tree) 자료구조를 활용하여 타이머 만료 이벤트를 스케줄링한다. 커널은 메인보드에 실장된 고정밀 하드웨어 타이머 장치인 HPET(High Precision Event Timer), LAPIC(Local Advanced Programmable Interrupt Controller) 또는 CPU의 TSC(Time Stamp Counter)를 클럭 소스로 삼아 나노초 해상도의 절대 시간 혹은 상대 시간을 측정한다.

새로운 타이머가 hrtimer API를 통해 설정되면, 커널은 트리의 가장 왼쪽에 위치한 만료 노드(Earliest Expiry Node)의 시간을 읽어 하드웨어 타이머의 다음 인터럽트 시점을 프로그래밍한다. 만료 시각에 도달하여 하드웨어 인터럽트가 발생하면 커널 모드의 인터럽트 서비스 루틴(Interrupt Service Routine, ISR)이 트리에서 해당 노드를 순차적으로 파기하고 등록된 만료 콜백 함수들을 발동시킨다. 이는 커널 내부(Kernel Space) 구조상 마이크로초 초반대 이내의 극도로 낮은 지연율(Latency) 모델을 물리적으로 구성할 수 있음을 의미한다.

2. 커널-유저 스페이스 경계선 상의 치명적 레이턴시 병목 현상

hrtimer 서브시스템이 나노초 단위의 정밀도를 담보한다는 명제는 ’타이머 인터럽트가 발생하는 시점’에 국한된 커널 스페이스(Kernel Space) 내의 진실이다. ROS2에 기반한 비행 제어 알고리즘은 유저 스페이스(User Space)의 스레드 환경에서 구동되므로, 커널의 만료 인터럽트가 유저 스레드의 깨어남(Wake-up)으로 전이되기까지 거대한 오버헤드 파이프라인을 통과해야 한다.

hrtimer 콜백이 가장 먼저 수행하는 작업은 유저 스페이스에서 대기 중이던 비행 제어 스레드의 상태를 TASK_INTERRUPTIBLE(대기 상태)에서 TASK_RUNNING(실행 가능 상태)으로 전환하여 런큐(Run-queue)에 삽입하는 것이다. 그러나 이 즉시 제어 연산이 실행되는 것은 아니다. 이후 커널 스케줄러가 개입하여 현재 CPU 코어를 점유 중인 스레드를 축출(Preempt)하고 비행 제어 스레드로 문맥을 전환(Context Switching)하는 과정이 반드시 동반된다.

이 때 발생하는 파이프라인의 오버헤드는 크게 세 가지로 분류된다.

  1. 시스템 호출(System Call) 반환 궤도 이탈로 인한 트랩(Trap) 오버헤드
  2. 하드웨어의 CPU 캐시(Cache) 및 TLB(Translation Lookaside Buffer) 플러시(Flush)에 따른 캐시 미스 가중치
  3. 가상 메모리 매핑 교체 소요 시간

이러한 레이턴시는 물리적 메모리 아키텍처의 상태에 따라 수십에서 수백 마이크로초(\mu\text{s})에 달하는 지연을 필연적으로 유발하며, 이는 제어 루프가 목표한 \Delta t를 고정적으로 유지하지 못하게 만드는 원초적 지터(Jitter)의 요인이 된다.

3. 로컬 인터럽트 비활성화 및 락(Lock) 경합에 따른 실효 동작 한계율

더 구조적인 취약점은 리눅스의 비선점형 임계 구역(Non-preemptible Critical Section)에서 발현된다. 비행 제어기가 고주파수 hrtimer를 가동하는 도중 동시에 로깅 시스템의 대용량 디스크 I/O나 고대역폭 네트워크 연산(Video Streaming 등)이 발생할 경우, 다른 커널 스레드 혹은 디바이스 드라이버가 스핀락(Spinlock)을 확보하거나 local_irq_disable() 매크로를 호출해 로컬 CPU의 인터럽트를 차단할 수 있다.

이러한 상태에서 hrtimer의 하드웨어 타이머 만료 시점이 도달하더라도, 인터럽트를 수신하는 즉각적인 ISR(Interrupt Service Routine) 호출이 차단되어(Pending) 해당 임계 구역 연산이 종료될 때까지 최대 수 밀리초(ms) 지연 상태에 빠지게 된다. 즉, ROS2 노드 내부에서 설정한 제어 주파수가 1,000Hz(1ms 주기)라 하더라도 커널 하위단의 동시성 제어 경합(Concurrency Control Contention)에 의해 개별 틱이 손실(Drop)되거나 연달아 폭주하는(Burst) 형태로 앨리어싱(Aliasing)이 축적된다.

결국 리눅스의 hrtimer 프레임워크는 유저 스페이스 프로그램 관점에서 볼 때 하드웨어 클럭의 명세 해상도(Resolution)와 달리, 실제로 보장받을 수 있는 정시성 즉 ‘동작 한계율(Operational Limit Rate)’ 측면에서 범용 스케줄러 오버헤드 및 임계 구역 차단이라는 거대한 장벽에 직면한다. 이는 ROS2 기반 제어 시스템이 소프트웨어 레벨 타이머에만 의존해서는 물리계의 안전 필수(Safety-critical) 다자유도 동역학 보상 임무를 확정적으로 책임질 수 없음을 방증하는 학술적 결론으로 귀착된다.
