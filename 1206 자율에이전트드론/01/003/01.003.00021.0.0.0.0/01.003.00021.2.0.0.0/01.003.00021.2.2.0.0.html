21.2.2 동적 틱리스(Tickless) 커널 정책 하의 컨텍스트 스케줄링 간섭 현상 규명

현대의 리눅스 커널은 모바일 및 서버 환경의 폭발적인 전력 효율화 요구에 부응하기 위해 구조적인 전환을 꾀하였으며, 그 중심에 동적 틱리스(Dynamic Tickless) 연산 정책 체계가 존재한다. 커널 컴파일 옵션인 CONFIG_NO_HZ_IDLE (또는 극단적 형태인 CONFIG_NO_HZ_FULL)로 활성화되는 이 메커니즘은 CPU가 유휴(Idle) 상태에 돌입할 경우 주기적인 시스템 틱(System Tick) 인터럽트를 과감히 중단(Suspend)시킴으로써 전력 소모를 비약적으로 절감한다. 그러나 400Hz 수준의 초고주파 연산 단위 위에서 비행 역학을 유지해야 하는 자율 에이전트 드론의 입장에서는, 이 틱리스 메커니즘이 오히려 컨텍스트 스케줄링 과정에 치명적이고 비결정론적인 간섭(Interference)을 유발하는 시스템 역행 요소로 작용하게 된다.

1. 틱리스(Tickless) 커널의 구조적 작동 기제

고전적인 커널 스케줄러는 시스템에 실행할 스레드가 존재하지 않아도, 설정된 CONFIG_HZ(예: 250Hz 혹은 1,000Hz) 주기에 맞추어 맹목적으로 하드웨어 타이머 인터럽트를 발동시킨다. 이는 CPU 코어를 유휴 전력 모드에서 지속적으로 벗어나게 하여 불필요한 배터리 소모를 야기했다. 이를 방지하고자 설계된 틱리스(Tickless, NO_HZ) 아키텍처는 런큐(Run-queue)가 완전히 비워진 코어에 한해 주기를 관장하는 스케줄링 인터럽트 자체를 완전히 꺼버린다(Tick off).

틱이 중지된 상태에서 커널은 다음으로 예약된 가장 이른 타이머 만료 시점(hrtimer 등)까지 시스템 틱을 ‘건너뛰기’ 할 수 있도록 로컬 APIC(Advanced Programmable Interrupt Controller) 타이머를 재프로그래밍(Reprogramming)한다. 이는 운영체제가 깨어나는 횟수를 근원적으로 줄일 수 있다는 측면에서 유용하지만, 커널의 기준 시간인 jiffies의 갱신 주기 자체가 고무줄처럼 변모하는 동적 시간 체계 시스템으로의 전환을 의미한다.

2. 유휴 복원 레이턴시(Wake-up Latency)와 타이머 지터(Jitter) 스파이크 폭증

문제는 틱리스 상태로 깊은 절전 상태에 빠진 CPU 코어가 ROS2 비행 제어 노드가 요구하는 타이머 만료 신호를 수신하여 런큐를 복원하고 유저 스레드로 진입하는 순간에 폭발적으로 발생한다. CPU 클럭 파이프라인의 전력 공급을 복구하고, 중지되었던 인터럽트 테이블의 컨텍스트를 커널 공간 메모리로 다시 로딩하는 과정(Wake-up sequence)은 본질적으로 수십에서 수백 마이크로초(\mu\text{s})에 달하는 하드웨어 복원 레이턴시(Exit Latency)를 동반하게 된다.

고주파수 비행 제어 루프는 비록 1ms 단위로 극히 짧은 연산을 수행하지만, 연산이 종료되는 즉시(Yield) 런큐가 일시적으로 비워지는 양상을 보이게 된다. 이때 틱리스 알고리즘은 오판을 일으켜 CPU를 절전 상태로 파견하려 시도하며 틱 주기 스케줄링을 재프로그래밍하는 오버헤드를 발생시킨다. 그리고 불과 1 \sim 2\text{ms} 뒤에 제어 루프가 다시 호출될 때 시스템은 절전 모드 해제 페널티를 거치며 실행 주기를 어긋나게 만든다. 결과적으로 틱리스 메커니즘 하에서 구동되는 고주파 타이머는 규칙적인 위상 지연이 아닌 돌발적인 레이턴시 스파이크(Spike)로 발현되며, 이로 인한 지터(Jitter)의 증폭은 제어계의 위상 여유(Phase Margin)를 순식간에 붕괴시킨다.

3. CFS의 오산과 프로세스 간섭(Interference)의 심화

더불어 틱리스 모드는 완벽하게 공정한 스케줄러(CFS)의 태스크 로드 균형(Load Balancing) 계산에도 간섭을 일으킨다. 시스템 틱이 일정한 주기로 울리지 않으므로, 커널은 각 프로세스가 소모한 가상 런타임(vruntime)을 인터럽트 시점마다 소급하여 계산(Catch-up)해야 한다. 이러한 비재귀적 소급 계산은 제어 스레드의 실행을 일시적으로 지연시키는 커널 공간 내의 연산 병목으로 작용한다.

또한 멀티 코어 환경에서 CONFIG_NO_HZ_FULL 설정을 통해 단일 코어를 무중단(Tickless) 모드로 전환하려 하더라도, RCU(Read-Copy-Update) 콜백, IPI(Inter-Processor Interrupt), 커널 레벨 워크큐(Workqueue) 등의 백그라운드 소프트 인터럽트(SoftIRQ)들이 틱리스 코어 내부로 침범하여 실행 흐름을 파편화시키는 간섭 현상을 완전히 차단할 수 없다. 즉, ROS2 제어 노드가 리눅스의 타이머 트리 위에서 아무리 나노초 단위의 정밀도를 요청하더라도, 절전과 효율에 경도된 동적 틱리스 스케줄링 정책 하에서는 일관된 컨텍스트 스위칭 윈도우를 확정받을 수 없다. 이는 자율 드론의 비행 역학이 요구하는 굳건한 \Delta t 이산 시간 모델이 소프트웨어 기반 환경에서는 심각하게 왜곡되고 간섭됨을 학술적으로 증명한다.
