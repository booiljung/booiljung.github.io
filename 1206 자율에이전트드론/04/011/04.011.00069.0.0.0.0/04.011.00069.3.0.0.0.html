69.3 YAML 명세서 기반 브리지 포트 매핑(Topic Mapping) 규칙 및 노드 네임스페이스(Namespace) 격리 메커니즘

1. 개요

ros_gz_bridge 환경에서 ROS 2 제어 노드와 Gazebo 시뮬레이터 간의 메시지 교류는 수많은 토픽(Topic)들의 실시간 교환을 수반한다. 단일 드론뿐만 아니라 다중 에이전트(Multi-Agent) 스웜(Swarm) 비행 시나리오에서는 브리지를 통과하는 데이터 스트림이 기하급수적으로 증가하며, 토픽 이름의 충돌 현상(Naming Collision)이 발생할 우려가 높다. 이를 체계적으로 관리하고 메모리 안정성을 확보하기 위하여, 브리지 노드는 CLI 기반의 임의 인자 전달 방식 대신 정형화된 YAML(YAML Ain’t Markup Language) 명세서를 통한 정적 포트 매핑 규칙을 채택하며, 네임스페이스(Namespace) 격리 메커니즘을 통하여 에이전트 간 통신 독립성을 보장한다.

2. YAML 명세서(Specification) 기반 포트 매핑 규칙

ros_gz_bridge는 초기화 시점에 --ros-args -p config_file:=<path_to_yaml> 옵션을 통하여 외부 конфигурация 파일을 로드한다. 이 파일은 배열(Array) 형태의 매핑 엔트리를 포함하며, 각 엔트리는 ROS 2 네트워크와 Gazebo 네트워크 양단의 토픽 종단점(Endpoint) 속성을 명시적으로 정의한다.

표준 매핑 규칙은 다음과 같은 필수 속성(Properties)을 상속한다.

  1. 식별자 및 경로 제어 (Identifier & Path Control):
  • ros_topic_name: ROS 2 DDS 공간 내부에 노출될 토픽 명칭. (예: /drone_1/cmd_vel)
  • gz_topic_name: Gazebo Ignition 전송 레이어 내부의 토픽 명칭. (예: /model/drone_1/cmd_vel)
  1. 데이터 타입 명세 (Data Type Declaration):
  • ros_type_name: 포장 해제 후 ROS 2에서 인식할 메시지 타입. (예: geometry_msgs/msg/Twist)
  • gz_type_name: 퍼블리시/서브스크라이브 시 사용되는 Protobuf 메시지 타입. (예: gz.msgs.Twist)
  1. 흐름 제어 (Directional Flow Control):
  • direction: 통신의 벡터 방향을 결정한다. ROS_TO_GZ, GZ_TO_ROS, BIDIRECTIONAL 중 하나를 채택한다. 자원 최적화를 위하여, 실제 데이터가 단방향으로만 전송되는 센서나 제어 명령의 경우 절대적으로 단방향 흐름 규칙을 적용하여 불필요한 루프백(Loopback) 오버헤드를 근절해야 한다.

3. 다중 에이전트 환경을 위한 노드 네임스페이스(Namespace) 격리 메커니즘

단일 브리지 노드 내에서 여러 대의 드론 모델을 운용할 경우, 예를 들어 /imu 토픽을 그대로 전방위적으로 매핑하게 되면 DDS 상에서 수많은 에이전트의 센서 정보가 중첩되어 제어 알고리즘의 발산을 초래한다. 이를 관리하기 위한 네임스페이스 격리 메커니즘은 로컬 프레임(Local Frame) 규칙과 런치(Launch) 파이프라인의 푸시다운(Push-down) 전략 기반 위에서 동작한다.

  1. 네임스페이스 결합 (Namespace Concatenation) 토폴로지:
    YAML 명세서 정의 시, Gazebo 월드(World)에서 할당한 모델의 이름(예: drone_alpha, drone_beta)을 ROS 2 측 ros_topic_name의 최상위 네임스페이스 접두어(Prefix)로 강제 할당한다.
    예: ros_topic_name: /drone_alpha/sensor/imu, gz_topic_name: /world/default/model/drone_alpha/link/base/sensor/imu/imu
  2. ROS 2 런처 내 네임스페이스 푸시 (Namespace Push in Launch):
    단일 거대 브리지 노드가 아닌, 에이전트 수만큼의 분산된 경량 브리지 노드를 선언할 경우, Node(package='ros_gz_bridge', ...) 선언 부에 namespace='drone_alpha' 파라미터를 인젝션한다. 이 경우 YAML 내부에 상대 경로(Relative Path)를 사용하더라도 ROS 2 RMW가 자동으로 네임스페이스 공간을 캡슐화(Encapsulation)하여 정보 교란을 차단한다.
  3. TF 브로드캐스팅(Transform Broadcasting) 접두어 적용:
    브리지가 Gazebo의 포즈 데이터(gz.msgs.Pose_V)를 ROS 2의 /tf 트리에 퍼블리시할 때, ros_gz_bridge는 각 객체의 frame_id에 네임스페이스 스트링을 연결(Append)하여 다중 에이전트의 기준 좌표계가 글로벌 월드(Global World) 상에서 독립적으로 연산되도록 정합성을 통제한다.

4. 런타임 검증 및 학술적 시사점

정립된 YAML 명세서와 네임스페이스 격리 구조는 시뮬레이터 구동 초기화 시점에 브리지 노드의 파싱 루틴을 거치며 문법(Syntax) 오류와 타입 불일치(Type Mismatch)가 일차적으로 검증된다. 이러한 정적 선언(Static Declaration) 기반의 아키텍처는 코드 변경 없이 다양한 센서 조합 및 에이전트 군집(Swarm) 패턴을 실험할 수 있는 유연성을 제공한다. 또한, 각 종단 모델 간 결합도(Coupling)를 낮추는 소프트웨어 공학의 단일 책임 원칙(Single Responsibility Principle)을 충족시킴으로써 대규모 자율 비행 시스템의 재현 가능성(Reproducibility) 확보에 중대한 학술적 기여를 한다.
