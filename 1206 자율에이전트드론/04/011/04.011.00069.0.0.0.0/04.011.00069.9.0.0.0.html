69.9 C++ 환경 내 커스텀 메시지 타입(Custom Message Type) 교환을 위한 인터페이스 정의 언어(IDL) 브리지 팩토리(Factory) 구현론

1. 개요 및 커스텀 메시지 교환의 필요성

Octuple 로터 시스템이나 다관절 로봇 팔을 장착한 에이전트 드론과 같이 특이점이 높은 자율 이동체 시스템은 ROS 2 시스템의 표준 메시지(Standard Messages, std_msgs, sensor_msgs 등)만으로는 제어 상태나 시뮬레이션 로직을 충분히 모사할 수 없다. 픽셀 클러스터 기반 객체 정보 목록, 특정 배터리 배열의 비대칭 방전 셀 상태, 혹은 고유한 BMS(Battery Management System) 진단 코드와 같은 사용자 정의 도메인 지식은 커스텀 메시지 타입(Custom Message Type)의 개발을 요구한다. 그러나 이러한 비표준 메시지가 곧바로 ros_gz_bridge를 통과할 수는 없으며, 양단(ROS 2 데이터 구조와 Gazebo 프로토콜 버퍼)의 정합성을 매핑하기 위해 C++ 인터페이스 정의 언어(IDL, Interface Definition Language) 및 브리지 팩토리(Bridge Factory) 컴파일러 확장이 수반되어야 한다.

2. 양방향 IDL(Interface Definition Language) 데이터 명세화

커스텀 타입의 교차 역직렬화 및 직렬화를 달성하기 위한 첫 단계는, 두 이종 미들웨어 환경 각각에 맞는 IDL 구조체를 정의하는 것이다.

  1. ROS 2 .msg 명세: 해당 드론 프로젝트의 msg 디렉토리 내에 커스텀 필드를 갖는 정의 파일(예: DroneDiagnostic.msg)을 작성해야 한다. 이 파일에는 표준 C++ 타입 매핑 규칙에 따라 uint8, float64[], string 등의 변수들을 나열한다.
  2. Gazebo .proto 명세: ros_gz_bridge는 상대 계층인 gz-transport와 소통하기 위하여 Google Protocol Buffers 컴파일러 프로토콜 형태(예: drone_diagnostic.proto)를 요구한다. 여기서는 syntax = "proto3"; 규약 하에 package gz.msgs; 네임스페이스를 할당하고 필드마다 고유 태그 번호를 부여하여 직렬화 무결성을 지킨다.

3. C++ 템플릿 변환 함수(Conversion Template Function)의 설계

두 프레임워크에 대한 데이터 명세가 완료된 후, 브리지 노드가 런타임에 메모리를 복사할 수 있도록 C++ 환경에서 변환기(Converter) 역할을 수행하는 코어 헤더(Header) 파일들을 작성해야 한다. 이 변환 함수들은 팩토리 패턴(Factory Pattern)의 핵심적인 이행 요소이다.

변환 인터페이스는 ros_gz_interfaces 패키지와 유사하게 두 개의 오버로드(Overload)된 템플릿 형태로 구현된다.

  • ROS 2 \rightarrow Gazebo 방향 (convert_ros_to_gz):
    입력된 ROS 2 커스텀 메시지 구조체(예: drone_msgs::msg::DroneDiagnostic)의 각 멤버 변수를 순회하며 대응되는 Protobuf 객체(예: gz::msgs::DroneDiagnostic)의 Mutator(설정형) 메서드(예: set_voltage())에 복사해 넣는다.
  • Gazebo \rightarrow ROS 2 방향 (convert_gz_to_ros):
    수신된 Protobuf C++ 파싱 객체의 Accessor(조회형) 메서드를 호출하여, ROS 2 C++ 구조체의 대응 멤버에 복제한다.

이러한 설계에서 동적 배열(Dynamically Sized Array)과 같은 복합적인 데이터는 C++ 표준 템플릿 라이브러리(STL)의 std::vector와 Protobuf의 RepeatedField 간 루프(Loop) 연산을 통하여 명시적으로 번역(Translation)되어야 메모리 누수나 크래시를 방지할 수 있다.

4. ros_gz_bridge 확장 팩토리(Factory) 플러그인 빌드 파이프라인

단순한 C++ 함수 객체의 생성만으로는 구동 엔진에 토픽이 노출되지 않는다. ros_gz_bridge의 팩토리 매니저가 새로 개발된 커스텀 타입을 동적으로 식별하도록 하려면 아래의 패키지 컴파일 및 인젝션 파이프라인을 구축해야 한다.

  1. CMake 의존성 선언: 확장할 패키지의 CMakeLists.txt 내에서 앞서 생성한 ROS 2 패키지와 Gazebo Protobuf 생성 타깃을 모두 find_package 또는 pkg-config로 링킹(Linking)한다.
  2. 공유 라이브러리 플러그인 링킹: ros_gz_bridge는 확장 기능을 외부 브리지 플러그인 라이브러리(.so 형태)로부터 검색하여 런타임에 삽입한다. 따라서 CMake 매크로에서 add_library(...)로 공유 라이브러리를 빌드하고, 이를 팩토리 레지스트리에 등록하는 C++ 매크로(예: GZ_REGISTER_BRIDGE_FACTORY)를 호출하도록 구성해야 한다.
  3. 환경 변수 구성: 최종적으로 ROS 2 작업 스페이스를 컴파일(예: colcon build)한 후, 터미널 환경에 ROS_GZ_BRIDGE_EXTENSION_PATH 환경 변수(Environmental Variable)를 로드하여 바이너리 런처가 컴파일된 공유 플러그인을 인지할 수 있도록 명시적으로 가이드한다.

이 C++ 브리지 팩토리 확장 프로세스는 시뮬레이션 개발자에게 높은 수동적 난이도를 부여하지만, 다중 로터 시뮬레이션의 도메인 특화 모델링과 자율주행 제어 루프의 완전한 동기화를 위한 핵심적인 학술적(Engineering) 해법이다.
