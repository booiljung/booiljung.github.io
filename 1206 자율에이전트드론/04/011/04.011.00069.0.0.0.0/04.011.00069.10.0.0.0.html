69.10 하드웨어-인-더-루프(Hardware-In-The-Loop, HIL) 실증 전환을 대비한 브리지 네트워크 통신 토폴로지(Topology) 스와핑(Swapping) 패턴

1. 개요 및 HIL(Hardware-In-The-Loop) 전환의 연속성 과제

자율 에이전트 드론의 개발 파이프라인은 논리적 알고리즘 검증을 위한 순수 소프트웨어 시뮬레이션(Software-In-The-Loop, SIL) 단계에서 물리적 탑재 제어기(Pixhawk 등)와 실제 이종 센서를 연동하는 하드웨어-인-더-루프(Hardware-In-The-Loop, HIL) 단계로 진화한다. 이 전환 과정에서 가장 큰 학술적, 공학적 과제는 기존에 Gazebo 물리 엔진 노드와 통신하던 ROS 2 알고리즘 스택 코드(예: 센서 융합 SLAM, 제어기)를 단 한 줄도 수정하지 않고 물리적 하드웨어의 I/O 스트림으로 원활하게 교체(Swapping)하는 재사용성(Reusability)의 확보다. ros_gz_bridge는 이러한 통신망 스와핑 토폴로지를 구현하는 핵심 분리 계층(Decoupling Layer)으로 작용한다.

2. 인터페이스 분리 원칙(ISP) 기반 브리지 퍼사드(Facade) 아키텍처

ROS 2 스택 내의 핵심 주행 노드들은 데이터를 제공하는 주체가 가상의 플러그인인지, 아니면 실제 하드웨어 센서 드라이버(예: V4L2 카메라 드라이버, urg_node 라이다 드라이버)인지 인지해서는 안 된다. 이를 구현하기 위해 브리지 계층을 퍼사드(Facade) 패턴으로 정립해야 한다.

  1. 일관된 의미론적 네임스페이스(Semantic Namespace):
    SIL 모드에서 ros_gz_bridge가 퍼블리시하는 토픽 이름(예: /drone/camera/image_raw, /drone/imu/data)과 QoS 프로파일은, 향후 HIL 모드에서 실제 하드웨어 드라이버 노드가 생성할 토픽의 명세와 바이트 단위까지 완전히 동일해야 한다.
  2. 데이터 타입 투명성(Data Type Transparency):
    Gazebo 고유의 포맷을 사용하지 않고 ros_gz_bridge를 통해 반드시 ROS 2의 sensor_msgs 및 nav_msgs 표준 인터페이스 포맷으로 변환함으로써, 수신하는 비전 제어 노드 코드가 msg.header.frame_id만을 참조하도록 캡슐화(Encapsulation)한다.

3. 런치 시스템(Launch System) 레벨의 통신 토폴로지 스와핑(Swapping)

실제 이행 단계에서는 ROS 2의 런치(Launch) 파이썬 라이브러리가 제공하는 실행 시점 조건부 파라미터 제어를 통해 SIL 브리지 토폴로지와 HIL 드라이버 토폴로지를 스위칭한다.

  1. 실행 변수(Launch Argument)를 활용한 브랜칭:
    런처 실행 시 use_sim 또는 is_hil과 같은 부울(Boolean) 파라미터를 인젝션한다.
  2. 가상 시뮬레이터 브리지 분기 (use_sim:=true):
    이 분기가 활성화되면 런치 시스템은 ros_gz_bridge 노드를 인스턴스화하고 YAML 포트 매핑을 가동한다. 동시에 글로벌 파라미터 use_sim_time=True를 모든 노드 스택에 전파하여 논리 시간에 동기화시킨다.
  3. HIL 센서 드라이버 분기 (use_sim:=false):
    ros_gz_bridge의 생성을 억제하고, 대신 실제 하드웨어 소켓 인터페이스(예: UART 패킷 캡처, 실제 라이다 이더넷 포트 청취)를 점유하는 물리 드라이버 노드들을 백그라운드에서 기동시킨다. 이때 use_sim_time=False로 하여 각 하드웨어의 실제 시스템 클럭(Wall-clock)을 기준으로 TF 트리를 구성하게 한다.

4. 모드 무결성(Integrity) 검증 및 브리지 멀티플렉싱(Multiplexing)

단순 토픽 교환을 넘어 HIL 환경에서는 일부 센서는 가상 엔진(Gazebo)에 의존하고, 제어기는 실제 하드웨어(Pixhawk)를 사용하는 하이브리드 스트리밍이 빈번하게 요구된다.

이 경우 ros_gz_bridge는 멀티플렉서(Multiplexer) 역할을 겸해야 한다. 예를 들어 cmd_vel 토픽은 ROS 2에서 Gazebo로 흐르는 브리지 규칙(ROS_TO_GZ)을 폐기하고 실제 시리얼 포트 변환 노드로 라우팅되는 반면, 값비싼 3D LiDAR 점군 데이터는 여전히 가상 엔진으로부터 수신받는(GZ_TO_ROS) 반쪽짜리 브리지 매핑 룰셋을 별도의 YAML 명세서(예: hil_bridge_mapping.yaml)로 한정 관리하여 적용한다.

결과적으로, 철저히 계획된 체계적 브리지 토폴로지 스와핑 메커니즘을 통하여, 연구 시뮬레이션 코드베이스와 상용 실기체 배포 코드베이스를 단일 버전 관리 리포지토리(Git)의 바이너리로 통합 운영(Single Source of Truth)할 수 있는 최상위 소프트웨어 공학적 유연성을 달성하게 된다.
