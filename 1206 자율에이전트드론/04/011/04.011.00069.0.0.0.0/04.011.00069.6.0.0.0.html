69.6 시뮬레이션 논리 시간(Simulation Time) 퍼블리셔(use_sim_time) 연동 및 글로벌 /clock 토픽 동기화(Synchronization)

1. 개요 및 시뮬레이션 논리 시간(Simulation Time)의 본질

자율 에이전트 드론의 내재적(Intrinsic) 소프트웨어 아키텍처는 센서 퓨전(Sensor Fusion), 궤적 계획(Trajectory Planning), 그리고 제어 루프(Control Loop)의 인과성을 유지하기 위하여 정교한 시계열(Time-series) 동기화를 요구한다. 실제 환경(Hardware)을 비행할 경우 시스템은 운영 체제의 벽시계 시간(Wall-clock Time) 또는 하드웨어 RTC(Real-Time Clock)를 참조한다. 그러나 Gazebo(Ignition)와 같은 디지털 트윈 시뮬레이터 환경에서는 컴퓨팅 자원의 동적 부하에 따라 물리 엔진의 연산 속도가 실제 시간보다 느려지거나(Real Time Factor < 1.0) 가속될 수 있다. 따라서 시뮬레이터와 교신하는 모든 ROS 2 노드는 물리적 시스템 시간이 아닌, Gazebo가 주도하는 시뮬레이션 논리 시간(Simulation Time)을 기준으로 동작해야 하며, 이를 구속하는 핵심 기반 노드가 글로벌 /clock 토픽 동기화 파이프라인이다.

2. Gazebo 시계열 발행(Publishing) 메커니즘과 /clock 토픽 브리징

Gazebo 물리 엔진은 스텝(Step) 연산이 한 사이클 완료될 때마다 내부 시뮬레이션 타임스탬프를 증가시키며, 이를 gz.msgs.Clock 포맷으로 gz-transport 상에 퍼블리시한다. 이 이기종 타임스탬프를 ROS 2의 공식 시간 정보로 치환하는 첫 번째 단계는 ros_gz_bridge 노드를 통해 글로벌 /clock 토픽을 단방향(GZ_TO_ROS)으로 매핑하는 것이다.

YAML 매핑 명세서의 구성은 다음과 같다.

  • ros_topic_name: /clock
  • gz_topic_name: /clock
  • ros_type_name: rosgraph_msgs/msg/Clock
  • gz_type_name: gz.msgs.Clock
  • direction: GZ_TO_ROS

브리지가 이 매핑을 활성화하면, Gazebo가 생성한 가상 시간 스텝 정보는 ROS 2 DDS 공간 내에서 브로드캐스트된다. 이때 타임스탬프는 초(Seconds)와 나노초(Nanoseconds)를 포함하는 rosgraph_msgs/msg/Clock 구조체로 역직렬화 및 직렬화되어 네트워크의 모든 에이전트에게 전달된다.

3. use_sim_time 파라미터 연동의 역학(Mechanics)

단순히 /clock 토픽이 DDS 상에 존재한다고 해서 ROS 2 노드들이 즉각적으로 시스템 시간을 버리고 논리 시간을 참조하지는 않는다. 로봇 제어, SLAM, VIO(Visual Inertial Odometry) 노드들이 논리 시간을 올바르게 추종(Follow)하게 하려면, 노드 초기화(Initialization) 단계에서 use_sim_time이라는 글로벌 ROS 파라미터를 명시적으로 활성화(True)해야 한다.

  1. 시계(Clock) 추상화 인터페이스 전환:
    ROS 2 C++ API(rclcpp::Node) 또는 Python API(rclpy.node.Node)는 내부에 관측 가능한 시계 객체(get_clock())를 소유한다. use_sim_time이 False(기본값)일 때 함수 now()는 OS 기반의 시스템 시간을 반환한다. 그러나 이 파라미터가 True로 바인딩되는 순간, 노드 코어는 내부적으로 /clock 토픽에 대한 서브스크립션을 활성화하고 now() 함수의 리턴값을 가장 최근 수신된 /clock 페이로드 데이터로 덮어쓴다(Override).
  2. 노드 실행 구문(Launch)에서의 글로벌 파라미터 주입:
    대규모 드론 소프트웨어 스택에서는 수평적으로 전개되는 수십 개의 노드 각각에 파라미터를 일일이 전달하는 것은 극히 비효율적이다. 따라서 XML/Python 런치 구조에서는 SetParameter 이벤트 핸들러를 사용하여 런치 스코프 내의 모든 자식 노드들에게 use_sim_time을 일괄 상속시킴으로써, 시뮬레이션 환경 내 시간의 결정론적(Deterministic) 정합성을 일거에 보장한다.

4. 논리 시간 동기화 부전(Desynchronization) 시 아노말리(Anomaly) 현상 진단

use_sim_time 파라미터가 누락되거나, 브리지 오버헤드로 인해 /clock 토픽의 전송 지연이 발생할 경우 자율 드론 제어 시스템에는 다음과 같은 치명적인 상태 전이 오류가 발생한다.

  • TF(Transform) 버퍼 만료 현상: ROS 2의 tf2_ros 라이브러리는 센서 데이터 포즈를 보관하는 타임스탬프 큐를 갖는다. Gazebo 플러그인에서 생성된 센서 데이터는 시뮬레이션 논리 시간을 헤더 스탬프에 기입하여 송출하는데, 수신하는 제어 노드가 런타임 시스템 시간(Wall-clock)을 기준으로 작동할 경우, 헤더의 시간이 시스템 시간과 크게 어긋나 TF 보간(Interpolation)에 실패하고 Extrapolation Exception을 끝없이 호출한다.
  • 제어 루프(Control Loop) 발산: 타이머 주기로 작동하는 PID 제어기나 LQR 최적화기는 미분항 \Delta t를 시계의 차이로 계산한다. 시뮬레이터가 잠시 멈추거나 느려졌을 때(가상 시간이 일시 정지 상태), 제어 속도(Velocity)와 가속도(Acceleration) 계산 루틴이 잘못된 \Delta t를 채택함으로써, 정상 상태로 복귀 시 엄청난 적분 오차(Wind-up)를 발생시켜 쿼드콥터가 추락(Crash)하는 현상을 유도한다.

따라서 시뮬레이터 중심의 아키텍처 통신 설계 시, 브리지 노드의 우선 설정 우선순위 1순위는 타 데이터 토픽보다 절대적으로 /clock 토픽의 가장 안정적인 QoS 정책(RELIABLE, TRANSIENT_LOCAL) 수립 및 use_sim_time 메타 계층 선언에 있음을 명심해야 한다.
