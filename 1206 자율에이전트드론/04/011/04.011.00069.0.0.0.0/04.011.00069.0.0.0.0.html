Chapter 69. ROS2와 Gazebo 물리 엔진 간의 ros_gz_bridge 메시지 브리지 포트 매핑 (출처: OSRF)

1. 개요 및 통신 아키텍처의 분리성(Decoupling)

ROS 2(Robot Operating System 2)와 Gazebo(Ignition/Harmonic 이후 버전)는 각각 독립적인 통신 미들웨어 생태계를 기반으로 작동한다. ROS 2는 DDS(Data Distribution Service) 표준을 채택하여 분산 노드 간 실시간 통신을 보장하는 반면, Gazebo는 gz-transport 라이브러리를 통해 ZeroMQ 백엔드 기반의 자체적인 게시/구독(Publish/Subscribe) 토픽 모델을 운용한다. 두 시스템 간의 메모리 공간과 프로토콜이 근본적으로 단절되어 있으므로, 시뮬레이션 환경의 물리 엔진 연산 결과(센서 데이터, 기구학적 상태 등)를 ROS 2 제어기로 전달하고 반대로 제어 명령(추력 분배 매트릭스, 목표 포즈 등)을 시뮬레이터로 피드백하기 위해서는 데이터 직렬화(Serialization) 형식을 상호 변환해 주는 양방향 게이트웨이(Gateway)가 필수적이다. 이를 담당하는 핵심 패키지가 Open Source Robotics Foundation (OSRF)에서 개발 및 유지보수하는 ros_gz_bridge이다.

2. ros_gz_bridge의 역할 및 직렬화 포맷 변환 논리

ros_gz_bridge 노드는 ROS 2 네트워크와 Gazebo 네트워크의 경계면에서 양단의 토픽(Topic)을 동시에 구독(Subscribe)하고 게시(Publish)하는 병렬 스레드를 운용한다. 페이로드(Payload) 교환 과정에서 양방향 변환기는 아래와 같은 기하학적, 물리적 속성의 마샬링(Marshalling) 및 언마샬링(Unmarshalling)을 수행한다.

  1. 메시지 타입(Type) 매핑: ROS 2의 표준 메시지 인터페이스(예: sensor_msgs/msg/Image, geometry_msgs/msg/Twist)와 Gazebo의 프로토콜 버퍼(Protocol Buffers, Protobuf) 기반 메시지 타입(예: gz.msgs.Image, gz.msgs.Twist) 사이의 필드 매핑 딕셔너리(Dictionary)를 통해 타입을 동적으로 변환한다.
  2. 타임스탬프(Timestamp) 동기화: Gazebo 시뮬레이션 세계의 가상 시간(Sim Time)과 ROS 2 시스템 시간 간의 오프셋(Offset) 불일치를 방지하기 위해 use_sim_time 파라미터를 강제하고, 메시지 헤더(Header)의 스탬프를 Gazebo의 주기율(Tick)에 맞추어 보정한다.
  3. 좌표계 교정 (Transform Frame Calibration): 경우에 따라 좌표계의 규약이 다를 수 있으므로(예: ROS 2 표준인 FLU(Forward-Left-Up) 모델과 일부 시뮬레이터 모델 간의 축 변동), 헤더의 frame_id 문맥을 유지하며 쿼터니언(Quaternion)의 사상(Mapping)을 일치시킨다.

3. 포트 매핑 (Port Mapping) 파라미터 규격 및 시방서

ros_gz_bridge 실행 시 어떤 토픽을 매핑할 것인지 정의하는 конфигурация(Configuration) 파일(YAML 포맷) 또는 인자(Argument) 인터페이스가 요구된다. 포트 매핑 시방서(Specification)는 양방향 트래픽 방향, 토픽 변환 규칙, QOS(Quality of Service) 정책을 포함한다.

OSRF 권장 설정에 따른 YAML 기반 매핑 명세의 기본 구조는 다음과 같다 (버전: Harmonic 호환 기준).

  • ros_topic_name: ROS 2 환경에서 사용할 토픽 명칭의 절대 경로.
  • gz_topic_name: Gazebo 내부 프레임워크 상의 식별자(보통 월드 이름과 모델 이름 포함).
  • ros_type_name: ROS 2 메시지 데이터형의 정규표현 (예: geometry_msgs/msg/Pose).
  • gz_type_name: Protobuf 메시지 형의 정규표현 (예: gz.msgs.Pose).
  • direction: 데이터 흐름의 방향성 제어. 단방향 대역폭 최적화를 위해 매우 중요하며 다음 중 하나를 갖는다.
  • ROS_TO_GZ: ROS 명령을 Gazebo로 보냄 (예: 제어 조이스틱 속도 입력).
  • GZ_TO_ROS: Gazebo 센서 데이터를 ROS로 수신 (예: LiDAR 점군 데이터, IMU 단위).
  • BIDIRECTIONAL: 양방향 교환 (루프백 현상에 주의).

4. 커스텀 인터페이스 및 런타임 통신 최적화 고려사항

복잡한 자율 드론의 경우, 단순 표준 메시지를 넘어서 비대칭 페이로드 윈치 제어나 배터리 BMS 상태와 같은 사용자 정의 시스템 식별 토픽을 개발해야 한다. 이 경우 ros_gz_bridge의 소스 코드 내부에 위치한 C++ 헤더 변환기(Factory Pattern)에 새 ROS 2 .msg 패키지와 .proto 파일을 주입하여 리빌드(Re-build) 과정을 거쳐야 한다.

또한 영상 스트리밍(Camera RGB/Depth)이나 3D LiDAR 포인트 클라우드와 같은 고대역폭 페이로드를 ros_gz_bridge로 이송할 경우, ROS-DDS 도메인과 Gazebo 도메인 사이의 메모리 복사 오버헤드(Memory Copy Overhead)가 프레임 저하를 초래한다. 이를 해결하기 위해 최근 버전에서는 브리지를 통하지 않고 ROS 2 노드가 직접 공유 메모리 레벨에서 Gazebo 플러그인에 접근하는 ros_gz_image 타입의 전용 제로-카피(Zero-Copy) 전송 노드를 사용할 것을 학술적, 실무적으로 권장하고 있다.
