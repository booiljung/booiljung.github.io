Chapter 66. SDF(Simulation Description Format) 문법 구조와 동역학 물리 속성 매핑

자율 에이전트 드론(UAV)의 개발 생태계에서 로봇 공학자들은 종종 기구학적 표현의 표준인 URDF(Unified Robot Description Format)와 시뮬레이션 환경의 표준인 SDF(Simulation Description Format) 사이에서 아키텍처적 선택의 기로에 놓인다. URDF가 ROS 생태계 내에서 단일 로봇 프레임의 위상 기하학적 트리(Tree)와 TF 연산을 정의하는 데 최적화되어 있다면, SDF는 OSRF(Open Source Robotics Foundation)에 의해 Gazebo 물리 엔진의 고유(Native) 언어로 설계되어 빛, 중력, 표면 마찰, 복수 로봇의 상호작용, 그리고 폐쇄 루프(Closed-loop) 기구학을 통합적으로 관장하는 거시적 환경 스키마이다. 이 장에서는 SDF의 고유 문법 구조와 이를 바탕으로 한 정밀한 동역학 물리 속성 매핑 메커니즘을 심도 있게 분석한다.

1. SDF의 최상위 XML 계층 구조

SDF 스키마는 단일 개체 서술에 국한되지 않고 전체 시뮬레이션 세계(World)를 캡슐화하는 계층적 XML(eXtensible Markup Language) 구조를 갖는다. 최상단에는 파일의 포맷 선언인 <sdf> 태그가 위치하며, 그 하위에 독립적인 개체들을 담아낸다.

<?xml version="1.0" ?>
<sdf version="1.9">
  <!-- 시뮬레이션 환경 전체 (조명, 물리 엔진 설정, 중력 등) -->
  <world name="drone_testing_arena">
    
    <!-- 정적 장애물 1 -->
    <model name="building_1"> ... </model>

    <!-- 자율 비행 드론 로봇 모델 -->
    <model name="autonomous_quadcopter">
      <link name="base_link"> ... </link>
      <link name="rotor_1"> ... </link>
      <joint name="joint_rotor_1" type="revolute"> ... </joint>
      <!-- 센서 플러그인 등 탑재 -->
      <plugin filename="libgazebo_ros_imu.so" name="imu_plugin" />
    </model>
    
  </world>
</sdf>

가장 핵심적인 차이점은 URDF의 최상위 요소가 <robot>인 반면, SDF는 <world> 컨테이너 안에 다수의 <model> 요소(로봇, 장애물, 지형 등)를 병렬적으로 수용한다는 점이다. 이는 멀티 에이전트 강화학습(Multi-Agent RL)이나 스웜(Swarm) 비행 시뮬레이션에 필수적인 기초 골격을 제공한다.

2. URDF 패러다임 극복: 폐쇄 루프(Closed-loop) 기구학 지원

URDF는 수학적으로 엄격한 ‘방향성 비순환 그래프(DAG)’ 법칙을 따르기 때문에, 4절 링크(Four-bar linkage) 기구나 복잡한 병렬 로봇(Parallel Manipulator) 형태의 폐쇄 회로 구조를 원천적으로 표현할 수 없다.

반면, SDF는 <joint> 태그가 반드시 부모 프레임 구조의 종속성을 요구하지 않고 어떠한 두 개의 <link>라도 그래프 위상에 제약 없이 엮을 수 있도록 허용한다. 따라서 드론의 전개식 랜딩 기어(Retractable Landing Gear) 메커니즘 등 복수의 결합점이 서로 맞물려 순환하는 기계적 구속(Constraint)을 물리 엔진의 솔버(Solver) 상에 다이렉트로 매핑할 수 있는 기구학적 유연성을 지닌다.

3. 동역학 접촉(Contact) 및 마찰(Friction) 물리 속성의 고도화된 매핑

SDF 문법은 외부 환경과의 접촉 동역학(Contact Dynamics)을 표현하기 위한 정밀한 태그 속성 파라미터 계층을 <collision> 블록 하위에 제공한다.

3.1 표면 마찰력(Surface Friction) 매핑

ODE(Open Dynamics Engine) 물리 엔진 기준으로 마찰력은 방향성을 지닌다. <surface> 태그 내 <friction> 요소를 통해 구속된다.

  • mu와 mu2: 로봇의 휠(Wheel)이나 드론의 스키드(Skid)가 지면에 닿았을 때 작용하는 쿨롱 마찰 계수(Coulomb friction coefficient)이다. 주행 방향(mu)과 측면 미끄러짐 방향(mu2)을 별도로 지정하여 이방성(Anisotropic) 마찰을 모사할 수 있다.

3.2 접촉 면의 반발과 연성(Bounciness & Softness)

탄성 충돌 및 비탄성 충돌 거동을 정의하여 물체 간의 에너지 손실률을 시뮬레이션한다.

  • <restitution_coefficient>: 반발 계수(e). 값 1.0은 완전 탄성 충돌(에너지 보존)을, 0.0은 완전 비탄성 충돌(에너지 흡수)을 의미한다. 추락하는 드론 하우징의 파손 모사 또는 그물망과의 충돌 시험에 적용된다.
  • kp와 kd: 접촉 강성도(Stiffness)와 감쇠도(Damping). 두 물체가 부딪힐 때 가상의 스프링-댐퍼 시스템으로 상호 침투(Penetration) 깊이를 제어한다. 매우 단단한 금속 강체라면 kp 범위를 10^7 이상으로 할당하여 물리적 튕겨냄을 극대화한다.

4. 센서 플러그인의 네이티브 탑재

URDF가 별도의 <gazebo> 익스텐션을 빌려 억지로 센서 환경을 플러그인해야 했던 구조적 모순을 탈피하여, SDF는 로봇의 특정 <link> 내부에 센서 태그 <sensor>를 네이티브 계층으로 품는다.

  • type="camera", type="ray", type="imu" 등 물리 엔진 렌더러와 직접 융합되는 센서 종류를 지정한다.
  • 센서의 업데이트 주기(<update_rate>), 시야각(<fov>), 신호에 끼어드는 가우시안 노이즈(Gaussian Noise) 프로파일을 태그 기반으로 상세하게 기입하여, 실제 드론 센서 체계의 하드웨어적 한계(Sensor Imperfection)를 가상 공간에 완벽하게 복제한다.

결과적으로 시뮬레이션을 전제로 한 차세대 HIL(Hardware-In-the-Loop) 테스팅 아키텍처에서는, URDF로 초기 로봇 형상을 구축한 후 ROS 도구에 의해 SDF로 변환(Compile)하거나, 처음부터 SDF만으로 로봇을 역설계(Reverse Engineering)하여 모델의 무결성과 시뮬레이션 정합성을 극대화하는 패턴이 주류를 이루고 있다.
