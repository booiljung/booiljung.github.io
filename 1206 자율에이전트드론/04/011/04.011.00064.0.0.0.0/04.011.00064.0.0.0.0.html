Chapter 64. Gazebo (Ignition/Harmonic 버전) 플랫폼 기본 아키텍처 개념

1. 분산 아키텍처로의 전환 배경

과거 클래식 Gazebo(이전 명칭: Gazebo 11 이하 버전)는 단일 모놀리식(Monolithic) 소프트웨어 아키텍처를 채택하여 물리 엔진 연산, 센서 측정 시뮬레이션, 그래픽 렌더링, 사용자 인터페이스(GUI)가 강하게 내부 결합되어 운용되었다. 이로 인해 메모리 가용성 한계, 특정 플러그인 충돌 시 전체 시스템의 결함 전이, 대규모 다중 에이전트(Multi-Agent) 환경 구성 시 수직 확장성(Vertical Scalability)의 제한이 지속적으로 관찰되었다. 이러한 구조적 문제를 근본적으로 타개하기 위해, ROS 생태계를 유지 보수하는 Open Source Robotics Foundation(OSRF) 등은 Ignition 프로젝트를 개시하여 철저한 모듈 기반 분산 아키텍처를 갖춘 신규 계열의 Gazebo(Harmonic 버전 포함) 플랫폼을 설계하였다. 이 분산 구조는 개발자가 필요한 특수 목적 라이브러리 집합과 통신 채널만을 선별하여 경량화된 개별 인스턴스로 구동할 수 있도록 하며, 고밀도 연산이 요구되는 자율 에이전트 훈련 모델을 다수의 프로세스 노드 상에서 효과적으로 분해(Decoupling) 및 적재할 수 있도록 지원한다.

2. Gazebo 라이브러리 생태계(Gz Ecosystem)의 해체 및 재구성

신규 Gazebo 플랫폼 아키텍처의 근간은 단일 거대 패키지가 아닌, 고유한 연산 목적을 갖는 독립적인 C++ 라이브러리 컬렉션들의 집합 패턴으로 분할되었다는 점에 있다. 이를 통상적으로 Gazebo 생태계(Gz Ecosystem)라 명명하며, 각 모듈 계층의 명칭에는 설계상 부여된 기능적 역할이 명확히 내포된다.

  • gz-math: 3차원 공간상의 좌표 해석, 다차원 행렬 연산, 쿼터니언(Quaternion) 대수, 리지드바디 운동 모델링을 위한 관성 텐서(Inertia tensor) 등 로봇공학에 특화된 수리 연산 인터페이스를 제공하는 기반 수학 라이브러리 계층이다.
  • gz-physics: 다중 물리 백엔드의 동역학 연산을 규격화하는 추상화 플러그인 인터페이스 계층이다. DART(Dynamic Animation and Robotics Toolkit), Bullet, TPE(Trivial Physics Engine) 등의 다양한 외부 백엔드 동역학 솔버(Solver)와 연동할 수 있는 단일화된 연산 호출 구조(API)를 정의한다.
  • gz-rendering: Ogre, Ogre2, OptiX 등 이종 그래픽 렌더링 백엔드 파이프라인을 포괄하는 추상화 계층이다. 머신비전용 카메라, 라이다(LiDAR)와 같은 탑재 센서 시스템이 환경 내 광학, 기하 데이터를 획득하기 위한 광선 추적(Ray tracing) 기반 모델링의 기저 프레임워크로 작용하게 된다.
  • gz-transport: 다수의 분산 Gazebo 노드 간 메시지 송수신 및 동기식 서비스 호출을 전담하는 경량 통신 미들웨어 계층이다. ZeroMQ 기반의 Publisher-Subscriber, Request-Response 패턴을 라이브러리 내부에 탑재하여 프로세스 간 통신(IPC) 및 분산 네트워크 통신 계층의 복잡도를 은닉한다.
  • gz-gui: Qt 인프라의 위젯 기반 그래픽 사용자 인터페이스(GUI) 프레임워크 계층으로, 구동 중인 시뮬레이션 내에 모니터링 컴포넌트를 동적 로드 형태로 플러그인 및 해제할 수 있는 모듈식 레이아웃 런타임 환경을 보장한다.

3. ECS (Entity-Component-System) 소프트웨어 아키텍처 전면 도입

개편된 Gazebo 플랫폼은 아키텍처의 기능적 상호 의존성을 극도로 억제하고 물리 및 센서 컴포넌트의 확장성을 극대화하기 위하여, 데이터 지향 설계(Data-Oriented Design)에 속하는 Entity-Component-System (ECS) 아키텍처를 시뮬레이터 연산 코어에 전면 도입하였다. 이 ECS 아키텍처 체계 내에서 시뮬레이션 프레임워크 내의 모든 객체 속성과 산출 로직은 다음과 같은 엄격한 설계 규칙에 종속된다.

3.1 Entity (실체)

Entity는 시뮬레이션 가상 공간상에 위치하는 객체들에 할당되는 64비트 정수 형태의 고유 식별자(ID)이다. 기체의 주요 프레임 모델, 부속 링크, 회전 조인트, 내장 센서 모듈, 심지어 월드 환경 단위조차도 독립적인 Entity로 취급된다. Entity 식별자 자체는 아무런 물리 속성 데이터나 구동 로직 인스턴스를 소유하지 않으며, 오로지 데이터 바인딩을 위한 논리적 참조점으로 작용한다.

3.2 Component (구성 요소)

Component는 객체의 물리적, 논리적 상태를 묘사하는 순수 데이터(Plain Old Data, POD) 모음에 해당한다. 질량 상수(Mass Component), 관성 텐서(Inertial Component), 실시간 기하 포즈(Pose Component) 등 객체의 제반 속성은 독립적인 Component들이 Entity 식별자에 키-밸류(Key-Value) 형태로 배열됨으로써 실질적 구성을 형성한다. 이러한 객체 지향 모델의 다형성 배제 방식은 메모리 상의 연속적 배열 저장을 유도하여 캐시 지역성(Cache Locality)을 획기적으로 향상시킴으로써 수천 기의 에이전트 동역학 처리에 요구되는 연산 대역폭을 보증한다.

3.3 System (시스템)

System은 명시적 로직만을 포함하고 연산을 수행하는 런타임 제어 주체이다. System은 자체적인 데이터 스토리지를 갖추지 않으며, 오직 대상 역할을 하는 특정 Component 묶음의 데이터를 읽은 후 다음 시뮬레이션 타임스텝의 상태로 강제 갱신(Update)하는 순수 함수 단위로 설계된다. 예컨대 Physics System 모듈은 시뮬레이션 파이프라인 주기마다 각 Entity의 Pose 및 Inertial Component 배열 데이터를 동시 참조하여 외부 힘 벡터를 뉴턴-오일러 방정식에 접목하고 다음 스텝의 가속도를 적분하여 Pose Component를 갱신하게 된다. 센서 데이터 스위프(Sweep) 과정 역시 Sensor System이 정해진 샘플링 주파수에 따라 관련된 Entity 데이터 집합을 갱신 및 발행하는 병렬 함수 구조로 구현된다.
