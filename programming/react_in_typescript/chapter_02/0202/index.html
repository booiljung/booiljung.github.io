Number 타입¶

TypeScript에서 Number 타입은 ECMAScript 표준과 동일하게 배정밀도 64비트 부동소수점 형식으로 표현된다. 정수와 실수 구분 없이 모든 숫자가 같은 형식으로 취급되며 특별한 제한 없이 연산이 가능하다. 명시적인 타입 선언이 필요한 경우에는 코드에서 number라는 키워드를 사용한다.

자바스크립트와 동일하게 양의 무한대, 음의 무한대, NaN(Not a Number)을 포함하며, 컴파일 단계에서는 숫자 범위를 제한하지 않는다. Java나 C++ 등의 언어처럼 정수형, 실수형이 구분되지 않으므로, 별도의 캐스팅 없이 동일하게 산술 연산을 수행할 수 있다.

정수 형태의 값을 다루더라도 타입 시스템상으로는 모두 부동소수점이므로, 0.1과 0.2를 더할 때와 같이 부동소수점 정밀도 문제에 주의해야 한다. 고정 소수점 정밀도가 필요한 상황이라면 BigInt나 다른 라이브러리를 활용해야 한다.

코드에서 Number 타입을 선언할 때는 다음과 같은 방식으로 사용된다.

let basicNumber: number = 42
let decimalNumber: number = 3.14
let binaryNumber: number = 0b1010
let octalNumber: number = 0o52
let hexNumber: number = 0x2A
let notANumber: number = NaN
let infinityNumber: number = Infinity

타입 추론이 가능하기 때문에 명시적 선언 없이 바로 값만 할당해도 Number 타입이 적용된다. 다만 팀 내부에서 코드의 가독성이나 유지보수를 위해 명시적으로 타입을 선언하는 경우도 많다.

나누기 연산에서 0으로 나누는 경우에도 자바스크립트와 동일한 결과가 나오므로, 예외 처리와 같은 별도의 안전장치를 마련하고 싶다면 런타임 로직으로 구현해야 한다.

엄밀성을 유지하기 위해 특정 상황에서 BigInt 타입을 고려하는 것도 중요하다. 예를 들어 매우 큰 정수를 다루거나, 정확도가 떨어지는 부동소수점 연산을 피해야 할 때 BigInt를 사용한다. 그러나 BigInt는 Number 타입이 아니므로, 혼합 연산에 제약이 따른다.

TypeScript의 Number 타입은 유연성이 높지만, 부동소수점 오차나 0으로 나눈 결과에 대한 유의점이 존재한다. 따라서 프로젝트 요구사항과 숫자 처리 방식에 따라 적절한 검증 로직을 병행하는 것이 바람직하다.

String 타입¶

TypeScript에서 String 타입은 문자 시퀀스 전체를 의미하며, 자바스크립트의 문자열과 동일하게 다룬다. 작은따옴표, 큰따옴표, 백틱(Template Literal) 모두 문자열 선언에 사용할 수 있으며, 어떤 방식을 사용해도 타입은 string으로 정의된다.

문자열은 불변이므로, 기존 문자열에 다른 문자열을 연결할 때 새로운 문자열 객체가 생성된다. 기존 문자열을 직접 수정하는 것이 아닌, 매번 새 객체를 만드는 방식을 고려하여 성능 이슈를 점검해야 한다.

Template Literal을 사용하면 문자열과 표현식을 손쉽게 결합할 수 있으므로, 동적 문자열을 생성할 때 가독성을 높일 수 있다. 다음 예시는 명시적 string 타입 선언과 템플릿 리터럴을 함께 보여준다.

let greeting: string = "Hello"
let target = "World"
let combined: string = `${greeting},${target}!`

문자열 메서드는 자바스크립트에서 제공하는 다양한 표준 API를 그대로 활용한다. 문자열 길이 확인부터 부분 문자열 추출, 대소문자 변경, 문자열 검색 등 모든 작업이 동일하다. TypeScript 컴파일러는 타입 체크만 수행할 뿐, 런타임 동작은 자바스크립트와 완전히 동일하게 이뤄진다.

문자열이 비었는지 확인하거나, 특정 문자열로 시작하는지 검사하는 상황에서도 자바스크립트 메서드를 그대로 사용한다. TypeScript는 타입 안정성을 위해 문자열 연산의 결과가 언제나 string 타입으로 추론된다고 간주한다.

엄밀한 검증이 필요한 영역에서 string 타입이 너무 포괄적이라면, Literal Type이나 Enum, 또는 Union Type을 사용하여 가능한 문자열 값을 제한하는 방법을 활용할 수 있다.

Boolean 타입¶

TypeScript의 Boolean 타입은 참과 거짓을 표현하며, 자바스크립트의 true와 false를 그대로 지원한다. 명시적으로 Boolean 타입을 설정하거나, 자바스크립트 런타임에서 나온 불리언 값을 할당할 수 있다. 다음 예시에서는 Boolean 타입을 명시적으로 선언한다.

let isDone: boolean = false
let isReady: boolean = true

자바스크립트의 Truthy, Falsy 개념(문자열이 비어 있으면 false로 간주, 0이면 false 등)은 런타임에서 표현되는 값의 평가 결과이며, TypeScript는 타입 계층에서 Boolean을 별도로 구분한다. 따라서 string 타입이나 number 타입 변수를 조건문에서 Truthy/Falsy로 사용해도 문법상 오류가 발생하지 않는다. 그러나 boolean 타입으로 명시하고 싶은 상황이라면 확실히 타입 선언을 통해 의도를 표현해야 한다.

TypeScript는 런타임 동작을 제한하지 않으므로, if ("") { ... }와 같은 코드는 여전히 컴파일 오류 없이 동작한다. 이는 자바스크립트와 동일한 동작을 유지하기 위해서다. 필요하다면 strictNullChecks나 strictBooleanExpressions 같은 컴파일 옵션을 사용하여 더 엄격한 검사 규칙을 적용할 수 있다.

Null과 Undefined¶

자바스크립트에서 null과 undefined는 실제로 서로 다른 값이지만, 많은 상황에서 유사하게 동작한다. TypeScript는 이 둘을 별도의 타입으로 정의하며, null 타입과 undefined 타입이 존재한다. 기본적으로는 strictNullChecks 컴파일 옵션에 따라 이들의 취급 방식이 달라진다.

null은 의도적으로 값이 없음을 표현하기 위한 값이며, undefined는 값이 아직 정의되지 않았음을 표현한다. 두 값은 자바스크립트 엔진에 의해 내부적으로도 구분 처리되지만, strictNullChecks 옵션을 끈 상태에서는 null이나 undefined를 다른 모든 타입에 할당할 수 있다.

strictNullChecks를 켠 상태에서는 number 타입 변수에 null이나 undefined를 대입할 수 없으므로, 만약 이러한 값이 가능하다면 number | null | undefined 형태의 유니언 타입으로 정의해야 한다. 예를 들어 다음과 같은 코드는 strictNullChecks 옵션이 켜져 있을 때는 오류가 발생한다.

let possibleNull: number = null

따라서 다음처럼 유니언 타입을 선언해야 한다.

let possibleNull: number | null = null

null이나 undefined를 체크하는 과정에서 Non-null Assertion Operator(!)를 사용할 수도 있다. 그러나 이는 컴파일러에게 해당 변수가 null 또는 undefined가 아님을 보증하는 것일 뿐, 실제 런타임에서 값이 없을 경우 오류가 발생할 수 있으므로 주의가 필요하다.

TypeScript에서 null과 undefined는 자바스크립트의 동작을 그대로 따르되, 컴파일 단계에서 개발자의 의도를 명확히 나타낼 수 있도록 별도의 타입 시스템을 두어 더욱 엄밀한 검증을 수행할 수 있게 한다.

Null과 Undefined의 활용 시 주의점¶

코드에서 null이나 undefined가 발생할 수 있는 모든 경로를 인지하고, 적절히 방어 코드를 작성해야 한다. 런타임 환경에서는 null과 undefined가 의도치 않은 오류를 야기하기 쉽다. 예를 들어, 어떤 함수에서 숫자를 반환할 것으로 기대했는데 null이 반환되는 경우, 산술 연산 시 오류가 발생할 수 있다.

TypeScript는 이러한 문제를 사전에 감지하도록 돕지만, 너무 많이 허용하거나(옵션을 끈 상태) 너무 엄격하게 제한하면(옵션을 켠 상태) 개발 편의성과 안전성 사이에서 균형을 잡기 어렵다. 팀의 코드 스타일과 프로젝트의 규모, 오류 발생 가능성을 고려해 strictNullChecks를 신중하게 설정해야 한다.

Null 병합 연산자(??)를 활용하면, null이나 undefined가 될 수 있는 변수를 쉽게 처리할 수 있다. 예를 들어 a ?? b는 a가 null이나 undefined가 아니라면 a를, 그렇지 않다면 b를 반환한다. 선택적 체이닝(Optional Chaining) ?. 또한 객체의 속성 접근 과정에서 null이나 undefined가 나올 수 있는 시점에 안전하게 단계를 건너뛰도록 해준다.

두 연산자 모두 TypeScript에서 지원되며, 런타임 동작은 자바스크립트와 동일하지만, 컴파일 시점에 에러를 방지하기 위해 타입 선언이 잘 되어 있어야 한다.

타입 계층 구조 예시¶

간단히 살펴보면 최상위 타입인 any와 unknown이 있고, 그 아래로 수많은 내장 타입이 존재한다. number, string, boolean, null, undefined 모두 별도의 타입으로 존재하지만, 상황에 따라 유니언 타입으로 묶이거나, strictNullChecks 옵션의 설정에 따라 서로 호환성이 달라진다.

아래 다이어그램은 아주 단순화된 형태로 몇 가지 주요 타입만을 표현한다. 실제 TypeScript의 타입 계층은 훨씬 복잡하지만, Number, String, Boolean, Null, Undefined가 어디에 위치하는지 시각적으로 참고할 수 있다.

flowchart LR Any(( any )) --> NumberType(( number )) Any(( any )) --> StringType(( string )) Any(( any )) --> BooleanType(( boolean )) Any(( any )) --> NullType(( null )) Any(( any )) --> UndefinedType(( undefined ))

TypeScript에서 Number, String, Boolean, Null, Undefined는 자바스크립트와 동일한 형태로 동작하되, 컴파일 단계에서 더욱 엄밀한 타입 검사를 수행한다. Number는 정수와 실수를 구분하지 않으므로 부동소수점 오차를 주의해야 하며, 매우 큰 정수를 다룰 때는 BigInt를 고려하는 것이 좋다. String은 문자열을 불변 객체로 취급하므로 문자열 연산 시 새로운 객체가 생성되며, Template Literal을 사용하면 동적 문자열 생성이 편리하다. Boolean은 true와 false 두 가지만 존재하나, 자바스크립트의 Truthy/Falsy 개념과 혼동하지 않도록 타입을 명시적으로 선언하는 편이 안전하다. Null과 Undefined는 런타임에서 자주 발생하는 값이지만, TypeScript에서는 strictNullChecks 설정에 따라 각 타입을 별도로 인식하고 엄격하게 검사할 수 있다.

React 18과 TypeScript를 함께 사용할 때도 이들 기본 타입에 대한 이해가 필수적이다. 컴포넌트에서 다양한 값과 상태를 다루는 과정에서, 타입 시스템을 통해 오류를 미리 방지하고 의도를 명확히 표현할 수 있기 때문이다. 엄격한 설정이 불편하다면 설정 단계를 낮추더라도 프로젝트 규모와 요구사항에 따라 조정할 수 있다. 코드를 더욱 견고하게 유지하려면 Null 병합 연산자와 선택적 체이닝을 적극적으로 활용하여 런타임 오류를 줄이는 방안을 고려해볼 수 있다.

TypeScript의 기본 타입은 자바스크립트와 긴밀하게 연동되며, 작은 규모의 프로젝트부터 대형 프로젝트에 이르기까지 안정성 확보에 도움을 주는 핵심적인 기능이다. Number, String, Boolean, Null, Undefined 각각의 특징과 주의점을 인지하고 사용한다면, React 18 프로젝트에서도 타입 안전성과 개발 생산성을 더욱 높일 수 있을 것이다.
