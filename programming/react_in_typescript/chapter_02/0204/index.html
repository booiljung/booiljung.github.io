기본 개념과 공통점¶

TypeScript에서 인터페이스와 타입 별칭은 모두 특정 객체 형태를 정의하거나, 함수 시그니처나 클래스 구조 등을 명세하는 데 사용된다. 자바스크립트 언어에는 존재하지 않는 문법적 개념이지만, 타이핑을 통해 코드의 안정성과 가독성을 향상시키기 위해 등장한 TypeScript만의 기능이다.

인터페이스와 타입 별칭 모두 객체의 구조가 어떻게 생겼는지 기술하거나, 함수나 클래스, 혹은 어떤 다양한 자료형을 추상화해 놓을 수 있는 점이 공통점이다. 예를 들어 객체가 name과 age라는 필드를 갖고 있다는 사실을 인터페이스로 선언할 수도 있고 타입 별칭으로 선언할 수도 있다. 코드 레벨에서 둘은 매우 유사한 형태를 가진다.

인터페이스의 특징과 확장성¶

인터페이스는 TypeScript의 대표적인 타입 설계 도구 중 하나이다. 자바나 C#과 유사하게 특정 규격을 정의하고 그 규격을 만족하도록 클래스를 구현할 수도 있다. 하지만 TypeScript의 인터페이스는 단순히 클래스와의 결합만 의미하지 않는다. 객체 리터럴 타입, 함수 타입 등의 형태로도 다양하게 활용된다.

TypeScript 인터페이스는 선언 병합(Declaration Merging)이 가능하다는 점이 특징이다. 동일한 이름의 인터페이스를 여러 곳에서 선언하면, 컴파일 시점에 모든 선언이 합쳐져서 하나의 인터페이스로 취급된다. 라이브러리의 타입 정의를 확장해야 하는 경우 선언 병합을 적극 활용할 수 있다. 예를 들어, 외부 라이브러리에서 제공하는 인터페이스에 필요한 속성을 추가하고 싶다면, 선언 병합이 매우 유용하게 작용한다.

인터페이스는 extends 키워드를 통해 다른 인터페이스를 확장할 수 있으며, 동시에 여러 개의 인터페이스를 extends 하는 것도 허용한다. 클래스가 여러 인터페이스를 구현하는 방식과 유사하지만, 여기서는 순수한 타입 레벨에서의 확장만 가능하다는 점에 주의해야 한다.

인터페이스를 활용하면 협업 환경에서 코드의 규칙을 명확히 공유하기 용이하다. 예를 들어 React 컴포넌트의 props를 기술할 때 props 인터페이스를 만들어두면, 다른 개발자가 해당 컴포넌트를 사용할 때 어떤 속성을 입력해야 하는지 코드 에디터의 자동 완성 기능을 통해 바로 알 수 있다.

타입 별칭의 특징과 유연성¶

타입 별칭은 type 키워드를 사용해 새 이름을 정의하고, 그 뒤에 대입되는 실제 타입이 무엇이든 해당 이름으로 대표하는 기능을 수행한다. 기본적으로 객체 리터럴 타입을 정의할 수도 있고, 유니언 타입이나 인터섹션 타입, 혹은 기타 다른 원시 타입 같은 것을 간단히 별칭으로 지정할 수도 있다.

타입 별칭은 어떠한 타입 구문에도 활용할 수 있기 때문에, 매우 유연하다는 장점을 갖는다. 예를 들어 어떤 함수 타입을 직접 인라인으로 작성하면 가독성이 떨어질 수 있는데, 이를 별칭으로 뽑아서 재사용하면 코드의 깔끔함을 유지하기 쉽다. 또한 제네릭 타입을 설계해 놓고 해당 제네릭에 타입 별칭을 여러 번 부여해 다양하게 활용하는 것도 가능하다.

다만 타입 별칭은 선언 병합이 불가능하다. 동일한 이름을 여러 번 선언하면 컴파일 오류가 발생한다. 이 때문에 특정 라이브러리 타입을 확장하거나, 동일 이름으로 여러 번 선언해 점진적으로 구체화해야 하는 경우에는 적절하지 않을 수 있다.

타입 별칭은 기본적으로 new 키워드가 허용되지 않고, 타입 체커 관점에서 정적인 타입 정보만 담는다. 클래스 구현과 관련된 확장성보다는, 유니언 타입이나 매핑된 타입 등 고급 타입 기법에서 더 큰 유연성을 발휘하기 위해 사용하는 경우가 많다.

인터페이스와 타입 별칭의 교차점과 차이점¶

인터페이스나 타입 별칭 모두 객체 구조를 정의하는 등 공통적으로 사용될 수 있는 영역이 있다. 예를 들어 React 컴포넌트의 props를 정의할 때 크게 차이가 없어 보인다. 그러나 몇 가지 결정적인 차이가 존재한다.

인터페이스는 선언 병합이 허용되므로, 라이브러리나 기존 프로젝트에서 이미 사용하고 있는 타입을 확장하기에 용이하다. 반면 타입 별칭은 확장보다는 다양한 타입 구성(유니언, 인터섹션, 조건부 타입, 매핑된 타입 등)에 더 큰 이점을 가진다. 예를 들어 유니언 타입을 다루거나, 복잡한 타입 구문을 재사용하기 위해서는 타입 별칭이 필수적이다.

타입 별칭은 어떤 타입이든 이름을 붙일 수 있다. 객체 형태가 아니더라도, 예를 들어 string | number와 같은 유니언 타입도, Pick와 같은 매핑된 타입도, 혹은 ReturnType과 같은 도구적(utility) 타입도 모두 별칭으로 정의할 수 있다. 반면 인터페이스는 오직 객체 형태와 클래스 구현을 전제로 하는 구조적 설계에 특화되어 있다.

인터페이스는 extends를 통해 상속에 유연하고, 타입 별칭은 & 기호를 활용한 인터섹션으로 다양한 형태를 결합할 수 있다. 그러나 extends는 병합을 통한 새로운 인터페이스를 정의하는 식으로 안정적으로 확장이 가능하고, 타입 별칭의 &는 좀 더 범용적인 방법으로 다양한 타입을 융합시킬 수 있다. 이 둘의 확장 방식이 미묘하게 다른 동작을 할 수 있으므로, 어떤 구조가 필요한지에 따라 적절히 선택해야 한다.

명시적 선언과 선언 병합 활용의 예시¶

외부 라이브러리를 사용할 때, 종종 기본 인터페이스에 추가 기능을 주입하고 싶을 때가 있다. 예를 들어 Express.js에서 Request 객체에 새로운 사용자 정보를 담는 기능을 적용할 수 있다. 이때 Request 인터페이스를 선언 병합으로 확장하면 별도의 재정의 과정 없이 기존 Request 타입에 내가 원하는 속성이 자연스럽게 추가된다. 그러나 타입 별칭은 이런 식의 선언 병합을 지원하지 않으므로, 새로운 이름으로 재정의해야만 한다. 이 경우 기존 레퍼런스 코드나 라이브러리 외부 선언이 타입 별칭을 사용하고 있다면, 수정을 위한 유연성이 인터페이스에 비해 부족해진다.

복잡한 타입 구성에서의 타입 별칭 사용¶

유니언 타입이나 조건부 타입, 매핑된 타입 등의 복잡한 구문이 필요한 경우에는 타입 별칭을 사용하는 편이 자연스럽다. 예를 들어 React에서 여러 형태의 이벤트 객체를 하나의 props 필드로 처리하고 싶다면, type 키워드로 MouseEvent | KeyboardEvent와 같은 유니언을 정의한 뒤 이를 그대로 props에 대입하는 식이 더 직관적이다. 또한 타입 별칭을 통해 중첩된 제네릭 형태를 설계해 놓고, 이를 재사용하거나 다른 타입 구문에 혼합할 수도 있다. 이런 복잡한 타입 구문은 인터페이스로는 전부 표현하기 어렵거나, 표현이 가능하더라도 코드의 가독성이 떨어질 수 있다.

React 환경에서의 실무적 판단¶

React 컴포넌트의 props를 정의할 때 단순한 객체 구조라면 인터페이스와 타입 별칭 모두 크게 차이가 없다. 협업을 고려해 props 구조가 명확해야 하고, 프로젝트 전체에서 해당 props 규칙을 많이 재사용하거나 확장할 여지가 있다면 인터페이스를 쓰는 것이 좀 더 안전하다. 선언 병합을 통해 글로벌하게 사용되는 타입들을 점진적으로 확장하기도 쉽다. 반면 복잡한 타입 조합이 필요한 경우, 예를 들어 여러 형태의 이벤트 객체나 유니언 타입을 사용해야 한다면 타입 별칭이 더 적합하다. 이벤트 객체가 서로 다른 형태를 가진 경우에 유니언 타입을 쓰면 훨씬 강력한 타입 체킹이 가능하다. 이렇게 컴포넌트가 인자로 받을 수 있는 값이 다변적이라면 타입 별칭이 가독성과 유지보수에 유리하다.

구현 관점에서의 한계와 디자인 선택¶

인터페이스와 타입 별칭 모두 자바스크립트 런타임 레벨에서는 사라지는 개념이므로, 실제 실행 중에 인터페이스나 타입 별칭이 동작한다거나 하는 일은 없다. 컴파일 타임에만 의미가 있으며, 이는 React 컴포넌트에서도 동일하다. 인터페이스가 좋고 타입 별칭이 나쁘거나, 혹은 그 반대가 아니라, 상황에 따라 더 적절한 도구가 있을 뿐이다. 상속 구조가 자연스럽게 이어지고 선언 병합으로 확장 가능해야 하는 경우에는 인터페이스가 훨씬 편리하다. 복잡한 타입 연산과 유니언, 조건부 타입, 인덱스 시그니처, 제네릭 등을 종합해서 설계해야 한다면 타입 별칭이 더 손쉽다.

인터페이스와 타입 별칭의 관계 예시 (mermaid)¶

아래 예시는 인터페이스와 타입 별칭을 동시에 사용하며, 서로 다른 확장 방식을 보여준다.

flowchart TB A[interface Animal] --> B[interface Dog] B --> C[type AdditionalDogInfo] C --> D[type DogWithBreedInfo] A --> E[interface Cat]

Animal 인터페이스를 상속받는 Dog 인터페이스가 있고, 별도의 타입 별칭으로 추가 정보를 지정해 Dog 타입에 & 기호를 사용하여 확장하고 있다. Cat 역시 Animal을 상속받지만, 별도의 타입 별칭 확장은 하지 않는다.

객체 구조 예시¶

간단한 사용자 정보를 표현하려 할 때, 동일한 구조를 인터페이스와 타입 별칭으로 각각 정의하면 아래와 같은 코드를 작성할 수 있다.

interface Person {
  name: string
  age: number
}

type PersonAlias = {
  name: string
  age: number
}

function greet(person: Person) {
  return `Hello, ${person.name}`
}

function greetAlias(person: PersonAlias) {
  return `Hello, ${person.age}살의${person.name}`
}

두 선언 모두 객체 구조를 정의한다는 점에서 기능이 비슷하다. Person과 PersonAlias는 모두 name과 age를 필수 속성으로 갖는 객체로서 동일하게 동작한다.

클래스 구현 예시¶

인터페이스는 클래스에서 implements 키워드를 사용해 구현할 수 있다. 타입 별칭은 클래스 구현과 직접적인 연결을 제공하지는 않지만, 타입 검증 용도로는 동일하게 사용할 수 있다.

interface Animal {
  name: string
  sound(): void
}

class Dog implements Animal {
  name: string

  constructor(name: string) {
    this.name = name
  }

  sound() {
    console.log("Bow-wow!")
  }
}

// 타입 별칭으로 같은 구조를 표현
type AnimalAlias = {
  name: string
  sound(): void
}

// 별칭을 직접 implements 할 수는 없지만, 변수를 선언해 할당 가능
class Cat {
  name: string

  constructor(name: string) {
    this.name = name
  }

  sound() {
    console.log("Meow!")
  }
}

const cat: AnimalAlias = new Cat("Kitty")

AnimalAlias를 implements 키워드로 사용하는 것은 허용되지 않지만, 변수에 타입을 지정해 클래스 인스턴스가 해당 구조를 만족시키는지 검사할 수 있다.

확장(상속) 예시¶

인터페이스는 extends 키워드를 사용해 다른 인터페이스를 확장하고, 타입 별칭은 & 연산자(인터섹션)를 사용해 기존 타입을 조합한다.

interface BaseUser {
  id: number
  username: string
}

interface AdminUser extends BaseUser {
  isAdmin: boolean
}

type BaseUserAlias = {
  id: number
  username: string
}

type AdminUserAlias = BaseUserAlias & {
  isAdmin: boolean
}

const admin1: AdminUser = {
  id: 1,
  username: "admin",
  isAdmin: true
}

const admin2: AdminUserAlias = {
  id: 2,
  username: "root",
  isAdmin: true
}

AdminUser는 BaseUser를 상속받아 isAdmin 속성을 추가하고, AdminUserAlias는 & 기호를 통해 BaseUserAlias와 새로운 속성을 결합한다.

함수 시그니처 예시¶

함수의 타입을 인터페이스나 타입 별칭으로 정의할 수도 있다.

interface Greeting {
  (name: string): string
}

type GreetingAlias = (name: string) => string

const greetByInterface: Greeting = (name) => `Hello, ${name}`
const greetByAlias: GreetingAlias = (name) => `Hi, ${name}`

인터페이스 Greeting은 호출 시그니처를 정의하며, 타입 별칭 GreetingAlias도 동일한 함수 형태를 명시한다.

선언 병합 예시¶

인터페이스는 선언 병합이 가능하다. 동일한 이름의 인터페이스가 여러 곳에서 선언되면, 컴파일 시점에 모두 하나로 합쳐진다. 타입 별칭은 이런 병합을 지원하지 않는다.

// 첫 번째 선언
interface Window {
  title: string
}

// 두 번째 선언
interface Window {
  version: number
}

// 병합 결과
// interface Window {
//   title: string
//   version: number
// }

이렇게 한 번에 정의할 수 없는 환경(예: 여러 타입 선언 파일)에서 Window 인터페이스를 확장할 수 있다. 반면 타입 별칭은 동일한 이름으로 여러 번 선언하면 오류가 발생한다.

type WindowAlias = {
  title: string
}

type WindowAlias = {
  version: number
} // 오류 발생

유니언과 인터섹션 예시¶

타입 별칭은 유니언 타입이나 조건부 타입 등의 복잡한 조합을 자주 다룰 때 유리하다.

type InputEvent = 
  | MouseEvent
  | KeyboardEvent
  | TouchEvent

function handleInputEvent(event: InputEvent) {
  if (event instanceof MouseEvent) {
    console.log("MouseEvent 처리")
  } else if (event instanceof KeyboardEvent) {
    console.log("KeyboardEvent 처리")
  } else {
    console.log("TouchEvent 처리")
  }
}

동일한 구조를 인터페이스로 표현하고 싶다면, 각각의 이벤트 유형에 대해 별도의 인터페이스와 식별 가능한 유니언 형태를 구성해야 한다. 하지만 타입 별칭을 사용하면 좀 더 간결하게 유니언으로 묶을 수 있다.

인터페이스와 타입 별칭 혼합 예시¶

아래 코드는 하나의 Animal 인터페이스를 두고, Dog를 인터페이스로 확장하면서 동시에 별도의 타입 별칭들을 & 연산자로 확장하는 예시이다.

interface Animal {
  name: string
}

interface Dog extends Animal {
  bark(): void
}

type AdditionalDogInfo = {
  breed: string
  age: number
}

type DogWithBreedInfo = Dog & AdditionalDogInfo

const myDog: DogWithBreedInfo = {
  name: "Buddy",
  bark() {
    console.log("Woof!")
  },
  breed: "Golden Retriever",
  age: 3
}

Dog 인터페이스는 Animal을 상속받아 bark 메서드를 추가한다. 이후 타입 별칭 AdditionalDogInfo에 & 연산자를 사용하여 Dog 타입을 더 확장한다.

React에서 TypeScript를 활용할 때, 인터페이스와 타입 별칭은 모두 중요한 역할을 한다. 객체의 구조를 선언하고, 팀 내에서 공유하며, 협업할 수 있는 명확한 계약을 제시하기에는 인터페이스가 직관적이다. 특히 선언 병합 기능을 통해 기존 타입을 확장해야 할 필요가 있는 경우 인터페이스가 필수적이다.

반면 타입 별칭은 더 폭넓은 타입 구문에 이름을 부여하고, 유니언 타입이나 조건부 타입, 인덱스 시그니처 등 다양한 복잡성을 간단히 처리하는 데 유리하다. 특정 상황에서 복잡한 타입 구조를 많이 다룬다면 타입 별칭을 통해 코드 구조를 깔끔하게 정리할 수 있다.

최종적으로는 프로젝트에서 어떤 방식으로 재사용과 확장을 요구하느냐, 다른 팀원이 어떻게 이해하고 유지보수할 수 있느냐 등을 종합적으로 고려해 선택해야 한다. 인터페이스와 타입 별칭 중 하나만을 고집할 필요는 없다. 타입 설계의 목적에 부합하도록 적절히 병행 사용하는 것이 TypeScript를 활용한 React 개발에서 가장 효율적인 전략이다.
