VSCode 확장 프로그램의 중요성¶

React 18과 TypeScript를 사용하여 개발할 때에는 프로젝트 구조와 코드 스타일을 유지하는 것이 매우 중요하다. 협업 환경에서는 더욱 엄격한 규칙과 스타일 가이드가 필요해지므로, 개개인의 취향과 무관하게 통일된 코드 포맷팅과 문법 규칙을 적용할 수 있는 자동화된 도구가 필수적이다. 이러한 기능을 VSCode 확장 프로그램을 통해 쉽게 구현할 수 있는데, 시Lint(ESLint)와 프리티어(Prettier)를 비롯한 다양한 확장을 적절히 조합하여 사용하면 작업 효율을 크게 높일 수 있다. 에디터 자체에서 오류를 실시간으로 확인하고, 저장 시점에 자동으로 포맷팅해주는 방식으로 코드를 관리하면, 코드 리뷰나 빌드 파이프라인에서 발생할 수 있는 불필요한 문제를 미리 차단할 수 있다.

ESLint 확장 프로그램¶

ESLint는 JavaScript 및 TypeScript 코드의 품질과 일관성을 유지하기 위한 린팅(Linting) 도구이다. 프로젝트 내부에 존재하는 잠재적 오류, 변수 사용 패턴의 문제점, 불필요한 import나 선언 등 다양한 부분을 자동으로 점검해주는 역할을 수행한다. 이는 TypeScript 컴파일러가 잡아내지 못하는 영역까지 커버할 수 있으므로, 코드 전반의 품질을 한층 더 끌어올려 준다. ESLint는 플러그인 구조로 이루어져 있어 React, TypeScript 등 특정 프레임워크나 언어에 최적화된 규칙 세트를 쉽게 추가할 수 있다.

React 개발 시에는 eslint-plugin-react, eslint-plugin-react-hooks, eslint-plugin-jsx-a11y 등의 공식 플러그인을 함께 사용하는 경우가 많다. TypeScript와 함께 사용할 때에는 @typescript-eslint/eslint-plugin, @typescript-eslint/parser를 반드시 설치해야 하며, ESLint 설정 파일에서 파서를 @typescript-eslint/parser로 지정하고, 플러그인 목록에 @typescript-eslint를 포함해야 한다. 이를 통해 ESLint가 JavaScript를 넘어 TypeScript 구문까지 올바르게 이해하고 린트를 수행할 수 있다.

VSCode는 ESLint 확장 프로그램을 제공한다. 이 확장을 설치하면, 저장 시점 혹은 실시간으로 문법 오류나 스타일 위반 사항을 편집기 내에 표시할 수 있게 된다. 예를 들어 변수 선언 후에 사용하지 않는 경우에 대한 경고, 사용이 권장되지 않는 API 호출, 올바르지 않은 React Hooks의 사용 방법 등을 코드 에디터 내에서 즉각적으로 확인할 수 있다. 프로젝트 규모가 커질수록 이러한 자동화된 점검 과정을 통해 생산성이 크게 향상된다.

ESLint 설정 파일(.eslintrc.js 혹은 .eslintrc.json 등)은 프로젝트에 대한 린트 규칙을 세부적으로 정의하는 곳이다. React와 TypeScript를 사용하는 상황이라면 다음과 같은 예시를 적용할 수 있다.

{
  "env": {
    "browser": true,
    "es2022": true
  },
  "parser": "@typescript-eslint/parser",
  "parserOptions": {
    "ecmaVersion": "latest",
    "sourceType": "module"
  },
  "plugins": [
    "react",
    "@typescript-eslint",
    "react-hooks"
  ],
  "extends": [
    "eslint:recommended",
    "plugin:react/recommended",
    "plugin:@typescript-eslint/recommended",
    "plugin:react-hooks/recommended"
  ],
  "rules": {
    "@typescript-eslint/no-unused-vars": "warn",
    "react/prop-types": "off"
  }
}

이 설정 예시는 브라우저 환경을 대상으로 React 18과 TypeScript 구문을 린트하는 것이 주요 목표이며, 자주 발생하는 오류 패턴을 사전에 방지한다. 프로젝트에서 요구되는 규칙에 따라 rules 속성을 추가 수정하거나, 특정 플러그인을 더 등록할 수도 있다.

Prettier 확장 프로그램¶

Prettier는 코드의 포매팅을 담당하는 도구로, 특정 스타일 가이드에 맞춰 코드를 자동으로 정렬하고 줄바꿈 등을 처리한다. 스타일 통일성을 유지하기 위해서는 매우 유용하며, ESLint 규칙과 충돌하지 않도록 설정하면 저장 시점마다 모든 코드가 특정 포맷으로 변환된다. 팀 단위의 협업에서 Prettier를 사용하면 코드 스타일 관련 논쟁을 줄일 수 있고, 코드 리뷰 시에는 구현 로직 그 자체에 집중할 수 있다.

Prettier는 .prettierrc 파일 혹은 package.json 등에 설정을 추가하여 다양한 포맷팅 옵션을 적용할 수 있다. 예를 들어 세미콜론 사용 여부, 작은따옴표와 큰따옴표 사용, 탭 너비, 줄바꿈 규칙 등의 세부 설정이 가능하다. React 18 프로젝트에서도 JSX, TSX 파일을 자동 정렬할 수 있도록 지원하므로, UI 컴포넌트 코드의 가독성이 향상된다.

Prettier 확장 프로그램은 VSCode에서 제공하는 포맷 기능과 연계되어 있다. 사용자 설정에서 "Format On Save" 옵션을 활성화하면 코드를 저장할 때마다 자동으로 Prettier 규칙에 맞춰 코드가 정리된다. ESLint와 동시에 사용할 경우에는 eslint-plugin-prettier나 eslint-config-prettier 등을 조합하여 린트 규칙과 포매팅 규칙이 충돌하지 않도록 설정할 수 있다. 즉, ESLint가 문법적 오류나 코드 품질에 집중하도록 하고, Prettier가 순수한 포매팅 역할만 수행하도록 분리하는 방식이다.

아래와 같은 예시 구성으로 Prettier 설정 파일을 작성할 수 있다.

{
  "semi": true,
  "singleQuote": false,
  "tabWidth": 2,
  "printWidth": 80,
  "trailingComma": "es5"
}

이와 같은 설정은 세미콜론 사용을 허용하고, 작은따옴표 대신 큰따옴표를 기본값으로, 탭 너비를 2로, 한 줄의 최대 길이를 80으로 설정하는 등 다양한 옵션을 적용한다.

ESLint와 Prettier의 결합¶

ESLint와 Prettier를 제대로 결합하기 위해서는 충돌을 최소화하는 설정이 필요하다. eslint-config-prettier 패키지를 사용하면 Prettier 관련 규칙이 ESLint 규칙보다 우선순위를 갖도록 구성할 수 있다. 또한 eslint-plugin-prettier 패키지를 추가하여 ESLint가 Prettier 포맷팅 검사까지 겸하도록 설정할 수도 있다.

프론트엔드 프로젝트에서 자주 사용되는 방식 중 하나는 ESLint가 문법 오류나 코드 품질을 주로 검사하도록 두고, Prettier가 포매팅을 전담하게 하는 전략이다. 이 경우에는 ESLint 설정 파일에서 Prettier가 관리해야 할 부분을 비활성화하고, Prettier 규칙이 우선 적용되도록 extends 항목에 'prettier'를 맨 마지막에 배치한다. 이렇게 하면 저장 시점이나 CI 파이프라인에서 린트 및 포매팅이 동시에 수행되고, 작업 효율이 높아진다.

VSCode 설정 및 단축키 연동¶

VSCode에서 ESLint와 Prettier를 설치한 뒤에는, 사용자 설정을 통하여 저장 시 자동 포매팅과 린트가 잘 동작하도록 구성해야 한다. settings.json 파일이나 VSCode 설정 GUI를 통해 "editor.formatOnSave": true, "editor.codeActionsOnSave": {"source.fixAll.eslint": true} 등을 활성화하면, 저장할 때마다 ESLint의 자동 수정 가능한 규칙이 먼저 적용되고, 그 후에 Prettier가 포매팅 작업을 수행하게 된다. 이를 통해 모든 파일이 일관된 스타일을 유지하면서도, 코드의 문제점을 실시간으로 확인할 수 있다.

React 18 기반 프로젝트를 진행한다면, 함수형 컴포넌트나 React Hooks 관련 린트 규칙도 실시간으로 체크해주는 환경이 매우 유용하다. 잘못된 useEffect 의존성 배열이나 useState 사용 등은 곧바로 에디터에 표시될 것이며, 이러한 문제를 빠르게 수정함으로써 유지보수성과 동작 안전성을 확보할 수 있다.

TypeScript 언어 지원 확장¶

VSCode는 기본적으로 TypeScript 언어 서비스를 내장하고 있다. 그러나 React 18 프로젝트에서 tsx 파일을 다루거나, 고급 TypeScript 기능을 활용하기 위해서는 내장된 언어 서비스와 함께 추가적인 기능을 제공하는 확장을 사용하는 편이 좋다. TypeScript 버전 관리, 자동 완성, 리팩토링 제안, 타입 선언 파일 업데이트 등이 잘 이뤄지려면 VSCode 환경을 최신 상태로 유지해야 한다.

TypeScript가 JS보다 엄격한 타입 검사를 제공한다는 점은 개발자를 오탈자나 API 오용으로부터 보호한다. 타입 정의 덕분에 에디터에서 자동 완성과 실시간 오류 탐지가 더욱 정확하게 동작한다. VSCode의 TypeScript 확장 프로그램이나 관련 플러그인은 이러한 언어 서비스를 더욱 강화하고, React의 JSX나 TSX 문법에도 완벽하게 대응하도록 도와준다.

추가 유용한 확장 프로그램¶

React 18 프로젝트뿐 아니라 전반적인 웹 프론트엔드 개발에 도움이 되는 확장 프로그램이 여러 가지 존재한다. 예컨대 GitLens는 Git 이력을 시각적으로 표시해 주어 협업 시 매우 편리하다. Path IntelliSense는 상대 경로나 모듈 경로를 자동으로 완성해 주어 import 구문을 편리하게 작성할 수 있다. 코드 스펠 체커(Code Spell Checker)는 문서화 주석이나 문자열 리터럴에서 오탈자를 찾아주는 기능을 제공하고, Auto Close Tag나 Auto Rename Tag 등은 JSX 태그와 HTML 태그를 편리하게 작성하도록 도와준다. Import Cost는 모듈을 import할 때 해당 모듈의 번들 크기를 표시해 주어 성능 최적화 관점에서 도움을 준다. 이러한 확장들은 프로젝트 전반에서 개발 편의성을 높이지만, 반드시 모든 확장을 설치해야 하는 것은 아니므로 팀의 문화나 프로젝트 특성에 맞춰 선택적으로 적용하면 된다.

정리와 권장 설정 파일 예시¶

React 18과 TypeScript를 사용하는 프로젝트에서 VSCode 확장 프로그램을 잘 활용하면, 코드 품질과 생산성을 동시에 높일 수 있다. ESLint는 코드 문법이나 React Hooks 사용 패턴의 오류를 미연에 방지하고, Prettier는 포매팅 규칙을 일관성 있게 적용한다. VSCode 설정 파일에서 ESLint와 Prettier를 함께 활용하도록 구성해 두면, 저장 시점에 모든 문제가 자동으로 해결되거나 표시되어 효율적인 개발 프로세스를 구축할 수 있다. 앞서 언급한 설정 예시와 함께, 프로젝트 규모나 팀의 스타일 가이드를 고려하여 세부 항목을 조정하면서 자신만의 이상적인 환경을 만들어 나가는 것이 중요하다.

아래는 .vscode/settings.json 파일에서 ESLint와 Prettier를 동시에 활용하고, 코드를 저장하자마자 자동 포매팅되도록 구성하는 예시이다.

{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "eslint.alwaysShowStatus": true,
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true
  }
}

이 설정은 "editor.defaultFormatter" 옵션으로 Prettier를 기본 포매터로 지정하고, "editor.codeActionsOnSave"를 통해 ESLint가 자동 수정 가능한 오류를 먼저 고치도록 만든다. 이후 Prettier가 코드 스타일을 일관되게 정리하여 최종적으로 통합된 포매팅 결과를 제공한다. 이러한 확장 프로그램 조합은 React 18과 TypeScript에 최적화된 개발 환경을 마련하는 데 핵심적인 요소이므로, 프로젝트에 맞춰 세부 사항을 계속 조정하고 모니터링하면서 최고의 개발 경험을 누릴 수 있도록 하길 권장한다.
