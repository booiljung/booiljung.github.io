CRA, Vite 등 프로젝트 생성 방식 개요¶

React 18과 TypeScript를 함께 사용하는 프로젝트를 시작할 때 가장 먼저 고민하게 되는 부분은 효율적이고 간편한 초기 세팅 방법이다. 전통적으로는 Webpack이나 Babel 설정을 직접 구성하는 방법이 있었지만, 프로젝트 설정과 구성 요소가 복잡해지면서 이러한 작업에 드는 시간이 상당히 커졌다. 이에 따라 Create React App(CRA)과 Vite 같은 도구들은 React 프로젝트를 쉽고 빠르게 시작하도록 돕는다. 이 도구들은 표준화된 디렉터리 구조, 초기 설정, 개발 서버, 빌드 스크립트 등을 제공하여 개발자가 핵심 비즈니스 로직에 집중할 수 있게 해준다.

CRA와 Vite 모두 React 18을 지원하며 TypeScript 템플릿을 제공하거나 TypeScript 설정을 간단히 추가할 수 있다. 이 장에서는 CRA와 Vite가 무엇인지, 어떤 특징을 가지는지, 그리고 React 18 + TypeScript 환경을 손쉽게 구축하기 위해 어떻게 활용하는지를 중점적으로 다룬다.

Create React App(CRA) 소개¶

CRA는 React 공식 문서에서 권장되는 프로젝트 부트스트래핑 툴 중 하나로, Webpack, Babel, ESLint 등 다양한 도구가 내장되어 있어 추가 설정 없이도 바로 개발을 시작할 수 있다. 내부적으로 Webpack 설정 파일이 존재하지만 기본적으로는 이를 직접 수정할 수 없도록 설계되었다. 그 대신 대부분의 React 애플리케이션에서 필요로 하는 로더, 플러그인, 환경 변수를 어느 정도 갖춘 상태로 제공되므로, 별도의 작업 없이도 표준적인 개발 환경을 구축하기 쉽다.

CRA에서 React 18 프로젝트를 생성할 때는 명령줄 도구를 활용한다. 예시로 TypeScript 템플릿을 사용하여 프로젝트를 생성하고 싶다면 다음과 같이 입력한다.

npx create-react-app my-app --template typescript

실행 후에는 디렉터리 구조가 자동으로 생성되고, package.json, tsconfig.json, App.tsx, index.tsx 등이 포함된 초기 상태가 생성된다. React 18의 새 API인 createRoot 함수를 사용하기 위해서는 index.tsx 파일에 react-dom/client 패키지의 createRoot를 임포트한 뒤, 기존의 ReactDOM.render 대신 createRoot를 사용하는 형태로 작성한다. 초기 템플릿에도 이미 ReactDOM.createRoot 사용 방식이 적용되어 있을 수 있으므로 반드시 해당 부분을 확인하고, 앱 전반에 걸쳐 React 18의 기능이 올바르게 동작하는지 점검한다.

CRA로 생성된 프로젝트는 npm start 명령을 통해 로컬 개발 서버를 구동하고, npm run build 명령으로 빌드 아웃풋을 생성한다. 빌드 과정은 Webpack을 통해 진행되며, babel-loader가 JS/TS 파일을 트랜스파일링하고, 필요한 CSS나 이미지 자원을 로드하는 설정도 포함되어 있다. 환경 변수는 REACT_APP_ 접두사를 붙여서 .env 파일 혹은 시스템 환경 변수로 정의할 수 있으며, 빌드 시점에 process.env.REACT_APP_XXX 형태로 접근 가능하다. 이는 보안 및 설정 관리를 단순화해주지만, 사용 가능한 환경 변수 이름이 제한된다는 단점이 있다.

CRA가 편리한 이유 중 하나는 대부분의 설정이 자동화되어 있고, 어느 정도 표준화된 방식으로 구축되어 있어 초기에 삽입되는 설정 오류를 최소화한다는 점이다. 반면에 커스텀 웹팩 설정이나 Babel 플러그인 추가와 같은 고급 설정을 하려면 react-app-rewired, craco 등을 별도로 도입해야 한다. 이러한 패키지를 활용하면 CRA의 내부 설정을 간접적으로 오버라이드할 수 있다. 그러나 이 과정에서 CRA의 업그레이드 호환성을 반드시 고려해야 하며, 경우에 따라 CRA를 사용하는 기본 취지(간소화된 설정)와 어긋날 수 있다는 점도 유의해야 한다.

CRA는 프로젝트 시작 시점에 React, TypeScript, Webpack, Babel, ESLint 등을 모두 세팅해주며, 별도의 설정 없이도 수많은 자잘한 구성 작업을 대체할 수 있다. 예컨대 디렉터리 구조, 초기 ESLint 규칙, Jest 기반의 테스트 환경 등 필요한 구성 요소 대부분이 포함되어 있어 즉시 코드 작성에 착수하기 쉽다. 그러나 프로젝트가 커지거나 특정 기능을 위해 세밀한 커스텀 빌드 설정이 필요해지는 순간이 오면 CRA 방식이 오히려 제약이 될 수 있다. 특히 빌드 성능이나 번들 크기를 최적화하고 싶을 때, 또는 Webpack이 아닌 다른 번들러를 쓰고 싶을 때는 다른 대안이 필요하다.

Vite 소개¶

Vite는 Evan You(Vue.js 창시자)가 개발한 차세대 프런트엔드 빌드 툴로, React 프로젝트도 빠르고 간단히 생성할 수 있다. 특히 esbuild, Rollup 등을 활용하여 개발 서버 부팅 및 HMR(Hot Module Replacement)이 매우 빠르고, 번들링 성능 역시 우수하다. CRA는 Webpack 기반으로 동작하기 때문에 초기 빌드와 리빌드에 많은 시간이 소요될 수 있지만, Vite는 모듈 단위로 ES 모듈을 직접 서빙하거나, 우선 개발 모드에서 필요한 부분만 빠르게 번들링하는 최적화 전략을 사용한다.

Vite로 React 프로젝트를 만들 때는 npm create vite@latest 명령을 사용하는 것이 일반적이다. React + TypeScript 템플릿을 선택하려면 대화식 프롬프트 혹은 인자 옵션을 사용할 수 있다. 예시로 아래와 같이 입력한다.

npm create vite@latest my-app -- --template react-ts

프로젝트 디렉터리로 이동한 뒤 npm install을 통해 의존성을 설치하고, npm run dev로 개발 서버를 실행하면 매우 빠른 속도의 HMR을 체감할 수 있다. 또한 Vite는 index.html 파일의 위치나 설정 방식이 CRA와 다소 다르므로, 초기 템플릿 구조를 살펴본 뒤 main.tsx 또는 main.jsx에서 ReactDOM.createRoot를 어떻게 구성하는지 확인하는 것이 좋다. React 18 프로젝트에서는 createRoot API를 사용하므로, 해당 부분에 대한 설정이 이미 반영되어 있는지도 함께 살펴본다.

Vite는 환경 변수 설정 방식도 CRA와 다르다. import.meta.env 객체를 통해 환경 변수에 접근하며, VITE_ 접두사를 붙여 선언한다. 예컨대 VITE_API_ENDPOINT 환경 변수를 .env 파일에 정의해두고 import.meta.env.VITE_API_ENDPOINT로 접근하는 식이다. 이는 CRA의 process.env.REACT_APP_XXX 방식과 동일한 개념이지만, 내부적으로는 esbuild를 사용하는 차이가 있고, 사용하는 접두사 역시 다르다.

Vite를 사용하면 번들링 도구로 Rollup을 사용하여 프로덕션 빌드를 생성한다. Rollup이 제공하는 Tree Shaking, 코드 스플리팅, 플러그인 시스템 등의 이점 때문에 더 세밀하고 효율적인 번들 최적화가 가능하다. 설정 파일(vite.config.js 혹은 vite.config.ts)을 통해 번들 동작, 플러그인, 환경 변수를 자유롭게 커스터마이징할 수 있다는 점도 큰 장점이다. 프로젝트 규모가 커질수록 이런 유연한 설정 능력이 도움이 될 수 있으며, TypeScript 설정(tsconfig.json) 역시 개발자가 직접 통제 가능하다.

CRA와 Vite를 함께 고려하기¶

CRA와 Vite는 모두 React 18 + TypeScript 기반의 프로젝트를 쉽게 시작하는 훌륭한 방법이다. CRA는 공식적으로 권장되는 표준화된 방식이며, 안정적이고 광범위한 커뮤니티 생태계를 갖춘다. Vite는 빠른 개발 서버와 빌드 속도를 제공하며, 간결하고 유연한 구성을 지향한다. 간단한 예제로 테스트를 해보면, Vite가 훨씬 빠른 초기 구동 시간을 보이는 경우가 많다.

CRA를 쓰는 경우 설정 파일을 명시적으로 다루지 않고도 빠르게 개발이 가능하며, 대다수의 React 튜토리얼이나 예제들이 CRA를 활용하기 때문에 학습 자료도 풍부하다. 그러나 커스텀 웹팩 설정을 요구하는 고급 시나리오에서는 한계를 느낄 수 있다.

Vite는 esbuild와 Rollup 기반의 체계를 가져 최신 스펙을 빠르게 지원하고, 설정 파일을 통한 세밀한 커스텀도 가능하다. 특히 Typescript 구성에 대한 제약이 적고, 빌드나 배포 과정을 따로 세부적으로 다룰 필요가 있는 경우 Vite가 유리하다. 더불어 각종 프레임워크별 템플릿이나 플러그인 생태계가 활발히 확장되고 있어, 점차 React를 포함한 다양한 프로젝트에서 Vite를 도입하는 추세가 늘고 있다.

프로젝트 생성 과정의 일반적 흐름¶

필요한 도구가 설치된 환경에서 React 18과 TypeScript 프로젝트를 초기화하는 단계는 크게 보아 몇 가지 공통 과정을 거친다. 초기화 단계에서 사용 도구(CRA, Vite 등)에 따라 명령어와 구성 파일이 달라지지만, 디렉터리 구조가 생성되고, React와 TypeScript가 설치되고, 개발 서버가 가동되며, 빌드가 이루어지는 순서는 본질적으로 유사하다. 아래는 프로젝트를 생성하는 일반적 흐름을 간단히 나타낸 예시다.

flowchart LR A(프로젝트 생성 명령 실행) --> B(기본 구조 및 설정 파일 생성) B --> C(React 18용 createRoot 확인) C --> D(TypeScript 및 빌드 설정 점검) D --> E(개발 서버 구동 및 테스트)

이 과정에서 가장 중요한 포인트는 React 18의 기능을 사용하는 부분을 올바르게 설정했는지 확인하는 것이다. CRA든 Vite든 템플릿에 따라 이미 createRoot 사용 방식이 적용되어 있을 수 있으나, 이전 버전의 React 프로젝트 구조를 재활용하거나 업그레이드하는 경우 기존의 ReactDOM.render 호출을 수정해야 한다.

TypeScript 설정(tsconfig.json) 역시 주의 깊게 살펴야 한다. CRA는 기본적으로 숨겨진 설정이 많기 때문에, 특정 컴파일 옵션을 바꾸고 싶다면 tsconfig.json에서 수정이 불가능할 수도 있고, react-app-env.d.ts 파일 등과의 호환 관계를 고려해야 한다. Vite에서는 비교적 자유롭게 tsconfig.json을 설정할 수 있지만, 별도의 path 설정이나 strict 모드 옵션 등을 활성화했을 때 프로젝트 전반의 코드를 모두 수정해야 할 수도 있다.

프로젝트 생성 방식 선택 시 고려할 점¶

CRA와 Vite 중 어떤 방식을 선택할지는 프로젝트 규모, 팀 구성, 그리고 설정 커스텀 정도에 따라 달라진다. 소규모 팀이 단기간에 React와 TypeScript를 이용해 프로토타입을 만들어야 한다면 CRA의 단순함이 매우 도움이 된다. 대규모 프로젝트에서 빌드 성능과 설정 유연성이 중요하다면 Vite가 더 유리할 수 있다. 기존에 CRA를 활용해온 코드베이스가 있거나, 사내 표준 템플릿으로 정착된 경우라면 CRA를 유지하는 것이 운영상 편리할 수 있다. 반대로 Vite가 제공하는 빠른 개발 서버와 다양한 플러그인 생태계의 이점을 체감하고 싶다면 Vite로 전환하는 것이 미래 지향적일 수 있다.

React 18과 TypeScript로 작성된 애플리케이션은 점진적으로 규모와 복잡도가 증가하기 마련이며, 이때 성능과 생산성 모두가 중요하다. 빌드 및 배포 환경을 장악하고 싶다면 Vite의 설정 파일을 적극적으로 활용하는 편이 낫고, 도구 설정보다는 코드 구현에 집중해야 하는 상황이라면 CRA가 낫다고 볼 수 있다.

CRA와 Vite는 모두 React 18과 TypeScript 개발 환경을 빠르게 구축하기 위한 강력한 툴이다. CRA는 오랜 시간에 걸쳐 공식 문서와 커뮤니티가 쌓아온 안정성과 풍부한 예제가 강점이며, 기본 설정을 따르는 한 큰 문제 없이 개발에 집중할 수 있다. Vite는 최신 번들링 기법을 통해 빠른 빌드와 HMR을 자랑하며, 유연한 설정 파일을 통해 세밀한 커스텀도 가능하다.

이 두 가지 도구 모두 React 18의 createRoot API를 완벽히 지원하고, TypeScript 템플릿 혹은 설정 옵션을 갖추고 있으므로, 취향과 요구사항에 맞춰 프로젝트를 시작하면 된다. 이 책의 이후 내용에서 다룰 실습 예제나 코드 스니핏은 상황에 따라 CRA 혹은 Vite를 사용하는 방법을 함께 제시할 것이며, 필요에 따라 어떤 점을 주의해야 하는지도 설명한다. 궁극적으로는 개발 편의성과 프로젝트 요구사항 사이에서 균형을 맞춰 도구를 선택하는 것이 가장 효율적인 시작점이 될 것이다.
