자바스크립트(ES6+)의 특징¶

ECMAScript 6 이후의 자바스크립트는 이전 버전에 비해 매우 큰 변화를 거쳤다. ES6(ECMAScript 2015)부터 시작하여 해마다 새로운 기능이 추가되고 사양이 발전함에 따라, 자바스크립트는 웹 브라우저뿐 아니라 서버 및 다양한 환경에서 활용되는 언어로 성장해 왔다. 블록 스코프를 지원하는 let과 const, 화살표 함수(Arrow Function), 클래스 문법, 모듈 시스템, 비동기 프로그래밍 모델 등이 대표적인 예시다. 이러한 기능은 코드 가독성을 높이고 유지보수를 용이하게 했으며, 모던 자바스크립트를 기반으로 동작하는 React 18에서도 적극적으로 활용된다.

블록 스코프와 let, const¶

기존의 var 키워드는 함수 스코프만을 지원했기 때문에 예기치 않은 재할당이나 호이스팅으로 인한 혼동이 잦았다. ES6에서 새롭게 도입된 let 키워드는 블록 스코프를 지원하므로, 중괄호로 감싸진 영역 내에서만 변수가 유효하고 재선언이 불가능하다. const 키워드는 상수(Constant)를 선언할 때 사용되며, 선언 시점에 반드시 값을 할당해야 하고 재할당이 허용되지 않는다. 이러한 스코프 관리 체계는 코드의 안정성을 높여주며, 복잡한 로직에서 오류를 줄여준다.

화살표 함수¶

화살표 함수는 가독성을 높이고 간결한 문법을 제공한다. 또한 화살표 함수 내의 this가 상위 스코프의 this를 유지한다는 점이 특징이다. 콜백 함수가 중첩되는 상황에서 자주 발생하던 this 바인딩 문제를 효과적으로 해결해 주며, React 컴포넌트에서 이벤트 핸들러나 콜백 함수를 작성할 때도 많이 활용된다.

const sum = (a: number, b: number): number => a + b;

위와 같은 화살표 함수는 간결함과 동시에 명확한 스코프 바인딩을 제공한다. ES6 이전에는 function 키워드와 bind 메서드를 통해 this를 컨트롤했지만, 이제는 이런 번거로움이 많이 사라졌다.

비동기 프로그래밍과 Promise, async/await¶

자바스크립트는 단일 스레드 기반의 이벤트 루프를 통해 비동기 동작을 처리한다. ES6 이전까지는 콜백 패턴으로만 비동기를 제어했으나, ES6부터 Promise가 도입되어 비동기 처리가 보다 구조화되었다. 이후 async/await 키워드가 추가되어 마치 동기식 코드처럼 비동기 로직을 작성할 수 있게 되었으며, 코드 가독성이 크게 향상되었다.

async function fetchData() {
  try {
    const response = await fetch("https://api.example.com/data");
    const result = await response.json();
    return result;
  } catch (error) {
    console.error(error);
    throw error;
  }
}

위 예시처럼 async/await를 활용하면 복잡한 비동기 흐름을 단순한 구조로 표현할 수 있다. 이 기법은 React 18의 서버 사이드 렌더링이나 데이터 패칭 로직에서 더욱 유용하게 사용된다.

모듈 시스템¶

ES6 모듈 시스템은 import와 export 키워드를 통해 모듈을 정의하고 가져올 수 있게 한다. 이전에는 CommonJS(주로 Node.js에서 사용), AMD(주로 브라우저에서 사용) 등 모듈화 방식이 혼재했으나, ES6 모듈은 표준화된 문법으로 코드 구조화를 도와준다. React 애플리케이션에서는 컴포넌트 단위로 모듈을 분리하고, 필요한 곳에서 import하여 재사용할 수 있다.

import React from "react";
export const MyComponent = () => <div>Hello, World!</div>;

ES6 모듈을 활용하면 코드 의존성을 명확히 파악할 수 있으며 트리 셰이킹(Tree Shaking)과 같은 번들 최적화 전략도 적용이 용이하다.

클래스 문법¶

ES6 클래스 문법은 프로토타입 기반의 객체 지향 모델을 보다 직관적으로 사용할 수 있도록 해준다. constructor, extends, super 등의 키워드를 지원하며, 정적 메서드와 정적 프로퍼티를 통해 구조화된 코드를 작성할 수 있다. 자바스크립트 내부적으로는 프로토타입 방식을 사용하지만, 클래스 구문 덕분에 전통적인 객체 지향 언어에 익숙한 개발자들도 자연스럽게 적응할 수 있다.

class Person {
  name: string;
  constructor(name: string) {
    this.name = name;
  }
  greet() {
    return `Hello, ${this.name}`;
  }
}

ES6 클래스 문법이 도입되면서 React 라이프사이클 메서드를 가진 클래스 컴포넌트가 등장했지만, React 16.8 이후 훅(Hooks)이 보편화되면서 함수형 컴포넌트가 더욱 주류가 되었다. 그럼에도 클래스 문법은 다양한 라이브러리나 프로젝트에서 여전히 유효하다.

TypeScript의 등장 배경¶

자바스크립트는 유연하지만, 대규모 애플리케이션 개발에서 점차 복잡성이 증가함에 따라 런타임 오류가 빈번하게 발생하고 코드를 유지보수하기가 어려워졌다. 이러한 문제를 해결하기 위해 Microsoft에서 자바스크립트의 상위 집합(Superset) 언어인 TypeScript를 발표했다. TypeScript는 자바스크립트 코드를 포함할 수 있는 동시에 정적 타입 검사(Static Type Checking)와 최신 ECMAScript 기능을 선제적으로 지원한다.

정적 타입 선언¶

TypeScript의 가장 큰 특징은 정적 타입 지원이다. 자바스크립트에서는 런타임에 타입이 결정되지만, TypeScript는 컴파일 시점에 타입 오류를 확인한다. 이를 통해 코드를 작성하는 과정에서 다양한 오류를 조기 발견할 수 있어 디버깅 시간이 단축되고, 대규모 프로젝트에서도 안전성이 높아진다.

function multiply(a: number, b: number): number {
  return a * b;
}

위 코드처럼 매개변수와 반환값에 타입을 지정하면, 잘못된 데이터 타입을 전달할 경우 컴파일 단계에서 오류가 발생한다. 자바스크립트에서는 런타임에만 오류가 감지되므로, 여러 단계를 거쳐야만 문제를 발견하는 경우가 잦았다.

인터페이스와 타입 별칭¶

TypeScript는 구조적 서브타이핑(Structural Subtyping)을 기반으로 작동하며, 인터페이스나 타입 별칭(Type Alias)을 사용해 객체의 구조를 정의할 수 있다. 인터페이스를 사용하면 객체의 형태를 사전에 약속할 수 있고, 코드 재사용성이 높아진다. 타입 별칭은 다양한 타입 조합을 간결하게 표현하거나 유니온, 인터섹션 타입을 정의할 때 유리하다.

interface User {
  id: number;
  name: string;
  isAdmin?: boolean;
}

type UserId = number | string;

function getUserInfo(user: User): UserId {
  return user.id;
}

위 예시에서 인터페이스 User는 id, name, isAdmin 프로퍼티를 가지도록 정의했다. isAdmin 프로퍼티는 물음표(선택적 프로퍼티)를 사용해 필수 사항이 아님을 나타낸다. 또한 UserId 타입 별칭을 통해 숫자와 문자열 중 하나를 사용 가능하다는 것을 지정할 수 있다. 이러한 타입 정의는 IDE에서 코드 자동 완성(IntelliSense)을 향상시키고, 협업 시에도 큰 도움을 준다.

고급 타입¶

TypeScript는 제네릭(Generic) 외에도 유니온(Union), 인터섹션(Intersection), 조건부(Conditional), 맵드(Mapped) 타입 등 다양한 고급 타입 체계를 제공한다. 이를 이용하면 함수를 오버로드하지 않고도 복잡한 타입 구조를 정확하게 표현할 수 있다. 또한 keyof, in, infer 같은 키워드를 통해 더욱 동적인 타입 정의도 가능하다.

type ApiResponse<T> = {
  data: T;
  error?: string;
};

고급 타입을 적극적으로 활용하면 React 18 프로젝트에서 Redux, React Query, Apollo Client 같은 다양한 상태 관리 또는 데이터 관리 라이브러리와 결합할 때 안전하고 견고한 코드를 작성할 수 있다.

제네릭(Generic)¶

TypeScript의 제네릭은 함수나 클래스를 선언할 때, 인자의 타입이나 반환 타입 등을 유연하게 처리하도록 도와준다. 자바스크립트에서 함수가 인수의 타입이나 개수를 엄격하게 제한하지 않는 반면, 제네릭을 사용하면 타입 변환 과정에서 발생할 수 있는 오류를 사전에 방지할 수 있다.

function identity<T>(value: T): T {
  return value;
}

이처럼 제네릭을 사용하면 어떤 타입이 들어와도 해당 타입을 그대로 반환한다는 사실이 보장되므로, 런타임 오류 가능성을 줄이고 코드 안정성을 높일 수 있다. React 18의 useState나 useReducer 훅도 내부적으로 제네릭이 활용되어, 상태 변수의 타입을 명시할 수 있다.

Decorator¶

Decorator는 클래스나 메서드, 접근자, 프로퍼티, 매개변수 등에 추가 기능을 주입할 수 있는 메타프로그래밍 기법이다. 자바스크립트의 정식 사양에 완전히 포함된 기능은 아니지만, TypeScript 컴파일러 설정에서 experimentalDecorators 옵션을 활성화하면 사용할 수 있다. React 18 환경에서 Decorator를 활용하는 경우는 드물지만, NestJS 같은 프레임워크에서는 의존성 주입(Dependency Injection)이나 메타데이터 주입 등에 Decorator가 광범위하게 사용된다.

컴파일 및 트랜스파일¶

TypeScript 파일(.ts, .tsx)은 반드시 자바스크립트로 트랜스파일(Transpile)된 후 실행된다. React 애플리케이션에서는 Babel이나 SWC 등의 도구와 함께 TypeScript 컴파일러(tsc)를 사용해 코드를 빌드한다. 빌드 과정에서 타입 검사와 코드 변환이 이루어지므로, 배포 파일은 순수 자바스크립트 코드로 만들어진다.

flowchart TB A["TypeScript Source (.ts, .tsx)"] --> B[TypeScript Compiler] B --> C["Transpiled JavaScript (.js)"] C --> D[Babel/SWC Bundling] D --> E[Browser or Node.js Execution]

위 다이어그램처럼 TypeScript 코드는 tsc 단계에서 자바스크립트로 변환되며, 이후 추가적인 번들러 단계를 거친다. React 18 프로젝트에서도 이러한 과정이 표준적으로 이뤄지며, tsconfig.json 파일을 통해 다양한 컴파일 옵션을 설정한다.

자바스크립트(ES6+)와 TypeScript의 차이점¶

자바스크립트(ES6+)와 TypeScript는 동일한 실행 환경에서 동작하고, ES6+가 제공하는 문법을 모두 활용할 수 있다. 다만 TypeScript는 정적 타입 시스템, 인터페이스, 고급 타입, 데코레이터 기능 등을 추가함으로써 대규모 애플리케이션을 안정적으로 작성할 수 있도록 지원한다. 따라서 자바스크립트보다 컴파일 과정이 추가되고, tsc를 통한 타입 검사 후 자바스크립트로 트랜스파일하는 단계를 반드시 거친다.

TypeScript는 자바스크립트의 상위 집합(Superset)이기 때문에 모든 자바스크립트 코드를 그대로 포함할 수 있다. 기존의 자바스크립트 프로젝트에 점진적으로 TypeScript를 도입해 타입 안정성을 확보할 수 있으며, 팀 규모가 커질수록 이러한 정적 타입 검사와 풍부한 도구 지원이 큰 장점이 된다. 다만 학습해야 할 개념이 추가되고, 컴파일 시간이 늘어날 수 있다는 점을 고려해야 한다.

TypeScript를 사용하면 코드 에디터나 IDE에서 풍부한 자동 완성과 타입 힌트를 제공받을 수 있다. 함수 시그니처나 인터페이스를 미리 정의해 두면, 예기치 않은 프로퍼티 접근 오류나 잘못된 함수 호출을 미리 방지할 수 있으며, 협업 시 코드의 의도를 명확히 드러낼 수 있다. 반면 자바스크립트는 타입 검사가 없으므로, 재사용성 높은 라이브러리나 프레임워크를 개발하거나, 복잡도가 높은 비즈니스 로직을 다룰 때 런타임 오류가 발생하기 쉽다.

TypeScript는 React 18 환경에서 더욱 많이 사용되고 있으며, .tsx 파일을 통해 JSX 문법에 타입을 결합할 수 있다. 컴포넌트의 props나 state에 타입을 지정하면, 에디터 차원에서 컴파일 단계부터 실수를 바로 잡을 수 있고, 리팩토링 또한 수월해진다. 이로써 코드의 품질이 향상되고 유지보수가 용이해지며, 팀 전체가 합의한 타입 정의를 통해 협업 과정에서의 충돌을 최소화할 수 있다.

정리하자면, 자바스크립트(ES6+)는 명시적 타입 선언 없이 신속하게 코드를 작성하기에 적합하며, 이미 웹 전체에서 사실상의 표준 역할을 하고 있다. 반면 TypeScript는 정적 타입 검사와 풍부한 언어 기능을 제공함으로써, 대규모 프로젝트나 장기 유지보수가 필요한 상황에 더욱 적합하다. React 18 개발에서도 TypeScript를 함께 사용하면, 컴포넌트 구조를 더욱 견고하게 설계할 수 있고, 애플리케이션의 안정성을 한층 높일 수 있다.
