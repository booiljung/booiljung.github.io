프로젝트 구조를 고민해야 하는 이유¶

프로젝트를 React 18과 TypeScript로 시작할 때에는 CRA(Create React App), Vite, Webpack과 같은 툴 중 어느 것을 사용하든 간에 폴더 구조를 명확하게 잡는 일이 중요하다. 유지보수성과 확장성이라는 두 가지 축을 만족시키려면 어느 위치에 어떤 파일이 들어가야 하는지를 미리 고민하는 과정이 반드시 필요하다. 한 번 정해진 구조는 프로젝트가 어느 정도 규모에 도달하면 바꾸기가 쉽지 않기 때문에, 초기 단계에서 목적과 요구사항에 맞는 구조를 잡아두는 것이 좋다. 모든 설정이 끝난 후에는 개발자가 의도한 위치에서 직관적으로 파일을 찾고, 추가하거나 제거하는 데 어렵지 않아야 하며, 팀 단위로 협업할 때에도 동일한 원칙으로 프로젝트를 이해할 수 있어야 한다. 특히 TypeScript를 도입하면 자바스크립트보다 파일의 역할이 더 명확해지고, 선언 파일(types), 인터페이스, 유틸리티 함수 등 구조적으로 나누어야 할 요소들이 늘어나므로 폴더 구조 설계가 훨씬 중요해진다.

도메인 기반 또는 계층(레이어) 기반 구조 설계¶

구조를 설계할 때에는 크게 도메인(domain) 중심으로 나누는 방법과 계층(layer) 중심으로 나누는 방법이 고려 대상이 된다. 도메인 중심 방법이란, 기능(예: 회원, 상품, 결제) 혹은 페이지 단위(예: Home, About, Dashboard)로 묶어서 폴더를 만들고 그 안에 필요한 컴포넌트, 훅, 타입, 상태 로직 등을 일괄적으로 배치하는 방식이다. 계층 중심 방법이란, 모든 컴포넌트를 한 폴더에, 모든 훅을 한 폴더에, 모든 유틸 함수를 또 다른 폴더에 모으는 식으로 역할별로 폴더를 구분하는 방식이다. React의 컴포넌트 구조는 재사용성과 모듈화를 극대화하기 위해 종종 도메인 중심으로 접근하기도 하며, 반대로 기능별 확장성을 중시하는 프로젝트라면 계층 중심이 유리할 수도 있다. 실제로는 두 방법을 혼합하여 사용하기도 하므로, 프로젝트의 성격에 맞춰서 유연하게 적용하는 것이 좋다.

폴더 구조 설계 시 혼합 전략¶

도메인 중심과 계층 중심을 단순히 어느 한 쪽만 선택하기보다, 도메인별로 대분류한 뒤 그 안에서 다시 역할별로 세분화하는 식의 혼합 전략을 사용할 수도 있다. 예를 들어 대규모 애플리케이션에서 특정 도메인(예: 사용자 관리)이 매우 복잡해진다면, 사용자 관리에 관한 컴포넌트, 상태 로직, API 호출 함수, 유틸 함수 등을 하나의 폴더 아래에 배치하되 내부적으로 적절히 레이어를 구분함으로써 가독성과 확장성 둘을 모두 잡을 수 있다. 이 과정에서 이름 충돌이나 역할 중복이 생기지 않도록 디렉터리와 파일 이름을 잘 정리하는 것이 핵심이다.

예시 폴더 구조¶

폴더 구조에 대한 예시를 시각적으로 나타내기 위해 아래 그림을 참고할 수 있다. 이는 CRA든 Vite든 Webpack이든 상관없이 공통적으로 React 18 + TypeScript 프로젝트에서 사용할 수 있는 하나의 예시일 뿐이며, 실제 프로젝트의 요구사항에 따라 유연하게 변형할 수 있다.

graph LR A[프로젝트 루트] --> B[src] B --> C[components] B --> D[pages] B --> E[hooks] B --> F[utils] B --> G[types] B --> H[store] A --> I[public] A --> J[.gitignore] A --> K[tsconfig.json] A --> L[package.json] A --> M[webpack or vite config]

상위 레벨에 src 폴더가 있고, 그 안에 components, pages, hooks, utils, types, store 등과 같이 React 애플리케이션의 주요 레이어를 구성할 수 있다. public 폴더는 정적 파일을 담기 위한 용도로 쓰이며, 빌드 설정 파일(tsconfig.json, webpack 또는 vite 관련 설정 파일)과 프로젝트 공통 설정 파일(.gitignore, package.json 등)은 루트에 위치한다. 이 구조는 가장 일반적으로 많이 쓰이는 형태 중 하나지만, 실제로는 더 세부적으로 도메인별 하위 폴더를 두거나, pages 내부를 또다시 기능별로 쪼개기도 한다.

컴포넌트 폴더 구성과 네이밍 컨벤션¶

React 18 프로젝트에서 컴포넌트는 주로 사용자 인터페이스를 구성하는 단위 요소이므로, 폴더링과 네이밍에 있어 일관성을 유지하는 것이 매우 중요하다. 대부분의 경우 컴포넌트 폴더 안에 각 컴포넌트별로 폴더를 따로 생성하여 관리한다. 이름을 정할 때에는 컴포넌트 이름을 PascalCase로 작성하는 것이 관례이다. 예를 들어 UserCard라는 컴포넌트는 src/components/UserCard 디렉터리 내부에 UserCard.tsx 파일과 함께 스타일 정의 파일, 테스트 파일, 그리고 필요한 서브 컴포넌트를 포함하는 식으로 배치한다. 컴포넌트 이름과 폴더 이름을 동일하게 설정하면 찾기가 수월해지고, 각 컴포넌트 단위로 독립적인 환경을 갖출 수 있다.

페이지 폴더 구성과 라우팅 전략¶

페이지 폴더에는 라우팅을 통해 직접 접근되는 화면 단위가 들어간다. 예를 들어 Home, About, Dashboard와 같은 각각의 페이지 컴포넌트가 존재한다. 이 페이지들은 종종 Next.js 같은 프레임워크를 쓰지 않는 이상 React Router를 사용하여 라우팅을 설정하게 되는데, 이때도 페이지 파일의 이름은 PascalCase 또는 기능에 따라 약간 변형된 케이스로 작성한다. 여러 화면을 관장하는 페이지 컴포넌트 내부에서 공통으로 사용하는 서브 컴포넌트가 많다면, 페이지 폴더 내부에 또 다른 components 폴더를 만들어서 관리하거나, 프로젝트 구조에 따라 별도의 도메인 폴더를 두고 그 안에 컴포넌트를 배치할 수도 있다.

훅과 유틸 함수의 분리¶

React에서 커스텀 훅(custom hook)은 로직을 재사용하는 핵심 수단이다. 훅을 한 곳에 몰아넣는 계층적 구조를 택할 수 있고, 도메인별로 분산하는 방식도 가능하다. 어떤 방식을 택하든 간에 커스텀 훅의 이름은 use로 시작하는 것이 컨벤션이며, 이것이 곧 훅 파일임을 직관적으로 나타낸다. 유틸 함수(utils)는 React와 상관없이 쓸 수 있는 범용 함수나 특정 도메인 로직을 담당하는 경우가 많으므로, 이 역시 별도의 utils 폴더로 분리해두는 편이 좋다. 유틸 함수는 이름을 지을 때 함수의 기능을 명확히 드러내되, 중복되지 않고 검색하기 용이하게 정한다.

타입 정의 파일과 인터페이스 관리¶

TypeScript를 사용하는 프로젝트라면, 분산된 컴포넌트와 API 함수들이 공유하는 타입 정의를 어떻게 관리할 것인지도 중요한 문제이다. types 폴더를 만들어 공통 타입(예: User, Product, Order)을 모아놓거나, d.ts 확장자를 가진 선언 파일을 따로 두어 전역 타입을 정의하기도 한다. 타입 선언 파일을 어디에 두어야 할지는 프로젝트 성격에 따라 달라진다. 공통적으로 사용되는 타입은 최상위 types 폴더 아래에 두고, 특정 도메인에서만 쓰이는 타입은 해당 도메인 폴더 안에 두는 식으로 분리하기도 한다. 인터페이스나 타입 이름은 PascalCase로 시작하는 것이 일반적이며, 어떤 팀은 인터페이스 앞에 I 접두사를 붙이기도 하지만, 최근에는 붙이지 않는 추세도 많다.

전역 상태 관리를 위한 store 폴더 구조¶

Redux나 Recoil, Zustand, MobX 같은 상태관리 라이브러리를 사용할 경우, store 또는 states라는 폴더를 만들어 전역 상태를 선언하거나 정의하는 파일을 배치한다. Redux를 예로 들면 rootReducer, store 설정, 미들웨어 설정, 액션/리듀서가 들어갈 폴더 구조를 만든다. 도메인별로 slice를 분리해두면 추후 유지보수가 편리해지고, slice 이름과 파일 이름을 일관성 있게 만들어두면 코드 탐색이 수월해진다. 예를 들어 userSlice.ts, productSlice.ts 등으로 분리해서 관리할 수 있으며, 전역 상태를 정의하는 파일 이름 또한 PascalCase 또는 camelCase 방식 중 일관되게 적용하면 된다.

빌드 설정 파일과 외부 설정 파일 위치¶

프로젝트 최상위 루트 디렉터리에는 package.json, tsconfig.json, .gitignore와 같은 기본 설정 파일들이 들어간다. CRA, Vite, Webpack 같은 툴을 사용할 때에는 이에 해당하는 환경 설정 파일(예: vite.config.ts, webpack.config.js)을 루트 또는 config라는 별도의 폴더에 두기도 한다. 빌드 설정 파일의 위치는 어느 정도 관례가 있지만, 팀 내 규칙이나 도구의 권장 사항에 따라 조금씩 달라질 수 있다. 중요한 것은 설정 파일이 어디에 있는지를 초기에 잘 문서화해두고, 추후 다른 개발자가 들어왔을 때 혼동을 겪지 않게 하는 것이다.

네이밍 규칙의 일관성 유지¶

컴포넌트, 훅, 유틸 함수, 타입, 전역 상태 등의 파일 이름은 결국 협업과 유지보수성을 결정짓는 요인이다. React 18 + TypeScript 프로젝트에서 다음과 같은 원칙을 준수하면 좋다. 첫 번째로 컴포넌트 이름은 PascalCase로 한다. 두 번째로 훅 이름은 use로 시작하고 camelCase로 작성한다. 세 번째로 유틸 함수 이름은 소문자로 시작하며, 함수의 기능을 반영하는 명확한 이름을 짓는다. 네 번째로 타입이나 인터페이스는 PascalCase를 쓴다. 다섯 번째로 테스트 파일을 작성할 때에는 원본 파일 이름에 .test 혹은 .spec을 붙이고, 어디에 배치할지 미리 정해둔다. 이러한 네이밍 규칙은 ESLint 설정이나 Prettier 설정, 또는 tsconfig 경로 별칭 설정과 연동하여 강제하거나 자동화할 수 있다. 이름이 달라졌을 때 생길 수 있는 오류나 혼란을 최소화하기 위해, 프로젝트 초기에 이런 규칙을 문서화해두는 것이 바람직하다.

규모 확장 시의 폴더 구조 리팩터링 고려¶

프로젝트가 초기에는 간단하더라도, 기능이 점점 늘어나고 페이지가 많아지다 보면 폴더 구조가 복잡해질 수 있다. 이 시점에서 리팩터링이 필요한 대표적인 징후로는 폴더 안에 파일이 지나치게 많아서 목적별 구분이 어려운 경우, 컴포넌트 재사용도나 분리 정도가 지나쳐서 오히려 찾기 어려운 경우, 네이밍이 혼재되어 일관성이 깨진 경우 등이 있다. React 18과 TypeScript를 사용한다면 컴파일 시간에 타입 충돌 경고가 자주 발생할 수 있는데, 이런 문제가 잦아진다면 구조 자체가 잘못되었을 가능성이 있으므로 폴더링을 재고해보는 편이 좋다. 리팩터링 시에는 핵심 도메인을 기준으로 폴더를 나누고, 계층 간 참조를 최소화할 수 있도록 리듀스(줄이는) 작업을 하며, 컴포넌트 간 결합도를 살펴서 독립적인 모듈로 변환하는 과정을 거친다.

폴더 구조 문서화와 자동화¶

마지막으로, 폴더 구조와 네이밍 컨벤션은 팀 전체가 공감대를 가지고 지켜야 하는 규칙이므로 문서화가 필수다. README.md나 별도의 Wiki 등을 통해 어떻게 폴더가 구성되어 있고, 어떤 방식으로 파일을 배치해야 하며, 네이밍 규칙은 무엇인지 명시하면 협업 시에 많은 충돌을 예방할 수 있다. ESLint, Stylelint, Prettier, Husky 등을 활용하면 커밋 단계에서부터 코드를 검사하고 포맷팅해줄 수 있으므로, 폴더 구조만큼이나 일관된 코드 품질도 기대할 수 있다. 프로젝트가 확장됨에 따라 폴더를 재배치하거나 네이밍 규칙을 변경할 경우, 반드시 문서를 동시에 업데이트해야 한다.

프로젝트 구조 설계를 체계적으로 진행하면, React 18과 TypeScript 프로젝트에서 발생할 수 있는 복잡도를 줄이고, 장기적으로 유지보수와 협업 효율을 높일 수 있다. 팀 규모나 프로젝트 규모가 클수록 이러한 구조적 설계는 더욱 중요해지며, 선택된 구조가 적절한지 꾸준히 검토하고 필요에 따라 개선하는 과정을 거쳐야 한다. 초기 설정 단계에서 시간을 조금 더 들이더라도, 명확하고 일관성 있는 폴더 구조와 네이밍을 지니고 있는 프로젝트가 결국 높은 생산성과 안정성을 보장한다.
