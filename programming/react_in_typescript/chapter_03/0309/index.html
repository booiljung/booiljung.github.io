기존 React 버전에서 호환성 점검¶

기존 프로젝트가 React 16 또는 17을 사용 중이었다면, 새로운 Concurrency Model과 관련된 코드가 예기치 않게 동작할 수 있는 부분을 먼저 찾아야 한다. 오래된 Class Component 기반의 라이프사이클 메서드가 새로워진 렌더링 전략과 어떻게 상호 작용하는지 살펴볼 필요가 있다. 특정 라이프사이클 메서드가 더 이상 권장되지 않거나 자동 배치(Automatic Batching) 로직과 충돌할 가능성이 있으므로, 이를 먼저 파악하는 작업이 중요하다. 동시에 외부 라이브러리나 서드파티 컴포넌트가 React 18 환경에서 정상 동작하는지 체크해야 한다. 이때 원하는 동작이 이뤄지지 않는다면 의존성 버전을 재확인하거나 대체 라이브러리를 모색하는 과정을 거치는 편이 안전하다.

React 18 설치 전 준비 사항¶

리액트 관련 패키지와 타입 정의 파일(@types/react)을 모두 최신 버전으로 업데이트해야 한다. 만약 TypeScript를 사용 중이라면 tsconfig 설정에서 JSX 변환 모드를 확인하고, React 18의 새로운 JSX 트랜스폼과 호환되는지 확인해야 한다. 만약 Babel을 쓴다면 .babelrc 설정에 포함된 플러그인 버전도 함께 점검하는 것이 좋다. 추가적으로 Webpack, Vite, Rollup 등 빌드 도구의 설정 및 플러그인들이 React 18의 JSX나 TS 타입 정의를 올바르게 처리하는지도 살펴봐야 한다. 이러한 사전 준비가 잘 되어 있어야 실제 마이그레이션 과정에서 발생할 수 있는 오류를 최소화할 수 있다.

자동 배치(Automatic Batching) 관련 변경 사항¶

React 18에서는 자동 배치 기능이 확장되었다. 과거에는 React 이벤트 핸들러 안에서만 배치가 이뤄졌지만, 이제는 setTimeout 또는 fetch와 같은 비동기 이벤트 콜백 내에서도 자동 배치가 적용된다. 이로 인해 여러 상태 업데이트가 한 번에 처리되어 불필요한 렌더링 횟수가 줄어드는 장점이 있지만, 기존에 비동기 호출마다 재렌더링이 발생하는 로직에 의존해 왔다면 예기치 않은 동작 변화를 초래할 수 있다. 자동 배치 로직이 적용되면서, 렌더링 타이밍이나 컴포넌트 상태 값이 바뀌는 순간이 달라질 수 있으므로 이 부분을 세밀하게 검증해야 한다. 로그를 찍어 가며 상태 업데이트의 시점이 바뀌지 않았는지 확인하거나, Transition API와 함께 사용해도 무리가 없는지 점검하는 과정을 거쳐야 한다.

Transition API 적용 여부¶

Transition API를 사용하면 상태 업데이트의 우선순위를 조정하고, 유저 인터랙션과 구분하여 화면을 업데이트할 수 있다. 예컨대 검색어 입력에 따른 자동완성 리스트를 렌더링할 때, 가볍게 처리해야 하는 인터랙션과 조금 느려도 괜찮은 데이터 요청이나 무거운 연산을 구분하여 성능을 높일 수 있다. 만약 이미 사용자 경험을 최적화하기 위해 setTimeout, requestAnimationFrame 또는 일부 서드파티 라이브러리를 사용하고 있었다면, 이를 Transition API로 교체하는 것이 좋은 선택이 될 수 있다. 마이그레이션을 진행하면서 Transition API로 분리 가능한 상태 업데이트가 있는지, 그리고 현재 프로젝트 구조에서 손쉽게 적용이 가능한지 살펴봐야 한다.

Server Components와 Streaming 추가 검토¶

React 18은 Server Components 개념을 도입하고, 서버 사이드 렌더링(SSR) 과정에서 HTML을 스트리밍으로 전송하는 기능을 공식적으로 지원한다. 이는 SSR에 대한 접근 방식을 바꾸고, 서버와 클라이언트 간의 데이터 교환 모델을 새롭게 설계하도록 유도한다. 이미 SSR 프레임워크(Next.js, Remix 등)를 사용하는 경우라면, 해당 프레임워크가 React 18과 호환되는지 우선 확인해야 한다. 그리고 서버 컴포넌트를 도입했을 때 얻을 수 있는 성능상의 이점과 유지보수 복잡도를 면밀히 검토해야 한다. 마이그레이션 초기에 모든 컴포넌트를 서버 컴포넌트로 전환하기보다는, 우선 성능에 민감한 페이지나 내부적으로 데이터 처리가 큰 부분부터 점진적으로 적용하는 전략이 권장된다.

새로운 Suspense 기능 및 서버 사이드 렌더링¶

React 18에서 Suspense는 데이터 페칭이나 로딩 중인 UI를 보다 직관적으로 다룰 수 있게 되었다. 과거에는 Suspense가 클라이언트 사이드 렌더링(또는 Code Splitting) 상황에서만 제한적으로 사용되었지만, 이젠 SSR과 병행하여 좀 더 폭넓은 활용이 가능하다. 예컨대 서버 사이드에서 렌더링 도중에 필요한 데이터를 받아오지 못했다면, Suspense 구문을 이용해 로딩 UI를 표시하고, 스트리밍으로 그 결과를 조금씩 내려줄 수 있다. 이 과정을 통해 사용자 경험을 자연스럽게 유지할 수 있지만, 기존에 특정 시점에만 데이터를 강제로 동기화하거나 로딩 UI를 표시했다면, Suspense를 적용하는 로직으로 리팩토링하면서 동작 타이밍이 달라질 수 있다. 특히 TypeScript를 사용할 때엔 Suspense 관련 타입이 정확히 선언되어 있는지, 비동기로 처리되는 모듈 임포트나 fetch 로직에서 타입 오류가 나지 않는지 꼼꼼히 점검해야 한다.

Concurrent Rendering이 가져오는 영향¶

새로운 Concurrent Rendering 모드는 UI를 좀 더 부드럽게 업데이트할 수 있는 장점을 제공한다. React는 이제 더 이상 모든 상태 변경을 즉시 동기적으로 처리하지 않고, 우선순위나 현재 CPU 사용량 등을 고려해 렌더링을 지연시키거나 중단했다가 재개할 수 있다. 단, 이로 인해 컴포넌트가 여러 번 렌더되는 구간에서 발생하는 사이드 이펙트(예: useEffect, useLayoutEffect 등)를 다시 확인해야 한다. 이전에는 거의 한 번에 진행되던 렌더 사이클이 나눠져서 일어나므로, 어떤 시점에 사이드 이펙트가 실행되는지 예측이 어려워질 수 있다. 필요하다면 Strict Mode와 함께 새로운 렌더링 과정을 면밀히 살펴보면서, 부적절한 API 호출이나 레이스 컨디션이 일어나지 않도록 조정해야 한다.

아래 예시는 Concurrent Rendering 과정을 단순화한 흐름이다.

flowchart LR A(사용자 인터랙션 발생) --> B(Concurrent Rendering 시작) B --> C(업데이트 스케줄링) B --> D(우선순위 확인) C --> E(렌더 시도) D --> E E --> F(컴포넌트 렌더 완료)

이런 모식도를 통해 알 수 있듯이, 업데이트 스케줄링 단계와 우선순위 확인 단계가 분리되어 있고, 그 결과로 렌더링이 나누어 진행된다. 이 과정에서 리액트는 여러 번 렌더를 중단하거나 다시 시작할 수 있으며, 이는 상태 값이 예기치 않은 시점에 결정되는 효과를 일으킬 수도 있다. 따라서 프로젝트 전반에 걸쳐 이 과정을 면밀히 파악하고, 독립된 테스트 환경에서 각 업데이트가 정상적으로 실행되는지 확인해야 한다.

ReactDOM.render에서 createRoot로의 전환¶

React 17까지는 ReactDOM.render 함수를 사용해 화면에 컴포넌트를 그려 왔다. React 18에서는 createRoot가 새로운 엔트리 포인트가 되었다. 만약 기존 코드를 그대로 두면 경고 메시지가 출력되거나 일부 기능이 의도대로 동작하지 않을 수 있으므로, 반드시 createRoot 방식을 적용해야 한다. 이 과정에서 hydrate 기능을 사용한다면, createRoot가 아닌 hydrateRoot 함수를 사용하여 서버 사이드 렌더링된 HTML과 동기화하는 방식을 택해야 한다. createRoot는 Concurrent Rendering을 활용하기 위한 기본 전환이므로, 이 작업을 빠르게 완료한 뒤 전체 코드가 정상적으로 동작하는지 우선 파악하는 것이 좋다.

Hook 동작 확인¶

React 18의 렌더링 방식이 달라지면서, useState, useReducer, useEffect 등 기본 훅의 호출 순서나 횟수가 미묘하게 변할 수 있다. 렌더링 중단이나 재시작이 발생하면 effect가 예상과 다르게 여러 번 실행될 수 있고, transition으로 설정한 상태 업데이트가 나중에 처리될 수도 있다. TypeScript 환경에서 훅을 사용한다면, 제네릭 타입을 사용하는 경우나 커스텀 훅을 만들 때의 타입 정의가 Concurrent Rendering과 잘 맞물려 있는지 확인해야 한다. 예컨대 일부 커스텀 훅이 특정 시점에만 한 번 실행된다고 가정하고 설계되었다면, Concurrent Rendering 상황에서도 확실히 동작하도록 재검토할 필요가 있다.

에러 경계 컴포넌트 점검¶

에러 경계(Error Boundary)는 클라이언트 렌더링에서만 적용되지만, Concurrent Rendering이 활성화되면 에러 발생 지점이나 잡아내는 시점이 달라질 가능성이 있다. 예컨대 렌더 중간에 다른 작업이 우선순위를 가져가면서, 에러가 예상치 못한 타이밍에 발생할 수 있다. 에러 경계 컴포넌트가 정상적으로 동작하도록 isMounted 체크나 타이머가 제대로 설정되어 있는지, 또 오류가 발생한 후에 사용자에게 재시도 버튼을 보여주거나 로그를 전송하는 로직이 변함없이 동작하는지 살펴봐야 한다. 에러 경계 컴포넌트는 TypeScript에서 props와 state 인터페이스를 명확히 정의해야 하는데, 이때 Concurrent Rendering을 고려해 추가 상태(예: 재시도 횟수)를 두는 등 리팩토링을 할 수도 있다.

Strict Mode 강화된 점검¶

React 18 Strict Mode는 개발 환경에서 훅이나 라이프사이클 메서드가 두 번 호출되는 시뮬레이션을 더욱 철저하게 진행한다. 이는 실제 운영 환경에서 발생할 수 있는 Concurrent Rendering 시나리오를 미리 체험하도록 만들어진다. 엄격 모드에서 나타나는 추가 렌더링이나 사이드 이펙트 호출이 문제가 되지 않는지 살펴보고, 예상치 못한 부작용이 없다면 그대로 운영 환경에서도 동일하게 동작할 가능성이 높다. Strict Mode에서 계속 문제가 발생한다면, 그 문제는 Concurrent Rendering을 비롯한 React 18의 다양한 기능을 사용할 때 실제로도 문제가 될 수 있으니 반드시 수정해야 한다.

Legacy API 폐지 및 대안 모색¶

React 18로 넘어오면서 일부 레거시 API가 곧 폐지될 예정이거나 권장되지 않는 방식이 되었다. 예컨대 UNSAFE_ 접두사가 붙은 라이프사이클 메서드나 ReactDOM.findDOMNode 같은 API는 가급적 사용하지 않는 편이 좋다. Concurrent Rendering과 맞물려 안정성을 보장하기 어렵거나, 향후 React 개발 방향과 맞지 않아서 사라질 가능성이 높기 때문이다. 프로젝트 내에서 이런 API가 사용되는 곳을 찾아 대체 방식을 적용해야 한다. findDOMNode가 필요한 경우에는 ref를 통해 DOM 요소를 직접 참조하고, UNSAFE_ 메서드는 적절한 대안(예: getDerivedStateFromProps 또는 componentDidUpdate + setState 로직 등)으로 치환하는 형태가 일반적이다.

TypeScript 환경 설정 및 React 18 타입 검사¶

마이그레이션 과정에서 TypeScript가 적극적으로 에러와 경고를 보여주어, 코드 품질을 유지하는 데 큰 도움이 될 수 있다. React 18의 새로운 타입을 활용하려면 @types/react와 @types/react-dom을 맞춰 설치하고, 버전 호환성을 확인해야 한다. JSX 변환 모드도 react-jsx인지, 혹은 react-jsxdev 모드인지 등을 세심히 체크해야 하며, tsconfig.json에 "skipLibCheck" 설정을 켜두었다면 이 설정을 끄고 에러를 직접 확인하는 편이 좋다. React 18과 관련된 타입 호환 오류가 발생한다면, 일단 @types 패키지 버전부터 맞춰 본 다음, 프로젝트 내에서 사용하는 Global JSX 타입 선언이 충돌하지 않는지도 확인해야 한다.

JSX 변환과 타입 선언¶

새로운 JSX 트랜스폼을 사용할 경우, Babel과 TypeScript 설정을 함께 조정해야 한다. .babelrc나 tsconfig.json에서 "jsx": "react-jsx" 설정을 활용하거나, "react-jsxdev"처럼 개발 환경에 특화된 옵션을 사용하는지 확인한다. 이런 옵션이 일치하지 않으면 React 18의 기능들이 정상 동작하지 않을 수 있고, IDE가 JSX 문법을 해석하지 못할 수도 있다. 그리고 프로젝트 내에서 JSX.Element나 ReactNode 등을 정의하는 부분이 있다면, React 18 기준으로 업데이트된 타입과 충돌이 없는지 살펴봐야 한다. Suspense나 Transition과 관련된 타입 정의를 잘못 해 놓으면 훅이 동작할 때 오류가 날 수 있으므로, 마이그레이션 과정에서 빠짐없이 점검해야 한다.

ESLint 및 Prettier 설정 재점검¶

코드 스타일과 품질 검사 도구를 사용하는 프로젝트라면, ESLint와 Prettier 설정도 함께 수정해야 한다. React 18 관련 플러그인(예: eslint-plugin-react, eslint-plugin-react-hooks 등)이 최신 버전인지 다시 확인하고, 설정 파일에서 "react version"을 18로 명시할 필요가 있는지 살펴봐야 한다. React Hooks 규칙이나 Concurrent Rendering 관련 규칙이 업데이트되었을 수 있으므로, 기존에 오류가 없던 코드가 새 규칙에 의해 경고 메시지를 출력할 수도 있다. 이 경우에는 규칙을 끄는 것보다 우선 코드를 리팩토링하여 규칙을 지키는 편이 React 18 환경에서 더 안정적이다.

테스트 및 성능 모니터링¶

마이그레이션 후에는 기존 테스트 케이스가 정상적으로 통과하는지 확인해야 한다. Jest, React Testing Library 등을 사용 중이라면, React 18 호환 버전으로 업데이트하여 Concurrent Rendering 환경에 대한 테스트까지 대응할 수 있도록 한다. 만약 e2e 테스트를 진행하는 프로젝트라면, 사용자 입장에서 주요 UI 흐름이 원활히 작동하는지 시나리오 전반을 재검증하는 과정을 거친다. 동시에 성능 모니터링 도구(React Profiler, Web Vitals 등)를 활용해, React 17 대비 렌더링 속도나 사용자 입력 지연이 개선되었는지 확인할 수 있다. Transition API를 적용한 곳이 있다면, 그 효과가 실제로 체감 가능한지 데이터를 모으고, 필요에 따라 로직을 미세 조정할 수도 있다.

라이브러리 호환성 재확인¶

redux, mobx, recoil, zustand 등 상태 관리 라이브러리나 react-router, formik, react-query 등의 서드파티 라이브러리가 React 18 환경에서 특별한 업데이트를 요구하는지 다시 한번 살펴야 한다. 대부분의 인기 라이브러리는 이미 React 18 지원을 완료했지만, 일부 기능이 자동 배치나 Concurrent Rendering에 대응하지 않을 수 있다. 상태 관리 라이브러리가 내부적으로 ReactDOM.render를 여전히 사용하고 있다면, createRoot 방식으로 마이그레이션을 해야 하거나 패치를 기다려야 할 수도 있다. 문제점을 발견하면 우선 해당 라이브러리 문서를 확인하거나, GitHub 이슈에서 React 18 마이그레이션 사례를 참고하는 편이 좋다.

성능 측정 기법과 도구¶

React Profiler를 통해 렌더링 횟수와 시간을 측정하고, 과도한 재렌더링이 발생하는 컴포넌트를 찾아서 최적화할 수 있다. Concurrent Rendering이 가능해진 이후에는 렌더링이 분할되고 우선순위가 달라질 수 있으므로, 이전과 같은 지표만으로는 성능 문제를 정확히 진단하기 어려울 수도 있다. Profiler의 flame graph나 ranked chart를 꼼꼼히 살펴보고, Transition이 적용되는 구간에서 예상치 못한 빈번한 렌더링이 일어나지는 않는지 확인해야 한다. 여기서 성능 측정 결과가 좋지 않다면, useMemo나 useCallback, memo 등 최적화 기법을 한 번 더 적용하거나, 상태 분리를 통해 컴포넌트가 불필요하게 렌더되지 않도록 개선해야 한다.

병렬 데이터 페칭 검증¶

Concurrent Rendering과 Suspense가 결합되면, 병렬로 여러 데이터를 fetch한 뒤 컴포넌트를 순차적으로 렌더할 수 있게 된다. 이 과정에서 기존 코드가 fetch -> setState -> 렌더링 순서에만 의존하고 있었다면, 동시에 여러 데이터를 불러오는 로직에서 race condition이나 중복 렌더링이 발생할 수 있다. 예컨대 두 개 이상의 비동기 호출이 한꺼번에 시작되었을 때, 먼저 도착한 데이터가 두 번째 데이터를 덮어쓰는 상황이 일어날 수도 있다. 이런 문제가 있다면 Suspense 안쪽에서 각각의 fetch를 독립적으로 관리하거나, Transition API를 이용해 중요도가 낮은 데이터를 지연 처리함으로써 로직 충돌을 피할 수 있다.

마이그레이션 이후 주의 사항¶

마이그레이션은 라이브러리 업데이트 자체로 끝나는 것이 아니라, 전체 애플리케이션 동작을 다시 점검하는 과정이다. React 18의 Concurrent Rendering과 새 API들이 도입됨에 따라, 렌더링의 우선순위나 상태 업데이트 타이밍이 달라질 수 있음을 잊지 않아야 한다. 기존에 임시방편으로 setTimeout 등을 활용했던 부분이 의도치 않게 더 자주 혹은 더 적게 호출될 수도 있다. 심지어 UI 테스트 자동화 도구가 업데이트 타이밍 변화를 예측하지 못해 실패하는 사례도 발생할 수 있다. 이런 다양한 부작용을 막기 위해서는 마이그레이션 직후부터 철저히 기능별 리그레션 테스트를 진행하고, 사용자 트래픽이 많은 시점에 성능 모니터링을 집중적으로 해야 한다.

코드베이스 리팩토링¶

새로운 기능을 단순히 도입하는 것뿐 아니라, 과거 코드가 자동 배치나 Transition API를 염두에 두지 않고 작성된 부분을 단계적으로 손보는 것이 필요하다. 이를테면 콘솔 로그나 analytics 이벤트를 특정 시점에 맞춰 호출하는 로직이, 이제는 배치가 일어나기 전후나 transition 경계 내부에서 호출되어 더 적절한 타이밍으로 옮겨질 수 있다. 이 과정에서 Hook의 의존성 배열이나 TypeScript 타입 정의가 함께 바뀔 수도 있고, 실제로 코드가 간결해지거나 유지보수가 쉬워지는 이점도 생긴다. 리팩토링 작업은 전체적인 코드 품질 향상과 함께 React 18의 장점을 극대화하는 방향으로 계획하는 것이 좋다.

미래 버전과의 호환성¶

React 18에 도입된 Concurrent Rendering과 Server Components는 앞으로 더 확대될 가능성이 크다. 향후 React 19 이상 버전에서도 이 모델이 기본이 되며, 점차 레거시 기능들이 제거될 수 있다. 이 때문에 현재 단계에서 가능한 한 최신 API를 적극적으로 수용하고, 레거시 API에 의존하지 않는 구조로 코드를 작성해 두어야 이후 업데이트에서도 큰 어려움을 겪지 않을 것이다. TypeScript 타입 정의 역시 React 공식 팀의 변경 사항에 맞춰 함께 변할 수 있기 때문에, 정기적으로 @types/react 패키지나 기타 유형 정의 파일을 업데이트하고, 빌드나 테스트 과정에서 발생하는 에러를 신속히 처리하는 습관을 들이면 장기적으로 더 안정적인 코드를 유지할 수 있다.

마이그레이션 과정에서 이러한 사항을 꼼꼼하게 확인해 두면, React 18의 모든 기능을 온전히 활용할 수 있고, 애플리케이션의 성능과 코드 품질이 한층 향상될 것이다. Concurrent Rendering, Suspense, Transition API, Server Components 등은 단순한 기능 추가가 아니라 React의 철학과 아키텍처가 진화하는 큰 변곡점이므로, 모든 요소를 세심히 살펴보는 작업이 필요하다. 이를 통해 기존 버전보다 유연하고 반응성이 뛰어난 UI를 구현할 수 있게 될 것이다.
