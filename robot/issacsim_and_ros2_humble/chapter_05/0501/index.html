ROS의 진화: ROS1에서 ROS2로¶

ROS의 태동과 한계¶

ROS(Robot Operating System)는 오픈 소스 로봇 소프트웨어 프레임워크로 시작되어, 로봇 개발자 및 연구자가 복잡한 시스템을 빠르게 개발하고 확장할 수 있게 한 중요한 토대였다. 초기 ROS(ROS1)는 분산 환경에서 메시지 교환 방식을 단순화하고, 센서나 액추에이터 노드 등을 광범위하게 재사용할 수 있도록 통합된 인터페이스를 제공하였다. 그러나 ROS1은 중앙 매개체 역할을 하는 roscore와 특정 통신 방식을 이용함으로써, 일부 성능 및 유지보수상의 한계를 노출하였다. 예를 들어, 실시간성 보장이나 네트워크 환경 변화 대응, 보안 측면에서 개선이 필요하다는 의견이 커졌다.

ROS2는 이러한 한계를 극복하고자 DDS(Data Distribution Service)를 기반으로 설계되었다. ROS1이 roscore에 의존하여 중앙 집중형 구조를 가진 것과 달리, ROS2는 분산 시스템을 좀 더 유연하게 구성하며 다양한 네트워크 상황에 대응하도록 개선하였다. 또한 기업 및 연구 기관의 협업을 통해 멀티 플랫폼 지원, 실시간성, 보안, QoS(Quality of Service) 등 미래 지향적 요구사항을 적극 반영하였다.

DDS 기반 아키텍처¶

ROS2의 가장 큰 특징은 DDS 프로토콜을 통한 강력하고 유연한 통신 계층이다. DDS는 퍼블리셔-서브스크라이버 모델을 효율적으로 구현하여, 데이터 배포 및 미들웨어를 확장 가능하게 만든다. 노드 간 통신에서 각 노드가 직접 브로커 없이 메시지를 송수신할 수 있으며, 네트워크 구성이 복잡해도 DDS의 Discovery 기능을 통해 동적으로 노드를 식별하고 연결을 구성한다. 따라서 ROS2 노드들은 다양한 물리적 위치에 분산되어 있어도, QoS 정책을 통해 요구사항에 맞춰 데이터 교환 방식을 손쉽게 제어할 수 있다.

QoS 매커니즘¶

ROS2는 DDS의 QoS 정책을 그대로 활용하여, 메시지 신뢰도나 전송 모드를 상황에 따라 조절할 수 있는 유연성을 제공한다. 예를 들어 실시간 처리가 필요한 센서 데이터 스트리밍에는 적절한 히스토리 정책(History), 신뢰성 정책(Reliability) 등을 설정하여 신속하고 안정적인 데이터 전송을 달성한다. 반면 단순 상태 보고나 로깅용 데이터는 상대적으로 낮은 신뢰성 정책을 적용하여 자원 사용량을 절감할 수 있다. 이를 통해 같은 네트워크 안에서도 노드별로 다양한 특성을 가지는 로봇 애플리케이션을 안정적으로 운영할 수 있다.

멀티 플랫폼 지원¶

ROS2는 Windows, macOS, Linux뿐 아니라 임베디드 OS까지 폭넓은 지원을 목표로 설계되었다. C++ 14 표준 지원과 함께, 멀티 플랫폼을 위한 colcon 빌드 시스템이 도입되어 기존 ROS1의 catkin보다 확장성과 편의성을 높였다. 이러한 개선점은 로봇 개발 환경이 데스크톱 시스템을 넘어 모바일 디바이스, 클라우드 서버, 임베디드 컨트롤러 등 다양화되는 추세에 효과적으로 대응한다.

실시간성 보장¶

ROS2는 실시간 시스템 설계를 돕기 위해, DDS의 RTPS(Real-Time Publish Subscribe) 프로토콜을 활용하고, ROS2 Executor 레벨에서 콜백 처리 방식을 확장 가능하게 구현하였다. 또한 실시간 커널에서의 스케줄링을 고려하도록 rclcpp/rcl 등 ROS 레이어가 계층적으로 구성되어 있다. 적절한 RTOS(Real-Time Operating System) 혹은 프리엠프형 Linux 커널을 사용하면, 네트워크 레이턴시와 노드 스케줄링 지연을 줄여 사실상의 실시간 처리에 근접할 수 있다.

보안 강화¶

ROS2는 DDS Security 스펙을 확장하여 메시지 암호화, 인증, 무결성 검증 등을 제공한다. 기존의 ROS1에서 보안이 별도의 도구나 튜닝에 의존했던 것과 달리, ROS2는 표준 미들웨어 계층에서 보안을 직접 다룬다. 안전-critical한 로봇 시스템(예: 산업용 로봇, 자율 주행, 의료용 로봇 등)의 요구사항에 부합하기 위해, DDS를 통해 전송 계층부터 인증, 접근 제어를 구조적으로 적용할 수 있다.

ROS1에서 ROS2로의 마이그레이션¶

ROS1에서 ROS2로 이전하는 과정은 익숙한 catkin 빌드 환경에서 colcon 빌드 시스템으로 옮겨 가는 것 외에도, 메시지 정의, launch 파일, API 차이를 이해하고 반영해야 한다. rclcpp/rclpy 등 새롭게 도입된 추상화 계층은 기존 ROS1 노드를 이식하는 작업을 단순화하지만, DDS 기반 아키텍처 특성을 고려하여 QoS 설정이나 노드 통신 구조를 재점검하는 것이 좋다. 기존 ROS1 패키지가 ROS2로 공식 포팅된 사례가 많으므로, 해당 패키지를 참고하거나 ros1_bridge를 활용하여 호환성을 검토할 수 있다.

ROS1 브리지(ros1_bridge)는 ROS1 토픽과 ROS2 토픽 사이를 연결하는 중간 다리 역할을 수행한다. 이를 통해 점진적으로 ROS2로 전환하면서도 기존 ROS1 노드를 잠시 유지할 수 있다. 다만 QoS 설정, 로깅 방식 차이 등 일부 제한 사항이 있으므로, 지속적인 유지보수 환경에서는 가급적 ROS2 환경으로 빠르게 이주하는 것이 권장된다.

로봇 변환 행렬 예시¶

ROS2 역시 로봇 관절 제어, 3D 공간 내 로봇 위치 계산 등에 동원되는 다양한 수학적 연산을 사용한다. 간단한 예시로, 동차 변환행렬(Homogeneous Transformation Matrix)을 통한 좌표 변환을 살펴보면 아래와 같다. $$ \begin{align} \mathbf{T} = \begin{pmatrix} \mathbf{R} & \mathbf{d} ??라인피드??\ \mathbf{0} & 1 \end{pmatrix} \end{align} $$ 위 행렬에서 \(\mathbf{R}\)은 회전행렬이며, \(\mathbf{d}\)는 위치 벡터를 의미한다. 예를 들어 로봇 베이스 좌표계에서 말단 링크(end-effector) 좌표계를 구하는 데 이 행렬이 사용된다. ROS2에서 TF(transform) 패키지를 활용하면 이 변환 과정을 노드 간에 실시간으로 교환하여, 로봇 각 부위 및 센서, 액추에이터 사이의 좌표 관계를 동적으로 추적할 수 있다.

ROS2 아키텍처 개념도¶

flowchart LR subgraph ROS2 Architecture A[노드] --> B[토픽] B --> C[퍼블리셔/서브스크라이버] C --> D[DDS 계층] end

DDS 계층을 기반으로 노드 간 직접 통신을 하며, Discovery 과정에서 네트워크 상의 노드를 자동으로 인식한다. 필요에 따라 QoS 정책을 달리 적용하여, 안정성, 대역폭, 지연시간 등을 조정할 수 있다.

ROS2 빌드 및 실행 예시¶

ROS2 패키지를 빌드하고 실행하는 과정은 colcon을 사용하여 수행한다.

cd ~/ros2_ws
colcon build --symlink-install
. install/setup.bash
ros2 run my_package my_node

위와 같이 colcon 명령어를 사용하면, ROS2 패키지를 시스템에 맞추어 빌드하고 실행할 수 있다. 빌드 후 환경 변수 설정을 위해 install/setup.bash 스크립트를 소스(source)하면, ROS2 실행에 필요한 경로가 자동으로 등록된다.

ROS2 노드 예제 (C++)¶

아래는 ROS2에서 간단한 퍼블리셔-서브스크라이버 노드를 보여주는 예시이다. 토픽 이름은 "my_topic"으로 정하고, std_msgs의 String 메시지를 주고받는다.

#include <memory>
#include <string>
#include "rclcpp/rclcpp.hpp"
#include "std_msgs/msg/string.hpp"

class MyPublisherNode : public rclcpp::Node {
public:
    MyPublisherNode() : Node("my_publisher_node") {
        publisher_ = this->create_publisher<std_msgs::msg::String>("my_topic", 10);
        timer_ = this->create_wall_timer(
            std::chrono::milliseconds(500),
            std::bind(&MyPublisherNode::timerCallback, this)
        );
    }
private:
    void timerCallback() {
        auto msg = std::make_shared<std_msgs::msg::String>();
        msg->data = "Hello from Publisher";
        publisher_->publish(*msg);
        RCLCPP_INFO(this->get_logger(), "Publishing: '%s'", msg->data.c_str());
    }
    rclcpp::Publisher<std_msgs::msg::String>::SharedPtr publisher_;
    rclcpp::TimerBase::SharedPtr timer_;
};

class MySubscriberNode : public rclcpp::Node {
public:
    MySubscriberNode() : Node("my_subscriber_node") {
        subscription_ = this->create_subscription<std_msgs::msg::String>(
            "my_topic",
            10,
            std::bind(&MySubscriberNode::topicCallback, this, std::placeholders::_1)
        );
    }
private:
    void topicCallback(const std_msgs::msg::String::SharedPtr msg) {
        RCLCPP_INFO(this->get_logger(), "Subscribed: '%s'", msg->data.c_str());
    }
    rclcpp::Subscription<std_msgs::msg::String>::SharedPtr subscription_;
};

int main(int argc, char ** argv) {
    rclcpp::init(argc, argv);
    auto publisher_node = std::make_shared<MyPublisherNode>();
    auto subscriber_node = std::make_shared<MySubscriberNode>();
    rclcpp::executors::SingleThreadedExecutor executor;
    executor.add_node(publisher_node);
    executor.add_node(subscriber_node);
    executor.spin();
    rclcpp::shutdown();
    return 0;
}

위 예시에서 퍼블리셔 노드는 0.5초 간격으로 문자열 메시지를 발행하고, 서브스크라이버 노드는 해당 메시지를 받아 콘솔에 출력한다. DDS를 기반으로 노드 간에 직접 연결이 성립되며, 중앙 집중형 roscore 없이 자동으로 토픽을 발견하고 연결된다.

ROS1에서 ROS2로 넘어오면서 내부 구조, 빌드 시스템, 통신 방식이 모두 개선되었다. 다만 마이그레이션 시에는 새롭게 도입된 DDS QoS, launch 시스템, 빌드 도구(colcon), 노드 실행 방식 등을 충분히 숙지하고, 필요하다면 ros1_bridge나 공식 포팅 패키지를 참고하여 안정적인 전환을 진행하는 것이 바람직하다. 이러한 ROS2의 진화로 인해 다양한 플랫폼에서 확장성, 실시간성, 보안을 보장하는 새로운 로봇 시스템을 보다 체계적으로 개발할 수 있다.
