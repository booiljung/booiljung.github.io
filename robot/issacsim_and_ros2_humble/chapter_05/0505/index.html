ROS2 워크스페이스 구성과 빌드 시스템(Colcon)¶

ROS2 개발 환경에서는 워크스페이스를 체계적으로 구성하고, 빌드 시스템을 활용하여 패키지를 효율적으로 생성·관리하는 것이 매우 중요하다. ROS2 Humble 버전에서도 이 기본 원칙은 동일하며, 이를 위해 주로 Colcon 빌드 시스템이 사용된다. ROS2는 Catkin과 달리 Colcon이라는 도구를 통한 통합 빌드와 멀티 패키지 관리를 강조한다. 또한 워크스페이스는 여러 개의 패키지를 모아서 관리하고, 빌드 및 설치 과정에서 유연하게 확장할 수 있도록 지원한다.

ROS2 워크스페이스 구조¶

ROS2 워크스페이스는 보통 src, build, install, log 네 가지 주요 디렉터리로 구성되는 것이 일반적이다. src 디렉터리는 ROS2 패키지의 소스 코드를 담는 곳이며, build와 install 디렉터리는 Colcon을 통해 생성되는 빌드 결과물, 설치 파일 등을 보관한다. log 디렉터리는 빌드 및 실행 과정에서 생성되는 로그 파일들을 저장한다.

아래 예시는 my_ws라는 이름의 워크스페이스를 구성할 때의 디렉터리 구조를 단순화하여 나타낸다.

graph LR A(ROS2 Workspace: my_ws)-->B(src) A-->C(build) A-->D(install) A-->E(log)

ROS2에서 워크스페이스를 구성하는 핵심 아이디어는 하나의 워크스페이스 안에 여러 패키지를 동시에 빌드하고, 빌드된 결과물들을 하나의 환경으로 만들어서 활용한다는 점이다. 이를 통해 각 패키지가 의존하는 다른 패키지들과의 호환성 검증이나 설치 과정을 투명하게 진행할 수 있다.

Colcon 빌드 시스템 개념¶

Colcon은 ROS2 Humble 환경에서 기본적으로 채택된 멀티 패키지 빌드 도구이다. 각 패키지가 어떤 언어로 작성되었는지와는 무관하게, Colcon은 빌드 타입(예: ament_cmake, ament_python 등)에 맞추어 전체 워크스페이스 내 패키지를 효율적으로 빌드한다. 이때 패키지 간 의존 관계를 스스로 파악하여 올바른 순서대로 빌드 순서를 결정한다.

Colcon을 사용하면 워크스페이스 최상단에서 단 한 번 명령을 실행해도 src 안에 들어 있는 모든 패키지를 빌드할 수 있다. 또한 동시에 병렬 빌드를 수행하거나, 특정 패키지 또는 특정 빌드 타입만 선택적으로 빌드할 수도 있다. 빌드가 완료되면 install 디렉터리에 ROS2 실행 환경이 갖추어지고, 이를 소스(source)하여 원하는 노드나 라이브러리를 사용할 수 있게 된다.

예제 패키지 생성¶

my_ws라는 이름의 디렉터리에 src라는 폴더를 생성하고, 그 안에 C++ 예제 패키지를 만들어 보자. 이 예시는 ament_cmake 빌드 타입을 사용하는 가장 기본적인 구조의 ROS2 패키지를 보여준다.

mkdir -p my_ws/src
cd my_ws/src
ros2 pkg create --build-type ament_cmake my_cpp_pkg

위 명령을 실행하면 my_cpp_pkg라는 패키지 디렉터리가 생성되고, 패키지 선언 파일(package.xml)과 CMakeLists.txt가 자동으로 들어간다. 이후 Colcon을 통해 전체 워크스페이스를 빌드한다.

cd ../..
colcon build

빌드가 성공하면 my_ws 디렉터리 내부에 build, install, log 디렉터리가 새롭게 생성된다. 그리고 ROS2 환경을 사용하기 위해 아래 명령을 통해 설치 설정을 반영한다.

. install/setup.bash

이제 터미널에서 ros2 run my_cpp_pkg <실행 가능한 노드> 형태로 노드를 실행할 수 있다.

C++ 예제 노드 살펴보기¶

my_cpp_pkg 내부에 간단한 C++ 노드를 작성해 볼 수 있다. 아래 코드는 src 디렉터리 안에 minimal_node.cpp라는 이름으로 저장한다고 가정한다.

#include <rclcpp/rclcpp.hpp>

class MinimalNode : public rclcpp::Node
{
public:
  MinimalNode() : Node("minimal_node")
  {
    RCLCPP_INFO(this->get_logger(), "MinimalNode has been started.");
  }
};

int main(int argc, char** argv)
{
  rclcpp::init(argc, argv);
  rclcpp::spin(std::make_shared<MinimalNode>());
  rclcpp::shutdown();
  return 0;
}

이 코드를 빌드하려면 CMakeLists.txt에 해당 소스 파일을 추가해준다. 간단히 add_executable과 ament_target_dependencies를 통해 rclcpp를 링크해 주면 된다. Colcon build를 다시 수행하면 minimal_node라는 실행 파일이 생성되고, ros2 run my_cpp_pkg minimal_node로 실행할 수 있다.

Workspace Overlay¶

ROS2에서는 하나의 워크스페이스에 국한되지 않고, 여러 개의 워크스페이스를 중첩(Overlay)하여 사용할 수 있다. 이는 상위 워크스페이스(기존에 구축된 환경)를 유지하면서, 하위 워크스페이스(개발 중인 패키지들)를 덧붙이는 방식이다. 상위 워크스페이스에서 제공되는 라이브러리나 실행 파일을 그대로 사용하고, 필요한 부분만 추가 또는 교체하며 개발할 수 있다.

예를 들어 /opt/ros/humble 경로에 설치된 ROS2 코어 라이브러리를 상위 워크스페이스로 보고, my_ws를 아래에 두어 my_ws에서 만든 패키지를 우선순위로 사용하고 싶다면 다음과 같은 방식으로 순서를 구성한다.

source /opt/ros/humble/setup.bash
cd my_ws
colcon build
. install/setup.bash

이렇게 하면 ROS2 기본 환경에 my_ws 내의 패키지들을 덧대어 사용하게 된다.

빌드 환경 변수¶

Colcon 빌드를 수행할 때, 환경 변수 설정도 중요하다. ROS2에서는 CMAKE_PREFIX_PATH, AMENT_PREFIX_PATH 등을 통해 install 디렉터리를 찾는다. Colcon은 빌드 시점에 이 정보를 자동 설정해주지만, 빌드 후에 ros2 명령어를 실행하려면 해당 워크스페이스를 매번 source해 주어야 한다. 이러한 환경 변수를 적절히 관리하면 패키지 종속성을 추적하고, 라이브러리를 올바르게 로드하는 것이 수월해진다.

결론¶

ROS2 Humble에서 워크스페이스를 구성하고 Colcon 빌드 시스템을 활용하는 방법은 ROS2 개발의 핵심이다. src 디렉터리에 패키지를 관리하고, build 및 install 디렉터리를 통해 빌드 결과물을 통합하며, 환경 변수를 적절히 설정하여 여러 워크스페이스를 겹쳐서 사용하면 복잡한 프로젝트도 유연하게 개발할 수 있다. 또한 Colcon은 모든 패키지의 의존성을 자동으로 탐색하고, 빌드 순서를 효율적으로 결정해주므로 대규모 ROS2 프로젝트에서도 직관적이고 간단한 빌드 관리가 가능하다.
