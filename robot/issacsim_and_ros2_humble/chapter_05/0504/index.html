ROS2 Humble은 로보틱스 분야에서 확장성과 안정성을 제공하기 위해 만들어진 ROS(Robot Operating System)의 최신 버전 중 하나로서, 이전 버전보다 개선된 기능과 장기 지원(LTS)을 제공한다. 로보틱스 시스템을 개발할 때 필수적인 오케스트레이션, 메시지 송수신, 하드웨어 추상화, 다양한 통신 미들웨어(RMW) 지원 등을 유연하게 구성할 수 있다. 특히 실시간성 강화, 보안 기능 향상, 분산 환경에서의 확장성 등을 목표로 발전해 왔으며, 산업 및 학술 환경 전반에서 폭넓게 활용되고 있다.

ROS2 Humble 개요¶

ROS2 Humble은 Ubuntu 22.04(Jammy Jellyfish)를 주요 목표 플랫폼으로 삼아 개발되었다. 따라서 ROS2 Humble이 공식적으로 지원하는 여러 운영체제 중에서도 우분투 22.04 환경에서 가장 안정적이고 간편한 설치 과정을 제공한다. 그러나 Windows 10/11, macOS 등에서도 빌드와 실행이 가능하도록 준비되어 있으며, 이러한 환경에서 로보틱스 애플리케이션을 구동할 수 있도록 광범위한 호환성을 갖춘다. Humble에서는 보안 수준이 높아진 DDS 통신 옵션을 지원하며, 아키텍처 수준에서 멀티 OS 지원과 멀티 언어(특히 C++와 Python)의 활용성을 더욱 강화한다. 또한 이전 버전 대비 다양한 성능 최적화가 적용되어 있고, 노드 라이프사이클(Lifecycle), 네임스페이스(Namespaces) 등 ROS2 핵심 기능에 대한 문서화와 유지보수가 더욱 견고해졌다.

다중 노드를 사용하는 복잡한 로보틱스 시스템에서 성능을 최대화하기 위해서는, 노드 간 통신 지연이나 데이터 충돌을 최소화해야 한다. 이를 위해 DDS 계층에서 QoS 설정을 세밀하게 조정하거나, RMW를 변경하여 성능 프로파일을 달리하는 방법이 제공된다. Humble에서는 rmw_fastrtps, rmw_cyclonedds, rmw_connextdds 등 다양한 RMW를 지원한다. 배포 환경에 따라 원하는 RMW를 선택함으로써, 작은 임베디드 기기부터 고성능 산업용 장비까지 폭넓게 시스템을 확장 가능하다.

공식 지원 운영체제 및 호환성¶

ROS2 Humble은 배포판마다 빌드 스크립트와 호환성 테스트를 진행한 뒤 릴리스된다. Ubuntu 22.04가 기본이지만, 사용 환경에 따라 Windows나 macOS, 그리고 임베디드 보드를 위한 크로스 컴파일 방식도 고려할 수 있다. 아래에서는 ROS2 Humble이 자주 사용되는 운영체제를 간략히 살펴본다.

Ubuntu 22.04 (Jammy Jellyfish)¶

Ubuntu 22.04는 ROS2 Humble의 공식 권장 플랫폼으로서, apt 패키지 관리자를 이용하여 쉬운 설치가 가능하다. Ubuntu 22.04는 장기 지원(LTS) 버전이므로, 로봇 운영 환경으로서 안정성을 중시할 때 가장 적합하다. 패키지 인덱스 및 보안 업데이트가 원활하게 지원되므로, ROS2를 장기간 운용하려는 프로젝트에도 유리하다. 또한 Humble의 모든 종속성 라이브러리와 도구 체인이 우분투 22.04를 기준으로 준비되어 있기 때문에, 의존성 관리나 버전 충돌 문제를 최소화하면서 개발을 진행할 수 있다.

Windows 10/11¶

Windows 환경에서도 ROS2 Humble을 사용할 수 있으며, Visual Studio를 활용해 C++ 기반 노드를 작성하고, PowerShell 환경에서 ROS2 명령어를 실행할 수 있다. Windows 패키지 설치 관리자(Choco) 또는 바이너리 압축 파일을 이용하여 쉽게 설치할 수 있고, IoT 디바이스나 산업용 소프트웨어를 Windows 환경에서 개발 중이라면 유용하다. 다만 일부 패키지는 아직 Linux 중심으로 최적화되어 있어, Windows로 포팅되지 않은 기능이 있을 수 있다는 점에 유의해야 한다. 또한 네트워크 인터페이스 설정이나 DDS 통신 설정이 Linux 대비 다소 복잡해질 수 있으므로, 설치와 설정 중에 세심한 검토가 필요하다.

macOS¶

macOS에서도 Homebrew 등을 이용하면 ROS2 Humble 설치가 가능하다. 개발 머신으로 Mac을 선호하는 연구자나 엔지니어라면, Python 혹은 C++ 노드를 macOS 환경에서 작성하고 빌드할 수 있으며, 간단한 시뮬레이션 정도는 로컬에서 테스트 가능하다. 그러나 고성능 GPU 자원을 활용하는 로봇 시뮬레이터(예: Isaac Sim)를 macOS에서 직접 구동하기에는 제약이 있을 수 있다. 이러한 경우는 Docker나 원격 서버 환경, 혹은 듀얼 부팅 등으로 Linux 환경을 별도 마련해야 할 수도 있다.

하드웨어 요구사항¶

ROS2 Humble은 소형 임베디드 보드에서부터 대규모 산업용 서버까지 다양한 하드웨어를 지원하도록 설계되었다. 그러나 일반적으로 64비트 아키텍처(x86_64)와 ARM64 아키텍처에서 정상 동작하며, 32비트 아키텍처 지원은 이전 ROS2 버전들에서도 제한적이었다. 더 높은 성능의 GPU나 CPU 자원이 필요한 시뮬레이션 및 데이터 처리 노드를 구동하는 경우, RAM 용량이나 GPU 사양을 충분히 고려해야 한다. Isaac Sim과 연동하여 3D 시뮬레이션을 수행할 계획이라면, NVIDIA GPU 및 CUDA 라이브러리에 대한 호환성을 신중히 확인해야 하며, 이를 만족하기 위해서는 Ubuntu 기반 환경이 사실상 필수적이다.

복잡도가 높은 로보틱스 시스템에서는 다중 센서 데이터를 실시간으로 처리해야 하므로, 멀티코어 CPU와 충분한 메모리가 권장된다. 예를 들어 LiDAR, 카메라, IMU 등을 모두 연결하고 대용량 데이터를 동기화하면서, 네트워크 상 여러 노드와 동시에 통신하려면, 최소 4코어 이상의 CPU와 8GB 이상의 RAM을 갖추는 것이 좋다. 네트워크 트래픽이 많이 발생하는 경우에는 유선 LAN 환경을 구축하거나, 무선이라도 안정적인 Wi-Fi 프로토콜을 사용하는 것이 데이터 누락을 줄이는 데 도움이 된다.

설치 방법 개요¶

ROS2 Humble을 설치하는 방법은 크게 바이너리 패키지를 이용하는 방식과 소스 코드를 직접 빌드하는 방식, 그리고 Docker 이미지를 활용하는 방법으로 나눌 수 있다. 각 방법은 설치 난이도와 시스템 최적화 가능 범위, 그리고 유지보수 편의성 등에서 차이가 있다. 일반적으로 우분투 22.04에서 로컬 개발 및 테스트만 한다면 바이너리 설치가 가장 간편하며, 특정 커스텀 기능이 필요하거나 ROS2의 내부 동작을 수정하고자 한다면 소스 빌드가 유리하다. Docker 이미지는 통합 개발 환경을 빠르게 구성하거나, 개발 팀원 간 동일한 환경을 공유하려는 목적에 활용할 수 있다.

바이너리 패키지 설치¶

Ubuntu 22.04라면 apt를 통해 ROS2 Humble의 핵심 패키지부터 데스크톱 전체 기능을 포함하는 metapackage까지, 다양한 구성을 쉽게 설치할 수 있다. 터미널에서 apt를 이용해 설치를 진행하면 관련 의존성 패키지들이 자동으로 풀 다운되어, 설치 과정을 단순화할 수 있다. 설치 후에는 환경 설정 스크립트인 setup.bash나 setup.zsh를 ROS2 작업 공간(workspace)에 맞게 불러와서, ros2 명령어나 colcon 빌드 도구 등을 즉시 사용할 수 있다. 이 방식은 대부분의 일반 사용자에게 가장 추천되는 접근 방식이며, ROS2 공식 문서에서 제시된 안정 버전을 이용할 수 있어 비교적 오류 발생 가능성이 낮다.

소스 컴파일¶

ROS2 Humble을 다양한 옵션으로 커스터마이징하거나, 최신 개발 버전을 추적해보고 싶다면 소스 코드를 직접 빌드하는 방식을 고려할 수 있다. Git에서 ros2.repos 파일을 가져와서 필요한 저장소를 선택적으로 내려받고, colcon 빌드 도구를 사용해 전체를 컴파일한다. 빌드 과정에서 \(-DBUILD_TESTING=OFF\)와 같은 CMake 옵션을 조정하거나, 특정 RMW만 활성화/비활성화하여 최적화할 수 있다. 시스템 종속 라이브러리(예: OpenCV, PCL, Fast DDS 등)에 대한 버전을 직접 관리하거나, 디버그 심볼을 활성화한 상태에서 테스트를 진행하려면 소스 빌드가 적합하다. 그러나 초기 세팅과 종속성 충돌 해결에 추가 시간이 소요될 수 있으며, 빌드 과정에서 CPU와 메모리를 상당히 소모할 수 있으므로, 여유 자원이 많은 환경에서 진행해야 한다.

Docker 환경 이용¶

로보틱스 소프트웨어 개발은 많은 종속 라이브러리를 필요로 하고, 다양한 버전 충돌이나 OS 설정 차이로 인해 문제가 발생하기 쉽다. Docker 이미지는 이러한 문제를 줄이면서, 팀원이 동일한 컨테이너 이미지를 공유하거나 클라우드에서 쉽게 배포할 수 있는 이점을 제공한다. NVIDIA GPU를 활용해야 하는 경우에는 nvidia-docker를 사용하여 그래픽 자원을 컨테이너 내부에서 활용할 수 있다. ROS2 Humble 전용 Docker 이미지가 이미 준비되어 있으므로, 이를 기반으로 자신의 프로젝트가 요구하는 추가 패키지나 설정을 Layer로 쌓아 맞춤형 이미지를 빌드할 수 있다. 컨테이너 내부에서 colcon을 실행하거나 RMW를 설정하고, 컨테이너 외부의 호스트 네트워크와 DDS 통신 포트를 잘 매핑해주면, 로컬 개발 환경을 깔끔하게 유지하면서도 로보틱스 애플리케이션을 실행할 수 있다.

추가 환경 설정¶

ROS2 Humble을 설치한 뒤에는 작업 공간 설정, 빌드 도구 활용, 노드 간 통신 설정 등을 세부적으로 조정해야 한다. 기본 상태에서도 ros2 run, ros2 topic 같은 명령어로 로컬 노드를 구동하고 토픽을 확인할 수 있지만, 실제 프로젝트 수준으로 확장하려면 추가 설정이 필수적이다.

colcon과 워크스페이스¶

ROS2에서는 catkin 대신 colcon이라는 빌드 도구를 주로 사용한다. colcon은 여러 패키지를 병렬로 빌드하고, 의존성 그래프를 자동으로 분석해서 빌드 순서를 결정한다. 작업 공간(workspace)은 colcon을 통해 빌드되는 모든 패키지의 루트 디렉터리로서, 일반적으로 src 디렉터리에 소스 코드를 두고, 빌드 산출물은 build, install, log 디렉터리에 배치된다. Humble 환경에서 작업 공간을 효율적으로 다루기 위해서는 setup.bash를 반복적으로 로드해야 하며, 여러 개의 workspace를 서로 다른 경로에 구성해두고 우선순위를 조절하는 방식도 가능하다. 예를 들어 외부 라이브러리나 서드파티 패키지를 한 workspace로 관리하고, 실제 프로젝트 코드를 또 다른 workspace로 구성하는 방식도 있다.

ROS2 build 환경 변수¶

ROS2 Humble에서 환경 변수를 설정할 때는 \(ROS_DOMAIN_ID\), \(RMW_IMPLEMENTATION\), \(COLCON_DEFAULTS_FILE\) 등을 고려해야 한다. \(ROS_DOMAIN_ID\)는 DDS 네트워크에서 서로 충돌 없이 구분할 수 있는 논리적 영역을 정하는 데 사용된다. 여러 대의 로봇이나 여러 연구팀이 같은 물리 네트워크를 공유하는 상황에서, 각 팀마다 고유한 도메인 ID를 부여하면 노드 간 메시지가 뒤섞이지 않는다. \(RMW_IMPLEMENTATION\)은 사용하는 RMW 계층을 지정하는 변수로서, 예를 들어 \(rmw_fastrtps_cpp\)나 \(rmw_cyclonedds_cpp\) 등을 명시할 수 있다. 시스템 성능이나 보안 요구사항에 따라 적절한 RMW를 선택할 수 있으므로, 이를 적절히 설정해야 한다.

RMW와 DDS 설정¶

DDS 계층은 ROS2에서 노드 간 통신의 핵심 역할을 담당한다. ROS2 Humble은 Fast DDS, Cyclone DDS, Connext DDS 등 다양한 DDS 구현체를 지원한다. Fast DDS는 일반적으로 ROS2에서 가장 널리 사용되며, 높은 커뮤니티 지원과 빠른 업데이트 주기가 장점이다. Cyclone DDS는 임베디드 환경까지 포함해 매우 다양한 플랫폼에 대응하고, Connext DDS는 상업용 환경에서 기술 지원과 보안 기능이 강력하다. DDS를 선택하면 메시지 전달 품질, 세분화된 QoS 정책, 보안 설정 등을 비교적 자유롭게 설정할 수 있다. \(rmw_fastrtps_cpp\)를 사용하는 경우, QoS 정책을 변경해 신뢰성 모드, 히스토리 깊이, 지연 시간 등에 대해 다양한 파라미터 조절이 가능하다. 이를 통해 네트워크 상태가 좋지 않은 환경에서도 적절한 성능과 신뢰도를 동시에 확보할 수 있다.

Isaac Sim과의 연동 고려사항¶

Isaac Sim은 NVIDIA Omniverse 기반의 고성능 로보틱스 시뮬레이터로, ROS2 Humble과 연동해 물리 엔진 및 센서 시뮬레이션을 수행할 수 있다. 이때는 Isaac Sim이 요구하는 GPU 사양과 드라이버를 먼저 충족해야 하며, CUDA 및 NVIDIA-Docker가 설치된 환경에서 정상적인 가속을 보장해야 한다. Isaac Sim 컨테이너에서 ROS2 Humble의 토픽을 퍼블리시하거나, Isaac Sim에서 센서 데이터를 ROS2 노드로 발행하도록 구성하려면, DDS 통신 포트와 네트워크 브리지 설정이 반드시 올바르게 설정되어야 한다. 또한 Isaac Sim이 Windows용 버전을 제공하기는 하지만, 대부분의 고급 기능은 Linux 환경에 최적화되어 있고, ROS2와의 자연스러운 호환성도 우분투 22.04에서 가장 우수하게 나타난다.

네임스페이스와 노드 구성¶

ROS2 Humble에서는 하나의 프로세스 내에서 여러 노드를 구동하거나, 여러 프로세스에 걸쳐 분산된 노드를 네임스페이스로 구분해 관리하는 것이 가능하다. 대규모 로봇 시스템에서는 센서 그룹, 제어 로직 그룹, 시뮬레이션 그룹 등으로 나누어 노드를 구성하고, 각각의 네임스페이스에 속한 토픽 이름이나 서비스 이름을 적절히 정의한다. 이를 통해 토픽 충돌을 방지하고, 노드 계층 구조를 직관적으로 파악할 수 있다. 예컨대 실제 로봇과 가상 로봇의 토픽 이름을 구분하고 싶다면, 네임스페이스를 /real_robot, /sim_robot와 같이 설정해 두면 된다. 특히 Isaac Sim과 물리 로봇을 동시에 통합 테스트하는 경우, 네임스페이스 설정이 매우 중요하다.

보안 및 인증서 설정¶

ROS2에서는 DDS 계층을 이용해 보안 프로토콜을 활성화할 수 있는데, 이는 노드 간 암호화 통신이나 데이터 무결성 검증 등을 수행하기 위함이다. 실제 산업 현장에서 로봇을 인터넷이나 외부 네트워크에 연결해야 하는 상황이라면, DDS 보안(ROS2 SROS2)을 사용해 인증서 기반 암호화를 적용하는 방안이 권장된다. 보안을 활성화하면 메시지 주고받기에 추가 오버헤드가 발생하지만, 잠재적인 해킹이나 데이터 노출 위험을 크게 줄여준다. ROS2 Humble에서는 이전 버전에 비해 보안 구성 절차가 간소화되고 문서가 잘 정리되어 있지만, 보안 인증서 생성, 정책 파일 작성, 각 노드에 대한 권한 설정 등은 여전히 주의가 필요하다.

--- 및 확장 방향

ROS2 Humble은 다양한 운영체제와 하드웨어에서 안정적으로 동작하며, Isaac Sim 같은 시뮬레이션 환경과도 원활히 연동되는 강력한 로보틱스 프레임워크이다. 설치를 진행하기 전에 시스템 요구사항, 운영체제 호환성, DDS 구성, 네트워크 환경, GPU 드라이버 등을 미리 점검해두면 추후 문제 발생을 줄일 수 있다. 또한 프로젝트 요구사항에 맞춰 바이너리 설치, 소스 빌드, Docker 활용 중 하나를 선택하여 환경을 구축할 수 있다. 실제 로봇 개발에서 확장성과 유지보수를 모두 고려한다면, ROS2 Humble의 생태계가 제공하는 패키지들과 연동하여 효율적인 로봇 애플리케이션을 구현하는 것이 가능하다. Isaac Sim과의 연동을 통해 가상 환경에서 안전하게 실험하고, 이후 물리 환경으로 전환하는 개발 프로세스 역시 점점 표준화되어 가고 있다. 로봇 제어, 자율주행, 매니퓰레이션 등 다양한 응용 분야에서의 성공적인 활용을 위해서는, ROS2 Humble의 설치와 시스템 요구사항을 철저히 파악하고 환경을 올바르게 구성하는 것이 무엇보다 중요하다.
