ROS2 아키텍처: DDS, QoS, 노드 개념¶

DDS 통신 구조¶

ROS2는 DDS(Data Distribution Service)를 기반으로 통신을 수행한다. DDS는 퍼블리셔(Publisher)-서브스크라이버(Subscriber) 모델을 구현하여 노드 간 데이터 교환을 지원한다. 이때 네트워크 상에서 중간 브로커(Broker)가 없는 분산(distributed) 방식으로 작동하므로, 각 노드는 직접 상대 노드의 존재 여부를 인지하고 메시지를 주고받는다. 결과적으로 고속의 실시간 통신이 가능하며, 노드의 가용성과 신뢰도를 높일 수 있다.

DDS 표준은 여러 구현체가 존재한다. Cyclone DDS, Fast DDS, RTI Connext DDS 등의 다양한 벤더별로 구현체를 선택할 수 있어, 애플리케이션 특성에 맞추어 적절한 DDS를 활용할 수 있다. ROS2는 이러한 DDS 구현체 위에 추상 레이어(RMW, ROS Middleware Interface)를 두어 DDS와의 직접적인 상호 작용 대신, 통일된 인터페이스를 제공한다.

다음 예시는 DDS 기반 노드 간 관계를 간단히 표현한 Mermaid 다이어그램이다.

QoS(품질 서비스) 개념¶

DDS 통신에서는 Quality of Service(QoS) 파라미터를 통해 메시지 전송 품질을 세밀하게 제어할 수 있다. 신뢰 모드(Reliability), 히스토리(History), 내구성(Durability), 데드라인(Deadline) 등 다양한 QoS 설정이 가능하다. 예를 들어 신뢰 모드를 Reliable로 설정하면 재전송 메커니즘을 통해 메시지 손실 없이 통신하고, Best Effort 모드에서는 재전송 없이 가능한 한 빨리 메시지를 전송한다.

QoS 파라미터 조절은 시스템 요구사항에 따라 달라진다. 실시간성이 중요한 로보틱스 분야에서는 특정 노드 간 트래픽에 Best Effort 방식을 쓰고, 핵심 제어 메시지는 Reliable로 설정하여 손실률을 낮추는 방식으로 구성할 수 있다. 송신 측 노드와 수신 측 노드의 QoS 정책이 호환되지 않으면, 해당 토픽(topic)에 대한 통신이 이루어지지 않으므로 QoS 설정 시 호환성을 고려해야 한다.

ROS2에서 QoS를 설정할 때는 프로파일(profile)을 지정하거나, 동적으로 파라미터를 변경할 수도 있다. 노드 생성 시 또는 퍼블리셔/서브스크라이버 생성 시 QoS 옵션을 인자로 주어 적용한다.

아래 간단한 수식을 통해 데이터를 전송할 때의 평균 지연(평균 전달 시간)을 나타낼 수 있다. DDS 통신에서 신뢰 모드나 버퍼링 전략에 따라 지연이 달라지므로, 시스템 설계 시 이를 계산하거나 추정하는 과정을 거친다. $$ \begin{align} t_\mathrm{avg} = \frac{\sum_{i=1}^{N} t_\mathrm{i}}{N} ??라인피드??\ \end{align} $$ 여기에서 tit_\mathrm{i}는 ii번째 전송 성공에 걸린 시간이며 NN은 전체 전송 횟수다. QoS 파라미터는 이 tit_\mathrm{i}의 분포와 편차를 결정하는 주요 인자로 작용한다.

노드(Node) 개념과 통신 흐름¶

ROS2에서 노드는 DDS 위의 추상화 계층을 통해 퍼블리셔, 서브스크라이버, 서비스 서버, 서비스 클라이언트 등 다양한 엔티티(entity)를 구성한다. 각 노드는 서로 독립된 프로세스로서, 특정 목표를 수행하기 위해 로직을 포함하고 메시지를 교환한다. ROS1에서 단일 마스터(Master)를 통해 통신 정보를 등록하고 조회했다면, ROS2는 DDS의 분산된 discovery 메커니즘에 의존한다.

노드는 서로 다른 머신에서 실행해도 자동으로 검색하고 통신할 수 있다. 이는 DDS의 discovery 프로토콜이 네트워크 상에서 동작하여, 노드가 새로 나타나거나 사라지는 상황을 탐지하고 동적으로 업데이트하기 때문이다. 따라서 로봇 센서 노드, 제어 노드, 시뮬레이션 노드 등 다양한 노드가 네트워크 환경에서 자유롭게 연결되어 시스템 전체를 구성한다.

ROS2 Humble에서 노드를 생성하고 퍼블리셔와 서브스크라이버를 등록하는 예시 코드를 간단히 Python으로 살펴보면 다음과 같다.

import rclpy
from rclpy.node import Node
from std_msgs.msg import String
from rclpy.qos import QoSProfile, ReliabilityPolicy

class SimplePublisher(Node):
    def __init__(self):
        super().__init__('simple_publisher')
        qos_profile = QoSProfile(depth=10)
        qos_profile.reliability = ReliabilityPolicy.RELIABLE
        self.publisher_ = self.create_publisher(String, 'chatter', qos_profile)
        self.timer = self.create_timer(0.5, self.timer_callback)

    def timer_callback(self):
        msg = String()
        msg.data = 'Hello ROS2'
        self.publisher_.publish(msg)
        self.get_logger().info('Publish: "%s"' % msg.data)

class SimpleSubscriber(Node):
    def __init__(self):
        super().__init__('simple_subscriber')
        qos_profile = QoSProfile(depth=10)
        qos_profile.reliability = ReliabilityPolicy.BEST_EFFORT
        self.subscription_ = self.create_subscription(
            String, 'chatter', self.listener_callback, qos_profile)

    def listener_callback(self, msg):
        self.get_logger().info('I heard: "%s"' % msg.data)

def main(args=None):
    rclpy.init(args=args)
    publisher_node = SimplePublisher()
    subscriber_node = SimpleSubscriber()
    try:
        rclpy.spin(publisher_node)
        rclpy.spin(subscriber_node)
    except KeyboardInterrupt:
        pass
    publisher_node.destroy_node()
    subscriber_node.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()

여기서 간단한 노드 두 개를 생성하고, 하나는 퍼블리셔로서 "chatter" 토픽으로 메시지를 전송하며, 다른 하나는 서브스크라이버로서 같은 토픽을 구독한다. 퍼블리셔는 신뢰 모드를 Reliable로, 서브스크라이버는 Best Effort로 설정했으므로 QoS 정책이 맞지 않아 실제 연결이 이루어지지 않을 수도 있다. 이를 의도적으로 다르게 설정하여 QoS 불일치를 테스트해볼 수 있다.

ROS2 아키텍처에서 노드의 설계와 DDS 기반 통신 구조 이해는 매우 중요하다. DDS의 분산 구조는 ROS2에 뛰어난 확장성과 실시간성을 부여하며, QoS 세분화를 통해 다양한 서비스 품질 요구에 대응할 수 있다. 이러한 유연성은 로보틱스나 시뮬레이션 환경에서 복잡한 시스템을 구성할 때 큰 강점으로 작용한다.
