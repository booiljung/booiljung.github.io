ROS2 도입의 역사적 배경¶

ROS는 로봇 시스템을 구성하는 여러 요소들을 유연하고 확장 가능하게 연결하는 목적으로 시작되었다. 초기 ROS1은 연구 및 프로토타이핑 관점에서 많은 장점을 가지고 있었지만, 산업계에서 요구하는 실시간성이나 보안성, 분산 환경에서의 확장성 같은 요소들이 부족하다는 평가가 있었다. 이러한 문제를 극복하기 위해 ROS2가 기획되었고, 지속적인 버전 업그레이드를 거치면서 개선된 아키텍처와 성능을 선보여 왔다. ROS2 Humble은 바로 이러한 ROS2 계열에서 추가적인 기능 개선과 안정화를 통해 새로운 이정표가 되었다고 할 수 있다.

ROS2 Humble이 등장하기 전에도 Foxy나 Galactic, Rolling 등 다양한 배포판이 존재했지만, Humble은 장기 지원(LTS) 버전 중 하나로서 상대적으로 긴 기간 동안 지속적인 유지보수를 받는다. 연구 현장과 산업 현장에서 ROS2 기반 솔루션이 자리를 잡아감에 따라 안정성과 장기 지원이 더욱 중요해졌는데, 이러한 요구를 충족하기 위한 전략적 버전이 바로 ROS2 Humble이다. Foxy 이후의 Galactic이나 Rolling에서는 아직 실험적인 기능이 활성화되는 경우가 많았고, API 변화가 빈번히 발생하여 프로젝트 유지보수에 부담이 컸다. Humble은 이전 버전에서 축적된 피드백과 현실적인 요구사항을 반영해 안정적이고 폭넓은 호환성을 지닌다는 점에서 주목을 받게 되었다.

ROS2는 DDS(데이터 분산 서비스)를 바탕으로 통신을 제공하는데, ROS1 시절의 마스터 서버 의존적 구조와 비교할 때 이미 상당히 분산화되고 유연한 방식으로 거듭났다. 그러나 DDS 구현체마다 세부 설정이 달라 성능 편차가 발생하거나, 예기치 못한 네트워크 환경에서 호환성 문제가 일어나는 사례가 보고되어 왔다. Humble에서는 특정 DDS 구현체에 대한 지원성이 한층 개선되어 네트워크 계층과 RMW(Robot Middleware) 계층 간의 호환성 테스트가 더욱 강화되었고, 배포판 차원에서도 실시간 성능 검증이 체계적으로 이루어졌다.

노드 구조와 라이프사이클 관리의 강화¶

ROS2의 핵심은 노드가 분산 환경에서 어떻게 구성되고, 주어진 작업을 어떻게 수행하고, 종료 혹은 재시작까지의 과정이 얼마나 유연하고 안전하게 관리되느냐에 달려 있다. ROS2 Humble에서는 기존 라이프사이클 노드의 개념이 더욱 확장되었으며, 노드 상태 전이 과정을 직관적으로 제어할 수 있도록 여러 API와 툴이 개선되었다. 이러한 라이프사이클 관리는 로봇 시스템이 실시간으로 동작하다가 문제가 생겼을 때 재시작 과정을 간단하게 처리하거나, 특정 조건이 발생했을 때 노드 상태를 특정 모드로 전환하는 등의 작업을 수월하게 만든다.

라이프사이클 노드는 노드가 unconfigured, inactive, active, finalized 상태 등 여러 상태를 오가며 동작하는 구조를 갖는다. 이를 통해 시스템에서 노드를 안전하게 구성하고, 활성화하는 과정을 명시적으로 표현할 수 있다. ROS2 Humble에서 더욱 선명해진 라이프사이클 관리는 자율주행 로봇, 협동 로봇과 같이 미션 크리티컬한 환경에서 연속적인 작업 수행과 복구 전략을 유연하게 적용하도록 돕는다.

실시간 성능과 보안 요구사항 대응¶

ROS2를 도입하는 가장 큰 이유 중 하나는 실시간 통신에 대한 니즈가 높기 때문이다. 로봇은 센서에서 빠르게 데이터를 수집하고, 이를 분석하여 제어 명령을 내리는 과정을 짧은 시간 안에 반복해야 한다. ROS1에서의 TCP 기반 통신은 프로토타입 단계에서는 충분히 유용했지만, 하드 실시간은 물론 소프트 실시간 요구사항에도 일부 한계가 존재했다. ROS2 Humble에서는 DDS를 활용하되, 실시간 성능이 특히 요구되는 상황에서 활용할 수 있는 다양한 설정값과 프로파일이 확충되었다.

DDS 계층에서 QoS 정책을 더 세밀하게 적용할 수 있도록 개선된 점도 눈여겨볼 만하다. 예를 들어, 납기 한계(deadline), 샘플 이력(history) 및 우선순위(priority) 같은 설정을 노드나 토픽마다 세부적으로 지정할 수 있게 됨으로써 타이밍 제약이 중요한 태스크와 그렇지 않은 태스크를 구분하여 자원을 효율적으로 배분한다. 이전 버전에서는 일부 DDS 벤더에서만 지원되던 확장 기능들이 Humble에서 표준화된 형태로 제공되는 경우가 많아져, 실시간 요구사항을 갖는 프로젝트에서 호환성 문제를 덜 겪게 되었다.

보안 측면에서도 DDS Security 스펙을 좀 더 간편하게 활용할 수 있도록 ROS2 레벨에서의 설정 인터페이스가 정돈되었다. 인증서 및 키 관리가 보다 직관적으로 개선되어, 네트워크 노출이 많은 로봇 시스템에서도 적절한 암호화를 통해 안전하게 노드 간 통신을 수행할 수 있다.

QoS 정책 확장과 상호 운용성¶

ROS2는 QoS(서비스 품질) 설정을 통해 노드 간의 통신 신뢰도, 지연, 안정성을 제어한다. 이를테면 Reliability, Durability, Deadline과 같은 요소들을 조절하여 토픽 전송 방식을 유연하게 결정할 수 있다. Humble은 기존 QoS 프로파일을 유지하면서도 새로운 정책이 추가되거나, 기존 정책들이 더욱 세밀하게 설정될 수 있도록 확장되었다. 실험적인 기능으로 간주되던 정책들이 이제는 안정적인 API로 자리 잡아 다양한 요구사항에 대응하기가 쉬워졌다.

특히 멀티 로봇 시스템에서 각 로봇이 다수의 센서를 장착하고, 이 센서들이 고속으로 데이터를 전송해야 할 경우가 많아진다. 이때 QoS 설정이 적절치 않으면 패킷 손실이 잦아지거나 네트워크 혼잡이 심화될 수 있다. Humble에서는 QoS 설정을 통해 이러한 문제를 효과적으로 완화하거나 방지할 수 있는 방안이 더욱 명확히 제시된다.

아래는 C++ 예제 코드를 통해 QoS 설정이 간단히 확장된 모습을 보여준다.

#include <rclcpp/rclcpp.hpp>
#include <std_msgs/msg/string.hpp>

class CustomQoSNode : public rclcpp::Node
{
public:
  CustomQoSNode() 
  : Node("custom_qos_node")
  {
    // QoS 설정 예시
    rclcpp::QoS custom_qos(rclcpp::QoSInitialization::from_rmw(rmw_qos_profile_default));
    custom_qos.reliability(RMW_QOS_POLICY_RELIABILITY_RELIABLE);
    custom_qos.durability(RMW_QOS_POLICY_DURABILITY_TRANSIENT_LOCAL);
    custom_qos.deadline(std::chrono::milliseconds(50));

    publisher_ = this->create_publisher<std_msgs::msg::String>("custom_topic", custom_qos);
    timer_ = this->create_wall_timer(std::chrono::milliseconds(100),
        [this]() {
          auto msg = std_msgs::msg::String();
          msg.data = "Custom QoS Message";
          publisher_->publish(msg);
        }
    );
  }

private:
  rclcpp::Publisher<std_msgs::msg::String>::SharedPtr publisher_;
  rclcpp::TimerBase::SharedPtr timer_;
};

int main(int argc, char ** argv)
{
  rclcpp::init(argc, argv);
  rclcpp::spin(std::make_shared<CustomQoSNode>());
  rclcpp::shutdown();
  return 0;
}

위 예시에서는 Reliability, Durability, Deadline과 같은 QoS 파라미터를 명시적으로 설정하여 메시지의 송수신 신뢰성을 높이고, 일정 시간 내에 메시지를 전송하거나 받지 못하면 이를 감지할 수 있는 방식을 취하였다.

개발 및 빌드 프로세스의 개선¶

ROS2 Humble에서는 개발 편의성을 대폭 높이기 위해 빌드 도구 colcon의 기능 향상과 함께, 패키지 관리를 위한 API 사용성이 개선되었다. 개발자들은 다양한 언어(C++, Python 등)로 노드를 작성하며, 필요에 따라 의존 라이브러리를 자동으로 설치하고 환경 변수를 설정해야 한다. Humble에서는 복잡한 의존 관계를 간소화하거나, 각 노드가 사용하는 메시지 타입을 자동으로 감지하여 빌드 절차를 단순화하는 작업이 이루어졌다.

CLI(Command Line Interface) 툴에서도 명령어 구성이 간결해지고, ros2 doctor 같은 점검 툴이 보강되어 패키지 구조나 통신 설정을 쉽게 확인할 수 있게 되었다. 이는 새로 ROS2를 학습하는 사용자부터 대규모 프로젝트를 다루는 산업계 종사자까지 모두에게 유용하다.

생태계 확장과 상호 운용성 강화¶

ROS2를 활용한 패키지나 라이브러리는 매 버전마다 폭발적으로 늘어나고 있다. Humble 시점에서는 로봇 하드웨어 드라이버, SLAM 라이브러리, 멀티 로봇 협업 프레임워크, 시뮬레이터 통합 도구 등에 대한 호환성이 높아지고, API 안정화가 이루어졌다. 기존에는 특정 버전의 ROS2에만 맞추어 개발된 패키지가 다른 버전과 호환되지 않아 애로사항이 많았지만, Humble 이후에는 안정적인 API 버전을 기준으로 패키지 호환성을 확보하려는 시도가 활발해졌다.

또한 Windows, macOS, Linux와 같은 다양한 OS에 대한 지원 범위가 확장되고, ARM 아키텍처에 대한 최적화가 진행되어 임베디드 환경에서도 ROS2 Humble을 더욱 손쉽게 적용할 수 있게 되었다. 이러한 이식성 확장은 로보틱스 업계에서 필수적인 요소이며, 최근 들어 주목받는 모바일 플랫폼 또는 소형 임베디드 장치에서도 ROS2 기능들을 활용할 수 있도록 만든다.

새로운 커뮤니티 동향과 교육 자원¶

ROS2 Humble의 등장으로 인한 또 다른 긍정적 효과는 교육 자원과 학습 자료가 이전보다 쉽게 확보된다는 점이다. ROS2의 아키텍처와 기능이 어느 정도 자리를 잡아가면서, Humble 버전을 기준으로 한 온라인 튜토리얼과 교재, 강의 자료가 대거 등장하고 있다. 이는 새로운 사용자나 연구자가 ROS2 환경을 빠르게 습득하고, 나아가 실무 프로젝트로 확장하는 데 중요한 역할을 한다.

다양한 공식 문서와 예제, 그리고 커뮤니티가 주도하는 수많은 학습 자료들이 Humble 버전에 맞추어 재정비되고 있다. 이는 ROS1 시절과 비교했을 때 훨씬 더 체계적이고 통일된 지식 체계를 형성하게 하며, 개발자들이 서로 다른 버전이나 환경 차이로 인한 문제를 겪는 상황을 크게 줄여 준다.

--- 및 전망

ROS2 Humble은 ROS2 계열에서 중요한 전환점이 되는 버전으로, 기존 버전에서 누적된 요구사항을 적극적으로 반영하여 실시간 성능, 분산 환경 호환성, API 안정성 등 핵심 부분들을 크게 향상시켰다. 이를 통해 업계와 학계 모두에서 ROS2의 도입과 활용이 가속화되고 있으며, 관련 생태계가 빠른 속도로 확장되는 추세다. 오픈소스 커뮤니티가 주도하는 로봇 소프트웨어가 산업 표준으로 자리 잡아가는 흐름 속에서, Humble은 지속가능한 장기 지원과 함께 앞으로의 ROS2 발전 방향을 제시하고 있다.

계속해서 진행될 차기 배포판에서는 Humble에서 발견된 이슈를 더욱 안정화하고, 일부 실험적 기능들을 표준화하며, 로봇 AI와 클라우드 연동, 시뮬레이터 통합 같은 영역과 긴밀히 협력하게 될 것으로 보인다. 더욱 폭넓은 분야에서 ROS2가 활용되는 시점에, Humble은 견고한 기반을 제공함으로써 실제 로봇 시스템이 요구하는 기능들을 충족시키고, 새로운 개발 패러다임을 안정적으로 이끌어 나갈 것이다.
