시뮬레이션 시간의 개념¶

Isaac Sim을 활용할 때 실제 물리 세계의 시간이 아니라 시뮬레이션 내부적으로 정의된 “시뮬레이션 시간(simulation time)”을 사용한다. 이는 실제 세계에서 1초가 흐르더라도 시뮬레이션 내부적으로는 임의의 스텝 크기(step size)에 따라 시간 단위가 달리 흐르거나, 사용자가 원하는 속도로 시간을 가속할 수 있음을 의미한다. 예를 들어 실제로는 단 1초가 경과했지만, 시뮬레이션 내부에서는 10초가 흐른 것으로 가정할 수 있다.

시뮬레이션 시간은 물리 엔진에서 각 시뮬레이션 스텝마다 \(\Delta t\)만큼 증가된다. 이 \(\Delta t\)는 매우 작은 단위일 수도 있고, CPU나 GPU의 처리 부하에 따라 가변적일 수도 있다. Isaac Sim에서는 이러한 시뮬레이션 시간을 기반으로 동역학, 충돌 검출, 센서 시뮬레이션 등이 계산된다.

시뮬레이션 환경이 복잡해지거나 리소스가 부족해지면 실제 시간(real time) 대비 시뮬레이션 시간이 천천히 흐르기도 한다. 반대로 계산 능력이 충분하다면, 시뮬레이션 시간을 실제 시간보다 빠르게 가속할 수도 있다. 이를 종종 ‘실시간 퍼포먼스(real-time factor)’로 표현하는데, 일반적으로 다음과 같이 정의한다.

\[ \begin{align} RTF &= \frac{\Delta t_\text{시뮬}}{\Delta t_\text{실시간}} \\ \Delta t_\text{시뮬} &= \text{시뮬레이션 상에서 경과한 시간} \\ \Delta t_\text{실시간} &= \text{현실 세계에서 경과한 시간} \end{align} \]

\(RTF > 1\)인 경우 시뮬레이션 시간이 실제 시간보다 빠르게 흐르고 있음을 의미하고, \(RTF < 1\)이면 그 반대를 의미한다.

ROS2 Humble에서의 Clock 사용¶

ROS2 Humble에서 시뮬레이션 시간을 활용하기 위해서는 노드가 “시뮬레이션 시간”을 사용하도록 설정해야 한다. ROS2는 \(\mathtt{/clock}\) 토픽을 통해 외부(Isaac Sim 등)에서 발행되는 시간 정보를 받을 수 있다. 기본적으로 ROS2 노드는 시스템 시간을 사용하지만, 파라미터를 통해 시뮬레이션 시간 사용 여부를 결정할 수 있다.

Isaac Sim에서 ROS2 Bridge를 통해 \(\mathtt{/clock}\) 토픽을 발행하면, ROS2 쪽에서는 이를 받아서 노드들이 사용하는 시간을 동기화한다. 노드 내부에서 \(rclcpp::Clock\) 혹은 Python의 Clock() 객체를 조회할 경우, 이 값이 실제 시간 대신 시뮬레이션 시간으로 바뀌게 된다.

이렇게 시뮬레이션 시간을 사용함으로써, 실제 장비를 연결하지 않고도 원하는 속도로 로봇 제어 알고리즘, SLAM, 네비게이션 등의 테스트를 수행할 수 있다.

Isaac Sim에서의 속도 제어¶

Isaac Sim에서 속도를 제어하는 방식은 크게 두 가지이다. 첫째는 시뮬레이션 내부 파라미터(physics step size)를 조정하는 것이고, 둘째는 시뮬레이션을 멈추거나 재생하는 등의 실행 모드(Play, Pause 등)를 제어하는 것이다.

Isaac Sim의 물리 엔진(예: PhysX)을 사용할 때, \(\Delta t\)를 직접 정의하거나, Isaac Sim에서 제공하는 리얼타임 스케일(real-time scale) 파라미터를 통해 시간을 가속 또는 감속할 수 있다. 이는 실제 물리 시뮬레이션을 빠르게 진행하고 싶을 때나, 매우 느리게 진행하여 세밀한 변화를 관측하고 싶을 때 유용하다. 엔진은 내부적으로 일정한 \(\Delta t\)를 기준으로 계산하지만, 외부에서 노출되는 시뮬레이션 시간의 변화 속도를 조절함으로써 체감되는 시뮬레이션 흐름을 바꿀 수 있다.

다음은 Isaac Sim의 Python 스크립트에서 시뮬레이션 스텝 크기와 시간을 제어하는 예시다.

import omni
from omni.isaac.core.simulation_context import SimulationContext

# SimulationContext 초기화
simulation_app = SimulationContext()

# 물리 엔진 스텝 설정
# 예: step_size를 0.01초(100Hz)로 설정
physics_dt = 0.01
simulation_app.set_simulation_dt(physics_dt)

# 실제 시간 대비 시뮬레이션 시간을 2배 빠르게 (scale=2.0) 설정
simulation_app.set_rendering_dt(1.0 / 60.0)     # 렌더링 빈도
simulation_app.set_time_scale(2.0)

# 시뮬레이션 실행
for _ in range(1000):
    simulation_app.step()

# 시뮬레이션 종료
simulation_app.close()

위 스크립트에서 set_time_scale(2.0)을 통해 실제 시간 1초가 지났을 때, 시뮬레이션에서는 2초가 흐른 것으로 간주하도록 설정한다. 물리 계산은 \(\Delta t = 0.01\)로 고정되어 있으므로, 한 프레임마다 Isaac Sim 내부적으로 물리 엔진 스텝이 반복 호출되어 물리를 계산한다.

고급 주제: 동적 스텝 크기 및 멀티스레딩¶

복잡한 시뮬레이션 환경에서 상호 작용이 증가하면, 스텝 크기 \(\Delta t\)를 고정하는 방식은 비효율적일 수 있다. 이에 따라 상황에 따라 \(\Delta t\)를 동적으로 조정하여 안정성을 보장하면서도 필요 없는 계산을 줄이는 기법이 사용되기도 한다. 예를 들어 충돌이 빈번하게 일어나는 순간에는 \(\Delta t\)를 줄여서 정확도를 높이고, 변화가 거의 없는 구간에서는 \(\Delta t\)를 늘려서 계산 부하를 줄이는 방식이다.

Isaac Sim은 물리 엔진 레벨에서 이러한 동적 스텝 크기(Dynamic Time Stepping)와 멀티스레딩 최적화를 지원할 수 있다. 멀티스레딩 기법을 사용하면 충돌 계산, 렌더링, ROS2 통신 등이 병렬로 진행되어 실시간 퍼포먼스(혹은 초실시간 시뮬레이션)가 가능하다. 다만, 멀티스레딩으로 인해 시뮬레이션 시간 동기화를 더욱 엄밀하게 처리해야 하므로, ROS2 Humble 환경과 연동할 때는 \(\mathtt{/clock}\) 토픽 발행 간격 및 시뮬레이션 스텝 동기화를 주의 깊게 설정해야 한다.

복잡한 물리 시뮬레이션 모델, 센서 플러그인, 여러 로봇의 동시 시뮬레이션 등을 고려할 때에는 시뮬레이션 시간과 실제 시간 사이의 차이를 모니터링하여, 필요하다면 \(RTF\)를 조절하거나 스텝 크기를 유동적으로 바꾸는 과정을 자동화하는 스크립트를 작성하는 것이 유용하다.

시뮬레이션 시간(Clock)을 적절히 제어함으로써 Isaac Sim에서 로봇 동작을 효율적으로 시각화하고, ROS2 Humble와 동기화하여 복합적인 로봇 애플리케이션을 개발할 수 있다. 이를 통해 시뮬레이션과 실제 현장의 차이를 최소화하고, 개발 효율과 안정성을 극대화할 수 있다.
