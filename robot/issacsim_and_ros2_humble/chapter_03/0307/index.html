시뮬레이션 환경에서 시간은 단순한 흐름 이상의 의미를 가진다. 물리 엔진에서 연산되는 스텝 단위의 시간을 어떻게 정의하고 어떤 방식으로 제어하느냐에 따라 로봇의 행동 결과, 센서 데이터의 타임스탬프, ROS2 토픽 간 메시지 동기화 등이 달라진다. Isaac Sim은 자체 엔진을 통해 시뮬레이션 시간을 정의하고 ROS2와 연동하는 메커니즘을 제공한다. 시뮬레이션의 성능과 정확도는 시간 처리 방식에 의해 크게 좌우되므로, 시뮬레이션 시간의 흐름을 엄밀히 이해하고 제어하는 것이 중요하다.

시뮬레이션 시간의 개념¶

시뮬레이션 시간은 실제 시간(real time)과 독립적으로 흐를 수 있다. 실제 시간의 1초가 시뮬레이션에서 1초로 표현될 수도 있지만, 그 배율을 바꾸거나 심지어 시뮬레이션 시간을 정지시키는 것도 가능하다. 이것이 시뮬레이션을 통해 반복 실험, 디버깅, 고속 시뮬레이션 등의 다양한 상황을 유연하게 제어할 수 있게 해주는 핵심 원리다.

특정 시점의 시뮬레이션 시간을 \(t_s\)라 하고 실제 시간을 trt_r라고 할 때, 시뮬레이션 시간은 다음과 같은 비례 관계를 가진다.

\[ \begin{align} t_s = \alpha , t_r \ \end{align} \]

이때 \(\alpha\)는 타임 스케일(Time Scale)이라고 부르며, 이를 변화시켜 시뮬레이션 속도를 제어한다. \(\alpha = 1\)일 경우 실제 시간과 동일하게 흐르며, \(\alpha > 1\)이면 시뮬레이션이 더 빠르게 진행되고 \(\alpha < 1\)이면 더 느리게 진행된다. Isaac Sim에서는 이 \(\alpha\)를 별도의 UI나 스크립트에서 변경함으로써 전체 시뮬레이션 시간의 흐름을 조절할 수 있다.

Isaac Sim에서의 시간 구조¶

Isaac Sim은 NVIDIA Omniverse 플랫폼 위에서 동작하며, 물리 엔진(Physics Engine)과 렌더링 엔진(Rendering Engine)이 서로 다른 주기로 동작한다. 렌더링은 사람이 인지하는 시각적 프레임을 그리기 위한 주기로 진행되는 반면, 물리 엔진은 동역학 계산과 충돌 검출 등을 수행하기 위한 스텝 단위의 틱(tick)으로 진행된다. 보통 시뮬레이션 시간은 물리 엔진의 틱을 기준으로 누적되어 정의된다.

ROS2 환경과 연동 시, Isaac Sim은 물리 엔진이 진행되는 각 스텝마다 내부적으로 시뮬레이션 시간을 갱신하고, 이를 ROS2의 /clock 토픽으로 퍼블리시할 수 있다. 따라서 ROS2에서 제공하는 \(\mathbf{Clock}\) 메시지는 Isaac Sim의 물리 엔진 시간이 반영된 시뮬레이션 시간을 나타내게 된다. 로봇 컨트롤러나 센서 플러그인은 이 시뮬레이션 시간을 기준으로 데이터를 처리하게 되므로, 실제 시간과 시뮬레이션 시간이 얼마나 동기화되어 있는지는 중요한 이슈가 된다.

ROS2 Humble 통합과 시뮬레이션 시간¶

ROS2 Humble은 rclcpp, rclpy 등의 라이브러리를 통해 /clock 토픽을 구독(subscribe)하고, 시뮬레이션 시간 또는 실제 시간을 사용할 것인지를 결정할 수 있다. 일반적으로 로봇 시뮬레이션에서 사용되는 시나리오에서는 use_sim_time 파라미터를 true로 설정해, 실제 시간 대신 시뮬레이션 시간을 사용하도록 한다.

Isaac Sim에서는 ROS2 Extension을 활성화하여 /clock 퍼블리싱을 허용한다. 이를 통해 ROS2 네임스페이스에서 시뮬레이션 시간 정보를 읽어올 수 있다. 시뮬레이션 시간이 제대로 퍼블리시되려면 Isaac Sim 측에서 타임 스텝(physics timestep)과 Clock 퍼블리시 주기가 정상적으로 설정되어야 한다.

시뮬레이션 속도 제어 기법¶

시뮬레이션 속도를 제어하는 주요 방식에는 타임 스케일(Time Scale) 조절과 물리 엔진의 스텝 크기 조절이 있다. 타임 스케일은 시뮬레이션 전체 시간이 실제 시간에 대해 상대적으로 얼마나 빨리 혹은 느리게 흐르는지를 결정한다. 물리 엔진 스텝 크기는 각 물리 연산 틱 사이의 시간 간격으로, 이를 줄이면 좀 더 세밀한 계산을 수행하지만 연산량이 증가한다.

시뮬레이션 시간을 \(t_s\), 실제 시간을 \(t_r\), 물리 엔진의 스텝 간격을 Δt\Delta t라 할 때, Isaac Sim의 내부에서는 다음과 같은 방식으로 시간이 업데이트된다.

\[ \begin{align} t_s \leftarrow t_s + \Delta t \times \alpha \ \end{align} \]

여기서 \(\alpha\)는 타임 스케일, \(\Delta t\)는 물리 엔진 스텝 크기다. 만약 타임 스케일을 2.0으로 설정하면 시뮬레이션 시간은 실제 시간보다 2배 빠르게 흐른다. 반대로 0.5로 설정하면 절반 속도로 진행된다. 물리 엔진 스텝 크기를 줄이면 시뮬레이션은 같은 물리 시간 구간에 대해 더 많은 계산을 수행하지만 더 정확한 충돌 및 역학 해석이 가능해진다.

Real-Time Factor(RTF)¶

시뮬레이션이 실제 시간과 비교하여 얼마나 빠른 속도로 진행되고 있는지를 나타내는 척도로 자주 사용되는 값이 실시간 계수(real-time factor, RTF)다. RTF는 단위 시간 동안 진행된 시뮬레이션 시간과 실제 시간을 비교하여 정의된다. 예를 들어 10초의 시뮬레이션을 실행하는 데 실제로 5초가 걸렸다면 RTF는 2.0이 된다. Isaac Sim에서는 계산 자원이나 물리 엔진의 복잡도에 따라 RTF가 달라질 수 있다. RTF가 1.0에 가깝다면 실제 시간과 거의 동일하게 흐르지만, 높은 해상도의 센서를 여러 개 장착하거나 복잡한 메쉬 충돌을 계산할 경우 RTF가 1 이하로 떨어질 수도 있다.

RTF는 시뮬레이션 속도를 어떻게 설정하였는지, 그리고 시스템의 하드웨어 성능이 어떠한지에 따라 달라지며, 일반적으로 다음과 같은 비례 관계를 갖는다.

\[ \begin{align} \text{RTF} = \frac{\text{시뮬레이션으로 진행된 시간}}{\text{실제로 경과된 시간}} \ \end{align} \]

Clock 동기화와 ROS2 메시지 타임스탬프¶

Isaac Sim에서 발행되는 센서 토픽(예: 카메라 이미지, LiDAR 데이터, IMU 값 등)은 ROS2 메시지 헤더에 시뮬레이션 시간의 타임스탬프를 갖는다. 이는 Isaac Sim의 /clock 메시지와 동기화되므로, ROS2 노드들이 시뮬레이션 환경에서 동일한 시간축을 공유할 수 있다. 시뮬레이션 시간 동기화가 활성화되지 않은 상태에서는 센서 토픽에 실제 컴퓨터 시스템 시간이 들어갈 수 있으므로, 적절한 실험 재현성 확보를 위해서는 Isaac Sim이 제공하는 시뮬레이션 시간으로 통일하는 것이 권장된다.

시뮬레이션 시간 동기화를 위해서는 ROS2 노드의 파라미터에 use_sim_time을 활성화하고, Isaac Sim 내의 ROS2 확장(Extension)에서 /clock 퍼블리시를 허용해야 한다. 이렇게 설정하면 ROS2의 rclcpp, rclpy 인터페이스에서 now() 함수를 호출할 때 시뮬레이션 시간을 얻게 된다.

물리 엔진(Physics) 스텝 제어¶

Isaac Sim은 물리 엔진을 매 프레임마다 자동으로 진행할 수도 있고, 특정 스크립트 명령에 의해 명령형으로 스텝을 진행할 수도 있다. 명령형 스텝(Manual stepping)은 시뮬레이션을 멈춘 상태에서 물리 연산을 하나씩 진행할 수 있게 해주므로, 디버깅이나 세밀한 분석에 유용하다.

Omniverse Kit를 기반으로 한 Isaac Sim 인터페이스에서 물리 엔진의 스텝 크기, 서브 스텝(substep), 타임 스케일 등을 설정할 수 있다. 특정 실험에서는 물리 스텝 크기를 매우 작게 설정하여 높은 정밀도의 시뮬레이션을 수행하기도 하며, 이때에는 실제 시간에 비해 시뮬레이션이 매우 느리게 진행될 수 있다. 반대로 물리 스텝 크기를 너무 크게 설정하면 역학 해석의 정확도가 떨어지지만, 빠른 시뮬레이션을 얻을 수 있다.

시뮬레이션 시간과 스크립트 제어¶

Python 스크립트에서 Isaac Sim과 ROS2의 시뮬레이션 시간을 동적으로 제어할 수 있다. 예를 들어 타임 스케일을 바꾸거나, 물리 엔진의 스텝 크기를 변경하거나, 시뮬레이션을 정지한 뒤 특정 시점에 한 스텝씩 진행하는 방식으로도 제어 가능하다. 이러한 스크립트 기반 제어는 자동화 테스트 파이프라인을 구축하거나 방대한 시나리오를 재현하는 데에 큰 도움이 된다.

아래 예시 코드는 Isaac Sim 내의 Python 스크립트에서 타임 스케일을 동적으로 변경하고, /clock을 퍼블리시하도록 설정한 뒤, 일정 시간 동안 시뮬레이션을 진행하는 간단한 구조를 보여준다. 실제로는 Isaac Sim의 옴니그래프(OmniGraph) 노드나 확장(Extension)을 통해 제어하는 방식과 결합하여 동작한다.

import omni
import time

# Isaac Sim의 physics_interface 가져오기
physics_interface = omni.physics.get_physics_interface()

# 타임 스케일 설정 함수
def set_time_scale(scale_value: float):
    # 이 함수는 Kit 또는 OmniGraph API로 접근 방식이 달라질 수 있음
    # 예시적으로 'physics_interface.set_time_scale(scale_value)' 형태를 가정
    physics_interface.set_time_scale(scale_value)

# 시뮬레이션 초기화
omni.kit.commands.execute('PhysicsPlay')

# 타임 스케일을 2.0으로 설정(시뮬레이션이 실제의 2배 속도로 진행)
set_time_scale(2.0)

# 5초 동안 시뮬레이션 진행
time.sleep(5)

# 시뮬레이션 정지
omni.kit.commands.execute('PhysicsPause')

위와 같은 방식으로 스크립트 내부에서 타임 스케일을 바꾸거나, 물리 엔진의 스텝 크기를 변경할 수 있다. 또한 /clock 퍼블리시를 위해서는 Isaac Sim의 ROS2 Extension 설정에서 Publish Clock 항목을 활성화해야 한다.

시뮬레이션 속도 제어 시 고려 사항¶

시뮬레이션 속도를 과도하게 높이면 물리 엔진 연산이 실제로 따라가지 못하여 RTF가 1 이하로 떨어지는 상황이 발생할 수 있다. 즉, 타임 스케일을 10으로 설정해도 실제로는 높은 연산 부하로 인해 Isaac Sim이 그 속도를 달성하지 못할 수 있다. 또한 센서 데이터의 타임스탬프가 급격하게 증가함에 따라 ROS2의 다른 노드들이 버퍼를 처리하지 못할 가능성도 생긴다.

시뮬레이션 속도를 낮출 때에는 매우 세밀한 계산이 필요한지, 혹은 디버깅 목적을 위해 한 스텝씩 물리 연산을 확인하려는 것인지 의도를 명확히 해야 한다. 타임 스케일을 인위적으로 0.1로 낮추는 것과 물리 스텝 크기를 줄이는 것은 서로 다른 효과가 있을 수 있다. 전자는 전체 시뮬레이션 진행 속도를 느리게 만드는 반면, 후자는 물리 엔진 스텝당 연산 정확도를 높이는 역할을 한다.

엄밀한 실험 재현과 Clock 동기화¶

ROS2 메시지 타임스탬프, 로봇 제어 알고리즘에서 사용하는 시계(clock) 동기화가 보장되어야 실험 재현이 가능하다. Isaac Sim에서 시뮬레이션 시간을 사용해 기록한 토픽 데이터를 향후 재생(rosbag play 등)하거나, 동일한 시뮬레이션 설정에서 반복 테스트를 할 때 타임스탬프가 정확히 일치해야 한다. Isaac Sim에서 시뮬레이션 속도를 조절할 때에는 /clock 토픽이 올바르게 퍼블리시되는지, ROS2 노드가 해당 Clock에 동기화되어 있는지를 항상 확인해야 한다.

시뮬레이션 시간이 정확히 흐르도록 만들기 위해서는 물리 스텝, 타임 스케일, ROS2 퍼블리시 레이트를 균형 있게 설정해야 한다. 특히 물리 스텝 계산이 완료되지 않은 상태에서 렌더링 프레임만 지나치게 증가한다면 시뮬레이션 시간과 센서 데이터의 시점이 어긋날 수 있다. 이를 방지하기 위해 Isaac Sim 내부에서 렌더링과 물리 연산을 동기화하거나, 특정 옵션을 통해 렌더링 속도와 물리 스텝이 균일하게 진행되도록 제어할 수 있다.

시뮬레이션 환경에서 시간은 로봇 실험을 재현하고 다양한 알고리즘을 테스트하는 데 있어 핵심적인 요소다. Isaac Sim과 ROS2 Humble을 통합하여 로봇의 센서 데이터와 제어 명령을 정확한 타임스탬프로 동기화하려면, 시뮬레이션 시간(Clock)에 대한 이해가 필수적이다. 물리 엔진 스텝, 타임 스케일, RTF, 그리고 ROS2의 /clock 사용 방식까지 심도 있게 파악해야 다양한 시나리오에서 신뢰할 수 있는 결과를 얻을 수 있다. 시뮬레이션 속도 제어 기법을 적절히 활용하면, 고속 테스트나 정밀 디버깅 같은 특수 목적에도 Isaac Sim 환경을 유연하게 적용할 수 있다. 이러한 시간을 엄밀히 다루는 역량이 Isaac Sim과 ROS2 Humble을 제대로 활용하는 핵심 역량이다.
