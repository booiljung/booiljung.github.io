DDS의 개념과 특성¶

DDS(Data Distribution Service)는 분산 환경에서 데이터를 효과적이고 실시간으로 교환하기 위한 미들웨어 표준이다. 객체 관리 그룹(OMG, Object Management Group)에서 정의한 데이터 교환 모델이며, 주로 임베디드, 실시간, 분산 시스템 등 다양한 분야에서 사용된다. DDS는 Pub-Sub(Publish-Subscribe) 모델을 따르며, 노드들 간 데이터를 주고받기 위해 RTPS(Real-Time Publish Subscribe) 프로토콜을 기반으로 동작한다.

  • Pub-Sub 모델:
  • 퍼블리셔(Publisher)는 특정 주제(Topic)나 데이터 타입에 대한 데이터를 발행한다.
  • 서브스크라이버(Subscriber)는 해당 데이터에 대한 구독을 요청하고, DDS 미들웨어는 적절히 매칭된 데이터만을 서브스크라이버에게 전달한다.
  • RTPS 프로토콜:
  • DDS가 네트워크 위에서 상호 통신할 때 사용하는 표준 프로토콜이다.
  • RTPS는 IP를 기반으로 동작하며, 자체적인 세션 관리, 디스커버리, 신뢰성 보장 방식을 제공한다.

DDS는 각 노드가 무엇을 발행하고, 어떤 정보를 구독하는지를 자동으로 파악하여 네트워크 상에서 노드 간 연결을 동적으로 구성한다. 이를 Discovery(디스커버리)라고 부르며, DDS는 이를 위해 Simple Discovery Protocol(SDP)과 Enhanced Discovery 방식을 사용한다. 각 벤더 DDS 구현에 따라 세부 구현이 다를 수 있지만, 기본 구조는 유사하다.

ROS2 Humble가 DDS를 선택한 이유¶

ROS(ROS1) 시절에는 메시지 교환이 자체 개발된(혹은 TCP/IP 기반의) Transport 레이어 위에서 이루어졌다. 하지만 네트워크 트래픽이 복잡해지고, 동시에 많은 노드가 서로 정보를 빠르고 유연하게 교환하기가 어려워지는 한계점이 있었다. ROS2에서는 이러한 문제를 해결하기 위해 다양한 실시간 시스템을 지원하고, QoS(Quality of Service) 설정이 자유로운 DDS를 통신 미들웨어로 채택하였다.

  • 실시간성과 낮은 지연:
  • DDS는 실시간 처리가 요구되는 환경에서 최적화된 통신 모델을 제공한다.
  • TCP 기반의 기존 ROS1 메시지 교환보다 지연 시간을 크게 줄일 수 있다.
  • QoS 설정의 유연성:
  • 신뢰성(RELIBILITY), 내구성(DURABILITY), 히스토리(HISTORY) 등 다양한 QoS 정책을 설정해 원하는 통신 품질을 보장받을 수 있다.
  • 다중 플랫폼 지원:
  • DDS는 이미 업계 표준으로 자리를 잡았고, 다양한 벤더 구현이 존재한다(eProsima의 Fast DDS, RTI Connext DDS, ADLINK의 OpenSplice 등).
  • 윈도우, 리눅스, macOS 등 주요 OS뿐 아니라 다양한 임베디드 RTOS도 지원할 수 있다.

이러한 이유들로 인해 ROS2는 DDS를 통해 미래의 로봇 분산 시스템, 특히 ROS2 Humble과 같은 버전에서도 신뢰성 높은 통신을 이어갈 수 있게 된다.

ROS2 Humble의 RMW 계층과 DDS¶

ROS2는 RCL(Robot Client Library)과 RMW(ROS Middleware)라는 계층 구조를 통해 DDS와의 의존성을 간접적으로 관리한다. 즉, DDS는 RMW 레이어에 의해 추상화되어 애플리케이션 코드(RCL, ROS2 API)에서 직접 DDS API를 호출하지 않고도 메시지 통신을 수행할 수 있다.

  • RCL(Robot Client Library):
  • ROS2의 최상위 레이어로, 개발자가 ROS2 API를 통해 노드를 생성, 토픽 발행과 구독, 서비스 호출과 응답 등을 구현한다.
  • RMW(ROS Middleware) 레이어:
  • DDS를 비롯한 다른 미들웨어를 추상화한다.
  • 벤더별 DDS 구현을 플러그인처럼 교체할 수 있게 하여 유연성을 부여한다.
  • DDS:
  • 실제 통신 로직 및 네트워크 디스커버리, 패킷 전송 등을 담당한다.
  • RMW 계층에 의해 메타 정보(토픽 타입, QoS 설정 등)를 전달받아 최적의 방식으로 데이터를 교환한다.

ROS2 Humble에서는 더욱 개선된 RMW 인터페이스가 제공되어, 다양한 DDS 벤더와의 호환성 및 성능 최적화가 이루어졌다.

Quality of Service(QoS)와 ROS2 Humble의 유연성¶

DDS에서 제공하는 QoS 정책을 ROS2 Humble에서도 그대로 적용할 수 있다. 예를 들어 다음과 같은 QoS 설정을 자유롭게 지정한다.

  • Reliability: Reliable / Best effort
  • Durability: Transient local / Volatile
  • History: Keep last / Keep all
  • Deadline: 메시지가 처리되어야 하는 기한
  • Liveliness: 노드나 퍼블리셔가 살아 있음을 증명하는 방식

개발자는 위 QoS 파라미터를 상황에 맞게 설정하여, 네트워크 트래픽을 최적화하거나 신뢰도를 높일 수 있다. 예를 들어 센서 데이터는 빠른 전송이 중요하므로 best effort 모드를 사용하고, 중요한 명령 메시지는 reliable 모드를 사용하여 손실 없이 전달하도록 설정할 수 있다.

다중 벤더 DDS 환경에서의 확장성¶

ROS2 Humble은 기본적으로 eProsima의 Fast DDS를 많이 사용하지만, 다른 DDS 벤더 구현체로도 쉽게 교체해볼 수 있는 구조를 갖추고 있다. 이를테면 RTI Connext DDS나 ADLINK의 OpenSplice DDS를 사용하여 성능이나 라이선스 측면에서 좀 더 적합한 솔루션을 찾을 수 있다.

DDS 벤더 교체 방법:

  • 원하는 DDS 벤더의 RMW 구현 패키지를 설치한다.
  • ROS2 Humble에서 RMW_IMPLEMENTATION 환경 변수를 통해 사용할 RMW를 지정한다.
  • 예:
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
  1. 노드를 실행하면 지정된 RMW 구현을 통해 DDS 통신이 이루어진다.

장단점 비교:

  • Fast DDS: 오픈소스 커뮤니티 지원이 활발, 업데이트가 빈번하고 가볍다.
  • RTI Connext DDS: 상용 솔루션, 성능 최적화가 잘 되어 있고 문서화가 풍부하지만 비용이 든다.
  • OpenSplice 등: 다른 벤더의 DDS 구현도 오랜 기간 산업에서 사용되어 안정성이 높다.

ROS2 Humble 사용자는 요구사항에 따라 DDS 벤더를 선택하여 성능과 비용, 기능을 조정할 수 있다.

DDS 디스커버리(Discovery) 상세 과정¶

DDS의 핵심 기능 중 하나인 디스커버리는 네트워크 상에서 퍼블리셔와 서브스크라이버(또는 관련 엔티티)들을 자동으로 탐지하고 적절히 연결해주는 과정이다. ROS2 Humble에서도 이 디스커버리 과정을 통해 노드들이 자동으로 서로를 인식하여 토픽 통신이 이뤄진다. 디스커버리는 주로 RTPS 프로토콜 상에서 진행되는데, 아래와 같은 단계를 포함한다.

  1. Participant Discovery
  2. DDS에서 노드는 보통 Participant라는 개념으로 표현된다. ROS2 Humble의 각 노드는 내부적으로 하나 이상의 Participant를 가진다.
  3. 네트워크에 새 Participant가 등장하면, 기존에 있던 Participant와 통신하여 각자에 대한 정보(Participant GUID, QoS 정보 등)를 교환한다.
  4. Endpoint Discovery
  5. Participant 간에 디스커버리가 완료되면, 각 Participant에 등록된 Publisher/Subscriber와 같은 Endpoint 정보가 교환된다.
  6. 이를 통해 실제 데이터 교환에 필요한 Topic, Type, QoS 등을 확인한다.
  7. Matching
  8. Publisher와 Subscriber가 디스커버리를 통해 서로를 인식하면, 설정된 QoS가 호환되는지 확인한다.
  9. QoS가 서로 호환될 경우, 네트워크상에서 실제 메시지 전송 경로가 설정되어 데이터가 교환된다.
  10. 반면 QoS가 호환되지 않는다면 연결이 성립되지 않으므로 메시지가 전달되지 않는다.

이 과정을 통해 ROS2 노드들은 추가 설정 없이도 네트워크 상에서 자동으로 서로를 찾고, 알맞은 토픽 데이터를 교환할 수 있다.

TypeSupport와 ROS2 Humble에서의 메시지 구조¶

ROS2 Humble에서 메시지 타입은 DDS의 타입과 직접적으로 연관되어 있다. 특히 ROS2가 제공하는 각 메시지 타입(sensor_msgs, std_msgs 등)은 DDS에서 사용할 수 있도록 별도의 IDL(Interface Definition Language) 파일과 C++ 코드를 생성한다. 이를 TypeSupport라고 부른다.

  • IDL 파일
  • DDS는 OMG가 정의한 IDL 인터페이스를 통해 구조체, 열거형, 배열, 시퀀스 등의 자료형을 기술한다.
  • ROS2 메시지(예: geometry_msgs/Twist)는 내부적으로 IDL로 표현되어, DDS가 해석 가능한 형태로 변환된다.
  • TypeSupport 클래스
  • DDS에서 메시지를 직렬화/역직렬화(Serialization/Deserialization)하고, 네트워크 전송에 필요한 형식으로 변환하기 위한 기능이다.
  • ROS2 Humble에서 메시지 패키지를 빌드하면 자동으로 생성되는 이 TypeSupport 코드를 통해, 노드 개발자는 별도 DDS API를 몰라도 메시지를 교환할 수 있다.

예를 들어, \(\mathbf{x} = \begin{bmatrix}1 \ 2 \ 3\end{bmatrix}\) 와 같은 벡터를 담는 메시지가 있다고 하면, IDL 파일에서 다음과 같이 정의될 수 있다(예시이며 실제 구조와 다를 수 있음).

struct Vector3 {
  float x;
  float y;
  float z;
};

이 정의가 DDS 레벨에서 TypeSupport 코드를 생성해주고, 퍼블리셔-서브스크라이버는 서로 같은 메시지 구조임을 자동으로 인지하여 데이터를 안전하게 주고받게 된다.

RTPS Submessage와 데이터 전송 기법¶

RTPS(Real-Time Publish-Subscribe)는 DDS가 실제 데이터 전송에 사용하는 표준 프로토콜이다. ROS2 Humble에서 DDS를 통해 메시지를 주고받는 과정은 RTPS Submessage라는 논리 단위로 이뤄진다.

  • Header
  • 버전 정보, RTPS 프로토콜 식별자, Submessage 개수 등
  • Submessage
  • Data: Publisher가 발행하는 실제 메시지가 포함된다.
  • Heartbeat: Reliable QoS에서 필요한 데이터 수신 상태를 확인하기 위한 신호.
  • AckNack: Subscriber가 수신하지 못한 메시지가 있음을 알리는 신호.
  • Gap: 특정 시퀀스 넘버 구간에 메시지가 존재하지 않음을 알리는 신호.

신뢰성 있는 통신을 위해 Heartbeat와 AckNack을 교환하는 과정을 통해, 유실된 패킷이 있으면 재전송을 요구하는 식으로 동작한다.

DDS 보안과 SROS2¶

ROS2 Humble에서는 DDS의 보안 확장을 활용한 SROS2(Security for ROS2) 기능을 제공한다. DDS 보안 표준인 DDS Security를 적용해 노드 간의 데이터 암호화, 인증, 무결성 보장 등을 수행할 수 있다.

  • DDS Security Plugin

  • DDS 벤더별로 제공하는 보안 플러그인을 통해 메시지 암호화, 키 교환, 접근 제어 등의 기능을 수행한다.

  • X.509 인증서와 같은 표준 인증 체계를 활용한다.

  • SROS2

  • ROS2에서 DDS 보안을 쉽게 설정하고 관리하기 위한 별도 툴과 라이브러리 집합이다.

  • 정책 파일, 키스토어(Keystore) 생성, 인증서 발급 등을 편리하게 수행할 수 있는 CLI 기능을 제공한다.
  • 예:
ros2 security create_keystore my_keystore
ros2 security create_key my_keystore /my_namespace/my_node

  • 적용 시 이점

  • 분산 로봇 시스템에서 중요한 명령·제어 메시지의 안전성과 무결성을 보장한다.

  • 네트워크 공격에 대응하고, 권한이 없는 노드가 임의로 메시지를 받거나 발행할 수 없도록 제어한다.

DDS 보안을 사용하면 실시간 통신에 부가적인 오버헤드가 있을 수 있지만, ROS2 Humble에서는 QoS와 보안 정책을 조정하여 적절한 균형을 찾을 수 있다.

Intra-process 통신에서의 DDS 활용¶

ROS2 Humble에서는 노드 내부(Intra-process) 통신을 효율적으로 처리하기 위한 메커니즘을 제공한다. Intra-process 통신은 네트워크를 경유하지 않고 프로세스 내부 메모리를 통해 메시지를 주고받을 수 있는 방법을 의미한다. DDS 기반의 기본 구현은 프로세스 간 통신(Inter-process)을 전제로 설계된 반면, Intra-process에서는 데이터 복사와 직렬화 과정을 최소화하려는 최적화가 필요하다.

  • Zero-copy 또는 Loaned message
  • DDS 벤더에 따라, 퍼블리셔가 메시지를 발행할 때 해당 메모리를 서브스크라이버가 재사용(Loan)할 수 있도록 하는 기능을 제공하기도 한다.
  • ROS2 Humble의 Intra-process 통신 모드는 이를 더욱 확장해, 노드 내부에서 불필요한 복사가 발생하지 않도록 해준다.
  • 잠재적인 이점
  • CPU 자원 절약: 직렬화/역직렬화 과정을 없애거나 최소화해, CPU 사용량을 줄일 수 있다.
  • 낮은 지연(latency): 인터럽트나 네트워크 I/O 없이 바로 데이터 버퍼를 공유하므로, 훨씬 빠른 통신이 가능하다.

Domain ID와 DDS 멀티 로봇 시스템¶

ROS2 Humble에서 다중 로봇을 운용하려면, DDS에서 제공하는 Domain ID 개념을 잘 이해해야 한다. Domain ID는 DDS 네트워크를 논리적으로 구분하는 식별자로서, 서로 다른 Domain ID의 노드들은 기본적으로 서로를 발견하지 못한다.

  • Domain ID 활용

  • 로봇마다 다른 Domain ID를 사용하면 서로 간의 트래픽이 격리된다.

  • 반대로 여러 로봇을 하나의 영역에서 협업하게 하고 싶다면 같은 Domain ID를 사용한다.

  • 환경 변수 설정

  • ROS2에서는 ROS_DOMAIN_ID 환경 변수를 사용하여 Domain ID를 설정한다.

  • 예:
export ROS_DOMAIN_ID=10

  • 이렇게 설정하면 DDS Participant를 생성할 때 내부적으로 Domain ID 10을 사용한다.

  • 응용 사례

  • 테스트 환경에서 여러 인스턴스(시뮬레이터, 실제 로봇 등)를 구분하거나, 하나의 로봇을 여러 그룹의 센서/액추에이터로 구분해 도메인을 구성할 수 있다.

WAN(광역 네트워크) 환경에서의 DDS¶

ROS2 Humble에서 DDS를 사용하는 경우, 주로 로컬 네트워크나 동일 서브넷 내에서 노드 간 자동 디스커버리가 이뤄진다. 하지만 로보틱스 시스템이 여러 지리적 장소에 분산되어 있거나, 클라우드와 연동해야 하는 경우에는 WAN을 통한 DDS 통신을 고려해야 한다.

  • WAN 통신 이슈
  • DDS의 기본 디스커버리 방식을 그대로 쓰면, 브로드캐스트/멀티캐스트 패킷이 WAN 환경에서 차단되거나 성능이 크게 저하될 수 있다.
  • 방화벽, NAT 환경 등에서 직접 DDS 세션이 맺어지기 어렵다.
  • 해결 방안
  • DDS Routing Service: RTI Connext DDS 등 상용 벤더에서 제공하는 라우팅 서비스 혹은 브릿지 노드를 사용해 WAN 통신을 중계할 수 있다.
  • VPN 사용: VPN 터널을 통해 로컬 네트워크와 유사한 멀티캐스트 구성이 가능하도록 환경을 세팅한다.
  • Edge 노드: 로컬 DDS 네트워크와 WAN 구간을 중계해주는 별도의 ROS2 노드를 구성하여, WAN 구간은 TCP/HTTP/웹소켓 등 다른 통신 방식을 사용할 수도 있다.

RMW 계층에서의 인스트루멘테이션(Instrumentation)¶

DDS 기반 RMW 계층은 다양한 디버깅과 모니터링 기능을 제공할 수 있다. ROS2 Humble에서는 노드 간의 트래픽, QoS 충돌, 메시지 수신 지연 등을 관찰하기 위해 ddsperf나 ROS2 tracing 툴을 사용한다.

  • ddsperf
  • DDS 벤더별로 제공하는 성능 측정 도구로, 퍼블리셔-서브스크라이버 간 메시지 레이트, 지연, 손실률 등을 확인한다.
  • 예: Fast DDS에서는 Fast-RTPS-Gen과 연동한 퍼포먼스 측정 샘플을 제공한다.
  • ROS2 tracing
  • ros2_tracing 패키지를 사용하면 LTTng 기반의 상세 추적(Trace)이 가능하다.
  • rclcpp, rcl, rmw 계층에서 일어나는 이벤트를 기록해 성능 병목 또는 QoS 미스매치를 진단할 수 있다.

노드 그래프와 토픽 시각화¶

ROS2 Humble은 DDS 덕분에 동적으로 노드를 연결할 수 있지만, 개발 시에는 어떤 노드가 어떤 토픽을 발행/구독하는지 시각적으로 파악하기가 중요하다. 이를 위해 다음과 같은 도구들이 사용된다.

  • rqt_graph
  • 전통적인 ROS1 시절부터 사용되던 GUI 툴이며, ROS2에서도 사용 가능하다.
  • 노드와 토픽 간의 관계를 그래프로 표시해준다.
  • ros2 topic list / ros2 node list
  • 커맨드라인 툴을 통해 현재 활성화된 노드와 토픽 목록을 확인할 수 있다.
  • 예시
  • DDS 토픽 관계를 표시
flowchart LR A["Node A"] --> T["Topic /example_topic"] B["Node B"] --> T T --> C["Node C"]

위처럼 노드 간의 퍼블리셔와 서브스크라이버 관계를 간단히 시각화할 수 있다.

고급 QoS 정책 적용과 실전 예시¶

ROS2 Humble에서 DDS를 사용할 때, QoS(품질 보장) 정책 설정은 시스템 요구사항에 따라 매우 다양하게 적용할 수 있다. 일반적으로 많이 쓰이는 Reliability, Durability, History 외에도 다음과 같은 정책들이 존재한다.

  • Latency Budget
  • 메시지가 노드에 도착하기까지 평균적으로 허용 가능한 지연 시간을 설정한다.
  • DDS 미들웨어는 이 값을 참고하여 네트워크 전송 전략이나 버퍼링 전략을 최적화한다.
  • Liveliness
  • 노드나 퍼블리셔가 활성(alive) 상태인지 검사하는 메커니즘이다.
  • Automatic 모드: DDS 미들웨어가 자동으로 liveliness를 체크한다.
  • Manual by Topic 모드: 사용자가 명시적으로 노드가 살아 있음을 알린다.
  • 로봇 시스템에서는 제어 명령 퍼블리셔가 죽었는지를 빠르게 감지해 안전 대처를 해야 할 때 활용한다.
  • Deadline
  • 서브스크라이버가 필요한 주기로 메시지를 받아야 하는 기한(Deadline)을 설정한다.
  • 퍼블리셔가 설정된 기간 내에 메시지를 발행하지 못하면 QoS 위반으로 판단되고, DDS는 관련 콜백을 트리거한다.
  • 예: 10Hz 주기로 읽어야 하는 센서데이터는 Deadline QoS를 100ms로 설정해 누락을 감지할 수 있다.

이러한 QoS 조합을 통해 실시간성과 신뢰성을 동시에 만족하는 분산 로봇 시스템을 구현할 수 있다. 예를 들어, Liveliness + Deadline 조합을 통해 특정 주기마다 메시지가 반드시 와야 하고, 퍼블리셔가 중단되면 곧바로 파악할 수 있다.

정적 디스커버리(Static Discovery) 기법¶

기본 DDS 디스커버리는 동적 방식이어서 추가 설정 없이 네트워크 내의 노드를 자동으로 찾는다. 그러나 대규모 혹은 제한된 네트워크 환경에서는 정적 디스커버리(Static Discovery) 파일을 사용하여 성능과 보안을 향상시킬 수 있다.

  • 정적 디스커버리 파일
  • 특정 Domain 내에서 참여할 Participant 정보, Topic, QoS 등을 미리 정의한 XML 파일 등을 DDS 미들웨어에 로드한다.
  • 디스커버리 트래픽이 줄어들고, 예측 가능한 네트워크 구성이 가능하다.
  • ROS2 적용 예
  • 큰 공장을 운영하는 로보틱스 시스템에서, 서로 통신하는 노드 목록이 고정적이라면 정적 디스커버리 방식을 써서 네트워크 트래픽을 줄일 수 있다.
  • WAN 환경에서 보안/방화벽을 고려해야 할 경우에도 정적 디스커버리를 이용해 포트와 IP 주소를 미리 고정할 수 있다.

DDS XML 프로파일 설정¶

DDS 벤더들은 XML을 통해 QoS, 디스커버리, 세부 동작 등을 설정하도록 지원한다. ROS2 Humble에서는 RMW 구현에 따라 이런 설정 파일을 로드할 수 있다.

  • Fast DDS 예시

  • FASTRTPS_DEFAULT_PROFILES_FILE 환경변수를 통해 XML 파일 경로를 지정한다.

  • 예:
export FASTRTPS_DEFAULT_PROFILES_FILE=/path/to/my_dds_profile.xml
  • ROS2 노드 실행 시 DDS가 해당 프로파일을 불러와 QoS, 디스커버리, 보안 설정 등을 반영한다.

  • RTI Connext DDS 예시

  • NDDS_CONFIG_FILE 환경변수를 통해 XML 파일 경로를 지정하며, Connext만의 설정 문법을 쓴다.

XML 프로파일을 통해 DDS 미들웨어 레벨에서 커스터마이징을 해두면, ROS2 코드 수정 없이도 QoS나 네트워크 설정을 유연하게 바꿀 수 있다.

성능 테스트와 벤치마크¶

ROS2 Humble 환경에서 DDS의 성능을 검증하고자 할 때, 여러 가지 툴과 벤치마크 방법이 존재한다.

  • ROS2 자체 성능 툴
  • ros2 topic pub / ros2 topic echo를 이용해 실제 메시지 왕복 시간과 throughput을 재본다.
  • ros2 launch로 다수의 노드를 띄워놓고 메시지 레이트를 높여가며 테스트해볼 수 있다.
  • DDS 벤더별 성능 툴
  • eProsima Fast DDS: Fast DDS Benchmark 샘플 프로젝트나 perf 툴 제공.
  • RTI Connext DDS: 실시간 모니터링 툴(Analyzer), recording 서비스 등을 통해 성능과 트래픽을 측정할 수 있다.
  • 실환경 시나리오 테스트
  • 여러 센서(카메라, LiDAR 등)를 연결한 실제 로봇에서 ROS2 토픽 데이터를 퍼블리시해보고, 원하는 FPS·대역폭이 만족되는지 확인한다.
  • 네트워크 지연, 패킷 손실 등을 인위적으로 가중시켜 견고성을 테스트한다.

DDS 확장 기능: Logging과 Monitoring¶

ROS2 Humble가 DDS 미들웨어를 사용하면, DDS가 제공하는 모니터링 및 로깅 기능도 적극 활용할 수 있다.

  • DDS Logging
  • DDS 스택 내부에서 발생하는 이벤트, 경고 메시지 등을 로깅 수준(INFO, DEBUG, ERROR)에 따라 출력한다.
  • 특정 시점에 DDS가 메시지를 드롭했는지, QoS 매칭이 실패했는지 등을 알 수 있어 디버깅에 유리하다.
  • Monitoring
  • DDS 샘플 수, QoS 충돌 횟수, Discovery 상태, 네트워크 트래픽 통계 등을 모니터링할 수 있다.
  • 상용 DDS 벤더들은 GUI 모니터링 툴을 제공해 직관적인 그래프와 알림을 지원한다.

ROS2의 rclcpp 레이어 로그와 결합해 분석하면, 애플리케이션 로직과 DDS 통신 레벨 양쪽에서 발생하는 문제를 파악하기가 더 쉽다.

대규모 시스템에서의 DDS 샤딩(Sharding)¶

여러 로봇이나 수십~수백 개 노드가 동시에 움직이는 대규모 시스템에서는, DDS 디스커버리가 부담스러울 수 있다. 이때는 샤딩(Sharding) 기법을 도입해 도메인을 물리적으로 구분하거나 노드 그룹별로 구성하여 확장성을 높일 수 있다.

  • 샤딩 개념
  • 여러 Domain ID를 두고, 로봇 또는 노드 그룹을 분할 배치한다.
  • 그룹 간 통신이 필요한 경우에는 브릿지 노드나 라우터 서비스를 통해 연결한다.
  • 장점
  • Discovery 트래픽이 각 그룹 내부로 제한되므로, 확장성이 향상된다.
  • 특정 그룹에서 QoS 설정을 다르게 적용할 수도 있어, 성능·안정성을 세분화할 수 있다.

예를 들어, Domain ID 0에서 센서 데이터 처리 노드들을 몰아넣고, Domain ID 1에서는 SLAM이나 AI 연산 노드들을 운영한 뒤, 필요한 토픽만 브리지 노드로 주고받게 할 수 있다.

DDS 기반 ROS2 네트워크 구조 간단 예시 (mermaid)¶

flowchart LR subgraph "Domain ID=0" A("Robot1 Sensor Node") --> T1("Topic /scan") A --> T2("Topic /camera") T3("Topic /cmd_vel") --> B("Robot1 Control Node") end subgraph "Domain ID=1" T1 --> C("SLAM Node") T2 --> D("Mapping Node") D --> T4("Topic /map") end BN["Bridge Node"] --> T1 BN --> T2 BN --> T3 T4 --> BN

위 구조에서 로봇 센서와 제어는 Domain ID=0에서 빠르게 주고받고, SLAM과 맵핑은 Domain ID=1에서 별도로 구동한다. Bridge Node가 Topic을 양쪽 도메인에 연결해준다.
