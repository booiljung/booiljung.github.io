ROS2 인터페이스의 기본 개념¶

ROS2는 노드 간 데이터를 주고받기 위해 다양한 통신 방식을 제공한다. 가장 대표적인 통신 방식으로 메시지, 서비스, 액션이 있으며, 이를 통합적으로 ‘ROS2 인터페이스(ROS2 Interface)’라고 부른다. 각 통신 방식마다 사용 목적이 다르고, 인터페이스 정의 파일의 형식 또한 다르다.

  • 메시지(Message): 단순히 정보를 송수신하는 자료 구조.
  • 서비스(Service): 요청(Request)과 응답(Response)이 한 쌍으로 이루어지는 동기적 통신 방식.
  • 액션(Action): 목표(Goal)를 설정하고 중간 피드백(Feedback)을 주고받으며 최종 결과(Result)를 얻는 비동기적 통신 방식.

메시지, 서비스, 액션 인터페이스는 모두 .msg, .srv, .action 파일로 정의하고, 빌드 과정에서 자동으로 C++/Python 등에 맞게 소스 코드가 생성된다. 이를 통해 인터페이스를 효율적으로 공유하고 재사용할 수 있다.

패키지 생성과 디렉터리 구조¶

메시지, 서비스, 액션 인터페이스를 정의하기 위해서는 ROS2 패키지를 새로 생성한 뒤, 패키지 내부에 msg, srv, action 디렉터리를 만들어 인터페이스 정의 파일을 위치시켜야 한다.

ROS2에서 자주 사용하는 빌드 도구는 colcon이며, 새 패키지는 ros2 pkg create 명령을 통해 생성할 수 있다. 예를 들어, my_robot_interfaces라는 이름의 패키지를 만들려면 다음과 같이 명령을 입력한다.

ros2 pkg create my_robot_interfaces --build-type ament_cmake

생성된 디렉터리 구조는 대략 다음과 같다.

my_robot_interfaces/
├── CMakeLists.txt
├── package.xml
├── include/
├── src/
└── ...

이후 메시지, 서비스, 액션 정의 파일을 배치할 디렉터리를 생성한다.

cd my_robot_interfaces
mkdir msg srv action

최종적으로 다음과 같은 구조를 가지도록 만든다.

my_robot_interfaces/
├── action/
├── msg/
├── srv/
├── CMakeLists.txt
├── package.xml
└── ...

이렇게 디렉터리를 준비한 뒤, 각각의 폴더에 .msg, .srv, .action 파일을 작성하고 CMakeLists.txt 및 package.xml 설정을 진행하면 된다.

메시지 인터페이스(.msg) 정의¶

메시지는 노드 간 송수신될 데이터를 표현한다. 예를 들어 센서에서 측정한 거리 정보를 송신하는 경우, 이를 나타내는 메시지를 정의할 수 있다.

메시지 정의 파일 기본 형식¶

메시지 정의 파일은 .msg 확장자를 가지며, 각 줄에 메시지 필드 정보를 작성한다. 예를 들어 다음과 같이 msg 디렉터리에 DistanceSensor.msg라는 메시지 파일을 작성한다고 해보자.

float32 min_range
float32 max_range
float32 current_distance

위 예시에서는 최소 측정 거리, 최대 측정 거리, 현재 측정 거리 세 개의 필드를 가진다. ROS2 빌드 시스템은 이 .msg 파일을 기준으로 언어별(예: C++/Python) 메시지 클래스 또는 모듈을 자동 생성한다.

메시지 타입 지원¶

ROS2의 .msg 파일에서는 다음과 같은 기본 타입을 지원한다.

  • 숫자형: bool, int8, uint8, int16, uint16, int32, uint32, int64, uint64, float32, float64
  • 문자열: string
  • 시간: time, duration
  • 복합 타입: 배열(Array), 다른 메시지 타입

배열이나 벡터를 표현하려면 [ ]를 사용한다. 예를 들어 $\mathbf{x} \in \mathbb{R}^{3}$에 해당하는 3차원 벡터를 담고 싶다면 다음과 같이 정의한다.

float64[3] position

또한 가변 길이 배열을 사용하려면 다음과 같이 표기한다.

float64[] points

정의된 .msg 파일은 이후 노드에서 #include(C++의 경우) 혹은 import(Python의 경우) 하여 사용하게 된다.

예시: 벡터와 행렬¶

만약 로봇 제어를 위해 3x3 크기의 매트릭스를 송수신해야 한다고 가정하면, 다음처럼 정의할 수 있다.

float64[9] transform_matrix

3x3 행렬은 총 9개의 원소로 표현할 수 있으므로 이를 일차원 배열로 정의하는 것이다. 실제로 사용 시에는 3x3 형태로 변환해서 쓰게 된다. 예를 들어 다음과 같이 수학적으로 표현할 수 있다. $$ \mathbf{M} = \begin{bmatrix} m_{11} & m_{12} & m_{13} \ m_{21} & m_{22} & m_{23} \ m_{31} & m_{32} & m_{33} \end{bmatrix} $$ 하지만 ROS 메시지에서는 이 행렬을 float64[9] transform_matrix 형태로 정의하여 사용한다.

서비스 인터페이스(.srv) 정의¶

서비스는 요청(Request)과 응답(Response)으로 구성된 통신 방식을 지원한다. 클라이언트에서 요청을 보내면, 서버는 이를 처리한 뒤 결과를 응답으로 되돌려 준다. 일반적인 메시지 통신(Publish/Subscribe)이 비동기적 송수신에 초점을 맞추는 반면, 서비스는 “질의-응답” 형태의 동기적 통신을 구현할 때 사용한다.

서비스 파일 기본 형식¶

서비스 정의 파일은 .srv 확장자를 가지며, “---”를 기준으로 위아래가 Request와 Response 영역으로 나뉜다. 예를 들어 다음과 같이 srv 디렉터리에 SetLedState.srv 파일을 작성할 수 있다.

uint8 led_number
bool state
---
bool success
string message
  • uint8 led_number와 bool state는 Request 영역에 포함되어, 클라이언트가 LED 번호와 ON/OFF 상태를 요청한다.
  • bool success와 string message는 Response 영역에 포함되어, 요청 처리 후 서버가 성공 여부와 메시지를 응답한다.

다양한 데이터 타입 사용¶

서비스 정의 파일 역시 메시지와 동일하게 다양한 기본 타입, 배열, 복합 타입 등을 사용할 수 있다. 예를 들어, 이동 명령을 요청하고 결과 좌표를 받는 서비스를 정의해 보자. srv 디렉터리에 MoveTo.srv라는 파일을 작성하면 다음과 같은 형식이 가능하다.

float64 x
float64 y
float64 z
---
float64[3] final_position
bool arrived
  • 위 Request 영역에는 이동하려는 목표 위치인 $(x, y, z)$가 정의된다.
  • Response 영역에는 최종 도달 좌표인 float64[3] final_position과 성공 여부 bool arrived가 정의된다.

서비스 사용 시 주의 사항¶

  • 서비스는 요청-응답 구조이므로 클라이언트가 요청을 보낸 후 서버의 응답을 기다려야 한다. 이로 인해 호출이 블로킹(Blocking)될 수 있음을 고려해야 한다.
  • 특정 노드가 서비스 서버 역할을 하지 않으면, 해당 서비스는 사용 불가능하다. 서비스 서버와 클라이언트가 모두 존재해야 정상 동작한다.
  • 메시지와 마찬가지로 .srv 파일 수정 후 빌드 과정을 거쳐야 한다.

액션 인터페이스(.action) 정의¶

액션은 목표(Goal)를 전달하고, 중간 피드백(Feedback)을 수신하며, 최종 결과(Result)를 수신하는 비동기적 통신 방식을 제공한다. 이동, 탐색처럼 장시간이 소요되는 작업을 수행할 때 유용하다.

액션 파일 기본 형식¶

액션 정의 파일은 .action 확장자를 가지며, 세 영역으로 구성된다. 각 영역은 “---”로 구분한다.

  1. Goal
  2. Result
  3. Feedback

예를 들어 action 디렉터리에 MoveRobot.action이라는 파일을 정의한다고 해보자.

# Goal
float64 x
float64 y
float64 z
---
# Result
bool success
string message
---
# Feedback
float64 progress

위와 같이 작성하면:

  • Goal 영역에는 로봇이 이동해야 할 목표 좌표 $(x, y, z)$가 정의된다.
  • Result 영역에는 작업 완료 시점에서 동작 성공 여부(bool success)와 메시지(string message)가 정의된다.
  • Feedback 영역에는 작업 도중 진행 상황(float64 progress)을 알려주는 정보를 담는다.

액션 사용 시 특징¶

  • Goal: 액션 클라이언트가 액션 서버로 목표를 보낼 때 사용하는 구조.
  • Result: 작업이 완료된 시점에 액션 서버가 액션 클라이언트에게 반환하는 결과.
  • Feedback: 작업이 진행되는 동안 액션 서버가 주기적으로 액션 클라이언트에게 전달할 수 있는 중간 상태 정보.

액션 클라이언트는 Goal을 전송한 후, 피드백을 구독(subscribe)하여 중간 상황을 모니터링하며, 최종적으로 작업이 끝나면 서버가 Result를 반환한다.

액션 메시지 타입¶

액션이 정의되면 ROS2 빌드 시스템은 해당 액션을 위한 메시지, 서비스, 그리고 필요한 기타 인터페이스를 자동으로 생성한다. 즉, .action 파일 하나에서 Goal, Result, Feedback에 필요한 구조체(또는 클래스)와 함께 다음의 구조도 생성된다.

  • Goal 요청을 위한 서비스
  • Goal 취소 요청을 위한 서비스
  • Goal 상태 확인을 위한 토픽
  • Feedback 데이터를 주고받기 위한 토픽

이를 통해 한 번의 인터페이스 정의로 액션 시스템에 필요한 모든 기능을 구현할 수 있다.

예시: 위치 이동 액션¶

위 예시 MoveRobot.action은 특정 위치로 이동하는 액션 인터페이스를 정의했다. 이 액션을 사용하는 로봇 노드가 있다면, 아래와 같은 흐름을 가지게 된다.

  1. 액션 클라이언트에서 (x, y, z)를 Goal로 설정해 액션 서버에 전송한다.
  2. 액션 서버는 해당 목표에 맞춰 로봇 이동을 시작하고, 일정 주기로 Feedback을 전송한다. 예를 들어 이동 진행 상황을 progress(0.0~1.0)로 나타낼 수 있다.
  3. 만약 다른 요인으로 인해 이동을 취소해야 한다면 액션 취소 서비스를 호출해 요청할 수 있다.
  4. 최종적으로 이동이 완료되거나 실패하면 액션 서버는 Result로 성공 여부와 메시지를 반환한다.

액션은 중간 피드백을 제공할 수 있다는 점에서 서비스와의 큰 차이를 보인다. 장시간 걸리는 작업에서 현재 진행 상황을 파악하고 취소를 처리할 수 있다는 이점이 있다.

빌드 설정과 의존성 관리¶

ROS2에서 메시지, 서비스, 액션 인터페이스를 정의했으면 이를 사용하기 위한 빌드 설정을 진행해야 한다. 다음과 같은 두 가지 파일이 핵심이다.

  1. CMakeLists.txt: CMake 기반 빌드 스크립트
  2. package.xml: ROS2 패키지 메타정보 및 의존성 관리

이 두 파일에 필요한 설정을 정확히 해줘야, .msg, .srv, .action 파일이 올바르게 빌드되고 다른 패키지에서 사용할 수 있는 형식으로 배포된다.

package.xml 설정¶

package.xml에는 메시지나 서비스, 액션을 빌드하고 사용하기 위해 필요한 의존성을 명시한다. 아래는 예시 스니펫이다.

<package format="3">
  <name>my_robot_interfaces</name>
  <version>0.0.1</version>
  <description>My custom interfaces for robot</description>
  <maintainer email="user@todo.todo">TODO: Maintainer Name</maintainer>
  <license>Apache License 2.0</license>

  <!-- 빌드 시 의존성 -->
  <buildtool_depend>ament_cmake</buildtool_depend>
  <build_depend>rosidl_default_generators</build_depend>
  <build_depend>rosidl_default_runtime</build_depend>

  <!-- 실행 시 의존성 -->
  <exec_depend>rosidl_default_runtime</exec_depend>

  <!-- 다른 의존성 패키지가 있다면 여기에 추가 -->
  <!-- 예: <build_depend>geometry_msgs</build_depend> 등 -->

</package>
  • <buildtool_depend>ament_cmake</buildtool_depend>: colcon 빌드 시스템과 함께 CMake를 사용하기 위해 필요한 의존성.
  • <build_depend>rosidl_default_generators</build_depend>: .msg, .srv, .action 인터페이스 파일을 C++/Python용으로 자동 생성하기 위한 빌드 의존성.
  • <exec_depend>rosidl_default_runtime</exec_depend>: 런타임에 메시지, 서비스, 액션 타입을 사용하기 위한 의존성.

CMakeLists.txt 설정¶

CMakeLists.txt 파일에서는 rosidl_default_generators를 이용해 .msg, .srv, .action 파일을 빌드 대상에 포함시킨다. 예시는 다음과 같다.

cmake_minimum_required(VERSION 3.8)
project(my_robot_interfaces)

find_package(ament_cmake REQUIRED)
find_package(rosidl_default_generators REQUIRED)

# 메시지, 서비스, 액션 파일 지정
set(MSG_FILES
  "msg/DistanceSensor.msg"
)

set(SRV_FILES
  "srv/SetLedState.srv"
)

set(ACTION_FILES
  "action/MoveRobot.action"
)

# 인터페이스 생성을 rosidl_default_generators에 넘김
rosidl_generate_interfaces(${PROJECT_NAME}
  ${MSG_FILES}
  ${SRV_FILES}
  ${ACTION_FILES}
)

ament_package()
  • find_package(rosidl_default_generators REQUIRED): 메시지, 서비스, 액션을 자동 생성하기 위한 패키지 검색.
  • rosidl_generate_interfaces(${PROJECT_NAME} ...): 지정된 .msg, .srv, .action 파일을 기반으로 언어별 인터페이스 소스 코드를 생성하도록 설정.

일반적으로는 메시지, 서비스, 액션을 정의하는 인터페이스 전용 패키지라면 위와 같이 최소한의 설정만으로 충분하다. 만약 메시지와 서비스를 사용하는 실제 소스 파일(C++/Python)이 패키지 내부에 포함되어 있다면, 그에 맞춰 추가 설정을 할 수 있다.

C++ 코드에서 인터페이스 사용 예시¶

인터페이스 패키지가 정상적으로 빌드된 후에는, 이를 사용하는 다른 패키지(또는 같은 패키지)의 C++ 노드에서 .msg, .srv, .action 타입을 임포트할 수 있다. 아래 예시는 메시지를 퍼블리시(publish)하고, 서비스를 제공(서버)하고, 액션 서버를 구현하는 간단한 코드 사례를 보여준다.

#include <rclcpp/rclcpp.hpp>
// my_robot_interfaces 패키지에서 정의한 메시지, 서비스, 액션 헤더
#include "my_robot_interfaces/msg/distance_sensor.hpp"
#include "my_robot_interfaces/srv/set_led_state.hpp"
#include "my_robot_interfaces/action/move_robot.hpp"

using namespace std::chrono_literals;

class MyRobotNode : public rclcpp::Node
{
public:
  MyRobotNode()
  : Node("my_robot_node")
  {
    // 메시지 퍼블리셔
    distance_sensor_pub_ = this->create_publisher<my_robot_interfaces::msg::DistanceSensor>(
      "distance_sensor", 10);

    // 서비스 서버
    set_led_state_srv_ = this->create_service<my_robot_interfaces::srv::SetLedState>(
      "set_led_state",
      std::bind(&MyRobotNode::setLedStateCallback, this, std::placeholders::_1, std::placeholders::_2));

    // 액션 서버
    move_robot_action_server_ = rclcpp_action::create_server<my_robot_interfaces::action::MoveRobot>(
      this,
      "move_robot",
      std::bind(&MyRobotNode::handleGoal, this, std::placeholders::_1, std::placeholders::_2),
      std::bind(&MyRobotNode::handleCancel, this, std::placeholders::_1),
      std::bind(&MyRobotNode::handleAccepted, this, std::placeholders::_1)
    );

    // 타이머로 주기적으로 메시지 퍼블리시
    timer_ = this->create_wall_timer(
      1s,
      std::bind(&MyRobotNode::publishDistanceSensor, this)
    );
  }

private:
  // 메시지 퍼블리시 예시
  void publishDistanceSensor()
  {
    auto msg = my_robot_interfaces::msg::DistanceSensor();
    msg.min_range = 0.1f;
    msg.max_range = 10.0f;
    msg.current_distance = 3.14f;
    distance_sensor_pub_->publish(msg);
  }

  // 서비스 서버 콜백
  void setLedStateCallback(
    const std::shared_ptr<my_robot_interfaces::srv::SetLedState::Request> request,
    std::shared_ptr<my_robot_interfaces::srv::SetLedState::Response> response)
  {
    RCLCPP_INFO(this->get_logger(), "LED %d을(를) %s 상태로 설정",
      request->led_number, request->state ? "ON" : "OFF");
    // 실제 로직은 생략...
    response->success = true;
    response->message = "LED state changed successfully.";
  }

  // 액션 서버: Goal 수락 처리
  rclcpp_action::GoalResponse handleGoal(
    const rclcpp_action::GoalUUID & uuid,
    std::shared_ptr<const my_robot_interfaces::action::MoveRobot::Goal> goal)
  {
    RCLCPP_INFO(this->get_logger(), "새 Goal 수신: (%.2f, %.2f, %.2f)",
      goal->x, goal->y, goal->z);
    // Goal을 수락할지 거절할지 결정
    (void)uuid;
    return rclcpp_action::GoalResponse::ACCEPT_AND_EXECUTE;
  }

  // 액션 서버: Goal 취소 처리
  rclcpp_action::CancelResponse handleCancel(
    const std::shared_ptr<rclcpp_action::ServerGoalHandle<my_robot_interfaces::action::MoveRobot>> goal_handle)
  {
    RCLCPP_INFO(this->get_logger(), "Goal 취소 요청 수신");
    (void)goal_handle;
    return rclcpp_action::CancelResponse::ACCEPT;
  }

  // 액션 서버: Goal 수락 시 비동기로 동작을 시작
  void handleAccepted(
    const std::shared_ptr<rclcpp_action::ServerGoalHandle<my_robot_interfaces::action::MoveRobot>> goal_handle)
  {
    using namespace std::placeholders;
    // 별도의 스레드에서 실행
    std::thread{std::bind(&MyRobotNode::executeMoveRobot, this, _1), goal_handle}.detach();
  }

  // 실제 액션 동작
  void executeMoveRobot(
    const std::shared_ptr<rclcpp_action::ServerGoalHandle<my_robot_interfaces::action::MoveRobot>> goal_handle)
  {
    rclcpp::Rate r(1);
    auto feedback = std::make_shared<my_robot_interfaces::action::MoveRobot::Feedback>();
    auto result = std::make_shared<my_robot_interfaces::action::MoveRobot::Result>();

    // 예: 5초 동안 진행 상황을 0.0에서 1.0까지 업데이트
    for (int i = 0; i <= 5; ++i) {
      // 취소 요청 확인
      if (goal_handle->is_canceling()) {
        RCLCPP_INFO(this->get_logger(), "Goal 취소됨");
        result->success = false;
        result->message = "Cancelled";
        goal_handle->canceled(result);
        return;
      }

      // 피드백 전송
      feedback->progress = static_cast<double>(i) / 5.0;
      goal_handle->publish_feedback(feedback);
      r.sleep();
    }

    // 완료 처리
    result->success = true;
    result->message = "MoveRobot 완료";
    goal_handle->succeed(result);
    RCLCPP_INFO(this->get_logger(), "MoveRobot 액션 성공");
  }

  rclcpp::Publisher<my_robot_interfaces::msg::DistanceSensor>::SharedPtr distance_sensor_pub_;
  rclcpp::Service<my_robot_interfaces::srv::SetLedState>::SharedPtr set_led_state_srv_;

  using MoveRobotActionServer = rclcpp_action::Server<my_robot_interfaces::action::MoveRobot>;
  rclcpp_action::Server<my_robot_interfaces::action::MoveRobot>::SharedPtr move_robot_action_server_;

  rclcpp::TimerBase::SharedPtr timer_;
};

int main(int argc, char **argv)
{
  rclcpp::init(argc, argv);
  auto node = std::make_shared<MyRobotNode>();
  rclcpp::spin(node);
  rclcpp::shutdown();
  return 0;
}
  1. #include "my_robot_interfaces/msg/distance_sensor.hpp"와 같은 형태로, ROS2 메시지 헤더를 사용할 수 있다.
  2. create_publisher, create_service, create_server 등 ROS2의 다양한 함수들을 통해 메시지, 서비스, 액션에 대한 핸들을 생성한다.
  3. 메시지 퍼블리시, 서비스 콜백, 액션 서버의 Goal/Feedback/Result 처리 방식을 각각 구현한다.

실제 로봇 제어나 센서 측정 로직은 프로젝트 요구 사항에 맞추어 작성하면 된다.

Python 코드에서 인터페이스 사용 예시¶

ROS2는 Python으로도 노드를 작성할 수 있으며, 메시지, 서비스, 액션 인터페이스는 모두 자동으로 Python 모듈이 생성된다. 일반적으로 my_robot_interfaces 패키지의 메시지, 서비스, 액션을 사용하려면 다음과 같이 import 할 수 있다.

import rclpy
from rclpy.node import Node

# 메시지, 서비스, 액션 타입 임포트
from my_robot_interfaces.msg import DistanceSensor
from my_robot_interfaces.srv import SetLedState
from my_robot_interfaces.action import MoveRobot
from rclpy.action import ActionServer, CancelResponse, GoalResponse

class MyRobotPyNode(Node):
    def __init__(self):
        super().__init__('my_robot_py_node')

        # 메시지 퍼블리셔
        self.distance_sensor_pub = self.create_publisher(DistanceSensor, 'distance_sensor', 10)

        # 서비스 서버
        self.set_led_state_srv = self.create_service(
            SetLedState,
            'set_led_state',
            self.set_led_state_callback
        )

        # 액션 서버
        self.move_robot_action_srv = ActionServer(
            self,
            MoveRobot,
            'move_robot',
            goal_callback=self.goal_callback,
            cancel_callback=self.cancel_callback,
            execute_callback=self.execute_callback
        )

        # 주기적으로 퍼블리시
        self.timer = self.create_timer(1.0, self.publish_distance_sensor)

    def publish_distance_sensor(self):
        msg = DistanceSensor()
        msg.min_range = 0.1
        msg.max_range = 10.0
        msg.current_distance = 3.14
        self.distance_sensor_pub.publish(msg)

    def set_led_state_callback(self, request, response):
        self.get_logger().info(f"LED {request.led_number}를 {'ON' if request.state else 'OFF'}로 설정")
        # 실제 로직은 생략...
        response.success = True
        response.message = "LED 상태 변경 성공"
        return response

    def goal_callback(self, goal_request):
        self.get_logger().info(f"새 Goal 수신: ({goal_request.x}, {goal_request.y}, {goal_request.z})")
        return GoalResponse.ACCEPT

    def cancel_callback(self, goal_handle):
        self.get_logger().info("Goal 취소 요청 수신")
        return CancelResponse.ACCEPT

    async def execute_callback(self, goal_handle):
        feedback_msg = MoveRobot.Feedback()
        result_msg = MoveRobot.Result()

        for i in range(6):
            if goal_handle.is_cancel_requested:
                self.get_logger().info("Goal 취소됨")
                result_msg.success = False
                result_msg.message = "Cancelled"
                goal_handle.canceled()
                return result_msg

            feedback_msg.progress = float(i) / 5.0
            goal_handle.publish_feedback(feedback_msg)
            await rclpy.sleep(1.0)

        result_msg.success = True
        result_msg.message = "MoveRobot 완료"
        goal_handle.succeed()
        return result_msg

def main(args=None):
    rclpy.init(args=args)
    node = MyRobotPyNode()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()
  1. from my_robot_interfaces.msg import DistanceSensor와 같이 생성된 메시지 클래스를 import 한다.
  2. Python 노드에서도 동일하게 create_publisher, create_service, ActionServer 등을 사용해 메시지 퍼블리시, 서비스 제공, 액션 서버 동작을 구현한다.
  3. Python에서는 async/await를 통해 비동기 액션 실행을 간단하게 표현할 수 있다.

인터페이스 변경 시 주의 사항¶

ROS2에서 .msg, .srv, .action 파일을 수정하면, 반드시 다시 빌드(colcon build)를 실행해야 한다. 그래야 수정된 인터페이스에 맞춰 C++/Python 등의 언어 바인딩 코드가 재생성된다. 빌드 후에는 패키지 내의 install/my_robot_interfaces/include/my_robot_interfaces/msg/, install/my_robot_interfaces/include/my_robot_interfaces/srv/, install/my_robot_interfaces/include/my_robot_interfaces/action/ 등으로 생성된 헤더 파일들을 확인할 수 있으며, Python 모듈은 install/my_robot_interfaces/lib/pythonX.X/site-packages/my_robot_interfaces/msg/ 등지에 생성된다.

만약 이미 빌드된 인터페이스를 다른 패키지가 사용하고 있다면, 해당 패키지들도 인터페이스 변경분을 반영하기 위해 재빌드가 필요하다. 특히 서비스나 액션 구조가 바뀔 경우(필드 이름, 필드 타입 추가/삭제 등), 기존 코드를 사용하는 모든 노드가 수정된 인터페이스에 맞춰져야만 정상 동작한다.

인터페이스 간 의존성¶

여러 개의 메시지, 서비스, 액션 파일에서 다른 패키지(예: geometry_msgs, sensor_msgs)에 정의된 타입을 참조할 수 있다. 예를 들어, geometry_msgs/msg/Point를 메시지 필드로 사용하려면 다음과 같이 .msg 파일에서 선언할 수 있다.

geometry_msgs/Point position

이 경우, package.xml과 CMakeLists.txt에 geometry_msgs를 의존성으로 추가해야 한다. 예를 들어 package.xml에는 다음처럼 의존성을 명시한다.

<build_depend>geometry_msgs</build_depend>
<exec_depend>geometry_msgs</exec_depend>

그리고 CMakeLists.txt에서는:

find_package(geometry_msgs REQUIRED)

rosidl_generate_interfaces(${PROJECT_NAME}
  ...
  DEPENDENCIES geometry_msgs
)

이를 통해 rosidl_default_generators가 geometry_msgs 타입 정보를 찾아, 해당 필드를 해석할 수 있게 된다.

인터페이스 활용 흐름(예시)¶

다음은 액션 클라이언트-서버 간 상호 작용 흐름을 mermaid로 간단히 표현한 예시다.

sequenceDiagram participant Client participant ActionServer Client->>ActionServer: Goal 요청(Goal) alt 서버가 Goal 수락 ActionServer-->>Client: GoalAccepted loop 작업 진행 중 ActionServer-->>Client: Feedback end ActionServer-->>Client: Result else 서버가 Goal 거부 ActionServer-->>Client: GoalRejected end
  1. 액션 클라이언트가 Goal을 서버로 전송한다.
  2. 액션 서버는 Goal을 수락(ACCEPT)하거나 거절(REJECT)한다.
  3. 수락 시, 서버는 작업을 진행하면서 중간 Feedback을 전송한다.
  4. 작업이 종료되면 서버가 최종 Result를 반환한다.

정리¶

이와 같이 메시지, 서비스, 액션 인터페이스를 정의하고, 패키지 빌드 설정을 마친 뒤, C++/Python 노드에서 각 인터페이스를 활용하여 통신을 구현할 수 있다. 인터페이스를 효율적으로 관리하면 여러 노드 간 데이터 교환 형식을 일관성 있게 유지하면서 손쉽게 확장할 수 있게 된다.
