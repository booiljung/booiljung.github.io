패키지 생성 기본 명령과 템플릿 구조¶

ROS2에서 패키지를 생성하기 위해서는 단순히 디렉터리를 생성하고 필요한 파일들을 배치하는 것만으로는 충분하지 않다. ROS2는 패키지를 구성할 때 일종의 표준적 규칙을 따른다. 이 규칙에는 패키지 메타정보(package.xml 파일)와 빌드 시스템 설정(CMakeLists.txt 등)이 포함되며, 패키지를 이용해 노드를 빌드하거나 실행할 때 필요한 정보를 제공한다.

여기서는 ROS2 Humble에서 가장 기초가 되는 패키지 생성 명령어와, 이를 통해 자동으로 생성되는 템플릿 구조에 대해 엄밀하게 살펴본다.

패키지 생성 명령¶

ROS2 Humble에서는 일반적으로 다음과 같이 ros2 pkg create 명령을 사용해 패키지를 생성한다.

ros2 pkg create <패키지명> [옵션들]

가장 기본적인 구조를 빠르게 생성하는 옵션은 다음과 같다.

  • --build-type
  • ament_cmake 또는 ament_python 등을 지정하여, 패키지가 어떤 빌드 시스템을 사용하는지 정한다.
  • --dependencies
  • 패키지가 의존할 ROS2 패키지들을 쉼표(,) 또는 공백으로 구분하여 지정한다.
  • --cpp-node-name / --python-node-name
  • 패키지 안에 기본 템플릿 노드(C++ 또는 Python)를 생성할 때 사용하는 옵션이다.

예시로, C++ 기반의 ament_cmake 패키지를 생성하려면 다음과 같은 명령을 사용할 수 있다.

ros2 pkg create my_cpp_pkg --build-type ament_cmake --dependencies rclcpp std_msgs

위 명령을 실행하면 my_cpp_pkg라는 새 디렉터리가 생성되고, package.xml, CMakeLists.txt, src, include 등의 디렉터리 혹은 파일들이 자동으로 생성된다. 또한 rclcpp, std_msgs를 의존 패키지로 명시하며, C++ 코드를 빌드할 수 있는 템플릿을 기본으로 제공한다.

기본 템플릿 구조¶

ros2 pkg create 명령으로 패키지를 생성하면, 아래와 같은 형태의 디렉터리 및 파일 구조를 확인할 수 있다(ament_cmake 예시).

my_cpp_pkg
├── CMakeLists.txt
├── package.xml
├── include
│   └── my_cpp_pkg
│       └── ...
└── src
    └── my_cpp_pkg
        └── ...
  • CMakeLists.txt ROS2 패키지 빌드 시스템에 필요한 설정을 선언하고 기술한다. 예를 들어 find_package(rclcpp REQUIRED), add_executable(...), ament_target_dependencies(...) 등을 포함한다.
  • package.xml 패키지의 메타정보를 기록한다. 패키지 이름, 버전, 라이선스, 의존 패키지, 유지보수자 정보 등 ROS2가 패키지를 인식하고 의존성을 파악하는 데 필요한 정보를 기입한다.
  • include 디렉터리 C++ 헤더 파일을 관리하는 디렉터리이다. my_cpp_pkg/include/my_cpp_pkg/ 형태로 구성되어, 이 디렉터리에 C++ 헤더(.hpp) 파일이 위치하게 된다.
  • src 디렉터리 C++ 소스 파일을 관리하는 디렉터리이다. my_cpp_pkg/src/my_cpp_pkg/ 형태로 구성되어, 이 디렉터리에 C++ 소스(.cpp) 파일이 위치하게 된다.

이후 필요에 따라 launch, test 등의 디렉터리를 추가로 생성해서 패키지를 확장할 수 있다.

package.xml 파일 구조¶

ROS2에서 필수적인 정보 중 하나는 package.xml이다. 자동 생성된 package.xml의 기본 구조는 아래와 비슷하다.

<package format="3">
  <name>my_cpp_pkg</name>
  <version>0.0.0</version>
  <description>Package Description</description>
  <maintainer email="user@todo.todo">user</maintainer>
  <license>Apache License 2.0</license>

  <buildtool_depend>ament_cmake</buildtool_depend>
  <build_depend>rclcpp</build_depend>
  <build_depend>std_msgs</build_depend>

  <exec_depend>rclcpp</exec_depend>
  <exec_depend>std_msgs</exec_depend>

  <test_depend>ament_cmake_gtest</test_depend>

  <export>
    <build_type>ament_cmake</build_type>
  </export>
</package>
  • : 패키지 이름을 나타낸다. 여기서는 my_cpp_pkg로 설정되어 있다.
  • : 패키지의 버전 정보를 나타낸다.
  • : 패키지에 대한 간단한 설명을 담는다.
  • : 패키지를 유지보수하는 사람의 정보를 담는다.
  • : 패키지에 적용되는 라이선스를 적는다. 예: Apache License 2.0, BSD, MIT, GPLv3 등.
  • : 빌드 도구 의존성을 명시한다. ament_cmake, ament_python 등이 주로 여기에 들어간다.
  • , : 패키지를 빌드하거나 실행하기 위해 필요한 ROS2 패키지 의존성을 나눈다. 예: rclcpp, std_msgs, sensor_msgs 등.
  • : 패키지 테스트에 필요한 의존성을 명시한다.
  • : 추가 정보를 제공한다. 예: <build_type>ament_cmake</build_type>를 통해 빌드 유형을 명시한다.

CMakeLists.txt 주요 항목¶

CMakeLists.txt는 ament_cmake 패키지에서 빌드를 구성하는 핵심 파일이다. 자동 생성된 템플릿에는 아래와 같은 내용이 포함된다.

cmake_minimum_required(VERSION 3.8)
project(my_cpp_pkg)

find_package(ament_cmake REQUIRED)
find_package(rclcpp REQUIRED)
find_package(std_msgs REQUIRED)

add_executable(talker src/publisher_member_function.cpp)
ament_target_dependencies(talker rclcpp std_msgs)

install(TARGETS
  talker
  DESTINATION lib/${PROJECT_NAME}
)

ament_package()
  • cmake_minimum_required(VERSION 3.8) CMake 버전을 지정한다.
  • project(my_cpp_pkg) 프로젝트 이름을 지정한다. ROS2 패키지 이름과 동일하거나 유사하게 작성한다.
  • find_package(...) 필요한 패키지를 찾는다. 이를 통해 CMake에서 해당 패키지의 헤더 파일, 라이브러리를 인식하고 연결할 수 있다.
  • add_executable(이름 소스파일) 실행 파일을 생성한다. C++ 소스 파일을 바탕으로 빌드된다.
  • ament_target_dependencies(타겟 패키지들) 타겟에 의존성을 연결한다. rclcpp, std_msgs 등 필요한 ROS2 라이브러리를 타겟에 반영한다.
  • install(...) 빌드 후 생성된 실행 파일을 설치할 위치를 지정한다. ROS2는 일반적으로 lib/<패키지명> 디렉터리 아래에 설치한다.
  • ament_package() ament_cmake 패키지에서 필수로 호출하는 매크로이며, ROS2 패키지로서 인식되기 위한 정보를 마무리 짓는다.

ament_cmake vs ament_python¶

ROS2 패키지는 빌드 방식에 따라 크게 ament_cmake와 ament_python(또는 colcon_python_setup) 방식으로 나뉜다.

  • ament_cmake
  • CMake 기반 프로젝트를 빌드하며, 주로 C++ 코드로 작성된 노드를 포함한다.
  • ament_python
  • Python 기반 프로젝트를 빌드하며, setup.py 등을 활용해 파이썬 스크립트를 설치 및 배포한다.

예시 명령어는 다음과 같다.

ros2 pkg create my_py_pkg --build-type ament_python --dependencies rclpy std_msgs

이 경우 my_py_pkg 디렉터리 내에 setup.py, package.xml 파일이 생성되고, Python 스크립트용 템플릿 구조가 포함된다.

템플릿 구조 시각화¶

아래는 ament_cmake 방식 패키지의 파일 구조를 간략하게 시각화한 다이어그램 예시다.

graph LR A[my_cpp_pkg] --> B[CMakeLists.txt] A --> C[package.xml] A --> D[include/] A --> E[src/] D --> F[my_cpp_pkg/] E --> G[my_cpp_pkg/]

ament_python 패키지 구조¶

앞서 살펴본 ament_cmake 방식과 달리, Python 기반 노드를 만들 때는 ament_python 빌드 유형을 사용한다. 다음은 ros2 pkg create를 통해 Python 기반 패키지를 생성할 때의 명령 예시다.

ros2 pkg create my_py_pkg --build-type ament_python --dependencies rclpy std_msgs

위 명령을 수행하면, my_py_pkg라는 디렉터리가 생성되고 그 안에 파이썬 패키지를 구성할 수 있는 기본 파일들이 자동으로 생성된다. 구조는 대략 아래와 같다.

my_py_pkg
├── package.xml
├── setup.py
├── my_py_pkg
│   └── __init__.py
└── resource
    └── my_py_pkg
  • package.xml ament_cmake 방식과 마찬가지로 패키지 메타정보가 들어 있다. 빌드 방식이 ament_python인 점만 다르며, rclpy 등 파이썬 패키지 의존성이 명시된다.
  • setup.py 파이썬 패키지 설치 정보를 담는다. 일반적으로 ROS2 파이썬 패키지에서 스크립트와 노드를 설치·관리하는 데 사용된다.
  • my_py_pkg/*init*.py 파이썬 패키지 초기화 파일이다. 보통은 비어 있거나, 패키지 전역에서 필요한 내용을 정의한다.
  • resource/my_py_pkg ROS2 빌드시 내부적으로 사용하는 리소스 파일이다. Python 패키지 식별자 역할을 하며, ament_python 환경에서 패키지로서 등록된다.

setup.py 상세 항목¶

ament_python 패키지에서 가장 중요한 파일은 setup.py이다. 이 파일에서는 패키지 이름, 버전, 파이썬 의존성 등을 정의하며, ROS2 런타임 환경과 연동하기 위한 설정도 추가된다. 자동 생성된 템플릿 예시는 다음과 비슷하다.

from setuptools import setup

package_name = 'my_py_pkg'

setup(
    name=package_name,
    version='0.0.0',
    packages=[package_name],
    data_files=[
        ('share/ament_index/resource_index/packages',
            ['resource/' + package_name]),
        ('share/' + package_name, ['package.xml']),
    ],
    install_requires=['setuptools'],
    zip_safe=True,
    maintainer='Your Name',
    maintainer_email='your_email@example.com',
    description='TODO: Package description',
    license='Apache License 2.0',
    tests_require=['pytest'],
    entry_points={
        'console_scripts': [
            'talker = my_py_pkg.talker:main',
            'listener = my_py_pkg.listener:main',
        ],
    },
)
  • packages=[package_name] 설치할 파이썬 모듈(패키지) 이름을 설정한다.
  • data_files ROS2 패키지를 인식하기 위해 필요한 resource/<package_name> 파일과 package.xml을 지정한다.
  • install_requires=['setuptools'] Python 설치 시 필요한 의존 패키지를 정의한다. 여기서는 setuptools가 필수로 포함된다.
  • entry_points ROS2 실행 가능 스크립트를 정의하는 부분이다. 예를 들어 talker = my_py_pkg.talker:main은 ros2 run my_py_pkg talker 명령을 통해 my_py_pkg/talker.py의 main() 함수를 실행하도록 설정한다.

ament_python 배포 및 실행¶

위 구조에서 별도의 빌드 과정 없이도, ROS2 패키지를 설치·배포하기 위해서는 colcon을 활용하여 다음과 같이 빌드한다.

colcon build --packages-select my_py_pkg

이후 아래 명령으로 파이썬 노드를 실행할 수 있다.

. install/setup.bash
ros2 run my_py_pkg talker

ament_python 패키지에서 정의한 스크립트가 실제 노드로 동작하게 되는 원리를 체험해볼 수 있다.

심화: package.xml에서의 추가 의존성 처리¶

ROS2에서는 단순히 rclcpp나 rclpy 같은 ROS2 핵심 패키지만이 아니라, 사용자 정의 메시지 타입 패키지, 외부 라이브러리 등 다양한 의존성을 활용할 수 있다. 예를 들어 아래처럼 sensor_msgs와 geometry_msgs를 추가로 의존하려면 package.xml에 다음과 같이 작성한다.

<build_depend>sensor_msgs</build_depend>
<build_depend>geometry_msgs</build_depend>
<exec_depend>sensor_msgs</exec_depend>
<exec_depend>geometry_msgs</exec_depend>

그리고 ament_cmake 패키지라면 CMakeLists.txt에도 다음과 같이 의존성을 추가한다.

find_package(sensor_msgs REQUIRED)
find_package(geometry_msgs REQUIRED)

add_executable(my_node src/my_node.cpp)
ament_target_dependencies(my_node rclcpp sensor_msgs geometry_msgs)

이처럼 package.xml과 CMakeLists.txt(또는 setup.py)에 일치하는 의존성을 설정해야 정상적인 빌드 및 실행이 가능하다.

메시지/서비스/액션 인터페이스와의 연계¶

ROS2 패키지에서 메시지, 서비스, 액션 인터페이스를 직접 정의할 수도 있다. 예를 들어, msg 디렉터리 안에 MyMessage.msg를 생성하여 사용자 정의 메시지를 추가하려면:

msg 디렉터리 생성:

mkdir -p my_cpp_pkg/msg

MyMessage.msg 파일 작성:

float32 x
float32 y
string name

package.xml 에 관련 의존성 추가:

<build_depend>rosidl_default_generators</build_depend>
<build_depend>rosidl_default_runtime</build_depend>
<exec_depend>rosidl_default_runtime</exec_depend>

CMakeLists.txt 수정:

find_package(rosidl_default_generators REQUIRED)

rosidl_generate_interfaces(${PROJECT_NAME}
  "msg/MyMessage.msg"
  DEPENDENCIES builtin_interfaces
)

ament_export_dependencies(rosidl_default_runtime)
ament_package()

위와 같은 설정을 통해, 새로 정의한 메시지 타입을 빌드 시 자동으로 C++ 또는 Python 등 언어별로 바인딩하여 사용할 수 있다. 이 과정을 포함해도 결국 패키지 구조의 근간은 package.xml과 빌드 시스템(ament_cmake/ament_python) 설정으로 일관성을 유지한다.

템플릿 활용 시 주의 사항¶

ros2 pkg create 명령으로 생성된 템플릿은 편리하지만, 프로젝트 요구 사항에 따라 세부 구성을 적절히 수정해야 한다. 예컨대:

  • 라이선스 정보 수정 기본 템플릿에 포함된 Apache License 2.0 대신 다른 라이선스를 적용하려면 package.xml과 setup.py(또는 CMakeLists.txt) 양쪽에 라이선스 정보를 일관되게 수정한다.
  • 유지보수자 정보 <maintainer> 태그의 이메일 주소나 이름 등을 실제 프로젝트 환경에 맞춰 변경해야 한다.
  • 패키지 버전 <version>0.0.0</version> 형태로 생성되므로, 버저닝 규칙에 따라 적절히 갱신한다.

로컬 및 글로벌 패키지 설치 위치¶

ROS2 패키지는 일반적으로 작업공간(workspace) 내에서 colcon을 통해 빌드된 뒤, install 디렉터리에 설치된다. 이와 별개로 시스템 전역 /opt/ros/humble 경로 등에 설치된 기본 ROS2 패키지들과 함께 사용될 수 있다.

  • 로컬 설치 동일한 workspace 안에서 개발한 ROS2 패키지들을 모아두고, colcon build를 수행하면, install/<패키지명> 구조로 결과물이 배치된다.
  • 글로벌 설치 ROS2 공식 저장소나 apt를 통해 설치한 패키지들은 /opt/ros/humble/... 아래에 존재한다. 일반적으로 로컬 workspace에서 빌드한 패키지와 중복되지 않는다면, 두 위치에서 설치된 패키지들을 함께 사용할 수 있다.

다양한 옵션 활용¶

ros2 pkg create 명령에서는 생성 시점에 다양한 옵션을 넣어 더 구체화할 수 있다. 예를 들어, 노드 파일을 자동으로 만들어주도록 요청할 수도 있다.

ros2 pkg create my_cpp_pkg --build-type ament_cmake --dependencies rclcpp std_msgs --cpp-node-name my_node

이 경우, my_node.cpp 템플릿 파일이 자동으로 생성되며, CMakeLists.txt에도 간단한 예시용 실행 파일 추가 코드가 들어간다.

아래는 --cpp-node-name 사용 시 생성되는 기본 템플릿(예시)이다.

#include <chrono>
#include <memory>
#include "rclcpp/rclcpp.hpp"

using namespace std::chrono_literals;

class MyNode : public rclcpp::Node
{
public:
  MyNode()
  : Node("my_node")
  {
    timer_ = this->create_wall_timer(
      1000ms, std::bind(&MyNode::timer_callback, this));
  }

private:
  void timer_callback()
  {
    RCLCPP_INFO(this->get_logger(), "Hello from my_node!");
  }
  rclcpp::TimerBase::SharedPtr timer_;
};

int main(int argc, char * argv[])
{
  rclcpp::init(argc, argv);
  rclcpp::spin(std::make_shared<MyNode>());
  rclcpp::shutdown();
  return 0;
}

노드의 이름, 타이머 콜백 예제 등이 미리 작성되어 있어, 빠르게 시작하기 유용하다.

패키지 명명 규칙과 구조 확장¶

ROS2 패키지를 생성할 때, 패키지 이름과 디렉터리 구조는 프로젝트의 유지보수와 협업에 큰 영향을 준다. ROS2는 엄격한 규칙을 강제하진 않지만, 다음과 같은 명명 규칙과 디렉터리 구조 확장에 대한 권장사항이 있다.

  • 패키지 이름
  • 알파벳 소문자와 언더스코어(_) 조합을 주로 사용하며, 숫자는 필요시에만 사용한다.
  • 가독성을 높이기 위해 길이가 너무 길지 않도록 한다.
  • 예: my_robot_controller, image_processing_utils, lidar_scan_processor
  • 디렉터리 구조 확장
  • launch 디렉터리: ROS2의 Launch 파일(.py 등)을 저장한다. 노드 실행, 파라미터 설정 등을 중앙에서 관리할 때 유용하다.
  • test 디렉터리: 단위 테스트, 통합 테스트 스크립트를 둔다. ament_cmake_gtest나 pytest 등을 사용한다.
  • scripts 디렉터리: 스크립트 형태로 동작하는 파일을 두는 곳이며, 주로 Python 기반 패키지에서 활용한다.
  • config 디렉터리: YAML, XML 등 노드가 참고하는 설정 파일을 두는 디렉터리다.
  • msg, srv, action 디렉터리: 사용자 정의 메시지, 서비스, 액션 인터페이스를 정의할 때 생성한다.

아래 Mermaid 예시는 my_cpp_pkg 패키지에 launch, test, config 디렉터리를 확장한 사례를 보여준다.

graph LR A[my_cpp_pkg] --> B[CMakeLists.txt] A --> C[package.xml] A --> D[include/] A --> E[src/] A --> F[launch/] A --> G[test/] A --> H[config/]

Launch 파일 템플릿¶

ROS2에서는 노드 실행을 중앙에서 제어하기 위해 Launch 파일을 널리 사용한다. 주로 Python 코드로 작성하며, 복수의 노드를 동시에 실행하거나 파라미터를 넘길 때 유용하다. 예시 템플릿은 다음과 같다.

# my_cpp_pkg/launch/example.launch.py
from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
    return LaunchDescription([
        Node(
            package='my_cpp_pkg',
            executable='talker',
            name='my_talker',
            output='screen'
        ),
        Node(
            package='my_cpp_pkg',
            executable='listener',
            name='my_listener',
            output='screen'
        ),
    ])

위 예시에서 my_cpp_pkg 내의 talker, listener 노드를 한꺼번에 실행할 수 있도록 Launch 파일을 구성했다. 이러한 Launch 파일을 통해 노드 간 의존관계, 파라미터 전달, remap 등을 한 곳에서 제어할 수 있다.

테스트 디렉터리 구성¶

ROS2에서 테스트를 진행할 때는 ament_cmake 또는 ament_python 기반으로 각각 다른 방식을 지원한다.

C++ (ament_cmake_gtest):

  • test 디렉터리에 GTest 기반 소스파일(.cpp)을 배치한다.
  • CMakeLists.txt에서 find_package(ament_cmake_gtest REQUIRED) 후 ament_add_gtest() 매크로를 사용한다.

Python (pytest):

  • test 디렉터리에 pytest 기반 스크립트(.py)를 배치한다.
  • setup.py에서 tests_require=['pytest']와 같은 설정을 통해 테스트 의존성을 정의한다.
  • colcon test 명령으로 테스트를 실행할 수 있다.

예시 CMakeLists.txt 설정(ament_cmake_gtest):

find_package(ament_cmake_gtest REQUIRED)
ament_add_gtest(test_mynode test/test_mynode.cpp)
if(TARGET test_mynode)
  target_link_libraries(test_mynode
    my_cpp_pkg_library  # 테스트 대상 라이브러리
  )
endif()

파라미터 및 설정 파일 관리¶

ROS2에서 노드 파라미터를 활용하기 위해 YAML 파일 등을 사용한다. 이를 보통 config 디렉터리에 배치하며, Launch 파일이나 노드 초기화 시점에 로드한다.

# config/my_node_params.yaml
my_node:
  ros__parameters:
    param_1: 42
    param_2: "Hello ROS2"

Launch 파일에서 이를 로드하려면 다음과 같이 작성한다.

# launch/param_example.launch.py
from launch import LaunchDescription
from launch_ros.actions import Node
import os
from ament_index_python.packages import get_package_share_directory

def generate_launch_description():
    config_path = os.path.join(
        get_package_share_directory('my_cpp_pkg'),
        'config',
        'my_node_params.yaml'
    )

    return LaunchDescription([
        Node(
            package='my_cpp_pkg',
            executable='my_node',
            name='my_node_with_params',
            parameters=[config_path],
            output='screen'
        )
    ])

CMakeLists.txt에서 라이브러리 구조 구성¶

C++ 패키지에서 복수의 노드를 작성할 때, 공용 기능을 별도의 라이브러리로 만들어 두고 노드들이 이를 링크하는 방식이 유지보수에 유리하다. 예시로, src/library/ 디렉터리에 공용 코드(.cpp)와 대응하는 헤더(.hpp)를 두고 다음과 같이 CMakeLists.txt를 구성할 수 있다.

# 공용 라이브러리 생성
add_library(my_cpp_pkg_library SHARED
  src/library/common_functions.cpp
)
target_include_directories(my_cpp_pkg_library
  PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
)
ament_target_dependencies(my_cpp_pkg_library rclcpp std_msgs)

# 라이브러리를 링크하는 실행 파일
add_executable(my_node src/my_node.cpp)
target_link_libraries(my_node my_cpp_pkg_library)
ament_target_dependencies(my_node rclcpp std_msgs)

install(TARGETS
  my_cpp_pkg_library
  my_node
  DESTINATION lib/${PROJECT_NAME}
)

이렇게 하면 my_cpp_pkg_library라는 별도 라이브러리를 빌드하고, my_node 실행 파일에서 이를 링크한다. 추후 다른 노드에서도 my_cpp_pkg_library를 공통으로 사용할 수 있다.

ROS2 패키지(ament_cmake)를 빌드하고 실행하는 과정을 시각적으로 나타내면 다음과 같다.

ament_lint와 코드 품질 관리¶

ROS2 패키지를 개발할 때, 단순히 코드를 빌드하고 실행하는 것뿐 아니라, 코드 품질과 일관성을 유지하는 것도 중요하다. ROS2에서는 ament_lint라는 툴셋을 통해 다양한 정적 분석 도구(lint, code style checker 등)를 쉽게 적용할 수 있다. 이를 활용하면 프로젝트 규모가 커졌을 때도 일관된 코드 스타일과 품질을 유지할 수 있다.

ament_lint 패키지 의존성 추가¶

먼저 package.xml에 ament_lint 관련 패키지를 의존성으로 추가해야 한다. C++ 프로젝트(ament_cmake)라면 예를 들어 다음과 같이 작성한다.

<test_depend>ament_lint_auto</test_depend>
<test_depend>ament_lint_common</test_depend>

CMakeLists.txt 설정¶

다음으로 CMakeLists.txt에 ament_lint 관련 매크로를 호출하여 자동으로 체크가 이루어지도록 설정할 수 있다.

find_package(ament_lint_auto REQUIRED)
find_package(ament_lint_common REQUIRED)

ament_lint_auto_find_test_dependencies()

ament_lint_auto_package()
  • ament_lint_auto_find_test_dependencies() ament_lint가 제공하는 lint, cppcheck, uncrustify, cpplint, flake8 등 다양한 분석 도구를 자동으로 불러온다.
  • ament_lint_auto_package() 실제로 lint 검사를 수행하기 위한 필요한 설정을 일괄 적용한다.

이렇게 구성해 두면, colcon test --packages-select <패키지명> 명령을 실행할 때 자동으로 lint 검사도 진행된다. 만약 특정 규칙을 커스터마이징하고 싶다면, .clang-format, .cppcheck, .style.yapf 등의 설정 파일을 패키지 최상단 디렉터리에 배치해두고 원하는 룰을 기입한다.

lint 검사 실행 예시¶

colcon test --packages-select my_cpp_pkg
colcon test-result --verbose
  • colcon test: 빌드 후, 단위 테스트와 함께 lint 검사도 수행한다.
  • colcon test-result --verbose: 테스트 및 lint 결과를 상세하게 확인한다.

만약 오류가 발생하면, 어떤 파일에서 어떤 문제(코드 스타일 위반, 포맷 문제, 잠재적 버그)가 발생했는지 구체적으로 확인할 수 있다.

CI/CD에서의 ROS2 패키지 빌드와 테스트¶

개발 환경에서 로컬로 빌드하고 테스트하는 것 외에, 원격 저장소(GitHub, GitLab 등)에서 자동으로 ROS2 패키지를 빌드하고 테스트하는 CI/CD 파이프라인을 구성할 수도 있다. 이때도 colcon build, colcon test를 사용하며, Docker 기반으로 ROS2 Humble 이미지를 이용하거나, setup-ros 같은 GitHub Actions용 액션을 사용하면 편리하다.

GitHub Actions 예시(간략)¶

name: Build and Test

on: [push, pull_request]

jobs:
  build_and_test:
    runs-on: ubuntu-22.04
    steps:
      - name: Checkout
        uses: actions/checkout@v2

      - name: Setup ROS
        uses: ros-tooling/setup-ros@v0.3
        with:
          distro: humble

      - name: Install dependencies
        run: sudo apt-get update && rosdep update && rosdep install --from-paths src --ignore-src -y

      - name: Build
        run: colcon build --event-handlers console_cohesion+

      - name: Test
        run: colcon test && colcon test-result --verbose
  • ros-tooling/setup-ros ROS2 특정 버전을 자동으로 설치·설정한다.
  • colcon build와 colcon test 로컬 환경에서와 동일한 명령으로 빌드와 테스트를 수행한다.

이렇게 하면, Pull Request가 생성될 때마다 패키지 빌드와 테스트, lint 검사 등을 자동으로 수행해 코드 품질을 유지하고 협업의 효율을 높일 수 있다.

외부 라이브러리 연동 시 고려 사항¶

ROS2 패키지에서 OpenCV, PCL(Point Cloud Library), Eigen 같은 외부 라이브러리를 사용하려면 아래와 같은 단계를 거친다.

의존 라이브러리 설치:

  • 패키지 배포판(apt) 등을 통해 미리 설치하거나, CMake FetchContent/ExternalProject를 통해 소스에서 가져올 수 있다.

CMakeLists.txt에서 find_package:

  • 예: find_package(OpenCV REQUIRED)
  • 예: find_package(Eigen3 REQUIRED)

target_link_libraries: 예:

target_link_libraries(my_node ${OpenCV_LIBS} Eigen3::Eigen)

package.xml에서의 의존 관계: 스템 전역 라이브러리를 사용한다면 ROS2 패키지 의존성에 직접 명시하지 않아도 되지만, rosdep으로 설치 가능한 라이브러리라면 <build_depend>와 <exec_depend>에 기입해두면 협업 시 편리하다.

일부 라이브러리는 ROS2 메시지 변환 기능을 함께 제공하므로, 그 경우 추가적으로 tf2, tf2_eigen, tf2_geometry_msgs 등 ROS2 전용 패키지 의존성을 정의할 수도 있다.

멀티 패키지 관리와 모놀리식 리포지토리¶

프로젝트 규모가 커지면, 여러 개의 ROS2 패키지를 하나의 Git 리포지토리에서 관리하기도 한다(모놀리식). 이 경우, 서로 다른 패키지 간에 공용 라이브러리를 두거나 상호 참조가 발생할 수 있다.

my_robot_ws
├── src
│   ├── my_robot_bringup
│   ├── my_robot_description
│   ├── my_robot_control
│   ├── my_robot_navigation
│   └── ...
├── build
└── install
  • 공용 메시지 인터페이스: my_robot_interfaces라는 별도 패키지를 만들어 메시지·서비스·액션을 집중 관리하고, 다른 패키지들이 이를 의존하도록 할 수 있다.
  • 공용 라이브러리: my_robot_utils와 같은 패키지를 만들어 반복 사용되는 함수를 라이브러리화하고, 로봇 bringup, 제어, 내비게이션 패키지에서 이를 링크한다.

colcon은 src 디렉터리 아래 존재하는 모든 패키지를 자동으로 검색하고, 빌드 의존 관계에 따라 순서대로 빌드한다. 따라서, 여러 패키지를 한꺼번에 빌드하고 테스트하기가 수월하다.

플랫폼별 빌드(ARM, 크로스 컴파일)¶

ROS2 Humble은 x86_64(AMD64) 플랫폼을 주로 지원하지만, 라즈베리파이(Raspberry Pi), NVIDIA Jetson과 같은 ARM 기반 보드에서도 동작한다. 크로스 컴파일을 설정하거나, 직접 해당 장치에 ROS2를 설치한 뒤 똑같이 colcon build를 돌릴 수 있다.

  • 로컬 빌드(ARM 디바이스에서 직접)
  • ARM 기반 장치에 Ubuntu 22.04 또는 ROS2가 지원되는 OS를 설치한 뒤, 동일하게 빌드한다.
  • 크로스 컴파일
  • x86_64 환경에서 ARM용 크로스 컴파일 툴체인(toolchain)을 세팅하고 colcon build 시 툴체인을 지정한다.

개발 편의성을 높이기 위해 Docker 이미지를 활용하여, 특정 플랫폼용 ROS2 빌드 환경을 컨테이너화하는 방법도 쓰인다.

추가 도구: ros2 pkg executables¶

생성한 패키지 안에 어떤 실행 파일(노드)들이 존재하는지 확인하려면, 다음 명령어가 유용하다.

ros2 pkg executables <패키지명>

예:

ros2 pkg executables my_cpp_pkg

이 명령을 통해 해당 패키지에 등록된 실행 파일과 그에 연결된 엔트리포인트를 간단히 나열할 수 있다. Python 패키지라면 setup.py에서 정의한 entry_points 목록이 표시되고, C++ 패키지라면 install 디렉터리에 설치된 실행 파일 기준으로 표시된다.

패키지 생성 시 권장되는 검증 절차¶

패키지 metadata 확인
