RINEX 포맷 개요

RINEX(Receiver Independent Exchange Format)는 서로 다른 제조사의 GNSS 수신기에서 관측한 데이터를 표준화하여 상호 교환할 수 있도록 정의된 ASCII 텍스트 기반 파일 형식이다. GNSS 해석 및 후처리에 필수적인 다양한 관측 정보를 공통된 구조로 관리하기 위해 국제 GNSS 서비스(IGS, International GNSS Service)와 국제천문연맹(IAU, International Astronomical Union) 등이 협의하여 표준안을 제정하였다. RINEX는 관측 데이터, 항법 메시지, 궤도 보정 파라미터 등을 동일한 형식으로 기술하여 GNSS 데이터를 정확하고 일관성 있게 활용할 수 있게 한다.

RINEX 파일 구조

RINEX 파일은 크게 헤더(Header) 부분과 본문(Body) 부분으로 구성된다. 헤더에는 관측 장소, 수신기와 안테나 정보, 시간 시스템 정보 등을 기록하여 파일을 이용하는 사람이 관측 환경과 후처리 시 필요한 기본 정보를 확인할 수 있게 한다. 본문에는 실제로 측정된 데이터(예: 위성별 신호 강도, 측정 시각, 코드 관측량, 위상 관측량 등)가 기록된다.

헤더(Header) 정보

  • 관측소 및 수신기 정보: 관측소 이름, 수신기 기종, 안테나 타입, 안테나 높이 등
  • 관측 타입(Observation Types): 예를 들어 GPS L1, L2, GLONASS L1, L2 등 어떠한 주파수 대역에서 어떤 종류의 관측을 수행했는지
  • 좌표 및 기준계 정보: 관측소 위치가 어떤 지구기준좌표계나 지역기준 좌표계에서 표현되는지
  • 시간 정보: GNSS 시각, 파일 시작 시각, 종료 시각 등
  • 기타: RINEX 파일 버전, 제작 소프트웨어 정보, 파일 형식 등

본문(Body) 정보

  • 에포크(Epoch)별 데이터 블록: 수신기가 특정 시각(에포크)에서 수집한 모든 위성 신호에 대한 관측량을 순차적으로 기록
  • 위성별 관측 정보: 코드 관측값, 위상 관측값, 도플러, 신호 강도(SNR) 등
  • 각 관측값의 품질지표: 예를 들어 LLI(Loss of Lock Indicator)나 신호 추적 상태

RINEX 버전

RINEX는 GNSS 신호체계 변화와 자료 처리 기술의 발전에 발맞춰 지속적으로 개정되어 왔다. 대표적인 버전은 2.x, 3.x가 있다.

  • RINEX 2.x: GPS와 GLONASS 중심으로 정의되었으며, 관측 타입 표기 방식이 상대적으로 단순하다.
  • RINEX 3.x: 2.x의 한계를 극복하고, Galileo, BeiDou, QZSS 등 다양한 위성항법시스템을 통합적으로 처리 가능하도록 개정되었다. 관측 타입 표기 규칙이 체계화되었고, 새로운 데이터 필드가 추가되었다.

GNSS 관측 데이터와 수식 표현

RINEX 본문에는 주로 코드 관측량과 위상(Phase) 관측량이 기록된다. 코드 관측량(가령 \(C_{1}\), \(C_{2}\) 등)은 위성–수신기 간 거리 추정을 위한 기본 측정치이고, 위상 관측량(예: \(L_{1}\), \(L_{2}\) 등)은 파장의 수 및 부분 위상으로부터 고정밀 위치를 추정하는 데 활용된다. 이에 대한 일반화된 측정 방정식을 간단히 쓰면 다음과 같은 형태로 표현할 수 있다. $$ P = \rho + c \cdot dT + I + T + \epsilon $$

  • \(P\): 코드 또는 위상 측정값
  • \(\rho\): 위성–수신기 간 기하학적 거리
  • \(c\): 빛의 속도
  • \(dT\): 위성 시계와 수신기 시계의 시간 오차
  • \(I\): 전리권 편차(ionospheric delay)
  • \(T\): 대류권 편차(tropospheric delay)
  • \(\epsilon\): 측정 잡음 및 모델링 오차

위상 측정값의 경우엔 파장 \(\lambda\)(예: L1 주파수의 파장)과 측정된 위상값(사이클 수 등)을 사용하여 다음과 같은 형태로 나타내기도 한다. $$ \phi = \frac{1}{\lambda} \left( \rho + c \cdot dT + I + T + N \cdot \lambda + \epsilon' \right) $$

  • \(\phi\): 위상관측으로 환산된 사이클(cycle) 수
  • \(N\): 위상 관측값에서 정수로 추정되는 불확정 정수(ambiguity)
  • \(\epsilon'\): 잔여 측정 잡음 및 모델링 오차

RINEX 본문은 위와 같은 여러 물리량을 규칙에 따라 배치하며, GNSS 후처리를 위해 필요한 정확한 자료 구조를 제공한다.

RINEX 관측 파일(Observation File)

일반적으로 확장자가 .yyO(RINEX 2.x) 또는 .rnx(RINEX 3.x)로 표시되며, 다음과 같은 특징이 있다.

  • 다양한 GNSS 관측량(코드, 위상, 도플러, 신호 강도 등)을 시간 순서대로 기록
  • 에포크(Epoch) 단위로 데이터를 정렬
  • 버전에 따라 관측량 표기 규칙이 다르므로, 소프트웨어 사용 시 해당 버전에 유의해야 함

RINEX 항법 파일(Navigation File)

위성 궤도력과 시계정보 등을 제공하는 파일로서, 확장자가 .yyN(GPS), .yyG(GLONASS), .yyP(Galileo) 등으로 구분되기도 한다.

  • 위성력(위성 궤도 파라미터), 위성 시계보정값, 기타 항법 메시지 정보를 수록
  • 후처리 시 위성의 위치를 복원하거나 시계 보정을 수행하는 데 활용

RINEX 기타 파일

  • Meteorological File: 기압, 온도, 습도 등 기상정보
  • Clock File: 위성 시계 및 지상국 시계 보정 정보
  • Sp3 (정밀 궤도 파일): IGS 등에서 제공하는 정밀 궤도 파라미터

RINEX 헤더 예시

헤더는 사람이 읽을 수 있는 ASCII 텍스트로 구성되어 있으며, 예시적으로 다음과 같은 구문을 포함한다(버전에 따라 차이가 있음):

     3.04           OBSERVATION DATA    M (Mixed)           RINEX VERSION / TYPE
Teqc  2019Feb25     University XX       20201219  203322 UTC PGM / RUN BY / DATE
MARKER NAME         EXAMPLE-STATION
MARKER NUMBER       1234
ANT # / TYPE        ABC1234567         NONE
REC # / TYPE / VERS GNSS-Receiver X 1.0
APPROX POSITION XYZ   1234567.890   234567.890   34567.890
ANTENNA: DELTA H/E/N       0.0000        0.0000        0.0000
SYS / # / OBS TYPES G   5 C1C L1C D1C S1C C2W
SYS / # / OBS TYPES R   5 C1C L1C D1C S1C C2P
...
TIME OF FIRST OBS   2020    12    19    20    33   22.0000000     GPS
...
END OF HEADER
  • RINEX VERSION / TYPE: 현재 RINEX 버전과 파일 형식
  • PGM / RUN BY / DATE: 생성 프로그램, 생성자, 생성 날짜
  • MARKER NAME / NUMBER: 관측소 이름과 내부 식별자
  • APPROX POSITION XYZ: 관측소의 대략적인 위치(GRS80/ITRF 계열 XYZ 좌표 등)
  • SYS / # / OBS TYPES: 어떤 GNSS에 대해 몇 가지 관측 타입을 기록하는지 기술
  • TIME OF FIRST OBS: 첫 관측 시각
  • 그 외: 수신기 안테나 정보, 수신기 기종, 파일 종료 표시(END OF HEADER)

RINEX 파일 이름 규칙

RINEX 형식의 파일은 일반적으로 관측소 식별자, 관측일, 세션 정보, 연도, 확장자 등을 조합하여 생성된다. RINEX 2.x의 경우 8+3 형태의 DOS 호환 파일명을 많이 사용하며, RINEX 3.x에서는 좀 더 유연한 작명법을 지원한다. 예컨대 RINEX 2.x 방식에서는 다음과 같은 예시가 있다.

ABCD0010.23O
  • ABCD: 관측소 이름(최대 4자)
  • 0: 세션 구분(0: 24시간 전체를 1일 1파일로 구성할 때 등)
  • 010: 연중 일련번호(Day of Year), 이 예시는 1월 10일을 의미
  • .23: 연도 마지막 두 자리(2023년)
  • O: RINEX 관측 파일(Observation)의 확장자

RINEX 3.x에서는 확장자가 .rnx인 경우도 흔히 볼 수 있고, 파일명에 관측소 이름을 9자까지 사용하는 등 다양한 조합이 가능하다.

RINEX 관측 본문 구조

RINEX Observation File의 본문(Body)에는 에포크별로 획득한 위성 관측 데이터가 기록되어 있다. 크게 에포크 정보를 기술하는 행과, 각 위성에 대한 관측값을 기술하는 행으로 나누어진다.

에포크(Epoch) 라인 구조

에포크 라인에는 다음과 같은 정보가 기록된다(버전에 따라 세부 차이는 존재):

  • 에포크 플래그(Epoch Flag): 해당 에포크의 데이터가 정상 측정인지, 혹은 사이클 슬립 발생 등을 나타내는 간단한 플래그
  • 연도, 월, 일, 시, 분, 초: 관측 시각
  • 해당 에포크에서 추적한 위성 수
  • GNSS 시스템 및 PRN(또는 SVN) 번호: 예) G12, R06, E18, C07 등(GPS, GLONASS, Galileo, BeiDou 등)
  • 그 외 수신 지연이나 스테이션 이벤트를 기록하는 부가 정보(선택적으로 존재)

대략적인 예시는 다음과 같이 표현될 수 있다(각 버전마다 스페이싱, 표기 규칙 차이 있음):

> 2020 12 19 20 33 22.0000000  0  7
G 12 G 25 R 06 E 18 C 07 C 13 G 01

위 예시에서 > 2020 12 19 20 33 22.0000000는 2020년 12월 19일 20시 33분 22초의 에포크라는 의미이고, 이어서 7개의 위성을 추적했다(G12, G25, R06, E18, C07, C13, G01). 이 후 각 위성별 관측치가 순서대로 나온다.

위성별 관측 라인 구조

관측 라인에는 각 위성에 대한 코드, 위상, 도플러, 신호 강도(SNR) 등이 특정 순서로 기록된다. 예를 들어 GPS에 대해 \(L1\) 코드(\(C1\)), \(L1\) 위상(\(L1\)), \(L2\) 코드(\(C2\)), \(L2\) 위상(\(L2\)) 등이 한 줄 또는 여러 줄로 배치될 수 있다(버전 2.x에서는 관측 타입이 2줄에 걸쳐 기록되기도 함). RINEX 3.x에서는 SYS / # / OBS TYPES 헤더에 정의된 관측량의 순서에 맞춰 본문이 구성된다.

예시(단순화된 구조):

    22564783.123  130628696.456          42.0       22564785.234  130628710.987          43.5
  • 앞쪽부터 \(C1\), \(L1\), 신호 강도(\(S1\)), \(C2\), \(L2\), 신호 강도(\(S2\)) 값이 기록
  • 일부 자리수 뒤에 LLI(Loss of Lock Indicator) 또는 신호 품질 지표가 붙기도 함

LLI (Loss of Lock Indicator)

위상 관측에서 위상 연속 추적이 중단되거나 사이클 슬립이 발생할 경우, 이를 표현하기 위한 지표가 LLI이다.

  • 0: 정상 추적
  • 1: 사이클 슬립 또는 추적 이상 발생
  • 버전마다, 수신기 구현마다 세부 해석이 달라질 수 있음

RINEX 항법(Navigation) 파일

항법 파일(Navigation File)에는 각 위성에 대한 궤도력과 시계 보정 파라미터 등이 기록되어 있다. GPS를 예로 들면:

  • 확장자: .yyN(RINEX 2.x), .rnx(RINEX 3.x), 경우에 따라 .nav라는 이름도 사용
  • 하나의 위성에 대한 궤도력(Ephemeris)이 여러 세트 들어 있으며, 각 세트는 특정 발행 시각에 대한 위성의 궤도 파라미터를 의미

항법 본문 예시

GPS 위성항법 메시지(RINEX 2.x 기준)는 위성 식별자, 에포크 시각, 오비탈 파라미터(Keplerian 요소 기반), 위성 시계 오차 등이 여러 줄에 걸쳐 기록된다. 예를 들어 (단순화 예시):

 2 20 12 19 20  0  0.0 -.814372E-04 -.335382E-11  0.000000E+00
     .790000E-08  .198100E+02  .170000E-06  .515367E+00
     ...
     ... (이하 줄 생략)
  • 첫 줄: 위성 식별자(2), 에포크 시각(2020년 12월 19일 20시 0분 0초), 위성 시계 계수 등
  • 이어지는 줄에 궤도력(정확한 파라미터, 편이각, 평균운동, 궤도 기울기, 승교점 적경, 적경 변동률 등)이 순차적으로 배치

RINEX 3.x에서는 G02처럼 시스템 식별자(G)와 PRN(02)을 함께 적어 구별하며, 일부 필드 구성이 달라질 수 있다.

GLONASS, Galileo, BeiDou 항법 파일 차이점

  • GLONASS: 빈도 채널 번호, 위성 건강상태, 궤도 파라미터 구조가 GPS와 다름
  • Galileo: I/NAV 메시지 기반, 특정 파라미터들이 GPS와 유사하지만 일부 지표(예: E1, E5) 구성이 다름
  • BeiDou: B1, B2 등 주파수별로 항법 메시지 구조가 조금 다르고, 궤도 파라미터 또한 GPS와 비슷하지만 세부 항목이 상이

RINEX와 다중위성시스템(Multi-GNSS) 호환성

RINEX 3.x의 큰 장점은 GPS, GLONASS, Galileo, BeiDou, QZSS 등 다양한 위성항법시스템을 하나의 파일에서 함께 다룰 수 있도록 설계되었다는 점이다.

  • SYS / # / OBS TYPES에서 각 시스템(G, R, E, C 등)에 대해 어떤 관측 타입이 기록되는지 명시
  • 에포크 라인에서 위성 시스템별 식별자를 구분하여 표기

RINEX와 스테이션 이벤트(Observing Session Events)

RINEX 파일에서는 측정 중 단말의 작동 상태나 관측 이벤트에 대한 간단한 정보도 표시할 수 있다. 예컨대 에포크 플래그를 통해 다음 상황을 표시한다:

  • 0: 정상 측정

  • 0: 사이클 슬립 강제 처리, 안테나 교체, 수신 중지 등 스테이션 이벤트

  • 에포크 라인 뒤에 별도 주석이나 표시가 들어갈 수도 있음

RINEX 파일의 처리 및 검증

다양한 GNSS 처리 소프트웨어(예: RTKLIB, GIPSY, Bernese, GAMIT 등)에서 RINEX 파일을 입력으로 받아 후처리를 수행한다. 이때 다음 사항을 주의해야 한다.

  • 버전 호환성: RINEX 2.x와 3.x는 문법과 데이터 구조가 다르므로 변환이 필요할 수 있음
  • 누락된 헤더 정보: 관측소 좌표나 안테나 정보가 없으면 처리 시 에러가 날 수 있음
  • 관측 타입 일관성: 후처리 소프트웨어가 요구하는 관측 타입이 파일에 기록되어 있는지 확인 필요
  • 에포크 간격 및 품질: 지연 측정(time lag), 사이클 슬립 등 품질 문제 발생 시 후처리에 영향

RINEX 파일의 변환: Hatanaka 압축(CRX)

GNSS 관측 데이터는 일반적으로 관측 간격이 짧고(예: 1초, 30초), 하루 24시간 데이터를 기록하면 파일 크기가 매우 커진다. 이를 효율적으로 보관·전달하기 위해 Hatanaka(하타나카) 방식의 압축을 적용하기도 하는데, 이를 흔히 “RINEX 압축”이라고 부른다.

  • 압축 원리: RINEX 본문에서 연속 에포크 간 관측값이 크게 변하지 않는다는 점을 활용한다. 에포크 간 차분 값을 이용하고, ASCII 문자로 재인코딩하여 저장한다.
  • 파일 확장자: 압축 전에는 .rnx(또는 .YYo 등), 압축 후에는 .crx 형태를 취하며, 추가적으로 gzip과 같은 2차 압축을 적용하여 .crx.gz로 저장하기도 한다.
  • 복원: 후처리 프로그램이나 전용 변환 도구(예: crx2rnx)로 .crx 파일을 다시 표준 RINEX 파일로 복원할 수 있다.

RINEX 파일 검증 툴

RINEX 파일의 품질이나 헤더 정보를 검증하고, 다양한 포맷 변환을 지원하는 툴이 존재한다. 대표적으로 다음이 널리 활용된다.

  • Teqc
  • UNAVCO에서 제공하던 툴. 관측 품질 체크, 포맷 변환(RINEX 2.x ↔ 3.x), Hatanaka 압축/해제 지원, 헤더 수정 등 다양한 기능을 제공
  • 명령어 기반으로 간단한 스크립트를 작성하여 대량의 RINEX 파일 처리 가능
  • gfzrnx
  • GFZ에서 개발한 고급 RINEX 처리 툴. RINEX 파일의 완전성 검사, 버전 변환, 품질 분석, 헤더 수정 기능 등을 제공
  • RTKLIB
  • 실시간 RTK와 후처리 기능을 모두 제공하는 오픈소스 패키지. RINEX 파일 변환, 소프트웨어 수신기 기능 등 다양한 모듈이 있음

이러한 툴을 통해 RINEX 파일에 누락된 헤더를 보완하거나, 불필요한 관측 타입을 제거하고, 중복 에포크를 정리하는 등 정비 과정을 수행함으로써 후처리 해석의 정확도와 안정성을 확보할 수 있다.

RINEX의 궤도/시계 파일(예: SP3, Clock RINEX 등)

관측 파일(Observation RINEX)과 항법 파일(Navigation RINEX) 외에도, GNSS 해석 시 정밀궤도와 정밀시계 정보를 별도 파일로 제공하는 경우가 많다.

  • SP3(정밀 궤도 파일)
  • IGS(International GNSS Service) 등에서 제공하는 표준 정밀 궤도 포맷
  • ASCII 텍스트 기반이며, 위성별 위치(XYZ)와 시계 정보가 일정 간격(예: 15분)으로 기록
  • 정확도가 수 cm~dm 수준으로, 정밀 위치결정(PPP 등)에 사용
  • Clock RINEX
  • 위성 시계와 지상국 시계에 대한 정밀 보정 값을 RINEX-like 형식으로 기술
  • PPP나 정밀후처리에서 시계 모델을 보완하기 위해 사용

RINEX 파일 간 상호 참조

일반적인 후처리 과정에서 다음과 같은 파일들이 함께 사용된다.

flowchart LR A("Observation RINEX <br>(.rnx, .YYo)") --> C(GNSS Processing <br> Software) B("Navigation RINEX <br>(.rnx, .YYn, .YYp)") --> C D(SP3, Clock Files) --> C C --> E(정밀 해석 결과)
  • Observation RINEX: 수신기에서 실제 측정한 코드, 위상, 도플러 등
  • Navigation RINEX: 위성 궤도력(브로드캐스트)과 시계 파라미터
  • 정밀 궤도/시계 파일: SP3 및 Clock 파일
  • GNSS Processing Software: Bernese, GIPSY, GAMIT, RTKLIB 등 다양한 패키지

이처럼 여러 RINEX 계열 파일을 동시에 활용하여 GNSS 해석이 진행되고, 필요에 따라 정밀 궤도 및 시계 정보를 추가로 적용하여 더 높은 정확도를 얻는다.

본 장에서는 RINEX의 핵심 구조와 종류, 관측 데이터 및 항법 파일의 기록 방식, 다양한 버전과 호환성, 압축 방식, 검증 툴 등을 다루었다. GNSS 후처리를 수행하기 위해서는 RINEX 파일의 헤더와 본문을 제대로 이해하고, 필요 시 적절히 변환·검증하는 과정이 필수적이다.

RINEX 3.x에서의 관측 타입 표기

RINEX 3.x는 다중위성시스템과 다양한 신호 대역을 지원하기 위해 관측 타입 표기를 더욱 세분화하였다. RINEX 헤더의 SYS / # / OBS TYPES 라인을 보면, 각 시스템마다 쓰이는 관측 타입이 여러 줄에 걸쳐 기술될 수 있다.

  • 관측 세부 식별
  • 첫 번째 문자: 측정 방식 예) C(코드), L(위상), D(도플러), S(신호강도) 등
  • 두 번째 문자: 주파수 대역 예) 1, 2, 5, 6, 7, 8 등 (시스템별로 의미가 달라짐)
  • 세 번째 문자: 신호 코드 예) C, P, X, Z 등 (구체적인 코드 식별)
  • 예: C1C (GPS L1 코드 관측 중 C/A 코드), L5X (GPS L5 위상 관측 중 I+Q 합성 신호)

예를 들어, GPS 위성의 경우 \(L1\) 대역에서 C/A 코드를 측정한 값은 C1C, 위상 관측은 L1C와 같이 표기하며, \(L2\) 대역의 P(Y) 코드는 C2P, 위상 관측은 L2P로 구분하는 식이다. Galileo, BeiDou, QZSS 등도 비슷한 규칙을 따르되, 각 위성시스템에서 정의하는 대역(예: E1, E5 등)에 맞추어 표기가 달라진다.

기타 RINEX 계열 포맷

  • Clock RINEX
  • 위성 시계 및 지상국 시계 측정 결과를 RINEX와 유사한 텍스트 형식으로 기록
  • PPP(Precise Point Positioning) 해석이나 시계 분석 연구에 활용
  • IONEX (Ionosphere Map Exchange)
  • 전리권 총전자량(TEC) 지도를 교환하기 위한 표준 포맷
  • GNSS 자료를 기반으로 전리권 모델을 생성하고, 시간·위치별 TEC 값을 2차원 격자로 표현
  • RINEX와 직접적으로 동일한 형식은 아니나, GNSS 해석에서 함께 활용됨

다중 신호(RINEX 3.x)의 활용 예

멀티밴드 GNSS 수신기의 경우 GPS, GLONASS, Galileo, BeiDou, QZSS, SBAS 등 다양한 위성을 동시에 추적하고, 각 시스템별로 복수 주파수 대역에서 코드를 측정할 수 있다. 이때 RINEX 3.x 본문에는 에포크마다 수십 개의 관측값(코드, 위상, 도플러, 신호 강도)이 기록될 수 있다.

이를 수학적으로 표현하면, 예컨대 GPS 기준으로 \(m\)개 대역, \(n\)개 위성을 추적한다면, 하나의 에포크에서 관측할 수 있는 코드 측정값들은 다음과 같이 벡터 형태로 나타낼 수 있다. $$ \mathbf{P}{\text{epoch}} = \begin{bmatrix} P{1,1} \ P_{1,2} \ \vdots \ P_{m,n} \end{bmatrix} $$ 각 \(P_{i,j}\)는 \(i\)번째 대역에서 \(j\)번째 위성에 대해 측정된 코드 관측치를 의미하고, 이들에 상응하는 위상 관측값 \(\mathbf{L}_{\text{epoch}}\)도 유사한 벡터 형태로 나타낼 수 있다.

측정 시각 동기와 RINEX 시간계

RINEX는 GNSS 시각을 표준으로 채택하되, 시스템별 시간 오프셋을 고려해야 한다. 예를 들어:

  • GPS 시각: 1980년 1월 6일부터 시작되며, 윤초가 적용되지 않음
  • Galileo 시각: 1999년 8월 22일부터 시작
  • GLONASS 시각: UTC(러시아) 기반
  • BeiDou 시각: 2006년 1월 1일부터 시작

RINEX 헤더에서 TIME OF FIRST OBS를 표기할 때, 보통은 GPS 시각을 쓰되, 시스템별 표기가 혼재할 수 있음을 유의해야 한다.

전처리 단계에서의 RINEX 활용

GNSS 후처리 소프트웨어로 자료를 투입하기 전, RINEX 파일을 여러 단계의 전처리에 부치는 경우가 많다. 예를 들어:

  1. 파일 병합: 하루 24시간 데이터를 1시간 단위로 저장했으면 이를 하나로 합치기
  2. 불량 데이터 제거: 측정 스푸리어(spurious) 값을 제거하거나 사이클 슬립이 과도한 에포크를 제거
  3. 간격 재샘플링: 1초 간격으로 관측된 대용량 파일을 30초 간격으로 재샘플링
  4. 헤더 보완: 관측소 정보, 안테나 정보, 좌표계 정의 등이 누락된 경우 추가 기입

이러한 전처리 과정을 통해 RINEX 파일 품질을 개선하면 후속 해석 단계에서 계산 안정성과 정확도가 향상된다.
