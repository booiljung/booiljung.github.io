소스 빌드를 통한 설치(Windows, Linux)

Windows 환경에서 PCL 소스 빌드

Windows에서 PCL을 소스 빌드하기 위해서는 CMake와 Visual Studio, 그리고 PCL 빌드를 위한 필수 라이브러리가 준비되어 있어야 한다. Visual Studio는 Community 버전 이상을 사용하면 되며, CMake 역시 최소 요구 버전에 맞춰 설치해야 한다. 또한 PCL에서 의존하는 다양한 라이브러리(Boost, Eigen, FLANN, VTK 등)도 함께 구성되어 있어야 안정적인 빌드가 가능하다. 일반적으로는 vcpkg나 Conan, 혹은 수동 방식으로 필요한 라이브러리를 설치할 수 있지만, 책의 예시에서는 CMake를 직접 구성하는 과정을 살펴본다.

CMake를 통해 소스 빌드를 진행하려면 PCL의 Git 저장소를 로컬 환경에 클론한다. 클론된 디렉터리에서 CMake GUI 혹은 명령어를 사용하여 프로젝트를 구성할 수 있다. Visual Studio 기반으로 생성을 진행하려면 CMake의 ‘Generator’를 Visual Studio에 맞게 지정한다. 예를 들어 Visual Studio 2022를 사용한다면 아래와 같이 명령어를 작성할 수 있다.

cmake -B build -G "Visual Studio 17 2022" -A x64 .

이때 소스 디렉터리를 현재 디렉터리로 가정하고, 생성되는 빌드 디렉터리를 build로 지정한다. CMake 명령 실행 시 여러 옵션을 직접 지정할 수도 있는데, 예를 들어 Boost나 Eigen 등의 경로가 자동으로 검색되지 않는 경우에는 -DBOOST_ROOT=경로 또는 -DEIGEN_INCLUDE_DIR=경로 같은 식으로 전달한다. 모든 라이브러리가 잘 구성되어 있다면, CMake가 성공적으로 프로젝트를 생성하고 Visual Studio 솔루션 파일을 만들어 낸다.

만약 Visual Studio에서 오프라인 설치된 라이브러리를 찾지 못하거나 버전이 상이한 경우, 빌드가 진행되는 도중에 심각한 오류가 발생할 수 있다. 이때는 CMake 구성 단계에서 경로나 버전을 다시 확인하고, 라이브러리가 설치된 위치를 정확히 지정해야 한다. 빌드 도중에 ‘unresolved external symbol’과 같은 링크 오류가 발생한다면, 환경 변수나 CMake 설정에서 라이브러리가 정상적으로 링크되지 않은 것일 가능성이 높다.

Visual Studio 솔루션 파일이 생성되면, Visual Studio IDE를 열고 해당 솔루션을 로드한다. 구성(Configuration)은 일반적으로 Release 모드로 진행하는 것을 권장하지만, 필요하다면 Debug 모드로도 빌드해볼 수 있다. IDE에서 빌드 시도 시에는 많은 양의 C++ 코드가 컴파일되므로 꽤 오랜 시간이 걸릴 수 있다. 빌드가 완료된 후에는 PCL 라이브러리와 실행 파일들이 지정한 빌드 출력 경로에 생성된다. PCL을 활용한 응용 프로그램을 제작하려면, 빌드된 PCL 라이브러리에 대한 경로를 프로젝트 설정에 추가해 주거나, CMake를 통해 별도의 find_package 설정을 진행하면 된다.

Windows 환경에서 PCL의 소스 빌드를 진행할 때, OpenMP나 CUDA 등을 활용해 병렬화 혹은 GPU 가속을 사용할 수도 있다. GPU 기능이 필요하다면, PCL의 CMake 옵션에서 CUDA 연동을 활성화해야 하며, GPU 연산을 테스트할 수 있는 하드웨어와 드라이버가 적절히 준비되어 있어야 한다. 이러한 기능은 포인트 클라우드 데이터 처리 시 계산량이 많은 경우에 상당한 성능 향상을 제공할 수 있다.

Linux 환경에서 PCL 소스 빌드

Linux 환경에서는 일반적으로 패키지 관리자를 통해 필요한 의존 라이브러리를 먼저 설치한다. 예를 들어 Ubuntu 계열 배포판에서는 apt 명령을 사용하여 Boost, Eigen, VTK, FLANN 등을 설치하고, 추가로 CMake와 Git 역시 준비한다. 패키지 관리자로 설치하기 어려운 라이브러리가 있다면 직접 소스 빌드를 하거나 별도의 패키지 저장소(PPA 등)를 사용할 수도 있다. 모든 라이브러리가 준비되었다면, Git을 통해 PCL 저장소를 클론한다.

소스 폴더 안에서 CMake를 통해 빌드 디렉터리를 생성하고, Makefile 혹은 Ninja 빌드 시스템을 이용해 컴파일한다. Ninja를 쓰려면 CMake의 -G Ninja 옵션을 주거나, Makefile을 쓰려면 별도의 설정 없이도 될 수 있다. 예를 들어 Makefile 기반으로 빌드한다면 아래와 같이 명령어를 입력한다.

cd pcl
mkdir build
cd build
cmake ..
make -j8

위 예시에서 -j8 옵션은 8개 스레드를 사용하여 병렬 빌드를 진행한다는 의미다. CPU 코어가 충분히 많다면 더 큰 숫자를 사용하는 것이 빌드 시간을 단축하는 데 유리하다. 만약 특정 라이브러리의 버전이 호환되지 않는다면, CMake 구성 단계에서 해당 라이브러리 경로를 직접 지정하여 문제를 해결해야 한다. Linux 환경에서도 Windows와 마찬가지로, 컴파일 도중에 경로 문제나 링크 오류가 발생하면 대부분 CMake 설정 혹은 의존 라이브러리 배포본에 원인이 있다.

빌드가 정상적으로 끝나면, make install 명령 등을 통해 PCL 라이브러리를 시스템 디렉터리 혹은 사용자가 지정한 경로에 설치할 수 있다. 시스템 전역에 설치하지 않고 로컬 유저 경로에 설치하기 위해서는 CMake 실행 시 -DCMAKE_INSTALL_PREFIX=/설치경로 옵션을 줄 수 있다. 설치가 완료되면, PCL을 사용하는 프로젝트에서 find_package(PCL 1.12 REQUIRED) 같은 식으로 필요한 버전에 맞춰 PCL을 찾아오도록 설정할 수 있다.

Linux 환경에서 GPU 가속을 사용하려면 CUDA 패키지와 드라이버를 설치한 뒤, CMake 옵션을 통하여 CUDA 지원을 활성화할 수 있다. 이는 대규모 포인트 클라우드 데이터 처리 시 상당한 연산 성능 상승을 기대할 수 있으며, OpenMP 등을 함께 사용하는 것도 좋은 방법이 될 수 있다.

추가 설정 옵션 및 트러블슈팅(Windows)

Windows에서 PCL을 소스 빌드할 때 CMake 옵션을 세분화해 보면, PCL에서 제공하는 여러 모듈(예: features, filters, io, kdtree, registration 등)을 개별적으로 활성화하거나 비활성화할 수 있다. 예를 들어, 특정 모듈에서만 GPU 가속을 사용하고 싶다면 CMake 단계에서 관련 옵션을 조정해야 한다. 일반적으로 CMake GUI에서 검색어를 ‘PCL_’로 입력하면 모듈별 설정 항목이 나타난다. 이 중 필요 없는 기능은 OFF로 설정해 빌드 시간을 단축할 수도 있다.

Windows에서는 수동으로 설치한 3rd-party 라이브러리를 CMake가 자동 인식하지 못하는 경우가 잦다. 예를 들어, Boost가 C:\local\boost_1_81_0 같은 폴더에 설치되어 있다면, CMake 구성 시 아래처럼 명시적으로 옵션을 줘야 한다.

cmake -B build ^
      -G "Visual Studio 17 2022" ^
      -A x64 ^
      -DBOOST_ROOT="C:/local/boost_1_81_0" ^
      -DBoost_NO_SYSTEM_PATHS=ON ^
      .

CMake로 구성해도 빌드 단계에서 링크 오류가 발생한다면, Visual Studio의 프로젝트 속성 또는 CMake의 고급 설정에서 링크러리 경로가 빠져 있지 않은지 확인한다. 링크러리 이름에 _s나 _mt 등의 suffix가 붙은 경우, PCL에서 찾는 라이브러리 이름과 정확히 일치하지 않아 문제를 일으키는 경우도 있다. 이럴 때는 관련 라이브러리가 서로 대응되는지, 혹은 다른 CRT(C Runtime Library) 설정으로 빌드된 것은 아닌지 살펴볼 필요가 있다.

OpenMP를 활성화하려면 Visual Studio 쪽에서 CMake가 -DUSE_OPENMP=ON을 전달하도록 설정하거나, GUI 환경이라면 PCL 내부 설정 중 WITH_OPENMP 항목을 체크한다. 빌드가 정상적으로 끝난 뒤, OpenMP가 동작하는지 확인하기 위해서는 Visual Studio에서 예제 코드를 만들어 스레드가 병렬로 실행되는지 테스트해 볼 수 있다. CUDA를 이용한 GPU 기능을 켜려면 WITH_CUDA 혹은 PCL_ENABLE_GPU와 같은 옵션이 필요할 수 있다. CUDA Toolkit 버전도 PCL과 호환되는지 미리 확인하는 것이 좋다.

WSL(Windows Subsystem for Linux) 환경에서 Linux 빌드 방식을 그대로 따라가는 것도 하나의 방법이다. WSL에 Ubuntu 또는 다른 배포판을 설치한 뒤, Linux용 빌드 지침을 그대로 진행하면, Windows와 Linux 간의 경로 설정 문제를 상당 부분 피할 수 있다. 다만 GUI를 사용하는 단계(예: CMake GUI) 대신 CLI 방식으로만 작업해야 하며, 하드웨어 가속이나 USB 장치 액세스를 포함한 몇몇 부분에서 제한이 발생할 수 있다.

추가 설정 옵션 및 트러블슈팅(Linux)

Linux 환경에서는 PCL 소스 빌드 시 기본적으로 Makefile이나 Ninja를 사용한다. 특정 모듈만 빌드하고 싶다면 CMake에서 PCL 모듈별 활성화 옵션을 켜거나 끌 수 있다. 예를 들어, -DBUILD_apps=OFF 같은 식으로 응용 프로그램(apps) 디렉터리에 포함된 예제 프로그램 빌드를 생략할 수 있다. 여러 모듈이나 툴 중 실제 프로젝트에서 사용하지 않는 것이 있다면, 빌드 시간을 크게 줄일 수 있다.

패키지 매니저로 설치한 라이브러리가 여러 버전 동시에 깔려 있으면, CMake가 구버전을 인식해 빌드에 실패할 수 있다. 이런 경우 -DCMAKE_PREFIX_PATH 옵션을 사용하여 새로 설치한 라이브러리의 경로를 우선 탐색하도록 설정할 수 있다. 예컨대, /home/user/local 아래에 최신 버전의 VTK가 설치되어 있다면 아래와 같은 식으로 명령을 입력한다.

cmake -DCMAKE_PREFIX_PATH=/home/user/local ..

이후 make -j8 명령으로 병렬 빌드를 수행할 때 CPU 사용량이 너무 높아 다른 작업이 느려질 수 있다면, 적절히 스레드 개수를 조절한다. 빌드가 완료된 뒤에는 $make install$을 통해 시스템 경로(기본적으로 /usr/local)에 PCL을 설치하거나, -DCMAKE_INSTALL_PREFIX로 특정 디렉터리를 지정해 로컬 설치로 운용할 수도 있다.

동적 링크(Shared Library)와 정적 링크(Static Library)를 어떻게 구성할지도 중요한 고려 사항이다. 정적 라이브러리로 PCL을 빌드하면 결과물이 커지지만, 응용 프로그램에서 추가로 동적 라이브러리를 찾을 필요가 없다. 반면에 동적 라이브러리는 실행 시에 LD_LIBRARY_PATH 등의 환경 변수를 통해 PCL 소스를 로드해야 한다. 프로젝트 규모와 배포 방식을 고려해 선택하는 것이 좋다.

OpenMP, CUDA 등의 병렬화 옵션은 Linux에서도 유사하게 적용된다. 다만 NVIDIA GPU를 활용하기 위해서는 호환 가능한 NVIDIA 드라이버와 CUDA Toolkit 버전이 설치되어 있어야 하며, CMake 설정에서 -DWITH_CUDA=ON과 같은 옵션을 통해 GPU 모듈이 빌드 대상에 포함되도록 해야 한다.

SSE나 AVX 같은 CPU 벡터화 기술을 최적화하려면, CMake 단계에서 -DCMAKE_CXX_FLAGS="-march=native" 같은 식의 플래그를 주어 CPU의 아키텍처별 최적화를 수행할 수 있다. 이는 포인트 클라우드 연산에서 회전, 변환, 거리 계산과 같은 연산 빈도가 높을 때 성능을 향상시킬 수 있다.

추가적인 CMake 설정 및 환경 변수 고려

소스 빌드를 진행할 때, Windows와 Linux 모두에서 CMake가 사용하는 환경 변수가 의존 라이브러리의 경로, 빌드 옵션, 컴파일러 설정 등에 관여할 수 있다. 예컨대 $CMAKE_PREFIX_PATH 환경 변수가 /opt/custom 같은 디렉터리를 포함한다면, CMake는 시스템 경로 대신 그 디렉터리 안에 있는 라이브러리를 먼저 검색한다. $CMAKE_LIBRARY_PATH나 $CMAKE_INCLUDE_PATH를 지정해 특정 라이브러리나 헤더 경로를 우선순위로 잡을 수도 있다.

PCL을 비롯해 여러 대규모 C++ 프로젝트에서는 의존성이 많은 경우가 흔하므로, CMake 설정 시 여러 변수를 확인해야 한다. Windows에선 주로 시스템 변수나 사용자 변수를 통해 설정하고, Linux에선 쉘 스크립트나 .bashrc, .zshrc 등을 통해 세션마다 자동으로 세팅하도록 구성할 수 있다.

실제 빌드 시 환경을 보다 체계적으로 관리하기 위해, Python 기반의 Conan 또는 vcpkg와 같은 패키지 관리자와 CMake를 혼합해서 사용할 수도 있다. 이 경우 PCL을 Conan 레시피 혹은 vcpkg 포트로 설치할 수 있고, 각종 라이브러리를 자동으로 호환 버전으로 맞춰주므로 수동 세팅보다 편리하다. 다만 책에서는 ‘소스 빌드 + 수동 구성’ 과정을 중심으로 다루기 때문에, 패키지 관리자 사용은 별도의 선택 사항으로 제시한다.

멀티 플랫폼 및 크로스 컴파일

Windows와 Linux뿐 아니라, ARM 기반의 임베디드 리눅스 환경 등 다른 플랫폼에서도 PCL 빌드를 시도할 수 있다. 이때는 크로스 컴파일 도구체인을 준비하고, CMake에 -DCMAKE_TOOLCHAIN_FILE=파일 옵션을 넘겨서 크로스 컴파일러와 관련 라이브러리 경로를 알려줘야 한다. 예를 들어 라즈베리 파이나 NVIDIA Jetson과 같은 ARM 기반 기기용으로 PCL을 빌드하려면, 호스트 머신(예: x86_64)에서 크로스 툴체인을 설정해 빌드를 진행하는 방법이 있다.

크로스 컴파일 시에는 의존 라이브러리도 모두 타겟 아키텍처용으로 빌드되거나 준비되어 있어야 한다. PCL은 자체적으로 매우 많은 기능을 제공하기 때문에, 모든 의존 라이브러리를 매번 크로스 컴파일해야 하는 상황이면 시간이 많이 걸릴 수 있다. 실제 제품 환경에서는 핵심 기능만 추려서 빌드하는 것이 일반적이다. 예컨대, GPU가속 모듈을 사용하지 않고 CPU 기반의 필터, 세그멘테이션, 피처 추출 모듈만 빌드하면 의존 라이브러리 수가 줄어든다.

32비트와 64비트 빌드 고려

Windows의 경우 Visual Studio 생성기에서 -A x64 옵션을 쓰지 않으면 32비트 기본 구성이 될 수도 있다. 32비트 환경에서는 더 적은 메모리를 사용하지만, 대규모 포인트 클라우드를 다루거나 많은 양의 데이터를 불러오는 작업에서는 한계가 빨리 찾아올 수 있다. 따라서 대부분 64비트 빌드를 권장한다. Linux 환경에서는 배포판이 기본적으로 64비트인 경우가 많아, 별도의 설정 없이도 64비트로 빌드된다.

32비트 빌드를 꼭 해야 하는 특수 상황(예: 오래된 운영체제, 혹은 메모리가 적은 임베디드 환경)에서는 CMake에서 -DCMAKE_CXX_FLAGS="-m32" 같은 식으로 컴파일러 플래그를 제공해야 한다. 나아가 32비트 라이브러리 버전도 따로 설치해야 하므로, 매우 번거로울 수 있다. PCL처럼 대규모 라이브러리의 32비트 빌드는 별도의 이유가 없다면 피하는 것이 일반적이다.

빌드 최적화와 런타임 최적화

PCL을 빌드할 때, 일반적인 Release 모드 외에도 런타임에 적용할 최적화 전략을 고려할 수 있다. 예컨대 GCC나 Clang에서 -O3, -march=native, -mtune=native 등을 지정하면, 사용하는 CPU의 아키텍처 특성을 적극 활용해 빌드된다. 이렇게 하면 AVX2, SSE4.2, AVX-512 등 CPU 벡터 명령을 사용할 수 있어, 포인트 클라우드 연산 속도가 크게 빨라진다.

한편, -march=native 플래그는 빌드 환경과 실행 환경의 CPU가 다르다면 호환성 문제가 생길 수 있다. 배포용 라이브러리를 만들 때는 -msse4.2처럼 특정 아키텍처 수준만 명시하거나, 아예 CPU 벡터 명령을 비활성화한 채로 빌드하기도 한다. 프로젝트 특성상 특정 서버나 특정 장비에서만 동작한다면, 최대한 CPU 최적화를 활용해 성능을 높이는 것이 유리하다.

디버깅 빌드와 릴리스 빌드

소스에서 디버깅 빌드를 수행하면 컴파일러가 디버그 심볼과 검사 코드를 삽입하기 때문에 라이브러리 용량이 커지고 실행 속도도 느려질 수 있다. 하지만 개발 단계에서 PCL 내부 함수를 추적하거나 문제점을 찾을 때는 반드시 디버그 빌드가 필요하다. Windows에서는 Visual Studio IDE에서 Debug/Release를 쉽게 전환할 수 있고, Linux에서는 -DCMAKE_BUILD_TYPE=Debug로 지정 후 make -j8을 돌려서 빌드하면 된다.

디버그 빌드가 완료된 후에는 디버거(gdb, lldb, Visual Studio 디버거 등)로 PCL 함수 내부를 살펴볼 수 있다. 포인트 클라우드 처리 알고리즘의 세부 동작을 이해하거나, 특정 부분에서 강제 중단점(breakpoint)을 걸어 검증할 때 유용하다. 하지만 디버그 빌드를 사용하는 동안에는 성능 프로파일링 결과도 왜곡될 수 있으므로, 최종 성능 측정이나 벤치마크는 Release 빌드를 쓰는 것이 좋다.

모듈 선택 빌드와 최소 의존 구성

PCL은 내부적으로 매우 많은 모듈을 포함하고 있으며, 이 모듈들이 다양한 외부 라이브러리(Boost, Eigen, FLANN, VTK, Qhull, OpenNI, OpenCV 등)와 연결된다. 그러나 모든 모듈을 한꺼번에 빌드하면 빌드 시간도 길어지고, 쓰지 않는 기능까지 포함되어 라이브러리 크기가 커진다. 따라서 실제로 필요하지 않은 모듈을 비활성화해 최소 구성만 빌드할 수 있다. 예컨대 3D 시각화를 사용하지 않는다면 VTK 모듈을 끄고, 시각화 관련 기능을 빼고 빌드할 수도 있다.

CMake에서 PCL 모듈을 개별적으로 제어하려면, BUILD_apps, BUILD_examples, WITH_QT 같은 옵션들을 OFF로 두거나, 특정한 모듈만 켜서 빌드해볼 수 있다. 예를 들어 VTK가 필요 없는 환경에서는 -DWITH_VTK=OFF를 추가해 CMake를 실행하면, 시각화 모듈이 자동으로 비활성화된다. 이때 일부 다른 모듈이 해당 모듈에 종속적인지 확인해야 한다. 시각화 모듈이 꺼지면, 이후 시각화 예제나 GUI 툴 관련 프로젝트는 빌드 대상에서 제외된다.

OpenNI, Qhull, Qt, CUDA 등의 라이브러리는 PCL 내부에서 선택적으로 사용되는 경우가 많다. 필요한 기능만 활성화해 빌드하면, 의존성이 줄어들고 빌드 과정도 단순해진다. 예컨대 OpenNI를 사용하지 않는다면, -DWITH_OPENNI=OFF 옵션을 주어 관련 드라이버와 레이어를 빼는 식이다. Qhull은 3D Convex Hull 계산이나 Advanced Geometry 기능을 지원하는 라이브러리인데, 이러한 기능이 전혀 필요치 않다면 Qhull을 제외하고 빌드할 수도 있다.

QVTK와 Qt 통합 빌드

PCL의 시각화(visualization) 기능은 VTK를 기반으로 동작하며, Qt와 연동해 GUI 어플리케이션 형태로 사용할 수 있다. 이때 QVTKWidget을 사용해, Qt 위젯 형태로 PCL 시각화 창을 배치하고, 마우스 이벤트나 키보드 이벤트를 처리할 수 있게 된다. 이를 위해선 Qt(주로 Qt5 이상)와 VTK가 빌드 시점에 제대로 연결되어 있어야 하며, CMake에서 -DWITH_VTK=ON, -DWITH_QT=ON 등을 설정해야 한다. 일부 시스템에서는 Qt와 VTK 버전 충돌이나 빌드 옵션 불일치로 문제가 발생하기 쉽다.

Windows에서 Qt를 별도로 설치했다면, Qt가 설치된 경로를 CMake에 알맞게 지정해야 한다. 예를 들어 Qt 5.15가 C:\Qt\5.15.2 같은 경로에 설치되어 있다면, 아래처럼 명시해줄 수 있다.

cmake -B build -G "Visual Studio 17 2022" -A x64 .
  -DCMAKE_PREFIX_PATH="C:/Qt/5.15.2/msvc2019_64"

Qt를 MinGW로 설치했다면, Visual Studio 툴체인과 호환이 되지 않아 빌드가 꼬일 수 있다. 이때는 빌드 툴체인을 Qt가 제공하는 MinGW로 통일하거나, Visual Studio 용 Qt를 사용해야 한다. Linux에서는 Qt 패키지(예: qtbase5-dev, qttools5-dev-tools 등)를 설치하면 CMake가 자동으로 찾는 경우가 많지만, 버전을 주의 깊게 확인하고 VTK와의 호환성도 확인해야 한다.

문서화와 예제 빌드

PCL은 풍부한 문서와 예제를 제공한다. 소스 빌드 시 Doxygen을 활용해 API 레퍼런스 문서를 생성할 수도 있는데, 이를 위해서는 CMake 옵션에서 BUILD_doc 같은 항목을 활성화해야 한다. Doxygen, Graphviz 등이 시스템에 설치되어 있어야 문서화 과정이 정상적으로 진행된다. 문서화 빌드는 소스 디렉터리 내 doc 폴더나 build/doc 폴더 아래에 HTML, LaTeX 문서가 생성되는 식이다.

PCL 예제 코드는 examples나 apps 디렉터리에 모아져 있는데, 이를 빌드해서 확인해보면 포인트 클라우드 로딩, 필터링, 정렬, 시각화, 알고리즘 구현 등에 관한 구체적인 예시를 살펴볼 수 있다. 예제를 빌드하려면 BUILD_examples나 BUILD_apps 같은 옵션을 ON으로 두어야 하며, 시각화 예제를 실행하려면 시각화 모듈 및 VTK, Qt 등이 필요하다. 예제 빌드를 통해 PCL 라이브러리가 제대로 동작하는지 시험해 보는 절차로 삼을 수 있다.

테스트 빌드

PCL 소스 레포지토리에는 자동화 테스트 코드가 포함되어 있다. CMake에서 BUILD_tests=ON으로 설정하고 빌드한 뒤, $ctest$ 명령이나 Visual Studio의 테스트 실행 기능을 사용해 유닛 테스트를 수행할 수 있다. 테스트 과정에서 여러 의존 라이브러리가 필요한 경우가 있으니, 미리 준비가 되어 있어야 한다. 테스트가 모두 통과하면 PCL이 정상적으로 동작할 가능성이 높다고 볼 수 있으나, 빌드 환경이 서로 다르거나 OS 버전에 따라 예기치 못한 문제가 발생할 수도 있다.

테스트 실행 시 시간이 꽤 소요될 수 있으므로, 간단히 필요한 기능만 확인하려면 임의로 특정 테스트만 골라 실행할 수도 있다. 예를 들어 $ctest -R filters를 입력해 filters 모듈 관련 테스트만 수행한다. 특정 모듈이 이상 동작하는지 빠르게 진단할 때 유용하다.

컨테이너(Docker) 기반 빌드

Linux 환경에서 의존 라이브러리 버전 충돌이나 배포판 간 차이 때문에 빌드 오류가 발생하는 경우, Docker를 이용해 빌드 환경을 고정시킬 수 있다. Docker 이미지를 직접 만들어 CMake, 컴파일러, 필요한 라이브러리 등을 일정 버전으로 미리 설치해 두고, 그 내부에서 PCL을 빌드하면 호환성 문제를 줄일 수 있다. 예를 들어 Ubuntu 20.04 기반 이미지를 사용해 PCL을 빌드한 뒤, 다른 호스트 시스템(예: Ubuntu 22.04나 CentOS 등)에서도 동일 이미지로 동일 결과물을 얻게 된다.

Dockerfile을 작성해 필요한 패키지를 설치한 뒤, Git으로 PCL 소스를 클론하고 CMake로 빌드하는 과정을 정의할 수 있다. 아래는 매우 단순화된 예시 Dockerfile이다.

FROM ubuntu:20.04

ENV DEBIAN_FRONTEND=noninteractive

RUN apt-get update && \
    apt-get install -y build-essential cmake git libboost-all-dev \
                       libeigen3-dev libflann-dev libvtk7-dev \
                       libqhull-dev libusb-1.0-0-dev libopenni2-dev

WORKDIR /root

RUN git clone --branch pcl-1.12.1 https://github.com/PointCloudLibrary/pcl.git

WORKDIR /root/pcl
RUN mkdir build && cd build && cmake .. && make -j8

이 Dockerfile로 이미지를 빌드하면, Docker 컨테이너 내부에서 PCL 소스 빌드가 수행된다. 완성된 이미지를 사용하면 의존 라이브러리 버전에 대한 고민 없이 바로 PCL 라이브러리가 포함된 환경을 실행할 수 있다. 필요하다면 CUDA를 쓰는 Docker 베이스 이미지(nvidia/cuda) 등을 활용해 GPU 지원도 설정할 수 있다. 이 방식을 이용하면 CI(Continuous Integration) 환경에서 자동으로 PCL 빌드를 돌려, 특정 커밋마다 성공 여부를 검증하거나 테스트 스위트를 실행해 볼 수도 있다.

간단한 빌드 과정 다이어그램

여러 플랫폼에서 PCL을 소스 빌드하는 과정을 단순화하면 다음과 같은 흐름으로 볼 수 있다.

flowchart TB A[소스 클론] --> B[의존 라이브러리 설치] B --> C[CMake 구성] C --> D[컴파일 & 링크] D --> E[테스트 & 예제 빌드] E --> F[설치 혹은 배포]

소스 클론 단계에서 특정 버전 태그(pcl-1.12.1 등)로 체크아웃하면, 빌드 안정성이 높아진다. 의존 라이브러리 설치 단계는 Windows, Linux, Docker 등 플랫폼마다 차이가 있지만, CMake가 이 라이브러리를 성공적으로 발견해야 다음 단계가 진행된다. CMake 구성 후에는 Make나 Ninja, Visual Studio 솔루션 등 빌드 시스템에 맞춰 컴파일과 링크가 이루어진다. 빌드가 완료되면 테스트와 예제 빌드를 통해 라이브러리가 정상 동작하는지 확인하고, 문제가 없다면 설치 혹은 배포 과정을 거친다.
