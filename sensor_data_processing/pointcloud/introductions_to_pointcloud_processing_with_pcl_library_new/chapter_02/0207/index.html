간단한 예제 프로젝트 생성 및 컴파일 확인

프로젝트 디렉터리 구성

PCL을 활용하기 위한 예제 프로젝트를 간단히 구성하려면 먼저 작업을 원하는 경로에 새 폴더를 만들고 그 안에 C++ 소스 파일과 CMake 설정 파일을 준비한다. 예를 들어 mkdir pcl_simple_example 명령으로 폴더를 생성한 뒤 이동하고, 다음과 같이 구조를 마련한다. 소스 코드는 src 디렉터리에 모아 두고, 빌드 디렉터리는 별도로 분리하여 CMake를 통해 생성된 파일들이 소스 트리와 섞이지 않도록 관리한다. 이는 유지보수와 여러 빌드 환경(디버그, 릴리스 등)에서의 설정을 쉽게 분리하기 위해 흔히 권장되는 방식이다.

예제 프로젝트를 위한 최소 구성은 다음과 같다.

pcl_simple_example
├─ CMakeLists.txt
└─ src
    └─ main.cpp

이때 build 디렉터리는 CMake를 실행할 때 자동으로 생성하거나, 미리 mkdir build 명령으로 만들어 둘 수 있다.

CMakeLists.txt 작성

CMake를 이용하여 프로젝트를 관리하려면 최상위 디렉터리에 CMakeLists.txt 파일을 준비해야 한다. 간단한 예제에서는 프로젝트 이름, 최소 요구 CMake 버전, PCL을 찾기 위한 find 패키지 설정, 그리고 링크 설정 정도만 작성하면 된다. PCL은 CMake에 의해 자동으로 설정 파일이 탐색되도록 구성되어 있으므로, FIND_PACKAGE(PCL 1.2 REQUIRED) 구문만 올바르게 들어가면 의존 라이브러리를 쉽게 불러올 수 있다.

아래 예시는 PCL 라이브러리를 사용하기 위한 매우 기본적인 CMakeLists.txt 예시다.

cmake_minimum_required(VERSION 3.10)
project(pcl_simple_example)

find_package(PCL 1.2 REQUIRED)

include_directories(${PCL_INCLUDE_DIRS})
link_directories(${PCL_LIBRARY_DIRS})
add_definitions(${PCL_DEFINITIONS})

add_executable(pcl_simple_example src/main.cpp)
target_link_libraries(pcl_simple_example ${PCL_LIBRARIES})

find_package(PCL 1.2 REQUIRED)에서 버전 숫자는 본인이 설치한 PCL 버전에 맞추어 설정해야 한다. 예컨대 PCL 1.10.1을 설치했다면 find_package(PCL 1.10 REQUIRED) 와 같이 적을 수 있으며, 명시적으로 특정 소수점까지 지정해야 하는 경우가 아니라면 대략적인 버전만 적어주어도 충분하다. 또한 PCL의 추가 모듈이 필요하다면 PCL_COMPONENTS 옵션을 사용하여 의존 관계를 세부적으로 설정할 수 있다. 단순 예제라면 모든 모듈을 링크해도 크게 문제될 부분은 없다.

샘플 코드 구현

소스 파일은 src/main.cpp와 같이 구성한다. 기본적인 예제로는 단순히 포인트클라우드 객체를 생성해 몇 개의 점을 삽입한 후, 콘솔에 점의 개수나 좌표를 확인해 보는 식으로 작성할 수 있다. 아래 예시는 PCL의 PointXYZ 포맷을 사용하는 방법을 보여준다.

#include <iostream>
#include <pcl/point_types.h>
#include <pcl/point_cloud.h>

int main(int argc, char** argv)
{
    pcl::PointCloud<pcl::PointXYZ>::Ptr cloud(new pcl::PointCloud<pcl::PointXYZ>);
    cloud->width = 5;
    cloud->height = 1;
    cloud->points.resize(cloud->width * cloud->height);

    for (std::size_t i = 0; i < cloud->points.size(); ++i)
    {
        cloud->points[i].x = static_cast<float>(i);
        cloud->points[i].y = static_cast<float>(i * 0.1);
        cloud->points[i].z = static_cast<float>(i * 0.01);
    }

    std::cout << "PointCloud size: " << cloud->points.size() << std::endl;
    for (std::size_t i = 0; i < cloud->points.size(); ++i)
    {
        std::cout << "Point " << i
                  << ": (" << cloud->points[i].x
                  << ", "   << cloud->points[i].y
                  << ", "   << cloud->points[i].z << ")\n";
    }
    return 0;
}

여기서 PCL의 pcl::PointCloud<pcl::PointXYZ> 자료형을 포인터 형태로 선언한 뒤, 폭(width)과 높이(height)를 설정하면 width \times height 크기의 포인트 배열이 할당된다. 만약 point cloud 데이터에 컬러 정보나 법선 벡터가 포함되어야 한다면, PointXYZRGBA나 PointNormal 등 적절한 자료형을 택하면 된다. 이 예제에서는 가장 기본 타입인 pcl::PointXYZ만 사용했다.

컴파일 및 링크 확인

프로젝트의 최상위 디렉터리에 있는 CMakeLists.txt 파일을 기반으로 실제 빌드를 진행하려면, 빌드 전용 디렉터리로 이동하여 cmake ..와 같은 명령을 실행한다. 예를 들어 다음과 같이 수행할 수 있다.

cd pcl_simple_example
mkdir build
cd build
cmake ..
make

정상적으로 PCL 라이브러리가 포함되어 있다면 컴파일 및 링크 과정에서 별다른 오류가 발생하지 않아야 한다. 빌드가 완료되면 실행 파일(pcl_simple_example)이 build 디렉터리에 생성된다. 해당 바이너리를 실행하면 콘솔에 포인트 수와 각 점의 좌표가 순차적으로 출력되며, 예제 프로그램이 제대로 동작하는지 확인할 수 있다.

PCL 헤더가 올바르게 포함되지 않았거나, FIND_PACKAGE(PCL 1.2 REQUIRED) 부분에서 PCL 경로를 찾지 못한다면 컴파일 단계에서 에러 메시지가 나타난다. 이런 경우에는 CMake가 PCL 설정 파일들을 올바르게 인지할 수 있도록 환경 변수를 확인해야 한다. Linux 환경에서는 pkg-config나 CMAKE_PREFIX_PATH 등을 수정해 주거나, Windows 환경에서는 Visual Studio에서 빌드할 때 환경 변수를 수동으로 설정해 줄 필요가 있다. 문제 해결을 위해서는 PCL이 설치된 경로의 include 디렉터리나 라이브러리 디렉터리를 CMake가 참조하고 있는지 점검해야 한다.

간단한 빌드 프로세스 흐름(mermaid 예시)

개발 환경에서 프로젝트가 어떻게 빌드되는지 흐름을 간단히 나타내면 다음과 같은 순서를 거친다.

flowchart TB A[CMakeLists.txt 준비] --> B(빌드 디렉터리 생성) B --> C[cmake .. 실행] C --> D(make 명령어로 컴파일 및 링크) D --> E(실행 파일 생성)

CMake를 통해 프로젝트를 설정하고 Makefile을 생성한 뒤, 해당 Makefile에 정의된 규칙에 따라 실제 컴파일이 수행된다. 작업 환경에 따라 ninja 빌드 시스템을 쓸 수도 있지만, 원리는 동일하다. 최종적으로 생성된 실행 파일을 ./pcl_simple_example 처럼 실행하여 결과를 확인할 수 있다.

Windows 환경에서의 CMake 설정

Windows에서 Visual Studio를 사용하여 PCL 예제 프로젝트를 빌드하려면 기본적으로 CMake 설정과 빌드 도구가 제대로 설치되어 있어야 한다. Visual Studio가 설치되어 있으면 CMake도 함께 설치되어 있는 경우가 많지만, 만약 없다면 별도로 CMake를 다운로드하여 설치한다. CMake GUI를 사용하면 환경 변수나 경로 설정을 조금 더 직관적으로 확인할 수 있으나, 명령줄에서 작업해도 큰 차이는 없다.

Windows에서 PCL이 설치된 경로(예: C:/Program Files/PCL 등)와 Visual Studio가 제공하는 컴파일러 환경이 일치해야 CMake가 PCL을 올바르게 찾는다. FIND_PACKAGE(PCL 1.2 REQUIRED) 구문이 제대로 동작하지 않는 경우에는 CMAKE_PREFIX_PATH나 PCL_DIR 환경 변수를 설정하여 CMake에게 PCL 패키지 정보가 있는 위치를 알려 줄 수 있다.

설정 예시로, PCL이 C:/PCL 경로에 설치되어 있다고 가정하면 Visual Studio 명령 프롬프트에서 다음 명령을 실행한다.

set CMAKE_PREFIX_PATH=C:/PCL;%CMAKE_PREFIX_PATH%

이렇게 설정하면 CMake가 C:/PCL 아래에서 PCL 관련 구성 파일(예: PCLConfig.cmake)을 찾으려고 시도한다. 만약 CMakeLists.txt 구성에서 찾고자 하는 PCL 버전(1.10 등)이 실제 설치 버전과 다르면 에러가 발생할 수 있으므로 버전을 맞춰 주거나, 버전 정보를 생략해 보고 찾는 방법도 시도해 볼 수 있다.

Visual Studio 솔루션 생성

Windows에서 CMake를 실행하면 Makefile 대신 Visual Studio 솔루션(.sln) 파일과 프로젝트(.vcxproj)가 생성된다. 예를 들어 pcl_simple_example/build 디렉터리에서 cmake .. -G "Visual Studio 17 2022" 와 같이 특정 버전의 Visual Studio를 지정하면, 해당 버전에 호환되는 프로젝트 파일이 만들어진다. 완료 후 pcl_simple_example.sln 파일을 Visual Studio에서 열면 IDE 환경에서 디버깅과 실행을 할 수 있다.

Visual Studio IDE를 사용하면 소스 파일을 편집하고, 디버거로 변수 값을 확인하거나 중단점을 설정하기가 편하다. 특히 PCL 코드를 처음 접한다면 IDE 환경에서 디버거를 통해 포인트클라우드 객체 내부를 관찰해 보는 것이 도움이 된다. 점이 올바르게 삽입되었는지, 좌표 값이 의도한 대로 변환되는지 등을 편리하게 확인할 수 있다.

런타임 라이브러리 설정

Windows 환경에서 PCL 라이브러리는 보통 동적 라이브러리(.dll) 형태로 제공된다. Visual Studio에서 빌드된 실행 파일은 PCL DLL 파일을 찾기 위해 시스템 경로를 참조하거나, 실행 파일이 있는 디렉터리에서 해당 DLL 파일을 검색한다. 빌드 후 실행 시에 "DLL을 찾을 수 없다"는 오류가 발생한다면, PCL DLL이 위치한 디렉터리를 PATH 환경 변수에 추가하거나, 실행 파일이 있는 위치로 해당 DLL을 복사해 두면 된다.

만약 정적 링크를 사용하기 위해 .lib 파일을 찾으려면, PCL을 빌드할 때 정적 라이브러리 옵션을 활성화해야 한다. 그러나 일반적인 Windows 설치 패키지에서는 동적 라이브러리로만 제공되는 경우가 많다. 정적 라이브러리를 직접 빌드하는 것은 프로젝트 규모가 커졌을 때, 배포 환경에 맞춰 라이브러리 종속성을 줄이기 위해 고려할 수 있는 선택지다.

디버그/릴리스 설정

Visual Studio를 사용해 CMake 프로젝트를 빌드하면 기본적으로 디버그( Debug )와 릴리스( Release ) 설정을 각각 독립적으로 구성할 수 있다. 예를 들어 디버그 모드에서 프로젝트를 빌드한 다음, 릴리스 모드로 전환해 빌드하면 별도의 출력 폴더에 최적화가 적용된 실행 파일이 생성된다. CMakeLists.txt에서 CMAKE_BUILD_TYPE 변수를 설정하거나, Visual Studio에서 빌드 구성을 직접 전환하여 모드를 선택한다.

PCL도 디버그 라이브러리와 릴리스 라이브러리를 구분하여 제공한다. 디버그 모드에서 빌드 시, 반드시 PCL 디버그 라이브러리를 찾아서 링크해야 링크 에러를 방지할 수 있다. Visual Studio에서 CMake를 사용하면 이 과정이 자동화되어 있지만, 직접 설정이 필요한 경우라면 FIND_PACKAGE(PCL REQUIRED COMPONENTS ...) 구문이나 target_link_libraries() 구문에 라이브러리 파일명을 명시해 주어야 한다.

macOS 환경에서의 프로젝트 구성

macOS에 PCL을 설치했다면 Homebrew를 통해 설치한 경우가 많다. 예를 들어 brew install pcl 명령으로 PCL을 설치할 수 있다. 이때 Homebrew가 /usr/local/Cellar 또는 /opt/homebrew/Cellar(Apple Silicon) 경로에 라이브러리와 헤더 파일을 배치한다. CMake가 Homebrew 경로를 자동으로 인식하지 못하면, 아래와 같이 CMAKE_PREFIX_PATH 등을 지정해 줄 필요가 있다.

export CMAKE_PREFIX_PATH=/opt/homebrew:$CMAKE_PREFIX_PATH

또한 macOS 환경에서는 Xcode나 CLion과 같은 IDE를 사용하여 PCL 프로젝트를 컴파일하고 디버깅할 수 있다. Xcode를 사용하기 위해서는 CMake 설정 단계에서 -G "Xcode"를 전달하여 Xcode 프로젝트를 생성한 뒤, 해당 .xcodeproj 파일을 열어 프로젝트를 빌드할 수 있다. CLion은 내부적으로 CMake를 사용하는 IDE이므로, PCL 경로 인식만 올바르게 이루어지면 별다른 추가 설정 없이도 예제 프로젝트를 빌드한다.

C++ 표준 설정

PCL을 사용하려면 C++11 이상의 표준이 요구되는 경우가 많다. 특히 템플릿 라이브러리를 다룰 때 컴파일러 지원이 충분해야 정상 동작하므로, 다음과 같이 CMakeLists.txt에 설정을 명시하는 것이 좋다.

set(CMAKE_CXX_STANDARD 14)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

이렇게 하면 프로젝트가 C++14 표준을 사용하도록 강제되며, 지원되지 않는 컴파일러인 경우 에러가 발생한다. 일부 최신 PCL 기능을 활용할 때는 C++17 또는 그 이상의 표준이 필요할 수 있으므로, 사용하는 PCL 버전에 맞추어 적절한 표준을 선택한다.

컴파일 최적화와 옵션

PCL 예제 프로젝트가 어느 정도 규모가 커지면, 컴파일 시간을 단축하고 실행 효율을 높이기 위해 최적화 옵션을 고려하게 된다. 예를 들어, 릴리스 빌드에서는 다음과 같은 플래그가 자동 설정될 수 있다.

CFLAGS=−O3,CXXFLAGS=−O3\text{CFLAGS} = -O3, \quad \text{CXXFLAGS} = -O3

디버그 빌드에서는 -g 플래그를 통해 디버깅 심볼을 포함하고, 최적화를 꺼서 개발 과정에서 변수 값이나 스택 프레임을 좀 더 쉽게 확인할 수 있게 한다. CMake에서는 다음과 같이 매개변수를 바꿔 가며 빌드를 반복할 수 있다.

cmake -DCMAKE_BUILD_TYPE=Debug ..
make

혹은

cmake -DCMAKE_BUILD_TYPE=Release ..
make

이렇게 하면 같은 소스 코드라도 컴파일러가 다른 설정으로 바이너리를 만들어 낸다.

예제 포인트클라우드 시각화

현재 예제는 단순히 콘솔에 포인트 좌표만 출력하지만, 시각적으로 확인하기 위해서는 PCL의 시각화 모듈을 사용할 수 있다. 예를 들어 pcl/visualization/cloud_viewer.h를 포함하면 간단하게 윈도우 창에 포인트들을 표시할 수 있다. CMakeLists.txt에는 PCL 컴포넌트 중 visualization 모듈 링크가 활성화되어야 하므로, 필요한 경우 다음과 같이 수정할 수 있다.

find_package(PCL 1.2 REQUIRED COMPONENTS common io visualization)
...
target_link_libraries(pcl_simple_example ${PCL_LIBRARIES})

샘플 코드를 약간 확장해 보려면 다음처럼 작성할 수 있다.

#include <pcl/point_types.h>
#include <pcl/point_cloud.h>
#include <pcl/visualization/cloud_viewer.h>

int main(int argc, char** argv)
{
    pcl::PointCloud<pcl::PointXYZ>::Ptr cloud(new pcl::PointCloud<pcl::PointXYZ>);
    cloud->width = 5;
    cloud->height = 1;
    cloud->points.resize(cloud->width * cloud->height);

    for (std::size_t i = 0; i < cloud->points.size(); ++i)
    {
        cloud->points[i].x = static_cast<float>(i);
        cloud->points[i].y = static_cast<float>(i * 0.1);
        cloud->points[i].z = static_cast<float>(i * 0.01);
    }

    pcl::visualization::CloudViewer viewer("Cloud Viewer");
    viewer.showCloud(cloud);

    while (!viewer.wasStopped())
    {
    }
    return 0;
}

pcl::visualization::CloudViewer 객체를 이용하면 간단한 라이브 3D 시각화 창이 뜨고, 프로그램 종료 전까지 점들을 화면에서 확인할 수 있다. 이 시각화 예제도 빌드와 실행이 성공한다면, 라이브러리와 헤더 경로, 링크 설정 모두가 정상적으로 구성되었다고 볼 수 있다.

프로젝트 확장 시 주의 사항

기본 예제 프로젝트가 동작하는 것을 확인했다면, 이후에는 여러 소스 파일을 추가하거나 서브디렉터리를 만들어 프로젝트를 확장할 수 있다. PCL은 다양한 모듈(예: filters, features, segmentation, registration, io, visualization 등)을 제공하기 때문에, 프로젝트가 커질수록 모듈별로 코드를 분리하는 것이 좋다. 이때는 CMakeLists.txt를 계층 구조로 작성하여 각 디렉터리에 맞는 설정을 부여하고, 상위 CMakeLists.txt에서 필요한 서브디렉터리를 add_subdirectory()로 포함하면 프로젝트 관리가 용이하다.

예컨대 pcl_simple_example 프로젝트가 다음과 같이 확장된다고 가정한다.

pcl_simple_example
├─ CMakeLists.txt
├─ src
│   ├─ CMakeLists.txt
│   ├─ main.cpp
│   ├─ filters.cpp
│   └─ visualization.cpp
└─ include
    ├─ filters.h
    └─ visualization.h

상위 CMakeLists.txt에서 add_subdirectory(src)를 통해 src/CMakeLists.txt를 빌드 단계에 포함시키고, 실제 구현부 filters.cpp, visualization.cpp가 각각의 헤더와 함께 배치되는 형태다. 이렇게 구조화하면 새로운 기능을 모듈별로 깔끔히 나누어 관리할 수 있으며, 각 모듈 내에서 PCL 기능을 적절히 가져다 쓰기 편리하다.

추가 의존성 라이브러리와의 연동

PCL은 자체적으로 수많은 모듈을 제공하지만, 다른 서드파티 라이브러리(예: OpenCV, Eigen, Boost 등)를 함께 사용해야 하는 경우가 많다. 특히 PCL은 내부적으로 Eigen, Boost를 광범위하게 활용하므로, CMake 구성 시 이들 라이브러리와 버전이 충돌하지 않도록 주의해야 한다. Homebrew나 vcpkg 같은 패키지 매니저를 이용하면 의존성 관리가 비교적 용이하다.

상황에 따라 다음과 같이 해당 라이브러리를 명시적으로 찾고 인클루드 디렉터리를 추가할 수도 있다.

find_package(Eigen3 REQUIRED)
include_directories(${EIGEN3_INCLUDE_DIR})

PCL에서 이미 Eigen3를 내부적으로 사용해도, 특정 버전 이상의 Eigen을 요구할 경우 명시적으로 find_package를 호출해 충돌을 방지하거나 올바른 경로를 지정해 주는 것이 좋다. 여러 버전이 공존할 때는 CMake가 어떤 경로를 우선으로 잡는지 확인해야 한다.

예외 처리 및 디버깅

PCL 코드를 작성하다 보면, 포인터와 템플릿 인스턴스가 많이 사용되기 때문에 빌드 에러나 런타임 에러를 만날 가능성이 있다. 예를 들어 포인트클라우드 타입이 pcl::PointXYZ인지 pcl::PointNormal인지 잘못 지정되면 빌드 중에 템플릿 오류가 발생할 수 있다. 컴파일러 에러 메시지가 길어지면, 템플릿 인스턴스 경로를 하나씩 추적해 보면서 문제점을 파악해야 한다.

또한 포인터를 pcl::PointCloud<T>::Ptr와 같은 똑같은 스마트 포인터로 처리하는 것이 권장된다. 기존에 std::shared_ptr 등 다른 스마트 포인터를 혼합해서 사용하다 보면 참조 카운트가 꼬이거나, 해제 시점이 어긋나서 예기치 않은 상황이 발생할 수도 있다. 디버깅 시에는 GDB, LLDB, Visual Studio 디버거 등을 활용하여 포인트클라우드 객체가 올바르게 생성되고 해제되는지 확인하는 습관을 들이는 것이 좋다.

CI/CD 환경에서의 PCL 빌드

프로젝트가 단순 예제를 넘어 실제 서비스나 솔루션에 적용되면, GitLab CI, GitHub Actions, Jenkins 등 CI/CD 환경에서 자동 빌드/테스트 파이프라인을 구성하는 경우가 많다. 이때도 PCL 설치 및 빌드 단계를 스크립트화해 두면 편리하다. 예를 들어 우분투 기반 Docker 컨테이너를 사용한다면, 다음과 같은 식으로 Dockerfile을 구성할 수 있다.

FROM ubuntu:20.04
RUN apt-get update && apt-get install -y build-essential cmake libpcl-dev
WORKDIR /app
COPY . /app
RUN mkdir build && cd build && cmake .. && make
CMD ["./build/pcl_simple_example"]

이 Docker 이미지를 빌드하면 PCL이 설치된 상태의 환경에서 예제 프로젝트가 자동 컴파일되고, CI 상에서 테스트도 진행할 수 있다. macOS나 Windows CI 환경에서도 비슷한 개념으로 스크립트를 구성하면, 여러 운영 체제에서의 호환성을 지속적으로 검증할 수 있다.
