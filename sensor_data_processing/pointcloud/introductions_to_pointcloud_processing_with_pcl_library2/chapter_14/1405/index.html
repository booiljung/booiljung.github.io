Doxygen을 이용한 자동 문서화와 환경 설정 기초¶

Doxygen은 C++ 코드를 기반으로 자동화된 문서를 생성하는 강력한 도구다. 포인트클라우드 라이브러리(PCL) 같은 대규모 라이브러리나 프로젝트에서, 코드 주석만으로도 함수와 클래스들의 사용 방법과 의존 관계를 포괄적으로 파악할 수 있다. 이를 위해 먼저 sudo apt-get install doxygen graphviz 같은 명령으로 필요한 패키지를 준비한다. 이후 프로젝트 루트 디렉터리에서 doxygen -g 명령을 실행하면 기본 Doxyfile을 생성할 수 있다. 이 파일은 Doxygen의 설정을 담고 있으며, 파라미터를 변경하여 HTML, PDF 등 문서화 형식을 조정한다. 예컨대 INPUT 섹션을 수정하여 소스 파일 경로를 지정하거나 GENERATE_LATEX 값을 YES로 전환해 LaTeX 문서를 생성하도록 설정할 수 있다.

코드 레이아웃과 주석 스타일¶

프로젝트 내에서 Doxygen으로 문서화를 극대화하기 위해서는 특정 주석 패턴을 준수해야 한다. 예를 들면 함수 선언부 위에 /// 또는 /** */ 형태로 설명을 붙이고, 파라미터와 반환값을 @param, @return 태그로 명확하게 기술한다. 특히 PCL과 같이 템플릿 기반, 다양한 자료구조가 혼재된 프로젝트에서는 클래스 템플릿 매개변수나 상속 구조에 대한 설명이 제대로 남아 있어야 유지보수 시 혼란이 줄어든다. 이러한 주석 스타일을 팀 차원에서 통일해 두면, 여러 개발자가 동시에 코드를 수정하더라도 생성된 문서의 일관성을 유지할 수 있다.

자동 문서화 흐름과 CI/CD 파이프라인 연동¶

지속적인 유지보수를 위해서는 Git이나 SVN 같은 버전 관리 시스템과 CI/CD 파이프라인에 Doxygen 실행 단계를 통합하는 방법이 자주 활용된다. 예컨대 GitHub Actions, Jenkins, GitLab CI 등에서 코드를 빌드할 때마다 doxygen Doxyfile을 실행하여 최신 문서가 자동으로 빌드되도록 설정할 수 있다. 그 결과물인 HTML 문서는 프로젝트 위키나 호스팅 서버에 배포되며, 팀원들은 언제든지 최신 API 문서를 열람할 수 있게 된다. 이런 자동화가 구축되면, 문서를 수동으로 따로 갱신하는 과정 없이도 코드 변동 사항이 곧바로 문서에 반영되어 편리하다.

UML 다이어그램과 Graphviz 통합¶

Doxygen은 Graphviz를 활용하여 클래스 상속 구조나 콜라보레이션 다이어그램을 시각적으로 표현한다. 이는 PCL 프로젝트처럼 복잡한 계층을 가진 라이브러리에서 특히 유용하다. mermaid 문법을 사용할 수도 있지만, Doxygen 문서 내부에서 UML 수준의 구조를 표현할 때는 dot 기반 그래프나 class diagram 태그를 주로 활용한다. 예를 들어, 다음과 같은 다이어그램을 별도의 마크다운 문서와 함께 관리하면서 Doxygen 문서에 링크시키는 방법도 가능하다.

classDiagram class PointCloud { +width +height +isOrganized() } class PCLAlgorithms { +computeNormals() +filterOutliers() } PointCloud --> PCLAlgorithms

이렇게 시각적으로 구조를 파악하면, 다양한 필터, 정렬, 세그멘테이션 알고리즘 간의 상호 의존 관계를 한눈에 이해할 수 있다.

매크로와 템플릿 문서화 기법¶

PCL 내부 코드에는 템플릿이나 매크로가 자주 등장한다. 이 경우 일반적인 함수와 달리 주석에 <typename T>와 같은 템플릿 인자를 명확히 표기해야 한다. 예를 들어 템플릿 클래스 선언부 위에 /** \tparam T 포인트 타입 */처럼 Doxygen 태그를 달면, API 문서에서 해당 템플릿의 의미가 분명하게 드러난다. 매크로의 경우도 \def나 \macro 같은 Doxygen 태그를 사용하면, 특정 매크로가 무엇을 의미하고 어떻게 확장되는지 설명을 추가할 수 있다. 여러 매크로가 연쇄적으로 연결되는 구조라면, 매크로 간 연결 관계를 파악할 수 있도록 \file 태그 등을 적극 활용해 문서를 풍부하게 만든다.

커스텀 태그와 예시 코드 문서화¶

복잡한 포인트클라우드 처리는 예제 코드가 매우 중요한 의미를 가진다. Doxygen의 \example이나 \snippet 태그를 활용하면, 특정 소스 파일이나 코드 조각을 직접 문서에 삽입하여 실제 사용 방법을 시각적으로 보여줄 수 있다. 예컨대 PCL의 특정 필터 함수를 사용하는 코드를 아래와 같이 삽입할 수 있다.

/**
 * \snippet src/filter_example.cpp example_filter_usage
 */

그리고 src/filter_example.cpp 내부에서 다음과 같은 구조의 주석을 추가하여 스니펫 범위를 지정하면 된다.

/// \snippet example_filter_usage start
// 예제 코드 시작
...
// 예제 코드 끝
/// \snippet example_filter_usage end

이런 방식으로 예제 코드를 문서에서 쉽게 확인할 수 있으며, 예제 내용을 수정해도 Doxygen 문서를 재빌드하면 최신 코드 예시가 자동으로 반영된다.

유지보수를 위한 코드 구조 리팩토링 전략¶

문서화를 단순히 코드 위에 덧붙이는 방식으로만 접근하면, 프로젝트가 커질수록 문서와 실제 구현 내용이 어긋날 가능성이 커진다. 따라서 중복 코드나 필요 없는 헬퍼 함수를 발견했을 때, 즉시 재정리하여 주석 내용과 구현을 일치시키는 작업이 중요하다. 또한, PCL처럼 외부 의존성이 큰 라이브러리는 특정 모듈이 다른 모듈에 종속되는 구조가 복잡하므로, 모듈 단위로 폴더를 재배치해 문서 내용이 더 명확히 구분되도록 해줄 필요가 있다. 예컨대 필터 관련 코드는 filters, 정렬 관련 코드는 registration, 세그멘테이션은 segmentation 폴더로 분리하고, 이에 맞춰 Doxyfile의 INPUT 섹션도 모듈별로 작성해주는 식이다.

코드 커버리지와 Doxygen 연동¶

프로젝트가 성장하면 단순히 API 문서만 잘 정리된다고 해서 품질이 보장되는 것은 아니다. 코드가 실제로 어느 부분까지 테스트되었는지를 가시화하는 작업도 함께 진행되어야 한다. 이런 맥락에서 코드 커버리지 도구(예: gcov, lcov, cobertura 등)의 결과물을 Doxygen에 통합 표시하거나, 문서화 파이프라인 후처리로 별도의 리포트를 생성해 배포하는 방식을 고려할 수 있다. 예컨대 lcov를 통해 생성된 HTML 커버리지 리포트를 Doxygen 문서 내부에서 링크하거나, CI/CD 파이프라인이 문서 빌드 후 결과물을 자동 업로드하여, 문서 사용자(개발자)가 API와 테스트 커버리지를 동시에 확인하도록 돕는다.

대규모 협업 환경에서의 충돌 방지 전략¶

여러 개발자가 동시에 포인트클라우드 알고리즘 구현을 수정하거나 확장하는 상황이 잦다. 이때 Doxygen 주석이나 함수 시그니처가 자주 바뀌면 머지 충돌이 일어날 수 있다. 이를 방지하기 위해서는 다음과 같은 흐름을 따르는 것이 좋다(본 문서에서는 나열 형태 대신 요약으로 서술한다).

  • Pull Request를 생성할 때 반드시 해당 함수의 문서화 내용이 최신 상태인지 검사하고, 팀 내 리뷰어는 주석이 새로운 구현 내용을 반영하는지 우선 확인한다.
  • 새로운 알고리즘이나 자료구조를 추가할 때, 설계 문서 혹은 UML 다이어그램에 반영된 내용을 Doxygen 주석에도 일관성 있게 반영한다.
  • 기존 함수 시그니처를 변경할 경우, 다른 부분에서 호출되고 있는지(특히 템플릿 인자나 매개변수 타입)에 대한 검토를 거친 뒤, 관련 주석도 즉시 고쳐야 한다.

템플릿 라이브러리 문서화 시 발생할 수 있는 이슈¶

PCL은 템플릿 기반의 자료형을 광범위하게 활용한다. 템플릿 코드가 많아질수록 Doxygen 문서에서 클래스나 함수 템플릿의 구체화 정보가 부분적으로 누락되거나, 인스턴스화되는 시점에 맞춰 다르게 해석될 수 있다는 점에 주의해야 한다. 예를 들어, 특정 템플릿 클래스가 float 타입과 double 타입에 대해 다른 특수화를 가지면, 이를 명시적으로 주석으로 구분해줄 필요가 있다.

  • 문서화를 보완하는 한 가지 방법은 \if 지시자를 이용해 컴파일러별, 템플릿 인자별로 다른 설명을 선택적으로 보여주는 것이다. 그러나 지나치게 복잡해지지 않도록, 가능하면 템플릿 인자의 다양한 사용 예시를 \example 태그를 통해 제시하는 편이 낫다.
  • 또한 Doxygen 설정에서 EXTRACT_ALL, EXTRACT_PRIVATE, EXTRACT_STATIC 같은 옵션을 적절히 조정해, 템플릿 메서드나 private 영역에 대한 문서화를 어느 범위까지 생성할지 팀 내에서 합의가 필요하다.

변경 이력(Changelog)와 버전 태그 관리¶

장기적으로는 API 변경 이력과 버전을 체계적으로 관리하는 것이 필수다. PCL 라이브러리처럼 기능이 확장될 때마다 함수 이름이 바뀌거나 시그니처가 달라지면, 별도의 Changelog 문서나 Git 태그, 릴리스 노트 등을 통해 변경 이력을 정리해야 한다.

  • Doxygen은 기본적으로 Changelog를 특별히 처리해 주지 않으므로, 별도의 마크다운 문서나 reStructuredText 문서 등을 함께 운영하며 주요 변경 사항을 기록하고, Doxygen 문서에서는 그 문서로 링크를 제공하는 방식이 이상적이다.
  • API 변경 주기가 짧은 프로젝트일수록 버전별 Doxygen 문서를 아카이빙하는 방안을 검토할 수 있다. 예를 들어 docs/v1.0, docs/v1.1 등 버전에 맞춰 아예 별도의 문서 디렉터리를 준비하고, CI/CD가 태그를 달 때마다 해당 버전에 대응하는 문서를 빌드해 보관한다.

지속적인 품질 개선을 위한 문서 리뷰 프로세스¶

Doxygen 문서가 단순히 자동 생성된다고 해서 내용이 항상 유효하거나, 실무에 유용한 수준으로 유지되는 것은 아니다. 정기적으로 문서 리뷰 프로세스를 갖추어야 하며, 특히 다음 사항을 집중 점검한다.

  • 새로운 API 추가 시, 제대로 된 설명과 @param, @return, @tparam 등의 태그가 누락되지 않았는지 확인한다.
  • 사용하지 않는 혹은 더 이상 유효하지 않은 함수나 클래스에 대한 문서가 남아 있지는 않은지 살핀다.
  • 같은 기능을 하는 여러 함수가 분산되어 있으면 하나로 통합할 것을 제안하고, 통합이 이루어지면 문서도 일원화한다.

소스 코드 예시: CMake와 Doxygen 스크립트 연동¶

Doxygen 문서화를 자동화하려면, 빌드 시스템인 CMake와의 연동이 유용하다. 예시로 최상위 CMakeLists.txt에 Doxygen 사용을 위한 옵션을 정의하고, 빌드 타겟을 추가하여 빌드 과정에서 자동으로 문서를 생성할 수 있다. 다음은 간단한 예시다.

# 프로젝트 최소 요구 CMake 버전 및 프로젝트 선언
cmake_minimum_required(VERSION 3.10)
project(PCL_Doxygen_Example)

# Doxygen 패키지 탐색
find_package(Doxygen REQUIRED)

# Doxygen 옵션을 담을 설정 파일 생성 경로
set(DOXYGEN_IN ${CMAKE_CURRENT_SOURCE_DIR}/Doxyfile.in)
set(DOXYGEN_OUT ${CMAKE_CURRENT_BINARY_DIR}/Doxyfile)

# configure_file 함수로 Doxyfile.in을 치환하여 Doxyfile을 만든다
configure_file(${DOXYGEN_IN}${DOXYGEN_OUT} @ONLY)

# doxygen 타겟을 구성
add_custom_target(doc_doxygen
    COMMAND ${DOXYGEN_EXECUTABLE}${DOXYGEN_OUT}
    WORKING_DIRECTORY ${CMAKE_CURRENT_BINARY_DIR}
    COMMENT "Generating Doxygen documentation..."
)

위 코드 예시에서 Doxyfile.in 파일에는 @PROJECT_NAME@, @CMAKE_SOURCE_DIR@ 등 치환을 위한 매크로가 들어 있으며, configure_file 함수가 이들을 실제 값으로 변환해 최종 Doxyfile을 만든다. 이렇게 하면 make doc_doxygen 명령만으로도 간단하게 Doxygen 문서를 빌드할 수 있고, CI/CD 파이프라인에서 doc_doxygen 타겟을 호출하여 자동으로 문서를 생성·배포하도록 설정할 수 있다. 특히 PCL처럼 다양한 모듈과 헤더 파일이 여러 경로에 분산되어 있다면, INPUT 지시자를 모듈별로 나누어 Doxyfile에 기재한 뒤, CMake 설정으로 통합 관리하는 것이 이상적이다.

포인트클라우드 프로젝트에서의 API 문서 체계 구조 설계¶

포인트클라우드 처리 알고리즘은 크게 필터링, 분할(Segmentation), 피처 추출, 정합(Registration) 등 여러 영역으로 나뉘며, 각 모듈마다 상호 의존성이 복잡하게 얽힌다. 이런 경우, Doxygen 문서를 단일 대형 API 레퍼런스로 만들기보다는, 영역별로 분할된 모듈 문서를 구성하여 상호 참조할 수 있게 하는 편이 유지보수에 유리하다. 예를 들어, PCL 내부 모듈 구성을 그대로 반영하여 filters, features, registration 섹션을 따로 두고, 공통 자료구조인 PointCloud나 KdTree 등은 별도의 공통 섹션으로 추출해 중앙에서 레퍼런스를 제공한다.

또한 PCL은 2차원 이미지 처리 라이브러리와 달리 3차원 공간 정보를 다루므로, 기하학적 연산과 좌표 변환 등이 API 전역에서 중요하게 취급된다. 이런 내용을 별도의 문서(예: Transformations or Geometry)로 마련해, 수학적 전제나 부동소수점 이슈, 혹은 난수 발생 기법 등 공통되는 세부 사항을 묶어두면 좋다. 팀원들이 각 기능 모듈을 개발하거나 수정할 때, 이 공통 문서를 통해 필수 전제와 수학적 조건을 재확인하도록 유도할 수 있다.

API 문서와 예제 테스트베드 운영¶

포인트클라우드 알고리즘은 입력 데이터나 파라미터 설정에 따라 결과가 크게 달라진다. 따라서 코드로만 구현 상세를 살펴보기보다는, 다양한 실제 예제 데이터를 통해 알고리즘의 동작을 직접 시험해 보는 과정이 중요하다. 이런 맥락에서, Doxygen 문서에 포함된 예제 코드를 독립된 테스트베드 형태로 운영하는 전략을 고려할 수 있다. 예컨대 examples 디렉터리를 만들어, filters_example.cpp, registration_example.cpp 등 각 기능 모듈에 상응하는 샘플 코드를 두고, CMake에서 별도의 실행 타겟을 만들어 누구나 쉽게 ./registration_example을 돌려볼 수 있게 만든다.

Doxygen 문서에서는 \example 태그를 통해 이 예제 소스가 어떻게 동작하는지, 주요 함수 호출 순서나 파라미터 설정 방법이 무엇인지, 그리고 실제로 어떤 포인트클라우드를 입력했을 때 어떤 결과가 나오는지를 상세히 설명한다. 시각화가 필요한 경우, PCL의 시각화 모듈(pcl_visualization)을 이용해 점군 데이터에 컬러를 입히거나, 포인트 사이즈를 조절하는 방식을 보여줄 수도 있다. 이런 과정을 통해 팀원 간 지식 전파가 더 쉽게 이뤄지고, 신입 혹은 외부 기여자도 API 문서를 보고 빠르게 프로젝트 전반을 이해할 수 있다.

참고 자료 관리와 린팅 도구¶

Doxygen 문서 뿐 아니라, 레퍼런스로 사용되는 학술 논문, 알고리즘 개요, 혹은 수학 공식을 따로 정리한 문서 등을 함께 관리하면, 팀원들이 특정 알고리즘의 근거 이론이나 최적화 기법에 대한 출처를 확인하기 용이하다. 예컨대 Fast Point Feature Histograms(FPFH) 알고리즘에 대한 논문, Iterative Closest Point(ICP)의 변형 알고리즘 레퍼런스 등을 한데 모아두고, Doxygen 문서에 참고 자료로 링크를 달아두면, 유지보수 시 해당 이론적 배경을 재확인하기 쉬워진다.

또한 C++ 코드 품질을 유지하기 위해 clang-tidy, cppcheck 같은 린팅 도구를 도입할 수 있다. 함수 인자 순서나 템플릿 특수화, 매크로 사용이 복잡해지면 의도치 않은 버그나 스타일 균열이 생길 수 있다. 린팅 도구로 코드를 점검한 뒤, Doxygen 주석에도 권장 스타일을 일관되게 반영하면 프로젝트 전반의 코드·문서 품질이 함께 향상된다.

고급 사용: Doxygen 내부 명령어 확장¶

Doxygen은 기본적인 \brief, \param, \return 등 태그 외에도, \cond, \endcond, \mainpage, \page, \section 등을 이용해 문서를 풍부하게 구성할 수 있다. 예를 들면, 특정 플랫폼 전용 코드(CUDA 연산, ROS 통합 코드 등)나 실험적 기능을 감추고 싶다면 \cond ~ \endcond 블록으로 감싸, 일반 문서에서는 노출되지 않게 처리할 수 있다. 그리고 문서 메인 페이지(\mainpage)에 프로젝트 전반 개요나 PCL과의 통합 구조, 빌드 방법 등을 소개해두면, 사용자가 처음 문서를 접할 때 한눈에 프로젝트의 기능 범위를 이해하게 된다.

만약 폴더 구조나 파일 단위로 문서를 작성할 필요가 있다면 \page 지시자를 통해 임의의 페이지를 구성하고, 그 안에 상세 설명이나 스크린샷을 포함할 수 있다. 이 방식은 모듈별로 별도의 하위 페이지를 두어, 예를 들면 \page filters_module 필터 모듈 안내 같은 식으로 구성을 나누고, 그 페이지에서 관련 클래스나 함수를 연결해주면 된다.

마크다운 문서와 Sphinx, MkDocs 등과의 비교¶

Doxygen 외에도 Python 생태계에서 Sphinx, JavaScript 기반의 MkDocs 등 마크다운 문서화 툴이 인기를 끌고 있다. 이런 툴들은 일반 텍스트 문서를 중심으로 구성되어, API 자동 추출보다는 설명서 혹은 가이드 문서를 작성하기에 편리하다. 반면 C++ 프로젝트에서의 API 자동 문서화를 생각한다면, Doxygen이 훨씬 직관적이고 강력하다. 따라서 대규모 PCL 기반 프로젝트에서 C++ API 도큐멘테이션이 주를 이루는 경우, Doxygen이 사실상 표준에 가깝다고 볼 수 있다. 다만 Sphinx와 Doxygen을 혼합해 사용하거나, Doxygen이 생성한 XML 출력물을 이용해 Sphinx에서 추가로 HTML 문서를 만들 수도 있다. 어떤 방식을 택하든, 궁극적으로는 팀 내 협업과 유지보수가 편리한 구조를 설계하는 것이 핵심이다.
