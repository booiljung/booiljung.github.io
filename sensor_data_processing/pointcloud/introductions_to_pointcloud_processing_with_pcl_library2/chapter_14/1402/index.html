Git 사용의 중요성과 확장 개념¶

PCL 기반 포인트클라우드 프로젝트에서 작업자가 여러 명이 되거나, 프로젝트 규모가 커짐에 따라 소스 코드를 안정적으로 관리하고 추적하기 위해서는 버전 관리 시스템이 필수적이다. 특히 Git은 분산 버전 관리 도구로서, 중앙 서버가 없어도 각자의 로컬 저장소에 전체 이력을 담을 수 있고, 병렬적인 작업도 유연하게 지원한다. 포인트클라우드 데이터가 크고, 관련 알고리즘의 최적화 시도가 자주 발생하며, 실험용 브랜치를 통한 시도와 폐기가 빈번한 상황에서는 Git이 제공하는 브랜치·머지 전략이 매우 효율적이다.

장기적으로 유지해야 할 기능, 단기적 실험으로 진행될 기능, 그리고 연구 단에서만 관리될 기능 각각이 독립 브랜치로 운영될 수 있기 때문에, 협업 중에도 서로의 작업이 섞이거나 충돌하는 것을 최소화할 수 있다. 또한 PCL에서 제공하는 여러 알고리즘(예: ICP, RANSAC 필터링, 분할 클러스터링 등)을 독립적인 기능으로 모듈화하여, 필요한 기능을 원하는 타이밍에 브랜치로 병합하거나 제외할 수 있다. 이는 프로젝트 복잡도가 증가해도 전체 구조를 통제하고 이력을 보존하기에 적합하다.

Git의 기본적인 구성요소인 저장소(repository), 작업 디렉터리(working directory), 스테이징(staging) 영역의 활용부터, 브랜치 전략, 태그(tag) 사용, CI/CD 연동까지 연결하는 과정을 체계화하면, 협업에서 발생하는 거의 모든 버전 관리 이슈를 제어할 수 있다. 복잡한 PCL 프로젝트에서 흔히 발생하는 문제는 “어떤 버전의 알고리즘 모듈이 어느 시점에 적용되었는가”를 명확히 기록하고 추적하기가 어렵다는 것인데, Git 리포지토리에 적절히 태그와 릴리즈 노트를 작성해두면 상황을 훨씬 깔끔하게 해결할 수 있다. 예컨대 새로운 최적화 알고리즘을 실험용 브랜치에 적용한 뒤, 이를 태그로 명시하거나 릴리즈 노트를 작성해두면, 코드가 아닌 문서 수준에서 변경의 맥락을 요약해둘 수 있다. 이후에 해당 알고리즘만 다시 필요한 경우에는 해당 태그나 릴리즈를 바로 체크아웃(checkout)하여 복원하면 된다.

실험적 브랜치가 다수 존재할 때는 브랜치 전환과 병합이 잦으므로, 잘못된 시점의 브랜치를 병합하거나, 소스 충돌이 발생했을 때 재빨리 문제 해결을 해야 한다. 이 상황에서 Git의 rebase, cherry-pick과 같은 기능을 적극 활용하면, 겹치는 내용이 있는 다른 브랜치에서 특정 커밋만 추출해오거나 히스토리를 단순화해 깔끔하게 병합할 수 있다. 이때 주의해야 할 점은, rebase를 무분별하게 사용하면 원본 커밋의 해시가 바뀌면서 협업자의 작업 이력과 충돌할 위험이 크다는 것이다. 따라서 팀 내부에서 “공유된 브랜치를 rebase하지 않는다”와 같은 규칙을 미리 설정해두고, 이 규칙을 문서화하여 준수해야 한다. 이는 작업자가 많을수록 필수적인 약속이 된다.

PCL 프로젝트를 진행하다 보면, 포인트클라우드 데이터를 버전 관리하려는 시도가 생길 수 있다. 그러나 포인트클라우드 파일은 대체로 용량이 크고, 작은 수정 사항에도 전체 파일이 변경되기 쉬우며, Git 리포지토리에 직접 포함하기엔 부적합한 경우가 많다. 이럴 때는 Git LFS(Git Large File Storage)나 별도 데이터 전용 서버를 고려할 필요가 있다. Git LFS를 통해 큰 파일을 효율적으로 관리하면, 실제 소스 파일과 분리된 상태로 보관해 네트워크 전송 부담을 줄이고, 필요할 때만 다운로드 받도록 할 수 있다. 다만 Git LFS를 사용하기 위해서는 협업 환경(예: GitHub, GitLab 등)이 Git LFS를 지원해야 하며, 팀원 모두가 이를 설정해야 하는 추가 절차가 따른다.

포인트클라우드 데이터가 아니라도, 예컨대 PCL에서 시각화에 필요한 영상 파일들, 테스트용 환경 파일들(예: .pcd 또는 .ply 형태의 예제 데이터)이 수십 MB 이상일 경우 Git LFS가 권장된다. 이렇게 하면 실제 코드와 데이터가 분리되어 관리되고, 협업 시에 굳이 필요 없는 자료를 모두 받지 않아도 되므로 작업 효율이 올라간다.

Branch 구조 예시와 가이드라인¶

프로젝트 규모가 커질수록 여러 브랜치를 어떻게 구성하느냐가 관건이다. 가장 일반적으로 많이 사용되는 Git-Flow 방식에서는 main 브랜치와 develop 브랜치를 중심으로, 기능별(feature) 브랜치를 develop에 병합하고, 릴리즈 시점에는 release 브랜치를 만들어 최종 점검 후 main에 병합하는 식으로 관리한다. 그러나 Git-Flow가 다소 복잡하다고 느끼는 팀이라면 main 브랜치를 중심으로, 안정 버전만 main에 유지하고, 그 외 기능 구현이나 실험은 topic 브랜치에 분산시키는 간소화된 모델을 도입하기도 한다.

PCL에서 특정 알고리즘 모듈을 개선하거나 교체할 때, 또는 GPU/CUDA 가속 기능을 추가할 때, 독립적인 브랜치를 만들어 실험하고, 어느 정도 검증이 끝난 시점에 main 혹은 develop 브랜치로 병합하는 전략이 선호된다. 예컨대 ICP 기반 정합 알고리즘을 GPU로 가속화하는 기능을 개발할 때에는 feature/gpu_icp라는 브랜치를 만들어 실험하고, 성능 테스트 결과가 만족스러운 수준이 된 후에 병합 과정을 거친다. 이때 병합 시점에는 코드리뷰, 자동 빌드, 테스트 자동화 등을 통해 충돌이나 성능 저하 여부를 점검하는 것이 좋다.

버그 수정 역시 별도의 hotfix 브랜치를 사용해 main 또는 release 브랜치에서 파생시켜 빠르게 수정하고, develop 브랜치로도 병합해두어야 한다. PCL 프로젝트는 대규모 데이터와 연산이 얽혀 있기에, 수치 오류나 오버플로(overflow), 누적 오차 같은 버그가 특정 하드웨어, 특정 컴파일 옵션에서만 발생하기도 한다. 따라서 다양한 환경에서의 테스트가 필수적이고, hotfix 브랜치가 생길 때마다 CI/CD 환경에서 자동으로 여러 OS와 여러 컴파일러, 다양한 PCL 버전에 대한 호환성을 점검하도록 구성해두면 좋다. 이는 프로젝트 안정성과 함께 팀 전반의 생산성을 높이는 핵심 포인트다.

Submodule과 Monorepo vs. Multirepo¶

PCL 프로젝트가 복잡해지다 보면 여러 개의 Git 리포지토리로 분리할지, 아니면 하나의 대형 리포지토리(Monorepo)로 합칠지 결정해야 하는 상황이 온다. Monorepo 구조란, 프로젝트 관련 모든 소스, 라이브러리, 도구, 스크립트 등을 하나의 리포지토리에 포함하는 방식이다. 반대로 Multirepo 구조는 기능별 또는 서브 프로젝트별로 독립 리포지토리를 운용하는 방식이다. PCL 프로젝트에서 다양한 시각화 모듈, 데이터 전처리 모듈, 센서 데이터 수집 모듈 등을 각각 별도 리포지토리로 관리할 수도 있고, 하나의 리포지토리에 통합할 수도 있다.

Submodule 기능은 Multirepo 접근법을 지원하는 대표적인 Git 기능으로, 특정 리포지토리를 다른 리포지토리 내부에 하위 모듈처럼 포함시켜버리는 방식을 제공한다. 예컨대 “PCL-Algorithms”라는 리포지토리를 주 프로젝트에 submodule로 연결해두면, 주 프로젝트를 클론(clone)할 때 submodule도 함께 가져올 수 있다. 이때 submodule은 독립적인 Git 리포지토리이므로, 해당 알고리즘 부분만 따로 버전 관리를 진행할 수도 있다. 변경 사항이 생기면 주 프로젝트 리포지토리에는 해당 submodule의 특정 커밋 해시를 참조하는 형태로 업데이트된다.

이는 PCL 라이브러리 내 특정 알고리즘을 크게 수정하거나 독립적인 로드맵이 필요한 경우, 혹은 모듈별로 협력 팀이 전혀 다를 때 유용하다. 예컨대 센서 인터페이스 모듈은 하드웨어 팀이 전담하고, 알고리즘 모듈은 소프트웨어 팀이 전담할 경우, 각각의 리포지토리를 submodule로 관리해두는 방식이 프로젝트 전반의 혼선을 줄인다. 반면 Monorepo 방식은 코드 검색과 빌드 환경 통합 측면에서 편리하며, 버전 태그를 통해 전체 모듈의 버전을 일괄 관리하기 쉬운 이점이 있다. 다만 규모가 커질수록 리포지토리의 용량이 커지고, 변경 이력도 방대해지기 때문에, 한 번의 clone이나 pull/push 연산에 시간이 많이 걸릴 수 있다.

모듈화 아키텍처 설계¶

Git으로 버전을 관리하더라도, 프로젝트 자체가 모듈화되어 있지 않다면 협업과 유지보수 측면에서 한계가 뚜렷하다. PCL 기반 프로젝트에서 모듈화를 설계할 때는 다음과 같은 측면을 고려해야 한다. 우선, 수집 모듈(센서 인터페이스, 데이터 파일 로딩 등), 전처리 모듈(필터링, 다운샘플링, 잡음 제거, 좌표 변환 등), 본 처리 모듈(특징 추출, 세그멘테이션, 분류, 정합 등), 시각화 및 결과 출력 모듈(3D 뷰어, 로그 출력, 파일 저장 등)을 논리적으로 구분해야 한다. 이때 각 모듈이 가져야 할 의존성 관계를 최소화함으로써, 한 모듈에서 발생한 변경이 다른 모듈에 미치는 영향을 줄일 수 있다.

예컨대 전처리 과정에서 Downsample 필터를 교체하거나 파라미터 튜닝을 해도, 후속 알고리즘 모듈에서는 동일한 인터페이스를 통해 결과를 받아 처리하도록 설계해두면, 전처리 모듈을 교체해도 다른 모듈들은 빌드 오류나 API 호환 문제를 최소화할 수 있다. 이런 구조를 코드 상에서 좀 더 명확히 드러내려면, 각 모듈마다 별도의 네임스페이스를 두거나(예: namespace preproc, namespace alg, namespace vis 등), 디렉터리 구조를 일관성 있게 분리해두는 것이 좋다. 이렇게 나누어진 모듈을 각각 독립적인 CMake 타깃으로 구성해두는 것도 권장된다. 필요하다면 아래와 같이 CMakeLists.txt를 구성해볼 수 있다.

cmake_minimum_required(VERSION 3.10)
project(PCL_Modular_Project)

# Find PCL
find_package(PCL 1.10 REQUIRED)

# 모듈별 디렉터리 추가
add_subdirectory(sensor_interface)
add_subdirectory(preprocessing)
add_subdirectory(algorithms)
add_subdirectory(visualization)

이렇게 분리된 구조에서 각 모듈 내부의 CMakeLists.txt도 독립적으로 관리할 수 있으며, Git submodule을 사용할 경우에는 sensor_interface, preprocessing 등 디렉터리가 각각 별도 리포지토리가 되어 있을 수도 있다. 변경 내역이나 추가 파일 등이 충돌 없이 분산 관리되므로 팀 구성원 간 작업 충돌을 최소화할 수 있다.

코드 계층이 서로 교차되지 않도록 하는 것이 중요하다. PCL에서 제공하는 헤더 파일 구조나 링크 라이브러리는 모든 모듈이 공통으로 참조할 수도 있지만, 각 모듈의 헤더를 지나치게 여기저기서 임포트하면, 종속성이 불필요하게 복잡해진다. 따라서 모듈 간 인터페이스를 간결하게 정의하고, 데이터 교환 형식 역시 표준화해야 한다. 예컨대 내부에서 Eigen 매트릭스를 사용하더라도, 모듈 간에는 \(\mathrm{std::vector}\) 형태의 포인트 배열이나 PCL 포맷 자체를 사용하는 식으로, 서로 약속된 포맷을 유지하는 편이 구현 상 안전하다.

테스트 자동화와 통합 전략¶

프로젝트가 모듈화되고, Git 브랜치 전략이 잘 구비되어 있다면, 마지막으로 중요한 것은 자동화된 테스트와 통합 전략이다. 포인트클라우드는 데이터 양이 방대하고, 출력 결과가 수치적으로만 검증되기 어려우며, 시각화 측면에서도 품질을 점검해야 할 때가 많다. 따라서 정성적 평가(visual check)와 정량적 평가(메트릭 비교)를 모두 자동화할 수 있는 테스트 스크립트를 작성해두는 것이 이상적이다.

예를 들어, PCL의 ICP 알고리즘 결과를 평가하려면, 정합 이후에 두 포인트클라우드의 최소 RMS 에러를 측정하거나, 몇 프레임에 걸쳐 누적된 변환 오차를 계산해볼 수 있다. 이를 위해 테스트 모듈에서 임의의 변환이 적용된 두 포인트클라우드를 준비하고, ICP로 정합한 뒤, 결과 변환 행렬 \(\mathbf{T}\)가 실제 변환 행렬 \(\mathbf{T}_{\mathrm{true}}\)에 얼마나 근접한지 비교하는 방식으로 스크립트를 작성한다. 예컨대 변환 행렬 차이를 다음과 같이 비교할 수 있다.

\[ \Delta = \| \mathbf{T} - \mathbf{T}_{\mathrm{true}} \|_{F} \]

여기서 \(| \cdot |_{F}\)는 Frobenius 노름을 의미한다. 이 값이 임계값 이하이면 테스트 통과, 초과하면 실패로 처리할 수 있다. 해당 테스트를 CI 스크립트 안에 포함시켜, pull request가 올라올 때마다 자동으로 테스트가 수행되도록 구성하면, 팀원들이 코드를 병합하기 전에 오류나 성능 퇴보를 즉시 파악할 수 있다.

테스트 커버리지를 높이기 위해서는, 단순한 결과 값 비교뿐 아니라 예외 상황(예: 포인트가 매우 적은 경우, 노이즈가 극단적으로 큰 경우, 센서가 서로 다른 좌표계를 사용하는 경우 등)도 테스트 케이스로 넣어야 한다. 또한 GPU 기반 알고리즘을 추가했을 때는 GPU가 없는 환경에서의 동작 여부나 CPU fallback 로직이 동작하는지도 검사해야 한다. 이를 자동화한 코드는 다음과 같이 작성될 수 있다.

#!/usr/bin/env bash

echo "Running ICP test..."

# Input data 준비
./generate_test_data.py --type=icp --noise-level=medium

# ICP 알고리즘 실행
./build/icp_test --input1=cloud1.pcd --input2=cloud2.pcd --output=transform.txt

# 결과 검증
python3 verify_icp.py --expected_transform=transform_gt.txt --actual_transform=transform.txt --threshold=0.01

if [ $? -eq 0 ]; then
    echo "ICP test passed."
    exit 0
else
    echo "ICP test failed."
    exit 1
fi

이런 식으로 셸 스크립트 또는 CTest 등 CMake 기반 테스트 프레임워크를 사용할 수 있다. GitHub Actions, GitLab CI/CD, Jenkins 등 어떤 자동화 도구를 쓰든, pull request나 merge request 시점에 위와 같은 스크립트를 실행하게 하면, 문제 발생을 사전에 감지하고 협업 중 발생하는 버그를 줄이는 데 큰 도움을 준다.

컨테이너화(Docker)와 일관된 개발 환경¶

포인트클라우드 프로젝트에서는 개발 환경에 따라 컴파일러 버전, 라이브러리 의존성, GPU 드라이버 등 다양한 요소가 충돌을 일으킬 수 있다. 특히 대용량 데이터를 다루고, PCL 버전이 여러 단계에 걸쳐 바뀌는 프로젝트라면, 환경 설정이 조금만 달라져도 빌드가 실패하거나 성능이 크게 달라질 수 있다. 이 문제를 완화하기 위해 Docker 같은 컨테이너화 기술을 도입하면, 운영체제와 라이브러리 버전이 동일한 “이미지” 안에서 일괄적으로 개발 및 테스트를 진행할 수 있으므로 재현 가능성(reproducibility)이 크게 향상된다.

Docker 환경을 Git과 결합하여 사용할 때는, 일반적으로 리포지토리 내부에 Dockerfile을 두어, 해당 프로젝트를 빌드하고 실행하는 방법을 명시해둔다. 예컨대 아래와 같은 Dockerfile을 두면, Ubuntu 20.04 기반 컨테이너에서 PCL과 기타 의존성 라이브러리를 자동으로 설치하고, 프로젝트를 빌드할 수 있다.

FROM ubuntu:20.04

ENV DEBIAN_FRONTEND=noninteractive
RUN apt-get update && apt-get install -y \
    build-essential cmake git libpcl-dev \
    python3 python3-pip && \
    rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY . /app

RUN mkdir build && cd build && cmake .. && make -j$(nproc)

CMD ["bash"]

이 Dockerfile을 프로젝트 최상위 디렉터리에 두고, Git 리포지토리로 함께 관리한다면, 팀원 누구든지 “Dockerfile이 있는 상태” 그대로 동일한 환경을 재현할 수 있다. 예를 들어 클라우드 기반 CI/CD에서 “docker build” 명령을 수행하면, 리포지토리에 담긴 Dockerfile을 기준으로 이미지를 빌드하고, 이후 자동 빌드 및 테스트를 동일한 환경에서 돌릴 수 있다. 개발자의 로컬 머신에서도 “docker run”으로 컨테이너를 구동해, 호스트 환경과 상관없이 일관된 개발 환경에서 작업할 수 있다.

컨테이너 환경은 Git submodule 접근과도 잘 어울린다. 예컨대 알고리즘 모듈 submodule을 추가로 가져와야 한다면, docker build 시점에 “git submodule update --init --recursive” 명령을 실행하는 스크립트를 Dockerfile 안에 삽입할 수 있다. 이렇게 하면, 컨테이너 빌드 과정에서 자동으로 하위 모듈들이 체크아웃되고, 의존성 설치와 빌드도 연쇄적으로 이루어진다. 결국 팀 전체가 “docker build”만 수행하면, 동일한 PCL 버전과 동일한 submodule 커밋 상태에서 코드를 테스트하고 개발할 수 있게 된다.

코드 커버리지(Code Coverage)와 품질 관리¶

모듈화가 잘 된 PCL 프로젝트라 해도, 얼마나 많은 코드가 테스트되고 있는지는 별도의 분석이 필요하다. 수치 연산이나 3D 처리 로직이 많을수록, 어떤 함수와 클래스가 테스트에서 전혀 다뤄지지 않는지 쉽게 파악이 안 될 수 있다. 따라서 Git 기반 CI/CD에 코드 커버리지 도구를 연결하여, 테스트가 실제로 어느 부분까지 커버하는지 정량화하는 전략이 중요하다. 예컨대 GCC의 gcov와 lcov, Clang의 llvm-cov 등을 활용할 수 있고, CMake에서 BUILD_TESTING 옵션과 함께 빌드 옵션을 설정해두면 빌드 시점에 커버리지 정보를 수집할 수 있다.

커버리지 리포트를 생성한 뒤, 이를 HTML 형식으로 변환하여 CI/CD 아티팩트로 남기거나, SonarQube 같은 품질 분석 플랫폼과 연동해두면, 협업자들이 어느 부분이 테스트 사각지대인지를 한눈에 확인할 수 있다. PCL 알고리즘에는 분기(조건문)와 예외 처리 로직이 많은데, 커버리지 보고서를 통해 주요 분기나 예외 처리가 실제 테스트로 커버되는지, 누락된 부분은 어디인지 파악하고, 누락된 테스트 케이스를 추가하도록 유도할 수 있다. 예컨대 ICP 정합에서 예측 변환이 특정 범위를 벗어났을 때 에러 처리를 하는 분기문이 테스트되지 않았다는 사실이 커버리지 리포트에 드러난다면, 해당 부분을 중점적으로 테스트하여 기능 신뢰도를 높이게 된다.

커버리지를 과도하게 추구하면, 실제로 무의미한 코드도 억지로 테스트하는 경우가 생길 수 있다. 특히 PCL 라이브러리를 직접 수정하는 경우, 라이브러리 내부 구현까지 100% 커버리지를 달성하려 하기보다는, 프로젝트에서 사용하는 부분을 중심으로 커버리지를 평가하는 것이 현실적이다. 이는 CMake나 빌드 스크립트에서 특정 모듈에 대해서만 커버리지를 수집하도록 제한할 수도 있고, 서브 프로젝트별로 구분된 범위 내에서만 측정하게 할 수도 있다.

퍼포먼스 테스트와 벤치마크¶

포인트클라우드 처리에서는 정합 오차나 필터링 품질도 중요하지만, 성능 역시 프로젝트 성공에 큰 영향을 미친다. Git으로 효율적으로 협업을 진행하더라도, 성능이 어느 시점에서 갑자기 떨어지거나, 특정 병합 후 GPU 버전의 처리 속도가 급격히 낮아지는 문제가 발생할 수 있다. 이를 방지하기 위해서는 정기적으로 벤치마크를 수행하고, 결과를 저장·분석할 체계가 필요하다.

벤치마크는 단순히 실행 시간을 측정하는 스크립트부터 시작할 수 있다. 예컨대 필터링 모듈이나 세그멘테이션 모듈에 대해 표준 데이터셋을 준비해둔 뒤, 각각의 처리 시간을 측정하여 로그를 남기는 것이다. 이런 스크립트를 CI/CD에 포함해두면, pull request 시점에 해당 변경 사항이 성능을 얼마나 바꾸는지 비교할 수 있다. 성능 결과를 전후 대비해 “10% 이상 성능이 하락하면 병합 전 리뷰가 필요하다” 같은 정책을 세울 수도 있다.

조금 더 고도화하려면, Google Benchmark 또는 Catch2 등의 프레임워크를 사용해 세밀한 단위 벤치마크를 작성할 수 있다. 예컨대 ICP 한 번의 iteration에 걸리는 평균 시간, KD-트리 검색 수행 시간, RANSAC 내적 반복 횟수 등의 통계를 수집하고, 이를 그래프로 시각화해두면, 특정 브랜치에서 알고리즘을 개선했을 때 실제 성능 개선폭이 어느 정도인지 바로 확인할 수 있다. 브랜치 병합 시점마다 이 벤치마크를 자동으로 돌리고, GitHub Actions나 GitLab CI/CD에서 “성능 보고서”를 남기게 하는 방식을 쓴다면, 프로젝트 규모가 커져도 성능 추이를 놓치지 않고 관리할 수 있다.

성능 테스트를 할 때는 하드웨어 스펙과 드라이버 버전도 고정시키거나, Docker 같은 컨테이너로 환경을 통일하는 것이 권장된다. GPU를 사용하는 경우, 드라이버나 CUDA 버전 변화가 성능에 직접적으로 영향을 미치므로, 벤치마크 결과를 신뢰하기 위해서는 동일한 컨테이너 기반에서 측정해야 한다. 아울러 대규모 데이터셋에 대한 테스트도 필수인데, 이때 Git 리포지토리에 대용량 데이터를 포함시키는 대신, 별도의 데이터 서버나 S3 같은 스토리지를 참조해 필요할 때만 다운로드하는 방식을 고려하는 것이 효율적이다.

고급 Git 기법: Partial Clone, Sparse Checkout¶

포인트클라우드 프로젝트가 모노레포 형태로 점점 커지면, 리포지토리를 완전히 클론받는 데 걸리는 시간이 급증할 수 있다. 이 문제를 완화하려면 Git의 Partial Clone, Sparse Checkout 기능을 활용해 볼 수 있다. Partial Clone은 큰 바이너리 파일이나 특정 경로의 파일만 필요한 경우, 나머지 이력은 원격에서 필요할 때만 가져오는 방식으로 클론 속도를 높인다. Sparse Checkout은 리포지토리 내 특정 디렉터리만 체크아웃하여 사용 가능하도록 해준다.

예컨대 시각화 모듈만 수정하는 팀원이라면, 전처리 모듈이나 알고리즘 모듈의 소스는 당장 필요 없을 수 있다. 이럴 때 Sparse Checkout을 통해 시각화 관련 디렉터리만 가져오면, 불필요한 파일 다운로드 없이 빠른 작업이 가능하다. PCL 프로젝트에 대용량 테스트 파일이 포함되어 있어도, Sparse Checkout으로 필요한 부분만 얻을 수 있다. GitHub 등 호스팅 서비스에서도 Partial Clone을 지원하기 때문에, 리포지토리가 거대해진 상황에서 협업 속도를 개선하는 데 매우 유용하다.

Git Hooks와 자동화된 규칙 적용¶

Git Hooks는 커밋, 푸시, 병합 등의 이벤트 발생 시 특정 스크립트를 자동 실행하도록 설정하는 메커니즘이다. 예컨대 “pre-commit” 훅을 이용하면, 커밋 직전에 코드 포매터(예: clang-format)를 실행해 자동으로 스타일을 정리하거나, 린터(예: clang-tidy, cpplint 등)를 통해 기본적인 오류나 스타일 경고를 사전에 잡을 수 있다. “pre-push” 훅으로는 간단한 테스트를 돌려서 실패 시 푸시를 막아버리는 식의 정책도 가능하다. 이는 PCL 라이브러리의 API 오용이나 매직 넘버 사용 등 사소한 실수를 초기에 방지하는 역할을 한다.

특히 포인트클라우드 데이터의 좌표 변환, 단위(metric) 문제 등은 코드 리뷰로도 쉽게 놓칠 수 있으므로, 훅 스크립트에서 정적 분석 도구나 커스텀 스크립트를 실행해 “잘못된 단위 변환 호출 여부” 같은 특정 패턴을 검사하도록 만들 수도 있다. 이처럼 Git Hooks를 적절히 활용하면, 협업 과정에서 놓치기 쉬운 부분을 자동화로 보완할 수 있다. 다만 Git Hooks는 로컬 설정에 의존하므로, 팀원 모두가 Hooks를 동일하게 설정해야만 일관성이 보장된다. 이를 위해서는 사내 정책이나 지침을 만들어, 리포지토리에 Hooks 예시 스크립트를 넣어두고, 이를 심볼릭 링크나 설치 스크립트로 연결하도록 안내하는 식의 체계가 필요하다.

다음과 같은 pre-commit 훅 예시를 들 수 있다.

#!/bin/bash

echo "Running clang-format..."
clang-format -i $(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(cpp|h|hpp)$')

echo "Running static analysis..."
clang-tidy $(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(cpp|h|hpp)$') -- -I./include

# 만약 오류가 있으면 exit 1 처리
# ...

이처럼 pre-commit 훅으로 스타일 및 린팅을 강제하면, 결국 팀 전체의 코드 품질과 스타일이 일정하게 유지되어, 협업 시 충돌이나 불필요한 변경(diff) 생성이 줄어든다. Git Hooks는 로컬 단에서 실행되는 것이므로, CI/CD에서 동일 작업을 한 번 더 수행해 최종적으로 검증하는 방식을 추천한다. 로컬에서 검사를 통과해도, CI/CD 환경에서 다시 한 번 검사하여 정책 위반이 없는지 최종 확인하는 이중 체크 구조가 안전하다.

태그(Tag)와 릴리즈(Release) 전략¶

PCL 기반 프로젝트에서 버전을 구분해 관리하려면, Git 태그를 적절히 사용하는 것이 좋다. 예컨대 개발이 일정 단계에 이르러 “베타 버전” 수준으로 안정화되었을 때, 해당 커밋에 “v0.9.0-beta”와 같은 이름으로 태그를 달 수 있다. 이후 문제점이 개선되고, 새로운 기능이 추가되어 정식 릴리즈를 할 때는 “v1.0.0”처럼 명료한 태그를 부여하면 된다. 이렇게 태그를 달아두면 특정 시점의 소스 코드를 체크아웃해 재현하기 쉬워지고, 프로젝트를 배포하거나 다른 팀에 전달할 때도 버전 호환성을 확실하게 알릴 수 있다.

상용 소프트웨어나 규모 있는 오픈소스 프로젝트에서는 “Semantic Versioning” 규칙이 널리 쓰이는데, \(\mathrm{major.minor.patch}\) 형태로 버전을 표현한다. 예컨대 “1.2.5”에서 “1”은 큰 틀을 변경하는 호환성 깨짐(이진 호환성, API 호환성 등)이 발생했을 때 올리고, “2”는 호환성을 깨지 않는 신기능 추가나 변경이 있을 때, “5”는 간단한 버그 수정이나 마이너 수정 시 올린다. 이런 방식을 PCL 프로젝트에 적용하면, 예컨대 ICP 정합 알고리즘의 핵심 API 시그니처가 바뀌는 대형 변경은 major 버전을 올리고, 세부 알고리즘 파라미터 추가처럼 호환성은 유지되지만 기능이 늘어난 경우 minor를 올린다. 오탈자 수정이나 성능 개선처럼 사용자 입장에서 인터페이스가 바뀌지 않는 경우는 patch를 올린다. 이런 명확한 규칙이 적용되어 있으면, 협업자가 “v2.3.1 -> v2.4.0”으로 넘어갔다는 사실만 보고도 “기능이 추가되었구나. 기존 API가 깨지진 않았겠군.” 하고 예측할 수 있다.

GitHub나 GitLab 등에서 릴리즈 기능을 사용하면, 태그와 함께 릴리즈 노트를 작성할 수 있고, 빌드 산출물(바이너리, 라이브러리 등)을 첨부해 다른 개발자가 편리하게 다운로드하도록 제공할 수 있다. PCL 프로젝트에서 특정 운영체제나 컴파일러 버전에 맞춰 빌드된 결과물을 릴리스 아티팩트로 등록해두면, 협업자가 소스 빌드 과정 없이 해당 바이너리를 받아 바로 실행해볼 수 있다. 이런 식으로 릴리즈를 배포해두면, 새로운 팀원이 참여하거나, 테스트 환경이 한정된 상태에서 빠르게 결과를 재현해야 할 때 유용하다. 다만 아티팩트가 대용량인 경우에는 릴리즈 페이지가 부담스러워질 수 있으므로, 별도의 아티팩트 저장소나 CDN을 사용하는 것도 방법이다.

커밋 메시지 정책과 코드 리뷰¶

Git 협업에서 커밋 메시지를 일관성 있게 유지하면, 히스토리 조회나 코드 추적이 훨씬 수월해진다. PCL 프로젝트처럼 알고리즘, 데이터 처리, 시각화 등 다양한 관점이 얽혀 있을 때는 커밋 메시지에 “어떤 모듈을 변경했고, 왜 변경했는가”를 짧지만 명확하게 써두는 것이 핵심이다. 예컨대 “[preprocessing] Downsample voxel 크기를 0.05에서 0.02로 조정”처럼, 모듈명이나 변경 요점을 대괄호나 콜론으로 구분해두면 협업자들이 한눈에 커밋 내용을 파악하기 좋다. 실제 변경량(diff)만 봐서는 파라미터가 바뀐 이유나 맥락을 알기 어려우므로, 커밋 메시지로 근거나 목적을 간결히 기술하는 습관이 중요하다.

코드 리뷰 문화도 Git를 통한 협업의 질을 높인다. Pull Request나 Merge Request를 통해 다른 팀원의 변경 사항을 리뷰할 때, 단순히 문법적 오류나 스타일 체크만 하기보다는, 해당 로직이 PCL의 동작 방식과 일관되는지, 수학적 오차 처리가 올바르게 되어 있는지, 모듈 간 의존성이 과하게 증가하지는 않았는지 등을 검토해주는 것이 이상적이다. 3D 포인트클라우드에서는 좌표 변환, 수치 안정성, 메모리 사용량 같은 부분이 예상치 못한 오류를 일으키기 쉬우므로, 리뷰 단계에서 팀원들이 서로 의견을 교환하고, 필요하면 추가로 테스트 케이스를 작성하거나 문서를 보강하도록 제안해야 한다.

코드 리뷰는 Git 플랫폼(예: GitHub, GitLab)의 GUI를 통해 진행할 수도 있고, Gerrit처럼 별도의 리뷰 툴을 사용할 수도 있다. 어떤 방식을 쓰든, 모듈별 전문 지식을 가진 리뷰어가 해당 부분을 책임감 있게 검수하도록 하는 것이 좋다. 예를 들어 시각화 모듈 수정이 있을 때는 그 모듈 담당자나 시각화에 경험이 많은 인원이 리뷰하면, 도메인 특화된 피드백을 줄 수 있다. 이런 전문 담당자가 배정되지 않으면, 실질적인 문제를 놓칠 위험이 있다.

Pull Request 템플릿과 체계화된 변경 절차¶

협업 과정에서 발생하는 혼선을 줄이려면, Pull Request나 Merge Request를 올릴 때 일정 양식에 맞춰 중요 정보를 기록하도록 하는 템플릿 방식을 도입할 수 있다. 예컨대 “변경 요약”, “실행 또는 테스트 방법”, “예상 영향 범위”, “관련 이슈 링크”, “추가 확인 사항” 등을 템플릿 형태로 두고, PR 작성 시 자동으로 해당 양식이 채워지도록 설정해두면, 리뷰어가 무엇을 봐야 하는지 쉽게 파악할 수 있다.

PCL 프로젝트에서, 예컨대 ICP 정합 파라미터 변경 Pull Request를 예로 들어보자. 템플릿에 따라, 변경 요약에 “ICP 최대 iteration 수를 50에서 100으로 늘림”, 테스트 방법에 “rosbag에서 가져온 라이다 데이터로 icp_test.launch 실행 후, 정합 결과를 rviz로 확인”, 예상 영향 범위에 “추가 연산 시간이 필요해 성능이 2~3% 떨어질 수 있음”, 관련 이슈 링크로 “#102: ICP 수렴 불안정 문제 보고됨” 등을 적어놓으면, 리뷰어가 바로 이 Pull Request가 어떤 목적을 가지고 있으며 어떤 영향을 줄지 이해하기 쉽다.

이런 템플릿은 GitHub에서 .github/pull_request_template.md 형식으로 두거나, GitLab에서는 Merge Request 템플릿을 프로젝트 설정에 추가해두면 된다. 팀이 합의한 “체계화된 변경 절차”를 문서화해두고, 모든 Pull Request가 이 절차를 따르게 하면, 프로젝트가 커지더라도 일관된 품질과 관리 체계를 유지하기 쉽다.

외부 라이브러리와 의존성 관리¶

PCL 프로젝트에서 PCL 자체 외에도 Eigen, Boost, VTK 등 다양한 라이브러리에 의존하게 되는데, 각각 버전이 다르거나 맞지 않는 경우 빌드 에러나 런타임 오류가 발생하기 쉽다. 이런 복잡성을 줄이기 위해 CMake의 FetchContent, ExternalProject, 혹은 Conan, vcpkg 같은 패키지 매니저를 사용하는 방법을 고려할 수 있다. 예컨대 Conan을 사용해 PCL과 필요한 라이브러리를 지정해두면, 개발 환경이 달라도 Conan이 알아서 의존 라이브러리를 올바른 버전으로 가져오고 빌드해주므로, Git 리포지토리 안에서 “어느 라이브러리 버전을 쓰는지”를 매번 수동으로 관리하지 않아도 된다.

다만 PCL은 이미 자체적으로 여러 라이브러리에 의존하기 때문에, 호환성 문제가 생길 수도 있다. 예컨대 특정 버전의 PCL이 Boost 1.70, VTK 8.2를 요구하는데, 다른 모듈은 Boost 1.73, VTK 9.0을 필요로 할 수 있다. 이런 충돌을 사전에 정리하기 위해서는 프로젝트 초기에 “지원할 라이브러리 버전 범위”를 정해두고, CI 스크립트에서도 해당 버전 조합만 테스트하도록 설정해두면 예측 가능성을 높일 수 있다. 여러 버전 조합을 동시에 지원하기는 쉽지 않으므로, 릴리즈마다 지원 버전을 단계적으로 업데이트하는 식으로 운영하는 편이 실무적이다.

의존성 관리를 Git submodule로 하는 경우도 있다. 예컨대 외부 라이브러리를 submodule 형태로 직접 소스 코드를 가져와 빌드하는 방법이다. 이는 인터넷 연결이 제한된 사내 환경에서 라이브러리를 외부에서 다운로드받기 어려울 때 사용할 수 있는 방법이지만, 리포지토리 자체가 거대해질 위험이 있고, 라이브러리 업데이트 시 관리 부담이 크다. 따라서 가능하다면 Conan이나 vcpkg 같은 패키지 매니저를 우선 고려하고, 내부 방화벽이나 접근 제한이 있는 환경이라면 사내에 별도 아티팩트 서버를 두어 패키지 매니저가 접근할 수 있도록 구성하는 것이 일반적이다.

문서화(Documentation)와 위키¶

Git를 통한 협업에서 놓치기 쉬운 부분은 문서화다. PCL 프로젝트가 모듈화되고, 여러 알고리즘 브랜치가 존재하며, 테스트 데이터와 벤치마크가 병행되는 상황이라면, “전체적인 구조가 어떻게 되어 있고, 각 모듈은 어떤 책임을 지며, 어느 브랜치에서 어떤 실험이 진행되고 있는지”를 팀원 모두가 알기 어렵다. 이를 보완하려면 Git 리포지토리와 연동된 위키나, docs 디렉터리에 Markdown 문서를 체계적으로 배치하는 전략을 쓸 수 있다.

예컨대 GitHub Wiki, GitLab Wiki, Confluence, 혹은 별도의 정적 사이트 생성기(예: MkDocs, Docusaurus)를 이용해 프로젝트의 구조, 빌드 방법, 모듈 설명, 실험 방법 등을 정리해두면, 새로 합류한 인원도 빠르게 온보딩할 수 있다. 모듈별로 사용된 수학 이론이나 주요 알고리즘 개념, 예제 코드 등을 문서화해두면, 코드를 직접 뜯어보지 않아도 대략적인 동작 방식을 알 수 있다. PCL의 경우, 예를 들어 NARF 특징 추출이나 FPFH 계산 로직 등을 수식과 함께 문서화해두면, 협업자가 해당 모듈을 파악하는 시간이 단축된다.

Git 리포지토리에 docs 디렉터리를 두고, .md 형식 문서를 저장해두며, CI/CD 파이프라인에서 이를 자동으로 HTML로 변환해 GitHub Pages나 GitLab Pages 등으로 배포할 수도 있다. 이렇게 하면 “docs” 브랜치나 “gh-pages” 브랜치에 문서만 모아두고, 실시간으로 웹 문서가 갱신되도록 유지할 수 있다. 코드 변경이 이루어질 때마다 관련 문서 역시 Pull Request로 동반 변경되게끔 하는 문화가 정착되면, “코드는 바뀌었는데 문서는 안 바뀐” 상황을 줄일 수 있다. 이는 포인트클라우드 데이터 처리 프로젝트에서 매우 중요하다. 알고리즘이나 파라미터가 시시각각 달라지는데, 문서가 이를 따라가지 못하면 실제 운영 환경에서 혼선이 생기기 쉽기 때문이다.

Fork 워크플로와 오픈소스 기여¶

내부 프로젝트가 아니라 오픈소스 성격으로 운영하는 경우, 협업자는 리포지토리를 포크(fork)한 뒤 Pull Request를 보내는 방식을 주로 사용한다. PCL 자체가 오픈소스이므로, 공식 PCL 리포지토리에 기여할 때도 이런 방식을 취한다. 오픈소스 커뮤니티에서는 주로 “fork -> clone -> branch -> commit -> push -> pull request” 순으로 진행하며, 리뷰나 CI 통과 후 main 브랜치에 병합되는 형태가 일반적이다. 내부 협업에서는 굳이 fork 방식을 쓰지 않고 한 리포지토리 내에서 branch를 생성해도 되지만, 외부 기여나 오픈소스 협업에서는 fork 방식이 표준처럼 굳어져 있다.

Fork 기반 협업에서도 모듈화 전략이 중요하다. 기여자가 특정 모듈만 수정하고 싶을 때, 해당 모듈 디렉터리에서만 작업을 진행하고, 다른 부분은 건드리지 않는 식으로 PR 범위를 한정하면 리뷰가 쉽고 충돌이 적다. 만약 프로젝트가 submodule로 분리되어 있다면, 기여자가 서브 프로젝트를 별도로 fork하고, 서브 프로젝트에 대한 PR을 만든 다음, 주 프로젝트 측에서 submodule 커밋 해시를 갱신하는 방식으로 이어질 수 있다. 이 과정은 조금 복잡해 보이지만, 모듈별로 협업 범위를 제한하고 책임 분리를 명확히 하는 이점이 있다.

오픈소스 프로젝트에서는 기여 가이드(contributing guide), 라이선스 정책, CLA(Contributor License Agreement) 같은 부분도 고려해야 한다. PCL 라이브러리의 ALv2(아파치 라이선스 2.0) 사용 여부 등과 호환되는지, 외부 기여자의 코드가 적절한 라이선스 하에 공유되는지 등을 관리해야 한다. 이는 Git 관리와 별개이지만, GitHub나 GitLab에서 Pull Request 시점에 자동으로 CLA 봇이나 라이선스 체크를 수행하도록 설정할 수도 있다. 이런 사전 절차가 없으면, 나중에 법적 문제나 라이선스 충돌이 발생할 수 있으므로 주의가 필요하다.

CI/CD에서의 배포 자동화¶

PCL 프로젝트가 일정 규모에 이르면, 코드를 단순히 빌드·테스트하는 것 외에도, 특정 브랜치나 태그가 업데이트될 때 자동으로 실행 파일, 라이브러리, Docker 이미지 등을 빌드해 배포하는 파이프라인이 필요해진다. 예컨대 GitHub Actions나 GitLab CI/CD에서 main 브랜치에 태그를 달면, 배포 스크립트가 작동해 Docker 이미지를 빌드하고, 특정 레지스트리에 push하거나, 바이너리를 압축해 릴리즈 아티팩트로 업로드하는 식의 자동화를 구성할 수 있다. 이런 자동화 파이프라인은 “소스 수정 -> CI 빌드 & 테스트 -> 승인 -> 배포”의 흐름을 안정적으로 유지하기 위한 핵심 도구다.

특히 PCL처럼 빌드 옵션과 의존성 설정이 복잡한 라이브러리 기반 프로젝트는, 동시다발적으로 여러 플랫폼(Windows, Ubuntu, macOS 등)에서 배포해야 할 때가 많다. 이를 수동으로 전부 빌드하고 업로드하면 시간이 많이 소요되는데, CI/CD에서 멀티 플랫폼 빌드를 병렬로 수행해주면, 모든 플랫폼의 배포용 바이너리를 한 번에 확보할 수 있다. 이렇게 생성된 결과물을 Git 태그나 릴리즈 노트와 연동해 배포하면, 협업자는 원하는 플랫폼 용 바이너리를 선택해 빠르게 설치하고 테스트할 수 있다.

PCL 프로젝트가 사내에서만 사용되는 경우라도, CI/CD를 통한 사내 패키지 레지스트리(예: Nexus, Artifactory)에 자동 업로드하는 식으로 관리해두면, 특정 버전의 라이브러리를 간편히 받아서 설치할 수 있으므로 협업 효율이 오른다. 예컨대 “pcl_project-1.0.0-ubuntu20.04.deb” 형태의 패키지를 만들어 두고, apt 저장소나 내부 서버에 올려두면, 팀원들이 apt-get install pcl_project 식으로 프로젝트를 설치할 수도 있다. 모듈화가 잘 되어 있고, 의존성도 패키지 매니저가 자동으로 처리해준다면, 새로 합류한 팀원이 복잡한 빌드를 거치지 않고도 프로젝트를 활용할 수 있다.

거대 데이터셋 핸들링과 확장 문제¶

버전 관리와 모듈화 전략을 아무리 잘 잡아도, 포인트클라우드 데이터 자체가 테라바이트 단위로 커지는 순간에는 Git만으로는 한계가 온다. Git LFS도 용량 제약이나 호스팅 비용 문제가 생길 수 있고, 협업자 중 일부는 매번 전체 데이터를 받을 필요가 없을 수 있다. 이럴 때는 별도의 대용량 데이터 호스팅 서비스를 사용하고, 코드 내에서는 데이터 링크나 다운로드 스크립트만 두고, 실제 데이터는 필요할 때만 받도록 하는 식으로 운영해야 한다.

예컨대 AWS S3, Google Cloud Storage, 사내 NAS 등을 활용해 포인트클라우드 파일을 관리하고, Git에는 샘플 데이터나 테스트에 필요한 소량만 포함해두는 식이다. CI/CD에서도 전체 대용량 데이터셋을 굳이 다운받을 필요가 없는 경우가 많으므로, 핵심 기능 테스트에 필요한 대표 샘플만 저장소나 LFS에 포함해두고, 나머지는 벤치마크 전용 파이프라인에서만 다운로드해 측정하게끔 설계할 수 있다. 이렇게 분산된 구조를 잘 문서화해두면, 협업자들이 혼동 없이 “어떤 데이터가 어디에 저장되어 있고, 어떻게 가져와야 하는지”를 파악할 수 있다.

이 과정에서 Git 서브트리나 submodule 대신, 데이터 관리용 별도 Git 리포지토리를 두는 방법도 있다. 예컨대 “pcl_data”라는 리포지토리를 만들어, 그 안에 LFS로 대용량 데이터를 관리하고, 실제 코드는 “pcl_project” 리포지토리에서 레퍼런스만 담아두는 식이다. 이때 두 리포지토리의 버전을 맞춰서 태그를 달아두거나, docs 문서에서 “pcl_data v1.0.0과 pcl_project v1.0.0이 호환된다”는 식으로 명시하면, 데이터와 코드의 동기화가 깨지지 않는다. 물론 이 방식도 완벽하지 않으므로, 팀 내부에서 명확한 협의와 지침을 마련해야 한다.

유지보수와 장기 브랜치 운영¶

PCL 기반 프로젝트가 프로덕션 단계로 들어가거나, 상용 장비에 탑재되는 등 장기 유지보수가 필요해질 때는, 오래된 버전에 대한 패치 브랜치를 어떻게 운영할지도 고민해야 한다. 예컨대 “release-1.0” 브랜치를 만들어, 프로덕션 환경에서는 이 브랜치를 사용하도록 고정한 뒤, 긴급한 버그가 발견될 때마다 “hotfix”를 적용해 릴리스 버전을 갱신하는 식이다. 이때 develop 브랜치에서는 새로운 기능 개발과 버전 2.0, 3.0 등에 대한 준비가 계속되지만, release-1.0 브랜치에는 호환성을 깨는 변경이 포함되지 않는다.

Git-Flow와 같은 모델에서는 release 브랜치와 hotfix 브랜치의 개념이 이미 정의되어 있지만, 실제 운영 중에는 브랜치가 너무 많아져 혼란을 줄 수 있다. 팀 규모가 작다면, main 브랜치를 안정 버전으로 삼고, 새로운 기능은 feature 브랜치에서만 개발하다가 충분히 안정화된 후 main에 합치는 간단한 모델로도 충분할 수 있다. 중요한 것은 “프로덕션 환경에서 사용하는 브랜치를 누구나 마음대로 건드릴 수 없게 한다”는 점이다. 즉, Pull Request나 Merge Request를 통해 승인된 변경만 들어가도록 권한 설정을 해두고, 태그를 달 때만 빌드·테스트 후 정식 배포가 이루어지게 만들면, 장기 운영 시에도 예측 가능하게 안정성을 유지할 수 있다.

현실적으로, PCL은 버전마다 API 호환성이 약간씩 달라지고, 사용 중인 GPU나 OS 버전에 따라 동작이 달라질 수 있다. 따라서 장기 브랜치를 운용하면서도, 특정 고객(또는 현장)이 요구하는 커스텀 패치가 필요할 수도 있다. 이 경우 별도의 고객 전용 브랜치를 만들고, 해당 브랜치에서만 유지보수 작업을 진행할 수 있다. 이렇게 커스텀 분기를 많이 늘리면 관리 복잡도가 커지므로, 가능한 한 메인 브랜치와 동기화하거나, 공용 코드 베이스 위에 최소한의 차이만 두도록 정리해야 한다.

향후 확장 고려¶

PCL 프로젝트가 머신러닝, SLAM, 로보틱스 등 다양한 기술과 접목되면, Git 리포지토리 안에서 텐서플로(TensorFlow)나 파이토치(PyTorch) 같은 프레임워크가 추가로 필요해지고, 데이터가 2D 이미지, 비디오, 라이다·레이다 시퀀스 등으로 확장될 수도 있다. 이때 모듈화 전략이 부실하면, 한두 명이 전체 코드를 모두 이해해야 하거나, 특정 부분이 변경될 때 전체가 재빌드되는 일이 빈번해진다. 반면 모듈 간 독립성이 잘 보장되어 있고, CI/CD에서 병렬 빌드와 테스트가 분리되어 있다면, 팀원들은 자신이 담당하는 영역에 집중하면서도 전체 프로젝트와 원활히 맞물려 동작할 수 있다.

장기적으로는 “각 모듈의 기술 스택과 의존성을 어떻게 분리하고, 어떤 공통 인터페이스로 상호작용하게 만들 것인가”가 프로젝트 유지보수와 확장의 핵심 과제가 된다. 포인트클라우드 처리와 딥러닝 파이프라인이 결합될 경우, 전처리된 데이터 형식을 서로 공유하는 규약을 정하거나, Rosbag 같은 미들웨어 또는 Flatbuffers·Protobuf 같은 직렬화 포맷을 채택해 모듈 간 데이터 교환을 체계화할 수도 있다. 이런 결정들이 Git 상에서 버전 추적이 잘되도록 설계하고, 문서화 및 자동화된 테스트와 연계하는 것이 “버전 관리(Git)와 모듈화 전략”의 궁극적 목표다.
