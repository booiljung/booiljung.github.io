Jira를 통한 이슈 추적 프로세스 구성¶

PCL 프로젝트에서 발생하는 다양한 작업은 단순히 코드를 작성하고 테스트하는 수준을 넘어, 기획 단계부터 분석, 구현, 검증, 배포까지 전 과정을 체계적으로 추적해야 효율적이다. Jira는 이러한 요구에 부합하는 강력한 이슈 트래킹 시스템을 제공한다. 먼저 프로젝트에 맞춰 커스텀 이슈 타입을 정의한다. 예를 들어 포인트클라우드 데이터를 전처리하는 작업, 알고리즘을 최적화하는 작업, 또는 특정 하드웨어 환경에서의 호환성 검증 작업 등을 개별 이슈로 구분할 수 있다. 각 이슈에 우선순위, 담당자, 예상 소요 시간, 진행 상태 등을 할당해 개발 진행 과정을 한눈에 파악한다.

워크플로우는 요구 사항 정의 → 구현 → 리뷰 → 최종 승인 → 배포와 같이, 프로젝트 팀의 실제 업무 단계를 반영해 구성한다. Jira에서 워크플로우를 세밀하게 설정하면 특정 단계에서 리뷰어에게 자동으로 알림이 가거나, 특정 이슈가 다른 이슈를 해결해야만 진행 상태가 변경될 수 있도록 제약을 두는 것도 가능하다. 이때 워크플로우 변경이 코드 저장소와도 긴밀히 연결되도록 Git이나 SVN과 연동하여, 이슈 상태가 변경될 때 자동으로 커밋 메시지나 브랜치가 매칭되도록 설정할 수 있다.

PCL 코드를 작성할 때 발생하는 수많은 실험 데이터 관리 역시 중요한 이슈다. Jira 이슈에 데이터를 첨부하거나, Confluence 문서와 연결해 실험 결과를 시각화된 그래프로 남기면, 이후 유사한 문제가 발생했을 때 참고 자료로 쉽게 찾을 수 있다. 이런 방식으로 Jira를 사용하는 것은 단순한 ‘할 일 목록’ 이상이 된다. 예를 들어, 포인트클라우드 필터링 기능을 향상시키기 위해 진행 중인 이슈가 있다면, 각 이슈에 해당하는 알고리즘의 \(\mathrm{O}(n \log n)\) 혹은 \(\mathrm{O}(n^2)\) 시간 복잡도를 분석한 결과를 첨부하거나, 특정 베이지안 필터(Bayesian Filter) 실험 과정을 이미지로 기록해둘 수 있다.

회사 내 다른 프로젝트와 연계할 때도 Jira 대시보드를 활용한다. 포인트클라우드 데이터 처리 파이프라인을 구성하는 하위 모듈들이 여러 개의 다른 팀(예: 센서 데이터 수집, 로봇 제어 등)과 연동되어야 한다면, Jira에서 여러 프로젝트 간의 의존관계를 설정해 통합 대시보드를 만든다. 이 대시보드를 통해 각 팀이 공유 자원을 어디까지 활용하고 있는지, 문제가 발생하면 어떤 이슈가 병목이 되는지 바로 파악한다. 일정 지연이 예측될 경우 일정 조정이나 리소스 재분배를 사전에 계획할 수 있기 때문에 업무 병목 현상을 최소화할 수 있다.

Confluence를 활용한 문서화 및 지식 관리¶

PCL 프로젝트에서 이루어지는 모든 작업 내용, 예컨대 메서드별 파라미터 설정 근거, 하드웨어별 호환성 이슈, 레이저 스캐너와 리다(라이더) 센서를 비교한 결과 등을 Confluence에 축적하면 팀원 간의 정보 격차를 줄일 수 있다. Confluence는 단순한 위키 수준을 넘어, Jira 이슈와 상호 참조가 가능하고 여러 플러그인을 통해 추가 기능을 확장할 수 있다. 예를 들어, “PCL 필터링 알고리즘 성능 분석”이라는 Confluence 페이지를 생성하고, 해당 페이지에서 Jira 이슈를 인용하면 Confluence에서 실시간으로 이슈 상태를 확인하게 된다.

포인트클라우드 시각화 결과를 공유하기 위해 3D 렌더링 이미지를 업로드하거나, 동영상 파일을 첨부해 실험 장면을 아카이빙할 수도 있다. 팀원들은 이러한 자료를 통해 단순히 텍스트로 설명된 자료보다 빠르고 직관적으로 내용을 파악한다. 더 나아가 Confluence 내부에서 매크로를 사용해 수식을 표현하거나, mermaid를 이용해 알고리즘 흐름도나 데이터 처리 파이프라인 다이어그램을 실시간으로 업데이트한다. 예컨대, 실험 설계 다이어그램을 작성할 때:

flowchart LR SensorData --> PreProcessing PreProcessing --> PCLFiltering PCLFiltering --> MLModel MLModel --> Visualization

이런 식으로 구성하면 프로젝트 이해도가 높은 팀원뿐 아니라, 신규 합류자가 봐도 전체 파이프라인에서 특정 모듈이 어떤 역할을 담당하는지 한눈에 확인할 수 있다. Confluence의 버전 관리 기능을 통해, PCL 라이브러리 버전에 따라 알고리즘의 일부 파라미터가 바뀔 때마다 문서 변경 이력을 확인한다. 과거 버전을 살펴볼 때는 단순한 텍스트 비교가 아니라, 다이어그램 변경이나 수식 변경까지 추적해 어떤 단계에서 성능 개선이 이루어졌는지 이해하게 된다.

서버 환경에서 Confluence를 운영할 때는, 사내 접근 권한 정책에 유의해야 한다. 포인트클라우드 데이터 처리와 관련된 연구용 데이터는 용량이 큰 경우가 많다. 이 데이터를 모두 문서에 직접 첨부하기보다는 사내 NAS(Network Attached Storage)나 클라우드 저장소와 연동해 주요 데이터만 링크 형태로 연결하고, 문서에는 결과 요약 혹은 대표 이미지를 남기는 것이 효율적이다. 이때 권한 설정을 잘못하면 외부에 공개되어선 안 되는 데이터가 유출될 수 있으니, Confluence의 스페이스(Space)나 페이지 단위로 권한을 세분화해 운영한다.

Jira와 Confluence를 결합해 사용하는 가장 큰 이점 중 하나는, 프로젝트 요구 사항 변경이 발생했을 때 문서와 이슈가 분리되지 않고 항상 동기화된다는 것이다. 예를 들어, 새로운 센서가 추가되어 포인트클라우드 데이터의 스캔 해상도가 올라갔을 때, Jira 이슈를 새로 작성하거나 기존 이슈에 댓글을 달면서 Confluence 문서의 파이프라인 설명 섹션을 함께 업데이트한다. 이렇게 하면 새로운 하드웨어 구성으로 실험이 이루어져도, “이전에 작성된 매뉴얼은 더 이상 유효하지 않을 것 같은데?”라는 의문을 제기할 필요가 없다. Confluence 문서를 통해 지속해서 최신 상태를 유지하고, Jira를 통해 모든 관련 이슈가 유기적으로 연결되어 있기 때문이다.

Jira 자동화 기능과 CI/CD 파이프라인 연계¶

PCL 프로젝트처럼 포인트클라우드 데이터를 다루는 경우, 코드의 빌드나 테스트가 단순히 컴파일 과정에 그치지 않고 대규모 데이터셋을 사용한 알고리즘 검증 단계까지 이어지는 경우가 많다. 이러한 과정에서 인력 개입을 최소화하고, 작업 흐름을 자동화하기 위해 Jira 이슈와 CI/CD(Continuous Integration/Continuous Delivery) 파이프라인을 연동할 수 있다. 예를 들어, 개발 팀에서 사용하는 Jenkins 혹은 GitLab CI, GitHub Actions 등과 Jira를 연결해두면, 특정 브랜치에 커밋이 올라갈 때마다 자동으로 PCL 모듈 빌드와 단위 테스트, 그리고 대규모 포인트클라우드 데이터셋에 대한 벤치마크 테스트가 진행된다. 이후 테스트 결과나 로그가 특정 이슈에 첨부되거나, 이슈 상태가 자동으로 ‘테스트 중’으로 변경되어 팀원들이 실시간으로 파악할 수 있다.

대규모 포인트클라우드 데이터 처리에는 상당한 컴퓨팅 리소스가 요구된다. 클라우드 환경에서 GPU 노드나 HPC(High-Performance Computing) 클러스터를 이용해 병렬 처리를 수행할 때, 이러한 외부 시스템 호출 또한 Jira 이슈 워크플로우에 통합할 수 있다. 예컨대, \(\mathrm{build_id}=n\)의 테스트가 성공적으로 수행된 후 벤치마크 스크립트가 실행되도록 설정하고, 이 결과가 S3나 NAS에 저장되면 Jira 이슈에 해당 결과 링크를 자동으로 추가한다. 또한 벤치마크 스크립트 내에서 \(\mathrm{accuracy}=f(\mathrm{build_id}, \mathrm{config_params})\) 등의 형태로 측정된 정확도나 처리 시간을 로그로 남기면, Jira 대시보드 상에서 여러 빌드 버전과 파라미터 조합에 따른 성능 변화를 일목요연하게 시각화할 수 있다. 이 과정에서 Jira Automation 기능을 활용해, 특정 커스텀 필드(예: “알고리즘 버전”이나 “필터 파라미터”)가 변경될 때마다 자동으로 CI 스크립트를 실행하거나, 이슈를 특정 담당자에게 재할당하는 규칙을 설정하기도 한다.

포인트클라우드 처리 도중 발생하는 병목점을 찾기 위해서는 다양한 로그 데이터가 필요하다. 예컨대, 알고리즘의 병렬화 수준에 따른 CPU와 GPU 사용량, 파라미터 세팅에 따른 메모리 증가량 등을 모니터링해 Jira 이슈에 첨부하는 방식을 사용한다. 만약 특정 이슈가 높은 우선순위로 지정되어 있으면, CI/CD 파이프라인에서 그 이슈에 해당하는 작업을 먼저 수행하거나, 리소스를 더 많이 할당하도록 스크립트를 작성할 수 있다. 이렇게 Jira와 CI/CD 시스템이 긴밀히 연계되면, PCL 프로젝트의 최적화 작업이 반복 실행될 때마다 자동으로 성능 측정이 이뤄지고, 결과가 이슈에 누적되므로 프로젝트의 품질 관리가 체계적으로 이루어진다.

Confluence 고급 활용 사례¶

Confluence는 단순히 개발 문서를 작성하는 것뿐만 아니라, 연구 논문의 초안이나 특허 문서 작성 과정에도 적용할 수 있다. PCL을 활용해 특정 센싱 모듈과 결합한 연구를 진행하는 도중, 실험 결과를 바탕으로 논문 초안을 작성한다면, 각 섹션별로 Confluence 페이지를 구성해 공동 작업이 가능하다. 특정 섹션에 추가적인 실험 데이터나 그래프를 삽입하려면, Confluence의 “attachments” 기능을 통해 이미지나 CSV 파일을 업로드하고, 다이어그램이나 LaTeX 매크로를 사용해 실시간으로 해당 데이터를 시각화한다. 가령 로봇 팔 끝단 센서에서 측정된 포인트클라우드 데이터가 다양한 각도에서 캡처되어, 수만 개의 포인트로 구성된 경우, Confluence에 2D 투영 그래프나 부분 샘플링 이미지를 실시간으로 게시해 팀원들과 함께 분석한다.

문서의 버전 관리와 비교는 Confluence에서 매우 중요한데, PCL 라이브러리 업데이트 시점마다 성능상 변화를 문서에 기록해 두면, 팀원들이 과거 어느 시점의 알고리즘 버전에 대해 궁금해질 때 손쉽게 찾아볼 수 있다. Confluence는 문서 내용이 크게 변동될 때마다 해당 페이지의 버전을 저장하므로, 과거 버전과의 차이점을 시각적으로 비교할 수 있다. 파라미터가 어떤 형태로 변경되었는지, 코드 스니펫이 어느 부분에서 수정되었는지 등을 빠르게 추적하는 것이다. 라이다(LiDAR) 센서를 이용해 취득한 포인트클라우드 데이터와 카메라 영상과의 정합(Matching)을 예시로 들면, 버전별로 정합 알고리즘이 다를 수 있는데, Confluence 페이지에 그 알고리즘의 흐름도나 PCL 내장 함수 사용 예시 코드를 디스플레이 코드 블록으로 정리해 두면, 버전 간 차이를 쉽게 이해할 수 있다.

Confluence에서 Jira 이슈를 페이지에 직접 임베드하면, Jira에 새로 등록된 이슈나 진행 상태 변경 사항이 실시간으로 문서에 표시된다. 이 방식을 통해, “포인트클라우드 노이즈 제거 모듈 최적화”라는 주제의 Confluence 페이지가 있다면, 해당 페이지 내에 관련된 모든 Jira 이슈 목록이 자동으로 연동되어 표시된다. 팀원들은 어떤 이슈가 진행 중이고, 어떤 이슈가 해결되었으며, 남아 있는 이슈는 무엇인지 문서를 읽으면서 동시에 확인한다. 이는 정보가 산재되어 있지 않고 한 곳에서 중앙집중적으로 관리되어야 효율이 극대화되는 포인트클라우드 프로젝트에 특히 유용하다. 현재 작업 중인 알고리즘의 이론적 배경, 실험 결과, 앞으로 진행해야 할 작업 목록 모두를 하나의 페이지에서 살펴볼 수 있기 때문이다.

고급 통합 전략: Jira, Confluence, 그리고 외부 협업 플랫폼 간의 연동¶

포인트클라우드 처리 프로젝트는 다양한 이해관계자(개발 팀, 연구팀, 현장 운영팀, 심지어 외부 파트너사) 간에 지식이 유기적으로 교류되어야 한다. Jira와 Confluence가 사내 협업을 위한 두 축이라면, Slack, Microsoft Teams, 혹은 이메일 시스템 같은 외부 커뮤니케이션 툴과의 연동도 고려해야 한다. 이를테면 Jira 이슈가 특정 상태(예: “고우선순위 버그 발생”)에 도달하면 Slack 채널에 자동 알림을 띄우거나, Confluence 페이지가 변경될 때마다 팀원들에게 이메일 또는 Teams 메시지가 전송되도록 설정할 수 있다. 이러한 추가 알림 시스템은 포인트클라우드 데이터 처리에 특화된 긴급 상황—예컨대 센서 오작동으로 인한 대규모 데이터 손실, 실험 환경의 결함 등—이 발생했을 때 더욱 중요하다. 문제 상황을 빠르게 감지하고 팀 전체가 같은 정보를 공유할 수 있기 때문이다.

프로젝트 규모가 커지면, Jira 이슈나 Confluence 페이지의 수도 기하급수적으로 늘어난다. 그래서 Atlassian Marketplace 등을 통해 제공되는 고급 플러그인이나 애드온을 활용하는 사례도 많다. 예컨대, “ScriptRunner” 같은 애드온을 이용하면 Jira 워크플로우에서 스크립트 단위로 복잡한 로직을 추가할 수 있다. PCL 프로젝트 내에서, 특정 이슈가 해결된 뒤 자동으로 문서(Confluence 페이지)의 특정 섹션에 시간과 담당자, 변경 내용을 기록하거나, 이슈가 ‘배포 대기’ 상태가 되었을 때 Slack 채널에 사전 정의된 메시지를 보내도록 설정할 수 있다. ScriptRunner를 이용하면 단순한 트리거 수준을 넘어, 실제 코드 저장소와 연동해 브랜치 전략도 자동으로 생성할 수 있다. 예컨대 이슈 키가 “PCL-123”인 경우, 저장소 브랜치를 feature/PCL-123 형태로 자동으로 만들어 개발자의 실수를 줄이고 워크플로우 표준화를 강제한다.

Confluence에서도 “Scroll PDF Exporter”나 “Scroll Versions” 같은 플러그인을 적용해 대규모 문서를 효율적으로 관리한다. PCL 프로젝트가 주로 소스 코드와 실험 데이터를 중심으로 이뤄지긴 하지만, 때로는 고객사 대상의 매뉴얼 작성이나, 특허 출원을 위한 사양서를 작성해야 할 때가 있다. Confluence에서 “Scroll PDF Exporter”를 통해 해당 페이지나 스페이스 전체를 PDF 문서로 변환해 배포할 수 있으며, “Scroll Versions”로는 동일한 문서의 여러 버전을 동시에 관리해, 어느 팀은 “버전 1.0” 기준으로 업무를 진행하고 다른 팀은 “버전 2.0”을 준비 중일 때 협업 충돌을 최소화한다. 예를 들어, “PCL 필터링 알고리즘 가이드” 페이지를 버전별로 나눠서 관리하면, 1.0 버전에선 기존 필터링 함수를 중점적으로 설명하고, 2.0 버전에선 새롭게 추가된 고성능 필터 함수를 상세히 서술하는 식이다.

포인트클라우드 처리 프로젝트에서는 하드웨어 구성(센서, GPU, CPU, 네트워크 자원 등)이 다변화될 수 있다. Confluence에 “하드웨어 구성 템플릿”을 만들어두면, 새롭게 도입되는 하드웨어나 시스템 아키텍처 정보를 일관되게 문서화할 수 있다. 예컨대 템플릿에 “센서 스펙, 데이터 포맷, 최대 처리 용량, 네트워크 대역폭 요구사항, 사용 라이브러리 버전” 같은 항목을 미리 정의해두고, 각 하드웨어가 도입될 때마다 문서를 복제해 세부 내용을 채운다. 이 템플릿은 Jira 이슈 생성과도 연결되어, 하드웨어 도입 이슈가 만들어지면 자동으로 Confluence 템플릿 페이지를 생성하도록 설정할 수 있다. 이를 통해 사내 표준화된 형식에 따라 신규 하드웨어 정보를 빠짐없이 기록하게 되고, 전체 포인트클라우드 파이프라인에서 어떤 하드웨어가 어떤 기능을 담당하는지 쉽게 파악할 수 있다.

포인트클라우드 도메인 특성상, 대규모 3D 데이터나 시뮬레이션 결과를 다루기 위해 Unity, Unreal Engine, RVIZ, ROS(ROS2 포함) 등의 툴과도 연동하는 사례가 많다. Jira에 ‘시뮬레이션 수행’ 이슈가 생성되면, Confluence 페이지에서 시뮬레이션 결과(동영상, GIF, 혹은 3D 모델 스크린샷)를 간단히 검토한 후, Slack 채널로 해당 결과를 공유하고 의견을 수렴하는 방식이다. 일부 기업은 개발 팀 외에도 기획, 마케팅, 현장 운영 팀이 같은 Confluence 페이지를 열람해, 프로젝트 결과를 직관적으로 이해하고 피드백을 주도록 유도한다. 이렇게 Jira, Confluence, 외부 협업 툴이 모두 연결되어 있으면 의사소통 속도가 비약적으로 빨라진다.

포인트클라우드 데이터의 특성상, 각종 파일(PLY, PCAP, LAS 등)의 용량이 크고 형식도 제각각일 수 있다. 효율적인 파일 관리를 위해서는 Confluence나 Jira에 직접 첨부하기보다는 사내 파일 서버나 AWS S3, Git LFS(대용량 파일 스토리지)와 연동하는 방식을 고려한다. 특히 Git LFS는 포인트클라우드 데이터를 버전 관리할 때 매우 유용한데, Jira 이슈에 S3 링크나 Git LFS 링크를 삽입해두면, 실제 파일이 별도의 스토리지에 저장되므로 성능 저하 없이 편리하게 활용할 수 있다. 더 나아가 Confluence에서는 특정 데이터세트에 대한 간단한 MD5 해시나 SHA256 해시 값을 표로 정리해두면, 팀원들이 동일한 데이터셋을 사용하고 있는지 교차 검증하기 쉽다.

PCL 라이브러리 자체도 빈번한 업데이트가 이루어지므로, 협업 툴을 통해 버전별 호환성 정보를 효과적으로 관리하는 것이 핵심이다. Confluence에 “PCL 버전 호환성 매트릭스” 같은 테이블을 작성해 OS(Windows, Linux 등), 컴파일러(Visual Studio, GCC, Clang 등), CUDA 버전, Eigen 버전 등의 호환성 여부를 표로 정리하고, Jira 이슈를 통해 해당 호환성 테스트가 언제, 어떤 환경에서 이루어졌는지 기록한다. 예컨대:

PCL OS Compiler CUDA Eigen TestResult
1.11.1 Windows 10 MSVC 2019 11.2 3.3.9 Passed
1.11.1 Ubuntu 20.04 GCC 9.3 10.1 3.3.7 Failed
... ... ... ... ... ...

이 테이블을 Confluence에서 관리하고, 각 행에 Jira 이슈 링크를 첨부해두면, 호환성 테스트가 실패한 이유나 해결 과정, 사용한 포인트클라우드 샘플 등 세부 정보를 손쉽게 찾아볼 수 있다.

CI/CD와 관리 지표(KPI)¶

포인트클라우드 데이터 처리 프로젝트에서 Jira와 Confluence를 연동해 협업을 극대화하려면, CI/CD(지속적 통합/지속적 배포) 프로세스에 대한 정량적 관리 지표(KPI, Key Performance Indicators)도 함께 운용하는 편이 좋다. Jira에 등록된 이슈를 기준으로 작업 시간을 추적하거나, 빌드와 테스트의 성공/실패 로그를 분석해 개발 효율을 측정할 수 있다. 예컨대 다음과 같은 지표를 고려할 수 있다.

첫째, “리드 타임(Lead Time)”을 추적해, 이슈가 생성된 뒤 실제 작업이 완료되는 데까지 걸리는 시간을 정량적으로 파악한다. 포인트클라우드 필터링 기능을 개선하는 작업이 1주일 안에 끝날 것으로 예상했는데 실제로 2주 이상 걸렸다면, 데이터셋의 크기나 사내 리소스 할당 문제가 있었는지 Jira 이슈와 CI 로그를 함께 분석한다. 이슈에 첨부된 벤치마크 결과나 Confluence 문서의 실험 과정을 재검토해, 어느 단계에서 병목이 발생했는지 구체적으로 확인한다. 이를 통해 다음 스프린트에서 작업 분량을 예측할 때 더 정확한 계획을 세울 수 있다.

둘째, “MTTR(Mean Time To Repair/Resolve)”를 추적해, 발생한 버그나 긴급 이슈가 해결되는 속도를 개선한다. PCL 라이브러리가 업데이트된 뒤 특정 운영체제나 특정 GPU에서만 발생하는 오류가 있다면, Jira 이슈 생성부터 문제 해결까지 모든 과정을 Confluence 문서와 연동해 기록한다. 이렇게 누적된 데이터를 바탕으로, 차후 비슷한 오류가 발생하면 과거 해결 과정을 신속히 참고해 MTTR을 줄이는 것이 목표다.

셋째, “빌드 안정성 지표”를 도입해, CI 파이프라인에서 포인트클라우드 처리 알고리즘 테스트가 얼마나 자주 실패하는지를 정량화한다. 복잡한 알고리즘이 추가되거나 새로운 하드웨어 환경이 투입되는 시점에 빌드 실패율이 급격히 높아지면, 이는 해당 환경에 맞는 컨테이너 설정이 미흡하거나, PCL 라이브러리 의존성 문제가 발생했을 가능성이 높다. Jira의 이슈를 통해 문제 상황을 기록하고, Confluence 문서에 환경 설정 가이드를 업데이트함으로써 빌드 안정성을 높인다.

이처럼 지표를 활용하면, 협업 프로세스 전반에 대한 과학적 통제와 예측이 가능해진다. 단순히 “개발 일정이 늦어지고 있다”는 느낌적 판단이 아니라, 실제 지표 기반으로 개선할 영역을 파악하고 대응 방안을 마련한다. Jira의 스프린트 보드나 대시보드에 커스텀 위젯을 설정해 팀의 현재 KPI 수준을 시각화하거나, Confluence에 “프로젝트 지표 현황” 페이지를 만들어 주요 데이터를 주기적으로 업데이트하면, 팀 전체가 동일한 목표 지향성을 유지하면서 협업을 진행하게 된다.

권한 관리와 보안 이슈¶

포인트클라우드 데이터는 산업 현장에서 민감한 정보(예: 공장 내부 구조, 지도 제작 정보, 건설 현장 스캔 데이터 등)를 포함하는 경우가 많다. 따라서 Jira와 Confluence를 사용할 때 권한 구조를 엄격하게 설계해, 프로젝트별 또는 스페이스별로 접근 범위를 제한해야 한다. 예컨대, 사외 파트너사와 특정 페이지를 공유해야 하는 상황이라면, 전체 Confluence 스페이스가 아니라 특정 페이지 혹은 그 하위 트리 구조만 접근 가능하도록 한다. Jira 이슈 역시, 민감 정보가 포함된 이슈는 보고 권한을 제한하거나, 이슈 내 첨부 파일에 대한 접근만 제한하는 방식을 취한다.

또한, 포인트클라우드 처리 과정에서 생성되는 중간 결과물(예: 특정 지역의 지형 데이터, 물체 인식 결과, 로봇 주행 경로)은 잘못 노출될 경우 기업 자산 혹은 공공 안전과 직결될 수 있다. Confluence 문서를 대외적으로 공유해야 한다면, PDF나 Word 형태로 추출한 뒤 민감 정보를 마스킹하거나 해시 형태로만 남기도록 사내 보안 정책을 마련하는 것이 좋다. Jira의 협업을 통해 업무 효율을 높이더라도, 특정 국방이나 산업 기밀에 해당하는 정보는 별도 보안 워크플로우로 관리하도록 사전 계획을 세워야 한다. 예를 들어, Jira에서 ‘보안 이슈’ 유형을 별도로 두고, 기본 워크플로우와 다른 승인 단계를 추가해 보안 관리자를 통과해야만 이슈가 열람 가능하도록 설정한다.

On-premise 방식으로 Jira나 Confluence를 운영하는 기업은 사내 방화벽 규칙, LDAP/SAML 인증 등과 연동해 물리적·논리적 보안을 강화한다. 클라우드 기반 Atlassian 서비스를 사용한다면, VPN 접속이나 IP 화이트리스트를 활용해 외부 접근을 통제하고, 사용자 액세스 로그를 상시 모니터링한다. 포인트클라우드 프로젝트는 하드웨어적으로도 실험 장비와 센서가 여러 장소에 분산될 수 있기 때문에, 어디서나 안전하게 Jira와 Confluence에 접근해 정보를 관리하도록 네트워크 보안 체계를 점검해야 한다.

협업 문화의 정착과 효율 극대화¶

Jira, Confluence 같은 협업 툴을 아무리 정교하게 설계해도, 실제 팀원들이 이를 적극적으로 활용하지 않으면 실질적인 효과를 거두기 어렵다. 포인트클라우드 데이터 처리를 전문으로 하는 팀은 대부분 R&D 성격이 강해서, 개발자가 혼자 깃허브 리포지토리를 파고들어 문제를 해결하는 경우가 잦다. 이런 방식이 단기적으로는 편해 보여도, 장기적 관점에서 팀 지식이 개인에게만 축적되어 버리면 프로젝트 지속 가능성에 악영향을 미친다. 따라서 모든 팀원이 Jira를 통해 자신의 작업을 투명하게 공유하고, Confluence 문서에 최신 정보를 남기는 문화가 필수적이다.

이를 정착시키기 위해서는 신규 입사자의 온보딩 과정에서부터 Jira와 Confluence 사용 방법을 체계적으로 교육하고, PCL 라이브러리 및 내부 알고리즘에 대한 문서를 Confluence에서 직접 찾아볼 수 있도록 안내한다. 팀 차원에서 어떤 작업이라도 이슈를 먼저 생성하고, 리뷰와 승인 과정을 거쳐야 최종적으로 머지(Merge)되도록 워크플로우를 설정하면, 자연스럽게 협업 툴 중심으로 소통이 이루어진다. 그리고 이러한 환경에서 일어나는 모든 기술적, 지식적 자산은 Confluence 문서와 Jira 이슈로 쌓여, 시간이 지날수록 팀 레벨의 역량이 향상된다.

개발 및 연구 과정에서 문서를 작성하는 일이 귀찮아도, 그 문서가 이슈 해결에 얼마나 기여하는지를 팀원들이 직접 체감하게 되면, 자연스럽게 문서화 습관이 자리 잡는다. 특히 PCL 프로젝트처럼 알고리즘 실험, 데이터 전처리, 대규모 배포 등 협업 범위가 넓은 경우라면, 협업 툴로부터 받는 이점이 더 두드러진다. Confluence에 간단히 “예시 데이터셋, 샘플 코드, 성능 결과, 적용 대상 환경”을 표로 요약한 뒤, Jira 이슈에 “이 표를 토대로 필터 파라미터를 조정하자”고 의견을 남기는 식으로, 일상적이고 즉각적인 커뮤니케이션을 반복하면 협업 효율이 극대화된다.
