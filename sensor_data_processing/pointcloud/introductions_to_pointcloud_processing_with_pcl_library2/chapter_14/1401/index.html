개요¶

대규모 포인트클라우드 처리 프로젝트를 진행할 때 폴더 구조와 빌드 스크립트를 체계적으로 구성하는 일은 협업과 유지보수에 중요한 핵심 요소가 된다. 예컨대 PCL 라이브러리를 중심으로 프로젝트를 설계할 경우, 폴더 구조는 단순히 소스와 헤더를 분리하는 데서 그치지 않고, 샘플 코드, 실험용 스크립트, 의존성 확인용 설정 파일 등을 모두 포괄해야 하며, 빌드 스크립트 역시 단일 플랫폼과 컴파일러만 가정하지 않고 확장 가능하도록 준비하는 것이 바람직하다. PCL은 다른 수많은 의존 라이브러리를 요구하기도 하고, 포인트클라우드 정합이나 필터링, 픽처 영역 분할 등의 고수준 기능을 제공하기 때문에, 중첩된 종속성 문제를 해결하기 위한 구조적 설계와 자동화가 필수적이다.

폴더 구조 설계의 기본 원칙¶

PCL 기반 프로젝트에서 폴더 구조를 설계할 때는 환경 설정, 소스 코드, 빌드 산출물, 문서 등을 구분해 놓는 것을 우선적으로 고려한다. 주로 소스 코드와 헤더 파일은 모듈별 또는 기능별로 분할해 놓고, 빌드 결과물을 한 데 모으는 전략을 사용한다. 예컨대 의존성이 서로 다른 외부 라이브러리를 각각 별도의 폴더에서 관리하거나, PCL 자체를 서브모듈 형태로 두어 버전을 명시적으로 제어할 수도 있다.

PCL 라이브러리는 다음과 같은 모듈적 구성을 제공한다. \(\mathrm{pcl_common}, \mathrm{pcl_io}, \mathrm{pcl_filters}, \mathrm{pcl_features}, \mathrm{pcl_segmentation}\) 등으로 이어지는 모듈 구조는 전체 프로젝트에서 필요한 부분만 선별하여 사용하는 것을 가능케 한다. 프로젝트 관점에서는 이 모듈들을 어떻게 링크할 것인지, 필요한 때에만 컴파일되도록 만들 것인지 등이 폴더 구조와도 직결된다.

폴더 구성을 도식화하면 아래와 같이 다이어그램으로 나타낼 수 있다.

flowchart TB A(Root) --> B(src) A(Root) --> C(include) A(Root) --> D(docs) A(Root) --> E(build) A(Root) --> F(third_party) B --> G(Algorithms) B --> H(Applications) C --> I(Headers for Algorithms) C --> J(Headers for Applications) F --> K(PCL Submodule) F --> L(Other libs)

여기서 Root 폴더 안에 src, include, docs, build, third_party 등의 서브 디렉터리를 두고, src에는 알고리즘 구현과 응용 코드를 기능 단위로 나누어 배치한다. include에는 공개해야 할 헤더 파일만 별도로 둔다. third_party 폴더는 외부 라이브러리나 서브모듈 형태로 관리하는 PCL을 비롯해, CUDA, OpenCV 등 다른 라이브러리들을 함께 모아둘 수 있다.

빌드 산출물은 \((\mathrm{.o}, \mathrm{.so}, \mathrm{.dll}, \mathrm{.exe})\)와 같은 모든 결과물을 분산시키지 않고 build 폴더 아래에 통합 관리하는 방식을 많이 채택한다. 이렇게 하면 빌드 폴더만 삭제하면 깨끗한 상태로 되돌릴 수 있기 때문에, 협업 시 충돌이나 중복 문제가 훨씬 적어진다. 또한 문서화에 필요한 텍스트 파일, Doxygen 설정 파일 등은 docs 폴더에 모아서 관리하면 유지보수 시 참고하기 쉽다.

서브모듈로 PCL을 직접 관리할 경우에는 third_party 안에서 PCL 버전을 명시적으로 고정하거나, 별도의 branch를 연결해두어 프로젝트 업데이트 시점에 맞추어 버전을 변경할 수 있다. 이는 포인트클라우드 처리에서 버전 차이로 인해 발생할 수 있는 API 호환성 문제를 줄여준다.

빌드 스크립트 설계의 개념적 접근¶

폴더 구조를 잘 잡아 놓았다면, 이제 이를 뒷받침하는 빌드 스크립트 역시 구조적으로 작성해야 한다. 일반적으로는 CMake를 많이 쓰며, 플랫폼에 따라 빌드 스크립트가 약간씩 달라질 수 있다. 예컨대 윈도우 환경에서는 Visual Studio 솔루션 파일을 자동 생성하도록 하고, 리눅스 계열에서는 make 기반의 Makefile을 생성하도록 한다. 근래에는 CMake가 가장 표준적으로 활용되는 추세이므로 CMake 설정 예시를 많이 다룬다.

CMakeLists.txt를 여러 개로 분리해, 모듈별로 관리하는 방식도 유용하다. 예를 들어 상위 루트 디렉터리의 CMakeLists.txt 안에서 하위 모듈 디렉터리를 add_subdirectory로 불러들이며, 하위 모듈마다 또 다른 CMakeLists.txt를 두는 식이다. 이런 구조를 통해 모듈별로 빌드 옵션을 달리 설정하거나, 특정 기능을 켜고 끌 수 있다.

빌드 스크립트에서 자주 활용되는 매크로나 함수들을 별도의 cmake 폴더에 두고, 프로젝트 공통으로 사용할 수 있도록 설계하기도 한다. 예컨대 find_package를 통해 PCL이나 Boost, Eigen 등의 라이브러리를 찾는 과정을 래핑하는 유틸리티 함수를 정의해 두면, 여러 모듈에서 이를 공유함으로써 중복을 줄이고 일관성을 높인다.

PCL 라이브러리를 빌드 스크립트에서 연결할 때는 `target_link_libraries(my_target pcl_common pcl_io ... )와 같이 필요한 PCL 모듈을 명시한다. 이때, PCL이 내부적으로 의존하는 다른 라이브러리(예: FLANN, VTK, Eigen 등)가 제대로 설정돼 있는지도 함께 확인해야 한다. CMake는 이를 자동으로 처리해주지만, 버전 불일치 문제나 환경 변수 누락으로 인해 빌드 에러가 발생할 수 있으므로 신중한 설정이 필요하다.

플랫폼별 빌드 환경과 스크립트 확장¶

프로젝트를 진행하다 보면 Windows, Linux, macOS 등에서 동시에 호환되는 코드를 작성해야 하는 경우가 많다. PCL 역시 멀티플랫폼을 지원하지만, OpenGL, VTK, CUDA 등의 라이브러리는 각 플랫폼에서 설치 및 설정 방식이 다를 수 있다. 빌드 스크립트에는 이런 플랫폼 의존적 부분을 분기처리하도록 설계해야 한다.

CMake에서는 $CMAKE_SYSTEM_NAME, $CMAKE_CXX_COMPILER_ID 등을 확인하여, 특정 컴파일러나 OS에 따른 옵션을 분기할 수 있다. 예컨대 리눅스에서만 적용해야 하는 매크로를 추가하거나, 윈도우에서만 활성화해야 하는 /MP 옵션 등을 설정하는 식이다. 또한 CUDA의 경우에는 $CMAKE_CUDA_COMPILER 같은 변수를 확인하여 GPU 컴파일 옵션을 활성화할 수 있다.

서로 다른 OS 환경에서 협업할 때는, 빌드 스크립트뿐만 아니라 폴더 구조도 플랫폼에 특화된 부분이 있을 수 있다. 예컨대 .dll이나 .lib 파일이 생기는 윈도우 환경을 고려해 build/win, 리눅스 환경을 고려해 build/linux처럼 플랫폼별로 폴더를 나누어 설정할 수도 있다. 이 방식은 cross-compiling 시에도 유용한 전략이다.

종속성 관리와 배포를 고려한 스크립트 활용¶

PCL 프로젝트를 배포할 때, 내부에서 사용한 모든 종속 라이브러리를 함께 패키징해야 하는 상황이 생길 수 있다. 예컨대 의존성으로 쓰인 Eigen, FLANN, Boost, VTK 등이 자동으로 포함되어야 최종 배포 환경에서 실행 파일이 제대로 동작한다. 이를 위해서는 CMake의 install 타깃 설정과 패키징 설정을 활용하거나, 별도의 스크립트(Python 등)를 통해 라이브러리를 복사하는 방법을 마련해둔다.

프로젝트 규모가 커지면, CMake만으로는 부족해서 배포 전용 스크립트를 따로 둘 수도 있다. 예컨대 scripts/deploy.sh 혹은 scripts/deploy.ps1 같은 파일을 두어, 빌드 결과물과 라이브러리, 기타 리소스 파일을 모아서 특정 디렉터리에 배치하고 아카이빙까지 진행하는 자동화 과정을 구축할 수 있다. 이는 협업 시 CI/CD 파이프라인과 결합하기도 용이하다.

빌드 스크립트를 확장하여 테스트 자동화도 진행할 수 있는데, 예를 들어 add_test를 사용하거나, GTest 같은 테스트 프레임워크를 연동해 CI 단계에서 유닛 테스트와 통합 테스트가 수행되도록 설계한다. PCL 함수를 직접 호출해 포인트클라우드 데이터 정렬, 정합, 다운샘플링 등의 기능을 테스트하는 경우, 샘플 포인트클라우드를 함께 관리하는 폴더를 만들어두어 테스트 스크립트가 일관된 데이터를 활용하도록 할 수 있다.

외부 파이프라인 연동 및 코드 포맷팅¶

포인트클라우드 처리 과정에서, 종종 ROS(Robot Operating System) 등의 외부 프레임워크와 연동하는 상황이 있다. 이때 ROS의 Catkin 빌드 시스템이나 Colcon 등과 CMake의 충돌을 최소화하기 위해, 별도의 폴더 구조를 갖추거나, 패키지 의존성을 명시적으로 구분하는 설계가 필요하다. 예컨대 PCL을 ROS 환경에서 사용하려면 sensor_msgs/PointCloud2 형식을 받아들여 pcl::PointCloud로 변환해야 하는 브리지 코드가 추가로 필요해지기도 한다.

협업에서 품질 관리를 위해 clang-format, cpplint 등을 연동해 코드를 정적 점검하고, 자동 포맷팅하도록 설정할 수 있다. 이를 위해 CMake나 별도의 스크립트에서 make format 같은 타깃을 정의하거나, git pre-commit 훅으로 포맷팅을 강제하는 방식을 쓸 수 있다. 이런 자동화가 곧 유지보수성과 협업 효율성을 높여준다.

폴더 구조 예시: 알고리즘 모듈화¶

조금 더 세부적으로 들어가, 알고리즘 단위로 모듈화하는 전략을 예시로 들어볼 수 있다. 예컨대 포인트클라우드 필터링 알고리즘을 모아 둔 src/Filters와 그에 대응하는 헤더 include/Filters를 만들어 두면, 이 디렉터리 안에는 PCL의 filter 모듈을 래핑하거나 확장한 로직이 담기게 된다. 반면, 특징 추출(Features) 파트는 src/Features, include/Features로 따로 두고, SIFT, ISS, Harris3D 등 다양한 특징 추출 기법의 구현체와 PCL 함수 호출 코드를 이 안에 배치한다.

메인 실행 파일은 src/Apps 같은 디렉터리에서 작성하며, 여기서 원하는 알고리즘 모듈을 선택적으로 호출할 수 있다. 이런 구조에서는 CMake에서 각 모듈을 타깃으로 만들어 두고, target_link_libraries(app Filters Features ...) 식으로 링크한다. 이렇게 모듈별로 분리하면, 필요 없는 기능은 빌드 대상에서 제외하기도 쉬워지고, 특정 알고리즘만 재컴파일하는 것도 빨라진다.

CMake 모듈 의존성과 컴파일 옵션¶

PCL 프로젝트 내에서 모듈별로 소스 코드를 구분하고 이를 CMake에서 add_subdirectory로 관리할 때, 각 모듈이 필요한 라이브러리와 컴파일 옵션을 어떻게 설정할지 명확히 정의하는 것이 중요하다. 예컨대 포인트클라우드 시각화 기능을 사용하는 모듈이라면, PCL 내에서도 \(\mathrm{pcl_visualization}\) 모듈 및 VTK 의존성이 추가로 필요하므로, 해당 CMakeLists.txt 안에서 find_package(VTK REQUIRED)를 호출하거나, target_link_libraries에 시각화 관련 라이브러리를 포함해야 한다.

컴파일 옵션은 최적화, 디버깅 여부, 경고 레벨 등의 측면에서 모듈별로 다를 수 있다. 예를 들어 필터링 모듈이나 대규모 데이터 처리 모듈에서는 고성능을 위해 $-O3 최적화를 활성화하고, 알고리즘 정확도를 확인해야 하는 테스트 빌드에서는 $-Og나 $-O0를 사용해 디버깅 가시성을 높일 수 있다. CMake 변수 중 $CMAKE_BUILD_TYPE나 $CMAKE_CONFIGURATION_TYPES(Visual Studio 환경) 등을 통해 이러한 옵션을 분기처리할 수 있으며, 특정 모듈에 대해서만 추가 매크로를 부여하는 식으로 세부 설정을 줄 수도 있다.

C++ 표준(예: C++14, C++17, C++20) 역시 모듈마다 원하는 기능이 다를 수 있으므로, $CXX_STANDARD를 모듈마다 지정하거나, 상위 프로젝트에서 일괄적으로 세팅하되 하위 모듈에서 필요하면 재설정하도록 허용할 수도 있다. 다만, 통일성을 위해 최대한 한 가지 표준으로 전체를 맞추는 편이 좋고, 그 외 특수한 경우가 있다면 CMake 옵션으로 토글할 수 있도록 유연성을 확보한다.

테스트 디렉토리의 구조¶

신뢰도 높은 포인트클라우드 처리를 위해서는 수많은 테스트 케이스가 필요하다. 특히 PCL 함수들을 wrapping하거나 커스텀 알고리즘으로 확장할 경우, 실제 센서 데이터나 시뮬레이션 데이터를 이용해 기능 검증이 이뤄져야 한다. 이를 위해 별도의 tests 디렉토리를 마련해, 유닛 테스트와 통합 테스트를 구분해 배치한다.

예컨대 tests/unit 아래에는 알고리즘 단위의 테스트를, tests/integration 아래에는 여러 모듈을 엮어 실제 파이프라인을 검증하는 테스트 코드를 둘 수 있다. CMake에서는 해당 테스트 디렉토리 안에 있는 CMakeLists.txt에서 add_executable 혹은 add_library로 테스트 파일을 빌드한 뒤, add_test 명령을 통해 테스트를 등록한다. GTest나 Catch2 같은 테스트 프레임워크를 사용할 경우, 이를 링크하고 테스트 함수를 구현하면 된다.

테스트에서 포인트클라우드 데이터를 다룰 때, 크기가 큰 데이터셋이 필요할 수 있다. 지나치게 대규모 데이터셋을 포함하면 저장소 용량이 부담되므로, 최소한의 샘플 데이터를 두고 중요한 케이스만 검증하도록 설계한다. 필요하다면, 외부 서버에서 테스트 데이터셋을 다운로드받아 사용하도록 스크립트를 만들 수도 있다. 이를 위해 tests/data 디렉토리를 두고, 여기에 .pcd, .ply 등의 예시 파일을 배치하거나, 특정 URL에서 받을 수 있도록 설정한다.

샘플 데이터 관리¶

포인트클라우드 처리 특성상 실험용 샘플 데이터가 필수적이며, 이를 어떻게 저장소에 포함시킬지, 혹은 별도로 관리할지 결정해야 한다. 작은 크기의 .pcd 파일들은 버전 관리 시스템에 직접 포함하기도 하지만, 대용량 파일은 저장소가 비대해지는 문제를 일으킨다. 이를 해결하기 위해 Git LFS(Git Large File Storage)를 적용하는 방법도 있고, 아예 샘플 데이터 전용 레포지토리를 따로 만들어 서브모듈로 연결한 뒤, 필요 시점에만 동기화하는 전략도 쓸 수 있다.

샘플 데이터는 알고리즘 검증과 시연을 위해서만 존재하는 경우가 많지만, 협업 상황에서 다양한 케이스를 재현하기 위해 반드시 공유되어야 한다. 이런 목적이라면 third_party/data 또는 resources처럼 별도의 폴더를 두고, 대략적인 용량과 용도를 기록한 README 파일을 곁들여 두면 유용하다. 그리고 빌드 스크립트나 테스트 스크립트에서 이 경로를 자동으로 참조해, 매뉴얼 설정 없이도 테스트가 동작하게 만든다.

PCL은 포인트클라우드 데이터를 읽고 쓰는 편의 기능을 제공하지만, 확장자를 인식하지 못하거나 커스텀 포맷을 사용해야 하는 경우가 생길 수 있으므로, 샘플 파일 형식을 다양화해 두면 좋다. 예컨대 .pcd, .ply, .obj, .bin 형태로 된 점 구름 파일 등을 함께 둠으로써, 추후 확장성을 높일 수 있다.

코드 재사용과 API 설계¶

PCL을 직접 쓰면서 추가로 작성한 필터링, 매칭, 세그멘테이션 함수 등을 일반화해 재사용 가능한 API 형태로 설계하면, 협업 시 다양한 팀원이 동일한 코드 베이스를 쉽게 활용할 수 있다. 이를 위해 include/ 디렉터리에 공개 API 헤더를 두어 모듈 간 의존성을 최소화하고, 불필요한 구현 세부사항은 src/ 내부에 감춘다. API 함수 설계 시에는 PCL의 자료구조, 예컨대 \mathrm{pcl::PointCloud 등의 타입을 그대로 노출할지, 래핑 타입을 사용할지를 결정해 일관성을 유지한다.

가령 확장된 필터링 함수를 다음과 같이 선언한다고 하자.

#include <pcl/point_cloud.h>
#include <pcl/point_types.h>

namespace custom_filters {
    void removeOutliers(
        pcl::PointCloud<pcl::PointXYZ>::Ptr input,
        pcl::PointCloud<pcl::PointXYZ>::Ptr output,
        float threshold
    );
}

이때 구현부는 src/Filters/remove_outliers.cpp 등에 두고, 헤더 파일에는 함수 시그니처와 간단한 주석만 제공한다. 이렇게 모듈화하면 다른 모듈(예: Features나 Segmentation)이 필요 시 이 함수를 호출해 곧바로 포인트클라우드를 후처리할 수 있다. API의 파라미터나 반환 타입을 단순화하거나 템플릿화를 통해, 다양한 포인트 타입을 지원하도록 설계하는 것도 가능하다. 예를 들어 RGB, Normal, Intensity를 포함한 점 클라우드 처리를 모두 지원하고 싶다면 템플릿을 사용하는 방식으로 확장할 수 있다.

Git 브랜치 전략과 태깅¶

협업 시에는 기능 단위로 브랜치를 파생시켜 개발 후 병합하는 Gitflow 전략 등을 사용할 수 있다. PCL 버전 호환성 문제가 발생할 때, 예컨대 PCL 1.11 버전을 사용한 브랜치와 1.12 버전 브랜치가 동시에 존재할 수 있는데, 이러한 호환성 이슈를 명확히 식별하기 위해 태그나 브랜치를 분명하게 구분하는 것이 좋다. 특히 특정 버전의 PCL에 최적화된 코드는 향후 버전에서 빌드가 깨지지 않도록 유의해야 한다.

프로덕션 환경에서 안정적으로 동작하는 시점에 릴리스 태그를 달아, 해당 시점의 PCL 버전, 내부 종속 라이브러리 버전 등을 문서화하면, 나중에 동일 환경을 재현하기가 쉽다. 이를 위해 CMake 설정에서 $PCL_VERSION$을 읽어와 빌드 시점에 기록하거나, CI/CD 단계에서 자동으로 버전을 추출해 배포물에 포함하는 식으로 추적 가능성을 높인다.

CI/CD 파이프라인 연동¶

프로젝트 규모가 커지면, CI(지속적 통합)와 CD(지속적 배포) 파이프라인을 구축해 코드 품질과 배포 과정을 자동화하는 것이 바람직하다. 예를 들어 GitHub Actions, GitLab CI, Jenkins 등을 활용해 pull request가 생성되면 자동으로 빌드를 수행하고, 테스트를 통과하는지 검증한다. 여기서 PCL 의존성을 어떻게 설치할지, OS별로 설치 스크립트가 달라질 수 있으므로 주의가 필요하다.

리눅스 환경에서는 apt-get install 등을 통해 PCL을 손쉽게 설치할 수 있으나, 특정 버전을 요구하는 경우에는 소스 빌드를 해야 할 수도 있다. 윈도우 환경에서는 vcpkg나 챠콜레이트(Chocolatey) 같은 패키지 매니저를 사용하거나, PCL 바이너리를 별도 설치해야 하는 경우도 있다. 이러한 설정 과정을 CI 스크립트에 반영해, 각 플랫폼별로 동일한 테스트가 수행되도록 조정한다.

배포 자동화까지 고려한다면, 테스트가 모두 통과했을 때 빌드 산출물(실행 파일, 라이브러리, 샘플 데이터 등)을 아티팩트로 저장하거나, Docker 이미지를 만들어 레지스트리에 푸시함으로써 다른 개발자가 손쉽게 환경을 재현하게 만들 수도 있다. 이렇게 CI/CD 환경을 잘 설계해 두면, 다수의 기여자가 함께 작업하더라도 빌드 및 테스트 결과를 빠르게 공유하며 작업 품질을 높일 수 있다.

폴더 구조와 빌드 스크립트 설계¶

빌드 스크립트 유지보수와 확장 전략¶

장기적으로 프로젝트를 운영하려면, 빌드 스크립트(CMakeLists.txt 등)를 잦은 변경 없이 확장 가능하도록 설계해야 한다. 이를 위해 상위 프로젝트의 CMakeLists.txt에 공통 설정(예: C++ 표준, 경고 레벨, 전역 매크로 등)을 최대한 모아서 정의하고, 하위 모듈에서는 필요한 설정만 추가적으로 Override하는 방식을 취한다.

모듈 간 중복 설정이나 충돌을 막기 위해서는, 다음과 같은 방법을 고려할 수 있다. 상위 CMakeLists.txt에서 $project\_wide\_options 같은 변수나 함수로 빌드 옵션을 설정하고, 하위 모듈 CMakeLists.txt에서는 이 변수를 참조해 컴파일 옵션을 재사용한다. 하위 모듈에서만 필요한 옵션은 로컬로 정의하고, 상위에 불필요하게 전파되지 않도록 주의한다.

프로젝트 규모가 매우 커져서 모듈 수가 많아질 경우, macro나 function 등을 활용해 반복되는 CMake 구문을 추상화하는 편이 좋다. 예컨대 PCL을 포함한 특정 라이브러리를 링크하는 과정을 link_pcl_and_others(my_target) 같은 함수로 미리 정의해 두면, 여러 곳에서 코드 중복을 줄이고 재사용성을 높일 수 있다.

코드 제너레이션과 스크립팅¶

포인트클라우드 처리 과정에서 반복적인 코드(예: 여러 포인트 타입을 대상으로 유사한 알고리즘을 구현해야 하는 경우)를 생성해야 할 수도 있다. 이를 C++ 템플릿만으로 처리하기에는 한계가 있다면, 별도의 코드 제너레이션 스크립트를 둘 수 있다. 예컨대 Python 스크립트를 통해 각 포인트 타입별로 C++ 소스 파일을 자동으로 생성하고, CMake에서 빌드 시점에 이 스크립트를 호출하도록 설정하는 식이다.

자동 생성된 파일들은 build/generated/처럼 따로 모아 관리해도 되고, 실제 소스 디렉터리에 포함시키되 .gitignore로 버전 관리를 제외할 수도 있다. 이런 방식을 잘 설계해 두면, 코드 중복이나 실수를 줄이면서 다양한 타입에 일관된 알고리즘 지원이 가능해진다.

안전한 멀티쓰레딩 고려¶

PCL 자체가 OpenMP나 TBB, GPU(CUDA) 병렬화를 지원하는 기능을 제공하기도 하지만, 프로젝트 수준에서 멀티쓰레딩을 고려해야 하는 경우가 있다. 예컨대 대규모 포인트클라우드 처리를 여러 쓰레드나 CUDA 커널에서 병렬화하는 코드를 작성할 때, CMake에서는 $FindOpenMP.cmake, $FindCUDA.cmake 등을 사용해 시스템 환경을 감지하고, 필요한 컴파일 옵션을 추가한다.

폴더 구조 측면에서도 멀티쓰레딩/병렬화와 관련된 코드를 src/Parallel 같은 별도 디렉터리에 분리해 두면, 병렬화 로직을 전역으로 확산시키지 않고 모듈화할 수 있다. 빌드 스크립트에서 $CMAKE_CUDA_ARCHITECTURES 등을 세팅해 GPU 아키텍처별 최적화를 진행할 수도 있다.

코드 예제와 튜토리얼 디렉터리¶

프로젝트 안에서 PCL 라이브러리 활용 예제나 샘플 코드, 튜토리얼 등이 별도로 필요하다면, examples 또는 tutorials 디렉터리를 둘 수 있다. 이곳에는 학습목적이나 데모용으로 작성된 간단한 코드를 배치하고, CMake에서 별도의 타깃으로 빌드하게 만들면 된다.

이를 통해 협업자나 신규 참여자가 PCL API와 프로젝트 구조를 빠르게 익히도록 도울 수 있다. 예컨대 examples/voxel_grid.cpp 파일 안에 다운샘플링 알고리즘 시연 코드를 두고, 빌드 후 실행하면 .pcd 파일을 읽고 voxelization 결과를 보여주도록 설정한다. 이런 예제들은 학습뿐 아니라 디버깅, 기능 검증에도 유용하다.

문서화와 Doxygen 설정¶

PCL 라이브러리 기반의 코드를 문서화할 때, Doxygen 등을 활용해 함수별 API 문서를 자동 생성하려면, 적절한 폴더 구조와 설정 파일이 필요하다. docs 폴더 아래에 Doxyfile(Doxygen 설정 파일)을 두고, INPUT 경로로 include와 src를 지정해 주면, 주석에 따라 문서를 생성할 수 있다.

또한 CMake에서 make doc 같은 타깃을 만들어, Doxygen 빌드를 자동화할 수 있다. 이때 docs/Doxyfile.in 파일을 템플릿으로 두고, CMake에서 변수를 주입해 프로젝트 버전 등을 동적으로 반영하도록 설정하기도 한다. 문서화가 잘 돼 있으면, 협업자나 사용자들이 PCL API를 확장한 프로젝트 코드를 보다 쉽게 파악할 수 있다.

협업을 위한 버전 고정 전략¶

프로젝트 내에서 종종 “어떤 버전의 PCL과 의존성 라이브러리를 사용하는지”를 명시해야 한다. 이를 위해 third_party/ 아래에 .cmake 혹은 .txt 파일로 의존 라이브러리 버전을 기록해 두거나, find_package(PCL 1.11 EXACT REQUIRED)처럼 CMake에서 정확한 버전을 요구하도록 설정할 수 있다.

또한, vcpkg, Conan 같은 패키지 매니저를 사용해 버전을 고정하는 방식도 있다. 예컨대 conanfile.txt나 vcpkg.json 등에 PCL과 Boost, Eigen, FLANN 등 필요 라이브러리 버전을 명시하면, 협업자가 동일한 버전을 자동으로 설치할 수 있다. 이때 CI 스크립트에서도 같은 패키지 매니저와 설정을 사용해 일관된 환경을 유지한다.

서브모듈 관리 시 유의점¶

git submodule을 통해 PCL 자체를 소스 형태로 포함하는 프로젝트도 있다. 이 경우, PCL 레포지토리의 특정 커밋 SHA를 가리켜 놓고, git submodule update --init --recursive 명령을 통해 동기화하는 방식을 쓴다. 장점은 PCL을 직접 커스터마이징하거나 특정 버전에 맞춰 패치를 적용할 수 있다는 것이고, 단점은 PCL 빌드 시간이 길어지거나 충돌이 발생할 위험이 있다는 점이다.

서브모듈을 활용할 때는 third_party/pcl 폴더에 submodule이 들어간다고 가정하면, 상위 CMakeLists.txt에서 add_subdirectory(third_party/pcl)로 빌드하도록 설정할 수도 있다. 이렇게 하면 외부 패키지 없이도 PCL을 바로 사용할 수 있지만, 플랫폼별 의존성을 추가로 맞춰야 하므로, CI 스크립트에도 이 부분을 반영해야 한다.

문서화와 유지보수 체계¶

PCL 라이브러리를 기반으로 한 프로젝트는, 시간이 흐르면서 점점 새로운 알고리즘과 기능, 최적화가 추가된다. 폴더 구조가 잘 잡혀 있어도, 이에 대한 문서화와 유지보수 체계가 없다면 혼란이 생길 수 있다. 각 모듈의 목적, 빌드 방법, 의존성, 테스트 커버리지 등을 문서나 Wiki 등에 정리해 두면, 협업과 버그 추적이 수월해진다.

특히, 포인트클라우드 처리에서는 센서 종류(LiDAR, Structured Light, Stereo Camera 등), 포인트 타입(XYZ, Normal, RGBA, Intensity 등), 운영체제, GPU 모델 등이 다양하게 달라질 수 있어, 호환성을 관리하는 문서도 필요하다. “어떤 조합이 공식적으로 지원되는지”를 명시하는 식이다.

이런 문서화가 빌드 스크립트와 폴더 구조와 합쳐져, 협업자나 신규 사용자에게 ‘온전히 세팅된’ 안내를 제공하게 된다. 결국, “이 폴더 안에는 어떤 모듈이 있고, 빌드하려면 어떻게 하면 되는지, 문제 발생 시 어디를 살펴봐야 하는지”를 일관성 있게 알려 주는 것이 목표다.
