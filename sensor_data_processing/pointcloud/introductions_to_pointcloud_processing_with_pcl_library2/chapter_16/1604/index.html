여기에서는 PCL(Point Cloud Library)을 활용하여 포인트클라우드 데이터를 보다 쉽고 체계적으로 다룰 수 있는 부록의 샘플 코드와 프로젝트 템플릿에 대해 구체적으로 다룬다. 이미 본문에서 다양한 예제와 활용 방법을 살펴보았지만, 현업 환경에서 직접 업무 또는 연구에 적용하려면 일정한 구조를 갖춘 프로젝트 템플릿과 재사용 가능한 코드 스니펫이 유용하다. 따라서 이 부록은 독자들이 좀 더 깊이 있는 수준에서 직접 활용하거나 수정할 수 있는 자료를 제공하는 데 초점을 맞춘다.

부록의 목적 및 구성¶

먼저 독자들이 PCL을 프로젝트에 연계해 사용할 때 고려해야 할 빌드 시스템, 디렉터리 구조, 공통 헤더 및 설정 파일 등을 포괄적으로 정리했다. CMake를 이용하여 프로젝트를 구성하는 방법과, 여기에 필요한 패키지 의존성 설정을 포함해 기본 템플릿을 구성해 두었다. 포인트클라우드 관련 알고리즘을 테스트하거나 빠르게 프로토타이핑할 수 있도록 최소 단위의 샘플 코드도 제공한다. 예를 들어 간단한 좌표 변환이나 필터링, 정렬 알고리즘 시연 등에 활용할 수 있는 모듈화된 코드 스니펫이 포함되어 있다.

PCL을 활용하는 과정에서 가장 중요한 부분은 여러 기능을 조합하는 작업 흐름을 어떻게 구조화하느냐인데, 이러한 측면을 반영하여 샘플 프로젝트 템플릿은 필터 모듈, 정합 모듈, 시각화 모듈 등을 독립된 디렉터리에 구성해 두었다. 하나의 대규모 프로젝트 내부에서 PCL과 관련된 여러 기능을 동시에 개발·관리하려면, 공통 인터페이스 계층과 독립된 구현 계층을 구분하는 방식으로 설계하는 것이 용이하다. 본문에서는 이를 단순화하여 설명했지만, 부록에서는 좀 더 고급 수준의 예시를 보여 주어, 실제로 여러 개발자가 동시에 작업할 때에도 효율적으로 유지보수할 수 있도록 했다.

샘플 코드의 특징¶

샘플 코드들은 각 기능별로 모듈화되어 있어, 필요한 부분만 발췌하여 다른 프로젝트에 바로 재활용할 수 있도록 구성했다. 예를 들어 대용량 포인트클라우드를 처리하는 병렬화 예시 코드, MeshLab이나 Blender 등 외부 3D 툴과 연동하기 위한 포맷 변환 예시 코드, 그리고 GPU 가속을 통해 일부 주요 알고리즘을 빠르게 실행하는 방법에 관한 예시가 준비되어 있다. 특히 GPU 가속 파트에서는 CUDA 혹은 OpenCL과 같이 직접적인 GPU 프로그래밍 환경을 다루기보다는, PCL에서 제공하는 GPU 모듈과 연계할 때의 주의 사항이나 워크플로를 중심으로 다룬다. 이를 통해 기존의 CPU 기반 코드를 GPU 모듈에 교체하거나 추가할 때 발생할 수 있는 메모리 전송 오버헤드나 동기화 문제 등의 해결책을 제시한다.

여기에 덧붙여, 다양한 센서로부터 획득한 포인트클라우드를 전처리하는 방식을 세분화해서 보여 준다. 센서 노이즈가 심한 경우 Outlier Removal 필터를 반복 적용하는 코드 스니펫, 반면 상대적으로 깔끔한 데이터에는 Voxel Grid 필터로 해상도를 단일화하는 예시 등을 별도로 분리했다. 더 나아가 질감(texture) 정보와 함께 제공되는 포인트클라우드를 다루기 위해, PCL 내에서 텍스처를 저장·관리하고 시각화하는 방법을 예시 코드에 포함했다. 이렇게 다양한 상황에 대응할 수 있는 예제들을 제공함으로써, 독자들이 자신이 다루는 데이터 유형과 응용 분야에 따라 필요한 스니펫만 선택적으로 조합해 사용할 수 있도록 했다.

프로젝트 템플릿의 구조¶

프로젝트 템플릿은 크게 공통 라이브러리 디렉터리, 모듈 디렉터리, 실행 코드 디렉터리, 그리고 테스트 디렉터리로 구성된다. 공통 라이브러리는 PCL을 비롯하여 기타 수학 라이브러리, 예컨대 Eigen 등의 기본 설정이나 공용 헬퍼 함수 등을 포함한다. 모듈 디렉터리는 포인트클라우드 필터링, 정합, 분할, 시각화 등 각 기능을 맡는 로직이 배치되는 공간이다. 실행 코드는 모듈을 종합하여 최종 결과를 내는 예제 바이너리를 제공하며, 테스트 디렉터리는 GTest 등으로 구성된 단위 테스트와 통합 테스트를 수행해 각 기능 모듈이 정상 동작하는지 자동으로 검증한다.

예를 들어, 다음은 CMakeLists.txt 예시다.

cmake_minimum_required(VERSION 3.10)
project(AdvancedPCLProject)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

find_package(PCL 1.10 REQUIRED COMPONENTS common io filters visualization)

include_directories(${PCL_INCLUDE_DIRS})
link_directories(${PCL_LIBRARY_DIRS})
add_definitions(${PCL_DEFINITIONS})

add_subdirectory(src/modules)
add_subdirectory(src/apps)
add_subdirectory(tests)

상기 예시에서는 PCL의 공통, 입출력, 필터, 시각화 모듈을 찾고, 이를 전역으로 include 및 link 설정한 뒤 각각의 기능을 모듈 형태로 src/modules에 분할 배치하도록 구성했다. src/apps 디렉터리는 최종 실행 파일을 생성하는 부분이며, tests 디렉터리는 단위 테스트를 위한 디렉터리다. 이때 모듈화 구조 덕분에 필요한 기능만 빌드하거나 테스트하는 것이 가능하여, 프로젝트 전체 컴파일 시간을 줄이고 유지보수 편의성을 높일 수 있다.

실제 샘플 프로젝트 템플릿 내에는 이보다 더 상세한 설정이 들어가며, 예를 들어 CUDA를 사용하는 경우에는 별도의 FindCUDA 스크립트를 통해 CUDA 컴파일러를 찾고, PCL의 GPU 관련 모듈을 사용할 때 주의해야 하는 점도 CMake 설정에 추가되어 있다. 또한 특정 운영체제에서만 사용할 수 있는 기능에 대해서는 조건부 컴파일 지시문을 삽입하기도 한다. 예를 들어 Windows에서만 동작하는 DirectX 기반 시각화 코드를 포함하고 싶다면, 다음과 같이 CMake 파일 내부에서 조건부 처리를 할 수 있다.

if(WIN32)
  find_package(DirectX REQUIRED)
  if(DirectX_FOUND)
    # DirectX 기반의 추가 라이브러리 설정
  endif()
endif()

이처럼 프로젝트 템플릿은 포인트클라우드 중심의 고급 기능들을 다루면서도, 여러 개발 환경에 유연하게 대응할 수 있도록 설계되었다. 이에 따라 독자들은 템플릿을 그대로 사용하거나, 혹은 일부 설정만 참조하여 자신의 프로젝트에 맞춰 커스텀하는 선택을 할 수 있다. PCL은 높은 자유도를 제공하지만, 반대로 다양한 옵션 때문에 설정 단계에서 난관에 봉착하기도 한다. 이 부록에서는 그러한 초기 설정 과정의 시행착오를 최소화하기 위해 다양한 OS, 다양한 빌드 환경을 고려한 템플릿 구성을 제시한다.

고급 변환 및 정렬 알고리즘 샘플¶

본 부록에서 제공하는 예제들 중에는 6자유도 변환을 다루는 고급 예제도 포함되어 있다. 기본적으로 회전 행렬과 병진 벡터를 조합하여 3차원 공간 상에서 포인트클라우드를 움직이는 방식은 다음과 같이 표현된다. 회전 행렬을 \(\mathrm{R} \in \mathbb{R}^{3\times3}\), 병진 벡터를 \(\mathbf{t} \in \mathbb{R}^{3}\)라고 할 때, 한 점 \(\mathbf{p} \in \mathbb{R}^{3}\)은 다음과 같은 변환을 거친다.

\[ \mathbf{p}' = \mathrm{R}\mathbf{p} + \mathbf{t} \]

여기에 쿼터니언(Quaternion)을 사용한 표현 방식을 도입하면 보다 안정적인 수치 계산이 가능하다. 예시 코드에서는 쿼터니언을 Eigen의 내장 함수를 통해 다루고, 이를 PCL의 PointCloud 객체에 적용하여 대규모 포인트클라우드 전체를 변경하는 과정을 보여 준다. 또한 ICP(Iterative Closest Point) 알고리즘이나 NDT(Normal Distributions Transform) 알고리즘으로 두 포인트클라우드를 정합할 때, 초기 변환값 설정으로 쿼터니언을 활용하는 예시도 함께 제시한다. 이를 통해 반복 계산 과정에서 발생할 수 있는 수치 오차를 최소화하고, 보다 빠른 수렴을 유도한다.

고급 변환 및 정렬 알고리즘 샘플에서는 특수한 케이스 처리 방법도 소개한다. 예를 들어, 대칭 구조를 가진 포인트클라우드나, 특정 축에 대해 종종 발생하는 회전 불확실성(예컨대 차체 스캔 데이터를 정합할 때 앞뒤 방향이 모호해지는 문제) 등을 해결하기 위한 사전 필터링 기법이 포함되어 있다. 이를 위해 표면 노멀(에지 방향) 기반으로 정렬 과정을 제어하는 스니펫도 제공하는데, 이는 PCL의 NormalEstimation 모듈과 연동하여 다음과 같은 식으로 표면 법선 벡터를 계산한다.

\[ \mathbf{n}_i = \mathrm{PCA}(\{\mathbf{p}_i\}_{i \in \mathrm{neighbor}})_{\mathrm{normal}} \]

표면 노멀을 추정한 뒤에는 이 정보를 ICP 알고리즘에서 가중치로 활용하거나, 혹은 데이터 전처리로써 노멀 방향에 따른 분할(Segmentation)을 수행하는 전략을 채택할 수도 있다. 이러한 예시들은 본문에서 다룬 일반적인 정합 알고리즘을 좀 더 심화하여, 다양한 분야의 3D 데이터 처리에 적용할 수 있도록 하는 실제 코드를 보여 준다.

GPU 및 병렬화 예시¶

CPU만으로는 처리 시간이 오래 걸리는 대용량 포인트클라우드에도 빠르게 접근하기 위한 병렬화 예시가 준비되어 있다. OpenMP 지시문을 사용해 각 포인트에 대해 독립적으로 수행 가능한 연산을 병렬화하는 기본 예시가 제공되고, 추가로 PCL의 GPU 모듈이 지원하는 알고리즘을 실제 프로젝트 템플릿에 통합하는 방법도 설명한다. 이를 위해 CMake에서 FindPCLGPU 모듈을 찾거나, 원하는 알고리즘만 부분적으로 활성화하는 기법이 샘플 코드에 포함되어 있다.

예를 들어, GPU 기반 Voxel Grid 필터를 적용하려면 다음과 같은 과정을 거친다. 포인트클라우드 데이터 \(\mathrm{PCD}_{\mathrm{input}}\)를 GPU 메모리로 전송하고, GPU 상에서 해시 테이블 기반으로 셀을 구성한 뒤, 각 셀에 속한 포인트들을 평균화(혹은 다른 통계량 계산)한 결과를 다시 CPU로 가져온다. 이때 전송에 소요되는 시간이 적지 않으므로, 전체 처리 시간 관점에서 언제 GPU를 활용하는 것이 유리한지 고려해야 한다. 부록에서는 이를 테스트하는 벤치마크 예시도 포함하고 있으며, 특정 입력 사이즈 이상일 때 GPU가 유리해지고, 그 이하에서는 오히려 전송 지연 때문에 CPU 방식이 더 빠를 수 있다는 점을 실제 타이밍 로그와 함께 보여 준다.

모든 GPU 예시 코드는 다음과 같이 샘플 CMakeLists.txt 파일에서 옵션으로 빌드할 수 있도록 되어 있다.

option(ENABLE_GPU "Enable GPU modules" ON)
if(ENABLE_GPU)
  find_package(PCL 1.10 REQUIRED COMPONENTS gpu_containers gpu_octree)
  # GPU 관련 설정 추가
  add_subdirectory(src/gpu_modules)
endif()

이를 통해 GPU를 활성화하거나 비활성화할 수 있고, 병렬화가 필요한 특정 알고리즘만 별도로 모듈로 관리함으로써 프로젝트를 가볍게 유지할 수 있다. 한편, GPU 코드와 CPU 코드를 함께 운영할 때는 같은 데이터를 중복 관리하게 되는 일이 많으므로, 이를 최소화하기 위한 데이터 구조 설계 예시도 포함했다. 예를 들어, 포인트클라우드를 CPU 메모리에 먼저 로딩한 뒤, 필요할 때마다 GPU에 스트리밍 방식으로 전송하여 동기화 비용을 줄이는 방법이 그 예다.

부록 소개(샘플 코드, 프로젝트 템플릿)¶

멀티 센서 데이터 융합 및 확장 예시¶

포인트클라우드 데이터 처리를 위해서는 종종 여러 센서로부터 획득된 데이터를 하나의 좌표계로 통합하는 작업이 필요하다. 예를 들어 라이다(LiDAR) 스캐너와 RGB 카메라, 심지어 레이더(Radar) 센서 등 서로 다른 특성을 가진 센서 조합으로부터 얻은 점군(point cloud), 이미지, 메타데이터 등을 하나의 프로젝트에서 같이 다루려면, 고유한 상호 보정(Calibration) 과정을 통해 맞춰 준 뒤에 PCL에서 제공하는 변환 모듈을 활용해 공동의 좌표계를 설정해야 한다. 부록에서는 이를 손쉽게 적용할 수 있도록, 카메라 행렬 및 왜곡 파라미터를 이용한 점 재투영(Reprojection) 샘플 코드, 그리고 두 개 이상의 3D 센서를 합치는 경우 필요한 교정 행렬 추정 알고리즘 예제를 함께 수록했다.

예를 들어, RGB 카메라에서 획득한 이미지를 포인트클라우드에 매핑할 때에는, 다음과 같은 행렬 연산을 통한다. 특정 픽셀 좌표 \((u, v)\)가 있다고 할 때, 이를 3D 점 \(\mathbf{p}\)와 연계하려면 카메라 내·외부 파라미터(내부 파라미터 행렬 \(\mathrm{K}\), 회전 \(\mathrm{R}\) 및 평행이동 \(\mathbf{t}\))가 필요하다. 3D 공간의 점 \(\mathbf{p}\)가 카메라 좌표계에서 표현되었을 때, 이미지 좌표로 변환하는 과정은 다음과 같은 투영 방정식으로 요약된다.

[uv1]=K(Rp+t)\begin{bmatrix} u \ v \ 1 \end{bmatrix} = \mathrm{K} \bigl(\mathrm{R}\mathbf{p} + \mathbf{t}\bigr)

샘플 코드에서는 OpenCV를 이용해 카메라 캘리브레이션 결과를 읽어 들인 뒤, PCL 포인트클라우드와 매칭하는 과정을 보여 준다. 이렇게 매칭된 텍스처 정보는 PCL의 PointXYZRGB 구조체 혹은 PointXYZRGBA 구조체에 할당되며, 다른 센서(예: 열화상 카메라, IR 카메라 등)로부터 획득한 데이터도 유사한 방식으로 매핑할 수 있다. 부록에 포함된 예시는 센서 간 시간 동기화 문제도 다루는데, 서로 다른 촬영 시점에서 획득된 점군과 이미지를 매칭하려면 센서 시간축을 동일 기준으로 보정해 줘야 한다. 이를 위해 타임스탬프를 활용하는 로직이 별도의 헬퍼 함수로 포함되어 있으며, 불규칙한 프레임 지연이나 드롭(drop)이 발생했을 때 대처하는 방법도 다룬다.

더불어 레이더 데이터처럼 다소 해상도가 낮은 센서에서 생성된 점군을 고해상도 라이다 데이터와 융합할 때에는, PCL의 radiusSearch 나 kdtreeSearch 기능을 적절히 활용하여 근방의 점들을 찾고, 상대적인 밀도 차이를 고려해 가중치 합을 통해 융합된 최종 점 정보를 계산한다. 부록의 샘플 코드에는 이러한 밀도 기반 보정 알고리즘이 구현되어 있으며, 직교 좌표계와 구면 좌표계(레이더에서 측정되는 방위각, 고각 기반) 간 변환 예시도 포함되어 있다. 예시 코드에서 K-D 트리를 사용해 근접 이웃을 탐색하는 방식은 다음과 같은 흐름이다.

// Pseudocode
pcl::KdTreeFLANN<pcl::PointXYZ> kdtree;
kdtree.setInputCloud(highResCloud);

for (auto& p : lowResCloud->points) {
    std::vector<int> indices;
    std::vector<float> sqrDistances;

    if (kdtree.radiusSearch(p, radius, indices, sqrDistances) > 0) {
        // 근방 점들과의 융합 로직 수행
    }
}

이러한 코드를 통해 고해상도 라이다 점군에 대한 근방 탐색 결과를 이용해 레이더 점군을 좀 더 세밀하게 보정할 수 있게 된다. 부록에서는 해당 코드를 프로젝트 템플릿에 손쉽게 통합할 수 있는 구조를 제시하므로, 복잡한 센서 융합 시나리오에서도 모듈 단위로 재활용이 가능하다.

품질 평가와 디버깅 전략¶

포인트클라우드를 다루면서 발생하는 알고리즘적·수치적 오류를 조기에 발견하기 위해서는 시각화나 통계 기반의 품질 평가 툴이 필수적이다. 본문에서는 기본적인 PCL 뷰어를 사용하여 점군을 확인하는 방법을 다뤘지만, 부록에서는 더 심화하여, 자동화된 품질 평가 모듈을 예시 코드로 제공한다. 예컨대 두 점군 간 평균 거리나 최대 오차, 혹은 정합 알고리즘의 반복 횟수에 따른 에러 감소 추이를 그래프로 기록·저장하는 기능 등이 있다.

이를 위해 PCL의 io 모듈과 함께 Matplotlib C++ Binding(또는 다른 plotting 라이브러리)를 연동하여, 정합 과정에서 매 이터레이션(iteration)마다 에러 함수를 로깅하고, 최종적으로 결과 그래프를 PNG나 PDF 등으로 출력하는 간단한 예시 코드를 포함한다. 개발 단계에서 특정 알고리즘이 수렴하지 않는 문제를 발견하면, 이 로깅 시스템을 통해 어느 시점에 에러가 급격히 증가했는지, 혹은 초기 변환이 불완전했는지를 쉽게 파악할 수 있다.

디버깅 측면에서도, PCL 내부에서 발생하는 예외나 경고 메시지를 세분화하여 표시해 주는 로깅 함수를 부록에 수록했다. 예를 들어 PCL에서 지원하지 않는 파일 포맷을 읽으려고 할 때 발생하는 예외나, 정규 벡터(normal) 계산 과정에서 분산 행렬이 특이행렬(singular)이 되어 오류가 나는 경우를 트리거 단위로 캡처해, 개발자가 어느 지점에서 오류가 났는지 로그를 남길 수 있도록 돕는다. 또한 Eigen 라이브러리에서 발생하는 수치 오차 경고나 OpenMP 병렬 처리 과정에서의 race condition 가능성도, 특정 로그 레벨에서만 출력하도록 설정할 수 있는 예시를 보여 준다.

품질 평가와 디버깅 관련 예시는 직접 실행해 볼 수 있도록 간단한 스크립트를 제공한다. 예를 들어 CMake로 컴파일 후, ./runQualityCheck --inputCloud A.pcd --referenceCloud B.pcd 같은 식으로 명령줄 옵션을 입력받아 두 점군 간 기준 통계량을 산출하고, 시각화까지 해 주는 미니 애플리케이션 형태로 구성되어 있다. 이를 통해 독자들은 본문의 알고리즘 이론을 구현할 때, 발생할 수 있는 각종 문제점을 현장에서 즉시 확인하고 수정할 수 있게 된다.

하이브리드 파이프라인 통합¶

현대의 3D 처리 워크플로는 단순히 PCL만 사용하는 경우는 드물다. 예를 들어 3D 메시 기반 처리(예: Meshlab이나 CGAL 라이브러리)를 일부 사용하거나, ROS(Robot Operating System)와 연동하여 로봇 시스템 내에서 실시간 포인트클라우드 처리를 하는 경우도 있다. 부록에서는 이러한 하이브리드 환경에서 PCL 모듈을 통합하는 방법을 다룬다. 예컨대 ROS의 sensor_msgs/PointCloud2 메시지를 PCL PointCloud로 변환하고, 필터나 정합 등의 처리를 수행한 뒤 다시 ROS 메시지로 변환하여 퍼블리시하는 기본 예시 코드를 제공한다.

ROS와 연동하는 경우, CMakeLists.txt 파일 내에서 find_package(catkin REQUIRED COMPONENTS roscpp sensor_msgs pcl_ros) 등을 통해 ROS 메시지 헤더와 PCL을 함께 불러와야 하며, 빌드 대상에 catkin 관련 설정을 추가해야 한다. 부록의 샘플 템플릿에는 catkin 패키지 구성 예시가 들어 있으며, ROS Kinetic, Melodic, Noetic 등 버전에 따른 호환 이슈도 간단히 정리해 두었다. Meshlab 통합 시에는 외부에서 OBJ, PLY 등 메시 파일을 읽어들인 뒤, PCL로 변환하는 예시가 수록되어 있다. 이를 통해 메시 기반의 표면 정보에 포인트클라우드가 결합된 하이브리드 데이터로 확장된 워크플로를 시도해 볼 수 있다.

특히 산업 현장이나 로보틱스 분야에서는 실시간성(real-time)과 정확도, 두 가지를 동시에 만족해야 하는 경우가 많으므로, 라이다 센서의 실시간 스트림을 ROS 토픽으로 받아서 PCL 필터 모듈을 거친 뒤, 추가로 Meshlab 서버에서 표면 리컨스트럭션(surface reconstruction)을 한 결과를 로봇 매니퓰레이터(manipulator)에 전달하는 식의 복합 파이프라인이 가능하다. 이러한 워크플로를 단계별로 분할해 하나의 프로젝트 템플릿 안에 담기에는 너무 복잡할 수 있으나, 부록의 예시는 핵심적인 데이터 변환·연결 과정을 중심으로 최소한의 구성 예시를 보여 주고 있으므로, 독자들이 원하는 수준으로 확장하거나 수정할 수 있다.

데이터 압축 및 스트리밍 기법¶

대용량 포인트클라우드를 다룰 때, 파일 크기가 수백 MB 내지 수십 GB에 달할 수도 있다. 이를 효율적으로 저장·전송하기 위해서는 PCL에서 제공하는 다양한 압축 기능(예: pcl::io::OctreePointCloudCompression)이나 외부 라이브러리(예: Draco, LASzip)를 활용할 수 있다. 부록에는 Octree 기반 압축 기법의 내부 동작 원리를 간략히 설명하고, 실제로 압축된 스트림을 네트워크를 통해 전송한 뒤 다시 복원(decompress)하는 실습 코드를 담았다.

압축 알고리즘은 다음과 같은 과정을 거친다. 우선 포인트클라우드를 옥트리(octree) 구조로 분할해, 분해 수준(depth)에 따라 공간을 8등분하면서 각 노드가 포함하는 점의 범위를 추적한다. 특정 노드가 포함하는 점의 수가 기준 이하로 줄어들면 해당 노드 전체를 하나의 대표 포인트로 축약하거나, 부분 노드에 대한 참조를 더 이상 세분화하지 않는다. 이 압축 과정은 적당한 임계값을 설정해 주면 수십 배 이상의 공간 절감 효과를 얻을 수 있으나, 너무 높은 압축률은 복원 시에 세부 정보 손실이 생길 수 있으므로 목적에 맞게 조정해야 한다.

예시 코드에서는 다음과 같은 압축·복원 함수가 사용된다.

pcl::io::OctreePointCloudCompression<pcl::PointXYZ> compressor;
std::stringstream compressedData;

// 압축
compressor.encodePointCloud(inputCloud, compressedData);

// 복원
pcl::PointCloud<pcl::PointXYZ>::Ptr decompressedCloud(new pcl::PointCloud<pcl::PointXYZ>());
compressor.decodePointCloud(compressedData, decompressedCloud);

이때 compressedData는 일반 텍스트나 바이너리 스트림 형태로 만들 수 있으며, 네트워크 소켓이나 로컬 디스크 파일로 전송·저장한 뒤 나중에 복원할 수 있다. 부록에서는 이러한 과정을 Docker 컨테이너 환경에서 구현한 예시까지 포함해, 클라우드 서버에서 포인트클라우드를 압축해 전송하고, 로컬 머신에서 복원해 시각화하는 전 과정의 데모 스크립트를 제공한다. 이를 통해 오프라인 환경이나 제한된 대역폭 환경에서 고효율로 대용량 포인트클라우드를 주고받는 방법을 이해할 수 있을 것이다.

매크로, 템플릿 메타프로그래밍 기법¶

마지막으로, 부록에서는 C++ 템플릿 메타프로그래밍 기법을 이용해 포인트클라우드의 특정 자료형에 구애받지 않는 범용 모듈을 구현하는 예시를 소개한다. 예컨대 pcl::PointXYZ 외에도 pcl::PointXYZI(intensity), pcl::PointNormal, pcl::PointXYZRGB, pcl::PointXYZRGBA 등 다양한 포인트 유형이 존재하는데, 필터 함수나 변환 함수가 모든 자료형별로 오버로드를 제공해야 하면 코드가 매우 비대해진다. 이를 회피하기 위해 C++ 템플릿을 이용해 “어떤 포인트 타입이라도” 동일한 필터 로직을 적용할 수 있는 형태를 구성할 수 있다.

샘플 코드에서는 SFINAE(Substitution Failure Is Not An Error) 기법과 type traits를 활용해, 특정 필드가 존재할 때만 수행되는 연산(예: RGB가 존재하면 색상 필터 적용)을 선택적으로 활성화하거나, intensity 필드가 없는 포인트 타입이라면 관련 로직을 제외시키는 식의 조건부 컴파일을 구현한다. 또한 매크로를 통해 여러 포인트 타입에 대한 인스턴스화를 자동으로 생성함으로써, 빌드 시점에 필요한 타입만 선택해서 컴파일 시간을 최적화하는 예시도 있다.

이 같은 템플릿 메타프로그래밍 기법은 PCL 내부 소스 코드에서도 폭넓게 활용되고 있는데, 부록의 예제는 이를 축소판 형태로 구현하여 독자들이 직접 연습할 수 있도록 준비했다. 예컨대 HAS_RGB_FIELD 같은 매크로를 정의해 두고, 특정 구조체에 rgb 또는 rgba 필드가 있는지를 체크한 뒤, 적절히 색상 필터 코드를 컴파일할지 말지를 결정하는 방식을 살펴볼 수 있다. 이를 통해 “포인트 타입별로 코드가 난잡해지는 문제”를 해결하고, 확장성 있는 모듈을 작성할 수 있다.
