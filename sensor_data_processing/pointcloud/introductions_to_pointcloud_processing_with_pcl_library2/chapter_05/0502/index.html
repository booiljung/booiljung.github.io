PCL에서 제공하는 핵심 함수 중 하나인 pcl::transformPointCloud는 주어진 포인트 클라우드에 임의의 3차원 좌표 변환을 적용할 때 사용된다. 포인트 클라우드의 좌표계를 재정의하거나, 서로 다른 기준 좌표계를 공유하도록 만들기 위해서는 기하학적 변환을 정확히 이해하고 다룰 수 있어야 한다. 이를 위해서는 먼저 3차원 회전과 평행이동을 표현하는 방법을 숙지한 다음, 해당 변환을 PCL의 포인트 클라우드 데이터 구조와 연결하는 과정을 살펴보아야 한다.

3차원 좌표 변환의 기초¶

3차원 공간에서 임의의 점은 일반적으로 \(\mathbf{p} = \begin{bmatrix} x \ y \ z \end{bmatrix}\) 형태로 표현된다. 이를 회전과 평행이동을 동시에 적용하기 위해서는 동차좌표계를 사용하여 다음과 같은 4차원 벡터로 확장할 수 있다.

\[ \mathbf{\tilde{p}} = \begin{bmatrix} x \\ y \\ z \\ 1 \end{bmatrix} \]

동차좌표계에서의 회전과 평행이동은 하나의 \(4 \times 4\) 행렬로 묶여 표현된다. 예를 들어, 회전행렬을 \(\mathbf{R}\)라 하고 평행이동 벡터를 \(\mathbf{t}\)라 할 때, 전체 변환행렬 \(\mathbf{T}\)는 다음과 같다.

\[ \mathbf{T} = \begin{bmatrix} \mathbf{R} & \mathbf{t} \\ \mathbf{0} & 1 \end{bmatrix} \]

이 행렬을 이용하여 점 \(\mathbf{\tilde{p}}\)를 변환하면,

\[ \mathbf{\tilde{p}}' = \mathbf{T}\mathbf{\tilde{p}} \]

의 결과로 \(\mathbf{\tilde{p}}'\)가 얻어진다. 여기서 \(\mathbf{R}\)는 \(3 \times 3\) 회전행렬, \(\mathbf{t}\)는 \(3 \times 1\) 형태의 이동 벡터, 그리고 아래 행은 \([0, 0, 0, 1]\)의 형태를 가진다.

pcl::transformPointCloud의 기본 사용법¶

PCL에서 pcl::transformPointCloud 함수를 사용하기 위해서는 먼저 입력 포인트 클라우드를 준비하고, 변환행렬 \(\mathbf{T}\)(또는 변환을 나타내는 Eigen::Affine3f, Eigen::Matrix4f 등)을 생성해야 한다. 기본적인 함수 시그니처는 다음과 비슷한 형태를 가진다.

pcl::transformPointCloud(
    const pcl::PointCloud<pcl::PointXYZ> &input,
    pcl::PointCloud<pcl::PointXYZ> &output,
    const Eigen::Matrix4f &transform
);

이 함수는 입력으로 주어진 input 포인트 클라우드의 각 점에 대해 transform을 적용하고, 변환된 결과를 output에 저장한다. 여러 가지 오버로드(overload)가 존재하지만, 가장 단순한 형태는 위 시그니처와 유사하다.

변환행렬 구성¶

3차원 변환행렬을 구성하기 위해서는 먼저 회전행렬 \(\mathbf{R}\)를 정의하고, 평행이동 벡터 \(\mathbf{t}\)를 정의한 다음, 이를 하나의 \(4 \times 4\) 행렬로 합쳐야 한다. Eigen 라이브러리를 사용하면, 다음과 같은 절차로 직접 행렬을 구성할 수 있다.

Eigen::Matrix4f transform = Eigen::Matrix4f::Identity();

// 예: z축으로 90도 회전하는 회전행렬 구성
float theta = M_PI / 2.0f;
transform(0,0) =  cos(theta);
transform(0,1) = -sin(theta);
transform(1,0) =  sin(theta);
transform(1,1) =  cos(theta);

// 예: x축 방향으로 1.0, y축 방향으로 2.0, z축 방향으로 3.0만큼 평행이동
transform(0,3) = 1.0f;
transform(1,3) = 2.0f;
transform(2,3) = 3.0f;

이 예시는 \(z\)축을 기준으로 시계 방향 90도 회전과 동시에 \((1, 2, 3)\)만큼 평행이동을 적용하는 변환행렬을 만든다. 이렇게 구성한 transform을 pcl::transformPointCloud에 전달하면, 입력 포인트 클라우드가 동일한 방식으로 변환되어 출력 클라우드가 생성된다.

변환의 올바른 이해¶

3차원 공간에서의 좌표 변환은 관점에 따라 물체 좌표계를 움직이거나, 카메라 좌표계를 움직이거나, 혹은 전체 장면의 기준좌표계를 움직이는 것과 동일하게 볼 수 있다. 실제로는 하나의 점에 적용되는 변환이지만, 그 해석은 “무엇이 움직이는가?”에 따라 달라진다. 이 점을 혼동하면 같은 변환행렬을 사용해도 예상과 다른 결과가 나올 수 있으므로, 변환 방향과 순서를 명확히 인지해야 한다.

예를 들어,

\[ \mathbf{p}' = \mathbf{R}\mathbf{p} + \mathbf{t} \]

로 표현되는 변환에서 \(\mathbf{R}\)과 \(\mathbf{t}\)는 “현재 좌표계에서 점을 회전시키고 이동시키는 연산”으로 해석할 수도 있지만, 반대로 “기존 좌표계를 \(\mathbf{R}^{-1}, -\mathbf{R}^{-1}\mathbf{t}\)만큼 움직인 것”으로도 해석할 수 있다. 이러한 중의성을 풀어내려면 최종적으로 구하고자 하는 목표 좌표계와 현재 점의 좌표계가 어떻게 연결되는지 꼼꼼히 따져야 한다.

코드 예시¶

다음은 PCL을 사용하여 실제로 포인트 클라우드에 변환을 적용하는 간단한 예시 코드다.

#include <pcl/point_types.h>
#include <pcl/point_cloud.h>
#include <pcl/common/transforms.h> // transformPointCloud가 들어 있음
#include <Eigen/Dense>

int main() {
    pcl::PointCloud<pcl::PointXYZ>::Ptr cloud_in(new pcl::PointCloud<pcl::PointXYZ>);
    pcl::PointCloud<pcl::PointXYZ>::Ptr cloud_out(new pcl::PointCloud<pcl::PointXYZ>);

    // cloud_in 초기화 과정을 가정
    // ...

    Eigen::Matrix4f transform = Eigen::Matrix4f::Identity();

    float theta = M_PI / 4.0f;
    transform(0,0) =  cos(theta);
    transform(0,1) = -sin(theta);
    transform(1,0) =  sin(theta);
    transform(1,1) =  cos(theta);

    transform(0,3) = 1.0f;
    transform(1,3) = 2.0f;
    transform(2,3) = 3.0f;

    pcl::transformPointCloud(*cloud_in, *cloud_out, transform);

    // cloud_out에 변환된 포인트 클라우드가 들어 있음
    return 0;
}

이 코드는 입력 포인트 클라우드 cloud_in의 각 점에 대해, \(z\)축 기준 45도(\(\pi / 4\)) 회전과 평행이동(1, 2, 3)을 적용한 결과를 cloud_out에 저장한다.

점들의 좌표 변화 확인¶

포인트 클라우드가 제대로 변환되었는지 확인하기 위해서는 변환 전후의 위치나 형태를 비교하는 시각화나 좌표 값 비교가 필요하다. 시각화 도구로는 PCL의 pcl::visualization::PCLVisualizer를 사용하거나, 다른 3D 뷰어를 통해 결과를 확인할 수 있다. 또한 특정 점들에 대한 좌표만 확인하고 싶다면 변환 전후의 대표 점을 출력하여 비교하는 것도 도움이 된다.

Eigen::Affine3f를 이용한 변환¶

Eigen 라이브러리에서 3차원 기하학적 변환을 표현하는 방법은 여러 가지가 존재한다. 그중 Eigen::Affine3f는 회전, 평행이동, 스케일 등을 포함한 3D 변환을 보다 직관적인 방식으로 다룰 수 있게 해 준다. Eigen::Matrix4f가 4×4 행렬로 모든 변환 정보를 직접 다루어야 하는 반면, Affine3f는 연산자 오버로딩 등을 통해 연속된 변환을 직관적으로 연결할 수 있다는 장점이 있다.

다음은 Affine3f를 활용하여 변환행렬을 구성하고, pcl::transformPointCloud에 전달하는 간단한 예시다.

#include <pcl/point_types.h>
#include <pcl/point_cloud.h>
#include <pcl/common/transforms.h>
#include <Eigen/Dense>
#include <Eigen/Geometry>

int main() {
    pcl::PointCloud<pcl::PointXYZ>::Ptr cloud_in(new pcl::PointCloud<pcl::PointXYZ>);
    pcl::PointCloud<pcl::PointXYZ>::Ptr cloud_out(new pcl::PointCloud<pcl::PointXYZ>);

    // cloud_in 초기화 과정을 가정
    // ...

    Eigen::Affine3f transform = Eigen::Affine3f::Identity();

    float theta = M_PI / 6.0f; // 30도 회전 예시
    transform.rotate(Eigen::AngleAxisf(theta, Eigen::Vector3f::UnitZ())); 
    transform.translate(Eigen::Vector3f(1.0f, 2.0f, 3.0f));

    pcl::transformPointCloud(*cloud_in, *cloud_out, transform);

    return 0;
}

이 예시에서 transform.rotate(...)와 transform.translate(...)로 연속된 변환을 하나의 Affine3f 객체에 담은 뒤, pcl::transformPointCloud 함수에 그대로 전달했다. 내부적으로는 Affine3f가 4×4 행렬로 해석되어 최종 변환이 적용된다.

여러 개의 변환을 체인으로 구성하기¶

연속된 다수의 변환을 적용해야 할 때는, Affine3f를 이용하면 직관적으로 변환을 누적할 수 있다. 예를 들어, 서로 다른 좌표축을 기준으로 회전한 뒤, 그 결과에 다시 평행이동을 적용하고, 마지막으로 추가적인 축 회전을 적용해야 하는 경우를 생각해 보자. 그런 경우, 다음과 같은 방식으로 작성할 수 있다.

Eigen::Affine3f compound_transform = Eigen::Affine3f::Identity();

compound_transform.rotate(Eigen::AngleAxisf(M_PI / 4.0f, Eigen::Vector3f::UnitX()));
compound_transform.rotate(Eigen::AngleAxisf(M_PI / 6.0f, Eigen::Vector3f::UnitY()));
compound_transform.translate(Eigen::Vector3f(0.5f, 1.0f, 2.0f));
compound_transform.rotate(Eigen::AngleAxisf(M_PI / 3.0f, Eigen::Vector3f::UnitZ()));

이와 같은 코드를 통해 회전과 평행이동을 원하는 순서대로 누적한 뒤, compound_transform을 사용하여 한 번에 포인트 클라우드를 변환할 수 있다. 변환의 순서가 달라지면 결과도 달라지므로, 어떤 순서로 연산을 누적할지 명확히 지정해야 한다.

변환 시 주의할 점¶

회전은 불변축(invariant axis) 주변에서 이루어지기 때문에, 회전을 어느 축을 기준으로 수행하는지 정확히 인지해야 한다. 또한 회전과 평행이동의 순서에 따라 결과가 크게 달라진다. 예를 들어, 어떤 점에 대해 먼저 평행이동을 적용한 뒤 회전하는 것과, 먼저 회전한 뒤 평행이동을 적용하는 것은 완전히 다른 최종 좌표를 낳는다. 이것은 행렬로 표현했을 때 곱셈 순서가 달라짐에 대응한다.

좌표 변환 시 다음과 같은 관점 차이도 주의해야 한다. 물체를 회전하고 평행이동하는 해석과, 관찰자(카메라)를 역방향으로 움직인다고 해석하는 방식이 서로 동등하지만, 두 관점의 변환행렬은 역행렬 관계일 수 있다. 따라서 지금 적용하고자 하는 변환이 “물체를 움직이는 것”인지, “카메라를 움직이는 것”인지를 명확히 구분해 두어야 한다.

색상 정보나 노멀이 있는 포인트 타입 변환¶

포인트 클라우드에서 점 하나를 \(\mathbf{p} = (x, y, z)\)라고 할 때, 그 점에 색상 정보나 노멀(normal) 정보가 포함되어 있을 수도 있다. pcl::PointXYZRGB, pcl::PointXYZRGBA, pcl::PointNormal, pcl::PointXYZRGBNormal 등과 같은 타입이 있다.

이 때 pcl::transformPointCloud 함수는 점의 좌표만 변환한다. 노멀이나 색상 정보는 좌표와는 별개이므로, 색상은 그대로 유지되지만 노멀 벡터는 별도의 방식으로 회전되어야 한다. 일반적으로 노멀 벡터는 물체 표면의 방향을 나타내므로 평행이동에 대해서는 변화가 없고, 회전에 대해서만 영향을 받는다. 색상 정보는 회전이나 평행이동과 직접적인 연관이 없으므로 그대로 유지된다. 즉, pcl::transformPointCloud는

\[ \mathbf{n}' = \mathbf{R} \mathbf{n} \]

와 유사한 방식으로 노멀을 회전하는 기능을 제공한다. 단, 일부 특정 오버로드에서만 노멀 벡터의 회전이 처리되므로, 사용하려는 함수가 올바른 템플릿으로 인스턴스화되는지 확인해야 한다.

메모리와 성능 고려¶

포인트 클라우드가 매우 큰 경우, 변환을 여러 번 반복해서 적용하는 것은 시간과 메모리 측면에서 비효율적일 수 있다. 가능한 한 연속된 변환들은 합성하여 한 번에 적용하는 편이 좋다. transformPointCloud 함수는 내부적으로 입력 클라우드의 모든 점을 순회하며 변환을 수행하므로, 결합할 수 있는 여러 변환을 미리 하나로 합성한 뒤에 한 번만 transformPointCloud를 호출하는 방식을 추천한다.

특히 실시간 처리(real-time)를 목표로 하는 시스템에서는 변환을 최소화하고, 필요하다면 GPU 연산을 활용하는 방법도 고려해야 한다. 하지만 기본적으로 PCL은 CPU 기반으로 동작하므로, 실행 환경에 따라 적절한 최적화가 필수적이다.

In-place 변환¶

pcl::transformPointCloud 함수를 사용할 때 입력 클라우드와 출력 클라우드가 다른 변수에 매핑되는 경우가 일반적이지만, 경우에 따라서는 동일한 변수를 입력과 출력으로 사용해야 할 수도 있다. 이처럼 기존 클라우드를 그대로 업데이트하기 위해서는 다음과 같은 방식이 가능하다.

pcl::transformPointCloud(*cloud_in, *cloud_in, transform);

이 방법을 쓰면 새로운 메모리를 추가로 할당하지 않고, 기존 포인트 클라우드를 덮어쓰게 된다. 포인트 클라우드 크기가 아주 크지 않은 경우에는 메모리를 절약할 수 있으나, 변환 도중에 다른 스레드에서 동일한 포인트 클라우드를 참조하고 있다면 예상치 못한 결과가 나올 수 있다. 다중 스레드 환경에서 이를 사용할 때는 충돌을 피하기 위해 적절한 동기화 또는 복제 과정을 고려해야 한다.

부분 클라우드에 대한 변환¶

때로는 포인트 클라우드 전체가 아니라 특정 구간, 예를 들어 레이블이 특정 조건을 만족하는 점들만 골라 변환하고 싶을 수도 있다. 이 경우에는 부분 포인트 클라우드를 추출하여 transformPointCloud를 적용한 뒤, 다시 전체 포인트 클라우드에 병합하는 방식으로 진행할 수 있다. PCL에서 특정 점들을 부분적으로 추출하기 위해서는 pcl::ExtractIndices나 pcl::PassThrough 등의 필터를 사용하는 방법이 일반적이다.

아래는 인덱스 필터를 사용하여 특정 점들만 뽑아낸 뒤 변환하고, 다시 합치는 사례를 단순화해 보여주는 예시다.

#include <pcl/filters/extract_indices.h>

// 인덱스가 담긴 vector<int> idxs가 있고,
// 원본 포인트 클라우드 cloud_in이 있다고 가정
pcl::PointIndices::Ptr indices(new pcl::PointIndices());
indices->indices = idxs;

pcl::ExtractIndices<pcl::PointXYZ> extractor;
extractor.setInputCloud(cloud_in);
extractor.setIndices(indices);
extractor.setNegative(false);

pcl::PointCloud<pcl::PointXYZ>::Ptr partial_cloud(new pcl::PointCloud<pcl::PointXYZ>);
extractor.filter(*partial_cloud);

// 필요하다면 partial_cloud를 별도로 변환
pcl::PointCloud<pcl::PointXYZ>::Ptr partial_cloud_out(new pcl::PointCloud<pcl::PointXYZ>);
pcl::transformPointCloud(*partial_cloud, *partial_cloud_out, transform);

// 원본 클라우드에 다시 합치는 작업은 사용 목적에 따라 여러 방법이 있으나,
// 특정 인덱스 위치에 그대로 덮어쓰거나, 별도의 데이터 구조를 만들어 반영한다.

이 과정을 통해 원하는 점들만 별도로 선택하여 변환한 뒤, 결과를 적절히 다시 합성할 수 있다. 이런 접근은 복잡한 조건이나 다양한 필터를 적용해야 할 때 유용하다.

회전 중심(pivot) 지정하기¶

좌표 변환 시, 회전의 기준점을 어디로 둘지 지정할 수 있다. 기본적인 회전변환은 좌표계의 원점(0,0,0)을 중심으로 하며, 평행이동과 회전을 적절히 조합하여 다른 임의의 점을 회전 중심으로 설정할 수 있다. 특정 점 \(\mathbf{c} = (c_x, c_y, c_z)\)를 기준으로 회전하고자 할 때, 이를 동차좌표계로 표현하면 아래와 같은 절차로 해석할 수 있다.

먼저, 회전 중심 \(\mathbf{c}\)를 원점으로 이동시키기 위해 \(-\mathbf{c}\)만큼 평행이동을 적용한 뒤 회전, 그 후에 다시 \(+\mathbf{c}\)만큼 역으로 평행이동을 적용하면 된다. 이 변환행렬을 구성하면

\[ \mathbf{T} = \begin{bmatrix} \mathbf{I} & \mathbf{c} \\ \mathbf{0} & 1 \end{bmatrix} \begin{bmatrix} \mathbf{R} & \mathbf{0} \\ \mathbf{0} & 1 \end{bmatrix} \begin{bmatrix} \mathbf{I} & -\mathbf{c} \\ \mathbf{0} & 1 \end{bmatrix} \]

이 된다. 실제 코드로 구현 시에는

Eigen::Affine3f T_center = Eigen::Affine3f::Identity();
T_center.translate(+c);

Eigen::Affine3f T_back = Eigen::Affine3f::Identity();
T_back.translate(-c);

Eigen::Affine3f T_rot = Eigen::Affine3f::Identity();
T_rot.rotate(Eigen::AngleAxisf(theta, Eigen::Vector3f::UnitZ()));

// 최종합성
Eigen::Affine3f T = T_center * T_rot * T_back;

이 과정을 거친 T가 실제로 \(\mathbf{c}\)를 기준으로 한 회전변환을 표현한다.

센서 오리진(origin)과 변환¶

PCL의 포인트 클라우드 구조체에는 포인트뿐 아니라 센서가 놓인 위치, 방향에 대한 정보(sensor_origin_, sensor_orientation_)가 포함되어 있을 때가 있다. 이는 스캔 센서(예: LiDAR, 카메라 등)의 위치가 포인트 클라우드에 기록되어 “이 데이터가 어떤 센서 좌표계에서 얻어졌는가?”를 명시한다.

pcl::transformPointCloud 함수는 기본적으로 각 점의 좌표만 변환해 주며, sensor_origin_과 sensor_orientation_은 별도로 업데이트해 주지 않는다. 따라서 센서의 위치 정보까지 함께 갱신하려면, 변환이 적용된 뒤 별도로 sensor_origin_과 sensor_orientation_을 업데이트해야 한다. 예를 들어,

pcl::PointCloud<pcl::PointXYZ> cloud;
cloud.sensor_origin_ = Eigen::Vector4f(0, 0, 0, 1.0f);
cloud.sensor_orientation_ = Eigen::Quaternionf::Identity();

Eigen::Affine3f T = ...; // 어떤 변환
pcl::transformPointCloud(cloud, cloud, T);

// 센서 원점과 방향도 변환
Eigen::Vector3f so = T.linear() * cloud.sensor_origin_.head<3>() + T.translation();
cloud.sensor_origin_ = Eigen::Vector4f(so(0), so(1), so(2), 1.0f);

Eigen::Quaternionf qo(T.linear());
cloud.sensor_orientation_ = qo * cloud.sensor_orientation_;

이런 방식으로 센서가 어떤 위치에서 어떤 방향으로 데이터를 수집했는지에 대한 메타데이터도 일치하도록 수정해 주어야, 이후에 다른 포인트 클라우드와 정렬하거나 센서 기반의 연산을 수행할 때 혼동이 줄어든다.

역변환(Inverse Transform) 활용¶

포인트 클라우드를 다른 좌표계로 변환한 뒤, 필요에 따라 다시 원래 좌표계로 되돌리는 상황이 자주 발생한다. 예를 들어 로봇의 베이스 좌표계에서 카메라 좌표계로 포인트를 변환했다가, 나중에 다시 베이스 좌표계로 돌려야 할 수도 있다. 이를 위해서는 원래 적용했던 변환행렬의 역행렬을 구해서 포인트 클라우드에 다시 적용하면 된다.

\(\mathbf{T}\)가 어떤 변환을 나타내는 \(4 \times 4\) 행렬이라면, 그 역행렬을 \(\mathbf{T}^{-1}\)라고 할 때,

\[ \mathbf{\tilde{p}}'' = \mathbf{T}^{-1} \mathbf{\tilde{p}}' \]

로 표현된다. 여기서 \(\mathbf{\tilde{p}}' = \mathbf{T} \mathbf{\tilde{p}}\)였으므로, 결국 \(\mathbf{\tilde{p}}'' = \mathbf{\tilde{p}}\)임을 알 수 있다. 즉, 역변환을 통해 원점으로 되돌리는 효과를 얻는다.

Eigen을 사용할 때는 inverse() 함수를 호출하거나, Affine3f의 경우 transform.inverse()를 통해 역행렬을 쉽게 구할 수 있다. pcl::transformPointCloud 호출 시 이 역행렬을 넘겨주면, 정확히 반대 방향의 변환이 적용된다.

Eigen::Matrix4f transform = ... // 어떤 변환
Eigen::Matrix4f transform_inv = transform.inverse();

pcl::transformPointCloud(*cloud_in, *cloud_in, transform_inv);

Eigen::Affine3f를 사용한다면,

Eigen::Affine3f T = ...;
Eigen::Affine3f T_inv = T.inverse();

pcl::transformPointCloud(*cloud_in, *cloud_in, T_inv);

이러한 방식으로 간단히 역변환을 적용할 수 있다. 회전행렬 \(\mathbf{R}\)와 평행이동 \(\mathbf{t}\)만 있다면, 역행렬은 아래와 같이 블록 형태로 유도할 수 있다.

\[ \mathbf{T} = \begin{bmatrix} \mathbf{R} & \mathbf{t} \\ \mathbf{0} & 1 \end{bmatrix}, \quad \mathbf{T}^{-1} = \begin{bmatrix} \mathbf{R}^{T} & -\mathbf{R}^{T}\mathbf{t} \\ \mathbf{0} & 1 \end{bmatrix} \]

여기서 \(\mathbf{R}^{T} = \mathbf{R}^{-1}\)임을 고려하면, 평행이동의 반대 방향이 \(\mathbf{R}^{T}\mathbf{t}\)만큼 적용되어야 함을 알 수 있다.

transformPointCloud와 좌표계 일관성¶

로봇 공학이나 3D 센싱 시스템을 다루다 보면, 여러 개의 센서가 서로 다른 좌표계를 사용하여 동일한 장면을 관측하고 있을 수 있다. 센서 A에서 수집한 포인트 클라우드를 센서 B의 좌표계로 일치시키거나, 특정 월드(world) 좌표계를 정의하고 모든 센서의 데이터를 그 기준으로 모으는 작업을 하게 된다.

이 때 각 센서가 월드 좌표계에 대해 어떤 위치와 자세(orientation)에 있는지, 즉 변환행렬 \(\mathbf{T}_{\text{A}\to\text{W}}, \mathbf{T}_{\text{B}\to\text{W}}\) 등을 정확히 알고 있어야 한다. 만약 센서 A의 데이터가 \(\mathbf{p}_A\)로 표현되어 있고, 월드 좌표계의 점을 \(\mathbf{p}_W\)로 표현한다면,

\[ \mathbf{p}_W = \mathbf{T}_{\text{A}\to\text{W}} \mathbf{p}_A \]

가 된다. 반대로 월드 좌표계의 점을 센서 B 좌표계로 표현하면,

\[ \mathbf{p}_B = \mathbf{T}_{\text{W}\to\text{B}} \mathbf{p}_W \]

이지만

\[ \mathbf{T}_{\text{W}\to\text{B}} = \left(\mathbf{T}_{\text{B}\to\text{W}}\right)^{-1} \]

이므로, 센서 A → 월드 → 센서 B로 이어지는 변환을 모두 명확히 파악해 둬야 좌표계를 올바르게 맞출 수 있다. 이러한 변환 관계를 시각화하면 더 명료해지므로 다이어그램을 간단히 들어 보이면 다음과 같다.

flowchart LR A(Sensor A) --> TW( T_A->W ) TW --> W(World) W --> TB( T_W->B ) TB --> B(Sensor B)

여기서 T_A->W는 센서 A 좌표계에서 월드 좌표계로의 변환, T_W->B는 월드 좌표계에서 센서 B 좌표계로의 변환을 의미한다.

transformPointCloud와 복합적인 정렬 작업¶

일반적인 응용에서 포인트 클라우드를 단순히 회전, 평행이동하는 것 이상의 작업이 필요할 수 있다. 예를 들어, Iterative Closest Point(ICP) 알고리즘으로 포인트 클라우드를 정렬하거나, 피처 매칭을 통해 초기 추정을 얻은 뒤 최적화 과정을 거치는 경우 등이 있다. 이러한 알고리즘들의 결과물은 결국 “이 포인트 클라우드를 다른 포인트 클라우드에 맞추기 위해 필요한 6자유도(6DoF) 변환”으로 표현된다.

이 변환이 바로 \(4 \times 4\) 형태의 최종 행렬이 되며, ICP의 최적 해로부터 얻은 변환행렬을 pcl::transformPointCloud에 전달하면, 알고리즘이 산출한 정렬 결과를 실제로 적용할 수 있다. PCL에서 ICP 사용 시에는 다음과 같은 형식의 함수 호출이 일반적이다.

pcl::IterativeClosestPoint<pcl::PointXYZ, pcl::PointXYZ> icp;
icp.setInputSource(cloud_source);
icp.setInputTarget(cloud_target);

pcl::PointCloud<pcl::PointXYZ> cloud_aligned;
icp.align(cloud_aligned);

Eigen::Matrix4f transformation_icp = icp.getFinalTransformation();

이렇게 얻은 transformation_icp는 source 클라우드를 target 클라우드에 정렬하기 위해 필요한 변환이다. 직접 cloud_aligned를 사용해도 되지만, 다른 목적상 필요하면

pcl::transformPointCloud(*cloud_source, *cloud_source, transformation_icp);

등과 같이 source 클라우드를 덮어써 사용할 수도 있다.

파이프라인에서 transformPointCloud의 활용¶

3D 포인트 클라우드 파이프라인을 설계할 때, 여러 종류의 선처리(pre-processing)나 후처리(post-processing) 단계에서 좌표 변환이 필요한 경우가 자주 발생한다. 예를 들어, 센서가 움직이는 환경에서 지속적으로 포인트 클라우드를 수집하고, 적절한 누적(aggregation) 또는 프레임 간 정렬(alignment)을 통해 전체 지도의 품질을 높이고 싶을 때, 각 단계마다 pcl::transformPointCloud가 활약한다.

센서로부터 새롭게 들어오는 포인트 클라우드가 월드 좌표계를 기준으로 어디에 위치하는지 바로 확인하기 위해, 매 프레임마다 새 포인트 클라우드에 적절한 변환행렬을 적용하는 방식이 대표적이다. 그 뒤, 이미 축적된 월드 좌표계의 포인트 클라우드와 병합하거나, 특정 영역만을 업데이트하는 식으로 파이프라인을 구성할 수 있다.

좌표계 변환이 파이프라인 중간에 다수 배치될 경우, 각 변환단계가 독립적으로 실행되는지, 혹은 상호 종속적인지를 확인해야 한다. 상호 종속적인 변환이라면, 중간 결과를 저장하지 않고 한 번의 합성행렬로 줄이는 것이 성능 측면에서 유리할 수 있다. 예를 들어, 새롭게 수집된 포인트를 먼저 로컬 필터로 노이즈를 제거한 뒤, 센서 좌표계 → 로봇 바디 좌표계 → 월드 좌표계로 연달아 변환해야 한다면, 이를 단계별로 각각 분리해서 호출하기보다는, 최종적으로 필요한 한 번의 변환으로 통합한 뒤 pcl::transformPointCloud를 호출하는 편이 효율적이다.

중간 결과 시각화가 필요한 경우에는 단계별로 변환을 적용한 상태를 확인해야 하므로, 합성행렬 대신 여러 차례 transformPointCloud를 호출하는 편이 코드의 가독성과 디버깅 편의성 면에서 나을 수 있다. 실무 환경에서 성능과 유지보수성 중 어느 쪽을 우선시할지는 프로젝트 요구 사항 및 시스템 제약에 따라 결정된다.

로봇이나 드론 등에서 실시간 처리를 목표로 할 때, CPU 사용량을 줄이기 위해 필요 최소 횟수만 변환을 수행하거나, 병렬 연산 기법을 도입하기도 한다. PCL 자체는 CPU 기반이지만, 하드웨어 가속 환경에서 Eigen 연산을 더 빠르게 수행할 수 있도록 최적화(예: Intel TBB, OpenMP 등)를 병행하기도 한다. 그 밖에 GPU 기반으로 행렬 연산을 처리하는 라이브러리와 결합하여 포인트 클라우드 변환 속도를 높이는 사례도 있으나, 이를 위해서는 별도의 커스텀 구현이 필요하다.

이렇듯 pcl::transformPointCloud는 다수의 센서가 동시 관측하는 장면을 종합하거나, 누적 지도를 실시간으로 구축하거나, 대규모 3D 데이터를 다양한 시점에서 확인하기 위한 파이프라인 전반에 걸쳐 광범위하게 사용된다.

고급 커스텀 포인트 타입과 transformPointCloud¶

PCL은 pcl::PointXYZ, pcl::PointXYZRGB, pcl::PointNormal 등 미리 정의된 포인트 타입 외에도, 사용자가 직접 구조체를 정의하여 포인트 필드를 확장할 수 있도록 템플릿 기반 아키텍처를 제공한다. 예를 들어, 다음과 같이 사용자 정의 포인트 타입을 만들 수 있다.

struct MyPointType {
    PCL_ADD_POINT4D;                  // x, y, z, w
    float intensity;
    float curvature;
    EIGEN_MAKE_ALIGNED_OPERATOR_NEW;
} EIGEN_ALIGN16;

이렇게 정의해 두면, pcl::transformPointCloud의 템플릿 인자로 MyPointType을 지정하여 같은 방식으로 좌표 변환을 적용할 수 있다. 단, 이 때는 해당 포인트 구조체가 PCL 매크로와 Eigen 정렬 매크로를 적절히 포함하고 있어야 한다. 변환 과정에서 실제로 회전이나 평행이동이 적용되는 필드는 \((x,y,z)\) 좌표뿐이며, 나머지 필드는 별도로 정의된 연산을 지원하지 않는 한 그대로 보존된다.

pcl::PointCloud<MyPointType>::Ptr mycloud_in(new pcl::PointCloud<MyPointType>);
pcl::PointCloud<MyPointType>::Ptr mycloud_out(new pcl::PointCloud<MyPointType>);

// ... mycloud_in 초기화

Eigen::Affine3f T = Eigen::Affine3f::Identity();
T.translate(Eigen::Vector3f(1.0f, 0.0f, 0.0f));
T.rotate(Eigen::AngleAxisf(M_PI/4.0f, Eigen::Vector3f::UnitX()));

pcl::transformPointCloud(*mycloud_in, *mycloud_out, T);

이 코드는 MyPointType 구조체에 저장된 점들의 \((x,y,z)\) 좌표값만을 회전 · 평행이동하고, intensity, curvature 같은 다른 필드는 그대로 보존한다.

다른 템플릿 함수와의 혼동¶

PCL에는 pcl::transformPointCloudWithNormals, pcl::transformPointCloudWithNormalsScale 등의 이름을 가진 함수도 일부 구현되어 있다. 이러한 함수들은 노멀 벡터까지 함께 변환하거나, 스케일링(배율) 정보를 함께 적용해야 할 때 사용된다. 그러나 PCL 버전에 따라 함수 시그니처나 지원 여부가 달라질 수 있으므로, 최신 PCL 문서를 확인해야 한다.

만약 노멀을 회전하려면, transformPointCloud 호출 이후에 별도의 코드를 작성하기보다는, 처음부터 노멀 정보를 지원하는 포인트 타입 예: pcl::PointXYZRGBNormal 등을 사용하고 해당하는 변환 함수를 호출하는 편이 편리하다. 예컨대 노멀을 올바르게 회전하기 위해서는

\[ \mathbf{n}' = \mathbf{R}\,\mathbf{n} \]

이 수행되어야 하며, 평행이동은 노멀에 영향을 미치지 않는다. 이런 부분을 자동으로 처리해 주는 템플릿 함수를 사용하면 코드가 단순해진다.

transformPointCloud 적용 시 에러 처리¶

좌표 변환 과정에서 발생할 수 있는 주요 에러나 예외 상황은 다음과 같다.

  • 변환 행렬이 유효하지 않은 경우(예: \(\mathbf{R}\) 부분이 정규직교행렬이 아니거나 행렬의 값이 NaN 등)
  • 입력 포인트 클라우드가 비어 있거나, 포맷이 잘못 지정된 경우
  • Eigen에서 역행렬을 구할 때 행렬이 특이(singular) 행렬인 경우

이런 상황을 방지하기 위해서는 변환 전후에 행렬을 검증하거나, 포인트 수가 0인지 아닌지 확인하고, 예외 처리를 추가해 두어야 한다. 특히 실시간 로봇 운영체제(ROS)나 대규모 지점 클라우드 처리 파이프라인에서는 비정상 입력값에 대해 실패를 빠르게 감지하고 대처하지 않으면 시스템이 다운되거나 다음 단계의 연산이 연쇄적으로 실패할 수 있다.

대규모 데이터 세트 처리 전략¶

수백만 개 이상의 포인트를 갖는 대규모 클라우드에 변환을 적용할 때는, 한꺼번에 모든 데이터를 메모리로 불러와 transformPointCloud를 실행하는 방식이 비효율적일 수 있다. 이런 경우엔 다음과 같은 전략을 고려한다.

  • 블록(block) 단위 스트리밍 포인트 클라우드를 적절한 크기의 블록으로 나누어 순차적으로 변환하고, 변환된 결과를 디스크나 네트워크로 내보내는 방식을 사용한다.
  • 병렬화 PCL 기본 함수들은 CPU 단일 스레드로 동작하지만, 데이터 병렬화를 통해 여러 코어가 동시에 부분 클라우드를 변환하도록 구성할 수 있다. OpenMP, TBB 등으로 루프를 분할하면 성능 향상을 기대할 수 있다.
  • 하드웨어 가속 GPU 상에서 대규모 병렬 연산을 수행할 수 있도록, cuBLAS 등 라이브러리를 활용하여 행렬-벡터 연산을 병렬화하고, 포인트를 대량으로 변환하는 접근 방법이 있다. 이를 위해서는 별도의 CUDA 기반 구현이 필요하지만, PCL 자체와의 호환성 유지에 주의해야 한다.

이처럼 pcl::transformPointCloud는 표준 CPU 환경에서 간편히 사용할 수 있는 범용 함수이지만, 데이터 규모가 커질수록 최적화가 필요하게 된다. 실제 시스템 설계에서는 데이터 양과 연산 자원을 균형 있게 고려하여, 스트리밍, 병렬화, 하드웨어 가속 등 여러 방법을 조합하는 것이 일반적이다.
