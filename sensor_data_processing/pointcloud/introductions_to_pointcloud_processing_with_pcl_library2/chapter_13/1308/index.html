프로젝트의 효율적인 개발과 유지보수를 위해서는 명확한 코드 구조와 견고한 빌드 스크립트가 필수적이다. 본 절에서는 여러 센서를 이용한 3D 스캔 프로젝트의 예제 코드 구조와 이를 지원하는 빌드 스크립트에 대해 심층적으로 분석한다. 특히, PCL 라이브러리를 기반으로 한 모듈화된 설계와 CMake를 활용한 빌드 시스템 설정에 중점을 둔다.

프로젝트 디렉토리 구조¶

프로젝트의 디렉토리 구조는 코드의 가독성과 유지보수성을 높이기 위해 논리적으로 구성되어야 한다. 아래는 예제 프로젝트의 디렉토리 구조이다.

my_3d_scan_project/
├── CMakeLists.txt
├── src/
│   ├── main.cpp
│   ├── sensor/
│   │   ├── sensor_manager.cpp
│   │   └── sensor_manager.h
│   ├── processing/
│   │   ├── point_cloud_processor.cpp
│   │   └── point_cloud_processor.h
│   └── visualization/
│       ├── visualizer.cpp
│       └── visualizer.h
├── include/
│   └── my_3d_scan_project/
│       ├── sensor_manager.h
│       ├── point_cloud_processor.h
│       └── visualizer.h
├── libs/
│   └── external_dependencies/
├── build/
└── tests/
    ├── CMakeLists.txt
    └── test_main.cpp

이 구조는 다음과 같은 원칙을 따른다:

  • src/: 주된 소스 코드 파일들을 포함하며, 기능별로 하위 디렉토리로 구분된다.
  • include/: 공용 헤더 파일들을 저장하여 다른 모듈에서 쉽게 접근할 수 있도록 한다.
  • libs/: 외부 종속성 라이브러리들을 포함한다.
  • build/: 빌드 결과물이 생성되는 디렉토리로, 소스 코드와 분리하여 관리한다.
  • tests/: 단위 테스트 코드와 관련 스크립트를 포함한다.

코드 모듈 설명¶

프로젝트는 주요 기능별로 모듈화되어 있어 각 모듈은 독립적으로 개발되고 테스트될 수 있다. 주요 모듈은 다음과 같다.

센서 모듈¶

src/sensor/ 디렉토리에는 다양한 센서와의 인터페이스를 담당하는 코드가 포함된다. sensor_manager.h 및 sensor_manager.cpp 파일은 여러 센서의 초기화, 데이터 수집, 동기화 등을 관리한다.

// sensor_manager.h
#ifndef SENSOR_MANAGER_H
#define SENSOR_MANAGER_H

#include <vector>
#include <memory>
#include "sensor_interface.h"

class SensorManager {
public:
    SensorManager();
    ~SensorManager();
    void initializeSensors();
    void collectData();
private:
    std::vector<std::shared_ptr<SensorInterface>> sensors_;
};

#endif // SENSOR_MANAGER_H

처리 모듈¶

src/processing/ 디렉토리는 수집된 포인트 클라우드 데이터를 처리하는 기능을 담당한다. point_cloud_processor.h와 point_cloud_processor.cpp 파일은 필터링, 정합, 병합 등의 작업을 수행한다.

// point_cloud_processor.h
#ifndef POINT_CLOUD_PROCESSOR_H
#define POINT_CLOUD_PROCESSOR_H

#include <pcl/point_cloud.h>
#include <pcl/point_types.h>

class PointCloudProcessor {
public:
    PointCloudProcessor();
    pcl::PointCloud<pcl::PointXYZ>::Ptr filterCloud(const pcl::PointCloud<pcl::PointXYZ>::Ptr& cloud);
    pcl::PointCloud<pcl::PointXYZ>::Ptr alignClouds(const pcl::PointCloud<pcl::PointXYZ>::Ptr& source,
                                                    const pcl::PointCloud<pcl::PointXYZ>::Ptr& target);
private:
    // 내부 처리 변수 및 메서드
};

#endif // POINT_CLOUD_PROCESSOR_H

시각화 모듈¶

src/visualization/ 디렉토리는 처리된 포인트 클라우드를 시각화하는 기능을 제공한다. visualizer.h와 visualizer.cpp 파일은 PCL의 시각화 도구를 활용하여 사용자에게 데이터를 표시한다.

// visualizer.h
#ifndef VISUALIZER_H
#define VISUALIZER_H

#include <pcl/visualization/pcl_visualizer.h>
#include <pcl/point_cloud.h>
#include <pcl/point_types.h>

class Visualizer {
public:
    Visualizer();
    void displayCloud(const pcl::PointCloud<pcl::PointXYZ>::Ptr& cloud);
private:
    pcl::visualization::PCLVisualizer::Ptr viewer_;
};

#endif // VISUALIZER_H

빌드 시스템 개요¶

프로젝트는 CMake를 빌드 시스템으로 채택하여 플랫폼 독립적인 빌드 구성을 지원한다. CMake의 모듈화된 특성을 활용하여 각 모듈의 종속성을 명확히 정의하고, 전체 프로젝트의 빌드 과정을 간소화한다.

CMake 스크립트 상세¶

루트 디렉토리에 위치한 CMakeLists.txt 파일은 전체 프로젝트의 빌드 구성을 관리한다. 주요 내용은 다음과 같다.

cmake_minimum_required(VERSION 3.10)
project(My3DScanProject)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# PCL 설정
find_package(PCL 1.8 REQUIRED)

include_directories(${PCL_INCLUDE_DIRS})
link_directories(${PCL_LIBRARY_DIRS})
add_definitions(${PCL_DEFINITIONS})

# 소스 파일 지정
set(SOURCE_FILES
    src/main.cpp
    src/sensor/sensor_manager.cpp
    src/processing/point_cloud_processor.cpp
    src/visualization/visualizer.cpp
)

add_executable(my_3d_scan ${SOURCE_FILES})

target_link_libraries(my_3d_scan ${PCL_LIBRARIES})

이 스크립트는 다음과 같은 단계를 포함한다:

  • CMake 최소 버전 및 프로젝트 이름 설정: cmake_minimum_required와 project 명령어를 통해 프로젝트의 기본 설정을 정의한다.
  • C++ 표준 설정: C++17 표준을 사용하도록 지정한다.
  • PCL 라이브러리 찾기 및 포함 경로 설정: find_package를 사용하여 PCL 라이브러리를 찾고, include_directories와 link_directories로 필요한 경로를 설정한다.
  • 소스 파일 목록 작성: set 명령어로 프로젝트에 포함될 소스 파일들을 나열한다.
  • 실행 파일 생성 및 라이브러리 링크: add_executable과 target_link_libraries를 통해 최종 실행 파일을 생성하고, 필요한 라이브러리를 링크한다.

모듈별 CMake 설정¶

프로젝트의 규모가 커짐에 따라, 루트 CMakeLists.txt 파일에 모든 설정을 포함시키는 것은 비효율적일 수 있다. 이를 해결하기 위해 각 모듈별로 독립적인 CMakeLists.txt 파일을 작성하여 모듈화된 빌드 구성을 유지한다. 아래에서는 각 모듈별 CMake 설정을 상세히 설명한다.

루트 CMakeLists.txt¶

루트 디렉토리의 CMakeLists.txt는 전체 프로젝트의 빌드 구성을 관리하며, 하위 디렉토리의 CMake 설정을 포함한다.

cmake_minimum_required(VERSION 3.10)
project(My3DScanProject)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# PCL 설정
find_package(PCL 1.8 REQUIRED)

include_directories(${PCL_INCLUDE_DIRS} include)
link_directories(${PCL_LIBRARY_DIRS})
add_definitions(${PCL_DEFINITIONS})

# 서브 디렉토리 추가
add_subdirectory(src)
add_subdirectory(tests)

이 설정은 다음을 수행한다:

  • C++ 표준 설정: C++17을 사용하도록 지정.
  • PCL 라이브러리 설정: PCL 라이브러리를 찾고 포함 디렉토리와 링크 디렉토리를 설정.
  • 서브 디렉토리 포함: src와 tests 디렉토리를 빌드 대상으로 추가.

src/CMakeLists.txt¶

src 디렉토리 내의 CMakeLists.txt는 각 모듈별로 하위 디렉토리를 추가하고, 공통 설정을 적용한다.

# src/CMakeLists.txt

# 센서 모듈
add_subdirectory(sensor)

# 처리 모듈
add_subdirectory(processing)

# 시각화 모듈
add_subdirectory(visualization)

# 실행 파일 생성
set(SOURCE_FILES
    main.cpp
)

add_executable(my_3d_scan ${SOURCE_FILES})

# 각 모듈의 라이브러리 연결
target_link_libraries(my_3d_scan
    sensor_module
    processing_module
    visualization_module
    ${PCL_LIBRARIES}
)

이 설정은 다음을 수행한다:

  • 모듈별 서브 디렉토리 추가: sensor, processing, visualization 모듈을 각각 추가.
  • 실행 파일 생성: main.cpp를 포함한 실행 파일 my_3d_scan 생성.
  • 라이브러리 링크: 각 모듈의 라이브러리와 PCL 라이브러리를 실행 파일에 링크.

src/sensor/CMakeLists.txt¶

센서 모듈의 빌드를 관리하는 CMakeLists.txt는 센서 관련 소스 파일을 컴파일하고 라이브러리를 생성한다.

# src/sensor/CMakeLists.txt

set(SENSOR_SRC
    sensor_manager.cpp
)

add_library(sensor_module STATIC ${SENSOR_SRC})

target_include_directories(sensor_module PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}/../include/my_3d_scan_project
)

target_link_libraries(sensor_module
    ${PCL_LIBRARIES}
)

이 설정은 다음을 수행한다:

  • 소스 파일 지정: sensor_manager.cpp를 센서 모듈의 소스 파일로 설정.
  • 정적 라이브러리 생성: sensor_module 정적 라이브러리 생성.
  • 헤더 파일 포함 디렉토리 설정: 공용 헤더 파일 경로를 포함.
  • 라이브러리 링크: PCL 라이브러리를 센서 모듈에 링크.

src/processing/CMakeLists.txt¶

처리 모듈의 CMakeLists.txt는 포인트 클라우드 데이터 처리 관련 소스 파일을 컴파일하고 라이브러리를 생성한다.

# src/processing/CMakeLists.txt

set(PROCESSING_SRC
    point_cloud_processor.cpp
)

add_library(processing_module STATIC ${PROCESSING_SRC})

target_include_directories(processing_module PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}/../include/my_3d_scan_project
)

target_link_libraries(processing_module
    sensor_module
    ${PCL_LIBRARIES}
)

이 설정은 다음을 수행한다:

  • 소스 파일 지정: point_cloud_processor.cpp를 처리 모듈의 소스 파일로 설정.
  • 정적 라이브러리 생성: processing_module 정적 라이브러리 생성.
  • 헤더 파일 포함 디렉토리 설정: 공용 헤더 파일 경로를 포함.
  • 라이브러리 링크: 센서 모듈과 PCL 라이브러리를 처리 모듈에 링크.

src/visualization/CMakeLists.txt¶

시각화 모듈의 CMakeLists.txt는 포인트 클라우드 시각화 관련 소스 파일을 컴파일하고 라이브러리를 생성한다.

set(VISUALIZATION_SRC
    visualizer.cpp
)

add_library(visualization_module STATIC ${VISUALIZATION_SRC})

target_include_directories(visualization_module PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}/../include/my_3d_scan_project
)

target_link_libraries(visualization_module
    processing_module
    ${PCL_LIBRARIES}
)

이 설정은 다음을 수행한다:

  • 소스 파일 지정: visualizer.cpp를 시각화 모듈의 소스 파일로 설정.
  • 정적 라이브러리 생성: visualization_module 정적 라이브러리 생성.
  • 헤더 파일 포함 디렉토리 설정: 공용 헤더 파일 경로를 포함.
  • 라이브러리 링크: 처리 모듈과 PCL 라이브러리를 시각화 모듈에 링크.

외부 종속성 관리¶

프로젝트는 여러 외부 라이브러리에 의존할 수 있으며, 이를 효율적으로 관리하기 위해 libs/ 디렉토리를 활용한다. 예를 들어, 추가적인 센서 SDK나 유틸리티 라이브러리가 있을 경우 이 디렉토리에 포함시킨다.

libs/CMakeLists.txt¶

# libs/CMakeLists.txt

add_subdirectory(external_dependencies)

libs/external_dependencies/CMakeLists.txt¶

외부 종속성의 빌드를 관리하는 CMakeLists.txt는 각 라이브러리에 대한 설정을 포함한다. 예를 들어, libxyz라는 외부 라이브러리를 추가한다고 가정할 때:

# libs/external_dependencies/CMakeLists.txt

add_subdirectory(libxyz)

# 다른 외부 라이브러리들도 동일한 방식으로 추가

이 방식은 외부 라이브러리를 독립적으로 관리하고, 필요에 따라 쉽게 추가하거나 제거할 수 있게 한다.

테스트 설정¶

프로젝트의 신뢰성을 높이기 위해 단위 테스트와 통합 테스트를 설정한다. tests/ 디렉토리는 이러한 테스트 코드를 포함하며, CMake를 통해 빌드 및 실행할 수 있다.

tests/CMakeLists.txt¶

# tests/CMakeLists.txt

enable_testing()

# Google Test 설정 (가정)
find_package(GTest REQUIRED)
include_directories(${GTEST_INCLUDE_DIRS})

# 테스트 소스 파일
set(TEST_SRC
    test_main.cpp
    test_sensor.cpp
    test_processing.cpp
    test_visualization.cpp
)

add_executable(run_tests ${TEST_SRC})

target_link_libraries(run_tests
    sensor_module
    processing_module
    visualization_module
    GTest::GTest
    GTest::Main
    ${PCL_LIBRARIES}
)

add_test(NAME AllTests COMMAND run_tests)

이 설정은 다음을 수행한다:

  • 테스트 활성화: enable_testing()을 통해 테스트 기능 활성화.
  • Google Test 찾기: find_package를 사용하여 Google Test 라이브러리를 찾고 포함.
  • 테스트 실행 파일 생성: run_tests 실행 파일에 테스트 소스 파일 포함.
  • 라이브러리 링크: 모듈 라이브러리와 Google Test, PCL 라이브러리를 테스트 실행 파일에 링크.
  • 테스트 등록: add_test를 통해 모든 테스트를 AllTests로 등록.

tests/test_main.cpp¶

// test_main.cpp

#include <gtest/gtest.h>

int main(int argc, char **argv) {
    ::testing::InitGoogleTest(&argc, argv);
    return RUN_ALL_TESTS();
}

이 파일은 Google Test의 메인 함수를 정의하며, 모든 테스트 케이스를 실행한다.

빌드 구성 최적화¶

빌드 시간을 단축하고, 다양한 빌드 설정을 지원하기 위해 CMake의 캐시 변수와 옵션을 활용할 수 있다.

디버그 및 릴리즈 설정¶

프로젝트는 디버그와 릴리즈 빌드 설정을 지원해야 하며, 각 설정에 맞는 컴파일러 옵션을 적용한다.

# 루트 CMakeLists.txt 내 추가

set(CMAKE_BUILD_TYPE Release CACHE STRING "Build type" FORCE)

if(CMAKE_BUILD_TYPE STREQUAL "Debug")
    add_definitions(-DDEBUG)
    set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -g")
elseif(CMAKE_BUILD_TYPE STREQUAL "Release")
    set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -O3")
endif()

이 설정은 다음을 수행한다:

  • 빌드 타입 설정: 기본 빌드 타입을 Release로 설정.
  • 컴파일러 플래그 추가: Debug 빌드 시 디버그 심볼을 포함하고, Release 빌드 시 최적화 옵션을 추가.

병렬 빌드 지원¶

CMake는 병렬 빌드를 기본적으로 지원하므로, 빌드 시 -j 옵션을 사용하여 여러 코어를 활용할 수 있다. 이는 빌드 시간을 크게 단축시킨다.

cmake --build build/ -- -j$(nproc)

위 명령어는 현재 시스템의 CPU 코어 수를 기반으로 병렬 빌드를 수행한다.

멀티 플랫폼 빌드 고려사항¶

프로젝트가 다양한 플랫폼에서 빌드될 수 있도록 설정을 최적화한다. 예를 들어, Windows와 Linux에서의 차이점을 처리한다.

# 루트 CMakeLists.txt 내 추가

if(WIN32)
    # Windows-specific 설정
    set(EXTRA_LIBS ws2_32)
elseif(UNIX)
    # Unix-specific 설정
    set(EXTRA_LIBS pthread)
endif()

target_link_libraries(my_3d_scan ${EXTRA_LIBS})

이 설정은 플랫폼에 따라 추가적인 라이브러리를 링크한다.

지속적 통합(CI) 설정¶

지속적 통합을 통해 코드 변경 시 자동으로 빌드와 테스트가 수행되도록 설정할 수 있다. 예를 들어, GitHub Actions를 사용하는 경우:

# .github/workflows/ci.yml

name: CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:

    runs-on: ubuntu-latest

    steps:
    - uses: actions/checkout@v2
    - name: Install dependencies
      run: |
        sudo apt-get update
        sudo apt-get install -y build-essential cmake libpcl-dev libgtest-dev
        cd /usr/src/gtest
        sudo cmake .
        sudo make
        sudo cp libg* /usr/lib/
    - name: Build
      run: |
        mkdir build
        cd build
        cmake ..
        make -j$(nproc)
    - name: Test
      run: |
        cd build
        ctest --output-on-failure

이 설정은 다음을 수행한다:

  • 의존성 설치: 필요한 빌드 도구와 라이브러리를 설치.
  • 빌드: 프로젝트를 빌드.
  • 테스트 실행: 모든 테스트를 실행하고 결과를 출력.

빌드 캐싱 및 최적화¶

CMake의 캐싱 기능을 활용하여 빌드 시간을 더욱 최적화할 수 있다. 예를 들어, 외부 라이브러리의 빌드 결과를 캐싱하거나, CMake의 ExternalProject 모듈을 사용하여 종속성 관리를 자동화할 수 있다.

# 루트 CMakeLists.txt 내 추가

include(ExternalProject)

ExternalProject_Add(
    libexternal
    GIT_REPOSITORY https://github.com/example/libexternal.git
    PREFIX ${CMAKE_BINARY_DIR}/external
    INSTALL_COMMAND ""
)

add_dependencies(sensor_module libexternal)

이 설정은 외부 라이브러리를 빌드 프로세스에 통합하고, 필요한 경우 자동으로 빌드 순서를 관리한다.

환경 변수 및 설정 파일 활용¶

빌드 시 다양한 환경 설정을 지원하기 위해 환경 변수나 설정 파일을 활용할 수 있다. 예를 들어, 센서의 설정을 외부 파일에서 로드하도록 구성할 수 있다.

// sensor_manager.cpp

#include <fstream>
#include <json/json.h>

void SensorManager::initializeSensors() {
    std::ifstream config_file("config/sensors.json");
    Json::Value config;
    config_file >> config;

    for(const auto& sensor_config : config["sensors"]) {
        // 센서 초기화 로직
    }
}

이와 같이, 빌드 스크립트에서 설정 파일의 경로를 지정하거나, 빌드 시 특정 설정을 적용할 수 있다.

의존성 관리 도구 통합¶

CMake와 함께 패키지 관리 도구인 vcpkg나 Conan을 통합하여 의존성 관리를 자동화할 수 있다. 예를 들어, vcpkg를 사용하는 경우:

# vcpkg 설치 및 설정
git clone https://github.com/microsoft/vcpkg.git
cd vcpkg
./bootstrap-vcpkg.sh
./vcpkg integrate install

# 필요한 패키지 설치
./vcpkg install pcl gtest

CMake에서 vcpkg를 인식하도록 설정:

# 루트 CMakeLists.txt 상단에 추가

set(CMAKE_TOOLCHAIN_FILE "${CMAKE_SOURCE_DIR}/vcpkg/scripts/buildsystems/vcpkg.cmake" CACHE STRING "")

이 설정은 vcpkg를 통해 설치된 라이브러리를 자동으로 찾아 빌드에 포함시킨다.

지속적 유지보수 및 코드 품질 도구 통합¶

빌드 스크립트에 코드 품질 도구를 통합하여 지속적인 코드 품질 관리를 수행할 수 있다. 예를 들어, clang-tidy나 cppcheck를 통합하는 방법은 다음과 같다.

# 루트 CMakeLists.txt 내 추가

find_program(CLANG_TIDY "clang-tidy")
if(CLANG_TIDY)
    set(CMAKE_CXX_CLANG_TIDY "${CLANG_TIDY};-checks=*")
endif()

이 설정은 clang-tidy를 사용하여 코드 정적 분석을 수행하도록 CMake를 구성한다.
