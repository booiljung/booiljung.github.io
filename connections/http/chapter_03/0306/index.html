HTTP 버전과 호환성¶

HTTP/0.9 시대¶

최초의 HTTP는 매우 단순한 형식을 지녔다. 한 줄의 요청만으로 HTML 문서를 내려받는 역할에 충실했으며, 헤더 개념도 명확히 존재하지 않았다. 서버에 전달되는 메시지는 오직 GET 메서드와 경로로 이루어져 있었고, 응답 또한 단순히 HTML 본문만 반환했다. 이 시기의 HTTP는 더 복잡한 요구 사항이나 콘텐츠 타입 협상 같은 기능을 전혀 고려하지 않았으므로, 대부분의 현대 브라우저나 서버 환경에서 사실상 사용되지 않는다.

구버전의 HTTP와 호환성을 논할 때, HTTP/0.9가 필요한 경우는 거의 없으나 역사를 이해하는 차원에서 언급된다. 최근에 와서는 HTTP/0.9와의 호환성을 제공하는 서버나 프록시는 찾아보기 어려우며, 대부분 HTTP/1.x 이후 버전만 지원한다.

HTTP/1.0 확립¶

인터넷이 급속도로 확산되면서, 단순히 문서만 전송하던 HTTP는 더 풍부한 요구 사항을 처리해야 했다. 그 결과 표준화 과정을 거쳐 처음으로 명확하게 정의된 버전이 HTTP/1.0이다. 주요 특징은 요청과 응답에 헤더를 추가함으로써 확장성을 확보했다는 점이다. 예를 들어 Host 헤더를 통해 가상 호스팅을 지원하고, Content-Type, Content-Length 등을 통해 다양한 콘텐츠를 보다 정교하게 전송할 수 있게 되었다.

HTTP/1.0은 여전히 연결 당 하나의 요청-응답만 처리할 수 있었다. 즉, 요청을 보낼 때마다 TCP 연결이 맺어지고 끊어지는 방식이었다. 이로 인해 왕복 횟수가 잦아져 성능 문제가 대두되었다. 그렇지만 HTTP/1.0은 이후 표준인 HTTP/1.1이 등장하기 전까지 널리 사용되었고, 부분적으로나마 호환성 유지가 가능하다. 많은 서버와 클라이언트가 HTTP/1.0 요청에 대응하는 방식을 여전히 구현하고 있기 때문이다.

HTTP/1.1 표준화¶

HTTP/1.1은 가장 오랫동안 웹 전반에 걸쳐 광범위하게 채택된 버전이다. 현재도 많은 시스템이 HTTP/1.1을 기본으로 사용하고 있다. 가장 큰 개선점은 유지 연결(persistent connection)과 파이프라이닝(pipelining)이 추가되어 한 번의 TCP 연결로 여러 요청을 순차적으로 처리할 수 있게 된 것이다.

이 버전에서 호스트 헤더는 필수로 지정되며, 캐싱 관련 헤더, 콘텐츠 인코딩, 전송 인코딩(Transfer-Encoding: chunked) 같은 다양한 기능이 확립되었다. 이러한 표준화로 인해 서버와 클라이언트가 보다 효율적으로 통신할 수 있게 되었으나, 파이프라이닝 기능의 실제 활용도는 제한적이었다. 동시에 무거운 헤더 구조로 인해 요청 수가 많아질 때 성능 저하가 발생하는 문제도 있었다.

HTTP/1.1은 HTTP/1.0과 대체로 호환된다. 만약 클라이언트가 HTTP/1.0 형식의 요청을 서버로 보낸다면, 대부분 서버는 최소한의 요청 헤더만으로도 처리를 시도한다. 반대로 HTTP/1.1 전용 기능을 사용하려면 해당 기능을 지원하는 서버와 클라이언트여야 한다.

HTTP/2 도입¶

HTTP/2는 기존 HTTP/1.1과 동일한 API와 세멘틱을 유지하면서도, 전송 계층에서 대규모 변화를 시도했다. 텍스트 기반의 헤더와 본문을 이진 프레이밍(binary framing)으로 변환하고, 단일 TCP 연결을 통해 여러 스트림(stream)을 동시에 전송 가능하게 만들었다. 이러한 다중화(multiplexing)를 통해 HTTP/1.1 시절의 문제였던 헤더 오버헤드와 다수의 병렬 요청으로 인한 연결 비용을 크게 완화시켰다.

또한 헤더를 효율적으로 압축하기 위해 HPACK 헤더 압축 방식이 도입되었다. 이를 통해 반복되는 헤더 데이터가 상대적으로 적은 자원으로 전송될 수 있게 되었고, 네트워크 대역폭 사용량이 줄어드는 효과를 볼 수 있었다.

HTTP/2의 호환성은 전반적으로 매우 뛰어나다. 기본적으로 상위 계층의 HTTP 메서드, URI, 상태 코드 등은 기존과 동일하다. 따라서 애플리케이션 로직을 크게 수정하지 않고도 HTTP/2 프로토콜로 전환할 수 있다. 다만 브라우저 지원 측면에서 대부분의 경우 TLS를 통한 암호화 세션을 활용하여 HTTP/2를 수행하며, 명시적으로 HTTP/2를 지원하는 웹 서버가 필요하다.

HTTP/3와 QUIC¶

HTTP/3는 기존의 TCP 기반 프로토콜을 벗어나 QUIC(Quick UDP Internet Connections)를 전송 계층으로 사용한다. QUIC은 UDP를 기반으로 하며, 지연 시간(latency) 감소와 패킷 손실 처리, 연결 핸드셰이크 속도 개선 등에 특화된 구조를 가진다. HTTP/3는 HTTP/2의 이진 프레이밍 및 스트림 개념을 그대로 계승하면서, QUIC을 통해 전송 계층 문제를 해결하려 시도했다.

기존 HTTP/2가 연결 설정 단계에서 TCP 핸드셰이크와 TLS 핸드셰이크 과정을 순차적으로 거쳐야 했던 것에 비해, HTTP/3는 QUIC 프로토콜에 TLS가 결합되어 초기에 보안 채널이 동시에 설정된다. 이를 통해 연결 지연을 줄이고, 네트워크가 불안정한 환경에서도 안정적으로 스트리밍이 가능하도록 설계되었다.

HTTP/3 또한 HTTP/2와 마찬가지로 애플리케이션 관점에서 대부분의 로직 변경을 요구하지 않는다. URI, 메서드, 헤더, 상태 코드, 쿠키 등의 의미 체계가 동일하기 때문이다. 하지만 네트워크 스택 수준에서 UDP 443 포트를 열어야 하고, QUIC 프로토콜을 구현하는 서버가 필요하므로 호환성 확보에는 추가적인 환경 구성이 요구된다.

버전 협상과 ALPN¶

HTTP 클라이언트와 서버는 연결을 맺을 때, 사용하는 프로토콜 버전을 협상한다. 보통 HTTPS를 통해 통신하는 경우, TLS 핸드셰이크 단계에서 ALPN(Application-Layer Protocol Negotiation) 확장 기능을 이용해 서버가 지원 가능한 HTTP 버전 리스트를 제시하고, 클라이언트가 이를 보고 최적의 버전을 선택한다.

예시로 TLS를 사용하는 환경에서, 다음과 같은 과정을 거칠 수 있다.

  • 클라이언트: ALPN 확장으로 ["h2", "http/1.1"]을 보냄
  • 서버: ["h2"] 선택

이 과정을 통해 HTTP/2로 통신이 시작된다. 서버가 HTTP/2를 지원하지 않는다면 자동으로 HTTP/1.1을 사용하도록 폴백(fallback)한다. HTTP/3의 경우 QUIC으로 전송 계층 자체가 달라지므로, 별도의 커넥션 설정 과정을 거치지만 개념적인 협상 과정은 유사하다.

상호 호환성 유지 전략¶

HTTP는 각 버전이 등장할 때마다 완전히 단절된 형태로 새로 설계되기보다, 이전 버전과의 상위 호환성을 최대한 유지하려 노력해 왔다. 그 결과, 다수의 웹 서버는 HTTP/1.0, HTTP/1.1, HTTP/2, 심지어 HTTP/3까지 여러 버전을 동시에 지원할 수 있는 구조를 갖추었다. 브라우저도 마찬가지로, 서버의 응답을 보고 적절한 방식으로 처리하려 한다.

다만 HTTP/3는 전송 계층의 변경이라는 커다란 차이가 있어서 기존 TCP 기반 인프라와 직접적인 호환이 이루어지지 않는다. 따라서 CDN이나 로드 밸런서 같은 네트워크 구성 요소가 QUIC을 지원해야 하며, 방화벽이나 게이트웨이도 UDP 트래픽을 허용하도록 설정되어 있어야 한다.

HTTP 버전별 주요 특징을 간단히 정리하면 아래와 같은 방식으로 요약할 수 있다.

  • HTTP/0.9: 매우 단순한 단일 요청 구조, 헤더 미포함
  • HTTP/1.0: 헤더 개념 도입, 연결 당 한 번의 요청만 가능
  • HTTP/1.1: 유지 연결, 파이프라이닝, 다양한 헤더와 캐시 기능, 가장 널리 보급
  • HTTP/2: 이진 프레이밍, 다중화, 헤더 압축, TLS(ALPN) 기반 협상
  • HTTP/3: QUIC 기반 전송, 빠른 핸드셰이크, 패킷 손실에 강함

결론¶

HTTP는 각 버전이 발전하면서도 가능한 한 이전 버전과의 호환성을 유지해 왔다. 초기의 HTTP/0.9는 현재 표준으로서의 의미가 거의 없지만, HTTP/1.x부터는 근간이 되는 프로토콜 형식이 크게 달라지지 않았다. HTTP/2는 전송 계층을 이진화하고 다중화를 도입해 성능을 개선했으며, HTTP/3는 QUIC을 적용함으로써 지연과 패킷 손실 문제를 대폭 완화하려고 시도하고 있다.

이처럼 HTTP는 오랫동안 단절 없이 발전해 왔고, 다양한 버전이 동시에 사용되는 혼합 환경에서도 상호 호환성을 최대한 유지하려는 특징을 보인다. 실제 운영 환경에서는 서버, 프록시, CDN, 브라우저 각 요소가 상호 지원 가능한 HTTP 버전을 바탕으로 협상하여 최적의 방식을 사용하게 된다. 높은 호환성 덕분에 버전이 다른 여러 클라이언트와 서버가 한데 뒤섞여도 웹이 원활히 동작한다. HTTP/3가 완전히 보편화되더라도, 하위 버전과의 호환성 요구는 당분간 이어질 것으로 예상된다.
