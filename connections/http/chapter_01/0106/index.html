상태 비저장(Stateless) 특성¶

HTTP에서의 상태 비저장 개념¶

HTTP(HyperText Transfer Protocol)은 요청(Request)과 응답(Response)이 단일한 독립 메시지로 처리되는 프로토콜이다. 전송되는 모든 HTTP 메시지는 그 자체로 완전한 의미를 가지며, 별도의 연결 상태나 이전 요청 정보를 프로토콜 차원에서 유지하지 않는다. 이러한 특성을 상태 비저장(Stateless)이라고 한다. 상태 비저장이라는 말은 서버가 각 클라이언트의 과거 요청 내역이나 현재 진행 상황 등을 지속적으로 저장·관리하지 않는다는 뜻이다.

HTTP가 상태 비저장을 구현하는 근간은 초기 설계에서부터 네트워크 확장성과 단순성을 우선시했기 때문이다. 프로토콜 계층에서 기본적으로 상태를 보존하지 않으면 서버와 클라이언트는 오버헤드가 줄어들고, 대규모 트래픽을 처리하기가 용이해진다. 반면 요청을 처리하는 시점마다 클라이언트 상태에 대한 정보를 별도로 확인하거나 전달해야 하므로, 애플리케이션 단에서는 세션 유지 기법 혹은 기타 보조 메커니즘이 필요하다.

요청과 응답의 독립성¶

HTTP 메시지는 매번 새로운 연결로 데이터를 전송하는 것을 전제로 하므로, 이전 통신 맥락이 현재 요청에 암묵적으로 전승되지 않는다. 예를 들어 클라이언트가 다음과 같은 요청을 보냈다고 할 때,

GET /hello HTTP/1.1
Host: www.example.com
Connection: close

서버는 단지 이 요청이 의미하는 바(예: /hello 라는 리소스를 요청함)를 해석하여 응답을 돌려줄 뿐, 이전에 어떤 요청이 있었는지, 클라이언트와 어떤 상태를 공유해야 하는지는 HTTP 자체의 규약으로는 고려하지 않는다. 서버는 응답을 보낸 직후 상태 정보를 별도로 유지하지 않는 것이 기본이며, 이 점이 HTTP가 상태 비저장 프로토콜임을 잘 보여준다.

상태 비저장이 가져오는 특징¶

상태 비저장 프로토콜로서의 HTTP는 여러 측면에서 장점과 단점을 동시에 가진다. 별도의 연결 상태를 유지하지 않으므로 서버에서 필요한 메모리나 시스템 자원의 사용이 상대적으로 적다. 분산 환경에서 다수의 서버를 사용해도 각 서버가 클라이언트별 세션 정보를 공유할 필요가 적어 확장성이 높아진다. 반면 사용자가 로그인을 하거나 특정 맥락에서 연속적으로 작업을 수행해야 하는 상황에서는, 서버가 상태를 직접 추적하지 않기 때문에 별도의 해결책을 사용해야 한다.

세션 유지 기법¶

HTTP가 Stateless하다는 것은 프로토콜 자체에서 상태를 관리하지 않는다는 것을 의미할 뿐, 웹 애플리케이션 수준에서 상태를 전혀 활용할 수 없다는 뜻은 아니다. 실제로는 다양한 세션 유지 기법이 사용된다. 대표적으로 쿠키(Cookie)가 널리 쓰이며, 다음과 같이 Set-Cookie 헤더를 통해 클라이언트에게 고유한 세션 식별자를 제공하고 이후 요청마다 이를 되돌려 받는다.

HTTP/1.1 200 OK
Set-Cookie: SESSIONID=abcdef123456; Path=/

<html>...</html>

서버는 쿠키로 전달받은 세션 식별자에 기초하여 사용자 정보를 연결하고, 그 결과 특정 사용자의 상태를 추적한다. 이처럼 상태 비저장 프로토콜 상에서 상태ful한 서비스를 구현하려면, 애플리케이션 단에서 쿠키, 세션, 토큰 등 별도의 매커니즘을 준비해야 한다.

상태 비저장 프로토콜과 상태ful 연결의 대비¶

HTTP는 본질적으로 상태 비저장적이지만, 다른 프로토콜들은 내부적으로 연결 상태를 유지하거나 클라이언트 세션을 추적하는 경우가 있다. 예를 들어 일부 TCP 기반 프로토콜은 연결이 확립된 동안 상태 정보를 교환하거나, 특정 세션에 대한 맥락을 서버가 보존하기도 한다. 그와 달리 HTTP는 한 번의 요청-응답이 끝나면 그 자체로 하나의 거래가 종료되는 것으로 간주하기에, 별도의 작업 없이 연결이 끊어져도 문제없이 독립 메시지로 통신할 수 있다.

확장성과 단순성의 이면¶

HTTP가 상태 비저장을 유지함으로써 얻는 이점은 매우 크다. 트래픽 규모가 갑작스럽게 확대되어도, 서버는 클라이언트별 세션 상태를 자율적으로 할당할 필요가 없어지는 경우가 많아 수평 확장(Scale-out)에 유리하다. 또 요청과 응답을 단순한 메시지 단위로 처리하기 때문에 애플리케이션 로직을 독립적으로 구성하기가 쉬워진다. 반면 사용자의 상태가 서비스에 매우 밀접하게 연관되어야 하는 애플리케이션이라면, 상태를 추적하기 위한 추가 기법이 불가피하다. 이는 쿠키나 세션, JWT(JSON Web Token) 등을 이용해 HTTP 프로토콜의 한계를 보완하는 식으로 해결한다.

정리¶

HTTP의 상태 비저장(Stateless) 특성은 프로토콜 설계의 핵심 원리로, 각 요청이 과거 맥락에 구애받지 않고 독립적으로 처리된다는 점에서 비롯된다. 이러한 특성은 단순성, 확장성 측면에서 큰 이점을 제공하지만, 동시에 사용자별 상태를 저장하고 활용해야 하는 웹 애플리케이션의 요구에 대응하기 위해 쿠키, 세션, 토큰 같은 추가적인 방법론을 필요로 한다. 결국 HTTP는 상태 자체를 저장하지 않는 구조적 특성을 유지하면서, 필요한 곳에서는 프로토콜 외부 수단으로 상태를 관리하는 방식이 일반적으로 정착되어 왔다.
