TCP와 IP의 간단한 소개¶

인터넷 상에서 이루어지는 모든 데이터 통신은 기본적으로 IP(Internet Protocol)와 TCP(Transmission Control Protocol)의 결합으로 이루어진 프로토콜 스택을 기반으로 작동한다. 흔히 TCP/IP 스택이라 부르며, 네트워크 계층에서 IP가, 전송 계층에서 TCP가 중요한 역할을 담당한다. 이는 HTTP가 동작하기 위해 필수적으로 필요하다. HTTP가 애플리케이션 계층에서 요청과 응답을 전송하는 반면, 그 하부에서는 IP와 TCP가 데이터가 제대로 목적지까지 도달하고 오류 없이 순서대로 전달되도록 신뢰성과 경로 설정을 담당한다.

IP의 역할¶

IP는 네트워크 계층에서 동작하며, 컴퓨터나 장치가 서로 다른 네트워크에 존재하더라도 데이터를 목적지까지 전달할 수 있도록 주소 지정과 라우팅을 책임진다. IP 주소는 인터넷에 연결된 장치를 식별하기 위해 사용되며, 패킷(Packet)이라는 단위로 데이터를 캡슐화한다. IP 패킷에는 헤더(출발지 IP, 목적지 IP 등)와 페이로드(실제 전송되는 데이터)가 포함된다.

인터넷은 수많은 라우터와 중간 네트워크 장치로 이루어져 있고, IP는 이러한 요소들을 적절히 활용해 최적의 경로를 찾아 패킷을 전달한다. 이 과정에서 IP가 확인하는 정보는 기본적으로 목적지 IP 주소이며, 라우터는 자신이 가진 라우팅 테이블에 따라 패킷을 차례차례 다음 라우터나 최종 목적지로 전달한다. IP 계층은 데이터의 순서 보장을 제공하지 않으며, 데이터가 손실 없이 도착할 것이라는 보장도 제공하지 않는다. 오직 베스트 에포트(Best Effort) 방식을 사용한다.

IP에는 여러 가지 버전이 존재하는데, 현재 가장 보편적으로 사용되는 버전은 IPv4와 IPv6다. IPv4는 32비트 주소 체계를 사용하며, IPv6는 주소 고갈 문제 등을 해결하기 위해 128비트 주소 체계를 사용한다. IP가 제공하는 기능은 상대적으로 단순하지만, TCP가 이 상위 계층에서 복잡한 신뢰성 보장 기능을 수행하도록 돕는다.

TCP의 역할¶

TCP는 전송 계층에서 동작하며, 네트워크 계층의 IP가 제공하지 않는 연결 지향형 통신, 신뢰성 보장, 흐름 제어, 혼잡 제어 등을 구현한다. IP 패킷이 라우팅 과정에서 지연 혹은 손실될 수 있고, 순서가 뒤바뀔 수도 있는데, TCP는 이를 해결하기 위해 세그먼트(Segment) 단위의 전송과 확인 응답(ACK) 메커니즘을 사용한다.

TCP에서는 연결을 맺기 위해 3-웨이 핸드셰이크(3-way handshake)를 수행한다. 연결이 성공적으로 맺어진 뒤에는, 송신 측이 전송한 세그먼트를 수신 측이 올바르게 받았는지 ACK로 확인하고, 손실된 세그먼트가 있으면 재전송을 시도한다. 이렇게 하여 애플리케이션 계층에서 전달되는 데이터가 목적지에 정확하고 순서에 맞게 도달하도록 보장한다.

TCP는 흐름 제어(Flow Control)를 통해 수신 측이 처리 가능한 만큼만 데이터를 전송하도록 제어하며, 혼잡 제어(Congestion Control)를 통해 네트워크 상황에 맞춰 전송 속도를 조절한다. 이러한 메커니즘은 네트워크 전반의 혼잡을 방지하고, 전체적인 데이터 전송 효율을 높이는 데 기여한다.

HTTP와의 상관관계¶

HTTP가 애플리케이션 계층에서 동작할 때, 실제 데이터 전송은 TCP를 통해 이뤄지며, IP 계층을 통해 패킷이 전달된다. 즉, HTTP 요청이나 응답은 TCP 세그먼트에 담겨 IP 패킷으로 캡슐화된 뒤 인터넷 상을 흐른다. 따라서 IP와 TCP가 없다면 HTTP 같은 높은 수준의 프로토콜은 동작할 수 없다.

TCP/IP는 범용적인 네트워크 프로토콜이므로, HTTP뿐 아니라 SMTP나 FTP 등 다양한 애플리케이션 계층 프로토콜이 동일한 전송/네트워크 계층을 공유한다. 이러한 구조 덕분에 인터넷에서 수많은 서비스가 단일한 네트워크 인프라를 활용해 유연하게 동작할 수 있다.
