HTTP 개요¶

요청과 응답의 기본 개념¶

HTTP는 웹 클라이언트와 서버가 서로 정보를 주고받기 위한 프로토콜이다. 이때 모든 정보 교환의 근간이 되는 구조는 요청(request)과 응답(response)이다. 클라이언트는 서버에게 명시적인 요구 사항을 요청 메시지 형태로 전달하고, 서버는 이에 대한 처리 결과를 응답 메시지 형태로 돌려준다. 이 과정을 통해 웹 브라우저가 페이지를 로드하거나, REST API가 특정 데이터를 얻거나, 파일을 업로드하거나 다운로드하는 등 다양한 기능이 가능해진다.

클라이언트의 요청 메시지는 크게 요청 줄(request line), 요청 헤더(request headers), 요청 본문(request body)으로 구성된다. 요청 줄에는 메서드, 요청 대상(주로 경로 형태로 기술), 그리고 HTTP 버전이 들어 있다. 예를 들어 GET /index.html HTTP/1.1과 같이 작성된다. 요청 헤더는 호스트(Host), 사용자 에이전트(User-Agent), 수락 가능한 미디어 타입(Accept) 등의 메타정보로 이루어지며, 필요에 따라 다양한 커스텀 헤더가 추가될 수 있다. 요청 본문은 메서드 종류에 따라 포함될 수도, 생략될 수도 있는데, 예컨대 POST 메서드를 사용하여 폼 데이터를 서버에 전송하는 경우 요청 본문에 해당 데이터가 담긴다.

서버는 클라이언트 요청을 받은 뒤, 이를 처리하고 결과를 응답 메시지로 내려보낸다. 응답 메시지는 상태 줄(status line), 응답 헤더(response headers), 응답 본문(response body)으로 이루어지며, 상태 줄에는 사용된 프로토콜 버전, 상태 코드, 그리고 해당 상태 코드를 설명하는 이유 구문(reason phrase)이 포함된다. 예시로 HTTP/1.1 200 OK 같은 형태이다. 응답 헤더 역시 요청 헤더와 유사하게 서버, 콘텐츠 유형(Content-Type), 콘텐츠 길이(Content-Length) 등에 관한 정보를 담는다. 응답 본문은 HTML, JSON, XML, 이미지 등 어떠한 형태의 데이터든 올 수 있으며, 클라이언트가 요청한 내용을 담아 전달한다.

HTTP가 요청과 응답이라는 두 가지 메시지 형태로 동작한다는 점에서 중요한 특징은 상태 유지가 없는(stateless) 프로토콜이라는 것이다. 요청과 응답 사이에 별도의 연결 정보가 유지되지 않으므로, 서버는 매 요청마다 클라이언트를 새로 인지하는 셈이다. 따라서 로그인을 포함한 사용자별 세션 관리가 필요한 경우, 쿠키나 세션, 토큰 등을 이용해 사용자를 식별하는 기법이 추가적으로 쓰인다.

다음 예시는 HTTP/1.1을 기준으로 한 간단한 요청과 응답 구조의 예시다.

GET /index.html HTTP/1.1
Host: www.example.com
Accept: text/html
Connection: close

위 요청은 클라이언트가 www.example.com 서버에 있는 /index.html 리소스를 GET 메서드로 요청하는 상황을 나타낸다. 특정 헤더 필드는 HTML을 받을 수 있음( text/html )을 전달하고, 요청 후 연결을 종료하겠다고(Connection: close) 명시한다.

서버는 아래와 유사한 형태로 응답할 수 있다.

HTTP/1.1 200 OK
Date: Tue, 23 Jan 2025 12:00:00 GMT
Server: Apache
Content-Type: text/html
Content-Length: 125

<html>
<head>
  <title>Example Page</title>
</head>
<body>
  <p>Hello World!</p>
</body>
</html>

응답의 첫 줄인 상태 줄에서 200 OK는 요청이 정상 처리되었음을 의미한다. 이어지는 헤더 필드는 서버 시각, 서버 종류, 콘텐츠 유형 및 길이 등의 정보를 표현한다. 마지막에 표시된 HTML 문서가 실제로 클라이언트에게 전송되는 응답 본문에 해당한다.

요청과 응답의 주고받음은 가장 기초적이면서도 핵심적인 HTTP 동작 방식이다. HTTP/1.1에서 한 번의 TCP 연결로 여러 요청을 파이프라인(pipelining) 방식으로 연달아 보낼 수 있도록 개선되었으며, HTTP/2나 HTTP/3에서는 더 효율적인 다중화(multiplexing)와 전송 방식이 추가되어 성능이 향상되었다. 하지만 이러한 변화에도 요청과 응답이라는 기본적인 개념 자체는 변함없이 유지된다.

이처럼 요청과 응답은 HTTP 프로토콜이 동작하는 데 있어 중심이 되는 요소다. 모든 추가 기능이나 확장 사양, 최적화 기법은 결국 요청을 어떻게 보내고, 응답을 어떻게 전달할지를 개선하거나 보완하는 형태로 설계된다. 따라서 HTTP에 대한 깊은 이해는 요청 메시지와 응답 메시지가 어떤 구조로 이루어져 있고, 실제 전달되는 데이터가 어떤 맥락에서 동작하는지를 파악하는 데서 시작한다고 할 수 있다.
